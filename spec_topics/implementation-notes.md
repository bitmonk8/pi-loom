# Implementation Notes

This page collects the parser, runtime, and orientation notes. Two more focused implementation pages are split out:

- [Runtime Value Model](./runtime-value-model.md) — how Loom values are represented at runtime, equality semantics, wire-name translation.
- [Pi Integration Contract](./pi-integration-contract.md) — the named SDK surface from `@mariozechner/pi-coding-agent` the runtime depends on.

## Parser

- **Contract.** The parser MUST emit diagnostics matching [Diagnostics](./diagnostics.md), including multi-error reporting across the whole `.loom` or `.warp` file and its transitively imported `.warp` modules before failing. Parse errors surface as structured diagnostics compatible with Pi's error reporting.
- **Implementation hint (non-normative).** The reference implementation uses Chevrotain (TypeScript-native, no separate lexer generator required), with the grammar derived from the Chevrotain TinyC example and extended with `@`...`` query templates, `schema`, and the type-context schema-inference pass. The choice of toolkit and grammar lineage have no observable effect on a loom author and may be replaced by any implementation that satisfies the contract above.

## Runtime

- Implemented in TypeScript as a Pi extension module
- Holds a reference to the target conversation (the caller's session for prompt mode; a freshly spawned isolated session for subagent mode) and drives it turn-by-turn
- Each `@`...`` query issues a user turn against that conversation, services any tool-call loop the model returns, and resolves to the final assistant response
- Typed queries (schema inferred from binding type, function parameter, return type, or explicit `@<Schema>`...``) lower the schema to the provider's structured-output / strict tool-input contract; the returned response is validated with **AJV** before being handed back to loom code. The behavioural contract for typed queries is in [Pi Integration Contract](./pi-integration-contract.md); alternative techniques satisfying that contract are permitted. The V1 reference implementation uses a synthesised one-shot tool named `__loom_respond_<slug>` (where `<slug>` is the canonical schema hash of the lowered query response schema, computed per [Schema Subset — Canonical schema hash](./schema-subset.md#canonical-schema-hash)). The tool's `parameters` is the lowered query schema wrapped via `Type.Unsafe<unknown>(loweredJsonSchema)` (per the bridging convention in [Pi Integration Contract — Per-loom registration](./pi-integration-contract.md)), the tool's `label` is the literal string `"Loom typed-query response"`, and the tool's `execute` records the validated args (after AJV-checking the `unknown`-typed `params` against the original lowered JSON Schema) and resolves the query's promise. The tool's *registration lifetime and visibility* is governed by [Pi Integration Contract — Tool-registration lifetime and visibility](./pi-integration-contract.md): in **subagent mode** the tool is passed to `createAgentSession({ customTools, tools, ... })` and lives only inside the spawned `AgentSession`; in **prompt mode** the tool is registered once per unique `<slug>` via the registration cache (subsequent typed queries with the same lowered schema reuse the same registration) and made visible for the typed-query turn via the `pi.setActiveTools` snapshot/restore protocol. Pi has no `unregisterTool` API; the spec deliberately does not call one. Tool-use is forced for the typed-query turn via the `before_provider_request` hook with provider-specific `tool_choice` payloads (`tool_choice: { type: "tool", name }` for Anthropic, `tool_choice: { type: "function", function: { name } }` for OpenAI). The `before_provider_request` handler the loom installs fires on every provider request in every session into which the loom extension is loaded — including the user's untyped turns, compaction calls, and other extensions' (or the loom's own) subagent sessions — so it MUST **fingerprint** each payload: it inspects `payload.tools` for an entry whose `name` matches a `__loom_respond_<slug>` for a currently-pending typed query, and only when found does it rewrite `tool_choice` to that name; all other payloads are returned unchanged. The currently-pending set is keyed by tool name (per-runtime, O(1) lookup) so the same handler covers concurrently-in-flight typed queries and the coercion follow-up turns that reuse the same `__loom_respond_<slug>` tool, and so subagent-mode typed queries reuse the prompt-mode handler with no second install path. Because `BeforeProviderRequestEvent.payload` and `BeforeProviderRequestEventResult` are both typed `unknown` in `@mariozechner/pi-coding-agent` and the concrete shape is owned by `@mariozechner/pi-ai`'s `streamSimple` request, this fingerprinting-and-rewrite step is version-coupled to `@mariozechner/pi-ai` and MUST be re-validated on every upgrade of that package; the no-match path returns `event.payload` unchanged (returning `undefined` is also accepted by the runner's `if (handlerResult !== undefined)` guard, but explicit pass-through is preferred so the handler survives a future tightening of that contract). The synthesised-tool turn is observably equivalent to a native structured-output turn: the user sees one assistant turn carrying the validated payload, matching the behavioural contract's "no extra user-visible turns" clause
- Tool calls from loom code (`<name>(args)`) dispatch through the per-loom callable table that was materialised at load time; resolution is by held reference, not by re-querying Pi's tool registry on every call (see [Parameters and Frontmatter — Resolution snapshot](./frontmatter.md)). For Pi tools the runtime invokes the captured `ToolDefinition`'s `execute(toolCallId, params, signal, onUpdate, ctx)` directly with a synthesized `toolCallId` and a no-op `onUpdate`, and awaits the resulting Promise before returning to loom code (non-blocking at the runtime level, sequential at the language level); for registered loom paths the runtime spawns a subagent invocation equivalent to `invoke<T>(path, ...)`. Pi tools registered in `tools:` are also wired into every `@`...`` query as model-callable tools; registered loom paths are lowered to a tool spec (params → input schema, inferred return → output schema, frontmatter `description` → tool description) and exposed alongside Pi tools to the model
- For subagent-mode looms, the spawned conversation's system prompt is taken from frontmatter `system:` (with `${param}` interpolation resolved at conversation-creation time) and applied to every query against that conversation
- Parameter schemas (frontmatter `params`) are likewise validated with AJV at invocation time
- For slash-command invocation, the runtime first runs the binder (per [Slash-Command Argument Binding](./binder.md)) unless the bypass condition holds. The binder is a one-shot ephemeral call to `bind_model` (resolved from frontmatter, falling back to the loom-extension setting `looms.binderModel`; see [Settings file reads](./discovery.md#settings-file-reads)) with `temperature: 0` and a fixed seed where supported. The runtime constructs the binder's response schema dynamically from the loom's `params:` schema (the three-arm envelope `ok | needs_info | ambiguous`), passes it as a strict structured-output contract, and validates the returned `args` (on `kind: "ok"`) with AJV before merging in defaults and starting the loom. Binder failures and non-`ok` envelopes are surfaced as one-line system notes in the user's session; the loom is not started
- **Schema validation.** Schema validation is provided by an injected `SchemaValidator` service (see [Pi Integration Contract](./pi-integration-contract.md) and the H2 DI seams). The validator's behavioural contract — what any conforming implementation must satisfy:
    - Validation MUST report every validation error in a single pass (no fast-fail) so [Diagnostics](./diagnostics.md) can present them all at once.
    - Validation MUST NOT coerce values and MUST NOT fill defaults — model output is taken as-is, and loom does no implicit type conversion beyond the validator's own pass.
    - `$ref` MUST resolve only within the lowered per-query schema document (the document produced by step 4 of the lowering algorithm in [Schema Subset — Lowering Algorithm](./schema-subset.md#lowering-algorithm)); external `$ref`s are not supported because lowered schemas never produce them.
    - Validation MUST silently accept any JSON-Schema `format` keyword that may appear in a value's schema. Loom-emitted schemas never use `format`, but model output can carry one; an implementation that raises on unknown formats violates this contract.
    - Validation results MUST be deterministic for a given (schema, value) pair.
  Architectural constraints carried by the DI seam itself: one `SchemaValidator` instance is constructed per runtime instance — never as a module-level global — and lives for the lifetime of that runtime. Two concurrent runtime instances (e.g. parallel tests) each get their own; sharing one across runtimes is disallowed. Cache invalidation on file change is a property of the validator service — the file watcher calls into the service's invalidate path; it does not reach into any module-scoped state.
- **Implementation hint (non-normative).** The reference implementation uses AJV v8 (`ajv ^8`, `ajv-formats ^3`), declared in loom's own `dependencies` rather than provided by Pi, with `strict: false`, `allErrors: true`, and `ajv-formats` registered as an instance-level side effect performed by each `SchemaValidator` on its own `Ajv`. These flag values are how AJV happens to satisfy the contract above: `allErrors: true` for one-pass multi-error reporting, the absence of AJV's `coerceTypes` / `useDefaults` options for no-coercion / no-defaults, AJV's default in-document `$ref` resolver for the ref-scope rule, and `ajv-formats` for silent format-keyword acceptance. The reference implementation also caches compiled validators per lowered-schema document, keyed by a stable content hash of that document plus the AJV major version; the file watcher MAY evict cache entries belonging to changed files as a memory-management optimisation, but cache hit/miss policy is an optimisation and does not affect the deterministic-validation contract above
- **Per-invocation single-threaded execution.** The loom interpreter runs as plain async TypeScript on Pi's event loop. There are no worker threads and no loom-level concurrency primitive in V1. Within a single invocation the interpreter is strictly sequential: every `await` (query, tool call, invoke) yields the event loop and the next loom expression cannot run until the awaited Promise resolves, so an invocation can never re-enter itself. Concurrent invocations are independent: two slash-command invocations of the same loom run sequentially against the user's session in prompt mode, and each gets its own private `AgentSession` in subagent mode — transcripts, captured `tools:` tables, and derived `AbortSignal`s are per-invocation and do not leak across siblings. The schema-validator cache and the file-watcher are shared across invocations but are read-mostly: compiled validators are immutable post-compile and cache writes only occur on initial compile or watcher-driven invalidation, so concurrent reads from sibling invocations are safe without locking. Sibling invocations can interleave on the event loop — this happens whenever the model emits parallel tool calls into the same registered subagent-mode loom (see [Tool Calls — Concurrency](./tool-calls.md))
- The loom's overall return value is the value of the last expression of its top-level block
