## Finding

The *Runtime construction obligation* paragraph in `diagnostic-shape.md` (the `session-shutdown-details-conventions` block) carries two layered prescriptions. The first is behavioural: the constructed `details.event` object MUST contain exactly the pinned fields, and the field set MUST be sourced "from this paragraph" rather than from the SDK-delivered `SessionShutdownEvent` payload. That invariant is what the operator-tooling consumers' strict-validation entitlement and the future-SDK-addition safety property both rely on, and the paragraph already states it cleanly with the "no … dynamic-field-set mechanism whose member set is sourced from the SDK-delivered `SessionShutdownEvent` rather than from this paragraph" clause.

The second layer is mechanism: enumerated bans on `...event` spread, `Object.assign(..., event, ...)`, and per-key copy loops over `Object.keys(event)`, plus the closing "with each pinned field assigned by an explicit static property name." These are stricter than the behavioural invariant. A static allow-list construction — e.g. an iteration over the literal tuple `["reason", "loom", "invocation_id"] as const`, a `_.pick(event, [...staticFieldList])` driven by a hard-coded list, or a builder over a static schema — achieves the spec-pinned field set without any SDK-sourced membership, but is forbidden by the per-key-loop ban and by the explicit-static-property-name mandate even though it preserves the property the paragraph cares about (a future `event.timestamp` cannot leak onto the wire because the iteration source is the static tuple, not `Object.keys(event)`).

Drop the mechanism mandate and keep the behavioural invariant. The "field set sourced from this paragraph, not from `SessionShutdownEvent`" clause already gives a future contributor and a future reviewer the right test to apply when judging a refactor, and any construction that satisfies it satisfies the operator-tooling entitlement by construction on the same footing as the currently-blessed `{ reason }` / `{ reason, loom, invocation_id }` object literals.

## Solution Space

**Shape:** single

### Recommendation

Edit the *Runtime construction obligation* paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` so it pins only the behavioural invariant. Concretely:

1. Keep the lead obligation that the runtime MUST construct `details.event` as a fresh object whose member set is sourced from this paragraph rather than from the SDK-delivered `SessionShutdownEvent`, and keep the per-code field-set enumerations (`{ reason }` for `loom/host/session-shutdown-runtime-degraded`, `{ reason, loom, invocation_id }` for `loom/runtime/cancelled-by-session-shutdown`).
2. Drop the parenthetical "(no `...event` spread, no `Object.assign(..., event, ...)`, no per-key copy loop over `Object.keys(event)`, and no other dynamic-field-set mechanism whose member set is sourced from the SDK-delivered `SessionShutdownEvent` rather than from this paragraph)" and the trailing clause "with each pinned field assigned by an explicit static property name."
3. Replace it with a single sentence pinning the static-source property — e.g. "The member set of the constructed object MUST be sourced statically from this paragraph; any construction whose member set could change without a spec edit to this paragraph (notably any mechanism that derives the member set from `Object.keys(event)`, `Reflect.ownKeys(event)`, a `...event` spread, or an equivalent SDK-payload-driven iterator) is non-conformant." This preserves the operator-tooling entitlement and the future-`event.timestamp` safety property without naming `Object.assign` / `...event` / property-name syntax specifically.
4. Leave the "future SDK addition (e.g. a hypothetical `event.timestamp`) does not silently leak onto the wire under a future contributor's spread-style refactor" rationale sentence untouched — it captures the property the relaxed pin still enforces.

Edge case for the implementer: the rephrased pin must still forbid a `Reflect.ownKeys(event)`-driven loop (and a `for…in event` loop), since both source the member set from the SDK payload. The static-source framing already covers them, but call it out explicitly so a reviewer auditing the diff does not mistakenly read the relaxation as opening that surface.

## Consequence

**Severity:** advisory

A future contributor who reaches for a static-allow-list construction (an iteration over a const tuple of field names, a `_.pick` with a static list, a TypeBox-style schema builder) gets blocked by the explicit-property-name mandate even though their construction preserves the load-bearing invariant. The likely outcome is a refactor either rejected at review on a mechanism-not-behaviour basis or routed through a spec edit that the relaxation would have avoided.
