## Finding

The implementation leaf `V11h` (Argument echo) declares `Deps. V11h-T, V11g, V2d, V2c`: it composes the canonical number renderer from `V2d` and obtains that renderer's `integer`-vs-`number` kind discriminator from the bound value's static type via the runtime value model (`V2c`). Its paired tests task `V11h-T` declares `Deps. V11g, V2d` — omitting `V2c`.

`V11h-T`'s behavioural test reproduces the BNDR-6 echo reference renderings (rows 6a–6x) exactly. Constructing the bound-`args` inputs those rows render requires the `V2c` value model: the enum rows (6w `Severity.High` → `High`, 6x `Severity.NeedsReview` → `"needs review"`) need an enum value carrying `V2c`'s declaring-enum tag, and the object/array rows (6g–6l) need `V2c`'s structural value representation. The tests task therefore has a direct prerequisite on `V2c` that its `Deps.` field does not record, while the paired implementation already does.

Because `V2c` lands in the early `V2` slice — far ahead of the `V11` binder slice — the omission does not break the dep-driven build order in practice; `V2c` is complete long before `V11h-T` becomes reachable. The defect is an incomplete `Deps.` declaration that diverges from the parity its paired implementation maintains, not an executability break.

## Issue introduction

**Verdict:** introduced-by-later-edit
**Introducing commit:** `4807958` (2026-06-17) — "pi-loom plan: resolve \"renderer input contract: integer-vs-number discriminator origin\""
**History:** `V11h-T-argument-echo.md` was created with `Deps. V11g, V2d` in `cb8737a` (2026-06-12, "Systemic leaf over-bundling across the plan corpus") and has never listed `V2c` — `git log -S 'V2c'` over the file returns no commit. `V2c` entered the paired implementation leaf `V11h` only in `4807958`, which added both `V2c` to `V11h`'s `Deps.` and the "kind discriminator … via the value model (`V2c`)" prose to its `Adds.`. That commit edited `V11h`, `V2d`, and `V5f` but did not touch `V11h-T`. The impl/test asymmetry therefore dates from `4807958`: the discriminator-origin edit added the value-model dependency to the implementation leaf without propagating it to the test leaf that reconstructs the same value-model inputs.

## Solution Space

**Shape:** single

### Recommendation

Add `V2c` to `V11h-T`'s `Deps.` field in `docs/plan_topics/V11h-T-argument-echo.md` so it reads `**Deps.** V11g, V2d, V2c`, mirroring the `V2c` dependency the paired `V11h` already lists. This records the value-model prerequisite the BNDR-6 enum rows (6w/6x) and structural-object rows (6g–6l) need to construct their bound-`args` test inputs.

Cite `V2c` exactly — the existing implementation leaf ID, not the `V2c-T` tests task — because `V11h-T` constructs value-model values that exist only once `V2c` is complete, consistent with the pickability predicate keying on the `<id>-complete` tag. The fix is internal to this plan leaf: do not edit the spec or any other leaf.

## Consequence

**Severity:** advisory

An implementer reading `V11h-T`'s `Deps.` sees only `V11g` and `V2d`, understating the leaf's real prerequisites: the BNDR-6 enum rows (6w/6x) and structural-object rows (6g–6l) cannot be set up as bound-`args` inputs without the `V2c` value model. In the canonical slice order `V2c` is already complete by the time `V11h-T` is picked, so no build-order break occurs; the cost is a `Deps.` field that does not match the test's actual inputs and diverges from the parity the paired implementation leaf maintains.
