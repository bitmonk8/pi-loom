## Finding

The *Wire serialisation* clause in `audit-wire-and-canary.md` packs each record as four tab-delimited tokens on one stdout line and obliges every field value — including `path` — to contain no ASCII tab (U+0009). The same sentence justifies that obligation for `path` by claiming "bare identifiers, package-qualified imported names, the literal `<n/a>` sentinel, integer `line` values, and file paths are tab-free by their pinned shapes." That justification holds for the other listed values (each is constrained elsewhere on the page) but fails for `path`: nothing on the page, in `audit-resolution.md` (*Audit scope*, *Cross-platform glob normalisation*, *Source-file encoding*), or in `audit-failures.md` (*Failure-surface contract*, *Per-family record-shape table*) constrains the *content* of an audited source path. POSIX permits U+0009 in pathname components, and TypeScript / `tsc` likewise accept it; a contributor who creates `src/foo\tbar.ts` (or whose tooling generates one) produces a `path` value with an embedded tab.

The downstream consequence is a silent corruption of the wire contract a CI parser is explicitly permitted to rely on. A tab-bearing `path` makes the leading-token-prefix routing rule still work (the discriminator is the first whitespace-separated field), but the "split each line on ASCII tab obtains four fields" rule the same paragraph promises fails: the parser obtains five or more fields, and the right-most-`:` rule used to recover `path` and `line` from segment (1) then runs against the wrong substring. The record is malformed under PIC's own contract, yet the audit emits no infrastructure-failure record — the *Infrastructure-failure handling* surface enumerates parse / read / import / crash failures, not "audited path contains a wire-illegal character," so the failure is silent-by-omission rather than fail-closed.

The defect is a missing precondition: the wire format's no-tab guarantee for `path` is asserted but not established anywhere in the audited-source-tree definition.

## Solution Space

**Shape:** single

### Recommendation

Add one normative sentence to `audit-resolution.md` *Audit scope* (the existing home for source-tree preconditions like UTF-8 encoding and symlink handling) stating that an audited source file's path MUST NOT contain the ASCII tab character (U+0009), and that the audit MUST surface a tab-bearing path under the *Infrastructure-failure handling* implementation-agnostic category (i) — on the same footing as a UTF-8 decode failure — with `path` carrying the offending pathname (the violating character is preserved verbatim in the failure record because the failure record is itself the place that escapes the wire contract; the alternative of stripping or replacing the tab inside the failure-record `path` would hide the offending byte from the contributor's diff-reading recovery). Replace the unsupported "file paths are tab-free by their pinned shapes" clause in `audit-wire-and-canary.md` *Wire serialisation* with a back-reference to the new *Audit scope* precondition.

Implementer-relevant edge cases the rule must cover:

- The detection step must run before record emission, so a tab-bearing `path` cannot escape via the *Pre-emission termination carve-out*; pinning it under the (i) implementation-agnostic category is what makes that ordering load-bearing.
- The infrastructure-failure-record's own `path` field, by design, carries a tab-bearing string when that is the failure being reported. That single record is therefore the only wire-permitted exception to the no-tab rule, and the *Failure-surface contract*'s per-line wire serialisation reads it as having five (or more) tab-delimited tokens — a CI parser MUST identify this record by the `audit/infra/...` leading discriminator and skip the four-field decode for it. PIC should state that explicitly rather than leave the parser to discover it.
- The rule applies uniformly to direct-tree files and to files reached via inclusion (any future include-glob widening or symlink-following exemption inherits the precondition without re-litigation).

Escaping the tab inside the `path` field instead (e.g. percent-encoding, backslash-tab) is rejected: it would require parser-side decoding to recover the on-disk path, would interact unpredictably with the segment-(1) right-most-`:` recovery rule, and would push complexity onto every CI consumer for a precondition the audited-source-tree scope can simply forbid at zero implementation cost on every realistic workflow.

## Consequence

**Severity:** correctness

A conformant audit implementation walking a source tree that happens to contain a tab-bearing path will emit a record that two reasonable CI-parser implementations decode into different field tuples (or into a different number of fields), defeating the documented "splitting each line on the ASCII tab obtains four fields" guarantee. The audit's gate would report green while emitting unparseable records, and the silent failure is exactly the surface the *Non-empty-scan canary* and *Infrastructure-failure handling* sections were designed to prevent.
