## Finding

`V17a` Adds describes itself as owning "the cancellation-runtime *semantics* of `SHUTDOWN_AWAIT_CAP_MS = 2000` … as a conceptual consumer", and then states the value "physically resides in the single source-of-truth pinned-constants block established by `V9a`" while naming only `V9g` (session-shutdown await cap) and `V9i` (subagent-disposal bound) as the leaves that "consume that same block constant". On its own surface, V17a presents itself as a semantics owner with no code-level reference to the constant.

`V9a` Adds, however, states the constant is the one "which `V18c`'s build-time literal-read assertion reads and `V9g`, `V9i`, and `V17a` import rather than redeclare" — explicitly listing `V17a` among the runtime *importers*. The two leaves contradict each other: V9a asserts V17a holds a code-level `import` of the constant, while V17a's own framing ("conceptual consumer") implies a purely logical/semantic dependency with no code reference. An implementer of V17a cannot tell whether to write an `import { SHUTDOWN_AWAIT_CAP_MS }` against V9a's block or to treat the constant as documentation of the contract that V17a never references in code. The phrase "as a conceptual consumer" is the locus of the ambiguity — it is undefined and admits both readings.

The actual cancellation mechanics resolve which reading is correct: the cap-bounded await is `await Promise.allSettled(activeInvocations.map(inv => inv.disposeBarrier))` bounded by `SHUTDOWN_AWAIT_CAP_MS`, defined at `patch-skew-degradation.md` §`session_shutdown` sub-step 3 and enforced by `V9g` (and covered for disposal by `V9i`). `V17a` ships the `loomAbort` controller and the late-settlement discard rules; it performs no bounded await and its Tests reference no cap. V17a therefore has no functional need to import the value — its role is to own the semantics, not to read the literal.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 622d0d3 — pi-loom plan: resolve "SHUTDOWN_AWAIT_CAP_MS physical home contradicts pinned-constants-block placement" (2026-06-11, Thomas Andersen); 2f4ecea — pi-loom plan: resolve "V9a Adds prescribes source-code layout instead of export/import obligations" (2026-06-17, Thomas Andersen)
**History:** Commit 622d0d3 rewrote V17a's Adds to call the leaf "a conceptual consumer" of the constant's semantics, moving the physical home to V9a's pinned-constants block and naming only V9g and V9i as the leaves that "consume that same block constant" — V17a was framed as a semantics owner, not a code importer. Six days later, commit 2f4ecea reworded V9a's Adds from the earlier "`V9g`, `V9i`, and `V17a` consume it without redeclaring it" to the stronger "`V9g`, `V9i`, and `V17a` import rather than redeclare", promoting V17a from conceptual consumer to code-level importer in V9a's prose without reconciling V17a's own "conceptual consumer" framing. The contradiction between the two leaves is the joint product of those two edits.

## Solution Space

**Shape:** single

### Recommendation

Make V17a and V9a state the same thing about V17a's relationship to the constant: V17a owns the cancellation-runtime semantics of `SHUTDOWN_AWAIT_CAP_MS` but does not itself read the literal in code; the runtime importers are `V9g` and `V9i`.

- In `docs/plan_topics/V17a-cancellation-core.md` Adds, strike the phrase `as a conceptual consumer` and replace it with a concrete statement of V17a's role: V17a defines/owns the cancellation-runtime semantics of `SHUTDOWN_AWAIT_CAP_MS` (what the cap means within the cancellation contract) but does not import or reference the constant in its own code; the value physically resides in V9a's pinned-constants block and is read at runtime by `V9g` (session-shutdown await cap) and `V9i` (subagent-disposal bound).
- In `docs/plan_topics/V9a-capability-probe.md` Adds, reconcile the importer clause so it no longer claims V17a imports the constant: the literal text `which V18c's build-time literal-read assertion reads and V9g, V9i, and V17a import rather than redeclare` should drop `V17a` from the importer list (runtime importers `V9g` and `V9i`; build-time literal-read by `V18c`), and may note that V17a owns the constant's semantics without holding a code-level reference.

Edge case for the implementer: if V17a's code is found to genuinely read the literal (for example, a shared cancellation-runtime helper that V17a owns and that itself consumes the cap), then reconcile in the other direction instead — keep `V17a` in V9a's importer list and state the import explicitly in V17a's Adds. The leaf's current scope (`loomAbort` + late-settlement discard rules, with the cap-bounded `Promise.allSettled` owned by V9g per `patch-skew-degradation.md` §`session_shutdown` sub-step 3) indicates no such code reference exists.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one reads V9a's "`V17a` import rather than redeclare" and writes a code-level import of `SHUTDOWN_AWAIT_CAP_MS` into V17a's module (an unused import, since V17a performs no bounded await), while another reads V17a's "conceptual consumer" framing and writes no reference at all. The leaves describe two different V17a implementations, and the plan asserts both.
