## Problem

The `V9a` and `V17a` Adds sections contradict each other about whether `V17a` holds a code-level reference to `SHUTDOWN_AWAIT_CAP_MS`. `V9a` Adds names `V17a` among the runtime importers ("`V9g`, `V9i`, and `V17a` import rather than redeclare"), asserting `V17a` carries a code-level `import` of the constant. `V17a` Adds frames itself as owning the constant's cancellation-runtime semantics "as a conceptual consumer", implying a purely logical dependency with no code reference. An implementer of `V17a` cannot tell whether to write an `import { SHUTDOWN_AWAIT_CAP_MS }` against `V9a`'s pinned-constants block or to treat the constant as contract documentation `V17a` never references in code.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 622d0d3 — pi-loom plan: resolve "SHUTDOWN_AWAIT_CAP_MS physical home contradicts pinned-constants-block placement" (2026-06-11, Thomas Andersen); 2f4ecea — pi-loom plan: resolve "V9a Adds prescribes source-code layout instead of export/import obligations" (2026-06-17, Thomas Andersen)
**History:** Commit 622d0d3 rewrote V17a's Adds to call the leaf "a conceptual consumer" of the constant's semantics, moving the physical home to V9a's pinned-constants block and naming only V9g and V9i as the leaves that "consume that same block constant" — V17a was framed as a semantics owner, not a code importer. Six days later, commit 2f4ecea reworded V9a's Adds from the earlier "`V9g`, `V9i`, and `V17a` consume it without redeclaring it" to the stronger "`V9g`, `V9i`, and `V17a` import rather than redeclare", promoting V17a from conceptual consumer to code-level importer in V9a's prose without reconciling V17a's own "conceptual consumer" framing. The contradiction between the two leaves is the joint product of those two edits.

## Solution approach

Rewrite the phrase `as a conceptual consumer` in `V17a` Adds (`docs/plan_topics/V17a-cancellation-core.md`) to state that `V17a` owns the cancellation-runtime semantics of `SHUTDOWN_AWAIT_CAP_MS` but does not import or reference the constant in its own code. In `V9a` Adds (`docs/plan_topics/V9a-capability-probe.md`), drop `V17a` from the importer clause `V9g, V9i, and V17a import rather than redeclare` so only `V9g` and `V9i` remain named as runtime importers. If `V17a`'s code is found to genuinely read the literal, reconcile in the other direction instead — state the import explicitly in `V17a` Adds and keep `V17a` in `V9a`'s importer list.
