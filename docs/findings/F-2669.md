## Problem

`type-system.md` §Type compatibility defines the normative `⊑` relation as a closed-for-V1 list (TYPE-1…TYPE-8). `array<T>` gets covariance via TYPE-7, but `Result<T, E>` — the other generic constructor — has no rule, participating only in reflexivity (TYPE-1). Core sites nonetheless require a `Result` compatibility check: the `?` operator (expressions.md §`?` operator), typed `Result` bindings, and explicitly-annotated `Result`-returning functions (functions.md FN-3) all check `Result<…> ⊑ Result<…>` pairs that differ only by payload. Because `Result` is non-lowerable (rejected in schema-feeding positions via `loom/parse/result-in-schema-position`), the §Type-compatibility "Operational definition" AJV safety net cannot decide these pairs either, leaving them undecidable — conforming implementations diverge between admitting covariant `Result` and rejecting all but reflexive `Result`.

## Issue introduction

**Verdict:** introduced by a single commit
**Introducing commits:** `0226b80` ("pi-loom plan: resolve \"V15c references an undefined compatibility relation\"", 2026-05-05)
**History:** The entire §Type compatibility section and its closed `⊑` list (then rules 1–8) were authored whole in `0226b80`; the corpus is git-tracked and `git log -L` over the "Structural cases the parser must recognise / closed for V1" line confirms that commit as its origin. The list shipped closed with no `Result` rule from inception, even though `Result<T, E>` was already the type system's second generic constructor. The omission grew load-bearing as later commits elaborated consumers that cite `⊑` for `Result`-typed positions — FN-3 success-payload LUB reconciliation (`08886d9`, 2026-06-07) and the relocated `?`-operator enclosing-scope compatibility text (`87dfa1d`, 2026-06-28) — but neither introduced the gap; both rely on a rule the closed list never provided.

## Solution approach

Add a covariant `Result` rule to the closed `⊑` list in type-system.md §Type compatibility, mirroring TYPE-7: `Result<T₁, E₁> ⊑ Result<T₂, E₂>` iff `T₁ ⊑ T₂ ∧ E₁ ⊑ E₂`, covariant on both type parameters. Carve the rule out of the §Type-compatibility "Operational definition" AJV-necessary-condition clause, since `Result` is non-lowerable and never reaches AJV — the parser's decision is authoritative for it.

- Out of scope: editing the citing sites — expressions.md §`?` operator and functions.md FN-3 already cite `⊑` against `Result`; the `?`-operator / FN-3 return-type phrasing is owned by the decision-dependent sibling finding.
