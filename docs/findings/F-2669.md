## Finding

`type-system.md` §Type compatibility defines the single normative relation `T₁ ⊑ T₂` and enumerates the structural cases the parser must recognise as rules TYPE-1 … TYPE-8. The list is declared **closed for V1**: "anything outside it that the parser cannot decide statically is reported as a type mismatch." `array<T>` — one of the two generic constructors — gets covariance via TYPE-7, but `Result<T, E>` — the other generic constructor — has no rule at all. The only relation it participates in is reflexivity (TYPE-1).

Multiple core sites nonetheless require a `Result` compatibility relation. The `?` operator (expressions.md §`?` operator) states the enclosing scope's return type "must therefore be compatible with `Result<U, QueryError>` for some `U` under [Type System — Type compatibility]" — an explicit `⊑` check against a `Result` type. A typed binding `let x: Result<Animal, QueryError> = okCat` (where `okCat : Result<Cat, QueryError>`) is a legal program whose RHS check is `Result<Cat, QueryError> ⊑ Result<Animal, QueryError>`. An explicitly-annotated function `fn f(): Result<Animal, QueryError>` whose body yields `Result<Cat, QueryError>` must type-check the same pair (FN-3 / functions.md §Loom return type). Under the closed list, every such pair is decidable only by reflexivity, so a covariant `Result` (the natural analogue of TYPE-7's covariant array) is rejected.

The runtime AJV safety net cannot rescue these cases. `Result` is explicitly non-lowerable — schema-subset.md and runtime-value-model.md both state `Result` carries no emission rule and is rejected in any schema-feeding position (`loom/parse/result-in-schema-position`), and it never reaches AJV. The §Type-compatibility "Operational definition" — that `T₁ ⊑ T₂` implies the lowering of `T₁` AJV-validates against the lowering of `T₂` — has no meaning for `Result`, because `Result` has no lowering. So a `Result` pair the parser cannot decide statically has neither a structural rule nor a runtime fallback: it is simply undecidable. Two implementers diverge — one admitting covariant `Result`, one rejecting everything but reflexive `Result` — both consistent with the spec as written.

## Issue introduction

**Verdict:** introduced by a single commit
**Introducing commits:** `0226b80` ("pi-loom plan: resolve \"V15c references an undefined compatibility relation\"", 2026-05-05)
**History:** The entire §Type compatibility section and its closed `⊑` list (then rules 1–8) were authored whole in `0226b80`; the corpus is git-tracked and `git log -L` over the "Structural cases the parser must recognise / closed for V1" line confirms that commit as its origin. The list shipped closed with no `Result` rule from inception, even though `Result<T, E>` was already the type system's second generic constructor. The omission grew load-bearing as later commits elaborated consumers that cite `⊑` for `Result`-typed positions — FN-3 success-payload LUB reconciliation (`08886d9`, 2026-06-07) and the relocated `?`-operator enclosing-scope compatibility text (`87dfa1d`, 2026-06-28) — but neither introduced the gap; both rely on a rule the closed list never provided.

## Solution Space

### Recommendation

Add a normative covariant `Result` rule to the closed `⊑` list in `type-system.md` §Type compatibility, mirroring the existing TYPE-7 array-covariance rule. Insert one new row into the §Type-compatibility structural-cases table stating:

`Result<T₁, E₁> ⊑ Result<T₂, E₂>` iff `T₁ ⊑ T₂ ∧ E₁ ⊑ E₂`.

Write the rule covariant on both type parameters. Although loom 1.0 fixes the error side to `QueryError` (error-model.md — user-defined error types are out of scope), so the `E₁ ⊑ E₂` conjunct is reflexive in practice, state it generally to keep the rule forward-compatible.

Add a note next to this rule carving it out of the §Type-compatibility "Operational definition" AJV-necessary-condition clause: the `Result` rule is exempt from the requirement that the lowering of `T₁` AJV-validates against the lowering of `T₂`, because `Result` is non-lowerable (schema-subset.md and runtime-value-model.md reject it in any schema-feeding position via `loom/parse/result-in-schema-position`). For this rule the parser's decision is authoritative; there is no runtime AJV net. Omitting this carve-out makes the rule self-contradictory, since it asserts a `⊑` pair that can never AJV-validate because it never lowers.

The citing sites need no edits: `expressions.md` §`?` operator and `functions.md` FN-3 already cite the `⊑` relation against `Result` types, and this rule supplies the relation they depend on.

## Consequence

**Severity:** correctness

A typed `let`/parameter/return position holding a `Result<T, QueryError>` of a narrower payload against a wider declared payload (e.g. `Result<Cat, …>` where `Result<Animal, …>` is annotated) is undecidable: one implementer admits covariance, another rejects all but the reflexive pair, and the non-lowerable `Result` denies any runtime AJV tie-breaker. Routine `?`-bearing and explicitly-annotated `Result`-returning functions therefore parse differently across conforming implementations.
