## Finding

The bare word *capability* labels three distinct closed sets, with no glossary entry to disambiguate: (1) the **SDK capability inventory** — the seven obligations PIC-15 pins (`CAPABILITY_OBLIGATIONS.length === 7`); (2) the **Step-0 capability probe** — the five sub-steps `(a)`–`(e)`, one of which iterates the **factory-probable capability subset** (a different "five", `FACTORY_PROBABLE_CAPABILITIES`); (3) the **`details.capability` registration-surface set** — the closed five-member enum `{pi.registerMessageRenderer, pi.registerCommand, pi.registerFlag, pi.on, pi.getCommands}`.

`glossary.md` already carries paired disambiguation entries for `caller`, `callable set`, and `loom / Loom / pi-loom`. The capability overload is the same shape but has no entry, silently violating the glossary's own contract ("Add new entries here when the spec coins a new term reused on more than one page"). No normative breakage today (each callsite is locally unambiguous); the cost is orientation.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `b4fbfac` (2026-05-04, glossary created without a `capability` entry); `2d2fc8e` (2026-05-06, `details.capability` set introduced); `79ea4e8` / `bf85092` (2026-06-06, grew the set to five).
**History:** `git log --follow docs/spec_topics/glossary.md` shows the glossary was first created in `b4fbfac` with no `capability` entry and no later commit added one (the closest amendment `d54b426` added `caller`, `.warp` file, etc. but skipped `capability`). The three closed sets grew on independent timelines: the seven-item inventory and five-substep probe were pinned in `126ff8a` (2026-05-06); the `details.capability` enum was introduced at size three in `2d2fc8e`, grew to four (`79ea4e8`) and five (`bf85092`). At no single commit did the corpus "introduce" the overload — the missing glossary entry is the persistent gap the interaction exposes.

## Solution Space

**Shape:** single

### Recommendation

Add a single paired-disambiguation entry to `glossary.md` modelled on the existing `caller` / `callable set` entries, naming the three senses (SDK capability/inventory + factory-probable subset; capability probe/Step-0; `details.capability`/diagnostic-payload) and pointing each at its canonical owner. Place it alphabetically between `canonical schema hash` and `caller`. Watch: (i) distinguish the two distinct "fives" (Step-0 sub-steps vs factory-probable inventory items); (ii) keep the `details.capability` membership consistent with the `<capability>` placeholder table (the source of truth) so the glossary does not become a second drift-prone enumeration. Do not edit the three canonical pages.

## Consequence

**Severity:** advisory

Every concrete callsite is locally disambiguated, so implementers can produce a correct system; the cost is reader orientation — first-time readers cross-check which "five" or "seven" a paragraph means, and contributors lack a single glossary anchor to cite.
