## Finding

The same `{ field: T, ... }` construct in type position is referred to by several different names across the spec corpus. `type-system.md`'s opening bullet list calls it an **Inline anonymous object**; `grammar.md` introduces the production `ObjectType` under the bold lead **Inline object types** but, in the same paragraph, then refers to "an anonymous object type"; the `Type` grammar's inline comment writes "inline anonymous object type"; `TYPE-8` in `type-system.md` uses **inline object type** (the term then propagated by the `9a4e078` rewrite); `influences.md` and `diagnostics/placeholder-rendering-a.md` say **inline anonymous object types**; `schema-subset.md` (Lowering step 2) and `diagnostics/code-registry-load.md` (`loom/load/schema-slug-collision`) talk about **anonymous inline object schemas**; and `runtime-value-model.md`'s table cell groups the type into "Object schema (named or anonymous)".

The grammar production is named `ObjectType` and the compatibility rule that owns its `⊑` semantics (`TYPE-8`) already settled on **inline object type**. The intro bullet and the surrounding references have not been updated to match, so an implementer reading top-down sees four different labels for one construct before reaching the rule that names it. The `schema-subset.md` / load-diagnostics usages are framing the lowered-schema side of the same construct (the `__inline_<slug>` `$defs` entry), so they read naturally as **inline object schema** — but they too should agree on adjective order (`inline` first, then `anonymous` only if the framing requires distinguishing it from a named one).

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**

- `fecb504` — `Split spec.md and plan.md into per-topic / per-phase files` (carried the bullet verbatim into `spec_topics/type-system.md`)
- `232a020` — `pi-loom spec: resolve "Type grammar omits inline anonymous object types"` (added `grammar.md`'s `ObjectType` production with the comment "inline anonymous object type" and the bold lead "**Inline object types.**" — body text "anonymous object type")
- `9a4e078` — `pi-loom spec: resolve "TYPE-8 field-wise compatibility scope ambiguous"` (rewrote `TYPE-8` to use the term "inline object type" and to anchor at `grammar.md#type-grammar`)

**History:** The intro bullet ("Inline anonymous objects") predates the formal grammar of inline object types by about a month. The two subsequent fixes (`232a020`, `9a4e078`) introduced the more careful terminology in their respective owning sections (`grammar.md` and `TYPE-8`) but did not sweep back to the intro bullet, `influences.md`, the lowering paragraph, the load diagnostic, the placeholder-rendering rule, or the runtime-value-model table. The four phrasings observed today are the deposited residue of those three uncoordinated commits, not a single defect.

## Solution Space

**Shape:** single

### Recommendation

Standardise on **inline object type** (singular) for the type-position construct, matching `TYPE-8` and `ObjectType`. Apply the following edits:

- `type-system.md` intro bullet: replace **Inline anonymous objects** with **Inline object types** (plural in the list-heading position to match the surrounding bullet style — `Primitive types`, `Named types`, `Generic types`, `Union types`, `Literal types`). Keep the rest of the bullet text.
- `grammar.md` — the `ObjectType` grammar comment: change `// inline anonymous object type; Field per Schema Declarations` to `// inline object type; Field per Schema Declarations`. In the **Inline object types** paragraph body, change "admits an anonymous object type" to "admits an inline object type"; the bold lead already matches.
- `influences.md` — TypeScript-surface bullet: change "inline anonymous object types" to "inline object types".
- `diagnostics/placeholder-rendering-a.md` — change "Inline anonymous object types" to "Inline object types".

For the lowered-schema framing (where the noun is **schema**, not **type**), standardise on **inline object schema** with adjective order `inline` first:

- `schema-subset.md` Lowering step 2: change "Hoists anonymous inline object schemas" to "Hoists inline object schemas". The contrast against named schemas is already carried by the surrounding step-1 / step-2 framing; the word "anonymous" is redundant.
- `diagnostics/code-registry-load.md` `loom/load/schema-slug-collision` trigger column: change "Two anonymous inline object schemas" to "Two inline object schemas".

For `runtime-value-model.md`'s top table row "Object schema (named or anonymous)", leave the parenthetical as-is or recast as "Object schema (named or inline)" — this cell deliberately groups both forms under one row and the parenthetical is a join, not a coined term. Treating it as out-of-scope for the sweep is acceptable.

Edge cases the implementer must watch:

- Do not rename the diagnostic codes (`loom/load/schema-slug-collision`, `loom/parse/empty-schema-body`, `__inline_<slug>`) — only the prose around them.
- Do not edit anchor IDs (`#type-grammar`, `#type-8`, `#canonical-schema-hash`, `#lowering-algorithm`). Inbound cross-references rely on them.
- The `binder/defaulting-system-note-echo.md` echo-policy passage talks about "first-field-value" rendering rules; it does not coin a term for the construct itself and should not be touched.

## Consequence

**Severity:** cosmetic

A first-time reader of `type-system.md`'s intro encounters one label, then sees three more by the time they reach `TYPE-8`, `grammar.md`'s production, and the lowering and diagnostics text. No behaviour diverges and no diagnostic name is affected, but the inconsistency forces the reader to verify that all four phrasings refer to the same construct.
