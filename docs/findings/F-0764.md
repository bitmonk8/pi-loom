## Finding

GOV-1, GOV-3, GOV-6 and GOV-24 all refer to a single "REQ-ID prefix table below" anchored at `req-id-prefix-table-active-a.md#req-id-prefix-table`. Physically that table is cut in half: `active-a.md` ends mid-table after `PIC`, and `active-b.md` opens at column zero with a bare GFM table data row (`| \`implementation-notes.md\` | \`IMPL\` |`) — no `| Page | Prefix |` header row, no continuation note, no GOV-3 / GOV-24 back-reference, and no statement on either page that the live table is the union of the two fragments.

The omission has two concrete consequences. (1) A consumer building the `<live-prefix-alternation>` regex from `active-a.md` alone — which is exactly what GOV-3 tells it to do, naming only that page's heading — produces an alternation that misses `IMPL`, `CEIL`, `CIO`, `GRAM`, `GOV` and every `(no IDs — narrative)` row on `active-b.md`; the GOV-3 *Unknown-prefix closure invariant* and GOV-6 *table-completeness invariant* are then trivially violated against any commit that emits, for example, a `GOV-` REQ-ID. (2) The fragment that lives on `active-b.md` is, in CommonMark / GFM terms, not a table at all — a GFM table requires a delimiter row immediately under the header row, and `active-b.md`'s leading data-row line stands alone, so most CommonMark renderers will render the page's twelve "rows" as a single paragraph of pipe-delimited text rather than as a continuation of the table on `active-a.md`. Both the byte-level extractor contract and the rendered contract are broken at the seam.

The split is mechanical — it was performed by the `docs/_tools/split_spec.py` line-range cut introduced by commit `f5e89f4` ("docs: enforce 100KB hard / 30KB recommended size cap") with PLANS entries `req-id-prefix-table-active-a (32–100)` / `req-id-prefix-table-active-b (101–162)` — and the splitter did not insert a header row, continuation note, or anchor at the seam.

## Issue introduction

**Verdict:** introducing-commit-identified
**Introducing commit:** `f5e89f4` — *docs: enforce 100KB hard / 30KB recommended size cap on spec set* (Thu Jun 4 10:38:09 2026)
**History:** Both fragment files are first-added at `f5e89f4` (`git log --diff-filter=A --follow` on either file returns only that commit). The split was performed by `docs/_tools/split_spec.py`'s PLANS table, which cuts `spec_topics/governance.md` at fixed line ranges — `req-id-prefix-table-active-a (32–100)` and `req-id-prefix-table-active-b (101–162)` — and writes each slice verbatim under its slug, prepending only a generated `# Req id prefix table active b` H1. The splitter does not detect that line 101 sits inside a GFM table, so it neither carries the `| Page | Prefix |` header / delimiter rows down to the new file nor inserts a continuation marker or back-reference at the seam. One subsequent commit (`1d773d6`, "GOV-3/GOV-6 subdirectory scan scope and prefix-table hub-stub bindings") edits two rows in `active-b.md` but does not touch the seam. The defect is therefore single-commit-introduced at `f5e89f4` and present since these two files first existed; before `f5e89f4` the table was a single unbroken GFM block inside the monolithic `governance.md`.

## Solution Space

**Shape:** single

### Recommendation

Treat the table as one logical structure split across two physical files, and make every part of the corpus that touches it self-describing.

1. **On `active-a.md`**, edit GOV-3's introductory sentence ("Pages whose row in the prefix table below…") to read explicitly: *"…the prefix table below — the live REQ-ID prefix table is the union of the rows on this page and on [Req id prefix table active b](./req-id-prefix-table-active-b.md), and the `<live-prefix-alternation>` in GOV-3's primary extractor is built from that union."* Apply the same union framing to GOV-1's "(table below)" parenthetical and to GOV-24's "absent from both this table and the *Retired prefixes* sub-table" clause (which today implicitly assumes single-file scope).

2. **On `active-b.md`**, restore the table's structural integrity at the seam by prepending, immediately after the H1, the two missing GFM table lines:

    ```
    | Page | Prefix |
    |---|---|
    ```

    followed by a one-line non-normative continuation note: *"(Continuation of the [REQ-ID prefix table](./req-id-prefix-table-active-a.md#req-id-prefix-table) on `req-id-prefix-table-active-a.md`. The live table is the union of the rows on both pages.)"* This makes the file render as a valid table standalone and tells the reader where the header lives.

3. **Audit `docs/_tools/split_spec.py`** (or add a follow-up note in `docs/_tools/`) so a future split that cuts inside a GFM table is either rejected or has the header carried down automatically — the current PLANS table is the proximate cause and is one line away from doing this again on the next size-cap pass. (This is a tooling fix, not a spec edit; flag it but do not block on it.)

Edge cases the implementer must watch:

- The GOV-3 *Unknown-prefix closure invariant* and the GOV-6 *table-completeness invariant* both read "live REQ-ID prefix table" — after the edit, audit those phrasings to make sure they unambiguously denote the union, not the file `active-a.md` alone.
- GOV-24's *Add prefix to existing page* / *previously-unused* checks must scan both files; the new union framing in step 1 covers this textually, but a reviewer should confirm no remaining clause says "this table" with single-file scope.
- Do not introduce a new `<a id>` on `active-b.md` for "the prefix table" — keep the single anchor `#req-id-prefix-table` on `active-a.md` so the existing 4+ cross-references in `docs/plan_topics/` and `docs/spec_topics/` remain valid.

## Consequence

**Severity:** correctness

A third-party REQ-ID extractor or table-completeness auditor that follows GOV-3 literally — "built from the live REQ-ID prefix table below" on `active-a.md` — produces an alternation that omits five live prefixes and seven narrative-page rows, then falsely reports GOV-3 / GOV-6 / GOV-22 violations against the corpus on the next commit that mentions e.g. a `GOV-` or `CEIL-` ID. In rendered form the seam currently shows as either a malformed table or a stray paragraph of pipe-delimited text on `active-b.md`, making the live prefix set difficult to read without source-diving.
