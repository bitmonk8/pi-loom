## Problem

The V1 diagnostic registry is sharded across four sibling pages, but the normative column legend (*Sev* / *Phase* / *Trigger* / *Spec rule* / *Hint* / *Message*), the closed *Phase* taxonomy, and the *E/W* severity carve-out live only on `code-registry-parse.md`, whose intro nonetheless claims "the column legend below applies to all four tables." A reader entering at a continuation page — the common deep-link path — sees columns whose semantics are defined on another page with no in-page pointer, and `diagnostic-shape.md`'s DIAG-4 keys on "the *Message* column" although no table lives on DIAG-4's home page. The `code-registry-host.md` header has drifted from the legend: *Sev* is spelled *Severity*, *Spec rule* is renamed *Anchor*, and the *Hint* column is dropped — two of those contradict the legend directly, so a deep-link reader hits column names that do not match the schema the legend governs.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `9559e19` (2026-05-07) — added the initial `loom/host/*` subsection with the divergent header `Code | Severity | Phase | Trigger | Anchor | Message`, alongside siblings that did not yet share a normative legend (divergence was then a stylistic choice, not a contradiction).
- `f5e89f4` (2026-06-04) — split the monolithic `diagnostics.md` into `diagnostic-shape.md` plus four `code-registry-*.md` pages; left the legend (and *Phase* taxonomy) on the `loom/parse/*` table only and the three new pages with bare headings and no intro. DIAG-4 carried to `diagnostic-shape.md` and lost the *Message* column to a sibling file.
- `3059a0e` (2026-06-05) — added "The column legend below applies to all four tables" without relocating the legend, retroactively making the host and runtime headers non-conformant.

**History:** `git log -S '*Phase* identifies which pipeline'` traces the taxonomy through `839db71` (single-file era, no inconsistency) → `f5e89f4` (split, latent) → `3059a0e` (made explicit). `git blame` confirms `9559e19` as the host header's origin, textually untouched since. None of the commits is individually defective — the durable fix is to relocate the legend and align the headers.

## Solution approach

Move the column legend, the closed *Phase* taxonomy, and the *E/W* severity carve-out from `code-registry-parse.md` into `diagnostic-shape.md`'s `## Code registry rules (normative)` section, using the canonical labels *Sev*, *Spec rule*, and *Message*. Replace the legend on `code-registry-parse.md` with a back-reference to the relocated legend and prepend the same back-reference on `code-registry-load.md`, `code-registry-runtime.md`, and `code-registry-host.md`. Re-anchor DIAG-4's "the *Message* column" reference at the relocated legend. Align the `code-registry-host.md` header and its alignment row to the legend's labels (`Sev`, `Spec rule`, and the restored *Hint* column).

## Solution constraints

- Preserve the *E/W* severity carve-out paragraph's existing cross-references (the Discovery failure-modes table and `loom/load/callee-has-errors`) when relocating it.

## Relationships

- `code-registry-runtime.md` "Message template" vs sibling "Message" — co-resolve (the runtime rename should land in the same sweep as the host-header alignment so the legend becomes true at once).
- `diagnostic-shape.md` `loom/typecheck/*` namespace governance misplaced on a runtime page — same-cluster (both move governance prose out of enumeration-only pages; resolve independently).
- `diagnostics.md` opening framing of what the registry rules apply to — same-cluster (resolve independently).
