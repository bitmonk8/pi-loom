## Finding

The V1 diagnostic registry is sharded across four sibling pages (`code-registry-parse.md`, `code-registry-load.md`, `code-registry-runtime.md`, `code-registry-host.md`). The intro paragraph on `code-registry-parse.md` defines the normative column legend — *Sev*, *Phase* (with its closed five-arm taxonomy `lex` / `parse` / `type` / `load` / `runtime`), *Trigger*, *Spec rule*, *Hint*, *Message* — and explicitly asserts that "the column legend below applies to all four tables." The legend itself, however, never leaves this page. A reader who lands on `code-registry-load.md`, `code-registry-runtime.md`, or `code-registry-host.md` (the common entry path, since cross-references from other shards link directly to specific rows) sees columns whose semantics are defined elsewhere, with no in-page pointer.

The breakage is not merely cosmetic. `diagnostic-shape.md` §**DIAG-4** is a normative rule keyed to "the *Message* column" — but DIAG-4's home page hosts none of the registry tables, and the column it names is defined on yet a third page. The closed *Phase* taxonomy is similarly orphaned: any reader auditing whether a sibling table's *Phase* values are in-domain has to navigate to `code-registry-parse.md` to discover the domain. The three sibling registry pages also accreted divergent header spellings (host uses "Anchor" / "Severity", runtime uses "Message template") under the same governance gap — those divergences are tracked as their own findings, but they exist because the legend is not collocated with the rules that govern it.

The legend, the *Phase* taxonomy, and the DIAG-4 *Message*-column reference are all governance for the registry as a whole and belong with the four DIAG rules in `diagnostic-shape.md`, not embedded in one arbitrary continuation table.

## Solution Space

**Shape:** single

### Recommendation

Move the column legend and the *Phase* taxonomy from `code-registry-parse.md` into `diagnostic-shape.md`, placing them under the existing `## Code registry rules (normative)` section adjacent to DIAG-1…DIAG-4 so the column DIAG-4 cites is defined on its own page. Specifically:

1. In `diagnostic-shape.md`, after the DIAG-1…DIAG-4 block (and before the naming-convention sentence), insert a `**Registry table columns.**` paragraph carrying the legend prose: the *Sev* / *Phase* / *Trigger* / *Spec rule* / *Hint* / *Message* definitions and the closed *Phase* taxonomy `lex` / `parse` / `type` / `load` / `runtime`, plus the *E/W* severity carve-out sentence currently on `code-registry-parse.md`. Use the canonical header spellings ("Sev", "Spec rule", "Message" — singular, no "template") so the legend is the unambiguous source the sibling tables must conform to.
2. In `code-registry-parse.md`, replace the legend paragraph with a one-line back-reference: e.g. *"Column semantics and the closed `Phase` taxonomy are defined in [Diagnostic shape — Code registry rules](./diagnostic-shape.md#code-registry-rules-normative)."* Keep the introductory "split across four pages" sentence on the parse page so cross-shard links into the registry still land on a self-explanatory intro.
3. In each of `code-registry-load.md`, `code-registry-runtime.md`, `code-registry-host.md`, prepend the same one-line back-reference immediately before the table heading. These three pages currently have no intro paragraph at all; a single cross-reference line is sufficient.
4. Update DIAG-4's "the *Message* column" wording to anchor the reference at the now-collocated legend (e.g. "the *Message* column defined above"), so the rule and the column it normates live on the same page.

Edge case: the *E/W* per-source carve-out in the legend names a specific Discovery table and a specific load-namespace code (`loom/load/callee-has-errors`); preserve those cross-references verbatim when moving the paragraph — they point into other shards and must continue to resolve.

## Consequence

**Severity:** advisory

Readers entering at any of the three continuation pages encounter undefined columns. DIAG-4's normative "the *Message* column" reference dangles off its own page. The governance gap creates the soil in which sibling-table header drift (host's "Anchor"/"Severity", runtime's "Message template") could accrue without contradicting any single-page rule.
