## Finding

In `docs/spec_topics/expressions.md` the heading `## Object construction, array construction, and operator rules` (line 148) carries no body: the immediately following line is another H2 sibling, `## Object construction` (line 150). The four topics the umbrella heading names — object construction, array construction, other arithmetic, ordering comparisons — are all emitted as H2 siblings of the umbrella, not as children nested beneath it. The `+`-operator rule has no heading of its own; it is folded into the `## Array construction` body. The result is a heading that promises a grouped section but introduces zero content, and a flat structure that does not express the parent/child relationship its title asserts.

The umbrella heading's anchor is load-bearing. Four inbound citations resolve to `#object-construction-array-construction-and-operator-rules`: an intra-page navigation bullet (`expressions.md:21`), the `concat` member's reference to the array-literal LUB rule (`expressions.md:110`), and two normative type-system rules — `TYPE-7` (`type-system.md:41`) and `TYPE-9` ternary common-type machinery (`type-system.md:50`). Any fix must keep that anchor resolvable; deleting the heading outright would break all four.

## Issue introduction
**Verdict:** single introducing commit
**Introducing commits:** `c1e5d64` (2026-05-05) "spec: REQ-ID infrastructure + Phase 12a heading promotion on five named pages (Decision #12, partial)"
**History:** The umbrella heading was created when `spec.md` was split into per-topic files (`fecb504`), at which point it had body content directly beneath it — a `**Object construction.**` bold inline-label paragraph. The empty-section defect entered at `c1e5d64`, whose "Phase 12a heading promotion" converted the bold inline labels (`**Object construction.**`, `**Array construction.**`, `**Other arithmetic.**`, …) into standalone `## Object construction`, `## Array construction`, `## Other arithmetic` H2 headings. Because the promotion lifted the first child's body out from under the umbrella heading without re-parenting it (e.g. into an H1, or by nesting the new headings one level deeper), the umbrella heading was left with zero body and its intended children became siblings. `git log -L148,150:docs/spec_topics/expressions.md` shows the `-**Object construction.** …` / `+## Object construction` hunk at `c1e5d64` as the transition point.

## Solution Space

**Shape:** single

### Recommendation

Keep `## Object construction, array construction, and operator rules` as the H2 group heading (it owns the `#object-construction-array-construction-and-operator-rules` anchor that four inbound sites cite) and demote its four intended children — `Object construction`, `Array construction`, `Other arithmetic`, `Ordering comparisons` — from H2 to H3 so they nest beneath it. This expresses the grouping the title asserts while leaving every existing anchor slug unchanged, so the four inbound citations continue to resolve with no repointing.

Edge cases for the implementer:
- Do **not** delete the umbrella heading and do not alter its text — its anchor slug is derived from the exact heading string and is referenced normatively from `type-system.md` (TYPE-7, TYPE-9). Changing or removing it forces edits at all four inbound sites.
- The `+`-operator rule currently lives inside the Array-construction body. Either leave it there or give it its own H3 (`### `+` operator`) under the umbrella; either is acceptable, but if a new heading is added, confirm no other page links to a `+`-operator anchor (none does today).
- The page already has a single H1 (`# Expression Sublanguage`); do not introduce a second H1 — demoting the children to H3 keeps one H1 and a clean H2→H3 hierarchy.

## Consequence

**Severity:** cosmetic

A reader reaching the umbrella heading finds no content and must infer that the next four H2s are its intended members. No implementer behaviour diverges and the cited anchors currently resolve, so the defect is pure document organisation.
