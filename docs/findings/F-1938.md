## Finding

After the prior peel-offs of `V15d` (prompt→prompt parent-suspend), `V15f` (invoke parse/load diagnostics), `V15g` (provenance seam), `V15h` (invoke-child swallowing-handler), and `V15j` (ceiling-#4 depth carrier), the residual `V15a` still ships several mechanisms that fail and ship independently of one another:

1. `realpath` + segment-boundary discovery-root containment on two channels (load-time diagnostic and invocation-time `InvokeInfraError{load_failure}`) — `INV-1`.
2. The static-resolution per-pass parse cache and its transitive parse/lower walk (each visited file parsed and lowered exactly once) — the `implementation-notes.md` / `invocation.md` static-resolution obligation and `cka-44`.
3. The fresh-vs-attach cross-mode matrix selection (child uses its own model/tools/system).
4. Callee final-value propagation to the `invoke` caller (present on success, absent on fail/cancel) — the `return.md` final-value contract.
5. The `cka-47` pre-`invoke` cancellation checkpoint facet, plus the `ERR-13` completed-callee-finality live-carrier witness.

`V15a` Ships-when enumerates five separate assertions across two abstraction layers: a static path/containment + cache-walk layer (items 1–2) and a runtime cross-mode-execution layer (items 3–5, which drive behaviour on top of the item-1/2 machinery). These are independently exercisable — a containment escape, a cache double-parse, a wrong fresh/attach selection, a dropped final value, and a missed checkpoint each red out on their own assertion against largely disjoint spec anchors and test fixtures. The plan already treats this granularity as the unit of work — `V14b` was carved out of `V14a` for the same reason, and `V15d`/`V15f`/`V15g`/`V15h`/`V15j` were peeled off this very leaf in earlier passes. The residual leaf is the last over-bundled increment in the cluster.

## Solution Space

**Shape:** single

### Recommendation

Split `V15a`/`V15a-T` into two paired leaves at the static-resolution-vs-runtime seam, keeping the `V15a`/`V15a-T` ID for the foundational half and allocating one new paired leaf (`<new>`/`<new>-T`) for the runtime half:

- **Foundational half (retains `V15a`/`V15a-T`):** `INV-1` load-time + invocation-time `realpath` + segment-boundary containment on both channels (diagnostic + `InvokeInfraError{load_failure}`), the static-resolution per-pass parse cache and its transitive parse/lower walk (`cka-44`), and the post-`realpath` parent-loom-path capture that `V15g` consumes. `INV-2`/`INV-3` (AST arg-list `style` and the open invoke-options struct) belong with this static half.
- **Runtime half (new `<new>`/`<new>-T`):** the fresh-vs-attach cross-mode matrix selection, callee final-value propagation to the caller (present on success, absent on fail/cancel) against the `return.md` final-value contract, the `ERR-13` completed-callee-finality live-carrier witness, and the `cka-47` pre-`invoke` checkpoint facet. This half lists `V15a` in its Deps (it drives behaviour on the foundational half's machinery).

Then propagate the seam:

- In `docs/plan_topics/coverage-matrix.md`, retarget each affected closing-leaf cell to the half that closes it: `INV-1/INV-2/INV-3` and `cka-44` to the foundational `V15a`; `ERR-13` (the `V15a` co-witness entry) and `cka-47` (the invoke-checkpoint-site entry) to the new runtime leaf's real ID once allocated.
- In `docs/plan_topics/H5b-warn-only-canary.md`, add the new leaf to the Deps range so the canary and the `H6a` release gate stay sequenced after it (required because the runtime half closes coverage-matrix-mapped obligations), per the *Transitive-completeness plan-maintenance* clause in `conventions.md` §REQ-ID discipline.
- Retarget the Deps of consumers that exercise the runtime half (`ERR-13` co-witness, cross-mode/final-value, or `cka-47` behaviour) to the new leaf; leave consumers of the static half (`V15e` parse-cache eviction, `V15g` parent-loom-path provenance) on `V15a`.

The new leaf-ID is the implementer's to allocate from the V15 slice's existing scheme; do not pre-assign a final ID here. Keep the `spec.md` topics read-only — every REQ-ID and code-keyed token involved (`INV-1`–`INV-3`, `ERR-13`, `cka-44`, `cka-47`) already exists; the fix is internal to the plan files.

## Consequence

**Severity:** advisory

`V15a` remains implementable — an implementer can still produce a working leaf — so this does not block pickup or risk spec divergence. The cost is granularity: a single tests-task / implementation-task pair carries five independently-failing assertions across a static-resolution layer and a runtime cross-mode-execution layer, enlarging the TDD increment and the review surface beyond the "smallest feature that can ship and be tested independently" the plan's leaf-sizing convention targets, and matching the bundling the plan already split out of sibling leaves.
