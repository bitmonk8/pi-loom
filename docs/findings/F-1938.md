## Problem

The `V15a`/`V15a-T` leaf pair still ships five independently-failing mechanisms in one increment: `realpath` + segment-boundary containment on both channels (`INV-1`), the static-resolution per-pass parse cache and its transitive parse/lower walk (`cka-44`), the fresh-vs-attach cross-mode matrix selection, callee final-value propagation to the caller, and the `cka-47` pre-`invoke` checkpoint facet plus the `ERR-13` completed-callee-finality witness. These span two abstraction layers — a static path/containment + cache-walk layer and a runtime cross-mode-execution layer — and each reds out on its own assertion against largely disjoint spec anchors and fixtures. The bundle enlarges the TDD increment and review surface beyond the plan's leaf-sizing convention, matching bundling already peeled into sibling leaves (`V15d`/`V15f`/`V15g`/`V15h`/`V15j`).

Two subsequent passes reduced but did not eliminate the bundling: `cb8737a` (2026-06-12, "Systemic leaf over-bundling across the plan corpus") peeled `V15d` off, and `5f4d271` (2026-06-17, "V15a bundles ~14 independent test obligations into one impl+T pair") peeled `V15f`/`V15g`/`V15h` off (those leaves gained their content in that commit). The residual five-mechanism bundle the finding describes has been present continuously since `c6a664e` and survives the partial peels. ## Solution approach

Split `V15a`/`V15a-T` at the static-resolution-vs-runtime seam into two paired leaves: keep `V15a`/`V15a-T` for the foundational static half (`INV-1` containment, the `cka-44` parse cache and transitive walk, `INV-2`/`INV-3`, and the post-`realpath` parent-loom-path capture that `V15g` consumes) and allocate one new paired leaf for the runtime half (fresh-vs-attach matrix, callee final-value propagation per `return.md`, the `ERR-13` witness, and the `cka-47` checkpoint facet), listing `V15a` in the runtime leaf's `Deps`. Retarget each affected closing-leaf cell in `docs/plan_topics/coverage-matrix.md` to the half that closes it, add the new leaf to the `Deps` range in `docs/plan_topics/H5b-warn-only-canary.md`, and retarget the `Deps` of consumers that exercise the runtime half to the new leaf while leaving static-half consumers on `V15a`.

## Solution constraints

- The `spec.md` topics are read-only for this fix — every REQ-ID and code-keyed token involved (`INV-1`–`INV-3`, `ERR-13`, `cka-44`, `cka-47`) already exists; the change is internal to the plan files.
- Allocate the new paired leaf ID from the V15 slice's existing `V<N><letter>` scheme (next free letter; never reuse a retired ID), as a `-T` tests task plus a bare implementation task per `docs/plan_topics/conventions.md`.
