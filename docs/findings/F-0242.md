## Finding

`V3e` and its paired `V3e-T` tests leaf both name the string-replace stdlib member as `String.replace`. The spec's expression sublanguage exposes the member on the lowercase composite type `string` and names it `replace(from, to)` in both the member table and the normative reference-vector heading (`docs/spec_topics/expressions.md` §Built-in methods and properties). The PascalCase `String` form is not a loom identifier — loom's primitive type is lowercase `string` with no `String` namespace — so `String.replace` denotes nothing in the language.

The mismatch appears in three places: the `V3e` **Adds** sentence ("the `String.replace` reference vectors and `concat`") and the first **Tests** bullet of both `V3e` and `V3e-T`, whose link label reads `expressions.md — String.replace reference vectors` against a spec section that titles the vectors `replace(from, to)` reference vectors. The section anchor (`#built-in-methods-and-properties`) resolves correctly, so the vectors remain reachable; the defect is confined to the label/identifier form.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (`pi-loom plan: build/update plan for spec.md + review`, 2026-06-10) — first introduced the `String.replace` label in `docs/plan_topics/V3a-T-expression-evaluator.md` / `V3a-expression-evaluator.md`. `cb8737a` (`pi-loom plan: resolve "Systemic leaf over-bundling across the plan corpus"`, 2026-06-12) relocated the stdlib surface into the new `V3e` / `V3e-T` leaves, carrying the label verbatim.
**History:** `git log -S 'String.replace' --all` shows the token first net-appearing at `c6a664e`, the earliest plan-corpus commit (prior commits are spec-only). The defect was never introduced by a later edit; the over-bundling split (`cb8737a`) and the subsequent stdlib-bullet reworks (`20b4689`, `026697d`) all preserved the original `String.replace` phrasing rather than correcting it. The mislabel has therefore been present since the plan's inception.

## Solution Space

**Shape:** single

### Recommendation

Replace the `String.replace` identifier with the spec's `replace(from, to)` phrasing in both leaves; the `#built-in-methods-and-properties` anchor already resolves and stays unchanged.

- `docs/plan_topics/V3e-expression-stdlib.md` **Adds**: change `the \`String.replace\` reference vectors and \`concat\`` to `the \`replace(from, to)\` reference vectors and \`concat\``.
- `docs/plan_topics/V3e-expression-stdlib.md` **Tests** (first bullet) and `docs/plan_topics/V3e-T-expression-stdlib.md` **Tests** (first bullet): change the link label `[expressions.md — \`String.replace\` reference vectors]` to `[expressions.md — \`replace(from, to)\` reference vectors]`, leaving the `../spec_topics/expressions.md#built-in-methods-and-properties` target intact.

## Consequence

**Severity:** cosmetic

The vectors stay reachable because the section anchor resolves, so no implementer is blocked. The residual risk is that a leaf author treats `String` as a distinct namespace or propagates the non-loom identifier form into test naming, diverging from the spec's `string` member surface.
