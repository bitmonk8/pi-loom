## Problem

PIC-50 (anchor `pic-50` in `conversation-drive.md`) maps a synchronous throw from `pi.sendUserMessage` to `Err(QueryError { kind: "transport", message: <error.message>, ... })`, sourcing `message` from a bare `.message` read on the caught thrown value. JavaScript permits any value to be thrown, so for a non-Error throw `<error.message>` evaluates to `undefined`, and for a `null`/`undefined` throw the property access raises a `TypeError`. PIC-50 names no coercion or fallback, so two conforming implementations diverge on the non-Error case and a conformance author has no expected value to assert. The gap is load-bearing because `transport` is in the always-log set (group A): the synthesised `undefined` propagates into `RuntimeEvent.message` (typed non-optional `string`) and into the `(kind, query_site, message, occurred_at)` dedup key, corrupting the exactly-once emission contract.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `bf85092` (2026-06-06, "resolve non-Error-throw coercion + session_start pi.getCommands() failure contract") in interaction with the pre-existing synchronous-throw transport mapping (present at least since `31ff060`, 2026-05-08).
**History:** The synchronous-throw transport mapping has carried the raw `message: <error.message>` read since before the spec was relocated into `docs/` (`31ff060`, 2026-05-08) ā€” predating the coercion convention by roughly a month. Commit `bf85092` (2026-06-06) introduced the `#underlying-error-coercion` rule in `placeholder-rendering-b.md` Ā§6 and the deterministic-payload convention, and subsequent edits propagated explicit citations of it to the sibling diagnostic sites (extension-bootstrap, host-incompatible/`details.cause`, capability-probe self-failure, code-registry-runtime, `CodeToolError`). That convention-introduction did not retrofit the pre-existing prompt-mode transport synchronous-throw read. Later commits that touched the mapping ā€” `dd09e20` (2026-06-25, RFC-2119 force on PIC-50/51) and `6f38d17` (2026-06-25, provider derivation) ā€” formalised it as the named PIC-50 rule while preserving the raw `.message` read. The defect is therefore the interaction between the long-standing raw read and the later corpus-wide coercion convention that left this one site behind, not a single introducing edit.

## Solution approach

Rewrite PIC-50's `message` derivation to route the caught throw through the underlying-error coercion at `placeholder-rendering-b.md#underlying-error-coercion` instead of the raw `.message` read, so a non-Error throw yields a deterministic non-null string. Add a non-Error-throw boundary acceptance criterion to PIC-50 covering the synchronous non-Error (and `null`/`undefined`) throw case and its resulting group-A `RuntimeEvent.message`. PIC-51's existing absent-`errorMessage` fallback (`pic-51`) is the symmetric prompt-mode precedent.
