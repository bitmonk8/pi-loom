## Finding

`cka-27` is the canonical code-keyed-area token that `coverage-matrix.md` assigns to the `session-only-degraded-state.md` obligation set (the Recovery-path prohibition, the state-independent drain-state tag write, the *Predicate split* rule, the *Seam-minimality* prohibition, and the *Inline triplet is normative* clause). Its sole closing leaf is `V9l`.

All four Tests bullets in `V9l` (mirrored in `V9l-T`) reference that obligation only by spec-section paraphrase — `session-only-degraded-state.md §degraded branch`, `§Recovery-path prohibition`, `§Predicate split`, `§Seam-minimality + *Inline triplet is normative*` — and never name `cka-27`. This diverges from the plan-wide convention for code-keyed-area leaves, where each Tests bullet cites its token inline: `V9g` cites `(code-keyed-area token cka-31)`, `V14b` cites `cka-13`, `V15i` cites `cka-48`. Because the token is absent from the leaf prose, the implementer has no instruction to embed `cka-27` in the test source, a reviewer cannot `grep cka-27` to confirm `V9l` coverage, and any cka-token index misses `V9l` entirely.

The defect is confined to traceability and convention-consistency: it does not redden a closing gate. `cka-27` is a single-leaf row, so the `H5f` per-facet citing-test arm (which fires only on multi-leaf rows) does not apply, and the `H5e` un-anchored-MUST arm is satisfied because the obligation is present in the coverage matrix with a named closing leaf.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 55ed974 — pi-loom plan: resolve "degraded-state obligations green over open spec contradiction" (2026-06-12, Thomas Andersen); 7259777 — pi-loom plan: resolve "code-keyed obligation areas lack a canonical citable token" (2026-06-18, Thomas Andersen)
**History:** `V9l` and `V9l-T` were created on 2026-06-12 (55ed974) citing the obligation only by spec-section paraphrase, before any `cka-<n>` token existed. The canonical code-keyed-area token scheme was added on 2026-06-18 (7259777), which introduced the `cka-27` row in `coverage-matrix.md` and migrated the sibling `V9g` and `V14b` Tests bullets to cite their tokens inline, but did not touch `V9l`/`V9l-T` — leaving them as the residue of an incomplete migration. A pickaxe search (`git log -S 'cka-27'`) over both leaf files confirms the token was never present in either.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V9l-session-only-degraded-branch.md` and its `-T` partner `docs/plan_topics/V9l-T-session-only-degraded-branch.md`, make each of the four Tests bullets cite `cka-27` inline alongside its existing `session-only-degraded-state.md §…` reference, so the token is present in both the leaf prose and the test source the implementer derives from it. Match the form already used by sibling code-keyed-area leaves — append `(code-keyed-area token `cka-27`)` to each bullet's spec-section reference, as `V9g`'s bullet does for `cka-31` and `V14b`'s for `cka-13`. Keep both leaf files' bullets identical, since `V9l-T` mirrors `V9l`. The `coverage-matrix.md` `cka-27` row already names `V9l` as the closing leaf and needs no change.

## Consequence

**Severity:** advisory

Without the `cka-27` citation the resulting test source omits the token, so a reviewer or cka-token index cannot associate `V9l`'s tests with the `cka-27` obligation, and the leaf diverges from the plan-wide convention that every code-keyed-area Tests bullet cites its `cka-<n>` token inline. No closing gate reddens — `cka-27` is a single-leaf row outside the `H5f` per-facet arm — so the gap is traceability and consistency, not a blocked pickup.
