## Finding

`M`'s `SLSH-2` happy path requires the in-process session double to **emit** an assistant response that appends as one prompt-mode turn — both `M` Adds ("a single untyped `@`-query … whose assistant response streams as one prompt-mode turn") and `M`/`M-T` Tests ("streams its **assistant response** into the user session as one appended turn") depend on the double producing assistant output. The plan, however, never states how that assistant response is supplied to the driven turn.

The only mechanism the plan defines for scripting what the double emits is `H4b`'s response-programming surface, whose category (a) is "script the sequence of assistant turns the double emits and the streamed fragments per turn." `M` depends on `M-T, H4a`; `M-T` depends on `H4a` only. Neither reaches `H4b`. `H4a` establishes the **session-double fidelity contract** (the double *reproduces* streamed-token order and single-turn append semantics) but explicitly hands the input-side scripting API to `H4b`: "The full scripted-injection **response-programming surface** the double's input side exposes is owned by `H4b`, which depends on this leaf."

So `M`'s happy path needs an emission capability that, as written, lives only in a leaf `M` does not depend on. An implementer picking up `M` is left to either silently add an `H4b` dependency or invent an ad-hoc canned-response mechanism in `H4a`/`M` — two divergent resolutions, neither sanctioned by the plan. The gap is the `M`-leaf manifestation of the broader undefined `H4a`/`H4b` response-emission boundary: `H4a`'s own fidelity self-check (items (i)/(ii)) already asserts the double emits "one streamed assistant response appended as a single prompt-mode turn," so a minimal emission capability must exist at `H4a` regardless — but the plan never says so, and never says `M` consumes it.

## Solution Space

**Shape:** multiple

### Option A — Locate a minimal emission capability in `H4a` that `M` consumes

**Approach.** State in `H4a` Adds that the harness double exposes a minimal response-emission capability — the ability to supply the assistant response/turn the double emits for a driven dispatch — and that `H4a`'s own fidelity self-check (items (i)/(ii)) already exercises it. `M` and `M-T` Adds then state that `M`'s happy-path `@`-query assistant response is supplied via that `H4a` capability; `M`/`M-T` `Deps` stay as-is. `H4b` Adds notes its response-programming surface category (a) *extends* the same `H4a` capability into the full scripted-injection surface.

**Plan edits.** `H4a` Adds: name the minimal emission capability and tie it to the existing self-check items (i)/(ii). `M` Adds / `M-T` Adds: state the assistant response is supplied through `H4a`'s minimal emission capability (no `H4b` dependency). `H4b` Adds: phrase category (a) as an extension of the `H4a` capability rather than a redefinition.

**Spec edits.** None.

**Pros.** Keeps `M` (the MVP proof) dependency-minimal; avoids pulling the full `H4b` surface — including categories (f)/(g) that model not-yet-authored `V4f`/`V9i` contracts — into the MVP. Resolves the related `H4a`/`H4b` emission-boundary gap in the same edit.

**Cons.** Requires `H4a` and `H4b` to describe one shared capability at two levels of detail; risk of two emission APIs forking if the "extends, not replaces" relationship is not stated.

**Risks.** Low. The capability already exists implicitly because `H4a`'s self-check asserts assistant emission.

### Option B — Add an `H4b` dependency to `M`/`M-T`

**Approach.** `M` and `M-T` add `H4b` to `Deps` and script the single happy-path assistant turn via `H4b` response-programming surface category (a).

**Plan edits.** `M` `Deps`: `M-T, H4a, H4b`. `M-T` `Deps`: `H4a, H4b`. `M`/`M-T` Adds: state the assistant response is scripted via the `H4b` surface category (a).

**Spec edits.** None.

**Pros.** Single owner for all emission scripting; no second emission API.

**Cons.** Inflates the MVP's dependency footprint onto a leaf that also carries advanced modeled-behaviour categories (f)/(g) unrelated to `M`'s happy path; reorders the MVP after `H4b` lands.

**Risks.** Couples the minimal end-to-end proof to a larger, later harness surface, partially defeating `M`'s purpose of proving the pipeline early.

### Recommendation

Adopt **Option A**. `H4a`'s own fidelity self-check already asserts the double emits one streamed assistant turn, so the minimal emission capability must exist at `H4a` regardless; routing `M` through it keeps the MVP dependency-minimal and resolves the broader `H4a`/`H4b` emission-boundary gap in the same edit. Resolve the `H4a`/`H4b` emission-boundary first — define the minimal emission capability in `H4a` and state that `H4b`'s category (a) extends it — then land `M`/`M-T`'s supply-path wording on that baseline. Watch the edge case that the `H4a` minimal capability and the `H4b`-extended surface are described as one capability (extend, not replace) so `M` and downstream `H4b` consumers do not fork two emission APIs.

## Consequence

**Severity:** correctness

`M` is pickable (its `Deps` of `M-T, H4a` are satisfiable) but cannot be implemented as specified without an assistant-response emission capability the plan leaves unlocated. Two reasonable implementers diverge — one adds an undeclared `H4b` dependency, another hard-codes a canned response into `H4a` or `M` — producing different dependency graphs and different harness emission surfaces for the same happy path.
