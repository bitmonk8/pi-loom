## Problem

The `Tests` sections of both `V10b-package-discovery.md` and its paired `V10b-T-package-discovery.md` carry three bullets that all cite the same bare REQ-ID `DISC-6` for three distinct bounded-walk facets: the core trip behaviour, settings-sourced bounds reaching the walk, and the per-read deadline scaling with the operator override. Only prose parentheticals distinguish them, so there is no stable per-facet handle to reference any one scenario. The `H5a` citing-test scanner is satisfied by the presence of any single `DISC-6` token, so it cannot witness that all three facets are individually covered.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e965634` (2026-06-11, "pi-loom plan: resolve \"V10c settings keys read but application not asserted\""); `750ecbb` (2026-06-12, "pi-loom plan: resolve \"V10b per-read-deadline formula variable t is undefined\"")
**History:** At plan inception (`c6a664e`, 2026-06-10) `V10b` carried a single `DISC-6` Tests bullet — no collision. `e965634` appended a second `DISC-6`-labelled bullet (settings-sourced bounds) while resolving an unrelated settings-application finding; `750ecbb` appended a third `DISC-6`-labelled bullet (per-read-deadline scaling) while resolving the per-read-deadline-formula finding. Neither follow-up sub-identified its new facet, so the triple collision accreted across two independent fix commits rather than entering in one edit.

## Solution approach

In both `docs/plan_topics/V10b-package-discovery.md` and `docs/plan_topics/V10b-T-package-discovery.md`, give each of the three `DISC-6` Tests bullets a distinct citable sub-identifier so every walk facet is independently referenceable — for example `DISC-6`, `DISC-6b`, `DISC-6c` for core-trip / settings-sourced-bounds / per-read-deadline-scaling respectively. This finding authorises the sub-ID split of the existing `DISC-6` REQ-ID under the registered `DISC` prefix. Apply the same labelling to both the implementation and `-T` leaves so they stay in sync.

## Solution constraints

- Keep the bare `DISC-6` token present on at least one bullet so the `H5a` citing-test scanner still recognises `DISC-6` closure (`governance.md`'s word-boundary-disjoint token rule treats `DISC-6b`/`DISC-6c` as distinct tokens from `DISC-6`).
- Out of scope: leave the `coverage-matrix.md` `DISC-5, DISC-6 | V10b` row unchanged — all three facets still close in `V10b`.
