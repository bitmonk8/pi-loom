## Finding

The manual real-host smoke gate (`real-host-smoke-gate.md`) fires on two triggers. Trigger-(1) (a Pi version bump) has a concrete, observable recovery: a confirmed behavioural-divergence finding blocks the bump and the prior Pi-SDK pin is restored before merge via `V18d`'s revert path. Trigger-(2) (a merge whose diff touches the four fidelity-contract axes) has no equivalent. Recognising a trigger-(2) event is a contributor obligation with no automatic detector — by design — so a contributor who does not recognise their diff as touching the four axes can merge a real double-vs-real-host divergence to `main`. The note states such an omission is "detectable only after the fact" via the `H6a` release-gate commit-message audit, but it never states what to do once such an already-merged divergence is found: there is no after-the-fact remediation analogous to trigger-(1)'s pin-restore.

Separately, the note defines an activation window: a trigger event that fires before `H7a` lands is "deferred" — its owner records the deferral in the triggering commit message and is obligated to re-run the smoke once `H7a` lands. But nothing makes that deferred re-run observable: there is no tracking that reconciles recorded deferrals against recorded re-runs, so a deferred trigger-(2) event can be recorded and then never re-run, with no signal at the `H6a` release gate.

The combined effect is an asymmetry: trigger-(1) divergences are caught and recovered pre-merge with a stated revert, while trigger-(2) divergences can silently reach `main` (via non-recognition or a dropped deferral) with no documented recovery and no audit reconciliation.

## Solution Space

**Shape:** multiple

This finding carries two independent obligations (a recognition-aid half and a recovery/tracking half that no single one-paragraph edit resolves). Both are required; address them in the order below so the second lands on a stable baseline.

### Option A — Four-axis seam touch-list recognition aid

- **Approach:** Extend the trigger-(2) definition in `real-host-smoke-gate.md` with a concrete touch-list of the path/identifier seams whose edit constitutes a trigger-(2) event, so a contributor checks their diff against a named list rather than re-deriving "touches the four fidelity-contract axes" from prose.
- **Plan edits:** `real-host-smoke-gate.md` — add, alongside the trigger-(2) clause, the production-seam and double-modelling path/identifier hints for (i) streamed-token order relative to `ctx.waitForIdle()` resolution, (ii) single-turn prompt-mode append, (iii) the `pi.on` cancel-forward subscription, and (iv) cancellation propagation.
- **Spec edits:** None.
- **Pros:** Shrinks the "did not recognise my diff" escape; bounds what counts as a trigger-(2) event, which the recovery half then builds on.
- **Cons:** The touch-list is hand-maintained and must move when the seams move.
- **Risks:** A stale list under-signals; pair with the existing per-axis owners so the list is reviewed when an axis seam relocates.

### Option B — After-the-fact remediation + deferred-run reconciliation

- **Approach:** State a recovery for an already-merged trigger-(2) divergence and make the deferred (pre-`H7a`) re-run observable.
- **Plan edits:** `real-host-smoke-gate.md` — (1) add an after-the-fact remediation: when the `H6a` commit-message audit (or any later observation) surfaces an unrecorded or skipped trigger-(2) run, re-run the smoke against the merged change, and on a confirmed behavioural-divergence finding land a revert or follow-up fix — mirroring the trigger-(1) recovery `V18d` owns; (2) make deferred runs observable: state that recorded deferrals carry forward and are reconciled against recorded re-runs at the `H6a` release-gate audit, reusing the same triggering-commit-message evidence convention. If the audit-side reconciliation is expressed in `H6a`, extend its release-gate evidence-record check correspondingly.
- **Spec edits:** None.
- **Pros:** Closes the silent-escape and never-re-run gaps and gives trigger-(2) recovery parity with trigger-(1).
- **Cons:** Recovery stays manual, consistent with the gate's stated non-CI-enforced nature.
- **Risks:** Without a mechanical hook the recovery remains a contributor obligation; the audit reconciliation is the only backstop.

### Recommendation

Land Option A first — it bounds what counts as a trigger-(2) event, which the recovery and deferred-reconciliation prose in Option B then reference — then Option B. The smoke remains a manual contributor obligation by design; these edits add guidance and an audit trail, not a CI hook. Edge cases the implementer must watch: keep trigger-(2)'s after-the-fact remediation distinct from `V18d`, which owns trigger-(1)'s pre-merge pin restore — trigger-(2) recovery is a post-merge re-run plus revert/follow-up, not a pin restore; and route the deferred-run reconciliation through the commit-message evidence record `H6a` already audits rather than introducing a separate artefact file.

## Consequence

**Severity:** advisory

A real double-vs-real-host fidelity divergence introduced by a loom change can reach `main` undetected (the contributor does not recognise the four-axis touch, or a pre-`H7a` deferral is never re-run) with no documented recovery once found, undermining the gate's stated purpose of catching divergences before they ship. No leaf is blocked and no mechanical coverage gate fails to fire — the smoke is a manual contributor obligation by design — so the gap is a guidance/recovery completeness defect in the governance note rather than a leaf-pickup blocker.
