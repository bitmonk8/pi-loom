## Problem

Inside `docs/spec_topics/pi-integration-contract/`, the file `audit-target-categories.md` carries the H1 `# Audit recognised shapes` and the body anchor `<a id="audit-recognised-shapes"></a>`, while the sibling `audit-recognised-shapes.md` carries the H1 `# Audit target categories` and the anchor `<a id="audit-target-surface-categories"></a>`. Filename, heading, and anchor name different things in each file. The intuitive link a writer would produce — `./audit-recognised-shapes.md#audit-recognised-shapes` — resolves to no anchor, because that anchor lives in the oppositely-named file. Any short-form reference to either file by filename alone (plan-leaf Spec field, commit message, review note) silently names the wrong document.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f5e89f4` (Thu Jun 4 2026, "docs: enforce 100KB hard / 30KB recommended size cap on spec set") seeded the mismatch; `7e3e94e` (Fri Jun 5 2026, "pi-loom spec: resolve 'audit-cluster files have H1 headings inverted'") cemented it.
**History:** `git log --follow` on both files shows only two relevant commits. In `f5e89f4` the spec-split tool (`docs/_tools/split_spec.py`) materialised both files at once, assigning each H1 to match its filename — but the body content placed under each filename was swapped (the "Recognised import/access shapes" paragraph landed in `audit-target-categories.md`, the "Target surface categories" paragraph in `audit-recognised-shapes.md`). `7e3e94e` then ran the per-finding fix loop on a review item that observed "H1 headings inverted" and resolved it by editing only the two H1 lines (and the matching link-text labels in `pi-integration-contract.md`'s child-page TOC) so each H1 matched its body. The local fix was correct, but it locked in the filename↔body mismatch as the new steady state and left the `audit-recognised-shapes` body anchor stranded in the file named `audit-target-categories.md`. The defect is the multi-commit interaction: a split that planted content under the wrong filenames, followed by an H1-only repair that did not chase the rename through.

## Solution approach

Realign the two sibling files so each filename matches its H1 and body anchor: move the `audit-recognised-shapes`-anchored body (H1 "Audit recognised shapes") into `audit-recognised-shapes.md`, and the `audit-target-surface-categories`-anchored body (H1 "Audit target categories") into `audit-target-categories.md`. Rewrite the cross-references that point at `./audit-recognised-shapes.md#audit-target-surface-categories` to `./audit-target-categories.md#audit-target-surface-categories`, and update the child-page TOC in `docs/spec_topics/pi-integration-contract.md` so each bullet's label and link name the same file. Update `docs/_tools/split_spec.py`'s shard tuples so a future re-split round-trips to the corrected filenames.

## Solution constraints

- Keep the body anchor IDs `audit-recognised-shapes` and `audit-target-surface-categories` unchanged; only their hosting file changes (the anchors are durable cross-link targets across the audit cluster).
