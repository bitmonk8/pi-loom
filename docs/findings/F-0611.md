## Finding

Inside `docs/spec_topics/pi-integration-contract/`, the file named `audit-target-categories.md` carries the H1 `# Audit recognised shapes` and the body anchor `<a id="audit-recognised-shapes"></a>` ("Recognised import/access shapes" paragraph), while the sibling file `audit-recognised-shapes.md` carries the H1 `# Audit target categories` and the anchor `<a id="audit-target-surface-categories"></a>` ("Target surface categories" paragraph). The body of `audit-target-categories.md` even contains an outgoing link to the sibling (`./audit-recognised-shapes.md#audit-target-surface-categories`), so every reader sees the swap at point-of-use: filename, heading, and outgoing link all name different things.

The most concrete failure surface is the anchor `audit-recognised-shapes`. The intuitive link a writer or reviewer would produce — `./audit-recognised-shapes.md#audit-recognised-shapes` — resolves to no anchor (that id lives in the sibling file `audit-target-categories.md`). The PIC topic index in `docs/spec_topics/pi-integration-contract.md` already reflects the collision by listing `[Audit target categories](./pi-integration-contract/audit-recognised-shapes.md)` and `[Audit recognised shapes](./pi-integration-contract/audit-target-categories.md)` — the TOC link labels are correct, but a contributor who shortens a reference to the filename alone (in plan files, cross-spec prose, or commit messages) will name the wrong document.

This is the same root defect catalogued one shard earlier under "audit-recognised-shapes.md / audit-target-categories.md — filenames transposed relative to H1 headings"; the finding above records the third corroborating site, where the in-body outgoing link makes the collision externally observable.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f5e89f4` (Thu Jun 4 2026, "docs: enforce 100KB hard / 30KB recommended size cap on spec set") seeded the mismatch; `7e3e94e` (Fri Jun 5 2026, "pi-loom spec: resolve 'audit-cluster files have H1 headings inverted'") cemented it.
**History:** `git log --follow` on both files shows only two relevant commits. In `f5e89f4` the spec-split tool (`docs/_tools/split_spec.py`) materialised both files at once, assigning each H1 to match its filename — but the body content placed under each filename was swapped (the "Recognised import/access shapes" paragraph landed in `audit-target-categories.md`, the "Target surface categories" paragraph in `audit-recognised-shapes.md`). `7e3e94e` then ran the per-finding fix loop on a review item that observed "H1 headings inverted" and resolved it by editing only the two H1 lines (and the matching link-text labels in `pi-integration-contract.md`'s child-page TOC) so each H1 matched its body. The local fix was correct, but it locked in the filename↔body mismatch as the new steady state and left the `audit-recognised-shapes` body anchor stranded in the file named `audit-target-categories.md`. The defect is the multi-commit interaction: a split that planted content under the wrong filenames, followed by an H1-only repair that did not chase the rename through.

## Solution Space

**Shape:** single

### Recommendation

Resolve by file rename (swap the two filenames), in one commit, co-edited with every cross-reference. Concretely:

1. Move the body of `audit-target-categories.md` (current H1 "Audit recognised shapes", anchor `audit-recognised-shapes`) into `audit-recognised-shapes.md`, and the body of `audit-recognised-shapes.md` (current H1 "Audit target categories", anchor `audit-target-surface-categories`) into `audit-target-categories.md` — i.e. swap the file contents so each filename matches its H1 and its anchor.
2. Update the eleven outgoing links currently pointing at `./audit-recognised-shapes.md#audit-target-surface-categories` (and its `../pi-integration-contract/audit-recognised-shapes.md#…` cousins) to `./audit-target-categories.md#audit-target-surface-categories`. Sites are listed under **Spec Documents** above; a single `grep -rn 'audit-recognised-shapes.md#audit-target-surface-categories' docs/` followed by mechanical rewrite covers them.
3. Update the two plan-leaf Spec fields in `V18b-inventory-audit.md` / `V18b-T-inventory-audit.md` and the `cka-20` row of `coverage-matrix.md` so they continue to cite both files (paths stay the same after rename; only the inline-body references that pair the H1 description with the filename need a sanity-read).
4. Update the topic index `docs/spec_topics/pi-integration-contract.md` so each TOC bullet's label and link point at the same-named file.
5. Update `docs/_tools/split_spec.py`'s shard tuples so a future re-split round-trips to the same filenames.

Edge cases for the implementer: (a) `version-bump-step2b.md` and `inventory-audit-intro.md` contain prose like "per *Target surface categories*" that already references the right anchor — only the path component changes; (b) `audit-target-categories.md`'s body contains an in-page self-reference via `./audit-recognised-shapes.md#audit-target-surface-categories` that becomes `./audit-target-categories.md#audit-target-surface-categories` after the swap, i.e. an intra-file fragment; (c) no `<a id="audit-recognised-shapes">` cross-link targets exist outside the file that owns the anchor (verified by `grep`), so the anchor itself needs no rename — only its hosting file does.

Picking the file-rename branch (rather than swapping H1s back) keeps the URL-bearing identifier — the filename — aligned with the durable identifier the audit ecosystem already references in its anchor IDs (`audit-target-surface-categories`, `audit-recognised-shapes`) and the labels the TOC chose to use; swapping H1s back would re-strand the anchor IDs.

## Consequence

**Severity:** correctness

Two reasonable contributors will diverge on which file is authoritative for "recognised shapes" vs "target categories," and the intuitive anchor link `audit-recognised-shapes.md#audit-recognised-shapes` resolves to nothing because that anchor lives in the file with the opposite name. Any short-form reference to either file by filename alone (plan-leaf Spec field, commit message, review note, mechanical sweep) silently names the wrong document, and the V18b implementer reading `V18b-inventory-audit`'s Spec field cannot tell which page owns recognised shapes without opening both files.
