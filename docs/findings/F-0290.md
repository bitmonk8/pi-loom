## Finding

`V14a` (`docs/plan_topics/V14a-tool-calls.md`) and its paired tests leaf `V14a-T` carry eight distinct, independently-testable concerns behind one compound **Ships when**: (1) the parse-time argument codes (`tool-arg-not-literal` / `tool-arg-arity` / `tool-arg-type-mismatch`); (2) the closed `CodeToolError` enum; (3) the four off-surface outcome routing paths (pre-eval throw, non-conforming return shape, non-settling promise, post-cancel late settlement); (4) both return-type rows (Pi-tool → `Ok(string)`, subagent-mode `.loom` → `Ok(T)`); (5) the `.loom`-callable failure surface (`Invoke*Error`); (6) the host-denial MUST from *No additional access channels*; (7) the ceiling-#4 depth-6 routing row; and (8) the swallowing-handler-attachment / three-channel late-settlement suppression assertion.

These concerns are drawn from four separate spec documents — `tool-calls.md`, `pi-integration-contract/conversation-drive.md`, `cancellation.md`, and `hard-ceilings/ceilings-3-and-4.md`. The leaf's **Tests** field is eight bullets long and its **Ships when** chains every one of them into a single condition. A vertical-slice leaf is defined (`conventions.md` §3) as "the smallest feature that can ship and be tested independently"; this leaf is well past that boundary. No reviewer can verify it in one pass, and the per-phase TDD ritual operates on a single test file that asserts all eight concerns at once.

Perform the decomposition before the reference pass so the new ID set and concern ownership are fixed and the retarget operates against a stable baseline. Use `<new>` placeholders for the sub-leaf IDs and pick finals consistent with the V14 slice's ID scheme (no invented IDs). The spec is read-only throughout; no spec edits.

**Decompose.** Replace the single `V14a` impl leaf and its `V14a-T` tests partner with a set of paired `-T`/impl sub-leaves, each owning one coherent concern group with its own **Ships when**, preserving the per-phase paired TDD structure. Cut the concerns into three groups: (1) parse-time argument codes (`tool-arg-not-literal` / `tool-arg-arity` / `tool-arg-type-mismatch`) + the closed `CodeToolError` enum + the base code-side `<name>(args)` call mechanism; (2) the four off-surface outcome paths (pre-eval throw, non-conforming return shape, non-settling promise, post-cancel late settlement) + both return-type rows (Pi-tool → `Ok(string)`, subagent-mode `.loom` → `Ok(T)`) + the `.loom`-callable failure surface (`Invoke*Error`) + the host-denial MUST from *No additional access channels* + the swallowing-handler-attachment / three-channel late-settlement suppression assertion; (3) the ceiling-#4 depth-6 routing/cancellation row. ## Consequence

**Severity:** advisory

The eight concerns share one compound **Ships when**, so no reviewer can check the leaf in a single pass and the TDD ritual reds/greens all eight at once; a concern can be under-implemented or mis-asserted without an isolated signal flagging it. Each concern is individually well-specified, so an implementer can still produce a working leaf, but review confidence and maintainability are degraded.
