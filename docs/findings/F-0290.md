## Finding

`V14a` (`docs/plan_topics/V14a-tool-calls.md`) and its paired tests leaf `V14a-T` carry eight distinct, independently-testable concerns behind one compound **Ships when**: (1) the parse-time argument codes (`tool-arg-not-literal` / `tool-arg-arity` / `tool-arg-type-mismatch`); (2) the closed `CodeToolError` enum; (3) the four off-surface outcome routing paths (pre-eval throw, non-conforming return shape, non-settling promise, post-cancel late settlement); (4) both return-type rows (Pi-tool → `Ok(string)`, subagent-mode `.loom` → `Ok(T)`); (5) the `.loom`-callable failure surface (`Invoke*Error`); (6) the host-denial MUST from *No additional access channels*; (7) the ceiling-#4 depth-6 routing row; and (8) the swallowing-handler-attachment / three-channel late-settlement suppression assertion.

These concerns are drawn from four separate spec documents — `tool-calls.md`, `pi-integration-contract/conversation-drive.md`, `cancellation.md`, and `hard-ceilings/ceilings-3-and-4.md`. The leaf's **Tests** field is eight bullets long and its **Ships when** chains every one of them into a single condition. A vertical-slice leaf is defined (`conventions.md` §3) as "the smallest feature that can ship and be tested independently"; this leaf is well past that boundary. No reviewer can verify it in one pass, and the per-phase TDD ritual operates on a single test file that asserts all eight concerns at once.

## Solution Space

**Shape:** multiple

This finding carries two coupled obligations — decompose the over-scoped leaf, then retarget every cross-reference that pins `V14a` as a coordinate. They cannot be resolved by one leaf edit; each is presented as its own option, and the recommendation fixes their order.

### Option A — Decompose `V14a` / `V14a-T` into concern-bounded sub-leaves

**Approach.** Replace the single `V14a` impl leaf and its `V14a-T` tests partner with a set of paired sub-leaves, each owning one coherent concern group and its own **Ships when**, preserving the per-phase paired `-T`/impl TDD structure. One viable cut groups: (1) parse-time argument codes + closed `CodeToolError` enum + the base code-side `<name>(args)` call mechanism; (2) the four off-surface outcome paths + both return-type rows + `.loom`-callable failure + host-denial MUST + swallowing-handler attachment; (3) the ceiling-#4 depth-6 routing / cancellation row.
**Plan edits.** New paired sub-leaf files under `docs/plan_topics/` (each with its `-T` partner); remove `docs/plan_topics/V14a-tool-calls.md` and `docs/plan_topics/V14a-T-tool-calls.md`; relink the V14 slice section in `docs/plan.md`.
**Spec edits.** None.
**Pros.** Each sub-leaf becomes reviewable in one pass; isolates the concerns that accreted; lets the four-site swallowing-handler / ceiling-#4 owners be named precisely.
**Cons.** Creates several new leaf files; the boundary between the "outcomes" and "ceiling-cancel" groups must be drawn carefully.
**Risks.** If the cut bisects a concern (cancellation appears in both off-surface-outcome suppression and ceiling-#4 routing), an assertion could fall through the gap.

### Option B — Retarget every cross-reference to the new sub-leaf IDs

**Approach.** Propagate the new IDs through every leaf and matrix row that currently pins `V14a`.
**Plan edits.** `coverage-matrix.md` TOOL row (line 85) and conversation-drive denial-surface row (line 90) → map to the owning sub-leaves; `H5b` Deps range `V14a`–`V14b` → enumerate the new IDs (and add each coverage-producing sub-leaf per `conventions.md` §REQ-ID discipline *Transitive-completeness*); `H7a` Deps + Adds provenance list + permitted-code list; `V17a` / `V17a-T` code-side `execute()` four-site owner → the sub-leaf owning swallowing-handler attachment; `V5e` / `V5e-T` `CodeToolError`-wrapping site owner → the sub-leaf owning ceiling-#4; `V13f` four-site set reference; `V14b` / `V14b-T` Deps; `V18d` Deps; `V4c` / `V4c-T` "not the live V14a surfaces" references.
**Spec edits.** None.
**Pros.** Keeps the cross-reference graph and the closing/transitive-completeness gates consistent.
**Cons.** Touches many files; depends on Option A's IDs being fixed first.
**Risks.** A missed reference leaves a dangling `V14a` ID; the `H5d` transitive-completeness gate and the closing gates can misfire if `H5b` Deps and the coverage-matrix closing-leaf columns fall out of lockstep.

### Recommendation

Resolve Option A first, then Option B. The decomposition is scope-bounding — it fixes the new ID set and the concern ownership that every retarget depends on — so landing it first lets the reference pass operate on a stable baseline. Use `<new>` placeholders for the sub-leaf IDs and let the implementer pick finals consistent with the V14 slice's ID scheme (no invented IDs). The spec is read-only throughout this fix. Edge cases to watch: keep the swallowing-handler / late-settlement-suppression assertion and the ceiling-#4 depth-6 cancellation routing within a single owner so the four-site routing model (`V17a`, `V13f`, `V9i`, `V5e`) names exactly one V14-side owner; add each coverage-producing sub-leaf to `H5b`'s Deps so the warn-only canary and the `H6a` release gate stay sequenced after complete coverage; and re-run the closing-gate reconciliation after retargeting so no coverage-matrix closing-leaf cell or `H5b` Deps entry references the retired `V14a` ID.

## Consequence

**Severity:** advisory

The eight concerns share one compound **Ships when**, so no reviewer can check the leaf in a single pass and the TDD ritual reds/greens all eight at once; a concern can be under-implemented or mis-asserted without an isolated signal flagging it. The leaf's size has already drawn three rounds of accretion. Each concern is individually well-specified, so an implementer can still produce a working leaf, but review confidence and maintainability are degraded.
