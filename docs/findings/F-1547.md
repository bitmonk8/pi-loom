## Finding

The paragraph immediately before **PIC-1** in `runtime-event-channel.md` (after the *Deduplication and lifetime rules* block) opens with a future-work enumeration:

> "Aggregation, latency histograms, per-loom token reports, retention policies, and a consumer-facing read API for the event stream are deferred to [Future Considerations](../future-considerations.md). loom 1.0 ships the always-log set and the payload shape only; the channel is intentionally write-only at the spec level, with downstream consumers free to read from Pi's session transcript via existing surfaces."

The first sentence is a catalogue of deferred features. That catalogue already has a normative owner: `future-considerations/surface-extensions.md` lists "**Richer runtime-event telemetry** — per-loom token aggregation, latency histograms, cost reporting, retention policies, and a consumer-facing read API for the `details: { event }` stream" against the same channel. Restating the deferred-item list inline in the channel's normative section duplicates that catalogue and creates a second site that must be kept in lock-step (the inline list already omits the catalogue's "cost reporting" item, demonstrating the drift risk).

The second sentence carries the normative boundary for loom 1.0 — that only the always-log set and payload shape ship, and that the channel is write-only at the spec level — and is the part a reader needs to implement the section correctly. Removing the leading deferred-work sentence leaves the normative boundary intact while eliminating the duplicated forward-looking enumeration.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** b3bc4ce — spec: pin failure-observability surface (Cluster A, Option C) (2026-05-04, Thomas Andersen)
**History:** The Runtime event channel section was introduced wholesale by b3bc4ce, which simultaneously added the inline deferral sentences (the "intentionally write-only at the spec level" / "consumer-facing read API for the event stream" prose, then in `pi-integration-contract.md`) and the `future-considerations.md` deferral entry. The duplication was therefore born with the section in a single commit; the later size-cap split (f5e89f4) only carried the inline copy into `runtime-event-channel.md` and the deferral entry into `surface-extensions.md` without removing the overlap.

## Solution Space

**Shape:** single

### Recommendation

Delete the first sentence ("Aggregation, latency histograms, per-loom token reports, retention policies, and a consumer-facing read API for the event stream are deferred to [Future Considerations](../future-considerations.md)."). Keep the second sentence ("loom 1.0 ships the always-log set and the payload shape only; the channel is intentionally write-only at the spec level, with downstream consumers free to read from Pi's session transcript via existing surfaces."), which states the loom 1.0 normative boundary. The deferred-telemetry enumeration remains owned by `future-considerations/surface-extensions.md`; no cross-reference need be added, since the retained sentence already conveys the write-only scope.

## Consequence

**Severity:** cosmetic

A redundant forward-looking enumeration sits in a normative section, duplicating the `future-considerations` catalogue. No implementer behaviour diverges; the only concrete effect is a second maintenance site that can drift out of sync with the canonical deferred-telemetry list (it already has — the inline copy omits "cost reporting").
