## Finding

`H7a` is a Horizontal-phase leaf that cites a `Convention.` field (phase categories — end-to-end harness) and openly self-describes as closing no new spec REQ-ID. Its substance is a cross-slice integration-regression gate: a committed golden transcript (exact turn order, count, and per-turn content), a committed golden diagnostics list, and a committed permitted-code list, all asserted against a single representative multi-feature fixture `.loom` run through the `H4a` harness. None of that machinery is requested by the spec. Per `GOV-18` binding scope (`spec_topics/governance/corpus-direction-and-scope.md#gov-18`), the spec corpus binds only the implementation target (arm a) and the spec corpus itself (arm b); a bespoke regression harness is plan-elected tooling outside both arms.

The leaf does carry one genuine spec-anchored obligation: its fourth `Tests` bullet drives a co-occurring ceiling #3 / runtime-class ceiling #2 breach through the live pipeline and asserts the `CIO-5` arbitration order, co-witnessing `CIO-5`'s cross-site never-interleaves property. `coverage-matrix.md` confirms this — `CIO-5` maps to `V16a`, `H7a`, with the rationale that `V16a` closes the seam-local arbitration while `H7a` co-witnesses the cross-site property. That arm is real and should stay.

The defect is that the leaf does not distinguish the genuine `CIO-5` co-witness obligation from the plan-elected regression scaffolding. It reads as if the entire harness were required, which obscures, for any future scope decision, which part of `H7a` carries spec footing and which is discretionary tooling the plan could trim or defer. The `H4a` manual real-host smoke (pass criterion (e)) and the `H6a` release-gate evidence record both consume `H7a`'s committed permitted-code list, so the discretionary machinery is load-bearing for those leaves as currently authored.

## Solution Space

**Shape:** multiple

### Option A — Mark the plan-elected boundary in place

**Approach:** Keep the harness and the `CIO-5` arm; add an explicit statement in `H7a`'s Adds that the golden-transcript, golden-diagnostics, and permitted-code regression machinery is plan-elected tooling outside the spec corpus's binding scope (`GOV-18`), distinct from the fourth `Tests` bullet's `CIO-5` cross-site co-witness obligation, which carries the leaf's only spec footing.
**Plan edits:** `docs/plan_topics/H7a-integration-acceptance.md` Adds — state that the regression machinery is plan-elected under `GOV-18` and that the `CIO-5` arm is the spec-anchored obligation (matching the `CIO-5` row and closure rationale already in `coverage-matrix.md`).
**Spec edits:** None.
**Pros:** Preserves `H4a` criterion (e) and the `H6a` release-gate evidence record, which consume `H7a`'s permitted-code list. Makes the discretionary/spec-anchored boundary explicit for future scope decisions. Smallest diff.
**Cons:** Retains discretionary tooling on the 1.0 path.
**Risks:** None material.

### Option B — Narrow H7a to the CIO-5 live-site obligation

**Approach:** Strip the golden-transcript / golden-diagnostics / permitted-code regression machinery and retain only the `CIO-5` co-occurring-breach live-site assertion.
**Plan edits:** `docs/plan_topics/H7a-integration-acceptance.md` — remove the first three Tests bullets and the corresponding Adds/Ships-when prose; retain the fourth Tests bullet. `docs/plan_topics/H4a-factory-shell-and-harness.md` — rework manual real-host smoke pass criterion (e), which references `H7a`'s committed permitted-code list. `docs/plan_topics/H6a-live-corpus-activation.md` — rework the Release-gate acceptance item and the `H7a` Deps rationale, both of which reference the committed permitted-code list and fixture.
**Spec edits:** None.
**Pros:** Removes all plan-elected tooling; leaves only the spec-anchored obligation.
**Cons:** Cascades into `H4a` and `H6a`, which currently depend on `H7a`'s committed permitted-code list and fixture as a single committed reference; loses the cross-slice integration regression coverage the harness provides.
**Risks:** Breaking the permitted-code-list referent without re-homing it would leave `H4a` criterion (e) and the `H6a` evidence check dangling.

### Recommendation

Take Option A. The `CIO-5` arm is genuine and the harness is already consumed by `H4a` and `H6a`, so removal is the larger and more disruptive change; an explicit `GOV-18` plan-elected annotation closes the disclosure gap while keeping the dependent leaves intact. Edge case for the implementer: keep the annotation consistent with the `CIO-5` closure rationale already stated in `coverage-matrix.md` so the two surfaces do not drift.

## Consequence

**Severity:** advisory

If the plan ships unchanged, a maintainer making a scope decision (for example, deferring plan-elected tooling off the 1.0 critical path) cannot tell from `H7a` which content is the genuine `CIO-5` cross-site co-witness obligation and which is discretionary regression scaffolding; the leaf reads as if the whole harness were spec-required. Implementers can still build a working leaf, so the impact is a guidance / disclosure gap rather than a divergence.
