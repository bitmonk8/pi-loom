## Finding

`binder-inference.md` introduces the per-attempt binder call with "The runtime populates one `complete(model, context, options)` call per binder attempt as follows:" and then enumerates the full option set: `model`, `context.systemPrompt`, `context.messages`, `context.tools`, `options.temperature`, `options.signal`, the conditional seed key, and `options.onResponse`. The framing presents this enumeration as the complete population list for the call.

The page's closing paragraph then states that loom "re-issues each budgeted retry immediately, delegating per-attempt backoff and any server-requested wait (e.g. an HTTP `Retry-After`) to this call's `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs`." This phrasing reads as loom relying on those two fields to govern backoff, yet neither field appears in the population enumeration, no value is assigned, and the page never says they are deliberately left at the `@earendil-works/pi-ai` library default. An implementer working from the enumeration alone never sets the fields; an implementer reading the closing prose infers loom takes a deliberate position on them. The two statements do not agree on whether `maxRetries` / `maxRetryDelayMs` are part of the binder call's deterministic population contract.

`StreamOptions` is owned by `@earendil-works/pi-ai`; the unresolved question is purely whether the spec leaves these fields unset (library defaults apply) or pins values, not whether loom should introduce its own timeout/backoff surface. The loom-level inter-attempt behaviour (immediate re-issue under the per-invocation retry budget) is already pinned; only the provider-level per-attempt fields are dangling.

## Issue introduction
**Verdict:** single-commit introduction
**Introducing commit:** `d0bcdb2` — "pi-loom spec: resolve \"binder per-invocation retry inter-attempt timing\""
**History:** The corpus is git-tracked. `git log -S` on both `maxRetryDelayMs` and the phrase `delegating per-attempt backoff` against `binder-inference.md` returns exactly one commit, `d0bcdb2`. `git show d0bcdb2` shows it rewrote the closing sentence of the structured-output-extraction paragraph: the prior text ended "…governed by the binder's [per-invocation retry budget]; inter-attempt timing is out of scope here," and the commit replaced the out-of-scope clause with "…which also pins inter-attempt timing: loom re-issues each budgeted retry immediately, delegating per-attempt backoff and any server-requested wait (e.g. an HTTP `Retry-After`) to this call's `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs`." The same commit did not touch the options enumeration above. Before `d0bcdb2` the enumeration carried no obligation regarding inter-attempt timing (it was explicitly out of scope), so there was no inconsistency; the defect entered when the new delegation phrase began referencing two `StreamOptions` fields the unchanged population list neither sets nor explicitly defers.

## Solution Space

**Shape:** single

### Recommendation

In the `complete()` options enumeration, add an explicit statement of how `StreamOptions.maxRetries` and `StreamOptions.maxRetryDelayMs` are handled. State that loom leaves both at the `@earendil-works/pi-ai` library default and imposes no per-attempt timeout on the binder call beyond `loomAbort.signal` cancellation, so the closing-paragraph delegation phrase describes reliance on the library's default backoff behaviour rather than loom-set values. Keep the loom-level inter-attempt rule (immediate re-issue under the per-invocation retry budget) as-is; do not introduce a loom-owned timeout/backoff knob layered over the external `StreamOptions` surface. If a future decision is to pin concrete values instead of accepting the defaults, those values belong in the enumeration too — the resolution is to make the enumeration and the delegation phrase agree on a single position, whichever it is.

## Consequence

**Severity:** advisory

An implementer who treats the enumeration as exhaustive leaves the fields at the pi-ai default; one who reads the closing prose as loom setting backoff policy may pin explicit values. Either path yields a working binder call, but the two diverge in per-attempt retry/backoff timing, and the deterministic-envelope acceptance criterion is left without a stated position on these fields.
