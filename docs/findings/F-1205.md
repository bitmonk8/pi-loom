## Finding

GOV-1 *Per-page progressive normalisation* in `req-id-prefix-table-active-a.md` defines the per-commit diff to evaluate as follows:

> taking the first-parent diff (in git's sense, per `git log --first-parent` / `git diff-tree --first-parent`) for **merge or squash commits**, and the diff against the commit's **sole parent** for **single-parent commits**

The two arms partition on incompatible axes. The first arm names *workflow shapes* ("merge", "squash"); the second names a *git structural property* (parent count = 1). The word "squash commit" has no formal git meaning: a commit produced by `git merge --squash` carries exactly one parent and is structurally indistinguishable from any other single-parent commit. It therefore satisfies the predicate "single-parent commit" simultaneously with being a "squash commit", and an implementer cannot tell which arm's diff machinery to apply.

In practice the diff bytes coincide for a single-parent commit regardless of which arm is invoked (`git diff-tree --first-parent <c>` and `git diff <c>^ <c>` produce the same patch when `<c>` has one parent), so the overlap is benign for the most common case. The defect is in the *classification language*, not the resulting behaviour: the rule mixes workflow vocabulary with structural vocabulary in a way that cannot be implemented as written and that GOV-22 and GOV-17's "Triggering and scope" clause both inherit by reference ("on the same per-page terms as GOV-1 *Per-page progressive normalisation*"), propagating the same ambiguity to two other rules.

## Solution Space

**Shape:** single

### Recommendation

Replace the workflow-name grouping with a structural (parent-count) grouping that partitions cleanly:

> At every commit on `main`, the *commit diff* this rule evaluates is determined by the commit's parent count: a commit with two or more parents (a merge commit) uses its first-parent diff, in git's sense (`git diff-tree --first-parent <c>`, equivalently `git log --first-parent`); a commit with exactly one parent (including a `git merge --squash` result, a rebase result, and any ordinary non-merge commit) uses its diff against that sole parent. Root commits (zero parents) are vacuously out of scope for this rule.

Notes for the implementer:

- Drop the words "merge", "squash", and "single-parent" as classification labels; the partition is on parent count alone (`≥2` vs `=1` vs `0`), which is decidable from `git rev-list --parents -1 <c>` without knowing how the commit was produced.
- The behavioural content of the rule is unchanged for every commit that actually appears on `main`: for a single-parent commit the first-parent diff and the sole-parent diff are byte-equal, so any commit previously read as "squash → first-parent arm" lands the same way under "single-parent → sole-parent arm".
- Add a parenthetical naming the root-commit case explicitly (`zero parents → vacuously out of scope`); the current text does not address it.
- GOV-22 and GOV-17 *Triggering and scope* inherit the grouping by phrase ("on the same per-page terms as GOV-1 *Per-page progressive normalisation*"); no edit is needed on those pages — they pick up the corrected partition automatically.

## Consequence

**Severity:** advisory

A scrupulous implementer reading the rule must invent an ordering rule to resolve the overlap (e.g. "structural arm wins") because the spec gives none. Two implementers will likely converge on the same diff for any real commit (the two arms produce identical bytes for single-parent commits, including squashes), so observable behaviour does not diverge — but the rule cannot be cited unambiguously in a review-time disposition or in a hypothetical CI gate, and GOV-22 / GOV-17's inheriting clauses are pulled into the same ambiguity.
