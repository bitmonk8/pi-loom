## Finding

The `node_modules/` root (#3) paragraph in `docs/spec_topics/discovery/package-and-settings.md` follows the robust mechanism rule (`lstat` → `isSymbolicLink()` → silently filter) with a concrete assertion about pnpm's on-disk layout: "pnpm's default isolated node-linker writes top-level `node_modules/<pkg>` entries as symlinks into `node_modules/.pnpm/…`, so looms shipped by pnpm-isolated dependencies are out of scope for this root; such projects install loom packages via `pi install` (roots 1 or 4) or switch pnpm to its hoisted node-linker mode." The sentence states this as a present-tense fact with no pnpm-version qualifier — neither the *isolated* default, the `.pnpm/` virtual-store path, nor the *hoisted* opt-out are pinned to a pnpm major.

Two of those facts are real-world version-dependent: pnpm's default `node-linker` value has been the *isolated* layout for the entire pnpm 7/8/9 line but is a configurable setting and has flipped before in pnpm history; the `.pnpm/` virtual-store directory name is the current convention but is itself controlled by pnpm settings (`virtual-store-dir`). Phrased as version-independent prose, the sentence reads as a normative claim about pnpm rather than as an illustrative observation, and any future pnpm release that changes the default linker or the virtual-store path would make the spec text wrong without changing what the extension actually has to do (the `lstat`/`isSymbolicLink()` filter and the recourse — `pi install` or hoisted mode — are entirely independent of pnpm's internal default).

The implementer-relevant rule that survives any pnpm change is the mechanism: symlinked immediate children of `node_modules/` are filtered out silently. The pnpm-specific paragraph is a non-normative explanation of *which real-world tool that filter discards entries from*, and should be framed that way.

## Solution Space

**Shape:** single

### Recommendation

Reshape the trailing pnpm sentence in `docs/spec_topics/discovery/package-and-settings.md` root-#3 paragraph so the normative content is *only* the mechanism and the version-dependent pnpm details are demoted to a non-normative example. Concretely:

- Keep the existing mechanism sentence intact: "Entries here are classified by the same `lstat`-based immediate-child rule used for the other roots; because `lstat` does not follow symlinks (per [PIC-13](../pi-integration-contract/host-interfaces-services.md#pic-13)), an entry that is a symlink rather than a real directory reports `isSymbolicLink()` true and `isDirectory()` false and is filtered out silently — it is neither a candidate package nor a scope directory."
- Replace the trailing pnpm sentence with a non-normative observation that scopes itself to "package managers that use symlinks for top-level `node_modules/` entries (e.g. pnpm in its symlink-based linker modes, the default at the time of writing)" and names the recourse generically: install loom packages via `pi install` (roots 1 or 4), or configure the package manager to materialise real directories under `node_modules/` (e.g. `node-linker=hoisted` on current pnpm).
- Mark the pnpm half clearly as illustrative — a sub-bullet labelled "(Non-normative.)" or an inline "for example" clause works; the goal is that a future pnpm change to the default linker, the virtual-store directory, or the linker-mode names does not invalidate the spec text.

Edge cases the rewrite must preserve: (a) `node_modules/.bin` and other immediate non-symlink entries that pnpm itself creates as real directories are still subject to the same `lstat` rule — they get inspected as candidate packages and fall out naturally when their `package.json` is missing or invalid (no special case needed); (b) Yarn PnP installs that have no `node_modules/` at all are already covered by the root-missing silent-skip rule at the head of the same paragraph and need no mention here.

## Consequence

**Severity:** advisory

Implementer behaviour is unaffected today: the mechanism the test gate enforces (`lstat` returns `isSymbolicLink() = true` → filter silently) is correct against every shipping pnpm version. The defect is that the prose explanation overcommits — a pnpm release that changes the default `node-linker` or moves the virtual store directory would silently make the spec text wrong while the implementation continued working, and a reader who treats the sentence as authoritative on pnpm's layout could be misled when diagnosing a project that has explicitly set `node-linker=hoisted` (where the paragraph's recourse advice — "switch pnpm to its hoisted node-linker mode" — is already moot).
