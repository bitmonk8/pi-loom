## Finding

The third `Tests.` bullet of [`H7a`](../../docs/plan_topics/H7a-integration-acceptance.md) is the one beginning "a **committed permitted-code list** is checked in alongside the fixture `.loom`…". Read as an acceptance criterion it asserts nothing the harness can green or red on: it states that the artifact exists ("is checked in alongside the fixture"), what it contains ("enumerates the union of `loom/...` codes the slices in **Deps** … *can* emit"), and what role it plays (the reviewed reference set that `H4a`'s smoke pass criterion (e) and the `H6a` release-gate evidence record check live codes against, "replacing a per-run by-hand reconstruction"). All of that defines what the artifact IS and WHY it exists — material that belongs in `Adds.`, alongside the existing "committed … derived … kept current under the same per-Deps-slice provenance obligation" prose that already lives there.

The only part of the bullet that is a verifiable property of the artifact is the relation that the permitted-code list is a **superset** of the golden diagnostics list (`golden ⊆ permitted`), so every code the deterministic run emits is permitted and the in-process gate and the real-host smoke cannot disagree on a code the fixture path actually emits. That property — and only that — is what a `Tests.` bullet should carry; the construction and role description should move up to `Adds.`.

As written, the leaf's `Adds.` partly describes the artifact but the bulk of its construction/membership/role text sits under `Tests.`, where it masquerades as an acceptance criterion while gating nothing.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H7a-integration-acceptance.md`, relocate the construction/membership/role description of the committed permitted-code list from the `Tests.` bullet into `Adds.`, leaving only a checkable assertion in `Tests.`:

- Into `Adds.` (it already states the list is "committed … derived … kept current under the same per-Deps-slice provenance obligation"): fold in the remaining construction and role text — that the permitted-code list is checked in alongside the fixture `.loom` next to the committed golden transcript and golden diagnostics list; that it enumerates the union of `loom/...` codes the Deps slices *can* emit; and that it is the single reviewed reference set `H4a`'s real-host smoke pass criterion (e) and the `H6a` release-gate evidence record check live emitted codes against, replacing a per-run by-hand reconstruction of the union.
- Keep in `Tests.` a bullet that asserts the one observable property: the committed permitted-code list is a superset of the committed golden diagnostics list (`golden ⊆ permitted`) — every code the deterministic fixture run emits appears in the permitted-code list.

Do not touch the spec for this fix (it is read-only here). Preserve `H4a` and `H6a`'s existing references to the permitted-code list unchanged — only the placement of text inside `H7a` moves. Honour the project's leaf-ID and `Convention:`-tag format per `conventions.md`.

Edge case the implementer must watch: the superset assertion must reference the **committed** golden diagnostics list (the codes the deterministic fixture path actually emits), not the broader emittable set, so the relation it checks is `golden ⊆ permitted` and not a tautology.

## Consequence

**Severity:** advisory

A reader scanning `H7a`'s `Tests.` section sees a paragraph that defines an artifact but states no pass/fail condition, so the section's gating intent is unclear; a reader scanning `Adds.` does not get the full construction/membership contract in the place that contract belongs. The implementer can still produce a working leaf — the artifact's existence is also noted in `Adds.` — but the misplacement obscures what the gate actually checks (the `golden ⊆ permitted` superset relation) versus what it merely describes.
