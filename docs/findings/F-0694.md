## Finding

The `concat(other)` row in [Expressions — Built-in methods](../../docs/spec_topics/expressions.md#built-in-methods-and-properties) gives the signature `(other: array<U>): array<T ⊔ U>` and pins the result element type as the LUB of `T` and `U` under `⊑`. The disjoint-element clause then narrows the picture to one case: *"Disjoint element types union exactly as the array-literal LUB rule (case 2) computes, so `array<integer>.concat(array<string>)` types as `array<integer | string>`."* But the array-literal LUB rule has three cases, not one. Case 2 covers primitive (and union-of-primitive) elements, where TYPE-5 / TYPE-6 yield a clean union. Case 3 — the rule that immediately follows in the same section — says *"Object schemas do not unify implicitly — an array containing two different named schemas yields `array<A | B>` only if some sink in scope expects a union; otherwise it is `loom/parse/array-no-common-type`."*

`array<Cat>.concat(array<Dog>)` (two disjoint nominal object schemas, [TYPE-10](../../docs/spec_topics/type-system.md#type-10)) falls squarely under case 3, not case 2. But case 3 is array-literal-shaped: it requires *a sink in scope* and, when none exists, raises an array-literal-specific diagnostic that does not name `concat`. A method call's receiver and argument do not supply a sink to each other, and there is no concat-specific diagnostic registered, so a conforming implementer is left to pick from three incompatible readings: (a) silently produce `array<Cat | Dog>` (overriding case 3's "no implicit union for named schemas" rule for the call case), (b) defer to case 3 and reject — but then with what diagnostic, since `loom/parse/array-no-common-type` is named for an array literal, not a method call, and (c) accept only when a third sink (the surrounding binding) supplies the union, otherwise reject.

The same fork recurs for inline object types ([TYPE-8](../../docs/spec_topics/type-system.md#type-8)) with non-equal field sets, and for any future `array<T>`-returning method that inherits this signature pattern. The cited sentence's "(case 2)" parenthetical reads as if only one disjointness case existed; in fact it silently elides the one case where the LUB story is contested.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commit:** `deea483` — *pi-loom spec: resolve "array<T>.concat(other) admissibility routed through ⊑"* (2026-06-04)
**History:** Before `deea483`, the row read `(other: array<T>): array<T>` with the gloss *"element type must match"* — disjointness was not addressed at all (a different completeness gap, which was the prior spec-review finding T11). `deea483` rewrote the row to route admissibility and result type through `⊑`, introducing the `T ⊔ U` signature and the sentence *"Disjoint element types union exactly as the array-literal LUB rule (case 2) computes, so `array<integer>.concat(array<string>)` types as `array<integer | string>`."* That sentence is the locus of the present defect: it selectively cites array-literal *case 2* while the array-literal section, in the same topic file, defines a *case 3* (named object schemas do not unify implicitly) that the new wording neither cites nor disposes of. The defect is therefore not pre-existing; it was introduced by `deea483`'s partial routing of the disjointness story, against an unchanged case-3 backdrop in the array-construction section. No other commit since `deea483` has touched the row or the array-construction cases.

## Solution Space

**Shape:** multiple

### Option A — Reject; concat-specific diagnostic

**Approach.** Treat `concat` consistently with array-literal case 3: when the receiver and argument element types are distinct named object schemas (TYPE-10), or distinct inline object types with non-equal field sets (TYPE-8), and `T ⊔ U` would therefore not be one of TYPE-1 / TYPE-2 / TYPE-5 / TYPE-6, reject at parse time. Register a concat-specific diagnostic (proposed: `loom/parse/concat-no-common-element-type`) so the message can name the method, the two element types, and the recommended remedy ("annotate the surrounding binding with `array<A | B>` and re-issue the call").

**Spec edits.**
- `expressions.md`: rewrite the `concat(other)` row's disjointness sentence to enumerate the three cases (primitive / union-via-TYPE-5,6 — produces LUB; identical or `integer⊑number` — produces TYPE-1 / TYPE-2 result; otherwise — `loom/parse/concat-no-common-element-type`). Drop the bare "(case 2)" parenthetical.
- `diagnostics/code-registry-parse.md`: add the new code with template `concat element types have no common type: array<<T>>.concat(array<<U>>); annotate the surrounding binding with array<<A> | <B>>` (or reuse phrasing parallel to `array-no-common-type`).

**Pros.** Preserves the array-literal case-3 invariant ("named schemas do not unify implicitly without an explicit sink") uniformly across array-construction sites; the diagnostic names `concat` so the remedy is actionable; soundness reading is unambiguous.

**Cons.** Adds a diagnostic code; users who want the union must hoist the call through an annotated binding (`let xs: array<A | B> = a.concat(b)`), which is mildly awkward for the common "I have two arrays of related-but-disjoint schemas, give me one array of either" case.

**Risks.** None beyond the routine code-registry sweep.

### Option B — Always union; concat is its own sink

**Approach.** Make `concat` an explicit carve-out from case 3: the call form `a.concat(b)` is itself the user's expression of intent to combine, so it acts as its own union sink. The result type is unconditionally `array<T ⊔ U>` where `T ⊔ U` is the union under TYPE-5 / TYPE-6 even when `T` and `U` are disjoint named or inline object types. The only rejection path remaining is `void` operands and pre-existing TYPE-9 unresolvable cases.

**Spec edits.**
- `expressions.md`: rewrite the `concat(other)` row's disjointness sentence to read approximately *"Disjoint element types union to `array<T | U>` unconditionally — `concat`'s explicit two-array form supplies the union sink that an array literal lacks, so the case-3 restriction on implicit unification of named schemas does not apply."* Cite TYPE-5 / TYPE-6 for the union, and TYPE-10 for the nominal identities on each side.
- No diagnostics registry change.

**Pros.** No new diagnostic code; the common "concatenate two typed arrays" call just works; the call-site form is genuinely more explicit than a bare literal, so the case-3 rationale does not transfer.

**Cons.** Introduces an asymmetry between array literals and `concat` that a reader of the array-construction section must learn from the `concat` row alone; future `array<T>`-returning methods (e.g. a hypothetical `array.flat` or `array.map`) inherit a precedent of "the method form acts as its own sink", which may not always be desired.

**Risks.** A user who concat-s two unrelated nominal schemas by accident loses the case-3 safety net; the resulting `array<A | B>` will only fail later at a typed binding or call site. Manageable, since the later site does diagnose; mention this in the row.

### Recommendation

Adopt **Option A**. The array-literal case-3 invariant — *named schemas do not unify implicitly without an explicit sink* — is a deliberate design choice in the spec, and `concat` is not meaningfully more "explicit" than `[a, b]`: both are two-element combinators. Carving `concat` out (Option B) sets a precedent that the next array-returning stdlib member will have to relitigate, whereas Option A composes: any future `array<T>`-returning method routes the same way and gets the same diagnostic for free.

Edge cases the implementer must watch:

- The TYPE-2 path (`array<integer>.concat(array<number>)`) is *not* rejected — it widens to `array<number>` because the LUB is one of the primitive cases. The new diagnostic fires only when no TYPE-1 / TYPE-2 / TYPE-5 / TYPE-6 LUB exists.
- `array<Cat>.concat(array<Cat>)` produces `array<Cat>` via TYPE-1; no diagnostic.
- When a sink *is* in scope and expects `array<A | B>` (e.g. `let xs: array<A | B> = a.concat(b)`), the call should be admitted — match the array-literal case-1 sink rule. Spell this out in the row so implementers do not raise the new diagnostic in the sink-narrowed case.
- The diagnostic's message template must render both element-type names; for unresolvable callees (TYPE-9), the check is skipped as the existing TYPE-9 paragraph already specifies.

## Consequence

**Severity:** correctness

Two reasonable implementers reading the current row diverge on `array<Cat>.concat(array<Dog>)`: one accepts and types it `array<Cat | Dog>`, the other rejects (and then disagrees with a third implementer on which diagnostic to raise). The resulting cross-implementation incompatibility is silent at the row's own level — both readings can claim conformance to "the same LUB the array-literal rule computes (case 2)" — and only surfaces when a real program mixes nominal element schemas under concat.
