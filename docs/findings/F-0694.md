## Problem

The `concat(other)` row in expressions.md gives signature `(other: array<U>): array<T ⊔ U>`, routing the result element type through the `⊑` LUB, then narrows disjointness to "(case 2)" of the array-literal LUB rule. But that rule has a case 3 — named object schemas do not unify implicitly without a sink — which the concat sentence neither cites nor disposes of. So `array<Cat>.concat(array<Dog>)` (disjoint TYPE-10 schemas, and likewise non-equal TYPE-8 inline object types) has no defined result: conforming implementers diverge between silently producing `array<Cat | Dog>`, rejecting (with no concat-specific diagnostic registered), or accepting only when a surrounding sink supplies the union.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commit:** `deea483` — *pi-loom spec: resolve "array<T>.concat(other) admissibility routed through ⊑"* (2026-06-04)
**History:** Before `deea483`, the row read `(other: array<T>): array<T>` with the gloss *"element type must match"* — disjointness was not addressed at all (a different completeness gap, which was the prior spec-review finding T11). `deea483` rewrote the row to route admissibility and result type through `⊑`, introducing the `T ⊔ U` signature and the sentence *"Disjoint element types union exactly as the array-literal LUB rule (case 2) computes, so `array<integer>.concat(array<string>)` types as `array<integer | string>`."* That sentence is the locus of the present defect: it selectively cites array-literal *case 2* while the array-literal section, in the same topic file, defines a *case 3* (named object schemas do not unify implicitly) that the new wording neither cites nor disposes of. The defect is therefore not pre-existing; it was introduced by `deea483`'s partial routing of the disjointness story, against an unchanged case-3 backdrop in the array-construction section. No other commit since `deea483` has touched the row or the array-construction cases.

## Solution approach

Rewrite the `concat(other)` row's disjointness sentence in expressions.md to dispose of all three array-literal cases for the method call — LUB via TYPE-5/TYPE-6 for primitive and union elements, the TYPE-1/TYPE-2 result for identical or `integer ⊑ number` widening, admission under a surrounding `array<A | B>` sink per case 1, and rejection otherwise — and drop the bare "(case 2)" parenthetical. Register a concat-specific diagnostic in diagnostics/code-registry-parse.md (proposed code `loom/parse/concat-no-common-element-type`) whose message names the method and the two element types and points to annotating the surrounding binding with `array<A | B>`.

## Solution constraints

- The result rule MUST NOT introduce implicit unification of disjoint named object schemas (TYPE-10) or non-equal inline object types (TYPE-8); concat must stay consistent with array-literal case 3 — no implicit union of named schemas without a sink in scope.
