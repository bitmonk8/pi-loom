## Finding

The closing paragraph of `ceiling-invariants-and-audit.md` (*Five-site list co-edit obligation.*) is a spec-maintenance instruction, not a runtime requirement. It states that ceiling #4's per-boundary table is the canonical enumeration of the AJV enforcement points and that "a future loom 1.0 leaf that adds, removes, or splits a row of that table **MUST** update the five-site enumeration in CIO-3 above … and the per-site reachable-mask-domain table in [Pi Integration Contract — PIC-1] … in the same commit, in addition to the GOV-30 aggregator update covered above."

The MUST is addressed to a future *spec editor* performing a ceiling-set change, not to a runtime implementer. It pins a cross-page lock-step co-edit obligation (ceiling-#4 table ↔ CIO-3 five-site enumeration ↔ PIC-1 reachable-mask-domain table) of exactly the kind GOV-30/GOV-31 already own. A runtime implementer reading the ceiling spec to build the validator gains nothing actionable from it and must parse a false implementer-facing MUST; meanwhile the corpus's lock-step co-edit discipline is fragmented away from its canonical governance home.

This is the same defect pattern as several sibling paragraphs in the file (the *Routing obligation for future ceilings*, *Release gate*, and *loom 1.0.0 rejected-candidate record* blocks), all of which are spec-editor procedural prose embedded in a normative ceiling page. This finding is the single-sentence instance at end of file.

The defect is therefore present since its introduction at `5d14719`.

## Solution Space

**Shape:** single

### Recommendation

Remove the *Five-site list co-edit obligation* sentence from `ceiling-invariants-and-audit.md` and relocate the cross-page lock-step co-edit obligation into the governance corpus alongside GOV-30 (`req-id-prefix-table-active-b.md#gov-30`), which already owns aggregator/owner-page lock-step co-edits — or into the dedicated ceiling-set-change-procedure governance page that the sibling cluster findings propose, if that page lands. Express it there in GOV-30's form: a ceiling-#4 per-boundary table row add/remove/split MUST co-edit, in the same commit, the CIO-3 five-site enumeration and the PIC-1 per-site reachable-mask-domain table.

If a runtime reader benefits from knowing the table is the canonical AJV-enforcement-point enumeration, that single declarative fact (without the spec-editor MUST) may stay; the co-edit MUST itself moves to governance. Watch that the relocated rule names all three co-edit targets exactly (ceiling-4 table, CIO-3 enumeration, PIC-1 reachable-mask-domain table) so the cross-page lock-step is not silently narrowed during the move.

## Consequence

A runtime implementer reading the ceiling spec encounters a MUST that is actually a spec-editor co-edit instruction, adding reading overhead and a false implementer-facing obligation, while the cross-page lock-step rule lives apart from GOV-30, the canonical co-edit home. No runtime behaviour diverges; the defect is pure organisation.
