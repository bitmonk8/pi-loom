## Finding

BNDR-4 (integer rendering) and BNDR-5 (number rendering), together with the numeric rows of the BNDR-6 reference-rendering table, are defined in `docs/spec_topics/binder/defaulting-system-note-echo.md` under *Echo policy › Format rules* — a user-facing section whose declared purpose is rendering the per-call binding-echo line.

However, those same rules are the authoritative numeric-serialisation algorithm for the **canonical schema hash**. `docs/spec_topics/schema-subset.md` (Canonical schema hash § step 2) pins: *"numeric literals embedded in the lowered fragment … are serialised by the `integer` / `number` rendering algorithm pinned in [Binder — Echo policy] *Format rules* (its [`integer`](#bndr-4) and [`number`](#bndr-5) value-rendering rules together with the [normative reference-renderings table](#bndr-6) …)."* The canonical schema hash is a pinned on-disk/on-wire contract: its 16-hex output (the **schema slug**) names `__inline_<slug>`, `__loom_respond_<slug>`, `__loom_callee_<slug>__<name>`, and `__loom_bind_<slug>`, and keys three slug-keyed caches.

The result is an ownership inversion: a central infrastructure contract sources its normative numeric serialisation from a section architecturally dedicated to UX echo display. An implementer or auditor reading `schema-subset.md` to build the hash must cross-navigate into the binder echo-formatter section to find the algorithm. The plan already recognises the rule as shared — leaf `V2d` ("Canonical integer/number renderer") factors it as "the canonical number-rendering pure function shared by the schema canonical-hash recipe (`V5f`) and the binder argument echo (`V11h`)" — so the spec-prose placement lags the design intent the plan already encodes.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** c5bb5e1 — pi-loom spec: resolve "Canonical schema hash numeric serialization underspecified" (2026-06-03, Thomas Andersen)
**History:** The numeric-rendering rules pre-existed as binder echo-formatter concerns. Commit c5bb5e1 created the ownership inversion by adding the Canonical schema hash step-2 clause that points the hash recipe at the binder *Echo policy › Format rules* `integer`/`number` rendering as its authoritative numeric-serialisation algorithm; before that commit no central contract depended on those rules and there was no inversion. A later edit (111e533, 2026-06-07) refined the cross-reference (per-anchor links, scope carve-out) but did not change the placement.

## Solution Space

**Shape:** multiple

### Option A — Relocate the numeric-rendering rules to the schema infrastructure

- **Approach:** Move the normative numeric-rendering rules (BNDR-4, BNDR-5, and the numeric rows of the BNDR-6 reference table) into `schema-subset.md`'s Canonical schema hash section; have the binder echo-policy *Format rules* cite that canonical home.
- **Spec edits:** Lift BNDR-4/BNDR-5 and the numeric BNDR-6 rows into a `schema-subset.md` subsection; replace them in `defaulting-system-note-echo.md` with a citation; repoint `V2d`'s Spec field and any inbound cross-links.
- **Pros:** Puts the rule with its most-central consumer (the on-wire hash contract); removes the inversion outright; aligns the spec home with `V2d`'s shared-renderer framing.
- **Cons:** Fragments the binder *Format rules* cluster (string/integer/number/boolean/null/array/object defined together) and the BNDR-6 reference table (mixed-type rows); migrates BNDR REQ-IDs out of the binder topic.
- **Risks:** Splitting BNDR-6's numeric rows from the rest of the table risks orphaned or duplicated sub-letter anchors and broken citations — `V11h` asserts the whole 6a–6x table.

### Option B — Extract numeric rendering to a neutral shared home

- **Approach:** Move BNDR-4/BNDR-5 and the numeric reference rows into a neutral value-rendering location both consumers cite — e.g. `runtime-value-model.md` (already the JS-host scalar-model owner) or a new dedicated subsection.
- **Spec edits:** Define the numeric-rendering rules once in the neutral home; have both `schema-subset.md` and the binder echo formatter cite it; repoint `V2d` and cross-links.
- **Pros:** Neither consumer "owns" the shared rule; symmetric for the hash recipe and the echo formatter; most directly matches `V2d`'s "shared pure function" framing.
- **Cons:** Introduces a third file into the citation graph; still fragments the binder *Format rules* cluster; more cross-links to maintain.
- **Risks:** A new owning section needs its own REQ-ID/anchor-scheme decision; anchor churn across three files.

### Option C — Keep the rules in the Format-rules cluster, reframe it as the shared canonical home

- **Approach:** Leave BNDR-4/5/6 in `defaulting-system-note-echo.md` but re-label the *Format rules* block as the canonical value-serialisation authority shared by the echo formatter and the canonical-hash recipe (a heading/intro-sentence edit), and add a one-line scope statement on the `schema-subset.md` side noting the hash recipe consumes the rule from there.
- **Spec edits:** Reframe the *Format rules* heading/intro in `defaulting-system-note-echo.md` to declare shared ownership; add a scope-statement sentence to `schema-subset.md`'s Canonical schema hash section. The existing cross-reference already carries the exact anchors and bounded scope.
- **Pros:** Smallest, lowest-risk edit; keeps the *Format rules* cluster and the BNDR-6 table intact; no REQ-ID migration; no broken citations; dissolves the perceived inversion by naming the shared ownership.
- **Cons:** The numeric rules still physically live under a binder/UX page title, so a reader navigating purely by file name may still find the location surprising.
- **Risks:** Minimal — purely additive prose; no anchor or REQ-ID churn.

### Recommendation

Adopt **Option C**. The plan already treats the numeric renderer as a shared pure function (`V2d`) consumed by both the hash recipe (`V5f`) and the echo (`V11h`); the defect is one of framing/ownership, not a missing or wrong rule, and the existing cross-reference is already precise and scope-bounded. Reframing the *Format rules* block as the canonical shared value-serialisation home and adding a scope note on the `schema-subset.md` side removes the inversion at the lowest risk, without fragmenting the BNDR-6 reference table that `V11h` asserts wholesale. Edge case: ensure the schema-subset scope note and the relabelled binder heading agree on exactly which rules are shared (BNDR-4, BNDR-5, and the numeric BNDR-6 rows) and that the borrowed-scope carve-out (the 120-code-point line cap and `(default)` suffix do **not** apply to hashing) remains intact.

## Consequence

**Severity:** advisory

A central infrastructure contract (the canonical schema hash) sources its normative numeric serialisation from a UX echo-formatter section, so every hash implementer must navigate out of the schema infrastructure into the binder display section to find the algorithm. No behavioural divergence results — the cross-reference carries exact anchors, worked examples (`1e21` → `1000000000000000000000`, `42.0` → `42`, `-0` → `0`), and an explicit carve-out of what is *not* borrowed — so the gap is one of discoverability and ownership clarity, not correctness.
