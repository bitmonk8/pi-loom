## Problem

The Grammar Appendix (`grammar.md`) declares its scope as surface-syntax forms no single topic page owns and the literal sublanguage. Its `## `array<T>` literal type-sink rule` section breaks that contract: it specifies expression-context element-type inference for array literals — the exhaustive type-sink set, the `for`-iterand carve-out, and the `loom/parse/array-no-common-type` diagnostic — none of which is surface syntax or literal-sublanguage admission. That subject is already normatively owned by `expressions.md` under `## Array construction`, which states the same `[]`/`[a, b, c]` inference and the same diagnostic. Two normative homes for one inference rule is a drift hazard: a later edit to one silently diverges from the other.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** f841b12 — spec: resolve Cluster 2 (literal sublanguage + grammar appendix) (2026-05-04, Thomas Andersen)
**History:** `expressions.md` "Array construction" already owned the element-type-inference rules and `loom/parse/array-no-common-type` once the spec was split into topic files (fecb504, 2026-05-04). Later that same day f841b12 authored the Grammar Appendix's literal sublanguage and added a second, overlapping `## `array<T>` literal type-sink rule` section — the exhaustive sink enumeration plus the `for`-iterand carve-out — into `grammar.md` rather than into the existing `expressions.md` home, introducing the misplacement and duplication. Subsequent edits (e.g. heading promotion in c1e5d64) only reshaped surrounding text and did not relocate the rule.

## Solution approach

Move the exhaustive type-sink enumeration, the `for`-iterand carve-out, and the `loom/parse/array-no-common-type` diagnostic out of grammar.md's `## `array<T>` literal type-sink rule` section and consolidate them into expressions.md's `## Array construction`, where the same element-type inference and diagnostic already live. Reduce the grammar.md section to a forward-link to `[Expression Sublanguage — Array construction](./expressions.md#array-construction)`, noting only that in literal positions an array literal's element type is supplied by the external sink (the `params:` declared type or the registered Pi-tool input schema). Remove the duplicate diagnostic definition from grammar.md so `loom/parse/array-no-common-type` is defined at a single site.
