## Problem

The `loom/load/missing-source` row in `code-registry-load.md` carries a *Sev* of `E/W`, which per the column-legend means the severity is resolved per-source by the *Missing path* column of the DISC-2 failure-modes table in `discovery/discovery-sources.md`. That column resolves only to `silent` (conventional locations) or `error` (explicit references) — it never resolves to `warning`. The `W` branch is therefore unreachable, and the row's Trigger prose has to spend a "warning never" clause explaining why. Sibling rows `unreadable-source` and `wrong-type-source` are genuinely `E/W`; only `missing-source` is mis-tagged.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `5b772b6` ("pi-loom spec: resolve 'Discovery source failure-mode policy is unspecified'", 2026-05-04)
**History:** `git log --all -S 'warning never'` and `git log --all -S 'missing-source'` both identify `5b772b6` as the commit that added the `loom/load/missing-source` row, with the `E/W` Sev cell and the "warning never — conventional locations … emit no diagnostic when missing" clause introduced together in the same hunk (which also added the failure-modes table to `spec_topics/discovery.md`). The row was given `E/W` for visual consistency with its two table neighbours (`unreadable-source` and `wrong-type-source`, which legitimately are `E/W`); the author noted the dead branch inline with the "warning never" phrase rather than tightening the Sev cell. The subsequent commit `f5e89f4` (2026-06-04) reflowed the spec across the `spec_topics/diagnostics/` shard files but preserved the row verbatim. No intervening commit altered the Sev cell, the Trigger prose, or the failure-modes table's *Missing path* column. The defect therefore has a clean single-commit origin in `5b772b6` rather than emerging from a multi-commit interaction.

## Solution approach

Narrow the `loom/load/missing-source` row's *Sev* cell from `E/W` to `E` in `code-registry-load.md`. Delete the now-redundant "warning never — conventional locations …" clause from that row's *Trigger* cell; the remaining prose already pins the firing surfaces (the explicit references). Keep the row's *Spec rule* link to DISC-2 (*Discovery — Failure modes*) — it remains the upstream authority for which conventional locations stay silent.

## Solution constraints

- Out of scope: the `loom/load/unreadable-source` and `loom/load/wrong-type-source` rows — both have reachable `E` and `W` branches and stay `E/W`.
- Out of scope: the column-legend prose in `code-registry-parse.md` and the GOV-15 *Loads-cleanly predicate* paragraph in `governance/source-language-stability.md` — both stay correct once `missing-source` leaves the `E/W` set.
