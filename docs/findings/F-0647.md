## Finding

The `loom/load/missing-source` row in `code-registry-load.md` (line 38) declares its *Sev* as `E/W`, which per the column-legend in `code-registry-parse.md` means "severity is not fixed at the code level but resolved per diagnostic by that row's own Trigger prose … per-source by the table in *Discovery — Failure modes*." But the per-source failure-modes table in `discovery/discovery-sources.md` §DISC-2 has only two outcomes in the *Missing path* column: `silent` (for the three conventional locations — global, project, package `looms/`) and `error` (for the three explicit references — `pi.looms`, settings `loomPaths`, `--loom`). No source produces a `warning` on a missing path; the Trigger acknowledges this directly with the phrase "warning never — conventional locations … emit no diagnostic when missing."

The result is a dead branch. The Sev cell promises a per-source `W` resolution, the failure-modes table the cell defers to never resolves to `W`, and the Trigger has to spend a clause explaining why. A reader following the legend's "look at the Trigger prose" instruction is sent on a goose chase; an implementer asserting test vectors against the registry has no W vector to assert for this code; and `source-language-stability.md`'s GOV-15 "loads-cleanly" predicate (which calls out `E/W` codes specifically as needing per-source resolution) inherits the same dead branch.

This row is anomalous within the discovery-source cluster. `loom/load/unreadable-source` and `loom/load/wrong-type-source` are genuinely `E/W` — both columns of the failure-modes table reach both severities. Only `missing-source` is mis-tagged.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `5b772b6` ("pi-loom spec: resolve 'Discovery source failure-mode policy is unspecified'", 2026-05-04)
**History:** `git log --all -S 'warning never'` and `git log --all -S 'missing-source'` both identify `5b772b6` as the commit that added the `loom/load/missing-source` row, with the `E/W` Sev cell and the "warning never — conventional locations … emit no diagnostic when missing" clause introduced together in the same hunk (which also added the failure-modes table to `spec_topics/discovery.md`). The row was given `E/W` for visual consistency with its two table neighbours (`unreadable-source` and `wrong-type-source`, which legitimately are `E/W`); the author noted the dead branch inline with the "warning never" phrase rather than tightening the Sev cell. The subsequent commit `f5e89f4` (2026-06-04) reflowed the spec across the `spec_topics/diagnostics/` shard files but preserved the row verbatim. No intervening commit altered the Sev cell, the Trigger prose, or the failure-modes table's *Missing path* column. The defect therefore has a clean single-commit origin in `5b772b6` rather than emerging from a multi-commit interaction.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/diagnostics/code-registry-load.md` line 38:

1. Change the *Sev* cell from `E/W` to `E`.
2. In the *Trigger* cell, drop the ", warning never — conventional locations (global, project, package `looms/`) emit no diagnostic when missing" clause. The remaining prose ("A discovery source's path does not exist. Severity is per the failure-modes table in *Discovery — Failure modes*: error for explicit references (`pi.looms` entries, settings `loomPaths` entries, `--loom` flags)") already pins the firing surfaces; the dropped clause was a workaround for the dead branch and is no longer needed.
3. Keep the *Spec rule* link to `Discovery — Failure modes` — DISC-2's "silent for conventional locations" rule is still the upstream authority for *which* conventional locations stay silent, and the link is the cross-reference for that.

Do not touch the `unreadable-source` or `wrong-type-source` rows: both genuinely have reachable `E` and `W` branches in their respective failure-modes columns. Do not touch the column-legend prose in `code-registry-parse.md`: the legend still applies to the two `E/W` discovery-source codes that remain. Do not touch the `source-language-stability.md` GOV-15 *Loads-cleanly predicate* paragraph: its "discovery-source codes whose Severity cell carries `E/W`" phrasing is set-based and stays correct once `missing-source` leaves that set.

Edge case for the implementer: the registry-closing gate test vectors for `missing-source` must enumerate exactly the three explicit-reference surfaces (`pi.looms`, settings `loomPaths`, `--loom`), all asserted at `E`. There is no fourth conventional-location vector to assert (those paths produce no diagnostic at all, which is itself a DISC-2 assertion belonging to `V10a-T`, not a `V7b-T` registry-row vector).

## Consequence

**Severity:** correctness

DIAG-2's "registry code with no asserting test fails the gate" arm becomes either toothless (if the gate is silently tolerant of unreachable severity branches) or actively wrong (if it demands a non-existent `W` test vector for this row). A second implementer reading the legend and the failure-modes table will reach different conclusions about whether they need to plumb a `W` code path for `missing-source`, and the GOV-15 "loads-cleanly" predicate inherits the ambiguity.
