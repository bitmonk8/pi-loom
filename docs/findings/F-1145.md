## Problem

In prompt mode the loom drives the user's existing Pi session via `pi.sendUserMessage`; which provider/model that turn routes through is a property of the user session, not the loom, and no pinned `ExtensionAPI` / `ExtensionCommandContext` member lets the loom override it for the duration of a query. Yet `conversation-drive.md` §`prompt-mode-error-detection` populates the synthesised `TransportError.provider` — for both the synchronous-`pi.sendUserMessage`-throw secondary mapping and the post-`waitForIdle()` `stopReason: "error"` primary mapping — from the loom's resolved `model:`, and `errors-and-results/queryerror-variants.md` §`provider` derivation restates it as the `Model<Api>.api` value of "the model the failing query routed through". When the user-session model differs from the loom's `model:` — or when the loom carries no `model:` and `ctx.model` is `undefined` — `provider` is either misattributed or undefined-by-construction, and the **Runtime event channel** always-log emission for `kind: "transport"` then carries that wrong or undefined value into operator-visible telemetry as the canonical "which provider failed" signal.

## Issue introduction

**Verdict:** present-since-derivation
**Introducing commits:**
- `2d2fc8e` (2026-05-06) — *pi-loom spec: resolve "Pi SDK capability calls have no failure contract"* — first introduced the "`provider` field is populated from the loom's resolved `model:`" clause on the prompt-mode bullet of `spec_topics/pi-integration-contract.md`, justifying it on the **Runtime event channel** always-log requirement.

**History:** Before `2d2fc8e` the prompt-mode bullet had no transport-error mapping clause at all. `2d2fc8e` added the `<resolved-model-provider>` placeholder and the "populated from the loom's resolved `model:`" derivation in a single edit; the assumption that the user-session turn routes through that model was never separately stated or audited. Subsequent edits refined the surrounding surface but left the derivation in place: `8c6cfec` (2026-05-07) clarified that `pi.sendUserMessage` returns `void` and re-shaped the primary/secondary surfaces while preserving the derivation verbatim; `d37c563` (2026-06-05) moved the primary surface from `AgentSession.errorMessage` to the trailing-`assistant`-message `stopReason: "error"` probe (still propagating the same `provider` derivation); `43bef30` (2026-06-05) hardened the `provider` value space by adding the cross-reference to `Model<Api>.api`, importing the same misattribution into `errors-and-results/queryerror-variants.md` §*`provider` derivation*. None of those edits revisited the underlying user-session-vs-loom-model premise.

## Solution approach

Derive the primary-surface `provider` from the same trailing `assistant` message the `stopReason: "error"` probe already reads at `#prompt-mode-error-detection`, using whichever `AssistantMessage` field (declared in `@earendil-works/pi-ai`'s `dist/types.d.ts`) carries the `Api`-shaped value, rather than from the loom's resolved `model:`. For the synchronous-throw secondary surface, which has no `assistant` message to read, pin a derivation from `ctx.model` with a defined sentinel for the `ctx.model: undefined` case. Decide whether prompt mode needs a model-unresolved gate counterpart to subagent mode's `loom/runtime/subagent-model-unresolved`, and state the outcome. Reconcile `errors-and-results/queryerror-variants.md` §`provider` derivation with the two-branch rule and drop the "the model the failing query routed through" phrasing.

## Solution constraints

- Out of scope: the §*Provider compatibility for typed queries* unsupported-provider synthesised `TransportError`, whose `provider` is genuinely the loom's resolved-`model:` `Api` value (synthesised at load/dispatch time before any user turn) — leave that derivation unchanged.
