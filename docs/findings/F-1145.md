## Finding

In prompt mode the loom drives the user's existing Pi `AgentSession` by calling `pi.sendUserMessage(...)`. Which provider/model Pi actually routes that user turn through is a property of *the user session* â€” set and changed by the user through Pi, not by the loom â€” and no loom-touched member of the pinned `ExtensionAPI` / `ExtensionCommandContext` surface (per `host-interfaces-core.md`) lets the loom override or pin the user-session model for the duration of a query. `pi.sendUserMessage` carries no `model` option; `ExtensionCommandContext.model` is read-only; `newSession` / `fork` / `switchSession` are explicitly in the **MUST NOT touch** set. The loom's frontmatter `model:` â€” when present, resolved at load time; when absent, "inherited" by snapshotting `ctx.model` at invocation time â€” is therefore a value the runtime *carries* but does not impose on the driven turn.

`conversation-drive.md` Â§*Conversation drive â€” prompt mode* nevertheless pins, for both the synchronous-`pi.sendUserMessage`-throw secondary mapping and the post-`waitForIdle()` `stopReason: "error"` primary mapping, that `QueryError.provider` "is populated from the loom's resolved `model:` regardless of which surface â€¦ detected the failure", and `errors-and-results/queryerror-variants.md` Â§*`provider` derivation* restates this as "the resolved `Model<Api>.api` value of the model the failing query routed through". The two clauses presuppose the user-session model is the loom's resolved `model:`. When the user has selected a different session model â€” and especially when the loom carries no `model:` and `ctx.model` happens to be `undefined` (a state the subagent path explicitly handles via the **Pre-spawn model guard** + `loom/runtime/subagent-model-unresolved` but prompt mode does not) â€” the populated `provider` is either misattributed (a different provider than the one whose transport actually failed) or undefined-by-construction. The downstream **Runtime event channel** always-log emission for `kind: "transport"` then carries a wrong or undefined `provider` string into operator-visible telemetry.

This propagates into the plan: `V9n` (and its test leaf `V9n-T`) cite *"the `provider` value sourced from the loom's resolved `model:` `Model<Api>.api` value"* as the contract a test must witness, so the test asserts the same misattribution.

## Solution Space

**Shape:** single

### Recommendation

Stop treating the loom's resolved `model:` as the source of truth for prompt-mode `TransportError.provider`, and derive `provider` from the *failing turn* the spec already mandates loom probe. Concretely:

1. **Primary surface (post-`waitForIdle()` `stopReason: "error"` probe).** Derive `provider` from the same trailing `assistant` message the `stopReason: "error"` / `errorMessage` read is taken from. The driven turn's `AssistantMessage` already carries provider attribution Pi uses for its own rendering; pin the field that supplies it (whichever member of the externally-owned `AssistantMessage` declaration in `@earendil-works/pi-ai`'s `dist/types.d.ts` carries the `Api`-shaped value â€” most likely a model/provider field already populated for the same reason the rest of the message is) and state that the `provider` field of the synthesised `TransportError` is read from that field, with the same `Api`-shaped value space and value-pin treatment `queryerror-variants.md` Â§*`provider` derivation* already gives it.

2. **Secondary surface (synchronous-throw branch from `pi.sendUserMessage`).** The call site has no `AssistantMessage` to read from, so pin a fallback rule explicitly: derive `provider` from `ctx.model` (the user-session model snapshot at the call site), falling back â€” when `ctx.model` is `undefined` â€” to a defined value (either the literal string `"unknown"` registered alongside the existing `Api`-union value space, or `null` if the `TransportError` schema is widened to admit `provider: string | null` on this branch only). State the chosen sentinel and the schema impact.

3. **Reconciliation.** Update `errors-and-results/queryerror-variants.md` Â§*`provider` derivation* to spell out the two branches in lock-step with `conversation-drive.md` and drop the "the model the failing query routed through" phrasing, which currently begs the question â€” replace with the explicit "read from the trailing `assistant` message's provider field (primary surface) / `ctx.model` (secondary surface, with sentinel `<â€¦>` when `undefined`)".

4. **Prompt-mode model-unresolved gate.** Decide whether prompt mode needs a counterpart to the subagent `loom/runtime/subagent-model-unresolved` guard. If the answer is *yes* (the synchronous-throw branch needs a non-`undefined` `provider` and there is no telemetry-acceptable sentinel), register a new prompt-mode `loom/runtime/*` code under `diagnostics/code-registry-runtime.md` and have prompt-mode invocation refuse to start when `model:` is absent and `ctx.model` is `undefined`. If *no* (the sentinel is acceptable), state explicitly that prompt mode tolerates the case and the binder / typed-query paths handle their own model-unresolved arms.

5. **Plan adjustments.** Repoint `V9n` / `V9n-T`'s acceptance text from "the loom's resolved `model:` `Model<Api>.api` value" to the trailing-turn-derived value (primary) plus the secondary-branch sentinel rule. The test fixture for the synchronous-throw scenario must drive the case where the user-session model differs from the loom's frontmatter `model:` and assert the trailing-turn value wins â€” that is the exact divergence the current contract masks.

Edge cases the implementer must watch: (a) a `stopReason: "error"` turn whose trailing `assistant` message is itself absent / malformed â€” fall through to the synchronous-throw branch's sentinel rule rather than reading the loom's `model:`; (b) a user session whose model was changed by the user mid-loom between two queries in the same invocation â€” each query's `provider` is derived afresh from its own driven turn, which is the correct behaviour and does not require any per-invocation pinning; (c) typed-query free-phase failures route through `pi.sendUserMessage` and therefore use the same primary/secondary derivation; (d) the typed-query *unsupported-provider* synthesised `TransportError` in Â§*Provider compatibility for typed queries* is unaffected â€” its `provider` is genuinely the loom's resolved-`model:` `Api` value, since the runtime synthesises that error at load/dispatch time before any user turn â€” keep that derivation as-is and note it as the one site where loom-resolved `model:` is the right source.

## Consequence

**Severity:** correctness

Two conforming implementations diverge on the `provider` field of every prompt-mode `TransportError`: one populates it from the loom's resolved `model:` per the spec letter (and produces wrong attribution whenever the user-session model differs, plus an undefined / `null` / `"undefined"` string when both `model:` is absent and `ctx.model` is `undefined`); the other reads the actual provider off the failing turn (e.g. from the trailing `assistant` message or session state) and produces a different value for the same failure. The wrong value flows into operator telemetry as the canonical "which provider failed" signal, defeating its purpose.
