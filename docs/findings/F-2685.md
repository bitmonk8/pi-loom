## Problem

The *Baseline acquisition* paragraph (`#bump-baseline-acquisition`) in `version-bump-step2.md` mandates, as a `MUST`, that the contributor run `npm pack @earendil-works/pi-coding-agent@<pinned-version>` and extract the tarball to obtain the canonical "from" side for every `dist/*.js` diff on the page, and explicitly forecloses `npm install --prefix`. The step assumes the pinned version stays retrievable from the registry — it states no precondition and no fallback source, so a yanked version, registry outage, or air-gapped CI makes `npm pack` fail and silently strips the baseline from every diff/JS-body audit that consumes it. The requirement is also over-prescribed: the only observable obligation is materialising the loom-1.0-pinned `dist/` tree without pulling its transitive dependencies, which any equivalent mechanism satisfies. Separately, the preamble enumerates only consumers (e), (r), and (aj), but item (am) (`#bump-checklist-tools-allowlist-suppression`) also reads against the same extracted tree.

The fix closed the under-specification gap by mandating `npm pack @earendil-works/pi-coding-agent@<pinned-version>` as a `MUST` and explicitly foreclosing `npm install --prefix` — over-correcting from "source unspecified" to "single command pinned, retrievability assumed," which is the present defect. The retrievability assumption and the over-prescription therefore entered the corpus together at `ccc6e38`; they were not present at the section's inception.

## Solution approach

Rewrite the `#bump-baseline-acquisition` paragraph to pin the observable outcome and its precondition rather than the single command: state that the loom-1.0-pinned `dist/*.js` tree must be obtainable, and name a fallback baseline source for when registry retrieval is unavailable — the pinned tree already installed under `node_modules/@earendil-works/pi-coding-agent/dist/`, captured into the scratch directory before step 1 overwrites it. Restate the requirement behaviourally (materialise the pinned `dist/` tree without its transitive dependency tree as the "from" side) and demote `npm pack @earendil-works/pi-coding-agent@<pinned-version>` and its `npm pack`-not-`npm install --prefix` rationale to a non-normative worked example. Correct the preamble's consumer enumeration to list all four dependent items — (e), (r), (aj), and (am).

## Solution constraints

- Keep the ephemerality and MUST-NOT-commit rules on the scratch directory and extracted tarball unchanged.
- The `node_modules`-copy fallback must stay gated to pre-step-1 execution; post-install the explicit `@<pinned-version>` registry path remains the only correct source.
