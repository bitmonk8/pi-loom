## Finding

The `session_shutdown` teardown sequence in `session-shutdown-semantics.md` defines sub-step 4 as: "Call `discoveryWatcher.close()` (the chokidar watcher established in step 5 above) and `settingsWatcher.close()` … ; cancel any pending debounce timer handles via `Clock.clearTimeout`." The call is *not* awaited. The **Per-step isolation** paragraph then wraps sub-step 4 in its own synchronous `try`/`catch`: "a throw from any of them MUST be caught, MUST emit exactly one `loom/host/session-shutdown-teardown-step-failed` … and MUST NOT prevent the remaining sub-steps from running," applied "independently to each of `discoveryWatcher.close()`, `settingsWatcher.close()`, and the debounce-handle `Clock.clearTimeout(...)` call (three potential emissions, one per failing call)." This whole machinery only works if `close()` surfaces its failure as a *synchronous* throw.

chokidar's `FSWatcher.close()` returns `Promise<void>` (chokidar v3 and v4), and the spec deliberately leaves the chokidar version unpinned (`host-prerequisites.md` — "version-unpinned `semver` / `chokidar` choices"). An async close rejection therefore does not surface synchronously: it escapes the synchronous `try`/`catch`, becomes an unhandled promise rejection on the about-to-be-invalidated runtime, the `loom/host/session-shutdown-teardown-step-failed` diagnostic is never emitted for that call site, and the no-throw-escapes-the-handler invariant the teardown contract relies on is silently defeated. The spec already demonstrates it knows how to await teardown work — sub-step 3 uses `await Promise.allSettled(...)` — but sub-step 4 has no await and no statement of close() semantics.

The gap also undercuts a downstream guarantee: `drain-state-contract.md` (the *idempotent* clause, *Read-failure fallback* clause, and *all-three-throw corner case*) repeatedly relies on "the per-call `try`/`catch` … routes any double-close throw through one additional `loom/host/session-shutdown-teardown-step-failed` row per failing call site." If chokidar's double-close rejects asynchronously rather than throwing synchronously, that routing never fires, and the re-entry / read-failure / all-three-throw acceptance branches that lean on it are unsound.

Compounding this, sub-step 4 names chokidar's `close()` directly, bypassing the normative `FileWatcher` seam (PIC-14 in `host-interfaces-services.md`), whose teardown surface is the *synchronous* `Unsubscribe = () => void` ("The returned `Unsubscribe` tears down the underlying watcher (chokidar's `close()` in production); calling it twice is a no-op"). The contract has a synchronous teardown abstraction available, but the teardown sequence reaches past it to the async production call.

The spec must state whether sub-step 4's watcher teardown is synchronous or asynchronous, and — if asynchronous — how an async close rejection routes to `loom/host/session-shutdown-teardown-step-failed` (or to some other defined surface) rather than escaping as an unhandled rejection.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** `5e8b499` (2026-05-06, "pi-loom spec: resolve \"Hot-reload teardown contract for the loom extension instance is unspecified\""); `6ce416d` (2026-05-07, "pi-loom spec: resolve \"Teardown sub-steps 1, 4, and 5 lack a per-step isolation rule\"")
**History:** `git log -S` over the spec corpus shows the defect arose from the interaction of two independently-landed edits, not a single faulty commit. `5e8b499` first introduced sub-step 4 ("Close watchers. Call `discoveryWatcher.close()` … and `settingsWatcher.close()`") with no statement of whether the calls are awaited or whether `close()` is synchronous. `6ce416d` then added the **Per-step isolation** rule wrapping each sub-step in a synchronous `try`/`catch` that routes throws to `loom/host/session-shutdown-teardown-step-failed` — at which point the latent synchronous-throw assumption became load-bearing: a synchronous catch over an un-awaited asynchronous `close()` can only ever capture a synchronous throw. Neither commit alone is the sole cause; the unsound assumption emerges only from their combination. Both blocks were later relocated verbatim into `session-shutdown-semantics.md` by `c3de47a` (F-1602), which preserved the defect without re-introducing it.

## Solution Space

**Shape:** multiple

### Option A — Route sub-step 4 through the synchronous `FileWatcher` seam and accept async-rejection routing as adapter-owned

**Approach:** Pin sub-step 4's watcher teardown as the synchronous `FileWatcher` `Unsubscribe = () => void` seam (PIC-14) rather than naming chokidar's async `close()`. State that the seam's teardown is synchronous and void-returning; the production chokidar adapter attaches a rejection handler to the underlying async `close()` and routes any late rejection through the watcher's existing `error` / PIC-55 transient-toast surface, never to `teardown-step-failed`. The Per-step isolation synchronous `try`/`catch` then only ever needs to catch synchronous throws out of the synchronous `Unsubscribe`, which is sound.

**Spec edits:** In `session-shutdown-semantics.md` sub-step 4, replace the direct `discoveryWatcher.close()` / `settingsWatcher.close()` calls with the synchronous seam teardown and state that it returns synchronously; in **Per-step isolation**, keep the `details.call` labels but note they wrap a synchronous teardown. Add a sentence (or cross-reference PIC-14 / PIC-55) pinning that the production adapter must not let an async close rejection become an unhandled rejection and that such a rejection routes to the watcher `error` surface, not `teardown-step-failed`. In `drain-state-contract.md`, reword the "double-close throw" routing to reference the synchronous seam teardown rather than an async close throw.

**Pros:** Aligns with the existing normative seam (`Unsubscribe = () => void`) and the corpus convention of deferring chokidar to the implementation layer; keeps teardown sub-steps 1/4/5 synchronous, so the Per-step isolation contract stays uniform; resolves the layering defect (naming chokidar directly) at the same time.

**Cons:** A genuine close failure no longer surfaces through `teardown-step-failed`; it is demoted to the transient-toast surface, which changes the diagnostic contract for watcher-close failures.

**Risks:** If the adapter forgets the rejection handler, the unhandled-rejection problem returns; the spec must make the handler obligation explicit.

### Option B — Await sub-step 4's `close()` calls and route async rejections to `teardown-step-failed`

**Approach:** Make sub-step 4 await the watcher `close()` calls (mirroring sub-step 3's `await Promise.allSettled`), wrap each await in its own `try`/`catch`, and specify that an async close rejection routes to `loom/host/session-shutdown-teardown-step-failed` on the same `(code, details.step, details.call)` footing as a synchronous throw.

**Spec edits:** In `session-shutdown-semantics.md` sub-step 4, state the calls are awaited; in **Per-step isolation**, reword the sub-step-4 clause so the per-call `try`/`catch` wraps an `await` and an async rejection produces the same `teardown-step-failed` row a synchronous throw would. Confirm the double-close routing in `drain-state-contract.md` is satisfied by an awaited rejection.

**Pros:** Preserves `teardown-step-failed` as the single observable surface for any watcher-close failure (sync or async); keeps the double-close routing claim literally true; consistent with sub-step 3's existing await pattern.

**Cons:** Introduces an await into the otherwise-synchronous teardown sub-steps 1/4/5, so the bounded-await discipline (which today is scoped to sub-step 3 and the `SHUTDOWN_AWAIT_CAP_MS` cap) now has a second, uncapped await; still names chokidar's async `close()` directly, leaving the PIC-14 seam-bypass layering defect unaddressed.

**Risks:** An unbounded await on a hung `close()` could stall teardown past the `SHUTDOWN_AWAIT_CAP_MS` budget the rest of the handler respects, unless a cap is also specified.

### Recommendation

Adopt **Option A**. PIC-14 already pins the normative watcher teardown as the synchronous `Unsubscribe = () => void` seam, and sub-step 4 reaching past it to chokidar's async `close()` is the root of the unsound synchronous-throw assumption; routing through the seam restores soundness to the Per-step isolation `try`/`catch` without threading a second uncapped await through the synchronous teardown sub-steps. Edge case the implementer must watch: the production adapter MUST attach a rejection handler to the underlying chokidar `close()` so a late rejection cannot surface as an unhandled rejection on the invalidated runtime, and the spec must say where that rejection goes (the PIC-55 watcher `error` surface) since it no longer reaches `teardown-step-failed`.

## Consequence

**Severity:** correctness

Under any realistic chokidar (v3/v4 `close()` returns `Promise<void>`), an async close rejection escapes the synchronous Per-step isolation `try`/`catch`: it becomes an unhandled promise rejection on the invalidated runtime, the `teardown-step-failed` diagnostic is not emitted for the failing call site, and the double-close routing the drain-state contract's re-entry / read-failure / all-three-throw branches depend on never fires. Two reasonable implementers diverge — one treats `close()` as the spec's literal synchronous call, the other awaits it — producing different diagnostic-emission counts and different unhandled-rejection behaviour for the same teardown failure.
