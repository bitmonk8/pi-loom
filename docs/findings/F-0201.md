## Problem

The `bind_context: session` truncation walk on `spec_topics/binder/binder-model-and-context.md` ¬ßSession-context truncation is a normative obligation with no numbered `PREFIX-N` REQ-ID and no `loom/...` registry code ‚Äî un-anchored GOV-22 residue that `conventions.md` *REQ-ID discipline* requires to be enumerated as a row in `coverage-matrix.md` under *Code-keyed obligation areas (no numbered REQ-IDs)* with a named closing leaf. That obligation is closed by `V11i`, but no such row exists, so `V11i` appears nowhere in the matrix. Because the walk's caps are phrased without a literal `MUST` token, the `H5a` recogniser cannot mechanically witness the gap; it surfaces only at the manual `GOV-15` residue sweep.

## Issue introduction

**Verdict:** introduced ‚Äî single commit
**Introducing commit:** `d0c127f` ‚Äî "pi-loom plan: resolve \"V11b bundles transcript renderer with truncation walk\"" (Thomas Andersen, 2026-06-16)
**History:** `git log --follow` shows `V11i-session-context-truncation.md` was created in exactly one commit, `d0c127f`, which split the truncation walk out of `V11b` into the new `V11i` leaf. `git show --stat d0c127f` confirms that commit touched `V11b`, `V11i`, `H5b`, `H7a`, `plan.md`, and `plan-review.md` ‚Äî and added `V11i` to `H5b`'s `Deps.` (`V11a`‚Äì`V11h` ‚Üí `V11a`‚Äì`V11i`) ‚Äî but did **not** touch `coverage-matrix.md`. `git log --follow -- coverage-matrix.md` does not list `d0c127f`. Before the split the truncation behaviour lived bundled inside `V11b` (already un-anchored / token-less and never given a dedicated matrix row); the split moved the obligation onto its own leaf without adding the corresponding matrix row, which is when `V11i` became a matrix-absent coverage-producing leaf.

## Solution approach

Add a row to the *Code-keyed obligation areas (no numbered REQ-IDs)* table in `docs/plan_topics/coverage-matrix.md` mapping the `binder/binder-model-and-context.md` ¬ßSession-context truncation walk obligation to closing leaf `V11i`, alongside the other `binder/...` rows.

## Solution constraints

- The spec page `binder/binder-model-and-context.md` is read-only for this fix; do not re-anchor the obligation as a new REQ-ID (a GOV-22 re-anchoring is a spec-side decision outside this plan rule).
- Out of scope: `H5b`'s `Deps.` (`V11i` is already present) and the `BNDR-7`/`BNDR-8`/`BNDR-9` ‚Üí `V11b` numbered rows (those cover the transcript rendering, not the walk).
