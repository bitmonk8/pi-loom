## Problem

"statically resolvable" / "static resolution" is a multi-page load-bearing qualifier — canonical at `invocation.md#static-resolution` and reused in `functions.md`, `tool-calls.md`, frontmatter, and several parse-diagnostic rows. It meets the glossary's admission criterion (a coined term reused on more than one page) but has no glossary entry, so a reader cannot get the single-lookup definition the glossary promises.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `b4fbfac` ("pi-loom spec: resolve \"no central glossary\"", 2026-05-04) — the commit that created the glossary.
**History:** The spec corpus is git-tracked. `git log --diff-filter=A -- '*glossary.md'` shows the glossary was first created in `b4fbfac`; `git log -S 'final value' --` and `git log -S 'statically resolvable'` against the glossary file (both current and pre-move paths) return no commits, i.e. neither term has ever had a glossary entry. The terms themselves predate the glossary (`final value`/`return value` and `statically resolvable` were already in use across `invocation.md`/`functions.md`/`overview.md` when `b4fbfac` landed), so the omission has existed continuously since the glossary was introduced — it is not a regression from any later edit.

## Solution approach

Add a glossary entry for **statically resolvable** (covering the noun form *static resolution*) to the alphabetised entry list, with `See:` pointing at `invocation.md#static-resolution`.
