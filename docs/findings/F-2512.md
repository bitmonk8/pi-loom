## Problem

The Indexed-access bullet in `expressions.md` defines the runtime out-of-bounds panic as an array index outside `0..arr.length`. That `..` notation is a Rust-style half-open range, but the spec never declares `..` as a half-open `[start, end)` interval and loom is positioned as a TypeScript subset, not a Rust dialect. A reader without the Rust convention cannot tell whether `arr.length` is itself a valid index (inclusive upper bound) or the first out-of-bounds index (exclusive upper bound), so the off-by-one behaviour of `arr[arr.length]` and the `loom/runtime/index-out-of-bounds` conformance test are underspecified.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** d6f17ca — pi-loom spec: resolve "indexed-access static semantics and runtime disposition" (2026-06-07, Thomas Andersen)
**History:** The Indexed-access bullet originally stated only the parse-time `loom/parse/non-indexable-receiver` rule and carried no runtime out-of-bounds prose. Commit d6f17ca extended the bullet with the runtime disposition sentence, introducing the bare `0..arr.length` range notation for the panic boundary without declaring the interval's exclusivity; the ambiguity entered the corpus with that single commit.

## Solution approach

Rewrite the panic-boundary clause of the Indexed-access bullet so it names both ends explicitly instead of the bare `0..arr.length` form: a valid index lies in the half-open interval `[0, arr.length)`, and an index that is negative or `≥ arr.length` panics with `loom/runtime/index-out-of-bounds`. Apply the same wording to any other site that reuses the `0..arr.length` form.
