## Finding

`docs/spec_topics/future-considerations/model-changes-and-non-goals.md` ends with `## Open contradictions (blocking, not deferrals)`, whose body records a present-tense, pre-implementation gate against the session-only degraded-state branch (SM-3b / SM-4 / SM-5 / SM-6). Every surrounding signal advertises the file as the opposite kind of content: the directory is `future-considerations/`, the parent index (`docs/spec_topics/future-considerations.md`) explicitly introduces the bucket as *"Features deliberately deferred from loom 1.0"* and as no-seam scope dispositions, the filename pairs *model changes* with *non-goals*, and the file's only other H2s (`## Model-level changes (no loom 1.0 seam expected)` and `## loom 1.0 non-goals`) carry the same forward-looking framing. The block's own self-correcting subtitle — "blocking, not deferrals" — concedes the mismatch but does not repair it.

The section's content also differs structurally from the rest of the page. The non-goals are durable, project-wide scope dispositions; the open contradiction is a single transient state record whose entire purpose is to be resolved and deleted. Co-locating the two confuses readers about both: a reader who treats the file as the canonical non-goals registry has to learn that one section is a different kind of entity, and a reader following an SM-3b/4/5/6 cross-link arrives in a directory titled "things loom 1.0 will not do" when looking for the gate that blocks them.

The directly-observable damage is bounded — the implementation gate is also surfaced by the per-page blockquote in `session-only-degraded-state.md` and the per-clause sentences in `session-model-and-appendix.md`, and the plan's §Blocked obligations entry on `V9l` correctly blocks on `V18c` — so an implementer is unlikely to silently start building SM-4/5/6. The defect is one of taxonomy and discoverability rather than misdirection: the canonical spec home for the *open question itself* is filed under the wrong cabinet.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commit:** `b8f004f` (2026-06-19, *"pi-loom spec-plan: fix F-0719 — SM-4/5/6/3b degraded-state implementation-gate contingency markers"*)
**History:** Prior to `b8f004f`, `model-changes-and-non-goals.md` carried only the `## Model-level changes (no loom 1.0 seam expected)` and `## loom 1.0 non-goals` H2s, both consistent with the surrounding `future-considerations/` framing. `b8f004f` appended the new `## Open contradictions (blocking, not deferrals)` H2 in a single `+8`-line hunk on this file (the same commit also added the *Implementation gate* blockquote to `session-only-degraded-state.md` and per-clause *Implementation gate* sentences on SM-3b/4/5/6 in `session-model-and-appendix.md`). The directory and filename framing the new section conflicts with were not edited by `b8f004f`, and `docs/spec_topics/future-considerations.md`'s "Features deliberately deferred from loom 1.0" intro was already in place. The categorisation mismatch was therefore introduced by `b8f004f` choosing this file as the home for a fundamentally different kind of entry rather than by drift across multiple commits; the parenthetical subtitle "(blocking, not deferrals)" shipped in the same hunk, signalling the author already perceived the tension but resolved it with disclaimer prose rather than relocation.

## Solution Space

**Shape:** multiple

### Option A — Extract to a dedicated `open-questions.md` outside `future-considerations/`

**Approach.** Create a new top-level spec topic `docs/spec_topics/open-questions.md` (or `governance/open-questions.md` if open-question stewardship is governance-owned) titled e.g. `# Open questions (implementation blockers)`. Move the `## Open contradictions (blocking, not deferrals)` section out of `model-changes-and-non-goals.md` into the new file as its first section, preserving the section body and the *Extension-instance survival across session-only `session_shutdown`* bullet verbatim. Add a `docs/spec.md` TOC entry for the new page and update inbound references (notably the SM-3b/4/5/6 per-clause sentences and the top-of-page *Implementation gate* on `session-only-degraded-state.md`) to point at the new file. Remove the `(blocking, not deferrals)` disclaimer from the heading once relocated.

**Spec edits.** New file `docs/spec_topics/open-questions.md`; delete the trailing `---` + `## Open contradictions` block from `model-changes-and-non-goals.md`; TOC line in `docs/spec.md`; re-point the *Implementation gate* citations on `session-only-degraded-state.md`, `session-model-and-appendix.md` (SM-3b/4/5/6), `host-prerequisites.md` clause (a), and `version-bump-step2.md` item (a) to the new file.

**Pros.** Filename and directory match content; discoverability becomes positive ("where do I find open implementation blockers?" → `open-questions.md`); the disclaimer subtitle disappears; future open questions have an obvious home rather than each picking a discordant host.

**Cons.** Adds a new top-level spec topic and TOC entry; touches more files than an in-place restructuring; commits this kind of entry as a recurring spec category (not a one-off).

**Risks.** Inbound link breakage if the section anchor is not preserved with an `<a id>` shim at both the new and old location during the transition.

### Option B — Move under `governance/` as `governance/open-questions.md`

**Approach.** As Option A, but place the file under `docs/spec_topics/governance/` and add it to the governance TOC rather than spec-level TOC, on the basis that the stewardship of unresolved spec contradictions is a governance concern (open-question lifecycle, resolution-owner pointers, lock-step with version-bump items).

**Spec edits.** Same as Option A except the new file is `docs/spec_topics/governance/open-questions.md`, and the TOC update is to `governance.md`'s contents list rather than `spec.md`.

**Pros.** Locates the registry beside the procedural machinery (GOV rules, version-bump procedure) that owns its resolution; matches the existing pattern where governance carries process artefacts.

**Cons.** Hides the registry from readers who follow spec content (versus process); the entries in it are spec-state records, not governance rules with GOV-IDs.

**Risks.** Same anchor-preservation risk as Option A.

### Option C — Rename the host file so its title admits both halves

**Approach.** Rename `model-changes-and-non-goals.md` to a title that explicitly admits its dual content (e.g. `model-changes-non-goals-and-open-questions.md` or split into two files: `model-changes-and-non-goals.md` keeps the non-goals; a sibling `open-questions.md` inside `future-considerations/` takes the contradictions). Update the parent index in `future-considerations.md` to broaden its framing past *"deliberately deferred"*.

**Spec edits.** File rename or split; parent-index intro rewrite to admit "and open questions blocking implementation"; inbound references to the file path updated.

**Pros.** Minimal section content disturbance; preserves the registry's history-of-edits on the existing file.

**Cons.** The `future-considerations/` directory framing still has to be relaxed, weakening the categorisation it currently provides; readers who took the directory's "deferred features" framing seriously lose that signal across the entire bucket; renames are mechanically heavier than extracts.

**Risks.** Path-change link breakage across the corpus; broadening the parent-index intro invites future drift of unrelated transient state into the same bucket.

### Recommendation

Take **Option A**. The defect's root is a mismatch between transient state records and durable scope dispositions sharing a directory whose intro paragraph explicitly forbids the former kind; extracting to a dedicated top-level `open-questions.md` resolves both the file-level and directory-level miscategorisation in a single move, retires the `(blocking, not deferrals)` disclaimer that already concedes the bug, and gives any future open question a discoverable home. Implementer edges to watch: preserve the section anchor with an `<a id>` shim on the new page so existing deep-links from `host-prerequisites.md` clause (a), `version-bump-step2.md` item (a), `session-only-degraded-state.md`, and the SM-3b/4/5/6 *Implementation gate* sentences continue to resolve; do not collapse the substantive contradiction prose from `host-prerequisites.md` clause (a) into the new file (that relocation is a separate question, owned by the F-1002 / SM-3b cluster) — the new page initially carries only the existing back-reference bullet.

## Consequence

**Severity:** advisory

A reader who navigates the spec by directory/file titles will not expect a present-tense implementation blocker inside `future-considerations/`, and a maintainer searching for "where the spec records open contradictions" will not find a hit in any file named *open-questions*, *blockers*, or *contradictions*. The implementation gate is independently carried on the affected pages and on `V9l`, so the categorisation defect does not cause work to start on a blocked branch; the cost is discoverability, taxonomy hygiene, and the conceptual confusion of mixing transient state records with durable scope dispositions inside the same canonical registry.
