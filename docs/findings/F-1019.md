## Finding

The five-row *Per-boundary surfaces* table under `## Depth Enforcement` → **Error shape** in `docs/spec_topics/schema-subset.md` prescribes carrier shape, destination, and (for the slash-load `params` row) the cross-ceiling routing for surfaces whose normative owners live elsewhere — `query.md` / `errors-and-results.md` (row #1), `query.md` (row #2), `tool-calls.md` + `errors-and-results.md` (row #3), `invocation.md` + `binder/determinism-cancellation-failure.md` (row #4), `invocation.md` (row #5). The paragraph that closes the section says only that "this section's table mirrors it [the per-boundary table for ceiling #4] with the depth-walk-specific carrier details" and cross-links `overview-and-orientation.md#hard-runtime-ceilings`.

Two problems compound. First, the cross-link target is the informative four-ceiling orientation aggregator on `overview-and-orientation.md`, which under [Governance — GOV-30](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-30) carries no normative weight; the actual normative ceiling-#4 per-boundary table is `hard-ceilings/ceilings-3-and-4.md#ceiling-4-table`. So "mirrors it" already targets the wrong layer of the spec. Second, even with the target corrected, the schema-subset table does not declare whether it is itself normative (a second normative authority that must be kept in lock-step with the hard-ceilings table) or informative (a reader convenience whose authority is the hard-ceilings table alone). The word "mirrors" leaves both readings open: an implementer auditing rows #2–#5 cannot tell which carrier-shape claims to trust if the two tables drift, and a future editor cannot tell whether changing one row obliges a change in the other.

A drift surface already exists: `hard-ceilings/ceilings-3-and-4.md`'s row for `params` validation does **not** carry the `<ajv-summary>` placeholder clause that schema-subset.md's row #4 spells out (`the row's <ajv-summary> placeholder carries this enforcement point's canonical issue …`). If schema-subset's table is informative, that clause is undocumented prescription living in the wrong file; if it is normative, the two tables are already out of step. The same authority question applies row-by-row to #2's `ModelToolError` exclusion, #3's `CodeToolError { cause: "validation" }` shape, and #5's `cause: "return_validation"` literal — each of which is also restated, not delegated, here.

## Solution Space

**Shape:** multiple

### Option A — Promote `hard-ceilings/ceilings-3-and-4.md#ceiling-4-table` to sole normative owner; demote the schema-subset table to a depth-walk-specific informative restatement

**Approach.** Treat `ceilings-3-and-4.md#ceiling-4-table` as the single normative owner of ceiling #4's per-boundary destination/surface contract. Tag the schema-subset.md table explicitly as *informative — depth-walk-specific restatement of [Per-boundary destination/surface table (ceiling #4)](./hard-ceilings/ceilings-3-and-4.md#ceiling-4-table); the hard-ceilings table is authoritative.* Replace the "mirrors it" sentence with that disposition. The schema-subset table's rows are then permitted to be a strict subset of the hard-ceilings table's content (limited to "depth-walk-specific carrier details" that the hard-ceilings table doesn't already say); any restated content beyond that subset is an editorial defect.

**Spec edits.**
- `schema-subset.md` **Error shape** — add an explicit *Informative restatement* label sentence opening the table block; rewrite the trailing paragraph to delegate authority to `hard-ceilings/ceilings-3-and-4.md#ceiling-4-table` instead of `overview-and-orientation.md#hard-runtime-ceilings`; relocate the slash-load `<ajv-summary>` placeholder clause that currently lives only on the schema-subset row #4 into the corresponding row of `hard-ceilings/ceilings-3-and-4.md#ceiling-4-table` (where it joins the existing slash-load arm prose), or delete it from schema-subset.md if it is already covered there.
- `hard-ceilings/ceilings-3-and-4.md` — confirm the `<ajv-summary>` placeholder clause is on the table's `params` row; if not, add it.
- Plan-side: update each affected leaf's **Spec** field (and any "delegated live-carrier witness" bullets) to cite `ceilings-3-and-4.md#ceiling-4-table` only, removing the schema-subset.md citation for the carrier-shape claim and keeping it only for the depth-walk algorithm.

**Pros.** Aligns with the existing GOV-30 informative-aggregator regime and with the *Five-site list co-edit obligation* on `ceiling-invariants-and-audit.md`, which already names `ceiling-4-table` as the canonical enumeration. Eliminates a duplicate normative source. Keeps the schema-subset table as a reader convenience for someone who arrives at the depth walk without going through the hard-ceilings page.

**Cons.** Requires moving the `<ajv-summary>` placeholder clause and any other row-specific prose currently anchored to the schema-subset table. The informative restatement still needs to be kept in rough sync (textually, not normatively) — drift becomes a documentation defect rather than a correctness defect, but the reviewer SHOULD-flag obligation under GOV-30 still applies.

**Risks.** A future editor who reads only the schema-subset table may treat its informative content as a green-light to invent new carrier-shape claims there. Mitigated by the explicit *Informative restatement* label.

### Option B — Make the schema-subset table normative for the five rows it carries and impose an explicit two-site lock-step obligation

**Approach.** Declare the schema-subset table normative (it owns ceiling #4's per-boundary carrier contract from the depth-walk's perspective), declare the hard-ceilings table normative (it owns the same contract from the ceiling-set perspective), and add a paragraph that pins the two as bidirectionally lock-stepped — modelled on the *Five-site list co-edit obligation* already in `ceiling-invariants-and-audit.md`. Replace "mirrors it" with the lock-step pin and a forward-link.

**Spec edits.**
- `schema-subset.md` **Error shape** — open the table block with an explicit *Normative — bidirectionally lock-stepped with [Per-boundary destination/surface table (ceiling #4)](./hard-ceilings/ceilings-3-and-4.md#ceiling-4-table)* label; rewrite the trailing paragraph to delegate the lock-step authority to the hard-ceilings owner and to retarget the cross-link to `ceilings-3-and-4.md#ceiling-4-table` (not the GOV-30 orientation aggregator).
- `hard-ceilings/ceiling-invariants-and-audit.md` — add a second co-edit obligation paragraph (sibling to *Five-site list co-edit obligation*) requiring rows of `ceiling-4-table` and rows of schema-subset.md's *Error shape* table to be co-edited in the same commit.
- Plan-side: update each affected leaf's **Spec** field to cite both tables and add an explicit "lock-step" co-edit note in the **Adds** prose.

**Pros.** Preserves the reader's local access to the carrier contract from the depth-walk page (which is where someone investigating a `maxDepth` violation arrives) without forcing a forward-jump. Pinning the lock-step explicitly is exactly the same move the corpus already uses elsewhere for cross-shard tables.

**Cons.** Introduces a second normative authority for content that is already authoritatively owned elsewhere — strictly more work for editors than Option A, and a new closure surface for future ceiling-#4 work. The existing `<ajv-summary>` drift between the two tables shows the lock-step is non-trivial to maintain even when explicitly named.

**Risks.** Editors who change one table without the other land a correctness defect (no longer a documentation defect, as in Option A). The reviewer-flag culture is the only mechanical backstop, on the same terms as GOV-30.

### Recommendation

Take **Option A**. The corpus already treats the ceiling-#4 per-boundary table on `hard-ceilings/ceilings-3-and-4.md` as the canonical enumeration — that file carries the `ceiling-4-table` anchor that `overview-and-orientation.md`, `query-tool-loop.md`'s worked examples, PIC-1's mask-domain table, and every affected plan leaf (`V5e`, `V14e`, `V15j`) already cross-link to as the authoritative carrier source. Adding a second normative site (Option B) is more authoritative duplication than the rest of the corpus tolerates and asks the *Five-site list co-edit obligation* pattern to grow a sibling for a row count it already handles. The schema-subset table earns its place as a reader convenience for someone tracing a `maxDepth` violation locally, but it should be tagged informative and trimmed to depth-walk-specific content that the hard-ceilings table doesn't already say.

Implementer edge cases to watch under Option A: (a) the slash-load `<ajv-summary>` placeholder clause currently appears only on schema-subset.md's row #4 — moving authority to `ceilings-3-and-4.md#ceiling-4-table` requires either relocating that clause or verifying it is already covered by the existing slash-load arm prose; (b) `V5e`'s Adds prose and Tests bullets currently lean on schema-subset.md for the carrier contract — the leaf's **Spec** citation set needs the schema-subset reference narrowed to the depth-walk algorithm only, not the carrier shapes.

## Consequence

**Severity:** correctness

Two implementers reading rows #2–#5 cannot agree on which table is canonical and therefore cannot agree on which file to update when (for example) `cause: "return_validation"` changes shape: one will edit schema-subset.md alone and consider the work done, the other will edit hard-ceilings/ceilings-3-and-4.md and leave schema-subset.md behind. The slash-load `<ajv-summary>` clause already demonstrates that drift can land without notice. The depth-walk's wire contract (the `schema_keyword`, message, and carrier `cause` for each of the five enforcement points) is part of the on-wire contract the spec pins, so a silent inter-table divergence on these rows is a correctness defect for any cross-shard reader.
