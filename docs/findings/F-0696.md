## Finding

The paragraph anchored at `#argument-hint-binder-only` in `frontmatter-fields-a.md` is declared "the owning definition of the `argument-hint` autocomplete contract" and is forward-linked from `pi-integration.md` and `slash-invocation.md`. In a single paragraph it does four things: (1) states the loom-side rule (the autocomplete dropdown shows only the registered `description`); (2) explains *why* by reaching into the Pi SDK ("Pi's `RegisteredCommand` API has no `argumentHint` slot for extension-registered commands — the field exists only on Pi's built-in `.md` prompt templates"); (3) prescribes an author workaround keyed to that gap ("authors who want hint-like text visible in the dropdown must include it in `description:`"); and (4) cross-references the Future Considerations entry that tracks the upstream fix.

Pieces (2) and (3) bind loom's normative surface to the current state of an external API. If Pi adds the `argumentHint` slot to `RegisteredCommand` — exactly the change tracked in `future-considerations/surface-extensions.md` — the loom-side rule (1) becomes *more* permissive while the prescriptive workaround (3) becomes wrong (authors who concatenated their hint into `description:` will get duplicate text in the dropdown). The forward-compat seam is recorded in Future Considerations, but the prose at the owning site reads as a permanent normative prescription rather than as a current-API observation.

The same paragraph also carries the row-level claim in the field-contract table (the `argument-hint` row), and the supporting diagnostic `loom/load/argument-hint-not-displayed` is registered at `diagnostics/code-registry-load.md`. The diagnostic's *trigger* is loom-internal ("argument-hint declared without description") and is correct independent of any Pi-API state; the *message* template ("Pi's autocomplete entry will be empty") is the same bleed of Pi-API state into a normatively-fixed loom artefact.

## Solution Space

**Shape:** single

### Recommendation

Split the existing `#argument-hint-binder-only` paragraph into two structurally distinct pieces at the owner site, preserving the anchor on the normative piece:

1. **Normative loom-side contract (keeps `#argument-hint-binder-only`).** State the rule in loom's own terms only: the loom 1.0 autocomplete dropdown surface for an extension-registered slash command shows the `description` text; the `argument-hint` frontmatter field is consumed by the binder as grounding (per `slash-invocation.md`) and is not displayed in the autocomplete dropdown in loom 1.0. Declaring `argument-hint` without `description` is `loom/load/argument-hint-not-displayed` (advisory). No mention of `RegisteredCommand`, no workaround prescription.

2. **Non-normative note (new sub-paragraph, separate heading or italicised lead-in such as *"Rationale — Pi extension-API gap"*).** Explain the *reason* loom 1.0 has the rule above: Pi's extension-registered command surface has no `argumentHint` slot at the loom 1.0 Pi-SDK pin; only Pi's built-in `.md` prompt templates carry one. Cross-link the Future Considerations entry that tracks closing the gap. State explicitly that this paragraph is non-normative and tracks the current Pi-API state — when Pi closes the gap, this paragraph and the field row are revisited together, but the loom 1.0 rule above is not contingent on the rationale.

The author-workaround sentence ("authors who want hint-like text visible in the dropdown must include it in `description:`") moves into the non-normative sub-paragraph (or is dropped — it is the only piece that becomes actively wrong if Pi closes the gap, and the diagnostic plus the normative rule already tell the author everything they can act on).

Apply the same separation to the `argument-hint` row of the field-contract table: the row's normative cell states the loom-side behaviour and cites the anchor; the Pi-`RegisteredCommand` parenthetical moves to the non-normative sub-paragraph or is dropped from the row.

Update the diagnostic message template for `loom/load/argument-hint-not-displayed` in `diagnostics/code-registry-load.md` to drop the "(Pi extension-registered commands have no `argumentHint` slot)" parenthetical from the user-facing message. The user-facing message states the observable outcome ("`argument-hint` is set but no `description:` is set; the autocomplete dropdown entry will be empty"); the *reason* for the empty-dropdown outcome lives in the spec rationale, not in the message template that the runtime emits at load time.

Forward-links from `pi-integration.md` and `slash-invocation.md` continue to point at the same `#argument-hint-binder-only` anchor and so need no change. The Future Considerations entry already exists and is the right carrier for the seam; no edit required there.

Edge cases the implementer must watch:
- The `loom/load/argument-hint-not-displayed` *trigger* condition stays exactly as it is today (declared `argument-hint:` with absent / empty `description:`). Only the message template's wording changes; tests asserting the diagnostic code and severity continue to pass.
- Do not relocate the `#argument-hint-binder-only` anchor — three forward-links cite it. Keep the anchor on the normative piece.

## Consequence

**Severity:** advisory

The current text is not implementer-ambiguous in loom 1.0 — every reading produces the same autocomplete behaviour today — but it binds two of loom's authored normative surfaces (the owning paragraph and a diagnostic message) to the present state of an external API. The forward-compat seam exists in Future Considerations, yet the owner paragraph reads as if the workaround were a permanent rule, so a future Pi upgrade that closes the gap will silently invalidate published author guidance until the spec is re-edited.
