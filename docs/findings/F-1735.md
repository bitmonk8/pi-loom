## Problem

Output (c) of `version-bump-triggers.md`'s closing outputs paragraph makes runtime evidence a merge-blocking acceptance gate ("a bump is therefore not considered landed until the runtime evidence in (c) is green … MUST NOT be merged at the candidate pin"), but the gate rests on three terms the spec never operationalizes: "the loom's end-to-end harness" (named only at this site, never identified as a runnable command), "representative integrated `.loom`" (never characterized beyond the six surfaces it must drive, with no canonical fixture named and no selection rule), and "green" (used as the pass/fail predicate of a normative MUST with no defined success criterion). Two implementers reading only the spec cannot agree on which harness invocation to run, which fixture counts as "representative", or what observable result counts as "green", so the gate is unfalsifiable from the spec alone. Its operational meaning currently exists only in the plan, leaving the spec's own landing gate non-self-contained.

## Issue introduction

**Verdict:** single-commit introduction (introduced with the clause)
**Introducing commit:** `81ab342` — "pi-loom plan: resolve \"version-bump runtime-evidence acceptance gate and revert path\"" (2026-06-10)
**History:** The spec corpus is git-tracked. Before `81ab342`, the `version-bump-triggers.md` outputs paragraph enumerated only outputs (a) and (b) — `git show 81ab342^:docs/spec_topics/pi-integration-contract/version-bump-triggers.md` shows "its outputs are (a) … and (b) …" with no (c) and zero occurrences of "end-to-end runtime evidence". Commit `81ab342` added output (c), introducing the clause "a representative integrated `.loom` exercised through the loom's end-to-end harness against the bumped SDK" together with the merge-blocking MUST; the clause named "the loom's end-to-end harness" without stating the provider-execution posture from its first appearance. `git log -S "green end-to-end runtime evidence"` and `git log -S "not considered landed until the runtime evidence"` both bottom out at `81ab342`, confirming it as the origin. The same commit authored the plan-side `V18d`/`V18c` leaves that pin the `H4a` double, so the answer was settled in the plan in the same change that left the spec silent. The clause was later relocated by `81bf52a` (2026-06-11, the spec/section → `spec_topics/` collapse) and refined by subsequent version-bump resolutions, none of which added the live-vs-recorded posture.

## Solution approach

In output (c) of `version-bump-triggers.md`, operationalize the three undefined terms. Pin "the loom's end-to-end harness" to a single stable named invocation and cite where its existence and fidelity contract are owned (`plan_topics/H4a-factory-shell-and-harness.md`) rather than presupposing it. Define "representative integrated `.loom`" via the falsifiable six-surface coverage requirement already listed (typed query, tool-call loop, `invoke(...)`, schema validation, binder, cancellation). Define "green" as that named run completing with each of the six surfaces exercised and all assertions passing.

## Solution constraints

- Out of scope: the provider-execution posture (live vs recorded) is a distinct gap tracked separately and MUST NOT be conflated into the definition of "green".
