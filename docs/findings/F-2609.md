## Problem

QRY-20 (*Observability of discarded results*, `query-escapes-stringification.md`) specifies that a discarded query's failure event carries the failing query's source location in the `RuntimeEvent` `discard_site` field, but its only pointer to the field's declaration is the link text "Pi Integration Contract — Runtime event channel" whose href targets the PIC index aggregator (`../pi-integration-contract.md`) rather than the `runtime-event-channel.md` sub-page that actually declares `discard_site` (its `{ file, line, column }` type, its present-only-on-discarded-query-events presence rule, and its relationship to `query_site` and the dedup tuple). No `RuntimeEvent` shape exhibiting `discard_site` appears anywhere in the query cluster, so an implementer must land on the index page and traverse a further hop to recover the field's type and its distinction from `query_site`. The coarse link is an inconsistency, not a convention: the same shard already cites the sub-page precisely elsewhere (`../pi-integration-contract/runtime-event-channel.md#pic-1`). The parallel "Runtime event channel" link in `query-tool-loop.md`'s worked-example payload paragraph carries the same imprecision.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `6afca020c5f8f35e25d2cf69c4a4cf989ba62de4` (2026-06-03, "pi-loom spec: resolve \"discard-site field double-specified on RuntimeEvent\"") and `f5e89f4fb8f7e3816d3f5aacb2750c27d34dbc1d` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set")
**History:** `git log -S 'discard_site'` shows the field and the inbound "Runtime event channel" cross-reference were authored together in `6afca02`, which split the discard-site location out of the mode-overloaded `query_site` slot into a distinct `discard_site` field on the then-monolithic `pi-integration-contract.md` and pointed `query.md`'s link at that same page. The link was accurate at that point — the page contained the `RuntimeEvent` declaration. The size-cap split `f5e89f4` extracted the Runtime event channel into the `pi-integration-contract/runtime-event-channel.md` sub-page and repointed most inbound links to the new sub-page anchors (e.g. `unknown-reason-rule`, `diagnostic-emission-isolation`, `capability-probe`, `binder-inference#success-side-null-policy`) but left the QRY-20 (and the `query-tool-loop.md` payload) "Runtime event channel" link aimed at the now index-only aggregator, stranding the `discard_site` declaration one hop away. The "no in-shard `RuntimeEvent` shape exhibits `discard_site`" aspect predates the split (present since `6afca02`) but was benign while the link still resolved to the shape. The durable fix therefore addresses the cross-reference precision left stale by the split, not a revert of either commit.

## Solution approach

Repoint QRY-20's "Pi Integration Contract — Runtime event channel" link from the index aggregator (`../pi-integration-contract.md`) to the sub-page that declares the `RuntimeEvent` shape, `../pi-integration-contract/runtime-event-channel.md` (anchored on its `#runtime-event-channel` heading), matching the precise sub-page-anchor citation style already used at `query-tool-loop.md`. Apply the same repoint to the parallel "Runtime event channel" link in `query-tool-loop.md`'s worked-example payload paragraph.

## Solution constraints

- Do not restate the field's presence rule or dedup-tuple rule in the query shard — those stay owned by the PIC `runtime-event-channel.md` sub-page; QRY-20 retains only the per-form population rule it already owns.
