## Finding

`V1a`'s **Adds** field claims the tokeniser implements "the closed statement-termination / newline-continuation trigger set", and the paired test bullet in both `V1a` and `V1a-T` reads: "`loom/parse/single-line-if`, `loom/parse/stray-backslash`: termination/continuation violations fire per the closed trigger table." The closed trigger table is normatively defined in [`spec_topics/grammar.md` §Newline continuation](../../../docs/spec_topics/grammar.md) and enumerates four positive continuation triggers — unmatched open bracket, trailing binary/ternary operator, trailing comma, leading binary/ternary operator — plus the "blank lines do not break a continuation" rule.

The two cited codes are the only *violation* codes in this area (`single-line-if` for the no-single-line-`if` rule, `stray-backslash` for the no-line-continuation-marker rule). Neither code, nor any other listed vector, exercises the four positive continuation triggers or the blank-line-spanning behaviour. The leaf therefore asserts coverage of the whole closed table while its **Ships when** gate ("fires each listed code at the offending span") greens after verifying only the two negative-space codes.

The continuation engine — the part of the tokeniser that decides a statement continues across a newline when a trigger holds — has no asserted test and can ship broken (or absent) while the gate passes. The wording overclaims relative to what the listed vectors verify.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** The plan corpus is git-tracked. `git log --follow` shows `V1a-lexer-core.md` and `V1a-T-lexer-core.md` were created in `c6a664e`. `git log -S "closed trigger table"` and `git log -S "newline-continuation trigger set"` both return only `c6a664e`, and `git log -L 13,13:docs/plan_topics/V1a-lexer-core.md` shows the test bullet was added at file creation against `/dev/null`. The later commit `fa18160` rewrote only the **Adds** byte-source clause and left the trigger-set claim and the under-covering test bullet untouched. The overclaim/under-test mismatch has existed since the leaf was authored.

## Solution Space

**Shape:** multiple

### Option A — Verify the claimed table

**Approach.** Keep the "closed ... trigger set" claim and add the missing coverage so the gate actually exercises it.

**Plan edits.** In `V1a-T-lexer-core.md` (and mirror the new vectors in `V1a-lexer-core.md`'s **Tests**), add positive continuation-acceptance vectors covering each row of [`grammar.md` §Newline continuation](../../../docs/spec_topics/grammar.md) — unmatched open bracket, trailing binary/ternary operator, trailing comma, leading binary/ternary operator — plus one blank-line-spanning vector (`let x =\n\n  foo` tokenises as a single continued statement). Update `V1a`'s **Ships when** so its observable condition references the continuation-acceptance vectors as well as the two violation codes.

**Spec edits.** None.

**Pros.** The gate then verifies the continuation engine the leaf says it Adds; the coverage claim becomes true.

**Cons.** Larger test surface in the `-T` leaf.

**Risks.** None material; the spec table is the authoritative enumeration to copy from.

### Option B — Scope the wording to what is tested

**Approach.** Narrow the claim to the two verified violation codes.

**Plan edits.** In `V1a-lexer-core.md`, change "the closed statement-termination / newline-continuation trigger set" to reference only the listed violation codes, and change both leaves' test bullet from "per the closed trigger table" to "for the two listed termination/continuation violation codes."

**Spec edits.** None.

**Cons.** Leaves the positive continuation engine unverified by `V1a`; the engine ships on the strength of downstream slices exercising it incidentally.

**Risks.** Defers continuation coverage with no leaf owning it.

### Recommendation

Take Option A. `V1a`'s **Adds** declares the continuation engine as in-scope for this leaf, so the closing gate should verify it rather than the wording be softened to exclude it. Enumerate the continuation-acceptance vectors directly from the four rows of `grammar.md` §Newline continuation and add the blank-line-spanning case; land the new vectors in `V1a-T` first per the TDD ritual, then satisfy them in `V1a`. Edge case the implementer must watch: the leading-operator trigger and the blank-line rule interact (`let x = a\n\n  + b` continues across the blank line) — include that combined vector, not just the single-newline form.

## Consequence

**Severity:** advisory

The continuation engine is fully specified, so a careful implementer following the spec produces correct behaviour; the gap is that the gate verifies neither the positive continuation triggers nor blank-line spanning, so a tokeniser that mishandles continuation still greens `V1a` and downstream multi-line constructs would only surface the defect indirectly in later slices. The leaf's coverage claim is broader than its evidence.
