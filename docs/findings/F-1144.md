## Finding

The *Hang handling* clause in `docs/spec_topics/pi-integration-contract/conversation-drive.md` (`**Conversation drive — prompt mode**`, single bullet item) states:

> `loomAbort.abort()` invokes `ctx.abort()` on the captured slash-command handler context (the `ExtensionCommandContext.abort` member **already in the forwarded set** per **`ExtensionContext` (member surface loom touches)** below) …

The parenthetical is wrong on its face. The cited section (`docs/spec_topics/pi-integration-contract/host-interfaces-core.md`, *Per-mode override semantics* table) lists `abort` as one of exactly **three overridden** members, alongside `signal` and `sessionManager`. The override wraps `abort` to call `loomAbort.abort()` rather than Pi's underlying `abort()`. There is no "forwarded set" containing `abort` — the table partitions members into overridden vs forwarded, and `abort` is on the overridden side in both prompt and subagent mode.

An implementer who reads the parenthetical at face value will call the *synthesised, wrapped* `ctx.abort()` from the bidirectional forwarding path. Because that wrapper's body is "call `loomAbort.abort()`", the bidirectional propagation re-enters `loomAbort.abort()` recursively. The one-shot flag the same paragraph pins ("a re-entrant `loomAbort.abort()` … does not double-cancel") then short-circuits the re-entry to a no-op, so Pi's real `abort()` is **never** reached, the active user run is never cancelled, and `await ctx.waitForIdle()` never resolves — the precise hang the clause is meant to bound. The symmetric statement in `docs/spec_topics/cancellation.md` (*Forwarding into `loomAbort`*, slash-command-entry bullet) inherits the same ambiguity: it says the runtime "calls `ctx.abort()` on the captured `ExtensionCommandContext`" without distinguishing the raw Pi-supplied context from any synthesised `ExtensionContext`.

The intended object is the **raw `ExtensionCommandContext` Pi passes to the slash-command handler**, captured for the loom's lifetime; that object's `abort()` is unwrapped Pi-supplied `abort()`. The override table only synthesises `ctx.abort()` on the tool-execution `ExtensionContext` handed to `tool.execute(...)`, never on the slash-command handler's own `ExtensionCommandContext`. The spec needs to say this explicitly so the right call site is unambiguous.

## Issue introduction

**Verdict:** single-commit

**Introducing commit:** `d54d798` — *pi-loom spec: resolve "waitForIdle resolution, error, and hang semantics not contracted"* (2026-05-06).

**History:** The per-mode override table that places `abort` on the *overridden* side of the partition was introduced earlier in `27ec53d` — *Resolve Cluster-A: per-mode tool registration* (2026-05-04), as part of the synthesised tool-execution `ExtensionContext`. That commit was internally consistent: it pinned `abort` as overridden in both modes and made no claim about a "forwarded set" containing it.

Two days later, `d54d798` added the *Hang handling* paragraph (and its symmetric statement in `cancellation.md`), introducing the bidirectional `loomAbort.abort()` → `ctx.abort()` propagation rule together with the parenthetical "the `ExtensionCommandContext.abort` member already in the forwarded set per **`ExtensionContext` (member surface loom touches)** below". The parenthetical references the override table that `27ec53d` had already established, but mis-states the table's content: at the time `d54d798` landed, the table already showed `abort` as overridden. The defect is therefore a single-commit drift — `d54d798` authored the wrong cross-reference into a sibling table that already said the opposite — rather than a multi-commit interaction or a present-since-inception ambiguity.

The token `"in the forwarded set"` does not appear in the corpus prior to `d54d798` (`git log -S 'in the forwarded set'` on the spec corpus returns `d54d798` as the first introduction, with no preceding rename or move).

## Solution Space

**Shape:** single

### Recommendation

Replace the parenthetical in `conversation-drive.md` *Hang handling* so it names the unwrapped Pi-supplied `abort()` on the raw `ExtensionCommandContext`, and explicitly contrasts it with the synthesised tool-execution `ctx.abort()` wrapper:

> `loomAbort.abort()` invokes the unwrapped, Pi-supplied `abort()` on the slash-command handler's captured `ExtensionCommandContext` — **not** the synthesised tool-execution `ctx.abort()` wrapper from the *Per-mode override semantics* table in **`ExtensionContext` (member surface loom touches)** below, which would re-enter `loomAbort.abort()` recursively. The captured `ExtensionCommandContext` is the raw object Pi passes to the slash-command `handler`; the override table only synthesises a wrapped `abort()` on the `ExtensionContext` handed to `tool.execute(...)`, never on the slash-command handler's own context.

Apply the same disambiguation in `cancellation.md` *Forwarding into `loomAbort`*, slash-command-entry bullet ("the runtime calls `ctx.abort()` on the captured `ExtensionCommandContext`"): rephrase to "calls Pi's unwrapped `abort()` on the captured slash-command handler's `ExtensionCommandContext`" with the same cross-reference back to the override-table partition.

Edge cases the implementer must watch:

- The captured object is the `ExtensionCommandContext` Pi passes to the slash-command `handler`, retained for the loom's lifetime per the *Per-mode override semantics* prompt-mode bullet in `host-interfaces-core.md`. It is **not** the synthesised `ExtensionContext` handed to `tool.execute(...)`, even though both share the underlying live host's `abort` member by forwarding-table identity outside the three overridden rows.
- The one-shot guard on `loomAbort.abort()` remains load-bearing: it prevents an `agent_end` observer that re-enters `loomAbort.abort()` from double-cancelling, but it is no longer the only thing standing between the bidirectional path and an infinite recursion — the call-target disambiguation above is what actually severs the recursion at the source.
- The `ctx.abort()` override row of the `host-interfaces-core.md` table cites `cancellation.md`'s *Forwarding-listener throw* clause to cover a throw from the underlying `loomAbort.abort()` invoked by the wrapper from loom-driven tool code. That citation continues to apply unchanged: it covers the wrapper's call site, which is orthogonal to the bidirectional path's call site clarified here.

## Consequence

**Severity:** correctness

A literal reading of the parenthetical wires the bidirectional cancellation path to the wrapped, synthesised `ctx.abort()` rather than Pi's underlying `abort()`. Combined with the one-shot guard, the user turn never cancels and the prompt-mode driver hangs on `waitForIdle()` indefinitely — exactly the failure mode *Hang handling* claims to prevent. Two implementers reading the same paragraph will diverge on which context object to call `abort()` on, with no spec-internal evidence to arbitrate.
