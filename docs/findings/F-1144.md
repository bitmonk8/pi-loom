## Problem

The *Hang handling* clause in `docs/spec_topics/pi-integration-contract/conversation-drive.md` (`**Conversation drive — prompt mode**`) claims `ExtensionCommandContext.abort` is "already in the forwarded set", but the *Per-mode override semantics* table in `host-interfaces-core.md` lists `abort` as one of exactly three overridden members in both prompt and subagent mode — there is no forwarded set containing `abort`. An implementer reading the parenthetical literally wires the bidirectional cancellation path to the synthesised, wrapped `ctx.abort()` (whose body calls `loomAbort.abort()`), so the one-shot guard short-circuits the re-entry, Pi's real `abort()` is never reached, and `await ctx.waitForIdle()` never resolves — the precise hang the clause is meant to bound. The symmetric slash-command-entry bullet under *Forwarding into `loomAbort`* in `cancellation.md` inherits the same ambiguity: it does not distinguish the raw Pi-supplied `ExtensionCommandContext` from the synthesised `ExtensionContext`.

## Issue introduction

**Verdict:** single-commit

**Introducing commit:** `d54d798` — *pi-loom spec: resolve "waitForIdle resolution, error, and hang semantics not contracted"* (2026-05-06).

**History:** The per-mode override table that places `abort` on the *overridden* side of the partition was introduced earlier in `27ec53d` — *Resolve Cluster-A: per-mode tool registration* (2026-05-04), as part of the synthesised tool-execution `ExtensionContext`. That commit was internally consistent: it pinned `abort` as overridden in both modes and made no claim about a "forwarded set" containing it.

Two days later, `d54d798` added the *Hang handling* paragraph (and its symmetric statement in `cancellation.md`), introducing the bidirectional `loomAbort.abort()` → `ctx.abort()` propagation rule together with the parenthetical "the `ExtensionCommandContext.abort` member already in the forwarded set per **`ExtensionContext` (member surface loom touches)** below". The parenthetical references the override table that `27ec53d` had already established, but mis-states the table's content: at the time `d54d798` landed, the table already showed `abort` as overridden. The defect is therefore a single-commit drift — `d54d798` authored the wrong cross-reference into a sibling table that already said the opposite — rather than a multi-commit interaction or a present-since-inception ambiguity.

The token `"in the forwarded set"` does not appear in the corpus prior to `d54d798` (`git log -S 'in the forwarded set'` on the spec corpus returns `d54d798` as the first introduction, with no preceding rename or move).

## Solution approach

The intended target is the unwrapped, Pi-supplied `abort()` on the raw `ExtensionCommandContext` Pi passes to the slash-command handler and captures for the loom's lifetime — not the synthesised tool-execution wrapper. Clarify the *Hang handling* parenthetical in `conversation-drive.md` to name that unwrapped `abort()` and contrast it with the synthesised `ctx.abort()` wrapper from the *Per-mode override semantics* table (`#extensioncontext-interface` in `host-interfaces-core.md`). Clarify the slash-command-entry bullet under *Forwarding into `loomAbort`* in `cancellation.md` the same way, distinguishing the captured `ExtensionCommandContext` from the synthesised `ExtensionContext`.

## Solution constraints

- Out of scope: the *Per-mode override semantics* table in `host-interfaces-core.md` is correct as written; edit only the two cross-references that misread it.
