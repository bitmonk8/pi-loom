## Finding

Four normative rules in the discovery and typed-query areas state requirements an implementation must satisfy, but each rule is missing the observable hook a conformance test would have to attach to. As written, two reasonable implementers could each satisfy the prose and still ship divergent — or completely absent — behaviour, and no `npm test` assertion can tell them apart.

1. **DISC-6 `loom/load/discovery-slow` `details` shape.** The DISC-6 prose says the warning "names the root being scanned and the cap that fired"; the registry row in `diagnostics/code-registry-load.md` only pins the rendered `Message` template (`package-discovery walk aborted at <root>: <cap> cap reached`). Unlike the sibling `loom/load/package-read-timeout` row — which pins `details.kind = "package-read-timeout"` — DISC-6 names no `details` field at all, so the per-row `details` shape on which `diagnostics/diagnostic-shape.md` says every row's *Trigger* prose is the source of truth is empty. Operator tooling that wants to discriminate the two cap arms structurally (rather than by parsing the rendered message) has no contract to key on. The dual-cap tie-break rule (DISC-6 explicitly fixes the file-count predicate to win when both fire at the same site, and the test vector in `diagnostics/placeholder-rendering-b.md` asserts the rendered `<cap>` token in that case) confirms the cap arm is observable — but only through string parsing.

2. **DISC-5 same-class insertion-order MUST.** DISC-5's final sentence — "When two patterns of the same class match one path, insertion order is preserved" — has no falsifiable conformance scenario in any of the four classes the same paragraph defines. Plain includes union into a set, `!` removes from a set, `+` re-admits into a set, `-` removes from a set with final precedence; in every class, two patterns of that class matching one path produce the same membership outcome regardless of order. There is no observable effect "insertion order is preserved" could pin, so an implementation that hashes the patterns into an unordered structure satisfies the MUST vacuously.

3. **DISC-1 `~user` outcome.** DISC-1 says the `~user` form "is **not** honoured — only the bare `~` followed by `/` (or end-of-string)". It does not state what *does* happen to a `~user`-prefixed path: whether it is passed through verbatim to the seam call site (where the host filesystem will produce its own `ENOENT` per the failure-modes table), whether it triggers a load-time diagnostic, whether it is rejected at parse time, or whether the leading `~user` segment is stripped. The downstream rule that "`loom/load/missing-source` is the path the seam reports" provides a candidate behaviour but is never wired to this case, and `V10a`'s test bullet asserts only the negative ("`~/` expands only via `FileSystem.homedir()` — no `~user`/env/platform branch"). The positive outcome — what the operator actually sees when they write `~alice/looms/foo.loom` — is unspecified.

4. **Right-trim vector for the forced-respond boundary.** `query/query-tool-loop.md` paragraph 2 states "The runtime right-trims any trailing U+000A bytes the rendered prompt text carries before inserting this separator … keeping the prompt-to-instruction gap exactly one line feed in every case." This is a normative MUST with no test vector — `V13c-T`'s asserted vectors cover the free-phase tool loop, `max_rounds:0`, exhaustion, and the depth-6 co-fire case, but not the right-trim. An author who writes a template ending `\n\n\n` against a template that ends `\n` would, under a non-trimming implementation, be observable in the model-facing turn; under a trimming implementation, byte-identical. No fixture proves the trim happened.

## Issue introduction

**Verdict:** multi-commit-interaction

- `1cb54c7` (2026-06-06, *"resolve Glob !/+/- precedence and matcher engine unspecified for pi.looms / loomPaths"*) — added the DISC-5 same-class "insertion order is preserved" MUST sentence into an already-set-shaped algebra where it has no observable.
- `475155c` (2026-05-05, *"V14m discovery walk omits scoped packages and has no upper bound"*) and `25e8a60` (*"T08 Per-package.json read timeout: overrun is unbounded"*) together established the DISC-6 cap+root prose and the `loom/load/discovery-slow` registry row; the row pinned the rendered Message template but never the `details` shape, leaving the cap arm observable only by string parsing.
- `bbe2602` (2026-06-05, *"Initial forced respond turn — instruction wording status undeclared"*) — landed the *Initial forced respond turn template* section and the right-trim MUST in the paragraph that references it. The template body became byte-exact in the same commit, but the right-trim itself never got a paired vector.

**History:** Each defect arrived as a single-commit normative fix that closed a different prior gap; none of the fixing commits paired the new MUST with the testability hook it required. The interaction is "fix-without-observable": four independent edits over a one-month span each landed a clarification that read complete in prose review but is unobservable to the per-leaf citing-test gate (`H5f`). The pattern — not any single one of these commits — is what produced the consolidated finding.

## Solution Space

**Shape:** multiple

The four obligations are independent and resolve through four small, separate spec edits. Per the bimodal-obligation rule, each is mapped as its own option below; the recommendation orders them so each edit lands on a stable baseline.

### Option A — DISC-6 `details` shape

**Approach.** Extend the DISC-6 paragraph (and mirror onto the `loom/load/discovery-slow` registry row's *Trigger* prose) to pin the structured `details` payload: `details.kind = "discovery-slow"`, `details.root` = the absolute filesystem path of the root the walk aborted at (the same string the rendered `<root>` placeholder carries), `details.cap` drawn from the closed enum `{"looms.scanPackagesMaxFiles", "looms.scanPackagesTimeoutMs"}` (already pinned for the `<cap>` placeholder in `placeholder-rendering-b.md` §7), and `details.elapsedMs` for the time-cap arm (so operator tooling can distinguish "barely over" from "stuck"). Existing `<cap>` rendering and the dual-cap tie-break rule already constrain which value goes where; this option only lifts those into the per-row `details` contract.

**Spec edits.** One sentence appended to DISC-6 ("The diagnostic carries `details: { kind, root, cap, elapsedMs? }` per the per-row shape rule in [diagnostic-shape.md]"); one corresponding `details` mention in the `loom/load/discovery-slow` row's *Trigger* prose; the existing rendered-Message vector in `placeholder-rendering-b.md` is unchanged.

**Pros.** Operator tooling gains a structured discriminator; the cap-arm test in `V10b-T` becomes a structured-payload assertion instead of message parsing; matches the established convention used by `loom/load/package-read-timeout`.

**Cons.** Widens the diagnostic's frozen-baseline surface — the `details` shape is now a wire contract subject to source-language-stability.

**Risks.** None significant; the row already lives in the location-less-diagnostics closed set, so the schema change is bounded.

### Option B — DISC-5 same-class insertion-order MUST

**Approach.** Demote the "When two patterns of the same class match one path, insertion order is preserved" sentence from a MUST to a *Note*, with the explicit reasoning that all four classes are idempotent under same-class overlap and the sentence therefore has no observable. This preserves the editorial intent (signalling determinism to authors who might worry about pattern reordering) without asserting an unfalsifiable obligation.

**Spec edits.** Replace the MUST sentence in DISC-5 with: "*Note.* The four override classes are each idempotent under same-class overlap — two `!` patterns matching the same path drop it once, two `+` patterns re-admit it once, etc. — so the within-class evaluation order is not operator-observable. The cross-class order pinned above (1→2→3→4) is the only ordering with observable effects."

**Pros.** Removes a vacuous MUST that would otherwise count against the `H5e` un-anchored-MUST gate by being un-citeable in tests; clarifies the determinism story for authors.

**Cons.** Loses a forward-compatibility seam — a future widening of the class semantics that *did* depend on insertion order would have to re-introduce the MUST and add an observable.

**Risks.** A future spec edit that introduces an ordering-sensitive operator (e.g. a per-pattern weight) would need to re-establish the rule explicitly.

### Option C — DISC-1 `~user` outcome

**Approach.** State the concrete outcome: a path beginning with `~<username>` (any non-`/` character following the tilde) is left **byte-verbatim** when passed to the `FileSystem` seam; expansion is performed only on the bare-`~`-then-`/` (or end-of-string) form. The downstream seam call site reports `ENOENT` per its normal contract, and the failure-modes table's *missing-path* row of the relevant source classifies the outcome (silent for conventional roots, error for explicit references). Cross-reference DISC-2's failure-modes table so the reader can trace the outcome without a separate enumeration.

**Spec edits.** Append one sentence to DISC-1: "A path whose leading `~` is followed by any character other than `/` or end-of-string (the `~user` form, or `~~`, `~.`, etc.) is passed verbatim to the seam call site — the leading `~user` segment is **not** stripped — and surfaces through the standard failure-modes table per DISC-2."

**Pros.** Makes `V10a-T`'s existing negative bullet ("no `~user` branch") completable with a positive companion assertion ("a `~user`-prefixed `--loom` flag emits `loom/load/missing-source` via DISC-2's CLI row"); uses the existing failure-modes machinery rather than introducing a new diagnostic.

**Cons.** None material — the verbatim-passthrough behaviour is the simplest interpretation already consistent with the "MUST NOT use any platform-conditional branch" rule.

**Risks.** None.

### Option D — Right-trim vector

**Approach.** Add a normative test vector pinning the right-trim behaviour to the *Initial forced respond turn template* section of `query/query-tool-loop.md`. The vector specifies: a typed query whose rendered prompt body ends in `"foo\n\n\n"` produces a forced-respond user turn whose body is exactly `"foo" + "\n" + "<rendered-template-body>"`, byte-equal to the same query whose prompt body ends in `"foo"` or `"foo\n"`. Phrase as a fenced-block test vector parallel to the depth-6 worked example already in the same file.

**Spec edits.** Insert a "*Right-trim vector (normative)*" sub-section after the *Initial forced respond turn template* block; add a corresponding `V13c-T` bullet citing the new anchor.

**Pros.** Pins an otherwise-silent regression surface; small edit (one fenced block); aligns with the existing pattern of "depth-6 worked example" as a normative vector.

**Cons.** None.

**Risks.** None.

### Recommendation

Resolve in the order **C → B → A → D**. Rationale:

- **C** (`~user` outcome) is a one-sentence addition that touches no other rule and stabilises DISC-1; doing it first removes ambiguity for any subsequent edit that cites DISC-1.
- **B** (insertion-order demotion) is a one-sentence in-place replacement in DISC-5; demoting before any structured edit to DISC-6 avoids two near-simultaneous touches of the same paragraph.
- **A** (DISC-6 `details` shape) is the largest edit — it touches both the spec topic and the diagnostic registry row, and adds a frozen-baseline surface element. Land it on a stable DISC-5/DISC-6 baseline.
- **D** (right-trim vector) is independent of the discovery edits and can land any time; placing it last keeps the discovery cluster atomic.

Implementer notes:

- For **A**, ensure `details.elapsedMs` is emitted only on the `looms.scanPackagesTimeoutMs` arm — under the dual-cap tie-break the file-count cap wins, and emitting `elapsedMs` on that arm would suggest the time predicate fired.
- For **C**, the seam-verbatim passthrough must hold on **every** `~`-eligible site DISC-1 enumerates (global discovery root, global package roots, global settings file, settings `loomPaths` entries, `--loom` CLI components) — not just one — so the failure-modes-table classification is uniform.
- For **D**, the right-trim is on **trailing U+000A bytes only**, not on trailing whitespace generally; the vector should include a case ending `"foo\n  "` to prove non-newline trailing whitespace is preserved (per the surrounding `query-forms.md` rule that newline-trim is newline-only).

## Consequence

**Severity:** correctness

Two compliant implementations can diverge in operator-observable ways on every one of the four points (structured `details` payload for `discovery-slow`, ordering of equivalent overlapping globs, the rendered behaviour of `~user`-prefixed paths, and the trailing-newline content of the forced-respond opening turn) without any conformance test catching the drift. The forced-respond right-trim in particular is the only thing keeping the prompt-to-instruction separator at exactly one U+000A — a regression there silently changes every typed query's opening turn shape against an LLM-facing wire contract the spec otherwise pins byte-exactly.
