## Finding

The `loom/parse/unsupported-feature` row in [`code-registry-parse.md`](../../docs/spec_topics/diagnostics/code-registry-parse.md) describes its trigger as "A loom 1.0-deferred or non-Loom syntactic construct (arrow function, spread, optional chaining, `===`, bitwise op, comma op, nested template, etc.) appears in source." The trailing "etc." is open-ended: the row is the only normative anchor for what fires this code, and it explicitly invites unlisted constructs.

The renderer in [`placeholder-rendering-a.md` §3](../../docs/spec_topics/diagnostics/placeholder-rendering-a.md) is closed in the opposite direction. The `<construct>` placeholder used by this code's `Message` template (`unsupported syntactic feature: <construct>`) is rendered from a fifteen-row closed token-name table (`arrow function`, `spread`, `optional chaining`, `nullish coalescing`, `strict equality`, `bitwise <op>`, `comma operator`, `nested template`, `new`, `typeof`, `instanceof`, `delete`, `void`, `yield`, `await`). §3 provides no fallback rule, and the file's opening §1 declares the closure governed by GOV-7 / GOV-8 — i.e. enforced at build time.

The two firing sites the plan already names sit outside the table: [`V1b`](../../docs/plan_topics/V1b-literals-and-paths.md) ships `loom/parse/unsupported-feature` for "reserved hex/octal/binary/underscore numeric forms", and [`V5d`](../../docs/plan_topics/V5d-reject-gate.md) ships it as the allowlist-reject diagnostic for "a JSON-Schema construct outside the permitted subset … absent from the enumerated unsupported-keyword list" — neither numeric forms nor JSON-Schema keywords correspond to any row in the closed table. A conformant implementation following §3 literally has no token to interpolate at these emission sites, so the normative `Message` template is unrenderable and the DIAG-4 obligation (every asserting test sources its expected string from the registry) cannot be discharged.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `839db71` (2026-05-04, *pi-loom spec: resolve "Diagnostic codes are declared as namespaces but never assigned to specific errors"*) — introduced the `loom/parse/unsupported-feature` registry row with the open "etc." trigger gloss, in the monolithic `spec_topics/diagnostics.md` that predates the diagnostics shard split.
- `61ea1a5` (2026-05-06, *pi-loom spec: resolve "Diagnostic message placeholders have no defined rendering"*) — introduced the §3 *Syntactic-construct placeholder* rule and the fifteen-row closed token-name table, including the closure-governance pin to GOV-7 / GOV-8.

**History:** `git log -S "etc.) appears in source"` and `git log -S "arrow function" -- spec_topics/diagnostics.md` both surface `839db71` as the first time the open trigger appears and `61ea1a5` as the first time the closed token-name table appears, with the size-cap split (`f5e89f4`, 2026-06-04) relocating the same text into the present per-shard files (`code-registry-parse.md`, `placeholder-rendering-a.md`) verbatim. Neither commit on its own introduces a defect — the trigger row alone is a reasonable summary, and the rendering rule alone is a reasonable closed list — but the join leaves an unrenderable case for any unlisted construct. Subsequent unrelated work (`72e2c31`, *Placeholder-rendering closure omits live placeholders*) tightened §1 closure governance without revisiting `<construct>`, so the defect has been present since 61ea1a5 and is durable in the current corpus.

## Solution Space

**Shape:** multiple

### Option A — Extend the §3 closed table to cover every firing site

**Approach.** Audit the plan and spec for every site that fires `loom/parse/unsupported-feature`, add one row per construct to the §3 table (so `0xFF`, `0o7`, `0b1`, underscore-separated digits, each rejected JSON-Schema keyword such as `pattern`/`format`/`minLength`/`exclusiveMinimum`, plus any other future-deferred forms named elsewhere), and rewrite the row's "etc." gloss as a cross-reference to the §3 table. Keep the closure governed by GOV-7 / GOV-8.

**Spec edits.**
- `code-registry-parse.md`: replace "(arrow function, …, etc.)" with "(see [`placeholder-rendering-a.md` §3 closed token-name table](./placeholder-rendering-a.md#3-syntactic-construct-placeholder))".
- `placeholder-rendering-a.md` §3: extend the token-name table with one row per construct mined from the audit; group by category if useful (`syntactic-construct`, `reserved-numeric-literal`, `unsupported-jsonschema-keyword`).

**Pros.** Preserves byte-identical-message determinism; preserves the closure posture §1 declares; one place to look for the full enumeration.

**Cons.** The table grows from ~15 rows to ~40+, and the JSON-Schema-keyword rows have to track the schema subset's deferred/permitted partition, coupling §3 to schema-subset evolution. Adding a future deferred construct becomes a spec-versioned breaking change governed by GOV-7 / GOV-8.

**Risks.** Easy to omit a construct mentioned only in a plan leaf or appendix; a missed audit row resurfaces the same defect silently. Mitigate by adding a cross-spec lint that every plan reference to `loom/parse/unsupported-feature` names a token present in the §3 table.

### Option B — Make `<construct>` two-tier: closed canonical name + open verbatim-source fallback

**Approach.** Keep the §3 table as a canonical-name lookup for the enumerated cases, and add a fallback rule for unlisted constructs that renders the offending source span verbatim — identical in shape to the `<expr>` rule already present in §3 for `default-not-literal` and `tool-arg-not-literal`. The trigger row stays open ("etc.") because the renderer now defines behaviour for any input.

**Spec edits.**
- `placeholder-rendering-a.md` §3: add a third bullet: "For `<construct>` in `loom/parse/unsupported-feature` at a site not enumerated in the table above, render the offending source span verbatim per the `<expr>` rule." Add a test vector (e.g. `0xFF` renders `unsupported syntactic feature: 0xFF`).
- `code-registry-parse.md`: leave the "etc." row largely intact; add a parenthetical "see [§3](./placeholder-rendering-a.md#3-syntactic-construct-placeholder) for the closed canonical names and the verbatim-source fallback".
- §1: relax the §3 closure statement to apply to the canonical-name rows only, since the fallback admits arbitrary source spans.

**Pros.** No table maintenance burden when new deferred constructs are added (they get the fallback); both `V1b` numeric forms and `V5d` JSON-Schema keywords render deterministically as their source text. Aligns `<construct>` with `<expr>` (one less rendering shape to learn).

**Cons.** Two constructs with the same canonical name and different source spans (`if (a === b)` vs `if (a !== b)`) render differently under the fallback but identically under the table — split behaviour creates a subtle conformance-test surface. Weakens the §1 closure claim for §3.

**Risks.** Implementers may inadvertently take the fallback path for a construct that does have a canonical row (rendering source verbatim where they should render `arrow function`); add a normative rule that the canonical table MUST be consulted first.

### Option C — Split `loom/parse/unsupported-feature` into namespace-scoped codes

**Approach.** Replace the single code with three: `loom/parse/unsupported-syntactic-construct` (the current §3-table cases), `loom/parse/unsupported-numeric-literal-form` (V1b reserved forms), and `loom/parse/unsupported-schema-keyword` (V5d reject-gate cases). Each gets its own row, trigger, and `Message` template; `<construct>` shrinks to the syntactic-construct table and stays closed; the other two codes carry placeholders sized to their domains (e.g. `<token>` for the numeric-literal form, `<keyword>` for the schema keyword).

**Spec edits.**
- `code-registry-parse.md`: split one row into three; update `Message` templates accordingly.
- `placeholder-rendering-a.md` §3: keep `<construct>` closed and unchanged; add placeholder rules for the two new placeholders (probably in §3 alongside `<construct>`, or §1 as new categories).
- Every plan leaf that names `loom/parse/unsupported-feature` (currently `V1b`, `V1b-T`, `V5d`, `V5d-T`) updates its code reference.
- DIAG-2 / DIAG-3 / DIAG-4 obligations reapply automatically once the split lands.

**Pros.** Each code has a domain-shaped closed set; no fallback is needed; diagnostics carry sharper machine-readable intent (`<keyword>` is more useful to a schema tool than a generic `<construct>`).

**Cons.** Largest blast radius — touches every plan leaf and every test that currently asserts the old code. Two new codes consume registry slots and require their own DIAG-2 / DIAG-3 / DIAG-4 enforcement. May still leak a fourth category later (e.g. unsupported regex flags) and require another split.

**Risks.** GOV-3 / DIAG-3 forbid renames in loom 1.0; this is a delete-plus-add, which the registry-closing gate must accept while consumers migrate. If any in-flight implementation already asserts the old code, the transition is observable. Schedule the split before the registry-closing gate (`V7b`) lands.

### Recommendation

Take **Option A**. The corpus already enforces closure of §3 under GOV-7 / GOV-8 and treats the byte-identical-message contract as load-bearing; extending the closed table preserves that posture and keeps the registry-closing gate's job simple. Option B trades determinism for maintenance ease at a site where the spec has explicitly chosen determinism elsewhere (the §1 closure statement was tightened, not relaxed, as recently as `72e2c31`), and Option C's payoff (sharper machine-readable intent) does not yet justify rewriting every consuming plan leaf before `V7b` lands.

When applying Option A, the implementer must (i) mine the audit from `V1b`, `V5d`, `schema-subset.md`, and `expressions.md`'s *Not supported* section, not only from the spec_topics/diagnostics directory; (ii) preserve the `bitwise <op>` row's existing sub-token rule and consider an analogous "numeric-literal-prefix `<prefix>`" row that absorbs `0x`/`0o`/`0b` in one row rather than three; (iii) leave the §1 governance pin to GOV-7 / GOV-8 untouched, since the closure posture is what makes the registry-closing gate enforceable.

## Consequence

**Severity:** correctness

Two implementers reading only the row will plausibly disagree on what tokens to interpolate for unlisted constructs (a verbatim source span, a synthetic name, the literal string `etc.`, an internal-error). Two implementers reading only §3 will reject the firing sites the plan already mandates (`V1b` numeric forms, `V5d` JSON-Schema keywords) as unrepresentable. The DIAG-4 obligation that every test sources its expected message from the registry is unsatisfiable at those sites, and the H5a registry-closing gate has nothing stable to enforce.
