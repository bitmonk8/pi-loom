## Problem

The `loom/parse/unsupported-feature` row in `code-registry-parse.md` describes its trigger with an open-ended "etc." that explicitly invites unlisted constructs, while the `<construct>` placeholder renderer in `placeholder-rendering-a.md` §3 is a closed fifteen-row token-name table governed by GOV-7 / GOV-8 with no fallback rule. The two named firing sites — `V1b`'s reserved hex/octal/binary/underscore numeric-literal forms and `V5d`'s rejected JSON-Schema keywords — correspond to no row in the closed table. A conformant implementation following §3 literally has no token to interpolate at those sites, so the `Message` template `unsupported syntactic feature: <construct>` is unrenderable and the DIAG-4 obligation (every asserting test sources its expected string from the registry) cannot be discharged.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `839db71` (2026-05-04, *pi-loom spec: resolve "Diagnostic codes are declared as namespaces but never assigned to specific errors"*) — introduced the `loom/parse/unsupported-feature` registry row with the open "etc." trigger gloss, in the monolithic `spec_topics/diagnostics.md` that predates the diagnostics shard split.
- `61ea1a5` (2026-05-06, *pi-loom spec: resolve "Diagnostic message placeholders have no defined rendering"*) — introduced the §3 *Syntactic-construct placeholder* rule and the fifteen-row closed token-name table, including the closure-governance pin to GOV-7 / GOV-8.

**History:** `git log -S "etc.) appears in source"` and `git log -S "arrow function" -- spec_topics/diagnostics.md` both surface `839db71` as the first time the open trigger appears and `61ea1a5` as the first time the closed token-name table appears, with the size-cap split (`f5e89f4`, 2026-06-04) relocating the same text into the present per-shard files (`code-registry-parse.md`, `placeholder-rendering-a.md`) verbatim. Neither commit on its own introduces a defect — the trigger row alone is a reasonable summary, and the rendering rule alone is a reasonable closed list — but the join leaves an unrenderable case for any unlisted construct. Subsequent unrelated work (`72e2c31`, *Placeholder-rendering closure omits live placeholders*) tightened §1 closure governance without revisiting `<construct>`, so the defect has been present since 61ea1a5 and is durable in the current corpus.

## Solution approach

Narrow the open `loom/parse/unsupported-feature` trigger gloss in `code-registry-parse.md` so it no longer invites unlisted constructs, rewriting the parenthetical as a forward-link to the §3 closed token-name table in `placeholder-rendering-a.md`. Add rows to that §3 table covering every site that fires the code — including `V1b`'s reserved numeric-literal forms and `V5d`'s rejected JSON-Schema keywords — so the trigger and renderer resolve to one closed set.

## Solution constraints

- The §1 closure governance pin to GOV-7 / GOV-8 stays in force: the §3 table remains closed and build-time-enforced; do not introduce an open fallback rendering rule.
