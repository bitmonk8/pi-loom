## Problem

The *Tool execution from loom code* section of `host-interfaces-core.md` (anchor `#tool-execution-from-loom-code`) specifies that a code-side `<name>(args)` call invokes a Pi tool's `execute(toolCallId, â€¦)` directly, passing a synthesised `loom-direct:`-prefixed UUID `toolCallId` that is never registered through Pi's dispatcher. For this design to be correct the receiving tool's `execute()` must treat the inbound `toolCallId` as **opaque** â€” it must not look the id up in a dispatcher-side registry, must not correlate it against a prior model-issued `tool_use` block, and must not validate its shape or provenance. The spec relies on this consumption posture for every code-side call but never states it, and `host-prerequisites.md`'s register of named host-behaviour presuppositions â€” each routed to the version-bump editorial-review checklist so a falsifying Pi minor is caught at bump time â€” carries no entry for it, leaving a hole in the bump-time safety net the corpus maintains for every other unpinned host reliance.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `27ec53d` (2026-05-04, "Resolve Cluster-A: per-mode tool registration") is the earliest commit where the direct out-of-band `execute(toolCallId, â€¦)` design with a synthesised `loom-direct:` id is textually locatable; that commit references the *Tool execution from loom code* prose as already-present, so the design â€” and the omission of the opaqueness presupposition â€” predates it.
**History:** The corpus is git-tracked. The out-of-band direct-`execute()` design has carried the unstated opaqueness/non-correlation consumption posture since the section was first authored. Two later edits refined the id's *generation* without ever surfacing the *consumption* posture: `0c6c347` (2026-06-25, F-1396) pinned the `toolCallId` UUID body format and routed minting through PIC-20, and `95d4dd9` (2026-06-27, F-1923) named `IdSource.newToolCallId()`. A history search for any opaqueness / non-dispatcher-correlation statement tied to `toolCallId` (`git log -S`/`-G` over `opaque`, `correlat`, `dispatcher`, `out-of-band` across the corpus) returns no commit that ever added one. The defect is therefore an omission present since inception, not one a specific later commit introduced.

## Solution approach

Add a named presupposition paragraph with its own anchor to the `host-prerequisites.md` presupposition register, in the same shape as the existing entries, asserting that at the loom 1.0 Pi-SDK pin a Pi tool's `execute(toolCallId, â€¦)` treats the `toolCallId` as opaque (no dispatcher-registry lookup, no `tool_use` correlation, no provenance/shape rejection) and citing `#tool-execution-from-loom-code` as the consumer. Add a corresponding item to the *Editorial-review checklist for unpinned host presuppositions* in `version-bump-step2.md` (items currently run (a)â€“(as)) pointing at the new presupposition anchor, so the per-bump audit re-confirms the posture against each candidate Pi minor. If an authoritative Pi SDK statement of the opaqueness contract can instead be located, cite it at the `#tool-execution-from-loom-code` call site in place of the presupposition.
