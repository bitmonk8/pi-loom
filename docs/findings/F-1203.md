## Finding

Two independent defects on the same registration / watcher surface, both phrased in implementation terms rather than as observable behaviour:

1. **Collision-source named set (`registration-steps.md`, step 3 blockquote).** The "loom 1.0 seam — Pi-owned subagents collision source set" MUSTs that the three-arm `SlashCommandSource` candidate list "be defined as a single named set inside the runtime, consulted by the collision check via membership test rather than open-coded as three separate string comparisons or a hard-coded `switch`." This is a structural pin on an internal runtime artefact: a passing-and-a-failing implementation are observationally identical — both drop the same looms with the same `loom/load/cross-format-collision` diagnostic against the same `(name, source)` inputs. The seam's load-bearing requirement is the *collision-candidate membership rule* (`source ∈ {"prompt","extension","skill"}`) and the *future-widening invariant* (adding `"subagent"` is the only loom 1.0 surface that perturbs); neither needs the "single named set / membership test" implementation shape to hold, and no conformance test in `V9b`/`V9b-T` cites it as an acceptance criterion. The clause reads as authoring guidance promoted to MUST.

2. **Watcher post-`error` posture (`host-interfaces-services.md` §PIC-14).** The chokidar adapter's bullet states that the adapter filters `error` (among others) out of the seam and routes its own `error` events through `ctx.ui.notify`. It does not state what the *watcher* does after that error: does the chokidar instance keep delivering subsequent `add`/`change`/`unlink` events, has the watcher died (leaving content silently stale until the next `/reload`), is the extension instance now degraded in the sense PIC's drain-state contract uses that word, and is `/reload` the operator recovery path? Chokidar's `error` callback is non-fatal by default but specific underlying syscall failures (e.g. an inotify watch limit, an unreadable mount) can stop event delivery on one or more roots without the watcher object surfacing a separate "closed" signal. The diagnostics page's transient-toast paragraph (`diagnostic-shape.md`) mentions the same surface but explicitly does not state post-throw watcher state, scan continuation, or message content — so neither owning page closes the loop. Two reasonable implementers will diverge: one wraps each chokidar root with a re-arm-on-`error` adapter and treats the toast as informational; another lets the watcher die silently and assumes `/reload` restarts it. Operators have no contract to lean on.

The two halves share a file pair but resolve independently: (1) is a one-paragraph demotion / rephrasing in `registration-steps.md`; (2) adds new normative text in `host-interfaces-services.md` (and a cross-reference from `diagnostic-shape.md`'s transient-toast paragraph) about continuation and recovery.

## Solution Space

**Shape:** multiple

### Option A — Demote / reframe the collision-source named-set MUST

**Approach.** In `registration-steps.md` step 3, rewrite the "loom 1.0 seam — Pi-owned subagents collision source set" blockquote so the normative content is the behavioural collision-candidate membership rule and the future-widening invariant, not the internal collection shape. Keep the seam anchor (`#loom-1-0-seam-pi-owned-subagents-collision-source-set` / `#v1-seam-pi-owned-subagents-collision-source-set`) — `future-considerations.md` links it.

**Spec edits.**
- Replace the MUST sentence with: "The `session_start` handler MUST treat a pending loom as colliding when an existing entry's `source` is one of `"prompt"`, `"extension"`, `"skill"` (the three arms of Pi's `SlashCommandSource` union at the loom 1.0 Pi-SDK pin); other source values do not collide."
- Move the "single named set / membership test rather than three string comparisons" sentence into an italicised *Implementation note* at the end of the blockquote (non-normative) — it is genuine maintainability guidance for the future-widening case and is worth keeping in some form.
- Preserve the existing sentence about widening to four arms when `SlashCommandSource` grows a `"subagent"` arm.

**Pros.** Removes an unobservable MUST. Keeps the future-considerations cross-reference and the widening invariant intact. One-paragraph edit, no plan-acceptance disturbance beyond reframing `V9b`'s `cka-23.getcommands` facet citation.

**Cons.** Implementation note's status as non-normative is itself a soft signal; readers used to MUSTs may treat it as a hint and skip it.

**Risks.** None material.

### Option B — Specify the post-`error` watcher posture

**Approach.** In `host-interfaces-services.md` §PIC-14, extend the chokidar-`error` bullet from "routes through `ctx.ui.notify`" to a closed two-arm rule: the adapter (i) classifies the underlying error as *transient* (event delivery continues) or *terminal* (chokidar will deliver no further events on at least one watched root); on (ii) the adapter MUST emit a single `loom-system-note` instructing the operator to `/reload`, and the runtime MUST transition the extension instance into the existing `"degraded-needs-reload"` drain-state arm so subsequent slash dispatches return the standard degraded note rather than dispatching against stale content. Classification follows chokidar's documented behaviour: any `error` whose `.code` is in the closed set `{"EMFILE", "ENOSPC", "ENFILE"}` (watch-limit / file-descriptor exhaustion) or whose `path` argument identifies a watched root that chokidar reports as no-longer-watched via `getWatched()` is terminal; everything else is transient and merely toasts via `ctx.ui.notify`. Cross-reference the new rule from the transient-toast paragraph in `diagnostic-shape.md`.

**Spec edits.**
- `host-interfaces-services.md` §PIC-14 bullet: split the existing chokidar-`error` sentence into a transient arm (current behaviour: `ctx.ui.notify` toast only) and a terminal arm (new: `loom-system-note` + `LoomRegistry.markRuntimeDegraded` transition + persistent `loom/runtime/watcher-terminated` diagnostic). Pin the terminal-arm classifier set.
- `diagnostic-shape.md` transient-toast paragraph: add the cross-reference to PIC-14 so the toast surface's post-throw recovery posture has an owner.
- `code-registry-runtime.md`: register the new `loom/runtime/watcher-terminated` diagnostic row (`details: { path, code, classification }`, channel `loom-system-note`).

**Pros.** Closes the silent-staleness path. Reuses the existing drain-state `"degraded-needs-reload"` arm and `/reload` recovery vocabulary, so it composes with PIC without adding a fourth drain-state tag. Gives `V8e` a testable acceptance criterion (fire a synthetic chokidar `error` with `.code = "EMFILE"`; assert the system note and the drain-state transition).

**Cons.** Pins one new diagnostic code and three `.code` literals against chokidar's contract; the literals would be re-audited on each chokidar bump (route through the existing per-package audit carve-out in `implementation-notes.md`).

**Risks.** Mis-classifying a transient error as terminal would degrade the extension unnecessarily; the closed `.code` set bounds that risk and the operator recovery is `/reload`, not extension reload.

### Recommendation

Apply Option A first, then Option B. Option A is a one-paragraph edit in a single file with no plan-acceptance churn; landing it on the existing baseline narrows the diff Option B's `V8e` / `V7d` reviewers see. Option B is the substantive fix and is the source of the `correctness` severity. The orchestrator should run them as two separate fix-loop entries.

Edge cases for Option B's implementer:
- The terminal arm's drain-state transition MUST be idempotent — a watcher that emits two terminal errors (e.g. once per root) MUST NOT emit two `loom-system-note`s; deduplicate on `loom/runtime/watcher-terminated`'s `(code, details.path)` pair the same way other PIC diagnostics dedup.
- The `Unsubscribe` returned by `watch(...)` MUST remain a no-op after a terminal error (the watcher is already done); the existing "calling it twice is a no-op" rule covers this if the adapter records the terminal state.
- `getWatched()` is a chokidar surface, not a Pi surface; the classifier may need to call it from inside the adapter's own `error` handler. This is fine — PIC-14 already permits the adapter arbitrary internal logic — but the adapter MUST NOT expose `getWatched()` through the `FileWatcher` seam.

## Consequence

**Severity:** correctness

Part 1 alone is cosmetic / advisory — a passing implementation under the looser rule satisfies every observer. Part 2 is the correctness load: a chokidar `error` whose underlying cause silences one root for the lifetime of the extension means content edits there stop reaching the registry-swap path, slash dispatches keep returning pre-error content, and no diagnostic or system note tells the operator that `/reload` is required. The composite severity is therefore `correctness`, driven by part 2.
