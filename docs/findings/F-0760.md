## Finding

Step 5 of `registration-steps.md` pins the watcher debounce window at exactly `250 ms` and re-asserts the value once more in the same paragraph, accompanied by the explicit hardening clause that the seam "exists to make burst-coalescing assertions deterministic and is **not a tunable runtime knob**." The same `250 ms` figure is then re-cited normatively from at least four other sites: `discovery/package-and-settings.md` §*Caching and reload* (the settings-watcher debounce), `discovery/package-and-settings.md` §ERR-7 emission-timing note, `pi-integration-contract/host-interfaces-services.md` (twice, in the `Clock.setTimeout` bullet and the chokidar-seam responsibilities bullet), `implementation-notes.md` (the load-bearing call-sites enumeration), and `binder/binder-model-and-context.md` (the settings-file debounce window for the `looms.binderModel` recovery note).

What is missing across all of these sites is any link between the magnitude `250` and an observable property of the bursts the window is supposed to absorb. The closest the prose gets is a parenthetical that the window is "to coalesce editor `change` + `rename` bursts" — a *purpose* statement, not a sizing argument. No measurement of an editor's typical change+rename inter-event delay is cited; no upper-bound rationale (why not 100 ms, why not 500 ms) is given; no host pin (e.g. a chokidar `awaitWriteFinish` default the loom value tracks) is referenced. The "not a tunable runtime knob" clause hardens this further: implementers, reviewers, and future spec-bump editors are told the value is normative and immutable, but offered no anchor against which to defend it under future audit pressure or to adjust it should chokidar's coalescing behaviour or a Pi minor change the underlying timing surface.

The defect is one of presentation: the value is almost certainly fine in practice, but as written it appears as a magic number with normative force, which the corpus has elsewhere taught readers to mistrust (cf. the de-prescription pattern around `defineTool`, `Map<…>` container shapes, and lowerCamelCase field spellings). Either anchor the number, or demote the figure to non-normative and keep only the behavioural obligation ("a window long enough to coalesce a typical editor change+rename burst, measured against the injected `Clock` seam") at MUST level.

## Solution Space

**Shape:** multiple

### Option A — Anchor `250 ms` with a measured/cited rationale, keep it normative

**Approach.** Keep `250 ms` as the normative window at every site, but add a short rationale clause at the *defining* citation (step 5 of `registration-steps.md`) and rely on cross-references everywhere else. The rationale must cite at least one of: (a) a measured editor change+rename inter-event delay loom validated against, (b) the chokidar `awaitWriteFinish` default the value tracks, or (c) an explicit lower-bound argument (smaller risks splitting bursts) paired with an upper-bound argument (larger degrades author edit-loop latency).

**Spec edits.**
- In `registration-steps.md` step 5, replace the parenthetical with one extra sentence: e.g. "The 250 ms window is sized to absorb an editor's typical change+rename burst (chokidar's own `awaitWriteFinish.stabilityThreshold` ships a 2 s default for partial-write detection; the watcher does not enable `awaitWriteFinish`, so a smaller burst-coalescing window is layered on top) while keeping the author-edit-loop reload latency below a perceptible threshold." Or, if a measurement was actually taken, cite the test or capture.
- Keep the "not a tunable runtime knob" clause; it remains correct under this option.
- Other citation sites stay as they are; their `250 ms` references resolve back to the now-anchored defining site.

**Pros.**
- Preserves the existing test suite (`V9b-T`, `V10d-T` keep their exact `250` assertions through the `FakeClock` seam).
- Honours the "not a tunable runtime knob" pin without weakening any normative force.
- Keeps `cka-23.swap` / `cka-36` coverage rows unchanged.

**Cons.**
- Requires the spec author to actually have (or produce) the anchor — option only ships if the rationale is real, not invented post-hoc. If no measurement was taken, this option degenerates into a plausible-sounding citation that future reviewers will challenge again.
- Adds a small amount of bytes to the shard-06 page that is already near the soft size cap.

**Risks.**
- A future Pi or chokidar minor that changes the underlying timing surface re-opens the question, since the spec is now committed to a specific defensible number.

### Option B — Demote the magnitude to non-normative; promote the behavioural obligation

**Approach.** Keep the normative obligation that "the watcher debounces watcher events into a single coalesced re-parse / swap, measured against the injected `Clock` seam"; demote the specific `250 ms` figure to a non-normative implementation note (e.g. "the reference implementation uses a 250 ms window; this magnitude is not part of the contract"). Drop the "not a tunable runtime knob" clause along with it — once the magnitude is non-normative, the "not tunable" framing is also non-normative.

**Spec edits.**
- `registration-steps.md` step 5: replace "the watcher debounces (250 ms window, to coalesce …; the 250 ms window is measured against the runtime's injected `Clock` seam …, the seam exists to make burst-coalescing assertions deterministic and is not a tunable runtime knob)" with "the watcher debounces watcher events into a single coalesced re-parse / swap, measured against the runtime's injected `Clock` seam (per `Clock` / `FakeClock` interface), with a window long enough to absorb an editor's change+rename burst" plus a trailing non-normative note "The reference implementation uses 250 ms; the exact figure is not part of the contract."
- `discovery/package-and-settings.md` §*Caching and reload* and the ERR-7 note: same shape — drop the literal `250 ms` from the normative sentence, retain it as an illustrative figure.
- `host-interfaces-services.md` and `implementation-notes.md`: drop the literal `250 ms` from the load-bearing call-sites bullets; keep "the watcher debounce" / "the settings-watcher debounce" as the obligation.
- Tests (`V9b-T`, `V10d-T`) keep asserting that *N events in one window coalesce to one reload* and *one further event past the window produces a second reload*, but stop asserting the exact `250` constant — they advance the `FakeClock` by whichever value the production wiring chose plus epsilon.

**Pros.**
- Aligns with the corpus's existing de-prescription pattern (the `Map<…>` container, the `ActiveInvocationRegistry` field spellings, the `defineTool` removal): observable behaviour pinned, implementation magnitude not.
- Removes a defect surface entirely rather than papering it over; no future reviewer will re-flag a magic number that is no longer normative.
- Frees the implementation to retune (e.g. raise to 400 ms under a future chokidar event-storm bug) without a spec bump.

**Cons.**
- Touches more sites than Option A (six spec files + two test leaves).
- Weakens the deterministic-assertion guarantee very slightly: the FakeClock-driven coalescing test now binds to a runtime-supplied constant rather than a spec-frozen one. Mitigation: declare the constant on the production wiring module and import it into the test so the test still asserts "advance by `WATCHER_DEBOUNCE_MS + 1` produces exactly one reload."
- Modifies `V9b-T` / `V10d-T` *Tests* / *Ships when* bullets — minor plan churn.

### Recommendation

**Option B.** The corpus already de-prescribes implementation magnitudes that are not observable contracts (container shapes, field spellings, `defineTool`), and the `250 ms` figure is structurally the same class of pin: nothing the watcher's consumers can observe distinguishes 200 from 250 from 300. Option A only ships well if a real measurement exists; absent that, the anchor risks being a plausible-sounding back-rationalisation that re-opens the same finding on the next pass. The implementer-relevant edge case to watch: when collapsing the literal `250` out of the test bullets, route the constant through the production wiring (a single exported `WATCHER_DEBOUNCE_MS`) so the `FakeClock` test still asserts the coalescing/post-window-second-reload property deterministically — the *determinism* of the assertion was the point of the original "not a tunable runtime knob" clause and must survive the demotion.

Resolution order: edit the defining site (`registration-steps.md` step 5) first, then propagate the shape change to the five cross-citing sites in one pass, then update `V9b` / `V9b-T` / `V10d` / `V10d-T` to consume the shared constant. The settings-file debounce reference in `binder-model-and-context.md` is a cross-reference, not a self-citation, and falls out automatically once `discovery/package-and-settings.md` lands.

## Consequence

**Severity:** advisory

Implementers can ship the spec as written and the runtime will work; the gap is not a divergence risk between two reasonable implementers. The downside is durability: the value sits in the spec with normative force ("not a tunable runtime knob") but nothing to defend it on the next spec-review pass or under a Pi/chokidar bump that touches timing. The same lens flagged sibling magic-shape pins (`Map<schemaSlug, registeredToolName>`, `ActiveInvocationRegistry`'s container/field spellings) in the same shard, so the corpus owes its readers a consistent treatment.
