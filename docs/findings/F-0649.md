## Finding

Step 0 (d) of the capability probe enumerates exactly four observable conditions that route to `kind: "peer-dep-malformed-version"` (per `capability-probe.md` §**(d) Peer-dep version**):

1. `MODULE_NOT_FOUND` from the main-entry resolution call;
2. The parent-directory walk reaches the filesystem root without finding a matching `package.json`;
3. The located, parsed `package.json` object has no own `version` field;
4. A `version` field is present but is not a syntactically valid SemVer string.

The `<observed>` placeholder for this `kind` is pinned in two normative locations — the per-`kind` table in `placeholder-rendering-b.md` and the long `Trigger` cell in `code-registry-load.md` — both of which describe it as *"the raw `version` string (rendered per §8, host-derived), or the literal `"<unresolvable>"` when the read failed"*. That phrasing maps cleanly onto conditions (1), (2), and (4): conditions (1)/(2) are "read failed" → `"<unresolvable>"`; condition (4) is "raw `version` string present". Condition (3) — the `package.json` was read and parsed successfully but has no `version` key — falls into neither arm. The read did not fail, so `"<unresolvable>"` does not obviously apply; there is no raw `version` string to render, so the "raw string" arm cannot apply either. Two implementers will plausibly diverge: one emits `"<unresolvable>"`, another emits the empty string, another emits a sentinel such as `"<absent>"` or the literal text `"undefined"`. The wire value of `details.observed` on this row is therefore unspecified for one of the four normative trigger conditions.

The defect surfaces in the rendered `Message` (`host incompatible (peer-dep-malformed-version): observed <observed>, required <range>`) and, more importantly, in the structured `details.observed` payload that operator tooling keys on. The conformance test vector required by `V7c` cannot be written for condition (3) until the spec pins one value.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `126ff8a` (2026-05-06, "pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B"), `fa9f2d5` (2026-06-02, "pi-loom spec: resolve \"host-incompatible per-kind observed/required rendering\""), `9371309` (2026-06-05, "pi-loom spec: resolve \"foreseeable shape failure clause leaves kind discriminator under-specified\"")
**History:** Commit `126ff8a` introduced the `peer-dep-malformed-version` kind and its bivalent `<observed>` rule (`<raw-string-or-"<unresolvable>">`) atop a single-clause definition: "a `version` field that fails `semver.valid`, a missing `version` field, a `MODULE_NOT_FOUND` on the resolution call, or any other foreseeable shape failure of the read". At that point the bivalent mapping was coherent only by collapsing "missing `version` field" into the foreseeable-shape arm — the spec gave no explicit mapping. Commit `fa9f2d5` lifted the `<observed>` / `<required>` strings into the per-`kind` enumeration table in `diagnostics.md` (later split into `placeholder-rendering-b.md` by the file-size split `f5e89f4`) and re-stated the bivalent rule as "the raw `version` string ... or the literal `\"<unresolvable>\"` when the read failed", inheriting the latent ambiguity. Commit `9371309` then resolved a sibling finding by replacing the catch-all "foreseeable shape failure" prose in `capability-probe.md` with the explicit four-condition enumeration above — which finally made condition (3) (`version` field absent on a successful read) a first-class trigger but did not update the `placeholder-rendering-b.md` table or the `code-registry-load.md` `Trigger` cell, both of which retain the "when the read failed" phrasing. The defect is the *interaction*: the bivalent rendering rule and the four-condition trigger now coexist without a mapping from condition (3) to either arm.

## Solution Space

**Shape:** single

### Recommendation

Pin condition (3) (`version` field absent on a successful read) to `details.observed = "<unresolvable>"` in both diagnostics-side tables, and broaden the table's prose so the discriminator names the *unreadable-version* condition rather than the *unreadable-file* condition.

Concretely:

- In `docs/spec_topics/diagnostics/placeholder-rendering-b.md`, change the `peer-dep-malformed-version` row's `<observed>` cell from

  > the raw `version` string (rendered per §8, host-derived), or the literal `"<unresolvable>"` when the read failed

  to

  > the raw `version` string (rendered per §8, host-derived) when condition (4) fired (a `version` field is present but is not a syntactically valid SemVer string); otherwise — conditions (1), (2), and (3) — the literal `"<unresolvable>"`. The four conditions are enumerated at [Step 0 (d)](../pi-integration-contract/capability-probe.md#entry-capability-probe).

- Apply the matching edit to the `peer-dep-malformed-version` clause in `docs/spec_topics/diagnostics/code-registry-load.md`'s `loom/load/host-incompatible` `Trigger` cell, so the bivalent description there cites the same four-condition partition.

- Add one test vector to the `V7c` plan leaf covering condition (3): `package.json` reads as `{ "name": "@earendil-works/pi-coding-agent" }`, expected rendered `Message` matches `^host incompatible \(peer-dep-malformed-version\): observed <unresolvable>, required ~.*$`, expected `details.observed === "<unresolvable>"`.

`"<unresolvable>"` is the right sentinel because (a) it is already in the closed value set for this row, (b) it correctly conveys "no SemVer string is available to compare against the pin range", and (c) the operator-side triage path documented in the `Hint` column ("the host's Pi installation is broken … workspace symlink, git-pinned dev install, corrupt `package.json`") covers all three of conditions (1)/(2)/(3) uniformly. Coining a new third sentinel for condition (3) (e.g. `"<absent>"`) would grow the closed rendering surface, force a sibling update to category 7's identifier/closed-enum sub-table, and offer the operator no additional triage information that `details.cause` / `details.package` does not already carry. Implementer edge cases: the four-condition partition is mutually exclusive by construction (each condition is a distinct failure point in the read pipeline), so the renderer needs no priority rule; condition (3) requires that the `package.json` be parsed successfully *and* the parsed top-level value be an object (per the `Self-failure` rule in `capability-probe.md`, a parsed non-object routes to `probe-failed`, not here), so the "no own `version` field" check is unambiguous.

## Consequence

**Severity:** correctness

Two conformant capability-probe implementations will emit different `details.observed` values when a peer package's `package.json` parses but lacks a `version` field — a perfectly plausible host condition (workspace symlink, hand-edited manifest, partially published package). Operator tooling that switches on `details.observed === "<unresolvable>"` to distinguish "couldn't read it" from "read it but value is unusable" will mis-route condition (3) on at least one of the two implementations, and the `V7c` byte-identical rendering test cannot be authored for this row until the value is pinned.
