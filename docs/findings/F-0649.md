## Problem

The `peer-dep-malformed-version` `<observed>` rendering is pinned in two normative sources — the per-`kind` enumeration table in `placeholder-rendering-b.md` (anchor `id="host-incompatible-observed-required"`) and the `loom/load/host-incompatible` `Trigger` cell in `code-registry-load.md` — both describing it as the raw `version` string or the literal `"<unresolvable>"` when the read failed. That bivalent rule maps three of the four trigger conditions enumerated at Step 0 (d) in `capability-probe.md` (`#entry-capability-probe`), but leaves condition (3) — a `package.json` parsed successfully yet carrying no own `version` field — mapped to neither arm: the read did not fail, and there is no raw `version` string to render. Conformant implementations therefore diverge on the wire value of `details.observed` for that condition, and the `V7c` byte-identical rendering test cannot be authored for it.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `126ff8a` (2026-05-06, "pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B"), `fa9f2d5` (2026-06-02, "pi-loom spec: resolve \"host-incompatible per-kind observed/required rendering\""), `9371309` (2026-06-05, "pi-loom spec: resolve \"foreseeable shape failure clause leaves kind discriminator under-specified\"")
**History:** Commit `126ff8a` introduced the `peer-dep-malformed-version` kind and its bivalent `<observed>` rule (`<raw-string-or-"<unresolvable>">`) atop a single-clause definition: "a `version` field that fails `semver.valid`, a missing `version` field, a `MODULE_NOT_FOUND` on the resolution call, or any other foreseeable shape failure of the read". At that point the bivalent mapping was coherent only by collapsing "missing `version` field" into the foreseeable-shape arm — the spec gave no explicit mapping. Commit `fa9f2d5` lifted the `<observed>` / `<required>` strings into the per-`kind` enumeration table in `diagnostics.md` (later split into `placeholder-rendering-b.md` by the file-size split `f5e89f4`) and re-stated the bivalent rule as "the raw `version` string ... or the literal `\"<unresolvable>\"` when the read failed", inheriting the latent ambiguity. Commit `9371309` then resolved a sibling finding by replacing the catch-all "foreseeable shape failure" prose in `capability-probe.md` with the explicit four-condition enumeration above — which finally made condition (3) (`version` field absent on a successful read) a first-class trigger but did not update the `placeholder-rendering-b.md` table or the `code-registry-load.md` `Trigger` cell, both of which retain the "when the read failed" phrasing. The defect is the *interaction*: the bivalent rendering rule and the four-condition trigger now coexist without a mapping from condition (3) to either arm.

## Solution approach

Clarify the `peer-dep-malformed-version` `<observed>` rule in both normative sources — the per-`kind` enumeration row in `placeholder-rendering-b.md` (`id="host-incompatible-observed-required"`) and the `loom/load/host-incompatible` `Trigger` cell in `code-registry-load.md` — so that all four Step 0 (d) conditions, including condition (3) (`version` absent on a successful read), map to a defined rendered `details.observed` value, keeping the two sources mutually consistent and aligned with the four-condition enumeration in `capability-probe.md`. Add a `V7c` test vector covering condition (3).

## Solution constraints

- The condition-(3) rendered value MUST be drawn from this row's existing closed `<observed>` set; coining a new sentinel widens a closed enum (a GOV-7/GOV-8 breaking change requiring a sibling update to category 7's closed-enum sub-table) and is out of scope for this fix.
