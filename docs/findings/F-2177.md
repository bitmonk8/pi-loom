## Problem

The *Strict-capability absence-under-the-probed-name arm* (`inventory-audit-intro.md#strict-capability-absence-under-probed-name`) extends the step-2(a) `strict-capability-probe` literal-read assertion to fail "if any **reachable** `Model<Api>` declaration at the model-registry surface exposes a `strictCapable` member under the probed name." The same "reachable `Model<Api>` declaration" phrasing recurs in the assertion's own definition at `version-bump-step2.md#bump-step-2-positive` and in the prose at `audit-target-categories.md#strict-capability-absence-pin`, but nowhere in the audit cluster — nor in the glossary — is "reachable" given a boundary in this context. The assertion's input set is therefore undecidable: it admits a single declaration resolved through the package's type exports, every declaration the TypeScript checker can reach transitively, or every declaration in some file set, and these readings are not equivalent. Two implementers writing the assertion and its negative fixtures cannot converge on the same pass/fail criterion, and the transitive reading additionally demands a whole-program type load the cluster's tooling floor declines to require.

## Issue introduction
**Verdict:** single-commit
**Introducing commit:** `6829a79` (2026-06-06, "pi-loom spec: resolve \"Pi-SDK hidden-assumption cluster (strictCapable absence, decode steps, pi.on origin)\"")
**History:** `git log -S'reachable \`Model<Api>\` declaration' -- docs/spec_topics/` returns `6829a79` as the oldest commit introducing the phrase; `git show 6829a79~1:…/binder-model-and-context.md` confirms it was absent immediately before. That commit authored the strict-capability absence-under-the-probed-name arm and coined the gate with the bare word "reachable" and no scan-boundary definition (its own finding-store entry had even recommended "Cite the actual `.d.ts` declaration site for `Model<Api>`", which the landed text did not do). Later commits `23d28a1` (2026-06-22, audit-cluster basename realignment) and `bf92fe6` (2026-06-25, relocation of the absence-arm MUST into `inventory-audit-intro.md`) moved and re-anchored the paragraph but carried the undefined "reachable" through verbatim. The defect is therefore inherent to the arm as first authored, not a product of the later relocations.

## Solution approach

Define what "reachable `Model<Api>` declaration" denotes for this gate, pinning it to a closed input set anchored at the single `Model<TApi>` interface declaration in `@earendil-works/pi-ai`'s `dist/types.d.ts` — the declaration already cited by *Binder-model matcher inputs* at `host-interfaces-core.md#binder-model-matcher-inputs` and pinned at `host-interfaces-core.md#model-registry-pin`. State that boundary once and reference it from the two other sites carrying the same phrasing (`version-bump-step2.md#bump-step-2-positive` and `audit-target-categories.md#strict-capability-absence-pin`) rather than restating it divergently.

## Solution constraints

- The pinned definition MUST NOT widen to require a whole-program TypeScript load — that contradicts the cluster's stated tooling floor (the inventory-closure audit's tooling-choice carve-out and step 2(a)'s category-(3) contributor-discipline note).
