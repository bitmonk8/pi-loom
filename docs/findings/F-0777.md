## Problem

Under **Deduplication and lifetime rules**, the *Tool-host emission ownership* bullet closes with `Renderers MUST NOT cross-correlate the two.` "Cross-correlate" is undefined anywhere in the spec corpus, and the rule names no observable artefact whose presence or absence would falsify it. The neighbouring bullets in the same list are observable — exactly-once emission at the origin, byte-identical cascade re-emission, the `(kind, query_site, message, occurred_at)` dedup tuple — so this non-observable MUST NOT sits awkwardly inside an otherwise testable rule set and is unreachable from any conformance fixture.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `b3bc4ce` (spec: pin failure-observability surface (Cluster A, Option C), 2026-05-04) — added the *Tool-host emission ownership* bullet, with the wire kinds named as `tool_call_error` and `tool_failure` at the time, inside a then-monolithic `pi-integration-contract.md`. The defect (a non-observable MUST NOT) is in the bullet as authored.
**History:** `git log --all -S 'cross-correlate' -- docs/` returns `b3bc4ce` as the first-introducing commit. `git show b3bc4ce` confirms the bullet was added with its current wording — the only material change since has been the wire-kind rename in `6750c81` (spec: consolidate QueryError variants; rename to InvokeInfraError / CodeToolError / ModelToolError, 2026-05-05) which mechanically swapped `tool_call_error`/`tool_failure` → `code_tool`/`model_tool`, and the spec-set split in `f5e89f4` (docs: enforce 100KB hard / 30KB recommended size cap on spec set) which carved the bullet out into the per-topic `runtime-event-channel.md` file unchanged. Neither follow-up touched the MUST NOT's observability. The bullet has been non-observable from the moment it entered the corpus; the testability gap is not the result of a later edit eroding an earlier observable form.

## Solution approach

Rewrite the trailing `Renderers MUST NOT cross-correlate the two.` sentence of the *Tool-host emission ownership* bullet so the obligation lands on the `Component` the `loom-system-note` renderer returns for a `code_tool` / `model_tool` `details: { event }` payload — an observable self-containment rule (no rendered link, parent/child grouping, or textual back-reference to a tool-host-internal diagnostic) in place of the undefined "cross-correlate". Keep the bullet's first ownership sentence intact.

## Solution constraints

- The rewritten rule's scope MUST stay on cross-correlation with a tool-host-internal diagnostic; it MUST NOT forbid legitimate grouping of two loom-side events sharing an `invocation_id`.
