## Finding

Under **Deduplication and lifetime rules**, the *Tool-host emission ownership* bullet closes with a normative prohibition:

> The loom runtime owns `code_tool` and `model_tool` events; the tool host owns whatever it emits internally. **Renderers MUST NOT cross-correlate the two.**

"Cross-correlate" is not defined anywhere in the spec corpus, and the rule states no observable artefact whose presence or absence would falsify it. A conforming renderer's output is a sequence of `Component`s rendered for `loom-system-note` payloads (per `PIC-21` and the renderer-registration block in `extension-bootstrap-and-per-loom.md`); from that observable surface a reviewer cannot decide whether two unrelated displayed items constitute an internal correlation between a loom-runtime `code_tool`/`model_tool` event and a tool-host-owned diagnostic, or are merely adjacent transcript entries. The neighbouring bullets in the same list are observable — exactly-once emission at the origin, byte-identical cascade re-emission, `(kind, query_site, message, occurred_at)` dedup tuple — so a non-observable MUST NOT sits awkwardly inside an otherwise testable rule set and is unreachable from any conformance fixture in `V9d` / `V9d-T`.

The underlying concern the bullet is reaching for is sound: a renderer that fabricated a shared container, parent/child relation, or back-reference between a loom-runtime `code_tool`/`model_tool` event and a tool-host-internal diagnostic would misrepresent the ownership split. That concern is expressible as an observable rule about the rendered `Component` for a single `loom-system-note` payload; the current MUST NOT is not.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `b3bc4ce` (spec: pin failure-observability surface (Cluster A, Option C), 2026-05-04) — added the *Tool-host emission ownership* bullet, with the wire kinds named as `tool_call_error` and `tool_failure` at the time, inside a then-monolithic `pi-integration-contract.md`. The defect (a non-observable MUST NOT) is in the bullet as authored.
**History:** `git log --all -S 'cross-correlate' -- docs/` returns `b3bc4ce` as the first-introducing commit. `git show b3bc4ce` confirms the bullet was added with its current wording — the only material change since has been the wire-kind rename in `6750c81` (spec: consolidate QueryError variants; rename to InvokeInfraError / CodeToolError / ModelToolError, 2026-05-05) which mechanically swapped `tool_call_error`/`tool_failure` → `code_tool`/`model_tool`, and the spec-set split in `f5e89f4` (docs: enforce 100KB hard / 30KB recommended size cap on spec set) which carved the bullet out into the per-topic `runtime-event-channel.md` file unchanged. Neither follow-up touched the MUST NOT's observability. The bullet has been non-observable from the moment it entered the corpus; the testability gap is not the result of a later edit eroding an earlier observable form.

## Solution Space

**Shape:** multiple

### Option A — Replace with an observable per-Component rule

**Approach.** Rewrite the trailing sentence so the obligation lands on the rendered `Component` for a single `code_tool` / `model_tool` `loom-system-note`: that `Component` MUST NOT include a shared container, parent/child relation, or back-reference linking the loom-runtime event to a tool-host-internal diagnostic. Keep the ownership statement (the first sentence of the bullet) verbatim.

**Spec edits.**
- In `runtime-event-channel.md`, in the *Tool-host emission ownership* bullet, replace `Renderers MUST NOT cross-correlate the two.` with text along the lines of: *The `Component` returned by the `loom-system-note` renderer for a `code_tool` or `model_tool` `details: { event }` payload MUST be self-contained — it MUST NOT include any rendered link, parent / child grouping, or textual back-reference to a tool-host-internal diagnostic. Tool-host-internal diagnostics, when present, render through whatever channel the host owns; the loom-side `Component` neither references nor groups them.*
- Add one conformance test to `V9d-T` / `V9d`'s test list: a fixture emitting a `code_tool` event (and separately a `model_tool` event) asserts the rendered `Component` contains no reference to any sibling Pi-tool-internal diagnostic identifier.

**Pros.** Preserves the bullet's intent; observable; reds an actual fixture; stays inside the renderer's own contract surface (`PIC-21` neighbourhood); no cross-topic moves.

**Cons.** Adds a fixture obligation to `V9d` / `V9d-T`. Slight risk of over-constraining a future renderer that legitimately wants to group consecutive tool events for visual readability — the rule must be scoped to *cross-correlation with a tool-host-internal diagnostic*, not to *any grouping*.

**Risks.** If the wording is loose, a reviewer could read it as banning ordinary same-`invocation_id` grouping of two loom-runtime events; the rewrite must keep "tool-host-internal diagnostic" as the load-bearing referent.

### Option B — Demote to non-normative guidance

**Approach.** Strip the MUST NOT and recast the closing sentence as a non-normative note that records the ownership intent without binding the renderer.

**Spec edits.**
- In `runtime-event-channel.md`, in the *Tool-host emission ownership* bullet, replace the closing sentence with: *(Non-normative.) Because ownership is split, renderers should not present the loom-side event and a tool-host-internal diagnostic as parts of a single failure record.*
- No plan-side change beyond removing the bullet from the testable surface (the leaves do not currently carry a test for it, so this is a no-op for `V9d` / `V9d-T` other than the test-list audit).

**Pros.** Smallest edit; honest about the observability gap; no new fixtures; the wire-level ownership split (group A / group B partition; dedup tuple; `code_tool` / `model_tool` ownership) carries the load on its own.

**Cons.** Loses the prohibition: a renderer that fabricated a shared container would not red any conformance test. The current bullet does not red one either, so this is a documentation-only loss, but it removes the ambient pressure on renderer design.

**Risks.** A future reader may reintroduce the MUST NOT thinking it was an accidental demotion; the non-normative framing must be explicit ("Non-normative." prefix).

### Recommendation

Take **Option A**. The bullet is in the testable section of `runtime-event-channel.md`, sits next to other observable rules, and is already plan-anchored to `V9d` / `V9d-T`; the cheap path is to give it an observable form rather than abandon the prohibition. Edge cases the implementer must watch: (i) keep the scope of the rule on *cross-correlation with a tool-host-internal diagnostic*, not on legitimate grouping of two loom-side events sharing an `invocation_id`; (ii) the rewrite stays compatible with `PIC-21`'s exception-safety contract — the renderer's failure mode on an internal throw is unchanged.

## Consequence

**Severity:** advisory

Two reasonable implementers will not diverge on user-observable runtime behaviour because of this bullet — the loom runtime emits exactly the `code_tool` / `model_tool` events the rest of the section pins, and the renderer's `Component` output is already constrained by `PIC-21` and the per-variant `display` / `content` matrix. What the bullet does is consume normative weight without contributing a falsifiable rule: a reader looking for conformance criteria sees a MUST NOT that no fixture in `V9d-T` can red, and a renderer implementer cannot derive a design rule from it. Left as is, the line is a small but durable defect in the testability discipline the rest of the section maintains.
