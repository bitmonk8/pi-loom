## Problem

The `loom/parse/empty-template` "Parse-time warning" bullet in `query-forms.md` §"Degenerate rendered templates" (anchor `#degenerate-rendered-templates`) defines the warning to fire when a template's static body is empty or whitespace-only, but never states whether the emptiness/whitespace predicate runs against the source-level literal segments or against the post-escape segments after `query-escapes-stringification.md` rewrites (`` \n `` → U+000A, etc.). The two readings diverge on the exact example the same bullet names as the suppression hatch, `` @`\n` ``: pre-escape, the body is the non-whitespace two-character sequence and the warning is suppressed; post-escape, the body is a single whitespace newline and the warning fires unsuppressably. Conforming implementers reading the one paragraph will therefore produce observably divergent parsers for the canonical suppression vector.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `82be125` ("spec: adopt P1 typed-query two-phase loop (Option A)", 2026-05-04)
**History:** Commit `82be125` is the first commit to add a "Degenerate rendered templates" section anywhere in the spec corpus (`git log -S 'explicit literal escape' --reverse`, `git log --diff-filter=A -- docs/spec_topics/query/query-forms.md`, and a `git log -L` walk of the section all bottom out at this commit). The introduced text already contained both halves of the contradiction in their current form: the "static body … after newline-trim and dedent" predicate and the "suppress … by writing an explicit literal escape (`` \n ``)" hatch, with no statement of order relative to escape processing. The later commit `f14a45e` pinned the *whitespace set* (ASCII vs `\s`) but did not touch the pre-/post-escape order. The defect has been present since the section was first authored.

## Solution approach

Clarify the "Parse-time warning" bullet so the empty/whitespace predicate's input is pinned to the source-level literal segments — evaluated before the escape rewrites of `query-escapes-stringification.md` are applied (newline-trim and dedent still apply to the surrounding template first). The pre-escape order is the only reading internally consistent with the bullet's own `` @`\n` `` suppression hatch, so name it explicitly.

## Solution constraints

- The runtime short-circuit's input is unchanged: it remains the fully-rendered (post-interpolation, post-newline-trim, post-dedent, post-escape) text. Only the parse-time predicate's input is pinned.
