## Problem

The `in-loop` glossary entry specifies normative destination/surface routing for ceiling #4's model-driven tool-call args row: that the firing is not re-surfaced to loom code (no `Err` flows), that no `loom-system-note` is rendered, and that both surfaces become reachable only on later ceiling-#2 exhaustion where `tool_loop_exhausted` flows. That routing law belongs on the canonical hard-ceilings page, not in a glossary entry whose charter promises a descriptive one-paragraph reminder with the canonical page as tie-break authority. The duplication is a maintenance hazard: if the rule later changes on only one side, the glossary asserts stale law that an implementer may read as authoritative.

**History:** The descriptive-only charter ("Each entry is descriptive â€” a one-paragraph reminder â€¦ the canonical page wins") was established at the glossary's inception in `b4fbfac` ("resolve 'no central glossary'"). ## Solution approach

Narrow the `in-loop` entry in `docs/spec_topics/glossary.md` to a definitional stub that names the concept (a hard-ceiling enforcement point that surfaces inside the tool-call loop as a tool-result fed back to the model rather than as an `Err` or `loom-system-note`) and forward-links to `./hard-ceilings/ceilings-3-and-4.md#ceiling-4-table`. Delete the firing/exhaustion routing prose from the entry; it is already carried by the canonical page's `#ceiling-4-table-reconciliation` paragraph.
