## Finding

The **Unknown-reason rule** introduces two handler-local variables by identifier — `capturedEventReason` and `pinnedConstantReadOk` — and proceeds to use those exact names as cross-document binding handles. `capturedEventReason` is pre-initialised to `"<unreadable>"`; `pinnedConstantReadOk` is pre-initialised to `false` and set to `true` on the success arm of the snapshot lookup. Both names are then consumed by name in adjacent normative prose: the *Predicate split* table in `session-only-degraded-state.md` quotes the literal expressions `pinnedConstantReadOk && capturedEventReason ∈ {"new","resume","fork"}` and `capturedEventReason ∉ {"quit","reload"}` as the predicates that govern the `runtime-degraded` emission and the `markRuntimeDegraded` tag write; `drain-state-contract.md` quotes the same `capturedEventReason ∉ {"quit", "reload"}` expression as the tag-transition trigger; and `version-bump-triggers.md` lists `capturedEventReason` in its `event.reason|SessionShutdownEvent|capturedEventReason|entry.shutdownReason` grep-recipe anchor set as a required string to grep for during a bump.

The observable contract is behavioural — a `"<unreadable>"` default for the captured shutdown reason, a boolean flag recording whether the snapshot lookup-and-`literals` read succeeded, and predicates over those two pieces of state. The identifiers `capturedEventReason` and `pinnedConstantReadOk` are private to whichever module ends up implementing the handler; nothing outside the runtime sees them. By naming them in spec prose and then re-quoting them across three sibling PIC pages, the spec elevates internal-variable spellings into contract surface: a perfectly conformant implementation that names the variables `lastReason` and `snapshotOk` would now diverge from spec text rather than from spec semantics, and a future rename inside the handler module triggers a multi-file inbound-reference sweep that produces no behavioural change.

The defect is the same shape as the sibling `active-invocation-registry.md` finding (internal container / field spellings pinned where only wire-side fields and insertion-order iteration are observable) and the `drain-state-contract.md` `drainStateTag` vs `tag` naming finding: the runtime's internal naming has leaked into the contract because the names happened to be convenient cross-page anchors.

## Solution Space

**Shape:** single

### Recommendation

Demote the two handler-local identifiers from spec-pinned names to spec-described roles, in the same edit across the cited files:

1. In `unknown-reason-rule.md`'s pre-init clause, replace the two named-variable introductions with role descriptions — e.g. "the handler-scoped *captured-reason slot*, pre-initialised to the literal string `"<unreadable>"`" and "the handler-scoped *snapshot-read-succeeded flag*, pre-initialised to `false` and set to `true` on the success arm of the snapshot lookup-and-`literals` read." Keep the behavioural pins (the literal `"<unreadable>"` default, the pre-init-before-either-read ordering, the success-arm transition, the unchanged-on-snapshot-failure invariant) verbatim — they are the contract.
2. In `session-only-degraded-state.md`'s *Predicate split* table, restate the two predicates by referring to the roles rather than the identifiers — e.g. "the snapshot-read-succeeded flag is set **and** the captured-reason slot holds a member of `{"new","resume","fork"}`" for the narrow predicate, and "the captured-reason slot's value is not in `{"quit","reload"}`" for the broad predicate. The *Seam minimality* clause's "no third seam beyond these two" prohibition is preserved by referring to the two roles, not the two names. Acceptance branches (d), (e), and the sub-step-2 stamp note in *Co-emission rules* similarly switch from `pinnedConstantReadOk = true / false` to "snapshot-read-succeeded flag set / unset."
3. In `drain-state-contract.md`'s `LoomRegistry.markRuntimeDegraded` paragraph, restate the tag-transition predicate in role terms.
4. In `version-bump-triggers.md`, drop `capturedEventReason` from the second-stage grep anchor list and re-derive the anchor set as the source-side reference shapes for `SessionShutdownEvent['reason']` values *without* committing to a specific handler-scoped capture name. The closed-enumeration discipline the paragraph imposes is preserved — what changes is that the enumeration documents "the handler-scoped capture site of the session-shutdown reason, whatever the implementation has named it" instead of grepping for a literal token the spec invented. The leaf-owned audit pass the paragraph permits then keys on observable shapes (the per-entry `shutdownReason` stamp site, the `details.event.reason` payload site) rather than on a private identifier.
5. In the three V9h / V9l plan-leaf **Spec** fields' "normative-cross-link closure" notes, replace "the unknown-reason rule's `pinnedConstantReadOk` seam" with "the unknown-reason rule's snapshot-read-succeeded flag seam" (or equivalent role phrasing).

Edge cases the implementer must watch:

- The roles MUST stay distinguishable as two separate seams — the *Seam minimality* clause's prohibition on a third seam keys on the two-piece state model, not on the two literal names, and the restated predicates must continue to reference exactly those two pieces.
- The `version-bump-triggers.md` rewrite is the load-bearing piece: that page currently uses `capturedEventReason` as a contributor-facing grep token, so demoting the name without re-grounding the grep recipe would weaken the inbound-reference sweep. Re-ground the recipe on the per-entry `shutdownReason` field and the `details.event.reason` payload, both of which are observable, before dropping `capturedEventReason`.
- Preserve the `pinnedConstantReadOk`-as-narrow-predicate-first-conjunct argument the *Seam minimality* clause makes (it pins the snapshot-failure-with-closed-set-`event.reason` exclusion against an alternative implementation that re-reads `event.reason` at the discriminator site). That argument is about *what state the predicate consumes*, not *what the variable is called* — it survives the rename in role form.

## Consequence

**Severity:** advisory

A conformant implementation that uses different identifier names diverges from spec text without diverging from spec semantics, and a contributor renaming the variables inside the handler now triggers a multi-page inbound-reference sweep that produces no behavioural change. The misalignment also locks in a contributor-side trap on the next Pi version bump: `version-bump-triggers.md`'s grep recipe treats the literal token `capturedEventReason` as a required search anchor, so an implementation that picked a different name would silently miss its own emission site during the bump audit.
