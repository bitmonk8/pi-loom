## Finding

The Parse-time warning bullet under `### Degenerate rendered templates` ends with:

> "The loom still loads. Authors who genuinely intend a whitespace-only prompt can suppress the warning by writing an explicit literal escape (`` \n ``): evaluating the predicate pre-escape, the body is the non-whitespace two-character sequence `` \n `` and the warning is suppressed (post-escape it would be a single whitespace newline, so only the pre-escape reading makes this suppression hatch effective)."

Read in isolation the phrase "authors who genuinely intend a whitespace-only prompt can suppress the warning…" frames the escape sequence as a working **escape hatch for sending a whitespace prompt** — i.e. a way to opt out of the empty-template defence and have the runtime issue the user turn with a `\n`-only body. That is not what happens. The very next bullet (Runtime short-circuit) defines a second, independent defence that inspects the *fully-rendered post-escape* text and short-circuits to `Err(QueryError { kind: "validation", cause: "empty_template", … })` whenever that rendered text is empty or ASCII-whitespace-only. A template whose only static content is the literal escape `\n` passes the parse-time predicate (which is evaluated pre-escape and so sees the two-character sequence `` \n ``) but fails the runtime predicate (which sees a single newline). The author still cannot send a whitespace-only prompt — the suppression only silences the parse warning, not the runtime refusal.

The fix is purely a re-wording of the bullet to identify the population the hatch actually serves (templates whose **static body is whitespace-only but whose rendered body will be non-empty** — typically because runtime interpolations contribute the substantive content), and to make explicit that the hatch is a warning suppression only and does not bypass the runtime short-circuit.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the trailing sentence of the Parse-time warning bullet to (a) name the actual beneficiary population — templates whose static body is whitespace-only but whose interpolations will contribute the substantive rendered content — and (b) state that the hatch only suppresses the parse warning, not the runtime short-circuit.

Concretely, replace the current trailing sentence:

> "Authors who genuinely intend a whitespace-only prompt can suppress the warning by writing an explicit literal escape (`\n`): evaluating the predicate pre-escape, the body is the non-whitespace two-character sequence `` \n `` and the warning is suppressed (post-escape it would be a single whitespace newline, so only the pre-escape reading makes this suppression hatch effective)."

with wording along these lines:

> "Authors whose **rendered** text will be non-empty (because runtime interpolations contribute the substantive content) but whose **static body** is whitespace-only can suppress the parse-time warning by writing an explicit literal escape (`\n`) in place of the literal whitespace: evaluating the predicate pre-escape, the static body is the non-whitespace two-character sequence `` \n `` and the warning is suppressed (post-escape it would be a single whitespace newline, so only the pre-escape reading makes this suppression hatch effective). The hatch suppresses the parse-time warning only; the runtime short-circuit (next bullet) still fires if the *fully-rendered* text is empty or whitespace-only, so this hatch cannot be used to dispatch a whitespace-only user turn."

Edge cases for the implementer of the fix:

- Keep the existing pre-escape predicate explanation intact — it is the technical correctness anchor added by commit `2637aec` and resolves a separate plan-review finding (F-0755). Only the audience-framing sentence and the closing disclaimer change.
- Do not add a second short-circuit cross-reference link target; an in-paragraph "(next bullet)" pointer is sufficient and avoids anchor churn.
- The bullet must remain a single paragraph (the section's bullets are siblings in a two-item list); do not promote any of the new clauses into nested sub-bullets.

## Consequence

**Severity:** advisory

Two reasonable implementers will not diverge on behaviour — both the parse-time and runtime predicates are pinned tightly in the surrounding sentences — but an implementer skim-reading the bullet may write a conformance test that asserts a whitespace-only turn is dispatched once `\n` is used, then be confused when the runtime short-circuit fires and the test fails. A spec author or reviewer reading the bullet for intent may also mis-summarise it elsewhere (the bullet sits adjacent to the runtime-short-circuit description, so the contradiction is locally detectable, but a cross-reference from another file that quotes only the parse-time bullet would carry the mis-statement forward).
