## Finding

`V1b`'s **Adds** promises "Quoted-string lexing with the full escape table (including `\u{…}`)" and its **Ships when** states `npm test` "lexes the escape table". The spec escape table (`lexical.md` §String literals) enumerates `\"`, `\'`, `\\`, `\n`, `\t`, `\r`, and `\u{XXXX}`. Of these, only `\u{…}` has a positive decode assertion: the bullet "[lexical.md — String literals] (LEX code-keyed area): a recognised `\u{…}` escape decodes to the correct Unicode scalar value." The remaining six entries (`\"`, `\'`, `\\`, `\n`, `\t`, `\r`) have no asserted decode result.

The other escape-related bullets are negative tests — `loom/parse/illegal-escape` (unrecognised escape fires) and `loom/parse/invalid-unicode-escape` (out-of-range/surrogate `\u{…}` fires). Neither demonstrates that the recognised non-Unicode escapes decode to their correct values. The gate therefore greens while the leaf's prose claims the whole table is verified.

The same Tests bullets appear in the authoritative paired tests leaf `V1b-T`, so the coverage gap and the overclaiming prose exist in both files.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e — pi-loom plan: build/update plan for spec.md + review (2026-06-10, Thomas Andersen)
**History:** The leaf's first commit (c6a664e) already carried the "full escape table" Adds claim, the "lexes the escape table" Ships-when, and a single positive-decode bullet limited to `\u{…}` ("String escape errors fire at the offending span; `\u{…}` decodes correctly"). A later same-day commit (b053e61) split that bullet into separate error-detection and decode bullets but preserved the single `\u{…}`-only positive decode, so the gap was present from inception and never closed.

## Solution Space

**Shape:** single

### Recommendation

Close the gap on the test side: in `V1b`'s **Tests** and the paired `V1b-T`'s **Tests**, add positive decode assertions covering the remaining escape-table entries the spec enumerates — `\"`, `\'`, `\\`, `\n`, `\t`, `\r` — asserting each decodes to its correct character (mirroring the existing `\u{…}` decode bullet). The escape set is fixed and small; take it from `lexical.md` §String literals, which is the authoritative source and is read-only for this fix (do not edit the spec to satisfy the leaf).

If full per-entry coverage is not wanted, the alternative is to narrow the prose instead: change `V1b`'s Adds "the full escape table" and Ships-when "lexes the escape table" to scope only the entries actually asserted, so the claim matches the coverage. Adding the vectors is preferred because the table is small and fixed and the broader claim is the one the spec supports.

## Consequence

**Severity:** advisory

A decode bug in any of `\"`, `\'`, `\\`, `\n`, `\t`, or `\r` ships green: the test suite asserts nothing about their decoded values while the leaf's Adds/Ships-when assert the full table is exercised. Two implementers reading the same leaf could reasonably test all entries or only `\u{…}`, and the wording invites the narrower reading.
