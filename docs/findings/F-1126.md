## Finding

Two independent gaps in `audit-wire-and-canary.md`, one per sub-paragraph of the audit's "fail-closed" surface:

1. **Wall-clock budget bound is missing.** The *Wall-clock budget* clause inside the *Infrastructure-failure handling* paragraph reads: *"The audit MUST enforce a wall-clock budget on its own execution (the implementation pins the budget value, e.g. 60 s); on budget expiry the audit MUST emit one record under a stable `<symptom>` token in the infrastructure-failure family and exit non-zero."* The MUST is normative but the budget value is delegated wholesale to the implementation with only a parenthetical "e.g. 60 s" as guidance. A conformant implementation that pins the budget at 1 ms satisfies the MUST while making every real audit run trip the timeout; a conformant implementation that pins it at 86 400 s satisfies the MUST while neutralising the fail-closed property the clause exists to provide (the outer `npm test`-side timeout the *Pre-emission termination carve-out* defers to will fire first and remove the routable infrastructure-failure record). Either extreme passes the spec, so the "fail-closed bound" the section depends on is unenforceable at the spec level. The same problem applies to the `npm test`-side timeout the carve-out names, with identical unboundedness.

2. **Green-path canary counter key names are not portable.** The *Green-path emission contract* clause specifies that the canary record's `proposed-resolution` field MUST carry the two counter values as the plain-ASCII key=value pair set `walked=<N> recognised=<M>` — *"or an equivalent implementation-chosen plain-ASCII key=value pair set whose pair names identify the same two counters this paragraph defines."* A CI parser that asserts `walked=` / `recognised=` works on implementation A but breaks on implementation B that picked, e.g., `files=` / `refs=`. Conversely, a CI parser that does not parse names at all has no spec-level guarantee on the order, so it cannot rely on positional decoding either (the prose says *"in that order"* but that constraint is buried inside the parenthetical license to substitute names, leaving open whether the positional guarantee survives substitution). The result is a wire contract that no portable CI parser can write against — the section pins the four-tab-delimited record shape and the `<n/a>` sentinel rule precisely, then leaves the only contentful field on the green path under-specified.

Both gaps share a single pattern (an implementation-chosen value sits behind a MUST with no enforceable shape constraint), but they sit in two separate paragraphs and resolve by two independent edits, so they are decomposed below.

## Solution Space

**Shape:** single

### Recommendation

Make two edits to `audit-wire-and-canary.md`: pin the green-path canary counter key names, then pin the audit's wall-clock budget. Make the counter-key edit first, because the wall-clock edit would otherwise re-touch lines adjacent to the canary-wire edit.

1. In the *Non-empty-scan canary* paragraph, *Green-path emission contract* clause, replace "The `proposed-resolution` field MUST carry the two counter values as the plain-ASCII key=value pair set `walked=<N> recognised=<M>` (or an equivalent implementation-chosen plain-ASCII key=value pair set whose pair names identify the same two counters this paragraph defines — the audited-files-walked counter and the in-scope-references-recognised counter, in that order)" with: "The `proposed-resolution` field MUST be exactly the plain-ASCII pair set `walked=<N> recognised=<M>`, with `<N>` the audited-files-walked counter and `<M>` the in-scope-references-recognised counter, the two pairs separated by a single ASCII space (U+0020), in that order, with no leading or trailing whitespace and no additional pairs." In the closing PIC-pins sentence of the same paragraph, change "the `walked=<N> recognised=<M>` `proposed-resolution` pair set or its implementation-chosen equivalent" to "the `walked=<N> recognised=<M>` `proposed-resolution` pair set".

2. In the *Infrastructure-failure handling* paragraph, *Wall-clock budget* clause, change "(the implementation pins the budget value, e.g. 60 s)" to "of 60 s" and drop the parenthetical, so the clause reads that the audit MUST enforce a wall-clock budget of 60 s on its own execution. In the *Pre-emission termination carve-out* clause, change "with the implementation pinning a `npm test`-side timeout to bound the overall run" to "with a 120 s `npm test`-side timeout that bounds the overall run; the 60 s audit budget above is the inner bound and fires first on a hung audit, producing a routable infrastructure-failure record".

Edge cases the implementer must watch:

- The `npm test`-side timeout must be pinned in the same edit as the audit budget (the carve-out depends on it), and it must be strictly greater than the 60 s audit budget so the audit's routable infrastructure-failure record fires before the outer `npm test` kill removes the routing surface.
- The *Wire serialisation* clause already forbids ASCII tab in any field value; the new pair-set wording inherits that prohibition (the single ASCII space between pairs is U+0020, not tab) and does not need to re-state it.
- The closing PIC-pins sentence MUST be updated in lockstep with the *Green-path emission contract* edit, or the page contradicts itself on whether substitution is permitted.

## Consequence

**Severity:** advisory

Two reasonable implementers will pick a sensible budget (60 s) and the canonical counter keys (`walked` / `recognised`), so the expected outcome is divergence in pathological cases rather than wrong behaviour on the common path. An adversarial or careless implementer can however satisfy both MUSTs with values that defeat the fail-closed protection and break cross-implementation CI tooling; the spec gives a reviewer no normative anchor to reject either choice, and gives a CI parser no portable contract to write against on the green path.
