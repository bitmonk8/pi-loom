## Finding

Two independent gaps in `audit-wire-and-canary.md`, one per sub-paragraph of the audit's "fail-closed" surface:

1. **Wall-clock budget bound is missing.** The *Wall-clock budget* clause inside the *Infrastructure-failure handling* paragraph reads: *"The audit MUST enforce a wall-clock budget on its own execution (the implementation pins the budget value, e.g. 60 s); on budget expiry the audit MUST emit one record under a stable `<symptom>` token in the infrastructure-failure family and exit non-zero."* The MUST is normative but the budget value is delegated wholesale to the implementation with only a parenthetical "e.g. 60 s" as guidance. A conformant implementation that pins the budget at 1 ms satisfies the MUST while making every real audit run trip the timeout; a conformant implementation that pins it at 86 400 s satisfies the MUST while neutralising the fail-closed property the clause exists to provide (the outer `npm test`-side timeout the *Pre-emission termination carve-out* defers to will fire first and remove the routable infrastructure-failure record). Either extreme passes the spec, so the "fail-closed bound" the section depends on is unenforceable at the spec level. The same problem applies to the `npm test`-side timeout the carve-out names, with identical unboundedness.

2. **Green-path canary counter key names are not portable.** The *Green-path emission contract* clause specifies that the canary record's `proposed-resolution` field MUST carry the two counter values as the plain-ASCII key=value pair set `walked=<N> recognised=<M>` — *"or an equivalent implementation-chosen plain-ASCII key=value pair set whose pair names identify the same two counters this paragraph defines."* A CI parser that asserts `walked=` / `recognised=` works on implementation A but breaks on implementation B that picked, e.g., `files=` / `refs=`. Conversely, a CI parser that does not parse names at all has no spec-level guarantee on the order, so it cannot rely on positional decoding either (the prose says *"in that order"* but that constraint is buried inside the parenthetical license to substitute names, leaving open whether the positional guarantee survives substitution). The result is a wire contract that no portable CI parser can write against — the section pins the four-tab-delimited record shape and the `<n/a>` sentinel rule precisely, then leaves the only contentful field on the green path under-specified.

Both gaps share a single pattern (an implementation-chosen value sits behind a MUST with no enforceable shape constraint), but they sit in two separate paragraphs and resolve by two independent edits, so they are decomposed below.

## Solution Space

**Shape:** multiple

Two independent obligations, resolved in this order: pin the counter key names first (single-paragraph clarification on the wire shape with no numeric judgement required), then bound the wall-clock budget (touches two coupled timeout values and demands a defensible number). The smaller/scope-bounding edit first keeps the wall-clock fix from landing on a moving canary-wire baseline.

### Option A — Pin the wall-clock budget to a single concrete value

**Approach:** Replace the parenthetical "e.g. 60 s" in the *Wall-clock budget* clause with a normative pin: *"the audit MUST enforce a wall-clock budget of 60 s on its own execution"*. Apply the same treatment to the `npm test`-side timeout one sentence later (pin a single value, e.g. 120 s, that is strictly greater than the audit's budget so the audit-side fail-closed surface fires first and produces a routable record). Optionally add a forward-compatibility note that a future minor revision MAY raise the budget if a non-pathological audit run is observed exceeding it.

**Spec edits:**

- In `audit-wire-and-canary.md`, *Infrastructure-failure handling* paragraph, *Wall-clock budget* clause: change *"(the implementation pins the budget value, e.g. 60 s)"* to *"of 60 s"* and drop the parenthetical.
- In the *Pre-emission termination carve-out* clause: change *"with the implementation pinning a `npm test`-side timeout to bound the overall run"* to *"with a 120 s `npm test`-side timeout that bounds the overall run; the 60 s audit budget above is the inner bound and fires first on a hung audit, producing a routable infrastructure-failure record"*.

**Pros:** Single number per timeout, trivially auditable, mirrors how the spec pins other numeric thresholds (counter floors, the `> 0` floor on the canary, the four-tab-delimited record shape, the `<n/a>` sentinel length).

**Cons:** Locks the budget against future SDKs / source trees that may warrant a larger value; a spec revision is required to raise it.

**Risks:** A grown audited source tree could legitimately exceed 60 s, forcing a spec revision rather than an implementation-side adjustment. Mitigated by the optional forward-compatibility note.

### Option B — Pin the canonical canary counter key names

**Approach:** Remove the "or an equivalent implementation-chosen plain-ASCII key=value pair set" license. Pin `walked=<N> recognised=<M>` (in that exact order) as the sole permitted `proposed-resolution` payload for a green-path canary record. The pair-name substitution allowance was the only surface a portable CI parser could not write against; removing it makes the wire format complete on the green path.

**Spec edits:**

- In `audit-wire-and-canary.md`, *Non-empty-scan canary* paragraph, *Green-path emission contract* clause: change *"The `proposed-resolution` field MUST carry the two counter values as the plain-ASCII key=value pair set `walked=<N> recognised=<M>` (or an equivalent implementation-chosen plain-ASCII key=value pair set whose pair names identify the same two counters this paragraph defines — the audited-files-walked counter and the in-scope-references-recognised counter, in that order)"* to *"The `proposed-resolution` field MUST be exactly the plain-ASCII pair set `walked=<N> recognised=<M>`, with `<N>` the audited-files-walked counter and `<M>` the in-scope-references-recognised counter, the two pairs separated by a single ASCII space (U+0020), in that order, with no leading or trailing whitespace and no additional pairs."*
- In the closing PIC-pins sentence of the same paragraph: change *"the `walked=<N> recognised=<M>` `proposed-resolution` pair set or its implementation-chosen equivalent"* to *"the `walked=<N> recognised=<M>` `proposed-resolution` pair set"*.

**Pros:** A single short edit makes the green-path canary record byte-portable across conformant implementations; a CI parser asserting on the canonical names ports unchanged. Closes the substitution surface without enlarging the wire contract.

**Cons:** Forecloses any future renaming of the counter keys by the implementation; a rename now requires a spec edit. Negligible — the names are already published and have no observable downside.

**Risks:** None material; the names are short, descriptive, and already the de facto contract.

### Recommendation

Apply both options, in the order **Option B then Option A**. Option B is a one-paragraph clarification on the wire shape, lands on a quiet baseline, and once it lands the green-path record is byte-portable. Option A then bounds the timeout pair on top of that stable baseline; doing it in the reverse order would mean Option B's edit re-touches lines that Option A just changed and risks rebase churn.

Edge cases the implementer must watch:

- For Option A, the `npm test`-side timeout in the *Pre-emission termination carve-out* must be pinned in the same edit (the carve-out depends on it), and it must be strictly greater than the audit budget so the audit's routable infrastructure-failure record fires before the outer `npm test` kill removes the routing surface.
- For Option B, the *Wire serialisation* clause already forbids ASCII tab in any field value; the new pair-set wording inherits that prohibition (the single ASCII space between pairs is U+0020, not tab) and does not need to re-state it. The closing PIC-pins sentence must be updated in lockstep or the page contradicts itself on whether substitution is permitted.

## Consequence

**Severity:** advisory

Two reasonable implementers will pick a sensible budget (60 s) and the canonical counter keys (`walked` / `recognised`), so the expected outcome is divergence in pathological cases rather than wrong behaviour on the common path. An adversarial or careless implementer can however satisfy both MUSTs with values that defeat the fail-closed protection and break cross-implementation CI tooling; the spec gives a reviewer no normative anchor to reject either choice, and gives a CI parser no portable contract to write against on the green path.
