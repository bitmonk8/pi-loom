## Problem

The Canonical schema hash recipe in `schema-subset.md` (`#canonical-schema-hash`) declares its canonical-form byte sequence an exact on-disk/on-wire breaking-change contract, so two conforming implementations must produce identical bytes — and therefore identical slugs — for the same source. The step-2 string-serialisation clause ("strings escaped per RFC 8259 minimal-escape rules") does not meet that bar: it leaves two degrees of freedom RFC 8259 explicitly permits. For C0 control characters (U+0000–U+001F) "minimal-escape" admits both the two-character short escapes (`\n`, `\t`, …) and the six-character `\uXXXX` form, and the `\uXXXX` form admits both uppercase and lowercase hex digits. Any `const`/`enum` string carrying a control character can therefore serialise to divergent canonical bytes and a divergent slug across implementations.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `e21449b` — "pi-loom spec: resolve \"__inline_<hash> hash function is not pinned\"" (2026-05-04)
**History:** `e21449b` authored the entire **Canonical schema hash** recipe, including the step-2 clause verbatim: "strings escaped per RFC 8259 minimal-escape rules (only the characters JSON requires escaping; no gratuitous `\u` escapes for printable ASCII)." The string-escaping under-pinning has existed since that introduction. A later commit `c5bb5e1` ("Canonical schema hash numeric serialization underspecified") tightened the adjacent *numeric* clause in the same byte-sequence block — delegating it to the binder `integer`/`number` rendering algorithm — but left the string-escaping line untouched (`git show c5bb5e1` carries it as unchanged context). `git log -S 'minimal-escape' --all` confirms no commit has since narrowed the escaping rule. The defect is thus original to the recipe, not introduced by a later edit; the numeric-clause fix simply isolated it as the last unpinned facet.

## Solution approach

Rewrite the step-2 string-escaping clause in the Canonical schema hash recipe (`#canonical-schema-hash`, step 2 "Canonical form") so it pins one byte sequence per source string, closing both degrees of freedom the current "RFC 8259 minimal-escape" wording leaves open: the C0 control-character escape form (short escape vs `\uXXXX`) and the `\uXXXX` hex-digit case. Either delegate to a named normative serialiser, as the adjacent numeric clause already delegates to the binder rendering algorithm, or state the escaping rule directly in-spec.
