## Finding

Pi discovers loom's extension by reading `package.json#pi.extensions = ["./extensions"]` and loading the module that resolves under that directory — `extensions/index.ts` (`H4a`'s thin delegating entry shim, which re-exports the `src/**` factory as the standard `default function (pi: ExtensionAPI)` export). No `npm test`-time check exercises that resolution path. `H1a`'s build-time assertion reads `package.json#pi.extensions` and asserts only that its value equals `["./extensions"]`; by its own wording it deliberately stops short of loadability ("rather than shipping green while the end-to-end harness imports the factory directly through a build-internal path"). `H4a`'s harness loads the extension to dispatch a command, but the plan explicitly permits that load to reach the factory through a build-internal path rather than through the resolved `extensions/index.ts`, so the harness self-check does not guarantee the on-disk shim exists, resolves under the manifest directory, or exposes a callable `default` export.

The shim's correctness is therefore confirmed by exactly two surfaces, both manual and both late: the manual `H6a` real-host smoke run, and the loom 1.0 release-time residue inspection (checklist item 8, "Entry-shim purity"), which is a reviewer reading `extensions/index.ts`. A shim with a wrong file path, a missing or misnamed `default` export, or a re-export that drifts from the `src/**` factory passes every mechanical gate green through the entire H- and V-leaf set and is caught only at that terminal manual review (or in a real host). A build-time test that resolves the manifest directory, imports the resolved entry file, and asserts its `default` export is a function would shift this detection from release-time manual inspection to per-commit `npm test`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e7c1b1e`, `4b3c2c0` (re-export shape established in `eafd4a0`)
**History:** The plan corpus is git-tracked. The gap is the joint product of two plan edits, neither of which alone is wrong. `e7c1b1e` (2026-06-16, "pi-loom plan: resolve \"Pi extension-discovery contract owned by no leaf\"") first assigned the discovery contract to `H1a`/`H4a` and authored `H1a`'s manifest assertion as a value-only literal-read; `git log -S "imports the factory directly through a build-internal path"` reports `e7c1b1e` as the sole introducing commit of the clause that consciously scopes that test to the manifest value and concedes the harness may import through a build-internal path — i.e. the discovery path is left mechanically unexercised at this point. `eafd4a0` ("resolve \"H4a factory framed as returning an extension object\"") later fixed the shim's re-export shape to `default function (pi: ExtensionAPI)` (`git log -S "default function (pi: ExtensionAPI)"` → `eafd4a0`). `4b3c2c0` (2026-06-17, "pi-loom plan: resolve \"gated-tree completeness claim rests on unverified entry-shim purity\"") then added conventions residue-inspection item 8 (`git log -S "Entry-shim purity"` → `4b3c2c0`) and the framing that the shim's purity and re-export are verified only at release-time manual inspection — cementing the absence of any mechanical existence/re-export gate. The gap was thus present from the contract's first assignment (`e7c1b1e`) and entrenched as a deliberate "manual-only" posture by `4b3c2c0`; this finding asks whether that posture should be supplemented by a cheap mechanical check.

## Solution Space

**Shape:** single

### Recommendation

Add a build-time (`npm test`) assertion to `H4a` — the earliest leaf at which `extensions/index.ts` exists — that reads `package.json#pi.extensions`, resolves the module under that directory, imports it, and asserts its `default` export is callable (ideally that it is the same factory the `src/**` root exports). Place it as a new `Tests.` bullet on `H4a` (the leaf that authors the shim and already owns the load-the-extension harness), distinct from `H1a`'s value-only `pi.extensions` literal-read: `H1a` keeps asserting the manifest value equals `["./extensions"]`; the new `H4a` bullet consumes that value to locate and actually import the resolved entry file, so a missing file, a wrong path, or a non-function / absent `default` export fails `H4a`'s own `npm test` rather than surfacing only at the `H6a` manual smoke or release-time residue item 8.

The bullet must drive resolution through the same loader/build path a real Pi host uses to load the manifest directory, so the file it imports tracks whatever the open `.ts`-vs-built-`.js` entry-load question settles (see Related Findings); state that the resolved entry file is whatever resolves under the `pi.extensions` directory rather than hard-coding a `.ts` path that a built-`.js` host would not load. Item 8 of the release-time residue inspection stays as the purity backstop; this bullet adds the existence/loadability/re-export gate item 8 currently carries alone.

## Consequence

**Severity:** advisory

A discovery shim that resolves to the wrong file, omits or misnames its `default` export, or drifts from the `src/**` factory ships green through every `npm test` gate and is caught only by the manual `H6a` real-host smoke or the release-time residue inspection (item 8) — the very last checkpoints before release. The defect is real but not blocking: a documented manual backstop exists, and no implementer is forced to diverge or guess. Adding a mechanical gate shifts detection of a broken host-discovery path from terminal manual review to per-commit CI.
