## Problem

PIC-8 ("Restore-failure protocol", anchor `pic-8`) clause (a) directs the runtime to "re-attempt the restore exactly once with the same snapshot (covering transient failures)". The trailing parenthetical admits two incompatible readings: as design rationale for an unconditional retry (the intended reading, reinforced by the adjacent "exactly once" / "on a second failure" / no-backoff language), or as a normative gating predicate that fires the retry only when the failure is classified transient. A literal reader taking the second reading would build a transient-vs-permanent error classifier over `pi.setActiveTools`'s exception surface — a taxonomy the spec never defines.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** 2d2fc8e — pi-loom spec: resolve "Pi SDK capability calls have no failure contract" (2026-05-06, Thomas Andersen)
**History:** The Restore-failure protocol (later PIC-8) was authored in 2d2fc8e, and the parenthetical "(covering transient failures)" appears verbatim in that same introducing commit alongside the "re-attempt the restore exactly once" clause. The ambiguity has been present since the rule first entered the corpus; later commits (the spec split f5e89f4 and the PIC-8 `/tools`-hint narrowing bef37f3) moved and trimmed the surrounding text but never touched the parenthetical.

## Solution approach

Rewrite PIC-8 clause (a) so the trailing parenthetical reads unmistakably as rationale for an unconditional retry rather than as a trigger condition, leaving the existing "exactly once" / "on a second failure" / no-backoff language intact.

## Solution constraints

- The fix MUST NOT introduce a transient-vs-permanent error taxonomy or classifier; the retry fires on any throw or rejection from the step-4 restore.
