## Finding

The family-(4) CJS-reach prohibition in `audit-recognised-shapes.md` extends from a finite syntactic enumeration (`require(...)`, direct `createRequire(...)(...)`, aliased import / namespace import / default import / re-export of `createRequire`, captured `const cr = createRequire`) into an open-ended data-flow obligation: *"any call to a function whose source binding **transitively resolves to** the `createRequire` named export of the Node `"module"` package when invoked with one of the in-scope package specifiers — any such call MUST NOT exist anywhere in the audited source tree, and the audit MUST enforce that prohibition under its own discriminator"*. "Transitively resolves to" has no bounded definition: it admits arbitrary chains through helper modules, returned-from-factory bindings, conditional re-exports, and reflection that no static-AST walker can decide without a whole-program type-and-value-flow analysis.

This obligation sits alongside three other audit rules whose authors anticipated exactly the same shape of unbounded edge — *Substitution limitation*, *Stale-citation limitation* (both in `audit-failures.md`), and *Generated-output-in-src limitation* (in `audit-resolution.md`) — and each of those rules is explicitly framed as an *"acknowledged soft surface"*: the spec pins the prohibition, names contributor-side mitigation, and explicitly states that no additional audit discriminator is owed at loom 1.0. The transitive-`createRequire` rule is silently the same shape (mechanically undecidable from source text), but is written as an absolute MUST with no caveat. Two reasonable implementers will produce different audits: one will ship the enumerated syntactic detector and treat the residual chains as out-of-scope, the other will block the leaf trying to satisfy the unbounded clause. The audit's *behavioural contract is invariant under tooling choice* sentence at the end of the paragraph is incompatible with an obligation whose satisfiability *does* depend on tooling power.

## Solution Space

**Shape:** single

### Recommendation

Bound the family-(4) `createRequire`-reach detection to a closed enumerated set of syntactic shapes and acknowledge deeper transitive resolution as a soft surface, mirroring the three sibling acknowledgments. Concretely, edit the CJS-reach clause in `audit-recognised-shapes.md` to (a) rewrite the *"any call to a function whose source binding transitively resolves to"* sentence to enumerate the closed set the audit MUST detect — direct named import, aliased import / namespace import / default import / default-namespace re-export of `createRequire` from the `"module"` package, per-symbol same-module re-export, and a same-module captured local (`const cr = createRequire`) — and to state that the detector resolves bindings within a single module's syntactic scope, not across module boundaries or through dynamic dispatch; and (b) add a parenthetical *Transitive-resolution limitation (acknowledged soft surface)* sub-bullet — sibling to the *Substitution limitation* and *Stale-citation limitation* sub-bullets in `audit-failures.md` and the *Generated-output-in-src limitation* in `audit-resolution.md` — stating that calls whose `createRequire` provenance reaches across module boundaries, through factory returns, through reflective lookup, or through any other shape outside the enumerated set are *not* mechanically detectable by the audit; that the mitigation is the contributor-side review obligation under the version-bump checklist; and that widening detection is a PIC edit deferred past loom 1.0.

Implementer-relevant edge cases:

- Keep the existing `.resolve(<specifier>)` carve-out exactly as it is: it narrows the rule itself rather than authorising a marker, and the soft-surface note must not be read as opening a marker path on family (4).
- The enumeration MUST be syntactically decidable from a per-module AST walk so that the *behavioural contract is invariant under tooling choice* claim in the closing sentence is actually true (the type-aware-variant allowance then becomes an implementation latitude, not a satisfiability requirement).
- The same-line `// allow-pi-surface:` marker remains forbidden on family (4) lines per *Exemption mechanism*; the soft-surface acknowledgment changes only the boundary of what the audit MUST detect, not the exemptibility of what it does detect.

## Consequence

**Severity:** correctness

Without bounding, the family-(4) detector's scope is ambiguous: implementers cannot decide whether shipping a syntactic-shape detector against the enumerated list satisfies the MUST or whether they must invest in flow analysis (which, in general, cannot exhaustively recover transitive bindings). Two reasonable implementations therefore differ on what they reject — the family-(4) record set diverges across implementations on the same `src/` tree — which directly breaks the surface-set-closure guarantee the audit exists to enforce. The risk is asymmetric: under-coverage silently launders CJS reach past the audit, while over-coverage flags benign indirection and blocks `V18b` from landing green.
