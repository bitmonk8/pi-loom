## Finding

The inventory-closure audit cluster repeatedly names `SDK_SURFACE_INVENTORY` and the two typebox sibling allow-lists (`{ Type }` named-import allow-list, `{ Unsafe }` member-access allow-list) as the source of truth that both the positive-direction literal-read assertion and the negative-direction audit consume. The typebox carve-out in `audit-target-categories.md` pins the allow-lists to "the same source-of-truth pinned-constants block in the extension module that already holds the four Step 0 … constants," and `inventory-audit-intro.md` pins `CAPABILITY_OBLIGATIONS` to "the extension module's pinned-constants block." Neither file gives a concrete module path or a citable anchor for "the extension module," and neither cross-references the one place a path appears (`extension-bootstrap-and-per-loom.md`, which names the extension entry point `extensions/index.ts`). An implementer building the audit and the inventory has no in-cluster pointer to where these constants are declared.

The omission is sharper than a missing cross-reference because the audited source tree is defined as the closed glob `src/**/*.ts` (per `audit-resolution.md` *Audit scope*), and `extensions/` is outside that root. The cluster never states whether the inventory-defining module is inside the audited tree. The answer has audit-behaviour consequences: if the constants module sits under `src/` it is itself walked, so any in-scope reference it contains (e.g. an `import { Type } from "typebox"`) must itself resolve through the inventory or allow-lists or be exempted; if it sits under `extensions/` it is not walked and those references are outside the surface-set-closure guarantee. Two implementers placing the module on opposite sides of the `src/` boundary produce audits that observe different reference populations.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commit:** `e5cd514` (2026-05-09) — "pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\""
**History:** The corpus is git-tracked. `git log -S` over `docs/` shows the inventory-closure audit and the "pinned-constants block in the extension module" framing were introduced together in `e5cd514`; the `SDK_SURFACE_INVENTORY` token itself predates that (first appearing 2026-05-06). No revision in the history ever pinned a concrete declaration path or anchor for the inventory / allow-list module (searches for path-pinning phrasing return no commits), and the current audit-cluster files (`inventory-audit-intro.md`, `audit-target-categories.md`) contain no `extensions/index.ts` or `src/extension` reference. The loose "the extension module" reference and the unstated in/out-of-`src/`-scope status are therefore co-extensive with the audit feature's introduction, not a later regression.

## Solution Space

**Shape:** single

### Recommendation

In the audit cluster, pin the declaration site of `SDK_SURFACE_INVENTORY` and the two typebox sibling allow-lists either by (a) naming the canonical constants-block module with a citable anchor the audit, the literal-read assertion, the Step 0 capability probe, and the typecheck-time brand-string assertion all reference, or (b) stating explicitly that the module path is implementation-owned and pinning the invariant that all those gates MUST import one shared module. Whichever form is chosen, add one sentence stating whether that module lies inside the `src/**/*.ts` audited tree, and if it does, that its own in-scope references (notably any `import { Type } from "typebox"`) resolve through the inventory / allow-lists or carry an `// allow-pi-surface:` marker on the same footing as any other audited file. Place the pin at the `inventory-audit-intro.md` *Inventory-closure audit* site (the cluster's source-of-truth anchor for `SDK_SURFACE_INVENTORY`), and have the `audit-target-categories.md` typebox carve-out cite it rather than restate "the extension module."

## Consequence

**Severity:** advisory

Implementers cannot locate from the audit cluster where to declare `SDK_SURFACE_INVENTORY` and the typebox allow-lists, and the cluster does not say whether that module is inside the `src/**/*.ts` audited tree. A working audit is reachable under either placement, but the spec leaves the in/out-of-scope decision implicit, so two implementers can diverge on whether the inventory module's own peer-package/typebox references are themselves audited.
