## Finding

The unblock condition for the session-only degraded-state work is described throughout the plan as the prose phrase "host-prerequisites clause (a)" / "version-bump editorial-review checklist item (a)". These are *positional* labels — the `(a)` enumerates a position inside a list, not a content-stable identifier. The references appear in four plan locations: `docs/plan.md` (the §V9 interleave note, the §V9 leaf-list entry for `V9l`, and the §Blocked obligations bullet), `V9h-unknown-reason-rule.md` (Adds), `V9l-session-only-degraded-branch.md` (Deps and Ships when), and `V9l-T-session-only-degraded-branch.md` (Deps).

The two referents the `(a)` phrases point at are asymmetric in their addressability. The *resolution owner* — the version-bump editorial-review checklist item (a) in `version-bump-step2.md` — already carries a content-stable anchor, `#bump-checklist-instance-survival`, and the §Blocked obligations bullet links to it. The *contradiction statement* — clause (a) under `host-prerequisites.md` §"Host prerequisites for the degraded-state branch" — has **no** dedicated anchor; it sits as a bare `(a)` sub-bullet under the section anchor `#degraded-state-host-prerequisites`, which spans clauses (a)–(d) collectively. References therefore resolve only to the section, and the precise sub-clause is pinned by prose position alone.

If `host-prerequisites.md`'s clause enumeration is reordered or a clause is inserted ahead of the present (a), every "clause (a)" reference in the plan silently mis-describes which clause gates the work — the markdown link still resolves to the section, so nothing breaks loudly. (Note: actual pickability is gated by the stable `V18c` Deps edges on `V9l`/`V9l-T`, not by this prose, so execution is unaffected; the defect is a documentation-traceability one. The original framing's "register it in ids.md" suggestion does not apply here — this corpus has no plan-index files; the only durable home for a clause-level identifier is a spec anchor.)

## Solution Space

**Shape:** multiple

### Option A — Dedicated spec anchor for the contradiction clause

**Approach.** Add an HTML anchor to the clause (a) sub-bullet in `host-prerequisites.md` (e.g. `<a id="degraded-state-host-prerequisite-a">` immediately before the `(a) *Extension-instance survival...*` bullet under §degraded-state-host-prerequisites), then rewrite the four plan references so each links to that anchor for the contradiction and to the existing `#bump-checklist-instance-survival` for the resolution, in place of the bare positional "clause (a)" / "item (a)" prose.

**Plan edits.** In `docs/plan.md` (§V9 interleave note, §V9 leaf-list `V9l` entry, §Blocked obligations bullet), `V9h-unknown-reason-rule.md` Adds, `V9l-session-only-degraded-branch.md` Deps + Ships when, and `V9l-T-session-only-degraded-branch.md` Deps: replace the positional clause-(a) phrasing with a link to the new `host-prerequisites.md` clause anchor.

**Spec edits.** `host-prerequisites.md`: insert one `<a id=...>` on the clause (a) sub-bullet. No normative wording of clause (a) changes.

**Pros.** Both referents (contradiction statement and resolution owner) become content-addressed; reordering the (a)–(d) enumeration no longer silently mis-points any plan reference.

**Cons.** Requires a spec edit (an editorial anchor-only change).

**Risks.** Low — anchor-only change; the plan-diff fixer must not alter clause (a)'s normative text.

### Option B — Re-pin plan references to the existing resolution-owner anchor

**Approach.** Treat `version-bump-step2.md#bump-checklist-instance-survival` (already stable) as the canonical block-condition identifier and rewrite the four plan references to cite it as the gating condition, dropping reliance on the positional "host-prerequisites clause (a)" phrase; keep the section-level `#degraded-state-host-prerequisites` link only as supporting context for the contradiction.

**Plan edits.** Same four locations as Option A, re-pointed to `#bump-checklist-instance-survival`.

**Spec edits.** None.

**Pros.** No spec edit; uses an anchor that already exists and is content-stable.

**Cons.** The contradiction-statement reference still resolves only to the section anchor, so a positional "(a)" within `host-prerequisites.md` remains in any prose that must name the contradiction clause specifically; partial elimination of the positional label.

**Risks.** Low.

### Recommendation

Take **Option A**. The block condition has two distinct referents — the contradiction (host-prerequisites clause (a)) and its resolution (version-bump item (a)) — and only the resolution side is content-addressed today. Adding a single anchor to the contradiction clause makes both referents stably citable and removes the positional "(a)" for both, which is the complete fix the finding asks for; Option B leaves the contradiction-clause reference positional. The spec change is anchor-only and must not touch clause (a)'s normative wording. Edge case: land the spec anchor before retargeting the plan links, so the plan references resolve at every intermediate state.

## Consequence

**Severity:** advisory

A reader or future editor who reorders the `host-prerequisites.md` clause enumeration silently invalidates every "clause (a)" reference in the plan without any link breaking or gate firing. Pickability itself is unaffected (the `V18c` Deps edges gate the leaves), so the impact is confined to cross-reference accuracy and auditability of the block condition.
