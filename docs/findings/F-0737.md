## Problem

DISC-5 (`discovery/package-and-settings.md#disc-5`) normatively pins `pi.looms` glob resolution — and the inherited `loomPaths` rule — to the `minimatch` engine, but the spec's runtime-dependency enumeration in `implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)" (anchor `#loom-package-implementation-dependencies-loom-1-0`) lists only `semver`, `chokidar`, and `yaml` — not `minimatch`. The corpus therefore names a load-bearing matcher in a normative rule without telling the implementer where it comes from, leaving two readings open: that loom declares its own `minimatch` dependency, or that it reaches into the copy bundled with `@earendil-works/pi-coding-agent` via the internal `applyPatterns` symbol (which carries no stability promise). The plan's `H1a` leaf has already chosen the own-dependency reading and cites DISC-5, but the impl-deps paragraph it defers to doesn't carry the `minimatch` entry, so the citation points at a paragraph that doesn't list the dependency it claims to source.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `1cb54c7` (2026-06-06, "pi-loom spec: resolve \"Glob !/+/- precedence and matcher engine unspecified for pi.looms / loomPaths\""); origin context `1ea0455` (2026-06-04, "pi-loom spec: resolve \"DI-seam vs reference-implementation placement boundary\""); missed-opportunity `1064946` (2026-06-10, "pi-loom plan: resolve \"YAML frontmatter parsing mechanism is never declared as a dependency\"")
**History:** `git log -G 'minimatch' -- docs/spec_topics` returns a single commit, `1cb54c7`, the sole site where `minimatch` enters the spec corpus — added to DISC-5 (and inherited by `loomPaths`) as the pinned glob engine, referencing Pi's internal `applyPatterns` symbol. `git log -S 'Loom-package implementation dependencies'` on `implementation-notes.md` shows the enumeration block was created at `1ea0455` (before `minimatch` was named in DISC-5) with `semver` and `chokidar` clauses; the `yaml` clause was added at `1064946` (after `minimatch` was named in DISC-5). The defect is the omission at `1cb54c7`: that commit pinned the engine in the discovery rule without extending the parallel enumeration in `implementation-notes.md`. The later `yaml`-parity edit at `1064946` is a missed second opportunity and confirms the omission is durable, not a transient mid-edit state. (Verified: `git log -S 'minimatch' -- docs/spec_topics/implementation-notes.md` returns no matches.)

## Solution approach

Add a `minimatch` clause to the impl-deps enumeration in `implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)", parallel to the existing `semver` / `chokidar` / `yaml` clauses: name `minimatch` as the loom 1.0 runtime dependency consumed by DISC-5's glob matcher (forward-link to `discovery/package-and-settings.md#disc-5`), declared in loom's own `dependencies` block on the same footing as the others. Follow the established "names the dependency, not its version" wording and add the inventory-closure carve-out sentence already used for `chokidar` and `yaml` (`minimatch` is neither a `@earendil-works/*` peer package nor `typebox`).
