## Finding

DISC-5 in `discovery/package-and-settings.md` pins `pi.looms` glob resolution to the `minimatch` engine — "the same package Pi applies to its own resource arrays in `@earendil-works/pi-coding-agent` (`dist/core/package-manager.js`, symbol `applyPatterns`)" — and `loomPaths`' `loompaths-entry-schema` section inherits that engine choice. Realising DISC-5 therefore requires loom to consume `minimatch` at runtime: the cited `applyPatterns` symbol lives under `dist/core/` of a peer package, is not part of any documented `@earendil-works/*` export surface, and cannot be relied on by an implementer.

The spec's own runtime-dependency enumeration — `implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)" — names `semver`, `chokidar`, and `yaml` as the loom package's runtime `dependencies`, each with a parallel clause stating the seam it backs. It does not mention `minimatch`. A reader cross-checking DISC-5 against that enumeration sees the engine named in the normative discovery rule but absent from the spec's authoritative list of runtime dependencies, leaving open whether loom takes its own `minimatch` dependency or reaches into Pi's bundled copy through a non-public dist-internal symbol.

## Issue introduction

**Verdict:** introduced-by-later-edit (multi-step)
**Introducing commits:** `1cb54c7` (2026-06-06, "pi-loom spec: resolve \"Glob !/+/- precedence and matcher engine unspecified for pi.looms / loomPaths\""); origin context `1ea0455` (2026-06-04, "pi-loom spec: resolve \"DI-seam vs reference-implementation placement boundary\"") and missed-opportunity `1064946` (2026-06-10, "pi-loom plan: resolve \"YAML frontmatter parsing mechanism is never declared as a dependency\"").
**History:** `git log -G 'minimatch' -- docs/spec_topics` returns a single commit, `1cb54c7`, which is the sole site where `minimatch` enters the spec corpus — added to DISC-5 (and inherited by `loomPaths`) as the pinned glob engine. `git log -S 'Loom-package implementation dependencies'` on `implementation-notes.md` shows the enumeration block was created at `1ea0455` (2026-06-04, before `minimatch` was named in DISC-5) with `semver` and `chokidar` clauses; the `yaml` clause was added at `1064946` (2026-06-10, after `minimatch` was named in DISC-5). The defect is the omission at `1cb54c7`: that commit pinned the engine in the discovery rule without extending the parallel enumeration in `implementation-notes.md`. The later `yaml`-parity edit at `1064946` is a missed second opportunity to close the gap and confirms the omission is durable, not a transient mid-edit state.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)", add a `minimatch` clause parallel to the existing `chokidar` and `yaml` clauses: state that the loom 1.0 runtime depends on the [`minimatch`](https://www.npmjs.com/package/minimatch) npm package as the glob-matcher engine realising the `pi.looms` resolution rule (override ordering `plain → ! → + → -` and exact-path operand treatment) anchored at `discovery/package-and-settings.md#disc-5`; that `minimatch` is declared in the loom package's `dependencies` block on the same footing as `semver`, `chokidar`, and `yaml`; that the clause names the dependency, not its version; and that, like `chokidar` and `yaml`, `minimatch` is neither one of the four `@earendil-works/*` peer packages nor `typebox`, so it is out of scope for the inventory-closure audit's *Target surface categories* and requires no `// allow-pi-surface:` exemption marker at any `minimatch` import site. The clause closes the "how loom obtains it" question by stating the loom package takes its own dependency — loom does not reach into Pi's bundled copy through the dist-internal `applyPatterns` symbol that DISC-5's parenthetical merely cites for behavioural reference. `minimatch` ships its own TypeScript bindings in current major versions, so a `@types/minimatch` `devDependencies` entry is not required (note the parallel with `yaml`, not with `semver`).

Edge cases: the cross-reference at DISC-5 to `applyPatterns` is a non-normative behavioural-reference parenthetical and stays as-is for this edit — its load-bearing-ness is the subject of the two sibling findings below, not of this enumeration gap.

## Consequence

**Severity:** advisory

An implementer following the plan provisions `minimatch` correctly; nothing is blocked. The defect is a documentation-parity gap: the spec's runtime-dependency enumeration is incomplete relative to the dependency set DISC-5 actually induces, weakening the enumeration's value as a cross-check reference and leaving the "own dependency vs. Pi-bundled copy" question implicit rather than stated.
