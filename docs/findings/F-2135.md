## Finding

Two normative pages assert that the multi-error `Diagnostic[]` batch is **sorted by `(file, line, col)`** and explicitly attribute that ordering to the Diagnostics page: both `invocation.md` (*Static resolution*) and `implementation-notes.md` (*Static-resolution load pass*) read "aggregates each visited file's diagnostics into the entry loom's drain (**sorted by `(file, line, col)` per [Diagnostics — Multi-error reporting](./diagnostics.md)**)". The cited owner — `diagnostics/diagnostic-shape.md`, sections *Multi-error reporting* and *Serialised content format* — contains **no ordering rule at all**. *Multi-error reporting* says the pass "collects all errors … before failing … the complete list in one `pi.sendMessage` call"; *Serialised content format* says "each `Diagnostic` becomes one such line block and successive blocks are separated by a single blank line" and that `details.diagnostics` "carries the same as a structured array". Neither pins the order of elements within the batch, in the rendered `content` blocks, or in `details.diagnostics`.

The result is a dangling normative attribution: the `(file, line, col)` sort key lives only as a parenthetical on two *consumer* pages that both point back to a Diagnostics section that does not define it. An implementer who treats `diagnostic-shape.md` as the canonical owner of the diagnostic wire shape and serialisation — its declared role — finds the batch order unspecified, and could legitimately emit diagnostics in source-walk / encounter order rather than the `(file, line, col)` order the rest of the corpus and the test plan assume. The ordering is also stated only for the load-pass aggregation drain; the section that owns the rendered `content` and `details.diagnostics` serialisation never restates that both representations inherit that single order.

A secondary gap: `(file, line, col)` is not a total order — two diagnostics at the same file/line/column (e.g. two distinct codes firing on the same token span) have no defined tiebreaker, yet the corpus requires the rendered/structured output to be deterministic across runs.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** `fae85d3` (2026-05-04, "Tighten spec for implementation-plan readiness") established the *Multi-error reporting* surface with no ordering rule; `333ba2f` (2026-05-04, "spec: define static resolution for invoke/tools callees (Option C)") added the static-resolution load pass carrying the back-citation "sorted by `(file, line, col)` per [Diagnostics — Multi-error reporting]" without adding the cited rule to the Diagnostics surface.
**History:** `git log -S` confirms *Multi-error reporting* entered at `fae85d3` and has never carried an ordering clause on the Diagnostics page. The `(file, line, col)` sort key, and its attribution "per [Diagnostics — Multi-error reporting]", entered with the static-resolution pass at `333ba2f` in `invocation.md` and `implementation-notes.md`. The defect is the interaction between the two: the section asserted to own the sort has never stated it. The durable fix addresses the missing rule on the owning page rather than either commit in isolation.

## Solution Space

**Shape:** single

### Recommendation

Add a normative ordering sentence to `diagnostic-shape.md` (in *Multi-error reporting*, with the serialisation consequence stated in *Serialised content format*): the assembled `Diagnostic[]` is ordered by ascending `(file, range.start.line, range.start.column)`, and this single order governs both the `content` line blocks and the `details.diagnostics` array identically. Use the same key the two citing pages already name, so the existing "sorted by `(file, line, col)` per [Diagnostics — Multi-error reporting]" references in `invocation.md` and `implementation-notes.md` resolve to a real rule. Pin a deterministic tiebreaker for diagnostics sharing the same `(file, line, col)` — `code` (lexicographic), then message — since the rendered/structured output must be byte-identical across runs; location-less diagnostics (no `file`/`range`) need a defined slot in the order (e.g. sorted last, or by `code`). Declaring the order non-normative is not an option here: it would contradict the two citing pages and the `V7a` acceptance assertion.

## Consequence

**Severity:** correctness

A consumer reading the canonical diagnostic-shape owner finds no batch-ordering rule, so two reasonable implementers diverge — one sorts by `(file, line, col)` (as `invocation.md` / `implementation-notes.md` and the `V7a` test assert), another emits in source-walk order — producing different rendered `content` and `details.diagnostics`. Exact-match consumers (the `V7a` ordering assertion, LSP integrations, test harnesses) break against a conforming-but-differently-ordered implementation, and same-position ties have no deterministic resolution.
