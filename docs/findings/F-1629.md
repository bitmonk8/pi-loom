## Finding

The **Unknown-reason rule** prose pins a normative test obligation for the `"literals-shape-invalid"` discriminator: *"A conformance test suite MUST cover each of these four sub-cases with its own fixture …"*, enumerating (1) `literals` not an array, (2) array with a non-string element, (3) empty array, and (4) array with an empty-string element. Each fixture must construct a synthetic snapshot via the static-import-preserving substitution mechanic borrowed from `patch-skew-degradation.md`, dispatch a `session_shutdown` event, and assert exactly one `loom/host/session-shutdown-pinned-constant-unreadable` with `details.failure === "literals-shape-invalid"` and no `loom/host/session-shutdown-reason-unknown`.

The file's only structured acceptance-criteria section — `## Acceptance criteria — "throw:<String(error)>" discriminator grammar` — decomposes a *different* prose MUST (the throw-discriminator grammar) into fixture items (a)–(d). The four `"literals-shape-invalid"` sub-case obligation, though structurally parallel ("MUST cover each … with its own fixture"), has no corresponding acceptance-criteria block. The file therefore presents one of its two co-located fixture MUSTs as a checklist and leaves the other buried in a multi-thousand-word prose paragraph.

The consequence is a review/audit discoverability hazard, not an ambiguity: a reviewer who treats the file's `## Acceptance criteria` section as the complete fixture inventory for the rule would mark it satisfied while the four `"literals-shape-invalid"` sub-case fixtures go unwitnessed. The section's narrow heading ("throw discriminator grammar") is itself the symptom — the spec coined a fixture-checklist for one prose MUST and not for its sibling.

**History:** `git log -S'"literals-shape-invalid"' -- docs/spec_topics/pi-integration-contract/unknown-reason-rule.md` and `git log -L8,16:docs/spec_topics/pi-integration-contract/unknown-reason-rule.md` together pin the timeline: the `"literals-shape-invalid"` clause and its conformance-suite obligation entered the body paragraph in `a7d2a80` (`docs/spec.md` had been split out of by `f5e89f4` already), and the file had no `## Acceptance criteria` section until `8008b91` added the throw-discriminator one as a peer of the body paragraph. The inconsistency is the cross-product of the two and surfaced the moment `8008b91` landed.

## Solution Space

**Shape:** single

### Recommendation

Add a fixture-checklist block for the four `"literals-shape-invalid"` sub-cases alongside the existing throw-grammar items, and widen the section heading so the new block is not mis-filed under "throw discriminator grammar."

- Rename `## Acceptance criteria — "throw:<String(error)>" discriminator grammar` to a heading that covers both `details.failure` arms (e.g. `## Acceptance criteria — details.failure discriminator coverage`), or add a sibling `## Acceptance criteria — "literals-shape-invalid" sub-case fixtures` section.
- Under it, add an item enumerating four fixtures — (e1) `literals` not an array, (e2) array with a non-string element, (e3) empty array, (e4) array with an empty-string element. For each, the fixture constructs a synthetic snapshot via the test-only static-import-preserving substitution mechanic from `patch-skew-degradation.md`, dispatches a `session_shutdown` event, and asserts the handler emits exactly one `loom/host/session-shutdown-pinned-constant-unreadable` with `details.failure === "literals-shape-invalid"` and that no `loom/host/session-shutdown-reason-unknown` fires.
- Cite `PIC-47` and substring (d) of `PIC-48` exactly as the throw-grammar block does, and state — as that block does — that the leaf owns fixture-shape while branch coverage and the exact discriminator string are normative.
- The block introduces no new diagnostic code or wire field; the `details.failure` literal and the two diagnostic codes are read from the anchor-stable contract surface (`PIC-48`), matching the existing block's framing.

## Consequence

**Severity:** advisory

The four-sub-case fixture MUST is fully and unambiguously pinned in prose and is already tracked by leaves `V9h`/`V9h-T`, so a correct, fully-tested implementation is reachable without any change. The defect is a discoverability gap: the file's sole acceptance-criteria checklist is scoped to the throw-grammar arm only, so an auditor reading that section as the rule's complete fixture inventory can sign it off while the `"literals-shape-invalid"` sub-case coverage is never verified.
