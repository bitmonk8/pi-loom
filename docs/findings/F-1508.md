## Finding

The spec owns a deliberately validator-neutral abstraction for schema validation: the injected `SchemaValidator` service (`pi-integration-contract/host-interfaces-services.md#schemavalidator-interface`) carries the behavioural contract (one-pass multi-error reporting, no coercion, no default-filling, `$ref` resolution within the lowered per-query document), and `implementation-notes.md` explicitly relegates the concrete library to a non-normative hint ("**Implementation hint (non-normative).** The reference implementation uses AJV v8…"). The validation-error rendering is likewise pinned as validator-neutral: `query/query-failure-and-repair.md` defines `<ajv-summary>` as "the **validator-neutral** observable summary … independent of which validator produced the issues," and `schema-subset.md` reasons at the draft level ("AJV has no `maxDepth` keyword (the JSON Schema 2020-12 vocabulary omits it)").

Against that abstraction, normative prose throughout the corpus pins the concrete library by name. `schema-subset.md` states "The walk runs **before** AJV at each site," "the depth check lives in the `SchemaValidator` service (the same component that owns the **AJV** cache)," and "the per-query **AJV** compiled-validator cache key." `runtime-value-model.md` says rebuilding happens "after **AJV** validation against the lowered schema" and outbound JSON is produced "before **AJV** validation." The binder files repeat the pattern (the "*AJV-on-`args` class*" failure class, "**AJV** validation of the binder's `args` failed"). The caller-observable contract is "validates against JSON Schema 2020-12 semantics via the `SchemaValidator` service" — which validator implements that contract is an implementer's choice the spec elsewhere protects. Naming AJV in the normative body over-constrains implementation and contradicts the spec's own validator-neutrality statements.

Scope note: this finding covers only library-name leakage into *normative prose*. Where "AJV" is fused into a coined token or contract surface — the `<ajv-summary>` template placeholder, the `schema_keyword` "not a literal AJV keyword" enumeration note, or any `ValidationIssue` ordering behaviour that genuinely depends on a validator's traversal order — that is a separate concern: if traversal order is part of the ordering contract it must be pinned *behaviourally* under the relevant ordering REQ-ID (ERR-14), not by naming the library. Those token-level surfaces are out of scope here.

## Solution Space

**Shape:** single

### Recommendation

In normative prose, replace bare references to "AJV" with the validator-neutral term — "the JSON Schema 2020-12 validator," "the `SchemaValidator` service," or "the validator" where context is clear — at every site that merely names the validator. Concretely:

- `schema-subset.md`: "runs **before** AJV at each site" → "runs **before** the validator at each site"; "the same component that owns the AJV cache" → "the same component that owns the compiled-validator cache"; "the per-query AJV compiled-validator cache key" → "the per-query compiled-validator cache key." Keep the draft-level reasoning ("AJV has no `maxDepth` keyword") rephrased as "JSON Schema 2020-12 has no `maxDepth` keyword."
- `runtime-value-model.md`: "after AJV validation against the lowered schema" → "after validation against the lowered schema"; "before AJV validation" → "before validation."
- Binder/query/invocation/type-system files: rename the "AJV-on-`args`" failure class and prose ("AJV validation of the binder's `args` failed") to validator-neutral wording (e.g. "schema-validation-on-`args`").

Retain "AJV" only in the existing non-normative `implementation-notes.md` hint (and any other paragraph explicitly tagged non-normative reference-implementation guidance). Leave coined tokens and contract surfaces untouched: the `<ajv-summary>` template placeholder, the `schema_keyword` "not a literal AJV keyword" enumeration note, and `ValidationIssue` ordering — those are out of scope (the ordering contract, if validator-traversal-dependent, is fixed behaviourally under ERR-14, not here).

Edge cases for the fixer: the `<ajv-summary>` placeholder name is a wire/template token, not prose — do not rename it. Where prose says a value "is not a literal AJV keyword," restate it relative to JSON Schema 2020-12 keywords rather than deleting the carve-out, since `maxDepth` is genuinely Loom-coined and not a draft keyword.

## Consequence

**Severity:** advisory

Any conformant JSON Schema 2020-12 validator wired through the `SchemaValidator` seam satisfies the contract, so an implementer is not driven to wrong runtime behaviour by the AJV mentions. The cost is over-prescription plus an internal inconsistency: normative prose pins a specific library while the abstraction layer and the `<ajv-summary>` rendering rule assert validator-neutrality, leaving a reader unsure whether AJV is a hard requirement or a reference choice.
