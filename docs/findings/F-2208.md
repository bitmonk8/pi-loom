## Problem

The debounce window is pinned to the exact value `250 ms` in the step-5 paragraph (`#watcher-hot-reload-registration`) and echoed in the `#pic-49` and `#structural-changes-no-unregister` paragraphs. The spec routes the window through the injected `Clock` seam to make burst-coalescing assertions deterministic, but never states whether the specific value `250 ms` is part of the observable contract or an implementer tuning choice. An implementer cannot tell whether `200 ms` or `300 ms` would be conformant.

## Issue introduction
**Verdict:** mixed — the framing defect was introduced by a specific later commit; the prescription defect is present-since-inception.
**Introducing commits:** `6d1a949` (framing); none identified for the 250 ms constant (predates the topic-file split).
**History:** The corpus is git-tracked. `registration-steps.md` was created at `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set") containing only steps 1–4; it carried no watcher/hot-reload content. Step 5 — including the "Registers a chokidar file watcher" framing and the 250 ms window — was added to this file at `6d1a949` (2026-06-05, "pi-loom spec: resolve \"session-shutdown sub-step 5 hot-reload split\""), which relocated the hot-reload sub-step out of `session-shutdown-semantics.md` and re-homed it as the fifth item of the registration list (`git log -S "Registers a chokidar file watcher"` reports `6d1a949` as the sole introducing commit for this path). That relocation is the origin of the registration-step framing defect. The 250 ms debounce constant itself is older than the topic-file split: `git log -S "250 ms"` traces it to the earliest spec/review-consolidation commits (e.g. `b87d4a6` "Add consolidated enriched spec review"), so the constant has been present since the spec corpus's inception and was carried verbatim through both `f5e89f4` and the `6d1a949` relocation rather than introduced by any single later edit.

## Solution approach

Clarify in the step-5 paragraph (`#watcher-hot-reload-registration`) whether the `250 ms` debounce window is a fixed contracted value or an implementer tuning choice, and apply that resolution consistently to the `#pic-49` and `#structural-changes-no-unregister` occurrences of the constant. If contracted, state that the window is a fixed value the conformance fixture asserts against. If tunable, restate the requirement behaviourally against the injected `Clock` seam (`#clock--fakeclock-interface`) rather than as the literal `250 ms` boundary.
