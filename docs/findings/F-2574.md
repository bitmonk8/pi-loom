## Problem

The hot-reload invalidation set is specified as "the changed file and every transitive `.warp` importer" in implementation-notes.md (Static-resolution load pass bullet), and registration-steps.md re-parses "the changed file plus every transitive `.warp` importer reached through the import graph", where step 5 (`id="watcher-hot-reload-registration"`) maintains a "`.warp` import graph". Read literally, the invalidation set is restricted to importers that are themselves `.warp` files. But the import relation is asymmetric: `.loom` files import `.warp` files and are never importable from each other (imports.md), so a `.loom` always sits at the root of its import chain. When a `.warp` schema or `fn` signature changes, the transitively-importing `.loom` entries are excluded from the invalidation set and keep stale cached type information (cross-loom return-type inference, `invoke(...)` argument-type checks, lowered query schemas, AJV validators) until a full reload.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** d38ffdb ("spec: hot-reload via in-process LoomRegistry swap; no ctx.reload() for content edits (Decision #8)", 2026-05-05)
**History:** The "transitive `.warp` importer" phrasing entered with the hot-reload mechanism itself in d38ffdb; `git log -S 'transitive \`.warp\` importer'` returns only that commit, so no later edit altered it. That commit's own message frames the design as tracking "`.warp` transitive importers via in-memory import graph", confirming the `.loom`-importer exclusion was present from the feature's introduction rather than a regression. The same commit rewrote the PIC registration step end-to-end, propagating the identical `.warp`-only phrasing to `registration-steps.md`.

## Solution approach

Rewrite the invalidation-set phrasing in implementation-notes.md's Static-resolution load pass bullet and in registration-steps.md's Hot-reload subsystem paragraph so the re-parse / validator-recompile / `pi.registerTool` staging set covers every file — `.warp` or `.loom` — that transitively imports the changed file. Redefine the "`.warp` import graph" at registration-steps.md's `watcher-hot-reload-registration` anchor as an import graph that records `.loom`→`.warp` edges and resolves a changed `.warp` to the root `.loom` entries in `LoomRegistry`. Keep both pages' phrasing aligned.
