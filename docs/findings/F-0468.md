## Finding

The cross-cutting governance note `docs/plan_topics/real-host-smoke-gate.md` names its own subject mechanism two different ways. The file title and filename call it the **"manual real-host smoke gate"** (`# Manual real-host smoke gate (release-gate / version-bump governance)`, filename `real-host-smoke-gate.md`), while the body's formal-definition sentence and every prose cross-reference call it the **"manual real-host smoke governance protocol"** (line 3 "It is the home of the **manual real-host smoke governance protocol**", line 5 "This note defines the **manual real-host smoke governance protocol** — a non-automated pre-merge gate", `docs/plan.md` line 226, and `H4a-factory-shell-and-harness.md` line 7). A third variant, "manual real-host smoke **run**", appears in `H6a-live-corpus-activation.md` line 5 to name a single execution.

Line 5 makes the conflation explicit: it defines "the manual real-host smoke governance **protocol**" as "a non-automated pre-merge **gate**", equating the two nouns in one sentence. There is no single canonical term a reader can search on, and the "gate / protocol" pair reads as if it might name two distinct artefacts on first encounter when it names one. The "governance protocol" prose actually outnumbers the "gate" usage (four prose occurrences vs the title/filename), so the file title is the outlier rather than the canonical anchor.

This is a wording-consistency defect in a governance note read by the bump/merge contributor and the `H6a` release-gate auditor; it does not change any acceptance criterion or implementable behaviour.

## Solution Space

**Shape:** single

### Recommendation

Adopt one canonical noun phrase for the mechanism and use it consistently across the note and its cross-references. The filename `real-host-smoke-gate.md` is linked from `H4a`, `H6a`, `H7a`, `V18d`, and `plan.md`, so renaming the file would force a propagation pass through every link; the lower-cost canonical anchor is the title/filename term **"manual real-host smoke gate"**.

Concretely:

- In `docs/plan_topics/real-host-smoke-gate.md`, change the line-3 phrase "the **manual real-host smoke governance protocol**" and the line-5 phrase "This note defines the **manual real-host smoke governance protocol**" to use **"manual real-host smoke gate"**. Line 5's "— a non-automated pre-merge gate" apposition can stand as the gloss.
- In `docs/plan.md` line 226, change "The **manual real-host smoke governance protocol** the bump/merge contributor runs…" to "The **manual real-host smoke gate**…".
- In `docs/plan_topics/H4a-factory-shell-and-harness.md` line 7, change "The manual real-host smoke governance protocol that gates a double-vs-real divergence…" to "The manual real-host smoke gate…".

`H6a`'s "manual real-host smoke **run**" names a single execution of the gate, a legitimately distinct referent; leave it unless the same edit pass wants to write "a **run** of the manual real-host smoke gate" for explicitness (option-dependent). The spec is read-only for this fix — no spec file uses either term.

## Consequence

**Severity:** cosmetic

A reader scanning for the mechanism cannot rely on one canonical term: the file is titled "gate", its definition calls it a "governance protocol", and `H6a` calls executions a "run". No implementer diverges and no acceptance criterion shifts; the cost is reader friction and the momentary ambiguity of whether "gate" and "governance protocol" are one artefact or two.
