## Finding

The *Transient toasts (auxiliary)* paragraph at `diagnostics/diagnostic-shape.md#transient-toasts` carries a cluster of normative MUST/MUST-NOT obligations that, by construction, carry **no** `loom/parse|load|runtime/*` registry code (the paragraph explicitly forbids routing coded diagnostics through this surface) and no numbered `DIAG-N` REQ-ID. Concretely, the paragraph pins:

- three failure modes routed through `ctx.ui.notify(message, "error")` — the chokidar watcher itself throwing, an unrecoverable settings-I/O exception that no `loom/load/settings-*` code covers, and an internal extension invariant violation;
- a swallow MUST — a synchronous throw from `ctx.ui.notify` on this last-resort surface is swallowed (mirroring the `console.error`-throw swallow rule) and **no further diagnostic is emitted** on the inner throw;
- three per-mode post-failure pins — **none** of the three arms writes the `LoomRegistry` drain-state tag (a toast-surface write would trip the `session_shutdown` handler-entry short-circuit); the chokidar-throw arm leaves the file-watching subsystem torn down while the registry stays dispatchable via arm (a); the settings-I/O arm retains last-known-good settings and keeps the settings watcher armed; the invariant-violation arm leaves the registry live and dispatchable with no drain-state write.

None of these obligations is mapped in `coverage-matrix.md`. The *Code-keyed obligation areas (no numbered REQ-IDs)* table (`cka-1` … `cka-48`) contains no row whose subject is `diagnostic-shape.md#transient-toasts`; sibling diagnostics-area rows cover adjacent but distinct surfaces (`cka-29` → `diagnostic-emission-isolation.md` teardown `console.error` isolation at `V9g`; `cka-22` → `drain-state-contract.md` `readDrainState` three-arm dispatch at `V9m`). The two existing `ctx.ui.notify` references in the plan are different reach paths: `V7d`/`V7d-T` and `V9p`/`V9p-T` reference the `ctx.ui.notify` arm of the persistent-channel *System-notes fallback chain* (`pi.sendMessage` throws → `ctx.ui.notify` → `loom/runtime/system-note-delivery-failed` → `console.error`), not the auxiliary bookkeeping surface this paragraph governs.

Because `diagnostic-shape.md` is a non-narrative page (it carries `DIAG-1` … `DIAG-4`), its un-anchored MUSTs are in scope for the `H5e` un-anchored-normative-MUST text-scan arm, which reddens when a `spec_topics/**` normative MUST carries no `PREFIX-N` REQ-ID, no `loom/...` code, and is absent from the *Code-keyed obligation areas* table with a closing leaf. The transient-toast obligations satisfy that trigger and are mapped nowhere.

## Solution Space

**Shape:** single

### Recommendation

Add one row to the *Code-keyed obligation areas (no numbered REQ-IDs)* table in `docs/plan_topics/coverage-matrix.md` whose *Token* is the next free `cka-<n>`, whose subject is `diagnostics/diagnostic-shape.md` §*Transient toasts (auxiliary)* (anchor `#transient-toasts`, tagged `un-anchored; GOV-22 residue`), and whose *Closing leaf(s)* cell names `V10d`. Extend `V10d`/`V10d-T` so the leaf asserts, with Tests bullets that cite the new `cka-<n>` token and `V10d` inline (per `H5f`):

- each of the three failure modes routes through `ctx.ui.notify(message, "error")`: a chokidar-watcher throw, a settings-I/O exception not covered by any `loom/load/settings-*` code, and an internal extension invariant violation;
- a synchronous throw out of `ctx.ui.notify` on this surface is swallowed and emits no further diagnostic (the toast surface is the bottom of the stack for these failures);
- none of the three arms writes the `LoomRegistry` drain-state tag;
- the chokidar-throw arm leaves the file-watching subsystem torn down while the registry stays dispatchable via `readDrainState` arm (a);
- the settings-I/O arm retains the last-known-good settings cache and keeps the settings watcher armed (per-event, self-clearing);
- the invariant-violation arm leaves the registry live and dispatchable, recovery via `/reload`.

`V10d` is the natural home: it already owns the watcher reload path and the `ReloadFailureInjector` arm set (registry-swap / re-parse / settings-re-merge), so modes (1) and (2) sit on seams it already drives. If folding all three modes plus the swallow MUST into `V10d` would over-grow it, allocate a `<new>` dedicated leaf instead and point the new `cka-<n>` row at it; either way the row and its closing leaf's citing test must land together so `H5e` and `H5f` both pass.

Edge case for the implementer: mode (1)'s chokidar-error case overlaps PIC-55's *continues-delivering* transient-toast arm at `registration-steps.md#pic-55`. Keep this row's mode-(1) assertion scoped to the auxiliary-surface bookkeeping behaviour (swallow + no-drain-state + subsystem-state pin) and let the PIC-55 watcher-lifecycle posture (continues-delivering toast vs terminal `loom/runtime/watcher-terminated` note) close under its own mapping, so the two do not double-claim the same obligation.
