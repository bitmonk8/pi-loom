## Finding

The runtime registry table in `docs/spec_topics/diagnostics/code-registry-runtime.md` declares its rightmost column as `Message template`. The shared column legend that governs all four registry pages — defined in `docs/spec_topics/diagnostics/code-registry-parse.md` ("The column legend below applies to all four tables. … *Message* is the rendered author-facing string template …") — names that column `Message`. The two other continuation tables that follow the legend, `code-registry-parse.md` and `code-registry-load.md`, both spell the header `Message`. The runtime page is the sole deviant among the legend-governed three.

The drift is also load-bearing for `DIAG-4` in `plan_topics/V7b-code-registry.md` and `plan_topics/V7b-T-code-registry.md`: that test asserts that "the **Message** column is normative — every asserting test sources its expected string from the registry, not prose." A literal column-name reader of the runtime table will not find a column named `Message`, weakening the closing gate's traceability and forcing readers to silently equate `Message` with `Message template`.

(The host table's deviation — `Message` is spelled the same there, but two other headers differ — is the subject of a sibling finding; this finding is scoped to the runtime page's column-header drift only.)

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `839db71` — *pi-loom spec: resolve "Diagnostic codes are declared as namespaces but never assigned to specific errors"* (2026-05-04)
**History:** Commit `839db71` created the initial registry as three tables inside the then-monolithic `spec_topics/diagnostics.md`. The parse and load tables were authored with the header `Message`, and the runtime table was authored with `Message template` in the same commit — so the divergence existed from the moment the tables existed. The `loom/host/*` table was added later in `9559e19` (2026-05-07) with its own header drift (`Message` is spelled the same there, but the other column names diverge — handled by a sibling finding). The corpus-wide split in `f5e89f4` (2026-06-04) moved each table to its own file but did not touch the header rows. The column legend was reworded in `3059a0e` (2026-06-05) to explicitly declare it "applies to all four tables" without reconciling the runtime page's deviant header. No subsequent commit has touched the runtime table's header row.

## Solution Space

**Shape:** single

### Recommendation

Rename the runtime table's rightmost column header from `Message template` to `Message`, matching the legend prose in `code-registry-parse.md` and the sibling parse/load tables. Edit is confined to the table header line of `docs/spec_topics/diagnostics/code-registry-runtime.md`; no cell contents and no surrounding prose change.

Implementer notes:

- The introductory paragraph above the runtime table refers to "its Message template" when discussing placeholder interpolation; this phrase is using *message template* as a common noun (the rendered string is, factually, a template), not naming the column. It can stay as-is, or be rephrased to "its Message" if the reviewer prefers the phrase to mirror the column name verbatim — neither version creates a fresh inconsistency.
- No other registry pages or downstream cross-references key on the literal string `Message template`; a corpus grep confirms the token appears only in this one header cell.

## Consequence

**Severity:** cosmetic

A test author wiring `DIAG-4` against the runtime registry must either rename the column header or special-case the runtime page in the registry-extraction routine; readers cross-checking against the legend hit a one-page discontinuity. No runtime, contract, or wire behaviour is affected.
