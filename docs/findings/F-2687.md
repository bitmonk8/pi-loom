## Problem

Version-bump checklist item (a) (`#bump-checklist-instance-survival`) records a one-shot open contradiction — whether loom's package-loaded `extensions/index.ts` survives a session-only `session_shutdown` or is torn down and rebound per Pi's documented lifecycle. The spec defines the machinery thoroughly (the two-verdict recording shape under `Outcome recording conventions`, the clause-(a) narrative, and the SM-3b/SM-4/SM-5/SM-6 and `session-only-degraded-state.md` gate markers that block on `provably-differs`), but it never states what loom 1.0 does at ship if neither verdict has been authoritatively recorded. Two readings are equally consistent with the current text: recording a verdict is a 1.0 release-blocking gate, or 1.0 may ship with the degraded-state branch descoped and SM-4/5/6 deferred. The spec corpus encodes no choice; only the plan's `§Blocked obligations` fallback silently picks the descope reading, leaving the normative spec and the plan out of step on a scope question.

## Solution approach

Add a statement of the loom-1.0 release disposition for an unresolved item (a) at the version-bump-step2.md item (a) site (`#bump-checklist-instance-survival`) and mirror it at host-prerequisites clause (a) (`#degraded-state-host-prerequisites`). Adopt the disposition the plan has already pre-designed in `plan.md §Blocked obligations` so spec and plan agree: that a recorded verdict is not a 1.0 release-blocking gate, the unresolved-at-ship state resolves to the descoped full-teardown behaviour, and SM-3b/SM-4/SM-5/SM-6 are deferred. Forward-link the SM-3b/4/5/6 gate markers and the `session-only-degraded-state.md` gate blockquote to that disposition so none implies a release-blocking reading.

## Solution constraints

- Do not add a verdict to item (a): the open question stays genuinely open; only its release disposition is being pinned.
