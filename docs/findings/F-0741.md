## Finding

The DISC-6 paragraph in [`discovery/package-and-settings.md`](../../../docs/spec_topics/discovery/package-and-settings.md) ends the per-read-timeout sentence with a parenthetical that fuses two clauses of different kinds under a single MUST: *"the in-flight read is abandoned (the file handle is dropped and GC'd; the read's eventual settlement MUST be silenced with `.catch(() => {})` and its result MUST NOT be re-routed back into the discovery pass)"*. The second clause ("MUST NOT be re-routed back into the discovery pass") is a behavioural rule with an observable consequence ŌĆö a conforming implementation can be falsified by a test that arranges the abandoned read to resolve late and asserts the walk's outcome is unchanged. The first clause ("MUST be silenced with `.catch(() => {})`") is an implementation-strategy assertion: it names a specific JavaScript idiom for attaching a no-op rejection handler, with no observable difference between an implementation that attaches `.catch(() => {})`, an implementation that wraps the abandoned read in `try { await ŌĆ” } catch {}`, an implementation that swallows via an `AbortController`-derived rejection, or an implementation that hands the dangling settlement to a registered `unhandledRejection` sink.

The same paragraph already takes care to leave the racing construct unprescribed ŌĆö *"the implementation construct used to await whichever resolves first is left to the implementer (e.g. an `AbortController` derived from `Clock.setTimeout`)"* ŌĆö so freezing the discard idiom is internally inconsistent with the prescription level chosen one sentence earlier. It also leaks past the spec's seam: the genuine obligation behind the `.catch(() => {})` clause is "the abandoned read MUST NOT crash the extension via an unhandled-rejection escape and MUST NOT be re-routed back into the discovery pass." Stating that behavioural obligation lets any of the above implementation strategies satisfy DISC-6 and gives V10b a falsifiable hook (assert no `unhandledRejection` event fires when the deadline races a delayed read); pinning the idiom does neither.

## Solution Space

**Shape:** single

### Recommendation

Replace the `.catch(() => {})` MUST with a behavioural MUST that captures the actual obligation, and demote the idiom to a parenthetical example matching the prescription level the same paragraph uses for the racing construct. Concretely, rewrite the parenthetical in DISC-6 from

> the in-flight read is abandoned (the file handle is dropped and GC'd; the read's eventual settlement MUST be silenced with `.catch(() => {})` and its result MUST NOT be re-routed back into the discovery pass)

to

> the in-flight read is abandoned (the file handle is dropped and GC'd; the read's eventual settlement MUST NOT escape as an unhandled rejection and its eventual result MUST NOT be re-routed back into the discovery pass ŌĆö e.g. attach a no-op rejection handler such as `.catch(() => {})`, or swallow the rejection via the cancellation mechanism the implementation chose above)

This keeps the two genuinely observable obligations (no process-level unhandled-rejection escape; no re-routing of the late result), removes the idiom-level MUST, and matches the prescription level the same paragraph applies to the deadline-race construct one sentence earlier. The V10b / V10b-T **Tests** lists gain one DISC-6 sub-test that drives a deadline-triggered read to settle late via `FakeClock` and asserts (a) no `process` `unhandledRejection` event fires and (b) the walk's emitted diagnostics and discovered-loom set are identical to a run where the read never settles. Implementer edge case: the assertion against `process.on('unhandledRejection', ŌĆ”)` must register the listener before the deadline fires (Node only queues the event on the next microtask after rejection), and must un-register in the test's teardown so it does not leak across the suite.

## Consequence

**Severity:** advisory

Two reasonable implementers reach the correct externally observable behaviour by different mechanisms ŌĆö `.catch(() => {})`, `try { await } catch {}`, `AbortController` cancellation, or a process-level `unhandledRejection` sink ŌĆö but only the first literally satisfies the MUST as written, so a conformance reviewer reading the spec strictly may red-flag a correct implementation. Conversely, an implementer who attaches `.catch(() => {})` but lets the resolved value flow back into a result accumulator satisfies the over-prescribed MUST while violating the genuine intent. Neither failure mode produces wrong runtime behaviour today, but both produce confused review traffic, and the second drains the testability the paragraph otherwise has.
