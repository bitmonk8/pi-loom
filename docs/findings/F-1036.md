## Problem

In the **Subagent state-isolation matrix** in `docs/spec_topics/pi-integration-contract/subagent.md`, the `params` cell ("captured `params` (load-time per-invocation values)") sits in the "Inherited from the loom's frontmatter" column. This misclassifies the cell two ways. The "load-time per-invocation values" wording is internally contradictory — `params` are validated by AJV at invocation time and supplied by the caller (binder, `invoke(...)` arguments, or the model's typed tool-call payload), not captured at load or frozen per loaded file, per the authoritative semantics in [Parameters and Frontmatter — `params`](../frontmatter/frontmatter-fields-a.md). And placing `params` in the frontmatter-inheritance column mis-implies the subagent inherits caller-independent frontmatter defaults — only the schema and defaults live in frontmatter; the values come from the invoking caller — which conflicts with the third column's own "caller's `params` and variable bindings" row.

No later edit has touched the cell. The defect is therefore co-extensive with the matrix itself — there is no earlier, correct version to revert to.

## Solution approach

Rewrite the `params` cell so its value semantics match the invocation-time, caller-supplied rule pinned in [Parameters and Frontmatter — `params`](../frontmatter/frontmatter-fields-a.md), dropping the words "captured" and "load-time" (both belong to the `tools:` [Resolution snapshot](../frontmatter/frontmatter-fields-b-and-templates.md#resolution-snapshot), not to `params`). Split the cell across the columns it straddles: keep the `params` schema and defaults in the frontmatter-inheritance column, and place the per-invocation values in the caller-forwarding column as AJV-validated caller-supplied arguments. Cross-reference [Parameters and Frontmatter — `params`](../frontmatter/frontmatter-fields-a.md) so the cell stays a summary.
