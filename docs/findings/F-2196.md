## Problem

The per-provider Overflow-signature table on `provider-error-mapping.md` expresses one matching operation in two vocabularies: the `anthropic-messages` and `openai-completions` rows match `errorMessage` (i.e. `AssistantMessage.errorMessage`), while the `mistral` and `amazon-bedrock` rows match "body" — a term never defined at or near the table. Only the `#classifier-input-surface` note, several paragraphs above, ties "body" to `errorMessage`; read in isolation the table invites matching against a parsed JSON error body that the spec asserts pi-ai does not surface, so two implementers can produce divergent classifiers. The `amazon-bedrock` row carries a second, independent ambiguity: it leads with `ValidationException` rather than an HTTP-status condition, leaving unstated whether `ValidationException` is a required substring precondition of the match or a descriptive label for the AWS error type. Because `amazon-bedrock` is the SDK-only path the `#classifier-input-surface` note classifies as network-level absent an `onResponse` firing, that row's predicate is the sole gate promoting such a response to `ContextOverflowError`, making the precondition question behaviourally load-bearing.

## Issue introduction
**Verdict:** introduced by a specific commit (incomplete edit)
**Introducing commit:** `513a5cc` (2026-06-09) — *resolve "overflow table predicates key on structured JSON fields errorMessage does not expose"*
**History:** `provider-error-mapping.md` was extracted in the spec-split commit `f5e89f4` (2026-06-04), at which point all four rows used structured-field framing (`anthropic-messages` keyed on `error.type`/`error.message`, `openai-completions` on `error.code`, and `mistral`/`amazon-bedrock` on "body matching"). Commit `513a5cc` (verified via `git log -S"errorMessage\` matching"` and `git show 513a5cc`) rewrote the `anthropic-messages` and `openai-completions` rows — and the *Overflow token-count extraction* prose — to read `errorMessage`, but left the `mistral` and `amazon-bedrock` rows untouched on the older "body matching" wording. The terminology split this finding reports is the residue of that partial conversion; the "body matching" phrasing for the two untouched rows predates the split (carried in at `f5e89f4`). The `ValidationException`-as-lead-token shape on the `amazon-bedrock` row also predates `513a5cc` and was never normalised.

## Solution approach

Rewrite the `mistral` and `amazon-bedrock` rows of the Overflow-signature table to express their match against `errorMessage` (the `AssistantMessage.errorMessage` surface the `#classifier-input-surface` note already names), bringing them onto the same vocabulary the other two rows use rather than the undefined "body". For the `amazon-bedrock` row, also resolve the `ValidationException` token: state explicitly whether it is a required substring precondition the matcher must check or a descriptive label for the AWS error type with no matching role.

## Solution constraints

- Keep the `amazon-bedrock` row's `errorMessage` predicate as the sole classification signal for that path; do not introduce an HTTP-status precondition, which the SDK-only `amazon-bedrock` path (resolving with `stopReason: "error"` and typically no `onResponse` firing) cannot supply.
