## Problem

Seven sites across `docs/spec_topics/` write the literal type `Result<_, QueryError>`, but `_` is not admissible in a `Type` position: per [Grammar Appendix — Type grammar](../../docs/spec_topics/grammar.md#type-grammar) the `Type` production does not enumerate `_`, and per [Lexical Structure](../../docs/spec_topics/lexical.md) `_` is the reserved discard binding that "cannot be used in expression or type position." The sites use `_` informally as a wildcard metavariable, but the spec has no such convention. The two `code-registry-parse.md` rows are the sharpest case: DIAG-4 makes the registered *Message* column byte-normative, so every conforming runtime must emit a diagnostic whose own text contains a token sequence the grammar rejects.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `53f5831` (2026-05-03, *spec: rewrite query primitive, system prompts, and schema syntax*)
**History:** `git log --reverse --all -G 'Result<\s*_\s*,\s*QueryError\s*>'` shows the literal `Result<_, QueryError>` was introduced into `spec.md` by `53f5831` as part of the original `Result` / `QueryError` / `?`-operator design — both the `?`-operator gate and the function-return paragraph carried the `_`-in-type-position from that first day. `8eea523` (2026-05-04, *spec: fill implementation-blocking gaps*) added a third occurrence in the same monolithic file. `fecb504` (2026-05-04, *Split spec.md and plan.md into per-topic / per-phase files*) moved the offending sentences into `spec_topics/functions.md` and `spec_topics/errors-and-results.md` without altering them. The grammar productions that make the violation textually visible were tightened later — `1c552ff` (2026-06-04, *resolve "Type grammar admits no generic application beyond array<T>"*) added the `GenericType ::= "Result" "<" Type "," Type ">"` line — but the `Type` production already excluded `_` in the May commits via the absence of `_` from `NamedType ::= Ident` and the lexical reservation of `_`. No commit ever introduced `_` as a type-position metavariable convention.

## Solution approach

Sweep every `Result<_, QueryError>` in `docs/spec_topics/` to `Result<T, QueryError>`, introducing `T` with a `where T is …` binding clause matching the convention already in force in `errors-and-results/error-model.md` and `tool-calls.md`. Rewrite the byte-normative *Message* and the *Description* / *Hint* columns of the `loom/parse/question-outside-result-fn` and `loom/parse/question-on-non-result` rows in `diagnostics/code-registry-parse.md` to the wildcard-free `Result<T, QueryError>` form.

## Solution constraints

- In `error-model.md`'s `?` gate, keep the success-payload binder `T` and the enclosing-return binder `R` distinct; do not collapse `Result<T, QueryError> ⊑ R` to a single binder.
- Do not introduce a type-position metavariable meaning for `_`; it stays reserved as the discard binding per [Lexical Structure](../../docs/spec_topics/lexical.md).
