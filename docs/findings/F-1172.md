## Finding

Seven sites across the spec write the type `Result<_, QueryError>` literally â€” `functions.md` (the `?`-requires-`Result`-return paragraph), `errors-and-results/error-model.md` (the `?`-operator gate and ERR-18 operand precondition), `expressions.md` (the postfix-`?` row in the operator table), `invocation.md` (the InvokeInfraError/InvokeCalleeError naming paragraph), and two rows of `diagnostics/code-registry-parse.md` (the `loom/parse/question-outside-result-fn` row â€” twice, in *Description* and the registered *Message* â€” and the `loom/parse/question-on-non-result` row). The token `_` is not admissible there. Per [Grammar Appendix â€” Type grammar](../../docs/spec_topics/grammar.md#type-grammar), `Result` is a `GenericType` constructor of arity 2 whose two arguments are `Type`s, and the `Type` production enumerates `PrimitiveType | NamedType | GenericType | ObjectType | Type "|" Type | LiteralType` â€” `_` is in none of them. Per [Lexical Structure â€” Reserved keywords](../../docs/spec_topics/lexical.md), `_` is the reserved discard binding, "not a regular identifier and cannot be used in expression or type position." So the literal string `Result<_, QueryError>` would fail `loom/parse/reserved-keyword-as-identifier` (or its discard-binding analogue) if any author copy-pasted it into a return annotation.

The sites are using `_` informally as a wildcard metavariable â€” "some unspecified success payload" â€” but the spec has no metavariable convention that allows `_` in a `Type` position, and several of the sites are themselves the normative pin for the rule (the `?` gate, ERR-18) where the syntactic confusion bites hardest. `code-registry-parse.md` makes it worse: the registered *Message* column for `loom/parse/question-outside-result-fn` is `'?' used in a scope whose return type is not Result<_, QueryError>`, and DIAG-4 makes that string byte-normative â€” every conforming implementation has to emit a diagnostic message that purports to name a Loom type but is not a syntactically valid one.

The neighbouring conventions already used by these same paragraphs supply the right fix: `error-model.md` immediately below the offending sentence writes `"implicitly returns `Result<T, QueryError>` where `T` is the type of its last expression"`, and `tool-calls.md` writes `"`Result<T, QueryError>` where `T` is the callee's inferred return type"`. The `T`-with-binding-clause convention is the one already in force.

## Solution Space

**Shape:** single

### Recommendation

Sweep every `Result<_, QueryError>` in `docs/spec_topics/` to `Result<T, QueryError>` and, at each site, introduce `T` with the same `where T is â€¦` binding clause the spec already uses in `error-model.md` (`"where `T` is the type of its last expression"`) and `tool-calls.md` (`"where `T` is the callee's inferred return type"`). The seven sites and their bindings:

1. `functions.md` (`?`-requires-`Result`-return paragraph) â€” `Result<T, QueryError>` *for some `T`* (the inferred-or-annotated payload type; the existing paragraph below already explains the inference).
2. `errors-and-results/error-model.md` (`?` gate) â€” `compatible with `Result<T, QueryError>` for some `T`` and `Result<T, QueryError> âŠ‘ R`.
3. `errors-and-results/error-model.md` (`ERR-18` operand precondition) â€” `MUST itself have Loom static type `Result<T, QueryError>` for some `T``.
4. `expressions.md` (postfix-`?` row) â€” `admitted only on a `Result<T, QueryError>`-typed operand for some `T``.
5. `invocation.md` (uniform-error-type paragraph) â€” `a single `Result<T, QueryError>` return type` (no binding clause needed; the sentence's point is the error arm).
6. `diagnostics/code-registry-parse.md` (`loom/parse/question-outside-result-fn` row): change *Description* to `â€¦ whose return type is not `Result<T, QueryError>` for any `T``, change *Hint* to `Declare the enclosing scope's return type as `Result<T, QueryError>``, and change the registered *Message* to `'?' used in a scope whose return type is not Result<T, QueryError>`.
7. `diagnostics/code-registry-parse.md` (`loom/parse/question-on-non-result` row) â€” change *Description* to `â€¦ whose Loom static type is not `Result<T, QueryError>` for any `T``. The registered *Message* (`'?' requires a Result operand; got <type>`) is already wildcard-free and needs no change.

In parallel, update the `Result<_, QueryError>` quotation in `docs/plan_topics/V4a-match-result.md` and `docs/plan_topics/V4a-T-match-result.md`'s **Tests** bullets so the plan does not re-encode the defect.

Edge cases for the implementer:

- The `Result<T, QueryError> âŠ‘ R` rendering in `error-model.md`'s `?` gate must keep `T` and `R` as distinct binders â€” `T` is the success payload, `R` is the enclosing return annotation; do not collapse them.
- Do *not* introduce a new spec-wide "underscore-as-type-metavariable" convention as a shortcut â€” `_` is already reserved as the discard binding and adding a second meaning in type position would create exactly the ambiguity the finding identifies.
- The change to the byte-normative *Message* in `code-registry-parse.md` is a spec-versioned wording change under DIAG-4; tests that assert on the rendered message must be updated in lock-step (the `V7b-T` diagnostics suite is the natural home).

## Consequence

**Severity:** correctness

The seven prose sites are recoverable by a careful reader, but the two `code-registry-parse.md` rows are not: DIAG-4 makes the *Message* column byte-normative, so every conforming runtime must emit a diagnostic whose own text contains a token sequence the grammar rejects. A conformance test that round-trips the message through the parser is the natural way to catch defects of this exact shape, and it would fire on its own spec.
