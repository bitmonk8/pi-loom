## Problem

`V2c` (`docs/plan_topics/V2c-value-model.md`) bundles three concerns into one leaf: the JS representation of loom values, structural `==` / `!=` equality, and the inbound/outbound wire-name translation boundary. The first two are pure interpreter-internal value semantics depending only on the type grammar (`V2a`); the third is a validation-boundary operation that needs `V5f`'s per-schema sidecar and an understanding of the AJV validation flow — a distinct mental model. Because the bundle pulls the whole leaf onto the schema-subset machinery, `V3a` and `V3e` — which depend on `V2c` only for value representation and equality — inherit a spurious transitive dependency on the schema reject gate and lowering pass (`V5d`/`V5f`) that their actual needs do not require.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e — pi-loom plan: build/update plan for spec.md + review (2026-06-10, Thomas Andersen)
**History:** `docs/plan_topics/V2c-value-model.md` has exactly one commit in its `git log --follow` history, c6a664e — the plan-genesis commit that built the leaf corpus from `spec.md`. The leaf bundled value representation, structural equality, and the wire-name translation boundary in that first authoring, and no later commit (including the many subsequent `pi-loom plan: resolve ...` fix commits that split other over-large leaves such as `V13c`, `V11f`, `V11b`, and `H5a`) touched the file. The over-bundling has therefore been present since the leaf entered the corpus.

## Solution approach

Split `V2c` into a value-representation-plus-equality leaf and a wire-name-translation leaf, each with its paired `-T` tests task per the TDD ritual. Point the value leaf's `Deps.` at `V2a` (no schema-pipeline dependency) and the wire leaf's `Deps.` at the sidecar producer. Retarget consumers off the removed `V2c` ID: `V3a` and `V3e` point to the value leaf, and `V13a` points to the wire leaf (plus the value leaf where it observes equality/representation). Update the RVM row in `docs/plan_topics/coverage-matrix.md` so its closing-leaf set is the union of the two new leaves.

## Solution constraints

- The spec topics (`runtime-value-model.md`, `schema-subset.md`) are read-only for this fix — the re-partitioning is internal to the plan corpus.
- Honour the leaf-ID scheme and `idAllocationPolicy` per `## Plan rules` (no reused/retired IDs, no numbering hole, no dangling `V2c` reference, each new leaf carries its `-T` pair).
- The `V5f` dependency edge attaches to the wire-name-translation leaf only, never the value-representation leaf — landing it on the value leaf re-introduces the schema coupling this split removes.
