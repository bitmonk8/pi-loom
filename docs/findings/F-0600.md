## Finding

The *Per-family record-shape table* in [`audit-failures.md`](../../docs/spec_topics/pi-integration-contract/audit-failures.md) pins the `proposed-resolution` value a family-(5) violation record MUST carry. That cell describes "whichever of [step 2(b) branch (4)]'s five sub-cases the marker falls under" and then enumerates five sub-cases inline. Only the last one carries an explicit ordinal — the parenthetical `(v)` immediately before *delete the marker and resolve the underlying surface via branch 1/2/3 (no-exemption-intended, malformed-marker only)*. The first four (`no-surface-on-line`; `surface-now-in-inventory`; `rewrite the marker's citation`; `rewrite the marker's grammar`) are presented as a comma-separated run with no `(i)`, `(ii)`, `(iii)`, `(iv)` ordinals.

This is structurally inconsistent within a single sentence and breaks two downstream contracts the spec itself sets up. First, the *Failure-surface contract* requires every negative-test fixture to assert that the `<family>` token it emits matches the leaf-side routing table for the sub-case the fixture seeds, and the prose above the table calls out family (5) as having "nine `<symptom>` tokens total: both stale sub-kinds (s1) and (s2) plus all seven malformed-marker clauses (a)–(g)" — the same enumerative-token discipline (s1/s2; a–g) is precisely what the sub-case run inside the family-(5) `proposed-resolution` cell omits. Second, the *Note on family (5) routing.* paragraph and [step 2(b) branch (4)](../../docs/spec_topics/pi-integration-contract/version-bump-step2b.md) both refer back to *sub-case (v)* by that ordinal as the *no-exemption-intended, malformed-marker only* arm; the cross-reference is well-formed but its antecedent enumeration is not, so a reader navigating from those references to the table cell cannot mechanically check that sub-case (v) is indeed the fifth bullet, nor identify sub-cases (i)–(iv) for symmetric cross-reference.

The cell is otherwise the normative description of the `proposed-resolution` field's value range on family-(5) wire records; an audit implementer pinning fixture assertions against sub-case identifiers (and a CI parser keying on the sub-case token a `proposed-resolution` field carries) has no closed enumeration to bind to for the first four arms.

## Solution Space

**Shape:** single

### Recommendation

Add explicit `(i)` / `(ii)` / `(iii)` / `(iv)` ordinals to the first four sub-cases inside the family-(5) `proposed-resolution` table cell, matching the existing `(v)` and the back-reference shape the *Note on family (5) routing.* paragraph already uses. Concretely, rewrite the cell's enumeration as:

> deletion-or-rewrite suggestion per [step 2(b) branch (4)] — naming whichever of that branch's five sub-cases the marker falls under: (i) delete the marker line (`no-surface-on-line` — covering both the surface-removed and the placement-error cases the *Stale sub-kinds* paragraph above collapses under the single (s1) `<symptom>` token); (ii) delete the marker (`surface-now-in-inventory`); (iii) rewrite the marker's citation; (iv) rewrite the marker's grammar; (v) delete the marker and resolve the underlying surface via branch 1/2/3 (`no-exemption-intended, malformed-marker only`).

Apply the same `(i)` through `(v)` ordinals symmetrically to step 2(b) branch (4) in `version-bump-step2b.md`, where the same five arms are described in narrative form without ordinals. Do not change the `<symptom>` token taxonomy (s1/s2 and the malformed-marker clauses a–g) and do not introduce a new wire-format obligation on the `proposed-resolution` field — the ordinal is an in-prose enumeration anchor, not a new field value.

Edge cases for the implementer:
- The cell rewrite must preserve the existing parenthetical commentary on (i) (the (s1) collapse) and on (ii) (the typebox-promotion arm) verbatim; only the leading ordinal is added.
- Step 2(b) branch (4)'s narrative ordering must match the table's ordinal assignment so a reader navigating from the table to the branch lands on the same numbered arm. The branch's existing prose ordering is (no-surface-on-line; surface-now-in-inventory; citation-rewrite; grammar-rewrite; no-exemption-intended-deletion), which already matches the recommendation above — preserve that order.
- The `(s1)` / `(s2)` stale-sub-kind tokens and the `(a)`–`(g)` malformed-marker clause letters live on a different axis (they are `<symptom>` tokens) and MUST NOT be conflated with the `(i)`–`(v)` sub-case ordinals being added here.

## Consequence

**Severity:** advisory

Two reasonable implementers do not diverge on observable wire output — the `<symptom>` token taxonomy (s1, s2, and the malformed-marker clauses) is closed elsewhere and is what the `<symptom>` field actually carries on the wire. They do diverge on how they parse a `proposed-resolution` field's free-text content and how the fixture suite names the five sub-cases, and a reader reconciling the *Note on family (5) routing.* paragraph's `(v)` back-reference against the table cannot mechanically locate the antecedent. The defect is editorial-shape consistency rather than a wire-format ambiguity.
