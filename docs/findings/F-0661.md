## Finding

The normative "Provider compatibility for typed queries" paragraph in `spec_topics/pi-integration-contract/conversation-drive.md` carries two pieces of non-normative content that do not belong in the paragraph that pins the supported-provider set and the unsupported-provider runtime behaviour:

1. An inline HTML comment between the "naming the model" clause and the loom-load behaviour sentence: `<!-- the warning's payload is exactly the model + provider carried by its diagnostics/code-registry-load.md message template; the former "listing the typed-query expression positions" clause was dropped as no positions format was ever pinned -->`. The first half restates that the diagnostic message template lives in `diagnostics/code-registry-load.md` (already authoritative there); the second half is a decision-log note recording why an earlier clause was removed — pure change-history, not a constraint a future implementer needs.
2. The paragraph's closing sentence, "Future widening (Gemini, OpenAI Responses, weak local models via JSON-mode fallback) is listed in [Future Considerations](../future-considerations.md).", duplicates the deferred-extension entry already pinned at `future-considerations/surface-extensions.md` (the *Typed-query support for providers without named-tool forcing* bullet), which lists the same three exemplar providers and the same JSON-mode / system-prompt fallback. The "loom 1.0 seam — typed-query supported provider set" blockquote immediately below the paragraph already cross-links to that Future Considerations entry as the seam's deferred-extension anchor, so the inline sentence is a third copy of a one-source-of-truth fact.

Both items widen the surface a future edit has to keep in sync without adding any normative constraint, and the inline future-provider list is the same divergence risk the corpus elsewhere keeps off the normative path by deferring exemplar lists to `future-considerations/surface-extensions.md`.

## Solution Space

**Shape:** single

### Recommendation

In the "Provider compatibility for typed queries" paragraph of `docs/spec_topics/pi-integration-contract/conversation-drive.md`:

- Delete the HTML comment in its entirety, including the surrounding semicolon space. The clause that precedes it ("naming the model") and the clause that follows it ("the loom still loads, …") stitch together as a single sentence; verify the resulting punctuation reads `... naming the model; the loom still loads, ...` with no doubled separators.
- Delete the closing sentence "Future widening (Gemini, OpenAI Responses, weak local models via JSON-mode fallback) is listed in [Future Considerations](../future-considerations.md).". The cross-link to Future Considerations is preserved by the *loom 1.0 seam — typed-query supported provider set* blockquote immediately below, which already anchors the deferred extension at `future-considerations/surface-extensions.md#mid-loom-user-session-replacement`'s sibling *Typed-query support for providers without named-tool forcing* bullet.

After both deletions the paragraph still pins (a) the supported-provider set, (b) the load-time warning code, (c) the runtime `Err(QueryError { kind: "transport", ... })` shape, and (d) the `<provider>` field's `Model<Api>.api` provenance. The deferred-extension cross-link is preserved exactly once, in the seam blockquote that owns it.

Implementer-relevant edge case: do not also delete the seam blockquote in the same edit — that blockquote's *single-named-constant* implementation note overlaps with the sibling "inline seam blockquotes restate normative MUSTs / future-work" finding and must be resolved under that finding's own decision, not as a side effect of this cleanup.
