## Finding

`coverage-matrix.md` maps every executable spec REQ-ID to its closing leaf across two tables (§Numbered REQ-IDs, §Code-keyed obligation areas). The `GOV-*` prefix family is deliberately excluded from both tables — `GOV-1, GOV-3, GOV-5 … GOV-31` govern the spec corpus itself (anchoring, retirement, cross-link form) and map to no runtime leaf. The note recording that exclusion is the final `## Governance REQ-IDs (GOV-*) — corpus governance, not runtime obligations` section, positioned after both data tables.

A scoping exclusion is only useful as a forward signal: it tells a reader, before they scan the data, which inputs are intentionally absent. Here the order is inverted. A contributor enumerating REQ-IDs from `governance.md`'s prefix table (as the matrix preamble directs) sees `GOV-*` prefixes missing from both tables with no early indication the omission is by design, and only learns it is intentional after reaching the tail section. The note is correct and complete; only its placement defeats its purpose.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** `docs/plan_topics/coverage-matrix.md` was created at fecb504 ("Split spec.md and plan.md into per-topic / per-phase files") with no `GOV-*` section. The `GOV-*` scoping-exclusion section was first authored in c6a664e, and in that same commit it was placed at the tail of the file — after both the §Numbered REQ-IDs and §Code-keyed obligation areas tables. The note has never existed in a non-tail position; the placement defect is coincident with the section's introduction.

## Solution Space

**Shape:** single

### Recommendation

Relocate the `GOV-*` scoping-exclusion content into the preamble so it precedes the §Numbered REQ-IDs table. Remove the trailing `## Governance REQ-IDs (GOV-*) — corpus governance, not runtime obligations` section and fold its body into the preamble immediately after the existing sentence "Use the REQ-ID prefix table in [`../spec_topics/governance.md`](../spec_topics/governance.md) to enumerate the REQ-IDs to cover." — the natural anchor, since that sentence is what sends the reader to the prefix table where the `GOV-*` prefixes appear.

Preserve the body text verbatim (the `GOV-1, GOV-3, GOV-5 … GOV-31` enumeration, the "govern the spec corpus itself" rationale, the arm-(b) corpus-invariant framing, and the pointer to `H5a` for the mechanizable slice). The edit is purely positional within `coverage-matrix.md`; no wording change to the data tables and no spec edit is required.

## Consequence

**Severity:** advisory

A contributor scanning the matrix top-down encounters the `GOV-*` absence before any explanation, risking either a spurious `GOV` row addition or wasted investigation of an intentional omission. No closing gate misfires and no implementation leaf is affected.
