## Finding

The `loom/runtime/internal-error` registry row in `code-registry-runtime.md` describes the runtime-defect surface twice over. Its leading clause is open-ended and engine-neutral — "The interpreter or an adapter it called threw an exception outside the closed loom 1.0.0 panic-source list" — and that clause alone already routes *any* JavaScript-exception-shaped throw to this code. The row then narrows to a closed, engine-specific enumeration: "also catchable host allocation failures — `RangeError: Invalid string length`, `RangeError: Invalid array length`, `RangeError: Maximum call stack size exceeded`." The same exhaustive three-string form appears in the canonical runtime-defect-surface prose at `errors-and-results/error-model.md` ("The catchable allocation-failure family (`RangeError: Invalid string length`, …) does reach the runtime-defect surface and routes here normally").

Naming those three failures as *the* catchable family bakes in two engine assumptions stated as fact: (1) that these specific V8 conditions remain catchable JavaScript exceptions — rather than crossing into the uncatchable `OOMErrorCallback` / `abort()` fatal path the row itself carves out — across the open-ended pinned Node range (`>=22.19.0`, no upper bound); and (2) that their V8-internal message strings are stable. The row matches on neither the strings nor the catchability flag — the open-ended trigger catches whatever throws and carries `error.message` through verbatim — so the enumeration adds an unverified engine-pinned classification without adding coverage.

The spec already demonstrates the engine-neutral framing in its own NOCEIL-3 carve-out (`hard-ceilings/ceiling-invariants-and-audit.md`): there the same three strings appear behind a generalising clause — "and any other JavaScript-exception-shaped allocation failure the host throws." That generalisation was dropped from the registry row and from the `error-model.md` prose, leaving the three strings reading as the exhaustive catchable set on the two surfaces an implementer is most likely to consult.

## Issue introduction
**Verdict:** single-commit introduction
**Introducing commit:** `cc240be` (2026-05-07) — "pi-loom spec: resolve \"No additional V1 runtime ceiling closing paragraph\""
**History:** The corpus is git-tracked. `git log -S "RangeError: Invalid array length"` and `git log -G "catchable host allocation failures"` over the spec content (excluding the review/store bookkeeping commits) trace the enumeration's first appearance to `cc240be`. Later commits `f5e89f4` (2026-06-04, "enforce 100KB hard / 30KB recommended size cap") and `81bf52a` ("collapse spec/ section pages into spec_topics/") relocated and condensed the text into its current `code-registry-runtime.md` home without restoring the generalisation. The defect is therefore present since `cc240be` on the two surfaces it concerns.

## Solution Space

**Shape:** single

### Recommendation

Reframe the three named `RangeError` strings as non-exhaustive illustration rather than the exhaustive catchable family, on both surfaces that currently present them as closed:

- In the `code-registry-runtime.md` `loom/runtime/internal-error` row, change "also catchable host allocation failures — `RangeError: Invalid string length`, `RangeError: Invalid array length`, `RangeError: Maximum call stack size exceeded`" to lead with the general clause and mark the strings illustrative, e.g. "also any catchable host allocation failure the JavaScript exception machinery delivers (illustratively `RangeError: Invalid string length`, `RangeError: Invalid array length`, `RangeError: Maximum call stack size exceeded`)."
- In `errors-and-results/error-model.md`, restore the same "and any other JavaScript-exception-shaped allocation failure the host throws" generalisation that NOCEIL-3 already carries, so the three strings read as examples behind the open-ended clause.

This rests the closure on the engine-neutral runtime-defect trigger ("a throw outside the closed panic-source list") that already does the routing, and aligns the two surfaces with the spec's own canonical NOCEIL-3 wording.

Preserve intact the catchable-vs-uncatchable partition itself — the `OOMErrorCallback` / `abort()` fatal-path carve-out that emits no diagnostic. That distinction is load-bearing and separately verified; only the *exhaustiveness* of the catchable enumeration is at issue.

## Consequence

**Severity:** advisory

No observable behaviour diverges: the open-ended trigger already routes any JavaScript-exception-shaped throw to `loom/runtime/internal-error` regardless of its message, so two implementers produce the same routing. The cost is a durable accuracy/robustness gap — the spec asserts an exhaustive, engine-pinned catchable set as fact, which can silently become wrong if a future Node/V8 in the open-ended pinned range alters the catchability or message contract of one of the three named conditions.
