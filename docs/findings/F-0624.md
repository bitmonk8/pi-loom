## Finding

Under `bind_context: session`, the runtime sends a bounded, most-recent slice of the caller's session transcript — rendered per [Compact-transcript format](../../docs/spec_topics/binder/binder-model-and-context.md#compact-transcript-format-normative) — to whatever provider backs the resolved binder model. That slice is not a sanitised projection: BNDR-7c pins that `loom-system-note` custom messages with `display: false` are still rendered into the transcript ("Custom messages with `display: false` are still surfaced in the transcript"), and BNDR-7e pins that assistant `ToolCall.arguments` are rendered verbatim under `[tool-call <name>(<args-json>)]`. The caller's session may therefore contain operator-only diagnostics and full tool-call argument JSON, both of which leave the caller's data path the moment the binder runs.

Independently of that, the non-normative cost guidance under [Binder model](../../docs/spec_topics/binder/binder-model-and-context.md#binder-model) actively steers authors toward "cheaper structured-output-capable models (e.g. Claude Haiku, GPT-4o-mini, Gemini Flash)," and the resolution chain (`bind_model:` → `looms.binderModel`) coins no relationship between the binder model's resolved provider and the caller's model's provider. The two can — and the guidance encourages them to — be different providers. Yet no rule on this page constrains the binder model's provider relative to the caller, and the [Trust boundary](../../docs/spec_topics/overview-and-orientation.md#non-functional-requirements) bullet (the only non-functional disposition that could plausibly own this) speaks only to host-process privilege — what code can do — not to where data goes through declared model calls. The result is a real cross-provider data flow with no acknowledgement in either the binder-context section that produces it or the trust-posture section that ought to frame it; authors and operators are left to discover the property by reading the transcript-rendering rules and the cost note together.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `83802db` (2026-05-04, "spec: collapse to single typed argument style; add LLM-driven slash-command binder")
**History:** The binder feature itself, the `bind_context: session` axis, and the cost guidance steering authors toward a cheaper tier-2 model all landed in the same commit (`git log --reverse -S "bind_context" -- '*.md'` and `git log --reverse -S "Haiku" -- '*.md'` both bottom out at `83802db`). That commit message lists "`bind_context` (none | session, default `none`)" and "cost note" as siblings under a single bullet, with no data-boundary statement accompanying either. Subsequent edits (the `binder-model-and-context.md` split at `f5e89f4`, the BNDR-7 transcript reference renderings at `1a41db2`, the automatic-context-escalation seam at `0594eea`) further fleshed out the transcript surface — making the egress more concrete (custom-message `display: false` rendering, verbatim `[tool-call …]` lines) — without ever adding the missing disposition. The defect is not the result of a later interaction; the omission has been continuously present since the binder section was first authored.

## Solution Space

**Shape:** single

### Recommendation

Add a normative author/operator-responsibility disposition to `binder-model-and-context.md`. Two edits, in this order:

1. **Under `## Binder context`, immediately after the `none` / `session` enumeration**, insert one paragraph stating that when `bind_context: session` is set, the bounded transcript slice the runtime sends to the binder model includes any custom-message and tool-call content rendered per [Compact-transcript format](../../docs/spec_topics/binder/binder-model-and-context.md#compact-transcript-format-normative) — including `display: false` system notes (per BNDR-7c) and verbatim `ToolCall.arguments` JSON (per BNDR-7e) — and that the binder model's provider may differ from the caller's model's provider, so authors and operators are responsible for ensuring the resolved binder-model provider is an acceptable destination for that content. State the disposition as author/operator responsibility, not as a runtime check; loom 1.0 imposes no provider-identity constraint between `bind_model` and the caller's `model`.

2. **In the non-normative cost note under `## Binder model`**, add a clause acknowledging that "cheaper structured-output-capable models" are typically reached through a different provider than the caller's model, with a forward-link to the disposition added in edit 1. The cost note remains non-normative; the disposition is normative.

Do **not** add a load-time diagnostic or a provider-equality gate. A cross-provider binder model is a legitimate configuration (it is exactly what the cost guidance steers authors toward); the gap is disclosure, not enforcement. Do **not** broaden `## Trust boundary` to cover this — that bullet pins host-process privilege and is owned by `overview-and-orientation.md`; cross-linking from the new binder-context paragraph back to `## Trust boundary` is sufficient if a link is desired at all.

Edge cases the implementer must cover when writing the paragraph: (a) the bypass-eligible cases (no-params bypass, single-string bypass per [Binder bypass](../../docs/spec_topics/binder/binder-bypass-and-envelope.md#binder-bypass)) do not call the binder and therefore route no caller content to the binder-model provider — the disposition applies only to the non-bypass path; (b) `bind_context: none` still routes the slash text and the loom's frontmatter to the binder-model provider, so the disposition is not zero-content under `none`, only narrower; (c) the disposition is per-invocation (re-entrancy per the [loom 1.0 seam — automatic context escalation](../../docs/spec_topics/binder/binder-model-and-context.md#loom-1-0-seam-automatic-context-escalation)) — a hot-reload changing `looms.binderModel` mid-session changes the destination provider on the next invocation, with no retroactive effect.

## Consequence

**Severity:** advisory

Two reasonable implementers will produce the same runtime; what differs is what authors and operators understand about it. An author wiring `bind_context: session` against a cross-provider binder model has no spec text telling them they are routing operator-only system notes and verbatim tool-call arguments through a second vendor's API, and an operator reading the spec to assess the loom-extension's data-egress surface will find no inventory of it. The exposure is real and shipping today; the gap is in disclosure rather than in behaviour.
