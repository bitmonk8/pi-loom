## Problem

`PIC-14` (`filewatcher-interface` in `host-interfaces-services.md`) routes the chokidar adapter's own `error` events onto the transient `ctx.ui.notify` toast surface and says nothing about the watcher's disposition after such an error — whether the adapter re-attaches, degrades to a terminal state, or is left undefined. The toast is non-persistent by construction (excluded from the transcript), and no `loom/runtime/*` or `loom/host/*` code in the registry covers a live watcher-backend death: `ERR-7` covers debounced rebuild failures of a still-live watcher and `loom/host/session-shutdown-runtime-degraded` covers the post-`session_shutdown` closed-watcher case, leaving the live-death case in the gap. When the host kernel pushes an error to chokidar (ENOSPC, EMFILE, root permission flip, FSEvents stream death), hot-reload silently stops: the operator sees one toast that scrolls out of the TUI, and every subsequent `.loom`/`.warp`/settings edit is silently ignored until they independently run `/reload`. Two conforming implementers diverge — one leaves the dead watcher in place, the other adds unbounded reattach attempts the seam does not forbid.

## Solution approach

Add a clause to `PIC-14`'s `FileWatcher` bullet list (`filewatcher-interface`) defining the watcher's disposition after a chokidar adapter `error`: a terminal degraded state for the remainder of the runtime instance's lifetime, with `/reload` the only recovery, modelled on the MUST-NOT-re-subscribe / MUST-NOT-poll precedent in `session-only-degraded-state.md`. Add a persistent diagnostic code to `code-registry-runtime.md` routed through the `loom-system-note` channel, co-emitted with the retained transient toast, that names the failing root and directs the operator to `/reload`; register it in the location-less-diagnostics set in `diagnostic-shape.md`, and have both the discovery-root and settings-file watcher paths emit it. Add a forward-cross-reference from registration-steps §5 (`watcher-hot-reload-registration`) to the new disposition clause.

## Solution constraints

- The watcher-backend-death surface MUST be disjoint from `ERR-7` — which covers debounced rebuild failures of a still-live watcher, not backend death — and not folded into it.
- An adapter `error` arriving during the `session_shutdown` teardown window (after sub-step 4's watcher close) MUST be swallowed without emission, since the watcher is already being torn down.
