## Finding

The preamble of `docs/spec_topics/glossary.md` declares the page descriptive: "Each entry is descriptive — a one-paragraph reminder of what a term means and where it is used. The canonical, normative definition lives on the feature page named in the entry's `See:` reference; if a glossary entry and its canonical page disagree, the canonical page wins." Three entries on the same page nonetheless carry full normative obligations addressed to spec authors and implementers:

- `callable set` — "Authors and implementers MUST use the term `callable set` (or, with an explicit scoping modifier, `frontmatter callable set` / `loom's declared callable set`) … the bare phrases `tools: set`, `tool set`, `loom's tools`, and `available tools` are spec-prose synonyms to be avoided".
- `schema slug` — "Authors and implementers MUST use `schema slug` (or the bare form `slug` when context is unambiguous). The synonyms `schema hash`, `schema-hash`, `sha12`, `lowered-schema hash`, and `lowered-schema content hash` are spec-prose drift to be avoided; a future grep gate uses this entry as its source of truth."
- `Pi tool` vs. `.loom callable` — "Authors and implementers MUST use the canonical term `.loom callable` … the synonyms `registered loom`, `registered subagent loom`, `registered-loom call`, and `loom callee` are spec-prose drift to be avoided."

Three failures follow. (1) Each `See:` target (`frontmatter.md`, `tool-calls.md`, `schema-subset.md`) is named as the canonical page yet does not carry the MUST, so under the preamble's "canonical page wins" tie-break a reader who consults the feature page sees no terminology obligation and a glossary-only MUST is, by the page's own rule, ignorable. (2) The audience for spec-authoring conventions is misdirected — glossary readers are looking up what a term means, while authors deciding which token to use would not normally consult the glossary. (3) `schema slug` further declares itself "the source of truth" for a future grep gate; pinning a build-time gate's source of truth to a page the preamble disclaims as carrying no binding obligations is internally incoherent.

## Solution Space

**Shape:** multiple

### Option A — Push each terminology MUST onto its canonical feature page

Approach: For each of the three entries, move the "Authors and implementers MUST use …" sentence and its accompanying synonyms-to-avoid list onto the page named in `See:` (callable-set → `frontmatter.md`'s `tools` paragraph or `frontmatter/frontmatter-fields-a.md` next to the existing *Naming convention* / *Binder-model root-word convention* paragraphs; `.loom callable` → `tool-calls.md`; `schema slug` → `schema-subset.md`, either at the *Canonical schema hash* / *Schema slug* anchor or in a sibling *Terminology* paragraph). Reduce each glossary entry to a one-paragraph descriptive reminder that ends with `See:` pointing at the new home.

Spec edits:
- `glossary.md`: delete the three MUST sentences and their synonyms-to-avoid lists; rewrite each entry as a descriptive paragraph.
- `frontmatter/frontmatter-fields-a.md` (or `frontmatter.md`): add a *Callable-set terminology* paragraph carrying the `callable set` MUST and its avoid-list (mirroring the existing *Naming convention* and *Binder-model root-word convention* paragraphs).
- `tool-calls.md`: add a parallel *`.loom` callable terminology* paragraph carrying the `.loom callable` MUST and its avoid-list.
- `schema-subset.md`: at or just below the *Schema slug* anchor add a *Schema-slug terminology* paragraph carrying the `schema slug` MUST, its avoid-list, and the future grep-gate source-of-truth claim.

Pros: aligns with the preamble's existing "canonical page wins" rule; readers consulting the feature page see the obligation; no new page introduced; the grep gate's source-of-truth claim lands on a page whose normative status is not self-disclaimed.

Cons: three small parallel paragraphs instead of one consolidated rule; future terminology MUSTs would each need a similar home on their feature page; the audit/grep tooling has three sites to grep rather than one.

Risks: forgetting to update one of the three feature-page paragraphs when a future split/merge changes ownership — equivalent in scale to other existing per-page conventions (`Naming convention`, `Binder-model root-word convention`).

### Option B — Create one corpus-wide terminology-conventions home

Approach: Author a single corpus-level conventions page (either a new `docs/spec_topics/governance/terminology-conventions.md` under the existing governance owner, or a new top-level `docs/spec_topics/conventions.md`) that lists each governed term with its canonical spelling, scoping modifiers, and synonyms-to-avoid, and that carries the future grep-gate source-of-truth claim. Reduce each glossary entry to a descriptive paragraph that `See:`-points at the conventions page row.

Spec edits:
- New page (governance/terminology-conventions.md preferred — `governance.md` already owns corpus-wide author/tooling conventions): one row per governed term (`callable set`, `.loom callable`, `schema slug`, and any future additions) with canonical form / allowed scoping modifiers / synonyms-to-avoid / `See:` back to the feature page.
- `governance.md` Contents list: add the new page.
- `glossary.md`: delete the three MUST sentences and synonyms-to-avoid lists; each entry retains its descriptive paragraph and gains a `See:` line pointing at the new conventions page row.
- Note `governance.md`'s declared audience ("spec maintainers and … automated tooling, not … implementers of the loom runtime"); the terminology-conventions page would either widen that audience locally or be hosted as a peer outside `governance/`.

Pros: single home for all terminology obligations, easy for a grep gate to point at; future additions need only a new row; the page can be cited from the feature pages too.

Cons: introduces a new file and a new governance sub-area; the conventions live one hop away from the feature page that defines the term; conflicts mildly with `governance.md`'s audience note if hosted under `governance/`.

Risks: drift between the conventions-page row and the feature page's actual usage of the term; requires a one-time cross-link audit so the feature pages forward-link.

### Recommendation

Take **Option A**. The spec corpus already uses per-feature-page terminology paragraphs (`Naming convention`, `Binder-model root-word convention`) and the glossary preamble already names the feature page as the normative owner, so this option closes the placement gap with the smallest disturbance and without introducing a new governance surface. Edge cases for the implementer: (i) the `schema slug` entry's "future grep gate uses this entry as its source of truth" clause must move with the MUST so the grep gate points at `schema-subset.md`; (ii) the glossary entries should retain the `(or, with an explicit scoping modifier, …)` allow-list locally too, since a glossary reader still needs to recognise the licit variants — only the MUST verb belongs on the feature page; (iii) update any cross-references that currently target the glossary entry for the obligation (none found in `plan_topics/`, but re-grep before finalising).

## Consequence

**Severity:** advisory

A spec author who reads only `frontmatter.md`, `tool-calls.md`, or `schema-subset.md` sees no terminology obligation, and the glossary preamble explicitly authorises ignoring an obligation that the feature page does not corroborate; new prose written today can therefore drift back into `tool set`, `schema hash`, `loom callee` etc. without violating any rule the author actually has to read. The planned `schema slug` grep gate inherits the same weakness — its source-of-truth page is, by its own preamble, non-binding.
