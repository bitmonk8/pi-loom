## Finding

`V3d` Adds declares that top-level `fn` declarations carry the restriction "no nested functions, closures, or functions-as-values", but neither `V3d` nor its paired `V3d-T` carries a test bullet that asserts the restriction. The four `V3d`/`V3d-T` Tests bullets cover return-type inference, empty-tail/`void`, the `loom/parse/unreachable-code` warning, and the final-value contract — none exercises a nested `fn`, a closure, or a function-name-as-value.

The spec makes the restriction observable through two registered diagnostic codes, not by silent by-construction rejection: `functions.md` FN-1 states that nested function definitions "surface as `loom/parse/nested-fn`" and that a function name used outside call position (bound to `let`, passed as an argument) "surfaces as `loom/parse/function-as-value`". Both are severity-`E` `parse`-phase rows in `diagnostics/code-registry-parse.md`. `coverage-matrix.md`'s `cka-7` row maps the FN / RET code-keyed obligation area to `V3d` as its closing leaf, so `V3d` is the leaf responsible for closing these codes.

Because the two codes are registered but asserted by no test anywhere in the plan, the `V7b` registry↔test parity gate (DIAG-2: "a registry code with no asserting test fails the gate") and the `H5a` closing-gate automation will flag them at the live-corpus footing activated by `H6a`. Separately, an implementer reading `V3d` with no test pinning the codes may enforce the restriction in the `V2a` grammar with a generic parse error rather than emitting the two spec-named codes, diverging from FN-1.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** `V3d-functions-and-return.md` was created in `c6a664e` with the "no nested functions, closures, or functions-as-values" claim already in Adds and no nested-`fn`/function-as-value test bullet. `git log -S "nested-fn"` and `-S "function-as-value"` over the file return no commits — the codes have never appeared in the leaf. The only later commit touching the file (`7a8565a`, 2026-06-11) reworked the final-value assertion and did not add the missing restriction tests. The gap has existed since the leaf's inception.

## Solution Space

**Shape:** single

### Recommendation

Add the two missing restriction assertions to `V3d-T`'s Tests (the tests task lands first under the TDD ritual) and mirror them in `V3d`'s Tests:

- A nested `fn` declaration (a `fn` declared inside another `fn` body or a block) fires `loom/parse/nested-fn`. Cite `functions.md` FN-1 and the `diagnostics/code-registry-parse.md` row.
- A function name used outside call position — bound to a `let` or passed as an argument — fires `loom/parse/function-as-value`. Cite the same FN-1 / code-registry rows.

Extend `V3d`'s **Ships when** so it names the two codes alongside the existing return-inference / empty-tail / final-value criteria.

The two codes are spec-named in FN-1, so the assertion must witness those exact codes firing; enforcing the restriction by-construction in `V2a` with a generic parse error does not satisfy FN-1 and does not clear the DIAG-2 parity gate, which requires an asserting test for each registered code regardless of where the rejection is implemented. If the rejection is implemented in the `V2a` grammar, the asserting tests still belong on `V3d`/`V3d-T` because `cka-7` names `V3d` as the FN closing leaf.

## Consequence

**Severity:** correctness

Two registered, spec-mandated parse codes (`loom/parse/nested-fn`, `loom/parse/function-as-value`) have no asserting test, so the `V7b`/`H5a` registry↔test parity arm reddens the closing gate at the `H6a` live-corpus footing. With no test pinning the codes, an implementer may enforce the restriction by-construction in `V2a` and emit a generic parse error instead of the two codes FN-1 names, shipping a leaf that does not match the spec.
