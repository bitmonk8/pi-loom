## Problem

In the `cancelled-by-session-shutdown` row of the runtime code registry, the closing sentence "the exclusion mirrors `cancelled` (operator-initiated, self-explanatory in context)" names the `cancelled` `QueryError` kind and its always-log exclusion rationale with no cross-reference. A reader on the registry page cannot resolve the referent in place and must backtrack off-page to confirm what `cancelled`'s exclusion is being mirrored.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 7f37dd7 (2026-05-07, "resolve Session-swap behaviour for in-flight loom invocations is under-specified"); 52b9101 (2026-05-11, "resolve T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state"); 47fe3cd (2026-06-05, "resolve exactly six panic sources references a tag/column that does not exist")
**History:** The corpus is git-tracked. `git log -S`/`-G --follow` across renames (the file was later split out under `docs/spec_topics/diagnostics/` by f5e89f4) shows the three defect facets entered at distinct commits and compounded. 7f37dd7 added the `cancelled-by-session-shutdown` row, including the bare "exclusion mirrors `cancelled`" sentence (facet 2). 52b9101 added the residual-gap "MAY both contribute … AND later emit" EXCEPT clause and its `reload-teardown-timeout` mirror (facet 1). 47fe3cd recast the intro from the loose "(the rows below tagged as panics)" wording into an explicit per-code accounting ("the table below carries one row per code … One further code — …") but never enumerated the already-present `cancelled-by-session-shutdown` row (facet 3); every subsequent "One further code —" addition (validator-cache-collision, watcher-terminated, custom-type-unsafe) perpetuated the gap. The omission is thus an interaction: a row introduced on one date became silently unaccounted-for when a later commit converted the intro into an exhaustive enumeration without revisiting it.

## Solution approach

Add a forward cross-reference from the "exclusion mirrors `cancelled`" sentence in the `cancelled-by-session-shutdown` row to the always-log-set exclusion list in `pi-integration-contract/runtime-event-channel.md` (where `cancelled` is enumerated as deliberately excluded as "user/operator-initiated and self-explanatory in context") and/or to the `CancelledError` variant definition at `errors-and-results/queryerror-variants.md#queryerror-variants`, so the claim is resolvable in place.
