## Finding

The `SessionContext` and the `.messages` element shape paragraph in `docs/spec_topics/pi-integration-contract/host-interfaces-core.md` (`#sessioncontext-shape`) declares itself "the loom-load-bearing subset the renderer reads … MUST be re-validated against the cited declarations on each Pi minor bump". The four bullets that follow are meant to enumerate, per `AgentMessage` arm, exactly which fields the compact-transcript renderer reads — so that a Pi-minor diff against the declarations is mechanical at bump time and so that a fresh implementer of the renderer knows what to walk.

Three of the four arms are pinned at that grain. The `AssistantMessage` bullet names `content` as `(TextContent | ThinkingContent | ToolCall)[]`, names the per-block disposition (concat `TextContent.text`, omit `ThinkingContent`, emit one `[tool-call …]` per `ToolCall` reading `ToolCall.name` and `ToolCall.arguments`), and cites the declaration file for each block type. The `CustomMessage` bullet names `customType` (string) and `content` (`string | (TextContent | ImageContent)[]`) and adds the transcript-safe constraint. The `UserMessage` bullet, by contrast, says only "its text content is read for the `[user]` body" — with no mention that `UserMessage.content` at the loom 1.0 Pi-SDK pin is the discriminated union `string | (TextContent | ImageContent)[]` (declared at `@earendil-works/pi-ai`'s `dist/types.d.ts`), and with no statement of how the array arm is walked or how a non-`TextContent` block (e.g. an `ImageContent`) contributes to the `[user]` body. The `ToolResultMessage` bullet says only "its `content` is a typed-block array; the renderer reads the text-block content for the `[tool]` body" — the bullet does not state that `ToolResultMessage.content` is `(TextContent | ImageContent)[]`, does not name the discriminator (`TextContent.type === "text"`) by which "the text-block content" is selected, and does not state the non-text disposition.

The renderer rules in `docs/spec_topics/binder/binder-model-and-context.md` rule 4 plus the BNDR-7a/7g/7h reference renderings constrain the byte output and so a conforming implementation is reachable; the gap is in the PIC pin itself, which is the file the Pi-version-bump procedure greps against (re-validation per the *Editorial-review checklist for unpinned host presuppositions*). An asymmetric pin defeats the version-bump audit: a future Pi minor that shifted `UserMessage.content` from `string | (TextContent | ImageContent)[]` to a richer union, or renamed `TextContent.type`, would slip past the bump checklist because the PIC paragraph does not name those fields for the `UserMessage` / `ToolResultMessage` arms in the first place.

## Issue introduction

**Verdict:** introduced

**Introducing commit:** `01c9db9` — "pi-loom spec: resolve "compact-transcript renderer references undeclared SessionContext/AgentMessage fields"" (2026-06-03).

**History:** The `SessionContext` and the `.messages` element shape paragraph did not exist in the corpus before `01c9db9`; that commit created it. The introducing diff (against `docs/spec_topics/pi-integration-contract.md` at the time, later split into the topic file by `f5e89f4`) added all four arm bullets in their current asymmetric form on the same line: the `AssistantMessage` bullet was authored at full grain (`content: (TextContent | ThinkingContent | ToolCall)[]`, per-block disposition, declaration cites), while the `UserMessage` bullet was authored as "its text content is read for the `[user]` body" and the `ToolResultMessage` bullet as "its `content` is a typed-block array; the renderer reads the text-block content for the `[tool]` body". Subsequent edits to the paragraph touched only `CustomMessage`: `9568906` ("custom:<type> role tag customType safe-character constraint") added the transcript-safe clause to the `CustomMessage` bullet, and `f5e89f4` re-homed the paragraph under the new topic directory without touching its prose. The under-pinned framing on the two arms has been carried through unchanged since the paragraph's introduction.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the two under-pinned bullets in `host-interfaces-core.md`'s `#sessioncontext-shape` paragraph so each names its `content`-field shape at the same grain as the `AssistantMessage` bullet. Concretely:

- **`UserMessage`** — replace the bullet body with: `UserMessage.content` is `string | (TextContent | ImageContent)[]` (declared at `@earendil-works/pi-ai`'s `dist/types.d.ts`). The renderer reads the string arm verbatim as the `[user]` body; for the array arm it concatenates each `TextContent.text` in array order (consistent with the `CustomMessage` array-arm rule below) and `ImageContent` blocks contribute no transcript bytes. `TextContent` and `ImageContent` are declared in the same `@earendil-works/pi-ai` `dist/types.d.ts`.
- **`ToolResultMessage`** — replace the bullet body with: `ToolResultMessage.content` is `(TextContent | ImageContent)[]` (declared at `@earendil-works/pi-ai`'s `dist/types.d.ts`). The renderer separates `TextContent` blocks (discriminator `type === "text"`) from non-text blocks per `binder-model-and-context.md` rule 4 `toolResult` arm: the `[tool]` body is the concatenation of each text block's `.text` in array order, with any non-text block `JSON.stringify`'d (no whitespace) and concatenated in the same array order.

Both edits land in the same paragraph and the same diff. The `binder-model-and-context.md` rules and the BNDR-7 reference renderings already encode the same dispositions; the PIC edit pulls them into the pin so the version-bump audit can grep all four arms uniformly. Edge cases the implementer must watch:

- The `UserMessage.content` `string` arm and the array arm yield identical `[user]: <text>` output when the array is exactly one `TextContent`; BNDR-7a (string arm) and the recommended array-arm fixture must agree byte-for-byte for that case.
- The `ImageContent`-contributes-no-bytes rule on the `UserMessage` array arm must match the existing `CustomMessage` array-arm rule and BNDR-7h, or the two custom/user paths drift.
- The `ToolResultMessage` discriminator wording must name `TextContent.type === "text"` explicitly so the non-text fallback is unambiguous when Pi adds a new block kind under the same union.

## Consequence

**Severity:** correctness

Two reasonable implementers can diverge on `UserMessage` and `ToolResultMessage` walks even while both pass the existing BNDR-7 fixtures. One reads `UserMessage.content` as always-string-text (the bullet's "text content" phrasing licenses it), drops the array arm on the floor, and silently emits an empty `[user]: ` body whenever Pi delivers the array form. Another walks the array but invents its own non-text disposition (`JSON.stringify`'d, dropped, or interleaved). The same divergence exists for `ToolResultMessage`: the PIC bullet does not name the discriminator the renderer uses to separate text blocks from non-text blocks, so two implementations can disagree on which blocks are "text" without contradicting the PIC paragraph. The fixtures detect the disagreement at conformance time, but the PIC pin is also the version-bump audit anchor and an asymmetric pin lets a Pi-minor shape change in either arm slip past the bump checklist.
