## Finding

Version-bump step 6 (`version-bump-triggers.md`) requires, when a new `@earendil-works/pi-ai` `Api` value appears, that the contributor "MUST consult pi-ai release notes **and** the new provider's request-adapter source to decide between a named field and `omitted`." The phrase *request-adapter source* names a concrete evidence artefact the contributor is obligated to inspect, but the spec never defines what that artefact is. `@earendil-works/pi-ai` is a published, closed npm package distributed as a compiled bundle; "source" has no single obvious referent in that context. At least three readings are live: (a) the compiled `dist/*.js` request-adapter modules materialised from the installed package, (b) the `dist/*.d.ts` type declarations, or (c) some pi-ai documentation artefact distinct from the release notes already named. Each can yield a different verdict for the same provider (a seed field may be visible in compiled adapter code but absent from declarations, or vice versa), so two contributors auditing the same candidate minor can reach different named-field/`omitted` conclusions.

The same sentence is additionally ambiguous on whether the two evidence sources are conjunctive or disjunctive: "consult pi-ai release notes **and** the new provider's request-adapter source" reads literally as *both required*, yet a contributor who finds the answer in the release notes alone has no instruction on whether the adapter-source check is still mandatory. The decision the clause governs is consequential — choosing `omitted` when a named seed field exists silently regresses a provider — and the build-time `Api`-coverage assertion does not adjudicate it (it only forces a row to exist), so the underspecified evidence target is the sole guard against a wrong default.

## Solution Space

**Shape:** single

### Recommendation

Define the evidence artefact and the conjunction explicitly in step 6's MUST sentence. State that the contributor consults two named sources to decide between a named seed field and `omitted`: (1) the candidate `@earendil-works/pi-ai` minor's release notes, and (2) the request-adapter artefacts in the installed package — naming the concrete files, i.e. the compiled `dist/*.js` request-adapter modules and the `dist/*.d.ts` type declarations of `@earendil-works/pi-ai` at the bumped pin. Make the relationship a disjunction-with-precedence: a seed field identified in **either** source justifies a named field; `omitted` is the conservative default only when **neither** source identifies one. This eliminates both the undefined "source" referent (it now names the package's distributed artefacts rather than implying unavailable upstream source) and the and/or ambiguity. Leave the existing "do not silently regress a provider for which pi-ai has added seed support" clause and the `Api`-coverage gate intact.

## Consequence

**Severity:** advisory

A contributor reconciling a newly-introduced provider against a candidate `@earendil-works/pi-ai` minor has no defined artefact to inspect and no rule on whether release notes alone suffice, so two reasonable contributors can pick different defaults for the same provider. A provider entered as `omitted` when adapter evidence would justify a named seed field is silently regressed; the divergence is bounded by the conservative-default posture and is recoverable on a later bump, so the working system can still be produced.
