## Finding

The `## Retired REQ-IDs` table on `docs/spec_topics/governance/anchor-scheme-and-retired.md` carries two rows whose `Retired in` cell holds the literal token `` `pending` ``: the GOV-12 row (retired in favour of GOV-30 / GOV-31) and the GOV-14 row (demoted to a non-normative editorial note). The trailing paragraph immediately under the same table states the column rule:

> The `Retired in` column carries either the 7-character abbreviated commit SHA or a release tag.

`pending` is neither. The earlier wording that explicitly admitted a placeholder ("the `<retirement commit>` placeholder is replaced with a concrete SHA at the moment of the retiring commit") was removed when the GOV-2 / GOV-10 / GOV-11 / GOV-4 / GOV-21 placeholders were backfilled, so the tightened rule now governs the whole sub-table and these two cells contradict it. Every other row in the same table carries a real 7-char SHA (`64cdc60`, `c253233`, `1d773d6`, `8e3ccb4`), so the breach is visible against its own siblings as well as against the column rule.

The retiring commits themselves are knowable: `fe8aa5c` retired GOV-12 (`pi-loom spec: resolve "GOV-12 bundles ≥3 independently-testable sub-obligations"`, 2026-06-05) and `5a772e3` retired GOV-14 (`pi-loom spec: resolve "GOV-14 two SHOULD-NOTs admit no mechanical witness"`, 2026-06-05). A `pending` cell at this point is therefore stale, not pending in any real sense — it is unfinished editing on commits whose SHAs are already in the history.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 3f00256 — pi-loom spec: resolve "Governance retirement tables ship unresolved commit-SHA placeholders" (2026-06-05, Thomas Andersen); fe8aa5c — pi-loom spec: resolve "GOV-12 bundles ≥3 independently-testable sub-obligations" (2026-06-05, Thomas Andersen); 5a772e3 — pi-loom spec: resolve "GOV-14 two SHOULD-NOTs admit no mechanical witness" (2026-06-05, Thomas Andersen)
**History:** Commit `3f00256` backfilled the earlier `<retirement commit>` placeholders for GOV-2 / GOV-10 / GOV-11 / GOV-4 / GOV-21 with concrete SHAs and, in the same edit, tightened the column-rule prose by deleting the placeholder-tolerance parenthetical, so from that point on the `Retired in` column normatively admits only a 7-char SHA or a release tag. Later the same day commit `fe8aa5c` added the GOV-12 retirement row with `` `pending` `` in the `Retired in` cell, and a few hours after that commit `5a772e3` added the GOV-14 retirement row with the same `` `pending` `` placeholder, neither of them updating the column-rule prose nor backfilling its own SHA in a follow-up commit. The defect is the interaction: the row-adding commits inserted a placeholder vocabulary the rule-tightening commit had just removed.

## Solution Space

**Shape:** single

### Recommendation

In a single follow-up edit to `docs/spec_topics/governance/anchor-scheme-and-retired.md`, replace the `` `pending` `` token in the `Retired in` cell of the GOV-12 row with `` `fe8aa5c` `` and the `` `pending` `` token in the `Retired in` cell of the GOV-14 row with `` `5a772e3` ``. Make no other change: leave the column-rule paragraph, the `Replaced by` cells, the `Reason` cells, the row order, and the surrounding sub-tables untouched. The two SHAs are the abbreviated hashes of the retirement commits that added their respective rows, so they satisfy the column rule directly without invoking the "release tag" arm or inventing a prose qualifier.

Edge cases the implementer must watch:

- The edit is a backfill commit; it does *not* itself retire any new ID, so it adds no new row and the `Retired in` value it writes for each row is the SHA of an *earlier* commit (the same backfill pattern commit `3f00256` used for GOV-2 / GOV-10 / GOV-11 / GOV-4 / GOV-21).
- Do not re-introduce the deleted "the `<retirement commit>` placeholder is replaced with a concrete SHA at the moment of the retiring commit" parenthetical. The tightened rule is intentional; any tolerance for placeholders should be re-introduced (if at all) as a separate finding, not folded into this fix.
- Re-confirm `fe8aa5c` and `5a772e3` against `git log --date=short --format='%h %ad %an %s' -- docs/spec_topics/governance/anchor-scheme-and-retired.md` before committing, in case rebases have rewritten the local hashes.

## Consequence

**Severity:** advisory

The two `pending` cells leave the sub-table self-inconsistent with the column rule it publishes one paragraph below. No runtime behaviour or implementer divergence is at stake — `GOV-*` REQ-IDs govern the spec corpus itself, not the loom — but a future GOV-15 source-language-stability review (the only consumer of this sub-table) sees two rows that the table's own value-space MUST reject and cannot tell whether the retirement is finalised. The defect is also a self-replicating bad example: the next retirement commit is invited to write `pending` too.
