## Finding

The inventory-closure audit is specified across the six-file audit cluster (`inventory-audit-intro.md`, `audit-target-categories.md`, `audit-recognised-shapes.md`, `audit-resolution.md`, `audit-failures.md`, `audit-wire-and-canary.md`). Two of those files pin obligations by reference to enumerations that live in *sibling* PIC topic files, not in the audit cluster:

- `inventory-audit-intro.md` (`#sdk-capability-inventory`) opens with "The runtime depends on the seven named SDK capabilities **enumerated below**." There is no enumeration in that file. The actual items 1–7 live in `capability-inventory-items.md`, which the PIC page (`pi-integration-contract.md` Contents) places *after* the entire audit cluster.
- `audit-target-categories.md` pins the `namespace-function` entry kind as "exclusively the nine function members of the factory-probable subset … (capabilities 1, 2, 3, 4, and 6)" and elsewhere refers to "the seven capability obligations enumerated below." The seven-item list is in `capability-inventory-items.md`; the nine-function-member enumeration is in `capability-probe.md` Step 0 (c).

The references to the **nine function members** are adequately resolvable: `audit-target-categories.md` links the dependency to `[Step 0 (c)](./capability-probe.md#entry-capability-probe)`, whose body enumerates the nine members and is self-declared as their canonical source of truth. The defect is concentrated on the **seven-capability enumeration**: `inventory-audit-intro.md`'s "enumerated below" carries no link at all, and the cross-references that are meant to reach the items (`[SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory)`, including the link in `audit-target-categories.md`) resolve to the `## SDK capability inventory` section *heading* in `inventory-audit-intro.md` — a paragraph that itself only says "enumerated below" — rather than to the actual items 1–7 in `capability-inventory-items.md`. `capability-inventory-items.md` exposes only per-item anchors (`#sdk-cap-slash-command-registration` … `#sdk-cap-binder-llm-model`); there is no anchor the "seven capabilities" references can target as a set, and no link from the intro to that file.

The enumerations do exist in the corpus and are reachable by reading further down the PIC page, so this is a wayfinding gap rather than a missing-content gap. But the relative "below" phrasing and the heading-targeted anchors mean an implementer chasing "which seven capabilities does `CAPABILITY_OBLIGATIONS` enumerate?" lands on a paragraph that redirects them onward with no link, six topic files before the answer.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** e5cd514 (`pi-loom spec: resolve "T35 SDK capability inventory closed-set negative-direction audit"`, 2026-05-09), f5e89f4 (`docs: enforce 100KB hard / 30KB recommended size cap on spec set`, 2026-06-04)
**History:** e5cd514 authored the SDK-capability-inventory + closure-audit material as a single in-page PIC section, where "the seven named SDK capabilities enumerated below" and the `#sdk-capability-inventory` anchor resolved to items physically present in the same page. f5e89f4 split the PIC monolith into per-topic files — creating `inventory-audit-intro.md`, `audit-target-categories.md`, and `capability-inventory-items.md` as separate files — and placed `capability-inventory-items.md` after the audit cluster in the Contents while leaving the `#sdk-capability-inventory` anchor and the "enumerated below" wording on `inventory-audit-intro.md`, without converting the now-cross-file relative references into resolving links. That split is what materialised the wayfinding gap. 23d28a1 (2026-06-22) later realigned audit-cluster filenames and swept cross-refs but did not introduce or close this gap.

## Solution Space

**Shape:** single

### Recommendation

Make the seven-capability enumeration directly resolvable from the audit cluster:

1. Add a set-level anchor to `capability-inventory-items.md` (e.g. `<a id="sdk-capability-inventory-items"></a>` immediately above item 1) so the list of items 1–7 has a single citable target.
2. In `inventory-audit-intro.md`, replace the bare phrase "the seven named SDK capabilities enumerated below" with a linked form pointing at that new anchor (e.g. "the seven named SDK capabilities enumerated in [Capability inventory items](./capability-inventory-items.md#sdk-capability-inventory-items)"). Do the same for the other "enumerated below" / "items 1–7 below" references in this file that are meant to reach the list rather than the section header.
3. In `audit-target-categories.md`, repoint the "seven capability obligations enumerated below" reference (and the `[SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory)` link where it is meant to reach the items, not the section heading) to the same `capability-inventory-items.md#sdk-capability-inventory-items` anchor.

The nine-function-member references already resolve via `[Step 0 (c)](./capability-probe.md#entry-capability-probe)`; leave them as-is, or make the link explicit at the `namespace-function` definition site in `audit-target-categories.md` for symmetry. Do not relocate `capability-inventory-items.md` earlier in the Contents — anchored links resolve regardless of page order and avoid disturbing the other "above"/"below" relative references that the split established.

Edge case for the fixer: `#sdk-capability-inventory` is the load-bearing anchor for `CAPABILITY_OBLIGATIONS` cardinality wording and is targeted by `spec.md` Orientation and the version-bump procedure; keep that anchor in place on `inventory-audit-intro.md` and add the new item-level anchor alongside rather than moving the existing one.

## Consequence

**Severity:** advisory

An implementer populating `CAPABILITY_OBLIGATIONS` (the seven capabilities) or the `namespace-function` entries from the audit cluster must navigate from a "enumerated below" pointer that carries no link, then locate `capability-inventory-items.md` further down the PIC page unaided. The data is unambiguous once found, so two implementers would not diverge on content; the cost is navigation friction and the risk of stopping at the redirecting intro paragraph.
