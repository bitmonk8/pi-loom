## Problem

`V15e`'s Adds pins all three path identities its cache-eviction drop matches on ‚Äî the watcher-delivered changed-file path (`V9b`), the `V15a` parse-cache key, and the `.warp` import-edge-graph node identity (`V15c`) ‚Äî to the single canonical `realpath`-then-forward-slash form defined under invocation.md ‚Äî Resolution. But `V15c`'s `.warp` resolution per imports.md IMP-1 is a relative `path.join` plus a byte-for-byte final-segment match against `readdir` output, with no `realpath` step and no Unicode normalisation, so `V15c`'s node identity is never in that canonical form. On a symlinked or case-variant host the two forms diverge, the transitive-importer walk fails to match importers of the changed `.warp` file, and their stale parse-cache entries survive the `LoomRegistry` swap ‚Äî the exact stale-cache serve `V15e` exists to prevent. No leaf currently owns reconciling the two path-identity forms.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `b63e915`
**History:** The defect entered with commit `b63e915` (2026-06-17, "pi-loom plan: resolve \"hot-reload eviction pins one canonical path identity\""), which rewrote `docs/plan_topics/V15e-hot-reload-cache-eviction.md`'s Adds to pin all three path identities ‚Äî including "the `.warp` import-edge-graph node identity (`V15c`)" ‚Äî to "the single canonical `realpath`-then-forward-slash path form defined under invocation.md ‚Äî Resolution" (`git log -S 'import-edge-graph node identity'` reports `b63e915` as the sole commit introducing that text). The immediately prior revision (`a3dd2f0`, 2026-06-16) referenced "the `.warp` import-edge graph (`V15c`)" with no canonical-form claim, so the form mismatch did not exist before `b63e915`. `V15c`'s own history (`git log --follow docs/plan_topics/V15c-imports.md`: `c6a664e` ‚Üí `2316cac` ‚Üí `f930314` ‚Üí `6b7c7e8`) never introduced a `realpath` step; its `.warp` resolution has always tracked imports.md IMP-1's byte-for-byte-against-`readdir` rule. The introducing commit was a single-identity-canonicalisation fix that over-reached by binding `V15c`'s node identity to a form `V15c` does not (and per IMP-1 should not) produce.

## Solution approach

Rewrite the `.warp` node-identity clause in `V15e`'s Adds: instead of asserting `V15c`'s node identity is in the canonical `realpath`-then-forward-slash form, state that it is in imports.md's relative-resolution / byte-for-byte form (IMP-1) and that `V15e` re-canonicalises it to the eviction-comparison form before the transitive-importer match runs, naming `V15e` as owner of that mapping. State in the same Adds what form `V15a`'s parse-cache key for reachable `.warp` files is in, so all three identities in the three-way match are well-defined. For the `unlink` arm ‚Äî a deleted or renamed `.warp` file whose path can no longer be resolved ‚Äî direct the mapping to match against the stored node identity rather than re-resolving the vanished path. Extend the non-canonical-form test vector in `V15e` (Tests / Ships when) and `V15e-T` (Tests) to exercise a changed `.warp` file whose importer's native-form node identity differs from the changed-file `realpath` form, asserting the importer is still found and its stale parse-cache entry dropped.

## Solution constraints

- Do not add a `realpath` step to `.warp` resolution in `V15c` or `V15a`: that contradicts imports.md IMP-1's byte-for-byte-against-`readdir` rule and the deliberate cross-host case-sensitivity guarantee.
- The spec is read-only for this fix (`fix_surface: plan`); no invocation.md / imports.md edit is required.
