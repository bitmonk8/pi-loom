## Finding

`V15e` (hot-reload static-resolution cache eviction) requires the three path identities a cache drop matches on — the watcher-delivered changed-file path (`V9b`), the static-resolution per-pass parse-cache key (`V15a`), and the **`.warp` import-edge-graph node identity** (`V15c`) — to all be compared in "the single canonical `realpath`-then-forward-slash path form defined under invocation.md — Resolution". That `realpath`-then-forward-slash form is, per invocation.md, scoped specifically to `invoke(...)` `.loom` paths and `.loom` `tools:` entries: the resolved callee path is run through `FileSystem.realpath` (full transitive symlink resolution, byte-stable canonical output) before discovery-root containment.

`V15c`'s `.warp` import resolution does not produce that form. Per imports.md IMP-1 and the `Resolver` interface, `.warp` resolution is a relative `path.join`-style computation (`resolve(spec, fromFile)` joins `spec` against the directory of `fromFile`) followed by a **byte-for-byte** final-segment match taken against the bytes `FileSystem.readdir` returns — explicitly with no `realpath` / symlink-normalisation step, and explicitly not folded against any Unicode-normalised form. The spec deliberately keeps `.warp` resolution byte-exact-without-realpath ("The byte-for-byte rule holds on every host regardless of the host filesystem's case-equivalence model"). So the `.warp` import-edge-graph node identity `V15c` constructs is in the imports.md relative-join form, not the invoke-side `realpath`-then-forward-slash form `V15e` assumes.

On a host where the two forms diverge — a symlinked path, or a case-variant entry on a case-insensitive host — the watcher-delivered changed-file path (canonicalised by `V15e` to the `realpath` form) will not match the import-edge-graph node identity (`V15c`'s imports.md form), so the transitive-importer walk fails to identify importers of the changed `.warp` file and their stale parse-cache entries survive the `LoomRegistry` swap. That silent eviction miss is the exact stale-cache-serve failure `V15e` claims to prevent. No leaf currently owns reconciling the two path-identity forms, so an implementer following `V15e` as written would build an eviction that under-evicts on precisely the non-canonical hosts the leaf's own test vector targets.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `b63e915`
**History:** The defect entered with commit `b63e915` (2026-06-17, "pi-loom plan: resolve \"hot-reload eviction pins one canonical path identity\""), which rewrote `docs/plan_topics/V15e-hot-reload-cache-eviction.md`'s Adds to pin all three path identities — including "the `.warp` import-edge-graph node identity (`V15c`)" — to "the single canonical `realpath`-then-forward-slash path form defined under invocation.md — Resolution" (`git log -S 'import-edge-graph node identity'` reports `b63e915` as the sole commit introducing that text). The immediately prior revision (`a3dd2f0`, 2026-06-16) referenced "the `.warp` import-edge graph (`V15c`)" with no canonical-form claim, so the form mismatch did not exist before `b63e915`. `V15c`'s own history (`git log --follow docs/plan_topics/V15c-imports.md`: `c6a664e` → `2316cac` → `f930314` → `6b7c7e8`) never introduced a `realpath` step; its `.warp` resolution has always tracked imports.md IMP-1's byte-for-byte-against-`readdir` rule. The introducing commit was a single-identity-canonicalisation fix that over-reached by binding `V15c`'s node identity to a form `V15c` does not (and per IMP-1 should not) produce.

## Solution Space

**Shape:** multiple

### Option A — Map `V15c`'s import-edge node identity onto the eviction-comparison form inside `V15e`

**Approach.** Keep `V15c`'s `.warp` resolution exactly as imports.md IMP-1 specifies (relative join + byte-for-byte final-segment match, no `realpath`). In `V15e`, stop asserting that `V15c`'s node identity is already in the `realpath`-then-forward-slash form; instead have `V15e` own an explicit re-canonicalisation that maps each import-edge-graph node's imports.md-form identity onto the form the changed-file path is compared in before the transitive-importer match runs.

**Plan edits.**
- `V15e` Adds: replace the clause asserting the `.warp` import-edge-graph node identity (`V15c`) is in the canonical `realpath`-then-forward-slash form with text stating that `V15c`'s node identity is in imports.md's relative-resolution / byte-for-byte form (IMP-1) and that `V15e` re-canonicalises it to the eviction-comparison form before matching; name `V15e` as the owner of that mapping.
- `V15e` Tests / Ships when and `V15e-T` Tests: extend the non-canonical-form vector so it exercises a changed `.warp` file whose importer's node identity, in `V15c`'s native form, differs from the changed-file path's `realpath` form, asserting the importer is still found and its entry dropped.

**Spec edits.** None.

**Pros.** Confines the change to `V15e`; leaves IMP-1's deliberate byte-for-byte-without-`realpath` `.warp` semantics and `V15c` untouched; no `.warp` resolution behaviour changes.

**Cons.** `V15e` must carry an explicit cross-form mapping step, and must state what form `V15a`'s parse-cache key for reachable `.warp` files is in so the three-way match is well-defined.

**Risks.** If `V15a`'s `.warp` cache-key form is left unstated, the mapping is under-specified for one of the three identities.

### Option B — Pin `V15c`'s `.warp` node identity (and `V15a`'s `.warp` parse-cache key) to the `realpath`-then-forward-slash form

**Approach.** Add a `realpath` normalisation step to the `.warp` import-edge-graph node identity in `V15c` (and the parse-cache key `V15a` uses for reachable `.warp` files) so all three identities natively share the invoke-side `realpath`-then-forward-slash form, making `V15e`'s comparison correct as written.

**Plan edits.**
- `V15c` Adds (and `V15c-T`): state that the import-edge-graph node identity is canonicalised through `realpath`-then-forward-slash, and reconcile that with IMP-1's byte-for-byte final-segment rule.
- `V15a`: state that the parse-cache key for reachable `.warp` files uses the same `realpath`-then-forward-slash form.

**Spec edits.**
- `docs/spec_topics/imports.md`: IMP-1 currently mandates byte-for-byte matching against `FileSystem.readdir` output with no `realpath` step, on every host regardless of case-equivalence model. Adding a `realpath` step to `.warp` resolution contradicts that rule, so the spec must be amended first (per the Spec-drift rule) before the plan can adopt it.

**Pros.** All three identities share one form; `V15e`'s comparison needs no mapping layer.

**Cons.** Changes `.warp` resolution semantics the spec deliberately defines as byte-for-byte-without-`realpath`; forces a spec edit and ripples into `V15c`/`V15a` resolution behaviour beyond the eviction concern.

**Risks.** Re-introducing `realpath` into `.warp` resolution risks regressing IMP-1's case-sensitivity guarantee, whose byte-exactness was a deliberate cross-host design choice.

### Recommendation

Take Option A: keep `.warp` resolution as imports.md IMP-1 specifies and give `V15e` an explicit re-canonicalisation that maps `V15c`'s import-edge-graph node identity onto the form the changed-file path is compared in, naming `V15e` as the owner. The fix must also state what form `V15a`'s parse-cache key for reachable `.warp` files is in, so all three identities in the match are well-defined. Edge case the implementer must watch: a deletion/rename of a `.warp` file (the `unlink` arm) cannot be `realpath`-canonicalised after the fact, so the mapping must match against the stored node identity rather than re-resolving a path that no longer exists.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one reads `V15e` literally and assumes `V15c` already yields a `realpath`-canonicalised node identity (it does not), and ships an eviction whose transitive-importer match silently misses on symlinked or case-variant hosts; another notices the mismatch and invents an ad-hoc reconciliation with no named owner. The first produces the exact stale-parse-cache serve `V15e` exists to prevent, surfacing only on hot reload on those hosts.
