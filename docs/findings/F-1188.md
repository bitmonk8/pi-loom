## Finding

The normative five-sub-step `session_shutdown` teardown sequence — the ordered list of sub-steps (1) "Stop accepting new work", (2) "Cancel in-flight invocations", (3) "Await subagent disposal", (4) "Close watchers", (5) "Detach forwarding listeners", together with the `**Per-step isolation**` paragraph that immediately follows it — is the core teardown algorithm the PIC commits the runtime to. It currently lives inside `docs/spec_topics/pi-integration-contract/patch-skew-degradation.md`, a file whose own title and opening paragraph are scoped to a different and narrower concern (the transient Pi-side / loom-side `SessionShutdownEvent['reason']` snapshot disagreement during a `pi-coding-agent` patch window).

The sibling file `docs/spec_topics/pi-integration-contract/session-shutdown-semantics.md` — which is the file an implementer looking up "what does the session-shutdown handler do?" by filename will land on — opens its first body paragraph with **"The five-sub-step sequence above defines the abort-and-await sequence; this paragraph closes the three obligations the sequence does not by itself state…"**. There is no five-sub-step sequence above in that file; the only headings above it inside the file are the `# Session shutdown semantics` title and the `<a id="session-shutdown-semantics">` anchor stub. The phrase is a dangling intra-file back-reference that points at content now located in a differently-named sibling file two clicks of cross-navigation away.

The downstream consequences are not limited to that one dangling word. `session-shutdown-semantics.md`'s subsequent bullets ("**Per-invocation operator visibility (clean-cancel path)**", "**Partial-append fate during teardown**", "**`invoke`-parent observation rule**") cite "sub-step 2", "sub-step 3", and "sub-step 5" by number; the `<a id="factory-ordering-pin">` paragraph at the bottom of the same file refers to "step 4 above" and "step 5 of the **Extension entry point** sequence, above"; the coverage-matrix row `cka-31` describes its scope as "`pi-integration-contract/patch-skew-degradation.md` §`session_shutdown` sub-step 3"; and `V9g-session-shutdown.md`'s implementer reading-list cites the *same algorithm* by two different filenames. The whole vocabulary of "sub-step *N*" has slipped one filename to the left of the page that owns the topic the names describe.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f5e89f4` (`docs: enforce 100KB hard / 30KB recommended size cap on spec set`, 2026-06-04) — proximate cause; built on `5e8b499` (`pi-loom spec: resolve "Hot-reload teardown contract for the loom extension instance is unspecified"`, 2026-05-06), `7f37dd7` (`pi-loom spec: resolve "Session-swap behaviour for in-flight loom invocations is under-specified"`, 2026-05-07), and `75a9626` (`pi-loom spec: resolve "SessionShutdownEvent.reason closed set has no build-time pin against the SDK type"`, 2026-05-09).
**History:** `5e8b499` introduced the `session_shutdown` handler section into `spec_topics/pi-integration-contract.md` as a single in-file block: the umbrella subscription paragraph immediately followed by the five-sub-step ordered list. `7f37dd7` appended the `<a id="session-shutdown-semantics">` anchor and the `*Session-swap behaviour for in-flight invocations.*` clarifying paragraph (whose body opens with "The five-sub-step sequence above defines…") inside the same file; at that point the "above" pointer resolved correctly because both the algorithm and the clarifier lived in the same file. `75a9626` inserted the `**Patch-skew degradation contract.**` paragraph between the *Unknown-reason rule* and the five-sub-step list. `f5e89f4` then mechanically split `pi-integration-contract.md` into per-section files to satisfy the 30 KB / 100 KB size cap; the splitter sectioned the file along its bolded paragraph titles, so the five-sub-step list inherited the section header `Patch skew degradation` from the contract paragraph that had been inserted immediately above it in `75a9626`, while the `session-shutdown-semantics` anchor block from `7f37dd7` was emitted into its own sibling file. The intra-file "above" pointer survived the split verbatim; it now reaches across files in a directory listing where the target file's title no longer advertises what the pointer is pointing at. The defect is the *interaction* of the four commits — each individually correct against the file shape it acted on — rather than any single one of them in isolation.

## Consequence

**Severity:** advisory

The algorithm is normative and complete in its current location, so a determined implementer reading the PIC end-to-end will eventually find it. The cost is paid by every reader who lands on `session-shutdown-semantics.md` expecting the session-shutdown algorithm — they read a paragraph that asserts a "five-sub-step sequence above" with no such sequence above it in the file, then have to discover, by guess or by directory listing, that the algorithm lives in a sibling whose title advertises an unrelated topic. The same misdirection bleeds into spec cross-references (six topic files cite sub-step ordinals authored in a file whose title does not mention sub-steps), into coverage-matrix descriptions, and into plan leaves. None of this blocks implementation; all of it slows comprehension and audit of one of the PIC's most cross-cited algorithms.
