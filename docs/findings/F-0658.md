## Finding

Two inline "loom 1.0 seam" blockquotes in `conversation-drive.md` — the *mid-loom user-session replacement* block and the *typed-query supported provider set* block — duplicate, on one side, prose that already appears in the surrounding paragraph and, on the other, the deferred-feature description that `future-considerations/surface-extensions.md` owns. The mid-loom block's "single factory-captured reference" MUST restates the immediately-preceding sentence ("The `pi` reference here is the `ExtensionAPI` captured by the extension factory … which the runtime holds for the lifetime of each loom invocation") and then re-describes the same `withSession` re-acquisition that `surface-extensions.md#mid-loom-user-session-replacement` defines as the deferred feature. The typed-query block's first sentence ("is referenced by the load-time `loom/load/typed-query-unsupported-provider` warning emitter and the runtime guard that synthesises the unsupported-provider `TransportError`") is a paraphrase of the *Provider compatibility for typed queries* paragraph directly above it, and its closing sentence re-describes the same deferred extension cross-referenced from `surface-extensions.md` lines 22–23 as "Typed-query support for providers without named-tool forcing". The typed-query block additionally carries an explicit `*Non-normative implementation note.*` ("loom 1.0 expects the provider set to be exposed as a single named runtime constant … a maintainability convention, not an observable conformance point") embedded inside a normative-prose section, where its non-normative status is signalled only by the italic prefix rather than by structural separation.

The pattern raises a divergence risk: each surface (PIC blockquote vs. surrounding paragraph vs. `surface-extensions.md` entry) can drift independently on the same fact (single-capture site, deferred re-acquisition shape, supported-provider set, JSON-mode fallback plan). Both blockquotes also carry named anchors (`v1-seam-mid-loom-user-session-replacement` / `loom-1-0-seam-mid-loom-user-session-replacement` and `v1-seam-typed-query-supported-provider-set` / `loom-1-0-seam-typed-query-supported-provider-set`) that are referenced from `surface-extensions.md`, `future-considerations/model-changes-and-non-goals.md`, `pi-integration-contract/tool-registration-lifetime.md`, and `pi-integration-contract/version-bump-step2.md`, so a bare deletion of the blockquotes would break four inbound deep-links; the fix must preserve those anchor IDs while removing the duplicated prose.

## Solution Space

**Shape:** single

### Recommendation

Reduce each blockquote to an anchor-bearing pointer so the duplicated prose is consolidated to its owner while inbound deep-links keep resolving.

For the *mid-loom user-session replacement* block: replace the current four-sentence blockquote with a one-sentence blockquote (or an inline anchor-bearing sentence) that carries both `<a id="loom-1-0-seam-mid-loom-user-session-replacement"></a>` and `<a id="v1-seam-mid-loom-user-session-replacement"></a>`, states only that *this is the single-capture seam for the deferred re-acquisition extension*, and cross-references `future-considerations/surface-extensions.md#mid-loom-user-session-replacement` for the deferred feature description. The "MUST read `pi` from a single factory-captured reference / MUST NOT re-acquire" obligation is already carried by the surrounding paragraph's "which the runtime holds for the lifetime of each loom invocation"; if the editor judges that prose insufficiently imperative, the MUST can be promoted there in the same edit — but it should appear once, not twice on the same page.

For the *typed-query supported provider set* block: replace it with the same anchor-only construction — preserve `<a id="loom-1-0-seam-typed-query-supported-provider-set"></a>` and `<a id="v1-seam-typed-query-supported-provider-set"></a>`, and either (a) move the non-normative single-named-constant maintainability note to `docs/spec_topics/implementation-notes.md` and cross-reference it, or (b) drop it entirely (the surrounding paragraph already lists the four providers, and no normative consumer depends on the constant being single). The "emitter and guard reference this set" sentence should not be repeated — the surrounding paragraph already specifies both consumers. The deferred-feature pointer should be the cross-reference to `future-considerations/surface-extensions.md` lines 22–23 rather than an inline re-description.

Edge cases the implementer must watch:

- Both legacy `v1-seam-…` anchors and the newer `loom-1-0-seam-…` aliases must survive on each replacement; `grep -rn 'v1-seam-mid-loom-user-session-replacement\|loom-1-0-seam-mid-loom-user-session-replacement\|v1-seam-typed-query-supported-provider-set\|loom-1-0-seam-typed-query-supported-provider-set' docs/` enumerates the inbound links to verify post-edit.
- The mid-loom block's "loom 1.0 looms never trigger replacement" invariant is already stated in the surrounding paragraph and is also independently anchored from `future-considerations/model-changes-and-non-goals.md` *No concurrent user sessions* via the same `#v1-seam-mid-loom-user-session-replacement` deep-link; the replacement pointer must therefore land at exactly the spot in the prose where that invariant is asserted, not before the surrounding paragraph (so the link target's neighbourhood still talks about the lifetime guarantee).
- The typed-query block's anchor sites in `tool-registration-lifetime.md` (and via it `version-bump-step2.md`) use the mid-loom anchor, not the typed-query anchor, so only inbound links to the typed-query anchor are from `surface-extensions.md` line 23 — verify post-edit but the surface is narrower than the mid-loom case.

## Consequence

**Severity:** advisory

No observable runtime behaviour changes either way; the prose is correct as written. The cost is editorial: three independent locations carry the same fact, so an editor updating one is not warned by structure to update the other two, and the typed-query block embeds a non-normative maintainability convention inside otherwise-normative prose without structural separation.
