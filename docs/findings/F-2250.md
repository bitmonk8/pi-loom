## Problem

Step 1 of the Pi version-bump procedure (`version-bump-intro.md`, *Re-typecheck against the new package*) describes the module-import-resolution carve-out (the closed family `TS2305` / `TS2307` / `TS2614` / `TS2724`) twice within the same paragraph, several sentences apart, with conflicting routing. The first mention is a parenthetical aside whose trailing "it is handled by step 5 below" clause can be read as routing the module-import family to the step-5 snapshot-`literals` edit, while the later authoritative passage routes the family through step 2(a)'s surface-inventory update and explicitly says it is "not covered by step 5". A contributor can over-count the carve-outs as three distinct exceptions, or attempt the wrong remediation arm (snapshot `literals` edit instead of import-specifier re-pin).

## Issue introduction
**Verdict:** defect-introduced-by-specific-commit
**Introducing commits:** `75a9626` (2026-05-09 — "pi-loom spec: resolve \"SessionShutdownEvent.reason closed set has no build-time pin against the SDK type\"")
**History:** The step-1 paragraph was authored at `75a9626` already carrying both mentions in their split form: the same added paragraph introduces the parenthetical "(the second is the module-import-resolution failure mode … discussed later in this step): it is handled by step 5 below" and, several sentences on, "is a second expected exception, and is **not** covered by step 5: route every TS code in that family through step 2's surface-inventory update." `git log -S` on both phrasings (`the second is the module-import-resolution`, `module-import-resolution failure mode`) returns `75a9626` as the earliest commit on the file's history. Later commits only refined the surrounding text — `6f898a9` (2026-06-05) reshaped the TS-code-family wording and `e785f77` (2026-06-07) addressed six clarity/scope gaps — without removing the dual-mention structure, which has therefore been present since the carve-out's inception.

## Solution approach

Describe the module-import-resolution carve-out exactly once, keeping the later passage as the single authoritative definition (second — not third — exception, not covered by step 5, routed through `#bump-step-2-positive`'s surface-inventory update with the import specifier re-pinned in the same edit). Rewrite the first parenthetical mention so the "handled by step 5 below" clause binds unambiguously to the type-equality-assertion exception only, and demote the module-import reference there to a forward pointer carrying no handling claim.
