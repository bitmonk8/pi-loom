## Finding

GOV-16's *Per-page inline-label prefix table* in `governance/stable-inline-labels.md` carries a live table for registered inline-label prefixes, then closes the block with a single prose sentence standing in for the retired-prefix sub-table:

> *Retired inline-label prefixes.* Empty in loom 1.0; the sub-table is append-only on the same terms as the *Retired prefixes* sub-table for REQ-IDs and is materialised here when the first inline-label prefix retires.

No actual sub-table exists at this location — only a "materialised here when…" announcement of content to be added later. This diverges from the corpus precedent for empty append-only retirement sub-tables: `governance/anchor-scheme-and-retired.md` materialises its *Retired anchor aliases* sub-table as a real header + separator row with zero data rows, accompanied by per-column definitions, rather than as a deferred-materialisation note. The REQ-ID *Retired prefixes* sub-table (`req-id-prefix-table-retired.md`) is likewise a concrete table.

The defect is a cruft/consistency one: the sentence mixes a current-state fact ("Empty in loom 1.0") and a real normative contract ("append-only on the same terms as the *Retired prefixes* sub-table") with a future-work announcement ("materialised here when the first inline-label prefix retires"). The append-only contract is worth keeping; the deferred-materialisation framing should be replaced with the empty table itself so the retirement surface is structurally present and consistent with sibling sub-tables.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `c4ad105` (2026-05-08, "pi-loom spec: resolve \"HC3-a–HC3-e and NOCEIL-N labels escape GOV-3/GOV-8\"")
**History:** `git log -S 'materialised here when the first'` traces the placeholder sentence to `c4ad105`, the commit that first introduced the GOV-16 inline-label identifier class in the then-monolithic `governance.md`. The block was authored from the outset as the prose announcement *"Empty in V1; … materialised here when the first inline-label prefix retires"* — the sub-table was never materialised. The later file split `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set") moved the block verbatim into `stable-inline-labels.md` (with the `V1` → `loom 1.0` spelling update); it did not introduce the defect. The placeholder has therefore been present since the GOV-16 feature's inception.

## Solution Space

**Shape:** single

### Recommendation

Replace the *Retired inline-label prefixes* placeholder sentence with a materialised empty append-only sub-table, mirroring the *Retired prefixes* sub-table in `req-id-prefix-table-retired.md`: a labelled sub-section heading, a header row plus separator row, and no data rows. Retain a one-sentence append-only contract statement ("append-only on the same terms as the *Retired prefixes* sub-table for REQ-IDs; a retired inline-label prefix is never reused") and drop the "materialised here when the first … retires" future-work clause.

Edge cases for the implementer:

- The *Inline-label grammar and closure invariant* requires that "retired inline-label prefixes carry the same pinned tail form they held when live." Make that recoverable from the retired sub-table by carrying a `Tail form` column alongside the `Retired prefixes` analog columns (`Inline-label prefix | Formerly | Retired in | Tail form`), rather than copying the REQ-ID three-column shape verbatim — the REQ-ID retired table has no tail-form concept.
- Keep the `Retired in` column's value-space identical to the *Retired prefixes* sub-table (7-character abbreviated commit SHA or release tag, no prose), so GOV-7 *Delete* / *Merge* retirements record consistently across both identifier classes.

## Consequence

**Severity:** cosmetic

A reader scanning the GOV-16 block for the retired-inline-label-prefix record finds a "to be added later" note instead of the empty table its siblings present, creating a minor structural inconsistency across the three append-only retirement sub-tables. No implementer or tooling behaviour diverges: the retired-prefix union is empty in loom 1.0 either way.
