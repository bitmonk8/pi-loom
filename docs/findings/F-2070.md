## Finding

The corpus uses the phrase **"on the same footing as [X]"** as a load-bearing normative connector — it asserts that the subject is governed by the *same rule, obligation, or constraint* as some referent X. It is not occasional: the exact phrase appears ~100 times across `docs/spec_topics/`, heaviest in `version-bump-step2.md` (≈39 occurrences) and `audit-recognised-shapes.md` (≈16), with further concentrations in `audit-target-categories.md`, `audit-resolution.md`, `version-bump-triggers.md`, and the session-shutdown PIC pages. (The original framing's "~30 times … most concentrated in `audit-failures.md`" undercounts and mis-attributes the concentration; the defect itself is real and wider than stated.)

The phrase is an analogy, not a definition. "Footing" does not name *which* properties of X are equated with the subject — the same rule? the same routing? the same severity? the same exemption mechanism? In the majority of sites the surrounding clause narrows the meaning adequately (e.g. "walked on the same footing as a hand-authored file" plainly means "subject to the same include-glob walk"). But in dense normative paragraphs that already reference several candidate rules, a bare "on the same footing as X" leaves the reader to guess which of the in-scope rules is "the footing," and two readers can reasonably equate different property-sets. The spec never defines "on the same footing as" as a term of art, so there is no canonical disambiguation a reader can fall back on.

## Issue introduction
**Verdict:** spread-since-introduction
**Introducing commits:** `e5cd514` ("pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\"", 2026-05-09) originated the idiom; it then proliferated across ~100 subsequent commits.
**History:** The spec corpus under `docs/` is git-tracked (first docs commit `b87d4a6`, 2026-05-04). `git log -S"on the same footing as" -- docs/` shows the phrase was absent from the inception commit and first entered five days later in `e5cd514`, where the inventory-closure audit cluster was authored — that commit already uses it repeatedly as a connector ("on the same footing as the capability members", "on the same footing as category (3)", etc.). It was never introduced as a single defect; it is a stylistic connector that was seeded in the audit-cluster authoring and accreted organically across the corpus as later commits reused the established idiom. No single later commit "broke" anything — the defect is the cumulative, undefined-by-design spread of the analogy.

## Solution Space

**Shape:** multiple

### Option A — Define "on the same footing as" once as a term of art

- **Approach:** Add a short glossary entry in `docs/spec_topics/glossary.md` pinning the connector's meaning: "*on the same footing as X* — the subject is governed by the identical normative rule(s), routing, and obligation that X is governed by, as established at X's defining site; it equates the governing rule, not merely the subject matter." Leave existing callsites untouched.
- **Spec edits:** one new glossary entry; no body-prose churn.
- **Pros:** Proportionate to a nit; single-site edit; preserves the existing terse phrasing the corpus relies on; gives every callsite a canonical fallback reading.
- **Cons:** A generic definition still does not, by itself, identify *which* of several candidate rules is "the footing" in a paragraph that cites more than one; the few genuinely-ambiguous dense sites remain ambiguous.
- **Risks:** Low. Risk is only that readers still mis-resolve the handful of multi-candidate paragraphs.

### Option B — Mechanically replace each occurrence with the explicit equated rule

- **Approach:** At every callsite, replace "on the same footing as X" with an explicit statement of the equated obligation (e.g. "subject to the same five-family routing as X").
- **Spec edits:** ~100 edits across the audit cluster and PIC pages.
- **Pros:** Eliminates the ambiguity everywhere; no reliance on a fallback definition.
- **Cons:** Very wide-surface, high-churn edit on a nit; high risk of introducing wording errors or mis-stating the equated rule at individual sites; inflates diff far beyond the defect's severity.
- **Risks:** Moderate-to-high — wide mechanical rewrites of dense normative prose are error-prone and hard to review.

### Recommendation

Adopt **Option A** (define the connector once in the glossary), and additionally rewrite only the small number of dense paragraphs where more than one candidate rule is in scope so the referent of "footing" is explicit. This gives every callsite a canonical reading at single-site cost while sparing the corpus the high-churn, error-prone wholesale rewrite of Option B. Edge case for the implementer: when disambiguating a dense site, equate the *governing rule* (routing/obligation), not the subject matter, consistent with the glossary definition — do not silently narrow or widen the rule the original analogy intended.

## Consequence

**Severity:** advisory

In most callsites context resolves the connector, so a working system is still producible. In the dense audit-cluster and version-bump paragraphs that already cite multiple rules, the bare analogy imposes interpretive burden and admits divergent readings of which obligation is being equated — a guidance gap rather than a hard correctness fork.
