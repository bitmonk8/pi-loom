## Finding

The **Serialised content format** paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` builds each per-`Diagnostic` block out of three line classes: a mandatory main line `"<file>:<line>:<col>: <code>: <message>"`, an *optional* two-space-indented `"\n  hint: <hint>"` continuation "when a hint is present", and *zero or more* two-space-indented related-site lines `"  <file>:<line>:<col>: <message>"` (one per `Diagnostic.related[]` element). The hint continuation is introduced as "optionally followed by"; the related-site lines are introduced separately as "appended on its own line". Both bullet-points use the verb "appended" and both produce two-space-indented continuations, but the paragraph never says which class comes first when *both* are present, nor whether related-site lines may interleave with the hint.

Per-block separation is pinned (a blank line separates successive `Diagnostic` blocks in a multi-error batch), so the ambiguity is intra-block, not inter-block. The `content` field is the normative renderer input *and* the textual artefact test harnesses, LSP integrations, and the loom system-note renderer parse. Two conformant implementations — one that emits hint-then-related, one that emits related-then-hint — both satisfy the prose verbatim yet produce byte-divergent `content` strings on every diagnostic that carries both a `hint` and at least one `related[]` entry (e.g. a collision diagnostic whose hint suggests a rename and whose `related` array points at the colliding declarations). DIAG-4 (Message column normative, character-for-character emission) is silent on inter-line composition of the surrounding `content` string and so does not absorb the gap.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `fae85d3` (2026-05-04, *Tighten spec for implementation-plan readiness*)
**History:** The original **Serialisation to Pi's flat shape** paragraph entered the then-monolithic `spec.md` in `fae85d3` as a single sentence: *"… optionally followed by `"\n  hint: <hint>"` when a hint is present. Related sites are appended as additional indented lines."* The two appended-line classes (hint continuation, related-site lines) were both authored "appended" without an order rule. Commit `fecb504` split `spec.md` into `spec_topics/` and carried the paragraph verbatim into `spec_topics/diagnostics.md`; `f5e89f4` (2026-06-04) split `diagnostics.md` into the subdirectory layout and carried the paragraph verbatim again into `spec_topics/diagnostics/diagnostic-shape.md`. Commit `8837a93` (2026-06-05, *resolve "Serialised content related-site line format is undefined"*) replaced the terse *"Related sites are appended as additional indented lines."* sentence with the current per-line format spec (`"  <file>:<line>:<col>: <message>"`, two-space indent matching the hint line, `range.start` endpoint, no `<code>` prefix), but kept the "appended" framing for both line classes and did not introduce an intra-block ordering rule. The ordering gap has therefore been present since the paragraph's first authoring; the `8837a93` enrichment closed the per-line *format* gap while leaving the relative-order question untouched.

## Solution Space

**Shape:** single

### Recommendation

Amend the **Serialised content format** paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` to pin the intra-block line order normatively: within a single `Diagnostic` block the lines render in the fixed sequence **(1) main line, (2) optional `hint` continuation (zero or one line, when `hint` is present), (3) related-site lines (zero or more, one per `Diagnostic.related[]` element, in array order)**. Related-site lines MUST NOT interleave with or precede the hint continuation. The blank-line separator between successive `Diagnostic` blocks in a multi-error batch is unchanged.

Edge cases the implementer must watch:

- A `Diagnostic` with `hint` absent and `related[]` non-empty: the main line is immediately followed by the related-site lines, with no intervening blank.
- A `Diagnostic` with `hint` present and `related[]` empty: the main line is immediately followed by the hint continuation, with no trailing blank inside the block.
- A `Diagnostic` with both absent: a single-line block (just the main line).
- Related-site order tracks `Diagnostic.related[]` array order verbatim — the spec already pins the per-element format but the renderer MUST preserve array order; state this explicitly alongside the new ordering rule so that test harnesses can assert on the sequence as well as on individual lines.
- Location-less diagnostics (the closed set enumerated in the **Location-less diagnostics** paragraph) drop the `<file>:<line>:<col>:` prefix on the main line only; the hint and related-site continuations are unaffected by that rule and the new ordering rule applies unchanged. (Related-site entries always carry a located site per the existing `{ file, range, message }` shape pin.)

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge on the byte sequence of `content` whenever a `Diagnostic` carries both `hint` and `related[]`. Downstream consumers (the loom system-note renderer, LSP integrations, conformance test harnesses parsing the `content` string) need a deterministic intra-block layout to round-trip or assert on the rendered text; without it, golden-file tests against `content` are non-portable across conformant implementations.
