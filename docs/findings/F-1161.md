## Finding

Four executable REQ-IDs each bundle multiple independent obligations under one anchor, defeating the cite-by-ID contract that GOV-1 / GOV-3 / GOV-8 / GOV-9 deliver. A reviewer or test that fails one half of the bundle cannot cite a narrower identifier than the whole, and a future GOV-8 *Split* will have to retire the bundled ID and reissue N fresh ones — exactly the lifecycle bookkeeping the previously-retired GOV-21 → {GOV-25..GOV-29} and GOV-12 → {GOV-30, GOV-31} splits established as the precedent for this pattern:

- **DISC-1** (`docs/spec_topics/discovery/discovery-sources.md#disc-1`, "Home-directory expansion") bundles (a) the positive obligation that `~/`-expansion MUST route through `FileSystem.homedir()`, and (b) two distinct negative obligations — implementations MUST NOT read `process.env.HOME` / `process.env.USERPROFILE` directly, and MUST NOT use any platform-conditional branch. The plan's `V10a-T` test bullet for DISC-1 already enumerates these as three sub-clauses ("`~/` expands only via `FileSystem.homedir()` — no `~user`/env/platform branch"), confirming the obligations are independently testable.
- **DISC-6** (`docs/spec_topics/discovery/package-and-settings.md#disc-6`, "Package walk is bounded") bundles (a) the global `looms.scanPackagesMaxFiles` / `looms.scanPackagesTimeoutMs` cap with its `loom/load/discovery-slow` warning and tie-break rule, (b) the per-read deadline race scheduled through `Clock.setTimeout` with its `loom/load/package-read-timeout` warning and per-read-vs-global ordering rule, and (c) the wholesale `looms.scanPackages: false` disable. `V10b-T` already splits its DISC-6 line into three sub-bullets and a misreading-red ("settings-sourced bounds reach the walk", "per-read deadline scales with the operator override"), each addressing a distinct sub-clause.
- **FRNT-1** (`docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md#frnt-1`, `tool_loop`) bundles (a) the `max_rounds` semantics (free-phase scope, per-query and per-callee independence, `0` disables, exhaustion → `QueryError{tool_loop_exhausted}`, exempt forced-respond terminator) with (b) the parse-time range-rejection rule (out-of-range `max_rounds` and `respond_repair.attempts` → `loom/load/frontmatter-value-out-of-range`, including the integer-ness rule). The latter is the same range-rejection diagnostic shared with `respond_repair.attempts`, an obligation FRNT-1 only happens to also house.
- **PIC-8** (`docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md#pic-8`, "Restore-failure protocol") bundles four sequential MUSTs labelled (a)–(d) inside one anchor: (a) one re-attempt with the same snapshot, (b) emit `loom/runtime/active-set-restore-failed` with templated `message` / `hint` on second failure, (c) emit a `display: true` `loom-system-note` with the verbatim template, and (d) propagate the original exception. `V9f-T` already cites these as discrete observables ("triggers one re-attempt, then `active-set-restore-failed` (E) + a `display:true` note, and propagates the original error"), so the test scaffolding to discriminate them already exists.

Separately, the DISC-1 cross-reference inside the rule body links to the `pi-integration-contract.md` hub page rather than to PIC-13's defining anchor (`pi-integration-contract/host-interfaces-services.md#pic-13`). Per GOV-9, every cross-page reference to a normative rule on another topic page MUST resolve as a `#prefix-n` URL fragment of the depended-upon REQ-ID; a link to the hub paraphrases past that contract. The defining anchor exists and is the canonical home for `FileSystem.homedir()` (it back-cites DISC-1 by name), so the retarget is a single-byte fix.

## Solution Space

**Shape:** multiple

This finding bundles five distinct edits (one xref retarget + four independent REQ-ID splits). Each split is a self-contained GOV-8 *Split* operation on a different page and a different test-leaf. Sequencing matters: the DISC-1 xref retarget is the smallest, scope-bounding edit and should land first so it does not get tangled into the larger DISC-1 split diff; the four splits then proceed in any order, each as its own commit, each closing under GOV-8 by retiring the bundled ID on the owning page's `## Retired REQ-IDs` table and appending N fresh IDs at the tail.

### Option A — DISC-1 hub-page xref retarget to PIC-13

**Approach.** In the DISC-1 paragraph (`docs/spec_topics/discovery/discovery-sources.md`, line ~37), replace the `[Pi Integration Contract — FakeFileSystem / FileSystem interface](../pi-integration-contract.md)` hub link with `[PIC-13](../pi-integration-contract/host-interfaces-services.md#pic-13)`. PIC-13 is the defining anchor for `FileSystem.homedir()` and already back-cites DISC-1 by name, so the retarget is bidirectional-symmetric.

**Spec edits.** Single link retarget; no anchor lifecycle event.

**Pros.** One-line, no REQ-ID retirement, immediately closes the GOV-9 violation. Scope-bounding for the four splits that follow.

**Cons.** None.

**Risks.** None.

### Option B — DISC-1 split into DISC-1a / DISC-1b / DISC-1c

**Approach.** Retire DISC-1 per GOV-8 *Split* (`anchor-scheme-and-retired.md` precedent: GOV-21 *Split* row). Append three fresh IDs at the tail of `discovery-sources.md`'s REQ-ID sequence:

- **DISC-1a (positive seam-routing MUST).** `~/`-prefixed paths MUST be expanded via `FileSystem.homedir()`. Enumerate the six binding sites (global discovery root, global package roots, global settings file, settings `loomPaths` entries, `--loom` CLI components, `~user` non-honour clause).
- **DISC-1b (env-var read prohibition).** Implementations MUST NOT read `process.env.HOME` or `process.env.USERPROFILE` directly.
- **DISC-1c (platform-branch prohibition).** Implementations MUST NOT use any platform-conditional branch — the seam is the single source of truth.

Record the retirement on `## Retired REQ-IDs` of `discovery-sources.md` per GOV-8. Update `coverage-matrix.md` row `DISC-1, DISC-2, DISC-3 | V10a` to enumerate DISC-1a/b/c.

**Spec edits.** One paragraph restructured into three dual-form anchor sites; one retirement row on the same page; one coverage-matrix row update.

**Pros.** The plan-side `V10a-T` already enumerates these three sub-clauses, so the test scaffolding does not change shape — only its `**Tests.**` bullet citations.

**Cons.** Three new anchors on one page; minor visual churn in the rendered document.

**Risks.** None beyond ordinary GOV-8 bookkeeping.

### Option E — PIC-8 split into PIC-8a / PIC-8b / PIC-8c / PIC-8d

**Approach.** Retire PIC-8 per GOV-8 *Split*. Append four fresh IDs at the tail of `tool-registration-lifetime.md`'s REQ-ID sequence, one per labelled MUST in the existing paragraph:

- **PIC-8a (single re-attempt).** On first restore throw / reject, re-attempt exactly once with the same snapshot. The retry MUST NOT chain back beyond the single re-attempt.
- **PIC-8b (`active-set-restore-failed` diagnostic).** On second failure, emit `loom/runtime/active-set-restore-failed` (E, runtime) with `message` carrying the underlying error's message and `hint` listing the snapshot tool names.
- **PIC-8c (`display:true` system note).** Emit a `display: true` `loom-system-note` whose `content` is the verbatim template `loom: failed to restore tool active-set after /<name>; the user session may have unexpected tools active. Run /reload to reset.` (only `<name>` substituted).
- **PIC-8d (original-error propagation).** Propagate the original exception / terminal `Err` the `finally` was protecting; restore failure does not mask the inner error, and the cascade-twin dedup treats the active-set-restore note as independent.

Record the retirement on `## Retired REQ-IDs` of `tool-registration-lifetime.md`. Update `coverage-matrix.md` row `PIC-8, PIC-19 | V9f` to enumerate PIC-8a/b/c/d.

**Spec edits.** One paragraph restructured into four dual-form anchor sites; one retirement row; one coverage-matrix row update. The subagent-mode carve-out sentence at the paragraph's tail attaches to the cluster as a closing note (it scopes all four sub-IDs equally).

**Pros.** `V9f-T`'s existing PIC-8 bullet already discriminates the four observables and maps 1:1.

**Cons.** Four new anchors on one page. Slightly more bookkeeping than the other splits.

**Risks.** None.

### Recommendation

Land Option A first as a single one-line retarget commit — it closes the GOV-9 violation immediately and removes that concern from the larger DISC-1 split's diff. Then land Options B, C, D, E in any order, each as its own GOV-8 *Split* commit on its own page, each updating its own `## Retired REQ-IDs` table and its own `coverage-matrix.md` row in the same commit. Each split is a textbook GOV-8 *Split* with the GOV-21 → {GOV-25..29} and GOV-12 → {GOV-30, GOV-31} entries in `anchor-scheme-and-retired.md#retired-req-ids` as worked precedent. The plan-side test leaves (`V10a-T`, `V10b-T`, `V6e-T`, `V9f-T`) already discriminate the sub-clauses in their `**Tests.**` bullets, so each split's plan-side fan-out is a citation update inside an already-correctly-scoped test — no test redesign is owed. The fresh IDs MUST be appended at each page's tail per GOV-3 (the next free integer under that prefix; numbering never collapses to fill holes), and the per-prefix retirement bookkeeping (`anchor-scheme-and-retired.md`) is a `## Retired REQ-IDs`-row obligation, not a prefix-retirement.

Edge cases the implementer must watch:

- Inbound `#disc-1` / `#disc-6` / `#frnt-1` / `#pic-8` fragment links elsewhere in the corpus (and in `plan_topics/`) MUST be repointed in the same diff as each split, or each becomes a dangling xref the moment its bundled ID retires. `grep -rn '#disc-6\b' docs/` is the audit recipe; the small known set of plan-leaf citations is enumerated in Plan Impact above.
- The DISC-1 paragraph also carries a `<a id="home-directory-expansion"></a>` heading-style anchor that PIC-13 cross-references back to it as `#home-directory-expansion`; the split MUST preserve that anchor on the DISC-1a fragment so the bidirectional back-link from PIC-13 keeps resolving.
- The PIC-8 paragraph's closing sentence about subagent-mode invocations (`Subagent-mode invocations are unaffected — …`) attaches to all four sub-IDs equally; restate it once after the PIC-8d clause as a non-normative scope note, not under any one of the four split IDs.
- PIC-19's first sentence cites "PIC-8 above governs only the step-4 restore"; after the split, retarget that to "PIC-8a..d above govern only the step-4 restore" (or to a stable cluster reference if one is coined).

## Consequence

**Severity:** advisory

A test reviewer who watches `V10b-T`'s per-read-deadline assertion fail cannot cite anything narrower than "DISC-6" in the diagnosis; the surrounding two DISC-6 obligations (the global cap warning and the wholesale-disable path) ride along under the same identifier. The same is true of DISC-1 (seam-routing vs env-var ban vs platform-branch ban), FRNT-1 (semantics vs range-rejection), and PIC-8 (four sequential restore-failure steps). Implementation still proceeds correctly because the plan-side test bullets already discriminate the sub-clauses informally; the cost is borne by traceability — the GOV-9 `#prefix-n` cross-link contract and the GOV-30 lock-step convention both rely on REQ-IDs being narrow enough to cite individually. The DISC-1 hub-page xref violates GOV-9 outright but is a one-line fix. None of this blocks shipping; it accumulates the same kind of GOV-8 *Split* debt the corpus has already paid down twice (GOV-21 → {GOV-25..29}, GOV-12 → {GOV-30, GOV-31}).
