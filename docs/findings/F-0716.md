## Finding

In `PIC-13`'s `FileSystem` interface declaration in `host-interfaces-services.md`, every read member carries an explicit Node-`.code` rejection annotation: `readText` enumerates `ENOENT`/`EACCES`/`EPERM` and a "other I/O failures surface their underlying `.code` unchanged" tail; `exists` pins the `ENOENT â†’ false`, all-other â†’ reject contract; `readdir` and `lstat` defer to "the same Node-style `.code` shape as `readText`"; `realpath` enumerates `ELOOP`/`ENOENT`/`EACCES`/`EPERM`. The seam's only write member, `writeText(path, contents): Promise<void>`, is bare â€” no `.code` enumeration, no deferral to the `readText` shape, no statement of behaviour on rejection. The very next bullet then names `readText`, `writeText`, and `exists` as "the load-bearing surface for [Directory Convention â€” Settings file reads] and for every other normative file read or write the runtime performs", which elevates the omission from a cosmetic asymmetry to a missing normative contract on the seam's only write member.

The `FakeFileSystem` mirror-rejection discipline carries the same asymmetry. The `realpath` bullet explicitly says the in-memory fake "applies the same `.code` rejections (`ELOOP` on a cycle in that table, `ENOENT` on a missing component, `EACCES` / `EPERM` on an injected permission failure) so a symlink-farm fixture exercises the same rejection surface through the fake that production reaches through Node." There is no analogous clause for `writeText`, so a conformance test for a write failure cannot route through `FakeFileSystem` with a known rejection shape, and `V8b`/`V8b-T`'s shared assertion that "the enumerated `FileSystem` members map Node `.code` values" has nothing to assert for `writeText`. The downstream surface is equally undefined: the spec does not say whether a `writeText` rejection becomes a persistent diagnostic, a `ctx.ui.notify` toast, a `loom/load/settings-*`-namespace code, or is silently swallowed.

Loom 1.0 has no normative caller of `writeText` today â€” settings rewrites are owned by Pi per the **Settings write-back key preservation** presupposition in `host-prerequisites.md` â€” so user-visible damage is bounded, but the gap leaves the seam's only write member unconformable and creates a future-caller hazard the first time loom code itself writes a file.

## Solution Space

**Shape:** single

### Recommendation

Annotate `writeText` in the `PIC-13` interface declaration with a Node-`.code` rejection shape that matches the production `fs.promises.writeFile` surface, extend the `FakeFileSystem` mirror-rejections clause to cover `writeText` symmetrically with the `realpath` template, and pin the loom 1.0 surface-disposition as "no normative caller; any 1.0.x caller MUST pin its diagnostic route in the same edit."

Concrete edits:

1. Replace the bare line inside the `interface FileSystem { â€¦ }` block with:
   `writeText(path: string, contents: string): Promise<void>;          // rejects with a Node-style error whose `.code` is `"ENOENT"` for a missing parent directory, `"EACCES"` / `"EPERM"` for permission failures, `"EROFS"` on a read-only filesystem, `"ENOSPC"` when the underlying device is full, and `"EISDIR"` when the path resolves to an existing directory; other I/O failures surface their underlying `.code` unchanged`
   The closed list above is the Node POSIX-wrapper closure for `fs.promises.writeFile` against an absolute path; the trailing "other I/O failures" clause keeps the seam open to Windows-specific codes (e.g. `"EBUSY"`) without requiring the spec to enumerate them, and is identical in form to the tail clause already on `readText` and `realpath`.
2. Extend the per-member `FakeFileSystem` mirror-rejections sentence (currently attached only to the `realpath` bullet) so `writeText` is covered symmetrically: state that the in-memory fake applies the same `.code` rejections (`ENOENT` for a missing parent directory in its in-memory tree, `EACCES`/`EPERM` on an injected permission failure, `EROFS` on a read-only-marked fixture, `ENOSPC` on a quota-marked fixture, `EISDIR` when the path resolves to an existing directory entry), so a write-failure fixture exercises the production surface through the fake.
3. Add one runtime-side surface sentence under the per-member bullets: loom 1.0 has no normative caller of `writeText` â€” settings writes are owned by Pi per the [**Settings write-back key preservation**](./host-prerequisites.md#settings-write-back-preservation-presupposition) presupposition â€” so a `writeText` rejection has no normative observable in loom 1.0, and any 1.0.x patch that introduces a runtime caller MUST, in the same edit, pin its diagnostic route (persistent diagnostic, `ctx.ui.notify` toast, or a `loom/load/settings-*`-namespace code).

Edge cases the implementer must watch:

- The `EISDIR` rejection is asymmetric with the read side: `readText` against a directory rejects with `EISDIR` on POSIX and surfaces `EPERM`/`EACCES` on Windows; Node smooths the difference inconsistently. The fake's contract is the source of truth â€” a fake that rejects with `EISDIR` on writes against a directory satisfies `V8b-T` regardless of which production OS surfaces it, and the open-tail clause covers the Windows variants.
- The "other I/O failures surface their underlying `.code` unchanged" tail must be retained verbatim from the `readText` parallel so a future Windows-only code like `EBUSY` does not silently violate a closed-list reading of the enumeration.
- `V8b`'s `Tests.` bullet already asserts "the enumerated `FileSystem` members map Node `.code` values"; once the spec pins `writeText`'s mapping, the test obligation discharges by widening the existing test surface to drive `writeText` rejections through the fake â€” no plan-leaf retitling is needed.

## Consequence

**Severity:** advisory

Two reasonable implementers will diverge on what a rejected `writeText` looks like â€” one fake may reject with `ENOSPC`/`EROFS`-style Node-style codes mirroring `fs.promises.writeFile`, another may reject with a string or a custom error â€” and conformance tests for the "load-bearing surface for every normative file write" claim cannot be authored against the seam. There is no loom 1.0 caller of `writeText` today (per the `host-prerequisites.md` write-back presupposition), so the immediate observer impact is bounded; the contract is needed to discharge `PIC-13`'s `V8b`/`V8b-T` test obligation and to keep the surface ready for the first loom-side write.
