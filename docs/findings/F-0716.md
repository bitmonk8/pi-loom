## Finding

In `PIC-13`'s `FileSystem` interface declaration in `host-interfaces-services.md`, every read member carries an explicit Node-`.code` rejection annotation: `readText` enumerates `ENOENT`/`EACCES`/`EPERM`; `exists` pins the `ENOENT â†’ false`, all-other â†’ reject contract; `readdir` and `lstat` defer to "the same Node-style `.code` shape as `readText`"; `realpath` enumerates `ELOOP`/`ENOENT`/`EACCES`/`EPERM`. Only `writeText(path, contents): Promise<void>` is bare â€” no `.code` enumeration, no deferral to the `readText` shape, no statement of behaviour on rejection. The very next bullet then names `readText`, `writeText`, and `exists` as "the load-bearing surface for [Directory Convention â€” Settings file reads] and for every other normative file write the runtime performs", elevating the gap from a cosmetic asymmetry to a missing normative contract on the seam's only write member.

The downstream contract that consumes write rejections is also undefined: the spec does not say whether a `writeText` rejection surfaces as a persistent diagnostic, a `ctx.ui.notify` toast, a `loom/load/settings-*`-namespace code, or is silently swallowed. The two paired plan leaves (`V8b` and `V8b-T`) both cite `PIC-13` as the source of the seam's `.code` mapping and ship a test asserting "the enumerated `FileSystem` members map Node `.code` values" â€” but `PIC-13` supplies no `writeText` mapping to assert.

The asymmetry is also dropped by the existing `FakeFileSystem` rejection-surface guarantee: the realpath bullet explicitly says the fake "applies the same `.code` rejections" so a fixture exercises the production surface through the fake; without an analogous pin on `writeText`, a conformance test for a settings-write failure cannot route through `FakeFileSystem` with a known rejection shape.

## Solution Space

**Shape:** single

### Recommendation

Annotate `writeText` in the `PIC-13` interface declaration with a Node-`.code` rejection shape that matches the production `fs.promises.writeFile` surface and the realpath bullet's `FakeFileSystem` mirror-rejections discipline, and add one bullet under the per-member bullets stating how the runtime surfaces a `writeText` rejection.

Concrete edits:

1. Replace the bare `writeText` line with:
   `writeText(path: string, contents: string): Promise<void>;          // rejects with a Node-style error whose `.code` is `"ENOENT"` for a missing parent directory, `"EACCES"` / `"EPERM"` for permission failures, `"EROFS"` on a read-only filesystem, `"ENOSPC"` when the underlying device is full, and `"EISDIR"` when the path resolves to an existing directory; other I/O failures surface their underlying `.code` unchanged`
   (the closed list above is the Node POSIX-wrapper closure for `fs.promises.writeFile` against an absolute path; the trailing "other I/O failures" clause keeps the seam open to Windows-specific codes such as `"EBUSY"` without requiring the spec to enumerate them).
2. Extend the `FakeFileSystem` mirror-rejections clause already attached to `realpath` to cover `writeText` symmetrically: state that the in-memory fake applies the same `.code` rejections (constructor-injected per-path) so a write-failure fixture exercises the production surface through the fake.
3. Add the runtime-side surface rule in one sentence: there is no normative caller of `writeText` in loom 1.0 (settings writes are owned by Pi per the **Settings write-back key preservation** presupposition on `host-prerequisites.md`), so a `writeText` rejection has no normative observable in loom 1.0; any loom 1.0.x patch that introduces a runtime caller of `writeText` MUST, in the same edit, pin its diagnostic route.

Edge cases the implementer must watch:
- The `EISDIR` rejection is asymmetric with the read side (`readText` against a directory rejects with `EISDIR` on POSIX and `EPERM`/`EACCES` on Windows; Node smooths the difference inconsistently). The fake's contract is the source of truth â€” a fake that rejects with `EISDIR` on writes against a directory satisfies `V8b-T` regardless of which production OS surfaces it.
- The clause "other I/O failures surface their underlying `.code` unchanged" must be retained verbatim from the `readText` parallel so a future Windows-only code like `EBUSY` does not silently violate the closed-list reading of the enumeration.
- `V8b`'s `Tests.` bullet already asserts "the enumerated `FileSystem` members map Node `.code` values"; once the spec pins a `writeText` mapping, the test obligation is automatically discharged, no plan-leaf retitling needed â€” only the test surface must be widened to drive `writeText` rejections through the fake.

## Consequence

**Severity:** advisory

Two reasonable implementers will diverge on what a rejected `writeText` looks like â€” one fake may reject with `ENOSPC`/`EROFS`-style codes mirroring Node's `fs.promises.writeFile`, another may reject with a string or a custom error, and conformance tests for the "load-bearing surface for every normative file write" claim cannot be written against the seam. Loom 1.0 itself has no normative caller of `writeText` (settings rewrites are owned by Pi per the **Settings write-back key preservation** presupposition), so user-visible damage is bounded today; the contract is needed to keep `PIC-13`'s test obligation discharged and to keep the surface ready for any loom-side write the runtime adds later.
