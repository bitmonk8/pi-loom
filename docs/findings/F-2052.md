## Finding

The `GOV-N (governance rule)` entry in `docs/spec_topics/glossary.md` states that the live governance-rule range is "currently running through `GOV-23`, with some IDs retired". The authoritative source, `docs/spec_topics/governance.md`, registers rules through `GOV-31` (the live set is `GOV-1`…`GOV-31` with several retired). The inline literal is therefore stale by eight IDs.

The same sentence directs readers to the authoritative source — "cite `governance.md` for the authoritative range rather than a fixed upper bound" — so the parenthetical contradicts its own guidance: it hardcodes exactly the fixed upper bound the following clause tells readers not to rely on. A reader treating the glossary as orientation will believe no `GOV-N` above 23 exists and read in-corpus citations of `GOV-24`…`GOV-31` as dangling.

The fixed number is redundant with the deferral clause and is structurally guaranteed to drift: it goes stale on every governance-rule allocation, with no mechanism keeping it in lock-step.

## Issue introduction
**Verdict:** stale-after-extension
**Introducing commit:** `96cfc08` (2026-06-04, "pi-loom spec: resolve \"Glossary GOV-N entry under-describes the live GOV range and scope\"")
**Staleness-introducing commits:** the later `governance.md` extensions that allocated `GOV-24`…`GOV-31` without co-updating the glossary literal — including `fe8aa5c` (2026-06-05, added `GOV-30`/`GOV-31` from the `GOV-12` split) and the intervening `GOV-24`…`GOV-29` allocations.
**History:** `git log -S 'through \`GOV-23\`' -- docs/spec_topics/glossary.md` shows the literal entered the glossary at `96cfc08`, replacing an earlier `(GOV-1 through GOV-8)` wording. At that commit `GOV-23` was the live maximum, so the literal was correct on the day it was written. Subsequent commits extended `governance.md` with new live `GOV-N` IDs (through `GOV-31`) without co-updating the glossary literal — even though `96cfc08`'s own forward-pointing clause ("cite `governance.md` for the authoritative range rather than a fixed upper bound") was authored precisely to discourage relying on that literal. The defect is drift across a sequence of single-page edits, not a single mis-statement; the authoring page anticipated the failure mode and embedded a hedge instead of removing the literal.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the parenthetical in the `GOV-N (governance rule)` entry of `docs/spec_topics/glossary.md` to remove the fixed numeric upper bound entirely, deferring to `governance.md` for the live set while retaining the existing forward-link to [Retired REQ-IDs](./governance/anchor-scheme-and-retired.md#retired-req-ids) for the retired sub-list. The entry already carries the correct deferral clause ("cite `governance.md` for the authoritative range rather than a fixed upper bound"); the edit aligns the entry with it.

Do not restate the maximum as `GOV-31` — substituting one fixed number for another recreates the same per-allocation lock-step drift this finding addresses. The live set must be described only by reference to `governance.md`.

## Consequence

**Severity:** advisory

A reader using the glossary entry as orientation is told the live range stops at `GOV-23` and will treat the eight live rules `GOV-24`…`GOV-31` (and their in-corpus citations) as nonexistent or dangling. No runtime behaviour is affected, but the entry actively misleads on a number while naming a different page as authoritative.
