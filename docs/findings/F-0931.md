## Problem

V6b's `loom/parse/non-trailing-default` rule and the default-vs-non-default arity
partition used for positional binding are both positional rules: the spec
(`frontmatter-fields-a.md`, Defaults paragraph) states them over `params:` *declaration
order*. Evaluating them correctly presupposes that the V6a frontmatter parser returns
`params:` entries in source declaration order, but neither V6a's Adds nor V6b's
Adds/Tests states that the positional rules are load-bearing on this. The standard `yaml`
parse happens to preserve source order, so the obvious implementation works — the gap is
that the dependence is undocumented and untested, so a later parser-shape change (or a
duplicate-key collapse) could break the rule with no signal at parse time.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** `V6b-params-defaults.md` was created in `c6a664e` already carrying the
"no-non-defaulted-after-defaulted ordering rule" in Adds and the corresponding Tests
bullet (`git show c6a664e:docs/plan_topics/V6b-params-defaults.md`). The registry code
`loom/parse/non-trailing-default` was added to that bullet later in `7678da2`
(`git log -S 'non-trailing-default'`), but the positional rule — and therefore the
unstated declaration-order dependence — has existed since inception. No commit ever added
a declaration-order / order-preservation statement to `V6a` or `V6b`
(`git log -S 'preserv' -G 'declaration order' -- V6a… V6b…` returns nothing). The gap is
original to the leaf, not introduced by a later edit.

## Solution approach

In `docs/plan_topics/V6b-params-defaults.md`, state that the `non-trailing-default` rule
and the arity partition are evaluated over `params:` entries in their source declaration
order as returned by V6a's parser, and extend the Tests so the order-sensitivity fixture
discriminates ordering rather than passing under any order. In
`docs/plan_topics/V6a-frontmatter-contract.md` Adds, record that the parser exposes
`params:` in source declaration order and that the chosen `yaml` parse shape must preserve
declaration order and define behaviour for duplicate `params:` keys.

## Solution constraints

- The spec is read-only for this fix; `frontmatter-fields-a.md` already encodes the
  `params:` declaration-order requirement, so no spec edit is needed.
