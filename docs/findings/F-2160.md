## Problem

The bare-`Type`-positions enumeration in grammar.md's Type-grammar section names "type-ascription contexts" as a position where a bare `Type` may appear, but no `TypeAscription` production exists, the phrase is never defined, and it carries no cross-reference. Loom 1.0 has only two inline angle-bracket type-annotation surfaces — the `invoke<Type>` return annotation and the query `@<Schema>` explicit-ascription form — and defines no general `expr as Type` cast operator, so a reader cannot map the phrase to a concrete surface or tell whether a distinct ascription form must be parsed. The same undefined term ("a type ascription") propagates into the `loom/parse/void-in-non-return-position` row of diagnostics/code-registry-parse.md, leaving that diagnostic's position list without a concrete syntactic site to test against.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `f841b12` ("spec: resolve Cluster 2 (literal sublanguage + grammar appendix)", 2026-05-04)
**History:** `git log -S 'type-ascription contexts' --reverse --all` identifies `f841b12` as the first commit to introduce the phrase, in the then-`spec_topics/grammar.md` Type-grammar section. It entered as part of the bare-`Type`-positions enumeration ("…and `invoke<Type>` / type-ascription contexts") and was never accompanied by a production or cross-reference. The phrase has carried forward unchanged through the `docs/` relocation (`31ff060`) and the void-grammar reconciliation (`4c76dfb`), the latter of which propagated the same undefined term ("a type ascription") into the new `void`-in-non-return-position enumeration. The defect has thus been present since the term was coined and has been amplified, never repaired, by subsequent edits.

## Solution approach

Clarify the grammar.md Type-grammar bare-`Type`-positions sentence by replacing the undefined "type-ascription contexts" phrase with the two concrete inline angle-bracket type-annotation surfaces it denotes — the `invoke<Type>` return annotation and the query `@<Schema>` explicit-ascription form — and add forward-links to [Invocation — Typed return](./invocation.md) and [Query — Explicit form](./query/query-forms.md#explicit-ascription-override). Rename the same paragraph's `void`-position "a type ascription" entry, and the matching "type ascription" item in diagnostics/code-registry-parse.md's `loom/parse/void-in-non-return-position` row, to the concrete `@<Schema>` query ascription so both name a cross-referenceable site. Optionally state that loom 1.0 defines no standalone `expr as Type` cast operator, bounding the term to these inline annotation forms.

## Solution constraints

- Keep grammar.md's lowered-schema `Result` carve-out — which currently admits `Result` in "`invoke<Type>` / type-ascription contexts" — consistent with the renamed phrase.
