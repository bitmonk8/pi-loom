## Finding

`docs/spec_topics/binder/defaulting-system-note-echo.md` §*System-note rendering* rule 2 stacks two requirements on the truncation behaviour that read against each other. The first sentence is a MUST: "When the rendered note exceeds 120 code points, the runtime MUST replace the overflow with a trailing `…` (U+2026) and the resulting note MUST be exactly 120 code points (the `…` counts toward the cap)." The third sentence is a MAY: "Implementations MAY additionally back the truncation point off to the nearest preceding extended grapheme cluster boundary as a rendering courtesy, provided the resulting note is still **≤120** code points."

A literal reading of the MUST forbids any rendered length other than exactly 120 on overflow. A literal reading of the MAY explicitly permits a result anywhere from 0 to 120 code points when an implementation chooses the cluster-courtesy back-off (e.g. a 117-code-point output when the boundary at scalar 119 falls inside a grapheme cluster that started at 117). The MAY does not say it overrides the MUST, and no precedence rule appears anywhere in the section. The tests-MUST-NOT-assert-cluster-aware-behaviour clause mitigates the test surface but does not repair the normative contradiction: two implementations conforming strictly to the MUST (always 120) and strictly to the MAY (cluster-aware back-off to ≤120) will produce different observable byte sequences on the same input, and a third reading — that the MAY is an authorised carve-out — is the most plausible authorial intent but is not stated.

The same ambiguity propagates to every site that flows through this rule: the success echo's line-level cap bullet at the end of *Echo policy*, the `needs_info` / `ambiguous` / malformed-envelope / transport-failure failure-arm notes that cite *System-note rendering*, and the `V11e` test leaf, which currently asserts "exactly 120 code points" without noting the cluster-back-off exception.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commit:** `6efbd8a` ("pi-loom spec: resolve 'System-note 120-codepoint cap truncation unit'", 2026-05-05)
**History:** Rule 2 originally read "Truncation operates on whole code points (or grapheme clusters) — never on UTF-16 code units" — a single permissive clause whose ambiguity the prior review caught. The fix at `6efbd8a` replaced that one clause with the current three-clause structure: a strict MUST pinning "exactly 120 code points" on overflow, a `≤120` no-`…` clause for the under-cap case, and the new MAY permitting cluster-boundary back-off "provided the resulting note is still ≤120 code points." The MUST and the MAY were authored in the same edit and shipped together, so the contradiction is not the residue of an interaction between two unrelated changes — it is intrinsic to the resolution as drafted. The file move at `f5e89f4` (binder.md → binder/defaulting-system-note-echo.md) carried the text across the split unchanged.

## Solution Space

**Shape:** multiple

### Option A — Pin "exactly 120" as the default; name the MAY as the sole authorised exception

**Approach.** Keep the MUST that on overflow the rendered note is exactly 120 code points, and rewrite the MAY clause to say it is the *only* permitted departure from that requirement, with the cluster-back-off case explicitly named as the carve-out.

**Spec edits.** In rule 2, replace the MAY sentence with: "As the sole permitted exception, implementations MAY back the truncation point off to the nearest preceding extended grapheme cluster boundary when the scalar-aligned truncation would split a grapheme cluster; in that case the rendered note MUST still be ≤120 code points and the trailing `…` MUST still be appended. This back-off is non-normative and tests MUST NOT assert cluster-aware behaviour." No change to the `V11e` rule-2 test bullet (which already says "exactly 120"); add a sentence acknowledging the cluster-back-off carve-out is not asserted.

**Pros.** Preserves the strong determinism for the overwhelming-majority case (ASCII + BMP text near the cap). Tests can keep pinning byte-exact output on representative inputs. Matches the most plausible authorial intent.

**Cons.** Spec text remains nuanced (MUST + named exception); implementers must still read the carve-out carefully.

**Risks.** None material; the carve-out is already in the spec, this option just labels it as the authorised override of the MUST.

### Option B — Demote the cap to a SHOULD on the exact length

**Approach.** Soften the "MUST be exactly 120" to "MUST be ≤120, SHOULD be exactly 120 where the scalar boundary at position 120 does not split a grapheme cluster."

**Spec edits.** In rule 2, change the second sentence to: "When the rendered note exceeds 120 code points, the runtime MUST replace the overflow with a trailing `…` (U+2026) and the resulting note MUST be ≤120 code points; the resulting note SHOULD be exactly 120 code points where doing so does not split an extended grapheme cluster." The MAY back-off sentence then becomes purely descriptive (it is one way the SHOULD can be honoured). Update `V11e`/`V11e-T` rule-2 bullet wording: assertion becomes "≤120 code points" with a `…` present whenever the pre-truncation length exceeded 120.

**Pros.** No internal contradiction at any reading level; rule 2 collapses to a single coherent obligation.

**Cons.** Loses the test surface for "exact 120 on overflow" across the board, not just on cluster-straddling inputs — even ASCII overflow is now only SHOULD-pinned at exactly 120, weakening byte-determinism guarantees for the success echo and the failure-arm notes.

**Risks.** Downstream renderers / fixture suites that rely on hash-stable output for the 120-cap case (e.g. snapshot tests over the success-echo line) lose a guarantee they may already be implicitly depending on.

### Recommendation

**Option A.** It preserves the byte-determinism the rest of the binder system-note discipline trades on (the *Reference rendering* paragraph immediately below rule 2, the BNDR-6 canonical-rendering table, and the V11e shipping assertion "exactly 120 code points" all rest on a strict cap), and the cluster-back-off scope is genuinely the only legitimate departure — there is no reason to weaken the cap in the universal ASCII/BMP case to accommodate it. Implementer-relevant edge cases: (i) the `…` (U+2026) is itself one code point, and the carve-out's "≤120" upper bound includes the `…`, so a cluster-back-off result of `<117 prefix scalars> + …` is the worst case; (ii) the rule-1 whitespace collapse runs before rule 2, so the 120-scalar measurement is taken over the rule-1 output (already pinned in the current text and should be preserved); (iii) the cluster carve-out applies only when scalar-aligned truncation would *split* a cluster — implementations that never invoke `Intl.Segmenter` always honour the MUST trivially and remain conforming.

## Consequence

**Severity:** correctness

Two reasonable implementers who read the rules strictly will produce different observable system-note byte sequences for any rendered note whose 120th-scalar boundary falls inside a multi-scalar grapheme cluster (ZWJ-joined family emoji, regional-indicator pairs, combining-mark sequences, etc.). The cluster-aware implementer can render at 117–119 code points; the scalar-only implementer always renders at exactly 120. Downstream renderers (Pi TUI, log scrapers, screenshot-anchored docs, hash-stable fixtures) will see a divergence that the spec text — as currently written — provides no normative basis to call non-conforming on either side.
