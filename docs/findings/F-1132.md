## Problem

`docs/spec_topics/binder/defaulting-system-note-echo.md` §*System-note rendering* rule 2 (the Length cap, under `id="system-note-rendering"`) stacks a strict MUST and a permissive MAY that read against each other. The MUST requires that on overflow the rendered note be exactly 120 code points (the `…` counting toward the cap); the MAY permits an implementation to back the truncation point off to the nearest preceding grapheme-cluster boundary, yielding a result below 120 code points. No precedence rule states which governs. A scalar-only implementation (always 120) and a cluster-aware implementation (≤120) both conform strictly yet produce different observable byte sequences for any note whose 120th-scalar boundary falls inside a multi-scalar grapheme cluster.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commit:** `6efbd8a` ("pi-loom spec: resolve 'System-note 120-codepoint cap truncation unit'", 2026-05-05)
**History:** Rule 2 originally read "Truncation operates on whole code points (or grapheme clusters) — never on UTF-16 code units" — a single permissive clause whose ambiguity the prior review caught. The fix at `6efbd8a` replaced that one clause with the current three-clause structure: a strict MUST pinning "exactly 120 code points" on overflow, a `≤120` no-`…` clause for the under-cap case, and the new MAY permitting cluster-boundary back-off "provided the resulting note is still ≤120 code points." The MUST and the MAY were authored in the same edit and shipped together, so the contradiction is not the residue of an interaction between two unrelated changes — it is intrinsic to the resolution as drafted. The file move at `f5e89f4` (binder.md → binder/defaulting-system-note-echo.md) carried the text across the split unchanged.

## Solution approach

Rewrite the cluster-back-off sentence in rule 2 so it reads as the sole authorised exception to the exactly-120 MUST, supplying the missing precedence: the scalar-aligned case stays bound to exactly 120 code points, and the grapheme-cluster back-off is the one permitted departure (resulting note ≤120 code points, with the trailing `…` still appended). Scope the carve-out to the case where scalar-aligned truncation would split a grapheme cluster.

## Solution constraints

- Preserve the existing "tests MUST NOT assert cluster-aware behaviour" clause; the rewrite must not make the back-off result test-observable.
- Out of scope: the plan's `V11e` test leaf — this is a spec-only edit (`fix_surface: spec`).
