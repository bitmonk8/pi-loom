## Finding

The `loom/runtime/internal-error` row in [`diagnostics/code-registry-runtime.md`](../../docs/spec_topics/diagnostics/code-registry-runtime.md) and the **runtime-defect surface** paragraph in [`errors-and-results/error-model.md`](../../docs/spec_topics/errors-and-results/error-model.md#runtime-panics) both describe how the runtime fills the diagnostic's `message` and `hint` fields with a literal property-access recipe: *"`message` is the underlying `error.message`; `hint` carries the underlying `error.stack` for operator triage (rendered as `"<no stack available>"` when `error.stack` is falsy)"*. The same phrasing appears verbatim in the system-note format string `"loom /<name> aborted with internal error: <error.message>"`. The recipe presumes the caught value is `Error`-shaped — an object with a string-valued `.message` and a string-valued `.stack`. JavaScript permits `throw <any value>`: strings, numbers, `null`, plain objects, proxies whose property accessors throw, objects with non-string `.stack`. The runtime owns the catch site at the boundaries enumerated in the runtime-defect surface (a `tool.execute(...)` adapter, the slash-command dispatch wrap, the `invoke` spawn-site wrap, post-probe SDK calls), so it cannot constrain the shape of what arrives.

The diagnostics registry's per-page preamble — *"Wherever a row below carries a caught throw's error message … in a `message` / `hint` field, that string is the throw's underlying-error string"* — and the [`§6 Underlying-error coercion`](../../docs/spec_topics/diagnostics/placeholder-rendering-b.md#underlying-error-coercion) rule together cover the *message-carrying* paths: an implementer who follows the coercion gets the same byte-identical result for a `throw "boom"`, a `throw null`, or a `throw new Error("boom")`. They do **not** cover the `hint` field on this row, because that field does not carry the throw's *message* — it carries its *stack*. The falsy-stack guard the row already states (`<no stack available>` when `error.stack` is falsy) handles the `throw "string"` and `throw null` cases by accident (a string has no `.stack`, `null` cannot be dereferenced — but the latter is the bug, not the guard), and leaves undefined behaviour on the throwing-property-access path (a proxy whose `.stack` getter throws), the non-string-stack path (an object whose `.stack` is a number or another object), and the synchronous `TypeError` path on `null` / `undefined` throws.

Two reasonable implementers reading the current prose will diverge: one routes the catch through the underlying-error coercion plus a parallel stack-tolerant rule and crashes on no input; the other writes the literal `e.message` / `e.stack` reads the prose describes and crashes the runtime-defect surface — the surface whose entire purpose is to absorb unexpected throws — on a non-Error throw. That regression is exactly the failure mode the §6 coercion was authored to prevent for `<error.message>` and `<dispose error first line>`; the rule simply does not reach this row's `hint` or the still-literal `error.message` / `error.stack` phrasing in the two pages above.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**

- `f8f4813` (2026-05-05) — *"pi-loom plan: resolve 'Panic routing has no defined path for unexpected interpreter exceptions'"* — first added the `loom/runtime/internal-error` row to the diagnostics registry and the runtime-defect-surface paragraph to `errors-and-results.md` with the literal `"<message> is the underlying error.message; <hint> carries the underlying error.stack (rendered as '<no stack available>' when error.stack is falsy)"` phrasing on both sites. The Error-shape assumption is present from the row's first appearance; no coercion rule yet exists in §6 to repair it.
- `bf85092` (2026-06-06) — *"pi-loom spec: resolve 'non-Error-throw coercion + session_start pi.getCommands() failure contract'"* — introduced the §6 *Underlying-error coercion* sub-rule in `placeholder-rendering-b.md` and the matching code-registry-runtime preamble (*"Wherever a row below carries a caught throw's error message … in a `message` / `hint` field, that string is the throw's underlying-error string"*). The coercion repaired the **message-carrying** paths (the `<error.message>` placeholder in the *Message template*, the `message` field, and any `hint` carrying a throw's *message*) but did not extend to `error.stack` extraction and did not edit the `internal-error` row's prose or the `error-model.md` paragraph to replace the literal `error.message` / `error.stack` field-access phrasing.

**History:** The defect was present-since-inception at the row level (f8f4813); a later, narrower coercion pass (bf85092) covered the message half of the same surface but left the stack half and the literal phrasing in the two source paragraphs. The remaining defect is the residue of that incomplete pass, not a regression.

## Solution Space

**Shape:** single

### Recommendation

State, on the `loom/runtime/internal-error` row and in the matching `error-model.md` runtime-defect-surface paragraph, that the runtime owns the catch site and coerces the caught value `v` through two rules before populating the diagnostic:

1. **Message field and slash-command system-note tail.** Replace every literal occurrence of `error.message` (in row prose, in the system-note format string `"loom /<name> aborted with internal error: <error.message>"`, and in the *Message template* token surround) with a reference to the [`§6 underlying-error string`](../../docs/spec_topics/diagnostics/placeholder-rendering-b.md#underlying-error-coercion): when `v` is an object whose `.message` is a string, that `.message`; otherwise `String(v)`, or `<unreadable>` when `String(v)` itself throws. The `<error.message>` placeholder in the template is already covered by §6 and the page preamble; the prose around it must stop describing it as a literal field access.

2. **Hint field stack extraction.** State the parallel stack-tolerant rule explicitly on the row (preferred — keeps the §6 coercion's scope unchanged): the `hint` carries the underlying `.stack` only when `v` is an object whose own `.stack` is a string, read defensively so that a throwing getter does not propagate out of the catch site; in every other case (non-object throw, missing or non-string `.stack`, throwing `.stack` access, the `String(v)`-coercion-throws fallback) `hint` renders the literal `<no stack available>`. The existing falsy-stack guard is the same rule narrowed to the string-`.stack` case; the new prose generalises it to the full throw domain rather than expanding §6, which is scoped to *error-message* strings, not stacks.

Edit both pages in the same change so the two paragraphs stay byte-aligned. Edge cases the implementer must watch: a proxy whose `.message` getter throws (the §6 coercion's `String(v)` fallback covers it; the result will be `String(v)` evaluated on the proxy, which can itself throw — hence the `<unreadable>` arm); a proxy whose `.stack` getter throws (the rule above renders `<no stack available>`, never propagates); a `throw new Error()` with no message (renders `<no message>` per §6); a non-Error object whose `.stack` is a non-string (renders `<no stack available>`).

## Consequence

**Severity:** correctness

Two implementers reading the row's `error.message` / `error.stack` prose literally — one routing through the page-preamble coercion, one not — produce divergent behaviour on every non-Error throw: a `RuntimeError` cascading out of the runtime-defect surface (the surface specifically designed to absorb unexpected throws), a `<unreadable>` rendering, an `undefined` interpolation, or a recovered diagnostic. The probability is non-zero in practice — host adapters routinely throw bare strings; proxies with throwing accessors arise when SDKs lazy-load — and a runtime-defect surface that itself defects on a non-Error throw forfeits its NOCEIL-3 guarantee that no host throw terminates the user session.
