## Problem

The `loom/runtime/internal-error` row in `diagnostics/code-registry-runtime.md` and the runtime-defect-surface paragraph under `#runtime-panics` in `errors-and-results/error-model.md` both describe filling the diagnostic's `message` from `error.message` and its `hint` from `error.stack` as literal property accesses; the same literal phrasing appears in the system-note format string `"loom /<name> aborted with internal error: <error.message>"`. That recipe presumes the caught value is `Error`-shaped, but JavaScript permits `throw <any value>` (strings, `null`, proxies whose `.stack` / `.message` getters throw, objects whose `.stack` is non-string), and the runtime owns the catch site so cannot constrain the throw shape. The page-preamble `#underlying-error-coercion` rule covers the message-carrying paths but does not reach the `hint` field's stack extraction or the still-literal `error.message` / `error.stack` phrasing. A non-Error throw therefore yields undefined behaviour on the very surface whose purpose is to absorb unexpected throws, forfeiting its NOCEIL-3 guarantee.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**

- `f8f4813` (2026-05-05) — *"pi-loom plan: resolve 'Panic routing has no defined path for unexpected interpreter exceptions'"* — first added the `loom/runtime/internal-error` row to the diagnostics registry and the runtime-defect-surface paragraph to `errors-and-results.md` with the literal `"<message> is the underlying error.message; <hint> carries the underlying error.stack (rendered as '<no stack available>' when error.stack is falsy)"` phrasing on both sites. The Error-shape assumption is present from the row's first appearance; no coercion rule yet exists in §6 to repair it.
- `bf85092` (2026-06-06) — *"pi-loom spec: resolve 'non-Error-throw coercion + session_start pi.getCommands() failure contract'"* — introduced the §6 *Underlying-error coercion* sub-rule in `placeholder-rendering-b.md` and the matching code-registry-runtime preamble (*"Wherever a row below carries a caught throw's error message … in a `message` / `hint` field, that string is the throw's underlying-error string"*). The coercion repaired the **message-carrying** paths (the `<error.message>` placeholder in the *Message template*, the `message` field, and any `hint` carrying a throw's *message*) but did not extend to `error.stack` extraction and did not edit the `internal-error` row's prose or the `error-model.md` paragraph to replace the literal `error.message` / `error.stack` field-access phrasing.

**History:** The defect was present-since-inception at the row level (f8f4813); a later, narrower coercion pass (bf85092) covered the message half of the same surface but left the stack half and the literal phrasing in the two source paragraphs. The remaining defect is the residue of that incomplete pass, not a regression.

## Solution approach

Rewrite the literal `error.message` references on the `loom/runtime/internal-error` row, in the `error-model.md` runtime-defect-surface paragraph, and in the system-note format string to reference the `#underlying-error-coercion` rule rather than describing a literal field access. Add a stack-tolerant rule on the row for the `hint` field: extract the underlying `.stack` only when the caught value is an object whose own `.stack` is a string, read defensively so a throwing getter does not escape the catch site, rendering `<no stack available>` in every other case. Edit both pages in the same change so the two paragraphs stay aligned.

## Solution constraints

- Keep the `#underlying-error-coercion` rule scoped to error-message strings; author the stack-tolerant rule on the row rather than expanding that rule's coercion scope.
