## Problem

ERR-15 makes the `kind`-typed-as-`string` type-openness obligation a MUST but offers only a runtime-emission conformance assertion. A closed-enum implementation passes that runtime test while still violating the type-openness MUST, so the type-level obligation is unfalsifiable as written. An ERR-15-conformant runtime can silently foreclose the deferred user-defined-error-type and `BinderError`-as-variant seam and still report passing.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 02d7ca5 — pi-loom spec: resolve "T02 surface-extension seam obligations not consistently marked" (2026-05-08, Thomas Andersen); 2180f59 — pi-loom spec: resolve "Mid-stream cancellation: no observable rule for loom-runtime conversation behaviour" (2026-05-08, Thomas Andersen)
**History:** The two obligations entered the corpus independently, each in its own same-day commit, and each was authored without a loom-side acceptance criterion from the outset. `git log -S` localises the ERR-15 type-openness seam MUST to 02d7ca5, which added the seam blockquote ("`kind` … MUST be typed as `string`") carrying only the runtime-emission conformance assertion. The mid-stream non-mutation MUST NOTs trace to 2180f59, which introduced the "Mid-stream cancellation, conversation state" paragraph (later split into ERR-8…ERR-12 by cd0e82c and relocated into `error-model.md` by the f5e89f4 size-cap split); that paragraph stated the prohibition against Pi-committed conversation state but never named a loom-side observable. Neither defect is an interaction between the two commits — they are two independent single-commit introductions bundled under one review heading.

## Solution approach

Extend the ERR-15 seam blockquote in `queryerror-variants.md` (anchor `id="err-15"`) with a type-level acceptance criterion that binds the type-openness MUST through a static type-level observable, distinct from the existing runtime-emission conformance assertion.

## Solution constraints

- Do not phrase the criterion in terms of `match`-exhaustiveness diagnostics; loom 1.0 performs no static exhaustiveness check (`expressions.md` **Exhaustiveness**).
