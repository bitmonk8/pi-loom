## Finding

`PIC-21. Renderer exception safety` (in `extension-bootstrap-and-per-loom.md`) instructs the
`loom-system-note` renderer that, on any internal failure, it MUST catch the error and "return a
minimal `Component` that renders the raw `message.content` when `display === true`, or `undefined`
when `display === false`." The renderer's `message: CustomMessage<T>` parameter carries a `content`
field typed `string | (TextContent | ImageContent)[]` (pinned in `host-interfaces-core.md` →
*`CustomMessage`*). "Render the raw `message.content`" is well-defined only for the `string` arm:
for the array arm the rule does not state which blocks are emitted, how `ImageContent` is disposed,
or what separator joins blocks.

In loom 1.0 the gap is latent rather than live. The per-variant `display`/`content` table in
`runtime-event-channel.md` pins `content` to a string for every variant loom emits on this channel
(diagnostic batch lines, the slash-dispatch failure template, structural/recovery note templates,
and the `display: false` empty-string case), so a loom-produced `loom-system-note` never carries the
array arm, and the renderer is registered only for the `loom-system-note` `customType`. The defect
is therefore a guidance gap: PIC-21 is written as if `content` were always a string, but the type it
operates on is a union, and the spec never tells the renderer it may assume the string arm. A
TypeScript implementer must still satisfy the union at the catch site and is given no rule for the
array case, so the "exception-safe by construction" contract is incompletely specified at exactly
the point it claims completeness.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `841a085` (2026-06-06) — "pi-loom spec: resolve \"renderer-throw during render has no defined failure mode\""
**History:** PIC-21 did not exist before `841a085`; that commit authored the entire *Renderer exception safety* clause, including the verbatim fallback "return a minimal `Component` that renders the raw `message.content` when `display === true`." The `CustomMessage.content` union (`string | (TextContent | ImageContent)[]`) was already present in `host-interfaces-core.md` at that time (carried through the `f5e89f4` size-cap split of 2026-06-04, which predates PIC-21). The fallback was thus written against an already-declared union but only addresses its `string` arm — the defect is co-extensive with PIC-21's introduction, not the result of a later edit or a drift between two independently-changed sites.

## Solution Space

**Shape:** single

### Recommendation

At the PIC-21 fallback, state that on the `loom-system-note` channel `message.content` is always a
`string` — every loom-produced variant pins a string `content` per the `runtime-event-channel.md`
per-variant table (including the `content: ""` case) — so the minimal `Component` renders that
string verbatim for `display === true`. Add that the array arm of the `CustomMessage.content` union
is unreachable on this channel and the renderer narrows it away defensively, rendering an empty body
(equivalently `undefined`/no-op) if a non-string `content` is ever observed, so the catch path
remains total without inventing block-linearisation, image-disposition, or separator rules. This
keeps the fallback a guaranteed-string projection and leaves `V7d`'s PIC-21 acceptance test
(renderer returns a minimal `Component` rendering raw `message.content` for `display === true`)
unchanged for the string case while making the union-narrowing obligation explicit.

## Consequence

**Severity:** advisory

Two implementers writing the catch-path fallback would handle the array arm differently (join text
blocks, coerce, drop, or emit `""`), but because loom never emits array `content` on this channel
the divergence has no observable effect in loom 1.0. The cost is implementer friction: the
"exception-safe by construction" obligation requires a fallback whose behaviour on a reachable
union arm is unspecified, leaving the implementer to invent an unwitnessed rule to type-check.
