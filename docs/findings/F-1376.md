## Finding

The `getCommands() snapshot read-only` Tests bullet in `V9b` (and its verbatim mirror in `V9b-T`) describes its oracle as a snapshot whose array mutators throw plus a length/element-order assertion, and then claims this makes "any in-place write fail the test rather than passing silently". The named mechanism does not deliver the universal "any". A throwing-mutator array traps method-form mutations (`push`, `splice`, `sort`, …), and the length/element-order check additionally traps index assignment and length writes. It does **not** observe mutation of an element object's own fields — e.g. `snap[0].name = "x"` — which writes through to a `SlashCommandInfo` member without calling an array mutator and without changing the array's length or element order.

The spec surface this bullet witnesses scopes read-only to the **array**: `registration-steps.md` step 2 says "the returned array is mutable in the type system but MUST be treated as read-only by convention (the loom 1.0 collision check is a single forward pass and never mutates it)". Element-internal mutation is therefore outside the obligation; the defect is purely the universal quantifier "any in-place write", which promises coverage the mechanism cannot provide.

## Solution Space

**Shape:** single

### Recommendation

In the `getCommands() snapshot read-only` Tests bullet of both `V9b-registration-reload-wiring.md` and `V9b-T-registration-reload-wiring.md`, replace the clause `so any in-place write fails the test rather than passing silently` with `so any structural in-place write to the snapshot array (mutator call, index assignment, or length write) fails the test rather than passing silently`. This is the exact set of writes the throwing-mutator array plus length/element-order assertion observes, and it matches the spec's array-scoped read-only obligation. Apply the identical edit to both leaves so the V9b/V9b-T pair stays in lockstep; leave the rest of the bullet (spec link, `cka-23.getcommands` facet citation, the single-forward-pass / instance-and-order description) unchanged.

## Consequence

**Severity:** advisory

The bullet promises that "any in-place write" to the snapshot fails the test, but the stated mechanism (throwing-mutator array + length/element-order assertion) cannot observe element-internal field mutation. An implementer taking the claim literally either writes an assertion the mechanism cannot satisfy or assumes a guarantee that does not hold. Because the spec scopes the read-only obligation to the array itself, no implementer diverges from the spec — this is a guidance gap, not a correctness defect.
