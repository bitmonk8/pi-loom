## Finding

The *Factory-ordering pin* in `session-shutdown-semantics.md` states a MUST: "The `session_shutdown` handler MUST be subscribed (step 4 above) only after the `LoomRegistry` and watcher handles the handler reads are constructed … a `session_shutdown` MUST NOT be reachable against a partially-constructed extension state." Its stated benefit — the closure-captured `LoomRegistry` handle "is never `undefined` and `LoomRegistry.readDrainState()` cannot throw a `TypeError` … in the gap between step 4 returning and registry/watcher construction completing" — is the observable consequence of the ordering rule, but it is written as a trailing rationale clause, not as a verifiable then-clause.

The result is a normative ordering constraint with no test scenario that falsifies a violation. An implementation that subscribes the handler before constructing the registry/watcher (violating the pin) would satisfy every existing acceptance criterion in the cluster, because none of them inject a `session_shutdown` into the step-4→step-5 window and assert the read-site behaviour. The behaviour the pin protects is observable — a `session_shutdown` delivered immediately after step 4 returns, before construction completes, must drive the handler to completion without a `TypeError` from the `readDrainState()` read site — but the spec never states that scenario as an acceptance criterion. The pin is therefore currently a behaviour the implementer is told to honour but not told how to demonstrate.

## Issue introduction
**Verdict:** introduced-by-commit
**Introducing commits:** `52b9101` ("pi-loom spec: resolve T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state", 2026-05-11)
**History:** `git log -S 'partially-constructed extension state'` and `git log -S 'cannot throw a \`TypeError\`'` over `docs/` both bottom out at `52b9101`, which authored the *Factory-ordering pin* into the then-monolithic `pi-integration-contract.md`. `git show 52b9101` confirms the pin entered with its consequence written as the rationale clause "is never `undefined` and `LoomRegistry.readDrainState()` cannot throw a `TypeError`" and no accompanying acceptance criterion. Later commits (`f5e89f4` size-cap reflow, and the subsequent split into `session-shutdown-semantics.md`) relocated the text but preserved the rationale-only framing; the testability gap has been present continuously since the pin's introduction.

## Solution Space

**Shape:** multiple

### Option A — Add an explicit observable acceptance criterion, keep the pin normative

- **Approach:** Retain the MUST and append a falsifiable then-clause plus fixture. State: "For any `session_shutdown` event delivered after step 4 registration, the handler-entry `LoomRegistry.readDrainState()` (step (I) of *Handler control-flow ordering*) MUST NOT throw a `TypeError`. A fixture that injects a `session_shutdown` immediately after step 4 returns — before registry/watcher construction (step 5) completes — MUST observe the handler running to completion without a `TypeError` from the read site."
- **Spec edits:** Convert the rationale clause in the *Factory-ordering pin* paragraph into a normative then-clause, and add the inject-in-the-gap fixture description; carry the criterion into `V9g-T`'s **Tests** and `V9g`'s **Ships when**.
- **Pros:** Preserves the crash-preventing MUST; makes a subscribe-before-construct regression detectable; the scenario is reproducible with the existing DI seams (the factory sequence and `Clock` seam are already test-controllable).
- **Cons:** Requires a fixture that exercises the factory mid-construction, which is more involved than a steady-state handler test.
- **Risks:** If the fixture cannot deterministically interpose between step 4 and step 5 with the current seams, an additional construction-order seam may be needed; verify the factory bootstrap is injectable before pinning the fixture.

### Option B — Demote the pin to a non-normative implementation note

- **Approach:** Reword the pin from a MUST to guidance ("the implementation SHOULD subscribe after construction so the read site cannot observe an undefined handle"), accepting that the ordering is an internal sequencing detail rather than a tested contract.
- **Spec edits:** Replace the MUST with a non-normative note; drop the pin from `cka-30`'s normative coverage and adjust `V9g`/`V9g-T` accordingly.
- **Pros:** Removes the untested MUST, eliminating the coverage gap by lowering the obligation rather than testing it.
- **Cons:** Weakens a constraint that prevents a real startup `TypeError`/crash; the `readDrainState` read-failure fail-safe (`PIC-31`) catches a thrown read at the handler-entry site, but relying on that fallback to mask an ordering bug contradicts the pin's "never `undefined`" guarantee-by-construction intent.
- **Risks:** A subscribe-before-construct implementation would be conformant under this option, shifting the burden onto the `readDrainState` fail-safe path for a condition the spec currently forbids by construction.

### Recommendation

Take **Option A**: keep the MUST and add the inject-in-the-step-4→step-5-gap acceptance criterion. The pin prevents a concrete startup crash and is already mapped to a closing leaf (`cka-30` → `V9g`), so the right fix is to make it falsifiable, not to weaken it. Implementer watch-point: confirm the factory bootstrap exposes a seam that lets the fixture deliver a `session_shutdown` after step 4 but before step 5 completes; if no such seam exists, add one as part of `V9b` rather than demoting the pin.

## Consequence

**Severity:** advisory

If shipped unfixed, the ordering MUST has no test that distinguishes a conformant factory (subscribe-after-construct) from a non-conformant one (subscribe-before-construct). A violating implementation passes the suite yet carries a startup race: a `session_shutdown` arriving in the step-4→step-5 gap reads an `undefined` registry handle and `readDrainState()` throws a `TypeError` at handler entry. The MUST itself is clearly stated, so a careful implementer can still produce a correct system; the gap is verification coverage, not behavioural ambiguity.
