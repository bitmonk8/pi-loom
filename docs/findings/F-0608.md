## Finding

`audit-resolution.md` (paragraph anchored at `audit-category-1-3-inventory-join-key`) justifies the rightmost-segment match for category-(1) and category-(3) by appealing to the carrier-leftmost `path` convention as a property of *existing* entries: *"the existing category-(1)/(3) entry shapes carry the carrier name as the leftmost dot-separated segment of the `path` field rather than the accessed member name"*. The same paragraph then offers `path: "pi.registerCommand"` and `path: "ExtensionContext.modelRegistry"` / `path: "ctx.modelRegistry"` as illustrative entries with no qualifier signalling they are still hypothetical.

`audit-recognised-shapes.md` (paragraph anchored at `audit-target-surface-categories`) describes the same kinds in the opposite direction: it requires that satisfying the three in-scope categories *"MUST extend `SDK_SURFACE_INVENTORY`'s entry-kind taxonomy beyond the existing kinds, with at minimum one new entry kind per category not already covered"*, and that *"The new kinds MUST be coined in the implementation's entry-kind enumeration as a precondition to the audit's first run on `main`"*. The two `SDK_SURFACE_INVENTORY` kinds that *do* exist today (`namespace-function` and `type-union-snapshot`) are both category-(2)-shaped — neither pins a category-(1) or category-(3) `path` convention.

A reader following `audit-resolution.md` first will look for entries that do not exist, or — worse — will read *"carrier name as the leftmost dot-separated segment"* as a fait accompli describing pinned-down kinds, when it is in fact a behavioural requirement the implementation is obligated to satisfy when it coins those kinds. The leftmost-segment-is-the-carrier convention is normative content; presenting it as descriptive lets it slip past as background.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e5cd514` (2026-05-09) "pi-loom spec: resolve 'T35 SDK capability inventory closed-set negative-direction audit'" — landed both halves of the eventual contradiction in the same edit: the `audit-target-surface-categories` paragraph with its *"new kinds MUST be coined"* obligation and the `audit-category-1-3-inventory-join-key` paragraph asserting the carrier-leftmost convention on *"existing category-(1)/(3) entry shapes the H1 leaf pins (per `h1-scaffold.md`)"*. At that point the "existing" framing was tempered by the explicit forward-pointer to `h1-scaffold.md`, which let a reader see the kinds were future-pinned rather than present-pinned. `9e53613` (2026-05-26) "pi-loom spec: resolve T03/T24a/T25/T26 corpus-direction sweep — drop H1 / plan_topics / plan-leaf-ID references from spec corpus" then deleted the `the H1 leaf pins (per [`h1-scaffold.md`](../plan_topics/h1-scaffold.md))` qualifier from the join-key paragraph, leaving the bare "existing" claim with no context. The defect is the *interaction* of the original sloppy "existing" wording with the later corpus-direction sweep that stripped the only hint that "existing" was forward-looking; neither commit on its own is the root cause.
**History:** the `81bf52a` (2026-06-11) "docs: collapse spec/ section pages into spec_topics/" move shifted the prose into per-topic files (`audit-resolution.md` and `audit-recognised-shapes.md`) but did not change the contradictory wording; the two paragraphs were already in separate sections of the same shard before the split, and the split is incidental to the defect.

## Solution Space

**Shape:** single

### Recommendation

Recast the `audit-category-1-3-inventory-join-key` paragraph so the leftmost-segment-is-carrier `path` convention is presented as a normative requirement on the kinds that are yet to be coined, not as a description of existing entries. Concretely:

- Replace *"the existing category-(1)/(3) entry shapes carry the carrier name as the leftmost dot-separated segment of the `path` field rather than the accessed member name — so a leftmost-segment match … cannot identify the accessed member, and the audit MUST instead match on the rightmost segment"* with *"the category-(1) and category-(3) entry kinds that the implementation coins per `audit-recognised-shapes.md` *Target surface categories* MUST carry the carrier name as the leftmost dot-separated segment of the `path` field — `pi.<member>` for category (1) and `ExtensionContext.<member>` / `ctx.<member>` for category (3) — and the audit MUST resolve those entries by rightmost-segment match against the accessed member."*
- Mark the `path: "pi.registerCommand"` and `path: "ExtensionContext.modelRegistry"` / `path: "ctx.modelRegistry"` strings as illustrative target shapes the implementation will produce, not as quotations of currently-pinned entries.
- Add a one-clause back-reference to the *"MUST be coined … as a precondition to the audit's first run on `main`"* sentence in `audit-recognised-shapes.md` so the two paragraphs are visibly co-load-bearing rather than appearing to describe two different worlds.

Edge cases for the implementer: the same paragraph already (correctly) treats category (2)'s `path: "SessionShutdownEvent.reason"` as a real, currently-pinned entry via the existing `type-union-snapshot` kind; the rewrite must not flatten that asymmetry — category (2) has one pre-existing audit-participating kind, categories (1) and (3) have none, and the new wording must keep those facts distinct.

## Consequence

**Severity:** advisory

An implementer wiring up the category-(1)/(3) entry kinds reads the paragraph as documenting an established convention rather than imposing one, and may either hunt for missing entries (wasted cycles, false bug reports against `V18a`) or coin kinds whose `path` shape places the member name at the leftmost segment (the natural reading for `pi.<member>` / `ctx.<member>`), at which point `V18b`'s rightmost-segment resolver silently fails to identify the accessed member.
