## Finding

Across the Pi-integration-contract version-bump cluster the spec repeatedly designates one module as the canonical home for a set of build-time-consumed constants — "the single source-of-truth pinned-constants block in the extension module." That block holds `CAPABILITY_OBLIGATIONS`, the four Step-0 capability-probe constants (Node floor, `AbortSignal`/`AbortController` member list, factory-probable capability list, peer-dep range), the `SessionShutdownEvent['reason']` closed-set snapshot entry, `SHUTDOWN_AWAIT_CAP_MS`, and the two typebox allow-lists (named-import `{ Type }` and member-access `{ Unsafe }`). Multiple independent build-time gates are required to read the *same* block: step 2(a)'s literal-read assertion (`CAPABILITY_OBLIGATIONS.length === 7`, the SDK surface-inventory check), the `npm run typecheck` `SessionShutdownEvent['reason']` brand-string assertion, the Step-0 runtime capability probe, and the typebox inventory-closure audit.

Yet no anchor or path names that module anywhere in the cluster. The only path-constrained artefact in this area is the *type-equality assertion file* (`src/extension/**/*.assert.ts` when co-located under `src/extension/`, per *Audit scope*); the constants block itself — the thing every gate must import — is referred to only by the recurring prose phrase "the extension module" and is never pinned. The closure audit walks `src/**/*.ts`, but whether the constants-defining module sits inside that walked tree (and is therefore itself audited) is also left unstated. Because the spec leans hard on the words "single source of truth" without anchoring a citeable location or pinning a cross-gate import invariant, the obligation that all gates resolve against one shared module rests entirely on prose emphasis rather than on a stated rule.

## Issue introduction
**Verdict:** present-since-inception

**Introducing commits:** `b0cdf1b` (2026-05-06, version-bump procedure / capability-probe authoring era — first spec-body appearance of the "source-of-truth pinned-constants block in the extension module" concept); amplified by `f5e89f4` (2026-06-04, the 100 KB/30 KB size-cap page split that created `version-bump-triggers.md` / `version-bump-step2.md` / `version-bump-intro.md`) and `81bf52a` (the `spec/` → `spec_topics/` collapse), which fanned the references across multiple files.

**History:** The block was designated a single source of truth without ever carrying a concrete module path or a page anchor, and at no point in the file history was a locating anchor (`id="pinned-constants…"` / `id="constants-block…"`) added to the spec body — such anchors appear only in transient review-store artefacts, never in the corpus. The defect therefore did not enter via a regression; it has been present since the constants-block concept was first written. The size-cap split is what made the omission bite, because spreading the gates' references across four files turned an in-page co-location into a cross-file citation gap with no shared anchor to point at.

## Solution Space

**Shape:** multiple

### Option A — Name the canonical constants-block module with a citeable anchor

- **Approach:** Pin a concrete module identifier (e.g. a dedicated constants module under the extension source tree) and attach a stable anchor to it, then repoint every "the extension module"/"single source-of-truth pinned-constants block" reference (version-bump steps 5 and 2(a), capability-probe, session-shutdown-semantics, audit-target-categories) at that anchor.
- **Spec edits:** Add an anchored sentence at the block's natural definition site (capability-probe.md Step 0, which already says "one source of truth inside the extension module") naming the module and stating whether it lies inside the `src/**/*.ts` audited tree; convert the recurring prose references into resolving links to that anchor.
- **Pros:** Maximally concrete; gives the audit and gates an unambiguous import target; resolves the "is the defining module itself walked?" question directly.
- **Cons:** Prescribes source-file layout, which the cluster has elsewhere deliberately left implementation-owned (e.g. the assertion file MAY sit under `test/` or `src/extension/`); risks over-constraining the implementation and inviting drift between the pinned path and the actual layout.
- **Risks:** A future refactor that moves the module silently invalidates the pinned path; the spec would then mis-state reality.

### Option B — Declare the path implementation-owned and pin the single-shared-module import invariant

- **Approach:** Keep the block's physical path implementation-owned (consistent with the cluster's existing posture) but add one normative invariant: all build-time gates and the Step-0 runtime probe MUST import the constants from a single shared module, and that module MUST reside inside the audited `src/**/*.ts` tree (or, if not, the audit scope MUST be widened to include it). Give the invariant an anchor so the scattered references can cite it.
- **Spec edits:** Add an anchored invariant sentence (most naturally at the capability-probe Step 0 "one source of truth" site, or in `inventory-audit-intro.md` alongside `SDK_SURFACE_INVENTORY`) stating the shared-module-import requirement and the in-audited-tree requirement; repoint the "the extension module" references at that anchor.
- **Pros:** Preserves the spec's implementation-owned-layout posture; pins the property that actually matters (one shared module, audit-visible) without dictating a filename; gives every cross-reference a stable citation target.
- **Cons:** Slightly less concrete than a named path; an implementer must still choose the module name.
- **Risks:** If the in-audited-tree clause is omitted, the "is the block itself walked" ambiguity persists; the invariant must explicitly cover the audit-scope question.

### Recommendation

Adopt **Option B**: pin a single-shared-module import invariant with a citeable anchor and require the module to live inside the audited `src/**/*.ts` tree, leaving the concrete filename implementation-owned. This matches the cluster's established treatment of source-file layout (the `*.assert.ts` placement is permissive, not pinned) and fixes the property that the gates depend on — one shared source of truth, audit-visible — without committing the spec to a path it would have to keep in sync. The implementer must watch the audit-scope clause: state explicitly whether the constants module is inside `src/**/*.ts` so the closure audit's coverage of the defining module is not left ambiguous.

## Consequence

**Severity:** advisory

An implementer can still build a working system — the emphatic "single source of truth" prose makes the intent (one shared module imported by every gate) clear enough that two implementers would both create one block. What is missing is a citeable anchor for the cross-references that point at the block and an explicit statement of whether the block's module is inside the audited `src/**/*.ts` tree, leaving the multi-gate shared-import requirement as an inference rather than a pinned obligation.
