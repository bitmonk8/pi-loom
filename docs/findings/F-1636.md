## Problem

DISC-5 (`#disc-5`) requires the `minimatch` engine for `pi.looms` glob resolution, but the canonical `**Loom-package implementation dependencies (loom 1.0).**` bullet in `implementation-notes.md` (anchor `loom-package-implementation-dependencies-loom-1-0`) lists `semver`/`chokidar`/`yaml` and omits `minimatch`. An implementer provisioning loom's `package.json` from that inventory would leave DISC-5's glob resolution unbacked. The dependency's version-skew posture against Pi's own `minimatch` is also unstated.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `1cb54c7` — pi-loom spec: resolve "Glob !/+/- precedence and matcher engine unspecified for pi.looms / loomPaths" (2026-06-06, Thomas Andersen); `1ea0455` — pi-loom spec: resolve "DI-seam vs reference-implementation placement boundary" (2026-06-04, Thomas Andersen); `1064946` — pi-loom plan: resolve "YAML frontmatter parsing mechanism is never declared as a dependency" (2026-06-10, Thomas Andersen)
**History:** The normative `minimatch` reliance in DISC-5, together with the `dist/core/package-manager.js` / `applyPatterns` private-symbol pin, entered the corpus in commit `1cb54c7` (2026-06-06) — `git log -S "applyPatterns"` and `git log -S "minimatch"` against `package-and-settings.md` each return exactly that one commit. The dependency inventory bullet in `implementation-notes.md` predates that reliance (created by `1ea0455`, 2026-06-04, for `semver`) and was subsequently extended for sibling packages (`chokidar`, then `yaml` via `1064946`, 2026-06-10) in response to parallel "X is never declared as a dependency" findings, but was never extended for `minimatch` — `git log -S "minimatch" -- docs/spec_topics/implementation-notes.md` returns nothing. The "undeclared dependency" gap thus arose from the interaction between the reliance-introducing commit and the separately-evolving inventory that closed the sibling cases but never the `minimatch` one.

## Solution approach

Add a `minimatch` entry to the dependency-inventory bullet (anchor `loom-package-implementation-dependencies-loom-1-0`) on the same footing as the existing `semver`/`chokidar`/`yaml` entries, naming `minimatch` as the glob matcher behind DISC-5's `pi.looms` resolution and declaring it in loom's `dependencies` block rather than pinning a version. State the version-skew posture: loom carries its own `minimatch` instance distinct from Pi's, so the DISC-5 parity claim is about dialect identity, not a shared runtime instance.
