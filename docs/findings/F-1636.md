## Finding

DISC-5 (`discovery/package-and-settings.md`) places normative weight on the `minimatch` engine: it pins the glob matcher loom uses to resolve `pi.looms` patterns ("Glob patterns are matched with the `minimatch` engine") and grounds the dialect by asserting parity with the package Pi applies to its own resource arrays. The override-ordering and exact-path operand rules at DISC-5 are loom-owned and depend on loom running this matcher itself.

The canonical dependency inventory â€” the `**Loom-package implementation dependencies (loom 1.0).**` bullet (anchor `loom-package-implementation-dependencies-loom-1-0`) in `implementation-notes.md` â€” enumerates `semver`, `chokidar`, and `yaml`, each declared in loom's own `dependencies` block and each marked out of scope for the inventory-closure audit's target categories. `minimatch` appears nowhere in that inventory, even though it is on the same footing as the three listed packages: a third-party runtime dependency loom carries itself rather than receiving from Pi.

The result is an internal inconsistency in the spec corpus: the package whose presence DISC-5 requires for `pi.looms` resolution has no entry in the document that is supposed to be the complete list of loom's implementation dependencies, and its version-skew posture (loom carrying its own `minimatch` instance distinct from Pi's) is unstated.

## Solution Space

**Shape:** single

### Recommendation

Add a `minimatch` entry to the `**Loom-package implementation dependencies (loom 1.0).**` bullet (anchor `loom-package-implementation-dependencies-loom-1-0`) in `implementation-notes.md`, on the same footing as the existing `semver` / `chokidar` / `yaml` entries:

- State that loom 1.0 depends on the `minimatch` npm package as the glob matcher behind DISC-5's `pi.looms` resolution, declared in the loom package's `dependencies` block, naming the dependency rather than a version.
- Note that loom carries its own `minimatch` instance distinct from Pi's (the DISC-5 parity claim is about dialect identity, not a shared runtime instance), so version skew between loom's `minimatch` and Pi's is a deliberate, tolerated posture.
- Mirror the chokidar/yaml audit-scope clause: `minimatch` is neither one of the four `@earendil-works/*` peer packages nor `typebox`, so it is out of scope for the inventory-closure audit's *Target surface categories* and requires no `// allow-pi-surface:` exemption marker at any `minimatch` import site.

Add a forward-link from DISC-5 (anchor `disc-5`) to the new inventory entry so the dialect-parity sentence anchors to the dependency record rather than leaving the sourcing implicit.

Edge case for the implementer: keep the entry's "names the dependency, not its version" framing consistent with the other three so the entry does not over-pin; the version-skew-with-Pi note is the only addition beyond the chokidar/yaml template.

## Consequence

**Severity:** advisory

An implementer provisioning loom's `package.json` from the canonical dependency inventory would omit `minimatch`, leaving DISC-5's `pi.looms` glob resolution unbacked. The gap is recoverable because DISC-5 itself names the engine and the plan's `H1a` leaf already declares it, but the authoritative inventory is incomplete and the version-skew posture against Pi's own `minimatch` is unstated, leaving the dependency-sourcing decision implicit.
