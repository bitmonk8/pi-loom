## Finding

The prompt-mode untyped-query bullet in `conversation-drive.md` describes the secondary error-detection surface — a synchronous throw from `pi.sendUserMessage(...)` — with the sentence:

> As a narrow secondary mapping, a synchronous throw from `pi.sendUserMessage` (the only failure mode the call surface itself can signal) **is mapped to** `Err(QueryError { kind: "transport", message: <error.message>, http_status: null, provider: <resolved-model-provider>, retryable: false })` — not to `loom/runtime/internal-error`.

The clause is in the indicative ("is mapped to"), not the imperative (no MUST / MUST NOT). The negative half ("not to `loom/runtime/internal-error`") is also descriptive rather than prohibitive. RFC-2119-style normative force is what loom's spec uses elsewhere to make a rule conformance-checkable; this rule has none.

Read strictly, an implementation that wraps the synchronous throw as `loom/runtime/internal-error`, lets it propagate as an uncaught throw out of the slash-command handler, or swallows it and returns `Ok("")` is not unambiguously non-conforming — the spec merely says what *the* mapping is, not what the implementer is required to produce. The paired `V9n-T` test leaf does encode the rule as a red-test, but that test exists against a prose claim the spec itself never elevated to a MUST; the test would still pass against a conforming implementation while a non-conforming one would escape detection if a different implementer skipped the test fixture.

The companion `stopReason: "error"` primary-surface clause in the same paragraph has exactly the same indicative phrasing ("A `stopReason` of `"error"` maps to …"); both clauses share the same defect and should be tightened together so the rule covers both detection branches with the same force.

## Issue introduction

**Verdict:** present-since-inception (regressed by a later rewrite)

The indicative phrasing and the descriptive negative half are present in this first authored form.
- `8c6cfec` — *pi-loom spec: resolve "pi.sendUserMessage returns void; prompt-mode transport-error detection path is mis-described"* (2026-05-07): rewrote the paragraph to split the surface into primary (`AgentSession.errorMessage` probe, later replaced by the `stopReason: "error"` trailing-`assistant` probe) and secondary (synchronous throw), and added the parallel primary-surface "maps to …" sentence. Both halves carried the indicative phrasing forward unchanged.
- `7f75947` and `d37c563` (post-`8c6cfec` rewrites of the *Error detection* surface) preserved the indicative phrasing across the `AgentSession.errorMessage` → `stopReason: "error"` migration; neither tightened the rule.

**History:** The defect is original to the first authored form of the synchronous-throw mapping (`2d2fc8e`) and survived every subsequent rewrite of the surrounding paragraph. No commit ever upgraded the clause to RFC-2119 normative force, and the parallel primary-surface sentence introduced in `8c6cfec` inherited the same indicative phrasing rather than fixing it.

## Solution Space

**Shape:** single

### Recommendation

Rewrite both error-detection mappings in the prompt-mode untyped-query bullet so each carries explicit RFC-2119 force, and pair each positive MUST with an explicit MUST NOT for the prohibited misrouting:

1. **Secondary surface (synchronous throw).** Replace "is mapped to" with "**MUST** be mapped to", and replace "not to `loom/runtime/internal-error`" with "**MUST NOT** be routed to `loom/runtime/internal-error`, **MUST NOT** be allowed to propagate as an uncaught throw out of the slash-command handler, and **MUST NOT** be swallowed (e.g. lowered to `Ok("")`)." Keep the synthesised `QueryError` shape and the `provider`-population rule verbatim.

2. **Primary surface (`stopReason: "error"` probe).** Apply the symmetric tightening to "A `stopReason` of `"error"` maps to …" — make it a MUST, and add an explicit MUST NOT against routing the same condition to `loom/runtime/internal-error` or to the `Ok(string)` extraction. The absent-`errorMessage` → `"provider transport failure"` fallback should likewise be a MUST ("the synthesised `message` **MUST** be the fixed string `"provider transport failure"`").

Implementer-relevant edge cases the rewritten text should still cover (these are already present in the surrounding paragraphs and must not regress):

- The cancellation short-circuit (`loomAbort.signal.aborted` → `kind: "cancelled"`) takes precedence over both detection branches; the new MUSTs apply only after that short-circuit is passed.
- The same dual-surface mapping applies when `pi.sendUserMessage` is the transport call inside a typed query's free phase; the forced respond turn is exempt because it runs off-session through pi-ai's `complete()` and is classified through `provider-error-mapping.md` instead.
- The synthesised `QueryError.provider` is sourced from the loom's resolved `model:` for both detection branches, per the same paragraph (the separate "user-session model vs loom's resolved `model:`" finding tracked below covers whether that source is the right one — this fix should keep the current sourcing rule intact and let that finding alter it independently).

The paired `V9n-T` / `V9n` leaf **Tests** / **Adds** bullets and the `coverage-matrix.md` `cka-24` row already paraphrase these clauses; their wording lift is mechanical once the spec text carries a MUST.

## Consequence

**Severity:** correctness

Two reasonable implementers following the current text can diverge: one writes the explicit transport-error mapping; another lets the synchronous throw escape as `loom/runtime/internal-error` or swallows it, and points at the indicative phrasing to defend the choice. Because the rule governs the user-visible `QueryError` shape returned from prompt-mode untyped queries (and the free-phase of typed queries), the divergence is observable to loom authors and to the runtime-event-channel always-log emission for `kind: "transport"`.
