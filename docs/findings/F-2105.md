## Problem

The probe-wide invariants PIC-3 ("The probe MUST NOT use any member it is itself checking") and PIC-4 ("The probe is `typeof`- and `in`-only — no arity sniffing, no return-shape sniffing, no construction") in `capability-probe.md` are framed purely as implementation prohibitions with no falsifiable observable. On a conformant host a probe that obeys them and one that violates them (e.g. constructing `new AbortController()` inside the `AbortController` check) produce identical output. The two diverge only on a non-conformant host: a violating probe touches or constructs the absent member, throws, and is trapped by PIC-6's try/catch, routing to `details.kind = "probe-failed"` instead of the specific kind the check was supposed to assign. No acceptance criterion ties PIC-3/PIC-4 compliance to that observable, so today they are verifiable only by code inspection.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** c5cc01b ("pi-loom spec: resolve \"Runtime version / capability mismatch: no failure contract\"", 2026-05-06)
**History:** The capability-probe failure contract first landed in c5cc01b, and the prohibition arrived with it in prose form ("The probe MUST be limited to `typeof` checks — no arity sniffing, no return-shape sniffing — and MUST NOT use any member it is itself checking"). The 2026-05-06 rewrite (126ff8a) restructured that prose into the bulleted **Probe-wide invariants** block, and 6d8bea5 (2026-06-04) attached the `PIC-N` REQ-IDs. Across all three commits the invariants were carried as bare implementation prohibitions; no commit ever attached an observable runtime predicate or acceptance criterion to them. The defect is therefore an original framing choice, not a regression introduced by a later edit.

## Solution approach

Add an acceptance criterion to the **Probe-wide invariants** block (or the diagnostics-side test contract it cross-references) that ties PIC-3/PIC-4 to the missing-member observable: given a host missing exactly one probe member, the emitted `details.kind` is the specific kind covering that member's check — `"abortsignal-shape"` for an `AbortSignal`/`AbortController` member, `"sdk-capability-missing"` with `details.member` naming the member for a step-(c) member — and NOT `"probe-failed"`. State the criterion per check family, since the `prototype-property` members (`aborted`, `reason`) are `in`-checked and route their observed/required substrings differently from the `typeof`-checked members, and `details.member` refines only the `sdk-capability-missing` kind.
