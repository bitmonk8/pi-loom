## Problem

The `RuntimeEvent` payload TS block in `runtime-event-channel.md` declares `loom: string` with the inline type-comment example `"/code-review"`, carrying a leading `/`. The authoritative lexical form for this field is owned by `active-invocation-registry.md`, whose `loom`-field rule pins the value as the invocation's canonical-key name with **no leading `/`**, matching every other wire site that emits a slash-name onto `details.event.loom`. An implementer following the inline example populates `RuntimeEvent.loom` as `"/code-review"` while one following the registry contract populates it as `"code-review"`, so operator-facing dedup keyed on `details.event.loom` buckets the same invocation under two distinct loom identities.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `b3bc4ce` — spec: pin failure-observability surface (Cluster A, Option C) (2026-05-04, Thomas Andersen); `52b9101` — pi-loom spec: resolve "T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state" (2026-05-11, Thomas Andersen)
**History:** `b3bc4ce` introduced the `RuntimeEvent` type with `loom: string // slash name of the loom that owned the failure ("/code-review")` in the then-monolithic `pi-integration-contract.md`. At that commit no normative authority on the lexical form of the wire `loom` existed, so the leading-`/` example was latent rather than contradictory. `52b9101` then added the `loom` field to the `ActiveInvocationRegistry` entry shape and pinned its canonical-key form with the explicit "**no leading `/`**" rule, but did not retro-fit the `RuntimeEvent` type-comment example to match — at which point the contradiction became normative. The later `f5e89f4` (2026-06-04) split routed the type block verbatim into `runtime-event-channel.md` and the registry rule verbatim into `active-invocation-registry.md`, freezing the disagreement across two shards. The defect is the interaction: the example was inert at introduction and only became wrong when the sibling page tightened the lexical-form rule a week later.

## Solution approach

Rewrite the `RuntimeEvent.loom` type-comment example in the payload TS block so it matches the canonical-key form — strip the leading `/` so the example reads `"code-review"`. Optionally cross-reference `active-invocation-registry.md` as the owner of the lexical-form rule so a reader of the type block does not have to chase the sibling shard to confirm the wire form.

## Solution constraints

- The canonical lexical form for `loom` stays owned by `active-invocation-registry.md`; the type-comment may cross-reference that owner but must not become a second normative definition of the no-leading-`/` rule.
