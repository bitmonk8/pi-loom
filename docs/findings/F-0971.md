## Problem

The typed-query bullet in `conversation-drive.md` says the two-phase protocol transitions into the forced respond turn "once the model emits a plain text turn," but the shard never defines "plain text turn." Two readings are admissible: a text-only assistant turn (no `tool_use` blocks), or any assistant turn that contains text even when it also emits a `tool_use` block. The second reading breaks the two-phase protocol — a still-tooling turn would either be short-circuited into the forced respond branch (dropping pending tool calls) or serviced and forced-responded both (double-resolving the query). The companion bullet in `query-tool-loop.md` step 2 disambiguates the same rule parenthetically, but the `conversation-drive.md` bullet reads as load-bearing prose on its own without that disambiguation.

## Issue introduction

**Verdict:** present-since-inception (with respect to the two-phase tool-loop architecture)
**Introducing commits:** `82be125` (2026-05-04, "spec: adopt P1 typed-query two-phase loop (Option A)")
**History:** The phrase "once the model emits a plain text turn" entered the spec corpus in `82be125`, which first introduced the two-phase tool-loop conversation in the then-monolithic `spec_topics/pi-integration-contract.md`. `git log -S "emits a plain text turn"` returns `82be125` as the earliest commit on any tracked spec path; the term has been carried verbatim across every subsequent edit. The 100KB-cap split (`f5e89f4`, 2026-06-04) moved the bullet into `pi-integration-contract/conversation-drive.md` without touching the wording. The companion paragraph in `query/query-tool-loop.md` step 2 was added concurrently and *did* carry the disambiguating "(provider stop reason `end_turn` / `stop`)" parenthetical; the conversation-drive bullet has never carried it. The defect is a single missing disambiguation that has been in place since the two-phase architecture was adopted, not a regression introduced by a later edit.

## Solution approach

Clarify the "plain text turn" predicate in the typed-query bullet under `conversation-drive.md`'s **Conversation drive — prompt mode.** section so it names the text-only / no-`tool_use`-blocks reading (provider stop reason `end_turn` / `stop`), matching the disambiguation `query-tool-loop.md` step 2 already carries. The finding's defect is the wording itself, so the disambiguating predicate is the value to land; leave its exact phrasing and any cross-reference to the stop-reason classification owner to the implementer.

## Solution constraints

- `query-tool-loop.md` step 2's parenthetical remains the normative source; do not delete or relocate it — the `conversation-drive.md` bullet stays a paraphrase that agrees on the predicate.
- The clarified predicate must not exclude an empty assistant turn (no text, no `tool_use`), which is a valid free-phase terminator at the `max_rounds: 0` boundary in `query-tool-loop.md` step 2.
