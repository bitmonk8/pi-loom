## Finding

`H5a`'s Adds says its warn-only live-corpus mode "runs the same **four** live-corpus reconciliation surfaces" and enumerates them as four arms: the live spec REQ-ID set, the live numbered-REQ-ID citing-test surface, the live `spec_topics/**` normative-MUST surface (`H5e`), and the live multi-leaf-row per-facet citing-test surface (`H5f`). `H5b`'s Adds, describing the identical machinery, says it "reconciles the same **three** live surfaces `H6a` hard-fails on — the live spec REQ-ID set, the live `spec_topics/**` normative-MUST set, and the live test corpus." The two leaves count the same gate differently because they are counting different things: `H5b` counts live **corpora** (3 — the test corpus feeds two distinct arms), while `H5a` counts reconciliation **arms / gap kinds** (4). Neither leaf states the relationship, so the bare "four" / "three" prose reads as a contradiction about the same surface.

The conflation is not confined to these two leaves. `H6a`'s Adds enumerates all four failure modes (unmapped executable REQ-ID; coverage-matrix-mapped numbered REQ-ID with no citing test; per-facet facet with no facet-naming citing test; un-enumerated un-anchored MUST) and then refers to "those **three** failure modes," carrying the same corpora-vs-arms slip. `H5b`'s own gap-kind parenthetical and Tests bullet, and `H6a`'s Tests bullets, all correctly enumerate four gap kinds — only the summary noun counts ("three"/"four") drift.

The defect is a terminology inconsistency, not a coverage hole: a reader who reads `H5b`'s full Adds (with its four-gap-kind parenthetical) and Tests still sees all four gap kinds. But the mismatched summary counts force a reader to reconcile "four surfaces" (`H5a`) against "three surfaces" (`H5b`/`H6a`) when wiring `H5b` against `H5a`, and the bare-noun count is a foreseeable place for a skimming reader to undercount the arms.

## Solution Space

**Shape:** single

### Recommendation

Make the count terminology internally consistent across `H5a`, `H5b`, and `H6a` by distinguishing the number of live **corpora reconciled** (three: the live spec REQ-ID set, the live `spec_topics/**` normative-MUST set, and the live test corpus) from the number of **reconciliation arms / gap kinds** (four: unmapped executable REQ-ID; coverage-matrix-mapped numbered REQ-ID with no citing test; per-facet facet with no facet-naming citing test; un-enumerated un-anchored MUST), noting that the live test corpus feeds two of the four arms (the numbered-REQ-ID citing-test arm and the per-facet citing-test arm).

Concretely:

- In `docs/plan_topics/H5a-closing-gate-automation.md` Adds, the "runs the same four live-corpus reconciliation surfaces" sentence: keep the four-arm enumeration but state that the four arms reconcile against three live corpora, so the count matches the sibling leaves' "three corpora" framing.
- In `docs/plan_topics/H5b-warn-only-canary.md` Adds, the "reconciles the same three live surfaces `H6a` hard-fails on" sentence: retain the three-corpora wording but state that the three corpora are reconciled by four arms yielding four gap kinds, matching this leaf's own four-gap-kind parenthetical and Tests bullet.
- In `docs/plan_topics/H6a-live-corpus-activation.md` Adds, the "until this leaf lands those three failure modes are exercised only against the seeded fixtures at `H5a`" sentence: change "three failure modes" to "four failure modes" to match the four failure modes the same sentence-group already enumerates.

The spec is read-only for this fix; the entire change is internal to the three plan leaf files.

## Consequence

**Severity:** advisory

An implementer wiring `H5b` against `H5a` must stop to reconcile "four reconciliation surfaces" against "three live surfaces" before concluding they describe the same gate run; the same slip recurs in `H6a`'s "those three failure modes." Because each leaf's parenthetical and Tests already enumerate all four gap kinds, no coverage is lost — the cost is reader friction and the risk that a reader trusting the bare "three" undercounts the per-facet (`H5f`) arm.
