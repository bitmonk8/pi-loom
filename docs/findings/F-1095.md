## Problem

The noun "seam" carries two categorically different meanings across the plan. In its dominant sense it names a constructor-injected host DI interface — the PIC-10…PIC-16 seams modelled by `V8a`–`V8e` and scaffolded by the `H3a` DI-seam skeleton. In a second sense it names a stateless function imported and consulted directly at call sites, never injected — `V16a`'s `arbitrate(candidate)` cross-ceiling arbitration "seam," which `conventions.md` §Leaf format classifies as a Class-2 named cross-leaf seam. `V16a`'s `Adds` disambiguates the two senses, but that note lives only in `V16a`; the consumer leaves that consult it (`V4e`, `V5e`, `V11f`, `V13c`, `V15b`, and `H7a`) cite "`V16a`'s cross-ceiling arbitration seam" without the "never injected" caveat, and `conventions.md` defines the Class-2 binding category without flagging the overload. No durable plan file reconciles the two senses.

**History:** `V16a` was created on 2026-06-10 (c6a664e); commit e2b7e81 the same day named the cross-ceiling arbitration unit a "seam," reusing the noun the DI host interfaces already carried and thereby introducing the overload. A later commit (6826ca0, 2026-06-17) added `V16a`'s inline DI-disambiguation in response to a prior "reuses spec's reserved seam noun" finding, but mitigated it only locally — the central-terminology gap and the unqualified consumer-leaf references remain.

## Solution approach

Add a durable terminology entry in `conventions.md` distinguishing the two senses of "seam": the constructor-injected host DI interface sense (PIC-10…PIC-16, `V8a`–`V8e`, `H3a`) and the Class-2 named cross-leaf seam sense (a stateless function consulted directly, never injected — `V16a`'s arbitration seam), and note that the No-globals/statics/singletons rule does not require the latter to be injected. Cross-reference that entry from `V16a`'s existing inline disambiguation note.
