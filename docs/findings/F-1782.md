## Finding

The binder transport-failure system note is pinned as `loom /<name>: argument binder unavailable (<provider>: <message>)` in the *Failure-mode templates (normative)* section of `determinism-cancellation-failure.md`. The section's classification sentence groups the transport row's `<provider>` and `<message>` together as "runtime- or classifier-/validator-derived content," then pins only two of the deterministic placeholders: it states "(`<provider>` and `<ajv-summary>` are pinned below)" and supplies a full field-and-rendering rule for each. `<message>` is named in the deterministic group but is given no derivation rule â€” neither a source field nor a rendering rule.

Two gaps follow. First, the placeholderâ†’field mapping is left to analogy: a renderer must infer that `<message>` interpolates `TransportError.message` from the parenthetical's parallel with `<provider>`. Second, and more consequential, the *source* of `TransportError.message` for the binder is unpinned. The only `message` derivation rule in the corpus lives in `queryerror-variants.md` and is explicitly scoped to **prompt mode** ("a transport failure detected after `waitForIdle()` sources `message` from the `errorMessage` field of the driven turn's trailing `assistant` message â€¦ falling back to the fixed string `provider transport failure` when that field is absent"). The binder does not run through `waitForIdle()`; it is an off-session `complete()` call (the same paragraph that pins `<provider>` confirms the binder pass resolves off-session through `complete()`). No rule states where `TransportError.message` comes from on that path.

By contrast the sibling placeholders are fully nailed down: `<provider>` is pinned to the classifier-produced `TransportError.provider` (`Model<Api>.api`, rendered verbatim) and `<ajv-summary>` has its own multi-clause rendering rule including the depth-walk fast-fail case. `<message>` is the lone deterministic placeholder in the row left without a source pin. A conformance test that injects a provider transport failure and asserts the exact rendered note cannot be written against the spec, because the spec does not say which value lands inside the parenthetical.

## Solution Space

**Shape:** single

### Recommendation

Pin `<message>` in the *Failure-mode templates (normative)* section in the same form as the adjacent `<provider>` pin, and supply the binder-path source derivation that the prompt-mode note does not cover:

- State that `<message>` in the transport-failure row is the `message` field of the classifier-produced `TransportError`, rendered verbatim and subject to System-note rendering rule 1 whitespace normalisation (the same rendering discipline the whole template already inherits).
- Pin the binder-path derivation of `TransportError.message`. The binder classifies off-session via `complete()`, so the prompt-mode `waitForIdle()` rule in `queryerror-variants.md` does not apply. Derive it from the same classifier input surface the binder already reads â€” the resolved `AssistantMessage.errorMessage` captured on the failing `complete()` (per *Classifier input surface* in `provider-error-mapping.md`) â€” falling back to the fixed string `provider transport failure` when that field is absent, mirroring the prompt-mode fallback so the two paths converge. Anchor this either inline in the binder section or by extending the existing `message` derivation note in `queryerror-variants.md` to cover the off-session/binder path alongside prompt mode.

Edge cases the implementer must watch: the network-level failure case where `onResponse` never fired and `complete()` rejected with no `AssistantMessage` (the `retryable: true, http_status: null` route) carries no `errorMessage`, so the fixed fallback string governs; and the rendered note still passes through the shared System-note rendering discipline, so whitespace normalisation applies to the final interpolated string, not to the raw `errorMessage`.

## Consequence

**Severity:** correctness

The interpolated `<message>` content is unspecified for the binder's off-session `complete()` path, so two conforming implementations can surface different strings for the same provider transport failure (e.g. the pi-ai `AssistantMessage.errorMessage`, the rejected-promise `.message`, or a fixed fallback). The rendered note is user-facing and its wording is declared a spec-versioned breaking change, so the divergence is observable and contractually significant, and no end-to-end conformance test can assert the exact rendered note.
