## Finding

`binder-inference.md` (*Binder inference call*, the paragraph beginning "The structured-output result is extracted…") specifies the extraction-vs-failure routing for the binder's one-shot `complete()` result entirely in dense prose. The rule partitions the returned `AssistantMessage` across several distinguishable input combinations: a matching `ToolCall` present (extract regardless of `stopReason`, `errorMessage`, or HTTP status); a matching `ToolCall` whose `arguments` fail JSON parse or schema validation (malformed-envelope class); no matching `ToolCall` — plain text or a `ToolCall` with a different `name` (also malformed-envelope); and a non-`"stop"`/`"tool_use"` `stopReason`, non-empty `errorMessage`, rejected promise, or 4xx/5xx/network failure routed through the provider-error mapping table.

The load-bearing and least-obvious case is the precedence rule: "Success extraction takes precedence over `stopReason`-based failure classification … a degenerate response carrying both a matching `ToolCall` and a non-empty `errorMessage` also extracts as success." This inverts the intuitive reading (a present `errorMessage` would normally route to the provider-error path) and is the exact point at which two test suites would diverge if the author worked from intuition rather than the prose.

No acceptance criteria anywhere pin these combinations as executable test material. The owning implementation/test leaves (`V9j`/`V9j-T`) assert only the provider→`QueryError` classifier table, the `complete()` envelope shape, and the unsupported-provider warning; the failure-taxonomy leaves (`V11f`/`V11f-T`) assert the retry caps, the six verbatim failure-mode templates, and the depth-walk fast-fail rendering. Neither enumerates the extraction-routing decision itself — most critically the `ToolCall`+`errorMessage`→success path and the parse-fail→malformed-envelope (vs. provider-error) split. The behaviour is fully determined by the prose, but it is not pinned as test material, so a conforming test suite can silently omit the non-obvious arms.

## Solution Space

**Shape:** single

### Recommendation

Add a compact extraction-routing decision table to `binder-inference.md`, placed immediately after the "The structured-output result is extracted…" paragraph, enumerating each input combination and its outcome class. At minimum, one row per:

1. matching `ToolCall` + `stopReason: "tool_use"` → extraction succeeds.
2. matching `ToolCall` + `stopReason: "length"` (or any non-`"stop"`/`"tool_use"` stop) → extraction succeeds (precedence over stop-reason classification).
3. matching `ToolCall` + non-empty `errorMessage` + `stopReason: "stop"` → extraction succeeds (the degenerate-success path).
4. matching `ToolCall` whose `arguments` fail JSON parse or schema validation → malformed-envelope class (not provider-error route).
5. no matching `ToolCall` (plain text, or a `ToolCall` with a different `name`) → malformed-envelope class.
6. no matching `ToolCall` + non-`"stop"`/`"tool_use"` `stopReason` / non-empty `errorMessage` / rejected promise / 4xx-5xx-network → provider-error mapping route.
7. multiple `ToolCall` entries, first matching wins, rest ignored.

Then extend the test leaves to cite the table: `V9j-T` asserts the extraction arms (rows 1–3, 7) and the parse-fail split (row 4); `V11f-T` asserts the no-`ToolCall`/provider-error classification arms (rows 5–6) it already partly covers. The table is a faithful enumeration of behaviour the prose already fixes — no new contract is introduced. Watch the row-4 vs row-6 boundary: a parse/schema failure on a present matching `ToolCall` is malformed-envelope, whereas the same surface error with no matching `ToolCall` is still malformed-envelope but a transport-level `errorMessage`/status with no `ToolCall` is provider-error; the table must keep these distinct.

## Consequence

**Severity:** advisory

The routing behaviour is unambiguous in prose, so an implementer can still build a correct extractor; the gap is in verification, not specification. Without enumerated boundary criteria the non-obvious arms — chiefly `ToolCall`+`errorMessage`→success and parse-fail→malformed-envelope — are easy to leave untested, so a regression that re-routes the degenerate success case to the provider-error path would ship green.
