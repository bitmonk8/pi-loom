## Problem

The `#binder-model-hot-reload` recovery-note paragraph (the BNDR-11 neighbour in `binder/binder-model-and-context.md`) defines `recovery.looms` membership as a conjunction: (a) the loom's original load failure was `loom/load/binder-model-unresolved`, AND (b) its binder-model two-step chain now re-resolves under the changed setting. Conjunct (b) is computable on demand — the recovery pass re-runs binder-model resolution alone — but conjunct (a) requires the runtime to know each previously-discovered loom's original `loom/load/*` failure code, a per-loom load-failure classification state the spec corpus never introduces. `LoomRegistry` (the `Map<slashName, parsedLoom>` in `registration-steps.md` step 5 / `drain-state-contract.md#loom-registry-drain-state-contract`) holds only successfully-loaded looms, and BNDR-11 pins only the successfully-loaded path, leaving the previously-failed-loom set undefined. Two implementers therefore diverge: one filters by original failure cause, another over-emits every now-resolving loom regardless of original cause, and both pass the current single-path recovery fixture.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commits:** `97430aa` (2026-06-06) — *pi-loom spec: resolve "Hot-reload recovery note over-promises /reload success"*
**History:** The paragraph was first authored in `f88698e` (2026-06-05) using membership prose "the looms whose re-resolution under the changed setting would now succeed" — a single conjunct (b) condition that the recovery pass's "re-run binder-model resolution alone" computes directly, with no need for any persisted per-loom failure classification. `97430aa` tightened the prose to "exactly the looms whose original load failure was `loom/load/binder-model-unresolved` and whose binder model now re-resolves to a model under the changed setting" while keeping the "by re-running binder-model resolution alone with no other load-pass step re-run" computation clause unchanged. That edit silently introduced conjunct (a) — a precondition on each loom's *prior* load-failure classification — without specifying where that classification is persisted or how the membership computation accesses it. The subsequent `BNDR-11` coining in `1b6d2f1` (2026-06-24) pinned the no-re-resolve-on-already-loaded-looms rule above the paragraph but did not touch the recovery-membership clause and so neither introduced nor cured the defect.

## Solution approach

Specify a persisted per-loom load-failure classification, keyed by the final derived slash name (per DISC-4), recording each loom's original `loom/load/*` failure code — sibling to the `LoomRegistry` `Map<slashName, parsedLoom>` named in `registration-steps.md`'s `#watcher-hot-reload-registration` step 5 and its `drain-state-contract.md#loom-registry-drain-state-contract` field list. Rewrite the `#binder-model-hot-reload` recovery-note membership clause so `recovery.looms` computes conjunct (a) from that classification rather than from an unspecified source, retaining the on-demand binder-model re-resolution for conjunct (b). Add a forward-link from that paragraph to the named state, and a pointer to it from the `recovery` row in `runtime-event-channel.md`. Pin the classification's lifetime so an entry clears on a successful `/reload` of its loom and drops when the loom unloads.

## Solution constraints

- Preserve BNDR-11: a `looms.binderModel`-only settings edit MUST NOT re-run binder-model resolution for any already-loaded loom — the new classification state must not introduce such a re-resolution.
