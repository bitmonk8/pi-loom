## Finding

The closing `## Template Interpolation` section of `frontmatter/frontmatter-fields-b-and-templates.md` carries two normative lexical rules about `@`...`` **query templates**: (1) the `@` character has a single lexical role — introducing a query template at top level — and never appears inside `${...}`; and (2) there is no bash-style argument-slice sugar (`${@:N}`, `$1`, `$@`, `$ARGUMENTS`). These are query-layer lexical constraints, not frontmatter behavior. The frontmatter page owns field contracts (`tools:`, `system:`, `respond_repair`, `tool_loop`); the `${...}`/`@` lexical surface is owned by the Query topic.

The Query subtree — `query/query-forms.md` and `query/query-escapes-stringification.md` (which carries QRY-17, the authoritative `@`-template body escape/lexical rules, and QRY-18, the stringification table) — states neither rule and carries no forward link to the frontmatter page. A lexer author building the `@`-template tokenizer from the Query topic alone has no statement that `@` is inert inside `${...}` and no prohibition on bash-style sugar; nothing in their cited pages forbids treating `@` as significant inside the slot or admitting `$ARGUMENTS`-style forms.

The content is currently discoverable only because the `system:` *Escapes* paragraph in the same frontmatter file forward-links to `#template-interpolation`. The fix is a relocation, not a rewrite: move the two lexical rules adjacent to QRY-17 in `query/query-escapes-stringification.md`, leave a forward-link (and a resolvable `#template-interpolation` anchor) on the frontmatter page for the `system:` Escapes reference.

## Solution Space

**Shape:** single

### Recommendation

Move the two lexical rules — (a) `@` has only one lexical role and never appears inside `${...}`, and (b) there is no bash-style argument-slice sugar (`${@:N}`, `$1`, `$@`, `$ARGUMENTS`) — into `query/query-escapes-stringification.md` adjacent to QRY-17, which already owns the `@`-template body escape and lexical rules. Replace the relocated content on the frontmatter page with a one-line forward-link to the Query topic, and keep the `#template-interpolation` heading anchor in `frontmatter-fields-b-and-templates.md` resolvable, because the `system:` *Escapes* paragraph links to it. The sentence noting that slash-command arguments are bound to typed `params` (rather than positional bash forms) travels with the prohibition. The cross-reference to the canonical stringification table (QRY-18) is already on the Query page, so no duplication is created by the move.

## Consequence

**Severity:** advisory

An implementer building the `@`-query-template lexer from the Query topic alone has no statement that `@` is inert inside `${...}` and no prohibition on bash-style argument-slice sugar, so a lexer author could admit `$ARGUMENTS`/`${@:N}` forms or treat `@` as significant inside the slot. The rule is still discoverable via the frontmatter page's forward-link, so a working system remains reachable — this is a guidance gap, not a hard divergence.
