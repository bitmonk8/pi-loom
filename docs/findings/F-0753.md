## Finding

The "Schema-validation respond-repair" section of `docs/spec_topics/query/query-failure-and-repair.md` makes two normative statements about `respond_repair.attempts` accounting that disagree on the same wire-observable outcome.

The general paragraph (third paragraph of the section) says:

> Each follow-up counts as one against `respond_repair.attempts` regardless of how many tool-call rounds it contains.

The "Non-validation failures during a respond-repair follow-up" sub-section then says the opposite for the non-validation arm:

> A follow-up that fails for a non-validation reason **does not consume an `attempts` slot**. `attempts` counts only follow-ups that produced an assistant response which was then re-validated …

The general sentence is unqualified — "each follow-up … regardless of …" — so a reader who stops there concludes every follow-up debits the budget. A reader who continues to the sub-section concludes only re-validated follow-ups debit it. On the same concrete trace — `respond_repair.attempts: 3`, validation failure on the original response, transport failure on the first follow-up, then transport recovers — the first reading reports `ValidationError.attempts: 1` already burnt (two follow-ups remain), the second reports `ValidationError.attempts: 0` (three follow-ups remain). The two readings also disagree on whether the *next* validation failure can still trigger respond-repair at all when only one nominal slot is left.

The sub-section is the rule the rest of the spec relies on (V13d's `Tests` cite "non-validation failure propagates the proximate variant and consumes no attempt"; `ERR-17` accounting also assumes the re-validated-only counting), so the general sentence is the defect.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 53f5831 — spec: rewrite query primitive, system prompts, and schema syntax (2026-05-03, Thomas Andersen); 06491ee — pi-loom spec: resolve "Coercion follow-up failure modes unspecified" (2026-05-04, Thomas Andersen)
**History:** `53f5831` introduced the unqualified general sentence ("Each follow-up counts as one against `retry.attempts` regardless of how many tool-call iterations it contains.") at a time when the only rule for non-validation outcomes was the trailing "Non-validation failures (`transport`, `tool_failure`, etc.) are not retried by the query primitive" — so the unqualified phrasing was consistent with the rest of the section. The next day, `06491ee` added the new "Non-validation failures during a coercion follow-up" sub-section introducing the "does not consume an `attempts` slot" exception, but left the earlier general sentence as-is; the two statements have contradicted each other ever since. Subsequent renames (`retry` → `respond_repair`, `coercion` → `respond-repair`) and the size-cap split into `docs/spec_topics/query/query-failure-and-repair.md` (`f5e89f4`) carried both passages forward verbatim without reconciling them.

## Solution Space

**Shape:** single

### Recommendation

Qualify the general sentence in the "Schema-validation respond-repair" section so it counts only re-validated follow-ups, and point the reader at the sub-section that owns the full rule. Concretely, replace

> Each follow-up counts as one against `respond_repair.attempts` regardless of how many tool-call rounds it contains.

with something equivalent to

> Each follow-up that produces an assistant response the runtime then re-validates counts as one against `respond_repair.attempts`, regardless of how many tool-call rounds it contains; follow-ups that fail for a non-validation reason are governed by [Non-validation failures during a respond-repair follow-up](#non-validation-failures-during-a-respond-repair-follow-up) and do not consume a slot.

The qualifier ("that produces an assistant response the runtime then re-validates") is the load-bearing change; the cross-reference is what keeps the two passages from drifting again. Keep the "regardless of how many tool-call rounds it contains" clause — that is the orthogonal in-loop accounting rule and is not in dispute.

Implementer edge cases the qualifier must continue to cover:

- The forced-respond-non-compliance path (ERR-17) still consumes one slot per attempt, because the runtime synthesises a `ValidationIssue` and feeds it into the re-validation pipeline — that path *does* produce an assistant response that is then validated (and rejected).
- An empty-template short-circuit observed on a follow-up does not consume a slot (already stated as "defensive" in the sub-section); the qualifier above preserves that because no provider round-trip happened.
- `respond_repair.methodology: none` / `attempts: 0` remains a no-op; the qualifier is vacuously satisfied because no follow-up is ever issued.

## Consequence

**Severity:** correctness

Two reasonable implementers reading the spec produce divergent wire behaviour for the same trace. The visible divergence is `ValidationError.attempts` on terminal exhaustion when any non-validation failure occurred mid-repair (one implementation reports the configured budget, the other reports a smaller number), and the count of follow-ups actually issued after an interleaved transport blip. Compliance tests written against one reading reject the other.
