## Problem

In the `## Schema-validation respond-repair` section of `docs/spec_topics/query/query-failure-and-repair.md`, two normative statements about `respond_repair.attempts` accounting contradict each other for the same wire-observable outcome. The third paragraph states unqualified that "each follow-up counts as one against `respond_repair.attempts` regardless of how many tool-call rounds it contains," while the `### Non-validation failures during a respond-repair follow-up` sub-section states that a follow-up failing for a non-validation reason does not consume an `attempts` slot. On a trace where a non-validation failure occurs mid-repair, the two readings produce divergent `ValidationError.attempts` values and a divergent count of follow-ups issued. The sub-section is the rule the rest of the spec relies on (V13d's tests and ERR-17 accounting assume re-validated-only counting), so the general sentence is the defect.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 53f5831 тАФ spec: rewrite query primitive, system prompts, and schema syntax (2026-05-03, Thomas Andersen); 06491ee тАФ pi-loom spec: resolve "Coercion follow-up failure modes unspecified" (2026-05-04, Thomas Andersen)
**History:** `53f5831` introduced the unqualified general sentence ("Each follow-up counts as one against `retry.attempts` regardless of how many tool-call iterations it contains.") at a time when the only rule for non-validation outcomes was the trailing "Non-validation failures (`transport`, `tool_failure`, etc.) are not retried by the query primitive" тАФ so the unqualified phrasing was consistent with the rest of the section. The next day, `06491ee` added the new "Non-validation failures during a coercion follow-up" sub-section introducing the "does not consume an `attempts` slot" exception, but left the earlier general sentence as-is; the two statements have contradicted each other ever since. Subsequent renames (`retry` тЖТ `respond_repair`, `coercion` тЖТ `respond-repair`) and the size-cap split into `docs/spec_topics/query/query-failure-and-repair.md` (`f5e89f4`) carried both passages forward verbatim without reconciling them.

## Solution approach

Rewrite the third-paragraph sentence in the `## Schema-validation respond-repair` section so the `respond_repair.attempts` debit applies only to follow-ups that produce an assistant response the runtime then re-validates, bringing it into agreement with the `### Non-validation failures during a respond-repair follow-up` sub-section. Add a forward-link from that sentence to `#non-validation-failures-during-a-respond-repair-follow-up` so the two passages stay reconciled.

## Solution constraints

- The in-loop accounting rule that each follow-up counts once against `respond_repair.attempts` regardless of how many tool-call rounds it contains MUST be preserved тАФ that clause is orthogonal to the re-validated-only counting and is not in dispute.
- The forced-respond-non-compliance path (ERR-17) MUST continue to consume one slot, since the runtime synthesises a `ValidationIssue` and feeds it into the re-validation pipeline (an assistant response that is then validated and rejected).
