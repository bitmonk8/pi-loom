## Finding

The `context.tools` bullet in `binder-inference.md` specifies the binder's structured-output tool by deferring to a section it names twice in bold тАФ "the **Per-loom registration** section above" (for the lowered-JSON-Schema тЖТ TypeBox bridge) and "the **Per-loom registration** field enumeration above" (for the required-field set). Neither mention is a hyperlink, and "Per-loom registration" is not a section on this page: it lives on the sibling page `extension-bootstrap-and-per-loom.md`, where its heading carries no `<a id>` anchor. The word "above" reflects cluster reading order (the PIC hub orders that page before `binder-inference.md`), but an implementer reading the binder page in isolation has no resolvable target for either reference.

The bullet then asserts the binder tool has "three remaining required `ToolDefinition` fields" тАФ `name`, `label`, `description` тАФ beyond `parameters`. That count is only verifiable against the off-page enumeration, which states the full required set is five fields: `name`, `label`, `description`, `parameters`, `execute`. After `parameters`, four fields remain, not three: the bullet silently omits `execute`. The binder tool is never executed (the runtime extracts the envelope directly from the returned `ToolCall.arguments`, per the same page's extraction paragraph), so `execute` has no behavioural role here тАФ but it is a required field of the materialised tool, and the spec never says what the binder tool supplies for it. The "three remaining" framing therefore both undercounts the required set and leaves the `execute` field of the binder tool unpinned.

A secondary type-attribution wrinkle compounds this: the binder call passes `context.tools` to pi-ai's `complete()`, whose entries are pi-ai `Tool` objects (declared at `dist/types.d.ts`), whereas the "Per-loom registration" enumeration the bullet borrows from describes Pi's `ToolDefinition` (the `@earendil-works/pi-coding-agent` type used for registered loom callables and subagent `customTools`). The bullet calls the binder tool's fields "`ToolDefinition` fields" while routing it through the `Tool` surface, so the correct required-field set for the binder tool is not even unambiguously the five-field `ToolDefinition` shape.

## Issue introduction

**Verdict:** single-commit

**Introducing commit:** `61592d7` (2026-06-05, "pi-loom spec: resolve 'Binder structured-output tool name and label undefined'")

**History:** Before `61592d7` the `context.tools` bullet pinned only `parameters` (wrapped via `Type.Unsafe<unknown>(envelopeSchema)`) and named no further fields. The commit, resolving a prior review finding that the binder tool's `name`/`label` were undefined, added the "three remaining required `ToolDefinition` fields" clause and the literal pins for `name` (`__loom_bind_<slug>`), `label` (`"Loom binder envelope"`), and `description`. The fix reached for the wrong type: the synthesised typed-query respond tool whose pin it mirrors is a `ToolDefinition` (registered via `pi.registerTool`), but `Context.tools` is `Tool[]` тАФ the opening paragraph of the same file had already pinned that typing two commits earlier under `f5e89f4` (2026-06-04, the spec-split commit that created the file). A follow-up commit `e63ba6c` (2026-06-05) edited the same bullet to add the `complete()` forced-tool presupposition cross-reference but did not touch the type framing. The defect has lived in its current form since `61592d7` and has not been re-edited since.

## Solution Space

**Shape:** single

### Recommendation

Make the binder tool's field set resolvable and complete on the binder page:

1. Add an `<a id="per-loom-registration">` anchor to the **Per-loom registration** heading in `extension-bootstrap-and-per-loom.md`.
2. In the `context.tools` bullet of `binder-inference.md`, convert both bold "Per-loom registration" mentions to hyperlinks to that anchor (`./extension-bootstrap-and-per-loom.md#per-loom-registration`), matching the hyperlinked cross-reference style the same bullet already uses for the envelope schema and the canonical-schema-hash recipe.
3. State the binder tool's full required-field set on the bullet rather than by an off-page count. Enumerate all required fields and pin the one currently omitted (`execute`): because the runtime extracts the envelope directly from the returned `ToolCall.arguments` and never dispatches the binder tool, fix `execute` to a defined value (e.g. an adapter that is never invoked / resolves trivially) so the materialised tool is complete and two implementers cannot diverge.
4. Reconcile the type attribution: the binder tool flows through pi-ai's `complete()` `context.tools` (pi-ai `Tool`), not Pi's `ToolDefinition` registration path. Either name the correct pi-ai `Tool` shape and its required fields, or state explicitly that the binder tool reuses the same five-field shape as the registered `ToolDefinition` and why that shape is what `complete()` consumes.

Edge case for the implementer: the binder tool's `name` is `__loom_bind_<slug>` from the canonical-schema-hash recipe тАФ already pinned in the bullet тАФ so only `execute` and the type-shape attribution are genuinely unspecified after the link is added.

## Consequence

**Severity:** advisory

An implementer building the binder `complete()` call cannot, from the binder page alone, confirm the complete required-field set of the tool it constructs, and the "three remaining" count omits `execute` тАФ a required field the spec never pins for the binder tool. The references are recoverable by hand-searching the PIC cluster, and a compiler would force `execute` to be supplied, so a working system is still reachable; the cost is navigability and the risk of two implementers diverging on what `execute` does (no-op vs. throw) for a tool that is never invoked.
