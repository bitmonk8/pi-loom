## Problem

`conventions.md` §Loom 1.0 release-time residue inspection enumerates a nine-item manual review that is the single named backstop for the residue classes the mechanical scans concede, and asserts it runs at the loom 1.0 release gate and that a defect found at any item blocks the release until fixed. No plan leaf is assigned to perform it: `H6a` — the loom 1.0 release gate — records the manual real-host smoke run in its **Adds** and **Ships when** but never references the nine-item residue inspection, and nothing else in the corpus claims to own or run it. Because no owner step exists and no observable artifact records a per-item clean/disposition result, the entire manual safety net can be silently skipped at release and completion is unverifiable — unlike the sibling manual real-host smoke run in the same gate, which carries a falsifiable evidence record committed in the gate-activation commit message.

## Issue introduction

**Verdict:** present-since-inception (of the consolidated residue-inspection surface)
**Introducing commit:** `e94ca10` (2026-06-16) — "pi-loom plan: resolve \"GOV-15 release-time reviewer inspection has no enumerated residue checklist\""
**History:** `git log -S '## Loom 1.0 release-time residue inspection' -- docs/plan_topics/conventions.md` returns exactly one commit, `e94ca10`; its parent (`e94ca10~1`) contains no occurrence of the heading or the phrase "release-time residue inspection". That commit authored the section as a single named manual-review surface ("runs at the loom 1.0 release gate", "blocks the loom 1.0 release until fixed") but pinned no owner leaf to perform it and defined no falsifiable evidence record — the gap this finding names. The contrast is not incidental: the sibling manual real-host smoke evidence record in `H6a` was already in place by 2026-06-11 (`eca63cf` "Manual real-host fidelity gate leaves no falsifiable record", `4099d0a` "Release-gate evidence artifact has no defined committed home or format"), so the residue-inspection surface was created five days later without the owner-pin/evidence-record treatment its sibling already carried. `git log -G 'residue' -- docs/plan_topics/H6a-live-corpus-activation.md` shows no commit ever added a residue-inspection reference to H6a.

## Solution approach

Pin `H6a` as the owner step for the nine-item residue inspection. Add the residue inspection to `H6a`'s **Ships when** as a release-gate precondition that records each item's clean-or-dispositioned result through the same gate-activation commit-message evidence channel `H6a` already uses for the manual real-host smoke run, so a skipped or mis-recorded inspection is detectable after the fact. Name the residue inspection among the release-gate confirmations in `H6a`'s **Adds**, alongside the existing manual-smoke-run clause. Optionally add a pointer in `docs/plan_topics/conventions.md` §Loom 1.0 release-time residue inspection naming `H6a` as the owner step.

## Solution constraints

- Out of scope: mechanising or re-anchoring any individual residue class (items 1–9) — each remains owned by the cross-cutting rule that concedes it; this fix adds the owner step and recorded outcome only.
