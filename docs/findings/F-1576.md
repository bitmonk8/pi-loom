## Finding

The `AgentSession` member-surface block in `host-interfaces-core.md` snapshots the driver method as:

```
sendUserMessage(text: string): Promise<void>;          // subagent-mode query driver; returns `Promise<void>` — unlike `pi.sendUserMessage`'s `void` shape
```

The Pi-owned shape at the pinned SDK (`dist/core/agent-session.d.ts:377`) is wider:

```
sendUserMessage(content: string | (TextContent | ImageContent)[], options?: { deliverAs?: "steer" | "followUp" }): Promise<void>;
```

The return type (`Promise<void>`) matches, but the snapshot narrows the first parameter from `string | (TextContent | ImageContent)[]` to bare `string` and drops the optional `options?: { deliverAs?: "steer" | "followUp" }` argument entirely. The block's own preamble declares each member's TypeScript signature a "non-normative snapshot of the Pi-owned shape at the loom 1.0 Pi-SDK pin, re-validated on each Pi minor bump." An inaccurate baseline defeats that re-validation: the version-bump procedure diffs the live SDK against this snapshot, so a snapshot that was never faithful produces a spurious "signature changed" diff on the next bump and obscures any real upstream change to the parameter shape or `deliverAs` modes.

The narrowing does not break the loom runtime's own call site (a bare `string` is a valid argument under the real union), which is why this is a snapshot-fidelity defect rather than a behavioural one.

## Issue introduction

**Verdict:** present-since-inception (of the snapshot block)
**Introducing commit:** `aa334c2` — "pi-loom spec: resolve "AgentSession consumed member surface not pinned"" (2026-06-05)
**History:** `git log -S` / `git log -L` on the `sendUserMessage(text: string)` line in `host-interfaces-core.md` return a single commit, `aa334c2`, which added the `AgentSession` member-surface snapshot. The line was authored with the simplified single-arg signature. At that same commit the Pi SDK pin (`host-prerequisites.md#pi-sdk-pin`) already read `~0.75.5`, whose `agent-session.d.ts` already declared the multi-arg `content` + `options` form — so the snapshot was inaccurate at the moment it was written, not a victim of later upstream drift. No subsequent commit touched the line.

## Solution Space

**Shape:** single

### Recommendation

Replace the snapshot's parameter list with the full Pi-owned shape so the baseline matches `agent-session.d.ts`:

```
sendUserMessage(content: string | (TextContent | ImageContent)[], options?: { deliverAs?: "steer" | "followUp" }): Promise<void>;
```

Keep the trailing comment's substantive point (subagent-mode query driver; returns `Promise<void>` unlike `pi.sendUserMessage`'s `void` shape). `TextContent` and `ImageContent` are Pi-owned content types from the same SDK surface; reproduce them verbatim as written in the `.d.ts` rather than substituting loom-local names. The member-set normativity statement and the per-bump re-validation note already cover the snapshot's status and need no change.

## Consequence

**Severity:** advisory

The snapshot is non-normative and the runtime call site (string argument) remains valid, so nothing breaks at implementation time. The cost is to the bump-time re-validation workflow: an inaccurate baseline yields a false-positive signature diff on the next Pi minor and can mask a genuine upstream change to the parameter shape or `deliverAs` modes.
