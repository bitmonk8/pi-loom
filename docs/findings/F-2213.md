## Finding

The single active REQ-ID prefix table is split across two files, `docs/spec_topics/governance/req-id-prefix-table-active-a.md` and `req-id-prefix-table-active-b.md`. The split is a pure file-size artifact, but nothing on either page records that. The `-a` page opens with `## REQ-ID prefix table`, GOV-1, GOV-3, and the first half of the prefix table; the `-b` page carries the remaining prefix rows plus GOV-5 through GOV-31. Neither page carries a preamble, heading note, or comment stating that the two are halves of one table, what the split criterion is, or which portion each holds. The `-a`/`-b` suffixes are the only signal, and they convey nothing about content.

The defect is sharper on `-b` than the heading suggests. After its auto-titled H1, `-b` begins immediately with bare table rows (`| \`implementation-notes.md\` | \`IMPL\` |` …) but **without the `| Page | Prefix |` header row or the `|---|---|` separator** — those live only on `-a`. In isolation (a deep link, a grep hit, or a renderer that opens `-b` directly) the opening rows do not even render as a table; they appear as a stray run of pipe-delimited text with no column meaning and no indication they continue `-a`.

The fix is a low-risk, self-contained legibility edit on the two pages; renaming the files is the higher-risk alternative because the filenames are cited as cross-link and anchor targets from both the spec corpus and the plan.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** `f5e89f4` ("docs: enforce 100KB hard / 30KB recommended size cap on spec set")
**History:** Both files were created (`--diff-filter=A --follow`) in `f5e89f4`, the one-shot mechanical split that broke the over-limit governance page into the subdirectory layout via `docs/_tools/split_spec.py`. `git show f5e89f4:…/req-id-prefix-table-active-b.md` confirms `-b` was born headerless — H1 followed immediately by orphan table rows, with no `| Page | Prefix |` header and no continuation preamble. The defect is therefore not a regression from an earlier well-formed state; it is an artifact of the size-cap split that the mechanical driver did not repair, present unchanged since that commit.

## Solution Space

**Shape:** single

### Recommendation

Make the split legible in place, without renaming the files:

1. On `req-id-prefix-table-active-b.md`, immediately under the H1, add a one-line continuation preamble that links back to `-a`, e.g. `_Continuation of [REQ-ID prefix table](./req-id-prefix-table-active-a.md#req-id-prefix-table) — this page carries the remaining prefix rows and GOV-5…GOV-31._`
2. On `req-id-prefix-table-active-b.md`, re-emit the table header above the first row (`| Page | Prefix |` then `|---|---|`) so the opening prefix rows render as a table when the page is viewed in isolation.
3. On `req-id-prefix-table-active-a.md`, add a one-line note at the end of the `## REQ-ID prefix table` table indicating the remaining rows live on `-a`'s sibling, e.g. `_Table continues in [req-id-prefix-table-active-b.md](./req-id-prefix-table-active-b.md)._`

Keep both filenames as they are. They are anchor/cross-link targets from the spec corpus and from `docs/plan_topics/*` (GOV-22/GOV-3/GOV-6/GOV-24 citations); renaming would force a corpus-wide repoint for no behavioural gain. The HTML-comment-only variant is insufficient on its own — a reader arriving via a rendered deep link never sees a comment, and the missing table header would remain — so prefer the visible preamble plus restored header.

## Consequence

**Severity:** cosmetic

A reader or maintainer arriving at `req-id-prefix-table-active-b.md` directly cannot tell it is the continuation of `-a`, and its leading rows do not render as a table at all (no header/separator), making the prefix entries on `-b` hard to read in isolation. No runtime, parse, or load behaviour is affected; this is a corpus-navigation and rendering defect.
