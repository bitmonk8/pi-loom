## Finding

DISC-5 in `docs/spec_topics/discovery/package-and-settings.md` pins the `pi.looms` matching contract by naming the `minimatch` package as the matcher engine and citing `@earendil-works/pi-coding-agent`'s `dist/core/package-manager.js` symbol `applyPatterns` (with a further reference to `matchesAnyExactPattern` for the `+`/`-` operand comparison). Both citations sit inside the normative paragraph that defines the `pi.looms` resolution rule. The same prescription propagates to the `loomPaths` schema, which says its glob/override semantics "follow the contract pinned at DISC-5."

The genuine requirement on the discovery surface is *behavioural parity with the Pi resource-array glob dialect and override ordering* (plain → `!` → `+` → `-`, with `-` final over `+`, `nocase` off, three match attempts, exact-path `+`/`-` operands). That parity is what keeps `pi.looms` indistinguishable from `pi.extensions` / `pi.skills` / `pi.prompts` / `pi.themes` for an author. Naming `minimatch` and `applyPatterns` over-fixes the contract on two axes that should not be normative:

- **Library identity.** A loom implementer who reads DISC-5 literally and depends on `minimatch` becomes locked to it; if Pi later swaps to `picomatch` or rolls its own matcher, loom diverges from Pi while remaining "compliant" with the spec text. The matcher choice is a Pi internal, not a wire fact loom needs to mirror byte-for-byte.
- **Pi-internal symbol path.** `dist/core/package-manager.js` and the unexported `applyPatterns` / `matchesAnyExactPattern` symbols are private to `@earendil-works/pi-coding-agent` — citing them inside a normative paragraph reads as if loom may import them, which it cannot (they are not in the SDK surface inventory).

The spec already softens the prescription mid-paragraph ("this paragraph pins loom's *use* of that matching, not minimatch's glob dialect"), but that disclaimer leaves the load-bearing references — engine name and internal symbols — inside the normative text.

## Solution Space

**Shape:** single

### Recommendation

Reframe DISC-5's matcher prescription as **behavioural parity with the Pi resource-array glob dialect and override ordering**, keeping every observable behaviour the current paragraph spells out (the four-step override order with `-` final over `+`, `nocase` off, the three match attempts — relative path / basename / POSIX-normalised absolute path, the exact-path treatment of `+`/`-` operands, and the empty-include "everything under root" starting set) and demoting the `minimatch` / `applyPatterns` / `matchesAnyExactPattern` references to a non-normative implementation note.

Concretely:

- Remove the embedded "with the `minimatch` engine — the same package Pi applies to its own resource arrays in `@earendil-works/pi-coding-agent` (`dist/core/package-manager.js`, symbol `applyPatterns`)" clause from the normative sentence; replace with "with a glob matcher that mirrors the Pi resource-array dialect" (or equivalent parity wording).
- Remove "mirroring Pi's `matchesAnyExactPattern`" from the `+`/`-` operand sentence; keep the operational rule (compared post-`~` expansion against the relative path, the basename, and the POSIX-normalised absolute path).
- Add a non-normative footnote or aside immediately after DISC-5: "Implementation note (non-normative): Pi's own resource arrays are matched by `minimatch` (see `@earendil-works/pi-coding-agent` internals); a loom implementation that delegates to `minimatch` with `nocase` off satisfies the parity requirement above." This keeps the engine name discoverable to implementers without making it the contract.
- Leave the `loomPaths` cross-reference to DISC-5 unchanged in shape — it inherits the reframed contract automatically.

Edge cases the implementer must watch:

- The parity target is **observable behaviour**, not glob *dialect* — `**`, brace expansion, and dotfile defaults must still match Pi's behaviour, even though they are owned by whichever engine Pi chooses. The non-normative note should make that clear so implementers know which `minimatch` options to set (or which behaviours to replicate in an alternative engine).
- The override-order pin (plain → `!` → `+` → `-`, with `-` final) stays normative — it is the part of T092's resolution that was actually load-bearing.
- Plan-leaf wording at `V10b` / `V10b-T` that says "minimatch with `!`/`+`/`-` ordering" should track the spec reframe in the same touch-up, otherwise the plan will read as more prescriptive than the spec.

## Consequence

**Severity:** advisory

A literal reading of DISC-5 brittles loom to Pi's current internal matcher choice: if Pi swaps `minimatch` or renames `applyPatterns`, a conforming loom implementation is now *out of parity* with Pi while still satisfying the spec letter. The citation to `dist/core/package-manager.js` also mis-suggests an internal symbol is part of the contract surface loom may rely on. Day-one implementers will likely converge on the right behaviour (because the override order is also spelled out in the paragraph), so this is not a correctness bug — it is a durability and contract-shape concern.
