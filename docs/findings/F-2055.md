## Finding

The SLSH-4 per-`kind` system-note table in `docs/spec_topics/slash-invocation.md` assigns the stable inline labels `SNK-a, SNK-b, … SNK-i, SNK-k` — it runs `a`→`i`, then jumps to `k`, omitting `SNK-j`. The introductory prose immediately above the table still reads "Each row below carries a stable inline label (`SNK-a` … `SNK-k`, in the `Label` column)", which asserts a span whose endpoints imply a contiguous `a`–`k` set, while the table itself is non-contiguous.

`SNK` is a GOV-16-governed inline-label prefix registered to this page. GOV-16's *Retirement section* clause is a MUST: "All inline-label retirements MUST be recorded in a trailing `## Retired inline labels` section on each page that owns inline-label prefixes, structurally parallel to the `## Retired REQ-IDs` section." `docs/spec_topics/slash-invocation.md` owns the `SNK` prefix but has **no** `## Retired inline labels` section anywhere in the file. The `SNK-j` gap therefore exists with no governance record explaining it: a maintainer or auditor reconciling the live `SNK` set against the page finds a label position (`j`) that is neither live nor accounted for by any retirement table.

The gap is not accidental. `SNK-j` formerly carried the `invoke_callee` template row; it was deliberately removed because SLSH-5's chain-attribution rule computes the descriptive text from the **leaf** `kind`, and every concrete `InvokeCalleeError` carries a required `inner: QueryError`, so there is no inner-less `invoke_callee` for `SNK-j` to format. That rationale is real and the removal is correct — but the retirement was never recorded at the site GOV-16 mandates, and the SLSH-4 prose range was never reconciled. The defect is the missing governance record plus the stale prose, not the absence of the row.

## Issue introduction
**Verdict:** introduced by a single later commit
**Introducing commit:** `4c2121d` (2026-06-27) — "pi-loom spec-plan: fix F-1752 — retire unreachable SNK-j row (SLSH-5 leaf-kind); drop plan SNK-j bullets + coverage-matrix note"
**History:** The `SNK-a … SNK-k` inline labels were coined contiguously by `27b2576` (2026-06-05), which added the labelled per-`kind` table including a live `SNK-j` row for `invoke_callee` — at that point the sequence had no gap. `4c2121d` then deleted the `SNK-j` row to resolve finding F-1752 (the row was unreachable under SLSH-5's leaf-kind rule) and updated the plan-side `coverage-matrix.md` retirement note and `V12b` test bullets, but it did **not** add the GOV-16-mandated `## Retired inline labels` section to `slash-invocation.md` and did **not** reconcile the SLSH-4 prose range. `git log` on the table region and a corpus grep confirm no `## Retired inline labels` section exists on the page in any commit, so the unrecorded gap entered the corpus at `4c2121d` and persists at HEAD. The defect is thus an incomplete-retirement residue of the F-1752 fix, not present since the labels' inception.

## Solution Space

**Shape:** single

### Recommendation

Complete the `SNK-j` retirement bookkeeping on `docs/spec_topics/slash-invocation.md`:

1. Add a trailing `## Retired inline labels` section, structurally parallel to the `## Retired REQ-IDs` sub-table (columns `| Label | Retired in | Replaced by | Reason |`), with one row:
   - **Label:** `SNK-j`
   - **Retired in:** `4c2121d` (the 7-character abbreviated SHA of the removal commit, per the `## Retired REQ-IDs` *Retired in* value-space — 7-char SHA or release tag)
   - **Replaced by:** SLSH-5 (chain-attribution leaf-`kind` rule)
   - **Reason:** the `invoke_callee` template is unreachable — every concrete `InvokeCalleeError` carries a required `inner: QueryError`, so SLSH-5 always recurses to the leaf `kind` and renders the leaf's row; there is no inner-less `invoke_callee` for a dedicated row to format.
2. Reconcile the SLSH-4 prose at line 35 so it no longer implies a contiguous set: keep the `SNK-a … SNK-k` span (its endpoints are correct) but add a clause noting `SNK-j` is retired (pointing at the new `## Retired inline labels` section), so the non-contiguity is explicit rather than apparently erroneous.

Do **not** renumber `SNK-k` (or any other `SNK` label) to close the gap: `SNK` labels are GOV-16-governed identifiers, renumbering breaks existing citations, and the retirement record is the prescribed way to account for a non-contiguous range. The `SNK-a … SNK-k` descriptions in `governance/stable-inline-labels.md` and `governance.md` name the span endpoints and remain correct as written once the retirement is recorded.

## Consequence

**Severity:** correctness

The spec self-contradicts: SLSH-4's prose asserts an `SNK-a … SNK-k` span while the table omits `SNK-j`, and GOV-16's mandatory `## Retired inline labels` record is absent. A maintainer or auditor reconciling the `SNK` label set against the page cannot tell whether `SNK-j` was retired, never existed, or is missing in error, and the plan's per-prefix numbering-hole gate would flag the unaccounted `j` position at the loom 1.0 release-gate footing.
