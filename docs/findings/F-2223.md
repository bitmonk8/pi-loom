## Finding

The *Discriminated unions* section of `schemas.md` justifies the `loom/parse/non-string-discriminator` reject rule with the sentence: *"provider grammar-constrained decoders are only validated against string `const`, and non-string tags degrade decoding quality — exactly the failure mode the discriminator-required rule was introduced to avoid."* This is written as a bare present-tense statement of fact about external provider decoders, with no marker distinguishing it from a normative spec obligation.

Everywhere else the corpus makes a claim about a provider's live capabilities, it is explicitly fenced as design context. `schema-subset.md` is the canonical example: its provider-alignment paragraphs carry `*(Non-normative provenance.)*` and spell out that the rationale is "design context, not a present-tense claim about any provider's live capabilities, and the subset is not re-derived from those capabilities if they later drift." The depth-ceiling and JSON-Schema-draft entries on the same page repeat that discipline verbatim.

The `schemas.md` sentence is the lone parallel claim that is *not* fenced. It reads as a live assertion that today's providers behave in a specific way, which (a) can silently go stale if a provider's grammar-constrained decoder changes, and (b) blurs the spec-owned reject rule — which stands on its own and does not consume any external provider fact — into something that looks contingent on provider behaviour. The fix is to apply the same non-normative-provenance treatment used in `schema-subset.md`: mark the provider-decoder sentence as design-context rationale, not a live provider claim. No provider-version pin and no presupposition block are warranted — the normative reject rule does not depend on the external fact.

## Solution Space

**Shape:** single

### Recommendation

Reframe the provider-decoder clause in the `schemas.md` *Discriminated unions* rationale so it carries the same non-normative-provenance treatment as `schema-subset.md`: keep the normative reject statement (numeric and boolean literal discriminators are rejected in loom 1.0 with `loom/parse/non-string-discriminator`), and mark the provider-decoder justification as design context rather than a live claim — e.g. wrap it with `*(Non-normative provenance.)*` and word it as the loom-1.0-authoring-time rationale ("the rule was chosen because the loom 1.0-supported providers' grammar-constrained decoders were validated against string `const`; this is design context, not a present-tense claim about any provider's live capabilities, and the reject rule is loom-owned and does not track later provider drift"). Do not add a provider-version pin or a presupposition/citation block — the reject rule is spec-owned and does not consume the external fact. Leave the `kind: "v1"` authoring guidance and the implicit/explicit-`by` scope clause intact.

## Consequence

**Severity:** advisory

The reject rule itself is unambiguous, so two implementers will not diverge on behaviour. The defect is a provenance/consistency gap: an unfenced present-tense claim about external provider decoders can go stale on provider drift and reads as a normative contingency, contradicting the corpus's own discipline of marking such claims non-normative.
