## Finding

The compact-transcript renderer (rule 4 / BNDR-8) emits each assistant `ToolCall` as a single line of the form `[tool-call <name>(<args-json>)]`, where `<name>` is `ToolCall.name` interpolated **verbatim**. BNDR-7 pins these renderings as MUST-reproduce-exactly, and rule 3 establishes that continuation lines carry no role-tag prefix — so the line's structure depends on `<name>` containing none of the bytes that would break a single line: U+000A (`\n`), U+000D (`\r`), the closing `]`, or the trailing `)`.

The spec recognises exactly this hazard for the *sibling* single-line interpoland on the same role-tag line: BNDR-9 declares a **transcript-safe** precondition for `CustomMessage.customType` (non-empty; none of `\n`, `\r`, `]`, `: `), justified explicitly by "the bytes whose appearance … would break the line, the closing `]`, or the `: ` role-tag separator and so violate BNDR-7's reproduce-exactly contract." Its rationale rests on the observation that "Pi types `CustomMessage.customType` only as `string` … and no upstream rule constrains its characters," so the binder MUST reject an out-of-class value and abort the invocation (`loom/runtime/custom-type-unsafe`) rather than interpolate it.

`ToolCall.name` is in the identical epistemic position. Per the Pi Integration Contract (`host-interfaces-core.md`, `AssistantMessage`/`ToolCall`), it is declared in `@earendil-works/pi-ai` and typed only as a `string`; no loom-side rule pins its charset, and the spec carries no host-prerequisite asserting that registered tool/function names are constrained to a transcript-safe class. A tool name containing `\n` would split the `[tool-call …]` line into a phantom continuation line; a name containing `]` or `)` would corrupt the line's structural delimiters. The renderer is total over the byte stream, so the bytes stay deterministic, but the byte-exact line *form* BNDR-7 guarantees — and the line-per-tool-call structure the binder model reads — is silently broken. The asymmetry is the defect: one interpoland on the role-tag line is guarded, its sibling is trusted, with no stated justification for the difference. (The `<args-json>` interpoland is largely protected because the canonical no-whitespace JSON serialisation escapes control characters, though a literal `)` or `]` inside a string value remains an unescaped reader-ambiguity rather than a line break; `ToolCall.name` is the unprotected case.)

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set"); `9568906` (2026-06-09, "pi-loom spec: resolve \"custom:<type> role tag customType safe-character constraint\"")
**History:** The corpus is git-tracked. `binder-model-and-context.md` was created at `f5e89f4` (the spec-sharding commit); the `[tool-call <name>(<args-json>)]` rendering with verbatim `ToolCall.name` interpolation has been present since that inception (`git log -S'[tool-call <name>'` over the file returns only `f5e89f4`, and `git log --diff-filter=A --follow` confirms `f5e89f4` is the file's add). At inception neither role-tag-line interpoland was guarded — `customType` and `ToolCall.name` were uniformly unprotected. The asymmetry the finding flags was introduced later by `9568906`, which added BNDR-9's transcript-safe precondition (and the `transcript-safe` constraint on rule 3's `<custom:<type>>` tag) for `customType` only, without extending the same guard to `ToolCall.name`. Verified via `git show 9568906` (adds the BNDR-9 block and the rule-3 transcript-safe clause; touches no `ToolCall.name` text) and `git grep -c 'tool-call <name>' 9568906^` (the verbatim interpolation already present at the parent). The defect is therefore the interaction of an inception-era unguarded interpolation with a later selective guard that covered only the sibling field.

## Solution Space

### Recommendation

Extend the BNDR-9 transcript-safe posture to `ToolCall.name`. In `binder-model-and-context.md`, add a BNDR-9-sibling clause (BNDR-9b) declaring a transcript-safe precondition over `ToolCall.name`: the name MUST be non-empty and MUST contain none of the bytes that would break the `[tool-call <name>(<args-json>)]` line form — `\n`, `\r`, `)`, and `]`. (The name sits before the `(` and is not on a `: ` role-tag separator, so the relevant breaking bytes are `\n`/`\r`/`)`/`]`, not BNDR-9's `: `.) Before rendering, the binder MUST reject any included assistant `ToolCall` whose `name` is not transcript-safe, abort transcript construction for that invocation (no proceed-and-drop), and surface a runtime diagnostic.

Register the diagnostic in `diagnostics/code-registry-runtime.md`, reusing the `loom/runtime/custom-type-unsafe` pattern with a new code (or generalising it to a transcript-unsafe-interpoland cause), and add the matching row to the Failure-mode templates in `determinism-cancellation-failure.md`. State in the BNDR-9b clause whether the `<args-json>` `)`/`]`-inside-string reader-ambiguity is accepted: it does not break the line, since control chars are JSON-escaped and the canonical no-whitespace serialisation keeps the bytes deterministic, so the accepted scope is the verbatim `ToolCall.name` interpoland only.

## Consequence

**Severity:** correctness

Absent a stated rule, two reasonable implementers diverge: one mirrors BNDR-9 and rejects/aborts on an out-of-class tool name, the other interpolates `ToolCall.name` verbatim. A name carrying `\n`/`]`/`)` then silently corrupts the binder's transcript input — splitting the `[tool-call …]` line or breaking its delimiters — degrading binder grounding and violating the BNDR-7 reproduce-exactly contract that the page's input-reproducibility claim depends on.
