## Problem

The compact-transcript renderer (rule 4 / BNDR-8) emits each assistant `ToolCall` as a single line `[tool-call <name>(<args-json>)]` with `ToolCall.name` interpolated verbatim, and BNDR-7 pins that line form as MUST-reproduce-exactly. Per the Pi Integration Contract `ToolCall.name` is typed only as `string` with no loom-side charset constraint, so a name containing `\n`, `\r`, `]`, or `)` silently splits the line into a phantom continuation or corrupts its structural delimiters. BNDR-9 already guards the sibling role-tag-line interpoland `CustomMessage.customType` against exactly this hazard (reject-and-abort rather than interpolate), but `ToolCall.name` carries no equivalent guard. The asymmetry — one interpoland on the line guarded, its sibling trusted with no stated justification — is the defect.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set"); `9568906` (2026-06-09, "pi-loom spec: resolve \"custom:<type> role tag customType safe-character constraint\"")
**History:** The corpus is git-tracked. `binder-model-and-context.md` was created at `f5e89f4` (the spec-sharding commit); the `[tool-call <name>(<args-json>)]` rendering with verbatim `ToolCall.name` interpolation has been present since that inception (`git log -S'[tool-call <name>'` over the file returns only `f5e89f4`, and `git log --diff-filter=A --follow` confirms `f5e89f4` is the file's add). At inception neither role-tag-line interpoland was guarded — `customType` and `ToolCall.name` were uniformly unprotected. The asymmetry the finding flags was introduced later by `9568906`, which added BNDR-9's transcript-safe precondition (and the `transcript-safe` constraint on rule 3's `<custom:<type>>` tag) for `customType` only, without extending the same guard to `ToolCall.name`. Verified via `git show 9568906` (adds the BNDR-9 block and the rule-3 transcript-safe clause; touches no `ToolCall.name` text) and `git grep -c 'tool-call <name>' 9568906^` (the verbatim interpolation already present at the parent). The defect is therefore the interaction of an inception-era unguarded interpolation with a later selective guard that covered only the sibling field.

## Solution approach

In `binder-model-and-context.md`, add a BNDR-9-sibling transcript-safe precondition over `ToolCall.name`: the name MUST be non-empty and contain none of the bytes that break the `[tool-call <name>(<args-json>)]` line form — `\n`, `\r`, `)`, `]` — and the binder MUST reject an out-of-class name and abort transcript construction (no proceed-and-drop) rather than interpolate it. Register the matching runtime diagnostic in `diagnostics/code-registry-runtime.md` mirroring the `loom/runtime/custom-type-unsafe` row, and add the corresponding row to the Failure-mode templates in `determinism-cancellation-failure.md`. State whether the `<args-json>` `)`/`]`-inside-string reader-ambiguity is in or out of scope for the new guard.
