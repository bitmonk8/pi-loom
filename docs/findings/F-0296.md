## Finding

`V16a`'s **Adds** describes the cross-ceiling arbitration seam as "a stateless function `arbitrate(candidate) → { surfaced, masked? }`" followed by an **Input** / **Output** signature block. The function name `arbitrate`, the parameter name `candidate`, and the exact call/return literal are an implementation-shape prescription, not an observable obligation.

Per `conventions.md` §leaf-format (**Adds.**), `Adds.` is descriptive by default. A mechanism named in `Adds.` binds only when it is a Class-1 observable behaviour traced to a normative obligation, or a Class-2 cross-leaf seam that a consumer binds. Here the *seam* is genuinely Class-2 — `V4e`, `V5e`, `V11f`, `V13c`, `V15b`, and `H7a` all reference "`V16a`'s cross-ceiling arbitration seam" and bind it via their `Deps` on `V16a`. But every consumer references it as *the seam* and binds only its observable contract: the cross-ceiling surfacing precedence and the `masked` enumeration. None names the function `arbitrate`, the parameter `candidate`, or the literal return shape. The output field names `surfaced` and `masked` are independently anchored (`masked` is pinned at `ceilings-3-and-4.md#masked-field`; both are asserted by the CIO-6 Tests bullet), so the binding semantics already live in Tests.

What remains in the signature block — the function name, the parameter name, and the `arbitrate(candidate) → { … }` call form — is Class-3 shape: it prescribes how the seam is coded rather than what it must observably do. It is illustrative under the convention, but its placement as a bold signature line risks being read as binding and over-constraining the implementer.

## Issue introduction

**Verdict:** introduced-by-later-edit
**Introducing commits:** `e2b7e81` (2026-06-10) — "pi-loom plan: resolve \"isolated cross-ceiling unit interface/authority undefined\""
**History:** The plan corpus is git-tracked. `V16a` was first authored in `c6a664e` (2026-06-10, initial plan build) with an **Adds** that described the CIO order, the at-most-one rule, and the `masked` enumeration in prose, with no explicit function signature. `git log -S 'arbitrate'` and `git show e2b7e81` show the `arbitrate(candidate) → { surfaced, masked? }` signature and the Input/Output block were added in `e2b7e81`, a remediation commit closing a prior finding that the isolated cross-ceiling unit's interface/authority was undefined. That fix over-corrected: it pinned an exact function signature to resolve an under-specification, introducing the Class-3 shape prescription this finding flags. Single introducing commit; the durable fix is to express the seam contract at the observable level the remediation was actually reaching for.

## Solution Space

**Shape:** single

### Recommendation

In `V16a`'s **Adds**, remove the implementation-shape prescription — the bold `arbitrate(candidate) → { surfaced, masked? }` function-signature clause and the **Input** / **Output** signature bullets — and describe the seam by its observable contract instead: a stateless cross-ceiling arbitration seam that, given the set of ceiling-candidates satisfied at a check site (tagged by check-site / ceiling class), yields the single `surfaced` ceiling identifier per the CIO order plus the optional `masked` enumeration of co-fired siblings (omitted when empty), as pinned at [`ceilings-3-and-4.md` — `masked` field](../spec_topics/hard-ceilings/ceilings-3-and-4.md#masked-field).

Retain the Class-2 seam framing ("cross-ceiling arbitration seam" consulted by `V4e`/`V5e`/`V11f`/`V13c`/`V15b` and `H7a`, each binding it via `Deps`) and the existing `surfaced`/`masked` output field names — those are bound by the CIO-6 Tests bullet and the spec `masked`-field anchor. The binding obligations already live in the CIO-1 … CIO-6 Tests bullets and the **Ships when** gate; do not duplicate them as a prescribed function signature.

## Consequence

**Severity:** advisory

An implementer reading the bold `arbitrate(candidate) → { surfaced, masked? }` line and the Input/Output block may treat the function name, parameter name, and exact call shape as required, over-constraining the seam's implementation even though only its observable `surfaced`/`masked` output is bound by Tests and the spec `masked`-field anchor. The leaf still ships a working seam; the cost is interpretive over-constraint, not divergence.
