## Finding

`V10d` adds two distinct deliverables: the `Clock`-driven 250 ms watcher-event reload debounce, and the settings-re-merge sub-arm of the **watcher-time reload failure-injection seam**. The leaf's `Adds` describes the arm concretely — a caller supplies a synthetic settings-re-merge failure to `ReloadFailureInjector.injectReloadFailure`, and the arm routes it onto the `loom-system-note` surfacing path with `triggerTurn:false` without standing up a live watcher. But `V10d`'s sole `Tests` bullet and its `Ships when` condition cover only the debounce coalescing behaviour through the `FakeClock` seam; neither references the settings-re-merge arm at all.

The arm's only assertion lives in `V4g` (`ERR-7` watcher-time reload-failure integration), which depends on `V10d` and exercises the arm together with `V9b`'s registry-swap and `.loom`/`.warp` re-parse arms. Because `V4g` sits downstream of `V10d` in the `Deps` chain, `V10d` can be tagged `V10d-complete` (its `Ships when` observable) while the arm it contributes has not been exercised by any landed test. A defect introduced in the arm then surfaces later as a red `V4g`, attributing the failure to the integration leaf rather than the leaf that owns the arm code.

This is a delegated-verification gap rather than a coverage hole: the arm is verified before release by `V4g`, and the delegation is named explicitly in both leaves. The defect is the absence of a per-leaf gate at the contributing leaf, which weakens defect attribution and lets `V10d` ship green with an unexercised deliverable.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** f14e929 — pi-loom plan: resolve "split V10c into settings-merge and reload-debounce leaves" (2026-06-16, Thomas Andersen)
**History:** `V10d` was created at f14e929 by splitting the reload-debounce and settings-re-merge sub-arm out of `V10c` into a new leaf. From that first commit the leaf's single `Tests` bullet and `Ships when` covered only the 250 ms debounce, while the settings-re-merge arm was delegated to the integration leaf (then `V4e`, later retargeted to `V4g` at 324813d). The per-leaf gate gap is therefore present in the leaf's inception commit and was never introduced by a later edit.

## Solution Space

**Shape:** single

### Recommendation

Add a `Tests` bullet to `V10d` (and the paired failing-test bullet to `V10d-T`) that gates the arm at its contributing leaf: assert that a synthetic settings-re-merge failure supplied to `ReloadFailureInjector.injectReloadFailure` routes onto the `loom-system-note` surfacing path with `triggerTurn:false`, without standing up a live watcher. Extend `V10d`'s `Ships when` so the observable completion condition references that assertion alongside the existing debounce-coalescing condition.

The arm-routing assertion binds to the `injectReloadFailure` interface `V9b` declares; if `V9b`'s parameter contract (the arm discriminator and synthetic-failure payload) is pinned, phrase the new bullet's `injectReloadFailure(...)` call against that pinned shape so it stays consistent with `V4g`'s injection. `V4g`'s `ERR-7` integration assertion is unchanged and remains the cross-leaf gate; the new bullet is the per-leaf gate, not a replacement.

## Consequence

**Severity:** advisory

The settings-re-merge arm `V10d` contributes can pass `V10d`'s `Ships when` (`V10d-complete`) without any landed test exercising it; a defect in the arm only reddens the later `V4g` integration leaf, mis-attributing the failure to the integration gate rather than the arm's owning leaf. Verification before release is not lost — `V4g` is a required leaf and exercises the arm — so an implementer still produces a working slice; only attribution and leaf self-completeness degrade.
