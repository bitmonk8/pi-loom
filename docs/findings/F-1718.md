## Finding

PIC-13's `realpath` bullet (`host-interfaces-services.md`) binds the **Case-canonicalisation guarantee** — "on a case-insensitive filesystem, two input paths that differ only in component or leaf case but resolve to one directory entry MUST yield byte-identical `realpath` output" — onto *both* the production `PiFileSystem` adapter and `FakeFileSystem`. It explicitly states that bare delegation to `fs.promises.realpath` is insufficient, because Node's `fs.promises.realpath` and `fs.promises.realpath.native` differ on leaf-case output, so the production adapter must canonicalise leaf case to satisfy the guarantee. The guarantee is therefore a real conformance obligation on the production adapter, not just on the fake.

The witnessing fixture — two case-variant paths that resolve to one directory entry — cannot be constructed against the production adapter on a case-sensitive filesystem. On Linux/ext4, the default `npm test` substrate, `Plan.loom` and `plan.loom` are *distinct* directory entries, so the inputs the assertion needs do not exist; only on macOS APFS or Windows NTFS are they alternate spellings of one entry where the assertion is meaningful. PIC-13 carries no platform-host requirement, no skip protocol, and no delegation to the manual real-host smoke gate (`real-host-smoke-gate.md`) that already houses comparable real-host-only behaviours. The project's "no silent test skipping" rule (`CLAUDE.md` §Testing) also forbids the obvious early-return/skip-macro workaround on Linux, since that would report success while having verified nothing.

The result is a normative MUST whose production-adapter half is not exercisable under `npm test` on the only CI substrate the project commits to, and is not routed to any other witness, leaving the leaf-case canonicalisation behaviour in the production `PiFileSystem` adapter unverified.

  - `6ab9f7b` (2026-06-25) "pi-loom spec-plan: fix F-1293 — pin PIC-13 realpath transitive resolution + byte-stability" — extended the same bullet with the **Byte-stable canonical-output guarantee** (of which case-canonicalisation is named the case-insensitive-FS instance) and the **Transitive-symlink-resolution guarantee**, again binding both halves to the production adapter without addressing the production-adapter testability gap.
**History:** the defect entered with the guarantee itself: `a765f6ca` is the only commit that has ever touched the **Case-canonicalisation guarantee** clause on this file (verified via `git log -S "byte-identical" -- docs/spec_topics/pi-integration-contract/host-interfaces-services.md`, which returns exactly the two commits above; the more general byte-stable phrasing arrived at `6ab9f7b`). Neither fix paired its normative bind-to-both-adapters with a testability protocol — no platform-host requirement, no skip-on-case-sensitive-host clause, no delegation to the real-host smoke gate, and no companion edit to `V8b` / `V8b-T` adding a production-adapter test bullet. The guarantee was therefore untestable against the production adapter under `npm test` from its first commit, and remains so on HEAD.

## Solution Space

**Shape:** multiple

### Option A — Require production-adapter case-canonicalisation test on a case-insensitive host with loud platform-skip

**Approach.** Amend PIC-13's `realpath` bullet so the case-canonicalisation guarantee's production-adapter conformance test MUST run on a case-insensitive filesystem and MUST loudly fail-skip on a case-sensitive host (per `CLAUDE.md` §Testing: no silent skips — surface the skip as a recorded outcome via `assert!`/`panic!`-style loud recording, not an early return). Add the assertion as a `V8b` / `V8b-T` **Tests** bullet that constructs `Plan.loom` and `plan.loom` in a temp directory, calls `PiFileSystem.realpath` on both, and asserts byte-identical output; on a case-sensitive host the bullet records an unverified-on-this-host outcome through a recognised channel rather than passing silently.

**Spec edits.**
- In PIC-13's `realpath` bullet, append after the **Case-canonicalisation guarantee** sentence: "*Production-adapter conformance test.* This guarantee's production-adapter half is verified by a conformance test that MUST run on a case-insensitive filesystem (Windows NTFS, macOS default APFS); on a case-sensitive host the test MUST loudly record an unverified-on-this-host outcome rather than silently pass."
- In `V8b` / `V8b-T`, add a **Tests** bullet: "`PIC-13` (case-canonicalisation, production adapter): with `Plan.loom` and `plan.loom` constructed in a temp directory, `PiFileSystem.realpath` on both inputs yields byte-identical output; the test loud-fail-skips on a case-sensitive host."

**Pros.**
- Keeps the production-adapter assertion inside the automated `npm test` surface on hosts where it is constructible.
- Detects a bare-`fs.promises.realpath` regression at first developer run on Windows/macOS, before merge.

**Cons.**
- On the project's current Linux-only CI the loud-skip permanently records "unverified-on-this-host" with no positive evidence — closer to documenting the gap than closing it.
- Adds the only host-platform branch in `V8b`'s leaf; every other PIC-13 assertion is host-agnostic.

**Risks.**
- The production-adapter half is never positively witnessed until a CI matrix or a Windows/macOS developer happens to run the suite, which may never occur in the loom 1.0 window.

### Option B — Scope `npm test` to the `FakeFileSystem`; delegate the production-adapter half to the real-host smoke gate

**Approach.** Add a non-normative platform note to PIC-13 stating that the production-adapter half is witnessed at the manual real-host smoke gate (`real-host-smoke-gate.md`), while the `FakeFileSystem` half (whose in-memory symlink table can simulate case-insensitive collapse on any host) stays under `npm test`. Add a smoke-gate criterion that, when run on a case-insensitive Pi host, exercises a `Plan.loom` / `plan.loom` fixture pair and verifies `PiFileSystem.realpath` returns byte-identical output. Update `V8b` / `V8b-T` **Tests** to assert the `FakeFileSystem` half and reference the smoke-gate delegation for the production half — mirroring how `V4c` / `V9i` delegate ERR-8's mid-stream-cancellation live-surface confirmation to the same gate.

**Spec edits.**
- In PIC-13's `realpath` bullet, append a *Production-adapter platform note* clause: "*Non-normative — production-adapter platform note.* The production-adapter half of the case-canonicalisation guarantee (and of the byte-stable canonical-output guarantee, of which it is the case-insensitive-FS instance) is witnessable only on case-insensitive hosts; `npm test` asserts the algorithmic shape against the `FakeFileSystem`, and the production `PiFileSystem` half is witnessed at the manual real-host smoke gate (`real-host-smoke-gate.md`) per its activation triggers."
- In `real-host-smoke-gate.md`'s Pass/fail criteria, add a criterion: "when the live Pi host's filesystem is case-insensitive, a `Plan.loom` / `plan.loom` fixture pair under a discovery root yields one byte-identical `realpath` output from `PiFileSystem.realpath`, witnessing the PIC-13 case-canonicalisation guarantee on the production adapter." Extend the smoke-gate's fidelity-axis trigger so an edit to `PiFileSystem.realpath`'s case-canonicalisation step also fires the smoke.
- In `V8b` / `V8b-T`, add a **Tests** bullet asserting the guarantee on `FakeFileSystem` (case-variant inputs resolve to one canonical path) and note that the production-adapter half is delegated to the smoke gate.

**Pros.**
- Works on Linux-only CI without a permanently-skipped automated assertion.
- Reuses the project's established convention for real-host-only behaviours (`V4c` / `V9i`), inventing no new spec mechanism.
- The `FakeFileSystem` assertion under `npm test` still covers algorithmic shape — DISC-3 and INV-1 consume the guarantee through the fake.

**Cons.**
- The production-adapter conformance becomes a manual gate; an absentminded contributor on a case-insensitive Pi host can skip the criterion (recoverable only post-hoc via the H6a evidence audit).
- Delays detection of a bare-`fs.promises.realpath` regression from `npm test` to the next smoke run.

**Risks.**
- Without the trigger extension above, a refactor of `PiFileSystem.realpath` that drops leaf-case canonicalisation would not by itself fire the smoke gate, so the gap would close only at the next version bump or fidelity-axis merge.

### Recommendation

Take **Option B**. It mirrors the convention this project already uses for real-host-only behaviours (`V4c` / `V9i` delegate ERR-8 the same way), so no new spec mechanism is invented, and it lets `npm test` produce a positive automated witness on the `FakeFileSystem` half — which is what every downstream consumer (DISC-3, INV-1) actually exercises through the seam — rather than a permanently-skipped Linux-CI placeholder. The contributor-obligation gap on the production adapter is real but bounded, and is the same exposure ERR-8 already accepts; the smoke-gate trigger extension narrows it. Implementer watch-points: (i) the `FakeFileSystem` test must drive a case-variant input pair through the in-memory symlink table, not bypass it, or the assertion regresses to a no-op; (ii) the smoke-gate criterion is conditional on the live host being case-insensitive — on a Linux Pi host it is an un-runnable-gate condition, not a failure; (iii) the spec edit must cover both the case-canonicalisation guarantee and the byte-stable canonical-output guarantee, since the former is the case-insensitive-FS instance of the latter.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one ships a production adapter that bare-delegates to `fs.promises.realpath` and silently violates the guarantee — uncaught by `npm test` on Linux CI, because the witnessing fixture cannot be built there — while another writes a custom adapter that canonicalises leaf case. The DISC-3 case-collision dedup and the INV-1 invocation-time containment re-check both rely on the production adapter honouring the guarantee, so a silently non-conforming adapter produces wrong behaviour at first contact with a case-insensitive host (false-negative case-collision dedup; sibling-case-variant invoke paths that should canonicalise to one root-contained path but do not).
