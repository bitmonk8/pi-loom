## Finding

The `<a id="loom-flag-namespace"></a>` paragraph in `discovery-sources.md` asserts three concrete behaviours of `pi.registerFlag('loom', …)` under the pinned SDK — (i) it does not throw on a flag-name overlap, (ii) registration is "first-load-wins", and (iii) a later sibling extension that registers a `'loom'` flag is "silently shadowed with no diagnostic", with loom's own `pi.getFlag('loom')` still observing loom's registration. None of these clauses cites the SDK source they describe, even though the rest of the contract corpus (e.g. the [Non-capability `pi.<member>` surfaces](../pi-integration-contract/inventory-audit-intro.md) paragraph that pins `pi.registerFlag` / `pi.getFlag` at `dist/core/extensions/types.d.ts`) routinely names the SDK file when it makes such a claim.

The behaviours are verifiable from the pinned SDK. In `@earendil-works/pi-coding-agent@0.75.5` `dist/core/extensions/loader.js`, `registerFlag` writes to a per-extension `extension.flags` Map and then writes the default into the shared `runtime.flagValues` only when `!runtime.flagValues.has(name)`; `getFlag` returns `undefined` unless the calling extension has the name in its own `extension.flags`, then reads from `runtime.flagValues`. So no-throw and first-load-wins are accurate. The "loom's `pi.getFlag('loom')` still observes loom's registration" clause is correct at the gate (loom's `extension.flags.has('loom')` remains true) but imprecise on the value: the shared `runtime.flagValues['loom']` may carry the *other* extension's default if that extension registered first, and any CLI `--loom` value (`dist/core/agent-session-services.js`'s `applyExtensionFlagValues`) overrides for both extensions equally.

The section's closing sentence explicitly disclaims normativity ("The check has no REQ-ID, no per-leaf test obligation, and emits no diagnostic"), so no implementation leaf depends on these clauses being right. The defect is therefore an unsourced empirical claim in a non-normative observation: the spec corpus elsewhere holds itself to citing the SDK pin when it characterises SDK behaviour, and the loom-flag-namespace paragraph is the visible exception.

## Solution Space

**Shape:** single

### Recommendation

Cite the SDK source for each behavioural clause in the `loom-flag-namespace` paragraph, mirroring the citation discipline already used in the [Non-capability `pi.<member>` surfaces](../pi-integration-contract/inventory-audit-intro.md) paragraph. Concretely, append to the existing sentence a parenthetical pinning the mechanism at `dist/core/extensions/loader.js`'s `registerFlag` (no-throw; shared `runtime.flagValues` only populated by the first-load default via the `!runtime.flagValues.has(name)` gate) and `getFlag` (per-extension `extension.flags.has(name)` gate followed by a shared `runtime.flagValues.get(name)` read), under the [loom 1.0 Pi-SDK pin range](../pi-integration-contract/host-prerequisites.md#pi-sdk-pin) already cited at the head of the paragraph.

While making that edit, sharpen the "loom's own `pi.getFlag('loom')` read still observes loom's registration" clause: under the SDK source, what loom observes is *the gate plus the shared `runtime.flagValues` entry* — that entry holds the first-load default until CLI `--loom` overrides it for both extensions equally. The current wording reads as "loom sees loom's default value", which is not what the loader implements. Re-state the clause as "loom's `pi.getFlag('loom')` continues to return the shared-runtime flag value (the first-registered default, or the CLI `--loom` value when supplied) — the loader does not strip loom's entry from `extension.flags` on a sibling collision".

Edge cases to preserve while editing:

- Keep the existing closing disclaimer ("no REQ-ID, no per-leaf test obligation, and emits no diagnostic"). Adding a citation does not promote the paragraph to normative — it remains a sourced observation.
- Do not add a new `loom/load/...` code for flag-name collision; the spec is explicit that none exists in loom 1.0.
- The SDK-source citation is itself a single-site claim — keep it textually adjacent to the existing `pi-sdk-pin` link so a future SDK bump that changes the loader contract is a one-paragraph audit, not a corpus-wide grep.

## Consequence

**Severity:** advisory

The defect is local: the section is explicitly non-normative, so no implementer is misled into a behavioural divergence. The cost is editorial — an empirical SDK claim that sits below the corpus's own citation discipline, and which a later reader cannot validate without re-reading the SDK source themselves. If a future Pi minor bump changes the dedup gate or `getFlag`-gate semantics, the unsourced clause is also the kind of line that silently becomes wrong without a single-site update.
