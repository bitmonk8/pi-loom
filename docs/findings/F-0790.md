## Problem

The thirteen acceptance items (a)–(m) under `## Acceptance criteria` describe scenarios and observable outcomes but carry no explicit pointer — by REQ-ID, anchor fragment, or stable section heading — to the MUST / MUST NOT obligation each one verifies; they key only on inline prose tokens (e.g. "the state-independent-write pin", "the interstitial-seam swallow rule"). None of the rule-bearing sections above the AC list carries a REQ-ID anchor, and no REQ-ID prefix is registered for the page. A test author cannot determine from an AC alone which spec rule a fixture asserts against, and a failing fixture maps to no specific obligation without manually scanning the preceding normative prose.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `52b9101` (2026-05-11, "pi-loom spec: resolve \"T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state\"") — first introduced the acceptance criteria block as (a)–(k); `2775382` (2026-06-05) extended it with (l)–(m); `f5e89f4` (2026-06-04) split the page into its own file without altering AC structure.
**History:** `git log -S "Positive — clean session-only reason"` shows two commits touch the AC block — `52b9101` (creation as (a)–(k) inside `pi-integration-contract.md`) and `f5e89f4` (the file split). The two-AC extension to (l)–(m) by `2775382` resolved a separate finding ("Two throwing-readDrainState MUSTs have no acceptance criteria") that added test coverage but did not re-litigate the back-reference shape. The defect entered with the original drafting in `52b9101`: from creation, the ACs have keyed on inline prose tokens rather than REQ-IDs or anchors, and no REQ-ID prefix was ever allocated for the page (`git log -S "DEGR-"` returns nothing in the spec corpus). The peer finding `session-only-degraded-state.md — Recovery-path prohibitions lack per-obligation REQ-IDs` is the upstream half of the same drafting choice; this finding is the downstream half (the ACs that would, post-DEGR-N allocation, cite those REQ-IDs).

## Solution approach

Once the peer finding registers per-obligation REQ-IDs for the page's MUST / MUST NOT sites, append a `(verifies …)` back-reference to each of (a)–(m) under `## Acceptance criteria`, citing the obligation that AC asserts against. For ACs whose obligation lives on a sibling page (notably (k), (l), (m)), use the explicit cross-page link form per GOV-9; for in-page obligations the bare REQ-ID reference suffices.

## Solution constraints

- Out of scope: allocating the page's REQ-ID prefix and the per-obligation REQ-ID anchors — owned by the peer recovery-path-prohibitions finding. This finding adds back-references only after those IDs exist; it MUST NOT mint a new REQ-ID prefix (GOV-7).
