## Finding

The thirteen acceptance items (a)–(m) under `## Acceptance criteria` describe scenarios and observable outcomes but carry no explicit pointer (by REQ-ID, by anchor fragment, or even by stable section heading) to the MUST / MUST NOT obligation each one is verifying. Most items name the rule only via inline prose tokens — "the state-independent-write pin", "the interstitial-seam swallow rule", "arm (c) of `LoomRegistry.readDrainState`", "the *Predicate split* clause" — and rely on a reader's linear knowledge of the page to map each AC back to its source rule.

The mapping problem is compounded by the page itself: none of the obligations under `## Drain-state tag write`, `## Recovery path and prohibitions`, `## Diagnostic emission`, `## Predicate split`, `## Co-emission rules`, or `## Idempotent re-entry and the all-three-throw corner case` carries a REQ-ID anchor. No prefix has been allocated for this page (the file contains zero `PREFIX-N` markers, and the [REQ-ID prefix table](../../docs/spec_topics/governance/req-id-prefix-table-active-a.md#req-id-prefix-table) carries no entry for it). The peer finding on the recovery-path prohibitions proposes a DEGR-1…DEGR-4 prefix for the four MUST NOTs; once that prefix exists (or is widened to cover the rest of the page's MUSTs), each AC needs a parenthetical back-reference like `(verifies DEGR-N)` or `(verifies #anchor)` so a test author dropping in fresh, can determine without ambient context which obligation each fixture is asserting against.

This is a traceability gap, not a behavioural one: every AC's observable expectation is concrete enough to write a fixture against, but a fixture passing on (b) while failing on (i) gives no per-obligation signal beyond "branch (b) failed".

## Solution Space

**Shape:** single

### Recommendation

This finding is the downstream half of the peer finding *Recovery-path prohibitions lack per-obligation REQ-IDs*; resolve in that order.

1. **Establish the REQ-ID prefix first** (owned by the peer finding). Allocate a DEGR-* family that covers, at minimum:
   - the four `## Recovery path and prohibitions` MUST NOTs (DEGR-1…DEGR-4, per the peer finding);
   - the state-independent drain-state tag write under `## Drain-state tag write` (one MUST);
   - the `## Diagnostic emission` section's emission obligation, isolation wrap, and the *Inline triplet is normative* clause (three independently-satisfiable MUSTs);
   - the *Predicate split* table and *Seam minimality (normative)* clause (one MUST plus one MUST NOT);
   - the `## Idempotent re-entry and the all-three-throw corner case` paragraph's MUST NOT against suppression seams.
   
   Register the DEGR prefix in the [REQ-ID prefix table](../../docs/spec_topics/governance/req-id-prefix-table-active-a.md#req-id-prefix-table) per GOV-1 / GOV-3, with dual-form anchor placement per GOV-1's *Dual-form layout*.

2. **Then back-reference each AC.** Once the REQ-IDs exist, append a parenthetical `(verifies DEGR-N[, DEGR-M])` to each of (a)–(m). The mapping is direct from the prose each AC already carries — for example:
   - (a) / (b) / (c) verify the diagnostic-emission MUST + the closed-set predicate;
   - (d) / (e) verify the broad tag-transition predicate (arms 2 and 3);
   - (f) / (g) verify the slash-arm observability rules;
   - (h) verifies the idempotent-clause short-circuit;
   - (i) verifies the state-independent-write MUST under `## Drain-state tag write`;
   - (j) verifies the broad short-circuit predicate `drained || tag !== undefined`;
   - (k) verifies the interstitial-seam swallow rule (cross-page link to `patch-skew-degradation.md#interstitial-seam-tag-write-swallow` because the rule itself lives on a sibling page);
   - (l) / (m) verify the [`drain-state-contract.md` Read-failure fallback](../../docs/spec_topics/pi-integration-contract/drain-state-contract.md#read-failure-fallback) MUSTs (cross-page links to that page's REQ-IDs).
   
   Where the obligation an AC verifies lives on a different page (notably (k), (l), (m)), use an explicit cross-page link `(verifies [REQ-ID](./path#anchor))` per GOV-9; in-page references can use the bare `(verifies DEGR-N)` form.

3. **Update the V9l / V9l-T `**Tests.**` bullets** in the same edit so they cite DEGR-N REQ-IDs instead of `§<section title> (un-anchored; GOV-22 residue)`, dropping the GOV-22-residue qualifier.

Implementer edge cases:

- The MUST under *Inline triplet is normative* binds a future spec edit ("the inbound-reference sweep MUST update this inline triplet in the same edit"), not a runtime obligation; it still needs a REQ-ID for governance traceability but it is not testable by a V9l-T fixture. Mark its AC slot as "spec-editing obligation; no fixture" rather than inventing a runtime test for it.
- AC (b)'s 3 × 8 = 24-fixture grid spans obligations across three sub-steps (1, 4, 5). The back-reference should cite the *Independence from sub-step 4* / sub-step-1 / sub-step-5 invariants jointly; do not collapse them into a single REQ-ID that elides the sub-step-1 and sub-step-5 analogues the AC's own prose pins.

## Consequence

**Severity:** advisory

A test author writing the V9l-T fixtures cannot determine from an AC alone which spec rule the fixture asserts; they must scroll back through the ~80 lines of preceding normative prose and pattern-match on inline phrases ("state-independent write", "Per-step isolation", "interstitial-seam swallow rule"). When a fixture fails, the operator cannot map the failure to a specific obligation without the same manual lookup. The risk is low-frequency drift: when a rule is reworded but its tests continue to pass, the silent decoupling can persist undetected because no machinery checks AC ↔ rule alignment.
