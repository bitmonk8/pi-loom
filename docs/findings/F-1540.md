## Finding

CTRL-1 (`docs/spec_topics/control-flow.md`) states that a `for ... in` iterand expression "MUST be evaluated exactly once, at loop entry, before the first iteration," and explicitly extends the once-only commitment to the degenerate case: where the iterand carries effects (a function-call iterand, an `@`-query iterand, or an `invoke` child iterand), "that effect commits exactly once at loop entry — **including when the resulting array is empty and the body never runs**." The empty-array branch is therefore a named normative obligation, not an inferred consequence.

The spec pins this obligation in prose only; it supplies no test vector that distinguishes a conforming implementation from a non-conforming one. The distinguishing behaviour is subtle: an implementation that short-circuits iterand evaluation when the snapshot is empty (treating an empty result as "no loop, nothing to evaluate") never runs the loop body — and neither does a conforming implementation. The two are observably identical *except* for the side effect of the iterand expression itself. Only an effectful iterand whose effect is observed after a zero-iteration loop separates the conforming case from the violating one, and nothing in the spec corpus exercises that path.

This is the spec's own established convention working against it: the diagnostics placeholder-rendering subsection (`docs/spec_topics/diagnostics/placeholder-rendering-a.md`, `placeholder-rendering-b.md`) carries normative test vectors and boundary cases precisely so that "two conformant implementations produce byte-identical strings." CTRL-1's empty-array effect-commit is exactly the kind of subtle, easily-skipped edge that warrants the same treatment but does not receive it.

## Solution Space

**Shape:** single

### Recommendation

Add a normative test vector to CTRL-1 in `docs/spec_topics/control-flow.md` pinning the empty-array effect-commit: a `for x in f()` loop whose iterand `f()` returns `[]` and has an observable side effect (e.g. increments a `let mut` counter). After the loop completes — zero iterations — the counter MUST equal exactly 1, demonstrating that the iterand committed its effect once despite the empty snapshot. Phrase the vector so it isolates iterand evaluation from body execution (the body must be capable of incrementing the same counter, so that a conforming run shows `1` and a body-driven run would show `0`), making the short-circuit-on-empty implementation observably non-conforming.

Edge cases the implementer must keep visible: the vector must use an effectful *iterand expression* (function call / `@`-query / `invoke` child), not an effect inside the body, since only the iterand path is at issue; and it should sit alongside, not replace, the existing non-empty once-only and `let mut`-reassignment guarantees CTRL-1 already describes.

## Consequence

**Severity:** advisory

CTRL-1's prose is unambiguous, so two careful implementers reading it would not diverge on the required behaviour; the gap is that the spec offers no vector to *prove* conformance, leaving the subtle empty-array effect-commit path unguarded against a plausible short-circuit optimisation. The paired plan leaves `V3c` / `V3c-T` already mandate a test that "proves an effectful empty-array iterand commits exactly once," so the implementation path is in practice covered — but the spec corpus itself remains the weaker artefact, pinning the obligation in prose while delegating the only conformance evidence to the plan.
