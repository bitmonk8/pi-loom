## Finding

The hub paragraph at the top of `docs/spec_topics/governance.md` enumerates the live and retired GOV-rule sets inline:

> the rules (GOV-1, GOV-3 through GOV-9, GOV-15, GOV-16, GOV-30, GOV-31; GOV-2, GOV-10, GOV-11, GOV-12, GOV-13, GOV-14 retired — see [Retired REQ-IDs](./governance/anchor-scheme-and-retired.md#retired-req-ids))

The canonical Retired REQ-IDs table at `governance/anchor-scheme-and-retired.md#retired-req-ids` lists eight retired IDs — `GOV-2`, `GOV-10`, `GOV-11`, `GOV-13`, `GOV-4` (retired in `1d773d6`, replaced by GOV-24), `GOV-21` (retired in `8e3ccb4`, replaced by GOV-25–GOV-29), `GOV-12`, and `GOV-14`. The hub paragraph omits both `GOV-4` and `GOV-21` from its retired set, and the inclusive range "GOV-3 through GOV-9" positively asserts `GOV-4` is active — directly contradicting the canonical table.

The hub paragraph also asserts a "see [Retired REQ-IDs]" pointer immediately after the stale list, so the reader sees the inline enumeration first and the contradicting table only on a follow-link, which is the worst arrangement for a self-contradicting spec.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `1d773d6` (2026-06-04, retired GOV-4); `8e3ccb4` (2026-06-05, retired GOV-21)
**History:**

`git log -L 5,5:docs/spec_topics/governance.md` shows the hub paragraph has been touched five times since its current shape was set in `64cdc60` (2026-05-26), and every edit since has only added new live IDs (`GOV-15`, `GOV-16`, `GOV-30`, `GOV-31`) or moved an ID between live and retired within the closed set `{GOV-2, GOV-10, GOV-11, GOV-12, GOV-13, GOV-14}` — most recently `5a772e3` (2026-06-05) which moved GOV-14 to retired.

`git show 1d773d6 --stat` reveals the retiring commit for GOV-4 edited `governance/anchor-scheme-and-retired.md`, `governance/req-id-prefix-table-active-a.md`, `governance/req-id-prefix-table-active-b.md`, `governance/stable-inline-labels.md`, and `hard-ceilings.md` — but did **not** touch `governance.md` or `session-model-and-appendix.md`. `git show 8e3ccb4 --stat` for GOV-21 likewise edited only `anchor-scheme-and-retired.md` and `release-version-naming.md`. Neither retirement commit propagated its effect into the hub paragraph's inline enumeration or the session-model appendix bullet that mirrors it.

The drift is therefore the joint residue of two independent retirement commits, each of which correctly updated the canonical table but left the editorial restatement on the hub page (and its appendix mirror) untouched. The current `session-model-and-appendix.md` was created at `81bf52a` (2026-06-11) by mechanical move from the pre-existing `spec/session-model-and-appendix.md`, inheriting the already-stale enumeration.

## Solution Space

**Shape:** single

### Recommendation

Update both editorial restatements to match the canonical Retired REQ-IDs table, and remove the inline retired enumeration from the hub paragraph and the appendix bullet in favour of a pointer to the canonical table, so future retirements cannot reintroduce the same drift.

Concrete edits:

1. On `docs/spec_topics/governance.md` line 5, replace the parenthetical inline enumeration with a form that does not list retired IDs and does not pretend to enumerate every live `GOV-N`. A minimal shape:

   > the rules listed in the [REQ-ID prefix table](./governance/req-id-prefix-table-active-a.md#req-id-prefix-table) under the `GOV` row (live IDs; see also [Retired REQ-IDs](./governance/anchor-scheme-and-retired.md#retired-req-ids) for the retired set)

   Either drop the inline live enumeration entirely as above, or — if the enumeration is editorially load-bearing — include every currently-live `GOV-N` (`GOV-1, GOV-3, GOV-5 … GOV-9, GOV-15 … GOV-20, GOV-22 … GOV-31`) and every currently-retired `GOV-N` (`GOV-2, GOV-4, GOV-10, GOV-11, GOV-12, GOV-13, GOV-14, GOV-21`), with explicit calls to `GOV-5 … GOV-9` rather than the misleading `GOV-3 through GOV-9` (since the range silently lies about GOV-4).

2. Apply the same fix to `docs/spec_topics/session-model-and-appendix.md` line 57 (Appendix → Governance bullet), which carries an identical drift. The simpler of the two shapes above — pointer to the canonical table — is preferable here too, since this is an Appendix entry whose role is wayfinding rather than re-stating governance.

Edge cases the implementer must watch:

- The hub paragraph also independently omits `GOV-15`/`GOV-16`/`GOV-19`/`GOV-20`/`GOV-22`/`GOV-23`/`GOV-24`/`GOV-25`/`GOV-26`/`GOV-27`/`GOV-28`/`GOV-29` from its live enumeration — but those omissions are out of scope for *this* finding (which is about the retired set and the false GOV-4-active assertion only). If the chosen shape is the explicit-enumeration variant rather than the pointer variant, the omitted live IDs surface as collateral that must also be folded into the same edit; if the pointer variant is chosen, the omissions evaporate. The related finding *`GOV-N` entry — live range "through GOV-23" stale* on the glossary covers the same drift pattern from a different page and should be resolved compatibly.
- The fix is purely editorial and does not constitute a `GOV-8` *Pure rewording* concern — the canonical Retired table is the source of truth and is unchanged; the edit only repairs an out-of-sync restatement.

## Consequence

**Severity:** advisory

A maintainer or downstream tool reading the governance hub paragraph receives a stale view of the retired GOV set and a positive (false) assertion that `GOV-4` is live. The canonical table is the source of truth and is correct, so no mechanical gate or runtime behaviour is affected; the harm is editorial — readers must cross-check the hub paragraph against the canonical table to discover the drift, and a downstream document built from the hub paragraph (rather than from the table) silently inherits the wrong retired set.
