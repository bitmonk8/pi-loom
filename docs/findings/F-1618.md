## Problem

The "Why the transcript bytes are pinned" sub-section of `binder-model-and-context.md` asserts an input-reproducibility contract: the same session must construct byte-identical binder input across runs and processes, which is the basis for the BNDR-7 reproduce-exactly pins. Those bytes are a function of the included-turn set chosen by the session-context truncation walk, which selects turns by summing per-message counts from the `TokenEstimator` seam (PIC-16), delegating in production to the `estimateTokens` import. Nothing in the spec asserts that production `TokenEstimator.estimate(message)` is a deterministic pure function of its `message` argument; PIC-16 only acknowledges that Pi's heuristic is version-nondeterministic, and the `estimateTokens` entry says Pi owns the algorithm and does not re-specify it. A production estimator whose counts vary across calls on identical input — or depend on hidden mutable state, time, or randomness — silently shifts the included-turn boundary and therefore the transcript bytes, defeating the input-reproducibility claim without violating any stated rule.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `8f7266c` (2026-06-03, "spec: resolve 'Binder Determinism reproducibility scope overreaches provider contract'"), `1f37b15` (2026-06-05, "spec: resolve 'Session-context truncation — no TokenEstimator DI seam for estimateTokens'")
**History:** `8f7266c` re-scoped the binder-determinism prose from a provider-side reproducibility claim to an *input-reproducibility* contract pinning the prompt bytes loom constructs, while explicitly disclaiming provider sampling. `1f37b15` then introduced the PIC-16 `TokenEstimator` seam and split production (`PiTokenEstimator` → `estimateTokens`) from tests (`FakeTokenEstimator`), motivating the seam as defending conformance tests against "Pi's version-nondeterministic estimation heuristic" — a phrasing that names the variability `loom` is exposed to but stops short of constraining it. The defect emerged at the interaction of these two commits: the input-reproducibility contract names the bytes it owns but does not name the per-message-determinism precondition the truncation walk silently relies on, and the seam definition names test-side determinism but does not assert the production adapter's input-determinism over the message argument. Subsequent commits (`05e84dd`, `f088698e`, `97ab51e`, `0ffc66e`) refined adjacent contracts (`.messages` ordering, BNDR-7 oracles, BNDR-8 canonical JSON, presupposition relocation) without revisiting this gap.

## Solution approach

Add a behavioural precondition at the PIC-16 `TokenEstimator` seam (`#pic-16` / `#tokenestimator-interface`) pinning the production `TokenEstimator.estimate(message)` (and the `estimateTokens` import it delegates to) as a deterministic pure function of its `message` argument at a fixed Pi-SDK pin. Add a forward-link from the input-reproducibility prose in `binder-model-and-context.md`'s "Why the transcript bytes are pinned" to that precondition. Add a host-presupposition register row at `host-prerequisites.md` mirroring the `.messages` chronological-ordering and leading-`user` presuppositions, and add a corresponding re-audit item to the Pi version-bump editorial checklist in `version-bump-step2.md`.

## Solution constraints

- The precondition binds per-message determinism only at a single fixed Pi-SDK pin; it must not assert stability of per-message counts across Pi minor bumps (Pi owns the estimation algorithm).
