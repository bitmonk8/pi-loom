## Finding

The `details` shape conventions paragraph for the `session_shutdown`
event payload in `diagnostic-shape.md` (§`session-shutdown-details-conventions`)
and the `loom/runtime/cancelled-by-session-shutdown` row in
`code-registry-runtime.md` together carry at least six normative
statements directed at parties the loom runtime does not own:

- *"consumers MUST validate the field against the per-code value space"*;
- *"operator-tooling consumers MUST whitelist `details.event.loom` on
  the `loom/runtime/cancelled-by-session-shutdown` code when applying
  the strict-validation entitlement"*;
- *"Consumers MUST treat the construction-site fallback
  `details.event.reason` value on either row as a separate value-space
  outside the closed `{"new", "resume", "fork"}` set"*;
- *"Consumers MUST treat fallback-path emissions as a separate wire
  shape from the structured emission and MUST NOT validate them
  against the structured presence-entitlement above"*;
- *"consumers MUST first discriminate the path before asserting
  presence and MUST key the construction-site-throw read on the second
  token"* (twice, once each for `details.event.loom` and
  `details.event.invocation_id`);
- *"consumers MUST first discriminate the path before asserting
  presence and MUST treat fallback-path emissions as carrying no
  observable `details.event.invocation_id` discriminator"*.

The runtime cannot observe whether an external consumer obeys any of
these — the loom emits the same wire bytes regardless of how the
operator's log pipeline parses them. They are advisory framing
masquerading as normative MUSTs, which both inflates the MUST surface
the spec must defend and makes the rules unfalsifiable from the
implementation side.

The same paragraph then embeds an implementation-guidance sentence
about test code: *"a test that asserts `details.event.loom` presence
without first discriminating the structured-emission path from the
two fallback paths over-fails on every multi-invocation shutdown
whose per-invocation notes route through either fallback"* (and the
matching sentence for `details.event.invocation_id`). Test-writing
advice belongs in a test-design note, not in the spec paragraph
defining the wire shape.

The producer-side facts the paragraph carries are correct and worth
keeping — the structured-emission shape (`details.event = { reason,
loom, invocation_id }` for the per-invocation note, `{ reason }` for
the runtime-degraded row), the two-token serialiser-throw fallback
`` `${code} ${detailsEventReason}` ``, and the three-token
construction-site-throw fallback
`` `${code} ${entry.loom} <unreadable>` `` — but the wrapping
"consumers MUST" framing dilutes that signal and conflates wire
contract with consumer contract.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `52b9101` (2026-05-11, "T24 — Fork-reason
watcher closure"), `63bddcd` (2026-05-22, cluster "T19a
invocation_id correlation key")
**History:** `git log -S` against each of the offending phrases
(`'Consumers MUST treat the construction'`, `'consumers MUST first
discriminate'`, `'operator-tooling consumers MUST whitelist'`,
`'tests are entitled to assert its presence'`, `'a test that
asserts'`) shows the consumer-directed MUST surface accreted
across two resolution passes: the T24 commit introduced the
session-only degraded-state branch and laid down the first
*"Consumers MUST treat the construction-site fallback …"* / *"Consumers
MUST treat fallback-path emissions …"* / *"operator-tooling consumers
MUST whitelist `details.event.loom` …"* / *"consumers MUST first
discriminate the path …"* clauses (plus the embedded *"a test that
asserts `details.event.loom` presence …"* sentence) when it added
the `runtime-degraded` row and the runtime-constructed sibling
carve-out for `details.event.loom`; the T19a-cluster commit then
duplicated the same pattern when it added the
`details.event.invocation_id` runtime-constructed sibling — repeating
*"tests are entitled to assert its presence"* and adding the
parallel *"consumers MUST first discriminate the path before
asserting presence"* / *"consumers MUST treat fallback-path emissions
as carrying no observable `details.event.invocation_id`
discriminator"* clauses, and extending the test-implementation
sentence to cover `invocation_id`. The June 2026 `f5e89f4`
re-shard split the per-page text out of the prior monolithic
`diagnostics.md` but did not change any of these clauses. The
defect is therefore the interaction of two *individually* defensible
resolutions that each chose the same flawed framing (consumer-side
MUST + embedded test-implementation guidance), not a single
introducing edit; the durable fix is at the paragraph level, not at
either originating commit.

## Solution Space

**Shape:** single

### Recommendation

Rewrite each consumer-directed obligation as a non-normative
contract note that documents what an operator-tooling consumer
*should* expect, then strip the embedded test-design sentences.
Concretely, in both files:

1. Replace the four `Consumers MUST …` / `consumers MUST …`
   sentences (the value-space-per-code MUST, the fallback-as-separate-
   wire-shape MUST/MUST NOT pair, and the two path-discrimination
   MUSTs in `code-registry-runtime.md`) with a single italicised
   *Consumer contract.* note that states the same facts descriptively
   ("Operator-tooling consumers parsing across the two codes encounter
   per-code value spaces — `loom/runtime/cancelled-by-session-shutdown`
   carries the wider `"quit"|"reload"|"new"|"resume"|"fork"|string|"<unreadable>"`
   union; `loom/host/session-shutdown-runtime-degraded` carries
   only `{"new","resume","fork"}` — and a separate wire shape on
   each of the two fallback paths …"). The note is descriptive of
   the producer's behaviour, not prescriptive of the consumer's.
2. Replace *"operator-tooling consumers MUST whitelist
   `details.event.loom` …"* (and the analogous future
   `invocation_id` whitelist obligation) with a *Consumer contract.*
   note that lists the runtime-constructed siblings consumers will
   observe on each code, so an operator that chooses to apply a
   closed-shape validator has the data it needs to whitelist them
   without being told it MUST.
3. Delete both *"a test that asserts `details.event.loom` …"* /
   *"a test asserting [`details.event.invocation_id`] presence …"*
   sentences outright. The producer-side fact those sentences exist
   to defend — that neither fallback path carries the field — is
   already stated in the surrounding sentences; the test-writing
   advice belongs in the leaf's test guidance, not in the spec.
4. Keep every producer-side MUST untouched: the *Runtime construction
   obligation* paragraph's `{ reason }` / `{ reason, loom,
   invocation_id }` field set, the spread-/`Object.assign`-/key-loop
   prohibition (subject to a separate finding), the two-token /
   three-token fallback wire-shape pins, the `"<unreadable>"`
   sentinel substitution rule, and the *"tests are entitled to assert
   on the specific code at every documented diagnostic site"*
   sentence in DIAG-1 (a producer-side entitlement, not a
   consumer-side MUST). The narrow per-row *"tests are entitled to
   assert its presence"* sentences attached to `details.event.loom`
   and `details.event.invocation_id` may stay or fold into the
   *Consumer contract.* note — either is fine, since they restate
   the producer-side presence guarantee rather than impose a
   consumer obligation.

Edge cases the implementer must watch:

- Do not silently weaken the `details.event` field-set closure
  while editing — the *"future SDK additions require a spec edit
  before they are permitted on the wire"* obligation is a
  producer-side MUST and stays.
- The two `MUST NOT` clauses (*"MUST NOT validate them against the
  structured presence-entitlement"* and the implicit "MUST NOT key
  on a structured `details.event.loom` field on construction-site-
  throw fallback") become *"will under-validate if …"* descriptive
  cautions in the *Consumer contract.* note, not deletions; the
  facts they encode are part of the wire contract operators rely on.

## Consequence

**Severity:** advisory

The unfalsifiable MUSTs do not block implementation — a runtime that
emits the documented wire shape satisfies the producer-side
obligation regardless of how the consumer MUSTs are phrased — but
they pollute the spec's normative surface (every MUST is something an
audit must defend, a test must cover, and a future contributor must
not silently weaken). The embedded test-writing sentences additionally
encourage readers to treat the spec as a test-design document, which
muddles ownership of the test suite.
