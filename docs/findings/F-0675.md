## Problem

The `details` shape conventions paragraph for the `session_shutdown`
event payload in `diagnostic-shape.md` (anchor
`session-shutdown-details-conventions`) and the matching prose for the
`loom/runtime/cancelled-by-session-shutdown` row in
`code-registry-runtime.md` carry several normative MUSTs directed at
external operator-tooling consumers — per-code value-space validation,
fallback-path discrimination, and `details.event.loom` whitelisting. The
loom runtime cannot observe whether any external consumer obeys these; it
emits the same wire bytes regardless of how a consumer parses them, so
the obligations are unfalsifiable from the implementation side and
inflate the spec's normative MUST surface. The same paragraph also
embeds test-implementation guidance (e.g. "a test that asserts
`details.event.loom` presence … over-fails …") that belongs in
test-design notes, not in the wire-shape definition.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `52b9101` (2026-05-11, "T24 — Fork-reason
watcher closure"), `63bddcd` (2026-05-22, cluster "T19a
invocation_id correlation key")
**History:** `git log -S` against each of the offending phrases
(`'Consumers MUST treat the construction'`, `'consumers MUST first
discriminate'`, `'operator-tooling consumers MUST whitelist'`,
`'tests are entitled to assert its presence'`, `'a test that
asserts'`) shows the consumer-directed MUST surface accreted
across two resolution passes: the T24 commit introduced the
session-only degraded-state branch and laid down the first
*"Consumers MUST treat the construction-site fallback …"* / *"Consumers
MUST treat fallback-path emissions …"* / *"operator-tooling consumers
MUST whitelist `details.event.loom` …"* / *"consumers MUST first
discriminate the path …"* clauses (plus the embedded *"a test that
asserts `details.event.loom` presence …"* sentence) when it added
the `runtime-degraded` row and the runtime-constructed sibling
carve-out for `details.event.loom`; the T19a-cluster commit then
duplicated the same pattern when it added the
`details.event.invocation_id` runtime-constructed sibling — repeating
*"tests are entitled to assert its presence"* and adding the
parallel *"consumers MUST first discriminate the path before
asserting presence"* / *"consumers MUST treat fallback-path emissions
as carrying no observable `details.event.invocation_id`
discriminator"* clauses, and extending the test-implementation
sentence to cover `invocation_id`. The June 2026 `f5e89f4`
re-shard split the per-page text out of the prior monolithic
`diagnostics.md` but did not change any of these clauses. The
defect is therefore the interaction of two *individually* defensible
resolutions that each chose the same flawed framing (consumer-side
MUST + embedded test-implementation guidance), not a single
introducing edit; the durable fix is at the paragraph level, not at
either originating commit.

## Solution approach

Demote the consumer-directed `MUST` / `MUST NOT` clauses in the
`session-shutdown-details-conventions` paragraph and the
`loom/runtime/cancelled-by-session-shutdown` row into descriptive
consumer-contract notes that state the same producer-observable facts
without imposing obligations the runtime cannot observe. Delete the
embedded test-implementation sentences describing how a test asserting
`details.event.loom` / `details.event.invocation_id` presence over-fails
on fallback paths.

## Solution constraints

- Out of scope: the producer-side MUSTs (the `{reason}` field-set
  closure, the *Runtime construction obligation*, the two-/three-token
  fallback wire-shape pins, and the future-SDK-addition obligation)
  remain normative and unchanged.
- Out of scope: the `details.event` construction-mechanism prohibition
  (no `...event` spread / `Object.assign` / per-key loop), owned by the
  same-cluster construction-over-prescription finding.
