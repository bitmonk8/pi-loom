## Finding

The diagnostic code `loom/load/cross-format-collision` is registered against a Trigger that explicitly covers two arms: a *same-format* arm (two `.loom` files at the same priority — two packages, two `loomPaths` settings entries, or two `--loom` components) and a *cross-format* arm (`.loom` vs Pi-owned `.md` prompt / `.md` skill / another extension's command). The code name names only the second arm. Discovery's owning rule is even broader: `discovery-sources.md` DISC-4 is titled "Slash-name collision rules" without the `cross-format` qualifier.

The current spec attempts to defuse the mismatch with one sentence inside the code's Trigger cell ("The code name retains the historical `cross-format` token as a stable diagnostics-contract identifier per DIAG-3; its scope is the full same-priority slash-name collision rule, not only the cross-format arm."). That gloss is reachable only by readers who open the registry row in full — it does not appear at the namespaces / naming-convention block where new contributors will first encounter the `cross-format` token, and it does not appear at DISC-4 itself. A reader who sees `loom/load/cross-format-collision` fired against two `.loom` files at the same priority will reasonably suspect a defect, and a contributor adding a new diagnostic site will not know that the code's *name* and the code's *scope* have intentionally diverged.

DIAG-3 (codes are stable identifiers) correctly forecloses renaming the code inside loom 1.x — the existing trigger-column note is the right disposition for the rename question. What is missing is a single discoverable place outside the trigger cell that explains the name/scope gap, so the gloss is reachable from the namespaces block, from DISC-4, and from anyone searching for "cross-format" in the corpus.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `d124b8c` (2026-05-05, "spec: same-priority slash-name collisions are uniform load-time errors (Decision #10)")
**History:** The code was authored at the original cross-format-only scope in commit `c5c4e79` (2026-05-04, "pi-loom spec: resolve 'Cross-format slash collision rule cannot achieve neither is registered'"), where its Trigger genuinely was cross-format only ("A `.loom` candidate's slash name collides with an already-registered Pi prompt template, subagent, or other extension's command"). The name and the scope agreed at that point. `d124b8c` then broadened the diagnostic to cover same-priority same-format collisions ("two packages, two settings entries, two `--loom` components") while explicitly noting in the commit message "no new code introduced" — at this commit the registered code name `cross-format-collision` ceased to name its own scope, and DISC-4 lost the `across formats` qualifier in `discovery-sources.md`. A later commit `7e440c9` (2026-06-06, "pi-loom spec: resolve 'cross-format-collision misnomer / DISC-4 heading scope'") attempted to close the gap by adding the DIAG-3 / loom-2.0-rename gloss inline to the Trigger cell, but the gloss reaches only readers who open the row; the present finding is the residual gap that mitigation left behind.

## Solution Space

**Shape:** single

### Recommendation

Promote the existing inline Trigger-column gloss into a single named paragraph in `diagnostic-shape.md`'s code-namespaces block, and forward-link it from both the registry row and from DISC-4. Concretely:

1. In `docs/spec_topics/diagnostics/diagnostic-shape.md`, add a short paragraph (or stable-anchored note) immediately after the existing `loom/load/*` namespace bullet — or alongside the *Naming convention* sentence below the namespaces list — that names the one stable-name carve-out: "`loom/load/cross-format-collision` is the registered identifier for the full DISC-4 same-priority slash-name collision rule (both same-format and cross-format arms); the `cross-format` token is retained as a stable diagnostics-contract identifier per DIAG-3 and a rename to `slash-name-collision` is deferred to loom 2.0 migration." Give the paragraph a stable HTML anchor so future references can cite it.
2. In `docs/spec_topics/diagnostics/code-registry-load.md`, replace the inline Trigger-column rationale clause with a one-clause forward-link to the new anchor in (1). Keep the Trigger column focused on what fires the code; defer the naming carve-out to the dedicated note.
3. In `docs/spec_topics/discovery/discovery-sources.md`, add a parenthetical at the DISC-4 opening (or at the `loom/load/cross-format-collision` mention inside DISC-4) that forward-links the new anchor — so a reader landing on DISC-4 from any direction reaches the name/scope gloss without round-tripping through the registry row.

Edge cases the implementer must watch:

- The registered code identifier `loom/load/cross-format-collision` MUST NOT change; DIAG-3 is the binding constraint and the existing `V7b-T` `DIAG-3` test would correctly fail on a rename.
- The Message template `slash name '<name>' collides at the same priority: <paths>` already names the broader scope correctly and MUST NOT be reworded under DIAG-4.
- The relocation of the gloss is a structural move, not a content change — the canonical text is the existing trigger-cell sentence; copy it forward so reviewers can confirm by inspection that the diagnostics contract is unchanged.

## Consequence

**Severity:** advisory

A contributor or downstream tooling author who searches the corpus for `cross-format-collision` lands on the code's row (gloss reachable) but not on the namespace block or on DISC-4; the name/scope gap stays opaque outside the row. No author-visible behaviour diverges — the registered code, the Trigger, the Message, and the DISC-4 rule are all internally consistent — so the cost is reviewer / maintainer surprise rather than divergent implementations. The DIAG-3 stability pin and the deferred loom-2.0 rename make this an editorial rather than a normative gap.
