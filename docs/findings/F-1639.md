## Finding

Two normative statements describe how a contributor resolves a family-(4) violation (an out-of-scope import/access shape), and they do not agree.

The *Failure-surface contract* family→step-2(b) routing table (`audit-failures.md`, `#audit-failure-surface-contract`) gives the family-(4) row three permitted arms: **branch (1) [delete]**, **branch (2) [promote]**, and **branch (5) [rewrite-shape]**, with an explicit exclusion that "branch (3) exempt MUST NOT appear here per the *Exemption mechanism* family (4) prohibition."

The *Exemption mechanism* prose (`audit-resolution.md`, `#audit-exemption-mechanism`) instead says: "a contributor confronted with such a shape MUST rewrite the source line into a recognised shape … **rather than annotate it, per step 2(b) branch (5)**." Read literally, this MUST narrows family-(4) resolution to a single arm — the branch-(5) rewrite — and does not accommodate the routing table's branch (1) [delete] (which removes the offending line outright, producing no recognised shape) or branch (2) [promote] (which adds an inventory entry, impossible for a *shape* until after a rewrite). The phrase is ambiguous between (a) "branch (5) is the only permitted non-exempt source change" (which contradicts the table) and (b) "make a source change rather than annotate it" (which is consistent with the table but is not what the sentence states).

The inconsistency compounds with branch (5)'s own definition. Branch (5) (`version-bump-step2b.md`, `#bump-step-2b-rewrite-shape`) says to rewrite the offending line into a recognised shape "and resolve the rewritten reference via **branch 1, 2, or 3** above as that branch's criterion fits." So after a branch-(5) rewrite, the rewritten reference may legitimately be exempted via branch (3). If the family-(4) row's branches (1) and (2) are meant to be the *terminal* arms reached through branch (5), then branch (3) should appear on the row too (branch (5) permits it); if instead branches (1) and (2) *directly* resolve a family-(4) red without a rewrite, then they are direct arms and branch (5)'s follow-on resolution is a separate matter. The spec mixes both models: it lists branches (1)/(2) on the row but asymmetrically excludes (3), and the prose names only branch (5). A contributor or reviewer cannot determine from the text whether a family-(4) line may be deleted directly (branch 1) or must first be rewritten (branch 5), nor why branch (2) is admitted on the family-(4) row while branch (3) is forbidden when both are only reachable after a rewrite.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commits:** `e5cd514` ("pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\"", 2026-05-09)
**History:** The five-family *Failure-surface contract* and the five-branch step-2(b) resolution set were authored together in a single commit, `e5cd514`, which introduced the closed-set negative-direction audit. `git log -S 'family (5)'`, `git log -S 'branch (5)'`, and `git log -S 'Resolve by exactly one of the following five branches'` (all `--reverse`, over `docs/**`) each first appear in `e5cd514`; no predecessor revision carried either ordinal space. The overload was therefore present from the feature's inception — both 1–5 spaces were coined with the same `(N)` notation in the same change and never disambiguated — rather than drifting in via a later edit. The later `81bf52a` page-collapse and `23d28a1` basename realignment relocated the prose but did not introduce or alter the notation collision.

## Solution Space

**Shape:** multiple

### Option A — Trim the family-(4) row to its direct arms; reconcile the prose

**Approach:** Treat the routing table as "arms that directly resolve *this family's* red." A family-(4) shape is resolved either by deleting the offending line outright (branch 1, when the runtime does not depend on the surface) or by rewriting it into a recognised shape (branch 5). Remove branch (2) from the family-(4) row, because a *shape* cannot be promoted to inventory directly — promotion only applies to the recognised-shape reference that a branch-(5) rewrite produces, which is a separate (family 1/2/3) red resolved on its own row. Soften the *Exemption-mechanism* prose from "MUST rewrite … per branch (5)" to "MUST resolve by a source change — delete the line (branch 1) or rewrite it into a recognised shape (branch 5) — rather than annotate it; branch (3) exempt never applies to a family-(4) line."
**Spec edits:** `audit-failures.md` family-(4) row → `branch (1) [delete], branch (5) [rewrite-shape]` (drop branch (2); keep the branch-(3) exclusion note). `audit-resolution.md` *Exemption mechanism* → reword the MUST to admit branch (1) and frame branch (5) as the rewrite path. Add one sentence to branch (5) in `version-bump-step2b.md` clarifying that its follow-on branch-1/2/3 resolution resolves the *rewritten* reference's own red, recorded against that reference's family row, not the family-(4) row.
**Pros:** Each routing-table row means exactly one thing (arms that resolve that family's red); the branch-(3) exclusion becomes consistent (neither (2) nor (3) is a direct family-(4) arm); the leaf-side test invariant has an unambiguous arm set.
**Cons:** Contributors must understand that promote/exempt of a rewritten reference is tracked as a separate family-(1)/(2)/(3) red, not as a family-(4) resolution.
**Risks:** The land-green sweep wording in `V18b` ("promoted … or rewritten") must be re-read so it does not imply direct promotion of a shape.

### Option B — Make the family-(4) row the transitive closure of reachable arms

**Approach:** Treat the routing table row as "every arm a contributor may touch while fixing a family-(4) violation." Keep branches (1), (2), (5) and **add branch (3)**, since branch (5) explicitly permits resolving the rewritten reference via branch 1, 2, *or* 3. Replace the absolute "branch (3) MUST NOT appear here" exclusion with a note that branch (3) is reachable only as branch (5)'s post-rewrite follow-on (never as a marker on the original family-(4) line). Reword the prose so "MUST rewrite … per branch (5)" reads as "the family-(4) line itself is non-exemptible; resolve it via the branches the row lists."
**Spec edits:** `audit-failures.md` family-(4) row → `branch (1), branch (2), branch (3), branch (5)`, with the exclusion note rewritten to "branch (3) applies only to the recognised-shape reference produced by a branch-(5) rewrite, never to the family-(4) line directly." `audit-resolution.md` prose reworded to drop the single-arm MUST.
**Pros:** Matches branch (5)'s "resolve via branch 1, 2, or 3" wording directly; no arm a contributor can legitimately reach is omitted from the row.
**Cons:** Collapses the per-family routability distinction — the row no longer says which arm resolves the family-(4) red itself versus a downstream red; conflicts with the dual-emission model elsewhere, where a rewritten reference's red is a *separate* record on its own family row.
**Risks:** Weakens the "one record routes to one family's arm set" property the *per-discriminator routability* obligation pins; the leaf-side test invariant becomes harder to assert cleanly.

### Recommendation

Option A. It keeps each routing-table row meaning "arms that resolve *this* family's red," which is the property the per-`<symptom>` fixture floor and the leaf-side routing-table test invariant depend on, and it removes the asymmetry by recognising that neither promote (branch 2) nor exempt (branch 3) is a *direct* family-(4) arm — both act on the recognised-shape reference a branch-(5) rewrite produces, which is a separate family-(1)/(2)/(3) red. Edge case the implementer must watch: a family-(4) shape that the runtime does depend on cannot be deleted; it must go through branch (5), and the rewritten reference's promote/exempt is then recorded against its own family row, co-committed per the existing dual-emission/co-commit discipline.

## Consequence

**Severity:** correctness

Two reasonable contributors fixing the same family-(4) violation would diverge: one reads the prose's MUST and rewrites the line into a recognised shape (branch 5); another reads the routing table and deletes the line outright (branch 1) or promotes after rewriting (branch 2). Reviewers checking branch-fit on the bump-commit diff, and implementers pinning the leaf-side family→step-2(b) routing table as a permanent test invariant, have no single authoritative arm set for family (4) — and the asymmetric exclusion of branch (3) (while branches 1/2 are admitted) cannot be justified from the text.
