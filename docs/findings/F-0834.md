## Finding

Items `(aa)` (`complete()` retry-timing), `(ab)` (`complete()` `options.signal` cancellation), and `(af)` (`complete()` `onResponse` delivery) each direct the bump contributor to "re-run the per-provider … fixtures against each loom 1.0-supported provider (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`)". The recipe enumerates four named providers and makes conformance hinge on the fixtures clearing for every one of them, but it nowhere states what the fixture run actually exercises — i.e. whether each per-provider run is a corpus replay or a live HTTP call. A reader is left to assume one or the other, and the wording naturally reads as "exercise the real provider", which silently presupposes the contributor holds working credentials, a paid account, and network reach to all four upstream APIs.

The two sibling items `(i)` (overflow-signature wording) and `(u)` (forced-tool behaviour) on the same page already settle the ambiguity the right way: each carries an explicit "fixtures match … against a maintained corpus of captured … bodies in the repo's `npm test` suite rather than against live provider responses; … re-capturing … is the responsibility of the contributor performing the bump" clause. `(aa)`, `(ab)`, and `(af)` were authored after that pattern was established (commits `3683345` and `44782e9`, both 2026-06-06, both later than `2ecf650` on 2026-06-05) but did not adopt it. The result is that three SHOULD-level checklist items in the same recipe operate under two unstated and incompatible execution models, only one of which (the corpus model) the loom 1.0 contributor obligation actually wants — and the version-bump procedure says nothing about who owns the run when a contributor lacks any one of the four providers.

The downstream effect is a contributor-discipline gate that conformance hinges on but that cannot be cleanly recorded as `pass` / `fail` / `not-performed`. A contributor without an Anthropic key has no defined disposition for `(af)`: `not-performed` is conformant per the preamble, yet the recipe implies the audit was supposed to be runnable, and nothing routes the missing-credential case to a named owner the way `real-host-smoke-gate.md`'s un-runnable-gate disposition does for the live-host smoke.

The defect is the interaction: the pattern existed at the moment the three later items landed, but they were authored independently and ended up with the silent live-credentials presupposition the earlier pair had explicitly dispelled.

## Solution Space

**Shape:** single

### Recommendation

Extend the `2ecf650` corpus-not-live framing to items `(aa)`, `(ab)`, and `(af)` so all five per-provider fixture items in the recipe execute under the same model. In each of the three items, append a sentence parallel to the one items `(i)` and `(u)` already carry: state that the fixtures exercise the loom-side path against a maintained corpus of captured per-provider `complete()` responses checked into the repo, not against live provider calls, and that re-capturing each provider's responses (retry/backoff inter-attempt timing for `(aa)`, `options.signal` cancellation behaviour for `(ab)`, `onResponse` `ProviderResponse { status, headers }` invocations for `(af)`) when the provider publishes an API-version change is the responsibility of the contributor performing the bump. Reuse the existing sibling-item wording verbatim where possible, swapping only the per-item observable; do not introduce a new corpus location, file layout, or owner — the contributor-discipline framing is already pinned for `(i)`/`(u)` and re-stating it twice for `(aa)`/`(ab)` and once for `(af)` keeps the policy single-sourced.

Edge cases the editor must watch:

- Items `(aa)` and `(ab)` cite the same upstream presupposition paragraph at `conversation-drive.md#complete-retry-and-cancellation-presupposition`; the corpus clause goes on each item independently rather than once on the parent presupposition, matching how `(i)`/`(u)` were patched.
- Item `(af)` audits `@earendil-works/pi-ai` (not `pi-coding-agent`); the corpus clause should retain the `pi-ai` package qualifier in the "publishes an API-version change" trigger so it does not silently widen to a pi-coding-agent trigger.
- The corpus that `(aa)`'s timing fixtures replay is not the same shape as `(i)`'s `errorMessage` corpus or `(u)`'s `complete()`-response corpus — timing requires recorded inter-attempt deltas, `(af)` requires recorded `onResponse` invocations and their status/header payloads. The spec only needs to commit to the corpus model; the concrete capture shapes are implementation-owned and ride on `V18c`'s capture-mechanism wiring without spec re-litigation.
- No "who owns the run when credentials are missing" clause is needed once the corpus model is universal across the five items — the live-credentials precondition disappears, and `not-performed` recording stops being the silent escape hatch for missing keys.

## Consequence

**Severity:** advisory

Two reasonable contributors will diverge on what the recipe asks for: one will treat the per-provider re-runs as corpus replays (consistent with siblings `(i)` and `(u)`), the other will read them as live API calls and either skip the audit for lack of credentials or burn account quota on each bump. Either way the audit's `pass` / `not-performed` record carries no signal about which mode was used, so a green commit-message row can mean "regression caught by replay", "no regression observed live", or "audit silently skipped because contributor lacked an Anthropic key" — three materially different facts.
