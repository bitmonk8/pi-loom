## Finding

The bullet enumeration in `binder-inference.md` that says "The runtime populates one `complete(model, context, options)` call per binder attempt as follows" lists `model`, `context.systemPrompt`, `context.messages`, `context.tools`, `options.temperature`, `options.signal`, the seed key, and `options.onResponse` â€” but never mentions `options.maxRetries` or `options.maxRetryDelayMs`. The same section's closing sentence then states that loom "re-issues each budgeted retry immediately, delegating per-attempt backoff and any server-requested wait (e.g. an HTTP `Retry-After`) to this call's `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs`." The closing sentence presupposes loom configures those fields, but the enumeration above â€” which is the only place fields on `options` are pinned for the binder call â€” leaves them unset.

This produces three concrete underspecifications. **(1) Configuration.** Whether loom passes `maxRetries` / `maxRetryDelayMs` at all, and to what numeric values, is undefined; a conformant implementation that omits both options (relying on `@earendil-works/pi-ai`'s defaults, whatever they are at the pinned version) and another that pins them to specific values would both read as conformant. **(2) Budget interaction.** Whether an SDK-internal retry inside a single `complete()` call (the SDK retrying a 5xx under its own `maxRetries`) consumes one of the HC3 transport-class slots â€” i.e. counts towards the "â‰¤3 binder LLM calls per slash invocation" rule pinned in `determinism-cancellation-failure.md#per-invocation-retry-budget` â€” is undefined. A reader can take HC3's "â‰¤3 LLM calls" literally (every wire attempt counts, so `maxRetries` must be 0 to keep the budget honest) or take it loom-level (a `complete()` invocation counts as one attempt regardless of how many wire retries pi-ai performs internally). **(3) Cardinality.** Whether loom issues one outer `complete()` call delegating both budgets to pi-ai, or one outer `complete()` per HC3 slot, is left to the implementer.

The companion paragraph in `conversation-drive.md` ("`complete()` retry-timing and cancellation behavioural presupposition") establishes that pi-ai *can* perform inter-attempt timing when `maxRetries` / `maxRetryDelayMs` are set; it does not establish that loom sets them, nor does it resolve whether SDK-internal retries are counted against the HC3 budget. The pre-`d0bcdb2` version of the binder-inference paragraph said "inter-attempt timing is out of scope here" and deferred entirely; the post-`d0bcdb2` paragraph added the delegation prose but left the option-bullet enumeration unchanged, so the call-shape now contradicts the prose.

## Solution Space

**Shape:** single

### Recommendation

Pin the SDK-level retry contract explicitly in `binder-inference.md`'s option enumeration, treating it as part of the call shape `V9j` implements. Specifically:

1. **Add an `options.maxRetries` bullet** to the "The runtime populates one `complete(model, context, options)` call per binder attempt as follows" list, pinning `options.maxRetries = 0`. Rationale: HC3 already grants exactly one transport-class retry at the loom level (`HC3-a`), and HC3's "â‰¤3 binder LLM calls per slash invocation" cap (`HC3-d`) is most naturally testable when each outer `complete()` invocation corresponds to exactly one wire attempt â€” so the count semantics in `V11f`'s tests stay unambiguous. Setting `maxRetries = 0` also keeps loom's retry policy single-sourced (HC3, not HC3-plus-pi-ai), which matches the existing `determinism-cancellation-failure.md` framing that "the runtime MUST issue each budgeted retry as soon as the preceding failure is classified, with no loom-side delay, backoff, or `Retry-After` wait inserted between a loom-level attempt and its retry."

2. **Add an `options.maxRetryDelayMs` bullet** in the same list, pinning it to whatever pi-ai treats as "no delay" when `maxRetries` is `0` (this is a no-op under `maxRetries = 0` but should be set explicitly so the call shape is fully observable in the `V9j-T` fixtures); the simplest stable choice is to omit the field entirely and add an explicit "loom does not set `options.maxRetryDelayMs`; it has no effect when `options.maxRetries` is `0`" note.

3. **Tighten the closing sentence** of the section so it no longer reads as a partial delegation. Current: "delegating per-attempt backoff and any server-requested wait (e.g. an HTTP `Retry-After`) to this call's `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs`." Replace with prose that states the concrete configuration and its consequence â€” e.g. "Because loom sets `options.maxRetries` to `0` (see the bullet above), pi-ai performs no SDK-internal retry of an underlying wire attempt: each HC3 budget slot corresponds to exactly one outer `complete()` invocation, which corresponds to exactly one wire attempt, and any server-requested wait such as an HTTP `Retry-After` is observed by loom only as the latency of the single failing attempt." This restores the "out of scope here" disposition for SDK-internal *backoff* (there is none) while keeping the version-bump editorial-review hook (item (aa)) for the more general retry-timing presupposition.

4. **Re-state HC3-d's count semantics** in `determinism-cancellation-failure.md` so that "at most 3 binder LLM calls per slash invocation" is defined as "at most 3 outer `complete()` invocations per slash invocation", with a parenthetical noting that loom's `maxRetries = 0` pin makes the outer / wire counts coincide. The current paragraph already says loom owns inter-attempt timing; adding the count-semantics clause closes the residual ambiguity that lets `V11f`'s `HC3-d` test be written two ways.

Implementer edge cases to watch: (a) `complete()` MAY interpret an absent `maxRetries` field as "use pi-ai default" rather than "no retries" â€” so the bullet must pin the field to `0` explicitly, not omit it; (b) the bullet should make clear that `maxRetries = 0` applies on every binder attempt (initial and HC3-budgeted retries), not just the initial call, so that an HC3-level retry does not implicitly inherit a different value; (c) `V9j-T`'s fixture must assert the literal `options.maxRetries === 0` on every captured `complete()` call, not merely "no internal retry observed", because the latter is vacuous on a non-failing fixture.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge on observable behaviour. One sets `options.maxRetries = 0` and relies entirely on the loom-level HC3 budget; another sets `options.maxRetries = 2` (matching the HC3 transport slot) and lets pi-ai do per-attempt backoff. Under transient 5xx pressure these differ in (a) total number of HTTP attempts per slash invocation, (b) total wall-clock latency seen by the user before the system note surfaces, (c) whether HC3-d's "â‰¤3 LLM calls" cap is observed as â‰¤3 outer `complete()` calls or â‰¤3 wire requests, and (d) whether `Retry-After` is honoured at all (it is only honoured by pi-ai when `maxRetries` â‰Ą 1). The `HC3-d` conformance test cannot be written without resolving the call-counting semantics first.
