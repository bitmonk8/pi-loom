## Finding

`frontmatter-fields-a.md` § *tools* fixes the default name for a `.loom` callable entry as "the file's basename without the `.loom` extension, with hyphens replaced by underscores" and then asserts that "loom identifiers must be lowercase-first identifier-shaped". The transformation can demonstrably produce a string that is *not* lowercase-first identifier-shaped per [Lexical — Identifiers](../lexical.md) (`[A-Za-z_][A-Za-z0-9_]*` with first-letter case enforcement). Two realisable cases:

- A digit-first basename such as `2fa.loom`. The discovery filename-validity regex (`^[a-z0-9][a-z0-9_-]*$`) accepts the stem, so `tools: ./2fa.loom` is a legal reference, but the derived default `2fa` is not a valid identifier (must start with a letter or `_`).
- An uppercase-first basename such as `Summarise.loom`. Discovery's slash-name regex would reject this as a top-level discovered loom, but `tools:` accepts any `.loom` path the resolver finds — including paths in subdirectories that discovery never walks (`./shared/Summarise.loom`). The derived default `Summarise` violates the lowercase-first rule.

The diagnostic surface around `tools:` covers only adjacent failures: `loom/load/invalid-tool-rename` is scoped to the explicit `as <name>` target (`"'as <name>' rename target must be lowercase-first; got '<name>'"`), and `loom/load/tool-name-collision` handles same-name collisions. Neither attaches to a *derived* default name that is itself invalid. The spec asserts the invariant ("loom identifiers must be lowercase-first identifier-shaped") without naming any rule or diagnostic that enforces it on the derived case, so two implementers can reasonably diverge: one rejects the entry at load time with an ad-hoc code; another registers the entry and fails opaquely later when bare-identifier resolution encounters a non-identifier name (or — if name-lookup is done by string — succeeds silently and exposes an unreachable callable).

## Issue introduction

**Verdict:** present-since-inception (for the unified `tools:` callable surface)
**Introducing commit:** `7cd9cc6` (2026-05-04) — *"spec: unify tools and loom-as-tool callables under tools:; add ToolCallError"*
**History:** Commit `7cd9cc6` introduced the `.loom` callable entry under `tools:` together with the default-name rule ("basename without the `.loom` extension, with hyphens replaced by underscores") and the lowercase-first invariant. Both clauses entered the corpus in the same edit; the derived-name case was never separately constrained or diagnosed. `git log -S "hyphens replaced by underscores"` against `docs/` and `spec.md` returns `7cd9cc6` as the originating change, and `git log -G "invalid-tool-rename"` confirms that the only later edits affecting the surrounding text (`c8191ce`, `f5e89f4`, `31ff060`) reformat or relocate the rule without touching the missing-diagnostic gap. The defect is therefore co-temporal with the feature, not introduced by a later refactor or interaction between commits.

## Solution Space

**Shape:** single

### Recommendation

Add a load-time rejection for a derived default name that is not loom-identifier-shaped and route it through `loom/load/invalid-tool-rename` (relaxing the code's description to cover both the explicit `as` target and the derived default; rename the code to `loom/load/invalid-tool-name` if the implementer prefers a name that reads neutrally over both triggers). Spell out, in the *tools* sub-paragraph, that the derived name is computed by replacing hyphens with underscores and then validated against the lowercase-first identifier rule from [Lexical — Identifiers](../lexical.md); on failure, the entry is rejected with `loom/load/invalid-tool-rename` and the hint directs the author to add an explicit `as <name>` clause. Examples to pin in prose: `./2fa.loom` (digit-first derived name) and `./shared/Summarise.loom` (uppercase-first derived name).

Update the existing `loom/load/invalid-tool-rename` row in `code-registry-load.md` to name both triggers ("`as <name>` target *or* derived default name from a `.loom` path basename is not loom-identifier-shaped") and broaden the message template to render the offending name regardless of source. Mirror the broadened scope into `V6c-tools-set.md` so the test obligation explicitly covers a derived-default failure (e.g. `tools: ./2fa.loom` rejected at load time) alongside the existing explicit-`as` case.

Do *not* introduce a silent normalisation (digit-prefix stripping, case-folding, automatic `_` prepending). Silent renaming yields a callable name the author never typed, defeats grep-driven authoring, and obscures the same problem at the wire-name layer where the model sees a tool name.

## Consequence

**Severity:** correctness

A `tools: ./2fa.loom` or `tools: ./shared/Summarise.loom` entry can load on one implementation and fail on another, with no spec-named diagnostic determining which behaviour is correct. Authors who hit it have no documented remediation pointer; the gap is small in surface area but reaches into the load-time error catalogue authors test against.
