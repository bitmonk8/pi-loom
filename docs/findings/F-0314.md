## Problem

The §Leaf format **Deps** *Pickability predicate* in `conventions.md` declares that a `Deps.` entry is satisfied "exactly when the §Per-phase TDD ritual completion tag for that entry is present" (`<id>-complete` / `<id>-T-complete`) and calls those git tags "the authoritative pickability signal," but never states *where* the tag must exist for the evaluation to be sound. Git tags are local until pushed and absent from a shallow / no-tags clone until fetched, so two evaluation environments disagree on the same predicate: an unpushed local tag leaves the dep unsatisfied for everyone else, and a fresh/shallow clone reads every completion tag as absent and judges every leaf un-pickable. Because tag presence is presented as authoritative, neither failure surfaces as an error — pickability is silently misread. The gap is the missing visibility precondition, not the tag mechanism, which is itself sound.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `4ee2700` (2026-06-17, "pi-loom plan: resolve \"Deps satisfied build-order step names no machine-checkable pickability predicate\"")
**History:** The `git log -S` walk over `docs/plan_topics/conventions.md` for the strings "authoritative pickability signal", "Pickability predicate", and "`-complete` git tag" returns exactly one commit, `4ee2700`. That commit introduced the entire git-tag-based pickability predicate to resolve a prior review finding (the build-order step previously had no machine-checkable predicate). The visibility-precondition gap is a co-introduced sibling of that enrichment: the moment pickability was keyed on `<id>-complete` / `<id>-T-complete` git tags, the unstated "where the tag must live / pushed / fetched" precondition entered with it. No earlier commit carried a git-tag pickability mechanism, so the defect is not present-since-inception — it dates from `4ee2700`.

## Solution approach

In `conventions.md`, §Leaf format **Deps** *Pickability predicate*, add the tag-visibility precondition adjacent to the "those git tags are the authoritative pickability signal" sentence: state that completion tags are pushed to the shared remote and that pickability is evaluated against a clone that has fetched tags. Cross-reference the §Per-phase TDD ritual phase exit gate ("tag commit `<id>-complete`"), where the push obligation naturally attaches, rather than restating the tagging step.
