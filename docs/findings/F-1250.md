## Finding

`V9i` (and its paired `V9i-T`) carries two qualitatively different concurrency
contracts in a single leaf. PIC-9 is a single-session lifecycle property —
one invocation owns one `AgentSession`, `dispose()` runs in `finally` on every
exit path, dispose is idempotent, and teardown never masks the original
`Err`/`Ok` — verified against a single-session fixture. PIC-22 is an
N=3 parallel-spawn ordering property — all N `createAgentSession` calls
complete and each session's `sendUserMessage` is entered before any blocked
call is released — verified against a distinct blocking-fake `AgentSession`
fixture and requiring the `Promise.all`-over-spawns step that carries the
`// allow: PIC-22` sequential-by-default allow-list comment.

The leaf also folds in the `return.md` final-value contract, the two
`subagent-*` diagnostic codes, the isolated-session assertion, and the ERR-8
delegated live-carrier witness (the live-subagent-cancellation re-assertion
`V4c` defers to `V9i`). The PIC-22 parallel-spawn ordering test and the
ERR-8 live-surface witness exercise concurrency/cancellation behaviour that is
independent of the PIC-9 lifecycle property and is reviewed against separate
fixtures; a reviewer verifying the PIC-9 lifecycle is not well placed to also
audit the PIC-22 interleaving in the same pass.

## Issue introduction

**Verdict:** introduced-by-commit (single)
**Introducing commit:** `7a3b907` (2026-06-10) — "pi-loom plan: resolve 'subagent parallel-initiation MUST has no closing leaf'"
**History:** The PIC-9 lifecycle obligation has lived in `V9i` since the leaf's inception. Commit `7a3b907` attached the PIC-22 parallel-spawn obligation to the existing PIC-9-owning leaf — adding the `Promise.all`-over-spawns parallel-spawn step in `V9i` Adds, the `PIC-22` Tests bullet in both `V9i` and `V9i-T`, and the combined `PIC-9, PIC-22 | V9i` coverage-matrix row — in order to give the subagent parallel-initiation MUST a closing leaf. That single edit created the bundling of two distinct concurrency contracts in one leaf; no earlier or later commit contributed.

## Solution Space

**Shape:** single

### Recommendation

Split the subagent concerns into two paired-leaf sets, preserving behaviour and
coverage:

- Retain the lifecycle obligations on the existing `V9i` / `V9i-T` pair (keeping
  the `V9i` ID for the lifecycle half so downstream `Deps` need no retargeting):
  the `PIC-9` Tests bullet, the `return.md` final-value-contract bullet, the
  isolated-session (no shared transcript or tool table) bullet, and the two
  diagnostic-code bullets (`loom/runtime/subagent-model-unresolved`,
  `loom/runtime/subagent-dispose-failure`). Drop the parallel-spawn step from
  `V9i` Adds.
- Move the parallel-spawn concern to a new paired-leaf set (suggested slug
  `subagent-parallel-spawn`; the implementer allocates the real `V9`-series ID
  under the project's leaf-ID scheme — use the `<new>` placeholder until then).
  The new impl leaf takes the `PIC-22` parallel-spawn step from Adds (including
  the `// allow: PIC-22 — pi-integration-contract/subagent.md` allow-list
  comment, which must travel with the `Promise.all` step so the Sequential-by-
  default rule's `Spec`-cite requirement still holds), the `PIC-22` Tests bullet
  with its blocking-fake fixture, and the `ERR-8` delegated live-carrier witness
  bullet. Its `Deps` list the lifecycle leaf. The matching `<new>-T` leaf takes
  the `PIC-22` (and `ERR-8`) red-first tests.
- Repartition `coverage-matrix.md`: split the `PIC-9, PIC-22 | V9i` row so
  `PIC-9` maps to the lifecycle leaf and `PIC-22` maps to the new parallel-spawn
  leaf; retarget the `V9i` token in the `ERR-8 … ERR-12 | V4c, V9i` row to the
  new parallel-spawn leaf, since the ERR-8 live-carrier witness moves there.
- Add the new leaf to `docs/plan.md` §Vertical slices V9 listing, and update the
  V9/V11 interleave-note seam reference where it names `V9i`/`V9i`-dependents if
  the parallel-spawn split changes the dependency seam.

Edge cases the implementer must watch: the new leaf's ID must be allocated under
the project's leaf-ID scheme (do not ship the `<new>` placeholder); the ERR-8
delegated witness gains a red-first home in the new `<new>-T` leaf (resolving the
related ERR-8 red-first gap below); and the downstream `Deps` on `V9i` (`V11b`,
`V11i`, `V9o` and their `-T` partners) reference the subagent session lifecycle,
so they remain correctly pointed at the retained `V9i` lifecycle leaf and need no
change as long as the lifecycle half keeps the `V9i` ID.

## Consequence

**Severity:** advisory

Both contracts are spec-pinned, so two implementers would not diverge on
behaviour; the cost is review quality. The PIC-22 parallel-spawn ordering and
the ERR-8 live-cancellation witness — each on its own fixture — receive less
scrutiny embedded in a long lifecycle leaf, and the leaf's single Ships-when
gate conflates two independent concurrency properties.
