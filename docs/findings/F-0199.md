## Problem

The `<new>` placeholder convention in `coverage-matrix.md` §*Code-keyed obligation areas (no numbered REQ-IDs)* enters an un-anchored normative MUST whose closing leaf does not yet exist as a one-row mapping listing the literal `<new>` in its closing-leaf column. Neither `conventions.md` §*REQ-ID discipline*, `coverage-matrix.md`, nor `H5a-closing-gate-automation.md` states whether a `<new>` row satisfies the H5a un-anchored-MUST arm's "enumerated with a closing leaf" condition — one reasonable reading greens (a row is present with a non-empty closing-leaf cell), the other reddens (the cell names no real leaf). On the lenient reading an un-anchored MUST represented only by a `<new>` placeholder rides to the loom 1.0 release gate with no real closing leaf and no asserting test, and because the `H5d` transitive-completeness arm explicitly excludes `<new>` rows, no other gate arm catches the gap.

## Issue introduction

**Verdict:** present-since-inception (of the `<new>` placeholder mechanism)
**Introducing commit:** `0603eb4` — "pi-loom plan: resolve \"un-anchored normative MUSTs invisible to closing gate\"" (2026-06-10)
**History:** The plan corpus is git-tracked. `git log -G'<new>' --reverse -- plan_topics/coverage-matrix.md` identifies `0603eb4` as the commit that first introduced the un-anchored-MUST closing-gate surface. `git show 0603eb4 -- plan_topics/coverage-matrix.md` confirms it added both the `<new>` placeholder convention ("a `<new>` placeholder where the obligation is currently uncovered") and the under-specified gate framing ("fails on any such MUST absent from this table with a closing leaf") in the same hunk — so the ambiguity over whether a `<new>` row counts as "with a closing leaf" has existed since the mechanism was created. A later commit `617a3ad` added the `H5d` arm's explicit exclusion of `<new>` rows, which deepened the gap (removing the transitive-completeness arm as a possible backstop) but did not introduce it.

## Solution approach

Make explicit that a `<new>` placeholder row — a row whose closing-leaf column names no real leaf — does not satisfy the "enumerated with a closing leaf" condition on the live-corpus footing, so the H5a un-anchored-MUST arm reddens for any un-anchored MUST whose only coverage-matrix entry is a `<new>` row. Clarify the rule in `conventions.md` §*REQ-ID discipline* and the `<new>` parenthetical in `coverage-matrix.md` §*Code-keyed obligation areas (no numbered REQ-IDs)*. Extend `H5a-closing-gate-automation.md`'s un-anchored-MUST arm coverage with a seeded violation fixture whose un-anchored MUST maps only to a `<new>` placeholder row, mirroring the existing seeded un-enumerated-MUST fixture. Pin the failure to the live-corpus footing activated by `H6a`, so seeded mid-authoring `<new>` placeholders do not redden.

## Solution constraints

- The spec corpus is read-only for this fix; it is internal to the plan's gate machinery.
- Out of scope: `H5d`'s deliberate exclusion of `<new>` rows from the transitive-completeness arm, owned by `H5d`.
