## Problem

The "Roots scanned" enumeration in `package-and-settings.md` obliges the extension, when `npmCommand` is configured, to read a global npm root that "Pi resolves" — instead of the literal `~/.pi/agent/npm/` path — yet names no API, seam, or context field through which the extension can obtain that resolved value. The mechanisms available to the extension are closed and enumerated: the `FileSystem` seam ([PIC-13](../pi-integration-contract/host-interfaces-services.md#pic-13)) returns no npm root, and the `ExtensionContext` member surface is normatively closed and exposes no global-npm-root member. The premise also appears to misread `npmCommand`, which `packages.md` §npm documents as governing only Pi's own npm lookup/install operations while global installs land under `~/.pi/agent/npm/` unconditionally. An implementer reading root #4 literally therefore cannot satisfy it when `npmCommand` is set.

## Issue introduction

**Verdict:** introduced-by-single-commit (fix-induced regression)
**Introducing commits:** `808c72b` — "pi-loom spec: resolve \"npmCommand-resolved npm root presupposes a subprocess seam\"" (2026-06-06)
**History:** The root #4 line was first authored when the spec moved into `docs/` (`31ff060`) and carried through the topic split (`f5e89f4`) reading "If `npmCommand` is configured … the extension uses the resolved global root **reported by that command** instead of the literal path" — which at least named the command as the source of the value (presupposing a subprocess seam through which the extension would invoke it). `git log -S 'already-resolved root'` and `git log -L 12,12` both isolate commit `808c72b` as the sole author of the current phrasing: that commit rewrote the line to "Pi resolves the global root and the extension reads that already-resolved root … the extension does not itself invoke `npmCommand`", explicitly to remove the subprocess-seam presupposition flagged by a sibling review finding. The rewrite eliminated the named source (the command) without substituting any other retrieval mechanism, which is precisely the gap this finding records — a defect introduced by the prior fix rather than present since inception.

## Solution approach

Rewrite the `npmCommand` sentence in root #4 of `package-and-settings.md` so the extension scans the literal `~/.pi/agent/npm/` path, `~`-expanded via [DISC-1](./discovery-sources.md#disc-1), exactly as root #5 does, and does not consult `npmCommand`. Ground the change in the fact that `npmCommand` governs only Pi's npm lookup/install operations, not the install root.
