## Finding

The "Roots scanned" enumeration in `package-and-settings.md` states for root #4 (`~/.pi/agent/npm/`): "When `npmCommand` is configured (per `packages.md` §npm), Pi resolves the global root and the extension reads that already-resolved root instead of the literal path; the extension does not itself invoke `npmCommand`." The rule obliges the extension to obtain a value — Pi's resolved global npm root — but names no API, seam, or context field through which to obtain it.

The mechanisms the extension actually has are closed and enumerated. The `FileSystem` seam ([PIC-13](../pi-integration-contract/host-interfaces-services.md#pic-13)) exposes only `readText`/`writeText`/`exists`/`homedir`/`cwd`/`readdir`/`lstat`/`realpath` (plus `readBytes`); none returns an npm root. The `ExtensionContext` member surface is normatively closed ("the runtime touches exactly the members enumerated"; omitted members are MUST-NOT-call) and exposes no global-npm-root member. So an implementer reading the spec literally cannot satisfy root #4 when `npmCommand` is configured: there is no path from "Pi resolves the global root" to a value the extension can read.

Compounding this, Pi's own `packages.md` §npm documents `npmCommand` as a wrapper command that pins "npm package lookup and install operations" (e.g. `mise exec node@20 -- npm`), and states user installs go under `~/.pi/agent/npm/` unconditionally. There is no documented Pi behaviour that relocates the global root when `npmCommand` is set, so the spec's premise that a separately-resolved root must be retrieved appears to rest on a misreading of what `npmCommand` does.

## Issue introduction

**Verdict:** introduced-by-single-commit (fix-induced regression)
**Introducing commits:** `808c72b` — "pi-loom spec: resolve \"npmCommand-resolved npm root presupposes a subprocess seam\"" (2026-06-06)
**History:** The root #4 line was first authored when the spec moved into `docs/` (`31ff060`) and carried through the topic split (`f5e89f4`) reading "If `npmCommand` is configured … the extension uses the resolved global root **reported by that command** instead of the literal path" — which at least named the command as the source of the value (presupposing a subprocess seam through which the extension would invoke it). `git log -S 'already-resolved root'` and `git log -L 12,12` both isolate commit `808c72b` as the sole author of the current phrasing: that commit rewrote the line to "Pi resolves the global root and the extension reads that already-resolved root … the extension does not itself invoke `npmCommand`", explicitly to remove the subprocess-seam presupposition flagged by a sibling review finding. The rewrite eliminated the named source (the command) without substituting any other retrieval mechanism, which is precisely the gap this finding records — a defect introduced by the prior fix rather than present since inception.

## Solution Space

### Recommendation

In `package-and-settings.md` root #4, remove the sentence stating that when `npmCommand` is configured Pi resolves the global root and the extension reads that already-resolved root. Replace it with a statement that the extension scans the literal `~/.pi/agent/npm/` path, `~`-expanded via the `FileSystem.homedir()` seam ([DISC-1](./discovery-sources.md#disc-1)), exactly like root #5, and does not consult `npmCommand`. Note that `npmCommand` governs only Pi's own npm lookup/install operations, not the install root: global installs land in `~/.pi/agent/npm/` regardless of `npmCommand` (per `packages.md` §npm).

Before deleting the clause, confirm against the pinned `@earendil-works/pi-coding-agent` that the global-install path is unaffected by `npmCommand`.

## Consequence

**Severity:** correctness

When `npmCommand` is configured, root #4 enumeration is unimplementable as written: two reasonable implementers diverge — one scans the literal `~/.pi/agent/npm/` path, one searches in vain for a non-existent resolution API, and one invokes `npmCommand` itself in direct violation of the "does not itself invoke `npmCommand`" clause. Global `pi install -g` packages are then discovered inconsistently (or not at all) across conforming implementations.
