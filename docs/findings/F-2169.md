## Finding

`host-prerequisites.md` (*Host prerequisites for the degraded-state branch*) flags clause (a) — *Extension-instance survival across session-only `session_shutdown`* — as an **open contradiction (blocking, not a deferral)** against Pi's documented teardown-and-rebind extension lifecycle at the loom 1.0 Pi-SDK pin, and states "The conflict MUST be resolved before the SM-4 / SM-5 / SM-6 (and SM-3b) degraded-state obligations are implemented." The session-only degraded-state branch presupposes that on `event.reason ∈ {"new", "resume", "fork"}` Pi keeps the same extension instance loaded against the swapped-in session, so the closed-over `LoomRegistry` the prior handler drained and tag-transitioned survives and the `"degraded-needs-reload"` slash note becomes observable on the new session.

The *Clause (a) contradiction narrative* sub-section then documents that Pi's published lifecycle (`@earendil-works/pi-coding-agent` `docs/extensions.md`, "Session replacement lifecycle and footguns") does the **opposite**: on `/new`, `/resume`, and `/fork` Pi fires `session_shutdown` for the old instance, tears the old runtime down, reloads-and-rebinds extensions for the new session, and fires `session_start`; captured session-bound objects are stale after replacement and throw if used. Under that behaviour the new instance starts with a fresh, non-drained registry and dispatches normally — hiding the degradation rather than surfacing it — and the CLAUDE.md no-globals/statics/singletons rule plus the contract's DI/closure routing rule out any process-global registry that could survive the rebind. The narrative concludes "there is no escape hatch."

The defect is that the spec presents SM-4/SM-5/SM-6 (and SM-3b) as a normative degraded-state contract whose foundational premise it simultaneously declares contradicted by Pi's own documentation, and leaves the deciding determination (`provably-differs` vs `governed-by-rebind`) formally open. The unblocking condition — a `provably-differs` verdict, recorded per item (a)'s authoritative-determination recording shape — has not been reached; the evidence already in the narrative points the other way (`governed-by-rebind`, "branch unreachable as written, SM-4/5/6 must be reworked"). Until the determination lands, the entire degraded-state branch is unimplementable and may be wholly unreachable.

## Solution Space

**Shape:** unresolved

### Reasoning

The genuine fix is the authoritative determination the spec already names: whether loom's package-loaded `extensions/index.ts` (auto-discovered from `package.json#pi.extensions`, distributed via `pi install`) is governed by the same teardown-and-rebind lifecycle `docs/extensions.md` documents for the general extension case, or provably differs from it for the session-only reasons. That answer requires a cited code/doc site about Pi's actual handling of package-loaded extensions across session-only swaps — external input a spec edit cannot manufacture — recorded per item (a)'s authoritative-determination recording shape (`version-bump-step2.md#bump-step-2-outcome-recording-conventions`), with the bump contributor as actor of record.

What would unblock it:
- A `provably-differs` verdict (cited evidence that the package-loaded path retains the same instance + registry across `{"new","resume","fork"}`) → SM-4/5/6 + SM-3b stand as written, `V9l` is unblocked.
- A `governed-by-rebind` verdict → the branch is unreachable as written; SM-4/5/6 + SM-3b must be reworked (the plan's pre-designed fallback collapses `V9l` to full teardown + `/reload` with no degraded tag write).

Note for whoever takes this up: the evidence already assembled in the *Clause (a) contradiction narrative* (Pi tears down and reloads-and-rebinds; stale captured objects throw; no process-global escape hatch under CLAUDE.md) points strongly toward `governed-by-rebind` / rework, but stops short of confirming the package-loaded placement specifically — that gap is precisely the open input. The interim quarantine is already in place (plan `V9l` blocked; `V18c` owns the resolution), so the residual defect is the unresolved determination itself, not the absence of a gating mechanism.

## Consequence

**Severity:** blocking

Implementation of the session-only degraded-state branch (SM-4/SM-5/SM-6, SM-3b; plan leaf `V9l`) cannot proceed: the spec itself gates it on resolving the contradiction, and the cited evidence indicates the closed-over `LoomRegistry` the branch depends on does not survive a session-only swap, so the branch as written may be unreachable. An implementer who missed the gate and built against the stated premise would produce a degraded-state surface that silently never fires (degradation hidden behind a fresh registry) rather than surfacing the `/reload` recovery note.
