## Finding

`extension-bootstrap-and-per-loom.md` routes the `loom/load/extension-bootstrap-failed` and `loom/load/host-incompatible` diagnostics through "the **System notes** fallback chain ÔÇö `sendSystemNote` Ôćĺ `ctx.ui.notify` Ôćĺ `console.error`", and the renderer-failure bullet degrades to "the `ctx.ui.notify` arm of the **System notes** fallback chain". `sendSystemNote` is used here as a spec-owned name for the first arm / the chain as a whole, but the file never defines it and there is no glossary entry for it. A reader of shard-08 alone cannot tell whether `sendSystemNote` is a loom-internal helper wrapping `pi.sendMessage`, a Pi SDK export, or an informal shorthand for the whole chain.

The gap is not local to this file. The canonical definition site is the **System notes** paragraph of `runtime-event-channel.md`, which pins the chain as the ordered steps `pi.sendMessage` Ôćĺ `ctx.ui.notify` Ôćĺ `loom/runtime/system-note-delivery-failed` diagnostic Ôćĺ terminal `console.error` ÔÇö but that paragraph describes the chain entirely in terms of the underlying SDK calls and never coins the identifier `sendSystemNote`. The name is nonetheless used normatively at that same site (PIC-54: "MUST NOT propagate out of the `sendSystemNote` fallback chain") and at several other reference sites (`capability-probe.md`, `active-invocation-registry.md`, `diagnostic-emission-isolation.md`). Every reference site therefore treats `sendSystemNote` as a defined term that is in fact defined nowhere.

The chain's behaviour is fully pinned by the ordered-step list, so this is a terminology gap rather than a behavioural one ÔÇö but it leaves the most-cited name in the System-notes machinery as an unbound referent. The durable fix is to coin `sendSystemNote` at its definition site (`runtime-event-channel.md`, **System notes** paragraph) with an anchor, then have shard-08's bare uses resolve to it via cross-reference.

## Solution Space

**Shape:** single

### Recommendation

Coin and define `sendSystemNote` at its canonical definition site, then make shard-08 reference it:

1. In the **System notes** paragraph of `runtime-event-channel.md`, immediately above the existing fallback step-list, add a one-sentence definition with an anchor (e.g. `<a id="send-system-note"></a>`): state that `sendSystemNote` is the spec-coined label for the loom-internal best-effort emission helper that drives the ordered chain (`pi.sendMessage` Ôćĺ `ctx.ui.notify` Ôćĺ `loom/runtime/system-note-delivery-failed` diagnostic Ôćĺ terminal `console.error`) for `loom-system-note` emissions ÔÇö explicitly **not** a Pi SDK export, and naming the *whole chain*, not solely the `pi.sendMessage` first step.
2. In `extension-bootstrap-and-per-loom.md`, link the bare `sendSystemNote` token in the routing paragraph to that anchor at first use (e.g. `[sendSystemNote](./runtime-event-channel.md#send-system-note)`), so a shard-08 reader resolves the term without leaving the file's link graph.

Edge cases for the implementer: keep the inline expansion consistent with the four-step definition ÔÇö the shard-08 routing paragraph currently abbreviates to three steps (`sendSystemNote` Ôćĺ `ctx.ui.notify` Ôćĺ `console.error`), dropping the `loom/runtime/system-note-delivery-failed` diagnostic step; once the term is defined and linked, prefer the link over re-spelling an abbreviated expansion. Do not coin a competing name; `sendSystemNote` is already the term used by PIC-54 and by the plan's `V7d` text.
