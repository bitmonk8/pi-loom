## Problem

The **`TransportError.retryable` population** rule (`#transport-error-retryable`) assigns `retryable` by transport-error class: `true` for network-level failures, HTTP 5xx, and HTTP 429; `false` for every non-429 4xx and for an HTTP-200 non-overflow body-envelope error. These arms partition only the status sub-domain `{network-level, 5xx, 429, non-429 4xx, HTTP-200}`, but the classifier reads `ProviderResponse.status` from the last `onResponse` firing and nothing constrains that status to the enumerated set. A response that fires `onResponse` with a residual status â€” a non-200 2xx (`201`/`202`/`204`) or a surfaced 3xx â€” and resolves with `stopReason: "error"` matches no arm, so `retryable` is left unpopulated for a `TransportError` the runtime still constructs. The gap compounds at the variant-selection sentence in the **Provider error mapping** rule (`#provider-error-mapping`), which routes only enumerated 4xx/5xx/HTTP-200/network-level responses to `TransportError`, leaving the residual status unrouted at both stages; conforming implementers diverge and the conformance suite cannot pin the field across the full status domain.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `8fb8f2b` ("pi-loom spec: resolve \"TransportError.retryable population rule\"", 2026-06-04)
**History:** The spec corpus is git-tracked. `git log -G 'every other \(non-429\) 4xx'` and `git log -S 'TransportError.retryable\` population'` both bottom out at `8fb8f2b`, the commit that first introduced the **`TransportError.retryable` population** rule (then on `pi-integration-contract.md`, before the size-cap split `f5e89f4` relocated it to `provider-error-mapping.md`). `git show 8fb8f2b` confirms the very first version enumerated only `true` for network-level/5xx/429 and `false` for "every other (non-429) 4xx" â€” non-total from inception. Later commits extended coverage without closing the hole: `47c0950` added the HTTP-200 non-overflow body-envelope arm and `ef4cfb8` added the stop-reason path, but no commit added a residual catch-all over the remaining status values. The defect is thus original to the rule's introduction, not regressed in by a later edit.

## Solution approach

In the **`TransportError.retryable` population** rule (`#transport-error-retryable`), rewrite the `false` clause into an explicit residual arm so every captured status not assigned `true` (network-level, 5xx, 429) maps to `false`, covering non-429 4xx, HTTP-200 non-overflow body-envelope errors, and any other status reaching the classifier including a non-200 2xx or a surfaced 3xx. Rewrite the variant-selection sentence in the **Provider error mapping** rule (`#provider-error-mapping`) to a total formulation so every classifier-reaching response that does not match an overflow signature maps to `TransportError` regardless of HTTP status.

## Solution constraints

- The residual `false` arm must not shadow an overflow match: overflow-signature precedence still runs first per the *Classifier input surface* note (`#classifier-input-surface`).
