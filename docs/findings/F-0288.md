## Finding

`V13a` ("Query render and escapes", `docs/plan_topics/V13a-query-render.md`) packs five separable query-surface concerns into a single feature leaf:

1. **Template-text rendering** — the trim → dedent pipeline reproducing the eight normative vectors (`query/query-forms.md` §"Dedent and newline-trim — normative behaviour").
2. **Escape handling** — `loom/parse/illegal-template-escape` and `loom/parse/unterminated-template` (`query/query-escapes-stringification.md` §Escapes).
3. **Interpolation stringification** — the stringify-by-loom-static-type table plus the static/runtime `loom/parse/interpolated-result` rejection (`query-escapes-stringification.md` §"Stringification of interpolated values").
4. **Result-discipline / discard observability** — the `loom/parse/discarded-query-result` parse error plus the runtime discard-observability event on `V9d`'s runtime-event channel (`query-escapes-stringification.md` §"Discarded query results…" and §"Observability of discarded results").
5. **Degenerate-template handling** — the `loom/parse/empty-template` warning and the runtime short-circuit to `ValidationError{empty_template, attempts:0}` that bypasses respond-repair (`query-forms.md` §"Degenerate rendered templates").

The plan's own convention (`conventions.md` §"Vertical slices") defines a leaf as "the smallest feature that can ship and be tested independently." Surface 4 (result-discipline / discard observability) is independent of how a template renders — it concerns the `Result` lifecycle and the `V9d` event channel, not template text — and surface 3 (stringification) is the value-to-text rule the frontmatter `system:` field reuses (`V6d` depends on `V13a` for exactly this). Bundling them with the render pipeline makes `V13a` materially larger than its V13 siblings (`V13b`, `V13c`), whose `Tests` bullets each cover a single coherent surface.

The bundling is original to the leaf, not introduced by a later edit.

## Solution Space

### Recommendation

Carve the result-discipline / discard-observability surface out of `V13a` into its own `<new>` leaf, leaving render + escapes + stringify + degenerate short-circuit in `V13a` as one cohesive rendering pipeline (escapes → stringify → trim/dedent → degenerate short-circuit), where each stage feeds the next. The discard surface concerns the `Result` lifecycle and the `V9d` runtime-event channel rather than template text, so it is the independent seam.

Plan edits:

- Carve a `<new>` leaf + paired `<new>`-T from `V13a` / `V13a-T`. Use the `<new>` placeholder for both IDs and let the implementer allocate the final IDs per `conventions.md` §"Leaf format"; do not invent a final ID.
- Move the `loom/parse/discarded-query-result` parse-error + runtime discard-observability event `Tests` content and its matching `Adds` clause from `V13a` onto the `<new>` leaf.
- Move the `V9d` `Deps` entry from `V13a` onto the `<new>` leaf, so the `V9d` dependency and the runtime-event assertions travel together.
- Add the `<new>` impl leaf to `H5b`'s `Deps` and to the `coverage-matrix.md` `query/` (QRY) row in the same change (per the *Transitive-completeness plan-maintenance* clause).

The spec is read-only for this fix; no spec edits.

## Consequence

**Severity:** advisory

The leaf is still shippable as written: every one of the five surfaces is enumerated in a `Tests` bullet and carries a paired `V13a-T` red-test obligation, so an implementer is not blocked and will not invent behaviour. The cost is a larger review/iteration surface than the V13 siblings and a single `Ships when` gate that names only two of the five surfaces (see related finding 142), so the done-condition reads as coarse relative to the asserted behaviours.
