## Finding

`V13a` ("Query render and escapes", `docs/plan_topics/V13a-query-render.md`) packs five separable query-surface concerns into a single feature leaf:

1. **Template-text rendering** — the trim → dedent pipeline reproducing the eight normative vectors (`query/query-forms.md` §"Dedent and newline-trim — normative behaviour").
2. **Escape handling** — `loom/parse/illegal-template-escape` and `loom/parse/unterminated-template` (`query/query-escapes-stringification.md` §Escapes).
3. **Interpolation stringification** — the stringify-by-loom-static-type table plus the static/runtime `loom/parse/interpolated-result` rejection (`query-escapes-stringification.md` §"Stringification of interpolated values").
4. **Result-discipline / discard observability** — the `loom/parse/discarded-query-result` parse error plus the runtime discard-observability event on `V9d`'s runtime-event channel (`query-escapes-stringification.md` §"Discarded query results…" and §"Observability of discarded results").
5. **Degenerate-template handling** — the `loom/parse/empty-template` warning and the runtime short-circuit to `ValidationError{empty_template, attempts:0}` that bypasses respond-repair (`query-forms.md` §"Degenerate rendered templates").

The plan's own convention (`conventions.md` §"Vertical slices") defines a leaf as "the smallest feature that can ship and be tested independently." Surface 4 (result-discipline / discard observability) is independent of how a template renders — it concerns the `Result` lifecycle and the `V9d` event channel, not template text — and surface 3 (stringification) is the value-to-text rule the frontmatter `system:` field reuses (`V6d` depends on `V13a` for exactly this). Bundling them with the render pipeline makes `V13a` materially larger than its V13 siblings (`V13b`, `V13c`), whose `Tests` bullets each cover a single coherent surface.

## Solution Space

**Shape:** multiple

The finding identifies a real atomicity gap, but the cut-line is a judgment call with two materially different boundaries.

### Option A — split off the value-rendering surface

**Approach.** Move the stringify-by-loom-type table plus `loom/parse/interpolated-result` (and optionally the degenerate short-circuit) into a `<new>` leaf; keep template-text rendering (trim → dedent vectors, escapes, `unterminated-template`) in `V13a`.

**Plan edits.** Carve a `<new>` leaf + paired `<new>`-T from `V13a` / `V13a-T`; move the stringify `Tests` bullet and the matching `Adds` clause; retarget `V6d` / `V6d-T` `Deps` from `V13a` to the `<new>` stringify leaf (the `system:` field reuses this table); add `<new>` to `H5b` `Deps` and to the `coverage-matrix.md` `query/` (QRY) row.

**Spec edits.** None.

**Pros.** Isolates the stringification table — the surface with a second consumer (`V6d`) — behind its own leaf and gate.

**Cons.** Stringification runs *before* dedent in the render pipeline (`query-escapes-stringification.md` notes stringified multi-line output participates in dedent), so this cut separates two stages that feed each other; forces a `V6d` `Deps` retarget.

**Risks.** A consumer (`V6d`) `Deps` retarget must land in the same change or `V6d`'s stringification dependency dangles.

### Option B — split off the result-discipline surface

**Approach.** Move `loom/parse/discarded-query-result` plus the runtime discard-observability event (on `V9d`'s channel) into a `<new>` leaf; keep render + escapes + stringify + degenerate short-circuit in `V13a` as the cohesive rendering pipeline.

**Plan edits.** Carve a `<new>` leaf + paired `<new>`-T from `V13a` / `V13a-T`; move the discard parse-error + runtime-event `Tests` content and its `Adds` clause; move the `V9d` `Deps` entry onto the `<new>` leaf; add `<new>` to `H5b` `Deps` and to the `coverage-matrix.md` `query/` (QRY) row.

**Spec edits.** None.

**Pros.** Cuts at the most independent seam — result-lifecycle/observability is orthogonal to template text — leaving the render pipeline (escapes → stringify → trim/dedent → degenerate short-circuit) whole, where each stage genuinely feeds the next. No `V6d` retarget needed.

**Cons.** `V13a` remains the larger of the two leaves (it keeps four of the five surfaces).

**Risks.** The `V9d` dependency and the runtime-event assertions must move together onto the `<new>` leaf.

### Recommendation

Take **Option B**. The discard / observability surface is the cleanest separation because it concerns the `Result` lifecycle and the `V9d` runtime-event channel rather than template text, whereas render, escapes, stringification, and the degenerate short-circuit are sequential stages of one rendering pipeline that feed each other and read more naturally as one leaf. Honour the existing leaf-ID scheme (`conventions.md` §"Leaf format"); do not invent a final ID for the new leaf or its `-T` pair — use the `<new>` placeholder and let the implementer allocate per `conventions.md`. Whichever cut is chosen, the paired `-T` leaf splits the same way, and per the *Transitive-completeness plan-maintenance* clause the `<new>` impl leaf must be added to `H5b`'s `Deps` and to the `coverage-matrix.md` `query/` (QRY) row in the same change. The spec is read-only for this fix.

## Consequence

**Severity:** advisory

The leaf is still shippable as written: every one of the five surfaces is enumerated in a `Tests` bullet and carries a paired `V13a-T` red-test obligation, so an implementer is not blocked and will not invent behaviour. The cost is a larger review/iteration surface than the V13 siblings and a single `Ships when` gate that names only two of the five surfaces (see related finding 142), so the done-condition reads as coarse relative to the asserted behaviours.
