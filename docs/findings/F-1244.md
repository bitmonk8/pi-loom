## Finding

The per-facet citing-test machinery names a single concept — *a test that cites both a multi-leaf coverage-matrix row's subject (its numbered REQ-ID or `cka-<n>` token) and one facet's closing-leaf-ID inline* — under two different terms. The dominant term across the corpus is **`facet-naming citing test`** (used in `H5a`'s Adds, `H5b`'s Tests/Ships-when, `H5f`'s Adds/Tests/Ships-when, `H6a`'s Adds/Tests/Ships-when, and `conventions.md`). A second term, **`per-facet citing test`**, names the same test object in `coverage-matrix.md`'s convention title (*"Multi-leaf-row per-facet citing tests"*) and in two of `H5f`'s own *Tests* references (the bullet label `(*REQ-ID discipline* — per-facet citing tests)` and the line *"(excluded, no per-facet citing test required)"*).

Both names appear on `H5f`'s page at once — the bullet label says "per-facet citing tests" while the same bullet's body says "facet-naming citing test" — with no equivalence note. The clash is narrow but real: the `per-facet` modifier is also used legitimately and consistently as the name of the *gate* (`per-facet citing-test arm`, `per-facet citing-test gate`, `per-facet citing-test scan`, `per-facet citing-test surface`), so a reader meeting bare "per-facet citing test(s)" for the *test object* cannot tell whether it names the same object as "facet-naming citing test" or a separate gate-level notion. `conventions.md` already keeps the distinction clean (test object = "facet-naming citing test"; gate = "per-facet citing-test arm"); the outliers that name the test object "per-facet citing test" are the `coverage-matrix.md` convention title and the two `H5f` bullet references.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `cda161b` (2026-06-17, "pi-loom plan: resolve \"split REQ-IDs pass closing gate on one facet citing test\""); `26a8794` (2026-06-18, "pi-loom plan: resolve \"split coverage-matrix rows can ship a facet untested\"")
**History:** `cda161b` coined the term `facet-naming citing test` in `conventions.md` (the *REQ-ID discipline* rule and release-time residue inspection item 7). The next day `26a8794` created the `H5f` leaf and the `coverage-matrix.md` *"Multi-leaf-row per-facet citing tests"* convention, and authored `H5f`'s *Tests* bullet labels as "per-facet citing tests" — introducing the second name for the test object alongside the already-established `facet-naming citing test` on the same page. The collision is durable since `26a8794`; the two terms have coexisted for the same object ever since. `git log -S` confirms no earlier occurrence of either token under `docs/plan_topics/`.

## Solution Space

**Shape:** multiple

### Option A — Reserve `per-facet` for the gate; name the test object `facet-naming citing test` everywhere

**Approach.** Use the single term `facet-naming citing test` for the test object (the test citing both the row subject and a facet's closing-leaf-ID inline). Keep `per-facet` only as the modifier on gate-level nouns (`per-facet citing-test arm` / `gate` / `scan` / `surface`). Fix the few sites that name the *test object* "per-facet citing test(s)".

**Plan edits.**
- `coverage-matrix.md`: retitle the convention *"Multi-leaf-row per-facet citing tests"* to name the test object consistently (e.g. *"Multi-leaf-row facet-naming citing tests"*), leaving the body's existing "facet-naming citing test" definition intact.
- `H5f-per-facet-citing-test-gate.md`: change the *Tests* bullet label `(*REQ-ID discipline* — per-facet citing tests)` and the partition bullet's `(excluded, no per-facet citing test required)` so both reference the test object as `facet-naming citing test`.
- `H5a`, `H5b`, `H6a`, `conventions.md`: no test-object rename — they already use `facet-naming citing test`; their `per-facet citing-test <noun>` usages are gate names and stay.

**Spec edits.** None.

**Pros.** Smallest churn (~3 edit sites). Preserves the gate-vs-test-object distinction `conventions.md` already maintains. Aligns with the majority form (the test object is "facet-naming citing test" at 15 sites vs 7). Leaves the `H5f` filename and gate title (`Per-facet citing-test closing-gate arm`) untouched.

**Cons.** Renames a convention title that other prose references by name; a stale cross-reference is possible.

**Risks.** Grep for the old convention-title string before renaming to catch any by-name reference (e.g. `H5f`'s **Convention.** field cites *"Multi-leaf-row per-facet citing tests"*).

### Option B — Standardise on `per-facet citing test` for both the gate and the test object

**Approach.** Replace every `facet-naming citing test` with `per-facet citing test` so one term serves both the gate and the test it looks for (the source review's original suggestion).

**Plan edits.** Replace all `facet-naming citing test` occurrences across `H5a`, `H5b`, `H5f`, `H6a`, and `conventions.md` (~15 sites).

**Spec edits.** None.

**Pros.** Single surface term; matches the `H5f` filename, gate title, and existing convention title already in place.

**Cons.** Much larger diff (~15 sites across 5 files). Collapses the gate-vs-test-object distinction, so "per-facet citing test" must denote both *the gate that runs per facet* and *a test, one per facet* — re-introducing the ambiguity at every `per-facet citing-test arm/scan/surface` site.

**Risks.** A reader can no longer tell from the bare term whether the gate or a test is meant.

### Option C — Add an equivalence note, leave both terms

**Approach.** At the `coverage-matrix.md` convention (the canonical definition site), state that `per-facet citing test` and `facet-naming citing test` denote the same test.

**Plan edits.** Add an equivalence clause to the *Multi-leaf-row per-facet citing tests* convention paragraph.

**Spec edits.** None.

**Pros.** Minimal diff; no renames.

**Cons.** Leaves two live names for one object; readers still meet both. Annotates the inconsistency rather than removing it.

**Risks.** The note can drift out of sync if either term later migrates.

### Recommendation

Take **Option A**. The test object becomes uniformly `facet-naming citing test` (already the majority form and the term `conventions.md` uses), while `per-facet` stays the gate modifier — so the touch is confined to the `coverage-matrix.md` convention title and `H5f`'s two test-object references, with no churn across `H5a`/`H5b`/`H6a` and no change to the `H5f` filename or gate title. Resolve in order: rename the `coverage-matrix.md` convention title first (the canonical definition), then sweep `H5f`'s two test-object usages to match. Edge case to watch: do **not** rename the gate-level usages (`per-facet citing-test arm` / `gate` / `scan` / `surface`) — only the bare references to the *test object* change; and update any by-name citation of the old convention title (e.g. `H5f`'s **Convention.** field) when the title changes.

## Consequence

**Severity:** advisory

An implementer building the `H5f` gate or authoring the per-facet tests meets the same test object under two names on one page and may read "per-facet citing test" and "facet-naming citing test" as two distinct test kinds, especially since `per-facet` elsewhere names the gate rather than the test. The inline definition is always supplied alongside each use, so behaviour will not diverge and no gate fires incorrectly; the cost is a consistency/guidance gap that slows comprehension of the most cross-referenced gate in the Horizontal phase.
