## Problem

DISC-4's detection-key parenthetical in `discovery/discovery-sources.md` says detection runs on the final derived name "after `pi.looms` mapping and `as` rename", but no loom-discovery rule defines an `as` rename. DISC-5's `pi.looms` grammar is purely path-shaped (paths/globs with `!`/`+`/`-` overrides) and has no aliasing clause, and no other discovery source (Global, Project, `loomPaths`, `--loom`, `looms/` fallback) introduces one. The only `as` rename in the spec is the `tools:` callable rename in `frontmatter/frontmatter-fields-a.md`, which changes how a parent loom calls its callees — not the callee's slash name or the detection key. The same dangling clause has propagated to `pi-integration-contract/active-invocation-registry.md`'s `loom`-field canonical-key paragraph (twice), so an implementer would expect a discovery-side rename hook that does not exist.

## Issue introduction

**Verdict:** present-since-inception (within the current spec corpus)
**Introducing commit:** d124b8c — `spec: same-priority slash-name collisions are uniform load-time errors (Decision #10)` (2026-05-05, Thomas Andersen)
**History:** The "Detection runs on the **final derived name** (after `pi.looms` mapping, `as` rename, and basename hyphen-normalisation)" sentence was authored whole by `d124b8c` when the same-priority collision rule was generalised in `spec_topics/discovery.md`; that commit's message itself lists the three-element parenthetical verbatim (`pi.looms mapping, as rename, basename hyphen-normalisation`) as the new detection key, so the `as` rename clause was inserted as a deliberate phrase rather than accreting. The sentence then moved into `discovery/discovery-sources.md` unchanged on `f5e89f4` (2026-05-26, the 100KB / 30KB size-cap split — a pure file split, no editorial change). `f2a82ad` (2026-06-06, *DISC-4 hyphen-normalise to same wire name transform contradicts Filename-validity*) dropped the third element — `and basename hyphen-normalisation` — after the lowercase-stem-only Filename-validity rule made it a no-op, but left the `as` rename clause intact in both `discovery-sources.md` and the inherited citation in `pi-integration-contract/active-invocation-registry.md`. The clause has therefore been present since the paragraph was written and survived the one targeted edit to its sibling list; no commit between `d124b8c` and HEAD audited the `as` rename clause for a discovery-side referent.

## Solution approach

Strike the `as` rename reference from DISC-4's detection-key parenthetical in `discovery/discovery-sources.md` so it names only the `pi.looms` mapping, with the filename stem taken verbatim for the discovery sources that have no `pi.looms` mapping. Propagate the same strike to both occurrences in `active-invocation-registry.md`'s `loom`-field canonical-key paragraph.

## Solution constraints

- Do not introduce a discovery-side `as` rename mechanism (e.g. an `as` clause in `pi.looms` entries, or wiring the `tools:` callable rename into the detection key); the fix is striking the dangling clause, not supplying the missing referent.
- Out of scope: the `loom/load/cross-format-collision` code-name wording (owned by F-0646) and the post-registration / superseded-entry-dispatch paragraph (owned by F-0979); edit only the detection-key parenthetical and its inherited citation.
