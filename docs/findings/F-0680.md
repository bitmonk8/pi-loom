## Finding

`docs/spec_topics/diagnostics/code-registry-load.md` carries two decision-log
fragments that record *how the file used to look* rather than anything an
implementer must know:

- A trailing HTML comment after the table (line 55):
  `<!-- `loom/runtime/*` registry now lives on code-registry-runtime.md; the
  former empty `#loom-runtime-namespace` stub anchor here was removed (no
  inbound links). -->`. HTML comments do not render and the message is purely
  an edit note explaining a prior anchor removal.
- A clause inside the `loom/load/unreadable-source` row's trigger cell
  (line 39): "*The per-read-timeout sub-trigger that previously rode this
  row now has its own code, `loom/load/package-read-timeout` (see the row
  below).*" This narrates a historical migration; the present row's contract
  no longer mentions per-read timeouts, so the cross-reference is forward
  guidance at best and decision-log carry-through at worst.

Neither fragment constrains the runtime, the wire shape, or operator
tooling. The forward pointer to `package-read-timeout` is the only piece
with arguable reader value, and it sits two rows below where any reader
hunting for the timeout case will already look.

## Issue introduction

**Verdict:** multi-commit-interaction

**Introducing commits:**
- `65989e2` (2026-06-01) — "pi-loom spec: resolve 'unreadable-source row
  carries two message formats'" — added the "*previously rode this row*"
  clause inside the `unreadable-source` row when splitting the per-read
  timeout out into the new `package-read-timeout` code.
- `3059a0e` (2026-06-05) — "pi-loom spec: resolve 'Diagnostic registry
  pointers cite the wrong file and overclaim completeness'" — replaced the
  empty `<a id="loom-runtime-namespace"></a>` anchor with the trailing
  HTML comment now on line 55.

**History:** Each fragment was introduced by a separate fix that, in
isolation, traded an older defect (a duplicated message-format note;
a dangling empty anchor) for a small explanatory note about *what
changed*. Neither commit added a decision-log fragment that the other
required; the cumulative effect — a registry file accumulating
"editorial residue" from every prior fix — only becomes visible when
the file is read end-to-end after both edits. No single revert is the
right remedy; the durable fix is to stop carrying the decision-log
prose forward at all.

## Solution Space

**Shape:** single

### Recommendation

Delete both fragments outright:

1. **Line 55** — remove the `<!-- `loom/runtime/*` registry now lives on
   code-registry-runtime.md … -->` comment in its entirety. The file already
   reads as the load-namespace registry; readers looking for runtime codes
   navigate via the sibling-files index in `diagnostic-shape.md` /
   `diagnostics.md`. There is no inbound link to preserve.
2. **Line 39** — in the `loom/load/unreadable-source` trigger cell, drop
   the sentence "*The per-read-timeout sub-trigger that previously rode
   this row now has its own code, `loom/load/package-read-timeout`
   (see the row below).*" The cell's preceding sentence is the full
   contract for the row in its current form, and the
   `package-read-timeout` row appears in the same table within visible
   scrolling distance.

If a forward pointer to `package-read-timeout` is genuinely wanted (it is
not strictly needed — the row sits twelve lines below), replace the
deleted sentence with a terse parenthetical on the existing sentence,
e.g. append "*(per-read timeouts during package discovery have their own
code, `loom/load/package-read-timeout`)*" to the trigger sentence. Do not
preserve the "previously rode" framing under any wording — that phrasing
is what makes it decision-log prose.

Edge cases for the implementer: confirm no other spec file links to
`#loom-runtime-namespace` before deleting the line-55 comment
(`grep -rn 'loom-runtime-namespace' docs/`); if a stray reference
appears, retarget it to the corresponding anchor on
`code-registry-runtime.md` rather than re-introducing the stub.

## Consequence

**Severity:** cosmetic

No observable behaviour is at stake; the cost is reader friction and a
weakly-justified precedent for parking decision-log prose inside normative
diagnostic-registry rows. Leaving the fragments in place gives later
maintainers cover to add more, eroding the registry's signal-to-noise.
