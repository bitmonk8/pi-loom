## Finding

Step 2(a) of the Pi version-bump procedure normatively edits and asserts against three named symbols — `SDK_SURFACE_INVENTORY`, `CAPABILITY_OBLIGATIONS`, and the literal-read assertion's per-`kind` branch table — but identifies the source-of-truth module that holds them only relationally: *"pinned in the same source-of-truth pinned-constants block in the extension module that holds the four Step-0 probe constants."* The "four Step-0 probe constants" pointer chains to `capability-probe.md`, which itself says only *"live in **one source of truth** inside the extension module that the build-time literal-read assertions also consume"* — another relational pointer. `version-bump-triggers.md` step 5 reuses the same wording when enumerating the four groupings the bump may touch.

The only place in the PIC corpus that pins a concrete on-disk location is `unknown-reason-rule.md`, which mentions in passing that the block is *"the static `SDK_SURFACE_INVENTORY` constant under `src/extension/`."* That citation is buried inside a sentence whose load-bearing claim is about the runtime handler's set-membership read, not about where the pinned-constants block lives, so a contributor working step 2(a) of the bump procedure has no anchored cross-link to it. The `audit-resolution.md` *Audit scope* paragraph independently pins `src/extension/**/*.assert.ts` as the suffix convention for the build-time assertion module — a third site referring to the same neighbourhood without an anchor that the version-bump step can link to.

The net effect: a contributor performing step 2(a) sees three named constants and a branch table they must co-edit on the same commit, but no canonical citation telling them which file (or named anchor) to open. They must either grep the codebase or follow a multi-hop relational chain through three spec topics, none of which is the canonical owner of the location pin.

## Solution Space

**Shape:** single

### Recommendation

Make `capability-probe.md`'s *"The four pinned constants the probe consumes … live in **one source of truth** inside the extension module that the build-time literal-read assertions also consume"* sentence the canonical citation site for the pinned-constants block. Attach a stable anchor to that sentence (e.g. `<a id="pinned-constants-block"></a>`) and edit the sentence to name the on-disk neighbourhood directly — `src/extension/` — using the same vocabulary `unknown-reason-rule.md` and `audit-resolution.md` already use. The spec MUST NOT pin a single fixed filename, since implementations may legitimately split the block across modules under `src/extension/`; the contract is the *directory* plus the *named symbols* (`SDK_SURFACE_INVENTORY`, `CAPABILITY_OBLIGATIONS`, `FACTORY_PROBABLE_CAPABILITIES`, `SHUTDOWN_AWAIT_CAP_MS`, and the implementation-owned `type-union-snapshot` entries), which together let a grep land on the block deterministically.

Then convert the four downstream relational references — `version-bump-step2.md` step 2(a) ("pinned in the same source-of-truth pinned-constants block in the extension module that holds the four Step-0 probe constants"), `version-bump-triggers.md` step 5 ("Four logical groupings live in the single source-of-truth pinned-constants block in the extension module"), `unknown-reason-rule.md`'s passing `src/extension/` mention, and any sibling occurrences — into anchor links pointing at the new `#pinned-constants-block` site. The version-bump-step2.md edit is the load-bearing one; the others ride along for consistency.

Edge cases the implementer must watch:
- The `src/extension/**/*.assert.ts` suffix convention in `audit-resolution.md` is a *separate* contract about the typecheck-only assertion module and MUST stay distinct from the runtime pinned-constants citation; do not collapse them.
- Do not re-introduce a plan-leaf citation (`V9a`, `V18a`, etc.) on the spec side — `9e53613`'s corpus-direction invariant remains binding and the fix above respects it by pinning the location via spec-intrinsic surface (directory + symbol names + anchor) rather than via a leaf reference.
- The anchored sentence in `capability-probe.md` enumerates the four Step-0 probe constants; `CAPABILITY_OBLIGATIONS` (a step-2(a)-side constant) is **not** one of them. The anchor's prose must therefore say the block "also holds" the step-2(a) constants (and the implementation-owned snapshot entries `version-bump-triggers.md` step 5 enumerates), so the cross-link from version-bump-step2.md lands somewhere whose scope actually includes the constants step 2(a) edits.

## Consequence

**Severity:** advisory

A contributor doing a bump can recover the location by grepping for `SDK_SURFACE_INVENTORY` or by following the relational chain into `unknown-reason-rule.md`, so the procedure is not blocked. The cost is friction and a documentation invariant that drifts silently: a future relocation of the constants would leave three independent spec paragraphs all referring to the home indirectly, with no single citation site to update. The defect compounds the cluster of step-2 citation-hygiene gaps already present (e.g. the `SM-4/5/6` compound-citation finding, the per-item recording schema, the buried baseline-acquisition block).
