## Finding

`V9i`'s **Adds** names "the pre-spawn binder-model guard" as one of the things the leaf ships. The two Tests bullets that exercise that guard — the `loom/runtime/subagent-model-unresolved` bullet in both `V9i-subagent-isolation.md` and `V9i-T-subagent-isolation.md` — shorten it to "the pre-spawn **model** guard", dropping the "binder-" qualifier.

Across the binder-model delta the canonical qualifier is consistently "binder-model": `V11a` resolves the "binder-model" and fires `loom/load/binder-model-unresolved`; `V9b`/`V6a` wire the same binder-model matcher. The runtime guard `V9i` adds is the subagent-spawn-time variant of that same binder-model resolution probe (its diagnostic code is `loom/runtime/subagent-model-unresolved`). The bare phrase "pre-spawn model guard" reads as a generic spawn precondition rather than the binder-model resolution probe specifically, leaving a reader who only sees the Tests bullet unsure whether the guard is checking the binder model or some other per-spawn model condition.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e (`pi-loom plan: build/update plan for spec.md + review`, 2026-06-10)
**History:** Both `V9i-subagent-isolation.md` and `V9i-T-subagent-isolation.md` were created in c6a664e. `git log -S` for "pre-spawn binder-model guard" (the Adds wording) and "pre-spawn model guard" (the Tests-bullet wording) each return only c6a664e across both leaf files; the qualifier mismatch was present in the leaf pair the moment it was authored, and no later commit touched either string.

## Solution Space

**Shape:** single

### Recommendation

In `V9i-subagent-isolation.md` and `V9i-T-subagent-isolation.md`, change the `loom/runtime/subagent-model-unresolved` Tests bullet so the guard is named "pre-spawn binder-model guard", matching the Adds wording. Concretely, the bullet currently reads:

> `loom/runtime/subagent-model-unresolved` fires when the pre-spawn model guard fails; `loom/runtime/subagent-dispose-failure` is advisory on a `dispose()` throw.

Replace "the pre-spawn model guard" with "the pre-spawn binder-model guard" in both files. No other wording in the bullet changes, and the Adds line is already correct.

## Consequence

**Severity:** advisory

A reader consulting the Tests bullet in isolation cannot tell that "pre-spawn model guard" is the binder-model resolution probe (`loom/runtime/subagent-model-unresolved`) rather than a generic spawn precondition; the test author may target the wrong guard or duplicate intent. The Adds wording disambiguates, so an attentive implementer recovers the meaning — this is a within-leaf-pair terminology drift, not a coverage or pickability gap.
