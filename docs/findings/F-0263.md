## Finding

Both `V7b` and `V7b-T` state the DIAG-3 acceptance criterion as:

> `DIAG-3`: codes are stable identifiers — a renamed code fails the gate (renames deferred to loom 2.0).

The clause before the parenthetical is the testable requirement: the closing gate must fail when a registered diagnostic code is renamed. The trailing parenthetical "(renames deferred to loom 2.0)" is future-roadmap rationale, not a property the test asserts. Folding it into the `Tests.` bullet invites an implementer to read the rename-stability gate itself as provisional — i.e. to treat "a renamed code fails the gate" as a temporary stance that relaxes at loom 2.0, rather than a permanent loom 1.0 invariant. In the spec the two ideas are kept distinct: DIAG-3 (`spec_topics/diagnostics/diagnostic-shape.md`, anchor `#diag-3`) states the stability rule normatively and separately notes that *performing* a rename is a breaking change deferred to a loom 2.0 migration. The plan leaf collapses both into one parenthetical glued to the test requirement.

The deferral context is already carried, in full, by the DIAG-3 spec rule the leaf cites; the leaf does not need to restate it to be complete.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V7b-code-registry.md` and `docs/plan_topics/V7b-T-code-registry.md`, strike the trailing parenthetical from the DIAG-3 `Tests.` bullet so it reads:

`- ``DIAG-3``: codes are stable identifiers — a renamed code fails the gate.`

Apply the identical edit in both files so the paired implementation/test leaves stay in sync. The deferral rationale need not be re-homed in the leaf: the cited DIAG-3 spec rule (`spec_topics/diagnostics/diagnostic-shape.md#diag-3`) already states that performing a rename is a breaking change deferred to loom 2.0. The spec is read-only for this fix.

## Consequence

**Severity:** advisory

The testable requirement ("a renamed code fails the gate") is stated clearly, so a working leaf can still be produced; the risk is interpretive. An implementer who reads the gate as temporary may under-build the rename detection or treat it as optional, weakening the stable-identifier guarantee DIAG-3 exists to enforce.
