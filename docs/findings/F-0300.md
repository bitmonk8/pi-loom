## Finding

`V18c`'s build-time literal-read assertion reads two constants out of `V9a`'s single source-of-truth pinned-constants block: `FACTORY_PROBABLE_CAPABILITIES` (the partition source `V18c` reconciles against) and `SHUTDOWN_AWAIT_CAP_MS` (the cancellation-runtime cap, value sourced from `patch-skew-degradation.md` §`session_shutdown` sub-step 3). Both symbols are owned and exported by `V9a` (`V9a-capability-probe.md` **Adds**). Yet `V18c`'s **Deps** field lists only `` `V18c-T`, `V18a`, `V18b` `` — no direct edge on `V9a`.

The per-leaf `Deps.` field, per `conventions.md`, names "other leaf IDs that must be complete first." Every other consumer of `V9a`'s pinned-constants block declares the edge directly: `V18a` lists `V9a`, and `V17a` (the semantics owner of `SHUTDOWN_AWAIT_CAP_MS`) lists `V9a`. `V18c` is the outlier. Today the read is only transitively ordered through `V18c → V18a → V9a`, and that chain is fragile for `SHUTDOWN_AWAIT_CAP_MS` specifically: `V18a` does not read `SHUTDOWN_AWAIT_CAP_MS` at all, so the only ordering guarantee for that constant rests on `V18a`'s incidental `V9a` dependency (acquired for the unrelated `FACTORY_PROBABLE_CAPABILITIES` partition reconcile). If `V18a`'s `V9a` edge is ever re-rooted or dropped, `V18c`'s literal read of `SHUTDOWN_AWAIT_CAP_MS` loses its ordering guarantee silently.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** b9de3ee (`pi-loom plan: resolve "V18a partition assertion imports undeclared V9a probe constant"`, 2026-06-11), 622d0d3 (`pi-loom plan: resolve "SHUTDOWN_AWAIT_CAP_MS physical home contradicts pinned-constants-block placement"`, 2026-06-11)
**History:** At `V18c`'s creation (c6a664e, 2026-06-10) the leaf mentioned neither `V9a`, `FACTORY_PROBABLE_CAPABILITIES`, nor `SHUTDOWN_AWAIT_CAP_MS`, and its `Deps` line was already `` `V18c-T`, `V18a`, `V18b` `` — correct for the then-uncoupled leaf. The `FACTORY_PROBABLE_CAPABILITIES` literal-read coupling to `V9a` was grafted into `V18c`'s **Adds** in b9de3ee; the `SHUTDOWN_AWAIT_CAP_MS` literal-read coupling was added in 622d0d3. Neither commit extended the `Deps` line, which has remained `` `V18c-T`, `V18a`, `V18b` `` unchanged since inception (`git log --follow`, `git log -S 'V9a'` and `-S 'SHUTDOWN_AWAIT_CAP_MS'` over `docs/plan_topics/V18c-version-bump-checklist.md`). The defect emerges from the interaction: the read couplings were authored without updating the dependency declaration they created an obligation for.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V18c-version-bump-checklist.md`, extend the `**Deps.**` field to add `` `V9a` `` to the existing list `` `V18c-T`, `V18a`, `V18b` `` (yielding `` `V18c-T`, `V18a`, `V18b`, `V9a` ``). This declares the direct edge for both constants `V18c`'s build-time literal-read assertion reads from `V9a`'s pinned-constants block (`FACTORY_PROBABLE_CAPABILITIES` and `SHUTDOWN_AWAIT_CAP_MS`), matching the direct `V9a` edge `V18a` and `V17a` already carry, and removing the dependence on the incidental transitive chain for `SHUTDOWN_AWAIT_CAP_MS`. No spec edit and no change to any test bullet or Ships-when gate is required; the edit is confined to the dependency declaration.

## Consequence

**Severity:** advisory

The transitive `V18c → V18a → V9a` chain orders the build correctly today, so `V18c` remains pickable; the gap is a stale `Deps` field that under-declares a direct literal-read edge. The brittle case is `SHUTDOWN_AWAIT_CAP_MS`, which `V18a` does not read — its ordering depends entirely on `V18a`'s unrelated `V9a` dependency, so re-rooting that edge would silently strand `V18c`'s literal read with no declared ordering.
