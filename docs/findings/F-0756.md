## Finding

`query-tool-loop.md` carries one normative worked-example transcript ("Worked example: depth-6 forced respond at `max_rounds`"). That vector uses `tool_loop.max_rounds: 2`, exercises two free-phase rounds, and reaches the forced respond turn via CIO-4's `max_rounds`-final branch firing *after* the second round's slot increment (`2 == 2`). The structurally-different `max_rounds: 0` case is pinned only in prose, in three coupled places: step 2 of *Typed queries are tool-loop-shaped* (the prompt-text is concatenated after the forced-respond instruction wording with a single U+000A separator; CIO-4 fires at typed-query start with `0 == 0`), the *Tool-call loop bound* paragraph (the forced respond turn is the exempt-routed terminator and does not consume a slot), and the closing sentence of the depth-6 example (with `max_rounds: 0`, `masked` is omitted on the surfaced event).

There is no normative input/output vector pinning what an implementer must actually emit on this path: the provider-visible message structure (exactly one provider call, whose user-message body is `<rendered-prompt> + U+000A + <forced-respond-instruction>` with the lowered schema interpolated into the template's `<schema-json>` placeholder), the CIO-4 `slot_count == max_rounds` evaluation at initialisation, the absence of any free-phase provider call, and the `masked`-omission on a `validation` event raised at that turn. A conformant implementation that issued a spurious zero-tool free-phase call before the forced respond turn â€” semantically wrong but observationally subtle, since the free-phase turn would carry no `tool_use` blocks and immediately stop â€” would still satisfy every prose MUST in the page and pass the depth-6 vector. The plan's `V13c-T` test leaf cites "[`max_rounds:0` boundary]" but has nothing normative behind that citation to assert against.

The same omission also breaks symmetry with the depth-6 vector's role as a fixed-point reference for the `RuntimeEvent`-shape conformance test and the typed-query suite (both cited as consumers of the depth-6 vector). The `masked`-omission rule for the `max_rounds: 0` arm of PIC-1's V1 reachable predicate currently has no fixture to consume.

## Solution Space

**Shape:** single

### Recommendation

Add a second normative worked-example transcript to `query-tool-loop.md`, sibling to the depth-6 vector, titled *Worked example: `max_rounds: 0` boundary* (with an anchor). Pin the following, all observable at the provider boundary:

- **Loom source.** A minimal typed query with `tool_loop: { max_rounds: 0 }`, `respond_repair: { attempts: 0 }`, an empty `tools:` list, and a one-line `@<T>`...`` body whose rendered post-trim form is a single short ASCII string (e.g. `Probe.`).
- **Slot-accounting precondition.** State explicitly that at typed-query start `slot_count == max_rounds` holds (`0 == 0`), so CIO-4's `max_rounds`-final branch fires at initialisation and the runtime dispatches the forced respond turn as the first and only turn â€” no free-phase provider call is issued.
- **Mock provider transcript.** Exactly one provider call. Show its user-message body byte-for-byte as the concatenation `<rendered-prompt> + U+000A + <forced-respond-instruction>`, where `<forced-respond-instruction>` is the *Initial forced respond turn template (normative)* rendered with `<slug>` substituted and the lowered response schema interpolated into the `<schema-json>` placeholder. Show `tool_choice` forced to `__loom_respond_<slug>`. Show the model invoking the respond tool with a schema-conformant payload.
- **Expected outcome â€” success arm.** The query resolves `Ok(<validated value>)`; no `RuntimeEvent` of `kind: "validation"` is emitted.
- **Expected outcome â€” failure arm (a depth-6 payload).** Mirror the depth-6 vector's `Err(QueryError { kind: "validation", cause: "schema_validation", attempts: 0, validation_errors: [...], ... })`, and pin the `RuntimeEvent` payload with `masked` **omitted** (per PIC-1's V1 reachable predicate: the round-after-increment slot count does not satisfy the predicate on this path, because no free-phase round ran).
- **Citation hooks.** Add the new vector to the citations already carried by the depth-6 example ("the `RuntimeEvent`-shape conformance test and the typed-query test suite both cite this vector"), so `V13c-T`'s `max_rounds:0` boundary test bullet has a single normative anchor to assert against.

Implementer edge case: the worked example must use the spec's post-trim / post-dedent / post-interpolation prompt form, not the source-text `@<T>`...`` form, so the byte-for-byte separator claim is verifiable; show both forms if needed for clarity, but only the post-trim form participates in the concatenation.
