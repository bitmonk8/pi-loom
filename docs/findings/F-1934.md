## Finding

Three leaves model behaviour that the plan itself classifies as an uncontracted real-Pi **consumption posture**, not a loom-side guarantee, because `@earendil-works/pi-coding-agent` pins no typed contract for it:

- `V14b` — the model-driven parallel tool-call batch shape (concurrent execution, settle-all-before-next-turn, independent per-sibling lowering). Its Adds explicitly tags this as an unpinned host presupposition whose detection "routes to editorial review per version-bump checklist item **(an)**".
- `V9i` — the `ERR-8` live surface (a mid-stream cancellation inside the real subagent `AgentSession` leaves committed turns unmutated), which the leaf states is a real-host-only behaviour witnessed only at the manual smoke gate.
- `V12a` — the `SLSH-2` user-visible streaming-ordering posture (streamed tokens observable before `ctx.waitForIdle()` resolves), routed to version-bump checklist item **(ak)**.

For all three, the only divergence-detection mechanisms the plan provides are *pre-merge, contributor-driven, manual* gates: editorial confirmation on a Pi version bump (`version-bump-step2.md` items (an)/(ak)) and the manual real-host smoke run (`real-host-smoke-gate.md`). Both gates fire only on a loom-side trigger — a Pi version bump, or a merge whose diff the contributor recognises as touching one of the smoke gate's four fidelity axes — and the smoke gate is, by its own statement, "a contributor obligation, not a CI-enforced gate" with "no automatic diff detector".

The deployed extension carries no runtime check or telemetry that observes whether the real host actually delivered the presupposed shape. If a presupposition is wrong — or if real Pi behaviour drifts without a loom-side change that trips a manual trigger — the divergence produces silently-wrong behaviour in production with no diagnostic, and recovery depends entirely on the cadence of the manual gates.

There is also a coverage asymmetry within the manual layer: the `V9i` `ERR-8` cancellation posture and the `V12a` `SLSH-2` streaming posture are both among the smoke gate's live-scored criteria (criterion (c) and fidelity-axis (i)/(iv) respectively), but `V14b`'s parallel-batch *delivery* shape is **not** one of the smoke gate's four fidelity axes — it is detected only by the editorial item (an) on a version bump, the weakest of the available paths.

## Solution Space

**Shape:** multiple

### Option A — Add an observe-and-log runtime divergence diagnostic

**Approach.** At the loom-observable sites, emit a lightweight diagnostic on the always-log `loom/runtime/*` channel (`V9d`'s always-log set) when the host's observed delivery/cancellation posture diverges from the presupposed shape — e.g. a model-issued multi-`tool_use` batch that is not delivered as a concurrent settle-all batch (`V14b`), or a mid-stream subagent cancellation after which committed turns are observed to have mutated (`V9i` `ERR-8` site). The diagnostic is observe-and-log only: it never alters control flow.

**Plan edits.** Add a Tests/Adds obligation at `V14b` (and the `V9i` `ERR-8` site, and optionally `V12a`) for the runtime divergence diagnostic; register the new `loom/runtime/*` code in `V9d`'s always-log set and in `V7b`'s diagnostic registry; add the matching registry row + asserting test to `coverage-matrix.md`. If the code does not fold into an existing leaf, allocate a new closing leaf (`<new>`).

**Spec edits.** None — the spec posture text is unchanged.

**Pros.** Gives production-time detection that the manual gates structurally cannot (they fire only on loom-side triggers, never on real-host drift). Closes the `V14b` blind spot where parallel-batch delivery is neither a smoke-gate axis nor runtime-checked.

**Cons.** The postures have no typed contract to check against, so any guard is heuristic; it must be tuned to avoid false-positive noise on the always-log channel.

**Risks.** A poorly-scoped guard could fire on benign host variance.

### Option B — Formalise the accepted-risk decision and close the coverage asymmetry

**Approach.** Treat the absence of runtime detection as a deliberate, documented decision rather than a gap. Add a cross-cutting host-presupposition rule to `conventions.md` §Cross-cutting rules that states unpinned host presuppositions are detected by editorial/manual gates only (no runtime guard, because no typed contract exists to check against), enumerates the cases (`V14b` parallel-batch → item (an); `SLSH-2` streaming → item (ak); `V9i` `ERR-8` → smoke gate), and explicitly records that `V14b`'s parallel-batch delivery shape is not among the smoke gate's four fidelity axes.

**Plan edits.** Add the rule to `conventions.md`; cross-reference it from `V14b`/`V9i`/`V12a` Adds in place of the per-leaf prose. No new diagnostic codes.

**Spec edits.** None.

**Pros.** Cheap; makes the residual production risk explicit and auditable; gives `V14b`'s asymmetry a named home.

**Cons.** Does not add any production-time detection — the silently-wrong-in-production risk remains, merely documented.

**Risks.** Future readers may treat the documented acceptance as closing a risk that is only acknowledged.

### Recommendation

Take Option A for the loom-observable sites (`V14b` parallel-batch delivery and the `V9i` `ERR-8` cancellation outcome), because it adds the one detection layer the manual gates cannot provide: a signal when the real host diverges in deployed use, including drift that occurs without any loom-side trigger. Scope the guard to observe-and-log on the always-log channel only — it must never change control flow, and its predicate should key off invariants the loom already observes (batch concurrency/settle ordering it controls; subagent committed-turn state before vs after a cancel) to avoid firing on benign live-model variance. Where a posture is genuinely not runtime-observable, fall back to Option B's documented-acceptance treatment for that posture rather than inventing a noisy heuristic.

## Consequence

**Severity:** advisory

A wrong host presupposition — or post-deployment drift in real-Pi parallel-batch delivery, subagent cancellation, or streaming order — produces silently incorrect behaviour for end users, because the deployed extension has no runtime detection and the only safeguards are pre-merge manual gates that fire on loom-side triggers a contributor must recognise. The leaves remain implementable and spec-faithful against the offline double, so this is a robustness/guidance gap rather than a blocker; the sharpest concrete hole is that `V14b`'s parallel-batch delivery shape sits outside both the smoke gate's four fidelity axes and any runtime check, leaving editorial item (an) as its sole detector.
