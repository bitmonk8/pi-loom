## Problem

`docs/plan_topics/conventions.md` names the test toolchain as load-bearing fact in two places: the §Per-phase TDD ritual depends on the TypeScript type-check / compile step (the "compiles and runs" gate, the "compile-time red" definition, and "type-check clean" in the Run step), and the §Cross-cutting rules "No silent test skipping" bullet names Vitest's `assert.fail` / `expect.unreachable`. Neither reference points to `H1a`, the leaf that provisions and pins that toolchain (declaring `vitest` and `typescript` in `devDependencies`, defining the `npm test` / `npm run typecheck` scripts, and owning the `tsc` floor). The page already cross-links the owning leaf for every other mechanical gate it names — `H2a`/`H3a` for the no-globals and ambient-primitive scans, `H5a`/`H5c`/`H5d` for the closing-gate and transitive-completeness checks — making the toolchain references the sole load-bearing infrastructure named without the parallel owning-leaf link. A reader following the page's own cross-link discipline cannot navigate from the test runner, assertion API, or compile step to the leaf that establishes them.

## Issue introduction

**Verdict:** introduced-by-later-edit
**Introducing commits:** `3edff0f` (2026-06-10, "resolve \"test runner and assertion API never named; non-JS panic token\""), `e7815d0` (2026-06-10, "resolve \"fail red for the intended reason undefined\"")
**History:** `conventions.md` was created at corpus inception by `fecb504` (2026-05-04, the spec/plan split) and did not name the toolchain at that point. The Vitest `assert.fail`/`expect.unreachable` reference in "No silent test skipping" entered with `3edff0f`; the type-check/compile-step references in the TDD ritual ("compile-time red", "type-check") entered with `e7815d0`, both during the same 2026-06-10 conventions-refinement session. Each commit added a load-bearing toolchain reference without the parallel `H1a` cross-link the page already used for its gate references. `git log -S "H1a" -- docs/plan_topics/conventions.md` returns no commits, confirming `H1a` has never been referenced in the file across its entire history; `H2a` (and the other gate-owning leaves) appear at their gate-naming sites. The defect is therefore an asymmetry that accreted as the toolchain references were added post-inception, not a present-since-inception gap.

## Solution approach

In `docs/plan_topics/conventions.md`, add a forward-link to `H1a` ([`H1a`](./H1a-scaffold-and-toolchain.md)) at each point the page names the test toolchain, mirroring the existing owning-leaf cross-links for the mechanical gates. Attribute the Vitest runner and assertion API to `H1a` at the §Cross-cutting rules "No silent test skipping" bullet, and attribute the `tsc` type-check / `npm run typecheck` toolchain to `H1a` at the §Per-phase TDD ritual's compile-step references ("compiles and runs", "compile-time red", and the "type-check clean" Run-step item).

## Solution constraints

- Out of scope: `H1a` itself is read-only for this fix — add links targeting it, do not edit it.
- The fix is confined to `docs/plan_topics/conventions.md`.
