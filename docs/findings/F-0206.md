## Finding

H2a's no-globals architectural gate describes one flagged form as "a top-level `const` whose initializer is a **mutable** object or array literal". The same sentence then states the scan's actual basis: "the AST scan decides the ban on the observable initializer shape, not on runtime reuse". These two clauses pull in opposite directions. "Mutable" is a runtime property — whether the object the literal produces is later mutated — which a static AST scan cannot determine. The parenthetical correctly says the decision rests on the *observable initializer shape*, i.e. whether the initializer is an object or array literal at all, independent of any later mutation.

The misleading qualifier appears twice in the same shape: in the **Adds.** bullet and in the *No globals, statics, singletons* **Tests.** bullet. A reader who takes "mutable" at face value could try to build a mutation/aliasing heuristic the AST scan is explicitly not meant to perform, or could exempt a never-mutated object/array-literal `const` that the intended rule means to flag. Striking "mutable" removes the contradiction and leaves the rule reading as the parenthetical already mandates: any top-level object or array literal initializer, outside any class or function, is flagged by shape.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H2a-cross-cutting-gates.md`, strike the word "mutable" from the object/array-literal qualifier in both places it occurs — the **Adds.** bullet and the *No globals, statics, singletons* **Tests.** bullet — so the flagged form reads "a top-level `const` whose initializer is an object or array literal, declared outside any class or function". This aligns the literal description with the parenthetical's stated AST-shape basis: any top-level object or array literal initializer is flagged regardless of whether the object it names is later mutated.

The **Ships when.** clause ("a module-level mutable binding of a flagged form") may stay as written — there "mutable binding" describes the category of binding the gate guards against, not a mutability test on a literal. `docs/plan_topics/conventions.md` needs no edit: it already describes the gate as covering "the module-level binding forms it can detect" without the misleading literal qualifier. The spec is read-only for this fix.

## Consequence

**Severity:** advisory

The leaf's own text is internally inconsistent: the literal description names a runtime property ("mutable") while the adjacent parenthetical says the scan decides on observable initializer shape, not runtime reuse. An implementer could read "mutable" as a directive to attempt a mutation/aliasing heuristic the AST scan cannot perform, or to exempt never-mutated object/array-literal `const`s, diverging from the intended ban-by-shape behaviour. The parenthetical disambiguates, so a careful implementer still produces a working gate.
