## Finding

Step 2(b) branch (4) (the stale/malformed-marker resolution arm) has exactly five sub-cases. `audit-failures.md` references those same five sub-cases at three sites, using two incompatible numbering conventions:

- In *Stale sub-kinds*, the first two sub-cases are cited with ordinal Arabic labels — "delete the marker line per step 2(b) branch (4) below **sub-case 1** (\"no-surface-on-line\")" and "deleting the marker per step 2(b) branch (4) below **sub-case 2** (\"surface-now-in-inventory\")" — and the closing sentence repeats "sub-case 1" / "sub-case 2".
- In the *Per-family record-shape table* (`#audit-per-family-record-shape-table`), the family-(5) `proposed-resolution` cell enumerates all five sub-cases inline but applies **no** numeric label to the first four (only the parenthetical names appear) and labels the fifth with the Roman numeral "**(v)**".
- The *Note on family (5) routing* likewise calls the fifth "Branch (4) **sub-case (v)**".

So the same closed five-element enumeration is referred to as "sub-case 1/2" (Arabic) in one section and "sub-case (v)" (Roman) in another, while four of the five are entirely unlabeled in the table. A reader cross-referencing the table's "(v)" against *Stale sub-kinds*' "sub-case 1"/"sub-case 2" has no shared label scheme to anchor on, and cannot tell from the table which inline item is sub-case (i)–(iv).

The defect is purely presentational: it does not change any audit behaviour. The wire discriminator carries an implementation-chosen `audit/<class>/<family>/<symptom>` token, never these sub-case ordinals; the numbers exist only to let humans cross-reference the prose.

## Solution Space

**Shape:** single

### Recommendation

Adopt a single Roman-numeral parenthetical scheme (i)–(v) for the five branch-(4) sub-cases throughout `audit-failures.md`:

- *Stale sub-kinds*: change "below sub-case 1 (\"no-surface-on-line\")" → "below sub-case (i) (\"no-surface-on-line\")", "below sub-case 2 (\"surface-now-in-inventory\")" → "below sub-case (ii) (\"surface-now-in-inventory\")", and the closing "(sub-case 1)" / "(sub-case 2)" → "(sub-case (i))" / "(sub-case (ii))".
- *Per-family record-shape table*, family-(5) `proposed-resolution` cell: prefix the first four inline items with "(i)"–"(iv)" so the existing "(v)" completes a contiguous (i)–(v) sequence — "(i) delete the marker line (no-surface-on-line …); (ii) delete the marker (surface-now-in-inventory); (iii) rewrite the marker's citation; (iv) rewrite the marker's grammar; (v) delete the marker and resolve the underlying surface via branch 1/2/3 …".
- *Note on family (5) routing*: "Branch (4) sub-case (v)" already conforms; leave it.

Roman (i)–(v) is the form to standardise on because the fifth sub-case already carries "(v)" and choosing Roman keeps the sub-case labels visually distinct from the family/branch Arabic ordinals (1)–(5) used elsewhere in the same cluster.

Edge case for the implementer: keep the branch-(4) sub-case labels (i)–(v) distinct from the family-(5) stale-sub-kind `<symptom>` tokens (s1)/(s2) — they are different enumerations. Sub-case (i) (no-surface-on-line) corresponds to `<symptom>` (s1) and sub-case (ii) (surface-now-in-inventory) corresponds to (s2); do not renumber or merge the (s1)/(s2) tokens, which are wire-relevant. Note also that `version-bump-step2b.md`'s branch (4) states its five sub-cases as prose conditionals with no numeric labels at all, so there is no existing label scheme there to mirror — this edit is confined to `audit-failures.md`.

## Consequence

**Severity:** cosmetic

A contributor routing a family-(5) red to its branch-(4) sub-case must reconcile two label schemes for one enumeration, and the table's unlabeled first four items give no anchor for the table's lone "(v)". No implementation diverges and no wire contract is affected; the cost is reader friction only.
