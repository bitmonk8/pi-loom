## Problem

`V15i` closes three normative export-visibility obligations from `imports.md` (the implicit auto-export of every top-level `schema`/`enum`/`fn`, the aliased `export … from` re-export form, and the negative rule that a plain `import` is not re-exported), none of which carries a numbered REQ-ID or registry code. Per the un-anchored-obligation discipline (GOV-22 residue), every such MUST on a non-narrative spec page must be enumerated in `coverage-matrix.md` under *Code-keyed obligation areas* with a named closing leaf. `V15i` has no row in either matrix table, and all three of its `Tests` bullets — duplicated into `V15i-T` — cite only prose paraphrases (`Visibility rule in imports.md`, `Re-exports rule`, `negative half of the Re-exports rule`) with no `cka-<n>` token. Because the §Visibility/§Re-exports text is phrased without literal `MUST` tokens, the `H5e` token scan does not surface these obligations either, so they ship untracked and pass the coverage gate vacuously.

## Issue introduction

**Verdict:** present-since-inception (of the `V15i` export-visibility Tests bullets)
**Introducing commits:** `2316cac` (`pi-loom plan: resolve "V15c asserts only import failure paths"`, 2026-06-11) authored the three export-visibility bullets with prose-paraphrase citations and added no coverage-matrix row; relocated verbatim into the new `V15i` / `V15i-T` leaves at `6b7c7e8` (`pi-loom plan: resolve "V15c bundles import resolution/diagnostics and export-visibility into one leaf"`, 2026-06-18), again with no row.
**History:** `git log -S "negative half of the Re-exports rule"` traces the bullets' first appearance to `2316cac`, which added them to `V15c-imports.md` to close a finding that `V15c` tested only import failure paths; they entered prose-only (`(Visibility rule in imports.md)`, `(Re-exports rule)`, `(negative half of the Re-exports rule)`) with no `cka-<n>` token. `git log -S "V15i" -- docs/plan_topics/coverage-matrix.md` returns nothing, and `git log -S "imports.md" -- docs/plan_topics/coverage-matrix.md` shows only the corpus relocation/scaffold commits — confirming `imports.md`'s export-visibility obligations have never had a code-keyed-area row. The `V15c → V15i` split at `6b7c7e8` carried the bullets unchanged into the dependent leaf. The untrackability gap is original to the bullets, not a later regression; no edit ever narrowed an earlier, correctly-anchored citation.

## Solution approach

Add a *Code-keyed obligation areas (no numbered REQ-IDs)* row to `coverage-matrix.md` covering the three `imports.md` (IMP) §Visibility + §Re-exports export-visibility obligations, taking the next free append-only token (the table currently stops at `cka-46`) and naming `V15i` as the closing leaf. Cite that token inline in the three `V15i` `Tests` bullets and the three matching `V15i-T` `Tests` bullets, replacing the prose paraphrases, following the code-keyed-area citation form already used elsewhere in the plan.

## Solution constraints

- `imports.md` and the spec are read-only for this fix; re-anchoring these obligations as numbered `IMP-*` REQ-IDs is a spec-side GOV-22 decision owned by a separate spec-coverage finding.
- The new row's closing-leaf cell must be backtick-delimited so `H5d`'s tokeniser reads it (per the *Closing-leaf column authoring* rule).
