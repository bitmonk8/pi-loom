## Problem

RET-1 mandates that `return expr` is type-checked against the enclosing scope's declared return type, and FN-3 mandates that a `fn` with an explicit return annotation type-checks its tail and every `return` operand against that annotation. Both obligations require a static `⊑` check that can fail, but neither names a diagnostic code for the failure, and the `loom/parse/*` registry contains no code for this site. The two adjacent registry codes cover different cases: `loom/parse/return-no-common-type` is the annotation-less / inference case and `loom/parse/bare-return-in-non-void` is the no-operand case, leaving the annotated-mismatch case orphaned. Under the closed diagnostic registry, conformance tests key on codes, so the rejection is currently untestable and implementations may invent divergent codes.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** none single; the obligation has lacked a named code since the return-statement semantics were first written. Earliest tracked appearance: `fecb504` (2026-05-04, "Split spec.md and plan.md into per-topic / per-phase files"), where `return.md` already reads "`return expr` is type-checked against the enclosing scope's declared return type" with no diagnostic named. The semantics predate the split (present in the `53f5831`/`83802db` rewrites of spec.md).
**History:** The defect is an omission carried through every subsequent diagnostic-naming pass without being closed. `839db71` ("Diagnostic codes are declared as namespaces but never assigned") assigned the bare-`return` case its code (`loom/parse/bare-return-in-non-void`) but not the `return expr` mismatch case. `648365c` ("Three subtype check sites name no diagnostic codes") introduced TYPE-9 naming codes for the typed-`let`, `fn`-argument, and ternary sites and explicitly scoped itself to "three" sites, excluding the annotated return-type check. `08886d9` (2026-06-07) added FN-3's "type-checks its tail and every `return` operand against that annotation" prose and registered the *inference-only* `loom/parse/return-no-common-type`, but added no code for the annotated-mismatch case. `c0c6853` (2026-06-25) coined the RET-1..3 identifiers, formalising the obligation in RET-1 while still naming no code. The token `loom/parse/return-type-mismatch` has never appeared in the spec corpus (only in review/store artefacts). The durable fix addresses the standing omission, not any single commit.

## Solution approach

Add a parse diagnostic for the case where, in a scope with an explicitly declared return annotation, the tail expression's type or any `return expr` operand's type is statically incompatible with the declared annotation. Register it as a row in the `loom/parse/*` table in `docs/spec_topics/diagnostics/code-registry-parse.md`. Extend TYPE-9's per-site enumeration in `docs/spec_topics/type-system.md` to cover the annotated return-type check and cite the new code. Name the code inline at RET-1 and at FN-3's tail/operand-against-annotation sentence so each obligation points to its code.

## Solution constraints

- The new code fires only when an explicit return annotation exists; do not displace `loom/parse/return-no-common-type`, which owns the annotation-less / inference case.
- Out of scope: bare `return` (owned by `loom/parse/bare-return-in-non-void`), the `void`-annotated tail-discard case (FN-4), and the `?`-operator `Err`-arm operand check (owned by `loom/parse/question-on-non-result`).
