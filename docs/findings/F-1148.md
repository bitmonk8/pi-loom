## Finding

Two clarity defects in the same paragraph of `diagnostic-emission-isolation.md` — independent in scope but both on the *details*-shape / count-semantics seam:

**(1) The `*details* nesting convention*` clause names the upstream pin as "the closed-at-`{reason}`-in-loom-1.0.0 pin governing additions".** This is a verbatim allusion to the bold heading `**Closed at \`{reason}\` in loom 1.0.0.**` in `diagnostics/diagnostic-shape.md`'s *`details` shape conventions for the `session_shutdown` event payload* paragraph, where `{reason}` is the literal `details.event.reason` field name (it is the *field* the closed-shape pin closes, not a stand-in to be resolved). Read in isolation — i.e. by anyone who has not already opened the cross-referenced anchor — the token reads as an un-substituted template placeholder, especially since the surrounding prose uses the `{reason}` form precisely where a Mustache-style reader would expect a placeholder slot. The cross-reference is the only thing telling the reader what the pin *does* (forbids additional SDK-derived `details.event.*` siblings between SDK-pin bumps); the catch-phrase token itself communicates nothing without that round-trip.

**(2) The *Count semantics — invocation-site framing* clause says emission-count bounds are "measured at the **handler invocation site** … for the same `session_shutdown` event", but the same clause's own enumeration explicitly includes "the **idempotent** clause above's at-most-once-per-extension-instance-lifetime bound on the same code" (i.e. on `loom/host/session-shutdown-runtime-degraded`).** A per-extension-instance-lifetime bound necessarily ranges over *multiple* `session_shutdown` events delivered to the same instance — exactly the multi-`session_shutdown` delivery that host-prerequisite (b) admits and that the **idempotent** clause's broadened short-circuit handles. The "for the same `session_shutdown` event" qualifier is correct for the per-event bounds in the same enumeration (`reason-unknown`, `pinned-constant-unreadable`, `teardown-step-failed`, `runtime-degraded` per-branch, `reload-teardown-timeout` per-event, per-invocation note per-invocation) but cannot describe the lifetime bound — and the clause never says how invocation-site counts compose across events to deliver "at most once per extension-instance lifetime". An implementer authoring fixtures for branch (h) of `session-only-degraded-state.md`'s acceptance criteria (which assert "zero additional rows" on follow-up `session_shutdown` events to preserve that lifetime bound) and the all-three-throw corner case (which explicitly admits one additional row per follow-up event for that case only) cannot read this clause to derive both behaviours; the cross-event composition rule has to be reconstructed from the **idempotent** clause's prose rather than read off the *Count semantics* pin that claims to govern the bound.

Neither defect is a logic error in the underlying contract — the upstream pin and the lifetime bound are well-specified at their owning sites (`diagnostic-shape.md`'s closed-shape paragraph and `drain-state-contract.md`'s idempotent-handler paragraph respectively). The defect is that `diagnostic-emission-isolation.md` adopts their conclusions in a form that requires the reader to round-trip to the upstream site to recover meaning the clause claims to convey.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `52b9101` (2026-05-11) — pi-loom spec: resolve "T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state"
**History:** Both defective phrasings entered the corpus in the same commit. Before `52b9101` the *Diagnostic-emission isolation* paragraph on `pi-integration-contract.md` covered four teardown-handler codes with a single bare-`code` serialiser-throw fallback and pinned no cross-event count semantics. `52b9101` (the T24 resolution) introduced the fifth teardown-handler code `loom/host/session-shutdown-runtime-degraded`, the nested `details.event.{reason}` wire-format convention with its closed-shape pin, the *Two-token fallback for nested-shape codes* sub-clause, the **idempotent** clause's at-most-once-per-extension-instance-lifetime bound, and the *Count semantics — invocation-site framing* clause — all in one edit. The "closed-at-`{reason}`-in-loom-1.0.0 pin governing additions" catch-phrase appears in this commit's `*details* nesting convention` clause; the *Count semantics* clause's "for the same `session_shutdown` event" qualifier and its enumeration item naming the lifetime bound also both appear here. The defect is therefore the same single commit: a large multi-concern edit that introduced both the lifetime bound and the count-semantics clause that names it, without reconciling the latter's "same event" framing against the former. Subsequent commits (`5e5f2f1` frozen-baseline reclass, `c326bfa` clarity sweep, `f5e89f4` 100KB split, `035d82e` per-invocation surface scope-fix) edited the surrounding paragraph but preserved both phrasings verbatim through `git show`.

## Solution Space

**Shape:** multiple

The finding bundles two independent obligations — paraphrasing the closed-`{reason}` cross-reference and disambiguating the per-event-vs-per-lifetime count framing. Resolve them in separate passes so each lands a small, separately-reviewable diff.

### Option A — Paraphrase the closed-`{reason}` cross-reference (fix first)

**Approach.** Rewrite the catch-phrase token at the one site that uses it. Replace "the closed-at-`{reason}`-in-loom-1.0.0 pin governing additions" with prose that names the contract in plain language and links to the owning anchor.

**Spec edits.** In `docs/spec_topics/pi-integration-contract/diagnostic-emission-isolation.md`, in the `*details* nesting convention*` sentence, replace the offending fragment with something on the order of: "the loom 1.0.0 closure that pins the `details.event.*` shape to the single SDK-derived sibling `reason` and forbids further SDK-derived siblings between Pi-SDK-pin bumps". Retain the existing markdown link to `diagnostics/diagnostic-shape.md#session-shutdown-details-conventions`. No semantic change — the underlying pin is unchanged at its owning site.

**Pros.** Tiny localised edit. Removes the only spec-corpus token that reads as an unresolved template. No downstream cross-references break.

**Cons.** None of substance; the rewrite is purely editorial.

**Risks.** A future contributor re-introducing the catch-phrase under a "tighten the prose" sweep. Mitigatable by leaving the descriptive form in place — it is harder to compress back into the catch-phrase.

### Option B — Disambiguate per-event vs per-extension-instance-lifetime in *Count semantics* (fix second, on top of A)

**Approach.** Split the "for the same `session_shutdown` event" qualifier into a per-bound-scope sentence rather than letting it govern the whole rule, and state explicitly how invocation-site counts compose across events to deliver the lifetime bound.

**Spec edits.** In the *Count semantics — invocation-site framing* clause of `docs/spec_topics/pi-integration-contract/diagnostic-emission-isolation.md`:
1. Restate the invocation-site framing as scope-agnostic ("the count is over the number of times the handler enters the leaf-owned serialisation-and-emission sequence the rule wraps").
2. Add an explicit "Scope of the count" sub-clause enumerating the three scopes the enumeration uses: (i) **per-event** (`reason-unknown`, `pinned-constant-unreadable`, `teardown-step-failed` per failing call site, `runtime-degraded` on the per-event-in-this-branch bound, `reload-teardown-timeout`); (ii) **per-cleanly-cancelled-invocation** (`cancelled-by-session-shutdown`); (iii) **per-extension-instance-lifetime** (`runtime-degraded`'s idempotent-clause bound), and state that the lifetime-scoped bound counts invocation-site enterings summed across **every** `session_shutdown` event the same instance receives, subject to the **idempotent** clause's short-circuit (which is how the bound is delivered — not via per-event suppression).
3. Cross-link the all-three-throw and `readDrainState`-throw carve-outs from `drain-state-contract.md` so the +1-per-follow-up-event behaviour is reachable from this clause without re-reading the idempotent paragraph.

**Pros.** Resolves the only correctness-relevant ambiguity in the paragraph. Branch (h) of `session-only-degraded-state.md`'s acceptance criteria and the all-three-throw exception become derivable from the *Count semantics* clause directly.

**Cons.** Lengthens an already-dense paragraph by one structured sub-clause.

**Risks.** Wording the lifetime scope precisely enough to admit the all-three-throw +1 carve-out without re-opening the bound. The carve-out text in `drain-state-contract.md` is the authoritative phrasing; cite it rather than restating.

### Recommendation

Land A first as a self-contained editorial fix; land B in a follow-up commit. Order matters because B's edits touch the same paragraph and a sequential land keeps each diff small enough for the per-finding fix loop to converge. The lifetime-bound disambiguation in B is the substantive change — it removes the cross-event composition question — and the implementer should explicitly cite both `drain-state-contract.md`'s **idempotent** clause anchor and the all-three-throw / `readDrainState`-throw carve-outs so the bound and its exceptions are co-located rather than scattered.

Edge cases the implementer must keep in view while landing B: (i) the per-invocation `cancelled-by-session-shutdown` bound is per-cleanly-cancelled-invocation, **not** per-event — the rewrite must not collapse those two scopes; (ii) the **Unknown-reason rule** emission that runs *before* sub-step 1 is still per-event and the bound preservation under swallow must still hold; (iii) the bare-`code` and two/three-token fallback forms remain part of the *same* wrapped sequence and do not count as second invocations against any scope.

## Consequence

**Severity:** correctness

Two reasonable implementers, reading the *Count semantics* clause without already knowing the **idempotent** clause's bound, will diverge on whether `loom/host/session-shutdown-runtime-degraded` emission counts compose across `session_shutdown` events: one will read "for the same `session_shutdown` event" as scoping the whole rule and write per-event-resetting fixtures (which then over-pass on follow-up events), the other will infer the lifetime bound from the idempotent-clause cross-reference. The closed-`{reason}` catch-phrase is a softer clarity defect — readers can recover the meaning via the cross-link — but it still costs every reviewer a round-trip on every encounter.
