## Problem

The transient list of parsed `.loom` looms that the factory builds at registration step 2 and that the `session_start` handler later drives through `pi.registerCommand` is named three different ways across the V9 plan files, with no equivalence note tying them together. `V9b` **Adds** calls it the bare **"pending list"**; `V9k` **Adds** and `coverage-matrix.md` (the `extension-bootstrap-and-per-loom.md` §Extension-bootstrap SDK failures row) call it **"pending-list drop"**. The spec fixes a canonical term — `registration-steps.md` step 2 and `extension-bootstrap-and-per-loom.md` both use **"pending-registration list"** — and both plan coinages diverge from it, so a reader cross-referencing V9b against V9k has no in-plan signal that the two name one structure.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** c6a664e — pi-loom plan: build/update plan for spec.md + review (2026-06-10, Thomas Andersen); 6df646a — pi-loom plan: resolve "extension-bootstrap SDK-failure rule has no closing leaf" (2026-06-12, Thomas Andersen)
**History:** V9b coined the bare "pending list" form when the plan was first built (c6a664e, 2026-06-10). Two days later, commit 6df646a added the V9k closing leaf and its coverage-matrix row using the spec-aligned "pending-registration list" and the adjectival "pending-list drop" forms, without reconciling them against V9b's pre-existing coinage. The divergence is the interaction of the original V9b naming and the later V9k/matrix naming.

## Solution approach

Adopt the spec's canonical term **"pending-registration list"** uniformly. Rename the step-2 token `pending list` in `V9b-registration-reload-wiring.md` **Adds** to the canonical form, rename the `pending-list drop` token in `V9k-extension-bootstrap-failures.md` **Adds** to the `pending-registration list` form, and rename the `pending-list drop` token in the `coverage-matrix.md` Extension-bootstrap SDK failures row likewise. `V9k`'s `pi.getCommands()` read-failure Test bullet already uses "pending-registration list" and needs no change.

## Solution constraints

- The spec is read-only for this fix — it is the reference for the canonical term and is not edited.
