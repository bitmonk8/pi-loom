## Finding

`runtime-event-channel.md` names `Clock.wallNow()` in four places — once in the `RuntimeEvent.occurred_at` TS comment (`Unix epoch ms, stamped at the originating emission site via Clock.wallNow()`) and three times in the best-effort fallback prose, including normative claims about its throw behaviour (`When the thrower is Clock.wallNow() during occurred_at stamping … the runtime MUST NOT re-invoke Clock.wallNow() in the fallback`). Nothing on the page introduces the `Clock` abstraction, names it as a DI seam, states the return contract of `wallNow()`, or links to where any of that lives.

The defining sites do exist: PIC-12 (`Clock` / `FakeClock` interface) at `host-interfaces-services.md#clock--fakeclock-interface` declares `wallNow(): number` as Unix epoch milliseconds, pins the per-runtime DI construction, and bans direct `Date.now` / `performance.now` / `Date.prototype.getTime` use outside the `WallClock` adapter; `implementation-notes.md` carries a parallel summary. Neither is cross-referenced from `runtime-event-channel.md`, so a reader landing on the runtime-event channel through a deep link (or after a `/tree` jump) has no path to the seam, no confirmation that `Clock` is the injected service rather than ad-hoc nomenclature, and no place where the "can throw" property the fallback relies on is grounded.

The defect is the interaction: not any single commit's edit, but the missing link that the split introduced and the subsequent expansion entrenched.

## Solution Space

**Shape:** single

### Recommendation

Add an in-shard cross-link from each occurrence of `Clock` / `Clock.wallNow()` in `runtime-event-channel.md` to PIC-12 at `host-interfaces-services.md#clock--fakeclock-interface`. Two link sites are sufficient because the four mentions cluster into two paragraphs:

1. At the `occurred_at` field comment in the `RuntimeEvent` TS block, change the trailing comment to reference the seam — e.g. `// Unix epoch ms, stamped at the originating emission site via [Clock.wallNow()](./host-interfaces-services.md#clock--fakeclock-interface)` (or, since Markdown links inside TS code fences are inert, put the link in a one-line prose sentence immediately following the code block: "`occurred_at` is stamped via [`Clock.wallNow()`](./host-interfaces-services.md#clock--fakeclock-interface), the per-runtime `Clock` DI seam defined at PIC-12.").
2. In the best-effort fallback paragraph, link the first `Clock.wallNow()` mention to the same anchor; subsequent mentions in the same paragraph need no further link. While editing that paragraph, lean on the seam's contract for the "can throw" property — e.g. "When [`Clock.wallNow()`](./host-interfaces-services.md#clock--fakeclock-interface) throws during `occurred_at` stamping …" — so the throw property is grounded in the linked definition rather than asserted naked.

No new normative content is required: PIC-12 already pins the return contract, the per-runtime DI construction, and the timing-source ban; the page only needs to point at it. Implementer edge case to watch: do not duplicate PIC-12's interface signature or its `Date.now`/`performance.now`/`getTime`/`setTimeout`/`clearTimeout` ban inline — both are PIC-12's to own, and a copy here would create a dual-anchor (GOV-1) problem the next review would flag.

## Consequence

**Severity:** advisory

An implementer reading the runtime-event channel page in isolation cannot tell, from this page alone, that `Clock` is the injected per-runtime DI seam, that `wallNow()` returns Unix epoch milliseconds, or that direct `Date.now()` use is banned. The "can throw" carve-out in the fallback hangs on a property never stated on-page. A careful reader will grep to PIC-12; a less-careful one may inline `Date.now()` and breach the timing-source ban without an in-shard cue.
