## Finding

`ERR-7` ("watcher-time reload failures") is enumerated at
`errors-and-results/error-model.md#err-7` and elaborated at
`discovery/package-and-settings.md#watcher-time-reload-failures` as the
pre-evaluation failure surface that fronts the watcher / hot-reload path. The
two surfacing paragraphs pin the channel contract (`loom-system-note`,
`triggerTurn: false`, no final value, not subject to cancellation) and the
two underlying arms (`loom/runtime/registry-swap-failed` for a thrown rebuild
step; a re-emitted `loom/load/*` / `loom/parse/*` batch for a watcher-time
parse / merge diagnostic) — but neither paragraph pins the operator-visible
`content` string that the `loom-system-note` actually carries on the wire.

The omission asymmetrically affects the two arms. The
re-parse / re-merge arm produces standard `loom/load/*` / `loom/parse/*`
diagnostics, so the `details: { diagnostics: Diagnostic[] }` row of
`pi-integration-contract/runtime-event-channel.md`'s per-variant
`display` / `content` table already covers it implicitly (serialised
`<file>:<line>:<col>: <code>: <message>` lines). The
`loom/runtime/registry-swap-failed` arm has no such coverage: the only
matching row in that table is the "runtime panic" row, whose
`"loom /<name> aborted: <message>"` template requires a `<name>` (the loom
the invocation is owned by), and a watcher-time registry-swap failure has no
owning invocation — it fires from the watcher's debounced rebuild outside any
slash dispatch. A conformance test can therefore verify that the note fires
and that `details.diagnostics[0].code === "loom/runtime/registry-swap-failed"`,
but it cannot assert on the bytes the operator reads, and two reasonable
implementers can ship visibly different operator surfaces (one borrowing the
panic template and substituting some sentinel for `<name>`, one inventing a
`"loom watcher: registry swap failed: <path>"` template parallel to the
existing `"loom watcher: <N> file(s) added or removed …"` structural-change
note) while both passing every existing assertion.

GOV-15's source-language equivalence promise extends to "equivalent
`loom-system-note` content strings" across the 1.x line; without a pinned
template the equivalence test for this surface has no left-hand side.

## Solution Space

**Shape:** single

### Recommendation

Pin a normative `content` template for both arms of `ERR-7` at the
`Watcher-time reload failures` section in
`docs/spec_topics/discovery/package-and-settings.md`, and cross-reference
those templates from the `ERR-7` list item in
`docs/spec_topics/errors-and-results/error-model.md` and from the
per-variant `display` / `content` table in
`docs/spec_topics/pi-integration-contract/runtime-event-channel.md`.

Concretely, extend the two arm bullets at
`#watcher-time-reload-failures` to spell out the `content`:

- **Registry-swap failure** arm — `content` is
  `"loom watcher: registry swap failed: <path>"`, where `<path>` is the
  failing file path the `loom/runtime/registry-swap-failed` registry row
  already names in its Message template (`registry swap failed: <path>`).
  The `"loom watcher: "` prefix mirrors the existing structural-change
  watcher note (`"loom watcher: <N> file(s) added or removed …"`) at
  `pi-integration-contract/registration-steps.md#watcher-hot-reload-registration`,
  giving operators a single recognisable watcher-prefix family. The
  emission is the `details: { diagnostics: [Diagnostic] }` shape carrying
  the single `loom/runtime/registry-swap-failed` element; `display: true`.
- **Re-parse / re-merge diagnostic** arm — `content` is the serialised
  `<file>:<line>:<col>: <code>: <message>` line(s) defined under
  `diagnostics/diagnostic-shape.md` *Serialised content format*, identical
  to the load / parse batch row of the per-variant table. The emission is
  the `details: { diagnostics: Diagnostic[] }` shape carrying the
  re-emitted `loom/load/*` / `loom/parse/*` batch; `display: true`.

Add a row (or two rows) to the per-variant `display` / `content` table at
`pi-integration-contract/runtime-event-channel.md#system-note-details-shapes`
for the watcher-time variants so the table is closed over the actually-shipped
shapes and the registry-swap arm is no longer covered only by the
panic row (whose `<name>` placeholder does not apply here).

Implementer edge cases to watch:

- The watcher-time `registry-swap-failed` note is fired outside any
  slash invocation; the `RuntimeEvent` payload's `loom` field has no
  natural value here. Renderers and conformance fixtures must treat the
  diagnostic-shape emission (`details.diagnostics`) as authoritative for
  this code and must not require the `details.event.loom` field to be
  populated for the watcher-time emission.
- `<path>` rendering MUST follow the same path-rendering rule the
  `loom/runtime/registry-swap-failed` registry row already uses (absolute
  path of the file whose change triggered the rebuild); cite the registry
  row from the new template rather than re-defining the rule.
- GOV-15's equivalence test for `loom-system-note` content now has a
  concrete anchor for `ERR-7`; ensure the `Source-language stability`
  paragraph at `overview-and-orientation.md` does not need a parallel
  edit (it cites the channel generically and should not need touching).

## Consequence

**Severity:** correctness

Without a pinned `content` template for `ERR-7` emissions, conformance
tests cannot fix the operator-visible surface, two reasonable implementers
ship visibly different `loom-system-note` bytes for the same registry-swap
failure, and GOV-15's 1.x source-language equivalence over
`loom-system-note` content has no anchor against which to compare. The
re-parse / re-merge arm is only implicitly covered by the existing
parse / load batch row in the per-variant table; the registry-swap arm is
uncovered by any row.
