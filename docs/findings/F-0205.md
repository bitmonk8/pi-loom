## Finding

H2a's `no-restricted-syntax` blocking-runtime check flags the statically named `*Sync`/`execSync` call forms over `src/**` "excluding `**/*.test.ts` and the `V8*` FileSystem seam-adapter modules that legitimately own file/process I/O." conventions.md repeats the same exclusion at the *Sequential by default* enforcement posture and at release-time residue-inspection item 3. But the exclusion is named only descriptively — no leaf declares the concrete file path or glob that identifies the "V8* FileSystem seam-adapter modules" the ESLint config must subtract from its scan. H2a does not state the path; V8b (which adds the `FileSystem` seam) does not publish where the adapter lives in `src/**`; H3a declares seam *interfaces* only, not adapter module locations.

The dependency chain makes this worse: H2a → H3a → V8b, so H2a (and the ESLint config it wires) is authored before any `V8*` adapter module exists. The implementer writing H2a's exclusion glob has no source to copy the path from, and the implementer writing V8b has no stated path to conform to. The two ends are not pinned to each other.

By contrast, the parallel *No globals, statics, singletons* ambient-access ban (H3a / conventions.md) takes a comment-keyed-only approach (`// allow-ambient: <primitive> — <seam>`) with no path-based exclusion, so it has no equivalent boundary-declaration gap. The blocking-runtime check is the only gate that subtracts a path region whose extent is never declared.

## Issue introduction

**Verdict:** introduced-by-a-single-commit
**Introducing commits:** `c33d309` — "pi-loom plan: resolve \"Blocking-runtime ban concedes enforcement to manual review\"" (Thomas Andersen, 2026-06-17)
**History:** `git log -S 'FileSystem seam-adapter' -- docs/plan_topics/` returns exactly one commit, `c33d309`; the same commit is the sole match for `git log -S 'allow-sync'` and for the H2a blocking-runtime-check additions. The diff (`git show c33d309 -- docs/plan_topics/H2a-cross-cutting-gates.md`) shows the `*Sync`/`execSync` mechanical check, the `// allow-sync:` exemption, and the "excluding ... the `V8*` FileSystem seam-adapter modules" phrasing all added together as `+` lines; before this commit the blocking-runtime ban was conceded to manual review with no mechanical gate and no path exclusion. The defect is an introduced-by-resolution side effect: converting the manual-review concession into a mechanical gate introduced a path-based exclusion without pinning the excluded region's module-path boundary. The phrasing has not been touched in any later commit.

## Solution Space

**Shape:** multiple

### Option A — V8b publishes the adapter path/glob, H2a references it

**Approach.** V8b's `Adds.`/`Ships when.` declares the concrete `src/**` path/glob its `FileSystem` seam-adapter module(s) occupy; H2a's exclusion and the two conventions.md sites reference that published path as the authoritative source.
**Plan edits.** Add the path declaration to `docs/plan_topics/V8b-filesystem-seam.md`; change H2a's exclusion and conventions.md to cite V8b's declared path rather than the bare descriptive phrase.
**Spec edits.** None.
**Pros.** The path lives at the leaf that owns the adapter; single source of truth.
**Cons.** Creates a forward reference — H2a's ESLint config (authored first, per `Deps. H1a`) must consume a path that only exists once V8b lands; the H2a leaf cannot be fully realised against a stable boundary at authoring time.
**Risks.** The forward coupling can leave H2a's wired glob stale if V8b later relocates the adapter.

### Option B — H2a declares the boundary; V8b conforms to it

**Approach.** H2a's `Ships when.`/`Adds.` states the concrete `src/**` glob the blocking-runtime check excludes; V8b's `Ships when.` adds that the `FileSystem` adapter is placed at that path.
**Plan edits.** Add the literal glob to `docs/plan_topics/H2a-cross-cutting-gates.md` (and the two conventions.md sites); add a conformance line to `docs/plan_topics/V8b-filesystem-seam.md` requiring the adapter live at that path.
**Spec edits.** None.
**Pros.** The gate and its excluded scope are pinned together at the leaf that wires the gate, with no forward reference; V8b has a concrete target to satisfy.
**Cons.** Couples the convention to a physical directory layout chosen at H2a time; relocating the adapter later requires editing both leaves.
**Risks.** If V8b's placement drifts from H2a's glob, the gate silently mis-scopes — needs the V8b conformance line to be observable in its tests.

### Option C — Drop the path exclusion; rely on `// allow-sync:` comments only

**Approach.** Remove the `V8*` FileSystem seam-adapter path exclusion entirely. The seam adapter's legitimate synchronous calls each carry an in-source `// allow-sync: <reason>` comment — the exemption mechanism H2a already defines — mirroring the comment-only *No globals, statics, singletons* ambient ban, which has no path exclusion.
**Plan edits.** Strike "and the `V8*` FileSystem seam-adapter modules" from H2a's Adds and the conventions.md *Sequential by default* enforcement posture; reword residue-inspection item 3's "outside the `V8*` FileSystem seam" so the only exemption channel is the `// allow-sync:` comment; drop the H2a Tests bullet clause "a `V8*` FileSystem seam-adapter module ... are unrestricted."
**Spec edits.** None.
**Pros.** Eliminates the undeclared boundary outright; no forward reference; consistent with the parallel ambient ban's comment-only model; the exemption is self-locating and grep-auditable.
**Cons.** Every legitimate sync call in the seam adapter must carry an `// allow-sync:` comment, which is more verbose if the adapter has many such call sites.
**Risks.** If the seam uses no `*Sync`/`execSync` forms at all (e.g. it routes all I/O through `fs.promises`), the exclusion was never needed and Option C is a pure simplification; verify the adapter's actual sync surface when applying.

### Recommendation

Apply Option C if the `FileSystem` adapter's synchronous-call surface is small (or empty), because it removes the boundary-declaration problem entirely and aligns the blocking-runtime ban with the comment-only ambient ban already in conventions.md. If the adapter genuinely needs many sync call sites, apply Option B so the gate and its excluded scope are pinned together without the Option A forward reference. Either way, the descriptive phrase "the `V8*` FileSystem seam-adapter modules" must resolve to a concrete, observable exemption channel (a declared path or an `// allow-sync:` comment) at the leaf that wires the check. Edge case the implementer must confirm: whether the seam actually issues any `*Sync`/`execSync` calls — that answer decides whether any exemption channel is needed at all.

## Consequence

**Severity:** correctness

The blocking-runtime ESLint exclusion cannot be configured against an undeclared boundary: an implementer picks some glob, and if it is broader than the real adapter the check over-excludes production code (false green — a synchronous `*Sync` stall ships in `src/**` and blocks Pi's shared event loop), while if it is narrower or mismatched it flags the seam's own legitimate sync calls (false red). Because H2a is authored before V8b, two reasonable implementers will diverge on the path, and the gate's correctness silently depends on the V8b adapter happening to land where H2a's glob guessed.
