## Finding

GOV-31 (the integer-count aggregator literal-preservation invariant, `governance/req-id-prefix-table-active-b.md#gov-31`) defines each integer-count aggregator's *source set* by reference to a set of named locations: "[`spec.md` — Scope](../overview-and-orientation.md#scope)", "[`spec.md` — Prerequisites](../overview-and-orientation.md#prerequisites)", "[`spec.md` — Hard ceilings](../overview-and-orientation.md#hard-runtime-ceilings)", and "The orientation prose literal that names the count on `spec.md`". The link *targets* resolve correctly to `overview-and-orientation.md`, where the Scope, Prerequisites, and Hard-ceilings sections actually live (verified: `docs/spec.md` is a three-link contents hub with no Scope/Prerequisites/Hard-ceilings sections, and `docs/spec_topics/overview-and-orientation.md` carries `## Orientation` → `### Prerequisites` / `### Scope` plus the `#hard-runtime-ceilings` anchor). The visible link text, however, asserts the host file is `spec.md`.

GOV-31's own prose compounds the mismatch: it speaks of "the aggregator's host paragraph on `spec.md`" and "the orientation prose literal that names the count on `spec.md`", but no such paragraphs exist on the hub. A maintainer applying GOV-31's literal-preservation MUST — locate the aggregator paragraph, confirm a parseable integer literal survives, confirm its value equals the source-set count — is told the paragraph lives on `spec.md` and would find nothing there. The resolving links rescue navigation, but the label and the surrounding prose disagree with where the content actually is.

This is one instance of a corpus-wide pattern in which `spec.md` is used as a label or referent for content that was split out into `overview-and-orientation.md` (and the other orientation topic pages). GOV-30 uses the same `spec.md`-denotes-aggregators framing, and GOV-23 makes the same claim about a non-existent `spec.md` Session Model section. Either the labels are made to match their resolved targets, or `spec.md` is given an explicit, pinned-once definition denoting the split orientation pages.

At that commit its source-set references already used the label-vs-target mismatch form — `[`spec.md` — Scope](../../spec/overview-and-orientation.md#scope)`, `[`spec.md` — Prerequisites](.../overview-and-orientation.md#prerequisites)`, `[`spec.md` — Hard ceilings](.../overview-and-orientation.md#hard-runtime-ceilings)` — so the defect entered with the rule itself rather than being introduced by a later edit. ## Solution Space

**Shape:** multiple

### Option A — Relabel GOV-31's source-set references to the actual host page

- **Approach:** Change the visible link text in GOV-31 from `spec.md` to `overview-and-orientation.md` (e.g. "[`overview-and-orientation.md` — Scope](../overview-and-orientation.md#scope)"), and revise GOV-31's prose ("the aggregator's host paragraph on `spec.md`", "the orientation prose literal … on `spec.md`") to name the orientation page. Link targets are already correct and unchanged.
- **Spec edits:** Edit GOV-31 only: the three bracketed source-set labels, the "host paragraph on `spec.md`" phrase, and the orientation-prose-literal phrase.
- **Pros:** Smallest, most local diff; label and target agree immediately; no new convention to maintain.
- **Cons:** Leaves GOV-30's identical `spec.md`-aggregator framing untouched, so GOV-30 and GOV-31 now describe the same aggregators using different host names — a fresh internal inconsistency unless GOV-30 is relabelled in the same pass (scope creep).
- **Risks:** Partial application that touches GOV-31 but not GOV-30/GOV-23 trades one mismatch for a cross-rule divergence.

### Option B — Pin a `spec.md`-denotes-orientation-pages definition once

- **Approach:** Add a single normative definition (e.g. in the governance preamble or alongside GOV-30) stating that, for aggregator-reference purposes, `spec.md` denotes the orientation topic pages split out of the former monolithic `spec.md` (`overview-and-orientation.md` et al.), with the per-section resolving links authoritative. GOV-31's existing labels then read correctly under that definition.
- **Spec edits:** One new definition clause; GOV-30 and GOV-31 reference it; GOV-31's labels need no per-site rewrite.
- **Pros:** Co-resolves GOV-30 and the GOV-23 / `schema-subset.md` instances of the same pattern with one pinned mapping; consistent with how `spec.md` is already used throughout the governance rules; no per-label churn.
- **Cons:** Introduces an indirection a reader must internalise ("`spec.md`" is shorthand, not a literal file); slightly larger conceptual surface than a direct relabel.
- **Risks:** The definition must be unambiguous about which pages `spec.md` denotes, or it merely relocates the ambiguity.

### Recommendation

Adopt Option B. GOV-30 already pervasively uses `spec.md` to denote the aggregator host pages and the orientation pages were literally split out of the former `spec.md`, so a single pinned denotation matches existing usage and co-resolves the sibling GOV-30 and GOV-23 instances; relabelling GOV-31 alone (Option A) would leave GOV-30 describing the same aggregators under a different host name. Watch the edge case that the definition must name the exact set of pages `spec.md` denotes (orientation/language-and-architecture/session-model split), since GOV-31's source sets also forward-link to non-orientation pages (`imports.md`, `future-considerations/…`) whose labels are already correct and must not be swept into the shorthand.

## Consequence

**Severity:** advisory

A maintainer applying GOV-31's literal-preservation MUST is directed to a "host paragraph on `spec.md`" that does not exist, and cannot tell from the labels alone which file actually hosts each integer-count aggregator. The resolving links still lead to the right sections, so the obligation remains followable; the cost is interpretive friction and a standing internal contradiction between GOV-31's labels/prose and the post-split file layout.
