## Finding

The `buildSessionContext` paragraph in `host-interfaces-core.md` closes with: "The runtime MUST NOT substitute `ctx.getContextUsage()` for per-turn token accounting; that API reports aggregate session usage, not per-turn counts, and is reserved for compaction triggers and footer rendering." The trailing clause states *what `getContextUsage` is used for* without naming the consumer, leaving two opposite readings: (a) loom *does* call it in loom 1.0 for its own compaction-trigger and footer-rendering features, or (b) Pi (and other Pi consumers) call it for compaction triggers and footer rendering; loom 1.0 calls it not at all.

The rest of the page resolves (b) by implication ŌĆö the `ExtensionContext` signature block's `getContextUsage` comment forbids per-turn substitution but says nothing about other loom uses; the per-mode override table forwards `getContextUsage` unchanged in both prompt and subagent mode (with no "loom calls" pin); and there is no callsite for `getContextUsage` anywhere in the spec corpus (only the runtime-event-channel cross-reference observes that custom-message contributions show up in `getContextUsage` *output*, not that loom reads it). But the disputed sentence itself never says so, and a reader landing on it can plausibly infer either reading. The two readings produce different implementations: (a) requires loom 1.0 to construct compaction-trigger and footer-rendering logic against the `ContextUsage | undefined` shape; (b) keeps loom out of that surface entirely.

The companion `compact` row in the same signature block already pins its own loom-1.0 disposition explicitly ("not invoked by loom in loom 1.0 (`await ctx.compact()` is a type-safe no-op ŌĆ”); ŌĆ” forwarded unchanged in both modes, listed here so the touched-surface inventory is exhaustive"), so the finding's title-level grouping of `compact` with `getContextUsage` is misleading ŌĆö `compact` is already unambiguous; only the `getContextUsage` prose is at fault.

## Solution Space

**Shape:** single

### Recommendation

Replace the trailing clause of the disputed sentence with a loom-side disposition that mirrors the `compact` row's pattern. Concretely, in `host-interfaces-core.md`, change

> The runtime MUST NOT substitute `ctx.getContextUsage()` for per-turn token accounting; that API reports aggregate session usage, not per-turn counts, and is reserved for compaction triggers and footer rendering.

to

> The runtime MUST NOT substitute `ctx.getContextUsage()` for per-turn token accounting; that API reports aggregate session usage, not per-turn counts. Loom 1.0 does not invoke `ctx.getContextUsage()` (it is consumed by Pi-side compaction-trigger and footer-rendering logic, not by loom); the member is forwarded unchanged in both modes per the override table below, listed here so the touched-surface inventory is exhaustive.

This brings the `getContextUsage` disposition into lockstep with the existing `compact` row's wording ("not invoked by loom in loom 1.0 ŌĆ” forwarded unchanged in both modes, listed here so the touched-surface inventory is exhaustive"), so the two non-load-bearing `ExtensionContext` members are described on the same footing.

The edit is local to the one sentence; do **not** modify the signature-block `getContextUsage` comment (which already carries the MUST-NOT-substitute pin and the cross-reference to `buildSessionContext`) and do **not** modify the override table (which already lists the member as forwarded unchanged in both modes). Watch for two existing cross-references to `getContextUsage` elsewhere in the corpus ŌĆö `runtime-event-channel.md`'s `convertToLlm` paragraph (which mentions that custom-message diagnostics "contribute to `ctx.getContextUsage()` and compaction decisions" as an *operator-facing* observation about Pi's state, not a loom callsite) and `subagent.md`'s state-isolation matrix row (which describes the forwarded `ExtensionContext` member) ŌĆö both remain consistent with the clarified reading and need no edits.

## Consequence

**Severity:** advisory

A reader of `host-interfaces-core.md` cannot determine from the disputed sentence alone whether loom 1.0 may call `ctx.getContextUsage()`. The signature-block comment and the override-table row corroborate the "loom does not call it" reading on careful reading, but a contributor implementing a footer-rendering or compaction-trigger feature in the loom runtime would have textual cover for the opposite reading and would not be caught by any obligation on the page. The cost is wasted implementation effort and the eventual need to retract a non-conformant surface, not silent runtime divergence.
