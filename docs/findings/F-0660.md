## Problem

In `conversation-drive.md`, both the *Error detection* paragraph (`#prompt-mode-error-detection`) and PIC-18 (`#pic-18`) prohibit using `pi.on("agent_end", …)` as a query-completion signal, but state the prohibition only as a technique-specific MUST NOT motivated by narrative ("cross-fires across concurrent looms or sibling subagents and resolves the wait on the wrong turn"). Neither pins the protected behaviour as a positive, black-box Then-clause keyed on session identity, so a test author cannot mechanically verify that a concurrent session's `agent_end` does not perturb the awaiting session's completion. An implementer who replaces `pi.on("agent_end", …)` with any other cross-session-reachable completion mechanism satisfies the technique-level MUST NOT while still shipping the silent cross-fire bug the rule exists to prevent.

## Issue introduction

**Verdict:** present-since-inception (compound)
**Introducing commits:** `fe67488` (2026-05-04, "pi-loom spec: resolve \"agent_end cross-fires across sessions\""); `0f81627` (2026-06-05, "pi-loom spec: resolve \"Turn-lifecycle subscription surface and pi.on never pinned\"")
**History:** The cross-fire MUST NOT first entered the corpus in `fe67488`, which swapped the prompt-mode completion mechanism from `agent_end` subscription to `await ctx.waitForIdle()` and added the *Error detection*-clause sentence ("The runtime MUST NOT subscribe to the global `pi.on("agent_end", …)` event for query completion: that event fires for every `AgentSession` in the process … so a global handler cross-fires across concurrent looms or sibling subagents and resolves the wait on the wrong turn."). That edit authored the cross-fire justification in declarative prose and added no Given/When/Then scenario.

The PIC-18 paragraph generalising the ban to the full five-event turn-lifecycle set was introduced separately by `0f81627`, which added the `<a id="pic-18">` block with the parallel cross-fire clause ("the prompt-mode driver MUST NOT use these events to resolve query completion … cross-fire from an unrelated session's turn event is harmless — it triggers only a re-check of a non-aborted captured signal"). This edit, too, carried no acceptance scenario.

Neither commit was a behavioural-rule edit gone wrong: both authored the obligation correctly. The testability defect is the silence on Given/When/Then in *both* edits — the spec gained two parallel MUST NOTs that share the same code-inspection-only surface — making this a compound present-since-inception defect rather than a later regression of an originally-testable rule.

## Solution approach

Add a concurrent-session non-interference assertion to `conversation-drive.md` stated as an observable property keyed on which `AgentSession` emitted `agent_end`: an `agent_end` for any other session MUST NOT resolve the awaiting session's `ctx.waitForIdle()`, trigger its trailing-turn `Ok(string)` extraction (`#untyped-query-ok-extraction`), or fire its post-`waitForIdle()` `stopReason`/`errorMessage` probe (`#prompt-mode-error-detection`). Add a forward-link from PIC-18's "MUST NOT use these events to resolve query completion" clause to the new assertion, leaving the existing `pi.on("agent_end", …)` mention as a named anti-pattern.
