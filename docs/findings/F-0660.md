## Finding

Two paragraphs in `conversation-drive.md` carry a MUST NOT against using
the global `pi.on("agent_end", …)` event as a query-completion signal:
the *Error detection* clause inside the untyped-query bullet, and the
parallel statement in **PIC-18. Prompt-mode turn-lifecycle event
subscription** generalising the ban to all five `pi.on` turn-lifecycle
events. Both rely on the same justification — `pi.on` events are
process-global and carry no per-session origin marker, so a global
listener cross-fires across concurrent looms or sibling subagent
sessions and would resolve the wait on the wrong turn.

Neither paragraph states the cross-fire behaviour as something an
implementer can mechanically test. There is no Given/When/Then, no
named scenario, and no pointer to one elsewhere — the prose makes the
obligation purely a code-inspection check ("does the source contain
`pi.on("agent_end", …)`?") rather than a behavioural test. The
testability gap is particularly load-bearing because PIC-18 itself
weakens the cross-fire concern for the non-completion uses
("cross-fire from an unrelated session's turn event is harmless — it
triggers only a re-check of a non-aborted captured signal"), so a
reviewer reading PIC-18 alone could mistake the MUST NOT as advisory.
A behavioural scenario — a concurrent unrelated `AgentSession` fires
`agent_end` mid-query, and the prompt-mode driver does not resolve
its query on that event — would pin the obligation to an observable
outcome rather than to a banned source-code shape.

## Issue introduction

**Verdict:** present-since-inception (compound)
**Introducing commits:** `fe67488` (2026-05-04, "pi-loom spec: resolve \"agent_end cross-fires across sessions\""); `0f81627` (2026-06-05, "pi-loom spec: resolve \"Turn-lifecycle subscription surface and pi.on never pinned\"")
**History:** The cross-fire MUST NOT first entered the corpus in `fe67488`, which swapped the prompt-mode completion mechanism from `agent_end` subscription to `await ctx.waitForIdle()` and added the *Error detection*-clause sentence ("The runtime MUST NOT subscribe to the global `pi.on("agent_end", …)` event for query completion: that event fires for every `AgentSession` in the process … so a global handler cross-fires across concurrent looms or sibling subagents and resolves the wait on the wrong turn."). That edit authored the cross-fire justification in declarative prose and added no Given/When/Then scenario.

The PIC-18 paragraph generalising the ban to the full five-event turn-lifecycle set was introduced separately by `0f81627`, which added the `<a id="pic-18">` block with the parallel cross-fire clause ("the prompt-mode driver MUST NOT use these events to resolve query completion … cross-fire from an unrelated session's turn event is harmless — it triggers only a re-check of a non-aborted captured signal"). This edit, too, carried no acceptance scenario.

Neither commit was a behavioural-rule edit gone wrong: both authored the obligation correctly. The testability defect is the silence on Given/When/Then in *both* edits — the spec gained two parallel MUST NOTs that share the same code-inspection-only surface — making this a compound present-since-inception defect rather than a later regression of an originally-testable rule.

## Solution Space

**Shape:** single

### Recommendation

Add an inline acceptance scenario to the PIC-18 paragraph (the
generalised, owning site for the cross-fire obligation), keyed by the
existing `<a id="pic-18">` anchor so the *Error detection* clause's
cross-fire sentence can either inherit it by cross-reference or be
collapsed into a PIC-18 pointer:

> *Acceptance.* Given a prompt-mode loom invocation that has issued
> `pi.sendUserMessage(text)` on the user session and is awaiting
> `ctx.waitForIdle()`, and a concurrently-live unrelated `AgentSession`
> (sibling subagent invocation or another extension's session) that
> fires `agent_end` for its own turn before the user session's turn
> completes, when the unrelated `agent_end` is delivered to every
> registered `pi.on("agent_end", …)` handler in the process, then
> `ctx.waitForIdle()` MUST NOT resolve on that event and the
> prompt-mode driver MUST NOT extract a trailing-turn `Ok(string)`,
> `stopReason: "error"` probe result, or any other completion signal
> from the cross-fired event. The same holds for the four other
> turn-lifecycle events PIC-18 enumerates (`tool_call`,
> `tool_result`, `message_update`, `turn_end`): a cross-fire from an
> unrelated session MUST NOT advance the prompt-mode driver's
> completion state.

Edge cases the implementer must watch:

- The witness requires a genuinely concurrent unrelated
  `AgentSession`, not a mock event injected through the
  registered handler — the latter would not exercise the
  process-global / no-per-session-origin-marker delivery property
  that is the actual MUST NOT's load-bearing premise.
- The scenario should be authored at the PIC-18 site (the
  generalised owning paragraph) and referenced from the *Error
  detection* cross-fire sentence rather than duplicated, so a
  future widening of the `pi.on` event set automatically extends
  the obligation without requiring a parallel edit at the *Error
  detection* site.
- `V9c-T`'s existing PIC-18 test bullet ("is used only for
  cancel-forwarding, never for completion") will need to be
  re-anchored on the new acceptance scenario; the bullet text
  itself stays valid, but its asserted property becomes the
  concurrent-cross-fire witness rather than an open source-grep.

## Consequence

**Severity:** advisory

The MUST NOT is real and the paired test leaves (`V9c-T` / `V9c`)
already obligate a PIC-18 test ("the prompt-mode `pi.on` subscription
is process-global with no per-session marker and is used only for
cancel-forwarding, never for completion"). Without a spec-side
acceptance scenario, however, two implementers could converge on
different witnesses — a structural source-grep on one hand, a
concurrent-session behavioural witness on the other — and the
weaker structural witness would not catch a regression where a
future contributor reintroduces global completion-resolution via a
different `pi.on(...)` path (e.g. `turn_end`) that the source-grep
on `agent_end` no longer covers.
