## Problem

`H4a`'s Adds pins the extension factory to the synchronous arm and declares "MUST NOT throw out of the factory body" the *complete* never-fault prohibition for the boundary. That completeness claim is load-bearing on an unstated precondition: that the registration/subscription calls the factory body makes complete — and take effect — synchronously, i.e. none returns a promise whose resolution is required for the registration to be durable. `H4a` neither records nor anchors that precondition. Its Adds also enumerates all five `pi.*` calls as factory-body side-effect calls, even though the spec (`registration-steps.md`) splits them by timing — only `pi.registerFlag`, `pi.registerMessageRenderer`, and the factory-time `pi.on` subscriptions run in the synchronous body, while `pi.registerCommand` and `pi.registerTool` fire later from the `session_start`/reload handlers.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 8a8af7d — pi-loom plan: resolve "H4a never-throw factory boundary leaves a returned rejected Promise unaddressed" (2026-06-16, Thomas Andersen)
**History:** The synchronous-arm pin entered `H4a` in commit `8a8af7d`, which resolved a prior finding by replacing the factory's `void | Promise<void>` return with the synchronous-arm pin ("returns `void` … no returned `Promise` arm that could later reject") and recasting "MUST NOT throw out of the factory body" as the *complete* never-fault prohibition. Before that commit the factory could return `Promise<void>`, so the synchronous-completion-of-registration property was not load-bearing; the pin made it load-bearing without recording it as a precondition. The five-call enumeration itself predates this (commit `eafd4a0`), but became a precondition surface only once `8a8af7d` pinned the synchronous arm.

## Solution approach

In `H4a`'s Adds, alongside the synchronous-arm pin, record that the pin (and its complete-never-fault claim) is load-bearing on the factory-body registration/subscription calls completing and taking effect synchronously. Anchor that property to the spec pin that already establishes it for the renderer call — `extension-bootstrap-and-per-loom.md` §"Renderer registration" (anchor `#renderer-registration`), which states `pi.registerMessageRenderer` returns `void`/synchronous with no separate Promise-rejection path and must be called before the factory returns. Scope the recorded dependency to the calls that actually run in the synchronous body — `pi.registerFlag`, `pi.registerMessageRenderer`, and the factory-time `pi.on` subscriptions — and reconcile H4a's blanket five-call enumeration to the timing split in `registration-steps.md`, noting that `pi.registerCommand` and `pi.registerTool` fire later from the `session_start`/reload handlers.
