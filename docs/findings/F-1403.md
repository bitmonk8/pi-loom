## Problem

The spec section [`binder-model-and-context.md#binder-model-hot-reload`](../spec_topics/binder/binder-model-and-context.md#binder-model-hot-reload) pins five normative requirements on the consolidated `looms.binderModel` recovery `loom-system-note`: the exact `content` template (for both `N>1` and `N=1`), ascending-Unicode ordering of `recovery.looms`, an empty-suppression `MUST NOT` when `recovery.looms.length === 0`, 250 ms settings-debounce coalescing, and re-resolve-only affected-set membership. The `V11a` / `V11a-T` `Tests.` field carries one recovery bullet that witnesses only that *a* note fires; it asserts none of the five MUSTs. No other leaf closes the gap — `V9d` witnesses only the channel triple and routing, and coverage-matrix rows `BNDR-11` and `cka-38` both map the recovery note to `V11a`, so the empty-suppression, ordering, and coalescing MUSTs have no verifying plan step.

## Solution approach

Add binder-side assertion bullets to the `Tests.` field of both `V11a-T-binder-model-resolution.md` (failing tests) and `V11a-binder-model-resolution.md` (mirrored), each citing [`binder-model-and-context.md#binder-model-hot-reload`](../spec_topics/binder/binder-model-and-context.md#binder-model-hot-reload), asserting: the exact `content` template for both `N>1` and `N=1`; ascending-Unicode ordering of `recovery.looms`; the empty-suppression `MUST NOT` (no emission when `recovery.looms.length === 0`); 250 ms debounce-window coalescing driven through the injected `Clock` seam; and re-resolve-only affected-set membership. Update the `Ships when.` clause on `V11a` so the observable condition covers these recovery-note assertions alongside the existing resolution-chain and matcher-instance assertions.
