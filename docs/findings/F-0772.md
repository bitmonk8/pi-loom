## Finding

The *Custom-message channel persistence and LLM-context entry* paragraph in `runtime-event-channel.md` mixes a present-tense normative observation (every `CustomMessage` is converted to a `user`-role context entry under the loom 1.0 Pi-SDK pin) with a conditional-future clause: *"a future Pi minor that adds such a field would be picked up by the `pi-version-bump-procedure` re-validation gate, at which point the runtime SHOULD set the opt-out on the canonical `sendSystemNote` call."* The clause uses an RFC-2119 modal (`SHOULD`) keyed to a `CustomMessage.excludeFromContext` field that does not exist at the loom 1.0 pin, addressed to a "runtime" that does not yet have a code path to host the obligation.

The clause is not a present-day requirement — there is no field to set — but its placement (interleaved with current MUSTs in the same `loom-system-note` delivery section) and its RFC-2119 modal both invite misreading: a contributor scanning for normative obligations will see a `SHOULD` and may try to wire something on the loom-side `sendSystemNote` call right now. The version-bump pipeline already covers the actual forward-compatibility concern: editorial-review checklist item (r) (`version-bump-step2.md#bump-checklist-convert-to-llm-context-entry`) re-audits the `convertToLlm` transformer and the `CustomMessage` declaration on every Pi minor bump and obligates the contributor to "surface the divergence on the bump commit" if a context-exclusion opt-out appears. The forward-looking SHOULD adds nothing the bump-time editorial gate does not already produce, and it is the only piece of forward-looking RFC-2119 prose embedded in this otherwise present-tense paragraph.

**Verdict:** present-since-inception (relative to the paragraph's introduction)

**History:** Subsequent edits touched only adjacent prose and did not alter the structural issue:

- `257c545` "Host presuppositions lack version-bump-procedure re-audit hooks" — appended the trailing sentence linking to `version-bump-step2.md#bump-checklist-convert-to-llm-context-entry`, which is the gate that now duplicates the SHOULD clause's forward-compat intent.

The SHOULD clause has therefore been in place since the paragraph was first authored; the later commits hardened the surrounding analogy and added the bump-time re-audit hook that now makes the SHOULD clause structurally redundant rather than introducing the defect.

## Solution Space

**Shape:** single

### Recommendation

Delete the conditional-future SHOULD clause from the *Custom-message channel persistence and LLM-context entry* paragraph. Concretely, remove the substring beginning with *"; a future Pi minor that adds such a field would be picked up by the `pi-version-bump-procedure` re-validation gate, at which point the runtime SHOULD set the opt-out on the canonical `sendSystemNote` call."* and terminate the preceding sentence after the `BashExecutionMessage.excludeFromContext` declaration-pin clause. Keep the present-tense observation (every `CustomMessage` enters provider context, no opt-out exists at the loom 1.0 pin) and keep the trailing sentence linking to `version-bump-step2.md#bump-checklist-convert-to-llm-context-entry`, which already owns the forward-compatibility behaviour the deleted clause was gesturing at.

Implementer edge cases to watch:

- The paragraph's anchor `custom-message-context-entry-presupposition` is referenced from `version-bump-step2.md` checklist item (r) ("[Custom-message channel persistence and LLM-context entry](./runtime-event-channel.md#custom-message-context-entry-presupposition)"). The anchor MUST stay on the same paragraph after the edit so the existing cross-link continues to resolve.
- The companion *non-normative note* paragraph two paragraphs below ("**Note (non-normative):** because of this transform, operators consuming the channel see all `loom-system-note` content (regardless of `display`) as durable session-context input rather than transient renderer output.") is already non-normative and already captures the operator-facing implication. No second-paragraph edit is required.
- Do **not** create a new bullet under `future-considerations/surface-extensions.md`. The forward-compatibility carrier for this surface is item (r) in the bump checklist (an editorial-review re-audit hook), not a `surface-extensions.md` seam — relocating the clause there would conflict with that page's "Carrier" convention and with the GOV-31 "Forward-compatibility seams" tally (the surface has no committed loom 1.0 seam to inventory).
- The companion `*Subagent-mode \`display: false\` cascades*` clause one paragraph above also embeds forward-looking framing, but it pins a present-day routing behaviour (the per-loom `sessionManager` swap) rather than reserving a non-existent field, so it falls outside the scope of this edit.

## Consequence

**Severity:** advisory

A conformant implementer can ship `sendSystemNote` correctly today either way — the clause is conditioned on a Pi-side field that does not exist, so it cannot generate divergent implementations. The cost is reader friction: an RFC-2119 modal in a current-tense normative section invites a misread that wastes implementer time looking for a non-existent opt-out field, and the redundancy with item (r) muddies the page's owner of forward-compatibility for this surface.
