## Problem

The *Custom-message channel persistence and LLM-context entry* paragraph (anchor `custom-message-context-entry-presupposition`) in `runtime-event-channel.md` interleaves a present-tense normative observation with a conditional-future clause: *"a future Pi minor that adds such a field would be picked up by the `pi-version-bump-procedure` re-validation gate, at which point the runtime SHOULD set the opt-out on the canonical `sendSystemNote` call."* The clause's RFC-2119 `SHOULD` is keyed to a `CustomMessage.excludeFromContext` field that does not exist at the loom 1.0 Pi-SDK pin and addresses a runtime that has no code path to host the obligation. Placed among present-tense MUSTs, the modal invites a contributor to wire a non-existent opt-out today. The actual forward-compatibility concern is already owned by version-bump editorial-review checklist item (r) (`version-bump-step2.md#bump-checklist-convert-to-llm-context-entry`), which re-audits the transform on every Pi minor bump, making the clause redundant.

## Issue introduction

**Verdict:** present-since-inception (relative to the paragraph's introduction)

**History:** Subsequent edits touched only adjacent prose and did not alter the structural issue:

- `257c545` "Host presuppositions lack version-bump-procedure re-audit hooks" — appended the trailing sentence linking to `version-bump-step2.md#bump-checklist-convert-to-llm-context-entry`, which is the gate that now duplicates the SHOULD clause's forward-compat intent.
- `158ed09` "Missing owning-declaration pins for bare Pi SDK symbols" — inserted the `dist/core/messages.d.ts` declaration-pin clause for `BashExecutionMessage.excludeFromContext` mid-sentence.

The SHOULD clause has therefore been in place since the paragraph was first authored; the later commits hardened the surrounding analogy and added the bump-time re-audit hook that now makes the SHOULD clause structurally redundant rather than introducing the defect.

## Solution approach

Delete the conditional-future `SHOULD` opt-out clause from the *Custom-message channel persistence and LLM-context entry* paragraph (anchor `custom-message-context-entry-presupposition`) in `runtime-event-channel.md`. Keep the present-tense observation (every `CustomMessage` enters provider context, no opt-out exists at the loom 1.0 pin) and keep the trailing sentence linking to `version-bump-step2.md#bump-checklist-convert-to-llm-context-entry`, which already owns the forward-compatibility behaviour.
