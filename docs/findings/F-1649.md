## Problem

In `provider-error-mapping.md`'s *Stop-reason classification* section (`#stop-reason-classification`), the second bullet labels an abnormal terminator a "non-terminal stop reason." This contradicts the section lead, which states a stop reason terminates the response, and contradicts the same bullet's own "content-filter terminator" example. The wording slip raises misreading risk in a normative classification bullet.

## Issue introduction

**Verdict:** single-commit

**Introducing commits:** `ef4cfb8` — "pi-loom spec: resolve \"Stop-reason → QueryError variant mapping is undefined\"" (2026-06-07).

**History:** The *Stop-reason classification* paragraph and both of its bullets — including the offending "Any other non-terminal stop reason" wording — were added in a single commit (`ef4cfb8`) that resolved a prior finding about the stop-reason → `QueryError` mapping being undefined. `git log -L` over the section shows no subsequent edits to these lines; the only earlier touch (`f5e89f4`) is the file's initial split-out of the surrounding anchors and added no stop-reason text. The contradiction was therefore present on first emission of the rule — a wording slip the resolving commit introduced rather than a drift between later edits.

## Solution approach

Rewrite the second bullet's opening qualifier in `#stop-reason-classification` to remove the contradictory term "non-terminal" and replace it with a consistent qualifier describing a stop reason outside the normal turn-boundary set (`end_turn` / `stop` / `tool_use`). Use that single qualifier consistently across the bullet.

## Solution constraints

- Out of scope: the `TransportError` / `retryable: false` mapping outcome the bullet governs and the `#transport-error-retryable` population rule it cross-references (F-1853 territory) — this is a wording-only fix that must not change the classification semantics.
