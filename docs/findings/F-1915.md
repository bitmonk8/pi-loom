## Finding

`V2e` (Wire-name translation boundary) declares three behaviours in its **Adds** field: a post-AJV loom-side rebuild (inbound), inbound enum-tag reattach via the `V5f`-produced per-schema sidecar, and outbound wire-name JSON. The spec makes all three normative — `runtime-value-model.md` states wire-name translation "happens in exactly two places" (inbound rebuild + tag reattach, and outbound loom-value → wire-named JSON before AJV), and the value-model table requires that a model-supplied enum value carry its declaring-enum tag so cross-enum equality holds (`Severity.High == OtherEnum.High` is `false`).

`V2e`'s single Tests bullet asserts only the inbound name rebuild and the defaults-bypass case: "inbound translation rebuilds loom-side names so loom code never sees wire names; defaults bypass inbound translation." Neither the outbound serialisation path nor the inbound enum-tag reattach is asserted by any vector, and the bullet is mirrored verbatim in `V2e-T`. `V2e`'s **Ships when** references only "the wire-translation vectors above", i.e. that one bullet, so the ship gate closes without observing the two unverified behaviours. `V2c` separately tests that `JSON.stringify` of an enum value yields the bare wire string and that cross-enum equality holds for loom-authored values, and `V5f` tests that the sidecar's two maps have the right shape — but no leaf asserts that `V2e` actually consumes the sidecar to reattach the tag at the boundary, nor that outbound construction emits wire names.

`cka-2` (the `runtime-value-model.md` code-keyed obligation row) lists `V2c`, `V2e` as a multi-leaf row, so `V2e` is one facet whose closing test must exist per the `H5f` per-facet citing-test gate. That gate is satisfied by the lone inbound bullet (its at-least-one-citing-test rule is met), so the coverage gate stays green while the outbound and reattach behaviours ship unexercised.

## Solution Space

**Shape:** single

### Recommendation

Extend `V2e`'s **Tests** with vectors covering the two unasserted behaviours, and mirror them into `V2e-T` per the paired-leaf TDD ritual:

- **Outbound:** a loom value whose object schema declares a renamed field serialises to wire-named JSON when constructing tool input / query-response payload / `invoke` arguments, before AJV validation (`docs/spec_topics/runtime-value-model.md`, *Wire-name translation* → *Outbound*).
- **Inbound enum-tag reattach:** a validated model-output enum value at a named-enum position has its declaring-enum tag reattached via the `V5f`-produced named-enum-position sidecar, so the rebuilt value compares equal to the same enum written as a loom-side literal (e.g. `Severity.High`) and compares unequal cross-enum (`docs/spec_topics/runtime-value-model.md`, *Inbound* clause (b) + *Equality*).

`V2e`'s **Ships when** already reads "asserts the wire-translation vectors above", so it extends automatically once the new vectors land — no separate Ships-when edit is needed. The fix is internal to the `V2e` / `V2e-T` leaf pair: the spec is read-only (it already fully specifies both behaviours), `V5f` remains the sidecar owner `V2e` consumes, and `V2c` keeps the value-model `JSON.stringify` / equality vectors. Honour the project's leaf-ID scheme and the coverage-matrix `cka-2` row as written; do not retarget the `cka-2` closing-leaf cell.

## Consequence

**Severity:** correctness

Two spec-normative behaviours — outbound loom-value → wire-named JSON and inbound enum-tag reattach — have no closing test, so an implementer can ship `V2e` with either path buggy or unimplemented and `npm test` plus the `cka-2`/`H5f` per-facet gate stay green. Downstream surfaces that build tool input, query responses, or `invoke` arguments would then emit loom-side names (or AJV-reject), and model-supplied enum values would fail to compare equal to loom-side literals, with no failing test pointing at the defect.
