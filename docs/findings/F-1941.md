## Problem

`V15e`'s **Adds** mandates that all three path identities a drop must match on — the watcher-delivered changed-file path (`V9b`), the `V15a` parse-cache key, and the `V15c` import-edge-graph node identity — be compared in the single canonical `realpath`-then-forward-slash form. But `FileSystem.realpath` (PIC-13) rejects with `ENOENT` on a missing tail, and the `FileWatcher` (PIC-14) delivers `"unlink"` events — and the old path of a rename — for files no longer on disk. Deletion is exactly the case where the gone file's parse-cache entry and its transitive importers' entries must be evicted, yet the mandated canonicalisation throws on the missing tail. `V15e` never states which event kinds drive canonicalisation, so as written the eviction either throws on every deletion or silently leaves stale entries behind the `LoomRegistry` swap.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** b63e915 — pi-loom plan: resolve "hot-reload eviction pins one canonical path identity" (2026-06-17, Thomas Andersen)
**History:** `V15e` was created at a3dd2f0 (2026-06-16) with an eviction step that did not mandate `realpath` canonicalisation of the changed-file path. b63e915 (2026-06-17) added the rule that all three path identities — including the watcher-delivered changed-file path — be compared in the canonical `realpath`-then-forward-slash form, introducing the unhandled-`unlink` defect: `realpath` throws `ENOENT` on a missing tail, but deletions still require eviction.

## Solution approach

In `V15e`'s **Adds**, state which watcher event kinds drive the `realpath`-then-forward-slash canonicalisation: scope it to `"add"`/`"change"` events (whose tail still exists), and specify that an `"unlink"` event — and the old path of a rename — is matched against the stored load-time-canonical `V15a` cache key directly, without re-`realpath`-ing the missing path. Add a `V15e` **Tests** bullet, mirror it in `V15e-T`, and add a corresponding **Ships when** clause asserting that an `"unlink"` of a cached file — including one with transitive `.warp` importers — drops the gone file's entry and its importers' entries on the swap, with no `realpath` against the missing path.
