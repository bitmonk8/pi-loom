## Finding

`V15e`'s eviction step compares three path identities a drop must match on — the watcher-delivered changed-file path (`V9b`), the static-resolution per-pass parse-cache key (`V15a`), and the `.warp` import-edge-graph node identity (`V15c`) — in the single canonical `realpath`-then-forward-slash path form defined under [`invocation.md` — Resolution](../spec_topics/invocation.md). The `FileWatcher` seam (PIC-14, `FileWatchEventKind`) delivers three event kinds: `"add"`, `"change"`, and `"unlink"`. `FileSystem.realpath` (PIC-13) rejects with `ENOENT` when any path component, including the final tail, is missing.

For an `unlink` event — and for the old path of a rename, observed by the watcher as a `removed`/`unlink` of the prior path — the changed-file path no longer exists on disk, so the mandated `realpath` canonicalisation throws rather than producing a cache key. Yet a deletion is exactly the case where the gone file's parse-cache entry (and the entries of every file that transitively imported it) must be evicted; otherwise the next load pass serves stale static-resolution state for a file that is no longer present.

`V15e` never states which event kinds drive the canonicalisation step, nor how a deletion/rename path is matched against the load-time-canonical cache key without re-`realpath`-ing a missing tail. As written, the eviction either throws on every deletion or silently fails to evict the deleted file, leaving stale entries behind the `LoomRegistry` swap.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** b63e915 — pi-loom plan: resolve "hot-reload eviction pins one canonical path identity" (2026-06-17, Thomas Andersen)
**History:** `V15e` was created at a3dd2f0 (2026-06-16) with an eviction step that did not mandate `realpath` canonicalisation of the changed-file path. b63e915 (2026-06-17) added the rule that all three path identities — including the watcher-delivered changed-file path — be compared in the canonical `realpath`-then-forward-slash form, introducing the unhandled-`unlink` defect: `realpath` throws `ENOENT` on a missing tail, but deletions still require eviction.

## Solution Space

**Shape:** single

### Recommendation

In `V15e`'s **Adds**, state which watcher event kinds drive the eviction canonicalisation: scope the `realpath`-then-forward-slash canonicalisation to `"add"`/`"change"` events (whose tail still exists), and specify that an `"unlink"` event — and the old path of a rename — is matched against the stored load-time-canonical cache key directly, without re-`realpath`-ing the now-missing path. The load-time-canonical key the entry was stored under (`V15a`) is the match target; the eviction looks the gone path up against that stored key rather than recomputing a canonical form from a path that no longer resolves.

Add a `V15e` **Tests** bullet (mirrored in `V15e-T`) and a corresponding **Ships when** clause asserting that an `"unlink"` of a cached file — and of a file with transitive `.warp` importers — drops the gone file's parse-cache entry and its importers' entries on the `LoomRegistry` swap, with no `realpath` call against the missing path, so the next load pass does not serve a stale entry for the deleted file.

Edge cases the implementer must watch: a rename arrives as an `"unlink"` of the old path plus an `"add"` of the new path (the old path matches by stored key; the new path canonicalises normally), and a deletion of a file that is itself a transitive-import target still drives the importer-graph walk to evict downstream entries.

## Consequence

**Severity:** correctness

On any `.loom`/`.warp` deletion or rename, the mandated `realpath` canonicalisation throws `ENOENT` on the now-missing tail, so the eviction either aborts or — if an implementer guards the throw — silently skips the deleted file. The deleted file's parse-cache entry and its transitive importers' entries then survive the `LoomRegistry` swap, and the next load pass re-resolves against a file that no longer exists. Two reasonable implementers would diverge (let it throw vs. invent an ad-hoc match), and neither reading is pinned by the leaf.
