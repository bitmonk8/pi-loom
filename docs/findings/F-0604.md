## Finding

The inventory-closure audit's entire marker-scope contract is line-keyed: *Marker scope is per source line* binds authorisation to "line N"; *Multi-line surface placement* speaks of references that "span multiple **physical lines**" and pins an *originating line* per shape; *Malformed-marker discriminator* clause (e) is defined against "the line above, on the line below, or as a standalone comment line"; *Stale sub-kind (s1)* fires when "the originating line … contains zero in-scope surface references at audit time"; and the wire format emits an integer `line` field per record. None of these rules names the set of code points that terminate a physical line.

The *Source-file encoding* paragraph in `audit-resolution.md` pins UTF-8 (BOM-stripped, strict decode) but encoding does not fix line boundaries. The candidate models diverge on real source: ECMAScript's `LineTerminator` production recognises LF (U+000A), CR (U+000D), the CRLF sequence, LS (U+2028), and PS (U+2029); TypeScript's `ts.getLineStarts` (the obvious AST-walker primitive given the audit's AST-recogniser shape) recognises only LF/CR/CRLF; a naive `String.prototype.split(/\r?\n/)` walker recognises only LF and CRLF; a `\n`-only walker treats a CR-only file as a single line. The audit binds the implementation to *some* AST walker without naming which line-boundary set that walker must agree on.

Two conformant audit implementations picking different models flip observable outputs on the same source tree: a `// allow-pi-surface:` marker placed after a U+2028 trailing an offending surface line either authorises the surface (ECMAScript model: marker is on the *next* line, fails clause (e)) or authorises it (TS-AST model: marker and surface are on the same line); a multi-line `pi\n  .registerCommand(...)` chain that uses LS instead of LF between the namespace and the property identifier picks a different *originating line* under each model, so the family-(5) `<symptom>` token emitted on a misplaced marker differs across walkers; and the family-(5) `(s1) no-surface-on-line` predicate flips for any line whose only in-scope reference straddles a contested terminator. The `<symptom>`-fixture floor pinned by the *Failure-surface contract* exercises these clauses by construction, so the divergence surfaces directly in the permanent test invariants.

## Solution Space

**Shape:** single

### Recommendation

Add a *Source-file line terminators* sentence to the *Source-file encoding* paragraph in `docs/spec_topics/pi-integration-contract/audit-resolution.md` pinning the line-terminator set the audit's "physical line" / "same-line" / `line N` rules are defined against, and the byte sequence that constitutes a single line boundary, before the encoding paragraph hands off to the walker. The pinned set MUST be the ECMAScript `LineTerminator` production — LF (U+000A), CR (U+000D), LS (U+2028), PS (U+2029) — with the CRLF sequence (U+000D U+000A) counted as a single boundary; this is the set TypeScript-source files are authored against and is the only set under which the audit's walker sees the same line numbering a contributor reading the file in an editor sees. State this normatively (MUST), not as guidance, since the per-`<symptom>` fixture floor in *Failure-surface contract* is keyed to it.

The new sentence also pins the `line` field's counting basis: the integer emitted in the wire-format `path:line` segment is the 1-based ordinal of the line the offending surface originates on under the same terminator set, so the wire field, the *Marker scope* `line N`, and the *Multi-line surface placement* *originating line* are the same number for the same reference. Edge cases the implementer must watch: (a) a trailing terminator at end-of-file does NOT introduce a phantom empty line (the `line` count is one less than the boundary count on a terminator-terminated file); (b) a lone CR followed by characters other than LF terminates a line (so a CR-only file is multi-line, not single-line); (c) U+2028 / U+2029 inside a string-literal or template-literal token still count as line terminators for the audit's purposes (the audit operates on the textual file, not on parsed token values, so the wire `line` field reflects the physical-line ordinal regardless of where the boundary falls syntactically) — this is the only place the audit's line counting deliberately disagrees with the host language's expression-level semantics, and is the rule that keeps the marker-placement contract decidable from the file text alone.

## Consequence

**Severity:** correctness

Two reasonable implementers can ship audits that disagree on which line a marker is on, which line a multi-line surface "originates" on, and whether a family-(5) `(s1) no-surface-on-line` record fires. The permanent test invariants the *Failure-surface contract* pins (per-`<symptom>` fixture floor; per-multi-line-shape fixture floor) sit directly on this divergence, so the same source tree produces different `audit/violation/.../...` records on different conformant walkers — defeating the byte-identical-output guarantee the *Wire serialisation* clause's deterministic-walk rule otherwise underwrites.
