## Problem

The machine-checkable registry of diagnostic codes that `V7b` delivers and the `H5a` closing gate consumes for DIAG-2/3/4 reconciliation is referred to under four distinct names across the plan corpus, all denoting the same single artifact: "diagnostic code registry" (`V7b` / `V7b-T` titles and the `plan.md` §V7 index link), "diagnostic registry" (`V7b` `Adds.`), "diagnostics registry" (`conventions.md` §Diagnostic message anchors, `coverage-matrix.md` preamble, `H7a` `Tests.`, `V4b` / `V4b-T` `Tests.`, `V6d` / `V6d-T` `Tests.`), and "diagnostics code registry" (`H5a` `Adds.`, whose own title reads "diagnostic-code"). A developer reading `V4b`'s, `V6d`'s, or `H5a`'s reference must infer that the referent is the registry `V7b` owns under a different name. The DIAG-4 test-citation discipline — every asserting test sources its expected *Message* string from the registry — is therefore stated under one name, owned by a leaf titled under another, and consumed by a gate named under a third, so nothing binds the discipline, its owning leaf, and its gate together. There is no mechanical name-consistency gate that catches the drift.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `c6a664e` — "pi-loom plan: build/update plan for spec.md + review" (2026-06-10)
**History:** The `git log -G` walk shows the "Diagnostic code registry" title form (`V7b`) and the "diagnostics registry" form (`conventions.md` §Diagnostic message anchors) both first appeared in `c6a664e`, the initial plan-build commit — they coexisted from the corpus's inception rather than being introduced by a later edit. Subsequent commits that touched the "diagnostics registry" form (e.g. `692cc8c` "resolve V7a bundles diagnostics primitive…", `2b8aed8` "resolve Prompt-mode system: rejection bullet cites no diagnostic code") propagated the already-present term into additional leaves without reconciling it against `V7b`'s title; they did not originate the divergence. `V7b`'s title has not changed since birth (only `f284534` touched the file, for an unrelated DIAG-2 scan resolution).

## Solution approach

Adopt `V7b`'s title term "diagnostic code registry" as the canonical name and rename every divergent site to it: `V7b` `Adds.`, `conventions.md` §Diagnostic message anchors, `coverage-matrix.md` preamble, `H5a` `Adds.`, and the `Tests.` references of `V4b` / `V4b-T` and `V6d` / `V6d-T`. Where the term is used attributively, as in `H7a` `Tests.`, render it hyphenated ("diagnostic-code-registry"). The `V7b` / `V7b-T` titles and the `plan.md` §V7 index link already use the term and need no change.

- Out of scope: `docs/spec.md` and `docs/spec_topics/**` are read-only for this fix; align names within the plan corpus only.
