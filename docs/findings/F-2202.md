## Problem

In `query/query-forms.md`, the schema-inference algorithm under the `qry-3` anchor closes with a `*Worked examples:*` block of six `input → outcome` pairs fixing the typed/untyped classification for representative queries. The block carries no normative designation, unlike the sibling example blocks on the same page: QRY-4's `*Test vectors (normative).*` and QRY-7's "The following input → output pairs are normative." Because the QRY-3 examples are informative only, a conforming implementation may classify an edge case differently from a worked pair and still claim conformance with the algorithmic prose, and a conformance suite cannot cite the pairs as executable pass/fail oracles.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `fae85d3` ("Tighten spec for implementation-plan readiness", 2026-05-04)
**History:** The schema-inference algorithm and its `*Worked examples:*` block were first authored in `fae85d3`, and the block carried no normative designation at that point. The visible asymmetry against the explicit-form examples appeared later: `9d0acbf` (2026-05-07, "explicit-schema-mismatch disagree not anchored to type compatibility") added the `*Test vectors (normative).*` block for QRY-4 while leaving the pre-existing QRY-3 worked examples informative. The `f5e89f4` size-cap split (which moved the content into `query/query-forms.md`) and `716a13b` (which coined the `QRY-N` labels) were mechanical and did not change the normative status of either block. The defect — QRY-3 examples lacking a normative designation — has existed since the examples were first written.

## Solution approach

Promote the QRY-3 `*Worked examples:*` block in `query/query-forms.md` to normative status, matching the designation register already used by the sibling QRY-4 and QRY-7 blocks on the page. The six pairs already agree with the algorithmic prose, so the elevation adds a normative designation without changing example content.
