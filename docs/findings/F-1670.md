## Finding

The page title `# Query escapes stringification` (in `docs/spec_topics/query/query-escapes-stringification.md`) has no conjunction joining its constituent topics. Read as English it parses most naturally as a subject-verb-object clause — "the query escapes stringification" — which inverts the intended meaning. The intended reading is a noun phrase naming the two subjects the page actually covers: query-template **escape sequences** (the `## Escapes` section, QRY-17) and **stringification** of interpolated values (the `## Stringification of interpolated values` section, QRY-18).

The ambiguity is amplified by reuse: the hub file `docs/spec_topics/query.md` reproduces the title verbatim as the navigation link label (`- [Query escapes stringification](./query/query-escapes-stringification.md)`), so every reader traversing the query subtree encounters the mis-parseable phrasing before opening the page.

The page additionally hosts QRY-19–QRY-21 ("Discarded query results are a parse error," "Observability of discarded results," and the interpolation-panic note), none of which concern escapes or stringification and none of which the title signals. Whether the title should enumerate those topics depends on whether that content stays on this page; that placement question is tracked by a separate finding (see Related Findings).

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set")
**History:** Before this commit, the discard parse rule, the discard-observability event, and the interpolation-panic subsection lived in the monolithic `docs/spec_topics/query.md` (see `git show 31ff060:docs/spec_topics/query.md` lines 134–189: `## Escapes`, `## Stringification of interpolated values`, `## Discarded query results are a parse error`, `## Observability of discarded results`, `### Panics during interpolation are not caught by let _ =`), where the page H1 was simply `# Query` and no naming mismatch existed. The `f5e89f4` splitter (`docs/_tools/split_spec.py`) carved `query.md` into four children keyed by a slug table; the entry `("query-escapes-stringification", 136, 192)` mechanically swept all four sections — Escapes, Stringification, Discarded, Observability (plus the Panics subsection) — into a single child whose slug, H1, and parent-Contents label were derived from only the first two section titles. The QRY-19 / QRY-20 / QRY-21 anchors did not yet exist at split time; they were later coined in place by `716a13b` (2026-06-25, "pi-loom spec-plan: fix F-1286 — coin QRY-1..21 across query subtree + repoint V13 leaves/coverage"), formalising the misplacement but not introducing it. The naming defect therefore enters at `f5e89f4`'s slug choice — no earlier commit ever named this page — and is a pure artefact of the mechanical split.

## Solution Space

**Shape:** single

### Recommendation

Retitle the page to **`# Query Escapes and Stringification`**, inserting the conjunction so the two-topic noun-phrase structure is explicit and the verb-phrase misreading is eliminated. Update the link label in `docs/spec_topics/query.md` to the same string so the hub and the page agree.

Edge case for the implementer: if the QRY-19–QRY-21 discarded-result content is relocated off this page (per the related placement finding), the two-topic title above is correct as written. If that content remains on the page, broaden the title to enumerate the third topic — e.g. `# Query Template Escapes, Stringification, and Discarded Results` — and mirror the chosen string in the hub link label. Resolve the title wording after, or jointly with, the placement decision so the two findings do not produce conflicting titles.

## Consequence

**Severity:** cosmetic

A reader scanning the spec or the hub link momentarily mis-parses the title and must open the page to recover the intended two-topic meaning. No implementer behaviour diverges; the defect is confined to navigation wording.
