## Finding

The `tools:` frontmatter field page hosts a "Resolution snapshot" subsection (anchor `#resolution-snapshot`, in `frontmatter-fields-b-and-templates.md`) that, beyond defining what load-time resolution produces, carries a block of runtime tool-call-dispatch normative obligations: (1) every call dispatches through the captured `ToolDefinition` strong reference and the runtime never re-queries Pi's tool registry by name during execution; (2) unregistering a Pi tool mid-run does not affect in-flight invocations; (3) the `unknown_tool` `CodeToolError` cause is reachable only via the file-watcher reload path; (4) file-watcher rebuilds invalidate the schema cache and rebuild `tools:` tables on the next invocation; and (5) a hot-reloaded extension leaves the captured `execute` closure orphaned, with full error-routing of the orphaned dispatch into `Err(CodeToolError { cause: "execution" })`, `loom/runtime/internal-error`, and the runtime-panics surface.

These are runtime tool-call-dispatch and error-model rules, not field-contract rules. The page that owns that surface is `tool-calls.md` — which already defines the `CodeToolError` cause enum (including the `unknown_tool` reachability fact) and the off-surface outcome routing, and which already cross-references *back* to the frontmatter page for the held-reference behaviour. The result is split ownership: consequence bullet (3) duplicates the `unknown_tool` reachability statement that `tool-calls.md` also makes (a drift risk if either is edited), and bullet (5) plants deep error-routing prose (CodeToolError / `loom/runtime/internal-error` / runtime panics) on a frontmatter field page rather than on the error-routing page it references.

The load-time half of the subsection — "resolution produces a frozen per-loom table of `{ post-rename name → resolved callable }` entries" — is intrinsically a `tools:` field semantic and belongs with the field. The defect is confined to the runtime-dispatch consequences co-located with it.

## Solution Space

**Shape:** multiple

### Option A — Move the whole subsection to `tool-calls.md`, leave a definitional stub on the field page

- **Approach:** Relocate the entire "Resolution snapshot" subsection (definition + all five consequence bullets) into `tool-calls.md`, which owns runtime dispatch. Replace it on the field page with a single sentence: "Resolution produces a frozen per-loom table of `{ post-rename name → resolved callable }` entries used for all dispatches during that loom's execution; see [Tool Calls — Resolution snapshot]." Move the `#resolution-snapshot` anchor to `tool-calls.md` and repoint the inbound cross-references in `tool-registration-lifetime.md`, `subagent.md`, and `slash-invocation.md`.
- **Spec edits:** cut subsection from `frontmatter-fields-b-and-templates.md`; paste into `tool-calls.md`; insert one-sentence stub + link on the field page; move anchor; repoint three inbound links.
- **Pros:** Single canonical home for both the snapshot definition and its dispatch consequences; eliminates the `unknown_tool` duplication outright.
- **Cons:** Removes the load-time "what resolving `tools:` yields" fact from the field that produces it, where readers of the field contract reasonably expect it; widest anchor-repointing surface.
- **Risks:** A missed inbound `#resolution-snapshot` link becomes a dangling cross-reference.

### Option B — Keep the snapshot definition on the field page, move only the runtime-dispatch consequences

- **Approach:** Retain the load-time definition ("resolution produces a frozen per-loom table … dispatch through the held reference; the runtime never re-queries Pi's registry by name during execution") and its `#resolution-snapshot` anchor on the field page, since that is the field's own load-time semantic. Move the runtime-dispatch consequence bullets — mid-run unregistration, `unknown_tool` reachability, file-watcher rebuild, and the hot-reload-orphan error routing — into `tool-calls.md` (the orphan-routing bullet naturally joins the off-surface outcome routing already there). On the field page, replace the moved bullets with a one-line pointer to the `tool-calls.md` dispatch-consequences section.
- **Spec edits:** keep definition paragraph + anchor in `frontmatter-fields-b-and-templates.md`; cut the four consequence bullets; paste them into the dispatch/outcome-routing area of `tool-calls.md`; add a one-line pointer; no anchor move, so inbound `#resolution-snapshot` links stay valid.
- **Pros:** Each fact sits with its owner — load-time semantics with the field, dispatch/error-routing with `tool-calls.md`; removes the `unknown_tool` duplication; no inbound-link repointing.
- **Cons:** Splits one subsection across two pages, requiring a clear pointer so the reader can follow the dispatch consequences.
- **Risks:** The pointer must be explicit enough that an implementer reading the field page does not assume dispatch semantics are unspecified.

### Recommendation

Take Option B. The frozen-snapshot definition is intrinsically a `tools:`-field load-time semantic and should stay where the field is defined (preserving the `#resolution-snapshot` anchor and its three inbound links), while the runtime-dispatch consequences — especially the hot-reload-orphan error routing — belong with `tool-calls.md`'s existing dispatch and off-surface outcome routing. Edge case for the implementer: when relocating the `unknown_tool` reachability bullet, fold it into `tool-calls.md`'s existing `CodeToolError` `unknown_tool` description rather than restating it, so the duplication is removed rather than merely moved.

## Consequence

**Severity:** advisory

A reader consulting the frontmatter field page encounters runtime-dispatch and error-routing obligations that the runtime-dispatch page also owns, and an implementer or auditor reading `tool-calls.md` to find the dispatch model is bounced to a frontmatter page for it. The duplicated `unknown_tool` reachability statement can drift between the two pages. No runtime behaviour is wrong today (the two pages currently agree), so implementers can still build a working system; the cost is interpretive burden and future divergence risk.
