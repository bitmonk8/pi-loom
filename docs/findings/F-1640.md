## Problem

Three glossary entries — `callable set`, `schema slug`, and `Pi tool` vs. `` `.loom` callable `` — host full naming-convention MUSTs (canonical term plus avoid-list), but `glossary.md` is classified `(no IDs — narrative)` in the REQ-ID prefix table, and its intro disclaimer states the page is descriptive-only with each term's normative definition living on the feature page named in the entry's `See:` link. On a narrative page these MUSTs cannot carry a REQ-ID anchor, so GOV-9's `#prefix-n` cross-link contract is unsatisfiable for them, and the "if a glossary entry and its canonical page disagree, the canonical page wins" doctrine has no canonical-page MUST to defer to — making the glossary the de facto source of truth (named so explicitly for `schema slug`'s future grep gate) while the prose insists it cannot be. The H5e un-anchored-MUST gate's live-corpus arm will recognise each of the three as an un-enumerated normative obligation.

## Issue introduction

**Verdict:** multi-commit-interaction

**Introducing commits:**
- `b4fbfac` (2026-05-04, "pi-loom spec: resolve \"no central glossary\"") — created `spec_topics/glossary.md` with the descriptive-only disclaimer in its intro paragraph; the initial `callable set` and `Pi tool` vs. `.loom callable` entries it shipped contained no MUSTs.
- `fec0870` (2026-05-06, "pi-loom spec: resolve \"callable-set prose synonyms erode no-inheritance invariant\"") — added the `MUST use the term \`callable set\`` clause and the `MUST use the canonical term \`.loom callable\`` clause inside the existing glossary entries.
- `9d99fe6` (2026-05-07, "pi-loom spec: resolve \"Schema-hash identifier is referred to by six surface names\"") — added the new `schema slug` entry already carrying its `Authors and implementers MUST use \`schema slug\`` clause and the "a future grep gate uses this entry as its source of truth" sentence.

**History:** `git log --follow -S "MUST use the term \`callable set\`"`, `-S "MUST use \`schema slug\`"`, and `-S "MUST use the canonical term"` on `docs/spec_topics/glossary.md` each return one commit (the two above plus `b4fbfac` for the page itself). The descriptive disclaimer at line 3 has been unchanged since `b4fbfac`. Neither of the two later commits touched the intro disclaimer or the prefix-table row that pins glossary.md as `(no IDs — narrative)`. The defect is the interaction: each later fix migrated a corpus-wide naming MUST into the entry it considered the canonical home (each subject was already cross-referenced from multiple feature pages), without revisiting whether the glossary's narrative-page framing still held after that MUST landed there. Each commit is locally well-motivated in isolation; the rule that breaks is one neither commit edited.

## Solution approach

Move each naming MUST off the glossary onto the non-narrative canonical feature page named in the entry's leftmost `See:` link, coining a fresh REQ-ID anchor under that page's already-registered prefix so each MUST satisfies GOV-9's `#prefix-n` cross-link contract. Relocate the `callable set` and `.loom callable` MUSTs (term plus avoid-list) to `frontmatter-fields-a.md` §`tools`, and the `schema slug` MUST (term, avoid-list, and the "future grep gate uses this entry as its source of truth" sentence) to `schema-subset.md`, converting that page's existing "pins the avoid-list" parenthetical into the normative site. Demote the three glossary entries to descriptive paragraphs by deleting the MUST sentences and avoid-lists, leaving each entry's prose and `See:` link. Update the `schema-subset.md` back-reference that currently cites the glossary entry so it cites the new local anchor.

## Solution constraints

- The glossary's `(no IDs — narrative)` prefix-table classification stays unchanged — do not promote `glossary.md` out of narrative status to host these MUSTs.
