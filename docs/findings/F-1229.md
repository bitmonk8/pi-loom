## Finding

`tool-registration-lifetime.md` (anchor `#snapshot-restore-pi-behavioural-preconditions`) lists two Pi *behavioural* preconditions that the page-pinned SDK type signature does not encode. Precondition (a) reads:

> the loom runtime treats `pi.setActiveTools(string[])` as synchronous-and-atomic on the JS event loop (a single swap or restore call cannot be interleaved with anything else)

The parenthetical gloss restates an invariant that every synchronous JS function call already satisfies by definition: while a synchronous call is running, the single-threaded event loop cannot interleave any other task. If `pi.setActiveTools` is synchronous (returns `void` without yielding), it is "atomic on the JS event loop" trivially; if it is not synchronous, no amount of "atomicity" framing rescues it. The wording therefore describes either a tautology or self-contradiction depending on which half of the conjunction is load-bearing, and does not pin a property that a future Pi minor could measurably weaken.

The PIC checklist item that consumes this precondition — `version-bump-step2.md` item (e), routed via the *Editorial-review checklist for unpinned host presuppositions* — needs to audit something concrete on each Pi bump. The actually-needed content is: `pi.setActiveTools(string[])` returns synchronously without yielding, and by the time the call returns the active-set change is committed and observable to a subsequent `pi.getActiveTools()` in the same task (so step-2's swap-install is committed before the query begins, and step-4's restore is committed before the `finally` block returns). Precondition (b) already covers cross-handler interleaving against the same session; precondition (a) needs to carry the synchronous-completion-and-commit property, not a restatement of JS single-threadedness.

## Issue introduction

**Verdict:** present-since-the-feature-was-introduced
**Introducing commits:** `e7ba4ba` ("pi-loom spec: resolve \"pi.setActiveTools snapshot/restore relies on an unstated Pi serialisation guarantee\"", 2026-05-06) introduced the two-precondition framing with the wording "`pi.setActiveTools(string[])` is synchronous and atomic on the JS event loop, so a single swap or restore call cannot be interleaved with anything else"; commit `68ff2d1` ("pi-loom spec: resolve \"Pi API surfaces asserted without .d.ts citations\"", 2026-05-07) refactored the same paragraph to add the `.d.ts` citation and SDK-pin framing and tightened the wording to "synchronous-and-atomic on the JS event loop (a single swap or restore call cannot be interleaved with anything else)" — the form the present finding cites.
**History:** The defect entered with the feature itself — the precondition was vacuous on day one of the snapshot/restore protocol's behavioural-precondition framing. Subsequent edits (`19b193b` adding PIC-19, `823ce54` adding the recovery-mutex contract, `4d8dc02` adjusting the hot-reload note) preserved the wording unchanged and built dependents on top of it (PIC-19's "step-2 throw leaves the install uncommitted" cites this precondition by anchor), so the wording is now load-bearing for at least one downstream normative claim, even though that claim quietly relies on the *useful* reading the wording fails to pin.

## Solution Space

**Shape:** single

### Recommendation

Rewrite precondition (a) to pin the synchronous-completion-and-commit property:

> (a) `pi.setActiveTools(string[])` returns synchronously and the active-set change is committed by the time the call returns — a subsequent `pi.getActiveTools()` in the same task (or after any number of microtask/await boundaries) observes the newly-installed set, and a swap-install call that throws leaves the active set unchanged.

Then audit downstream consumers in the same edit:

- In the same paragraph: change the wording at the PIC-19 anchor reference if the surrounding sentence loses fluency.
- In `capability-probe.md#post-probe-sdk-shape-drift` (consumed by PIC-19): the "step-2 throw leaves the install uncommitted under the synchronous-and-atomic precondition" justification still reads correctly with the new wording (the rewording explicitly carries the "throw leaves the set unchanged" clause, so this dependent claim becomes more clearly grounded, not less).
- In `version-bump-step2.md` item (e) audit recipe: confirm the recipe is auditing the synchronous-return-and-commit property (it should be); if its current language was tracking the vacuous wording, tighten it to the new property in the same commit.
- In `V15d-prompt-suspend-snapshot.md`'s **Spec** field: replace the "synchronous-and-atomic `pi.setActiveTools`" paraphrase with "synchronous-completion-and-commit `pi.setActiveTools`" (or the chosen short label) so the plan leaf's spec gloss matches.

Edge case the implementer must watch: do not weaken the precondition to "synchronous return" alone. The "commit on return" half is what justifies PIC-19's "no restore is owed" rule for a step-2 throw and PIC-17's reliance on the snapshot taken at step 1 being the right restore target at step 4. Both halves must travel together.

## Consequence

**Severity:** advisory

Two reasonable implementers reading precondition (a) today can infer different audited properties on the next Pi version bump — one auditing "the call doesn't internally `await`" (the useful property), another auditing "no other task interleaves during the call" (vacuous under JS single-threadedness). The downstream consumer of the precondition (`version-bump-step2.md` item (e)) and PIC-19's "step-2 throw leaves the install uncommitted" justification both rest on the *useful* reading, so the spec ships internally consistent today; the cost is a precondition that cannot be operationalised as a per-bump audit step and that gives no leverage if Pi later switches `pi.setActiveTools` to an async signature whose return promise resolves after the active-set change is committed (a change the current wording would not flag, but the synchronous-completion-and-commit wording would).
