## Problem

The `bind_context: session` truncation walk sources its message list from `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages` at slash-invocation time (`binder-model-and-context.md` *Session-context truncation*). The spec pins three behavioural preconditions on that snapshot — chronological ordering (`#messages-chronological-order-presupposition`), the leading-`user`-message biconditional (`#messages-leading-user-message-presupposition`), and per-message estimator determinism (`#token-estimator-determinism-presupposition`) — but none pins whether the triggering slash-command user turn (the `/<name> <args>` message that fired the invocation) is present at the array tail when the walk begins. That tail turn is the seed of the newest-to-oldest cap walk, so the two cases produce different included-turn sets and different transcript bytes for the same session. Because BNDR-7 declares the compact-transcript reference renderings MUST-reproduce-exactly and grounds that on "the same session always constructs the same prompt bytes" (`#why-the-transcript-bytes-are-pinned`), the indeterminate membership defeats the byte-reproducibility contract rather than being cosmetic.

None of the five edits introduces or removes any statement about whether the triggering slash turn is in the walked snapshot — the silence is original to the paragraph. The pre-split sibling text in the consolidated `spec.md` (visible at `git show fae85d3 -- spec.md`, the *Tighten spec for implementation-plan readiness* commit dated 2026-05-04) carried the same gap under an `ctx.getContextUsage()`-sourced earlier draft, so the defect predates the per-topic split.

## Solution approach

Add a named consumption-posture presupposition on `host-prerequisites.md` alongside the existing `AgentMessage[]` ordering and leading-`user`-message presuppositions, modelled on `conversation-drive.md`'s `#driven-turn-commit-ordering-presupposition`, pinning whether Pi has committed the triggering slash-command user turn to the session-entry store the binder's invocation-time `buildSessionContext(ctx.sessionManager.getEntries(), …)` read observes. Cite the new presupposition from the *Session-context truncation* section so the included-turn set is reproducible across conforming implementations. Add a matching item to the *Editorial-review checklist for unpinned host presuppositions* in `version-bump-step2.md` so a Pi minor that changes slash-turn commit timing is re-audited. Where the pinned answer admits the in-flight turn, resolve the duplication against the `User arguments:` line (`binder-bypass-and-envelope.md` *System-prompt structure* item 5); where it excludes the turn, pin how the runtime identifies it deterministically.

## Solution constraints

- The pinned answer MUST stay consistent with the BNDR-7 reference renderings (`bndr-7a`–`bndr-7j`) and the `bndr-7i` void-truncation case; if the implied included-turn set changes any oracle, reconcile that oracle in the same edit.
