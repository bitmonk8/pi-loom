## Problem

`H5b`'s `**Deps.**` field ends with a long italic parenthetical note that does two non-dependency jobs: it *defines* the "coverage-producing set" membership rule (every leaf that can introduce an executable REQ-ID, a numbered-REQ-ID citing test, or an un-anchored normative MUST — the closing-leaf set named in `coverage-matrix.md`) and it restates the transitive-completeness plan-maintenance obligation. Per `conventions.md` §Leaf format the `Deps.` field is an enumeration of leaf IDs, not a home for normative convention prose, so the standing membership rule is buried in one leaf's dependency footnote instead of governing all shards. The misplacement is also mechanically risky: `H5d`'s transitive-completeness gate arm parses `H5b`'s `Deps.`, and the footnote's embedded backtick leaf-IDs and cross-group spans (e.g. `V1a`–`V18d`, whose endpoints do not share a `<group>` number) make it ambiguous whether the prose is part of the parsed dependency set and could break `H5d`'s within-group range expander. The standing obligation already has a home — the **Transitive-completeness plan-maintenance** paragraph in `conventions.md` §REQ-ID discipline — where the membership-rule definition should sit instead.

## Issue introduction

**Verdict:** present-since-inception (of the `H5b` leaf)
**Introducing commit:** `ea6b1da` — "pi-loom plan: resolve \"Live-corpus gate activation has no documented rollback\"" (2026-06-11)
**History:** `git log --follow --diff-filter=A` records `ea6b1da` as the commit that created `docs/plan_topics/H5b-warn-only-canary.md`, and `git log -S "dependency set is the complete coverage-producing set"` shows the italic membership/transitive-completeness note was present in that same creation commit — the defect has existed since the leaf was first written. Four minutes later, `37733fd` ("H6a transitive-completeness rule parked in Deps, not conventions", 2026-06-11) corrected the analogous pattern in `H6a`'s `Deps.` by moving the rule to conventions, but did not propagate the sweep to `H5b`; subsequent commits (`25b911f`, `bc228eb`, `edf5e66`) only adjusted the enumerated IDs/ranges within the note rather than relocating it.

## Solution approach

Strip the trailing italic parenthetical note from `H5b`'s `**Deps.**` field so the field carries only leaf-ID tokens and ranges. Relocate the membership-rule definition it carried — which leaves comprise the coverage-producing set, that this set is `H5b`'s `Deps.`, and that `H6a` inherits it by depending on `H5b` — into `conventions.md` §REQ-ID discipline, adjacent to the **Transitive-completeness plan-maintenance** paragraph that already states the standing obligation. A brief non-normative why-this-structure pointer may remain in `H5b`'s `Adds.`, but the normative definition lives in conventions, not in `Deps.`.

## Solution constraints

- Do not introduce into `H5b`'s `Deps.` any leaf ID not already present in its canonical enumeration (no invented IDs, per the plan's `H<n><letter>` / `V<N><letter>` leaf-ID scheme).
- The relocated convention prose must not reintroduce a non-leaf-ID range form (e.g. the cross-group `V1a`–`V18d` span) into `H5b`'s `Deps.` or any other field `H5d` parses; such descriptive spans are safe only in `conventions.md` prose, which the gate does not parse.
- Out of scope: `H6a`'s analogous `Deps.` note, owned by the same-cluster sibling finding that resolves independently.
