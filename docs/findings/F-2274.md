## Finding

The `caller` glossary entry (`docs/spec_topics/glossary.md`, `#caller`) carries a normative usage mandate: *"Authors and implementers SHOULD use the disambiguated terms `slash caller` and `invoke parent` whenever the bare word `caller` is ambiguous in context."* This is the only place in the corpus where the disambiguation convention is stated as a directive.

The glossary's own preamble declares the page non-normative: *"Each entry is descriptive — a one-paragraph reminder of what a term means and where it is used. The canonical, normative definition lives on the feature page named in the entry's `See:` reference; if a glossary entry and its canonical page disagree, the canonical page wins."* A SHOULD-level authoring/implementation mandate placed inside a page that disclaims normativity is structurally self-contradictory: by the corpus's own precedence rule the mandate is overridable by canonical pages that say nothing about it.

The pages that motivate the distinction — `slash-invocation.md` (which uses *slash caller* and *invoke parent* in SLSH-3) and `invocation.md` (which uses *invoke parent*) — are exactly the `See:` targets the `caller` entry points to, yet neither carries the usage mandate. An author writing prose in those pages, or a contributor reviewing them in isolation, never encounters the directive. The mandate is discoverable only by reading the glossary entry it lives in.

This is one instance of a recurring pattern in the glossary: several entries (`schema slug`, `.loom callable`, `callable set`) likewise embed MUST/SHOULD usage mandates that the descriptive glossary cannot normatively host.

## Issue introduction
**Verdict:** single-commit (introduced at coinage)
**Introducing commits:** d54b426 ("pi-loom spec: resolve \"Glossary cluster: six independent defects in spec_topics/glossary.md\"", 2026-05-07)
**History:** `git log -S "disambiguated terms" -- '*glossary.md'` resolves to exactly one commit, d54b426, which authored the entire `caller` glossary entry as a new bullet. The diff shows the SHOULD disambiguation sentence present in the entry's first version; no earlier revision of the entry exists. The placement defect therefore co-arrived with the coinage of `slash caller` / `invoke parent`: the disambiguated terms and their usage mandate were introduced directly into the descriptive glossary, and the canonical `See:` pages (`slash-invocation.md`, `invocation.md`) were never given the directive. It was not a later regression — the mandate has only ever lived in the glossary.

## Solution Space

**Shape:** multiple

### Option A — Relocate the mandate to the canonical page

**Approach:** Move the normative SHOULD sentence to `slash-invocation.md` (and/or `invocation.md`), at the point where the two `caller` senses are first distinguished (near SLSH-3, which already uses both terms). Reduce the glossary `caller` entry's clause to a descriptive reminder that points at the canonical statement (e.g. "Spec prose disambiguates these as *slash caller* and *invoke parent* — see [Slash-Command Invocation]").

**Spec edits:**
- `slash-invocation.md`: add the SHOULD usage directive in the prose that introduces the slash-caller / invoke-parent distinction.
- `glossary.md` `caller` entry: replace the SHOULD sentence with a descriptive cross-reference.

**Pros:** Conforms to the glossary's own precedence rule (normative content on the canonical page). Puts the directive where authors of the affected pages will see it. Matches how sibling usage-mandate findings (`callable set`, `.loom callable`, `schema slug`) should resolve.

**Cons:** The disambiguation convention spans two pages (`slash-invocation.md`, `invocation.md`); a single canonical home must be chosen and the other cross-referenced, or the directive duplicated.

**Risks:** If duplicated across both pages, the two copies can drift; pin one as canonical and cross-reference.

### Option B — Demote the mandate to non-normative guidance

**Approach:** Rewrite the glossary clause so it states the disambiguation as a descriptive observation rather than a SHOULD directive (e.g. "Prose distinguishes these as *slash caller* and *invoke parent* where context is otherwise ambiguous"), coining no normative obligation anywhere.

**Spec edits:**
- `glossary.md` `caller` entry: drop the SHOULD modal, leaving descriptive prose only.

**Pros:** Smallest edit; removes the normativity/placement contradiction in one spot; no new canonical-page content needed.

**Cons:** Loses an authoring convention the corpus otherwise treats as worth enforcing (its sibling term entries use MUST-level mandates). Leaves term-usage drift uncontrolled.

**Risks:** Inconsistent with the corpus direction of pinning per-term usage conventions; may be re-flagged later as a missing convention.

### Recommendation

Option A. The disambiguation is a convention the corpus elsewhere treats as normative (the sibling term entries pin equivalent MUST mandates), and `slash-invocation.md` already exercises both terms in SLSH-3, so it is the natural normative home; the glossary then carries only a descriptive pointer, consistent with its declared role. Edge case for the implementer: pick one canonical page (`slash-invocation.md` is the stronger anchor since SLSH-3 depends on the distinction) and have `invocation.md` and the glossary cross-reference it rather than restating the directive.

## Consequence

**Severity:** advisory

Authors and reviewers working on `slash-invocation.md` or `invocation.md` in isolation never see the disambiguation directive, so the bare word `caller` can keep accreting in ambiguous contexts on the very pages the convention is meant to govern. No runtime behaviour diverges; the cost is prose drift and a normative rule that the corpus's own precedence rule renders non-binding where it sits.
