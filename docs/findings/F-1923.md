## Problem

Spec `PIC-20` defines the `IdSource` seam with two members ‚Äî `newInvocationId()` and `newToolCallId()` ‚Äî and `host-interfaces-core.md` ¬ß*Tool execution from loom code* mandates minting each code-side `loom-direct:` `toolCallId` through the seam, with no `crypto.randomUUID()` call outside the `CryptoIdSource` adapter. The plan describes only the first member: `V8d`'s **Adds** lists `IdSource (newInvocationId() UUID)`, and `V8d-T`'s `PIC-20` test asserts `newInvocationId()` is the only *direct* `crypto.randomUUID` reference. A conforming two-method `CryptoIdSource` adapter delegates both members and so has two direct references, meaning the test as worded fails against a conforming adapter or forces a non-conforming single-method one. `V14a`, the code-side `toolCallId` minting owner the spec names, has no seam member to call and does not list `V8d` in its `Deps`.

## Issue introduction

**Verdict:** spec-evolution divergence (plan stale against a later spec change)
**Introducing commit:** `0c6c347` ‚Äî "pi-loom spec-plan: fix F-1396 ‚Äî pin toolCallId UUID body format + route minting through PIC-20" (2026-06-25)
**History:** The plan corpus is git-tracked. The `IdSource` seam was authored at the plan's inception (`c6a664e`, 2026-06-10, then in the bundled `V8b-clock-fs-id-watch-token-seams.md`) describing a single member, `IdSource (newInvocationId() UUID)`, with the `PIC-20` test asserting `newInvocationId()` as the only `crypto.randomUUID` site. That description was **correct at the time** ‚Äî the spec's `PIC-20` had only `newInvocationId()`. The second member `newToolCallId()` was added to the spec later, at `0c6c347` (F-1396), which also pinned the `loom-direct:` `toolCallId` minting through the seam. The plan was never updated to follow that spec change; `git log -S 'newToolCallId'` over the plan corpus returns no commit, confirming the member has never appeared in any plan file. The over-bundling split (`cb8737a`, 2026-06-12) carried the stale single-member form forward into the present `V8d`/`V8d-T` leaf files unchanged.

## Solution approach

Rewrite `V8d`'s **Adds** so the `IdSource` description names both seam members, `newInvocationId()` and `newToolCallId()`, each minted through the seam per `PIC-20`. Rewrite `V8d-T`'s `PIC-20` test bullet so the exempt ambient site is the `CryptoIdSource` adapter rather than the lone `newInvocationId()` method, permitting both delegating sites and flagging any other direct `crypto.randomUUID` reference, and update `V8d`'s **Ships when** in lockstep so it no longer asserts `newInvocationId()` as the sole direct site. Add `V8d` to `V14a`'s `Deps` so the `newToolCallId()` member lands before or with its code-side `toolCallId` consumer.

## Solution constraints

- The spec is read-only for this fix ‚Äî `PIC-20` and `host-interfaces-core.md` already define and mandate both members; the gap is plan-internal.
- Out of scope: the `H3a` identifier-keyed scan, which sources seam signatures from the spec page and needs no edit.
