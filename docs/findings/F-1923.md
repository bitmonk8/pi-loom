## Finding

Spec `PIC-20` (`host-interfaces-services.md#pic-20`) defines the `IdSource` seam with **two** members â€” `newInvocationId()` and `newToolCallId()` â€” and its `CryptoIdSource` adapter delegates *each* to `crypto.randomUUID()`. The accompanying normative usage constraint (binding under `GOV-18` arm (a) even though the member *signatures* are not) requires the runtime to mint both the per-invocation `invocationId` **and** each code-side `loom-direct:` `toolCallId` UUID body through this seam, and to make no `crypto.randomUUID()` call outside the adapter. `host-interfaces-core.md` Â§*Tool execution from loom code* pins the second consumer explicitly: the `toolCallId` is "minted fresh per code-side tool call through the `IdSource` seam (PIC-20)".

The plan describes only the first member. `V8d` (`V8d-clock-id-seams.md`) lists `IdSource (newInvocationId() UUID)` in its **Adds**, and its paired test leaf `V8d-T` (`V8d-T-clock-id-seams.md`) asserts under `PIC-20` that "`IdSource.newInvocationId()` is the only *direct* `crypto.randomUUID` reference". `newToolCallId` appears in no plan file. The single-reference assertion directly conflicts with the spec adapter's second `crypto.randomUUID()` call site: a conforming `CryptoIdSource` that delegates both methods has **two** direct references in the adapter, so the test as worded either fails against a conforming adapter or forces a non-conforming single-method one.

The downstream consumer is `V14a` (`V14a-tool-calls.md`), the code-side `<name>(args)` dispatch owner that the spec names as the `toolCallId` minting site. `V14a` neither references the seam member nor lists `V8d` in its `Deps`, so its implementer is left to invent an unplanned `IdSource` member or to call `crypto.randomUUID()` directly â€” which trips the `H3a` identifier-keyed ambient-access scan (and the `H2a` no-globals gate). Two reasonable implementers would diverge here, and at least one path produces a build that fails the ambient gate.

## Issue introduction

**Verdict:** spec-evolution divergence (plan stale against a later spec change)
**Introducing commit:** `0c6c347` â€” "pi-loom spec-plan: fix F-1396 â€” pin toolCallId UUID body format + route minting through PIC-20" (2026-06-25)
**History:** The plan corpus is git-tracked. The `IdSource` seam was authored at the plan's inception (`c6a664e`, 2026-06-10, then in the bundled `V8b-clock-fs-id-watch-token-seams.md`) describing a single member, `IdSource (newInvocationId() UUID)`, with the `PIC-20` test asserting `newInvocationId()` as the only `crypto.randomUUID` site. That description was **correct at the time** â€” the spec's `PIC-20` had only `newInvocationId()`. The second member `newToolCallId()` was added to the spec later, at `0c6c347` (F-1396), which also pinned the `loom-direct:` `toolCallId` minting through the seam. The plan was never updated to follow that spec change; `git log -S 'newToolCallId'` over the plan corpus returns no commit, confirming the member has never appeared in any plan file. The over-bundling split (`cb8737a`, 2026-06-12) carried the stale single-member form forward into the present `V8d`/`V8d-T` leaf files unchanged.

## Solution Space

**Shape:** single

### Recommendation

Reconcile the plan's `IdSource` seam description to match `PIC-20`'s two-member adapter. The spec is read-only for this fix â€” `PIC-20` already defines both members and `host-interfaces-core.md` Â§*Tool execution from loom code* already mandates minting the `toolCallId` through the seam; the gap is entirely plan-internal.

- In `docs/plan_topics/V8d-clock-id-seams.md` **Adds**, change the `IdSource` member description so it names both seam members, `newInvocationId()` and `newToolCallId()`, both minted through the seam (the `CryptoIdSource` adapter delegating each to `crypto.randomUUID()`), per `PIC-20`. The `newToolCallId()` member is the code-side `loom-direct:` `toolCallId` UUID-body source named by `host-interfaces-core.md` Â§*Tool execution from loom code*.

- In `docs/plan_topics/V8d-T-clock-id-seams.md`, change the `PIC-20` test bullet so the exempt ambient site is the `CryptoIdSource` *adapter* rather than the lone `newInvocationId()` method: both `newInvocationId()` and `newToolCallId()` delegate to `crypto.randomUUID()` inside the adapter, each carrying its own same-line `// allow-ambient: crypto.randomUUID â€” IdSource` comment, and the `H3a` identifier-keyed scan permits exactly the adapter's two sites and flags any other direct reference. Update `V8d`'s **Ships when** in lockstep so it no longer asserts `newInvocationId()` as the *sole* direct `crypto.randomUUID` site.

Edge cases for the implementer: `H3a` declares seam signatures by sourcing them from the spec page, so it already picks up both members and needs no edit â€” the change is confined to `V8d`'s behavioural description and `V8d-T`'s test wording. Separately, `V14a` is the consumer the spec names for `toolCallId` minting yet does not list `V8d` in its `Deps`; the implementer of `V14a` will need the `newToolCallId()` member to exist, so the seam member must land before or with `V14a`.
