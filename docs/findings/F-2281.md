## Finding

The *Wire serialisation* clause of `audit-wire-and-canary.md` pins the audit's machine-parseable stdout stream to byte-level exactness: records are "line-delimited UTF-8 text … one record per line, terminated by a single `\n`", fields are tab-delimited with an explicit no-tab obligation, `proposed-resolution` is "plain printable ASCII text (no tabs, no newlines, no non-ASCII characters)", the `<n/a>` sentinel is exact, and two runs over the same tree "emit byte-identical output". Every byte of the wire is pinned **except** the behaviour of the stream that carries it: the clause never states that the audit writes its records without platform newline translation.

The contract therefore silently assumes the single `\n` terminator survives to the CI parser unmolested. On a host whose runtime applies text-mode newline translation, a written `\n` becomes `\r\n`, so each record line ends with a stray `\r` immediately before the terminating `\n`. That `\r` lands inside the trailing `proposed-resolution` field — a control character the field's own "no non-printable" intent forbids — and breaks any CI parser doing exact comparison (e.g. the `<n/a>` sentinel match, or the canary's `walked=<N> recognised=<M>` pair-set match). The "byte-identical across runs" guarantee then also fails to hold across hosts.

The audit's pinned runtime (a TypeScript walker run under `npm test`, i.e. Node) does not perform such translation on `process.stdout`, so the obvious implementation is unaffected in practice. But the spec never pins the runtime to that guarantee, never pins binary / no-translation stream mode, and explicitly leaves harness wiring (including spawning the audit "as a child process") an implementation choice. For a wire contract that pins every other byte, the line terminator's exactness against CRLF translation is the one byte left to assumption.

## Solution Space

**Shape:** single

### Recommendation

In the *Wire serialisation* clause, pin that the audit emits its stdout records as raw UTF-8 bytes with the single `\n` terminator preserved on every host — i.e. the audit MUST NOT write through a stream that performs platform newline translation, so no record line carries a `\r` before its terminating `\n`. State this on the same footing as the existing no-tab and `<n/a>`-sentinel obligations, so the byte-exact wire contract is closed end-to-end.

Edge cases for the implementer:
- The prohibition applies to every record class (violation, `infra`, and the once-per-invocation `canary` record), since all share the wire shape.
- A bare `\r` (U+000D) is a control character and is already inconsistent with the `proposed-resolution` "plain printable ASCII" intent; the pin makes that explicit at the terminator level rather than relying on the field-content rule.
- The CI-parser-side mitigation ("tolerate an optional trailing `\r`") is inferior here: the CI parser is a deferred / out-of-scope artefact at loom 1.0, and pinning the emitter keeps the byte-identical-across-runs guarantee literally true rather than parser-conditional.

## Consequence

**Severity:** advisory

If the audit is ever realised on a runtime that text-mode-translates stdout, every record gains a trailing `\r` and exact-match CI parsing fails; the byte-identical-output guarantee then holds only within a single host class. The pinned Node runtime does not translate, so implementers following the obvious path produce a working system — the gap is an unstated precondition rather than a live divergence.
