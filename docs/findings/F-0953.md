## Problem

The Tests fields of `V10b-package-discovery.md` and `V10b-T-package-discovery.md` cite two bullets as `` `DISC-6b` `` and `` `DISC-6c` `` — plan-coined sub-labels written in the same backtick `PREFIX-N` shape as a genuine REQ-ID. `DISC` is a registered spec REQ-ID prefix, but no `DISC-6b` or `DISC-6c` exists in the spec corpus; the bounded package walk is owned by the single registered REQ-ID `DISC-6`. The shape is one the spec's identifier governance rejects (a REQ-ID prefix paired with a non-numeric tail), so an implementer looking these tokens up finds nothing and may conflate them with real REQ-IDs. The assertion text in each bullet is complete, so the defect is identifier hygiene, not a behavioural gap.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** fea9f7f — pi-loom plan: resolve "Three V10b Tests bullets share the bare DISC-6 identifier" (2026-06-17, Thomas Andersen)
**History:** The two affected bullets originally cited the bare `DISC-6` REQ-ID (each already carrying its distinguishing parenthetical). Commit `fea9f7f` renamed them to `DISC-6b` / `DISC-6c` across both `V10b-package-discovery.md` and `V10b-T-package-discovery.md` to resolve a prior review finding that flagged three Tests bullets sharing the bare `DISC-6` identifier. That edit removed the duplicate-identifier symptom but introduced the present defect — minting REQ-ID-shaped tokens for identifiers the spec never registered.

## Solution approach

In the Tests fields of both `V10b-package-discovery.md` and `V10b-T-package-discovery.md`, strike the coined tokens `` `DISC-6b` `` and `` `DISC-6c` `` and re-cite both bullets against the registered REQ-ID `DISC-6`. Retain each bullet's existing scenario parenthetical as the human-readable handle that distinguishes the sibling `DISC-6` bullets, so the resolution does not reintroduce the bare-`DISC-6` duplication the rename was meant to fix.

## Solution constraints

- The spec is read-only for this fix — `DISC-6` remains the sole DISC anchor for the bounded walk; no spec edit is required.
- Do not introduce a `cka-N` token: `cka-N` code-keyed areas are reserved for un-anchored normative MUSTs, and `DISC-6` is an anchored numbered REQ-ID.
