## Problem

The *Sequential by default* cross-cutting rule in `conventions.md` says an ESLint `no-restricted-syntax` rule "enumerates a per-site allow-list," but never binds "per-site allow-list" to a concrete representation at its own rule site. Two readings stay open: the allow-list is a separate config-file registry kept in sync with the source comments, or the allow-list is exactly the set of `// allow:` comments the scan discovers across `src/**`. The two readings build different CI infrastructure. Its two sibling rules — *No globals, statics, singletons* and *Specific exception types only* — each disambiguate this in their own text, leaving *Sequential by default* the only one of the three whose allow-list representation is unstated locally.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `15f69aa` (2026-05-26, "pi-loom plan: finish scaffold/template re-pivot from commit 657ee76") introduced the terse "enumerates a per-site allow-list" phrasing; `358b5df` (2026-06-12, "resolve \"ambient-access allow-list representation\"") and `506d71c` (2026-06-12, "resolve \"no-broad-catch allow-list has no defined representation or location\"") added the explicit "denotes exactly the set of those same-line comments … not a separate registry file or artefact" disambiguation to the two sibling rules without propagating it to *Sequential by default*.
**History:** `git log -S "enumerates a per-site allow-list" -- plan_topics/conventions.md` shows the phrase has been present unchanged since the scaffold re-pivot at `15f69aa`. `git show 358b5df` and `git show 506d71c` add the disambiguating clause to the *No globals, statics, singletons* and *Specific exception types only* rules respectively; neither commit touches the *Sequential by default* allow-list sentence, which still reads as it did at `15f69aa`. The actionable defect — the lone undisambiguated allow-list among three otherwise-parallel rules — is the residue of those two June-12 disambiguation passes leaving *Sequential by default* behind, on top of the base ambiguity present since inception.

## Solution approach

In the *Sequential by default* rule, add to the "enumerates a per-site allow-list" sentence the disambiguating clause its two sibling rules already carry: state that the allow-list denotes exactly the set of those same-line `// allow:` comments the scan discovers across `src/**`, not a separate registry file or artefact. This aligns the local statement with the comment-keyed reading the *No globals, statics, singletons* cross-reference and `H2a`'s tests already assume.

## Solution constraints

- Out of scope: the *No globals, statics, singletons* and *Specific exception types only* sibling rules, and the `H5a` closing-gate token-resolution sentence — all already correct.
