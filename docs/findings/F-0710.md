## Problem

`host-interfaces-core.md` reproduces four Pi-owned type surfaces inline — `ExtensionContext` (`#extensioncontext-interface`), `ExtensionCommandContext` (`#extensioncommandcontext-interface`), `AgentSession` (`#agentsession-interface`), and the `AgentMessage`-arm enumeration under `SessionContext` (`#sessioncontext-shape`) — as fenced `ts` blocks or per-arm field walks. Each is wrapped only in a "MUST be re-validated against the cited `.d.ts` on each Pi minor bump" sentence; none carries an explicit marker that the reproduction itself is illustrative rather than authoritative. The sibling internal DI seams in `host-interfaces-services.md` each open with a fixed-form non-normative-reference disclaimer, so the absence on the Pi-owned blocks reads as deliberate pinning. A reader scanning for what the spec normatively pins on Pi cannot, from local signals, distinguish these externally-owned reproductions from spec-authored normative shapes.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `0e080df` (2026-05-06), `aa334c2` (2026-06-05), `01c9db9` (2026-06-03), `860e989` (2026-06-05)
**History:** The Pi-owned blocks were authored across three independent commits: `0e080df` introduced the `ExtensionContext` and `ExtensionCommandContext` inline shapes (resolving an earlier review finding about missing member-surface signatures); `aa334c2` added the `AgentSession` member-surface block; `01c9db9` added the `SessionContext`/`AgentMessage`-arms enumeration. Each carried the "MUST be re-validated against the cited declarations on each Pi minor bump" wrapper but no explicit non-normative-reference marker — at authoring time there was no in-page precedent for such a marker, so the omission was unremarkable. The asymmetry was introduced by `860e989` (T10+T11), which retrofitted the *internal* DI seams in `host-interfaces-services.md` with the fixed-form GOV-18-arm-(a) non-normative-reference disclaimer. That commit made the absence of an analogous marker on the Pi-owned blocks salient. The subsequent file-split commit `81bf52a` moved the blocks into `host-interfaces-core.md` (then `f5e89f4`'s prior 100KB-cap split had already separated the host-interfaces material) without altering the wrapping prose. The defect is therefore a cross-commit interaction — no single commit introduced an explicit gap; the gap emerged when the sibling-page marker convention landed without being applied symmetrically to the Pi-owned blocks.

## Solution approach

Add an explicit non-normative-reference marker to each of the four blocks (`#extensioncontext-interface`, `#extensioncommandcontext-interface`, `#agentsession-interface`, `#sessioncontext-shape`), modelled on the `host-interfaces-services.md` sibling pattern but adapted for externally-owned types. Each marker should declare the cited `.d.ts` (in the relevant `@earendil-works/pi-*` package) the authoritative declaration and the inline shape an illustrative reproduction of the loom-load-bearing subset, retaining the existing re-validation obligation.

## Solution constraints

- Do not cite GOV-18 arm (a): Pi-owned types fall outside both binding arms (`gov-18-binding-prohibition` in `docs/spec_topics/governance/corpus-direction-and-scope.md`), so the sibling page's arm-(a) basis is wrong here — either cite `gov-18-binding-prohibition` or frame the marker on the "authoritative declaration is the cited `.d.ts`" basis.
- Leave each block's per-member behavioural comments, loom-side MUSTs, and the "Members loom MUST NOT touch in loom 1.0" enumerations normative and untouched — the marker disclaims the reproduction, not the loom-side rules attached to it.
- Out of scope: the `ExtensionUIContext` / `ModelRegistry` / `Model<Api>` / `CompactOptions` / `CompactionResult` references that cite a `.d.ts` without reproducing an inline member-by-member shape — the marker is only for the four reproduction blocks.
