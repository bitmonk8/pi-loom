## Finding

`host-interfaces-core.md` reproduces four Pi-owned type surfaces inline — `ExtensionContext` (`#extensioncontext-interface`), `ExtensionCommandContext` (`#extensioncommandcontext-interface`), `AgentSession` (`#agentsession-interface`), and the `AgentMessage` arm enumeration under `SessionContext` (`#sessioncontext-shape`) — as fenced `ts interface { … }` blocks or bulleted per-arm field walks. Each carries per-member signatures and behavioural JSDoc-style comments, and each is wrapped only in a sentence of the form "the inline shape below is the loom-load-bearing subset and MUST be re-validated against [the cited `.d.ts`] on each Pi minor bump." No block carries an explicit marker that the reproduction itself is illustrative; a reader has to infer from the re-validation clause that the cited `.d.ts` (not the inline block) is authoritative.

The asymmetry is salient because the *sibling* DI-seam blocks in `host-interfaces-services.md` — `Checkpoint`, `SchemaValidator`, `Clock`, `FileSystem`, `FileWatcher`, `TokenEstimator`, `IdSource` — each open with a fixed-form marker: *"The TypeScript declaration below is a non-normative reference illustrating one conforming decomposition; per [GOV-18 arm (a)](…) this internal DI seam's member set is not itself binding."* Those seams are loom-internal (arm (a)) and the spec deliberately disclaims their inline shape. The Pi-owned blocks in `host-interfaces-core.md` are a stronger case for the same marker — they reproduce types loom does not author at all — yet they carry no equivalent disclaimer. The result is that a reader scanning the page for what the spec is pinning cannot tell, from local signals, that the per-member signatures and comments are an illustrative subset of an authoritative Pi declaration rather than a spec-authored normative shape.

The loom-side MUSTs attached to these blocks — `MUST NOT capture the wider `SessionManager` instance`, `MUST NOT substitute `ctx.getContextUsage()` for per-turn token accounting`, `MUST NOT call them in loom 1.0` for the omitted-member tail, and the re-validation obligation itself — are loom-side constraints on loom's use of Pi's surface, not redefinitions of Pi's types. They are unambiguously normative regardless of how the surrounding reproduction is framed and would survive the marker addition unchanged.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `0e080df` (2026-05-06), `aa334c2` (2026-06-05), `01c9db9` (2026-06-03), `860e989` (2026-06-05)
**History:** The Pi-owned blocks were authored across three independent commits: `0e080df` introduced the `ExtensionContext` and `ExtensionCommandContext` inline shapes (resolving an earlier review finding about missing member-surface signatures); `aa334c2` added the `AgentSession` member-surface block; `01c9db9` added the `SessionContext`/`AgentMessage`-arms enumeration. Each carried the "MUST be re-validated against the cited declarations on each Pi minor bump" wrapper but no explicit non-normative-reference marker — at authoring time there was no in-page precedent for such a marker, so the omission was unremarkable. The asymmetry was introduced by `860e989` (T10+T11), which retrofitted the *internal* DI seams in `host-interfaces-services.md` with the fixed-form GOV-18-arm-(a) non-normative-reference disclaimer. That commit made the absence of an analogous marker on the Pi-owned blocks salient. The subsequent file-split commit `81bf52a` moved the blocks into `host-interfaces-core.md` (then `f5e89f4`'s prior 100KB-cap split had already separated the host-interfaces material) without altering the wrapping prose. The defect is therefore a cross-commit interaction — no single commit introduced an explicit gap; the gap emerged when the sibling-page marker convention landed without being applied symmetrically to the Pi-owned blocks.

## Solution Space

**Shape:** single

### Recommendation

Prepend an explicit non-normative-reference marker to each of the four blocks, modelled on the `host-interfaces-services.md` pattern but adapted for externally-owned types. A suitable form is:

> The TypeScript declaration below is a non-normative reference reproducing the loom-load-bearing subset of `<TypeName>` for orientation; the authoritative declaration is `dist/<path>.d.ts` in `@earendil-works/pi-<pkg>` at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin). The inline shape MUST be re-validated against that file on each Pi minor bump per [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) (item `(…)`). The loom-side MUSTs attached to individual members below remain normative.

Apply to all four sites:

- `#extensioncontext-interface` — citing `dist/core/extensions/types.d.ts` in `@earendil-works/pi-coding-agent`.
- `#extensioncommandcontext-interface` — same `.d.ts`, with the `extends ExtensionContext` clause noted as part of the illustrative reproduction.
- `#agentsession-interface` — citing `dist/core/agent-session.d.ts` (re-validation already carried by version-bump item `(n)`).
- `#sessioncontext-shape` — the marker covers the bulleted `AgentMessage`-arm enumeration as a non-normative reproduction of the per-arm field walk; cite `@earendil-works/pi-agent-core`'s `dist/types.d.ts` for `AgentMessage` itself plus the per-arm declaration sites already named in the bullets.

Do **not** cite GOV-18 arm (a) verbatim — arm (a) governs loom's own implementation surface; Pi-owned types fall outside both binding arms by [GOV-18-binding-prohibition](../governance/corpus-direction-and-scope.md#gov-18-binding-prohibition), which is the governance hook that makes the marker correct here. Either cite GOV-18-binding-prohibition or frame the marker on the "authoritative declaration is the cited `.d.ts`" basis without a GOV citation; both routes are conformant.

Implementer edge cases:

- Leave every per-member behavioural comment and every loom-side MUST untouched. The marker disclaims the *reproduction*, not the loom-side rules attached to it.
- Leave the explicit "Members loom MUST NOT touch in loom 1.0" enumerations under each block normative — those are loom-side constraints on loom's surface use, not spec-authored Pi types.
- Do not retroactively add the marker to the `ExtensionUIContext` / `ModelRegistry` / `Model<Api>` / `CompactOptions` / `CompactionResult` references that already use the "declared at … MUST be re-validated" prose without an inline shape; the marker is for the four blocks that reproduce a member-by-member surface.

## Consequence

**Severity:** advisory

A maintainer editing one of the inline blocks to "fix a Pi type" risks treating the reproduction as the source of truth rather than the cited `.d.ts`, drifting the spec away from Pi's actual surface without surfacing through the version-bump checklist. A reviewer auditing what the spec normatively pins on Pi cannot, from local signals, distinguish the externally-owned reproductions from spec-authored normative shapes; the contrast with the explicitly-disclaimed internal DI seams makes the absence read as deliberate pinning rather than omission.
