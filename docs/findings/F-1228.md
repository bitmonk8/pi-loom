## Problem

`tool-registration-lifetime.md` makes two host-behaviour assumptions that nothing in the spec's host-presupposition machinery or SDK capability inventory pins, so the version-bump editorial-review checklist never re-audits them. First, the opening `**Tool-registration lifetime and visibility.**` paragraph asserts registry entries "persist for the process lifetime", and the whole per-mode cache-reuse design depends on it тАФ yet capability-inventory item 4 records only the *absence* of `pi.unregisterTool`, not positively that registrations stay valid and reachable. Second, PIC-8 (anchor `pic-8`) clause (b) directs operators to "manually restore via Pi's `/tools` interface", but the loom 1.0 Pi-SDK pin ships no built-in `/tools` command. Every other host behaviour this section relies on is a named presupposition with a matching checklist row; these two are not.

## Issue introduction

**Verdict:** multi-commit-interaction

**Introducing commits:**
- `27ec53d` (2026-05-04, *Resolve Cluster-A: per-mode tool registration*) тАФ authored the **Tool-registration lifetime and visibility** subsection, including the opening sentence "Pi's tool registry has no `unregisterTool` API тАФ `pi.registerTool` is the only mutation, and entries persist for the process lifetime" and the design that rests on it (the `Map<schemaSlug, registeredToolName>` cache, snapshot/restore as "the only control surface for visibility", and the closing "if Pi later adds `pi.unregisterTool`" note).
- `2d2fc8e` (2026-05-06, *Pi SDK capability calls have no failure contract*) тАФ authored PIC-8's **Restore-failure protocol** paragraph in a single hunk, including clause (b)'s phrase "so an operator can manually restore via Pi's `/tools` interface". The PIC-8 anchor was later added by `d4ac81b` without touching the rule text, and the spec-corpus split (`f5e89f4`, 2026-06-04) moved the paragraph from the monolithic `spec_topics/pi-integration-contract.md` into `tool-registration-lifetime.md` without altering it.

**History:** Both load-bearing assumptions were added in this order before the host-presupposition framework existed. The named-presupposition pattern in `host-prerequisites.md` тАФ each presupposition routed to a specific `(a)`тАУ`(al)` row in `version-bump-step2.md`'s editorial-review checklist тАФ was retrofitted over the following month (`60fba43`, `803106d`, both 2026-06-07, were the wave that promoted other implicit Pi assumptions into the presupposition list). The registry-persistence property and the `/tools` operator-interface reference were not migrated in that wave, leaving them as the residual unpinned assumptions on this page. The defect is the *interaction* of (a) the original authoring commits and (b) the later retrofit not sweeping these two sentences, not either commit in isolation.

## Solution approach

Narrow PIC-8 clause (b) in `tool-registration-lifetime.md`: drop the phrase directing operators to manually restore via Pi's `/tools` interface, leaving the snapshot tool names in the `hint` without an operator-action suggestion. Separately, add a registry-persistence presupposition to `host-prerequisites.md` pinning that `pi.registerTool` entries remain present, valid, and dispatchable for the loom-extension-instance lifetime absent an explicit `pi.unregisterTool` call, add forward-links to it from capability-inventory item 4 in `capability-inventory-items.md` and from the opening `**Tool-registration lifetime and visibility.**` paragraph, and add a matching editorial-review checklist row to `version-bump-step2.md` mirroring the existing `(ao)` `registerFlag` row.

## Solution constraints

- Out of scope: the *Consequences* bullet referencing `/tools` listings in `tool-registration-lifetime.md` тАФ it describes `pi.getAllTools()` listing contents, not the existence of a `/tools` surface, and stays unchanged.
