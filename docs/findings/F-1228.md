## Finding

`tool-registration-lifetime.md` makes two distinct behavioural assumptions about Pi that nothing in the spec's host-presupposition machinery or capability inventory pins, and that the version-bump editorial-review checklist therefore never re-audits.

The first is the opening sentence of **Tool-registration lifetime and visibility** — "Pi's tool registry has no `unregisterTool` API — `pi.registerTool` is the only mutation, and entries persist for the process lifetime." The whole per-mode wiring design (the prompt-mode `Map<schemaSlug, registeredToolName>` cache that reuses prior registrations across invocations, the snapshot/restore being "the only control surface for visibility", and the cosmetic-accumulation acceptance for `pi.getAllTools()` / `/tools` listings) depends on that property. The SDK capability inventory item 4 records the *absence* of `pi.unregisterTool` but does not record positively that registrations remain valid and reachable for the lifetime of the process — a Pi minor that started garbage-collecting unused registrations, invalidated them on session swap, or scoped them per extension instance would silently break cache-hit reuse with no SDK-surface signal. Every other host behaviour the section depends on (snapshot/restore atomicity, per-session slash-handler serialisation, model-registry population timing, settings write-back preservation, operator-always-present, `registerFlag` overlap behaviour) is enumerated in `host-prerequisites.md` as a named **presupposition** with a matching row in the `version-bump-step2.md` editorial-review checklist; registry persistence is the only one missing from that pattern.

The second is PIC-8's restore-failure protocol clause (b), which directs the runtime to emit a `hint` "listing the snapshot tool names so an operator can manually restore via Pi's `/tools` interface". Pi's `docs/usage.md` slash-command table at the loom 1.0 Pi-SDK pin enumerates `/login`, `/model`, `/settings`, `/reload`, `/quit`, etc., but no built-in `/tools` command. The only `/tools` reference in the Pi tree is the example extension `examples/extensions/tools.ts`, which contributors register themselves via `pi.registerCommand("tools", …)`. PIC-8's hint therefore directs an operator to an interface the loom 1.0 SDK pin does not guarantee exists in the operator's session, and that directive ships verbatim inside the cited operator hint rather than as a softened "if your Pi installation exposes a tools-listing command" hedge.

## Issue introduction

**Verdict:** multi-commit-interaction

**Introducing commits:**
- `27ec53d` (2026-05-04, *Resolve Cluster-A: per-mode tool registration*) — authored the **Tool-registration lifetime and visibility** subsection, including the opening sentence "Pi's tool registry has no `unregisterTool` API — `pi.registerTool` is the only mutation, and entries persist for the process lifetime" and the design that rests on it (the `Map<schemaSlug, registeredToolName>` cache, snapshot/restore as "the only control surface for visibility", and the closing "if Pi later adds `pi.unregisterTool`" note).
- `2d2fc8e` (2026-05-06, *Pi SDK capability calls have no failure contract*) — authored PIC-8's **Restore-failure protocol** paragraph in a single hunk, including clause (b)'s phrase "so an operator can manually restore via Pi's `/tools` interface". The PIC-8 anchor was later added by `d4ac81b` without touching the rule text, and the spec-corpus split (`f5e89f4`, 2026-06-04) moved the paragraph from the monolithic `spec_topics/pi-integration-contract.md` into `tool-registration-lifetime.md` without altering it.

**History:** Both load-bearing assumptions were added in this order before the host-presupposition framework existed. The named-presupposition pattern in `host-prerequisites.md` — each presupposition routed to a specific `(a)`–`(al)` row in `version-bump-step2.md`'s editorial-review checklist — was retrofitted over the following month (`60fba43`, `803106d`, both 2026-06-07, were the wave that promoted other implicit Pi assumptions into the presupposition list). The registry-persistence property and the `/tools` operator-interface reference were not migrated in that wave, leaving them as the residual unpinned assumptions on this page. The defect is the *interaction* of (a) the original authoring commits and (b) the later retrofit not sweeping these two sentences, not either commit in isolation.

## Solution Space

### Recommendation

Make two edits to pin the two unbacked host assumptions on this page.

**Drop the `/tools` operator-interface phrase from PIC-8 clause (b)** in `tool-registration-lifetime.md`. Remove the wording that directs the operator to "manually restore via Pi's `/tools` interface" and ship the snapshot tool names alone in the `hint`, with no operator-action suggestion. The user-visible recovery path (`/reload`) is carried by the existing `loom-system-note` template; operators who have a tools-listing command in their Pi session can read the snapshot names from the diagnostic regardless of whether the hint names a specific command. Leave the *Consequences* bullet referencing `/tools` listings unchanged — it refers to what would appear in such a listing if one is present (e.g. via `pi.getAllTools()`), which is a true statement about the listing's contents and does not assert the listing surface exists.

**Register registry-persistence as a named presupposition.** Add a new entry to `host-prerequisites.md`: `<a id="registry-persistence-presupposition"></a> **Registration-cache persistence and validity.**`, pinning (i) `pi.registerTool` entries remain present and dispatchable for the loom-extension-instance lifetime once registered, and (ii) no Pi-side path invalidates a prior registration without an explicit `pi.unregisterTool` call (which the loom 1.0 SDK pin does not expose). Cross-reference this entry from capability-inventory item 4 in `capability-inventory-items.md` (which already documents the absence of `unregisterTool`) and from the opening paragraph of *Tool-registration lifetime and visibility* in `tool-registration-lifetime.md`. Add a corresponding `(ap)` row to `version-bump-step2.md`'s editorial-review checklist, mirroring the structure of the existing `(ao)` row (the `registerFlag` flag-overlap presupposition): a SHOULD-level audit pointing at the bundler-output site that exposes the registry — the `pi.registerTool` call site and any registry-lifecycle plumbing in `dist/core/extensions/loader.js`. Apply the same **known-fragile evidence** caveat the neighbour presuppositions use to that authoritative-source pointer, because the registry lifecycle is not a typed SDK surface and is not textually locatable at a stable path.

Sequence the work with the PIC-8 clause (b) edit first (one sentence, fully scoped to `tool-registration-lifetime.md`, no cross-page dependency), then the presupposition registration on the resulting baseline (`host-prerequisites.md`, `version-bump-step2.md`, plus cross-reference text in `tool-registration-lifetime.md` and `capability-inventory-items.md`).

## Consequence

**Severity:** advisory

A future Pi minor could weaken either property — start GC'ing registrations or remove a previously-present `/tools` operator command — with no signal at the SDK surface-inventory test and no editorial-review checklist row to catch it at bump time. In the registry case, the cache-hit `pi.registerTool` reuse would silently miss its dedup contract; in the `/tools` case, PIC-8's emitted hint directs the operator to a command Pi never shipped, degrading the restore-failure UX without breaking the runtime invariant. Neither is implementation-blocking at the loom 1.0 pin.
