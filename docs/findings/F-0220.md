## Finding

The manual real-host smoke gate records its evidence "in the triggering change's commit message (the bump commit for trigger (1), the merge commit for trigger (2)), with no separate artefact file" (`H4a`), and the loom 1.0 release gate records the same evidence "in this leaf's gate-activation commit message" (`H6a`). Both gates are audited *after the fact*: `H6a`'s Ships-when treats "has recorded its evidence record in this leaf's gate-activation commit message" as binding evidence, and `H4a` states a skipped or stale-pin run "is detectable only after the fact, by reading the triggering commit message for the recorded run."

This rests on an unstated assumption: that the commit whose message carries the evidence is the commit the auditor later reads. Nothing in the plan or the spec convention it mirrors constrains the project's merge workflow. Under a squash- or rebase-merge of the feature branch, the originating commit message (where the evidence was written) can be dropped or rewritten, so the evidence does not necessarily survive onto the commit that lands on `main`. The same assumption underlies `V18c`'s editorial audit of the version-bump checklist outcomes recorded "in the bump commit message."

The spec's governance rules already confront this exact ambiguity for a different purpose — GOV-1 and GOV-22 explicitly take "the first-parent diff … for merge or squash commits" — so the corpus has a precedent for naming which commit on `main` is authoritative, but the smoke-evidence convention never adopts it.

## Issue introduction

**Verdict:** introduced-by-later-edit

**Introducing commits:** `4099d0ac662f0ac421945799416ceeb1d9ee9fb5` (2026-06-11 14:29:03 +0200) — "pi-loom plan: resolve \"Release-gate evidence artifact has no defined committed home or format\""

**History:** The plan corpus is git-tracked. Tracing the cited prose with `git log -S` over `H4a` and `H6a`: the predecessor commit `eca63cf` (2026-06-11 07:46:58) established that the manual real-host fidelity gate requires a *falsifiable record*, but did not fix its home. Commit `4099d0a` then chose the commit message as that home — introducing the "evidence committed in this leaf's gate-activation commit message — no separate artefact file" text in `H6a` and the "in the triggering change's commit message … with no separate artefact file" text in `H4a`, both explicitly mirroring the pre-existing spec version-bump convention (`version-bump-step2.md` / `version-bump-triggers.md`). The persistence assumption entered with that home-choosing edit: the commit selected the commit message as the evidence store without addressing whether that message survives the merge workflow, and inherited the same unaddressed assumption from the spec convention it copied.

## Solution Space

**Shape:** multiple

Ownership boundary (applies to both options): the spec topics are the contract being implemented. If the plan-diff-fix loop treats the spec as read-only, Option A is out of scope for the fixer and Option B is the plan-only path; if spec edits are in scope for this finding (it is an assumptions defect that may force a spec edit), Option A is available.

### Option A — Pin the persistence requirement at the spec evidence convention

**Approach.** State, in the spec convention that owns the "in the bump commit message / produces no separate artefact file" rule, that the recorded evidence must reside in the commit that lands on `main` and survives the project's merge workflow — equivalently, that the auditor reads the first-parent commit on `main`, consistent with the first-parent-diff semantics GOV-1 / GOV-22 already use for merge and squash commits.

**Plan edits.** None beyond ensuring `H4a`, `H6a`, and `V18c` reference the spec clause rather than restating the rule independently.

**Spec edits.** Add the persistence statement to `docs/spec_topics/pi-integration-contract/version-bump-triggers.md` (the "produces no separate artefact file" sentence) and `docs/spec_topics/pi-integration-contract/version-bump-step2.md` (the "in the bump commit message" recording sentence).

**Pros.** Single canonical home; `H4a`/`H6a`/`V18c` inherit by reference, so the three plan leaves cannot drift. Reuses the corpus's existing first-parent-diff precedent.

**Cons.** Requires a spec edit; out of scope if the spec is read-only for the plan fixer.

**Risks.** None beyond keeping the plan leaves' cross-references pointed at the amended clause.

### Option B — Pin the persistence requirement in the plan leaves

**Approach.** State the same persistence requirement directly in `H4a`'s manual-smoke governance protocol (the convention's plan home), have `H6a`'s release-gate evidence record reference it, and apply it to `V18c`'s bump-commit audit, without touching the spec.

**Plan edits.** `docs/plan_topics/H4a-factory-shell-and-harness.md` evidence-record prose; `docs/plan_topics/H6a-live-corpus-activation.md` release-gate evidence record; `docs/plan_topics/V18c-version-bump-checklist.md` Ships-when audit.

**Spec edits.** None.

**Pros.** Stays inside the plan; available even when the spec is read-only.

**Cons.** The persistence rule is restated across three plan leaves, which can drift; the spec's own version-bump convention retains the same latent assumption (a separate spec-review concern).

**Risks.** Plan/spec divergence — the plan would constrain commit-message persistence while the spec convention it cites does not.

### Recommendation

Adopt Option A when spec edits are in scope: the commit-message / no-separate-artefact convention is spec-owned, `H4a` explicitly mirrors it, and pinning the persistence requirement once at the spec clause keeps `H4a`, `H6a`, and `V18c` as inheritors and prevents drift across the three plan leaves. Align the requirement with the existing GOV-1 / GOV-22 first-parent-diff semantics so the "read the commit message" audit step is unambiguous about which commit on `main` it reads. If the plan-diff-fix loop treats the spec as read-only, fall back to Option B and pin the requirement in the plan leaves, accepting the plan/spec divergence as a flagged follow-up for spec review.

## Consequence

**Severity:** advisory

If the project merges feature branches by squash or rebase, the commit message carrying the smoke / release-gate evidence can be rewritten or dropped before it reaches `main`, so the after-the-fact auditor — and `H6a`'s Ships-when gate — may find no evidence to read even though the run was performed and recorded. The recorded-evidence convention is only as reliable as an unstated assumption about the merge workflow.
