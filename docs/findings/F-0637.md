## Finding

Step 0 (d) ("Peer-dep version (lock-step, four packages)") in `capability-probe.md` writes the per-package version read as a normative algorithm: "resolve the package's main entry via `createRequire(import.meta.url).resolve("<pkg>")` … then walk parent directories from the resolved entry path up to the first `package.json` whose `name` field equals the iteration's `<pkg>` and read its `version` field." Two of the four enumerated `peer-dep-malformed-version` routing conditions are then defined in terms of that exact mechanic's failure modes — condition (1) "a `MODULE_NOT_FOUND` thrown by the main-entry resolution call" and condition (2) "a parent-directory walk that reaches the filesystem root without finding a `package.json` whose `name` field equals the iteration's `<pkg>`" — rather than in terms of an observable property of the host installation (the named package's `package.json` is unresolvable / its parsed object has no `version` field / its `version` is not valid SemVer / its `version` is out of the pinned range).

What the runtime contract actually owes its downstream consumers is the four-failure-category routing into `peer-dep-out-of-range` / `peer-dep-malformed-version` / `probe-failed` plus the `details.observed` / `details.required` / `details.package` payload shape — `code-registry-load.md`, `placeholder-rendering-b.md`, and the orchestration of (c)+(d) all key off those, not off `createRequire` semantics. The same step's SemVer comparison is treated as a *Recommended recipe (non-normative)* explicitly to leave room for a future swap, and the immediate parent step (a) Node-floor uses the same recipe / contract split. Step (d)'s algorithm is the same kind of implementation detail but is written into the normative body, with the failure taxonomy bound to its internal symbols. A future implementation that locates the four peers by a different mechanism (a workspace-aware reader, a synchronous walk rooted at `process.cwd()`, an `import.meta.resolve()` once stable, a Pi-supplied helper, a vendored loader, etc.) cannot be conformant by reproducing the observable contract — it has to either reproduce the `createRequire` + walk algorithm itself or invent a re-mapping from its native errors into "a `MODULE_NOT_FOUND` thrown by the main-entry resolution call," which makes the spec brittle against tooling drift in a way the rest of Step 0 deliberately avoids.

The fix is to restate (d)'s normative body in observable terms and demote the algorithm to a recommended-recipe callout on the same footing as (a) and (d)'s own SemVer recipe; the four-category routing, the iteration order, the short-circuit, and the payload-field shape all stay normative.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 126ff8a — pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B (2026-05-06, Thomas Andersen); d2a6d6f — pi-loom spec: resolve "Capability probe verifies only one of four lock-step peer-dep versions" (2026-05-07, Thomas Andersen); 4c27f32 — pi-loom spec: resolve "Step 0 (d) peer-dep version probe relies on a subpath export" (2026-06-03, Thomas Andersen); 9371309 — pi-loom spec: resolve "foreseeable shape failure clause leaves kind discriminator under-specified" (2026-06-05, Thomas Andersen)
**History:** The Step-0 rewrite under 126ff8a introduced (d) as a single-package read with `createRequire(...).resolve("@mariozechner/pi-coding-agent/package.json")` already written into the normative body, paired with a separated non-normative SemVer recipe. d2a6d6f generalised (d) to the four-package lock-step iteration but propagated the `createRequire`-subpath form into the per-iteration text unchanged. 4c27f32 then changed the mechanic itself — main-entry resolution + parent-directory walk — to avoid `ERR_PACKAGE_PATH_NOT_EXPORTED`, embedding the new algorithm and its specific throw sites into the normative routing prose. 9371309 closed the discriminator gap by enumerating the four `peer-dep-malformed-version` conditions explicitly, but two of those four conditions are written as failures of the 4c27f32 mechanic ("`MODULE_NOT_FOUND` thrown by the main-entry resolution call"; "parent-directory walk reaches the filesystem root") rather than as observable properties of the host installation. The defect is the *interaction*: the SemVer-recipe split established alongside the algorithm in 126ff8a was never extended to the read-mechanic itself, and each subsequent revision tightened the algorithm-and-taxonomy coupling rather than separating it.

## Solution Space

**Shape:** single

### Recommendation

Restate Step 0 (d)'s per-package read in observable terms and move the `createRequire` + parent-walk algorithm into a non-normative recipe callout on the same footing as the existing SemVer recipe at `#step-0-d-recommended-recipe`.

Concretely, in `capability-probe.md`:

- Replace the normative sentence "resolve the package's main entry via `createRequire(...).resolve("<pkg>")` … then walk parent directories … and read its `version` field" with an observable statement: *"For each package in turn, the probe MUST read the installed `version` field from the named peer package's own `package.json` by any `exports`-independent mechanic that, for an iteration whose target package name is `<pkg>`, returns the `version` field of the `package.json` whose `name` field equals `<pkg>` and which sits in the dependency tree resolvable from the loom factory's module."* Keep the "resolution context is the loom factory's module, since the extension may be installed globally" clause as a normative precondition on the mechanic.
- Re-cast the four `peer-dep-malformed-version` conditions in installation-observable terms, e.g.: (1) the named peer's own `package.json` is not resolvable from the loom factory's module; (2) no `package.json` in the resolution path has a `name` field equal to the iteration's `<pkg>`; (3) the located `package.json` parses but has no own `version` field; (4) the located `version` field is not a syntactically valid SemVer string. Keep (3) and (4) as-is (they already read in observable terms); only (1) and (2) need rephrasing off their algorithm-internal symbols. The `kind` set, the iteration order, the short-circuit, and the `details.observed` / `details.required` / `details.package` payload shape stay normative and unchanged.
- Add a new *Recommended recipe (non-normative)* callout (or extend the existing `#step-0-d-recommended-recipe` callout) that gives the `createRequire(import.meta.url).resolve("<pkg>")` + parent-directory-walk algorithm as the recommended implementation, with the existing `ERR_PACKAGE_PATH_NOT_EXPORTED` rationale and the existing "swap permitted as long as the contract is preserved" clause.
- In *Self-failure*, keep the existing carve-out as-is — "every other throw inside step (d) … routes to `kind: "probe-failed"`" already keys off "is none of the four conditions above" rather than the algorithm, so no edit is needed once the four conditions are re-stated in observable terms.

In `docs/spec_topics/diagnostics/code-registry-load.md`, mirror the wording change in the `peer-dep-malformed-version` clause: replace "the `package.json` is unresolvable via `MODULE_NOT_FOUND`, or the parent-directory walk reaches the filesystem root without a matching `package.json`" with the same observable phrasing the spec uses ("the named peer's `package.json` is not resolvable from the loom factory's module, or no resolved `package.json` carries a `name` field equal to the iteration's package"). `placeholder-rendering-b.md` does not need an edit — its `<observed>` / `<required>` table already states wire values, not throw sites.

Implementer-relevant edge cases:

- The recommended recipe MUST stay textually present, because the family-(4) CJS-reach carve-out in `audit-target-categories.md` is currently scoped to `createRequire(...).resolve(<peer-specifier>)`; dropping the recommended mechanic entirely would either widen that carve-out or strand the only currently-documented conformant recipe.
- The observable phrasing of condition (2) MUST still pin the `name`-field equality check, not a path-shape heuristic — a vendored install where four peers share one `node_modules/<pkg>` directory with a wrapper `package.json` whose `name` does not match must still route to condition (2), and an implementation that uses path-component matching instead of `name`-field equality would diverge on this case.
- The interaction with the sibling finding "probe (d) assumes discrete on-disk peer packages with readable `package.json`" should be resolved by stating the deployment precondition once at the head of (d) (per that finding); the observable phrasing here then reads as gating that precondition rather than defining it.

## Consequence

**Severity:** advisory

A second-source implementation that locates the four peers by an `exports`-independent mechanic other than `createRequire` + parent-walk would have to either reproduce the algorithm or invent a re-mapping from its native error surface into the spec's algorithm-internal failure tokens (`MODULE_NOT_FOUND` from "the main-entry resolution call"; "parent-directory walk reaches the filesystem root"). The current text is also internally asymmetric with (a) and with (d)'s own SemVer-comparator recipe split, which is the precedent contributors will reach for when revisiting (d). Nothing breaks at loom 1.0 with the recommended mechanic in place; the cost is paid by future drift.
