## Problem

Step 0 (d) ("Peer-dep version (lock-step, four packages)") in `capability-probe.md` writes the per-package version read as a normative `createRequire(import.meta.url).resolve("<pkg>")` + parent-directory-walk algorithm, and two of the four `peer-dep-malformed-version` routing conditions — `MODULE_NOT_FOUND` on the resolution call, and a parent-walk reaching the filesystem root — are defined in terms of that algorithm's internal failure modes rather than observable properties of the host installation. A second-source implementation that locates the four peers by a different `exports`-independent mechanic cannot be conformant by reproducing the observable contract; it must reproduce the algorithm or re-map its native errors into the spec's algorithm-internal failure tokens. The step also never states the deployment-shape precondition the algorithm presupposes — that each of the four `@earendil-works/*` peers exists as a discrete, on-disk package carrying a synchronously-readable `package.json` reachable by Node's CommonJS resolver — so a bundled or vendored loom build would have its peers misclassified as `peer-dep-malformed-version` on an otherwise conformant host. The sibling step (a) and (d)'s own SemVer comparison already use a normative-contract / non-normative-recipe split that (d)'s read mechanic does not.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `126ff8a` (2026-05-06, "pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B", Thomas Andersen); `d2a6d6f` (2026-05-07, "pi-loom spec: resolve \"Capability probe verifies only one of four lock-step peer-dep versions\"", Thomas Andersen); `4c27f32` (2026-06-03, "pi-loom spec: resolve \"Step 0 (d) peer-dep version probe relies on a subpath export\"", Thomas Andersen); `9371309` (2026-06-05, "pi-loom spec: resolve \"foreseeable shape failure clause leaves kind discriminator under-specified\"", Thomas Andersen)
**History:** The Step-0 rewrite under `126ff8a` introduced (d) as a single-package read with `createRequire(...).resolve("@mariozechner/pi-coding-agent/package.json")` already written into the normative body, paired with a separated non-normative SemVer recipe — and baked the discrete-on-disk-install assumption into the mechanic from the moment it landed, without stating it as a precondition. `d2a6d6f` generalised (d) to the four-package lock-step iteration but propagated the `createRequire`-subpath form into the per-iteration text unchanged, carrying the unstated assumption forward. `4c27f32` then changed the mechanic itself — main-entry resolution + parent-directory walk — to avoid `ERR_PACKAGE_PATH_NOT_EXPORTED`, embedding the new algorithm and its specific throw sites into the normative routing prose and adding the root-bound-walk failure condition (still presupposing discrete on-disk packages). `9371309` closed the discriminator gap by enumerating the four `peer-dep-malformed-version` conditions explicitly, but two of those four are written as failures of the `4c27f32` mechanic rather than as observable properties of the host installation. The defect is the *interaction*: the SemVer-recipe split established alongside the algorithm in `126ff8a` was never extended to the read-mechanic itself, each subsequent revision tightened the algorithm-and-taxonomy coupling, and the discrete-on-disk deployment shape the algorithm presupposes was never promoted from an implicit assumption to a stated precondition.

## Solution approach

Add a deployment-shape precondition at the head of Step 0 (d) stating that the four `@earendil-works/*` peers are presupposed to be installed as discrete, on-disk packages with synchronously-readable `package.json` files reachable by Node's CommonJS resolver from the loom factory's module, and forward-link `host-prerequisites.md#pi-sdk-pin-manifest-lock-step` as the corpus source of the packaging model. Rewrite (d)'s normative read sentence in observable terms — the version field read from the named peer's own `package.json` by any `exports`-independent mechanic — and recast routing conditions (1) and (2) as installation-observable properties (the named peer's `package.json` is unresolvable from the loom factory's module; no resolved `package.json` carries a `name` field equal to the iteration's package). Demote the `createRequire` + parent-walk algorithm into the `#step-0-d-recommended-recipe` callout alongside the existing SemVer recipe, retaining its `ERR_PACKAGE_PATH_NOT_EXPORTED` rationale and swap-permitted clause. Mirror the same observable phrasing in `code-registry-load.md`'s `loom/load/host-incompatible` `peer-dep-malformed-version` clause.

## Solution constraints

- Keep a `createRequire(...).resolve("<peer-specifier>")`-based recipe textually present in (d): the family-(4) CJS-reach carve-out in `audit-target-categories.md` is scoped to it, and removing it strands the only documented conformant recipe.
- Preserve condition (2)'s `name`-field equality check in the observable rewrite; do not substitute a path-shape heuristic.
