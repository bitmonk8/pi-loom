## Finding

The `**Loom return type.**` paragraph in `docs/spec_topics/functions.md` (the `#loom-return-type` anchor) says the inferred return type "reconciles the tail-expression type … with the operand type of every **reachable** early `return expr` (per [Return Statement](./return.md))". `reachable` is the load-bearing modifier — it determines which `return` operands enter the least-upper-bound computation — but it is not defined anywhere in the spec corpus, and the cross-reference into `return.md` does not pin it either. `return.md` only states the dual rule that *code after* a `return` in the same block is unreachable and emits the `loom/parse/unreachable-code` warning; it never defines what makes a `return` itself reachable, nor how (or whether) `return`s emitting that warning participate in inference.

The ambiguity is observable. Consider:

```loom
fn f() {
  return "hi"
  return 42      // loom/parse/unreachable-code (warning, not error)
}
```

Under one reading `f` infers `string`; under the other it infers `string | integer`. Both readings are consistent with the current text. The same question recurs for `return`s under a statically-false `if` branch, under an infinite `while (true)`, and after an earlier `?` whose `Err` arm desugars to `return Err(e)`. Because `loom/parse/unreachable-code` is a warning (not an error), a body that compiles and ships still has unresolved meaning for its inferred type, and two conforming implementations can diverge on (a) whether `loom/parse/return-no-common-type` fires, and (b) what type cross-loom `invoke<Schema>` static checks see at the call site.

The page leans on the analogous reconciliation for `match` arms, ternary branches, and array literals, but none of those carry an unreachability concept — every arm/branch/element is by construction live. Inheriting their LUB discipline does not transfer a reachability rule, because they have none to transfer.

## Solution Space

**Shape:** single

### Recommendation

In the `**Loom return type.**` paragraph, replace "every **reachable** early `return expr`" with a definition-bearing form, and adjust the wrapping paragraph in lock-step:

- Pin the rule normatively: *every* early `return expr` in the body contributes its operand type to the inference, including `return`s whose enclosing position is statically dead and that emit `loom/parse/unreachable-code`. Reachability analysis in loom 1.0 is not a sound type-system input — the warning is editorial — so excluding warned-on `return`s would require a separate normative reachability predicate the spec does not define and the analyser is not asked to compute.
- Drop the bare word "reachable"; the paragraph now reads "the operand type of every early `return expr` (per [Return Statement](./return.md), including `return`s whose statement position is statically unreachable — `loom/parse/unreachable-code` remains an editorial warning and does not exclude the operand from inference)".
- The `?`-desugared `return Err(e)` is just an early `return expr` under this rule — the existing `Err`-arm-desugar sentence is consistent with the clarification and needs no edit.
- Add one sentence at the bottom of the same paragraph (or as a parenthetical) explicitly noting the parallel `match`/ternary/array-literal LUB sites carry no analogous reachability concept, so the inheritance of their LUB discipline does not import a reachability rule.

Edge cases the implementer must handle:

- A body whose tail is statically unreachable (e.g. `return x; tail_expr`) — the tail's type still participates (the empty-tail rule is the only way to elide a tail contribution), per the dual symmetry with this rule.
- A body whose every `return` is statically unreachable but whose tail is present — equivalent to a no-`return` body for inference purposes only by accident; the rule above does not special-case it.
- Update `V3d` / `V3d-T` to add at least one fixture per case: (i) reachable `return string` + unreachable `return integer` infers `string | integer` (or fires `return-no-common-type` if there is no LUB); (ii) the `?`-desugar case continues to wrap as today.

The alternative — "unreachable `return`s do not contribute" — is rejected because it forces every conforming parser to implement a reachability analysis the spec otherwise neither defines nor requires, and creates a class of programs whose inferred type silently changes when a `loom/parse/unreachable-code` warning is suppressed or surfaced.

## Consequence

**Severity:** correctness

Two reasonable implementers infer different return types for the same body, which propagates to different `loom/parse/return-no-common-type` firings, different `Result<T, …>` wrap payloads, and different cross-loom `invoke<Schema>` static-check outcomes. Conformance fixtures cannot be written for the affected bodies without first picking the rule.
