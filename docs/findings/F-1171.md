## Problem

The `**Loom return type.**` paragraph at `#loom-return-type` in `functions.md` says inference reconciles the tail-expression type with the operand type of every **reachable** early `return expr`. The word `reachable` is load-bearing — it determines which `return` operands enter the least-upper-bound computation — but it is defined nowhere in the spec corpus, and the cross-reference into `return.md` does not pin it (`return.md` only states that code *after* a `return` is unreachable and emits `loom/parse/unreachable-code`, never what makes a `return` itself reachable or whether a warned-on `return` contributes to inference). Two conforming implementations can therefore infer different return types for the same body, diverging on whether `loom/parse/return-no-common-type` fires and on what type cross-loom `invoke<Schema>` static checks observe at the call site.

## Issue introduction

**Verdict:** introduced-by-fix
**Introducing commits:** 08886d9 (`pi-loom spec: resolve "loom return-type inference does not reconcile early-return operand types"`, 2026-06-07)
**History:** The `Loom return type` paragraph was a single sentence ("the type of its tail expression, wrapped in `Result<T, QueryError>` if any `?` appears in the body") with no `return`-operand reconciliation at all until commit `08886d9` rewrote it to introduce the LUB discipline. That rewrite added the word **reachable** as a qualifier on early `return expr` but did not define it, and made no companion edit in `return.md` (whose only reachability mention remains the pre-existing `loom/parse/unreachable-code` warning on *code after* a `return`). The defect was created by the same commit that closed the prior "does not reconcile early-return operand types" finding: the reconciliation was added, the reachability qualifier was added with it, and the qualifier's definition was the dropped stitch. No other commit has touched the paragraph since (`git log --follow` shows `08886d9` as the most recent edit to `functions.md`).

## Solution approach

Clarify the `**Loom return type.**` paragraph at `#loom-return-type` so the `reachable` qualifier is defined, directing the inference to include every early `return expr` operand regardless of static reachability — loom 1.0 defines no reachability predicate the type system consumes, and `loom/parse/unreachable-code` is an editorial warning rather than an inference input. Drop the bare word `reachable` and keep the cross-reference to `return.md`. Clarify that the parallel `match`-arm, ternary, and array-literal LUB sites carry no analogous reachability concept, so inheriting their LUB discipline imports no reachability rule.
