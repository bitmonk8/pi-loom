## Problem

`H4a`'s Adds prose asserts that, because the factory body returns synchronously with no awaited work, `"MUST NOT throw out of the factory body"` is the **complete** never-fault prohibition for the factory boundary. That completeness claim is explicitly stated to be load-bearing on every factory-body call "completing — and taking effect — synchronously: none returns a `Promise` whose resolution is required for the registration to be durable." The factory body, per the same paragraph, runs three call types: `pi.registerFlag`, `pi.registerMessageRenderer`, and the three factory-time `pi.on` subscriptions (`resources_discover`, `session_start`, `session_shutdown`).

The only spec backing cited is for one of those three. The Adds sentence points to [`extension-bootstrap-and-per-loom.md` §"Renderer registration"](../spec_topics/pi-integration-contract/extension-bootstrap-and-per-loom.md#renderer-registration), which establishes the synchronous-void / no-separate-rejection property for `pi.registerMessageRenderer` alone — that section pins the signature as returning `void` and states "Failures propagate as synchronous throws … there is no separate Promise rejection path." For `pi.registerFlag` and the three `pi.on` subscriptions the same synchronous-and-void / no-unawaited-rejecting-`Promise` property is asserted with no citation. Nothing in the spec pins those calls' return shape on the same footing; the bootstrap-failure preamble of the same spec page in fact speaks generically of "a throw **or rejection** from any of the registration calls," which leaves a rejecting-`Promise` arm open rather than foreclosed.

`H4a`'s Tests exercise the synchronous-throw path and the returned-value-not-thenable path (the factory's own return value is `undefined`), but neither rules out a fire-and-forget rejecting `Promise` produced by `registerFlag` or a `pi.on` subscription — precisely the residual the word "complete" claims to foreclose. The completeness claim therefore spans three call types while carrying spec backing for only one.

The current shape of the defect (a completeness claim spanning three call types but backed for one) is the joint product of the earlier broad claim and the later partial-backing edit; the durable fix reconciles the claim and its backing rather than reverting either commit.

## Solution approach

Pin the synchronous-void property in the spec for all three factory-body call types, then point `H4a`'s claim at that backing.

In `extension-bootstrap-and-per-loom.md`, add a clause — on the same footing as the existing §"Renderer registration" wording — establishing that `pi.registerFlag` and the three factory-time `pi.on` subscriptions (`resources_discover`, `session_start`, `session_shutdown`) return `void` synchronously, with failures propagating only as synchronous throws and no separate `Promise`-rejection path. This property holds at the pinned `@earendil-works/pi-coding-agent` `~0.75.5` surface (resolving to `0.75.5`), confirmed at both the type and implementation level:

- **Type surface** (`dist/core/extensions/types.d.ts`, `ExtensionAPI`): every `on(...)` overload, `registerFlag`, and `registerMessageRenderer` is declared returning `: void` — none returns a `Promise`.
- **Implementation** (`dist/core/extensions/loader.js`, `createExtensionAPI`): `on(event, handler)`, `registerFlag(name, options)`, and `registerMessageRenderer(customType, renderer)` each perform only a synchronous `runtime.assertActive()` guard plus synchronous `Map`/array writes that store the handler/flag/renderer. None constructs a `Promise`, awaits anything, or spawns a fire-and-forget task; the handler is stored, not invoked, at registration time. The sole failure mode is a synchronous `throw` from `assertActive()` (a plain `if (...) throw`), which a synchronous per-call `try`/`catch` catches and which the host's `await factory(api)` wrapper also catches.

This confirms there is no background-failure (un-awaited rejecting-`Promise`) path for these three call types, so the "complete" framing is sound once the backing exists. Then in `H4a`'s Adds, broaden the citation so `pi.registerFlag` and the `pi.on` subscriptions reference the new clause covering all three call types, not only the renderer's, and reword the bootstrap-failure preamble so it no longer leaves a generic "throw **or rejection**" arm open for these calls.

## Solution constraints

- The new spec clause must match exactly what the pinned `0.75.5` surface guarantees and assert nothing beyond it; it covers the registration calls at factory time only. The event handlers passed to `pi.on(...)` may themselves be async and reject when the event later fires — that is outside the factory boundary and outside this clause's scope.
- Per `conventions.md` §"Spec drift", author the spec clause in `extension-bootstrap-and-per-loom.md` first, then land the `H4a` citation/preamble edits that depend on it.
- The grounded synchronous-void property is inherited by `V9k`/`V9p`, which wrap exactly these calls in synchronous `try`/`catch`; the spec clause must keep that synchronous-`try`/`catch` assumption valid (no rejected-`Promise` handling is required for these calls).
