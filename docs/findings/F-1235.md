## Finding

`H4a`'s Adds prose asserts that, because the factory body returns synchronously with no awaited work, `"MUST NOT throw out of the factory body"` is the **complete** never-fault prohibition for the factory boundary. That completeness claim is explicitly stated to be load-bearing on every factory-body call "completing — and taking effect — synchronously: none returns a `Promise` whose resolution is required for the registration to be durable." The factory body, per the same paragraph, runs three call types: `pi.registerFlag`, `pi.registerMessageRenderer`, and the three factory-time `pi.on` subscriptions (`resources_discover`, `session_start`, `session_shutdown`).

The only spec backing cited is for one of those three. The Adds sentence points to [`extension-bootstrap-and-per-loom.md` §"Renderer registration"](../spec_topics/pi-integration-contract/extension-bootstrap-and-per-loom.md#renderer-registration), which establishes the synchronous-void / no-separate-rejection property for `pi.registerMessageRenderer` alone — that section pins the signature as returning `void` and states "Failures propagate as synchronous throws … there is no separate Promise rejection path." For `pi.registerFlag` and the three `pi.on` subscriptions the same synchronous-and-void / no-unawaited-rejecting-`Promise` property is asserted with no citation. Nothing in the spec pins those calls' return shape on the same footing; the bootstrap-failure preamble of the same spec page in fact speaks generically of "a throw **or rejection** from any of the registration calls," which leaves a rejecting-`Promise` arm open rather than foreclosed.

`H4a`'s Tests exercise the synchronous-throw path and the returned-value-not-thenable path (the factory's own return value is `undefined`), but neither rules out a fire-and-forget rejecting `Promise` produced by `registerFlag` or a `pi.on` subscription — precisely the residual the word "complete" claims to foreclose. The claim is an assumption about the pinned `@earendil-works/pi-coding-agent` surface's current behaviour, asserted without backing for two of the three call types.

The current shape of the defect (a completeness claim spanning three call types but backed for one) is the joint product of the earlier broad claim and the later partial-backing edit; the durable fix reconciles the claim and its backing rather than reverting either commit.

## Solution Space

**Shape:** multiple

### Option A — Pin the synchronous-void property in the spec for all three factory-body call types

**Approach.** After verifying against the pinned `@earendil-works/pi-coding-agent` `ExtensionAPI` surface that `pi.registerFlag` and the factory-time `pi.on` subscriptions return `void` synchronously with no separate `Promise`-rejection path, extend `extension-bootstrap-and-per-loom.md` so that property is stated for those calls on the same footing as the renderer's, then point `H4a`'s Adds citation at the backing for all three call types.

**Plan edits.** In `H4a` Adds, broaden the citation so `registerFlag` and the `pi.on` subscriptions reference a spec anchor that establishes their synchronous-void return shape, not only the renderer's.

**Spec edits.** In `extension-bootstrap-and-per-loom.md`, add a clause establishing that `pi.registerFlag` and the factory-time `pi.on` subscriptions return `void` synchronously with no separate `Promise`-rejection path (the renderer section's wording is the model).

**Pros.** Keeps the "complete" claim literally true; gives `V9k`/`V9p`'s synchronous `try`/`catch` a grounded basis; single authoritative home for the property.

**Cons.** Requires a spec edit (cross-document) and an SDK-surface verification pass.

**Risks.** If the pinned SDK does not actually guarantee synchronous-void for these calls, the new clause would be false — verification against the pin is mandatory before authoring it.

### Option B — Narrow the completeness claim in the plan and assign the residual to the SDK-surface audit

**Approach.** Scope the "complete" prohibition in `H4a` Adds to the renderer (the only spec-backed call), and state that for `pi.registerFlag` and the three `pi.on` subscriptions the synchronous-void property is a pinned-SDK assumption whose verification is owned by the `V18` SDK-surface audit (`V18a`/`V18b`).

**Plan edits.** In `H4a` Adds, replace the unqualified "complete … for this boundary" with text that distinguishes the spec-backed renderer property from the assumed-and-residual `registerFlag` / `pi.on` property and names the `V18` audit as the residual owner. In `V18a`, optionally add an `SDK_SURFACE_INVENTORY` assertion that the `pi.registerFlag` / `pi.on` rows carry the synchronous-void return shape.

**Spec edits.** None.

**Pros.** Plan-internal; no spec change; honest about the residual.

**Cons.** Leaves a real residual unverified until `V18`; weakens the "complete" framing.

**Risks.** If `V18a`/`V18b` do not in fact assert the return shape of these calls, the residual is left with no actual owner.

### Recommendation

Take **Option A** when verification against the loom 1.0 Pi-SDK pin confirms `pi.registerFlag` and the factory-time `pi.on` subscriptions are synchronous-and-void (the renderer section's signature pin and the `ExtensionAPI` surface convention both point that way); this reconciles the completeness claim with its backing at a single authoritative home and grounds the synchronous-`try`/`catch` assumption `V9k`/`V9p` inherit. If verification finds either call can return a thenable, fall back to **Option B**. In either case, do not assert the synchronous-void property — in the spec or the plan — without first confirming it against the pinned SDK surface; the spec is read-only for this fix unless Option A is chosen.

## Consequence

**Severity:** correctness

The completeness claim is declared load-bearing, yet two of the three factory-body call types it covers have no backing for the synchronous-void property it rests on. If `pi.registerFlag` or a factory-time `pi.on` subscription can return a thenable that later rejects, that rejection escapes the synchronous per-call `try`/`catch` and faults outside the never-throw boundary unobserved — and `H4a`'s Tests would not catch it. Two reasonable implementers diverge: one trusts "complete" and writes only synchronous `try`/`catch`; another adds rejected-`Promise` handling for these calls. The same unverified assumption is inherited by `V9k`/`V9p`, which wrap exactly these calls in synchronous `try`/`catch`.
