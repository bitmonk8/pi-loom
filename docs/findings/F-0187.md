## Finding

`docs/plan.md` "How to use this plan" step 2 enumerates the three transitive-completeness triggers verbatim — "an executable REQ-ID, a citing test that *closes* a coverage-matrix-mapped numbered REQ-ID, or an un-anchored normative MUST" — before pointing at `conventions.md` §REQ-ID discipline (*Transitive-completeness plan-maintenance*) "which states that obligation once and governs its exact triggers." The sentence asserts single-sourcing ("states that obligation once") while itself being a second copy of the trigger list, so the assertion is self-contradicting.

The canonical owner is the *Transitive-completeness plan-maintenance* clause in `conventions.md` §REQ-ID discipline, which defines the same three triggers with their precise scoping (e.g. the closing-leaf-mapping qualifier that excludes a leaf merely re-citing an already-closed REQ-ID). The trigger text now lives in three places: that canonical clause, the `conventions.md` *Deps* leaf-format field, and plan.md step 2. The step-2 copy carries none of the canonical clause's qualifiers, so if the trigger set is ever refined the abridged step-2 list silently drifts out of agreement with the rule it claims to merely reference.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** ab8e297 — pi-loom plan: resolve "Transitive-completeness obligation invisible at leaf authoring" (2026-06-11, Thomas Andersen)
**History:** Step 2 originally ended at "Link the new leaf into the appropriate section below." Commit ab8e297 resolved a prior finding that the transitive-completeness obligation was invisible at leaf-authoring time by appending the trigger sentence to step 2 (and, in the same commit, to the `conventions.md` *Deps* field). That fix surfaced the obligation correctly but did so by restating the full trigger triple inline rather than carrying only a pointer, creating the duplicate surface this finding flags.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan.md` step 2, replace the inline trigger enumeration with a pointer-and-cue to the canonical clause, preserving the leaf-authoring visibility that commit ab8e297 added while leaving only one statement of the triggers.

Strike the sentence beginning "If the new leaf can introduce an executable REQ-ID, …" and replace it with a cue that (a) tells the author to check whether the new leaf trips the transitive-completeness obligation and (b) names the destination (`H5b`'s **Deps**) without re-listing the triggers. Suggested literal text:

> Before moving on, check the *Transitive-completeness plan-maintenance* clause in [`conventions.md`](./plan_topics/conventions.md) §REQ-ID discipline — which states the obligation and its exact triggers once — and, if the new leaf trips it, add the leaf to [`H5b`](./plan_topics/H5b-warn-only-canary.md)'s **Deps** as that clause directs.

This keeps the canonical clause as the sole trigger surface. The `conventions.md` *Deps* leaf-format field carries a third copy of the same triple (also added by ab8e297); it is the same drift class and the fixer may collapse it to a pointer in the same pass, but it is outside this finding's scope and not required to resolve step 2. `conventions.md` is otherwise read-only for this fix (the canonical clause is the pointer target, not an edit site).

## Consequence

**Severity:** cosmetic

A leaf author still reaches the canonical clause via the existing pointer, so build behaviour is unchanged today. The risk is drift: a future refinement of the trigger set in `conventions.md` leaves step 2's abridged copy stale, and an author who reads only step 2 would apply a trigger list that no longer matches the rule it cites.
