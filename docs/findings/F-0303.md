## Problem

`V18d` owns the revert path that restores the prior Pi-SDK pin before merge — a multi-step procedure that reverts step 4's pin edit, re-runs `V18c`'s static build-time gates against the restored prior pin, and widens the revert if any gate stays red (catching a co-edited operand left at its candidate value). This entire flow is described only in `V18d`'s **Adds** prose. The **Tests** and **Ships when** of both `V18d` and `V18d-T` cover only the forward runtime-evidence acceptance gate (`H4a` against the bumped pin); no test drives the simulated red-run → restore → `V18c`-green sequence and no acceptance criterion observes that the revert was performed and is complete. `V18d` can therefore be tagged complete with the revert flow — including its fragile "widen the revert" branch — entirely unverified, allowing an inconsistent post-revert state to merge.

## Issue introduction

**Verdict:** multi-commit-interaction (regression introduced while resolving a prior review finding)
**Introducing commits:**
- `8af3204` (2026-06-11) "pi-loom plan: resolve \"V18c bundles static gates with runtime-evidence acceptance gate\"" — split `V18d` out of `V18c`; the **Tests** block was authored covering only the forward runtime-evidence acceptance gate, and the revert path lived solely in **Ships when** prose, never as a Test. The Tests-level gap is present from this inception.
- `80d16ef` (2026-06-16) "pi-loom plan: resolve \"V18d Ships-when overloads single-observable gate with revert-path obligations\"" — in resolving the "overload" finding, removed the revert-verification narrative ("re-run `V18c` gates green … widen if red") from both `V18d` and `V18d-T` **Ships when**, relocating the mechanics into `V18d`'s **Adds** prose. This stripped the residual (prose-level) acceptance criterion that had partially compensated, leaving the revert path with no Test and no Ships-when gate.

**History:** The revert path's verification language oscillated in **Ships when** across `8af3204` → `eed7975` (2026-06-11, "revert path re-asserts gates return green") → `fd15386` (2026-06-11, "V18d revert path re-runs V18c gates but V18c absent from Deps") before `80d16ef` removed it. The Tests block never gated the revert at any point in the file's history (`git log --follow` / `git show <rev>:docs/plan_topics/V18d-version-bump-acceptance.md`).

## Solution approach

Add an observable acceptance criterion for the revert sequence to the `V18d` / `V18d-T` pair. Add a failing test to `V18d-T`'s **Tests** that exercises the restore → `V18c`-green re-run against the `H4a` / `V18c`-gate doubles, covering both the complete-revert (all gates green) and incomplete-revert (a co-edited operand left at its candidate value reddens at least one gate) signals. Add the paired impl-time assertion to `V18d`'s **Tests**, and surface the revert-verification criterion in `V18d`'s **Ships when** so completion is self-contained rather than buried in **Adds** prose.

## Solution constraints

- The spec is read-only for this fix — `version-bump-triggers.md` and `version-bump-intro.md` already carry the merge-blocking obligation and acceptance-gate scope; do not edit the spec.
- The static-gate definitions are owned by `V18c` / `V18c-T`; wire `V18d`'s revert test to those gates as doubles and do not redefine them in `V18d`.
- Honour the plan's test-first ritual (land `V18d-T`'s failing test before `V18d`'s impl) and cite only existing leaf IDs.
