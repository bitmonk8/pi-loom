## Finding

`host-prerequisites.md`'s clause (a) under **Host prerequisites for the degraded-state branch** records, as the spec's own text, that the session-only degraded-state branch relies on a premise that **directly contradicts** Pi's published extension lifecycle at the pinned SDK version (`@earendil-works/pi-coding-agent ~0.75.5`). The branch presupposes that on a session-only `session_shutdown` (`reason ∈ {"new","resume","fork"}`) the same extension instance — and its closed-over `LoomRegistry`, drained flag, and `drainStateTag` — survives the swap, so that a subsequent slash invocation on the swapped-in session observes the `"degraded-needs-reload"` tag the handler wrote. Pi's `docs/extensions.md` (§*Session replacement lifecycle and footguns*; §`session_shutdown`) documents the opposite: on `/new`, `/resume`, and `/fork`, Pi fires `session_shutdown`, **tears the old runtime down**, then **reloads and rebinds extensions** for the new session, then fires `session_start`. A new instance with a fresh, non-drained `LoomRegistry` would dispatch normally, silently hiding the degradation rather than surfacing it — which is the exact failure mode the `/reload`-only recovery framing depends on not occurring. CLAUDE.md's no-globals/statics/singletons rule and the contract's DI/closure routing eliminate any process-global escape hatch.

The spec marks this an *open contradiction* rather than an unpinned presupposition (it explicitly distinguishes (a) from (b)–(d) on this footing) and obligates an authoritative determination — with two and only two dispositive verdicts, `provably-differs` or `governed-by-rebind` — recorded under `version-bump-step2.md` item (a)'s recording shape. Until that record exists, every dependent obligation downstream is dead-by-construction or wrong-by-construction: the session-only degraded branch in `session-only-degraded-state.md`, the `markRuntimeDegraded` / `"degraded-needs-reload"` transition, the `loom/host/session-shutdown-runtime-degraded` emission, the post-handler operator-facing slash note, and the SM-3b session-swap semantics plus SM-4/5/6 session-only degraded-state obligations in `session-model-and-appendix.md`. The plan already mirrors this: `V9l` is marked **blocked** in `plan.md` §Blocked obligations, naming `V18c` (the version-bump-checklist leaf) as the resolution owner.

The contradiction is documented *at the current pin*, not as a hypothetical future-minor concern, and is therefore not amenable to deferral via the standard "editorial review at next Pi bump" gate the other presuppositions use. It demands an out-of-band investigation now.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f5e89f4` (Jun 4 2026, "docs: enforce 100KB hard / 30KB recommended size cap on spec set") for the original presupposition-(a) authoring carried over from the pre-split monolith; `7447d1e` (Jun 10 2026, "pi-loom plan: resolve 'Session-only degraded-state presupposition (a) contradicts Pi's documented lifecycle'") for the reclassification from "unpinned presupposition" to **open contradiction**.
**History:** Before commit `7447d1e`, clause (a) was authored on the same footing as (b)–(d) — an unpinned host-behaviour presupposition with editorial-review-at-next-bump as its gate. The `git diff` at `7447d1e` shows the rewrite: the four behaviours were re-grouped so that (b)–(d) remained presuppositions while (a) was reframed as "**open contradiction** against Pi's documented teardown-and-rebind extension lifecycle … at the [loom 1.0 Pi-SDK pin], not at some hypothetical future minor" and given the explicit `MUST be resolved before SM-4/5/6 are implemented` gate. The presupposition itself dates back further — the host-prerequisites file as a separate page was created by the `f5e89f4` corpus split on Jun 4 2026, lifted from the prior monolithic `spec_topics/pi-integration-contract.md` and earlier `spec/pi-integration-contract.md`; the per-event.reason lifecycle text it consolidated was substantively edited by `bfef26e` (Jun 5 2026, "resolve 'Per-event.reason session lifecycle asserted as fact with no probe or cited contract'") and `52b9101` (May 11 2026, "T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state"), neither of which yet recognised the contradiction. The defect is therefore the *interaction*: the spec text correctly identifies the contradiction (after `7447d1e`) and correctly gates implementation behind a recorded determination, but no determination has yet been recorded in `version-bump-step2.md` item (a) — so the gate is live and unresolved on every commit since `7447d1e`, including the current HEAD.

## Solution Space

**Shape:** multiple

### Option A — Land a `provably-differs` determination

**Approach.** Perform the authoritative investigation into whether loom's package-loaded `extensions/index.ts` (auto-discovered via `package.json#pi.extensions`, distributed via `pi install`) is governed by the same teardown-and-rebind lifecycle `docs/extensions.md` documents for `/new`/`/resume`/`/fork`, or whether the package-loaded path provably differs from it for those three reasons. Investigation surfaces: `@earendil-works/pi-coding-agent`'s extension loader source (`dist/core/extensions/loader.js` and the session-swap path that drives `session_shutdown` → reload), a real-host probe registering a marker on the extension instance and observing whether the same instance object receives the post-swap `session_start`, and the documented intent in `docs/extensions.md`. If the investigation concludes that package-loaded extensions are *not* rebound on session-only reasons (against the documented behaviour), record `provably-differs` in `version-bump-step2.md` item (a) per its authoritative-determination recording shape: cited code/doc sites, the bump-contributor as actor of record, evidence sufficient to fail under `governed-by-rebind`.
**Spec edits.** No semantic edits to `session-only-degraded-state.md` or SM-3b/4/5/6; remove the *Implementation gate* blockquote on `session-only-degraded-state.md` and the equivalent gate framings in `session-model-and-appendix.md`; rewrite the (a) clause to point at the recorded determination and drop the "open contradiction" label.
**Pros.** Preserves the entire degraded-state design as written; unblocks `V9l`/`V9l-T` without re-architecting; honours the explicit `/reload`-only recovery framing.
**Cons.** Requires real-host evidence that the spec currently does not have. If `docs/extensions.md`'s lifecycle wording is accurate for package-loaded extensions too, this option is unreachable.
**Risks.** A `provably-differs` recording that turns out to be wrong silently re-introduces the original failure mode; the determination's evidentiary standard MUST be tight.

### Option B — Land a `governed-by-rebind` determination and rework SM-3b/4/5/6

**Approach.** Record `governed-by-rebind` in `version-bump-step2.md` item (a) (the verdict the published Pi docs presently support), then rework the four SM obligations to operate on the rebind lifecycle: the closed-over `LoomRegistry`'s drained flag and `"degraded-needs-reload"` tag are not observable post-swap because the new instance starts fresh, so the operator-facing degraded note and `loom/host/session-shutdown-runtime-degraded` emission paths must either move to a different signal (e.g. a teardown-time diagnostic only, with no cross-instance survival expectation), be retired, or be re-targeted at the cases the rebind lifecycle does cover (e.g. `quit`/`reload` teardown). SM-3b session-swap semantics rewrite to describe the rebind path; SM-4/5/6 either collapse into a single "no degraded-state branch under session-only reasons; recovery is implicit via the new instance" obligation or are retired.
**Spec edits.** Rewrite `host-prerequisites.md` clause (a) to point at the `governed-by-rebind` record; rewrite or retire the bulk of `session-only-degraded-state.md` (drain-state tag write, predicate split, emission rules, recovery prohibitions); reshape `drain-state-contract.md`'s degraded tag set; remove the *Implementation gate* blockquotes; reshape SM-3b/4/5/6 in `session-model-and-appendix.md`; reshape `patch-skew-degradation.md`'s interstitial-seam swallow rule; refresh cross-references throughout PIC.
**Pros.** Aligns with Pi's actual documented behaviour at the pin; no real-host investigation required to begin work; the smallest set of contract surfaces survive (those that remain observable across the rebind).
**Cons.** Largest spec edit surface of any option; loses the explicit `/reload`-as-operator-recovery framing the current text builds on; risks under-specifying the new world (e.g. what happens to in-flight invocations across the rebind, which the current text relies on `disposeBarrier` to drain).
**Risks.** A rework done before the investigation under Option A would discard a design that may yet be re-enabled by a `provably-differs` finding — wasted work and lost design intent.

### Recommendation

Pursue Option A first as a time-boxed investigation owned by the version-bump editorial reviewer (the named owner already pinned by item (a)'s recording shape): the determination is dispositive in one commit either way, and the asymmetry of edit cost makes Option B's wholesale rework prohibitively expensive to attempt speculatively. Order: (1) record the determination in `version-bump-step2.md` item (a); (2) on `provably-differs`, perform the small spec edits Option A names and unblock `V9l`/`V9l-T`; (3) only on `governed-by-rebind` proceed to Option B's rework, treating that as a distinct multi-leaf fix. Edge cases the investigator must watch: package-loaded extensions may behave differently from in-development extensions in `pi extension install --link` mode; the `previousSessionFile` carry-through that `docs/extensions.md` documents on `session_start { reason: … }` is a strong hint that *some* state is preserved across the swap, but it is session-level state, not extension-instance state — that distinction is exactly what the determination must pin.

## Consequence

**Severity:** blocking

Until clause (a) has an authoritative determination recorded, `V9l`/`V9l-T` cannot proceed and SM-3b/4/5/6 are not implementable as written — `plan.md` already classifies `V9l` as **blocked** on this exact resolution. Worse, the larger session-only-degraded-state page reads as if its dozens of normative MUSTs were buildable; an implementer who skips the *Implementation gate* blockquote will produce code against contradicted behaviour, and the resulting tests will lock the wrong contract in. Either dispositive verdict materially reshapes the runtime: `provably-differs` unblocks the branch as-written; `governed-by-rebind` removes the degraded-state branch entirely and reworks recovery semantics around the rebind lifecycle.
