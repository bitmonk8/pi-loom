## Problem

The Error shape paragraph in `schema-subset.md` defines the observable value space of `ValidationIssue.schema_keyword` negatively against AJV ("the only `schema_keyword` value Loom emits that is not a literal AJV keyword"), so the wire enumeration that loom code branches on is pinned to whatever the bundled AJV version calls its keywords rather than to the JSON Schema Draft 2020-12 validation-keyword vocabulary the page already pins as the lowered-document dialect. AJV's `ErrorObject.keyword` includes non-draft entries (branch keywords, applicator strings, AJV extensions) and can change across a validator swap or AJV major bump without any change to the lowered schema or the depth-walk behaviour. Loom code branching on `schema_keyword` â€” including ERR-14's code-point ordering of `validation_errors` â€” therefore has no spec-supplied stable surface for any string other than `"maxDepth"`, which is the only entry the spec actually pins.
