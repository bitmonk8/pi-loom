## Finding

The `pi.getCommands()` read-failure Tests bullet in `V9k` and its test mirror `V9k-T` writes the term unhyphenated: "the handler swallows the throw and MUST NOT set drain state (drain state is owned by `V9m`'s `LoomRegistry` contract)". That is the only place in the plan corpus where the term appears without a hyphen.

Everywhere else the corpus uses the hyphenated compound `drain-state`: the `V9m` leaf is named `V9m-drain-state-contract.md` and uses "drain-state contract", "drain-state three-arm dispatch", "drain-state arms"; `V9l`, `V9b`, the coverage matrix, `docs/plan.md`, and even `V9k`'s own **Adds** bullet ("`getCommands()` read-failure pending-list drop with no drain-state write") all hyphenate it. The two unhyphenated instances in the getCommands bullet are the lone deviation, and they sit on the one line that disclaims the no-write obligation pointing at `V9m`.

## Issue introduction

**Verdict:** present-since-inception

**Introducing commit:** `6df646a` — "pi-loom plan: resolve \"extension-bootstrap SDK-failure rule has no closing leaf\"" (2026-06-12)

**History:** `V9k-extension-bootstrap-failures.md` and `V9k-T-extension-bootstrap-failures.md` were both created in `6df646a`; `git log --follow` shows no earlier revision for either file. `git show 6df646a:docs/plan_topics/V9k-extension-bootstrap-failures.md` shows the getCommands bullet already carried the unhyphenated "MUST NOT set drain state (drain state is owned by ...)" wording at creation (then pointing at `V9b`'s `LoomRegistry` contract). The later `f264d3a` ("resolve \"V9k getCommands read-failure handler missing V9m drain-state dependency\"") touched the same line but only retargeted the parenthetical owner from `V9b` to `V9m`, preserving both unhyphenated forms. `git log -S 'set drain state'` on both files returns only `6df646a`, confirming the deviation entered with the leaves themselves and no later edit introduced or removed it. The plan corpus is git-tracked.

## Solution Space

**Shape:** single

### Recommendation

In the `pi.getCommands()` read-failure Tests bullet of both `docs/plan_topics/V9k-extension-bootstrap-failures.md` and `docs/plan_topics/V9k-T-extension-bootstrap-failures.md`, change the substring `MUST NOT set drain state (drain state is owned by` to `MUST NOT set drain-state (drain-state is owned by`, so both occurrences match the hyphenated `drain-state` used corpus-wide (including the same leaf's **Adds** "drain-state write" and the `V9m` drain-state contract).

The spec is read-only for this fix — the canonical term already lives in `../spec_topics/pi-integration-contract/drain-state-contract.md`; do not touch any spec file. No leaf-ID, **Deps**, or **Adds** edits are required; this is a pure wording alignment on the two existing bullets.

## Consequence

**Severity:** cosmetic

A reader or implementer grepping the canonical token `drain-state` to trace the read-failure no-write obligation (which explicitly defers ownership to `V9m`) misses these two instances. No behaviour diverges; the cost is a terminology snag on the very line that cross-references the `V9m` `LoomRegistry` contract.
