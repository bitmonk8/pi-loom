## Finding

The closing paragraph of `version-bump-triggers.md` makes output (c) a merge-blocking landing gate: "green end-to-end runtime evidence at the new pin: a representative integrated `.loom` exercised through the loom's end-to-end harness against the bumped SDK … not the build-time surface-inventory assertions of output (a) alone," reinforced by the normative "a bump whose runtime-evidence run is red MUST NOT be merged at the candidate pin." The gate therefore depends on two artefacts existing before any bump can be validated: (1) an *end-to-end harness* capable of driving a loom against the bumped SDK, and (2) a *representative integrated `.loom`* that exercises the six named surfaces (typed query, tool-call loop, `invoke(...)`, schema validation, the binder, cancellation).

Neither artefact is established anywhere in the spec corpus. A grep across `docs/spec_topics/` finds the phrases "end-to-end harness" and "representative integrated `.loom`" only in this one sentence; no spec section requires the harness or the fixture to be built, defines their coverage, or marks them as a presupposed prerequisite. By contrast, `host-prerequisites.md` explicitly frames comparable dependencies as *presuppositions* ("loom 1.0 treats it as a **presupposition** …"). Output (c) does not. A reader implementing the version-bump procedure from the spec alone encounters a normative MUST whose satisfaction depends on infrastructure the spec never requires anyone to produce and never points at an owner for.

The defect is not that the harness/fixture do not exist — the plan builds them (the `H4a` harness, the `H7a` representative multi-feature fixture `.loom`, and the `V18d` acceptance gate). It is that the spec text silently assumes them: the establish-before-use discipline the rest of the corpus follows is skipped here, leaving the gate's dependency implicit.

**History:** The defect is co-extensive with output (c) itself. `git log -S "representative integrated"` and `git log -S "end-to-end harness"` over `version-bump-triggers.md` and `spec.md` both return `81ab342` as their only commit, and `git show 81ab342^:…/version-bump-triggers.md` contains zero occurrences of either phrase — the parent's outputs paragraph listed only (a) the green build-time test run and (b) the single bump commit, with no runtime-evidence arm at all. Commit `81ab342` added output (c) (the diff replaces the two-arm `(a)`/`(b)` sentence with the three-arm version) in the same change that introduced the `V18d` plan leaf and the `coverage-matrix.md` `cka-19` row that own the harness-backed gate. The spec sentence was thus authored presupposing the harness and representative `.loom`, while the obligation to build them was recorded only on the plan side; the spec text never gained a corresponding presupposition or owner reference. The defect has been present continuously since output (c) was written.

## Solution Space

**Shape:** single

### Recommendation

In output (c), make the dependency explicit: state that the runtime-evidence gate **presupposes** an existing end-to-end harness and a representative integrated `.loom` covering the six listed surfaces (typed query, tool-call loop, `invoke(...)`, schema validation, the binder, cancellation), using the same "presupposition" framing `host-prerequisites.md` already applies to comparable prerequisites. Note that the harness and fixture are implementation-owned artefacts the version-bump procedure consumes rather than authors — paralleling the existing §Non-goals (a) distinction in `version-bump-intro.md`, which already separates the in-scope acceptance gate from the out-of-scope recovery behaviour.

Keep the spec edit to declaring the presupposition and its consume-not-author posture; do not designate a concrete runnable command, name a specific fixture file, or define "green" as an exit status — those are test-infrastructure choices owned by the implementation/plan (`H4a`/`H7a`/`V18d`), and pinning them in the behaviour spec would overreach. The one-sentence presupposition makes the gate's dependency self-evident without the spec specifying infrastructure it does not own.

## Consequence

**Severity:** advisory

An implementer reading the version-bump procedure from the spec alone sees a merge-blocking MUST whose precondition (a runnable harness plus a representative `.loom` covering six surfaces) is never required or located by the spec, so the gate reads as un-actionable until the reader infers, out-of-band, that the harness and fixture are separate deliverables. Implementers can still produce a working system (the plan demonstrably does), so the gap is a guidance defect rather than a correctness or blocking one.
