## Finding

`V9k` closes the *Extension-bootstrap SDK failures* granularity rule by mapping each registration call in steps 1–5 to its own observable behaviour. The leaf's Tests section carries five bullets, each a distinct control-flow circuit with no shared assertion surface:

- `pi.registerMessageRenderer` failure — degrade-and-proceed (renderer dropped, factory completes, system notes permanently fall back to `ctx.ui.notify`).
- `pi.registerCommand` failure — per-loom partial drop (only the failing loom drops, siblings register).
- `pi.registerFlag` failure — whole-extension abort (steps 2–5 do not execute).
- `pi.on(...)` failure — whole-extension abort (remaining steps do not execute).
- `session_start`-time `pi.getCommands()` read failure — pending-registration-list drop with the throw swallowed and, distinctly from the four write-side surfaces, *no* drain-state write (drain state owned by `V9m`).

The five surfaces span three behaviour classes (two fatal/abort, two partial-drop, one degrade) plus a read-side no-write obligation that is explicitly contrasted against the four write-side surfaces. A reviewer must hold all five circuits in one pass, and the single compound Ships-when ("`npm test` exercises each of the five bootstrap-failure surfaces") does not localise which surface a regression touches.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `6df646a` (2026-06-12) — "pi-loom plan: resolve \"extension-bootstrap SDK-failure rule has no closing leaf\""
**History:** `V9k` was created in `6df646a` to give the *Extension-bootstrap SDK failures* rule a closing leaf, and it already carried all five test bullets (including the `getCommands` read-failure surface) at that first commit. A later systemic over-bundling sweep, `cb8737a` ("resolve \"Systemic leaf over-bundling across the plan corpus\""), edited the file but only retargeted the drain-state owner reference and did not split it; `f264d3a` likewise only adjusted the `V9m` drain-state dependency. The five-surface bundling has therefore been present since the leaf's inception and survived a prior over-bundling pass.

## Solution Space

**Shape:** multiple

The defect is the number of independent behaviours under one leaf; the decision is the split boundary. Two viable decompositions exist with a material trade-off. New leaf IDs are left as `<new>` placeholders for the implementer to allocate per the plan's `no-invented-ids` discipline; the `V9k` slug rename (if the surviving half no longer matches `extension-bootstrap-failures`) is at the implementer's discretion. The split is internal to the plan: do not touch `docs/spec.md` or the spec topics, and the single `loom/load/extension-bootstrap-failed` code and its un-anchored coverage-matrix obligation must remain closed by the union of the resulting leaves.

### Option A — two-way split (abort vs non-abort)

**Approach:** Keep one leaf for the two whole-extension abort surfaces (`registerFlag`, `pi.on`) and move the three non-abort surfaces (`registerMessageRenderer` degrade, `registerCommand` partial-drop, `getCommands` read-failure partial-drop) to a second leaf. This is the grouping the source finding proposes.

**Plan edits:** Partition the five Tests bullets across `V9k` and one `<new>` leaf (plus their `-T` partners); split the compound Ships-when so each leaf names only its own surfaces; update the `coverage-matrix.md` line-88 closing-leaf cell to list both leaf IDs; update the `docs/plan.md` §V9 listing.

**Spec edits:** None.

**Pros:** Smallest diff; one new leaf pair. Groups all three pending-registration-list / degrade behaviours (including the `getCommands` read-side no-write obligation contrasted against `registerCommand`) in one place where the write-vs-read distinction is reviewed together.

**Cons:** The non-abort leaf still carries three surfaces across two behaviour classes (degrade + partial-drop), so its review surface is larger than the abort leaf's.

**Risks:** Low — no downstream `Deps` reference `V9k`.

### Option B — three-way split (by behaviour class)

**Approach:** One leaf per behaviour class: fatal/abort (`registerFlag`, `pi.on`); partial-drop (`registerCommand`, plus the `getCommands` read-failure no-write surface); degrade (`registerMessageRenderer`).

**Plan edits:** Partition the five bullets across `V9k` and two `<new>` leaves (plus `-T` partners); split Ships-when per leaf; update the `coverage-matrix.md` row to list all three leaf IDs; update the `docs/plan.md` §V9 listing.

**Spec edits:** None.

**Pros:** Each leaf is single-behaviour-class; smallest per-leaf review surface; the read-side no-drain-state obligation sits beside the one write-side partial-drop it is contrasted with.

**Cons:** Two new leaf pairs (four files) and a three-way coverage-matrix cell; more plan churn for an advisory-severity issue.

**Risks:** Low.

### Recommendation

Take **Option A**. It removes the abort/non-abort review-coupling — the largest cognitive jump in the leaf — with one new leaf pair, and is the lower-churn fix proportionate to an advisory finding. Resolve the leaf-structural split before relabelling the per-event `pi.on` bullet (finding 131), so that ambiguity fix lands on the already-partitioned baseline. Edge case for the implementer: the `getCommands` read-failure bullet's "MUST NOT set drain state — drain state owned by `V9m`" contrast against the write-side surfaces must survive the split intact; keep it co-located with the `registerCommand` write-side bullet so the read-vs-write distinction stays legible.

## Consequence

**Severity:** advisory

A single reviewer must verify five unrelated control-flow circuits in one pass; the read-side no-drain-state obligation (contrasted against four write-side surfaces) is the easiest to under-review when stacked with the others, and the compound Ships-when does not pinpoint which surface a failing run implicates. The leaf still produces a working implementation, so this is review-burden / organisational, not a correctness gap.
