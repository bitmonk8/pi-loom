## Problem

§7's *Parse-time literal-value* sub-rule for `<value>` renders the scalar per category 5's `<key>` rule (bare when identifier-shaped, double-quoted otherwise) "except that YAML scalars with no enclosing source quoting (e.g. `mode:` values) render unquoted regardless of identifier shape." "Enclosing source quoting" is undefined: the carve-out keys on the source-text YAML scalar style, but never says which of YAML's five scalar styles (plain, single-quoted, double-quoted, literal-block, folded-block) count, and the post-parse string handed to the renderer has lost the source style. Separately, §7's *Test vectors* carry no vector for an unquoted YAML scalar that fails the identifier-shape predicate — the case the carve-out exists to handle (e.g. `mode: foo-bar`). Together these leave the most common malformed-`mode:` renderings implementation-defined, contradicting §7's byte-identical framing and leaving the DIAG-4 contract unverifiable for the `unknown-mode-value`, `unknown-methodology-value`, `unknown-bind-context-value`, and `model-unresolved` rows.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** f930818 — pi-loom spec: resolve "placeholder rendering exemption open-ended" (2026-05-07, Thomas Andersen)
**History:** `f930818` introduced the *Parse-time literal-value* bullet in `spec_topics/diagnostics.md` with the exact "no enclosing source quoting (e.g. `mode:` values)" wording and only the `unknown-mode-value` / `duplicate-enum-value` / `duplicate-discriminator-value` row enumeration, with no test vector for the non-identifier-shaped unquoted case. A later commit `5f5c487` (2026-06-03) extended the row enumeration to the `unknown-methodology-value` / `unknown-bind-context-value` / `model-unresolved` rows, widening the rule's blast radius without addressing either ambiguity; `f5e89f4` (2026-06-04) split `diagnostics.md` into `placeholder-rendering-b.md` and carried the defective clause across verbatim. Both defects were present in the bullet from its first commit.

## Solution approach

Rewrite the trailing exception of §7's *Parse-time literal-value* sub-rule so it names the YAML scalar node styles that gate the carve-out, replacing the undefined "no enclosing source quoting" phrasing — the renderer must be told which YAML node style observed at parse time triggers the unquoted rendering. Add a test vector to §7's *Test vectors* exercising a plain unquoted non-identifier-shaped scalar (e.g. `mode: foo-bar`) alongside its quoted counterpart, pinning byte-identical renderings for the affected parse-time rows.
