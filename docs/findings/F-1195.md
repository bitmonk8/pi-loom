## Finding

§7's *Parse-time literal-value* sub-rule for `<value>` says the scalar renders per category 5's `<key>` rule (bare when identifier-shaped, double-quoted otherwise) "except that YAML scalars with no enclosing source quoting (e.g. `mode:` values) render unquoted regardless of identifier shape." Two defects sit in that one clause:

1. **"Enclosing source quoting" is an undefined term.** Category 5's `<key>` predicate is a runtime regex check on the post-parse string value (the spec spells this out: *"The identifier-shape predicate is a runtime check on the key string, not a parse-time grammar production"*). The §7 carve-out, by contrast, looks at the *source-text* YAML scalar style — but it never says which YAML scalar styles count as "no enclosing source quoting." YAML 1.2 admits five flow scalar styles: plain (unquoted), single-quoted, double-quoted, folded-block (`>`), and literal-block (`|`). The intent of the `mode:` example is clearly "plain scalar," but two reasonable implementers can disagree on the block scalar styles, and the post-parse string handed to the renderer has lost the source style — the renderer needs to be told explicitly which YAML node-style observed at parse time gates the carve-out, and the rule needs to name how that style flag is plumbed from the YAML parser to the diagnostic site.

2. **No test vector covers the non-identifier unquoted scalar.** §7 carries one test vector ladder for `<value>` (the reload-teardown / discovery-slow / cross-source-shadow trio), none of which exercise an unquoted YAML scalar that fails the identifier-shape predicate. Without a vector, the most material case the carve-out exists to handle — an unquoted scalar like `mode: foo-bar` (`foo-bar` is plain, unquoted, non-identifier-shaped) — has no normative byte-identical rendering anchored anywhere. DIAG-4's byte-identical contract is therefore unverifiable for the `loom/load/unknown-mode-value`, `loom/load/unknown-methodology-value`, `loom/load/unknown-bind-context-value`, and `loom/load/model-unresolved` rows whenever the offending value is a plain non-identifier scalar.

Both defects together leave the rendering of the most common malformed-`mode:` shapes implementation-defined, which directly contradicts §7's framing as a byte-identical rendering surface.

## Solution Space

**Shape:** single

### Recommendation

Tighten the §7 *Parse-time literal-value* clause to name YAML node styles directly and add the missing test vector.

**Spec edits in `docs/spec_topics/diagnostics/placeholder-rendering-b.md` §7:**

1. Replace the trailing exception of the *Parse-time literal-value* bullet with a precise node-style predicate:

   > … rendered as the literal source text verbatim per category 5's `<key>` (bare-when-identifier-shaped, double-quoted otherwise) — **except that values parsed from a YAML *plain* scalar (no enclosing `'…'`, `"…"`, `|`, or `>` indicator at the YAML node) render unquoted regardless of identifier shape. The YAML scalar style is the node-style flag the `yaml` parser reports at parse time (per [implementation-notes.md](../implementation-notes.md) §"Loom-package implementation dependencies"); the frontmatter parser MUST plumb that flag through to the diagnostic site so the renderer sees the original node style, not just the post-parse string. Single-quoted, double-quoted, literal-block, and folded-block scalars all fall back to category 5's `<key>` rule (bare-when-identifier-shaped, double-quoted otherwise).**

2. Add a test vector to §7 *Test vectors* covering the non-identifier plain case:

   > - A `loom/load/unknown-mode-value` for the frontmatter line `mode: foo-bar` (plain unquoted YAML scalar; not identifier-shaped) renders `unknown 'mode:' value 'foo-bar'; expected 'prompt' or 'subagent'` — the plain scalar's `<value>` is rendered unquoted by the carve-out, and the surrounding single quotes come from the registry template.
   > - The same field re-authored as `mode: "foo-bar"` (double-quoted YAML scalar) renders `unknown 'mode:' value '"foo-bar"'; expected 'prompt' or 'subagent'` — the quoted scalar falls back to category 5's `<key>` rule, which double-quotes a non-identifier-shaped string.

**Edge-case watchlist for the implementer:**

- The non-identifier plain-vs-quoted distinction is only observable when the parser exposes the YAML node style. The frontmatter parser already runs via the `yaml` dependency (see `V6a`); the seam to surface node-style is small but must be wired before `V7c` can assert the new vector.
- Block scalars (`|` / `>`) are uncommon in loom frontmatter but legal; the proposed rule pins them to the fallback (double-quoted-when-non-identifier) rather than treating them as plain.
- The four affected rows (`unknown-mode-value`, `unknown-methodology-value`, `unknown-bind-context-value`, `model-unresolved`) all consume the same parser path; one node-style plumbing change covers all four.

## Consequence

**Severity:** correctness

Two implementers of the renderer can produce divergent byte sequences for the most likely real-world malformed-`mode:` shapes (e.g. `mode: foo-bar` vs `mode: "foo-bar"`) and both can claim conformance, because "enclosing source quoting" admits multiple readings (plain only? plain + block? does flow-mapping context matter?) and no test vector pins the non-identifier plain case. The downstream V18 byte-identical conformance gate cannot mechanically enforce a rule it cannot disambiguate.
