## Finding

`H4b`'s second Tests bullet (the per-category functional-effect self-check) asserts, for injection category (b): "a scripted `isError: true` tool-result surfaces as an error tool-result in the transcript, **and a mixed-success parallel `tool_use` batch lowers each sibling call's outcome independently**." The first half names a concrete observable (an error tool-result in the transcript); the second half does not — it restates the mechanism ("lowers … independently") without naming what the self-check must observe.

"Lower" is established plan vocabulary (the compiler-style sense of converting a higher representation to a lower one), so the term itself is not undefined. The concrete observable it denotes here is, however, spelled out only in the consuming vector leaf `V14b`, whose Tests bullet reads: "each sibling's outcome is lowered independently — the failing sibling becomes that `tool_use` block's `isError: true` tool-result fed back alongside the successful siblings' results." `H4b`'s Adds also ties category (b) to that vector ("a mixed-success parallel `tool_use` batch (the `V14b` tool-calls vector)"). The `H4b` self-check bullet is therefore anchored, but only by cross-reference: read in isolation it is terser than its sibling clauses (a) and (c)–(e), each of which names the observable the scripted input must yield. The fix is to state category (b)'s observable inline, matching `V14b`'s authoritative phrasing.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H4b-response-programming-surface.md`, in the second Tests bullet, rewrite category (b)'s second clause so it names the observable inline instead of restating the mechanism. Replace `a mixed-success parallel `tool_use` batch lowers each sibling call's outcome independently` with a clause that states what the transcript must show — for example: `a mixed-success parallel `tool_use` batch surfaces each sibling call's outcome as its own `tool_use` tool-result in the transcript, the failing sibling as an `isError: true` result fed back alongside the successful siblings' results`. This mirrors the observable `V14b`'s Tests bullet already pins ("the failing sibling becomes that `tool_use` block's `isError: true` tool-result fed back alongside the successful siblings' results"), so the owning self-check and its consuming vector assert the same observable. The Adds field's existing "(the `V14b` tool-calls vector)" cross-reference stays as-is.

## Consequence

**Severity:** advisory

An implementer writing `H4b`'s self-check for category (b) reading the Tests bullet alone sees only "lowers each sibling call's outcome independently" and must consult `V14b` (cross-referenced from Adds) to recover the concrete observable — that a mixed-success batch yields each sibling's own `tool_use` result, the failing sibling as an `isError: true` result alongside the successful siblings' results. The observable is recoverable, so this degrades the self-check bullet's self-containment rather than blocking the leaf or risking two implementers diverging.
