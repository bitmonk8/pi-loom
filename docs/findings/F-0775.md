## Finding

The best-effort emission contract in `docs/spec_topics/pi-integration-contract/runtime-event-channel.md` is built on a flat behavioural assertion about Pi's `pi.sendMessage` surface: it "returns `void` (synchronous); the runtime MUST NOT `await` it and MUST NOT attach a `.catch` handler", and the always-log fallback chain ("`ctx.ui.notify` → `loom/runtime/system-note-delivery-failed` → `console.error`") catches *synchronous* throws only. The same assumption is mirrored at `docs/spec_topics/diagnostics/diagnostic-shape.md` ("`pi.sendMessage` returns `void`; only synchronous throws are observable"), at `docs/spec_topics/diagnostics/code-registry-runtime.md`'s `loom/runtime/system-note-delivery-failed` row ("the surface returns `void`, so there is no rejection path"), and is consumed by the exactly-once emission guarantee for the always-log set.

The corresponding Pi surface lives on `ExtensionAPI.sendMessage` at `dist/core/extensions/types.d.ts`, declared as `void`-returning at the loom 1.0 Pi-SDK pin (`@earendil-works/pi-coding-agent@~0.75.5`). A silent Pi-minor change that retypes `sendMessage` to `Promise<void>` — or that keeps the `void` declaration but begins surfacing deferred-delivery failures via an asynchronous rejection or a separate channel — would not be caught by anything in the spec corpus: the build-time SDK surface-inventory literal-read for capability 6 (`sdk-cap-custom-message-renderer`) checks only that `pi.sendMessage` is present, and *none* of the SHOULD-level editorial-review items `(f)`–`(ak)` in `version-bump-step2.md` re-validate `sendMessage`'s return type, the absence of a deferred-delivery channel, or the synchronous-only failure surface. The same page already routes analogous Pi-behavioural presuppositions through that checklist (e.g. `(g)`/`(j)` for slash-handler promise lifecycle, `(m)` for `AssistantMessage` fields, `(n)` for `AgentSession` member surface, `(r)` for `convertToLlm`, `(u)` for `complete()`), so the omission is asymmetric: every other behavioural premise the runtime-event-channel page builds on has a named editorial-review re-validation item, except this one.

The consequence under a future Pi minor that makes `sendMessage` asynchronous: a deferred delivery failure would never enter the fallback chain (the runtime is explicitly forbidden from attaching `.catch`), the `loom/runtime/system-note-delivery-failed` diagnostic would never fire, and the always-log exactly-once guarantee for both group-A `RuntimeEvent`s and group-B panic diagnostics would silently degrade — with no in-band signal to the operator and no surface for the conformance test to red on.

## Issue introduction

**Verdict:** present-since-inception (with later structural reinforcement)

**Introducing commits:**

- `11b9199` (2026-05-04, "pi-loom spec: resolve \"pi.sendMessage failure leaves system notes silently dropped\"") — introduced the best-effort fallback chain itself; the original text framed the fallback as triggering on "any thrown or rejected value", so synchronous-only-ness was not yet asserted but was also not pinned.
- `29c29b2` (2026-05-06, "pi-loom spec: resolve \"pi.sendMessage is synchronous (void); spec treats it as returning a Promise\"") — added the synchronous-only assertion verbatim ("`pi.sendMessage` returns `void` (synchronous); the runtime MUST NOT `await` it and MUST NOT attach a `.catch` handler. The best-effort fallback below covers synchronous throws only.") and rewrote the fallback to drop the "rejected" arm. This is the commit that fixed `sendMessage`'s failure surface as synchronous-only without routing the behavioural premise to a re-validation gate.
- `257c545` (2026-06-05, "pi-loom spec: resolve \"Host presuppositions lack version-bump-procedure re-audit hooks\"") — built the editorial-review checklist machinery that hosts items `(f)`–`(ak)`; multiple sibling presuppositions on the same page acquired re-validation hooks here, but the `sendMessage` synchronous-failure-surface assumption was not added to the checklist in this pass and has remained unhooked since.

**History:** The defect is *present-since-inception* in the structural sense — the spec has always taken Pi's synchronous-throw failure surface as a flat fact — and `29c29b2` is the commit that crystallised it as a verbatim assertion. The structural omission (no re-validation hook) became reviewable only after `257c545` established the checklist pattern that should have caught it; subsequent re-audit passes added items `(f)`–`(ak)` to that checklist but did not include this presupposition, so the gap is a consistent omission across the re-audit campaign rather than a single-commit regression. The fix therefore targets the omission, not any specific commit.

## Solution Space

**Shape:** single

### Recommendation

Add a new SHOULD-level item to the editorial-review checklist in `docs/spec_topics/pi-integration-contract/version-bump-step2.md` (e.g. `(al)`, slotted after `(ak)`) that re-validates `pi.sendMessage`'s failure-surface contract against each candidate Pi minor. The item MUST:

1. Confirm against the candidate `@earendil-works/pi-coding-agent` minor that `ExtensionAPI.sendMessage` declared at `dist/core/extensions/types.d.ts` still has return type `void` (not `Promise<void>` or any other type that would create an async rejection path).
2. Confirm that the candidate minor's `sendMessage` implementation in `dist/core/agent-session.js` / `dist/core/extensions/loader.js` (read against the `Baseline acquisition` paragraph's extracted `package/dist/` tree, on the same footing as items `(r)` and `(aj)`) still surfaces all delivery failures as *synchronous* throws — i.e. there is no deferred-rejection arm, no separate failure-callback channel, and no buffered delivery queue whose drop would not throw at the call site.
3. Carry the standard SHOULD-level escalation clause ("escalates to a build-time pin once Pi exposes a typed `sendMessage` return-type / failure-surface contract that step 2(a)'s surface-inventory probe can mechanically verify") so it parallels every sibling item from `(f)` onward.

Co-edit the cited paragraph in `runtime-event-channel.md` to cross-reference the new item by anchor (e.g. `bump-checklist-sendmessage-sync-failure-surface`), parallel to how the `convertToLlm` paragraph already cross-references item `(r)`. The mirrored assertions at `docs/spec_topics/diagnostics/diagnostic-shape.md` and `docs/spec_topics/diagnostics/code-registry-runtime.md` need no edit beyond ambient consistency — the re-validation pointer at the normative owner is sufficient.

Edge cases for the implementer:

- A Pi minor that keeps `void` on the typed surface but begins surfacing deferred-delivery failures via a side channel (e.g. a new `ExtensionAPI.onSendMessageError` event) MUST be caught — phrase the item so it audits the *failure-surface* contract, not just the return-type declaration.
- The new item is SHOULD-level on the same footing as `(f)`–`(ak)`; a recorded `fail` does not block the bump per `V18c`'s "Ships when" gate. That is the right posture: the audit's purpose is to surface the divergence on the bump commit so the spec can be amended, not to gate the merge.
- The new item must also be added to the `V18c-version-bump-checklist.md` plan leaf's enumeration so the leaf's "Ships when" sentence's `(a)`–`(ak)` range extends to cover it; the range is currently `(a)`–`(ak)` and must become `(a)`–`(al)` in the same edit.

## Consequence

**Severity:** correctness

A future Pi minor that retypes `pi.sendMessage` to `Promise<void>` (or otherwise surfaces deferred delivery failures asynchronously) would silently break the always-log exactly-once guarantee for both group-A `RuntimeEvent`s and group-B panic diagnostics: rejections would land unhandled, the `ctx.ui.notify` / `loom/runtime/system-note-delivery-failed` fallback would never fire, and operators would lose the only observable signal that a system-note delivery failed. No build-time SDK surface-inventory probe and no current editorial-review item catches the drift, so the regression is invisible until a real failure is dropped in production.
