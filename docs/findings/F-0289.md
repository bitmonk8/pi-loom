## Finding

`V13a` (`docs/plan_topics/V13a-query-render.md`) enumerates seven distinct behaviors across its three `Tests` bullets: the eight-vector trim→dedent order, the `loom/parse/illegal-template-escape` and `loom/parse/unterminated-template` diagnostics, the per-loom-type stringify table, the `loom/parse/interpolated-result` rejection on a `Result`-valued interpolation, the `loom/parse/discarded-query-result` parse error on a bare expression-statement, the runtime discard-observability event on an explicit `Err` discard, and the degenerate-template short-circuit to `ValidationError{empty_template, attempts:0}`.

The leaf's `Ships when` reads only "`npm test` renders the eight dedent vectors and short-circuits the empty template." It names just two of those seven behaviors. The stringify table, all four parse diagnostics (`illegal-template-escape`, `unterminated-template`, `interpolated-result`, `discarded-query-result`), and the runtime discard-observability event have no anchor in the gate prose.

The paired test leaf `V13a-T` (`docs/plan_topics/V13a-T-query-render.md`) does carry the full set as red-first obligations, so the behaviors are not unowned; the gap is that V13a's acceptance gate is narrower than the surface its own `Tests` block asserts. A reader using `Ships when` as the done-checklist would consider the leaf complete after the two named behaviors pass, with the remaining five resting only on the implementer also honouring the `Tests` block.

The parity gap has been present since the leaf was created.

## Solution Space

**Shape:** single

### Recommendation

Extend the `Ships when` field of `docs/plan_topics/V13a-query-render.md` so the gate prose covers the same behaviors its `Tests` block asserts. Beyond the already-named eight dedent vectors and empty-template short-circuit, the gate should also assert that `npm test` exercises the per-loom-type stringify table, the four parse diagnostics (`loom/parse/illegal-template-escape`, `loom/parse/unterminated-template`, `loom/parse/interpolated-result`, `loom/parse/discarded-query-result`), and the runtime discard-observability event on an explicit `Err` discard. Pull the diagnostic codes verbatim from the existing `Tests` bullets so the gate and the test driver stay in lockstep.

Edge case for the implementer: if finding 141 (split V13a into render + stringify leaves) is actioned first, the stringify-table and discard surface migrate to the new stringify leaf, and this gate text should follow the behaviors into whichever leaf owns them rather than naming surfaces V13a no longer implements.

## Consequence

**Severity:** advisory

The `Ships when` gate undercounts the leaf's own asserted behaviors, so an implementer treating it as the acceptance checklist could mark V13a done with only the dedent vectors and empty-template short-circuit verified, leaving the stringify table, four parse diagnostics, and discard-observability event unconfirmed at the gate. The red-first `V13a-T` driver still holds those tests, so a divergence is unlikely rather than guaranteed — the defect is a prose-level parity gap, not an unowned behavior.
