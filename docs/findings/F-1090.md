## Finding

V4c and V4f both model a completed/subagent callee on the infrastructure `H4a` provides, but they name that infrastructure differently. The `ERR-12` Tests bullet in `V4c-T-terminal-outcomes.md` and `V4c-terminal-outcomes.md` reads "the `H4a` harness modelling a subagent-mode callee", while the `ERR-13` Tests bullet in `V4f-no-rollback.md` and `V4f-T-no-rollback.md` reads "modelled through the `H4a` session double".

`H4a` itself is explicit that these are two distinct deliverables, not synonyms: it "establishes a reusable end-to-end test harness that loads the extension against an in-process Pi session double". The harness is the loader/dispatch driver; the **session double** is the in-process Pi mock that reproduces host behaviour and carries the scripting points a modelled callee uses (V4f even cites one — "via its completed-invoke-child scripting point"). The component that *models a callee* is therefore the session double, so V4c's "harness modelling a … callee" attaches the modelling role to the wrong `H4a` component.

The original finding also proposed adding a glossary ("terms.md") entry for "harness". No `terms.md` / glossary file exists in this plan corpus, so that arm is not actionable and is not part of the fix; `H4a` already states the harness-vs-double distinction inline.

Commit 098ec80 then split the `ERR-13` no-rollback proof out into the new `V4f` leaf, carrying "session double" with it and leaving "harness" behind in V4c's `ERR-12`, which turned the intra-leaf dual naming into the present cross-leaf inconsistency.

## Solution Space

**Shape:** single

### Recommendation

Standardise V4c on the `H4a` session double, matching V4f and `H4a`'s own wording. In the `ERR-12` Tests bullet of both `docs/plan_topics/V4c-T-terminal-outcomes.md` and `docs/plan_topics/V4c-terminal-outcomes.md`, replace `the \`H4a\` harness modelling a subagent-mode callee` with `the \`H4a\` session double modelling a subagent-mode callee`. The surrounding clauses (the "not the live `V9i` surface" / delegated-live-carrier text) stay as-is. No `Deps.` change and no spec edit; the spec topics are read-only for this fix. Do not add a glossary/`terms.md` entry — none exists in this corpus and `H4a`'s Adds already states the harness-vs-double distinction.

## Consequence

**Severity:** advisory

A reader of V4c's `ERR-12` bullet could believe `H4a` exposes a separate "harness" facility for callee modelling distinct from the session double. In practice both V4c and V4f resolve to the same `H4a` deliverables and the modelling/scripting capability lives only on the double, so a test author converges on the double regardless; the cost is a guidance/terminology gap rather than a divergent implementation.
