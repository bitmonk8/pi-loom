## Finding

`active-invocation-registry.md` specifies that all three insertion sites populate the entry's `loom` field via the **same canonical-name resolver**, which *reads* the callee's final-derived slash-name from the resolved catalog entry — "the slash-name field the [Invocation — Static resolution] cache entry and the [Parameters and Frontmatter — `tools`] snapshot entry **already carry** on those entries". For slash-registered looms this grounds out cleanly against the `LoomRegistry` `Map<slashName, parsedLoom>` key. For the two callee-only arms it does not: neither cited owner declares such a field.

- `invocation.md` §Static resolution defines the per-load-pass parse cache as holding the callee's parsed/lowered source (for argument-type checking, return-type inference, cycle detection) and its aggregated diagnostics. No slash-name / canonical-key field is declared on a cache entry.
- `frontmatter-fields-a.md` §`tools` defines the resolution snapshot's entry **name** as the in-loom callable name — the basename with hyphens replaced by underscores (`./code-review.loom` → `code_review`), or the `as` override. That is *not* the slash-name the registry needs: the Discovery final-derived form takes the stem **verbatim** (hyphens preserved, no leading `/`). The only name the snapshot declares is the underscore-remapped callable name.

So the registry asserts a field exists on two entry shapes whose owning pages either declare no such field (the parse cache) or declare a *different* name under the only name field they do carry (the `tools:` snapshot). Compounding the gap, the resolver pins a mechanism — "the `loom` field is **read** from the resolved catalog entry's slash-name field rather than constructed at the insertion site" — even though it concedes re-derivation "would produce the identical canonical form". The only observable contract is that all three arms set `details.event.loom` to the same final-derived canonical form; the read-vs-construct choice is not observable.

## Solution Space

### Recommendation

In `active-invocation-registry.md`, restate the `loom`-field resolver as a behavioural contract. Replace the "both of which carry the same canonical-key field" / "already carry on those entries" / "read … rather than constructed at the insertion site" assertions with a single normative obligation: all three insertion arms set `details.event.loom` to the same final-derived canonical form per Discovery. Each arm obtains that form from the resolved callee — the `LoomRegistry` `Map<slashName, parsedLoom>` key for slash-registered looms, and the Discovery final-derived form of the resolved callee for callee-only `.loom` files — rather than reading a presupposed slash-name field on the static-resolution cache entry or the `tools:` resolution snapshot entry.

Move the read-vs-construct preference into a non-normative implementation note; do not pin it as normative. That note must require every arm to apply the identical Discovery final-derived rule (post-`pi.looms` mapping, stem verbatim, no leading `/`) and must forbid arm-specific re-derivation that diverges on `pi.looms` mapping order, so two conforming implementations cannot converge on different `details.event.loom` values and split the operator dedup buckets the registry exists to keep aligned.

Make no edits to `invocation.md` or `frontmatter-fields-a.md` — neither owner page needs to declare a new slash-name field.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the `details.event.loom` wire value for the callee-only arms: one looks for the asserted slash-name field and, finding only the `tools:` snapshot's underscore-remapped callable name, emits that (`code_review`); another applies the Discovery final-derived rule and emits the verbatim stem (`code-review`). The registry exists to keep `details.event.loom` identical across all three arms so operators can dedup on it; an undeclared/mis-identified source field splits those dedup buckets between conforming implementations — exactly the failure the resolver was written to prevent.
