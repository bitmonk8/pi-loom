## Finding

DISC-5 (`docs/spec_topics/discovery/package-and-settings.md` §"Per-package resolution") normatively pins `pi.looms` glob resolution to the `minimatch` engine, and the same paragraph also references Pi's internal `@earendil-works/pi-coding-agent` `dist/core/package-manager.js` symbol `applyPatterns`. The companion paragraph that enumerates loom's own runtime npm dependencies — `docs/spec_topics/implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)" (anchor `#loom-package-implementation-dependencies-loom-1-0`) — already names `semver`, `chokidar`, and `yaml` on the same footing ("declared in the loom package's `dependencies` block"). `minimatch` is absent from that enumeration.

The spec corpus therefore names a third-party package as a load-bearing matcher in a normative rule but never tells the implementer where it comes from. Two readings are left open: that loom declares its own `dependencies` entry on `minimatch` (the model `semver` / `chokidar` / `yaml` follow), or that loom reaches into the copy bundled with `@earendil-works/pi-coding-agent` (suggested by the `dist/core/package-manager.js` / `applyPatterns` parenthetical, even though `applyPatterns` is an internal symbol Pi makes no stability promise about). The plan side (`docs/plan_topics/H1a-scaffold-and-toolchain.md`) has already chosen the first reading — `minimatch` is declared in loom's runtime `dependencies` and the leaf cites DISC-5 — but the spec impl-deps paragraph that the plan defers to (`per implementation-notes.md §"Loom-package implementation dependencies (loom 1.0)"`) doesn't carry the `minimatch` entry, so the plan's choice is not anchored in the spec it points at.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 1cb54c7 — pi-loom spec: resolve "Glob !/+/- precedence and matcher engine unspecified for pi.looms / loomPaths" (2026-06-06, Thomas Andersen)
**History:** Commit 1cb54c7 rewrote DISC-5 to pin the matcher engine to `minimatch` and to reference Pi's internal `applyPatterns` symbol; that commit touched only `docs/spec_topics/discovery/package-and-settings.md` and did not extend the parallel enumeration paragraph in `docs/spec_topics/implementation-notes.md` (which at that point listed `semver` / `chokidar` and was later extended for `yaml` in 1064946 on 2026-06-10). No subsequent commit folded `minimatch` into that paragraph (verified with `git log -S 'minimatch' -- docs/spec_topics/implementation-notes.md`, which returns no matches), so the asymmetry — a normative `minimatch` citation in DISC-5 with no matching impl-deps line — has been present since 1cb54c7.

## Solution Space

**Shape:** single

### Recommendation

Extend `docs/spec_topics/implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)" with a `minimatch` clause on the same footing as the existing `semver` / `chokidar` / `yaml` clauses. The clause should:

- Name `minimatch` as the loom runtime dependency consumed by DISC-5's glob matcher (link to `discovery/package-and-settings.md#disc-5`).
- State that `minimatch` is declared in **loom's own** `dependencies` block (loom does not reach into the copy bundled with `@earendil-works/pi-coding-agent`, and the `dist/core/package-manager.js` / `applyPatterns` reference in DISC-5 is an editorial pointer at Pi's matching shape, not an import target).
- Follow the established "names the dependency, not its version" wording the other three clauses use, leaving the pinned range to `package.json`.
- Add the standard inventory-closure carve-out sentence already used for `chokidar` and `yaml`: `minimatch` is neither one of the four `@earendil-works/*` peer packages nor `typebox`, so it is out of scope for the inventory-closure audit's *Target surface categories* and requires no `// allow-pi-surface:` exemption marker at any `minimatch` import site.

Implementer edge case: the existing `H1a` text already cites this paragraph as the source for `minimatch`'s placement in `dependencies`; after the edit the citation lands on a clause that actually exists, so no plan-side rewording is needed.

## Consequence

**Severity:** advisory

A second implementer working from the spec alone (without consulting `H1a`) has no anchored answer to whether loom ships its own `minimatch` runtime dependency, omits the `dependencies` entry on the assumption that Pi supplies one, or imports an internal Pi symbol that carries no stability promise. The defect does not block the current build — `H1a` independently records the right answer — but it leaves the spec internally inconsistent (`semver` / `chokidar` / `yaml` enumerated, `minimatch` not) and makes the plan's `H1a` citation point at a paragraph that doesn't actually list the dependency it claims to source.
