## Finding

`docs/spec_topics/discovery/package-and-settings.md` §"Package discovery" lists five roots and then states (Roots scanned paragraph): "The extension enumerates each of the following directories and inspects each immediate child as an installed package …", with a follow-on paragraph that pins the candidate-enumeration rule as "every immediate child directory whose name does **not** begin with `@` is a candidate package, and every immediate child directory whose name **does** begin with `@` is a scope directory whose own immediate children are candidate packages."

That rule fits roots 1 (`.pi/npm/`), 3 (`node_modules/`), and 4 (`~/.pi/agent/npm/`) — flat (and one level of `@scope/`) — but does not fit roots 2 and 5, both of which the spec itself spells with a nested template: `.pi/git/<host>/<path>/` and `~/.pi/agent/git/<host>/<path>/`. Pi clones git packages to `<git-root>/<host>/<path>` (e.g. `~/.pi/agent/git/github.com/user/repo/`); the "immediate children" of `.pi/git/` are hostname directories such as `github.com`, not packages. Worse, the `<path>` component can be `user/repo` *or* a deeper path like `user/sub/repo` depending on the source shorthand pi accepts (`pi install git:github.com/user/repo`, `pi install ssh://git@host/path`), so even one fixed level of descent below the host directory would not exhaust the layout.

The spec therefore has no implementable enumeration procedure for the two git roots: literally applying the candidate-package predicate inspects host directories for `package.json` and finds none, contributing zero looms from any git-installed package; correctly walking the variable-depth layout is left implicit. The host (`@earendil-works/pi-coding-agent`'s `DefaultPackageManager`) already maintains the authoritative inventory — `listConfiguredPackages()` returns `{source, scope, installedPath}` for each configured package, and `getGitInstallPath(parsed, scope)` is the internal source of truth for the `<host>/<path>` mapping — but no seam delegating that knowledge to the extension is named on the page. The companion finding "`npmCommand`-resolved global npm root: no API/seam named for reading it" identifies the same gap for npm root 4.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** d6cbb37 — pi-loom spec: resolve "pi.looms is an extension-owned convention" (2026-05-04, Thomas Andersen)
**History:** d6cbb37 introduced the Package discovery section in one stroke: it added the five-root list (including the nested `.pi/git/<host>/<path>/` and `~/.pi/agent/git/<host>/<path>/` templates) and the "every immediate child whose `package.json` parses successfully is treated as a candidate package" enumeration rule in the same commit, with no per-root override for the git layout. Later edits to this section (f7a0dae rewording the candidate-enumeration rule to handle `@scope`; 1cb54c7 pinning the `!`/`+`/`-` ordering; 27e914a addressing pnpm-isolated entries on `node_modules/`) refined the npm-shaped roots without revisiting the git template, so the mismatch present at d6cbb37 has carried through unchanged. The defect is a single-commit oversight at the section's introduction.

## Solution Space

**Shape:** multiple

### Option A — Delegate enumeration to Pi via a new host-interface seam

**Approach.** Define a new `PackageInventory` member on the Pi host interface (`host-interfaces-services.md`) returning the inventory Pi already maintains — one entry per configured package with `{source, scope, installedPath, type: "npm" | "git"}` — and rewrite the Roots scanned paragraph to enumerate via that seam for the two git roots (and root 4's `npmCommand` case, per the sibling finding). The `@scope`-unwrapping candidate rule continues to apply to the three flat npm roots, which the extension still walks directly.

**Spec edits.**
- Add a `PackageInventory` (or equivalent) entry to `host-interfaces-services.md` returning `installedPath`s for git-scoped packages, grounded in Pi's `PackageManager.listConfiguredPackages()` + `getInstalledPath()` shape.
- Rewrite the Roots scanned paragraph to: walk the three npm roots directly with the existing candidate rule; obtain root 2 and root 5 candidate paths from the seam.
- Reconcile with the `npmCommand` finding (sibling): the same seam covers root 4's resolved global npm root, removing the literal-path fallback ambiguity.

**Pros.**
- Single source of truth: Pi already knows where each package lives, including the variable-depth `<host>/<path>` mapping it constructed.
- Robust to future install channels Pi may add (Pi's inventory grows; the extension follows).
- Co-resolves the `npmCommand` finding.

**Cons.**
- Requires a new PIC seam and an SDK-version commitment from Pi.
- Larger spec surface change; adds a host-prerequisite line item.

**Risks.**
- The seam must exist at the pinned Pi SDK version, or the spec needs a fallback paragraph for versions that lack it.

### Option B — Define the git-root traversal in-spec

**Approach.** Keep the extension walking the filesystem itself; pin a git-root traversal that descends to find directories containing `package.json`. Recommended rule: at each git root, walk depth-first, treating any directory that contains `package.json` as a candidate package and not descending into it further; cap the descent at a small fixed depth (e.g. 6) to bound the walk against pathological layouts and surface a `loom/load/discovery-slow`-style diagnostic on cap.

**Spec edits.**
- Replace "inspects each immediate child as an installed package" with a per-root predicate: roots 1/3/4 use the flat-plus-`@scope` rule; roots 2 and 5 use the "first `package.json`-bearing directory wins, descend no further" rule.
- Pin the depth cap and its diagnostic, fitting it under the existing `looms.scanPackagesMaxFiles` / `looms.scanPackagesTimeoutMs` budget (the per-`package.json`-open accounting already covers cost).
- State the candidate-package path is the directory containing `package.json`, used as-is for `pi.looms` glob-base resolution.

**Pros.**
- No new PIC seam; no Pi-side SDK change.
- Self-contained: the extension can be implemented and tested without a host stub for the new seam.

**Cons.**
- Duplicates Pi's `<host>/<path>` knowledge; if pi ever changes the on-disk git layout, both sides must change.
- Requires defining and testing an in-spec walk for a layout Pi already enumerates.
- Does not address the sibling `npmCommand` finding — that one still needs a separate fix.

**Risks.**
- A first-`package.json`-wins rule misses nested workspaces (`user/repo/packages/sub-pkg`); whether that is desired or a defect depends on whether pi clones workspace repos.
- Depth cap chosen without measuring real `<path>` depths in the wild may either truncate valid installs or burn budget on hostile inputs.

### Recommendation

Option A. Pi is the only system that authoritatively knows where each git package lives — the install path is a function of the parsed source shorthand (`getGitInstallPath` in `dist/core/package-manager.js`), not a stable pattern the extension can re-derive. Delegating to a host seam keeps the extension correct across any future change to Pi's git-clone layout and lets the same seam resolve the `npmCommand`-resolved npm root issue cited in the related finding. Implementer must watch: the seam member's contract must distinguish project-scope from user-scope entries so the existing project-before-global priority order is preserved, and must hand back paths post-`~` expansion so the `FileSystem.homedir()` rule (DISC-1) is not duplicated.

## Consequence

**Severity:** correctness

Two implementers reading the spec literally will produce a working npm-package walk and a silently broken git-package walk: every git-installed loom package contributes zero looms, with no diagnostic and no way for the user to discover the omission. Because git packages are a first-class install channel in Pi (`pi install git:…`), the omission is observable end-to-end on any project that installs looms via git.
