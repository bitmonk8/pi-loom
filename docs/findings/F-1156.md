## Problem

`docs/spec_topics/discovery/package-and-settings.md` §"Package discovery" tells the extension to enumerate each root's immediate children as candidate packages (flat directories, plus one level of `@scope/`). That rule fits the three npm roots (1 `.pi/npm/`, 3 `node_modules/`, 4 `~/.pi/agent/npm/`) but not roots 2 (`.pi/git/<host>/<path>/`) and 5 (`~/.pi/agent/git/<host>/<path>/`), whose immediate children are hostname directories such as `github.com`, not packages. The `<path>` component is variable-depth (`user/repo` or `user/sub/repo` depending on the install shorthand), so no fixed descent rule walks it. Literally applying the candidate predicate to the git roots finds zero packages, so every git-installed loom contributes nothing with no diagnostic.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** d6cbb37 — pi-loom spec: resolve "pi.looms is an extension-owned convention" (2026-05-04, Thomas Andersen)
**History:** d6cbb37 introduced the Package discovery section in one stroke: it added the five-root list (including the nested `.pi/git/<host>/<path>/` and `~/.pi/agent/git/<host>/<path>/` templates) and the "every immediate child whose `package.json` parses successfully is treated as a candidate package" enumeration rule in the same commit, with no per-root override for the git layout. Later edits to this section (f7a0dae rewording the candidate-enumeration rule to handle `@scope`; 1cb54c7 pinning the `!`/`+`/`-` ordering; 27e914a addressing pnpm-isolated entries on `node_modules/`) refined the npm-shaped roots without revisiting the git template, so the mismatch present at d6cbb37 has carried through unchanged. The defect is a single-commit oversight at the section's introduction.

## Solution approach

Add a host-interface seam member to `docs/spec_topics/pi-integration-contract/host-interfaces-services.md` that returns Pi's authoritative package inventory — per configured package, its source, scope, and resolved install path — so the extension obtains git-package locations from Pi instead of re-deriving the variable-depth `<host>/<path>` layout. Rewrite the Roots scanned paragraph in §"Package discovery" so enumeration splits by root type: walk the flat npm roots (1, 3, 4) directly under the existing flat-plus-`@scope` candidate rule, and obtain candidate paths for the two nested git roots (2 and 5) from the new seam.

## Solution constraints

- The seam contract MUST distinguish project-scope from user-scope entries so the existing project-before-global priority order is preserved.
- The seam MUST return paths post-`~` expansion so the DISC-1 `FileSystem.homedir()` rule is not duplicated in the extension.
