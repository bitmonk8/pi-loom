## Finding

The Audit-methodology subsection of `docs/spec_topics/hard-ceilings/ceiling-invariants-and-audit.md` uses the same italicised parenthetical-roman marker style — `*(i)*`, `*(ii)*`, `*(iii)*`, `*(iv)*` — for three independent enumerations packed into a few adjacent paragraphs:

1. **Four-axis closure list bullets** (lines 30–33): each of the four axis bullets uses `*(i)* **axis** — *(ii)* `NOCEIL-N` — *(iii)* justification` as poor-man's column markers, so `(i)`/`(ii)`/`(iii)` identify *columns within each bullet*.
2. **Four retirement forms** in the *Routing obligation* paragraph (line 37): `(i) **succeed** … (ii) **fall through to a documented `NOCEIL-N`** … (iii) **fall through to another ceiling** … (iv) **non-terminating absent cancellation**`. These are cross-referenced as `forms (i)–(iv)` by both the Release-gate paragraph (line 43) and the aggregator-co-edit paragraph (line 45).
3. **Three required components** every per-axis entry must name, also at line 37, immediately after the retirement-forms list: "must name (i) the axis, (ii) the routing destination, and (iii) a one-clause justification". This reuses `(i)`/`(ii)`/`(iii)` for the *same conceptual fields* as scheme 1 but reads as a fresh enumeration.

Because the three schemes co-exist in a single dense methodology block — and because external prose in the same file refers to "forms (i)–(iv)" without ever saying *which* `(i)–(iv)` — a reader landing on a cross-reference cannot disambiguate retirement-form `(ii)` ("fall through to a documented `NOCEIL-N`") from closure-list-column `(ii)` (the `NOCEIL-N` column of a four-axis bullet) without re-walking the paragraph. The collision is structurally invited by the column-marker style of scheme 1: a four-axis bullet that needs three columns and a retirement enumeration that needs four items both default to `(i)…(iv)`, and the existing prose has already proven willing to coin a third reuse one sentence later.

## Solution Space

**Shape:** single

### Recommendation

Re-mark schemes 1 and 3 with a distinguishable style, leaving scheme 2 (the retirement forms) on `(i)–(iv)` so the existing cross-references "forms (i)–(iv)" in the *Release gate* paragraph and the aggregator co-edit paragraph continue to resolve unambiguously to the only `(i)–(iv)` enumeration in the section.

Concrete edits in `docs/spec_topics/hard-ceilings/ceiling-invariants-and-audit.md`:

- **Closure-list bullets (lines 30–33).** Convert from `*(i)* axis — *(ii)* `NOCEIL-N` — *(iii)* justification` to a markdown table with explicit column headers `Axis | Routing destination | Justification`. A table both removes the marker collision and makes the per-axis-entry contract that the *Routing obligation* paragraph imposes (three named fields per entry) self-evident from the rendering rather than carried by prose.
- **Per-axis-entry required-components clause (inside the *Routing obligation* paragraph, line 37).** Drop the inline `(i)/(ii)/(iii)` markers and reword to name the three fields by label: "must name the **axis** (one of the four above, or a newly introduced axis), the **routing destination** (the new ceiling itself, or the specific `NOCEIL-N` claim whose documented fall-through carries the resource), and a **one-clause justification** tying the axis to the routing destination". This aligns the prose with the table column headers from the first edit and removes the second collision site.
- **Cross-references to "forms (i)–(iv)"** (in the *Release gate* paragraph at line 43 and in the closing aggregator co-edit paragraph at line 45) require no change — once schemes 1 and 3 stop using parenthetical-roman markers, `(i)–(iv)` is unambiguous on its single remaining referent (the retirement forms).

Edge cases for the implementer:

- Preserve the existing `<a id="four-axis-closure-list">` anchor on the table heading or the paragraph immediately above the table; the *Operational definitions* clause in `docs/spec_topics/governance/source-language-stability.md` line 21 links to it via `#four-axis-closure-list`.
- The four post-retirement runtime trajectories enumerated at the *Retire* clause of `docs/spec_topics/governance/source-language-stability.md` are cited from this file as "forms (i)–(iv)"; keep the retirement-forms enumeration ordered exactly as today (`succeed` / `fall through to a documented NOCEIL-N` / `fall through to another ceiling` / `non-terminating absent cancellation`) so the ordinals continue to correspond.
- The renumbered-clause case `(2) split-axis case` / `(3) new-axis case` in the *Routing obligation* paragraph uses arabic-in-parens, not roman, and is not part of this collision; leave it untouched.

## Consequence

**Severity:** cosmetic

A reader following a `forms (i)–(iv)` cross-reference must mentally distinguish three same-shaped enumerations in one section; no implementer-visible behaviour diverges, but reviewer-inspection at release time (the GOV-15 step the Release-gate paragraph cites) is slowed and a future spec-editor extending the retirement form list or the per-axis-component list risks introducing a marker collision that *would* be substantively ambiguous.
