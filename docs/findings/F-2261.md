## Finding

The Pi version-bump procedure is written end-to-end in npm-specific idioms but never declares that an npm-style toolchain is a precondition. Step 1 (`version-bump-intro.md`) runs `npm run typecheck`; step 3 (`version-bump-step2b.md`) and several checklist items in `version-bump-step2.md` (e.g. items (i), (u)) and the trigger steps in `version-bump-triggers.md` run `npm test` and `npm run typecheck`; the *Baseline acquisition* paragraph in `version-bump-step2.md` runs `npm pack`, contrasts it with `npm install --prefix`, and then reads the candidate-minor `dist/*.js` bodies directly from "the locally-installed `node_modules/@earendil-works/pi-coding-agent/dist/` tree" as the diff's "to" side.

Two layered assumptions are baked in without being named: (1) the contributor's package manager exposes the `npm pack`/`npm install` CLI and the `npm run typecheck` / `npm test` script-runner surface, and (2) installed dependencies materialise as a flat, directly-readable `node_modules/<pkg>/dist/` directory. Both fail to hold on package managers the spec corpus elsewhere explicitly anticipates. `host-prerequisites.md` (Pi SDK pin, typebox single-instance precondition) is written around the fact that "package managers (pnpm/yarn) … do not auto-deduplicate transitive peer-dep ranges," and `discovery/package-and-settings.md` documents that "pnpm's default isolated node-linker writes top-level `node_modules/<pkg>` entries as symlinks into `node_modules/.pnpm/…`." Under that default pnpm layout, the `node_modules/@earendil-works/pi-coding-agent/dist/` path the baseline step reads is a symlink into a content-addressed store rather than the flat tree the prose describes, and `npm run`/`npm pack`/`npm test` are not the contributor's invocation surface.

The procedure is internally complete and correct for an npm host; the gap is that it never states it assumes one, so a contributor on a different toolchain has no signal that the acquisition and JS-body-read steps must be adapted.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** none identified (the omission is continuous); the explicit `node_modules/.../dist/` directly-readable-tree read was added by ccc6e38 — "pi-loom spec: resolve 'Item (e)/(r) dist/*.js baseline source unspecified'" (2026-06-07, Thomas Andersen)
**History:** The npm-idiom steps (`npm run typecheck`, `npm test`) have been present in the version-bump procedure's lineage since at least 0e12140 (2026-05-08), and a package-manager precondition was never authored alongside them, so the unstated-assumption gap has existed since the procedure was first written. The narrower "directly-readable `node_modules/.../dist/` tree" half of the assumption was sharpened later by ccc6e38 (2026-06-07), which added the *Baseline acquisition* paragraph reading the installed `dist/` tree as the diff's "to" side — again without an accompanying package-manager precondition.

## Solution Space

**Shape:** single

### Recommendation

Add a one-sentence package-manager precondition to the version-bump procedure's entry point (the `version-bump-intro.md` preamble, before step 1), stating that the procedure assumes an npm-style toolchain: the `npm pack` / `npm install` CLI, the `npm run typecheck` / `npm test` script-runner surface, and a flat, directly-readable `node_modules/@earendil-works/pi-coding-agent/dist/` tree as the installed "to" side of the `dist/*.js` diffs. Note that a contributor on a different package manager — notably pnpm in its default isolated node-linker mode (which, per `discovery/package-and-settings.md`, writes `node_modules/<pkg>` as a symlink into `node_modules/.pnpm/…`), or yarn — MUST adapt the acquisition and JS-body-read steps accordingly, resolving the real `dist/` location and substituting the equivalent script invocation.

Edge cases the implementer must watch: under pnpm's isolated linker the `dist/` path is a symlink target (the read still works through the link, but the "flat tree" mental model in the *Baseline acquisition* prose does not); and the precondition should be stated at the single procedure entry point rather than repeated at each `npm`-invoking step, to avoid an N-site sweep on a future toolchain change.

## Consequence

**Severity:** advisory

A contributor running the version bump on pnpm (default isolated node-linker) or yarn finds the `node_modules/@earendil-works/pi-coding-agent/dist/` read resolves through a symlink into the content-addressed store rather than a flat tree, and the `npm pack`/`npm run`/`npm test` invocations are not their toolchain's surface. They can adapt, but the procedure gives no signal that adaptation is required, so the acquisition and JS-body-diff steps may be misapplied or silently skipped.
