## Finding

In `docs/spec_topics/query/query-forms.md`, the "Schema inference rules" section (around line 29) pins a **closed** three-element list of sink positions: the binding annotation, the declared return type of the enclosing function, and the declared parameter type of the enclosing call site. The "Schema inference algorithm" section (around line 41) then explains the shallow walk and, in the bullet describing array-literal element transparency, parenthesises the gating condition as "*if and only if* the literal has a sink (binding annotation, parameter type, etc.)".

That parenthetical does two things at once — both unintentional. First, the "etc." hedges an enumeration that the preceding rules section has already closed, inviting a reader to imagine sink positions beyond the three listed (e.g. a `match`-arm annotation, an `as`-cast, a struct-field type). Second, the parenthetical itself drops one of the three real sinks (the declared return type / `return`-operand position), so even read as illustrative it under-enumerates: nothing in the section signals that omitting the return-type sink is editorial rather than substantive. The same pattern recurs identically in the analogous bullet for ternary branches earlier in the same line (`*if and only if* the ternary itself has a sink`), where the gating condition has no parenthetical at all, increasing the inconsistency between the two sibling clauses.

The net effect is that a careful implementer reading only the algorithm section gets a fuzzier picture of the closed sink set than a reader of the rules section directly above it, and the algorithm section is the one the worked examples and downstream test vectors are anchored against.

## Solution Space

**Shape:** single

### Recommendation

In the "Schema inference algorithm" bullet for `Crossed (transparent)` positions, replace the parenthetical "(binding annotation, parameter type, etc.)" with a back-reference to the closed enumeration immediately above:

> the elements of an array literal `[a, b]` *if and only if* the literal has a sink (per the closed set in [Schema inference rules](#schema-inference-rules) above).

Apply the identical treatment to the sibling ternary clause earlier in the same line ("the branches of a ternary `cond ? a : b` *if and only if* the ternary itself has a sink") for consistency — either add the same back-reference to both, or add it to neither but introduce a single lead-in sentence ahead of the bullet stating that "has a sink" means "occupies one of the sink positions enumerated in the rules section above". The back-reference form is preferred because it keeps the bullet self-anchoring; the single lead-in form is acceptable if section length is a concern.

Edge cases the implementer must watch:
- Do not in-line a fresh three-element list; that would create a second normative copy of the closed set, which the next consistency lens pass would flag as drift-prone.
- The fix is purely editorial; no `<a id>` anchors move, no cross-link from `query-escapes-stringification.md` (which currently links to `#degenerate-rendered-templates`, not to this section) breaks. Verify with `docs/_tools/check_links.py` after the edit regardless.
- The "Stopped (opaque)" bullet on the next line ends with "(`+`, `==`, `!`, etc.)" — that "etc." is **not** in scope. The operators list is genuinely open (any operator the grammar adds in future is opaque by default), so the hedge is correct there; do not touch it.

## Consequence

**Severity:** advisory

Two reasonable implementers reading only the algorithm section could disagree on whether the sink set is closed or extensible, with the second implementer admitting positions the rules section forbids. The mismatch is small because the closed enumeration is stated unambiguously two paragraphs above, but the algorithm section is also where the worked examples live, so a divergent reading there propagates into test design.
