## Finding

`conventions.md` §"Cross-cutting rules (every leaf)" closes its scope paragraph with an absolute guarantee: "Because the entry file holds no broad `catch`, no global / static / singleton, no ambient-primitive read, and no `Promise` combinator of its own, no rule-subject behaviour falls outside the gated tree." That guarantee is conditional on `extensions/index.ts` staying a **pure re-export shim**. But `extensions/index.ts` sits outside `src/**`, and every mechanical gate in the plan reads `src/**` and no other tree: the `H2a` no-globals architectural test, the `no-broad-catch` and `no-restricted-syntax` lint rules, the `H3a` ambient-primitive scan, and the `H5a` / `H5c` closing-gate allow-list reconciliation. So none of them ever inspects the entry shim.

The two non-mechanical surfaces do not close the gap either. `H4a`'s **Tests** assert the never-throw factory contract, the harness, and the session-double fidelity contract — all against the `src/**` factory — but no `H4a` test asserts that `extensions/index.ts` itself carries no broad catch, global, ambient read, or `Promise` combinator. The `conventions.md` release-time residue inspection enumerates six checklist items, all scoped to `src/**` (plus repo-root doc artifacts for item 4 and `spec_topics/**` for item 5); none reads `extensions/index.ts`.

The completeness claim therefore overstates its footing: it is presented as a mechanically-closed guarantee, when in fact it rests on contributor discipline that no gate, no `H4a` test, and no release-time residue-inspection item observes. A later edit that adds a broad catch, an ambient read, a global, or a `Promise` combinator to the entry shim would silently break the "no rule-subject behaviour falls outside the gated tree" invariant with nothing to catch it.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commits:** `2f18db6` (`pi-loom plan: resolve "reconcile src/** production-code root with extensions/index.ts entry point"`, 2026-06-16)
**History:** The plan corpus is git-tracked. `git log -S` over `conventions.md` shows the entire scope paragraph — the `thin delegating entry shim` description and the closing "no rule-subject behaviour falls outside the gated tree" guarantee — was added (not modified) in commit `2f18db6`. `git log -S "thin delegating entry shim"` over `H4a-factory-shell-and-harness.md` returns the same single commit. The overclaim was therefore born with the reconciliation fix itself: that fix introduced the `src/**`-scope reconciliation and the entry-shim concept together, asserting the completeness guarantee on a prose-only property without backing it with a check or disclosing it as discipline. The durable fix addresses what that reconciliation commit established, not a later regression.

## Solution Space

**Shape:** multiple

### Option A — Disclose the property as discipline, not a mechanical guarantee

**Approach.** Reword the `conventions.md` scope paragraph so the closing sentence no longer claims an unconditional "no rule-subject behaviour falls outside the gated tree." State instead that the entry shim's purity (no broad catch / global / singleton / ambient read / `Promise` combinator) is a contributor-discipline property verified by the `H4a` self-review rather than by any of the `src/**`-scoped mechanical gates, and that the `src/**` scope is sound only while `extensions/index.ts` stays a pure re-export.

**Plan edits.** Edit the final sentence of `conventions.md` §"Cross-cutting rules" scope paragraph (the `extensions/index.ts` paragraph) to attribute the entry-shim purity to contributor discipline / the `H4a` self-review and qualify the completeness claim accordingly.

**Spec edits.** None.

**Pros.** Smallest change; makes the prose honest about its actual footing; matches the existing pattern where `conventions.md` concedes residues and names the surface that owns them.

**Cons.** Leaves the entry shim genuinely unobserved — nothing newly verifies it; relies on the `H4a` implementer remembering a property about a file outside the gated tree.

**Risks.** A future entry-shim edit can still silently break the invariant; the disclosure documents the gap rather than closing it.

### Option B — Route the entry-shim purity to the release-time residue inspection

**Approach.** Treat the entry shim's purity as a conceded mechanical-scan residue (the gates read only `src/**`, not `extensions/index.ts`) and route it to the existing `conventions.md` loom 1.0 release-time residue inspection, mirroring how items 1–6 own each other conceded residue. Add a checklist item that reads `extensions/index.ts` and confirms it carries no broad catch, no global / static / singleton, no ambient-primitive read, and no `Promise` combinator (i.e. that it remains a pure re-export of the `src/**` factory), then reword the completeness claim to cite that item as its closing surface.

**Plan edits.** Add a release-time residue-inspection checklist item in `conventions.md` covering `extensions/index.ts` purity with its "clean when" condition, and update the scope paragraph's closing sentence to cite that item rather than asserting an unconditional mechanical guarantee. Optionally cross-reference the new item from `H4a` so the leaf that authors the shim points at the surface that audits it.

**Spec edits.** None.

**Pros.** Closes the actual gap — something now verifies the shim; reuses the established conceded-residue → residue-inspection routing pattern; keeps the completeness claim true rather than merely disclosed.

**Cons.** Slightly larger edit; adds a seventh manual-review item to maintain.

**Risks.** The new item is a manual review like the other six, so it inherits their reliance on a reviewer actually running the inspection at release time.

### Recommendation

Take Option B. `conventions.md` already establishes the pattern that each mechanical-scan residue routes to the loom 1.0 release-time residue inspection; the entry-shim purity is exactly such a conceded residue (the gates scan `src/**`, never `extensions/index.ts`), so the durable fix is to give it the same owning surface and then cite it from the completeness claim — which both honours the existing structure and makes the "no rule-subject behaviour falls outside the gated tree" statement true rather than aspirational. Option A is acceptable if a no-new-checklist-item outcome is required, but it documents the gap instead of closing it. Either way the rewritten claim must stop asserting unconditional mechanical coverage it does not have. Edge case for the implementer: the new residue-inspection item reads a file outside `src/**`, so its "clean when" wording must name `extensions/index.ts` explicitly rather than inheriting the inspection's default `src/**` scope.

## Consequence

**Severity:** advisory

The completeness claim reads as a mechanically-enforced guarantee but is backed only by unobserved contributor discipline; a future edit that puts a broad catch, ambient read, global, or `Promise` combinator into `extensions/index.ts` would falsify "no rule-subject behaviour falls outside the gated tree" with no gate, `H4a` test, or residue-inspection item to flag it. Implementers can still produce working leaves; the defect is an overstated guarantee, not a blocked or divergent leaf.
