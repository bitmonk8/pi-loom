## Problem

The trailing sentence of the `INV-4` Tests bullet in `V15b-invoke-depth-cycle.md` (and its verbatim twin in `V15b-T-invoke-depth-cycle.md`) admits two materially different test designs. The grammar attaches "reaches the 32-frame boundary and fires at 33 > 32" to the cross-file `.warp fn` chain while "exercised across all three countable frame classes" reads as if all three reach the boundary, so two reasonable implementers resolve it differently. The spec (`invocation.md` §INV-4) defines the cap as a count of countable frames on a single shared per-chain counter; one reading verifies each frame kind's contribution at the cap, the other leaves the boundary behaviour of the `invoke` and `tools:` frame kinds unverified, letting a per-class counting defect in either ship undetected.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 084f171
**History:** The `INV-4` bullet had no "three frame classes" language at all until commit `084f171` ("pi-loom plan: resolve \"V15b counts .warp fn frames introduced by V15c but declares no dependency on it\"", 2026-06-10). That commit added the `V15c` dependency and, in the same edit, appended the sentence "The counter is exercised across all three countable frame classes, including a cross-file `.warp fn` call chain … that reaches the 32-frame boundary and fires at 33 > 32." to both `V15b-invoke-depth-cycle.md` and `V15b-T-invoke-depth-cycle.md` (`git log -S` on the defect token resolves to that single commit). The clause was introduced to make the warp-fn frame class explicit but carried the boundary-coverage ambiguity in with it; no later commit touched the sentence.

## Solution approach

Rewrite the trailing sentence of the `INV-4` Tests bullet so the boundary chain that trips the cap unambiguously exercises all three countable frame classes — a direct `invoke(...)` frame, a `.loom`-via-`tools:` frame, and a cross-file `.warp fn` frame — within the chain that reaches 33 > 32, so each frame kind's contribution to the single shared per-chain counter is observed at the boundary. Apply the identical edit to both `V15b-invoke-depth-cycle.md` and `V15b-T-invoke-depth-cycle.md`, since the bullet is duplicated verbatim.

## Solution constraints

- The spec (`invocation.md` §INV-4) is read-only; the defect is confined to the plan's `INV-4` test bullet.
- Out of scope: introducing a new leaf or altering the `coverage-matrix.md` `INV-4 | V15b` mapping.
