## Problem

The `Tool execution from loom code` section of `host-interfaces-core.md` (anchor `tool-execution-from-loom-code`) carries loom-runtime execution semantics: the `params` JSON-lowering rule, the `onUpdate` no-op policy, the `AgentToolResult` content-lowering procedure, the UTF-8 4096-byte code-point-boundary truncation algorithm, the *Outcome routing summary*, and the swallowing-handler obligation. The page's declared scope is Pi-owned host-interface snapshots re-validated on each Pi minor bump; these rules instead describe loom's own normative behaviour consuming the tool-execution boundary, not what Pi exposes. An implementer or auditor working from the code-side tool-call surface in `tool-calls.md` — which owns the closed `CodeToolError` enum and the code-side `<name>(args)` call — finds only forward-links and must navigate into the host-interface page for the load-bearing detail.

## Issue introduction

**Verdict:** present-since-inception

**Introducing commits:** `fae85d3` *Tighten spec for implementation-plan readiness* (2026-05-04) introduced the §*Tool execution from loom code* surface verbatim into the monolithic `spec.md`, already mixing the host-interface bullets (`toolCallId`, `params`, `signal`, `onUpdate`, `ctx`) with the runtime-side lowering rule (`{ content, isError }` → string return value; `Err(QueryError { kind: "tool_call", cause: "execution", … })`) under a single section.

**History:** `git log --reverse -G "Tool execution from loom code"` lands on `fae85d3` as the introducing commit; subsequent edits accreted the runtime-side content rather than relocating it. `8f8acbf` (2026-05-05) added the 4096-byte UTF-8 truncation rule (resolving "tool-error message truncation rule sloppy") at the same site; `11ef3b9` (resolving "Tool failure modes beyond throw / isError: true unspecified") introduced the non-settling / non-conforming-shape language; `2ebe288` (2026-05-07) added the post-cancel late-settlement discard and the swallowing-handler MUST; `5f04646` added the non-text-discard `Ok("")` paragraph; `482098b` (2026-06-04) split the late-settlement discard into the CNCL-1/2/3 sub-anchors but kept the rule in PIC. `fecb504` split `spec.md` into per-topic files (the section moved into `spec_topics/pi-integration-contract.md` unchanged), `31ff060` moved that file under `docs/`, and `f5e89f4` (2026-06-04) split the 576KB `pi-integration-contract.md` into the index+children layout, creating the present `host-interfaces-core.md` file with the section's content carried forward verbatim. The scope mixing is therefore traceable to the section's first appearance: every subsequent commit treated PIC as the canonical home for the runtime rule and reinforced the placement rather than challenging it.

## Solution approach

Move the loom-runtime execution rules out of `host-interfaces-core.md` § `tool-execution-from-loom-code` and onto the code-side tool-call surface in `tool-calls.md`, which already owns the `CodeToolError` enum and the code-side `<name>(args)` call; give the relocated content a stable anchor on `tool-calls.md`. Relocate the `params` JSON-lowering rule, the `onUpdate` no-op policy, the `AgentToolResult` content-lowering procedure, the truncation algorithm and worked example, the *Outcome routing summary*, and the swallowing-handler obligation. Leave in `host-interfaces-core.md` only the Pi-interface content — the `execute(toolCallId, params, signal, onUpdate, ctx)` call signature and the `AgentToolResult = { content, details, terminate? }` shape pin — plus a forward-link to the relocated semantics. Repoint the `V14a` / `V14a-T` **Spec.** citations and coverage notes to the new anchor.
