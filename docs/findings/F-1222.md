## Problem

Rule 1 of subagent.md's **Conversation drive — subagent mode** spawn block specifies the Pi built-in tool name→factory mapping only by two examples (`read` → `createReadToolDefinition(cwd)`, `bash` → `createBashToolDefinition(cwd)`), not as a closed enumeration. Pi exposes seven built-in tools at the loom 1.0 SDK pin (`read | bash | edit | write | grep | find | ls`, the `ToolName` union declared in `dist/core/tools/index.d.ts`), and `tools:` admits all seven by name, so the five remaining name→factory pairings and every factory's argument signature must be inferred from the two examples. The naming convention is conventional, not mechanical (e.g. `find` vs `findFiles`, `ls` vs `list`), so a wrong-guess factory name is a runtime TypeError inside `lowerLoomToolsToPi(...)` at `createAgentSession` rather than a load-time diagnostic.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commit:** `e279e0c` (2026-06-06) — *pi-loom spec: resolve "built-in Pi tool name → ToolDefinition resolution names no concrete API"*
**History:** Before `e279e0c`, rule 1 named the built-in-tool resolution surface as "resolved by name from the model registry / extension API" — fully unpinned. `e279e0c` replaced that disjunction with the current per-tool factory pattern and gave two concrete examples (`read → createReadToolDefinition(cwd)`, `bash → createBashToolDefinition(cwd)`) plus a pin to the factory family's declaration directory (`dist/core/tools/*.d.ts`) and package-root re-export. The improvement closed the original under-specification (which was *which* API to call) but stopped at exemplary enumeration, leaving the remaining five name→factory pairings (`edit`, `write`, `grep`, `find`, `ls`) and the per-factory argument signature unpinned. The defect is the residue of that single edit; subsequent commits on the file (e.g. `077afb5` on tools-allowlist suppression, `7a3b907` on PIC-22) do not touch the lowering rule.

## Solution approach

Rewrite rule 1's two-example parenthetical so the Pi built-in tool name→factory mapping is given as the closed set of seven names rather than illustrated by two examples, naming each factory function and its call signature. Ground the closed-set claim against the `ToolName` union and `allToolNames` set in `dist/core/tools/index.d.ts` at the loom 1.0 Pi-SDK pin (`host-prerequisites.md#pi-sdk-pin`). The five names left unpinned by the current examples are `edit`, `write`, `grep`, `find`, and `ls`; verify each factory's spelling and argument shape against the SDK declarations rather than the conventional pattern, and capture that loom 1.0 forwards only `cwd` (the forwarded `ctx.cwd`), never `options`.

## Solution constraints

- The enumerated built-in tool set MUST match the `ToolName` union / `allToolNames` set at the loom 1.0 Pi-SDK pin (`dist/core/tools/index.d.ts`) and stay consistent with the `tools:`-admitted name set in `frontmatter-fields-a.md#tools`; do not enumerate a tool name absent from that union.
