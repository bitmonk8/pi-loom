## Finding

`docs/spec_topics/pi-integration-contract/subagent.md` rule 1 of the **Conversation drive â€” subagent mode** spawn block requires `lowerLoomToolsToPi(loom.tools)` to materialise every Pi built-in tool in the loom's callable set to a `ToolDefinition` via "the corresponding per-tool factory function re-exported from the `@earendil-works/pi-coding-agent` package root", and gives two examples: `read â†’ createReadToolDefinition(cwd)` and `bash â†’ createBashToolDefinition(cwd)`. Pi exposes seven built-in tools at the loom 1.0 SDK pin â€” `read | bash | edit | write | grep | find | ls` (the `ToolName` union and `allToolNames` set declared at `dist/core/tools/index.d.ts`) â€” and `tools:` admits all seven by name per [Parameters and Frontmatter â€” `tools`](../frontmatter/frontmatter-fields-a.md#tools) and rule 2's own enumeration of the default suppression set (`read / bash / edit / write`). Five of the seven nameâ†’factory pairings are therefore not pinned anywhere on the spawn surface and must be inferred from the two examples.

The implicit pattern (`<name>` â†’ `create<TitleCase(name)>ToolDefinition(cwd, options?)`) is regular enough to guess, but the spec never states the rule, never names the closed set of seven tool names, and never pins each factory's argument signature. The closure-defect surfaces sharply at the spawn site: rule 4 already pins that the `customTools` field is not factory-probed, so a typo (`createListToolDefinition`, `createFindFilesToolDefinition`) or a wrong-shape call (`createBashToolDefinition()` with missing `cwd`) is a TypeError or undefined-reference thrown synchronously inside `lowerLoomToolsToPi(...)` and routed through the unspecified-spawn-throw path rather than caught by static surface validation. A `loom/load/unknown-tool` covers the *frontmatter* shape (no such Pi tool name) but does nothing for the lowering-side factory name once a known tool name is in hand.

## Solution Space

**Shape:** single

### Recommendation

Replace rule 1's two-example parenthetical with an inline seven-row table mapping each `ToolName` to its factory function and call signature, anchored at the `ToolName` declaration in `dist/core/tools/index.d.ts`. The table is the closed set; any future Pi-added built-in tool surfaces as a `loom/load/unknown-tool` until the table is widened. Concrete shape:

| `tools:` name | Factory function (re-exported from package root) | Call signature |
| --- | --- | --- |
| `read` | `createReadToolDefinition` | `(cwd: string, options?: ReadToolOptions)` |
| `bash` | `createBashToolDefinition` | `(cwd: string, options?: BashToolOptions)` |
| `edit` | `createEditToolDefinition` | `(cwd: string, options?: EditToolOptions)` |
| `write` | `createWriteToolDefinition` | `(cwd: string, options?: WriteToolOptions)` |
| `grep` | `createGrepToolDefinition` | `(cwd: string, options?: GrepToolOptions)` |
| `find` | `createFindToolDefinition` | `(cwd: string, options?: FindToolOptions)` |
| `ls` | `createLsToolDefinition` | `(cwd: string, options?: LsToolOptions)` |

State at the table that loom 1.0 passes only `cwd` (the forwarded `ctx.cwd` from the state-isolation matrix), never `options`, so `options`-default behaviour is the loom's contracted shape. Anchor the table's source-of-truth claim to `dist/core/tools/index.d.ts`'s `ToolName` union and `allToolNames` set at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin), and fold the table itself into the same `dist/core/tools/*.d.ts` re-validation obligation rule 1 already names (no separate checklist item needed â€” the table's correctness is exactly what that item already audits, now with a denser surface to diff against).

Edge cases the implementer must watch:
- A future Pi minor that adds an eighth built-in tool fails the version bump's `ToolName`-union diff and surfaces as an unpinned name; the table must be widened in lockstep with `tools:`'s admitted name set in `frontmatter-fields-a.md`.
- Pi exposes a name-driven dispatcher `createToolDefinition(toolName, cwd, options?)` at `dist/core/tools/index.d.ts`; loom 1.0 deliberately does **not** consume it (the table dispatches per-tool by name in loom code rather than delegating the closed-set assertion to Pi), so a future Pi rename of an individual factory function is caught by the per-row pin rather than masked by the dispatcher's runtime lookup.

## Consequence

**Severity:** correctness

Two implementers reading rule 1 will likely produce factory dispatchers that agree on `read`/`bash` and disagree on at least one of `edit`/`write`/`grep`/`find`/`ls` â€” the naming convention is conventional, not mechanical (e.g. `find` vs `findFiles`, `ls` vs `list`), and the SDK declarations are not cited at site. A wrong factory name is a runtime TypeError at `createAgentSession`, not a load-time diagnostic, so the bug ships in any loom that lists one of the five unpinned built-ins in `tools:`.
