## Problem

PIC-20 forbids minting `invocationId` and code-side `toolCallId` UUID bodies through any ambient UUID source outside the production adapter, but states no build-time-check conformance criterion — unlike PIC-12's timing-primitive ban and PIC-13's `homedir` bullet, which both carry one. PIC-20 also bans "`crypto.randomUUID()` (or any other ambient UUID source)", an open clause that cannot be enumerated or scanned, so the ban is not falsifiable.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** e9b0597 — pi-loom spec: resolve "invocationId has no injectable id-source seam" (2026-06-05); a765f6ca — fix F-0977: pin PIC-13 realpath case-canonicalisation guarantee for DISC-3 dedup (2026-06-22); 4556623 — fix F-1178: PIC-12 timing-primitive ban gets observable build-time conformance criterion (2026-06-25)
**History:** PIC-20 was introduced whole by e9b0597 (the only commit touching the `any other ambient UUID source` clause, per `git log -S`), already lacking any conformance criterion. a765f6ca added the PIC-13 `realpath` case-canonicalisation guarantee binding the production adapter with no stated test path. The PIC-20/PIC-12 asymmetry then crystallised on 2026-06-25 when 4556623 retrofitted the build-time-check conformance criterion onto PIC-12 (and PIC-13's `homedir` bullet by reference) but not PIC-20 — even though PIC-20 was edited the same day by 0c6c347 (fix F-1396, toolCallId minting) — leaving PIC-20 the only ambient-primitive ban in the section without a criterion.

## Solution approach

Add to PIC-20 a build-time-check conformance criterion by reference to PIC-12's canonical statement at `#pic-12`, mirroring how PIC-13's `homedir` bullet already cites it, scoped to direct lexical references to the banned UUID-source identifiers. Replace the open `(or any other ambient UUID source)` clause with a closed, enumerated identifier set — at minimum `crypto.randomUUID` — so the criterion is falsifiable.
