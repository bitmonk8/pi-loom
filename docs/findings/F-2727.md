## Problem

The *Code-keyed obligation areas (no numbered REQ-IDs)* table in `docs/plan_topics/coverage-matrix.md` interleaves ten retired rows — `cka-7`, `cka-12`, `cka-18`, `cka-21`, `cka-22`, `cka-23`, `cka-24`, `cka-25`, `cka-28`, `cka-29` — among the live obligation rows (~21% of the table). Each retired row records no live obligation: its obligations have migrated to numbered `PREFIX-N` REQ-IDs already mapped in the *Numbered REQ-IDs* table, and it carries a multi-sentence retirement rationale, a "(GOV-22 drain)" process label, and a `*(numbered above)*` closing-leaf placeholder. The retired rows are shape-identical to live rows and distinguishable only by parsing each row's prose for "retired", so a reader enumerating the active code-keyed obligation set must filter ten inert rows on every pass. Their only durable function is to keep the append-only `cka-<n>` token sequence intact so retired tokens are never reused.

## Issue introduction

**Verdict:** accreted-across-commits
**Introducing commits:** `716a13b` (first retired-row instance), then `1acdc55`, `941e52c`, `a9fb344`, `02b11c9`, `47915a2`, `ffe2cc9`
**History:** The retired-code-keyed-area-row convention — the "retired as a code-keyed-area row … (GOV-22 drain) … `*(numbered above)*`" form — was first authored in `716a13b` (2026-06-25, "fix F-1286 — coin QRY-1..21 across query subtree + repoint V13 leaves/coverage"), which drained the `query/` subtree to numbered `QRY-1 … QRY-21` and retired `cka-12` (`git log --reverse -S "retired as a code-keyed-area row" -- docs/plan_topics/coverage-matrix.md` reports `716a13b` as the earliest). The clutter then accreted one obligation area at a time as the GOV-22 draining campaign progressed: `1acdc55` (cka-18/24/25), `941e52c` (cka-23), `a9fb344` (cka-28), `02b11c9` (cka-21), `47915a2` (2026-06-27, bulk "retire wholly-numbered cka rows"), and `ffe2cc9` (2026-06-28, cka-7). No single commit introduced the defect — it is the cumulative residue of a multi-commit draining campaign, each commit correctly retiring a row in place while leaving its full rationale and process labels inline.

## Solution approach

In `docs/plan_topics/coverage-matrix.md`, lift the ten retired rows out of the live *Code-keyed obligation areas (no numbered REQ-IDs)* table and record their `cka-<n>` tokens in a dedicated retired-token subsection, keeping for each token only a one-line marker naming where its obligations migrated and dropping the multi-sentence rationale, the `(GOV-22 drain)` labels, and the `*(numbered above)*` placeholders. Update the preamble's append-only sentence to point the reader at the retired-token subsection for the full allocated-token set. Phrase each relocated marker so the relocated entry reads as "migrated to numbered REQ-IDs" rather than reusing the word "retired".

## Solution constraints

- Out of scope: `conventions.md` and the gate leaves (`H5d` / `H5f` / `H5a`, `H5c`) are read-only; confine the edit to `coverage-matrix.md`'s table and preamble. The spec corpus is untouched.
- The retired-token record must sit outside the two closing-leaf tables that `H5d` / `H5f` / `H5a` machine-read — author it as a list/section, not a third closing-leaf table — and must still list every retired `cka-<n>` token so the preamble's append-only "next free token" rule stays computable.
- Confirm no `src/**` `// allow:` / `// allow-broad-catch:` comment cites a retired `cka-<n>` token before relocating, since a token outside the live table no longer resolves as an exemption justification.
