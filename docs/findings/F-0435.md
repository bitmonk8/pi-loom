## Finding

The "Pre-designed fallback" sentence in `docs/plan.md` §Blocked obligations prescribes leaf-level implementation scope for `V9l`: what `V9l` ships when clause (a) resolves *branch unreachable as written* (collapse to full teardown + `/reload` with no degraded-tag write, degraded-branch obligations reworked spec-side), and which obligations are unaffected and stay in `V9h` (unknown-reason / closed-set / snapshot-failure). This is concrete leaf-scope content — it tells the `V9l` implementer what to build under the fallback branch.

`docs/plan.md` §How to use this plan states the reading model explicitly: "An implementer working on a single leaf only needs to open that leaf's page plus the spec topics it references" (step 3: "Open that leaf's page; read only the leaf and the spec topic(s) listed under its **Spec** field"). The fallback prescription sits in the master index, which the single-leaf implementer is not directed to open. `V9l`'s **Ships when** does cross-reference it ("the pre-designed fallback in `plan.md` §Blocked obligations governs what ships"), so the leaf is not wholly blind — but the prescriptive content the implementer must act on is one hop away in a document outside the stated reading scope, and the reader has to leave the leaf page to obtain it.

## Issue introduction

**Verdict:** introduced-by-specific-change (single commit)
**Introducing commits:** 9c42f2d
**History:** `docs/plan.md` §Blocked obligations was added in `55ed974` (2026-06-12), where the fallback was terse and carried an explicit "(in `V9l`)" parenthetical pointing the reader to the leaf. Commit `9c42f2d` (2026-06-16, "pi-loom plan: resolve \"V9l fallback path belongs in §Blocked obligations\"") expanded the fallback into the full leaf-scope prescription inline in the index and removed the "(in `V9l`)" pointer — relocating the prescriptive scope wholly into the master index, which is the state this finding flags. A prior review thus deliberately centralised this content; the present finding takes the opposite position, so the two are in direct tension.

## Solution Space

**Shape:** multiple

### Option A — Relocate the prescription into the leaf

**Approach.** Move the fallback's leaf-scope prescription into `V9l-session-only-degraded-branch.md` (Adds, or a labelled fallback note): the shipped shape under *branch unreachable* (full teardown + `/reload`, no degraded-tag write, degraded-branch obligations reworked spec-side in `session-only-degraded-state.md` / `SM-4`/`SM-5`/`SM-6`/`SM-3b`), and that the unknown-reason / closed-set / snapshot-failure obligations stay in `V9h`. Reduce §Blocked obligations to the block trigger, the resolution owner (version-bump editorial-review checklist item (a) / the `V9l`→`V18c` Deps edge), and a pointer to the leaf.
**Plan edits.** `docs/plan_topics/V9l-session-only-degraded-branch.md`: add the prescription text. `docs/plan.md` §Blocked obligations: strike the prescription sentences, retain trigger + owner + a pointer to the leaf.
**Spec edits.** None — the spec is read-only for this fix.
**Pros.** Single source of the prescription, located inside the leaf the implementer opens; matches the plan's single-leaf reading model.
**Cons.** §Blocked obligations loses its at-a-glance summary of what the fallback collapses to.
**Risks.** The retained §Blocked obligations pointer must resolve to the new leaf location.

### Option B — Keep it centralised; make the leaf self-sufficient

**Approach.** Leave the prescription in §Blocked obligations (preserving the centralised cross-leaf view) but inline the concrete shipped shape into `V9l`'s Ships when / Adds so the leaf reader needs nothing beyond the leaf page.
**Plan edits.** `docs/plan_topics/V9l-session-only-degraded-branch.md`: expand Ships when / Adds with the fallback's concrete shipped shape. `docs/plan.md`: unchanged.
**Spec edits.** None — the spec is read-only for this fix.
**Pros.** Retains a single centralised place that lists every blocked obligation and its fallback.
**Cons.** Duplicates the prescription across the index and the leaf, creating drift risk between the two copies.
**Risks.** The two copies can diverge on a later edit.

### Recommendation

Take **Option A**: the duplication Option B introduces is the larger long-term hazard, and Option A aligns the content with the plan's own single-leaf reading model. Edit the leaf first to carry the prescription, then trim §Blocked obligations to trigger + owner + pointer, so the index is never left pointing at content that has not yet landed. Preserve the "obligations stay in `V9h`" assignment and the resolution-owner pointer when trimming the index; the spec stays read-only for this fix.

## Consequence

**Severity:** advisory

A single-leaf implementer of `V9l` who follows the plan's stated reading model opens only the leaf page and the spec topics it cites; the concrete fallback shape (collapse-to-teardown, no degraded tag, the V9h obligation split) is not there. The leaf's **Ships when** cross-reference points the reader to the index, so the content is reachable rather than lost, which keeps this a guidance/placement gap rather than a divergence risk.
