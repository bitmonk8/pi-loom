## Finding

Two adjacent normative sections of `schema-subset.md` reach past observable behaviour into the runtime's internal data-structure layout, owning component, and traversal algorithm:

- **Lowering Algorithm, step 5 ("Per-schema sidecar")** specifies that the lowering pass captures, per `$defs` entry, "a sidecar with two maps". The behavioural obligations the rest of the spec actually consumes are (a) given a renamed field, the validation pass post-decode and the construction pass pre-encode can both recover the loom-side ↔ wire-name correspondence, and (b) given a validated JSON position whose source type was a *named* `enum` declaration (and not an anonymous `"a" | "b"` string-literal union), the inbound translation pass in `runtime-value-model.md` can recover the declaring-enum name to reattach as a tag. That a *single* sidecar carrying *exactly two* maps is the carrier is a structure choice; future widening (a third map for, say, default-value placement metadata) would silently violate the count without changing any observable behaviour.

- **Depth Enforcement, "Enforcement point"** declares that "the depth check lives in the `SchemaValidator` service (the same component that owns the AJV cache)" and then specifies that "implementation is a recursive descent over the parsed JSON value with a depth counter; the first node whose depth would exceed `5` short-circuits". The observable obligations are: a loom-owned depth walk runs at the five enumerated sites; it runs *before* AJV at each of those sites (so a depth-violating value never reaches AJV); the failure carries `schema_keyword: "maxDepth"`, the canonical message, and `cause: "schema_validation"`; the JSON-Pointer `path` is to the first node whose containing-subtree depth exceeds `5`. Whether the walk shares a class with the AJV cache, recurses, iterates with an explicit stack, or fuses the count into a streaming parser is unobservable. The `SchemaValidator` service identity is *separately* governed by [PIC-11](./pi-integration-contract/host-interfaces-services.md#pic-11) under [GOV-18 arm (a)](./governance/corpus-direction-and-scope.md#gov-18-arm-a) as an internal DI seam whose member set is explicitly non-binding, so naming it as the depth-walk owner in a normative voice contradicts that carve-out's intent — the placement claim is structurally the same kind of statement PIC-11 itself fences as non-normative.

These three over-prescriptions are pure leakage: the spec already states the behaviour each one is shaped around, and the structural commitments add no implementer-observable constraint while constraining internal refactors and the V5f/V5e leaf shapes downstream.

- `7f217a8` (2026-06-07) "pi-loom spec: resolve 'Language-core hidden assumptions: enum-tag sidecar, AJV-config silence, Markdown claim'" — replaced the prior single-map step-5 ("Wire-name translation is captured in a sidecar map per schema") with the current "sidecar with two maps" formulation that calls out the named-enum-positions map as a sibling.

The over-prescription is the cumulative effect: each commit reached one step past the behaviour it needed to pin and committed to a specific internal structure or owning component. Neither commit is the locally-wrong edit; the cumulative wording is what the prescription lens now flags, so a revert of either commit is not the right fix.

## Solution Space

**Shape:** single

### Recommendation

Re-shape the three offending passages so they pin behaviour and fence (or drop) the structural commitments. Three local edits, in this order:

1. **Lowering Algorithm step 5** — rename from "Per-schema sidecar" to "Per-schema lowering metadata" (the prose name); re-state its content as two behavioural obligations the lowering pass discharges, without specifying their carrier shape. The two obligations to keep, verbatim in substance:
   - For each renamed field, the lowering pass records the loom-name ↔ wire-name correspondence so the validation pass (post-decode) and the construction pass (pre-encode) can both recover it; the lowered JSON Schema only ever sees wire names.
   - For each JSON Pointer position into the lowered schema fragment whose source type was a *named* `enum` declaration, the lowering pass records the declaring `enum`'s loom-side name so the inbound translation pass in [Runtime Value Model — Wire-name translation](./runtime-value-model.md) can reattach the declaring-enum tag. Anonymous string-literal-union positions (`"a" | "b"`) carry no such record.

   Drop "a sidecar with two maps". Drop the literal "(1) … (2) …" map enumeration. The "iff its source type was a named `enum` declaration" predicate is behavioural and stays.

2. **Depth Enforcement, "Enforcement point" paragraph** — replace the placement sentence and the algorithm-shape sentence:
   - Strike "The depth check lives in the `SchemaValidator` service (the same component that owns the AJV cache; see [Implementation Notes — Runtime](./implementation-notes.md#runtime))." Replace with: "The depth check is owned by the loom runtime (not by AJV; the JSON Schema 2020-12 vocabulary omits `maxDepth`). It runs at every site where a Loom-declared schema is validated against runtime JSON:" The reference to the AJV cache's owning component is not needed for the behavioural claim that follows.
   - Keep "The walk runs **before** AJV at each site: it is a cheap fast-fail and avoids feeding pathologically deep payloads into the validator." (behavioural — observable through the no-AJV-error-on-depth-violation property).
   - Strike "Implementation is a recursive descent over the parsed JSON value with a depth counter; the first node whose depth would exceed `5` short-circuits and produces the failure." The behavioural counterpart already lives in the Counting algorithm and the Error shape table (which pin the JSON-Pointer-to-first-too-deep-node `path` shape).

3. **Edge cases bullet "a uniform implementation across all five sites"** — soften to "the depth-walk obligation is installed at the `params` boundary unchanged" — uniformity of *implementation* is a structure claim; uniformity of *obligation* is the behavioural one the bullet actually relies on.

Implementer edge cases to watch:

- The Error shape table's per-row carrier specs (`ValidationError`, `CodeToolError`, `InvokeInfraError`, model-feedback row, slash-load row) are observable and must be left intact; only the prose around the table changes.
- The `V5f` test that asserts the "two-map shape" needs to be re-phrased to assert behavioural recoverability of (wire-name translation; named-enum reattachment) — keep the test green by exercising the two queries against the lowering pass's output, not by introspecting a literal carrier.
- The `V5e` leaf's **Adds** field reads "the loom-owned recursive JSON-depth walk"; drop "recursive" to match the spec rephrase, leaving "the loom-owned JSON-depth walk".
- PIC-11 stays untouched — it already owns the `SchemaValidator` contract under the GOV-18-arm-(a) non-binding-member-set carve-out, and de-coupling depth from that contract is precisely what removes the local contradiction.

## Consequence

**Severity:** advisory

Implementers can build a working system from the current text; the over-prescriptions are internally consistent. They do, however, lock loom's internals against refactors the spec has no behavioural reason to forbid (fusing the depth walk into a single-pass validator wrapper; carrying the lowering-pass per-`$defs` metadata as one record rather than two maps; sharing the depth walk's owning class across a different validator decomposition) and they leak structure into `V5f`/`V5e` tests as assertions on the shape rather than the behaviour, calcifying the internal contract through the test suite. The placement claim ("lives in the `SchemaValidator` service") also locally contradicts the GOV-18 arm (a) posture that PIC-11 itself adopts for internal DI seam member sets.
