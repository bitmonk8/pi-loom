## Finding

The *Serialised content format* rule in `diagnostic-shape.md` fixes the `content` string of each `loom-system-note` to the line format `"<file>:<line>:<col>: <code>: <message>"` (with the related-site variant `"  <file>:<line>:<col>: <message>"` and the file-only / location-less reductions). The `<file>` token is the `Diagnostic.file` field, declared in the same file as `file?: string, // absolute path`.

This format uses `:` simultaneously as the `file`/`line`/`col`/`code` field separator and as a character that occurs *inside* the `file` value on Windows, where an absolute path begins with a drive-letter colon (`C:\path\to\file.loom`). A consumer that splits the rendered line on `:` to recover the location triple mis-segments any Windows path: the drive letter is read as `<file>`, the rest of the path as `<line>`, and so on. The defect is latent — it surfaces only when an absolute path containing `:` is rendered — but the format as written presents itself as a structured, delimited line without acknowledging the collision or restricting how it may be consumed.

The spec already provides a clean structured channel: the same diagnostics are carried as a typed `Diagnostic[]` in `details.diagnostics` "for the renderer and for downstream consumers (LSP integrations, test harnesses) that want the typed shape rather than the rendered string." That sentence frames the structured array as *available to* machine consumers but stops short of (a) declaring the rendered `content` line display-only and (b) directing machine consumers away from parsing it. The gap is the missing explicit boundary between the human-display string and the machine-readable channel.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `fae85d3` ("Tighten spec for implementation-plan readiness", 2026-05-04)
**History:** The `Diagnostic` shape and its serialisation rule first entered the corpus in `fae85d3`; the prior commit (`8eea523`) has no Diagnostic shape or serialisation prose. That single commit introduced both the colon-delimited rendered line (`"<file>:<line>:<col>: <code>: <message>"`) and the `file: // absolute path` field together (`git log -S '<file>:<line>:<col>'` and `-S 'absolute path'` both bottom out at `fae85d3`), so the delimiter/path-content collision was present the moment the construct was coined. Subsequent edits preserved it: `8837a93` added the related-site line reusing the same `<file>:<line>:<col>` triple, `b7d383a` added the location-less reduction, and `f5e89f4` split `diagnostics.md` into `diagnostic-shape.md` verbatim. No later commit introduced the defect.

## Solution Space

**Shape:** multiple

### Option A — Declare the rendered line display-only; route machine consumers to `details.diagnostics`

- **Approach:** In *Serialised content format*, state that the `content` line format is human-display-only and that machine consumers (LSP integrations, test harnesses, any tooling needing the location) MUST read `details.diagnostics[i].{file, range}` rather than parse the rendered string. Tighten the existing "for downstream consumers … that want the typed shape" sentence from optional availability to a normative directive.
- **Spec edits:** Add one or two sentences to the *Serialised content format* paragraph in `diagnostic-shape.md` immediately after the line-format definition; no change to the rendered format itself.
- **Pros:** Aligns with the existing design, which already separates the renderer-input `content` string from the typed `details.diagnostics` array; cheapest edit; no new escaping grammar to specify, test, or get wrong.
- **Cons:** Leaves the rendered line technically ambiguous for any tool that ignores the directive.
- **Risks:** None of consequence — the structured channel already exists and is unaffected.

### Option B — Pin a colon-robust delimiter or escaping scheme for the rendered line

- **Approach:** Define an escaping/quoting rule for the `<file>` token (e.g. quote the path, or use a separator that cannot appear in a path) so the rendered `content` line is itself unambiguously parseable even when the path contains `:`.
- **Spec edits:** Add an escaping grammar to *Serialised content format*, plus the corresponding decode rule, and define behaviour for the related-site and file-only variants.
- **Pros:** Makes the rendered string a sound machine-readable format in its own right.
- **Cons:** Adds parsing/escaping complexity to a string whose stated role is renderer input (display); there is no identified consumer that needs to parse the rendered line rather than read `details.diagnostics`; larger normative surface and more test obligations.
- **Risks:** Escaping rules invite their own edge cases (paths containing the quote/escape character), reintroducing ambiguity the option is meant to remove.

### Recommendation

Adopt Option A. The spec already treats `content` as renderer input and exposes the typed `Diagnostic[]` for tooling, so promoting that to an explicit display-only declaration plus a "machine consumers MUST read `details.diagnostics`" directive closes the gap with a one-sentence edit and no new format machinery. Edge case to watch: the directive must cover the related-site lines and the file-only / location-less reductions too, since all reuse the same colon-delimited triple.

## Consequence

**Severity:** advisory

A downstream tool or test harness that parses the rendered `content` line on `:` mis-extracts the file/line/col triple for any colon-bearing (Windows) absolute path, yielding a wrong location. A working system is still achievable because `details.diagnostics` carries the typed shape, but the rendered format silently overstates its own parseability and the spec gives no rule steering consumers to the structured channel.
