## Finding

`V9f`'s **Adds** field describes the materialised `ToolDefinition.label` derivation as "the loom file's basename with interior hyphens preserved and only the leading character capitalised (`code-review.loom` → `"Code-review"`)". The prose enumerates three operations — take the basename, preserve interior hyphens, capitalise the leading character — none of which removes the `.loom` extension. Standard `path.basename("code-review.loom")` returns `"code-review.loom"` (the extension is only dropped when a suffix argument is supplied), so an implementer translating the three listed operations directly would produce `label: "Code-review.loom"`. The worked example in the same parenthetical (`→ "Code-review"`) shows the extension stripped, contradicting the literal operation list.

The owning spec text (`extension-bootstrap-and-per-loom.md` §Per-loom registration, `label` field derivation) uses the same colloquial "basename" wording but pins the intent with two worked examples (`summarise.loom` → `"Summarise"`, `code-review.loom` → `"Code-review"`), both showing the extension dropped. The intended behaviour is therefore unambiguous at the spec; the gap is that `V9f`'s prose lists the stem-extraction operations explicitly while silently omitting the extension-stripping step that the example depends on.

The same omission is present in the paired `V9f-T` leaf, whose Tests bullet asserts `code-review.loom` yields `label: "Code-review"` with the qualifier "(interior hyphen preserved, leading character capitalised)" — again no extension-stripping statement, though the literal expected value pins the result.

## Issue introduction

**Introducing commit:** `cfd76f7` — "pi-loom plan: resolve \"Per-loom registration ToolDefinition field-derivation MUSTs unnamed by any leaf\"" (Thomas Andersen, 2026-06-12)

The `ToolDefinition.label` derivation clause — Adds prose plus the `V9f-T` Tests bullet — was added in `cfd76f7`, and the extension-stripping omission was present from that authoring (the operation list never named the strip step, while the worked example showed it). The later `8bd82be` reworded the adjacent snapshot/restore clause only and did not touch the basename prose. The defect therefore entered with `cfd76f7` and has persisted unchanged since.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V9f-tool-registration-lifetime.md`, in the **Adds** field, change the derivation description so the extension-stripping step is stated alongside the other operations. Replace "the loom file's basename with interior hyphens preserved and only the leading character capitalised" with "the loom file's base name with the `.loom` extension stripped, interior hyphens preserved, and only the leading character capitalised". Leave the `(code-review.loom → "Code-review")` example as-is.

Reconcile the paired `docs/plan_topics/V9f-T-tool-registration-lifetime.md` Tests bullet identically: its parenthetical qualifier "(interior hyphen preserved, leading character capitalised)" should also name the extension-stripping step, so the two leaves describe the same derivation (paired-leaf reconciliation per `conventions.md`). The literal expected values (`"Code-review"`, `"Loom typed-query response"`) are correct and stay unchanged.

The spec page is read-only for this fix: its two worked examples already make the intent unambiguous, and no spec rule is wrong — only the plan's operation list is incomplete.

## Consequence

**Severity:** advisory

An implementer following `V9f`'s three-operation prose literally with `path.basename` produces `"Code-review.loom"` instead of `"Code-review"`. The colocated example and the unambiguous `V9f-T` assertion (`code-review.loom` → `"Code-review"`) both pin the correct value, so a wrong implementation reddens its own test rather than shipping; the cost is a wasted iteration and reader friction, not a shipped defect.
