## Problem

In the §7 "Identifier-shaped" sub-rule of `placeholder-rendering-b.md`, `<slash-name>` and `<slug>` are listed among placeholders "rendered unquoted in identifier form per [Lexical — Identifiers]". The Lexical identifier grammar is `[A-Za-z_][A-Za-z0-9_]*`, which forbids `-` and a leading digit, but both placeholders carry values that grammar cannot represent: a slash name matches the discovery regex `^[a-z0-9][a-z0-9_-]*$` (admitting `-` and a leading digit, e.g. `code-review`, `2fa`), and a `<slug>` is a 16-lowercase-hex schema slug that may begin with a digit. "Render in identifier form" is therefore undefined for these values — an implementer cannot tell whether to render verbatim, normalise to fit the grammar, or treat the value as an error. The sub-rule already carves out the analogous `<uuid>`/`<invocation-id>` (hex form) and `<callee>` cases but leaves these two without a matching carve-out.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** f930818 — pi-loom spec: resolve "placeholder rendering exemption open-ended" (2026-05-07, Thomas Andersen)
**History:** The §7 "Identifier-shaped" sub-rule was authored in f930818, which already binned `<slash-name>` and `<slug>` into "rendered unquoted in identifier form per [Lexical — Identifiers]" while carving out `<uuid>`/`<invocation-id>` (hex form) and `<callee>` but not these two. The sub-rule later moved verbatim into `placeholder-rendering-b.md` at f5e89f4 (2026-06-04) during the diagnostics-page size-cap split, which only relocated the text. The slash-name discovery regex the rule contradicts predated f930818, so the defect was present from the sub-rule's first commit and was never reconciled against the slash-name grammar.

## Solution approach

Narrow the §7 "Identifier-shaped" sub-rule to carve `<slash-name>` and `<slug>` out of the "identifier form per [Lexical — Identifiers]" clause, mirroring the existing `<uuid>`/`<invocation-id>` carve-out, stating that they render verbatim with no identifier-grammar normalisation. Ground the verbatim values in their source grammars — `<slash-name>` as the registered slash name matching `^[a-z0-9][a-z0-9_-]*$`, `<slug>` as the literal 16-lowercase-hex schema slug — and keep `<slash-name>` consistent with the existing `<callee>` clause's leading-`/`-from-template treatment.
