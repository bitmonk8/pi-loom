## Finding

The §Leaf format **Deps.** field description in `conventions.md` ends with: "An implementation task MUST list its paired `<id>-T` tests task here. A new leaf *may additionally need adding* to [`H5b`]'s `Deps.`; the *Transitive-completeness plan-maintenance* clause in §REQ-ID discipline below owns the exact triggers." The "may additionally need adding" phrasing is read-twice ambiguous: the author intends the epistemic reading ("it is possible the leaf qualifies, depending on the trigger"), but a reader can take the deontic reading ("adding it is optional / at the implementer's discretion").

The authoritative clause it points to is unambiguous and normative: the *Transitive-completeness plan-maintenance* clause states that whenever a new leaf carries the citing test that *closes* a coverage-matrix-mapped numbered REQ-ID, or an un-anchored normative MUST, "that leaf **MUST** be added to [`H5b`]'s `Deps.`". So the Deps-field summary softens a conditional **MUST** into a discretionary-sounding "may need", inviting an implementer who stops at the field description to treat the H5b update as optional.

This is a summary-line wording defect, not a missing obligation. The MUST is correctly stated in its owning clause, and the omission is mechanically backstopped by `H5d`'s transitive-completeness gate for any closing-leaf cell *none* of whose listed leaves reaches `H5b`'s `Deps.` — but `H5d` is a per-cell **at-least-one** test, so a secondary facet-closer omitted while a sibling facet-closer for the same cell is already present slips past it to the warn-only canary and the release-time residue inspection. The fix is to phrase the summary as a conditional MUST while keeping its deferral to the trigger clause for the exact conditions.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `73b695d`
**History:** The "may additionally need adding to [`H5b`]'s `Deps.`" phrasing was introduced in `73b695d` (2026-06-20, "pi-loom spec-plan: fix F-0588 — Deps-field trigger summary defers to authoritative clause") in `docs/plan_topics/conventions.md`; `git log -S 'may additionally need adding'` reports that commit as the sole introducer. That commit's diff replaced a prior, more declarative summary — "A new leaf whose `Tests.` bullets carry an executable REQ-ID, a citing test that *closes* a coverage-matrix-mapped numbered REQ-ID, or an un-anchored normative MUST **is additionally added to** [`H5b`]'s `Deps.` per the *Transitive-completeness plan-maintenance* clause … which owns the exact triggers." — with the current looser sentence. The F-0588 fix intentionally stopped the summary from restating the exact triggers (the prior restatement listed "an executable REQ-ID" among them, mismatching the authoritative clause), correctly deferring to the owning clause; in doing so it swapped the declarative "is additionally added to" for the discretionary-sounding "may additionally need adding to", which is the ambiguity this finding records.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md`, rewrite the final sentence of the **Deps.** field description so the H5b obligation reads as a conditional MUST while still deferring to the *Transitive-completeness plan-maintenance* clause for the exact triggers. Replace:

> A new leaf may additionally need adding to [`H5b`](./H5b-warn-only-canary.md)'s `Deps.`; the *Transitive-completeness plan-maintenance* clause in §REQ-ID discipline below owns the exact triggers.

with text to the effect of:

> A new leaf that meets the *Transitive-completeness plan-maintenance* triggers in §REQ-ID discipline below MUST additionally be added to [`H5b`](./H5b-warn-only-canary.md)'s `Deps.`; that clause owns the exact triggers.

Keep the deferral intact — do **not** re-enumerate the trigger conditions in the Deps-field summary (the F-0588 fix removed that restatement precisely because it had drifted from the owning clause); the §REQ-ID discipline clause remains the single source of the exact triggers. The edit is confined to this one sentence in `conventions.md`; `H5b`, `H5d`, and `H6a` are not touched.

## Consequence

**Severity:** advisory

An implementer who reads only the Deps-field summary may treat adding a coverage-producing leaf to `H5b`'s `Deps.` as optional and skip it, leaving the warn-only canary and the `H6a` hard-fail release gate sequenced before a coverage-producing leaf. The `H5d` per-cell at-least-one gate catches this for single-closer rows but stays green for a secondary facet-closer omitted from a multi-facet split row whose sibling is already present, so a residual omission can pre-flight coverage incomplete until release-time review.
