## Finding

The preamble paragraph of `docs/spec_topics/diagnostics/code-registry-runtime.md` (the prose before the code table) restates normative behavioral content whose canonical home is `docs/spec_topics/errors-and-results/error-model.md#runtime-panics`. Three facts are duplicated rather than merely cross-referenced:

1. The closed panic-source **count** — "loom 1.0.0 has exactly six **panic sources**" — which the `Runtime panics` section of `error-model.md` already establishes as the closed list (and which `error-model.md` itself treats as a lock-step-governed assertion).
2. The **invoke-parent routing envelope** — `Err(QueryError { kind: "invoke_infra", cause: "panic", ... })` — whose authoritative statement is the `Runtime panics` "Panics surface to the loom's caller as" list and the per-cause caller-surface table in `error-model.md`.
3. The **delivery channels** — "slash-command system note for top-level looms; … to an `invoke` parent" — also owned by the same `error-model.md` section.

The preamble even acknowledges the split by linking to `[Errors and Results — Runtime panics]`, yet still re-asserts the content inline instead of confining itself to registry-specific material (what each code means and where its row lives). The registry page is the right home for code-to-row binding and message templates; it is not the home for the closed panic count or the cross-surface routing/delivery contract. Because the duplicated facts are normative and the count is explicitly an evolving, lock-step-governed value, the two pages can drift independently — the registry preamble would have to be hand-updated every time the panic catalogue or the routing envelope changes on its canonical page.

## Issue introduction
**Verdict:** single introducing commit
**Introducing commits:** `839db71` — "pi-loom spec: resolve 'Diagnostic codes are declared as namespaces but never assigned to specific errors'" (2026-05-04)
**History:** The diagnostic code registry was authored at `839db71`, and that commit planted the preamble in its restating form — the same paragraph then read "V1 has exactly five panic sources. Each surfaces through the channels defined in [Errors and Results — Runtime panics] (slash-command system note for top-level looms; `Err(QueryError { kind: "invoke_failure", reason: "panic", ... })` to an `invoke` parent)." The inline restatement of the closed panic count plus the routing/delivery-channel rules has therefore been present since the registry first existed, while the canonical `Runtime panics` section in the Errors-and-Results corpus already owned that content. The count later changed from "five" to "six" at `833d8d5` (2026-05-05, invoke-depth added to the catalogue), and the envelope was renamed to `kind: "invoke_infra", cause: "panic"`; commit `47fe3cd` (2026-06-05) replaced the broken "(the rows below tagged as panics)" qualifier with the closed-list cross-reference but left the inline restatements in place. Those are evolutions of the same duplicated prose, not the point of introduction. The defect entered with the registry itself at `839db71`.

## Solution Space

**Shape:** single

### Recommendation

Reduce the `code-registry-runtime.md` preamble to registry-specific content: identify which rows correspond to the closed panic-source list (by matching each row's *Code* cell against the canonical list) and state where each code's row lives, but do not re-assert the panic count, the `Err(QueryError { kind: "invoke_infra", cause: "panic", ... })` routing envelope, or the slash-note/invoke-parent delivery channels. Replace those with a single cross-reference sentence pointing at `../errors-and-results/error-model.md#runtime-panics` as the authoritative source for the closed count and the cross-surface routing/delivery contract.

Edge cases for the fixer:
- Keep the `loom/runtime/internal-error` runtime-defect-surface framing and the non-panic delivery-failure code descriptions — those are registry-specific and are not duplicated on `error-model.md`.
- Do not introduce a new routing-envelope spelling while trimming; the canonical page is the only place the `kind`/`cause` shape should be asserted (this also avoids re-stamping the `QueryError` vs `InvokeInfraError` spelling that the related finding flags).
- Preserve the existing link target `../errors-and-results/error-model.md#runtime-panics` (the deep anchor) rather than the bare `../errors-and-results.md` hub used elsewhere in the same paragraph.

## Consequence

**Severity:** advisory

No implementer diverges today — the registry preamble cites the canonical page, so both surfaces agree. The cost is a maintenance hazard: the panic count, the `cause: "panic"` envelope, and the delivery-channel rules now live in two places with no lock-step pin, so a future change on `error-model.md` (e.g. adding a panic source or altering the routing envelope) can silently leave the registry preamble stale.
