## Finding

The diagnostic-registry column legend in `diagnostic-shape.md` (`#column-legend`) defines the `Phase` column as exactly one value drawn from a closed set, phrased with an exclusive "or": *"`Phase` identifies which pipeline stage emits the diagnostic: `lex` … `parse` … `type` … `load` … or `runtime`."* The grammar admits no comma-separated or multi-valued form.

The `loom/load/invoke-path-escape` row in `code-registry-load.md` violates that grammar: its `Phase` cell reads `load, runtime`. This is the only row across all four sharded registry tables (`parse`, `load`, `runtime`, `host`) that carries two phase values; every other row carries exactly one. The row's `Trigger` prose confirms the dual value is intentional rather than a typo — the code fires both at load time (the static `invoke`/`tools:` containment walk) and at runtime (the open re-check for a symlink that resolves outside every active discovery root at invocation time, even if it was inside at load time).

Because the legend is the normative definition of the column and is silent on multi-valued cells, an implementer cannot tell whether `load, runtime` is (a) a sanctioned multi-phase form the legend simply forgot to describe, (b) shorthand for two distinct diagnostic codes, or (c) an authoring slip. The ambiguity propagates into the test layer: a conformance assertion does not know whether to expect `Phase == "load"`, `Phase == "runtime"`, or the literal string `"load, runtime"`, and whether other rows are permitted to adopt the multi-phase form. The legend and the row must be reconciled so the column's value domain is unambiguous.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** c8b8790 — pi-loom plan: resolve "invoke-path-escape parse-only TOCTOU" (2026-05-05, Thomas Andersen)
**History:** The `loom/load/invoke-path-escape` row originally carried the single `Phase` value `load`. Commit c8b8790 added the runtime open re-check to its `Trigger` and changed the cell to `load, runtime`. At that same commit the column legend already framed `Phase` as a single value drawn from the set with an exclusive "or", so the divergence was introduced by adding the second phase value to one row without reconciling the legend, and has persisted (across the later `81bf52a` page split and the `a37d4ef` legend relocation) ever since.

## Solution Space

**Shape:** multiple

### Option A — Make `Phase` an explicit one-or-more-values column

**Approach:** Amend the legend sentence to define `Phase` as one or more values from the closed set, written as a comma-separated list when a code is emitted from more than one pipeline stage, and state that a row's `Phase` enumerates every stage that may emit that code. Leave the `loom/load/invoke-path-escape` row as `load, runtime`.
**Spec edits:** In `diagnostic-shape.md#column-legend`, replace "identifies which pipeline stage emits the diagnostic: … or `runtime`" with a form that closes on "one or more of `lex` / `parse` / `type` / `load` / `runtime`, comma-separated in source-stage order when a single code is emitted from multiple stages". Add one clause pinning the test-comparison rule (the `Phase` cell is the set of permitted emitting stages; an asserting test matches if the code's observed emit stage is a member).
**Pros:** Preserves the single diagnostic code and its stable identifier (no DIAG-3 rename/split); matches the `Trigger`'s already-documented dual-stage reality; one-line legend change plus a test-rule clause.
**Cons:** Introduces a multi-valued column form that the `V7b` gate must now parse and compare set-membership-style rather than string-equality.
**Risks:** If the gate's comparison rule is left implicit, the closing gate could still diverge; the edit must pin the membership semantics explicitly.

### Option B — Split into two single-phase diagnostic codes

**Approach:** Keep `Phase` single-valued and give the runtime open re-check its own code (e.g. a `loom/runtime/invoke-path-escape`), leaving the load-time containment check as `loom/load/invoke-path-escape` with `Phase: load`.
**Spec edits:** Add a new row to `code-registry-runtime.md` for the runtime arm, move the runtime re-check `Trigger` prose to it, and trim the `load` row's `Trigger` back to the load-time walk. Cross-link the two rows.
**Pros:** Honours the existing single-valued legend with no grammar change; each code maps to exactly one pipeline stage, simplifying gate comparison.
**Cons:** Coins a new diagnostic code (wider surface, new namespace entry, new test); fragments one conceptual defect (path escapes its discovery root) across two codes that callers must treat together.
**Risks:** Two codes for one author-facing condition can confuse the recovery `Hint` and the `Message` template; downstream consumers keying on a single code break.

### Recommendation

Adopt **Option A**. The `invoke-path-escape` `Trigger` already describes one coherent condition (a callee path escaping every active discovery root) that is genuinely checked at two stages, so a single code with a multi-valued `Phase` reflects reality and avoids a wire-contract-expanding new code under the DIAG-3 stability regime. The implementer must watch one edge case: the `V7b` closing gate's `Phase` comparison must be specified as set membership, not string equality, so the multi-valued cell is checkable.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the encoding and test semantics of the `Phase` column for this row (single value vs. comma-joined string vs. two separate codes), and on whether other rows may legally adopt a multi-phase form. The `V7b` closing gate, which reconciles test-asserted codes against the registry, has no normative rule telling it how to compare a multi-valued `Phase` cell.
