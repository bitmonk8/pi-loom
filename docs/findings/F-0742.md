## Finding

`docs/spec_topics/discovery/package-and-settings.md` carries the operator-tunable rationale for the package-walk caps twice within the same page. DISC-6 (the §"Package discovery" paragraph) opens with: *"Package counts and walk time vary by install — monorepos and global-package-heavy setups can exceed the default caps without the user having any other knob to relieve the pressure — so the bounds are operator-tunable rather than hardcoded. The defaults are *upper bounds*, not target performance …"*. A few paragraphs later, the §"Settings file reads" → "Keys read" bullet for `looms.scanPackages` ends with a near-verbatim restatement: *"The three `looms.scanPackages*` keys below are operator-tunable in loom 1.0 rather than hardcoded because package counts and walk time vary by install; the defaults are *upper bounds*, not target performance."*

The second sentence carries no obligation that DISC-6 does not already pin: tunability is encoded by the keys themselves (which the bullet enumerates immediately), the upper-bound framing is already in DISC-6, and "monorepos / global-package-heavy" is the exact same justification. Keeping both copies is editorial duplication that invites future drift — a later edit to one phrasing can leave the other stale, and the GOV-8 "single defining site" expectation argues for one canonical rationale paragraph.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 52219c4 (2026-05-06) — "Address all 41 spec-review findings"
**History:** `git log -G 'three .looms.scanPackages.* keys below are operator-tunable'` shows the duplicated trailing sentence on the `looms.scanPackages` bullet first appears in 52219c4, alongside the new DISC-6 rationale clause ("Package counts and walk time vary by install … so the bounds are operator-tunable rather than hardcoded"). The same commit's diff against `spec_topics/discovery.md` shows both insertions side-by-side: lines 102 (DISC-6 rewrite) and 111 (Keys-read bullet) gained the rationale text in a single change to "Address all 41 spec-review findings". The earlier baseline pre-52219c4 had neither sentence — the bullet read only *"`looms.scanPackages` — boolean, default `true`. When `false`, the package-discovery walk is skipped wholesale …"*. Subsequent edits (a6d82db's prose `V1` → `loom 1.0` rename; f5e89f4's split into the topic subdirectory layout) carried both copies forward unchanged. The defect is a single-commit over-edit: the spec-review fix that authored the DISC-6 rationale also restated it on the settings bullet, instead of citing DISC-6.

## Solution Space

**Shape:** single

### Recommendation

Delete the trailing rationale sentence from the `looms.scanPackages` bullet in §"Settings file reads" → "Keys read", leaving only the functional content: *"`looms.scanPackages` — boolean, default `true`. When `false`, the package-discovery walk is skipped wholesale (see [Package discovery](#package-discovery) → 'Edge cases')."* DISC-6 already owns the operator-tunable / upper-bound rationale; no cross-link is needed because the surviving sentence already forward-links to §"Package discovery". Do not edit DISC-6 itself, and do not relocate the rationale.

Edge cases to watch:

- The two sibling bullets (`looms.scanPackagesMaxFiles`, `looms.scanPackagesTimeoutMs`) already describe themselves only in terms of their normative bound and trip diagnostic; leave them untouched.
- The deletion does not affect any anchor (`disc-6`, `disc-7`, the per-key headings) — those live on the surviving paragraphs.
- Grep `docs/` for "operator-tunable in loom 1.0 rather than hardcoded" before and after the edit to confirm only the one duplicated occurrence is removed; DISC-6's "operator-tunable rather than hardcoded" wording is intentionally distinct and must remain.

## Consequence

**Severity:** cosmetic

No implementer can be misled today: both copies of the rationale agree. The risk is drift — a future edit tightening one phrasing (e.g. raising a default, restating the upper-bound contract, or migrating away from "operator-tunable" wording) leaves the other copy quietly out of sync, and a reader hitting the stale one first will conclude the spec contradicts itself.
