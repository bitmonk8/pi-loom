## Problem

`runtime-value-model.md` carries a bold-led **Reference encoding (non-normative).** paragraph sandwiched between the normative representation table and the **Equality (`==`)** paragraph. It states two concrete reference-implementation shapes — the enum tag as a non-enumerable `__loomEnum` string property and `Result<T, E>` as `{ ok: true, value: T }` / `{ ok: false, error: E }` — which are implementation details not reachable from loom code, absent from any wire schema, and changeable without a spec revision. Placing them inline beside the normative rows invites readers to pin an implementation to them, re-introducing the over-prescription those rows were already rewritten to remove. The paragraph also ends with a conditional MUST addressed to a future spec editor ("If a future host-interop surface ever exposes a live `Result` or enum value to JS … this section MUST be revisited before that surface ships") — an action item, not an implementer obligation, and a model-level deferral that is invisible from the Future Considerations buckets where a reader would look for it.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `0a7884a` — "pi-loom spec: resolve \"JS representation of Result and enum brand is over-prescribed\"" (2026-05-04)
**History:** Before `0a7884a`, the concrete JS shapes (the `__loomEnum` non-enumerable brand and the `{ ok: true, value }` / `{ ok: false, error }` `Result` object) were stated inline in the normative representation table rows on `runtime-value-model.md`. That commit resolved an over-prescription finding by demoting those shapes out of the normative rows — but planted them as a new standalone **Reference encoding (non-normative).** paragraph on the same page and added the conditional-MUST future-revision reminder inline in that paragraph, introducing both halves of this defect in one edit. `git log -S` on both the `Reference encoding (non-normative)` and `MUST be revisited before that surface ships` tokens shows `0a7884a` as the sole spec-corpus content commit; the only later commit to touch the paragraph (`31ff060`) relocated the entire corpus into `docs/` without altering the text. (Subsequent `git log -S` hits are spec-review tooling/store commits, not spec-content edits.)

## Solution approach

Delete the **Reference encoding (non-normative).** paragraph from `runtime-value-model.md` and redistribute its two halves. Add a non-normative **Implementation hint (non-normative).** bullet under the **Runtime** list on `implementation-notes.md` (alongside the existing Chevrotain and AJV hints) carrying the concrete `__loomEnum` and `{ ok, value }` / `{ ok, error }` reference shapes and their implementation-detail qualifier. Add a bullet under **Model-level changes (no loom 1.0 seam expected)** on `future-considerations/model-changes-and-non-goals.md` recording that exposing a live `Result` or enum value to JS would change the runtime value model and is not anticipated by loom 1.0, dropping the conditional-MUST framing.

## Solution constraints

- Out of scope: the normative representation-table rows on `runtime-value-model.md`, including the observable "tag MUST NOT appear in JSON output" contract — only the concrete reference-shape detail moves.
- Out of scope: the **Equality (`==`)** paragraph (anchor `id="equality"`) that follows the deleted paragraph.
