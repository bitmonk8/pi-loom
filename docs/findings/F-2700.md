## Problem

The spec mandates a bidirectional prompt-mode cancellation linkage: when `loomAbort.abort()` fires during an in-flight prompt-mode user turn, the runtime MUST invoke the unwrapped, Pi-supplied `abort()` on the raw `ExtensionCommandContext` (not the synthesised tool-execution `ExtensionContext` whose `ctx.abort()` wrapper itself re-calls `loomAbort.abort()`) to tear down the user run and unblock `await ctx.waitForIdle()`. This is pinned in `conversation-drive.md` ┬¦*Hang handling* and `cancellation.md` ┬¦*Forwarding into `loomAbort`*. No plan leaf asserts this teardown direction: `V17a` covers forwarding *into* `loomAbort` and explicitly excludes the reverse propagation (`CNCL-4`); `V9c`'s Adds and its Tests (`PIC-2`, `PIC-17`, `PIC-18`, `PIC-53`) cover only cancel-forward; `V9n` covers the post-resolution short-circuit but presupposes the wait already resolved. An implementer can ship a green `V9c` whose driver omits or mis-wires the reverse teardown, surfacing only as a hung prompt-mode cancel.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (`pi-loom plan: build/update plan for spec.md + review`, 2026-06-10)
**History:** `V9c` was authored in `c6a664e`, the initial plan build for `spec.md`, and neither it nor any other leaf has ever carried an assertion or Adds mention of the bidirectional prompt-mode teardown. `git log -G`/`-S` over `docs/plan_topics/*.md` for the teardown clause (`unwrapped ŌĆ” abort`, `bidirectional`, `loomAbort.abort()`) returns only a scaffold-reset commit and spec-side commits ŌĆö never a plan-leaf assertion. The spec obligation (`conversation-drive.md` ┬¦*Hang handling* and `cancellation.md` ┬¦*Forwarding into `loomAbort`*) predates and outlives `V9c`; the gap is an omission present since the leaf's inception, not a regression a later edit introduced.

## Solution approach

Extend `V9c`'s **Adds** (`docs/plan_topics/V9c-conversation-drive.md`) to cover the reverse teardown alongside the existing `pi.on` cancel-forward: a `loomAbort.abort()` during an in-flight prompt-mode user turn invokes the unwrapped raw-`ctx` `abort()` to unblock `await ctx.waitForIdle()`. Add a paired citing **Tests** assertion to `V9c` and `V9c-T` (`docs/plan_topics/V9c-T-conversation-drive.md`) that fires `loomAbort.abort()` with a turn in flight and asserts the unwrapped raw-`ctx` `abort()` drives teardown (not the synthesised tool-execution wrapper) and that `await ctx.waitForIdle()` subsequently resolves, citing `conversation-drive.md` ┬¦*Hang handling* and `cancellation.md` ┬¦*Forwarding into `loomAbort`* as spec source.

## Solution constraints

- Out of scope: forwarding *into* `loomAbort` across the three entry points and `CNCL-1..6` are owned by `V17a`; only the reverse teardown direction belongs to this fix.
- The spec is read-only for this fix.
