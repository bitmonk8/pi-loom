## Problem

The `⊑` "Operational definition" in `docs/spec_topics/type-system.md` treats AJV-validation as a *necessary* condition for compatibility, and `docs/spec_topics/runtime-value-model.md`'s inbound *Wire-name translation* bullet rebuilds loom-side values from JSON validated "after AJV validation against the lowered schema". Both arguments hold only if AJV runs in a specific mode at every inbound boundary — no value coercion, no default-filling, and a fixed JSON Schema draft — properties pinned by PIC-11 and the Schema Subset page but cited by neither consumer page. A reader of either page has no path from the AJV reference to the configuration that makes the necessary-condition / inbound-rebuild claim true, and an implementation wiring AJV with `coerceTypes: true` or a different draft would silently break both the parser's delegation and the inbound walk's reliance on a structurally faithful validated value.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `0226b80` (2026-05-05, "pi-loom plan: resolve \"V15c references an undefined compatibility relation\"") for type-system.md; `fecb504` ("Split spec.md and plan.md into per-topic / per-phase files") for runtime-value-model.md
**History:** The type-system "Operational definition" paragraph was authored in `0226b80` as the section that first introduced `T₁ ⊑ T₂` and the "AJV-validates" framing; the bare "AJV-validates against the lowering of `T₂`" wording, with no pointer to the validator-mode pin, has been present from that commit. `git log -S "AJV-validates against the lowering" -- docs/spec_topics/type-system.md` yields exactly two later edits (`9a4e078` strengthened the framing to a necessary condition and added the nominal-vs-AJV note; `b93334a` adjusted "named schemas" → "object-schema named types" for TYPE-11), neither of which touched the missing citation. The runtime-value-model `after AJV validation against the lowered schema` clause traces back through `git log -S "after AJV validation"` to the original split commit `fecb504`; the only subsequent edit on the line (`7f217a8`, "Language-core hidden assumptions: enum-tag sidecar, AJV-config silence, Markdown claim") added the *Named-enum positions* sidecar pointer but, despite the commit subject naming "AJV-config silence", did not add the missing citation to PIC-11 / Schema Subset. The defect is therefore a single durable omission that has co-existed with the AJV references on both pages from the moment the pages were first written, not a regression introduced by a later edit.

## Solution approach

Add forward-link cross-references from both consumer sentences to the pins they presuppose: extend type-system.md's "Operational definition" AJV-validation sentence and runtime-value-model.md's inbound *Wire-name translation* bullet so each points to [PIC-11](./pi-integration-contract/host-interfaces-services.md#pic-11) (validator behaviour) and [Schema Subset — Draft](./schema-subset.md) (the pinned draft). The links sit alongside the existing [Schema Subset — Lowering Algorithm](./schema-subset.md#lowering-algorithm) reference and route the reader from the AJV claim to the contract that pins the validator mode and draft those claims depend on.

## Solution constraints

- The edits are citation-only — do not restate PIC-11's no-coercion / no-default-filling obligations or Schema Subset's Draft 2020-12 pin inline; those remain single-sourced at their defining sites.
