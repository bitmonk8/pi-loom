## Finding

A single concept — the informational note the discovery-root watcher emits when a `.loom` / `.warp` file is added or removed within a debounce window — is named three non-equivalent ways inside the `V9b` leaf, and the divergence also propagates to the paired `V9b-T` test leaf:

- **Adds** (`V9b`): "the step-5 structural-change `loom-system-note`".
- **Tests** bullet header (`V9b` and `V9b-T`): "registration-steps.md — structural-change watcher note".
- **Ships when** (`V9b`): "suppresses the structural-change `loom-system-note` on an empty debounce window".

The spec's canonical short name for this note is **"structural-change note"** — used verbatim in `package-and-settings.md` ("the informational structural-change note"), `runtime-event-channel.md` ("watcher structural-change notes"), and `session-shutdown-semantics.md` ("the structural-change note"). `loom-system-note` is a distinct concept in the spec: it is the Pi `customType` / message-renderer channel (`pi.registerMessageRenderer("loom-system-note", …)`, per `diagnostics/diagnostic-shape.md`) on which many different notes are delivered. Naming the note "structural-change `loom-system-note`" conflates the note with the channel it rides on; "structural-change watcher note" introduces a third spelling that matches neither the channel name nor the canonical note name.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** b722ce7 — pi-loom plan: resolve "registration-steps.md has normative MUSTs but no coverage-matrix row" (2026-06-17, Thomas Andersen)
**History:** All three divergent forms entered in commit b722ce7, the F-0151 coverage-matrix fix that added the structural-change Tests bullet to `V9b` and `V9b-T` and the matching Adds / Ships-when mentions in `V9b`. The bullet was authored with "structural-change watcher note" in the Tests headers and "structural-change `loom-system-note`" in Adds / Ships-when, neither matching the spec's pre-existing canonical "structural-change note"; the divergence was introduced atomically rather than accreting across commits.

## Solution Space

**Shape:** single

### Recommendation

Adopt the spec's canonical "structural-change note" for the note's name across the four sites, reserving `loom-system-note` for references to the delivery channel / `customType`:

- `V9b` **Adds.** — replace `the step-5 structural-change `loom-system-note`` with `the step-5 structural-change note` (the note is emitted *on* the `loom-system-note` channel; if the channel reference is wanted, phrase it as "the step-5 structural-change note (emitted on the `loom-system-note` channel)").
- `V9b` **Tests** bullet header — replace `structural-change watcher note` with `structural-change note` in the link text `[registration-steps.md — structural-change watcher note](…)`. A `(watcher)` parenthetical may be retained if the watcher provenance is worth surfacing.
- `V9b` **Ships when.** — replace `the structural-change `loom-system-note`` with `the structural-change note`.
- `V9b-T` **Tests** bullet header — apply the same `structural-change watcher note` → `structural-change note` replacement so the test leaf matches its implementation leaf.

Leave every reference to the `loom-system-note` channel / `customType` itself (e.g. "emits no `loom-system-note`", the renderer registration) unchanged — those correctly name the channel, not the note.

## Consequence

**Severity:** advisory

An implementer reading the leaf sees three names for one note and may infer "loom-system-note" is the note's proper name rather than the delivery channel's `customType`, or treat "watcher note" as a separate surface. The behaviour is unambiguous from the worked `content` literal, so no two implementers would diverge on what to build; the cost is comprehension friction and a terminology drift away from the spec's canonical "structural-change note".
