## Problem

The language-core normative prose names **AJV** — a specific npm package
— as the oracle for what "validates against the lowered schema" means:
`type-system.md` defines `T₁ ⊑ T₂` in terms of "AJV-validates" (Operational
definition, Structural cases, Unresolvable operands), `runtime-value-model.md`
gates wire-name translation on "AJV validation", and the query, binder,
invocation, tool-calls, schema-subset, hard-ceilings, and diagnostics shards
carry the same pattern (`<ajv-summary>` placeholders, "before AJV", "AJV
admit set"). The normative validator contract is actually the abstract
`SchemaValidator` service in PIC-11 (`#pic-11`), whose behavioural contract —
together with the JSON Schema 2020-12 "validates"-semantics sentence already
pinned in `schema-subset.md` — is what `⊑` and the boundary-validation rules
depend on; AJV is only the non-normative reference implementation named in
`implementation-notes.md`. Because normative prose names AJV directly, the
meaning of "validates" silently follows the resolved AJV version rather than
the documented PIC-11 contract, so a version upgrade or a permitted validator
swap shifts the language semantics with no spec edit.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `fae85d3` (2026-05-04 11:20 +0200, "Tighten spec for implementation-plan readiness") — first introduced "AJV" by name into the wire-name translation prose and the binder-`args` validation prose, and explicitly described AJV configuration ("AJV v8 with `strict: false` and `allErrors: true`") as a normative item; `d826fae` (2026-05-04 17:20 +0200, "Schema cache ownership is unspecified, inviting a module-level singleton") and `27583dd` (2026-05-04 17:25 +0200, "Implementation toolkit prescribed in place of behavioural contract") — together introduced the abstract `SchemaValidator` seam and demoted the AJV-flag enumeration to a non-normative implementation hint, but did not sweep the existing AJV-by-name occurrences in normative prose; `0226b80` (2026-05-05 16:23 +0200, "V15c references an undefined compatibility relation") — authored the **Type compatibility** section in `type-system.md` against the library name ("AJV-validates", "the AJV reading is the safety net", "runtime AJV check is the safety net") roughly a day *after* the `SchemaValidator` abstraction landed, deepening the divergence.
**History:** `git log -S 'AJV-validates' --reverse --all` and `git log -G 'AJV' --follow -- docs/spec_topics/runtime-value-model.md docs/spec_topics/type-system.md` show the defect's lineage. The original spec tightening pass (`fae85d3`) wrote AJV directly into normative prose because no validator abstraction existed yet. Six hours later, `d826fae` / `27583dd` introduced the `SchemaValidator` service and moved the AJV name to a non-normative hint, *but only for the new bullets they added in `implementation-notes.md`*; the AJV-by-name occurrences in the wire-name pass were not retargeted. The next day, `0226b80` authored a brand-new normative section (**Type compatibility**) against the library name rather than the new abstraction. Subsequent topic-split and resolve commits (`fecb504`, `7f217a8`, `9a4e078`, `b93334a`, …) propagated and elaborated the AJV phrasing through binder/query/schema-subset/hard-ceilings shards without ever reconciling it against `SchemaValidator`/PIC-11. The defect is the *interaction*: the introduction of the abstraction did not include a corpus-wide sweep, and every later edit honoured the dominant in-corpus convention (name AJV) rather than the newer, narrower abstraction.

## Solution approach

Rewrite the normative AJV-by-name references across the corpus to point at
the abstract `SchemaValidator` contract (`#pic-11`) and the JSON Schema
2020-12 "validates"-semantics already pinned in `schema-subset.md` — covering
`type-system.md` (Operational definition, Structural cases, Unresolvable
operands), `runtime-value-model.md` (Wire-name translation), `schema-subset.md`,
the query, binder, invocation, tool-calls, hard-ceilings, and diagnostics
shards, `frontmatter-fields-a.md`, `glossary.md`, and
`errors-and-results/queryerror-variants.md`. Rename the `<ajv-summary>`
placeholder and its render-sites to a validator-neutral spelling. Leave the
non-normative reference-implementation hint in `implementation-notes.md`
intact as the single admission that AJV is what loom 1.0 ships.

## Solution constraints

- Out of scope: the non-normative reference-implementation hint in `implementation-notes.md` and the `ajv` / `ajv-formats` dependency declarations in `H1a-scaffold-and-toolchain.md` — these are the legitimate package-level pins and must not be swept.
- The JSON Schema 2020-12 "validates"-semantics sentence in `schema-subset.md` must remain in place as the authoritative semantics anchor.
