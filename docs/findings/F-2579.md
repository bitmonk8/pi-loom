## Problem

INV-4 (`id="inv-4"`) asserts the invoke-depth counter crosses the subagent `customTools` / `AgentSession` boundary "carried by the invoke trampoline" but names no concrete carrier — the term "invoke trampoline" appears nowhere else in the corpus, and the only per-invocation runtime structure the spec defines, the `ActiveInvocationRegistry` entry, carries no depth field. A subagent-mode child runs in a freshly spawned, detached `AgentSession`, and two concurrent subagent siblings spawned from the same parent must each see the parent's depth as their starting count while remaining independent of one another. Because only the invariant ("crosses the boundary unchanged") is stated and not the propagation mechanism, two reasonable implementers can pick incompatible carriers (ambient module/extension-instance counter vs. per-spawn-threaded value) and diverge on whether the cap holds under concurrency — an ambient counter is corrupted by interleaving siblings on the event loop.

**Verdict:** introduced-by-commit
**Introducing commits:** `2b0def6` — "pi-loom spec: resolve \"invoke-chain depth-32 cap: counting origin and subagent-mode boundary semantics undefined\"" (Thu May 7 2026)
**History:** The corpus is git-tracked. `git log -S "carried by the invoke trampoline" -- spec_topics/invocation.md` and `git log --follow` resolve the defect prose to a single content-introducing commit (the later `31ff060` only relocated the file into `docs/`). Commit `2b0def6` resolved a prior review finding that the depth cap's counting origin and subagent-boundary semantics were undefined; its fix added the paragraph establishing the per-chain semantics and, in the same edit, introduced the under-specified carrier sentence ("carried by the invoke trampoline … not part of the wire-level data"). The carrier gap is therefore a regression-on-fix: the resolution pinned the behavioural answer but left the propagation mechanism — and the now-orphaned "trampoline" term — unspecified. No earlier revision contained this prose, so the defect is not present-since-inception.

## Solution approach

Rewrite INV-4's subagent-boundary paragraph to name a concrete per-invocation depth carrier and its propagation rule across the spawn boundary, replacing the orphaned "invoke trampoline" prose. Thread the parent's chain depth into each spawn by value — e.g. a depth field on the per-invocation runtime invoke-options or on the `ActiveInvocationRegistry` entry — so each spawned `AgentSession` (and each `tools:`-dispatched `.loom` / cross-file `.warp` `fn` frame) captures the parent's depth at spawn rather than reading ambient state, and state that concurrent subagent siblings each capture it independently. Resolve the orphaned "invoke trampoline" term (drop or define it).

## Solution constraints

- The depth carrier MUST remain correct when two concurrent subagent siblings spawned from the same parent interleave on the event loop — each independently sees the parent's depth as its starting count.
