## Problem

The `<construct>` placeholder in `loom/parse/unsupported-feature` is rendered by `placeholder-rendering-a.md` §3's closed 15-row token-name table, which carries no fallback rule and demands byte-identical output across conformant implementations. The trigger set actually routed to that code is strictly larger than the 15 tabled constructs: the registry row's *Trigger* column in `code-registry-parse.md` ends in "etc.", and `lexical.md` (reserved hex/octal/binary numeric literals and underscore digit separators), `schema-subset.md` (the reject-by-default JSON-Schema keyword allowlist), and `expressions.md` (field/index mutation, callback-taking higher-order array methods, `match` in interpolation) each route further constructs that have no §3 row. Because §3 carries no "any other construct" fallback and its table is GOV-7/GOV-8 spec-versioned, an implementer hitting any non-tabled trigger has no conformant rendering — byte-identicality is unobtainable and adding a row is itself a breaking change.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `839db71` (2026-05-04, "pi-loom spec: resolve 'Diagnostic codes are declared as namespaces but never assigned to specific errors'") and `61ea1a5` (2026-05-06, "pi-loom spec: resolve 'Diagnostic message placeholders have no defined rendering'")
**History:** `839db71` introduced the `loom/parse/unsupported-feature` registry row with the trigger phrase "A V1-deferred or non-Loom syntactic construct (arrow function, spread, optional chaining, `===`, bitwise op, comma op, nested template, etc.)" — the "etc." was present at first authoring, making the trigger surface verbally open from day one. Two days later, `61ea1a5` added the §3 *Syntactic-construct placeholder* rendering rule with the 15-row closed token-name table and no fallback clause; the rendering author did not reconcile the new closed table against the prior row's open "etc." nor against the existing routes from `lexical.md` (reserved numeric literals, already in place at the spec split) or `schema-subset.md` (the reject-by-default allowlist). The follow-up `f930818` (2026-05-07, "placeholder rendering exemption open-ended") tightened the closure paragraph for the eight categories overall but did not touch §3's row set or reconcile the registry row's "etc." with §3's closure. The later split into `placeholder-rendering-a.md` (`f5e89f4`, 2026-06-04) was a mechanical file split that preserved both surfaces unchanged. The defect is the interaction between two independently-correct edits: the registry row that kept its trigger open and the placeholder rule that closed its rendering surface, with no audit step that would have caught the mismatch.

## Solution approach

Add a fallback rule to `placeholder-rendering-a.md` §3 covering every construct routed to `loom/parse/unsupported-feature` that the closed 15-row token-name table does not enumerate, retaining the table for constructs that have no single source-span anchor. Rewrite the `loom/parse/unsupported-feature` *Trigger* column in `code-registry-parse.md` so its open "etc." no longer admits triggers the renderer cannot serve, and cross-link it to the §3 fallback so the two surfaces line up. Add normative §3 test vectors covering the non-tabled trigger families (reserved numeric literals, rejected JSON-Schema keywords, and a non-tabled JavaScript form).

## Solution constraints

- Out of scope per GOV-7/GOV-8: adding, retiring, or moving a placeholder, and expanding the closed §3 token-name table — non-tabled triggers are served by the fallback rule, not by new table rows.
