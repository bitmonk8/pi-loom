## Finding

The `<construct>` placeholder used by `loom/parse/unsupported-feature` is governed by the byte-identical rendering contract pinned at `placeholder-rendering-a.md` §3, which supplies a 15-row closed token-name table (`arrow function`, `spread`, `optional chaining`, `nullish coalescing`, `strict equality`, `bitwise <op>`, `comma operator`, `nested template`, `new`, `typeof`, `instanceof`, `delete`, `void`, `yield`, `await`) and no fallback rule. Two conformant implementations are required to produce byte-identical strings, with no implementation-defined tail.

The trigger set the registry row in `code-registry-parse.md` actually admits is strictly larger than 15:

- The row's own *Trigger* column lists the canonical examples and closes with "etc.", so the trigger is verbally open-ended on the registry side while the renderer side is closed.
- `lexical.md` explicitly routes reserved hex / octal / binary numeric literals and underscore digit separators to `loom/parse/unsupported-feature`; the leaf `V1b — String, number, and path literals` tests this. None of these tokens has a row in the §3 table.
- `V5d — Schema-subset reject gate` fires `loom/parse/unsupported-feature` both for each rejected JSON-Schema keyword (`pattern`, `format`, `minLength`, `multipleOf`, `oneOf`, `allOf`, `not`, `if`/`then`/`else`, etc., enumerated in `schema-subset.md`) and, by its reject-by-default allowlist semantics, for any JSON-Schema construct outside the permitted subset that is not in the enumerated list. None of these keywords has a row in the §3 table either.
- `expressions.md` §"Not supported" lists further constructs that route to this code unless a more specific code applies — field-/index-level mutation, callback-taking higher-order array methods (`map`/`filter`/`reduce`), and `match` inside `${...}` interpolation. None has a row in the §3 table.
- Genuine JavaScript forms an author might write but the spec does not call out individually — `class`, `function*` generators, regex literals, `try`/`catch`/`throw`, `do`/`while`, `for`-`of`/`for`-`in`, destructuring patterns, tagged templates — all collapse to this code with no §3 entry.

Because §3 carries no "any other construct renders as <X>" fallback and the §3 table is itself spec-versioned (GOV-7/GOV-8 per the page's "Closure" paragraph), an implementer encountering any of the above has no conformant rendering available: byte-identicality is unobtainable for these triggers, and adding a row is a breaking change. The renderer must either invent an off-spec token (drifting between implementations) or panic into `loom/runtime/internal-error`, which is wrong for a parse-time defect the author owns.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `839db71` (2026-05-04, "pi-loom spec: resolve 'Diagnostic codes are declared as namespaces but never assigned to specific errors'") and `61ea1a5` (2026-05-06, "pi-loom spec: resolve 'Diagnostic message placeholders have no defined rendering'")
**History:** `839db71` introduced the `loom/parse/unsupported-feature` registry row with the trigger phrase "A V1-deferred or non-Loom syntactic construct (arrow function, spread, optional chaining, `===`, bitwise op, comma op, nested template, etc.)" — the "etc." was present at first authoring, making the trigger surface verbally open from day one. Two days later, `61ea1a5` added the §3 *Syntactic-construct placeholder* rendering rule with the 15-row closed token-name table and no fallback clause; the rendering author did not reconcile the new closed table against the prior row's open "etc." nor against the existing routes from `lexical.md` (reserved numeric literals, already in place at the spec split) or `schema-subset.md` (the reject-by-default allowlist). The follow-up `f930818` (2026-05-07, "placeholder rendering exemption open-ended") tightened the closure paragraph for the eight categories overall but did not touch §3's row set or reconcile the registry row's "etc." with §3's closure. The later split into `placeholder-rendering-a.md` (`f5e89f4`, 2026-06-04) was a mechanical file split that preserved both surfaces unchanged. The defect is the interaction between two independently-correct edits: the registry row that kept its trigger open and the placeholder rule that closed its rendering surface, with no audit step that would have caught the mismatch.

## Solution Space

**Shape:** multiple

### Option A — Add an open-ended fallback token rule to §3

**Approach.** Append a final clause to the §3 rule: "For any other construct routed to `loom/parse/unsupported-feature` that is not enumerated in the closed table above, render `<construct>` as the construct's source-keyword form per `lexical.md` — Reserved keywords (e.g. `class`, `function*`) or, for non-keyword forms (reserved numeric-literal prefixes `0x`/`0o`/`0b`, underscore digit separator, rejected JSON-Schema keyword names, callback-taking higher-order method names like `map`), the offending source token verbatim copied per the same byte-for-byte rule as the §3 `<expr>` clause." Keep the existing 15-row table as the pinned mapping for constructs that have no single source-span anchor.

**Spec edits.** New paragraph in `placeholder-rendering-a.md` §3 after the closed table; new normative test vectors covering one reserved-numeric example, one JSON-Schema-reject example, and one non-tabled JS form (e.g. `class`). Add a sentence to the row in `code-registry-parse.md` cross-linking to the §3 fallback. Remove "etc." from the row's *Trigger* column or replace it with "and other constructs admitted by the §3 fallback rule".

**Pros.** Closes the byte-identicality gap immediately, requires no audit machinery, scales to future triggers without spec-version churn, preserves the existing 15-row table as the source of truth for anchorless-span constructs.

**Cons.** Weakens the §3 closure: the rendering surface for the fallback arm is implementer-discoverable from the source token, not enumerated. Two implementations could still differ on a construct whose "source keyword" form is itself ambiguous (e.g. a regex literal — `/regex/`, `regex literal`, or the literal slash-bounded source span?). The fallback's test vector burden is open-ended.

**Risks.** A future contributor adds a new spec rule that fires `loom/parse/unsupported-feature` and forgets to ship a test vector covering its §3 rendering; the closing CI gate has no signal because the fallback admits everything.

### Option B — Enumerate the full trigger set in §3 and gate via the closing CI

**Approach.** Mechanically enumerate every construct in the corpus that routes to `loom/parse/unsupported-feature` — by reading `lexical.md`, `expressions.md`, `schema-subset.md`, `grammar.md`, and any other page that names the code — and add one row to the §3 table per trigger with its pinned token name. Reinforce the closure: the table is the single source of truth, and a new trigger requires a co-edit on the §3 table under GOV-7/GOV-8. Add a closing-CI gate (under `V7b`'s registry-closure gate or a new sibling) that asserts every page-side route to `loom/parse/unsupported-feature` has a matching §3 row.

**Spec edits.** Expand the §3 table by ~15–25 rows (reserved-numeric prefixes, rejected JSON-Schema keywords by name, `class`, `function*`, regex literal, `try`/`catch`/`throw`, `do`/`while`, `for`-`of`/`for`-`in`, destructuring, tagged template, the higher-order method names enumerated in `expressions.md`, `match` inside `${...}`, the field/index assignment forms). Replace "etc." in the `code-registry-parse.md` row with "(see `placeholder-rendering-a.md` §3 for the closed set)". Add the closing-gate test obligation to `V7b-T` (or a new V7c-T bullet) that statically reconciles the routes against the table.

**Pros.** Preserves the strong byte-identicality closure of §3 with no fallback arm. The CI gate mechanically prevents the same drift from re-introducing itself. Test vectors are bounded (one per row) and authorable.

**Cons.** Larger surface to maintain — every future trigger requires a co-edit under the GOV-7/GOV-8 breaking-change posture. The mechanical reconciliation gate has to parse prose ("X is `loom/parse/unsupported-feature`" mentions across the corpus) which is brittle. Reserved-keyword tokens (`class`, etc.) have an obvious rendering; constructs with no keyword (regex literal, destructuring) still need an authored token name decided up-front.

**Risks.** The CI gate could over-fire on prose mentions that are documentation cross-references rather than routing rules, requiring an opt-out marker that itself becomes a footgun.

### Recommendation

Take **Option A** first, then layer Option B's CI gate on top in a follow-up. The fallback clause is the smaller, scope-bounding edit and immediately makes the spec implementable; it is also the only option that scales cleanly when a future spec rule adds a one-off trigger (e.g. a new reserved keyword in V2). The 15-row table stays as the authoritative pin for anchorless-span constructs (the renderer cannot derive `arrow function` from `=>` alone). Implementer edge cases to call out in the fallback clause: (i) reserved numeric-literal triggers render the source-token verbatim including the prefix (`0x42`, `1_000`); (ii) JSON-Schema-keyword triggers render the keyword name as it appears in the source JSON Schema, not a translated form; (iii) constructs whose source has no single token (`map(...)`-callback, regex literal) render via the §3 `<expr>` verbatim-source-span rule rather than inventing a token name. Drop "etc." from the registry row's *Trigger* column in the same edit so the two surfaces line up.

## Consequence

**Severity:** correctness

Two conformant implementations of the V1 diagnostic surface will produce divergent byte-exact messages for any `loom/parse/unsupported-feature` trigger outside the 15-row §3 table — most pressingly the reserved-numeric-literal triggers and the JSON-Schema reject-gate triggers, both of which the spec actively routes to this code from other pages. The DIAG-4 byte-identical conformance test that `V7c-T` is supposed to ship cannot be written for those triggers as the spec stands.
