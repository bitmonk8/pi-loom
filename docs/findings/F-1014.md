## Finding

Four small artefacts of recent spec-edit passes have accumulated across the registration-and-conversation cluster of pages. None changes observable runtime behaviour, but each is the kind of historical/governance debris that distracts readers from the normative obligations they sit beside.

1. **conversation-drive.md, *Provider compatibility for typed queries*.** The `loom/load/typed-query-unsupported-provider` sentence carries an inline HTML comment — `<!-- the warning's payload is exactly the model + provider carried by its diagnostics/code-registry-load.md message template; the former "listing the typed-query expression positions" clause was dropped as no positions format was ever pinned -->` — that mixes a hidden positive claim about the warning's payload (which belongs in normative prose if it is true) with a decision-log breadcrumb explaining what an earlier draft used to say.
2. **registration-steps.md step 5, *In-flight invocation rule*.** The clause "that contract supersedes the prior single-arm `drained`-keyed short-circuit and is the sole authority on which non-dispatch system note (the `"shutting-down"` arm's legacy `"loom /<name>: extension shutting down"` template, …)" narrates the history of an older single-arm design that no other live page references, and labels the `"shutting-down"` arm's currently-normative template "legacy" — even though that template is one of the two live system notes the three-arm `readDrainState` enumeration in `drain-state-contract.md` and `patch-skew-degradation.md` still prescribes.
3. **runtime-event-channel.md, Group B bullet.** The parenthetical "per the **Edge cases** bullet under **Extension entry point** sub-step 4 above" is a stale prose reference. The "Extension entry point" sub-step 4 sequence (the `session_shutdown` subscription) is no longer on this page — its `console.error`-routed teardown diagnostics are owned by `active-invocation-registry.md`'s *Edge cases* bullet and by `registration-steps.md` step 4 — so the in-file "above" pointer dangles.
4. **runtime-event-channel.md, same Group B bullet.** The editorial directive "A new diagnostic added to either of those `console.error` sinks MUST be added to this exclusion list in the same edit." is an instruction to spec editors, not to runtime implementers; it has no observable acceptance criterion and is a governance/maintenance note that the corpus normally handles through GOV rules, not through inline normative MUSTs.

Each item is a small lift on its own. Taken together they represent the kind of low-value normative surface area that, left in place, gets cited by future readers as if it were load-bearing.

**History:** Each of the four cruft fragments traces to a distinct recent fix commit. None of the commits is itself the defect — each was a legitimate normative edit — but each left a small editorial residue (an explanatory HTML comment, a historical "supersedes" narration, a "legacy" label that survived a template rename, an in-file "above" pointer that survived the move that invalidated it, and a spec-editor sync-MUST that should have been a GOV-rule). The defect is the cumulative residue of recent fix-loop passes, not any one of them in isolation.

## Solution Space

**Shape:** single

### Recommendation

Make four targeted prose edits in a single cleanup commit:

- In `conversation-drive.md` *Provider compatibility for typed queries*, delete the inline HTML comment entirely. If the positive claim it carries ("the warning's payload is exactly the model + provider carried by its `diagnostics/code-registry-load.md` message template") is worth preserving, fold it into the sentence as normative prose, citing `code-registry-load.md` — do not retain it as a hidden comment.
- In `registration-steps.md` step 5, *In-flight invocation rule*, drop the "that contract supersedes the prior single-arm `drained`-keyed short-circuit and is" clause; the sentence reads naturally as "… below (including that contract's *Methods* paragraph's *Read-failure fallback* clause) is the sole authority on which non-dispatch system note …". In the same parenthetical, remove the word "legacy" from "the `"shutting-down"` arm's legacy `"loom /<name>: extension shutting down"` template" — the template is currently-normative under arm (b) of `readDrainState`, not legacy.
- In `runtime-event-channel.md`'s Group B bullet, replace "per the **Edge cases** bullet under **Extension entry point** sub-step 4 above and the **Persistent diagnostics** paragraph in [Diagnostics](../diagnostics.md)" with a direct hyperlink to [`active-invocation-registry.md` *Edge cases*](./active-invocation-registry.md#active-invocation-registry) (which is where the `console.error`-routed teardown diagnostics are actually enumerated) alongside the existing Diagnostics link.
- In the same Group B bullet, delete the sentence "A new diagnostic added to either of those `console.error` sinks MUST be added to this exclusion list in the same edit." It is a spec-editor governance obligation, not a runtime conformance point, and has no acceptance criterion an implementation can satisfy. If the maintenance discipline is worth preserving, lift it to a GOV rule that names this exclusion list as a co-edit target alongside the diagnostics registry.

Edge cases the implementer must watch:

- The `"loom /<name>: extension shutting down"` string itself is normative — `patch-skew-degradation.md` sub-step 1 pins the substring `"shutting down"` as a grep key for operator tooling. Only the "legacy" qualifier comes out; the template literal stays.
- The conversation-drive HTML comment sits inside a long sentence with em-dashes and `<provider>` substitution — verify after the delete that the surrounding clause still parses (the cleanest cut is the entire `<!-- … -->` span plus any leading/trailing whitespace it leaves behind; a backstop is to re-read the rendered sentence end-to-end).
- The runtime-event-channel sync-MUST deletion removes one sentence; the next sentence ("The `RuntimeEvent` payload shape and the `(kind, query_site, message, occurred_at)` dedup key …") must continue to follow the cross-reference clause without a stranded paragraph break.

## Consequence

**Severity:** cosmetic

No runtime behaviour shifts under any of the four edits. The cost of leaving them is reader confusion: an HTML decision-log comment looks like a hidden specification, a "legacy" label invites an implementer to skip the live `"shutting-down"` template, a dangling "sub-step 4 above" pointer wastes a corpus-wide search to resolve, and an editorial sync-MUST risks being implemented as a runtime conformance check.
