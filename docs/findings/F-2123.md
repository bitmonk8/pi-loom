## Finding

PIC-50 ("Prompt-mode synchronous-throw transport mapping") carries a normative MUST for the `ctx.model === undefined` edge: when no user-session model is selected at the moment `pi.sendUserMessage` throws synchronously, the synthesised `QueryError.provider` MUST be the fixed sentinel string `"unknown"` rather than `undefined`, an empty string, or the loom's own resolved `model:` value. This is a distinct branch from the normal provider-attribution path (where `provider` is the `Api`-shaped `ctx.model.api` value), reachable only when `ctx.model` is unset â€” and it is the only path on this page that has no model-unresolved gate to fall back on, since prompt mode spawns no `AgentSession` and the `"unknown"` sentinel is the sole handling of the `undefined` case.

No fixture pins this branch. `conversation-drive.md` carries no acceptance-criteria section, and the page's PIC-50/51 fixture floor lives entirely in the plan leaves `V9n` / `V9n-T`. Those leaves exercise the synchronous-throw `kind: "transport"` mapping, the `stopReason: "error"` mapping, the absent-`errorMessage` â†’ `"provider transport failure"` fallback, and the `loomAbort.signal.aborted` cancellation short-circuit â€” but none drives a `ctx.model === undefined` scenario and asserts the `provider: "unknown"` sentinel. A tester has no pinned scenario verifying that the field carries the exact sentinel string rather than `undefined`, `""`, or the loom's own model string.

**Introducing commits:** 6f38d17 â€” "pi-loom spec-plan: fix F-1145 â€” derive prompt-mode TransportError.provider from ctx.model + sentinel" (2026-06-25, Thomas Andersen)
**History:** `git log -S '"unknown"'` and `git log -S 'pic-50'` over `conversation-drive.md` both isolate `6f38d17` as the commit that coined the PIC-50 anchor's `ctx.model`-derived `provider` and added the `ctx.model === undefined` â†’ `provider: "unknown"` sentinel branch. The coverage gap is therefore co-extensive with the sentinel's introduction.

## Solution Space

**Shape:** single

### Recommendation

Add the missing sentinel fixture to the existing prompt-mode transport-error mapping leaves, matching the page's plan-based fixture convention (PIC-50/51 fixtures live in `V9n` / `V9n-T`, not in an in-spec acceptance-criteria section):

- In `V9n-T` Tests, add a failing test keyed on PIC-50: given `ctx.model === undefined` at the moment `pi.sendUserMessage` throws synchronously, the synthesised `Err(QueryError)` has `kind: "transport"`, `provider: "unknown"` (the exact sentinel string), `http_status: null`, and `retryable: false`.
- In `V9n` Tests, add the matching green assertion, and extend the `V9n` **Ships when** clause to drive the `ctx.model === undefined` throw and observe the `provider: "unknown"` sentinel.

Implementer note: assert the sentinel is the literal string `"unknown"`, not merely non-empty â€” the value is consumed by the Runtime event channel always-log path, so an empty-string or `undefined` regression must fail red. Keep this fixture orthogonal to the routing-presupposition concern on the same anchor (see Related Findings); the sentinel value is fixed and does not depend on which model the driven turn routes through.

## Consequence

**Severity:** advisory

An implementation could emit `undefined`, an empty string, or the loom's resolved `model:` value for `QueryError.provider` on the `ctx.model === undefined` edge with no test catching the divergence. The Runtime event channel always-log emission for `kind: "transport"` requires the `provider` field, so a wrong or empty value would surface in operator-visible diagnostics; the obligation itself is fully specified, so the gap is coverage, not behavioural ambiguity.
