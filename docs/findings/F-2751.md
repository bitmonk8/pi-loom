## Problem

`V2b`'s **Adds** promises "a runtime AJV safety-net for statically-unresolvable operands" and its **Ships when** gate defers those operands to that net — a branch normatively required by `type-system.md` §"Unresolvable operands" (when either side of a `T₁ ⊑ T₂` check is past the parser's static view, the parse-time `*-type-mismatch` check is skipped and the runtime AJV check is the safety net). But `V2b`'s eleven test bullets `TYPE-1…TYPE-11` all assert *static* compatibility outcomes; none exercises the unresolvable-operand path. The leaf can therefore satisfy its **Ships when** gate with the entire safety-net branch unimplemented or silently broken, because every authored assertion is about the static path.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (pi-loom plan: build/update plan for spec.md + review, 2026-06-10)
**History:** `V2b-type-compat-engine.md` was created in `c6a664e` already carrying the **Adds** "runtime AJV safety-net for statically-unresolvable operands" claim and a **Ships when** that "defers unresolved operands to runtime AJV," alongside `TYPE-1…TYPE-10` (later extended to `TYPE-11` in `b93334a`), none of which assert that path. `git log -G` over the leaf finds no commit that ever added or removed a test bullet covering the skip-and-defer-to-AJV behaviour. The defect — claim present, no asserting test — has existed since the leaf's first commit; no later edit introduced it.

## Solution approach

Add a test bullet to `V2b`'s **Tests** list asserting the safety-net path: a statically-unresolvable operand bypasses the parse-time `*-type-mismatch` codes, and a value violating the expected type is instead caught at the runtime AJV check. Cite `type-system.md` §"Unresolvable operands" as the closed obligation.
