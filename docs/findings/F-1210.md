## Problem

The *Canonical form* step of the [Canonical schema hash](#canonical-schema-hash) recipe enumerates every byte-shape rule that converts a lowered JSON Schema fragment into hash input (key sort order, whitespace, numeric rendering, string escaping), and an adjacent HTML comment justifies the keywords it omits (`default`, the rejected integer structural keywords). It is silent on `description`. Unlike those omitted keywords, `description` is lowered into the `$defs` entry (per `descriptions.md`), so it reaches the canonical-form input — yet whether the recipe hashes those bytes or strips them is unstated. The choice is observable: it determines whether two inline schemas differing only by a `///` comment dedup to one `$defs` entry / cached validator / registered tool or split into two, and whether the [Schema-slug collision posture](#schema-slug-collision-posture) byte-equality check spuriously raises `loom/load/schema-slug-collision` / `loom/runtime/validator-cache-collision` / `loom/runtime/registration-cache-collision` on `///`-only differences.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `e21449b` (2026-05-04, `pi-loom spec: resolve "__inline_<hash> hash function is not pinned"`)
**History:** Commit `e21449b` introduced the *Canonical schema hash* section in `spec_topics/schema-subset.md`, including the *Canonical form* serialisation step. That initial wording phrased the numeric rule loosely ("numeric literals in their JSON Schema integer/number form …") and made no statement about `description`. Two subsequent edits tightened the same step without ever touching `description`: `c5bb5e1` (2026-06-03, *"Canonical schema hash numeric serialization underspecified"*) replaced the loose numeric rule with the explicit `const`-/`default`-/`enum`-position list delegated to the binder number renderer, and `111e533` (2026-06-07, *"Canonical hash recipe cites keywords the lowered subset never emits"*) dropped `default` from that list and added the HTML-comment justification for why `default` and the rejected structural keywords are absent. Neither edit considered `description`, which sits in a third category — a keyword that *is* lowered, unlike the ones the comment carves out — so the silence has been continuous since the section first existed.

## Solution approach

Clarify the *Canonical form* step of the [Canonical schema hash](#canonical-schema-hash) recipe to state that `description` values are present in the canonical-form bytes and contribute to the hash with no special treatment, escaped under the same string-escape rule. Extend the adjacent HTML comment that enumerates absent keywords to account for `description` as the keyword that is present and lowered. Keep the [Schema-slug collision posture](#schema-slug-collision-posture) byte-equality check operating on the same bytes the hash sees, so it needs no special-case stripping.
