## Finding

The *Canonical form* step of the [Canonical schema hash](../../docs/spec_topics/schema-subset.md#canonical-schema-hash) recipe enumerates every byte-shape rule that turns a lowered JSON Schema fragment into the hash input: key sort order, whitespace, numeric rendering (`const` / `enum` positions, delegated to the binder's number renderer), and string escaping. An adjacent HTML comment explicitly accounts for the keywords the recipe *omits* — `default` (never lowered) and the rejected integer structural keywords (`minLength`, `maxItems`, …) — but `description` is mentioned nowhere on the page.

`description` is the one keyword in the lowered subset whose status under the canonical form is genuinely ambiguous: it is text the lowering pass emits into the `$defs` entry (per [`descriptions.md`](../../docs/spec_topics/descriptions.md), which says `///` lowers byte-for-byte to JSON Schema `description`), so it *does* reach the canonical-form input — unlike `default` and the rejected structural keywords. Whether the recipe is supposed to hash those `description` bytes or strip them before hashing is unstated.

The choice is observable. The canonical schema hash drives three slug-keyed sites — `__inline_<slug>` `$defs` dedup, the per-query AJV validator cache, and the prompt-mode `pi.registerTool` registration cache — each of which performs a byte-equality verification step on slug match (the [Schema-slug collision posture](../../docs/spec_topics/schema-subset.md#schema-slug-collision-posture)). If `description` is identity-bearing, two inline object schemas that differ only by a `///` comment occupy two distinct `$defs` entries, two distinct cached validators, and two distinct registered tools, and a trivial `///` whitespace edit invalidates downstream caches. If `description` is stripped, those same schemas dedup to a single entry — and the collision-detection byte-equality check must strip on the same rule, or it will spuriously raise `loom/load/schema-slug-collision` / `loom/runtime/validator-cache-collision` / `loom/runtime/registration-cache-collision` on every dedup-eligible pair whose source `///`s diverge.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `e21449b` (2026-05-04, `pi-loom spec: resolve "__inline_<hash> hash function is not pinned"`)
**History:** Commit `e21449b` introduced the *Canonical schema hash* section in `spec_topics/schema-subset.md`, including the *Canonical form* serialisation step. That initial wording phrased the numeric rule loosely ("numeric literals in their JSON Schema integer/number form …") and made no statement about `description`. Two subsequent edits tightened the same step without ever touching `description`: `c5bb5e1` (2026-06-03, *"Canonical schema hash numeric serialization underspecified"*) replaced the loose numeric rule with the explicit `const`-/`default`-/`enum`-position list delegated to the binder number renderer, and `111e533` (2026-06-07, *"Canonical hash recipe cites keywords the lowered subset never emits"*) dropped `default` from that list and added the HTML-comment justification for why `default` and the rejected structural keywords are absent. Neither edit considered `description`, which sits in a third category — a keyword that *is* lowered, unlike the ones the comment carves out — so the silence has been continuous since the section first existed.

## Solution Space

**Shape:** multiple

### Option A — `description` is identity-bearing

**Approach.** State in the *Canonical form* step that `description` values are serialised into the canonical-form bytes on the same RFC 8259 minimal-escape rule used for all other strings, with no special treatment. The `__inline_<slug>` dedup rule in *Lowering Algorithm* step 2 and the *Schema-slug collision posture* byte-equality check then both see the description text, so two inline object schemas that differ only by `///` produce distinct slugs, distinct `$defs` entries, distinct cached validators, and distinct registered tools by construction.

**Spec edits.** Add a fourth bullet (or extend the existing string-escape bullet) in the *Canonical form* step: *"`description` values are present in the canonical-form bytes and contribute to the hash; they are escaped per the same RFC 8259 minimal-escape rule above."* Add a parenthetical to the HTML comment that already enumerates absent keywords, naming `description` as the keyword that *is* present and pointing back to the new bullet.

**Pros.**
- Trivial to implement: no stripping pass, hash the fragment as-emitted.
- Aligns "byte-for-byte emission" (from `descriptions.md`) with "byte-for-byte hashing": one rule, no carve-outs.
- The collision-detection byte-equality check needs no special-case logic; it compares the same bytes the hash saw.
- Two `///`s that an author wrote distinctly stay distinct in `$defs`, preserving the descriptions exactly as authored at every callsite.

**Cons.**
- Trivial `///` edits (whitespace touch-ups, typo fixes) invalidate slug-keyed caches and grow `$defs` until the next full load.
- Inline schemas that an author intends as "the same" but documents twice with cosmetically different `///`s do not dedup; the lowered output is larger than necessary.

**Risks.** Authors who refactor a description independently in two callsites (intending to keep them in sync but drifting) will silently get two `$defs` entries; this is visible in payload size and in `/tools` listings but is not flagged.

### Option B — `description` is stripped before hashing

**Approach.** State in the *Canonical form* step that all `description` keys are removed from the fragment before the keys-sorted whitespace-free serialisation runs, so the hash is computed over the fragment-minus-descriptions. The *Schema-slug collision posture* byte-equality check must apply the same strip — store the stripped canonical-form bytes alongside the keyed artefact, and compare stripped-against-stripped on a slug match. Add a retention rule that pins which `description` survives in the emitted `$defs` entry on a dedup (the first-encountered one, per loom-source declaration order across the file's lowering pass).

**Spec edits.** Add a new fourth bullet in the *Canonical form* step: *"all `description` keys are removed from the fragment before serialisation; the hash is computed over the fragment-minus-descriptions."* Extend the *Schema-slug collision posture* paragraph: *"the canonical-form bytes the site stores are the same fragment-minus-descriptions bytes used for hashing, so the byte-equality check disregards `description` on the same rule."* Add a one-sentence retention rule in *Lowering Algorithm* step 2: *"on dedup, the surviving `$defs` entry carries the `description` of the first inline schema encountered in source order; later occurrences' `description`s are discarded."*

**Pros.**
- Trivial `///` edits do not invalidate slug-keyed caches.
- Inline schemas that an author "documents twice" dedup cleanly; lowered output is minimal.
- Schema-slug collision diagnostics fire only on structural collisions, not on `///` drift.

**Cons.**
- A retention rule must be specified and tested ("which `description` wins") — silently discarding the loser is observable when the two `///`s disagreed on purpose.
- Two carve-outs (hash strip + byte-equality strip) instead of one rule.
- Cache hits across `///` edits hide a class of authoring bug where the author thought their `///` change took effect but the runtime is still serving a cached validator built from the old description text — except descriptions live in the schema's *runtime* transmission to the model too, so this also bears on what the model actually sees.

**Risks.** Implementations must keep the *hash strip* and the *byte-equality strip* exactly in lockstep; any divergence reintroduces the spurious-collision class Option A avoids. The retention-rule choice influences which description the model is fed when the same inline shape is documented twice.

### Recommendation

**Option A** (descriptions are identity-bearing). The recipe's stated rationale — "two source-level inline schemas that lower to the same JSON Schema fragment produce the same slug" — uses "lower to the same JSON Schema fragment" as its dedup predicate; under `descriptions.md`'s byte-for-byte emission rule, two inline schemas with different `///`s *do not* lower to the same fragment, so Option A preserves the recipe's own stated invariant without further carve-out. The collision-detection byte-equality check then needs no special-case logic. The implementer-relevant edge case is that the description text on the carrier node — wherever the companion finding *"`description:` emission missing from the lowering algorithm and subset keyword set"* places it for each schema shape — must be present in the lowered fragment by the time the hash is computed (i.e. emission precedes hashing, not the other way round); the canonical-form rule then applies uniformly.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge: one will hash the lowered fragment as-emitted (`description` bytes included), the other will strip `description` before hashing under the assumption that documentation text is "cosmetic". Their `$defs` shapes, their per-query validator-cache hit rates, and — under the byte-equality verification on slug match — whether `loom/load/schema-slug-collision` and the two `loom/runtime/*-cache-collision` codes fire on `///`-only differences will all diverge. None of these surfaces is hidden; every one of them is in the on-disk / on-wire contract the recipe section explicitly pins as "a breaking change for any cached artefact, fixture snapshot, or replayable provider payload."
