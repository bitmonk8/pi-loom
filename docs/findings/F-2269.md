## Finding

Version-bump step 5 in `version-bump-triggers.md` defines the *Inbound-reference sweep for session-only-reason subsets* and backs it with a "second-stage" grep filter pinned as a **closed, normative anchor set**: `event.reason|SessionShutdownEvent|capturedEventReason|entry.shutdownReason`. The step then makes that set a MUST-maintained artefact in two directions: "Any future runtime-side reference shape the spec adds ‚Ä¶ **MUST extend this second-stage anchor set** in the same edit," and "The implementation **MUST implement a build-time audit pass** over the same three grep targets ‚Ä¶ and second-stage anchor filter so that a result-delta ‚Ä¶ fails `npm test` red."

Two of the four anchors are externally-observable SDK surface ‚Äî `event.reason` (the property the host delivers) and `SessionShutdownEvent` (the SDK type). The other two are loom-internal binding names: `capturedEventReason` is a handler-scoped local capture, and `entry.shutdownReason` is the field stamped onto each `ActiveInvocationRegistry` entry. By pinning these in the normative filter that a build-time gate keys on, the spec couples a mechanical `npm test` gate (and the spec's own MUST-extend obligation) to the implementation's choice of internal identifier names: if an implementation names the capture seam anything other than `capturedEventReason`, or stamps the field under a name other than `shutdownReason`, the mandated audit silently elides those reference sites ‚Äî the exact failure the sweep exists to prevent. The over-prescription is that the spec ties the gate's correctness to literal internal identifiers rather than to the behaviour ("an inbound site that reads or stamps a reason value").

The coupling is not wholly arbitrary: both names are pinned elsewhere in the corpus ‚Äî `capturedEventReason` by the *Seam minimality (normative)* clause in `session-only-degraded-state.md` ("MUST NOT introduce any handler-scoped state seam beyond `pinnedConstantReadOk` and `capturedEventReason`") and `shutdownReason` by the `ActiveInvocationRegistry` entry's `Set<{ ‚Ä¶ shutdownReason: string | undefined ‚Ä¶ }>` signature in `active-invocation-registry.md`. But step 5 nowhere cites those pins as the sanction for embedding the names in a normative gate; the coupling reads as incidental rather than deliberate, and it inherits whatever fragility the upstream seam-name pinning carries (see Related Findings).

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** `52b9101` ("pi-loom spec: resolve \"T24 ‚Äî Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state\"", 2026-05-11)
**History:** The corpus is git-tracked. `git log -G 'MUST extend this second-stage'` and `git log -G 'capturedEventReason.*entry\.shutdownReason'` over `docs/spec/**` and `docs/spec_topics/**` both bottom out at `52b9101`. At that commit the step-5 content lived in `docs/spec_topics/pi-integration-contract.md`; `git show 52b9101~1:‚Ä¶/pi-integration-contract.md | grep -c second-stage` returns `0` while the commit itself returns `1`, and the diff adds the *Inbound-reference sweep* clause, the `second-stage anchor set`, the `MUST extend this second-stage` obligation, and 25 first occurrences of `capturedEventReason` / 8 of `entry.shutdownReason` in one move. The defect was therefore born with the feature: the same T24 resolution that introduced the fork-reason degraded-state branch (and with it the internal seam `capturedEventReason` and stamped field `entry.shutdownReason`) simultaneously authored the inbound-reference sweep whose normative filter pins those internal names. Later commits (`2797e1a`, `8c0cdaa`, the `81bf52a` spec‚Üíspec_topics relocation, and the size-cap split `f5e89f4`) refined the sweep clause and strengthened the audit from MAY to MUST but carried the name-coupling forward unchanged.

## Solution Space

**Shape:** multiple

### Option A ‚Äî Make the existing normative pins the explicit sanction; keep the closed set

**Approach:** Retain the closed, normative second-stage anchor set (the build-time gate needs a concrete, deterministic filter), but add a cross-reference establishing that `capturedEventReason` and `entry.shutdownReason` are admitted into the filter *because* they are already normatively pinned identifiers ‚Äî `capturedEventReason` by the *Seam minimality (normative)* clause in `session-only-degraded-state.md`, and `shutdownReason` by the `ActiveInvocationRegistry` entry shape in `active-invocation-registry.md`. State that the MUST-extend obligation tracks those pinned identifier names, so an implementation is already bound to use them.

**Spec edits:** In step 5's *Enumeration mechanism* paragraph, where each anchor is glossed, add the resolving link to its pinning site (the gloss already names the *Predicate split* clause and the registry entry ‚Äî promote those mentions to explicit "the name pinned by ‚Ä¶" cross-references and call out that the gate's correctness rests on the implementation using the pinned identifiers).

**Pros:** Smallest diff; preserves the deterministic `npm test` gate exactly as specified; turns an implicit coupling into a sanctioned one.

**Cons:** Does not remove the spec‚Üíinternal-identifier coupling; it formalises it. Inherits any fragility in the upstream seam-name pinning (Related Findings at lines 3059 / 3513), so if those are later demoted to non-normative, this option must be revisited.

**Risks:** If the upstream seam-minimality / handler-variable pins are themselves reframed behaviourally, the explicit cross-reference here becomes dangling.

### Option B ‚Äî Express the sweep obligation behaviourally; make the concrete anchor list implementation-owned

**Approach:** State the obligation as a behavioural contract ‚Äî the bump-time sweep MUST surface every inbound runtime site that reads or stamps a `SessionShutdownEvent['reason']` value, and the mandated build-time audit MUST redden when such a site is added without being reconciled. Demote the concrete `event.reason|SessionShutdownEvent|capturedEventReason|entry.shutdownReason` list to a non-normative example of the current filter, and make the concrete filter (and the obligation to extend it) implementation-owned, on the same footing as the implementation-owned discriminator names and fixtures already in step 5.

**Spec edits:** Reword the *Enumeration mechanism* paragraph so the anchor list reads "for example, at loom 1.0 the runtime-side reference shapes are ‚Ä¶"; relocate the "MUST extend this set" obligation onto the behavioural statement and assign filter maintenance to the implementation that owns the audit pass.

**Pros:** Removes the spec‚Üíinternal-identifier-name coupling entirely; the spec stops needing byte-alignment with internal bindings.

**Cons:** Weakens the spec's leverage over the gate ‚Äî a purely behavioural obligation cannot be verified by inspecting the spec, and the deterministic "result-delta against the documented enumeration" framing (which the audit currently keys on) must move into implementation-owned territory; larger diff touching the audit-pass wording.

**Risks:** Pushing filter maintenance to the implementation risks the documented-site enumeration and the implementation's filter drifting apart unless the audit's reconciliation contract is restated carefully.

### Recommendation

Adopt **Option A**. The two internal names are already normatively pinned elsewhere, so the implementation is bound to use them regardless; making that the explicit sanction closes the coupling gap with a minimal edit while preserving the deterministic build-time gate the spec deliberately mandates. Watch the edge case that this option is contingent on the upstream seam-name pins surviving ‚Äî if the related findings at source lines 3059 and 3513 demote `capturedEventReason` / `pinnedConstantReadOk` to non-normative illustration, this clause must be re-resolved toward Option B in the same edit.

## Consequence

**Severity:** advisory

A working system is still achievable ‚Äî implementers simply use the four pinned anchor names. But the spec mandates a build-time audit gate whose correctness silently depends on the implementation choosing those exact internal identifiers, and an implementer who renames a local or field (without realising it is load-bearing for the sweep) produces a gate that under-reports inbound references and passes green while leaving a session-only literal rename able to no-op a `session-shutdown-runtime-degraded` emission. The coupling is also a maintenance hazard: the spec must be kept byte-aligned with internal binding names across every future runtime-reference addition.
