## Finding

The *No globals, statics, singletons* rule in `conventions.md` bans direct
ambient access to the spec's WallClock timing primitives and enumerates them
inline — `Date.now`, `performance.now`, `Date.prototype.getTime`, `setTimeout`,
`clearTimeout` — with the parenthetical "(sourced directly from the
`host-interfaces-services.md` `Clock`/`WallClock` ban so the two stay in
lockstep)". `H3a`'s Convention test bullet repeats the same enumerated set and
the same claim ("so the scan and the spec stay in lockstep").

The enumerated set is a **manual transcription** of the spec ban. The spec
states the ban in prose ("The runtime MUST NOT call `Date.now`,
`performance.now`, `Date.prototype.getTime`, or the global `setTimeout` /
`clearTimeout` outside the `WallClock` adapter") and exposes no machine-readable
list. No lint rule, architectural test, or closing gate compares the H3a scan's
banned set against the spec text. The two lists match today, so there is no live
gap — but "stay in lockstep" asserts a maintained synchronisation that nothing
enforces. If the spec ban later gains or drops a primitive, the copied list can
silently diverge, and the wording invites a maintainer to assume the sync is
automatic.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** c6824f20a6d3b4539e10ad4b57ba53fe07bcd9a7 ("pi-loom plan: resolve \"ambient-primitive scan omits spec-banned clock primitives\"", 2026-06-12)
**History:** `git log -S "stay in lockstep"` over `conventions.md` and `H3a-di-seam-skeleton.md` returns this single commit. The prior state of the *No globals* bullet listed only `process.env`, `process.cwd`, `crypto.randomUUID`, `Date.now`, `setTimeout`. The cited commit — resolving an earlier review finding that the scan omitted spec-banned clock primitives — expanded the enumeration to the full WallClock set and introduced the "so the two stay in lockstep" parenthetical (and the parallel "so the scan and the spec stay in lockstep" wording in `H3a`) without adding any mechanism to maintain that lockstep. The overclaim entered the corpus with that fix.

## Solution Space

**Shape:** single

### Recommendation

Soften the lockstep claim at both surfaces so it states what is actually true:
the enumerated set is a manual copy of the spec ban, no mechanism detects
divergence, and a change to the spec ban must be re-propagated by hand.

- In `conventions.md`, *No globals, statics, singletons* rule, replace the
  parenthetical "(sourced directly from the `host-interfaces-services.md`
  `Clock`/`WallClock` ban so the two stay in lockstep)" with wording that
  states the list is copied from `host-interfaces-services.md` and that no
  mechanism detects divergence, so the list must be re-synced by hand if the
  spec ban changes (a spec-ban change is a *Spec drift* event under the
  same-named rule).
- In `docs/plan_topics/H3a-di-seam-skeleton.md`, the Convention ambient-scan
  test bullet, apply the same softening to "(the enumerated set is sourced
  directly from the `host-interfaces-services.md` `Clock`/`WallClock` ban so the
  scan and the spec stay in lockstep)".

The spec page is read-only for this fix; the spec ban itself is correct and
needs no change. Keep both surfaces' wording consistent so they do not drift
from each other.

## Consequence

**Severity:** advisory

The lists currently match, so the H3a scan covers the full spec ban today.
The defect is that "stay in lockstep" claims an enforced synchronisation that no
mechanism provides: if the spec WallClock ban changes, the hand-copied set in
`conventions.md` and `H3a` can drift unflagged, leaving a newly-banned primitive
unscanned while the wording reassures a maintainer the sync is automatic.
