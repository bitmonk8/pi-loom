## Finding

The *Loads-cleanly predicate* paragraph on `source-language-stability.md` says of the no-`E` selector:

> The no-`E` predicate alone selects GOV-15's input set; **it is not co-extensive with** whether the runtime registers the file as a loom (a `W`-severity load can leave the file unregistered — e.g. `loom/load/unreadable`).

"Co-extensive" is a set-theoretic term of art — two predicates are co-extensive iff they pick out exactly the same set — and the spec coins it nowhere and uses it nowhere else as a defined term. A maintainer who skims past the term is liable to flatten it to "is not related to" or "does not affect", losing the precise claim the sentence is making (that the GOV-15 input set is defined by the no-`E` predicate *alone*, independent of the orthogonal question of whether the runtime ends up registering the file). The example in the parenthetical — a `W`-severity load that leaves the file unregistered via `loom/load/unreadable` — illustrates one direction of the non-equivalence; the term itself carries the bidirectional set-equality framing the sentence relies on.

The fix is purely lexical: substitute a plain-English phrase that preserves the "the two predicates pick out different sets" reading without inventing new jargon.

The defect is therefore inception-state in the paragraph itself, not a regression.

## Solution Space

**Shape:** single

### Recommendation

Replace `it is not co-extensive with whether the runtime registers the file as a loom` with `it does not pick out the same set as whether the runtime registers the file as a loom` (or, if a shorter form is preferred, `it is not the same predicate as whether the runtime registers the file as a loom`).

Both replacements preserve the bidirectional set-equality reading that "co-extensive" carries — the two predicates select different sets of inputs — and stay consistent with the parenthetical example that follows. The originally suggested `it does not imply that the runtime registers the file as a loom` is intentionally not the recommended form: "does not imply" is one-directional and loses the framing that the runtime's registration outcome is a *separate* predicate from GOV-15's input selector (the no-`E` one), which is the property the surrounding paragraph leans on when it asserts that the no-`E` predicate *alone* selects GOV-15's input set.

No other text on the page needs to move. The discovery-source `loom/load/unreadable` example in the parenthetical remains valid under either rephrasing.

## Consequence

**Severity:** cosmetic

A maintainer who reads "co-extensive" as a synonym for "implied by" or "related to" — rather than as "picks out the same set as" — may misapply the *Loads-cleanly predicate* when reasoning about which inputs fall inside GOV-15's equivalence promise. The mistake is recoverable by reading the parenthetical example, so no implementer-visible behaviour diverges; the cost is reader friction on a paragraph that is already structurally dense.
