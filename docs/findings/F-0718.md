## Finding

Version-bump checklist item (a) (`#bump-checklist-instance-survival` in `version-bump-step2.md`, mirrored in `host-prerequisites.md#degraded-state-host-prerequisites` clause (a)) is a MUST-level gate that blocks implementation of the session-only degraded-state branch (SM-3b / SM-4 / SM-5 / SM-6). It requires the contributor to "record an authoritative determination of whether loom's package-loaded `extensions/index.ts` is governed by [Pi's documented teardown-and-rebind] lifecycle … or whether the package-loaded path provably differs from it for the session-only reasons." Neither qualifier — *authoritative*, *provably* — is defined. The text names no acceptable evidence (a specific Pi documentation paragraph, a traced `dist/` code path, a witness from a real-host probe, a Pi-side written ruling), no actor empowered to issue the determination, and no acceptance criterion that distinguishes a conformant record from a non-conformant one.

The effect is that two contributors can record opposite verdicts on the same Pi minor and both be formally conformant: one reads `docs/extensions.md`'s "Session replacement lifecycle and footguns" and records *governed by teardown-and-rebind → branch unreachable*; another points at any divergence between the package-loaded auto-discovery path and the hot-reload path and records *provably differs → branch implementable as written*, with neither obliged to cite a specific paragraph, line, or trace. The MUST gates an editorial step whose Then-clause is silent: there is no observable predicate that fails when the record is empty, vague, or contradicted by the cited Pi sources.

Item (a) is the only MUST-level checklist item whose outcome flips a normative branch on or off in the spec — by `V9l-session-only-degraded-branch.md` and `plan.md` §Blocked obligations, "branch unreachable" collapses `V9l` to teardown + `/reload` with no degraded tag write and triggers spec-side rework of SM-4/5/6/SM-3b — so the absence of an evidentiary standard at this site is load-bearing rather than incidental.

## Solution Space

**Shape:** single

### Recommendation

Extend item (a) in `version-bump-step2.md` (and mirror in `host-prerequisites.md` clause (a)) with a recording shape that pins all three currently-undefined dimensions and attaches an acceptance criterion to each branch of the determination. Concretely:

1. **Cited-evidence requirement.** The contributor MUST cite at least one of the following in the determination rationale, by specific reference: (i) a paragraph of `@earendil-works/pi-coding-agent`'s `docs/extensions.md` at the candidate Pi minor, named by its section heading and verbatim quoted clause; (ii) a traced `dist/*.js` code site in the extracted `package/dist/` tree produced by *Baseline acquisition* (`#bump-baseline-acquisition`), named by file path + symbol, that exercises the package-loaded `extensions/index.ts` path on a `{"new","resume","fork"}` `session_shutdown`. Citing neither makes the record non-conformant and the item-(a) outcome `fail` under the conservative-posture rule already in force for item (e). The two evidence kinds are equally acceptable; (i) suffices when Pi's documentation is unambiguous about the package-loaded path, (ii) is required when (i) is silent or covers only the hot-reload path.

2. **Recording token.** Map the binary determination onto the existing pass/fail/N/A outcome slot by adding an item-(a)-specific convention to *Outcome recording conventions* (`#bump-step-2-outcome-recording-conventions`):
   - `pass — governed-by-teardown-and-rebind` records "the package-loaded `extensions/index.ts` is governed by the documented lifecycle"; branch unreachable; triggers the `plan.md` §Blocked obligations pre-designed fallback for `V9l` and the spec-side rework of SM-4/5/6/SM-3b in the same edit as the bump.
   - `pass — package-loaded-path-differs` records "the package-loaded path provably differs from the documented lifecycle for the session-only reasons"; the branch is implementable as written; the cited evidence MUST be of kind (ii) (a traced `dist/*.js` site), since "differs from documentation" cannot be established by quoting documentation.
   - `fail` records that neither sub-verdict can be supported by cited evidence at the candidate minor.
   - `N/A` is not available on item (a); unlike item (e), there is no Pi-side mechanism that can supersede the determination.

3. **Named actor.** State that the contributor performing the bump is the actor of record and that the bump PR's reviewer is the second pair of eyes empowered to reject the record on cited-evidence grounds; no separate authority is invoked. This matches the existing editorial-review framing for items (a)–(e) and avoids importing a new role.

4. **Branch-side acceptance criteria.** Cross-reference the resulting recording from the two affected leaves so the implementer reading there sees the gate. In `session-model-and-appendix.md` at SM-4 add a contingency note pointing at item (a)'s recording shape; in `session-only-degraded-state.md`'s degraded branch state that the branch is reachable only when item (a)'s last-recorded outcome is `pass — package-loaded-path-differs`. (The cross-references are the implementability companion of the recording shape; without them an implementer reading SM-4 sees no indication that an item (a) outcome gates the branch.)

Edge cases for the implementer: a Pi minor where `docs/extensions.md` is silent about the package-loaded path (only the hot-reload path is described) cannot reach `pass — governed-by-teardown-and-rebind` on evidence kind (i) alone — kind (ii) is required either way; a Pi minor that documents the package-loaded path explicitly but contradicts itself across paragraphs is `fail` until the contradiction is resolved upstream; the recording shape MUST be re-evaluated on every minor bump (item (a) is per-bump, not one-and-done).

## Consequence

**Severity:** correctness

A MUST-level gate whose oracle is two undefined qualifiers is not a gate: two contributors will diverge on whether the same Pi minor "provably differs" from the documented lifecycle, and both records are conformant against the current text. Because the gate's resolution decides whether SM-3b / SM-4 / SM-5 / SM-6 are implemented as written or reworked, the divergence selects different spec content under different reviewers without any of them violating the procedure.
