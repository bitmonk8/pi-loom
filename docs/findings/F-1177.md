## Problem

The Newline-continuation trigger table in `docs/spec_topics/grammar.md` (the `## Newline continuation` section, "Trailing binary or ternary operator" row) lists `?` in its operator set without qualification, but Loom has two `?` operators with opposite continuation semantics: the ternary head `?`, which must continue the statement across a newline, and the postfix error-propagation `?` (ERR-18), a complete-expression terminator that desugars to `return Err(e)` and must close the statement. Under the table as written, a line ending in postfix `?` reads as one continued statement; under the `?`-operator semantics elsewhere in the spec it is two statements, so two compliant implementers diverge. The trigger row is the normative source — `lexical.md`'s "Statement terminators." prose cedes the closed set to it and the plan leaves cite it verbatim — so the disambiguation must land in this row, not in a downstream gloss.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `f841b12` (2026-05-04, "spec: resolve Cluster 2 (literal sublanguage + grammar appendix)")
**History:** The Newline-continuation prose was first added in `fae85d3` (2026-05-04, "Tighten spec for implementation-plan readiness") as the bullet *"A binary or ternary operator at the *end* of a line continues the statement to the next line"* — unambiguous because postfix `?` is neither binary nor ternary, and the prose enumerated no operators. `f841b12` (later the same day) moved the rule into the new `grammar.md` appendix and **converted the prose into a tabulated character list** including `? :` raw; that conversion dropped the binary/ternary qualifier in favour of the literal enumeration, and the enumeration silently swept in postfix `?`. The downstream commit `38e399f` (2026-06-05, "trailing-continuation operator set disagreement") only reconciled `lexical.md`'s gloss with the table; it did not revisit the table itself. `git log -L '/^| Trailing binary/,+1:docs/spec_topics/grammar.md'` shows the row has been edited exactly once — at `f841b12` — since its introduction.

## Solution approach

Clarify the trailing-operator trigger row in `grammar.md`'s `## Newline continuation` table so `?` triggers continuation only as the ternary head, and the postfix error-propagation `?` never triggers continuation. Add a forward-link from the qualifying text to the existing StmtBlock tail-`?` carve-out paragraph (which cites `errors-and-results/error-model.md#err-18`) so the two normative statements of postfix-`?` behaviour are co-located.

## Solution constraints

- Out of scope: `lexical.md`'s "Statement terminators." gloss and `expressions.md`'s precedence table — both already encode the correct distinction; the fix lands solely in `grammar.md`'s trigger row.
