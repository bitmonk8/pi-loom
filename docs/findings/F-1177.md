## Finding

The Newline-continuation trigger table in `docs/spec_topics/grammar.md` enumerates the trailing-operator trigger as *"the line ends with one of `+ - * / % == != < <= > >= && || ? :`"*. The `?` entry is unqualified, but Loom has two distinct `?` operators with opposite continuation semantics:

- The **ternary head** `?` (precedence 9 in `expressions.md`'s table) — a true binary/ternary trigger; `let x = cond ?\n  a : b` is legal and must continue across the newline.
- The **postfix error-propagation** `?` (precedence 1, the topmost row of `expressions.md`'s precedence table; semantics in `errors-and-results/error-model.md` ERR-18 and the `StmtBlock` tail-`?` carve-out in `grammar.md` itself) — a complete-expression terminator that desugars to `return Err(e)`. A line ending in postfix `?` MUST close the statement.

Under the table as written, `let x = foo?\nlet y = bar` is one continued statement (`let x = foo?let y = bar`, which is itself a parse error of a different kind); under the postfix-`?` semantics elsewhere in the spec it is two statements. Two compliant implementers reading only the trigger table will diverge from two compliant implementers reading the `?`-operator rules. The same row reuses `:` (ternary tail) without qualification; `:` happens to be unambiguous because every other Loom use of `:` (object field, schema field, type ascription) sits inside an open `(` / `[` / `{` already covered by the bracket trigger, but the table does not say so.

The trigger row is the normative source — `lexical.md`'s "Statement terminators." prose explicitly cedes the closed set to it, and the `V1a` / `V1a-T` plan leaves cite it verbatim — so the disambiguation has to land in this row, not in a downstream gloss.

## Issue introduction

**Verdict:** single-commit

The downstream commit `38e399f` (2026-06-05, "trailing-continuation operator set disagreement") only reconciled `lexical.md`'s gloss with the table; it did not revisit the table itself. `git log -L '/^| Trailing binary/,+1:docs/spec_topics/grammar.md'` shows the row has been edited exactly once — at `f841b12` — since its introduction.

## Solution Space

**Shape:** single

### Recommendation

In the trailing-operator trigger row, qualify `?` to mean the ternary head and exclude the postfix error-propagation operator. Concretely:

- Replace the trigger-row character list `+ - * / % == != < <= > >= && || ? :` with `+ - * / % == != < <= > >= && || :`, and add a follow-on sentence to the row (or a footnote): *"A trailing `?` triggers continuation only when it is the ternary head; the postfix error-propagation `?` (ERR-18) is a complete-expression terminator and never triggers continuation. The two are distinguished by the same rule as in expression parsing: postfix `?` follows an expression-terminating token (identifier, `)`, `` ` ``, `]`, or another postfix `?`); ternary `?` follows a boolean-typed sub-expression and is paired with a later `:`."*
- The `StmtBlock` tail-`?` paragraph already in `grammar.md` (the carve-out generalised to ERR-18 operands by `4e656ac`) is the cross-reference target — link to it from the new sentence so the two normative statements of postfix-`?` behaviour are explicitly co-located.
- No change to `lexical.md` (it already defers to the appendix). No change to `expressions.md`'s precedence table (postfix `?` is already at precedence 1, ternary at 9; the distinction the trigger row needs is already in the precedence table).
- Add the missing test vector to `V1a` / `V1a-T`: postfix `?` at end of line does not continue (`let x = foo?\nlet y = bar` is two statements). The existing "trailing binary/ternary operator" positive-path vector should be tightened to use the ternary head explicitly (e.g. `let x = cond ?\n  a : b`).

Edge cases the implementer must watch:

- A line ending in the bare token `?` whose prior token is itself a postfix `?` (chained `expr??`) — both `?`s are postfix; the line terminates. There is no "double-ternary head" form.
- A ternary head at end of line whose condition is itself a `Result`-typed sub-expression (`let x = isErr(r) ?\n  fallback : extract(r)`) — the trailing `?` here is ternary because the operand of the trailing `?` is the boolean `isErr(r)`, not a `Result`; the distinguishing rule (operand type / preceding-token shape) handles this correctly without backtracking.
- The `:` token retained in the trigger row is the ternary `:`; the spec should note (or the existing bracket-trigger row should be cross-referenced) that field-declaration `:` and type-ascription `:` always sit inside an open `(` / `[` / `{` and so do not reach this row.

## Consequence

**Severity:** correctness

Two reasonable implementers — one reading the trigger table literally, one reading the `?`-operator semantics — produce different token streams for any source where a postfix `?` lands at end of line. The divergence is silent on the lexer side (the lexer cannot raise a diagnostic for a contradiction it cannot detect) and surfaces downstream as parse failures with no shared error code, or worse, as quietly-different parses if the next line begins with something that happens to extend the prior expression.
