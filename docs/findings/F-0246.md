## Problem

The `V4c` implementation leaf (`docs/plan_topics/V4c-terminal-outcomes.md`) and its paired `V4c-T` test leaf (`docs/plan_topics/V4c-T-terminal-outcomes.md`) each carry two `Tests.` bullets both labelled `ERR-13` — one enumerating the five no-rollback authoring sites and a second tagged `ERR-13 (completed-callee finality)`. `ERR-13` is a single spec REQ-ID (`error-model.md` §No rollback; `coverage-matrix.md` maps the `ERR-8`…`ERR-13` range to `V4c`), so two bullets sharing the label collide on one obligation. An implementer cannot tell whether the second bullet restates the first or is a distinct build obligation, which also weakens coverage tracking against the single `ERR-13` REQ.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** db918a2 — pi-loom plan: resolve "ERR-13 no-rollback / completed-callee finality over-claimed in V4c" (2026-06-11, Thomas Andersen)
**History:** Both leaves originally carried a single `ERR-13` bullet. Commit db918a2, resolving a prior review finding about over-claiming, split the no-rollback assertion and appended a second `ERR-13 (completed-callee finality)` bullet to both `V4c-terminal-outcomes.md` and `V4c-T-terminal-outcomes.md`, reusing the `ERR-13` REQ-ID for the new bullet rather than folding it into the existing one. A later edit (de19068) expanded the first bullet to the five authoring sites but left the duplicate label in place.

## Solution approach

In both `docs/plan_topics/V4c-terminal-outcomes.md` and `docs/plan_topics/V4c-T-terminal-outcomes.md`, merge the two `ERR-13` `Tests.` bullets into a single `ERR-13` bullet so one REQ-ID maps to one bullet, folding the completed-callee-finality scenario into the surviving bullet as an additional named vector. Keep the merged bullet text mirrored across the implementation leaf and its `-T` test pair.

## Solution constraints

- The spec is read-only for this fix; do not edit `error-model.md`, `cancellation.md`, or the `ERR-8`…`ERR-13` mapping in `coverage-matrix.md`.
- Do not mint an `ERR-13b` or otherwise split `ERR-13` into sub-IDs — `ERR-13` is a single spec-governed REQ-ID (`idAllocationPolicy: governed-by-spec`).
