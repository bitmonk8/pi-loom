## Finding

The provider-detected context-overflow failure surface has one canonical name across the spec and most of the plan: `ContextOverflowError`. The spec declares it as the `QueryError` variant `ContextOverflowError` (`kind: "context_overflow"`) in `queryerror-variants.md`, and the binder failure taxonomy in `determinism-cancellation-failure.md` §Failure-class taxonomy explicitly states "a classifier output of `ContextOverflowError` is treated as transport-class for retry purposes … There is no `ContextOverflowError`-specific row" — i.e. the spec names the full type even when describing the transport-class fold. The plan honours this in `V9j`/`V9j-T` ("context-overflow is extracted to `ContextOverflowError`"), `V4d` ("`ContextOverflowError` is the sole token-domain failure surface"), and `V13d`/`V13d-T`.

`V11f` and `V11f-T` break the convention. Their shared Tests bullet reads "the six failure templates render verbatim … ; ContextOverflow folds into the transport class" — bare `ContextOverflow`, no `Error` suffix and no code formatting, presented as a failure-class label rather than a type. Nothing in either leaf states that this `ContextOverflow` is the same surface as the `ContextOverflowError` it depends on through `V9j`.

The two leaves describe exactly the spec's transport-class fold, so the underlying behaviour is unambiguous once the cited spec page is read; the defect is the divergent spelling, which invites a reader of `V11f` alone to treat `ContextOverflow` as a distinct concept from the `ContextOverflowError` type owned by `V4d`/`V9j`.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** The bare `ContextOverflow` token in both `V11f-binder-retry-taxonomy.md` and `V11f-T-binder-retry-taxonomy.md` was present in the commit that first created those files (`git log -S"ContextOverflow folds into the transport class"` and the `--diff-filter=A` creation commit both resolve to `c6a664e`). The canonical `ContextOverflowError` spelling in `V9j-binder-inference-provider-mapping.md` was introduced in the same commit (`git log -S"ContextOverflowError"` on the V9j file → `c6a664e`). The divergent spelling was therefore authored simultaneously with the canonical one during initial plan authoring; no later edit introduced or widened the inconsistency.

## Solution Space

**Shape:** single

### Recommendation

In `V11f-binder-retry-taxonomy.md` and `V11f-T-binder-retry-taxonomy.md`, change the trailing clause of the "six failure templates render verbatim" Tests bullet from `ContextOverflow folds into the transport class` to use the canonical spelling: `` `ContextOverflowError` folds into the transport class ``. Apply the identical edit to both the tests-task leaf and the implementation leaf so their Tests bullets stay verbatim-equal.

This aligns the two leaves with the spelling already used in `V9j`, `V9j-T`, `V4d`, `V13d`, `V13d-T`, and with `determinism-cancellation-failure.md` §Failure-class taxonomy (which itself describes the transport-class fold in terms of `ContextOverflowError`). The `ContextOverflowError` type is already owned by `V4d` and declared in the spec, so no new registry entry or equivalence note is required — only the spelling correction. Do not add a new index/registry file for this; the canonical home is `V4d` plus the spec variant declaration.

## Consequence

**Severity:** advisory

A reader of `V11f`/`V11f-T` in isolation cannot tell from the plan text that `ContextOverflow` is the `ContextOverflowError` type owned by `V4d` and surfaced by `V9j`; the cited spec page disambiguates it, so a careful implementer recovers, but the inconsistent spelling weakens the index-level and cross-leaf traceability of the token-overflow surface.
