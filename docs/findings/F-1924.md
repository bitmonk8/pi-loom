## Finding

The test-only `ReloadFailureInjector` interface and its `injectReloadFailure` method are plan-introduced â€” neither name appears anywhere in `spec.md` or `spec_topics/`. `V9b` is named the **single declaration site** for the interface ("all three arms â€¦ inject through the one `injectReloadFailure` method declared here"), so it is the only place the seam's shape can be pinned.

`V9b`'s `Adds` field names the method but never states its contract: there is no parameter shape, no discriminator that routes a single `injectReloadFailure` call onto one of the three failure arms (registry-swap, `.loom`/`.warp` re-parse, settings-re-merge), no per-arm failure-payload type, and no return type. The seam has two consumer leaves that bind against this same shape: `V10d` contributes the settings-re-merge arm "against the test-only `ReloadFailureInjector` interface declared by `V9b`" and explicitly "does not declare its own interface", and `V4g` "imports the interface from `V9b` (via `Deps`)" to exercise `ERR-7`.

Because both consumers must compile and call against a shape that the single declaration site never fixes, two reasonable implementers picking up `V10d` and `V4g` would each invent a different signature, discriminator encoding, and payload shape. The result is a seam whose three arms do not compose through one method and an `ERR-7` watcher-time-reload failure-injection test wired against a method shape that does not match the one `V9b` ends up emitting.

## Issue introduction

**Verdict:** present-since-introduction
**Introducing commits:** `d64dce5` (`pi-loom plan: resolve "Watcher-time reload failure-injection seam under-specified and ungated"`, 2026-06-11)
**History:** The named `ReloadFailureInjector.injectReloadFailure` interface was coined in `d64dce5`, which replaced an unnamed "watcher-time reload failure-injection seam" prose phrase with the named test-only interface and method. That commit introduced the method name but never pinned its parameter shape, arm discriminator, per-arm payload, or return type. Later refining commits repeatedly touched the seam and its consumers without closing the gap: `fdce2a2` (made `V9b` the single declaration site and added `V10d`'s settings-re-merge arm), `f14e929` (split `V10c` into settings-merge and reload-debounce leaves), and `324813d` (split the `ERR-7` scenario out of `V4e`'s routing checks). The signature gap has existed continuously since the interface was first named.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V9b-registration-reload-wiring.md`, extend the `Adds` field's `ReloadFailureInjector` description so the single declaration site pins the contract its two consumers bind against:

- the `injectReloadFailure` parameter â€” the synthetic failure descriptor a caller hands in, carrying a closed arm discriminator over exactly the three arms `{ registry-swap, `.loom`/`.warp` re-parse, settings-re-merge }`;
- the per-arm failure payload each arm carries;
- the method's return type.

State which arm each leaf contributes or exercises so the discriminator's three values are unambiguous: `V9b` wires the registry-swap (`loom/runtime/registry-swap-failed`) and `.loom`/`.warp` re-parse arms, `V10d` contributes the settings-re-merge arm, and `V4g` exercises all three via `ERR-7`. The exact TypeScript form of the descriptor and return type is the implementer's; the `Adds` prose must fix the four facets above (parameter shape, the closed three-value discriminator, per-arm payload, return type) so `V10d`'s and `V4g`'s call sites resolve to one shape. The spec stays read-only for this fix.

## Consequence

**Severity:** correctness

If the plan ships unfixed, the `V10d` and `V4g` implementers each guess `injectReloadFailure`'s parameter, discriminator, payload, and return type independently, producing a seam whose three arms do not compose through one method and an `ERR-7` failure-injection test mis-wired against a method shape `V9b` never declared.
