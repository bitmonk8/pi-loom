## Problem

GOV-15's *Operational definitions* permit a *Tighten* within loom 1.x "only when no input in the loom 1.x conformance fixture suite (once it ships) is made to fail," and forbid a tightening that "would fail any input in the eventually-shipped suite." Both arms key solely on the suite's *existence* — there is no floor on what the suite must contain to be adequate to gate a *Tighten*. The moment any suite ships, the gate flips from "Tighten absolutely forbidden" to "Tighten permitted when no fixture fails"; a suite with a single trivial fixture would vacuously satisfy "no input is made to fail," unlocking *Tighten* while providing no real protection for the cross-release equivalence promise GOV-15 exists to defend.

## Issue introduction
**Verdict:** single-commit introduction
**Introducing commits:** 2ff03c2 (2026-05-10, "pi-loom spec: resolve \"Source-language stability bullet leaves equivalence undefined at the aggregator and silently contradicts the open-set ceiling clause\"")
**History:** `git log -S 'inverse of relax'` and `git log -S 'no input in the loom 1.x conformance fixture suite'` both first surface the *Tighten* operational definition in 2ff03c2, the commit that authored the entire *Operational definitions* section (the four ceiling-set verbs) and the `gov-15-fixture-suite` anchor. `git show 2ff03c2` confirms the added text already keyed the *Tighten* unlock on the fixture suite "(once it ships)" / "the eventually-shipped suite" with no adequacy floor — the defect entered together with the clause itself, not via a later edit. Subsequent edits touching the file (a6d82db V1→loom 1.x rename, 7995d3c file relocation, 8d730c5 DIAG carve-out) preserved the gate text verbatim and never added a coverage criterion; the current wording still reads "once it ships" / "the eventually-shipped suite."

## Solution approach

Narrow the *Tighten* permission clause so the unlock keys on a minimum-adequacy criterion for the conformance fixture suite rather than its bare existence, and cross-reference that criterion from the `gov-15-fixture-suite` anchor. Express the criterion in terms of the suite's coverage content and artifact status — a published, versioned artifact frozen against the loom 1.0.0 baseline that exercises GOV-15's three observables (return values, ordered diagnostic-code sequences, `loom-system-note` content) across the corpus's normative examples. Until a suite meeting that floor ships, the pre-suite absolute *Tighten* prohibition remains in force.

## Solution constraints

- The criterion MUST constrain only the suite's coverage content and artifact status, not release-process or CI-audit mechanics — GOV-18 owns whether or how any release process audits GOV-15.
