## Problem

DISC-2's third implementation-notes bullet in `discovery-sources.md` and the `loom/load/wrong-type-source` row in `code-registry-load.md` both describe the wrong-type trigger as a path that is "neither a `.loom` file nor a directory containing them". That phrasing leaves a real branch undecided: an empty directory — or one whose entries are all non-`.loom` files — *is* a directory but is not "a directory containing them". Two reasonable implementers diverge: one treats such an entry as wrong-type and emits `loom/load/wrong-type-source`, the other treats it as a valid root enumerating zero looms. The failure-modes table cannot disambiguate because both implementers can cite the same row.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 5b772b6 — pi-loom spec: resolve "Discovery source failure-mode policy is unspecified" (2026-05-04, Thomas Andersen)
**History:** Commit `5b772b6` introduced DISC-2 from scratch, authoring both the failure-modes table and the third implementation-notes bullet ("the wrong-type rule fires only when the path exists but is neither a `.loom` file nor a directory containing them"). The same commit added the parallel `loom/load/wrong-type-source` row to `spec_topics/diagnostics.md` with the matching wording. Both sites have used the "containing them" phrasing verbatim since their first appearance; `git log -S "containing them"` over the discovery spec shows only the introducing commit and later file-move commits (`f5e89f4`, `31ff060`) that carried the text unchanged. The defect is therefore present-since-DISC-2's-inception, scoped to a single authoring commit.

## Solution approach

Clarify the DISC-2 third implementation-notes bullet in `discovery-sources.md` so it states that a directory entry is always a valid path regardless of contents, and that an empty directory (or one whose entries are all non-`.loom` files) enumerates zero looms successfully and emits no diagnostic — the wrong-type rule fires only when the path resolves to something that is neither a regular `.loom` file nor a directory. Apply the same clarification to the `loom/load/wrong-type-source` row description in `code-registry-load.md` so the two sites stay consistent.
