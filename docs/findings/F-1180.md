## Problem

DISC-5 in `discovery/package-and-settings.md` (anchor `#disc-5`) pins glob resolution for `pi.looms` — and, by inherited reference, `loomPaths` — to the `minimatch` engine, making `minimatch` a normatively-required loom-package runtime dependency. The canonical dependency inventory, `implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)" (anchor `#loom-package-implementation-dependencies-loom-1-0`), enumerates `semver`, `chokidar`, and `yaml` but never names `minimatch`. An implementer treating the inventory clause as the single source of dependency truth would omit `minimatch` from the package's `dependencies`, and DISC-5's glob contract would then fail at first `pi.looms` resolution.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `1cb54c7` (2026-06-06, "pi-loom spec: resolve \"Glob !/+/- precedence and matcher engine unspecified for pi.looms / loomPaths\""); origin context `1ea0455` (2026-06-04, "pi-loom spec: resolve \"DI-seam vs reference-implementation placement boundary\""); missed-opportunity `1064946` (2026-06-10, "pi-loom plan: resolve \"YAML frontmatter parsing mechanism is never declared as a dependency\"")
**History:** `git log -G 'minimatch' -- docs/spec_topics` returns a single commit, `1cb54c7`, the sole site where `minimatch` enters the spec corpus — added to DISC-5 (and inherited by `loomPaths`) as the pinned glob engine. `git log -S 'Loom-package implementation dependencies' -- docs/spec_topics/implementation-notes.md` shows the enumeration clause was authored at `1ea0455` (before `minimatch` was named in DISC-5) with `semver` and `chokidar` entries. `git log -S 'yaml' -- docs/spec_topics/implementation-notes.md` shows the `yaml` clause was added at `1064946` (after `minimatch` had landed in DISC-5) in response to a sibling missing-dependency finding for the frontmatter parser; that resolver added `yaml` on the chokidar template but did not sweep for other inventory gaps. The defect is therefore a multi-commit interaction: `1cb54c7` introduced the normative dependency without enrolling it in the inventory, and `1064946` had the opportunity to close the gap (was already editing the same paragraph for a structurally identical omission) but did not.

## Solution approach

Add a `minimatch` clause to `implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)" (anchor `#loom-package-implementation-dependencies-loom-1-0`) naming `minimatch` as a loom-package `dependencies` entry on the same footing as the existing `semver`/`chokidar`/`yaml` entries, and forward-linking `discovery/package-and-settings.md#disc-5` as the glob engine the dependency satisfies.

## Solution constraints

- Out of scope: pinning a `minimatch` version — F-1155 owns version pinning, and any pin it lands attaches to this clause.
- Out of scope: editing DISC-5 itself; the inventory clause is the alignment-closing edit.
