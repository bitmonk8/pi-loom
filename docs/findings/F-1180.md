## Finding

`discovery/package-and-settings.md` §DISC-5 pins glob resolution for `pi.looms` (and, by inheritance, `loomPaths`) to the `minimatch` engine — naming the package directly, fixing the override order, the three match targets, and `nocase: off`. The `loomPaths` *Glob patterns and exclusions* bullet defers to that same DISC-5 contract by reference. `minimatch` is therefore a normatively-required runtime dependency of the loom package.

The canonical dependency inventory — `implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)" (anchor `#loom-package-implementation-dependencies-loom-1-0`) — enumerates the loom runtime's package deps with a single additive paragraph: `semver` is the load-bearing entry, then "additionally depends on" `chokidar`, then "additionally depends on" `yaml`, each on the same footing. `minimatch` is not named anywhere in that paragraph (nor in the adjacent AJV implementation-hint bullet, which is non-normative and scoped to schema validation). An implementer working from this section as the single source of dependency truth would not declare `minimatch` in the package's `dependencies`, even though plan leaf `H1a` already (correctly) lists it.

The result is a doc-alignment break: a normative spec requirement (DISC-5's glob engine) names a package that the spec's own dependency-inventory clause omits. The plan happens to carry the right answer, but the spec corpus must be self-consistent — the inventory clause is the canonical home, and DISC-5 has no forward-link compensating for the omission.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `1cb54c7` (2026-06-06, "pi-loom spec: resolve \"Glob !/+/- precedence and matcher engine unspecified for pi.looms / loomPaths\""); origin context `1ea0455` (2026-06-04, "pi-loom spec: resolve \"DI-seam vs reference-implementation placement boundary\""); missed-opportunity `1064946` (2026-06-10, "pi-loom plan: resolve \"YAML frontmatter parsing mechanism is never declared as a dependency\"")
**History:** `git log -G 'minimatch' -- docs/spec_topics` returns a single commit, `1cb54c7`, the sole site where `minimatch` enters the spec corpus — added to DISC-5 (and inherited by `loomPaths`) as the pinned glob engine. `git log -S 'Loom-package implementation dependencies' -- docs/spec_topics/implementation-notes.md` shows the enumeration clause was authored at `1ea0455` (before `minimatch` was named in DISC-5) with `semver` and `chokidar` entries. `git log -S 'yaml' -- docs/spec_topics/implementation-notes.md` shows the `yaml` clause was added at `1064946` (after `minimatch` had landed in DISC-5) in response to a sibling missing-dependency finding for the frontmatter parser; that resolver added `yaml` on the chokidar template but did not sweep for other inventory gaps. The defect is therefore a multi-commit interaction: `1cb54c7` introduced the normative dependency without enrolling it in the inventory, and `1064946` had the opportunity to close the gap (was already editing the same paragraph for a structurally identical omission) but did not.

## Solution Space

**Shape:** single

### Recommendation

Add a `minimatch` clause to `implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)" (anchor `#loom-package-implementation-dependencies-loom-1-0`) immediately after the existing `yaml` clause, following the established template the `chokidar` and `yaml` clauses already use:

- Open with "The loom 1.0 runtime additionally depends on [`minimatch`](https://www.npmjs.com/package/minimatch) as the glob-matching engine behind DISC-5", forward-linking `discovery/package-and-settings.md#disc-5`.
- State that `minimatch` is declared in the loom package's `dependencies` block on the same footing as `semver`, `chokidar`, and `yaml`.
- Close with the existing "this clause names the dependency, not its version" wording, mirroring chokidar/yaml.
- Note (in the same clause, matching the chokidar/yaml pattern) that `minimatch` is neither one of the four `@earendil-works/*` peer packages nor `typebox`, so it is out of scope for the inventory-closure audit's *Target surface categories* and requires no `// allow-pi-surface:` exemption marker at `minimatch` import sites.
- Do not pin a major version in this clause; the sibling DISC-5 option-set finding owns version pinning and the option enumeration.
- Do not edit DISC-5 itself; the sibling over-prescription finding owns whether DISC-5 keeps naming `minimatch` directly or demotes to "behavioural parity with Pi's resource-array glob dialect". If that demotion lands first, rephrase this clause from a normative dependency entry into a reference-implementation note in the same location — the *placement* in the inventory paragraph is what closes the alignment gap; the framing follows from how DISC-5 ends up reading.

Edge case for the implementer: the related "per-package audit carve-out duplicated for chokidar/yaml" finding may collapse the per-package carve-out paragraphs into a single general statement deriving from `audit-target-categories.md`. If that collapse lands first, drop the per-package carve-out sentence from this new clause and let the general statement cover `minimatch`; if it lands second, the per-package sentence added here is harmless and will be folded in by that finding's edit.

## Consequence

**Severity:** advisory

Two implementers reading only the canonical inventory will produce a package that omits `minimatch`, and DISC-5's normative glob contract will then fail at first `pi.looms` resolution because the engine it pins is not on the dependency list. The plan leaf carries the right answer, so anyone who reaches `H1a` recovers — but the spec is the dependency contract's source of truth, and an implementer is entitled to rely on the inventory clause alone.
