## Finding

The normative *Field contract* table on `docs/spec_topics/frontmatter/frontmatter-fields-a.md` pins twelve frontmatter fields — `mode`, `description`, `argument-hint`, `model`, `bind_model`, `bind_context`, `bind_echo`, `tools`, `system`, `respond_repair`, `tool_loop`, `params`. Each row carries a per-field MUST cluster (load-time diagnostic codes, recognised-key / unrecognised-value splits, absent-vs-present-but-bad partitions, no-warning carve-outs). None of the twelve table rows carries an HTML anchor at its row.

The only per-field anchor in the section's normative surface is `<a id="tools"></a>` on the *prose* paragraph below the table (added incidentally by the PIC step-2 fix at commit `3536c67`, so PIC's tool-registration owner could forward-link to the "ambient tools are deliberately not inherited" invariant). The `argument-hint` row similarly has `<a id="argument-hint-binder-only"></a>` on its owning prose paragraph (added at `4da6c6c`), but again, not at the table row itself. The other ten fields have neither a row anchor nor an owning-prose anchor.

Two consequences follow. (1) A cross-page citation that needs to pin "the `mode:` recognised-key / unrecognised-value split" (currently invoked by `bind_context`'s and `methodology`'s rows on the same page, plus prose elsewhere) has no fragment link to resolve to — the rule is paraphrased instead, and the paraphrase drifts silently under future rewording. (2) The `FRNT` REQ-ID prefix is already registered against `frontmatter/` in `req-id-prefix-table-active-a.md` and currently carries exactly one obligation (`FRNT-1`, the `tool_loop` rule on `frontmatter-fields-b-and-templates.md`); the per-field MUSTs on `frontmatter-fields-a.md` could be lifted into the same prefix at any time, but until they are, GOV-1 / GOV-9 cross-link mechanics simply do not apply to them.

## Issue introduction

**Verdict:** present-since-inception (with later partial-anchor incidents)
**Introducing commits:** `fb54586` (2026-05-04, "Frontmatter field required/optional/default contract is unspecified")
**History:** `fb54586` introduced the *Field contract* table with all twelve rows un-anchored. The first per-row reference target was added at `3536c67` (2026-05-21, "PIC step 2 setActiveTools union-vs-replace contradiction"), which placed `<a id="tools"></a>` on the `tools` prose paragraph as a side-effect of giving PIC a forward-link target — not as a deliberate row-level anchor pass. `4da6c6c` (2026-06-06, "argument-hint field contract is defined once and duplicated") added `<a id="argument-hint-binder-only"></a>` on the `argument-hint` owning-prose paragraph for the same reason. The asymmetry has therefore been present since the table existed; subsequent commits only anchored individual rows when an unrelated forward-link demanded it.

## Solution Space

**Shape:** single

### Recommendation

Add a per-row stable anchor at every row of the *Field contract* table, mirroring the `tools` and `argument-hint` precedent: place `<a id="frontmatter-field-<name>"></a>` immediately before the bold field-name code span in each row's first cell (or, for rows whose row prose lives in the bullet block below — `params`, `model`, `tools`, `system`, `respond_repair`, `tool_loop` — duplicate the anchor at the matching prose paragraph so both surfaces resolve). Use the field name verbatim, hyphenated for compound names (`frontmatter-field-mode`, `frontmatter-field-argument-hint`, `frontmatter-field-bind-model`, ...); the existing `#tools` and `#argument-hint-binder-only` anchors stay where they are (they are cited elsewhere and GOV-26 alias-permanence-style stability applies, even though they predate this rule).

In the same edit, lift each row's per-field MUSTs into the `FRNT` prefix as `FRNT-2`, `FRNT-3`, ... in field order, dual-anchored as `<a id="frnt-N"></a> <a id="frontmatter-field-<name>"></a>` per GOV-1 *Dual-form layout*. The `FRNT` prefix is already registered in `req-id-prefix-table-active-a.md` and currently holds only `FRNT-1`, so no prefix-table change is needed. Once the IDs exist, the `mode:`-row "recognised-key / unrecognised-value split" can be cited as a fragment from the rows that already invoke it by paraphrase (`bind_context`, `methodology`) instead of restating the rule.

Edge cases the implementer must watch:

- The `mode:` row contains two distinct obligations (`loom/load/missing-mode` for absent; `loom/load/unknown-mode-value` for present-but-bad). GOV-8 *Split* prefers one ID per independently-testable obligation; assign `FRNT-2a` / `FRNT-2b` (or two peer IDs) rather than collapsing.
- The `bind_model` row threads through three diagnostic codes plus a strict-capability tri-state (`E` / `W` / silently-registers); follow the same split.
- The `argument-hint` row's "[`argument-hint` autocomplete contract](#argument-hint-binder-only)" link inside the cell already targets the existing `argument-hint-binder-only` anchor on the owning prose; keep that link unchanged so the established cross-references on `slash-invocation.md` and `pi-integration.md` still resolve.
- Per-row anchors and per-row `FRNT-N` IDs are independently useful: if the REQ-ID assignment is contentious, ship the anchors first and add IDs in a follow-up. Both halves fit inside one spec-side commit.

## Consequence

**Severity:** advisory

Implementers can still build to the table — each row is self-contained prose. The cost is downstream: cross-page citations to per-field MUSTs paraphrase the rule instead of citing it by fragment, and the GOV-9 / GOV-30 cross-link machinery (and the closing-gate `un-anchored MUST` scan) has nothing to verify against. As the surrounding spec adopts more `FRNT-N` REQ-IDs on adjacent pages, the asymmetry — anchored on `-b`, un-anchored on `-a` — grows.
