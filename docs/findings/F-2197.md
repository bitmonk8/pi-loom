## Finding

`docs/spec_topics/pi-integration-contract/provider-error-mapping.md` normatively defines two surfaces that sit on opposite sides of the request/response divide:

1. **Provider error-response classification** (`#provider-error-mapping`, `#transport-error-retryable`, `#stop-reason-classification`) — a *response-handling* path that maps HTTP status class, provider error-body wording, and stop-reason signals onto `ContextOverflowError` / `TransportError` / `stopReason` variants.
2. **Provider seed-field mapping** (`#provider-seed-field-mapping`) — a *request-construction* path: a static table keyed on the resolved binder model's `api` that governs which JSON request field (if any) carries the binder's determinism seed.

The two share an organizing principle — both are per-provider mapping tables version-coupled to `@earendil-works/pi-ai` and re-validated on each upgrade — but they have no normative interdependency, and the seed-field mapping is the request-side companion to a binder requirement whose seed *value* derivation lives in the binder cluster (`binder/determinism-cancellation-failure.md#determinism`). The page presents the two sections back-to-back with no seam, no scope note, and no non-goals statement declaring why they are co-delivered or what the page deliberately does not cover (e.g. runtime detection of new pi-ai `Api` values beyond the build-time coverage assertion). The H1 (`Provider error and seed-field mapping`) already names both surfaces, so the title is accurate; what is missing is an explicit boundary that tells a reader the co-location is intentional rather than incidental, and that the seed-field mapping is not part of the error-classification contract.

## Solution Space

**Shape:** multiple

### Option A — Relocate the seed-field mapping to the binder cluster

- **Approach:** Move the `provider-seed-field-mapping` section (table + `Api`-coverage-assertion prose) into the binder cluster — alongside `binder/determinism-cancellation-failure.md#determinism`, or as a new `binder/binder-seed-mapping.md` — and leave a cross-reference anchor from `provider-error-mapping.md`. Narrow the page H1 back to error/classification scope.
- **Spec edits:** Cut the `#provider-seed-field-mapping` block from `provider-error-mapping.md`; paste into the binder destination; repoint the existing inbound links (`binder-inference.md`'s seed-placement bullet, `version-bump-intro.md` step 6, and the `#determinism` reference's reverse link) to the new anchor; revise the H1.
- **Pros:** Co-locates the seed *field* mapping with the seed *value* derivation it serves; the error-mapping page becomes single-concern.
- **Cons:** Splits the seed-field table from the version-bump-procedure re-validation step (step 6) and the build-time `Api`-coverage assertion narrative, both currently adjacent in the pi-integration-contract cluster; requires repointing several cross-links.
- **Risks:** Link rot if any inbound anchor is missed; the `Api`-coverage assertion narrative may read as orphaned from the pi-ai-version-coupling context it currently shares with the error-mapping re-validation gate.

### Option B — Keep co-located, declare the boundary

- **Approach:** Leave both sections on the page (the shared organizing principle — per-provider tables version-coupled to pi-ai and re-validated per the same bump procedure — justifies co-location) but add a short scope/non-goals note near the top that (a) states the page owns two distinct surfaces (response-path classification and request-path seed-field mapping), (b) notes they are co-delivered because both are pi-ai-version-coupled provider tables with no runtime interdependency, and (c) names what is out of scope (e.g. runtime detection of pi-ai `Api` additions beyond the build-time coverage assertion).
- **Spec edits:** Insert a `**Scope.**` / `**Non-goals.**` paragraph after the H1, before `#provider-error-mapping`; optionally add a one-line seam sentence before `#provider-seed-field-mapping` pointing to the binder determinism requirement it serves.
- **Pros:** No cross-link churn; preserves the pi-ai-version-coupling context shared with the version-bump procedure; makes the deliberate bundling explicit so the boundary is no longer implicit.
- **Cons:** Leaves the request-path table on a page whose name leads with "error"; does not heal the seed field/value split across clusters.
- **Risks:** Minimal; a non-goals note can drift if `Api`-detection scope changes, but it tracks the same surface the coverage assertion already pins.

### Recommendation

Take **Option B**. The two tables share a real organizing principle — both are provider-keyed mappings version-coupled to `@earendil-works/pi-ai`, re-validated through the same Pi version-bump procedure — and the page H1 already names both, so a documented boundary resolves the implicit-scope defect at lower risk than a relocation that would split the seed-field table from its `Api`-coverage assertion and step-6 re-validation context. Watch that the non-goals note stays consistent with the build-time `Api`-coverage assertion's stated scope.

## Consequence

**Severity:** cosmetic

Both sections are internally complete and individually clear, so no implementer produces wrong behaviour from the bundling. The cost is organizational: a reader scanning `provider-error-mapping.md` for the error-classification contract encounters an unrelated request-path table with no declared boundary, and the seed-field mapping is separated from its binder-cluster value-derivation spec without a stated rationale.
