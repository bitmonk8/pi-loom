## Problem

The statement-form control-flow block rule in `grammar.md` (the `if` / `for` / `while` paragraph after the `BlockExpr` grammar) says a present tail `Expr`'s value is discarded "except that a tail `Expr` ending in the postfix `?` operator … still triggers `?` early-return on failure." The "ending in" qualifier is imprecise: it does not say whether it means the tail's outermost operation is `?`, its last lexical token is `?`, or that a `?` appears anywhere in the tail. The carve-out describes a control-flow side effect, but the postfix `?` (per `expressions.md` `#err-18` / `#question-operator`) early-returns wherever it is evaluated, independent of its position — so in `bar(foo()?)` the nested `foo()?` early-returns before `bar` is called. The phrasing invites an implementer to special-case the carve-out to the outermost-`?` tail and silently drop a nested `?`'s early-return, producing divergent runtime control flow between conforming implementations.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 4e656ac — pi-loom spec-plan: fix F-0707 — StmtBlock tail ? carve-out generalised to ERR-18 operands (2026-06-20, Thomas Andersen)
**History:** Before 4e656ac the carve-out read "a tail `Expr` of the `@…?` form", a query-template-specific surface that was narrow but unambiguous. F-0707 (4e656ac) broadened it to "a tail `Expr` ending in the postfix `?` operator — applied to any operand `?` admits" so it would cover every `Result`-typed operand; that generalisation introduced the imprecise "ending in" phrasing this finding targets. A later commit (87dfa1d) only re-pointed the ERR-18 cross-reference and left the phrasing intact.

## Solution approach

Rewrite the statement-form control-flow block-tail sentence in `grammar.md` so the carve-out is stated in terms of evaluation rather than the positional "ending in" predicate: the tail `Expr` is fully evaluated and only its successfully-produced value is discarded, and any postfix `?` evaluated within it still early-returns its `Err`, regardless of whether `?` is the tail's outermost operation. Keep the existing cross-references to `expressions.md#err-18` and `return.md`.
