## Finding

In `docs/spec_topics/query/query-failure-and-repair.md` §"Failure modes", the second paragraph (the one introducing `ModelToolError` as the model-loop counterpart to `CodeToolError`) ends with the parenthetical `(see **Tool calls during a query** above)`. There is no section by that name above it in this file — `## Failure modes` is the file's first body section after `## Options surface`. The referenced "Tool calls during a query" heading lives in a sibling file, `docs/spec_topics/query/query-tool-loop.md` (its `## Tool calls during a query`, line 3). The pointer is therefore both directionally wrong ("above" suggests the same file) and link-less.

The same paragraph then restates the `ModelToolError` ↔ `CodeToolError` relationship — "covers a non-recoverable adapter-layer failure of the model's tool-call loop … carries `tool_call_id` plus a `raw_response` … while `CodeToolError` covers a tool that **loom code** invoked directly … shapes diverge because the contexts diverge". That text is a paraphrase of the canonical paragraph already carried by `docs/spec_topics/tool-calls.md` (the long paragraph beginning "`CodeToolError` is distinct from `ModelToolError`" right after the `**Failures.**` bold lead, which itself pins the same per-context divergence, the per-variant fields, and the author-side `match`-arm pattern). Two prose copies of the same normative distinction live in the spec, with no single canonical owner — a small drift surface for future edits.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the offending paragraph in `docs/spec_topics/query/query-failure-and-repair.md` §"Failure modes" so that (a) the broken "above" pointer becomes an explicit hyperlink and (b) the duplicated distinction collapses to a one-sentence cross-reference, with `tool-calls.md` retained as the canonical owner.

Concrete edit:

- In `docs/spec_topics/tool-calls.md`, add an explicit anchor `<a id="failures"></a>` immediately before the `**Failures.**` bold lead (line 25) so the cross-reference target is stable and survives later heading-level reorganisation. (`tool-calls.md` already uses inline `<a id>` for `concurrency` and `parallel-tool-batch-delivery-presupposition`, so the convention is established.)
- In `docs/spec_topics/query/query-failure-and-repair.md`, replace the entire second paragraph of §"Failure modes" (currently the eight-line "`ModelToolError` is the model-loop counterpart…") with a single sentence:

  > For the `ModelToolError` ↔ `CodeToolError` divergence — the per-context distinction, the per-variant fields (`tool_call_id`, `raw_response`), and the in-loop tool-failure → tool-result feedback rule — see [Tool Calls — Failures](../tool-calls.md#failures) and [Query — Tool calls during a query](./query-tool-loop.md#tool-calls-during-a-query).

  This excises the duplicate, hyperlinks both cross-references, and removes the bare "above" relator in one stroke.

Edge cases the implementer must watch:

- The plan leaves `V13d` and `V14a` cite this paragraph indirectly. `V13d`'s **Spec** field already includes `query-failure-and-repair.md`; no change. `V14a`'s `CodeToolError` ↔ `ModelToolError` test (`asserts the closed CodeToolError enum is distinct from ModelToolError`) currently cites `tool-calls.md`; that citation remains correct and becomes the sole canonical site.
- `docs/_tools/check_links.py` should be run after the edit to confirm the two new fragment anchors (`tool-calls.md#failures`, `query-tool-loop.md#tool-calls-during-a-query`) resolve. The latter heading is `## Tool calls during a query`, which slugifies to `tool-calls-during-a-query` under the project's anchor conventions.
- Do not delete or rewrite the canonical "`CodeToolError` is distinct from `ModelToolError`…" paragraph in `tool-calls.md`; the fix consolidates *into* it, not *over* it.

## Consequence

**Severity:** cosmetic

A reader following the bare "above" pointer finds no antecedent in the file and must grep the corpus to discover the target lives in a sibling page; the duplicated `ModelToolError` / `CodeToolError` paragraph is a future drift hazard (an editor tightening one copy can leave the other stale). Neither defect changes the normative surface — the canonical owner in `tool-calls.md` is unambiguous and a careful reader reconstructs the intent.
