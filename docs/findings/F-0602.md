## Finding

In `audit-recognised-shapes.md` §*Target surface categories* the obligation reads:

> Satisfying the three in-scope categories MUST extend `SDK_SURFACE_INVENTORY`'s entry-kind taxonomy beyond the existing kinds, with at minimum one new entry kind per category not already covered: a category (3) kind for canonical-`ctx` member access …; a category (1) kind for `pi.<member>` references that are not function-member capabilities (`pi.registerFlag`, `pi.getFlag`, …); and a category (2) kind for named imports from the four `@earendil-works/*` peer packages **where not already covered by an existing entry kind** (the existing `type-union-snapshot` entry for `SessionShutdownEvent.reason` and the existing `namespace-function` entries for the nine factory-probable function-member entries are the two existing-coverage cases).

The trailing "where not already covered by an existing entry kind" attaches only to the category-(2) clause and admits two incompatible readings:

1. *Conditional* — the MUST applies only to the residual surface the two existing kinds (`type-union-snapshot`, `namespace-function`) do not reach; if a contributor argues every category-(2) reference is covered by an existing kind, no new category-(2) kind is required at all.
2. *Unconditional with carve-out* — a new category-(2) kind MUST be coined; it merely scopes its membership to category-(2) surface *other than* the two existing-coverage cases.

The two readings yield different inventory taxonomies and different `V18a` Ships-when fixtures. Category-(1) and category-(3) carry no analogous hedge, which compounds the ambiguity — a reader cannot tell whether the asymmetry is deliberate (category (2) genuinely conditional, the others unconditional) or vestigial. Downstream pages reinforce reading (2) implicitly — `audit-resolution.md`'s *Category-(2) inventory join key* speaks of "new category-(2) entry kinds coined per *Target surface categories* above" without conditionality — but the source paragraph never makes that pin.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the category-(2) clause so the obligation is unconditional and the existing-coverage cases are an explicit membership carve-out rather than a conditional trigger. Replace:

> a category (2) kind for named imports from the four `@earendil-works/*` peer packages where not already covered by an existing entry kind (the existing `type-union-snapshot` entry for `SessionShutdownEvent.reason` and the existing `namespace-function` entries for the nine factory-probable function-member entries are the two existing-coverage cases).

with (suggested wording — adjust to match surrounding voice):

> a category (2) kind for named imports from the four `@earendil-works/*` peer packages. This new kind's membership excludes the two existing-coverage cases — the `type-union-snapshot` entry for `SessionShutdownEvent.reason` and the `namespace-function` entries for the nine factory-probable function members of capabilities 1, 2, 3, 4, and 6 — which continue to resolve via their existing entry kinds; every other category-(2) named import resolves via the new kind.

Edge cases to watch:
- The rewrite must not weaken the implicit existing-coverage claim: a contributor MUST NOT promote a peer-package named import into either of the two named existing kinds to dodge coining the new kind (the `namespace-function` kind is already pinned in the same paragraph as "exclusively the nine function members of the factory-probable subset", and the `type-union-snapshot` kind is similarly pinned to `SessionShutdownEvent.reason` at the `h1-scaffold.md` cite — keep both pins intact).
- The same "Satisfying the three in-scope categories MUST extend …" sentence carries the precondition that the new kinds land before the audit's first run on `main`. Rewriting the conditional must not perturb that precondition's scope (it still ranges over all three new kinds).
- Downstream paragraphs (`audit-resolution.md` *Category-(2) inventory join key*: "New category-(2) entry kinds coined per *Target surface categories* above MUST carry a `path` field …") already assume reading (2); no edits there are required once the source paragraph is unconditional.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge: one coins no new category-(2) kind (reading every peer-package import as covered by `namespace-function` or `type-union-snapshot`, since every category-(2) reference the audit currently knows about — the nine factory-probable function members and `SessionShutdownEvent` — falls into one of the two existing-coverage cases), and the audit fails open for any future or currently-overlooked category-(2) named import. The other coins a category-(2) kind and constrains its membership, matching `audit-resolution.md`'s downstream assumption. The first implementer's audit silently passes contributor-introduced peer-package imports that the spec intended to surface.
