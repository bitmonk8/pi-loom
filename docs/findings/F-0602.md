## Problem

In `audit-recognised-shapes.md` `<a id="audit-target-surface-categories">` *Target surface categories*, the obligation to coin a new category-(2) entry kind carries a trailing "where not already covered by an existing entry kind" hedge that admits two incompatible readings: *conditional* — no new kind is required if every category-(2) reference resolves via the existing `type-union-snapshot` and `namespace-function` kinds; or *unconditional with carve-out* — a new kind MUST be coined and merely scopes its membership to exclude the two existing-coverage cases. Categories (1) and (3) carry no analogous hedge, so a reader cannot tell whether the asymmetry is deliberate or vestigial. The two readings yield different inventory taxonomies and different `V18a` Ships-when fixtures; under the conditional reading the audit fails open for any future or overlooked category-(2) named import. Downstream `audit-resolution.md` *Category-(2) inventory join key* assumes the unconditional reading without the source paragraph pinning it.

## Solution approach

Rewrite the category-(2) clause of `audit-target-surface-categories` so the new-entry-kind obligation is unconditional, expressing the two existing-coverage cases — `type-union-snapshot` for `SessionShutdownEvent.reason` and `namespace-function` for the nine factory-probable function members — as an explicit membership carve-out rather than a conditional trigger. Align the clause's modality with the un-hedged category-(1) and category-(3) clauses in the same sentence.

## Solution constraints

- Out of scope: do not weaken the existing `namespace-function` pin ("exclusively the nine factory-probable function members") or the `type-union-snapshot` pin (`SessionShutdownEvent.reason`); the rewrite must not allow promoting a peer-package named import into either existing kind to avoid coining the new kind.
- The "new kinds MUST be coined before the audit's first run on `main`" precondition in the same sentence must continue to range over all three new category kinds.
