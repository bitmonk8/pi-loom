## Finding

The **Provider error mapping** paragraph on `provider-error-mapping.md` opens with a normative classification rule (the table mapping, the `TransportError` catch-all, and the version-coupling MUST), and then — without paragraph break, blockquote, or any other typographic boundary — continues inline with the italicised sub-heading *Re-validation gate (loom 1.0.0).* and two sentences whose semantic load is *deferral* rather than normative obligation:

> the fixtures exist in `npm test` and a contributor performing the bump SHOULD run them … which does not yet enumerate the run as a mechanical step of that procedure. Wiring this re-validation into the bump procedure as a mechanical gate is a recognised post-loom 1.0.0 maintenance follow-up.

That second sentence is pure process-improvement self-commentary — a confession that the MUST above lacks a mechanical gate today and a promise to revisit after 1.0.0 — wedged into the same paragraph as the MUST it shadows. The corpus has an established convention for exactly this content class: forward-compatibility / deferred-feature commentary lives either in `future-considerations/` (referenced from the topic page by a single linked sentence — see e.g. how `imports.md`, `cancellation.md`, `tool-calls.md`, etc. handle deferred features) or, when it must remain on the owning page, inside a `> **loom 1.0 seam — …**` blockquote that visually segregates seam / deferred content from MUST/SHOULD text (see `invocation.md`, `binder/binder-model-and-context.md`, `errors-and-results/queryerror-variants.md`, …). The re-validation-gate paragraph follows neither convention; it is the only inline "post-loom 1.0.0 maintenance follow-up" prose in the spec.

The same defect repeats one paragraph later: the *Provider-owned-wording presupposition* closes with a pointer sentence — "See *Re-validation gate (loom 1.0.0)* above for the corresponding post-loom 1.0.0 maintenance follow-up." — that adds no normative content beyond the cross-reference itself. It exists only because the deferred note it points at is buried in the prior paragraph rather than carrying its own anchor / link target a reader can find unaided.

## Solution Space

**Shape:** single

### Recommendation

Wrap the *Re-validation gate (loom 1.0.0).* sentences in a dedicated `> **loom 1.0 seam — re-validation gate.**` blockquote — the standing corpus convention for forward-compatibility / deferred-feature content that remains on its owning page (see e.g. `invocation.md`'s *per-call timeout* seam blockquote and `errors-and-results/queryerror-variants.md`'s *discriminator type-openness* seam blockquote). Give the blockquote a stable `<a id="re-validation-gate"></a>` anchor so any future cross-reference can target it directly. Then delete the dangling pointer sentence "See *Re-validation gate (loom 1.0.0)* above for the corresponding post-loom 1.0.0 maintenance follow-up." from the *Provider-owned-wording presupposition* paragraph — once the seam blockquote carries its own anchor, the pointer sentence's only function (telling the reader where the deferred note lives) is served by the layout.

Do **not** relocate the deferred note to `future-considerations/`: the note's purpose is to flag, *at the very point where the version-coupled MUST is stated*, that loom 1.0.0 has no CI-wired enforcement of that MUST. Moving it to `future-considerations/` would force a reader of the MUST to follow a link to discover the gap; keeping it adjacent (but typographically demarcated) preserves the local-context value while removing the placement defect. The blockquote treatment is the convention the corpus already uses for exactly this trade-off.

Implementer edge cases:
- The **Provider seed-field mapping** paragraph at the bottom of the same file does **not** carry the parallel inline deferred-note text any more (it was rewritten to describe the now-wired `Api`-coverage assertion); no co-edit is needed there.
- Preserve the existing cross-references from `version-bump-step2.md` item `(i)` to *Re-validation gate (loom 1.0.0)*; the new anchor (`#re-validation-gate`) should be picked up by the link, not the prose label, so anchor stability is required.
- The seam blockquote must keep the existing `provider-overflow-wording-presupposition` and `classifier-input-surface` anchors on their respective paragraphs intact — only the **Provider error mapping** opening paragraph splits.

## Consequence

**Severity:** cosmetic

A reader scanning the **Provider error mapping** paragraph for the classification rule has to skip past four sentences of process / deferral commentary mid-paragraph to reach the table, and an implementer auditing the spec for MUSTs cannot trivially separate normative load-bearing prose from acknowledged-gap self-commentary because the two share a paragraph. The classification semantics itself is unaffected; the cost is purely organisational and reader-attention.
