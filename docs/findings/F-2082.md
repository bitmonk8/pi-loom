## Finding

The *Wire serialisation* clause in `audit-wire-and-canary.md` pins the audit's stdout output as line-delimited records — "one record per line, terminated by a single `\n`" — and packs the four record fields (`path`, `line`, `symbol`, `proposed-resolution`) into tab-delimited segments on that line. It then constrains the field values asymmetrically:

- All four fields "MUST NOT contain the ASCII tab character," justified by the assertion that "file paths are tab-free by their pinned shapes."
- Only `proposed-resolution` carries an explicit newline prohibition ("MUST be plain printable ASCII text (no tabs, no newlines, no non-ASCII characters)").

No field other than `proposed-resolution` is forbidden from containing a newline, and the `path`-specific assertion addresses only tabs. The record-terminator is itself a newline, so a `path` value containing an LF or CR would inject a spurious line break into the middle of a record, producing two (or more) physical lines where the contract promises one. A CI parser routing by leading-token prefix would then see a malformed continuation line with no `audit/<class>/<family>/<symptom>` discriminator, mis-route it, or treat it as a parse failure. The `path` value originates from the audited source tree, whose in-scope set is defined purely by the glob `src/**/*.ts` (`audit-resolution.md` — *Audit scope*); a POSIX filename may contain any byte except `/` and NUL, so a newline-bearing path can match the glob and reach the wire layer. The "tab-free by their pinned shapes" claim is the only stated defence and it is silent on line terminators.

## Solution Space

**Shape:** single

### Recommendation

Add one rule to the *Wire serialisation* clause, on the same footing as the existing tab prohibition: a walked in-scope file whose `path` value contains an ASCII tab **or** any line terminator (LF `\n` or CR `\r`) MUST surface as an infrastructure failure under a stable `audit/infra/<family>/<symptom>` discriminator (fail-closed, non-zero `npm test`) rather than being emitted into a record field. Prefer detection-and-fail-closed over a bare "paths are newline-free" assertion: the in-scope set is defined by the `src/**/*.ts` glob, which does not pin filenames to be tab/newline-free, so an assertion would be unfounded for the same reason the existing tab assertion is. Detection is also consistent with the audit's existing fail-closed infrastructure-failure surface, which already routes audit-side path anomalies through `audit/infra/...`.

Edge cases for the implementer: (a) handle CR and CRLF as well as bare LF, since a record terminator is a single `\n` and an embedded CR would still corrupt downstream parsing; (b) the new discriminator belongs in the `infra` class (not a violation family), since a pathological filename is an audit-input anomaly, not an inventory-closure violation; (c) close the gap for `path` and `symbol` together — `proposed-resolution` already forbids newlines, but `symbol` carries only the no-tab prohibition.

This is one coherent defect (the wire layer accepts a record-field value that can contain the record terminator). Although the fix mirrors the tab handling and touches both the prohibition text and the infra-discriminator list, it resolves a single problem and is admitted as `single`.

## Consequence

**Severity:** advisory

On a normal repository no source path contains a line terminator, so the audit and any CI parser work as intended. But if a walked in-scope file's path contains an LF or CR, the emitted record silently splits across physical lines, corrupting the sole machine-parseable surface a CI parser is permitted to consume — a fail-open outcome the audit's fail-closed posture otherwise forbids. The wire contract has an unhandled input case that two implementers would resolve differently (emit raw, escape, or fail-closed).
