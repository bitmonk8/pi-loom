## Problem

An object schema with two fields that share the same loom-side identifier and no `as` rename (`schema X { a: string, a: integer }`) has no explicitly stated disposition in `schemas.md`. The `### Wire-name renaming` rule (`loom/parse/wire-name-collision`) is framed entirely around the explicit `as`-rename mechanism, so a reasonable implementer can read it as scoped to renamed fields, leaving a plain duplicate loom-side name with no applicable diagnostic in the closed parse registry. The asymmetry sharpens the gap: enum variant-name duplication has a dedicated `loom/parse/duplicate-enum-variant-name` with stated precedence, while object-schema field duplication has no parallel. The same `Field` form is reused by inline object types in `grammar.md`, so whichever resolution is chosen applies there transitively.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `53f5831` (2026-05-03, "spec: rewrite query primitive, system prompts, and schema syntax") established the object-schema field semantics; `de4aa88` (2026-05-04, "spec: clarify framing, ownership, influences; add wire-name renaming") added the `wire-name-collision` rule that only inferentially covers the pure-duplicate case.
**History:** `git log -S` over `docs/` confirms no `loom/parse/duplicate-field-name` token, and no "duplicate field" rule, has ever existed in the corpus. The first definition of object-schema field semantics (`53f5831`, via `git log --follow` on `schemas.md` back through the `fecb504` topic split) specified field-required/`additionalProperties:false` behaviour but never field-name uniqueness. The wire-name-renaming rule introduced at `de4aa88` carried both the "share a wire name" and "wire name collides with another field's loom name" clauses, which arguably shadow the pure-duplicate case but were authored for the `as`-rename mechanism, not for plain duplicates. No subsequent commit added an explicit rule. The gap is therefore present since the object-schema concept was first written.

## Solution approach

Clarify in `schemas.md` `### Wire-name renaming` that the collision check operates on each field's effective wire name — the explicit `as` name when present, or the loom identifier when unrenamed — so a pure loom-side duplicate participates and is reported as `loom/parse/wire-name-collision`; alternatively register a dedicated diagnostic with stated precedence against `wire-name-collision`. Cover the mixed case (`schema X { a: string, a as "b": U }`) so its disposition is not left to inference.
