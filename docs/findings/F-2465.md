## Finding

An object schema with two fields that share the same loom-side identifier and no `as` rename — `schema X { a: string, a: integer }` — has no explicitly stated disposition in `schemas.md`. The Object-schema section defines field-required semantics and the Wire-name renaming section defines `loom/parse/wire-name-collision`, whose rule reads: "Two fields in the same schema cannot share a wire name. A wire name cannot collide with another field's loom name in the same schema." Because every field's default wire name equals its loom name (the spec's own example annotates `age: integer // no rename — wire name is "age"`), a careful reader can infer that two fields named `a` share the wire name `a` and so collide. But that coverage is purely inferential: the rule's framing is entirely about the explicit `as`-rename mechanism, and a reasonable implementer can read the wire-name rules as scoped to renamed fields, leaving a plain duplicate loom-side name with no applicable diagnostic.

The asymmetry sharpens the gap. Enum variant-name duplication is handled with an explicit, named diagnostic (`loom/parse/duplicate-enum-variant-name`) plus a stated precedence ordering against `loom/parse/duplicate-enum-value`. Object-schema field-name duplication has no parallel: there is no `loom/parse/duplicate-field-name` in the closed parse registry, and `schemas.md` never states whether the pure-duplicate case is a `wire-name-collision`, a distinct error, or undefined. The diagnostic registry's stock message for `wire-name-collision` (`wire name '<name>' collides with another field on schema '<schema>'`) also reads oddly for an author who never wrote a wire name at all.

The spec should state the behaviour explicitly: either confirm that `loom/parse/wire-name-collision` covers a pure loom-side duplicate (default wire name participates) with an example, or register a dedicated diagnostic with stated precedence against `wire-name-collision`. The same `Field` form is reused by inline object types (`grammar.md` §Inline object types), so whichever resolution is chosen applies there transitively.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `53f5831` (2026-05-03, "spec: rewrite query primitive, system prompts, and schema syntax") established the object-schema field semantics; `de4aa88` (2026-05-04, "spec: clarify framing, ownership, influences; add wire-name renaming") added the `wire-name-collision` rule that only inferentially covers the pure-duplicate case.
**History:** `git log -S` over `docs/` confirms no `loom/parse/duplicate-field-name` token, and no "duplicate field" rule, has ever existed in the corpus. The first definition of object-schema field semantics (`53f5831`, via `git log --follow` on `schemas.md` back through the `fecb504` topic split) specified field-required/`additionalProperties:false` behaviour but never field-name uniqueness. The wire-name-renaming rule introduced at `de4aa88` carried both the "share a wire name" and "wire name collides with another field's loom name" clauses, which arguably shadow the pure-duplicate case but were authored for the `as`-rename mechanism, not for plain duplicates. No subsequent commit added an explicit rule. The gap is therefore present since the object-schema concept was first written.

## Solution Space

**Shape:** single

### Recommendation

In `schemas.md` §Wire-name renaming, state that the wire-name collision check operates on each field's **effective wire name** — the explicit `as` name when present, or the field's loom identifier when unrenamed. As a consequence, two fields that share a loom-side identifier with no rename (`schema X { a: string, a: integer }`) share the effective wire name `a` and are reported as `loom/parse/wire-name-collision`. Add this as a worked example in the section.

Also cover the mixed case `schema X { a: string, a as "b": U }` in the example or rule text: the two fields still share loom-side name `a`, so the same `loom/parse/wire-name-collision` fires — make this explicit so the result is not left to inference.

This aligns with the spec's existing default-wire-name = loom-name model and reuses the existing diagnostic code without adding to the closed parse registry. Optionally adjust the `wire-name-collision` message wording in `code-registry-parse.md` so it reads sensibly when no `as` rename was written (the current `wire name '<name>' collides with another field on schema '<schema>'` text is counter-intuitive for an author who never wrote a wire name).

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one reads default-wire-name participation and rejects `schema X { a: string, a: integer }` with `wire-name-collision`; the other finds no applicable rule in the closed registry and silently takes last-wins (`a: integer`), producing a different lowered schema, `required` list, and validation/equality semantics. The closed diagnostic registry means the second implementer has no obvious code to emit even once the hazard is noticed.
