## Problem

`V13b`'s third Tests bullet (mirrored in `V13b-T`) asserts the `schema-subset.md` step-4 per-query `$defs`-pruning obligation but annotates it with the prose `(SUBS code-keyed area)`, which is not a machine-resolvable token. The canonical token for `schema-subset.md` (SUBS) obligations is `cka-10`, while `V13b` itself is a facet of the unrelated `cka-12` (`query/`, QRY) row; the `H5f` per-facet citing-test gate and the `H5a` scan resolve only literal `cka-<n>` tokens, so the annotation resolves to neither. `V13b`'s bullet is the only live witness for the step-4 pruning obligation (`V5f` asserts only step 5), yet the `cka-10` closing-leaf cell lists only `` `V5d`, `V5e`, `V5f` ``, leaving the obligation untracked under the token it actually witnesses. At the loom 1.0 release gate the `cka-10` facet partition shows no facet-citing test for the pruning obligation, so it can ship unverified while the gate stays green.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `ba7da00` ("pi-loom plan: resolve \"V5d lowering-pass outputs bound by no Tests bullet\"", 2026-06-10), `7259777` ("pi-loom plan: resolve \"code-keyed obligation areas lack a canonical citable token\"", 2026-06-18)
**History:** The initial plan build (`c6a664e`, 2026-06-10) gave `V13b` no per-query-pruning bullet. `ba7da00` added the step-4 `$defs`-pruning bullet (and the matching `V13b-T` bullet, plus a `V5d` Deps edge) with the prose label `(SUBS code-keyed area)`, placing the SUBS live witness inside the QRY leaf `V13b`; at that time no `cka-<n>` token system existed, so a prose area label was the only available form. `7259777` later introduced the canonical code-keyed-area tokens — `cka-10` for `schema-subset.md`, `cka-12` for `query/` — authoring the `cka-10` row's closing-leaf cell as `` `V5d`, `V5e`, `V5f` `` and never sweeping the pre-existing `(SUBS code-keyed area)` prose in `V13b`/`V13b-T` into a `cka-10` citation nor recording `V13b` as a `cka-10` contributor. The defect crystallised at `7259777`: the token became machine-resolvable, the prose label became stale and unresolvable, and the pre-existing `V13b` contribution fell outside the new `cka-10` cell.

## Solution approach

Repoint the third Tests bullet's `(SUBS code-keyed area)` annotation in `V13b` and `V13b-T` to cite the `cka-10` token, whose linked `schema-subset.md` page owns the obligation the bullet asserts. In `coverage-matrix.md`, add `V13b` to the `cka-10` row's closing-leaf cell as a co-witness, using the literal `co-witness` annotation token so `H5f`/`H5d` exclude it from the facet partition rather than creating a new per-facet obligation. Leave `V5f` as the owner of the step-4 pruning logic.

## Solution constraints

- In the `cka-10` closing-leaf cell, leaf-IDs stay backtick-delimited and the co-witness annotation prose carries no backtick span (write `$defs` bare), per `coverage-matrix.md` *Closing-leaf column authoring*.
- Out of scope: re-citing `V13b`'s other (non-SUBS) Tests bullets to the `cka-12` QRY token — that is owned by the same-cluster `cka-12` prose-label finding.
