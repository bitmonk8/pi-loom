## Finding

In `spec_topics/pi-integration-contract/subagent.md`, the **Conversation drive — subagent mode** block defines the untyped-query result as the assistant text assembled from the terminal `agent_end` event's `messages` array (anchor `subagent-untyped-query-ok-extraction`). The extraction has no transport-failure pre-check: it short-circuits only on the `willRetry: true` retry guard and on cancellation. A turn that fails at the provider level still emits a terminal `agent_end` whose `messages` array contains an `assistant` entry with `stopReason: "error"` and no usable text; under the rule as written, that turn resolves the query to `Ok("")` — silently, with no `QueryError` raised.

The prompt-mode driver already has the symmetric probe (`conversation-drive.md` anchor `prompt-mode-error-detection`): after `waitForIdle()` resolves, the runtime reads the trailing `assistant` message's `stopReason` and maps `"error"` to `Err(QueryError { kind: "transport", ... })`. That rule does not transfer to subagent mode for free, because it is written against `ctx.sessionManager.getEntries()` / `getLeafId()` — `ReadonlySessionManager` surfaces on the prompt-mode `ExtensionCommandContext` — whereas the subagent extraction reads the `messages: AgentMessage[]` payload delivered on the terminal `agent_end` event itself (consumed via `session.subscribe(...)` per the spawn block, and explicitly *not* via `AgentSession.messages` or any sessionManager surface). The subagent block therefore needs its own analogue, anchored on the same terminal-event payload it already consumes.

The same gap applies to a subagent-mode typed query's free phase, which routes through `session.sendUserMessage(...)` on the same agent_end-event extraction path; the typed query's forced respond turn runs off-session through pi-ai's `complete()` and is already covered by [Provider error mapping](pi-integration-contract/provider-error-mapping.md).

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `024215d` (2026-06-01) — *pi-loom spec: resolve "untyped-query Ok(string) extraction mechanism (prompt + subagent)"* — moved the subagent `Ok(string)` extraction to read from the terminal `agent_end` event's `messages: AgentMessage[]` payload (the `subagent-untyped-query-ok-extraction` anchor); at that point the prompt-mode side had an `AgentSession.errorMessage` post-`waitForIdle()` probe and the subagent side had no symmetric probe.
- `d37c563` (2026-06-05) — *pi-loom spec: resolve "Prompt-mode errorMessage probe unreachable from specified surface"* — replaced the prompt-mode `AgentSession.errorMessage` probe with the `stopReason: "error"` probe of the trailing `assistant` message read from `ctx.sessionManager.getEntries()` / `getLeafId()` (the `prompt-mode-error-detection` anchor). The new rule is written against the prompt-mode `ReadonlySessionManager` surface; the subagent extraction was not updated in the same edit, so the failure-detection asymmetry persists across two distinct delivering surfaces.

**History:** `git log -S "agent_end" -- docs/spec.md docs/spec_topics/pi-integration-contract.md docs/spec_topics/pi-integration-contract/subagent.md` and `git log -S 'stopReason: "error"'` locate the two edits above; subsequent commits to `subagent.md` (e.g. `cb3a2f2`, `7238428`, `61b4063`) touch unrelated parts of the spawn block and do not address the gap. The defect is the interaction between the two: each commit is internally consistent for the side it edits, but together they leave the subagent driver without the probe the prompt-mode driver now has.

## Solution Space

**Shape:** single

### Recommendation

In `subagent.md`'s **Conversation drive — subagent mode** block, insert a transport-failure short-circuit immediately before the `Ok(string)` assembly in the `subagent-untyped-query-ok-extraction` paragraph, on the same terminal `agent_end` event the extraction already consumes:

- The runtime inspects the trailing `assistant` message of the terminal `agent_end` event's `messages: AgentMessage[]` array (the same array the assistant-text concatenation walks) for an `AssistantMessage.stopReason` of `"error"`. The trailing-`assistant` boundary is the same one the assistant-text rule already presupposes; no new turn-grouping rule is introduced.
- If `stopReason: "error"` is present, the query resolves to `Err(QueryError { kind: "transport", message: <errorMessage>, http_status: null, provider: <resolved-model-provider>, retryable: false })`, with `message` falling back to `"provider transport failure"` when `errorMessage` is absent and `provider` populated from the loom's resolved `model:` `Model<Api>.api`. The synthesised `QueryError` is identical in shape to the prompt-mode mapping, so the `[Query]`, `queryerror-variants.md`, and **Runtime event channel** always-log obligations carry over unchanged.
- Order the short-circuits explicitly: (1) `loomAbort.signal.aborted` → `Err(QueryError { kind: "cancelled" })`; (2) `stopReason: "error"` on the terminal event's trailing `assistant` message → `Err(QueryError { kind: "transport", ... })`; (3) otherwise → assemble `Ok(string)` per the existing rule. This matches the prompt-mode ordering ("cancellation short-circuit and the `stopReason: "error"` transport short-circuit ... have both been passed").
- Apply the same rule on the subagent free-phase path of a typed query; the forced respond turn is unaffected (it routes through pi-ai's `complete()` and the [Provider error mapping](pi-integration-contract/provider-error-mapping.md) table already covers it).
- Add a sentence noting that the existing [`AgentMessage[]` leading-`user` and chronological-ordering presuppositions](pi-integration-contract/host-interfaces-core.md#messages-leading-user-message-presupposition) cover the new probe — no new presupposition is needed — and extend the [Pi version bump procedure](pi-integration-contract/version-bump-step2.md) item (m) "Prompt-mode transport-failure `AssistantMessage` fields" so its re-validation scope explicitly covers the subagent-mode terminal `agent_end` `messages` delivering surface too (or, if cleaner, mint a sibling subagent-mode item adjacent to (m)).

Implementer note: the probe reads the same payload the assistant-text concatenation already walks, so a single pass over the terminal event's `messages` array yields both the failure-detection verdict and the success string; the `willRetry: true` retry guard above the extraction is unchanged. A turn with `stopReason: "error"` and no trailing `assistant` message is not expected under the leading-`user` / chronological-ordering presuppositions but, defensively, should also map to the transport `Err` with the fallback `message`.

## Consequence

**Severity:** correctness

A subagent-mode loom whose query fails at the provider level returns `Ok("")` instead of `Err(QueryError { kind: "transport" })`. The loom proceeds as if the model answered the empty string, the failure is never logged on the **Runtime event channel** always-log path for `kind: "transport"`, and any downstream `?` / `match` arm that distinguishes success from failure takes the success branch on a transport-failed turn. Two reasonable implementers will diverge here — one will copy the prompt-mode probe across, the other will follow the spec literally and produce the silent-empty-string bug — and the prompt-mode/subagent symmetry the spec elsewhere asserts ("same final-assistant-text rule on both sides") is broken on the error side.
