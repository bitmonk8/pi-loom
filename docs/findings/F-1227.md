## Problem

The subagent-mode untyped-query extraction (anchor `subagent-untyped-query-ok-extraction` in `subagent.md`) assembles its `Ok(string)` result from the terminal `agent_end` event's `messages` array but has no transport-failure pre-check ‚Äî it short-circuits only on the `willRetry: true` retry guard and on cancellation. A provider-level turn failure still emits a terminal `agent_end` whose trailing `assistant` message carries `stopReason: "error"` and no usable text, so the query silently resolves to `Ok("")` with no `QueryError` raised. The prompt-mode driver has the symmetric probe (anchor `prompt-mode-error-detection` in `conversation-drive.md`) but it is written against the `ReadonlySessionManager` surface and does not transfer to the subagent terminal-event payload. The same gap affects the subagent free-phase path of a typed query.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `024215d` (2026-06-01) ‚Äî *pi-loom spec: resolve "untyped-query Ok(string) extraction mechanism (prompt + subagent)"* ‚Äî moved the subagent `Ok(string)` extraction to read from the terminal `agent_end` event's `messages: AgentMessage[]` payload (the `subagent-untyped-query-ok-extraction` anchor); at that point the prompt-mode side had an `AgentSession.errorMessage` post-`waitForIdle()` probe and the subagent side had no symmetric probe.
- `d37c563` (2026-06-05) ‚Äî *pi-loom spec: resolve "Prompt-mode errorMessage probe unreachable from specified surface"* ‚Äî replaced the prompt-mode `AgentSession.errorMessage` probe with the `stopReason: "error"` probe of the trailing `assistant` message read from `ctx.sessionManager.getEntries()` / `getLeafId()` (the `prompt-mode-error-detection` anchor). The new rule is written against the prompt-mode `ReadonlySessionManager` surface; the subagent extraction was not updated in the same edit, so the failure-detection asymmetry persists across two distinct delivering surfaces.

**History:** `git log -S "agent_end" -- docs/spec.md docs/spec_topics/pi-integration-contract.md docs/spec_topics/pi-integration-contract/subagent.md` and `git log -S 'stopReason: "error"'` locate the two edits above; subsequent commits to `subagent.md` (e.g. `cb3a2f2`, `7238428`, `61b4063`) touch unrelated parts of the spawn block and do not address the gap. The defect is the interaction between the two: each commit is internally consistent for the side it edits, but together they leave the subagent driver without the probe the prompt-mode driver now has.

## Solution approach

Add a transport-failure short-circuit to `subagent.md`'s `subagent-untyped-query-ok-extraction` paragraph, reading the trailing `assistant` message's `stopReason` from the same terminal `agent_end` `messages` array the extraction already walks and mapping `"error"` to `Err(QueryError { kind: "transport", ... })` with the same shape as the prompt-mode mapping at `prompt-mode-error-detection`. Order the cancellation and transport short-circuits to match the prompt-mode ordering. Apply the same probe on the subagent free-phase path of a typed query. Extend the version-bump procedure item `bump-checklist-prompt-mode-error-detection-fields` (m) in `version-bump-step2.md` so its re-validation scope covers the subagent-mode terminal `agent_end` `messages` delivering surface.

## Solution constraints

- Out of scope: the typed-query forced respond turn, which routes through pi-ai's `complete()` and is already covered by [Provider error mapping](pi-integration-contract/provider-error-mapping.md).
