## Finding

The `*Default-literal rendering.*` paragraph in `docs/spec_topics/binder/binder-bypass-and-envelope.md` pins the surface form of the `default=<literal>` token inside each `Parameters:` per-field line. The pin states a MUST: `<literal>` is the default rendered in the [Loom literal sublanguage](../grammar.md#loom-literal-sublanguage) surface syntax. The paragraph then enumerates round-trip examples for a primitive enum value (`Severity.High`), a string (`"hello"`), a non-empty array (`[1, 2, 3]`), and the empty array (`[]`).

The literal sublanguage admits two more default-position forms: `BareObjectLit` (an object literal whose schema is supplied by the LHS `params:` field type, e.g. `pet: Author = { name: "Ada" }`) and `NamedObjectLit` (a named-schema or variant constructor, e.g. `pet: Animal = Cat { name: "Whiskers" }` — the only admitted form for discriminated-union variants). Both shapes are explicitly legal `params:` defaults — `frontmatter/frontmatter-fields-a.md` uses the variant form as its worked example. Neither shape has a Default-literal reference rendering, and no other section in the corpus pins a canonical literal printer.

Surface syntax alone does not pick a byte sequence for these shapes. The grammar declares field order free (`Field rules` in `grammar.md`), and admits a trailing comma; whitespace between tokens is unconstrained by the grammar. A loom-1.0 implementer rendering `pet: Animal = Cat { name: "Whiskers" }` could emit `default=Cat { name: "Whiskers" }`, `default=Cat{name:"Whiskers"}`, `default=Cat {name: "Whiskers"}`, or any permutation of fields when the schema declares more than one — each is a conforming reading of the MUST. Item 4 of the system-prompt structure list and the *Parameter-line reference renderings* table both depend on byte-deterministic per-field lines, and the four reference-rendering rows pin those bytes only for scalar/array defaults; object/variant defaults sit in an under-pinned gap.

## Issue introduction

**Verdict:** present-since-inception (of the *Default-literal rendering* paragraph)
**Introducing commit:** `5a4ec51` — *pi-loom spec: resolve "Binder system prompt information content lacks verifiable field set"* (2026-05-06)
**History:** The *Default-literal rendering* paragraph and the *Type display* / structured-items list were added together in `5a4ec51`. The paragraph enumerated round-trip examples only for `Severity.High`, `"hello"`, `[1, 2, 3]`, and `[]` from the moment it landed; no object- or variant-default example was ever included. Three later commits touched the surrounding text — `52219c4` (broad spec-review pass, dropped a "V16a parses" phrase), `4188412` ("Parameters block: indentation and per-field token order are not normative", added the *Parameter-line reference renderings* table without extending the default-literal set), and `f5e89f4` (split the monolithic `spec_topics/binder.md` into `binder/binder-bypass-and-envelope.md` and siblings) — but none added object/variant coverage. The gap is therefore an inception defect of the paragraph that has persisted across every subsequent edit.

## Solution Space

**Shape:** multiple

### Option A — Pin a canonical literal printer and add reference renderings

**Approach.** Author a normative canonical-literal printer in `grammar.md` (alongside the existing literal-sublanguage productions) that fixes whitespace, field-ordering, and trailing-comma behaviour for every `Literal` production, then have the *Default-literal rendering* paragraph cite it. Add two reference-rendering rows covering one `BareObjectLit` and one `NamedObjectLit` (variant) default to nail the bytes down.

**Spec edits.**
- `grammar.md` (Loom literal sublanguage): add a `### Canonical literal rendering` sub-section pinning, for each `Literal` arm, the deterministic surface form — for `BareObjectLit` / `NamedObjectLit`: fields in the schema's declared order, exactly one U+0020 between `Ident` head and `{`, exactly one U+0020 after `{` and before `}` when non-empty (none when empty), `<key>: <value>` separated by `, ` with no trailing comma, recursion into nested literals. Cite `BNDR-4` / `BNDR-5` as the in-corpus precedent for numeric tokens.
- `binder/binder-bypass-and-envelope.md` (*Default-literal rendering*): replace "rendered in the Loom literal sublanguage surface syntax" with "rendered by the canonical literal printer (grammar.md §Canonical literal rendering)"; extend the round-trip example list to include a `Cat { name: "Whiskers" }` variant default and an inline-object `{ x: 1, y: 2 }` bare-object default with the exact byte sequence each produces.

**Pros.** Single source of truth for byte-determinism; reusable from any future site that needs to render a literal (system-note echo already needs much of this for BNDR-4/5/6); resolves the related composite/nested type-display gap by precedent.
**Cons.** Edits two pages; the canonical printer is a new normative artefact that future readers must learn.
**Risks.** The printer rules must be consistent with what AJV/the binder-envelope schema actually accept after round-trip; field-ordering rules in particular have to match the schema's declared-order discipline used elsewhere in the corpus.

### Option B — Add reference renderings inline, defer the general printer

**Approach.** Extend the *Default-literal rendering* paragraph with a normative reference rendering for one bare-object default and one variant default, pinning whitespace and field order inline in `binder-bypass-and-envelope.md` rather than via a grammar-level printer. Use the same byte-exact MUST style the *Parameter-line reference renderings* table already uses.

**Spec edits.**
- `binder/binder-bypass-and-envelope.md` (*Default-literal rendering*): after the existing scalar/array round-trip sentences, append two reference renderings stating the exact emitted bytes for a named bare-object default (with schema `Pt { x: integer, y: integer }`, default `{ x: 1, y: 2 }` → `default={ x: 1, y: 2 }`) and a variant default (`Animal = Cat | Dog`, default `Cat { name: "Whiskers" }` → `default=Cat { name: "Whiskers" }`), and add a normative sentence pinning field order to the schema's declared order, the trailing-comma policy (none), and the single-space whitespace inside braces.

**Pros.** Localised edit; no new abstraction; resolves the immediate byte-determinism gap.
**Cons.** Re-pins whitespace rules that the related composite/nested-type-display finding also needs; risks duplication if Option A lands later.
**Risks.** Inline normative bytes accumulate as more literal shapes ship (nested objects, arrays of objects, unions of objects); each becomes a new reference rendering rather than a printer application.

### Recommendation

Take **Option A**. The composite/nested type-display finding (see Related Findings) names the same byte-determinism gap one layer up the rendering stack, and the existing BNDR-4/5/6 numeric-rendering rules are already a partial canonical printer for scalars — the corpus is converging on a canonical-printer pattern, and extending it once in `grammar.md` is cheaper than re-pinning whitespace twice. Implementer edge cases to call out in the new sub-section: (1) field order in `BareObjectLit` / `NamedObjectLit` follows the LHS (or named) schema's *declared* order, not source order from the `params:` site; (2) the empty bare-object form `{}` and variant constructor `V {}` produce `{}` and `V {}` with no inner space; (3) recursion into nested literals re-applies the canonical printer at every level, including nested arrays/objects/variants; (4) cite `BNDR-4`/`BNDR-5` for numeric children rather than re-stating them.

## Consequence

**Severity:** correctness

Two reasonable V11d implementations will emit different bytes for the `default=<literal>` token whenever a `params:` field declares an object or variant default. That divergence falsifies the byte-reproducibility goal that the system-prompt-structure list and the reference-rendering table establish for the surrounding per-field line, and it exposes the binder to non-deterministic system-prompt hashes when authors use a variant default — the very pattern `frontmatter-fields-a.md` advertises.
