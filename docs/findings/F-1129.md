## Problem

The *Default-literal rendering* paragraph in `binder/binder-bypass-and-envelope.md` pins `default=<literal>` only for scalar and array defaults (`Severity.High`, `"hello"`, `[1, 2, 3]`, `[]`). The Loom literal sublanguage also admits `BareObjectLit` and `NamedObjectLit` defaults (e.g. `pet: Animal = Cat { name: "Whiskers" }`), which are legal `params:` defaults but have no canonical reference rendering. Because `grammar.md`'s `Field rules` leave field order free, allow a trailing comma, and leave inter-token whitespace unconstrained, two conforming implementations can emit different byte sequences for an object or variant default. That divergence falsifies the byte-reproducibility the *System-prompt structure* item 4 and the *Parameter-line reference renderings* table establish for the per-field line.

## Issue introduction

**Verdict:** present-since-inception (of the *Default-literal rendering* paragraph)
**Introducing commit:** `5a4ec51` — *pi-loom spec: resolve "Binder system prompt information content lacks verifiable field set"* (2026-05-06)
**History:** The *Default-literal rendering* paragraph and the *Type display* / structured-items list were added together in `5a4ec51`. The paragraph enumerated round-trip examples only for `Severity.High`, `"hello"`, `[1, 2, 3]`, and `[]` from the moment it landed; no object- or variant-default example was ever included. Three later commits touched the surrounding text — `52219c4` (broad spec-review pass, dropped a "V16a parses" phrase), `4188412` ("Parameters block: indentation and per-field token order are not normative", added the *Parameter-line reference renderings* table without extending the default-literal set), and `f5e89f4` (split the monolithic `spec_topics/binder.md` into `binder/binder-bypass-and-envelope.md` and siblings) — but none added object/variant coverage. The gap is therefore an inception defect of the paragraph that has persisted across every subsequent edit.

## Solution approach

Add a canonical literal-rendering rule to `grammar.md`'s `#loom-literal-sublanguage` that pins a deterministic surface form for `BareObjectLit` and `NamedObjectLit` defaults — fixing field ordering, inter-token whitespace, and trailing-comma behaviour so any two implementations emit identical bytes, including for nested literals. Rewrite the *Default-literal rendering* paragraph in `binder/binder-bypass-and-envelope.md` to cite that rule instead of the bare surface-syntax reference, and extend its round-trip example list to cover one variant default and one bare-object default.

## Solution constraints

- The canonical rendering rule governs printing only; it MUST NOT alter the free-field-order parse acceptance in `grammar.md`'s `Field rules`.
