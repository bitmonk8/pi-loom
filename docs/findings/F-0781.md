## Problem

The standalone paragraph closing `runtime-value-model.md` § JavaScript engine assumptions (line 47) asserts the Node-exclusivity scope decision as a free-standing normative claim ("loom 1.0 targets Node exclusively … Adding a non-Node host is … out of loom 1.0 scope"). That disposition is canonically owned by `model-changes-and-non-goals.md` § loom 1.0 non-goals item "No non-Node JavaScript host support", whose *Recorded at:* trailer designates this section as a back-reference recording site, not a primary owner. A reader of the value-model page in isolation cannot tell the paragraph is a back-reference rather than a primary normative claim, and a maintainer reworking the non-goals item must remember to also update a third site that does not advertise itself as part of the lock-step group.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `52219c4` (2026-05-06, "Address all 41 spec-review findings"); `0e12140` (2026-05-08, "pi-loom spec: resolve \"Non-goals are not consolidated into a single section\"")
**History:** Commit `52219c4` migrated host-runtime obligation 4 out of `spec.md` and created the new `## JavaScript engine assumptions` section in `runtime-value-model.md`; the Node-exclusivity paragraph (then *"V1 targets Node exclusively. Bun, Deno, browser embeds, and other JavaScript hosts are out of V1 scope; …"*) shipped as the second paragraph of that new section, where it was the only normative statement of the scope decision in the corpus. Two days later commit `0e12140` introduced the consolidated `V1 non-goals` list in `future-considerations.md` (later moved to `future-considerations/model-changes-and-non-goals.md`) and added the orientation-grade restatement *"loom 1.0 targets Node exclusively (Bun, Deno, browser-embed are out of loom 1.0 scope)."* to `spec.md`'s Host runtime block (now `overview-and-orientation.md` § Prerequisites — Host runtime). At that point the scope decision acquired a canonical owner (the non-goals item) and a designated orientation site (Host runtime), but the original `52219c4` paragraph in `runtime-value-model.md` was not demoted to a cross-reference; it remained an independent normative-sounding statement. The defect is the interaction of the two commits — neither was wrong in isolation — rather than a single introducing edit. Subsequent `V1` → `loom 1.0` rename sweeps (`a6d82db`, `f5e89f4`, `81bf52a`) re-wrote the prose without changing its placement.

## Solution approach

Demote the line-47 paragraph in `runtime-value-model.md` § JavaScript engine assumptions to an engine-prerequisite back-reference: have it presuppose the Node host floor and point to the canonical owner at `model-changes-and-non-goals.md#loom-1-0-non-goals`, replacing the free-standing normative scope assertion with that pointer. Leave the engine-prerequisite framing intact so the surrounding non-checked-invariant prose still reads as contingent on the Node floor.

## Solution constraints

- Out of scope: the canonical non-goals item in `model-changes-and-non-goals.md` and the orientation-grade restatement in `overview-and-orientation.md#orientation` — both are owned elsewhere and already correct.
- Do not empty the section's recording-site content: the non-goals item's *Recorded at:* trailer names this section as a lock-step recording site (GOV-30), so a back-reference must remain present here.
