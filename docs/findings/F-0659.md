## Finding

The opening paragraph of `docs/spec_topics/pi-integration-contract/conversation-drive.md` (anchor `#no-extra-mediation`) carries three independent defects bundled into one block of prose:

1. **Scope/placement.** The paragraph states a runtime-wide trust-boundary disposition — "the runtime interposes no privilege layer," "MUST NOT introduce additional access channels (sandbox, capability filter, mediated proxy) in any loom 1.x release," and the host-side denial-surface contract. None of this is conversation-drive material; it is the runtime's cross-cutting trust/security posture, which the overview already advertises under `## Non-functional requirements › **Trust boundary**` and which `future-considerations/model-changes-and-non-goals.md › No per-loom sandbox or capability model` cross-references as the owning normative pin. An operator or security reviewer scanning `overview-and-orientation.md` cannot reasonably be expected to navigate into a page titled "Conversation drive" to find a cross-version `MUST NOT`; the Trust boundary bullet currently forward-links *into* the misplaced paragraph rather than owning it.

2. **Untestable `MUST NOT`.** "The runtime MUST NOT introduce additional access channels (sandbox, capability filter, mediated proxy) in any loom 1.x release" has no observable Then-clause and no acceptance criterion. The categories named (sandbox, capability filter, mediated proxy) are framings, not surfaces — there is no fixture or test a `V14d`-class leaf could write to assert that the runtime "has not introduced a sandbox." The companion denial-surface sentence in the same paragraph (thrown / `isError: true` → `Err(QueryError { kind: "code_tool", cause: "execution" })`, silent success forbidden) **is** observable, and is the only piece that `V14d` / `V14d-T` actually consume.

3. **"Widening or narrowing" ambiguity.** "Widening or narrowing this rule is a major-version concern" uses lattice vocabulary against a `MUST NOT` whose direction the lattice does not name: widening the prohibition (covering more access channels) tightens the runtime's posture; narrowing it (covering fewer) loosens it. The current wording treats the two as symmetric major-version triggers, but does not say which direction is the headline concern (loosening the trust posture is a breaking change for security reviewers in a way that tightening it is not) and does not anchor the rule against `GOV-15` source-language-stability vocabulary, which is where versioning posture for the loom-author-observable surface is otherwise pinned.

The three defects are independent: a fix for any one of them leaves the other two intact. They are also bundled — addressing all three in a single edit would produce a wide-surface diff covering placement, normative shape, and wording, against three different downstream consumers (the plan's `V14d` / `V14d-T` leaves, the Trust boundary aggregator, the GOV-15 versioning posture).

## Solution Space

**Shape:** multiple

The finding bundles three independent obligations. Each gets its own option, and the recommendation orders them so the smallest, scope-bounding edit lands first and the others build on a stable baseline.

### Option A — Relocate the access-channels rule to `overview-and-orientation.md › Non-functional requirements › Trust boundary`

**Approach.** Move the **No additional access channels** paragraph out of `conversation-drive.md` into `overview-and-orientation.md § Non-functional requirements`, attaching it to the existing **Trust boundary** bullet. Carry the `<a id="no-extra-mediation"></a>` anchor with it so all inbound cross-references keep resolving.

**Spec edits.**

- Delete the opening paragraph (and the preceding `<a id="no-extra-mediation"></a>` anchor) from `docs/spec_topics/pi-integration-contract/conversation-drive.md`; the file now opens on **Conversation drive — prompt mode**.
- In `docs/spec_topics/overview-and-orientation.md`, fold the normative text into the **Trust boundary** bullet at the point where it currently paraphrases "The runtime itself interposes no additional access channels …" Replace the paraphrase + cross-reference with the relocated normative text, preserving the `<a id="no-extra-mediation"></a>` anchor inline.
- Repoint the citations in `docs/spec_topics/future-considerations/model-changes-and-non-goals.md` (the `No per-loom sandbox` bullet and its `Recorded at:` line), `docs/spec_topics/tool-calls.md § Failures`, `docs/spec_topics/pi-integration-contract/host-interfaces-core.md § Tool execution from loom code`, `docs/plan_topics/V14d-tool-calls-host-denial.md`, `docs/plan_topics/V14d-T-tool-calls-host-denial.md`, and `docs/plan_topics/coverage-matrix.md` row `cka-18` to `overview-and-orientation.md#no-extra-mediation`.
- Consolidate the Trust boundary bullet's existing paraphrase against the relocated normative text — per GOV-30 aggregator-vs-source lock-step, the rule lives in exactly one place.

**Pros.** Places the trust-boundary disposition where security reviewers expect to find it; reduces `conversation-drive.md` to its actual subject; resolves the duplication between the orientation paraphrase and the misplaced normative source.

**Cons.** Touches six consumer files; requires care to preserve the anchor id verbatim so deep links keep resolving.

**Risks.** Anchor renames would break the `V14d` / `V14d-T` plan-leaf spec citations; the constraint to preserve `no-extra-mediation` mitigates this.

### Option B — Replace the untestable `MUST NOT` with an observable acceptance criterion (or demote)

**Approach.** The "MUST NOT introduce additional access channels (sandbox, capability filter, mediated proxy)" framing is not directly observable — but its operational content *is* observable: when loom code or a model-driven turn calls a Pi-hosted tool, the call arguments reach Pi unmodified and the tool's return value reaches loom unmodified (no runtime-side filtering, redaction, or proxy rewrite). Rewrite the `MUST NOT` as a positive pass-through MUST on that observable, or — if the paragraph's intent is purely a design goal for the runtime's authors rather than a runtime-observable property — demote it to a SHOULD / design-goal note and rely on the denial-surface clause for the observable contract.

**Spec edits.** Replace the untestable sentence with a pass-through MUST of the form: "For every model-driven and code-driven tool-call dispatch, the runtime MUST pass the call arguments to Pi unmodified and MUST surface the tool's return value to loom code unmodified — no runtime-side filter, redaction, or proxy rewrite of the args or the result. Authoring a sandbox, capability filter, or mediated proxy that violates this pass-through is a major-version concern." Author a `V14d` / `V14d-T` companion bullet asserting the pass-through property over both call paths.

**Pros.** Closes the un-anchored-MUST gate; gives `V14d-T` a second assertion that fires on a real implementation regression; aligns with the existing denial-surface clause's observational shape.

**Cons.** Requires deciding whether "no sandbox / capability filter / mediated proxy" is genuinely a runtime-observable property (in which case the pass-through formulation suffices) or a design-time author-facing constraint (in which case the right answer is demotion to a SHOULD / design-goal note). The choice is the implementer's, but it must be made.

**Risks.** A pass-through MUST has to be careful not to forbid the typed-query lowered-response-schema rewrite, the binder's compact-transcript synthesis, or other runtime-internal rewrites that are not "interposing a privilege layer."

### Option C — Disambiguate "widening or narrowing" against the rule's direction

**Approach.** Rephrase the version-governance sentence to name the direction of edit and to anchor it against GOV-15 source-language-stability vocabulary, rather than leaving "widening or narrowing" as a symmetric lattice statement.

**Spec edits.** Replace "widening or narrowing this rule is a major-version concern" with explicit text such as: "Loosening this rule — introducing any new runtime-side access channel that mediates loom-to-Pi calls — is a breaking change deferred to loom 2.0 (per GOV-15). Tightening the rule — removing an existing access channel exposure — is also a breaking change for any loom that relied on the exposure, and is likewise deferred." (Exact wording is the implementer's; the constraint is that *loosen* / *tighten* replace *widen* / *narrow*, and that the GOV-15 anchor is cited.)

**Pros.** Aligns the paragraph with the GOV-15 vocabulary used elsewhere for major-version dispositions; removes the directional ambiguity.

**Cons.** Cosmetic on its own; only valuable once Options A and B have been applied (otherwise the rephrased sentence is still in the wrong file, and still attached to an untestable `MUST NOT`).

**Risks.** None beyond the wording itself.

### Recommendation

Resolve in order A → B → C.

A is the scope-bounding edit: it determines which file the rule lives in and which anchor downstream consumers cite, so deferring it forces B and C to be re-applied across two files. Land A first, in its own commit, with the anchor preserved verbatim so `V14d` / `V14d-T` and `coverage-matrix.md cka-18` keep resolving.

B lands second, on the relocated paragraph. The implementer must decide whether the access-channels rule is runtime-observable (pass-through MUST) or design-time-only (SHOULD / design-goal demotion); both close the un-anchored-MUST gate, but only the pass-through form gives `V14d-T` a new assertion. The denial-surface clause already in the paragraph is untouched by B — it stays verbatim because the `V14d-T` test consumes it directly.

C is cosmetic and lands third, once the sentence is in its final home and its normative shape is settled. Address it then or defer to a clarity-pass commit — but do not combine it with A or B, since the diff surface for each is already broad enough on its own.

The audit's decision-log discipline applies: do not add `<!-- previously placed in conversation-drive.md -->` comments or "rephrased from 'widening/narrowing'" annotations to the relocated paragraph; the git history carries that.

## Consequence

**Severity:** correctness

The denial-surface clause that `V14d` and `V14d-T` consume is correct and testable, so the implementation slice is not blocked. But the surrounding `MUST NOT introduce additional access channels` cannot be discharged by any leaf — an audit pass asking "which leaf closes this MUST?" finds none, and the H5e un-anchored-MUST text-scan gate will flag it. A future contributor who adds a thin proxy around a Pi API (for telemetry, retry, etc.) has no observable rule to consult and no test that fires, while a security reviewer scanning the orientation NFR is silently redirected into a conversation-drive page to find the rule they care about. The "widening/narrowing" wording independently misleads anyone running the version-bump procedure about which direction of edit triggers a major-version concern.
