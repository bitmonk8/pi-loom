## Finding

`V18d`'s **Adds** field embeds a standalone `**Blast radius.**` paragraph: "A divergent pin's blast radius is every runtime leaf — it is real-host-only and is **not** witnessed by the double-backed acceptance gate above, which is why the pre-merge real-host smoke is the gate that closes the window." This is a risk / scope qualification, not a description of what the leaf introduces, and `Blast radius.` is not a leaf-format field (the format defines `Spec.` / `Adds.` / `Tests.` / `Deps.` / `Ships when.` per `leaf-template.md` and `conventions.md` *Leaf format*). Lodged inside `Adds.` as a bolded pseudo-heading, it reads as a deliverable to a contributor scanning for what to build, while a reviewer scanning for risk notes does not find it under any recognized heading.

The scope-limit the paragraph asserts is already carried elsewhere in the leaf: the first **Tests** bullet states "a green surface-inventory run alone does not satisfy this gate … a green double-backed run is not real-host coverage," and the **Adds** body's closing sentence ("The static surface-inventory / `engines.node` / peer-dep / reason-snapshot gates are owned by `V18c`") already records the V18c cross-attribution. The `Blast radius.` paragraph therefore adds non-conforming structure without adding normative content.

## Solution Space

**Shape:** single

### Recommendation

Strike the `**Blast radius.**` heading and its sentence from `V18d`'s **Adds** field — the literal span `**Blast radius.** A divergent pin's blast radius is every runtime leaf — it is real-host-only and is **not** witnessed by the double-backed acceptance gate above, which is why the pre-merge real-host smoke is the gate that closes the window.` Leave the following sentence ("The static surface-inventory / `engines.node` / peer-dep / reason-snapshot gates are owned by `V18c`.") in place; it states cross-leaf ownership and belongs in Adds. Deleting the struck span loses no normative content: the "a green double-backed run is not real-host coverage" scope-limit is already asserted in the first **Tests** bullet. If the author wants the divergent-pin risk framing retained for reviewers, it belongs as a non-normative note outside the leaf-format deliverable fields rather than as a bolded pseudo-field inside `Adds.`

## Consequence

**Severity:** advisory

A non-conforming bolded pseudo-field inside `Adds.` splits the reader's attention: deliverable-scanners treat a risk note as build scope, and risk-scanners cannot locate the scope-limit under a recognized field. Because the "double-backed run is not real-host coverage" limit is already stated in the Tests bullets and the V18c ownership in the Adds body, no implementer diverges; the cost is leaf-format drift and redundant prose.
