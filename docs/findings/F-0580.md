## Finding

`V18d`'s `Adds.` field carries a sentence opening **"Blast radius."**: *"A divergent pin's blast radius is every runtime leaf — it is real-host-only and is **not** witnessed by the double-backed acceptance gate above, which is why the pre-merge real-host smoke is the gate that closes the window."* Per the leaf template, `Adds.` is descriptive-by-default and states what the leaf *introduces* (the new module / surface / gate). The "Blast radius." sentence introduces nothing; it is a risk / scope-of-impact rationale explaining *why* the pre-merge real-host smoke exists. Its substantive claim — that a divergent pin is not caught by the double-backed `H4a` acceptance gate and so the real-host smoke is the closing gate — is already carried by `V18d`'s `Ships when` (which gates the revert on a "confirmed real-host-smoke behavioural-divergence finding") and by the `Tests.` bullet noting "a green double-backed run is not real-host coverage."

The field then closes with *"The static surface-inventory / `engines.node` / peer-dep / reason-snapshot gates are owned by [`V18c`]."* That ownership boundary is already conveyed earlier in the same `Adds.` field, which enumerates those same `V18c` gates when describing the post-revert re-run obligation, so the closing pointer restates the `V18c`/`V18d` ownership split a second time.

This is an organisational defect localised to one leaf: an explanatory aside lives in a field reserved for deliverable description.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V18d-version-bump-acceptance.md`, strike the **"Blast radius."** sentence from the `Adds.` field — the run beginning `**Blast radius.** A divergent pin's blast radius is every runtime leaf …` through `… which is why the pre-merge real-host smoke is the gate that closes the window.`

The substantive claim it carries (a divergent pin is real-host-only, the double-backed `H4a` acceptance gate does not witness it, and the pre-merge real-host smoke is the closing gate) is already stated by the `Ships when` field and the second `Tests.` bullet, so no relocation is strictly required. If the scope-of-impact note is judged worth preserving as an explicit caveat, fold it into the `Ships when` field alongside the existing real-host-smoke-divergence clause rather than reintroducing it in another descriptive field.

The closing cross-reference `The static surface-inventory / engines.node / peer-dep / reason-snapshot gates are owned by V18c.` states a cross-leaf ownership boundary and may stay in `Adds.` as plain prose; if kept, it should not be duplicated against the earlier enumeration of those same `V18c` gates in the same field.

## Consequence

**Severity:** cosmetic

The aside is descriptive rather than normative, so no implementer diverges and no gate mis-fires; the cost is a heavier `Adds.` field that mixes deliverable description with risk rationale and restates the `V18c` ownership boundary, slightly degrading scan-ability of the leaf.
