## Problem

The *Transitive-completeness plan-maintenance* clause in `conventions.md` §REQ-ID discipline closes with the claim that the `H5d` gate arm and the plan-maintenance rule "stay in lockstep," and `H5d`'s `Adds.` echoes it ("mechanically backs the *Transitive-completeness plan-maintenance* obligation"). But the `H5d` arm is a per-cell **at-least-one** test: it fails CI only when *none* of a closing-leaf cell's listed leaves is present in `H5b`'s `Deps.`, whereas the rule requires *every* genuine closing leaf. For genuine multi-facet split rows where each listed leaf closes a distinct facet (e.g. `ERR-17 | V4d, V13d`, `DISC-4 | V10a, V9m`), the rule demands all of them in `H5b`'s `Deps.` while the arm stays green when only one is present. The gate is therefore strictly weaker than the rule it claims lockstep with — it cannot catch a secondary facet-closer omitted from `H5b`'s `Deps.` while a sibling facet-closer for the same cell remains.

## Issue introduction

**Verdict:** present since the mechanism's introduction
**Introducing commits:** `08bd641` ("pi-loom plan: resolve \"H5b coverage Deps completeness has no mechanical backstop\"", 2026-06-11); propagated to the `H5d` leaf by `617a3ad` ("pi-loom plan: resolve \"H5a bundles gate arms spanning four data sources\"", 2026-06-16)
**History:** `git log -S"so this rule and the gate stay in lockstep" -- docs/plan_topics/conventions.md` returns a single commit, `08bd641`. That commit added the mechanical backstop for the transitive-completeness obligation: it introduced the at-least-one gate semantics (`stays green as long as one listed leaf is present`) and the unqualified "so this rule and the gate stay in lockstep" claim in the same edit. The overclaim has thus existed since the gate itself was first authored — the at-least-one arm was never tighter than the rule. `617a3ad` later split the gate machinery out of `H5a` into the `H5d` leaf and carried the parallel "mechanically backs … so a co-witness leaf … does not redden the arm" wording into `H5d`'s `Adds.` (`git log -S"stays green as long as one listed leaf is present"` and `git log -S"mechanically backs the" -- docs/plan_topics/H5d-transitive-completeness-gate.md`). No commit ever tightened the gate to match the rule's every-genuine-closer obligation.

## Solution approach

Rewrite the closing "stay in lockstep" assertion in `conventions.md` §REQ-ID discipline *Transitive-completeness plan-maintenance* clause so the prose describes the per-cell at-least-one invariant the `H5d` arm actually enforces and discloses the multi-facet split-row residue it does not catch. Apply the same qualification to the "mechanically backs the *Transitive-completeness plan-maintenance* obligation" sentence in `H5d`'s `Adds.`. Name the residue's real backstops — the `H5b` warn-only canary and the loom 1.0 release-time residue inspection — as the channel that surfaces every-facet-closer discipline.

## Solution constraints

- The spec corpus (`spec_topics/**`) is read-only for this fix.
- Do not weaken the `H5d` arm's actual at-least-one semantics; tightening the arm to require every facet-closer is a separable design change, out of scope for this finding.
