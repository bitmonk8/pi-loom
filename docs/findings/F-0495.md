## Finding

The `H2a` leaf (`Cross-cutting lint and architectural gates`) delivers four
independent enforcement mechanisms in a single leaf: (1) the custom
`no-broad-catch` ESLint rule, (2) the `no-restricted-syntax`
sequential-by-default rule banning unguarded `Promise.all`, (3) the
`no-restricted-syntax` blocking-runtime rule flagging `*Sync`-suffixed calls,
and (4) the `src/**` AST-scan architectural test that fails on a module-level
mutable binding. The first three are ESLint configuration/rule work; the fourth
is a distinct AST-traversal test with its own detection model. They are coupled
only by sharing `Deps: H1a` and being wired into the same `npm test` invocation —
there is no ordering or data dependency between them.

A reviewer picking up `H2a` must evaluate four separate enforcement strategies in
one pass. The leaf is still executable and reviewable in a single session, so no
implementer is blocked or misled; the concern is review granularity only.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "build/update plan for spec.md + review" — created `H2a` already bundling three enforcement mechanisms); `c33d309` (2026-06-17, "resolve 'Blocking-runtime ban concedes enforcement to manual review'" — added the fourth mechanism, the `*Sync` blocking-runtime `no-restricted-syntax` check)
**History:** `H2a` bundled multiple cross-cutting enforcement mechanisms from its creation in `c6a664e` — the `no-broad-catch` ESLint rule, the `Promise.all` sequential-by-default `no-restricted-syntax` rule, and a `src/**` architectural test (then phrased as a global/static/singleton check). The fourth mechanism, the `*Sync` blocking-runtime `no-restricted-syntax` rule, was added later in `c33d309` while resolving an unrelated enforcement-strength finding, raising the bundled count to four. The multi-mechanism bundling is therefore an inception-era structural property of the leaf, not an artefact a single later edit introduced; `c33d309` only grew the count.

## Solution Space

**Shape:** multiple

### Option A — Keep H2a as a single cross-cutting-gates leaf

**Approach:** No structural change. `H2a` continues to deliver all four mechanisms.
**Plan edits:** None.
**Spec edits:** None.
**Pros:** Zero churn; the four mechanisms share one coherent "cross-cutting gates" theme tied to `conventions.md`, all gate `src/**` and all wire into `npm test`; the leaf is reviewable in one session.
**Cons:** A reviewer evaluates four enforcement strategies in one pass.
**Risks:** None.

### Option B — Split the architectural test into its own leaf

**Approach:** Keep the three ESLint rules in `H2a`; move the module-level
mutable-binding AST-scan test into a new sibling leaf.
**Plan edits:**
- In `docs/plan_topics/H2a-cross-cutting-gates.md`: retain the `no-broad-catch`,
  `Promise.all` sequential, and `*Sync` blocking-runtime clauses in **Adds**, their
  three matching **Tests** bullets, and the lint portion of **Ships when**. Strike
  the architectural-test clause ("…and the `src/**` architectural test that fails
  on a module-level mutable binding…" through its closing parenthetical), the
  fourth **Tests** bullet (`*No globals, statics, singletons*`), and the
  module-level-mutable-binding clause of **Ships when**.
- Create `docs/plan_topics/<new>-*.md` holding the struck architectural-test
  content (Adds clause, the `*No globals, statics, singletons*` Tests bullet, and a
  Ships-when on the architectural assertion), with **Deps** `H1a` and the
  **Convention.** field naming *No globals, statics, singletons*.
- In `docs/plan.md` §Horizontal phases, add a link line for the new leaf adjacent
  to the existing `H2a` bullet.
- In `docs/plan_topics/H3a-di-seam-skeleton.md`, change the **Deps** line from
  `H2a` to `H2a, <new>` so the seam skeleton still sequences after both gate halves.
**Spec edits:** None.
**Pros:** Each leaf is a single enforcement strategy; finer review granularity.
**Cons:** Adds a leaf and a `Deps` retarget for negligible verification value.
**Risks:** The `H3a` `Deps` retarget must land or `H3a` could be picked with only half the gate infrastructure present.

### Recommendation

Take Option A — keep `H2a` as one leaf. The four mechanisms cohere as the
project's cross-cutting `src/**` gate set, all depend only on `H1a`, and the leaf
is reviewable in one session; the split buys review granularity without changing
what is verified. If review burden is judged material and Option B is taken,
create the new architectural-test leaf first, then retarget `H3a`'s `Deps` to cite
both halves, so the seam skeleton never sees a partially-present gate set.

## Consequence

**Severity:** cosmetic

If the plan ships unfixed, `H2a` remains a single-session, multi-mechanism review:
a reviewer assesses four unrelated enforcement strategies at once, reducing review
granularity. No implementer is blocked, no gate fires incorrectly, and the leaf
stays executable as written.
