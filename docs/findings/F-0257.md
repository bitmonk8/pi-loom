## Finding

`V6c` (`tools` callable set and resolution snapshot) lists "the default for absent/empty `tools:`" among the behaviours its **Adds** field promises to implement, but the leaf never states what that default actually is. An implementer reading only `V6c` / `V6c-T` cannot tell what an absent `tools:` resolves to, whether an explicitly-empty `tools: []` differs from an absent field, or whether the resulting callable set is frozen like the populated one.

The answer is unambiguous in the spec topic `V6c` already cites under its **Spec** field: `frontmatter-fields-a.md#tools` states that "If frontmatter omits `tools`, the loom runs with an **empty callable set**" and that "`tools: []` and an absent `tools:` field are equivalent" (also tabulated in the frontmatter field table as default = "empty callable set"). So the behaviour is fully specified — the gap is purely that `V6c` names the obligation without carrying the observable result inline or pointing at the clause, and neither the **Tests** bullets nor the **Ships when** condition exercise the absent/empty equivalence. The frozen-snapshot test asserts no-ambient-inheritance for a resolved set but does not cover the absent/empty case.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V6c-tools-set.md`, make the **Adds** clause carry the observable default instead of merely naming it: state that an absent `tools:` field and an explicitly-empty `tools: []` are equivalent and both resolve to an empty, frozen callable set (the model cannot make tool calls and loom code has no `<name>(...)` callables), and anchor it to `frontmatter-fields-a.md#tools`, which `V6c` already cites under **Spec**.

Add a **Tests** bullet to `V6c-T-tools-set.md` (the paired TDD driver) asserting that both an absent `tools:` field and `tools: []` resolve to the same empty, frozen callable set, and name that observable in `V6c`'s **Ships when** so the default is gated rather than implied. Mirror the bullet into `V6c`'s **Tests** as the existing bullets are mirrored between the two leaves.

## Consequence

**Severity:** advisory

The observable default for absent / empty `tools:` is unverified and not restated at the leaf, so an implementer must navigate to the cited spec topic to recover it; nothing in the leaf's gate fails if the absent-vs-empty equivalence or the empty-frozen-set behaviour is implemented wrongly. The behaviour is spec-determined, so two implementers are unlikely to diverge, but the leaf's coverage of one of its own declared obligations is silent.
