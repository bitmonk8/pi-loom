## Finding

GOV-15's equivalence promise binds the runtime to produce, for a given input across two loom 1.x releases, identical `(b) ordered diagnostic-code sequences`. The reviewer-inspection step the same rule pins is supposed to decide that equality on a release diff. But the corpus nowhere defines *what order* the sequence is in — no rule sequences diagnostic codes across the lifetime of a single load+run for the purpose GOV-15(b) needs. "Identical ordered sequence" is therefore not a decidable comparison: a reviewer (or the deferred conformance-fixture suite GOV-15 forward-links) is free to project the same multiset of emitted codes onto two different sequences and reach opposite verdicts.

The ordering rules that do exist are scoped to other surfaces and do not aggregate into a GOV-15(b)-usable sequence:

- `V7a` (and the `implementation-notes.md` *Static-resolution load pass*) pins a `(file, line, col)` sort across the parse-time `Diagnostic[]` aggregated per entry-`.loom`, but that batch is delivered in a single `loom-system-note` per `.loom` and covers only `loom/parse/*` / `loom/load/*` codes assembled at load time.
- `ERR-14` (`errors-and-results/queryerror-variants.md`) pins the canonical sort on a single `validation_errors[]` array — entries *inside one diagnostic*, not codes across the run.
- `pi-integration-contract/runtime-event-channel.md` pins the always-log-set's exactly-once-per-occurrence guarantee, `occurred_at` stamping (via `Clock.wallNow()`), and the dedup tuple `(kind, query_site, message, occurred_at)`. It explicitly does *not* assert a total order over emissions, and `occurred_at` is wall-clock, which the same rule warns can collide and which GOV-15 itself excludes as an independent observable.

There is no spec-side bridge from these per-surface rules to the cross-cutting "ordered diagnostic-code sequence" GOV-15(b) names. Until one exists, the equivalence verification step the rule mandates has no operational footing.

## Solution Space

**Shape:** multiple

### Option A — Define a single canonical run-wide emission-time order on `diagnostic-shape.md`, forward-link from GOV-15(b)

**Approach.** Add a new normative paragraph on `diagnostics/diagnostic-shape.md` (e.g. *Run-wide diagnostic-code sequence*) that pins the canonical projection: the ordered sequence of *codes* emitted by a single load+run is the sequence of `code` values from the consecutive `loom-system-note` envelopes the runtime emits for that invocation, on the channel pinned by `runtime-event-channel.md`, in **emission order on the always-log set's origin emissions only** (boundary cascade-twin re-emissions and the multi-error batched `loom/parse/*` / `loom/load/*` envelope per `.loom` are projected by their already-pinned sub-orders: the parse-time batch contributes its `(file, line, col)`-ordered codes per `V7a`/IMPL, the boundary re-emission is dropped from the projection per the dedup tuple already defined). Replace observable (b) on GOV-15 with a forward-link to the new paragraph; restate the orientation bullet on `overview-and-orientation.md` in GOV-30 lock-step.

**Spec edits.**
- `diagnostics/diagnostic-shape.md`: new paragraph and anchor.
- `governance/source-language-stability.md` GOV-15: observable (b) becomes "the run-wide diagnostic-code sequence defined at [Diagnostics — Run-wide diagnostic-code sequence](...)".
- `overview-and-orientation.md` source-language-stability bullet: same forward-link in lock-step.

**Pros.** Single canonical home; reusable by the *Ceiling-set carve-out* and *Attribution test* without re-deriving an order; preserves observable (b) as a *sequence* (richer than a multiset, catches order-perturbing regressions).

**Cons.** Wall-clock-derived emission time is the natural projector but GOV-15 already excludes wall-clock as an independent observable; the rule must be careful to project on emission *order* without leaking wall-clock *values* into the equivalence test. Requires a tie-break rule for emissions stamped at the same `occurred_at`.

**Risks.** Under the always-log set's exactly-once-per-origin guarantee, sibling invocations interleaving on the event loop (per `implementation-notes.md` *Per-invocation single-threaded execution*) can produce a non-deterministic interleaving across invocations; the rule has to scope the sequence to a single load+run-of-a-single-input, matching GOV-15's "for any given input" framing.

### Option B — Narrow observable (b) to a multiset, drop "ordered"

**Approach.** Reword GOV-15(b) from "ordered diagnostic-code sequences" to "the multiset of emitted diagnostic codes" (or "the set of emitted diagnostic codes plus per-code emission counts"). This drops the ordering obligation entirely and binds only the set/multiset.

**Spec edits.**
- `governance/source-language-stability.md` GOV-15: observable (b) replaced with multiset language.
- `overview-and-orientation.md` source-language-stability bullet: lock-step rewording.
- Audit the *Ceiling-set carve-out*'s "multiplicity (number of emissions per query) changes solely as a behavioural consequence of the ceiling-set change" clause — already framed in multiplicity terms, so the rewording aligns.

**Pros.** Simplest fix; eliminates the need for any new ordering rule; the *Ceiling-set carve-out* already reasons in multiplicity terms.

**Cons.** Loses the ability to catch a regression that re-orders the same codes (e.g. a re-architected pipeline that emits a `loom/runtime/internal-error` *before* the `loom/runtime/cancelled-by-session-shutdown` it caused, instead of after). The carve-out's *Attribution test*, which keys on per-emission message content rather than position, mostly tolerates this — but a reviewer loses one signal.

**Risks.** Future GOV-15 consumers (the deferred conformance-fixture suite, future audit rules) that wanted to rely on order would have to be retrofitted; the rule's expressive power is permanently reduced.

### Option C — Split observable (b) by phase: ordered for the parse-time batch, multiset for runtime

**Approach.** Replace observable (b) with two sub-observables: (b₁) the per-`.loom` parse-time `Diagnostic[]` batch ordered by `(file, line, col)` per `V7a`/IMPL; (b₂) the multiset of emitted runtime-channel codes. Each sub-observable forward-links to its existing canonical home.

**Spec edits.**
- `governance/source-language-stability.md` GOV-15: observable (b) split into (b₁) and (b₂) with the two forward-links.
- `overview-and-orientation.md` source-language-stability bullet: lock-step rewording.
- No new ordering rule is needed in `diagnostics/diagnostic-shape.md`.

**Pros.** Re-uses every ordering rule already in the corpus; introduces no new normative content beyond a definitional split; respects the actual structural difference between the two emission surfaces (one batched per `.loom`, one per-occurrence on a channel).

**Cons.** Re-numbers observables (still three letters: (a), (b₁) + (b₂), (c) — or relabel as (a), (b), (c), (d)); the carve-out and *Attribution test* must be re-anchored to the new sub-labels in the same commit.

**Risks.** Lock-step churn across the rule, the orientation aggregator, the *Ceiling-set carve-out*'s "(b) and (c)" wording, and the *Diagnostic-registry carve-out*'s "observable (b) sequence and observable (c) content" wording. Manageable with a single coordinated diff but the diff is wider than Options A or B.

### Recommendation

Adopt **Option C**. The defect's root cause is that observable (b) bundles two structurally different emission surfaces (the per-`.loom` parse-time batch the runtime emits as a single envelope, and the per-occurrence always-log set delivered over the lifetime of a run) under one ordering word, when only the first has a canonical order pinned and the second already has its only stable invariants pinned in multiplicity terms. Splitting (b) along that pre-existing structural seam re-uses both canonical homes (V7a/IMPL for the parse-time batch; the always-log-set's per-occurrence emission contract for the runtime channel) without inventing a new ordering rule, and aligns observable (b) with how the *Ceiling-set carve-out* already reasons about runtime multiplicity. Edge cases the implementer must watch: (i) the boundary cascade-twin re-emission rule in `runtime-event-channel.md` produces a second wire-visible emission with the same `occurred_at` — the multiset projection must collapse these per the existing dedup tuple, otherwise observable (b₂) double-counts; (ii) the parse-time batch's `(file, line, col)` order is per-entry-`.loom`, so a single run that loads multiple `.loom`s must project per-`.loom` batches in load order to keep (b₁) well-defined across the run; (iii) the lock-step diff must touch GOV-15, the orientation aggregator, the *Ceiling-set carve-out*'s "(b) and (c)" references, and the *Diagnostic-registry carve-out*'s "(b) sequence and (c) content" references in the same commit per GOV-30.

## Consequence

**Severity:** correctness

The reviewer-inspection step GOV-15 names cannot return a decidable verdict on observable (b): two reviewers walking the same release diff can each pick a different sequencing of the same emitted codes and reach opposite conclusions on equivalence. The deferred conformance-fixture suite, which GOV-15 already forward-links and which the *Attribution test* and the *Ceiling-set carve-out*'s in-scope predicate both lean on for post-hoc verification, inherits the same gap — it cannot mechanically diff "the three observables" against a frozen loom 1.0.0 baseline while one of them has no canonical projection. Downstream rules (the carve-out's `(b)/(c)` coverage, the *Attribution test*'s "minimal edit" predicate keyed on canonical-message content) lose their grounding.
