## Problem

BNDR-9 pins the detection rule for a non-transcript-safe `CustomMessage.customType` and requires the binder to abort transcript construction while emitting `loom/runtime/custom-type-unsafe`, but it never states what the binder invocation does next: whether it proceeds with the rest of the system prompt (dropping the offending Session-context content) or fails outright with a user-facing system note. The companion `loom/runtime/custom-type-unsafe` row in `code-registry-runtime.md` asserts that the affected invocation's argument binding does not proceed, contradicting BNDR-9's silence. The failure-mode templates table in `determinism-cancellation-failure.md` (the canonical source for user-facing binder system notes per BNDR-3) carries no row for this cause, so even the "loom never starts" reading leaves the user-facing string undefined. The V11b / V11b-T conformance leaves can assert the diagnostic fires but have no oracle for the surrounding invocation outcome.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `9568906` (2026-06-09) — *pi-loom spec: resolve "custom:<type> role tag customType safe-character constraint"*
**History:** `git log -S "custom-type-unsafe" -- docs/` returns a single content-introducing commit, `9568906`, which added BNDR-9 to `binder-model-and-context.md` and the `loom/runtime/custom-type-unsafe` row to `code-registry-runtime.md` in the same change. That commit introduced both the "abort transcript construction" phrasing in BNDR-9 and the divergent "argument binding does not proceed" phrasing in the diagnostic row, and did not touch `determinism-cancellation-failure.md` to add a failure-mode-templates row. `git log --follow` on `binder-model-and-context.md` shows no subsequent commit narrowing BNDR-9's silence or reconciling the registry-row text. The gap — undefined invocation-level outcome, contradiction with the registry row, and missing failure-mode-templates row — has been present since BNDR-9 first existed.

## Solution approach

Clarify BNDR-9 to define the binder's invocation-level outcome when it rejects an unsafe `customType` — whether the binder call proceeds (omitting the Session-context block, for which BNDR-7i's void-truncation rendering already supplies the omission) or fails the invocation. Reconcile the `code-registry-runtime.md` `loom/runtime/custom-type-unsafe` row's `the affected slash/prompt invocation's argument binding does not proceed` clause so both sites agree on the chosen outcome. Extend the V11b / V11b-T acceptance assertions to oracle the surrounding invocation outcome, not only that the diagnostic fires.

## Solution constraints

- If the chosen outcome fails the invocation, the user-facing system note MUST resolve through a `determinism-cancellation-failure.md` failure-mode-templates row per BNDR-3, so the string is not left undefined.
