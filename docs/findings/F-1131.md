## Finding

BNDR-9 (in `binder-model-and-context.md`) pins the detection rule for a non-transcript-safe `CustomMessage.customType` and requires the binder to "abort transcript construction for that invocation" while emitting `loom/runtime/custom-type-unsafe`. It says nothing about what the binder call *does next*: two equally defensible readings sit side by side. Reading A is that "abort transcript construction" is local to the renderer — the offending custom message (and, conservatively, the whole Session-context block) is dropped and the binder proceeds with the remainder of the system prompt, reusing the BNDR-7i void-truncation rendering. Reading B is that the binder call itself is aborted, the loom never starts, and the operator sees a user-facing system note.

The companion registry row in `code-registry-runtime.md` explicitly takes Reading B — "the affected slash/prompt invocation's argument binding does not proceed" — but BNDR-9 itself never says so, and the failure-mode templates table in `determinism-cancellation-failure.md` (which BNDR-3 pins as the canonical source for user-facing binder system notes) carries no row for this cause. There is therefore both an internal contradiction (BNDR-9 silent vs registry row decisive) and a downstream gap (no template, so the user-facing string is undefined even under Reading B). The V11b/V11b-T conformance leaves can assert that `loom/runtime/custom-type-unsafe` fires, but have no oracle for the surrounding invocation outcome.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `9568906` (2026-06-09) — *pi-loom spec: resolve "custom:<type> role tag customType safe-character constraint"*
**History:** `git log -S "custom-type-unsafe" -- docs/` returns a single content-introducing commit, `9568906`, which added BNDR-9 to `binder-model-and-context.md` and the `loom/runtime/custom-type-unsafe` row to `code-registry-runtime.md` in the same change. That commit introduced both the "abort transcript construction" phrasing in BNDR-9 and the divergent "argument binding does not proceed" phrasing in the diagnostic row, and did not touch `determinism-cancellation-failure.md` to add a failure-mode-templates row. `git log --follow` on `binder-model-and-context.md` shows no subsequent commit narrowing BNDR-9's silence or reconciling the registry-row text. The gap — undefined invocation-level outcome, contradiction with the registry row, and missing failure-mode-templates row — has been present since BNDR-9 first existed.

## Solution Space

**Shape:** multiple

The two viable readings each pick a different invariant. Pick one and reconcile both sites; do not leave BNDR-9 silent.

### Option A — Bind-proceeds-without-context (drop the block, finish the bind)

**Approach.** Have BNDR-9 say that on detecting any non-transcript-safe `customType` in the candidate context, the binder drops the entire Session-context block, emits `loom/runtime/custom-type-unsafe`, and proceeds with the remainder of the binder system prompt unchanged. The transcript rendering reuses BNDR-7i's void-truncation rule: the opening line, body, and terminating blank line of the Session-context block are all omitted. The binder call itself then runs normally; the loom starts.

**Spec edits.**
- BNDR-9: append a sentence stating that transcript construction aborts by omitting the entire Session-context block per BNDR-7i, the binder call proceeds, and `loom/runtime/custom-type-unsafe` is emitted as an operator-visible companion diagnostic (`display: false`).
- `code-registry-runtime.md` `loom/runtime/custom-type-unsafe` row: replace "the affected slash/prompt invocation's argument binding does not proceed" with text matching the proceed-with-omitted-block outcome.
- `determinism-cancellation-failure.md` failure-mode templates: no new row needed (the binder succeeds or fails through its existing classes).

**Pros.** Smallest blast radius; reuses an existing rendering rule (BNDR-7i); does not invent a new user-facing system-note template; an unsafe `customType` in a single session message does not gratuitously block the whole loom invocation.

**Cons.** Lossy without surfacing to the user — the binder grounds against a smaller context than the caller's session implies, and the operator only learns through `display: false` diagnostics.

**Risks.** Authors may forget that `bind_context: session` is silently downgraded to `bind_context: none`-equivalent for this turn; worth a non-normative authoring note.

### Option B — Bind-fails (the loom does not start)

**Approach.** Have BNDR-9 say that on detecting any non-transcript-safe `customType`, the binder aborts the binder call before the LLM request goes out, the loom does not start, and a user-facing system note is rendered. Add a row to the failure-mode templates table in `determinism-cancellation-failure.md` for this cause.

**Spec edits.**
- BNDR-9: append a sentence stating the binder call aborts, the loom does not start, and the user-facing system note is rendered per the new failure-mode-templates row.
- `code-registry-runtime.md` `loom/runtime/custom-type-unsafe` row: keep the existing "argument binding does not proceed" phrasing.
- `determinism-cancellation-failure.md` failure-mode templates: add a seventh row, e.g. `unsafe customType in session context` → `loom /<name>: argument binding failed — session context contains an unsupported message type`.

**Pros.** Consistent with the registry row's current wording; the operator sees the failure prominently; the binder never silently grounds against a reduced context.

**Cons.** Larger blast radius — adds a new template row, expands the failure-mode taxonomy, and means a single malformed custom message anywhere in the visible session-context window blocks the slash invocation entirely.

**Risks.** Authors / hosts that legitimately emit custom messages with unusual `customType` strings can deny-of-service their own slash commands; needs a clear authoring note that all in-session custom messages MUST keep transcript-safe `customType` values.

### Recommendation

Take **Option A**. The detection rule is a defensive transcript-byte guard, not a user-facing failure semantics; coupling it to "loom does not start" makes the binder fragile against well-behaved custom messages with awkward `customType` strings. Reuse BNDR-7i's existing whole-block omission so no new rendering rule is introduced, and reconcile the `loom/runtime/custom-type-unsafe` registry row's "argument binding does not proceed" sentence in lockstep so the two sites agree. Extend the V11b / V11b-T acceptance assertions to cover both the diagnostic firing *and* the whole-block omission. Edge cases to watch: when a single oversized newest turn already empties the context (BNDR-7i case) the unsafe-`customType` check still applies before that walk, so the order of evaluation should be pinned (recommend: detect unsafe `customType` over the post-truncation set, so excluded turns cannot trigger the diagnostic).

## Consequence

**Severity:** correctness

Two reasonable implementations diverge on whether a `/foo`-style invocation with one unsafe `customType` in its session context completes (with the session-context block omitted) or fails outright with a user-facing system note. The divergence is observable to the operator and to test oracles, and the failure-mode templates table has no row to render whichever outcome wins, so even the "loom never starts" reading leaves the user-facing string undefined.
