## Problem

The *Transitive-completeness plan-maintenance* clause in `conventions.md` §REQ-ID discipline states the trigger for adding a leaf to `H5b`'s `Deps.` as a three-item disjunction whose first item ("carries an executable REQ-ID") contradicts the carve-out attached to its second item. Read literally, item (1) fires for any leaf whose `Tests.` bullets cite an executable `PREFIX-N` REQ-ID — including a co-witness leaf that re-cites an already-closed REQ-ID — while item (2)'s carve-out ("a leaf that merely re-cites an already-closed REQ-ID does not qualify") excludes exactly that case. Two reasonable plan-maintainers therefore reach opposite conclusions about whether a future co-witness leaf must enter `H5b.Deps`, yielding divergent `Deps.` sets and divergent release-gate sequencing, each defensible against the clause as written. The intended semantics are the narrow (item-2) reading — `H5b.Deps` is the coverage-producing closing-leaf set named in `coverage-matrix.md` — which the `H5d` per-cell at-least-one gate already presupposes.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `d7173b3` ("resolve 'Transitive-completeness rule trigger broader than H5b coverage set'", 2026-06-11), with `1cd23a6` ("resolve 'Transitive-completeness trigger verb \"can introduce\" is ambiguous'", 2026-06-18) a contributing later edit.
**History:** `git log -S 'merely re-cites an already-closed' -- docs/plan_topics/conventions.md` returns a single commit, `d7173b3`. Before it the trigger read uniformly broad ("a new leaf … that can introduce an executable REQ-ID, a numbered-REQ-ID citing test, or an un-anchored normative MUST") — three items with no narrowing parenthetical, so no internal contradiction. `d7173b3` narrowed item (2) to "the citing test that *closes* a coverage-matrix-mapped numbered REQ-ID" and added the "merely re-cites … does not qualify" carve-out, but left item (1)'s broad "introduce an executable REQ-ID" wording intact, introducing the contradiction. `1cd23a6` later rephrased item (1) from "can introduce an executable REQ-ID" to "whose `Tests.` bullets carry an executable REQ-ID" — still the broad form — sharpening rather than removing the literal item-(1) reading. Each prior fix addressed one facet of the trigger; their interaction left item (1) and the item-(2) carve-out in mutual contradiction.

## Solution approach

Rewrite the *Transitive-completeness plan-maintenance* clause's trigger so the closing-leaf-mapping condition is the only REQ-ID-side arm — drop the standalone "carries an executable REQ-ID" item so a leaf qualifies on the REQ-ID side only when `coverage-matrix.md` names it as a numbered REQ-ID's closing leaf, not when its `Tests.` merely re-cite an already-closed REQ-ID. Retain the un-anchored-normative-MUST arm. The intended-boundary parenthetical already in item (2) states this semantics and can carry it.
