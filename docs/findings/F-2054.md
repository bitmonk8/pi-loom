## Finding

The `callable set` glossary entry (`docs/spec_topics/glossary.md`, line 15) closes with a three-item `See:` list whose third link reads `[Pi Integration Contract — Tool-registration lifetime and visibility](./pi-integration-contract.md)`. The link **text** names a specific sub-page section, but the **URL** resolves to the Pi Integration Contract hub `./pi-integration-contract.md`, not to the section's actual home, `./pi-integration-contract/tool-registration-lifetime.md`.

The named section exists as a dedicated sub-page with a stable anchor: `docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md` carries `<a id="tool-registration-lifetime-and-visibility"></a>**Tool-registration lifetime and visibility.**`. Other inbound references already cite it directly — e.g. `host-prerequisites.md` links `./tool-registration-lifetime.md#tool-registration-lifetime-and-visibility`, and the sibling `occurrence vs. origin` glossary entry uses the corresponding sub-page URL for its target. An implementer following this entry's link lands on the hub index and must navigate one further hop to reach the section the link text promised.

## Issue introduction
**Verdict:** introduced-by-later-commit (page-split refactor that left an inbound reference stale)
**Introducing commit:** `f5e89f4` — "docs: enforce 100KB hard / 30KB recommended size cap on spec set" (2026-06-04)
**History:** The `callable set` glossary entry, with this exact `See:` link to `./pi-integration-contract.md`, was authored in `fec0870` ("resolve callable-set prose synonyms erode no-inheritance invariant", 2026-05-06). At that time the Pi Integration Contract was a single monolithic page `pi-integration-contract.md` that *contained* the `tool-registration-lifetime-and-visibility` section, so the hub URL was the correct target. `f5e89f4` split that monolith into sub-pages — relocating the section out to `pi-integration-contract/tool-registration-lifetime.md` (the same commit added that sub-page file) — but did not repoint this inbound `See:` link, leaving the link text naming a sub-page while the URL still resolves to the now-hub page. A later fix (`d75f2a8`) repointed the analogous `runtime-event-channel` references but did not touch this one.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/glossary.md`, in the `callable set` entry's `See:` list, change the third link's URL from `./pi-integration-contract.md` to `./pi-integration-contract/tool-registration-lifetime.md#tool-registration-lifetime-and-visibility`. The `#tool-registration-lifetime-and-visibility` anchor exists on the sub-page and matches the form used by `host-prerequisites.md`; if matching the bare sub-page-path style of the sibling `occurrence vs. origin` entry is preferred, the anchor may be dropped, but the path component must change from the hub to the sub-page. Link text is already correct and needs no edit.

## Consequence

**Severity:** cosmetic

A reader following the glossary link lands on the PIC hub rather than the named section and must navigate further; no implementation behaviour or correctness diverges. The mismatch also breaks the corpus convention — visible in the sibling `occurrence vs. origin` entry — that a `See:` whose text names a sub-page links that sub-page directly.
