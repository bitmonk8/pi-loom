## Finding

Four distinct passages in the pi-integration-contract shard pin a *mechanism* (a specific variable name, a specific control-flow shape, a specific coercion call, a specific data-structure type) where the contract that operators and the build are actually entitled to observe is a *predicate or wire payload*. The four are independent — they live in three different files and were added at four different commits — but they share a single failure mode: they reach past the observable boundary into implementation tissue, then add a normative MUST around the tissue.

The four sites:

1. **`session-only-degraded-state.md` §"Seam minimality (normative)"** — the `session_shutdown` handler "MUST NOT introduce any handler-scoped state seam beyond `pinnedConstantReadOk` and `capturedEventReason` — no `branchTakenWasClosedSet` flag, no per-sub-step success roll-up, …". Variable count and name are not observable; what is observable is the *Predicate split* table outcome (which `(reason, snapshot-OK)` cells fire the degraded tag write, the diagnostic, both, or neither). An implementation that achieves the same predicate-split outcomes with a different number of local variables is non-conformant under the current text but indistinguishable from the air.
2. **`unknown-reason-rule.md` §*Lookup-failure-to-discriminator routing* arm (1)** — the handler "MUST guard the lookup with an explicit `SDK_SURFACE_INVENTORY === undefined` check **before** invoking the iteration primitive". The observable obligation is that arm (1) (live-binding gap / circular-init `undefined` inventory) routes to discriminator `"missing-entry"`, not to the throw arm. The `=== undefined` guard is one of several mechanisms that satisfy that routing — a `typeof` check, an `if (!SDK_SURFACE_INVENTORY)` early-return, a defensive iteration primitive that treats `undefined` as a missing key, or a wrapping `try` whose catch arm distinguishes `TypeError: Cannot read … of undefined` from other throws all reach the same `details.failure === "missing-entry"`.
3. **`unknown-reason-rule.md` §`details.observed` coercion** — "Pin the coercion to `String(event.reason)` explicitly — a template-literal coercion … re-introduces the symbol-throw bug `String()` is chosen to avoid." The rationale (template-literal symbol-throw) is legitimate but the obligation it pins is again mechanical. The observable contract is: `details.observed` is a string whose construction does not itself escape the handler, with the literal `"<unreadable>"` fallback when coercion throws. An implementation that uses `try { return String(x) } catch { return "<unreadable>" }` and one that uses a custom `safeToString(x)` helper with the same behaviour both satisfy the wire contract; pinning the function-call shape blocks the second without distinguishing it from the first observably.
4. **`tool-registration-lifetime.md` *Prompt mode* paragraph** — "The runtime maintains an extension-scoped registration cache: `Map<schemaSlug, registeredToolName>`, where …". The observable contract is the cache's behaviour: keyed by the canonical schema slug from [Schema Subset — Canonical schema hash](../spec_topics/schema-subset.md#canonical-schema-hash), storing the registered tool name **and** the canonical-form bytes (so the byte-equality verification on a slug-cache hit can run without re-serialisation, per the very next paragraph), and producing the documented `pi.registerTool` call pattern and `registration-cache-collision` diagnostic. `Map<K, V>` is one container choice; a `Record<string, …>`, a pair of arrays, an LRU with infinite capacity, or any other associative structure are all conformant. The pinned type signature also misstates the cache's value — the next paragraph requires the cache to store canonical bytes alongside the name, which the `Map<schemaSlug, registeredToolName>` signature does not carry — so the signature is not only over-prescriptive, it is internally inconsistent with the obligation immediately below it.

These are four independent obligations, each fixable by one edit to one paragraph. None of the four prevents the other three from being shipped first.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `27ec53d` (2026-05-04, "Resolve Cluster-A: per-mode tool registration") added the `Map<schema-hash, registeredToolName>` signature for the prompt-mode registration cache; `6cd6c56` (2026-05-07, "details.event.reason coercion is unspecified for non-string Pi values") added the "Pin the coercion to `String(event.reason)` explicitly" sentence; `75a9626` (2026-05-09, "SessionShutdownEvent.reason closed set has no build-time pin against the SDK type") added the `SDK_SURFACE_INVENTORY === undefined` guard MUST as part of the four-arm lookup-failure-to-discriminator routing; `52b9101` (2026-05-11, "T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state") added the seam-minimality "MUST NOT introduce any handler-scoped state seam beyond …" clause.
**History:** Each commit's stated intent was sound — pin a specific bug-class (symbol-coercion throw, circular-init `undefined`, branch-flag drift, ambient-cache visibility) by naming the mechanism that avoids it. The defect is not in any single commit: each one named a real failure mode and chose a mechanism that closes it. The defect emerges from the cumulative pattern — four sites where the rationale supports an observable predicate but the prose pins the mechanism — which only becomes visible when the four sites are read together as a class. The cluster therefore reads as a missing **prescription-vs-observable** authorial-checklist gate at spec-edit time rather than as any one commit's regression.

## Solution Space

**Shape:** multiple

The four obligations are independent and each is a one-paragraph edit. They are listed in recommended landing order — narrowest scope and smallest blast first, so the wider seam-minimality rewrite in option A lands on the smallest possible baseline.

### Option D — Re-state the registration cache by behaviour, not type signature (`tool-registration-lifetime.md`)

**Approach.** Replace the type-signature pin with a behavioural specification: a per-extension-scope associative collection keyed by canonical schema slug (sourced via [Schema Subset — Canonical schema hash](../spec_topics/schema-subset.md#canonical-schema-hash)), whose entry value carries at minimum the registered Pi tool name and the canonical-form schema bytes the byte-equality check immediately below consumes. Drop the `Map<schemaSlug, registeredToolName>` literal; keep every other paragraph (the content-addressed name format, the `pi.registerTool`-once protocol, the byte-equality verification, the `registration-cache-collision` disambiguation, the disambiguated-name format) unchanged.

**Spec edits.** One sentence in the *Prompt mode* paragraph. Approximate replacement: *"The runtime maintains a per-extension-scope associative cache keyed by the canonical schema slug; each entry carries the registered Pi tool name and the canonical-form schema bytes (consumed by the byte-equality check below). On first encounter of a unique slug the runtime calls `pi.registerTool` once …"*

**Pros.** Smallest scope (one sentence, one file); resolves the value-omission defect (canonical bytes now in the entry contract instead of only in the next paragraph's prose); does not touch any acceptance test.

**Cons.** None observable.

**Risks.** None — the next paragraph already pins the byte-equality verification; option D only aligns the cache value with what that paragraph already requires.

### Option C — Replace the `String(event.reason)` pin with a coercion-contract predicate (`unknown-reason-rule.md`)

**Approach.** Replace "Pin the coercion to `String(event.reason)` explicitly — a template-literal coercion … re-introduces the symbol-throw bug `String()` is chosen to avoid" with an observable obligation on `details.observed`: the value MUST be a string whose construction does not escape the handler, and that under each of the closed set of input shapes (symbol, `undefined`, `null`, number, boolean, ordinary object, throwing-`toString` object, throwing property access) produces the documented payload (per-shape value, or the `"<unreadable>"` sentinel where the rule already documents it). Keep the existing inner `try`/`catch` clause and `"<unreadable>"` fallback — those *are* observable contract surface — and keep the rationale parenthetical ("a template-literal coercion re-introduces the symbol-throw bug") as advisory prose, not as a normative MUST on the call shape.

**Spec edits.** One sentence in the `details.observed` coercion sub-paragraph; the inner-throw `"<unreadable>"` fallback clause already pinned later in the paragraph is unchanged.

**Pros.** Implementer-choice between `String(x)`, a `safeToString(x)` helper, or any other coercion routine preserved without ambiguity; symbol-throw failure mode still pinned (it is one of the input shapes the predicate enumerates).

**Cons.** Implementers lose one named-mechanism cue; the per-shape conformance table needs to be either explicit in prose or cited from a fixture.

**Risks.** If the per-shape table is not made explicit, "does not escape the handler" by itself is weaker than the current text — symbol-coercion throw could re-emerge under a coercion choice that swallows it differently. Mitigation: enumerate the eight input shapes inline in the same paragraph (the rule already names seven of them; the eighth — `null` — is currently implicit under "objects").

### Option B — Replace the `=== undefined` guard MUST with a routing obligation on arm (1) (`unknown-reason-rule.md`)

**Approach.** Replace "the handler MUST guard the lookup with an explicit `SDK_SURFACE_INVENTORY === undefined` check **before** invoking the iteration primitive (a naive `SDK_SURFACE_INVENTORY.find(…)?.literals` pattern routes this case to the throw arm instead …)" with an obligation on the arm (1) routing outcome: when `SDK_SURFACE_INVENTORY` resolves to `undefined` (the circular-init / live-binding-gap case), the handler MUST route to `details.failure === "missing-entry"`, **not** to a `"throw:TypeError: …"` value. The mechanism that achieves the routing — a guarded read, a `typeof` check, a defensive iteration primitive, an outer `try` whose catch arm classifies `TypeError: Cannot read … of undefined` — is leaf-owned, on the same footing as the iteration primitive itself (which the very same paragraph already declares leaf-owned).

**Spec edits.** One sentence in arm (1) of the *Lookup-failure-to-discriminator routing* sub-paragraph; the symmetry with the existing "iteration primitive … is **leaf-owned** … routing is **spec-owned**" framing makes the rewrite a single-sentence substitution.

**Pros.** Symmetric with the routing-vs-primitive separation the same paragraph already pins for the iteration primitive; keeps the operator-deduplication guarantee intact (the routing-to-`"missing-entry"` obligation is the load-bearing pin, not the `===` operator).

**Cons.** Conformance fixture for arm (1) must construct the live-binding-gap state, which is unchanged from today — the existing acceptance test for arm (1) already requires it.

**Risks.** None — the routing obligation is strictly weaker than the current text *only* in the implementation-mechanism dimension; the observable arm-(1)-to-`"missing-entry"` contract is unchanged.

### Option A — Re-state seam minimality as a Predicate-split-table outcome (`session-only-degraded-state.md`)

**Approach.** Replace the "MUST NOT introduce any handler-scoped state seam beyond `pinnedConstantReadOk` and `capturedEventReason` — no `branchTakenWasClosedSet` flag, no per-sub-step success roll-up, and no separate `!pinnedConstantReadOk` disjunct or `"<unreadable>"` sentinel-string equality check to back arms (2)/(3)" sentence with an observable predicate over the Predicate-split table the same page already defines: the handler's diagnostic-emission and tag-transition outcomes on each `(event.reason, snapshot-OK)` cell of the closed product space MUST match the documented Predicate-split outcome, including the snapshot-failure-with-closed-set-`event.reason` exclusion and the snapshot-failure-with-`quit`/`reload` harmlessness rationale immediately above. Keep the two **load-bearing choices** bullet list (the `pinnedConstantReadOk` rationale and the captured-string-triplet-check rationale) — those *are* contract surface because they pin what each predicate is computing from — but reframe them as informative explanations of *why* the predicate-split outcomes are what they are, not as named-variable MUSTs.

**Spec edits.** The "Seam minimality (normative)" paragraph plus its two load-bearing-choice bullets. The acceptance test `V9l` currently carries (`§Seam-minimality + *Inline triplet is normative* (un-anchored; GOV-22 residue): no handler-scoped state seam is introduced`) becomes a Predicate-split-table-coverage test against the same cells.

**Pros.** Largest blast-radius reduction of the four; aligns the rule with the wire / diagnostic contract it actually enforces; resolves the `V9l` acceptance-test wording at the same time.

**Cons.** Touches the most prose; the predicate-split-table outcome enumeration must be tight enough to recover the snapshot-failure-with-closed-set-`event.reason` exclusion that the current `pinnedConstantReadOk` variable name carries implicitly.

**Risks.** The current text's `pinnedConstantReadOk` and `capturedEventReason` variable names are referenced by `unknown-reason-rule.md` (via the *Predicate split* clause and the `pinnedConstantReadOk` seam cross-reference) and by `V9h-unknown-reason-rule.md` / `V9l-session-only-degraded-branch.md`. The rewrite MUST preserve those cross-references — either keep the variable names as illustrative inline anchors (with the observable-predicate framing layered above) or sweep both the unknown-reason-rule paragraph and the two plan leaves in the same commit per the unknown-reason-rule "anchor-stable substrings" inbound-reference-sweep discipline.

### Recommendation

Land the four options as four separate commits in the order D → C → B → A. The order is dictated by blast radius: D is a one-sentence edit to a single paragraph with no inbound cross-references; C and B are one-sentence edits to a single paragraph each, with no cross-references to the changed wording; A is the largest rewrite and the only one that owes an inbound-reference sweep against `unknown-reason-rule.md` and the two plan leaves. Landing A last means it rebases onto a stable D-C-B baseline rather than the other way round, and means the per-finding spec-diff-fix loop sees one observable-vs-mechanism reframing per round instead of four bundled. Each commit should also delete the dependent acceptance-test wording that today names the mechanism (the `V9l` "no handler-scoped state seam is introduced" Test bullet; the `V9h` test wording around the `=== undefined` guard) and replace it with the corresponding observable-outcome assertion. Implementer edge case to watch on option A: the load-bearing `pinnedConstantReadOk` rationale must survive the rewrite as informative prose — silently dropping it re-opens the snapshot-failure-with-closed-set-`event.reason` exclusion the variable encodes.

## Consequence

**Severity:** advisory

A conformant implementation cannot be distinguished from a non-conformant one by any observable an operator, test, or build sees — both produce the same diagnostics, the same wire payloads, and the same registration-cache behaviour. The harm is forward: a reasonable implementer who chooses a different mechanism (a `typeof` guard, a `safeToString` helper, a `Record` for the cache, a single-variable encoding of the snapshot-OK state) is non-conformant on a wording technicality, and a reviewer who reads the spec literally will reject the PR without being able to point to an observably-broken test. The `Map<schemaSlug, registeredToolName>` signature also independently misstates the cache value — it omits the canonical bytes the next paragraph requires — which is a documentation defect even before the over-prescription concern.
