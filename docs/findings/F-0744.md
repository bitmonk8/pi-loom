## Problem

The settings-file shape-failure surface carries two coordinated gaps. First, `package-and-settings.md` §"Failure modes" defines only the missing/unreadable (`loom/load/settings-unreadable`) and invalid-JSON (`loom/load/settings-invalid-json`) outcomes, while §"Top-level shape validation" inspects the `loomPaths` / `looms` keys presupposing the parsed JSON root is an object; a settings file whose root parses as a non-object JSON value (`[1, 2, 3]`, `42`, `true`, `"hello"`, bare `null`) matches neither rule, so its handling is undefined. Second, `loom/load/settings-value-out-of-range`'s trigger now fires on top-level shape failures whose offending value can be a JSON array or object, but the §8 parsed-scalar carve-out in `placeholder-rendering-b.md` enumerates `<observed>` rendering only for `number` / `boolean` / `null` / `string`, leaving the byte string for `array` / `object` undefined and DIAG-4's byte-identicality unsatisfiable.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `043762e` (2026-05-04) — introduced the §"Failure modes" block with exactly two arms, implicitly assuming an object-shaped root.
- `72e2c31` (2026-06-05) — introduced the §8 parsed-scalar carve-out enumerating `number`/`boolean`/`null`/`string` only; coherent then, because the row's *Trigger* covered only `looms.*` scalar keys.
- `4c00462` (2026-06-06) — extended `code-registry-load.md`'s *Trigger* to include top-level `loomPaths`/`looms` shape failures (value can be array/object) and added the `Top-level shape validation` paragraph (framed as per-key inspection presupposing an object root), leaving the §8 carve-out untouched.

**History:** The non-object-root gap is the interaction of `043762e` and `4c00462`; the non-scalar-rendering gap is the interaction of `72e2c31` and `4c00462`. `git blame` confirms `72e2c31` as author of the §8 carve-out lines and `4c00462` as author of the extended trigger. Neither commit is wrong in isolation.

## Solution approach

Clarify `package-and-settings.md`'s §"Failure modes" and §"Top-level shape validation" to define a non-fatal outcome for a valid-JSON non-object root (`[1, 2, 3]`, `42`, `true`, `"hello"`, bare `null`), including which diagnostic surfaces and whether the per-key `loom/load/settings-value-out-of-range` cascade is suppressed for such a file. Add `array` and `object` rendering to the §8 parsed-scalar `<observed>` carve-out in `placeholder-rendering-b.md`, consistent with the §7 `<kind>` closure's existing `array` / `object` tokens, and add the corresponding test vectors.

## Solution constraints

- A non-object root in one settings file MUST NOT suppress a well-formed value of the same key in the other file (the per-file isolation invariant the §"Top-level shape validation" *Merge semantics* rule already holds).
