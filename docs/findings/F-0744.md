## Finding

`docs/spec_topics/discovery/package-and-settings.md` §"Settings file reads" enumerates exactly two failure modes for a settings file: missing/unreadable (`loom/load/settings-unreadable`) and "present but not valid UTF-8 JSON" (`loom/load/settings-invalid-json`). The companion §"Top-level shape validation" paragraph then handles per-key malformations of `loomPaths` and `looms` — but it presupposes that the *root* JSON value is an object whose keys can be looked up.

A settings file whose body parses as a JSON value other than an object — `[1, 2, 3]`, `42`, `true`, `"hello"`, or the bare literal `null` — satisfies neither rule. It is valid UTF-8 JSON (so `settings-invalid-json` does not fire as currently defined), and there is no "top-level container kind" check before the per-key inspection runs. The two-line failure-modes block and the per-key shape paragraph between them never name the behaviour, so the spec does not say whether such a file is treated as `{}`, whether it crashes the loader as it attempts to read `.loomPaths` from a non-indexable root, or which diagnostic surfaces. Two conforming implementations will diverge.

The neighbouring rules establish the *idiom* the missing rule should follow — per-file isolation, non-fatal diagnostic, fall through to the other file's value and to built-in defaults — but stop short of authoring it for this case.

**History:** Commit 043762e introduced the §"Settings file reads" → "Failure modes" block with exactly two arms (missing-or-unreadable and invalid-JSON), implicitly assuming any parsed JSON root would be object-shaped. Commit 4c00462 later added the "Top-level shape validation" paragraph to handle per-key wrong-typed `loomPaths` / `looms`, but framed it as key-by-key inspection that presupposes an object root rather than a top-level container-kind check. The two paragraphs together cover the (parse-fail) and (object root with bad keys) cases but leave (valid JSON, non-object root) in the gap between them — the defect is the interaction, not either commit alone.

## Solution Space

**Shape:** multiple

### Option A — Reuse `loom/load/settings-invalid-json`, broaden its definition

**Approach.** Extend the "Failure modes" bullet for `loom/load/settings-invalid-json` to cover "present, parses as JSON, but the parsed root is not a JSON object." Add a sentence to §"Top-level shape validation" of `package-and-settings.md` stating that a settings file whose parsed root is not a JSON object is treated as `{}` for the merge, contributes no keys, suppresses the per-key `settings-value-out-of-range` cascade for that file, and logs exactly one `loom/load/settings-invalid-json` diagnostic per file.

**Spec edits.**
- `package-and-settings.md` §"Failure modes": reword the second bullet to "File present but its body is not a JSON object — `loom/load/settings-invalid-json`. This covers two sub-cases: (a) the body is not valid UTF-8 JSON, and (b) the body parses as JSON but the root value is not an object (array, number, boolean, string, or the bare literal `null`)."
- `package-and-settings.md` §"Top-level shape validation": prepend "If the parsed root is not a JSON object, the file is treated as absent in its entirety per the Failure modes rule above and no per-key `settings-value-out-of-range` cascades for that file."
- `code-registry-load.md`: update the `loom/load/settings-invalid-json` *Trigger* column to match.

**Pros.** Zero new diagnostic codes; one place for an author to learn that a settings file was unusable as a key-value source; placeholder-rendering surface unchanged.

**Cons.** Conflates two distinct authoring mistakes (syntax error vs. wrong root shape) under one code; the *Message* column then needs a per-sub-case rendering rule or generic prose.

**Risks.** Existing tests that grep for the exact "not valid UTF-8 JSON" message string break; the rendering surface enumerated in `placeholder-rendering-b.md` may need a §8 follow-up if the message gains a variable substring.

### Option B — Add a dedicated `loom/load/settings-not-object` code

**Approach.** Introduce a third settings-load diagnostic specifically for the valid-JSON / non-object-root case, alongside `settings-unreadable` and `settings-invalid-json`. Keep `settings-invalid-json` scoped to UTF-8 / JSON parse failures.

**Spec edits.**
- `package-and-settings.md` §"Failure modes": add a third bullet "File present and valid UTF-8 JSON but its parsed root is not a JSON object (array, number, boolean, string, or `null`) — `loom/load/settings-not-object`."
- `package-and-settings.md` §"Top-level shape validation": add the same prepended sentence as Option A, citing the new code.
- `code-registry-load.md`: register `loom/load/settings-not-object` (severity `warning`, phase `load`) following the row style of its two siblings.
- `placeholder-rendering-b.md`: confirm the new row carries only the `<path>` placeholder (category 5) and needs no new rendering rule.

**Pros.** Each code names exactly one failure mode; downstream tooling (the `loom/load/*` registry consumers, the diagnostic-shape audit) can distinguish them; message rendering stays single-shape per code.

**Cons.** Adds a new diagnostic code, which `GOV-7`/`GOV-8` count for spec-version churn purposes; one more row for the registry-audit gates (`V18b`, `V18o`) to ratify.

**Risks.** Minor: the audit must verify the new code is unique across all four registry shards and that no rendering rule is needed beyond category 5.

### Recommendation

Take **Option B**. The two failure modes are semantically distinct — a syntax error is an author typo, a non-object root is a structural authoring mistake — and the registry already pays the per-code listing cost for the same distinction in the frontmatter family (e.g. `frontmatter-invalid-yaml` vs `frontmatter-value-out-of-range`). Reuse of `settings-invalid-json` muddies the *Trigger* column and forces an interpolated `<observed>`-style placeholder where today the row has none. Edge cases the implementer must watch: bare `null` is a valid JSON value and must trip the new code (not be confused with the missing-file arm); the `settings-value-out-of-range` cascade must be fully suppressed for that file so the diagnostic count for a `[1,2,3]` file is exactly one, not one-per-recognised-key; and the per-file isolation rule still holds — a non-object root in one file MUST NOT suppress a well-formed value of the same key in the other file.

## Consequence

**Severity:** correctness

A user who ships `.pi/settings.json` as a JSON array — or who accidentally overwrites it with `null` from an editor or scripted edit — will trigger implementation-defined behaviour: one loader treats it as `{}`, another raises `TypeError: Cannot read properties of null`, a third silently returns no settings and skips the diagnostic. The other settings file's contribution and the global merge outcome diverge correspondingly. Since the surrounding failure-modes block specifically promises per-file isolation and non-fatal degradation, the silent crash or silent absence breaks the contract two readers above and below this paragraph rely on.
