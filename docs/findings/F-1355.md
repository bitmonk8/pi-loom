## Finding

The BNDR-9 oracle in `V11b` describes the unsafe-`customType` failure outcome two ways within the same leaf. The Tests bullet states the binder "**fails the affected slash/prompt invocation**", while the Ships-when clause states it fires `custom-type-unsafe` "while **failing the affected invocation**". The compound noun `slash/prompt invocation` is introduced in the Tests bullet without a glossary entry and is then silently shortened to the bare `invocation` in Ships when.

The same `slash/prompt invocation` phrasing also appears verbatim in the paired `V11b-T` Tests bullet, so the qualifier is consistent across both Tests bullets and diverges only in the `V11b` Ships-when summary. This is a within-leaf-pair wording inconsistency, not a semantic gap — the failure outcome is the same in both places.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** f41af5d — pi-loom spec-plan: fix F-1131 — define BNDR-9 unsafe-customType invocation outcome as failure (no proceed-and-drop), add the custom-type-unsafe Failure-mode-templates row, and extend V11b/V11b-T oracles to assert invocation-fails + system-note-matches (2026-06-24, Thomas Andersen)
**History:** The F-1131 fix in `f41af5d` was the edit that first introduced the invocation-failure language into both the Tests bullets and the Ships-when clause of `V11b`/`V11b-T`. In that single commit the Tests bullets were authored with `fails the affected slash/prompt invocation` while the Ships-when clause was authored with `failing the affected invocation`; the qualifier divergence is an artifact of that one edit rather than later drift. `git log -S` for both phrases against the leaf returns only `f41af5d`.

## Solution Space

**Shape:** multiple

### Option A — Carry the qualifier through to Ships when

**Approach:** Align the one divergent location to the two consistent ones.
**Plan edits:** In `V11b` Ships when, change `failing the affected invocation` to `failing the affected slash/prompt invocation`.
**Spec edits:** None.
**Pros:** Single-location, single-file edit; preserves the more specific wording already used in both Tests bullets.
**Cons:** Retains a compound term (`slash/prompt invocation`) with no glossary entry.
**Risks:** None.

### Option B — Simplify to the unqualified form everywhere

**Approach:** Drop `slash/prompt` from all three sites so every BNDR-9 mention reads `invocation`.
**Plan edits:** In `V11b` Tests, `V11b-T` Tests, and `V11b` Ships when, change `slash/prompt invocation` / `affected invocation` to the unqualified `affected invocation`. BNDR-9 binding applies to both slash and prompt modes (subagent mode is rejected at parse by `loom/parse/bind-context-session-on-subagent`), so the unqualified form is accurate.
**Spec edits:** None.
**Pros:** Removes the un-glossaried compound term entirely; uniform wording.
**Cons:** Touches three sites across two files; discards the mode-naming detail.
**Risks:** None.

### Recommendation

Apply Option A: in `docs/plan_topics/V11b-bind-context-transcript.md`, change the Ships-when phrase `failing the affected invocation` to `failing the affected slash/prompt invocation`, matching the wording already present in both `V11b` and `V11b-T` Tests bullets. This is the minimal edit (one line, one file) and keeps the more precise mode wording. No other leaf references the compound term, so no further propagation is needed.

## Consequence

**Severity:** cosmetic

A reader treating Ships when as the authoritative done-condition sees a slightly different phrase than the Tests bullets; no implementer would produce a divergent leaf, since the failure outcome (argument binding does not proceed, the loom does not run) is stated identically in both. Pure wording-consistency defect.
