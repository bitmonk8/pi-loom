## Finding

Four acceptance criteria inside the version-bump procedure are written as obligations on the bump-commit record (or on a build-time smoke check) but state their pass/fail bar in terms that two careful reviewers can read incompatibly:

1. **`version-bump-step2.md` item (a) — `provably-differs` evidentiary standard.** "A record is conformant iff it carries exactly one of the two verdicts with a cited evidence site (and, for `provably-differs`, evidence that addresses session-only instance survival specifically)". "Addresses session-only survival specifically" is the load-bearing gate: a contributor cites *some* lifecycle evidence, a reviewer reads it as bearing on rebind in general rather than on the session-only sub-case, and the record is simultaneously claimed conformant and non-conformant. Item (a) is one-shot and dispositive (it gates SM-4/SM-5/SM-6 via `V9l`), so the ambiguity has merge-blocking reach.
2. **`version-bump-step2.md` "Outcome recording conventions" — item (e) N/A definition.** The `N/A` arm is satisfied when "the candidate Pi minor has surfaced an explicit per-session dispatch-lock mechanism — i.e. a Pi-side construct (named per-session lock, single-consumer per-session dispatch channel, **or other documented Pi-public surface**)". The trailing "or other documented Pi-public surface" is an open-ended catch-all: a contributor naming an `ExtensionAPI` member, a `pi.on` event payload, or any other typed-and-documented Pi surface can plausibly claim N/A without any predicate that ties the surface to the non-overlap property item (e) is actually keyed on.
3. **`version-bump-step2b.md` branch (3) exempt — "spec section that authorises".** Branch (3) requires a `// allow-pi-surface: <REQ-ID-or-PIC-anchor> — <justification>` marker "with the citation naming the spec section **that authorises** the unenumerated reference", and the tie-break makes "the spec authorises" the gate for picking branch (3) over branches (1) and (2). "Authorises" is itself the term under dispute: nothing in the surrounding text says whether the cited section must contain a normative MUST naming the symbol, an SHOULD-level allowance, a normative example, or merely a sentence that mentions the surface. The (1)∧(3) and (2)∧(3) tie-breaks resolve nothing when the disagreement is over whether the cited section qualifies as authorising.
4. **`version-bump-step2.md` step 2(a) — category-(3) smoke-check "type-side analogue".** The literal-read assertion's branch-table arm for category-(3) entries SHOULD include a smoke check that every entry's path member-name segment "still resolves on the imported `ExtensionContext` and `ExtensionCommandContext` types — for example, a per-entry `"<member>" in <importedExtensionContextLikePrototype>`-equivalent check, **or its type-side analogue** — so a Pi minor that drops or renames a category-(3) member the inventory enumerates surfaces as a literal-read failure". The runtime form has a clear pass/fail bar (`in`-check on the imported prototype, fails if the member is missing). The "type-side analogue" alternative has none: a `keyof`-typed identity, a satisfies-clause, or an empty conditional could all be claimed as conformant analogues while never failing when the member is dropped, leaving the SHOULD-smoke arm structurally weaker than the runtime arm without flagging the demotion.

Each clause is independently fixable. Together they form a pattern — the procedure tightened its mechanical gates over several months but left these four conformance bars as natural-language judgement calls, undermining the merge-gate posture `V18c`'s Ships-when places on the recorded outcomes.

## Issue introduction

**Verdict:** multi-commit-interaction

**Introducing commits:**

- `e5cd514` (2026-05-09) — *pi-loom spec: resolve "T35 SDK capability inventory closed-set negative-direction audit"* — introduced branch (3) "spec section that authorises the unenumerated reference" and the category-(3) smoke-check "type-side analogue" wording into the predecessor monolithic `pi-integration-contract.md`.
- `57d8235` (2026-05-11) — *pi-loom spec: resolve "T23 per-session slash-handler serialisation citation"* — introduced the item (e) N/A definition's "or other documented Pi-public surface" catch-all.
- `81bf52a` — *docs: collapse spec/ section pages into spec_topics/* — preserved all four clauses verbatim across the split that produced today's `version-bump-step2.md` / `version-bump-step2b.md` pair.
- `80fcd63` (2026-06-20) — *pi-loom spec-plan: fix F-0718+F-0823 — item (a) authoritative-determination evidentiary standard + recording shape* — introduced the item (a) `provably-differs` "evidence that addresses session-only instance survival specifically" gate as the latest of the four.

**History:** The four non-mechanical bars were authored under four different review threads spread across six weeks, each focused on resolving its own thread (closed-set audit; per-session serialisation citation; the file split itself; item (a) authoritative-determination). Each commit tightened one clause's wording in isolation, and none surfaced the cumulative pattern that the bump procedure's `pass/fail/N/A` recording shape — central to `V18c`'s merge-gate posture — was acquiring four independent natural-language judgement calls at its margins. The defect is the interaction: no single commit *introduced* "non-mechanical conformance criteria" as a class, and the same fix pattern (replace the judgement bar with a closed predicate) applies in four places that share no other coupling.

## Solution Space

**Shape:** multiple

Each clause is independently editable and the fixes do not interact. Tackle them in the order below so each landing leaves a stable baseline for the next; do not bundle them. The recommendation orders by ascending blast radius: the SHOULD smoke check is local to step 2(a); branch (3) and the N/A definition each touch one paragraph of step 2 / step 2b; item (a)'s evidentiary standard sits inside the load-bearing one-shot determination and is best landed last so its replacement predicate can lean on the now-tightened surrounding text.

### Option A — Tighten the category-(3) smoke-check "type-side analogue"

**Approach.** Either drop the "or its type-side analogue" alternative entirely and pin the runtime `"<member>" in <importedExtensionContextLikePrototype>`-shaped check, or keep the type-side option but state the predicate it must satisfy: the type-side construct MUST be one whose `tsc` evaluation produces a compile error when the named category-(3) member is absent from the imported `ExtensionContext` / `ExtensionCommandContext` declaration (e.g. a `satisfies` clause whose right operand asserts member presence, or an indexed access `T["<member>"]` whose lookup must resolve).

**Spec edits.** Replace the "or its type-side analogue" clause in the step-2(a) category-(3) smoke-check paragraph with the compile-error-on-missing-member predicate (or remove it). No cross-file edits required.

**Pros.** Smallest scope; only the contributor implementing the SHOULD-arm needs the predicate; does not perturb any recorded-outcome shape.

**Cons.** Leaves the SHOULD-not-MUST framing in place, so the smoke check remains optional; an implementation that omits it entirely is still conformant.

**Risks.** Negligible — this paragraph has no downstream readers in the plan beyond `V18c`'s step-2(a) test.

### Option B — Replace branch (3) "spec section that authorises" with a closed predicate

**Approach.** Define what "authorises" means in the `// allow-pi-surface:` grammar: the cited section MUST contain a normative obligation (a MUST, MUST NOT, SHOULD, or SHOULD NOT, per the spec's existing normative-keyword conventions) that names the exact symbol being exempted, or — if the spec deliberately authorises ranged exemptions — a normative obligation that names the exact surface pattern (e.g. "every `ctx.<member>` access in file `X`"). A reviewer rejects the marker if neither the symbol nor a containing pattern appears in a normative sentence at the cited site.

**Spec edits.** Tighten branch (3)'s wording and the (1)∧(3) / (2)∧(3) tie-breaks in `version-bump-step2b.md` so the "authorising section" criterion is the named-symbol-in-a-normative-sentence test above. Update the *Exemption mechanism* in `inventory-audit-intro.md` to match.

**Pros.** Resolves the tie-breaks mechanically; eliminates the bulk of the reviewer-disagreement failure mode the surrounding step already acknowledges.

**Cons.** A few existing exemption sites may need their citations re-pointed if the originally-cited section was advisory rather than normative; the contributor performing the next bump absorbs that.

**Risks.** A future legitimate exemption whose justification is genuinely informative (e.g. a single-callsite test bridge) needs the spec to spawn a normative carve-out for it. Acceptable trade.

### Option C — Replace item (e) N/A's "other documented Pi-public surface" catch-all with a closed mechanism predicate

**Approach.** Drop the open-ended trailing clause and pin the N/A predicate to a behavioural property the Pi-side mechanism must establish: the cited Pi mechanism MUST, by its documented contract, serialise concurrent slash-handler dispatches against the same `AgentSession` such that two `_tryExecuteExtensionCommand` handler bodies cannot overlap their snapshot/restore windows; and the contributor's rationale MUST cite the exact line of Pi-side documentation that states this serialisation property. A Pi surface that exists but does not document the property is not eligible for N/A.

**Spec edits.** Rewrite the *Item (e) `N/A` definition* paragraph under "Outcome recording conventions"; replace the "(named per-session lock, single-consumer per-session dispatch channel, or other documented Pi-public surface)" parenthetical with the behavioural-property predicate above.

**Pros.** Restores the mechanical bar: a candidate-minor Pi-doc line either documents the non-overlap property or it does not. Aligns the N/A arm with the conservative-posture rule already pinned for the fail arm.

**Cons.** The pass-evidence rule already requires a cited code site for `pass`; this option adds a symmetric cited-doc-site requirement for `N/A`, which a contributor may experience as friction when the supersession is obvious. The friction is the point — silent obvious-supersession claims are the failure mode.

**Risks.** Low. Pi minors that actually expose such a mechanism will document it (or the loom can refuse the N/A and add the recovery serialisation defensively).

### Option D — Tighten item (a) `provably-differs` "addresses session-only survival specifically"

**Approach.** Replace the natural-language "addresses … specifically" gate with a closed evidentiary predicate: a `provably-differs` rationale is conformant only if its cited site demonstrates (or, in the loom 1.0 case, observes from the package-loaded `extensions/index.ts` load path) that the same `ExtensionAPI` reference (i.e. the same factory closure) is invoked across a `session_shutdown` whose `event.reason ∈ {"new", "resume", "fork"}` without the extension factory being re-invoked. State that an evidence site that only demonstrates instance survival on `{"quit", "reload"}` does not satisfy the predicate, and state that a site that demonstrates extension re-invocation on the session-only reasons falsifies the `provably-differs` verdict (i.e. forces `governed-by-rebind`).

**Spec edits.** Rewrite the *Acceptance criterion* sentence under *Item (a) authoritative-determination recording shape* with the closed predicate above. No other paragraph needs to move.

**Pros.** Removes the load-bearing ambiguity from the one-shot determination that gates `V9l`'s implementation. Makes the bump-commit review of a `provably-differs` record a check that one specific behavioural property is or is not observed at one specific site.

**Cons.** None material; the new wording is longer but the existing wording is already long.

**Risks.** Low. If the loom 1.0 determination ends up `provably-differs`, the tighter predicate makes the determination harder to defend on weak evidence — which is the intended effect.

### Recommendation

Land all four. Order: **A → B → C → D**. The smoke-check edit (A) is purely additive to a SHOULD arm and has no downstream readers; branch (3) (B) and the item (e) N/A definition (C) each touch one paragraph and do not interact; item (a) (D) is the most consequential and lands cleanest on a baseline where the surrounding step is already tightened. Each lands as its own commit. Watch for: (i) any existing `// allow-pi-surface:` markers in `src/` (none yet, since `V18b` has not landed) that the B edit would invalidate; (ii) the loom 1.0 implementation commit that resolves item (a) — its `provably-differs` rationale (if that is the verdict reached) MUST be authored against the tightened predicate from D, not the original wording.

## Consequence

**Severity:** correctness

A contributor performing a Pi minor bump and a reviewer evaluating the bump commit can reach opposing conformance verdicts on the same record without either party having misread the spec. Item (a) gates the SM-4/SM-5/SM-6 implementation path via `V9l`; item (e)'s N/A arm gates whether the defensive recovery serialisation needs to be added; branch (3)'s exempt vs delete/promote choice gates what lands in `SDK_SURFACE_INVENTORY`; and the smoke-check's type-side analogue gates whether a dropped category-(3) member surfaces at `npm test` or only at runtime. Each of the four divergence points can produce a merged bump whose conformance the next pass disputes.
