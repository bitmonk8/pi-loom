## Finding

The *Runtime construction obligation* paragraph on
[`diagnostics/diagnostic-shape.md`](../../docs/spec_topics/diagnostics/diagnostic-shape.md)
pins the runtime's implementation idiom rather than the wire contract: it
bans `...event` spread, `Object.assign(..., event, ...)`, "per-key copy
loop over `Object.keys(event)`", and "any other dynamic-field-set
mechanism whose member set is sourced from the SDK-delivered
`SessionShutdownEvent`", and mandates that each field be "assigned by
an explicit static property name". The observable property the spec
needs is closure of the wire set under all execution paths
(`details.event` carries exactly the pinned fields for each of the two
nested-shape codes, with no SDK leakage on future SDK additions). That
property is a wire invariant; the spread / `Object.assign` / key-loop
list is a JavaScript-specific implementation prohibition that admits
many conforming constructions a future contributor or a non-JS reader
would have no way to evaluate against the stated rule.

The same page's *Out of scope: `loom/typecheck/*`* paragraph
([`#loom-typecheck-namespace`](../../docs/spec_topics/diagnostics/diagnostic-shape.md))
embeds two further obligations on a wire-format page where neither
belongs. The *Naming convention* and *Definition site* sub-clauses are
**spec-authoring MUSTs** governing how spec authors introduce future
build-time brand strings (kebab-case rule-name component, no
`<...>`-style placeholders, MUST be defined inline at the introducing
spec-prose site, no central registry); their normative home is the
governance corpus that already owns naming, anchor, and definition-site
conventions ([`docs/spec_topics/governance/`](../../docs/spec_topics/governance/)).
The *Audit-gate carve-out* sub-clause is a **tooling MUST** that
constrains the [`V18b`](../../docs/plan_topics/V18b-inventory-audit.md)
inventory-closure audit walker (the recogniser MUST treat
`loom/typecheck/` as a prefix-disjoint namespace) and is mirrored by
[`H5a`](../../docs/plan_topics/H5a-closing-gate-automation.md)'s
REQ-ID / diagnostic-code closing gate; both are runtime-tooling
obligations whose normative home is the implementation-tooling page
([`docs/spec_topics/implementation-notes.md`](../../docs/spec_topics/implementation-notes.md))
that already carries audit / lint / build-time-gate rules of this kind,
not the wire-format page.

Compounding this, the `diagnostics.md` index page that gates the
shape / placeholder / registry sub-pages carries no `## Non-goals`
section. The wire-format page therefore acts as a de-facto sink for any
diagnostic-adjacent obligation that has no obvious other home — the
`loom/typecheck/*` paragraph is the visible symptom — because there is
no enumerated "things this surface deliberately does not own" list to
discipline new authoring against.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `75a9626` (2026-05-09) — pi-loom spec: resolve "SessionShutdownEvent.reason closed set has no build-time pin against the SDK type" — added the *Out of scope: `loom/typecheck/*`* paragraph (with the *Naming convention*, *Definition site*, and *Audit-gate carve-out* sub-clauses) to the wire-format page.
- `52b9101` (2026-05-11) — pi-loom spec: resolve "T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state" — added the *Runtime construction obligation* paragraph with the `...event` / `Object.assign` / key-loop prohibition.
- `f5e89f4` (2026-06-04) — docs: enforce 100KB hard / 30KB recommended size cap on spec set — split the monolithic `diagnostics.md` into the index + seven children under `diagnostics/`, relocating both paragraphs above onto the new `diagnostics/diagnostic-shape.md` child and reducing the parent `diagnostics.md` to a thin index page (the page that now lacks a `## Non-goals` section).

**History:** `git log -S` against the defect tokens (`"Runtime construction obligation"`, `"loom-typecheck-namespace"`, `"...event"` spread prose) on `diagnostics.md` and `diagnostics/diagnostic-shape.md` shows two distinct prior fixes — each individually defensible — that landed adjacent obligations on the same paragraph (75a9626 added the namespace carve-out to anchor the build-time brand alongside the runtime-diagnostic namespace; 52b9101 added the construction obligation to defend the wire shape against SDK leakage). The June split (f5e89f4) carried both forward verbatim. The defect is the *accumulation* on a wire-format page, not either edit in isolation: each prior author chose this page because the immediately-adjacent paragraph already discussed the relevant code, and neither had a *Non-goals* section on the index page to push them elsewhere. The four-obligation finding is the natural way to unwind that accumulation.

## Solution Space

**Shape:** multiple

The finding bundles four independent obligations against the same
paragraph cluster. Each is small, scope-bounded, and trivially
loopable on its own; bundling them produces a much larger diff than
any half alone, which gives the next-pass lenses a much larger
critique surface. Resolve them in the order below so each obligation's
edit lands on a stable baseline.

### Option A — Soften the construction obligation to a wire invariant

**Approach.** Replace the *Runtime construction obligation* paragraph's
JS-idiom prohibition with a wire invariant stated in observable terms:
on every emission of `loom/host/session-shutdown-runtime-degraded` and
`loom/runtime/cancelled-by-session-shutdown`, the structured wire
payload's `details.event` member set equals the pinned per-code field
set exactly — `{ reason }` for the former and
`{ reason, loom, invocation_id }` for the latter — under every input
shape Pi can deliver under the loom 1.0 Pi-SDK pin range, including
future SDK additions to `SessionShutdownEvent` whose member set the
spec has not pre-authorised.

**Spec edits.** Rewrite the *Runtime construction obligation* paragraph
to state the wire-set-equality invariant and the rationale (a future
SDK field MUST NOT silently appear on the wire under any refactor).
Delete the enumerated `...event` / `Object.assign` / key-loop / "any
other dynamic-field-set mechanism" prohibition list. Retain the
per-code field-set enumeration (it is normative). Retain the
cross-references to the *Construction-site wrap* sub-clause of
[`diagnostic-emission-isolation.md`](../../docs/spec_topics/pi-integration-contract/diagnostic-emission-isolation.md)
and to PIC's *Unknown-reason rule* — those pins are unaffected.

**Pros.** States the property the spec needs in observable terms; lets
the cross-cutting wire-equality property be discharged by tests rather
than by per-construction inspection; removes JS-specific surface area;
makes a future move to a non-JS host re-readable without re-authoring.

**Cons.** Loses the implementer-facing concrete-mechanism hint. The
spec is JS-targeted in loom 1.0 and the prohibition was carrying that
hint cheaply.

**Risks.** A reviewer used to the explicit prohibition may not
recognise that the wire invariant covers the same defect class. Add a
non-normative implementer note ("a `...event` spread or
`Object.keys(event)` loop will breach this invariant when Pi adds a
field") to retain the hint without restoring the MUST.

### Option B — Relocate the `loom/typecheck/*` spec-authoring MUSTs to governance

**Approach.** Move the *Naming convention* and *Definition site*
sub-clauses out of `diagnostic-shape.md` and into the governance
corpus, which already owns naming conventions and definition-site
rules. Replace them on `diagnostic-shape.md` with a one-sentence
cross-reference back to the governance home.

**Spec edits.** Add a new sub-paragraph (or a new short page) under
[`docs/spec_topics/governance/`](../../docs/spec_topics/governance/) —
likely co-located with
[`stable-inline-labels.md`](../../docs/spec_topics/governance/stable-inline-labels.md)
or as a sibling — owning the `loom/typecheck/*` naming-rule and
definition-site contract. Land the new
[`req-id-prefix-table-active-b.md`](../../docs/spec_topics/governance/req-id-prefix-table-active-b.md)
or `governance.md` aggregator update if a new GOV-* anchor is needed
(consistent with the GOV-29 / GOV-30 aggregator-lock-step convention).
Strip the *Naming convention* and *Definition site* sentences from
`diagnostic-shape.md`'s *Out of scope* paragraph and replace them with
"naming convention and definition-site rules for this namespace are
owned by [governance home]".

**Pros.** Puts the spec-authoring MUST on the page where future spec
authors will go looking for naming rules. Removes a runtime-vs-author
mode-mix from the wire-format page.

**Cons.** Splits a paragraph that today reads cohesively for a reader
investigating the build-time brand. Requires a new governance anchor.

**Risks.** A reviewer reading `diagnostic-shape.md` may miss the new
home unless the cross-reference is prominent. Keep the *Out of scope*
top-line ("the namespace is parallel to runtime codes, no registry
row, no delivery channel, out of scope for the Code registry rules
below") on `diagnostic-shape.md` — the *runtime-diagnostic* claim
belongs there.

### Option C — Relocate the audit-gate carve-out MUST to implementation-notes

**Approach.** Move the *Audit-gate carve-out* sub-clause out of
`diagnostic-shape.md` and onto
[`docs/spec_topics/implementation-notes.md`](../../docs/spec_topics/implementation-notes.md),
which already owns runtime-tooling, lint, and build-time-gate
obligations. Strip the sub-clause from `diagnostic-shape.md` and add
the cross-reference.

**Spec edits.** Add a sub-paragraph to
[`implementation-notes.md`](../../docs/spec_topics/implementation-notes.md)
pinning the prefix-disjoint carve-out under whichever existing section
covers the [`V18b`](../../docs/plan_topics/V18b-inventory-audit.md) /
[`H5a`](../../docs/plan_topics/H5a-closing-gate-automation.md) audit
walker (these leaves already cite `implementation-notes.md` and would
gain a single-link reciprocal). On `diagnostic-shape.md`, delete the
*Audit-gate carve-out* sentence and add "audit-gate carve-out is owned
by [implementation-notes link]".

**Pros.** Co-locates the audit-gate MUST with the other audit-walker
rules. Removes a tooling-vs-wire mode-mix from `diagnostic-shape.md`.

**Cons.** Same paragraph-splitting cost as Option B.

**Risks.** If Option B does not land first, the *Out of scope*
paragraph will end up with two trailing "owned by …" cross-references
back-to-back. Land B and C in either order, but land both or neither;
landing one in isolation leaves the paragraph asymmetric.

### Option D — Add a `## Non-goals` section to `diagnostics.md`

**Approach.** Add a top-level `## Non-goals` section to the
`diagnostics.md` index page enumerating what the diagnostics surface
deliberately does *not* own (build-time brand-string surfacing;
spec-authoring conventions for namespaces; audit-gate / closing-gate
walker rules; anything else that has historically accreted under
*Out of scope* sub-clauses on the wire-format page). Each bullet
cross-references the owning home (e.g. governance for spec-authoring,
implementation-notes for tooling, version-bump-procedure for the
build-time brand) so future authors land their MUSTs on the right
page.

**Spec edits.** Add `## Non-goals` to
[`docs/spec_topics/diagnostics.md`](../../docs/spec_topics/diagnostics.md)
between *Contents* and the end of the file. Authors of new
diagnostic-adjacent surface MUSTs cite this section for "is this in
scope?" decisions.

**Pros.** Closes the sink: future authors get a disciplined list of
what does *not* belong on the diagnostics surface. The discipline is
self-reinforcing once present.

**Cons.** A new section adds bytes to the index page (well under the
30KB recommended cap, but still real). Requires future edit discipline
to keep the bullet list aligned with the actual carve-outs on child
pages.

### Recommendation

Address in the order **D → B → C → A**. Land D first (the *Non-goals*
section), because it bounds the scope of every subsequent edit and
makes the receiving-site choices in B and C reviewable against an
explicit "what belongs on diagnostics" list. Land B and C next, in
either order — both relocate paragraphs without changing any wire
contract, so they are byte-for-byte traceable and the rebases are
mechanical. Land A last; it has the largest single-paragraph diff and
benefits from landing on a baseline where the *Out of scope* paragraph
has already been simplified, so the construction-obligation rewrite is
not entangled with adjacent relocations. Implementers should watch for
two interaction edges: (i) Option A's rewrite must preserve the
cross-reference to the *Construction-site wrap* sub-clause of
[`diagnostic-emission-isolation.md`](../../docs/spec_topics/pi-integration-contract/diagnostic-emission-isolation.md)
that pins the catch-arm wire forms — that pin is doing work and the
wire-invariant restatement must remain compatible with it; (ii)
Options B and C each strip one sub-clause from the *Out of scope*
paragraph — verify after both land that the residual paragraph still
reads as a coherent "out of scope for this page" notice rather than as
two trailing cross-references with no governing prose.

The orchestrator should resolve these as four sequential per-finding
fix passes rather than one bundled edit, so each pass's diff stays
small enough that the next pass's lenses converge.

## Consequence

**Severity:** advisory

The wire behaviour the spec needs is already deducible to careful
readers, so two reasonable JS implementers will land on equivalent
contracts. The cost is reviewer drag (the wire-format page conflates
three normative surfaces — wire contract, spec-authoring convention,
tooling rule), language-specific brittleness in the construction
prohibition (the list does not cover, e.g., proxy traps or
property-descriptor-driven shapes), and a wire-format page that quietly
grows as a sink because `diagnostics.md` declares no scope boundary.
None of this gates implementation.
