## Problem

The inline comment on the `kind` field of the normative `RuntimeEvent` type in `runtime-event-channel.md` lists the `loom/runtime/*` code for panics as a legal `kind` value. This contradicts the group-A/group-B routing partition established in the same file: `RuntimeEvent` is the group-A payload (`details: { event: RuntimeEvent }`), while every `loom/runtime/*` panic is group B (`details: { diagnostics: Diagnostic[] }`) and never constructs a `RuntimeEvent`. An implementer reading the comment in isolation could populate a `RuntimeEvent` with a panic code and emit it through the group-A shape the partition forbids.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `b3bc4ce` (`spec: pin failure-observability surface (Cluster A, Option C)`, 2026-05-04)
**History:** The corpus is git-tracked. `git log -S` for the comment phrase `the loom/runtime/* code for panics` resolves to exactly two commits: `b3bc4ce`, which first introduced the `RuntimeEvent` type definition (then in the monolithic `docs/spec.md`), and `f5e89f4` (2026-06-04), which only relocated the block into `runtime-event-channel.md` during the 100KB spec-set size-cap split. `git show b3bc4ce` shows the same commit that added the `kind` comment also established the contradicting routing â€” panics flow through `details: { diagnostics: [...] }` "rather than `details: { event: ... }`" â€” so the comment was internally inconsistent with its own page from the moment it was written. No later commit altered the comment text; the defect has been present since the construct's inception.

## Solution approach

Rewrite the `kind` field comment in the `RuntimeEvent` type block so it no longer lists panic codes as a legal `kind` value â€” the value enumeration should name the group-A `QueryError.kind` and the binder failure cause, but not the `loom/runtime/*` panic codes. The defect is the comment wording itself, so the corrected enumeration is the deliverable.
