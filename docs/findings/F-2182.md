## Finding

INV-4 (*Invocation depth bound*) defines a `.warp` `fn` call as a *cross-file*
countable frame when caller and callee reside in different source files, and
pins residence to the declaration site: "A `fn`'s residence is its declaration
site — the `.warp` file in which the `fn` is declared — so re-exports
(`export { foo } from "./other.warp"`) and `as`-aliased imports per
[Imports](./imports.md) do not change a `fn`'s residence for this cross-file
test." (`docs/spec_topics/invocation.md`, INV-4 countable-frame paragraph.)

The rule exists only in prose. No normative test vector demonstrates that
calling a re-exported or `as`-aliased `fn` from a different `.warp` file counts
as a cross-file frame (depth+1) identical to calling it directly from its
declaring file, nor that calling a `fn` declared in the caller's own file is
not a countable frame. An implementation that determines cross-file status from
the *import site* (the file the binding is named in) rather than the
*declaration site* would miscount frames for re-exported and aliased `fn`s, yet
would pass every test vector the spec and plan currently pin — the existing
mixed-frame vector in `V15b-T` exercises only a plain cross-file `fn` (caller
and callee in different files with no re-export/alias indirection), which both
the correct declaration-site rule and the incorrect import-site rule satisfy
identically.

## Issue introduction
**Verdict:** introduced-by-commit
**Introducing commits:** `1b540e9` ("pi-loom spec: resolve \"T13a — define cross-file qualifier in invocation.md countable-frame paragraph\"", 2026-05-20)
**History:** `git log -S "residence is its declaration site" -- docs/spec_topics/invocation.md` returns a single commit, `1b540e9`. Its diff shows the countable-frame paragraph rewritten from a bare "any cross-file `.warp` `fn` call" to the full cross-file qualifier plus the residence-is-declaration-site sentence with the `export { foo } from` and `as`-alias examples. The residence rule and the testability gap entered together in that commit: the prose rule was added without an accompanying normative test vector, and no later commit (latest touching the file is `9d3f258`) supplied one. The defect is therefore not present-since-inception of INV-4 but coincident with the introduction of the residence prose itself.

## Solution Space

**Shape:** single

### Recommendation

Add a normative test vector to INV-4 (and mirror it into `V15b-T`'s `Tests`)
covering residence resolution through re-export and alias:

- Calling `fn foo` from `.warp` file B, where `foo` is *declared* in `.warp`
  file A and re-exported from B (`export { foo } from "./a.warp"`), counts as
  one cross-file countable frame (depth+1) — identical to calling `foo`
  directly from A. The same holds for an `as`-aliased import of `foo`.
- Calling `fn bar` declared in the same `.warp` file as the caller is not a
  countable frame (depth unchanged), regardless of whether `bar` is also
  re-exported elsewhere.

State the vector against the declaration site so that an import-site
implementation fails it. Keep the existing mixed-three-class boundary vector;
this addition is orthogonal to it.

## Consequence

**Severity:** advisory

Two implementers reading the prose can produce a correct system, but the
declaration-vs-import-site distinction is exactly the kind of detail an
implementation can get wrong while staying green against the current test set,
making the residence rule effectively unenforced by the test suite. The defect
is recoverable from prose, so it degrades test coverage rather than blocking
implementation.
