## Finding

The closing paragraph of `version-bump-step2.md`'s editorial-review checklist obliges the contributor extending the spec corpus:

> Any future presupposition added to this page or to `future-considerations.md` that routes its detection to editorial review under this procedure MUST be added to this checklist in the same edit; landing such a presupposition without extending the checklist leaves the detection mechanism reliant on contributor recall and is non-conformant with this step.

The MUST fires only on presuppositions whose detection is "routed to editorial review under this procedure," but neither this page nor any of the source pages it ranges over (`host-prerequisites.md`, `conversation-drive.md`, `runtime-event-channel.md`, `provider-error-mapping.md`, `tool-registration-lifetime.md`, `host-interfaces-core.md`, `host-interfaces-services.md`, `registration-steps.md`, `subagent.md`, `capability-inventory-items.md`, `future-considerations/model-changes-and-non-goals.md`) carries any formal or textual marker indicating that a given presupposition note opts into editorial-review routing. The existing presupposition paragraphs items (a)–(ak) cite use a free mix of headings — *consumption posture*, *behavioural presupposition*, *open question*, *Host prerequisite*, *premise* — none of which is a load-bearing routing signal; the routing claim is asserted only on the consuming side here at the checklist, never on the source side. A reviewer adding a new presupposition note therefore cannot mechanically decide whether the MUST applies: every presupposition whose detection is not already mechanically gated (i.e., the modal universe the gate exists to guard) is by construction a candidate, and the gate's binding bite depends entirely on whether the author retrospectively classifies their own paragraph as "editorial-review-routed."

The practical effect is that the MUST collapses to the very contributor-recall failure mode its own conformance clause declares non-conformant: a new presupposition can land on any of the source pages, be left off the checklist, and the omission is detectable only by a reviewer's intuition about whether the new paragraph belonged in the (a)–(ak) sequence.

## Solution Space

**Shape:** multiple

### Option A — Pin a required marker phrase on each editorial-review-routed presupposition

**Approach.** Coin a fixed marker the source-side presupposition note MUST carry to opt into the editorial-review-routing predicate — for example, a trailing sentence of the form "*Detection routed to editorial review per `version-bump-step2.md` item (`<letter>`).*" (with the `<letter>` slot filled once the checklist item lands), or an `<a id="bump-checklist-routed-…">` anchor pattern, applied consistently across every existing (a)–(ak) source paragraph. Re-state the closing sentence of `version-bump-step2.md` to bind the MUST on that marker: "Any future presupposition that carries the *editorial-review-routed* marker MUST be added to this checklist in the same edit; conversely, adding a presupposition without that marker is the contributor's positive declaration that its detection is covered elsewhere (build-time gate, typed contract, existing checklist item) and that declaration is itself reviewable."

**Spec edits.** Add the marker convention to `version-bump-step2.md`'s preamble (one paragraph). Backfill the marker on the existing presupposition paragraphs items (a)–(ak) currently cite (the 32 source-side anchors listed in §Spec Documents). Reword the closing gate sentence to key on the marker.

**Pros.** Makes the predicate textually decidable by `grep`. Inverts the failure mode — a new presupposition is either marked (and the gate fires) or unmarked (and the author has positively declared the detection is covered elsewhere, which is itself a reviewable claim). The marker doubles as a navigation anchor when amending the checklist on a later bump.

**Cons.** Requires a one-time backfill across ~32 anchors; the marker phrasing has to land in the same edit that updates the gate sentence, so the diff is wide. The marker is contributor-discipline at the point of authoring a new presupposition (the author still has to remember to apply it), but the failure mode of omitting it is now a reviewable artifact rather than an undecidable predicate.

**Risks.** The marker convention itself must be unambiguous; a fuzzy marker (e.g., "*Behavioural presupposition.*") would just relocate the same problem. The marker must be distinct from the existing "*consumption posture*" / "*behavioural presupposition*" framings already in use, so reviewers do not assume legacy headings auto-satisfy it.

### Option B — Invert the gate: every new presupposition is in-scope unless an explicit carve-out cites the covering mechanism

**Approach.** Drop the "routes its detection to editorial review under this procedure" qualifier entirely. Re-state the closing sentence so it ranges over *every* new presupposition added to the listed source pages, and require any author who believes the new presupposition is out of scope to cite the covering mechanism (a specific build-time gate, a typed contract pinned in PIC, or the existing checklist item that subsumes it) in the same edit. The default scope is "in," and the contributor's positive citation is what removes the obligation.

**Spec edits.** Reword only the closing sentence of `version-bump-step2.md`; no source-side backfill required.

**Pros.** Smallest diff. Eliminates the undecidable predicate by removing it. Self-correcting: a presupposition the author *forgets* to add to the checklist is, by default, non-conformant, so the failure mode the gate exists to catch is caught.

**Cons.** Slightly over-inclusive — a presupposition that is genuinely covered by an existing build-time gate now requires an explicit carve-out citation to escape the MUST, which is mild authorial overhead. Does not provide the navigation-anchor benefit Option A's marker gives.

**Risks.** The carve-out citation itself becomes the new reviewability surface; a hand-wavy "covered elsewhere" carve-out would re-introduce the same softness, so the spec edit must require a specific citation (gate name or PIC anchor).

### Recommendation

Take **Option A**. The marker-phrase convention pays for its one-time backfill cost by giving both the gate and the bump-time amending contributor a textually-grounded anchor — the marker on the source page and the matching `<a id="bump-checklist-…">` on the checklist item form a discoverable pair that survives spec rewordings. Option B is cheaper but trades the over-inclusive default against the carve-out's softness, which leaves a smaller but qualitatively similar undecidability surface. The edit should land in two parts in the same commit: (1) the gate sentence rewording naming the marker phrase verbatim; (2) the backfill of the marker on each of the existing (a)–(ak) source paragraphs, so the convention is closed at the moment it is introduced rather than carrying a transitional "this convention applies to new presuppositions only" caveat.

Edge cases the implementer must watch:

- The marker must be applied to the *presupposition paragraph on the source page*, not merely to its `<a id>` anchor; an anchor without the marker text is `grep`-invisible from the source side.
- A presupposition that is co-cited by multiple checklist items (e.g., `host-prerequisites.md#degraded-state-host-prerequisites` is referenced by items (a), (b), (c), and (x)) takes one marker that enumerates all matching letters, not a per-item duplication.
- Item (e)'s sub-items `(e.i)` / `(e.ii)` should be addressable by the marker convention without forcing future sub-items to coin a new shape; the marker grammar should allow `(e.i)` / `(e.ii)` style suffixes.

## Consequence

**Severity:** advisory

The gate exists to convert "did the contributor remember to extend the checklist?" from a recall problem into a reviewable conformance MUST, but because its triggering predicate is itself only recoverable by contributor intuition, the gate provides essentially no extra enforcement over the pre-existing review discipline. A new presupposition can land on any of the listed pages, be left off the checklist, and the omission is non-detectable by any mechanical or textually-grounded review pass — the failure mode the closing sentence's "non-conformant with this step" wording was meant to make catchable.
