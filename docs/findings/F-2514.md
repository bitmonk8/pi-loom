## Problem

The `?`-operator section of `expressions.md` states that a function or loom using `?` "implicitly returns `Result<T, QueryError>` where `T` is the type of its last expression." This is narrower than, and contradicts, the authoritative return-type rule in `functions.md` FN-3 (`#loom-return-type`), which defines the inferred success payload as the least upper bound (under `⊑`) of the tail-expression type and the success payload of every early-`return` operand in the body. For a body that early-returns differently-typed values, the `?`-section rule yields a too-narrow `T` and skips the `loom/parse/return-no-common-type` diagnostic that FN-3 mandates. The two normative sections must agree on a single rule.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 53f5831 (`spec: rewrite query primitive, system prompts, and schema syntax`, 2026-05-03) and 08886d9 (`pi-loom spec: resolve "loom return-type inference does not reconcile early-return operand types"`, 2026-06-07)
**History:** The phrase "`?` … implicitly returns `Result<T, QueryError>` where `T` is the type of its last expression" entered the corpus in 53f5831, in the then-monolithic `spec.md`. At that point return-type inference said only that "the same inference rule that applies to a tail expression applies to `return`'s argument" — no cross-`return` reconciliation — so the simplification was internally consistent. The contradiction emerged in 08886d9, which introduced the FN-3 least-upper-bound-of-tail-plus-every-`return`-operand rule but left the `?`-section phrase unchanged, making it stale. A later commit (87dfa1d, 2026-06-28) relocated the stale sentence verbatim from `error-model.md` into its current home in `expressions.md`, carrying the defect forward without correcting it. The defect is thus the product of the FN-3 refinement not being propagated to the `?`-section phrasing, not a single faulty edit.

## Solution approach

In the `§ Error propagation — the `?` operator` section of `expressions.md`, rewrite the "where `T` is the type of its last expression" clause so it defers to FN-3 for how `T` is computed, forward-linking to `functions.md#loom-return-type`. The `?` section should describe only the early-return/unwrap semantics and link out for the return-type rule.

## Solution constraints

- Do not re-derive the FN-3 least-upper-bound discipline inline in the `?` section; the single source of truth for return-type inference stays at `#loom-return-type`.
