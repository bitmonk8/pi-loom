## Finding

`plan.md` §How to use this plan, step 2 is the leaf-authoring entry point: it tells an author when a newly-added leaf must be added to [`H5b`](docs/plan_topics/H5b-warn-only-canary.md)'s `Deps.`. Its trigger summary reads "a citing test that *closes* a coverage-matrix-mapped numbered REQ-ID (not a mere re-citation of an already-closed REQ-ID, which does not qualify)". The parenthetical carries two ambiguities:

- **The relative "which" is danglingly attached.** "…a mere re-citation of an already-closed REQ-ID, which does not qualify" leaves the antecedent of "which" undetermined between the *re-citation act* (intended — the act does not qualify) and the *already-closed REQ-ID* (a misreading — the REQ-ID category does not qualify). The two readings classify different leaves as in/out of the H5b-Deps obligation.
- **"already-closed" has no fixed referent.** It reads as either (i) "the coverage matrix already lists a closing leaf for this REQ-ID" or (ii) "the closing leaf's `<id>-complete` git tag has already been pushed". Reading (ii) makes the H5b-Deps obligation depend on git-tag timing rather than on the static coverage-matrix mapping.

The authoritative clause in [`conventions.md`](docs/plan_topics/conventions.md) §REQ-ID discipline (*Transitive-completeness plan-maintenance*) resolves both: it states the trigger "turns on the coverage-matrix closing-leaf mapping, not on whether the citation is the REQ-ID's first textual occurrence, so a leaf that merely re-cites an already-closed REQ-ID does not qualify", and pins the closing leaf as "the leaf the coverage matrix lists as that REQ-ID's closing leaf". The plan.md summary — which an author reads first, and which explicitly forwards to that conventions.md clause as the governing statement — is less precise than the rule it summarises.

## Issue introduction

**Verdict:** introduced-by-later-edit
**Introducing commits:** `fa85571` (2026-06-24, "pi-loom spec-plan: fix F-1247 — strike over-broad H5b-Deps trigger disjunct in plan step 2")
**History:** The step-2 H5b-Deps trigger was added at `ab8e297` (2026-06-11) as "If the new leaf can introduce an executable REQ-ID, a citing test that *closes* a coverage-matrix-mapped numbered REQ-ID, or an un-anchored normative MUST". `ef9d7a7` (2026-06-20) re-cast the trigger verb to a present-state predicate ("`Tests.` bullets carry…"). The ambiguous parenthetical "(not a mere re-citation of an already-closed REQ-ID, which does not qualify)" entered at `fa85571` (2026-06-24), which struck the over-broad "an executable REQ-ID" disjunct and added the parenthetical as the narrowing clarification — but in a form looser than the conventions.md clause it summarises. The ambiguity is therefore not present-since-inception; it was introduced by that specific narrowing edit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan.md` §How to use this plan, step 2, replace the imprecise parenthetical "(not a mere re-citation of an already-closed REQ-ID, which does not qualify)" with wording that mirrors the conventions.md *Transitive-completeness plan-maintenance* clause: tie the trigger to the coverage-matrix closing-leaf mapping (the leaf the matrix lists as that REQ-ID's closing leaf), not to first textual occurrence or to git-tag state, and state that a leaf which merely re-cites a REQ-ID the matrix maps to a different closing leaf does not qualify.

A literal replacement for the parenthetical that carries this content:

`(the trigger turns on the coverage-matrix closing-leaf mapping — the leaf the matrix lists as that REQ-ID's closing leaf — not on first textual occurrence or completion-tag state; a leaf that merely re-cites a REQ-ID the matrix maps to a different closing leaf does not qualify)`

Keep the existing forward-pointer to `conventions.md` §REQ-ID discipline (*Transitive-completeness plan-maintenance*) as the governing statement; this edit only brings the summary into agreement with it. Do not touch `conventions.md` (it is already authoritative) or any leaf file.

## Consequence

**Severity:** advisory

A leaf author reading only `plan.md` step 2 may mis-classify whether a newly-added leaf must enter `H5b`'s `Deps.` — either by mis-attaching "which does not qualify" or by reading "already-closed" as the git-tag-pushed state (ii) rather than the static coverage-matrix closing-leaf mapping (i). The authoritative conventions.md clause and the `H5d` per-cell mechanical gate bound the harm (a working leaf is still producible), but the entry-point summary diverges from the binding rule it forwards to, and `H5d`'s per-cell at-least-one check cannot catch a secondary facet-closer omitted from `H5b.Deps` while a sibling closer for the same cell remains.
