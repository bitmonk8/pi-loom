## Finding

The `## Template Interpolation` section in `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md` (line 65) carries normative rules that govern every `@`...`` query template, not the frontmatter surface it sits under: it pins the meaning of `${...}` (a Loom expression from the expression sublanguage, evaluated up to the matching `}`); it pins the unique lexical role of `@` and its absence inside `${...}`; and it carries the spec's **only** explicit statement that no bash-style argument-slice sugar (`${@:N}`, `$1`, `$@`, `$ARGUMENTS`) exists in loom 1.0. None of that is frontmatter-specific. The `system:` field on the same page reuses the rules via a back-reference, which is the correct direction; the rules themselves should live where every other query-template normative — escapes, stringification table, dedent, degenerate-template short-circuit — already lives, in `docs/spec_topics/query/`.

The mis-placement materially harms discoverability of the "no bash sugar" rule. A reader checking whether `$1`/`$ARGUMENTS` are recognised inside a `@`...`` template will naturally consult `query/query-forms.md` or `query/query-escapes-stringification.md`; both files are silent on the question, and the answer is two directories away in a sub-page that *appears* to be about frontmatter parameters. The expression sublanguage page (`expressions.md`) likewise only mentions `${...}` interpolation in passing (line 3); it does not declare the lack of bash sugar. The grammar page mentions "Template interpolation `${...}` and `@`...`` query templates" in a list (line 51) without semantics. The result is a single load-bearing paragraph hidden under the wrong topic.

Cross-references already assume the destination home: `frontmatter-fields-b-and-templates.md` line 45 treats the `\${` escape as "the same escape used in `@`-template bodies" and links to the local section — which would link more honestly to the query area. `query/query-escapes-stringification.md` line 16 ("The same rule applies to the bare-path `${param}` / `${param.field}` form in the frontmatter `system:` field") already runs the rules in the canonical direction from query to frontmatter, leaving only this one section flowing the wrong way.

## Solution Space

**Shape:** single

### Recommendation

Move `## Template Interpolation` from `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md` to `docs/spec_topics/query/query-escapes-stringification.md`, placed immediately before the existing `## Escapes` section so the natural reading order is *what `${...}` is → how it stringifies → what escapes apply*. The relocated section keeps its two-paragraph body verbatim, including the "no bash-style argument-slice sugar" sentence, the cross-link to `binder.md` for slash-command argument binding, and the back-reference to `query.md`'s stringification table (which is now in the same file and becomes a same-file anchor link).

Leave a one-sentence stub in the original location reading roughly: *"Query-template `${...}` interpolation is specified in [Query escapes — Template interpolation](../query/query-escapes-stringification.md#template-interpolation). The `system:` field reuses that grammar restricted to the [Path](#parameters-and-frontmatter-system-interpolation) production above."* Retain the existing `\${` escape paragraph at line 45 but repoint its bracketed `(see [Template Interpolation](#template-interpolation))` link to the new query-area anchor.

Update collateral:

- Add an `<a id="template-interpolation">` anchor at the relocated section so existing in-page links (`frontmatter-fields-b-and-templates.md` line 45) continue to resolve after the cross-file repoint.
- `language-and-architecture.md` line 36: replace "template interpolation" in the frontmatter entry with the more honest "`system:` Path-only interpolation"; add a parallel mention under the query entry pointing at the new home.
- `plan_topics/V13a-query-render.md` and `V13a-T-query-render.md`: no Tests/Ships-when changes; the **Spec.** pointer already cites `query/query-escapes-stringification.md`, which is now the home.
- `plan_topics/V6d-system-interpolation.md` and `V6d-T`: no Tests/Ships-when changes; the existing cross-reference to `query/query-escapes-stringification.md` already points at the canonical stringification table and now also covers the `${...}` semantics implicitly.

Edge cases for the implementer:

- The relocated paragraph's reference to `binder.md`'s "Slash-Command Argument Binding" must remain a *typed-params* statement, not a *slash-command-only* statement. The "no bash sugar" rule applies to every `@`...`` template regardless of how the loom was invoked (slash, `invoke(...)`, callable). Word the relocated sentence accordingly.
- Choosing `query-escapes-stringification.md` over `query-forms.md` is deliberate: `query-forms.md` already owns the untyped/typed/explicit-form taxonomy and the dedent vectors and is near the 15 KB mark; adding the interpolation grammar there would push it toward the 30 KB recommended cap. `query-escapes-stringification.md` is 7 KB and topically tighter.
- The single existing internal link `[Template Interpolation](#template-interpolation)` on line 45 must be updated to `[Template Interpolation](../query/query-escapes-stringification.md#template-interpolation)`; a grep for `#template-interpolation` across `docs/` finds no other in-corpus consumers, but re-grep after the move as a safety check.

## Consequence

**Severity:** advisory

A discoverability degradation, not a correctness one. The rules are present in the corpus and an implementer who reads the whole spec will encounter them. But implementers who navigate by topic — checking the query area for query-template semantics, or grep-searching for `$ARGUMENTS` in the query/ directory — will miss the only normative statement that bash-style argument-slice sugar is not part of loom 1.0, and may either invent a stub for it or leave the lexer permissive in ways that produce silent mis-parses of `${...}` bodies containing `@`, `:`, or bare `$N` tokens.
