## Problem

`H7a` describes its committed permitted-code list as a **superset** of the committed golden diagnostics list, so that "the in-process gate and the real-host smoke cannot disagree on a code the fixture path actually emits." The `golden ⊆ permitted` relation backing that claim is established only by construction in prose: both lists are checked-in artifacts, but no test asserts containment between them, so a maintainer who adds a golden code without permitting it leaves `npm test` green while the property silently breaks. Separately, the union's completeness — that the permitted list captures every code each `Deps` slice can emit — is a manual per-slice provenance obligation with no emittable-code registry to verify against, so a stale union can spuriously fail `H4a`'s real-host smoke pass criterion (e) on a benign emission.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 052b019 — pi-loom plan: resolve "Real-host smoke pass criterion (e) names a permitted code set with no committed source" (2026-06-11, Thomas Andersen)
**History:** `H7a` was created earlier (e7e51cc), but the committed permitted-code list and its "superset" / "cannot disagree" wording, together with the manually-maintained per-Deps-slice provenance obligation and no mechanical containment check, were introduced as a unit in 052b019 when criterion (e) was given a committed source. Later edits (542395a widened the `Deps` union; eaa5000 touched the same line) preserved the overclaim without adding a gate.

## Solution approach

Add to `H7a`'s in-process Tests a verifiable containment assertion that reads the committed golden diagnostics list and the committed permitted-code list and asserts `golden ⊆ permitted`, reading both checked-in artifacts directly rather than re-deriving either from a pipeline run. Separately, rewrite the permitted-code-list Adds prose and Tests bullet in `H7a` so the "union … *can* emit" / per-Deps-slice-provenance language flags the union as a manually-maintained, best-effort set whose completeness is not mechanically verified — and align the corresponding restatements in `H4a`'s criterion (e) and `H6a`'s release-gate evidence record to that softened wording.

## Solution constraints

- The spec is read-only for this fix; edits are confined to the plan leaf files (`H7a`, `H4a`, `H6a`).
- The containment gate must compare the two checked-in artifacts, not live-run output.
