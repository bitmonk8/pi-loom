## Finding

Inside the *single-source-page* sub-clause of GOV-30 (`req-id-prefix-table-active-b.md`), the navigation-pointer-cross-link parenthetical first declares that a heading-rename which silently breaks an aggregator-side cross-link "is a documentation defect, not a normative defect", and then in the same parenthetical states "the PR that renames the heading SHOULD repoint each affected aggregator-side cross-link in the same commit, caught by reviewer / grep". The token `SHOULD` is uppercase, matches no surrounding code-span or quotation fence, and therefore reads as RFC-2119 — a reading the rest of the corpus actively reinforces: GOV-22 enumerates uppercase `SHOULD` as one of the six normative-modal tokens that mark a "defining obligation site", and the retired-prefix table at `governance/anchor-scheme-and-retired.md` records GOV-13's retirement under the heading "Modal weakening (RFC-2119 SHOULD → non-binding …)". The paragraph thus simultaneously claims the obligation is non-normative and writes it in the corpus's own normative-modal vocabulary.

A maintainer who breaks such a cross-link in a rename PR cannot resolve the contradiction from the text alone: under the "documentation defect" framing the lapse is reviewer-flagged housekeeping with no spec-defect status, whereas under the RFC-2119 reading it is a (lowercase-severity but still) normative `SHOULD` whose violation is a spec defect on the same terms as any other unconditional `SHOULD`. The two readings disagree on whether downstream auditing (GOV-18 arm (b), coverage-matrix authorship, review checklists) is required to track and report the lapse at all.

The same paragraph immediately afterwards uses uppercase `SHOULD` in a clearly normative slot ("reviewers SHOULD flag it on the PR that introduced the drift"), so a maintainer cannot infer from local context that the navigation-pointer `SHOULD` was meant casually — the rule already shows it can use uppercase `SHOULD` normatively when it intends to.

**Verdict:** present-since-introduction
**Introducing commits:** `0e12140` (`pi-loom spec: resolve "Non-goals are not consolidated into a single section"`, 2026-05-08)
**History:** The navigation-pointer-cross-link parenthetical was introduced wholesale in `0e12140`, which rewrote the predecessor rule (then `GOV-12`) on `docs/spec_topics/governance.md` to add the *single-source-page* shape. `git log -G "SHOULD repoint each affected aggregator"` and `git log -G "documentation defect, not a normative defect"` agree on `0e12140` as the first commit carrying either phrase; `git show 0e12140^:docs/spec_topics/governance.md` confirms neither phrase exists in the parent. The contradiction — uppercase RFC-2119 `SHOULD` adjacent to the explicit "not a normative defect" disclaimer — was therefore present in the very first version of this sub-clause. Commit `fe8aa5c` (2026-06-05) renumbered the host rule from GOV-12 to GOV-30 (and split out the integer-count invariant as GOV-31), and `f5e89f4` (2026-06-04) relocated the paragraph from `docs/spec_topics/governance.md` to `docs/spec_topics/governance/req-id-prefix-table-active-b.md` during the 100KB-cap split; neither commit altered the contradictory wording, both preserved it verbatim.

## Solution Space

**Shape:** single

### Recommendation

In the GOV-30 navigation-pointer-cross-link parenthetical, downcase `SHOULD` to `should` so the sentence reads as advisory editorial guidance consistent with the "documentation defect, not a normative defect" framing immediately preceding it. The minimal edit is to change "the PR that renames the heading SHOULD repoint each affected aggregator-side cross-link in the same commit" to "the PR that renames the heading should repoint each affected aggregator-side cross-link in the same commit".

Implementer edge cases:

- The other uppercase `SHOULD` in the same paragraph — "reviewers SHOULD flag it on the PR that introduced the drift" — is genuinely normative (it pins the review-time enforcement mechanism for the lock-step MUST) and must stay uppercase. Touch only the navigation-pointer occurrence.
- Do not introduce a co-located REQ-ID anchor for the downcased clause; lowercase `should` is outside GOV-22's defining-obligation-site predicate, so no GOV-22 trigger fires. (Uppercase would also be exempt today because the host paragraph already carries the `gov-30` anchor, but downcasing removes the latent ambiguity altogether.)
- Do not rephrase the surrounding clauses ("this is a documentation defect, not a normative defect, and the rename-PR author's responsibility", "caught by reviewer / grep") — they are doing real work in framing the obligation's force and audience, and the per-finding fix should be confined to the modal token.

## Consequence

**Severity:** advisory

Two reviewers reading the same rename PR will disagree on whether a missed aggregator-side repoint is a spec defect (uppercase-`SHOULD` reading) or a housekeeping miss outside the spec's normative scope (parenthetical-framing reading). No runtime behaviour is affected — the rule binds editors, not the loom runtime — but the same-paragraph contradiction undermines GOV-22's invariant that uppercase normative-modal tokens have a single, stable meaning across the corpus.
