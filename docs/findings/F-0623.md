## Problem

The second paragraph of `## Binder model` in `binder/binder-model-and-context.md` opens with a one-clause statement that `looms.binderModel` hot-reload re-resolves on the next loom load, then — with no sub-heading, callout, or topic break — defines a full operator-facing recovery-notification feature (the `loom watcher: …` system-note template, the `details: { recovery: { looms } }` payload, content/substitution/suppression/debounce rules). That body is a coherent operational feature, not a clarifying detail about binder-model resolution, yet the enclosing subsection's declared subject is load-time resolution and the strict-capability probe. A reader looking for "how do I know when settings changes unblock previously-failed looms?" cannot find it by heading or cross-reference target name — the only handle is the inline `<a id="binder-model-hot-reload">` anchor, which `runtime-event-channel.md`'s shape table back-links into mid-paragraph.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `0fa81fa` (2026-05-05) seeded the hot-reload sentence as a one-line tail to the binder-model paragraph; `319228c` (2026-05-05) appended the unstructured "single consolidated `loom-system-note`" clause; `f88698e` (2026-06-05) expanded that clause into the present operational-feature paragraph without re-shaping the enclosing `## Binder model` subsection to declare or contain it.
**History:** the original sentence at `0fa81fa` was small enough to live as a parenthetical inside `## Binder model`. The placement defect emerged not at any single commit but from the interaction of (a) the original embedding decision in `0fa81fa`, (b) the `319228c` append that grew the surface from "re-resolves on next load" to "emits a note," and (c) the `f88698e` rewrite that pinned a full operational-notification feature in the same paragraph. A later edit (`97430aa`, 2026-06-06) softened the over-promise about `/reload` but did not revisit the placement.

## Solution approach

Split the `## Binder model` second paragraph at the existing `<a id="binder-model-hot-reload">` anchor: keep the lead clause about re-resolution on the next loom load in `## Binder model`, and promote the recovery-note body into its own sub-section under a heading, carrying the `binder-model-hot-reload` anchor verbatim onto that new heading so the feature becomes discoverable in the file's heading structure. This is a single-file edit to `binder/binder-model-and-context.md`.
