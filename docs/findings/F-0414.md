## Finding

V18c's **Adds** field closes with a sentence describing the contributor's editorial-review checklist: "Separately — and *not* a build-time gate — the contributor's editorial-review checklist (items `(a)`–`(ak)`): a manual per-bump audit of each host presupposition against the candidate Pi minor, MUST-level for `(a)`–`(e)` and SHOULD-level-audit-but-MUST-record for `(f)`–`(ak)`." The **Adds** field is reserved for the code / infrastructure the leaf delivers — the SDK surface-inventory tests, the `engines.node` three-way equality gate, the `peerDependencies` pin assertion, the capability-probe constants, the provider seed-field table, and the strict-capability probe. The editorial-review checklist is none of these: it is a manual per-bump governance process, expressly flagged "*not* a build-time gate."

V18c's **Ships when** field already carries the editorial-review checklist correctly, as an acceptance condition confirmed by editorial review rather than `npm test`: it specifies that the bump commit message records a per-item outcome for every item `(a)`–`(ak)` (item `(e)` recording its two sub-outcomes) and that no MUST-level item `(a)`–`(e)` carries a `fail`. The checklist therefore appears twice — once as a descriptive aside in **Adds** and once as the operative gate in **Ships when** — with the **Adds** copy classifying a manual governance obligation among the leaf's implementation artefacts.

The defect is organisational, not mechanical: the **Ships when** field fully and correctly states the gate, so the implementer's deliverable and acceptance criteria are intact. The **Adds** sentence is redundant placement that mislocates a manual process among code deliverables.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V18c-version-bump-checklist.md`, **Adds** field, strike the editorial-review-checklist sentence ("Separately — and *not* a build-time gate — the contributor's editorial-review checklist (items `(a)`–`(ak)`): a manual per-bump audit of each host presupposition against the candidate Pi minor, MUST-level for `(a)`–`(e)` and SHOULD-level-audit-but-MUST-record for `(f)`–`(ak)`."), leaving **Adds** to enumerate only the build-time code/infrastructure deliverables.

The editorial-review checklist's acceptance is already stated in V18c's **Ships when** field (confirmed by editorial review, not `npm test`; per-item outcomes `(a)`–`(ak)` recorded in the bump commit message; no MUST-level `(a)`–`(e)` `fail`). If the MUST-level vs SHOULD-level distinction for `(a)`–`(e)` / `(f)`–`(ak)` is not already conveyed in **Ships when**, fold that distinction into the existing **Ships when** clause so no information is lost when the **Adds** sentence is removed. The checklist's normative definition lives in `version-bump-step2.md` (spec), which V18c already cites under **Spec**; no spec edit is required.

## Consequence

**Severity:** cosmetic

The editorial-review checklist is described in both **Adds** and **Ships when**, classifying a manual governance process among V18c's code/infrastructure deliverables. Because the sentence is explicitly flagged "*not* a build-time gate" and the **Ships when** field carries the operative acceptance criteria intact, no implementer is misled into building the checklist as code; the cost is duplicated prose and a field-classification inconsistency.
