## Finding

The `<failure>` carve-out paragraph in `placeholder-rendering-b.md` ends with the sentence:

> The canonical membership predicate operator tooling MUST use against a `details.failure` value `d` is `d === "missing-entry" || d === "literals-shape-invalid" || d.startsWith("throw:")` (or an equivalent expression that admits the same three discriminator forms…)

This is a normative obligation aimed at an externally-owned actor — operator triage tooling that consumes loom's diagnostics — and the runtime emits no signal that would let a conformance test distinguish a consumer that uses the predicate above from one that uses any other expression matching the same three discriminator forms. There is nothing observable on loom's own surface to falsify or witness this MUST, so it cannot anchor any conformance criterion.

The producer-side facts the runtime is actually accountable for — that `details.failure` carries exactly one of the two literals `"missing-entry"` / `"literals-shape-invalid"` or a `"throw:<…>"` template-family value, and that the wire format is pinned at the construction site — are already stated authoritatively in the same paragraph (and in the `loom/host/session-shutdown-pinned-constant-unreadable` row of `code-registry-host.md`, which redundantly carries the same wire-format pin). The membership-predicate sentence adds no producer-side obligation; it prescribes the consumer's source code.

The same anti-pattern recurs twice more in the surrounding cluster — `code-registry-host.md`'s row prose says "operator triage tooling MUST key on `startsWith("throw:")` against the discriminator", and `unknown-reason-rule.md` pins the same MUST inside its `"throw:<…>"`-grammar paragraph. A coherent fix demotes all three operator-directed MUSTs uniformly; demoting only the placeholder-rendering-b sentence leaves the identical normative claim standing upstream.

## Solution Space

**Shape:** single

### Recommendation

In all three sites, demote the operator-directed MUST to a non-normative note that documents the three-arm discriminator shape and gives the canonical predicate as an *example* parse, while keeping the producer-side wire-format obligation intact and normative.

Concretely:

- In `placeholder-rendering-b.md`, rewrite the membership-predicate sentence as informative guidance — e.g. "Operator tooling parsing `details.failure` typically uses `d === "missing-entry" || d === "literals-shape-invalid" || d.startsWith("throw:")` (or any expression equivalent over the same three discriminator forms); the wire format pinned above admits this parse." Keep the surrounding facts — that the discriminator set is not a `Set<string>`-membership-checkable enumeration, that tests on this row MAY use byte-identical equality for the two literals and `startsWith("throw:")` for the throw-template family, that a future §8 tightening MUST NOT change `<failure>`'s wire format — exactly as they stand. Those are producer-side and stay normative.
- In `code-registry-host.md`'s `session-shutdown-pinned-constant-unreadable` row, demote "operator triage tooling MUST key on `startsWith("throw:")` against the discriminator" to "operator triage tooling parsing this discriminator keys on `startsWith("throw:")` against the throw-template family" — informative rationale for why the `"throw:"` prefix is preserved on the `"throw:<unreadable>"` coercion-failure fallback. The producer-side pin (that the runtime emits the `"throw:"` prefix even on coercion failure, so the deduplication key `(code, details.failure)` does not split buckets) stays normative.
- In `unknown-reason-rule.md`, apply the same demotion to the parallel "operator triage tooling MUST key on `startsWith("throw:")`" sentence inside the discriminator-grammar paragraph.

Edge cases for the implementer: (1) the TypeScript closed template-literal union type `"missing-entry" | "literals-shape-invalid" | `throw:${string}`` in `unknown-reason-rule.md` is a static-typing aid for consumers, not an operator-side MUST — leave it as-is. (2) The build-time prohibition on strict-equality assertions over §8 placeholder rows is unaffected by this demotion; it targets test code that loom itself owns, not external operator tooling.

## Consequence

**Severity:** advisory

A reviewer trying to ship loom against the spec cannot write a conformance test that fails on a non-conformant operator-tooling implementation, because the runtime exposes no surface that distinguishes one predicate from another. The MUST therefore reads as authoritative guidance with no enforcement story, which is exactly the failure mode the testability lens flags. Two reasonable loom implementers would not diverge over this — the leak is editorial, not behavioural.
