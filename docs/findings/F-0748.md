## Problem

Three sites in the `session_shutdown` diagnostics cluster carry a normative MUST prescribing how external operator triage tooling must parse a `details.failure` value: `placeholder-rendering-b.md`'s §8 `<failure>` carve-out pins the canonical membership predicate operator tooling MUST use; `code-registry-host.md`'s `session-shutdown-pinned-constant-unreadable` row and `unknown-reason-rule.md`'s discriminator-grammar paragraph each pin "operator triage tooling MUST key on `startsWith("throw:")`". The runtime exposes no surface that distinguishes a consumer using that predicate from one using any other expression matching the same three discriminator forms, so no conformance test can witness or falsify these MUSTs. The producer-side facts the runtime is accountable for — the closed discriminator set and the construction-site wire-format pin — are already stated normatively in the same sites; the operator-predicate MUSTs add no producer obligation.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `75a9626` (2026-05-09, "pi-loom spec: resolve \"SessionShutdownEvent.reason closed set has no build-time pin against the SDK type\"")
**History:** The carve-out paragraph — including the sentence "The canonical membership predicate operator tooling MUST use against a `details.failure` value `d` is `d === \"missing-entry\" || d === \"literals-shape-invalid\" || d.startsWith(\"throw:\")`…" — was introduced in a single commit on the original `docs/spec_topics/diagnostics.md`. `git log --follow -S "canonical membership predicate" -- docs/` returns exactly two commits: `75a9626` (the introducing commit) and `f5e89f4` ("docs: enforce 100KB hard / 30KB recommended size cap on spec set", 2026-06-04), which mechanically split `diagnostics.md` into the `diagnostics/` subdirectory and moved the paragraph verbatim into `placeholder-rendering-b.md` without touching its wording. The sibling operator-directed MUSTs in `code-registry-host.md` and `unknown-reason-rule.md` were introduced in the same `75a9626` change as part of the same single-source-of-truth tightening — the commit added the operator-predicate prescription across the cluster simultaneously.

## Solution approach

Demote the operator-tooling-directed MUST in `placeholder-rendering-b.md`'s §8 `<failure>` carve-out paragraph, `code-registry-host.md`'s `session-shutdown-pinned-constant-unreadable` row prose, and `unknown-reason-rule.md`'s `"throw:<String(error)>"` discriminator-grammar paragraph from a normative obligation on external operator tooling to non-normative guidance, presenting the `startsWith("throw:")` predicate as an example parse rather than a requirement. Keep the producer-side wire-format obligations normative — the closed discriminator set, the construction-site truncation/escape rules, and the `"throw:"`-prefix preservation on the coercion-failure fallback.

## Solution constraints

- Out of scope: the `details.failure` TypeScript closed template-literal union type in `unknown-reason-rule.md` (a consumer static-typing aid, not an operator-directed MUST).
- Out of scope: §8's build-time prohibition on strict-equality assertions over placeholder rows (it targets loom-owned test code, not external operator tooling).
