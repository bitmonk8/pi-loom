## Finding

PIC-10 (`host-interfaces-services.md#pic-10`) carries an explicit negative-completeness rule: "Implementations MUST NOT add `before(...)` calls at any site not enumerated above," where the enumerated set is the five `CheckpointKind` sites fixed by the **Granularity** rule in `cancellation.md` (each loop iteration boundary, each `@`-query, each tool call, each `invoke`, the binder LLM call). Both `V8a` and `V8a-T` close PIC-10 with a single Tests bullet — "a checkpoint awaits at each defined cancel site with the correct yield kind and adds no extra sites" — and `V8a`'s Ships-when repeats "no extra sites."

The positive half (a checkpoint fires at each driven site, with the correct yield kind) is observable: a behavioural test drives each checkpoint kind and asserts the `before(...)` call and its yield. The negative half — "adds no extra sites" — is not. A behavioural test only observes the call sites it actually drives; a stray `before(...)` planted on an un-exercised code path is never invoked, so it never appears in the observed call log and the "no extra sites" assertion passes vacuously. The leaf names a verification method that cannot, by construction, back the MUST-NOT half of the obligation it claims to close.

The consequence is that the PIC-10 negative-completeness rule has no enforcing signal. An implementation that adds an out-of-enumeration `before(...)` at a site the test corpus does not exercise ships green while the leaf asserts conformance.

## Solution Space

**Shape:** multiple

### Option A — Static enumeration scan reconciling all `before(...)` call sites

**Approach.** Add a static (source-level) scan that finds every `before(...)` call site in `src/**` and asserts each one corresponds to one of the five enumerated `CheckpointKind` / **Granularity** sites — the same scan shape `H3a` already uses for the ambient-access ban. This backs the negative half independently of which paths the behavioural tests drive.

**Plan edits.** Either extend `H3a`'s architectural scan to also cover `before(...)` call sites against the enumerated `CheckpointKind` set, or add the scan to `V8a`; in `V8a`/`V8a-T` reword the `PIC-10` Tests bullet so the "no extra sites" half is closed by the static scan and the behavioural test closes only the per-site yield-kind half. Keep `V8a` and `V8a-T` in lockstep.

**Spec edits.** None. PIC-10 already states the MUST-NOT.

**Pros.** Actually enforces the PIC-10 MUST-NOT; catches a stray checkpoint regardless of test coverage; mirrors an enforcement pattern the corpus already trusts (`H3a`).

**Cons.** A source-token scan catches only direct `before(...)` call forms; indirect forms (aliased/wrapped invocations) escape it, the same residue `H3a`'s scan acknowledges.

**Risks.** If folded into `H3a`, widens `H3a`'s scope and its Tests/Ships-when; if added to `V8a`, introduces a second scan harness.

### Option B — Soften the claim to what behaviour can observe

**Approach.** Restate the `PIC-10` Tests bullet so it claims only the property a behavioural test can establish: every *exercised* checkpoint is a defined site with the correct yield kind, and the corpus does not assert the absence of un-driven sites.

**Plan edits.** In `V8a` and `V8a-T`, replace "adds no extra sites" with wording scoped to driven checkpoints (e.g. "every checkpoint the test drives fires at a defined cancel site with the correct yield kind"); strike "no extra sites" from `V8a`'s Ships-when. Keep the pair identical.

**Spec edits.** None.

**Pros.** The leaf's claim matches its verification method; no new harness.

**Cons.** Leaves PIC-10's "MUST NOT add `before(...)` at un-enumerated sites" un-gated by the plan — the negative-completeness rule ships on self-review alone.

**Risks.** A future stray checkpoint goes uncaught; the gap is then owned only by the *Per-phase TDD ritual* self-review step.

### Recommendation

Take Option A. PIC-10 states the negative-completeness rule as a normative MUST-NOT, so the plan should carry a mechanical signal for it rather than retreating to behavioural-only coverage; the `H3a` architectural scan already establishes that a source-level scan reconciling call sites against a spec-sourced enumerated set is an accepted enforcement shape here, and the `Checkpoint`-`before(...)`-site set is exactly such an enumeration. Edge cases the implementer must watch: keep the scanned enumeration sourced from the `CheckpointKind` / **Granularity** set so scan and spec stay in lockstep; the source-token scan catches only direct `before(...)` calls (indirect/wrapped invocations remain self-review residue, as with `H3a`); and apply the reworded `PIC-10` bullet identically in `V8a` and `V8a-T`.

## Consequence

**Severity:** correctness

PIC-10's "MUST NOT add `before(...)` calls at any site not enumerated above" ships without an enforcing check: a stray checkpoint on a code path the behavioural test does not drive passes the "no extra sites" assertion vacuously, so a non-conforming implementation can ship green while the leaf claims to have proven the negative. Two implementers reading the leaf disagree on whether the bullet is a real gate or an aspiration.
