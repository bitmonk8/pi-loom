## Finding

The category-(1) carrier rule in `audit-target-categories.md` §*Target surface categories* defines the canonical-`pi` carrier as "a function or method parameter declaration whose name is the literal identifier `pi` and whose explicit type annotation is the literal `ExtensionAPI` (qualified or unqualified, including `import type` re-imports, but resolved against the textual annotation rather than against an inferred type)". The category-(3) clause carries the same construction for `ctx: ExtensionContext` / `ExtensionCommandContext`. The parenthetical is internally inconsistent: a *qualified* annotation such as `pi: someNS.ExtensionAPI` is, textually, **not** the literal `ExtensionAPI`. Under "qualified or unqualified" it is a carrier; under "resolved against the textual annotation rather than against an inferred type" (read as literal-text matching of the bare identifier — the only mechanical reading that does not require a TypeScript program load, which is the explicit non-goal pinned two sentences later) it is not. The rule's purpose is precisely to let a static-AST walker decide carrier status without invoking the type checker, which means the textual-annotation clause is the operative one and the "qualified or unqualified" clause has no implementable referent.

The contradiction is not academic: every downstream prohibition in `audit-recognised-shapes.md` already keys on the bare literal. The off-canonical-annotation clauses prohibit "any function/method parameter declaration whose name is the literal identifier `pi` but whose explicit type annotation is not the literal `ExtensionAPI`", and the wrapped/aliased/generic-applied clause explicitly forbids "aliased forms (`pi: API` where `API` is introduced by an `import type { ExtensionAPI as API } from "@earendil-works/pi-coding-agent"` rename specifier…) — only the three bare textual literals `ExtensionAPI`, `ExtensionContext`, and `ExtensionCommandContext` are permitted as the canonical carrier annotation". A qualified-namespace form falls squarely into the same shape these clauses forbid, so accepting `someNS.ExtensionAPI` as a carrier under the parenthetical would simultaneously fire the family-(4) prohibition under `audit-recognised-shapes.md`.

Two implementers reading the rule will partition the same source tree differently: implementer A admits `someNS.ExtensionAPI` (and every `pi.<member>` on it) into category-(1) coverage; implementer B rejects the parameter as a carrier entirely (so its member accesses fall out of category-(1) coverage and the off-canonical-annotation rule fires under family (4)). The carrier-detection step is the root predicate of every subsequent category-(1) / category-(3) decision — resolution, exemption handling, family-(1)/(3)/(4) routing — so the two implementations produce materially different violation sets on the same source, breaking the surface-set-closure guarantee the audit exists to provide.

## Solution Space

**Shape:** single

### Recommendation

Strike "qualified or unqualified, including `import type` re-imports" from both the category-(1) and category-(3) carrier clauses in `audit-target-categories.md` §*Target surface categories*, and replace it with an explicit literal-text rule. Concretely, rewrite the category-(1) parenthetical to read along the lines of: *"…whose explicit type annotation is the bare identifier `ExtensionAPI` (matched as the literal token `ExtensionAPI`, not as a namespace-qualified form such as `Ns.ExtensionAPI` and not as a renamed re-import — both of which fall under the off-canonical-annotation prohibition in `audit-recognised-shapes.md`), resolved against the textual annotation rather than against an inferred type"*. Apply the symmetric edit to the category-(3) clause for `ExtensionContext` / `ExtensionCommandContext`.

Two implementer-relevant edge cases to pin in the same edit:

- **`import type { ExtensionAPI } from "@earendil-works/pi-coding-agent"`** under the unrenamed (`as`-less) form leaves the textual annotation in parameter position as the bare literal `ExtensionAPI` and remains a carrier — this is the case the original "including `import type` re-imports" clause was reaching for and is preserved by the literal-text rule without needing a separate carve-out.
- **`import type { ExtensionAPI as API } from "@earendil-works/pi-coding-agent"; function f(pi: API)`** is already prohibited by the aliased-import-specifier clause and the wrapped/aliased-annotation clause in `audit-recognised-shapes.md`; the carrier rule's literal-text matching simply does not fire, and family (4) catches the parameter under off-canonical-annotation.

No spec change is needed in `audit-recognised-shapes.md` — the bare-literal prohibition is already pinned there and is the consistency anchor this edit aligns the carrier rule to. The plan-side fixture floor on `V18b-T` should gain one negative test per category for the `someNS.ExtensionAPI` / `someNS.ExtensionContext` qualified-annotation shape so the carrier-rule wording change is mechanically witnessed.

## Consequence

**Severity:** correctness

Two reasonable implementations of the carrier detector will disagree on which parameters are category-(1)/(3) carriers when the annotation is namespace-qualified, producing different violation sets for the same `src/` tree. Because carrier detection gates every downstream category-(1)/(3) decision, the disagreement propagates into resolution outcomes, family-(4) routing, and the surface-set-closure invariant `V18b` is built to enforce — i.e. the gate can pass on one implementation and fail on the other against identical source.
