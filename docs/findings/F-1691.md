## Problem

PIC-52 in `pi-integration-contract/trust-boundary.md` reads "The runtime MUST NOT introduce additional access channels (sandbox, capability filter, mediated proxy) in any loom 1.x release." This is a prohibition on a class of implementation structures, not on an observable outcome: no black-box test can distinguish a conforming runtime from one that has interposed an extra capability filter or mediated proxy. The paragraph's companion denial-surface sentence (denials map to `Err(QueryError{kind:"code_tool", cause:"execution"})`) is observable, but a capability-filter layer that routes its extra denials through the same envelope passes every denial-surface check while violating the MUST NOT. The architectural half thus carries normative-grade weight with no closing test or review gate, unlike GOV-15's comparable claim, which is explicitly framed as a release-process goal rather than a normative obligation.

## Solution approach

Split the PIC-52 paragraph so its remaining MUST is bound to what black-box tests can witness: keep the denial-surface obligation on the `no-extra-mediation` anchor and demote the architectural prohibition (no-privilege-layer / no-additional-channels prose, including "sandbox, capability filter, mediated proxy") out of normative status. Place the demoted content in a non-normative paragraph or fold it into the `model-changes-and-non-goals.md` "No per-loom sandbox or capability model" bullet, which already restates the claim. Delete the "widening or narrowing this rule is a major-version concern" clause that pinned the demoted half. Retarget the `overview-and-orientation.md` Trust-boundary forward-link so it points at the demoted non-normative location while keeping the architectural claim discoverable.

## Solution constraints
