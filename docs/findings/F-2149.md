## Finding

The spec-owned schema name is `QueryError` (capital `Q`, capital `E`). Two heading strings spell it with a lowercase `e`:

- The page title of `docs/spec_topics/errors-and-results/queryerror-variants.md` is `# Queryerror variants`.
- The hub Contents entry in `docs/spec_topics/errors-and-results.md` reproduces the same casing: `[Queryerror variants](./errors-and-results/queryerror-variants.md)`.

Both are inconsistent with the canonical schema name used everywhere else, including the section heading immediately below the page title in the same file (`## QueryError variants`, correctly cased) and the `schema QueryError = …` union declaration. The lowercase spelling reads as a sentence-cased rendering of the filename stem rather than the schema identifier. The Markdown link target is unaffected (the file path is lowercase regardless); only the rendered link text and the rendered page title are wrong.

## Issue introduction
**Verdict:** introduced-by-commit
**Introducing commits:** f5e89f4 (`docs: enforce 100KB hard / 30KB recommended size cap on spec set`, 2026-06-04)
**History:** `git log --diff-filter=A --follow` shows `queryerror-variants.md` was created by the spec-splitting commit f5e89f4; `git show f5e89f4` confirms the same commit added both `# Queryerror variants` (page title, lowercase `e`) and the hand-written `## QueryError variants` (correct casing) in one diff. `git log -S 'Queryerror variants'` against both the topic file and the hub `errors-and-results.md` returns only f5e89f4, so the hub Contents link was added with the same casing in that commit. The lowercase spelling is the sentence-cased filename stem (`queryerror-variants` → "Queryerror variants") that the split produced automatically, while the body section heading was authored by hand with the correct schema casing. Later commits touching the file (`f094b5b`, `67fc9a1`) are spec-review store operations, not the source of the casing.

## Solution Space

**Shape:** single

### Recommendation

Change the page title in `docs/spec_topics/errors-and-results/queryerror-variants.md` from `# Queryerror variants` to `# QueryError variants`, and change the hub Contents link text in `docs/spec_topics/errors-and-results.md` from `[Queryerror variants]` to `[QueryError variants]`. Leave both link/file paths (`queryerror-variants.md`) unchanged — only the rendered text is corrected.

## Consequence

**Severity:** cosmetic

A reader sees the schema's defining page titled with a different casing than the schema name itself, slightly undermining the spec's own naming discipline. No runtime behaviour, link resolution, or implementer decision is affected.
