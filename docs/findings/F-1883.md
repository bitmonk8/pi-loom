## Finding

The *Sequential by default* rule in `conventions.md` now governs **two** distinct same-line exemption-comment forms: `// allow: <REQ-ID-or-code-keyed-area-token> — <spec-page>` for the four banned Promise combinators (`Promise.all`/`race`/`allSettled`/`any`), and `// allow-sync: <reason>` for the blocking-runtime ban (`fs.readFileSync`, `execSync`, and every other callee whose name ends in the literal suffix `Sync`). Both forms are introduced and enforced under this one rule.

The carve-out for test code, however, is phrased in the singular: "Test code (`**/*.test.ts`) is unrestricted but MUST NOT carry the exemption comment (so `grep`-based audits stay clean)." With two exemption-comment forms in scope, "the exemption comment" does not say whether the prohibition covers only `// allow:` (the form discussed in the immediately-preceding sentence), only `// allow-sync:`, or both. The sentence sits between the `// allow:` paragraph and the blocking-runtime / `// allow-sync:` paragraph, which makes the narrower reading (just `// allow:`) plausible.

`H2a` is the leaf that implements both ESLint surfaces (the `no-restricted-syntax` Promise-combinator allow-list and the blocking-runtime `Sync`-suffix check). Its implementer must decide which exemption-comment forms a `**/*.test.ts` file is forbidden to carry, and the rule does not pin the answer.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md`, in the *Sequential by default* bullet, edit the test-code carve-out sentence so it names both exemption-comment forms as forbidden in `**/*.test.ts`. Replace "MUST NOT carry the exemption comment" with text equivalent to: "MUST NOT carry either exemption-comment form — neither `// allow:` nor `// allow-sync:`". Keep the parenthetical rationale ("so `grep`-based audits stay clean") intact. No spec edit and no change to `H2a`'s Ships-when are required; `H2a`'s existing "`**/*.test.ts` is unrestricted" Tests bullets remain accurate (unrestricted as to the combinator/blocking bans themselves), and the clarified rule resolves which comment forms its lint must reject in test files.

## Consequence

**Severity:** correctness

Two reasonable `H2a` implementers diverge: one prohibits only `// allow:` in `**/*.test.ts` (leaving `// allow-sync:` comments to pass through test files), the other prohibits both forms. The grep-cleanliness purpose — that exemption tokens in production-code audits are never diluted by stray test-code occurrences — is met only by the both-forms reading, so the narrower implementation silently undermines the rule's stated rationale.
