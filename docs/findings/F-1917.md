## Finding

`V4e` asserts load-time pre-evaluation failure routing for `ERR-1тАжERR-6` plus `ERR-16`, and its `Deps` field drives each cause's failure through a real producer leaf: `ERR-1` (host-incompatible) through `V9a`, `ERR-3` (frontmatter) through `V6a`, `ERR-5` (binder arg-binding, ceiling #3) through `V11f`, and `ERR-16` (slash-load `params` ceiling #4) through `V16a`/`V5e`. The test methodology is therefore "drive a real failure through its producer", not "inject a synthetic failure signal".

`ERR-6` (`tools:` resolution failure) breaks that pattern. Its producer is `V6c` (`tools` callable set and resolution snapshot), which owns every `tools:` load-time rejection code (`loom/load/unknown-tool`, `loom/load/unresolvable-loom-path`, `loom/load/tool-name-collision`, `loom/load/prompt-mode-callable`, `loom/load/invalid-tool-rename`). `V6c` appears in neither `V4e.Deps` nor `V4e-T.Deps`, and it is not pulled in transitively: no leaf in `V4e`'s dependency closure depends on `V6c` (nothing in the plan depends on `V6c` at all except the all-leaves `H5b` canary). Under the pickability predicate, `V4e`/`V4e-T` become pickable before `V6c`'s `tools:`-resolution code exists, so the `ERR-6` arm cannot drive a genuine `tools:` resolution failure the way the sibling arms drive theirs.

The `ERR-4` half of the original concern does not hold: `ERR-4`'s producer is `V11a` (binder-model resolution, owner of `loom/load/binder-model-unresolved`), and `V11a` is already in `V4e`'s transitive closure via `V4e тЖТ V11f тЖТ V9j тЖТ V11a`. So `ERR-4` is reachable and benign; the gap is `ERR-6`/`V6c` only.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** `V4e`/`V4e-T` were authored in `c6a664e` with the `ERR-6` test bullet and a `Deps` line that already omitted `V6c`; `git log -S 'V6c'` over `V4e-pre-evaluation-failures.md` returns no commit, so `V6c` was never present in the leaf's `Deps`. Later edits (`887c440` added `V5e` for `ERR-16`; `324813d` split `ERR-7` out to `V4g`) reshaped other arms but never touched the `ERR-6` producer edge. The asymmetry has existed continuously since the leaf was first created.

## Solution Space

**Shape:** single

### Recommendation

Add `V6c` to the `Deps` field of both `docs/plan_topics/V4e-pre-evaluation-failures.md` and `docs/plan_topics/V4e-T-pre-evaluation-failures.md`, so the `ERR-6` arm drives a real `tools:` resolution failure through `V6c`'s producer, matching the real-producer pattern the other arms already follow (`ERR-1`тЖТ`V9a`, `ERR-3`тЖТ`V6a`, `ERR-5`тЖТ`V11f`, `ERR-16`тЖТ`V16a`/`V5e`).

No `V11a` edge is needed for `ERR-4`: `V11a` is already reachable via `V4e тЖТ V11f тЖТ V9j тЖТ V11a`. Adding `V6c` is acyclic тАФ `V6c`'s closure (`V6a`, `V15a`, `V9f`, and through `V15a` the `V4f`/`V17c` chain) does not reach `V4e`. If the `Deps` cross-reference rule in How-to-use step 2 / `conventions.md` applies to this edit, apply it as written; the spec is read-only for this fix.

## Consequence

**Severity:** correctness

`V4e`/`V4e-T` are pickable before `V6c` lands, so the `ERR-6` test cannot drive a real `tools:` resolution failure the way every sibling arm drives its cause through its producer. An implementer following the established pattern either stalls or stubs a synthetic `tools:`-failure signal that diverges from `V6c`'s actual routing, leaving the `ERR-6` routing claim under-exercised while the rest of the leaf's gate passes green.
