## Problem

`V10c`'s **Adds** promises "the settings-source reads (the five keys), the deep-merge precedence … and validation," but its **Tests** name only `DISC-7` (deep-merge precedence) and `loom/load/settings-invalid-json` (malformed-JSON diagnostic). The leaf's owned spec section, `discovery/package-and-settings.md` §"Settings file reads", defines further validation behaviours that no Tests bullet asserts: `loom/load/settings-unreadable` fall-through, top-level shape validation, scalar-key validation for the four `looms.*` scalar keys, and per-entry `loom/load/settings-invalid-entry` rejection. Because `V10c`'s **Ships when** requires only the deep-merge precedence and "a malformed-settings diagnostic", the leaf can be marked done with merge + invalid-JSON proven and every other validation behaviour unverified.

## Solution approach

Add Tests bullets to `docs/plan_topics/V10c-settings-merge.md` covering the remaining validation behaviours of spec §"Settings file reads" — `loom/load/settings-unreadable` fall-through, top-level shape validation via `loom/load/settings-value-out-of-range`, scalar-key validation for the four `looms.*` scalar keys, and per-entry `loom/load/settings-invalid-entry` rejection — and mirror the same bullets into the paired test leaf `docs/plan_topics/V10c-T-settings-merge.md`. Narrow `V10c`'s **Ships when** so the closing gate cannot be satisfied without those validation assertions, rather than merge + invalid-JSON alone.
