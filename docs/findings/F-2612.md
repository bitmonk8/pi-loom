## Problem

The `qry-6` parse-time-warning bullet describes suppressing the `loom/parse/empty-template` warning by writing an explicit literal escape, rendered in the parenthetical as a bare `` `\n` `` glyph. That glyph is the same one the spec uses elsewhere for the line-feed character U+000A (QRY-7 rule 4, and the escapes table in `query-escapes-stringification.md`), so the two readings are indistinguishable in rendered Markdown. A reader who imports the U+000A reading reaches the inverted conclusion that a whitespace-only newline would not suppress the warning, making the hatch appear ineffective. The surrounding "non-whitespace two-character sequence" prose disambiguates on full read, so this is a glyph-level clarity gap rather than a contradiction.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 82be125 (2026-05-04, "spec: adopt P1 typed-query two-phase loop (Option A)")
**History:** The corpus is git-tracked. `git log -S 'suppress the warning by writing an explicit literal escape'` and `git log -S 'loom/parse/empty-template'` trace the empty-template warning and its suppression hatch to 82be125, the commit that first authored the two-layer degenerate-template defence. That commit already wrote the prose hatch as `(`\n`)` while the diagnostics-table hint in the same commit wrote `\\n`, so the glyph overload existed at the feature's inception. A later commit (2637aec, 2026-06-20, "fix F-0755 — empty-template predicate pinned to pre-escape source segments") expanded the explanation to add "the non-whitespace two-character sequence `` \n ``" but retained the ambiguous bare `\n` in the parenthetical, so the defect was never the product of a single regression — it has been carried forward since 82be125.

## Solution approach

Clarify the suppression-hatch clause in the `qry-6` parse-time-warning bullet so the first occurrence of the escape is self-disambiguating as the two-character source sequence (backslash followed by `n`) rather than the U+000A character. Keep the existing pre-escape predicate semantics unchanged.
