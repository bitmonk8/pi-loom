## Finding

The Echo policy's *Object values* bullet defines first-field rendering for discriminated-union values as: *"the variant's declared fields are used in the variant's own source order; the discriminator field is included in that order if it appears there."* The trailing conditional reads as if the discriminator's appearance in the variant's declared fields were optional, leaving the absent-discriminator case unspecified and inviting an implementer to guess what `{first-field-value, …}` should render when the discriminator is missing.

The rest of the spec forecloses the conditional's "absent" arm. `schemas.md` §Discriminated unions requires the discriminator to *"Be present in every variant"* and *"live at the top level of each variant"* (`loom/parse/nested-discriminator` rejects nested discriminators; `loom/parse/missing-discriminator` rejects unions where no candidate qualifies). Together those rules guarantee that the discriminator is always one of the variant's declared top-level fields — exactly the field set this echo rule iterates. The "if it appears there" hedge therefore guards a case the schema layer has already eliminated; the wording reads as live optionality only because the echo rule doesn't say so.

The defect is plain — the conditional should be removed (or replaced with a statement that the discriminator is always present in source order) so the first-field selection is fully determined on every conforming variant. The BNDR-6h reference rendering already exercises this case (`Cat { name: "Whiskers" }` of `schema Animal = Cat | Dog` renders `{cat, …}`, picking the implicit `kind` discriminator declared first), but the prose's hedge undermines what the table demonstrates.

**History:** Commit `edc2e1b` introduced the *Object values* bullet's first-field definition and added the sentence *"For a value whose static type is a discriminated union, the variant's declared fields are used in the variant's own source order; the discriminator field is included in that order if it appears there."* The "if it appears there" conditional has been present verbatim since that commit. Subsequent edits to the bullet — `800b3f9` (composite first-field recursion), `71e8313` (inline-object first-field rule), `e44c234` (illustrative example), and the corpus-wide split into `binder/defaulting-system-note-echo.md` at `f5e89f4` — left the conditional unchanged. No commit ever revisited the wording against the `schemas.md` always-present + top-level invariants that make the conditional dead.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/binder/defaulting-system-note-echo.md`, §Echo policy *Object values* bullet, replace

> the discriminator field is included in that order if it appears there.

with

> the discriminator field appears in that order (per [Schemas — Discriminated unions](../schemas.md), the discriminator MUST be present at the top level of every variant).

The cross-reference makes it explicit that the always-present property is enforced by the schema layer and that the echo rule consumes the invariant rather than restating an obligation. No table rows or other rules need to change — BNDR-6h already pins the intended rendering — and the existing parenthetical pointers in this section follow the same `[Schemas](../schemas.md)`-style reference shape, so the edit is local to one sentence.

## Consequence

**Severity:** advisory

Two conforming binder implementations could disagree on the first-field rendering of a discriminated-union value if either reads "if it appears there" as live optionality and invents a fallback (e.g. emit `{…}`, fall back to the variant's second declared field, or use the discriminator's literal value regardless of declaration order). Because the schema layer already guarantees the discriminator is present at the top level of every variant, the wording is dead in practice — but as written it is the kind of hedge an implementer can act on, and the BNDR-6h reference rendering is the only thing pinning the intended behaviour.
