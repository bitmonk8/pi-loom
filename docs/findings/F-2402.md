## Problem

The capability-probe page's single-source-of-truth note declares that the four
probe constants (Node floor, `AbortSignal`/`AbortController` member-with-kind
list, factory-probable capability list, peer-dep range) "live in one source of
truth inside the extension module that the build-time literal-read assertions
also consume." Two build-time consumers — the step-2(a) surface-inventory
literal-read assertion and the `CAPABILITY_OBLIGATIONS.length === 7` cardinality
assertion — import that block and evaluate it mechanically. The spec never
states the precondition that makes this sound: that the block must be reachable
by a side-effect-free import that does not invoke the
`default function (pi: ExtensionAPI)` factory or trigger any registration or
import-time side effect. An implementation that co-locates the block with
import-time side effects breaks both assertions (or makes them register host
bindings) without violating any spec text.

## Solution approach

Clarify the single-source-of-truth note in `capability-probe.md` to add the
importability precondition: the pinned-constants block MUST be reachable by a
side-effect-free import — importing it to read the constants MUST NOT invoke the
`default function (pi: ExtensionAPI)` factory or perform any `pi.register*` /
`pi.on` host-binding or other import-time side effect. State it behaviourally
(side-effect-free static reachability) so an implementation may satisfy it by
top-level constants in the extension module or by a dedicated constants module.
