## Problem

`V4b`'s `Adds.` field catalogues the runtime-defect surface — the closed six-source runtime-panic set, the `internal-error` surface for unexpected throws, and the rule that panics bypass `?`/`match` — but omits the NOCEIL-3 host-fatal carve-out: that a host-fatal *uncatchable* condition emits no diagnostic at all (no `loom/runtime/internal-error`). That inverse behavioural rule appears only in a Tests bullet, in both `V4b` and `V4b-T`. The `Adds.` field is the leaf's normative scope catalogue, so a binding test asserts a rule the catalogue never states, leaving the prose surface inventory and the binding tests asymmetric.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** `docs/plan_topics/V4b-runtime-panics.md` was created in c6a664e with the `Adds.` field already naming the `internal-error` surface but not the NOCEIL-3 no-diagnostic carve-out, while a Tests bullet already asserted that carve-out — the placement asymmetry existed from the file's first revision. The only subsequent edit, 2ff06f2 (2026-06-16, "resolve 'V4b internal-error Tests bullet inverts lead-code-fires pattern'"), reworded the carve-out Tests bullet for clarity and did not touch `Adds.`, so it neither introduced nor resolved the gap.

## Solution approach

Extend the `Adds.` field in `docs/plan_topics/V4b-runtime-panics.md` so its scope catalogue also names the NOCEIL-3 host-fatal carve-out — that a host-fatal uncatchable condition emits no diagnostic at all (no `loom/runtime/internal-error`) — matching the behaviour the existing `V4b` Tests bullet asserts.

- Out of scope: `V4b-T` — no change to the tests leaf is required.
