## Problem

`docs/spec_topics/query/query-failure-and-repair.md` opens with a top-level `## Options surface` section before its `# Query failure and repair` page produces any failure or repair content, yet the section's entire body is a single non-normative `loom 1.0 seam — per-call timeout / per-query overrides` blockquote documenting deferred extensions. The heading promises a current options API the query primitive accepts today, misleading a reader scanning for one that does not exist in loom 1.0. The seam also sits on the wrong page under a dedicated wrapper heading, where the sibling primitives carry their analogous seam inline next to the primitive body (`tool-calls.md`, `invocation.md` INV-3). The blockquote is load-bearing: `future-considerations/surface-extensions.md` cites `[Query — Options surface](../query.md)` as the sole anchor for the "Per-call timeouts" and "Per-query overrides" inventory bullets, and those citations already point at a `query.md` anchor that no longer exists post-split.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `09c42c0` (2026-05-05, *pi-loom spec: resolve "Surface-extension Seam annotations carry V1 obligations from non-goals page"*); `f5e89f4` (2026-06-04, *docs: enforce 100KB hard / 30KB recommended size cap on spec set*)
**History:** `git log -S "## Options surface" -- spec_topics/query.md docs/spec_topics/query/query-failure-and-repair.md` returns exactly these two commits. `09c42c0` introduced `## Options surface` into the then-monolithic `spec_topics/query.md`, inserted between the "Untyped form" `Ok`-payload paragraph and the `## Failure modes` heading — adjacent to forward-compatibility material, with `## Failure modes` as its natural downstream neighbour. `f5e89f4` then split `query.md` into the subdirectory + index layout (this is the size-cap split that produced the four-page subtree `query-forms.md` / `query-escapes-stringification.md` / `query-tool-loop.md` / `query-failure-and-repair.md`); the `## Options surface` section was routed onto `query-failure-and-repair.md` as its first section rather than onto `query-forms.md` next to the primitive's introduction. The heading text "Options surface" — non-ideal but not misleading while it sat between two query-options-adjacent neighbours — became actively misleading once it was the opening section of a page whose H1 is "Query failure and repair"; the surface-extensions cross-references `[Query — Options surface](../query.md)` were not repointed in `f5e89f4` (they still target `query.md`, where the anchor no longer exists), confirming the split was the routing event that crystallised the placement defect. The sibling primitives' seam pages (`tool-calls.md`, `invocation.md`) were not affected by the size-cap split and retained their inline placement, which is why only the query surface exhibits this defect.

## Solution approach

Move the `loom 1.0 seam — per-call timeout / per-query overrides` blockquote out of `query-failure-and-repair.md`'s `## Options surface` section and into `query-forms.md` against the primitive introduction (`## Untyped form` / `## Typed form`), matching the inline seam placement on `tool-calls.md` and `invocation.md`. Delete the emptied `## Options surface` heading so the page opens on `## Failure modes`. Add a stable `<a id="…">` to the relocated blockquote and repoint the two `[Query — Options surface](../query.md)` citations in `future-considerations/surface-extensions.md` to that anchor.

## Solution constraints

- Out of scope: deleting the seam — it is the sole corpus anchor for the per-call-timeout and per-query-override inventory bullets and feeds the GOV-31 forward-compatibility-seam tally.
