## Problem

`docs/spec_topics/diagnostics/code-registry-runtime.md` spells the same `kind: "invoke_infra"` invoke-parent panic surface two ways. The panic-sources channel-summary paragraph (under `### loom/runtime/*`) writes it as `Err(QueryError { kind: "invoke_infra", cause: "panic", ... })`, while the `loom/runtime/internal-error` and `loom/runtime/subagent-model-unresolved` rows write it as `Err(InvokeInfraError { kind: "invoke_infra", ... })`. The two are the same envelope — `InvokeInfraError` is the `kind: "invoke_infra"` variant of the `QueryError` union per `errors-and-results/queryerror-variants.md` — but the page neither uses one spelling consistently nor cross-links the canonical definition, so a reader confined to this page cannot tell whether `QueryError` and `InvokeInfraError` are one type or two.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** 8eea523 (`spec: fill implementation-blocking gaps`, 2026-05-04) — channel-summary prose using the `Err(QueryError { kind: ... })` spelling for the invoke-parent panic surface; f8f4813 (`pi-loom plan: resolve "Panic routing has no defined path for unexpected interpreter exceptions"`, 2026-05-05) — the `loom/runtime/internal-error` routing row using the `Err(InvokeInfraError { kind: ... })` spelling.
**History:** `git log -S` over both literal forms (and their earlier discriminant spellings `invoke_failure` → `invoke_infra_error` → `invoke_infra`) shows the divergence is not the product of any single edit. The two halves of the panic-routing contract were authored one day apart in separate commits using different type-name conventions: the channel-summary line adopted the union name `QueryError` (8eea523), while the per-code routing row adopted the variant name `InvokeInfraError` (f8f4813). The inconsistency then survived every subsequent discriminant rename — ff9986d (`invoke_failure` → `invoke_infra_error`, 2026-06-04), 8455d5e (`invoke_infra_error` → `invoke_infra`, 2026-06-04) — and the size-cap file split f5e89f4 (2026-06-04) that carved this content into `code-registry-runtime.md`, each of which faithfully preserved both spellings without harmonising them. The defect is the durable interaction of the two original authoring conventions, not a regression introduced by any later commit.

## Solution approach

Rename the channel-summary paragraph's `Err(QueryError { kind: "invoke_infra", cause: "panic", ... })` to use `InvokeInfraError`, so all three usages on the page match the variant-naming convention of `queryerror-variants.md`. At that first use, add a forward-link to `queryerror-variants.md#invoke-variants` identifying `InvokeInfraError` as the `kind: "invoke_infra"` variant of the `QueryError` union returned by `invoke`.

## Solution constraints

- Do not alter the `kind: "invoke_infra"` discriminator value or any `cause` value — those are the stable wire contract; only the loom-side schema-identifier spelling is being unified.
