## Problem

BNDR-9 (`#bndr-9` in `binder-model-and-context.md`) specifies the detection rule, the diagnostic code `loom/runtime/custom-type-unsafe`, and that the binder "abort[s] transcript construction for that invocation", but it says nothing about what the invocation does next — whether the binder proceeds with the Session-context block omitted or the invocation fails and surfaces a user-facing system note. The `loom/runtime/custom-type-unsafe` registry row in `code-registry-runtime.md` adds "the affected slash/prompt invocation's argument binding does not proceed", which contradicts BNDR-9's silence, while the failure-mode templates table (which BNDR-3 pins as the canonical source for user-facing binder system notes) carries no row for this cause. So the user-facing outcome is undefined and a conformance test (V11b) has no oracle for whether the invocation completes or for what the user sees; two reasonable implementations diverge on both.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `9568906` (2026-06-09) — *pi-loom spec: resolve "custom:<type> role tag customType safe-character constraint"*
**History:** BNDR-9 was added to `binder-model-and-context.md` in a single commit that simultaneously added the `loom/runtime/custom-type-unsafe` row to `code-registry-runtime.md`. That commit introduced both the "abort transcript construction" phrasing in BNDR-9 and the "argument binding does not proceed" phrasing in the diagnostic row, with no row added to the failure-mode templates table in `determinism-cancellation-failure.md` (which the same commit did not touch). The gap — undefined invocation-level outcome and missing failure-mode-templates row — has been present since BNDR-9 first existed; no subsequent commit narrowed the silence.

## Solution approach

Clarify BNDR-9 (`#bndr-9`) to define what the invocation does after an unsafe `customType` is detected: have the binder proceed with the Session-context block omitted, reusing BNDR-7i's (`#bndr-7i`) void-truncation rendering rather than minting a new user-facing system-note template. Reconcile the `loom/runtime/custom-type-unsafe` row in `code-registry-runtime.md` in lockstep so its "argument binding does not proceed" wording matches the chosen invocation outcome. Extend the V11b acceptance assertions to exercise both the diagnostic firing and the Session-context-block omission.
