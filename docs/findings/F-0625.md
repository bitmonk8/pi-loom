## Finding

BNDR-9 (`binder-model-and-context.md`) specifies three things on encountering a non-transcript-safe `CustomMessage.customType`: the detection rule, the diagnostic code (`loom/runtime/custom-type-unsafe`), and that the binder "abort[s] transcript construction for that invocation." It says nothing about what the invocation does *next* — whether the binder proceeds with the Session-context block omitted (graceful degradation analogous to BNDR-7i's void-truncation result), or whether the invocation fails and surfaces a user-facing system note.

The diagnostic registry row for `loom/runtime/custom-type-unsafe` (`diagnostics/code-registry-runtime.md`) adds a stronger fragment — "the affected slash/prompt invocation's argument binding does not proceed" — and gives an internal Message template (`custom-message type is not transcript-safe: '<value>'`) routed through the `loom-system-note` channel. But this contradicts the more permissive reading of BNDR-9, and the failure-mode templates table in `determinism-cancellation-failure.md` (which BNDR-3 establishes as the canonical source for *user-facing* binder system notes) carries no row for this cause. So the user-facing wire string is undefined: the diagnostic Message template is for operator-visible diagnostic payloads, not for the failure-mode-templates `loom /<name>: …` system-note prefix family that BNDR-3 pins as breaking-change-protected.

A conformance test cannot write a complete Then-clause: it can assert that `loom/runtime/custom-type-unsafe` fires (and V11b's Ships-when does exactly that and no more), but it has no normative oracle for whether the invocation completes or for what the user sees. Two reasonable implementations diverge on both axes.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `9568906` (2026-06-09) — *pi-loom spec: resolve "custom:<type> role tag customType safe-character constraint"*
**History:** BNDR-9 was added to `binder-model-and-context.md` in a single commit that simultaneously added the `loom/runtime/custom-type-unsafe` row to `code-registry-runtime.md`. That commit introduced both the "abort transcript construction" phrasing in BNDR-9 and the "argument binding does not proceed" phrasing in the diagnostic row, with no row added to the failure-mode templates table in `determinism-cancellation-failure.md` (which the same commit did not touch). The gap — undefined invocation-level outcome and missing failure-mode-templates row — has been present since BNDR-9 first existed; no subsequent commit narrowed the silence.

## Solution Space

**Shape:** multiple

### Option A — Fail the invocation; add a failure-mode-templates row

**Approach.** Treat unsafe `customType` as a binder-input precondition failure that prevents the loom from running, matching the diagnostic registry's existing "does not proceed" wording.

**Spec edits.**
- In `binder-model-and-context.md` BNDR-9, append a normative outcome clause: *"The affected slash/prompt invocation MUST NOT proceed: the runtime surfaces the `Compact-transcript custom-type-unsafe` row of the [failure-mode templates](../binder/determinism-cancellation-failure.md#failure-mode-templates-normative) and does not invoke the loom body."*
- In `determinism-cancellation-failure.md` add a new row to the *Failure-mode templates (normative)* table, e.g. `| Compact-transcript custom-type-unsafe (no retry) | `loom /<name>: argument binding aborted — custom-message type is not transcript-safe (<value>)` |`. Pick `<value>`'s rendering rule per the same `<value>` category used by the diagnostic registry, and pin it in the table's preamble like the existing `<provider>` / `<ajv-summary>` placeholders.
- In `code-registry-runtime.md`'s `loom/runtime/custom-type-unsafe` row, replace "the affected slash/prompt invocation's argument binding does not proceed" with a cross-reference to the new failure-mode-templates row so the two wordings cannot drift.
- Per the retry-budget paragraph that already exists, declare this cause not retry-eligible (it is a deterministic input-validation failure; retrying would re-fail).

**Pros.** Honours the diagnostic-registry's existing intent. The failure mode is already structurally identical to the malformed-envelope class (no possible recovery via re-issue). Gives test authors an exact user-facing string. Single user-visible surface (the existing failure-mode-templates family) for every binder-input failure.

**Cons.** Adds a new breaking-change-protected wire string. The session-context block is grounding context, not arguments — failing the invocation when grounding can be silently omitted may surprise authors.

**Risks.** Wording of the new row needs to be reviewed alongside the other six rows for prefix consistency under BNDR-3.

### Option B — Graceful degradation; omit the Session-context block

**Approach.** Treat unsafe `customType` as making the session-context input un-renderable, drop to the existing void-rendering pathway (BNDR-7i / item 6 of *System-prompt structure (normative)*), and let the binder run.

**Spec edits.**
- In `binder-model-and-context.md` BNDR-9, replace "aborting transcript construction for that invocation" with "aborting transcript construction for that invocation and proceeding as if [Session-context truncation](#session-context-truncation-bind_context-session) had produced zero included turns: per BNDR-7i / [System-prompt structure (normative) item 6](./binder-bypass-and-envelope.md#system-prompt-structure-normative), the entire Session-context block is omitted and the binder runs without it."
- In `code-registry-runtime.md`'s `loom/runtime/custom-type-unsafe` row, drop the "argument binding does not proceed" clause and replace with "the Session-context block is omitted for this invocation; the loom body still runs."
- No row needed in the failure-mode templates table; the diagnostic remains the operator-visible signal.

**Pros.** Reuses the existing void-rendering oracle; no new user-facing system-note template, no new breaking-change-protected wire string. Keeps looms running when only the grounding context is broken. Cheaper test (the BNDR-7i oracle already exists; the new assertion is just "fires the diagnostic AND emits no Session-context block").

**Cons.** Contradicts the existing diagnostic-registry "does not proceed" wording — that row must be edited to match. A future audit reading only BNDR-9 might miss that the loom still ran (the diagnostic alone does not say so explicitly).

**Risks.** Custom-messages with unsafe types are likely an extension-author bug; silently dropping the grounding could mask the bug from the loom author (the diagnostic surfaces it, but at `display: false` on the operator channel rather than as a user-facing note).

### Recommendation

Take **Option B (graceful degradation)**. The session-context block is grounding-only — every other void-context pathway (empty session, single oversized newest turn, `bind_context: none`) already runs the binder without it via BNDR-7i, so this reuses an existing, tested rendering oracle and avoids minting a new breaking-change-protected system-note template. Test the *absence* of the Session-context block using the BNDR-7i oracle, and add the diagnostic-firing assertion that V11b already carries. Implementer must edit the `loom/runtime/custom-type-unsafe` registry row in lockstep so its "does not proceed" wording is replaced with the "Session-context block is omitted" phrasing; the V11b acceptance bullets should grow a second clause asserting the omission so the two oracles are exercised together.

## Consequence

**Severity:** correctness

A test author has no oracle for the post-detection behaviour and the user-visible system note is undefined, so the V11b acceptance criteria are satisfied by implementations that diverge on whether the loom runs at all. The diagnostic-registry entry's "argument binding does not proceed" line contradicts BNDR-9's silence and is not where conformance authors are directed to look for binder failure semantics (BNDR-3 points at the failure-mode templates table), so the contradiction can persist undetected through code review.
