## Finding

GOV-3 on `docs/spec_topics/governance/req-id-prefix-table-active-a.md` operates at byte granularity in two distinct places. The extractor "operates on raw Markdown source bytes, not on rendered HTML" before applying the `[A-Z]{2,4}` prefix regex. The narrative-exclusion cell carries an explicit byte specification: "open paren, `no IDs`, ASCII space, U+2014 EM DASH, ASCII space, `narrative`, close paren", with the closing MUST that "the cell-value comparison that classifies a row as narrative MUST be byte-exact." Both clauses are written in terms of bytes, not Unicode scalars.

Nothing in the spec corpus, however, pins the encoding of the spec-corpus markdown files themselves. `lexical.md` §Encoding states that `.loom` / `.warp` source files are UTF-8 with an optional leading BOM, and a handful of pages cite UTF-8 for other artefacts (canonical schema-hash serialisation in `schema-subset.md`, audit wire format in `pi-integration-contract/audit-wire-and-canary.md`, settings files in `discovery/package-and-settings.md`). None of these binds the spec corpus. `corpus-direction-and-scope.md` GOV-17 defines the corpus as `docs/spec.md` and `docs/spec_topics/**/*.md` but says nothing about their encoding.

The U+2014 EM DASH literal is the sharpest witness: it is the 3-byte sequence `E2 80 94` in UTF-8, the 2-byte sequence `14 20` (or `20 14`) in UTF-16, and unencodable in ASCII. A maintainer who re-saved one of these pages as UTF-16 (or UTF-8 with a BOM, where the BOM appears before the first table cell) would silently change the byte sequence that GOV-3's narrative-cell comparison and the extractor's regex inputs range over. The byte-exact MUSTs become incoherent: the rule binds an output (the bytes the comparison sees) without pinning the input (which encoded form of the source file produces those bytes). The spec corpus needs an arm-(b) invariant — pinning the corpus encoding as UTF-8, no BOM — that the byte-level GOV-3 clauses (and, by extension, GOV-6's prefix-token scan and the extractor used by H5e's text-scan gate) can range over coherently.

## Solution Space

**Shape:** single

### Recommendation

Add a one-sentence arm-(b) invariant on `docs/spec_topics/governance/req-id-prefix-table-active-a.md` GOV-3, sited at the head of the rule before "REQ-ID prefixes are byte-exact uppercase ASCII tokens…", that pins the spec-corpus encoding the byte-level clauses range over:

> *Corpus encoding.* Files in the spec corpus (`docs/spec.md` and `docs/spec_topics/**/*.md`, per [GOV-17](./corpus-direction-and-scope.md#gov-17)) MUST be encoded as UTF-8 with no byte-order mark. The byte-level scans this rule defines — the prefix regex over raw Markdown source bytes and the narrative-cell byte-exact comparison below — range over the file's UTF-8 byte sequence on this footing.

Place the sentence on GOV-3 itself rather than on GOV-17. GOV-3 is the rule whose semantics fail open without the invariant, and the closing-gate consumer in plan-side prose already cites GOV-3 (not GOV-17) as the byte-token authority; co-locating the invariant with the dependent MUSTs is the lower-coupling site. Allow no BOM rather than mirroring `lexical.md`'s "BOM consumed and ignored" rule for `.loom` / `.warp` files: the spec corpus has no read step that strips a BOM before scanning, so allowing one would re-open the same byte-exact-comparison defect for files saved with `EF BB BF` prefixed.

Watch one edge case at implementation time: a contributor who saves a governance page with a BOM today would, under this rule, render that file a spec defect. Run a one-off `grep -rlP '^\xef\xbb\xbf' docs/spec.md docs/spec_topics/` (or equivalent) at the commit that lands the invariant to confirm no current file already violates it; if any do, strip the BOMs in the same commit so the invariant lands true on `main` from its first appearance.

## Consequence

**Severity:** advisory

Two readers building a third-party REQ-ID extractor against the spec corpus would both pick UTF-8 by default — the defect is not implementer-diverging in practice. The cost is concrete the moment a contributor's editor or a Windows tool re-saves a governance page with a BOM or in UTF-16: the byte-exact narrative-cell comparison and the prefix-regex scan silently change their inputs, and GOV-3 has no rule it can cite to reject the file. The invariant is cheap to state and closes the gap before such a save ever lands.
