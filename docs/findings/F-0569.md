## Problem

`V15e`'s hot-reload cache-eviction step evicts the static-resolution per-pass parse cache (`V15a`) only for the changed file and its transitive `.warp` import-edge parents (`V15c`). But that cache is populated over a larger transitive graph: `invocation.md` §Static resolution walks `invoke` literals and `.loom` entries in `tools:`, and pins the hot-reload contract as re-running the static-resolution pass for "every parent that has the callee in its reachable graph." A `.loom` parent that reaches a changed callee only through an `invoke` literal or a `tools:` entry is not a `.warp` importer, so `V15e`'s walk never visits it and it is served stale cross-callee checks (arg-type, `invoke<T>` return-type, cycle) against the pre-change callee. As written `V15e` under-evicts relative to `invocation.md`, or the premise that makes `.warp`-only eviction sufficient is left unstated.

## Solution approach

Broaden the eviction set in `V15e`'s Adds/Tests/Ships-when from "transitive `.warp` importers" to also include every parent reaching the changed file through `invoke` literals and `.loom` `tools:` entries — the same transitive graph `V15a`'s static-resolution walk builds — to satisfy `invocation.md`'s "every parent in the reachable graph" mandate. Add a `V15e-T` case covering a changed `.loom` callee whose signature change forces a re-check at an `invoke` / `tools:`-edge parent. Update the `coverage-matrix.md` `cka-44` parenthetical so its scope description matches the broadened walk. Alternatively, if `.warp`-only eviction is in fact sufficient, state that load-bearing premise explicitly in `V15e` rather than broadening.
