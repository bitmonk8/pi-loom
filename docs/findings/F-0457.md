## Finding

`coverage-matrix.md` §"Code-keyed obligation areas" defines the closing-leaf recogniser's verdict in terms of resolution against the **"real plan leaf-ID universe"**: a closing-leaf token equal to the literal bare `<new>` is the uncovered-obligation placeholder; a token that resolves to *an existing plan leaf* satisfies the closing-leaf condition; and any token resolving to neither — a typo'd or non-existent ID such as `V99z` — reddens the gate as a defect. The whole three-way classification therefore turns on what counts as "an existing plan leaf," yet neither `coverage-matrix.md` nor `conventions.md` states how that set is enumerated.

`H5e`'s Adds re-cites the same predicate ("the recogniser resolves each closing-leaf token against the real plan leaf-ID universe per `coverage-matrix.md`'s *Code-keyed obligation areas* placeholder convention") and defers the definition back to `coverage-matrix.md` — which never supplies it, leaving the reference circular.

Two mechanically-distinct enumeration sources are equally plausible and produce different membership: (1) a filename glob of `docs/plan_topics/<id>-*.md` (a leaf exists iff its file exists), versus (2) the set of leaf IDs linked from `plan.md`. The two diverge for a leaf file that exists but is unlinked from `plan.md`, or for a `plan.md` link carrying a stale/renamed ID. Under those conditions the same closing-leaf token classifies as a real leaf under one source and as a `V99z`-style defect under the other.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `f3bb326` ("pi-loom plan: resolve \"<new> placeholder recognition rule pinned to a literal token\"", 2026-06-18)
**History:** `git log -G 'real plan leaf-ID universe'` over the plan corpus returns exactly one plan-side commit, `f3bb326`. That commit introduced the `<new>`-placeholder recognition rule and, with it, the phrase "resolves each closing-leaf token against the real plan leaf-ID universe" into both `coverage-matrix.md` §"Code-keyed obligation areas" and `H5e`'s Adds. The recogniser predicate was added without ever defining the enumeration source for the universe it resolves against, so the gap is present since the predicate's inception rather than introduced by a later edit.

## Solution Space

**Shape:** single

### Recommendation

In `coverage-matrix.md` §"Code-keyed obligation areas (no numbered REQ-IDs)", adjacent to the first use of "real plan leaf-ID universe", add a sentence pinning the single authoritative enumeration source: the universe is the set of leaf IDs derived from `docs/plan_topics/*.md` leaf filenames (each leaf file contributes its `<id>` prefix), read from the filesystem — **not** from `plan.md`'s link set. This makes the recogniser independent of `plan.md` link freshness and aligns with the existing closing-gate machinery, which already reads plan files directly.

`H5e`'s existing back-reference to this convention then resolves to a defined source and needs no separate definition; leave its cross-reference as-is.

Edge cases the implementer must get right:
- `<id>` extraction must preserve the `-T` tests-task suffix so a tests-task file (`M-T-minimal-slash-command.md`, `V9c-T-conversation-drive.md`) contributes `M-T` / `V9c-T`, not `M` / `V9c` — a naive split on the first hyphen mis-derives these.
- Non-leaf files under `plan_topics/` (`conventions.md`, `coverage-matrix.md`, `leaf-template.md`, `real-host-smoke-gate.md`) carry no `<id>-*` leaf-ID shape and must be excluded from the universe.

## Consequence

**Severity:** correctness

The `H5e` recogniser cannot deterministically classify a closing-leaf token as a real leaf, a benign `<new>` placeholder, or a `V99z`-style defect without knowing how the leaf-ID universe is enumerated. Two reasonable implementers (filename glob vs `plan.md`-link set) produce divergent live-corpus gate verdicts for an unlinked-but-present leaf or a stale link: one reddens a valid closing-leaf token as a spurious defect, the other passes a stale token as a real closing leaf, masking an uncovered obligation.
