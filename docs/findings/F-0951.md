## Finding

The fourth bullet in the **Tests** list of both `V11c` and `V11c-T` reads:
"The envelope is a relaxed copy (`maxLength:500` as a model budget, not a user
cap); the bypass path skips the LLM call." Unlike the three sibling bullets
(`BNDR-1`, `BNDR-2`, `BNDR-3`), it carries no REQ-ID label and is phrased as a
declarative fact rather than a checkable assertion. It states no input, no
expected output, and no observable property a test would exercise.

Both facts it asserts already appear in `V11c`'s **Adds** line ("the binder
bypass path and the dynamic envelope schema … the relaxed copy with
`maxLength:500`"). So as written the bullet contributes no test specification —
it re-states design content in the one section that is supposed to enumerate
red tests, which misleads a reader into treating it as a covered property when
nothing verifies it.

The two underlying properties are nonetheless real, spec-pinned, and currently
uncovered. `docs/spec_topics/binder/binder-bypass-and-envelope.md` pins the
`message` field's `maxLength:500` budget in the envelope schema, and pins two
binder-bypass cases (no-params and single-string) that skip the binder call
entirely. Neither `BNDR-1` (three-arm discriminator), `BNDR-2`
(`ambiguous.candidates` retention), nor `BNDR-3` (distinct template prefixes)
exercises the `maxLength:500` budget or the zero-call bypass behaviour.

## Solution Space

**Shape:** multiple

### Option A — Strike the bullet

**Approach.** Remove the fourth Tests bullet from both `V11c` and `V11c-T`,
leaving `BNDR-1`/`BNDR-2`/`BNDR-3`.

**Plan edits.** Delete "The envelope is a relaxed copy (`maxLength:500` as a
model budget, not a user cap); the bypass path skips the LLM call." from the
Tests list of `docs/plan_topics/V11c-bypass-envelope.md` and
`docs/plan_topics/V11c-T-bypass-envelope.md`.

**Spec edits.** None.

**Pros.** Minimal diff; the Tests list becomes exclusively labelled assertions;
the facts survive in `V11c` Adds, so nothing about the design is lost.

**Cons.** The `maxLength:500` envelope budget and the zero-call bypass behaviour
remain untested even though both are spec obligations.

**Risks.** Silently drops two genuinely test-worthy spec properties from the
leaf's red-test set.

### Option B — Convert to explicit assertions

**Approach.** Replace the declarative bullet with labelled assertion(s) that
exercise the two real, currently-uncovered properties.

**Plan edits.** In the Tests list of both leaves, replace the fourth bullet with
assertions to the effect of: the envelope's failure-arm `message` field carries
`maxLength:500`; and the binder-bypass path (no-params and single-string cases)
issues zero binder/LLM calls. Concrete literal text, e.g.
"`BNDR-4`: the envelope's `message` field carries `maxLength:500`." and
"`BNDR-5`: the no-params and single-string bypass paths issue zero binder
calls." (REQ-ID choice is the fixer's; do not coin a spec REQ-ID that the spec
does not register — use plan-governed labels if no spec REQ-ID exists for these).

**Spec edits.** None — the assertions are derived from existing pins in
`docs/spec_topics/binder/binder-bypass-and-envelope.md`.

**Pros.** Closes coverage on two spec-pinned properties; the Tests list stays
all-assertions; the design facts remain in Adds.

**Cons.** Larger diff than Option A; introduces new test labels the fixer must
keep distinct from spec-registered REQ-IDs.

**Risks.** Mislabelling the new assertions with a spec-style REQ-ID prefix that
the spec does not register (cf. the `DISC-6b`/`DISC-6c` finding); use only
labels the plan owns.

### Recommendation

Take **Option B**. The `maxLength:500` envelope budget and the zero-call bypass
behaviour are both pinned in `docs/spec_topics/binder/binder-bypass-and-envelope.md`
and are not exercised by `BNDR-1`/`BNDR-2`/`BNDR-3`, so striking the bullet
(Option A) would silently retire two real, otherwise-uncovered obligations.
Watch that any new test label is plan-governed and not confusable with a
spec-registered REQ-ID, and keep the new assertions identical in `V11c` and
`V11c-T` since the leaves mirror each other.

## Consequence

**Severity:** advisory

The bullet reads as a tested property but specifies no test, so a reader trusts
coverage that does not exist; the `maxLength:500` envelope budget and the
zero-call bypass behaviour — both spec-pinned — can ship unverified. No leaf is
blocked, and a careful implementer can still produce a working `V11c`, so the
gap is guidance-level rather than blocking.
