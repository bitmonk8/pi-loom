## Finding

PIC-55 (`registration-steps.md#pic-55`, under step 5) is an executable runtime obligation: on a chokidar `error` route firing or a synchronous throw out of the watcher, the runtime must either surface a transient `ctx.ui.notify` toast (continues-delivering) or — when one or more watched roots stop delivering — leave the watcher torn down, emit a single persistent `loom/runtime/watcher-terminated` `loom-system-note` prompting `/reload`, keep the `LoomRegistry` live and dispatchable, and write **no** drain-state tag. No plan leaf implements either arm. `grep` over `docs/plan_topics/` returns zero hits for `PIC-55`, `watcher-terminated`, or the recovery posture; `V8e` (the only watcher-touching leaf) is scoped to the `FileWatcher.watch` seam shape (PIC-14/16 — `Unsubscribe` and the three change kinds) and explicitly does not own the adapter error-route recovery posture.

This is simultaneously a coverage gap and a code-registry parity gap. `loom/runtime/watcher-terminated` is a registered runtime diagnostic (severity E, `code-registry-runtime.md`) but no leaf asserts it, so the DIAG-2 arm ("a registry code with no asserting test fails the gate", owned by `V7b`/`H5a`) has nothing to satisfy it. PIC-55 is also a numbered REQ-ID with no `coverage-matrix.md` row, so `H5a`'s live-corpus unmapped-executable-REQ-ID reconciliation has no mapping for it.

The defect is purely plan-internal to resolve: the spec contract for PIC-55 exists and is cited; what is missing is a plan leaf that closes it, the coverage-matrix row that maps it, and the asserting test for the `watcher-terminated` code.

## Solution Space

**Shape:** single

### Recommendation

Add a new vertical-slice leaf (and its paired `<new>-T`) that closes PIC-55 and asserts the `loom/runtime/watcher-terminated` code, citing `registration-steps.md#pic-55`, `code-registry-runtime.md` (the `watcher-terminated` row), and `host-interfaces-services.md` (PIC-14 `watch`) in its `Spec.` field, and depending on `V8e` (the `FileWatcher` seam), `V7b` (registry/DIAG-2 gate), and `V7d` (`loom-system-note` channel).

The leaf's `Tests.` cover the stopped-delivering terminal arm — the adapter leaves the watcher torn down, the runtime emits exactly one persistent `loom/runtime/watcher-terminated` `loom-system-note` prompting `/reload`, the `LoomRegistry` stays live and dispatchable through arm (a), and **no** drain-state tag is written — and the continues-delivering arm (transient informational toast, no further recovery action). One test must assert the `watcher-terminated` code with its expected message string sourced from the registry (DIAG-4), so the DIAG-2 parity arm is satisfied.

Register the PIC-55 row in `coverage-matrix.md`'s Numbered REQ-IDs table pointing at the new leaf's backtick-delimited ID, and link the new leaf into the appropriate Vertical slices section of `plan.md`. Because the leaf's citing test closes a coverage-matrix-mapped numbered REQ-ID, add the new leaf to `H5b`'s `Deps.` per How-to-use step 2 (*Transitive-completeness plan-maintenance*).

Edge cases the implementer must watch: the continues-delivering transient-toast arm overlaps the chokidar-throw surface of the separate transient-toasts coverage finding (see Related Findings) — coordinate so the toast arm is asserted once, not twice; and the no-drain-state-tag MUST is load-bearing for the `session_shutdown` handler-entry short-circuit, so its assertion must be explicit rather than implied.

## Consequence

**Severity:** blocking

PIC-55's watcher-recovery behaviour ships unimplemented: a dead watcher silently strands hot-reload with no operator prompt and no persistent note. At the loom 1.0 release-gate activation (`H6a`), `H5a` hard-fails twice over this gap — once on PIC-55 as an unmapped executable REQ-ID, once on `watcher-terminated` as a registered code with no asserting test — so the release gate cannot be flipped green until a closing leaf exists.
