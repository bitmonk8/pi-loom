## Finding

`docs/spec_topics/schema-subset.md` Canonical schema hash, step 2 (Canonical form) carries a developer-rationale HTML comment as a bullet-position element of the normative recipe list:

```markdown
   - Рђд(its `integer` and `number` value-rendering rules together with the normative
     reference-renderings table, which fix the otherwise-ambiguous cases Рђћ e.g. `1e21`
     serialises as `1000000000000000000000`, `42.0` as `42`, `-0` as `0`).
     Only that rendering algorithm is borrowed: the binder's 120-code-point line cap
     and `(default)` suffix are echo-formatter concerns and do not apply to the
     canonical-hash recipe;
   <!-- `default` and the integer-valued structural keywords (`minLength`, `maxItems`, Рђд)
        are deliberately absent here: `default` is a runtime-filled surface-syntax feature
        the binder never lowers, and the structural keywords are rejected at parse time
        per the top-of-file enumeration, so no such keyword reaches the lowered fragment. -->
   - strings escaped per RFC 8259 minimal-escape rulesРђд
```

Rendered Markdown elides the comment entirely, so the explanation of *why* the recipe no longer enumerates `default`, `minLength`, `maxItems`, Рђд is invisible to every spec reader; only someone reading the raw source file sees it. The carrying content is genuine rationale rather than a TODO or author note, but its placement makes it functionally absent from the spec.

## Issue introduction

**Verdict:** targeted-revision-introduced
**Introducing commits:** `111e533` (2026-06-07) Рђћ "pi-loom spec: resolve \"Canonical hash recipe cites keywords the lowered subset never emits\""
**History:** `git log -S` for the comment text returns a single commit. The commit's diff against `docs/spec_topics/schema-subset.md` shows the prior recipe bullet was edited in place to drop `default` from the numeric-literal enumeration (and to drop the trailing structural-keywords sentence), and a fresh `<!-- Рђд -->` block was inserted immediately after that bullet to record the rationale for the deletions. The comment has been carried unchanged since, surviving the `81bf52a` directory move into `docs/spec_topics/`. The defect therefore entered as part of a corrective revision that intended to add rationale but used the wrong inline-comment syntax for a rendered spec.

## Solution Space

**Shape:** single

### Recommendation

Convert the rationale to rendered Markdown so readers actually see it. Replace the `<!-- Рђд -->` line in step 2 with a sub-bullet under the same numeric-literal-rendering bullet, e.g.:

```markdown
   - Рђдdo not apply to the canonical-hash recipe. (`default` and the integer-valued
     structural keywords Рђћ `minLength`, `maxItems`, Рђд Рђћ are deliberately absent from
     this enumeration: `default` is a surface-syntax feature the binder never lowers,
     and the structural keywords are rejected at parse time per the
     [top-of-file enumeration](#Рђд), so neither reaches the lowered fragment.);
   - strings escaped per RFC 8259 minimal-escape rulesРђд
```

Either a parenthetical tail on the same bullet (as above) or an indented sub-bullet under it is acceptable; what matters is that the explanation renders. Keep the wording verbatim from the existing comment so the rationale `111e533` paid for is preserved. Do not delete it outright Рђћ the recipe section is the natural home for an explanation of which lowered-fragment keywords feed the hash and which do not, and a future reader landing on this section to debug a slug mismatch is exactly the audience the explanation was written for.

Edge case to watch: ensure the converted prose does not re-introduce a normative-sounding enumeration of `default` / `minLength` / `maxItems` that re-creates the very defect `111e533` resolved (the bullet must still say these keywords are *absent* from the lowered fragment, not that they are rendered some particular way).

## Consequence

**Severity:** cosmetic

No implementer behaviour changes Рђћ the surrounding bullets already prescribe the recipe completely. The defect is purely that the rationale paid for in the previous targeted revision (explaining why `default` / structural keywords were dropped from the enumeration) is invisible in the rendered spec.
