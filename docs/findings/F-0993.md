## Finding

The grammar appendix's `///` placement production reads:

```
DocComment   ::= ("///" RestOfLine "\n")+
```

Every `DocComment` alternative requires a literal `\n` after each `RestOfLine`. The companion `RestOfLine` definition in [`lexical.md` — `RestOfLine` terminal](../../docs/spec_topics/lexical.md#rest-of-line) explicitly captures only characters that contain no `\n`, so the line and its terminator are split between the two productions, with `\n` carried by `DocComment` itself.

Nowhere in the corpus is the EOF case carved out. The encoding/normalisation rules in `lexical.md` (UTF-8 decode, `\r\n` → `\n`, bare `\r` → `\n`) operate on what is already in the byte stream — they never *synthesise* a trailing `\n` at EOF. A `.loom` or `.warp` file whose final source line is a bare `/// last fragment` with no terminating newline therefore satisfies neither `("///" RestOfLine "\n")+` (no `\n` after the last `RestOfLine`) nor any other production for `///` content. The descriptions topic page (`descriptions.md`) describes joining and dedent semantics, but says nothing about whether a final unterminated `///` line is admitted, rejected, or implicitly terminated.

Two reasonable implementers will diverge: one treats EOF as equivalent to `\n` for `DocComment` termination (accepting the file and emitting the description); another follows the grammar literally, leaving the trailing `///` unmatched and surfacing `loom/parse/doc-comment-misplaced`, an `unterminated`-style error, or a generic parse failure. Authors and gate-writers cannot decide which is conformant.

The defect is therefore inherent to the appendix as it was first written; no later commit either introduced or repaired the EOF gap.

## Solution Space

**Shape:** single

### Recommendation

Anchor the EOF carve-out on the `lexical.md` normalisation step rather than the grammar production, so it composes uniformly with every spec rule that mentions `\n` (statement separators, query templates, doc-comment joining, etc.) and the grammar appendix is left as the single normative source for `DocComment`.

Concretely: extend the existing `**Newline normalisation.**` bullet in `docs/spec_topics/lexical.md` with a sentence stating that, after the `\r\n`/`\r` normalisation step, the lexer treats the EOF position as equivalent to a single `\n` for the purpose of every production that names `\n`, regardless of whether the source bytes actually end with a newline. Add an explicit clause that this rule does **not** introduce any new line of content (the synthesised newline terminates the final line but does not start a new empty one), so trailing-blank-line dedent and `///` joining behave identically on `…\n` and `…` inputs.

In `docs/spec_topics/grammar.md`, leave the `DocComment` production unchanged but add a one-sentence prose note immediately under the production block: "The terminal `\n` is supplied by the lexical normalisation step's EOF-equivalence rule (see [Lexical Structure — Newline normalisation](./lexical.md)) when the file's final `///` line is not followed by an explicit newline; a final unterminated `///` line therefore tokenises identically to the same file with a trailing newline." This keeps the grammar's `+` quantifier honest and gives implementers one obvious place to look.

Implementer-relevant edge cases the fix must spell out:

- A file consisting solely of `///fragment` (no trailing newline) is a valid single-line `DocComment` token, which then surfaces `loom/parse/doc-comment-misplaced` because no eligible production follows it — not an `unterminated-doc-comment` or generic parse failure.
- A file ending in `\n///fragment` (one prior newline, then an unterminated `///` line) is a two-line `DocComment` and is treated identically to the same file with a trailing newline.
- The rule must not retroactively terminate other productions that consume newlines explicitly: e.g. an unterminated string literal at EOF must remain `loom/parse/unterminated-string` (the existing rule wins because a regular string literal does not consume `\n` as part of its body). Adding the carve-out at the lexical-normalisation layer cleanly preserves that ordering because the synthesised `\n` is delivered to the lexer after string-scanning has already faulted on the missing closing quote.

## Consequence

**Severity:** correctness

A `.loom` file authored on a platform or editor that omits the final newline (a common configuration on Windows and in some build-pipeline outputs) may load on one implementation and fail on another, with no spec basis for either decision. The same file could lower its terminal `///` into a JSON Schema `description:` under one runtime and surface a parse diagnostic under another. The blast radius is narrow per occurrence but compounds with the existing `loom 1.x source-language equivalence` goal (GOV-15), which requires identical diagnostic-code sequences across conforming implementations.
