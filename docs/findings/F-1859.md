## Finding

The *Wall-clock budget* clause in `audit-wire-and-canary.md` requires the audit to "enforce a wall-clock budget on its own execution (the implementation pins the budget value, e.g. 60 s)" and, on expiry, to emit one infrastructure-failure record under a stable `<symptom>` token and exit non-zero. The "e.g. 60 s" is illustrative and non-binding; the spec pins only that *some* budget exists and that the implementation chooses its value. No spec-side floor or ceiling constrains that value — any figure from 1 ms to 24 h satisfies the MUST.

The behavioural contract on expiry (record emission + non-zero `npm test` exit) is fully specified and testable once a budget value is known. The gap is the value itself: a conformance test working from the spec alone cannot assert the budget falls within any reasonable range, and a test that wants to drive the expiry branch must already know the implementation's pinned constant to construct a run that exceeds it. A degenerate small value would make the audit time out on every invocation (caught indirectly when the canary/green-path emission never completes), and a degenerate large value would render the safety valve inert, relying entirely on the `npm test`-side timeout the *Pre-emission termination carve-out* mentions. Neither failure mode is detectable from the spec text.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `e5cd514` (2026-05-09, "pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\"")
**History:** The *Wall-clock budget* clause entered the corpus with the audit's first authoring in `e5cd514`. That first version already delegated the budget value to another owner ("the H1 leaf pins the budget value, e.g. 60 s") with the figure illustrative and no spec-side floor or ceiling. A later edit (`9e53613`, "drop H1 / plan_topics / plan-leaf-ID references from spec corpus") reworded the owner from "the H1 leaf" to "the implementation" but left the missing floor/ceiling unchanged. The defect has been present since the clause's inception.

## Solution Space

**Shape:** multiple

### Option A — Pin a minimum budget floor in PIC

- **Approach:** Keep the MUST that a budget exists and is implementation-pinned, and add a spec-side floor, e.g. "the implementation MUST pin the budget to at least 30 s." Optionally pin a ceiling too.
- **Spec edits:** Amend the *Wall-clock budget* clause to state the minimum (and any maximum), replacing the bare "e.g. 60 s" illustration with the pinned bound.
- **Pros:** Preserves the fail-closed intent at full strength; gives a reviewer and conformance test a concrete numeric boundary to check the pinned constant against; lightest-touch change.
- **Cons:** A spec-level conformance test still cannot know the exact pinned value, only that it clears the floor; the floor is a judgement call with no derivation.
- **Risks:** A floor chosen too high could mask a genuinely slow audit on constrained CI hardware.

### Option B — Demote the wall-clock budget to a non-binding recommendation

- **Approach:** Change the budget MUST to a SHOULD, relying on the existing `npm test`-side timeout (named in the *Pre-emission termination carve-out*) plus the pre-emission termination path as the fail-closed backstop for a hung audit.
- **Spec edits:** Reword the *Wall-clock budget* clause so the in-audit budget is recommended rather than required; keep the carve-out's red-exit-satisfies-fail-closed statement as the binding guarantee.
- **Pros:** Removes the untestable obligation entirely; the genuine fail-closed property (red `npm test` on a hung run) is already guaranteed by the harness timeout.
- **Cons:** Loses the clean in-audit expiry record (`<symptom>` token + infrastructure-failure record) for a hung run; a contributor then gets a bare timeout kill rather than a routable record.
- **Risks:** Weakens the diagnostic surface for hung-audit triage; the carve-out's recovery path (re-run with diagnostic-sink redirection) becomes the only signal.

### Recommendation

Take Option A: pin a minimum floor (a modest value such as 30 s) in the *Wall-clock budget* clause. It keeps the in-audit expiry record and full fail-closed strength while giving review and conformance a concrete lower bound, and is a smaller change than reworking the fail-closed guarantee. Watch that the floor is not set so high it masks a slow-but-functional audit on constrained CI.

## Consequence

**Severity:** advisory

Two reasonable implementers can pin wildly different budgets (1 ms vs. 24 h) and both pass; the spec offers no range a reviewer or conformance test can check the pinned constant against, and the expiry branch cannot be exercised from the spec alone. A working audit is still producible — the budget is a safety valve on hung runs, backstopped by the `npm test`-side timeout — so this degrades testability and review traction rather than blocking implementation.
