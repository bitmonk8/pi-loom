## Finding

`H7a`'s **Adds** and **Tests** describe a terminal integration pipeline that runs `typed query → tool loop → code-tool invoke → schema lowering/validation → binder → cancellation` in a single run. The plan distinguishes two binder-cancellation behaviours: `V17a` owns the pre-binder checkpoint abort (the `binder-call` `CheckpointKind` firing immediately *before* the binder's LLM call is issued), while `V11j` owns in-flight binder-call cancellation forwarding (an abort landing *during* the binder's provider call, on the initial attempt or a budgeted retry, surfacing the cancelled-binder system note). `H7a`'s **Deps** name `V17a` but not `V11j`, and `V11j` is not transitively reachable from any other Deps entry — the only leaf depending on `V11j` is `H5b`, which is not in `H7a`'s Deps. The golden-transcript / golden-diagnostics / permitted-code derivation enumeration in `H7a`'s **Adds** likewise lists `V17a` and omits `V11j`.

The terminal "cancellation" stage is therefore unqualified: it does not say whether it drives an in-flight binder abort (which would need `V11j`) or only the pre-binder checkpoint abort (which `V17a` already supplies). Under the plan's dep-driven build order ("pick the next leaf whose Deps are satisfied"), a scheduler can place `H7a` before `V11j` because nothing forces `V11j` to precede it. If the integration fixture genuinely exercises in-flight binder-call cancellation, the gate is missing a dependency and its golden artifacts cannot derive the cancelled-binder behaviour from any Deps slice. If it does not, the prose still misleads the next reader.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e7e51cc` ("pi-loom plan: resolve \"Plan has no terminal end-to-end integration-acceptance leaf\"", 2026-06-10) authored `H7a` with the `binder → cancellation` pipeline prose and a Deps list citing `V17a` but not `V11j`; at that commit `V11j` did not yet exist (in-flight binder-call cancellation forwarding was still bundled into `V11f`). `66b3815` ("pi-loom plan: resolve \"V11f bundles binder retry budget with in-flight cancellation forwarding\"", 2026-06-16) split `V11j` out as the distinct owner of in-flight binder-call cancellation forwarding but did not retarget `H7a`'s Deps or qualify its terminal cancellation stage.
**History:** `git log --follow` / `git log -S 'binder → cancellation'` / `git show` over `docs/plan_topics/H7a-integration-acceptance.md` and `docs/plan_topics/V11j-binder-call-cancellation.md` confirm the cancellation stage is present-since-`H7a`-inception (`e7e51cc`) while `V11j` was created later (`66b3815`, with `e7e51cc` an ancestor); the ambiguity became a missing-dependency / unclear-scope gap when the `V11f → V11j` split landed without propagating into `H7a`.

## Solution Space

**Shape:** multiple

### Option A — Declare the in-flight dependency

- **Approach:** Treat the terminal "binder → cancellation" stage as exercising in-flight binder-call cancellation forwarding.
- **Plan edits:** Add `V11j` to `H7a`'s **Deps**; add `V11j` to the golden-derivation Deps enumeration in **Adds** and to the permitted-code-list union (the parenthetical Deps lists in the third Tests bullet); fold `V11j`'s cancelled-binder note (`loom /<name>: argument binding cancelled`) into the golden transcript / golden diagnostics derivation where the fixture path emits it.
- **Spec edits:** None.
- **Pros:** The integration gate genuinely covers the in-flight binder-cancellation composition; the ordering gap closes because `H7a` becomes dep-ordered after `V11j`.
- **Cons:** Broadens `H7a`'s derivation surface and golden artifacts.
- **Risks:** Enlarges the `H7a` diff and the next pass's critique surface.

### Option B — Clarify the stage is V17a-scoped

- **Approach:** Treat the terminal cancellation as the pre-binder checkpoint abort `V17a` already owns; the gate does not drive an in-flight binder abort.
- **Plan edits:** In `H7a`'s **Adds** pipeline prose and the **Tests** / **Ships when** wording, qualify the terminal "cancellation" stage as the `V17a` pre-binder checkpoint (pre-call), distinct from `V11j`'s in-flight forwarding; leave **Deps** as-is.
- **Spec edits:** None.
- **Pros:** Minimal diff; aligns the prose with the existing golden-derivation enumeration, which already omits `V11j`; lands on a stable baseline.
- **Cons:** The integration gate does not cover the in-flight binder-cancellation composition (it remains covered in isolation by `V11j`).
- **Risks:** Low.

### Recommendation

Adopt Option B unless the terminal fixture is intended to drive an abort *during* the binder's in-flight provider call: the existing golden-derivation Deps enumeration already excludes `V11j`, so qualifying the stage as the `V17a` pre-binder checkpoint aligns the prose with the artifacts the gate actually derives. If the integration fixture must exercise in-flight binder cancellation, adopt Option A and add `V11j` to both `H7a`'s **Deps** and the golden-derivation / permitted-code enumerations so the cancelled-binder behaviour is sourced from `V11j`'s established expected output. The spec is read-only for this fix — do not edit spec topics. `V11j` is the only in-flight binder-call cancellation-forwarding leaf; cite it as-is and do not introduce a new ID.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one reads "binder → cancellation" as exercising in-flight binder-call cancellation, wires it, and finds `V11j` missing from Deps (so `H7a` is schedulable before the leaf that supplies the behaviour, and its golden artifacts cannot derive the cancelled-binder note from any Deps slice); the other treats the stage as `V17a`'s pre-binder checkpoint abort and ships a correct but unstated-scope gate. The divergence affects what the terminal integration gate actually covers.
