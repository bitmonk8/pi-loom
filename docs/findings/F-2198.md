## Finding

The *Classifier input surface* note in `provider-error-mapping.md` reads the HTTP-status class "from `ProviderResponse.status`, delivered through `@earendil-works/pi-ai`'s `StreamOptions.onResponse` callback (`ProviderResponse = { status: number; headers: Record<string, string> }`), declared at `dist/types.d.ts`". The parenthetical writes the type as a **closed equation** тАФ `ProviderResponse = { тАж }` тАФ restating the complete field list of a type loom neither owns nor declares ("loom redefines neither `StreamOptions.onResponse` nor `ProviderResponse`").

`ProviderResponse` is an external `@earendil-works/pi-ai` surface. loom consumes only `.status` as its HTTP-class signal and incidentally joins `.headers`. Pinning the *whole* shape with `= {тАж}` couples the spec to a structure it does not control: if pi-ai widens `ProviderResponse` (adds a field, refines `headers`), the equation silently becomes a false restatement of the SDK's declaration, and the spec carries a second source of truth competing with `dist/types.d.ts`. The note already names the authoritative declaration site, so the closed shape adds coupling without adding information loom depends on.

The correct form for an externally-owned type loom reads is a read-basis reference тАФ name the fields loom actually consumes plus the declaration site тАФ not a complete-shape equation.

## Solution Space

**Shape:** single

### Recommendation

Demote the closed-shape parenthetical to a read-basis reference. Replace `(`ProviderResponse = { status: number; headers: Record<string, string> }`)` with a phrasing that names only the fields loom consumes plus the declaration site, e.g.: "loom reads `ProviderResponse.status` (and joins `.headers`); `ProviderResponse` is declared at `@earendil-works/pi-ai`'s `dist/types.d.ts`". Drop the `= {тАж}` complete-shape form so the spec cites the SDK's declaration as the single source of truth instead of mirroring it.

Edge case: if the inline field list is retained for reader convenience, it MUST be tagged `**Non-normative.**` so it is unambiguously a courtesy snapshot of the upstream declaration and not a loom-owned shape constraint. The surrounding sentence already states loom "redefines neither `StreamOptions.onResponse` nor `ProviderResponse`", so no normative content is lost by dropping the equation.

## Consequence

**Severity:** advisory

The closed `= {тАж}` equation duplicates an SDK declaration loom does not own; an upstream `ProviderResponse` widening makes the spec's restatement stale without any build-time signal, and a reader may treat `headers` as a shape loom enforces rather than a field it incidentally joins. Behaviour does not diverge тАФ loom only reads `.status` тАФ so an implementer still builds a working classifier, but the spec carries an avoidable over-coupling to an external surface.
