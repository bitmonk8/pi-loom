## Finding

`active-invocation-registry.md` pins three layers of internal detail as normative
text: the container type (`Set<{ … }>`), the exact **five-member** entry shape
(`loomAbort`, `disposeBarrier`, `shutdownReason`, `loom`, `invocationId`), and
the **lowerCamelCase** spellings of those members. The section then
contradicts itself in the same paragraph: it states the registry name is
internal, asserts that tests assert only on observable side effects (entry
counts via probe seams, ordered `loomAbort.abort()` calls, `disposeBarrier`
settlement), and explicitly carves out the snake_case wire-side names
(`details.event.invocation_id`, `RuntimeEvent.invocation_id`,
`<invocation-id>` placeholder) as the only externally observable surface.
Despite that, the `disposeBarrier` resolver storage clause re-pins the
five-field shape as load-bearing for "observers of the entry (tests, audit
fixtures)", elevating an admittedly internal data structure into the
audit-fixture surface.

The result is a prescription that two conforming runtimes can fail
gratuitously: a runtime that stores the entry as a `Map`-backed record, a
class instance with private fields, or any object with differently-spelled
internal members (`abortController`, `barrier`, `reasonStamp`) is non-conformant
on the literal text yet emits byte-identical wire payloads, byte-identical
diagnostics, and byte-identical observable cancellation behaviour. Every
sibling DI seam in the same page (`PIC-12` `Clock`, `PIC-13` `FileSystem`,
`PIC-14` `FileWatcher`, `PIC-16` `TokenEstimator`, `PIC-20` `IdSource`)
already uses the GOV-18 arm (a) carve-out — "the interface members below are
a non-normative reference; per GOV-18 arm (a) the exact member signatures of
this internal DI seam are not themselves binding, while the usage constraints
stated on each member remain normative" — to demote exactly this kind of
illustrative shape. This section is the outlier.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `5e8b499` (2026-05-06) — *pi-loom spec: resolve "Hot-reload teardown contract for the loom extension instance is unspecified"*: introduced the `ActiveInvocationRegistry` with the literal `Set<{ loomAbort: AbortController; disposeBarrier: Promise<void> }>` type and the "The registry name is internal" bullet, but with only two pinned members the over-prescription was minor and the "internal" disclaimer was congruent with the pinned shape.
- `e4a3882` (2026-05-07) — *pi-loom spec: resolve "Session model: rewrite of concurrent-invocations and ActiveInvocationRegistry framing"*: expanded the "internal" bullet to enumerate observable side effects ("entry counts via probe seams, ordered `loomAbort.abort()` calls, `disposeBarrier` settlement"), which clarified the test surface but left the typed entry shape pinned.
- `75a9626` (2026-05-09) — *pi-loom spec: resolve "SessionShutdownEvent.reason closed set has no build-time pin against the SDK type"*: added `shutdownReason: string | undefined` to the typed entry, the first commit to grow the pinned shape past the original two members.
- `63bddcd` (2026-05-22) — *pi-loom spec: resolve cluster "T19a invocation_id correlation key" (+2 members)*: added `loom: string` and `invocationId: string` (reaching the now-pinned five-member shape) and introduced the explicit "registry-side identifier is spelled `invocationId` in lowerCamelCase (matching this section's other runtime-state members `loomAbort`, `disposeBarrier`, `shutdownReason`)" sentence — the first commit that explicitly elevated the lowerCamelCase spellings to contract status.

**History:** `git log -S "Set<{ loomAbort: AbortController"` against `docs/` (and its `spec_topics/` predecessor before the `31ff060` move) shows the four commits above, in that order. The defect did not enter at any single point; the first commit pinned a small shape that the "registry name is internal" disclaimer covered comfortably, and each later commit (driven by an unrelated finding) added one more member without re-examining whether the pinned-shape framing still squared with the disclaimer. By `63bddcd` the pinned surface had grown to a five-member literal with explicit lowerCamelCase spellings while the disclaimer text was unchanged, producing the present internal-vs-contract contradiction. None of the four resolving findings flagged the over-prescription themselves; each treated the shape pin as load-bearing for its own concern (teardown sequencing, snapshot brand, correlation-id propagation).

## Solution Space

**Shape:** single

### Recommendation

Apply the same GOV-18 arm (a) carve-out the sibling DI seams on
`host-interfaces-services.md` already use. Concretely:

1. Replace the typed `Set<{ … }>` declaration in the intro paragraph with a
   prose description: "an extension-instance-scoped collection of
   per-invocation entries, one per in-flight loom invocation". Move the
   illustrative type (`Set<{ loomAbort: AbortController; disposeBarrier:
   Promise<void>; shutdownReason: string | undefined; loom: string;
   invocationId: string }>`) into a **non-normative reference** sentence
   that opens with the same boilerplate as PIC-12/13/14/16/20: *"The entry
   shape below is a non-normative reference illustrating one conforming
   internal decomposition; per [GOV-18 arm (a)] the exact member set and
   spellings of this internal data structure are not themselves binding."*
2. Keep as normative the per-field **behavioural** rules already in the
   paragraph: that the registry carries a per-entry abort controller (used
   by sub-step 2 of the `session_shutdown` handler), a per-entry barrier
   promise the per-invocation `finally` settles (awaited by sub-step 3), a
   handler-stamped shutdown-reason string the per-invocation `finally`
   reads, a canonical loom-name field sourced from the catalog at insertion
   time, and an `IdSource`-minted invocation id read by downstream emission
   sites. Each of those rules can be stated without naming the member or
   pinning lowerCamelCase.
3. Pin the **wire-side** names (`details.event.invocation_id`,
   `RuntimeEvent.invocation_id`, the `<invocation-id>` placeholder,
   `details.event.loom`, `details.event.reason`) and the **observable
   side effects** (insertion-order iteration, abort dispatch over every
   entry, barrier settlement) as the externally-binding contract — these
   are already separated out in the existing prose and stay normative.
4. Edit the `disposeBarrier` resolver storage bullet to drop "observers of
   the entry (tests, audit fixtures) see exactly the five members the entry
   literal pins"; replace with a no-leakage rule keyed on the resolver
   itself ("the resolver is closure-scoped and not exposed through the
   registry-entry surface").

Edge cases for the implementer:

- The Diagnostics page's `details` construction prose (`details.event` is
  built with the static property name `invocation_id` assigned from
  `entry.invocationId`) is the one place that names the registry-side
  member. Reword it to read the value through whatever accessor the
  implementation chooses, with the externally-binding contract on the
  wire-side property name; or, equivalently, mark the `entry.invocationId`
  spelling as the example side of an *illustrative* assignment and pin only
  the static property name `invocation_id`.
- `V9e` / `V9e-T` will need their **Adds** and **Tests** fields rewritten
  in the same commit (the `V9e` plan leaf currently anchors on "the
  five-field entry"). The accompanying coverage-matrix row `cka-26`
  (which already cites un-anchored / GOV-22 residue text) can be left
  unchanged.

## Consequence

**Severity:** advisory

Two implementers can converge on differently-named internal members or a
non-`Set` container and still emit byte-identical wire payloads and
diagnostics, yet the literal text marks one of them non-conformant. The
existing audit-fixture pin in the `disposeBarrier` resolver storage bullet
also forces a test surface against fields the same paragraph admits are
internal, defeating the seam-isolation pattern the rest of the page uses for
DI seams.
