## Finding

Step 0 (a) of the capability probe carries the normative prose: *"a Node nightly or release-candidate build (e.g. `"23.0.0-nightly20250101abcdef"`) whose stable equivalent satisfies the floor MUST satisfy the check."* The term **"stable equivalent"** has no spec-resident definition. Two reasonable interpretations are available — *strip the prerelease identifier and compare the resulting `MAJOR.MINOR.PATCH` triple*, or *the nearest published GA release on that line* — and they answer differently right at the floor boundary.

The clause is doubly slippery because the non-normative *Recommended recipe* immediately below — `semver.satisfies(process.versions.node, ">=22.19.0", { includePrerelease: true })` — implements a **third** semantic, one that contradicts the "version-triple" reading. Under SemVer precedence (which `semver` follows even with `includePrerelease: true`), a prerelease of `X.Y.Z` sorts *below* `X.Y.Z`; so `semver.satisfies("22.19.0-nightlyXXXX", ">=22.19.0", { includePrerelease: true })` returns `false`, while a "version-triple" reading would admit it. The example the spec actually picks (`"23.0.0-nightly20250101abcdef"` against `>=22.19.0`) lies safely above the boundary and so masks the disagreement on every reading — leaving no in-spec witness that pins which interpretation is intended.

Two implementers consulting Step 0 (a) and the recipe will therefore disagree on whether a Node `22.19.0-nightlyXXXX` build is host-incompatible, and the `V9a` probe leaf has no test vector that forces the question one way or the other.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `126ff8a` (2026-05-06, "pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B"), then `cb6cf60` (2026-05-07, "pi-loom spec: resolve 'semver is referenced as a production dependency but is missing from package.json'")
**History:** `git log -S"stable equivalent"` on the spec corpus identifies `126ff8a` as the first commit to introduce the phrase. In that revision Step 0 (a) read *"Compare `process.versions.node` against `>=20.6.0` using `semver.satisfies(... { includePrerelease: true })` ... `includePrerelease: true` is load-bearing: it permits Node nightly … builds (e.g. `"21.0.0-nightly..."`) whose stable equivalent satisfies the floor."* — i.e. "stable equivalent" was a **gloss anchored to the inline `includePrerelease: true` recipe call**, not an independent prose contract. The defect was latent: the meaning was anchored by the cited library semantic. `cb6cf60` then split the section into a *normative prose* paragraph and a separate non-normative *Recommended recipe* paragraph (to decouple the spec from a specific library, in service of a different finding). That split promoted "stable equivalent" from a recipe-anchored gloss to a stand-alone normative term, while the recipe (still using `{ includePrerelease: true }`) drifted into a paragraph the prose no longer cites for semantics. `ae529db` (2026-05-27) bumped the floor literal from `>=20.6.0` to `>=22.19.0` but did not change the wording or example structure. The defect is therefore the *interaction* of the gloss-introduction commit with the prose/recipe split commit; reverting either alone would not be the durable fix.

## Solution Space

**Shape:** multiple

### Option A — Drop the gloss; name SemVer-with-prerelease-eligible precedence as the contract

**Approach.** Replace the "stable equivalent" sentence with a contract stated directly in SemVer-precedence terms: *"A `process.versions.node` value MUST satisfy `>=22.19.0` under SemVer precedence with prerelease versions eligible — equivalently, `semver.satisfies(process.versions.node, ">=22.19.0", { includePrerelease: true })` returns `true`. A prerelease of `X.Y.Z` sorts below `X.Y.Z` under this precedence; an implementation MUST NOT admit `22.19.0-<tag>` as satisfying `>=22.19.0`."*

**Spec edits.** Edit Step 0 (a) in `capability-probe.md`; add one normative sentence pinning the boundary case. Leave the *Recommended recipe* paragraph as-is — it already matches. Adjust the `loom/load/host-incompatible` row in `diagnostics/code-registry-load.md` to refer to "SemVer precedence with prereleases eligible" (drop "stable equivalent" if it appears anywhere else; a `grep` of the corpus shows only `capability-probe.md` §(a) carries it today).

**Pros.** Eliminates the divergence by making the recipe's semantic the normative one. The recipe stops being a parallel-but-non-citable specification. A conformance test for the boundary case (`22.19.0-nightlyXXXX` → refused) is now grounded in spec text.

**Cons.** Slightly tightens the contract relative to whatever an implementer might have hoped "stable equivalent" meant; a host running a `22.19.0`-line nightly during the GA-day window is refused.

**Risks.** Low. The change matches the long-standing recipe and the V9a leaf's existing structure; no other section depends on the version-triple reading.

### Option B — Pin the version-triple reading and align the recipe to it

**Approach.** Take the source review's suggested fix at face value: replace the sentence with *"a `process.versions.node` value whose [major, minor, patch] triple (ignoring the prerelease identifier) satisfies the floor MUST satisfy the check."* Then re-author the recipe paragraph so the recommended call yields the same answer — e.g. recommend `semver.satisfies(semver.coerce(process.versions.node).version, ">=22.19.0")` or an explicit triple-extraction-then-compare snippet — because the current `{ includePrerelease: true }` call no longer satisfies the contract at the boundary.

**Spec edits.** Edit Step 0 (a) prose **and** §(a) *Recommended recipe* in `capability-probe.md`. Edit the `loom/load/host-incompatible` row in `diagnostics/code-registry-load.md` to swap "SemVer ordering with prerelease tags eligible" for the triple semantic and to update the recipe snippet it inlines.

**Pros.** More permissive on GA-day boundaries (admits late nightlies of the floor patch).

**Cons.** Wider edit surface (prose + recipe + diagnostics row). Loses the `semver` library's idiomatic call shape. Forces every future SemVer-implementation swap (the spec explicitly contemplates this) to re-implement triple extraction rather than rely on a flag the library already exposes.

**Risks.** Easy to leave the recipe and prose subtly out of sync again on a future edit, since the natural `semver` API does not express this semantic in one call.

### Recommendation

**Option A.** The recipe paragraph has lived as `{ includePrerelease: true }` since `126ff8a` and is what every downstream cross-reference (`diagnostics/code-registry-load.md`, `placeholder-rendering-b.md`) already describes; promoting it to the normative reading is a one-paragraph edit that ratifies the corpus as it stands rather than rippling a contradicting semantic outward. Resolve **`capability-probe.md / audit-target-categories.md` — mandated probe `createRequire(...).resolve("<peer>")` collides with the family-(4) CJS-reach prohibition** first, and **probe (d) peer-dep version-read mechanic over-prescribed** before this one, since both rewrite adjacent Step 0 (d) text; landing them first gives this edit a stable §(a) baseline. The implementer must add one negative test vector to `V9a-T`: `process.versions.node = "22.19.0-nightlyXXXX"` MUST emit `loom/load/host-incompatible` with `details.kind = "node-floor"`.

## Consequence

**Severity:** correctness

Two implementers will diverge at the floor boundary. One follows the recipe verbatim and refuses `22.19.0-nightlyXXXX` as `loom/load/host-incompatible (node-floor)`; the other follows the "stable equivalent" prose under the version-triple reading and admits the same build. The defect is narrow in surface (one sentence) but produces an observable host-acceptance divergence that conformance tests cannot adjudicate from the spec text alone.
