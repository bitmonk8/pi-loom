## Problem

Step 0 (a) Node floor in `capability-probe.md` carries the normative phrase *"whose stable equivalent satisfies the floor"*, but "stable equivalent" has no spec-resident definition and admits at least two readings that disagree exactly at the floor boundary (strip the prerelease tag and compare the `MAJOR.MINOR.PATCH` triple, vs the nearest published GA release). The adjacent non-normative *Recommended recipe* — `semver.satisfies(process.versions.node, ">=22.19.0", { includePrerelease: true })` — implements a third semantic: under SemVer precedence a prerelease of `X.Y.Z` sorts below `X.Y.Z`, so a `22.19.0-<tag>` build fails `>=22.19.0`, contradicting the version-triple reading. The example the spec picks (`"23.0.0-nightly20250101abcdef"` against `>=22.19.0`) sits above the boundary and masks the disagreement, leaving no in-spec witness that pins the intended meaning. Two implementers therefore diverge on whether a `22.19.0-nightlyXXXX` build is host-incompatible.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `126ff8a` (2026-05-06, "pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B"), then `cb6cf60` (2026-05-07, "pi-loom spec: resolve 'semver is referenced as a production dependency but is missing from package.json'")
**History:** `git log -S"stable equivalent"` on the spec corpus identifies `126ff8a` as the first commit to introduce the phrase. In that revision Step 0 (a) read *"Compare `process.versions.node` against `>=20.6.0` using `semver.satisfies(... { includePrerelease: true })` ... `includePrerelease: true` is load-bearing: it permits Node nightly … builds (e.g. `"21.0.0-nightly..."`) whose stable equivalent satisfies the floor."* — i.e. "stable equivalent" was a **gloss anchored to the inline `includePrerelease: true` recipe call**, not an independent prose contract. The defect was latent: the meaning was anchored by the cited library semantic. `cb6cf60` then split the section into a *normative prose* paragraph and a separate non-normative *Recommended recipe* paragraph (to decouple the spec from a specific library, in service of a different finding). That split promoted "stable equivalent" from a recipe-anchored gloss to a stand-alone normative term, while the recipe (still using `{ includePrerelease: true }`) drifted into a paragraph the prose no longer cites for semantics. `ae529db` (2026-05-27) bumped the floor literal from `>=20.6.0` to `>=22.19.0` but did not change the wording or example structure. The defect is therefore the *interaction* of the gloss-introduction commit with the prose/recipe split commit; reverting either alone would not be the durable fix.

## Solution approach

Rewrite the "stable equivalent" sentence in `capability-probe.md` §(a) Node floor to state floor membership directly in SemVer-precedence terms, ratifying the existing non-normative *Recommended recipe* semantic (prerelease tags eligible, as `{ includePrerelease: true }`) as the normative one. Pin the boundary case explicitly: a prerelease of `X.Y.Z` sorts below `X.Y.Z`, so a `22.19.0-<tag>` build does not satisfy `>=22.19.0`. If any other spec consumer (e.g. the `loom/load/host-incompatible` row in `diagnostics/code-registry-load.md`) describes the floor via "stable equivalent", realign it to the same SemVer-precedence wording.

## Solution constraints

- The *Recommended recipe (non-normative)* paragraph in §(a) already implements the intended semantic; do not rewrite it.
