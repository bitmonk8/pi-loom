## Finding

`H4a`'s manual real-host smoke governance protocol fires on two triggers. Trigger (2) is "any merge whose diff **touches the four fidelity-contract axes**", which the leaf then operationalises as: "observably, a commit that edits either the in-process double's modelling of, or the production seam realising, any of (i) streamed-token order relative to `ctx.waitForIdle()` resolution, (ii) single-turn prompt-mode append semantics, (iii) the `pi.on` cancel-forward subscription, or (iv) cancellation propagation."

The word "edits" carries no boundary. Two reasonable contributors land on different trigger sets:

- **File-touch reading** — any diff that touches a file containing the double's modelling code or the production seam realising one of the four axes triggers the smoke, including pure refactors, renames, and comment/format-only changes to those files.
- **Behavioural reading** — only a diff that changes the observable behaviour the named axis produces triggers; a refactor or comment change that leaves all four axes' observable behaviour unchanged does not, while a behaviour-affecting change landing in an otherwise-unrelated file does.

These sets diverge in both directions: the file-touch reading over-triggers on no-op edits to the seam files, and the behavioural reading triggers on axis-affecting changes outside the obvious files. The leaf already disclaims *mechanical* enforcement — "the trigger-(2) condition … has no automatic diff detector — recognising it is itself part of the contributor's obligation" — but that disclaimer is about who/what detects a qualifying diff, not about *what qualifies*. A contributor performing the obligation in good faith still has no rule that decides whether a given diff is in scope. Because loom 1.0 has no mechanical detector and `H6a`'s release-gate audit reads the recorded run only after the fact, a contributor who classifies a behaviour-changing diff as "not editing the modelling" silently skips the only real-host divergence check loom has.

## Issue introduction

**Verdict:** present-since-inception (of the manual-smoke protocol)
**Introducing commit:** `e7f14dd` ("pi-loom plan: resolve \"Real-host fidelity of the session double has no reproducible detection point\"", 2026-06-11)
**History:** The manual real-host smoke protocol, and with it the trigger-(2) operationalisation "a commit that edits either the in-process double's modelling of, or the production seam realising, any of (i)–(iv)", was introduced wholesale in `e7f14dd`; no earlier `H4a` revision contained the protocol or the trigger, so the boundary ambiguity is coextensive with the protocol's inception. The `15eb44c` rewrite (2026-06-11) restructured the gate from a post-merge "acceptance trigger" into a pre-merge "required pre-merge gate" but copied the "edits either … modelling of … production seam realising …" clause forward verbatim; the later commits touching this paragraph (`ad23856` relocating the protocol from Tests to Adds, `2f18db6` reconciling the `src/**` root, among others) left the wording unchanged. The ambiguity has therefore persisted in every version of the protocol.

## Solution Space

**Shape:** multiple

### Option A — observable-behaviour-change criterion

**Approach.** Scope trigger-(2) to a diff that changes the *observable behaviour* of one of the four named axes (in either the double's modelling or the production seam); a pure refactor, rename, comment, or provably-unreachable edit that leaves all four axes' observable behaviour unchanged does not trigger.
**Plan edits.** In `H4a`'s Adds, replace the "observably, a commit that edits either the in-process double's modelling of, or the production seam realising, any of (i)–(iv)" clause with one that scopes on observable-behaviour change. Candidate text: "observably, a commit whose diff changes the observable behaviour of either the in-process double's modelling of, or the production seam realising, any of (i)–(iv) — a change to the logic that produces the named-axis observable; a pure refactor, rename, comment, or provably-unreachable edit that leaves all four axes' observable behaviour unchanged does not trigger."
**Spec edits.** None.
**Pros.** Aligns with the clause's existing "edits the modelling of [axis]" framing and the protocol's design as a contributor-judgment obligation; minimises wasted runs on no-op edits.
**Cons.** "Observable behaviour unchanged" still needs contributor judgment; not mechanically decidable.
**Risks.** A contributor under-estimates a diff's behavioural effect and skips a run.

### Option B — file/module proximity criterion

**Approach.** Trigger-(2) fires iff a diff touches an enumerated set of source modules that realise the double's modelling or the production seam for the four axes, regardless of whether behaviour changes.
**Plan edits.** In `H4a`'s Adds, replace the clause with one that names the enumerated module set and adds that list (the harness/double module plus the production-seam modules realising axes (i)–(iv)).
**Spec edits.** None.
**Pros.** Near-mechanical, low contributor judgment; cannot miss a behavioural change that lands in a listed module.
**Cons.** Over-triggers on refactors and comment-only edits to the listed modules; the module list needs maintenance as the codebase evolves and can miss a behaviour change landing outside the list.
**Risks.** A stale module list silently narrows the trigger, with no gate to catch the drift.

### Recommendation

Adopt Option A. It matches the clause's existing behavioural framing and the protocol's stated design as a contributor-judgment obligation with no mechanical detector, and it adds no enumerated-list maintenance burden that loom 1.0 has no gate to keep current. Pair the criterion with a "when in doubt, run the smoke" default so a contributor uncertain whether a diff changes a named axis's observable behaviour errs toward running it. The spec is read-only for this fix — the smoke protocol is plan-internal governance the spec does not request — and `H6a`'s release-gate audit text stays unchanged, since it audits the recorded run, not the trigger boundary.

## Consequence

**Severity:** advisory

Two contributors apply different trigger-(2) sets, so the protocol fires inconsistently: the file-touch reading wastes manual smoke runs on no-op edits, while the behavioural reading lets an author who under-estimates a diff's effect on a named axis skip the run. Since loom 1.0 has no mechanical detector and `H6a` audits the recorded run only after merge, the under-trigger case is the real exposure — a genuine double-vs-real divergence introduced by a loom change reaches `main` with no smoke run, defeating the protocol's purpose. It degrades rather than blocks: a conscientious contributor can still run the smoke when uncertain.
