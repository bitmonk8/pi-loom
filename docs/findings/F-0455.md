## Finding

`coverage-matrix.md`'s *Closing-leaf column authoring* rule requires every leaf-ID in a *Closing leaf(s)* cell to be backtick-delimited, because [`H5d`](docs/plan_topics/H5d-transitive-completeness-gate.md) tokenises each cell by taking **exactly its backtick-delimited spans** as leaf-ID tokens. A leaf-ID that an author forgets to wrap in backticks is therefore not seen as a token at all — it is dropped from the tokenised set with no signal. In a single-leaf cell this surfaces loudly (the cell tokenises empty and H5d reddens with a spurious failure). In a **multi-leaf** cell it fails silently: H5d's per-cell *at-least-one* rule stays green as long as one other listed leaf reaches `H5b`'s `Deps.`, so a dropped facet-closer leaves the gate green while that facet's closure goes unverified.

The plan documents the risk (both the authoring rule and H5d's *Adds* note it), but every backstop nominally responsible for catching a dropped facet derives its facet partition from the **same already-corrupted cell**, so none of them can see the missing facet. [`H5f`](docs/plan_topics/H5f-per-facet-citing-test-gate.md) derives each multi-leaf row's facet partition "from the cell alone" by reusing H5d's backtick-delimited tokens as facet keys — a leaf that produced no token is never a "remaining facet" it checks. The manual [`conventions.md`](docs/plan_topics/conventions.md) release-time residue inspection item 7 likewise derives the partition "from the cell alone." The result: a single forgotten pair of backticks removes a spec facet from coverage, and no mechanical gate (H5d, H5f) nor the manual sweep (item 7) can detect that the facet was ever supposed to be there.

The only place the defect is observable is the **raw cell text before tokenisation** — a leaf-ID-shaped token (`V<n><letter>`, `H<n><letter>`, or a named horizontal singleton) sitting bare in a *Closing leaf(s)* cell. No current arm inspects the cell at that level; every consumer reads the post-tokenisation set, by which point the evidence is gone.

## Issue introduction

**Verdict:** introduced-by-commit (single introducing commit; reinforced-but-not-closed by a later commit)
**Introducing commit:** `8b69cd9` — "pi-loom plan: resolve \"H5d tokeniser depends on unstated backtick authoring convention\"" (2026-06-18)
**History:** The plan corpus is git-tracked. `git log -S 'silently removing a facet-closer'` and `git log -G 'at-least-one rule keeps it green'` over `docs/plan_topics/` both return exactly `8b69cd9` as the sole commit that introduced this wording; `git log -S 'Closing-leaf column authoring'` confirms `8b69cd9` first introduced the authoring-rule heading. That commit established the backtick tokenisation convention and explicitly documented the dropped-facet residue ("silently removing a facet-closer from a multi-leaf cell while the at-least-one rule keeps it green and the dropped leaf unverified") but added no mechanical detector for it. A later commit `26a8794` — "resolve \"split coverage-matrix rows can ship a facet untested\"" (2026-06-18) — added the [`H5f`](docs/plan_topics/H5f-per-facet-citing-test-gate.md) per-facet citing-test arm and the *Multi-leaf-row per-facet citing tests* convention, which read as a backstop but derive the facet partition from the same closing-leaf cell and so do not close the dropped-facet hole. The defect has therefore been present since the backtick convention entered at `8b69cd9`.

## Solution Space

**Shape:** single

### Recommendation

Add a mechanical pre-tokenisation check that operates on the **raw text** of each *Closing leaf(s)* cell in both `coverage-matrix.md` tables and reddens when a leaf-ID-shaped token appears **un-backticked** in the cell. The check is the only mechanism that can detect the defect, because it runs before the backtick tokeniser discards the evidence.

- Recognise a bare leaf-ID-shaped token as one matching the plan's leaf-ID forms — `V<n><letter>`, `H<n><letter>`, and the named horizontal singletons (`H1a`, `H5a`, `M`) the plan already uses — appearing in a *Closing leaf(s)* cell outside any `` `…` `` span. Such a token reddens the gate as a missing-backtick authoring defect.
- The literal `<new>` placeholder token is exempt (it is intentionally bare and tokenises to the empty set, per *Code-keyed obligation areas*); the comma separators and the parenthetical annotation prose carry no leaf-ID-shaped tokens, so they are unaffected.
- Host the check in `H5d-transitive-completeness-gate.md` (its arm already reads every *Closing leaf(s)* cell and depends on the *Closing-leaf column authoring* precondition, so the bare-token scan is the natural enforcement of that precondition), or as a separate `<new>` arm registered under the [`H5a`](docs/plan_topics/H5a-closing-gate-automation.md) unified closing-gate machinery; either lands the same observable. Because the scan reads only ever-present plan files it has no live-corpus flip, mirroring H5d's existing footing.
- In `coverage-matrix.md`'s *Closing-leaf column authoring* paragraph, replace any implication that the dropped-facet case is caught by residue item 7 / H5f with a pointer to this mechanical scan; item 7 and H5f derive their partition from the post-tokenisation cell and cannot witness a dropped facet.

Edge case for the implementer: an annotation that legitimately names a leaf-ID in prose would currently trip the scan; require such mentions to be backtick-delimited (so they become tokens, where the existing spurious-token guidance applies) rather than left bare, keeping the "no bare leaf-ID-shaped token in a cell" invariant total.

## Consequence

**Severity:** correctness

A facet-closer whose backticks are omitted is silently removed from a multi-leaf coverage-matrix cell; H5d's at-least-one rule keeps the gate green, H5f never sees the facet (its partition derives from the same corrupted cell), and the manual residue item-7 sweep derives from that cell too — so a spec facet ships with no per-facet citing test and no gate fires. The closing-gate net passes vacuously for that facet on an authoring slip, with no mechanical or manual detection path.
