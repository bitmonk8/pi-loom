## Finding

The H1 of `docs/spec_topics/binder/defaulting-system-note-echo.md` is the bare string `Defaulting system note echo`. The file's own body is partitioned into three independent `##` sections — `## Defaulting`, `## System-note rendering`, and `## Echo policy` — and the H1 simply concatenates those three section names with the hyphens of the filename replaced by spaces. There is no connective grammar (no commas, no "and", no shared subject) and no unifying concept named.

The result is that the H1 is not a title at all but a typographically de-slugged filename. A reader landing on the page from a cross-reference cannot tell from the H1 what the page is *about* (the binder's post-return / pre-display processing of the model's response); they can only tell which three things it covers. The other three binder sub-pages — `Binder model and context`, `Binder bypass and envelope`, `Determinism cancellation failure` — share the same affliction in the abstract, but only this one packs three independently-named top-level concerns into the page; the others enumerate aspects of a single named subject.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commit:** `f5e89f4` — `docs: enforce 100KB hard / 30KB recommended size cap on spec set` (2026-06-04, Thomas Andersen)
**History:** Before this commit the three sections `## Defaulting`, `## System-note rendering`, and `## Echo policy` lived as three of the seventeen top-level sections of a single `docs/spec_topics/binder.md` page under the H1 `# Slash-Command Argument Binding` — a unified, well-named title. The cited commit split that page into a `binder.md` index plus four child pages to enforce the 30KB-per-file cap; the splitter generated each child page's H1 mechanically from its filename slug (filename `defaulting-system-note-echo.md` → H1 `Defaulting system note echo`). The three section names had never been intended to compose into a title, so the mechanical de-slugging produced a non-title for this child alone. The commit message confirms the split was driven by the size cap and was anchor-preserving / link-repointing in scope; no editorial pass over the generated H1s was performed. `git log -S 'Defaulting system note echo'` returns only this one commit.

## Solution Space

**Shape:** multiple

### Option A — Retitle in place

**Approach.** Replace the H1 with a name for the shared concern the three sections embody — the binder's runtime-side post-processing of a validated envelope before the loom starts. A workable title: `# Binder post-processing: defaults, note rendering, and echo` (alternative phrasings: `# Binder result processing`, `# Post-binder processing`). Optionally add a one-sentence lede paragraph immediately under the H1 stating that scope ("This page covers what the runtime does between the binder's return and the loom's start: filling defaults, rendering system notes, and emitting the argument echo.") so the orientation cue is not load-bearing on the title alone.

**Spec edits.**
- Replace the H1 of `docs/spec_topics/binder/defaulting-system-note-echo.md`.
- Update the link text in `docs/spec_topics/binder.md` `## Contents` to match.
- (Optional) Insert a one-sentence lede paragraph between the H1 and `## Defaulting`.
- Leave filename, anchors, and all inbound links untouched.

**Pros.** Zero churn on plan leaves, coverage matrix, or anchor map. Keeps the three sections — which cross-reference each other heavily (Defaulting ↔ Echo policy, Echo policy ↔ System-note rendering) — colocated. File is ~18 KB, well under the 30 KB recommended cap, so the size argument that drove the original split does not push back toward a re-split.

**Cons.** A two- or three-noun title is still longer and less crisp than the sibling pages' titles. Picking the title requires one editorial judgement call.

**Risks.** None material.

### Option B — Split into three sibling pages

**Approach.** Promote each `##` section to its own child page under `docs/spec_topics/binder/`: `defaulting.md`, `system-note-rendering.md`, `echo-policy.md`. Update the `binder.md` `## Contents` list to list all three. Repoint every inbound link (plan leaves, coverage matrix, cross-shard back-references, intra-binder back-references) to the new paths.

**Spec edits.**
- Create three new child pages, each with a singular H1.
- Move the three sections' bodies (and their anchors) into them.
- Delete `defaulting-system-note-echo.md`.
- Update the `binder.md` `## Contents` list.
- Rewrite every `defaulting-system-note-echo.md#<anchor>` reference across `docs/` (plan leaves, coverage matrix, other binder shards, findings) — the same map-driven rewrite the `f5e89f4` splitter ran, but in reverse-ish.

**Pros.** Each child page would have a crisp, single-concept H1 matching the sibling-page convention.

**Cons.** Order-of-magnitude more churn than Option A: four plan leaves, the coverage matrix, several findings under `docs/findings/`, and the intra-page back-references inside the three sections themselves would all need link updates. The three sections cross-reference each other constantly (the `Defaulting` section links to `Echo policy` and to `System-note rendering`; the `Echo policy` section links back to both) — splitting forces every such reference to become a cross-file link.

**Risks.** Link breakage during the rewrite (one of the explicit costs the `f5e89f4` commit message flags). A future reader losing the "these three things happen in this order on the same payload" colocation cue the single-page layout currently provides.

### Recommendation

**Take Option A.** Retitle to `# Binder post-processing: defaults, note rendering, and echo` (or an equally explicit alternative naming the shared subject) and update the matching link text in `binder.md`'s `## Contents`. The original split was driven by a 30 KB size cap that this page is already comfortably under, and the three sections are tightly cross-referenced — there is no editorial reason to pay Option B's repoint cost. Edge cases for the implementer: keep the filename and all `<a id>` anchors unchanged (they are load-bearing for the entire plan and coverage matrix); make sure the new link text in `binder.md` `## Contents` matches the new H1 verbatim.

## Consequence

**Severity:** cosmetic

A reader navigating to the page from a TOC or a cross-shard back-link gets no orientation cue beyond "this page covers three things"; the H1 carries no information the URL did not already carry. No implementer behaviour diverges — every normative obligation is anchored on its `##` heading, not on the H1.
