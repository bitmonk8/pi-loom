## Finding

The *Source priority* paragraph in `discovery/discovery-sources.md` states that "when the same slash name resolves from multiple sources, the higher-priority source wins and `loom/load/cross-source-shadow` is emitted naming both paths," and pins the rendered message to the `loom/load/cross-source-shadow` row's two-slot template `'<higher>' wins over '<lower>'`. Both the prose ("naming both paths") and the template encode a binary winner/loser pair.

The discovery model admits five distinct-priority sources (CLI flag, Settings, Project, Packages, Global). When the same slash-name stem resolves from three or more of them simultaneously — e.g. `plan.loom` present in CLI, Settings, and Project — the binary framing has no defined behaviour. It is unspecified whether the loader emits one diagnostic naming all shadowed paths (which the two-slot `<lower>` template cannot render), one diagnostic per pairwise (winner, loser) comparison (yielding multiple warnings for a single name), or one diagnostic naming only the winner and a single representative loser. The sibling `loom/load/cross-format-collision` code (for same-priority collisions) explicitly "lists every colliding path," so the spec already has a precedent for the N-way shape — but the cross-source (different-priority) path was never reconciled with it.

The winning loom is unambiguous in every case (highest priority registers); the gap is confined to the shape, count, and rendered content of the warning diagnostic, which is operator-observable.

## Solution Space

**Shape:** single

### Recommendation

Specify that a slash-name shadowed across N≥2 distinct-priority sources produces **exactly one** `loom/load/cross-source-shadow` warning naming the winning (highest-priority) path and **every** shadowed lower-priority path, mirroring the `cross-format-collision` rule that "lists every colliding path." Update the *Source priority* paragraph to say the diagnostic names the winner and all shadowed paths (not "both paths"), and revise the `loom/load/cross-source-shadow` row's message template so it renders the winning path followed by the full list of shadowed paths rather than a single `<lower>` slot.

Edge cases the implementer must pin:
- One diagnostic per shadowed slash name — not one per pairwise (winner, loser) comparison.
- The shadowed-path list MUST be deterministically ordered (priority order high-to-low, then source-walk order within a tie) so the rendered `message` is byte-identical across runs, consistent with the diagnostics channel's determinism requirements.
- The two-source case must remain a strict special case of the N-source rendering (no separate template).

## Consequence

**Severity:** correctness

Two reasonable implementers would emit divergent operator-facing diagnostics for a ≥3-source shadow — one warning listing all shadowed paths, three pairwise warnings, or a single winner/loser pair — so the observable diagnostic surface is non-deterministic across conformant implementations. Which loom registers is unaffected.
