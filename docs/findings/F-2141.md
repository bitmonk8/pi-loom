## Finding

The `LoomRegistry` drain-state contract exposes a closed four-method surface, two members of which write the `drainStateTag` field: `initDrainStateTag()` (sets `"shutting-down"`) and `markRuntimeDegraded()` (sets `"degraded-needs-reload"`). The two write methods are named on incompatible conventions. `initDrainStateTag` embeds the target field name (`DrainStateTag`) plus a lifecycle verb (`init`), making the field relationship explicit at the call site. `markRuntimeDegraded` names the domain-level outcome (`RuntimeDegraded`) with no reference to the field it mutates, so it is not apparent from the method name alone that its effect is `drainStateTag = "degraded-needs-reload"`.

A reader learning the API from the method names cannot tell that `markRuntimeDegraded` and `initDrainStateTag` write the same field. The surface already mixes conventions further: `drain()` is a bare verb, `readDrainState()` embeds the field prefix. The result is that the closed four-method surface — explicitly designated the source of truth for these names and required to be swept in lock-step on any rename — carries no single naming convention an author can apply when reasoning about which method touches which field.

The page's non-normative naming-rationale paragraph discusses the `drainStateTag`/`tag` long-vs-short split and names `initDrainStateTag` and `readDrainState` as the two methods embedding the longer field name, but it does not address or justify the asymmetry between the two *write* methods.

**Verdict:** present-since-inception (of the two-write-method surface)
**Introducing commit:** `52b9101` — *pi-loom spec: resolve "T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state"* (2026-05-11)
**History:** The spec corpus is git-tracked. `git log -S` over both method tokens (across all files, since the contract predates the `pi-integration-contract.md` → topic-file split) reports `52b9101` as the earliest commit introducing each of `initDrainStateTag` and `markRuntimeDegraded`; the diff of `52b9101` adds both names in the same change, alongside the new degraded-state branch and the `markRuntimeDegraded` call site. The naming asymmetry therefore entered the corpus at the moment the second write method was coined next to the first — it was never a later regression. No subsequent commit reconciled the two conventions.

## Solution Space

**Shape:** multiple

The two write methods can be reconciled in either direction; both fully resolve the finding and they are mutually exclusive (one naming convention is adopted for the surface).

### Option A — Field-name-surfacing convention

- **Approach:** Make both write methods embed the field name. Keep `initDrainStateTag` and rename `markRuntimeDegraded` → `setDrainStateTagDegraded` (or equivalent field-embedding name).
- **Spec edits:** Rename the method in the *Methods* paragraph; sweep the four reference sites the page's own rename rule enumerates; update `session-only-degraded-state.md`'s interstitial-seam reference.
- **Pros:** Preserves symmetry with the existing `drainStateTag` field name and with `readDrainState`; both write methods visibly name their target field.
- **Cons:** Produces longer, less readable call-site names; pulls against the deliberate short-`tag` snapshot-key decision the page already records; `drain()` remains a bare verb, so the surface is still not uniform.
- **Risks:** Wider rename surface (the longer name appears in more prose); the `details.call` label `"loomRegistry.markRuntimeDegraded"` (cross-reference-only) must also be renamed.

### Option B — Semantic-outcome convention

- **Approach:** Make both write methods name the outcome. Keep `markRuntimeDegraded` and rename `initDrainStateTag` → `markShuttingDown` (or equivalent outcome name).
- **Spec edits:** Rename the method in the *Methods* paragraph; sweep the four reference sites the page's rename rule enumerates (notably the `details.call` label `"loomRegistry.initDrainStateTag"` in the **Per-step isolation** set, sub-step 1, the **idempotent** clause, and the `reason: "new" | "resume" | "fork"` edge-case bullet); update the non-normative naming-rationale paragraph that currently names `initDrainStateTag` as one of the two methods embedding the field name.
- **Pros:** Aligns the three mutating methods (`drain`, `markShuttingDown`, `markRuntimeDegraded`) on a verb-outcome convention; more concise call-site API consistent with the short `tag` snapshot key.
- **Cons:** Loses the explicit field-name link on the write methods (the reader relies on the field definitions for the mapping); the rationale paragraph's claim about which methods embed the field name must be re-stated.
- **Risks:** The `details.call` label sweep is load-bearing — fixtures key on the label `"loomRegistry.initDrainStateTag"`, so the `V9m`/`V9m-T` and `V9l`/`V9l-T` acceptance criteria must be updated in lock-step.

### Recommendation

Adopt **Option B**: rename `initDrainStateTag` → `markShuttingDown`. It gives the three mutating methods one verb-outcome convention while leaving `readDrainState` as the sole reader, and it matches the page's existing decision to keep dispatch-site surfaces concise. The implementer must perform the full four-site sweep the spec's own rename rule mandates — including the `details.call` label in the **Per-step isolation** set and the fixture labels in `V9m-T`/`V9l-T` — and update the non-normative naming-rationale paragraph so it no longer cites `initDrainStateTag` as a field-name-embedding method.

## Consequence

**Severity:** advisory

The method surface is fully specified behaviourally and is explicitly declared non-observable, so the inconsistency does not cause two implementers to diverge on runtime behaviour. It is a readability/discoverability gap: an author or implementer scanning the method names cannot infer the field-write relationship uniformly, raising the risk of mis-citing or misusing the write methods in prose and code.
