## Finding

The spec fixes the source location stamped into the `discard_site` field of the runtime discard-observability event differently for the two discard forms. For the explicit-discard form it is "the location of the discarding `let _ =` binding"; for the void-tail-function form it is "the location of the tail `` @`...` `` expression (**its start**, for a tail expression spanning multiple source lines)" (`docs/spec_topics/query/query-escapes-stringification.md` §discard observability).

`V13g` and its paired `V13g-T` describe the same stamping as "the location of the discarding `let _ =` binding (or tail `` @`...` `` expression for the `void`-tail form)" in both their **Tests** bullets, and `V13g` repeats the abbreviated form in **Ships when**. The "**its start**, for a tail expression spanning multiple source lines" qualifier is dropped. For a single-line tail expression the position is unambiguous, so the gap is invisible. For a tail expression spanning multiple source lines — which the spec explicitly calls out — the plan leaves the stamped position under-constrained: nothing in the leaf tells the implementer to stamp the expression's start rather than, say, the position of the closing backtick or the last line.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 4e395ef — pi-loom plan: resolve "V13g discard-observability tests assert only that the event fires, not payload or channel" (2026-06-18, Thomas Andersen)
**History:** The `discard_site` stamping clause did not exist in `V13g` until commit 4e395ef, which expanded the **Tests** and **Ships when** bullets to assert the payload and channel. That commit authored the abbreviated "(or tail `` @`...` `` expression for the `void`-tail form)" parenthetical and omitted the spec's "its start, for a tail expression spanning multiple source lines" qualifier, which had already been present in `docs/spec_topics/query/query-escapes-stringification.md` since commit 13e58fe (2026-06-06). The plan therefore diverged from the spec at the moment the stamping clause was first written.

## Solution Space

**Shape:** single

### Recommendation

In the `discard_site` stamping clause of the **Tests** bullet of both `docs/plan_topics/V13g-query-discard-observability.md` and `docs/plan_topics/V13g-T-query-discard-observability.md`, carry the spec's multi-line qualifier for the `void`-tail form. Replace the parenthetical "(or tail `` @`...` `` expression for the `void`-tail form)" with text that pins the position to the start of the tail expression for multi-line tails, e.g.:

> stamps `discard_site` with the location of the discarding `let _ =` binding for the expression-statement form, and the start position of the tail `` @`...` `` expression — its start when the tail expression spans multiple source lines — for the `void`-tail form

Apply the same expansion to the `discard_site` mention in `V13g`'s **Ships when** bullet so the acceptance gate and the test bullet stay aligned. The spec is the authority for the wording and is read-only for this fix — mirror it; do not edit the spec topics.

## Consequence

**Severity:** correctness

Two reasonable implementers can stamp different `discard_site` positions for a multi-line `void`-tail discard (start of the tail expression vs. closing backtick vs. last line), and the leaf's acceptance gate accepts any of them because the test bullet never pins the position. A leaf can ship green while stamping a location the spec disallows, so log scrapers / replay tools / `/tree` navigation point operators at the wrong source line.
