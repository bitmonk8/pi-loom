## Problem

The V9 and V11 slice interleave notes in `docs/plan.md` both assert that `V11a` (Binder-model resolution) is a prerequisite of `V9c`/`V9i`/`V9j` and draw the seam `V9b → V11a → V9c`/`V9i`/`V9j`. The authoritative `Deps` contradict this for two of the three named leaves: `V9c` does not list `V11a` and consumes no binder-model resolution, and `V9i`'s pre-spawn guard reads the resolved `model:` via `V6a` rather than the `bind_model:` resolution `V11a` owns. Only `V9j` actually declares `V11a`. The notes therefore misstate the DAG at the `V9b → V11a` integration seam.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `b9b427d` (2026-06-10, "pi-loom plan: resolve \"slice-ordering narrative overpromises DAG order\""); `8490e2c` (2026-06-25, "pi-loom spec-plan: fix F-1327 — relabel V9i pre-spawn model guard, swap V11a->V6a dep")
**History:** `b9b427d` authored both interleave notes, introducing the `V11a … is a prerequisite of \`V9c\`/\`V9i\`/\`V9j\`` claim as added lines. `git log -S 'V11a'` over `docs/plan_topics/V9c-conversation-drive.md` returns no commits, so `V9c` never carried `V11a` in its `Deps` — the `V9c` strand of the claim was wrong at authoring. The `V9i` strand became wrong later: `8490e2c` (F-1327) swapped `V9i`'s `V11a` dependency to `V6a` when relabelling its pre-spawn model guard, but did not propagate that correction into the interleave notes, leaving the `V9i` reference stale. The current full defect is the join of both: an overbroad claim at authoring (`V9c`) plus an un-propagated dep correction (`V9i`).

## Solution approach

Narrow the `V11a`-prerequisite claim in both interleave notes (V9 slice note and V11 slice note in `docs/plan.md`) so it names only `V9j`, updating the drawn seam to `V9b → V11a → V9j` to match. Leave `V9j`'s reference intact and change no leaf `Deps`.
