## Finding

The step-2 preamble mandates that every editorial-review item record an outcome of `pass` / `fail` / `N/A` / `not-performed` with a one-line rationale, and item (e) builds an explicit merge gate on its `fail` arm (the recovery serialisation must land "in the same edit as the bump," and a non-positive second-pass outcome "does not clear the merge gate"). The SHOULD-level items `(f)`–`(ak)` carry no analogous disposition. Some of them — `(g)`, `(j)`, `(q)`, `(v)`, `(ad)`, `(ah)` — close with a "surface the divergence on the bump commit so [the cited paragraph] can be amended in the same edit" clause, but the majority of the SHOULD-level items end at "would surface as <runtime symptom> with no SDK surface-inventory signal" and never say what the contributor does next when the audit records `fail`. The preamble never globalises any disposition either.

Two contributors auditing the same `fail` outcome on, say, item `(f)` (no "amend" clause) or item `(s)` (no "amend" clause) reach incompatible conclusions: one treats the recorded `fail` as a non-blocking note carried in the commit message, the other treats it as merge-blocking by analogy to item (e). The plan side has already chosen the answer — `V18c`'s **Ships when** clause states that "a `fail` on any MUST-level item `(a)`–`(e)` blocks the bump, whereas a recorded `fail` / `not-performed` on a SHOULD-level item `(f)`–`(ak)` is a conformant non-blocking outcome, per `version-bump-step2.md`" — but the cited spec carries no such sentence, so the plan's "per `version-bump-step2.md`" reference is to a disposition the spec does not actually state.

The asymmetric per-item "amend in the same edit" clauses compound the ambiguity: where they appear they imply a tighter posture than "non-blocking," and where they are absent the spec is silent, with no preamble rule explaining which posture governs which items.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e89e42e` (June 2026, "T21 Pi-side slash-handler promise lifecycle taken as given" — first SHOULD-only item, `(g)`, with per-item "amend in the same edit" clause but no preamble disposition); subsequent SHOULD-item-adding commits in the `T2*` / shard-10 cluster (`aebec4e`, `b077c07`, `7f07a96`, `fdc5115`, `6829a79`, `79fa3b9`, `d16c23a`, `d37c563`, `aa334c2`, `5d55c12`, `1f37b15`, etc.) each extended the preamble's `(f) through (.)` range and added a SHOULD item without backfilling a uniform disposition rule, and inconsistently included or omitted the per-item "amend" clause; `e785f77` ("Version-bump procedure six clarity/scope gaps (T061)") tightened the mandatory-recording shape ("omitting the row is non-conformant") without adding the disposition; `828b7d3` (June 16, 2026, "V18c editorial-checklist MUST/SHOULD blocking distinction missing from exit gate") added the disposition to `V18c`'s **Ships when** clause citing back to `version-bump-step2.md`, codifying on the plan side a rule the spec still does not carry.
**History:** The defect is the cumulative product of items (f)–(ak) being added one or two at a time over ~12 commits between June 2026 and the current pin, each touching only its own item and the preamble's range terminator. No single commit introduced the gap; no single commit ever owned the preamble's disposition slot. The plan-side resolution (`828b7d3`) closed the V18c exit gate by naming a disposition, but did so by reading a rule into `version-bump-step2.md` that never landed there.

## Solution Space

**Shape:** multiple

### Option A — Backfill preamble disposition matching `V18c` (recommended)

**Approach.** Add a single disposition sentence to the step-2 preamble (immediately after the existing mandatory-recording sentence) stating: a recorded `fail` on a MUST-level item `(a)`–`(e)` blocks the bump per item (e)'s merge-gate prose (item (e)'s second-pass clearance is the worked example), whereas a recorded `fail` or `not-performed` on a SHOULD-level item `(f)`–`(ak)` is a conformant non-blocking outcome — the row stands in the commit message as the disposition record. Leave the per-item "amend in the same edit" clauses on `(g)`, `(j)`, `(q)`, `(v)`, `(ad)`, `(ah)` in place as item-specific tightenings of the SHOULD-level non-blocking baseline (they describe what the contributor *does* alongside recording the `fail`, not what blocks the merge).

**Spec edits.** One preamble sentence in `version-bump-step2.md` (after the "in the bump commit message:" line). Optionally normalise the per-item "amend" clauses to a single shared phrase or extract them to the **Outcome recording conventions** block so the asymmetry is visible at a glance. No `V18c` edit needed — its **Ships when** cite resolves cleanly.

**Pros.** Matches the plan-side acceptance criterion already shipped in `V18c`. Smallest spec surface change. Preserves existing per-item amendment obligations.

**Cons.** Leaves the asymmetric per-item "amend" clauses readable as ambiguous (the contributor still cannot tell from the SHOULD baseline which items expect in-same-edit amendment on `fail` without scanning every item). The cross-axis "non-blocking but amend in the same edit" framing is subtle.

**Risks.** Future SHOULD items may forget the per-item amendment clause where one is warranted, recreating the asymmetry.

### Option B — Generalise "amend in the same edit" across all SHOULD items

**Approach.** Lift the per-item "amend in the same edit" clause into the preamble as the default disposition for every SHOULD-level item: a recorded `fail` is non-blocking provided the cited paragraph is amended in the same edit as the bump (or a defect-record is added that explicitly defers the amendment). Delete the asymmetric per-item clauses in `(g)`, `(j)`, `(q)`, `(v)`, `(ad)`, `(ah)` since the preamble now states the rule.

**Spec edits.** New preamble disposition sentence; deletion of six per-item "amend" clauses; matching wording adjustment to `V18c`'s **Ships when** to include the in-same-edit amendment expectation alongside the non-blocking framing.

**Pros.** Uniform disposition. Eliminates the asymmetric boilerplate the sibling finding *"items (f)–(ak) repeated escalation boilerplate (~31×) / no triage"* flags. Stronger spec posture: the bump never lands with a known-falsified presupposition still cited in unchanged prose.

**Cons.** Materially broader scope: every existing SHOULD item without an "amend" clause now carries an implicit amendment obligation on `fail` that its author did not write. Requires re-checking every cited paragraph to confirm "amend in the same edit" is even meaningful (some items cite invariants that have no single-paragraph anchor to amend). Forces a `V18c` rewording.

**Risks.** Some SHOULD-level `fail` cases legitimately want deferred amendment (e.g. a provider-fixture refresh blocked on credentials access per the sibling *"provider-fixture re-capture assumes live credentials/network"* finding); a blanket in-same-edit rule pushes contributors toward `not-performed` to avoid the obligation, degrading the audit signal.

### Recommendation

Option A. Address the smaller, scope-bounding obligation first: a one-sentence preamble disposition that names the plan-side rule the spec is already cited as carrying. This closes the correctness gap with no side effects on adjacent items, leaves the asymmetric per-item "amend" clauses available for the cleanup that the sibling boilerplate finding owns, and keeps the `V18c` cite valid without a coordinated plan edit. Implementer edge cases: place the new sentence after the existing "omitting the row is non-conformant" clause and before the colon that opens the items list, so the disposition reads as a continuation of the recording schema; cite item (e)'s merge-gate prose as the worked MUST example so the new sentence does not duplicate (e)'s own gate; do not weaken (e)'s `fail` arm — the recovery-serialisation second-pass clearance is part of the MUST disposition the new sentence summarises.

## Consequence

**Severity:** correctness

Two reasonable contributors record a SHOULD-level `fail` and reach incompatible merge conclusions: one merges the bump with the `fail` row preserved in the commit message, the other blocks the bump and demands a spec amendment land in the same edit. The plan-side acceptance criterion (`V18c` **Ships when**) silently relies on a disposition the spec never states, so a strict reading of the spec leaves the contributor with no rule to follow at exactly the point the editorial review is supposed to terminate.
