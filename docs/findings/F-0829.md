## Problem

The step-2 editorial-review checklist preamble mandates that every item record a `pass` / `fail` / `N/A` / `not-performed` outcome, and item (e) builds an explicit merge gate on its `fail` arm. The SHOULD-level items `(f)`–`(ak)` carry no analogous disposition: the preamble never states whether a recorded `fail` on a SHOULD-level item blocks the bump, and only some items (`(g)`, `(j)`, `(q)`, `(v)`, `(ad)`, `(ah)`) close with an "amend in the same edit" clause while the rest are silent. Two contributors auditing the same SHOULD-level `fail` therefore reach incompatible merge conclusions. The plan-side `V18c` **Ships when** clause already cites `version-bump-step2.md` for a MUST-blocks / SHOULD-non-blocking disposition that the spec does not actually state.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e89e42e` (June 2026, "T21 Pi-side slash-handler promise lifecycle taken as given" — first SHOULD-only item, `(g)`, with per-item "amend in the same edit" clause but no preamble disposition); subsequent SHOULD-item-adding commits in the `T2*` / shard-10 cluster (`aebec4e`, `b077c07`, `7f07a96`, `fdc5115`, `6829a79`, `79fa3b9`, `d16c23a`, `d37c563`, `aa334c2`, `5d55c12`, `1f37b15`, etc.) each extended the preamble's `(f) through (.)` range and added a SHOULD item without backfilling a uniform disposition rule, and inconsistently included or omitted the per-item "amend" clause; `e785f77` ("Version-bump procedure six clarity/scope gaps (T061)") tightened the mandatory-recording shape ("omitting the row is non-conformant") without adding the disposition; `828b7d3` (June 16, 2026, "V18c editorial-checklist MUST/SHOULD blocking distinction missing from exit gate") added the disposition to `V18c`'s **Ships when** clause citing back to `version-bump-step2.md`, codifying on the plan side a rule the spec still does not carry.
**History:** The defect is the cumulative product of items (f)–(ak) being added one or two at a time over ~12 commits between June 2026 and the current pin, each touching only its own item and the preamble's range terminator. No single commit introduced the gap; no single commit ever owned the preamble's disposition slot. The plan-side resolution (`828b7d3`) closed the V18c exit gate by naming a disposition, but did so by reading a rule into `version-bump-step2.md` that never landed there.

## Solution approach

Add a disposition statement to the step-2 preamble distinguishing the MUST-level items `(a)`–`(e)` — where a recorded `fail` blocks the bump, with item (e)'s merge-gate prose as the worked example — from the SHOULD-level items `(f)`–`(ak)`, where a recorded `fail` or `not-performed` is a conformant non-blocking outcome recorded in the bump commit message. Leave the per-item "amend in the same edit" clauses on `(g)`, `(j)`, `(q)`, `(v)`, `(ad)`, `(ah)` in place as item-specific tightenings of the SHOULD-level non-blocking baseline. This aligns the spec with the disposition the plan-side `V18c` **Ships when** clause already cites.

## Solution constraints

- Do not weaken item (e)'s `fail`-arm merge gate (the recovery-serialisation second-pass clearance) when summarising the MUST disposition.
- Out of scope: normalising or removing the per-item "amend in the same edit" clauses — that cleanup is owned by the items `(f)`–`(ak)` escalation-boilerplate sibling finding.
