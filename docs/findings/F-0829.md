## Finding

The step-2 editorial-review-checklist preamble mandates that the contributor record a per-item outcome — `pass` / `fail` / `N/A` / `not-performed` with a one-line rationale — for every item `(a)`–`(an)`, and item `(e)` carries an explicit merge gate on its `fail` arm (the recovery-serialisation second-pass clearance). The SHOULD-level items `(f)`–`(an)` carry no analogous merge-gate disposition: the preamble never states whether a recorded `fail` on a SHOULD-level item blocks the bump or is conformant non-blocking, and the item bodies are asymmetric — `(g)`, `(j)`, `(q)`, `(v)`, `(ad)`, `(ah)` close with an "amend the cited paragraph in the same edit" clause while the remaining ~26 SHOULD-level items are silent on what a `fail` means procedurally.

The disposition is in fact already codified, but only on the plan side: `V18c`'s **Ships when** clause states that "a `fail` on any MUST-level item `(a)`–`(e)` blocks the bump, whereas a recorded `fail` / `not-performed` on a SHOULD-level item `(f)`–`(al)` is a conformant non-blocking outcome, per `version-bump-step2.md`." That citation reads a rule into `version-bump-step2.md` that the spec page does not actually carry, so two contributors reading only the spec page reach incompatible merge conclusions on the same SHOULD-level `fail`, and any future divergence between the spec and the plan-side gate goes uncaught.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e89e42e` (June 2026, "T21 Pi-side slash-handler promise lifecycle taken as given" — first SHOULD-only item, `(g)`, with per-item "amend in the same edit" clause but no preamble disposition); subsequent SHOULD-item-adding commits in the `T2*` / shard-10 cluster (`aebec4e`, `b077c07`, `7f07a96`, `fdc5115`, `6829a79`, `79fa3b9`, `d16c23a`, `d37c563`, `aa334c2`, `5d55c12`, `1f37b15`, etc.) each extended the preamble's `(f) through (.)` range and added a SHOULD item without backfilling a uniform disposition rule, and inconsistently included or omitted the per-item "amend" clause; `e785f77` ("Version-bump procedure six clarity/scope gaps (T061)") tightened the mandatory-recording shape ("omitting the row is non-conformant") without adding the disposition; `828b7d3` (June 16, 2026, "V18c editorial-checklist MUST/SHOULD blocking distinction missing from exit gate") added the disposition to `V18c`'s **Ships when** clause citing back to `version-bump-step2.md`, codifying on the plan side a rule the spec still does not carry.
**History:** The defect is the cumulative product of items (f)–(ak) being added one or two at a time over ~12 commits between June 2026 and the current pin, each touching only its own item and the preamble's range terminator. No single commit introduced the gap; no single commit ever owned the preamble's disposition slot. The plan-side resolution (`828b7d3`) closed the V18c exit gate by naming a disposition, but did so by reading a rule into `version-bump-step2.md` that never landed there.

## Solution Space

**Shape:** single

### Recommendation

Add a single MUST-level disposition sentence to the step-2 editorial-review-checklist preamble in `version-bump-step2.md`, immediately after the existing per-item recording-shape paragraph (the one ending "omitting the row is non-conformant; the (f)–(an) audits stay SHOULD-level, but the per-item recording is mandatory either way"). The sentence states:

> A recorded `fail` on any MUST-level item `(a)`–`(e)` blocks the bump (item `(e)` failing on either sub-outcome `(e.i)` / `(e.ii)`, including a non-positive verdict the conservative-posture rule records as `fail`; item `(e)`'s `fail`-arm merge gate — clearance via the recovery-serialisation second-pass outcome — is the worked example). A recorded `fail` or `not-performed` on a SHOULD-level item `(f)`–`(an)` is a conformant non-blocking outcome; the contributor records it on the bump commit message and proceeds. The per-item "amend the cited consumption-posture paragraph in the same edit" clauses already carried by items `(g)`, `(j)`, `(q)`, `(v)`, `(ad)`, `(ah)` are item-specific tightenings of this SHOULD-level non-blocking baseline, not exceptions to it. Any future SHOULD-level item added to this checklist inherits this same non-blocking disposition unless the item body declares an explicit tightening.

Place this sentence in the preamble (not in `Outcome recording conventions`) so a reader hits it before descending into the item list, and key the wording to the current `(f)`–`(an)` range terminator (the range-terminator fix is owned by its own sibling finding; whichever lands second mechanically refreshes the other). Edge cases the implementer must watch: keep item `(e)`'s existing `fail`-arm merge-gate prose intact rather than collapsing it into the new sentence (the recovery-serialisation second-pass clearance is item-specific and must remain locally authoritative); leave the per-item "amend in the same edit" clauses untouched (their normalisation belongs to the escalation-boilerplate sibling finding); after landing, audit `V18c`'s and `V18c-T`'s **Ships when** clauses to point at the new preamble anchor rather than at the file as a whole, and confirm the `(f)`–`(al)` / `(f)`–`(an)` range agreement.

## Consequence

**Severity:** correctness

Two reasonable contributors auditing the same SHOULD-level `fail` reach incompatible merge conclusions reading only the spec. The plan-side `V18c` **Ships when** clause is the de-facto authority, but it is reached only by a contributor who follows the bump procedure into the plan; a contributor working from `version-bump-step2.md` alone either blocks the bump conservatively (over-strict, against the intended posture) or merges with no recorded `fail`-handling rationale (under-strict, indistinguishable from a missed audit). The plan-side citation back into the spec also silently goes stale if the spec's tier boundary or letter range shifts without revisiting `V18c`'s **Ships when**.
