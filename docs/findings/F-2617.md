## Problem

The best-effort `loom-system-note` emission model in `runtime-event-channel.md` rests entirely on one host property: `pi.sendMessage` returns `void` synchronously and never returns a rejecting `Promise`. The no-`await`/no-`.catch` rule, the swallow-synchronous-throw fallback chain, and the exactly-once accounting all depend on it, yet the property is left un-pinned on both axes the spec otherwise uses to protect such presuppositions. It carries no concrete `.d.ts` declaration pin — it lives only as `conversation-drive.md` prose, and `capability-inventory-items.md` item 6 (`#sdk-cap-custom-message-renderer`) names the call shape but not the return type — and no entry in the version-bump editorial-review checklist (`version-bump-step2.md`, items (a)–(as)), which carries dedicated items for directly analogous never-rejecting / void behaviours such as item (al) for `ctx.waitForIdle()`. A Pi minor that made `pi.sendMessage` async-rejecting would pass every gate while rejections escape as `unhandledRejection`s, and the absence also violates the checklist's own closing rule that any editorial-review-routed presupposition MUST be enrolled in the same edit.

## Issue introduction

**Verdict:** single-commit (introduced; remediation gap perpetuated)
**Introducing commit:** `29c29b2` (2026-05-06) — *pi-loom spec: resolve "pi.sendMessage is synchronous (void); spec treats it as returning a Promise"*
**History:** The corpus is git-tracked. `git log -S` over the defect tokens shows the prose "`pi.sendMessage` returns `void` (synchronous); the runtime MUST NOT `await` it and MUST NOT attach a `.catch` handler" was introduced verbatim by `29c29b2`, the commit that flipped the model from treating `pi.sendMessage` as a `Promise` to treating it as synchronous `void`. That commit added the behavioural assertion but no `.d.ts` declaration pin and no version-bump re-validation hook. Two later corpus-wide sweeps closed the analogous gap for sibling surfaces but passed over this one: `158ed09` (2026-06-05, *Missing owning-declaration pins for bare Pi SDK symbols*) added `.d.ts` declaration pins for bare Pi symbols, and `257c545` (2026-06-05, *Host presuppositions lack version-bump-procedure re-audit hooks*) added the editorial-review checklist hooks for host presuppositions — neither touched `pi.sendMessage`'s void / never-rejecting property (their diffs reference `pi.sendMessage` only incidentally). The defect therefore dates to the introduction of the synchronous-void model and survived the two passes that established the very pin-and-enroll discipline it is now missing against.

## Solution approach

At the no-`await` rule in `runtime-event-channel.md` and on capability 6 (`#sdk-cap-custom-message-renderer`) in `capability-inventory-items.md`, add a declaration citation for `pi.sendMessage`'s signature and `void` return naming its `.d.ts` declaration site at the loom 1.0 Pi-SDK pin (`host-prerequisites.md#pi-sdk-pin`), the same footing the spec already gives sibling Pi surfaces. Add one editorial-review item to the (a)–(as) list in `version-bump-step2.md`, modelled on item (al) (`#bump-checklist-waitforidle-no-rejection`), confirming against each candidate Pi minor that `pi.sendMessage` still returns `void` and never a rejecting `Promise`, anchored to the `runtime-event-channel.md` best-effort model.

## Solution constraints

- The new bump-checklist entry MUST be an editorial-review item, not a mechanical step-2(a) `SDK_SURFACE_INVENTORY` row: the surface-inventory literal-read is presence-only and the Step-0 capability probe is `typeof`-only (PIC-4), so neither can witness a return type.
- Out of scope: the `capability-probe.md` silent-shape-drift disposition (the runtime-time outcome); the new item closes the orthogonal bump-time detection gap and is complementary, not a replacement for it.
