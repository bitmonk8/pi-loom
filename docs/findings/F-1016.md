## Problem

In `docs/spec_topics/schema-subset.md`, step 5 (`#synthesised-names`) declares itself the "source of truth for the full set" of four slug-keyed synthesised-name forms (`__inline_<slug>`, `__loom_respond_<slug>`, `__loom_callee_<slug>__<post-rename-name>`, `__loom_bind_<slug>`). The immediately following `#schema-slug-collision-posture` block states the universal byte-verify no-aliasing rule and enumerates "the three slug-keyed sites" with their diagnostics. Two of the four name forms — `__loom_respond_<slug>` and `__loom_bind_<slug>` — are not bound to any of the three enumerated sites by name, so the no-aliasing contract is unstated for them. The "three slug-keyed sites" count also contradicts the four-name-form set declared one paragraph above.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**

- `61592d7` (2026-06-05) — *pi-loom spec: resolve "Binder structured-output tool name and label undefined"* — added `__loom_bind_<slug>` to step 5's enumeration, taking the count from 2 to 3.
- `c577ab3` (2026-06-06) — *pi-loom spec: resolve "Glossary schema slug name enumeration cannot be complete"* — added `__loom_callee_<slug>__<post-rename-name>` and the normative "source of truth for the full set" phrasing, taking the count to 4.
- `2128e5f` (2026-06-07) — *pi-loom spec: resolve "Schema-slug collision posture pinned only for registerTool cache"* — introduced the "Schema-slug collision posture" block with the universal byte-verify rule and the "three slug-keyed sites" enumeration.

**History:** Each commit individually resolves a prior review finding, and each is internally consistent with the wording in place at the time it landed. The defect is the cross-edit drift between them: `2128e5f`'s "three slug-keyed sites" count was authored against the *site* inventory (inline-dedup, validator cache, registerTool cache) rather than the *name-form* inventory, and it landed one day after `c577ab3` had finalised the four-form enumeration directly above it. Neither commit cross-checks the other; the gap between "full set of four" and "three slug-keyed sites" only becomes legible when both are read in the post-2128e5f order. `61592d7` is implicated because it is the commit that first made the name-form count diverge from the (then-future) site count, but the defect is not localised to any single commit — it is the unreconciled juxtaposition produced by the three together.

## Solution approach

In the `#schema-slug-collision-posture` registration-cache bullet, clarify that the registration cache covers both `__loom_callee_<slug>__<post-rename-name>` and `__loom_respond_<slug>`, cross-linking `tool-registration-lifetime.md` (which already pins the `__loom_respond_<slug>_<n>` counter-suffix disambiguation). Add a statement that `__loom_bind_<slug>` is non-cached — attached as a single per-`complete()`-call `context.tools` entry with no slug-keyed cache or dedup table — cross-linking `binder-inference.md`. Narrow the enumeration lead away from the fixed "three slug-keyed sites" count to count-free phrasing so it no longer contradicts step 5's four-name-form `#synthesised-names` set. Update plan leaf `V9f` Tests to name both forms the registration-cache diagnostic disambiguates.

## Solution constraints

- Do not coin a second registration-cache diagnostic for the respond form; the existing `loom/runtime/registration-cache-collision` is shared by design, and a respond-specific code would violate the "originating cache identifiable from the code alone" intent.
- Do not expand the binder tool's lifetime to a cross-call cache while making the non-cached claim normative; any such change is an independent decision owned by the binder pages, out of scope here.
