## Finding

PIC-10's Rules list pins a normative prohibition — *"Implementations MUST NOT add `before(...)` calls at any site not enumerated above."* The rule states what is forbidden (a checkpoint observation at any node kind outside the enumerated five: loop-iteration boundary, `@`-query, tool call, `invoke`, binder LLM call) but supplies no acceptance criterion a conformance test can assert against. The only nearby observability statement — *"Test wiring is the only consumer that observes calls"* — describes who watches the seam, not what a passing test must prove about the call set.

The gap is concrete: nowhere in the cluster is the total `before(kind, site)` call count per canonical test script enumerated, so a counting fake has no spec-stated expected value to compare against and would have to re-derive expected counts from script structure — exactly the kind of independent re-derivation that defeats a conformance check. The two sibling seams that coin structurally identical "ambient surface" bans both close the gap explicitly: PIC-12's timing-primitive ban and PIC-13's `process.env`/`process.cwd()` ban each mandate *"A conforming implementation MUST carry a build-time check that rejects any direct lexical reference"* and name that check as the conformance criterion. PIC-10's MUST NOT is the odd one out — a forbidden behaviour with no stated pass/fail gate.

Note the verification shape differs from PIC-12/PIC-13: their bans are static-lexical (grep/lint-detectable absence of a token), whereas PIC-10's prohibition is a runtime/structural property (the *set of sites that fire* `before`), so the criterion must be observation-based — a per-script enumerated call count asserted by the test `Checkpoint` fake — rather than a build-time lexical scan.

**Introducing commits:** `033b8e3` (2026-05-05) — *pi-loom spec: resolve "Race semantics: no injectable seam to land an abort between an Ok return and the next checkpoint"*
**History:** `git log -S` over `host-interfaces-services.md` for both the MUST NOT clause (`"MUST NOT add"`, `"at any site not enumerated"`) and the `"Test wiring is the only consumer"` rule resolves to a single authoring commit, `033b8e3`, which introduced the entire `Checkpoint` seam (then resident in `pi-integration-contract.md`). `git show 033b8e3` confirms both bullets entered together in that commit. The seam was later relocated into `host-interfaces-services.md` by the size-cap split `f5e89f4`, but `git log -L` shows the MUST NOT line was never substantively revised after introduction. The missing acceptance criterion is therefore co-extant with the rule from its first authoring — the seam shipped with a verification-criterion gap that the parallel PIC-12 criterion (added later via `F-1178`/`4556623`) was never extended to cover.

## Solution Space

**Shape:** single

### Recommendation

In PIC-10, immediately after the *"Test wiring is the only consumer that observes calls"* bullet, add an observation-based conformance criterion: a conforming implementation MUST verify, for each canonical test script in the cancellation suite, that the total number of `before(kind, site)` calls observed by the test `Checkpoint` fake equals the count of enumerated checkpoint events for that script — one `before` per enumerated checkpoint kind per loop iteration / `@`-query / tool call / `invoke` / binder LLM call in the script's execution trace. A surplus call is a conformance failure.

State the criterion as an observed-call-count equality, not a build-time lexical scan: unlike PIC-12/PIC-13, the forbidden thing here is a runtime checkpoint firing at an un-enumerated site, which a static token scan cannot see. Leave the count derivation anchored to the **Granularity** enumeration in `cancellation.md` so the expected value is spec-traceable rather than re-derived by the test author. Edge case for the implementer: the `loop-iter` macrotask-yield path still emits exactly one `before` per iteration (the yield is in resolution, not an extra call), and `invoke` parent/child each own a separate `Checkpoint` instance — the count is asserted per instance, not summed across the parent/child pair.

## Consequence

**Severity:** advisory

The behavioural rule itself is unambiguous, so implementers will not diverge on runtime behaviour; the loss is purely in conformance coverage. The plan's existing "no extra sites" assertion is best-effort and self-admittedly cannot catch a checkpoint emitted at a node kind outside the test corpus, so a regression that adds a spurious `before(...)` call at an un-driven site can pass the suite undetected.
