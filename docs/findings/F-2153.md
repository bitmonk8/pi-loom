## Problem

The frontmatter Field-contract table defines a type/value load diagnostic for every recognised **scalar** field carrying the wrong shape (`mode:`, `bind_context:`, `model:`, `respond_repair.methodology:`, and the `tool_loop.max_rounds` / `respond_repair.attempts` range rule), but no rule covers the inverse: a recognised **block-valued** field (`respond_repair`, `tool_loop`, `params`) present with a non-mapping YAML value, or `system:` present with a non-string value. Concretely `respond_repair: 3`, `tool_loop: "x"`, `params: "foo"`, `params: [..]`, and `system: 42` name a recognised key with a wrong container type, yet the spec pins no detection or surfacing. Conforming implementations therefore diverge — rejecting at load, silently coercing, treating the field as absent and applying the default, or crashing on member access. The settings side already closes this gap via `loom/load/settings-value-out-of-range` (type and range); the frontmatter side carries only the range half.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** none singularly — the omission has existed since the frontmatter field contract was first authored (the corpus began at `9a53d9c`, 2026-05-01; the contract has carried per-field type rules since early May 2026).
**History:** The corpus is git-tracked. The scalar-field type/value diagnostics accreted incrementally and each addressed a single scalar surface — e.g. `loom/load/frontmatter-value-out-of-range` entered on 2026-05-07 (`24a8079`, `36cc8d9`) for the `tool_loop`/`respond_repair` integer leaves, with the `mode:`, `bind_context:`, `model:`, and `methodology:` value-splits added in later commits. No commit ever authored a parallel rule for the block-valued fields' container type or for `system:` being non-string: a `git log -S` across all history finds the strings `non-mapping` and `present with a non-string` never appear in any revision. The defect is thus an inception-era omission that survived every subsequent scalar-rule addition, not a regression introduced by a specific later edit.

## Solution approach

Widen `loom/load/frontmatter-value-out-of-range` to cover container-type mismatches — a recognised block-valued field (`respond_repair`, `tool_loop`, `params`) present with a non-mapping value, and `system:` present with a non-string value — stating the load-time rejection in `frontmatter-fields-a.md`'s Field-contract table or adjacent prose, mirroring the dual type-and-range role `loom/load/settings-value-out-of-range` plays on the settings side. Reconcile the `loom/load/frontmatter-value-out-of-range` row in `code-registry-load.md`, whose current scope ("a frontmatter scalar covered by the non-negative-integer rule") the widening contradicts. Extend the `<observed>` rendering rule in `placeholder-rendering-b.md` so a non-scalar offending value renders by its kind, mirroring the existing `settings-value-out-of-range` `array` / `object` cases.

## Solution constraints

- The existing exclusions stay intact: `respond_repair: {}`, `tool_loop: {}`, and `params: {}` remain equivalent to absence and must not fire the widened rule, and `params: null` keeps its dedicated `loom/load/params-null` code.
- Per DIAG-4 the *Message* column is normative and test-asserted, so the registry row, the `V7b` gate, and any asserting tests must change in lockstep with the field-contract widening.
