## Finding

`V5f` (Schema lowering and canonical hash) lists "per-query `$defs` pruning" in its **Adds** as one responsibility of its load-time lowering pass, but no `V5f`/`V5f-T` Tests bullet asserts it and the `V5f` **Ships when** clause omits it (it enumerates only the canonical-hash recipe, slug-collision detection, and the per-schema sidecar two-map shape).

Per-query `$defs` pruning is **Lowering Algorithm step 4** in [`schema-subset.md`](../../docs/spec_topics/schema-subset.md): "*Per-query schema document is built lazily: when a typed query fires, the runtime extracts the query's response schema as the document root and copies in only the `$defs` entries transitively reachable from it. Unused `$defs` are pruned…*". This is explicitly a **query-time / runtime** operation, distinct from the load-time lowering pass (steps 2, 3, 5, 6), which the spec describes as "*performed once per loom-file load … purely a function of the parsed source (no runtime values)*".

That step-4 obligation is already owned and asserted elsewhere: `V13b` (Query schema inference) carries a Tests bullet citing *Lowering Algorithm step 4 (per-query `$defs` pruning)* — "*a typed query's request schema document contains only the `$defs` transitively reachable from its response-schema root; unreachable `$defs` are pruned*" — and `V13b`'s **Ships when** asserts "*per-query `$defs` pruning to the reachable definitions*". `V13b-T` authors the same failing test. So the observable pruning behaviour does **not** ship unverified.

The defect is therefore an attribution/duplication error, not a coverage gap: `V5f`'s Adds bundles a query-time step-4 obligation into its description of the load-time lowering pass, while the obligation is owned and tested by `V13b`. An implementer reading only `V5f`'s Adds could build pruning logic into the load-time lowering pass, duplicating or diverging from `V13b`'s query-time implementation, and the Adds text itself misstates spec step 4 (which makes pruning lazy/query-time).

## Solution Space

**Shape:** single

### Recommendation

Strike the `per-query `$defs` pruning` item from `V5f`'s **Adds** field. The Adds currently lists, within its description of the load-time lowering pass: "…the per-schema two-map sidecar (…) whose shared type `V5f` owns as the seam `V2e` consumes, **per-query `$defs` pruning**, and the canonical-hash → 16-hex-slug recipe…". Remove that one list item so the Adds describes only the load-time responsibilities `V5f` actually owns and tests: the `__inline_<slug>` hoist, auto `$defs`/`$ref`, the per-schema two-map sidecar, and the canonical-hash → 16-hex-slug recipe.

Per-query `$defs` pruning is `schema-subset.md` Lowering Algorithm step 4 — a query-time operation owned and asserted by `V13b`/`V13b-T`. If a pointer is wanted in `V5f`, attribute the step to `V13b` rather than claiming it as a `V5f` lowering-pass output. No `V5f` Tests bullet or Ships-when change is required, and `V13b`/`V13b-T` need no change (they already cite step 4 and assert reachable-only pruning).

## Consequence

**Severity:** advisory

If shipped unfixed, a `V5f` implementer reading the Adds claim may build per-query `$defs` pruning into the load-time lowering pass, duplicating or diverging from `V13b`, which owns the step-4 query-time pruning and already asserts it; the Adds text also misstates spec step 4 (lazy/query-time, not load-time). No coverage gate fails — `V13b-T` asserts the reachability/pruning property — so this is a guidance/attribution defect, not an unmet closing obligation.
