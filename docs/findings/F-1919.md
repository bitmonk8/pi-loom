## Problem

`V5f`'s **Adds** lists "per-query `$defs` pruning" as one output of its load-time lowering pass, but that is `schema-subset.md` Lowering Algorithm step 4 — an explicitly query-time/runtime operation, distinct from the load-time lowering pass. The step-4 obligation is already owned and asserted by `V13b`/`V13b-T`, which cite step 4 and assert reachable-only pruning; no `V5f` Tests bullet asserts it and `V5f`'s **Ships when** omits it. An implementer reading only `V5f`'s Adds could build pruning into the load-time lowering pass, duplicating or diverging from `V13b`, and the Adds text itself misstates step 4 (which is lazy/query-time).

## Issue introduction

**Verdict:** introduced by a single commit (leaf-split carve-out)
**Introducing commits:** `cb8737a` ("pi-loom plan: resolve \"Systemic leaf over-bundling across the plan corpus\"", 2026-06-12)
**History:** `git log -S'per-query \`$defs\` pruning' -- docs/plan_topics/V5f-subset-lowering-hash.md` returns a single commit, `cb8737a`, which created the `V5f` file. At creation the Adds already listed "per-query `$defs` pruning" with no matching Tests bullet or Ships-when clause; the defect is present since the leaf's inception with no earlier clean `V5f` revision. The step-4 per-query pruning obligation was already owned and asserted by `V13b`/`V13b-T` as of `ba7da00` (2026-06-10, "resolve \"V5d lowering-pass outputs bound by no Tests bullet\""), which predates the `V5f` carve-out — so the over-bundling split copied an obligation into `V5f`'s Adds that another leaf already tested.

## Solution approach

Delete the "per-query `$defs` pruning" item from `V5f`'s **Adds** so the field describes only the load-time responsibilities `V5f` owns and tests. If a pointer is wanted, attribute step-4 pruning to `V13b` rather than claiming it as a `V5f` lowering-pass output.

## Solution constraints

- Out of scope: per-query `$defs` pruning (Lowering Algorithm step 4) is owned and asserted by `V13b`/`V13b-T` — do not add a duplicating Tests bullet or Ships-when clause to `V5f`.
- The spec is read-only for this fix.
