## Finding

The §Doc updates rule in `conventions.md` instructs each leaf to stage `README.md`, `CHANGELOG.md`, and `notes.md` explicitly with `git add`, and justifies that instruction by asserting as fact that "the orchestrator's `commitAddPaths` (`.pi/project-config.md`) auto-stages only `docs/`." The asserted value is correct today — `.pi/project-config.md` declares `commitAddPaths: [docs/]` — but the plan states the value inline as a load-bearing premise without it being self-evident from the plan text, so the rationale is coupled to a value that lives in a separate orchestrator-config file.

The procedural instruction itself (stage the three root files explicitly) is sound regardless of the auto-stage scope: an explicit `git add` of repository-root files is correct whether or not the orchestrator also auto-stages them. Only the *justification clause* depends on the asserted scope. If `commitAddPaths` is ever broadened (for example to include the root files the doc-update rule touches), the "auto-stages only `docs/`" rationale silently goes stale while the surrounding instruction keeps working, leaving a future maintainer reading a justification that no longer matches the config.

Before `a9f8409` the §Doc updates rule ended at the artifact-shape descriptions; that commit appended the clause asserting that `commitAddPaths` "auto-stages only `docs/`, so the leaf stages `README.md`, `CHANGELOG.md`, and `notes.md` explicitly (`git add`)." The asserted-value coupling therefore entered the corpus with the edit that resolved the earlier "commitAddPaths omits root files" review finding: that resolution added the explicit `git add` guidance and justified it by restating the external config value inline, rather than phrasing the guidance to stand independent of the auto-stage scope.

## Solution Space

**Shape:** single

### Recommendation

In `conventions.md` §Doc updates, reword the justification clause so the explicit-staging instruction stands on its own and does not assert a specific `commitAddPaths` scope as its load-bearing premise. Replace the current clause —

> These three artifacts live at the repository root, outside `docs/`; the orchestrator's `commitAddPaths` (`.pi/project-config.md`) auto-stages only `docs/`, so the leaf stages `README.md`, `CHANGELOG.md`, and `notes.md` explicitly (`git add`) as part of its commit.

— with text whose correctness does not depend on the auto-stage scope, e.g.:

> These three artifacts live at the repository root, outside `docs/`. Because they fall outside the orchestrator's `docs/`-scoped auto-staging (`commitAddPaths` in `.pi/project-config.md`), the leaf stages `README.md`, `CHANGELOG.md`, and `notes.md` explicitly (`git add`) as part of its commit; the explicit `git add` is correct regardless of the configured auto-stage scope.

Keep the parenthetical pointer to `.pi/project-config.md` as an informative cross-reference rather than the rule's premise. `.pi/project-config.md` is read-only for this fix — do not edit the config to match the prose; reconcile the prose to the config and to the scope-independent rationale.

## Consequence

**Severity:** advisory

If `commitAddPaths` is later broadened, the §Doc updates rationale becomes a stale assertion that contradicts the orchestrator config; a maintainer trusting it could conclude the explicit `git add` is unnecessary or mis-reason about staging. The per-leaf procedure still produces a correct commit, so no leaf is blocked — the defect is a rationale-to-config coupling that can drift unobserved.
