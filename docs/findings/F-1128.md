## Finding

The `loom 1.0 seam — automatic context escalation` blockquote on `docs/spec_topics/binder/binder-model-and-context.md` pins, with normative force, that the binder-invocation path is **re-entrant per loom turn** — every binder call constructs the input record and the resolved binder-model handle afresh, with no cached state that would prevent a second call from observing a different `bind_context` snapshot. The companion sentence ("This re-entrancy invariant is a settled loom 1.0 binder-path property in its own right …") explicitly elevates it to a "settled loom 1.0 binder-path property" detached from the extension's unsettled shape. Yet the obligation carries no `BNDR-N` ID, no `<a id="bndr-…"></a>` HTML anchor, and no inline `**BNDR-N.**` marker.

This violates the page's own REQ-ID discipline (`binder/` → `BNDR` per the prefix table on `req-id-prefix-table-active-a.md`; GOV-1's *Canonical form* requires the inline `**PREFIX-N.**` marker at the defining site of every live REQ-ID). It also creates an asymmetry with the deferred binder-refinement seam in the sibling file `binder-bypass-and-envelope.md`, where the same blockquote pattern (`> **loom 1.0 seam — binder refinement loop.**`) introduces three carriers that *do* each carry a fresh `BNDR-N` ID (BNDR-1, BNDR-2, BNDR-3) — exactly the pattern the re-entrancy invariant fails to follow.

The companion BNDR-1/2/3 carrier inventory itself is well-formed: BNDR-1/2/3 are properly indexed from `future-considerations/surface-extensions.md` (the *Binder refinement loop* bullet's `*Anchored at:*` line names "the three loom 1.0 carriers — envelope-schema three-arm discriminator, `ambiguous.candidates` field, distinct failure-arm template prefixes — are pinned with MUST there"), and the binder refinement loop sits in the *Surface extensions (loom 1.0 leaves a seam)* registry rather than the *non-goals* registry precisely because it reserves a forward-compatibility seam. Those three obligations are not the gap; the gap is the un-anchored re-entrancy MUST on the *other* binder page.

## Solution Space

**Shape:** single

### Recommendation

Coin a new `BNDR-N` (next free number is BNDR-10 — neither `BNDR-10` nor anything in `BNDR-1[0-9]` appears in the corpus today) and attach it to the re-entrancy invariant on `docs/spec_topics/binder/binder-model-and-context.md`. The carrier sentence is the existing one ("The binder-invocation path is **re-entrant per loom turn**: …, with no cached state that would prevent a second call from observing a different `bind_context` snapshot."), tightened into a stand-alone obligation paragraph immediately under the blockquote in the GOV-1 dual-form layout used by the sibling BNDR-1/2/3 carriers:

```markdown
<a id="bndr-10"></a> **BNDR-10.** The binder-invocation path MUST be re-entrant per loom turn: the binder's input record (parameter table, raw slash text, optional session-context block) and the resolved binder-model handle MUST be constructed afresh on every binder call, with no cached state that would prevent a second call from observing a different `bind_context` snapshot.
```

Edge cases for the implementer:

- The blockquote's framing prose ("seam — automatic context escalation", "settled loom 1.0 binder-path property", "not contingent on the extension's unsettled shape") stays as non-normative seam orientation; only the carrier MUST is lifted into the BNDR-10 paragraph. The blockquote remains the place where the seam's *purpose* is explained — it just stops carrying the normative obligation in implicit prose form.
- The two existing HTML anchor arms on the blockquote (`loom-1-0-seam-automatic-context-escalation`, `v1-seam-automatic-context-escalation`) stay where they are; they index the seam framing, not the carrier MUST. The BNDR-10 anchor is additional, not a replacement.
- Update `docs/plan_topics/coverage-matrix.md` *Numbered REQ-IDs* table: add a row `| BNDR-10 | \`V11b\` |` (or merge into the existing `BNDR-7, BNDR-8, BNDR-9` row if the closing leaf stays `V11b`). `V11b` is the right closer because the re-entrancy property is exercised by the same `bind_context`/transcript-renderer wiring `V11b` builds.
- Update `surface-extensions.md`'s `automatic context escalation` bullet *Carrier* line to cite the new BNDR-10 alongside the existing seam-blockquote anchor, mirroring how the *Binder refinement loop* bullet cites BNDR-1/2/3.
- Do **not** touch the BNDR-1/2/3 carriers or the *Binder refinement loop* registry placement: those are correctly anchored under *Surface extensions* and correctly indexed.

## Consequence

**Severity:** correctness

A normative MUST with no REQ-ID is opaque to the closing-gate machinery: the [`H5d`](`H5d-transitive-completeness-gate.md`) transitive-completeness gate and the [`H5e`](`H5e-un-anchored-must-gate.md`) un-anchored-MUST gate both key off the BNDR prefix at this page, and a test that regresses the re-entrancy property has no REQ-ID to cite per the *REQ-ID discipline* rule on `conventions.md`. Implementers and reviewers reading the blockquote cannot distinguish the explicitly-pinned loom 1.0 invariant from non-normative seam orientation. The downstream automatic-context-escalation extension also loses its anchoring story: the *Carrier* line on `surface-extensions.md` defensively cites this blockquote but cannot quote a BNDR-N, mirroring the very gap the [Surface extensions without a dedicated topic-page seam](docs/spec_topics/future-considerations/surface-extensions.md#surface-extensions-no-dedicated-seam) sub-bucket exists to call out.
