## Finding

H1a's `pi.extensions` Tests bullet defines a build-time literal-read assertion that reads `package.json#pi.extensions` and asserts it equals `["./extensions"]`. Its justification prose claims the assertion makes "a manifest a real Pi host cannot discover fail `H1a`'s own `npm test` rather than shipping green while the end-to-end harness imports the factory directly through a build-internal path."

The string-equality check cannot back that claim. It verifies only that the manifest *value* is `["./extensions"]`; it does not verify discoverability — that a module resolves under that directory (`extensions/index.ts`) and exports a valid `default function (pi)`. Discoverability is a property of the factory file plus the host's loader, not of the manifest string.

The gap is concrete at H1a's place in the order. H1a carries `Deps: -`; the `extensions/index.ts` factory file is pinned by H4a (`Deps: H4a` chain), so it does not exist when H1a runs. The literal-read test therefore passes green for the *correct* manifest value at a point where a real Pi host still cannot discover any factory. The bullet's promised property ("real Pi host cannot discover → red") does not hold; the test only catches a wrong/absent manifest value.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H1a-scaffold-and-toolchain.md`, in the `pi.extensions` literal-read Tests bullet, restate the consequence clause to the property the equality check actually delivers. Replace the clause "so a manifest a real Pi host cannot discover fails `H1a`'s own `npm test` rather than shipping green while the end-to-end harness imports the factory directly through a build-internal path" with wording scoped to the manifest value — e.g. "so a `pi.extensions` value other than `["./extensions"]` (absent, a file path, or any other directory) fails `H1a`'s own `npm test`." Leave factory-resolution / discoverability to the `extensions/index.ts` file pin owned by H4a and the green `npm install` witness in *Ships when*; do not assert in H1a that the gate proves a real host can load the factory.

Keep the existing sentence that the manifest directory value MUST stay consistent with the `extensions/index.ts` factory file path H4a pins — that is accurate and unaffected. The H4a leaf and the spec page `host-prerequisites.md §"Extension entry point"` are read-only for this fix.

## Consequence

**Severity:** advisory

A reader trusting the bullet's prose believes H1a's `npm test` gates real-host extension discoverability, when it gates only the manifest literal. No divergent H1a implementation results (the assertion itself is correct), but the false coverage claim can suppress the addition of an actual discovery/re-export gate, leaving the `extensions/index.ts` resolution path mechanically unverified until the manual H6a smoke.
