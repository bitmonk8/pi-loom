## Problem

`V15c` (Imports â€” `.warp` library files) bundles two independently-shippable concern groups under a single `Ships when` gate. The first is the import **resolution and diagnostics** path: the `Resolver` seam, `IMP-1` (`loom/load/unresolvable-warp-path`), the parse/load diagnostics (`loom/parse/warp-top-level-statement`, `loom/parse/import-non-warp-extension`, `loom/load/import-cycle` with `import-unknown-symbol` / `import-name-collision`), and the resolver success baseline. The second is **export-visibility semantics**: auto-export of every top-level `schema`/`enum`/`fn`, aliased re-export (`export { A as B } from`), and the negative plain-`import`-not-re-exported rule. Visibility observes the resolved file's declarations so it depends on resolution, but resolution and its diagnostics ship and test with no dependency on visibility â€” the two are genuinely separable, leaving 8+ separately-assertable scenarios under one gate and partial completion unobservable. The defect is granularity, not correctness (`conventions.md` Â§phase categories smallest-shippable-feature convention).

## Solution approach

Split `V15c` so the resolution/diagnostics path retains the `V15c` ID and the export-visibility semantics move to a new dependent leaf. Keep the `Resolver` seam, `IMP-1`, the parse/load diagnostics (`warp-top-level-statement`, `import-non-warp-extension`, `import-cycle`, `import-unknown-symbol`, `import-name-collision`), and the resolver success baseline in `V15c`, and split `V15c-T` to match. Add a new vertical leaf (with its paired `-T`) depending on `V15c` that owns the `imports.md` Visibility and Re-exports rules â€” auto-export of top-level declarations, aliased `export â€¦ from`, and the negative plain-`import` rule â€” and link it into `docs/plan.md` "V15 â€” Invocation and imports" after `V15c`. Add the new leaf to `H5b`'s Deps so the coverage-producing set stays transitively complete.
