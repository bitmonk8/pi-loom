## Finding

Three independent paragraphs in the host-interfaces shard impose normative
constraints on the runtime's *internal* structure — code factoring, in-memory
storage layout, and an opaque identifier prefix — whose violation no
conformance test can witness. Each is the residue of an independent decision
made on its own timeline; they are bundled here only because they share a
shape, not because they share a fix.

1. **Collision-source-set seam — structural MUST on code factoring.**
   The *loom 1.0 seam — Pi-owned subagents collision source set* blockquote
   in `host-interfaces-services.md` (anchor
   `#loom-1-0-seam-pi-owned-subagents-collision-source-set`) carries the
   sentence *"MUST be defined as a single named set inside the runtime,
   consulted by the collision check via membership test rather than
   open-coded as three separate string comparisons or a hard-coded
   `switch`."* A conformant implementation that open-codes
   `source === "prompt" || source === "extension" || source === "skill"`
   drops the same pending looms, fires the same
   `loom/runtime/cross-format-collision` diagnostics, and registers the
   same survivors as one that defines the set as a named constant. The
   MUST therefore binds GOV-18 arm (a) on a property no test can witness.
   The structurally-identical sibling MUST in the *typed-query supported
   provider set* blockquote in `conversation-drive.md` was already demoted
   to a non-normative implementation note in commit `8362824` ("Demote
   over-prescriptive and non-testable MUSTs on unobservable
   implementation structure"); this twin was missed in that pass.

2. **PIC-11 cache — MUST on validator-cache storage layout.**
   The final sentence of PIC-11's bullet on the compiled-validator cache
   reads *"The cache stores the canonical-form bytes alongside the
   compiled validator so the check is a byte comparison, not a
   re-serialisation."* The byte-equality check itself is the load-bearing
   obligation — it is what prevents a schema-slug collision from serving
   the wrong cached validator — but how the runtime arranges its memory
   to perform that check (store the bytes, re-derive them, hash them, key
   the cache by a content-addressed tuple, …) is not externally
   observable. The sentence over-prescribes a storage tactic in a section
   whose surrounding boilerplate already pins, via GOV-18 arm (a), that
   "this internal DI seam's member set is not itself binding".

3. **`loom-direct:` `toolCallId` prefix — literal asserted without a
   contract.** The *Tool execution from loom code* bullet in
   `host-interfaces-core.md` states *"`toolCallId` is a synthesised UUID
   prefixed `loom-direct:`."* The literal is asserted with no stated
   contract — neither what externally-observable property the prefix
   satisfies (uniqueness, namespace separation from Pi-issued ids,
   transcript-grep affordance, anything) nor a declaration that the
   prefix is an internal implementation choice. A reader cannot tell
   whether changing the prefix string (to `loom-code:`, to
   `<empty>`, to a non-`UUID` suffix shape) breaks any contract. Two
   sibling findings in the active store (F-0711 *prefix vocabulary
   divergence*, F-0807 *format/minting/uniqueness undefined*) already
   propose specific contracts here; this finding's lens is narrower —
   the prefix as currently written carries no observable obligation at
   all, and the spec should either say what observable property it
   satisfies or fence it as opaque.

The three defects sit on three different paragraphs in two different
files, were introduced in three different commits across roughly a
month, and their fixes do not interact mechanically. They are decomposed
into separate options below so each can resolve independently in the
per-finding fix loop.

## Solution Space

**Shape:** multiple

Three independent obligations against three different paragraphs in two
files. Resolved in sequence so each fix lands against a stable baseline
of the prior one (the orchestrator should drive them in the order A → B
→ C).

### Option A — Demote the collision-source-set structural MUST

**Approach.** Apply the same `8362824`-shaped demotion to the
*loom 1.0 seam — Pi-owned subagents collision source set* blockquote in
`host-interfaces-services.md` that was applied to the sibling
*typed-query supported provider set* blockquote in `conversation-drive.md`.
Preserve the behavioural framing (the set content `{"prompt", "extension",
"skill"}` under the loom 1.0 Pi-SDK pin; the forward-compatibility
statement that the deferred subagents extension widens it to four arms);
demote only the source-shape sentence.

**Spec edits.**
- In `docs/spec_topics/pi-integration-contract/host-interfaces-services.md`,
  in the *loom 1.0 seam — Pi-owned subagents collision source set*
  blockquote, split the existing sentence into two: (i) a behavioural
  sentence stating the collision-candidate set as `{"prompt",
  "extension", "skill"}` at the loom 1.0 Pi-SDK pin, and (ii) a trailing
  *Non-normative implementation note* recording that loom 1.0 expects
  the set to be exposed as a single named runtime constant consulted via
  membership test, framed explicitly as a maintainability convention
  rather than a conformance point.
- Preserve both `<a id="loom-1-0-seam-pi-owned-subagents-collision-source-set">`
  and `<a id="v1-seam-pi-owned-subagents-collision-source-set">` anchors
  on the blockquote label line so the three live cross-references from
  `registration-steps.md`, `host-prerequisites.md`, and
  `surface-extensions.md` resolve byte-equal under GOV-25 / GOV-27.
- Preserve the blockquote label *loom 1.0 seam — Pi-owned subagents
  collision source set* verbatim so the GOV-31 *Forward-compatibility
  seams* count in `surface-extensions.md` remains 12.

**Pros.** Verbatim precedent already in the corpus (`8362824` in
`conversation-drive.md`). Anchors and label survive. Behavioural
content stays normative.

**Cons.** Removes a sentence some readers may treat as protecting against
future-Pi widening (it does not — the named-set factoring is internal,
the forward-compat statement is the surface that actually protects
widening, and it is preserved).

**Risks.** None mechanical. Editorial only: the surrounding paragraph
phrasing must keep the GOV-31 seam-count invariant tight; the blockquote
label is what GOV-31 counts on, and the label does not change.

### Option B — Demote the PIC-11 cache storage sentence

**Approach.** Delete or demote the trailing sentence *"The cache stores
the canonical-form bytes alongside the compiled validator so the check
is a byte comparison, not a re-serialisation."* The preceding
byte-equality obligation — the runtime MUST verify byte-equality on a
cache hit and recompile on mismatch — is the externally-observable
contract and stays normative.

**Spec edits.**
- In `docs/spec_topics/pi-integration-contract/host-interfaces-services.md`,
  PIC-11 §SchemaValidator interface, in the bullet beginning *"The
  runtime's per-query compiled-validator cache is keyed by the schema
  slug…"*, delete the trailing storage-tactic sentence. Optionally
  retain it as a `*Non-normative implementation note.*`-prefixed sentence
  at the end of the bullet if the implementation hint is judged useful;
  the existing PIC-12 *Non-normative implementation note* on the
  grep-test ban is the in-corpus shape to follow.
- Leave the byte-equality MUST and the `validator-cache-collision`
  emission unchanged.

**Pros.** Smallest possible edit — one sentence. Preserves the
externally-observable contract (byte-equality, recompile,
`validator-cache-collision` emission). Aligns the bullet with PIC-11's
own GOV-18-arm-(a) boilerplate ("this internal DI seam's member set is
not itself binding").

**Cons.** Loses an implementation hint a reader may find useful. The
*Non-normative implementation note* retention sub-option mitigates this
cleanly.

**Risks.** None observable. The plan-side `V8c` *Ships when* clause
already asserts the byte-mismatch emission, not the storage tactic.

### Option C — Pin a contract for the `loom-direct:` prefix, or fence it as opaque

**Approach.** Choose one of two sub-options:

- *C1 (pin a contract).* Replace the bullet
  *"`toolCallId` is a synthesised UUID prefixed `loom-direct:`"* with a
  three-clause obligation on format, minting seam, and uniqueness scope,
  modelled on the `invocationId` precedent at PIC-20 / active-invocation-
  registry. This is the shape F-0807 proposes, and F-0711's optional
  prefix rename (`loom-direct:` → `loom-code:`) can land additively
  against it. The contract is observable: a UUID-shape regex test, a
  test that the call site never reaches an ambient `crypto.randomUUID`,
  and an assertion of per-call distinctness.
- *C2 (fence as opaque).* Replace the bullet with *"`toolCallId` is a
  runtime-internal opaque string the loom 1.0 spec imposes no
  observable contract on; consumers MUST NOT depend on its lexical
  shape or prefix."* Pi's `tool_execution_*` correlation keys on the
  string identity, not on its shape, so this is implementable;
  uniqueness becomes a private invariant the runtime enforces by
  construction without spec witness.

**Spec edits.**
- In `docs/spec_topics/pi-integration-contract/host-interfaces-core.md`,
  *Tool execution from loom code*, edit the `toolCallId` bullet per C1
  or C2.
- In `docs/spec_topics/tool-calls.md`, *loom 1.0 seam — per-call
  timeout* blockquote, update the parenthetical *"the synthesised
  `loom-direct:` `toolCallId`"* to match the chosen sub-option (point
  at the pinned contract under C1, drop the prefix mention under C2).
- Under C1 only, plan-side `V14a` *Ships when* grows a regex /
  distinctness test bullet; under C2, `V14a` MUST NOT grow such a test.

**Pros.** C1 closes F-0711 and F-0807 in one move and makes the prefix
a real contract. C2 closes the same gap with one sentence and stops the
prefix from looking like normative content it is not.

**Cons.** C1 is the larger edit and re-opens the rename debate
(`loom-direct:` vs `loom-code:`) that F-0711 articulates separately. C2
is editorially simpler but discards the transcript-grep affordance the
prefix provides for operators.

**Risks.** C1 must be sequenced against F-0711 if both land — resolve
the rename first so this fix picks up the renamed prefix string. C2
breaks any existing operator workflow that greps logs for `loom-direct:`;
because loom 1.0 is pre-release this is moot today but worth recording.

### Recommendation

Resolve in the order **A → B → C**. The three obligations do not
interact mechanically, but sequencing smaller and stand-alone-clean
edits first gives the fix loop progressively narrower diffs.

- *Option A first* because the precedent (`8362824`) is already in-corpus
  byte-equal and the edit is mechanical: split one sentence, keep both
  anchors and the GOV-31-counted label. No downstream coordination.
- *Option B second* because it is a single-sentence delete (or one
  sentence demoted with a *Non-normative implementation note* prefix)
  in a different bullet of the same file Option A just touched, so the
  surface is hot and the diff is small.
- *Option C last* because it is the option with a downstream coupling
  (to F-0711 and F-0807) and the option whose sub-choice (C1 vs C2)
  most benefits from the spec being otherwise quiescent. Take **C1**
  if F-0807 is the authoritative resolution of the same surface in the
  active findings store; take **C2** only if the rename / minting-seam
  pin in F-0807 is rejected. Either way, sequence after F-0711's
  rename if F-0711 lands, so the prefix string the contract pins is
  the final one.

Edge case for the implementer (all three options): GOV-25 / GOV-27
require the existing `v1-…` alias anchors to survive byte-equal on the
collision-source-set blockquote (Option A); the byte-equality MUST in
PIC-11 must remain reachable from the schema-slug collision posture in
`schema-subset.md` (Option B); and the `loom-direct:`-containing
blockquote in `tool-calls.md` is one of the 12 typed/structural seams
under GOV-31, so its label must survive (Option C).

## Consequence

**Severity:** advisory

Each of the three sentences phrased as a MUST whose violation no test can
witness invites the same two-sided failure mode the spec corpus elsewhere
guards against: a reviewer rejecting a conformant implementation on a
factoring grievance, and the inverse — implementers learning to treat MUST
as advisory after running into one whose violation produces no observable
divergence. The runtime's externally-observable behaviour (collision
diagnostics fired, byte-mismatch recompiles emitting
`validator-cache-collision`, code-side tool-call `toolCallId` flowing
through `execute()`) is unaffected by any of the three either way. The
specific concrete risk on Option C is downstream: the literal prefix is
already cited from `tool-calls.md` and `implementation-notes.md`, and two
active findings (F-0711, F-0807) want to pin a contract here — leaving the
prefix as an unfenced literal leaves those findings reasoning about a
sentence that does not say what property it is asserting.
