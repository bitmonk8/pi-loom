## Finding

`V18c`'s **Ships when** gate has two structurally different done-conditions joined into one criterion. The first is observable at implementation time and verified mechanically: `npm run typecheck` runs the brand-string assertion green on `main`, and `npm test` runs the six static build-time gates (step-2(a)/2(b) surface-inventory, `engines.node` three-way, peer-dep literal-read, reason-snapshot literal-array consistency, provider seed-field `Api`-coverage, strict-capability probe) green on `main`. The second is a per-bump editorial process: the bump commit message records a per-item outcome for every editorial-review checklist item `(a)`–`(al)` and no MUST-level item `(a)`–`(e)` carries a `fail` — explicitly "confirmed by editorial review — not by `npm test`."

The editorial-review checklist is a recurring contributor procedure that runs at each real Pi version bump, not an artefact produced when the leaf is first implemented. Folding its acceptance into the leaf's **Ships when** means the leaf has no single externally observable "done" that an implementer can reach during loom 1.0 development: the automated half is reachable by writing the gates and running CI, but the editorial half cannot be exercised because no version bump occurs at implementation time. Two reasonable implementers diverge — one tags `V18c-complete` once the gates are green and the checklist is documented, another holds the tag waiting for an editorial review that never happens during the build.

The conflation is also load-bearing for the `V9l` blocking design. `V9l` (and `V9l-T`) declare `V18c` as the clause-(a) resolution owner and stay blocked until the editorial-review checklist item (a) resolves the host-prerequisites clause (a) open contradiction; the plan tracks this "in plan terms by `V18c`'s Deps edge." Because that anchor is the `V18c-complete` tag, and `V18c-complete` is reachable from the automated half alone, the blocking predicate can fire on green gates rather than on the actual clause-(a) resolution — `V9l` is either mis-unblocked before clause (a) is resolved, or `V18c` is never taggable during development and every `V18c` dependent stalls. The editorial-acceptance done-condition needs to live on a leaf distinct from the automated-gate implementation so each has its own coherent completion signal.

## Solution Space

**Shape:** single

### Recommendation

Separate the one-time automated-gate implementation deliverable (verified by `npm test` / `npm run typecheck`) from the recurring per-bump editorial-review acceptance (verified by editorial review), so each has its own leaf-completion signal, while preserving `V9l`'s blocking anchor.

- In `docs/plan_topics/V18c-version-bump-checklist.md`, reduce **Ships when** to the mechanically observable conditions only: the `npm run typecheck` brand-string assertion green on `main`, and the `npm test` run of the step-2(a)/2(b) surface-inventory, `engines.node` three-way, peer-dep literal-read, reason-snapshot literal-array consistency, provider seed-field `Api`-coverage, and strict-capability probe gates green on `main`. Strike the trailing editorial-review-acceptance clause — everything from "and the editorial-review checklist's acceptance is confirmed by editorial review — not by `npm test` — that …" through the MUST/SHOULD blocking distinction — from `V18c`'s **Ships when**. `V18c` keeps its `V18c-T`, `V18a`, `V18b` Deps and its automated-gate role.
- Move the editorial-review checklist acceptance to a new leaf `<new>` (the per-bump contributor procedure): its **Ships when** is the editorial-review acceptance — the bump commit message records a per-item outcome for every item `(a)`–`(al)`, and no MUST-level item `(a)`–`(e)` carries a `fail` — with no code/test. `<new>` owns the clause-(a) resolution. The checklist itself stays defined in `version-bump-step2.md`; `<new>` owns only the leaf-completion semantics for its acceptance. Link `<new>` into the appropriate `docs/plan.md` section.
- Trim `V18c`'s **Adds** so its editorial-checklist sentence previews the checklist as the per-bump contributor procedure that `<new>` owns, rather than restating the MUST/SHOULD severity and block-or-not gate conditions.
- Re-point the clause-(a) resolution owner from `V18c` to `<new>`: in `docs/plan.md` §Blocked obligations and the V9 interleave note, and in `V9l`/`V9l-T`'s **Deps** prose, the clause-(a) blocking anchor becomes `<new>`. `V9l`/`V9l-T` keep their existing `V18c` Deps edge (for the `session-shutdown-reason-snapshot` brand-string constant) and add `<new>` as the clause-(a) resolution owner that gates them.

Edge cases the implementer must watch:

- `V9g`, `V9h`, `V9h-T`, `V18d`, and `V18d-T` cite `V18c` only for the brand-string constant or the static-gate re-run on the revert path — both of which remain in `V18c` — so they need no retarget. Keeping the automated half under the `V18c` id (rather than moving it to `<new>`) is what keeps that retarget surface empty.
- Do not invent the final ID for `<new>`; the implementer allocates it under the plan's leaf-ID scheme (`conventions.md` §Plan structure).
- `<new>` carries no REQ-ID-closing or un-anchored-MUST citing test (its acceptance is editorial), so the *Transitive-completeness plan-maintenance* trigger for adding it to `H5b`'s Deps (conventions.md §REQ-ID discipline) most likely does not fire; the implementer confirms this against that clause's exact triggers before deciding.

## Consequence

**Severity:** correctness

`V18c` has no single implementation-time-observable "done": its **Ships when** requires an editorial review of the 38-item checklist that only runs at a real Pi version bump, so two implementers diverge on when `V18c-complete` is taggable. Worse, `V9l`'s blocking predicate is anchored on `V18c-complete`, which the automated half alone can satisfy — so `V9l` is either unblocked before clause (a) is actually resolved or `V18c` never completes during loom 1.0 development and stalls all of its dependents.
