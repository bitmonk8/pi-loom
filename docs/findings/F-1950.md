## Problem

`V18c`'s **Ships when** joins two structurally different done-conditions: a one-time, implementation-time-observable half (the `npm run typecheck` brand-string assertion and the `npm test` static build-time gates green on `main`) and a recurring per-bump editorial-review acceptance (the bump commit records a per-item outcome for items `(a)`–`(al)` with no MUST-level `(a)`–`(e)` `fail`, confirmed by editorial review, not by `npm test`). The editorial half runs only at a real Pi version bump, so the leaf has no single externally observable "done" reachable during loom 1.0 development, and two implementers diverge on when `V18c-complete` is taggable. The conflation is load-bearing for `V9l`/`V9l-T`, which anchor their clause-(a) blocking predicate on `V18c-complete` — reachable from the automated half alone — so `V9l` is either mis-unblocked before clause (a) is resolved or stalls all `V18c` dependents.

## Issue introduction

**Verdict:** single-commit-introduction (compounded by later edits)
**Introducing commits:** `f1cda48` — *pi-loom plan: resolve "V18c mislabels editorial-review checklist as a static build-time gate"* (2026-06-11)
**History:** At leaf inception (`c6a664e`), `V18c`'s **Ships when** was purely the automated gates — "`npm test` runs the step-2(a)/2(b), `engines.node`, peer-dep, and reason-snapshot gates green on `main`" — with no editorial-review condition and no atomicity defect. `f1cda48` correctly reclassified the editorial-review checklist as *not* a static build-time gate, but in doing so folded its acceptance ("confirmed by editorial review — not by `npm test`") into `V18c`'s **Ships when**, introducing the automated-vs-editorial mix. `828b7d3` (2026-06-16) added the MUST/SHOULD blocking distinction for `(a)`–`(e)` vs `(f)`–`(al)` to the exit gate, deepening the interleave. `1ca13ec` (2026-06-19, F-0662) added item `(al)` and the `V18c` recording gate, growing the checklist to its current 38-item form.

## Solution approach

Narrow `V18c`'s **Ships when** in `V18c-version-bump-checklist.md` to the mechanically observable conditions only — the `npm run typecheck` brand-string assertion and the `npm test` static-gate run on `main` — striking the editorial-review-acceptance clause and its MUST/SHOULD blocking distinction, and trim `V18c`'s **Adds** so its editorial-checklist sentence previews the per-bump contributor procedure rather than restating the gate conditions. Add a new plan leaf owning the editorial-review checklist acceptance (the per-bump procedure verified by editorial review, not `npm test`) and the clause-(a) resolution, and link it into `docs/plan.md`. Re-point the clause-(a) resolution owner from `V18c` to the new leaf in `docs/plan.md` §Blocked obligations and the V9 interleave note, and in `V9l`/`V9l-T`'s **Deps** prose. Keep `V18c`'s automated-gate role under its existing id so `V9g`, `V9h`, `V9h-T`, `V18d`, and `V18d-T` need no retarget.

## Solution constraints

- The spec is read-only; the editorial-review checklist definition stays in `version-bump-step2.md` — the new leaf owns only its leaf-completion semantics, not the checklist text.
- Allocate the new leaf's ID under the plan leaf-ID scheme (`conventions.md` §Plan structure); do not invent an off-scheme ID or reuse a retired one.
- `V9l`/`V9l-T` retain their existing `V18c` Deps edge (the `session-shutdown-reason-snapshot` brand-string constant); only the clause-(a) resolution anchor moves to the new leaf.
