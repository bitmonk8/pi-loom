## Finding

`V2c` (Runtime value model, equality, and wire-name translation) performs inbound
wire-name translation with "enum-tag reattach **via sidecar**" (V2c **Adds.**). That
per-schema sidecar — a two-map structure: a wire-name translation map plus a
named-enum-position map keyed by JSON Pointer into the lowered fragment — is produced
by `V5f`'s lowering pass (V5f **Adds.** "wire-name and named-enum-position sidecars";
V5f **Tests.** "the lowering pass captures, per `$defs` entry, a two-map sidecar …",
citing `schema-subset.md` Lowering Algorithm step 5). `V5f` is therefore the producer
of the data structure `V2c` consumes.

`V2c`'s **Deps.** list `V2c-T, V2a, V5d`, and `V2c-T`'s **Deps.** list `V2a, V5d`.
Neither names `V5f`. `V5d` is the schema-subset **reject gate** — its own **Adds.**
states plainly that "the lowering pass and the canonical-hash recipe … are owned by
`V5f`", so `V5d` does not produce the sidecar. The dependency edge that should point at
the sidecar producer points at a sibling leaf that produces nothing `V2c` reads.

Two distinct consequences follow. First, because `V5f` is not in `V2c`/`V2c-T` Deps,
the dependency graph permits `V2c` to be picked and built before `V5f` exists (the
sidecar contract `V2c` consumes need not be defined when `V2c` lands). Second, no leaf
is designated as the owner of the shared sidecar *type*: `V5f` describes the producer
side and `V2c` describes the consumer side, but with no Class-2 seam binding between
them (conventions.md §Leaf format: a seam binds only when the consumer lists the
producing leaf in **Deps.** and names the seam in its own **Adds.**), the producer and
consumer code can independently define divergent in-shard sidecar shapes.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** `git log --follow` shows `V2c-value-model.md` and `V2c-T-value-model.md` were both created in `c6a664e` and have no later commits. `git show c6a664e:docs/plan_topics/V2c-value-model.md` confirms the inception revision already carried both halves of the defect: **Adds.** named "enum-tag reattach via sidecar" and **Deps.** were `V2c-T, V2a, V5d` (V5d, not V5f). `git log -G "Deps.*V5d" -- docs/plan_topics/V2c-value-model.md` returns only `c6a664e`, so the V5d edge has never been the producer edge. Later `V5f`/`V5d` commits (`ba7da00`, `cb8737a`, `fdccb5d`) refined where the sidecar is described but never corrected `V2c`'s edge or assigned a type owner. The mis-targeted edge and the undesignated type owner have been latent since the plan's first build.

## Solution Space

**Shape:** single

### Recommendation

Retarget the `V2c`/`V2c-T` producer dependency to the sidecar producer and give the
shared sidecar type a single owning leaf. Make both edits, applying the **Deps.**
retarget first so the type-ownership statement lands against a stable producer edge.

Retarget the dependency edge:
- In `docs/plan_topics/V2c-value-model.md` **Deps.**, strike `V5d` and insert `V5f`,
  yielding `V2c-T, V2a, V5f`.
- In `docs/plan_topics/V2c-T-value-model.md` **Deps.**, strike `V5d` and insert `V5f`,
  yielding `V2a, V5f`.

This points each consumer at the leaf that produces the sidecar it reads. `V5f`'s own
**Deps.** already include `V5d`, so `V5d` stays transitively reachable; this retargets
the existing cross-slice backward edge rather than adding a dependency. It removes the
false producer edge, prevents `V2c` being scheduled before `V5f`, and satisfies the
conventions Class-2 seam binding precondition — the consumer lists the producing leaf in
**Deps.** and names the sidecar in **Adds.**.

Designate the type owner:
- In `docs/plan_topics/V5f-subset-lowering-hash.md` **Adds.**, state that `V5f` owns the
  shared per-schema sidecar type consumed by downstream leaves — the wire-name
  translation map plus the JSON-Pointer-keyed named-enum-position map.
- In `docs/plan_topics/V2c-value-model.md` **Adds.**, phrase the enum-tag reattach so it
  names the `V5f`-produced sidecar as the consumed seam.

This binds producer and consumer to one definition rather than two independently
authored shapes, closing the divergent-sidecar risk; the Class-2 seam binding then makes
`V5f`'s declaration the binding contract for `V2c`. No spec edits are required.

If `V5f` is later split (related finding "101"), target both the retargeted edge and the
type-ownership statement at the resulting sub-leaf that carries the lowering pass
emitting the sidecar.

## Consequence

**Severity:** correctness

Because `V2c`/`V2c-T` depend on the reject gate (`V5d`) rather than the sidecar
producer (`V5f`), the build graph allows `V2c` to be implemented before its actual
producer exists, and — with no leaf owning the shared sidecar type — the producer
(`V5f`) and consumer (`V2c`) can settle on divergent two-map sidecar shapes. The result
is a silent data-shape mismatch at the in-shard producer/consumer boundary that does not
surface at either leaf's own gate; two reasonable implementers would diverge on the
sidecar structure.
