## Finding

`V2c` (Runtime value model, equality, and wire-name translation) performs inbound
wire-name translation with "enum-tag reattach **via sidecar**" (V2c **Adds.**). That
per-schema sidecar — a two-map structure: a wire-name translation map plus a
named-enum-position map keyed by JSON Pointer into the lowered fragment — is produced
by `V5f`'s lowering pass (V5f **Adds.** "wire-name and named-enum-position sidecars";
V5f **Tests.** "the lowering pass captures, per `$defs` entry, a two-map sidecar …",
citing `schema-subset.md` Lowering Algorithm step 5). `V5f` is therefore the producer
of the data structure `V2c` consumes.

`V2c`'s **Deps.** list `V2c-T, V2a, V5d`, and `V2c-T`'s **Deps.** list `V2a, V5d`.
Neither names `V5f`. `V5d` is the schema-subset **reject gate** — its own **Adds.**
states plainly that "the lowering pass and the canonical-hash recipe … are owned by
`V5f`", so `V5d` does not produce the sidecar. The dependency edge that should point at
the sidecar producer points at a sibling leaf that produces nothing `V2c` reads.

Two distinct consequences follow. First, because `V5f` is not in `V2c`/`V2c-T` Deps,
the dependency graph permits `V2c` to be picked and built before `V5f` exists (the
sidecar contract `V2c` consumes need not be defined when `V2c` lands). Second, no leaf
is designated as the owner of the shared sidecar *type*: `V5f` describes the producer
side and `V2c` describes the consumer side, but with no Class-2 seam binding between
them (conventions.md §Leaf format: a seam binds only when the consumer lists the
producing leaf in **Deps.** and names the seam in its own **Adds.**), the producer and
consumer code can independently define divergent in-shard sidecar shapes.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** `git log --follow` shows `V2c-value-model.md` and `V2c-T-value-model.md` were both created in `c6a664e` and have no later commits. `git show c6a664e:docs/plan_topics/V2c-value-model.md` confirms the inception revision already carried both halves of the defect: **Adds.** named "enum-tag reattach via sidecar" and **Deps.** were `V2c-T, V2a, V5d` (V5d, not V5f). `git log -G "Deps.*V5d" -- docs/plan_topics/V2c-value-model.md` returns only `c6a664e`, so the V5d edge has never been the producer edge. Later `V5f`/`V5d` commits (`ba7da00`, `cb8737a`, `fdccb5d`) refined where the sidecar is described but never corrected `V2c`'s edge or assigned a type owner. The mis-targeted edge and the undesignated type owner have been latent since the plan's first build.

## Solution Space

**Shape:** multiple

This finding carries two independent obligations: (A) retarget the dependency edge from
the reject gate to the real producer, and (B) give the shared sidecar type a designated
owning leaf. They land on different leaves and are not a single one-paragraph edit, so
they are presented as separate obligations to be resolved in order — not as competing
alternatives. Both are required.

### Option A — Retarget the V2c / V2c-T dependency edge to the sidecar producer

**Approach:** Replace `V5d` with `V5f` in the **Deps.** of `V2c` and `V2c-T`, so the
consumer depends on the leaf that produces the sidecar it reads. `V5f`'s own **Deps.**
already include `V5d`, so `V5d` remains transitively reachable; this is a retarget of the
existing cross-slice backward edge, not a new dependency.
**Plan edits:**
- `docs/plan_topics/V2c-value-model.md` **Deps.**: strike `V5d`, insert `V5f` (yielding `V2c-T, V2a, V5f`).
- `docs/plan_topics/V2c-T-value-model.md` **Deps.**: strike `V5d`, insert `V5f` (yielding `V2a, V5f`).
**Spec edits:** None.
**Pros:** Removes the false producer edge; prevents `V2c` being scheduled before `V5f`; satisfies the conventions Class-2 seam binding precondition (consumer lists the producing leaf in **Deps.** and names the sidecar in **Adds.**, which `V2c` already does).
**Cons:** On its own does not state which leaf owns the shared type definition.
**Risks:** If `V5f` is later split (see related finding "101"), the edge must target whichever resulting leaf carries the lowering pass that emits the sidecar.

### Option B — Designate V5f as the owner of the shared sidecar type

**Approach:** Make `V5f` the explicit owner of the shared sidecar type that `V2c`
consumes, so producer and consumer bind to one definition rather than two independently
authored shapes. `V5f` already declares the two-map sidecar in **Adds.** and asserts its
shape in **Tests.**; state in `V5f` **Adds.** that it owns the shared sidecar type
exported to consumers (the wire-name map plus the JSON-Pointer-keyed named-enum-position
map), and have `V2c` **Adds.** reference that V5f-owned sidecar as the seam it consumes.
**Plan edits:**
- `docs/plan_topics/V5f-subset-lowering-hash.md` **Adds.**: state that `V5f` owns the shared per-schema sidecar type consumed by downstream leaves.
- `docs/plan_topics/V2c-value-model.md` **Adds.**: phrase the enum-tag reattach so it names the V5f-produced sidecar as the consumed seam.
**Spec edits:** None.
**Pros:** Closes the divergence risk by giving the in-shard sidecar data shape a single owning leaf; the conventions Class-2 seam binding then makes `V5f`'s declaration the binding contract for `V2c`.
**Cons:** Touches two leaves' **Adds.** prose.
**Risks:** Low; both leaves already describe the sidecar, so this formalizes existing intent.

### Recommendation

Resolve **Option A first**, then **Option B** on the resulting baseline. Option A is the
scope-bounding edit (two **Deps.** strikes/inserts) and establishes the producer
relationship the type-ownership statement rests on; once the `V2c`/`V2c-T` → `V5f` edge
exists, Option B's type-owner designation lands against a stable graph and the Class-2
seam binding between `V5f` (producer) and `V2c` (consumer) holds. Both obligations are
required. Watch the interaction with related finding "101": if `V5f` is split, target the
edge and the type-ownership statement at the sub-leaf that owns the lowering pass emitting
the sidecar.

## Consequence

**Severity:** correctness

Because `V2c`/`V2c-T` depend on the reject gate (`V5d`) rather than the sidecar
producer (`V5f`), the build graph allows `V2c` to be implemented before its actual
producer exists, and — with no leaf owning the shared sidecar type — the producer
(`V5f`) and consumer (`V2c`) can settle on divergent two-map sidecar shapes. The result
is a silent data-shape mismatch at the in-shard producer/consumer boundary that does not
surface at either leaf's own gate; two reasonable implementers would diverge on the
sidecar structure.
