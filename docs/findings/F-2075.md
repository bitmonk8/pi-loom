## Finding

The *Category-(2) inventory join key* paragraph closes with a hard rule for the multi-segment collision case: "For legal multi-segment matches (e.g. `Foo.bar` and `Foo.baz` both resolving against `import { Foo }`, where neither entry is a bare `Foo`), the audit's `proposed-resolution` field reports the entry with the lexicographically-smallest `path` for determinism." This pins one specific tie-break algorithm (lex-min on `path`) where the stated goal — determinism — is satisfied by *any* run-stable tie-break.

The only cross-run guarantee the audit actually pins is same-implementation reproducibility. The *Wire serialisation* paragraph in `audit-wire-and-canary.md` requires the implementation to "pin to a deterministic ordering, e.g. lexicographic on POSIX-form path, so two runs against the same source tree emit byte-identical output" — note that the file-iteration ordering, an exactly parallel determinism concern, is left to the implementation to pin (with lexicographic offered only as an example). No cross-implementation byte-identical guarantee is stated anywhere in the audit cluster. The `proposed-resolution` field is human-facing remediation guidance; CI routes on the leading `audit/<class>/<family>/<symptom>` discriminator token, not on this field.

So the lex-min mandate narrows implementer choice without a stated need, and it is internally inconsistent with the file-iteration ordering rule one page over, which treats the same determinism requirement permissively. Any deterministic, run-stable tie-break the implementation pins would satisfy every guarantee the spec actually makes.

## Solution Space

**Shape:** single

### Recommendation

Relax the final sentence of *Category-(2) inventory join key* so the multi-segment collision tie-break is left to the implementation to pin deterministically, matching the formulation the file-iteration ordering rule already uses in `audit-wire-and-canary.md`. For example: "For legal multi-segment matches … the `proposed-resolution` field reports a single entry chosen by a deterministic, run-stable tie-break that the implementation pins (e.g. the lexicographically-smallest `path`)." This preserves the same-implementation byte-identical guarantee without mandating a specific algorithm.

Edge case the editor must watch: if a *cross-implementation* byte-identical `proposed-resolution` is in fact intended (it is not stated anywhere in the audit cluster today), do not relax — instead keep the lex-min pin and add the cross-implementation reproducibility requirement explicitly as its justification, so the prescription is backed by a stated need rather than left implicit.

## Consequence

**Severity:** advisory

Shipping unfixed forces every audit implementation to use lexicographically-smallest-`path` selection in the rare multi-segment collision case, even though any deterministic tie-break meets the spec's only stated guarantee (same-implementation byte-identical output). Nothing breaks; the rule simply over-constrains and reads inconsistently against the permissive file-iteration ordering rule on the same surface.
