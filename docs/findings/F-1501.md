## Problem

§2 Runtime-value placeholders (governing `<scrutinee summary>` and `<value>`) layers an 80-Unicode-code-point cap on the `string` row of the canonical interpolation-stringification table but layers no length bound on the `array<T>` or schema-typed-object rows, which delegate verbatim to compact `JSON.stringify`. A `<scrutinee summary>` / `<value>` emission on a large or deeply-nested object/array runtime value therefore produces an unbounded normative message string, which flows unchanged through the panic-message routing surfaces (user-facing system-note, `InvokeFailure.message`, operator diagnostic channel) and is byte-asserted by conformance tests. This is the lone unbounded normative emission site for a Category 2 `<…>` placeholder, so two conformant implementations diverge on whether to truncate.

## Issue introduction

**Verdict:** single-introducing-commit
**Introducing commits:** `61ea1a5` (2026-05-06) — `pi-loom spec: resolve "Diagnostic message placeholders have no defined rendering"`
**History:** Commit `61ea1a5` introduced §2 Runtime-value placeholders into the then-monolithic `spec_topics/diagnostics.md`, delegating object/array rendering to the canonical interpolation-stringification table (which already specified compact `JSON.stringify` for arrays and schema-typed objects, added in `0c9ce54` on 2026-05-04) and layering the string-only 80-code-point truncation rule on top. Object and array scrutinees have been governed by an unbounded `JSON.stringify` path since that commit. Commit `055ab2f` (2026-06-01, *§2 Runtime-value placeholder rule contradicts its own Cat { name: "fluffy" } test vector*) made the gap visually evident by replacing the misleading `Cat { name: "fluffy" }` test vector with the schema-typed-object form `{"name":"fluffy"}` that actually exercises the unbounded JSON path, but did not address it. Commit `f5e89f4` (2026-06-04) split `diagnostics.md` into the subdirectory layout and carried §2 across into `placeholder-rendering-a.md` byte-identically.

## Solution approach

Generalise the §2 length bound so it applies to the rendered output of the canonical stringification table for every row that produces a textual rendering (`string`, `array<T>`, schema-typed object, and the recursive `<inner>` of `Result<T, E>`), preserving the existing 80-code-point cap, the 77-code-point prefix, the literal `...` suffix, and the Unicode-code-point counting basis; rewrite the `**String truncation.**` bullet in `placeholder-rendering-a.md` §2 to reflect that generalised scope. Add a §2 boundary test vector for a schema-typed-object rendering whose compact JSON exceeds the cap. In `placeholder-rendering-b.md` Edge cases *Category 2 boundary*, broaden the boundary obligation from "string scrutinee" to any §2-rendered scrutinee.
