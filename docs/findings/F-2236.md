## Finding

The `session_shutdown` teardown and drain-state machinery wrap several write and read operations in per-call `try`/`catch` arms and enumerate, parenthetically, the throw causes those arms are meant to absorb. Across three files the enumerations include an out-of-memory condition as one of the catchable causes:

- `session-shutdown-semantics.md` sub-step 2 (stamp write): "*on a stamp throw (frozen entry, proxied entry, write-rejecting setter, OOM during property assignment) the per-entry `catch` arm proceeds to the next entry*".
- `session-shutdown-semantics.md` interstitial seam (sub-steps 4→5 tag write): "*a throw from the tag write (frozen `LoomRegistry`, proxied setter, OOM during property assignment) MUST be caught and silently swallowed*".
- `drain-state-contract.md` PIC-31 *Read-failure fallback* (read side): "*to absorb a throw out of the read (a frozen-but-mutated-import edge case, a future Proxy-backed field with a getter trap, OOM during the snapshot construction, or any other read-side throw …)*".
- `session-only-degraded-state.md` residual gap 3 and acceptance criterion (l) restate the same write-side and read-side examples.

The other enumerated causes are genuine catchable JavaScript throws: assigning to a frozen object's property throws a `TypeError` in strict/module code, a `Proxy` `set`/`get` trap or a write-rejecting setter can throw, and a getter trap can throw. A true V8/Node heap out-of-memory condition is not in that class. When the JavaScript heap is exhausted, V8 invokes its fatal-error handler and aborts the process ("FATAL ERROR: … JavaScript heap out of memory") — it does not deliver a catchable exception to the surrounding `try`/`catch`. A trivial string property assignment (`entry.shutdownReason = …`) or the small object literal `readDrainState` returns cannot themselves provoke a *catchable* allocation error; the narrow allocations that do throw a catchable `RangeError` (e.g. over-limit array/string lengths, `ArrayBuffer` allocation past the engine cap) are not what these sites perform.

The enumerations therefore assert a runtime fact that does not hold: the listed `catch` arm would never run for a genuine heap-OOM at these sites, because the process is already gone. The example is misleading rather than load-bearing — the `catch` arms still correctly handle the real catchable causes — but it embeds an incorrect runtime assumption in normative prose and can mislead an implementer or fixture author into believing a catchable OOM at these sites is a path to design or test against.

## Issue introduction
**Verdict:** introduced-by-commit

**Introducing commits:** `75a9626` (2026-05-09, *resolve "SessionShutdownEvent.reason closed set has no build-time pin against the SDK type"*) — authored the sub-step 2 per-entry `try`/`catch` stamp prose with the catchable-throw list `(frozen entry, proxied entry, write-rejecting setter, OOM during property assignment)`; `52b9101` (2026-05-11, *resolve "T24 — Fork-reason watcher closure…"*) — added the read-side *Read-failure fallback* enumeration with `OOM during the snapshot construction`.

**History:** `git log -S "OOM during property assignment"` and `git log -S "OOM during the snapshot construction"` show the defect was co-extensive with the feature prose: each `OOM …` example was present in the same commit that first authored the corresponding `try`/`catch` isolation clause (write-side in `75a9626`, read-side in `52b9101`), never introduced by a later regression edit. `c3de47a` (2026-06-27) subsequently relocated the universal teardown prose into `session-shutdown-semantics.md`, carrying the existing write-side example along unchanged. The six current occurrences are propagations of these two original authorings, not independent introductions.

## Solution Space

**Shape:** multiple

### Option A — Drop OOM from the catchable enumerations

- **Approach:** Treat fatal OOM as out of scope of the per-call isolation guarantee and remove the `OOM during property assignment` / `OOM during (the) snapshot construction` clauses from all six enumerations, leaving the genuinely-catchable causes (frozen/proxied entry, write-rejecting setter, getter/setter trap, frozen-but-mutated import).
- **Spec edits:** Delete the OOM clause from each of the six sites: `session-shutdown-semantics.md` lines 10 and 15; `drain-state-contract.md` lines 5 and 14; `session-only-degraded-state.md` lines 87 and 107. No surrounding sentence needs restructuring — each clause is a comma-separated list item.
- **Pros:** Removes the incorrect assumption entirely; smallest, purely-subtractive diff; the remaining examples are all accurate; no new runtime claim to verify.
- **Cons:** Loses any acknowledgement that heap exhaustion at these sites is simply uncatchable (an implementer might still wonder).
- **Risks:** None of substance; the `catch` arms' contract is unchanged.

### Option B — Requalify the example with the verified catchable condition

- **Approach:** Keep an allocation-failure example but correct it to a genuinely catchable one and state the condition, e.g. replace OOM with "a `RangeError` from an over-limit allocation" or add a one-clause note that genuine heap-OOM is process-fatal and out of scope of this isolation.
- **Spec edits:** At each of the six sites, replace `OOM during …` with the corrected catchable example and/or a parenthetical noting heap-OOM is uncatchable/out of scope.
- **Pros:** Preserves an allocation-failure illustration while making the runtime claim accurate; documents the out-of-scope boundary explicitly.
- **Cons:** Larger, repeated edit; introduces a new runtime claim that must itself be correct; the corrected condition (over-limit allocation) does not actually arise at these trivial write/read sites, so the example remains somewhat artificial.
- **Risks:** A sloppy requalification could substitute one inaccurate example for another.

### Recommendation

Take **Option A**. The illustrative list exists only to show the *kind* of throw the `catch` arms absorb; the frozen/proxied/setter/import-mutation cases already convey that, and removing the OOM clause is the minimal edit that eliminates the false assumption without adding a new runtime claim to defend. Apply it to all six occurrences in the same edit so the corpus stays internally consistent; do not leave the read-side `OOM during (the) snapshot construction` variants behind.

## Consequence

**Severity:** advisory

No runtime behaviour diverges: the `try`/`catch` arms still absorb the real catchable throws (frozen/proxied/write-rejecting writes, getter/setter traps), and a genuine heap-OOM crashes the process regardless of the prose. The cost is a false runtime assumption stated in normative-adjacent prose, which can mislead an implementer's mental model or prompt a fixture author to attempt an unrealizable "catchable OOM" test at these sites.
