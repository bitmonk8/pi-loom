## Finding

DISC-5 (`docs/spec_topics/discovery/package-and-settings.md`) specifies `pi.looms` glob resolution by naming a specific third-party library in normative prose: *"Glob patterns are matched with the `minimatch` engine ‚Äî the same package Pi applies to its own resource arrays in `@earendil-works/pi-coding-agent` (`dist/core/package-manager.js`, symbol `applyPatterns`)."* The behavioural contract that actually matters is already stated alongside it and is fully sufficient: loom MUST resolve patterns under the same glob dialect Pi applies to its own resource arrays (`pi.extensions` / `pi.skills` / `pi.prompts` / `pi.themes`), with the fixed `!`/`+`/`-` override ordering (plain ‚Üí `!` ‚Üí `+` ‚Üí `-`), `nocase` off, exact-path operand treatment for `+`/`-`, and matching attempted against the package-root-relative path, basename, and POSIX-normalised absolute path.

Naming `minimatch` as *the* engine adds nothing to that observable contract while coupling the spec to Pi's current dependency choice: if Pi changes the package backing `applyPatterns`, the spec's "same package" claim silently goes stale, and loom's correct obligation (track Pi's dialect) is left ambiguously entangled with a now-wrong library identity. The paragraph itself half-acknowledges this when it says it "pins loom's *use* of that matching, not minimatch's glob dialect, which the package owns" ‚Äî but then it still designates the package by name as the normative engine, which is precisely the coupling.

The fix is narrow and behaviour-preserving: lead with the dialect/ordering contract as the binding requirement, and demote the `minimatch` identity to a non-normative realisation note. The override ordering and exact-path operand rules that DISC-5 legitimately owns are unchanged.

## Solution Space

**Shape:** single

### Recommendation

In DISC-5, state the binding requirement as the observable dialect contract: loom MUST resolve `pi.looms` glob patterns under the same glob dialect and `!`/`+`/`-` override ordering Pi applies to its own resource arrays (`pi.extensions` / `pi.skills` / `pi.prompts` / `pi.themes`), preserving the already-pinned specifics ‚Äî fixed order (plain ‚Üí `!` ‚Üí `+` ‚Üí `-`), `+`/`-` as exact-path operands compared against relative path / basename / POSIX-normalised absolute path, `nocase` off, insertion order preserved within a class. Then demote the library identity to a non-normative realisation note, e.g. "Pi currently realises this matcher with `minimatch` (`applyPatterns` in `@earendil-works/pi-coding-agent`, `dist/core/package-manager.js`); loom mirrors that dialect rather than the package."

Keep the override-ordering and exact-path rules normative ‚Äî they are loom's own contract, not Pi's, and must not be softened. Retain the SDK symbol reference (`applyPatterns`) only as a grounding pointer inside the non-normative note, not as the engine mandate. Do not revert the introducing change wholesale: the override-ordering and dialect-identity pins it added are the legitimate fix for the original under-specification and must stay.

## Consequence

**Severity:** advisory

No implementer is misled about behaviour ‚Äî the dialect and override ordering are fully pinned independently of the library name. The cost is durability: the "same package" coupling goes stale if Pi swaps its matcher, and the spec over-constrains the realisation without a behavioural justification.
