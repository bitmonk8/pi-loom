## Finding

PIC-1 clause (c) enumerates three co-fire-eligible runtime-event surfaces. Sub-clause (i) reads "the `validation` event when its `cause` is `"schema_validation"`," and sub-clause (ii) reads "the `code_tool` event when its underlying `CodeToolError.cause` is `"validation"`." The sibling phrasings are deliberately asymmetric in a way that misleads. The `RuntimeEvent` declared just above PIC-1 carries `kind`, `code?`, `loom`, `invocation_id`, `query_site?`, `discard_site?`, `message`, `attempts?`, `tokens_used?`, `masked?`, and `occurred_at` — no `cause` field. Read literally, (i) names a non-existent property on the event payload; read charitably (and by analogy with (ii)) it points at the underlying `ValidationError.cause` sub-discriminator declared in [`errors-and-results/queryerror-variants.md`](../errors-and-results/queryerror-variants.md). Two reasonable implementers diverge here: one extends `RuntimeEvent` with `cause?: string` so the predicate can be evaluated against the event itself (a wire-shape change, and an additive break against PIC-1's "additive-only" contract because it adds a payload field downstream consumers will then condition on); the other reaches through to the originating `ValidationError.cause` (correct, but invisible from the prose). Because the predicate gates an observable surface — whether `masked: ["ceiling#2"]` populates — the choice is not benign.

The defect is purely a wording asymmetry. The `cause` sub-discriminator lives on `ValidationError` and on `CodeToolError`, never on `RuntimeEvent`; the predicate is evaluated at the origin site where the typed `QueryError` is in hand, not from the lowered runtime-event payload. Bringing (i) into lockstep with (ii)'s explicit "underlying `<Variant>Error.cause`" framing eliminates the divergence without touching the wire shape.

The durable fix is the wording alignment in clause (c)(i), not a revert of either commit.

## Solution Space

**Shape:** single

### Recommendation

Bring PIC-1 clause (c)(i) into lockstep with (c)(ii) by qualifying the sub-discriminator with its owning type. Rewrite the sub-clause to read:

> (i) the `validation` event when its underlying `ValidationError.cause` is `"schema_validation"` (ceiling #4 surface at the typed-query response / model-driven `tool_use` args / `params` validation / `invoke<T>` return value boundary, per [CIO-3](../hard-ceilings/ceilings-3-and-4.md#cio-3))

The `RuntimeEvent` type block stays untouched — `cause` is not a wire field on the event and adding one would (1) widen the additive-only payload contract for no observable gain, (2) duplicate state the originating `QueryError` already carries, and (3) invite downstream consumers to condition on a sub-discriminator that the spec deliberately surfaces only via `kind` plus the variant-specific `ValidationError`/`CodeToolError` carriers.

Implementer-relevant edge cases:

- The predicate is evaluated **at the origin site** — where the typed `QueryError` is constructed and the `RuntimeEvent` is being lowered — not by consumers reading the emitted event. The wording fix makes that location explicit.
- Sibling sub-clause (ii) (`CodeToolError.cause`) already uses this phrasing; no edit needed there.
- The cascade-twin verbatim-copy rule in clause (f) remains undisturbed: the boundary re-emission copies the originating event verbatim, so the predicate is never re-evaluated post-origin and the `cause` lookup never needs to be reachable from the wire payload.
- The `V1 reachable predicate` in clause (d) names the `validation` event "with `cause: "schema_validation"`" — the same lookup. Apply the qualifier consistently if (d)'s shorthand reads ambiguously to the same reader; one occurrence is enough if (c)(i) clearly establishes the referent.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one widens `RuntimeEvent` with `cause?: string` (changing the wire shape and inviting downstream consumers to condition on a field PIC-1 never authorised), the other correctly sources the sub-discriminator from the originating `ValidationError`. The first path produces a non-conforming wire surface; the second is right but unobvious. Either way the predicate that gates the only non-empty `masked` payload in loom 1.0 (`["ceiling#2"]`) is misdescribed at its sole authoritative source.
