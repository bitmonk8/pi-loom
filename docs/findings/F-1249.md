## Finding

`V2e`'s **Adds** claims the leaf delivers the *inbound/outbound* wire-name translation boundary, explicitly naming "outbound wire-name JSON" as part of its deliverable. The spec backs a two-directional boundary: `runtime-value-model.md` §Wire-name translation states translation happens in exactly two places — *Inbound* (model output → loom value, rebuild loom-side names + reattach enum tags) and *Outbound* (loom value → JSON, walk the loom-side value and produce wire-named JSON when constructing tool input, query response payloads, or `invoke` arguments).

The leaf's single Tests bullet (mirrored in `V2e-T`) asserts only the inbound direction: "inbound translation rebuilds loom-side names so loom code never sees wire names; defaults bypass inbound translation." No assertion owned by `V2e`/`V2e-T` exercises the outbound direction. An outbound-rename regression — emitting the loom-side field name instead of the declared `field as "Wire"` wire name, or emitting a tagged enum object instead of the bare wire string — would turn no test red within this leaf, even though the leaf's Adds claims to close that behaviour.

`V2e` closes a code-keyed area (`cka-2`, `runtime-value-model.md` RVM) rather than a numbered REQ-ID, so there is no per-REQ-ID coverage-matrix row that would otherwise flag the missing direction; the only gate on the outbound behaviour is the leaf's own Tests list, which omits it.

The `d4f6e5c` split carried the gap through unchanged rather than introducing it — the defect is original to the plan's first authored form of this boundary.

## Solution Space

**Shape:** single

### Recommendation

Add one outbound assertion to the `V2e`/`V2e-T` pair, citing `runtime-value-model.md` §Wire-name translation (*Outbound*). Author it red-first in `V2e-T` and add the matching bullet to `V2e`, then extend `V2e`'s **Ships when** so it names the outbound vector alongside the existing inbound one.

The new Tests bullet asserts that a loom-side value carrying a `field as "Wire"` rename serialises outbound to JSON under the wire name (not the loom-side name), and that an enum value emits its bare wire string. Suggested bullet text: "[runtime-value-model.md — Wire-name translation](../spec_topics/runtime-value-model.md) (RVM code-keyed area): outbound translation serialises a `field as \"Wire\"` value to JSON under its wire name, and an enum value emits its bare wire string."

`V2e` is the correct home: its Adds already claims ownership of the outbound boundary, and no other leaf currently asserts outbound runtime serialisation (the construction sites — tool input, query response payloads, `invoke` arguments — consume the boundary but do not test it). If a future maintainer instead routes the outbound assertion to one of those downstream construction-site leaves, that leaf must be named explicitly in `V2e`'s Adds and must carry the assertion; do not leave the outbound claim in `V2e`'s Adds with the assertion owned nowhere.

## Consequence

**Severity:** correctness

The leaf can be marked complete with the outbound translation unverified. An implementation that serialises loom-side names (or tagged enum objects) instead of wire-named JSON would ship green, breaking the model/tool-facing wire contract precisely where `V2e` claims to own it; the regression would surface only far downstream at a tool-input / query-response / `invoke`-argument construction site, with no test attributing it to this boundary.
