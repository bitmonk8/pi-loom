## Finding

The `V6b` / `V6b-T` Tests block carries the bullet "`params` are validated through AJV against their lowered schema." Every sibling bullet in the same block pairs a concrete input with an observable outcome — a non-trailing default fires `loom/parse/non-trailing-default`, a non-literal default fires `loom/parse/default-not-literal`, a forward-referenced named type "resolves and validates correctly," an unresolvable named type fires `loom/parse/unresolved-named-type`. The AJV bullet alone names only the validation *mechanism*; it states neither what malformed input is fed nor what accept/reject outcome the test must observe.

Under the per-phase TDD ritual (land `V6b-T` red first, then `V6b` green), this wording lets a conforming implementation satisfy the bullet with a test that merely exercises the AJV code path without asserting a reject/accept boundary. A test that constructs a validator and calls it — passing whatever it returns — reads as satisfying "validated through AJV" while verifying nothing about the schema-conformance contract the bullet exists to gate.

The spec contract is unambiguous (`frontmatter-fields-a.md` §`params`: "`params` are validated with AJV at invocation time"); the gap is purely in the plan's test-requirement wording, which does not pin the observable the test must assert. The bullet lives canonically in `V6b-T` (the authoritative test-requirement source) and is duplicated verbatim into `V6b`.

## Solution Space

**Shape:** single

### Recommendation

In `V6b-T-params-defaults.md` (the authoritative test-requirement source), replace the third Tests bullet "`params` are validated through AJV against their lowered schema." with a bullet that names a concrete input vector and the observable accept/reject outcome — e.g. a `params` payload that violates its lowered schema is rejected and a conforming payload is accepted, asserted against a fixture pair. Where the rejection surfaces a defined diagnostic/error (per `frontmatter-fields-a.md` §`params`, AJV validation at invocation time), name that observable outcome in the bullet.

Apply the same replacement to the duplicated bullet in `V6b-params-defaults.md` so the impl leaf's Tests text stays consistent with its `-T` partner. (See the related duplication finding: if impl-leaf Tests are collapsed to a pointer at `V6b-T`, the edit lands only in `V6b-T`.)

## Consequence

**Severity:** advisory

The bullet names a mechanism rather than a reject/accept boundary, so a TDD-ritual implementer can land a red→green test that invokes AJV without asserting that a schema-violating `params` payload is rejected and a conforming one accepted. Two reasonable implementers diverge — one writes a fixture-pair reject/accept assertion, another writes a path-exercising smoke test — and the latter leaves the `params`-validation contract effectively ungated while the leaf still reports done.
