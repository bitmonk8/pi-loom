## Finding

In `docs/spec_topics/diagnostics/placeholder-rendering-a.md` §3 (*Syntactic-construct placeholder*), the rule for `<expr>` in the two literal-sublanguage parse codes reads:

> render the offending source span verbatim, copied byte-for-byte from the source file between the offending sub-expression's start and end token positions (post-newline-normalisation per [Lexical — Encoding](../lexical.md)), with internal whitespace preserved.

The English phrase "between X and Y" is ambiguous on endpoint inclusion — it is routinely used to mean both `[X, Y]` (closed) and `(X, Y)` (open), and `[X, Y)` (half-open) is the dominant convention in token-range / source-span APIs. The §3 prose never disambiguates this, so two conformant implementations can legitimately render different byte sequences for the same defect: e.g. for a `params:` default `a + b`, the closed reading produces `a + b` (matching the §3 test vector), while a half-open `[start, end)` reading drops the trailing token and produces `a + `.

This ambiguity collides directly with the byte-identical contract that DIAG-4 (and the opening paragraph of this very subsection) imposes on rendered diagnostic messages, pinned by GOV-15. The §3 test vector for `default-not-literal` (`a + b → ... offending sub-expression: a + b`) implies the closed reading, but a normative test vector cannot substitute for a normative rule: the rule has to self-contain the interpretation, otherwise an implementer whose tokeniser produces `[start, end)` half-open spans (the common case) will read the prose to permit their tokeniser's native convention and only discover the conflict via the test vector — too late to inform their span representation. The defect surfaces on both codes that use `<expr>`: `loom/parse/default-not-literal` and `loom/parse/tool-arg-not-literal`.

## Issue introduction

**Verdict:** present-since-inception (at §3 surface)
**Introducing commits:** `61ea1a5` (2026-05-06, *pi-loom spec: resolve "Diagnostic message placeholders have no defined rendering"*)
**History:** Commit `61ea1a5` introduced the eight placeholder-rendering categories into the then-monolithic `spec_topics/diagnostics.md`, including the `<expr>` rule with the wording "between the offending sub-expression's start and end token positions". The wording was carried verbatim through the May-2026 file split (`f5e89f4`, *docs: enforce 100KB hard / 30KB recommended size cap on spec set*) that created the present `placeholder-rendering-a.md`, and has not been edited since. `git log -S "start and end token position" -- docs/` returns only these two commits plus the unrelated `004dfef` review-doc edit and the `31ff060` directory move; no later commit attempted to clarify endpoint inclusion. The defect therefore traces to the original authoring of §3 — not to a later regression — and the spec-prose surface has been ambiguous on this point since the placeholder-rendering category was first defined.

## Solution Space

**Shape:** single

### Recommendation

Replace the ambiguous "between … start and end token positions" phrasing in the §3 `<expr>` bullet with an explicitly closed-interval formulation that names both endpoints as included. Suggested wording:

> For `<expr>` in the two literal-sublanguage codes, render the offending source span verbatim, copied byte-for-byte from the source file **from the first byte of the offending sub-expression's start token through the last byte of its end token, inclusive of both endpoints** (post-newline-normalisation per [Lexical — Encoding](../lexical.md)), with internal whitespace preserved.

Implementer-relevant edge cases the prose should leave undisturbed (already covered today, but worth re-checking after the edit):

- *Single-token spans* (e.g. a bare identifier `x`): start token == end token; the closed-interval reading degenerates to that one token's byte range, which matches the existing single-token expectation.
- *Trailing punctuation*: the end token is the last token of the sub-expression itself, not the following separator (`,`, `)`, newline). The wording "the offending sub-expression's … end token" already pins this; do not loosen it.
- *Internal comments / whitespace*: covered by "with internal whitespace preserved"; the new phrasing leaves that clause intact.
- *Newline normalisation*: the parenthetical citing Lexical — Encoding stays where it is, so the byte-for-byte copy is taken from the normalised source, not the on-disk bytes.

## Consequence

**Severity:** correctness

Two implementers reading §3 in isolation can produce different byte sequences for the same defect (closed `[start,end]` vs half-open `[start,end)`), breaking the byte-identical guarantee on every multi-token `<expr>` site for both `loom/parse/default-not-literal` and `loom/parse/tool-arg-not-literal`. The §3 test vector catches the case eventually, but the rule is meant to be normative on its own.
