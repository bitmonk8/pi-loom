## Problem

In `docs/spec_topics/diagnostics/placeholder-rendering-a.md` §3 (*Syntactic-construct placeholder*), the `<expr>` rule renders the offending source span "between the offending sub-expression's start and end token positions". The phrase "between X and Y" is ambiguous on endpoint inclusion — it can mean closed `[X, Y]`, open `(X, Y)`, or half-open `[X, Y)` (the dominant token-range convention). Two conformant implementations can therefore render different byte sequences for the same defect (e.g. `a + b` under the closed reading vs `a + ` under half-open), breaking the byte-identical contract pinned by GOV-15. The defect surfaces on both `loom/parse/default-not-literal` and `loom/parse/tool-arg-not-literal`; the §3 `a + b` test vector implies the closed reading but cannot substitute for a self-contained normative rule.

## Issue introduction

**Verdict:** present-since-inception (at §3 surface)
**Introducing commits:** `61ea1a5` (2026-05-06, *pi-loom spec: resolve "Diagnostic message placeholders have no defined rendering"*)
**History:** Commit `61ea1a5` introduced the eight placeholder-rendering categories into the then-monolithic `spec_topics/diagnostics.md`, including the `<expr>` rule with the wording "between the offending sub-expression's start and end token positions". The wording was carried verbatim through the May-2026 file split (`f5e89f4`, *docs: enforce 100KB hard / 30KB recommended size cap on spec set*) that created the present `placeholder-rendering-a.md`, and has not been edited since. `git log -S "start and end token position" -- docs/` returns only these two commits plus the unrelated `004dfef` review-doc edit and the `31ff060` directory move; no later commit attempted to clarify endpoint inclusion. The defect therefore traces to the original authoring of §3 — not to a later regression — and the spec-prose surface has been ambiguous on this point since the placeholder-rendering category was first defined.

## Solution approach

Rewrite the §3 `<expr>` rule bullet to state explicitly that both endpoints are included — a closed-interval reading consistent with the existing `a + b` test vector. The disambiguation must make the rule self-contained on endpoint inclusion without relying on the test vector to settle it.
