## Finding

Capability inventory Item 3 (`sdk-cap-subagent-isolated-session`) in `capability-inventory-items.md` carries a self-contained restatement of the subagent spawn-call satellite-type pins: it declares `SessionManager.inMemory` as `static inMemory(cwd?: string): SessionManager` on the `SessionManager` class at `dist/core/session-manager.d.ts`, names the loom-constructed `ResourceLoader` adapter's declaration at `dist/core/resource-loader.d.ts`, and specifies that the on-bump re-validation splits across version-bump editorial-review checklist items `(o)` (signature) and `(aj)` (no-persistence behaviour). In the same breath it says both satellites "are pinned at [spawn-block satellite type pins](./subagent.md#subagent-spawn-satellite-types)."

That cross-referenced block in `subagent.md` (`#subagent-spawn-satellite-types`) is the canonical owner of exactly the same material: it pins the identical `static inMemory(cwd?: string): SessionManager` signature to `dist/core/session-manager.d.ts`, pins the `ResourceLoader` interface to `dist/core/resource-loader.d.ts`, and states the re-validation MUST against those declarations on each Pi minor bump. The result is two normative homes for the same type signatures and declaration paths. The two currently agree, so no implementer would diverge today; the defect is structural — ownership is split, and a future edit to the authoritative `subagent.md` block (a signature change, a declaration-path move) that is not mirrored into Item 3 leaves the corpus self-contradicting with no rule indicating which location governs.

There is also a latent asymmetry that any consolidation must preserve: Item 3 routes re-validation to *both* checklist items `(o)` and `(aj)`, whereas the `subagent.md` satellite-pin block names only item `(o)`. The behavioural transcript-discard audit `(aj)` is therefore documented only on the capability-item side today.

**Verdict:** single-commit (with later accretion)
**Introducing commit:** `5d55c12` — *pi-loom spec: resolve "Subagent-spawn satellite types ResourceLoader and SessionManager.inMemory not pinned"* (2026-06-05)
**History:** The spec corpus is git-tracked. `git log -S 'static inMemory(cwd?: string): SessionManager' -- capability-inventory-items.md` and `git show 5d55c12` show that `5d55c12` introduced Item 3's satellite-pin prose: prior to it, Item 3 ended at the `AgentSession` member-surface pin, and the commit appended the full `static inMemory(cwd?: string): SessionManager` signature plus the `dist/core/session-manager.d.ts` / `dist/core/resource-loader.d.ts` declaration paths while also creating the cross-referenced `#subagent-spawn-satellite-types` anchor (originally on `provider-error-mapping.md`, later relocated into `subagent.md` at `6932599`). That commit established the dual normative home in one step. Two later commits widened the duplicated footprint inside Item 3 without removing it: `252acda` (2026-06-05) added the transcript-privacy presupposition text, and `9ced9f1` (2026-06-08) added the item `(o)`/`(aj)` re-validation-split prose. The originating defect — restating, rather than purely referencing, an owned pin — is `5d55c12`.

## Solution Space

**Shape:** single

### Recommendation

Reduce Item 3's satellite-pin sentence in `capability-inventory-items.md` to a pure cross-reference, e.g. "the spawn-call satellite type pins for `SessionManager.inMemory` and the loom-constructed `ResourceLoader` adapter — with their on-bump re-validation obligation — are owned by [spawn-block satellite type pins](./subagent.md#subagent-spawn-satellite-types)." Remove from Item 3 the restated `static inMemory(cwd?: string): SessionManager` signature, the `dist/core/session-manager.d.ts` and `dist/core/resource-loader.d.ts` declaration paths, and the duplicated `(o)`/`(aj)` re-validation specification, so the only normative declaration of these types lives in `subagent.md`. Retain in Item 3 the genuinely capability-scoped content: the transcript-privacy presupposition statement (that the in-memory session discards rather than persists the transcript and that the loom asserts no independent behavioural check).

Edge case the fixer must handle: `subagent.md`'s satellite-pin block currently names only re-validation item `(o)`, while Item 3 names both `(o)` and `(aj)` (the behavioural no-persistence audit). Before deleting the obligation from Item 3, confirm the full `(o)` + `(aj)` re-validation pointer is present in the `subagent.md` block (or in the version-bump checklist it cites); do not drop the `(aj)` behavioural-audit routing in the move.

## Consequence

**Severity:** advisory

Two reasonable implementers reading either page in isolation would today produce the same pin, so nothing breaks at loom 1.0. The cost is a maintenance hazard: the duplicated signature and declaration paths can silently drift on a future edit, and a reader cannot tell which of the two normative statements governs.
