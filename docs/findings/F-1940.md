## Finding

The `loom/load/invocation-cycle` test bullet in `V15b` (and its byte-identical twin in `V15b-T`) reads: "a static-resolution cycle fires at parse time; an unresolvable callee is a leaf (undetected until fixed)." Both halves of the trailing clause are underspecified for the implementer authoring the test.

"a leaf" is meant as *terminal node in the static-resolution graph* (no outgoing edges), but the word collides head-on with the plan's pervasive use of "leaf" to mean a plan task — `V15b` is itself a "leaf". A reader of the test bullet cannot tell which sense is intended, and the graph-node sense is never stated. "undetected until fixed" omits what is fixed, what re-triggers detection, and which prior diagnostic stands in for the suppressed cycle diagnostic.

The spec already pins this exactly in `invocation.md` §Cycle detection: an unresolvable callee is one that produced a `loom/load/callee-has-errors` diagnostic; the walker treats that node as a terminal node and the existing `callee-has-errors` diagnostic at every parent invoke / `tools:` site is the visible trace; a cycle routed through such a node is not detected until the underlying file is fixed and the watcher re-walks the graph. The test bullet should mirror that language so the assertion is unambiguous.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the `loom/load/invocation-cycle` test bullet in both `V15b` and `V15b-T` (keep them identical, as they are today) to mirror `invocation.md` §Cycle detection instead of the compressed jargon. Replace the current bullet text:

> `loom/load/invocation-cycle`: a static-resolution cycle fires at parse time; an unresolvable callee is a leaf (undetected until fixed).

with text along the lines of:

> `loom/load/invocation-cycle`: a static-resolution cycle (e.g. `A → B → A`) fires `loom/load/invocation-cycle` at parse time; an unresolvable callee — one that produced a `loom/load/callee-has-errors` diagnostic — terminates its own walk arm and is treated as a terminal node in the static-resolution graph, so a cycle routed through it is not detected until the underlying file is fixed and the graph is re-walked (the existing `callee-has-errors` diagnostic at each parent site is the visible trace meanwhile).

Drop the bare word "leaf" for the graph-node sense and the bare "undetected until fixed" clause. Naming `loom/load/callee-has-errors` ties the test to the spec's own discriminator for "unresolvable".

## Consequence

**Severity:** advisory

Two implementers authoring the `loom/load/invocation-cycle` test could diverge on what the unresolvable-callee arm asserts — or assert nothing for it — because "a leaf" reads as a plan task and "undetected until fixed" names no trigger. The spec is cited in the leaf's `Spec.` field and resolves the behaviour, so a careful implementer recovers the intent; the gap is guidance, not a blocker.
