## Problem

PIC-12 (host-interfaces-services.md, anchor `pic-12`) makes the timing-primitive ban enforceable by requiring that "a conforming implementation MUST carry a build-time check that rejects any direct lexical reference to those timing primitives outside the `WallClock` adapter … the presence of such a check and the references it rejects are the conformance criterion." The verb "carry" states only that a check exists, not that it runs: an implementation can contain a check script that is never wired into the build or CI, never executes, and never fails anything — and still satisfy "MUST carry a build-time check" while providing zero enforcement. The criterion never pins the property that makes the gate meaningful: that the check executes on every build (or CI run) and fails the build when it finds a violation, so two conformance reviewers can reasonably disagree on whether a present-but-unexecuted check conforms. The implementation-notes.md Clock bullet restates the same "MUST carry a build-time check" wording and inherits the identical gap.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `4556623` (2026-06-25, "pi-loom spec-plan: fix F-1178 — PIC-12 timing-primitive ban gets observable build-time conformance criterion")
**History:** The `Clock` / PIC-12 timing-primitive ban predates the defect. Before commit `4556623`, the build-time grep-test was explicitly **non-normative**: "*Non-normative implementation note.* The reference implementation enforces this ban with a build-time grep-test …; the grep-test mechanism is an implementation choice, **not an observable conformance point**." Commit `4556623` promoted the check to a normative requirement ("A conforming implementation MUST carry a build-time check … the presence of such a check and the references it rejects are the conformance criterion") and added the parallel "MUST carry a build-time check" wording into the implementation-notes.md Clock bullet. The "carry"/enforcement-cadence ambiguity entered the corpus with that promotion; `git log -S` for both "carry a build-time check" and "direct lexical reference" returns only this commit across both files.

## Solution approach

Rewrite PIC-12's "MUST carry a build-time check that rejects …" sentence (anchor `pic-12`) so the enforcement cadence and effect are pinned: a conforming implementation's check runs as part of every build / CI run and fails the build when it finds a direct lexical reference to the banned timing primitives outside the `WallClock` adapter. Mirror the same cadence wording in the implementation-notes.md Clock bullet so the two statements do not drift.

## Solution constraints

- The `WallClock` adapter's own `wallNow()` → `Date.now()` and timer delegations are the single permitted exemption and MUST remain carved out of the check.
- The rewrite pins enforcement cadence only: preserve the existing freedom of mechanism (grep / lint / equivalent) and the direct-lexical-reference-only scope (indirect/aliased/destructured/computed forms remain conceded to plan-side review).
