## Problem

`runtime-event-channel.md`'s `RuntimeEvent` payload declares `loom: string` with a type comment that names no source, whereas the sibling `invocation_id` comment was tightened to pin its `ActiveInvocationRegistry` source for every group-A event. The only place `loom` is pinned to the registry is the binder-scoped **Binder-failure sourcing** paragraph; read against the silence on the other five group-A kinds (`transport`, `model_tool`, `code_tool`, `tool_loop_exhausted`, `invoke_infra`), the carve-out framing creates a negative implicature that those kinds source `loom` elsewhere — e.g. from a `query_site`-derived or call-site name that can diverge from the registry's canonical-key form. `active-invocation-registry.md`'s downstream-consumer enumeration names `RuntimeEvent.invocation_id` but not `RuntimeEvent.loom`, so no page carries a general statement that every group-A `RuntimeEvent.loom` is registry-sourced. The divergence splits operator dedup buckets on `details.event.loom`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** b3bc4ce — spec: pin failure-observability surface (Cluster A, Option C) (2026-05-04, Thomas Andersen); 63bddcd — pi-loom spec: resolve cluster "T19a invocation_id correlation key" (+2 members) (2026-05-22, Thomas Andersen); 20f9645 — pi-loom spec: resolve "Binder-failure RuntimeEvents have no pinned source for invocation_id / loom" (2026-06-06, Thomas Andersen)
**History:** b3bc4ce introduced the `RuntimeEvent` type with `loom: string` and a type comment that pinned no source. 63bddcd added `invocation_id: string` and tightened *its* type comment to name the `ActiveInvocationRegistry` source, but did not do the same for the existing `loom` line — opening the asymmetry. 20f9645 then closed the binder-specific half by adding the **Binder-failure sourcing** paragraph that pins both fields to the registry for binder events, framed as a workaround for the missing `query_site` — which by exclusion leaves the five other group-A kinds without a general statement and creates the negative implicature this finding turns on.

## Solution approach

Tighten the `loom: string` type comment in the `RuntimeEvent` block to name its source — the `ActiveInvocationRegistry` entry's `loom` field — matching the sibling `invocation_id` comment's pattern, and correct its example to drop the leading `/` (which contradicts the registry's no-leading-`/` canonical-key rule). Add a general sourcing statement covering every group-A `RuntimeEvent` that reads `loom` from the registry entry, and demote the existing **Binder-failure sourcing** paragraph to the binder-specific consequence of that general rule. In `active-invocation-registry.md`'s *Registry contract* downstream-consumer enumeration, add `RuntimeEvent.loom` alongside the already-named `RuntimeEvent.invocation_id`.

## Solution constraints

- Out of scope: the `ActiveInvocationRegistry` container/field spelling changes owned by the sibling registry-contract finding; add only the `RuntimeEvent.loom` consumer entry.
- The canonical-key derivation rule is owned by `active-invocation-registry.md`; reference it rather than restating the full derivation.
