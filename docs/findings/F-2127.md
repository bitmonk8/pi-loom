## Finding

BNDR-5 pins a distinct rule for integral values of static type `number`: "an integral value MUST NOT carry a trailing `.0` (an integral `number` renders as `42`, not `42.0`)". The BNDR-6 reference-rendering table (`bndr-6a` … `bndr-6x`) is the normative oracle set conforming implementations MUST reproduce exactly, yet it carries no row that isolates this rule at an ordinary magnitude. The relevant `number`-typed rows are:

- `bndr-6q` — `3.14` (number) → `3.14`: non-integral, exercises the fractional-digit pin, not the no-`.0` rule.
- `bndr-6r` — `1e21` (number) → `1000000000000000000000`: integral, but its purpose is the forbidden large-magnitude scientific-notation expansion; the no-`.0` behaviour is incidental and conflated with full fixed-point expansion.
- `bndr-6s` — `1e-8` (number) → `0.00000001`: non-integral small-magnitude expansion.
- `bndr-6o` — `42` (integer) → `42`: static type `integer`, governed by BNDR-4, not the BNDR-5 `number` rule.

An implementer building a test or coverage-matrix leaf for "integral `number` must not carry `.0`" at a non-scientific-notation magnitude has no dedicated, citable pin. The closest integral-`number` row (`bndr-6r`) couples the rule to scientific-notation expansion, so a renderer that wrongly emitted `42.0` for a `number`-typed `42.0` would pass every existing `number` row except by inference. The inline `42` example inside the BNDR-5 prose is not a normative table row and carries no stable sub-letter anchor, so it cannot be cited as an oracle.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `e3ddb6f` (introduced the scalar non-string reference-rendering pairs, including the `3.14` (number) and `42` (integer) rows, with no integral-`number` row); the gap persisted through `a9e48a2` (added the `1e-8` small-magnitude row), `291447a` (converted the pairs to the sub-lettered `bndr-6a … bndr-6x` table), and `e2dc186` (number-rendering derivation recipe), none of which added an ordinary-magnitude integral-`number` row.
**History:** `git log -S '\`3.14\` (number)'` traces the reference pair to `e3ddb6f`; the pre-image of `291447a` confirms the `42` (integer) / `3.14` / `1e21` / `1e-8` rows already existed without a dedicated integral-`number` row. The omission has been present in the reference set since the number rows were first introduced and was carried forward unchanged across every later revision of the table.

## Solution Space

**Shape:** single

### Recommendation

Append one new BNDR-6 reference row exercising an ordinary-magnitude integral `number`: value `42.0` (declared static type `number`, integral value) → renders as `42`. Per the table's own append rule ("a later row append takes the next free sub-letter and existing sub-letters are not renumbered"), assign it the next free sub-letter `bndr-6y` and place it at the end of the table; do not renumber existing rows. Update the BNDR-6 intro sentence's anchor range from "`bndr-6a` … `bndr-6x`" to "`bndr-6a` … `bndr-6y`". This gives the no-trailing-`.0` rule a dedicated, citable oracle at a non-scientific-notation magnitude, symmetric with the existing non-integral (`bndr-6q`) and large-magnitude integral (`bndr-6r`) coverage. Downstream, `V11h`'s "6a–6x" reproduction range becomes "6a–6y" and `V2d` gains the `42.0` → `42` vector.

## Consequence

**Severity:** advisory

The behavioural contract is unambiguous in BNDR-5 prose, so two implementers will not diverge on what to emit. The gap is a missing normative test oracle: there is no citable BNDR-6 row that exercises the integral-`number` no-`.0` rule in isolation at an ordinary magnitude, so a regression that emits `42.0` could pass the reference set by inference rather than direct assertion.
