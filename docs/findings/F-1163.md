## Problem

The opening *Implementation gate* blockquote scopes the block by file ("the obligations on this page"), making it the only gate in the page family that scopes by page rather than by clause. Read literally it sweeps every normative clause on the page — including the *Drain-state tag write* setter contract for `LoomRegistry.markRuntimeDegraded` and the *Isolation* cross-reference target — yet two sibling pages (`drain-state-contract.md`, `patch-skew-degradation.md`) already encode portions of that same surface as unblocked, buildable contract shipping under `V9m`/`V9g`. An implementer cannot tell which page-local definitions are inside the gate, which of the acceptance-criteria fixtures (a)–(m) may be written before clause (a) resolves, or which page-local definitions sibling pages would need to relocate should clause (a) resolve `governed-by-rebind`.

The PIC-page gate landed as the bare sentence *"The obligations on this page are blocked pending resolution of the open contradiction …"* without any enumeration of which page-local clauses it scopes; `80fcd63` later appended the `provably-differs`-verdict clearance condition but did not narrow the "obligations on this page" referent. `git log --follow` confirms the sibling pages' relevant clauses (`drain-state-contract.md`'s `markRuntimeDegraded` definition, `patch-skew-degradation.md`'s `interstitial-seam-tag-write-swallow` rule) predate `b8f004f` and were left untouched, so the scope mismatch is purely a property of the new gate's wording rather than a change to the siblings.

## Solution approach

Rewrite the *Implementation gate* blockquote so it enumerates which page-local clauses are gated versus non-gated, rather than scoping the block by file. Classify the *Drain-state tag write* paragraph (as the method-level `LoomRegistry.markRuntimeDegraded` setter contract) and the *Isolation* cross-reference target as non-gated — these are the API/seam contracts `V9m`/`V9g` and the sibling pages consume — and the page's branch-behaviour clauses and acceptance fixtures (a)–(m) as gated. Add a normative rider that any newly-added clause defaults to gated unless explicitly enumerated as non-gated, so the scope cannot drift back into ambiguity.

## Solution constraints

- Out of scope: `drain-state-contract.md` and `patch-skew-degradation.md` — do not edit them; their existing inbound references must resolve into the non-gated subset.
- Out of scope: the clause-(a) release condition on `host-prerequisites.md` (owned by F-1166); this edit changes only the gate's scope, not its release condition.
