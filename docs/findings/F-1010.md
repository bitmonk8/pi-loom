## Problem

The parse-time `loom/parse/empty-template` predicate names *every literal segment between interpolations* (plural, non-contiguous) and then names *newline-trim and dedent* (operations defined over a single string), but does not say how the two compose once interpolations split the body. At least three readings are consistent with the present prose and fire the warning on different templates — the dedent step is load-bearing, because dedent's common-prefix algorithm is defined over the line set of one string, and the line set of the segments is not the line set of the concatenation once an interpolation sits mid-line. The runtime short-circuit in the same section is already pinned to operate on the fully-rendered post-interpolation string and so has no ambiguity; the parse-time side inherited the unresolved composition question.

## Issue introduction

**Verdict:** present-since-inception

**History:** The "Degenerate rendered templates" subsection was added wholesale in `82be125`, with the parse-time bullet already worded as *"every literal segment between interpolations, after newline-trim and dedent — is empty or whitespace-only"*. The composition question across non-contiguous segments was unspecified from that first commit. Two later refinements touched adjacent ambiguities in the same bullet without addressing this one: `f14a45e` (2026-06-05, *empty-template / slash-argument whitespace set unpinned*) pinned the ASCII whitespace set, and `2637aec` (2026-06-20, *F-0755 — empty-template predicate pinned to pre-escape source segments*) pinned the pre-escape evaluation order. Neither commit altered the "every literal segment … after newline-trim and dedent" phrasing.

## Solution approach

Clarify the `loom/parse/empty-template` parse-time bullet in the `#degenerate-rendered-templates` section to specify how newline-trim and dedent compose across non-contiguous literal segments. Define the static body as the source text between the backticks with each `${…}` interpolation source span replaced by the empty string, then evaluate the predicate by running the existing newline-trim→dedent pipeline (pinned under `Dedent and newline-trim — normative behaviour`) on that single string and testing the result for emptiness or whitespace-only against the ASCII set already cited at `#system-note-rendering` rule 1. Add a worked vector that locks the chosen reading observationally.

## Solution constraints

- Out of scope: the runtime short-circuit bullet, which already operates on the fully-rendered post-interpolation string — do not alter its semantics.
- Preserve the pre-escape evaluation order and the literal-escape suppression hatch already pinned in the bullet (the predicate evaluates on the source before escape rewrites).
