## Problem

PIC-8's restore-failure protocol pins a byte-verbatim `display: true`
`loom-system-note` template whose only substitutable token, `<name>`, is never
defined — the paragraph states "only `<name>` is substituted" but does not say
what `<name>` denotes. The corpus carries several plausible candidate bindings
(the slash-command name without leading `/`, the `loom` canonical-key field on
the `ActiveInvocationRegistry` entry, a filename-derived stem, the registered
tool name), each yielding a different operator-log string for the same restore
failure. Because the whole string is byte-stable outside this placeholder, two
conforming implementers can emit literally different operator strings for the
same event, violating the operator-side source-language-stability observable.
The companion `code-registry-runtime.md` row for
`loom/runtime/active-set-restore-failed` quotes a parallel `<name>` token that
is likewise unbound and not cross-referenced.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `2d2fc8e` ("pi-loom spec: resolve \"Pi SDK capability calls have no failure contract\"")
**History:** `git log -S 'failed to restore tool active-set after'` against the
spec corpus identifies `2d2fc8e` as the commit that first authored the PIC-8
restore-failure paragraph (then in the pre-shard
`spec_topics/pi-integration-contract.md`, later moved into
`pi-integration-contract/tool-registration-lifetime.md` by the
shard-collapse commit `81bf52a`). The introducing commit added the verbatim
template `loom: failed to restore tool active-set after /<name>; …` together
with the "only `<name>` is substituted" substitution rule, but did not bind
`<name>`. Every subsequent edit to the paragraph (`d0083d7`, `19b193b`,
`4d8dc02`, `823ce54`) reshapes neighbouring sentences without touching the
placeholder's definition. The defect is therefore co-introduced with the
PIC-8 paragraph itself rather than regressed in later editing.

## Solution approach

Bind `<name>` in the PIC-8 restore-failure paragraph (`id="pic-8"`) to the
invocation's canonical-key slash name — the `loom` field on the active
`ActiveInvocationRegistry` entry (`active-invocation-registry.md`) — rendered
per the `<slash-name>` rule in `placeholder-rendering-b.md`. Reconcile the
parallel `<name>` token on the `loom/runtime/active-set-restore-failed`
Message template in `code-registry-runtime.md` to the same binding so both
surfaces resolve to byte-identical identifiers for the same event.

## Solution constraints

- The PIC-8 `loom-system-note` template and the `code-registry-runtime.md`
  Message template are distinct wire strings; reconcile only the `<name>`
  binding, not the surrounding template text.
