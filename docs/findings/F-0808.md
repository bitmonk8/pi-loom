## Finding

PIC-8 defines a `display: true` `loom-system-note` whose `content` is the verbatim
template

> `loom: failed to restore tool active-set after /<name>; the user session may have unexpected tools active. Run /reload to reset.`

with the explicit substitution rule *"only `<name>` is substituted — every other
character ships verbatim."* The paragraph then never says what `<name>` denotes.
The corpus carries several plausible candidates for a `<name>`-shaped slot —
the slash-command name without leading `/` (the convention used by every
sibling `loom /<name>` template in `defaulting-system-note-echo.md`,
`determinism-cancellation-failure.md`, `error-model.md`, and
`runtime-event-channel.md`), the `entry.loom` canonical-key field on the
`ActiveInvocationRegistry` entry, a frontmatter / filename-derived stem, or
the registered tool name on a synthesised tool — each yielding a different
operator-log string for the same restore failure.

The companion `code-registry-runtime.md` registry row for
`loom/runtime/active-set-restore-failed` quotes a different *Message* template
(`failed to restore tool active-set after /<name>: <error>`) and the
placeholder-rendering rules in `placeholder-rendering-b.md` §7 govern only
`<slash-name>`, not the bare `<name>` token used here. Nothing in the PIC-8
paragraph cross-references either of those, and the substitution-rule
sentence's *"only `<name>` is substituted"* framing actively discourages an
implementer from reading the answer out of an adjacent template.

Because the entire string is pinned as byte-verbatim except this one
placeholder, the placeholder's binding is part of the byte-stable wire
contract: two implementers picking different reasonable resolutions
(`code-review` vs. `Code-review` vs. `code-review.loom`) emit literally
different operator strings for the same event and silently violate the
operator-side source-language-stability observable for this note.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `2d2fc8e` ("pi-loom spec: resolve \"Pi SDK capability calls have no failure contract\"")
**History:** `git log -S 'failed to restore tool active-set after'` against the

restore-failure paragraph (then in the pre-shard
`spec_topics/pi-integration-contract.md`, later moved into
`pi-integration-contract/tool-registration-lifetime.md` by the
shard-collapse commit `81bf52a`). The introducing commit added the verbatim
template `loom: failed to restore tool active-set after /<name>; …` together
with the "only `<name>` is substituted" substitution rule, but did not bind
`<name>`. Every subsequent edit to the paragraph (`d0083d7`, `19b193b`,
`4d8dc02`, `823ce54`) reshapes neighbouring sentences without touching the
placeholder's definition. The defect is therefore co-introduced with the
PIC-8 paragraph itself rather than regressed in later editing.

## Solution Space

**Shape:** single

### Recommendation

Bind `<name>` to the **invocation's canonical-key name** — the same final
derived slash name (post-`pi.looms` mapping, post-`as` rename, stem taken
verbatim, **no leading `/`**) that the `ActiveInvocationRegistry` entry's
`loom` field already carries per
`active-invocation-registry.md`, and that the surrounding `loom /<name>`
template family (panic notes, binder failure rows, `cancelled-by-session-shutdown`
template) uses uniformly. Concretely, add a parenthetical to the PIC-8
paragraph's substitution-rule sentence:

> `<name>` is the invocation's canonical-key slash name (the `loom` field on the
> active `ActiveInvocationRegistry` entry — post-`pi.looms` mapping, post-`as`
> rename, no leading `/`), rendered per the `<slash-name>` rule in
> [Diagnostics — Placeholder rendering §7](../diagnostics/placeholder-rendering-b.md#7-identifier--descriptor--and-closed-enum-placeholders);
> the leading `/` in the template is the template's, not the placeholder's.

Reconcile the parallel `<name>` token on the
`loom/runtime/active-set-restore-failed` *Message* template in
`code-registry-runtime.md` to use the same binding (either rename the
placeholder to `<slash-name>` to match §7's closed enumeration, or carry the
same parenthetical), so both surfaces resolve to byte-identical identifiers
for the same event.

Implementer edge cases:

- For a prompt → prompt cross-mode `invoke(...)` whose snapshot/restore
  window belongs to the *child* prompt-mode body, the rendered name is the
  child's canonical-key name, since that is the entry whose snapshot/restore
  `finally` threw — not the slash-command name the operator dispatched.
- Subagent-mode invocations cannot trigger PIC-8 (the paragraph already
  excludes them), so the binding never needs to resolve against a
  subagent-only or `tool.execute(...)`-only canonical key.

## Consequence

**Severity:** correctness

Two reasonable conforming implementers will produce literally different
`loom-system-note.content` strings for the same restore failure (e.g.
`… after /code-review; …` vs. `… after /Code-review; …` vs.
`… after /code-review.loom; …`). The note is pinned as byte-verbatim outside
the one placeholder, so the divergence is observable to operators and to the
operator-side source-language-stability observable that
`overview-and-orientation.md` already pins. The downstream `V9f` test row
cannot author a passing assertion without picking a resolution unilaterally.
