## Finding

`session-only-degraded-state.md` canonicalises one half of the teardown-vs-session-swap `event.reason` partition as **"session-only reasons"** (used uniformly for `{"new","resume","fork"}` across the opener, the *Predicate split* table, the *Inline triplet is normative* paragraph, the acceptance criteria, and the *Emergent fast-path* clause), but its symmetric `{"quit","reload"}` complement is named three different ways within the same file: **"always-tear-down reasons"** (acceptance criterion (c)), **"`quit`/`reload` sub-cases"** (the *Harmlessness* paragraph in *Predicate split*), and **"always-tear-down branch"** (*Emergent fast-path*). The companion files use yet two more variants — **"always-tear-down half"** in `host-prerequisites.md` presupposition (d) and `version-bump-step2.md` checklist item (x), and **"teardown half"** in `host-prerequisites.md` (d) — so the corpus-wide variant count is five.

Both halves of the partition are co-equal load-bearing concepts: presupposition (d) of the host-prerequisites contract is literally *about* the partition assignment, the predicate split's broad tag-transition predicate is the complement check `capturedEventReason ∉ {"quit","reload"}`, and the version-bump inbound-reference sweep audits both halves with symmetric greps. The asymmetric naming (one half pinned, the other variant-named) makes prose harder to scan, weakens the rhetorical pairing the predicate split depends on, and risks the next editor introducing a sixth variant rather than noticing the canonical term is missing.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `52b9101` (2026-05-11, "pi-loom spec: resolve T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state") and `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set")
**History:** `git log -G "always-tear-down"` shows the term family entered the corpus in `52b9101` — the same commit that introduced the degraded-state branch and canonicalised "session-only reasons" for the `{"new","resume","fork"}` half. That commit already shipped three variants in parallel ("always-tear-down reasons", "always-tear-down branch", "always-tear-down half"), so the asymmetric naming is present-since-inception of the branch contract itself; the canonical name on one half and the variant family on the other arrived in lockstep. The later split commit `f5e89f4` (verified by `git log -G "Harmlessness on"`, which returns only that commit) extracted `session-only-degraded-state.md` from the monolithic `pi-integration-contract.md` and, in the condensation pass, introduced a fourth variant — "`quit`/`reload` sub-cases" in the new *Harmlessness* paragraph — without unifying any of the inherited three. The defect is therefore the *interaction* of those two commits: `52b9101` permitted the asymmetry by canonicalising only one half, and `f5e89f4` widened it under a refactor that touched the surrounding paragraphs without auditing the complement-name family.

## Solution Space

**Shape:** single

### Recommendation

Coin **"teardown reasons"** as the canonical name for `{"quit","reload"}`, introduce it once at the file opener of `session_topics/pi-integration-contract/session-only-degraded-state.md` (parallel to the existing inline definition of "session-only reasons"), and sweep every variant phrase in the four files listed under *Spec Documents* to that one term. Concretely:

- In `session-only-degraded-state.md`: rename "always-tear-down reasons" → "teardown reasons" in acceptance criterion (c); rename "`quit`/`reload` sub-cases" → "teardown sub-cases" (or "teardown reasons") in the *Harmlessness* paragraph; rename "always-tear-down branch" → "teardown branch" in the *Emergent fast-path*.
- In `host-prerequisites.md` presupposition (d): rename "teardown half" and "always-tear-down half" → "teardown reasons" (keeping "half" only where the prose is genuinely about the *partition halves* as a pair).
- In `version-bump-step2.md` checklist item (x) and `version-bump-triggers.md` step 5: rename "always-tear-down half" → "teardown reasons".

Edge cases the implementer must watch:

- The literal set `{"quit","reload"}` is normative and MUST NOT be replaced by the prose name at any of the seam sites — the inline `{"new","resume","fork"}` triplet is explicitly pinned by the *Inline triplet is normative* paragraph as a normative runtime literal, and the complement seam at the `markRuntimeDegraded` call site (`capturedEventReason ∉ {"quit","reload"}`) is similarly load-bearing. The sweep replaces only the *prose label* for the set; every existing site that already writes `{"quit","reload"}` stays as-is.
- "Session-only reasons" is the established term for the other half — do not rename it. The goal is symmetry, not a fresh coinage on both halves.
- `version-bump-triggers.md`'s inbound-reference-sweep paragraph audits the `{"quit","reload"}` complement via a symmetric grep; that grep recipe keys on the literal strings, not the prose label, so it is unaffected by the rename.
- "Always-tear-down" appears once in `version-bump-triggers.md` inside the parenthetical *example* "(e.g. making `"reload"` session-only, or `"fork"` always-tear-down)" — that phrase is an adjective describing what Pi would do to the literal, not a label for the complement set, and should stay unchanged.

## Consequence

**Severity:** cosmetic

A reader scanning the file cannot use a single search term to find every site that discusses the `{"quit","reload"}` half, and the asymmetry with the pinned "session-only reasons" gives the next editor cover to invent a sixth variant. No runtime behaviour, predicate, diagnostic code, or test fixture changes either way — the load-bearing seams use the literal set `{"quit","reload"}`, not these prose labels.
