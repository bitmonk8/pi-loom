## Finding

GOV-15's *Tighten* clause makes *Tighten* permissible within loom 1.x only "when no input in the loom 1.x conformance fixture suite (once it ships) is made to fail by the change," and pins the present-state prohibition with "Until that suite ships, *Tighten* is forbidden under loom 1.x." Neither sentence defines what counts as "ships" for the suite: a merged commit on `main`, a tagged loom 1.x release that includes the suite, a CI gate wired into the release pipeline, or some other observable.

The suite itself is named only as a deferred post-loom 1.0.0 follow-up at the `gov-15-fixture-suite` anchor on this same page, with no companion definition of its shipping state and no forward-pointer to a release-process artefact that would distinguish "exists" from "shipped." A maintainer evaluating whether a proposed *Tighten* edit is permissible in some future loom 1.x release therefore cannot mechanically determine whether the gate has been crossed; two maintainers can reasonably disagree.

The predicate also gates the secondary clause "Once *Tighten* becomes permitted (per the conformance-fixture-suite condition above)" that inherits the GOV-30 lock-step and *Audit methodology* release-gate obligations onto *Tighten*, so the same vagueness reaches the procedural-inheritance trigger.

## Solution Space

**Shape:** single

### Recommendation

Replace the undefined `ships` predicate on both sites with a release-anchored crisp condition keyed to a stable spec anchor for the suite. Concretely, in the *Tighten* clause of `docs/spec_topics/governance/source-language-stability.md`:

- Define a single named predicate inline (e.g. *suite-shipped*): "the conformance fixture suite is *shipped* once a loom 1.x release tag exists whose tree contains the suite as named at [GOV-15 fixture suite](#gov-15-fixture-suite), and that suite is referenced from `Future Considerations` as the active loom 1.x conformance baseline."
- Rewrite the two affected sentences to use the named predicate: "*Tighten* is permitted within loom 1.x only when no input in the *shipped* conformance fixture suite is made to fail by the change" and "Until the suite is *shipped*, *Tighten* is forbidden under loom 1.x …"
- Carry the same predicate name into the trailing inheritance sentence: "Once *Tighten* becomes permitted (per the *suite-shipped* condition above) …"

Edge cases the implementer must watch:

1. The predicate must pin to a *release tag*, not merely a `main` commit, so a suite present in development but not yet released does not silently flip the gate.
2. The predicate must survive suite-page renames: anchor the definition to the `#gov-15-fixture-suite` anchor on this page, not to a file path that may later move.
3. Leave the `(once it ships)` parenthetical out entirely once the inline predicate is named — the parenthetical was the surface that admitted the original ambiguity.
4. No GOV-30 lock-step aggregator update is required: the aggregator bullet at `overview-and-orientation.md#source-language-stability` already defers the *Tighten* gate to GOV-15 by reference rather than restating it.

## Consequence

**Severity:** advisory

In loom 1.0 itself *Tighten* is unconditionally forbidden, so the defect produces no immediate behavioural divergence. The risk realises in a future loom 1.x release that proposes a *Tighten* edit: two reviewers can legitimately disagree on whether the conformance fixture suite has "shipped," yielding inconsistent admissibility verdicts on the same proposed edit. The same vagueness silently delays the secondary inheritance trigger that brings *Tighten* under GOV-30 lock-step and the *Audit methodology* release gate.
