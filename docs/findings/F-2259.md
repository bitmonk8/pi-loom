## Problem

The version-bump editorial-review checklist routes four audits at the same fragile evidence surface: items (e), (r), (aj), and (am) each diff or read a Pi-internal `dist/core/*.js` body against the loom 1.0-pin Baseline-acquisition snapshot (`#bump-baseline-acquisition`). Only item (e) carries a conservative-posture rule forbidding a `pass`/`N/A` outcome when that evidence trail has drifted and the body cannot be located or diffed. Items (r), (aj), and (am) carry no such guard, and the generic per-item recording shape permits `pass` on textual-locatability grounds alone. A contributor who cannot perform the diff/read therefore has spec licence to record a false `pass`, silently clearing a real behavioural regression with no build-time SDK surface-inventory signal.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** `57d8235` (2026-05-11, "resolve T23 per-session slash-handler serialisation citation") established the item-(e) conservative-posture rule; the evidence-trail-fragile sibling audits accreted afterwards without an equivalent guard — `257c545` (2026-06-05, "Host presuppositions lack version-bump-procedure re-audit hooks") added item (r); `9ced9f1` (2026-06-08, "subagent transcript-discard not routed to bump checklist") added item (aj); `077afb5` (2026-06-20, "tools-allowlist suppression pinned to new bump checklist item (am)") added item (am).
**History:** The defect is not a single-commit slip. The conservative-posture recording guard was authored as item-(e)-local prose. Each later commit added a new JS-body audit reading against the same `dist/*.js` Baseline-acquisition snapshot but reused only the diff/read instruction, not the item-(e) recording guard, so the asymmetry widened with every sibling added. The durable fix targets the interaction — a shared recording guard covering every `dist/*.js`-fragile audit — rather than any one introducing commit.

## Solution approach

Add a cross-item recording guard, stated once, that applies the conservative-posture rule to every audit reading or diffing against the Baseline-acquisition `dist/*.js` snapshot — items (e), (r), (aj), (am). Anchor it where both arms already reference: either the **Baseline acquisition** paragraph (`#bump-baseline-acquisition`) or the **Outcome recording conventions** block (`#bump-step-2-outcome-recording-conventions`). Mirror item (e)'s symmetric pass-evidence requirement so a `pass` on (r)/(aj)/(am) is valid only when its rationale cites a specific candidate-minor code site supplying the behavioural evidence.

## Solution constraints

- Do not promote (r)/(aj)/(am) from SHOULD to MUST: `not-performed` with a rationale remains a conformant non-blocking outcome; the guard forbids only a textual-locatability `pass`/`N/A`.
- Out of scope: item (e)'s own existing MUST-level rule and the recovery-serialisation merge gate it feeds — generalise its recording arm, do not weaken or replace (e)'s stronger obligations.
