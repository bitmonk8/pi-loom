## Problem

The inbound translation pass (`runtime-value-model.md`, *Wire-name translation* → *Inbound*) walks the AJV-validated JSON value and, at each string position, consults the lowering pass's *Named-enum positions* sidecar to decide whether to reattach a declaring-enum tag. That sidecar is keyed by JSON Pointer into the lowered schema fragment (`schema-subset.md` Lowering Algorithm step 5), but the inbound pass walks the data value; the two address spaces do not coincide and the procedure mapping a data position to its schema pointer is never specified. The gap is concrete at three crossings the walk must make: array indices versus the single `items` subschema, `anyOf` arm selection with no emitted `discriminator`, and `$ref`/`$defs` boundaries the data walk passes through without a structural marker. Because the correlation is undefined, two conforming implementations can arrive at different schema pointers for the same string and so diverge on enum tagging — and on the wire-name rebuild (step (a)), which rides the same descent.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** fae85d3 — Tighten spec for implementation-plan readiness (2026-05-04, Thomas Andersen); 072d3fe — pi-loom spec: resolve "Inbound walk does not state when enum brands get reattached" (2026-05-04, Thomas Andersen); 7f217a8 — pi-loom spec: resolve "Language-core hidden assumptions: enum-tag sidecar, AJV-config silence, Markdown claim" (2026-06-07, Thomas Andersen)
**History:** The inbound walk first appeared at fae85d3, already rebuilding loom-side names "using each schema's translation map" without defining how a data position correlates to a schema — so the correlation gap is present in the section's first substantive commit. 072d3fe layered enum-tag reattachment onto the same walk, and 7f217a8 reformulated the consumed structure as a sidecar **keyed by JSON Pointer into the lowered schema**, making the data-space/schema-space mismatch acute. No commit ever specified the descent across `items`, `anyOf` arm selection, or `$ref`/`$defs`; the defect deepened by accretion but was never introduced by a regression.

## Solution approach

Rewrite the `runtime-value-model.md` *Wire-name translation* → *Inbound* bullet to specify how a data position correlates to the schema pointer keying the *Named-enum positions* sidecar: a schema-guided parallel descent of the validated value and the lowered schema that maintains the current schema position as a canonical `$defs`-rooted JSON Pointer matching the sidecar's key form. Resolve the three crossings the walk must make — array indices collapsing onto the single `items` subschema, deterministic `anyOf` arm selection, and `$ref`/`$defs` resolution into canonical pointer form. State that the same descent drives the wire-name rebuild (step (a)) so both consumers share one traversal.
