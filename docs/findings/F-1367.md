## Finding

`V11a` Adds pins a specific binder-model resolution outcome: a bare `modelId` matching models under more than one provider is ambiguous and "resolv[es] to no model, not pick-first". This is normative — the spec's binder-model parse rule (`binder-model-and-context.md#binder-model-parse-rule`) states the resolver "treats a bare `modelId` that matches models under more than one provider as ambiguous, resolving to no model (not pick-first)", and that an ambiguous string "fails to load with `loom/load/binder-model-unresolved`".

The only `V11a` / `V11a-T` Tests bullet that covers this code reads: "`loom/load/binder-model-unresolved` fires when no exact model reference matches." That phrasing asserts only the zero-match case. A binder-path implementation that pick-firsts on a multi-provider bare `modelId` (rather than rejecting it) would still satisfy this bullet, because such an input is not a "no exact match" case — it is an over-match the bullet never names. The "not pick-first" decision pinned in Adds therefore has no direct binder-path assertion behind it.

The outcome is caught only transitively: `V6a-T`'s `loom/load/model-unresolved` bullet does enumerate "a bare `modelId` ambiguous across providers", and the single-matcher instance-identity wiring forces the binder path to inherit `V6a`'s resolver behaviour. The defect is the gap between `V11a` Adds and its own test bullet, not an end-to-end coverage hole — but a reader auditing the binder path in isolation cannot see the ambiguity outcome being exercised.

The gap is the un-propagated half of that commit: the obligation was added to Adds and to the sibling `V6a-T` leaf without tightening `V11a`'s own assertion.

## Solution Space

**Shape:** single

### Recommendation

In both `docs/plan_topics/V11a-binder-model-resolution.md` and `docs/plan_topics/V11a-T-binder-model-resolution.md`, tighten the `loom/load/binder-model-unresolved` Tests bullet so it names the ambiguous-bare-`modelId` input alongside the existing zero-match input and asserts the "resolves to no model, not pick-first" outcome directly on the binder path — mirroring the enumeration already present in `V6a-T`'s `loom/load/model-unresolved` bullet.

Concretely, the current bullet text

> `loom/load/binder-model-unresolved` fires when no exact model reference matches.

should be replaced with a bullet that asserts the code fires both when no exact model reference matches and when a bare `modelId` matches models under more than one provider — the latter resolving to no model (not pick-first) — e.g.

> `loom/load/binder-model-unresolved` fires when no exact model reference matches, and when a bare `modelId` matches models under more than one provider (the ambiguous case resolves to no model, not pick-first).

Apply the same edit to the paired `V11a-T` bullet so the failing-test leaf and its implementation leaf stay in lockstep. No spec edit is required — the spec already pins the outcome at `binder-model-and-context.md#binder-model-parse-rule`; this is solely a plan-side test-bullet tightening.

## Consequence

**Severity:** advisory

A faithful implementer writing the `V11a` binder-model test from the leaf alone could assert only the zero-match firing and leave the multi-provider-bare-`modelId` over-match path unverified on the binder side; the pinned "not pick-first" decision is then witnessed only through the shared-matcher inheritance from `V6a-T`, invisible to anyone reviewing the binder path in isolation. Production behaviour stays correct as long as the single-matcher wiring holds, so this is a verification-clarity gap rather than a correctness break.
