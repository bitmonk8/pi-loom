## Finding

`V2a`'s **Adds** field promises "`array<T>` type-sink resolution over the exhaustive sink set." The spec enumerates that set in [`grammar.md` §`array<T>` literal type-sink rule](../spec_topics/grammar.md) as four positive sink kinds: (1) a binding annotation (`let xs: array<T> = ...`), (2) a function parameter type at a call site, (3) the declared type of a surrounding constructor field, and (4) the element type of an array-typed sink the literal is nested in (recursive descent) — plus the explicit negative that a `for` iterand is **not** a sink.

`V2a`'s **Tests** and **Ships when** verify only the negative: `loom/parse/array-no-common-type` fires for an `[]` with no resolving sink, with a parenthetical that a `for` iterand is not a sink. No vector exercises the *success* path for any of the four enumerated sink kinds — i.e. that an array literal actually resolves its element type against a binding annotation, a call-site parameter type, a constructor field, or a nested array sink.

This gap is not caught downstream. Positive sink resolution succeeds silently — it fires no diagnostic — so the code-keyed closing gate (`grammar.md` (GRAM) → `V2a` in [`coverage-matrix.md`](./coverage-matrix.md)), which closes when green tests assert the listed `loom/parse/*` codes, cannot observe whether positive resolution works at all. The leaf can therefore go green while resolution against one or more sink kinds is missing or wrong.

## Solution Space

**Shape:** multiple

### Option A — Add a positive resolution vector per enumerated sink kind

**Approach.** Extend `V2a-T`'s and `V2a`'s **Tests** so that positive `array<T>` resolution is verified against each of the four sink kinds enumerated in [`grammar.md` §`array<T>` literal type-sink rule](../spec_topics/grammar.md) (binding annotation, call-site parameter type, constructor field, recursive array-element descent), and update **Ships when** to assert that positive resolution.
**Plan edits.** `V2a-T-type-grammar.md` and `V2a-type-grammar.md`: add the positive resolution vectors to the **Tests** bullets and reflect them in **Ships when**.
**Spec edits.** None.
**Pros.** Backs the "exhaustive sink set" claim with observable coverage; closes the blind spot where positive resolution fires no diagnostic and so escapes the code-keyed closing gate.
**Cons.** Enlarges `V2a`'s test surface, which is already flagged as oversized (finding #73).
**Risks.** If `V2a` is later split per #73, the new vectors must move with the sink-resolution concern.

### Option B — Scope the Adds claim to what the tests verify

**Approach.** Rephrase `V2a`'s **Adds** so it no longer claims verified resolution "over the exhaustive sink set," citing instead the spec-enumerated sink set and the no-sink / `for`-exclusion behaviour the tests actually exercise.
**Plan edits.** `V2a-type-grammar.md`: reword the **Adds** sink-resolution clause.
**Spec edits.** None.
**Pros.** Minimal diff; removes the overclaim without growing the test set.
**Cons.** Leaves positive resolution unverified anywhere in the plan; because success fires no diagnostic, a missing or wrong resolution path for any sink kind ships undetected by the code-keyed gate.
**Risks.** Positive resolution silently unimplemented or incorrect for one or more sink kinds.

### Recommendation

Take Option A. Positive sink resolution emits no diagnostic on success, so the code-keyed closing gate (`grammar.md` → `V2a`) cannot detect a missing or wrong resolution path; only explicit positive vectors covering each enumerated sink kind close that gap, and Option B leaves it open. Cite [`grammar.md` §`array<T>` literal type-sink rule](../spec_topics/grammar.md) as the source for the kinds to cover. Edge case the implementer must watch: the recursive-descent sink kind (an array literal nested as an element of an array-typed sink) must be exercised distinctly from the plain binding-annotation kind — they share the binding-annotation entry point but resolve through different walk steps.

## Consequence

**Severity:** correctness

Two reasonable implementers could ship `V2a` green while disagreeing on — or omitting — positive resolution for the constructor-field or recursive-descent sink kinds, because success fires no diagnostic and the only acceptance vector is the no-sink failure case. The "exhaustive sink set" claim in **Adds** is unbacked by any test, so a leaf that resolves only the binding-annotation kind would still satisfy **Ships when**.
