## Problem

`V9n`'s **Adds** field states the synthesised `transport` error's `provider` field "is sourced from `V9j`'s provider-error-mapping surface, not re-derived here." But the prompt-mode transport paths `V9n` owns (the post-`waitForIdle()` `stopReason:"error"` probe and the synchronous-`pi.sendUserMessage`-throw secondary mapping) issue no pi-ai `complete()` call, so `V9j`'s provider-error classifier never runs there to supply the value. `V9n`'s own first **Tests** bullet (and `V9n-T`) already state the correct source: the loom's resolved `model:` `Model<Api>.api` value (resolution owned by `V6a`). The Adds clause therefore contradicts the leaf's asserted behaviour and leaves the prompt-mode `provider` field with no wired runtime source.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 0c3a2a2 — pi-loom plan: resolve "V9c bundles three concern clusters in one leaf pair" (2026-06-17, Thomas Andersen)
**History:** The misattribution entered with commit 0c3a2a2, which split `V9n` out of `V9c` and authored `V9n`'s new **Adds** clause ("`provider` … is sourced from `V9j`'s provider-error-mapping surface"). One commit earlier the same day, c04b0a6 ("resolve 'prompt-mode TransportError.provider cites V9j's classifier'") had fixed exactly this misattribution in the `V9c`/`V9c-T` Tests bullets, re-pointing them at the resolved `model:` `Model<Api>.api` value. The split carried the corrected Tests bullet into `V9n`/`V9n-T` but reintroduced the old wrong attribution in the newly-written Adds prose — the c04b0a6 fix was not propagated into the Adds sentence.

## Solution approach

In `docs/plan_topics/V9n-transport-error-mapping.md`, rewrite the **Adds** clause so the prompt-mode `provider` value is the loom's resolved `model:` `Model<Api>.api` value (resolution owned by `V6a`), with `V9j` named only as the owner of the shared api-shaped value space the field's value space is keyed on — not a runtime supplier on this path. Align the prose with `V9n`'s existing first **Tests** bullet and `V9n-T`.

## Solution constraints

- Out of scope: the `V9n` Deps edge — `V9j` stays in `V9n`'s Deps as owner of the shared api-shaped value space and the Provider-error-mapping table; do not re-open it.
- The spec is read-only for this fix (`conversation-drive.md`, `queryerror-variants.md` already state the correct derivation).
