## Finding

`V9n`'s **Adds** field states that the synthesised `transport` error's `provider` field "is sourced from `V9j`'s provider-error-mapping surface, not re-derived here." On the prompt-mode transport path that `V9n` actually owns — the post-`waitForIdle()` `stopReason:"error"` probe and the synchronous-`pi.sendUserMessage`-throw secondary mapping — no pi-ai `complete()` call is issued, so `V9j`'s provider-error classifier is never entered there. `conversation-drive.md` (`#prompt-mode-error-detection`) detects the failure off the driven turn's trailing `assistant` message; the binder's `complete()` classifier (`V9j`) is on an unrelated path.

The spec defines the prompt-mode `provider` value as the loom's resolved `model:` `Model<Api>.api` value: `queryerror-variants.md` §`TransportError` provider-derivation says the field "is the resolved `Model<Api>.api` value of the model the failing query routed through — the same `api`-shaped field the Provider error mapping table is keyed on," and `conversation-drive.md` says the field "is populated from the loom's resolved `model:` regardless of which surface … detected the failure." `V9j`'s per-provider table is merely *keyed on* that same api-shaped value space; it is not a runtime supplier of the value on the prompt-mode path.

`V9n`'s own first **Tests** bullet (and `V9n-T`'s) already state the correct source: the `provider` value "sourced from the loom's resolved `model:` `Model<Api>.api` value (the frontmatter `model:`, or the inherited `ctx.model` when `model:` is absent; resolution owned by V6a)." The Adds clause therefore contradicts the leaf's own asserted behaviour, and an implementer who follows the Adds attribution literally finds the prompt-mode `provider` field has no wired runtime source — `V9j`'s classifier never runs on this path to produce it.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 0c3a2a2 — pi-loom plan: resolve "V9c bundles three concern clusters in one leaf pair" (2026-06-17, Thomas Andersen)
**History:** The misattribution entered with commit 0c3a2a2, which split `V9n` out of `V9c` and authored `V9n`'s new **Adds** clause ("`provider` … is sourced from `V9j`'s provider-error-mapping surface"). One commit earlier the same day, c04b0a6 ("resolve 'prompt-mode TransportError.provider cites V9j's classifier'") had fixed exactly this misattribution in the `V9c`/`V9c-T` Tests bullets, re-pointing them at the resolved `model:` `Model<Api>.api` value. The split carried the corrected Tests bullet into `V9n`/`V9n-T` but reintroduced the old wrong attribution in the newly-written Adds prose — the c04b0a6 fix was not propagated into the Adds sentence.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V9n-transport-error-mapping.md`, replace the Adds clause "the synthesised `transport` error's `provider` field is sourced from `V9j`'s provider-error-mapping surface, not re-derived here" with a statement that the prompt-mode `provider` value is the loom's resolved `model:` `Model<Api>.api` value (resolution owned by `V6a`), and that `V9j` only defines the api-shaped value space the field shares — not a runtime supplier of the value on this path. This aligns the Adds prose with `V9n`'s existing first Tests bullet and `V9n-T`.

Scope constraints: fix the attribution only. Do **not** re-open the Deps edge — `V9j` stays in `V9n`'s Deps as the owner of the shared api-shaped value space and the Provider-error-mapping table the field's value space is keyed on. The spec is read-only for this fix; `conversation-drive.md` and `queryerror-variants.md` already state the correct derivation.

## Consequence

**Severity:** correctness

The Adds clause contradicts `V9n`'s own Tests bullet: an implementer wiring from Adds would look to `V9j`'s classifier to supply `provider` on the prompt-mode path, but `V9j` is never invoked there, so the field has no source — while the leaf's test expects the resolved-`model:` value. Two reasonable implementers diverge, and a `kind:"transport"` error synthesised with a missing/wrong `provider` value also breaks the runtime-event-channel always-log emission for transport errors, which requires the field populated.
