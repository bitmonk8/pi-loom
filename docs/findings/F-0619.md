## Finding

The Binder system prompt is a single construct, but its normative content is split across two sibling pages with no single owning section. `binder-bypass-and-envelope.md` carries the *Binder system prompt* H3 with the illustrative fenced rendering, the *System-prompt structure (normative)* sub-section enumerating the eight MUST items, the *Type display* table, the *Default-literal rendering* rule, and the *Parameter-line reference renderings* table. The body of item 6 (Session-context block) — the *Compact-transcript format (normative)* sub-section and the BNDR-7/8/9 reference renderings (BNDR-7a–BNDR-7i) — lives in `binder-model-and-context.md`, under the §Session-context truncation parent.

The two halves cross-reference each other six times (item 6 forward-links the compact-transcript renderer; the renderer back-links item 6's opening-line and terminating-blank-line contract, and BNDR-7i back-links item 6 for the void-truncation case). An implementer assembling the rendered system prompt must hold both files open simultaneously to satisfy one obligation; a coverage-matrix author has already split the system-prompt obligations across two leaves (`cka-45` cites `binder-bypass-and-envelope.md` for `V11d`; the compact-transcript renderings are cited at `cka-39`/the BNDR-7 row of `V11b`) even though the system prompt is what they jointly produce.

The placement is also taxonomically wrong: the system prompt is neither a bypass case (the bypass paths skip the binder entirely and never construct a prompt) nor a property of the envelope schema (the envelope is the binder's *return* type). It sits in `binder-bypass-and-envelope.md` only because the file-split commit (`f5e89f4`) packed it there when the monolithic `binder.md` was carved into four size-capped children.

## Issue introduction

**Verdict:** specific-commit

**Introducing commit:** `f5e89f4` — "docs: enforce 100KB hard / 30KB recommended size cap on spec set" (2026-06-04)

**History:** The pre-split monolith `docs/spec_topics/binder.md` (last seen at `f5e89f4^`) held the entire binder construct as one file, with §Binder bypass, §Binder envelope, §Binder system prompt, and §System-prompt structure (normative) all sitting alongside §Binder context and §Session-context truncation / §Compact-transcript format. The split commit's plan (recorded in `docs/_tools/split_spec.py` at that commit) carved `binder.md` at four line ranges by file size, not by construct boundary: `binder-model-and-context` (lines 7–87) and `binder-bypass-and-envelope` (lines 88–236). The §Binder system prompt H3 lived at original line 164 — inside the 88–236 bypass/envelope window — while §Compact-transcript format (normative) lived at line 39, inside the 7–87 model/context window. The split therefore mechanically separated the two halves of the system-prompt construct in the same operation that fixed the size cap; the six surviving cross-links between the halves are the rewritten artefacts of references that were intra-file before `f5e89f4` and became cross-file after it. No subsequent commit has re-evaluated the construct boundary; downstream resolutions in this area (e.g. `1a41db2`, `ec86d31`, `2fb5783`, `361a6ea`) have only added pins on top of the inherited split.

## Solution Space

**Shape:** multiple

### Option A — Extract a dedicated `binder-system-prompt.md` child

**Approach.** Add a fifth sibling under `docs/spec_topics/binder/` carrying the entire system-prompt construct: §Binder system prompt (illustrative rendering + lead prose), §System-prompt structure (normative) with the eight items, Type display, Default-literal rendering, Parameter-line reference renderings, §Compact-transcript format (normative), BNDR-7/8/9 and BNDR-7a–BNDR-7i. Update `binder.md`'s Contents to list the new child. Anchors `system-prompt-structure-normative`, `compact-transcript-format-normative`, `bndr-7` (and 7a…7i), `bndr-8`, `bndr-9` travel verbatim to the new file (their `<a id>` form is location-independent). The six inter-half cross-links collapse to intra-file links. Internal in-page anchors `binder-envelope` (referenced by *Recent session context* discussion only via prose, not links) and `binder-model` (referenced only from `binder-model-and-context.md` itself) stay where they are.

**Spec edits.**
- Create `docs/spec_topics/binder/binder-system-prompt.md` with the moved sections.
- Remove §Binder system prompt and below from `binder-bypass-and-envelope.md`.
- Remove §Session-context truncation's *Compact-transcript format (normative)* sub-section (the H4 and everything below it through BNDR-9) from `binder-model-and-context.md`. The walk-defining prose under §Session-context truncation stays — it is a binder-context concern, not a renderer concern — with a forward-link to the new file's renderer section.
- Repoint the four cross-file links flagged above: in `pi-integration-contract/binder-inference.md` (`#system-prompt-structure-normative`) and in the four `V11b*`/`V11d*` plan leaves (and `coverage-matrix.md` `cka-39`/`cka-45`), change `binder-bypass-and-envelope.md#system-prompt-structure-normative` and `binder-model-and-context.md#compact-transcript-format-normative` (etc.) to `binder-system-prompt.md#…`.
- Add an entry to `binder.md`'s Contents.

**Pros.** Single authoritative location matching the construct's name. Eliminates all six inter-half cross-references. Keeps `binder-model-and-context.md` under the 30 KB recommended cap (it shrinks ~17 KB) and produces a new file well under the cap (~17 KB by the same accounting). Symmetric with the existing four-file binder layout (each construct gets its own file).

**Cons.** Adds a new file (now five binder children instead of four). Requires repointing ~6 cross-references across the corpus.

**Risks.** Anchor coverage: the spec uses `<a id>` rather than slug-derived anchors, so anchors survive the move; the repointing surface is enumerable and small.

### Option B — Consolidate into `binder-model-and-context.md`

**Approach.** Move §Binder system prompt and the four following sub-sections out of `binder-bypass-and-envelope.md` and into `binder-model-and-context.md`, appended below §Session-context truncation so the renderer flows directly into the system-prompt structure that consumes it. Rename `binder-model-and-context.md` to reflect the new scope (e.g. to `binder-model-context-and-system-prompt.md`) or accept the misnomer.

**Spec edits.**
- Append the §Binder system prompt construct after the existing compact-transcript section in `binder-model-and-context.md`.
- Remove the corresponding sections from `binder-bypass-and-envelope.md`.
- Either rename the destination file (and repoint every inbound link to it) or leave the name unchanged.
- Cross-references in `pi-integration-contract/binder-inference.md` and the plan leaves repoint to the new file.

**Pros.** No new file added (stays at four binder children). Compact transcript and its consuming system-prompt item 6 sit physically adjacent.

**Cons.** `binder-model-and-context.md` grows past 30 KB (it is already at the recommended cap; this would push it to ~47 KB and very close to the 100 KB hard cap), partially undoing what `f5e89f4` did. The destination file's name no longer describes its contents (system-prompt structure is neither "model" nor "context"). Same number of cross-file links survive in net (just rerouted), since the system prompt is consumed by sites in `pi-integration-contract/binder-inference.md`.

**Risks.** Size-cap regression — the `docs/_tools/check_links.py`/`fingerprint.py` tooling and the cap that triggered `f5e89f4` would need re-evaluation, or the file would need a follow-up split.

### Recommendation

**Option A** — extract to `binder-system-prompt.md`. The construct is named "Binder system prompt" everywhere it is referenced, every other binder construct (model, context, bypass, envelope, defaulting, determinism, cancellation, failure) is paired with a file whose name carries its concept, and the option respects the size cap that drove the original split. The naming follows the spec's existing pattern of giving each binder construct its own filename slot, so the rename is mechanical rather than judgement-bound.

Edge cases the implementer must watch:

- Preserve the verbatim `<a id="system-prompt-structure-normative">`, `<a id="compact-transcript-format-normative">`, `<a id="bndr-7">`/`bndr-7a`–`bndr-7i`, `<a id="bndr-8">`, `<a id="bndr-9">` anchors. The spec uses explicit `<a id>` tags, so anchors are location-independent; do not rename them.
- The link to *System-prompt structure (normative)* from `pi-integration-contract/binder-inference.md` appears twice (lines 12 and 13 of that file as of writing); both must be repointed.
- `binder.md`'s Contents bullet list must add the new file in the order it appears under the new layout.
- The four `V11b*`/`V11d*` plan leaves and `coverage-matrix.md` rows `cka-39` and `cka-45` carry the file paths the closing leaf cites for spec coverage; update them in the same change so coverage-matrix `npm test` integrity does not red-flag.
- The §Session-context truncation walk prose (the FakeTokenEstimator paragraph, the worked-example block) stays in `binder-model-and-context.md`. Only the compact-transcript H4 and below moves.

## Consequence

**Severity:** advisory

An implementer or test author writing the system-prompt builder must navigate two files to assemble one construct, and the conceptual location ("system prompt") matches the title of neither owning file. Behaviour is fully specified by the existing prose plus the six cross-links, so this is a navigability/maintainability cost rather than a correctness gap; the working coverage matrix already has both leaves citing both files.
