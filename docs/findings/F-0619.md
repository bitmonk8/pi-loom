## Problem

The Binder system prompt is a single construct, but its normative content is split across two sibling pages with no single owning section. `binder-bypass-and-envelope.md` carries the §Binder system prompt H3, the §System-prompt structure (normative) sub-section, the Type display table, the Default-literal rendering rule, and the Parameter-line reference renderings table; item 6's body — §Compact-transcript format (normative) and BNDR-7/8/9 (BNDR-7a–BNDR-7i) — lives in `binder-model-and-context.md`. The two halves cross-reference each other six times, so an implementer assembling the rendered prompt must hold both files open to satisfy one obligation. The placement is also taxonomically wrong: the system prompt is neither a bypass case nor a property of the envelope schema; it sits in `binder-bypass-and-envelope.md` only because the size-cap split commit packed it there.

## Issue introduction

**Verdict:** specific-commit

**Introducing commit:** `f5e89f4` — "docs: enforce 100KB hard / 30KB recommended size cap on spec set" (2026-06-04)

**History:** The pre-split monolith `docs/spec_topics/binder.md` (last seen at `f5e89f4^`) held the entire binder construct as one file, with §Binder bypass, §Binder envelope, §Binder system prompt, and §System-prompt structure (normative) all sitting alongside §Binder context and §Session-context truncation / §Compact-transcript format. The split commit's plan (recorded in `docs/_tools/split_spec.py` at that commit) carved `binder.md` at four line ranges by file size, not by construct boundary: `binder-model-and-context` (lines 7–87) and `binder-bypass-and-envelope` (lines 88–236). The §Binder system prompt H3 lived at original line 164 — inside the 88–236 bypass/envelope window — while §Compact-transcript format (normative) lived at line 39, inside the 7–87 model/context window. The split therefore mechanically separated the two halves of the system-prompt construct in the same operation that fixed the size cap; the six surviving cross-links between the halves are the rewritten artefacts of references that were intra-file before `f5e89f4` and became cross-file after it. No subsequent commit has re-evaluated the construct boundary; downstream resolutions in this area (e.g. `1a41db2`, `ec86d31`, `2fb5783`, `361a6ea`) have only added pins on top of the inherited split.

## Solution approach

Give the Binder system prompt its own file: create `docs/spec_topics/binder/binder-system-prompt.md` and move the entire construct into it — the §Binder system prompt H3 and everything below it in `binder-bypass-and-envelope.md`, plus the §Compact-transcript format (normative) H4 through BNDR-9 from `binder-model-and-context.md`. Add a forward-link from `binder-model-and-context.md`'s retained §Session-context truncation prose to the new file's renderer section, and add a `binder-system-prompt.md` entry to `binder.md`'s Contents list. Repoint the inbound cross-file references — `binder-inference.md`, the `V11b*`/`V11d*` plan leaves, and `coverage-matrix.md` rows `cka-39`/`cka-45` — from the old `binder-bypass-and-envelope.md#…` / `binder-model-and-context.md#compact-transcript-format-normative` anchors to `binder-system-prompt.md#…`.

## Solution constraints

- Preserve the existing `<a id="…">` anchor tags (`system-prompt-structure-normative`, `compact-transcript-format-normative`, `bndr-7`/`bndr-7a`–`bndr-7i`, `bndr-8`, `bndr-9`) verbatim when moving the sections; do not rename them.
- Out of scope for the move: the §Session-context truncation walk-defining prose (FakeTokenEstimator paragraph and worked-example block) stays in `binder-model-and-context.md` — only the §Compact-transcript format (normative) H4 and below moves.
