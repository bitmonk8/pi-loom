## Finding

`host-prerequisites.md` justifies the `"typebox": "*"` peer-dep choice by asserting that "`Type.Unsafe`'s call shape is stable across the TypeBox `0.x` → `1.x` line and therefore does not need install-time minor-skew detection" — a forward-looking, cross-major behavioural claim that no probe, fixture, or editorial-review gate anywhere in the corpus verifies. The same rationale is re-stated under *Deliberate deviation from Pi's bundled-package convention* on the same page to explain why typebox is excluded from the four-entry tilde-pinned lock-step.

The runtime backstop the page cites is the Step-0(e) host-shape probe in `capability-probe.md`, which checks only `typeof Type.Unsafe === "function"` — it does not pin arity, generic-signature shape, the lowered-JSON-Schema → TypeBox bridge contract, or any other aspect of `Type.Unsafe<unknown>(loweredJsonSchema)` call validity. A TypeBox 1.x release that renamed, retyped, or replaced the signature while leaving `Type.Unsafe` callable would pass the probe and surface as a runtime error at the first `pi.registerTool` / synthesised typed-query / binder-envelope construction site — exactly the failure mode `host-prerequisites.md` says the host-shape probe is there to prevent ("rather than to a runtime-time `TypeError` at the first tool-registration call"). The "Silent shape drift" carve-out at `capability-probe.md#post-probe-sdk-shape-drift` admits that such drift is undefined behaviour the runtime is not required to detect, but that carve-out covers post-probe drift on members the probe has already gated; it is not licence for the up-front rationale to overstate what the probe witnesses.

`version-bump-step2.md` enumerates editorial-review items (a)–(an) for every other unpinned host presupposition `host-prerequisites.md` and its sibling topic pages depend on, including ones (like `AgentSession.dispose()` mid-`abort()` safety, item (y)) with comparable behavioural-stability shape. The typebox cross-major stability claim has no parallel checklist item, so a TypeBox major-version bump landing on a host with `"typebox": "*"` resolves can ship undetected. The narrow defect: the rationale's scope outruns the verification it claims.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `f072903` (2026-05-08, *pi-loom spec: resolve "T09 typebox host-shape probe + diagnostic"*) — introduced Step 0 (e) and the `loom/load/host-incompatible` `typebox-shape` kind, narrowing the existing typebox sub-paragraph to delegate the host-shape check to the new probe. Prior to this commit the typebox paragraph asserted "The runtime depends only on `Type.Unsafe`, which is stable across the TypeBox 0.x → 1.x line; no version pin is warranted" as a standalone justification for `"*"`. The commit removed that exact sentence from the typebox sub-paragraph itself.
- `d8da74b` (2026-05-08, *pi-loom spec: resolve "peerDependencies ranges silently deviate from Pi documented * convention"*) — landed the same day; introduced the *Deliberate deviation from Pi's bundled-package convention* paragraph, which re-stated the cross-major stability claim verbatim ("whose call shape is stable across the TypeBox `0.x` → `1.x` line and therefore does not need install-time minor-skew detection") as the rationale for why typebox follows the convention while the four `@earendil-works/*` peers do not.
**History:** The narrow defect (rationale outruns verification) is the interaction between the two commits, not either in isolation. `f072903` deleted the standalone stability claim from the typebox sub-paragraph because the new probe (Step 0 (e)) was meant to replace it as the load-bearing justification; on the same day `d8da74b` re-introduced the same claim under a different paragraph addressing a different question (deviation rationale vs. probe scope). Subsequent edits — `5e73072` (2026-05-30, scope rename `@mariozechner` → `@earendil-works`), `f985a17` (Pi pin bump to `~0.74.1`), and the `81bf52a` collapse of the section pages into `spec_topics/` — preserved both paragraphs unchanged. The `version-bump-step2.md` editorial-review checklist accreted items (b)–(an) over later commits but no item was ever added for the typebox cross-major claim, so the gap has stood since `d8da74b` re-introduced the unbacked rationale.

## Solution Space

**Shape:** single

### Recommendation

Narrow the *Deliberate deviation from Pi's bundled-package convention* paragraph's typebox clause to observed-at-the-pin language, and add a paired editorial-review item to `version-bump-step2.md` mirroring the (b)–(an) shape used for every other unpinned host presupposition.

Concretely, in `host-prerequisites.md` under *Deliberate deviation from Pi's bundled-package convention*, replace "whose call shape is stable across the TypeBox `0.x` → `1.x` line and therefore does not need install-time minor-skew detection" with a clause that:
- names the bounded property the runtime actually consumes (the `Type.Unsafe<unknown>(loweredJsonSchema)` wrap surface used by `extension-bootstrap-and-per-loom.md`'s `parameters` derivation and `binder-inference.md`'s binder-envelope construction);
- pins the property as observed at the typebox version Pi bundles at the loom 1.0 Pi-SDK pin (not as a cross-major claim);
- forward-references the new editorial-review item below as the gate that re-confirms it on each Pi minor bump.

In `version-bump-step2.md`, append an item (following the (b)–(an) framing) — e.g. `(ao) *Bundled-typebox `Type.Unsafe` call-shape stability* — host-prerequisites.md *Deliberate deviation from Pi's bundled-package convention* typebox clause` — that obligates the contributor to confirm, against the typebox version the candidate Pi minor bundles, that `Type.Unsafe<unknown>(jsonSchema)` still produces a `TSchema` consumable by `pi.registerTool`'s `parameters` field on the same footing as the loom 1.0 pin. The recommended evidence shape mirrors item (n)'s editorial-read pattern (read the bundled typebox's `Type.Unsafe` declaration at its source-of-truth path; record pass / fail / N/A with a one-line rationale in the bump commit message). The item is SHOULD-level rather than MUST-level, consistent with the bulk of the (b)–(an) presupposition checklist.

The `typebox: "*"` peer-dep declaration itself, the `H1a` literal-read assertion, the `V9a` Step-0(e) probe, the `typebox-shape` `details.kind` enumeration, and the typebox carve-out under *Recognised import/access shapes* and *Target surface categories* all stay unchanged — the fix is editorial, scoped to the two paragraphs in `host-prerequisites.md` that overstate the rationale and the version-bump checklist that omits the parallel review item.

Edge case the editor must watch: do not let the narrowed rationale weaken the existing typebox-vs-four-entry distinction the *typebox MUST NOT be folded into the four-entry tilde-pinned group* MUST relies on. The deviation rationale (Pi's convention prescribes `"*"`, the host bundles typebox, the install-time-skew argument that justifies pinning the four `@earendil-works/*` does not apply because typebox arrives through a single bundled instance) is unaffected by the cross-major-claim narrowing and stays in place.

## Consequence

**Severity:** advisory

A TypeBox major-version bump that ships under the host's bundled `typebox` and reshapes `Type.Unsafe` enough to break the lowered-JSON-Schema bridge (but leaves it callable) surfaces as a runtime error at the first tool-registration / typed-query / binder construction site, not at probe time. Pi minor bumps that re-resolve the bundled typebox version are caught by no loom-side gate — the contributor checklist names every other unpinned host-behavioural presupposition and silently omits this one. The blast radius is bounded by Pi's own bundling discipline (typebox is shipped by Pi), but the rationale outruns the probe and the omission is a real future-bump trap.
