## Problem

The `ExtensionContext (member surface loom touches)` block's opener asserts the runtime "touches exactly" the members enumerated below, but the signature block mixes two kinds without distinguishing them: members the runtime actively calls, and three members (`compact`, `shutdown`, `getSystemPrompt`) flagged "not invoked / not consumed by loom in loom 1.0" yet kept in the block for re-validation. "Touches" is therefore overloaded — it means both "actively called at a defined callsite" and "re-validated on each Pi minor bump even though never called". The version-bump re-validation checklist (`#pi-version-bump-procedure`) reads from this set, so an implementer building the table from the opener omits the three uncalled rows while one building it from the row annotations includes them; both are textually defensible.

Did not introduce the underlying contradiction; it amplified it.

**History:** `git log -S "touches exactly the members" -- docs/` and `git log -S "completion / failure observed via"` over the spec corpus, plus `git show 0e080df -- spec_topics/pi-integration-contract.md` (the pre-split path; the file was later moved into `docs/spec_topics/pi-integration-contract/host-interfaces-core.md` by the corpus-split commits `f5e89f4` and `81bf52a` with no semantic change to this block).

## Solution approach

Rewrite the opener of the `#extensioncontext-interface` block so it distinguishes the two member sets the signature block already contains — members the runtime actively calls, and re-validation-only members the runtime never invokes in loom 1.0 but the Pi-version-bump procedure (`./version-bump-intro.md#pi-version-bump-procedure`) still re-validates. Clarify each signature-block row to the set it belongs to, where `compact`, `shutdown`, and `getSystemPrompt` are the re-validation-only rows and the remainder are actively-called. Keep the existing statement that members outside both sets are non-load-bearing and may be revised by Pi without spec changes.

## Solution constraints

- The touched-surface inventory the Pi-version-bump procedure re-validates MUST remain the union of both member sets, not the actively-called set alone.
