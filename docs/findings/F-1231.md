## Finding

The version-bump procedure rests on two un-stated infrastructure preconditions:

1. **Registry-fetchability of the pinned SDK.** The *Baseline acquisition* paragraph (`version-bump-step2.md`, anchor `bump-baseline-acquisition`) mandates `npm pack @earendil-works/pi-coding-agent@<pinned-version>` as the sole materialisation path for the loom 1.0 Pi-SDK pin's `dist/*.js` tree — the canonical "from" side every `dist/*.js` diff in items (e), (r), and (aj) reads against. The paragraph names no fallback for the case where `<pinned-version>` is unpublished, yanked, or otherwise un-resolvable from the registry, even though it explicitly tells the contributor to use a literal `@<pinned-version>` specifier rather than registry resolution from `package.json` (because step 1 has already overwritten the local copy). With the pinned tree neither retained in-repo (the same paragraph forbids committing the tarball) nor obtainable from the local install (step 1 has clobbered it), a yanked pin leaves every `dist/*.js`-diff and JS-body-read item (e, r, aj) un-runnable, blocking the bump on an external-availability event the procedure does not acknowledge.

2. **`npm run typecheck` / `npm test` contract.** The spec pages name `npm run typecheck` and `npm test` as the harnesses that run the brand-string type-equality assertion (`version-bump-intro.md` step 1; `version-bump-triggers.md` step 5), the SDK surface-inventory literal-read (`version-bump-step2.md` step 2(a)), the literal-array consistency check, the inbound-reference sweep audit, the `engines.node` three-way equality, and the provider-error / forced-tool fixture corpora — but nowhere enumerates which script runs which assertion family, nor pins the existence of the scripts themselves. The scripts are populated in the plan (`H1a` defines `npm test`, `npm run build`, `npm run typecheck`; `V18c`'s *Ships when* names the per-gate green requirements), so the spec implicitly assumes the plan-side wiring without citing it; an implementer working from the spec alone cannot determine which `package.json#scripts` entries are load-bearing for which obligations.

## Issue introduction

**Verdict:** multi-commit-interaction

**Introducing commits:**
- `ccc6e38` ("pi-loom spec: resolve \"Item (e)/(r) dist/*.js baseline source unspecified\"") introduced the `Baseline acquisition` paragraph and the `npm pack @earendil-works/pi-coding-agent@<pinned-version>` recipe that obligation 1 keys on. The fetchability failure-mode (yanked / un-resolvable pin) was not considered in the resolving edit.
- Subsequent commits — `d4d72fb`, `5d73cc3`, `c5f97f6`, `9ced9f1` — extended the consumer list of the baseline (adding items (aj) and re-wording the prose) without revisiting the precondition. The 2026-06-07 introduction therefore went unchallenged through ~five later edits to the surrounding paragraph.

**History:** The `npm run typecheck` / `npm test` script names (obligation 2) predate `ccc6e38`; they were carried into the version-bump pages at the corpus split (`81bf52a docs: collapse spec/ section pages into spec_topics/`) and were already present in the pre-split monolith. The plan-side definition in `H1a` (`H1a-scaffold-and-toolchain.md`) was authored independently and the spec/plan cross-link was never threaded — neither commit chain (the baseline-acquisition chain or the original script-name chain) attempted to enumerate the script→assertion-family mapping in the spec. The defect is the *interaction* of these two histories: the version-bump pages accreted assertion families and harness-script references in parallel, with neither chain owning their alignment.

## Solution Space

**Shape:** multiple

This finding bundles two independent obligations (a registry-availability precondition + fallback, and a spec/plan script-contract reconciliation). Per the bimodal rule, each obligation is shaped as its own option so the orchestrator can sequence them as separately-loopable fixes.

### Option A — State the fetchability precondition and a fallback in the `Baseline acquisition` paragraph

**Approach.** Extend the `bump-baseline-acquisition` paragraph in `version-bump-step2.md` with a two-sentence precondition + fallback clause. Precondition: the pinned `@<pinned-version>` MUST be resolvable from the contributor's configured npm registry at bump time; the contributor MUST confirm `npm view @earendil-works/pi-coding-agent@<pinned-version> version` resolves before running `npm pack`. Fallback: when the pin is unpublished or yanked, the contributor MUST acquire the same `dist/` tree from one of (i) the corresponding `@earendil-works/pi-coding-agent` git tag (e.g. `git archive` of the upstream repo at the tag matching `<pinned-version>`), or (ii) a previously-cached tarball at the pin. The fallback path MUST produce an extracted `package/dist/` tree byte-identical to what `npm pack` at that version would have produced (or surface the divergence on the bump commit); the contributor MUST record the fallback path used in the bump commit message so the audit trail captures the substitution.

**Spec edits.** Append a precondition + fallback sub-paragraph to the existing `Baseline acquisition` block in `version-bump-step2.md`; no other page changes.

**Pros.** Closes the gap at its source. Localised edit (one paragraph). Preserves `npm pack` as the recommended path. The byte-identity criterion gives a verifiable acceptance.

**Cons.** Adds a new "MUST acquire byte-identical tree" obligation the contributor cannot mechanically verify without a checksum the spec does not currently pin.

**Risks.** The byte-identity criterion may be impossible to satisfy under realistic registry-yank conditions (a yanked version's `npm pack` output is typically gone with the version); softening to "best-effort equivalent tree, with divergences recorded" may be needed.

### Option B — Add a spec→plan cross-reference for the harness-script contract; do not duplicate the wiring spec-side

**Approach.** Add a short *Harness scripts* preamble paragraph to `version-bump-intro.md` stating that the `npm run typecheck` and `npm test` invocations named throughout the version-bump pages resolve to `package.json#scripts` entries whose existence, command, and per-assertion-family wiring are plan-owned (cross-link `H1a-scaffold-and-toolchain.md` for the script definitions and `V18c-version-bump-checklist.md` for the per-gate green-on-`main` requirements). Do not enumerate the per-assertion-family mapping in the spec; the mapping is implementation- and plan-owned by design (the spec's *implementation owns the literal strings* convention is consistent with this).

**Spec edits.** A single preamble paragraph in `version-bump-intro.md` (the page that opens the procedure), plus optionally a one-line forward-link at the head of `version-bump-step2.md` and `version-bump-triggers.md` to the same paragraph.

**Pros.** Honours the existing spec/plan separation (spec defines obligations, plan/implementation owns harness wiring). Minimal edit. Makes the spec self-contained-modulo-cross-references on the script question.

**Cons.** Acknowledges rather than closes obligation 2; an implementer building from the spec alone still must follow a cross-link.

**Risks.** None material — this is the convention already in use elsewhere (e.g. `audit-wire-and-canary.md` says `npm test` is the entry harness PIC pins, with the concrete command and wiring implementation-owned).

### Recommendation

Resolve Option B first, then Option A. Option B is the smaller, scope-bounding edit (one preamble paragraph; reconciles a long-standing spec/plan ownership question) and lands on a stable footing without touching the baseline-acquisition surface. Option A then lands on top, addressing the genuine procedural gap in the `Baseline acquisition` paragraph without entangling the script-contract question.

Implementer edge cases:
- For Option A: the byte-identity criterion is aspirational; phrase the fallback so divergences caused by registry-yank-and-rebuild are surface-able on the bump commit rather than treated as a blocker.
- For Option A: confirm that no other paragraph on this page silently relies on a registry-resolvable pin (the sweep over `npm view` / `npm install` / `npm pack` recipes is one-pass) so the fallback prose covers every site.
- For Option B: the preamble must not duplicate the per-gate green requirements `V18c`'s *Ships when* already pins; the cross-link is the contract.

## Consequence

**Severity:** advisory

Obligation 1: a yanked pinned version mid-bump strands the contributor — the procedure offers no documented escape (alternate registry, vendored tarball, prior-checkout reuse, git-archive of a Pi tag), so the contributor is forced to improvise off-spec under time pressure, with no convention recording what they did. Obligation 2: an implementer building the version-bump gates from the spec alone cannot determine the script contract without reading the plan; the gap rarely bites at run time because the plan already pins the scripts, but it makes the spec non-self-contained for the version-bump surface.
