## Problem

The Step-0 opening paragraph of `capability-probe.md` (anchor `entry-capability-probe`) declares "a synchronous probe of **three host preconditions** in the **fixed order (a) → (b) → (c)**" and claims canonical-source-of-truth status for every probe rule. The same page, however, enumerates five lettered sub-steps `(a)`–`(e)` and states the reconciled short-circuit order as `(a) → (b) → (c)+(d) → (e)`, while PIC-5 reinforces "Probes MUST NOT be added beyond the **five** enumerated checks". The opening's `(a) → (b) → (c)` range therefore omits `(d)` peer-dep version and `(e)` `typebox` host-shape outright, in the paragraph that claims to override the others. A reader who anchors on it can conclude the gating probe ends at `(c)`, letting a mis-versioned peer or a host missing callable `Type.Unsafe` pass Step 0 and fail later as an uncaught runtime `TypeError` instead of refusing with `loom/load/host-incompatible`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `126ff8a` (2026-05-06, "pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B"); `f072903` (2026-05-08, "pi-loom spec: resolve \"T09 typebox host-shape probe + diagnostic\"")
**History:** `git log --follow` / `git show` over the Step-0 prose (tracked across the `spec.md` → `spec_topics/pi-integration-contract.md` → `spec_topics/pi-integration-contract/capability-probe.md` splits) shows the defect is the accreted residue of two commits, neither of which revised the opening sentence. At `126ff8a` the rewrite authored the opening as "three host preconditions in the **fixed order (a) → (b) → (c)**" while that same commit's body already carried sub-step `(d)` Peer-dep version, with the brevity paragraph reading "four sub-steps `(a)`–`(d)`" and the invariant reading "Probes MUST NOT be added beyond the **four** enumerated checks" — so the opening's `(a) → (b) → (c)` range was already one short of the body at authoring. `f072903` then added sub-step `(e)` `typebox` host-shape, bumping the brevity paragraph to "five sub-steps `(a)`–`(e)`", the invariant to "**five** enumerated checks", the `details.kind` enumeration to include `typebox-shape`, and the short-circuit order to `(a)→(b)→(c)+(d)→(e)` — but again left the opening sentence untouched, widening the gap from one to two omitted sub-steps. The opening was last consistent with the body before sub-step `(d)` entered the probe; every subsequent expansion updated the reconciliation/invariant prose while the opening enumeration was carried forward unrevised.

## Solution approach

Rewrite the Step-0 opening sentence at anchor `entry-capability-probe` so its sub-step enumeration matches the body's reconciled `(a) → (b) → (c)+(d) → (e)` order and PIC-5's "five enumerated checks". If the "three host preconditions" conceptual grouping is retained, decouple it from the sub-step lettering so the count is not equated with the `(a) → (b) → (c)` letter range.

## Solution constraints

- Out of scope: do not alter the `(d)`/`(e)` short-circuit semantics — only the opening's description of the ordered set.
- Out of scope: the `overview-and-orientation.md` "preconditions 1–3" orientation framing — leave it as-is; no forward-link relabel is required.
