## Problem

The `Baseline acquisition` block (anchor `#bump-baseline-acquisition`) makes `npm pack @earendil-works/pi-coding-agent@<pinned-version>` the sole channel for producing the canonical "from" side that audit items (e), `#bump-checklist-convert-to-llm-context-entry` (r), and `#bump-checklist-inmemory-transcript-discard` (aj) diff and read against, and it forbids retaining the extracted tree as an in-repo snapshot. This makes registry retrievability of the pinned version an implicit precondition for those three items, but the precondition is never stated and the procedure never says what outcome to record when acquisition fails — the routine `npm pack` failure modes (the version unpublished or yanked, the contributor offline or behind a mirror lacking that version, a corrupt download, registry auth failure). On any of these, items (e)/(r)/(aj) become unrunnable and two reasonable contributors will diverge for the same external state, recording `fail`, `not-performed`, `N/A`, or a silent skip and reaching different merge dispositions.

## Issue introduction

**Verdict:** present-since-inception (of the `Baseline acquisition` block)
**Introducing commits:** `ccc6e38` (2026-06-07, *pi-loom spec: resolve "Item (e)/(r) dist/\*.js baseline source unspecified"*) — introduced the `Baseline acquisition` block, the `npm pack`-based recipe, and the no-commit rule, all in one diff covering items (e) and (r). `9ced9f1` (2026-06-08, *pi-loom spec: resolve "subagent transcript-discard not routed to bump checklist"*) extended the same block's scope to item (aj) (adding the JS-body read of `SessionManager.inMemory` in `dist/core/session-manager.js`), inheriting the same gap unchanged.
**History:** `git log -L '/bump-baseline-acquisition/,+1:docs/spec_topics/pi-integration-contract/version-bump-step2.md'` shows two edits to the anchored line: the introducing commit `ccc6e38` and the (aj) extension `9ced9f1`. Neither commit's diff adds a precondition statement or a failure-disposition arm for unretrievable baselines; the prose has carried the same shape (recipe + no-commit rule, with no precondition / disposition) since `ccc6e38`. The gap is durable in the introducing commit rather than the product of later interaction.

## Solution approach

Clarify the `#bump-baseline-acquisition` block to state the registry-retrievability precondition the three items depend on, and add a failure-disposition arm for items (e)/(r)/(aj) when that precondition does not hold. Keep both inside the block so the items inherit them by reference rather than each repeating the disposition. Align the disposition's outcome with item (e)'s conservative-posture rule (record `fail`, not `N/A`, when the property cannot be positively demonstrated) and with the fail-closed posture at `audit-wire-and-canary.md#audit-infrastructure-failure-handling`.

## Solution constraints

- Preserve the block's existing ephemeral / `MUST NOT be committed` rule unchanged.
- Do not redefine the established outcome tokens: item (e)'s `N/A` stays reserved for the superseded-by-Pi-mechanism case defined at `#bump-step-2-outcome-recording-conventions`, and `not-performed` stays "the SHOULD-level audit was skipped".
