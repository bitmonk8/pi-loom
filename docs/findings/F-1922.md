## Finding

`V7b`'s `DIAG-2` and `DIAG-4` Tests bullets (mirrored verbatim in `V7b-T`) describe the registry↔test parity gate as if it witnesses semantic-fidelity properties that the underlying scan cannot deliver:

- `DIAG-2`: "a code asserted by a test with no registry row fails the gate (and, per the `H5a` gate's `Adds.`, a registry code with no asserting test also fails the gate)."
- `DIAG-4`: "the Message column is normative — every asserting test sources its expected string from the registry, not prose."

The gate `V7b` consumes is owned by `H5a`, and `H5a`'s own citing-test arm declares its limit inline: it is "a best-effort inline-`PREFIX-N`-citation scan over the test sources [that] certifies citing-test existence, not the assertion's semantic fidelity." A token-presence scan cannot witness that a test *asserts* on the code (versus mentioning the code token in a comment or an unrelated string), nor that a rendered-message test *sourced* its expected string from the registry (versus hardcoding an identical literal). Yet `V7b`'s bullets phrase both as if the gate mechanically establishes them.

The sibling closing-gate arms state this limit explicitly. `H5a` carries it on its per-REQ-ID citing-test bullet; `H5f` repeats it ("a best-effort inline-citation existence scan … not that the cited test's assertion is semantically faithful … that per-facet faithfulness stays a TDD and self-review obligation"). `V7b` alone omits the caveat, leaving its parity bullets reading as a stronger guarantee than the machinery it inherits from `H5a` provides.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `c6a664e` (2026-06-10), `f284534` (2026-06-10)
**History:** The plan corpus is git-tracked. `git log --follow` on `docs/plan_topics/V7b-code-registry.md` shows two commits. `c6a664e` ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10 08:16) introduced the `DIAG-4` bullet "every asserting test sources its expected string from the registry, not prose" verbatim — an over-claim of semantic fidelity present since plan inception. `git show f284534` ("pi-loom plan: resolve \"DIAG-2 src/** emission scan vs closing gate\"", 2026-06-10 11:47) rewrote `DIAG-2` from "a code emitted by `src/**` with no registry row fails the gate" to "a code asserted by a test … (and … a registry code with no asserting test also fails the gate)" and changed Ships-when to "reconciles test-asserted codes", introducing the "asserting test" parity framing without importing the best-effort-scan caveat the sibling `H5a`/`H5f` arms carry. The current defect is the joint residue of the inception-era `DIAG-4` over-claim and the later `f284534` `DIAG-2` rewrite, neither of which stated the scan's limit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V7b-code-registry.md` and the mirrored `docs/plan_topics/V7b-T-code-registry.md`, qualify the `DIAG-2` and `DIAG-4` Tests bullets so each states the scan's limit inline, the way `H5a`'s per-REQ-ID citing-test bullet and `H5f`'s "best-effort existence" bullet already do.

- On the `DIAG-2` bullet, after the parity clause "a registry code with no asserting test also fails the gate", append wording to the effect that the reconciliation is a best-effort inline code-token existence scan: it certifies that a test cites the code token, not that the test asserts on that code rather than merely mentioning it — assertion fidelity staying a TDD and self-review obligation. Phrasing such as `; this is a best-effort inline code-token scan that certifies the code token is cited by a test, not that the test asserts on it — assertion fidelity stays a TDD/self-review obligation` is suitable.
- On the `DIAG-4` bullet, append wording that the scan cannot witness that the expected string was sourced from the registry versus a hardcoded identical literal, so the "sources its expected string from the registry, not prose" requirement is a TDD/self-review obligation the gate does not mechanically prove. Phrasing such as `; the scan cannot distinguish a registry-sourced string from an identical hardcoded literal, so registry-sourcing stays a TDD/self-review obligation` is suitable.

The exact wording is content, not shape — match it to the surrounding bullet style. Apply the same edit to both leaves so the `V7b`/`V7b-T` pair stays identical. Do not edit `H5a` or `H5f` (they already carry the caveat) and do not touch the spec `DIAG-2`/`DIAG-4` rules.

## Consequence

**Severity:** advisory

An implementer reading `V7b` in isolation may build (or believe the gate enforces) a mechanical check that a test genuinely *asserts* on each code and that each message string was *sourced* from the registry — neither of which a presence scan can do — then either over-engineer the gate to chase fidelity it cannot reach or silently leave the TDD/self-review obligation undocumented. The gate still functions as the presence scan `H5a` owns, so no leaf is blocked; the cost is a guidance gap and a parity mismatch with the sibling arms that do state the limit.
