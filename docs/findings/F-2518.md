## Problem

When a `.loom` is exposed as a tool via another loom's `tools:`, the field-derivation list under the **Per-loom registration** heading in `extension-bootstrap-and-per-loom.md` asserts all five `ToolDefinition` fields are required by Pi's `ToolDefinition` interface and derives `description` from the loom's frontmatter `description`. But the field-contract table in `frontmatter-fields-a.md` marks frontmatter `description` as optional (`required: no`, default `null`, with the note "internal-only looms legitimately omit this"). A tool-exposed loom that omits `description` therefore derives a `null` (absent) value for a Pi-required field, and the spec defines no behaviour for that case ŌĆö no synthesised default, no load-time diagnostic refusing to expose a description-less loom as a tool, and no requiredness assertion scoped to tool-exposed looms.

All three obligations are present-since-inception of the field-derivation block.

## Solution approach

Clarify the `description` derivation in the **Per-loom registration** field-derivation list to pin what the runtime does when a tool-exposed loom's frontmatter `description` is absent, reconciling the "all five fields required" assertion against the optional-`description` row in `frontmatter-fields-a.md`'s field-contract table.

## Solution constraints

- Preserve the optional `description` contract for non-tool-exposed looms in `frontmatter-fields-a.md`'s field-contract table (`required: no`, default `null`, "internal-only looms legitimately omit this"); the fix scopes only to looms exposed via `tools:`.
