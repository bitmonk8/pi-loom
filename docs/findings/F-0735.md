## Finding

DISC-5 in `docs/spec_topics/discovery/package-and-settings.md` defines the default starting set for the `pi.looms` glob pipeline as: *"When no plain (un-prefixed) include pattern is present, the starting set is every path under the package root; otherwise the starting set is the union of the paths matched by the plain include patterns."* The phrase "every path under the package root" does not state whether enumeration descends into subdirectories or stops at the package root's immediate children.

The ambiguity is decision-relevant because the starting set is the input to the `!` / `+` / `-` override pipeline; the surviving members then feed the per-match contribution rule (`.loom` file → register; directory → non-recursive scan for `*.loom`; other → silently filtered). A flat enumeration delivers only top-level files and immediate-child directories to the pipeline (and so `!` cannot ever subtract a nested file because none was ever in the set); a recursive enumeration delivers every descendant file and directory and behaves very differently under both `!` and the file/directory contribution rule. Two implementers reading the same paragraph will register materially different sets of looms for any nested package layout.

The follow-up phrasing in the parked Suggested fix ("and how `**/*.loom` patterns then work") is a hint that the recursive reading is intended — a `**/*.loom` plain include only meaningfully selects nested files if the enumeration the matcher walks is itself recursive — but DISC-5 never states this, so the question is left open.

## Solution Space

**Shape:** single

### Recommendation

Amend DISC-5 to state that the default starting set is **every file and directory reachable by recursive descent from the package root** (subject to the per-match contribution rule that already filters non-`.loom` files silently). State this once, in the existing sentence, e.g.:

> When no plain (un-prefixed) include pattern is present, the starting set is every path under the package root **enumerated by recursive descent** (every file and directory at any depth); otherwise the starting set is the union of the paths matched by the plain include patterns, **matched against the same recursive enumeration**.

The second clause is the load-bearing half — it pins that the matcher walks a recursive enumeration, so a plain include of `**/*.loom` selects nested `.loom` files at any depth and a plain include of `*.loom` selects only top-level `.loom` files. Without that pin the matcher-input set is itself ambiguous.

No further change is required at the `loomPaths` site: its glob bullet already defers wholesale to DISC-5 via cross-link, so tightening DISC-5 carries through. The `looms/` fallback paragraph remains untouched — it is explicitly non-recursive and that asymmetry (manifest entry = glob-driven recursive; convention directory = flat) is intentional and matches how Pi treats its own `pi.<resource>` arrays vs convention directories.

Edge cases the implementer must watch:

- The recursive enumeration MUST honour the same `lstat` / symlink rule already pinned for root-level child classification (per PIC-13), so a symlinked subdirectory inside a package does not silently leak out-of-package paths into the starting set.
- The bounded-walk caps in DISC-6 (`scanPackagesMaxFiles`, `scanPackagesTimeoutMs`) are defined against `package.json` reads, not against the per-package recursive enumeration; a deeply-nested package with thousands of files does not currently trip those caps. If that becomes a concern it is a separate finding against DISC-6, not part of this fix.
- The `pi.looms`-vs-`looms/` asymmetry (recursive vs flat) should be noted briefly, otherwise a reader who normalised on "non-recursive everywhere" will be surprised. A single parenthetical at the DISC-5 sentence ("the `looms/` fallback below remains non-recursive — the asymmetry mirrors Pi's manifest-vs-convention split") is sufficient.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on which looms a nested-layout package contributes. A package shipping `pkg/sub/foo.loom` with no `pi.looms` plain includes (just `!`/`+`/`-` overrides, or none) registers `foo.loom` under the recursive reading and contributes nothing under the flat reading. The defect also silently propagates to `loomPaths`, whose entry schema delegates its glob contract back to DISC-5.
