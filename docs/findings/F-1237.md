## Finding

`H4a` defines an **in-memory fixture-supply mechanism**: the harness "supplies the fixture `.loom` content to the extension in memory so a harness-driven leaf's discovery reads from the harness-provided source rather than the real filesystem â€” no ambient `src/**` filesystem read, no `FileSystem` seam (`V8b`) dependency, keeping the `H3a` ambient-access scan clean." The route by which that in-memory content actually reaches the discovery code is never specified. `H3a`'s injected seam set (`Checkpoint`, `SchemaValidator`, `Clock`, `FileSystem`, `FileWatcher`, `TokenEstimator`, `IdSource`) contains no "discovery source" seam, and the production discovery walk (`V10a`) reads files through the `FileSystem` seam (`~/` expansion via `FileSystem.homedir()`, `readdir`/`readText`). So the mechanism asserts that fixture content reaches discovery while bypassing both ambient reads *and* the only seam through which discovery reads files â€” without naming the affordance that bridges the gap.

For `M` / `M-T` the tension is contained: both perform a bespoke single-source happy-path discovery that `M` explicitly says "diverges from the production filesystem-backed discovery walk deepened by `V10a`," so neither exercises the production read path. But `H7a` drives a representative multi-feature fixture `.loom` through the `H4a` harness against the integrated pipeline, and the route by which that fixture reaches whatever loading/discovery code the integrated run uses is left unstated. The "no `FileSystem` seam dependency" claim cannot be reconciled with a fixture that must be discovered/loaded unless the injection point is named.

Two reasonable implementers diverge: one supplies the fixture via an in-memory implementation of the existing `FileSystem` seam (which contradicts the leaf's own "no `FileSystem` seam dependency" claim), the other invents a harness-only discovery-source override distinct from the seam set. The leaf neither picks nor names either route.

## Solution Space

**Shape:** multiple

### Option A â€” Named harness discovery-source injection point (distinct from the seam set)

- **Approach:** Add to `H4a` an explicitly named injection affordance â€” a harness-supplied discovery source the factory/discovery accepts in harness mode â€” distinct from the `H3a` seam set, by which in-memory fixture `.loom` content reaches discovery without an ambient read or a `FileSystem` seam call.
- **Plan edits:** In `H4a` Adds, name the injection point and state that harness-driven discovery reads from it; keep the "no `FileSystem` seam dependency" claim (it stays true). In `H7a` Adds, state its integrated-run fixture reaches the pipeline via the same `H4a` injection point. `M` / `M-T` Adds keep their current wording, optionally citing the named injection point.
- **Spec edits:** None.
- **Pros:** Preserves the "no `FileSystem` seam, clean `H3a` ambient-access scan" invariant verbatim; one shared route for all harness-driven discovery consumers.
- **Cons:** Introduces a harness-only injection concept outside the seam set that production discovery code must accommodate (a discovery-source override threaded into discovery).
- **Risks:** If production discovery cannot accept the override without branching, the harness path stays divergent from the production walk (already true for `M`; would persist for `H7a`).

### Option B â€” In-memory `FileSystem`-seam implementation

- **Approach:** Supply the fixture via an in-memory implementation of the existing `FileSystem` seam injected at the runtime root, so discovery's normal `FileSystem` reads resolve against in-memory content.
- **Plan edits:** In `H4a` Adds, restate the mechanism as an in-memory `FileSystem`-seam fake and strike the "no `FileSystem` seam (`V8b`) dependency" claim (the ambient-access-scan note stays â€” a seam fake is still no ambient read). Update `M` / `M-T` Adds to drop "no `FileSystem` seam dependency" and add `V8b` to their Deps. `H7a`'s integrated run uses the same fake.
- **Spec edits:** None.
- **Pros:** No new injection concept; harness-driven discovery exercises the real production discovery walk (`V10a`) rather than a bespoke path, improving fidelity.
- **Cons:** Removes the current "no `FileSystem` seam dependency" claim; `M` / `M-T` gain a `V8b` Deps edge.
- **Risks:** `V8b` lands after `H4a`/`M` in dep order, so `M` / `M-T` / `H7a` would need `V8b` in Deps â€” a sequencing change.

### Recommendation

Option A. It keeps `H4a`'s stated invariants intact and matches the current text's intent (a harness affordance distinct from the seams), confining the edit to naming the injection point in `H4a` and pointing `H7a` at it. Watch that the named injection point is the single route all harness-driven discovery consumers (`M`, `M-T`, `H7a`) use, and that `H7a`'s integrated-run fixture is explicitly wired through it rather than left implicit.

## Consequence

**Severity:** correctness

The `H4a` implementer must invent the route by which harness fixture content reaches discovery; two implementations diverge, and one of them (an in-memory `FileSystem`-seam fake) directly contradicts the leaf's stated "no `FileSystem` seam dependency / clean `H3a` ambient-access scan" invariant that the architectural gate relies on. `H7a`'s integrated multi-feature run has no stated loading route at all, so its fixture would be wired ad hoc.
