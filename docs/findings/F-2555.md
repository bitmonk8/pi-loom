## Problem

The Loom literal sublanguage's **Field rules** in `grammar.md` state that "Every declared field of the LHS schema (or the named variant schema) MUST be present" (omissions ŌåÆ `loom/parse/missing-object-field`) and that "the is-literal check runs at parse time," applying these uniformly across both `BareObjectLit` positions. But the Pi-tool argument position draws its type from the Pi tool's registered input schema, which is only available after callee resolution against the load-time callable table ŌĆö not at parse time. This contradicts `tool-calls.md`, whose **Argument shape** section states a Pi-tool argument that does not match the tool's input schema "is never a parse error" and "is always caught by the runtime AJV check" surfacing as `Err(CodeToolError { cause: "validation", ... })`. The blanket parse-time rule also presupposes the loom-schema model (every declared field required); an externally-owned JSON Schema with its own `required` subset, `additionalProperties`, and possibly non-object root does not obey that model, so applied literally it would wrongly reject calls that omit optional Pi-tool fields.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f841b12` (2026-05-04, "spec: resolve Cluster 2 (literal sublanguage + grammar appendix)"); `4385d86` (2026-06-04, "pi-loom spec: resolve \"Pi-tool argument type-checking timing\"")
**History:** `git log -S` over the cited tokens traces the defect to the interaction of two commits, not a single inception. `f841b12` introduced the entire Loom literal sublanguage in `grammar.md`, including the `BareObjectLit` Pi-tool position rule, the blanket "Every declared field of the LHS schema ŌĆ” MUST be present ŌåÆ `loom/parse/missing-object-field`" field rule, and the "is-literal check runs at parse time ŌĆ” does not require a separate parser" assertion; at that point `grammar.md` and `tool-calls.md` were mutually consistent ŌĆö both routed Pi-tool arguments through the parse-time literal sublanguage. `4385d86` then edited `tool-calls.md` (and `diagnostics.md`) to make a Pi-tool argument that does not match the tool's input schema "never a parse error ŌĆ” always caught by the runtime AJV check," keeping only `tool-arg-not-literal`/`tool-arg-arity` as parse checks ŌĆö but did **not** touch `grammar.md` (confirmed: `git show 4385d86 --stat` lists no `grammar.md` change). That asymmetric edit severed the prior consistency, leaving `grammar.md`'s blanket parse-time field-presence rule applying to the Pi-tool `BareObjectLit` position that `tool-calls.md` had moved to runtime. The durable fix addresses the interaction (reconciling `grammar.md`'s field rule with the runtime-only Pi-tool arm) rather than reverting either commit.

## Solution approach

Narrow `grammar.md`'s **Field rules** so the parse-time field-presence check (`loom/parse/missing-object-field`) applies only to the loom-schema-backed positions ŌĆö `params:` defaults and `NamedObjectLit` ŌĆö and clarify that the Pi-tool argument position's only parse-time checks are the is-literal check (`loom/parse/tool-arg-not-literal`) and arity (`loom/parse/tool-arg-arity`). State that field presence, optionality, non-object roots, and `additionalProperties` on the Pi-tool arm are validated at runtime by AJV against the tool's own schema, surfacing as `Err(CodeToolError { cause: "validation", ... })`. Forward-link `tool-calls.md`'s **Argument shape** section as the owner of the runtime split.
