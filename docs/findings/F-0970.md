## Finding

The prompt-mode driver picks between two `pi.sendUserMessage` shapes — the no-options form when idle, and `{ deliverAs: "steer" }` when mid-stream — by reading `ctx.isIdle()` and then calling `pi.sendUserMessage(...)`. Per Pi's `examples/extensions/send-user-message.ts`, calling the no-options form while the session is mid-stream throws synchronously; that throw is the *only* failure surface the bullet enumerates apart from the post-`waitForIdle()` trailing-`assistant` probe.

The spec does not state the behaviour the `isIdle()`-then-`sendUserMessage` sequence relies on: that Pi cannot transition from idle to mid-stream (or vice versa) between the `isIdle()` read and the matching `sendUserMessage` call, *or* that any such transition is intentionally absorbed by the synchronous-throw → `kind: "transport"` secondary mapping rather than retried with the other form. Without that statement, an implementer reading the bullet has no basis to decide between (a) treating the two-step as atomic and trusting the form-selection, (b) wrapping the call in a retry that flips to `deliverAs: "steer"` on a `isIdle()`-disagreement throw, or (c) treating every `sendUserMessage`-time throw as terminal transport failure on the user's behalf.

This is the same shape of gap the file already pins for several other Pi behaviours that lack typed contracts — the `complete()` forced-tool, retry-timing, and cancellation presuppositions; the `waitForIdle()`-no-rejection consumption posture; the user-visible-streaming ordering posture; the driven-turn session-commit ordering — each surfaced as a named "consumption posture" or "behavioural presupposition" anchor and each routed to an editorial-review checklist item under [Pi version bump procedure](../spec_topics/pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure). The `isIdle()`→`sendUserMessage` form-selection race is unpinned in the same way and has no checklist item.

## Issue introduction

**Verdict:** introduced-at-commit
**Introducing commits:** `c993401` ("pi-loom spec: resolve \"Prompt-mode driver calls a non-existent `ctx.sendUserMessage`\"", 2026-05-04)
**History:** Earlier prose (pre-`c993401`) stated the mid-stream form by simple condition — "Issuing untyped queries via `ctx.sendUserMessage(text)` (or `{ deliverAs: "steer" }` if the agent is mid-stream)" — with no observable check-then-call site, so no TOCTOU surface existed. `c993401` fixed the wrong-method correctness bug (`ctx.sendUserMessage` → `pi.sendUserMessage`) and in the same edit introduced the explicit two-step "the per-handler `ctx` is consulted only for `ctx.isIdle()` to choose between the two forms (calling the no-options form mid-stream throws — see Pi's `examples/extensions/send-user-message.ts`)". That edit created the `isIdle()`→`sendUserMessage` race window. Subsequent edits to this bullet (`d37c563`, `aa334c2`, `0f81627`, `6829a79`, `803106d`, `c326bfa`, `85fc906`, `1ca13ec`) added the post-`waitForIdle()` error probe, the `waitForIdle()`-no-rejection presupposition, and the user-visible-streaming and driven-turn-commit consumption postures, but never pinned the form-selection-race behaviour the `c993401` edit's two-step relies on. The path-only rename in `f5e89f4` is non-substantive.

## Solution Space

**Shape:** multiple

### Option A — Pin the form-selection race as a named consumption posture with an editorial-review checklist item

**Approach.** Treat the `isIdle()`→`sendUserMessage` two-step on the same footing as the four other consumption postures already pinned in this file. Add a named anchor immediately after the `isIdle()`-then-`sendUserMessage` sentence (suggested: `<a id="sendusermessage-form-selection-race-presupposition"></a>*Form-selection race (consumption posture).*`) that:

1. names the behaviour the bullet relies on — "the `isIdle()` read and the matching `pi.sendUserMessage(...)` call observe the same agent-state generation, so the form selected by the read is the form Pi expects at the call site";
2. records the absorption rule — a lost race surfaces as a synchronous throw from `pi.sendUserMessage`, which is mapped to `Err(QueryError { kind: "transport", message: <error.message>, http_status: null, provider: <resolved-model-provider>, retryable: false })` per the existing secondary-mapping clause above; the runtime MUST NOT silently retry with the other `deliverAs` form;
3. notes the loom-side rationale — `@earendil-works/pi-coding-agent` types `isIdle()` as `() => boolean` and `sendUserMessage` as `void`, neither encoding the form-selection-race behaviour, and the property carries no build-time SDK surface-inventory signal;
4. routes detection to editorial review with a new lettered item appended to the checklist in `version-bump-step2.md` (next available letter past `(an)`).

**Spec edits.**

- `conversation-drive.md`: insert the named anchor + paragraph between the existing `*Form selection*` sentence and the *Resolution* clause.
- `version-bump-step2.md`: append a new SHOULD-level item to the *Editorial-review checklist for unpinned host presuppositions* mirroring the `(u)` / `(ak)` / `(al)` shape (with cross-reference back to the new anchor); bump the preamble's "items (f) through (an)" range to include the new letter.
- `version-bump-step2.md` preamble: the *all five* enumeration at line 3 already routes per-item recording; only the range-end token needs updating.

**Pros.**
- Symmetric with the four sibling presuppositions already pinned in the same file; one consistent maintenance discipline.
- Preserves the existing secondary-mapping disposition unchanged — implementer has one fewer behavioural choice to make.
- Detection routes through the existing version-bump editorial-review machinery; no new gate infrastructure.

**Cons.**
- Adds a sixth presupposition anchor to a file that already carries five; further accretion makes the file harder to scan.
- Extends the editorial checklist (already `(a)`–`(an)`) by one item; each new item has audit cost on every Pi bump.

**Risks.**
- The "MUST NOT silently retry" clause closes a future optimisation — if a Pi minor introduces a typed `idle-or-steer` form-agnostic send, the spec would need to revisit this clause.

### Option B — State the race-absorption disposition inline on the existing bullet, without a separate presupposition anchor

**Approach.** Append a one-sentence clause to the existing `isIdle()`-then-`sendUserMessage` sentence stating only the absorption disposition: "A state transition between the `isIdle()` read and the `pi.sendUserMessage` call surfaces as a synchronous throw at the call site and is mapped through the secondary `kind: "transport"` mapping above; the runtime MUST NOT retry with the other `deliverAs` form." No new anchor, no editorial-review checklist item.

**Spec edits.**

- `conversation-drive.md`: append the single sentence to the existing parenthetical.

**Pros.**
- Smallest possible edit; no new editorial-review burden.
- Implementer disambiguation is achieved directly at the read site.

**Cons.**
- Loses parity with the five other consumption-posture anchors in the same file — the form-selection race becomes the one unpinned-presupposition family member without an editorial-review hook.
- A Pi minor that changes the synchronous-throw behaviour (e.g. makes the no-options form's mid-stream rejection asynchronous) silently falsifies the absorption clause with no scheduled audit.

**Risks.**
- The asymmetry with the sibling presuppositions invites a future reviewer to "fix" the asymmetry, reopening this finding.

### Recommendation

Take **Option A**. The form-selection race is structurally identical to the four other consumption postures the file already pins — same unpinned-Pi-behaviour pattern, same falsified-on-Pi-minor failure mode, same lack of SDK surface-inventory signal — and the existing editorial-review machinery is exactly the home for it. The cost is one anchor and one checklist item; the benefit is uniform maintenance discipline across all five presuppositions in the file.

Edge cases the implementer must watch:

- The `(u)` / `(aa)` / `(ab)` / `(ak)` / `(al)` items all carry an *Authoritative source* clause naming the `dist/...` files where the behaviour is observed at the loom 1.0 Pi-SDK pin. The new item should cite `dist/core/extensions/types.d.ts` for the `isIdle()` and `sendUserMessage` declarations and `dist/core/sdk.js` (or the equivalent runtime module at the pin) for the synchronous-throw behaviour, with the **known-fragile evidence** caveat the sibling presuppositions already use.
- The "MUST NOT silently retry with the other `deliverAs` form" clause must be reconciled against `V9n`'s existing synchronous-`pi.sendUserMessage`-throw test — that test asserts the `kind: "transport"` mapping fires; under Option A it additionally pins that no retry intervenes.
- The new anchor letter — the checklist currently ends at `(an)`; the next item is `(ao)`. Bump the "items (f) through (an)" range token in the preamble to `(ao)` in the same edit.

## Consequence

**Severity:** advisory

A literal implementer can still produce a working driver — calling the unconditional `pi.sendUserMessage` shape selected by the `isIdle()` read, observing any synchronous throw, and routing it as `kind: "transport"` per the secondary mapping the bullet already pins. The gap is that two reasonable implementers can disagree on whether a lost `isIdle()` race is a transport failure to surface to the user (no retry; correct under the current secondary mapping) or a benign re-selection to absorb internally (silent retry with the other `deliverAs` form). A Pi minor that changes the idle→mid-stream transition timing — for instance, by making `isIdle()` reads weaker, or by making `pi.sendUserMessage`'s mid-stream throw lazy rather than synchronous — would silently shift which implementation choice is correct, with no build-time SDK surface-inventory signal.
