## Problem

The §5 `<descriptor>` rule in `placeholder-rendering-b.md` defines the `cli-flag` kind's value as "the CLI flag string", which admits two readings: just the path argument the operator wrote (e.g. `/proj/a/plan.loom`), or the full flag-plus-argument token (e.g. `--loom /proj/a/plan.loom`). The §7 `loom/load/cross-source-shadow` test vector silently chooses the flag-token reading (`cli-flag:"--loom /proj/a/plan.loom"`), but §5 — where the rule is defined — never states which. Two implementers reading only §5 will produce non-byte-identical messages for every `cli-flag` descriptor, defeating the byte-identical rendering contract. The `--loom` flag also accepts a multi-path argument joined by `path.delimiter`, so the per-component descriptor for a multi-path invocation is likewise undefined.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `61ea1a5` (2026-05-06) — "pi-loom spec: resolve \"Diagnostic message placeholders have no defined rendering\"" — introduced the §5 `<descriptor>` rule with the ambiguous phrase "the CLI flag string".
- `f930818` (2026-05-07) — "pi-loom spec: resolve \"placeholder rendering exemption open-ended\"" — added the `loom/load/cross-source-shadow` test vector that silently locks in reading (b) (flag-token-included), making the §5 ambiguity concrete and observable.

**History:** `git log --follow` on `docs/spec_topics/diagnostics/placeholder-rendering-b.md` (and on its pre-split ancestor `spec_topics/diagnostics.md`, since `f5e89f4` split the file) shows the §5 `<descriptor>` bullet has stood verbatim since its introduction in `61ea1a5`; `git log -G 'cli-flag.*--loom'` shows the contradicting §7 test vector first appears in `f930818`. The ambiguity was present at the §5 inception commit but was latent until the §7 vector at `f930818` chose one of the two readings without amending §5 to match. Neither commit alone is the durable fault; the interaction is — §5's prose and §7's vector were each internally reasonable when written but together produce two readings.

## Solution approach

Clarify the `cli-flag` clause in §5's `<descriptor>` bullet so the value is unambiguously the full `--loom <path>` flag token (flag name plus path argument, with the leading `--` retained), matching the existing §7 `cross-source-shadow` vector. In the same bullet, define the per-component descriptor for a multi-path `--loom` invocation (paths joined by `path.delimiter`): each derived component renders as `cli-flag:"--loom <that-component>"`, reconstructed per component rather than carrying the operator's literal multi-path argument.
