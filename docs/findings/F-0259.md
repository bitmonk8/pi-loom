## Finding

In both `V6d` and `V6d-T`, the **Tests** section carries the bullet:

> The `Result<T, E>` row does not arise from this surface (`params:` types never include `Result`).

This is a design-scope exclusion, not an assertion. It states *why a row of the stringification table is out of scope* for the `system:` surface — that `params:` types can never be `Result`, so the renderer never receives one. There is no observable behaviour to drive red-then-green here; nothing is exercised, asserted, or witnessed. A Tests bullet should name an assertion the leaf's test suite makes, and a `V6d-T` author scanning Tests for vectors to write has nothing to act on for this line.

The content is correct and worth keeping as a boundary qualifier — it tells the reader the stringification table is deliberately not exhausted across every Loom type. Its home is **Adds** (where the `system:` interpolation surface and the stringification table are described), as a scope note on that surface, rather than in the assertion list.

Commit 3625ee0 added the per-type stringification Tests bullet and, alongside it, the `Result<T, E>` exclusion line — placing the non-testable scope note inside the Tests list of both leaves at the same time the testable stringification vectors were introduced.

## Solution Space

**Shape:** single

### Recommendation

In both `docs/plan_topics/V6d-system-interpolation.md` and `docs/plan_topics/V6d-T-system-interpolation.md`, strike the Tests bullet:

> The `Result<T, E>` row does not arise from this surface (`params:` types never include `Result`).

and fold the same fact into **Adds** as a scope qualifier on the `system:` interpolation surface / stringification table — e.g. note that the stringification table covers the param-resolvable Loom static types and that `Result<T, E>` is excluded because `params:` types never include `Result`. The Tests sections retain only the per-type stringification vectors that are actually driven.

## Consequence

**Severity:** cosmetic

A non-testable design fact sits in the assertion list of both leaves, so a `V6d`/`V6d-T` author scanning Tests for vectors to author encounters a line with no observable to drive. No implementer is blocked or driven to divergence; the defect is organisational placement.
