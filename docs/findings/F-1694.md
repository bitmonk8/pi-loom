## Finding

The token `sendSystemNote` is used throughout the Pi-integration-contract cluster as if it were an established, defined name â€” "the `sendSystemNote` fallback chain", "the canonical `sendSystemNote` call", "MUST NOT propagate out of the `sendSystemNote` fallback chain", "MUST NOT be invoked from the teardown handler". It appears as a coined term in at least six files (`runtime-event-channel.md` including PIC-54, `diagnostic-emission-isolation.md`, `active-invocation-registry.md`, `session-shutdown-semantics.md`, `extension-bootstrap-and-per-loom.md`, and the `code-registry-*.md` diagnostic rows). Yet no site ever binds the name `sendSystemNote` to a referent.

The natural definition site is the **System notes** section at the top of `runtime-event-channel.md`. That section defines the single `pi.sendMessage({ customType: "loom-system-note", â€¦ }, { triggerTurn: false })` call shape and the renderer registration â€” but it never names that call shape `sendSystemNote`. Other files explicitly point at it as the definition (`active-invocation-registry.md`: "the best-effort chain defined under **System notes**"), so the reader is sent to a section that does not actually carry the name. The first arm of the documented fallback chain (`sendSystemNote` â†’ `ctx.ui.notify` â†’ `console.error`) is therefore an undefined identifier presented as a defined one.

A reader can recover the intended referent â€” `sendSystemNote` is the `pi.sendMessage`-on-`loom-system-note` call shape, degrading through `ctx.ui.notify` and finally `console.error` â€” only by inference from surrounding prose. The defect is that a name carrying normative weight (it is the subject of MUST/MUST-NOT clauses and the head of a routing chain cited from multiple pages) has no anchor, no binding sentence, and no glossary entry.

## Issue introduction

**Verdict:** single-commit (with later propagation)

## Solution Space

**Shape:** single

### Recommendation

Bind the name at its canonical definition site. In `runtime-event-channel.md`'s **System notes** section, add an anchor (e.g. `<a id="send-system-note"></a>`) on the paragraph that introduces the `pi.sendMessage({ customType: "loom-system-note", â€¦ })` call shape, and add one sentence naming that call shape `sendSystemNote` and stating that the best-effort fallback chain `sendSystemNote` â†’ `ctx.ui.notify` â†’ `console.error` is rooted on it. This makes **System notes** the single source of truth the other files already assume it is.

The bare uses in the other cluster files then become references to a defined term; they need no rewrite, but the first use in `runtime-event-channel.md` outside **System notes** (and ideally the cross-referencing bullet in `active-invocation-registry.md`) should link to the new anchor. A glossary entry is optional and secondary â€” the binding sentence at **System notes** is what the normative MUST/MUST-NOT clauses resolve against.

Edge case for the implementer: keep the new anchor stable, since `diagnostic-emission-isolation.md` and `session-shutdown-semantics.md` reach the chain by reference and a future cross-link will target it.

## Consequence

**Severity:** advisory

An implementer or auditor reading any single cluster file in isolation encounters a named helper (`sendSystemNote`) that resolves to nothing, and following the cross-reference to **System notes** finds the call shape but not the name. The referent is recoverable from context and no behavioural divergence results, so this is a navigation/guidance gap rather than a correctness fault.
