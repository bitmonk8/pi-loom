## Finding

`bindings.md` and `grammar.md` together define `let x = e` and the `LetStmt` grammar, and `loom/parse/immutable-rebinding` covers a *reassignment* (`x = …`) of a non-`mut` binding. Neither file states what a *second* `let x = e` (or `let x: U = e`) inside the same block does. The grammar admits the second `LetStmt` and the diagnostic registry has no code that fires on it; the prose closest to the question — expressions.md §"Identifier resolution", which says local bindings "shadow everything else lexically, the same as in Rust or TypeScript" — is about identifier resolution across the four namespaces (locals, top-level `fn`/import, callable set), not about whether two `let` statements may introduce the same name in the same scope. The simile is also self-cancelling: Rust accepts same-scope shadowing and rebinds; TypeScript rejects it with a duplicate-declaration error.

A conforming implementation can therefore pick either Rust-style shadowing (the second `let` introduces a fresh binding, possibly with a new type, that masks the first for the remainder of the block) or TypeScript-style rejection (a same-scope second `let` of an in-scope name is a parse error) and remain consistent with every cited rule. The two behaviours are observable: under shadowing, `let x = 0; let x: string = "a"; f(x)` typechecks against `f(s: string)`; under rejection, the second `let` fails to parse and no later sites see the new binding. A conformance test can pin neither outcome today.

## Solution Space

**Shape:** multiple

### Option A — Rust-style same-scope shadowing

**Approach.** A second `let x = e` (or `let mut x = e`) in the same lexical scope introduces a fresh binding that *masks* any earlier same-named binding for the remainder of the scope; the new binding's type is independent of the old one's. `loom/parse/immutable-rebinding` continues to fire only on reassignment (`x = …` without `let`), unchanged.

**Spec edits.**
- In `bindings.md`, after the `let` / `let mut` intro, add a `**Shadowing.**` paragraph: a `let`/`let mut` whose pattern names an identifier already bound in the same or an enclosing scope introduces a fresh binding that masks the prior one for the remainder of the introducing scope; the new binding's type is inferred or annotated independently; the old binding is no longer reachable by name from inside the new binding's scope (but values previously captured remain reachable). Include a worked example showing a type change across the shadow.
- In `bindings.md`, in the `immutable-rebinding` discussion (the existing code-block comment and the prose if any), restate that the diagnostic targets `x = …`, never `let x = …`, so the reader does not conflate the two.
- In `grammar.md` §`let form`, append one sentence pinning the same disposition for grammatical clarity: a second `LetStmt` introducing an in-scope name is legal and shadows.
- In `expressions.md` §"Identifier resolution", change "shadow everything else lexically, the same as in Rust or TypeScript" to drop the TypeScript half (TypeScript does not permit same-scope shadowing).

**Pros.** No new diagnostic; no implementation work beyond the parser already required for `let`. Matches the Rust heritage `bindings.md` already invokes ("Loom follows Rust's *immutable-by-default, opt-in mutability* convention"). Lets authors narrow a value's type by re-`let`-ing it post-validation without an intermediate name.

**Cons.** Shadowing across `let mut` and `let` (e.g. `let mut x = 0; let x = freeze(x)`) is a known foot-gun in Rust too; the spec inherits the same hazard. The benefit is small because Loom has no `if let` / `while let` pattern-binding form where Rust uses shadowing heavily.

**Risks.** A reader skimming `immutable-rebinding` may still assume the second `let` is "rebinding" in the colloquial sense; the prose must explicitly disambiguate.

### Option B — TypeScript-style same-scope duplicate-binding rejection

**Approach.** A `let`/`let mut` whose pattern names an identifier already bound in the **same** lexical scope is `loom/parse/duplicate-binding` (new E/parse code). Shadowing across nested scopes (an inner block introducing a same-named binding) remains permitted — this is the standard scope-walk behaviour and is what `expressions.md`'s "shadow everything else lexically" already implies for cross-scope lookup.

**Spec edits.**
- In `bindings.md`, add a `**Same-scope redeclaration.**` paragraph: a `let`/`let mut` whose pattern names an identifier *already bound by an earlier `let`, `let mut`, function parameter, `for` iteration variable, or `match` pattern in the same lexical scope* is `loom/parse/duplicate-binding`; cross-scope shadowing (inner block introducing the same name) is permitted and resolves per `expressions.md` §Identifier resolution.
- Register `loom/parse/duplicate-binding` in `docs/spec_topics/diagnostics/code-registry-parse.md` (E, parse): "second `let`/`let mut` introducing a name already bound in the same lexical scope", with rendered template `cannot redeclare binding '<name>' in the same scope; use a different name or wrap in a nested block to shadow`, owner link back to `bindings.md`.
- In `grammar.md` §`let form`, add the same disposition pinned to the grammar.
- In `expressions.md` §"Identifier resolution", change "shadow everything else lexically, the same as in Rust or TypeScript" to "shadow non-local namespaces lexically; same-scope redeclaration is `loom/parse/duplicate-binding` (see Bindings)".

**Pros.** Closes the foot-gun statically; matches TypeScript, which `expressions.md` cites alongside Rust as a heritage language. Loom's existing collision-error vocabulary (`import-name-collision`, `mut-on-immutable-context`, `tools:`-vs-`fn` registration failure) already trends toward rejecting silent shadowing within a single namespace; same-scope `duplicate-binding` is the consistent next step.

**Cons.** Introduces one new diagnostic that V3b/V3b-T must assert. Authors who want a type-narrowed second binding (`let x = parse(s); let x: Foo = validate(x)?`) must rename or wrap in a `{ … }` block expression.

**Risks.** The line between "same lexical scope" and "nested scope" must be pinned precisely so the diagnostic does not catch `for x in xs { let x = … }` patterns the spec explicitly permits (`for` iteration variable is itself a binding, but the `let` is in the body's nested scope). The fix paragraph must enumerate the source scopes the rule counts (top-level loom, function body, `if`/`else`/`while`/`for` body block, `match`-arm block expression, bare block expression).

### Recommendation

Option B — same-scope duplicate-binding rejection. Loom already rejects every other same-namespace name collision it can detect (`immutable-rebinding`, `import-name-collision`, `tools:`-shadows-`fn`, `mut-on-discard`, `mut-on-immutable-context`); admitting silent same-scope shadowing of `let` is the lone outlier and undercuts the spec's stated "no implicit shadowing" stance in `imports.md`. Loom has no idiom (no `if let`, no `while let`) that depends on Rust-style shadowing, so the cost is low. Implementer edge cases to watch: (i) the rule fires on the *same* lexical scope only — an inner block legitimately introduces a fresh binding; (ii) the rule applies symmetrically to function parameters and `for` / `match` bindings as the colliding *prior* binding, not only to a prior `let`; (iii) the per-scope check must run after `Pattern` is resolved so `let _ = …` followed by `let _ = …` is permitted (the discard sigil is not a name).

## Consequence

**Severity:** correctness

Two reasonable implementers will produce divergent parsers: one accepts `let x = 0; let x: string = "a"` and rebinds, the other rejects it. The divergence is silent (no diagnostic differs in name or shape — under the shadowing reading there is no diagnostic at all, under the rejection reading a presently-unnamed code fires), so cross-implementation `.loom` portability and any test suite asserting either behaviour are at risk.
