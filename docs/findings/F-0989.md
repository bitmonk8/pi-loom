## Problem

`bindings.md` and `grammar.md` define `let x = e` and the `LetStmt`
production, and `loom/parse/immutable-rebinding` covers *reassignment* of a
non-`mut` binding — but neither file states what a second `let x = e` (or
`let x: U = e`) inside the same block does. The grammar admits the second
`LetStmt`, no diagnostic fires on it, and the nearest prose
(`expressions.md` §Identifier resolution: locals "shadow everything else
lexically, the same as in Rust or TypeScript") is about cross-namespace
resolution, not same-scope redeclaration — and is self-cancelling, since
Rust rebinds while TypeScript rejects. A conforming implementation can
therefore pick either Rust-style shadowing or TypeScript-style rejection
and stay consistent with every cited rule, so a conformance test can pin
neither outcome.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `53f5831` (2026-05-03, *spec: rewrite query primitive, system prompts, and schema syntax*) for `bindings.md` content; `f841b12` (2026-05-04, *spec: resolve Cluster 2 (literal sublanguage + grammar appendix)*) for the `## let form` grammar section.
**History:** `git log --follow docs/spec_topics/bindings.md` and `git log --follow docs/spec_topics/grammar.md` show the `let`/`let mut` paragraph and the `LetStmt ::= "let" "mut"? Pattern (":" Type)? "=" Expr` production were authored in those two commits respectively and have not been substantively edited since (subsequent commits on each file address adjacent rules — initialiser-required diagnostic naming, deferral-note consolidation, file moves). `git log -S 'duplicate-binding' -S 'redeclar' -S 'same-scope let'` over the whole corpus history returns no matches, confirming the question has never been addressed in any direction. The defect is therefore a coverage gap baked into the topic pages at first authorship rather than a regression: every release that has ever shipped a `let` rule has left the second-`let`-in-the-same-block case undefined.

## Solution approach

Resolve the underspecification by specifying in `bindings.md` that a
`let`/`let mut` naming an identifier already bound in the same lexical
scope is a parse error, and coin `loom/parse/duplicate-binding` for it in
`code-registry-parse.md`. Pin the same disposition in `grammar.md`
§`let form`, and rewrite the self-cancelling Rust/TypeScript simile clause
in `expressions.md` §Identifier resolution to point at the bindings rule
rather than leaving same-scope redeclaration to the reader.

## Solution constraints

- Cross-scope shadowing (an inner block introducing a same-named binding, e.g. `for x in xs { let x = … }`) MUST remain permitted; the new diagnostic fires on same-lexical-scope redeclaration only.
