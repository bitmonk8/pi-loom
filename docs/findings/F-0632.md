## Finding

Step 0 (d) of the capability probe (`capability-probe.md` §`entry-capability-probe`) **requires** the factory to read each of the four `@earendil-works/*` peer packages' installed `package.json#version` by resolving `createRequire(import.meta.url).resolve("<pkg>")` and walking parents to the matching `package.json`. The factory module lives at `extensions/index.ts` and is wired into `src/**` by `H4a` (the factory delegates into the `src/**` implementation, where the probe code itself lives — see `H4a` Adds: "the factory implementation lives under `src/**`"). The inventory-closure audit (`audit-target-categories.md` §`audit-recognised-shapes`) is scoped to `src/**/*.ts` (`audit-resolution.md` §`audit-scope`) and prohibits, under family (4) with its own discriminator, "any function call whose call site uses `createRequire` imported from the Node `"module"` package and whose result is invoked with one of the in-scope package specifiers" — with no exemption for `.resolve()`, no exemption for the probe, and no path-discriminator carving out the factory module. Family (4) is the non-exemptible family in `audit-exemption-mechanism`; the same-line `// allow-pi-surface:` marker is explicitly disallowed on it.

Worse, the same paragraph designates a hard `src/` precondition: "the implementation's land-green precondition … MUST verify both this `package.json#type` value and the absence of CJS-reach shapes in `src/` before the audit's first run on `main`." That precondition is also wired into the plan (`V18b` Adds: "in the same commit, every existing `src/` Pi-side reference the new audit would surface is deleted, promoted to an `SDK_SURFACE_INVENTORY` entry, marked with an `// allow-pi-surface:` exemption, or rewritten into a recognised shape"). The spec therefore both *requires* the probe's `createRequire` call to exist in the factory's `src/**` implementation and *requires* its absence as the audit's land-green precondition. The two obligations cannot both be honoured.

A secondary ambiguity hangs on the same paragraph: the prohibition's text says `createRequire(...)` "**plus** `req("<pkg>")` style indirection", which reads as gating the prohibition on the value-loading call, but the broader rule below (`createRequire`'s "result is invoked with one of the in-scope package specifiers") makes no distinction between `.resolve()` (returns a path string, loads no module) and a bare call (loads the module). An implementer cannot tell whether `.resolve()` alone is in scope of family (4); two reasonable readers will disagree.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `126ff8a` (2026-05-06) — "pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B" — introduced Step 0 (d) with the `createRequire(import.meta.url).resolve("@<scope>/pi-coding-agent/package.json")` mechanic. Family-(4) CJS-reach prohibition did not yet exist in the audit prose, so no conflict at this commit.
- `e5cd514` (2026-05-09) — "pi-loom spec: resolve 'T35 SDK capability inventory closed-set negative-direction audit'" — added the inventory-closure audit's *Recognised import/access shapes* paragraph, including the absolute family-(4) prohibition on any `createRequire`-rooted call reaching the four peers, with no carve-out for the Step 0 (d) probe mechanic that had been live for three days. This is the commit that created the conflict.
- `4c27f32` (2026-06-03) — "pi-loom spec: resolve 'Step 0 (d) peer-dep version probe relies on a subpath export'" — replaced the `.resolve("<pkg>/package.json")` subpath form with `.resolve("<pkg>")` + parent-walk to avoid `ERR_PACKAGE_PATH_NOT_EXPORTED`. The conflict survived this edit: the rewritten probe still uses `createRequire` reaching an in-scope peer specifier, so it remains a family-(4) shape.

**History:** The defect is the **interaction** between (a) the probe's CJS-reach mechanic that 126ff8a introduced as the only `exports`-independent way to discover `package.json` and (b) the absolute family-(4) prohibition e5cd514 layered on top without surveying existing probe-side citations. Each edit was locally sensible; neither author treated the other surface as in-scope of their patch. The durable fix lives in the audit prose or in the probe mechanic, not in either commit individually.

## Solution Space

**Shape:** multiple

### Option A — Narrow audit carve-out for the probe's `createRequire(...).resolve(<peer>)` shape

**Approach.** In `audit-target-categories.md` §`audit-recognised-shapes`, extend the family-(4) `createRequire(...)` clause with a single carve-out: a call shape of `createRequire(<arg>).resolve(<specifier>)` (i.e. `.resolve()` on the `createRequire`-returned function, never a bare value-loading invocation) against one of the four peer specifiers is exempt from the family-(4) prohibition iff (i) the call's lexical site is the Step 0 (d) probe in the factory's pinned-constants probe module, and (ii) the call is followed only by `package.json` discovery (no module load). Cross-reference the carve-out from `capability-probe.md` §Step 0 (d). Add an exhaustive rule that `.resolve()` returns a path string and does not "reach" the peer in the family-(4) sense the prohibition targets (value-loading reach), so the prohibition's text now disambiguates the secondary question the finding raises.

**Spec edits.**
- `audit-target-categories.md`: insert the carve-out clause inline in the family-(4) `createRequire` paragraph; restate the value-loading vs path-resolution distinction.
- `capability-probe.md` §Step 0 (d): add a back-reference to the carve-out anchor so a reader landing on the probe sees the audit is aware of it.
- `audit-exemption-mechanism`: no change — the carve-out is at the rule level, not the marker level (family (4) remains non-exemptible by marker).

**Pros.** Smallest spec surface touched. Preserves the existing probe code shape `V9a` is wired to. Removes the secondary `.resolve()`-in-scope ambiguity in the same edit.

**Cons.** Introduces a site-conditional rule the audit walker must check (call-site lexical location), modestly enlarging the audit's discriminator surface. Carve-out is narrowly tied to one probe; another caller that has a legitimate `.resolve()` use later needs a fresh edit.

**Risks.** A site-conditional rule keyed on "the probe module" needs a syntactic anchor the audit can verify (e.g. a same-line trailing marker the audit recognises only on this one call shape, or a pinned `path:line` location in the audit-shared pinned-constants block). If the anchor design is fuzzy the carve-out becomes an open back-door.

### Option B — Re-mechanic Step 0 (d) so the probe holds no `createRequire` shape at all

**Approach.** Replace step (d)'s `createRequire(...).resolve(...)` + parent-walk with a static JSON-import of each peer's `package.json` (e.g. `import pkg from "@earendil-works/pi-coding-agent/package.json" with { type: "json" }`), or with a `node:fs`-backed read against a path the build pins. The probe's observable contract (route the four enumerated failure conditions to `peer-dep-malformed-version`, etc.) is unchanged; only the mechanic moves. The factory's `src/**` source tree no longer carries any `createRequire` shape, so the audit and the land-green precondition stay coherent without any audit-side carve-out.

**Spec edits.**
- `capability-probe.md` §Step 0 (d): replace the prescribed mechanic with the alternative (or restate the step in outcome-only terms — "read each peer's installed `version` by any `exports`-independent mechanic that holds no `createRequire`-rooted call against an in-scope peer specifier" — and demote the concrete mechanic to a non-normative recipe).
- `audit-target-categories.md`: no change.

**Pros.** Eliminates the conflict structurally rather than by exemption — the family-(4) prohibition stays absolute, easier to audit. Composes with the sibling finding *"probe (d) peer-dep version-read mechanic over-prescribed"* (which separately argues the mechanic is over-prescribed). The audit's `src/` land-green precondition becomes a real, uniform property.

**Cons.** Each replacement mechanic carries its own caveats: `with { type: "json" }` import-attribute support across the Node floor (`>=22.19.0`) needs verifying, and a static-import path bypasses Node's `exports`-map resolution which was the original problem 4c27f32 solved. A `node:fs`-backed read needs a path-discovery story that does not itself reintroduce `createRequire`. Implementers across the four peers must converge on one alternative.

**Risks.** A subtly-different observable shape (e.g. JSON-import failure throws a different code than the `MODULE_NOT_FOUND` step (d) currently routes by) silently changes which of the four `peer-dep-malformed-version` conditions a given failure routes through, perturbing the diagnostics-side test contract in `placeholder-rendering-b.md`.

### Recommendation

**Take Option A**, with `.resolve()`-vs-bare-call disambiguation included in the same edit. The probe mechanic is already once-rewritten under 4c27f32 to dodge `ERR_PACKAGE_PATH_NOT_EXPORTED`; rewriting it a second time to dodge the audit risks chasing the next undiscovered mechanic-side caveat (per the sibling finding noting that the probe assumes discrete on-disk `package.json`s — Option B does not help with that). The audit prose is where the rule that names `createRequire` lives, and the same paragraph already enumerates several site-shape carve-outs (the `as`-rename / namespace-import / aliased-rebinding sub-rules, the `"module"`-package "MUST NOT be read as authorising by omission" clause), so one more narrow carve-out is in keeping with the paragraph's existing structure.

Edge cases the implementer must watch:
- The carve-out MUST key on the call shape `createRequire(<arg>).resolve(<peer-specifier>)` specifically — a bare `createRequire(<arg>)(<peer-specifier>)` value-loading call (with or without an intermediate `req` binding) MUST remain a family-(4) violation.
- The carve-out's lexical-site anchor MUST be machine-checkable by both a static-AST walker and the type-aware variant (`audit-target-categories.md` permits both); a syntactic marker (e.g. a stable `// probe-step-0d:` comment the audit recognises in this exact position, or a pinned `path:line` in the same source-of-truth pinned-constants block that already holds the Node floor, capability list, etc.) avoids tying the audit to a TypeScript-program load.
- The `.resolve()` vs bare-call distinction MUST land in the same edit; leaving it implicit invites Option A to be the source of the next ambiguity finding.
- `V18b`'s land-green `src/` sweep instruction needs a cross-reference update: the sweep MUST NOT delete the probe's `createRequire` call, only every other `createRequire`-reach shape.

## Consequence

**Severity:** blocking

`V9a` and `V18b` cannot both ship green as currently specified: the probe call `V9a` is required to emit is a family-(4) shape `V18b` is required to refuse, with no permitted exemption. The contradiction surfaces the first time the audit runs on `main` after `V9a` lands (or, symmetrically, on `V18b`'s land-green sweep), where the spec demands two incompatible source-tree states in the same commit. Implementers either rewrite the probe in a way the spec does not currently sanction, or wire a carve-out the spec does not currently authorise, or stall.
