## Problem

The capability probe's Step 0 (d) (`capability-probe.md` §`entry-capability-probe`) requires the factory's `src/**` implementation to read each of the four `@earendil-works/*` peer packages' installed `package.json#version` via a `createRequire(import.meta.url).resolve("<pkg>")` call plus parent-directory walk. The inventory-closure audit's family-(4) clause (`audit-target-categories.md` §`audit-recognised-shapes`) prohibits any `createRequire`-rooted call reaching one of the in-scope peer specifiers anywhere in the audited `src/` tree, with no exemption for the probe and no permitted `// allow-pi-surface:` marker; the same paragraph additionally makes the absence of CJS-reach shapes in `src/` a land-green precondition. The spec therefore simultaneously requires the probe's `createRequire` call to exist and to be absent — the two obligations cannot both be honoured. Separately, the paragraph does not say whether a `.resolve()` call (which returns a path string and loads no module) is in family-(4) scope or only a bare value-loading invocation is; two readers will disagree.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `126ff8a` (2026-05-06) — "pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B" — introduced Step 0 (d) with the `createRequire(import.meta.url).resolve("@<scope>/pi-coding-agent/package.json")` mechanic. Family-(4) CJS-reach prohibition did not yet exist in the audit prose, so no conflict at this commit.
- `e5cd514` (2026-05-09) — "pi-loom spec: resolve 'T35 SDK capability inventory closed-set negative-direction audit'" — added the inventory-closure audit's *Recognised import/access shapes* paragraph, including the absolute family-(4) prohibition on any `createRequire`-rooted call reaching the four peers, with no carve-out for the Step 0 (d) probe mechanic that had been live for three days. This is the commit that created the conflict.
- `4c27f32` (2026-06-03) — "pi-loom spec: resolve 'Step 0 (d) peer-dep version probe relies on a subpath export'" — replaced the `.resolve("<pkg>/package.json")` subpath form with `.resolve("<pkg>")` + parent-walk to avoid `ERR_PACKAGE_PATH_NOT_EXPORTED`. The conflict survived this edit: the rewritten probe still uses `createRequire` reaching an in-scope peer specifier, so it remains a family-(4) shape.

**History:** The defect is the **interaction** between (a) the probe's CJS-reach mechanic that 126ff8a introduced as the only `exports`-independent way to discover `package.json` and (b) the absolute family-(4) prohibition e5cd514 layered on top without surveying existing probe-side citations. Each edit was locally sensible; neither author treated the other surface as in-scope of their patch. The durable fix lives in the audit prose or in the probe mechanic, not in either commit individually.

## Solution approach

Reconcile the contradiction in the audit prose where the family-(4) `createRequire(...)` clause lives (`audit-target-categories.md` §`audit-recognised-shapes`), narrowing it so the Step 0 (d) probe's version-read call is no longer prohibited. In the same edit, clarify whether a `.resolve()` call — which returns a path string and loads no module — is in family-(4) scope or only a bare value-loading invocation is. Add a forward-link from `capability-probe.md` §Step 0 (d) (anchor `entry-capability-probe`) to the reconciled audit clause.

## Solution constraints

- The carve-out MUST stay narrow: only the path-resolution shape `createRequire(<arg>).resolve(<peer-specifier>)` is exempted; a bare value-loading `createRequire(...)("<peer>")` invocation (with or without an intermediate `req` binding) MUST remain a family-(4) violation.
- Family (4) MUST remain non-exemptible via the same-line `// allow-pi-surface:` marker per `audit-exemption-mechanism`; the reconciliation lands at the rule level, not by authorising the marker on family (4).
