## Problem

The typed-query reference-implementation bullet in `implementation-notes.md` states the concrete provider-native `tool_choice` wire payloads that pi-ai's per-provider adapter maps `complete()`'s `options.toolChoice` to (Anthropic `tool_choice: { type: "tool", name }`; OpenAI Completions `tool_choice: { type: "function", function: { name } }`; "Mistral and Bedrock map analogously") as flat spec fact. These payload shapes are owned externally — by each provider's published HTTP API and by pi-ai's per-provider adapter — yet they carry no pointer to those owners and no provider-owned/may-drift hedge, and the "map analogously" clause asserts a shape it does not spell out. A provider reshaping its `tool_choice` payload would leave the bullet silently stale, because the change is invisible to both the pi-ai version pin and the SDK surface-inventory test. The only normatively load-bearing fact is that the runtime forces tool choice through `complete()`'s `options.toolChoice`; the per-provider shapes are illustrative.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `82be125` — "spec: adopt P1 typed-query two-phase loop (Option A)" (2026-05-04)
**History:** `git log -S`/`-G` over `implementation-notes.md` shows the provider-native payload sentence ("Anthropic `tool_choice: { type: "tool", name }`; OpenAI Completions `tool_choice: { type: "function", function: { name } }`; Mistral and Bedrock map analogously") first entered the corpus in `82be125`, as part of the new typed-query reference-implementation bullet that replaced the prior typed-query description. It was introduced without a provider-docs/pi-ai pointer or drift hedge from the outset. Later commits that touched the same bullet — `7f75947` ("resolve Typed-query forced-respond tool choice delivery channel", which rerouted the forced turn through `complete()`) and the `docs/` relocation `31ff060` — carried the assertion through verbatim without adding the missing pointer/hedge. The defect has therefore been present since the assertion's inception in `82be125`, not introduced by a later edit.

## Solution approach

Demote the per-provider wire payloads in the typed-query reference-implementation bullet to a non-normative illustration, keeping the normative claim as the `complete()` `options.toolChoice` channel and adding a pointer to pi-ai's per-provider adapter and each provider's published API docs as the payload owners. Route drift re-check to the existing `complete()` forced-tool behavioural presupposition at `conversation-drive.md#complete-forced-tool-presupposition` (already covered for all four providers by editorial-review checklist item (u) at `version-bump-step2.md#bump-checklist-complete-forced-tool`), mirroring `provider-error-mapping.md#provider-overflow-wording-presupposition`. Drop the unspecified "map analogously" clause rather than spelling out the Mistral/Bedrock shapes.

## Solution constraints

- The normative claim that forced tool choice is delivered exclusively through `complete()`'s `options.toolChoice` for the forced respond turn MUST survive the rewrite — that channel restriction is load-bearing and may not be weakened.
