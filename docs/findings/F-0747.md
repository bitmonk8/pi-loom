## Finding

`loom/load/settings-value-out-of-range`'s *Trigger* (in `docs/spec_topics/diagnostics/code-registry-load.md`) now fires on two top-level shape failures in addition to the four `looms.*` scalar keys: a top-level `loomPaths` that is not a JSON array (e.g. an object, string, number, boolean) and a top-level `looms` that is not a JSON object (e.g. an array, string, number, boolean). The companion rule `Top-level shape validation` in `docs/spec_topics/discovery/package-and-settings.md` confirms this: a `loomPaths`-as-object or `looms`-as-array logs exactly one `settings-value-out-of-range` per malformed top-level key per file.

The `<observed>` token in this row's *Message* template (`settings key <key> value is out of range; got <observed>`) is governed by the parsed-scalar carve-out paragraph in §8 of `docs/spec_topics/diagnostics/placeholder-rendering-b.md`. That paragraph enumerates rendering rules only for the scalar JSON kinds: `number` (canonical decimal), `boolean` (`true`/`false`), `null` (literal `null`), and `string` (category-5 `<key>` rule). It does not cover `array` or `object` kinds, even though the trigger now routes such values through `<observed>`. A producer faced with `loomPaths: { "a": 1 }` or `looms: [1, 2]` has no normative answer for what byte string `<observed>` substitutes, and DIAG-4's byte-identicality guarantee fails: two conformant implementations could legitimately render `[object Object]`, `{"a":1}`, `object`, `<unreadable>`, or anything else, all without violating any pinned rule.

The asymmetry inside the carve-out itself confirms the gap: §7 already maintains a separate `<kind>` closure for `loom/load/settings-invalid-entry` and `loom/load/manifest-invalid` whose enumeration explicitly includes `array` and `object` (alongside the four scalars), because those two rows render the JSON-kind token rather than the offending value. The §8 carve-out for `settings-value-out-of-range` made a different choice — render the value, not the kind — but only finished the table for scalars.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `72e2c31` (2026-06-05, "Placeholder-rendering closure omits live placeholders"); `4c00462` (2026-06-06, "top-level loomPaths/looms shape failures have no surfacing rule")
**History:** `git log -G "settings-value-out-of-range" -- docs/spec_topics/` shows two relevant commits after the row's introduction (`cb82bbf`, 2026-06-01, scalar-only trigger). `72e2c31` introduced the §8 parsed-scalar carve-out paragraph in `placeholder-rendering-b.md` enumerating rules for `number`/`boolean`/`null`/`string` only — coherent at the time, because the row's *Trigger* covered only `looms.*` scalar keys. The next day `4c00462` extended `code-registry-load.md`'s *Trigger* to include top-level `loomPaths`/`looms` shape failures (where the offending value can now be a JSON array or object) and added the matching `Top-level shape validation` paragraph in `discovery/package-and-settings.md`, but left the §8 carve-out untouched. The defect is the interaction: neither commit is wrong in isolation, but their combination leaves the non-scalar arm of the trigger with no `<observed>` rendering rule. `git blame` on the carve-out paragraph confirms `72e2c31` as the still-current author of the lines that need extending.

## Solution Space

**Shape:** multiple

### Option A — Render the JSON kind for non-scalar values

**Approach.** Extend the §8 parsed-scalar carve-out so the rendering for `array` and `object` values is the JSON-kind token (`array`, `object`), matching the established convention used by the §7 `<kind>` closure for `loom/load/settings-invalid-entry` and `loom/load/manifest-invalid`. Scalars continue to render their value verbatim per the existing rules.

**Spec edits.**
- In `placeholder-rendering-b.md` §8, add to the parsed-scalar carve-out paragraph: "an `array` renders the literal token `array`; an `object` renders the literal token `object`. This rendering applies only on the top-level `loomPaths`/`looms` shape-failure arm of `loom/load/settings-value-out-of-range` (per [Discovery — Top-level shape validation](../discovery/package-and-settings.md)); the frontmatter row never observes `array`/`object` inputs."
- Add a test vector: a `loomPaths: { "a": 1 }` input renders `settings key loomPaths value is out of range; got object`; a `looms: [1, 2]` input renders `settings key looms value is out of range; got array`.
- No change to `code-registry-load.md`'s *Message* template.

**Pros.**
- Reuses the existing `<kind>` token vocabulary; readers already encounter it for the sibling rows.
- Wire format is fully byte-identical with no implementation freedom; the existing strict-equality exclusion paragraph stays correct.
- One-paragraph edit, no template change.

**Cons.**
- Two rendering modes coexist on one row (value for scalars, kind for non-scalars); the *Message* "got `<observed>`" reads slightly differently in the two arms — `got 25` (a value) vs `got object` (a kind).
- Loses the contents of the malformed value (operator sees "got object", not which object).

**Risks.** Minimal. The operator-information loss is already accepted for the sibling rows.

### Option B — Render compact JSON for non-scalar values

**Approach.** Extend the carve-out so `array` and `object` render as compact `JSON.stringify` of the parsed value, matching the [canonical stringification table](../query/query-escapes-stringification.md#stringification-of-interpolated-values)'s rule for `array<T>` and schema-typed objects in query interpolation.

**Spec edits.**
- In `placeholder-rendering-b.md` §8, add to the carve-out: "an `array` or `object` renders as compact `JSON.stringify` of the parsed value, with no pretty-printing and no wire-name translation (the JSON keys are operator-visible settings-file paths, not loom-side identifiers)."
- Add test vectors: `loomPaths: { "a": 1 }` renders `... got {"a":1}`; `looms: [1, 2]` renders `... got [1,2]`; a deeply nested object renders its full serialisation.
- Decide whether to truncate (e.g. the 80→77+`...` runtime-value rule from category 2) to bound output; if not, document the unbounded case.

**Pros.**
- Preserves operator-visible information about *which* malformed value caused the diagnostic.
- Aligns with an existing pinned stringification table.

**Cons.**
- Larger spec surface: must decide on truncation, key ordering for objects (the BNDR-9 toolResult ordering finding is an open question on the same shape), and what happens with cycles (the parsed JSON cannot cycle, so this is mostly theoretical).
- Two byte-identicality regimes on the same placeholder: scalars by the §8 carve-out, non-scalars by the JSON-stringify table.

**Risks.** The key-ordering question forks into a separate decision the spec does not currently take for this surface; resolving it here creates a precedent that may bind future rows.

### Recommendation

Take **Option A**. The matching `<kind>` convention is already pinned for the two sibling rows (`settings-invalid-entry`, `manifest-invalid`) and a one-paragraph carve-out extension closes the gap with no new decisions to take. Option B's operator-information advantage is real but small: the *Trigger* row already names which top-level key is malformed via `<key>`, and the practical operator action ("set `loomPaths` to a JSON array") does not depend on the contents of the malformed value. Edge case for the implementer: the rendering modes split per *Trigger* arm (top-level shape vs `looms.*` scalar), not per JSON kind globally — the row never observes a stringly-typed scalar in the top-level arm and never observes an array/object in the scalar arm, so the disjunction is well-defined.

## Consequence

**Severity:** correctness

Two conformant implementations of the runtime can render the same `loomPaths: { "a": 1 }` input as visibly different `<observed>` substrings, defeating DIAG-4's byte-identicality contract for this row and breaking the related build-time-enforced exclusion from §8's strict-equality prohibition (tests are told they MAY assert this row's full *Message* by byte-identical equality, but the byte string is undefined for these inputs). A conformance test for the non-scalar arm cannot be written from the spec.
