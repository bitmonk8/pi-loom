## Finding

The `bind_context: session` truncation walk runs *before* the binder LLM call is issued: it sums per-message token counts through the injected `TokenEstimator` seam (PIC-16) and sources its message list from `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages`, then hands the selected turns to the compact-transcript renderer that builds the binder's system prompt. This is the binder's *input-construction* phase, and it can fail in ways that never reach the network: `estimateTokens` (the production `TokenEstimator` adapter) raising, or `buildSessionContext` returning a shape the per-variant compact-transcript renderer cannot consume (e.g. a message whose variant falls outside the `user` / `assistant` / `toolResult` / `custom` union the renderer is required to be total over).

The binder's failure model does not account for this phase. The *Failure-class taxonomy (normative)* partitions outcomes into transport, malformed-envelope, AJV-on-`args`, and context-overflow — every one of which presupposes a binder call was issued and a provider response (or network failure) was classified. The *Failure-mode templates (normative)* table then enumerates "the seven user-facing shapes" (needs_info, ambiguous, transport failure, malformed envelope, AJV, cancelled, custom-type-unsafe) and the section opens by asserting "every shape below" is rendered through the shared system-note discipline. Of these, only the custom-type-unsafe row (BNDR-9) is an input-construction failure, and it is surfaced precisely because BNDR-9 explicitly routes it to that row. A `TokenEstimator` throw or a non-consumable `buildSessionContext` shape has no analogous routing: it produces no binder envelope, no transport / AJV / malformed classification, and matches none of the seven shapes — yet the table's framing presents itself as the complete enumeration of binder failure surfaces.

The spec already adopts a coherent posture for host-seam misbehaviour: the chronological-ordering, leading-`user`-message, and `TokenEstimator` per-message-determinism preconditions are all framed as unpinned behavioural *presuppositions* taken on a best-effort basis and re-audited on each Pi minor bump — i.e. their violation is undefined behaviour loom does not defend against at runtime. The gap is that those presuppositions cover *value* correctness (ordering, determinism, leading element) and say nothing about a seam *throwing* or returning an out-of-union shape, and the failure-mode table never states that its enumeration is bounded to outcomes *after* a well-formed binder input has been constructed. Which contract owns the pre-call input-construction failure path is therefore left implicit.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `83802db` ("spec: collapse to single typed argument style; add LLM-driven slash-command binder")
**History:** The spec corpus is git-tracked. `git log -S` over the binder failure-mode enumeration ("Binder failures are runtime-handled") and the session-context token-budget walk ("8000") converges on a single originating commit: `83802db` introduced the LLM-driven slash-command binder, adding *in the same commit* both the `bind_context: session` context-injection walk (then "~20 turns or ~8000 tokens") and the "Failure modes … the five user-facing shapes" table. From that first version the table enumerated post-binder-call outcomes plus cancellation and never carved out the pre-call input-construction phase, so the omission is structural to the binder's inception. Later commits elaborated the input-construction seams without reconciling the table's scope — `1f37b15` (the `TokenEstimator` DI seam), `d69ec89` (the `buildSessionContext` citation), `f41af5d`/F-1131 (the seventh, custom-type-unsafe, row — the one input-construction failure that *was* given a surface), and `341bcd9`/F-1618 plus `05e84dd`/`ce4e476` (the determinism, ordering, and leading-`user`-message presuppositions). None of these widened or qualified the failure-mode table's exhaustiveness claim. The gap therefore traces cleanly to `83802db` and has persisted unaddressed through every subsequent binder edit (file split at `f5e89f4` carried the content forward unchanged).

## Solution Space

**Shape:** single

### Recommendation

Do not add a new failure-mode row or a new `QueryError` / diagnostic variant for the external seam's behaviour. Instead narrow the exhaustiveness framing in `determinism-cancellation-failure.md`:

- In the *Failure-mode templates (normative)* section (and/or the *Failure modes* intro that says "every shape below"), state that the table enumerates only the user-facing outcomes that arise *after* a well-formed binder input has been constructed and a binder attempt has been classified — explicitly excepting the BNDR-9 custom-type-unsafe row, which is the one input-construction failure the binder deliberately routes into the table.
- Add one sentence pinning that a host-seam precondition violation *during* input construction — `TokenEstimator`/`estimateTokens` raising, or `buildSessionContext` returning a shape outside the `AgentMessage` union the compact-transcript renderer is total over (the renderer-totality, leading-`user`-message, chronological-ordering, and per-message-determinism preconditions catalogued under [host-prerequisites.md]) — is governed by the existing host-presupposition posture (undefined behaviour loom does not surface as a binder failure shape) and is out of the table's scope.

This pins which contract owns the path and removes the over-claim without authoring any new normative error surface about an externally-owned seam. Edge case the implementer must keep consistent: the custom-type-unsafe row stays *in* scope (it is loom's own pre-render rejection, not a host-seam violation), so the carve-out must name it as the explicit exception rather than blanket-excluding all input-construction failures.

## Consequence

**Severity:** advisory

If `estimateTokens` throws or `buildSessionContext` yields a non-consumable shape during input construction, the spec gives no defined user-facing surface: two implementers could diverge between letting the exception crash the slash handler, swallowing it, or surfacing an ad-hoc note. Because loom's general posture for host-presupposition violations is undefined behaviour, the practical defect is a guidance/scope gap — the failure-mode table over-claims completeness — rather than a behaviour an implementer cannot ship without.
