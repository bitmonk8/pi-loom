## Finding

In `audit-failures.md` → *Stale sub-kinds*, family-(5) sub-kind (s2) is coined with the `<symptom>` token `all-in-inventory`. The token's literal reading asserts that the originating line's references "now resolve to an `SDK_SURFACE_INVENTORY` entry." But the predicate is broader: it fires when every in-scope reference resolves to the inventory **or** to a typebox sibling allow-list entry (`{ Type }` for named imports, `{ Unsafe }` for `Type.<member>` member-access), per the *Target surface categories* typebox carve-out. The token names only the first of the two resolution paths.

The spec is aware of the mismatch and discloses it inline: "the token is more precisely `all-in-inventory-or-allow-list` — PIC retains the shorter `<symptom>` token to keep the wire-discriminator stable, and the parenthetical here is the normative scope." That stability rationale has no force here. The token is coined for the first time at this point in the corpus; there is no prior wire contract for a shorter name to stay stable against. The spec is therefore choosing an admittedly-imprecise name and then justifying it by a stability concern that does not yet exist — when the accurate name could simply be coined now.

The defect is contained: the parenthetical fully states the normative scope, so the predicate itself is unambiguous, and the `<symptom>` strings are implementation-owned per the *Failure-surface contract*. The cost is a misleading canonical label plus a self-acknowledged "more precisely X" caveat that exists only because the chosen name is wrong.

## Solution Space

**Shape:** single

### Recommendation

Rename sub-kind (s2)'s `<symptom>` token to a form that does not claim inventory-only resolution, and delete the self-acknowledging stability caveat. Recommended token: `all-resolved` — neutral, kebab-case, and covers both the `SDK_SURFACE_INVENTORY` and typebox-allow-list paths without implying either exclusively. (The precise `all-in-inventory-or-allowlist` the spec itself names is also acceptable; `all-resolved` is preferred for brevity and because it does not bake the two-path enumeration into the token.) With an accurate token, drop the parenthetical clauses "so the token is more precisely `all-in-inventory-or-allow-list` — PIC retains the shorter `<symptom>` token to keep the wire-discriminator stable"; retain only the normative scope statement (resolution to inventory or typebox allow-list).

Edge cases the fixer must preserve:
- The *Failure-surface contract* requires the (s1) and (s2) `<symptom>` tokens to be **distinct**; the replacement must not collide with (s1) `no-surface-on-line`.
- `<symptom>` strings are implementation-owned per the *Failure-surface contract*; the spec's (s1)/(s2) names are the canonical illustrative coinage, so the rename is a spec-text edit only and does not change the implementation-ownership rule.
- The per-`<symptom>` negative-test-fixture floor (one fixture per family-(5) sub-kind) is unaffected; only the token spelling changes.

## Consequence

**Severity:** cosmetic

A CI parser routes by the `audit/<class>/<family>/<symptom>` token prefix and never interprets the symptom name's semantics, and the parenthetical states the full normative predicate, so no implementer diverges in behaviour. The cost is a misleading canonical label and a redundant "more precisely X" disclaimer carried in a normative paragraph.
