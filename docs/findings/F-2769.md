## Problem

The `V7b` Ships-when ("`npm test` ... fails on a missing/renamed code") and the `DIAG-3` Tests bullet on both `V7b` and `V7b-T` ("a renamed code fails the gate") promise that the `H5a` closing gate detects a renamed diagnostic code. That gate's diagnostic arm is a point-in-time parity reconciliation holding no prior-version baseline: it reds only on a one-sided rename (registry row renamed while the asserting test still cites the old code, or vice versa) — which is exactly the `DIAG-2` failure already covered. A consistent rename, renamed in both the registry row and every asserting test in one edit, leaves parity satisfied and passes silently. The cross-version-stability property `DIAG-3` exists to protect is therefore never witnessed by this gate; `diagnostic-shape.md#diag-3` itself frames a rename as a loom-2.0-deferred contract change carried by contributor discipline / release review, not a loom 1.0 mechanical check.

## Issue introduction

**Verdict:** present-since-inception

**History:** The "fails on a missing/renamed code" Ships-when clause and the `DIAG-3` "a renamed code fails the gate" Tests bullet were both present at the V7b leaf's first commit (`c6a664e`, the initial plan build). The later `f284534` ("resolve DIAG-2 src/** emission scan vs closing gate") rewrote the same Ships-when line but only changed "emitted codes" to "test-asserted codes"; the "renamed" overclaim predates it. The defect was never introduced by a later edit — it shipped with the leaf's inception.

## Solution approach

Rewrite the `V7b` Ships-when so it no longer claims the gate fails on a renamed code, stating instead the parity outcome the `H5a` gate actually delivers — a one-sided/inconsistent rename reds, a consistent rename is a loom-2.0-deferred contract obligation carried by contributor discipline / release review. Rewrite the `DIAG-3` Tests bullet on both `V7b` and `V7b-T` so the asserting test it names is one the parity gate can red on (an inconsistent rename), and name the consistent-rename stability property as the review-time loom-2.0-deferred obligation. Keep the `DIAG-3` cross-version-stability obligation named distinctly so the bullet does not collapse into a restatement of `DIAG-2`.

## Solution constraints

- Out of scope: the `DIAG-2` reverse-direction (registry-code-with-no-asserting-test) framing, owned by the same-cluster sibling finding.
