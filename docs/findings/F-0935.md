## Finding

`V9b` is named the single declaration site of the test-only `ReloadFailureInjector` interface for the whole watcher-time reload failure-injection seam, and all three arms route through its one `injectReloadFailure` method: the registry-swap arm and the `.loom`/`.warp` re-parse arm (wired in `V9b`), plus the settings-re-merge arm (contributed by `V10d`). `V4g` imports the interface to exercise `ERR-7`. The leaf gives only the method name and surrounding prose ("a caller supplies a synthetic registry-swap or re-parse failure to `ReloadFailureInjector.injectReloadFailure`"); it never states the parameter shape — how a caller selects which arm to inject against, and what synthetic-failure payload it passes.

Because the parameter contract is unstated, the `V9b` implementer invents the signature. The risk is bounded by the single declaration site (`V10d` and `V4g` import the interface from `V9b`, so they bind to whatever shape `V9b` produces), but `V9b` itself wires only two of the three arms; a signature designed for the registry-swap and re-parse arms alone may not cleanly admit `V10d`'s later settings-re-merge arm, forcing rework of the interface and `V9b`'s tests once `V10d` lands. Two reasonable implementers of `V9b` would also produce divergent signatures from the same prose.

## Issue introduction

**Verdict:** present-since-introduction

**History:** `git log -S "ReloadFailureInjector"` over `docs/plan_topics/` first introduces the `ReloadFailureInjector` interface and the `injectReloadFailure` method name in commit `d64dce5`. The introducing diff describes the method only in prose ("whose `injectReloadFailure` method a caller invokes to hand it a synthetic failure"); no parameter shape, arm discriminator, or payload type was ever stated. The gap is therefore present from the construct's first authoring and persists through the subsequent owner-consolidation edits (`fdce2a2`, `f14e929`, `324813d`); none of those added a parameter contract.

## Solution Space

**Shape:** single

### Recommendation

State the `injectReloadFailure` parameter contract in `V9b`'s **Adds** field, alongside the existing sentence that names the interface as the single declaration site. Pin two things:

- The closed arm selector the caller passes to choose which arm to inject against, covering exactly the three arms the seam declares: registry-swap, `.loom`/`.warp` re-parse, and settings-re-merge (the literal arm names `V9b`/`V10d`/`V4g` already use, e.g. `registry-swap` / `loom-warp-reparse` / `settings-remerge`).
- The synthetic-failure value the caller hands in for the selected arm (the value the seam routes onto the `loom-system-note` surfacing path).

Pinning the selector as a closed set over all three arms ensures the shape `V9b` declares already accommodates `V10d`'s settings-re-merge arm and `V4g`'s `ERR-7` injection, so the consumers bind to a stable contract rather than one shaped around only the two arms `V9b` wires. The contract is internal to the plan's test-only interface (no spec page declares it), so the edit is confined to plan files.

## Consequence

**Severity:** correctness

With no pinned parameter shape, the `V9b` implementer invents the `injectReloadFailure` signature from prose, and two implementers diverge. Worse, a signature shaped for the two arms `V9b` wires (registry-swap, `.loom`/`.warp` re-parse) need not accommodate the settings-re-merge arm `V10d` adds later against the same interface, forcing rework of `V9b`/`V9b-T` and `V10d` when the third arm lands.
