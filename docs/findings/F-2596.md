## Problem

`package-and-settings.md` § "Package discovery" specifies a single flat candidate-enumeration rule that applies "within each root": every immediate child directory not beginning with `@` is a candidate package, and every `@`-prefixed child is a scope directory unwrapped one level. That rule reaches packages in the three npm-shaped roots (depth 1, scoped at depth 2) but cannot reach the two git roots, whose listed layout is `.pi/git/<host>/<path>/` and `~/.pi/agent/git/<host>/<path>/` — a multi-segment path where the clone directory holding `package.json` sits two or more levels below the root. Under the flat rule the immediate children of a git root are host directories (`github.com`, …); they do not begin with `@`, so each is mis-classified as a candidate package whose `package.json` parse fails, and the walk never descends to the real clone directory. The page commits to both the multi-segment git layout and a depth-1 enumeration that cannot reach it, with no fallback (the extension walks roots itself and does not delegate to Pi), so git-installed loom packages are silently undiscoverable.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** d6cbb37 — pi-loom spec: resolve "pi.looms is an extension-owned convention" (2026-05-04, Thomas Andersen)
**History:** Commit d6cbb37 authored the package-discovery section, introducing the multi-segment git roots (`.pi/git/<host>/<path>/`) and the flat "every immediate child" enumeration rule in the same change; the two were never reconciled, so git roots were unreachable from the section's first commit. Commit 475155c (2026-05-05) later extended the flat rule with `@scope` unwrapping for scoped npm packages — touching the enumeration rule without closing the git-depth gap — and f7a0dae (2026-06-08) deduplicated two contradictory statements of the rule, again leaving the git case unaddressed.

## Solution approach

Split the candidate-enumeration rule in § "Package discovery" by root kind. Keep the existing flat immediate-child rule and one-level `@scope` unwrap for the npm roots unchanged. For the two git roots, specify a presence-driven descent through the `<host>/<path>` tree that treats the first directory along each path containing a `package.json` as the candidate clone directory, with `@scope` unwrap not applying. Cross-reference `packages.md` § git as the authority for the `<host>/<path>` clone layout.

## Solution constraints

- The DISC-6 walk bounds (`looms.scanPackagesMaxFiles`, `looms.scanPackagesTimeoutMs`) apply unchanged to the git descent — every `package.json` probe and the descent walk time count against them.
