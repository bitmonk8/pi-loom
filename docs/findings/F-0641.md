## Finding

NOCEIL-1 is a citable conformance claim: it asserts that "the absence [of a wall-clock timeout] is enforced at parse time by rejecting any `timeout:` field at every site where a future per-call timeout could land." The four-axis closure list one paragraph down restates the same claim ("the absence is enforced at parse time by `NOCEIL-1`'s `timeout:`-field rejection"). Neither phrasing tells an implementer (or a conformance-fixture author) which parse sites those are, nor names the diagnostic code that is supposed to fire on rejection.

The information exists elsewhere in the corpus. `cancellation.md` names the three sites — "a query, tool call, or invoke" — and the diagnostic code `loom/parse/timeout-field-rejected`. `diagnostics/code-registry-parse.md` carries the registry row for that code with the same site list. The three per-surface "loom 1.0 seam — per-call timeout" blockquotes in `query/query-failure-and-repair.md`, `tool-calls.md`, and `invocation.md` pin each options-surface seam that the future per-call-timeout extension will land in. NOCEIL-1 forward-links to `cancellation.md` for the deferral rationale but to nothing for the rejection contract it purports to enforce.

The consequence is a conformance gap entirely within NOCEIL-1's own framing: an implementer or test author grounding the claim from `ceiling-invariants-and-audit.md` alone must reconstruct the site list from prose ("query / tool-call / invoke" appearing only in the *deferral* sentence), guess that an undisclosed diagnostic code carries it, and find by grep that `loom/parse/timeout-field-rejected` is the intended code. A guessed-incomplete site list (e.g. forgetting that frontmatter-declared `timeout:` on a `tools:` entry counts as the tool-call site, or omitting the invoke options record) lets `timeout:` slip through without firing the seam NOCEIL-1 names as its sole enforcement mechanism.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commits:** `cc240be` (2026-05-07, "pi-loom spec: resolve \"No additional V1 runtime ceiling closing paragraph\"")
**History:** The "every site where a future per-call timeout could land" phrasing entered the corpus in `cc240be`, which split the prior single-paragraph "No additional V1 runtime ceiling applies" claim into the four NOCEIL-N inline labels. The pre-split prose ("V1 imposes no wall-clock timeout per query / tool-call / invoke … enforced at parse time by rejecting any `timeout:` field") at least named the three sites inline; the NOCEIL-1 rewrite dropped that enumeration in favour of the parametric "every site where a future per-call timeout could land" phrase, without picking up the diagnostic code or a cross-reference to `cancellation.md`'s sentence that names it. The diagnostic row itself (`loom/parse/timeout-field-rejected`) had landed three days earlier in `839db71` (2026-05-04, "Diagnostic codes are declared as namespaces but never assigned to specific errors"); NOCEIL-1's introduction inherited the existing code but did not cite it. The later move from `spec.md` into `spec_topics/hard-ceilings.md` (`1c7f550`, 2026-05-07) and the subsequent split into `ceiling-invariants-and-audit.md` (`81bf52a`) preserved the under-specified phrasing unchanged. The defect is therefore traceable to a single authoring decision in `cc240be`: trading the inline three-site enumeration for a more abstract closure phrasing without picking up a forward link to the owner sites or the diagnostic code.

## Solution Space

**Shape:** single

### Recommendation

Rewrite NOCEIL-1's enforcement clause to name both the site set and the diagnostic code, with cross-references to their owner pages, in two places (the NOCEIL-1 bullet itself and the `(i)` time-axis entry in the four-axis closure list — they share the rewrite by GOV-30 lock-step):

> Per-call timeouts are deferred per [Cancellation](../cancellation.md) and [Future Considerations](../future-considerations.md). The absence is enforced at parse time by rejecting any `timeout:` field at each of the three sites where a future per-call timeout would land — the `@`-template query options surface ([Query — Options surface](../query/query-failure-and-repair.md#options-surface)), the code-side tool-call options surface ([Tool Calls](../tool-calls.md)), and the invoke options surface ([Invocation — INV-3](../invocation.md#inv-3)) — with the rejection surfacing as `loom/parse/timeout-field-rejected` per [Diagnostics — Parse codes](../diagnostics/code-registry-parse.md) and [Cancellation](../cancellation.md).

The three options-surface seams (Query / Tool Calls / Invocation) are already pinned as the loom 1.0 closed enumeration of the locations a `timeout:` field could attach to — they are the same three surfaces the matching deferred extension in `future-considerations/surface-extensions.md` enumerates as its anchor sites — so NOCEIL-1 inherits the closure from those seams rather than minting a new one. Frontmatter is *not* a fourth site: the only `timeout:` field NOCEIL-1 is concerned with is the per-call options-record key, not a hypothetical frontmatter knob, and the `V6a` test obligation that currently lives in the frontmatter leaf is exercising the frontmatter-parser's rejection of an unknown options-shaped key, which is downstream of the same `loom/parse/timeout-field-rejected` registry row.

Edge cases the implementer must watch:

- The diagnostic-code citation must point at the registry row's owner, not at `cancellation.md`'s mention of it; `cancellation.md` already redirects readers to `diagnostics.md` for the code.
- The four-axis closure list's `(i)` entry currently says "enforced at parse time by `NOCEIL-1`'s `timeout:`-field rejection" — that phrasing self-refers and must be updated in lock-step (per the methodology paragraph's GOV-30 co-edit obligation) so the two NOCEIL-1 surfaces do not drift.
- The three site cross-references must be checked against the anchors actually published on each owner page; `query/query-failure-and-repair.md` carries the `## Options surface` heading verbatim, `tool-calls.md` and `invocation.md` use blockquoted "loom 1.0 seam — per-call timeout" callouts inside their bodies (use `#inv-3` for the invocation site since it is the only one of the three that already has a stable anchor).

## Consequence

**Severity:** advisory

The defect is local to NOCEIL-1's framing: a careful implementer can grep the corpus and reconstruct the site set and diagnostic code from `cancellation.md` plus the three options-surface blockquotes. The risk is the silent variant — a conformance-fixture author writes a NOCEIL-1 test that exercises only one or two sites, the suite passes, and a real `timeout:` declaration on the unguarded site reaches the runtime intact. The four-axis Audit methodology presents NOCEIL-1 as a self-contained, citable closure claim; that role requires it to carry its own enforcement mechanism on its face.
