## Problem

NOCEIL-1 in `ceiling-invariants-and-audit.md` is a citable conformance claim: the absence of a wall-clock timeout "is enforced at parse time by rejecting any `timeout:` field at every site where a future per-call timeout could land." Neither NOCEIL-1 nor its matching `(i)` time-axis entry in the four-axis closure list names which parse sites those are or the diagnostic code that fires on rejection. The site set ("a query, tool call, or invoke") and the code `loom/parse/timeout-field-rejected` exist in `cancellation.md` and `diagnostics/code-registry-parse.md`, but NOCEIL-1 forward-links to neither for the rejection contract it claims to enforce. A conformance-fixture author grounding the claim from this page alone must reconstruct the site list from prose and guess the code, and an incomplete reconstruction lets a `timeout:` field slip through the seam NOCEIL-1 names as its sole enforcement mechanism.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commits:** `cc240be` (2026-05-07, "pi-loom spec: resolve \"No additional V1 runtime ceiling closing paragraph\"")
**History:** The "every site where a future per-call timeout could land" phrasing entered the corpus in `cc240be`, which split the prior single-paragraph "No additional V1 runtime ceiling applies" claim into the four NOCEIL-N inline labels. The pre-split prose ("V1 imposes no wall-clock timeout per query / tool-call / invoke … enforced at parse time by rejecting any `timeout:` field") at least named the three sites inline; the NOCEIL-1 rewrite dropped that enumeration in favour of the parametric "every site where a future per-call timeout could land" phrase, without picking up the diagnostic code or a cross-reference to `cancellation.md`'s sentence that names it. The diagnostic row itself (`loom/parse/timeout-field-rejected`) had landed three days earlier in `839db71` (2026-05-04, "Diagnostic codes are declared as namespaces but never assigned to specific errors"); NOCEIL-1's introduction inherited the existing code but did not cite it. The later move from `spec.md` into `spec_topics/hard-ceilings.md` (`1c7f550`, 2026-05-07) and the subsequent split into `ceiling-invariants-and-audit.md` (`81bf52a`) preserved the under-specified phrasing unchanged. The defect is therefore traceable to a single authoring decision in `cc240be`: trading the inline three-site enumeration for a more abstract closure phrasing without picking up a forward link to the owner sites or the diagnostic code.

## Solution approach

Rewrite NOCEIL-1's enforcement clause to name the three options-surface rejection sites with forward-links to their owner pages — the query options surface ([Query — Options surface](../query/query-failure-and-repair.md#options-surface)), the tool-call options surface ([Tool Calls](../tool-calls.md)), and the invoke options surface ([Invocation — INV-3](../invocation.md#inv-3)) — and name the diagnostic code `loom/parse/timeout-field-rejected` with a forward-link to its registry row ([Diagnostics — Parse codes](../diagnostics/code-registry-parse.md)). Apply the same rewrite to the `(i)` time-axis entry of the four-axis closure list in lock-step per the methodology paragraph's GOV-30 co-edit obligation, so the two NOCEIL-1 surfaces do not drift.
