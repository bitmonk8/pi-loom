## Problem

`H5a`'s Adds defines a warn-only live-corpus mode that "emits each gap as a CI finding without failing the build," and `H5b`'s Tests assert the canary "reports" each seeded gap while the run stays green. Neither leaf states what a "CI finding" concretely is — a returned findings collection, captured stdout, or a structured log artifact — yet each representation demands fundamentally different `H5b` verification code (in-process return assertion vs. output capture vs. filesystem read). Two conforming `H5a` implementations could each satisfy the prose while forcing mutually incompatible `H5b` assertions, so the `H5b` author cannot write the seeded-gap checks without first inventing the contract `H5a` left unspecified.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commit:** `ea6b1da` — "pi-loom plan: resolve \"Live-corpus gate activation has no documented rollback\"" (2026-06-11)
**History:** The warn-only live-corpus mode and the "CI finding" term entered both `H5a` (Adds) and `H5b` (Tests) in the single commit `ea6b1da`, which added the warn-only pre-flight while resolving the gate-activation rollback finding. `git log -G "CI finding"` on each leaf file bottoms out at `ea6b1da` with no earlier occurrence. The reporting contract was undefined from the moment the concept was authored, and no later commit narrowed "CI finding" to a concrete representation. The emission side (`H5a`) and the verification side (`H5b`) co-entered in this one commit, so the defect is not a multi-commit interaction.

## Solution approach

In `H5a`'s Adds, where the warn-only mode "emits each gap as a CI finding," specify a single concrete, observable reporting representation for the warn-only output. Then rewrite `H5b`'s Tests assertion — replacing the undefined "reports it" — and `H5b`'s Ships when so they observe that same representation per seeded gap kind while the run's exit status stays green.

## Solution constraints

- The warn-only reporting/non-failing path must stay distinct from `H6a`'s hard-fail footing, whose signal over the identical surfaces remains a non-zero `npm test` exit status.
