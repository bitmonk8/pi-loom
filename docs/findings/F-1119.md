## Finding

Three unrelated PIC sections each pin an implementation mechanism where only a behavioural requirement is needed; the page text in each case acknowledges (explicitly or by tone) that the mechanism is internal, yet still bolts a normative literal to it.

1. **`ActiveInvocationRegistry` entry shape** (`active-invocation-registry.md`). The opening sentence types the registry as `Set<{ loomAbort: AbortController; disposeBarrier: Promise<void>; shutdownReason: string | undefined; loom: string; invocationId: string }>`, and the *Registry contract* further pins "the pinned five-field entry shape is unchanged … so observers of the entry (tests, audit fixtures) see exactly the five members the entry literal pins." The same section also pins iteration order to "the V8 `Set` insertion-order invariant" and labels the registry name "internal — not part of the loom-facing or extension-facing contract; tests assert on observable side effects." The five-member pin and the `Set` engine reference contradict that explicit internality: an audit fixture asserting "exactly five members" is asserting on the implementation symbol, not on a behavioural side effect.

2. **`loom-direct:` `toolCallId` prefix** (`host-interfaces-core.md` §*Tool execution from loom code*). The first bullet states "`toolCallId` is a synthesised UUID prefixed `loom-direct:`." No consumer of the prefix is named anywhere in the corpus — Pi's `execute(...)` does not branch on the leading bytes of `toolCallId`, no loom-side diagnostic templates render it, and no audit fixture grep-asserts the literal. The behavioural requirement at this site is exhausted by "a synthesised id that is unique across the call site's lifetime"; the prefix literal is decoration.

3. **`250 ms` debounce literal** (`registration-steps.md` step 5, restated in `host-interfaces-services.md` `Clock` and in `session-shutdown-semantics.md` *Factory-ordering pin* prose). The spec already names the behavioural requirement: "coalesce editor `change` + `rename` bursts" through the injected `Clock` seam, and explicitly notes the seam "exists to make burst-coalescing assertions deterministic and is **not** a tunable runtime knob." Pinning the specific value `250 ms` as a normative MUST adds nothing to that contract — burst coalescing through `FakeClock` is asserted with whatever window the runtime chose, and any later editor study that motivates retuning would have to thread a spec edit it doesn't need to.

Each item is a separate edit on a separate page with no shared anchor or shared text run, so the three are addressed as three independent obligations rather than one bundle.

## Solution Space

**Shape:** multiple

The three obligations are decomposed into one option per file so the orchestrator can land them as three small, separately-loopable fixes. The recommended resolution order minimises diff surface for the next loop pass: start with the smallest single-bullet edit (Option B), then the contained-section edit (Option C), then the larger cross-page sweep (Option A).

### Option A — Demote the `ActiveInvocationRegistry` shape pins (in `active-invocation-registry.md`)

**Approach.** Reframe the registry as a behavioural contract instead of a typed-literal pin. Replace the opening `Set<{ ... }>` type with prose: "an extension-instance-scoped, insertion-ordered collection of per-invocation records, one per in-flight loom invocation, each carrying `loomAbort`, `disposeBarrier`, `shutdownReason`, `loom`, and `invocationId`." Strike the *Registry contract*'s "observers of the entry (tests, audit fixtures) see exactly the five members the entry literal pins" sentence and the parenthetical "the V8 `Set` insertion-order invariant" — keep the substantive "insertion order is observable to the `loom/runtime/reload-teardown-timeout` `<list>` rendering" claim, which is the actual observable.

**Spec edits.**
- Opening paragraph: replace the `Set<{...}>` type expression with the named-field-list prose above.
- *Registry contract* `disposeBarrier resolver storage` bullet: keep the resolver-is-closure-scoped rule; drop the "pinned five-field entry shape" cardinality assertion.
- *Registry contract* `Iteration order` bullet: state "insertion order, observable on the `loom/runtime/reload-teardown-timeout` `<list>` rendering"; drop the "V8 `Set` insertion-order invariant" cross-engine appeal.

**Pros.** Aligns the page with its own "registry name is internal" stance; removes a fixture-cardinality assertion that pins implementation surface; preserves every behavioural observable (insertion order, per-entry fields named individually).

**Cons.** Loses the at-a-glance `Set<{...}>` type signature that some readers find clarifying; downstream pages (`drain-state-contract.md`, `session-shutdown-semantics.md`) that reference the `Set` shape need their cross-references tightened to "the per-invocation record" or similar.

**Risks.** A future audit fixture written against the demoted text might over-relax and accept a sixth field (e.g. a debug-only `createdAt`). Mitigate by retaining the per-field list and an explicit "no further fields are observable to spec-compliant code" remark — this keeps the closed-shape obligation while shedding the cardinality literal.

### Option B — Strike the `loom-direct:` `toolCallId` prefix pin (in `host-interfaces-core.md`)

**Approach.** Rewrite the `toolCallId` bullet under §*Tool execution from loom code* to require a "synthesised, per-call unique id" without dictating the lexical form.

**Spec edits.** Replace the bullet "`toolCallId` is a synthesised UUID prefixed `loom-direct:`." with "`toolCallId` is a per-call synthesised id, unique across the call site's lifetime; loom 1.0 reads it back from no normative surface, so the lexical form is not load-bearing."

**Pros.** Smallest possible diff (one bullet); removes a literal with no enumerated consumer; aligns with the existing `IdSource` seam that already mints UUIDs for `invocationId`, suggesting the production implementation will reuse that seam here too.

**Cons.** Loses a piece of operator-side log-grep affordance (an operator inspecting Pi traces could currently scan for `loom-direct:` to find loom-originated tool calls). If that affordance is in fact wanted, the right home is a normative observable in `runtime-event-channel.md` or `diagnostic-shape.md`, not a buried bullet on a host-surface page.

**Risks.** None at the implementation level. The only second-order risk is that the prefix is silently relied on by some out-of-tree tooling; loom 1.0 is pre-release, so this is acceptable.

### Option C — Demote the `250 ms` debounce literal to non-normative (in `registration-steps.md` + `host-interfaces-services.md`)

**Approach.** State the behavioural requirement ("the watcher coalesces bursts of events on the same path through the injected `Clock` seam, so that an editor `change` + `rename` sequence within a short window produces one downstream re-parse and one downstream notification") and move the `250 ms` figure to a non-normative parenthetical naming it as the reference-implementation default.

**Spec edits.**
- `registration-steps.md` step 5: rewrite the parenthetical to "(the watcher coalesces bursts through the injected `Clock` seam — reference-implementation window: `250 ms`, not a normative pin; the seam exists to make burst-coalescing assertions deterministic and is not a tunable runtime knob)".
- `host-interfaces-services.md` `Clock.setTimeout`/`clearTimeout` bullet: replace "The watcher's 250 ms debounce (step 5 above) and the settings-watcher debounce …" with "The watcher debounce and the settings-watcher debounce both schedule through this seam (window value per the reference-implementation parenthetical in `registration-steps.md` step 5); the debouncer holds the most recent timer handle and clears it on each new event."
- `session-shutdown-semantics.md` *Factory-ordering pin* cross-reference: change "the chokidar file watcher, the 250 ms debounce, …" to "the chokidar file watcher, the watcher debounce, …".
- `patch-skew-degradation.md` sub-step 4 cross-reference: change "any pending debounce timer handles" — already neutral, no edit needed.

**Pros.** Removes a literal MUST that hard-codes a UI-latency tuning decision; preserves the observable (coalescing-through-`Clock`) the plan leaves actually need; the reference-implementation parenthetical leaves a single source of truth for the test constant in `V10d`.

**Cons.** Slight expansion of two paragraphs to explain the demotion; `V10d`'s test text already uses `250 ms` and will either keep the literal as a test constant or read it from a shared production-wiring constant. Either is a small follow-up.

**Risks.** A reader who skims only the bullet might not notice the window is non-normative; mitigate by labelling the parenthetical "reference-implementation window" explicitly so the demotion is unambiguous.

### Recommendation

Apply all three options. Resolve in the order **B → C → A**:

1. **B** is a single-bullet edit on a single page with no cross-references depending on the prefix — it shrinks the diff baseline that the next pass's lenses will critique.
2. **C** is a same-section sweep across two pages plus two read-only cross-reference fixups; it leaves the `Clock` seam contract intact and only demotes the literal.
3. **A** is the largest edit (opening paragraph rewrite plus two *Registry contract* bullets) and benefits from landing on a baseline that B and C have already cleaned, so the cross-references to `Set`-shape framing on neighbouring pages don't churn twice.

Edge cases the implementer must watch:

- For **A**, do *not* drop the named-field list — the closed-shape obligation ("no further fields are observable to spec-compliant code") is load-bearing for `V9e` and the `loom/runtime/cancelled-by-session-shutdown` emission rule. The over-prescription is the cardinality literal ("exactly five members") and the `Set` engine reference, not the field set.
- For **C**, ensure `V10d`'s **Ships when** clause is read after the edit lands — if it currently asserts the `250 ms` literal as the *spec window*, it must be reworded to "the reference-implementation `250 ms` window" or read the value from a production-wiring constant. Otherwise the leaf's text drifts from the spec's text and the next plan-review pass flags the inconsistency.
- For **B**, sweep any downstream pages that *might* mention the `loom-direct:` token — the grep at the time of authoring shows only the one bullet, but verify before merging.

## Consequence

**Severity:** advisory

No two implementers would build observably different runtimes from the present text — the over-prescriptions are mostly inert. The cost is to spec hygiene and to the next round of audit-rule design: a `spec-lens-prescription` lens correctly flags these as MUST-on-mechanism, and leaving them in place trains future authors to keep pinning internal symbols.
