## Finding

The **Per-step isolation** paragraph in `session-shutdown-semantics.md` opens with: "Sub-steps 1, 4, and 5 each run inside their own `try`/`catch`; a throw from any of them MUST be caught, MUST emit exactly one `loom/host/session-shutdown-teardown-step-failed` … and MUST NOT prevent the remaining sub-steps from running." Read on its own, this licenses a single `try`/`catch` per sub-step: one handler wrapping the whole of sub-step 1, one wrapping sub-step 4, one wrapping sub-step 5.

That per-sub-step framing contradicts the per-call granularity the rest of the same paragraph pins. Three sentences later: "Within sub-step 1 the rule applies independently to each of `LoomRegistry.drain` … and `LoomRegistry.initDrainStateTag` … (two potential emissions, one per failing call)"; sub-step 4 likewise carries "three potential emissions, one per failing call"; and the closed `details.call` label set enumerates each call site individually. The intended contract is per-*call* isolation, not per-*sub-step*.

The two readings diverge observably. Under the per-sub-step reading, a single `try` wrapping both sub-step-1 acts means a throw from `LoomRegistry.drain` skips `LoomRegistry.initDrainStateTag` entirely (control jumps to the shared `catch`) and emits one combined `teardown-step-failed` row; under the intended per-call reading, both acts run independently and each emits its own row keyed on a distinct `details.call`. The defect is confined to the lead sentence: the normative content is complete and correct once the full paragraph is read, but the topic sentence — which a reader reasonably treats as the rule statement — states the wrong granularity.

The f5e89f4 size-cap split and the c3de47a F-1602 relocation carried the text verbatim into `session-shutdown-semantics.md` without altering the wording.

## Solution Space

**Shape:** single

### Recommendation

Rephrase the opening sentence of the **Per-step isolation** paragraph so it states per-call granularity directly, matching the rule the rest of the paragraph already pins. For example: "Within sub-steps 1, 4, and 5, each individual call site runs inside its own `try`/`catch`; a throw from any one call MUST be caught, MUST emit exactly one `loom/host/session-shutdown-teardown-step-failed` … and MUST NOT prevent the remaining calls or sub-steps from running." This removes the "each sub-step … its own try/catch" phrasing that licenses one combined handler per sub-step. The subsequent per-sub-step clarifications and the closed `details.call` set then read as enumerations of the same per-call rule rather than as corrections to a contradictory lead. No normative content changes; the edit is a wording tightening confined to the one sentence.

## Consequence

**Severity:** advisory

An implementer who treats the lead sentence as the normative rule and the elaboration as commentary could wrap each sub-step in a single `try`/`catch`, dropping the second sub-step-1 act when the first throws and emitting one combined `teardown-step-failed` row instead of one per failing call. The full paragraph and the implementing plan leaf both pin the per-call contract, so a careful reader still produces the correct system; the defect is a misleading topic sentence that invites a skim-level error rather than a genuine gap.
