## Finding

`V12b` (Top-level `Err` formatting and chain attribution) carries a cross-slice forward dependency on `V15g` (the invoke invocation-record provenance seam): its `SLSH-5` chain-attribution suffix renders `from <callee> invoked at <parent>:<line>` from `V15g`'s per-frame invocation record, and `V15g` is correctly listed in `V12b`'s `Deps.` (`V12b-T`, `V12a`, `V4d`, `V15g`). `V15g` lives in the V15 slice ‚Äî three slices ahead of V12 ‚Äî so `V12b` cannot be built until that V15-slice leaf has shipped.

The V12 slice interleave note (`docs/plan.md`, the `> **Interleave note.**` blockquote under `### V12 ‚Äî Slash invocation`) advertises only the `V12a`/`V12a-T ‚Üí V13c` edge ("the V12 slice lands after that V13 leaf"). It does not mention the `V12b ‚Üí V15g` forward dependency, even though `V15g` lands later than `V13c` and is therefore the true gating leaf for completing the V12 slice. The note is the implementer's summary of the slice's out-of-order build points; omitting the largest forward cross-slice edge under-states where the V12 slice actually lands.

This is a narrative-completeness gap, not a DAG defect: the `Deps. ‚Ä¶ V15g` edge binds correctly and the note's own "Sequence by **Deps**, not slice number" closing sentence keeps a Deps-following implementer correct. The omission is inconsistent with the sibling V14 and V15 notes, which do call out their cross-slice deps (`V14a ‚Üí V15a`, `V15b ‚Üí V16a`). The leaf body of `V12b` is unchanged by the fix; only the slice narrative in `plan.md` is incomplete.

**History:** The V12 interleave note did not exist until `15c2dcf` added it as part of the F-0584 missing-interleave-notes fix; that commit captured only the `V12a ‚Üí V13c` edge. The `V12b ‚Üí V15`-slice provenance dependency already existed at that point ‚Äî earlier commit `5f4d271` ("resolve 'V15a bundles ~14 independent test obligations into one impl+T pair'", 2026-06-17) had split `V15g` out of `V15a` and retargeted `V12b`'s `SLSH-5` provenance consumption and `Deps` from `V15a` to `V15g`; before that split, `V12b` depended on `V15a` for the same record. The cross-slice forward dep therefore predates the note, and the note was incomplete from the moment it was introduced ‚Äî the `V12b ‚Üí V15g` edge was never recorded.

## Solution Space

**Shape:** single

### Recommendation

Extend the V12 slice interleave note in `docs/plan.md` (the `> **Interleave note.**` blockquote directly under `### V12 ‚Äî Slash invocation`) so it also records `V12b`'s cross-slice forward dependency on `V15g`. The note currently states only that `V12a`/`V12a-T` depend on `V13c`; add that `V12b` consumes `V15g`'s per-frame invocation record (the invoke invocation-record provenance seam) for its `SLSH-5` chain-attribution suffix, so `V12b` lands after that V15-slice leaf. Use the same phrasing the sibling notes already use ‚Äî the V14 note ("The V14 series depends on `V15a` ‚Ä¶ so the V14 slice lands after that V15 leaf") and the V15 note (`V15b ‚Üí V16a`). Keep the existing "Sequence by **Deps**, not slice number" closing sentence intact.

No leaf file (`V12b`, `V15g`) and no spec file changes: the edge already binds via `V12b`'s `Deps. ‚Ä¶ V15g`; only the slice narrative is incomplete.

## Consequence

**Severity:** advisory

An implementer relying on the V12 interleave note as the slice's sequencing summary would learn only that V12 lands after `V13c`, not that `V12b` is additionally blocked on `V15g` (three slices ahead) ‚Äî under-stating the slice's true landing point. Nothing breaks at build time: the `Deps. ‚Ä¶ V15g` edge binds and the note's "Sequence by Deps" disclaimer keeps a Deps-following reader correct, so the impact is a guidance gap and a consistency mismatch with the V14/V15 notes that do surface their cross-slice deps.
