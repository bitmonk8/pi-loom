## Finding

The **Granularity** paragraph in `docs/spec_topics/cancellation.md` prefaces its
five-bullet checkpoint list with "The interpreter checks the cancellation
signal at exactly these points and no others", then includes the binder-call
bullet — "immediately before issuing the slash-command argument binder's LLM
call (and the signal is forwarded to the binder model's provider invocation,
so an abort observed *during* the binder call also surfaces)." But the binder
runs at **slash-dispatch / load time**, before the loom body is interpreted at
all; it is not an interpreter-loop site. The companion page
`docs/spec_topics/binder/determinism-cancellation-failure.md` §Cancellation
already attributes the same check to the runtime ("The runtime checks
`ctx.signal` immediately before issuing the binder call …"), and the
`Checkpoint` seam contract at
`docs/spec_topics/pi-integration-contract/host-interfaces-services.md#pic-10`
explicitly calls this out — "the binder checkpoint fires even though the
binder runs outside the loom body".

The four pre-existing bullets (loop iteration, `@`-query, tool call, `invoke`)
are genuinely interpreter sites; the umbrella sentence fit them. The
binder-call bullet was appended later under the same sentence without
revising the actor, leaving cancellation.md the only page that calls the
binder check an interpreter check. The result is a single-sentence
contradiction with two adjacent pages a reader is expected to cross-read,
and an inconsistent reading of "the interpreter" within cancellation.md
itself (every other reference to the interpreter loop excludes the binder).

## Issue introduction

**Verdict:** single-commit

**Introducing commit:** `7cb4cc3` *("pi-loom spec: resolve 'Binder LLM call has no cancellation checkpoint'")*

**History:** The umbrella sentence + four-bullet list ("for/while iteration,
`@`-query, tool call, `invoke`") was introduced by commit `e090161`
(*"resolve 'Cancellation granularity contradicts its own checkpoint list'"*).
At that point all four enumerated sites were genuinely interpreter-loop
sites, so "The interpreter checks the cancellation signal at exactly these
points and no others" was correct. Commit `7cb4cc3` then appended the
fifth bullet — the binder-call checkpoint — to the same list without
revising the umbrella attribution, even though the same commit added the
**Cancellation** sub-section to `binder.md` (now `determinism-cancellation-failure.md`)
using "The runtime checks `ctx.signal` immediately before issuing the
binder call …". The two pages were edited in the same commit but with
different actor words, and the inconsistency has persisted since.

## Solution Space

**Shape:** single

### Recommendation

Split the binder-call bullet out of the interpreter list and attribute it
to the runtime explicitly. Concretely, edit
`docs/spec_topics/cancellation.md` §Granularity as follows:

1. Drop the binder-call bullet from the existing list so the umbrella
   sentence ("The interpreter checks the cancellation signal at exactly
   these points and no others") again covers exactly the four
   interpreter-loop sites (loop iteration, `@`-query, tool call,
   `invoke`).
2. Immediately after the list (before the *loop-iteration checkpoint
   releases the event loop* paragraph), add a one-sentence runtime
   attribution along the lines of: "Additionally, the **runtime** checks
   the cancellation signal immediately before issuing the slash-command
   argument binder's LLM call, and forwards the signal to the binder
   model's provider invocation so an abort observed *during* the binder
   call also surfaces; this checkpoint fires at slash-dispatch time,
   outside the loom body (see [Slash-Command Argument Binding —
   Cancellation](./binder/determinism-cancellation-failure.md#cancellation)
   and the [`Checkpoint` seam](./pi-integration-contract/host-interfaces-services.md#checkpoint-seam))."
3. Verify that the count "five sites" used in `V17c` and in
   `host-interfaces-services.md` §PIC-10 ("each `for`/`while` iteration,
   each `@`-query, each tool call, each `invoke`, and the binder LLM
   call") is preserved — the set is unchanged, only the actor attribution
   splits. The umbrella's "and no others" exhaustiveness claim must now
   read as covering the union of the interpreter four plus the runtime
   one; the simplest phrasing is to keep "and no others" on the
   interpreter list (it is exhaustive *over interpreter-loop sites*) and
   make the runtime sentence the sole runtime-side checkpoint by
   construction.
4. Update `plan_topics/V17c-checkpoint-granularity.md` **Adds** field to
   mirror the new split actor wording so the leaf stays in lockstep.

Implementer-relevant edge cases: the `Checkpoint` seam (PIC-10) treats the
binder-call checkpoint identically to the other four (a `CheckpointKind`
literal `"binder-call"` with the same `before(...)` semantics), so the
actor-split is purely a prose-attribution change — no `CheckpointKind`
revision, no new seam, no change to the five-element checkpoint set
tested by `V17c` or the binder-call cancellation test owned by `V11j`.

## Consequence

**Severity:** cosmetic

A first-time reader of cancellation.md walks away believing the
binder-call check is issued inside the interpreter loop, then encounters
the binder topic page and the PIC-10 seam saying the opposite and has to
re-derive which is authoritative. Implementation is not at risk (the
runtime-side wiring is fully pinned by PIC-10 and the binder topic page),
but the umbrella sentence is the kind of "exactly these points and no
others" prose readers and downstream documents quote verbatim, so the
defect propagates if not fixed at source.
