## Problem

The §5 `<key>` sub-rule in `placeholder-rendering-b.md` defines its quoting predicate as "not identifier-shaped per [Lexical — Identifiers] (i.e. would not match `[A-Za-z_][A-Za-z0-9_]*`)". The Lexical — Identifiers definition is more than that bare character class: it layers a parse-time first-letter case constraint and a reserved-keyword exclusion on top, so a reader who imports the full Lexical semantics would quote reserved-keyword-valued keys (rendering `"if"` instead of `if`). That contradicts the Edge cases "Category 5 identifier predicate" note, which states reserved-keyword-valued keys are still identifier-shaped and render bare. The cross-reference is imprecise: the operative predicate is the bare runtime regex applied to the key string value, with neither the parse-time case constraint nor the reserved-keyword exclusion in force.

## Issue introduction

**Verdict:** present-since-inception (of the rule)
**Introducing commit:** `61ea1a5` (2026-05-06, *pi-loom spec: resolve "Diagnostic message placeholders have no defined rendering"*)
**History:** The `<key>` bullet was authored verbatim in `61ea1a5` as part of the initial introduction of §5 ("Source-derived placeholders") into `spec_topics/diagnostics.md`, with the misleading "(i.e. would not match `[A-Za-z_][A-Za-z0-9_]*`)" gloss in place from the first version. The same commit added the disambiguating Category 5 Edge Cases bullet, so the internal tension has been present since the rule existed. The 100KB split (`f5e89f4`, 2026-06-04, *docs: enforce 100KB hard / 30KB recommended size cap on spec set*) relocated the §5 block from `diagnostics.md` into `diagnostics/placeholder-rendering-b.md` byte-for-byte without amending it. Subsequent commits to `placeholder-rendering-b.md` (`72e2c31`, `bf85092`, `d3ef0cf`, `89eabf1`, `62d49d6`, …) extended §7 and §6 but never revisited the §5 `<key>` wording.

## Solution approach

Rewrite the §5 `<key>` sub-rule so the operative quoting predicate is named inline as the runtime regex `^[A-Za-z_][A-Za-z0-9_]*$` applied to the key string value, rather than carried by the "identifier-shaped per Lexical — Identifiers" cross-reference. Clarify that the predicate is purely syntactic over the runtime string — case-blind and keyword-blind — so a reserved-keyword-valued key renders bare. Add a forward-link from the §5 `<key>` sub-rule to the Edge cases "Category 5 identifier predicate" note, keeping that note as the single source of truth for the reserved-keyword case.
