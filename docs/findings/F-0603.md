## Problem

The `strict-capability-absence-pin` sub-paragraph in
`audit-recognised-shapes.md` uses the phrase "the probed name" as if it were
a defined term or a cross-reference, but neither the page nor the cited
step 2(a) target defines, names, or anchors it. The operative clause
"exposes a `strictCapable` member under the probed name" is also
self-contradictory read literally, since a `strictCapable` member
necessarily has the name `strictCapable`. Because this absence gate and the
rename-detection arm at `version-bump-triggers.md` step 7 are meant to read
the same constant, leaving the binding's source unanchored invites two
implementers to diverge on where the literal actually lives.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commits:** `6829a79` (2026-06-06) — *pi-loom spec: resolve "Pi-SDK hidden-assumption cluster (strictCapable absence, decode steps, pi.on origin)"*
**History:** The `strict-capability-absence-pin` sub-paragraph that contains the "under the probed name" phrasing was added to `audit-recognised-shapes.md` as a single one-line insertion by `6829a79`; `git log -S "the probed name" -- docs/spec_topics/pi-integration-contract/audit-recognised-shapes.md` returns only that commit, and `git log -G "MUST additionally fail" -- docs/spec_topics/pi-integration-contract/audit-recognised-shapes.md` confirms the assertion sentence has had no subsequent edits. The defect entered with the sub-paragraph itself: the same commit pins the absence claim, names the citation site for it, and references `strict-capability-probe`'s literal-read assertion — but introduces the bare phrase "the probed name" without an inline definition or a cross-reference anchor.

## Solution approach

Rewrite the `strict-capability-absence-pin` operative clause to anchor "the
probed name" to the `probedName` payload field on the
`strict-capability-probe` row of `SDK_SURFACE_INVENTORY`, dropping the inner
`strictCapable` occurrence so the gate reads as a member whose name equals
that payload field. Clarity grounding can name the loom 1.0 literal inline so
a reader of `audit-recognised-shapes.md` alone need not chase the
cross-reference to recover the behaviour.

## Solution constraints

- The absence gate here and the `version-bump-triggers.md` step 7
  rename-detection arm MUST consume the same `probedName` payload, not two
  independently-pinned copies of the literal.
- The cross-reference targets the existing implementation-owned `probedName`
  payload field, not a new spec-pinned constant — the "implementation owns
  the literal kind names and per-kind payload field shapes" delegation is
  preserved.
