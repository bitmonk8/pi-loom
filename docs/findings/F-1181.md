## Finding

`docs/spec_topics/invocation.md` line 85 (the INV-4 countable-frames paragraph) renders the depth-overflow diagnostic string with a parenthetical "(matching [Diagnostics — code registry](./diagnostics/code-registry-parse.md#code-registry))". The cited file is the `loom/parse/*` shard — it does not carry the `loom/runtime/invoke-depth-exceeded` row whose *Message template* the parenthetical claims to match. That row lives on `docs/spec_topics/diagnostics/code-registry-runtime.md` (the `loom/runtime/*` shard), under the section heading `### loom/runtime/* — runtime panics, runtime-defect surface, and delivery failures`.

The parallel cite in `docs/spec_topics/errors-and-results/error-model.md:121` does point at the runtime shard with the correct deep anchor (`../diagnostics/code-registry-runtime.md#loomruntime-runtime-panics-runtime-defect-surface-and-delivery-failures`), so the two cross-references for the same template — INV-4 in invocation.md and the *Panic message string (normative)* paragraph in error-model.md — disagree about where the registered template lives. The `#code-registry` fragment on the parse shard does resolve (that page's `## Code registry` umbrella heading is the four-shard preamble), which makes the link silently navigate to a real page that does not contain the row, rather than producing a visibly broken anchor.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commit:** `f5e89f4` — *docs: enforce 100KB hard / 30KB recommended size cap on spec set* (2026-06-04).
**History:** Before this commit, INV-4 cited `./diagnostics.md#code-registry`, which targeted the single pre-split `diagnostics.md` page that held all four phase tables under one anchor (`git log -L` shows the citation stable from its introduction in `2b0def6` "invoke-chain depth-32 cap…" through `1b540e9` and `bf9d0a8`). The size-cap commit split `diagnostics.md` into four sibling shards (`code-registry-{parse,load,runtime,host}.md`). In the same commit, `errors-and-results/error-model.md` was newly authored with the correct cite to `code-registry-runtime.md`, while INV-4's pre-existing link was retargeted to `code-registry-parse.md` instead of `code-registry-runtime.md`. The two callsites have disagreed since that single commit.

## Solution Space

**Shape:** single

### Recommendation

Retarget the INV-4 parenthetical link to the runtime shard with the same deep anchor the error-model.md cite already uses, so both callsites for the `loom/runtime/invoke-depth-exceeded` template point at the row that owns it.

Concretely, in `docs/spec_topics/invocation.md` line 85, replace

```
(matching [Diagnostics — code registry](./diagnostics/code-registry-parse.md#code-registry))
```

with

```
(matching [Diagnostics — code registry](./diagnostics/code-registry-runtime.md#loomruntime-runtime-panics-runtime-defect-surface-and-delivery-failures))
```

Edge cases for the implementer:

- The anchor slug for the runtime-shard section is the GitHub-Markdown-derived slug `loomruntime-runtime-panics-runtime-defect-surface-and-delivery-failures` (the leading backticks and slash in `` `loom/runtime/*` `` collapse out). Use the exact slug already in production at `error-model.md:121`; do not invent a new one.
- Do not also change the visible link text — "Diagnostics — code registry" remains accurate for the runtime shard and keeps the two callsites' prose phrasing aligned.
- No other invocation.md citation needs touching; the only other registry mention in the file (under the cycle-detection paragraph) cites `implementation-notes.md`, not the registry.

## Consequence

**Severity:** cosmetic

A reader following the INV-4 parenthetical lands on the parse-shard table and does not find the row they were promised, which slows comprehension and erodes the four-shard registry's navigability. Implementers are not misled about the registered template (it is restated inline in the parenthetical, and the parallel error-model.md cite reaches the right shard), and the depth-bound contract is unchanged.
