## Problem

The **Cancellation** paragraph and the *Failure-mode templates* sixth-row **Cause** cell in `docs/spec_topics/binder/determinism-cancellation-failure.md` name `ctx.signal` as the signal the runtime checks before the binder call and forwards as `options.signal`. `docs/spec_topics/cancellation.md` is the single source of truth for the cancellation surface and pins `loomAbort.signal` as the only signal downstream components see ("never `ctx.signal` directly"), requiring the runtime to tolerate `ctx.signal` being `undefined` at slash-command entry; `docs/spec_topics/pi-integration-contract/binder-inference.md` already wires `options.signal` to `loomAbort.signal`. The binder page is the lone divergent reading ‚Äî an implementer following only it would read `ctx.signal` at slash entry, observe `undefined`, forward `undefined`, and lose every binder-call cancellation.

## Issue introduction

**Verdict:** present-since-inception (with respect to this page) / single-commit (with respect to the binder corpus)
**Introducing commit:** `7cb4cc3` (2026-05-04 17:59:15 +0200) ‚Äî *pi-loom spec: resolve "Binder LLM call has no cancellation checkpoint"*
**History:** Commit `7cb4cc3` introduced the binder **Cancellation** paragraph and the new `argument binding cancelled` failure-mode row, both naming `ctx.signal`, into the then-monolithic `spec_topics/binder.md`. At that moment `cancellation.md` still described per-entry-point "signal sources" in `ctx.signal` terms (e.g. "Slash-command invocation: the loom receives `ctx.signal` from the Pi command handler"), so the binder wording was internally consistent. Roughly three hours later on the same day, commit `8395c03` (2026-05-04 20:13:49 +0200, *resolve "Cancellation source ctx.signal is undefined when a slash-handler starts"*) replaced the per-entry-point signal-source paragraph with the new `loomAbort` controller model ‚Äî introducing both "`loomAbort.signal` ‚Äî never `ctx.signal` directly" and the explicit "MUST tolerate `ctx.signal` being `undefined` at slash-command entry" obligation. The binder page was not edited in that commit, leaving the `ctx.signal` wording stranded. Subsequent commits on the binder page (e.g. `e4d2c7d`, `fb3474f`, `f5e89f4` whole-corpus reshape into `docs/spec_topics/binder/determinism-cancellation-failure.md`, `2fb5783`) refactored neighbouring content but never revisited this paragraph. The page-level cancellation finding has therefore existed since `7cb4cc3` and became a cross-page contradiction at `8395c03`.

## Solution approach

Rename `ctx.signal` to `loomAbort.signal` at two sites on `docs/spec_topics/binder/determinism-cancellation-failure.md`: the **Cancellation** paragraph's "checks `ctx.signal` immediately before issuing the binder call" clause, and the *Failure-mode templates* sixth-row **Cause** cell ("`ctx.signal` aborted before or during the binder call"). The remainder of both sites ‚Äî the forwarding clause into the binder inference call and the `loom /<name>: argument binding cancelled` system note ‚Äî is already accurate.

- Out of scope: adding visible content (e.g. a cross-reference parenthetical) to the *Failure-mode templates* table cells beyond the token replacement ‚Äî those templates are byte-stable normative output (renderers MUST emit the surrounding template text verbatim).
