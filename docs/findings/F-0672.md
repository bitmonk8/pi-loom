## Finding

The *Hoist obligation* clause in `diagnostic-emission-isolation.md` pins a specific source-level mechanism — "the runtime MUST hoist that `details.event.reason` value into a local string binding (e.g. `const detailsEventReason: string = ...`) **before** entering the wrapped serialisation-and-emission sequence … and the catch arm MUST interpolate the pre-hoisted local rather than re-reading from the partially-constructed structured object or from `entry.shutdownReason` directly." The MUST is on *where the value lives* and *which expression the catch arm reads from*, both of which are unobservable from outside the handler. The supporting prose already states the observable contract a sentence later: the catch arm must emit `` `${code} ${detailsEventReason}` `` carrying a string equal to the `details.event.reason` the structured payload would have carried, even when the serialiser throws part-way through constructing that payload — i.e. the catch-arm reason token must not depend on the order in which the structured payload's fields were assembled.

A conforming implementation that snapshots the reason into a closure, a wrapper-object field, a parameter to the emission helper, or that proves at type level that the structured object's `details.event.reason` slot is written before the serialiser is entered, all deliver the same observable two-token output on every failure path the clause enumerates (mid-construction serialiser throw, throwing-property-access captured value, `"<unreadable>"` snapshot-failure sentinel). Pinning the `const detailsEventReason` local — and forbidding a re-read of `entry.shutdownReason` from the catch arm even though that field's lifetime guarantee (set on entry creation, never mutated thereafter) is pinned by `active-invocation-registry.md` precisely so the catch arm can rely on it — over-constrains the seam without changing what an operator-tooling consumer sees. The same surface already includes a behaviour-shaped sibling clause, the *Construction-site wrap*, that pins the observable (a construction-site throw must not escape the handler, and the per-row fallback token count must be 2 or 3 as enumerated) without naming the variable.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `52b9101` (2026-05-11) — *pi-loom spec: resolve "T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state"*
**History:** Before `52b9101`, the *Diagnostic-emission isolation* paragraph in `docs/spec_topics/pi-integration-contract.md` covered only the four teardown-handler codes and pinned a single bare-`code` fallback (`console.error(code)`) on serialiser throw — no nested-shape codes, no two-token form, no hoist. `52b9101` introduced the fifth (per-invocation) code `loom/runtime/cancelled-by-session-shutdown` and the nested `details.event.{reason, loom}` shape, and along with that added the *Two-token fallback for nested-shape codes*, *Hoist obligation*, and *Construction-site wrap* sub-clauses in a single edit. The mechanism MUST has been carried verbatim through every subsequent rewrite (`c326bfa`, the size-cap split into `diagnostic-emission-isolation.md` at `f5e89f4`, and the per-invocation-scope clarification at `035d82e`); no later commit revisited the mechanism-vs-behaviour distinction. The defect is therefore originating in `52b9101` rather than introduced by interaction or by a later regression.

## Solution Space

**Shape:** single

### Recommendation

Replace the *Hoist obligation* clause with a behaviour-shaped restatement of the same invariant:

> *Catch-arm reason-token stability.* The two-token catch arm of the wrapped serialisation-and-emission sequence MUST emit a `details.event.reason` token equal to the string the structured payload would have carried on a successful emit, on every failure path the *Two-token fallback for nested-shape codes* clause covers — including a serialiser throw that interrupts construction of the structured payload mid-way through stamping `details.event.reason`. Equivalently: the catch-arm reason token MUST NOT be sourced from a read of the partially-constructed structured object. Implementations MAY satisfy this by snapshotting the reason into a local binding before entering the wrapped sequence, by writing `details.event.reason` first under a documented construction order, by passing the reason as a separate argument to the emission helper, or by any other means that delivers the same observable.

Keep three things from the current clause as **non-normative notes** because they retain implementer value without re-pinning a mechanism:

1. The `const detailsEventReason: string = ...` snapshot pattern is the simplest way to satisfy the obligation and is the form the V9g reference test fixture is written against.
2. On a sub-step-2 stamp-throw entry, the captured value is the `"<unreadable>"` sentinel per `session-only-degraded-state.md` — implementations that read `entry.shutdownReason` directly (permitted by `ActiveInvocationRegistry`'s lifetime guarantee) get the same sentinel byte-for-byte.
3. The reason this matters is dedup-bucket preservation: without it, every session-only reason collapses into a single bare-`code` bucket on serialiser throw. (This justification is already in the clause and is what the behavioural restatement implicitly preserves; keep it as the explanatory follow-on.)

Edge cases the implementer must still cover, and which the V9g test suite must assert against the behavioural form:

- A serialiser throw triggered between the construction of `details.event.reason` and the `console.error` invocation must still produce a two-token form whose reason matches the structured payload's reason — i.e. a "writes-`details.event.reason`-last" implementation is non-conformant on this path and the test fixture must include it.
- A catch arm that re-reads from `entry.shutdownReason` on the per-invocation note path must observe the same `"<unreadable>"` sentinel on a sub-step-2 stamp-throw entry, not `undefined`; this is already guaranteed by the residual-gap paragraph but the acceptance criterion should call it out so the rewrite does not regress.
- The *Construction-site wrap* clause's `` `${code} <unreadable>` `` / `` `${code} ${entry.loom} <unreadable>` `` fallbacks remain unchanged — they live structurally outside the wrapped sequence and are already behaviour-shaped.

## Consequence

**Severity:** advisory

Two compliant implementations diverge in source shape (a hoisted local versus an equivalent snapshot in a different binding, or a catch arm that reads `entry.shutdownReason` directly given the registry's lifetime guarantee) while producing byte-identical operator output across every enumerated failure path. Reviewers and implementers spend cycles defending against the mechanism MUST rather than against the observable contract, and the V9g test suite has no way to distinguish the two — the acceptance criteria are inherently behavioural. No operator-visible behaviour is at risk; the cost is review friction and an over-narrow conformance gate.
