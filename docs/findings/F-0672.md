## Problem

The *Hoist obligation* clause in `diagnostic-emission-isolation.md` pins a source-level mechanism: the runtime MUST hoist `details.event.reason` into a `const detailsEventReason` local before entering the wrapped serialisation-and-emission sequence, and the catch arm MUST interpolate that pre-hoisted local rather than re-reading from the structured object or from `entry.shutdownReason`. Those MUSTs constrain where the value lives and which expression the catch arm reads — neither is observable from outside the handler. The observable contract is already stated a sentence later (the two-token catch arm must carry a reason equal to the string the structured payload would have carried, on every enumerated failure path). Implementations that snapshot into a different binding, write `details.event.reason` first under a documented order, pass the reason as a helper argument, or read `entry.shutdownReason` directly under its lifetime guarantee all deliver byte-identical operator output, yet the mechanism MUST rejects them.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `52b9101` (2026-05-11) — *pi-loom spec: resolve "T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state"*
**History:** Before `52b9101`, the *Diagnostic-emission isolation* paragraph in `docs/spec_topics/pi-integration-contract.md` covered only the four teardown-handler codes and pinned a single bare-`code` fallback (`console.error(code)`) on serialiser throw — no nested-shape codes, no two-token form, no hoist. `52b9101` introduced the fifth (per-invocation) code `loom/runtime/cancelled-by-session-shutdown` and the nested `details.event.{reason, loom}` shape, and along with that added the *Two-token fallback for nested-shape codes*, *Hoist obligation*, and *Construction-site wrap* sub-clauses in a single edit. The mechanism MUST has been carried verbatim through every subsequent rewrite (`c326bfa`, the size-cap split into `diagnostic-emission-isolation.md` at `f5e89f4`, and the per-invocation-scope clarification at `035d82e`); no later commit revisited the mechanism-vs-behaviour distinction. The defect is therefore originating in `52b9101` rather than introduced by interaction or by a later regression.

## Solution approach

Rewrite the *Hoist obligation* clause as a behaviour-shaped restatement of the same invariant: the two-token catch arm MUST emit a `details.event.reason` token equal to the string the structured payload would have carried on a successful emit, on every failure path the *Two-token fallback for nested-shape codes* clause covers — including a serialiser throw that interrupts construction mid-way through stamping `details.event.reason`. State the negative observable (the catch-arm token MUST NOT be sourced from a read of the partially-constructed structured object) and leave the mechanism to the implementer (hoisting into a local, documented construction order, a separate helper argument, or reading `entry.shutdownReason` under its registry lifetime guarantee are all permitted). Preserve the dedup-discriminator rationale and the `"<unreadable>"`-sentinel behaviour the clause already carries.

## Solution constraints

- Out of scope: the *Construction-site wrap* / *Scope* / construction-site catch-arm clauses' normative content and their `` `${code} <unreadable>` `` / `` `${code} ${entry.loom} <unreadable>` `` fallback forms — they govern the distinct construction-site-throw path and are already behaviour-shaped.
