## Problem

"final value" (canonical at `functions.md` FN-5, anchor `final-value-language-definition`) and "return value" (used in the glossary `top-level loom` entry, `overview.md`, and `implementation-notes.md`) name the same programmatic-caller-facing value, but the glossary records no equivalence between them. A reader cannot confirm from the glossary that the two terms are synonyms.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `b4fbfac` ("pi-loom spec: resolve \"no central glossary\"", 2026-05-04) — the commit that created the glossary.
**History:** The spec corpus is git-tracked. `git log --diff-filter=A -- '*glossary.md'` shows the glossary was first created in `b4fbfac`; `git log -S 'final value' --` and `git log -S 'statically resolvable'` against the glossary file (both current and pre-move paths) return no commits, i.e. neither term has ever had a glossary entry. The terms themselves predate the glossary (`final value`/`return value` and `statically resolvable` were already in use across `invocation.md`/`functions.md`/`overview.md` when `b4fbfac` landed), so the omission has existed continuously since the glossary was introduced — it is not a regression from any later edit.

## Solution approach

Tie "final value" and "return value" together in exactly one place in the glossary, naming `functions.md#final-value-language-definition` as the canonical definition and "return value" as the programmatic-caller-facing synonym used in `overview.md` / `implementation-notes.md` / the `top-level loom` entry. Either add a new glossary entry for "final value" carrying the equivalence, or fold the equivalence note into the existing `top-level loom` entry — the implementer chooses the single home.
