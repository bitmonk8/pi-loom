## Problem

`V9a` owns the single source-of-truth pinned-constants block (`FACTORY_PROBABLE_CAPABILITIES` and `SHUTDOWN_AWAIT_CAP_MS`). `V9g` reads `SHUTDOWN_AWAIT_CAP_MS` from that block and `V9h` reads it at runtime via the `pinnedConstantReadOk` snapshot seam, yet neither leaf declares a direct `V9a` edge in **Deps** (`V9g`: `V9g-T`, `V9e`, `V9h`, `V17a`; `V9h`: `V9h-T`, `V9m`, `V18c`). Build order is still transitively guaranteed through the `V9e`/`V9m → V9b → V9a` chains, so the defect is silent today. It is inconsistent with `V9b`, `V9i`, and `V17a`, which each declare `V9a` directly for the same block consumption, and a future re-rooting of the `V9e`/`V9m` chains would break read-before-use ordering with no `Deps` edge to surface it.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** b9de3ee — pi-loom plan: resolve "V18a partition assertion imports undeclared V9a probe constant" (2026-06-11, Thomas Andersen); 622d0d3 — pi-loom plan: resolve "SHUTDOWN_AWAIT_CAP_MS physical home contradicts pinned-constants-block placement" (2026-06-11, Thomas Andersen)
**History:** At plan inception (c6a664e), V9g already read `SHUTDOWN_AWAIT_CAP_MS` and V9h already read the pinned-constants snapshot, but V9a did not yet own a single source-of-truth pinned-constants block, so there was no V9a read edge to declare. Commit b9de3ee established V9a as the block's physical owner (the `FACTORY_PROBABLE_CAPABILITIES` home), turning V9h's snapshot read into a V9a read. Commit 622d0d3 then moved `SHUTDOWN_AWAIT_CAP_MS` into that block (making V9g's use a V9a read) and propagated the corresponding direct `V9a` `Deps` edge to V17a — but not to V9g or V9h, leaving their read edges undeclared.

## Solution approach

Add `V9a` to the **Deps** field of `V9g` (`docs/plan_topics/V9g-session-shutdown.md`) and `V9h` (`docs/plan_topics/V9h-unknown-reason-rule.md`), making each leaf's direct read of V9a's pinned-constants block explicit rather than transitively implied through the `V9e`/`V9m` chains. This aligns both leaves with `V9b`, `V9i`, and `V17a`, which already declare `V9a` directly for the same consumption.

## Solution constraints

- Out of scope: the V18c `SHUTDOWN_AWAIT_CAP_MS` literal-read test gap named in the title — this fix addresses only the V9g and V9h `Deps` edges.
