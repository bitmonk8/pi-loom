## Problem

`grammar.md`'s Loom literal sublanguage defines `NamedValueLit ::= Ident "." Ident` with the gloss "head is an enum name in scope, tail a declared variant", and separately asserts "The is-literal check runs at parse time against the Loom AST; it does not require a separate parser." These are in tension: deciding the head is an in-scope enum and the tail a declared variant is name/scope resolution, which a pure AST-shape check cannot perform. The spec never says whether that validation is folded into the parse-time is-literal gate or deferred to a downstream resolution pass, and it names no diagnostic for a literal-position dotted pair whose head does not resolve to an enum (the tail-not-a-variant case has `loom/parse/unknown-variant` but no stated firing site or ordering). Because diagnostic codes are an observable, version-pinned surface, conforming implementations can emit different codes for the same source.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** f841b12 — spec: resolve Cluster 2 (literal sublanguage + grammar appendix) (2026-05-04, Thomas Andersen)
**History:** The Loom literal sublanguage and its grammar appendix were authored whole in f841b12 (resolving spec-review Cluster-2 findings #6/#10/#5/#17). That same commit introduced both the `NamedValueLit ::= Ident "." Ident` production with its "head is an enum name in scope, tail a declared variant" semantic gloss and the "The is-literal check runs at parse time against the Loom AST; it does not require a separate parser" claim — so the tension between the semantic (resolution-requiring) constraint and the pure-AST-parse-time assertion, and the absent head-not-an-enum diagnostic, have been present since the section's first commit. No later commit reconciled them (`git log -S 'NamedValueLit'` shows only the f841b12 authoring and the docs/ relocation move).

## Solution approach

In `grammar.md`, clarify the is-literal check as a pure AST-shape gate and rewrite the `NamedValueLit` production gloss and the "runs at parse time against the Loom AST" sentence so enum-in-scope and declared-variant validation is scoped out of the parse-time gate, with a forward-link to where that validation runs (the whole-file resolution pass that already resolves `params:` `NamedType` references and owns `loom/parse/unknown-variant`). In `code-registry-parse.md`, state that `loom/parse/unknown-variant` fires in literal positions for a dotted pair whose tail is not a declared variant, and name a diagnostic for a literal-position dotted pair whose head resolves to a non-enum symbol or to nothing. Pin the phase ordering so a non-`Ident "." Ident` member-access shape fires the position's not-literal code (`loom/parse/default-not-literal` / `loom/parse/tool-arg-not-literal`) before resolution, and resolution-pass checks run second.

## Solution constraints

- Out of scope: the `loom/parse/unknown-variant` definition site on `schemas.md` (Variant access); `code-registry-parse.md` references it rather than redefining it.
