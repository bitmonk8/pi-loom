## Finding

`grammar.md` (Loom literal sublanguage) defines `NamedValueLit ::= Ident "." Ident` with the gloss *"Enum.Variant access; head is an enum name in scope, tail a declared variant"*, and separately asserts *"The is-literal check runs at parse time against the Loom AST; it does not require a separate parser."* These two statements are in tension. Deciding that the head `Ident` is an in-scope enum and the tail `Ident` is a declared variant of that enum is name/scope resolution — it cannot be performed by a pure AST-shape check. The spec never says whether the enum-in-scope / declared-variant validation is folded into the parse-time is-literal gate or deferred to a downstream resolution pass, so two implementers diverge on *when* the check runs and *which* diagnostic it produces.

The diagnostic assignment is correspondingly under-specified for the failure cases in a literal position (a `params:` default RHS or a Pi-tool call argument):

- A literal-position `Enum.Variant` whose **tail** is not a declared variant: `loom/parse/unknown-variant` exists (owned by `schemas.md` — Variant access, and registered in the parse code-registry), but the spec never states that it fires inside the literal sublanguage, nor in what order relative to the is-literal shape gate. The is-literal gate's failure codes (`loom/parse/default-not-literal` / `loom/parse/tool-arg-not-literal`) are the competing candidates, and nothing disambiguates.
- A literal-position dotted pair whose **head** does not resolve to an enum (a `let` binding, a schema or `fn` name, or an unresolved identifier): no diagnostic is named at all. The forbidden-forms list says "member access on anything other than `Enum.Variant`" is rejected — but classifying `obj.field` as *not* `Enum.Variant` itself requires resolving that `obj` is not an enum, which is exactly the resolution step whose phase is unstated. The candidate outcomes (`default-not-literal`/`tool-arg-not-literal`, `loom/parse/unknown-identifier`, or an unnamed not-an-enum code) are never reconciled.

Because diagnostic codes are an observable, version-pinned surface, leaving the phase model and the head-not-an-enum disposition unspecified lets conforming implementations emit different codes for the same source.

## Solution Space

### Recommendation

Split the `NamedValueLit` validation into two phases with a pinned ordering.

In `grammar.md`, make the is-literal check a pure AST-shape gate: it admits any `Ident "." Ident` node as a syntactically well-formed `NamedValueLit` without consulting scope. Replace the `NamedValueLit` production comment's "head is an enum name in scope, tail a declared variant" wording with a shape-only gloss, and add a note that enum-in-scope and declared-variant validation is a downstream resolution-pass check. Scope the "runs at parse time against the Loom AST; does not require a separate parser" sentence to AST shape only, and forward-link the resolution-pass check.

Perform enum-in-scope and declared-variant validation in the existing whole-file resolution pass — the same pass that resolves `params:` `NamedType` references (`frontmatter-fields-a.md`) and already owns `loom/parse/unknown-variant`. In `code-registry-parse.md`, state that `loom/parse/unknown-variant` fires in literal positions for a dotted pair whose tail is not a declared variant, and register a head-not-an-enum code (or designate `loom/parse/unknown-identifier` for the case where the head resolves to nothing) for a literal-position dotted pair whose head resolves to a non-enum symbol or to nothing.

Pin the phase ordering: the AST-shape gate runs first — a non-`Ident "." Ident` member-access shape (e.g. `a.b.c`, member chains) fires the position's not-literal code (`loom/parse/default-not-literal` / `loom/parse/tool-arg-not-literal`) before resolution runs — and the resolution check runs second. Edge case the implementer must watch: a literal referencing an enum declared later in the same file must resolve under whole-file resolution rather than fail as not-a-literal.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on which diagnostic fires for a literal-position `Enum.Variant` whose head is not an enum or whose variant is unknown (`loom/parse/default-not-literal`/`tool-arg-not-literal` vs `loom/parse/unknown-variant` vs `loom/parse/unknown-identifier`), and on whether the check happens in the parse-time literal gate or a later resolution pass. Diagnostic codes are an observable, version-pinned surface, so the same source produces different conformance-visible output across implementations.
