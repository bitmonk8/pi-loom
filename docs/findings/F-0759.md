## Finding

`docs/spec_topics/errors-and-results/queryerror-variants.md:1` opens with the H1 `# Queryerror variants` (lowercase `e`), and the parent index `docs/spec_topics/errors-and-results.md:8` cites it as `- [Queryerror variants](./errors-and-results/queryerror-variants.md)` with the same lowercase form. Every other occurrence in the corpus — the H2 `## QueryError variants` immediately below the H1, the schema declaration `schema QueryError = …`, the prose, the cross-references from `cancellation.md`, `tool-calls.md`, `slash-invocation.md`, `query/query-failure-and-repair.md`, `hard-ceilings/*`, `glossary.md`, and the `plan_topics/V4d-*`, `V12b-*`, `V13d-*` Spec fields — spells the type name `QueryError` (camelCase).

The two miscapitalised strings are presentational labels only (page title and TOC link text); the file path stays `queryerror-variants.md` and the in-page anchor `#queryerror-variants` is unaffected. No cross-link target breaks. The defect is purely a reader-facing inconsistency: the hub TOC entry and the page H1 do not name the type the page is *about*.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `f5e89f4` (`docs: enforce 100KB hard / 30KB recommended size cap on spec set`, 2026-06-04).
**History:** Before `f5e89f4`, `errors-and-results.md` was a single ~300-line file with the H2 `## QueryError variants` (correct camelCase). The split commit introduces `docs/_tools/split_spec.py`, which mints child pages from slug entries — for this section the slug literal `"queryerror-variants"` (see `split_spec.py:84`) — and mechanically derives titles via `title_from_slug` (`split_spec.py:114-115`): `s.replace("-", " ")[:1].upper() + s.replace("-", " ")[1:]`. That helper only uppercases the first character of the slug, so `queryerror-variants` → `Queryerror variants` rather than `QueryError variants`. The same helper writes both the child H1 (`split_spec.py:154`) and the parent's Contents link text (`split_spec.py:163`), which is why both presentational sites carry the defect in lock-step. Subsequent edits to `queryerror-variants.md` (`d510a55`, `c3ea1d1`, `8455d5e`, `d37c563`, `43bef30`, `33d959a`, `0e2eb05`, `34517a0`, `f7a505e`, `4af6fb3`, `901d49c`) all touch body content and never revisit the H1.

## Solution Space

**Shape:** single

### Recommendation

Retitle the page H1 and the index TOC link text to `QueryError variants` (camelCase), matching the H2 on the same page and the type name used everywhere else:

- `docs/spec_topics/errors-and-results/queryerror-variants.md` line 1 — change `# Queryerror variants` to `# QueryError variants`.
- `docs/spec_topics/errors-and-results.md` line 8 — change `- [Queryerror variants](./errors-and-results/queryerror-variants.md)` to `- [QueryError variants](./errors-and-results/queryerror-variants.md)`.

Leave the filename `queryerror-variants.md`, the directory layout, and the in-page anchor `#queryerror-variants` unchanged — renaming any of those would break the 18+ inbound cross-references catalogued under spec and plan files. The slug stays lowercase by convention (filenames and anchors are kebab-cased throughout the corpus); only the human-readable label is being corrected.

While editing, also patch `docs/_tools/split_spec.py:114-115` so a future re-run of `split_spec.py` does not silently regenerate the same defect — either special-case `queryerror-variants` in a slug→title override map alongside the existing entries, or change the helper to title-case each hyphen-separated word *and* leave any pre-mapped overrides intact. Without the tooling fix, any re-driven split would overwrite the H1 back to `Queryerror variants`.

## Consequence

**Severity:** cosmetic

A reader scanning the errors-and-results index sees a label that disagrees with the type name used throughout the corpus and with the page's own H2; an implementer searching for the canonical `QueryError` declaration may scroll past the TOC entry. No normative obligation, anchor, or link target is affected.
