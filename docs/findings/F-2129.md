## Problem

PIC-25's *Hoist obligation* clause in `diagnostic-emission-isolation.md` mandates a specific code construction — hoisting the `details.event.reason` value into a named local before the wrapped serialisation-and-emission sequence, and forbidding the catch arm from re-reading `entry.shutdownReason` directly — rather than the observable contract it guards. Forbidding a catch-arm re-read of the *partially-constructed structured object* is sound, since a mid-construction serialiser throw can leave that field unbuilt. But forbidding a re-read of `entry.shutdownReason` over-prescribes: an implementation that re-reads `entry.shutdownReason` and re-applies the same unset → `"<unreadable>"` substitution produces byte-identical output to the hoisted-local approach. The same section's *Construction-site wrap* clause already permits the catch arm to re-read immutable registry state (`entry.loom`), undercutting a blanket re-read prohibition for that channel.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** 52b9101 — pi-loom spec: resolve "T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state" (2026-05-11, Thomas Andersen)
**History:** The *Hoist obligation* clause was authored wholesale in 52b9101, then resident in `pi-integration-contract.md`; its added lines carry the prescriptive "MUST hoist … into a local string binding … before entering the wrapped serialisation-and-emission sequence" and the catch-arm "MUST interpolate the pre-hoisted local rather than re-reading from … `entry.shutdownReason` directly" verbatim. The over-prescription was baked into the clause at the moment it was written — `git log -S 'detailsEventReason'` and `git log -G 'MUST hoist'` across all paths show no earlier appearance. Later commits only relocated and re-numbered it: f5e89f4 split the clause into `diagnostic-emission-isolation.md` under the 100KB size cap, and 82856ab coined the PIC-24..28 anchors; neither altered the prescriptive phrasing.

## Solution approach

Rewrite PIC-25's *Hoist obligation* clause as a behavioural contract: the `details.event.reason` token the catch arm emits MUST equal the string the structured payload would have carried — always a string, never `undefined`, with the unset-`entry.shutdownReason` → `"<unreadable>"` substitution applied, regardless of construction order. Drop the mandate to bind the value into a named local before the wrapped sequence and the blanket prohibition on re-reading `entry.shutdownReason`. Retain the one prohibition that carries behavioural force: the catch arm MUST NOT read `details.event.reason` from the partially-constructed structured object. Demote the hoist-into-a-local pattern to a non-normative implementation note.

## Solution constraints

- The behavioural restatement MUST preserve the `details.event.reason` value space — including the literal `"<unreadable>"` substitution on the sub-step-2 stamp-throw entry (`#session-only-reason-degraded-state`) — so the two-token fallback does not collapse distinct reasons into a single bare-`code` dedup bucket.
