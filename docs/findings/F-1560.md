## Finding

DISC-6 (`docs/spec_topics/discovery/package-and-settings.md`, *Package discovery*) describes three observable trip paths for the `loom/load/discovery-slow` warning: the file-count cap (`looms.scanPackagesMaxFiles`) firing alone, the wall-clock cap (`looms.scanPackagesTimeoutMs`) firing alone, and the simultaneous-cap tie-break. The warning is said to name "the root being scanned and the cap that fired." For the simultaneous case the paragraph pins the field value verbatim — "the warning's `cap` field is `looms.scanPackagesMaxFiles`" — but the two single-cap sentences pin nothing: they assert only the behavioural phrase "the cap that fired" without inlining either of the two settings-key strings the `cap` field is supposed to carry.

The value set is in fact closed elsewhere. `docs/spec_topics/diagnostics/placeholder-rendering-b.md` §7 defines `<cap> ∈ {looms.scanPackagesMaxFiles, looms.scanPackagesTimeoutMs}`, and `docs/spec_topics/diagnostics/code-registry-load.md` rows the `discovery-slow` *Message* template as `package-discovery walk aborted at <root>: <cap> cap reached`, with a worked render vector in §7's test-vector list. A conformance-test author can therefore recover the token a single-cap trip must emit — but only by chasing two cross-references out of the DISC-6 site and pairing the closed enum against the "names the cap that fired" verb to infer the predicate-to-token mapping.

The defect is the asymmetry itself: a reader of DISC-6 in isolation gets a verbatim asserted value for one of three trip paths and a bare behavioural description for the other two. The local statement reads as if the tie-break case is uniquely worth pinning, inviting the question "why is the single-cap value not stated?" Either all three trip paths should state the `cap` value at the DISC-6 site, or none should (deferring uniformly to the closed enum). Today neither holds.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `475155c` (pi-loom plan: resolve "V14m discovery walk omits scoped packages and has no upper bound", 2026-05-05) and `0378193` (pi-loom spec: resolve "T12 dual-cap simultaneous-breach tie-break", 2026-05-18)
**History:** Commit `475155c` introduced the `loom/load/discovery-slow` registry row, the bounded-walk paragraph, and the *Message* template `package-discovery walk aborted at <root>: <cap> cap reached`, with the prose "names the root being scanned and the cap that fired". At that point all three trip paths were equally implicit — the `<cap>` value had to be inferred from context for every case. Commit `0378193` (resolving T12) then added the tie-break sentence pinning `cap` = `looms.scanPackagesMaxFiles` for the simultaneous case, but did not back-fill verbatim values for the two single-cap cases. The asymmetry this finding flags is the interaction of those two edits: the original paragraph treated all paths symmetrically (none stated, all inferable), and the tie-break patch broke the symmetry by stating one. The closed-enum entry at `placeholder-rendering-b.md` §7 and the registry-row test vector were added on later commits and would have made a full pinning straightforward at the time of the tie-break edit.

## Solution Space

**Shape:** single

### Recommendation

Make the three trip paths render symmetrically at the DISC-6 site. Replace the "names the root being scanned and the cap that fired" clause with an explicit predicate-to-token binding: the `cap` field carries `looms.scanPackagesMaxFiles` when the file-count predicate trips alone, `looms.scanPackagesTimeoutMs` when the wall-clock predicate trips alone, and `looms.scanPackagesMaxFiles` for the simultaneous tie-break (per the existing tie-break rule). Retain the existing tie-break sentence as the local explainer for *why* `maxFiles` wins on a simultaneous trip, without restating the value a second time.

Add a parenthetical cross-reference to the `<cap>` closed enum in `placeholder-rendering-b.md` §7 so a reader who spots the inlined values knows the rendering surface is closed and spec-versioned under GOV-7/GOV-8. Do not reword or delete the §7 enum entry itself: the inlined list at DISC-6 is the predicate-to-token binding; the §7 entry remains the rendering-surface contract.

Update the `V10b-T` `DISC-6` test bullet so all three `cap` token assertions are explicit — maxFiles-only trip → `cap = "looms.scanPackagesMaxFiles"`; timeoutMs-only trip → `cap = "looms.scanPackagesTimeoutMs"`; simultaneous trip → `cap = "looms.scanPackagesMaxFiles"`. The existing "file-count is checked before time on a tie" bullet covers tie-break ordering but is silent on the two single-cap token bindings.

## Consequence

**Severity:** cosmetic

The `cap` token is fully recoverable by composing the §7 closed enum with the "names the cap that fired" verb, so no two implementations diverge on the emitted value and no observable behaviour is at risk. The cost is reader-side: the DISC-6 paragraph carries a false signal that only the tie-break value needs pinning, and a test author asserting the single-cap token must chase two cross-references rather than read the binding at the rule site.
