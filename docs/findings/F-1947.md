## Finding

The `V17b` (and the paired `V17b-T`) **Tests.** bullet ends its edge-case clause with: "the `session_shutdown` teardown-iteration path is governed by its own sub-step-2 swallow rule and is out of scope for this steady-state trap." The phrase "its own sub-step-2 swallow rule" carries no document or section anchor — "sub-step-2" indexes into no document the leaf cites. The leaf's **Spec.** field cites `cancellation.md`, `host-interfaces-core.md`, and `host-prerequisites.md`; none of those *defines* the rule. `cancellation.md` (the *Forwarding-listener throw* clause the bullet is built from) only *references* it — "the `session_shutdown` sub-step 2 swallow rule remains the governing rule for the teardown-iteration path" — again without an anchor. The actual rule lives in `pi-integration-contract/patch-skew-degradation.md`, in the numbered `session_shutdown` teardown sequence at sub-step 2 ("Cancel in-flight invocations"): a per-entry `try`/`catch` around the `shutdownReason` stamp and the `loomAbort.abort(reason)` call that proceeds to the next entry on a throw and emits no diagnostic. That file is not cited anywhere in `V17b`/`V17b-T`.

The implementer is told a behaviour is out of scope but is given no traversal path to confirm what that behaviour is or that it truly differs from the steady-state trap under test. The plan already uses a concrete citation form for the adjacent sub-step in the same teardown sequence — `V9a`, `V17a`, `V9g`, `V9g-T`, and `V18c` all write "[`patch-skew-degradation.md` §`session_shutdown` sub-step 3](../spec_topics/pi-integration-contract/patch-skew-degradation.md)" — so the un-anchored sub-step-2 reference is an inconsistency against an established in-plan convention rather than an unavoidable gap.

## Solution Space

**Shape:** single

### Recommendation

In both `docs/plan_topics/V17b-forwarding-listener-throw-trap.md` and `docs/plan_topics/V17b-T-forwarding-listener-throw-trap.md`, in the **Tests.** bullet's edge-case clause, replace the un-anchored phrase "is governed by its own sub-step-2 swallow rule" with a concrete cross-reference to where the rule is defined — `patch-skew-degradation.md` `§session_shutdown` sub-step 2 — mirroring the existing in-plan citation form used for sub-step 3 (the linked `[\`patch-skew-degradation.md\` §\`session_shutdown\` sub-step 2](../spec_topics/pi-integration-contract/patch-skew-degradation.md)` style). The cited rule is the per-entry `try`/`catch` in sub-step 2 ("Cancel in-flight invocations") that swallows a stamp-throw or `loomAbort.abort()`-throw, proceeds to the next registry entry, and emits no diagnostic; naming that behaviour inline alongside the link lets the implementer confirm the out-of-scope boundary without further traversal.

Apply the same edit to both leaves — their Tests bullets are byte-identical, so the fix is one coherent change reproduced across the implementation/test pair. Keep the clause's existing "out of scope for this steady-state trap" framing; only the reference is being made resolvable.

## Consequence

**Severity:** advisory

The implementer can still write the steady-state trap test (the clause only marks the `session_shutdown` path as out of scope), but cannot quickly verify the exclusion is correct or locate the rule it defers to, and a reviewer cannot grep for the governing document. The risk is wasted traversal time and an unverifiable scope boundary, not a divergent or spec-violating leaf.
