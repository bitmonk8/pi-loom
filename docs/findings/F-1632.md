## Problem

The `loom-typecheck-namespace` paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` closes with an *Audit-gate carve-out* sub-clause that imposes a normative MUST on build/CI audit-gate tooling, directing how a code-registry-reconciliation gate must scope its scan of `loom/...` literals. `diagnostic-shape.md` is the runtime-diagnostics contract (the `Diagnostic` wire shape, delivery channels, and the Code registry rules DIAG-1…DIAG-4); this sub-clause is the lone build-tooling obligation in the file and constrains nothing the file's readers implement at runtime. The carve-out is the operational corollary of the namespace boundary the paragraph legitimately establishes — that `loom/typecheck/*` brands carry no registry row and are out of scope for the Code registry — so the defect is one of framing: it is phrased as a MUST aimed at "audit gates" rather than as a scope property of the registry it bounds.

## Solution approach

Rewrite the *Audit-gate carve-out* sub-clause in `diagnostic-shape.md` so it states the disjointness as a scope property of the Code registry — that the registry does not cover `loom/typecheck/*` and that any reconciliation of `loom/...` literals against it treats `loom/typecheck/` as a prefix-disjoint namespace — rather than as a MUST directed at external audit gates. Add a stable anchor on the reframed clause so downstream consumers can cite it directly.

## Solution constraints

- Keep the clause in `diagnostic-shape.md` adjacent to the Code registry it bounds; out of scope to relocate it to `source-language-stability.md` (GOV-15) or `inventory-audit-intro.md`, whose carve-outs and audit cover different concerns.
