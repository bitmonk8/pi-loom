## Finding

The `Conversation drive — subagent mode` block in `docs/spec_topics/pi-integration-contract/subagent.md` accumulates four under-supported claims around the `createAgentSession` call.

**(1) Helper / `params` anchors.** The illustrative `ts` snippet calls two loom-internal helpers — `lowerLoomToolsToPi(loom.tools)` and `renderLoomSystemPrompt(loom.frontmatter.system, params)` — and a free variable `params`, none of which is anchor-linked or signature-pinned anywhere in the spec corpus. A `grep -rn "lowerLoomToolsToPi\|renderLoomSystemPrompt" docs/` returns only the spawn-block usage site. An implementer cannot tell from the spec which page owns these helpers' contracts or what `params` is bound to at this site (the load-time-resolved per-invocation values from `Parameters and Frontmatter`, but the snippet does not say so).

**(2) `ResourceLoader` member-set completeness.** The "Spawn-block satellite type pins" paragraph asserts that the rule‑4 adapter's "loom-load-bearing member surface is exactly the set rule 4 enumerates" — a *load-bearing* claim, not a *completeness* claim. The adapter is constructed as an object literal that must satisfy the structural `ResourceLoader` interface (declared at `dist/core/resource-loader.d.ts` and shipping nine required members at the loom 1.0 Pi-SDK pin: `getExtensions`, `getSkills`, `getPrompts`, `getThemes`, `getAgentsFiles`, `getSystemPrompt`, `getAppendSystemPrompt`, `extendResources`, `reload`). If a Pi minor adds a tenth required member, the loom's object-literal adapter fails to type-check at the spawn site with no value to supply; nothing in the spec or in version-bump-step2.md item (o) explicitly says "the nine listed members ARE the full required member set." Item (o) audits that the *enumerated* members "still carr[y] the members the rule-4 adapter implements with compatible shapes" but does not flag the addition of a new required member.

**(3) Empty-resource tolerance presupposition.** Rule 4 has the adapter return `{ extensions: [], errors: [] }`, empty arrays for skills/prompts/themes/agentsFiles, and no-op `extendResources`/`reload`. That `createAgentSession` accepts this all-empty/no-op resource shape and produces a working session is a behavioural presupposition of the Pi-SDK pin (no type encodes "empty resource sets are tolerated"); nothing in the spec records it as a version-bump-audited posture on the footing of items (am)/(aj). A future Pi minor that started requiring a non-empty `extensions` array, or that crashed when `reload()` was an unawaited no-op, would surface as a spawn-site runtime failure with no version-bump signal.

**(4) UI-non-attachment causal attribution.** The block carries the sentence "The in-memory `SessionManager` is the mechanism that prevents subagent output from reaching Pi's user-facing UI." This is a factual claim about *why* the spawned subagent's transcript does not stream into the operator's TUI. `dist/core/session-manager.d.ts` documents `SessionManager.inMemory(cwd?)` only as "Create an in-memory session (no file persistence)" — its purpose is no on-disk persistence, not UI suppression. The TUI displays the *user* session it subscribes to; the spawned subagent is a *separate* `AgentSession` instance whose events the TUI never subscribes to. The actual UI-non-attachment mechanism is the separate-`AgentSession`-instance fact (combined with the spec's prohibition on the global `pi.on("agent_end", …)` and on routing `loom-system-note` emissions from a subagent), not the choice of in-memory persistence. A loom that used `SessionManager.create(cwd)` (file-persisting) for the spawned `AgentSession` would still keep its output off the TUI; the persistence mode does not gate UI attachment. Item (aj) audits the *transcript-non-retention* behaviour of `SessionManager.inMemory`, reinforcing that the in-memory factory's contract is about persistence, not UI routing — so the spec is naming the wrong mechanism for the operator-isolation guarantee.

## Solution Space

**Shape:** multiple

The finding bundles four independent obligations on the same spawn block. They cannot all be resolved by a single paragraph edit — option A is local in-block cleanup, option B adds new version-bump-checklist entries to a different file, and option C corrects a substantive factual claim and ripples into a sibling capability-inventory page. Order them smallest first so each lands on a stable baseline.

### Option A — Anchor the helpers and pin the `ResourceLoader` surface as complete

**Approach:** Edit only `docs/spec_topics/pi-integration-contract/subagent.md` — the spawn block and the immediately following *Spawn-block satellite type pins* paragraph.

**Spec edits:**
- In the ts snippet, add a leading comment that the snippet is illustrative and that `params` is the load-time-resolved per-invocation values from `Parameters and Frontmatter` (anchor-link to the existing `frontmatter.md` section), and that `lowerLoomToolsToPi` / `renderLoomSystemPrompt` are loom-internal helpers whose contracts are owned by `extension-bootstrap-and-per-loom.md` §Per-loom registration (ToolDefinition derivation rules) and `frontmatter.md` (system-prompt interpolation rules) respectively. Two anchor-links and one sentence; do not pin the helper names themselves (the "Shard-09 over-prescription" finding wants the names demoted to illustrative, which is compatible with this anchor wording).
- In the *Spawn-block satellite type pins* paragraph, replace "its loom-load-bearing member surface is exactly the set rule 4 enumerates" with "the nine members rule 4 enumerates ARE the complete required member set of the `ResourceLoader` interface at the loom 1.0 Pi-SDK pin; a Pi minor that adds a tenth required member surfaces as a spawn-site type-check failure rather than at the SDK surface-inventory assertion, and is caught by item (o)'s editorial audit which the *Adds* below extends to cover required-member-set growth, not just shape compatibility of the enumerated members."
- Extend item (o) in `version-bump-step2.md` by one clause: "(o) … and that no *additional* required member has been introduced beyond the nine enumerated above; an added required member is caught here." This is the minimum edit to make the completeness pin enforceable.

**Pros:** Local; touches one normative file and one one-clause addition to a sibling. No content fact-checking required.
**Cons:** Does not address the UI-attribution defect or the empty-resource-tolerance gap.
**Risks:** Conflicts with the "Shard-09 over-prescription & naming" finding if that finding wants helper names demoted to *non-normative* — the anchor wording here treats them as illustrative variable names with anchored contracts elsewhere, which is the compatible reading.

### Option B — Add empty-resource-tolerance as a version-bump-audited presupposition

**Approach:** Add a new entry to the *Editorial-review checklist for unpinned host presuppositions* in `version-bump-step2.md`, on the same footing as items (am) and (aj). Anchor it from the spawn block's rule 4 with the same "re-validated on each Pi minor bump per item (XX)" sentence pattern rule 2 already uses.

**Spec edits:**
- Insert a new lettered item (placement-wise after (o) since it concerns `ResourceLoader` adapter behaviour): "*Empty-resource tolerance of `createAgentSession` against the loom's `ResourceLoader` adapter* — rule 4 of [Conversation drive — subagent mode](./subagent.md). The contributor SHOULD confirm against the candidate `@earendil-works/pi-coding-agent` minor that `createAgentSession` produces a working spawned `AgentSession` when the supplied `ResourceLoader` returns the empty / no-op values rule 4 fixes (empty `extensions` / `errors`, empty skills / prompts / themes / agentsFiles, and no-op `extendResources` / `reload`). The empty-resource tolerance is neither factory-probed nor enumerated by step 2(a)'s seven-capability literal-read assertion, so a silent change that started rejecting an empty-extensions adapter or that required `reload()` to do real work would surface as a spawn-site failure rather than at load time. SHOULD-level at loom 1.0; escalates to a build-time pin once Pi exposes a typed contract that step 2(a)'s probe can mechanically verify."
- In subagent.md rule 4, append "Empty-resource tolerance of this adapter is re-validated on each Pi minor bump per item (XX) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below" (paralleling rule 2's existing anchor to item (am)).

**Pros:** Closes the empty-resource gap on the same audited footing as the other un-typed-contract presuppositions (am / aj / ai). No factual claims to re-research.
**Cons:** Touches two files; needs a new lettered item which collides with the "step 2 label space" naming finding in shard-09 (low-importance, separately tracked).
**Risks:** None substantive.

### Option C — Correct the UI-non-attachment causal attribution

**Approach:** Replace the load-bearing causal sentence with a correct one identifying the actual mechanism, and re-route the operator-isolation guarantee in `capability-inventory-items.md` item 3 to that mechanism.

**Spec edits:**
- In subagent.md, replace "The in-memory `SessionManager` is the mechanism that prevents subagent output from reaching Pi's user-facing UI; user-visible behaviour is specified in [Slash-Command Invocation — User-visible streaming](../slash-invocation.md#slsh-2)." with "The spawned subagent's output does not reach Pi's user-facing UI because the spawned `AgentSession` is a *separate* session instance from the TUI-subscribed user session, combined with the runtime's mandate above to use the session-local `subscribe` API rather than the global `pi.on('agent_end', …)` event. The choice of `SessionManager.inMemory(cwd)` is owned by the transcript-non-retention guarantee (per capability-inventory item 3 and version-bump checklist item (aj)), not by the UI-non-attachment guarantee — a file-persisting `SessionManager` on the same separate `AgentSession` would equally keep subagent output off the TUI. User-visible streaming on the parent prompt-mode surface is specified in [Slash-Command Invocation — User-visible streaming](../slash-invocation.md#slsh-2)."
- In `capability-inventory-items.md` item 3, separate the two guarantees: "transcript privacy (no file persistence)" — `SessionManager.inMemory(cwd)` — from "operator-non-attachment" — the spawned `AgentSession` being a distinct session instance the TUI never subscribes to.
- Audit the state-isolation matrix's "caller's transcript" row (which currently attributes both privacy and isolation to `SessionManager.inMemory(cwd)`) and split similarly if it conflates the two.

**Pros:** Removes the only factually-wrong sentence in the cluster. Aligns the spec with what `inMemory()` actually does at the pin (Pi's own d.ts comment says "no file persistence" only).
**Cons:** Substantive re-wording that ripples into a sibling page; needs the reviewer to confirm the mechanism description against the Pi codebase (this finding's verification already did so against `dist/core/session-manager.d.ts` and the spec's prohibition on global `pi.on("agent_end", …)`).
**Risks:** The state-isolation matrix's "caller's transcript" row may need re-splitting; touching the matrix is a higher-impact edit than just the prose sentence and could draw lens scrutiny on the next review pass. Mitigation: keep the matrix row intact (it correctly cites `SessionManager.inMemory(cwd)` for transcript privacy) and only edit the standalone causal sentence + capability-inventory item 3.

### Recommendation

Address all three options in the order **A → B → C**. Option A is the smallest scope-bounding edit and removes the implementer-confusion sources without touching factual claims. Option B is also additive and lands cleanly on top of A. Option C is the substantive correction; landing it last means the prose change is reviewed against a stable surrounding context. Each option is independently loopable through `spec-diff-fix-loop`; bundling them empirically widens the lens-critique surface enough to risk divergence. Implementers must watch the edge case where the state-isolation matrix's "caller's transcript" row currently double-attributes privacy and isolation to `SessionManager.inMemory(cwd)` — under option C, keep the matrix row's attribution and only re-target the standalone causal sentence and the capability-inventory item 3 enumeration.

## Consequence

**Severity:** correctness

Three of the four sub-claims (helper anchors, completeness pin, empty-resource tolerance) leave implementers guessing in ways that compile-and-link but surface as runtime spawn failures on a Pi minor that adds a required `ResourceLoader` member or tightens its empty-resource tolerance. The fourth sub-claim (the in-memory-`SessionManager`-as-UI-mechanism sentence) is actively wrong: it names a mechanism that does not perform the duty the sentence assigns it, so an implementer reasoning from the spec about how to extend the subagent (e.g. swapping the session manager, adding a debugging TUI sink) draws the wrong conclusion about which property keeps the operator-isolation guarantee live. Two reasonable implementers will diverge on which mechanism is load-bearing.
