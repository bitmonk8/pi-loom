## Problem

PIC-22 *Parallel subagent spawn initiation* on `subagent.md` is a behavioural spawn-initiation MUST (initiate `createAgentSession` for all N parallel subagent invocations before any one returns), but its body also carries a multi-sentence resource-ownership NFR run pinning that runtime-value heap, OS file-descriptor / socket / child-process slots, and provider rate-limit / quota are owned by the host JavaScript runtime, the host OS, and the LLM provider respectively, together with the per-call surfacing routes (V8 `RangeError` family / OS-exhaustion throws → `loom/runtime/internal-error`; provider 429 → `TransportError`; uncatchable host fatals → process termination). That NFR does not constrain spawn-initiation behaviour, so it inflates the behavioural requirement and buries the resource-ownership boundary at the tail of a MUST where an operator reasoning about resource limits is unlikely to find it. The `#no-invocation-cap` concurrency-disposition block immediately above already states the no-cap / no-scheduler stance the NFR run elaborates.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `a389af0` (2026-05-13), `7a3b907` (2026-06-10)
**History:** The resource-ownership prose was authored at `a389af0` ("rewrite T20 as positive ownership-boundary statement"), which resolved the T20 finding about undisclaimed non-memory resource exhaustion. At that point it lived in the `no-invocation-cap` concurrency-disposition paragraph in the then-`implementation-notes.md`, attached to the no-cap/no-scheduler disposition — not yet a distinct anchored MUST. The embedding defect crystallised at `7a3b907` ("subagent parallel-initiation MUST has no closing leaf"), which carved the `<a id="pic-22">` anchor out of the combined concurrency paragraph: it split the spawn-initiation MUST into the standalone PIC-22 requirement but carried the resource-ownership sentences into the new PIC-22 body rather than leaving them on the `concurrent-invocation-isolation` / `no-invocation-cap` paragraph that owns the no-cap disposition. The subsequent `f5e89f4` 100KB-cap split only relocated the page; it did not introduce the bundling.

## Solution approach

Move the resource-ownership NFR run out of the PIC-22 body and into the `#no-invocation-cap` concurrency-disposition block immediately above, continuing that block's no-cap / no-scheduler disposition. Leave PIC-22 as a purely behavioural spawn-initiation requirement, and repoint its "does not promise resource unboundedness" clause to the relocated note.

## Solution constraints

- Keep the `#no-invocation-cap` and `#pic-22` anchors at their current positions — SM-7d's owner pointer, the future-considerations "No admission cap" entry, and the GOV aggregator inventory resolve to them.
- The relocation is organisational only: preserve the resource-ownership normative content — resource ownership, the per-call routing to `loom/runtime/internal-error` / `TransportError` / process-termination, and the major-version stability tier — unchanged, do not rewrite it.
- Out of scope: the PIC-22 conformance-test-strictness concern owned by F-1682.
