## Finding

IMP-1 (`docs/spec_topics/imports.md`, "resolver failure contract") enumerates exactly three arms under which a `.warp` import spec is *unresolvable* and surfaces `loom/load/unresolvable-warp-path`: (1) the spec is non-relative; (2) after relative resolution against the importing file's directory, no directory entry's name matches the path literal's final segment byte-for-byte; (3) a byte-exact matching entry exists but is not readable (`EACCES`/`EPERM` on that entry, or a broken symlink on that entry). The non-normative implementation note then tells implementers the conformant strategy is to `FileSystem.readdir` the resolved parent directory once and compare final segments byte-for-byte.

What IMP-1 never enumerates is the failure mode of *that* `readdir` call itself. `FileSystem.readdir` (`docs/spec_topics/pi-integration-contract/host-interfaces-services.md` line 100) is declared to reject with the same Node-style `.code` shape as `readText`, so the parent-directory enumeration can fail with `EACCES`/`EPERM` (no list-permission on the directory), `ENOTDIR` (a path component on the way to it is not a directory), `ENOENT` (the parent directory itself does not exist), or a generic I/O error. Under IMP-1's three enumerated arms, none of these match: the implementer never reaches the byte-exact comparison, so arm (2) does not apply; nothing is determined about a matching entry's readability, so arm (3) does not apply; and the spec is relative, so arm (1) does not apply. The opening sentence of IMP-1 does say "loom 1.0's load pipeline treats a throw from `resolve` as a resolution failure" — which an implementer reading charitably could use to route the `readdir` rejection through the same diagnostic — but the *taxonomic definition* that follows, and the matching `loom/load/unresolvable-warp-path` registry row (`docs/spec_topics/diagnostics/code-registry-load.md`), both enumerate the failure cases as a closed list. A second implementer reading the same text could equally plausibly conclude that `readdir` rejection is a host-failure outside the resolver's `MUST … by throwing` contract, leaving the load-pipeline outcome undefined.

Discovery already addresses the analogous concern at the *source-walk* level — the "clean leaf-`ENOENT`" rule in `docs/spec_topics/discovery/discovery-sources.md` classifies `EACCES`/`EPERM`/`ENOTDIR` mid-walk as unreadable rather than missing. IMP-1 needs the same coverage at the *import-resolution* level so that the unresolvable arm is exhaustive over the `FileSystem.readdir` rejection surface IMP-1 itself directs implementers to use.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `9c90227` 2026-06-06 — `pi-loom spec: resolve ".warp import resolution has no diagnostic code or resolver failure contract"`
- `9aa73b1` 2026-06-07 — `pi-loom spec: resolve "Filesystem case-sensitivity for .warp basenames and invoke/tools containment"`

**History:** `git log -S "IMP-1" -- docs/spec_topics/imports.md` returns `9c90227` as the sole IMP-1-introducing commit. That commit's text defined the unresolvable set as "non-relative … or when relative resolution points at a `.warp` path that does not exist or is not readable" — a broad-enough phrasing that "not readable" arguably absorbed any `FileSystem` rejection on the way to the file. Follow-up commit `9aa73b1` (verified via `git log -S "byte-for-byte" -- docs/spec_topics/imports.md`) replaced that phrasing with the current three-arm enumeration: byte-for-byte final-segment match against `readdir` output, plus a specific "matching entry exists but is not readable — `EACCES` / `EPERM`, or a broken symlink on that entry" arm. The specialization sharpened the case-sensitivity model and the byte-exact comparison surface (the bug it was resolving) but narrowed the fs-error mapping from the original blanket "not readable" to a single entry-level arm, dropping the parent-directory enumeration failure case in the process. The defect is therefore the *interaction* between the two commits — the second commit's tightening of a phrasing the first had left deliberately broad.

## Solution Space

**Shape:** single

### Recommendation

Extend IMP-1's enumeration of "unresolvable" with a fourth arm covering parent-directory enumeration failure, and update the matching `loom/load/unresolvable-warp-path` row in `docs/spec_topics/diagnostics/code-registry-load.md` so the trigger column stays in lock-step. Concretely, after the existing "matching entry exists but is not readable" sentence, add a sentence of the form:

> A failure to enumerate the resolved parent directory itself — `FileSystem.readdir` rejecting with `EACCES` / `EPERM` (no list-permission on the directory), `ENOTDIR` (an intermediate component is not a directory), `ENOENT` (the parent directory does not exist), or any other I/O failure — is likewise unresolvable.

Then add the parallel clause to the registry row's trigger column (which currently ends "…or the byte-exact matching entry exists but is not readable. The importing file is not registered.") so the closed-set enumeration matches.

Implementer-relevant edge cases:

- `ENOENT` on the parent directory specifically (as opposed to `ENOENT` returned for an entry inside a successfully enumerated parent) should land in the same arm; do not mint a separate diagnostic for it. The user-relevant fact is "the import cannot be resolved", and the diagnostic message template `cannot resolve .warp import '<path>'` already accommodates the case.
- The non-normative implementation note further down IMP-1 should be left to its existing wording — it already says the resolver "enumerates the resolved parent directory once via `FileSystem.readdir`", which is consistent with the added arm; no edit there.
- Resist the temptation to mint a new code such as `loom/load/warp-path-enumeration-failed`. The user-visible action is identical to the other arms ("fix the path or its permissions"), the diagnostic-shape gate (`H5a`) and the V7b registry close-out cost a code, and the placeholder-rendering rules for `<path>` apply unchanged.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on what happens when the resolved parent directory cannot be enumerated. One catches the `readdir` rejection inside the resolver and rethrows so IMP-1's throw-equals-unresolvable opening sentence routes it to `loom/load/unresolvable-warp-path`; the other lets the rejection propagate as an uncaught load-pipeline error (because it does not match any enumerated arm of "unresolvable"), aborting the entire load pass rather than failing just the importing file. The user-visible difference is a single bad permission on a directory containing a `.warp` either takes down one file or takes down the project's whole load pass.
