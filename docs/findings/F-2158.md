## Problem

FN-3 (`functions.md#fn-3`) specifies loom return-type inference as a multi-component algorithm: it reconciles the tail-expression type (or the implicit `null` of an empty-tail body, per FN-4) with the operand type of every syntactically-present `return expr` under the least-upper-bound discipline, then conditionally wraps the result in `Result<T, QueryError>` when any `?` appears or any contributing operand is itself `Result`-typed. The page states the rule entirely in prose and supplies no worked vectors mapping a concrete body to its inferred type, so a conformance tester cannot mechanically determine the expected type for a novel body and two implementers reconstructing the wrapping/LUB interaction have no shared oracle to converge on. The corpus already establishes a normative-vector convention for comparably mechanical surfaces (the placeholder-rendering vectors); FN-3 lacks the equivalent treatment.

**History:** The loom return-type inference rule first entered the corpus in 8eea523, which never carried test vectors. 08886d9 added early-`return` operand reconciliation and b97fab3 coined the FN-1..5 labels (including FN-3), each widening the algorithm's interacting-component surface without ever introducing accompanying vectors. The defect is co-extensive with the rule's existence: the inference surface has lacked normative test vectors at every revision since it was written.

## Solution approach

Add a normative, conformance-binding `(body → inferred type)` test-vector table to FN-3 in `functions.md`, covering the wrapping/LUB edge interactions — including the `?`-only tail, a `Result`-typed operand mixed with a plain tail, the empty-tail `null` and `Err`-only `Result<null, …>` cases (referencing FN-4), and the no-common-upper-bound diagnostic outcome `loom/parse/return-no-common-type`. Mark it so `V3d-T` can cite individual rows as test cases.
