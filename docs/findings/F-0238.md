## Finding

`expressions.md` §Other arithmetic specifies two *distinct* non-panic arithmetic dispositions. The first: division and modulo by zero produce IEEE-754 `Infinity` / `-Infinity` / `NaN` and do not panic. The second, separate property: an `integer`-typed result of binary `-`, `*`, `%`, or unary `-` whose computed magnitude exceeds the safe-integer bound (`|value| > 2^53 - 1`) is computed in double precision, silently loses precision, **does not panic**, and **retains the static `integer` type** rather than widening to `number`. Both are reinforced by `diagnostics/code-registry-runtime.md`, which states integer overflow is "deliberately not in loom 1.0.0's panic catalogue."

The authoritative test leaf `V3a-T`'s `Other arithmetic` bullet — mirrored verbatim in `V3a` — asserts only "`integer ⊑ number` widening and div/mod-by-zero produce `Inf`/`NaN` without panic." It carries no vector that drives an out-of-range integer arithmetic result. `V3a`'s **Ships when** likewise asserts "arithmetic-widening behaviour" and div/mod, but not the overflow disposition.

`V3a` is the matrix-mapped closing leaf for the `expressions.md` (EXPR) code-keyed area (`coverage-matrix.md`). Because the EXPR area is code-keyed with no numbered REQ-ID gating, the missing vector is not caught by any closing-gate automation: the area's coverage is satisfied while this specified property goes unexercised. An implementation that panics on integer overflow, or that widens the overflowed result to `number`, would pass the leaf's tests as written.

**History:** The `Other arithmetic` Tests bullet in both `V3a-T-expression-evaluator.md` and `V3a-expression-evaluator.md` was created in `c6a664e` — the commit that first authored these leaves — and has covered only div/mod-by-zero from inception (`git log -S 'div/mod-by-zero produce'` on both files returns only `c6a664e`). No later edit narrowed the bullet; the gap is original to the leaf authoring against a spec that already carried the property.

## Solution Space

**Shape:** single

### Recommendation

Extend the `Other arithmetic` Tests bullet in `docs/plan_topics/V3a-T-expression-evaluator.md` (the authoritative tests leaf, landed first per the per-phase TDD ritual) and the matching bullet in `docs/plan_topics/V3a-expression-evaluator.md` to cover the integer-overflow disposition, and extend `V3a`'s **Ships when** to assert it.

Add a vector that evaluates an `integer`-typed binary `-`, `*`, or `%` (or unary `-`) whose result magnitude exceeds `2^53 - 1` and asserts:

- no `loom/runtime/*` panic fires;
- the result is the IEEE-754 double-precision value (silent precision loss is expected, not an error); and
- the result retains the static `integer` type and does **not** widen to `number`,

citing `expressions.md` §Other arithmetic as the source. `/` is outside this rule (it always produces `number`); the vector must use one of `-`, `*`, `%`, or unary `-`.

Edge case for the implementer: the type-retention half (overflowed `integer` stays `integer`) is the part most likely to be implemented wrong, since the natural instinct is to widen on overflow — the vector should assert the static type of the result expression, not only the absence of a panic.

## Consequence

**Severity:** advisory

A distinct normative property (integer overflow does not panic and retains the static `integer` type) has no asserting vector, so the test net would not catch an implementation that panics on out-of-range integer arithmetic or that widens the result to `number`. The EXPR coverage row closes vacuously with respect to this property because the area is code-keyed and no gate exercises it. The implementer still has guidance — `V3a`'s **Adds** prose names "no integer-overflow panic" — which is why this degrades validation rather than forcing divergence.
