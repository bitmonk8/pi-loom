## Finding

The per-type stringification `Tests.` bullet in both `V6d` (`docs/plan_topics/V6d-system-interpolation.md`) and its paired tests leaf `V6d-T` (`docs/plan_topics/V6d-T-system-interpolation.md`) ends with a scope-limitation clause stating that row-level correctness stays owned by `V13a` and that this leaf only routes the resolved value through the shared renderer. That clause is a descriptive statement of what the leaf does *not* verify and which sibling owns the excluded behaviour — it asserts no test.

Per `conventions.md` §Leaf format, a `Tests.` bullet is one assertion per REQ-ID the leaf claims to close; the descriptive "what the leaf introduces / what it excludes" prose is the `Adds.` field's job. A boundary statement about ownership belongs in `Adds.` as a scope qualifier, not folded into the tail of a test assertion where it dilutes the bullet's testable claim. The same misplacement appears verbatim-in-intent in both members of the leaf pair.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 3625ee0 — pi-loom plan: resolve "system: interpolation per-type stringification not witnessed" (2026-06-11, Thomas Andersen)
**History:** The scope-limitation clause did not exist in either `V6d` or `V6d-T` before commit 3625ee0. That commit added the new per-type stringification `Tests.` bullet to both leaves and, in the same edit, appended the "row-level correctness stays owned by `V13a`" boundary sentence to the tail of that bullet. The misplacement therefore entered the corpus wholesale in a single later edit, not at plan inception — the clause was authored directly into `Tests.` rather than into `Adds.`.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V6d-system-interpolation.md`, strike the trailing scope sentence from the per-type stringification `Tests.` bullet — the text `Row-level correctness stays owned by \`V13a\`; this leaf only routes the resolved value through the shared renderer.` — and carry the same ownership/scope statement into that leaf's `Adds.` field as a scope qualifier (e.g. note that `V6d` adds only the `system:` resolve-then-route path and that row-level rendering correctness remains owned by `V13a`).

Apply the parallel edit in `docs/plan_topics/V6d-T-system-interpolation.md`: strike its variant trailing sentence — `Row-level correctness stays owned by \`V13a\`; these vectors witness only that the \`system:\` resolve-then-stringify path feeds each type into the shared renderer.` — from the stringification `Tests.` bullet, and reflect the same scope boundary in that leaf's `Adds.`.

Keep `V13a` as the named owner in the relocated `Adds.` prose; `V13a` is already an existing `Deps.` edge on both leaves, so no new dependency or leaf is introduced. The spec is read-only for this fix — no `spec_topics/*.md` edits.

## Consequence

**Severity:** cosmetic

The clause is descriptive organisation only; relocating it changes neither what `V6d`/`V6d-T` test nor what the implementer builds. Left as-is it slightly muddies the testable claim of the stringification bullet by appending a non-asserting ownership statement, but no implementer diverges and no coverage gate is affected.
