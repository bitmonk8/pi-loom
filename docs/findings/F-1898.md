## Finding

`H1a`'s **Adds** declares `minimatch` in loom's runtime `dependencies` "as the glob matcher engine for `pi.looms` resolution — the `!`/`+`/`-` override ordering and exact-path operand treatment — grounded in `discovery/package-and-settings.md` §DISC-5". This sentence attributes the entire `!`/`+`/`-` override behaviour to the library, but the spec it cites draws a sharper line.

DISC-5 pins the matcher engine and the override ordering as loom's own concern, *not* minimatch's: "The matcher engine and the `!`/`+`/`-` override ordering are pinned here rather than deferred to Pi, because the extension resolves these patterns itself … this paragraph pins loom's *use* of that matching, not minimatch's glob dialect, which the package owns." Minimatch supplies only the glob dialect used to test plain-include and `!`-exclude patterns against a path. The four-class precedence (plain → `!` → `+` → `-`, with `-` taking final precedence over a `+` re-admission) is loom's own layered algorithm. Furthermore DISC-5 states the `+`/`-` operands "are exact paths (compared post-`~` expansion against the relative path, the basename, and the POSIX-normalised absolute path, mirroring Pi's `matchesAnyExactPattern`), not glob patterns" — so the exact-path operand treatment is explicitly *not* minimatch matching at all.

The H1a prose therefore conflates three distinct things — minimatch's glob dialect (library-owned), the override ordering (loom-owned), and the exact-path `+`/`-` operand treatment (loom-owned, via Pi-style exact matching) — into a single clause crediting minimatch with all of them. `V10b` (which closes DISC-5/6) carries the same shorthand in its Adds ("minimatch with `!`/`+`/`-` ordering"), though it at least spells the order out inline.

## Issue introduction

**Verdict:** present-since-introduction
**Introducing commit:** `8279cb15cff893b288c3d02afcab893426a66797` (2026-06-11) — "pi-loom plan: resolve \"minimatch runtime dependency required by V10b is never provisioned\""
**History:** `git log -S` over the H1a leaf shows the clause "the `!`/`+`/`-` override ordering and exact-path operand treatment" first appeared in `8279cb1`, the same commit that introduced the `minimatch` dependency to H1a's **Adds** to close an unrelated provisioning gap. The mischaracterisation has existed unchanged since that single authoring; no later commit touched it. The defect was introduced incidentally by the prose written to justify the new dependency, not by a subsequent edit or a multi-commit interaction.

## Solution Space

**Shape:** single

### Recommendation

In `H1a`'s **Adds**, rewrite the `minimatch` clause so the library is credited only with the glob dialect, and the override ordering and exact-path operand treatment are attributed to loom's own layer per DISC-5. Replace the current text —

> `minimatch` belongs in loom's own `dependencies` as the glob matcher engine for `pi.looms` resolution — the `!`/`+`/`-` override ordering and exact-path operand treatment — grounded in `discovery/package-and-settings.md` §DISC-5 (consumed in `V10b`)

— with text conveying that `minimatch` is the glob-dialect engine loom uses to match plain-include and `!`-exclude patterns, while the `!`/`+`/`-` four-class override ordering and the exact-path `+`/`-` operand treatment (Pi-style `matchesAnyExactPattern`, not glob matching) are loom's own layer pinned in `discovery/package-and-settings.md` §DISC-5, consumed in `V10b`. Keep the §DISC-5 anchor and the `V10b` consumption pointer.

The spec (`package-and-settings.md` §DISC-5) is read-only for this fix — it already states the split correctly; do not edit it. If `V10b`'s Adds shorthand is touched in the same pass, mirror the same attribution there ("loom's `!`/`+`/`-` override ordering layered over minimatch glob matching"), but the load-bearing edit is the H1a clause.

## Consequence

**Severity:** advisory

The dependency declaration (`minimatch` in runtime `dependencies`) is correct regardless, so nothing fails to build. The risk is a guidance gap: an implementer reading H1a's clause could expect minimatch to provide the `+`/`-` precedence and exact-path operand handling, when those are loom's own layer; the authoritative DISC-5 contract that V10b cites already states the split, so a careful implementer recovers it, but the plan prose contradicts its own cited source.
