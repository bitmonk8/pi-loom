## Finding

The glossary `operator` entry (`docs/spec_topics/glossary.md`) closes with a two-item `See:` list:

> See: [Pi Integration Contract — System notes](./pi-integration-contract.md), [Pi Integration Contract — Runtime event channel](./pi-integration-contract.md).

The second link's text names a specific sub-page — *"Pi Integration Contract — Runtime event channel"* — but its URL resolves to the PIC hub `./pi-integration-contract.md`, not to the sub-page `./pi-integration-contract/runtime-event-channel.md` where that content actually lives. A reader following the link lands on the hub's table-of-contents and must navigate one more hop to reach the section the link text promised.

The glossary already demonstrates the correct target: the `occurrence` vs. `origin` entry links the same content as `[Pi Integration Contract — Runtime event channel](./pi-integration-contract/runtime-event-channel.md)`, and the PIC hub's own Contents list points `Runtime event channel` at `./pi-integration-contract/runtime-event-channel.md`. The `operator` entry's link is simply stale against the sub-page split.

This is the same defect pattern as the `always-log set` entry's single `See:` link (tracked separately); here it occurs inside the `operator` entry's multi-item `See:` list. The fix is confined to the Runtime-event-channel link; the sibling `System notes` link in the same list is out of scope for this finding (there is no dedicated `system-notes` sub-page, so its hub target is not the same defect).

## Issue introduction
**Verdict:** introduced-by-later-edit (size-cap sub-page split; never swept here)
**Introducing commits:** 52219c4 — "Address all 41 spec-review findings" (2026-05-06) authored the `operator` entry with the hub link; f5e89f4 — "docs: enforce 100KB hard / 30KB recommended size cap on spec set" (2026-06-04) extracted the Runtime-event-channel content into its own sub-page, making the existing link stale.
**History:** `git log -L` over the `operator` entry shows the `See:` list has pointed both links at `./pi-integration-contract.md` since the entry was first introduced in 52219c4. At that time the link was correct: the Runtime-event-channel content lived inline in the monolithic `pi-integration-contract.md`. The size-cap split f5e89f4 relocated that content to `./pi-integration-contract/runtime-event-channel.md` without repointing the glossary `See:` links, which is the moment the link text and its URL diverged. A later edit (d75f2a8) corrected the sibling `occurrence` vs. `origin` entry to the sub-page URL but did not sweep the `operator` entry, so the stale link survives.

## Solution Space

**Shape:** single

### Recommendation

In the `operator` entry's `See:` list, change the Runtime-event-channel link's URL from `./pi-integration-contract.md` to `./pi-integration-contract/runtime-event-channel.md`, leaving the link text unchanged:

> See: [Pi Integration Contract — System notes](./pi-integration-contract.md), [Pi Integration Contract — Runtime event channel](./pi-integration-contract/runtime-event-channel.md).

Edit only the Runtime-event-channel link. Leave the `System notes` link untouched: there is no `system-notes` sub-page, so its hub target is not part of this defect.

## Consequence

**Severity:** cosmetic

An implementer or auditor following the `operator` entry's Runtime-event-channel link lands on the PIC hub instead of the named sub-page and must navigate one further hop. No behaviour diverges and no implementation outcome changes; the cost is reader navigation time and an internally inconsistent citation style across glossary entries.
