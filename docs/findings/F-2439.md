## Problem

BNDR-12's normative MUST is framed around a "second call" that never occurs in loom 1.0 — the runtime issues exactly one binder call per slash invocation — so its load-bearing no-stale-`bind_context`-snapshot clause cannot be falsified by any 1.0 conformance test. The "re-entrant per loom turn" phrasing is itself ambiguous between concurrency and statelessness. The result is a guidance gap: an implementer still builds a working system (fresh construction per call), but a conformance author cannot write a passing/failing test for the stated requirement.

## Solution approach

Rewrite the BNDR-12 MUST as the observable single-call statelessness property — on each binder call the binder's input record (parameter table, raw slash text, optional session-context block) and the resolved binder-model handle are constructed afresh from current inputs with no state cached from any prior binder call or from loom load — replacing the "re-entrant per loom turn" wording so the requirement is testable on the one 1.0 call. Move the "second call serves a non-stale snapshot" property and the automatic-context-escalation rationale into the adjacent non-normative seam note as forward-compat justification, so no normative MUST depends on an unreachable second call.
