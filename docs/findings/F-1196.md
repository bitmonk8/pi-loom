## Problem

`placeholder-rendering-b.md` §8's **Rule** paragraph carries a closed enumeration of the `<error>` placeholder occurrences that bind a *caught thrown value* and therefore must run through the §6 [underlying-error coercion](#underlying-error-coercion) before the first-line truncation rule applies. The enumeration names two rows but omits a third caught-throw row: `loom/host/session-shutdown-teardown-step-failed`, whose `<error>` interpolates the value caught by PIC's **Per-step isolation** `try`/`catch`. An implementer wiring the §8 renderer from this enumeration alone treats that row's `<error>` as §8's host-supplied-string default class and skips the coercion, so a non-Error throw renders the literal `undefined` or trips a synchronous `TypeError` in first-line truncation.

## Issue introduction

**Verdict:** present-since-section-inception
**Introducing commits:** `bf85092` — *pi-loom spec: resolve "non-Error-throw coercion + session_start pi.getCommands() failure contract"* (2026-06-06)
**History:** Commit `bf85092` introduced the §6 **Underlying-error coercion** paragraph and the §8 **Rule** sentence enumerating the caught-throw `<error>` rows. That same commit names exactly the two rows (`loom/load/extension-bootstrap-failed`, `loom/runtime/active-set-restore-failed`) the present-day text names; `loom/host/session-shutdown-teardown-step-failed` already existed in `code-registry-host.md` at that commit (it is one of the three rows whose row prose `bf85092` simultaneously updated to link to the new `#underlying-error-coercion` anchor — diff hunk on the teardown-step-failed row replaces a `String(error)` clause with the cross-reference to the coercion rule). The enumeration in §8 was therefore incomplete from the moment it was authored: the row whose prose `bf85092` was reconciling against the new coercion rule was the one omitted from §8's enumeration of rows that use it. No subsequent commit has touched the §8 enumeration sentence (`git log -S "extension-bootstrap-failed and " -- docs/spec_topics/diagnostics/placeholder-rendering-b.md` returns only `bf85092`).

## Solution approach

Extend the §8 **Rule** paragraph's caught-throw enumeration in `placeholder-rendering-b.md` to name the third caught-throw `<error>` row, `loom/host/session-shutdown-teardown-step-failed`, alongside the two it already lists. That row's `<error>` binds a value caught by PIC's **Per-step isolation** `try`/`catch`, placing it in the §6 [underlying-error coercion](#underlying-error-coercion) class rather than §8's host-supplied-string default.
