## Finding

`placeholder-rendering-b.md` §8 carries a closed enumeration of the §8 `<error>` placeholder occurrences that bind *caught thrown values* and therefore must run through the §6 underlying-error coercion before the first-line truncation rule applies:

> When the §8 placeholder binds a *caught thrown value* — `<error>` on `loom/load/extension-bootstrap-failed` and `loom/runtime/active-set-restore-failed` — the value is first coerced to the underlying string per the [underlying-error coercion](#underlying-error-coercion) in §6 above, then rendered by the first-line truncation rule.

The enumeration names two rows, but a third row's `<error>` is also a caught-throw placeholder: `loom/host/session-shutdown-teardown-step-failed`, whose *Message* template `session_shutdown teardown step <step> failed at <call>: <error>` interpolates the value caught by the per-step `try`/`catch` mandated by PIC's **Per-step isolation** paragraph (see the row's *Trigger* prose in `code-registry-host.md`). The row prose now (since the §6 coercion was introduced) links its own `details.error` description out to `#underlying-error-coercion` directly, so a producer that reads the registry row gets the right contract — but the §8 closure prose at the placeholder-rendering page implies the row's `<error>` is in §8's host-supplied-string default class, not the caught-throw class, and an implementer wiring the renderer from §8 alone would therefore omit the coercion for this row.

The fix is a one-line edit in §8: extend the caught-throw enumeration to name all three rows.

## Solution Space

**Shape:** single

### Recommendation

Extend the §8 caught-throw enumeration in `placeholder-rendering-b.md` to name all three caught-throw `<error>` rows. The minimal edit replaces the inline enumeration

> `<error>` on `loom/load/extension-bootstrap-failed` and `loom/runtime/active-set-restore-failed`

with

> `<error>` on `loom/load/extension-bootstrap-failed`, `loom/runtime/active-set-restore-failed`, and `loom/host/session-shutdown-teardown-step-failed`

in §8's **Rule** paragraph (the sentence that begins "When the §8 placeholder binds a *caught thrown value*…"). Nothing else in §8 or §6 needs to change: the §6 coercion rule already covers the third row's payload class, the row prose in `code-registry-host.md` already links to `#underlying-error-coercion`, and the §8 anchored partial-match build-time prohibition already covers the row because its *Message* template contains the `<error>` token.

Two edge cases the implementer must hold:

- `V7c`/`V7c-T` should add a §8 test vector for `loom/host/session-shutdown-teardown-step-failed` under a non-Error throw (`throw "boom"`) and under the `String(v)`-throws fallback (`throw { toString() { throw new Error("nope") } }`), so the renderer is exercised on both arms of the §6 coercion through the third row. The existing host-incompatible / active-set-restore vectors do not cover this row, and a row in the enumeration without a vector falls back into the same drift that produced this finding.
- The §8 enumeration is now a closed three-element list. Any future row whose `<error>`/`<message>` interpolation binds a caught throw must extend the enumeration in the same commit that adds the row — analogous to the `<step>`/`<call>` closed-set discipline §7 already imposes via PIC's **Per-step isolation** paragraph. A non-normative note to that effect on §8's enumeration sentence is optional but consistent with §7's posture.

## Consequence

**Severity:** correctness

Two implementers reading §8 strictly will diverge on `loom/host/session-shutdown-teardown-step-failed`'s rendered `<error>` for non-Error throws: one applies §6's coercion and produces a stable string; the other treats the host-supplied-string default as governing and either reaches for an `.message` that is `undefined` (rendering the literal string `undefined`) or hits a synchronous `TypeError` inside the renderer's first-line truncation when the caught value is not string-shaped. The producer-site discipline in `code-registry-host.md` keeps `details.error` correct on the wire, but the rendered `message` byte form is what `DIAG-4` and the closing gate enforce, so the divergence is observable in the rendered system-note channel.
