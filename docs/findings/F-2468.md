## Problem

The prompt-mode *Error detection* probe (`#prompt-mode-error-detection`) asserts that every non-synchronous-throw failure of a driven user turn surfaces as a trailing `assistant` message with `stopReason: "error"`, and PIC-51 maps only that member to a transport `Err`. Any other terminator — `stopReason: "length"` (context overflow), a content-filter stop reason, or an absent/empty trailing `assistant` message — falls through to PIC-53 and is extracted as a successful `Ok(string)` (often the empty string), silently reporting success for a turn that actually failed. The sibling off-session path in `provider-error-mapping.md`'s *Stop-reason classification* (`#stop-reason-classification`) already classifies these non-`"error"` terminators, but the driven-turn probe ignores them. Unlike the three adjacent consumption-postures on the same page, this load-bearing failure-universe claim has no named presupposition block and no editorial-review hook in the version-bump checklist.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commits:** `d37c563` (2026-06-05, `pi-loom spec: resolve "Prompt-mode errorMessage probe unreachable from specified surface"`)
**History:** The prompt-mode error-detection probe was reworked twice. At `8c6cfec` (2026-05-07, `pi-loom spec: resolve "pi.sendUserMessage returns void; prompt-mode transport-error detection path is mis-described"`) the probe moved from a (mistaken) throw/reject-of-`sendUserMessage` surface to reading `AgentSession.errorMessage`, mapping any non-empty value to `transport` — already an exhaustive single-field failure test, but keyed on `errorMessage` non-emptiness. `d37c563` then migrated the probe to read the trailing `assistant` message's `stopReason` / `errorMessage` from the `ReadonlySessionManager` surface and narrowed detection to `stopReason: "error"`, adding the "opaque apart from the `"error"` member it keys on" wording — this is the present concrete form of the defect and the point at which the `"length"` / content-filter blind spot was introduced. Two days later `ef4cfb8` (2026-06-07, `pi-loom spec: resolve "Stop-reason → QueryError variant mapping is undefined"`) added the *Stop-reason classification* (`length` → overflow, other → transport) to `provider-error-mapping.md` for the off-session `complete()` path, formalising the very non-`"error"` terminators the driven-turn probe ignores — but never propagated that classification to the driven turn. The sibling consumption-postures were authored separately (`3683345`, 2026-06-06, commit-ordering; `1ca13ec`, no-rejection item `(al)`) without ever adding a posture for the failure-universe→`stopReason: "error"` mapping. The gap has carried from `d37c563` through the present.

## Solution approach

Broaden the *Error detection* probe at `#prompt-mode-error-detection` so the trailing `assistant` message's `stopReason` is classified against `provider-error-mapping.md#stop-reason-classification` rather than keyed on the single `"error"` member, and give the probe an explicit disposition for an absent trailing `assistant` message. Clarify PIC-53's `Ok(string)` extraction precondition so it runs only after that broadened classification passes. Add a named presupposition block and an editorial-review checklist hook covering the failure-universe → terminal-stop-reason mapping, matching the three sibling consumption-postures on the page.

## Solution constraints

- Reuse only the stop-reason arm of `provider-error-mapping.md#stop-reason-classification`; do not reproduce its HTTP-status arm on the driven-turn path — the HTTP-status arm is not reachable from the `ReadonlySessionManager` read surface.
