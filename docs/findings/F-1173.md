## Finding

The Future Considerations hub introduces a four-bucket taxonomy of deferred work. Category 2 ("Surface extensions — loom 1.0 leaves a seam") and category 3 ("Model-level changes — no loom 1.0 seam expected") are distinguished by a single qualitative phrase: a deferred feature falls into category 3 when it "changes the runtime value model, evaluation model, or tool-result contract **enough that** loom 1.0 is not expected to anticipate them." The threshold word "enough that" is not operationalised anywhere on the page or in `model-changes-and-non-goals.md` / `surface-extensions.md`, and no test is given for which side of the line a candidate feature sits on.

The classification is load-bearing because it determines whether loom 1.0 must reserve and document a forward-compatibility seam now (category 2 requires one — every category 2 bullet on `surface-extensions.md` carries an *Anchored at:* line pointing at a topic-page MUST; the seams are counted in the "Forward-compatibility seams" tally on `overview-and-orientation.md#scope`) or whether it may ship nothing and accept that adoption will require a migration (category 3). Two reviewers could read the same candidate feature — for example "Richer untyped-query return shape", currently listed in category 3 — and one could argue it ships under the existing options-struct seam (category 2) while the other accepts the current placement, with neither able to point at an objective test to resolve the disagreement.

The downstream effect is that authors editing Future Considerations have no rule to apply when they add a new entry, and reviewers have no rule to apply when they challenge a placement. Misclassification has asymmetric cost: a category 3 item that should have been category 2 ships loom 1.0 without the seam it actually needs, which is exactly the failure mode the taxonomy exists to prevent.

## Solution Space

**Shape:** single

### Recommendation

Replace the "enough that loom 1.0 is not expected to anticipate them" qualifier on category 3 with an operational test stated in terms of loom 1.0's existing carriers. Specifically, define category 3 as: *a deferred feature whose adoption would alter the return type, routing class, or normative behavioural contract of a construct already expressible in loom 1.0, in a way that no extant loom 1.0 seam (options struct, `QueryError` discriminator, `Resolver` interface, frontmatter unknown-key policy, or other carrier listed in `surface-extensions.md`) can absorb without rewriting call sites.* Equivalently — and more useful as a triage rule — *if a candidate feature can be added by extending an inventoried loom 1.0 seam, it is category 2; if no such seam exists or the change would force the type of a return value or the dispatch class of an existing construct to change, it is category 3.*

Authoring notes the implementer should observe:

- Re-audit the current category 3 list against the new test. The "Richer untyped-query return shape" bullet self-identifies as widening the `Result<string, QueryError>` Ok payload to a union, which under the new test is correctly category 3 (return type change). The remaining bullets (first-class loom values, concurrency primitives, streaming partial results, structured tool outputs, `mut` parameters, `break expr`, match guards / rest patterns, non-decimal literals, integer-division operator, additional stdlib methods) should each be re-checked once against the operational test; any that turn out to be expressible via an existing seam move to category 2 and acquire an *Anchored at:* line.
- Keep the category 2 vs category 3 axis distinct from the category 3 vs category 4 axis already pinned in the intro paragraph (non-goals are scope dispositions, not model-level changes). The new threshold replaces only the 2/3 boundary.
- The "additional string and array stdlib methods" bullet on `model-changes-and-non-goals.md` is a known edge case: its own prose says the method surface is "non-breaking by design" because unknown methods are parse-time errors, which under the new test reads as "no extant seam absorbs the addition" (each new method is a closed-set extension) and is therefore correctly category 3. Confirm this reading when re-auditing rather than leaving it implicit.

## Consequence

**Severity:** advisory

A future author or reviewer cannot mechanically place a deferred feature into category 2 vs category 3, so placements rest on individual judgement and can drift. The asymmetric failure is a category-3 placement of an item that actually needs a category-2 seam: loom 1.0 then ships with a missing forward-compatibility carrier and a post-loom 1.0 implementation of that feature requires a migration that a clearer threshold would have avoided. Implementers of loom 1.0 itself are not directly blocked — the existing bullets are placed and seams are anchored — so this is a guidance gap rather than a correctness defect.
