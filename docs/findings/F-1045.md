## Finding

Three independent defects in the step-2 editorial-review checklist's recording and merge-gate prose. Each can be fixed without touching the other two; the original groups them only because they share a file.

1. **Item (ah) has no recording path for a half-completed audit.** Item `(ah)` (`#bump-checklist-pi-on-process-global`) requires two independently falsifiable halves to be audited together: (i) `pi.on` events are delivered to every subscribed extension regardless of which session is active, and (ii) no event payload carries a per-session origin field. The item states that a recorded `pass` requires *both* halves to have been covered, that `fail` / `N/A` "reflect the joint result", and that `not-performed` is reserved for skipping the audit entirely. The step-2 preamble's recording vocabulary is `pass` / `fail` / `N/A` / `not-performed`. A contributor who performs half the audit — say, confirms (i) but cannot reach a verdict on (ii) before merge — has no conformant recording option: `pass` is barred, `fail` / `N/A` would mis-attribute a verdict to the un-audited half, and `not-performed` denies the work that was done. Item (e) solves the same shape by carving out atomic sub-outcomes `(e.i)` / `(e.ii)`; item (ah) does not, even though the spec text explicitly calls the halves "independently falsifiable for distinct PIC-18 arguments".

2. **Item (e)'s merge gate has no disposition for an `N/A` second-pass outcome.** When `(e.i)` or `(e.ii)` records `fail`, the contributor adds the recovery serialisation and re-runs the sub-outcome against the loom-side source. The spec says "the bump MAY merge only when that second-pass outcome is `pass` under the pass-evidence rule above". The valid verdict set on the second pass is the same `pass` / `fail` / `N/A` triple, but the merge clause names only `pass`. The `N/A` arm is governed by the named-Pi-mechanism definition, which on the second pass is incoherent (the recovery is loom-side, not a Pi mechanism), so the disposition is presumably "second-pass `N/A` is unreachable" — but the spec does not say so, and a contributor who reads the clause literally cannot tell whether a second-pass `N/A` blocks the merge, clears it, or is itself non-conformant.

3. **Item (e) uses "in the same edit as the bump" where neighbouring obligations say "in the same commit".** Item (e)'s two recovery-serialisation cues both read "in the same edit as the bump". Sibling co-edit obligations in this file and across the bump-procedure pages use both phrasings interchangeably (14 occurrences of "in the same edit" and 3 of "in the same commit" in `version-bump-step2.md` alone; the split is similar in `version-bump-step2b.md` and `version-bump-triggers.md`). The two phrases mean the same thing — a single landed change — but the inconsistency is most visible at item (e), where the recovery-serialisation contract sits next to the merge-gate language that talks about commits.

## Issue introduction

**Verdict:** multi-commit-interaction

**Introducing commits:**
- `c0259ca` (2026-06-07, "pi-loom spec: resolve \"Version-bump procedure: four obligations have no verifiable acceptance criterion\"") — introduced both the `same edit as the bump` phrasing at item (e) and the second-pass merge-gate sentence ("the bump MAY merge only when that second-pass outcome is `pass`"); the merge-gate clause names only `pass` from inception.
- `a62bed0` (2026-06-20, "pi-loom spec: resolve F-0824 — item (ah) bare MUST rebound to SHOULD-gated conditional") — rewrote item (ah)'s "MUST audit both" into "a recorded `pass` requires both halves to have been covered, while a recorded `fail` or `N/A` likewise reflects the joint result; recording `not-performed` for the item — skipping the audit entirely — remains a conformant outcome", which is the wording that creates the partial-audit recording gap.

**History:** The three sub-defects are not the residue of a single edit. The merge-gate `N/A` gap and the `edit`-vs-`commit` mismatch entered together in `c0259ca` when the second-pass re-run and the recovery-serialisation co-edit obligation were first articulated. The `(ah)` partial-audit gap entered later in `a62bed0`, when an earlier finding (F-0824) was resolved by softening the bare MUST into a `pass`-requires-both-halves rule that left no slot for half-coverage. The edit/commit terminology drift is older than either commit and is corpus-wide — it predates the current revisions of these pages and propagated from siblings like `version-bump-step2b.md` and `version-bump-triggers.md`. No single commit can be reverted to remove all three; the durable fix addresses each sub-defect on its own footing.

## Solution Space

**Shape:** multiple

### Option A — Item (ah) partial-audit recording slot

Approach. Either add `(ah.i)` / `(ah.ii)` sub-outcomes on the same footing as item (e)'s `(e.i)` / `(e.ii)` carve-out, or extend the recording vocabulary on item (ah) with a `partial — <which-half> covered, <which-half> not-performed` shape, in either case scoped to (ah) only with a forward-extension hook for any future item that develops the same pattern.

Spec edits.
- `version-bump-step2.md` item (ah): replace the "a recorded `pass` requires both halves to have been covered, while a recorded `fail` or `N/A` likewise reflects the joint result" sentence with either (a) two atomic sub-items `(ah.i)` *process-global delivery* and `(ah.ii)` *no-per-session-origin-marker*, each recording `pass` / `fail` / `N/A` / `not-performed` with a one-line rationale, mirroring item (e)'s `(e.i)` / `(e.ii)` shape; or (b) a `partial` recording variant that names which half was covered.
- *Outcome recording conventions*: if option (a), extend the "same two-sub-outcome shape applies to any future checklist item that carries independently falsifiable sub-checks" sentence to add item (ah) alongside item (e) as a worked example; if option (b), define the `partial` recording shape's rationale grammar.
- `V18c.Ships when.` ("the bump commit message records a per-item outcome for every item `(a)`–`(al)` (item `(e)` recording its two sub-outcomes)"): extend with item (ah)'s two sub-outcomes (option a) or with the `partial` arm (option b).

Pros. Closes a real recording gap; aligns (ah) with the (e) pattern the spec already names as the worked example for independently falsifiable sub-checks.

Cons. Option (a) adds another two-anchor block to a checklist that is already dense; option (b) introduces a recording shape unique to one item.

Risks. None beyond editorial review of the modified item.

### Option B — Item (e) second-pass `N/A` disposition

Approach. State explicitly that a second-pass `N/A` is unreachable, or — preferred — define which side of the merge gate it falls on if it does arise.

Spec edits.
- `version-bump-step2.md` item (e), the merge-gate sentence "the bump MAY merge only when that second-pass outcome is `pass` under the pass-evidence rule above": add a parenthetical or a follow-on clause stating that a second-pass `N/A` is unreachable on this arm (the `N/A — superseded by <named Pi mechanism>` verdict is defined only against a Pi-side mechanism, not against the loom-side recovery serialisation just added), so a contributor whose re-run produces `N/A` has mis-recorded the verdict and the auditor MUST re-record it.

Pros. Removes an unbounded reading of the merge gate; small edit; no plan-leaf surface change beyond `V18c`.Ships-when prose tracking the literal merge-gate phrasing.

Cons. Embeds another conditional in an already-busy paragraph.

Risks. None.

### Option C — Normalise "edit" vs "commit" on item (e)

Approach. Replace "in the same edit as the bump" with "in the same commit as the bump" at item (e)'s two recovery-serialisation occurrences. Do **not** sweep the whole corpus in this edit; both phrasings are in active use across `version-bump-*.md`, the wider sweep is a separate clean-up that would touch ~25 sites, and item (e) is the only locality where the inconsistency is reader-visible against an adjacent merge-gate clause that talks about commits.

Spec edits.
- `version-bump-step2.md` item (e), two occurrences of "in the same edit as the bump" inside item (e)'s recovery-serialisation prose: change to "in the same commit as the bump".

Pros. Two-character edit; restores local consistency with the merge-gate sentence that immediately follows.

Cons. Leaves the corpus-wide `edit`/`commit` inconsistency unaddressed.

Risks. None.

### Recommendation

Apply all three options as **three independent edits**, in the order C → A → B. Option C is a near-zero-surface terminology fix that lands cleanly on the baseline. Option A is a localised recording-shape addition that mirrors an existing worked example. Option B is the smallest of the three and sits inside a paragraph the other two also touch, so resolving it last lets it land on text the previous two edits have already stabilised. Bundling all three into one edit re-creates the bimodal-finding hazard this split is meant to avoid; sequencing them separately keeps each diff small enough for the next reviewer pass to evaluate on its own.

The corpus-wide `edit` / `commit` normalisation sweep is out of scope here — it is a separate editorial finding against `version-bump-step2.md` / `version-bump-step2b.md` / `version-bump-triggers.md` collectively, and would be much larger.

## Consequence

**Severity:** advisory

A contributor running the bump procedure in good faith hits each defect at a different point: the (ah) partial-audit case forces a non-conformant record or an unfounded `not-performed`; a second-pass `N/A` on item (e) (rare but possible if the conservative-posture rule is misapplied) leaves the merge gate underspecified; the `edit` / `commit` mix is editorial drag a reviewer must mentally normalise. None of the three blocks implementation, but each pushes the editorial-review acceptance gate the `V18c` leaf consumes onto contributor discipline that the prose was specifically introduced to eliminate.
