## Problem

In `version-bump-step2.md` step 2(a), the TS error family routing import-specifier re-pinning is labelled "TS230x-family", a numeric-range glob that reads as TS2300–TS2309, yet two of its four enumerated codes (`TS2724`, `TS2614`) live in the TS26xx/TS27xx ranges — a direct contradiction with the codes listed beside it. The trailing "any other TS code in the same family" inherits the same ambiguity (numeric range vs conceptual category). The same defect tokens appear in `audit-wire-and-canary.md`'s `id="audit-infrastructure-failure-handling"` clause. Sibling pages (`version-bump-intro.md` step 1, `version-bump-triggers.md` step 5) already use "module-import-resolution family" / "closed family" phrasing, leaving these two sites as the lone outliers.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `75a9626` (2026-05-09) — *pi-loom spec: resolve "SessionShutdownEvent.reason closed set has no build-time pin against the SDK type"*
**History:** `git log -G "TS230x"` shows `75a9626` as the first commit in which the `TS230x` token appears anywhere in the corpus. That commit coined the phrase in the same paragraph it introduced the enumerated four-code set (`TS2305` / `TS2307` / `TS2614` / `TS2724`), so the mislabel was born together with the surface it claims to cover — not introduced by a later refactor that lost track of the codes. The token was subsequently propagated to `audit-wire-and-canary.md` ("the TS230x category step 1 above already names"). Independent later edits to sibling callsites (`version-bump-intro.md` step 1 and `version-bump-triggers.md` step 5) replaced the numeric label with "module-import-resolution family"/"closed family" phrasing without renaming the original sites — so the corpus drifted internally, leaving `version-bump-step2.md` and `audit-wire-and-canary.md` as the two surviving sites of the original mislabel.

## Solution approach

Rename the `TS230x` label out of `version-bump-step2.md` step 2(a) and `audit-wire-and-canary.md`'s `id="audit-infrastructure-failure-handling"` clause (including the later "TS230x routing carve-out" sentence in the same paragraph), replacing it with the "module-import-resolution family" / "closed family" phrasing that `version-bump-intro.md` step 1 and `version-bump-triggers.md` step 5 already carry. Keep the enumerated four-code set `TS2305` / `TS2307` / `TS2614` / `TS2724` intact at both sites.

## Solution constraints

- The four codes are the closed family at the loom 1.0 TypeScript pin; do not broaden the open arm to "any present/future module-import-resolution TS code" — that recast deletes the build-toolchain-floor editorial-review gate (`host-prerequisites.md#tsc-toolchain-floor`).
