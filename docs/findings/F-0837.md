## Finding

In `version-bump-step2.md` step 2(a), the TS error family that routes import-specifier re-pinning is described as a "TS230x-family module-import-resolution failure (`TS2305`, `TS2724`, `TS2307`, `TS2614`, or any other TS code in the same family)". The `TS230x` label is a numeric-range glob that naturally reads as TS2300–TS2309, yet two of the four enumerated codes (`TS2724` for rename-suggestion, `TS2614` for default-vs-named-import) live in the TS26xx/TS27xx ranges. The trailing "any other TS code in the same family" inherits the same ambiguity: is the open arm a numeric range (TS230x), a broader numeric span covering everything enumerated, or the conceptual category "module-import resolution"?

The conceptual category is what the surrounding prose actually wants — the routing key is the category of failure (a Pi-side rename/drop on the package-root re-export surface that the import statement cannot resolve), not a numeric range — and the rest of the corpus already says so. `version-bump-intro.md` step 1 phrases the same set as "the closed family `TS2305` / `TS2307` / `TS2614` / `TS2724`" with the explicit caveat that the four-code set is the closed family at the loom 1.0 pin and any TypeScript bump that could reshape it is re-confirmed under the build-toolchain-floor editorial-review gate; `version-bump-triggers.md` step 5 likewise says "module-import-resolution family" without the misleading numeric label. The `TS230x` label is the lone outlier and produces a direct contradiction with the codes listed beside it.

The same defect tokens have leaked sideways into `audit-wire-and-canary.md` under the *Infrastructure-failure handling* clause, which now says "a module-import-resolution failure (the TS230x category step 1 above already names — TS2305 / TS2307 / TS2614 / TS2724 and any other diagnostic of the same category)". Whichever fix lands here MUST sweep that callsite in the same commit, otherwise the open-arm wording diverges across siblings and the contradiction multiplies.

**Verdict:** present-since-inception
**Introducing commit:** `75a9626` (2026-05-09) — *pi-loom spec: resolve "SessionShutdownEvent.reason closed set has no build-time pin against the SDK type"*
**History:** `git log -G "TS230x"` shows `75a9626` as the first commit in which the `TS230x` token appears anywhere in the corpus. That commit coined the phrase in the same paragraph it introduced the enumerated four-code set (`TS2305` / `TS2307` / `TS2614` / `TS2724`), so the mislabel was born together with the surface it claims to cover — not introduced by a later refactor that lost track of the codes. The token was subsequently propagated to `audit-wire-and-canary.md` ("the TS230x category step 1 above already names"). Independent later edits to sibling callsites (`version-bump-intro.md` step 1 and `version-bump-triggers.md` step 5) replaced the numeric label with "module-import-resolution family"/"closed family" phrasing without renaming the original sites — so the corpus drifted internally, leaving `version-bump-step2.md` and `audit-wire-and-canary.md` as the two surviving sites of the original mislabel.

## Solution Space

**Shape:** single

### Recommendation

Drop the `TS230x` numeric label from both surviving callsites and replace it with the "module-import-resolution family" phrasing the sibling pages already use, in a single commit.

Concrete edits:

1. In `docs/spec_topics/pi-integration-contract/version-bump-step2.md`, step 2(a), rewrite "a TS230x-family module-import-resolution failure (`TS2305`, `TS2724`, `TS2307`, `TS2614`, or any other TS code in the same family) routed here from step 5 below — including one …" as:

   > a module-import-resolution failure — surfacing as one of the closed family `TS2305` / `TS2307` / `TS2614` / `TS2724` that the pinned TypeScript toolchain (per [Host prerequisites — Build-toolchain floor](./host-prerequisites.md#tsc-toolchain-floor)) emits for a Pi-side rename/drop on the package-root re-export surface, where this four-code set is the closed family at the loom 1.0 pin and any TypeScript bump that could reshape it is re-confirmed under the build-toolchain-floor editorial-review gate — routed here from step 5 below, including one …

   This is the wording `version-bump-intro.md` step 1 already uses; lifting it verbatim into step 2(a) eliminates the contradiction and aligns the three callsites on one canonical form.

2. In `docs/spec_topics/pi-integration-contract/audit-wire-and-canary.md`, *Infrastructure-failure handling*, rewrite "a module-import-resolution failure (the TS230x category step 1 above already names — TS2305 / TS2307 / TS2614 / TS2724 and any other diagnostic of the same category)" as:

   > a module-import-resolution failure (the closed family step 1 above names — `TS2305` / `TS2307` / `TS2614` / `TS2724` — and any other diagnostic of the same category at a future TypeScript pin re-confirmed under the build-toolchain-floor editorial-review gate)

   Same callsite "TS230x routing carve-out" sentence later in the same paragraph: rewrite "The TS230x routing carve-out — a module-import-resolution failure of the TS230x category above originating at the bare-package import …" to drop the `TS230x` token and say "The module-import-resolution routing carve-out — a module-import-resolution failure of the closed family above originating at the bare-package import …".

Edge cases the implementer must watch:

- Keep the "routing keys on the TS error code, not on the message-text quote" sentence in `version-bump-step2.md` step 2(a) intact — that property is what protects the four-code list against TypeScript prose drift and is independent of the label change.
- Do not over-broaden the open arm. The corpus deliberately keeps the four codes as a *closed* family at the loom 1.0 TypeScript pin and routes any reshaping through the [Host prerequisites — Build-toolchain floor](./host-prerequisites.md#tsc-toolchain-floor) editorial-review gate; an edit that recasts the open arm as "any module-import-resolution TS code, present or future" would silently delete that gate.
- The `TS230x` token does not appear elsewhere in the corpus; `grep -rn "TS230x" docs/spec_topics/` should return zero matches after the edit.

## Consequence

**Severity:** advisory

A literal reading of "TS230x-family" excludes `TS2724` and `TS2614`, which are exactly the diagnostics step 2(a) is meant to catch on a rename or a default-vs-named-import drift; a contributor who keys routing on the numeric label rather than the four-code enumeration could (a) misroute a `TS2724` rename failure to step 1's spec-prose-update path instead of step 2(a)'s import-specifier re-pin path, or (b) treat a future TS27xx-coded module-import-resolution diagnostic as out of scope. Both outcomes are recoverable at review time — the failure mode is "wrong fix-routing for one bump" rather than runtime breakage — but the contradiction sits inside a procedure whose whole point is to be mechanically routable without judgement.
