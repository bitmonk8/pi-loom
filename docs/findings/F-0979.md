## Problem

DISC-4's final paragraph in `docs/spec_topics/discovery/discovery-sources.md` restates runtime dispatch consequences owned elsewhere: that Pi exposes no `pi.unregisterCommand` so the loom's `pi.registerCommand` registration survives, that `/<name>` still routes to the loom's slash handler, and that a dispatch of the orphaned `/<name>` returns a fixed superseded system note. That normative content already lives in `drain-state-contract.md#superseded-entry-dispatch` and `registration-steps.md#structural-changes-no-unregister`, so DISC-4 reads as a dispatch-behaviour summary rather than a discovery rule and ownership becomes ambiguous — a future edit to the superseded-note text, the arm-(a) routing, or the registration-survival rationale would have to sweep DISC-4 with no marker indicating it is downstream. The prose is consistent with its sources today; the defect is placement, not contradiction.

## Issue introduction

**Verdict:** introduced-in-commit
**Introducing commits:** `4a10cde` ("pi-loom spec: resolve \"DISC-4 de-registers has no Pi-side mechanism\"", 2026-06-06)
**History:** The DISC-4 final paragraph through commit `f2a82ad` (2026-06-05) and earlier read simply: "the next `session_start` cycle re-evaluates and de-registers the previously-registered loom, emitting the same diagnostic." Commit `4a10cde` replaced "de-registers" with the longer rewrite that (i) clarified the drop is a `LoomRegistry`-side entry-table removal rather than a Pi command-router unregistration, (ii) explained `pi.registerCommand` survival because `pi.unregisterCommand` does not exist, and (iii) introduced the cross-reference to the new `superseded-entry dispatch` paragraph it simultaneously added to `drain-state-contract.md`. That single commit added the cross-reference (good) alongside the behavioural restatement (the defect this finding raises). The subsequent commit `da4bd57` ("DISC-4 collision rule split into two scoped arms") restructured the opening of DISC-4 but left the final paragraph untouched, so it inherits the placement decision from `4a10cde` and has not been revisited since.

## Solution approach

Rewrite DISC-4's final paragraph so it retains the discovery-side drop statement (the `LoomRegistry` entry removal that inbound cross-references depend on) and replaces the post-drop dispatch restatement with a forward-link to `drain-state-contract.md#superseded-entry-dispatch` and `registration-steps.md#structural-changes-no-unregister`. Delete the trailing loss-asymmetry restatement, which is already established higher up in DISC-4.

## Solution constraints

- Out of scope: `drain-state-contract.md` and `registration-steps.md` — DISC-4 owns only the discovery-side drop; the dispatch behaviour and registration-survival rationale stay in those files.
