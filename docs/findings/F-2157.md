## Finding

FRNT-1 (`tool_loop`) in `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md` closes with two sentences:

> loom 1.0 provides no operator-level override for `max_rounds`; the value is per-loom only. Adding a `looms.toolLoopMaxRounds` settings key is deferred per [Future Considerations](../future-considerations.md).

The first sentence is the normative statement an implementer needs: loom 1.0 has no operator-level override, `max_rounds` is per-loom only. The second sentence names a hypothetical future settings key (`looms.toolLoopMaxRounds`) for a release that does not yet exist and announces it as "deferred" ŌĆö future-work commentary embedded inside a normative field contract.

The deferred extension is already catalogued at its canonical home: `docs/spec_topics/future-considerations/surface-extensions.md` carries an **Operator-level `looms.toolLoopMaxRounds` settings override** bullet that states the same per-loom-only fact, names the future key, and cites its carrier. The inline FRNT-1 sentence therefore duplicates content the future-considerations register already owns, and the named key is the kind of unimplemented-release detail that future-considerations exists to hold out of the normative pages.

## Issue introduction
**Verdict:** single-commit introduction
**Introducing commit:** `24a8079` (2026-05-07, "pi-loom spec: resolve 'tool_loop.max_iterations validation rules and diagnostic surface unspecified'")
**History:** The FRNT-1 `tool_loop` prose carried no deferred-override sentence before this commit; the field paragraph ended at "the cap is a ceiling, not a floor." Commit `24a8079` appended the inline deferral sentence to `spec_topics/frontmatter.md` and, in the *same commit*, added the canonical "Operator-level `looms.toolLoopMaxIterations` settings override" bullet to `spec_topics/future-considerations.md`. The inline forward-reference was therefore redundant from the moment it was introduced ŌĆö its canonical home was created in the same change. None of those later commits touched the defect.

## Solution Space

**Shape:** single

### Recommendation

Delete the second sentence of FRNT-1's closing text ŌĆö "Adding a `looms.toolLoopMaxRounds` settings key is deferred per [Future Considerations](../future-considerations.md)." ŌĆö leaving the normative first sentence "loom 1.0 provides no operator-level override for `max_rounds`; the value is per-loom only." in place. The deferred future key remains fully recorded in `future-considerations/surface-extensions.md`, so no information is lost.

## Consequence

**Severity:** cosmetic

A normative field contract carries a duplicate forward-reference to an unbuilt release feature. Nothing diverges at implementation time, but the inline note will drift against the future-considerations register (the two already disagreed on token spelling once) and adds noise to the `max_rounds` contract.
