## Finding

`V16a` and its paired tests leaf `V16a-T` use the phrase **runtime-class ceiling** as a formal category — `CIO-1` arbitrates "#3 over any co-present runtime-class ceiling", and the *Not witnessable* bullet repeats "before any runtime-class ceiling fires" — but neither leaf defines the term or points to where its membership is pinned. The membership criterion lives in the spec: the four-ceiling list in [`overview-and-orientation.md#hard-runtime-ceilings`](../spec_topics/overview-and-orientation.md#hard-runtime-ceilings) tags ceilings #1 (`invoke` depth), #2 (`tool_loop.max_rounds`), and #4 (JSON-document depth) as `*(runtime)*` and ceiling #3 (binder per-class retry budget) as `*(load-time; not on the trichotomy)*`, and `ceilings-3-and-4.md#cio-1` uses the same "runtime-class" adjective. So "runtime-class ceiling" = the three `*(runtime)*`-tagged ceilings (#1/#2/#4), as distinct from ceiling #3 (slash-load).

The gap is that V16a's **Spec** field cites `hard-ceilings.md`, `ceilings-3-and-4.md`, and `ceiling-invariants-and-audit.md` — but **not** `overview-and-orientation.md`, where the `*(runtime)*` / `*(load-time)*` classification actually lives. An implementer reading only V16a's cited topics sees "runtime-class ceiling" used in `CIO-1` (both in the leaf prose and in the spec's `ceilings-3-and-4.md#cio-1`) but has no in-scope page that enumerates which ceilings belong to the class; membership is recoverable only by inference (V16a's own check-site→ceiling-class mapping plus `CIO-1` contrasting the class against #3).

`H7a` also uses the phrase, but qualifies it inline with a concrete instance — "a runtime-class ceiling (the `tool_loop.max_rounds` round boundary, ceiling #2)" — so its usage is self-disambiguating and needs no change.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V16a-ceiling-order-masked.md`, at the first use of "runtime-class ceiling" (the Adds *Input* bullet's check-site→ceiling-class mapping, or the first *Not witnessable* bullet), anchor the term: state that runtime-class ceilings are the three ceilings the spec tags `*(runtime)*` — ceiling #1 (`invoke` depth), ceiling #2 (`tool_loop.max_rounds`), and ceiling #4 (JSON-document depth) — as distinct from ceiling #3 (binder per-class retry budget), tagged `*(load-time)*` and evaluated at slash-load. Cite `../spec_topics/overview-and-orientation.md#hard-runtime-ceilings` as the membership anchor, and add that anchor to V16a's **Spec** field so the classification is in-scope for an implementer reading only V16a's cited topics.

Keep `V16a-T`'s `CIO-1` wording consistent with the definition introduced in `V16a`, per the paired-leaf convention.

`H7a` needs no change — its single use already qualifies the term inline with the concrete ceiling-#2 instance.

Do not target the projection-index `terms.md` (it is regenerated, not authored); the durable home for the definition is the V16a leaf.

## Consequence

**Severity:** advisory

An implementer building the `arbitrate()` seam must know which ceiling-candidates are "runtime-class" to construct the `CIO-1` precedence test, but no in-scope topic enumerates the class; they recover membership by inferring "everything except ceiling #3" from V16a's own mapping. Two implementers would most likely converge on #1/#2/#4, so the seam still ships — the cost is unanchored terminology that forces inference rather than a read.
