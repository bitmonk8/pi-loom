## Finding

The `V9j` leaf names a single deliverable — the mechanism that turns a provider error response into a `QueryError` variant — three different ways within one page. The leaf title (and the plan.md V9 list entry) call it the **"provider-error mapping"**, matching the spec page `provider-error-mapping.md` and its canonical lead term "Provider error mapping". The **Adds** field introduces it as the "provider-error → `QueryError` **classifier**", and the first **Tests** bullet (duplicated verbatim into the paired `V9j-T` leaf) calls it the "provider **classifier**". One concept, three surface names inside the same leaf.

The spec page does use both nouns, but with a distinction the leaf collapses: "mapping" names the rule/table, while "classifier" / "Classifier input surface" names the runtime component that reads the input surface and applies the rule. The leaf uses "mapping" and "classifier" interchangeably for the whole deliverable, so the title/body drift is genuine rather than a deliberate echo of that distinction.

No implementer would build a different artefact because of this; the referents are unambiguous from context. It is a readability/consistency defect, not a correctness one.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** Both `V9j-binder-inference-provider-mapping.md` and `V9j-T-binder-inference-provider-mapping.md` were added in the single plan-authoring commit `c6a664e`; `git log --follow` shows no subsequent edits. The mixed "mapping"/"classifier" wording was present in the leaf files' first and only committed revision, so the inconsistency entered with initial plan authoring rather than from any later edit.

## Solution Space

**Shape:** single

### Recommendation

Use the title/spec-page deliverable name "provider-error mapping" consistently in the leaf body, replacing the "classifier" occurrences:

- In `V9j-binder-inference-provider-mapping.md` **Adds**, change `the provider-error → \`QueryError\` classifier` to `the provider-error → \`QueryError\` mapping`.
- In `V9j-binder-inference-provider-mapping.md` **Tests** bullet 1, change the opening `The provider classifier maps each provider response to its \`QueryError\` variant and \`retryable\` flag; …` to `The provider-error mapping resolves each provider response to its \`QueryError\` variant and \`retryable\` flag; …`.
- Apply the identical edit to **Tests** bullet 1 of `V9j-T-binder-inference-provider-mapping.md` (its Tests bullets mirror `V9j`'s).

This is a leaf-internal wording alignment only; do not edit `provider-error-mapping.md` or any other spec page. The spec's own "Classifier input surface" note describes the runtime input-reading component and is unaffected — the leaf is not obliged to suppress the word "classifier" where it genuinely refers to that component, but the deliverable named in the leaf's title, Adds, and Tests should read as the single term "provider-error mapping".

## Consequence

**Severity:** cosmetic

A reader scanning the leaf sees the title's "mapping" and the body's "classifier" and must infer they denote the same deliverable; a naming-consistency check keyed on the title term misses the body occurrences. Pure wording; no implementer is led to produce a divergent or incorrect artefact.
