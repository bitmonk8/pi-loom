## Finding

In `docs/spec_topics/binder/binder-model-and-context.md`, the *Compact-transcript format (normative)* sub-section pins the binder's transcript bytes as a MUST-reproduce-exactly input-reproducibility contract. The §*Why the transcript bytes are pinned* paragraph spells out that any wobble in the rendered bytes can change the binder's sampled output even at `temperature: 0`, and that this is precisely why every BNDR-7 reference rendering is normative. BNDR-8 then makes the consequence explicit for tool-call arguments: `JSON.stringify` alone is not key-order-stable across SDK property-insertion orders, so `<args-json>` MUST emit object keys in ascending Unicode code-point order at every nesting level (and BNDR-7e is the multi-key reference rendering that pins the contract).

Rule 4 of the same sub-section, in the `toolResult` bullet, says: *"`ToolResultMessage.content` is a typed-block array; the body is the concatenation of its text-block content in array order, with any non-text block `JSON.stringify`'d with no whitespace and concatenated in the same array order."* The phrase "with no whitespace" is the only stability constraint — there is no key-sort obligation. A multi-key non-text block (any plain object the SDK's tool-result type family carries) therefore renders to bytes whose key order is whatever the producing SDK happens to insert at runtime, in violation of the same input-reproducibility contract BNDR-8 was added to satisfy. The lone existing reference rendering for the case (BNDR-7g) uses a single-key block (`{"chartId":7}`), which masks the gap rather than exposing it: a conformance test built from BNDR-7g cannot tell a conforming renderer from a property-order-sensitive one.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `01c9db9` (2026-06-03) — *pi-loom spec: resolve "compact-transcript renderer references undeclared SessionContext/AgentMessage fields"* — rewrote rule 4's `toolResult` bullet from the prior `string-vs-structured` shape to the current typed-block-array shape that introduces the unbounded *non-text block `JSON.stringify`* phrase.
- `1a41db2` (2026-06-06) — *pi-loom spec: resolve "Compact-transcript BNDR-7 oracle coverage and assistant byte-determinism (T034/T106)"* — added BNDR-8 (the explicit key-sort canonicalisation) plus BNDR-7e (the multi-key tool-call rendering that exercises it), and added BNDR-7g as the `toolResult` mixed-content rendering — but with a single-key non-text block, leaving rule 4's `JSON.stringify` phrasing unchanged.

**History:** The defect is the interaction of these two commits, not either one in isolation. `01c9db9` opened the non-text-block rendering path with bare `JSON.stringify`; `1a41db2` introduced the key-sort canonicalisation for the sibling `[tool-call …]` path while leaving the rule-4 `toolResult` clause and BNDR-7g blind to the same hazard. `git log --follow -- docs/spec_topics/binder/binder-model-and-context.md` shows no later edits to rule 4's `toolResult` bullet or BNDR-7g.

## Solution Space

**Shape:** single

### Recommendation

Strengthen rule 4's `toolResult` bullet so that non-text blocks pass through the same canonical no-whitespace + ascending-code-point key-sort serialisation BNDR-8 defines (rather than a bare `JSON.stringify`), name BNDR-8 as the canonicaliser for both call sites, and replace BNDR-7g's non-text block with a multi-key shape (and ideally a nested object) so the reference rendering actually witnesses the sort. Concretely:

- In rule 4, replace *"any non-text block `JSON.stringify`'d with no whitespace and concatenated in the same array order"* with *"any non-text block serialised by the canonical no-whitespace, ascending-code-point key-sort defined in [BNDR-8](#bndr-8), concatenated in the same array order"*. Apply the canonicalisation at every object nesting level (matching BNDR-8's existing scope) while preserving array element order verbatim.
- Restate BNDR-8 so its scope sentence covers both `ToolCall.arguments` (the existing pin) and `ToolResultMessage.content` non-text blocks (the new pin), and so the "`JSON.stringify` alone is not key-order-stable …" rationale sentence is co-cited by both sites. The renamed obligation does not need a new REQ-ID — BNDR-8 already is the byte-determinism pin, this just makes its scope honest.
- Update BNDR-7g's non-text block from `{"chartId":7}` (single-key) to a shape whose unsorted JSON would differ from the sorted JSON — e.g. one whose unsorted `JSON.stringify` form would be `{"width":640,"chartId":7}` but whose required rendered form is `{"chartId":7,"width":640}`. A nested object (e.g. `{"chartId":7,"meta":{"y":1,"x":2}}` → `{"chartId":7,"meta":{"x":2,"y":1}}`) is strongly preferred since it also pins the *every nesting level* clause for this code path. Keep the surrounding text-block bytes (`Rows: ` / ` (rendered)`) unchanged so the rendering still exercises the in-array-order concatenation rule.

Implementer-relevant edge cases:

- BNDR-9's transcript-safety reasoning does not extend to non-text-block bytes: BNDR-9 governs the `[custom:<type>]` role-tag interpolation. The non-text block's serialised bytes may legitimately contain `]`, `:` and space; what they MUST NOT contain is implementation-dependent key ordering. The fix here is canonicalisation, not rejection.
- The canonicaliser is the same function the assistant path already uses for `ToolCall.arguments` — implementers should share one routine across both call sites rather than duplicating the key-sort. `V11b-T`'s assertions for BNDR-7e and the strengthened BNDR-7g should pin the shared behaviour.
- The Pi-SDK tool-result type family owns the non-text block's internal shape ([Pi Integration Contract — `SessionContext` and the `.messages` element shape](../spec_topics/pi-integration-contract/host-interfaces-core.md#sessioncontext-shape)). The canonicalisation runs over whatever JSON the block ultimately serialises to; loom does not need to enumerate or validate the block's schema, only canonicalise its serialised bytes.

## Consequence

**Severity:** correctness

Two conforming implementers can ship binders whose `bind_context: session` transcript bytes differ on any session whose history contains a `toolResult` with a multi-key non-text content block, because nothing in the spec forbids relying on SDK property-insertion order for those bytes. Under the [Determinism](../spec_topics/binder/determinism-cancellation-failure.md#determinism) contract that pins `temperature: 0` and a stable seed, the divergent prompt bytes can produce divergent binder output for the same loom and the same caller transcript. The current BNDR-7g rendering cannot detect the regression because its non-text block has a single key.
