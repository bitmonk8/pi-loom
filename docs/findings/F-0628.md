## Problem

In `docs/spec_topics/binder/binder-model-and-context.md`, the *Compact-transcript format (normative)* sub-section pins the binder's transcript bytes as a MUST-reproduce-exactly input-reproducibility contract, and BNDR-8 makes that explicit for tool-call arguments by requiring ascending-code-point key-sort because `JSON.stringify` alone is not key-order-stable. Rule 4's `toolResult` bullet, however, serialises any non-text content block with bare `JSON.stringify` "with no whitespace" and no key-sort obligation, so a multi-key non-text block renders to bytes whose key order is whatever the producing SDK happens to insert at runtime — the exact hazard BNDR-8 exists to close. The lone reference rendering for the case (BNDR-7g) uses a single-key block (`{"chartId":7}`), which masks the gap: a conformance test built from it cannot distinguish a conforming renderer from a property-order-sensitive one. Two conforming binders can therefore emit divergent `bind_context: session` bytes, and under the Determinism contract divergent bytes can produce divergent binder output for the same transcript.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `01c9db9` (2026-06-03) — *pi-loom spec: resolve "compact-transcript renderer references undeclared SessionContext/AgentMessage fields"* — rewrote rule 4's `toolResult` bullet from the prior `string-vs-structured` shape to the current typed-block-array shape that introduces the unbounded *non-text block `JSON.stringify`* phrase.
- `1a41db2` (2026-06-06) — *pi-loom spec: resolve "Compact-transcript BNDR-7 oracle coverage and assistant byte-determinism (T034/T106)"* — added BNDR-8 (the explicit key-sort canonicalisation) plus BNDR-7e (the multi-key tool-call rendering that exercises it), and added BNDR-7g as the `toolResult` mixed-content rendering — but with a single-key non-text block, leaving rule 4's `JSON.stringify` phrasing unchanged.

**History:** The defect is the interaction of these two commits, not either one in isolation. `01c9db9` opened the non-text-block rendering path with bare `JSON.stringify`; `1a41db2` introduced the key-sort canonicalisation for the sibling `[tool-call …]` path while leaving the rule-4 `toolResult` clause and BNDR-7g blind to the same hazard. `git log --follow -- docs/spec_topics/binder/binder-model-and-context.md` shows no later edits to rule 4's `toolResult` bullet or BNDR-7g.

## Solution approach

Rewrite rule 4's `toolResult` bullet so non-text blocks pass through the same canonical no-whitespace, ascending-code-point key-sort serialisation defined by [BNDR-8](#bndr-8) (applied at every object nesting level) rather than bare `JSON.stringify`, preserving array element order. Restructure BNDR-8 so its scope covers both `ToolCall.arguments` and `ToolResultMessage.content` non-text blocks, with the `JSON.stringify`-is-not-stable rationale co-citing both sites. Rewrite BNDR-7g's non-text block to a multi-key (and preferably nested) shape whose unsorted JSON differs from its required sorted rendering, so the reference rendering actually witnesses the sort.

## Solution constraints

- Out of scope: BNDR-9's transcript-safety rule governs `[custom:<type>]` role-tag interpolation only; do not extend it to reject non-text-block bytes (which may legitimately contain `]`, `:`, and space) — the fix is canonicalisation of those bytes, not rejection.
