## Problem

The static-resolution load pass owns one in-process parse cache, which the spec names canonically in `implementation-notes.md` (the *Static-resolution load pass*) as a single per-pass parse cache. The two plan leaves that own (`V15a`) and consume (`V15e`) this cache refer to it with at least four divergent surface phrasings ("static-resolution parse cache", "shared per-pass cache", "static-resolution per-pass parse-cache", "per-pass static-resolution cache"), and no plan site uses the canonical spec term. A maintainer cross-referencing the cache between its owner and consumer hits a search miss in both directions, and the divergent qualifiers invite the false inference that there are two distinct caches when there is one.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `c6a664e` ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10); `249cec5` ("pi-loom plan: resolve \"implementation-notes.md (IMPL) unmapped in coverage matrix\"", 2026-06-11); `a3dd2f0` ("pi-loom plan: resolve \"hot-reload cache-eviction sequenced before parse cache and .warp import edges\"", 2026-06-16)
**History:** The plan corpus is git-tracked. `git log -S "static-resolution parse cache" -- docs/plan_topics/V15a-invocation-core.md` returns `c6a664e` as the sole commit — the "static-resolution parse cache" phrasing in `V15a`'s Adds was present at the leaf's inception. `git log -S "shared per-pass cache" -- docs/plan_topics/V15a-invocation-core.md docs/plan_topics/V15a-T-invocation-core.md` returns `249cec5`, where the "shared per-pass cache" variant entered `V15a`/`V15a-T`'s implementation-notes Tests bullet as part of an unrelated coverage-matrix mapping resolution. `git log --follow -- docs/plan_topics/V15e-hot-reload-cache-eviction.md` and `git log -S "per-pass parse-cache" -- docs/plan_topics/V15e-hot-reload-cache-eviction.md` both return `a3dd2f0` as the sole commit — `V15e` (and `V15e-T`) was created with the "static-resolution per-pass parse-cache" / "per-pass cache entry" variants. Each of the three commits introduced a different surface phrasing for the one cache the spec already named, and no commit ever reconciled them; the defect is the un-reconciled interaction of three independent edits, not a single regression.

## Solution approach

Adopt the canonical term `static-resolution per-pass parse cache` for this cache across all plan sites, harmonising with the spec's "per-pass parse cache" in `implementation-notes.md`. Rename the divergent variants in `V15a` (owner), `V15a-T`, `V15e`, and `V15e-T` to that single term, keeping any "entry"/"entries" suffix where the prose refers to a per-file cache entry. If the `implementation-notes.md` (IMPL) row of `coverage-matrix.md` is touched, align its cache phrasing to the same canonical form.

## Solution constraints

- Out of scope: `docs/spec_topics/implementation-notes.md` is read-only and authoritative for the term — do not edit it.
