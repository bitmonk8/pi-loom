## Finding

`audit-wire-and-canary.md` permits two wiring choices for the surface-set-closure audit's fail-closed gate: in-process to the harness (e.g. a Vitest test case whose `expect(...).toBe(0)` asserts on the audit's exit code) or as a spawned child process. The *Wire serialisation* clause separately pins that the audit emits `audit/<class>/<family>/<symptom>` line-delimited records on **stdout** as the sole machine-parseable surface a CI parser may consume, and that "the audit's record emission MUST reach a sink whose contents survive the harness wrapping". That sink-survival rule is written as an obligation on the audit, but the in-process variant pushes the obligation onto the harness: Vitest (and most test runners) capture child `console.log` / `process.stdout.write` calls into per-test buffers, reformat them with test-case headers and progress markers, or attach them to a reporter slot. A green `npm test` running the audit in-process under a default Vitest configuration can easily ship a stdout stream where the `audit/<class>/...` lines are interleaved with `✓ inventory audit (12ms)`-shaped reporter output, prefixed with file-and-line headers, or buffered behind a non-default reporter that elides them entirely — none of which violates the audit's own obligations, all of which break the CI parser.

Two preconditions are required for the in-process variant to satisfy the wire contract end-to-end, and neither is stated: (1) the harness MUST be configured so the audit's raw stdout bytes pass through to the `npm test` process's stdout unmodified — no per-test capture, no reformatting, no buffering behind a reporter that suppresses passing-test output; and (2) no harness-emitted line (reporter banners, test names, progress dots, summary tables) may share the `audit/<class>/<family>/<symptom>` leading-token shape, since the CI parser routes by leading-token prefix match. Without either precondition pinned, two reasonable implementers wire `npm test` differently and produce different stdout streams from the same audit.

## Solution Space

**Shape:** multiple

### Option A — State the in-process precondition

**Approach.** Add a sentence to the *step-2(b) wiring* paragraph that pins, for the in-process variant only, the two preconditions: (1) the harness MUST be configured to pass the audit's raw stdout bytes through to the `npm test` process's stdout without capture, reformatting, or per-test buffering; and (2) no harness-emitted line may share the `audit/<class>/<family>/<symptom>` leading-token shape (a reserved-prefix rule for harness output). Either precondition that the implementation cannot meet under its chosen harness disqualifies the in-process variant and forces the child-process variant.

**Spec edits.** One sentence inserted into `audit-wire-and-canary.md`'s final paragraph immediately after "An implementation wiring that interprets empty stdout as 'no failures' violates this obligation.", phrased as a MUST against the in-process variant. No edit to the *Wire serialisation* clause, since the precondition is wiring-shape-specific rather than a record-format change.

**Pros.** Preserves both wiring options. Makes the implicit harness contract explicit, so an implementer choosing Vitest sees what to configure (e.g. disable per-test stdout capture, pin a reporter whose output cannot collide with the `audit/` prefix). Costs nothing in flexibility.

**Cons.** Adds one more obligation to a paragraph already dense with them. The reserved-prefix rule (2) is a transitive obligation on the harness's reporter choice, which an implementer may not initially notice spans every Vitest reporter the project might pick up later.

**Risks.** A reporter swap in a later leaf — pulling in a CI-friendlier reporter that happens to print `audit/...`-shaped progress lines — could silently break the gate; the precondition lands the obligation but leaves enforcement to manual review unless `V18b-T` adds a test that greps the live `npm test` stdout for stray `audit/` prefix lines outside the audit's own records.

### Option B — Restrict the machine-parseable contract to the child-process wiring

**Approach.** Drop the in-process variant's eligibility for the per-discriminator routability obligation. The audit MAY still run in-process for fail-closed exit-code propagation, but the *Wire serialisation* and per-discriminator routing contracts bind only when the audit runs as a spawned child process whose own stdout is the sink CI parses. The in-process variant satisfies fail-closed only; CI parsers expecting `audit/<class>/...` lines MUST run the child-process variant.

**Spec edits.** Two edits to `audit-wire-and-canary.md`'s final paragraph: (i) rephrase "either wiring is permitted as long as the fail-closed property holds end-to-end" to "either wiring is permitted for the fail-closed exit-code property; the per-discriminator wire contract pinned by *Wire serialisation* above binds only the child-process wiring"; (ii) delete or rewrite the sink-survival sentence so it no longer pretends to cover the in-process case.

**Pros.** Eliminates the harness-stdout precondition entirely by removing the variant that needs it. Simpler invariant for CI parser authors — there is exactly one wire shape they care about. No risk of a future reporter swap silently breaking the gate.

**Cons.** Removes a wiring choice some implementations would prefer (in-process is faster, simpler, and keeps the audit visible in the standard test-runner output). Forces every implementation that wants machine-parseable output to take the child-process spawn cost on every `npm test` run.

**Risks.** A leaf that wires in-process for fail-closed-only ships green CI but has no machine-parseable audit output; a contributor seeing `npm test` red and no `audit/<class>/...` lines on stdout has to discover the carve-out before reaching for the recovery path the *Pre-emission termination carve-out* sentence already names for genuinely killed audits.

### Recommendation

**Option A.** The in-process variant is genuinely valuable (lower spawn cost, the audit's output is co-located with the rest of the test output, no orchestration friction), and the precondition is concrete enough to state in one sentence. Option B fixes the defect by removing the surface, which is a larger contract change than the defect warrants. Implementer edge cases under Option A: (i) Vitest's default `pool: 'threads'` and per-test stdout capture must be reviewed — `V18b` should pin the specific Vitest configuration switches it relies on (e.g. `printConsoleTrace: false` and a reporter that does not capture passing-test output, or running the audit case under a `pool: 'forks'` boundary); (ii) the reserved-prefix rule should be enforced in `V18b-T` by an explicit fixture that asserts no non-audit line on the live `npm test` stdout matches `^audit/`, so a future reporter swap reds the test rather than silently breaking CI.

## Consequence

**Severity:** correctness

A green in-process `npm test` whose Vitest configuration captures or reformats stdout — the default in many Vitest setups — ships an `audit/<class>/...` stream a CI parser cannot consume, defeating the per-discriminator routability obligation the same page pins. The audit still fails-closed on exit code, so the spec's primary safety property holds, but the contributor-facing routing table the *Failure-surface contract* and *Non-empty-scan canary* clauses build on is silently disabled — broken CI integration without a build-time signal.
