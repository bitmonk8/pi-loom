## Finding

The spec coins the string `loom-direct:` as the synthesised `toolCallId` prefix the runtime supplies when loom code calls a Pi tool's `execute(...)` directly. The prefix is asserted normatively in `pi-integration-contract/host-interfaces-core.md` §*Tool execution from loom code* ("`toolCallId` is a synthesised UUID prefixed `loom-direct:`.") and cited from `tool-calls.md` in the *loom 1.0 seam — per-call timeout* blockquote. It meets the glossary's inclusion criterion ("Add new entries here when the spec coins a new term reused on more than one page"). No such entry exists.

The two uses are textually consistent, so the omission is not corrupting the corpus today. The defect is wayfinding: a reader has no glossary anchor to confirm it as a spec-coined token, no canonical-page pointer, and no governed guard against future drift to a near-synonym. The PIC declaration in `host-interfaces-core.md` is the canonical home; the `tool-calls.md` blockquote is a cross-reference.

### Recommendation

Add a `loom-direct:` entry to `glossary.md` (alphabetised), defining it as the synthesised `toolCallId` prefix the runtime supplies for loom-code direct tool execution (suffix is a UUID), and `See:`-pointing at `pi-integration-contract/host-interfaces-core.md` §*Tool execution from loom code* as canonical. Do not add a second definition; the entry is a pure orientation aid. Keep the `tool-calls.md` blockquote as the cross-reference it is.
