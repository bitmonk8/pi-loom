## Problem

In `expressions.md` the `?`-operator section states a function that uses neither `?` nor an explicit `Result` return type "is required to handle every query failure with `match` (or to discard explicitly)". The phrase "every query failure" reads as scoped to `@`-query failures, but `ERR-18` admits `?` on any `Result<T, QueryError>` operand — a `@`-query, an `invoke(...)`, or a code-side tool call (which lowers to `CodeToolError`, a `QueryError` variant per `host-interfaces-core.md`). The only enforcing diagnostic, `loom/parse/discarded-query-result`, scopes itself in its registry row and in QRY-19 to a "Bare `@`...`` expression-statement", so two conforming front-ends diverge on whether a bare tool-call or `invoke` statement that drops its `Result` is a parse error or a silently-accepted discard.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 53f5831 ("pi-loom spec: rewrite query primitive, system prompts, and schema syntax", 2026-05-03)
**History:** `git log -S 'required to handle every query'` traces the sentence to 53f5831; its parent (the initial scaffold) contains no such text, so the construct entered the corpus at 53f5831 already carrying the "every query failure" framing while the discard parse error was scoped to bare `@`...`` statements in the same commit. The phrasing was carried verbatim through the F-2145 relocation (87dfa1d, 2026-06-28) that moved the `match`/`?` material from the error-model into the expressions cluster. The `CodeToolError` / `invoke` Result-producing forms were consolidated shortly afterward (around 2026-05-04/05) without reconciling this sentence's scope, so the scope gap has been latent in the text since the sentence was first authored.

## Solution approach

Rewrite the mandatory-handling sentence in `expressions.md` so the obligation applies to every expression of static type `Result<T, QueryError>` — `@`-queries, code-side tool calls, and `invoke(...)` alike — rather than only query failures. Close the enforcement gap by widening `loom/parse/discarded-query-result` to cover any dropped `Result<T, QueryError>` expression-statement, adjusting its registry-row trigger text in `code-registry-parse.md` and the QRY-19 wording in `query-escapes-stringification.md` away from the `@`...``-only framing. Extend QRY-20's operator-facing always-log observability for discarded query `Err`s to discarded `CodeToolError` / `invoke` `Err`s on the same footing.

## Solution constraints

- The `let _ =` escape hatch and the `void`-tail-function discard (QRY-19 / FN-4) already apply to any expression; do not change their scope.
