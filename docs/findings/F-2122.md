## Finding

The *Provider compatibility for typed queries* paragraph in `conversation-drive.md` carries an HTML comment spliced into the middle of a normative sentence, between the clause that names the warning diagnostic and the clause that states the loom still loads:

> … the runtime emits `loom/load/typed-query-unsupported-provider` (warning) naming the model; `<!-- the warning's payload is exactly the model + provider carried by its diagnostics/code-registry-load.md message template; the former "listing the typed-query expression positions" clause was dropped as no positions format was ever pinned -->` the loom still loads, and any typed query at runtime returns `Err(QueryError { … })`.

The comment is a decision-log record: it explains that a prior "listing the typed-query expression positions" clause was removed because no positions format was ever pinned, and restates what the warning payload carries. Neither half is a normative constraint — the payload contract belongs to the `diagnostics/code-registry-load.md` message template, and the rationale for a past deletion is history that lives in git, not in the spec body. Embedding it mid-sentence in normative prose is exactly the decision-log cruft pattern the spec's editorial discipline rejects.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** bb841d4 — pi-loom spec: resolve "typed-query-unsupported-provider listing positions has no pinned format" (2026-06-06)
**History:** The paragraph originally stated the warning names the model "and listing the typed-query expression positions". Commit bb841d4 resolved a prior finding that no positions format was ever pinned by deleting that clause — but instead of removing it cleanly, it left an HTML comment in the clause's place recording why the deletion happened and what the payload now carries. The decision-log cruft is therefore the residue of that resolution commit, not present since the file's inception.

## Solution Space

**Shape:** single

### Recommendation

Delete the entire HTML comment, leaving the sentence as "… emits `loom/load/typed-query-unsupported-provider` (warning) naming the model; the loom still loads, and any typed query at runtime returns `Err(QueryError { … })`." The warning-payload contract is already owned by the `diagnostics/code-registry-load.md` message template, and the rationale for the past clause removal need not survive in the spec body.

## Consequence

**Severity:** cosmetic

The comment does not render and carries no normative weight, so no implementer behaviour diverges. Its only effect is reader friction and the risk that a future editor mistakes the embedded payload restatement for a live constraint that competes with the authoritative `code-registry-load.md` template.
