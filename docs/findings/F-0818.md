## Finding

The **Baseline acquisition** paragraph in step 2 of `version-bump-step2.md` (anchor `bump-baseline-acquisition`) phrases the contributor's MUST as the literal command form — "MUST materialise … by running `npm pack @earendil-works/pi-coding-agent@<pinned-version>` and extracting the resulting tarball" — rather than as the observable property the procedure actually relies on. Three things matter at this point: that the [loom 1.0 Pi-SDK pin]'s `dist/*.js` tree is materialised as the "from" side of every `dist/*.js` diff items (e)/(r)/(aj) prescribe, that the materialised tree is ephemeral and is not committed (the paragraph already states this), and that the acquisition does not drag in the lock-step siblings' transitive dependency tree (the paragraph already explains *why* `npm pack` is preferred over `npm install --prefix`).

By spelling the recipe at command-string granularity, the MUST now binds to `npm pack` itself rather than to those properties. A contributor who reaches for an equivalent that satisfies all three (`npm view … dist.tarball | tar -x`, a registry mirror, a CI cache restoration, a pre-fetched `.tgz` artefact, etc.) is technically non-conformant against the literal MUST even when the resulting `package/dist/` tree is byte-identical to what `npm pack` would have produced. The rationale-comment immediately after the command ("The `npm pack` form is used rather than `npm install --prefix` so the lock-step siblings' transitive dependency tree is not pulled pointlessly") already exposes the behavioural intent — the obligation should be re-anchored to that intent, with the `npm pack` invocation kept as a worked example.

## Solution Space

**Shape:** single

### Recommendation

Restate the baseline-acquisition MUST behaviourally, then keep the existing `npm pack` invocation as a non-normative worked example. Concretely, replace the opening of the **Baseline acquisition** paragraph so that the obligation reads roughly:

> "Before running the `dist/*.js` diffs items (e), (r), and (aj) prescribe, the contributor MUST materialise the [loom 1.0 Pi-SDK pin]'s `dist/*.js` tree into a scratch directory as an ephemeral 'from' side of the diff — by any means that retrieves the pinned `@earendil-works/pi-coding-agent` package's `dist/` subtree without resolving its transitive dependencies and without persisting the result in-repo. The canonical worked example is `npm pack @earendil-works/pi-coding-agent@<pinned-version>` followed by extracting the resulting tarball; the extracted `package/dist/` tree is then the canonical 'from' side …"

Retain unchanged:

- The "from" / "to" side definitions and the loom-1.0-pin → candidate-minor diff direction.
- The `<pinned-version>` capture rule (pre-bump value read before step 4 rewrites the literal).
- The step-1 ordering caveat ("MUST either run before that install, or — if ordered after — pass the explicit `@<pinned-version>` specifier").
- The ephemeral / MUST-NOT-be-committed clause.

The rationale sentence that currently reads "The `npm pack` form is used rather than `npm install --prefix` so the lock-step siblings' transitive dependency tree is not pulled pointlessly" should be folded into the new behavioural phrasing as the *reason* for the no-transitive-deps property, rather than as a comparison between two specific commands — that prevents the same prescription-lens issue from re-appearing one paragraph later.

Edge case the implementer must watch: do not weaken the **ephemeral, MUST NOT be committed** clause when softening the surrounding obligation — it is what makes the no-in-repo-snapshot rule auditable, and is independent of which command produced the tree.

## Consequence

**Severity:** cosmetic

A literal-minded contributor or reviewer can read the current text as binding to `npm pack` rather than to the underlying behaviour, generating spurious conformance-debate when an equivalent acquisition method is used. No runtime behaviour changes and no two implementers will diverge on the audit's outcome; this is a prose-prescription tightening, not a correctness issue.
