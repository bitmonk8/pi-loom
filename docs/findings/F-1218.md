## Finding

Three normative MUSTs that the version-bump procedure relies on for its drift-detection contract are written in a form that no automated check can falsify, and that the spec itself acknowledges as contributor-discipline backstopped only by editorial review of the bump commit. Each is structurally independent — different file, different paragraph, different fix surface — and they fail open in distinct ways:

1. **Category-(3) drift MUST** in `version-bump-step2.md` step 2(a) requires the contributor to cross-reference the candidate Pi minor's `.d.ts` declarations of `ExtensionContext` and `ExtensionCommandContext` against every category-(3) entry of `SDK_SURFACE_INVENTORY` on each bump. The spec admits in the same paragraph that this is "contributor-discipline rather than a type-aware mechanical gate" and offers only a SHOULD-level smoke check that probes member-presence for entries already present in the inventory — the smoke check cannot witness the failure mode the MUST exists to catch (an SDK-side member that the runtime starts consuming in the same bump but that the contributor forgets to promote to a category-(3) entry). The obligation also sits outside the per-item `pass`/`fail`/`N/A` recording shape that governs items (a)–(ao), so there is no commit-message slot a reviewer can inspect to confirm it was performed.

2. **Item (a) ordering MUST** in `version-bump-step2.md` item (a) carries an open contradiction whose authoritative determination "MUST be resolved before SM-4/5/6 are implemented", and item (a)'s one-shot recording shape is the dedicated outcome a conformant determination takes. No build-time gate blocks an implementer from starting SM-4/5/6 against the unresolved branch; plan-side, the ordering is enforced only via `V9l`'s `V18c` **Deps** edge — a spec-side mechanical gate does not exist, and the spec leans on editorial review of the bump commit message and the plan's blocked-obligations table to hold the line.

3. **Second-stage anchor-set MUST** in `version-bump-triggers.md` step 5 requires that any future runtime-side reference shape reading or stamping a `SessionShutdownEvent['reason']` value be added to the grep's second-stage anchor filter (`event.reason|SessionShutdownEvent|capturedEventReason|entry.shutdownReason`) in the same edit that introduces the new shape. The build-time audit pass the same step now mandates compares hits against the documented enumeration — but a new anchor name that is omitted from the filter is excluded by the filter itself, so the hit never reaches the delta check. The audit can detect a stale documented enumeration relative to a fixed filter; it cannot detect a stale filter, which is the failure mode the MUST is written to prevent.

The unifying gap is the same: each MUST encodes a real correctness need but routes its only enforcement through contributor recall, with no observable signal a reviewer or CI can key on. The bump procedure already pins multiple paths to recording obligations and build-time gates for sibling MUSTs (item-(e) two-sub-outcome recording, item-(a) authoritative-determination shape, the `SessionShutdownEvent['reason']` snapshot brand-string assertion, the inbound-reference build-time audit gate); these three are the residue that did not get the same treatment.

- `c0259ca` (2026-06-07) — *pi-loom spec: resolve "Version-bump procedure: four obligations have no verifiable acceptance criterion"* — promoted the inbound-reference sweep to require a build-time audit pass over the documented enumeration, but left the second-stage anchor filter itself outside the audit's reach (the audit reports deltas against the enumeration after the filter has run, so a filter-omitted anchor cannot surface as a delta); this is the commit that fixed the sibling testability gap on the enumeration without closing the self-verification gap on the filter.
- `7447d1e` (2026-06-10) — *pi-loom plan: resolve "Session-only degraded-state presupposition (a) contradicts Pi's documented lifecycle"* — wrote the "*open question that MUST be resolved before SM-4/5/6 are implemented*" wording into item (a), establishing the ordering MUST with editorial-review-only enforcement; subsequent refinements (`80fcd63`, 2026-06-19) added the authoritative-determination recording shape and evidentiary standard but did not introduce a build-time gate.

**History:** `git log -S` searches against the three defect tokens (`"Category-(3) drift contributor obligation"`, `"extend this second-stage anchor set"`, `"MUST be resolved before SM-4/5/6 are implemented"`) each land on a single introducing commit, but the three commits are independent: they entered the corpus in three different resolve-passes against three different upstream findings, none of which surfaced the others. The defect is a multi-commit interaction in the durable sense that the three contributor-discipline MUSTs have accumulated in the same procedure over six weeks without anyone noticing they share a structural pattern (MUST-as-recall) — none of the introducing commits is wrong in isolation, and a fix that reverts any one of them would not address the pattern. The Category-(3) obligation has been carried forward unchanged across the file split (`81bf52a`) and all subsequent step-2 edits; the second-stage anchor set was tightened in `c0259ca` to add the build-time audit gate on the enumeration but the filter-completeness gap survived that tightening; item (a) has accreted recording shape and evidentiary standard refinements but no build-time gate.

## Solution Space

**Shape:** multiple

Three independent obligations with three independent fix surfaces. The orchestrator should resolve them sequentially in the order below so each fix lands on a stable baseline: the recording-shape change for item (a) is the smallest and scope-bounding (no implementation change required, only a recording-slot extension and a `V18c` acceptance-check tightening), the second-stage-filter fix is mid-sized (requires authoring a derivation rule and a new gate), and the Category-(3) fix is the most-substantive (requires authoring a structural type-side gate). Treating them as one finding would force a single fixer to write one prose edit covering all three vectors and produce a much larger diff than any half alone.

### Option A — Item (a) ordering MUST: add a recording slot to the per-item table and gate `V18c` acceptance on it

**Approach.** Extend the per-item recording shape so item (a)'s `governed-by-rebind` / `provably-differs — <cited evidence>` verdict has a dedicated, machine-locatable slot in every bump commit message (or a dedicated `docs/` artefact the bump touches); tighten `V18c`'s editorial-review acceptance check so a bump whose commit message lacks an item-(a) verdict (or whose verdict is `provably-differs` without a cited evidence site) is non-conformant on the same footing as a missing per-item `pass`/`fail`/`N/A` outcome. Cross-reference the new acceptance check from `plan.md` §"Blocked obligations" so the V9l-blocked-on-V18c edge keys on the recording rather than on contributor recall.

**Spec edits.** In `version-bump-step2.md`, give item (a)'s authoritative-determination recording shape a top-level position in the "*Outcome recording conventions*" block (it currently sits at the end of that block) and add a normative MUST that the verdict, when recorded, names which of SM-4/SM-5/SM-6 work is unblocked by it; in `V18c`'s acceptance prose, extend the "no MUST-level item (a)–(e) carries a `fail` outcome" check to include "and item (a) carries exactly one of the two verdicts with a cited evidence site, or the bump does not land any SM-4/5/6 implementation work".

**Pros.** Smallest diff; no new build-time machinery; surfaces the ordering MUST as a reviewable record.
**Cons.** Still editorial-review-enforced, not mechanical; relies on `V18c`'s reviewer noticing.
**Risks.** None substantive; the ordering MUST is fundamentally a process gate that cannot be made fully mechanical without a typed `branch-unreachable` resolution channel from Pi.

### Option B — Second-stage anchor-set MUST: derive the filter from a single source of truth so omission becomes a structural error

**Approach.** Rewrite the second-stage anchor set as a closed enumeration sourced from a runtime-side declaration (e.g. an array of inbound-reference shape names co-located with the handler-scoped capture and the per-entry stamp field) and have the build-time audit pass read its filter tokens from that declaration rather than from the spec paragraph's prose list. Adding a new reference shape then either extends the declaration (and the filter follows) or is silently absent (and a separate completeness gate — a structural assertion that every runtime-side `SessionShutdownEvent['reason']` read/stamp site is enumerated in the declaration — reddens). Pair this with a build-time gate (a small AST/grep pass over `src/extension/`) that enumerates every site reading `event.reason` or stamping `shutdownReason` and asserts each appears in the declaration.

**Spec edits.** In `version-bump-triggers.md`'s "*Enumeration mechanism — grep target*" paragraph, replace the inline filter prose with a forward-reference to the runtime-side declaration and add a normative MUST that the build-time audit pass reads its filter tokens from that declaration; add a sibling normative MUST that a build-time completeness gate asserts every runtime site reading or stamping a `SessionShutdownEvent['reason']` value is enumerated in the declaration. Cross-reference the declaration's source-of-truth site from `unknown-reason-rule.md` (`capturedEventReason` pin) and `active-invocation-registry.md` (`entry.shutdownReason` pin) so a future rename touches the declaration in lock-step.

**Pros.** Closes the self-verification gap mechanically; the filter cannot drift from the runtime sites it filters for.
**Cons.** Authors a new completeness gate (modest implementation surface); requires picking a runtime-side declaration site.
**Risks.** The completeness gate's coverage is bounded by what the AST/grep pass can recognise — a future inbound-reference shape that does not lexically read `event.reason` or stamp `shutdownReason` (e.g. one that reads through an alias) would be missed; mitigate by pinning the audit's recognised shapes alongside the declaration.

### Option C — Category-(3) drift MUST: add a structural type-side gate over `ExtensionContext` / `ExtensionCommandContext`

**Approach.** Replace the contributor-discipline obligation with a build-time type-side assertion that walks `ExtensionContext` and `ExtensionCommandContext`'s `.d.ts` member sets and asserts every member name appears as the rightmost-dot-separated-segment of exactly one category-(3) `SDK_SURFACE_INVENTORY` entry, with an exemption list for SDK-side members the runtime does not consume. The assertion is type-aware (consumes the `.d.ts`) rather than a literal-read; this is the requirement step 2(a) currently declines to commit to ("a mechanical gate would commit step 2 to a TypeScript-program load PIC declines to require"), so the option carries an explicit posture change.

**Spec edits.** In `version-bump-step2.md`'s step 2(a) "*Category-(3) drift contributor obligation*" sub-paragraph, replace the contributor MUST with a build-time MUST owning the type-side assertion; reword the smoke-check SHOULD as the now-redundant fallback or delete it; in `inventory-audit-intro.md` §"Inventory-closure audit", add a cross-reference to the new gate as the category-(3) drift counterpart to the existing closure-audit's category-(1)/(2) coverage. In `V18a` or `V18c`, name the new assertion under the appropriate **Tests** bullet (V18a owns the inventory population, V18c owns the build-time gates — natural home is V18c on the same footing as the seven-capability literal-read).

**Pros.** Closes the drift gap mechanically; aligns category-(3) with the existing closure-audit posture for category-(1)/(2).
**Cons.** Reverses an explicit spec-side declination ("PIC declines to require a TypeScript-program load"); adds build-time TypeScript-program-load infrastructure if not already present.
**Risks.** A TypeScript-program load adds non-trivial build-time cost; the exemption list for SDK-side members the runtime does not consume must be maintained, and a fresh contributor obligation accretes around it. Verify that the existing `loom/typecheck/session-shutdown-reason-snapshot` brand-string assertion already loads enough of the SDK's `.d.ts` to make this gate near-free (in which case the spec's declination is now stale and the option is essentially free).

### Recommendation

Implement A first, then B, then C — in that order. A is editorial-only and lands cleanly without touching the implementation; B closes the most-likely failure mode (a future inbound-reference site silently elided) and authors the build-time machinery the catch-all anchor extension MUST currently lacks; C is the heaviest fix and rests on a posture change PIC has explicitly declined, so it should land last after B has established the precedent that build-time gates can replace contributor-discipline MUSTs in this procedure.

Edge cases the implementer must watch:

- For A, item (a)'s evidentiary standard already requires a cited site for `provably-differs`; the new acceptance check must not weaken that — the recording slot is an additional gate, not a substitute. Confirm the recording slot survives a `governed-by-rebind` verdict (which carries no evidence-site obligation under the existing standard).
- For B, the runtime-side declaration site must outlive `V9h`'s implementation order — author it at `V18a` time or earlier so `V9h` consumes it rather than the reverse. The completeness gate's recognised-shape pin must be co-located with the declaration so additions stay in lock-step.
- For C, the exemption list (SDK-side members the runtime does not consume) is the new failure surface; pin its update obligation under `V18c`'s checklist so an SDK member the runtime starts consuming triggers both a category-(3) entry promotion and an exemption-list removal in the same edit.

## Consequence

**Severity:** advisory

A conformant contributor can complete a bump without performing the category-(3) cross-reference, without recording item-(a)'s authoritative determination before SM-4/5/6 work begins, and after introducing a new inbound-reference shape that the second-stage anchor filter silently elides — and the build will stay green in every case. The first failure mode lands an SDK-consumed surface absent from the inventory and the closure audit (silent drift); the second permits implementation work against an unresolved blocking contradiction; the third causes a session-only literal rename on the next bump to no-op its `session-shutdown-runtime-degraded` emission at the newly-added site. None is correctness-fatal on its own — editorial review of the bump commit is the documented backstop — but each defeats the build-time-gate posture the rest of the version-bump procedure works hard to establish.
