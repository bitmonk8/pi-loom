## Finding

Inside the prompt-mode bullet of `conversation-drive.md`, the *Resolution* clause states that `ctx.waitForIdle()` "never rejects, so the runtime MUST NOT wrap the `await` in a `try`/`catch` expecting rejection as a failure signal," and the immediately following *Error detection* clause builds the entire prompt-mode transport-failure surface on the same property ("Because `pi.sendUserMessage` returns `void` (no rejection path) and `waitForIdle()` never rejects, every transport- or provider-level failure … surfaces on the driven turn's trailing `assistant` message …"). The *Hang handling* clause then layers cancellation onto the same assumption. The "never rejects" property is a behavioural commitment of an external Pi method — `@earendil-works/pi-coding-agent`'s `ExtensionCommandContext.waitForIdle()` — and is not encoded by the typed surface: `Promise<void>` constrains only the resolution value, not whether the promise can reject. The mirror inline annotation in `host-interfaces-core.md`'s `ExtensionCommandContext` block (`// resolves once Pi emits agent_end … never rejects (see Conversation drive — prompt mode above)`) restates the same prose claim without pinning it.

Every other Pi/pi-ai behavioural property the same prompt-mode section relies on follows an established two-part pattern: (1) the property is captured under a named `<a id="...-presupposition"></a>` paragraph framed as a "loom-side consumption posture," and (2) detection of a future Pi minor that silently regresses it is routed to a specific lettered item in the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](docs/spec_topics/pi-integration-contract/version-bump-step2.md). The siblings on this page — driven-turn session-commit ordering (item (ac)), user-visible streaming ordering (item (ak)), `complete()` forced-tool and off-session-turn (item (u)), `complete()` retry-timing (item (aa)) and `options.signal` honouring (item (ab)) — and the cancellation-source promise-lifecycle presuppositions on `host-interfaces-core.md` (items (g), (j), (v), (ad)) all carry this treatment. The `waitForIdle()` never-rejects property carries neither half: no named anchor, no checklist item. The bump checklist's closest neighbour, item (j) *Slash-handler turn-liveness*, covers only that a terminal `agent_end` eventually arrives to settle the wait, not whether the wait can reject; the absence note at the end of step 2 ("Any future presupposition added to this page … that routes its detection to editorial review … MUST be added to this checklist in the same edit") confirms the checklist is meant to be exhaustive.

The consequence is that a future `@earendil-works/pi-coding-agent` minor that made `waitForIdle()` reject under some condition — for example, after a Pi-internal hard timeout, or on a re-engineered shutdown path — would silently invalidate both the `MUST NOT wrap the await in a try/catch` rule and the post-`waitForIdle()` error-detection probe, with no build-time SDK surface-inventory signal (the surface-inventory check is a presence check, not a behavioural one) and no editorial-review prompt at the bump. The loom-side symptom would be a Node `unhandledRejection` escaping the prompt-mode driver — the exact failure mode the inline `MUST NOT` is written to prevent — and, on the secondary path, the loss of the trailing-turn error probe for whichever failures Pi started signalling through rejection instead of through `stopReason: "error"`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `d54d798` (2026-05-06, "pi-loom spec: resolve \"waitForIdle resolution, error, and hang semantics not contracted\""); subsequent commits established the consumption-posture / editorial-review pattern around it (`e89e42e` 2026-05-17 "T21 Pi-side slash-handler promise lifecycle taken as given"; `aebec4e` 2026-06-04; `3683345` 2026-06-06; `803106d` 2026-06-07; `85fc906` 2026-06-10) without back-filling this one.
**History:** `git log -S "never rejects"` on the spec corpus shows the *Resolution / Error detection / Hang handling* triplet, which is what introduces the "never rejects" claim and the dependent `MUST NOT` and post-`waitForIdle()` probe, first lands in `d54d798`. At that point the editorial-review-checklist machinery for unpinned host presuppositions did not yet exist in its current form: the *Pi-side slash-handler promise lifecycle (consumption posture)* paragraph and its routed bump-checklist items were introduced in `e89e42e` eleven days later, and the catalogue of named `*-presupposition` anchors with bump-checklist routing was then expanded incrementally by the later commits enumerated above — each one adding (or relocating) the sibling presuppositions that now surround the unrouted `waitForIdle()` claim. The current defect is therefore the interaction between (a) the inline-MUST-NOT framing chosen in `d54d798` before the framework existed and (b) the framework's later adoption everywhere on the same page except this one site; no single commit introduced an inconsistency on its own.

## Solution Space

**Shape:** single

### Recommendation

Lift the *Resolution* clause's "never rejects" claim into a named consumption-posture paragraph and add a corresponding bump-checklist item, matching the exact treatment the sibling presuppositions on the same page already receive.

Concretely:

- In `conversation-drive.md`, add an anchored paragraph (e.g. `<a id="waitforidle-never-rejects-presupposition"></a>*`waitForIdle()` no-rejection (consumption posture).*`) immediately after the existing `*Hang handling.*` clause, modelled on the *Driven-turn session-commit ordering (consumption posture)* paragraph that already follows. The paragraph should: (i) name the property — `ctx.waitForIdle()` returns `Promise<void>` and resolves rather than rejects in every settlement path Pi uses, including the `loomAbort.abort() → ctx.abort()` cancellation tear-down; (ii) explain that the typed surface (`Promise<void>`) does not encode this and that the loom-side `MUST NOT` against a wrapping `try`/`catch` and the post-`waitForIdle()` error-detection probe both depend on it; (iii) explain that loom records this as a loom-side consumption posture because `@earendil-works/pi-coding-agent` encodes no typed no-rejection contract; (iv) state that a Pi minor that rejected `waitForIdle()` would surface as an unhandled rejection from the prompt-mode driver and would route real failures away from the `stopReason: "error"` probe with no SDK surface-inventory signal; and (v) route detection to a new lettered item in the editorial-review checklist.

- Inside the `*Resolution.*` clause (and, optionally, the mirror annotation on `host-interfaces-core.md`'s `ExtensionCommandContext` `waitForIdle()` row) replace the bare "it never rejects" with a forward-link to the new anchor.

- In `version-bump-step2.md`, add a new editorial-review checklist item with an anchor and structure copying item (ac)'s shape. Place it adjacent to items (ac)/(ak) since it concerns the same `waitForIdle()` surface. Wording: *(al) `waitForIdle()` no-rejection — [Conversation drive — `waitForIdle()` no-rejection (consumption posture)](./conversation-drive.md#waitforidle-never-rejects-presupposition) above. The contributor SHOULD confirm against the candidate `@earendil-works/pi-coding-agent` minor that `ExtensionCommandContext.waitForIdle()` still resolves rather than rejects on every settlement path Pi uses — clean `agent_end`, `ctx.abort()`-induced cancellation tear-down, and any internal error path — so that the loom-side MUST NOT against a wrapping `try`/`catch` and the post-`waitForIdle()` `stopReason: "error"` probe both stay valid. This rejection-vs-resolution behaviour is not encoded by the `Promise<void>` return type and is not enumerated by step 2(a)'s seven-capability literal-read assertion, so a silent regression would surface as a Node `unhandledRejection` from the prompt-mode driver and as a missed `stopReason: "error"` probe on the regressed failure modes with no load-time or SDK surface-inventory signal. SHOULD-level at loom 1.0; escalates to a build-time pin once Pi exposes a typed no-rejection contract that step 2(a)'s surface-inventory probe can mechanically verify.*

Edge cases the implementer must watch:

- Item-letter allocation: pick the next free letter after (ak); if the parked-finding queue includes other new bump items targeting the same release, sequence the letters in a single edit to avoid renumbering churn.
- The preamble of step 2 currently reads "items (a)–(ak)" and "items (f) through (aj)" in the SHOULD range. Both of those ranges, and the per-item-recording sentence in `V18c-version-bump-checklist.md`'s **Ships when** clause, must be widened in the same commit, or the new item will not be picked up by the per-item-outcome recording gate.
- The `V9c` plan leaf cites the *Conversation drive — prompt mode* section but does not currently enumerate the no-rejection presupposition; verify whether `V9c`'s harness fidelity contract (and `H4a`'s session-double) needs an explicit fidelity-axis bullet — the double's `waitForIdle()` must never reject — given that adjacent fidelity axes (streamed-token order, single-turn append, `pi.on` cancel-forward, cancellation propagation) are spelled out individually.
- Resist the temptation to weaken the inline `MUST NOT` to a SHOULD on the grounds that the contract is "merely presupposed." The `MUST NOT` is correct at the pin; the consumption-posture paragraph and the bump-checklist item are the surfaces that handle drift, not a downgraded normative claim.

## Consequence

**Severity:** advisory

At the loom 1.0 Pi-SDK pin the property holds and the spec, the MUST NOT, and the harness all work. The defect is a gap in the version-bump re-validation machinery: a future pi-coding-agent minor that started rejecting `waitForIdle()` would silently falsify a load-bearing prompt-mode contract — both the no-try/catch MUST NOT and the trailing-turn error-detection design — with no SDK-surface-inventory signal and no editorial-review prompt at the bump.
