## Finding

The phrase `closing gate` is a load-bearing canonical term in the plan: it names the single CI automation gate owned by `H5a` (`H5a-closing-gate-automation.md`, "REQ-ID / diagnostic-code closing-gate automation") and is used throughout `conventions.md` as "the loom 1.0 closing gate" — the gate that reconciles the spec REQ-ID set and the diagnostics code registry against `coverage-matrix.md` and the asserting tests.

`V7b`'s heading — "Diagnostic code registry and closing gate" — reuses that exact phrase for a different concept. `V7b`'s own `Adds.` is explicit that the leaf delivers "the closed-set + stable-id enforcement that the `H5a` gate **consumes**", and its `Tests.` bullets defer the gating verb to "the `H5a` gate's `Adds.`". `V7b` therefore owns the diagnostic-code registry and its closed-set/stable-id enforcement substrate — an input that `H5a`'s closing gate consumes — not the closing gate itself.

A reader or reviewer who knows the canonical term will read `V7b`'s heading as a claim that `V7b` owns `H5a`'s gate, mis-attributing gate ownership. The paired test leaf `V7b-T` carries the identical suffix, and the `docs/plan.md` §V7 table-of-contents entry mirrors the heading verbatim, so the collision is reproduced in three places.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** The plan corpus is git-tracked. `git log --follow` on `V7b-code-registry.md` shows two commits: the file was created in `c6a664e` with the heading `# `V7b` — Diagnostic code registry and closing gate` already present (`git log -L1,1:` confirms the first revision carries that exact line), and the later `f284534` edited only the DIAG-2 `Tests.`/`Ships when` wording, leaving the heading untouched. `H5a-closing-gate-automation.md` — the leaf establishing the canonical `closing gate` term — was created in the same `c6a664e` commit (`git show --stat c6a664e` lists both files added together). The term overload therefore existed from the moment both leaves were first authored; no later edit introduced or aggravated it.

## Solution Space

**Shape:** single

### Recommendation

Disambiguate `V7b`'s heading so it no longer reuses the bare canonical `closing gate` phrase, in all three locations that mirror it:

- In `docs/plan_topics/V7b-code-registry.md`, change the H1 heading suffix from `closing gate` to a term that names the registry substrate — e.g. `# `V7b` — Diagnostic code registry and code-registry gate` (or `... closed-registry enforcement`).
- In `docs/plan_topics/V7b-T-code-registry.md`, apply the same suffix change to the paired heading `# `V7b-T` — Diagnostic code registry and closing gate (tests)`, preserving the `(tests)` qualifier (paired-leaf convention requires the two headings stay in lockstep).
- In `docs/plan.md` §V7 — Diagnostics (the TOC list entry at the `V7b` line), update the link text to match the new heading.

Leave `V7b`'s `Adds.`/`Tests.` body references to "the `H5a` gate" and "fails the gate" unchanged — those correctly point at `H5a`'s closing gate, which `V7b` feeds. The fix is confined to the heading/TOC term; `H5a` and `conventions.md` remain read-only.

## Consequence

**Severity:** advisory

A reviewer working from the canonical `closing gate` term will read `V7b`'s heading as claiming `V7b` owns `H5a`'s CI gate, when `V7b` owns only the diagnostic-code registry substrate that gate consumes. The leaf bodies state the ownership correctly, so no implementer diverges; the cost is mis-attributed gate ownership during review and navigation.
