## Finding

The four co-sibling closing-gate arm leaves split out of `H5a` share a naming pattern: `H5d` ("Transitive-completeness plan-structural closing-gate **arm**"), `H5e` ("Un-anchored normative-MUST text-scan closing-gate **arm**"), and `H5f` ("Per-facet citing-test closing-gate **arm**") all end their headings in "arm". `H5c`'s heading alone ends in a different noun — "`no-broad-catch` allow-list closing-gate **reconciliation**".

`H5c` is the same kind of artifact as its siblings: its own body calls it an arm ("The CI gate **arm** that reconciles…", "This **arm** runs as part of the unified closing-gate machinery `H5a` establishes"). "Reconciliation" is the action this arm performs on the allow-list, not the leaf type; putting it in the heading's leaf-type slot makes `H5c` read as a different category of leaf from the three arms it sits beside in the Horizontal-phases list. The same divergent phrase is mirrored in `plan.md`'s Horizontal-phases link text for `H5c`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** cb8737a — pi-loom plan: resolve "Systemic leaf over-bundling across the plan corpus" (2026-06-12, Thomas Andersen); 617a3ad — pi-loom plan: resolve "H5a bundles gate arms spanning four data sources" (2026-06-16, Thomas Andersen); 5f33832 — pi-loom plan: resolve "H5a bundles seven gate arms behind a single Ships-when" (2026-06-18, Thomas Andersen); 26a8794 — pi-loom plan: resolve "split coverage-matrix rows can ship a facet untested" (2026-06-18, Thomas Andersen)
**History:** `H5c` was created in cb8737a with the heading ending in "reconciliation"; at that point no sibling "arm" closing-gate leaves existed, so the heading was internally consistent. The "...closing-gate arm" naming convention was established later by the `H5a` splits that produced `H5d` (617a3ad), `H5e` (5f33832), and `H5f` (26a8794). `H5c`'s heading line has never changed since cb8737a, so the divergence is the product of those later sibling commits adopting "arm" while `H5c` was not retro-fitted.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H5c-broad-catch-allow-list-gate.md`, change the line-1 heading's terminal word from `reconciliation` to `arm`, so the heading reads `` # `H5c` — `no-broad-catch` allow-list closing-gate arm ``. Apply the same change to the matching link text in `docs/plan.md` §Horizontal phases, so the `H5c` list entry reads `` [`H5c` — `no-broad-catch` allow-list closing-gate arm](./plan_topics/H5c-broad-catch-allow-list-gate.md) ``.

Leave the body's "reconciles" / "broad-catch allow-list reconciliation" wording in Adds, Tests, and Ships-when as-is — those describe the action the arm performs and are correct usages; only the leaf-type noun in the heading and the plan.md link text is the inconsistency.

## Consequence

**Severity:** cosmetic

A reader scanning the Horizontal-phases list sees `H5c` named with a different terminal noun ("reconciliation") from its three sibling "arm" leaves, suggesting it is a different kind of leaf when it is the same kind. No implementer is misled about behaviour and no gate is affected; the cost is labelling inconsistency within the H5 closing-gate family.
