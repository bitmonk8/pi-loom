## Finding

Both interleave notes in `docs/plan.md` — the one in §V9 (Extension host integration) and the one in §V11 (Binder) — present the seam `V9b → V11a → V9c`/`V9i`/`V9j`, naming `V9c` alongside `V9i` and `V9j` as the leaves gated on `V11a`'s binder-model resolution. The grouping is wrong for `V9c`. `V9c`'s `Deps` are `V9c-T, V9a, V8a, V17a` — no `V11a` (and no `V6a`) edge — and its `Adds` (the prompt-mode driver: `sendUserMessage`/`waitForIdle`, trailing-turn `Ok(string)` extraction, the `stopReason:"error"` probe, the cancel-forward `pi.on` subscription, and the active-set gating window) consumes no binder-model resolution. By contrast `V9i` (`Deps: … V11a …`) and `V9j` (`Deps: V9j-T, V4d, V11a`) both correctly declare the `V11a` edge, which is exactly why they belong in the seam.

So the note over-declares: a reader following the seam string concludes `V9c` cannot be picked up until `V11a` lands, while `V9c`'s own `Deps` (and the note's own closing instruction, "Sequence by **Deps**, not slice number") say it can land in parallel with binder work. The leaf text supports correcting the note rather than adding an edge — `V9c` has never declared `V11a` in its history and its `Adds` does not touch the binder-model contract.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** b9b427d — pi-loom plan: resolve "slice-ordering narrative overpromises DAG order" (2026-06-10, Thomas Andersen)
**History:** Both interleave notes were absent from `docs/plan.md` before b9b427d (the §V9 and §V11 notes are added as new `+` lines in that commit's diff) and were authored there with the `V9c`/`V9i`/`V9j` grouping already in place. `V9c`'s `Deps` never contained `V11a` (`git log -S "V11a" -- docs/plan_topics/V9c-conversation-drive.md` returns nothing), so the note has contradicted `V9c`'s declared edges from the moment it was written.

## Solution Space

**Shape:** single

### Recommendation

Drop `V9c` from both interleave notes in `docs/plan.md`, in both the prerequisite phrasing and the seam string:

- §V9 note: change "is itself a prerequisite of `V9c`/`V9i`/`V9j`, so the seam runs `V9b → V11a → V9c`/`V9i`/`V9j`" to "is itself a prerequisite of `V9i`/`V9j`, so the seam runs `V9b → V11a → V9i`/`V9j`".
- §V11 note: change "is a prerequisite of `V9c`/`V9i`/`V9j`, so it lands mid-V9 (seam `V9b → V11a → V9c`/`V9i`/`V9j`)" to "is a prerequisite of `V9i`/`V9j`, so it lands mid-V9 (seam `V9b → V11a → V9i`/`V9j`)".

Leave `V9i` and `V9j` in both notes — their `V11a` `Deps` edges are real. Do not edit `V9c`'s `Deps`.

Edge case for the implementer: if prompt-mode conversation drive is later changed to consume binder-model resolution, the correct response is the opposite — add `V11a` to `V9c`'s `Deps` and keep it in the seam. At the current leaf text `V9c`'s `Adds` consumes no binder-model resolution, so the note correction is the right fix now.

## Consequence

**Severity:** advisory

A reader following the interleave-note seam would defer `V9c`'s pickup behind `V11a`'s binder-model resolution, even though `V9c`'s `Deps` permit it to land in parallel with binder work and the note itself directs sequencing by `Deps`. No execution breakage results — `V9c`'s `Deps` are correct — but the note seeds doubt about whether `V9c` is missing a dependency edge.
