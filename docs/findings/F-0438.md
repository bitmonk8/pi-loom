## Finding

The non-`-T` half of a TDD leaf pair is referred to by two different nouns across the plan corpus, with no statement that they denote the same node. The §Per-phase TDD ritual in `conventions.md` defines the concept as the **"Implementation task"** (paired with the **"Tests task"**), and `leaf-template.md` reuses that "tests task / implementation task" framing. Elsewhere the same node is called the **"implementation leaf"**: `coverage-matrix.md`'s intro maps every executable REQ-ID "to the **implementation** leaf (`<id>`)", `plan.md`'s slice intro uses both terms in one sentence ("a paired `<id>-T` tests task and `<id>` implementation task; only the implementation leaf is linked"), and every `<id>-T` leaf's **Adds** field reads "Failing tests for the paired `<id>` implementation leaf."

The two terms are used interchangeably for one object — `plan.md` proves this by switching between them inside a single clause — so no task-vs-leaf distinction is actually maintained. A reader encountering "implementation task" in the TDD ritual and "implementation leaf" in the coverage matrix has no in-corpus signal that the two name the same thing.

(The original finding cited an "index canonical term"; the plan corpus has no separate terms index, so the inconsistency is grounded directly in the cross-cutting files above rather than against a glossary.)

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f9c5354` (established "implementation task"), `c6a664e` (introduced divergent "implementation leaf")
**History:** `git log -S 'Implementation task' -- docs/plan_topics/conventions.md` traces the term "implementation task" to `f9c5354` (2026-06-09, "add scope section + two-task TDD pairing convention"), which authored the §Per-phase TDD ritual and its "tests task / implementation task" vocabulary. `git log -S 'implementation leaf' -- docs/plan_topics/coverage-matrix.md` and the same probe over `docs/plan_topics/V10a-T-discovery-walk.md` and `docs/plan.md` all point to the next day's `c6a664e` (2026-06-10, "build/update plan for spec.md + review"), which introduced "implementation leaf" pervasively across the coverage matrix, every `<id>-T` Adds field, and plan.md's slice intro. The drift is the interaction of the two: `c6a664e` adopted "implementation leaf" as the object-noun without reconciling against the "implementation task" term `f9c5354` had defined the day before. Neither commit was wrong in isolation; the inconsistency is the unreconciled overlap.

## Solution Space

**Shape:** single

### Recommendation

Make the two names explicitly equivalent at the point where "Implementation task" is defined. In `docs/plan_topics/conventions.md` §Per-phase TDD ritual, step 2 ("**Implementation task** — leaf ID `<id>` …"), state that the implementation task's plan node is the same one referred to elsewhere as the **"implementation leaf"** — for example by appending to that definition a clause naming "implementation leaf" as the synonym used in `coverage-matrix.md`, the `<id>-T` Adds fields, and `plan.md`.

This resolves the ambiguity at its source while preserving the "tests task / implementation task" pairing symmetry the ritual relies on, and avoids re-terming the ~60 `<id>-T` Adds lines plus `coverage-matrix.md` and `plan.md` that already read "implementation leaf". The edit is confined to the cross-cutting `conventions.md` definition; do not touch the spec, and honour the existing leaf-ID scheme and `<new>`-placeholder conventions when editing.

## Consequence

**Severity:** cosmetic

Pure terminological drift: two nouns for one TDD-pair node, with no maintained distinction. An implementer following the TDD ritual and one reading the coverage matrix see different names for the same object; comprehension is essentially unaffected, but the inconsistency invites a reader to hunt for a task-vs-leaf distinction that does not exist.
