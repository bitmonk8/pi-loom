## Problem

Step 5's build-aside-then-publish swap (anchor `watcher-hot-reload-registration`) defines its failure behaviour exclusively for the *throw* path: if a rebuild step throws, the staging set is discarded, the prior `LoomRegistry` snapshot / validator cache / registration cache stay live, and `loom/runtime/registry-swap-failed` surfaces. But a staged re-parse of a `.loom` / `.warp` file does not throw on a schema-lowering or structural-subset rejection — those are *returned* as `loom/parse/*` / `loom/load/*` diagnostics. So the dominant failure mode — a staged file re-parses without throwing but yields `E`-severity diagnostics — has no defined swap-boundary outcome: it is unspecified whether the swap aborts (staging discarded, prior registry stays live) or the staged set publishes anyway, and whether `loom/runtime/registry-swap-failed` co-fires on this path. The `watcher-time-reload-failures` block in `discovery/package-and-settings.md` pins that such diagnostics are surfaced but is silent on the staging-set consequence.

## Issue introduction

**Verdict:** multi-commit-interaction

**Introducing commits:**

- `9e6bb1f` (2026-05-05) — *"pi-loom plan: resolve V18f watcher swap is not transactional and has no failure-mode test"* — introduced the build-aside-then-publish paragraph in step 5 with a single failure clause keyed to `throws`: *"If any rebuild step throws, the staging set is discarded …"*. The defect was present at this commit because parse-phase failures are diagnostic-bearing, not throwing, so the rule's domain was incomplete from the moment it was authored.
- `d488eaa` (2026-06-06) — *"pi-loom spec: resolve ERR-7 defining anchor + collision payload field"* — added the *Watcher-time reload failures* block in `discovery/package-and-settings.md` with the two-bullet ERR-7 elevation (registry-swap-failure vs re-parse / re-merge diagnostic). This commit pinned that re-parse diagnostics *are surfaced* but did not propagate a matching swap-side rule back into step 5, leaving the two pages in apparent agreement while still silent on the staging-discard / `registry-swap-failed` co-fire question.

**History:** `git log -G "If any rebuild step throws" -- docs/ spec_topics/` shows `9e6bb1f` as the introducing commit for the throw-keyed clause, with every subsequent edit on that paragraph preserving the throw-only wording verbatim. `git log -G "Re-parse / re-merge"` returns `d488eaa` as the sole authoring commit for the diagnostic-surfacing bullet, with no companion edit to step 5 in the same commit or any later commit. The corpus-split commit `f5e89f4` is purely structural (file split for the 100KB size cap) and does not touch the semantics. The defect is the *interaction* between the two paragraphs — neither author was wrong in isolation, but the union under-specifies one of the two dominant failure modes of the staged re-parse.

## Solution approach

Clarify the swap boundary in step 5 (`#watcher-hot-reload-registration`) to specify the diagnostic-only outcome: state whether an `E`-severity `loom/parse/*` or `loom/load/*` diagnostic emitted during the staged re-parse counts as a rebuild failure (staging set discarded, prior `LoomRegistry` / validator cache / registration cache stay live) or is published anyway, and state whether `loom/runtime/registry-swap-failed` co-fires on this path or stays the throw-only marker so the two failure modes remain distinguishable in operator logs. Anchor the `E`-vs-`W` cut on GOV-15's loads-cleanly predicate (`#gov-15-loads-cleanly`). Propagate the same swap-side consequence into the `#watcher-time-reload-failures` re-parse-diagnostic bullet in `discovery/package-and-settings.md` so the two pages agree, and into the `loom/runtime/registry-swap-failed` trigger column in `code-registry-runtime.md` so its scope contrasts the diagnostic-only path.

## Solution constraints

- Use GOV-15's effective-severity loads-cleanly predicate (`#gov-15-loads-cleanly`) for the `E`/`W` cut; do not introduce a new severity comparator.
