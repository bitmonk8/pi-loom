## Finding

Step 5 of *Registration steps* defines the watcher's build-aside-then-publish swap exclusively in terms of *throws*: "If any rebuild step throws … the staging set is discarded and the prior `LoomRegistry` snapshot, AJV validator cache, and prompt-mode registration cache all remain live; the failure surfaces as a single `loom/runtime/registry-swap-failed` diagnostic." But re-parsing a `.loom` or `.warp` file does not throw on a schema-lowering, structural-subset, or any other parse-phase rejection — by [`implementation-notes.md` Parser contract](../spec_topics/implementation-notes.md) and the entire `loom/parse/*` / `loom/load/*` registry, those rejections are *returned* as structured diagnostics, not raised. So the dominant failure mode of the staged re-parse — the staged file (or one of its transitive `.warp` importers) re-parses cleanly enough to *not* throw but yields one or more `E`-severity diagnostics — has no defined outcome at the swap boundary.

The adjacent rule in [`discovery/package-and-settings.md` — Watcher-time reload failures](../spec_topics/discovery/package-and-settings.md#watcher-time-reload-failures) gets one half of this right: its second ERR-7 bullet pins that re-parse / re-merge diagnostics *are surfaced* through the `loom-system-note` channel. But it does not state what happens to the staging set when those diagnostics fire: is the swap aborted (staging discarded, prior snapshot stays live, registry coherent), or is the staged set published anyway (operator's registry now contains a loom that re-parsed with `E`-severity errors)? Two implementers reading step 5 plus the package-and-settings ERR-7 elevation can defensibly land on either behaviour. The matching question — does `loom/runtime/registry-swap-failed` co-fire on the diagnostic-only path, or is it reserved for the throw path so the two failure modes stay distinguishable in operator logs — is likewise unanswered.

The fix is to state, at the swap boundary, that an `E`-severity `loom/parse/*` or `loom/load/*` diagnostic emitted during the staged re-parse counts as a rebuild failure: the staging set is discarded, the prior `LoomRegistry` snapshot / validator cache / registration cache stay live, the diagnostics themselves are the operator surface, and `loom/runtime/registry-swap-failed` does **not** co-fire on this path (it remains the throw-only marker, so an operator can tell the two failure modes apart from the code alone). `W`-severity diagnostics from the staged re-parse do not block the swap, mirroring the existing first-load *loads-cleanly* predicate.

## Issue introduction

**Verdict:** multi-commit-interaction

**Introducing commits:**

- `9e6bb1f` (2026-05-05) — *"pi-loom plan: resolve V18f watcher swap is not transactional and has no failure-mode test"* — introduced the build-aside-then-publish paragraph in step 5 with a single failure clause keyed to `throws`: *"If any rebuild step throws, the staging set is discarded …"*. The defect was present at this commit because parse-phase failures are diagnostic-bearing, not throwing, so the rule's domain was incomplete from the moment it was authored.
- `d488eaa` (2026-06-06) — *"pi-loom spec: resolve ERR-7 defining anchor + collision payload field"* — added the *Watcher-time reload failures* block in `discovery/package-and-settings.md` with the two-bullet ERR-7 elevation (registry-swap-failure vs re-parse / re-merge diagnostic). This commit pinned that re-parse diagnostics *are surfaced* but did not propagate a matching swap-side rule back into step 5, leaving the two pages in apparent agreement while still silent on the staging-discard / `registry-swap-failed` co-fire question.

**History:** `git log -G "If any rebuild step throws" -- docs/ spec_topics/` shows `9e6bb1f` as the introducing commit for the throw-keyed clause, with every subsequent edit on that paragraph preserving the throw-only wording verbatim. `git log -G "Re-parse / re-merge"` returns `d488eaa` as the sole authoring commit for the diagnostic-surfacing bullet, with no companion edit to step 5 in the same commit or any later commit. The corpus-split commit `f5e89f4` is purely structural (file split for the 100KB size cap) and does not touch the semantics. The defect is the *interaction* between the two paragraphs — neither author was wrong in isolation, but the union under-specifies one of the two dominant failure modes of the staged re-parse.

## Solution Space

**Shape:** single

### Recommendation

Add one paragraph to step 5 of `registration-steps.md`, immediately after the existing throw-clause, of the following shape:

> An `E`-severity `loom/parse/*` or `loom/load/*` diagnostic emitted while building the staging set is treated as a rebuild failure: the staging set, staging validator cache, and staging registration-cache entries are discarded; the prior `LoomRegistry` snapshot, AJV validator cache, and prompt-mode registration cache remain live; and the diagnostics surface through the standard persistent-diagnostic channel per [Diagnostics](../diagnostics.md). `loom/runtime/registry-swap-failed` does **not** co-fire on this path — it is reserved for the throw-clause above, so an operator can distinguish throw-side from diagnostic-side failures from the code alone. `W`-severity diagnostics from the staged re-parse do not block the swap (mirroring the [*Loads-cleanly predicate*](../governance/source-language-stability.md#gov-15-loads-cleanly)).

Update `discovery/package-and-settings.md` *Watcher-time reload failures*'s second bullet to state the same swap-side consequence ("the staging set is discarded; prior `LoomRegistry` remains live; `registry-swap-failed` does not co-fire"), so the two pages agree and neither is the sole source of truth on the swap-side rule.

Update the `loom/runtime/registry-swap-failed` row's *Trigger* column in `code-registry-runtime.md` to explicitly say "the watcher's build-aside-then-publish rebuild *threw* before publish" (it already implies this, but the contrast with the new diagnostic-only path should be made textual so the row's scope is unambiguous).

Edge cases the implementer must watch:

- The `E`-vs-`W` cut MUST use **effective severity** after any per-code `E/W` resolution rule, the same predicate `GOV-15`'s loads-cleanly check uses; do not invent a new severity comparator.
- A rebuild that produces both an `E`-severity diagnostic *and* later throws (e.g. parse succeeds with errors, then AJV recompile throws) takes the throw path: `registry-swap-failed` fires *in addition to* the surfaced parse diagnostics. The two failure-mode markers can co-exist within one rebuild; only the discard-and-revert outcome is shared.
- The `.warp` transitive-importer set is computed against the pre-swap import graph for the discard decision (so a file removed from the import graph by the failed rebuild does not change which entries get reverted). The post-swap graph rebuild that `V15e` keys its cache-eviction walk to is a separate, swap-conditional step.

## Consequence

**Severity:** correctness

Two compliant implementations will diverge at the diagnostic-only re-parse path: one discards the staged set and keeps the prior `LoomRegistry` live; the other publishes a registry in which a loom re-parsed with `E`-severity errors is now the dispatch target. The two operator-visible surfaces (next-call dispatch behaviour, which diagnostic codes co-fire) differ irreconcilably, and conformance tests cannot tell which is correct.
