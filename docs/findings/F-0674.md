## Finding

`diagnostics/diagnostic-shape.md`'s **Location-less diagnostics** paragraph pins the wire shape for codes that omit `file` and `range`, and closes the set to eight codes: the four `loom/load/*` codes (`host-incompatible`, `extension-bootstrap-failed`, `discovery-slow`, `package-read-timeout`) and the four `loom/host/session-shutdown-*` teardown codes. The closure rule is normative: "adding or removing a location-less code requires a spec edit to this paragraph (and, for an addition, the new code's registry row) before the omitted-fields form is permitted on the wire." `loom/runtime/reload-teardown-timeout` is not in that set.

But the same opening paragraph of `diagnostic-shape.md` names `loom/runtime/reload-teardown-timeout` as one of the five `console.error`-routed teardown carve-outs, and `code-registry-runtime.md`'s row for it describes a diagnostic emitted once per `session_shutdown` event after the bounded await on N in-flight `disposeBarrier`s exhausts its `SHUTDOWN_AWAIT_CAP_MS` budget. There is no single source location to attach: the diagnostic represents a handler-scoped timeout across an arbitrary number of invocations, and `code-registry-runtime.md` neither cites a `file`/`range` source nor exempts the row from carrying one. The structured `Diagnostic` envelope therefore requires fields the runtime has no value for.

`pi-integration-contract/diagnostic-emission-isolation.md` already exposes the contradiction: its *Wire format* clause says of all five `console.error` teardown codes that "the other optional fields of the `Internal diagnostic shape` — `file`, `range`, `related?`, `masked?` — are likewise not pinned by any of the five codes and MAY be omitted." That clause grants the omission permission to `loom/runtime/reload-teardown-timeout` directly; the diagnostic-shape.md closure rule denies it. Two implementers reading the two specs in either order arrive at different wire shapes, and a consumer that strictly validates the envelope against the location-less set as it stands today will reject every `loom/runtime/reload-teardown-timeout` emission.

## Issue introduction

**Verdict:** present-since-inception

**Introducing commits:** `b7d383a` (2026-06-06, "pi-loom spec: resolve \"Diagnostic envelope unsatisfiable for location-less codes\"")

**History:** The **Location-less diagnostics** paragraph and the closed eight-code set entered `docs/spec_topics/diagnostics/diagnostic-shape.md` in `b7d383a` as a single new block; `git show b7d383a -- docs/spec_topics/diagnostics/diagnostic-shape.md` lists exactly the four `loom/load/*` codes and four `loom/host/session-shutdown-*` codes carried today, with no `loom/runtime/*` member at any later commit (`git log --all -S 'location-less' -- docs/spec_topics/diagnostics/diagnostic-shape.md` shows `b7d383a` as the sole content-modifying commit on that token; subsequent commits to the file did not touch the set). The "five carved-out exceptions" framing that names `loom/runtime/reload-teardown-timeout` alongside the four `loom/host/session-shutdown-*` codes predates the location-less set (it has been in `diagnostic-shape.md`'s opening paragraph since the teardown contract landed in `5e8b499`, 2026-05-06), so the closure paragraph was authored against an already-present asymmetry between the five-code `console.error` carve-out set and the four-code `loom/host/session-shutdown-*` subset it chose to enumerate, and `loom/runtime/reload-teardown-timeout` was omitted at the location-less set's introduction rather than dropped later.

## Solution Space

**Shape:** single

### Recommendation

Add `loom/runtime/reload-teardown-timeout` to the closed location-less set in `diagnostic-shape.md`'s **Location-less diagnostics** paragraph, naming it as the one `loom/runtime/*` member alongside the four `loom/load/*` and four `loom/host/session-shutdown-*` members already enumerated. The set's closure sentence ("This set is closed in loom 1.0.0: adding or removing a location-less code requires a spec edit to this paragraph …") stands unchanged — the set remains closed; its membership just grows by one. The cross-reference in the same paragraph from the optional-marked `file?` / `range?` fields of the **Internal diagnostic shape** block already covers the new member by reference; no edit to the diagnostic shape block itself is required.

The `loom/runtime/reload-teardown-timeout` row in `code-registry-runtime.md` already implies the location-less posture by describing a handler-scoped timeout across N invocations with no per-invocation source span, and PIC's *Wire format* "MAY be omitted" clause already grants the omission permission for the five `console.error` teardown codes including this one; both then read consistently with the broadened set rather than against it. The opening paragraph's "five carved-out exceptions" framing in `diagnostic-shape.md` is independent of the location-less set's membership and does not need to be touched here.

Edge case the implementer must watch: the *Serialised content format* rule's carve-out — "When a diagnostic omits `file` and `range` (the location-less codes enumerated under **Location-less diagnostics** above), its line drops the leading `<file>:<line>:<col>:` segment and renders as `\"<code>: <message>\"`" — extends to `loom/runtime/reload-teardown-timeout` once it joins the set, but the row routes through `console.error` rather than the persistent-channel renderer, so the rendered-line carve-out is latent rather than load-bearing for this code; the change is correctness for the structured-envelope contract, not a renderer behaviour change.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one follows PIC's *Wire format* "MAY be omitted" clause and emits `loom/runtime/reload-teardown-timeout` with `file`/`range` absent; another follows `diagnostic-shape.md`'s closure rule and either fabricates a sentinel location to satisfy the envelope or stalls on the contradiction. Consumers that adopt the closed location-less set as their validation discriminator (the use the closure rule explicitly invites) reject every conformant emission.
