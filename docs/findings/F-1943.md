## Problem

The `V15f` arity Tests bullet (and the byte-identical bullet in `V15f-T`) states that more than the total `params:` count is "always `invoke-arity-too-many`, even when the callee is not statically resolvable," but drops the parse-time phase anchor and rationale that `invocation.md` §Argument arity ("Too many arguments") carries. Read in isolation the clause is ambiguous on two axes: what count is compared (and against what) when the callee is not statically resolvable, and whether "always" means "always at parse time" or "fires at runtime once resolved." The sibling too-few clause explicitly splits parse-time from runtime AJV fallback, so the unqualified "always" on the too-many clause is the weaker reading. The leaf is therefore strictly less precise than the spec it implements, and two implementers can construct divergent (or absent) too-many test arms.

## Issue introduction

**Verdict:** present-since-inception (of the `V15f` leaf)
**Introducing commits:** `5f4d271` (`pi-loom plan: resolve "V15a bundles ~14 independent test obligations into one impl+T pair"`, 2026-06-17), which created `V15f`/`V15f-T` by splitting the invoke-diagnostics obligations out of `V15a`.
**History:** `git log -S 'invoke-arity-too-many' -- docs/plan_topics/V15f-invoke-diagnostics.md docs/plan_topics/V15f-T-invoke-diagnostics.md` returns only `5f4d271`; `git show 5f4d271:docs/plan_topics/V15f-invoke-diagnostics.md` shows the arity bullet was authored in its current abbreviated form at the leaf's creation — the spec's "extra positional arguments cannot be matched to any param, and there is no runtime safety net possible" rationale and the explicit "Always a parse error" phase anchor were never present in the plan leaf. The spec sentence carrying that fuller wording predates the split (`git log -S 'even when the callee is not statically resolvable' -- docs/spec_topics/invocation.md` traces it to the `31ff060` docs reorg and earlier). The defect is original to `V15f`: the leaf was born under-specifying the spec it mirrors, not narrowed by a later edit.

## Solution approach

Rewrite the arity-too-many clause in the `V15f` Tests bullet (and the byte-identical clause in `V15f-T`) to restore the disambiguating content it dropped from `invocation.md` §Argument arity's "Too many arguments" bullet: pin the parse-time phase explicitly, carry the "no runtime safety net possible" rationale for why surplus positional arguments cannot defer to the runtime AJV check, and make the contrast with too-few (which does defer) legible. Mirror the spec's existing normative claim rather than coining a new one.

## Solution constraints

- The spec (`docs/spec_topics/invocation.md`) is read-only for this fix; reproduce its existing claim and do not resolve the implicit question of how the parser obtains the callee's `params:` count for a non-statically-resolvable callee by diverging from the spec — that is a spec-review matter.
- Apply the same edit to both `V15f` and `V15f-T`; their arity Tests bullets must remain byte-identical.
