## Problem

In the `system:` interpolation *Path grammar* prose, the opening sentence "There is no depth bound — arbitrary chains of `.Ident` are accepted" reads as unqualified syntactic and semantic acceptance. The per-step reachability rule that actually gates each `.Ident` — head `Ident` must name a declared `params` entry, each subsequent `.Ident` must name a reachable field of an *object* schema — is presented as a separate following statement rather than as part of the same acceptance claim. A reader who anchors on the first sentence may momentarily treat path acceptance as unconditional. The constraint is fully specified and backed by `loom/parse/system-interp-unknown-param` and `loom/parse/system-interp-bad-field`; the defect is presentational.

Later edits preserved the juxtaposition verbatim: 00681e0 (F-0700, 2026-06-20) rewrote only the trailing wire-name clause, and the shard split / size-cap reflow relocated the block into `frontmatter-fields-b-and-templates.md` unchanged. No earlier defect-free version of the passage exists.

## Solution approach

Rewrite the *Path grammar* depth statement so the field-reachability constraint is inseparable from it: scope "no depth bound" to syntactic chain length and fold head/step validation into the same acceptance rule, naming `loom/parse/system-interp-unknown-param` (head) and `loom/parse/system-interp-bad-field` (step). Behaviour is unchanged.
