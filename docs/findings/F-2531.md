## Problem

The canonical interpolation-stringification table (QRY-18, anchor `qry-18` in `query/query-escapes-stringification.md`) renders a `${expr}` interpoland by its Loom static type and enumerates exactly nine rows (`string`, `integer`, `number`, `boolean`, `null`, `Enum variant`, `array<T>`, `Schema-typed object`, `Result<T, E>`), with no row for a union type. Union-typed interpolands are reachable on every surface that feeds the table: a discriminated-union `params` field as `${param}` in `system:`, an `${obj[k]}` whose static type is a field-type union, and any `T | null` or primitive union (`string | integer`) binding in `@`-query templates. Because the table is keyed on static type and a union matches none of the nine rows, the rendered output is undefined and implementers diverge (compact `JSON.stringify` vs `String(...)` vs panic). The rendered text is fed verbatim to the model, so divergent rendering breaks the table's invariant that a value renders the same regardless of surface.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 0c9ce54 ("pi-loom spec: resolve \"Stringification rule for ${expr} interpolations is unspecified\"", 2026-05-04)
**History:** The corpus is git-tracked. `git log -S "Schema-typed object"` traces the stringification table to its sole introducing commit 0c9ce54, which added the QRY-18 table to the then-monolithic `spec_topics/query.md`. The diff for that commit shows the table was created with exactly the rows `string`, `integer`, `number`, `boolean`, `null`, `Enum variant`, `array<T>`, `Schema-typed object`, and `Result<T, E>` — no union row at any point. The later split into `query/query-escapes-stringification.md` (`git log --follow`) carried the table verbatim. The defect is therefore an omission present since the table first existed, not a regression introduced by a later edit.

## Solution approach

Add a union row to the QRY-18 table in `query-escapes-stringification.md` that defines union-typed interpoland rendering by dispatch on the resolved runtime value's concrete arm type, so the union reuses the nine existing rows rather than introducing its own rendering algorithm. The `system:` reference in `frontmatter/frontmatter-fields-b-and-templates.md` already defers to this canonical table, so it inherits the new row without a separate edit.

## Solution constraints

- A union arm typed `Result<T, E>` in interpolation position MUST still fire `loom/parse/interpolated-result` — the existing non-interpolable-`Result` invariant survives the union row.
