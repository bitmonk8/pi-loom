## Problem

`patch-skew-degradation.md` is presented in the PIC index TOC as "Patch skew degradation", but beyond its patch-skew contract paragraph it also carries the complete normative spec of the five-sub-step `session_shutdown` teardown handler, the **Per-step isolation** paragraph (with its closed eight-entry `details.call` label set), and the `interstitial-seam-tag-write-swallow` swallow rule. All three run unconditionally for every `event.reason` and are not patch-skew-specific behaviour. Every reader-driven discovery path for the teardown contract — the PIC index, cross-references citing `patch-skew-degradation.md §session_shutdown`, and downstream plan/coverage citations — leads away from where the rules actually live, and co-edits of the teardown surface land in a file whose name disclaims them.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `5e8b499` (2026-05-06) — pi-loom spec: resolve "Hot-reload teardown contract for the loom extension instance is unspecified"
- `75a9626` (2026-05-09) — pi-loom spec: resolve "SessionShutdownEvent.reason closed set has no build-time pin against the SDK type"
- `f5e89f4` (2026-06-04) — docs: enforce 100KB hard / 30KB recommended size cap on spec set

**History:** `5e8b499` introduced the five-sub-step `session_shutdown` handler as step 4 of the monolithic `spec_topics/pi-integration-contract.md` "Extension entry point" sequence, correctly nested under registration steps. `75a9626` then inserted the **Patch-skew degradation contract** paragraph immediately above the handler prose in the same file (the introductory sentence of the patch-skew paragraph ends with "The handler runs in fixed order regardless of `event.reason`:" followed by the five sub-steps). At that point the two paragraphs were physically adjacent inside one monolithic file with no internal heading boundary between them. `f5e89f4` (the 100KB-cap split commit) created `patch-skew-degradation.md` as a new file by mechanically partitioning that monolithic source; because no heading separated the patch-skew prose from the teardown-sequence prose that followed it, the splitter swept both — plus the trailing **Per-step isolation** paragraph and the interstitial-seam swallow rule — into the single new child file. The misplacement is therefore not authored as such; it is an emergent artefact of (i) two unrelated edits landing without a section break between them and (ii) a later mechanical splitter that respected only existing heading boundaries.

## Solution approach

Split the five-sub-step `session_shutdown` handler, the **Per-step isolation** paragraph, and the `interstitial-seam-tag-write-swallow` rule out of `patch-skew-degradation.md` into a new dedicated child page under `docs/spec_topics/pi-integration-contract/`, leaving the patch-skew contract paragraph behind with a forward-reference to the new page. Add the new page to the PIC index TOC in `pi-integration-contract.md`. Rewrite the inbound cross-references and plan-side citations that key on the old location — in `session-only-degraded-state.md`, `drain-state-contract.md`, `unknown-reason-rule.md`, the plan leaves, and `coverage-matrix.md` row `cka-31` — to point at the new page.

## Solution constraints

- Preserve every moved `<a id="...">` anchor (the `substep-1-*` set and `interstitial-seam-tag-write-swallow`) at its current fragment id so inbound cross-references continue to resolve.
- Out of scope: the `drain-state-contract.md` *Read-failure fallback* anchor and the `session-only-degraded-state.md` acceptance-criterion (l) surface, each owned by its own sibling finding.
