## Problem

The loom-owned depth walk (`V5e`) emits a `maxDepth` `ValidationIssue` whose `path` is "the JSON Pointer to the first too-deep node" — a normative contract value surfaced at all five ceiling-#4 boundaries and rendered by the `V11f` fast-fail `<ajv-summary>`. Neither the spec's §Depth Enforcement nor `V5e` pins the order in which a node's object members are visited, so for a payload with more than one sibling subtree that exceeds depth 5, "first" is non-deterministic: two conformant implementations emit different `path` pointers for identical input. Every current fixture uses single-chain payloads, so the divergence is invisible to the test suite.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `216130d` — "pi-loom spec: resolve \"Depth ≤5: counting algorithm and enforcement point unspecified\"" (2026-05-04)
**History:** The normative depth-enforcement section was authored into `spec_topics/schema-subset.md` at `216130d`, lifting the "recursive descent … the first node whose depth would exceed 5 short-circuits" text and the `path: <JSON Pointer to first too-deep node>` error shape out of the prior spec-review proposal. That commit pinned the depth-counting algorithm and the five enforcement points (the finding it closed) but never specified object-member visitation order, so the JSON-Pointer determinism gap has been present since the section first entered the normative corpus. Later edits to the section (`git log --follow` / `git log -S "first node whose depth would exceed"` show `f5e89f4` size-cap reflow, `31ff060` / `81bf52a` doc restructuring) carried the text forward unchanged. The plan inherited the gap when the V11f depth-walk fast-fail rendering bullet referencing "the path to the first too-deep node" was authored at `62d0251` (2026-06-11) and re-touched at `151f5b8` (2026-06-12); neither plan commit introduced the ambiguity, they propagated the pre-existing spec wording.

## Solution approach

In `V5e`'s **Adds** (`docs/plan_topics/V5e-depth-enforcement.md`), specify the deterministic node-visitation order the loom-owned walk uses — object members in parsed source/insertion order and array elements in ascending index — so the `path` for the first too-deep node is reproducible. Add a Tests bullet to `V5e-T` (mirrored in `V5e`) that drives a payload with two sibling depth-6 branches through the walk and asserts the emitted `path` is the first branch's pointer.

## Solution constraints

- The spec (`docs/spec_topics/`, incl. `schema-subset.md`) is read-only for this fix — pin the visitation order in the plan's `V5e` leaf for the loom-owned walk, not in the spec.
- Keep the `V5e` and `V5e-T` Tests sections mirrored per the paired-leaf convention.
