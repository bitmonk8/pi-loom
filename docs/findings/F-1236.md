## Finding

`V14b`'s `Adds.` field appends a ~100-word rationale block to its single descriptive sentence. The block reframes the concurrent settle-all batch as a real-Pi "consumption posture" rather than a loom-side guarantee, observes that `@earendil-works/pi-coding-agent` pins no typed contract for it and the `H4a` session double does not enumerate parallel-batch delivery, and states that detection of a drift "routes to editorial review per version-bump checklist item **(an)**… mirroring the way SLSH-2's real-Pi streaming-ordering behaviour routes to item (ak)."

This is governance-process context plus a cross-leaf analogy, not an implementation requirement. Per `conventions.md` §Leaf format, `Adds.` is descriptive by default and the binding obligations of the leaf are its `Tests.` REQ-ID bullets and `Ships when` gate — here the `cka-13` Tests bullet (settle-all-before-next-turn, independent per-sibling lowering, failing sibling lowered as `isError: true`) fully captures the normative loom-side content. The appended paragraph adds nothing an implementer must act on.

The routing it describes is already owned authoritatively elsewhere: `version-bump-step2.md` item **(an)** (`#bump-checklist-parallel-tool-batch-delivery`) enumerates the parallel-tool-batch-delivery presupposition and prescribes the editorial-review audit, and `tool-calls.md#parallel-tool-batch-delivery-presupposition` defines the consumption posture itself. The `Adds.` paragraph therefore restates two external authorities inside a leaf's descriptive field, creating a two-source maintenance burden: if item (an) or the consumption-posture anchor changes, this prose goes stale with no gate to catch it. The paired `V14b-T` leaf does not carry the paragraph (its `Adds.` is the standard "Failing tests for the paired V14b implementation leaf.").

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V14b-tool-calls-parallel-batch.md`, strike the appended governance/rationale sentence from the `Adds.` field — the run beginning "The concurrent-execution, settle-all-before-next-turn, and independent-per-sibling-lowering batch shape this leaf models is a real-Pi parallel-tool-mode **consumption posture**…" through "…mirroring the way SLSH-2's real-Pi streaming-ordering behaviour routes to item (ak)." Leave the first descriptive sentence (the `Promise.allSettled` / `cka-13` sentence) intact.

No relocation is required: the editorial-review routing is already owned by `version-bump-step2.md` item (an) (`#bump-checklist-parallel-tool-batch-delivery`) and the consumption posture by `tool-calls.md#parallel-tool-batch-delivery-presupposition`, both read-only for this fix. The deleted prose adds no obligation those authorities do not already carry. No edit to `V14b-T` (its `Adds.` carries no such paragraph), and no change to `Tests.`, `Deps.`, `Ships when`, or the coverage matrix.

## Consequence

**Severity:** advisory

No implementer is misled, because `Adds.` is non-binding and the normative behaviour lives in the `cka-13` Tests bullet and `Ships when`. The cost is a maintenance/staleness hazard: the paragraph duplicates the routing authoritatively owned by `version-bump-step2.md` item (an) and the spec's consumption-posture anchor, so a future edit to either can silently desynchronise the leaf prose, and it inflates `Adds.` past the one-sentence descriptive convention.
