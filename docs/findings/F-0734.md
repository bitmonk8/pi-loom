## Finding

DISC-5 says: "When no plain (un-prefixed) include pattern is present, the starting set is every path under the package root; otherwise the starting set is the union of the paths matched by the plain include patterns." The phrase "every path under the package root" does not say whether the enumeration is recursive (every transitive descendant of the root) or flat (only the root's immediate children). The rest of the spec is uniformly non-recursive — the global `.pi/looms/` / `~/.pi/agent/looms/` walks, the `looms/` fallback inside a package, and a directory-shaped match inside DISC-5 itself are each explicitly scanned **non-recursively** for `*.loom` children — so the absence of any non-recursion qualifier on the no-plain-includes default reads as a deliberate departure, but the paragraph does not state which direction it departs in.

The two readings produce materially different registered sets for a nested package. Under the recursive reading, a package shipping `subdir/foo.loom` with `pi.looms: ["!subdir/**"]` would have `subdir/foo.loom` enter the starting set and then be excluded; under the flat reading, the same exclude pattern is vacuous because `subdir/foo.loom` was never in the starting set in the first place. The interaction with `**/*.loom`-style patterns elsewhere in the `!`/`+`/`-` machinery is correspondingly undefined — DISC-5 pins the four-step override order but never pins what the override steps are acting *over*.

The Pi reference symbol DISC-5 cites for behavioral provenance (`applyPatterns` in `@earendil-works/pi-coding-agent` `dist/core/package-manager.js`) is operated on a `collectFilesFromPaths` output that walks each directory **recursively** (via `collectResourceFiles` → `collectFiles`). If "parity with Pi" is the intent, the recursive reading is the implied one — but DISC-5 does not name parity here, and recursion contradicts the corpus-wide non-recursion stance, so an implementer cannot tell which premise wins.

## Solution Space

**Shape:** multiple

### Option A — Flat (match the corpus-wide non-recursion stance)

**Approach.** Pin the no-plain-includes default to the package root's immediate children only, mirroring the non-recursive walks already specified for `.pi/looms/`, `~/.pi/agent/looms/`, and the `looms/` fallback. A `**/*.loom` plain include pattern is then the only way to register `.loom` files in nested subdirectories.

**Spec edits.** In the DISC-5 paragraph, replace "the starting set is every path under the package root" with "the starting set is every immediate child of the package root (the package root scanned non-recursively, matching the global non-recursion rule at the top of this file)". Add a clarifying sentence: "Patterns such as `**/*.loom` reach nested files by being authored as plain include patterns, which take their matches from a recursive enumeration scoped by the pattern itself rather than from the default starting set."

**Pros.** Internally consistent with every other walk in the spec. Default behaviour for a package that ships no `pi.looms` field (which falls through to the `looms/` non-recursive fallback) and a package that ships `pi.looms` with overrides-only entries now produce the same shape of registered set. Implementer cannot accidentally surface deep `.loom` files just by writing `pi.looms: ["!secret.loom"]`.

**Cons.** Diverges from Pi's `applyPatterns` semantics, where the default starting set is recursive. Packages that today rely on Pi-like recursion (none yet exist for `.loom` — `loom 1.0` is unreleased) would need an explicit `**/*.loom` include.

**Risks.** Future "behavioural parity with Pi" claims in DISC-5 become harder to make truthfully; the over-prescription finding elsewhere in this review may want to invoke parity, and would have to carve this case out.

### Option B — Recursive (match Pi's `applyPatterns` reference)

**Approach.** Pin the no-plain-includes default to a recursive enumeration of every descendant under the package root, matching the behaviour of Pi's `collectFilesFromPaths` → `collectResourceFiles` → `collectFiles` chain that DISC-5 already cites for the matching contract.

**Spec edits.** In the DISC-5 paragraph, replace "the starting set is every path under the package root" with "the starting set is every path under the package root, enumerated recursively across all subdirectories (departing from the corpus-wide non-recursion rule because DISC-5 inherits its starting-set semantics from Pi's `applyPatterns`; the non-recursion rule still applies to the *per-match contribution* step — a directory match continues to be scanned non-recursively for `*.loom` children — and to the `looms/` fallback path)". Add an example or anchor explaining that `**/*.loom`, `*.loom`, and `subdir/*.loom` all behave as in Pi's resource arrays.

**Pros.** Preserves behavioural parity with Pi's resource-array dialect, which DISC-5 already names as its reference. Override patterns like `!subdir/**` and `+keep.loom` work uniformly regardless of nesting depth. Packages can ship `.loom` files anywhere in their tree without authoring an explicit recursive include.

**Cons.** The package-discovery walk's depth differs from every other discovery walk in the spec, requiring a callout each time the non-recursion rule is invoked elsewhere. The DISC-6 bounded-walk machinery (`scanPackagesMaxFiles`, per-read deadlines) was framed against `package.json` reads, not `.loom` enumerations; an implementer may need to confirm whether the recursive enumeration is itself bounded or whether DISC-6's bounds happen to dominate.

**Risks.** Authors of small packages may accidentally register `.loom` files buried in `node_modules`-like nested test fixtures; the `!` / `-` machinery is the only escape hatch.

### Recommendation

Adopt **Option A (flat)**. Three independent forces converge on it: (i) every other walk in the spec is explicitly non-recursive, so a flat default fits the corpus's existing reader model with no carve-out; (ii) the `looms/` fallback that fires when `pi.looms` is absent is non-recursive, so making the `pi.looms`-present default flat keeps the with-vs-without-manifest behaviours symmetric; (iii) authors who genuinely want deep coverage can write `pi.looms: ["**/*.loom"]`, which is the explicit, audit-able way to opt in. The Pi-parity argument is the only force pulling toward recursive, and DISC-5 already qualifies its Pi reference as "pins loom's *use* of that matching, not minimatch's glob dialect, which the package owns" — the same hedge legitimises diverging on the starting-set shape.

Edge cases the implementer must watch: a `pi.looms` entry like `["!secret.loom"]` (override-only) under Option A produces an *empty* starting set, hence registers no looms; this is the same observable as the package shipping no `pi.looms` field but no `looms/` directory either. The corresponding `loom/load/manifest-invalid` / per-entry diagnostics are unchanged; "no `.loom` files registered" is silent today and should remain silent here.

## Consequence

**Severity:** correctness

Two reasonable implementers would diverge on every package that ships `.loom` files outside its package root's immediate level. The DISC-5 `!`/`+`/`-` override machinery becomes partially inert under one reading and fully active under the other, and the parity-with-Pi reading (recursive) directly contradicts the surrounding non-recursive ethos, so the choice is non-obvious.
