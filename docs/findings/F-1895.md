## Finding

The retired `cka-12` row in `coverage-matrix.md`'s *Code-keyed obligation areas* table carries the closing-leaf cell `*(numbered above)*`. The *Closing-leaf column authoring* rule requires every leaf-ID in a closing-leaf cell to be backtick-delimited, and recognises exactly one empty-cell form — the literal bare (unbackticked) `<new>` placeholder. `*(numbered above)*` is neither: it contains no backtick-delimited span and is not the `<new>` token.

The `H5d` transitive-completeness gate tokenises each closing-leaf cell by taking exactly its backtick-delimited spans, so `*(numbered above)*` tokenises to the empty set. `H5d` then requires at least one listed leaf to be a member of `H5b`'s expanded `Deps.` and reddens CI on "a cell none of whose listed leaves is in that expanded set"; the empty set satisfies that failure predicate vacuously. `H5d` excludes only the literal `<new>` placeholder rows — and the *Closing-leaf column authoring* rule is explicit that an empty non-`<new>` cell becomes a spurious gate failure, not a benign exclusion. Because `H5d` reads only ever-present plan files, it is not deferred to the live-corpus flip; the spurious failure fires the moment the arm runs against the real matrix.

The row no longer owns any code-keyed obligation: the `query/` subtree's obligations are coined as numbered `QRY-1 … QRY-21` REQ-IDs and mapped per closing leaf (`V13a`, `V13b`, `V13c`, `V13d`, `V13g`, `V13h`) in the *Numbered REQ-IDs* table. The `cka-12` token is cited by no plan leaf and by no `// allow:` / `// allow-broad-catch:` exemption — it appears only in the row itself.

The non-conforming cell entered the corpus at that commit and was an active defect immediately, since the governing rule and gate predate it.

## Solution Space

**Shape:** single

### Recommendation

Delete the entire `cka-12` row from the *Code-keyed obligation areas* table in `docs/plan_topics/coverage-matrix.md`. The `query/` subtree's obligations are fully carried by the numbered `QRY-1 … QRY-21` rows in the *Numbered REQ-IDs* table (closing leaves `V13a`, `V13b`, `V13c`, `V13d`, `V13g`, `V13h`, covering all 21 IDs), so removing the retired code-keyed row strands no obligation: `H5d` no longer sees an empty cell, `H5e` sees no un-anchored MUST, and `H5f` derives no facets.

Edge cases and constraints:
- Preserve the *Token* column's append-only discipline: `cka-13 … cka-48` keep their existing tokens unchanged and `cka-12` is not reissued. The resulting gap in the `cka-<n>` sequence is acceptable because the tokens are row-unique identifiers, not positional.
- Reference-safe to delete: no plan leaf, no `// allow:` / `// allow-broad-catch:` exemption, and no `H5f` facet currently cites `cka-12` (the token appears only in this row), so deletion breaks no resolver.

## Consequence

**Severity:** blocking

Once `H5d`'s transitive-completeness arm ships, every CI run scans the real `coverage-matrix.md`, tokenises the `cka-12` cell to the empty set, and reddens the closing gate vacuously — a false failure on a legitimately-retired row that prevents the gate from going green and blocks merges behind it.
