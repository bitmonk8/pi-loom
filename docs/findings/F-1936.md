## Finding

The `V13f` leaf is titled "`@`-query provider swallowing-handler per-site routing" in its heading, in its `plan.md` table-of-contents entry, and in its paired `V13f-T` leaf, and its body is scoped entirely to the per-site swallowing-handler routing for the `@`-query provider abandonable-Promise — the `@`-query entry in the four-site routing set `V17a` delegates to (`V14f`, `V13f`, `V15h`, `V9o`). Its filename, however, is `V13f-query-cancellation-routing.md` (and `V13f-T-query-cancellation-routing.md`), using the broader "cancellation-routing" stem.

This breaks the sibling naming pattern shared by the other three members of the same four-site routing set, whose filenames carry the "swallowing" scope token: `V9o-subagent-swallowing-handler.md`, `V14f-tool-calls-swallowing-handler.md`, and `V15h-invoke-ceiling-swallowing.md`. The `V13f` files are the only members whose filename advertises a wider scope ("cancellation-routing") than the leaf actually owns.

The leaf-ID `V13f` is correct and is what every `Deps.` field and prose cross-reference uses; the only filename-level reference outside the two files is the markdown link target in `docs/plan.md`'s slice-V13 table of contents.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `eaa5000` (2026-06-16) — `pi-loom plan: resolve "V13c bundles four independent spec obligations into one leaf"`
**History:** `git log --follow --diff-filter=A` shows `V13f-query-cancellation-routing.md` was added in `eaa5000`; `git show eaa5000:docs/plan_topics/V13f-query-cancellation-routing.md` confirms the heading was already "`@`-query provider swallowing-handler per-site routing" at that commit, and the `plan.md` TOC link (`git log -S 'V13f-query-cancellation-routing' -- docs/plan.md`) was introduced in the same commit. The filename/title mismatch therefore entered the corpus when the leaf was first authored; no later commit changed the title away from the filename.

## Solution Space

**Shape:** single

### Recommendation

Rename the two files to carry the swallowing-handler scope token, matching the sibling pattern, and retarget the single TOC link:

- Rename `docs/plan_topics/V13f-query-cancellation-routing.md` → `docs/plan_topics/V13f-query-swallowing-handler.md`.
- Rename `docs/plan_topics/V13f-T-query-cancellation-routing.md` → `docs/plan_topics/V13f-T-query-swallowing-handler.md`.
- In `docs/plan.md` slice-V13 table of contents, update the link target from `./plan_topics/V13f-query-cancellation-routing.md` to `./plan_topics/V13f-query-swallowing-handler.md`.

The leaf-ID `V13f`/`V13f-T` stays unchanged, so no `Deps.` field needs editing — every `Deps.` and prose cross-reference already uses the backtick-delimited leaf-ID, not the filename. The `docs/plan.md` TOC link is the only filename-level cross-reference in the plan corpus that requires retargeting.

## Consequence

**Severity:** cosmetic

A reader scanning the plan-topics directory sees `V13f` advertise a "cancellation-routing" scope broader than its actual swallowing-handler-only content, and the file breaks the otherwise-uniform `*-swallowing-handler*` / `*-swallowing*` naming of the four-site routing set. No mechanical gate keys on the filename (the leaf-ID `V13f` is unaffected and `Deps.` references use the ID), so no implementer is blocked; the cost is navigational/organisational only.
