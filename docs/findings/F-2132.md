## Problem

The `Diagnostic.range` shape in `diagnostic-shape.md` pins `column` as "1-indexed; end exclusive" but never states what unit `column` counts, and the *Serialised content format* renders that column into the `<col>` of every diagnostic line and each `related` entry line. The `lexical.md` *Diagnostic spans* bullet fixes line/column to logical lines on the BOM-stripped, newline-normalised stream but is likewise silent on the column unit. The spec corpus uses two different units in adjacent normative surfaces — string `length` and ordering are UTF-16 code units (`expressions.md`, "matches JS `.length`") while message truncation and the §8 placeholder caps are Unicode code points — so a reader cannot infer the column basis by analogy. For any token following a non-BMP (or non-ASCII) character, implementations counting UTF-16 code units, Unicode code points, or UTF-8 bytes diverge on the reported column, observable through DIAG-4's character-for-character message and in `details.diagnostics` consumed by LSP integrations.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** fae85d3 — Tighten spec for implementation-plan readiness (2026-05-04, Thomas Andersen)
**History:** The `range: { start: { line, column }, end: { line, column } } // 1-indexed; end exclusive` shape was first authored in fae85d3 and already lacked any column-unit statement; `git log -S 'code unit'` over `diagnostic-shape.md` returns nothing, confirming the unit was never present. Later commits only relocated the shape — fecb504 split the monolithic spec into per-topic files and f5e89f4 carved `diagnostic-shape.md` out under the size cap — without ever pinning the unit. The omission has been continuous since the shape's first authored form.

## Solution approach

State that `column` — on `Diagnostic.range`, on `related[].range`, and in the rendered `<col>` — is counted in UTF-16 code units, 1-indexed, the same basis as string `length` and string ordering in `expressions.md`. Add this to the `range?` field comment in `diagnostic-shape.md` alongside the existing "1-indexed; end exclusive" note. Extend the `lexical.md` *Diagnostic spans* bullet with the same code-unit statement so the lexer's column-advance rule and the renderer agree at a single normative basis.
