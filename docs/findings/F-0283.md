## Problem

The `V10a` discovery-walk leaf and its paired `V10a-T` tests leaf each carry two `Tests.` bullets that both cite `DISC-2`: the canonical spec failure-modes rule ([`discovery-sources.md` §DISC-2](../spec_topics/discovery/discovery-sources.md#disc-2)) and a later-added cross-leaf integration assertion that a `loomPaths` entry from `V10c`'s merged settings reaches the Settings discovery source. These are two materially different test obligations sharing one identifier, and the spec defines `DISC-2` as a single rule with no provenance for the settings-application assertion. The coverage-matrix row `| DISC-1, DISC-2, DISC-3 | V10a |` cannot indicate which bullet is the `DISC-2` closure, and a reader cannot tell whether the second bullet adds work beyond the first. The duplication exists byte-for-byte in both leaves.

## Issue introduction

**Verdict:** single-commit-regression
**Introducing commit:** `e965634` — "pi-loom plan: resolve \"V10c settings keys read but application not asserted\"" (2026-06-11)
**History:** The canonical `DISC-2` *Failure modes* bullet has been present on `V10a` / `V10a-T` since plan inception (`c6a664e`, 2026-06-10). `git log -S` over the distinctive string "Settings source applies the merged value" shows the second `DISC-2` bullet was added by `e965634`, the fix that closed the "V10c settings keys read but application not asserted" gap. That fix correctly introduced a settings-application integration assertion (and added `V10c` to `Deps.`) but reused the existing `DISC-2` label instead of giving the new assertion an unambiguous identity — the collision entered with the otherwise-sound coverage addition. The defect is therefore localised to the new bullet, not to the pre-existing failure-modes bullet.

## Solution approach

Leave the first bullet as the sole `DISC-2` citation on both `V10a` and `V10a-T`, and rewrite the second (settings-application) bullet so it no longer carries the `DISC-2` prefix, retaining its full assertion text as an uncited integration note-bullet so the coverage `e965634` added is not lost. The spec carries no numbered REQ-ID for this `loomPaths`/`V10c` union-membership assertion, so it stays uncited rather than re-pointed at a spec rule.

## Solution constraints

- Keep `docs/plan_topics/V10a-discovery-walk.md` and `docs/plan_topics/V10a-T-discovery-walk.md` byte-for-byte identical after the edit — they are red/green mirrors.
- The spec is read-only for this fix.
- Do not mint a `DISC-2a` / `DISC-2b` sub-identifier on the plan leaves; the spec's `DISC-2` is one rule and a sub-label would imply a spec sub-rule that does not exist (REQ-ID / inline-label discipline, `conventions.md`).
- If the settings-application bullet is instead re-cited to a numbered REQ-ID, keep `coverage-matrix.md` and the `H5b` / `H5d` transitive-completeness obligations consistent with the new co-witness.
