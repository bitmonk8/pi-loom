## Problem

In `cancellation.md`'s *Slash-command entry* bullet, the `agent_end`-driven cancellation trigger ("an `agent_end` event reporting a user-cancelled turn aborts `loomAbort`") is a *distinct* trigger from the `ctx.signal`-aborted arm — CNCL-4 gives it its own synthesised reason precisely because it has no source `AbortSignal` — so it cannot reduce to a `ctx.signal.aborted` check. But the page never states which field or shape of the `agent_end` event signals a "user-cancelled turn." PIC-18's forward-reference points back at these forwarding rules for the cancellation-relevant predicate, yet the rules never define it, so the reference closes on nothing. The `AgentEndEvent` payload carries no top-level `reason`/`cancelled`/`aborted` discriminator, leaving the predicate entirely to the implementer.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** 8395c03 — pi-loom spec: resolve "Cancellation source ctx.signal is undefined when a slash-handler starts" (2026-05-04, Thomas Andersen); f4b6738 — pi-loom spec: resolve "loomAbort ↔ ctx.signal linkage and reason-propagation" (2026-05-07, Thomas Andersen)
**History:** Commit 8395c03 first wrote the *Slash-command entry* bullet, including the clause "an `agent_end` event reporting a user-cancelled turn aborts `loomAbort`" — verbatim its current wording — without ever defining how a user-cancelled `agent_end` is recognised. Commit f4b6738 then added the CNCL-4 *Abort-reason propagation* rule, whose "`agent_end`-driven trigger has no source `AbortSignal` … so the runtime synthesises a reason `\"loom cancelled by agent_end\"`" clause cemented the arm as a *distinct* trigger that cannot reduce to the `ctx.signal`-aborted path. The undefined-predicate gap was latent from 8395c03 and became load-bearing once f4b6738 made the trigger non-reducible.

## Solution approach

Rewrite the *Slash-command entry* bullet's `agent_end` clause to define the predicate explicitly: a slash-command-path `agent_end` signals a user-cancelled turn when the trailing `assistant` message in the event's `messages` array carries `stopReason: "aborted"`, mirroring the existing prompt-mode `stopReason: "error"` trailing-`assistant` probe. Record loom's dependence on the externally-owned `AssistantMessage.stopReason: "aborted"` value as a consumption posture routed to the Pi version-bump editorial-review checklist, alongside the other `AssistantMessage`-field postures.

## Solution constraints

- The `agent_end` trigger MUST NOT be implemented by reading `ctx.signal.aborted` — it fires when there is no aborted source signal (including when `ctx.signal` is `undefined`), which is why it exists as an arm distinct from the `ctx.signal`-aborted trigger.
- The trigger MUST re-check against the active invocation's captured state, because `pi.on` `agent_end` events are process-global and carry no per-session origin marker (PIC-18), so an unrelated session's aborted turn must not spuriously fire `loomAbort.abort()`.
