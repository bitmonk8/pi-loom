## Finding

`H1a`'s **Ships when** states that the green `npm install` "is the witness that the in-tree `eslint-plugin-loom-local` package at `./tools/eslint-plugin-loom-local` resolves **and loads**". `npm install` of a `file:` specifier resolves the path and reads the target package's `package.json` (it links/copies the package into `node_modules`), but it does not execute the plugin's entry module. Module loadability — that the entry exports the shape ESLint expects, with no ESM/CJS mismatch or import-time throw — is therefore not exercised by `npm install`.

`H1a` wires no ESLint config and runs no lint pass (the `no-broad-catch` rule and its wiring into `npm test` are owned by `H2a`, `Deps: H1a`), so the plugin entry module is first actually loaded at `H2a`. The leaf's own parenthetical already concedes the architectural devDependencies-key test "checks only the manifest key, not loadability" — but the green `npm install` it then points at does not close that gap either. A placeholder plugin that resolves but fails to load (bad `rules` export shape, ESM/CJS mismatch, import-time throw) passes `H1a` green and only reddens at `H2a`.

## Issue introduction

**Verdict:** introduced by a later edit (single commit)
**Introducing commits:** `220df7e` — "pi-loom plan: resolve \"eslint-plugin-loom-local in-tree plugin package has no creating leaf\"" (2026-06-12)
**History:** Before `220df7e`, `H1a`'s Ships-when read only "A fresh checkout runs `npm install && npm run build && npm run typecheck && npm test` green with zero production source files" — no loadability claim at all. That commit added the in-tree plugin-creation obligation to **Adds** and, in the same edit, appended the "the green `npm install` is the witness that the in-tree `eslint-plugin-loom-local` package … resolves and loads" clause to Ships-when. The overclaim entered with that clause: `npm install` resolves the `file:` specifier but never executes the entry module, so the "loads" half of the witness was untrue from the moment it was written. Subsequent edits (`ce48e09`, `4b4382f`) only re-pointed citations elsewhere in the leaf and left the Ships-when clause verbatim.

## Solution Space

**Shape:** multiple

### Option A — Add a loadability assertion at `H1a`

**Approach.** Add an `H1a` **Tests** bullet that imports `eslint-plugin-loom-local` from loom's `node_modules` and asserts the module loads and exposes a `rules` export (an object), so a bad export shape, an import-time throw, or an ESM/CJS mismatch fails `H1a`'s own `npm test`. Keep the existing "resolves and loads" Ships-when wording, which then holds.

**Plan edits.** In `docs/plan_topics/H1a-scaffold-and-toolchain.md` **Tests**, add a `Convention:` (*phase categories*) bullet: a test imports the installed `eslint-plugin-loom-local` and asserts it loads without throwing and exposes a `rules` export. Leave the Ships-when sentence unchanged.

**Spec edits.** None.

**Pros.** Preserves the loadability guarantee the witness clause was added to provide; catches export-shape / ESM-CJS faults at `H1a` rather than at `H2a`; makes the Ships-when accurate.

**Cons.** Adds a test that executes the plugin module rather than only reading the manifest.

**Risks.** The placeholder package exports an empty rule set, so the assertion must accept a present-but-empty `rules` object (assert presence/type, not non-emptiness), or it couples `H1a` to a shape `H2a` later populates.

### Option B — Soften the Ships-when wording to claim only resolution

**Approach.** Change the Ships-when so the green `npm install` witnesses only `file:` resolution at `H1a`; first loadability is exercised at `H2a`.

**Plan edits.** In `docs/plan_topics/H1a-scaffold-and-toolchain.md` **Ships when**, strike "and loads" so the clause reads that the green `npm install` witnesses the `eslint-plugin-loom-local` package "resolves"; adjust the trailing parenthetical accordingly.

**Spec edits.** None.

**Pros.** Minimal change; the witness claim becomes literally true.

**Cons.** Drops any `H1a`-level loadability guarantee; a plugin that resolves but fails to load ships `H1a` green and reddens only at `H2a`.

**Risks.** None beyond the deferred detection point.

### Recommendation

Take Option A. The "resolves and loads" clause was added to give `H1a` a loadability guarantee for the in-tree plugin it now creates; a real import-and-`rules`-export assertion makes that guarantee true and surfaces a broken entry module at `H1a` instead of `H2a`. Watch the edge case that the `H1a` placeholder exports an empty rule set — assert that `rules` is present and an object, not that it is non-empty.

## Consequence

**Severity:** advisory

A broken `eslint-plugin-loom-local` entry module (wrong export shape, ESM/CJS mismatch) ships `H1a` green under a Ships-when that asserts the plugin "loads", with the failure surfacing only later at `H2a` when ESLint first loads it. The leaf still produces a working scaffold, so no implementer is blocked; the defect is an inaccurate witness claim that defers detection of a plugin-load fault one leaf downstream.
