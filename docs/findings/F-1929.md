## Finding

Several leaf test bullets carry a parenthetical governance-tracking tag — `(GOV-22 un-anchored residue)` or `(un-anchored; GOV-22 residue)` — embedded inside the normative requirement text. The tag records that the spec section the bullet cites is an un-anchored GOV-22 obligation (one carrying neither a numbered `PREFIX-N` REQ-ID nor a `loom/...` registry code). That classification is plan-authoring bookkeeping whose authoritative home is `coverage-matrix.md`'s *Code-keyed obligation areas (no numbered REQ-IDs)* table, where each such obligation is already enumerated with its closing leaf (e.g. `cka-17 → V9f`, `cka-27 → V9l`, `cka-46 → V14a`, each row itself tagged `(un-anchored; GOV-22 residue)`).

Carrying the same tag into the leaf's `Tests.` bullet adds no assertable behaviour: the bullet's normative content (the field-derivation MUSTs, the degraded-branch obligations, the envelope-lowering mechanics) is self-contained and stands without the tag. The implementer reading the bullet must parse around process metadata to reach the requirement. The annotation also duplicates a fact already owned by the coverage matrix, creating a second site that can silently diverge if an obligation is later re-anchored.

The affected bullets are: `V9f` (the `ToolDefinition.label` derivation bullet, tagged `(GOV-22 un-anchored residue)`); `V9l` and its tests partner `V9l-T` (all four §degraded-branch / §Recovery-path / §Predicate-split / §Seam-minimality bullets, tagged `(un-anchored; GOV-22 residue)`); and `V14a` and its tests partner `V14a-T` (the `host-interfaces-core.md — Tool execution from loom code` bullet, tagged `(un-anchored; GOV-22 residue)`). `V9f-T`'s corresponding bullet already omits the tag, so the impl/tests pair is internally inconsistent for `V9f`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `cfd76f7` (2026-06-12, "resolve 'Per-loom registration ToolDefinition field-derivation MUSTs unnamed by any leaf'" — added the `V9f` `ToolDefinition.label` bullet with the `(GOV-22 un-anchored residue)` tag); `55ed974` (2026-06-12, "resolve 'degraded-state obligations green over open spec contradiction'" — created `V9l`/`V9l-T` with the four `(un-anchored; GOV-22 residue)` bullets present at file creation); `77f2e88` (2026-06-18, "resolve 'code-side tool-call envelope-lowering MUSTs unanchored'" — added the `V14a`/`V14a-T` `(un-anchored; GOV-22 residue)` bullet).
**History:** The annotation was not present at plan inception. It entered incrementally: each commit resolved a prior plan-review finding that a particular un-anchored MUST was unnamed by any leaf, and in each remediation the author both assigned the obligation a closing-leaf Tests bullet and copied the coverage-matrix `GOV-22 residue` classification into that bullet. The two slightly different tag spellings (`(GOV-22 un-anchored residue)` in `V9f` vs `(un-anchored; GOV-22 residue)` elsewhere) reflect the separate remediation events rather than a single authoring pass.

## Solution Space

**Shape:** single

### Recommendation

Strike the GOV-22 residue parenthetical from each affected Tests bullet, leaving the normative content intact:

- `V9f-tool-registration-lifetime.md`: remove `(GOV-22 un-anchored residue)` from the `extension-bootstrap-and-per-loom.md` §Per-loom registration `ToolDefinition.label` bullet, so it reads `` - `extension-bootstrap-and-per-loom.md` §Per-loom registration `ToolDefinition.label` derivation: materialising … `` (matching the already-untagged `V9f-T` bullet).
- `V9l-session-only-degraded-branch.md` and `V9l-T-session-only-degraded-branch.md`: remove `(un-anchored; GOV-22 residue)` from each of the four §degraded-branch / §Recovery-path / §Predicate-split / §Seam-minimality bullets, leaving the `session-only-degraded-state.md §<section>: <obligation>` text.
- `V14a-tool-calls.md` and `V14a-T-tool-calls.md`: remove `(un-anchored; GOV-22 residue)` from the `host-interfaces-core.md — Tool execution from loom code` bullet.

Leave the identical tags in `coverage-matrix.md` untouched — that table is the authoritative GOV-22 residue ledger and the `H5e` un-anchored-MUST arm reads it. Do not touch the spec. The closing-leaf mapping for each obligation already lives in `coverage-matrix.md` (`cka-17`, `cka-27`, `cka-46`), so no traceability is lost by removing the in-leaf annotation.

## Consequence

**Severity:** cosmetic

If the plan ships unfixed, implementers read process-tracking metadata interleaved with normative test requirements and the `V9f`/`V9f-T` pair stays inconsistently tagged. The obligations remain implementable and the closing gates are unaffected; the cost is reader noise plus a duplicated classification that can drift from the coverage-matrix entry that owns it.
