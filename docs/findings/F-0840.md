## Finding

The four sub-files of the Pi version bump procedure under
`docs/spec_topics/pi-integration-contract/` are named on a uniform
step-number convention — `version-bump-intro.md`,
`version-bump-step2.md`, `version-bump-step2b.md` — except for
`version-bump-triggers.md`, whose filename is a topic label naming
only step 5's "trigger" concept even though the file holds the full
prose of steps 5, 6, and 7.

The file's own H1 already names the full span ("Version bump steps
5–7"), and the parent index entry in `pi-integration-contract.md`
reads "[Version bump steps 5–7](./pi-integration-contract/version-bump-triggers.md)";
the URL is the only artefact still pointing at the old, narrower
scope. A contributor scanning the directory listing or the index URL
will not connect "triggers" to steps 6 and 7 (provider seed-field
re-validation; binder strict-capability probe), neither of which is
about a trigger.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** f5e89f4 — docs: enforce 100KB hard / 30KB recommended size cap on spec set (2026-06-04, Thomas Andersen); a7a172c — pi-loom spec: resolve "version-bump-triggers.md H1 mis-scopes page contents" (2026-06-05, Thomas Andersen)
**History:** Commit `f5e89f4` split the over-size `pi-integration-contract.md` into per-topic sub-files and at that point the file was created as `version-bump-triggers.md` with the matching H1 "Version bump triggers" — internally consistent though already scope-narrow against its step-5/6/7 contents. The next day commit `a7a172c` resolved a separate review finding by renaming the H1 (and the index entry) to "Version bump steps 5–7" but left the filename unchanged, creating the present filename-vs-title divergence; the H1-rename commit is the proximate cause, but it only exposed a latent scoping problem the split commit had already baked into the filename.

## Solution Space

**Shape:** single

### Recommendation

Rename the file to `version-bump-steps-5-7.md` so the URL matches the
H1 and the sibling step-number convention (`version-bump-intro.md`,
`version-bump-step2.md`, `version-bump-step2b.md`). In the same
commit:

- Move the file (preserve git history with `git mv`).
- Update the index entry in `pi-integration-contract.md` to point at the new path.
- Sweep every inbound reference (12 files outside the renamed file itself, per the grep above) — three in `version-bump-intro.md`, one each in `audit-recognised-shapes.md`, `patch-skew-degradation.md`, `H6a-live-corpus-activation.md`, `real-host-smoke-gate.md`, `coverage-matrix.md`, three in `V18c-T-version-bump-checklist.md`, four in `V18c-version-bump-checklist.md`, three in `V18d-T-version-bump-acceptance.md`, four in `V18d-version-bump-acceptance.md`, and two in `findings/F-0469.md`.
- Update `docs/_tools/split_spec.py` if it carries the old filename in its split map (one match reported).
- Run `docs/_tools/check_links.py` (the in-repo link validator the size-cap split commit added) to confirm no inbound link reddens.

Edge cases for the implementer:

- The H1 anchor `version-bump-triggers` itself does not appear to be
  used as a `#anchor` target by inbound links (every cited reference
  is the bare file URL); confirm with a `grep -rn
  'version-bump-triggers#' docs/` before merge. If any
  `path#anchor` style reference exists, it is also covered by the
  sweep above.
- The intra-page `<a id="bump-step-5-trigger-i">` and
  `<a id="bump-step-5-trigger-ii">` anchors are unaffected — they
  key on `bump-step-5-*`, not on the filename.
- This is a pure rename: no prose edit. The H1 already reads
  "Version bump steps 5–7" and stays as-is.

## Consequence

**Severity:** cosmetic

A contributor navigating the version-bump procedure by directory
listing or by `pi-integration-contract.md`'s index URL sees a single
file whose URL name ("triggers") does not match its real span (steps
5–7), and may not connect step 6 (provider seed-field re-validation)
or step 7 (binder strict-capability probe) to that page at all. No
behaviour is at risk; the cost is purely discoverability and naming
hygiene against the sibling-file convention.
