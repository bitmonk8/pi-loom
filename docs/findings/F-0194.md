## Finding

The tests-task bullet in `conventions.md` §"Per-phase TDD ritual (mandatory)" states: *"A test that would pass when prerequisites are missing is a defect — fix it before the implementation task."* The word "prerequisites" is never defined at this site and admits two readings with different actionable consequences:

- **Test-level setup** — the test's own fixtures, harness setup, or the seams the tests-task is permitted to stub/declare are absent or unconfigured. Under this reading the defect is a vacuous-pass: the test greens without exercising anything.
- **Dependency leaves** — the upstream `Deps` leaves (the production code under test, owned by the implementation task or by other leaves) have not landed. Under this reading the sentence is about build ordering rather than test design.

The same section already defines the precise vocabulary that resolves this: the *"Fail red for the intended reason" (defined)* block distinguishes a test that "reds out on its own primary assertion because the implementation under test is absent or incomplete" (the legitimate red) from one red "on a missing or unconfigured fixture, a harness/setup throw, or an unrelated exception." That definition makes clear the intended scope of "prerequisites" is test-level setup (fixtures, declared seams) — but the tests-task bullet uses the bare, undefined word instead of the defined term, leaving the two surfaces unreconciled. The identical phrase "when prerequisites are missing" also appears in the cross-cutting *No silent test skipping* bullet, so the undefined term spans two rules.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `288f191` (2026-05-04) "Add implementation plan with horizontal/MVP/vertical-slice phases"
**History:** The plan corpus is git-tracked. `git log -S 'pass when prerequisites are missing' --all` traces the phrasing to the original plan-authoring commit `288f191`, where the then-"Tests first" step read *"A test that would pass when prerequisites are missing is a defect — fix it before writing code,"* and the *No silent test skipping* rule simultaneously used *"`assert.fail` / `panic` when prerequisites are missing."* The undefined word entered with the plan itself. The later commit `e7815d0` (2026-06-10) "resolve 'fail red for the intended reason undefined'" added the *"Fail red for the intended reason" (defined)* block adjacent to the tests-task bullet but did not rewrite the bullet's "prerequisites" wording to reference the new defined term, so the ambiguity persisted unreconciled rather than being newly introduced by that edit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md` §"Per-phase TDD ritual (mandatory)", rewrite the tests-task bullet's defect sentence so "prerequisites" is replaced by the test-level concepts the adjacent *"Fail red for the intended reason" (defined)* block already names. Replace:

> A test that would pass when prerequisites are missing is a defect — fix it before the implementation task.

with text to the effect of:

> A test that would pass when its own fixtures, harness setup, or declared seams are absent or unconfigured is a defect (it would not *fail red for the intended reason* as defined above) — fix it before the implementation task.

This ties the sentence to the defined vocabulary and excludes the "dependency leaves / build ordering" reading.

Edge case for the fixer: the same undefined phrase "when prerequisites are missing" appears in the *No silent test skipping* bullet under §"Cross-cutting rules (every leaf)". For consistency, apply the same fixtures/setup framing there (it carries the identical 288f191-origin wording); leaving it unchanged would re-introduce the ambiguity on a sibling surface.

## Consequence

**Severity:** advisory

Two implementers can read "prerequisites" differently — one hardening the test against vacuous fixture/setup passes, the other treating the sentence as a build-ordering note — so the guardrail does not reliably steer test design. Both readings still push toward "make the test red," so a working leaf can still be produced; the cost is inconsistent application of the anti-vacuous-pass discipline.
