## Problem

The *Specific exception types only* rule in `conventions.md` pins the `// allow-broad-catch:` closing-gate predicate so that every cited token must resolve to one of four arms, including a concrete `loom/...` diagnostics-registry code present in the diagnostics registry. The worked example for site (3) (the *Unknown-reason rule* `event.reason` read) then cites `loom/host/session-shutdown-*` — a glob, not a registry code: no such entry exists, and the gate's "code present in the registry" resolver cannot match a wildcard family. The rule never states whether the `loom/...`-arm token carries a literal concrete code or a wildcard family, leaving the example unsatisfiable and the resolver's contract ambiguous.

## Issue introduction

**Verdict:** present-since-inception (of the broad-catch token-resolution worked example)
**Introducing commit:** `96563f9` — "pi-loom plan: resolve \"broad-catch allow-comment predicate unsatisfiable for exempt sites\"" (2026-06-10)
**History:** `git log -S "session-shutdown-*" -- docs/plan_topics/conventions.md` returns exactly one commit, `96563f9`, which authored the entire "Each site cites a token the predicate above admits…" clause — including the `loom/...` registry-code arm added so "site (3) needs" a satisfiable token, and the `loom/host/session-shutdown-*` glob in that same sentence. The glob has been present unchanged since; no later commit touched the token string. The defect is intrinsic to the worked example as first written: the commit added the registry-code arm to make site (3) satisfiable but illustrated it with a glob that the gate's "code present in the registry" resolver cannot satisfy.

## Solution approach

Clarify the *Specific exception types only* rule so the `loom/...`-arm token resolves to one or more concrete diagnostics-registry codes rather than a glob or wildcard family. Correct the site (3) worked-example clause to cite the concrete code the `event.reason` read emits (`loom/host/session-shutdown-reason-unknown`), accounting for the spec-permitted single widened handler-entry `try`/`catch` that may also emit `loom/host/session-shutdown-pinned-constant-unreadable`. Reconcile the H5c and H2a token-resolution gate fixtures with whatever grammar the corrected rule lands on.

## Solution constraints

- The diagnostics registry (`spec_topics/diagnostics.md`) is read-only for this fix — cite the existing concrete codes; do not add a glob entry to the registry to make the example resolve.
- Preserve the gate resolver's wildcard-rejection guarantee: each cited `loom/...` token must resolve to a concrete code present in the registry, never a glob family.
