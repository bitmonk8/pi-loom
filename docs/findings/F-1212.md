## Finding

`schemas.md` §Object schema says, in one sentence, two things that pull in different directions:

> Every declared field is **required** (the lowered JSON Schema's `required` lists every property; `additionalProperties: false` is always emitted). Optional fields are expressed as `T | null` — there is no `field?: T` shorthand. The non-existence and the explicit-`null` cases are conflated, matching strict-mode provider behaviour.

The "conflated" clause admits two incompatible readings:

1. **Validation-level reading.** For a field typed `T | null`, the runtime accepts a wire payload in which that property is absent and treats it as equivalent to `null`. This reading contradicts the invariant in the same sentence (the lowered schema's `required` lists *every* property, `additionalProperties: false` is always emitted) and contradicts [`PIC-11` — `SchemaValidator` interface](./pi-integration-contract/host-interfaces-services.md#pic-11) ("Validation MUST NOT convert values and MUST NOT fill defaults"). Under this reading the runtime would have to either drop nullable fields from `required` at lowering, or coerce absent → `null` post-decode — neither of which is specified anywhere else.

2. **Language-level reading.** `required` still lists every field, an absent property still fails AJV, and "conflated" only describes the *loom surface syntax*: there is no `field?: T` shorthand, so a nullable field is always written and read as `T | null`, with no language-level distinction between "field present with value `null`" and "field absent" — the latter is simply not expressible in loom-side code.

Every other normative anchor in the corpus (`schema-subset.md` Lowering Algorithm step 3 *Object* — `"required": [...every wire name...]`; `V5d` reject-gate's "objects all-required + `additionalProperties:false`"; `PIC-11` no-coercion / no-defaults) is consistent only with reading 2, so reading 1 is wrong — but the prose does not say so, and an implementer reading this section alone has no way to discriminate.

## Solution Space

**Shape:** single

### Recommendation

Replace the trailing sentence of the cited paragraph with a version that names the *language-level* scope of "conflated" explicitly and reaffirms the validation-level contract. A concrete rewrite:

> Optional fields are expressed as `T | null` — there is no `field?: T` shorthand, so loom-side code never distinguishes "field absent" from "field present with value `null`" (the two cases are conflated at the language surface, matching strict-mode provider behaviour). At the wire / validation level the field stays in `required` and `additionalProperties: false` still applies: an absent property fails AJV exactly as it does for a non-nullable field, and the runtime performs no absent→`null` coercion (per [`PIC-11`](./pi-integration-contract/host-interfaces-services.md#pic-11)'s no-conversion / no-defaults contract).

The fixer should keep the existing first two sentences of the paragraph (the `required`/`additionalProperties: false` invariant and the `T | null` shorthand prohibition) unchanged — those already state the wire-side rule correctly; only the third sentence ("The non-existence and the explicit-`null` cases are conflated, matching strict-mode provider behaviour.") is being replaced. The cross-reference to `PIC-11` is the load-bearing piece: it anchors the no-coercion claim to the only normative owner so a future reader has a single follow-link rather than another local restatement.

Edge cases for the implementer:

- The cross-ref target is `host-interfaces-services.md#pic-11`; do not link to the `schemavalidator-interface` anchor immediately above it, which is an org anchor, not the REQ-ID anchor.
- The "matching strict-mode provider behaviour" clause should be kept (or paraphrased) — it is the *rationale* for the language-level choice and was the original sentence's only forward-looking content; do not delete it when removing the ambiguous "conflated" claim.

## Consequence

**Severity:** advisory

A solo reader of `schemas.md` could come away believing that the runtime accepts an absent property for any `T | null` field and so build a validator that drops nullable fields from `required` (or post-coerces absent → `null`) — producing wire payloads that other compliant runtimes reject and a `SchemaValidator` that violates `PIC-11`'s no-coercion contract. The other spec sections pin the correct behaviour firmly enough that a careful implementer cross-reading them will not actually diverge, which keeps this short of `correctness`; but the unforced two-reading ambiguity sits on a load-bearing sentence and should not be left to cross-page reconciliation.
