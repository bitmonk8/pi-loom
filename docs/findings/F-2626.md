## Finding

The lowering pass synthesises four content-addressed names — `__inline_<slug>`, `__loom_respond_<slug>`, `__loom_callee_<slug>__<post-rename-name>`, and `__loom_bind_<slug>` ([Schema Subset — Synthesised names](docs/spec_topics/schema-subset.md#synthesised-names)). Of these, `__loom_callee_<slug>__<post-rename-name>` and `__loom_respond_<slug>` are not internal artefacts: the prompt-mode registration cache passes them to `pi.registerTool` and they cross the wire to the provider as tool/function names ([Tool-registration lifetime and visibility](docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md#tool-registration-lifetime-and-visibility)). The supported providers' strict modes (the OpenAI Structured Outputs and Anthropic tool-use surfaces the subset was chosen to satisfy) cap function/tool names — Anthropic's `input_schema` tool name is `^[a-zA-Z0-9_-]{1,64}$`, OpenAI's function name is likewise capped at 64 characters — yet nothing in the spec bounds the synthesised composite against any such limit.

The composite `__loom_callee_<slug>__<post-rename-name>` has a fixed 32-character head (`__loom_callee_` = 14, the 16-hex slug, and the `__` separator), so the `<post-rename-name>` component has only 32 characters of headroom before a 64-character cap is breached. `<post-rename-name>` is author-controlled — it is a `.loom` basename (hyphens remapped to underscores) or an explicit `as` rename — and is constrained only to be a lowercase-first loom identifier ([Parameters and Frontmatter — `tools`](docs/spec_topics/frontmatter/frontmatter-fields-a.md#tools)); its length is unbounded. A loom whose callee filename or `as` rename exceeds that headroom produces a name the provider rejects at request time. The runtime-collision disambiguation path makes this worse: it inserts a `_<n>` segment (`__loom_callee_<slug>_<n>__<post-rename-name>`, [PIC-44](docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md#pic-44)), lengthening an already-borderline name at runtime.

The charset dimension is, in fact, already satisfied — the head is fixed ASCII, the slug is lowercase hex, and `<post-rename-name>` is identifier-shaped — but the spec never states this, so an implementer cannot tell whether charset was reasoned about or merely overlooked. The length dimension is a genuine gap: no maximum length is pinned, and no behaviour (reject, truncate, or pass-through-and-let-the-provider-fail) is specified when a synthesised name overflows.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `27ec53d` (2026-05-04, "Resolve Cluster-A: per-mode tool registration")
**History:** The corpus is git-tracked. `27ec53d` introduced the per-mode tool-registration mechanism and the synthesised `__loom_respond_<slug>` / loom-callee names, wiring them into `pi.registerTool` and `createAgentSession` without any registration-target name-length or charset bound. The subsequent refinements that consolidated and named these forms — `e21449b` (2026-05-04, pinned the `__inline_<hash>` hash function), `9d99fe6` (2026-05-07, collapsed the six surface names for the slug), and `c577ab3` (2026-06-06, made the synthesised-name enumeration the source of truth) — each touched the synthesised-name surface but none added a length/charset bound or an overflow rule. A `git log -S` over the synthesised-name files for `name length`, `too long`, `overflow`, `truncate`, `64 char`, `character limit`, and `charset` returns zero commits. The omission has therefore been present since the construct's introducing commit and was never subsequently addressed.

## Solution Space

### Recommendation

Pin a loom-owned maximum synthesised-name length (64, framed the way the `Depth ≤ 5` ceiling is — a conservative value chosen at authoring time to sit at or below the supported providers' published name caps, loom-owned and not tracking later provider drift). When a synthesised name would exceed the ceiling, fail at load with a registry diagnostic `loom/load/synthesised-name-too-long` naming the offending callee.

Compute the length check against the worst-case `_<n>`-disambiguated form (`__loom_callee_<slug>_<n>__<post-rename-name>`), not the base form, so that a clean load cannot later overflow on a runtime collision.

Spec edits:

- In `schema-subset.md`, at the Synthesised names enumeration: add the length ceiling, and add an explicit note that the charset dimension is satisfied by construction (fixed ASCII head + hex slug + identifier-shaped `<post-rename-name>`) so the rule is length-only.
- In `tool-registration-lifetime.md`, at the tool-registration site: add the reject rule and register the `loom/load/synthesised-name-too-long` diagnostic, and register it in the load diagnostic registry.

The slug-as-identity contract and the `_<n>` disambiguation are untouched.

## Consequence

**Severity:** correctness

A prompt-mode loom whose callee basename or `as` rename pushes the synthesised tool name past the registration-target cap produces a name the provider rejects, surfacing as an opaque provider-side request failure at query time with no loom-side diagnostic. Two reasonable implementers diverge on the unstated overflow behaviour — reject at load, deterministically truncate, or pass through and let the provider 400 — yielding observably different behaviour for the same loom.
