## Problem

Two of the four synthesised names enumerated at `schema-subset.md#synthesised-names` cross the wire to the provider as tool/function names: `__loom_callee_<slug>__<post-rename-name>` and `__loom_respond_<slug>` (passed to `pi.registerTool` per `tool-registration-lifetime.md#tool-registration-lifetime-and-visibility`). The supported providers' strict modes cap function/tool names at 64 characters, yet nothing in the spec bounds the synthesised composite. The `__loom_callee_` form leaves only ~32 characters of headroom for the author-controlled `<post-rename-name>` component, which is itself length-unbounded, and the `#pic-44` runtime-collision path inserts a further `_<n>` segment — so no maximum length is pinned and no overflow behaviour (reject, truncate, or pass-through-and-let-the-provider-fail) is specified. The charset dimension is in fact already satisfied by construction, but the spec never states this.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `27ec53d` (2026-05-04, "Resolve Cluster-A: per-mode tool registration")
**History:** The corpus is git-tracked. `27ec53d` introduced the per-mode tool-registration mechanism and the synthesised `__loom_respond_<slug>` / loom-callee names, wiring them into `pi.registerTool` and `createAgentSession` without any registration-target name-length or charset bound. The subsequent refinements that consolidated and named these forms — `e21449b` (2026-05-04, pinned the `__inline_<hash>` hash function), `9d99fe6` (2026-05-07, collapsed the six surface names for the slug), and `c577ab3` (2026-06-06, made the synthesised-name enumeration the source of truth) — each touched the synthesised-name surface but none added a length/charset bound or an overflow rule. A `git log -S` over the synthesised-name files for `name length`, `too long`, `overflow`, `truncate`, `64 char`, `character limit`, and `charset` returns zero commits. The omission has therefore been present since the construct's introducing commit and was never subsequently addressed.

## Solution approach

At `schema-subset.md#synthesised-names`, pin a loom-owned maximum synthesised-name length (framed as the `Depth ≤ 5` ceiling is — a conservative authoring-time value at or below the supported providers' published caps, not tracking later provider drift), and state that the charset dimension is satisfied by construction so the rule is length-only. At `tool-registration-lifetime.md#tool-registration-lifetime-and-visibility`, specify the load-time behaviour when a synthesised name overflows the ceiling, checking against the worst-case `_<n>`-disambiguated form from `#pic-44` so a clean load cannot later overflow on a runtime collision.

## Solution constraints

- Out of scope: the schema-slug recipe and the `#pic-44` `_<n>` disambiguation rule — bound the synthesised name, do not alter how it is constructed.
