## Finding

`V13b`'s third *Tests* bullet closes the SUBS lowering-algorithm step 4 (per-query `$defs` pruning) using the obligation-closing code-keyed-area citation format — a `schema-subset.md` link annotated `(SUBS code-keyed area)`, the same shape `V5f`'s third bullet uses for SUBS step 5 (per-schema sidecar). But the SUBS code-keyed obligation area is `cka-10` in `coverage-matrix.md`, and that row lists only `V5d`, `V5e`, `V5f`. `V13b` is not a listed SUBS facet; it sits in the `cka-12` (QRY) row. A SUBS obligation-closing citation therefore lives on a leaf the SUBS coverage row does not name.

This is ambiguous to a reviewer or auditor reconciling SUBS coverage. Either (a) the `cka-10` row is incomplete and `V13b` is a fourth SUBS facet, or (b) `V13b`'s bullet is integration evidence for the lowering-pass pruning `V5f` already owns — `V5f`'s *Adds* explicitly names "per-query `$defs` pruning" as part of the lowering pass — in which case the bullet should not carry the obligation-closing SUBS citation format. The two readings disagree on which leaf authoritatively closes the per-query-pruning SUBS obligation.

The downstream risk is that the per-query `$defs` pruning step has no single, unambiguous closing leaf: `V5f`'s *Adds* claims the behaviour but `V5f`'s *Tests* cover only step 5, while the only obligation-closing assertion for step 4 lives on `V13b` outside the SUBS facet set. Two implementers can diverge on where (and whether) the SUBS pruning obligation is asserted as such.

## Issue introduction

**History:** The `cka-10` (SUBS) coverage row has listed `V5d`, `V5e`, `V5f` and never `V13b` since the matrix's origin, and `V13b` has always sat in the `cka-12` (QRY) row. Commit ba7da00 — resolving an unrelated finding about `V5d`'s unbound lowering-pass outputs — added the SUBS-step-4 per-query `$defs` pruning obligation-closing *Tests* bullet (with the `(SUBS code-keyed area)` annotation) to `V13b` and added `V5d` to `V13b`'s Deps, but did not add `V13b` to the `cka-10` row, introducing the mismatch between `V13b`'s SUBS obligation-closing citation and its absence from the SUBS facet set.

## Solution Space

### Recommendation

Move the SUBS lowering-algorithm step-4 (per-query `$defs` pruning) obligation-closing citation onto `V5f`, which owns the lowering pass and already names "per-query `$defs` pruning" in its *Adds*, and convert `V13b`'s bullet 3 into typed-query integration evidence for that seam.

Resolve in this order so the obligation has a confirmed home before the integration bullet is reworded:

1. In `V5f` *Tests*, add a bullet citing `schema-subset.md` Lowering Algorithm step 4 (per-query `$defs` pruning) annotated `(SUBS code-keyed area)`. This is the single obligation-closing citation for step 4.
2. In `V13b` *Tests* bullet 3, reword the bullet to reference `V5f`'s per-query `$defs` pruning seam as typed-query integration evidence. Pin the seam by `V5f`'s leaf-ID (a Class-2 consumer-bound seam) and drop the `(SUBS code-keyed area)` obligation-closing annotation, so the bullet is not read as a second obligation-closing citation.

Leave the `coverage-matrix.md` `cka-10` row unchanged: its facet set stays the three schema-subset leaves `V5d`, `V5e`, `V5f`. `H5b`'s Deps already lists `V13b` (within the `V13a`–`V13d` range), so no `H5b` Deps change is required.

Ensure only `V5f` reads as the closer of the SUBS step-4 obligation — `V5f`'s new bullet and `V13b`'s reworded bullet must not both claim to *close* it — and that `V13b`'s reworded bullet binds `V5f`'s lowering-pruning seam by leaf-ID.

## Consequence

**Severity:** correctness

The per-query `$defs` pruning SUBS obligation has no unambiguous closing leaf: its only obligation-closing citation lives on `V13b`, which the `cka-10` SUBS facet set does not name, while `V5f` (whose *Adds* owns the lowering-pass pruning) tests only step 5. Two reasonable implementers diverge — one treats `V13b` as the SUBS closer and one treats `V5f` as the closer — risking the pruning behaviour being asserted twice with divergent assertions or, at `V5f`, not at all.
