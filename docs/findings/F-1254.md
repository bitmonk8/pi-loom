## Finding

`V13b`'s third *Tests* bullet closes the SUBS lowering-algorithm step 4 (per-query `$defs` pruning) using the obligation-closing code-keyed-area citation format — a `schema-subset.md` link annotated `(SUBS code-keyed area)`, the same shape `V5f`'s third bullet uses for SUBS step 5 (per-schema sidecar). But the SUBS code-keyed obligation area is `cka-10` in `coverage-matrix.md`, and that row lists only `V5d`, `V5e`, `V5f`. `V13b` is not a listed SUBS facet; it sits in the `cka-12` (QRY) row. A SUBS obligation-closing citation therefore lives on a leaf the SUBS coverage row does not name.

This is ambiguous to a reviewer or auditor reconciling SUBS coverage. Either (a) the `cka-10` row is incomplete and `V13b` is a fourth SUBS facet, or (b) `V13b`'s bullet is integration evidence for the lowering-pass pruning `V5f` already owns — `V5f`'s *Adds* explicitly names "per-query `$defs` pruning" as part of the lowering pass — in which case the bullet should not carry the obligation-closing SUBS citation format. The two readings disagree on which leaf authoritatively closes the per-query-pruning SUBS obligation.

The downstream risk is that the per-query `$defs` pruning step has no single, unambiguous closing leaf: `V5f`'s *Adds* claims the behaviour but `V5f`'s *Tests* cover only step 5, while the only obligation-closing assertion for step 4 lives on `V13b` outside the SUBS facet set. Two implementers can diverge on where (and whether) the SUBS pruning obligation is asserted as such.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** ba7da00 — pi-loom plan: resolve "V5d lowering-pass outputs bound by no Tests bullet" (2026-06-10, Thomas Andersen)
**History:** The `cka-10` (SUBS) coverage row has listed `V5d`, `V5e`, `V5f` and never `V13b` since the matrix's origin, and `V13b` has always sat in the `cka-12` (QRY) row. Commit ba7da00 — resolving an unrelated finding about `V5d`'s unbound lowering-pass outputs — added the SUBS-step-4 per-query `$defs` pruning obligation-closing *Tests* bullet (with the `(SUBS code-keyed area)` annotation) to `V13b` and added `V5d` to `V13b`'s Deps, but did not add `V13b` to the `cka-10` row, introducing the mismatch between `V13b`'s SUBS obligation-closing citation and its absence from the SUBS facet set.

## Solution Space

**Shape:** multiple

### Option A — Add `V13b` as a fourth `cka-10` facet

- **Approach:** Treat `V13b` as a genuine SUBS facet. Add `V13b` to the `cka-10` closing-leaf cell, keeping `V13b`'s bullet 3 as the obligation-closing SUBS citation.
- **Plan edits:** In `coverage-matrix.md`, the `cka-10` row's *Closing leaf(s)* cell becomes `` `V5d`, `V5e`, `V5f`, `V13b` `` (new token backtick-delimited per the *Closing-leaf column authoring* rule). `V13b` and `V5f` *Tests* unchanged. Because `cka-10` is now a four-leaf multi-leaf row, the `H5f` per-facet citing-test arm requires `V13b`'s SUBS test to cite both `cka-10` and `V13b` inline (a facet-naming citing test per the *Multi-leaf-row per-facet citing tests* convention).
- **Spec edits:** None.
- **Pros:** Keeps `V13b`'s existing bullet; records that pruning is exercised in the typed-query context.
- **Cons:** SUBS (schema-subset) obligations now span a QRY-slice leaf; widens the `cka-10` facet partition to four and creates a new `H5f` facet-naming citing-test obligation on `V13b`.
- **Risks:** The `H5f` per-facet arm reddens unless `V13b`'s test is updated to cite `cka-10` and `V13b` inline.

### Option B — Own per-query pruning on `V5f`; make `V13b` integration-only

- **Approach:** Place the SUBS step-4 obligation-closing citation on `V5f` (which already names per-query `$defs` pruning in its *Adds* and owns the lowering pass), and rewrite `V13b`'s bullet 3 as a cross-leaf integration-seam reference to `V5f`'s pruning that does not use the obligation-closing citation format.
- **Plan edits:** In `V5f` *Tests*, add a bullet citing `schema-subset.md` Lowering Algorithm step 4 (per-query `$defs` pruning) with the `(SUBS code-keyed area)` annotation. In `V13b` *Tests* bullet 3, reword to reference `V5f`'s per-query `$defs` pruning seam as typed-query integration evidence, pinning the seam by `V5f`'s leaf-ID (Class-2 consumer-bound seam) and dropping the `(SUBS code-keyed area)` obligation-closing annotation. `coverage-matrix.md` `cka-10` row unchanged.
- **Spec edits:** None.
- **Pros:** SUBS obligations stay owned within the schema-subset (`V5*`) leaves, matching `V5f`'s *Adds*; the `cka-10` facet partition stays the three schema-subset leaves; no new `H5f` obligation on `V13b`.
- **Cons:** `V13b` loses an obligation-closing citation (becomes integration-only); requires editing two leaves.
- **Risks:** Only `V5f` must read as the obligation closer for step 4; `V13b`'s reworded bullet must bind `V5f`'s seam by leaf-ID so it is not read as a second obligation-closing citation.

### Recommendation

Take **Option B**. `V5f`'s *Adds* already names "per-query `$defs` pruning" as part of the lowering pass it owns, and the SUBS code-keyed area's facets are the schema-subset (`V5*`) leaves; the step-4 obligation-closing citation belongs on `V5f`, with `V13b` exercising pruning as typed-query integration evidence. Resolve in order: first add the step-4 obligation-closing *Tests* bullet to `V5f`, then reword `V13b`'s bullet 3 to a `V5f`-seam integration reference — so the bullet's rewrite lands after the obligation has a confirmed home on `V5f`. `H5b`'s Deps already lists `V13b` (within the `V13a`–`V13d` range), so no `H5b` Deps change is required under either option. Edge case: ensure `V5f`'s new bullet and `V13b`'s reworded bullet do not both claim to *close* the SUBS step-4 obligation — only `V5f` closes it — and that `V13b`'s reworded bullet pins `V5f`'s lowering-pruning seam by leaf-ID.

## Consequence

**Severity:** correctness

The per-query `$defs` pruning SUBS obligation has no unambiguous closing leaf: its only obligation-closing citation lives on `V13b`, which the `cka-10` SUBS facet set does not name, while `V5f` (whose *Adds* owns the lowering-pass pruning) tests only step 5. Two reasonable implementers diverge — one treats `V13b` as the SUBS closer and one treats `V5f` as the closer — risking the pruning behaviour being asserted twice with divergent assertions or, at `V5f`, not at all.
