## Problem

In `docs/plan_topics/conventions.md`, the §Per-phase TDD ritual tests-task bullet instructs "Write the failing tests for *every* spec REQ-ID the feature introduces." The verb "introduces" admits two readings against the page's many-to-many coverage matrix: first-occurrence (test only REQ-IDs no prior leaf has tested) versus claims-to-close (test every coverage-matrix-mapped REQ-ID the leaf closes). The §Leaf format **Tests** bullet on the same page uses the unambiguous closure phrasing "one bullet per REQ-ID the leaf claims to implement," requiring the second reading. The page's most-followed actionable sentence thus conflicts with its own normative leaf format.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** fecb504 — Split spec.md and plan.md into per-topic / per-phase files (2026-05-04, Thomas Andersen)
**History:** The conventions page was split out of `plan.md` at fecb504 (2026-05-04). The TDD-ritual "introduces" phrasing (predecessor "the leaf introduces") and the Leaf-format "REQ-ID the leaf claims to implement" wording with the many-to-many coverage-matrix note have coexisted since the page's earliest form. Commit f9c5354 (2026-06-09, two-task TDD pairing rework) refined "leaf introduces"→"feature introduces" but did not create the divergence; both terminologies were already present, so the conflict cannot be localised to a single later commit.

## Solution approach

In the §Per-phase TDD ritual tests-task bullet, rewrite the obligation to name the leaf's coverage-matrix closing mapping unambiguously, using the §Leaf format **Tests** bullet's existing phrase "every spec REQ-ID the leaf claims to implement" so both sections name the obligation with one term and the claims-to-close reading is forced.
