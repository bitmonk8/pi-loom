## Problem

The two arity rows `loom/parse/invoke-arity-too-few` and `loom/parse/invoke-arity-too-many` in `code-registry-parse.md` cover both `invoke(...)` callsites and `.loom`-callable tool calls (the dual surface is normative per `invocation.md#argument-binding` and `tool-calls.md`), but their `invoke-` code-name prefix and the `invoke '<callee>' passes too few/many arguments…` *Message* templates attribute the diagnostic to `invoke` only. On a `.loom`-callable tool-call site (`mytool(args)` rather than `invoke("./mytool.loom", args)`), a reader sees `invoke 'mytool' passes too many arguments` with no syntactic `invoke(...)` form in source to ground the message against. The sibling type-mismatch codes already split this exact partition — `loom/parse/invoke-arg-type-mismatch` for `invoke(...)` and `loom/parse/tool-arg-type-mismatch` for `.loom`-callable calls — so holding split type-mismatch naming beside merged-and-`invoke`-prefixed arity naming leaves the diagnostics contract internally asymmetric for the same call-site partition.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `833d8d5` (`spec: resolve five invoke edge cases (Decision #4)`, 2026-05-05)
**History:** The two codes — including their `invoke-`-prefixed names and the `invoke '<callee>' passes…` *Message* templates — were added by `833d8d5` together with the *Argument arity* prose in `invocation.md`. That same commit's *Argument arity* paragraph already concluded with "The same arity rules apply to registered-loom calls through `tools:` entries", so the dual-surface scope and the invoke-only naming/template landed in the same commit; the asymmetry has been baked in since the codes' birth. A later commit `fec0870` (`pi-loom spec: resolve "callable-set prose synonyms erode no-inheritance invariant"`, 2026-05-06) rewrote "registered-loom" to "`.loom` callable" across the rule, the table *Trigger* prose, and the `loom/runtime/invoke-depth-exceeded` row, but did not touch the code names or the *Message* template prefix — the synonym sweep made the dual-surface scope more visible in the *Trigger* prose without resolving the naming inconsistency it surfaces. The sibling type-mismatch split (`invoke-arg-type-mismatch` / `tool-arg-type-mismatch`) that this finding contrasts against was already present in the diagnostics table before `833d8d5` landed (added by `333ba2f` / `839db71`), so the arity rows landed *into* a table whose precedent for this partition was to split. No subsequent commit revisited the naming.

## Solution approach

Clarify the *Trigger* column of both `loom/parse/invoke-arity-too-few` and `loom/parse/invoke-arity-too-many` rows in `code-registry-parse.md` to acknowledge the dual surface (literal-`invoke(...)` callsites and `.loom`-callable tool calls) and to record that the `invoke-` prefix is retained per DIAG-3 and the leading `invoke '<callee>'` Message wording per DIAG-4, both deferred to loom 2.0 migration. Mirror the in-place gloss `loom/load/cross-format-collision` already carries for the same DIAG-3 envelope. Name the loom 2.0 successor candidates — neutral rename or a split matching the `invoke-arg-type-mismatch` / `tool-arg-type-mismatch` partition — without picking one in the 1.x edit.

## Solution constraints

- Out of scope in loom 1.x: renaming either code — DIAG-3 pins code renames as breaking changes excluded from the diagnostic-registry carve-out.
- Out of scope in loom 1.x: rewording the *Message* template column for either row (even cosmetically) — DIAG-4 binds the template string character-for-character.
