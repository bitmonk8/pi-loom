## Finding

The two arity rows `loom/parse/invoke-arity-too-few` and `loom/parse/invoke-arity-too-many` in [`code-registry-parse.md`](../../docs/spec_topics/diagnostics/code-registry-parse.md#code-registry-parse) describe their own *Trigger* as covering both `invoke(...)` and `.loom`-callable tool-call sites, and the prose that owns the rule in [`invocation.md` — *Argument arity*](../../docs/spec_topics/invocation.md#argument-binding) confirms it ("The same arity rules apply to `.loom` callable calls through `tools:` entries"). The same dual-surface claim is also asserted in [`tool-calls.md` — *Argument shape*](../../docs/spec_topics/tool-calls.md) and relied on by [`placeholder-rendering-b.md` — Category 7 `<callee>` fallback](../../docs/spec_topics/diagnostics/placeholder-rendering-b.md). The `invoke-` prefix and the rendered *Message* template that begins `invoke '<callee>' passes too few/many arguments…` therefore mis-attribute the diagnostic when the offending call site is a `.loom`-callable tool call (`mytool(args)` rather than `invoke("./mytool.loom", args)`). A reader who sees `invoke 'mytool' passes too many arguments` in a system note has no syntactic `invoke(...)` form in the source to ground the message against.

The inconsistency is reinforced by the sibling type-mismatch codes, which the same shard already split along exactly this seam: `loom/parse/invoke-arg-type-mismatch` for `invoke(...)` callsites and `loom/parse/tool-arg-type-mismatch` for `.loom`-callable tool calls. Holding both naming patterns side by side in one table — split for type mismatches, merged-and-`invoke`-prefixed for arity — leaves the public diagnostics contract internally asymmetric for the same partition of call sites.

A rename or message reword to drop the `invoke` keyword is, however, foreclosed within loom 1.x: code renames are pinned as breaking changes by [DIAG-3](../../docs/spec_topics/diagnostics/diagnostic-shape.md#diag-3), *Message*-template rewords by [DIAG-4](../../docs/spec_topics/diagnostics/diagnostic-shape.md#diag-4), and both are explicitly excluded from the [diagnostic-registry carve-out](../../docs/spec_topics/governance/source-language-stability.md) that admits in-place registry edits within a loom 1.x minor release — rename/reword are deferred to loom 2.0 migration. The in-1.x repair surface is therefore documentation only: the *Trigger* column has to acknowledge the full scope (the same shape already used in-place by `loom/load/cross-format-collision`'s "retains the historical `cross-format` token… its scope is the full same-priority slash-name collision rule, not only the cross-format arm" gloss). The 2.0 rename or split is then named as the durable fix and parked there.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `833d8d5` (`spec: resolve five invoke edge cases (Decision #4)`, 2026-05-05)
**History:** The two codes — including their `invoke-`-prefixed names and the `invoke '<callee>' passes…` *Message* templates — were added by `833d8d5` together with the *Argument arity* prose in `invocation.md`. That same commit's *Argument arity* paragraph already concluded with "The same arity rules apply to registered-loom calls through `tools:` entries", so the dual-surface scope and the invoke-only naming/template landed in the same commit; the asymmetry has been baked in since the codes' birth. A later commit `fec0870` (`pi-loom spec: resolve "callable-set prose synonyms erode no-inheritance invariant"`, 2026-05-06) rewrote "registered-loom" to "`.loom` callable" across the rule, the table *Trigger* prose, and the `loom/runtime/invoke-depth-exceeded` row, but did not touch the code names or the *Message* template prefix — the synonym sweep made the dual-surface scope more visible in the *Trigger* prose without resolving the naming inconsistency it surfaces. The sibling type-mismatch split (`invoke-arg-type-mismatch` / `tool-arg-type-mismatch`) that this finding contrasts against was already present in the diagnostics table before `833d8d5` landed (added by `333ba2f` / `839db71`), so the arity rows landed *into* a table whose precedent for this partition was to split. No subsequent commit revisited the naming.

## Solution Space

**Shape:** single

### Recommendation

Extend the *Trigger* column of both `loom/parse/invoke-arity-too-few` and `loom/parse/invoke-arity-too-many` rows in `docs/spec_topics/diagnostics/code-registry-parse.md` with an explicit acknowledgement, mirroring the in-place pattern `loom/load/cross-format-collision` already uses for the same DIAG-3 envelope. The added prose must state, in this order:

1. that the code covers both literal-`invoke(...)` callsites and `.loom`-callable tool calls through `tools:` entries (the dual surface is normative — `invocation.md` and `tool-calls.md` already say so);
2. that the `invoke-` prefix is retained as a stable diagnostics-contract identifier per [DIAG-3](../../docs/spec_topics/diagnostics/diagnostic-shape.md#diag-3) and the *Message* template's leading `invoke '<callee>'` is retained per [DIAG-4](../../docs/spec_topics/diagnostics/diagnostic-shape.md#diag-4) — both are deferred to loom 2.0 migration per the [diagnostic-registry carve-out](../../docs/spec_topics/governance/source-language-stability.md) (rename and reword are explicitly outside the carve-out);
3. the suggested loom 2.0 successor shape: either neutral rename (`loom/parse/callee-arity-too-few`/`too-many`) or split along the same partition as `invoke-arg-type-mismatch` / `tool-arg-type-mismatch` (`loom/parse/invoke-arity-*` / `loom/parse/tool-arity-*`), with the corresponding *Message*-template reword that drops the `invoke` keyword on the tool-call arm. The 1.x edit picks *neither* successor — it only records that both are open candidates for the 2.0 migration to choose between.

Implementer-relevant edge cases:

- The `<callee>` placeholder's rendering on a tool-call site is already pinned by [`placeholder-rendering-b.md` — Category 7 `<callee>` fallback](../../docs/spec_topics/diagnostics/placeholder-rendering-b.md) (slash-name when bound, else path-literal fallback) — no edit there is required, and the test vectors that page cites against `loom/parse/invoke-arity-too-few` remain valid.
- The `<required>` and `<provided>` integer rendering is pinned by [`placeholder-rendering-a.md` — Scope of `<required>` / `<provided>`](../../docs/spec_topics/diagnostics/placeholder-rendering-a.md) — also unchanged.
- The *Spec rule* link target stays `invocation.md#argument-binding` for both rows; the *Argument arity* prose already covers the dual surface.
- Do **not** edit the *Message* template column in 1.x even cosmetically (e.g. surrounding `invoke` in quotes or swapping word order) — DIAG-4 binds the string character-for-character. The gloss lives in *Trigger* only.

## Consequence

**Severity:** advisory

Implementers and authors reading the rendered diagnostic on a `.loom`-callable tool-call site see the word `invoke` in a context where no `invoke(...)` form appears in source, costing triage time and inviting confusion about which surface the check covers. No behaviour is wrong — the codes fire on the correct trigger and tests assert the correct firings — but the public diagnostics contract reads as if the arity check is invoke-only, contradicting the normative *Trigger* prose and the sibling type-mismatch codes' split-naming convention.
