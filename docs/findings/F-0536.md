## Finding

`V9j` delivers two structurally independent units in one leaf. Its **Adds** field couples (1) the pi-ai `complete()` inference call — forced-tool envelope, `temperature:0`, the per-provider seed-field mapping, and the `onResponse` capture callback (specified in `binder-inference.md`) — with (2) the provider-error → `QueryError` classifier — HTTP-status / `errorMessage` classification, the `retryable` rule, context-overflow extraction, and stop-reason classification (specified in `provider-error-mapping.md`).

The leaf's two test scenarios exercise these as independent code paths: one asserts the deterministic `complete()` envelope, the other asserts the provider→`QueryError` table. The only coupling between the two concerns is the per-provider seed-field table (`provider-error-mapping.md` §Provider seed-field mapping), which both the `complete()` options enumeration (the request payload's seed key) and the classifier page co-own.

Both halves are independently designable from their cited spec pages and the leaf is reviewable in a single session, so this is an organisational observation about review-surface size, not a correctness or pickability defect.

## Solution Space

**Shape:** single

### Recommendation

Keep `V9j` as a single leaf at its current size. The two concerns are bound by the shared per-provider seed-field table in `provider-error-mapping.md` §Provider seed-field mapping — read both by the `complete()` request-payload seed key and by the classifier — so co-locating the inference call and the classifier in one leaf is coherent rather than accidental, and the leaf remains reviewable in one pass.

If later spec growth materially expands the provider-error table (additional providers, additional overflow signatures, or additional stop-reason classes), record the split trigger so a future maintainer acts on it: in `V9j` **Adds**, note that the provider-error → `QueryError` classifier may be promoted to a dedicated leaf depending on `V9j` should the provider-error mapping outgrow single-session review. No edit is required for correctness now.

## Consequence

**Severity:** cosmetic

Bundling two independent code paths in one leaf enlarges the single-session review surface but does not block pickup, change acceptance criteria, or cause two implementers to diverge — each concern is fully specified by its own spec page and the two test scenarios are independently expressed. No downstream leaf is affected.
