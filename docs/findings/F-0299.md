## Finding

`V17a`'s *Checkpoint granularity* test bullet pairs a definite presence arm ("a checkpoint fires immediately before each of the five sites") with a negative arm introduced as "A best-effort negative arm also asserts the seam witnesses no checkpoint at the non-checkpoint categories the test drives". The qualifier "best-effort" is doing two incompatible jobs at once. Read one way it is a normative hedge — the negative assertion is desirable but optional, so an author could write a thin assertion or none at all and still claim to have honoured the bullet. Read the other way it is meant to flag a *scope* limitation — that the assertion only covers the node kinds the test actually drives and cannot witness a stray checkpoint on an un-exercised path. The bullet already states that scope limitation explicitly in its tail clause ("a checkpoint emitted at a node kind outside the test corpus is not caught by this assertion"), so "best-effort" adds nothing but ambiguity about whether the negative arm is mandatory.

The same "best-effort negative arm" wording is mirrored in `V17a`'s *Ships when* gate and in the companion `V17a-T` test-leaf bullet, so the ambiguity is carried across all three places that an implementer reads when deciding what the negative arm must do. The negative arm's content is in fact fully pinned (no checkpoint inside a primitive operation — arithmetic, comparison, field/index access — and none at a straight-line statement boundary, for the categories the test drives), matching the spec's *Granularity* rule in `cancellation.md` ("No checkpoint fires inside a primitive operation"). The defect is purely the hedging qualifier sitting in front of a requirement whose content is otherwise definite.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 07555ea — pi-loom plan: resolve "completeness overclaims over partial-coverage mechanisms" (2026-06-10, Thomas Andersen)
**History:** Before 07555ea the bullet carried a strong exhaustivity claim — the presence arm asserted a checkpoint fires before each of the five sites "and at no other site", and "The exhaustivity arm asserts *absence* ... not merely presence at the five." Commit 07555ea, resolving an overclaim finding, replaced that unfalsifiable exhaustivity claim with the weaker "best-effort negative arm" phrasing across `V17a`'s Tests and Ships when, and added the mirrored bullet to `V17a-T`, in a single edit. That edit correctly removed the overclaim but introduced the vague "best-effort" qualifier this finding flags.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V17a-cancellation-core.md`, in the *Checkpoint granularity* **Tests** bullet, strike the "best-effort" hedge and state the negative arm as a definite assertion while keeping the explicit out-of-corpus scope note. Replace the sentence

> A best-effort negative arm also asserts the seam witnesses no checkpoint at the non-checkpoint categories the test drives — inside a primitive operation (arithmetic, comparison, field/index access) and at a straight-line statement boundary; a checkpoint emitted at a node kind outside the test corpus is not caught by this assertion.

with

> A negative arm asserts — for the driven primitive operations (arithmetic, comparison, field/index access) and straight-line statement boundaries — that the seam records zero checkpoints; node kinds outside the test corpus are not covered by this assertion.

In the same file's **Ships when** sentence, strike "best-effort" from "together with its best-effort negative arm (no checkpoint at the primitive operations or straight-line statement boundaries the test drives)" so it reads "together with its negative arm (no checkpoint at the primitive operations or straight-line statement boundaries the test drives)".

Apply the same strike-and-restate to the mirrored *Checkpoint granularity* **Tests** bullet in `docs/plan_topics/V17a-T-cancellation-core.md` so the test-companion leaf stays in lockstep with `V17a`. The out-of-corpus scope clause is retained in each case — it is a genuine, honest limitation of a behavioural test and must not be dropped; only the "best-effort" qualifier that conflates "optional" with "scope-limited" is removed.

## Consequence

**Severity:** advisory

The negative arm's content is enumerated and its *Ships when* gate names it, so an implementer cannot silently ship `V17a` without the assertion. But "best-effort" reads as permission to write a token assertion or skip it, so two implementers could diverge on how thorough the negative arm is, and a reviewer has no firm criterion to push back. It degrades the bullet from a checkable requirement to a suggestion.
