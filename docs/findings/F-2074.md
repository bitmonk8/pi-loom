## Finding

The *surface-set closure* property — defined identically in `overview-and-orientation.md` (Orientation → Prerequisites) and `inventory-audit-intro.md` — is stated as: *every Pi-side surface reference in the **audited source tree** resolves to an `SDK_SURFACE_INVENTORY` entry, a typebox sibling allow-list, or a declared `// allow-pi-surface:` exemption.* The audited source tree is in turn defined in `audit-resolution.md` (*Audit scope*) as the closed file set matching the include glob `src/**/*.ts` minus enumerated exclusions, with files outside `src/` and files inside `src/` whose extension is not bare `.ts` (`.tsx`, `.cts`, `.mts`, `.d.cts`, `.d.mts`, `.js`, `.mjs`, `.cjs`, …) declared out of scope "by not matching the include glob and need no separate exclusion."

As scoped, the closure property is trivially complete over the audited source tree — but the value the spec actually leans on (e.g. the trust-boundary reasoning, and `spec.md`'s framing that "the runtime's hand-authored TypeScript runtime source under `src/`" *equals* the audited source tree) is the stronger claim that the audit closes over **all** Pi-touching runtime source. That stronger claim depends on an unstated, unverified premise: that no Pi-touching hand-authored runtime source exists outside `src/**/*.ts`. The premise is never asserted as a precondition and nothing detects its violation. A runtime module authored as `src/foo.mts` or `src/foo.cts` (both valid Node/TS module extensions), a `.tsx` runtime file, or any Pi-touching source rooted outside `src/` would carry live `pi.<member>` / peer-package / `ctx.<member>` references that the audit never walks — silently escaping closure with no audit signal.

The *Audit scope* paragraph does carry a forward-looking governance rule: a future change adopting an unenumerated convention (it names `*.tsx`, `*.gen.ts`, `**/fixtures/**`, etc.) "MUST extend either the include glob or the exclusion list … in the same edit." But that rule is contributor-discipline only — there is no mechanical detector — and it governs only *newly adopted* conventions; it neither asserts the loom 1.0 starting precondition (that the present Pi-touching footprint is wholly within `src/**/*.ts`) nor covers out-of-`src/`-root carriers. The closure guarantee is therefore sound only by assumption, and the assumption is invisible to a reader who trusts the property as stated.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `e5cd514` (2026-05-09, "pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\"")
**History:** The negative-direction audit machinery — the `src/**/*.ts` *Audit scope* definition, the *audited source tree* concept, the *inventory-closure audit*, and the *surface-set closure* property — was introduced as a single unit in `e5cd514` (then living in `spec.md`, before the corpus was split into `spec_topics/` at `81bf52a`). `git log -S` confirms the closed `src/**/*.ts` glob and the closure property first appear together in that commit, and that the spec equated "audited source tree" with "the runtime's hand-authored TypeScript runtime source" from the outset. No later commit (the `589d9a4` surface-set-closure glossary entry, the `f5e89f4` size-cap edit, the `81bf52a` topic split) ever added a precondition asserting that the entire Pi-touching footprint resides within that glob; the gap has existed unchanged since the audit feature was first specified.

## Solution Space

**Shape:** multiple

### Option A — State the loom 1.0 closure precondition explicitly

**Approach:** In `audit-resolution.md` (*Audit scope*), add an explicit precondition, verified at adoption rather than merely asserted, that at loom 1.0 the entire Pi-touching hand-authored runtime source footprint resides under `src/` with the bare `.ts` extension — and that consequently no Pi-touching surface lives in a `.tsx`/`.cts`/`.mts`/other-extension file or outside the `src/` root. Bind this to the existing future-change governance rule so any later adoption of such a carrier must extend the include glob in the same edit. Update the `spec.md` / `inventory-audit-intro.md` closure framing to cite this precondition as the bridge from "audited source tree" to "all Pi-touching runtime source."

**Spec edits:** one new precondition sentence (plus a one-line adoption-verification note) in the *Audit scope* paragraph; a cross-reference from the surface-set-closure definition in `overview-and-orientation.md` / `inventory-audit-intro.md`.

**Pros:** Minimal, proportionate edit; introduces no new mechanical surface; consistent with the spec's existing posture of acknowledging contributor-discipline-only "soft surfaces" (the generated-output-in-`src/` limitation already sits in the same paragraph) and with the explicit deferral of `scope-audit/generated-source-in-tree` past loom 1.0.

**Cons:** The precondition remains a static, adoption-time check; a future `.mts` runtime file would still escape unless the contributor honours the future-change governance rule. The soundness of closure stays partly contributor-enforced.

**Risks:** Low. The verification reduces to a one-time confirmation of the source-tree shape at adoption.

### Option B — Widen the scope definition or add a canary to mechanically detect out-of-root / non-`.ts` Pi-surface carriers

**Approach:** Extend the audit (or its non-empty canary) so an out-of-`src/`-root or non-bare-`.ts` file carrying a recognised Pi-surface shape is detected and surfaced rather than silently skipped — e.g. a sibling scope-audit check (outside the `audit/` three-class namespace, per the existing `scope-audit/` carve-out) that walks the wider tree for Pi-surface tokens and fails on any carrier outside the include glob.

**Spec edits:** widen the include-glob / scope definition in *Audit scope*, or pin a new `scope-audit/<symptom>` discriminator and its fail condition; reconcile with the static-AST tooling floor.

**Pros:** Mechanical and durable; closure soundness no longer rests on contributor discipline; future carriers are caught automatically.

**Cons:** Defining what counts as a "Pi-surface carrier" in files the spec has deliberately declared out of scope reintroduces the detection problem the static-AST floor avoids; adds a new discriminator/check and broadens `V18b`'s scope; the spec already explicitly defers the analogous `scope-audit/generated-source-in-tree` check past loom 1.0, so this option cuts against a stated scope decision.

**Risks:** Moderate — scope creep into a class of check the spec deliberately deferred, and potential tension with the "satisfiable by a pure static-AST walker" claim.

### Recommendation

Take **Option A**: state the loom 1.0 closure precondition explicitly and verify it at adoption. It closes the soundness gap with a one-paragraph edit, stays consistent with the spec's existing treatment of audit "soft surfaces" and its explicit deferral of sibling `scope-audit/` checks past loom 1.0, and avoids inventing a new mechanical scope-audit surface the spec has already decided not to build for 1.0. The implementer must ensure the precondition text binds all three escape vectors — non-bare-`.ts` TS extensions (`.mts`, `.cts`, `.tsx`), out-of-`src/` roots, and the generated-output-in-`src/` case already noted — and links to the existing future-change governance rule so a later carrier cannot be adopted silently.

## Consequence

**Severity:** correctness

The spec asserts a closure guarantee that is stronger than what the audit actually enforces. An implementer can build the audit exactly as specified and still ship a system where Pi-touching surface in a `.mts`/`.cts`/`.tsx` file (or any out-of-`src/` carrier) is wholly unaudited, defeating the surface-set-closure property that the trust-boundary and version-bump reasoning rely on — without any test going red.
