## Finding

`functions.md`'s **Loom return type.** block (anchor `#loom-return-type`) declares that "a `fn` with an explicit return annotation type-checks its tail and every `return` operand against that annotation instead … rather than inferring", and `return.md` restates that "`return expr` is type-checked against the enclosing scope's declared return type." These two sentences together establish a fourth parse-time `T₁ ⊑ T₂` check site beyond the three TYPE-9 already enumerates — the `fn`-return-annotation site, exercised both by the tail expression and by every `return expr` operand against an explicit non-`void` annotation.

`type-system.md` §TYPE-9 names a diagnostic code for each of the other static check sites (`loom/parse/let-rhs-type-mismatch`, `loom/parse/fn-arg-type-mismatch`, and the ternary/array common-type codes routed through `loom/parse/array-element-type-mismatch` / `loom/parse/array-no-common-type`), but TYPE-9 does not enumerate the `fn`-return-annotation site at all, and `diagnostics/code-registry-parse.md` carries no row for it. The two registered return-adjacent codes do not cover this case either: `loom/parse/return-no-common-type` fires only for an *annotation-less* body whose contributing types share no LUB, and `loom/parse/invoke-return-type-mismatch` is a caller-side check at `invoke<Schema>(…)`, not a callee-side check of `fn foo(): Schema { … }`'s tail or `return` operand.

Concretely, `fn foo(): string { 42 }` and `fn foo(): string { if cond { return 42 } else { "ok" } }` both fail TYPE-9's `T₁ ⊑ T₂` check, but no `loom/parse/*` code is named at the check site or registered in the parse-code table — so a conformance test asserting "this loom emits *which* code?" has no answer, and the closing gate ([V7b](docs/plan_topics/V7b-code-registry.md)) cannot reconcile the assertion against the registry.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `648365c` (2026-06-06, *resolve "Three subtype check sites name no diagnostic codes"*) — added TYPE-9 with exactly three sites (`let`-RHS, plain `fn`-arg, ternary), and registered `loom/parse/let-rhs-type-mismatch` and `loom/parse/fn-arg-type-mismatch` in `code-registry-parse.md`. The `fn`-return-annotation site was simply not in scope of that finding.
- `08886d9` (2026-06-07, *resolve "loom return-type inference does not reconcile early-return operand types"*) — rewrote `functions.md`'s **Loom return type.** block and added the explicit sentence "a `fn` with an explicit return annotation type-checks its tail and every `return` operand against that annotation instead … rather than inferring", promoting the `fn`-return-annotation site from implicit to explicitly normative. That edit registered `loom/parse/return-no-common-type` (for the annotation-less LUB-failure case) but did **not** back-port a row for the annotation-bearing mismatch case nor extend TYPE-9.

**History:** `return.md`'s "`return expr` is type-checked against the enclosing scope's declared return type" bullet existed at corpus inception (`fecb504`, 2026-05-04), but no code was named for it then either — pre-TYPE-9, *no* static check site named a code, so the absence wasn't yet a TYPE-9-shaped gap. The gap as currently visible — TYPE-9 enumerating three of four sites, with a registered code for each *except* the return-annotation one — is the joint product of `648365c` defining TYPE-9 over the three sites then-explicit in the corpus and `08886d9` promoting the fourth site to explicit-normative the next day without coordinating an entry into TYPE-9 or the registry.

## Solution Space

**Shape:** single

### Recommendation

Register `loom/parse/return-type-mismatch` and extend TYPE-9 to cite it.

1. Add a new row to `docs/spec_topics/diagnostics/code-registry-parse.md`'s parse table:
   `| loom/parse/return-type-mismatch | E | type | The tail expression of an explicitly-annotated `fn` body, or the operand of a `return expr` inside such a body, has a static type that is not compatible with the declared return annotation under [Type System — Type compatibility](../type-system.md#type-compatibility), where the operand type is statically resolvable. | [Return Statement](../return.md) | — | `return value type mismatch in fn '<name>': expected <expected>, got <actual>` |`
2. Extend `type-system.md` §TYPE-9 from "Three of the compatibility check sites …" to "Four of the compatibility check sites …", and add a sentence: "The tail expression and every `return expr` operand of a `fn` with an explicit non-`void` return annotation report `loom/parse/return-type-mismatch` on a static failure." Update the trailing registry citation to mention the new row alongside the `let`- and `fn`-arg-site rows.
3. In `functions.md`'s **Loom return type.** block, append to the "type-checks its tail and every `return` operand against that annotation" sentence a parenthetical "(`loom/parse/return-type-mismatch` on a static failure; see [Type System — TYPE-9](./type-system.md#type-9))".
4. In `return.md`'s first Rules bullet, after "type-checked against the enclosing scope's declared return type", add "(`loom/parse/return-type-mismatch` on a static failure under [Type System — Type compatibility](./type-system.md#type-compatibility))".

Edge cases the implementer must watch:

- The new code is **callee-side**, distinct from the caller-side `loom/parse/invoke-return-type-mismatch`. Both can fire for the same conceptual mismatch on different sides of an `invoke` boundary; the spec text must keep their triggers crisply distinguishable in the registry **Trigger** column.
- `void` return annotation does not participate in `⊑` (per the existing "`void` does not participate in `⊑`" paragraph in `type-system.md`) — the new code only fires for non-`void` annotations. `void` discard of a tail value is governed by the **Empty-tail body** block, and bare `return` in a non-`void` annotated fn is already covered by `loom/parse/bare-return-in-non-void`.
- The unresolvable-operand carve-out from `type-system.md` applies here too: a `return expr` whose operand type is past the parser's static view skips the parse-time check and falls through to the runtime safety net, exactly as the `invoke` side does. The new registry row's **Trigger** must include the "statically resolvable" qualifier verbatim, matching the wording on the `let`/`fn-arg` rows.
- Per GOV-22, no new co-located REQ-ID anchor is needed on `type-system.md` because TYPE-9 already carries `<a id="type-9">` — the edit extends the existing obligation rather than coining a new one.

## Consequence

**Severity:** correctness

Two implementers will diverge: one will route a return-annotation tail/operand mismatch through `loom/parse/return-no-common-type` (misleading — that code is reserved for annotation-less LUB failure), another will coin an ad-hoc `loom/parse/return-type-mismatch`, and a third will surface the closest cousin `loom/parse/invoke-return-type-mismatch` at the wrong site. The `H5a` closing gate ([V7b](docs/plan_topics/V7b-code-registry.md)) cannot pass a test asserting *any* code at this site because none is registered, so the obligation is untestable as written.
