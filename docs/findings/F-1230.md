## Problem

`functions.md`'s `#loom-return-type` block and `return.md`'s first Rules bullet together make the `fn`-explicit-return-annotation site a normative parse-time `T₁ ⊑ T₂` check — the tail expression and every `return expr` operand are checked against the declared annotation. `type-system.md` §TYPE-9 enumerates only three compatibility check sites and names a diagnostic code for each, but does not enumerate the return-annotation site, and `diagnostics/code-registry-parse.md` carries no row for it. The two return-adjacent registered codes do not cover this case: `loom/parse/return-no-common-type` fires only for an annotation-less LUB failure, and `loom/parse/invoke-return-type-mismatch` is a caller-side check. A conformance test asking which code `fn foo(): string { 42 }` emits has no registered answer.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `648365c` (2026-06-06, *resolve "Three subtype check sites name no diagnostic codes"*) — added TYPE-9 with exactly three sites (`let`-RHS, plain `fn`-arg, ternary), and registered `loom/parse/let-rhs-type-mismatch` and `loom/parse/fn-arg-type-mismatch` in `code-registry-parse.md`. The `fn`-return-annotation site was simply not in scope of that finding.
- `08886d9` (2026-06-07, *resolve "loom return-type inference does not reconcile early-return operand types"*) — rewrote `functions.md`'s **Loom return type.** block and added the explicit sentence "a `fn` with an explicit return annotation type-checks its tail and every `return` operand against that annotation instead … rather than inferring", promoting the `fn`-return-annotation site from implicit to explicitly normative. That edit registered `loom/parse/return-no-common-type` (for the annotation-less LUB-failure case) but did **not** back-port a row for the annotation-bearing mismatch case nor extend TYPE-9.

**History:** `return.md`'s "`return expr` is type-checked against the enclosing scope's declared return type" bullet existed at corpus inception (`fecb504`, 2026-05-04), but no code was named for it then either — pre-TYPE-9, *no* static check site named a code, so the absence wasn't yet a TYPE-9-shaped gap. The gap as currently visible — TYPE-9 enumerating three of four sites, with a registered code for each *except* the return-annotation one — is the joint product of `648365c` defining TYPE-9 over the three sites then-explicit in the corpus and `08886d9` promoting the fourth site to explicit-normative the next day without coordinating an entry into TYPE-9 or the registry.

## Solution approach

Register a new callee-side parse code (e.g. `loom/parse/return-type-mismatch`) in `code-registry-parse.md`'s parse table for a tail expression or `return expr` operand whose statically-resolvable type is incompatible with an explicit non-`void` `fn` return annotation. Extend `type-system.md` §TYPE-9 to enumerate this fourth check site and cite the new code. Add forward-links to the new code from `functions.md`'s `#loom-return-type` block and `return.md`'s first Rules bullet.

## Solution constraints

- Keep the new callee-side code's registry **Trigger** distinct from the caller-side `loom/parse/invoke-return-type-mismatch`; both can fire across an `invoke` boundary for the same conceptual mismatch.
- The new code fires only for non-`void` annotations (`void` does not participate in `⊑`); bare `return` in a non-`void` annotated `fn` remains covered by `loom/parse/bare-return-in-non-void`.
