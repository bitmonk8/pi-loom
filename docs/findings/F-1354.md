## Problem

The BNDR-9 Tests bullet in `V11b` and `V11b-T` (and the matching `V11b` Ships when clause) asserts that the user-facing system note "renders verbatim from the custom-type-unsafe-cause row" of `determinism-cancellation-failure.md`. That row is a parameterised template carrying `<name>` and `<value>` substitution markers, so "renders verbatim from the … row" admits two distinct, mutually exclusive oracles: comparing the emitted note's `content` against the placeholder-substituted text, or comparing it against the raw template literal with `<…>` markers intact. The bullet does not say which string is compared, so two implementers would write divergent BNDR-9 tests from the same wording — one of which asserts against a string the runtime never emits.

## Issue introduction

**Verdict:** clean-introduction
**Introducing commits:** f41af5d2b2e402927f200172cdf09170f87bf325 (2026-06-24)
**History:** The "renders verbatim from the custom-type-unsafe-cause row" phrasing did not exist before commit f41af5d. `git log -S "renders verbatim"` and `git log -S "custom-type-unsafe-cause row"` against both `V11b` and `V11b-T` return only that commit — the F-1131 fix that defined the BNDR-9 unsafe-customType outcome as failure, added the custom-type-unsafe Failure-mode-templates row, and extended the V11b/V11b-T oracles to assert invocation-fails + system-note-matches. The ambiguity is a side effect of introducing the parameterised template row and the "renders verbatim" oracle wording in the same change, without stating which string (rendered vs raw) the oracle compares.

## Solution approach

Clarify the BNDR-9 Tests bullet in `V11b` and `V11b-T`, and the `V11b` Ships when clause, to name the compared string: the oracle compares the emitted system note's `content` against the **rendered** custom-type-unsafe row — the template with its `<name>` and `<value>` placeholders substituted — not the raw template literal. Apply the same disambiguation across all three sites so the implementation leaf and its paired test leaf agree.
