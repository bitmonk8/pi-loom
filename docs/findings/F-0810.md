## Finding

The prompt-mode paragraph in `pi-integration-contract/tool-registration-lifetime.md` pins the registration cache to a specific in-memory data structure — `Map<schemaSlug, registeredToolName>` — in continuous normative prose:

> The runtime maintains an extension-scoped registration cache: `Map<schemaSlug, registeredToolName>`, where the slug is the schema slug of the lowered tool `parameters` …

What is actually observable to an implementer's tests, to Pi, and to the runtime-event channel is only: (i) the slug → registered-name lookup is content-addressed on a SHA-256-derived 64-bit slug; (ii) a cache hit reuses the prior `pi.registerTool` registration; (iii) before reuse, byte-equality of the cached canonical-form schema bytes against the new entry is verified; (iv) a byte-mismatch raises `loom/runtime/registration-cache-collision` and registers under a counter-suffixed disambiguated name; and (v) the cache is dropped on `ctx.reload()`. Whether the runtime implements that contract with `Map`, with a `Record<string, …>`, with two parallel arrays, or with an LRU is invisible from outside the runtime.

The `Map<…>` spelling is therefore an implementation hint promoted to normative prose. It is also inconsistent with how the same shard treats the sibling `chokidar` over-prescription finding and how the page already treats `defineTool` (de-prescribed as illustrative). The glossary entry for **schema slug** restates the same `Map<schemaSlug, registeredToolName>` spelling and must be co-edited so the two sites do not drift.

## Solution Space

**Shape:** single

### Recommendation

In `tool-registration-lifetime.md`'s prompt-mode paragraph, replace the data-structure spelling with a behavioural description. Concretely, rewrite the opening sentence as:

> The runtime maintains an extension-scoped registration cache that maps each schema slug (produced by the recipe in [Schema Subset — Canonical schema hash](../schema-subset.md#canonical-schema-hash)) to the registered tool name returned by `pi.registerTool`, alongside the canonical-form schema bytes used to verify byte-equality on cache hits.

Keep the surrounding obligations verbatim — the first-encounter `pi.registerTool` call, the content-addressed synthesised names (`__loom_callee_<slug>__<post-rename-name>`, `__loom_respond_<slug>`), the byte-equality guard, the `loom/runtime/registration-cache-collision` emission, the counter-suffixed disambiguation, and the `ctx.reload()` cache-drop — since each of those is independently observable and load-bearing.

Co-edit the **schema slug** entry in `glossary.md` to drop the `(`Map<schemaSlug, registeredToolName>`)` parenthetical, replacing it with `(the prompt-mode registration cache defined in [Pi Integration Contract — Tool-registration lifetime](./pi-integration-contract/tool-registration-lifetime.md#tool-registration-lifetime-and-visibility))` so the two sites do not drift.

Edge cases for the implementer to keep in mind: (i) the byte-equality guard requires storing the canonical-form bytes alongside the registered name — the behavioural rewrite must keep this requirement explicit so an implementer does not store only the slug → name pair; (ii) the per-slug disambiguation counter is observable through the synthesised name suffix and must remain monotonically increasing per slug across reuse, which is part of the contract independent of how the counter is held; (iii) the `Map<slashName, parsedLoom>` spelling for the unrelated `LoomRegistry` in `drain-state-contract.md` and `registration-steps.md` is a different concern (governed by its own per-method call surface) and is out of scope for this edit.

## Consequence

**Severity:** cosmetic

An implementer reading the page may take `Map<schemaSlug, registeredToolName>` as a contract on the container type and over-constrain their implementation (refusing, for example, a `Record`-based cache during review). No observable runtime behaviour, diagnostic, or test outcome differs across container choices, so the spec text mildly misleads without producing divergent implementations.
