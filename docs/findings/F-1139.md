## Finding

The `loom/load/missing-source` row in `docs/spec_topics/diagnostics/code-registry-load.md` carries severity `E/W`. The column legend in `docs/spec_topics/diagnostics/diagnostic-shape.md` defines `E/W` as "the severity is not fixed at the code level but resolved per diagnostic by that row's own *Trigger* prose", and gives this row as one of its motivating examples ("per-source by the table in [Discovery — Failure modes]"). For an `E/W` row, both severity arms are reachable by construction — that is the whole point of the column-level convention.

This row's Trigger prose, however, says outright that the W arm never fires: "error for explicit references (`pi.looms` entries, settings `loomPaths` entries, `--loom` flags), **warning never** — conventional locations (global, project, package `looms/`) emit no diagnostic when missing." The failure-modes table at `docs/spec_topics/discovery/discovery-sources.md` (DISC-2) confirms this: the *Missing path* column reads `silent` for all three conventional sources and `error` for all three explicit sources — there is no row anywhere for which a missing path produces a warning.

The sibling rows `loom/load/unreadable-source` and `loom/load/wrong-type-source` are genuinely `E/W` (their `Unreadable path` and `Path is wrong type` columns in DISC-2 carry both `error` and `warning` entries). `missing-source` is the odd one out: it is a single-severity `E` code that has been mistakenly typed as `E/W` by analogy with its two sibling discovery-source codes.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `5b772b6` — *pi-loom spec: resolve "Discovery source failure-mode policy is unspecified"* (Mon May 4 2026)
**History:** The row was added by `5b772b6` in `spec_topics/diagnostics.md` already carrying `Sev: E/W` together with the "warning never" Trigger clause, in the same edit that introduced the DISC-2 failure-modes table whose *Missing path* column has no warning entries. The mismatch has therefore existed since the row first appeared. Commit `f5e89f4` (Thu Jun 4 2026) later relocated the row verbatim into `docs/spec_topics/diagnostics/code-registry-load.md` when the diagnostics page was split for the 30/100 KB caps; no edit since has touched the severity column or the "warning never" clause (`git log -L /missing-source/,+1:docs/spec_topics/diagnostics/code-registry-load.md` reports only the file-split commit). The two sibling discovery-source codes added in the same `5b772b6` edit (`unreadable-source`, `wrong-type-source`) are *genuinely* `E/W` against DISC-2, which is consistent with the row having been typed `E/W` by sibling-analogy rather than by checking the table.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/diagnostics/code-registry-load.md`, change the `loom/load/missing-source` row's *Sev* column from `E/W` to `E`, and rewrite the Trigger prose to drop the dead-W disjunct so the row reads as a single-severity error code that documents an adjacent silence rule, not as a two-arm severity rule one arm of which is empty. Suggested Trigger replacement:

> A discovery source path explicitly named by configuration — a `pi.looms` entry, a settings `loomPaths` entry, or a `--loom` flag — does not exist. Per the failure-modes table in [Discovery — Failure modes](../discovery.md), missing paths under the three conventional discovery roots (global, project, package `looms/`) emit no diagnostic at all; this code fires only for the three explicit-reference sources.

Leave `loom/load/unreadable-source` and `loom/load/wrong-type-source` untouched — both are genuinely `E/W` against DISC-2, so the column-legend example in `diagnostic-shape.md` ("for the discovery-source codes") remains accurate and needs no edit. Implementer edge case: the `Message` column template (`discovery source path does not exist: <descriptor>`) is unchanged, and the row already carries the correct closed `<descriptor>` set in DISC-2's *All warnings and errors above are emitted via the standard diagnostics channel* paragraph — no descriptor enumeration migrates between row and table as part of this fix.

## Consequence

**Severity:** cosmetic

The Trigger prose pins behaviour unambiguously ("warning never"; per-source mapping in DISC-2), so a careful implementer will produce the right wire output. The damage is reader cost and a small risk of a registry-driven implementation wiring a never-taken severity branch behind the row, plus the implicit promise the column legend makes (`E/W` ⇒ both arms reachable) being broken at one of the two rows it cites as its motivating example.
