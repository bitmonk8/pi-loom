## Finding

The final paragraph of `docs/spec_topics/cancellation.md` reads:

> Per-call timeouts (a separate cancellation source independent of the user) are **deferred to a later release**; declaring a `timeout:` field on a query, tool call, or invoke is `loom/parse/timeout-field-rejected`. See [Future Considerations](./future-considerations.md) and [Diagnostics](./diagnostics.md).

The paragraph carries a genuine loom 1.0 normative rule — a `timeout:` field at any of the three sites is rejected with `loom/parse/timeout-field-rejected` — but packages it as a subordinate clause inside "deferred to a later release" future-work prose. The normative content is the lead obligation an implementer must act on; the deferral status is rationale. The current ordering inverts that, presenting the rule as an aside to the roadmap note.

The deferral framing is also weakly anchored: the sentence points readers to `future-considerations.md`, but that page does not list per-call timeouts (the deferred-surface rationale lives instead in the open-struct seam notes in `invocation.md` (INV-3) and `tool-calls.md`). The diagnostic-code home, `docs/spec_topics/diagnostics/code-registry-parse.md`, already states the rejection normatively in present tense ("a per-call timeout field is rejected"), so the topic page should match that posture rather than re-narrate the deferral.

## Solution Space

**Shape:** single

### Recommendation

Restructure the final paragraph so the normative rejection is the lead clause and the deferral status becomes a brief non-normative tail. For example:

> A `timeout:` field declared on a query, tool call, or invoke is rejected with `loom/parse/timeout-field-rejected`; loom 1.0 has no per-call timeout (a cancellation source independent of the user). The seam reserved for it is recorded in [Future Considerations](./future-considerations.md).

Keep present-tense rejection wording consistent with the `loom/parse/timeout-field-rejected` row in `code-registry-parse.md` ("a per-call timeout field is rejected"). Drop the "deferred to a later release" phrasing. Retain the `Future Considerations` pointer only if it resolves to a section that actually documents the per-call-timeout seam; if it does not, point instead at the seam's true home (the open-struct notes in `invocation.md`/`tool-calls.md`) or drop the link rather than dangling it. The `Diagnostics` link is redundant once the code is stated normatively in-line and may be dropped.

## Consequence

**Severity:** cosmetic

No behaviour or acceptance criterion changes: the rejection rule (`timeout:` → `loom/parse/timeout-field-rejected`) is already stated in present tense and is unambiguous regardless of the surrounding framing. The defect is editorial — a normative rule presented as roadmap prose, with a `See:` link to a page that does not carry the referenced deferred-surface content.
