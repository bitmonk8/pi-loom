## Problem

The `loom/parse/invoke-arity-too-few` / `loom/parse/invoke-arity-too-many` Tests bullet in `V15f` and `V15f-T` asserts that more than the total `params:` count is always `invoke-arity-too-many`, "even when the callee is not statically resolvable." This is internally contradictory: the `params:` count comes from the callee's frontmatter, and per `invocation.md` §Static resolution a callee is *not statically resolvable* precisely when its file is unreadable, unparseable, or fails its own structural checks — exactly the case where the `params:` count cannot be obtained. So the comparison operand the rule depends on does not exist in the branch the rule claims to cover. The asymmetry with the sibling too-few clause (which correctly distinguishes the statically-resolvable parse-error path from the runtime AJV fallback) makes the gap conspicuous, and because the leaf transcribes `invocation.md` §Argument arity verbatim the contradiction lives in both the spec and the plan leaf.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 833d8d5 — spec: resolve five invoke edge cases (Decision #4) (2026-05-05, Thomas Andersen)
**History:** The contradictory "Too many arguments … Always a parse error `loom/parse/invoke-arity-too-many`, even when the callee is not statically resolvable" bullet entered `invocation.md` §Argument arity in 833d8d5 (2026-05-05); pickaxe (`git log -S 'even when the callee is not statically resolvable' --follow`) localises it to that single commit. The plan side is a faithful transcription, not an independent introduction: the phrase first reached the plan in cddd2b4 (2026-06-11, on the V15a leaves) and was carried into the dedicated `V15f`/`V15f-T` leaves when they were split out in 5f4d271 (2026-06-17). The defect therefore originates in the spec and propagated into the plan unchanged.

## Solution approach

Per the **Spec drift** rule in `conventions.md`, rewrite the *Too many arguments* clause in `invocation.md` §Argument arity to gate the arity check on static resolution exactly as the too-few clause is gated, then bring the `V15f` and `V15f-T` Tests bullet into agreement. The corrected content: more than the total `params:` count is `loom/parse/invoke-arity-too-many` **when the callee is statically resolvable**; when the callee is not statically resolvable the param count is unknown so no arity parse error fires — the site already carries `loom/load/callee-has-errors`. Drop the unqualified "always … even when the callee is not statically resolvable."

## Solution constraints

- Unlike the too-few clause's runtime AJV fallback, the not-statically-resolvable too-many case has no runtime safety net for excess positional arguments — do not introduce one; `loom/load/callee-has-errors` is the only surface for that case.
