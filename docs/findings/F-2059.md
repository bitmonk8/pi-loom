## Problem

The *Registry contract* "Removal" bullet in `active-invocation-registry.md` names the removal site as "the same `finally` block that disposes the subagent `AgentSession`" — an action that only occurs in subagent mode. A prompt-mode entry has no `AgentSession` to dispose, so the bullet names no removal block for prompt-mode entries. This is self-inconsistent with the same section's *Per-mode concurrency invariants* bullet, which affirms the registry tracks both prompt-mode and subagent-mode entries, and with the section's intro paragraph, which already frames removal in terms of "any subagent session" settling "immediately (prompt mode)". Read in isolation, the contract bullet is locally incomplete for half the entry classes it governs.

## Issue introduction
**Verdict:** introduced
**Introducing commit:** `e4a3882` (2026-05-07, "pi-loom spec: resolve \"Session model: rewrite of concurrent-invocations and ActiveInvocationRegistry framing\"")
**History:** The corpus is git-tracked. `git log -S '**Removal** happens in the same'` and a parent/child blob comparison show the *Registry contract* "Removal" bullet did not exist in `e4a3882`'s parent and was net-added in `e4a3882` (then to `spec_topics/pi-integration-contract.md`). At that same commit the enclosing *ActiveInvocationRegistry* intro paragraph already contained "disposes any subagent session" and "settles after `AgentSession.dispose()` returns (subagent mode) or immediately (prompt mode)" (confirmed present in `e4a3882^:spec_topics/pi-integration-contract.md`). The bullet was therefore born subagent-only despite its own section already acknowledging prompt mode. A later commit `14424b4` (2026-06-06) added the explicit *Per-mode concurrency invariants* bullet affirming "The registry tracks both prompt-mode and subagent-mode entries," which widened the mismatch but did not touch the "Removal" bullet (its current blame is `f5e89f4`, the 2026-06-04 topic split — unchanged since introduction). The defect is single-commit in origin (`e4a3882`); `14424b4` only sharpened its visibility.

## Solution approach

Rewrite the *Registry contract* "Removal" bullet so its removal site is the mode-agnostic per-invocation `finally`, with subagent `AgentSession` disposal named as the subagent-only sub-case rather than as the defining action. Retain the "after `disposeBarrier` settles" ordering clause — it is already mode-agnostic (settles after `AgentSession.dispose()` in subagent mode, immediately in prompt mode), so it covers both entry classes once the removal site itself is mode-agnostic.
