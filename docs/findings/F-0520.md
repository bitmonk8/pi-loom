## Problem

The `DIAG-3` Tests bullet in `V7b-code-registry.md` and `V7b-T-code-registry.md` claims "a renamed code fails the gate," but the only named mechanism — the `H5a` closing gate's present-state test↔registry reconciliation — reddens only on an *unsynchronised* rename (a registry row or an asserting test renamed without the other). A *coordinated* rename of the registry row and every asserting test in one commit stays mutually consistent and passes the gate green. The spec classifies a registered-code rename as a breaking change deferred to loom 2.0 and assigns its detection to the GOV-15 reviewer inspection of the registry diff (`source-language-stability.md#diagnostic-registry-carve-out`), not a mechanical CI gate. The bullet therefore over-states the gate's reach and obscures that GOV-15 review is the real rename backstop.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** The DIAG-3 bullet "codes are stable identifiers — a renamed code fails the gate (renames deferred to loom 2.0)" has been present verbatim in both `V7b-code-registry.md` and `V7b-T-code-registry.md` since the commit that first authored the plan-topics leaves. `git log -S"a renamed code fails the gate"` and `git log -G"DIAG-3"` over both leaves return only `c6a664e`; the sole subsequent commit touching `V7b-code-registry.md` (`f284534`, "resolve DIAG-2 src/** emission scan vs closing gate") did not alter the DIAG-3 line. The overclaim entered with the leaf and was never modified.

## Solution approach

Reword the `DIAG-3` Tests bullet so it claims only what the `H5a` reconciliation delivers — the gate fails an *unsynchronised* rename — and points a *coordinated* rename at the GOV-15 reviewer-inspection backstop (`source-language-stability.md#diagnostic-registry-carve-out`) rather than implying CI catches every rename. Apply the same edit to the bullet in `docs/plan_topics/V7b-T-code-registry.md` (the authoritative source) and `docs/plan_topics/V7b-code-registry.md` (which duplicates the `-T` Tests verbatim).

## Solution constraints

- The spec is read-only for this fix; do not add a mechanical rename detector or code-id manifest, since the spec assigns coordinated-rename detection to GOV-15 review.
- Out of scope: the `H5a` gate's `Adds.` reconciliation, which already describes the contract correctly — confine the edit to the `DIAG-3` bullet wording.
