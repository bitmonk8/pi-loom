## Problem

The *Signature* block under **Renderer registration** pins the `MessageRenderer` signature, return type, and lifecycle as the loom-load-bearing subset re-validated against `dist/core/extensions/types.d.ts` on each Pi minor bump, but two host shapes the renderer body depends on are never pinned to that footing. The component-construction entry point is unnamed: the block sketches the `Component` interface and cites its declaration site but never names how a renderer instantiates one (the SDK's exported `Text` class), an asymmetry against the `defineTool(...)` helper pinned for the tool path. The `theme: Theme` parameter is named only by prose with no member surface and no declaration site, and the surrounding "Component lives in pi-tui" framing implies `Theme` is also a pi-tui type, whereas it is declared in `@earendil-works/pi-coding-agent`. Neither shape has a recorded subset for the page's re-validation discipline to diff a future Pi release against, so the renderer cannot be implemented to spec without out-of-band SDK reading and the framing actively misdirects the `Theme` lookup.

The asymmetry has been present since that block's inception; the later page-split commit (78a2f94) carried it verbatim into `extension-bootstrap-and-per-loom.md`.

## Solution approach

Extend the *Signature* block under **Renderer registration** to pin the renderer's two remaining host dependencies on the same loom-load-bearing, re-validated-each-Pi-minor-bump footing as the signature, attaching both to the section preamble's existing re-validation obligation. Name `pi-tui`'s exported `Text` class (declared at `@earendil-works/pi-tui`'s `dist/components/text.d.ts`) as the `Component` construction entry point, with the loom-load-bearing constructor shape `new Text(text?: string, paddingX?: number, paddingY?: number, customBgFn?: (text: string) => string)`. Pin the `Theme` member subset the renderer reads — `fg(color: ThemeColor, text: string): string` together with the `ThemeColor` literals it uses (the `"dim"` member backing the existing dim-text example, plus system-note colours such as `customMessageText` / `customMessageLabel`) — and correct its declaration site to `@earendil-works/pi-coding-agent` (`dist/modes/interactive/theme/theme.ts`), not pi-tui. Ensure the `#pic-21` fallback `Component` constructs through the same named entry point.
