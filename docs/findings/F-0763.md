## Finding

In step 5 of `docs/spec_topics/pi-integration-contract/registration-steps.md` (the **Structural changes** paragraph), the rule that gates emission of the `loom-system-note` is stated formally as `added.length + removed.length === 0` and is reinforced by an explicit rename carve-out ("A rename observed within the same debounce window as `removed` of path P followed by `added` of path P still has `added.length + removed.length === 2` and MUST emit the note"). The same paragraph then enumerates illustrative empty-window examples and includes the bullet "a chokidar burst over discovery roots that **does not net any added or removed** `.loom` / `.warp` files."

The verb phrase "does not net" is the colloquial sense of *arithmetic netting* — i.e. additions and removals cancelling each other out. Under that reading, a same-window rename (1 add, 1 remove) **does** "net" to zero added and zero removed, which would land in the suppressed set. That contradicts both the formal `added.length + removed.length === 0` rule directly above and the rename carve-out directly below. The example bullet is the only sentence in the paragraph that drifts away from the disjoint-by-role array model the rest of the paragraph carefully pins.

The defect is local to one example bullet; the surrounding normative rule is correct. A careful implementer who reads the whole paragraph will reconstruct the right behaviour, but the example bullet is the kind of phrasing a reviewer will paraphrase out of context and a test author may anchor a misleading assertion on.

## Solution Space

**Shape:** single

### Recommendation

Replace the offending example bullet's verb phrase with one that matches the disjoint-by-role array model the rest of the paragraph uses. Concretely, in the **Structural changes** paragraph of step 5, change

> a chokidar burst over discovery roots that does not net any added or removed `.loom` / `.warp` files

to

> a chokidar burst over discovery roots that resolves to zero entries in both the `added` and `removed` path lists (`added.length + removed.length === 0`)

so the example bullet uses the same `added.length + removed.length === 0` predicate the formal rule pins, and so the rename carve-out that immediately follows reads as a coherent contrast (`=== 2`, not `=== 0`) rather than as a correction.

Edge cases the implementer must watch:

- The fix is purely textual within the one example bullet; the normative rule (`MUST NOT be emitted when added.length + removed.length === 0`) and the rename carve-out are correct and must stay verbatim.
- Mirror the same wording change in `docs/plan_topics/V9b-registration-reload-wiring.md` and `docs/plan_topics/V9b-T-registration-reload-wiring.md` (the `cka-23.note-empty` facet bullet currently paraphrases the defective sentence as "a burst that nets no added/removed `.loom`/`.warp` file") so the spec and the test description remain aligned.
- Do not touch the other two empty-window examples in the same enumeration (the byte-identical-`loomPaths` and `looms.binderModel`-only cases); they are unambiguous and not in scope.

## Consequence

**Severity:** advisory

Two careful readers can already converge on the correct behaviour because the formal `added.length + removed.length === 0` rule and the rename carve-out are both present in the same paragraph. The risk is paraphrase drift: an implementer or test author who quotes only the colloquial example bullet (e.g. into a leaf, a test name, or a PR description) imports the arithmetic-netting reading and could write a test that wrongly *suppresses* the note on a same-window rename. The `V9b` / `V9b-T` test bullets already carry the same paraphrase, which is exactly the risk path materialising.
