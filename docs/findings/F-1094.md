## Finding

The entire `npm test` surface — the `H4a` session-double fidelity-contract self-check, the `H7a` cross-slice integration-acceptance run, and the `V18d` version-bump runtime-evidence gate — executes against the in-process Pi session double, never a live Pi host. `H4a` states explicitly that the harness "cannot detect the double diverging from real Pi" and that "loom 1.0 has no mechanical real-host fidelity gate." Consequently, if the double models any of the four fidelity axes incorrectly (streamed-token order relative to `ctx.waitForIdle()`, single-turn prompt-mode append semantics, the `pi.on` cancel-forward subscription, cancellation propagation), every loom invocation can misbehave against a real host while CI stays uniformly green.

The sole backstop is the manual real-host smoke run defined in `real-host-smoke-gate.md`, which drives `H7a`'s single committed composite fixture `.loom` against a live host on two triggers (a Pi version bump, or a merge whose author recognises it touches the four axes). That backstop has two coverage limits the plan concedes but does not close: (1) the live run scores the four axes only as they happen to be exercised by one composite fixture, with no requirement that each axis be witnessed individually, so an axis the composite path does not stress can pass without ever being checked; and (2) the gate is purely pre-merge and contributor-driven — there is no post-release detection path, so a divergence that escapes the manual gate (a skipped run, an unrecognised fidelity-touching diff, or an axis the fixture under-exercises) surfaces only as silent real-host misbehaviour with no operator-visible signal.

This is the residual-risk posture deliberately adopted when double fidelity was reclassified as an editorial-review presupposition rather than a mechanical real-host backstop. The finding does not dispute that posture; it flags that the single manual fixture leaves both per-axis pre-release coverage and any post-release detection unaddressed.

## Solution Space

**Shape:** multiple

This finding asks the plan for two independent obligations: tightening the pre-release manual gate's per-axis coverage, and adding a post-release detection mechanism. They touch different surfaces (a governance note vs. a new runtime diagnostic with a spec dependency) and should be resolved separately rather than as one combined edit.

### Option A — Per-axis real-host fixture coverage in the manual smoke gate

- **Approach:** Strengthen the manual real-host smoke pass criterion so each of the four fidelity axes is individually witnessed by the live run, rather than only as the single composite `H7a` fixture happens to exercise them.
- **Plan edits:** In `real-host-smoke-gate.md`, the pass/fail criterion (d) currently requires "the four session-double fidelity-contract axes … hold"; add that the live run must witness each of the four axes individually (streamed-token order relative to `ctx.waitForIdle()`; single-turn prompt-mode append; the `pi.on` cancel-forward subscription; cancellation propagation), and that a composite fixture that does not exercise an axis does not satisfy that axis. Reflect the same per-axis witnessing in `H6a`'s Ships-when evidence record (the recorded smoke result must attest each axis).
- **Spec edits:** None.
- **Pros:** Closes the pre-release coverage half entirely within plan files; no new runtime mechanism; no spec change; directly auditable through the `H6a` gate-activation commit-message evidence record.
- **Cons:** Still pre-merge and contributor-driven; does nothing for divergences that escape the gate (skipped run, unrecognised fidelity-touching diff).
- **Risks:** If the single composite fixture cannot stress all four axes, an additional or extended fixture may be needed, increasing `H7a` fixture-maintenance burden.

### Option B — Post-release runtime divergence-detection telemetry

- **Approach:** Add a runtime divergence-detection path so an encoded host-shape assumption that fails at runtime surfaces as an operator-visible diagnostic rather than silent misbehaviour.
- **Plan edits:** A new leaf (`<new>`) that emits a `loom/runtime/*` diagnostic when an observed host-shape assumption fails, and a citing closing path in the coverage matrix.
- **Spec edits:** The spec must first define the new runtime divergence diagnostic and its REQ-ID; a runtime detection surface is a new mechanism and falls under the **Spec drift** rule (fix the spec before planning the leaf). Until the spec owns it, the plan cannot cite it.
- **Pros:** Provides the only signal for divergences that bypass the manual gate; converts silent real-host misbehaviour into an operator note.
- **Cons:** Introduces a new runtime surface that may brush against the loom 1.0 non-goals; cannot be authored from plan files alone — requires a spec decision first.
- **Risks:** Scope creep if pursued without a spec owner; a runtime self-diagnosis of host-shape assumptions can itself be unreliable (false positives/negatives) and needs its own design.

### Recommendation

Resolve Option A first: it is scope-bounded, plan-internal, requires no spec change, and lands a concrete improvement (per-axis live witnessing) on a stable baseline. Then evaluate Option B separately. Option B introduces a new runtime diagnostic surface and therefore depends on a prior spec decision (per the **Spec drift** rule) about whether loom 1.0 should own a runtime divergence-detection mechanism at all given its non-goals — do not let an Option-B runtime mechanism be added by a plan edit before the spec owns it. Sequencing A before B keeps the larger, spec-gated obligation off the smaller fix's baseline. Edge case the implementer must watch: a composite fixture that silently fails to exercise an axis must not be allowed to satisfy that axis's per-axis criterion under Option A.

## Consequence

**Severity:** advisory

If the double diverges from real Pi on any fidelity axis the single composite smoke fixture does not individually stress, loom 1.0 can ship with that axis never witnessed against a live host, and any post-release divergence is invisible to operators because there is no runtime detection signal — the failure manifests only as silent real-host misbehaviour. Implementers can still produce working leaves; this is a release-risk coverage gap in the manual backstop, not a per-leaf correctness divergence.
