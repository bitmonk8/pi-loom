## Problem

`H4a` establishes the manual real-host smoke governance protocol as a required pre-merge gate on two triggers (every Pi version bump, and any merge touching the four session-double fidelity-contract axes), both active as soon as `H4a` lands — `H4a`'s only `Deps` entry is `H3a`. But the smoke run is defined entirely in terms of `H7a`-owned artefacts: it drives `H7a`'s committed multi-feature fixture `.loom`, and pass criterion (e) scores emitted `loom-system-note` codes against the committed permitted-code list checked in alongside that fixture. `H7a` lands much later (its `Deps` name 20 leaves), so neither input exists in the `H4a`→`H7a` window. A trigger firing in that window therefore activates a required gate whose committed reference set and fixture are absent, and the protocol text gives the contributor no stated behaviour for that case.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 15eb44c — pi-loom plan: resolve "real-host divergence pre-merge gate (posture i)" (2026-06-11, Thomas Andersen); e7f14dd — pi-loom plan: resolve "Real-host fidelity of the session double has no reproducible detection point" (2026-06-11, Thomas Andersen); 052b019 — pi-loom plan: resolve "Real-host smoke pass criterion (e) names a permitted code set with no committed source" (2026-06-11, Thomas Andersen)
**History:** 15eb44c introduced the two-trigger required-pre-merge-gate framing (firing on every Pi bump and every fidelity-axis merge). e7f14dd then anchored the smoke run to reuse `H7a`'s committed fixture `.loom`, and 052b019 anchored pass criterion (e) to `H7a`'s committed permitted-code list. The triggers (always-on from `H4a`) and the `H7a`-only inputs were thus established by separate commits; none added an activation-window clause, so the ordering gap is the interaction of those changes (`git log -S "from H7a onward"` finds no commit ever introducing such scoping).

## Solution approach

In `H4a`'s **Adds** paragraph that establishes the manual real-host smoke governance protocol, add activation-window scoping so the two triggers are dormant until `H7a`'s committed fixture `.loom` and permitted-code list exist. State that a trigger event occurring in the `H4a`→`H7a` window is deferred — re-run once `H7a` has landed — rather than scored against absent inputs as a fail or behavioural-divergence finding.

## Solution constraints

- `H7a` remains the sole producer of the fixture `.loom` and the permitted-code list; out of scope to move or duplicate those artefacts into an earlier leaf.
