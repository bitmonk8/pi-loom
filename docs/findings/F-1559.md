## Finding

DISC-6 (the bounded package walk in `docs/spec_topics/discovery/package-and-settings.md`) specifies the per-read timeout path as: on per-read timeout the in-flight read is abandoned and "the read's eventual settlement **MUST be silenced with `.catch(() => {})`** and its result MUST NOT be re-routed back into the discovery pass." The genuine observable requirement is twofold and idiom-independent: (a) the abandoned read's late settlement must not surface as a Node `unhandledRejection`, and (b) its result must not re-enter the discovery pass. Naming `.catch(() => {})` as the MUST-level mechanism pins a specific code form rather than the behaviour it is meant to guarantee.

The sibling rule for the same class of abandoned promises — the swallowing-handler rule in `cancellation.md` ("Race semantics — swallowing-handler attachment on every abandonable Promise") — states the identical concern as a behavioural obligation: a late rejection is "silently absorbed and does not surface as a Node `unhandledRejection` process event," and it offers `.catch(() => {})` only as an illustrative example (it even notes the attachment-timing caveat that a lazily-registered `.catch(() => {})` will not catch an already-queued rejection). DISC-6 and `cancellation.md` therefore frame the same suppression requirement two different ways: one mandates the literal idiom, the other mandates the behaviour and demotes the idiom to an example.

The two are behaviourally equivalent in the happy path, so no runtime divergence results; the defect is that DISC-6's MUST over-constrains the implementation to one promise-suppression spelling and is inconsistent with the canonical behavioural framing the spec already uses for the same problem elsewhere.

## Solution Space

**Shape:** single

### Recommendation

Restate the DISC-6 silencing clause as a behavioural obligation and demote the idiom to a non-normative example, matching `cancellation.md`'s framing. Replace "the read's eventual settlement MUST be silenced with `.catch(() => {})` and its result MUST NOT be re-routed back into the discovery pass" with a form that requires: the abandoned read's eventual settlement MUST be suppressed so it raises no Node `unhandledRejection`, and its result MUST NOT re-enter the discovery pass — with `.catch(() => {})` offered as one acceptable realisation. Preserve the existing `details.kind = "package-read-timeout"` and the no-suppression ordering rule that follows; only the silencing-mechanism sentence changes. Edge case to keep visible: the suppression handler must be attached so a late rejection arriving after abandonment cannot escape (the attachment-timing caveat already documented in `cancellation.md`), so the behavioural restatement should not be read as permitting a handler registered too late.

## Consequence

**Severity:** advisory

An implementer reading DISC-6 literally is bound to a single suppression idiom even where an equivalent mechanism (an `AbortController`-derived handler, a shared swallowing handler attached at construction time, etc.) achieves the same observable result. The inconsistency with `cancellation.md`'s behavioural framing also risks an implementer applying a lazily-registered `.catch(() => {})` that misses an already-queued rejection — the exact failure mode `cancellation.md` warns against. No working system is precluded, so this is guidance-level.
