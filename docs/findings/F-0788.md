## Finding

The Appendix bullet for *Governance* in `docs/spec_topics/session-model-and-appendix.md` enumerates the active and retired REQ-IDs inline:

> the rules (GOV-1, GOV-3 through GOV-9, GOV-15, GOV-16, GOV-30, GOV-31; GOV-2, GOV-10, GOV-11, GOV-12, GOV-13, GOV-14 retired) that govern REQ-ID coining, anchoring, and retirement.

The canonical Retired REQ-IDs table at `docs/spec_topics/governance/anchor-scheme-and-retired.md#retired-req-ids` carries **eight** rows: GOV-2, GOV-10, GOV-11, GOV-13, GOV-4, GOV-21, GOV-12, GOV-14. The Appendix bullet enumerates only six retired IDs — it omits GOV-4 and GOV-21 — and the active-range shorthand "GOV-3 through GOV-9" positively asserts GOV-4 is live, directly contradicting the Retired table row that retires GOV-4 in commit `1d773d6` (replaced-by GOV-24).

The two omissions are not symmetric editorial slippage: each corresponds to a retirement that landed without a matching update to this Appendix mirror, so the bullet has been progressively drifting from the canonical table on each retirement commit. The same drifted enumeration also appears in the `governance.md` hub paragraph (tracked as a separate finding); resolving them independently keeps each fix scoped to one file.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `1d773d6` (2026-06-04, "GOV-3/GOV-6 subdirectory scan scope and prefix-table hub-stub bindings" — retired GOV-4, replaced-by GOV-24); `8e3ccb4` (2026-06-05, "GOV-21 bundles five independently testable sub-clauses" — retired GOV-21, replaced-by GOV-25…GOV-29).
**History:** `git log -S` on the Appendix bullet shows it has been updated exactly twice since the file came to rest at `docs/spec_topics/session-model-and-appendix.md`: `fe8aa5c` (2026-06-05) folded GOV-30/GOV-31 into active and moved GOV-12 to retired, and `5a772e3` (2026-06-05) moved GOV-14 to retired. Neither of the two intervening retirement commits — `1d773d6` for GOV-4 and `8e3ccb4` for GOV-21 — touched this bullet (or its `governance.md` hub twin), even though both updated `governance/anchor-scheme-and-retired.md`. The defect is therefore the interaction between two independent retirement PRs each failing to ripple into this Appendix mirror; no single commit "introduced" the inconsistency, and no commit has yet reconciled it. The duplicated mirror site on `governance.md` accumulated the same drift on the same commits, which is why both pages now read the same wrong enumeration.

## Solution Space

**Shape:** multiple

### Option A — Replace the inline enumeration with a pointer to the canonical table

**Approach.** Rewrite the *Governance* bullet so it names only the rule family, links to `governance.md` for the hub overview, and points at `anchor-scheme-and-retired.md#retired-req-ids` for the authoritative retired set. Concretely, drop the inline parenthetical "(GOV-1, GOV-3 through GOV-9, GOV-15, GOV-16, GOV-30, GOV-31; GOV-2, …, GOV-14 retired)" and leave a single forward-link.

**Spec edits.** One bullet rewrite in `session-model-and-appendix.md` Appendix.

**Pros.** Removes a recurring drift surface; future retirements need not be mirrored here at all; aligns with the *single canonical home* discipline GOV-30 already applies to the lock-step convention.

**Cons.** Reader loses the at-a-glance scan of which GOV-N IDs exist while standing in the Appendix; one extra click to enumerate them.

**Risks.** None — pointer-only edits cannot themselves drift.

### Option B — Re-enumerate to match the canonical table now

**Approach.** Rewrite the parenthetical to read:
"(GOV-1, GOV-3, GOV-5 through GOV-9, GOV-15, GOV-16, GOV-30, GOV-31; GOV-2, GOV-4, GOV-10, GOV-11, GOV-12, GOV-13, GOV-14, GOV-21 retired)"
— i.e. drop GOV-4 from the active range (now "GOV-3, GOV-5 through GOV-9"), and add GOV-4 and GOV-21 to the retired list, keeping the `(… retired — see [Retired REQ-IDs](…))` cross-link form the `governance.md` hub already uses.

**Spec edits.** One bullet rewrite in `session-model-and-appendix.md` Appendix.

**Pros.** Preserves the inline scan; matches the cadence on the `governance.md` hub paragraph if its sibling finding is resolved the same way.

**Cons.** Recreates the drift surface — every future retirement (and every newly-coined active GOV-N) must remember to update this mirror site, and the multi-commit history above shows that two of the four most recent retirement PRs already failed that obligation.

**Risks.** Identical drift will re-accumulate on the next un-mirrored retirement unless an aggregator lock-step rule (GOV-30 style) is added to bind this enumeration to the canonical table — which is out of scope here.

### Recommendation

Take **Option A** — replace the inline enumeration with a pointer. The multi-commit-interaction history makes the structural argument: this mirror site has already silently drifted twice in fourteen days under unrelated retirement PRs, and Option B reseats the same drift trap. The Appendix's role is wayfinding, not enumeration, so collapsing it to a forward-link is consistent with how the rest of the Appendix bullets treat the Glossary, Grammar, and Related Work pages (one-line description + link, no inline content mirror).

Edge cases for the implementer: keep the cross-link form readers expect (`[Retired REQ-IDs](./governance/anchor-scheme-and-retired.md#retired-req-ids)`); leave the bullet's "spec-corpus governance, not implementation spec" framing intact; do not touch `governance.md`'s sibling paragraph in the same edit (its fix is owned by the related finding below and may legitimately pick Option B on different grounds since it is the hub, not a mirror).

## Consequence

**Severity:** advisory

A reader navigating from the Session Model appendix to *Governance* takes away that GOV-4 is a live rule and that the retired set has six members. Cross-checking against the canonical table corrects the impression, but until then any reference, citation, or downstream tooling that treats the Appendix as authoritative will mis-classify GOV-4 (currently retired and replaced by GOV-24) and miss GOV-21 entirely. No runtime behaviour diverges.
