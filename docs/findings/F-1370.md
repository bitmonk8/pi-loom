## Problem

`V14b`'s **Adds** field carries a multi-sentence disclosure that classifies the parallel-tool-mode batch shape as an uncontracted real-Pi "consumption posture" and then routes its detection to editorial review per version-bump checklist item **(an)**, mirroring how SLSH-2's streaming-ordering routes to item (ak). This is risk classification plus a release-engineering process-routing rule, not a description of what the leaf ships; per `conventions.md` §Leaf format, **Adds** is descriptive and names what the implementation leaf introduces. A version-bump routing rule is cross-cutting governance a release engineer must be able to discover and maintain alongside the other bump-checklist routings, yet it is buried inside one implementation leaf's prose. Sibling leaves `V12a` and `V16a` keep their host-behaviour dependencies as fidelity-contract references and cite the cross-cutting home rather than defining policy inline, making `V14b` the outlier.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 2123dc9 — pi-loom spec-plan: fix F-0560 — surface V14b parallel-tool-mode presupposition via new bump checklist item (an) (2026-06-21, Thomas Andersen)
**History:** The "unpinned host presupposition" paragraph was appended to `V14b`'s Adds in commit 2123dc9. Before that commit `V14b`'s Adds was a single descriptive sentence about settle-all batch handling; the same commit added the routing target (bump checklist item (an)) to `version-bump-step2.md` and the corresponding anchors to `tool-calls.md`. The placement defect entered together with the F-0560 fix that surfaced the presupposition.

## Solution approach

Add a cross-cutting "host presuppositions" rule to `conventions.md` §Cross-cutting rules that defines an unpinned host presupposition (an uncontracted real-Pi consumption posture a leaf models) and routes its detection to editorial review via the version-bump checklist, citing `version-bump-step2.md` as the routing destination and naming the existing parallel cases (the `V14b` parallel-tool-batch posture → item (an); the SLSH-2 streaming-ordering posture → item (ak)). Narrow `V14b`'s **Adds** back to its delivery description (the settle-all-before-next-turn and independent-per-sibling-lowering batch handling, retaining the existing `cka-13` / `tool-calls.md#concurrency` citations) plus a one-line callout that the batch shape is an unpinned host presupposition per that `conventions.md` rule, carrying the `version-bump-step2.md#bump-checklist-parallel-tool-batch-delivery` and `tool-calls.md#parallel-tool-batch-delivery-presupposition` anchors on the callout.

## Solution constraints

- The spec is read-only for this fix — `version-bump-step2.md` item (an) already exists; only the plan-side prose moves.
- Honour the leaf-ID scheme and `no-invented-ids` policy: cite only `V14b`, `V12a`, `H4a` as they already appear.
