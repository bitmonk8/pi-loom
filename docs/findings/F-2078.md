## Problem

The *Green-path emission contract* (anchor `audit-non-empty-scan-canary`, *Non-empty-scan canary*, in `audit-wire-and-canary.md`) requires the canary record's `proposed-resolution` field to carry the two scan counters as the pair set `walked=<N> recognised=<M>` or an implementation-chosen equivalent whose pair names identify the same counters "in that order". The trailing "in that order" is ambiguous between a serialization-order constraint (the walked-counter pair MUST appear before the recognised-counter pair) and a parenthetical identifying clause that merely names the counters in introduction order and imposes no emission-order constraint. A CI parser that consumes the values positionally requires the first reading and would silently mis-map the counters against an implementation that emits them in reverse order under the second reading. The same clause is explicit elsewhere when it pins a fixed order ("in this fixed order" for the four record fields), so the loose phrasing here reads as an oversight rather than a deliberate pin.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** e5cd514 — pi-loom spec: resolve "T35 SDK capability inventory closed-set negative-direction audit" (2026-05-09, spec author)
**History:** The *Green-path emission contract* clause, including the `walked=<N> recognised=<M>` pair set and the trailing "in that order", was introduced fully formed in e5cd514, the commit that first authored the SDK surface-inventory audit's canary section. The ambiguous phrasing has been carried verbatim since that introduction; later commits (the `spec/` → `spec_topics/` collapse 81bf52a and the size-cap pass f5e89f4) only relocated and reworded surrounding context and did not introduce the defect.

## Solution approach

Clarify the trailing "in that order" in the *Green-path emission contract* clause so that the two counters are identified by their pair names (`walked` / `recognised`, or the implementation-chosen equivalent names), not by ordinal position, and so their relative emission order within the `proposed-resolution` field is not significant. Keep `walked=<N> recognised=<M>` as an illustrative example rather than an order-normative one.
