## Problem

The *Runtime construction obligation* clause in `diagnostic-shape.md` (under anchor `id="session-shutdown-details-conventions"`) states the runtime MUST construct `details.event` as a fresh object with each pinned field assigned by an explicit static property name, banning `...event` spread, `Object.assign(..., event, ...)`, and `Object.keys(event)` copy loops. This pins *how* the runtime assembles the object rather than the observable wire result. Under the loom 1.0 SDK pin `SessionShutdownEvent` carries only `reason`, so every banned mechanism and the mandated explicit-assignment style produce byte-identical wire output and no type-valid V1 conformance test can distinguish them; the prohibition only bites against a hypothetical future SDK sibling, a case the page already gates behind a required spec edit. The clause also reaches past the page's self-described scope ("only the wire-format nesting convention") into runtime-handler-internal construction style.

## Solution approach

Demote the construction-method enumeration ("no `...event` spread, no `Object.assign`, no `Object.keys` loop, … explicit static property name") in the *Runtime construction obligation* clause from a normative MUST to a non-normative implementation note, framed as forward-compat rationale. Retain the observable wire-shape obligation — `details.event` is exactly `{reason}` for `loom/host/session-shutdown-runtime-degraded` and exactly `{reason, loom, invocation_id}` for `loom/runtime/cancelled-by-session-shutdown`, with no `SessionShutdownEvent`-derived sibling beyond `reason` on the wire — as the sole normative pin. Co-locate the demoted note with the runtime `session_shutdown` handler material, optionally cross-referencing [Pi Integration Contract — Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation), which already owns the *Construction-site wrap* construction of the same payload.
