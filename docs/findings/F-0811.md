## Finding

The `⊑` relation in `type-system.md` is explicitly closed for loom 1.0: a pair the parser cannot decide statically via TYPE-1…TYPE-10 is reported as a type mismatch. Walk the closed list against the spec's own documented "enum-as-alias" idiom — `schema Severity = "low" | "medium" | "high"`, used in both [`schemas.md` §Type-alias / union schema](../../docs/spec_topics/schemas.md) and `frontmatter` examples — and ask whether `"low" ⊑ Severity` holds:

- **TYPE-1** (reflexivity) requires identical types.
- **TYPE-3** (literal-to-primitive) gives `"low" ⊑ string`, not `"low" ⊑ Severity`.
- **TYPE-4** (variant-to-union) is scoped to *discriminated* unions — i.e. object unions with a string-literal discriminator field. A primitive- or string-literal-union alias is not a discriminated union (`schemas.md` says detection runs only on unions whose variants are all object schemas), so its member literals don't qualify as variants.
- **TYPE-5/6** (union widening/distribution) operate on the union type itself; `Severity` is a *named alias to* that union, not the anonymous union, and the rules don't unfold the alias.
- **TYPE-10** (nominal named schemas) explicitly scopes itself to "a `NamedType` whose declaration is an object schema `schema X { ... }`", saying nothing about alias schemas, and asserts that named-schema participation in `⊑` runs only through TYPE-1/TYPE-4/TYPE-5/TYPE-6 — none of which fire here.

So `"low" ⊑ Severity` is decided by none of the closed cases, the parser must therefore report `T₁ ⋢ T₂`, and the closed-for-loom-1.0 wording forbids reading transparency in. Two conforming implementations are then forced to diverge — one folds the alias and admits the literal, the other refuses it nominally — on the primary idiom the schema-declarations page documents for inline enumerations. The same hole applies to every other `schema X = ...` alias form (`schema StringOrNumber = string | number` etc.): there is no rule that lets a value of static type `string` flow into a `StringOrNumber`-typed position even though `string` is an arm of the aliased union.

The hole is exactly between TYPE-10 (nominal for object-schema named types) and TYPE-5/TYPE-6 (structural for anonymous unions); alias-form `NamedType`s fall through both.

## Solution Space

**Shape:** multiple

### Option A — Transparent alias-fold rule (TYPE-11)

**Approach.** Add one rule to the closed table: a `NamedType` whose declaration is an alias schema `schema X = R` participates in `⊑` by unfolding to its RHS — `T ⊑ X` iff `T ⊑ R`, and `X ⊑ T` iff `R ⊑ T`. Recurse if `R` itself names another alias (cycle detection already exists per `type-alias-cycle`). Object-schema named types remain nominal under TYPE-10.

**Spec edits.**
- `type-system.md`: insert TYPE-11 row in the closed table, anchored `#type-11`; cross-reference from TYPE-10's "participates in `⊑` only through …" enumeration so the nominal paragraph reads "object-schema named types participate only through TYPE-1/4/5/6; alias-schema named types participate through TYPE-11 plus the rules its RHS triggers."
- `type-system.md` **Operational definition.** paragraph: note that AJV admits the alias-folded pair (lowering an alias just emits its RHS), so this rule keeps the parser's admit set strictly inside AJV's admit set as the operational definition requires.
- No grammar or construction-syntax changes; no diagnostic-registry changes (the existing `let-rhs-type-mismatch` / `invoke-arg-type-mismatch` / `fn-arg-type-mismatch` codes continue to fire on a `T ⋢ R` failure, now reported against the alias-typed site).

**Pros.**
- Preserves the documented `schema Severity = "low" | "medium" | "high"` idiom verbatim; no source-level edits needed in author code or in spec examples.
- Single-rule edit; no new diagnostic codes; no construction-syntax surface to design.
- Matches the AJV lowering's behaviour (the lowered schema for an alias *is* the RHS lowering), so the parse-time admit set stays a subset of the AJV admit set, consistent with the Operational definition.

**Cons.**
- Introduces a second flavour of named type into `⊑` (nominal for object schemas, transparent for aliases). The split must be re-asserted everywhere TYPE-10's "Named schemas are nominal" sentence currently reads as universal.

**Risks.**
- A future change that lets an alias resolve to an object schema (`schema X = Author`) would need a tie-breaker against TYPE-10. The natural rule — alias-fold wins because the alias's *declaration* is `=`, not `{ … }` — should be stated explicitly in TYPE-11 to forestall a re-litigation.

### Option B — Nominal alias schemas with an explicit construction form

**Approach.** Keep alias-schema NamedTypes nominal (consistent with TYPE-10's posture). Add an explicit construction syntax — e.g. `Severity("low")` or `Severity.from("low")` — that ascribes an alias type to a value whose static type already satisfies the alias's RHS, mirroring how TYPE-10 directs authors to "assign through an explicit construction or a typed `let` that names the target schema" for object schemas.

**Spec edits.**
- `type-system.md`: extend TYPE-10 to enumerate alias-schema NamedTypes alongside object-schema NamedTypes ("Named schemas — both object and alias — are nominal"); add a construction-form rule (TYPE-11) admitting `X(expr)` (or whatever surface is chosen) iff `T_expr ⊑ R`.
- `grammar.md`: add the construction-call production to the expression grammar and to `NamedType`'s usage rules; pick a syntax that does not collide with `fn` calls (an `Identifier "(" Expr ")"` form is already a function call).
- `schemas.md`: update the `schema Severity = "low" | "medium" | "high"` example to show the construction site (`let s: Severity = Severity("low")`); update the discriminated-union prose to clarify the new construction admission.
- `frontmatter/frontmatter-fields-a.md`: rewrite alias-typed defaults (`severity: Severity = Severity.Medium`-style entries) to use the construction form, or migrate them to `enum`.
- `diagnostics/code-registry-parse.md`: add a `loom/parse/alias-construction-type-mismatch` row (or document that the existing `*-type-mismatch` codes cover the inner expression) and a `loom/parse/alias-construction-on-non-alias` row for misuse.

**Pros.**
- Uniform nominal posture across every named schema; one rule to teach.
- Construction is observable in source, so an `invoke<Severity>(...)` boundary is unambiguously author-intended.

**Cons.**
- Breaks every existing example that assumes the alias is transparent (frontmatter defaults, schema-declarations examples); every author who writes the documented idiom must learn and adopt the construction form.
- Requires designing a construction surface that doesn't collide with function calls or with the `Enum.Variant` form already used for `enum` types.
- Larger spec surface: grammar, schemas, frontmatter, and diagnostics pages all gain new prose.

**Risks.**
- The construction form must round-trip through the binder's `Loom literal sublanguage` (`frontmatter/frontmatter-fields-a.md` `params:` defaults must accept it); the binder's echo-policy renderer (`binder/defaulting-system-note-echo.md`) likewise needs a rendering rule. Forgetting either site reintroduces the same gap on the frontmatter side.

### Recommendation

Adopt **Option A** (transparent alias-fold, TYPE-11). Resolve the smaller, scope-bounding fix first — it lands as one table row plus a clarifying sentence in TYPE-10, leaves every author-facing example untouched, and keeps the parse-time admit set provably inside the AJV admit set the Operational definition pins. The construction-form option's only durable advantage (uniformly nominal teaching surface) does not outweigh breaking the documented enum-as-alias idiom and the `frontmatter` default examples that depend on it. When writing TYPE-11, state the alias-vs-object tie-breaker explicitly (the alias is identified by `schema X = R`'s `=`, not by what `R` resolves to) so a later edit that aliases an object schema cannot reopen the TYPE-10 conflict; also note that TYPE-11 composes with the existing `type-alias-cycle` detector (cycle detection runs before lowering, so the unfold terminates).

## Consequence

**Severity:** correctness

Two compliant implementations decide `"low" ⊑ Severity` opposite ways: one folds the alias (the literal flows into an alias-typed `let`, `params:` default, function argument, or `invoke<Severity>` return); the other reports `T₁ ⋢ T₂` on every such site. Authors writing the documented `schema Severity = "low" | "medium" | "high"` idiom against the second implementation cannot construct any alias-typed value short of going through the inline union, defeating the reason the alias was introduced. Frontmatter examples like `severity: Severity = Severity.Medium` would also break under the nominal reading unless `Severity` is upgraded to an `enum` (a different construct with different runtime semantics).
