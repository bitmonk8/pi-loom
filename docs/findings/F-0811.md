## Problem

The closed `⊑` compatibility list (TYPE-1…TYPE-10) in `type-system.md` decides no case for `"low" ⊑ Severity` where `Severity` is an alias schema `schema X = R` (e.g. `schema Severity = "low" | "medium" | "high"` from `schemas.md` §Type-alias / union schema). TYPE-10 scopes nominal participation to object-schema named types and routes them only through TYPE-1/4/5/6; the union rules (TYPE-5/TYPE-6) operate on the anonymous union, not on the named alias, and never unfold the alias. Because the list is closed for loom 1.0, an alias-typed position must report `T₁ ⋢ T₂`, so two conforming implementations diverge — one folds the alias and admits the value, the other refuses it — on the documented enum-as-alias idiom. The same hole applies to every `schema X = ...` alias form (e.g. `schema StringOrNumber = string | number`).

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `0226b80` ("pi-loom plan: resolve "V15c references an undefined compatibility relation"", 2026-05-05) introduced the type-compatibility section and its closed-for-V1 enumeration with rules 1–8; `9a4e078` ("pi-loom spec: resolve "TYPE-8 field-wise compatibility scope ambiguous"", 2026-06-07) added TYPE-10's nominal-named-schemas paragraph and re-scoped TYPE-8 to inline object types.
**History:** `git log -S "T₁ ⊑ T₂" -- docs/spec_topics/type-system.md` and the earlier path `spec_topics/type-system.md` shows the closed table arrived in `0226b80` containing only rules 1–8; subsequent commits `3343890` (ID-coded rule numbering), `648365c` (TYPE-9 per-site diagnostics), and `9a4e078` (TYPE-10 nominal-named-schemas + TYPE-8 inline-only scope) each grew the table but never added a rule covering named alias schemas. The `schema X = ...` alias form itself predates the table by months (the `schema Severity = "low" | "medium" | "high"` example is in `schemas.md` from `53f5831` / `fecb504`, well before `0226b80`), so the gap was present the moment the closed-list `⊑` table was introduced and was preserved verbatim through every subsequent edit. TYPE-10's later object-schema scoping makes the gap more visible but is not what created it.

## Solution approach

Add a compatibility rule to the closed structural-cases table in `type-system.md` that makes an alias-schema `NamedType` (`schema X = R`) transparent in `⊑` by unfolding to its RHS, recursing through nested aliases. Rewrite TYPE-10's `Named schemas are nominal` statement and its participates-only-through enumeration to scope to object-schema named types, cross-referencing the new alias rule. Identify the alias by the `schema X = R` `=` form, not by what `R` resolves to, so aliasing an object schema does not reopen TYPE-10's nominal case.

## Solution constraints

- Object-schema named types MUST remain nominal under TYPE-10; the new rule applies only to alias-schema named types.
- The rule MUST keep the parser's static admit set within AJV's admit set, per the **Operational definition.** paragraph in `type-system.md`.
