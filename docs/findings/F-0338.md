## Problem

`H5a` (closing-gate automation) declares `**Deps.** H4a`, which transitively serialises it behind `H2a`, `H3a`, and `H4a` via the horizontal chain. But `H5a` consumes none of their outputs: its static-reconciliation CI gate runs under `npm test` over docs and a dedicated seeded test-fixtures root, and its only real prerequisite is the test-runner scaffold `H1a` establishes. The `H4a` edge is therefore a false sequencing constraint — it does not break executability but needlessly blocks `H5a` from being picked up until `H4a` (and transitively `H2a`/`H3a`) completes, losing build-order parallelism.

The over-constraining edge is original to the leaf, not introduced by any later edit.

## Solution approach

In `docs/plan_topics/H5a-closing-gate-automation.md`, re-point the `**Deps.**` field from `H4a` to `H1a`. `H1a` is `H5a`'s true floor — it provides the `npm test` runner the gate hooks into, and `H5a` consumes nothing from `H2a`, `H3a`, or `H4a`.
