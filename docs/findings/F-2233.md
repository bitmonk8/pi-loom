## Problem

The diagnostic code `loom/runtime/reload-teardown-timeout` in `code-registry-runtime.md` is named as though it is gated on a reload-triggered shutdown, but it fires whenever the `session_shutdown` handler's bounded await on every in-flight `disposeBarrier` exhausts the `SHUTDOWN_AWAIT_CAP_MS` budget — which runs reason-agnostically across `"quit"`, `"reload"`, `"new"`, `"resume"`, `"fork"`, and the unknown-reason `string` arm. The row's *Trigger* cell specifies the reason-agnostic firing condition correctly, so behaviour is fully specified; the defect is that an operator or test author reading only the code identifier or rendered *Message* is led to believe the diagnostic is conditioned on `event.reason === "reload"`. It is the lone misnamed registry row lacking the in-row self-disclosure clause the sibling `loom/load/cross-format-collision` row already carries.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commit:** `5e8b499` (2026-05-06, *pi-loom spec: resolve "Hot-reload teardown contract for the loom extension instance is unspecified"*)
**History:** `git log -S 'reload-teardown-timeout' --all` over the spec corpus shows the token first appears in `5e8b499`. `git show 5e8b499` confirms that same commit introduced both the `loom/runtime/reload-teardown-timeout` registry row (with the `reload-` token and the `reload teardown timed out …` *Message* template) and, on `spec_topics/pi-integration-contract.md`, the reason-agnostic framing that the `session_shutdown` handler "runs in fixed order regardless of `event.reason`" and "treats every reason identically" — explicitly enumerating `/reload`, `/new`, fork, and quit as the reasons it must cover. The name/trigger mismatch was therefore present at the row's first appearance, not introduced by a later widening of the trigger. Subsequent commits (`9559e19` 2026-05-07 unknown-reason rule; `52b9101` 2026-05-11 fork-reason watcher closure) widened the reason enumeration around the code but did not edit the row name or message and did not introduce the mismatch.

## Solution approach

Clarify the `loom/runtime/reload-teardown-timeout` row's *Trigger* cell with a self-disclosure clause mirroring the `loom/load/cross-format-collision` precedent: state that the code retains the historical `reload-` token as a stable diagnostics-contract identifier per [DIAG-3](diagnostic-shape.md#diag-3) and that its scope is the full `session_shutdown` bounded-await timeout across every `event.reason` arm, not only `reload`. Note that the *Message* literal is preserved verbatim under [DIAG-4](diagnostic-shape.md#diag-4) and that operators cannot infer the firing `event.reason` from the rendered message.

## Solution constraints

- Out of scope: the *Code* cell and the *Message* template — both frozen by DIAG-3 / DIAG-4; only the *Trigger* cell may change.
