## Finding

The Notes section at the bottom of `docs/spec_topics/errors-and-results/queryerror-variants.md` carries an HTML comment recording a past structural edit:

```html
<!-- ERR-14 (`ValidationIssue` ordering) relocated to ### Query-time variants, co-located with the ValidationError schema, for discoverability. Anchors carried verbatim. -->
```

This is an editorial breadcrumb — a decision-log note explaining *that* the ERR-14 block was moved out of Notes and into the Query-time variants subsection, and *why*. It conveys no normative content. ERR-14 now lives at its destination site (just below the `ValidationError` schema, carrying its `#err-14` and `#validation-issue-ordering` anchors), so the relocation is complete and self-evident from the current document structure. The comment is invisible in rendered output but persists in source, where it adds reading noise for any author or auditor working the file and will only grow more confusing as the document drifts further from the moment of the move.

## Issue introduction
**Verdict:** introduced-by-commit
**Introducing commits:** c3ea1d1 ("pi-loom spec: resolve \"ERR-14 ValidationIssue ordering buried in Notes\"")
**History:** `git log -S 'relocated to ### Query-time variants' -- docs/spec_topics/errors-and-results/queryerror-variants.md` returns exactly one commit, c3ea1d1, confirmed via `-G` as well. `git show c3ea1d1` on the file shows ERR-14 added in full at the Query-time variants site and, in the same hunk, the original ERR-14 block in Notes replaced by this HTML comment. The breadcrumb was therefore created as the by-product of that single relocation edit; it has not been touched since. The defect is not present-since-inception — it entered with the relocation commit and is the leftover marker that commit chose to leave behind.

## Solution Space

**Shape:** single

### Recommendation

Delete the HTML comment outright. The relocation it documents is complete and the destination block (with its `#err-14` / `#validation-issue-ordering` anchors) stands on its own; nothing references the comment and removing it changes no rendered output and no normative content. No replacement text is needed.

## Consequence

**Severity:** cosmetic

No implementer or runtime observer is affected — the comment does not render and constrains nothing. It is pure source-organisation residue that adds reading friction and risks future confusion as the surrounding structure evolves.
