## Finding

The corpus canonical term for the typed-query schema-repair mechanism is **`respond-repair`** (hyphenated in prose, `respond_repair` as a YAML key). `glossary.md` defines exactly one entry — `respond-repair` — and explicitly states the prose/YAML convention. No glossary entry, no anchor, and no other normative page defines `response-repair`; the token has no contract.

Two normative locations still use the undefined synonym `response-repair`, both as a leftover from the rename pass that replaced `coercion:` with `respond_repair:` (commit 708a42f, see Issue introduction):

- `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md` line 57 (the `respond_repair` bullet): "The block configures **respond-repair** — the response-repair mechanism specified in [Query — Schema-validation respond-repair](../query.md)." The sentence introduces `respond-repair` and then defines it in terms of `response-repair`, contradicting itself in the same clause.
- `docs/spec_topics/glossary.md` line 21 (the `respond-repair` entry): "Loom performs no implicit type conversion in expressions; the response-repair loop above is the only 'repair' mechanism the spec defines." The entry whose heading establishes `respond-repair` then refers back to itself with the wrong spelling.

Both occurrences are pure orthographic drift — there is no second mechanism being named — but they read as if the spec defines two distinct concepts.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commits:** `708a42f` ("pi-loom spec: resolve 'coercion overloaded: schema-validation repair vs. type conversion'", 2026-05-06)
**History:** The frontmatter bullet was first authored in `13cf76d` (2026-05-05, "spec: rename retry: -> coercion: frontmatter key") as `"The block configures **coercion** — the response-repair mechanism specified in [Query — Schema-validation coercion](./query.md)"`. At that point `coercion` was the canonical term and `response-repair` was an informal descriptor for what the field configured — internally consistent. Commit `708a42f` then renamed `coercion` to `respond_repair`/`respond-repair` corpus-wide (`spec_topics/frontmatter.md`, `spec_topics/glossary.md`, and 21 other files in that single commit), but the substitution pass left the two `response-repair` tokens unchanged. After this commit, `respond-repair` and `response-repair` co-exist in the same sentences with no defining entry for the latter. Commit `f5e89f4` (2026-06-04) subsequently split the monolithic `frontmatter.md` into `frontmatter/frontmatter-fields-a.md` + `frontmatter/frontmatter-fields-b-and-templates.md`, carrying the defect forward verbatim into its current location. The glossary entry was edited again in `d54b426` ("Glossary cluster: six independent defects") which fixed the heading from `respond_repair` to `respond-repair` but did not notice the trailing `response-repair` token in the entry body.

## Solution Space

**Shape:** single

### Recommendation

Replace `response-repair` with `respond-repair` at both sites:

- `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md` line 57: change `"The block configures **respond-repair** — the response-repair mechanism specified in [Query — Schema-validation respond-repair](../query.md)"` to `"The block configures **respond-repair** — the schema-validation repair mechanism specified in [Query — Schema-validation respond-repair](../query.md)"`. (Dropping the redundant `respond-repair` from the descriptor — the bold `respond-repair` already names the mechanism — avoids the awkward `"the respond-repair … respond-repair"` repetition. A literal `s/response-repair/respond-repair/` is also acceptable; the repetition is ugly but accurate.)
- `docs/spec_topics/glossary.md` line 21: change `"the response-repair loop above is the only 'repair' mechanism the spec defines"` to `"the respond-repair loop above is the only 'repair' mechanism the spec defines"`.

No anchors, REQ-IDs, diagnostic codes, or cross-references move. Verify afterwards that `grep -rn 'response-repair' docs/` returns no hits.

## Consequence

**Severity:** cosmetic

An implementer reading either sentence in isolation may briefly believe the spec distinguishes a `response-repair` concept from `respond-repair`; cross-referencing the glossary resolves the confusion. Nothing about runtime behaviour, diagnostic codes, or schema validity is affected.
