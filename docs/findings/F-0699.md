## Finding

The corpus canonical term for the typed-query schema-repair mechanism is **`respond-repair`** (hyphenated in prose, `respond_repair` as a YAML key). `docs/spec_topics/glossary.md` defines exactly one entry for the concept — `respond-repair` — and explicitly pins the prose/YAML spelling convention. No glossary entry, anchor, or other normative page defines `response-repair`; the token has no contract.

Two normative sentences still use the undefined synonym `response-repair`, both as leftovers from the corpus-wide rename that replaced `coercion:` with `respond_repair:`:

- `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md` line 57 (the `respond_repair` bullet): *"The block configures **respond-repair** — the response-repair mechanism specified in [Query — Schema-validation respond-repair](../query.md)."* The sentence bolds `respond-repair` as the defined term, then defines it in terms of `response-repair`, contradicting itself in a single clause.
- `docs/spec_topics/glossary.md` line 21 (the `respond-repair` glossary entry itself): *"Loom performs no implicit type conversion in expressions; the response-repair loop above is the only 'repair' mechanism the spec defines."* The entry whose heading establishes `respond-repair` refers back to itself with the wrong spelling.

Both occurrences are pure orthographic drift; there is no second mechanism being named. They read, however, as if the spec distinguishes two concepts.

Commit `f5e89f4` (2026-06-04) subsequently split the monolithic `frontmatter.md` into `frontmatter/frontmatter-fields-a.md` + `frontmatter/frontmatter-fields-b-and-templates.md`, carrying the defect forward verbatim into its current location. The glossary entry was edited again in `d54b426` ("Glossary cluster: six independent defects") which fixed the heading from `respond_repair` to `respond-repair` but did not notice the trailing `response-repair` token in the entry body.

## Solution Space

**Shape:** single

### Recommendation

Replace `response-repair` with `respond-repair` at both sites:

- `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md` line 57 — rewrite the offending clause as *"The block configures **respond-repair** — the schema-validation repair mechanism specified in [Query — Schema-validation respond-repair](../query.md)."* Dropping the redundant noun (rather than literal `s/response-repair/respond-repair/`) avoids the awkward "the respond-repair … respond-repair" repetition; the bolded `respond-repair` already names the mechanism, so the descriptive phrase only needs to say *what kind* of mechanism it is. A literal substitution is acceptable as a fallback if the rephrase is disliked.
- `docs/spec_topics/glossary.md` line 21 — change *"the response-repair loop above is the only 'repair' mechanism the spec defines"* to *"the respond-repair loop above is the only 'repair' mechanism the spec defines."*

No anchors, REQ-IDs, diagnostic codes, or cross-references move. After the edit, `grep -rn 'response-repair' docs/spec_topics/` MUST return zero hits (it is currently the only `docs/spec_topics/` consumer of the undefined token); the existing matches under `docs/findings/` are quoted defect descriptions and out of scope.

## Consequence

**Severity:** cosmetic

An implementer reading either sentence in isolation may briefly think the spec distinguishes a `response-repair` concept from `respond-repair`; cross-referencing the glossary resolves the confusion within seconds. No runtime behaviour, diagnostic code, schema validity, or REQ-ID is affected.
