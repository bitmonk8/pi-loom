## Finding

`H4a`'s **Adds** defines an "in-memory fixture-supply mechanism": the end-to-end harness supplies the fixture `.loom` content to the extension in memory so a harness-driven leaf's discovery reads from the harness-provided source "rather than the real filesystem — no ambient `src/**` filesystem read, no `FileSystem` seam (`V8b`) dependency, keeping the `H3a` ambient-access scan clean." The leaf names the mechanism and its invariants but never pins the architectural injection point: what entry the factory exposes in harness mode, and by what channel discovery is told to read the in-memory content instead of walking the filesystem.

This matters because the only route through which production discovery reads files is the `FileSystem` seam. `V10a` (discovery walk) reads exclusively through that seam (`FileSystem.homedir()`, `readdir`, `lstat`, etc.), and the `H3a` seam set contains no "discovery source" seam. So the mechanism asserts that fixture content reaches discovery while bypassing the only seam through which discovery reads files — with no stated alternative path. `M` and `M-T` (both `Deps: H4a`) consume this mechanism for their single-source happy-path discovery, and `H7a` drives a multi-feature fixture through the same harness against the integrated pipeline, also with no stated loading route.

Two reasonable implementers diverge. One supplies the fixture via an in-memory implementation of the `FileSystem` seam — directly contradicting the leaf's own "no `FileSystem` seam dependency / clean `H3a` ambient scan" claim. Another invents a harness-only discovery-source override (most likely hung off the `resources_discover` handler), which then risks diverging from the production seam-backed discovery path that `M` is meant to be the happy-path proof of.

That commit resolved an earlier finding that `M`'s happy-path discovery had no defined source under the ambient-access ban; it introduced the "in-memory fixture-supply mechanism" wording into `H4a` and the matching consumer notes in `M`/`M-T`, asserting the mechanism bypasses both ambient reads and the `FileSystem` seam, but never specified the injection point by which harness content reaches discovery. The present gap was introduced together with the mechanism in that single commit.

## Solution Space

**Shape:** single

### Recommendation

In `H4a`'s **Adds**, name the injection affordance the factory exposes in harness mode — a harness-supplied discovery source that the factory accepts and from which harness-driven discovery reads the fixture `.loom` content — and state that this source is distinct from the `H3a` seam set. State that harness-driven discovery reads its fixture content from this named affordance rather than the filesystem.

In `H7a`'s **Adds**, state that the integrated-run multi-feature fixture reaches the pipeline through this same `H4a` injection affordance.

`M` and `M-T` can then cite the named injection affordance in their existing happy-path discovery text without otherwise rewording it.

Constraints the fix must honour:
- The injection affordance must not be an in-memory implementation of the `FileSystem` seam (`V8b`); the leaf's "no `FileSystem` seam dependency / clean `H3a` ambient-access scan" invariant must remain intact.
- The spec is read-only for this fix — author the affordance entirely in the plan leaves.

## Consequence

**Severity:** correctness

The injection point is named but unpinned, so two implementers produce incompatible harnesses: one wires an in-memory `FileSystem`-seam fake (breaking the leaf's "no `FileSystem` seam dependency" invariant and dirtying the `H3a` ambient-access scan), the other invents a harness-only discovery override that diverges from the production seam path `M` is meant to validate. The divergence propagates to every consumer — `M`/`M-T`'s happy-path discovery and `H7a`'s terminal integration gate all read fixtures through whatever route the `H4a` implementer happened to choose.
