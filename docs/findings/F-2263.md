## Finding

Version-bump step 3 ("Re-confirm the `engines.node` floor") in `version-bump-step2b.md` pins a build-time test that asserts a three-way equality across (i) the loom `package.json#engines.node` literal, (ii) the in-repo pinned floor in the `pi-engines-node` row of `SDK_SURFACE_INVENTORY`, and (iii) the `engines.node` floor read **live** at build time from the installed `@earendil-works/pi-coding-agent` `package.json` under `node_modules`. Operand (iii) is the only live read.

The step specifies what to do when (iii) differs from (i)/(ii) — the upstream floor has moved, so the test reddens and the contributor re-pins (i) and (ii). It does not specify what (iii) evaluates to, or how the equality behaves, when the installed package's `engines.node` field is **absent**. A live read of a missing field yields `undefined` (or a missing key), and the spec gives no rule for comparing `undefined` against the two pinned string literals: the gate's outcome in that case is implementation-dependent rather than defined by the contract.

`engines.node` is not a mandatory `package.json` field — a future Pi minor could ship without it, or with a non-string shape. Under the current wording, one implementer's three-way equality would redden (treating the absent field as unequal to the pinned literal), while another's might throw, coerce, or silently treat two absent/undefined operands as "equal" and pass. The gate exists precisely to catch upstream floor drift at `npm test` time rather than at user runtime; an undefined-operand path that can silently pass defeats that purpose.

## Issue introduction
**Verdict:** introduced-by-commit
**Introducing commit:** `6798428` — *pi-loom spec: resolve "Step 3 engines.node test describes mutually exclusive right-hand operands"* (2026-06-05)
**History:** `version-bump-step2b.md` is git-tracked. Before `6798428`, step 3 asserted a **cross-package equality** between the loom `package.json#engines.node` literal and the upstream floor *sourced from the in-repo `pi-engines-node` row of `SDK_SURFACE_INVENTORY`* — i.e. it compared two in-repo literals, with no live read of the installed Pi `package.json`. Because nothing was read live from the installed dependency, the "installed package omits `engines.node`" assumption did not exist in that form. `6798428` replaced the two-operand cross-package compare with the current three-way equality and introduced operand (iii), "the `engines.node` floor read live at build time from the installed `@earendil-works/pi-coding-agent` `package.json`," without specifying its value when the field is absent — the point at which the silent assumption entered. (`git log -S 'read live at build time from the installed'` confirms `6798428` as the first commit carrying the live-read phrasing in this file; the later plan-side commit `c8564bb`, 2026-06-12, that provisioned the SDK into the build-test environment post-dates it and only operationalises the read.)

## Solution Space

**Shape:** single

### Recommendation

In step 3, state what operand (iii) yields when the installed `@earendil-works/pi-coding-agent` `package.json` lacks an `engines.node` field (or carries a non-string value), and pin the gate's behaviour: a missing or non-string operand (iii) MUST redden the test with a defined cause distinct from the ordinary "floor moved" red, rather than entering an implementation-dependent comparison against `undefined`. A missing upstream floor can never byte-equal the pinned floor literal, so the equality cannot hold; making that explicit keeps the gate failing loudly instead of relying on incidental coercion. Mirror the `package.json` read/parse failure handling already pinned in `capability-probe.md`'s *Self-failure* clause so the two `package.json`-reading surfaces treat anomalies consistently.

Edge cases the implementer must watch: distinguish "field absent" from "field present but empty/whitespace" and from "field present with a non-SemVer string"; ensure the distinct-cause red is itself deterministic and byte-stable across runs; and confirm the absent-field red does not get misread by a contributor as a normal floor-move that warrants re-pinning operands (i)/(ii).

## Consequence

**Severity:** correctness

Two reasonable implementers would diverge when a candidate Pi version omits `engines.node`: one reddens the gate, another silently passes by comparing `undefined` against `undefined` or by throwing. A silent pass lets an upstream floor anomaly slip through the bump gate that exists to catch exactly that drift. The defect is latent — the currently installed Pi declares `engines.node` (`">=22.19.0"`), so no live divergence exists today — but it becomes live on the first candidate version that drops or reshapes the field.
