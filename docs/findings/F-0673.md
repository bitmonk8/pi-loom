## Finding

`diagnostic-shape.md` makes the `Diagnostic` envelope's `file?` / `range?` fields optional only for a **closed set** of location-less codes — the four `loom/load/*` codes and the four `loom/host/session-shutdown-*` teardown codes. The paragraph is explicit that the set is closed in loom 1.0.0 and that "adding or removing a location-less code requires a spec edit to this paragraph (and, for an addition, the new code's registry row) before the omitted-fields form is permitted on the wire". For every other registered code the optional-marker is therefore a documentation convenience, not an authorisation to omit: the envelope effectively requires `file` and `range`, and the **Serialised content format** rule unconditionally prepends `<file>:<line>:<col>:` to the rendered line.

`loom/runtime/cancelled-by-session-shutdown` is registered in `code-registry-runtime.md` and is not in the closed location-less set. Its *Trigger* prose says it is emitted "from the invocation's own `finally` block (not from the teardown handler), exactly once per cleanly-cancelled invocation" after `loomAbort.abort()` fires on session swap. The cancellation is a lifecycle event delivered by the `session_shutdown` handler — there is no source span in the loom program to attribute it to, and the row defines no `file` / `range` source either. Implementers therefore have no normative answer for what `file` and `range` should carry on the wire and what the serialised content line's `<file>:<line>:<col>:` prefix should render: omit (violates the closed-set obligation), fabricate a sentinel (the envelope paragraph explicitly forbids "a sentinel `file` or a zero-valued `range`"), or pin to the cancelled invocation's `.loom` path with a synthetic range.

The defect is the symmetric twin of the sibling finding on `loom/runtime/reload-teardown-timeout` (immediately preceding in the source review): that code has the same problem and the same closed-set omission, but spans N invocations rather than one, so its resolution is unambiguously the location-less arm. This finding is the one-invocation variant where pinning to the cancelled invocation's `.loom` path is technically representable, so the resolution requires a choice rather than being forced.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `7f37dd7` (2026-05-07) — added the `loom/runtime/cancelled-by-session-shutdown` registry row; `b7d383a` (2026-06-06) — relaxed `file` / `range` to optional and introduced the closed location-less enumeration without including this code.
**History:** `git log -S "cancelled-by-session-shutdown" -- docs/` shows the code entered the registry in `7f37dd7` ("resolve Session-swap behaviour for in-flight loom invocations is under-specified"), at which point the `Diagnostic` envelope still required `file` / `range` for every code — so the row was already unsatisfiable, but the defect was hidden by the absence of a location-less enumeration to omit-from. `b7d383a` ("resolve Diagnostic envelope unsatisfiable for location-less codes") then created the closed location-less set, enumerating the four `loom/load/*` and four `loom/host/session-shutdown-*` codes but omitting both runtime-side teardown codes (`reload-teardown-timeout` and `cancelled-by-session-shutdown`); the same commit added the **Serialised content format** drop-the-prefix carve-out keyed on the same closed set. The combination locks the contradiction in place: the row remains in the registry, the envelope still requires `file` / `range` for any code outside the closed set, and the emission site still has no source span to provide. The defect is not attributable to either commit in isolation — `7f37dd7` introduced the unspecified-location row, `b7d383a` created the mechanism that could have absolved it but did not — and the durable fix targets that interaction (close the enumeration over both runtime teardown codes) rather than reverting either commit.

## Solution Space

**Shape:** multiple

### Option A — Add to the closed location-less set

**Approach.** Extend `diagnostic-shape.md`'s **Location-less diagnostics** paragraph to enumerate `loom/runtime/cancelled-by-session-shutdown` (and, per the sibling finding, `loom/runtime/reload-teardown-timeout`) alongside the existing eight codes. Reconcile the paragraph's "closed in loom 1.0.0" framing to cover the now-ten-code set.

**Spec edits.**
- `diagnostic-shape.md` **Location-less diagnostics** paragraph: enumerate the additional code; restate the count ("the four `loom/load/*` codes, the four `loom/host/session-shutdown-*` teardown codes, and the two `loom/runtime/*` session-shutdown-related codes").
- `code-registry-runtime.md`: no row-body edit required — the row continues to define no `file` / `range` source.
- The **Serialised content format** drop-the-prefix carve-out already keys on "the location-less codes enumerated under **Location-less diagnostics** above" and therefore picks up the addition automatically.

**Pros.**
- Mirrors the sibling resolution for `reload-teardown-timeout` exactly, keeping the two session-shutdown-related runtime codes on the same footing.
- The emission site genuinely has no source span; modelling that honestly avoids fabricated locations.
- One paragraph edit; no row-prose change; no test churn beyond loosening any prematurely-written file/range assertion at `V9g`.

**Cons.**
- The location-less set widens beyond its original `loom/load/*` + `loom/host/*` shape into `loom/runtime/*`, weakening the namespace-level invariant a reader might have inferred from the previous enumeration.

**Risks.**
- Operator-tooling consumers that hard-code the eight-code set as a constant need to pick up the addition; the closed-set future-addition obligation already says they must.

### Option B — Pin `file` to the cancelled invocation's `.loom` path

**Approach.** Define `file` as the cancelled invocation's `.loom` source path (resolvable from the `ActiveInvocationRegistry` entry the per-invocation `finally` already holds, via the `loom` canonical-key field plus the discovery-time path mapping) and `range` as a normative whole-file or `{1,1}`–`{1,1}` sentinel that the paragraph carves out explicitly. The carve-out would need to override the current "MUST NOT carry a sentinel `file` or a zero-valued `range`" prohibition for this row only.

**Spec edits.**
- `code-registry-runtime.md`: extend the `loom/runtime/cancelled-by-session-shutdown` *Trigger* prose to pin both `file` (cancelled invocation's `.loom` source path) and `range` (the chosen sentinel shape).
- `diagnostic-shape.md` **Location-less diagnostics** paragraph: add a "carved-out exception" sub-clause for this row that admits the sentinel `range` and the whole-file `file`.
- `diagnostic-shape.md` **Serialised content format** paragraph: confirm the standard `<file>:<line>:<col>:` prefix renders for this row (no carve-out needed).

**Pros.**
- Preserves the location-less set's current namespace-coherent shape (only `loom/load/*` and `loom/host/*`).
- The cancelled invocation's `.loom` path is information operators want anyway.

**Cons.**
- Requires carving an explicit exception into the "MUST NOT carry a sentinel" rule the envelope just established in `b7d383a` — a structural weakening of a recently-pinned invariant.
- Diverges from the sibling `reload-teardown-timeout` resolution, which has no single invocation to attribute and must take Option A; the two session-shutdown-related codes then resolve asymmetrically.
- The `range` is a fabrication regardless of which sentinel is chosen — the cancellation has no meaningful source position even within the chosen file.

**Risks.**
- Future runtime codes emitted from `finally` blocks on lifecycle events face the same decision and may inconsistently pick one arm or the other unless the spec also pins a general rule.

### Recommendation

**Option A.** Add `loom/runtime/cancelled-by-session-shutdown` (and, co-resolved with the sibling finding, `loom/runtime/reload-teardown-timeout`) to the closed location-less set. The emission has no source span by construction, and modelling that honestly via the existing location-less mechanism is symmetric with the sibling timeout code and avoids carving an exception into the just-pinned "no sentinel file / no zero-valued range" envelope rule.

Edge cases the implementer must watch:
- The **Serialised content format** paragraph's drop-the-prefix carve-out keys on the closed-set enumeration; the addition picks it up automatically and no second edit is required there.
- The `V9g` leaf's `Tests` line that asserts `loom/runtime/cancelled-by-session-shutdown` is "emitted per in-flight invocation" must not also assert `file` / `range` presence; if a prematurely-written test fixture pinned those fields, it needs updating.
- Operator-tooling consumers that strict-validate the envelope against a hard-coded eight-code location-less set need the addition; this is the closed-set future-addition obligation in action and is in-scope per [GOV-15 diagnostic-registry carve-out](../../../docs/spec_topics/governance/source-language-stability.md#diagnostic-registry-carve-out) as a registry-content addition within loom 1.x.
- The `details.event.{reason, loom, invocation_id}` payload the row already pins is unaffected — it lives under `details`, not under `file` / `range`, and continues to discriminate cause and per-invocation identity even though no source span is carried.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the wire shape: one omits `file` / `range` (relying on the optional marker, violating the closed-set obligation), one fabricates a zero-valued range or a sentinel path (explicitly forbidden by the envelope paragraph), and a third pins to the cancelled invocation's `.loom` path. The serialised content line's leading `<file>:<line>:<col>:` segment renders three different ways across those implementations, breaking byte-exact assertions on the persistent-diagnostic channel and any operator-tooling consumer that validates the envelope strictly.
