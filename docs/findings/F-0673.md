## Problem

`diagnostic-shape.md`'s **Location-less diagnostics** paragraph makes the `Diagnostic` envelope's `file?` / `range?` omitted-fields form lawful only for a closed set of codes — the four `loom/load/*` and four `loom/host/session-shutdown-*` codes — and forbids a sentinel `file` or zero-valued `range` for any code outside it. `loom/runtime/cancelled-by-session-shutdown` is registered in `code-registry-runtime.md` but is not in that closed set, yet it is a lifecycle event emitted from the cancelled invocation's `finally` block with no source span to attribute. Implementers therefore have no normative answer for what `file` / `range` carry on the wire and what the serialised content line's `<file>:<line>:<col>:` prefix renders: omitting violates the closed-set obligation, fabricating a location is explicitly forbidden, and pinning to the cancelled invocation's `.loom` path with a synthetic range is an unforced third choice. Two reasonable implementers diverge on the wire shape, breaking byte-exact assertions on the persistent-diagnostic channel and strict-validating operator-tooling consumers.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `7f37dd7` (2026-05-07) — added the `loom/runtime/cancelled-by-session-shutdown` registry row; `b7d383a` (2026-06-06) — relaxed `file` / `range` to optional and introduced the closed location-less enumeration without including this code.
**History:** `git log -S "cancelled-by-session-shutdown" -- docs/` shows the code entered the registry in `7f37dd7` ("resolve Session-swap behaviour for in-flight loom invocations is under-specified"), at which point the `Diagnostic` envelope still required `file` / `range` for every code — so the row was already unsatisfiable, but the defect was hidden by the absence of a location-less enumeration to omit-from. `b7d383a` ("resolve Diagnostic envelope unsatisfiable for location-less codes") then created the closed location-less set, enumerating the four `loom/load/*` and four `loom/host/session-shutdown-*` codes but omitting both runtime-side teardown codes (`reload-teardown-timeout` and `cancelled-by-session-shutdown`); the same commit added the **Serialised content format** drop-the-prefix carve-out keyed on the same closed set. The combination locks the contradiction in place: the row remains in the registry, the envelope still requires `file` / `range` for any code outside the closed set, and the emission site still has no source span to provide. The defect is not attributable to either commit in isolation — `7f37dd7` introduced the unspecified-location row, `b7d383a` created the mechanism that could have absolved it but did not — and the durable fix targets that interaction (close the enumeration over both runtime teardown codes) rather than reverting either commit.

## Solution approach

In `diagnostic-shape.md`'s **Location-less diagnostics** paragraph, extend the closed set to enumerate `loom/runtime/cancelled-by-session-shutdown` (alongside its sibling `loom/runtime/reload-teardown-timeout`), so the `file?` / `range?` omitted-fields form is authorised for it, and restate the set's count to cover the now-enumerated codes. Keep the "closed in loom 1.0.0" framing over the widened set.

## Solution constraints

- Out of scope: the `loom/runtime/cancelled-by-session-shutdown` row in `code-registry-runtime.md` — it continues to define no `file` / `range` source and is not edited.
- Out of scope: the **Serialised content format** paragraph — its drop-the-prefix carve-out already keys on the codes enumerated under **Location-less diagnostics**, so it picks up the addition without a second edit.
