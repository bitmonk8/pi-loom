## Problem

In `conventions.md` §REQ-ID discipline, the *executable spec REQ-ID* definition enumerates the diagnostics-registry code namespace as the closed triple `loom/{parse,load,runtime}/*`, while every other plan surface that resolves a registry code uses the open `loom/...` form — the *Specific exception types only* broad-catch predicate, `H5c`, and the `H5a` registry↔asserting-test reconciliation. The registry actually has a fourth shard: the spec's `diagnostic-shape.md` defines `loom/host/*` and `code-registry-host.md` registers concrete `loom/host/*` codes. The plan relies on those codes resolving as registry codes in two places (the `V9h` broad-catch allow-list tokens and the `H5a` parity scan), so read literally the closed triple contradicts the rest of the plan and the spec — two good-faith implementers reading the contradictory clauses would diverge on whether `loom/host/*` is a registry code.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `6bf8f6c` (2026-06-11) "pi-loom plan: resolve \"executable spec REQ-ID never defined as a predicate\"" — authored the *executable spec REQ-ID* predicate in `conventions.md` §REQ-ID discipline, enumerating the diagnostics-registry codes as the closed triple `loom/{parse,load,runtime}/*`.
- `8a6c9f6` (2026-06-11) "pi-loom plan: resolve \"un-anchored-MUST recogniser diverging registry scopes\"" — migrated the sibling un-anchored-MUST recogniser clause in the same section from the closed triple to the open `loom/...` form (adding the `loom/typecheck/*` carve-out) but did not reconcile the *executable spec REQ-ID* enumeration, leaving the closed triple as the lone holdout.

**History:** The closed-triple phrasing first entered `conventions.md` on 2026-06-10 (`0603eb4`) inside the un-anchored-obligation-class definition, then was duplicated into the new executable-REQ-ID predicate on 2026-06-11 (`6bf8f6c`). Same-day commit `8a6c9f6` open-formed the un-anchored recogniser arm to fix a diverging-registry-scope finding but touched only that clause; the executable-REQ-ID enumeration retained the closed triple through every subsequent edit (`git log -G` on the surviving phrase shows it unchanged from `6bf8f6c` to HEAD). The broad-catch predicate and `H5c` independently use the open `loom/...` form, so the divergence is the executable-REQ-ID clause being the single surface never migrated.

## Solution approach

In `conventions.md` §REQ-ID discipline, rewrite the closed-triple enumeration `loom/{parse,load,runtime}/*` in the *executable spec REQ-ID* definition to the open `loom/...` form that `H5a`, `H5c`, and the *Specific exception types only* broad-catch predicate already use, so the restatement admits the spec-defined `loom/host/*` shard. This is the only remaining closed-triple occurrence in the bullet.

## Solution constraints

- Resolve the contradiction by aligning the §REQ-ID discipline enumeration to the open form — do not narrow the `H5c` broad-catch resolver or rename the `V9h` `loom/host/*` codes, which are correct as written; the spec diagnostics namespace (`diagnostic-shape.md`, `code-registry-host.md`) is read-only for this fix.
- Preserve the existing `loom/typecheck/*` build-time brand-string carve-out already stated later in §REQ-ID discipline.
