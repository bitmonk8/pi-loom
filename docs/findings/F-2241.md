## Finding

`session-shutdown-semantics.md` sub-step 3 (*Await subagent disposal*) defines the bounded-await cap inline (`SHUTDOWN_AWAIT_CAP_MS = 2000`) and closes with: "The `SHUTDOWN_AWAIT_CAP_MS` constant lives in the same source-of-truth pinned-constants block as the Step 0 capability-probe constants so the build-time literal-read assertions can assert it." This sentence imposes a real placement obligation — the constant MUST be declared in one shared module so the build-time literal-read assertion has a single source to read — but identifies that module only by prose description ("the Step 0 capability-probe constants block"). No path, file, or anchor in these files resolves to it.

The Step 0 block is described in `capability-probe.md` (e.g. "the member name in the extension module's pinned-constants block") but is likewise never given a citable anchor or named module path anywhere in the corpus. An implementer reading sub-step 3 in isolation therefore has no link to follow to the declaration site; the build-time literal-read co-location guarantee the sentence asserts rests on a cross-reference that cannot be resolved.

## Issue introduction
**Verdict:** introduced-by-named-commits

The descriptive-only reference has been preserved verbatim across the spec split (`f5e89f4`, the 100KB/30KB size-cap split) and the relocation into `session-shutdown-semantics.md` (`c3de47a`, fix F-1602); only the trailing clause was edited (`9e53613` dropped the `H1` plan-leaf reference in favour of "the build-time literal-read assertions"). The referenced constants block has never been given a citable spec path or anchor, so the defect has been present since the sentence's inception.

## Solution Space

**Shape:** single

### Recommendation

Add a stable anchor (e.g. `<a id="pinned-constants-block"></a>`) at the canonical Step 0 capability-probe pinned-constants block in `capability-probe.md`, then replace sub-step 3's prose-only reference with a resolving link to that anchor. If the corpus position is that the block's module path is intentionally implementation-owned (no spec home), instead state that explicitly at the sub-step-3 site and pin the co-location invariant directly: `SHUTDOWN_AWAIT_CAP_MS` MUST be declared in the same single module the Step 0 probe constants occupy, so the build-time literal-read assertion reads one source. Either way the sub-step-3 reader must be able to resolve the declaration site without searching. Keep the anchor/cross-reference consistent with the broader unnamed-block fix so a single named target serves every callsite that cites this block.

## Consequence

**Severity:** advisory

An implementer following only the session-shutdown sub-step-3 contract cannot locate the declaration site and may introduce a fresh `SHUTDOWN_AWAIT_CAP_MS` constant in the session-shutdown module instead of co-locating it with the Step 0 probe constants, defeating the single-source-of-truth the build-time literal-read assertion depends on. The plan compensates (it pins the import-not-redeclare rule on `V9a`), so a working system is still reachable; the gap is a spec self-containment defect rather than a runtime hazard.
