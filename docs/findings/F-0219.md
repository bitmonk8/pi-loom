## Finding

`H6a` carries the loom 1.0 release-gate acceptance item for the `H4a` manual real-host smoke run. Its **Ships when** treats the run as recorded once the gate-activation commit message captures four evidence fields — the Pi-SDK pin literal the run executed against, the named owner who ran it, the run date, and the observed result against `H4a`'s narrowed model-output-invariant criterion — and the leaf states a skipped, mis-recorded, or stale-pin run is "detectable after the fact by reading the commit message."

That detection is entirely human. Nothing in `H6a`, `H5a`, or the closing-gate machinery asserts that the four fields are actually present in the commit message, nor that the recorded pin literal matches the single-source-of-truth shipping pin at [`host-prerequisites.md#pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin). The "recorded" obligation — which the leaf elevates to a release-blocking gate ("the release does not pass until that gate-activation commit message records the run") — is therefore enforced only by a reviewer reading prose, with no failing signal when a field is omitted or the pin is stale.

This is a disclosed, by-design property of the manual gate rather than a contradiction in the plan: loom 1.0 deliberately ships with no mechanical real-host fidelity gate. The gap is that even the cheap, mechanically-checkable half — presence of the four fields and equality of the recorded pin against the shipping pin — is left to human reading.

## Solution Space

**Shape:** multiple

### Option A — Accept as disclosed by-design residue

**Approach:** Treat the absence of a mechanical check as intentional and make the disclosure explicit. Extend the existing manual-gate disclosure in `H6a`'s Release-gate acceptance item to state that the presence of the four evidence fields and the pin-match are themselves verified only by release-time human reading, not by `npm test`.
**Plan edits:** Add a clause to `H6a`'s Release-gate acceptance (manual real-host smoke) text recording that the evidence-record presence and pin-match are contributor/reviewer discipline, not a mechanical gate.
**Spec edits:** None.
**Pros:** No new machinery; no coupling of the gate to commit-message text; the one-shot release-time gate stays simple.
**Cons:** The recorded-evidence obligation remains mechanically unenforced; a missing field or stale pin is caught only by a careful human reader.
**Risks:** None mechanical.

### Option B — Add a commit-message lint as a release-gate arm

**Approach:** Add an arm asserting that the gate-activation commit message contains the four evidence fields and that the recorded Pi-SDK pin literal equals the single-source-of-truth shipping pin at `host-prerequisites.md#pi-sdk-pin`.
**Plan edits:** Name the lint in `H6a`'s Ships when (and/or add the arm to `H5a`'s closing-gate machinery), with its assertion described as: the four fields are present and the recorded pin literal matches the shipping pin.
**Spec edits:** None (the lint reads `host-prerequisites.md#pi-sdk-pin` read-only).
**Pros:** Turns a skipped, mis-recorded, or stale-pin run from an after-the-fact human catch into a hard `npm test` failure for the mechanically-checkable half.
**Cons:** Couples the gate to commit-message text and format; fires once at release time, so the added machinery has low reuse.
**Risks:** A commit-message lint inherits the squash/rebase fragility raised in finding 60 — if the recording commit message is dropped or rewritten at merge, the lint reads the wrong message. Option B is only sound once finding 60's message-persistence question is settled.

### Recommendation

For a one-time, release-time manual gate that the plan already discloses as manual, Option A is the lower-risk path and resolves the finding by closing the disclosure gap. Adopt Option B only if finding 60 is resolved first, so the lint has a stable commit message to read. If Option B is taken, the implementer must agree the pin's single-source-of-truth location (`host-prerequisites.md#pi-sdk-pin`) with the lint so the pin-match compares against the canonical literal rather than a copy.

## Consequence

**Severity:** advisory

The release gate ships as a manual gate whose recorded-evidence obligation has no mechanical backstop: an omitted evidence field, or a recorded pin that does not match the shipping pin, passes silently and is caught only if a release reviewer reads the commit message carefully. The closing gate still functions as designed; the residue is that the cheapest verifiable half of the obligation is left unenforced.
