## Finding

The manual real-host smoke gate's pass criterion (e) requires that "the emitted `loom-system-note` codes are a **subset** of the **committed permitted-code list** … a manually-maintained, best-effort union of the `loom/...` codes the slices in `H7a`'s **Deps** can emit." The criterion compares two sets — emitted codes and the permitted `loom/...` codes — but never states how the runner derives a "code" from an emitted `loom-system-note`, nor that the derived values are guaranteed to be in the `loom/...` namespace the permitted list is keyed on.

The spec shows this is not a trivial read. A `loom-system-note` carries one of four disjoint `details` payload shapes (`runtime-event-channel.md` §System notes): `details: { diagnostics: Diagnostic[] }`, `details: { event: RuntimeEvent }`, `details: { structural: … }`, and `details: { recovery: … }`. Only the `diagnostics` variant carries `Diagnostic.code` (always a `loom/...` code per the Code-namespaces section of `diagnostic-shape.md` and DIAG-1). The `event` variant carries a `RuntimeEvent` whose `kind` is a `QueryError.kind` or binder failure cause — **not** a `loom/...` code — and whose `code?` is optional ("diagnostic code when one applies"). The `structural` and `recovery` variants carry no code at all. H7a's Deps include slices that emit group-A `details: { event }` notes (e.g. `V11f` binder-retry taxonomy, `V13c`/`V13d` query failure paths), and H7a's own final Tests bullet drives a binder-retry-class breach through the fixture, so the smoke run will emit `event`-shaped notes whose `kind` is outside `loom/...`.

Without a stated extraction rule, two auditors running the same live fixture score (e) differently: one reads `details.diagnostics[i].code` and silently ignores every `event`/`structural`/`recovery` note; another treats `event.kind` as the note's "code" and over-fails (a binder-failure `kind` is never in the `loom/...` permitted list). The criterion's pass/fail verdict is therefore not well-defined for the very notes the H7a fixture path produces.

None of these added a rule for deriving a code from a `loom-system-note` payload. The defect is a present-since-inception gap in the criterion, not a regression introduced by the move.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/real-host-smoke-gate.md`, in the Pass/fail criterion's clause (e), state how the runner derives the emitted-code set from each `loom-system-note` it observes, keyed on the note's `details` payload variant:

- `details: { diagnostics: Diagnostic[] }` — each element's `code` field is an emitted code (always a `loom/...` code per `diagnostic-shape.md` Code namespaces and DIAG-1).
- `details: { event: RuntimeEvent }` — the optional `event.code` field is an emitted code **only when present** (a `loom/...` code); `event.kind` is a `QueryError.kind` / binder failure cause, is not in the `loom/...` namespace, and is **not** compared against the permitted list.
- `details: { structural: … }` and `details: { recovery: … }` — informational notes that carry no code and contribute nothing to (e).

Then confirm that every code so extracted is in the `loom/...` namespace, matching the namespace the permitted-code list is keyed on. Cross-reference the authoritative surfaces rather than restating them: `runtime-event-channel.md` §System notes (the four `details` variants and `RuntimeEvent.code?`) and `diagnostics/diagnostic-shape.md` (the `Diagnostic.code` field and Code namespaces). The smoke gate already cites `V7d` and `H7a`; the added cross-references pin the emission shape (V7d delivers `Diagnostic[]` in `details.diagnostics`) and the namespace closure.

Edge case the implementer must keep: because group-A `event`-shaped notes are in scope for the run (H7a's Deps emit them and H7a's final Tests bullet drives a binder breach), the extraction rule must explicitly account for the `event` variant — silently scoring only the `diagnostics` variant would leave the binder-failure notes unscored.

## Consequence

**Severity:** correctness

Two reasonable auditors scoring criterion (e) on the same live smoke run diverge: one extracts only `details.diagnostics[i].code` and ignores all `event`-shaped notes, the other compares `event.kind` against the `loom/...` permitted list and over-fails on every binder-failure / `QueryError`-kind note. The criterion's pass/fail boundary is undefined for the `event`-shaped notes the H7a fixture path provably emits, so (e) can spuriously fail a clean run or vacuously pass a divergent one.
