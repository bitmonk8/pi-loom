## Problem

`cancellation.md` describes the `Checkpoint` seam's production wiring in two places that disagree about the `loop-iter` kind. The **Granularity** section requires the loop-iteration checkpoint to yield one macrotask turn before reading the signal, because a compute-bound `for`/`while` body with no genuine `await` drains only the microtask queue and would never let the Pi-dispatched abort macrotask run. The **Edge cases** bullet then asserts a blanket "Production wiring is a no-op (an already-resolved promise per checkpoint)"; an already-resolved promise resolves on the microtask queue, so read literally — "per checkpoint" includes `loop-iter` — it contradicts the Granularity requirement. The seam owner PIC-10 (`#checkpoint-seam`) already states the reconciled rule; an implementer anchoring on the stale edge-case aside could wire `loop-iter` as a microtask-only no-op and leave compute-bound loops uncancellable by Esc.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** `8cd3c4a` (2026-06-03, "resolve 'Loop-iteration checkpoint does not yield the event loop'"); predisposing commit `033b8e3` (2026-05-05, "resolve 'Race semantics: no injectable seam to land an abort between an Ok return and the next checkpoint'")
**History:** `git log -S` on `cancellation.md` shows the edge-case bullet "Production wiring is a no-op (an already-resolved promise per checkpoint)" entered in `033b8e3`, when the `Checkpoint` seam was first introduced and *every* checkpoint was a microtask no-op — the blanket "per checkpoint" claim was accurate at that point. The `loop-iter` macrotask-yield carve-out ("yields one macrotask turn … the `loop-iter` case of the `Checkpoint` seam") was added later by `8cd3c4a`; `git show 8cd3c4a -- cancellation.md` confirms that commit touched only the Granularity paragraph (a 2-line insertion) and did not revisit the pre-existing edge-case bullet. The contradiction is thus an interaction artefact: `8cd3c4a` introduced the carve-out without reconciling the prior aside. Subsequent commits `397d37e` (2026-06-05, relocate the `#checkpoint-seam` definition) and `fdc5115` (2026-06-06, single-macrotask abort-flip presupposition) refined PIC-10 / presupposition (vi) into the correct carve-out form on the seam-owner page but likewise left `cancellation.md`'s blanket edge-case wording stale.

## Solution approach

Rewrite the **Edge cases** production-wiring bullet in `cancellation.md` so it carves `loop-iter` out of the blanket microtask no-op claim and mirrors the seam owner PIC-10 (`#checkpoint-seam`). Also soften the adjacent "imposes no observable behaviour on production code beyond the always-`await`ed no-op" clause so it admits the single per-iteration macrotask turn rather than re-asserting the over-broad claim.

- Out of scope: re-deriving the `loop-iter` macrotask-yield primitive (`Clock.setTimeout(fn, 0)`, PIC-12 timer surface) in `cancellation.md` — PIC-10 owns it; cite it by reference only.
