## Finding

GOV-30 and GOV-31 are framed around "`spec.md` aggregator paragraphs": GOV-30's title is "*`spec.md` aggregator paragraphs are informative; lock-step convention*", its lock-step trigger says "*a PR that adds, removes, or renames an item on a source page … MUST update the corresponding `spec.md` aggregator in the same commit*", and its closing clause states "*previous load-bearing MUSTs scattered across `spec.md` aggregator paragraphs … are removed in favour of citing GOV-30*". GOV-31 carries the same framing, with prose references to "the orientation prose literal that names the count on `spec.md`" and to "the integer-count aggregators currently enumerated above" hosted on `spec.md`.

Current `docs/spec.md` is a six-line stub whose entire body is a three-link table of contents into `spec_topics/overview-and-orientation.md`, `spec_topics/language-and-architecture.md`, and `spec_topics/session-model-and-appendix.md`. The aggregators GOV-30/31 actually govern live on those topic pages: the Scope bullets, the Prerequisites Host-runtime paragraph, the Hard-ceilings bullet, the cross-cutting NFR bullets, and the SDK-capabilities list are all on `overview-and-orientation.md`; the loom 1.0 non-goals aggregator is on `language-and-architecture.md`. GOV-30/31's own inline cross-references already point at those pages (`../overview-and-orientation.md#scope`, `…#prerequisites`, `…#hard-runtime-ceilings`, the language-and-architecture non-goals anchor) — the prose framing and the link targets are out of sync within the same rule.

The result is an internal contradiction in GOV-30: the natural-language statement of the lock-step MUST names the wrong file, while the embedded links name the right ones. A maintainer who reads only the prose ("update the corresponding `spec.md` aggregator") will go searching `spec.md` for an aggregator that is not there; a maintainer who follows the links will land on the orientation pages and discover the prose is wrong. Two reasonable readers can disagree on which file the lock-step co-edit MUST touches, and reviewers enforcing the rule have no single authoritative locus. The orientation pages themselves reinforce the contradiction: every aggregator-paragraph subhead on `overview-and-orientation.md` is annotated "*Orientation aggregator (per [Governance — GOV-30](./governance.md))*", anchoring the convention to a paragraph the convention's own prose excludes.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f5e89f4` (2026-06-04, `docs: enforce 100KB hard / 30KB recommended size cap on spec set`) interacting with `126ff8a` (2026-05-06, `pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B`) and `fe8aa5c` (2026-06-05, `pi-loom spec: resolve "GOV-12 bundles ≥3 independently-testable sub-obligations"`).
**History:** The aggregator lock-step convention was authored in `126ff8a` as a single rule (`GOV-12`) whose prose was correct at the time — `docs/spec.md` was then a monolith that hosted the Scope, Prerequisites, Hard-ceilings, NFR, and loom 1.0 non-goals paragraphs the rule governed. The size-cap split in `f5e89f4` (later refined by `81bf52a` on 2026-06-11, which renamed `spec/` to `spec_topics/`) moved those aggregator paragraphs off `spec.md` into `spec/overview-and-orientation.md` and `spec/language-and-architecture.md`, reducing `spec.md` to a three-link TOC. The split repointed 1571 cross-references mechanically but did not rewrite the GOV-12 prose itself, so the rule's natural-language framing — "`spec.md` aggregator paragraphs", "the corresponding `spec.md` aggregator", "the orientation prose literal that names the count on `spec.md`" — was left referring to a host that no longer exists. The subsequent GOV-12 → GOV-30/31 split in `fe8aa5c` (2026-06-05) reflowed the prose into two rules but preserved the obsolete `spec.md` framing in both. Neither commit was individually wrong: `f5e89f4`'s scope was layout, and `fe8aa5c`'s scope was a GOV-12 sub-obligation split; the defect is the unmet co-edit obligation between them — GOV-30/31's own lock-step rule applied to itself.

## Solution Space

**Shape:** multiple

### Option A — Re-anchor GOV-30/31 prose to name the actual aggregator-host pages

Approach
- Rewrite GOV-30/31's natural-language framing so every occurrence of "`spec.md` aggregator paragraph(s)", "the corresponding `spec.md` aggregator", and "the orientation prose literal that names the count on `spec.md`" names the actual host page (`overview-and-orientation.md` for most, `language-and-architecture.md` for the loom 1.0 non-goals aggregator).
- Introduce a one-line definition near the top of GOV-30 — "*orientation aggregator pages*" or "*orientation aggregator hosts*" — covering the closed set of post-split files (`overview-and-orientation.md`, `language-and-architecture.md`, and `session-model-and-appendix.md` as it carries the SM-N forward-link inventory).
- Use that defined term in GOV-30's lock-step MUST: "*MUST update the corresponding aggregator on its orientation aggregator host page in the same commit*".
- Apply the same rename to GOV-31's literal-preservation MUST and to its `currently:` enumeration, where every "`spec.md` — Scope/Prerequisites/Hard ceilings" cross-reference already points at `overview-and-orientation.md` — the link targets become consistent with the prose for the first time since the split.
- Sweep parallel "`spec.md` aggregator" phrasings in GOV-23, GOV-29, the `stable-inline-labels.md` "GOV-30 lock-step coverage" bullet, the `glossary.md` GOV-N entry, and the `req-id-prefix-table-retired.md` GOV-12-split annotation in the same commit.

Spec edits
- `docs/spec_topics/governance/req-id-prefix-table-active-b.md`: GOV-30 and GOV-31 prose rewrites; introduce the orientation-host-page definition.
- `docs/spec_topics/governance/stable-inline-labels.md`: "*update the corresponding `spec.md` aggregator*" → defined-term phrasing.
- `docs/spec_topics/governance/anchor-scheme-and-retired.md`: GOV-23 framing review (the Session Model section forward-link inventory is on `session-model-and-appendix.md`, not `spec.md`).
- `docs/spec_topics/governance/release-version-naming.md`: GOV-29's "`docs/spec.md` + `docs/spec_topics/*.md`" framing is technically still accurate (the union covers everything), but the "known instance" parenthetical and the "GOV-30 lock-step normative-owner site" phrasing can be reconciled with the new definition.
- `docs/spec_topics/glossary.md`: GOV-N entry's "`spec.md` aggregator lock-step" phrase updated.

Pros
- Removes the contradiction at its source — the rule's prose names the same files its links already target.
- Preserves the natural-language framing of every aggregator's annotation on `overview-and-orientation.md` and `language-and-architecture.md` ("*Orientation aggregator (per Governance — GOV-30)*") without forcing those subheads to acquire a new collective label.
- Future-proof against a third orientation-host page being added later: extend the defined-term enumeration once, no further prose churn.

Cons
- Wide-surface edit (≥6 files); each rewrite touches dense, single-paragraph GOV rules where the rewording risk is non-trivial.
- The new defined term ("orientation aggregator host") adds a coinage to the governance vocabulary that readers have to learn.

Risks
- A partial sweep that leaves one or two "`spec.md` aggregator" phrasings behind reintroduces the same contradiction in a smaller surface. Mitigation: grep `spec\.md` aggregator` and `corresponding `spec\.md`` across `spec_topics/` after the edit.

### Option B — Redefine "`spec.md`" in GOV-30/31 to denote the orientation-host set

Approach
- Add an explicit definitional sentence at the top of GOV-30: "*Throughout GOV-30 and GOV-31, '`spec.md`' denotes the orientation aggregator pages the monolith was split into — `overview-and-orientation.md`, `language-and-architecture.md`, and `session-model-and-appendix.md` — not the post-split TOC stub at `docs/spec.md`.*"
- Leave the remaining prose untouched: every "`spec.md` aggregator paragraph" reads correctly under the new definition.
- Add a parallel sentence to the `glossary.md` GOV-N entry and to the `stable-inline-labels.md` "GOV-30 lock-step coverage" bullet so the same convention applies wherever the phrase recurs.

Spec edits
- `docs/spec_topics/governance/req-id-prefix-table-active-b.md`: one-sentence definitional preamble in GOV-30 (GOV-31 inherits via its `[GOV-30](#gov-30)` back-references).
- `docs/spec_topics/glossary.md` and `docs/spec_topics/governance/stable-inline-labels.md`: matching definitional notes.

Pros
- Minimal-surface edit: three files, one paragraph each.
- Preserves the GOV-30/31 prose byte-for-byte beyond the preamble — no risk of inadvertently breaking the dense bidirectional-referent-identity machinery in the rule's body.

Cons
- "`spec.md`" is then used in two distinct senses across the corpus: GOV-30/31 (orientation-host set) vs. every other site (the literal stub file). A reader encountering the phrase outside GOV-30/31 has to recall which sense applies.
- Visually contradicts GOV-29, which uses "`docs/spec.md` + `docs/spec_topics/*.md`" to mean the literal file plus the topic tree — the orientation-host set is a third interpretation.
- The aggregator-host annotations on the orientation pages still say "*Orientation aggregator (per Governance — GOV-30)*" without scare-quoting "`spec.md`", so a reader who lands on those pages first never learns the redefinition exists.

Risks
- A future editor restating GOV-30 prose elsewhere (e.g. in a release-notes summary or onboarding doc) may use the literal sense of "`spec.md`" and silently reintroduce the contradiction.

### Recommendation

Adopt Option A. The contradiction is fundamentally that the prose names a file the rule does not govern; renaming the prose to match the link targets it already carries is the durable fix and produces a corpus where every site that talks about an aggregator names the right file. Option B preserves the contradiction by overloading "`spec.md`" with a second meaning, which a corpus already burdened with `docs/spec.md` vs. `spec_topics/*.md` distinctions does not need.

Edge cases the implementer must watch:
- GOV-30's "Session Model section's forward-link inventory" sub-clause currently reads as if that inventory lives on `spec.md`; it lives on `session-model-and-appendix.md`, so the rename must include that page in the orientation-host enumeration.
- GOV-31's in-code-constant carve-out paragraph mentions "*the orientation prose literal that names the count on `spec.md`*" — the rename target here is `overview-and-orientation.md` Prerequisites paragraph specifically, not the orientation-host set as a whole.
- The annotation phrase used 14× on `overview-and-orientation.md` and `language-and-architecture.md` ("*Orientation aggregator (per [Governance — GOV-30](./governance.md))*") is correct as written under Option A — those subheads are the aggregators, no rewording needed.
- GOV-29 also says "`docs/spec.md` + `docs/spec_topics/*.md`": that phrasing means the union and remains accurate; the rename does not extend to GOV-29 unless the implementer reconciles the "normative-owner site" / "aggregator copy" framing with the new defined term in the same pass.
- Apply Option A's rename to the GOV-12-split annotation row in `req-id-prefix-table-active-b.md` (line 19, the `Retirement` column for GOV-12) so the retired-rule provenance reads consistently with the renamed live rules.

## Consequence

**Severity:** correctness

The lock-step co-edit convention that GOV-30 exists to operationalise is mis-anchored: contributors and reviewers cannot read the rule in isolation and arrive at the correct file. Drift between the orientation aggregators and the topic pages they aggregate is the exact failure mode GOV-30 is designed to catch, and that failure mode is now itself made more likely by the rule's own mis-statement. The defect is documentation-level — no runtime behaviour changes — but the spec corpus's self-governance discipline degrades until the rename or redefinition lands.
