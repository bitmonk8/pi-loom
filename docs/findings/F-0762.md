## Finding

The PIC corpus already routes the file-watch surface through a library-agnostic `FileWatcher` seam ([host-interfaces-services.md §PIC-14](../spec_topics/pi-integration-contract/host-interfaces-services.md#pic-14)) and treats `chokidar` as an implementation choice ([host-prerequisites.md §Loom-package implementation dependencies](../spec_topics/pi-integration-contract/host-prerequisites.md), [implementation-notes.md §Loom-package implementation dependencies (loom 1.0)](../spec_topics/implementation-notes.md#loom-package-implementation-dependencies-loom-1-0)). Despite this, several normative sites in the PIC subtree still name `chokidar` directly as the watcher being registered or closed, mixing the production-adapter identity into prose whose obligation is on the seam, not on any specific library:

- [registration-steps.md §watcher / hot-reload registration (step 5)](../spec_topics/pi-integration-contract/registration-steps.md#watcher-hot-reload-registration): "Registers a chokidar file watcher over the discovered roots…", "On a chokidar event for an **existing** loom or `.warp` file the watcher debounces…", and the *Structural changes* paragraph's "a chokidar burst over discovery roots".
- [session-shutdown-semantics.md §Factory-ordering pin](../spec_topics/pi-integration-contract/session-shutdown-semantics.md#factory-ordering-pin): "the chokidar file watcher" in the watcher-lifecycle restatement.
- [patch-skew-degradation.md §sub-step 4 — Close watchers](../spec_topics/pi-integration-contract/patch-skew-degradation.md): "Call `discoveryWatcher.close()` (the chokidar watcher established in step 5 above)" — the parenthetical pins the library inside the teardown-call MUST.
- [registration-steps.md §step 4 teardown contract](../spec_topics/pi-integration-contract/registration-steps.md): the teardown rationale names "chokidar file handles" alongside SDK objects.
- [active-invocation-registry.md](../spec_topics/pi-integration-contract/active-invocation-registry.md) edge cases name "`AbortController`, chokidar `FSWatcher`, `AgentSession`" as the directly-touched SDK objects.

This is inconsistent with the corpus's own de-prescription pattern at sibling sites where construction or matching identity is similarly an implementation choice: `defineTool` was demoted from normative prose to a *Non-normative implementation note* in [extension-bootstrap-and-per-loom.md](../spec_topics/pi-integration-contract/extension-bootstrap-and-per-loom.md), and `minimatch` is consistently flagged as an implementation choice in [implementation-notes.md](../spec_topics/implementation-notes.md). The PIC normative obligation here is on the watcher contract — coalescing bursts in the 250 ms debounce window, exposing disposable `.close()`-style handles, emitting `add`/`change`/`unlink` per PIC-14 — none of which requires any one library.

(`pi-integration.md` is named in the original heading but contains no `chokidar` reference; its file-watcher bullet is already library-agnostic. The finding's actual surface is the PIC subtree files enumerated above.)

## Issue introduction

**Verdict:** present-since-inception (with later partial-sweep history)
**Introducing commits:** `d38ffdb` (2026-05-05, "spec: hot-reload via in-process LoomRegistry swap; no ctx.reload() for content edits (Decision #8)") — the commit that first introduced the hot-reload prose with `chokidar` named as the watcher.
**History:** `git log -G chokidar -- docs/spec_topics/` and `git log --follow` over the relevant files show `chokidar` entered the normative prose with `d38ffdb` (then living in the un-split `pi-integration-contract.md`) and was carried forward verbatim through the file split (`fecb504`, `81bf52a`) into today's `registration-steps.md` / `session-shutdown-semantics.md` / `patch-skew-degradation.md`. Three later editorial passes established the de-prescription convention at sibling sites without sweeping the watcher-registration prose: `4cfe858` (2026-06-04, "resolve 'chokidar named but never declared as a dependency'") added the explicit `chokidar` enumeration in the dependency clause; `1ea0455` (2026-06-04, "DI-seam vs reference-implementation placement boundary") moved that enumeration to `implementation-notes.md`, isolating `chokidar` as an implementation choice behind the library-agnostic `FileWatcher` seam; `8362824` (2026-06-04, "Demote over-prescriptive and non-testable MUSTs on unobservable implementation structure") demoted the parallel `defineTool` naming to a *Non-normative implementation note*. None of those passes touched `registration-steps.md` step 5, the `session-shutdown-semantics.md` *Factory-ordering pin* restatement, or the `patch-skew-degradation.md` sub-step 4 parenthetical, leaving the `chokidar` token in normative prose that the rest of the corpus has since classified as library-agnostic surface. A separate sub-line of history (`6d1a949`, 2026-06-05, "session-shutdown sub-step 5 hot-reload split") moved the watcher-lifecycle prose between `session-shutdown-semantics.md` and `registration-steps.md` but preserved the `chokidar` token rather than de-prescribing it. The defect is therefore best characterised as a missed sweep across a multi-commit de-prescription pattern, not a single regressive introduction.

## Solution Space

**Shape:** single

### Recommendation

Replace every normative-prose mention of `chokidar` in the PIC subtree with a library-agnostic noun keyed to the existing seam, leaving the `chokidar` identity intact only where the corpus has already isolated it (the [host-prerequisites.md *Loom-package implementation dependencies*](../spec_topics/pi-integration-contract/host-prerequisites.md) cross-reference and the [implementation-notes.md §Loom-package implementation dependencies (loom 1.0)](../spec_topics/implementation-notes.md#loom-package-implementation-dependencies-loom-1-0) canonical enumeration).

Concrete sweep:

- **`registration-steps.md` §step 5 watcher / hot-reload registration.** "Registers a chokidar file watcher over the discovered roots…" → "Registers a file watcher (the [`FileWatcher`](./host-interfaces-services.md#filewatcher-interface) seam) over the discovered roots…". "On a chokidar event for an **existing** loom or `.warp` file" → "On a file-watcher event for an **existing** loom or `.warp` file". *Structural changes* paragraph: "a chokidar burst over discovery roots" → "a file-watcher burst over discovery roots".
- **`registration-steps.md` §step 4 teardown contract.** "chokidar file handles" → "file-watcher handles".
- **`session-shutdown-semantics.md` §Factory-ordering pin.** "the chokidar file watcher" → "the file watcher (the [`FileWatcher`](./host-interfaces-services.md#filewatcher-interface) seam)".
- **`patch-skew-degradation.md` §sub-step 4 — Close watchers.** "the chokidar watcher established in step 5 above" → "the discovery-root file watcher established in step 5 above".
- **`active-invocation-registry.md`** teardown-handler edge case: "`AbortController`, chokidar `FSWatcher`, `AgentSession`" → "`AbortController`, file-watcher handles, `AgentSession`".

Edge case the implementer must watch — the *load-bearing chokidar-behaviour* prose in [`drain-state-contract.md`](../spec_topics/pi-integration-contract/drain-state-contract.md) and [`session-only-degraded-state.md`](../spec_topics/pi-integration-contract/session-only-degraded-state.md) (the recurring "chokidar's unpinned / acknowledged-unpinned double-close behaviour" framing) MUST NOT be silently neutralised by the same token-swap. Those passages encode a real loom 1.0 acceptance: a second `.close()` against the same file-watcher handle may throw, and the per-call `try`/`catch` of the *Per-step isolation* paragraph routes any such throw through `loom/host/session-shutdown-teardown-step-failed`. Re-phrase those sites as "the file-watcher seam's unpinned double-close behaviour" (or equivalent) and keep the substantive double-close acceptance intact; do not delete the behavioural clause. If a single contiguous file-watcher contract for double-close is wanted, place it in [`host-interfaces-services.md` §PIC-14](../spec_topics/pi-integration-contract/host-interfaces-services.md#pic-14) and back-reference from the two drain-state sites — but that is an optional follow-up, not part of this de-prescription sweep.

After the sweep, the only surviving `chokidar` mentions in the spec corpus MUST be the two implementation-side anchors above, plus any non-normative cross-reference to them. A grep over `docs/spec_topics/` for the literal token `chokidar` is the implementer-visible acceptance check.

## Consequence

**Severity:** cosmetic

The watcher contract is already correctly library-agnostic at PIC-14 and the dependency clause; the normative-prose mentions of `chokidar` at sibling sites are editorial residue that an implementer could (reasonably but wrongly) read as pinning the library in the conformance contract itself. The defect is corpus-internal inconsistency — the cleanup that demoted `defineTool` and isolated `minimatch` as implementation choices never swept these PIC sites. No two reasonable implementers diverge on observable behaviour because of this prose alone; the seam still drives the contract.
