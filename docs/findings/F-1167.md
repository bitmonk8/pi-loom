## Finding

Item (a) in the **Editorial-review checklist for unpinned host presuppositions** has fundamentally different semantics from every other entry in the same list, yet is rendered as a peer of those entries. Items (b)–(ao) are *recurring per-bump audits* whose conformant outcome on every bump is normally `pass` / `fail` / `N/A` / `not-performed`; item (a) is a *one-shot, dispositive open question* that "MUST be resolved before SM-4/5/6 are implemented" and that — per the **Item (a) authoritative-determination recording shape** convention at the bottom of the page — does not even take the per-item recording shape the preamble describes (it takes one of two verdicts: `governed-by-rebind` or `provably-differs — <cited evidence>`). The blocking nature is therefore knowable only by reading two widely-separated paragraphs of the page back-to-back; an implementer scanning the checklist sees row (a) sandwiched between (b) and (c) and treats it as "one of the audit items".

Three concrete drift symptoms follow. First, the ordering MUST ("MUST be resolved before SM-4/5/6 are implemented") has no mechanical enforcement: nothing in `V18c`'s build-time gates or `V9l`'s acceptance criteria fails red if the SM-4/5/6 implementation lands without an item-(a) verdict in the bump commit message — the gate is "editorial review" on a row in a 41-item list and the per-item recording for the rest of that list explicitly tolerates `not-performed`, so a missing item-(a) verdict is indistinguishable on the surface from a deliberately skipped SHOULD-level row. Second, "SM-4/5/6" at its first occurrence in the item-(a) body is plain text — the cross-link to `session-model-and-appendix.md#sm-4-session-only-degraded-state` exists, but only in the **Outcome recording conventions** paragraph ~100 lines further down. Third, the page-level title ("Version bump step2") and the preamble describe a per-bump audit procedure; nothing in either page-level surface signals that a present-tense blocker also lives here.

The peer finding "host-prerequisites.md — clause (a) (root of the contradiction)" frames the same defect from the canonical location of the open contradiction; this finding is its `version-bump-step2.md` reflection. The placement defect on `version-bump-step2.md` is partially independent — even if `host-prerequisites.md` clause (a) is left exactly as-is, the row on `version-bump-step2.md` is still mis-shaped for its job and the ordering MUST is still mechanically un-enforced.

## Solution Space

**Shape:** single

### Recommendation

Keep the item-(a) row at position (a) in the `(a)`–`(ao)` enumeration on `version-bump-step2.md` — do not re-letter, so every existing intra-page and cross-page anchor is preserved — but hard-segregate it from the recurring per-bump audit shape:

- Rewrite the checklist preamble's "all five of which route their detection to editorial review under this procedure" clause so item (a) is listed as a *separate, blocking category* rather than as the lead element of an `(a)`–`(e)` MUST-audited group. The remaining MUST count becomes four items (b)–(e), and the existing "item (a) is the exception" parenthetical disappears as redundant — re-word both the count language and the `(a)`–`(e)` range references in the preamble accordingly.
- Prefix the item-(a) row body with a bold **BLOCKING — one-shot pre-implementation gate** marker.
- Hoist the **Outcome recording conventions** §**Item (a) authoritative-determination recording shape** sub-paragraph from the bottom of the page to inline-immediately-after the item-(a) row body, co-locating item (a)'s prose with its `governed-by-rebind` / `provably-differs — <cited evidence>` recording shape.
- Add the `SM-4/5/6` cross-link to `session-model-and-appendix.md#sm-4-session-only-degraded-state` inside the row body at its first occurrence (today it appears only in the recording-conventions paragraph ~100 lines down). `host-prerequisites.md` clause (a)'s existing link to `./version-bump-step2.md#bump-checklist-instance-survival` is unaffected, so no cross-file anchor edit is required.

Update `V18c-version-bump-checklist.md`'s **Ships when** so the item-(a) verdict's presence is asserted *as its own line item*, not folded into the existing "every item `(a)`–`(ao)`" recording check. `V18c`'s **Ships when** today says "no MUST-level item `(a)`–`(e)` carries a `fail` outcome"; item (a)'s verdict is `governed-by-rebind` / `provably-differs`, not `pass` / `fail`, so correct this clause in the same edit so it no longer treats item (a) inconsistently.

Add a build-time stub gate so the ordering MUST ("MUST be resolved before SM-4/5/6 are implemented") becomes a typecheck-or-test red rather than editorial review of a row in a 41-row list. At `V18c` build time, assert that a marker file or pinned constant — `INSTANCE_SURVIVAL_VERDICT: "governed-by-rebind" | "provably-differs"`, placed in the same pinned-constants block that holds `CAPABILITY_OBLIGATIONS` — carries a non-placeholder value before `V9l`'s degraded-branch obligations may compile against the symbol. This puts the ordering MUST on the same footing as the `loom/typecheck/session-shutdown-reason-snapshot` brand-string gate `V18c` already owns. Do not demote the MUST to a SHOULD: the gate is genuinely blocking and the spec is correct to state so. The stub-gate addition is independently landable from the placement edits and may be sequenced second.

The canonical blocker prose itself is left to land on `model-changes-and-non-goals.md` (or a new `open-questions.md`) under the sibling finding that resolves where it ultimately lives; `version-bump-step2.md` keeps only the bump-procedure obligation the bump contributor actually consults.

## Consequence

**Severity:** correctness

An implementer scanning `version-bump-step2.md` reads item (a) as a normal SHOULD-record-or-`not-performed` audit row, fails to discover the present-tense pre-implementation gate it actually carries, and ships `V9l` (and `SM-4`/`SM-5`/`SM-6`/`SM-3b`) against the un-resolved open contradiction with no build-time test reddening — exactly the failure mode `plan.md`'s §Blocked obligations entry is meant to prevent. The placement also amplifies the cross-cutting cluster's other defects (the session-model page lists SM-4/5/6 inline with active obligations, the model-changes-and-non-goals page hides a blocker in a non-goals file): every page that touches the blocker presents it as a routine entry.
