## Problem

PIC-1 clause (d) in `runtime-event-channel.md` states the *V1 reachable predicate* as an iff: a `validation` event (`ValidationError.cause: "schema_validation"`) at the typed-query response boundary carries `masked: ["ceiling#2"]` iff the forced respond turn's post-CIO-4-increment `tool_loop` slot count equals `max_rounds`; in every other case `masked` MUST be omitted. Only the positive arm is pinned by a binding fixture (the normative depth-6 worked example in `query-tool-loop.md`). No binding fixture pins the negative arm at its boundary (forced respond at slot count strictly below `max_rounds`) or the non-typed-query-response omit sites in the clause (c) per-site table. A conformance suite built strictly to the cited vectors would pass an implementation that emits `["ceiling#2"]` whenever ceiling #4 fires — ignoring the slot-count condition — which the iff exists to forbid.

## Issue introduction
**Verdict:** single introducing commit
**Introducing commit:** `7598f50` — *pi-loom spec: resolve "CIO-6 hard-ceiling co-fire: predicate, test vector, and normative ownership"* (2026-05-07)
**History:** The spec corpus is git-tracked. `git log -S` for the iff phrasing and for the `V1 reachable predicate` token, traced through the two later pure-relocation commits (`31ff060` "Move spec, plan, and review docs into docs/" and `f5e89f4` the 100KB split into the subdirectory layout), resolves to `7598f50`. That commit introduced PIC-1 clause (d) (the iff predicate), the clause (c) per-site reachable mask domain, the clause (e) pure-read MUST, **and** the normative "Worked example: depth-6 forced respond at `max_rounds`" in `query.md` in a single change. The worked example pinned the positive arm only; no negative-arm boundary fixture or non-site omission fixture was authored then or in any later commit (`d75f2a8`, `F-0776`/`85daf5a`, and the other runtime-event-channel edits touched (c)(i) qualification, (e) wording, and Group-A/B routing, never the omit-side test surface). The testability gap is therefore present-since-introduction of the predicate itself: the rule and its positive-only test vector landed together, and the complementary omit-side vectors were never added.

## Solution approach

Add a binding acceptance-criteria block, anchored in PIC-1, pinning the omit side of the clause (d) iff as normative vectors: a forced respond turn whose post-CIO-4-increment `tool_loop` slot count is strictly less than `max_rounds` (with `max_rounds: 0` folded in as a sub-case) producing an emitted `validation` event with `masked` absent (field omitted, not `[]`); and the non-typed-query-response sites of the clause (c) per-site table (code-driven `<name>(args)` AJV, `invoke` entry, tool-call-round boundary) likewise omitting `masked`. Cite the existing depth-6 worked example for the positive arm rather than restating it, and cross-reference the negative-arm vector from `query-tool-loop.md`, mirroring how the positive vector is already cross-cited.
