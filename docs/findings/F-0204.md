## Finding

`H1a`'s **Adds** declares six packages — `ajv`, `ajv-formats`, `semver`, `chokidar`, `yaml`, `minimatch` — in loom's own runtime `dependencies`. `H1a`'s **Tests** section carries architectural presence assertions for every other manifest group it owns: `vitest` in `devDependencies`, the three lint-toolchain keys (`eslint` / `@typescript-eslint/parser` / `eslint-plugin-loom-local`) in `devDependencies`, the four `@earendil-works/pi-*` packages in both `peerDependencies` and `devDependencies`, `typebox` in `peerDependencies`, `typescript` in `devDependencies`, plus `engines.node` and `pi.extensions`. The runtime `dependencies` group is the only declared manifest section with no corresponding presence assertion.

Because `H1a`'s **Ships when** condition (`npm install && npm run build && npm run typecheck && npm test` on an empty `src/**` tree) exercises no production source, a runtime dependency that is missing, typo'd, or accidentally dropped during a manifest edit is never imported at `H1a` and ships green. The omission surfaces only much later, when a downstream slice first imports the package — `minimatch` at `V10b`, `yaml` at `V6a`, `ajv`/`ajv-formats` behind the `SchemaValidator` reference validator — at which point the failure is reported far from its cause.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e1194227df0702609667f3ee8be2e8c6b2 (2026-06-10) — initial plan build
**History:** The runtime `dependencies` group (`ajv`/`semver`/`chokidar`) was declared in `H1a`'s **Adds** in the initial plan commit c6a664e, and that commit's **Tests** section already asserted the peer-dependency line but carried no runtime-dependency presence assertion. Subsequent commits added presence assertions for each `devDependencies` group as those groups were introduced (test runner, lint toolchain, SDK `devDependencies`, `typescript`), but never added a parallel assertion for the runtime `dependencies`. The runtime set was later extended — `yaml` (1064946, 2026-06-10), `minimatch` (8279cb1, 2026-06-11), `ajv-formats` (ca5c2fc, 2026-06-12) — each added to **Adds** without a presence assertion, widening the pre-existing asymmetry rather than introducing it. `git log -S` over the assertion phrasing confirms no runtime-dependency presence test has existed at any revision.

## Solution Space

**Shape:** single

### Recommendation

Add an architectural test bullet to `H1a`'s **Tests** section that reads `package.json#dependencies` and asserts the six runtime keys — `ajv`, `ajv-formats`, `semver`, `chokidar`, `yaml`, and `minimatch` — are declared, on the same footing as the existing `devDependencies` presence assertions. The assertion fails when any of the six keys is absent from `dependencies`, so a missing or mis-keyed runtime dependency fails `H1a`'s own `npm test` rather than surfacing when a downstream slice (`V10b`/`V6a`/the `SchemaValidator` consumers) first imports it.

The six-key list is the set already enumerated in `H1a`'s **Adds**; `docs/spec_topics/implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)" is the spec grounding for that set and stays read-only. Keep the new assertion scoped to key presence (consistent with the existing `devDependencies` key checks); it need not assert version ranges, since the runtime deps carry no spec-anchored floor.

## Consequence

**Severity:** advisory

A runtime dependency absent or mis-keyed in `package.json#dependencies` passes `H1a` (empty `src/**` imports none of them) and is detected only when a downstream slice — `V10b` (`minimatch`), `V6a` (`yaml`), or a `SchemaValidator` consumer (`ajv`/`ajv-formats`) — first resolves the module. The gap is a coverage hole in `H1a`'s own provisioning checks, not a defect that blocks any leaf from being picked up; `H1a` already gates the parallel `devDependencies` group this way.
