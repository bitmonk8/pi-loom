## Problem

The Slash-command discovery bullet in `pi-integration.md` asserts as fact that the `description` field populates the autocomplete entry Pi's slash-command UI shows, and that each loom appears in autocomplete as `/<filename>` "exactly mirroring Pi's prompt-template behaviour". Both are behavioural claims about Pi's current slash-command UI — what it renders, from which field, and whether extension-registered commands share the surface with prompt-template commands — yet neither is tied to a named host-behaviour presupposition or a bump-time re-audit step. What the contract actually pins for `description` is the registration call shape (`pi.registerCommand(name, { description, handler })` at `capability-inventory-items.md#sdk-cap-slash-command-registration`) — that the field is accepted as an argument, not that the dropdown renders it. The asymmetry with the sibling `argument-hint` clause, whose extension-API gap is fully cited and forwarded, makes the unbacked claim conspicuous.

## Issue introduction

**Verdict:** present-since-introduction
**Introducing commits:** `53f5831` ("spec: rewrite query primitive, system prompts, and schema syntax", 2026-05-03) introduced the unsourced `description` → autocomplete claim in its current form when it first added the slash-command-discovery bullet (`"The description and argument-hint from frontmatter populate the autocomplete entry."`). `1e103d5` ("spec: rescope argument-hint to binder grounding only (Decision #7)", 2026-05-05) then asymmetrically split the bullet — giving `argument-hint` a fully-cited SDK-gap framing (forwarded to Future Considerations and the `frontmatter-fields-a.md` autocomplete-contract anchor) while *retaining* the `description` half as a bare behavioural assertion, which is the precise asymmetry this finding flags.
**History:** `git log -L '/Slash-command discovery/,+1:docs/spec_topics/pi-integration.md'` and `git log -G 'populate the autocomplete entry'` show a four-commit chain (`fecb504` → `d6cbb37` → `1e103d5` → `4aef0e2` → `4da6c6c`): the bullet originated in `53f5831` against the top-level `spec.md`, was carried into the split `spec_topics/pi-integration.md` by `fecb504`, and the description-vs-argument-hint split was performed in `1e103d5` (the only commit that touched the rendering language); subsequent commits only adjusted the discovery-sources clause and the `argument-hint` cross-reference. No commit ever added a host-behaviour anchor for `description`. The defect is therefore present-since-introduction in `53f5831` and made asymmetric (and more conspicuous) by `1e103d5`'s targeted rescope of the sibling clause.

## Solution approach

Promote the `description`-autocomplete-rendering behaviour to a named host-behaviour presupposition with its own anchor in `host-prerequisites.md`, following the sibling presuppositions there (`operator-always-present-presupposition`, `pi.getCommands()` completeness, `pi.registerTool` post-startup acceptance). The presupposition should cover both behavioural claims: that Pi renders the `description` argument in the autocomplete entry, and that extension-registered commands appear in the same autocomplete surface as prompt-template commands. Add a bump-checklist editorial-review item to `version-bump-step2.md` keyed to the new presupposition so each Pi minor bump re-audits the rendering claim. Rewrite the Slash-command discovery bullet in `pi-integration.md`, replacing the bare `description`/autocomplete and "mirroring Pi's prompt-template behaviour" assertions with a forward-link to the new presupposition, matching how the sibling `argument-hint` clause forwards to `frontmatter-fields-a.md#argument-hint-binder-only`.
