## Finding

The *Audit scope* paragraph in `audit-resolution.md` defines the *audited source tree* as "the closed file set matching the positive include glob `src/**/*.ts` … minus the excluded files enumerated here," and elsewhere pins that the build target (`dist/` or equivalent) is "outside the included root by construction." Both the include glob `src/**/*.ts` and the `dist/`-outside-`src/` prohibition are written as if `src/` and `dist/` were absolute, named directories — but the paragraph never states what directory the glob is anchored against.

Every other geometric property of the scan (case-sensitivity, symlink handling, extension-suffix matching, source-file encoding, exclusion list, generated-output prohibition) is pinned with implementer-grade precision, yet the load-bearing assumption underneath all of them — *where the glob walk starts* — is left implicit. An audit invocation from the package root walks one closed file set; an invocation from `src/extension/` walks `src/extension/src/**/*.ts` (typically empty); an invocation from a parent of the repo walks a superset. The "same closed file set for a given source tree" guarantee the *Cross-platform glob normalisation* sub-paragraph leans on, and the `dist/` prohibition's "outside the included root by construction" claim, both presuppose a specific anchor directory that the prose never names.

The natural anchor — the package root containing `package.json#peerDependencies` and the `engines.node` field the surrounding obligations already key on — is consistent with the rest of PIC and with `H1a-scaffold-and-toolchain.md`'s npm-flat-`node_modules` install posture, but consistency-by-implication is not the same as pinning it.

## Solution Space

**Shape:** single

### Recommendation

Append a single sentence to the *Audit scope* paragraph (immediately after the "minus the excluded files enumerated here" sentence that introduces the include glob, before the *Cross-platform glob normalisation* sub-paragraph) pinning the anchor directory:

> The include glob is anchored at the loom package root — the directory containing the `package.json` whose `peerDependencies` block is consumed by `Step 0 (d)` of the capability probe and by [`bump-step-2(a)`](./version-bump-step2.md#bump-step-2-positive). The audit MUST resolve `src/` (and the symmetric `dist/`-outside-`src/` prohibition) against that directory regardless of the process working directory at invocation time, so an audit invoked from any subdirectory or from a CI step that has `cd`'d elsewhere walks the same closed file set as one invoked from the package root.

Edge cases the implementer must watch:

- The anchor is the *loom* package root (the one carrying loom's `peerDependencies`), not a parent workspace / monorepo root — the four-`@earendil-works/*` peer-dep group and the `engines.node` literal-read assertion already key on this same package root, so the audit inherits its anchor from those existing pins rather than introducing a new one.
- The anchor MUST be resolved from a stable signal (e.g. walking parents from the audit module's own location until a `package.json` whose `name` matches loom's published package name is found, mirroring the recipe `capability-probe.md` §(d) uses for the four peer packages), not from `process.cwd()` — using `process.cwd()` would re-introduce the very subdirectory-invocation divergence this sentence is meant to close.
- The `dist/`-outside-`src/` prohibition's "outside the included root by construction" claim should be re-read with this anchor in mind; it does not require a separate edit, but it implicitly inherits the new anchor and the *Generated-output-in-src limitation* soft-surface acknowledgement keeps its current shape.

## Consequence

**Severity:** advisory

Two contributors implementing the audit will both reach for "the package root" by analogy with the surrounding `package.json`-anchored obligations, so the practical risk of divergent walked file sets on a clean checkout is low. The cost is that the *Audit scope* paragraph's "same closed file set for a given source tree" guarantee and the `dist/`-outside-`src/` prohibition rest on an unstated premise, leaving a Windows / non-cwd-from-package-root invocation, a monorepo workspace placement, or a CI step that `cd`s elsewhere unprotected by anything in the spec; the audit fails open (walks an empty or wrong tree, exits green) rather than fail-closed.
