## Problem

BNDR-2 (`id="bndr-2"`) annotates `ambiguous.candidates` with "(`array<string> | null`; the binder may emit it; AJV accepts `null`)". The clause "the binder may emit it" is ambiguous: it can read as *the key is optional* (the binder may include or omit it) or as *the key is always present but its value may be `null` or an array*. The authoritative JSON schema block in the same section resolves this against the optional reading — the `ambiguous` arm lists `candidates` in `"required"` with `"type": ["array", "null"]`, so the key is always present and its value is `null` or an array of strings. The prose pulls against the schema it annotates, and a reader who weights the prose could drop `candidates` from `required` — which BNDR-2 itself calls a breaking change.

## Solution approach

Rewrite the "the binder may emit it" clause in BNDR-2 so it matches the schema's `required` membership and `["array", "null"]` type: state that the `candidates` key is always present in the `ambiguous` arm and that the binder sets its value to `null` when it records no candidates, otherwise to an array of strings. Because the defect is the wording itself, the corrected meaning (always-present key, nullable value) is the target.
