## Problem

The Baseline acquisition block (`#bump-baseline-acquisition`) binds the contributor with a MUST to the specific command `npm pack @earendil-works/pi-coding-agent@<pinned-version>`. The binding contract is behavioural: the diff's "from" side must be the registry-published pinned minor's `dist/*.js` tree, materialised separately so the candidate minor's locally-installed tree does not clobber it. The particular tool used to fetch that tree is incidental, but the command-as-MUST invites a contributor to treat the command rather than the resulting tree as the obligation.

Before that commit the diff baseline was described only abstractly as the loom 1.0 Pi-SDK pin's `dist/*.js` snapshot, with no concrete acquisition mechanism. `ccc6e38` created the **Baseline acquisition** block, introducing in one edit both the `npm pack @...@<pinned-version>` command-as-MUST and the ephemeral / "not retained as an in-repo snapshot" / MUST-NOT-commit rule, while resolving a prior review finding about the unspecified baseline source. Both defects — the command-verb-as-MUST and the unstated fetchability precondition with no fallback — entered together at the block's inception.

## Solution approach

Demote the `npm pack @earendil-works/pi-coding-agent@<pinned-version>`-and-extract command sequence in the Baseline acquisition block (`#bump-baseline-acquisition`) to a non-normative recipe. Rewrite the MUST as the behavioural obligation: the contributor MUST materialise the loom 1.0 Pi-SDK pin's `dist/*.js` tree (the registry-published pinned minor) as the diff "from" side, separate from the candidate minor's locally-installed tree. The existing "rather than `npm install --prefix`" rationale moves with the command sequence into the recipe.
