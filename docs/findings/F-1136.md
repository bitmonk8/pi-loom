## Problem

The *Reassignment* paragraph in `bindings.md` admits the compound forms `+=`, `-=`, `*=`, `/=`, `%=` on `let mut` bindings with the sole rule that "the RHS must be compatible with the binding's declared or inferred type per [Type System — Type compatibility](./type-system.md#type-compatibility)". The page never states whether `a OP= b` desugars to `a = a OP b`, so "RHS" is ambiguous between the literal operand `b` and the synthesised expression `a OP b`. The two readings disagree on legal programs: under the literal reading `count /= 1` is legal for `count: integer`, while under the desugared reading it fires `loom/parse/integer-narrowing` because `integer / integer` produces `number` per `expressions.md` *Other arithmetic*. The divergence reaches every compound form whose operator carries operand-type or widening rules, and silently leaves the in-page `count += 1` example dependent on an unstated rule.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `53f5831` *spec: rewrite query primitive, system prompts, and schema syntax* (2026-05-03)
**History:** The *Bindings and Mutability* section first entered the corpus in `53f5831` already carrying the compound-form list `+=`, `-=`, `*=`, `/=`, `%=` and the single-sentence RHS rule, with no desugaring statement and no delegation to the per-operator type rules; the per-operator paragraphs in `expressions.md` were authored in the same commit but never cross-linked from the compound-form rule. A later edit in `0226b80` *pi-loom plan: resolve "V15c references an undefined compatibility relation"* (2026-05-05) replaced the original phrase "the RHS must type-match the binding's declared or inferred type" with "the RHS must be compatible with the binding's declared or inferred type per [Type System — Type compatibility]" — this swap fixed an unrelated dangling-reference defect (the `V15c` relation) but did not address the desugaring gap, and the "RHS" pronoun stayed equally ambiguous between the literal and synthesised readings. The defect is therefore inherent to the section's original framing, not introduced by a later regression.

## Solution approach

Rewrite the *Reassignment* paragraph in `bindings.md` to make the desugaring `a OP= b ≡ a = a OP b` normative and delegate the operator's operand-type, result-type, widening, and diagnostic behaviour to the relevant `expressions.md` sections (`+` operator, Other arithmetic) applied unchanged to the synthesised expression, with the synthesised result required to be compatible with the binding's type per `type-system.md#type-compatibility`. Clarify that the plain form `a = b` checks `b` directly against the binding's type by the same relation.
