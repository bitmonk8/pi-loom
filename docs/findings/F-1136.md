## Finding

`bindings.md` admits the compound-assignment forms `+=`, `-=`, `*=`, `/=`, `%=` on `let mut` bindings with the sole rule that "the RHS must be compatible with the binding's declared or inferred type per [Type System — Type compatibility](./type-system.md#type-compatibility)". The page never states the desugaring `a OP= b ≡ a = a OP b`, and it never says that the per-operator operand-type rules and diagnostics defined in `expressions.md` (e.g. the `+` / `-` / `*` / `/` / `%` paragraphs) apply unchanged.

The omission is observable, not editorial, because the two readings disagree on legal programs:

- *Literal-RHS reading.* "RHS" means just `b`; `b` is checked for compatibility with the binding's type and that is the whole rule. Under this reading, given `let mut count: integer = 0`, the statement `count /= 1` is legal (the literal `1` is `integer ⊑ integer`), `count += " items"` is rejected only because `string` is not `⊑ integer`, and the per-operator diagnostics of `OP` never enter the picture.
- *Synthesised-RHS reading (desugaring).* "RHS" is the synthesised `a OP b`; the operator-arm rules in `expressions.md` apply unchanged and the result must be compatible with the binding's type. Under this reading, `count /= 1` is rejected as `loom/parse/integer-narrowing` because `/` is normatively `(integer, integer): number` (no integer-division operator in loom 1.0 — see `expressions.md` *Other arithmetic*), so the synthesised RHS has type `number` and `number ⋢ integer`. Similarly `let mut s = "" ; s += " items"` is legal (string concatenation), `let mut n: number = 0 ; n %= 0` produces `NaN` and stays `number`, and the `loom/parse/mixed-plus-operands` diagnostic of `+` applies to `+=`.

Two reasonable implementers will diverge on every program that exercises `/=` against an `integer` binding, on `+=` against a `string` binding, and on any compound form whose `OP` carries operand-type or widening rules. The wording also leaves the `count += 1` example in the page silently dependent on the missing rule.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `53f5831` *spec: rewrite query primitive, system prompts, and schema syntax* (2026-05-03)
**History:** The *Bindings and Mutability* section first entered the corpus in `53f5831` already carrying the compound-form list `+=`, `-=`, `*=`, `/=`, `%=` and the single-sentence RHS rule, with no desugaring statement and no delegation to the per-operator type rules; the per-operator paragraphs in `expressions.md` were authored in the same commit but never cross-linked from the compound-form rule. A later edit in `0226b80` *pi-loom plan: resolve "V15c references an undefined compatibility relation"* (2026-05-05) replaced the original phrase "the RHS must type-match the binding's declared or inferred type" with "the RHS must be compatible with the binding's declared or inferred type per [Type System — Type compatibility]" — this swap fixed an unrelated dangling-reference defect (the `V15c` relation) but did not address the desugaring gap, and the "RHS" pronoun stayed equally ambiguous between the literal and synthesised readings. The defect is therefore inherent to the section's original framing, not introduced by a later regression.

## Solution Space

**Shape:** single

### Recommendation

Edit the *Reassignment* paragraph in `bindings.md` to make the desugaring normative and delegate per-operator behaviour to `expressions.md`. Replace the current single-sentence RHS rule with text along these lines:

> Each compound form `a OP= b` is normatively equivalent to `a = a OP b`: the operator `OP`'s operand-type rules, result-type rules, widening, and diagnostics defined in [Expressions — `+` operator / Other arithmetic](./expressions.md) apply unchanged to the synthesised expression `a OP b`, and the synthesised result must be compatible with `a`'s declared or inferred type per [Type System — Type compatibility](./type-system.md#type-compatibility). The plain form `a = b` checks `b` directly against `a`'s type by the same compatibility relation.

This makes the worked cases observable consequences of rules already on the books rather than requiring `bindings.md` to re-state them:

- `let mut count: integer = 0 ; count /= 1` fires `loom/parse/integer-narrowing` (because `integer / integer` produces `number` per `expressions.md` *Other arithmetic*).
- `let mut s = "" ; s += " items"` is legal as string concatenation (because `string + string` is concatenation per `expressions.md` *`+` operator*).
- `let mut n: number = 0 ; n %= 0` produces `NaN` at runtime and stays `number` (because `% 0` produces `NaN` and does not panic).
- `let mut count: integer = 0 ; count += " x"` fires `loom/parse/mixed-plus-operands` rather than a binding-side type-mismatch code.

Edge cases the implementer must watch: the desugaring is type-checking-only — it must not duplicate side effects of `a` at run-time when `a` is a bare identifier (the only `a` form admissible under loom 1.0's binding-level-only mutability rule, since member/index assignment is `loom/parse/assignment-to-member-or-index`), so the synthesised-form framing matches the actual evaluation order and no second-eval hazard arises. The `V3b` / `V3b-T` leaves need test cases that pin the synthesised-RHS reading; without them a parser that implements only the literal-RHS check passes the current acceptance set.

## Consequence

**Severity:** correctness

Two implementers can ship parsers that disagree on whether `count /= 1` and `s += 1` (for `count: integer`, `s: string`) are accepted, because the page admits both readings and neither contradicts the cited type-compatibility relation. The same ambiguity propagates to every compound form whose operator carries operand-type or widening rules, and silently makes the in-page `count += 1` example dependent on a rule the page does not state.
