## Finding

The `session_shutdown` sub-step 3 in `session-shutdown-semantics.md` arms its bounded await with the call `Clock.setTimeout(SHUTDOWN_AWAIT_CAP_MS, ...)` — delay (a number) first, callback second. This is the inverse of the canonical seam signature published in `host-interfaces-services.md` PIC-12, which declares `setTimeout(fn: () => void, ms: number): TimerHandle` — callback first, delay second. The sibling call example in the same PIC-12 page (the `loop-iter` checkpoint macrotask yield, `setTimeout(fn, 0)`) also follows callback-first order. The corpus therefore carries two mutually contradictory example invocations of the same seam method, in two different files, with no binding statement reconciling them.

The contradiction is sharper than "argument order is asserted only by usage": PIC-12 explicitly marks the member *signatures* as non-normative (per GOV-18 arm (a), "the exact member signatures of this internal DI seam are not themselves binding, while the usage constraints stated on each member ... remain normative"). Argument order is not stated anywhere as a binding usage constraint — it is conveyed only through the signature (non-binding) and the two example call sites, which disagree. An implementer cannot resolve the order from the spec without picking one example over the other.

An implementer who builds the `Clock` seam from the PIC-12 signature (callback-first) and then implements sub-step 3 by copying the `Clock.setTimeout(SHUTDOWN_AWAIT_CAP_MS, ...)` example verbatim will pass a number where a callback is expected and a callback where a delay is expected: the `SHUTDOWN_AWAIT_CAP_MS` cap never fires correctly and the bounded await silently degrades to unbounded (or fires immediately), defeating the absolute-deadline guarantee the sub-step is built around.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** `bc50e69` (2026-05-05, "pi-loom spec: resolve \"H2 Clock seam — watcher debounce, scanPackagesTimeoutMs, RuntimeEvent occurred_at\""); `52219c4` (2026-05-06, "Address all 41 spec-review findings")
**History:** The corpus is git-tracked (1766 commits). `git log -S` traces the divergence to two independently-authored commits one day apart. `bc50e69` authored the canonical callback-first seam signature `setTimeout(fn: () => void, ms: number)` when the `Clock` seam was first specified. The next day, `52219c4` introduced the `session_shutdown` shutdown-await machinery, writing the call site as delay-first `Clock.setTimeout(SHUTDOWN_AWAIT_CAP_MS, ...)`; the diff for that commit adds the delay-first sub-step text without touching the previously-authored signature. Neither commit reconciled the two. Subsequent commits (`f5e89f4` size-cap reorg, `c3de47a` F-1602 relocation) only relocated the already-divergent text into the current topic files; they did not originate the contradiction. The defect is the unreconciled interaction between the two authoring sites, not a regression in any single later commit.

## Solution Space

**Shape:** single

### Recommendation

Make the argument order a binding usage constraint at PIC-12 (since the signature alone is non-normative under GOV-18 arm (a)): add an explicit sentence to the `setTimeout` member bullet pinning the order as `setTimeout(fn, ms)` — callback first, delay second — matching the existing `loop-iter` `setTimeout(fn, 0)` example. Then correct the sub-step 3 call site in `session-shutdown-semantics.md` to the callback-first form (e.g. `Clock.setTimeout(() => …, SHUTDOWN_AWAIT_CAP_MS)`) and link it to the PIC-12 `#clock--fakeclock-interface` anchor so the order is contract-backed rather than example-backed. The two sites must then agree. Watch the diagnostics/test wording that asserts the timeout handle is armed and cleared (`FakeClock.advance` / `clearTimeout`) so it references the corrected order.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one follows the PIC-12 callback-first signature and the `loop-iter` example, the other copies the delay-first sub-step 3 example. If the seam adapter and the call site disagree on order, `SHUTDOWN_AWAIT_CAP_MS` is passed as the callback and the callback as the delay, so the shutdown-await cap silently never fires and the bounded await loses its deadline.
