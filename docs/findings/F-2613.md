## Problem

The *Worked example: depth-6 forced respond at `max_rounds`* in `query-tool-loop.md` (anchor `worked-example-depth-6-forced-respond`) declares itself normative — cited as the fixture for the `RuntimeEvent`-shape conformance test and the typed-query suite, and the canonical PIC-1 exhibit for V1's only non-empty `masked` payload (`["ceiling#2"]`). Two of its fields the spec renders deterministic are left as prose placeholders: `validation_errors[].path: "<JSON Pointer to first too-deep node>"`, fixed at `/deeply/nested/value/a/b` for the pinned payload under `schema-subset.md`'s depth-counting algorithm, and the top-level `message: "<the validation message>"` (in both the `Err(QueryError {...})` and `RuntimeEvent` blocks), whose normative status is unstated. Two reasonable test authors therefore produce divergent conformance oracles, so the vector fails to pin the payload it claims to.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 7598f50 — pi-loom spec: resolve "CIO-6 hard-ceiling co-fire: predicate, test vector, and normative ownership" (2026-05-07, Thomas Andersen)
**History:** The worked example, its `RuntimeEvent` vector, and the `path: "<JSON Pointer to first too-deep node>"` placeholder were all introduced together in 7598f50, which authored the CIO-6 co-fire test vector and placed it in the (then-monolithic) `query.md`. Later commits preserved the placeholder untouched: 6cc4288 (2026-06-05) corrected the payload's labelled depth (7 → 6) and f5e89f4 (2026-06-04, doc size-cap split) moved the section into `query/query-tool-loop.md`, but neither concretised the deterministic fields.

## Solution approach

Replace the `path` placeholder in the `Err(QueryError {...})` block with the concrete JSON Pointer `/deeply/nested/value/a/b` — the first node at depth 6 under `schema-subset.md`'s counting algorithm — confirming the descent semantics name the depth-6 scalar rather than its depth-5 parent before pinning the literal. Resolve the top-level `message` field's normative status in both the `Err` and `RuntimeEvent` blocks, either inlining the asserted string or marking it illustrative-only, so the conformance author knows whether to assert on it. Distinguish the deterministic asserted fields from the genuinely run-dependent ones so the vector's testable surface is unambiguous. Keep the example in `query-tool-loop.md`.
