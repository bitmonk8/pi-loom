## Finding

`docs/spec_topics/future-considerations/surface-extensions.md` holds two of the four top-level future-considerations categories defined in the hub preamble, in sequence:

1. `## Tooling deferrals (no loom 1.0 impact)` — category 1 (items that ship as new tools/commands with no loom 1.0 runtime decision: LSP support, `loom test`, richer telemetry, automated Pi-patch re-import cadence).
2. `## Surface extensions (loom 1.0 leaves a seam)` — category 2 (items that extend a loom 1.0 type/struct/call shape).

The file name, its H1 (`# Surface extensions`), and the hub's Contents link (`docs/spec_topics/future-considerations.md` → `## Contents` → `[Surface extensions](./future-considerations/surface-extensions.md)`) all name only category 2. Nothing at the navigation surface signals that category-1 (Tooling deferrals) content lives inside this file. A reader who follows the hub's explicit four-category framing to locate category 1 has no Contents entry, no filename, and no H1 pointing here; the only thread is the `<a id="tooling-deferrals-no-loom-1-0-impact">` redirect stub (mirrored in the hub just above `## Contents`), which is an anchor, not a discoverable label.

The page is non-normative orientation/future-work material, so the defect is one of discoverability and organisation rather than runtime or contract correctness, but it is real: the canonical name of the file silently drops half its content.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** f5e89f4 — docs: enforce 100KB hard / 30KB recommended size cap on spec set (2026-06-04, Thomas Andersen)
**History:** Before f5e89f4 all four future-considerations categories lived in a single `future-considerations.md` whose name correctly covered every section, so no misname existed. f5e89f4 split the over-limit corpus files into a subdirectory+index layout; it grouped category 1 (Tooling deferrals) and category 2 (Surface extensions) into one child page but named that page `surface-extensions.md` after only category 2, while categories 3–4 went to `model-changes-and-non-goals.md`. The category-1 content has been present under a category-2 name in every commit of this file since its creation; the dual-section content has not changed materially since.

## Solution Space

**Shape:** multiple

### Option A — Rename the combined file to name both categories

**Approach:** Rename `surface-extensions.md` to `tooling-deferrals-and-surface-extensions.md`, change the H1 to match, and retitle the hub Contents link (e.g. "Tooling deferrals and surface extensions"). Keep both sections and their anchors in place.

**Spec edits:**
- Rename the file; update H1.
- Update the hub `## Contents` link text and URL.
- Repoint every `surface-extensions.md` path reference across `docs/` (≈50 occurrences) to the new filename.

**Pros:**
- Single file, no content relocation; section anchors (`tooling-deferrals-no-loom-1-0-impact`, `surface-extensions-loom-1-0-leaves-a-seam`, and their aliases) stay put.
- Both categories visible in the filename and Contents label.

**Cons:**
- High cross-reference churn: all ≈50 path references to `surface-extensions.md` must be repointed, the bulk of which target category-2 content that is not the source of the defect.
- Long, compound filename.

**Risks:** A missed path-reference repoint among the ≈50 sites yields a dangling link.

### Option B — Split Tooling deferrals into its own page

**Approach:** Extract `## Tooling deferrals (no loom 1.0 impact)` (and its anchors) into a new `docs/spec_topics/future-considerations/tooling-deferrals.md`; leave `surface-extensions.md` holding only category 2. Add a separate hub Contents entry for the new page.

**Spec edits:**
- Create `tooling-deferrals.md` with the category-1 section and move the `tooling-deferrals-no-loom-1-0-impact` / `tooling-deferrals-no-v1-impact` anchors into it.
- Add a `## Contents` entry for it in the hub; repoint the hub's redirect-anchor stub.
- Repoint the two cross-references to the tooling-deferrals anchor (`host-prerequisites.md`, `patch-skew-degradation.md`) to the new file. Category-2 references to `surface-extensions.md` are unchanged.

**Pros:**
- One category per file, matching the hub's four-category model.
- Materially lower cross-reference churn — only the ~2 tooling-deferrals anchor references move; the ≈50 `surface-extensions.md` path references stay valid.
- Removes the misleading top-of-file redirect anchor from `surface-extensions.md`.

**Cons:**
- Adds a file and a Contents entry; requires relocating a section and its anchors.

**Risks:** Both anchor alias spellings (`-no-loom-1-0-impact` and `-no-v1-impact`) must move together; missing one breaks a redirect.

### Recommendation

Option B. Splitting matches the hub's one-category-per-file model, touches only the ~2 anchor references plus the hub Contents/stub, and leaves the ≈50 category-2 path references untouched — materially less churn than renaming. Watch the two anchor alias spellings: relocate `tooling-deferrals-no-loom-1-0-impact` and `tooling-deferrals-no-v1-impact` together and repoint both `host-prerequisites.md` and `patch-skew-degradation.md`.

## Consequence

**Severity:** cosmetic

A reader following the hub's four-category framing cannot locate the Tooling-deferrals category by filename, H1, or Contents link, and must discover it incidentally by opening the category-2 file. No normative content, runtime behaviour, or implementer decision is affected.
