## Problem

The `cka-13` coverage-matrix row (`tool-calls.md` / TOOL) is a four-leaf row whose facets are `V14a`, `V14b`, `V14c`, `V14e`; under the multi-leaf-row per-facet `H5f` gate each facet's citing test must name both the `cka-13` token and the facet's own leaf-ID inline. Only `V14b` cites the token in its Tests bullet (`(code-keyed-area token `cka-13`)`); `V14a`/`V14a-T` annotate their Tests bullets with `(TOOL code-keyed area)` prose, `V14c`/`V14c-T` cite no `cka` token in the off-surface-routing bullet, and `V14e`/`V14e-T` cite only the live-carrier-witness annotation with no `cka` token. Separately, the `V14a` fourth bullet (the `host-interfaces-core.md` §*Tool execution from loom code* execute()-envelope bullet) carries a stale `(un-anchored; GOV-22 residue)` label even though the `cka-46` row now anchors that obligation with `V14a` as its closing leaf. An implementer following the three prose-labelled facets must independently consult `coverage-matrix.md` to find the token and may write tests omitting the inline `cka-13`/`cka-46` citation, reddening `H5f`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 7259777 — pi-loom plan: resolve "code-keyed obligation areas lack a canonical citable token" (2026-06-18, Thomas Andersen); 77f2e88 — pi-loom plan: resolve "code-side tool-call envelope-lowering MUSTs unanchored" (2026-06-18, Thomas Andersen)
**History:** The `(TOOL code-keyed area)` prose labels predate the token convention — they were authored in the initial plan build c6a664e (2026-06-10) for `V14a` and propagated to `V14c`/`V14e` when 5a1915d (2026-06-17, "V14a bundles eight independently-testable concerns") peeled those leaves off `V14a`. The labels were not defects until commit 7259777 introduced the `cka-<n>` token convention and the four-leaf `cka-13` row (`V14a, V14b, V14c, V14e`, confirmed via `git show 7259777:docs/plan_topics/coverage-matrix.md`): that commit swept `V14b` (and `V9g`) to cite `cka-13` inline but left the sibling `cka-13` facets `V14a`/`V14c`/`V14e` on their prose labels, concretising the inconsistency. Commit 77f2e88 then added the single-leaf `cka-46` row and `V14a`'s execute()-envelope bullet (`git log -S 'cka-46'` and `git log -S 'un-anchored; GOV-22 residue' -- V14a` both point at 77f2e88) while labelling that bullet `(un-anchored; GOV-22 residue)` rather than citing the token it had just created. Both commits are token-rollout edits that failed to complete the sweep across the cited leaves.

## Solution approach

In `V14a-tool-calls.md` and `V14a-T-tool-calls.md`, replace each `(TOOL code-keyed area)` annotation with `(code-keyed-area token `cka-13`)` matching `V14b`, and replace the execute()-envelope bullet's `(un-anchored; GOV-22 residue)` annotation with `(code-keyed-area token `cka-46`)`. In `V14c-tool-calls-off-surface-routing.md` / `V14c-T-tool-calls-off-surface-routing.md` and `V14e-tool-calls-depth-ceiling.md` / `V14e-T-tool-calls-depth-ceiling.md`, add the `cka-13` token citation to the off-surface-routing and ceiling-#4 Tests bullets respectively, retaining `V14e`'s existing live-carrier-witness annotation. Edit each implementation leaf and its paired `-T` tests leaf so the test sources `H5f` scans carry the citation.

## Solution constraints

- Out of scope: the `coverage-matrix.md` `cka-13` and `cka-46` rows already list the correct closing leaves and need no change.
- The spec files (`tool-calls.md`, `host-interfaces-core.md`) are read-only for this fix.
