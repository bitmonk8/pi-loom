## Problem

`schema-subset.md` §Lowering Algorithm step 2 and the `#schema-slug-collision-posture` it cross-links prescribe the inline-schema dedup test against two different byte-strings: step 2 says two inline schemas collapse to one `$defs` entry only when their *lowered JSON Schema fragments are byte-identical*, while the posture's first bullet defines its byte-equality check on the *canonical-form bytes* (the keys-sorted SHA-256 input from `#canonical-schema-hash` step 2). The two differ for field-order-permuted inline objects: their canonical-form bytes are identical (keys sorted) but their lowered fragments differ, because the emitted `properties`/`required` preserve loom-source declaration order. Two implementers reading step 2's two sentences side-by-side cannot agree whether such objects collapse silently, raise `loom/load/schema-slug-collision`, or land in distinct `__inline_<slug>` entries.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** e21449b — pi-loom spec: resolve "__inline_<hash> hash function is not pinned" (2026-05-04, Thomas Andersen); 2128e5f — pi-loom spec: resolve "Schema-slug collision posture pinned only for registerTool cache" (2026-06-07, Thomas Andersen)
**History:** e21449b introduced the canonical-form-vs-emitted-fragment split by pinning the canonical schema hash (sorted keys) while leaving step 2 phrased as "lower to byte-identical JSON Schema fragments" and adding the closing note that source-level field order does not change the canonical hash. At that point step 2's "byte-identical" predicate already disagreed with the canonical-form behaviour, but no cross-reference existed. 2128e5f then wired step 2 explicitly to the new *Schema-slug collision posture* paragraph — rewriting step 2 to "the lowering pass establishes that identity by the byte-equality check the [Schema-slug collision posture] below mandates" — while writing the posture's check against "canonical-form bytes". That cross-link is what turned the latent inconsistency into a contradiction visible on a single screen: step 2 prescribes one byte-string, the posture it points at prescribes another.

## Solution approach

Rewrite §Lowering Algorithm step 2 so its inline-schema dedup predicate names the canonical-form bytes (per `#canonical-schema-hash` step 2) as the dedup key — and so the `loom/load/schema-slug-collision` trigger it describes is a canonical-form-byte mismatch on a slug match — aligning step 2 with the byte-string the `#schema-slug-collision-posture` first bullet already uses, instead of "lowered JSON Schema fragments are byte-identical".
