## Problem

The `#audit-category-1-3-inventory-join-key` paragraph resolves a category-(1)/(3) member access by exact match between `<member>` and the rightmost dot-separated segment of an inventory entry's `path`, against the entire `SDK_SURFACE_INVENTORY`, with no clause restricting the candidate set to entries of the matching carrier. Consequently a `pi.<member>` access can resolve against a category-(3) entry (a `pi.modelRegistry` access is authorised by the existing `ctx.modelRegistry` / `ExtensionContext.modelRegistry` entry, and symmetrically for `ctx.<member>` against `pi.` entries), defeating the canonical-carrier rule the chapter pins under *Target surface categories*. The symmetric `literal-read/duplicate-rightmost-segment` uniqueness clause is likewise unscoped, so disjoint surfaces that share a member name across carriers — e.g. `pi.shutdown` and `ctx.shutdown` — collide spuriously at literal-read time. The carrier name is already present as the leftmost dot-separated segment of `path` on both sides of the join, but the rule consumes only the rightmost segment.

## Issue introduction

**Verdict:** present-since-introduction

**Introducing commit:** `e5cd514` — *"pi-loom spec: resolve T35 SDK capability inventory closed-set negative-direction audit"* (Sat May 9 17:31:30 2026)

**History:** The *Category-(1)/(3) inventory join key (per-category specialisation)* paragraph was introduced in `e5cd514`, the commit that first coined the audit-resolution semantics. The paragraph entered the corpus already in its current unscoped form — both the rightmost-segment match and the symmetric uniqueness clause omit any carrier predicate on the inventory side from the first line of git history. Subsequent edits to the file (notably the 100KB-cap split in `f5e89f4` that moved the paragraph from `pi-integration-contract.md` into the `pi-integration-contract/audit-resolution.md` child page, and `3e6ce2e` which only retuned the example's "ten" → "nine" factory-probable function-member count) preserved the wording verbatim. The defect is not the residue of a later refactor; the join key was designed without carrier scoping.

## Solution approach

Rewrite the `#audit-category-1-3-inventory-join-key` paragraph so category-(1)/(3) resolution scopes candidate inventory entries by carrier — using the leftmost dot-separated segment of `path` as the carrier discriminator — before applying the rightmost-segment member match: a `pi.<member>` access resolves only against entries whose carrier segment is `pi`, and a `ctx.<member>` access only against entries whose carrier segment is one of `ctx`, `ExtensionContext`, or `ExtensionCommandContext`. Narrow the symmetric rightmost-segment uniqueness clause so it applies within each carrier partition independently rather than across the whole inventory, leaving `pi.shutdown` and `ctx.shutdown` legal as disjoint entries.
