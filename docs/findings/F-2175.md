## Problem

The `loom/load/typed-query-unsupported-provider` gate keys on the loom's "resolved `model:`" and runs at loom-load time, which presupposes the model is known then. But `model:` is optional: when omitted, the loom inherits Pi's session default at invocation time with no load-time registry call, and re-resolves afresh each invocation. For the common inherited-model case the load-time provider gate cannot run — the provider is unknown until dispatch — and the "Provider compatibility for typed queries" contract names no invocation-time substitute. It is therefore unpinned whether the runtime re-evaluates provider compatibility at dispatch and synthesises the same `transport` error, or whether an incapable inherited provider falls through the forced respond turn into the respond-repair / `schema_validation` non-compliance path.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** 82be125 — spec: adopt P1 typed-query two-phase loop (Option A) (2026-05-04, Thomas Andersen); 616f046 — pi-loom spec: resolve "Frontmatter model — loom's lifetime vs per-invocation inheritance" (2026-06-04, Thomas Andersen)
**History:** 82be125 introduced the typed-query two-phase loop and the `loom/load/typed-query-unsupported-provider` gate keyed on the loom's *resolved* `model:`, written as a load-time check. 616f046 later pinned that an absent `model:` inherits Pi's session model at invocation time with "no load-time registry call," and was scoped to the frontmatter model-inheritance semantics (`docs/spec_topics/frontmatter.md` only) without touching the typed-query provider gate. The interaction of the two — a load-time-only provider gate over a model that, in the inherited case, is not resolved until invocation — leaves the inherited-model typed-query disposition unpinned; neither commit reconciled them.

## Solution approach

Clarify the "Provider compatibility for typed queries" contract in `conversation-drive.md` to pin the inherited-`model:` path: state that the load-time `loom/load/typed-query-unsupported-provider` warning does not fire when the provider is unknown at load time, and that the runtime instead evaluates provider compatibility against the inherited model resolved at typed-query dispatch, returning the same `transport` `QueryError` the present-`model:` runtime path returns so the disposition is uniform. Add forward cross-links to this inherited-model disposition from QRY-14 in `query-tool-loop.md` and the typed-query bullet in `implementation-notes.md`, alongside the existing binder-model and provider-compat references.

## Solution constraints

- Out of scope: adding a new model-capability load-time gate for the main `model:`; the binder's strict-capability probe is binder-only, and model-level non-compliance within a supported provider stays on the existing `#pic-typed-query-noncompliance` runtime disposition.
