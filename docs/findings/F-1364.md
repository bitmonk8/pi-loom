## Problem

`SUBS-1` is a live, numbered, executable REQ-ID anchored in `spec_topics/schema-subset.md` §Lowering Algorithm step 3 as a normative union-lowering MUST (primitive-only union → `{ "type": [...] }`; any non-primitive arm → `{ "anyOf": [...] }`; source-order arms with `"null"` last). The **Numbered REQ-IDs (runtime obligations)** table in `coverage-matrix.md` carries no `SUBS` row — the only SUBS entry, `cka-10`, sits under **Code-keyed obligation areas** and cannot close a numbered REQ-ID. No `V5f`/`V5f-T` Tests bullet cites `SUBS-1` inline; the existing bullet covers step 5 (the per-schema sidecar), not step-3 union-lowering. The obligation is therefore both unmapped and uncited, so the union-lowering split and canonical arm ordering ship unverified.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `6892aa3` (2026-06-20)
**History:** `SUBS-1` first appeared as a numbered REQ-ID in `6892aa3` ("fix F-0786 — pin primitive-union/nullability lowered form (SUBS-1) for canonical-hash byte-identity"), which edited only `docs/spec_topics/schema-subset.md` (and removed one line in `docs/spec-plan-review.md`); it touched neither `coverage-matrix.md` nor any V5 leaf. `git log -S "SUBS-1" -- docs/plan_topics/coverage-matrix.md` is empty — the Numbered REQ-IDs table has never carried a SUBS row — and `git log -S "SUBS-1"` over `V5d`/`V5f`/`V5f-T` is empty — no leaf has ever cited it. The mapping-and-citing-test gap has therefore existed since `SUBS-1`'s inception, not been introduced by a later edit.

## Solution approach

Add a `SUBS-1` → `V5f` row to the **Numbered REQ-IDs (runtime obligations)** table in `coverage-matrix.md`, leaving the existing `cka-10` code-keyed row unchanged. Add a Tests bullet to `V5f-subset-lowering-hash.md` and its paired `V5f-T-subset-lowering-hash.md` that cites `SUBS-1` inline and asserts the union-lowering split (primitive-only → `{ "type": [...] }`; any non-primitive arm → `{ "anyOf": [...] }`; source-order arms with `"null"` last, `null` treated as primitive). The spec's vectors are available as references (`string | number` → `{ "type": ["string", "number"] }`; `string | null` → `{ "type": ["string", "null"] }`; `string | Author` → anyOf).

## Solution constraints

- `schema-subset.md` and the rest of the spec are read-only for this fix.
- Out of scope: the `cka-10` code-keyed row, which continues to close `schema-subset.md`'s `loom/*` diagnostic codes.
