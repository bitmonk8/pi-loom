## Finding

H4a's `Adds` field carries a standalone second paragraph describing the manual real-host smoke governance protocol (its two triggers, named owners, activation window, live-host precondition, pass/fail criterion, and merge-blocker / revert path). The paragraph's purpose is the opposite of an `Adds` entry: it states that the protocol is **not** an H4a deliverable, is addressed to the bump/merge contributor and the H6a release-gate auditor rather than the H4a implementer, and lives in `real-host-smoke-gate.md`. It is a scope-exclusion / cross-reference note sitting in the field reserved for what the leaf produces.

The note carries no normative content for the H4a implementer and duplicates a pointer that already appears twice elsewhere: in H4a's own `Tests` third bullet ("…it is defined as a release-gate / version-bump governance protocol in `real-host-smoke-gate.md`") and in `plan.md` §Release gate ("The **manual real-host smoke governance protocol**… is a cross-cutting governance note, not a leaf: see `real-host-smoke-gate.md`"). Because it leads with "The manual real-host smoke governance protocol…" in an `Adds` field, it reads as though the protocol is part of H4a's output until the qualifying clause arrives mid-sentence.

## Solution Space

**Shape:** single

### Recommendation

Remove the standalone scope-exclusion paragraph from H4a's `Adds` field (the paragraph beginning "The manual real-host smoke governance protocol that gates a double-vs-real divergence…" and ending "…consumes this leaf's harness only as a downstream reader."). The same exclusion and pointer to `real-host-smoke-gate.md` already survive in H4a's `Tests` third bullet and in `plan.md` §Release gate, so no information is lost.

If an explicit boundary statement is still wanted inside H4a, attach it to the `Deps` field as a scope-exclusion note rather than to `Adds` — `Deps`/boundary text is where "this leaf does not own X; X lives in Y" belongs. The `real-host-smoke-gate.md` reference is read-only for this fix; do not edit the governance note or `plan.md` §Release gate.

## Consequence

**Severity:** cosmetic

A scope-exclusion note in the `Adds` field momentarily presents the real-host smoke protocol as an H4a deliverable before the qualifying clause corrects it. No implementer is driven to wrong code — the protocol is correctly disclaimed and the same pointer survives in H4a's Tests bullet — so the defect is organisational, not behavioural.
