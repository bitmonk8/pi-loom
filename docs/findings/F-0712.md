## Finding

`host-prerequisites.md` is the dedicated home for the "loom 1.0 treats it as a presupposition" framing: it collects the four degraded-state-branch clauses (a)–(d), the model-registry-population presupposition, the settings-write-back-preservation presupposition, and the operator-always-present presupposition, and each sibling presupposition is cross-named (("in the same sense as the degraded-state prerequisites … and the settings write-back key preservation prerequisite above")) so a reader of that page sees the closed set.

But a sizeable set of additional presuppositions using the very same framing — and routed to the same per-bump editorial-review checklist in `version-bump-step2.md` — live only on the interface page that defines the surface they constrain, with no inbound pointer from `host-prerequisites.md`:

- `host-interfaces-core.md`: `#messages-chronological-order-presupposition`, `#messages-leading-user-message-presupposition`, and the six-clause `#pi-slash-handler-promise-lifecycle-presupposition` (clauses (i)–(vi)).
- `host-interfaces-services.md`: `#cwd-project-root-presupposition`.
- `registration-steps.md`: `#getcommands-completeness-presupposition`, `#register-tool-post-startup-presupposition`.
- `conversation-drive.md`: `#driven-turn-commit-ordering-presupposition`, `#user-visible-streaming-ordering-presupposition`, `#complete-forced-tool-presupposition`, `#complete-retry-and-cancellation-presupposition`.
- `subagent.md`: `#dispose-mid-abort-presupposition`.
- `provider-error-mapping.md`: `#provider-overflow-wording-presupposition`.
- `runtime-event-channel.md`: `#custom-message-context-entry-presupposition`.

An auditor enumerating loom's unpinned host-behavioural presuppositions from `host-prerequisites.md` sees only the seven it owns; the rest are reachable only by walking out from `version-bump-step2.md`'s items (g)/(h)/(i)/(j)/(k)/(l)/(m)/(o)/(r)/(s)/(u)/(v)/(w)/(y)/(z)/(aa)/(ab)/(ac)/(ad)/(ag)/(ak) or by reading the topic pages directly. The dedicated collection page silently undercounts.

## Issue introduction

**Verdict:** present-since-inception (for the cluster) / multi-commit-accretion (for the specific gap)
**Introducing commits:** the establishment of `host-prerequisites.md`'s "treats it as a presupposition" framing predates the spec's current size-cap split (`f5e89f4` of 2026-06-04 mechanically split `pi-integration-contract.md` into the topic-page tree the finding observes), but the scattered presuppositions accumulated across multiple later resolve commits — notably `aebec4e` (2026-06-04, added `#messages-chronological-order-presupposition`), `cb3a2f2` (2026-06-04, widened it to surface (2) and added the subagent surface), `ce4e476` (2026-06-06, added `#messages-leading-user-message-presupposition`), `e89e42e` (2026-05-17, established `#pi-slash-handler-promise-lifecycle-presupposition` on the pre-split file), `7f07a96` (2026-06-05, added presupposition (v)), `fdc5115` (2026-06-06, added presupposition (vi)), and similar commits that authored `#cwd-project-root-presupposition`, `#provider-overflow-wording-presupposition`, `#complete-*`, `#dispose-mid-abort-presupposition`, `#register-tool-post-startup-presupposition`, `#getcommands-completeness-presupposition`, and `#custom-message-context-entry-presupposition` on their respective topic pages.
**History:** Each presupposition was authored on the topic page that defines the SDK surface it constrains (the natural local home) and was reflexively wired into `version-bump-step2.md`'s checklist as the per-bump detection gate. `host-prerequisites.md`'s own presupposition cluster (degraded-state (b)–(d), model-registry-population, settings-write-back, operator-always-present) grew on the same timeline (`79fa3b9`, `66da202`, `bfef26e`, `6caacfe`, `60fba43` — all 2026-06-04/05/07), and the "in the same sense as the … prerequisite above" chain was maintained inside that cluster but never extended to back-reference the topic-page-resident siblings. `803106d` (2026-06-07, "Pi behavioural presuppositions lack authoritative behavioural pointers") added *Authoritative source* / known-fragile-evidence sentences to most of these presuppositions, confirming that the corpus-wide pattern was recognised, but did not address the inbound-pointer gap. The defect is therefore an accretion gap: each commit added a presupposition correctly on its owning page and correctly to the version-bump checklist, but the dedicated collection page on `host-prerequisites.md` was never updated to index the topic-page-resident set.

## Solution Space

**Shape:** multiple

### Option A — Index paragraph on `host-prerequisites.md`

Approach: append a single paragraph to `host-prerequisites.md` (after `#operator-always-present-presupposition`) that enumerates the topic-page-resident presuppositions by anchor link, grouped by owning topic page, and names `version-bump-step2.md` as the per-item per-bump checklist gate. Each topic-page presupposition remains where it is and continues to carry its existing "in the same sense as the degraded-state prerequisites …" framing.

Spec edits:
- `host-prerequisites.md`: one new paragraph titled something like *Topic-page-resident behavioural presuppositions (index)* with the bullet list of the ~12 anchors enumerated under **Finding** above; one sentence stating that the per-bump editorial-review checklist in `version-bump-step2.md` is the detection gate for all of them.
- Topic pages: no edits required; the existing "treats it as a presupposition" framing on each one remains the normative declaration.

Pros: each presupposition stays adjacent to the SDK-surface declaration it constrains (a reader of the interface page sees the constraint inline); minimal edit surface; the index paragraph can be extended in lock-step with future presupposition additions by the same author who is already editing `host-prerequisites.md` for the "in the same sense as" chain; no anchor renames, no inbound-link breakage.

Cons: `host-prerequisites.md` becomes an index + a normative owner of seven, rather than the normative owner of all of them — a reader still has to follow a link to read each presupposition's text.

Risks: the index must be kept in lock-step with new presuppositions; the next presupposition author may forget to extend the index. Mitigated by adding the index update to the closing-paragraph obligation in `version-bump-step2.md` ("Any future presupposition added to this page … MUST be added to this checklist in the same edit") — extend that sentence to require an `host-prerequisites.md` index update in the same edit.

### Option B — Relocate every behavioural presupposition into `host-prerequisites.md`

Approach: move the full text of each topic-page-resident presupposition into `host-prerequisites.md`, leaving behind a one-sentence "owned by `host-prerequisites.md` §<anchor>" pointer on the original page. The anchors move with the text; every inbound link in `version-bump-step2.md` and the topic pages updates to the new location.

Spec edits:
- `host-prerequisites.md`: absorb ~12 paragraphs in addition to its current seven.
- Each topic page: replace the in-place presupposition with a one-sentence forward pointer.
- `version-bump-step2.md`: retarget every item that currently links to a topic-page anchor (items (g), (h), (i), (j), (k), (l), (m), (o), (r), (u), (v), (w), (y), (z), (aa), (ab), (ac), (ad), (ag), (ak)) to the new `host-prerequisites.md` anchor.
- Plan leaves and `coverage-matrix.md`: scan and retarget any link that hit the old anchor.

Pros: a single page is the sole normative owner; an auditor reads one file end-to-end.

Cons: a reader of the interface page (the natural entry point for "what does loom presuppose about `AgentSession.dispose()`?") sees only a pointer rather than the constraint; `host-prerequisites.md` grows substantially (it is already against the 30 KB recommended cap per `f5e89f4`); the edit surface is N anchors × all referrers, much larger than Option A; the "in the same sense as the degraded-state prerequisites …" cross-naming chain becomes long and brittle.

Risks: anchor-rename breakage across the corpus (plan leaves, coverage matrix, sibling spec pages); risk of pushing `host-prerequisites.md` past the file-size cap; the relocated paragraphs cite their interface page heavily ("the binder's `complete()` call", "the spawn block above", "the watcher's debounce above") — every such forward reference becomes a longer indirect citation rather than the in-page "above" / "below" the current author wrote.

### Recommendation

Option A. The topic-page presuppositions earn their location by being declarative consequences of the SDK surface declared on that page; relocating them is mostly an editorial cost with no observable benefit (the version-bump checklist is already the mechanical gate). The index paragraph closes the discoverability gap at one edit site, preserves the local readability the current placement buys, and avoids enlarging an already file-cap-pressed `host-prerequisites.md`. The index MUST list every presupposition enumerated under **Finding** above (twelve anchors across seven topic pages). Update `version-bump-step2.md`'s closing "Any future presupposition added to this page …" sentence in the same edit to require an index update on `host-prerequisites.md` whenever a new topic-page presupposition is authored.

Edge cases for the implementer: keep the index grouped by owning topic page (not by checklist letter) so a reader scanning by SDK surface finds them quickly; do not duplicate the existing seven `host-prerequisites.md`-owned presuppositions in the index (they are already visible on the page); make each bullet `[<owning page>](./<path>)` `§` `[<presupposition name>](./<path>#<anchor>)` so a follow-up rename of either side is grep-detectable.

## Consequence

**Severity:** advisory

An auditor who reads only `host-prerequisites.md` misses roughly two-thirds of loom's unpinned host-behavioural presuppositions, weakening the page's stated role as the dedicated collection. The version-bump editorial-review checklist still reaches every item, so no presupposition is silently dropped at bump time; the gap is in spec discoverability, not in the mechanical gate.
