## Finding

The clause governing how the handler-captured `event.reason` string reaches each per-invocation `finally` is currently phrased as two flat prohibitions:

> … the `finally` MUST NOT use a runtime singleton (forbidden by the no-globals rule) or the `AbortSignal.reason` payload (Pi-side shape not pinned).

Both halves prohibit specific mechanisms (a runtime singleton; the `AbortSignal.reason` payload). Neither states the underlying observable contract — that the value each invocation's `finally` reads via `entry.shutdownReason` is exclusively the value sub-step 2 of *that invocation's* governing `session_shutdown` handler stamped onto its own entry, with no cross-contamination from any prior, concurrent, or re-entrant shutdown.

Without an observable criterion, the conformance surface is hollow: an implementation that smuggles the captured string through a process-wide variable but happens to overwrite it correctly on every shutdown is mechanism-non-conformant yet behaviourally indistinguishable from a conformant one (and vice versa — a conformant per-entry channel could be erroneously failed by a fixture that probes for the *absence* of a singleton rather than for the per-entry isolation property the spec actually cares about). The downstream `V9h-T` / `V9g-T` test inventories carry no assertion for this property, so the gap is not closed elsewhere.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commits:** `75a9626` (2026-05-09) — *pi-loom spec: resolve "SessionShutdownEvent.reason closed set has no build-time pin against the SDK type"*
**History:** The `shutdownReason` per-entry channel and the two-prohibition sentence were introduced together in `75a9626`. Before that commit the file held no `shutdownReason` field on `ActiveInvocationRegistry` and no captured-value delivery rule of any kind — the per-invocation `finally` of the **Per-invocation operator visibility (clean-cancel path)** rule was simply assumed to have the reason string in scope. The fix that added the delivery channel also introduced the mechanism-only framing; subsequent edits (`aa2f9d7`, `a7d2a80`, `c326bfa`, `f5e89f4`) extended adjacent rules (coercion-throw fallback, literals-shape sub-cases, page splits) but did not revisit the prohibition's observability shape. The defect therefore entered the corpus as a side-effect of resolving a different finding, in a single commit, and has been stable since.

## Solution Space

**Shape:** single

### Recommendation

Reframe the sentence as an observable per-entry isolation requirement on `entry.shutdownReason`, with the two existing mechanism prohibitions demoted to a non-normative implementation note (both are already forbidden by other rules — process-globals by the CLAUDE.md / DI-routing rule, and `AbortSignal.reason` by the absence of a Pi-pinned shape — so the prohibitions carry no independent normative weight once the observable is in place).

Concretely, replace the prohibition sentence with an observable along these lines:

> The `finally` MUST read the captured reason exclusively from `entry.shutdownReason` of the same `ActiveInvocationRegistry` entry whose `finally` is running. The value MUST be the string sub-step 2 of the governing `session_shutdown` handler stamped onto that entry (one of the four captured-value cases enumerated above); no other write path to `entry.shutdownReason` exists, and the value MUST NOT be sourced from any process-, extension-, or handler-shared cell whose lifetime exceeds the single registry entry.
>
> *Conformance witness.* A fixture in which two `session_shutdown` events deliver distinct `event.reason` values against the same extension instance — whether re-entrant within one in-flight teardown (per host-prerequisites presupposition (b)) or sequentially after a prior session-only handler has returned — and in which both shutdowns reach at least one in-flight `ActiveInvocationRegistry` entry MUST observe: every emitted `loom/runtime/cancelled-by-session-shutdown` row's `details.event.reason` matches the `event.reason` of the shutdown whose sub-step 2 stamped that specific entry, with no row carrying the other shutdown's reason. (`Implementation note: a runtime singleton or an AbortSignal.reason-payload channel could not satisfy this fixture; the per-entry stamp channel is what does.`)

Pair the spec change with a `V9h-T` test entry asserting the witness, and a `V9h` ships-when clause covering it. Implementer edge cases to surface:

- The witness scopes to *the same extension instance* — the cross-rebind case is governed by host-prerequisites clause (a) and is out of scope here.
- The residual-gap path from `session-only-degraded-state.md` (`substep-2-stamp-throw-residual-gap`) where the stamp threw and the field carries the pre-init `"<unreadable>"` sentinel is already a witness arm of the captured-value enumeration — call it out explicitly so the test does not falsely treat that sentinel as a cross-contamination failure.

## Consequence

**Severity:** advisory

Two reasonable implementers can both satisfy the prohibitions as written (avoid a singleton; avoid `AbortSignal.reason`) and still disagree on what cross-shutdown isolation guarantees their `entry.shutdownReason` actually delivers — e.g. on whether stamping is permitted to be deferred past `loomAbort.abort()`, on whether a second `session_shutdown` may re-stamp an already-stamped entry, or on whether a shared mutable cell behind a closure (technically not a "singleton") counts. The current contract resolves none of these by inspection; the test suite cannot witness the property the prohibitions exist to protect.
