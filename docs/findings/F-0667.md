## Problem

System-note rendering rule 4 ("Empty model-supplied content") classifies two cases as malformed envelopes: a `message` empty after rule-1 stripping, and a `candidates` array whose every entry is empty after stripping. A binder response in which some `candidates[i]` are empty post-strip and others are not falls in neither bucket — the rule is silent on whether such a partial-empty array is malformed. Because the malformed-vs-ambiguous classification still determines which failure-mode row fires and whether the malformed-envelope retry budget is engaged, two conforming implementations can route the same response to different failure rows with different retry behaviour, contradicting BNDR-7's reproduce-exactly contract. The all-entries-empty sub-clause is itself orphan-shaped: rule 5 forbids loom 1.0 from surfacing `candidates`, so it gates classification on content the runtime never reads.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `989b93e` — *pi-loom spec: resolve "needs_info / ambiguous message strings flow into system notes without bounds"* (2026-05-04)
**History:** Rule 4 was authored in `989b93e` with its current two-case wording ("a `message` that is empty…" and "a `candidates` array whose every entry is empty after stripping"); the partial-empty case has been unaddressed since that introduction. Commit `0fa81fa` (2026-05-05, *Decisions #5, #6*) subsequently rewrote rule 5 to defer all `candidates` rendering to a future revision, which left rule 4's candidates sub-clause behaviourally moot for user-visible output but did not retract or refine it. No later commit touched the rule-4 candidates branch (`git log -S 'every entry is empty'` returns only the introducing commit `989b93e` and the spec-split commit `f5e89f4`, which only relocated the prose). The gap is therefore original to rule 4's authoring, not a regression from a later edit.

## Solution approach

Delete rule 4's `candidates`-array malformedness sentence in `docs/spec_topics/binder/defaulting-system-note-echo.md`'s System-note rendering section, leaving rule 4 a single predicate on `message`. Clarify rule 5 that the malformed-vs-ambiguous classification does not depend on `candidates` content in loom 1.0, deferring per-entry malformedness rules to the future binder-refinement-loop revision that re-introduces `candidates` rendering.
