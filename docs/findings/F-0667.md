## Finding

System-note rendering rule 4 ("Empty model-supplied content") classifies two cases as malformed envelopes: a `message` that is empty after rule-1 stripping, and a `candidates` array "whose every entry is empty after stripping." A binder response in which some `candidates[i]` are empty post-strip and others are not falls in neither bucket — the rule is silent on whether such a partial-empty array is malformed, whether the empty entries should be filtered, or whether the array should pass through as-is.

The gap matters even though rule 5 declines to render `candidates` on the user-facing note in loom 1.0: the malformed-vs-ambiguous classification still determines which failure-mode row fires (`argument binding failed — could not parse arguments` vs `ambiguous arguments — <model's message>`) and whether the malformed-envelope per-class retry budget is engaged at all. Two conforming implementations can therefore route the same binder response to two different failure rows with two different retry behaviours, contradicting the BNDR-7 reproduce-exactly contract for the failure-class taxonomy.

A separate but related observation: because rule 5 forbids loom 1.0 from surfacing `candidates`, the existing all-entries-empty sub-clause in rule 4 is itself orphan-shaped — it gates a malformed classification on content the runtime never reads. The fix for the partial-empty gap should either close the orphan as well or explicitly justify keeping it.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `989b93e` — *pi-loom spec: resolve "needs_info / ambiguous message strings flow into system notes without bounds"* (2026-05-04)
**History:** Rule 4 was authored in `989b93e` with its current two-case wording ("a `message` that is empty…" and "a `candidates` array whose every entry is empty after stripping"); the partial-empty case has been unaddressed since that introduction. Commit `0fa81fa` (2026-05-05, *Decisions #5, #6*) subsequently rewrote rule 5 to defer all `candidates` rendering to a future revision, which left rule 4's candidates sub-clause behaviourally moot for user-visible output but did not retract or refine it. No later commit touched the rule-4 candidates branch (`git log -S 'every entry is empty'` returns only the introducing commit `989b93e` and the spec-split commit `f5e89f4`, which only relocated the prose). The gap is therefore original to rule 4's authoring, not a regression from a later edit.

## Solution Space

**Shape:** multiple

### Option A — Retire the candidates sub-clause from rule 4

**Approach.** Delete the second sentence of rule 4 ("The same applies to a `candidates` array whose every entry is empty after stripping.") entirely. Rule 5 already forbids loom 1.0 from surfacing `candidates`, so rule 4's malformed-vs-ambiguous gating on candidates content has no user-visible effect; removing it leaves a single explicit rule (only `message` content drives malformed classification) and aligns the rule with the loom 1.0 "don't render `candidates`" stance.

**Spec edits.**
- In `docs/spec_topics/binder/defaulting-system-note-echo.md`, rule 4: drop the trailing sentence. The remaining rule reads as: "A `message` that is empty after rule 1's stripping — the binder returned only whitespace — is treated as a malformed envelope, not as an empty note: surface via the malformed-envelope row in the failure-modes table."
- Add a one-clause note to rule 5 that the malformed-vs-ambiguous classification does not depend on `candidates` content in loom 1.0 (since the field is not surfaced); the future binder-refinement-loop revision that re-introduces rendering will own the per-entry malformedness rules at that time.

**Pros.**
- Removes a behavioural rule the loom 1.0 runtime never observes through user-visible output.
- Eliminates the partial-empty ambiguity by removing the entire branch.
- Reduces the spec surface that V11e tests must cover by one rule.

**Cons.**
- Loses the schema-content sanity check that all-empty candidates is structurally degenerate; a misbehaving binder can return `candidates: ["", "", ""]` and the envelope proceeds as ambiguous.
- The deferred future revision will need to re-author this rule when it re-introduces rendering.

**Risks.**
- A future plan leaf that re-introduces candidates rendering must remember to re-specify malformedness; carries a single deferred-design hand-off.

### Option B — Tighten to per-entry classification

**Approach.** Reword rule 4's candidates clause to fire on any empty entry, mirroring how rule 1 treats each `candidates[i]` independently: "A `candidates` array containing any entry that is empty after rule 1's stripping is treated as a malformed envelope." This makes the classification deterministic across all empty patterns (none / some / all empty) and keeps the schema-content sanity check.

**Spec edits.**
- In `docs/spec_topics/binder/defaulting-system-note-echo.md`, rule 4: replace "whose every entry is empty after stripping" with "containing any entry empty after stripping".
- Optionally add a reference-vector row in the rule's normative-rendering paragraph to lock down the partial-empty case.

**Pros.**
- Closes the gap with the strictest, most testable predicate.
- Preserves the existing structural sanity check on candidates content.
- Symmetric with how rule 1 already treats each candidate entry individually.

**Cons.**
- Tightens the binder's room to manoeuvre (one stray empty entry rejects the whole envelope), while loom 1.0 never renders candidates anyway — so the tightening protects no user-visible behaviour.
- Keeps a malformed-classification branch that depends on data the user-facing note never reads, leaving the rule-4 / rule-5 tension intact.

**Risks.**
- A future revision that re-introduces `candidates` rendering must decide whether the "any empty entry → malformed" rule is still the right floor or whether per-entry filtering is preferable; the decision is forced earlier than necessary.

### Recommendation

Option A. Rule 5's loom 1.0 deferral makes the entire candidates-content sanity check in rule 4 behaviourally invisible to users; removing it both closes the partial-empty gap and retires the orphan, leaving rule 4 as a single tight predicate on `message`. The future binder-refinement-loop revision (currently deferred per the seam note in `binder-bypass-and-envelope.md`) will re-author the rule under whatever rendering model it picks. The V11e test that previously asserted all-empty-candidates → malformed-envelope is dropped; V11e-T continues to assert the empty-`message` case.

Edge cases the implementer must watch: the envelope schema (BNDR-2) still accepts `candidates: array<string> | null` and does not constrain element content, so a binder returning `candidates: ["", "", ""]` produces a structurally valid `ambiguous` envelope that proceeds to the failure-mode `ambiguous` row with only `<model's message>` rendered — this is the intended loom 1.0 behaviour after the edit.

## Consequence

**Severity:** correctness

Two implementers reading rule 4 today can disagree on whether `candidates: ["x", ""]` is malformed: the literal text reserves "malformed" for fully-empty arrays, but a defensive reader can extend the empty-content protection per entry. The two readings produce different failure-row outputs and different retry-budget behaviour for the same binder response, breaking the BNDR-7 reproduce-exactly contract on a code path V11e's tests are required to lock down.
