## Problem

The final "Not inherited from the caller" row of the **Subagent state-isolation matrix** in `docs/spec_topics/pi-integration-contract/subagent.md` cites a backticked `withActiveTools` identifier that corresponds to nothing in the spec vocabulary: it is not a Pi SDK member (the SDK surface at the loom 1.0 pin exposes only `getActiveTools` / `setActiveTools`), not a glossary term, not an anchor, and it appears in no other cell or topic file. The mechanism the row actually points at is the PIC-17 active-set snapshot/restore protocol defined in `tool-registration-lifetime.md`. An implementer reading the cell must either guess the row is a loose reference to PIC-17 or hunt for a helper that does not exist.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `52219c4` ("Address all 41 spec-review findings", 2026-05-06)
**History:** `git log -S "withActiveTools" -- docs/spec_topics/ spec_topics/ spec.md docs/spec.md` returns exactly two commits on `main`: `52219c4` (the spec-review fix sweep that first authored the **Subagent state-isolation matrix** table, with all six "Not inherited" rows present in the introducing diff) and `f5e89f4` ("docs: enforce 100KB hard / 30KB recommended size cap on spec set", 2026-06-04), which only relocated the matrix from `spec_topics/pi-integration-contract.md` into the new `docs/spec_topics/pi-integration-contract/subagent.md` shard without altering the row text. The phantom identifier is therefore an **authoring slip in the matrix's own genesis commit**: the introducing commit's body describes the bullet that produced the matrix as "Subagent state-isolation matrix (3-column)" under "PIC additions", with no separate sub-bullet for a `withActiveTools` wrapper, indicating the name was coined at the moment the row was written rather than imported from any pre-existing spec or plan vocabulary. (A plan-side `PiExtensionAPI.withActiveTools(set, fn)` helper was introduced two days earlier in `27ec53d` "Resolve Cluster-A: per-mode tool registration" as the runtime's prompt-mode snapshot/restore wrapper, and was carried in the plan corpus until `657ee76` wiped the plan; the matrix row was likely written with that internal-helper name in mind, but no current spec or plan file defines it, so the back-tick stranded the name.)

## Solution approach

Rewrite the offending matrix cell to drop `withActiveTools` and use the page's established terminology — the "active-set snapshot/restore" scope — and replace the bare **Tool-registration lifetime and visibility** reference with a forward-link to the owning `#pic-17` anchor in `tool-registration-lifetime.md`. The name `withActiveTools` should disappear from the corpus rather than be retroactively defined.

## Solution constraints

- Do not introduce a glossary entry or any new defining site for `withActiveTools`; the phantom name is to be removed, not legitimised.
