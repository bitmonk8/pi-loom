## Finding

The four per-site swallowing-handler leaves — `V9o`, `V13f`, `V14f`, `V15h` (and their `-T` partners) — carry a byte-identical "Swallowing-handler attachment at this site" Tests bullet that asserts two things: (1) the abandonable Promise "attaches its swallowing handler at the Promise-construction site (before the first microtask boundary)", and (2) "a build that bypasses the substrate reddens this leaf's tests". The only test vector the bullet (and the impl-leaf Ships-when) actually describes lands **a late settlement** via the `Checkpoint` seam *after* the checkpoint has surfaced `cause: "cancelled"`, and asserts three-channel suppression (no `unhandledRejection`, no second `RuntimeEvent`, no diagnostic).

A late-settlement-after-cancellation vector only proves a handler is attached *by the time the late settlement lands* — it does not witness attachment at the construction site before the first microtask boundary. The spec is explicit that lazy attachment is the failure mode being guarded against: "a `.catch(() => {})` registered lazily after cancellation fires will not catch a rejection that has already been queued for `unhandledRejection`" ([cancellation.md — *Race semantics*](../../docs/spec_topics/cancellation.md)). A build that attaches the handler lazily — anytime after construction but before the late settlement arrives — passes all three channels green yet violates the spec MUST ("The handler MUST be attached at the same site that constructs the Promise, before the first microtask boundary"). The timing claim is therefore unwitnessed by the planned test.

Separately, the suppression assertions are purely outcome-based, so the clause "a build that bypasses the substrate reddens this leaf's tests" over-claims: a build that bypasses the `Checkpoint`-seam substrate but reproduces identical three-channel suppression by other means is not detected by an outcome-only test. Finally, in `V15h`/`V15h-T` the bare phrase "the substrate" is ambiguous between the swallowing-handler substrate and the `Checkpoint`-seam substrate (both are named in the leaf's Adds), and "bypasses" is left undefined.

## Issue introduction

**Verdict:** introduced by a single commit
**Introducing commits:** `75a9bcd` (2026-06-11, "pi-loom plan: resolve \"V17a substrate-level suppression test claims four-site coverage\"")
**History:** The over-claiming bullet — the construction-site / first-microtask attachment assertion, the single late-settlement-only test vector, and the "a build that bypasses the substrate reddens this leaf's tests" clause — entered the corpus in `75a9bcd`, the plan-review fix that distributed `V17a`'s four-site swallowing-handler coverage onto the then-host leaves `V14a`/`V13c`/`V15a`/`V9i` (`git log -S 'bypasses the substrate'` / `-S 'before the first microtask boundary'` both bottom out at `75a9bcd`; no earlier commit contains either string). The wording was subsequently propagated verbatim into the dedicated per-site leaves when they were carved out — `V13f`/`V14f`/`V9o` via `6d7661d` (2026-06-17) and `V15h` via `323d361` (2026-06-18) — with no change to the asserted property. The defect is thus a single-origin wording choice, not an interaction between independent edits; the later split commits only relocated it.

## Solution Space

**Shape:** multiple

### Option A — Add an early-rejection vector that witnesses construction-site attachment

**Approach.** Keep the construction-site / first-microtask attachment as an asserted property, and add a test vector that actually witnesses it: construct the abandonable Promise, reject it within the first microtask boundary (before any lazy-attach point), and assert no `unhandledRejection` fires. A construction-site build catches the rejection (pass); a lazily-attaching build leaves the rejection queued for `unhandledRejection` (red), per the spec's own failure-mode description.

**Plan edits.** In each of the eight leaves, add an early-rejection vector to the "Swallowing-handler attachment at this site" Tests bullet (and reflect it in the impl-leaf Ships-when) alongside the existing late-settlement vector. In all eight, drop or relabel the unfalsifiable clause "a build that bypasses the substrate reddens this leaf's tests" — an outcome-only assertion cannot witness substrate bypass; state instead the property the vectors deliver (handler attached before the abandoned Promise can settle; late settlement suppressed on all three side channels). In `V15h`/`V15h-T`, replace the bare "the substrate" with the explicit referent (the `Checkpoint`-seam swallowing-handler substrate).

**Spec edits.** None.

**Pros.** Preserves verification of a genuine spec MUST (construction-site / first-microtask attachment). Uses the `Checkpoint` seam the spec already designates as the test substrate. Closes the lazy-attach gap.

**Cons.** Adds a second test vector to each of the four sites.

**Risks.** Implementer must reject within the first microtask without an intervening `await` that would itself create an attach point; the spec's lazy-`.catch` note is the reference for getting the timing right.

### Option B — Soften the prose to the property the late-settlement vector delivers

**Approach.** Accept the single late-settlement vector as the test and rewrite the bullet to claim only what that vector witnesses, dropping the construction-site / first-microtask / bypasses-the-substrate language entirely.

**Plan edits.** In all eight leaves, replace the bullet's claim with the delivered property: "a handler is attached before the abandoned Promise can settle; a late settlement is suppressed on all three side channels (no `unhandledRejection`, no second `RuntimeEvent`, no diagnostic of any severity)." Drop the "construction site (before the first microtask boundary)" and "a build that bypasses the substrate reddens this leaf's tests" phrasing from the Tests bullet, the impl-leaf Ships-when, and the `V15h` Adds.

**Spec edits.** None.

**Pros.** Smallest diff; the test wording becomes honest with no new vectors.

**Cons.** The spec's construction-site / first-microtask attachment MUST then has no leaf-level test asserting it; the property ships verified only by review.

**Risks.** A future lazy-attach regression goes undetected by `npm test`.

### Recommendation

Take **Option A**. The spec states construction-site / first-microtask attachment as a MUST and explicitly designates the `Checkpoint` seam as the substrate for testing it without depending on microtask scheduling; dropping the assertion (Option B) would leave a real spec obligation untested. Add the early-rejection vector to each of the eight leaves so the construction-site claim is witnessed, drop the outcome-only "bypasses the substrate" clause (no suppression-only test can witness substrate bypass), and in `V15h`/`V15h-T` name the `Checkpoint`-seam swallowing-handler substrate explicitly so "the substrate" is unambiguous. Edge case for the implementer: the early-rejection vector must land the rejection before the first microtask boundary with no intervening attach point, mirroring the spec's lazy-`.catch` failure description.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one writes only the planned late-settlement vector and ships green, believing the construction-site / first-microtask timing is verified; the other notices the gap. The spec's construction-site attachment MUST ships effectively unverified — a lazily-attaching build (the exact defect the spec calls out) passes all three channels, and the "bypasses the substrate" claim cannot fire because the assertions are outcome-only. The leaf's gate passes vacuously for the construction-site property it claims to witness.
