## Finding

The typed-query "Forced respond turn" step in `docs/spec_topics/query/query-tool-loop.md` introduces a normative right-trim rule:

> The runtime right-trims any trailing U+000A bytes the rendered prompt text carries before inserting this separator … keeping the prompt-to-instruction gap exactly one line feed in every case.

The rule applies on *every* typed-query provider call (the boundary between the rendered template body and the forced-respond instruction wording). It is the load-bearing guarantee that an author who writes one or more trailing blank lines inside the `@<T>` template still produces a single-U+000A gap before the schema-bearing instruction sentence.

The rule has no normative input/output vector. The companion file `docs/spec_topics/query/query-forms.md` pins eight rendered-text vectors for the dedent/newline-trim normalisations, including subtle cases such as "newline followed by trailing whitespace before the closing backtick is not trimmed" (vector 8). The right-trim rule sits in a different phase (typed-query dispatch, after template render) and is not covered by any of those eight vectors, nor by the depth-6 worked example, nor by the `max_rounds: 0` paragraph. An implementation that omits the right-trim, trims a single U+000A instead of all of them, or trims any whitespace rather than only U+000A is not caught by any cited test vector.

## Solution Space

**Shape:** single

### Recommendation

Add a short normative input/output vector table to `docs/spec_topics/query/query-tool-loop.md` immediately after the right-trim sentence in step 2 of "Typed queries are tool-loop-shaped", in the same shape as the eight `query-forms.md` vectors. Pin the boundary cases the rule was added to close:

- Input rendered prompt text → forced-respond user-message body (the leading content of the user-message body, separated from the [Initial forced respond turn template](#initial-forced-respond-turn-template-normative) by a single U+000A).
- At minimum three rows: zero trailing U+000A (no-op), exactly one trailing U+000A (trim one), and two or more trailing U+000A (trim all). Use a fixed instruction-sentence stand-in (e.g. `<INSTR>`) so the table pins only the prompt-side and the separator, not the schema interpolation.
- A fourth row covering a trailing `\t` or trailing ASCII space *after* a U+000A should pin that the rule trims only U+000A bytes, not whitespace generally (the inverse of `query-forms.md` vector 8's "trailing newline + spaces is not trimmed" interaction).

Cite the new vector from the `Tests` bullets of `V13c` and `V13c-T` ("typed-query dispatch right-trims trailing U+000A from the rendered prompt before the single-LF separator; the four right-trim vectors pass") so the leaf gate enforces conformance. Edge case the implementer must watch: the trim is on the rendered text (post-newline-trim, post-dedent), not on the template source — vector wording must say "rendered prompt text" to avoid suggesting a parse-time edit. The interaction with `query-forms.md` vector 8 is observable: an author who writes `\n  ` before the closing backtick produces a rendered text that ends in `\n` (the trailing whitespace-only line normalises to an empty line), which the dispatch-time right-trim then removes — pin this in the commentary so the two phases compose correctly.

## Consequence

**Severity:** correctness

Two implementers reading the prose can diverge: one right-trims all trailing U+000A bytes, another trims at most one, another trims trailing ASCII whitespace, and the conformance suite catches none of them. The divergence is observable in the boundary case the rule was added to close — an author writing one or more trailing blank lines inside the `@<T>` template — and so the provider sees a different user-turn body across conforming implementations. Blast radius is narrow (every typed-query call uses this path, but the visible defect is one or more spurious U+000A bytes in the gap).
