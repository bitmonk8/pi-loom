## Finding

`conventions.md` is the one document every implementer reads in full before picking up any leaf — plan.md's "How to use this plan" step 1 instructs: "Read `conventions.md` once." Its body (phase categories, the per-phase TDD ritual, the leaf format, the cross-cutting rules) is written for that implementer audience. The `## Loom 1.0 release-time residue inspection` section appended at the end is different in kind: it is a 9-item manual-review checklist addressed to the **loom 1.0 release-gate auditor** — a reviewer reading `src/**`, the repository-root doc artifacts, the `spec_topics/**` corpus, the coverage matrix, and `extensions/index.ts` at release time and recording a clean/disposition verdict per item. That reader is not the per-leaf implementer following the TDD ritual.

The plan already has an established pattern for release-auditor protocols: `real-host-smoke-gate.md` is a standalone cross-cutting governance note (carrying no leaf ID, closing no REQ-ID) for exactly such a protocol, and plan.md and the H4a/H6a/H7a/V18d leaves reference it by file. The residue inspection is the same shape of artefact — a release-time auditor protocol — but is carried inline in the implementer-facing conventions doc rather than in its own file.

The counter-pull is real and constrains the fix: the section is deliberately co-located with the cross-cutting rules that feed it. Each rule (*No globals…*, *Specific exception types only*, *Sequential by default*, *Doc updates*, *REQ-ID discipline*) routes its conceded mechanical-scan residue to this inspection via an intra-document anchor (`#release-time-residue-inspection`) and a "(checklist item N)" reference, and the section opens with "The cross-cutting rules **above** each concede a residue … enumerated **here**." Five plan files outside conventions.md (`coverage-matrix.md`, `H3a`, `H4a`, `H5d`, `H5f`) also link to the anchor.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commits:** `e94ca10` — "pi-loom plan: resolve 'GOV-15 release-time reviewer inspection has no enumerated residue checklist'" (2026-06-16)
**History:** The `## Loom 1.0 release-time residue inspection` section was authored directly into `conventions.md` by `e94ca10`, which resolved a finding that the GOV-15 release-time review had no enumerated residue checklist. The checklist was placed in `conventions.md` from the section's inception and has never lived in any other file. Later commits (`6ad39e0`, `cda161b`, `2ad25ab`, `1c9ede1`, `4b3c2c0`) grew it from 5 to 9 items and added the cross-file anchor references in `H3a`/`H4a`/`H5d`/`H5f`/`coverage-matrix.md`, but none relocated it. The auditor-vs-implementer audience mismatch is therefore co-extensive with the section's creation, not a later regression.

## Solution Space

**Shape:** multiple

### Option A — Relocate to a dedicated governance note

**Approach.** Move the `## Loom 1.0 release-time residue inspection` section out of `conventions.md` into a new `docs/plan_topics/release-time-residue-inspection.md`, a cross-cutting governance note parallel to `real-host-smoke-gate.md` (no leaf ID, closes no REQ-ID).

**Plan edits.**
- Create `docs/plan_topics/release-time-residue-inspection.md` holding the section content — the `<a id="release-time-residue-inspection"></a>` anchor, the introductory paragraphs, and the 9 numbered items (1–9) — with the same anchor id and item numbers preserved so existing "checklist item N" references stay valid.
- In `conventions.md`, replace the moved section with a short pointer to the new file (mirroring how plan.md points to `real-host-smoke-gate.md`), and reword the section's opening deixis ("the cross-cutting rules **above** … enumerated **here**") so it reads correctly from a separate file.
- Retarget every `[loom 1.0 release-time residue inspection](#release-time-residue-inspection)` reference in `conventions.md`'s cross-cutting rules to `./release-time-residue-inspection.md#release-time-residue-inspection`.
- Retarget the cross-file references in `coverage-matrix.md`, `H3a`, `H4a`, `H5d`, `H5f` from `./conventions.md#release-time-residue-inspection` to `./release-time-residue-inspection.md#release-time-residue-inspection`.

**Spec edits.** None.

**Pros.** Matches the established `real-host-smoke-gate.md` precedent; gives the auditor protocol its own home; removes auditor-only material from the doc implementers read in full.

**Cons.** Broad diff across six plan files; severs the physical adjacency between each cross-cutting rule and the residue item it feeds; forces rewording the "rules above / enumerated here" framing.

**Risks.** A missed link retarget leaves a dangling intra-doc anchor. The "(checklist item N)" references and the anchor id must be preserved verbatim or the rule-to-item routing breaks.

### Option B — Keep in `conventions.md`, mark the audience in place

**Approach.** Leave the section co-located with the rules that feed it, but add an explicit audience marker so the reader knows it is an auditor protocol, not implementer ritual.

**Plan edits.**
- Prepend to the `## Loom 1.0 release-time residue inspection` heading or its first paragraph an explicit statement of audience and timing — that the checklist is performed by the loom 1.0 release-gate auditor at the release gate, not by the per-leaf implementer following the TDD ritual — paralleling the audience sentence `real-host-smoke-gate.md` carries ("Its audience is the bump/merge contributor and the H6a release-gate auditor, not the H4a implementer …").

**Spec edits.** None.

**Pros.** Single-file, minimal diff; preserves the deliberate adjacency between each conceding rule and its residue item and the intra-document anchor wiring; no cross-file link churn.

**Cons.** The auditor protocol still physically lives in the implementer-read doc; does not adopt the `real-host-smoke-gate.md` file-per-protocol precedent.

**Risks.** Negligible.

### Recommendation

Take **Option B**. The audience mismatch the finding identifies is real but cosmetic, and the section is structurally bound to `conventions.md`: every cross-cutting rule routes its conceded residue to a specific checklist item through an intra-document anchor, and the section's own prose ("the cross-cutting rules above … enumerated here") is written for that adjacency. An in-place audience marker — modelled on the audience sentence already in `real-host-smoke-gate.md` — resolves the mismatch without the six-file cross-link retarget or the deixis rewording that Option A's relocation forces. Choose Option A only if maintainers want strict file-per-auditor-protocol parity with `real-host-smoke-gate.md` and accept that churn; if they do, the anchor id `release-time-residue-inspection` and the 1–9 item numbering must be preserved verbatim so the rule-to-item references survive the move. The fix is confined to plan files; the spec (including the GOV-15 reviewer-inspection step the inspection runs alongside) is read-only here, and any new note carries no leaf ID per the `real-host-smoke-gate.md` convention.

## Consequence

**Severity:** cosmetic

A different-audience checklist sits at the tail of the doc implementers are told to read in full; the implementer reads auditor-only material once, and an auditor must open the implementer doc to find the protocol. No implementer produces a wrong leaf and the checklist stays reachable via its anchor, so the impact is organisational only.
