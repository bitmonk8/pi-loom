## Finding

`H5f`'s `Tests.` block carries three bullets. The first two name concrete, assertable gate behaviours against seeded fixtures: bullet 1 — the gate reddens when a facet leaf of a seeded multi-leaf fixture row has no facet-naming citing test and passes when every facet leaf does; bullet 2 — the gate derives the facet partition from the closing-leaf cell, dropping `co-witness`-annotated leaves. The third bullet — "the recogniser is a best-effort inline-citation scan over the test sources and certifies per-facet citing-test existence, not the assertion's semantic fidelity" — asserts no behaviour. It restates, almost verbatim, a scope limitation already stated in the leaf's **Adds** paragraph ("Like the `H5a` per-REQ-ID citing-test scan it is a best-effort inline-citation existence scan: it certifies that a facet-naming citing test exists for each facet, not that the cited test's assertion is semantically faithful to the facet's obligation").

The sibling leaf `H5a` handles the identical scope-limit signpost differently: rather than a standalone non-asserting bullet, it appends the clause onto its real asserting bullet ("…and passes when every mapped REQ-ID has a citing test; the recogniser is a best-effort inline-`PREFIX-N`-citation scan … certifies citing-test existence, not the assertion's semantic fidelity"). `H5f`'s standalone third bullet is the divergent shape — it occupies the `Tests.` block, where a TDD implementer reads each entry as a behaviour to assert, but offers nothing assertable.

## Solution Space

**Shape:** single

### Recommendation

Delete the third `Tests.` bullet of `docs/plan_topics/H5f-per-facet-citing-test-gate.md` — the `` `Convention:` (*REQ-ID discipline* — best-effort existence) the recogniser is a best-effort inline-citation scan over the test sources and certifies per-facet citing-test existence, not the assertion's semantic fidelity. `` line. The scope limitation it carries is not lost: the leaf's **Adds** paragraph already states the same best-effort / existence-only / no-semantic-fidelity property, so the `Tests.` block is left with only its two genuine assertions (the fixture-redden/pass behaviour and the facet-partition derivation).

Edge case for the implementer: if the scope-limit signpost is wanted *inside* the `Tests.` block rather than only in Adds, the established sibling pattern is `H5a`'s — append the clause onto the first asserting bullet (the existence-scan assertion) rather than reintroducing a standalone non-assertion bullet. Either way the deliverable is that no `Tests.` bullet stands as pure scope-limit prose.

## Consequence

**Severity:** cosmetic

A TDD implementer reading `H5f`'s `Tests.` block finds a third "test" that names no behaviour to assert and duplicates the Adds scope limit; at worst they author a vacuous or no-op test against it. The shipped gate behaviour is unaffected, so no two implementers diverge on what the arm does.
