## Problem

`docs/plan_topics/coverage-matrix.md`'s *Numbered REQ-IDs (runtime obligations)* table maps only `PIC-1 … PIC-23` and `PIC-40 … PIC-43` to closing leaves, omitting the other live numbered `PIC-N` anchors — `PIC-24 … PIC-39`, `PIC-44 … PIC-48`, and `PIC-50 … PIC-54` — each of which is a live executable spec REQ-ID whose obligation already has an implementing leaf. Several are currently reachable only through `cka-N` rows in the *Code-keyed obligation areas (no numbered REQ-IDs)* table, mis-classifying numbered obligations as code-keyed, while `PIC-33/PIC-34`, `PIC-44`, and `PIC-54` appear nowhere in the matrix. Per `conventions.md` §*REQ-ID discipline*, the `H5a`/`H6a` closing gate treats any executable spec REQ-ID without a Numbered REQ-IDs mapping as a CI failure, so on the live-corpus footing the loom 1.0 gate reddens for all ~26 omitted PICs even though their behaviour ships, and the gate's per-PIC citing-test scan never fires for them. The parallel `PIC-40 … PIC-43` drain (mapped to `V9i`) is the contrast case establishing the correct shape these drains omitted.

## Issue introduction

**Verdict:** introduced incrementally across the `PIC` GOV-22-drain commit series (multi-commit pattern)

**Introducing commits:**
- `82856ab` — coin PIC-24..28 on diagnostic-emission-isolation.md (no `coverage-matrix.md` edit) → V9g obligations left unmapped in the numbered table
- `3137321` — coin PIC-29..32 on drain-state-contract.md (no `coverage-matrix.md` edit) → V9m obligations unmapped
- `02b11c9` — coin PIC-33..35 on host-prerequisites.md; only repointed `cka-21` to PIC-35, leaving PIC-33/PIC-34 unmapped and PIC-35 filed in the code-keyed table
- `941e52c` — coin PIC-36..39 on registration-steps.md; cited in `cka-23`/V9b only (no numbered row)
- `a829ff0` — coin PIC-44 on tool-registration-lifetime.md (no `coverage-matrix.md` edit)
- `a9fb344` — coin PIC-45..48 on unknown-reason-rule.md; cited in `cka-28`/V9h only
- `1acdc55` — coin PIC-52/53 and close `cka-18`/`cka-24`/`cka-25` (PIC-50/51 via `cka-24`); no numbered rows
- `cfb3943` — coin PIC-54 (no `coverage-matrix.md` edit)

**History:** The `PIC` prefix has never had a retired-ID entry; `PIC-1 … PIC-55` are all live. Each commit above is a GOV-22 drain that converted an un-anchored MUST on a `pi-integration-contract/` page into a numbered `PIC-N` REQ-ID, but mapped it (when at all) through a code-keyed `cka` row or no row, rather than adding a *Numbered REQ-IDs* mapping. The sibling drain `3253d16` (coin PIC-40..43) is the contrast case: it added the numbered-table row `PIC-9, PIC-22, PIC-23, PIC-40, PIC-41, PIC-42, PIC-43 | V9i`, establishing the correct shape the other drains omitted. The defect is therefore the accumulated residue of the drain series failing to follow `3253d16`'s pattern, not a single regression.

## Solution approach

In `docs/plan_topics/coverage-matrix.md`, add a *Numbered REQ-IDs (runtime obligations)* row for each absent live `PIC-N` that has an implementing leaf, mapping it to the leaf whose tests already assert the obligation; use the existing closing-leaf assignments on the `cka-18`/`cka-21`/`cka-22`/`cka-23`/`cka-24`/`cka-25`/`cka-28`/`cka-29` rows as the source of truth for which leaf closes each. Add the inline `PIC-N` citation to each closing leaf's `Tests.` (or `Convention.`-cited tests) bullet so the gate's per-REQ-ID citing-test scan resolves. Reconcile the code-keyed rows that stood in for these now-numbered obligations so each obligation is classified exactly once — retire a row whose obligations are wholly numbered (as `cka-12` was retired after the QRY drain), or narrow a mixed row to its residual un-anchored / `loom/...` obligations.

## Solution constraints

- The spec `pi-integration-contract/` pages are read-only for this fix — the `PIC-N` anchors already exist and are correct; this is a plan-only change.
- Out of scope: `PIC-49` and `PIC-55` — they have no implementing leaf and are tracked by their own findings.
- Map only to leaves that already exist; do not invent leaf IDs, and treat the `cka-N` *Token* column as append-only (retired tokens are not reused).
