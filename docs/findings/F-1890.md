## Finding

`docs/plan_topics/coverage-matrix.md`'s *Numbered REQ-IDs (runtime obligations)* table maps only `PIC-1 … PIC-23` and `PIC-40 … PIC-43` to closing leaves. Every other live numbered `PIC-N` anchor — `PIC-24 … PIC-39`, `PIC-44 … PIC-48`, and `PIC-50 … PIC-54` — is absent from that table even though each is a live, anchored numbered REQ-ID in the spec (`<a id="pic-N"></a> **PIC-N.**` on the `pi-integration-contract/` pages, whose prefix `PIC` is a numbered-REQ-ID prefix per `req-id-prefix-table-active-a.md`) and each already has an implementing leaf. The behaviour is implemented; only the numbered-table mapping is missing.

Per `conventions.md` §*REQ-ID discipline*, an *executable spec REQ-ID* is "a numbered `PREFIX-N` REQ-ID … that denotes a runtime/parse/load obligation the loom implements — every numbered REQ-ID except the corpus-governance `GOV-*` family." All `PIC-N` anchors qualify. The closing gate "treats an **executable** spec REQ-ID without a coverage-matrix mapping … as a CI failure," and the *Code-keyed obligation areas (no numbered REQ-IDs)* table is explicitly reserved for obligations that carry "neither a numbered `PREFIX-N` REQ-ID nor a `loom/...` registry code." The omitted PICs are therefore mis-classified: several are reachable today only through code-keyed (`cka-N`) rows whose table is, by its own header, "no numbered REQ-IDs" (`PIC-35` in `cka-21`; `PIC-36 … PIC-39` in `cka-23`; `PIC-45 … PIC-48` in `cka-28`; `PIC-50/PIC-51` in `cka-24`; `PIC-52` in `cka-18`; `PIC-53` in `cka-25`); the obligations of `PIC-24 … PIC-28` (`cka-29` page) and `PIC-29 … PIC-32` (`cka-22` page) are covered only at the page level by their `cka` rows without the PIC tokens appearing at all; and `PIC-33/PIC-34`, `PIC-44`, `PIC-54` appear nowhere in `coverage-matrix.md`.

The correct shape already exists for the parallel `PIC-40 … PIC-43` drain (mapped as a Numbered REQ-IDs row to `V9i`), which is what makes the sibling omissions a defect rather than a design choice. (`PIC-49` and `PIC-55` are out of scope here — they have no implementing leaf at all and are tracked by their own findings.)

## Issue introduction

**Verdict:** introduced incrementally across the `PIC` GOV-22-drain commit series (multi-commit pattern)

**Introducing commits:**
- `82856ab` — coin PIC-24..28 on diagnostic-emission-isolation.md (no `coverage-matrix.md` edit) → V9g obligations left unmapped in the numbered table
- `3137321` — coin PIC-29..32 on drain-state-contract.md (no `coverage-matrix.md` edit) → V9m obligations unmapped
- `02b11c9` — coin PIC-33..35 on host-prerequisites.md; only repointed `cka-21` to PIC-35, leaving PIC-33/PIC-34 unmapped and PIC-35 filed in the code-keyed table

**History:** The `PIC` prefix has never had a retired-ID entry; `PIC-1 … PIC-55` are all live. Each commit above is a GOV-22 drain that converted an un-anchored MUST on a `pi-integration-contract/` page into a numbered `PIC-N` REQ-ID, but mapped it (when at all) through a code-keyed `cka` row or no row, rather than adding a *Numbered REQ-IDs* mapping. The sibling drain `3253d16` (coin PIC-40..43) is the contrast case: it added the numbered-table row `PIC-9, PIC-22, PIC-23, PIC-40, PIC-41, PIC-42, PIC-43 | V9i`, establishing the correct shape the other drains omitted. The defect is therefore the accumulated residue of the drain series failing to follow `3253d16`'s pattern, not a single regression.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/coverage-matrix.md`, add a *Numbered REQ-IDs (runtime obligations)* row for every live `PIC-N` currently absent from that table whose obligation has an implementing leaf, mapping each to the leaf that already closes it:

- `PIC-24 … PIC-28` → `V9g`
- `PIC-29 … PIC-32` → `V9m`
- `PIC-33` → the leaf that owns the four-package lock-step assertion (`H1a`); `PIC-34` → the leaf that owns the widen-on-bump build-time gate (`V18c`, with `V18d` if it co-closes the procedure)
- `PIC-35` → `H1a`
- `PIC-36 … PIC-39` → `V9b`
- `PIC-44` → `V9f`
- `PIC-45 … PIC-48` → `V9h`
- `PIC-50/PIC-51` → `V9n`
- `PIC-52` → `V14d`
- `PIC-53` → `V9c`
- `PIC-54` → `V7d`

Pin each mapping to the leaf whose tests actually assert the obligation; use the existing `cka`-row closing-leaf assignments as the source of truth for which leaf that is, and do not invent new leaf IDs.

Each numbered-row closing leaf's `Tests.` (or `Convention.`-cited tests for `H1a`) must carry an inline `PIC-N` citation so the gate's per-REQ-ID citing-test scan resolves; the obligations are already implemented, so this is adding the `PIC-N` token to the existing assertions, not new behaviour.

Reconcile the code-keyed rows that currently stand in for these now-numbered obligations (`cka-18`, `cka-21`, `cka-22`, `cka-23`, `cka-24`, `cka-25`, `cka-28`, `cka-29`) so each obligation is classified exactly once: an obligation that carries a numbered `PIC-N` REQ-ID belongs in the *Numbered REQ-IDs* table, not in the "no numbered REQ-IDs" table. Where a `cka` row's obligations are now wholly numbered, retire the row the way `cka-12` was retired after the QRY drain; where a row mixes numbered-PIC obligations with genuinely un-anchored MUSTs or `loom/...` registry codes, narrow it to retain only the residue that still belongs in the code-keyed table. The `cka-N` *Token* column is append-only, so retired tokens are not reused.

Constraints: the spec `pi-integration-contract/` pages are read-only for this fix — the `PIC-N` anchors already exist and are correct. Honour the existing leaf-ID scheme; the only new rows are numbered-REQ-ID rows pointing at leaves that already exist.

## Consequence

**Severity:** blocking

The `H5a`/`H6a` closing gate enumerates the live executable REQ-ID set (every numbered `PIC-N`) and reddens on any member without a Numbered REQ-IDs mapping; on the live-corpus footing activated by `H6a` this fires for all ~26 omitted PICs even though their behaviour ships, so the loom 1.0 release gate cannot pass. The numbered-REQ-ID citing-test arm also never fires for these obligations (their tests cite `cka-N` tokens or spec paraphrase, not the `PIC-N` inline citation the gate scans for), so a regression in any of them would go unwitnessed. The gate is thus simultaneously over-firing (false-reddening implemented obligations) and under-covering (no per-PIC citing-test reconciliation), which defeats the purpose of the live-corpus reconciliation.
