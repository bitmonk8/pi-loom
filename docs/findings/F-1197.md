## Finding

`docs/spec_topics/diagnostics/placeholder-rendering-b.md` carries a per-`kind` `<observed>` / `<required>` enumeration table (anchor `#host-incompatible-observed-required`) whose seven rows pin literal values — `"function"`, `"absent"`, `"present"`, `"<unreadable>"`, the Node floor `>=22.19.0`, the Pi-SDK pin range, the `typeof <path>` string form — that are also pinned inline inside each Step 0 sub-step of `docs/spec_topics/pi-integration-contract/capability-probe.md`. Step 0 (b) names `"absent"`/`"present"` for the `prototype-property` members; step (c) names `"function"`; step (d) names the pin range and the `"<unresolvable>"` sentinel; step (e) names `"function"`; *Self-failure* names `"<unreadable>"`. The probe prose declares itself the **canonical source of truth** for these literals; the table declares itself a "restatement" sourced from the probe prose.

The two sites carry the literal strings independently. Each probe sub-step ends with a cross-reference to the diagnostics-side table, and the table cross-references each probe sub-step back, but the spec has no mechanical pin tying their byte content together (no build-time literal-read assertion, no Single-source paragraph). A future edit to a probe-side literal — for instance widening `"function"` to admit a class-constructor surface, swapping `"absent"`/`"present"` for a `Boolean` rendering, or revising the `peer-dep-malformed-version` `<observed>` cases — can land in `capability-probe.md` without the table updating, leaving the two sites silently inconsistent. The condition (4) row in the table already shows how easy this drift is: the table compresses conditions (1)–(3) into "the literal `\"<unresolvable>\"`" while the probe prose enumerates them as three distinct installation-observable conditions; an edit to the probe condition list does not automatically propagate into the table's verbal compression.

Because the table is the conformance-test sourcing point for diagnostic-message substrings, drift here surfaces as conformance tests that pass against a stale contract — exactly the failure mode the spec's other source-of-truth pins (the four-package iteration list, the nine-member factory-probable list, the `SHUTDOWN_AWAIT_CAP_MS` constant) are explicitly engineered to prevent via shared pinned-constants blocks.

## Solution Space

**Shape:** multiple

### Option A — Delete the table; cross-reference probe prose from §8

**Approach.** Remove the `host-incompatible-observed-required` table from `placeholder-rendering-b.md`. Replace it with a brief §8 paragraph that, per `kind`, names the canonical anchor in `capability-probe.md` where the `<observed>` / `<required>` literals are pinned (Step 0 (a) for `node-floor`; Step 0 (b) for `abortsignal-shape`; Step 0 (c) for `sdk-capability-missing`; Step 0 (d) for the two `peer-dep-*` kinds; Step 0 (e) for `typebox-shape`; *Self-failure* for `probe-failed`). Move the existing diagnostics-side test vectors that depend on the literals (the four `host incompatible (...)` entries already in §8's *Test vectors.* block plus the `peer-dep-malformed-version` vector) up to sit alongside the new cross-reference paragraph; they remain in `placeholder-rendering-b.md` because they exercise rendering, not probe behaviour.

**Spec edits.** Delete the per-`kind` table and the surrounding two-sentence paragraph. Replace with a single paragraph cross-referencing the probe sub-steps. Edit `capability-probe.md`'s "restates these as the diagnostics-side test contract" sentences (at the ends of Step 0 (b), (c)) into self-standing pins (the literals are already there; only the cross-link disposition changes). Update `V7c-T-placeholder-rendering.md` and `V7c-placeholder-rendering.md` to cite `capability-probe.md` anchors instead of the table anchor.

**Pros.** Removes the dual-source structure entirely; `capability-probe.md` becomes the sole pin. Aligns with the spec's existing pattern (e.g. the `<reason>` closed set in §7 is pinned only at PIC's Unknown-reason rule). The "restates as the diagnostics-side test contract" framing already implies the probe is primary.

**Cons.** Conformance-test authors lose a single consolidated table; they must read across the five probe sub-steps to assemble the seven-row substring contract. Modest scrollage cost.

**Risks.** None on wire behaviour.

### Option B — Keep the table; pull probe-side literals into table cross-refs

**Approach.** Make the table the single pin. Edit each probe sub-step to replace the inline `details.observed` / `details.required` literals with a cross-reference to the table row, of the form "see the per-`kind` enumeration entry for `<kind>` in [Placeholder rendering b](../diagnostics/placeholder-rendering-b.md#host-incompatible-observed-required)". The runtime contract obligation (what the probe writes into the payload) is unchanged; the values are read from the diagnostics-side table.

**Spec edits.** In `capability-probe.md`: remove the literal strings from Step 0 (b)'s observed/required paragraph, Step 0 (c)'s `sdk-capability-missing` paragraph, Step 0 (d)'s `peer-dep-malformed-version` `details.observed` enumeration (keeping the four installation-observable conditions but referring to the table for the per-condition rendering), Step 0 (e)'s `typeof Type.Unsafe` literal, and *Self-failure*'s `"<unreadable>"` literal. Replace each with a table cross-reference. Reword the table's "canonical source of truth" framing to drop the "restates" framing — the table is now primary.

**Pros.** Test-authoring ergonomic stays: one table, seven rows.

**Cons.** Inverts the established "canonical source of truth" framing in `capability-probe.md`. The probe-side prose becomes harder to read in isolation: a reader who lands on Step 0 (b) cannot see what `details.observed` is without following a link to a different shard. Step 0 (d)'s four-condition enumeration is hard to factor cleanly: it pairs *installation conditions* with *rendered values*, and only the rendered-value half belongs in the table.

**Risks.** A reader who edits Step 0 prose without noticing the table-side pin would silently introduce a runtime/contract mismatch — the same risk that motivates the finding, transposed.

### Recommendation

Option A. The duplication the finding flags is asymmetric — `capability-probe.md` is and reads as the runtime authority, with the table sitting downstream as a test-contract consolidation — and the cleanest fix removes the consolidation rather than inverting the authority. The cost is modest: §8's *Test vectors.* block already covers six of the seven kinds inline, so deleting the table loses only the at-a-glance grid. Implementers must watch one edge case: `peer-dep-malformed-version`'s `<observed>` is currently rendered per Step 0 (d) condition (which routes the four installation-observable conditions to either `"<unresolvable>"` or the raw `version` string); when collapsing the table, preserve that two-case rendering rule explicitly in §8 prose so test authors do not infer a single `"<unresolvable>"` literal from a too-terse cross-reference.

## Consequence

**Severity:** advisory

A subsequent edit to one of the seven probe-side literal pins can land without the diagnostics-side table updating, producing a silent contract divergence. Conformance tests sourced from the table would then assert a stale substring while the runtime emits the new one; the build-surface gate that catches probe-prose drift does not see the table.
