## Problem

The per-`kind` `<observed>` / `<required>` literals for `loom/load/host-incompatible` are pinned at two independent sites: the enumeration table at anchor `#host-incompatible-observed-required` in `placeholder-rendering-b.md` §8, and inline in each Step 0 sub-step of `capability-probe.md`. The probe prose declares itself the canonical source of truth; the table declares itself a restatement, but no mechanical pin ties their byte content together. An edit to a probe-side literal can land without the table updating, leaving the two sites silently inconsistent — and because the table is the conformance-test sourcing point for diagnostic substrings, the drift surfaces as tests asserting a stale contract.

## Issue introduction

**Verdict:** introduced-then-extended
**Introducing commits:** `fa9f2d5` (2026-06-02) — `pi-loom spec: resolve "host-incompatible per-kind observed/required rendering"`
**History:** Commit `fa9f2d5` introduced the per-`kind` `<observed>` / `<required>` enumeration table in the then-monolithic `docs/spec_topics/diagnostics.md` (anchor `#host-incompatible-observed-required`) together with the `<observed>` carve-out paragraph that points at it, in response to finding T16 (`loom/load/host-incompatible` — `<required>` placeholder rendering is undefined). The same finding required pinning per-`kind` `<observed>` / `<required>` for the previously-uncovered discriminators (`abortsignal-shape`, `sdk-capability-missing`, `probe-failed`). The fix chose to consolidate them into a new diagnostics-side table; `capability-probe.md` already carried the same literals inline in Step 0 (a)–(e) prose at that point, so the table landed as a deliberate restatement guarded only by cross-references and a "canonical source of truth" label. Commit `a33bd49` extended the table the same week by adding the `sdk-capability-missing` discriminator row (resolving the parallel finding T14). Commit `f5e89f4` (2026-06-04) split the monolithic `diagnostics.md` into the index + child layout and relocated the table verbatim into `docs/spec_topics/diagnostics/placeholder-rendering-b.md`; the dual-source structure was preserved. Subsequent edits — `4c27f32`, `9371309`, `d8d5ad0`, `89eabf1` — each modified probe-side literals (peer-dep version-read mechanics, `"<unresolvable>"` cases, settings non-object-root observed rendering) and re-edited the table to match; the table did stay in sync, but only because each fix author noticed both sites needed updating, which is the manual discipline the dual-source structure makes load-bearing.

## Solution approach

Make `capability-probe.md` the sole pin for the `loom/load/host-incompatible` per-`kind` `<observed>` / `<required>` literals and remove the downstream consolidation. In `placeholder-rendering-b.md` §8, delete the `#host-incompatible-observed-required` table and its introducing paragraph, replacing them with a per-`kind` paragraph that forward-links to the canonical Step 0 anchors in `capability-probe.md` (`#entry-capability-probe`) where each literal is pinned, while retaining the diagnostics-side rendering test vectors that depend on the literals. In `capability-probe.md`, update the cross-references that point at the now-removed table so the probe prose stands as the sole pin. Update the `V7c-T-placeholder-rendering` and `V7c-placeholder-rendering` plan leaves to cite the `capability-probe.md` anchors rather than the table anchor.

## Solution constraints

- The `peer-dep-malformed-version` two-case `<observed>` rendering rule — the raw `version` string for the present-but-invalid-SemVer condition versus the `"<unresolvable>"` literal for the unresolvable conditions — MUST survive the collapse.
