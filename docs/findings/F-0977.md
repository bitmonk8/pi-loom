## Problem

DISC-3's *Non-canonical extension case* paragraph in `discovery/discovery-sources.md` deduplicates the `loom/load/non-canonical-extension` warning by byte-equality of `realpath` outputs, presupposing that two case-variant paths pointing at the same directory entry produce identical `realpath` output. PIC-13's `realpath` bullet in `host-interfaces-services.md` specifies only its `.code` rejection contract and documents the production adapter as a passthrough to `fs.promises.realpath`; it states no case-canonicalisation guarantee. Node's `fs.promises.realpath` and `fs.promises.realpath.native` differ on leaf-case output, so two conforming seam implementations can disagree on whether `Plan.LOOM` and `Plan.loom` dedup — which changes whether the warning fires on case-insensitive filesystems.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `76c649f` (2026-06-04) "pi-loom spec: resolve \"realpath required by Resolution but absent from FileSystem seam\"" — added `realpath` to `PIC-13`'s interface with its `.code` rejection contract, but did not state any case-canonicalisation guarantee.
- `9dfbcee` (2026-06-05) "pi-loom spec: resolve \"non-canonical-extension dedup keys on realpath but claims same-inode siblings\"" — reframed DISC-3's dedup from inode equality to "canonical-path equality via `realpath`", thereby pulling the dedup correctness onto a `realpath`-output property that `76c649f` did not pin.

**History:** The defect emerges from the interaction of the two commits, not from either alone. `76c649f` introduced the seam member with a deliberately narrow contract (rejections only); `9dfbcee` then made the discovery walk's dedup outcome depend on a case-canonicalisation property of that member's output that was never added. Each commit was internally consistent against its own review prompt; neither audited the other's surface.

## Solution approach

Add a clause to PIC-13's `realpath` bullet (`host-interfaces-services.md`, anchor `id="pic-13"`) pinning a case-canonicalisation guarantee: on case-insensitive filesystems, two input paths that differ only in component or leaf case but resolve to one directory entry must yield byte-identical `realpath` output. State that the guarantee binds every conforming seam implementation — both the production `PiFileSystem` adapter and the `FakeFileSystem`. Rewrite DISC-3's *Non-canonical extension case* paragraph (`discovery/discovery-sources.md`, anchor `id="disc-3"`) to cross-reference that PIC-13 clause in place of its prose-local `realpath`-canonicalisation justification.
