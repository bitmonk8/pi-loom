## Finding

The final sentence of `FRNT-1` reads: *"Adding a `looms.toolLoopMaxRounds` settings key is deferred per [Future Considerations](../future-considerations.md)."* This is a deferred-feature note mixed into the body of an in-scope normative requirement. The preceding clause already pins the in-scope rule ("loom 1.0 provides no operator-level override for `max_rounds`; the value is per-loom only"), and the deferred extension itself is already catalogued, with the same name and forward-link target, as a bulleted item in `docs/spec_topics/future-considerations/surface-extensions.md` ("Operator-level `looms.toolLoopMaxRounds` settings override").

The inline sentence therefore adds no obligation for a loom 1.0 implementer — the operator-level value does not exist in loom 1.0 — and exists only to point at the deferral catalogue, which is the catalogue's job. Its presence inside a long FRNT-1 paragraph that already carries the field's bounds, range rule, exhaustion behaviour, `max_rounds: 0` semantics, per-callee/per-repair semantics, and cancellation rule lengthens the requirement's surface for no normative gain and creates two co-equal mention sites for the deferred key (the FRNT-1 paragraph and the surface-extensions bullet), inviting drift if the deferred name or its mirror-of (`bind_model` → `looms.binderModel`) is ever revised.

## Solution Space

**Shape:** single

### Recommendation

Delete the trailing sentence ("Adding a `looms.toolLoopMaxRounds` settings key is deferred per [Future Considerations](../future-considerations.md).") from the FRNT-1 paragraph in `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md`. The preceding clause pinning that loom 1.0 provides no operator-level override stays — that is the in-scope rule. No new content is added to `surface-extensions.md`; the "Operator-level `looms.toolLoopMaxRounds` settings override" bullet already there is the deferred feature's owning record and needs no change.

Edge cases for the implementer:

- Do not also delete the preceding "loom 1.0 provides no operator-level override for `max_rounds`; the value is per-loom only" sentence — it is the in-scope normative statement and is not duplicated in `surface-extensions.md`.
- Leave FRNT-1's `<a id="frnt-1"></a>` anchor and the surrounding bullet structure intact; only the trailing sentence is removed.

## Consequence

**Severity:** cosmetic

Two implementers will produce the same observable behaviour with or without the sentence — the operator-level key does not exist in loom 1.0 either way. The cost is duplicated narration of a deferred feature inside an in-scope requirement paragraph, plus a drift-vulnerable second mention site if the deferred name is ever revised.
