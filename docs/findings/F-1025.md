## Problem

`session-model-and-appendix.md`'s `## Session Model` section mints `SM-N` and `SM-N<letter>` identifier tokens (`SM-1`…`SM-8`, plus sub-letters `SM-3a`/`SM-3b` and `SM-7a`…`SM-7e`) that are used as REQ-IDs are used elsewhere — bolded `**SM-N — title.**` defining markers and cross-page back-references — yet the `SM` prefix appears in neither the live REQ-ID prefix table (`req-id-prefix-table-active-a.md` / `-b.md`), the *Retired prefixes* sub-table, nor the inline-label prefix table on `stable-inline-labels.md`. `SM-1`…`SM-8` match GOV-3's `\b[A-Z]{2,4}-[1-9][0-9]*\b` token shape with no registered prefix, so the GOV-3 *Unknown-prefix closure invariant* is currently false against the live corpus and GOV-6's *table-completeness invariant* is breached on the same grounds. GOV-23 reads as if it governed the SM-N markers but is explicit that it governs only the page-local `sm-N-…` HTML anchors and that those are "not REQ-IDs and not inline labels", leaving the prose tokens with no governance home.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `40ffdc5` (2026-05-27, "pi-loom spec: resolve 'T19a — Replace session-model paragraph with eight SM-N sub-units'") and `9271e6a` (2026-05-30, "pi-loom spec: resolve 'SM-1…SM-8 block sits under Orientation despite owning normative contracts'"); compounded by `64cdc60` (2026-05-26, "pi-loom spec: resolve 'governance.md plan-corpus dependency + corpus-direction articulation' (T27 + T28 cluster)") and `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set").
**History:** `git log -S "SM-1" -- docs/` shows `40ffdc5` is the commit that first introduced `**SM-1 — Single-active-session binding.**` and its seven siblings into `docs/spec.md`, replacing the prior monolithic Session-model paragraph with eight bolded sub-obligations under the same REQ-ID-shaped lexical convention (`**PREFIX-N.**`) that GOV-1 uses for actual REQ-IDs; the diff added the tokens without touching either prefix table. The defect's *closure-invariant* half was authored four days earlier in `64cdc60`, where `git show 64cdc60 -- docs/spec_topics/governance.md` shows the GOV-3 *Unknown-prefix detector* clause being tightened into the present-day *Unknown-prefix closure invariant* — a strict membership obligation rather than a gate-only check. The two commits independently are coherent; the conflict is between them. `9271e6a` then re-framed the Session-model section on `spec.md` from "informative aggregator" to "mixed disposition, authoritative-in-place for the obligations no topic page restates" (per its `docs/spec.md` diff), which is what promoted the SM-N markers from orientation labels into load-bearing identifier tokens — making the closure-invariant violation matter. `f5e89f4` later relocated the block from `docs/spec.md` into `docs/spec_topics/session-model-and-appendix.md` as part of the 100KB-cap split, which brought the tokens into the `spec_topics/**/*.md` scope GOV-6's table-completeness invariant explicitly binds. The defect is thus the interaction of (a) GOV-3's closure invariant tightening, (b) the minting of REQ-ID-shaped `SM-N` markers, (c) their promotion to authoritative-in-place status, and (d) their relocation into the governed glob — no single commit alone produces it.

## Solution approach

Register the `SM` prefix in one of the two governed identifier-class tables — the REQ-ID prefix table on `req-id-prefix-table-active-a.md` / `-b.md`, or the inline-label prefix table on `stable-inline-labels.md` — so GOV-3's closure invariant and GOV-6's table-completeness invariant hold against the live corpus. Reconcile the chosen class with the sub-letter tokens (`SM-3a`/`SM-3b`, `SM-7a`…`SM-7e`), since GOV-3's numeric tail rejects alphabetic tails and GOV-16 pins one tail form per inline-label prefix; closing that gap may require a GOV-8 *Split* of `SM-3`/`SM-7`. Update GOV-23's "not REQ-IDs and not inline labels" disclaimer on `anchor-scheme-and-retired.md` to match the class chosen, leaving its anchor-scheme-stability rule for the page-local `sm-N-…` HTML anchors in force.

## Solution constraints

- `SM` may be registered in exactly one identifier class — REQ-ID or inline-label — not both; dual-classing breaks GOV-5's disjoint-prefix invariant.
- The page-local `sm-N-…` HTML anchors stay orientation-only under GOV-23 and MUST NOT be repurposed for a different obligation.
- Registering `SM` MUST NOT convert the Session Model section from informative-orientation (per GOV-30) into authoritative-in-place.
