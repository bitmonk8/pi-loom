## Problem

Real-host smoke pass criterion (e) in `real-host-smoke-gate.md` scores a behavioural-divergence fail whenever the live run emits a `loom-system-note` code outside the committed permitted-code list that `H7a` maintains alongside its fixture `.loom`. That list is described as a "manually-maintained, best-effort" union whose completeness "is not mechanically verified," and the only thing keeping it current is the unanchored prose phrase "kept current by a per-Deps-slice provenance obligation" — no leaf, owner, or trigger is charged with appending a newly-introduced code. When a slice in `H7a`'s **Deps** legitimately begins emitting a benign new code the stale union omits, the next smoke run scores criterion (e) as a merge/bump-blocking fail with no documented recovery, indistinguishable from a genuine divergence. The plan carries an analogous declared trigger for the `H5b` canary (plan.md How-to-use step 2) but none for the permitted-code list.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `052b019` (2026-06-11) — "resolve Real-host smoke pass criterion (e) names a permitted code set with no committed source": introduced the committed permitted-code list as criterion (e)'s reference set. `f340c3b` (2026-06-17) — "resolve permitted-code-list cannot-disagree guarantee unenforced": authored the "completeness … is not mechanically verified" disclosure and the "kept current by a per-Deps-slice provenance obligation" phrasing.
**History:** `git log --follow` on `H7a-integration-acceptance.md` and `real-host-smoke-gate.md` shows the permitted-code list was added at `052b019` and refined across several later commits (`542395a` widened its Deps, `f340c3b` added the cannot-disagree guarantee). The maintenance-ownership gap was present from the mechanism's inception: `052b019` introduced the committed list without a concrete update trigger, and `f340c3b` hardened the disclosure that completeness is unverified and asserted the abstract "provenance obligation" without pinning who updates the list, when, or how a spurious (e) fail recovers. No commit ever closed the gap.

## Solution approach

Add a concrete maintenance-trigger step to plan.md How-to-use, parallel to the existing `H5b`-Deps trigger (step 2), that names an owner and the event obliging an emitted code to be appended to `H7a`'s committed permitted-code list. Replace the bare "kept current by a per-Deps-slice provenance obligation" wording in `H7a-integration-acceptance.md` with a pointer to that trigger. Record a recovery path in `real-host-smoke-gate.md` criterion (e) for the stale-list case: an offending code that an in-`Deps` slice legitimately emits but the stale list omitted is an out-of-date-list condition recoverable by appending and re-running, not a behavioural-divergence finding.

## Solution constraints

- The recovery path must stay falsifiable: it applies only to a benign code an in-`Deps` slice legitimately emits — an unexpected code with no owning `Deps` slice, or one a slice does not legitimately produce, still scores criterion (e) as a behavioural-divergence fail and must not be waved through as a stale list.
