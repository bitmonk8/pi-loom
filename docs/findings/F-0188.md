## Problem

The canonical build order is dep-driven — `docs/plan.md` "How to use this plan" step 3 and the §Vertical slices preamble both instruct the reader to "pick the next leaf whose **Deps** are satisfied" — but no document states the *satisfaction test*: the observable condition under which a named Deps entry counts as satisfied. The `conventions.md` **Deps** field definition (Leaf format) says "Other leaf IDs that must be complete first" without connecting "satisfied" to the per-phase TDD ritual's `<id>-complete` / `<id>-T-complete` tags, and the plan carries competing completion signals (the `README.md` status row, the dated `CHANGELOG.md` line) an orchestrator could equally read. With no stated predicate, two reasonable orchestrators can disagree on pickability, making the build order non-deterministic. The predicate must also distinguish which tag satisfies which Deps-entry form, since a Deps list can name either an implementation leaf `<id>` or a tests task `<id>-T`.

**History:** `git log -S` over `docs/plan.md` and `docs/plan_topics/conventions.md` (and the pre-`docs/` move at `31ff060`, root `plan.md`) shows the **Deps** definition carried the "must be complete first" framing from the plan's first commit, and no commit in the corpus ever introduced a "satisfied iff `<id>-complete` git tag present" predicate (no match for `satisfied iff`, `-complete\` git tag`). The defect is an omission present continuously since inception, later made more load-bearing by the two 2026-06-10 build-order edits.

## Solution approach

In the `conventions.md` **Deps** field definition (Leaf format), state the pickability predicate against the per-phase TDD ritual's tags: a Deps entry naming an implementation/horizontal leaf `<id>` is satisfied when its `<id>-complete` git tag is present, and a Deps entry naming a tests task `<id>-T` is satisfied when its `<id>-T-complete` tag is present, making those git tags the authoritative completion signal for pickability (the `README.md` status row and `CHANGELOG.md` line remain doc-update artifacts, not the pickability source). Point `docs/plan.md` "How to use" step 3 and the §Vertical slices preamble at that `conventions.md` predicate so the two surfaces reference rather than independently restate it. `conventions.md` is the authoritative home — it already defines both the **Deps** field and the `<id>-complete` / `<id>-T-complete` tags.

## Solution constraints

- The tag predicate composes with, and does not override, the existing §Blocked obligations gating: a leaf flagged blocked (e.g. `V9l`) stays gated by its declared Deps edge on the resolution-owner leaf regardless of tag presence.
