## Problem

`descriptions.md` mandates that `///` doc comments lower to a JSON Schema `description:` field emitted byte-for-byte into the lowered schema, so the lowering pass emits `description` for every loom-declared schema carrying doc comments. The authoritative emitted-subset enumeration in `schema-subset.md`, introduced by "The normative subset loom emits and enforces is:", never lists `description` — it neither admits it as an emitted keyword nor places it on the "Explicitly **not** supported" list — and the Lowering Algorithm step 3 emission rules and the Canonical-schema-hash recipe's absent-keyword HTML comment are likewise silent. Because the subset is consumed as a reject-by-default allowlist, an implementer building the emit set or accept gate from this enumeration would not account for `description`, and the canonical-hash recipe leaves undefined whether emitted `description` text participates in the slug-determining canonical bytes.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 53f5831 (`spec: rewrite query primitive, system prompts, and schema syntax`, 2026-05-03)
**History:** The Schema Subset normative enumeration was first authored in `spec.md` at 53f5831, where the `### Schema Subset` section already opened with "the normative subset is:" and listed Types/Validation/Objects/Arrays/etc. without `description`, while the Descriptions section in the same revision already stated `///` "lower to JSON Schema `description:` fields." The omission has carried through unchanged ever since: the enumeration was reaffirmed at 8eea523, split into the topic file `descriptions.md`/`schema-subset.md` at fecb504, relocated at 81bf52a, and amended by later subset/hash findings (111e533 "Canonical hash recipe cites keywords the lowered subset never emits", 6892aa3 SUBS-1), none of which added `description`. `git log -S "description"` over `docs/spec_topics/schema-subset.md` reports no commit that ever added or removed the token, confirming `description` was never present in that page's enumeration. The defect is thus a co-introduced gap between the emit-set enumeration and the description-lowering rule that has existed since the subset was first written, not a regression from any later edit.

## Solution approach

Add `description` to the emitted-subset enumeration in `schema-subset.md` as an emitted annotation keyword (lowered from `///` doc comments, no validation semantics) so the reject-by-default allowlist admits it and Lowering Algorithm step 3 accounts for it. Reconcile the Canonical-schema-hash recipe to state whether emitted `description` text participates in the canonical-form bytes that determine the schema slug — either stating it participates, or adding `description` to the absent-keyword HTML comment if it is stripped before hashing.
