## Problem

Both interleave notes in `docs/plan.md` — the one under `### V9` and the one under `### V11` — assert the seam `V9b → V11a → V9c`/`V9i`/`V9j`, making `V11a` (Binder-model resolution) a prerequisite of `V9c`. That holds for `V9i` and `V9j`, whose Deps list `V11a`, but not for `V9c`, whose Deps are `V9c-T, V9a, V8a, V17a` with no `V11a` (and none of those transitively pull it in). Because the canonical build order is dep-driven, an implementer sequencing strictly by `V9c`'s satisfied Deps may complete `V9c` before `V11a` exists, contradicting the documented seam. The prose over-claims an ordering edge the machine-checkable Deps field does not enforce.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** b9b427d — pi-loom plan: resolve "slice-ordering narrative overpromises DAG order" (2026-06-10, Thomas Andersen)
**History:** Commit b9b427d authored both the V9 and V11 interleave notes, introducing the seam phrase `V9b → V11a → V9c`/`V9i`/`V9j` that lists `V9c` as a `V11a` consumer. `V9c`'s Deps file (`docs/plan_topics/V9c-conversation-drive.md`) has never contained `V11a` (`git log -S 'V11a'` against that file returns no commits), so the prose/Deps contradiction was born with the note in that single commit; the `V9c` leaf's Deps were already in their current `V11a`-free form when the note was written.

## Solution approach

Rewrite both interleave notes (under `### V9` and `### V11` in `docs/plan.md`) so the seam reads `V9b → V11a → V9i`/`V9j`, dropping `V9c` from the seam arrow and from the `V11a`-prerequisite clause in each, leaving only `V9i`/`V9j` as `V11a` consumers. Apply the same correction to both notes so they remain reconciled.

## Solution constraints

- Reconcile by correcting the prose; do not add `V11a` to `V9c`'s authoritative Deps (`docs/plan_topics/V9c-conversation-drive.md`).
