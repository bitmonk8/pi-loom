## Finding

Both interleave notes — the one under `## Vertical slices → ### V9` and the one under `### V11` in `docs/plan.md` — assert the build seam `V9b → V11a → V9c`/`V9i`/`V9j`, i.e. that `V11a` (Binder-model resolution) is a prerequisite of `V9c`. That claim holds for `V9i` and `V9j`, whose authoritative Deps both list `V11a` (`V9i`: `… V11a …`; `V9j`: `V9j-T, V4d, V11a`). It does **not** hold for `V9c`: its Deps are `V9c-T, V9a, V8a, V17a`, with no `V11a`, and none of `V9a`/`V8a`/`V17a` transitively pull in `V11a`.

Because the canonical build order is dep-driven ("pick the next leaf whose Deps are satisfied"), an implementer sequencing strictly by `V9c`'s Deps may legitimately complete `V9c` before `V11a` exists, directly contradicting the documented seam. The narrative prose and the machine-checkable Deps field disagree about the same ordering edge.

`V9c`'s own scope reinforces that the omission is likely correct rather than an oversight: `V9c` is the prompt-mode conversation driver and active-set gating leaf (Spec: `conversation-drive.md`, `host-interfaces-core.md`, `subagent.md#pic-2`); it does not touch binder-model resolution, which is the concern of the binder-dependent leaves `V9i`/`V9j`. The contradiction is a stale over-claim in the prose, duplicated identically across the two interleave notes, so both must be reconciled the same way.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** b9b427d — pi-loom plan: resolve "slice-ordering narrative overpromises DAG order" (2026-06-10, Thomas Andersen)
**History:** Commit b9b427d authored both the V9 and V11 interleave notes, introducing the seam phrase `V9b → V11a → V9c`/`V9i`/`V9j` that lists `V9c` as a `V11a` consumer. `V9c`'s Deps file (`docs/plan_topics/V9c-conversation-drive.md`) has never contained `V11a` (`git log -S 'V11a'` against that file returns no commits), so the prose/Deps contradiction was born with the note in that single commit; the `V9c` leaf's Deps were already in their current `V11a`-free form when the note was written.

## Solution Space

**Shape:** multiple

### Option A — Add `V11a` to `V9c`'s Deps

**Approach:** Treat the seam as authoritative; make `V9c` genuinely depend on `V11a`.
**Plan edits:** In `docs/plan_topics/V9c-conversation-drive.md`, change the Deps line `**Deps.** `V9c-T`, `V9a`, `V8a`, `V17a`` to include `V11a`.
**Spec edits:** None.
**Pros:** Leaves the prose untouched; the seam becomes machine-checkable for `V9c`.
**Cons:** Asserts a binder-model dependency that `V9c`'s scope (prompt-mode driver / active-set gating, no binder concern) does not evidence; risks a fabricated edge that blocks `V9c` pickup behind `V11a` for no functional reason.
**Risks:** If the dependency is spurious it inverts the defect — `V9c` becomes artificially blocked, and a future reviewer strikes it again.

### Option B — Correct both interleave notes to drop `V9c` from the seam

**Approach:** Treat `V9c`'s Deps as authoritative; the prose over-claimed.
**Plan edits:** In `docs/plan.md`, edit both interleave notes (the `### V9` one and the `### V11` one) so the seam reads `V9b → V11a → V9i`/`V9j` and the prerequisite clause names only `V9i`/`V9j` (remove `V9c` from both the "prerequisite of" list and the seam arrow). Apply the identical correction in both notes.
**Spec edits:** None.
**Pros:** Aligns the narrative with the actual, scope-consistent Deps of `V9c`, `V9i`, `V9j`; no leaf gains a dependency it does not need.
**Cons:** Requires editing two prose locations identically (the duplication this finding already flags).
**Risks:** Low; the edge being removed is the unbacked one.

### Recommendation

Take Option B: correct both `docs/plan.md` interleave notes so the seam reads `V9b → V11a → V9i`/`V9j` and `V9c` is removed from the `V11a`-prerequisite list in each note. `V9c`'s scope (prompt-mode conversation drive / active-set gating, Spec pages with no binder-model content) and its existing Deps both indicate it does not require `V11a`, whereas `V9i`/`V9j` do and already list it. Apply the change verbatim in both notes so they stay reconciled. Edge case for the implementer: confirm no other leaf or the coverage narrative re-derives a `V11a → V9c` edge before removing it.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one follows the prose seam and builds `V11a` before `V9c`; the other follows `V9c`'s satisfied Deps and ships `V9c` first, violating the documented ordering. The plan's own "sequence by Deps" rule sanctions the second path, so the seam claim for `V9c` is silently unenforceable as written.
