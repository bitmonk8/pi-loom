## Finding

The *Malformed-marker discriminator* paragraph in `docs/spec_topics/pi-integration-contract/audit-failures.md` states that "The audit MUST surface every malformed marker under family (5)'s discriminator" — the singular *discriminator* implies one shared `<symptom>` token across all seven clauses (a)–(g). The sibling *Failure-surface contract* in `docs/spec_topics/pi-integration-contract/audit-wire-and-canary.md` says the opposite: the per-`<symptom>` fixture floor explicitly enumerates nine family-(5) `<symptom>` tokens ("the family has nine tokens total: both stale sub-kinds (s1) and (s2) plus all seven malformed-marker clauses (a)–(g); one fixture exercises one token") and forbids collapsing multiple tokens onto a single fixture. The two paragraphs cannot both be honoured: either malformed markers route to a single `<symptom>` (and the per-clause fixture floor is unmeetable), or each clause has its own `<symptom>` (and the *Malformed-marker discriminator* sentence is wrong about how many tokens there are).

Even granting the per-clause reading the *Failure-surface contract* prescribes, the *Malformed-marker discriminator* gives no evaluation order for clauses (a)–(g). The clauses are not disjoint: a marker like `/* Allow-Pi-Surface PIC#x  no colon` fails (a) (missing trailing colon), (c) (separator-tolerance — the recognised separator has no required surrounding spaces in this candidate), (f) (uppercase `Allow-Pi-Surface`), and (g) (block-comment shape) at once. Two conforming implementations will emit different `<symptom>` tokens for the same source line, breaking the per-`<symptom>` routability obligation the *Failure-surface contract* pins. The parallel `(s1)` / `(s2)` rule in *Stale sub-kinds* explicitly requires distinctness of those two tokens; the same distinctness discipline must be specified for the seven malformed-marker clauses, together with first-failure order.

## Solution Space

**Shape:** single

### Recommendation

Edit the *Malformed-marker discriminator* paragraph in `audit-failures.md` to align with the per-`<symptom>` floor the *Failure-surface contract* already pins:

1. Replace the singular "The audit MUST surface every malformed marker under family (5)'s discriminator" with: "The audit MUST surface every malformed marker under family (5), with one stable `<symptom>` token per clause (a)–(g) — seven malformed-marker `<symptom>` tokens in total, parallel to the `(s1)` / `(s2)` distinctness rule under *Stale sub-kinds* above and counted by the *Failure-surface contract*'s nine-token total for family (5). The seven `<symptom>` tokens MUST be distinct so a contributor routing to step 2(b) branch (4) can identify which clause the marker failed without re-reading the source."
2. Add a normative first-failure evaluation order immediately after the clause list: "When a candidate fails more than one of clauses (a)–(g), the audit MUST evaluate the clauses in the listed order (a) → (b) → (c) → (d) → (e) → (f) → (g) and emit the `<symptom>` token for the first clause that fails; subsequent clause failures on the same marker MUST NOT produce additional family-(5) records." The literal order (a)→(g) is chosen because it matches the order the clauses are already authored in; any other deterministic order would do equally well, but the chosen order MUST be pinned by PIC rather than left implementation-owned, because per-`<symptom>` CI routing depends on it.
3. Preserve the existing dual-emission obligation (the malformed marker plus the underlying out-of-inventory reference under family (1)/(2)/(3)) unchanged — it is orthogonal to the within-family-(5) order rule.

Edge cases the implementer must watch:

- The dual-emission obligation already pinned in the same paragraph is *not* affected: first-failure order governs only which family-(5) `<symptom>` is emitted, not whether the family-(1)/(2)/(3) record co-emits.
- The fixture catalogue must seed each of the seven clauses *in isolation* (a marker that fails only clause (x)) so the first-failure rule never has to break a tie inside a fixture; a fixture that seeds multiple clauses simultaneously would be testing the order rule rather than the per-clause routing, and should be added separately if at all.
- Clause (g) (block-comment shape) is structurally entangled with (a) (missing colon after `allow-pi-surface`) and (f) (uppercase spelling) on the `/* Allow-Pi-Surface ...` family of inputs; the first-failure order resolves this entanglement deterministically without requiring the clauses to be re-disentangled into disjoint predicates.
- The *Per-family record-shape table* row for family (5) already uses the literal `<n/a>` sentinel in `symbol`; no record-shape edit is required by this fix.

## Consequence

**Severity:** correctness

Two reasonable H1 implementers will diverge on (a) whether family (5) has one malformed-marker `<symptom>` or seven, and (b) which `<symptom>` token to emit for a marker that fails multiple clauses simultaneously. The first divergence makes the per-`<symptom>` fixture floor either trivially met (one shared token) or strictly unmeetable from the audit-failures wording (which says there is only one); the second divergence makes per-`<symptom>` CI routing — the entire point of the four-segment `audit/<class>/<family>/<symptom>` discriminator scheme — non-portable across implementations.
