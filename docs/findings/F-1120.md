## Problem

The *Malformed-marker discriminator* paragraph in `docs/spec_topics/pi-integration-contract/audit-failures.md` states "The audit MUST surface every malformed marker under family (5)'s discriminator" — the singular *discriminator* reads as one shared `<symptom>` token across all seven malformed-marker clauses (a)–(g). The sibling *Failure-surface contract* on the same page (`id="audit-failure-surface-contract"`) instead enumerates nine family-(5) `<symptom>` tokens — the two stale sub-kinds (s1)/(s2) plus all seven clauses (a)–(g), one fixture per token — and forbids collapsing multiple tokens onto a single fixture; the two paragraphs cannot both be honoured. Separately, the discriminator paragraph gives no evaluation order for clauses (a)–(g), which are not disjoint (a single marker can fail (a), (c), (f), and (g) at once), so two conforming implementations emit different `<symptom>` tokens for the same source line, breaking the per-`<symptom>` routability the *Failure-surface contract* pins.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `e5cd514` ("pi-loom spec: resolve T35 SDK capability inventory closed-set negative-direction audit", 2026-05-09)
**History:** The *Malformed-marker discriminator* paragraph (with the singular "surface every malformed marker under family (5)'s discriminator") and the *Failure-surface contract* (with the per-`<symptom>` fixture floor and the explicit "nine tokens total: both stale sub-kinds (s1) and (s2) plus all seven malformed-marker clauses (a)–(g)" sentence) were introduced together in `e5cd514`, into the then-monolithic `docs/spec_topics/pi-integration-contract.md`. Both halves of the contradiction landed in the same edit; the later `f5e89f4` size-cap split (which created the present-day `audit-failures.md` and `audit-wire-and-canary.md` from the monolith) only moved the contradictory text apart across pages, it did not introduce it. The defect is therefore self-contained to `e5cd514`'s drafting of the audit-discriminator scheme.

## Solution approach

Rewrite the *Malformed-marker discriminator* paragraph's family-(5) surfacing sentence so each clause (a)–(g) commits to its own stable `<symptom>` token, parallel to the (s1)/(s2) distinctness rule under *Stale sub-kinds* and consistent with the nine-token count the *Failure-surface contract* already carries. Add a PIC-pinned (not implementation-owned) first-failure evaluation order over clauses (a)–(g) so a marker failing multiple clauses emits exactly one deterministic family-(5) `<symptom>`.

## Solution constraints

- Preserve the paragraph's existing dual-emission obligation (the malformed marker plus the underlying family-(1)/(2)/(3) record on the same run) unchanged — the first-failure order governs only which family-(5) `<symptom>` emits, not the cross-family co-emission.
- Out of scope: the *Failure-surface contract*'s nine-token enumeration and per-`<symptom>` fixture floor — align the discriminator paragraph to that count rather than editing the count.
