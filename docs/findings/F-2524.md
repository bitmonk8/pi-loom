## Problem

The frontmatter field contract makes value-shape-dependent decisions — the `mode:`/`bind_context:`/`model:` recognised-value vs non-string-scalar splits, and FRNT-1's rule that integer-ness is judged on the parsed numeric value (`25` and `25.0` both accepted, `"25"` rejected) — that all presuppose a specific YAML scalar-resolution outcome for a given source token. Whether a bare token resolves to a string, boolean, number, or null depends on the YAML schema the parser applies: YAML 1.1 resolves `yes`/`no`/`on`/`off` to booleans and admits octal/sexagesimal integers, while YAML 1.2 core leaves those as strings under a narrow numeric grammar. The contract pins the consequences but never names the scalar-resolution schema they rest on, so tokens like `bind_context: none`, `model: no`, or `max_rounds: 010` fire different diagnostics (or none) under two conforming parser configurations — the same `.loom` file is accepted by one runtime and rejected by another.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 24a8079 (2026-05-07) — first reliance on a parsed-numeric scalar model (the `max_rounds` integer-ness rule, `25`/`25.0` both accepted); 5f5c487 (2026-06-03) — extended the typed-scalar reliance with the `mode:`/`bind_context:`/`model:` non-string-scalar recognised-value splits; 1064946 (2026-06-10) — named the `yaml` dependency but scoped itself to "names the dependency, not its version", a near-miss that left the scalar-resolution schema unpinned.
**History:** The typed-scalar premise has been load-bearing since the frontmatter contract first encoded value-shape-dependent diagnostics. `git log -S` over the cited tokens shows the parsed-numeric rule entering at 24a8079 and the non-string-scalar splits at 5f5c487, neither accompanied by a named YAML schema. The one commit that touched the adjacent gap (1064946, resolving "YAML frontmatter parsing mechanism is never declared as a dependency") added the `yaml` dependency clause but deliberately declined to pin a version, and pinned no scalar-resolution schema either — so the omission was visible at that point and left open. The defect is an inception-level omission rather than a regression introduced by a single later commit.

## Solution approach

Pin the YAML scalar-resolution schema as a behavioural conformance point in `frontmatter-fields-a.md`'s Field-contract section (or equivalently the `frontmatter.md` YAML-frontmatter intro), stating the schema independent of which library or version implements it. Have FRNT-1 (`frontmatter-fields-b-and-templates.md`) and the parsed-scalar rendering note in `placeholder-rendering-b.md` cross-reference the pinned schema rather than restating scalar-resolution behaviour.

## Solution constraints

- The pin states observable scalar-resolution behaviour only; the parsing-library version remains an implementation choice (preserve the `implementation-notes.md` "names the dependency, not its version" stance).
