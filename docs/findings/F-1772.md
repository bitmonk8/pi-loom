## Problem

PIC-43's subagent transport-failure short-circuit maps a trailing-`assistant` `stopReason` of `"error"` to `Err(QueryError { kind: "transport", …, provider: <resolved-model-provider>, … })`, but `<resolved-model-provider>` is a bare placeholder with no derivation rule — it appears once and is described only as "the same shape as the prompt-mode mapping." Prompt mode derives `provider` from `ctx.model` (the caller's user-session model), whereas a subagent query runs on the spawned `AgentSession`, whose model is the loom's resolved `model:`. Borrowing the prompt-mode placeholder without restating the source leaves two divergent readings: `ctx.model` (mis-attributing the failure to the caller's provider) versus the loom's resolved `model:` (the session that actually failed). The field is mandatory: the Runtime event channel's always-log emission for `kind: "transport"` requires it.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `222aacd` (2026-06-23, "fix F-1227 — subagent untyped-query transport-failure short-circuit (+ version-bump item m)"); `6f38d17` (2026-06-25, "fix F-1145 — derive prompt-mode TransportError.provider from ctx.model + sentinel")
**History:** The subagent transport-failure short-circuit and its bare `provider: <resolved-model-provider>` placeholder entered `subagent.md` in `222aacd` (the token's only introducing commit per `git log -S`). At that time `queryerror-variants.md` carried only a generic `provider` value-space (added `43bef30`, 2026-06-05) with no per-surface source clause, so the placeholder had no owner to cite. `6f38d17` then authored the per-surface derivation in `queryerror-variants.md` — distinguishing the loom-resolved-model surfaces (including the subagent driven turn → resolved model `.api`) from the prompt-mode user-session source (`ctx.model.api`) — but did not back-link `subagent.md` PIC-43 to it or expand the placeholder. The defect is the interaction: the placeholder authored before the owning text existed was never rewired once `6f38d17` created that owner, leaving PIC-43's local crutches ("`<resolved-model-provider>`" + "same as prompt-mode") pointing away from the now-authoritative derivation.

## Solution approach

Expand the `<resolved-model-provider>` placeholder in PIC-43 to name the resolved `Model<Api>.api` value of the loom's resolved `model:` — the same value passed to `createAgentSession` and the same `<provider>` derivation used in "Provider compatibility for typed queries" in `conversation-drive.md`. Clarify that this is **not** `ctx.model`, which in subagent mode is the caller's user-session model rather than the spawned session that failed. Note that the resolved model is necessarily defined when this short-circuit can fire — PIC-40's pre-spawn guard fails an unresolved spawn with `loom/runtime/subagent-model-unresolved` — so no `"unknown"` sentinel is required here.
