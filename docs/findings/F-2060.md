## Problem

The `ActiveInvocationRegistry` section establishes "the three insertion sites" as its canonical label for the three locations a registry entry is created (slash-command handler entry, `tool.execute(...)` adapter entry, `invoke` spawn-site entry), and the *Registry contract* bullet list reuses that term. But the normative rule pinning the setup `try`/`catch` is titled **Dispatch-site setup wrap**, and that "dispatch-site" framing is back-referenced from the intro paragraph's `invocationId` sentence and the `disposeBarrier` resolver bullet. "Dispatch-site" misdescribes two of the three arms — the `invoke` spawn-site is a callee-spawn (parent suspended at the call site, "not dispatched") and the `tool.execute(...)` adapter is a host-driven tool call. The result is two labels for the same three locations within one section; behaviour is not in doubt (the rule body enumerates all three with a `MUST`), but the heading misnames two of the arms it governs.

## Issue introduction
**Verdict:** single-commit introduction (effectively present since this file's inception)
**Introducing commits:** `f5e89f4` (2026-06-04) — "docs: enforce 100KB hard / 30KB recommended size cap on spec set"
**History:** The `ActiveInvocationRegistry` section was authored into `docs/spec_topics/pi-integration-contract/active-invocation-registry.md` at the size-cap split commit `f5e89f4`. Both the canonical "three insertion sites" intro label and the "Dispatch-site setup wrap" rule heading were introduced together in that commit — `git show f5e89f4:…/active-invocation-registry.md` carries both. The predecessor `docs/spec.md` `ActiveInvocationRegistry` text contained neither term: `git log -G "three insertion sites" -- docs/spec.md` and `git log -G "[Dd]ispatch-site" -- docs/spec.md` both return no match. The two labels have coexisted unchanged since: `git log -S "Dispatch-site setup wrap" -- <file>` lists only `f5e89f4`, and later commits that edited the surrounding paragraph (`e9b0597`, `14424b4`, `1d6d03e`, …) never reconciled the heading with the canonical term.

## Solution approach

Rename the *Registry contract* rule heading and its two in-section back-references so the section uses the canonical "insertion-site" term: the heading `- **Dispatch-site setup wrap.**` becomes `- **Insertion-site setup wrap.**`, the intro paragraph's "inside the **Dispatch-site setup wrap** `try`/`catch`" updates to match, and the `disposeBarrier` resolver bullet's "held in the dispatch-site insertion-site closure" reduces to "held in the insertion-site closure".

## Solution constraints

- Out of scope: the section's other "dispatch" uses that correctly describe slash-handler dispatch semantics — "the slash-command handler's dispatch lookup", "dispatch off the `LoomRegistry` arm", and "the parent is suspended at the call site, not dispatched". The rename retires only "dispatch-site" as the umbrella label for the three-site set.
