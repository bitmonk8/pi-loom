## Finding

Both `docs/spec_topics/discovery/discovery-sources.md` (line 37, DISC-1's *Home-directory expansion* paragraph) and `docs/spec_topics/discovery/package-and-settings.md` (line 45, the *Settings file reads* paragraph) carry a Markdown link whose anchor text is `` `FakeFileSystem` / `FileSystem` interface `` and whose href is `(../pi-integration-contract.md)` — a fragment-less reference to the Pi-integration-contract **hub page**. That hub page (`docs/spec_topics/pi-integration-contract.md`) is now a pure index — a one-paragraph framing plus a Contents list of 30 child pages — and contains no `` `FakeFileSystem` / `FileSystem` interface `` section. The interface those two xrefs are trying to reach lives at `docs/spec_topics/pi-integration-contract/host-interfaces-services.md#pic-13` (the `PIC-13. FakeFileSystem / FileSystem interface` block, anchored as `<a id="pic-13">` and `<a id="fakefilesystem--filesystem-interface">`).

A reader following either link lands on the TOC and has to hunt the right child page to find the cited interface — and the link text actively misleads them, because the hub page does not contain the section it names. The same defect-shape is back-cited cleanly elsewhere in the corpus: DISC-2's *clean leaf-ENOENT* bullet on the same `discovery-sources.md` page links to `../pi-integration-contract/host-interfaces-services.md#pic-13` (added by F-0976 / commit `dc6d69a`), and `package-and-settings.md`'s `Clock` / `cwd()` citations also resolve directly to `host-interfaces-services.md` anchors. Only these two `FakeFileSystem` / `FileSystem` xrefs were missed.

(The original review report frames this through a fabricated `#filesystem-seam` fragment that appears only in the review tool's xrefs.md projection — neither source file actually emits that fragment. The real defect is the unhelpful link target, not the projection artefact.)

## Solution Space

**Shape:** single

### Recommendation

In both files, retarget the link to `../pi-integration-contract/host-interfaces-services.md#pic-13`, matching the form already used by the DISC-2 *clean leaf-ENOENT* bullet on the same `discovery-sources.md` page (and by F-0976's back-citation pattern). Specifically:

- `docs/spec_topics/discovery/discovery-sources.md` line 37 — change `[Pi Integration Contract — \`FakeFileSystem\` / \`FileSystem\` interface](../pi-integration-contract.md)` to `[Pi Integration Contract — \`FakeFileSystem\` / \`FileSystem\` interface](../pi-integration-contract/host-interfaces-services.md#pic-13)`.
- `docs/spec_topics/discovery/package-and-settings.md` line 45 — change `[\`FakeFileSystem\` / \`FileSystem\` interface](../pi-integration-contract.md)` to `[\`FakeFileSystem\` / \`FileSystem\` interface](../pi-integration-contract/host-interfaces-services.md#pic-13)`.

Edge cases:

- Prefer `#pic-13` over the equivalent `#fakefilesystem--filesystem-interface` anchor — `#pic-13` is the governed REQ-ID under GOV-23 anchor scheme and is the form sibling xrefs (DISC-2, the V8b plan leaf, F-0716 / F-0723 / F-0976 / F-0977 / F-1000 / F-1001) already use, keeping the back-citation pattern uniform.
- The fix is pure-editorial; no spec rule, ID, or behavioural surface changes. No new `<a id>` needs to be minted in `host-interfaces-services.md` (PIC-13 is already anchored).
- Do not add a `#filesystem-seam` anchor to the hub page — that fragment is a review-tool projection artefact, not a real link target, and the hub page is deliberately a thin index.

## Consequence

**Severity:** advisory

A reviewer or test author following the DISC-1 or *Settings file reads* citation reaches a hub page that does not contain the named section, forcing them to scan the Contents list and guess which child page (`host-interfaces-services.md`) actually owns the interface. Two normative passages — the `FileSystem.homedir()` rule for `~/` expansion and the `FileSystem`-routed settings-file read — rely on these xrefs for their seam grounding. No behaviour is in question; the cost is reader friction and a corpus consistency blemish (the sibling DISC-2 xref already points at `host-interfaces-services.md#pic-13`).
