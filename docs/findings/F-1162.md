## Problem

The `` `FakeFileSystem` / `FileSystem` interface `` cross-references at DISC-1's *Home-directory expansion* paragraph in `discovery-sources.md` and at the *Settings file reads* paragraph in `package-and-settings.md` both link to the fragment-less hub page `../pi-integration-contract.md`. That hub page is now a pure index (framing paragraph plus a Contents list) and contains no such section — the cited interface lives at `host-interfaces-services.md#pic-13`. A reader following either link lands on the TOC, is misled by link text naming an absent section, and must hunt the owning child page. Sibling xrefs on the same surface (DISC-2's clean leaf-`ENOENT` bullet and the `Clock` / `cwd()` citations) already resolve directly to `host-interfaces-services.md` anchors.

## Issue introduction

**Verdict:** multi-commit-interaction

**Introducing commits:**

- `043762e` (2026-05-04, "pi-loom spec: resolve 'looms.binderModel is an extension-owned settings key, not a Pi-recognised setting'") — first wrote the fragment-less link `[FakeFileSystem / FileSystem interface](./pi-integration-contract.md)` in `spec_topics/discovery.md` for the *Settings file reads* passage.
- `2793da3` (2026-05-05, "pi-loom plan: resolve 'M ~/.pi/agent/looms/ path expansion unspecified for Windows'") — added the DISC-1 *Home-directory expansion* paragraph with the same fragment-less link `[Pi Integration Contract — FakeFileSystem / FileSystem interface](./pi-integration-contract.md)`.
- `8a6d15f` (2026-05-06, "pi-loom spec: resolve 'FileSystem seam interface signature lives outside the spec corpus'") — added the actual `**FakeFileSystem` / `FileSystem` interface.**` section (with `<a id="fakefilesystem--filesystem-interface">`) to `spec_topics/pi-integration-contract.md`. At this point the two pre-existing xrefs at least landed on a page that contained the section, even though the fragment had never been written in.
- `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set") — split `pi-integration-contract.md` (576 KB) into a TOC index plus 30 child pages, lifting the `FakeFileSystem` / `FileSystem` interface block to `pi-integration-contract/host-interfaces-services.md` under `<a id="pic-13">`. The commit's map-driven rewrite repointed 1571 cross-references across `docs/`, but because these two xrefs carried no fragment they were treated as "still resolves to a valid file" and were left pointing at the (now-empty) hub.

**History:** `git log -G "FakeFileSystem.*FileSystem.*interface"` over `docs/spec_topics/discovery*` confirms the defect was *latent* between `043762e`/`2793da3` and `f5e89f4` (the link target had the right section, just no fragment) and became *visible* at `f5e89f4` when the hub was emptied. The sibling `discovery-sources.md` xref that DISC-2 needed for the same surface was fixed later at `dc6d69a` (2026-…, F-0976), but only the DISC-2 line was retargeted in that pass — the DISC-1 and *Settings file reads* xrefs were not part of F-0976's scope and survived untouched.

## Solution approach

Retarget both links to `../pi-integration-contract/host-interfaces-services.md#pic-13`, matching the form already used by the DISC-2 *clean leaf-`ENOENT`* bullet on the same `discovery-sources.md` page. Use the `#pic-13` REQ-ID anchor rather than the equivalent `#fakefilesystem--filesystem-interface` alias, keeping the back-citation pattern uniform with sibling xrefs. The PIC-13 anchor already exists; no new anchor is minted.
