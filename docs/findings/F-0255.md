## Finding

`V5f`'s **Adds** owns the lowering pass's anonymous-inline-object hoist and automatic `$defs`/`$ref` generation — steps 2–3 of the spec's Lowering Algorithm (`schema-subset.md#lowering-algorithm`): an anonymous inline object schema appearing in any type position is hoisted into `$defs` under the synthesised name `__inline_<slug>`, and its original position is rewritten to `{ "$ref": "#/$defs/__inline_<slug>" }`. Two inline schemas collapse to one shared `$defs` entry only when their lowered fragments are byte-identical.

V5f's (and V5f-T's) **Tests** block asserts the canonical-hash recipe, the `loom/load/schema-slug-collision` posture, and the per-schema sidecar two-map shape. None of these asserts the structural transform itself: no vector confirms that an inline object is lifted into a `$defs/__inline_<slug>` entry, that the in-place reference is rewritten to the matching `$ref`, or that two byte-identical inline schemas resolve to a *single shared* `$defs` entry. The existing slug-collision bullet's "byte-identical ones dedup silently" clause asserts only that no collision diagnostic fires — it does not assert the deduped document contains one shared entry with correct `$ref` wiring.

The hoist/`$ref` behaviour is therefore exercised only incidentally: the hash-recipe vector needs a lowered fragment to hash, but it does not assert the emitted document's `$defs`/`$ref` structure. V5f's **Ships when** likewise names only the hash recipe, slug-collision detection, and sidecar shape.

**Verdict:** present-since-inception

The gap survived `ba7da00` ("V5d lowering-pass outputs bound by no Tests bullet"), which added the per-schema sidecar vector but not the hoist/`$ref` one, and `cb8737a` ("Systemic leaf over-bundling…"), which split the lowering content out of the V5d pair into `V5f`/`V5f-T` (rename `V5d-T-subset-lowering.md` → `V5f-subset-lowering-hash.md`) without altering the Tests coverage. The missing-vector defect has been continuously present since the leaf first existed.

## Solution Space

**Shape:** single

### Recommendation

Add a hoist/`$ref`/dedup vector to the **Tests** block of `V5f-T-subset-lowering-hash.md` (the red-first TDD driver) and mirror it in `V5f-subset-lowering-hash.md`'s **Tests** so the two stay in lockstep, then name it in `V5f`'s **Ships when** alongside the existing hash-recipe, slug-collision, and sidecar assertions.

The vector must assert, against the lowered document for a fixture containing an anonymous inline object schema in a type position (cite `schema-subset.md#lowering-algorithm` steps 2–3 as the spec source):

- the inline object is lifted into a `$defs/__inline_<slug>` entry whose `<slug>` is the canonical-hash slug of the lowered fragment;
- the original in-place position is rewritten to `{ "$ref": "#/$defs/__inline_<slug>" }`;
- two byte-identical inline schemas resolve to a single shared `$defs` entry referenced by both sites — i.e. the deduped document contains exactly one `__inline_<slug>` entry, distinct from the existing assertion that no collision diagnostic fires.

## Consequence

**Severity:** advisory

The hoist + auto-`$ref` rewrite is the core structural output of the lowering pass, yet no gate observes it. An implementation that emits the inline object in place without lifting it, targets the wrong `$ref`, or fails to collapse two byte-identical fragments to one shared `$defs` entry would ship green — the leaf's coverage gate passes while the behaviour V5f's Adds promises is unverified. Downstream consumers of the lowered document (per-query `$defs` pruning, AJV validation) inherit any structural defect undetected.
