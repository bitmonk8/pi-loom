## Problem

`subagent.md` rule 2 rests the subagent tool-isolation invariant on an unencoded behavioural claim about `createAgentSession`'s `tools` field ŌĆö that an explicit array (including the empty `tools: []`) acts as a closed allowlist suppressing Pi's default `read` / `bash` / `edit` / `write` built-ins, and that omitting `tools` re-enables them. This is the only mechanism enforcing the "ambient Pi tools NOT inherited" invariant that `subagent-state-isolation-matrix` delegates to, yet `CreateAgentSessionOptions.tools` is typed `string[] | undefined` and encodes none of these semantics. The behaviour has no detection path: Step 0's capability probe does not exercise it, and step 2(a)'s surface-inventory read asserts only the member's presence. Sibling behavioural presuppositions on the page are pinned to named editorial-review items ((n), (o), (y), (aj)), but the `tools`-allowlist suppression behaviour has no corresponding (a)ŌĆō(ak) entry, so a future Pi minor that changed the default-tool or empty-allowlist semantics would silently re-enable ambient built-ins inside every spawned subagent with no load-time or bump-time signal.

## Issue introduction

**Verdict:** single-commit-introduced
**Introducing commits:** `69b9c12` (2026-05-04) ŌĆö *pi-loom spec: resolve "createAgentSession example passes tool definitions through the wrong field"*
**History:** Before `69b9c12`, the spawn-block prose was a one-liner that referenced `createAgentSession({ tools, ... })` without claiming anything about how the field shaped Pi's default tool set. `69b9c12` expanded the block into the present three-rule structure (now grown to six rules) and introduced both the comment `// explicit allowlist suppresses Pi's default built-ins` and the prose "omitting `tools` would re-enable Pi's default built-in `read` / `bash` / `edit` / `write` tools regardless of what the loom declared." `git log -S "explicit allowlist suppresses"` and `git log -S "ambient Pi tools NOT inherited"` both bottom out at `69b9c12`; the surrounding behavioural-presupposition apparatus on the page (items (n), (o), (y), (aj)) was added by later commits that pinned other satellite types but never circled back to pin this one.

## Solution approach

Add a new editorial-review checklist item under the version-bump procedure's "Editorial-review checklist for unpinned host presuppositions" (the next free letter in the (a)ŌĆō(ak) sequence) on the same SHOULD-level footing as items (n) / (o) / (y) / (aj), auditing the `tools`-allowlist suppression behaviour against the `#bump-baseline-acquisition` snapshot on each Pi minor bump. The item must cover the three sub-properties that can each regress independently: an explicit non-empty `tools` array restricts the active set to exactly those names, `tools: []` yields an empty active set distinct from omission, and omitting `tools` enables Pi's default built-ins. Add a back-reference from `subagent.md` rule 2 to the new item, and note `noTools`'s existence in the item so a future reshape of `tools: []` defaulting is caught by the same pass.

## Solution constraints

- Out of scope: do not add a `noTools` field to the `createAgentSession` spawn call in `subagent.md` ŌĆö the spec mandates an explicit `tools` array, and consuming `noTools` would broaden the consumed Pi surface.
