## Finding

`subagent.md`'s rule 2 turns the tool-isolation invariant entirely on a behavioural claim about how Pi's `createAgentSession` treats the `tools` field: that passing an explicit array — including the empty array — acts as a closed allowlist that suppresses Pi's default `read` / `bash` / `edit` / `write` built-ins, and that omitting `tools` re-enables them. That is the *only* mechanism the spec offers for enforcing the "ambient Pi tools NOT inherited" invariant, and the state-isolation matrix's "ambient Pi tool set" row delegates to it.

The `CreateAgentSessionOptions.tools` field is typed `string[] | undefined` at `dist/core/sdk.d.ts`; nothing in that type encodes the suppression semantics. The behaviour is a JSDoc-documented runtime property of `createAgentSession`'s body (at the loom 1.0 Pi-SDK pin, `dist/core/sdk.js` derives `allowedToolNames = options.tools ?? (options.noTools === "all" ? [] : undefined)` and seeds `initialActiveToolNames` from `defaultActiveToolNames = ["read","bash","edit","write"]` when both `tools` and `noTools` are absent), so the rule holds today — but it is a load-bearing behavioural presupposition with no detection path. Step 0's seven-capability factory probe does not exercise it. Step 2(a)'s SDK surface-inventory literal-read asserts only the presence of the `tools` member, not its semantics. Sibling behavioural presuppositions on the same page are pinned to named editorial-review items under [Pi version bump procedure](../spec_topics/pi-integration-contract/version-bump-intro.md) — `SessionManager.inMemory` no-persistence to item (aj), the non-probed `AgentSession` members to item (n), `dispose()`-mid-`abort()` to item (y), the spawn-block satellite *types* (`ResourceLoader`, `SessionManager.inMemory` signature, `DefaultResourceLoader`) to item (o) — but the `tools`-allowlist suppression behaviour has no corresponding (a)–(ak) entry. A future Pi minor that swapped the default for "no built-ins unless requested" (the very change the new `noTools: "builtin" | "all"` option already gestures at), or that treated `tools: []` as "no allowlist; apply defaults," would silently re-enable Pi's `read` / `bash` / `edit` / `write` inside every spawned subagent — exactly the leakage rule 2 exists to prevent — and would surface only as user-visible cross-loom tool contamination, not at load time, not at the SDK surface-inventory assertion, and not at the spawn site.

## Issue introduction

**Verdict:** single-commit-introduced
**Introducing commits:** `69b9c12` (2026-05-04) — *pi-loom spec: resolve "createAgentSession example passes tool definitions through the wrong field"*
**History:** Before `69b9c12`, the spawn-block prose was a one-liner that referenced `createAgentSession({ tools, ... })` without claiming anything about how the field shaped Pi's default tool set. `69b9c12` expanded the block into the present three-rule structure (now grown to six rules) and introduced both the comment `// explicit allowlist suppresses Pi's default built-ins` and the prose "omitting `tools` would re-enable Pi's default built-in `read` / `bash` / `edit` / `write` tools regardless of what the loom declared." `git log -S "explicit allowlist suppresses"` and `git log -S "ambient Pi tools NOT inherited"` both bottom out at `69b9c12`; the surrounding behavioural-presupposition apparatus on the page (items (n), (o), (y), (aj)) was added by later commits that pinned other satellite types but never circled back to pin this one.

## Solution Space

**Shape:** single

### Recommendation

Add a new editorial-review checklist entry under [Pi version bump procedure](../spec_topics/pi-integration-contract/version-bump-step2.md) — appending it as the next available letter in the (a)–(ak) sequence — that audits the `tools`-allowlist suppression behaviour on each Pi minor bump, on the same SHOULD-level footing as items (n) / (o) / (y) / (aj). The new item must cover all three sub-properties the spec depends on, since each can regress independently:

1. An explicit non-empty `tools: [...]` array restricts the subagent's active tool set to exactly those names, with no Pi default built-ins added.
2. `tools: []` produces an empty active tool set (no Pi default built-ins added), distinct from omitting `tools`.
3. Omitting `tools` (and not setting `noTools`) enables Pi's default built-ins (`read`, `bash`, `edit`, `write`) — the property the spec uses to justify "always pass an explicit allowlist."

The audit recipe is the same shape items (aj) and (r) use: read the `tools` / `noTools` branches in `dist/core/sdk.js` against the [Baseline acquisition](../spec_topics/pi-integration-contract/version-bump-step2.md#bump-baseline-acquisition) snapshot and confirm the three sub-properties hold. Phrase the obligation SHOULD-level at loom 1.0 with the standard escalation clause (escalates to a build-time pin once Pi exposes a typed tool-defaulting contract step 2(a) can mechanically verify).

In `subagent.md` rule 2, add a back-reference to the new item — matching how rule 4 cross-references item (o) and PIC-9's `dispose()`-mid-`abort()` paragraph cross-references item (y) — and note that the three sub-properties are individually load-bearing (so the audit MUST confirm all three rather than spot-checking only the "always pass an allowlist" half).

The new `noTools: "all" | "builtin"` field that appears on `CreateAgentSessionOptions` at the loom 1.0 pin is not consumed by loom (the spec mandates an explicit `tools` array), but the audit item SHOULD note its existence so a future Pi reshape that, e.g., made `tools: []` mean "fall back to `noTools` defaulting" is caught by the same review pass. Do not add `noTools` to the spawn call — that would broaden the consumed surface unnecessarily.

Edge cases the implementer must watch:
- The `noTools` field already documents `"builtin"` semantics ("disable the default built-in tools but keep extension/custom tools enabled"). The audit item must distinguish "`tools: []` and `noTools: "all"` both yield an empty active set" from "`tools: []` ≡ `noTools: "builtin"`" — the former is the loom-relied property, the latter is not.
- The audit item is *behavioural*, not signature; pairing it with the existing item (o) (which audits the `tools` field's *signature* alongside the `ResourceLoader` / `SessionManager.inMemory` / `DefaultResourceLoader` signatures) is wrong on the same footing item (aj) is separated from item (o) for `SessionManager.inMemory`. Add it as its own letter.

## Consequence

**Severity:** correctness

A silent Pi-side change to the default-tool / empty-allowlist semantics turns the subagent's promised tool isolation into ambient inheritance of `read` / `bash` / `edit` / `write`, breaking the [state-isolation matrix](../spec_topics/pi-integration-contract/subagent.md#subagent-state-isolation-matrix)'s "ambient Pi tool set" row and the capability-3 isolation posture without any build-time or load-time signal. Two reasonable implementers reading the rule today will also disagree on whether `tools: []` is the conformant way to say "no tools at all" or a request for Pi's defaults — the rule asserts the former, but the SDK type encodes neither reading.
