## Problem

The **Adds** field of `docs/plan_topics/V9e-active-invocation-registry.md` lists "the setup wrap" as a first-class deliverable, but never defines it in the leaf body, and the **Spec** field links `active-invocation-registry.md` as a whole page rather than the bullet that defines the construct. The spec defines this construct under a sharper canonical name — the **Dispatch-site setup wrap** in that page's *Registry contract* section — as the per-insertion-site `try`/`catch` around the pre-evaluation setup sequence whose `catch` arm routes setup throws through the runtime-defect surface. An implementer scoping V9e from its Adds alone cannot recover that scope, and the bare shorthand does not match the spec's canonical name, so it is invisible to naming/term checks.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** `docs/plan_topics/V9e-active-invocation-registry.md` was created in c6a664e (`git log --diff-filter=A --follow`), and the bare phrase "the setup wrap" was present in its Adds enumeration from that first commit (`git log -S'setup wrap' --follow` returns only c6a664e). The three subsequent edits to the file (75b6a9b, 659aa21, cb8737a) did not touch the term. The defect is original to the leaf, not introduced by a later edit.

## Solution approach

In the V9e **Adds** field, rename the bare "the setup wrap" to the spec's canonical term **Dispatch-site setup wrap** (matching the *Registry contract* bullet in `active-invocation-registry.md` that the Spec link already reaches). Add a short gloss describing it as the per-insertion-site `try`/`catch` around the pre-evaluation setup sequence whose `catch` arm routes setup throws through the runtime-defect surface — the insertion-time wrapper only, distinct from the per-invocation `finally`'s entry removal.
