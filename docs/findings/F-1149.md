## Finding

For a runtime-panic `loom-system-note` (the `loom/runtime/*` rows in
`code-registry-runtime.md`, including `loom/runtime/internal-error`),
three distinct strings appear in the spec for the same emission, and
`diagnostic-shape.md` — the canonical home of both DIAG-4 and the
**Serialised content format** rule — never reconciles them:

1. **`Diagnostic.message`** — the row's normative *Message template*
   under DIAG-4 (e.g. `internal error: <error.message>`).
2. **The `content` line** under "Serialised content format" —
   `"<file>:<line>:<col>: <code>: <message>"` (plus optional hint /
   related lines).
3. **The bespoke slash-surface string** named in
   `code-registry-runtime.md`'s prose for the panic-source rows
   (and `internal-error`'s row): `"loom /<name> aborted: <message>"`
   / `"loom /<name> aborted with internal error: <message>"`.

A reader staying inside the diagnostics shard cannot tell which of
the three is on the wire as `content` for a runtime-panic note. The
reconciliation does exist — `pi-integration-contract/runtime-event-channel.md`
**System notes** scopes the `<file>:<line>:<col>: …` format to
parse / load / type batches and pins the `"loom /<name> aborted…"`
framing as the panic `content` — but `diagnostic-shape.md`'s
**Serialised content format** paragraph states the line format
unconditionally for "each `loom-system-note`", with no carve-out and
no link back to the runtime-event-channel scoping. Two implementers
reading the diagnostics shard alone will diverge: one emits the
serialised triple-prefixed line as `content` for panic notes; the
other emits the bespoke `"loom /<name> aborted…"` framing. Both can
honestly cite the spec.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `ac8c3af` (2026-05-04) — *resolve "LoadExtensionsResult.errors is the wrong delivery channel for loom diagnostics"*; `f8f4813` (2026-05-05) — *resolve "Panic routing has no defined path for unexpected interpreter exceptions"*; `5c8d782` (2026-05-06) — *resolve "Panic routing through loom-system-note is under-specified across two pages"*
**History:** `ac8c3af` introduced the **Serialised content format**
paragraph in `spec_topics/diagnostics.md`, stating the
`"<file>:<line>:<col>: <code>: <message>"` line format
unconditionally as the `content` of "each `loom-system-note`". The
next day `f8f4813` added the `loom/runtime/internal-error` registry
row carrying, in its Trigger prose, the bespoke slash-surface string
`"loom /<name> aborted with internal error: <message>"` — a third
string for the same emission, with no edit to the **Serialised
content format** paragraph above. The day after, `5c8d782`
performed the cross-page reconciliation in
`spec_topics/pi-integration-contract.md`: it scoped the
serialised-line format to parse / load / type batches in the
`details`-variant list, added the runtime-panic row to the
per-variant `display` / `content` matrix with the `"loom /<name>
aborted…"` framing, and rewrote the panic-emission bullet under
**Deduplication and lifetime rules** to pin the framing as the
panic `content`. That reconciliation was never back-ported into
`diagnostic-shape.md`'s **Serialised content format** paragraph,
which still reads as unconditional today (carried forward through
the `spec_topics/diagnostics.md` → `spec_topics/diagnostics/diagnostic-shape.md`
split). The defect is the omitted edit at the diagnostics shard's
own source-of-truth paragraph, not any single one of the three
commits in isolation.

## Solution Space

**Shape:** single

### Recommendation

Amend `diagnostic-shape.md`'s **Serialised content format**
paragraph to scope the `"<file>:<line>:<col>: <code>: <message>"`
line format to parse / load / type batches and to pin that for a
runtime-panic note (a single-element `Diagnostic[]` carrying a
`loom/runtime/*` code) the `content` field is instead the
user-facing framing defined in
`pi-integration-contract/runtime-event-channel.md` **System notes**
and `errors-and-results/error-model.md` **Runtime panics** — the
existing `"loom /<name> aborted: <message>"` form (or `"loom
/<name> aborted with internal error: <message>"` for
`loom/runtime/internal-error`), with `<message>` interpolated from
the registered DIAG-4 Message template. Frame this explicitly as
three nested, non-competing roles for the three strings:
`Diagnostic.message` carries the registered template
character-for-character; the framing wraps that template to produce
`content` for runtime-panic notes; the
`<file>:<line>:<col>:`-prefixed serialised line is the `content`
form for parse / load / type batches only. Link the runtime-panic
clause out to `runtime-event-channel.md`'s per-variant
`display` / `content` matrix and the panic-emission bullet under
**Deduplication and lifetime rules** rather than restating the
framing rule (those pages are the existing source of truth — keep
the single owner). Touch `code-registry-runtime.md` only to delete
the parenthetical wire-format prose in the `internal-error` and
panic-source rows that re-states the framing, replacing it with a
pointer to the same matrix, so the framing string lives at one
site rather than three.

Edge cases the implementer must watch: the `display: false` panic
case does not exist in loom 1.0 (all runtime-panic group-B
emissions are `display: true` per the matrix), but the carve-out
must remain shape-disjoint from the `details: { event }`
group-A `display: false` + `content: ""` rule; the fallback chain
to `ctx.ui.notify` and `loom/runtime/system-note-delivery-failed`
re-emits `content` verbatim, so a scoping edit that changes the
panic `content` shape would silently change what
`ctx.ui.notify(content, "error")` shows — leave the framing
literal unchanged from the runtime-event-channel pin; the five
`console.error`-routed teardown-handler codes (`reload-teardown-timeout`,
the four `loom/host/session-shutdown-*`) are out of scope here —
their wire shape is owned by **Diagnostic-emission isolation** and
its per-row construction-site / serialiser-throw fallback clauses,
not by the **Serialised content format** paragraph.

## Consequence

**Severity:** correctness

An implementer holding only `diagnostic-shape.md` and
`code-registry-runtime.md` open will produce one of two
wire-incompatible `content` values for every panic-source note — the
serialised `<file>:<line>:<col>: <code>: <message>` line or the
bespoke `"loom /<name> aborted…"` framing — and a conformance test
asserting either will let the other through. `internal-error` is
the most exposed surface (the runtime-defect catch site is the most
likely to be exercised by hostile input), but every one of the
six panic-source rows shares the defect.
