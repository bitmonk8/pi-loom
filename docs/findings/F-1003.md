## Problem

`host-prerequisites.md` item 4 (*Pi-supplied `AbortSignal`*) opens with an unqualified claim that Pi delivers an `AbortSignal` to "every extension entry point loom uses", naming `ctx.signal` on the slash-command `ExtensionCommandContext` alongside `tool.execute(...)`'s `signal` parameter. That qualification is wrong for the slash-command surface: `host-interfaces-core.md` declares `ExtensionContext.signal: AbortSignal | undefined` and its **Cancellation source** rule establishes that a slash-command handler fires from idle, so `ctx.signal` is `undefined` at handler entry — which is why the runtime wraps a fresh `loomAbort` controller rather than using `ctx.signal` directly. A reader consuming the prerequisite top-down without reaching **Cancellation source** has no reason to guard against `undefined` and may infer `ctx.signal` can be plumbed straight into adapter call sites.

## Issue introduction

**Verdict:** present-since-inception (of the four-prerequisite block)
**Introducing commits:** `022bff9` (2026-05-06) — *pi-loom spec: resolve "Orientation prerequisites + Host runtime + pi-integration-contract Host prerequisites"*
**History:** `git log -G "Pi delivers an" -- docs/spec_topics/pi-integration-contract/host-prerequisites.md` (and the predecessor path `spec_topics/pi-integration-contract.md` traced via `--follow`) shows the four-prerequisite **Host prerequisites** block first appearing in `022bff9`. The introducing wording — *"Pi delivers an `AbortSignal` to every extension entry point loom uses — `ctx.signal` on the slash-command `ExtensionCommandContext`, the `signal` parameter of `tool.execute(...)`, and the `signal` option of `createAgentSession({ signal })`"* — was already unqualified at that commit, while the same commit's **Cancellation source** paragraph (preserved unchanged from the prior single-page layout) already documented *"Pi documents `ctx.signal` as `undefined` in idle / non-turn contexts … a slash-command handler fires from idle, so the runtime MUST NOT use `ctx.signal` as the loom's authoritative signal"*. The contradiction is therefore present from the block's first commit. Subsequent commits touched item 4 without addressing it: `307927d` (2026-05-06) split the Host runtime paragraph upstream in `spec.md` and incidentally enumerated the `AbortSignal` member set in obligation 2 there but did not edit this page; `5c8e630` (2026-05-07) rewrote item 4's `createAgentSession` clause for the subagent-no-`signal`-option fix without touching the slash-command clause's qualification; `f5e89f4` (2026-06-04) split `spec_topics/pi-integration-contract.md` into the current subdirectory layout (`host-prerequisites.md` as one of the children) without content change. The defect predates the file split.

## Solution approach

Rewrite item 4's `ctx.signal` clause in `host-prerequisites.md` to pin the same `AbortSignal | undefined` typed surface and the "undefined at handler entry / supplied during an active turn" temporal shape that `host-interfaces-core.md` already pins, anchor-citing that page's `ExtensionContext.signal` declaration (`#extensioncontext-interface`) rather than restating the typed surface. Keep the `tool.execute(...)` `signal` clause stated as unconditional delivery, and let **Cancellation source** retain ownership of the `loomAbort` wrapping rule.

## Solution constraints

- The `tool.execute(...)` `signal` clause MUST remain stated as unconditional per-call delivery — the asymmetry against the SDK's `AbortSignal | undefined` typing is real and MUST NOT be weakened to conditional delivery.
- Out of scope: item 4's trailing hand-off sentence (`this prerequisite is what guarantees the inputs that rule consumes`) and its Step 0 (b) anchor citation (`#entry-capability-probe`) for the WHATWG `AbortSignal`/`AbortController` shape — both are consumed downstream and MUST survive the rewrite intact.
