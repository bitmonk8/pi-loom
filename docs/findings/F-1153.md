## Finding

The **Transient toasts (auxiliary)** paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` enumerates three named trigger sites — "the chokidar watcher itself throwing, an unrecoverable settings I/O exception that no `loom/load/settings-*` code covers, an internal extension invariant violation" — and specifies a single thing about them: that they reach the user via `ctx.ui.notify(message, "error")`, and that an inner throw from `ctx.ui.notify` is swallowed. Surfacing is pinned; **what happens next** at each of the three sites is not.

Three independent normative gaps live in that bundle:

1. **Watcher-throw recovery posture.** When the chokidar watcher throws, the paragraph is silent on whether hot-reload survives. A reasonable implementer can leave the watcher subscribed and continue (a transient `notify` is purely informational), tear it down and continue without hot-reload (next reload requires `/reload`), or tear down and mark the extension `"degraded-needs-reload"` via the existing `LoomRegistry` drain-state tag. Each is consistent with the surfacing rule; each gives the operator a different post-throw system. The companion finding *"registration-steps.md & host-interfaces-services.md — testability: collision-source named-set MUST and watcher post-error posture stated in implementation terms"* identifies the same gap on the chokidar-adapter `error`-event path that routes into this same toast surface (per `host-interfaces-services.md` PIC-14).

2. **Settings-IO exception effect on the in-progress scan.** "An unrecoverable settings I/O exception that no `loom/load/settings-*` code covers" can be raised mid-scan from the settings-file read path (the file existed at first-stat, then disappeared / EIO'd before parse; a non-`loom/load/settings-unreadable` exception class that no catch arm in `discovery/package-and-settings.md` enumerates). The paragraph does not state whether the scan aborts, continues against the built-in `settings.json` defaults, or continues against the last successfully merged settings snapshot. The choice is observable: it determines whether discovery completes with a partial loom registry, with an empty registry, or not at all.

3. **Toast `message` content.** `ctx.ui.notify(message, "error")` is the only user-visible string the loom extension emits along this entire surface, and the paragraph carries no contract for it — no template, no enumeration of allowed substrings, no "non-normative freeform" carve-out. Two conformant implementations can ship `message`s that share no tokens, and a conformance test has nothing to assert against.

Each gap is independently resolvable; their solutions touch different sibling pages (post-throw posture belongs near the watcher registration in `pi-integration-contract/registration-steps.md` and the watcher seam in `pi-integration-contract/host-interfaces-services.md`; the scan-continuation rule belongs in `discovery/package-and-settings.md` alongside the other settings-read failure modes; the `message` contract is local to `diagnostic-shape.md`). Bundling them in one paragraph hides three under-specifications behind one umbrella.

## Solution Space

**Shape:** multiple

This finding bundles three independent obligations; each is resolvable on its own and each touches a different sibling page. They are sequenced so the smaller, scope-bounding edits land first and the wider-surface watcher-posture edit lands on a stable baseline.

### Option A — Pin the toast `message` contract (smallest, local edit)

**Approach.** Amend the *Transient toasts (auxiliary)* paragraph to fix the `message` content contract for each of the three trigger sites. Two viable shapes:
- **A1 (normative templates).** Define a short normative template per trigger site (e.g. `"loom watcher: <error.message>"`, `"loom settings: <error.message>"`, `"loom internal: <description>"`), with the same first-line-truncation rule the §8 placeholder-rendering category uses for `<host-tail>` so non-finite host strings cannot blow the toast line.
- **A2 (declare freeform).** State explicitly that the `message` content is non-normative freeform diagnostic text, that conformance tests MUST NOT assert against its bytes, and that the only normative obligation is the `type === "error"` argument plus the `notify`-throw swallow rule already pinned.

**Spec edits.** `diagnostics/diagnostic-shape.md` only. No other page touched.

**Pros.** Smallest diff; resolves the testability gap (a conformance harness either has bytes to assert against, or an explicit "don't assert" carve-out); is a precondition for resolving B and C without re-litigating the same paragraph.

**Cons / Risks.** A1 forces a template choice the original author may have intentionally left to the implementer; A2 declares a contract gap rather than closing it, which a stricter reviewer can reject as ducking the obligation.

### Option B — Pin the settings-IO scan-continuation rule

**Approach.** Decide what the in-progress discovery scan does when a settings-file read raises an exception class that no `loom/load/settings-*` code covers, and state the rule on the page that owns the settings-read failure modes (`discovery/package-and-settings.md`, in or adjacent to the *Failure modes* table and the *Watcher-time reload failures* paragraph). Recommended default: **continue the scan against the last successfully merged settings snapshot** (or the built-in defaults if none has been merged yet); the scan emits its normal `loom/load/*` diagnostics for everything else and the toast fires once for the uncovered exception. Add a one-sentence back-reference from the *Transient toasts* paragraph in `diagnostic-shape.md` so the rule is discoverable from the surfacing site.

**Spec edits.** `discovery/package-and-settings.md` (the new continuation rule); `diagnostics/diagnostic-shape.md` (back-reference). The settings-re-merge re-run arm of the same rule fits naturally into the *Watcher-time reload failures* paragraph already on that page.

**Pros.** Removes one of the two non-local observability divergences; aligns with the existing pattern in `package-and-settings.md` where each settings-read failure class has a continuation rule (silent absent, `loom/load/settings-*` non-fatal, *Top-level shape validation* "treated as absent").

**Cons / Risks.** Choosing the "continue against last snapshot" arm means the scan can complete against settings the user already edited; that is consistent with the rest of the file's "treated as absent" pattern but is a policy choice and should be called out as such in the rule.

### Option C — Pin the watcher-throw post-error posture (largest surface)

**Approach.** Decide the post-throw lifecycle of the chokidar watcher and state the rule at the watcher-registration site. Three viable arms:
- **C1 (continue).** The watcher remains subscribed; the toast is purely informational. Cheap to specify; assumes chokidar's own recovery is sufficient (the load-bearing assumption is worth stating explicitly).
- **C2 (tear-down + `/reload`).** The watcher's `Unsubscribe` runs; hot-reload is dead until the user runs `/reload`; the toast `message` directs the operator to run `/reload`. Mirrors the existing structural-change-note pattern (which already prompts `/reload`).
- **C3 (degraded tag).** The watcher tears down and the runtime calls `LoomRegistry`'s `"degraded-needs-reload"` setter (the tag already exists per `drain-state-contract.md`), so the next slash invocation surfaces the `"loom /<name>: extension degraded; /reload to recover"` arm rather than dispatching against a stale registry.

**Spec edits.** `pi-integration-contract/registration-steps.md` step 5 (the rule) and `pi-integration-contract/host-interfaces-services.md` PIC-14 (the chokidar-adapter `error`-event route to the toast surface, which currently states the route without stating the posture). Either page can own the rule; the natural home is `registration-steps.md` because step 5 already owns the watcher lifecycle. `diagnostics/diagnostic-shape.md` gains a back-reference. This same edit resolves the watcher-posture half of the companion *"registration-steps.md & host-interfaces-services.md — testability"* finding.

**Pros.** Closes the most consequential of the three divergences (a silently dead watcher behaves indistinguishably from a live one until the user notices nothing reloads, which is exactly the operator-visibility class the toast surface is supposed to defend against).

**Cons / Risks.** C1's "continue" arm depends on a chokidar behavioural property the spec does not currently pin (whether the watcher recovers from an internal throw); auditing that is itself a Pi-prerequisite-style obligation. C3's "degraded tag" arm interacts with the *"drain-state-contract.md — no precedence/transition rule between the two drain-state tag values"* finding — a new writer of `"degraded-needs-reload"` increases the urgency of that finding's precedence rule. Of the three arms, **C2 is recommended**: it mirrors the existing structural-change-note `/reload` pattern, sidesteps the chokidar-recovery assumption, and does not depend on the drain-state-tag precedence rule being resolved first.

### Recommendation

Resolve the three obligations as **three separately loopable edits**, in the order **A → B → C**, so each fix lands on a baseline the previous one has already stabilised:

1. **A first** (smallest, local; recommended sub-shape **A1** with the §8 first-line-truncation borrow — it actually closes the contract, where A2 only declares it open). The diff is one paragraph and unblocks meaningful conformance assertions on the surfacing rule.
2. **B second**, with the "continue against last merged snapshot, defaults if none" arm called out explicitly. The diff is two paragraphs in `package-and-settings.md` plus a one-line back-reference in `diagnostic-shape.md`.
3. **C third**, recommended arm **C2** (tear down + `/reload` prompt). The diff is one paragraph in `registration-steps.md` step 5, one sentence updating the PIC-14 `error`-route prose, and one back-reference in `diagnostic-shape.md`. Watch the *"in-flight invocation behaviour under watcher hot-reload/removal is unowned"* finding — C2's "watcher torn down" arm is precisely the case that finding asks an owner for, so the two edits should be sequenced so this one does not retract a posture the other finding then has to re-author.

The implementer addressing each obligation should preserve the existing surfacing rule and the `notify`-throw swallow rule verbatim; only the recovery/continuation/message obligations are new.

## Consequence

**Severity:** correctness

Two conformant implementations can diverge observably on each of the three sites: one can shut hot-reload down on a watcher throw while another keeps it live; one can drop the scan on an uncovered settings-IO error while another continues against defaults; both can ship `notify` text that shares no tokens. None of those divergences is caught by any of the existing `loom/load/*` / `loom/runtime/*` registry rows, because by construction this surface fires when no registry row applies. Discovery completeness, hot-reload survival, and operator-actionability of the user-visible string all rest on choices the spec currently leaves to the implementer.
