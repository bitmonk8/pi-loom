## Finding

Spec [`type-system.md` TYPE-11](../spec_topics/type-system.md) (anchor `#type-11`) defines alias-schema transparency and then closes a specific loophole: the alias is identified solely by the `schema X = R` `=` form, *not* by what `R` resolves to, so aliasing an object schema (`schema Y = SomeObjectSchema`) "unfolds to that object schema, which then participates under the nominal rules of TYPE-10, so alias transparency never reopens TYPE-10's nominal case."

The plan splits TYPE-11 across the paired leaves asymmetrically. The tests leaf `V2b-T` carries the loophole-closing clause — its TYPE-11 bullet ends "…aliasing an object schema still re-enters TYPE-10's nominal case." The implementation leaf `V2b` does not: its TYPE-11 bullet states only "a `schema X = R` named type unfolds to its RHS for `⊑`, recursing through nested aliases (identified by the `=` form, not by what `R` resolves to)." A reader of `V2b`'s bullet alone sees unconditional, transparent unfolding and could implement structural admission through an object-schema alias — exactly the cross-named structural admission TYPE-10 forbids.

The omission is a description gap between two paired leaves and against the spec, not a missing test: `V2b-T` already names the nominal re-entry as a TYPE-11 sub-case, and `coverage-matrix.md` correctly maps `TYPE-1 … TYPE-11` → `V2b`. The fix aligns `V2b`'s TYPE-11 prose to the contract its partner test and the spec already state.

The asymmetry — the impl leaf omitting the nominal re-entry clause its test partner carries — has been present since TYPE-11 was first introduced, originating in that one commit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V2b-type-compat-engine.md`, extend the `TYPE-11` Tests bullet so it states the object-schema alias re-entry case, matching the spec's TYPE-11 loophole and `V2b-T`'s bullet. Append to the existing bullet the clause: `when R is itself a named object-schema type, alias unfolding re-enters TYPE-10's nominal case — no cross-named structural admission through an alias`. Keep the existing "identified by the `=` form, not by what `R` resolves to" phrasing; the new clause is the consequence of that identification for the object-schema RHS. No spec edit is required (TYPE-11 already states this) and the `coverage-matrix.md` mapping is already correct.

## Consequence

**Severity:** advisory

An implementer reading only `V2b`'s TYPE-11 bullet could implement unconditional structural unfolding through an object-schema alias, admitting a cross-named structural pair TYPE-10 forbids. The paired `V2b-T` TYPE-11 sub-case ("aliasing an object schema still re-enters TYPE-10's nominal case") is authored red-first under the TDD ritual and backstops a green ship, so a wrong leaf cannot actually pass; the live defect is that `V2b`'s prose diverges from both spec TYPE-11 and its own test partner, weakening the impl leaf as a standalone contract.
