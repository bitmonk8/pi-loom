## Finding

`schema-subset.md` §"Canonical schema hash" step 2 (the canonical-form bullet for numeric literals in `const`/`enum` positions) does not specify the integer/number rendering algorithm in place. Instead it borrows it by cross-reference from a binder sub-page — [Binder — Echo policy](../spec_topics/binder/defaulting-system-note-echo.md#echo-policy) *Format rules*, citing BNDR-4 / BNDR-5 / BNDR-6 — and then has to disclaim, in the same bullet, which parts of that page do *not* travel with the borrow ("Only that rendering algorithm is borrowed: the binder's 120-code-point line cap and `(default)` suffix are echo-formatter concerns and do not apply to the canonical-hash recipe").

The need for that disclaimer is the symptom: a cross-cutting, on-disk/on-wire serialisation rule has been parked inside one of its consumers (the binder argument-echo formatter) rather than at a neutral location, leaving every other consumer to (a) reach across the spec into a feature-specific page and (b) fence off the parts of that page that don't apply. The schema canonical hash and the binder echo are the two known consumers today; any future consumer that needs the same "shortest round-tripping fixed-point, no scientific notation, `-0`→`0`" pin (e.g. a diagnostic placeholder renderer, a fixture serialiser, a deterministic-replay payload formatter) will inherit the same awkwardness, and any change to the binder page risks accidentally moving the canonical hash's on-wire bytes.

The plan corpus has already factored the renderer as `V2d` (a shared pure function consumed by both `V5f` and `V11h`), which confirms the same architectural reading: there is one rule with two callers and it deserves one owner.

**Verdict:** single-commit-introduction (within a multi-commit history)
**Introducing commit:** `c5bb5e1` — *pi-loom spec: resolve "Canonical schema hash numeric serialization underspecified"* (2026-06-03)
**History:** The §"Canonical schema hash" section was first authored by `e21449b` (2026-05-04, resolving *"__inline_<hash> hash function is not pinned"*); at that point step 2 specified numeric serialisation in place with a single underspecified bullet ("numeric literals in their JSON Schema integer/number form (no trailing zeros, no exponent unless necessary to represent the value)"). `c5bb5e1` (2026-06-03) replaced that bullet with the present cross-reference to Binder — Echo policy plus the "Only that rendering algorithm is borrowed…" disclaimer, fixing the underspecification by borrowing the binder's pinned algorithm. That commit chose the binder echo page as the canonical home rather than extracting the rule to a neutral location, and that choice is what the present finding objects to. `111e533` (2026-06-07, resolving *"Canonical hash recipe cites keywords the lowered subset never emits"*) refined the bullet further — dropped `default` from the list of borrowing positions and added the HTML-comment justification — but did not move ownership; the borrow-with-disclaimer shape persists.

## Solution Space

**Shape:** single

### Recommendation

Extract the integer/number rendering algorithm to a neutral normative location and have both `schema-subset.md` §Canonical schema hash and `binder/defaulting-system-note-echo.md` §Echo policy cite it.

- **New home.** Add a short normative sub-section (working title *Canonical numeric serialisation*) to `docs/spec_topics/lexical.md`, immediately after **Number literals**. Lexical is the natural host because it already owns the `integer`/`number` literal *types* and the rule is a pure value→bytes function with no feature-specific framing; the new sub-section sits adjacent to its only structural dependency (the literal-type definitions) and is reachable from every consumer without crossing into a feature page.
- **Content of the new sub-section.** Promote, verbatim, the normative substance currently authored at BNDR-4 (canonical base-10 integer rendering; `-0`→`0`; no leading zeros / separators / exponent) and BNDR-5 (shortest round-tripping fixed-point for `number`; both `String(n)` exponential switches expanded; `-0`→`0`; integral `number` carries no `.0`; non-integral carries ≥1 fractional digit), plus the rule that the algorithm takes an explicit `integer`-vs-`number` kind discriminator from its caller (never from runtime integrality — preserving the discriminator pattern `V2d` already encodes). The BNDR-5 non-normative *Derivation recipe* moves with the rule.
- **Anchors.** Give the sub-section two stable page-local sub-letter anchors for the integer and number rules so existing BNDR-4 / BNDR-5 callers can rewire without losing per-rule cite granularity.
- **`binder/defaulting-system-note-echo.md` §Echo policy.** Replace the BNDR-4 / BNDR-5 *Format rules* bullets with one-line forward-references to the new lexical anchors. Keep BNDR-6 (the reference-renderings table) on the echo page: most rows are echo-specific framing (`(default)` tagging, unquoted-vs-quoted strings, array `…+N more` truncation, the 120-code-point line cap), and rows that exercise the numeric pins explicitly (`BNDR-6o`–`BNDR-6s`) become per-implementation cross-traffic that the citing-test gate `H5f` already covers.
- **`schema-subset.md` §Canonical schema hash step 2.** Replace the borrow-plus-disclaimer bullet with a direct cite to the new lexical sub-section. The disclaimer disappears entirely — there is nothing echo-formatter-shaped to fence off.
- **Plan adjustment.** `V2d`'s **Spec.** field switches from the binder echo page to the new lexical sub-section; `V5f` and `V11h` keep their per-citation links but now reach the renderer rule directly rather than via the binder page. No test or **Ships when.** clause changes — the renderings asserted at `V2d-T`, `V5f-T`, and `V11h-T` (BNDR-6 reference renderings, the `1e21` / `1e-8` / `5e-8` interior vectors, the `-0`→`0` pin) all remain the same bytes.

Edge cases the implementer must watch:

- Preserve the discriminator-from-caller invariant: the lexical-side rule must say the algorithm receives the kind discriminator from the caller (`V5f` derives it from the JSON-Schema `type` of the enclosing `const`/`enum` position; `V11h` derives it from the bound value's static type via the value model), so that integrality of the numeric value at runtime never drives BNDR-4-vs-BNDR-5 selection.
- Keep the schema-subset bullet's note that the canonical hash sorts object keys but never reorders array elements — that is *not* part of the numeric rule and must stay at schema-subset, not migrate to lexical.
- The HTML comment in schema-subset step 2 (about `default` and integer-valued structural keywords being deliberately absent from the recipe) is independent of this finding and is handled separately (see *Related Findings*).

## Consequence

**Severity:** advisory

No implementer is presently misled — the cross-reference is explicit and the disclaimer is precise — so no concrete behaviour diverges. The cost is architectural: the canonical-hash recipe and the binder echo are coupled through a non-obvious shared dependency that lives inside one of them, the disclaimer must be maintained on every future borrow, and the binder page's evolution is silently constrained by an on-wire contract owned elsewhere.
