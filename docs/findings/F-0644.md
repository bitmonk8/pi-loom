## Finding

`code-registry-parse.md` introduces the four-page registry with the sentence "The column legend below applies to all four tables", then defines the columns as `Code | Sev | Phase | Trigger | Spec rule | Hint | Message`. The `loom/load/*` continuation page faithfully uses the same seven-column header. The `loom/host/*` page, however, declares its table as `Code | Severity | Phase | Trigger | Anchor | Message` — *Sev* is spelled out as *Severity*, *Spec rule* is renamed *Anchor*, and the *Hint* column is omitted entirely.

Two of those three deviations contradict the legend directly: the legend pins *Sev* and *Spec rule* as the header labels for every table, and a reader landing on `code-registry-host.md` from a deep link encounters columns whose names do not match the schema the legend says governs them. The third deviation (the missing *Hint* column) is defensible — none of the four host-namespace rows carries an author-facing hint, since these codes route via `console.error` to operators rather than through the author-facing diagnostic channel, so the column would be uniformly empty. But that is an editorial choice the legend has not authorised, and it leaves DIAG-4's "Message column" referent ambiguous on this page in the same way DIAG-3's stability claim becomes ambiguous when the column carrying the codes is silently renamed.

## Issue introduction

**Verdict:** multi-commit-interaction

**Introducing commits:**
- 9559e19 (2026-05-07) "pi-loom spec: resolve \"session_shutdown unknown reason: no closed-set validation, no fail-safe\"" — added the initial `loom/host/*` subsection with header `Code | Severity | Phase | Trigger | Anchor | Message`.
- 3059a0e (2026-06-05) "pi-loom spec: resolve \"Diagnostic registry pointers cite the wrong file and overclaim completeness\"" — rewrote the parse-page intro to introduce the "The column legend below applies to all four tables" claim, retroactively making the pre-existing host-table headers (and the runtime-table's *Message template*) non-conformant.

**History:** The host table's header row entered the corpus on 9559e19 alongside three sibling tables that did not yet share a normative legend; at that point the divergence was a per-table stylistic choice, not a contradiction. Commit 3059a0e a month later added the four-tables legend (resolving a separate finding about pointer accuracy and completeness) without sweeping the existing host-table and runtime-table headers into the new schema. `git blame` on `code-registry-host.md`'s header line confirms 9559e19 as its origin; the line has been textually untouched across the subsequent splits (f5e89f4 moved the subsection into its own file) and host-namespace additions. The defect is the interaction of these two commits — neither is individually defective — so the durable fix is to align the headers with the legend (or vice versa) rather than to revert either commit.

## Solution Space

**Shape:** single

### Recommendation

Edit the `loom/host/*` table header in `docs/spec_topics/diagnostics/code-registry-host.md` to read `| Code | Sev | Phase | Trigger | Spec rule | Hint | Message |`, with the markdown alignment row widened to seven columns, and add a `—` (em-dash) cell to the *Hint* column of every host-table row (all four current rows carry no author-facing hint, matching the existing `loom/load/*` convention for hint-less rows). The renames are pure header-row edits; the *Anchor* → *Spec rule* change does not affect the linked targets in the renamed column (each host row already cites a Pi Integration Contract anchor in the form the legend expects). No row data, no diagnostic semantics, no cross-references outside the table are touched.

Implementer-relevant edge cases:
- The fix must land in the same change as the sibling `code-registry-runtime.md` *Message template* → *Message* rename so the four tables become uniform in a single sweep; a partial sweep leaves the legend's "all four tables" claim still locally false on the un-swept page.
- Any test or extractor that keys on the host page's current header labels (`Severity`, `Anchor`) must be repointed to the legend labels in the same commit. The `V7b-T` test leaf's harness is the most likely consumer.
- The `placeholder-rendering-b.md` cross-references to "[Code registry — host]" cite the page, not the column labels, and need no edit.

## Consequence

**Severity:** advisory

A reader or table-extraction tool that lands on `code-registry-host.md` from a deep link or cross-page reference sees columns whose names do not match the legend that supposedly governs them, and DIAG-4's "Message column" referent has no on-page anchor for the host namespace. No diagnostic emission changes, but the legend's "applies to all four tables" claim is locally false and a header-keyed registry extractor would either fail or silently mis-parse the host page.
