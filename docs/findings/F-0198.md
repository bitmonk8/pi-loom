## Finding

Several `coverage-matrix.md` rows map one numbered REQ-ID to two or more closing leaves, where each leaf closes a **distinct facet** of the obligation rather than re-closing the whole rule. The split rows are: `DISC-4` (`V10a` collision-detection facet + `V9m` superseded-entry-dispatch facet), `CIO-1` (`V16a` + `V4e` + `V11f`), `CIO-5` (`V16a` + `H7a`), `ERR-17` (`V4d` + `V13d`), `ERR-19` (`V4d` + `V13c`), `PIC-17, PIC-18` (`V9c` + `V15d`), and `CNCL-4` (`V17a` + `V9g`).

Neither closing-gate arm asserts that every facet of such a row is independently covered. The H5a citing-test reconciliation (`conventions.md` §REQ-ID discipline) is a best-effort inline-`PREFIX-N`-citation scan that "certifies that a citing test exists for each mapped REQ-ID" — one citing test anywhere in the corpus satisfies the per-REQ-ID check, regardless of which facet it exercises. The H5d transitive-completeness arm is explicitly weaker still: it requires "AT LEAST ONE of the leaf IDs that closing-leaf cell lists" to be reachable from `H5b`'s `Deps.`, and documents that a multi-leaf cell "stays green as long as one listed leaf is present."

Consequently a split REQ-ID can ship with one named facet fully tested and the other facet carrying no citing test at all, and both gates stay green. The matrix records the facet split (some rows even annotate it parenthetically), but nothing reconciles that the recorded facets are each closed. Partial coverage passes as full coverage.

**Verdict:** present-since-inception (of the citing-test gate)
**Introducing commits:** `953e3fa` (2026-06-10, "pi-loom plan: resolve \"closing gate checks numbered-REQ-ID matrix mapping, not test assertion\""); reinforced by `617a3ad` (2026-06-16, "pi-loom plan: resolve \"H5a bundles gate arms spanning four data sources\"")
**History:** `git log -S "certifies that a citing test exists" -- docs/plan_topics/conventions.md` shows the citing-test reconciliation entered the corpus in a single commit, `953e3fa`. From that first introduction it was scoped per-REQ-ID ("certifies that a citing test exists for each mapped REQ-ID"), never per-facet — the joint-completeness gap is intrinsic to the granularity the gate has always had, not a regression a later edit introduced. The multi-leaf split rows it fails to fully reconcile (e.g. `DISC-4`, `ERR-17`) predate that commit in `coverage-matrix.md`. `617a3ad` later split the transitive-completeness arm into `H5d` with an explicit at-least-one multi-leaf rule ("stays green as long as one listed leaf is present"), entrenching the same per-row-not-per-facet semantics by design. No commit ever narrowed either arm to per-facet granularity.

## Solution Space

### Recommendation

Add a new item to the `conventions.md` §"Loom 1.0 release-time residue inspection" checklist (alongside the existing items 1–6) covering a split-row joint-completeness sweep. The item is clean when every facet of every multi-leaf `coverage-matrix.md` row has its own distinct, facet-naming citing test on that facet's closing leaf. To determine a row's facet partition, the reviewer reads the row's parenthetical facet annotations where present and otherwise takes the closing-leaf set minus any co-witness leaf as the facet partition.

Apply the partition rule per row: rows carrying parenthetical facet annotations (`DISC-4`, `PIC-17/18`) supply the facet list verbatim; rows without annotations (`ERR-17`, `ERR-19`, `CIO-1`, `CNCL-4`) take the closing-leaf set as the facet partition, excluding any leaf that is a co-witness rather than an independent facet (e.g. `CIO-5`'s integration co-witness `H7a`, and `CIO-1`'s `V4e`/`V11f`, which the `H5d` arm already exempts from `H5b` `Deps.`).

In the §REQ-ID discipline citing-test paragraph, add a pointing clause noting that per-REQ-ID citing-test existence does not certify per-facet completeness for multi-leaf rows, routing that residue to the new inspection item.

No spec edits are required. This places fidelity for the joint-completeness gap with the release-time reviewer, consistent with the citing-test scan already being conceded as existence-only with fidelity routed to manual review.

## Consequence

**Severity:** correctness

A multi-leaf split REQ-ID can ship with one named facet entirely untested while the gate stays green on another facet's citing test, so partial coverage passes as full coverage. Two reasonable implementers diverge — one writes a facet-naming citing test on each closing leaf, the other relies on the single per-REQ-ID gate check and leaves a facet unverified — and the gap rides to the loom 1.0 release gate undetected.
