## Finding

Several `coverage-matrix.md` rows map one numbered REQ-ID to two or more closing leaves, where each leaf closes a **distinct facet** of the obligation rather than re-closing the whole rule. The split rows are: `DISC-4` (`V10a` collision-detection facet + `V9m` superseded-entry-dispatch facet), `CIO-1` (`V16a` + `V4e` + `V11f`), `CIO-5` (`V16a` + `H7a`), `ERR-17` (`V4d` + `V13d`), `ERR-19` (`V4d` + `V13c`), `PIC-17, PIC-18` (`V9c` + `V15d`), and `CNCL-4` (`V17a` + `V9g`).

Neither closing-gate arm asserts that every facet of such a row is independently covered. The H5a citing-test reconciliation (`conventions.md` §REQ-ID discipline) is a best-effort inline-`PREFIX-N`-citation scan that "certifies that a citing test exists for each mapped REQ-ID" — one citing test anywhere in the corpus satisfies the per-REQ-ID check, regardless of which facet it exercises. The H5d transitive-completeness arm is explicitly weaker still: it requires "AT LEAST ONE of the leaf IDs that closing-leaf cell lists" to be reachable from `H5b`'s `Deps.`, and documents that a multi-leaf cell "stays green as long as one listed leaf is present."

Consequently a split REQ-ID can ship with one named facet fully tested and the other facet carrying no citing test at all, and both gates stay green. The matrix records the facet split (some rows even annotate it parenthetically), but nothing reconciles that the recorded facets are each closed. Partial coverage passes as full coverage.

## Issue introduction

**Verdict:** present-since-inception (of the citing-test gate)
**Introducing commits:** `953e3fa` (2026-06-10, "pi-loom plan: resolve \"closing gate checks numbered-REQ-ID matrix mapping, not test assertion\""); reinforced by `617a3ad` (2026-06-16, "pi-loom plan: resolve \"H5a bundles gate arms spanning four data sources\"")
**History:** `git log -S "certifies that a citing test exists" -- docs/plan_topics/conventions.md` shows the citing-test reconciliation entered the corpus in a single commit, `953e3fa`. From that first introduction it was scoped per-REQ-ID ("certifies that a citing test exists for each mapped REQ-ID"), never per-facet — the joint-completeness gap is intrinsic to the granularity the gate has always had, not a regression a later edit introduced. The multi-leaf split rows it fails to fully reconcile (e.g. `DISC-4`, `ERR-17`) predate that commit in `coverage-matrix.md`. `617a3ad` later split the transitive-completeness arm into `H5d` with an explicit at-least-one multi-leaf rule ("stays green as long as one listed leaf is present"), entrenching the same per-row-not-per-facet semantics by design. No commit ever narrowed either arm to per-facet granularity.

## Solution Space

**Shape:** multiple

### Option A — Manual split-row joint-completeness sweep

- **Approach:** Add a checklist item to `conventions.md` §"Loom 1.0 release-time residue inspection" (alongside the existing items 1–6) that sweeps every multi-leaf row in `coverage-matrix.md` and confirms each distinct facet the row names has its own facet-naming citing test on its closing leaf.
- **Plan edits:** In `conventions.md`, add a new release-time-residue-inspection item covering the split-row joint-completeness sweep — clean when every facet of every multi-leaf coverage-matrix row has a distinct citing test on its facet's closing leaf (the reviewer reads a row's parenthetical facet annotations where present, and otherwise takes the closing-leaf set minus any co-witness leaf as the facet partition). Add a pointing clause to the §REQ-ID discipline citing-test paragraph noting that per-REQ-ID citing-test existence does not certify per-facet completeness for multi-leaf rows, routing that residue to the new inspection item.
- **Spec edits:** None.
- **Pros:** Consistent with the established residue-inspection pattern (the citing-test scan is already conceded as existence-only, with fidelity routed to manual review); reviewer judgment absorbs the primary/co-witness asymmetry naturally (e.g. `CIO-5`'s integration co-witness `H7a`, which `H5d` already exempts from `H5b` `Deps.`); small diff.
- **Cons:** Manual; runs only at the loom 1.0 release gate; depends on reviewer diligence; no mechanical fire at earlier footings.
- **Risks:** A missed facet escapes if the reviewer skips a row.

### Option B — Mechanical per-facet citing-test gate arm

- **Approach:** Extend the H5a citing-test reconciliation (or add a sibling H5-series arm) so that, for each multi-leaf row designated all-facets-cite, every listed closing leaf must carry a citing test for that REQ-ID, with co-witness rows annotated to exempt the co-witness leaf from the per-leaf check.
- **Plan edits:** In `coverage-matrix.md`, annotate each multi-leaf row to distinguish all-facets-cite splits from primary+co-witness rows. In `H5a` (and, if the reconciliation reuses the transitive-completeness reader, `H5d`), extend the arm's `Adds.`/`Tests.` to read those annotations, fail when a designated facet leaf has no citing test for the row's REQ-ID, and add the corresponding seeded fixtures.
- **Spec edits:** None.
- **Pros:** Mechanical and deterministic; fires at every footing like the other plan-structural arms.
- **Cons:** Requires a new annotation scheme on `coverage-matrix.md` plus gate logic to read it; larger diff; over-fire risk on co-witness rows; higher fix-loop churn.
- **Risks:** Annotation drift; a mis-tagged co-witness reddens CI.

### Recommendation

Take **Option A**. The plan already routes best-effort gate residue to the release-time residue inspection, and the citing-test scan is explicitly conceded as existence-only with fidelity left to manual review; the joint-completeness gap is the same class of residue. The primary/co-witness asymmetry across the split rows — `DISC-4`'s two facets both warrant citing tests, while `CIO-5`'s `H7a` and `CIO-1`'s `V4e`/`V11f` are co-witnesses the `H5d` arm already exempts from `H5b` `Deps.` — makes a clean mechanical predicate costly relative to the gap's severity, and reviewer judgment handles that distinction directly. Edge cases for the reviewer: rows carrying parenthetical facet annotations (`DISC-4`, `PIC-17/18`) supply the facet list verbatim; rows without annotations (`ERR-17`, `ERR-19`, `CIO-1`, `CNCL-4`) take the closing-leaf set as the facet partition, excluding any leaf that is a co-witness rather than an independent facet.

## Consequence

**Severity:** correctness

A multi-leaf split REQ-ID can ship with one named facet entirely untested while the gate stays green on another facet's citing test, so partial coverage passes as full coverage. Two reasonable implementers diverge — one writes a facet-naming citing test on each closing leaf, the other relies on the single per-REQ-ID gate check and leaves a facet unverified — and the gap rides to the loom 1.0 release gate undetected.
