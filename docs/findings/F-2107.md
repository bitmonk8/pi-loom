## Finding

The `(e)` `typebox` host-shape sub-step in `capability-probe.md` closes with a design-motivation sentence: *"The asymmetry with `AgentSession.prototype.abort` in `(c)` is removed by adding this check: both single-symbol Pi-bundled imports now surface absence as `loom/load/host-incompatible` rather than as a runtime-time `TypeError` at the first tool-registration call."* This is rationale for why the check was added — it constrains nothing an implementer must do.

The normative content of `(e)` is already fully stated earlier in the same paragraph: the `typeof Type.Unsafe === "function"` check, its position in the short-circuit order after `(d)`, the `kind: "typebox-shape"` failure route with `details.observed` / `details.required`, and the no-construction invariant. The trailing sentence adds only an explanation of the historical asymmetry it resolves, which an implementer does not need to produce a conformant probe.

(The original report framed this as a "rationale opener" / "first sentence"; the sentence is in fact the paragraph's trailing sentence. The requirement already begins with "Check `typeof Type.Unsafe === \"function\"`…". The defect — a non-normative rationale sentence embedded in a normative sub-step — is the same regardless of position.)
