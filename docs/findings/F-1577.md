## Problem

In the *Tool execution from loom code* paragraph (`#tool-execution-from-loom-code`), `AgentToolResult`'s declaration is attributed to `dist/core/extensions/types.d.ts` of `@earendil-works/pi-coding-agent`, but that file only re-exports the type; the interface is declared as `AgentToolResult<T>` in `@earendil-works/pi-agent-core`'s `dist/types.d.ts`. This is the lone outlier against the same page's convention, which attributes peer-package types (`AgentMessage`, `UserMessage`, `ToolResultMessage`, `Model<TApi>`) to `@earendil-works/pi-agent-core`'s (or `@earendil-works/pi-ai`'s) `dist/types.d.ts`. Because the sentence drives a build-time surface-inventory assertion and a version-bump re-validation premise, naming the wrong owning package means the re-validation watches the wrong package: a `@earendil-works/pi-agent-core` shape change can land without a `@earendil-works/pi-coding-agent` minor bump and silently miss the drift the assertion promises to catch.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `68ff2d1` — *pi-loom spec: resolve "Pi API surfaces asserted without .d.ts citations"*
**History:** The corpus is git-tracked. `git log -G` / `git log -S` over the envelope-declaration sentence and the phrase "declared on the tool-result type family" show the citation was added by `68ff2d1`, the commit that resolved a finding about Pi API surfaces lacking `.d.ts` citations. The task note that commit resolved (visible in its diff) located the declaration at `dist/core/extensions/types.d.ts` by where the `isError` token appears (lines 323/538/634/729), and the commit carried that file forward into the envelope sentence — citing the file where the token surfaces rather than the file where the `execute()`-return type is declared. The wrong-file citation has been present continuously since that commit; subsequent edits to the same paragraph (`ec6ab5c`, `f5e89f4`, and the scope-rename and SDK-pin sweeps) preserved the citation unchanged. The defect was introduced when the citation was first added, not by a later interaction.

## Solution approach

Rewrite the declaration attribution in the *Tool execution from loom code* paragraph to name `AgentToolResult`'s true declaration site — `@earendil-works/pi-agent-core`'s `dist/types.d.ts` (interface `AgentToolResult<T>`), re-exported through `@earendil-works/pi-coding-agent`'s `dist/core/extensions/types.d.ts` — matching the convention the page already uses for `AgentMessage` and `Model<TApi>`. Reconcile the surface-inventory / version-bump sentence so the field-set re-validation is keyed to the `@earendil-works/pi-agent-core` sibling pin (held transitively via `@earendil-works/pi-coding-agent`'s `dependencies` as a caret range) rather than solely the `@earendil-works/pi-coding-agent` Pi-SDK pin.
