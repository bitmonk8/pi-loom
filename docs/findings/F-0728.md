## Finding

The glossary pins two foundational concepts whose only distinguishing mark is letter case: `loom` (the file unit — what a slash command resolves to, what `invoke(...)` targets, what `mode:` is declared on) and `Loom` (the scripting language whose grammar/semantics the spec defines, shared by `.loom` and `.warp`). Case is a structurally unreliable disambiguator in CommonMark prose: every sentence-initial occurrence, every list-bullet-initial occurrence, and every heading word is capitalised by typographic convention, irrespective of which sense the author intends. The corpus already exhibits this collapse:

- `spec_topics/influences.md` opens "Loom borrows from two languages…" (sentence-initial; the language sense is intended and only inferable from the surrounding "Rust / TypeScript / Loom" enumeration) and immediately follows with the third bullet `**Original to loom**:` (lowercase mid-sentence) whose content — `schema` / `enum` declarations, the `@`...`` query primitive, frontmatter-declared mode, the `.loom`/`.warp` split, the unified callable set — is unambiguously language-level material that should bind to the `Loom` sense.
- `spec_topics/grammar.md` carries `## Loom literal sublanguage` (heading-initial capitalisation forced) and refers to "a legal Loom expression", "the Loom AST", "Loom distinguishes…" throughout — uses that are individually correct under the glossary rule but indistinguishable in form from the same words appearing sentence-initially when the file-unit sense is intended.
- `spec_topics/runtime-value-model.md` carries `| Loom type |` (table-cell capitalisation), "Loom values are represented…", and "Loom code observes `Result` only…" interleaved with body prose that uses lowercase `loom code` for the same referent.

A reader cannot recover the intended sense from a citation, a quoted fragment, or a heading-anchored cross-link, and `grep` over either spelling silently misses the other. The glossary's case-only convention therefore degrades to "rely on the surrounding paragraph", which is exactly what the rest of the glossary is structured to avoid for `caller`, `arm`, and the other ambiguous-when-bare terms — those carry distinct alternative names (`slash caller` / `invoke parent`; `canonical slug` / `alias slug` is the suggested resolution for `arm`). `loom` / `Loom` is the only remaining overload that relies on case alone.

## Issue introduction

**Verdict:** present-since-inception (for the file-unit / language split as a *two-entry, case-distinguished* pair)

**Introducing commit:**

- `53b8970` — *pi-loom spec: resolve "loom / Loom / pi-loom — three referents, no glossary disambiguation"* (Tue May 5 22:33:43 2026 +0200). This commit added the `loom (file unit)` and `Loom (language)` entries to `spec_topics/glossary.md` in their current form, including the rule "Capitalised when used as a proper noun … the lowercase form is reserved for sense 1 above." The commit also retitled `overview.md`'s `### Scope of a Loom File` → `### Scope of a loom file` to enforce the new rule, demonstrating that the case-only disambiguator was load-bearing from its first commit.

**History:** Before `53b8970` the glossary contained no entries for either sense (`git show 53b8970^:spec_topics/glossary.md` lists `binder`, `callable set`, `coercion`, `loom-side name`, `prompt mode`, `Pi tool` / `.loom callable`, `top-level loom` — no `loom` or `Loom` entry). The two glossary entries and the rule that distinguishes them were introduced together by `53b8970`; the prose collision documented above (`influences.md` sentence-initial `Loom`, `grammar.md` `## Loom literal sublanguage`, `runtime-value-model.md` table headers and prose) all pre-dates the rule and was inherited rather than created, so the rule did not introduce the collision so much as fail to resolve it. The current finding records that the chosen resolution (case-only) was structurally insufficient from its first commit and remains so. No later commit has attempted a stronger disambiguator on either sense.

## Solution Space

**Shape:** multiple

### Option A — Rename the language sense to a heading-stable form

**Approach.** Adopt a fixed phrase for the language sense — `the Loom language` when standing alone, `Loom-language <noun>` (e.g. `Loom-language expression`, `Loom-language AST`, `Loom-language value`) when used adjectivally — and reserve bare `loom` / `Loom` for the file-unit sense in every position (including sentence-initial, heading-initial, and table-header). Update the `Loom (language)` glossary entry to record the rename, mark the bare `Loom` spelling as a non-canonical alias retained for one release under the existing alias machinery (GOV-20 / GOV-26 supply the prose-token-alias and dual-anchor patterns), and add a usage note immediately under the `loom (file unit)` entry pinning the rule: "If a heading, table cell, or sentence-initial position would force capitalisation of the file-unit sense, restructure or pluralise (`looms`) to keep the bare word lowercase."

**Spec edits.**

- `docs/spec_topics/glossary.md`: rewrite `Loom (language)` entry; add the usage note under `loom (file unit)`.
- `docs/spec_topics/influences.md`: "Loom borrows…" → "The Loom language borrows…"; `**Original to loom**:` → `**Original to the Loom language**:`; "Loom is much smaller than either parent" → "The Loom language is much smaller than either parent".
- `docs/spec_topics/grammar.md`: `## Loom literal sublanguage` → `## Loom-language literal sublanguage`; "a legal Loom expression" → "a legal Loom-language expression"; "the Loom AST" → "the Loom-language AST"; "Loom distinguishes…" → "The Loom language distinguishes…"; "Statements … are not expressions in Loom" → "Statements … are not expressions in the Loom language".
- `docs/spec_topics/runtime-value-model.md`: table header `| Loom type |` → `| Loom-language type |`; "Loom values are represented…" → "Loom-language values are represented…"; "Loom code observes `Result`…" → "Loom-language code observes `Result`…".
- `docs/spec_topics/expressions.md`, `bindings.md`, `lexical.md`: apply the same rule to each `Loom <verb>` / `Loom <noun>` site (a closed list under one editorial pass).
- Add an alias-retirement table row under GOV-20/GOV-26 machinery recording bare `Loom` as a release-bounded alias for `Loom language`.

**Pros.** Disambiguator is structural, not positional — survives headings, sentence-initial, table cells, and quoted fragments. `grep "Loom"` returns only language-sense hits; `grep "loom"` returns only file-unit-sense hits (after the rename completes). Headings and anchors are stable to citation. Pattern matches the existing two-name resolution used for `slash caller` / `invoke parent`.

**Cons.** Wide-surface edit across ~7 topic files; every existing `Loom <noun>` collocation rewords. The compound `Loom-language` reads slightly heavier than bare `Loom` in body prose.

**Risks.** Mechanical find-and-replace cannot distinguish "Loom" (proper-noun language) from "Loom" (sentence-initial file-unit), so the edit must be human-driven case-by-case; an over-eager pass would rewrite the few legitimate sentence-initial file-unit uses and quietly change the referent.

### Option B — Rename the file-unit sense to a marked form, keep `Loom` bare

**Approach.** Reserve bare `Loom` (capitalised everywhere) for the language; rename the file-unit sense to `loom file` or `.loom unit` in every prose position and retain bare `loom` only inside fixed compounds (`top-level loom`, `loom code`, `loom 1.0`, `pi-loom`, diagnostic codes `loom/...`) where it is already locked by other governance.

**Spec edits.**

- `docs/spec_topics/glossary.md`: rename `loom (file unit)` → `loom file`; update the `Loom (language)` rule to "bare `Loom` is the language; the file unit is `loom file` (or `.loom unit` when the file extension is salient)".
- `docs/spec_topics/overview.md`: `### Scope of a loom file` is already aligned; "Each `.loom` file defines a **loom**" → "Each `.loom` file defines a **loom file**"; "A loom therefore alternates…" → "A loom file therefore alternates…".
- `docs/spec_topics/invocation.md`, `docs/spec_topics/frontmatter.md`, `docs/spec_topics/glossary.md` cross-references to the file-unit sense: rename in place.
- Diagnostic codes (`loom/parse/...`, `loom/load/...`) and the `pi-loom` extension name remain untouched — they are not in the renamed scope.

**Pros.** Bare `Loom` retains its current ergonomic in language-prose-heavy pages (`grammar.md`, `expressions.md`, `runtime-value-model.md`); the renamed form (`loom file`) lands on the conceptually less-frequently-named sense.

**Cons.** "Loom file" reads as a generic compound — a casual reader may not register it as a coined term, weakening the rename's force. The fixed-compound carve-out (`top-level loom`, `loom code`) keeps `loom`-bare in active use, partially reintroducing the original ambiguity in exactly those compounds. Touches a similar number of files to Option A.

**Risks.** "Loom file" collides ambiguously with phrasings like "the `.loom` file" (lowercase backtick form), where authors may not realise they have written the coined term. The carve-out for `loom code` is itself ambiguous (the language? code written in a `.loom` file? the file unit's code body?) and may need a third rename.

### Recommendation

Take Option A. The asymmetry of frequency — language-sense references are concentrated in a few topic pages (`grammar.md`, `expressions.md`, `bindings.md`, `runtime-value-model.md`, `influences.md`), while file-unit-sense references are spread across the whole corpus — means renaming the language sense moves the edit surface to the smaller, more localised set; renaming the file unit (Option B) touches every page that talks about discovery, invocation, frontmatter, slash dispatch, or diagnostics. Option A's compound (`Loom language`, `Loom-language <noun>`) is also structurally immune to position: a heading `## Loom-language literal sublanguage` is unambiguous on first read; `## Loom literal sublanguage` is not.

Edge cases the implementer must watch:

- Sentence-initial file-unit uses ("Loom files are discovered…", "Loom code never sees wire names…") must be restructured or pluralised (`looms`), not capitalised — the usage note pinned under `loom (file unit)` covers this.
- The `pi-loom (extension)` entry is unaffected — it is already a distinct lexical form (hyphenated, lowercase, never confused with either sense).
- The fixed compound `loom code` is a per-author choice between "code written in a loom file" (file-unit) and "code in the Loom language" (language); after Option A lands, prefer "Loom-language code" when the language sense is intended and "code in a loom" / "the loom's body" when the file-unit sense is intended.
- Diagnostic-code namespace `loom/...` is governance-locked and out of scope of the rename.

## Consequence

**Severity:** correctness

Two reasonable implementers — or two reviewers reading the same paragraph — can disagree on whether a heading, a list bullet, or a cross-page citation is binding the file-unit or the language sense; for normative prose (e.g. `runtime-value-model.md`'s "Loom code observes `Result` only…", which constrains language-level semantics) the divergence is observable in what they implement and what they test. The defect is not localised to the glossary: it propagates anywhere either sense is referenced, and it forecloses `grep`-driven audits over the corpus because both spellings collide on the same token under case-insensitive matching and miss each other under case-sensitive matching.
