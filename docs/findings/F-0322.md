## Finding

The committed permitted-code list (`H7a`, checked in alongside the fixture `.loom`) is a manually-maintained, best-effort union of the `loom/...` codes the `H7a`-`Deps` slices *can* emit. Only `golden ⊆ permitted` containment is mechanically gated (by `H7a`'s containment check). By the plan's own statement, the union's completeness — that it captures every code each slice can emit — is *not* mechanically verified, so a stale union can omit a benign code a slice emits.

The manual real-host smoke run scores criterion (e) by asserting the live run's emitted `loom-system-note` codes are a **subset** of that committed permitted list, and `real-host-smoke-gate.md` classifies "an emitted code outside the permitted set" as a **fail** — a confirmed behavioural-divergence finding, hence a merge blocker (it blocks the bump under trigger (1) or the fidelity-axis-touching merge under trigger (2)). Consequently, if a slice begins emitting a new benign code without a corresponding permitted-list update, a legitimate run scores (e) as a divergence and spuriously blocks the merge. Nothing ties the union back to the codes the `H7a`-`Deps` slices can actually emit, and neither `H7a` nor `real-host-smoke-gate.md` states a recovery for the stale-list false positive.

The plan already carries a structurally identical artifact: the `H3a` ambient-primitive scan's timing-primitive list is likewise a manually-maintained best-effort superset (`scan list ⊇ spec ban`) with no mechanical reconciliation, and the plan gives it a release-time reconciliation backstop — `conventions.md` §"Loom 1.0 release-time residue inspection" item 9, which re-reconciles the list against the live `Clock`/`WallClock` ban and records deliberate superset entries. The permitted-code list has no analogous backstop among the inspection's nine items.

## Issue introduction

**Verdict:** present-since-inception (of the committed permitted-code-list mechanism)
**Introducing commits:** `052b019` (2026-06-11) "pi-loom plan: resolve \"Real-host smoke pass criterion (e) names a permitted code set with no committed source\""; `f340c3b` (2026-06-17) "pi-loom plan: resolve \"permitted-code-list cannot-disagree guarantee unenforced\""
**History:** `git log -S` over the cited plan files shows `052b019` introduced the committed permitted-code list as the single reviewed reference criterion (e) scores against; at that introduction the completeness limitation was already present and unbacked (no reconciliation step). `f340c3b` later added the explicit "completeness is not mechanically verified … a stale union can omit a benign code a slice emits" / "best-effort union" wording to `H7a`, making the limitation textual without adding a reconciliation backstop. No commit in the history of `conventions.md`'s residue-inspection section or `real-host-smoke-gate.md`'s criterion (e) ever added the permitted-list analogue of item 9; the gap has therefore existed continuously since the artifact was introduced.

## Solution Space

**Shape:** single

### Recommendation

Add a reconciliation item to `conventions.md` §"Loom 1.0 release-time residue inspection", mirroring the existing item 9 (the `H3a` timing-primitive-list reconciliation), that reconciles the committed permitted-code list against the `loom/...` codes the `H7a`-`Deps` slices can emit. The item's "Clean when" condition: every code an `H7a`-`Deps` slice can emit is present in the permitted list, and any deliberate omission is recorded as dispositioned residue. State the recovery in the same item: append the slice's newly-emitted code to the committed permitted list and re-record the smoke (the falsifiable run record lives in `H6a`'s gate-activation commit message, per `H6a`'s existing evidence convention).

In `real-host-smoke-gate.md` criterion (e), add a cross-reference that an emitted code outside the permitted set traceable to permitted-list staleness is reconciled via this release-time item and recovered by appending the code, distinguishing the stale-list false positive from a genuine behavioural-divergence fail.

The reconciliation belongs to the same release-time review surface and owner that performs the rest of the nine-item inspection (the `H6a` release-gate audit); do not introduce a separate mechanical gate — the plan deliberately routes this class of manually-maintained-superset residue to the release-time inspection rather than to CI. The `H7a`-`Deps` slice set the reconciliation reads is the one `H7a` already lists; do not duplicate or restate that list in `conventions.md`.

## Consequence

**Severity:** advisory

A slice that starts emitting a new benign `loom/...` code without a permitted-list update makes a legitimate real-host smoke run fail criterion (e), spuriously blocking the bump/merge; with no reconciliation step and no stated recovery, the owner running the gate has no guidance to distinguish the stale-list false positive from a genuine behavioural divergence. The defect is bounded and recoverable (extend the list, re-run the smoke), but the plan leaves both the maintenance step and the recovery unstated, while the structurally identical timing-primitive list is reconciled by residue-inspection item 9.
