## Finding

The opening **Persistent diagnostics (default)** paragraph of `diagnostic-shape.md` scopes the persistent `loom-system-note` channel to three namespaces — "All `loom/parse/*`, `loom/load/*`, and `loom/runtime/*` diagnostics are delivered via the channel below" — and then enumerates "five carved-out exceptions" routed through `console.error` from the `session_shutdown` handler. Four of those five (`loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-teardown-step-failed`, `loom/host/session-shutdown-runtime-degraded`) live in `loom/host/*`, a fourth namespace the same paragraph never claimed for the persistent channel. The **Code namespaces** bullets later in the same file confirm this: the `loom/host/*` bullet says these codes are "Emitted via `console.error` since the typical observation site is mid-teardown when the persistent-diagnostic channel may already be invalidated" — i.e. `loom/host/*` is `console.error`-by-default at the *namespace* level, not a per-code exception to a `loom-system-note`-by-default rule that was never written for it.

Calling the four `loom/host/*` codes "exceptions" to a rule whose stated domain excludes their namespace is contradictory on its face, and it sets up a wrong mental model for a future contributor adding a new `loom/host/*` row: the text suggests they would need a fresh "exception" sentence to keep their code off the persistent channel, when in fact the `loom/host/*` bullet already routes the whole namespace through `console.error`. The genuine cross-namespace carve-out is exactly one row — `loom/runtime/reload-teardown-timeout` — a single `loom/runtime/*` code that opts out of its namespace's default delivery sink because it is emitted from the same teardown handler.

The count "five" is also vulnerable for the same structural reason: it conflates per-code carve-outs (one) with whole-namespace routing (four), and will keep drifting upward whenever a `loom/host/*` row is added or removed even though no new exception to the persistent-channel rule has been introduced. Per the `cancelled-by-session-shutdown` `file`/`range` finding immediately below it in the source review, at least one further per-invocation `loom/runtime/*` code on the same teardown path is already under-specified on this surface, which compounds the framing's brittleness.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `5e8b499` (2026-05-06) "pi-loom spec: resolve 'Hot-reload teardown contract for the loom extension instance is unspecified'" — introduced the framing as **"one carved-out exception"** covering `loom/runtime/reload-teardown-timeout` alone. At this point the sentence was internally coherent: a single `loom/runtime/*` code carved out of a `loom/parse/*` ∪ `loom/load/*` ∪ `loom/runtime/*` default.
- `9559e19` (2026-05-07) "pi-loom spec: resolve 'session_shutdown unknown reason: no closed-set validation, no fail-safe'" — first cross-namespace drift. Bumped the count to **two** and added `loom/host/session-shutdown-reason-unknown`, simultaneously adding the `loom/host/*` namespace bullet whose own text already routes the whole namespace through `console.error` ("emitted via `console.error` since the typical observation site is mid-teardown"). The contradiction between the headline ("exception" to a three-namespace rule) and the namespace bullet (whole-namespace default) was born here.
- `6ce416d` (2026-05-07) — bumped to **three** by adding `loom/host/session-shutdown-teardown-step-failed`.
- `75a9626` (2026-05-09) — bumped to **four** by adding `loom/host/session-shutdown-pinned-constant-unreadable`.
- `52b9101` (2026-05-11) — bumped to **five** by adding `loom/host/session-shutdown-runtime-degraded`.

**History:** `git log -G "carved-out exception" -- docs/spec_topics/diagnostics/diagnostic-shape.md` (resolved across the pre-split `spec_topics/diagnostics.md` ancestor via `--follow`) shows the count grew monotonically `1 → 2 → 3 → 4 → 5` across the five commits above; the `f5e89f4` mass-split and `158ed09` pin-clarification commits in between only repointed links and did not change the headline framing. The defect is genuinely a multi-commit interaction: the original sentence (`5e8b499`) was correct in isolation, and the `loom/host/*` namespace bullet introduced with the first cross-namespace addition (`9559e19`) is also correct in isolation; what is wrong is the failure to update the headline's domain claim when a fourth namespace started being enumerated under it. Each subsequent count-bump commit silently reinforced the bad framing rather than introducing it.

## Solution Space

**Shape:** single

### Recommendation

Reshape the **Persistent diagnostics (default)** paragraph to separate two structurally distinct rules:

1. **Persistent-channel default** — keep the existing `loom-system-note` envelope rule but scope it to the three namespaces that actually default to it: `loom/parse/*`, `loom/load/*`, and `loom/runtime/*`. State the genuine per-code exception within that domain: `loom/runtime/reload-teardown-timeout` is routed through `console.error` because it fires from the `session_shutdown` handler after `pi.sendMessage` may already be invalidated. Drop the "five carved-out exceptions" enumeration here.
2. **`loom/host/*` namespace routing** — state, as a separate sentence (or short paragraph) in the same opening block, that all `loom/host/*` codes are emitted via `console.error` from the `session_shutdown` handler, for the same `ExtensionRuntime.invalidate(...)` reason. Cross-link to the `loom/host/*` bullet later in **Code namespaces** as the canonical home of this rule. This makes the four `loom/host/session-shutdown-*` codes consequences of namespace routing, not per-row exceptions.

Keep the existing `console.error`-throw swallow obligation (the "Each of these … is itself wrapped per [Pi Integration Contract — Diagnostic-emission isolation] …" sentence) but reframe it to apply to "each `console.error` emission from the `session_shutdown` handler" — i.e. to the union of (1) the one carved-out `loom/runtime/*` code and (2) all `loom/host/*` codes — rather than to a fixed count of five carve-outs. This preserves the obligation's coverage even as `loom/host/*` rows are added or removed within the diagnostic-registry carve-out (`GOV-15`) without forcing an edit to the headline sentence.

The **Location-less diagnostics** paragraph already enumerates "the four `loom/host/session-shutdown-*` teardown codes" as a group; that enumeration can keep its current form (it is sized by the location-less property, not by the `console.error`-routing property) or be re-expressed as "every `loom/host/*` code" for symmetry — the latter is preferable because it removes another count that drifts whenever the namespace gains or loses a row.

Edge cases the implementer must watch:
- The `loom/runtime/cancelled-by-session-shutdown` row (see the immediately-following finding in the source review) is also emitted from the `session_shutdown` teardown but is a structured `loom-system-note`, not a `console.error` carve-out; the rewrite must not accidentally sweep it into the `console.error` group.
- The `details.event.*` *Construction-site wrap* sub-clauses currently lean on the "five carve-outs" framing when describing fallback wire forms; check that their `loom/host/session-shutdown-runtime-degraded` and `loom/runtime/cancelled-by-session-shutdown` references still resolve cleanly after the headline is reshaped.
- `code-registry-host.md`'s table rows and `V9g`'s `Adds` paragraph cite the per-namespace `console.error` rule; the rewrite should leave those cites pointing at the new namespace-routing sentence (or its anchor) rather than at the obsolete "five carve-outs" enumeration.

## Consequence

**Severity:** advisory

A contributor reading the persistent-channel paragraph in isolation gets a self-contradictory rule (three-namespace domain plus four out-of-domain "exceptions") and a wrong mental model for extending the `loom/host/*` family — they would expect to add a per-row exception sentence rather than relying on the `loom/host/*` namespace bullet's standing `console.error` routing. The runtime contract is not actually at risk because the per-row Trigger / routing text on each `loom/host/*` registry row pins delivery directly, but the headline framing will keep generating editorial churn (the carve-out count has already drifted `1 → 2 → 3 → 4 → 5` over six commits) and the next reviewer pass will keep finding it.
