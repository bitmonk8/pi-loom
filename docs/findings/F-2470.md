## Problem

A typed query's two-phase loop transitions from its free phase to the forced respond turn when the model emits a plain-text (non-tool-use) final turn that normalises to provider stop reason `end_turn` / `stop` — a behavioural property of `@earendil-works/pi-ai`'s per-provider adapters across the four loom 1.0-supported providers, not part of pi-ai's typed surface. Every sibling host-behavioural presupposition on `conversation-drive.md` carries an explicit presupposition block plus a routed item in `version-bump-step2.md`'s *Editorial-review checklist for unpinned host presuppositions* (items (a)–(as)); the free-phase `end_turn` / `stop` trigger normalisation has neither. A provider or pi-ai change that surfaced a plain-text final turn under another stop reason would silently misfire the transition — the typed query would loop to `max_rounds` or misclassify as a `TransportError` — with no SDK surface-inventory signal or editorial-review step scheduled to catch the regression.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** f5e89f4 — docs: enforce 100KB hard / 30KB recommended size cap on spec set (2026-06-04, Thomas Andersen); e63ba6c — pi-loom spec: resolve "Per-provider complete() forced-tool re-validation gate" (2026-06-05, Thomas Andersen); b119f5b — pi-loom spec-plan: fix F-1147 — enumerate both forced-respond entry points in phase-2 trigger (2026-06-25, Thomas Andersen)
**History:** The free-phase plain-text-turn trigger has named provider stop reasons `end_turn` / `stop` since `query-tool-loop.md` and `conversation-drive.md` were carved out of the monolithic spec (f5e89f4, 2026-06-04), and in the predecessor monolith before that; no presupposition or editorial-review hook was ever attached to that normalisation. The asymmetry the finding flags deepened as sibling host-behavioural presuppositions were retrofitted around the same surface — the `complete()` forced-tool hook (e63ba6c, 2026-06-05) and, through 2026-06-25, the second forced-respond entry point (b119f5b) alongside the `stopReason: "error"` / commit-ordering / streaming / no-rejection postures — each gaining its own editorial-review item while the free-phase trigger normalisation was left uncovered.

## Solution approach

Add a presupposition block to `conversation-drive.md` adjacent to the `#complete-forced-tool-presupposition` block, stating that the free-phase→forced-respond transition presupposes a plain-text final turn normalises to `AssistantMessage.stopReason` `end_turn` or `stop` across the four loom 1.0-supported providers. Add a routed editorial-review item to `version-bump-step2.md`'s items-(a)–(as) checklist parallel to item (m), and cross-reference it from the new block.

## Solution constraints

- Out of scope: re-pinning the `AssistantMessage.stopReason` field, already owned by `version-bump-step2.md` item (m); the new block adds only the `end_turn` / `stop` value-space presupposition.
