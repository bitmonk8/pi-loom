## Finding

QRY-20 (*Observability of discarded results*, `query/query-escapes-stringification.md`) guarantees that the operator-facing runtime event for a discarded `Err` "fires exactly once per discarded `Err`, regardless of how many tool-call rounds or respond-repair follow-ups the underlying query consumed." The multi-attempt case — a discarded typed query (`let _ = @<T>…`) that runs several respond-repair follow-ups before exhausting its budget — is the exact scenario the "regardless of how many … respond-repair follow-ups" clause exists to constrain.

That scenario has no normative test vector anywhere in the query topic. The neighbouring `query-forms.md` and the diagnostics pages carry explicit `*Test vectors (normative).*` blocks, and the discard-observability behaviour is leafed for tests (`V13g-T`), but neither QRY-20 nor the test leaf pins a multi-attempt fixture. An implementation that fires the event once per failed follow-up attempt (rather than once for the whole discarded query) violates QRY-20, yet passes every fixture the spec currently states — the single-attempt discard cases do not distinguish the two implementations. The one untested edge is precisely the edge the rule was written to fix.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `b3bc4ce` (2026-05-04, "spec: pin failure-observability surface (Cluster A, Option C)")
**History:** The corpus is git-tracked. `git log -S 'exactly once per discarded'` traces the *Observability of discarded results* rule to `b3bc4ce`, which introduced the discard-observability contract with the multi-attempt clause already present ("fires exactly once per discarded `Err`, regardless of how many tool-call iterations or coercion follow-ups the underlying query consumed"). No test vector accompanied the rule at inception. Subsequent restructuring — `f5e89f4` (2026-06-04, split of `query.md` into the `query/` subtree) and `716a13b` (2026-06-25, coined the `QRY-1..21` identifiers) — carried the rule (and the "coercion follow-ups" → "respond-repair follow-ups" wording) forward verbatim without adding a vector. The omission has existed continuously since the surface was first pinned.

## Solution Space

**Shape:** single

### Recommendation

Add a normative test vector to the *Observability of discarded results* section of `query-escapes-stringification.md`: a discarded typed query (`let _ = @<T>…`) with `respond_repair.attempts: 3` whose final response fails AJV validation on all three follow-ups MUST produce exactly **one** `loom-system-note` runtime event with `display: false`, regardless of the attempt count, and that single event's `attempts` field MUST be `3`.

Edge cases the vector should make explicit so it actually discriminates the wrong implementation:
- The assertion is on event *count* across the whole discarded query (exactly one), not per follow-up turn.
- The `attempts` field reflects the terminal-exhaustion count (3), consistent with QRY-11's slot accounting — non-validation failures do not consume a slot, so a clean three-validation-failure run is the right fixture.
- Pair it with the existing single-attempt discard vector so the suite covers both the zero-follow-up and the multi-follow-up paths.

The paired test leaf `V13g-T` should gain this fixture in its **Tests** acceptance criteria.

## Consequence

**Severity:** advisory

An implementer who reads QRY-20 literally produces correct behaviour; the gap is a missing conformance vector, not an ambiguity in the rule. But without the multi-attempt fixture, a wrong implementation (one event per failed follow-up) ships green, and the divergence surfaces only as duplicated operator-facing discard notes under respond-repair retry.
