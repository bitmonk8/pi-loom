## Finding

The `loom/load/package-read-timeout` row in [`code-registry-load.md`](../../../docs/spec_topics/diagnostics/code-registry-load.md) pins the Message template `package '<name>' package.json read exceeded <deadline>ms during package discovery`. The `<name>` token is governed by §5 of [`placeholder-rendering-b.md`](../../../docs/spec_topics/diagnostics/placeholder-rendering-b.md), whose **Rule** is "render verbatim from the source as the offending text would appear there" and whose `<name>` sub-bullet treats it as identifier-shaped, drawn from the package's own `name` field.

The trigger for this code, however, is exactly the case in which the `package.json` read **did not complete**: the per-read deadline derived from `looms.scanPackagesTimeoutMs` expired before the file's bytes were available, so the `name` field was never observed. The §5 rule has no fallback for an unobservable source value (unlike §6's `<no message>` rule for empty error strings, or §8's `<unreadable>` / `<unresolvable>` sentinels for `host-incompatible`), so two conformant implementers face an undocumented choice — render the empty string, render `<unreadable>`, render the candidate directory path, throw at the diagnostic site — and the row's normative Message contract is unsatisfiable as written. The same defect contaminates DIAG-4 (the Message column is normative): no asserting test can pin a byte-identical expected string for this code.

The fix needs to land in two places: the placeholder rendering page must admit a fallback for `<name>` on this row (`<path>`-shaped per §5, sourced from the walk's candidate directory), and the registry row must declare which `details` field carries that fallback value so the renderer's substitution input is unambiguous.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `61ea1a5` (2026-05-06) — *"Diagnostic message placeholders have no defined rendering"* — introduced the placeholder rendering categories that classify `<name>` as source-derived identifier-shaped with no fallback for an unobservable source value.
- `25e8a60` (2026-05-07) — *"T08 — Per-package.json read timeout: overrun is unbounded"* — first attached the message template `package '<name>' package.json read exceeded <deadline>ms during package discovery` to the read-timeout trigger (then folded into `loom/load/unreadable-source` via `details.kind = "package-read-timeout"`).
- `65989e2` (2026-06-01) — *"unreadable-source row carries two message formats"* — split the read-timeout sub-trigger into its own `loom/load/package-read-timeout` code, carrying the same `<name>`-using template over without addressing the rendering gap.

**History:** the `<name>` rendering rule and the read-timeout Message template were authored by independent commits a day apart and never reconciled; the 2026-06-01 split into a dedicated code preserved the template byte-for-byte, so the latent contradiction surfaced as a discrete row rather than being inherited from a multi-trigger parent. The defect lives in the interaction between the two pages — neither commit, read in isolation, looks wrong — and the durable fix has to touch both surfaces.

## Solution Space

**Shape:** single

### Recommendation

Add a fallback rule on §5's `<name>` sub-bullet keyed to the specific row, and pin the `details` source on the registry row so the renderer's input is contractual:

1. In [`placeholder-rendering-b.md`](../../../docs/spec_topics/diagnostics/placeholder-rendering-b.md) §5, extend the `<name>` sub-bullet with a row-specific carve-out: on `loom/load/package-read-timeout`, `<name>` falls through to category 5's `<path>` rule and renders the **candidate package directory's absolute path** (verbatim, no realpath normalisation, no symlink resolution — matching the `<path>` rule the row would have used had the file been opened and discovered to be unreadable rather than slow). Add a test vector: a timeout on a candidate at `/home/op/proj/node_modules/@scope/pkg` renders `package '/home/op/proj/node_modules/@scope/pkg' package.json read exceeded 200ms during package discovery`.
2. In the [`loom/load/package-read-timeout` row](../../../docs/spec_topics/diagnostics/code-registry-load.md), extend the existing `details.kind = "package-read-timeout"` sentence to also pin `details.path` (the candidate directory's absolute path) as the source of `<name>`'s rendered value, mirroring the `details.kind`/`details.member` discipline used by `loom/load/host-incompatible`.

Edge cases the implementer must watch:
- The candidate enumeration in [DISC-6](../../../docs/spec_topics/discovery/package-and-settings.md) yields directory paths *before* any `package.json` is read; `details.path` is therefore always available at the emit site, including the `@scope/<pkg>` unwrapping case.
- The `name` field is also unobservable on a slow read that *eventually* completes after the deadline; the diagnostic must still fire (and use the directory-path fallback) — the race is decided by the deadline, not by whether bytes ever arrive.
- The fallback applies only to this row. Other `<name>`-bearing rows (e.g. `loom/load/manifest-invalid`, `loom/load/manifest-escapes-package`) keep the §5 identifier-shaped rendering because their triggers fire *after* a successful `package.json` parse, so `name` is observable.

## Consequence

**Severity:** correctness

The Message column is normative under DIAG-4 and reachable by the V7c byte-identical-rendering tests; with no rule for `<name>` on the timeout case, two reasonable implementers diverge on the rendered string for the only trigger this code admits, and the V7b registry-closing gate has no expected string to check against. The diagnostic is non-fatal and only emitted on a degenerate-walk path, but the spec-level contract collapses for that path.
