## Problem

The `loom/load/package-read-timeout` row in `code-registry-load.md` pins the Message template `package '<name>' package.json read exceeded <deadline>ms during package discovery`, but §5 of `placeholder-rendering-b.md` governs `<name>` as identifier-shaped, sourced from the package's own `name` field. This code's only trigger is the case where the `package.json` read did not complete before its per-read deadline expired, so the `name` field is never observed, and §5 provides no fallback for an unobservable source value (unlike §6's `<no message>` rule or §8's `<unreadable>` / `<unresolvable>` sentinels). The row's normative Message contract is therefore unsatisfiable, and because the Message column is normative under DIAG-4, two conformant implementers diverge on the rendered string and no asserting test can pin a byte-identical expected value.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `61ea1a5` (2026-05-06) — *"Diagnostic message placeholders have no defined rendering"* — introduced the placeholder rendering categories that classify `<name>` as source-derived identifier-shaped with no fallback for an unobservable source value.
- `25e8a60` (2026-05-07) — *"T08 — Per-package.json read timeout: overrun is unbounded"* — first attached the message template `package '<name>' package.json read exceeded <deadline>ms during package discovery` to the read-timeout trigger (then folded into `loom/load/unreadable-source` via `details.kind = "package-read-timeout"`).
- `65989e2` (2026-06-01) — *"unreadable-source row carries two message formats"* — split the read-timeout sub-trigger into its own `loom/load/package-read-timeout` code, carrying the same `<name>`-using template over without addressing the rendering gap.

**History:** the `<name>` rendering rule and the read-timeout Message template were authored by independent commits a day apart and never reconciled; the 2026-06-01 split into a dedicated code preserved the template byte-for-byte, so the latent contradiction surfaced as a discrete row rather than being inherited from a multi-trigger parent. The defect lives in the interaction between the two pages — neither commit, read in isolation, looks wrong — and the durable fix has to touch both surfaces.

## Solution approach

Extend §5's `<name>` sub-bullet in `placeholder-rendering-b.md` with a fallback for `loom/load/package-read-timeout` that renders the candidate package directory path via §5's `<path>` rule. In the `loom/load/package-read-timeout` row of `code-registry-load.md`, pin which `details` field carries that directory path as the renderer's substitution input, mirroring the `details.kind` / `details.member` discipline `loom/load/host-incompatible` already uses.

## Solution constraints

- The `<name>` fallback is scoped to `loom/load/package-read-timeout` only; other `<name>`-bearing rows (e.g. `loom/load/manifest-invalid`, `loom/load/manifest-escapes-package`) retain §5's identifier-shaped rendering, since their triggers fire after a successful `package.json` parse.
