## Problem

DISC-6 bounds the package-discovery walk on a file-count cap and pins the cap-check site (*before each new candidate-package read attempt*), but never states when the file-count counter increments relative to a candidate read's lifecycle. The verb "inspected" admits three divergent readings — increment at read initiation, at read completion, or only on a successful open — which genuinely diverge once the per-read deadline path is in play, since a timed-out or errored candidate is a distinct outcome class. The result is an observable behavioural fork: two conforming implementations scan a different number of packages and trip `loom/load/discovery-slow` at a different point on any install where package reads time out.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 475155c — pi-loom plan: resolve "V14m discovery walk omits scoped packages and has no upper bound" (2026-05-05, Thomas Andersen); 25e8a60 — pi-loom spec: resolve "T08 — Per-package.json read timeout: overrun is unbounded" (2026-05-07, Thomas Andersen)
**History:** 475155c first wrote the bounded-walk rule with the verb "inspected" as the counter for `looms.scanPackagesMaxFiles`, at a point in the corpus when every candidate either parsed (counted) or hit a synchronous filesystem failure (not counted). 25e8a60 then added the per-read deadline and the abandonment / `package-read-timeout` path, introducing a third candidate outcome that "inspected" had no rule for. No subsequent revision (through the current head at `443ee4e`) has reconciled the counter against that third outcome, so the silence is the residue of the two changes layering.

## Solution approach

Clarify DISC-6 (anchor `id="disc-6"`) to pin the file-count counter's increment point at candidate-read initiation: a candidate is charged toward `looms.scanPackagesMaxFiles` when its read is attempted, regardless of whether the read later succeeds, errors, or times out. State this adjacent to the existing cap-check-site clause.
