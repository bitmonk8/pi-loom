## Problem

SLSH-5 ("Chain attribution", `slash-invocation.md#slsh-5`) is a normative MUST: for an `invoke_callee` cascade, renderers MUST append per hop the byte-exact suffix ` from <callee_path> invoked at <parent_path>:<line>` (byte-exact under SLSH-4). Of the three interpolated values, only `<callee_path>` has a carrier — `InvokeCalleeError` (`errors-and-results/queryerror-variants.md`) declares `callee_path: string`. `<parent_path>` and `<line>` are carried nowhere: the `InvokeCalleeError` schema is `{ kind, message, callee_path, inner }`, and the variant Notes state each variant "carries only its meaningful fields". SLSH-5 sources the two missing values from "the immediate parent's invocation record" / values "recorded at the invocation site (per Invocation, Resolution)", but no such per-hop record is defined anywhere in the spec — the phrase appears only inside SLSH-5, referring forward to a structure never established.

## Issue introduction

**Verdict:** introduced by a single later commit
**Introducing commit:** `2d217b1` (2026-05-06) — "pi-loom spec: resolve \"Call chain suffix is illustrative, not normative\""
**History:** The chain suffix originated as illustrative prose — an ellipsis-prefixed, parenthesised example (`"... from child.loom invoked at parent.loom:42"`) carrying no normative obligation, so no data-model carrier was implied. `2d217b1` promoted it to the normative SLSH-5 MUST (` from <callee_path> invoked at <parent_path>:<line>`), introducing the per-hop `<parent_path>`/`<line>` rendering obligation while attributing the values only to paths "recorded at the invocation site (per Invocation, Resolution)" — a section that defines callee-path `realpath` normalisation but no per-hop record. The `InvokeCalleeError` schema has carried `{ kind, message, callee_path, inner }` and no source-position fields since inception (`git log -S "InvokeCalleeError"` shows only the later `8455d5e` wire-kind rename, which left the field set unchanged). A subsequent commit `1ccbb75` (2026-06-07) refined the `<line>` definition (call-site token vs receiving binding, `.loom`-callable bare-identifier form) but still sourced it from "the immediate parent's invocation record" without ever defining that record. The gap is therefore the normalisation step `2d217b1` performed on the renderer side without a matching extension of the data model.

## Solution approach

Establish a per-`invoke`-hop carrier for `<parent_path>` and `<line>` in `invocation.md` — the natural owner is alongside the existing "Resolution" / "Failures" sections that already own `invoke` path semantics — and add an anchor so it can be cited. Repoint SLSH-5's "recorded at the invocation site" / "the immediate parent's invocation record" references to that anchor, leaving `<callee_path>` sourced from the `InvokeCalleeError` wrapper.

## Solution constraints

- Out of scope: do not add `parent_path` or `line` fields to the `InvokeCalleeError` wire schema in `queryerror-variants.md` — the record is a runtime-side seam distinct from the wire-contract error type, honouring the "each variant carries only its meaningful fields" design.
