## Finding

Step 0 (c) of the capability probe describes `AgentSession.prototype.abort` as probed
"against the imported `AgentSession` class on the `@earendil-works/pi-coding-agent`
namespace" and then justifies the import-path-load-bearingness with: "`AgentSession`
is re-exported by `dist/core/index.d.ts` of `@earendil-works/pi-coding-agent` Б─╕ from
`./agent-session.js`, and the loom runtime depends on the re-export specifically, not
on direct consumption from `@earendil-works/pi-agent-core` (where the underlying class
lives), so a Pi minor that drops the re-export surfaces at this probe as
`loom/load/host-incompatible` rather than as a silent fallback to a different import
path."

At the loom 1.0 Pi-SDK pin (`@earendil-works/pi-coding-agent ~0.75.5`), the class is
declared **directly** in `@earendil-works/pi-coding-agent`, at
`dist/core/agent-session.d.ts` (`export declare class AgentSession`). The token
`AgentSession` does not appear anywhere in `@earendil-works/pi-agent-core` at all Б─■
that package exports `Agent`, `AgentLoop`, `AgentHarness`, `AgentMessage`, `AgentEvent`,
etc., but no `AgentSession` class and no re-export of one. The two re-exports that do
exist are both intra-`pi-coding-agent`: `dist/core/index.d.ts` re-exports
`AgentSession` from `./agent-session.ts`, and the package root `dist/index.d.ts`
re-exports it from `./core/agent-session.ts`. The cross-package re-export the spec
claims is load-bearing simply does not exist, and the "Pi minor that drops the
re-export" fragility scenario therefore points at a non-existent dependency.

The downstream behavioural claim Б─■ that a missing `AgentSession` or a missing `abort`
member is caught by the probe and surfaces as `loom/load/host-incompatible` rather
than as a runtime `TypeError` Б─■ is still correct and still load-bearing; only the
provenance sentence and the named-package fragility rationale are wrong.

**History:** Before `68ff2d1`, the `AgentSession.prototype.abort` bullet named only the probe expression and the namespace ("imported `AgentSession` class on the `@mariozechner/pi-coding-agent` namespace") and made no claim about where the class is declared. `68ff2d1` rewrote the bullet to add the "import path is load-bearing" rationale, introducing both the `dist/core/index.d.ts` re-export citation and the parenthetical "(where the underlying class lives)" pointing at `@mariozechner/pi-agent-core` Б─■ the defect this finding reports. The later scope rename to `@earendil-works/*` (`5e73072`) and the SDK-pin bumps (`f985a17`, etc.) carried the wrong attribution through verbatim without re-checking it; none of them introduced the defect, they only preserved it. `git log -G "underlying class lives"` and `git log -S "where the underlying class lives"` both isolate the introducing commit to `68ff2d1`.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the parenthetical and the fragility clause inside the `AgentSession.prototype.abort` bullet of Step 0 (c) to reflect actual provenance, without disturbing the surrounding sentence structure or the probed expression. Specifically:

- Replace "`AgentSession` is re-exported by `dist/core/index.d.ts` of `@earendil-works/pi-coding-agent` (at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin)) from `./agent-session.js`, and the loom runtime depends on the re-export specifically, not on direct consumption from `@earendil-works/pi-agent-core` (where the underlying class lives)" with text that records that the class is declared in `@earendil-works/pi-coding-agent` at `dist/core/agent-session.d.ts` (citing the same file `host-interfaces-core.md` б╖`AgentSession` (member surface loom touches) already pins), and that the import path the loom runtime uses is the package root specifier `@earendil-works/pi-coding-agent`, which resolves the class through the package's own `dist/index.d.ts` re-export of `./core/agent-session` rather than through a deep import of `./dist/core/agent-session`.
- Reframe the fragility clause as: a Pi minor that drops the package-root re-export (forcing consumers to deep-import) Б─■ or relocates the class into a different `@earendil-works/*` package Б─■ surfaces at this probe as `loom/load/host-incompatible` rather than as a silent import resolution to a different specifier the runtime does not consume. Keep the "missing or renamed `abort` member" sentence verbatim; it is independent of provenance.

Cross-check that the corrected text is consistent with the `host-interfaces-core.md` б╖`AgentSession (member surface loom touches)` block (which already attributes the declaration to `dist/core/agent-session.d.ts in @earendil-works/pi-coding-agent`) and with the editorial-review checklist items in `version-bump-step2.md` that re-audit `AgentSession`-related declarations on each Pi minor bump (notably item (n) under host-interfaces-core's surface block). No probe expression, no `details.kind` value, no leaf acceptance criterion changes.

Edge cases: the same paragraph carries the load-bearing pin against `@earendil-works/pi-coding-agent` for the `typeof` target and the load-bearing pin against `@earendil-works/pi-agent-core` for `AgentMessage` (used elsewhere in the corpus). The fix must not delete the `@earendil-works/pi-agent-core` token from the corpus or weaken the loom 1.0 lock-step claim across the four `@earendil-works/*` packages; it only corrects the per-symbol provenance for `AgentSession`.

## Consequence

**Severity:** correctness

A reader auditing the import path (a likely activity for the next Pi minor bump, since the bump procedure re-validates pinned Pi-side declarations) follows the citation to `@earendil-works/pi-agent-core` looking for the underlying class, finds nothing, and either re-derives the provenance themselves or treats the spec as drifted. The fragility scenario the paragraph names ("Pi drops the cross-package re-export") cannot occur, so the rationale for pinning the import path against it does not stand up. A correct rationale (same-package re-export at the package root vs. deep import from `./dist/core/agent-session`, or class relocation across packages in a future minor) is available; the current text just does not express it.
