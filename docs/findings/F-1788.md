## Problem

The *Source-file encoding* paragraph on `audit-resolution.md` makes a strict UTF-8 decoder's reporting of the zero-based first-invalid-byte offset a normative MUST, but a decode failure is routed through *Infrastructure-failure handling* (i) onto the four-field stdout record (`path`, `line`, `symbol`, `proposed-resolution`) pinned by *Wire serialisation* on `audit-wire-and-canary.md`. None of those four fields is allocated to carry the decimal byte offset: `line` and `symbol` are both `<n/a>` for a pre-decode failure, and `proposed-resolution` is constrained to a printable-ASCII resolution-arm reference. Stderr is permitted only for non-machine-parseable text, so it is not a surface a CI parser may consume. The offset therefore has no wire slot a fixture can read, conformant implementations may legitimately drop or misplace it, and the MUST cannot be falsified end-to-end.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e5cd514` (2026-05-09) for the four-field wire format on the audit; `6829a79` (2026-06-06) for the strict-UTF-8 decoder MUST with the first-invalid-byte offset obligation.
**History:** `git log -S 'tab-delimited record segments' -- docs/` lands on `e5cd514` ("T35 SDK capability inventory closed-set negative-direction audit") as the commit that introduced the four-field `path` / `line` / `symbol` / `proposed-resolution` wire shape on what became `audit-wire-and-canary.md`. `git log -S 'strict UTF-8 decoder' -- docs/` and `git log -S 'first invalid byte sequence' -- docs/` both land on `6829a79` ("resolve 'Pi-SDK hidden-assumption cluster (strictCapable absence, decode steps, pi.on origin)'") as the commit that added the *Source-file encoding* decoder MUST to `audit-resolution.md`. Diffing `6829a79~1` against `6829a79` on `audit-resolution.md` confirms the MUST is net-new there; the wire-shape contract was already in place from a month earlier and was not revisited in `6829a79` to allocate a slot for the new offset obligation. Neither commit alone produces the defect — `e5cd514` predates the offset MUST, and `6829a79` did not touch the wire format — so the two changes interact to create a normative obligation whose surfacing surface has no allocated slot.

## Solution approach

Rewrite the *Source-file encoding* offset obligation on `audit-resolution.md` to pin where a decode failure surfaces the offset on the four-field stdout record — naming which field carries the decimal byte offset, which carry `<n/a>`, and which names the re-save-as-UTF-8 resolution arm. Clarify the per-field `<n/a>` rule under `#audit-infrastructure-failure-handling` on `audit-wire-and-canary.md` so the decode case's designated field carries the offset rather than `<n/a>`.

## Solution constraints

- Out of scope: the `symbol = <n/a>` rule for every other pre-decode-resolution infra failure — any carve-out is scoped to the decode case only.
- Preserve the non-UTF-8-BOM `offset = 0` semantics mirroring the runtime `loom/load/invalid-encoding` diagnostic.
