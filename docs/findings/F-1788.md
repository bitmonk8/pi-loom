## Finding

The *Source-file encoding* paragraph of `audit-resolution.md` imposes a hard reporting obligation: when the strict UTF-8 decoder raises on an in-scope file, the implementation "MUST report the zero-based byte offset of the first invalid byte sequence in the original (pre-normalisation) file content (for a non-UTF-8 BOM the reported offset is `0`)," explicitly "mirroring the offset semantics pinned for the runtime `loom/load/invalid-encoding` diagnostic." That decode error is the routed mechanism that produces a parse-failure outcome under *Infrastructure-failure handling* (i) on `audit-wire-and-canary.md`.

But the wire contract those failures must use has no field for the offset. The *Wire serialisation* clause pins every infrastructure-failure record to exactly four tab-delimited fields — the `audit/<class>/<family>/<symptom>` discriminator, `path:line`, `symbol`, and `proposed-resolution` — and *Infrastructure-failure handling* reuses that shape unchanged. For an encoding parse failure on the file header, `line` and `symbol` are each the `<n/a>` sentinel (the failure precedes line-level resolution), and `proposed-resolution` carries the contributor's resolution arm. There is no integer-offset slot anywhere in the record. `line` is a line number, not a byte offset, and is `<n/a>` on this path in any case.

The result is a self-contradiction: one section pins a MUST-report obligation for a value the other section's fixed record shape cannot carry. The two surfaces never reconcile, so the byte offset's destination is unspecified — and the only machine-parseable surface a CI parser may consume (per the *Wire serialisation* sole-surface rule) has nowhere to put it.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e5cd514` (2026-05-09) for the four-field wire format on the audit; `6829a79` (2026-06-06) for the strict-UTF-8 decoder MUST with the first-invalid-byte offset obligation.
**History:** `git log -S 'tab-delimited record segments' -- docs/` lands on `e5cd514` ("T35 SDK capability inventory closed-set negative-direction audit") as the commit that introduced the four-field `path` / `line` / `symbol` / `proposed-resolution` wire shape on what became `audit-wire-and-canary.md`. `git log -S 'strict UTF-8 decoder' -- docs/` and `git log -S 'first invalid byte sequence' -- docs/` both land on `6829a79` ("resolve 'Pi-SDK hidden-assumption cluster (strictCapable absence, decode steps, pi.on origin)'") as the commit that added the *Source-file encoding* decoder MUST to `audit-resolution.md`. Diffing `6829a79~1` against `6829a79` on `audit-resolution.md` confirms the MUST is net-new there; the wire-shape contract was already in place from a month earlier and was not revisited in `6829a79` to allocate a slot for the new offset obligation. Neither commit alone produces the defect — `e5cd514` predates the offset MUST, and `6829a79` did not touch the wire format — so the two changes interact to create a normative obligation whose surfacing surface has no allocated slot.

## Solution Space

**Shape:** multiple

### Option A — Surface the offset in `proposed-resolution` as a plain-ASCII key=value

**Approach:** Amend the *Infrastructure-failure handling* sub-paragraph (and/or the *Source-file encoding* paragraph) to pin that, for the invalid-encoding infrastructure `<symptom>`, the `proposed-resolution` field carries the offset as a plain-ASCII `byte-offset=<N>` key=value token, directly parallel to the *Non-empty-scan canary*'s established `walked=<N> recognised=<M>` `proposed-resolution` convention. `path` carries the failing file; `line` and `symbol` remain `<n/a>` per the existing per-field rules.

**Spec edits:** One sentence in `audit-wire-and-canary.md` pinning the `byte-offset=<N>` encoding for the invalid-encoding infra `<symptom>`; a back-reference from `audit-resolution.md`'s *Source-file encoding* clause so "MUST report" names the wire slot it lands in.

**Pros:** Honours the existing MUST-report obligation and its stated parity with the runtime `loom/load/invalid-encoding` diagnostic; reuses an established `proposed-resolution` key=value convention, so no new field and no break to the four-field tab shape; gives a test leaf a stable, machine-parseable assertion target.

**Cons:** Slightly overloads `proposed-resolution` (a contributor-resolution field) with a diagnostic datum; another consumer of that field's encoding to keep coherent.

**Risks:** The `byte-offset=` token must be plain printable ASCII and tab/newline-free (it is, being an integer); the non-UTF-8-BOM case must still emit `byte-offset=0`, not `<n/a>`, so the offset-is-zero case is not silently elided.

### Option B — Demote the MUST-report obligation to a non-normative note

**Approach:** Relax the *Source-file encoding* clause so the byte offset is a human-facing diagnostic aid (permitted on stderr per the *Wire serialisation* stderr-MAY allowance) rather than a MUST-report value, removing the obligation the four-field record cannot satisfy.

**Spec edits:** Change "MUST report the zero-based byte offset…" to a non-normative note (e.g. "the implementation MAY surface the byte offset on stderr for diagnosis"); drop the "mirroring the offset semantics pinned for the runtime `loom/load/invalid-encoding` diagnostic" parity claim, or restate it as informative.

**Pros:** Removes the contradiction with the smallest wire-format footprint; keeps `proposed-resolution` single-purpose.

**Cons:** Sheds the offset's machine-readability and breaks the deliberate parity with the runtime decode diagnostic that the clause currently asserts; loses a concrete debugging affordance for the most opaque failure class (a binary file that won't decode).

**Risks:** If any downstream consumer or test was written expecting the offset, demotion silently invalidates it.

### Recommendation

Adopt **Option A**. The *Source-file encoding* clause explicitly ties the audit's offset semantics to the runtime `loom/load/invalid-encoding` diagnostic, signalling the offset is meant to be a first-class reported value, not incidental stderr text; the canary's `walked=<N> recognised=<M>` precedent shows `proposed-resolution` is already the spec's home for plain-ASCII key=value diagnostics on non-violation records. Pin `byte-offset=<N>` for the invalid-encoding infra `<symptom>` and ensure the non-UTF-8-BOM case emits `byte-offset=0` rather than the `<n/a>` sentinel.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on where the mandated offset surfaces: one packs `byte-offset=<N>` into `proposed-resolution`, another emits it only on human-readable stderr, a third drops it entirely — each defensibly "conformant" against one section and in breach of the other. The audit still fails closed (non-zero `npm test` exit), so the gate's safety property holds; the divergence is confined to the diagnostic content of the emitted record. A test leaf asserting the offset is present in a stable location cannot be written against the spec as it stands.
