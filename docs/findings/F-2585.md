## Problem

The automated build-time inventory-closure audit is deferred to post-loom-1.0, but the deferral is stated only at the audit's own normative sites (`inventory-audit-intro.md`, where it is a `MAY`) and in one orientation parenthetical (`overview-and-orientation.md`). It has no entry in Future Considerations, the spec's self-described canonical inventory of features deliberately deferred from loom 1.0. The audit is a post-1.0 build-time tooling mechanization that fits the **Tooling deferrals (no loom 1.0 impact)** category exactly — alongside the existing LSP / `loom test` / telemetry / patch-cadence entries — yet it appears nowhere in Future Considerations, so a reader enumerating loom 1.0's deferred work from the canonical inventory cannot discover it. It is not a loom 1.0 non-goal: it has a defined forward path, not a no-seam disposition requiring migration.

## Solution approach

Add a bullet for the automated build-time inventory-closure audit to the **Tooling deferrals (no loom 1.0 impact)** list (anchor `tooling-deferrals-no-loom-1-0-impact`) in `future-considerations/surface-extensions.md`, describing it as a post-1.0 build-time tooling mechanization of the surface-set-closure obligation that loom 1.0 enforces by other means. Forward-link the normative owner at `../pi-integration-contract/inventory-audit-intro.md#sdk-cap-inventory-closure-audit`. Optionally retarget the inline parenthetical in `overview-and-orientation.md` to forward-link the new entry, keeping orientation consistent with the inventory.
