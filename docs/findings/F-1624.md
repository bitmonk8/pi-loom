## Problem

The **`invoke`-parent observation rule** in `session-shutdown-semantics.md` (under the `id="session-shutdown-semantics"` anchor) is normative prose with no accompanying test obligation. It asserts two things about a parent invocation whose child is concurrently in the `ActiveInvocationRegistry` at `session_shutdown`: that the parent unwinds without re-entering interpreter dispatch after its `loomAbort` fires in the same sub-step-2 pass, and that the parent's `disposeBarrier` settles independently of whether its child surfaced an `Err`. The sibling shard-12 topic files (`session-only-degraded-state.md`, `drain-state-contract.md`) carry per-branch `## Acceptance criteria` sections, but the paired test leaf `V9g-T` covers no concurrent parent+child teardown scenario, so the nested-invocation path has no conformance anchor.

The defect entered with the paragraph itself in `7f37dd7` and has never been touched in the conceptual sense intended by the finding.

## Solution approach

Add a `## Acceptance criteria` block to `session-shutdown-semantics.md`, co-located with the `invoke`-parent observation rule and matching the sibling shard-12 files' per-branch fixture style, and mirror it as a Tests bullet on `V9g-T`. Express the criterion in observable terms only — the "no re-entering interpreter dispatch" claim is interpreter-internal and must be pinned through observable proxies: both `disposeBarrier` promises settling within `SHUTDOWN_AWAIT_CAP_MS`, no post-abort loom-result, and no additional system-note beyond the parent's own `loom/runtime/cancelled-by-session-shutdown`.

## Solution constraints

- Measure emission counts at the handler/`finally` invocation site, not at the operator sink — matching the count semantics the sibling shard-12 acceptance sections pin.
