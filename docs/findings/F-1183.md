## Finding

Two consumption sites in the core-language spec hang load-bearing observable behaviour on properties of PIC-13 `FileSystem` members that PIC-13 itself does not pin:

1. **`realpath` — invocation.md `Resolution`.** The discovery-root containment check (load-time and the INV-1 invocation-time re-check) decides escape by the *byte-exact output of* `FileSystem.realpath` applied to both the resolved callee path and each active discovery root. Correctness of this rule requires two `realpath` properties: (a) **full transitive symlink resolution** — every symlink anywhere on the path, not just the leaf, is followed (a symlink farm whose interior link aims outside a discovery root must be detectable); and (b) **byte-stable canonical output for the same on-disk target** — two inputs that resolve to the same directory entry must produce identical bytes, otherwise byte-for-byte segment-boundary containment can fire spuriously or fail to fire. PIC-13's `realpath` bullet today specifies its `.code` rejections (`ELOOP`/`ENOENT`/`EACCES`/`EPERM`) and a case-canonicalisation guarantee for case-insensitive filesystems; its only "follows symlinks" pin is the inline comment "DOES follow symlinks, unlike `lstat`", which leaves *transitive* resolution implicit and says nothing about byte-stability beyond the case-only carve-out. Neither half of the contract that invocation.md relies on is asserted as a normative MUST at the seam.

2. **`readdir` — imports.md IMP-1.** IMP-1's unresolvable-import rule is decided by a byte-for-byte match of the import literal's final segment against "the bytes [`FileSystem.readdir`](./pi-integration-contract/host-interfaces-services.md#pic-13) returns for the resolved parent directory, not against any Unicode-normalised form of the path literal (NFC vs NFD spellings are not folded)". This is sound only if PIC-13's `readdir` guarantees its returned strings are (i) the raw on-disk bytes interpreted as UTF-8 (or otherwise pre-defined) and (ii) not subjected to any Unicode normalisation by the seam. PIC-13's `readdir` bullet today says only "entry names only (no full paths); rejects with the same Node-style `.code` shape as `readText`". No encoding pin, no normalisation pin. Two conforming `FileSystem` implementations — one that hands names through verbatim (Node's `fs.promises.readdir` default behaviour) and one that NFC-normalises on the way out (a plausible "be helpful" cross-platform adapter) — would diverge on whether an NFD-spelled `import` literal matches an NFC on-disk entry.

The defect is symmetric across the two members: the seam is the single source of truth for what the runtime is allowed to assume about the filesystem, and the consumption sites already lean on guarantees the seam never made. The case-canonicalisation clause F-0977 already added to `realpath` is the template — extend it on `realpath` to cover full transitive symlink resolution and byte-stability, and add an analogous clause on `readdir` covering UTF-8 byte preservation and no Unicode normalisation.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `8a6d15f` (2026-05-06) "pi-loom spec: resolve \"FileSystem seam interface signature lives outside the spec corpus\"" — introduced PIC-13's `readdir` member with the deliberately narrow "entry names only … rejects with the same Node-style `.code` shape as `readText`" contract; no encoding or normalisation pin.
- `76c649f` (2026-06-04) "pi-loom spec: resolve \"realpath required by Resolution but absent from FileSystem seam\"" — added `realpath` to PIC-13 with the narrow `.code` rejection contract and the inline "DOES follow symlinks" comment, but no normative pin on transitive resolution or byte-stability.
- `9aa73b1` (2026-06-07) "pi-loom spec: resolve \"Filesystem case-sensitivity for .warp basenames and invoke/tools containment\"" — rewrote IMP-1 to decide unresolvability against the byte-for-byte `readdir` return value with explicit NFC-vs-NFD non-folding, and rewrote invocation.md *Resolution* to decide containment on the "byte-exact output of `FileSystem.realpath`". This is the commit that lifted both consumption sites onto contract surfaces the seam never pinned.
- `a765f6c` (later, F-0977 fix) "pin PIC-13 realpath case-canonicalisation guarantee for DISC-3 dedup" — partially closed the `realpath` half by pinning byte-identity across case-variant inputs on case-insensitive filesystems; left full-transitive-resolution and general byte-stability unpinned, and did not touch `readdir` at all.

**History:** The defect emerges from the interaction across the seam-introduction commits and the consumption-site sharpening in `9aa73b1`. Each PIC-13 member was added with a narrow rejection-only contract; `9aa73b1` then routed two independent observable rules — invocation-time symlink-escape detection and import byte-exact match — through properties of those members that the narrow contracts never pinned. F-0977's `a765f6c` follow-up closed one corner of the `realpath` half (case-only canonicalisation), establishing the template the remaining clauses should follow but not extending it to the still-unpinned `realpath` properties or to `readdir` at all.

## Solution Space

**Shape:** multiple

This finding bundles two independent seam obligations on two different PIC-13 members. Each is resolvable by a single bullet-level edit at PIC-13, with at most a one-sentence anchored cross-ref added at the consumption site; they share no shared edit surface and the fix templates differ (one pins symlink/byte-stability behaviour, the other pins encoding/normalisation behaviour). Per the bimodal-obligation guidance they are split into separately-loopable options.

### Option A — Pin `realpath` transitive resolution + byte-stability on PIC-13

**Approach.** Extend PIC-13's `realpath` bullet (anchor `id="pic-13"`, `host-interfaces-services.md`) with a clause stating:
- **Transitive symlink resolution:** `realpath` MUST resolve every symlink on the path (intermediate components as well as the leaf), so a symlink anywhere in the chain that aims outside a discovery root is observable to callers of the containment check.
- **Byte-stable canonical output:** two input paths that name the same on-disk target MUST yield byte-identical `realpath` output, beyond the case-only carve-out the existing case-canonicalisation clause already pins. Frame this as the general invariant of which the F-0977 case-canonicalisation clause is one corner.

State the clause binds every conforming seam implementation (the production `PiFileSystem` adapter and the `FakeFileSystem`), parallel to the case-canonicalisation clause's framing. At the consumption site, rewrite invocation.md's *Resolution* paragraph to cite this clause by anchored link the first time it names `FileSystem.realpath`, replacing the bare-term reference.

**Spec edits.**
- `host-interfaces-services.md` — append a transitive-resolution + byte-stability MUST to the existing `realpath` bullet, sibling to the case-canonicalisation paragraph.
- `invocation.md` — promote the first `FileSystem.realpath` mention in *Resolution* to an anchored link to PIC-13's `realpath` bullet (`./pi-integration-contract/host-interfaces-services.md#pic-13`).

**Pros.** Closes a load-bearing security-boundary gap; reuses the F-0977 case-canonicalisation precedent verbatim; adds one bullet plus one anchored cross-ref. Conformance is testable through the existing `FakeFileSystem` symlink-table fixture by adding an interior-symlink case.

**Cons.** Tightens the production adapter's contract — `PiFileSystem` may need a post-call canonicalisation step when delegating to `fs.promises.realpath` (the case-canonicalisation clause already flagged this for leaf case; the byte-stability extension generalises that obligation).

**Risks.** A host whose native `realpath` does not perform transitive resolution (none of the loom 1.0 target platforms behave this way) would require an explicit chain-walking adapter; the spec already accepts this kind of adapter-layer obligation in the case-canonicalisation clause.

### Option B — Pin `readdir` UTF-8 byte preservation + no-Unicode-normalisation on PIC-13

**Approach.** Extend PIC-13's `readdir` bullet (anchor `id="pic-13"`, `host-interfaces-services.md`) with a clause stating:
- **UTF-8 byte preservation:** entries MUST be returned as the raw on-disk filename bytes interpreted as UTF-8 strings (sibling to the `readBytes` UTF-8 framing already pinned on `pic-13-readbytes` and to `lexical.md` *Encoding*).
- **No Unicode normalisation:** the seam MUST NOT apply NFC, NFD, NFKC, or NFKD normalisation on the way out; an entry whose on-disk bytes spell `café` in NFD is returned in NFD.

State the clause binds every conforming seam implementation, parallel to the `realpath` clauses. IMP-1's existing `[FileSystem.readdir](.../host-interfaces-services.md#pic-13)` link already targets the right anchor and needs no edit.

**Spec edits.**
- `host-interfaces-services.md` — append a UTF-8 + no-normalisation MUST to the existing `readdir` bullet.

**Pros.** Closes the IMP-1 byte-for-byte contract gap with a one-bullet edit; the consumption-site citation is already in place. Aligns `readdir` with the encoding posture already pinned for source-file bytes via `readBytes` and *Lexical — Encoding*.

**Cons.** Tightens conforming adapter behaviour against any "be helpful, normalise to NFC" implementation; this is rare in Node but possible in a cross-platform abstraction layer.

**Risks.** None beyond Option A's adapter-shape obligation.

### Recommendation

Adopt both options. Resolve **Option B first**, then **Option A**.

- Option B is strictly smaller (one bullet, one member, no consumption-site edit) and scope-bounding: it closes the IMP-1 gap with no ripple. Landing it first establishes a stable baseline for Option A's larger edit.
- Option A then extends the F-0977 case-canonicalisation precedent in the same bullet it already lives in, plus one anchored cross-ref at the invocation.md consumption site. Sequencing this second keeps invocation.md's edit landing on a `readdir`-clean PIC-13.

Edge cases the implementer must watch:
- The `realpath` byte-stability clause must explicitly subsume the case-canonicalisation clause as the general rule of which case-canonicalisation is one corner, not contradict it; phrasing as "in addition to the case-canonicalisation guarantee above" is safer than rewriting that clause.
- The `readdir` UTF-8 clause should explicitly cross-reference `lexical.md` *Encoding* so the encoding posture is single-sourced; it should not re-derive UTF-8 framing.
- Neither edit changes the existing `.code` rejection list — keep that list intact and append the new clauses as separate sentences.

## Consequence

**Severity:** correctness

Two reasonable implementers of the `FileSystem` seam — one verbatim, one that NFC-normalises `readdir` output or that uses a `realpath` variant which canonicalises only the leaf component — would silently diverge on (a) whether a symlink farm whose interior link aims outside a discovery root is rejected by INV-1, a load-bearing security boundary, and (b) whether an NFD-spelled `import "./café.warp"` resolves on a host that stores the entry as NFC, an authoring-experience and portability boundary. Both behaviours are observable and conformance-testable, and neither has a normative seam clause to test against today.
