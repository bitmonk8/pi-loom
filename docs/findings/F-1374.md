## Problem

`cancellation.md` §Granularity states a normative obligation — the interpreter checks the cancellation signal immediately before each of five checkpoint sites (each `for`/`while` iteration, each `@`-query dispatch, each tool call, each `invoke`, and the slash-command argument binder's LLM call) and at no other node kinds, together with the loop-iteration macrotask-yield property. That obligation has no row in `coverage-matrix.md`: the *Numbered REQ-IDs* table maps `CNCL-1`…`CNCL-6` to `V17a` (none of them §Granularity), and the two `cancellation.md` *Code-keyed obligation areas* rows (`cka-32`, `cka-33`) cover §Forwarding-listener and §Race semantics, not §Granularity — so its closing leaf `V17c` is named by no row at all. Compounding this, both the `V17c` and `V17c-T` `Tests` bullets cite only prose section links with no machine-resolvable token (no REQ-ID, no `cka-<n>`), inconsistent with the sibling pattern where `V14b` cites its row inline as `(code-keyed-area token cka-13)`.

## Issue introduction

**History:** The cancellation coverage was first authored in c6a664e, which created the `CNCL-1`…`CNCL-6` rows pointing at `V17a` but never created a row for the §Granularity obligation (`git log -S 'Granularity' -- docs/plan_topics/coverage-matrix.md` is empty across all history). The gap stayed latent while `V17a` was the single cancellation leaf, since §Granularity rode along inside `V17a`'s bundle. Commit 5b95143 peeled §Granularity onto a new dedicated leaf — adding `V17c-checkpoint-granularity.md` and `V17c-T-checkpoint-granularity.md` with prose-only `Tests` citations and editing `V17a` — without touching `coverage-matrix.md`, concretising the gap: the obligation's now-dedicated closing leaf has no traceability row at all and no machine-resolvable token in its tests.

## Solution approach

Add one row to the *Code-keyed obligation areas (no numbered REQ-IDs)* table in `docs/plan_topics/coverage-matrix.md` whose spec-area cell names `cancellation.md` (CNCL) §Granularity (the fixed five-site checkpoint set and the loop-iteration macrotask-yield property) and whose closing-leaf cell is `` `V17c` ``, taking the next free append-only `cka-` token (current table tail is `cka-46`). Then retarget the prose-only citations in both the `V17c` and `V17c-T` `Tests` bullets to cite that token inline alongside the existing seam reference, matching the `V14b` `(code-keyed-area token cka-13)` pattern. No `H5b` `Deps.` change is needed — `V17c` is already reachable through the `V17a`–`V17c` range entry, so `H5d` resolves the new row once it exists.

## Solution constraints

- The *Token* column of the *Code-keyed obligation areas* table is append-only: take the next free token and never reuse or renumber an existing one.
- Honour the *Closing-leaf column authoring* rule — `V17c` must be the only backtick-delimited span in the new row's closing-leaf cell, and any code identifiers in the spec-area annotation prose are written bare.
