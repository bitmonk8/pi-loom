## Finding

`conversation-drive.md` carries normative requirements across three distinct capability surfaces: prompt-mode driving (`pi.sendUserMessage` / `waitForIdle`, PIC-50/51/53), the typed-query two-phase tool loop (free phase plus forced respond turn), and provider-compatibility enforcement (the load-time `loom/load/typed-query-unsupported-provider` warning and the runtime unsupported-provider `TransportError`). Each loom 1.0 boundary the page commits to is stated only at its own site and in its own form: `deliverAs: "followUp"` "is reserved for future considerations and is not emitted by loom 1.0" appears inline mid-paragraph; mid-loom user-session replacement and the typed-query supported-provider set are each a visually distinguished "loom 1.0 seam" blockquote; the forced-tool non-compliance limitation is a separate "loom 1.0 diagnostic limitation" block.

There is no single place on the page — nor any single place that aggregates this page — from which a reader can enumerate what loom 1.0 deliberately excludes here. The corpus does maintain a central non-goals page (`future-considerations/model-changes-and-non-goals.md#v1-non-goals`), but it records only the cross-cutting single-active-session disposition that touches this page; it does not enumerate the page-local deferred seams (`deliverAs: "followUp"` non-emission, typed-query provider-set widening). An implementer or auditor asking "what does conversation-drive deliberately not do in loom 1.0?" must read the page end-to-end and recognise each scattered seam box, inline aside, and limitation block individually.

This is an organisation/discoverability gap, not an under-specification: every boundary is stated normatively somewhere. The defect is that the boundary set for a multi-capability page is distributed across four heterogeneous in-text devices with no collected entry point.

## Solution Space

**Shape:** multiple

### Option A — Page-local boundary orientation aside (link, don't restate)

- **Approach:** Add a short orientation aside near the top of `conversation-drive.md` (e.g. a "loom 1.0 boundaries" note) that enumerates the page's exclusions in one place — `deliverAs: "followUp"` not emitted; mid-loom user-session replacement deferred; typed-query supported-provider-set widening (Gemini, OpenAI Responses, JSON-mode fallback) deferred; no forced-tool provider-vs-model non-compliance separation — and links to each canonical site (the two seam blockquotes, the inline `deliverAs` clause, the `loom 1.0 diagnostic limitation` block, and the central `#v1-non-goals` entry). The aside points; it does not duplicate the normative text.
- **Spec edits:** One new short subsection at the top of `conversation-drive.md`; no change to the existing seam boxes or normative clauses.
- **Pros:** Directly answers the finding's concern (in-page discoverability); a reader of this page gets the boundary set without hunting; keeps the canonical normative text single-sourced at its existing sites.
- **Cons:** Introduces a per-page boundary-index device the rest of the corpus does not use; if it restates rather than links it risks the duplication the cruft/placement lenses elsewhere flag.
- **Risks:** The aside drifts out of sync if a future seam is added without updating it.

### Option B — Extend the central non-goals page with the page-local seams

- **Approach:** Add the two page-local deferred seams not currently recorded centrally (`deliverAs: "followUp"` non-emission; typed-query supported-provider-set widening) as entries in `model-changes-and-non-goals.md#v1-non-goals`, each with a `Recorded at:` backlink to its `conversation-drive.md` site, matching the established centralization-plus-backlink pattern already used for the single-active-session and mid-loom-replacement dispositions.
- **Spec edits:** Two new bullets in the central non-goals list with backlinks; no change to `conversation-drive.md` beyond ensuring anchor targets exist.
- **Pros:** Aligns with the corpus's existing single-home-for-non-goals convention; one place to audit all loom 1.0 non-goals corpus-wide; no new per-page device.
- **Cons:** Does not give a reader *of `conversation-drive.md`* a local entry point — they must already know the central page exists; the page-internal "read every seam box" friction the finding names remains.
- **Risks:** Conflating page-local *deferred seams* with cross-cutting *non-goals* on the central page, which currently distinguishes the two categories explicitly.

### Recommendation

Option A. The finding's concern is intra-page discoverability of a multi-capability page's boundaries, which only a page-local entry point resolves; keep it a linking index (not a restatement) so the canonical normative text stays single-sourced and the corpus's anti-duplication discipline is preserved. Watch the edge case that the aside must link to, not copy, the seam-box and limitation-block wording, and that every anchor it references already exists on the page.

## Consequence

**Severity:** advisory

A reader cannot enumerate the page's loom 1.0 exclusions without reading it in full and recognising each scattered seam box, inline aside, and limitation block; nothing breaks at runtime and implementers can still produce a working system because every boundary is stated normatively at its site. The cost is review and onboarding friction on a load-bearing multi-capability page.
