## Finding

`H5a`'s **Adds** describes the gate machinery's warn-only live-corpus mode as one "that runs the same four live-corpus reconciliation surfaces", then enumerates them: the live spec REQ-ID set and the live numbered-REQ-ID citing-test surface "this leaf owns", the live `spec_topics/**` normative-MUST surface "the `H5e` arm contributes", and the live multi-leaf-row per-facet citing-test surface "the `H5f` arm contributes". The verb attaches the complete four-surface run to the warn-only mode `H5a` itself exposes.

But `H5e` and `H5f` both declare `Deps. H5a` — they land *after* `H5a`. At `H5a` completion only the two surfaces this leaf owns exist; the un-anchored-MUST surface and the per-facet citing-test surface materialise only once `H5e` and `H5f` register their arms into the shared machinery, and the four-surface run is first consumed by `H5b` (whose `Deps` name `H5a`, `H5e`, and `H5f`). The build order is sound — nothing is mis-sequenced and no dependency is missing — but the prose presents a four-surface aggregator as a thing `H5a` ships, where `H5a`'s own `Tests` and `Ships when` correctly exercise only its two owned arms against the seeded fixtures. An implementer reading the `Adds` sentence in isolation could read the H5e/H5f surfaces as in-scope for `H5a` and attempt to pull that work forward, against a run that cannot compile until those arms exist.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H5a-closing-gate-automation.md` **Adds**, revise the warn-only-mode sentence so it states the temporal ownership the dependency graph already enforces: `H5a` establishes the warn-only framework plus the two reconciliation surfaces it owns (the live spec REQ-ID set and the live numbered-REQ-ID citing-test surface), into which the `H5e` un-anchored-MUST arm and the `H5f` per-facet citing-test arm register their surfaces when those leaves land; the complete four-surface run therefore materialises only once `H5e`/`H5f` are in place and is first consumed by the `H5b` canary. Keep the existing per-surface arm attributions (`H5e` contributes the normative-MUST surface, `H5f` the per-facet surface) — they are correct; only the framing that the four-surface run exists *at `H5a`* needs correcting. No `Deps` change: `H5a` continues to depend only on `H1a`, and its `Tests`/`Ships when` continue to exercise its two owned surfaces against the seeded fixtures.

## Consequence

**Severity:** advisory

An implementer reading only the `Adds` warn-only sentence may treat the `H5e`/`H5f`-contributed surfaces as `H5a`-scoped and attempt to implement a four-surface run at `H5a`, which cannot exist until those dependent arms land. The leaf's own `Tests` and `Ships when` constrain it to its two owned surfaces, so a working `H5a` is still producible; the cost is wasted forward-pull effort and reviewer confusion, not a broken build.
