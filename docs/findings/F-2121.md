## Problem

PIC-53 (`#untyped-query-ok-extraction`) fixes the final turn's left boundary as "the last `user` message (the loom-issued `pi.sendUserMessage` turn)" and PIC-51 (`#pic-51`)'s `stopReason: "error"` probe reuses that boundary. Both silently presuppose that the loom-issued `pi.sendUserMessage` — including the `deliverAs: "steer"` form injected mid-stream into an already-running turn whose original `user` and partial `assistant` content already sit in the entry stream — materializes in the `ReadonlySessionManager` entry stream as the trailing `user` entry that demarcates the turn. If Pi instead folds a steered message into the in-progress user turn or records it as a non-`user` entry, "the last `user` message" resolves to the pre-steer turn, and the extracted `Ok(string)` and the `stopReason: "error"` probe target silently span the wrong boundary with no build-time signal. The assumption carries identical weight to the sibling consumption postures already recorded on the same page (`#driven-turn-commit-ordering-presupposition`, `#waitforidle-no-rejection-presupposition`) but is left implicit inside the PIC-53 prose.

**History:** `git log -S 'the last \`user\` message (the loom-issued'` over the corpus returns the boundary phrase as a net addition only in `024215d`, the commit that first authored the untyped-query `Ok(string)` extraction mechanism; there is no prior occurrence. The defect is therefore co-extensive with the feature: the extraction was specified keying on "the last `user` message (the loom-issued `pi.sendUserMessage` turn)" without ever pinning that the steered loom message materializes as that trailing `user` entry. Two later commits touched the text without introducing or curing the gap — `f5e89f4` (2026-06-04, the 100KB/30KB size-cap split) relocated the paragraph verbatim into `conversation-drive.md`, and `1acdc55` attached the `PIC-53` REQ-ID label and `#pic-53` anchor to the pre-existing paragraph. The sibling consumption-posture paragraphs that should have set the precedent for this one were added separately (the `waitForIdle` no-rejection posture in `1ca13ec`), never extended to cover steer materialization.

## Solution approach

Add a loom-side consumption-posture paragraph to `conversation-drive.md` with its own `<a id="...">` anchor, modelled on the sibling `#driven-turn-commit-ordering-presupposition` and `#waitforidle-no-rejection-presupposition` paragraphs, recording that PIC-53's trailing-turn boundary (`#untyped-query-ok-extraction`) and PIC-51's reused probe (`#pic-51`) presuppose the loom-issued `pi.sendUserMessage` — in both the idle and `deliverAs: "steer"` forms — materializes as the trailing `user` entry, and that `@earendil-works/pi-coding-agent` encodes no typed contract guaranteeing this. Forward-cross-reference PIC-51 and PIC-53 from the new paragraph. Add a corresponding item to the *Editorial-review checklist for unpinned host presuppositions* under `version-bump-step2.md` / `#pi-version-bump-procedure`, alongside the existing commit-ordering and no-rejection items.
