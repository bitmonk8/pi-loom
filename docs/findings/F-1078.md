## Finding

H4a pins the factory to its synchronous arm and concludes that, because the factory body returns synchronously with no awaited work, *"MUST NOT throw out of the factory body" is the complete never-fault prohibition for this boundary.* The leaf states this completeness is load-bearing on every factory-body registration/subscription call "completing â€” and taking effect â€” synchronously: none returns a `Promise` whose resolution is required for the registration to be durable." It then enumerates five factory-body calls: `pi.registerFlag`, `pi.registerMessageRenderer`, and the three factory-time `pi.on` subscriptions (`resources_discover`, `session_start`, `session_shutdown`).

The synchronous-durability property is cited from the spec for exactly **one** of those five calls. The leaf points to [`extension-bootstrap-and-per-loom.md` Â§"Renderer registration"](../../../docs/spec_topics/pi-integration-contract/extension-bootstrap-and-per-loom.md#renderer-registration), which pins `pi.registerMessageRenderer` as returning `void` (synchronous) with "no separate Promise rejection path." No equivalent pin exists for `pi.registerFlag` or the factory-time `pi.on` subscriptions. Worse, the spec's own **Extension-bootstrap SDK failures** rule opens by contemplating "A throw **or rejection** from any of the registration calls in steps 1â€“5," so the corpus elsewhere treats an asynchronous rejection from these calls as a live possibility rather than a foreclosed one.

The H4a Tests bullet asserts only that the *factory's own* return value is `undefined` and not a thenable. It does not assert anything about the return values of the individual `registerFlag` / `pi.on` calls. So a `registerFlag` or `pi.on` call that returned a later-rejecting `Promise` would satisfy every H4a test bullet yet escape the per-call synchronous `try`/`catch` as an unhandled rejection â€” exactly the fault the "complete never-fault prohibition" claims is impossible. The claim is sound for the renderer and asserted-without-backing for the other four.

## Solution Space

**Shape:** multiple

### Option A â€” Extend the spec's synchronous-durability pin to `registerFlag` and the factory-time `pi.on` subscriptions

- **Approach.** Verify the declared return types of `pi.registerFlag` and the factory-time `pi.on` subscriptions at the loom 1.0 Pi-SDK pin (`dist/core/extensions/types.d.ts`). If they are void-returning synchronous registration sinks, add a void/no-rejection clause for them in `extension-bootstrap-and-per-loom.md` parallel to the renderer's existing "returns `void` (synchronous) â€¦ there is no separate Promise rejection path" sentence, then widen H4a's citation to cover all five factory-body calls.
- **Plan edits.** In `H4a` Adds, replace "The spec establishes that property for the renderer call at â€¦" with a citation that covers `registerFlag` and the factory-time `pi.on` subscriptions as well as the renderer.
- **Spec edits.** In `extension-bootstrap-and-per-loom.md`, add the void/no-rejection synchronous-durability clause for `pi.registerFlag` and the three factory-time `pi.on` subscriptions.
- **Pros.** Keeps the completeness claim globally true; matches the renderer precedent and the direction `a93b838` already started; no extra test surface.
- **Cons.** Requires a spec edit; only valid if the declared signatures are genuinely void/synchronous at the pin.
- **Risks.** If `pi.on` or `pi.registerFlag` returns a thenable at the pin, the clause would be false and Option A is unavailable for that call.

### Option B â€” Narrow the completeness claim and guard the unbacked calls' returned values

- **Approach.** Restrict the "complete never-fault prohibition" wording in H4a to the spec-backed renderer call; for `registerFlag` and the factory-time `pi.on` subscriptions, the per-call `try`/`catch` is supplemented by a returned-value non-thenable guard so a returned `Promise` cannot escape unobserved.
- **Plan edits.** In `H4a` Adds, narrow the completeness wording to the renderer; in `H4a` Tests, add an assertion that the `registerFlag` and the three `pi.on` calls' return values are non-thenable (or are guarded), not only the factory's own return.
- **Spec edits.** None.
- **Pros.** No spec edit; robust even if a future Pi minor makes one of these calls return a thenable.
- **Cons.** More test surface; the completeness wording becomes conditional; `V9k` may need to model a returned-value-rejection arm.
- **Risks.** `V9k` owns the per-surface abort granularity for `registerFlag` / `pi.on`; under this option its Ships-when must also exercise the returned-value-rejection arm, so the two leaves must be kept consistent.

### Recommendation

Verify the declared return types of `pi.registerFlag` and the factory-time `pi.on` subscriptions at the loom 1.0 Pi-SDK pin. If both are void/synchronous registration sinks â€” as the renderer is â€” take **Option A**: extend the spec's void/no-rejection clause and the H4a citation to cover them, so the completeness claim stays globally backed (consistent with the precursor `a93b838` direction, which already routed the renderer this way). If either call returns a thenable at the pin, take **Option B** for that call. Edge case: whichever option is chosen must stay consistent with `V9k`'s per-surface abort granularity for `registerFlag` / `pi.on` â€” under Option B, `V9k`'s Ships-when must also exercise the returned-value-rejection arm.

## Consequence

**Severity:** correctness

The factory's headline safety guarantee â€” that nothing faults out of the boundary â€” is established only for `registerMessageRenderer`; for `registerFlag` and the three factory-time `pi.on` subscriptions it is an unbacked generalisation. An implementer who trusts the completeness claim and one who reads the spec's "throw or rejection" framing would build different boundaries: the former relies solely on the synchronous `try`/`catch`, the latter also guards the returned value. A `registerFlag`/`pi.on` call that returns a rejecting `Promise` then escapes as an unhandled rejection, and the H4a test (which checks only the factory's own non-thenable return) stays green.
