## Problem

The plan splits `no-broad-catch` enforcement across two surfaces — the `H2a` ESLint rule and the `H5c` closing-gate reconciliation arm — but the three places that define their division of labour disagree on whether the **H2a lint resolves the cited exemption token** or only checks that the `// allow-broad-catch:` comment is **present**. `H2a` Adds (the `no-broad-catch` clause) frames the rule as presence-only, while the `H2a` first Tests bullet and the `conventions.md` §*Specific exception types only* Enforcement-posture sentence both require the cited token to *resolve* against `coverage-matrix.md` and the diagnostics registry. The resolution reading duplicates `H5c` (whose entire purpose is the token-resolution arm) and cannot pass at `H2a`'s build position: `H2a` depends only on `H1a` and runs on every `npm test` long before those corpora are populated or the live-corpus footing `H5c`/`H6a` own is established.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `96563f9` (2026-06-10, "resolve broad-catch allow-comment predicate unsatisfiable for exempt sites") and `6ad39e0` (2026-06-16, "resolve Broad-catch exemption: category-membership enforcement posture is unstated")
**History:** The defect is not present at inception; it accreted as successive findings were resolved, each touching one surface without reconciling the H2a-lint-vs-H5c division. `96563f9` introduced the four-arm token enumeration into the `H2a` first Tests bullet (turning the H2a pass criterion into a resolution check). `506d71c` (2026-06-12) gave `H2a` Adds its presence-only "no separate allow-list file read" phrasing, which does not require resolution. `6ad39e0` then added the `conventions.md` Enforcement-posture sentence that lumps the H2a lint together with the closing-gate token-resolution arm as both verifying "whose cited token resolves." The contradiction is the interaction of these surfaces: Adds (presence) vs Tests + conventions Enforcement-posture (resolution).

## Solution approach

Pin the `H2a` `no-broad-catch` lint to comment presence and well-formed-comment shape only, and confine token resolution to `H5c`. In the `H2a` first Tests bullet, strike the four-arm token-resolution enumeration from the passing-fixture criterion so the pass condition is a well-formed same-line `// allow-broad-catch:` comment. In `H2a` Adds (the `no-broad-catch` clause), state that the lint checks comment presence and shape and does not resolve the cited token. In the `conventions.md` §*Specific exception types only* Enforcement-posture sentence (and the checklist-item-6 conflation that repeats it), split requirement (a) so that only the `H5c` closing-gate arm verifies the cited token resolves, with the `H2a` lint verifying comment presence.

## Solution constraints

- Token resolution against `coverage-matrix.md` and the diagnostics registry stays owned solely by `H5c`; the `H2a` lint must not read those corpora.
- The four admitted arms and their resolvers (the `cka-<n>` "matches exactly one row's *Token* cell" resolver, the concrete-`loom/...`-code resolver, and the `pi-sdk-boundary` token) stay defined in `conventions.md`/`H5c` unchanged.
