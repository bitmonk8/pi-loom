## Finding

The plan describes the `no-broad-catch` enforcement as two surfaces — the `H2a` ESLint rule and the `H5c` closing-gate reconciliation arm — but the three places that define their division of labour disagree on whether the **H2a ESLint rule resolves the cited exemption token** or only checks that the comment is **present**.

- `H2a` Adds (`docs/plan_topics/H2a-cross-cutting-gates.md`) frames the rule as presence-only: it permits a broad catch "at an exempt-site shape whose same line carries the `// allow-broad-catch:` comment, the set of those comments across `src/**` being the lint allow-list, with no separate allow-list file read." Nothing here requires the lint to resolve the token against any other file.
- `H2a` first Tests bullet contradicts that: the passing fixture's comment must cite "any token the widened gate predicate admits — a coverage-matrix REQ-ID, a `cka-<n>` token (matching exactly one row's *Token* cell in `coverage-matrix.md`), a concrete `loom/...` registry code, or the structural `pi-sdk-boundary` token." A pass keyed on "matching exactly one row's *Token* cell" or "a concrete registry code" is a *resolution* check, not a presence check — it forces the ESLint rule to read `coverage-matrix.md` and the diagnostics registry.
- `conventions.md` §*Specific exception types only* (Enforcement-posture sentence) reinforces the resolution reading and attributes it to both surfaces: "The two mechanical surfaces — the `H2a` `no-broad-catch` lint and the loom 1.0 closing-gate token-resolution arm above — verify only requirement (a), that each exempt site carries a `// allow-broad-catch:` comment whose cited token resolves."

This duplicates `H5c`, whose entire purpose (`docs/plan_topics/H5c-broad-catch-allow-list-gate.md`) is the token-resolution arm — scanning the `// allow-broad-catch:` comments and failing when a cited token resolves to none of the four admitted arms, on the seeded-fixture-then-live-corpus footing that `H5a`/`H6a` own. Worse, `H2a`'s only dependency is `H1a`, and its lint runs on every `npm test` from that point on — long before `coverage-matrix.md` and the diagnostics registry are populated, and well before `H5c`/`H5a` establish the live-corpus footing. An ESLint rule that resolved `cka-<n>` and `loom/...` tokens against those corpora at H2a's build position would either redden continuously or require an unusual cross-file capability that re-implements `H5c`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `96563f9` (2026-06-10, "resolve broad-catch allow-comment predicate unsatisfiable for exempt sites") and `6ad39e0` (2026-06-16, "resolve Broad-catch exemption: category-membership enforcement posture is unstated")
**History:** The defect is not present at inception; it accreted as successive findings were resolved, each touching one surface without reconciling the H2a-lint-vs-H5c division. `96563f9` introduced the four-arm token enumeration into the `H2a` first Tests bullet (turning the H2a pass criterion into a resolution check). `506d71c` (2026-06-12) gave `H2a` Adds its presence-only "no separate allow-list file read" phrasing, which does not require resolution. `6ad39e0` then added the `conventions.md` Enforcement-posture sentence that lumps the H2a lint together with the closing-gate token-resolution arm as both verifying "whose cited token resolves." The contradiction is the interaction of these surfaces: Adds (presence) vs Tests + conventions Enforcement-posture (resolution).

## Solution Space

**Shape:** single

### Recommendation

Pin the `H2a` `no-broad-catch` lint to comment **presence** (and well-formed-comment shape) only, and confine token **resolution** to `H5c`.

- In `docs/plan_topics/H2a-cross-cutting-gates.md` first Tests bullet, strike the four-arm token enumeration from the passing-fixture criterion. The passing fixture's pass condition is that the exempt site carries a well-formed `// allow-broad-catch: <token> — <spec-page>` comment on the same line; the failing fixture is a broad catch with no such comment. Remove the "cites any token the widened gate predicate admits — a coverage-matrix REQ-ID, a `cka-<n>` token (matching exactly one row's *Token* cell …), a concrete `loom/...` registry code, or the structural `pi-sdk-boundary` token" clause. That four-arm resolution enumeration already lives in `H5c`'s Tests, where it belongs.
- In `docs/plan_topics/H2a-cross-cutting-gates.md` Adds, the `no-broad-catch` clause is already presence-only; state explicitly that the lint checks comment presence and shape and does not resolve the cited token against `coverage-matrix.md` or the diagnostics registry — that resolution is `H5c`'s arm.
- In `docs/plan_topics/conventions.md` §*Specific exception types only* Enforcement-posture sentence, split requirement (a) across the two surfaces rather than attributing "whose cited token resolves" to both: the `H2a` `no-broad-catch` lint verifies the exempt site carries a well-formed `// allow-broad-catch:` comment; the loom 1.0 closing-gate token-resolution arm (`H5c`) verifies the cited token resolves to one of the four admitted arms. The checklist-item-6 sentence ("the `H2a` `no-broad-catch` lint and the loom 1.0 closing-gate token-resolution arm verify only that the comment's cited token resolves") carries the same conflation and should be reconciled to the same split on touch.

Token resolution against `coverage-matrix.md` and the diagnostics registry remains owned solely by `H5c`; the `H2a` lint must not read those corpora (it depends only on `H1a` and runs before they are populated). The existing project pins hold: the four admitted arms, the `cka-<n>` "matches exactly one row's *Token* cell" resolver, the concrete-`loom/...`-code (no glob) resolver, and the `pi-sdk-boundary` token stay defined in `conventions.md`/`H5c` unchanged.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge at `H2a`: one reads the Adds and builds a presence-only lint; the other reads the Tests bullet and the conventions Enforcement-posture sentence and builds a lint that resolves `cka-<n>`/`loom/...` tokens against `coverage-matrix.md` and the diagnostics registry — an arm that cannot pass at H2a's build position (those corpora are unpopulated, and the live footing is `H5c`/`H6a`'s) and that duplicates `H5c`. The H2a Tests four-arm enumeration would also be authored as a resolution test, colliding with `H5c`'s ownership of that check.
