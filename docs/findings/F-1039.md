## Problem

The "Panics surface to the loom's caller as:" enumeration in `error-model.md` lists only two routing surfaces — the slash-command system note and the `invoke` parent — but `tool-calls.md` §Outcome enumeration pins a third surface for the `.loom`-callable parallel-tool-batch adapter, whose two MUSTs require translating a captured pre-evaluation setup throw into a clean `{ isError: true, ... }` value and forbid letting it propagate as an unhandled rejection. A PIC implementer working the `active-invocation-registry.md` *Dispatch-site setup wrap* and following its cross-reference into `error-model.md` never reaches that third surface. They can therefore satisfy both PIC pages by emitting the diagnostic plus system note and rethrowing, which silently breaks the dispatch into Pi's parallel-tool-call batch — the exact failure the second MUST forbids.

## Issue introduction

**Verdict:** present-since-inception (within the current cross-cut)
**Introducing commits:**
- `fe866f1` (2026-05-07) — "pi-loom spec: resolve 'T16 — Pre-evaluation setup failures: dispatch-site routing'"

**History:** `git log -L "/Pre-evaluation setup throw/,+1:docs/spec_topics/tool-calls.md"` shows the *Pre-evaluation setup throw inside the `.loom`-callable adapter* bullet — carrying both MUSTs verbatim — was introduced in `fe866f1` on 2026-05-07. The same commit added the *Dispatch-site setup wrap* paragraph to `spec_topics/pi-integration-contract.md` (the predecessor of `active-invocation-registry.md`, which was later split out by `f5e89f4` "docs: enforce 100KB hard / 30KB recommended size cap on spec set") and updated the runtime-defect surface paragraph in `spec_topics/errors-and-results.md` to enumerate the three dispatch-setup throw sources. The cross-cut as authored placed the two adapter MUSTs in `tool-calls.md` and the dispatch-setup-wrap obligation in PIC, but did not add the parallel-tool-batch adapter return-path surface to the "Panics surface to the loom's caller as:" enumeration in error-model.md — that enumeration has carried the same two channels (slash-command system note, `InvokeInfraError` to an `invoke` parent) since well before `fe866f1`. The visibility gap is therefore an inception defect of this cross-cut: the third surface was named in `tool-calls.md` from day one but never registered on the per-surface enumeration the PIC pages cross-reference.

## Solution approach

Add a third bullet to the "Panics surface to the loom's caller as:" enumeration in `error-model.md`, alongside the existing slash-command and `invoke`-parent bullets, naming the `.loom`-callable parallel-tool-batch adapter return-path surface and stating that it MUSTs return a clean `{ isError: true, ... }` value rather than propagate an unhandled rejection. Forward-link from that bullet to `tool-calls.md` §Outcome enumeration (the *Pre-evaluation setup throw inside the `.loom`-callable adapter* bullet) for the canonical per-surface form.

## Solution constraints

- Out of scope: the canonical per-surface form and the `<name>` token-selection rationale stay in `tool-calls.md` — do not duplicate them in `error-model.md`, and do not move the two adapter MUSTs out of `tool-calls.md`.
