## Finding

The `loom/load/settings-value-out-of-range` row's non-object-root arm in [`code-registry-load.md`](../../docs/spec_topics/diagnostics/code-registry-load.md) names the document root as `<key>` "rendered as `(root)`", and its sibling prose in [`discovery/package-and-settings.md` — Top-level shape validation](../../docs/spec_topics/discovery/package-and-settings.md) repeats the same wording. The token bound into `<key>` is therefore the seven-character literal `(root)`.

`<key>` is a category-5 placeholder in [`placeholder-rendering-b.md` §5](../../docs/spec_topics/diagnostics/placeholder-rendering-b.md), whose rule is normative and runtime-evaluated: render the key string bare iff it matches `^[A-Za-z_][A-Za-z0-9_]*$`, otherwise double-quote it. `(root)` contains the parenthesis characters and does not match that regex, so a strict implementer of §5 renders the *Message* as `settings key "(root)" value is out of range; got array`. An implementer who reads the producer-side wording "rendered as `(root)`" as a final-rendered literal renders it bare: `settings key (root) value is out of range; got array`. Both readings are defensible from the current text and produce byte-different *Message* strings — exactly the divergence DIAG-4's byte-identical contract is meant to prevent.

§5's *Test vectors* do not include the synthetic-label case (only `wibble`, `~/work/looms`, `my-key`, and `kind`), and no §7 closed-enum sub-rule covers `(root)` either — it is a loom-emitted synthetic label, not a host key. The producer-side prose was added in the same commit (89eabf1, fix F-0744) that introduced the non-object-root arm, alongside two `array` / `object` `<observed>` test vectors but no `(root)` `<key>` test vector — the synthetic-label rendering question slipped through the gap between the §5 predicate's authoring (the source-of-truth for `<key>` quoting) and the producer site (which carries the only mention of `(root)`).

## Issue introduction

**Verdict:** single-commit defect — introduced
**Introducing commits:**
- `89eabf1` (2026-06-22) `pi-loom spec-plan: fix F-0744 — define settings non-object-root outcome + array/object observed rendering`

**History:** `git log -G '\(root\)' -- docs/spec_topics/` returns a single commit, `89eabf1`, which is also the only commit touching the producer-side wording in `code-registry-load.md` and `discovery/package-and-settings.md`. That commit added the non-object-root arm, the producer-side "rendered as `(root)`" phrasing in both files, and two new test vectors for the `<observed>` array / object kinds — but did not extend §5's `<key>` rule or its test-vector list to cover the new synthetic label. The `<key>` identifier-shape predicate itself predates this commit (it has been the runtime regex `^[A-Za-z_][A-Za-z0-9_]*$` since `f5e89f4` and the explicit "runtime check" reading was added in `72e2c31`), so the predicate is stable; the defect is the unilateral introduction of a non-conforming key token at the producer site without a coordinating §5 edit.

## Solution Space

**Shape:** single

### Recommendation

In `placeholder-rendering-b.md` §5, extend the `<key>` rule with a one-sentence carve-out pinning the synthetic-label case, and add a matching test vector:

- Append to the `<key>` bullet (after the existing identifier-shape sentence): "When `<key>` binds a loom-emitted synthetic label of the form `(<label>)` (the only loom 1.0 instance is `(root)`, emitted by `loom/load/settings-value-out-of-range`'s non-object-root arm), the predicate does not apply and the label is rendered verbatim as the literal parenthesised token (bare, no double quotes). The synthetic-label form is closed under GOV-7 / GOV-8: new `(<label>)` tokens are a breaking change to the rendering surface."
- Append to §5's *Test vectors*: "A `loom/load/settings-value-out-of-range` whose offending value is a top-level JSON array renders `settings key (root) value is out of range; got array` — the synthetic `(root)` label is emitted bare, not double-quoted, per the synthetic-label carve-out above."

This pins the bare rendering (which matches the producer-side wording "rendered as `(root)`" already shipped in `code-registry-load.md` and `discovery/package-and-settings.md`, so no producer-side edit is needed), preserves §5's runtime-predicate framing for genuine host keys, and gives the V7c-T conformance test a normative byte-string to assert against. `V10c-T`'s settings-value-out-of-range top-level-shape test gains the same vector as its acceptance criterion's *got `array` / `object`* arm; cite §5's new vector by reference rather than duplicating bytes.

Watch-outs for the implementer: do **not** rename the synthetic token to bare `root` (drops the parens that the producer site already pins); do **not** widen the carve-out into a general "labels containing parens" rule (the GOV-7 / GOV-8 closure is what keeps the surface auditable); the §8 parsed-scalar carve-out for `<observed>` is unrelated and stays as-is.

## Consequence

**Severity:** correctness

Two reasonable implementations diverge on the *Message* byte-string for the non-object-root `loom/load/settings-value-out-of-range` diagnostic — one emits `settings key (root) value is out of range; got array`, the other `settings key "(root)" value is out of range; got array`. The divergence defeats DIAG-4's byte-identical contract on a single registry row and silently breaks any operator tooling that keys on either form by equality.
