## Problem

The `loom/load/settings-value-out-of-range` non-object-root arm binds the seven-character literal `(root)` into `<key>`, but §5's `<key>` rule in `placeholder-rendering-b.md` is a runtime identifier-shape predicate (`^[A-Za-z_][A-Za-z0-9_]*$`): `(root)` fails the regex, so a strict §5 implementer double-quotes it (`settings key "(root)" value is out of range; got array`), while an implementer reading the producer-side "rendered as `(root)`" wording emits it bare (`settings key (root) value is out of range; got array`). Both readings are defensible from the current text and produce byte-different *Message* strings, defeating DIAG-4's byte-identical contract on that row. §5's *Test vectors* cover only host keys (`wibble`, `~/work/looms`, `my-key`, `kind`), and no §7 closed-enum sub-rule covers the synthetic `(root)` label.

## Issue introduction

**Verdict:** single-commit defect — introduced
**Introducing commits:**
- `89eabf1` (2026-06-22) `pi-loom spec-plan: fix F-0744 — define settings non-object-root outcome + array/object observed rendering`

**History:** `git log -G '\(root\)' -- docs/spec_topics/` returns a single commit, `89eabf1`, which is also the only commit touching the producer-side wording in `code-registry-load.md` and `discovery/package-and-settings.md`. That commit added the non-object-root arm, the producer-side "rendered as `(root)`" phrasing in both files, and two new test vectors for the `<observed>` array / object kinds — but did not extend §5's `<key>` rule or its test-vector list to cover the new synthetic label. The `<key>` identifier-shape predicate itself predates this commit (it has been the runtime regex `^[A-Za-z_][A-Za-z0-9_]*$` since `f5e89f4` and the explicit "runtime check" reading was added in `72e2c31`), so the predicate is stable; the defect is the unilateral introduction of a non-conforming key token at the producer site without a coordinating §5 edit.

## Solution approach

Clarify §5's `<key>` rule in `placeholder-rendering-b.md` so the rendering of a loom-emitted synthetic label — the `(root)` token emitted by `loom/load/settings-value-out-of-range`'s non-object-root arm — is specified: bare, not double-quoted, matching the producer-side "rendered as `(root)`" wording already shipped in `code-registry-load.md` (so no producer-side edit is needed). Add a matching entry to §5's *Test vectors* giving the conformance test a normative byte-string to assert. The `<key>` runtime-predicate framing for genuine host keys stays unchanged.

## Solution constraints

- Preserve the parenthesised `(root)` token; do not rename it to bare `root` (the producer site pins the parenthesised form).
- Keep the synthetic-label form closed under GOV-7 / GOV-8; do not widen the rule into a general "labels containing parentheses" carve-out.
