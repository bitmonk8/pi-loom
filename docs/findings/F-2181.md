## Finding

INV-4 states normatively that the per-chain invoke-depth counter "crosses subagent-mode boundaries unchanged" — a `subagent → subagent` or `prompt → subagent` invocation does **not** reset the count, because the spawned `AgentSession` is a continuation of the same call chain even though it owns a fresh conversation. This is a subtle rule: the natural implementation instinct is to treat a freshly spawned `AgentSession` as a new root (depth 0), which would reset the budget on every subagent hop.

The spec corpus embeds normative test vectors / worked examples for comparable subtle runtime rules — e.g. the byte-identical message-rendering vectors in `diagnostics/placeholder-rendering-a.md` and `placeholder-rendering-b.md`, and the binder context-walk worked examples in `binder/binder-model-and-context.md`. INV-4's only embedded example is the message-rendering vector "A 33-deep `invoke` chain renders `invoke chain depth exceeded: 33 > 32`" (`diagnostics/placeholder-rendering-a.md`), which pins the rendered string but says nothing about which frames count toward the 33 or whether a subagent hop resets the budget.

Consequently the rule's primary scenario — a deep cross-mode chain (`prompt → subagent → subagent → …`) that reaches the cap precisely because the counter never resets — is unpinned by any example in the spec. An implementation that resets the counter on each `AgentSession` spawn would permit arbitrarily deep nesting through subagent-mode invocations, silently violating INV-4, with no spec-level conformance demonstration that distinguishes the conformant from the non-conformant behaviour.

**History:** Commit 2b0def6 added the INV-4 cross-mode accounting paragraph to `invocation.md`, introducing the normative non-reset rule ("crosses subagent-mode boundaries unchanged") that resolved the prior under-specification of subagent-boundary counting semantics. The rule entered as prose only, with no accompanying test vector or worked example; the testability gap is therefore co-extant with the rule itself and has persisted unchanged since that commit (`git log -S` on the non-reset wording localises to this single commit, modulo the later `docs/` relocation in 31ff060).

## Solution Space

**Shape:** single

### Recommendation

Add a normative test vector to INV-4 (immediately after the cross-mode non-reset paragraph) demonstrating that a cross-mode chain reaches the cap because the counter does not reset: a chain of `prompt → subagent → subagent → …` invocations totalling 33 countable frames (across any mix of modes) MUST raise `loom/runtime/invoke-depth-exceeded` on the frame that would bring the count to 33, rendering `invoke chain depth exceeded: 33 > 32`, with the panic surfacing to the nearest `invoke` parent as `Err(InvokeInfraError { cause: "panic", … })`.

Pin the discriminating property explicitly: at least one subagent-mode boundary MUST be crossed strictly before the cap is reached, so that an implementation resetting the budget on `AgentSession` spawn would fail to trip the cap (and thus fail the vector). State the depth-0 origin (the slash-invoked top-level loom) and the legal range `1 ≤ depth ≤ 32` so the 33rd frame is the breaching frame, consistent with the existing INV-4 accounting prose. The `V15b` / `V15b-T` leaves already enumerate a cross-subagent-boundary test in their `Tests` blocks; align their wording with the new spec vector so the leaf test explicitly exercises a multi-hop subagent chain rather than a single-hop one.

## Consequence

**Severity:** advisory

The non-reset rule is normatively explicit, so a careful implementer can build a conformant system from the prose alone; the gap is a missing conformance demonstration rather than an ambiguity. Without a worked example pinning a cross-mode chain that reaches the cap, a counter-reset-on-spawn defect can pass every spec-level demonstration and slip into a subagent-recursion depth bypass.
