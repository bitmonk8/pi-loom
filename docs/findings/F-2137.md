## Finding

The "Transient toasts (auxiliary)" paragraph in `diagnostic-shape.md` names the file-watching library `chokidar` three times inside normative behavioural prose: the example list ("the **chokidar** watcher itself throwing"), arm (1) of the three-mode subsystem-state enumeration ("(1) When the **chokidar** watcher itself throws, the file-watching subsystem is left torn down…"), and the cross-reference to the watcher-terminated posture ("the transient toast above stays reserved for the informational **chokidar**-error case"). In every case the binding requirement is the observable post-failure subsystem state — the file-watching subsystem is left torn down rather than re-armed, the `LoomRegistry` stays live and dispatchable through arm (a) of `readDrainState`, no drain-state tag is written, and operator recovery is one `/reload` away. The identity of the watcher library is irrelevant to that contract.

This contradicts the corpus's own architecture. `host-prerequisites.md` ("Loom-package implementation dependencies") explicitly classifies `chokidar` as an "implementation-side choice rather than part of the Pi-supplied contract", and PIC-14 in `host-interfaces-services.md` defines a library-neutral `FileWatcher` / `FakeFileWatcher` DI seam whose member set is declared non-binding (GOV-18 arm (a)) precisely so the watcher implementation is swappable. Anchoring behavioural requirements in `diagnostic-shape.md` to a specific library — when the rest of the spec treats that library as an interchangeable adapter behind a seam — couples a contract page to a packaging decision documented elsewhere as non-normative.

The fix is a confined terminology rename within this one paragraph; the observable state/recovery requirements stay byte-for-byte intact. The deliberate single-declaration sites that legitimately name the library (`host-prerequisites.md`, PIC-14, `implementation-notes.md`) are out of scope.

## Solution Space

**Shape:** single

### Recommendation

Within the "Transient toasts (auxiliary)" paragraph of `diagnostic-shape.md`, replace each `chokidar` reference with a seam-neutral descriptor, leaving all observable state/recovery requirements unchanged:

- "the **chokidar** watcher itself throwing" → "the file watcher itself throwing" (or "the file-watching subsystem throwing").
- "(1) When the **chokidar** watcher itself throws, the file-watching subsystem is left torn down…" → "(1) When the file watcher itself throws, the file-watching subsystem is left torn down…".
- "the informational **chokidar**-error case" → "the informational file-watcher-error case".

Keep the edit confined to this paragraph. Do **not** touch the deliberate single-declaration sites that name the library as the production adapter — `host-prerequisites.md` ("Loom-package implementation dependencies"), PIC-14 in `host-interfaces-services.md`, and `implementation-notes.md` — nor the `FileWatcher`-seam references; those are where the `chokidar` binding is correctly documented and cross-referenced.

## Consequence

**Severity:** cosmetic

A reader wiring a non-`chokidar` watcher behind the `FileWatcher` seam encounters normative behavioural prose anchored to a library the spec elsewhere declares interchangeable, forcing a mental translation to recover the seam-level contract. No runtime observer is affected and no two implementers would diverge on behaviour, since the post-failure state requirements are fully specified independently of the library name.
