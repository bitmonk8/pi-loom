## Problem

`lexical.md` §Encoding requires the runtime to detect non-UTF-8 bytes (non-UTF-8 BOMs, arbitrary invalid UTF-8 including lone surrogates), report the zero-based byte offset of the first invalid byte in the original file content as `loom/load/invalid-encoding`, and fail fast on a UTF-16 save without transcoding — all of which presuppose access to the raw, pre-decode byte sequence. The only spec-pinned filesystem read seam, PIC-13's `FileSystem` in `host-interfaces-services.md`, is text-shaped: its read member is `readText(path: string): Promise<string>`, with no byte-shaped equivalent. With only `readText` in scope, a conforming runtime has no specified mechanism to observe invalid bytes or recover original byte offsets — the underlying decoder has already thrown, substituted, or passed the bytes through before loom code sees the string. The defect is the missing connective tissue between what §Encoding requires loom to report and how PIC-13 lets loom read the filesystem, so conforming implementations diverge on every invalid-encoding case.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `5d053ff` (2026-05-04, "pi-loom spec: resolve \"file encoding, newline normalisation, and path resolution unspecified\""), `8a6d15f` (2026-05-06, "pi-loom spec: resolve \"FileSystem seam interface signature lives outside the spec corpus\""), with `6829a79` (2026-06-06, "pi-loom spec: resolve \"Pi-SDK hidden-assumption cluster (strictCapable absence, decode steps, pi.on origin)\"") tightening the byte-offset wording without closing the seam gap.
**History:**

- `git log --follow -- docs/spec_topics/lexical.md` shows the `**Encoding.**` bullet first added at `5d053ff` (the resolution of "file encoding, newline normalisation, and path resolution unspecified"). That commit introduced the requirement that the runtime detect non-UTF-8 bytes and report a "byte offset of the first invalid byte" — but, at the time, no `FileSystem` seam interface existed in the spec corpus, so the read-shape question simply did not arise.
- `git log -G "readText" -- spec_topics/pi-integration-contract.md docs/spec_topics/pi-integration-contract*` shows the `FileSystem` interface block first emerging at `8a6d15f` (resolution of "FileSystem seam interface signature lives outside the spec corpus"). That commit added `readText(path: string): Promise<string>` as the sole read member and *closed* the seam's member set as normative ("The interface shape is normative"), without re-opening `lexical.md` §Encoding to reconcile the now-incompatible text-vs-bytes shape. The pair `5d053ff` + `8a6d15f` is the moment the spec became internally inconsistent.
- `git log -L /loom\/load\/invalid-encoding/,+1:docs/spec_topics/lexical.md` shows the later refinement at `6829a79`, which tightened "byte offset" to "zero-based byte offset of the first invalid byte in the original (pre-normalisation) file content … populated by the loom-side UTF-8 decode step", and the parallel edit to `diagnostics/code-registry-load.md`. This tightening sharpened the obligation that the runtime own the decode but still did not introduce a bytes-shaped seam member; the contradiction with `8a6d15f` deepened rather than resolving.
- `git log -S readBytes -- docs/plan_topics/` shows the plan absorbing the gap at `fa18160` ("pi-loom plan: resolve \"V1a presupposes a pre-decode byte source that no seam supplies\"") and `0bfc784`, with `V1a` and `V8b` citing a `readBytes` member that the spec corpus never names — a downstream witness that this defect had already been observed and routed around at plan-author time without flowing back to the spec.

The defect is the *interaction* between two independently well-intentioned edits: the lexical rule assumes a byte-oriented read path; the seam pin closes the read path on a string. Neither commit, taken alone, would be a defect; together they leave the spec contradictory.

## Solution approach

Add a byte-shaped read member — the implementation plan already consumes one named `readBytes` returning raw bytes — to the PIC-13 `FileSystem` interface in `host-interfaces-services.md`, carrying the same Node-style `.code` rejection mapping already pinned for `readText`, and add a normative per-member bullet establishing it as the load-bearing surface for `.loom` / `.warp` source reads. Add a forward-link under `lexical.md` §Encoding stating that source files are obtained as raw bytes through that seam and UTF-8-validated by the runtime before decoding. Re-anchor the `loom/load/invalid-encoding` diagnostic row's "loom-side UTF-8 decode step" reference in `diagnostics/code-registry-load.md` to the new bullet.

## Solution constraints

- Preserve `lexical.md` §Encoding's existing observables unchanged: the leading UTF-8 BOM is consumed, a non-UTF-8 BOM reports offset `0`, and the runtime performs no transcoding (UTF-16 / Latin-1 fail fast).
- Out of scope: the `FileSystem.writeText` rejection contract (owned by a sibling PIC-13 finding).
