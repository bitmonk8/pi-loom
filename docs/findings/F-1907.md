## Finding

`H5a` establishes the warn-only live-corpus mode's reporting contract: the gate machinery **returns** an in-process findings collection to its caller — "one structured entry per gap, each tagged with the gap kind it represents". `H5b` consumes that returned collection (it names it "the in-process representation `H5a`'s Adds fixes") and its Tests bullet asserts only that the collection "contains a structured entry tagged with that gap kind".

The four gap kinds are enumerated (unmapped executable REQ-ID; coverage-matrix-mapped numbered REQ-ID with no citing test; multi-leaf coverage-matrix-row facet whose closing leaf carries no facet-naming citing test; un-enumerated un-anchored MUST), but each entry's only specified field is the gap-kind tag. Neither leaf pins the field that identifies *which* offending item the entry refers to — the offending REQ-ID, the offending `spec_topics/**` page, or the offending facet + closing leaf. Surfacing *which* coverage is missing before the `H6a` hard flip is the canary's stated purpose ("surfaces live coverage gaps to contributors before the hard flip can redden `main`"), and a gap-kind tag alone cannot point a contributor at the item to fix. Because `H5b`'s test asserts only the gap-kind tag, a minimal implementation that emits tag-only entries passes both leaves' gates while leaving the canary unable to name the offending item.

## Issue introduction

**Verdict:** introduced-by-single-commit
**Introducing commit:** `7b88876` (`pi-loom spec-plan: fix F-0922 — warn-only canary CI-finding reporting contract`, 2026-06-21).
**History:** `git log --follow` over `H5a-closing-gate-automation.md` and `H5b-warn-only-canary.md` shows the warn-only mode previously "emits each gap as a CI finding without failing the build" with no structured-entry concept at all. Commit `7b88876` authored the reporting contract — switching to a **returned** in-process findings collection of "one structured entry per gap, each tagged with the gap kind" and updating `H5b`'s Adds/Tests/Ships-when to assert a "structured entry tagged with that gap kind". That same commit pinned the gap-kind tag but stopped short of specifying any offending-item identifier field, so the under-specification is original to the contract rather than a later regression that narrowed an earlier, more complete schema. No subsequent commit (`cd27b78`, which only added the `H5f` per-facet surface to the enumeration) touched the entry schema.

## Solution Space

**Shape:** single

### Recommendation

In `H5a`'s Adds — at the clause that fixes the returned findings-collection representation ("one structured entry per gap, each tagged with the gap kind it represents") — extend the entry contract so each entry also carries an identifier of the offending item, keyed to its gap kind:

- unmapped executable REQ-ID, and coverage-matrix-mapped numbered REQ-ID with no citing test → the offending `PREFIX-N` REQ-ID;
- un-enumerated un-anchored MUST → the offending `spec_topics/**` page (with its anchor where one exists);
- multi-leaf coverage-matrix-row facet with no facet-naming citing test → the offending facet's closing-leaf ID together with its facet key.

State that the entry is self-identifying — the gap-kind tag plus the offending-item identifier above — so `H5b` and any auditor consume one shared entry contract and the canary can name which coverage is missing. `H5b` inherits the contract by reference (it already points at "the in-process representation `H5a`'s Adds fixes"); its seeded-gap Tests bullet may additionally assert the offending-item identifier of the seeded gap, but the authoritative schema statement belongs in `H5a`'s Adds.

## Consequence

**Severity:** advisory

The canary can ship returning entries tagged only by gap kind — `npm test` stays green and `H5b`'s gate composes against `H5a`'s contract — yet a contributor reading a finding learns only that a gap of some kind exists, not which REQ-ID / page / facet+leaf is offending, defeating the canary's stated purpose of surfacing which coverage is missing ahead of the `H6a` hard flip. Two implementers would diverge on what (if anything) the entry carries beyond the tag, but nothing is blocked and the gate still fires, so the impact is degraded diagnostic value rather than a broken gate.
