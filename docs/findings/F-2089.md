## Finding

The binder system-prompt builder renders each `params:` field's per-field line with a `<requirement>` token that is either `required` or `default=<literal>` (item 4 of *System-prompt structure (normative)*, `binder/binder-bypass-and-envelope.md`). The *Default-literal rendering* paragraph makes the `<literal>` token normative: it "MUST be the field's default value rendered in the [Loom literal sublanguage] surface syntax — the same notation accepted on the RHS of `params:` defaults." That is a byte-exact obligation — the *Parameter-line reference renderings* table immediately below pins per-field lines as exact byte sequences that conforming implementations MUST reproduce.

The worked renderings supplied cover only `Severity.High` (enum), `"hello"` (string), `[1, 2, 3]` (array), and `[]` (empty array) — all primitives and flat arrays. But the literal sublanguage (`grammar.md#loom-literal-sublanguage`) admits `BareObjectLit` and `NamedObjectLit` as `params:` defaults (a `BareObjectLit` is explicitly "admitted only when an external schema supplies the type — the LHS of a `params:` default supplies it"), so a param may carry an object default such as `Cat { name: "Whiskers", color: "red" }`, a nested object, or an array-of-objects. The literal grammar makes two choices free on input that a *rendering* must canonicalise: field order is explicitly free ("Field order is free; the parser does not require declared order"), and inter-token whitespace is unconstrained. The *Default-literal rendering* paragraph pins neither a canonical field order nor a canonical whitespace form for object and nested-composite defaults.

Consequently the byte sequence for `default=<literal>` is undefined whenever a param's default is an object or a composite containing one. Two conforming implementations could emit `default=Cat { name: "Whiskers", color: "red" }` and `default=Cat {color:"red",name:"Whiskers"}` and each claim conformance, defeating the byte-exact reference-rendering contract for this class of field.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** 5a4ec51 — "pi-loom spec: resolve \"Binder system prompt information content lacks verifiable field set\"" (2026-05-06)
**History:** The *Default-literal rendering* paragraph — including its enumerated worked renderings that cover only `Severity.High`, `"hello"`, `[1, 2, 3]`, and `[]` — was first authored in `5a4ec51` (then in `spec_topics/binder.md`; later relocated verbatim into `binder/binder-bypass-and-envelope.md` by the size-cap split `f5e89f4`, which made no semantic change to this paragraph). The `BareObjectLit` / `NamedObjectLit` literal-sublanguage forms admitted as `params:` defaults already existed at that point — introduced earlier in `f841b12` ("spec: resolve Cluster 2 (literal sublanguage + grammar appendix)", 2026-05-04). The defect is therefore the incomplete enumeration introduced by `5a4ec51`: it pinned a byte-exact rendering MUST whose worked set omits the object/nested-composite literal forms the grammar already admitted.

## Solution Space

**Shape:** multiple

### Option A — Extend *Default-literal rendering* with canonical rules and reference renderings

- **Approach:** Add explicit canonicalisation rules for object and nested-composite literals: render fields in **declared schema field order** (not author/source order, since input order is free), with discriminated-union-variant discriminator fields omitted (they are implicit per `grammar.md`), and pin canonical whitespace matching the existing worked examples (named-object form `Name { key: value, key: value }` with a space after `{` and before `}`, `: ` between key and value, `, ` between fields; arrays keep the existing `[1, 2, 3]` form; recurse the same rules for nested objects/arrays). Add ≥1 reference rendering each for a `BareObjectLit`, a `NamedObjectLit`, and a nested composite to the worked-example set.
- **Spec edits:** Extend the *Default-literal rendering* paragraph in `binder/binder-bypass-and-envelope.md` with the canonical field-order and whitespace rules and the new worked renderings.
- **Pros:** Keeps a single coherent byte-exact rendering contract across every literal form the grammar admits; preserves the per-field-line reference-rendering discipline; gives `V11d-T` a determinable oracle for object defaults.
- **Cons:** Adds normative surface; the canonical field-order rule must be stated carefully against the grammar's "field order is free" input rule.
- **Risks:** Missing a recursive case (array-of-objects, object-in-array) would leave a residual gap; the new rules must align with the `Cat { name: "Whiskers", color: "red" }` shape used elsewhere in the spec.

### Option B — Scope the byte-exact MUST to the enumerated cases; declare the rest non-normative-but-deterministic

- **Approach:** State that `default=<literal>` rendering is normative (byte-exact) only for the enumerated primitive/array forms, and that object/nested-composite defaults are rendered in any valid Loom literal sublanguage surface syntax chosen by the implementation, required only to be internally deterministic.
- **Spec edits:** Amend *Default-literal rendering* to carve out the non-enumerated forms, and reconcile the carve-out with the byte-exact framing of the *Parameter-line reference renderings* table (note that those object/nested forms are exempt from cross-implementation byte equality).
- **Pros:** Minimal normative addition; avoids pinning a canonical field order against a grammar that leaves input order free.
- **Cons:** Two implementations' object-default lines diverge byte-for-byte; weakens the per-field-line contract and any cross-implementation reproducibility expectation for this field class.
- **Risks:** A carve-out here invites the same carve-out for the sibling *Type display* gap (see Related Findings); the two must be reconciled or the spec acquires two inconsistent rendering-normativity policies.

### Recommendation

Take **Option A**. The per-field line is already a byte-exact reference-rendering contract, and the spec admits object literals as `params:` defaults; extending the canonical rules keeps one coherent rendering policy rather than splitting the field set into byte-pinned and implementation-chosen halves. Watch these edge cases: render in declared schema field order (input order is free); omit implicit discriminator fields in variant constructors; recurse the same whitespace/order rules through nested objects and array-of-object defaults; an empty object default cannot occur (an empty object body is a parse error and partial defaults are unsupported), so no `{}` rendering rule is needed.

## Consequence

**Severity:** correctness

A `params:` field with an object or nested-composite default has no canonical `default=<literal>` byte form, so two reasonable implementers diverge (field order, inner whitespace) while each satisfies the stated rules. This breaks the byte-exact per-field-line contract for that field class and leaves the paired conformance test (`V11d-T`) without a determinable oracle.
