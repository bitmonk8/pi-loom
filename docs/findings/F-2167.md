## Finding

PIC-12 bans the runtime from calling `Date.now`, `performance.now`, `Date.prototype.getTime`, or the global `setTimeout` / `clearTimeout` outside the `WallClock` adapter, and makes that ban observable with a conformance criterion: "A conforming implementation MUST carry a build-time check that rejects any **direct lexical reference** to those timing primitives outside the `WallClock` adapter." The criterion then scopes itself with a partition: it "covers direct lexical references only — indirect forms (aliased, destructured, computed, helper-wrapped, or re-exported reads) are out of scope for the spec criterion and are conceded to plan-side review."

`globalThis`-qualified and `global`-qualified access — `globalThis.Date.now()`, `globalThis.setTimeout()`, `global.setTimeout()` — falls into neither side of that partition. It is not a *bare-identifier* lexical reference to the primitive (the access flows through a property read on the global object, so an AST check keyed on the identifier `setTimeout` as a free variable does not match it), and it is not one of the five enumerated indirect forms (aliased, destructured, computed, helper-wrapped, re-exported). It is a statically-visible static member access that the enumeration simply does not name.

The result is two defensible readings of the conformance criterion. A grep-realised check matches `setTimeout` as a substring of `globalThis.setTimeout` and rejects it; an AST-realised check that looks for the bare global identifier does not see a property access on `globalThis` and lets it pass. Both are "conforming" build-time checks under the current prose, yet they disagree on whether a trivially-reachable evasion of the timing-primitive ban (`globalThis.setTimeout`) is a conformance failure. Because the same "direct lexical reference" wording is the canonical criterion that PIC-13 (`process.env.HOME`/`process.cwd()`) and PIC-20 (`crypto.randomUUID`) inherit by reference, the gap is not confined to the timing seam.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** `4556623` (2026-06-25, "pi-loom spec-plan: fix F-1178 — PIC-12 timing-primitive ban gets observable build-time conformance criterion")
**History:** `git log -S "direct lexical reference"` and `git log -S "indirect forms"` against `docs/spec_topics/pi-integration-contract/host-interfaces-services.md` each return exactly one commit, `4556623`. That commit introduced both the "direct lexical reference" criterion and the five-item indirect-form carve-out — i.e. the entire enumeration whose incompleteness this finding reports. `git log -G "globalThis"` against the same file returns nothing: the token has never appeared in the file's history, confirming the qualified-access case was never classified when the partition was authored. Before `4556623` the ban was behavioural-only with no enumeration, so the classification gap could not exist; it entered with the criterion itself in that single commit.

## Solution Space

**Shape:** multiple

### Option A — Fold `globalThis`/`global`-qualified access into the direct-lexical criterion

**Approach:** Amend the PIC-12 criterion sentence so that a static member access of a banned primitive on the global object (`globalThis.<primitive>`, `global.<primitive>`, including the `globalThis.Date.now` / `globalThis.Date.prototype.getTime` chains) is explicitly a *direct lexical reference* the build-time check MUST reject outside the `WallClock` adapter — distinct from the conceded indirect forms.

**Spec edits:**
- In PIC-12, extend the criterion clause to read that the check rejects direct lexical references "including a static member access of a banned primitive on `globalThis` or `global`" outside the adapter.
- Leave the indirect-forms carve-out (aliased, destructured, computed, helper-wrapped, re-exported) unchanged; `globalThis['setTimeout']` (computed) remains out of scope, consistent with the existing partition.
- No edit needed at PIC-13/PIC-20: they inherit the canonical PIC-12 criterion, so the clarification propagates by reference.

**Pros:** Closes the trivially-reachable evasion; `globalThis.setTimeout` is statically obvious and cheap to detect in either grep or AST realisations; keeps the behavioural MUST NOT and its conformance check in agreement (the MUST NOT already bans "the global `setTimeout`," which `globalThis.setTimeout` is).
**Cons:** Slightly widens what every conforming check must detect; an AST-only implementer must add a member-access rule alongside the free-identifier rule.
**Risks:** Must scope precisely to static-member access on `globalThis`/`global` to avoid re-absorbing the genuinely-indirect computed form the spec intends to concede.

### Option B — Classify it as an out-of-scope indirect form, conceded to plan-side review

**Approach:** Add `globalThis`/`global`-qualified access to the enumerated indirect-forms list, explicitly placing it out of scope for the spec criterion and conceding it (like the other indirect reads) to the `conventions.md` item-2 reviewer inspection.

**Spec edits:**
- In PIC-12, extend the indirect-forms parenthetical to "(aliased, destructured, computed, helper-wrapped, re-exported, or `globalThis`/`global`-qualified reads)".
- Confirm `conventions.md` item 2's manual-inspection scope already covers it (it enumerates alias/destructure/computed/helper/re-export reads; add the qualified-access case if needed — that is a plan-side edit, out of scope here).

**Pros:** Preserves the existing "direct lexical references only" criterion verbatim; no added obligation on the mechanical check.
**Cons:** Concedes the cheapest, most static evasion to manual review even though it is far easier to detect mechanically than the truly-indirect forms; leaves a static, grep-visible ban-bypass outside the automated gate.
**Risks:** Manual review reliably misses exactly this kind of low-salience qualified access; the behavioural MUST NOT and the conformance gate stay out of step.

### Recommendation

Adopt **Option A**. `globalThis.setTimeout` *is* the global `setTimeout` the MUST NOT already forbids, and it is statically detectable in both grep and AST realisations, so conceding it to manual review (Option B) would leave a trivial automated-gate bypass for the most reachable evasion. Pin the criterion to reject static-member access on `globalThis`/`global`; keep the computed form (`globalThis['setTimeout']`) in the conceded indirect set so the direct/indirect boundary stays decidable. Implementers must watch the `globalThis.Date.now()` / `globalThis.Date.prototype.getTime()` chains, where the banned primitive is the rightmost segment of a qualified member chain.

## Consequence

**Severity:** correctness

Two reasonable implementers building the mandated conformance check diverge: a substring/grep check rejects `globalThis.setTimeout`, an identifier-AST check accepts it. Under the second reading a runtime can reach the global timer surface through `globalThis.`/`global.` qualification and still pass the conformance gate, silently defeating the seam-injection discipline PIC-12 exists to enforce. The divergence is confined to the build-time check's accept/reject surface, but it propagates to the PIC-13 and PIC-20 bans that cite the PIC-12 criterion as canonical.
