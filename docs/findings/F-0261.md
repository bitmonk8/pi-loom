## Finding

The `V7a` (and mirrored `V7a-T`) Multi-error test bullet states that the
emitted `Diagnostic[]` "is ordered by `(file, line, col)` across an entry
`.loom` and ≥2 transitively-imported `.warp` modules (per
[implementation-notes.md — Static-resolution load pass] IMPL area, which
aggregates each visited file's diagnostics into the entry loom's drain in this
order)." The trailing pronoun "in this order" has two readable antecedents:
the global lexicographic `(file, line, col)` sort named in the main clause, or
the traversal order in which the load pass visits each file ("aggregates each
visited file's diagnostics … in this order"). Those two readings coincide only
when the transitive-import traversal order happens to equal the lexicographic
file-path order; they diverge otherwise.

The spec resolves the ambiguity unambiguously — `implementation-notes.md`
§Runtime *Static-resolution load pass* says the pass "aggregates each visited
file's diagnostics into the entry loom's drain (sorted by `(file, line, col)`
per [Diagnostics — Multi-error reporting])", i.e. a single global sort. The
plan sentence, however, still permits the traversal-bucket reading. The
practical risk is in test-fixture construction: a fixture whose import
traversal order coincides with path order would let the ordering assertion pass
under both a correct global sort and an incorrect per-visited-file bucket
concatenation, so the gate cannot distinguish the two implementations.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 249cec5 — pi-loom plan: resolve "implementation-notes.md (IMPL) unmapped in coverage matrix" (2026-06-11, Thomas Andersen)
**History:** Before 249cec5 the Multi-error bullet in `V7a` and `V7a-T` carried no ordering clause at all. That commit, while mapping the `implementation-notes.md` (IMPL) §Static-resolution load pass area into the coverage matrix, added the `(file, line, col)` ordering sentence to both leaves with the trailing "in this order" parenthetical; the ambiguous pronoun entered with that single edit.

## Solution Space

**Shape:** single

### Recommendation

In the `Tests.` "Multi-error" bullet of both
`docs/plan_topics/V7a-diagnostics-primitive.md` and
`docs/plan_topics/V7a-T-diagnostics-primitive.md`, remove the ambiguous
trailing pronoun and state the order directly. Replace the parenthetical
clause `which aggregates each visited file's diagnostics into the entry loom's
drain in this order` so the drain is described as a single global sort on
`(file-path-lexicographic, line, col)` across all visited files — matching the
`sorted by (file, line, col)` wording the cited `implementation-notes.md`
§Static-resolution load pass already uses. The `(file, line, col)` reference to
`implementation-notes.md` remains read-only; this fix is confined to the plan
leaves.

In the same bullet, add the requirement that the multi-error fixture's
transitive-import traversal order differ from the lexicographic file-path
order, so the ordering assertion distinguishes a global sort from
per-visited-file bucket concatenation. Edge case for the implementer: the
fixture needs at least one `.warp` module whose path sorts before a module
visited earlier in the traversal, so the discriminating reorder is observable.

## Consequence

**Severity:** advisory

A test author can satisfy the ordering bullet with a fixture whose import
traversal order equals lexicographic path order, in which case the multi-error
ordering assertion passes regardless of whether the implementation globally
sorts the drain or merely concatenates per-visited-file buckets — the gate
fires without exercising the discriminating case. Implementation behaviour
itself is not at risk because the spec pins the global sort; the weakness is a
non-discriminating acceptance test.
