## Finding

The **Conversation drive — subagent mode** paragraph in `docs/spec_topics/pi-integration-contract/subagent.md` fully authors the *success* arm of a subagent untyped query: it pins that the runtime awaits `session.subscribe(event => { if (event.type === "agent_end") … })`, ignores `willRetry: true` events, reads `messages: AgentMessage[]` from the terminal `agent_end` payload, and assembles the `Ok(string)` value by the same final-assistant-text rule the prompt-mode driver uses. The companion *failure* arm is missing entirely — neither a detection surface nor a `QueryError` mapping is named anywhere in the file for a subagent untyped query whose driven turn fails for transport / provider reasons.

The prompt-mode counterpart in `conversation-drive.md` is explicit on both halves: a synchronous throw at the `pi.sendUserMessage` call site maps to `Err(QueryError { kind: "transport", … })`, and the **primary** transport-failure surface is the post-`waitForIdle()` probe of the driven turn's trailing `assistant` message (`stopReason: "error"` → `kind: "transport"`, with the `errorMessage` / `"provider transport failure"` fallback). The subagent paragraph has none of this. Subagent mode differs from prompt mode on every relevant axis: `AgentSession.sendUserMessage` returns `Promise<void>` (so a *rejection* — not just a synchronous throw — is observable at the call site), the driver awaits a session-local `subscribe` rather than `ctx.waitForIdle()`, and `ctx.sessionManager` is the parent's user-session manager rather than the spawned in-memory one, so the prompt-mode probe formula (`buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId())`) does not transfer verbatim. The page therefore cannot just be read as "same rule as prompt mode by implication": the symmetric clause is normative content the spec is missing.

Two implementers reading the section in isolation would disagree on observable behaviour: whether to attach a `.catch` to the awaited `session.sendUserMessage(text)`, whether to inspect the terminal `agent_end` event's trailing assistant message for `stopReason: "error"` before invoking the success-side extraction, and which `QueryError` kind (`"transport"`, `"internal_error"`, or none — silently producing an `Ok("")`) a failed subagent driven turn surfaces as.

## Issue introduction

**Verdict:** multi-commit interaction (regression by omission)
**Introducing commits:**
- `024215d` (2026-06-01) `pi-loom spec: resolve "untyped-query Ok(string) extraction mechanism (prompt + subagent)"` — added the success-arm extraction symmetrically on both sides.
- `d37c563` (2026-06-05) `pi-loom spec: resolve "Prompt-mode errorMessage probe unreachable from specified surface"` — replaced the prompt-mode `AgentSession.errorMessage` probe with the trailing-assistant-message `stopReason` / `errorMessage` probe and anchored it as `#prompt-mode-error-detection`. The same edit deliberately scoped the subagent forced-respond-turn discussion (`complete()` lives off-session), but did **not** restate the new probe symmetrically on the subagent untyped arm.

**History:** Before `024215d` the success arm itself was unpinned on both sides — the symmetry defect did not exist because nothing was pinned. `024215d` made the success arm symmetric but did not touch the failure arm (it was authored as a focused success-extraction fix, mirroring T23/T24 of the then-current spec review, which scoped only the success path). `d37c563` then meaningfully rewrote the prompt-mode failure-detection surface (the old `AgentSession.errorMessage` probe was unreachable through `ExtensionContext`; the new probe reads the trailing assistant message via `ctx.sessionManager`). That rewrite extended the prompt-side authority of the failure arm but did not propagate the symmetric clause back into `subagent.md` — partly because the new prompt-mode formula does not transfer verbatim (the subagent driver awaits a session-local `subscribe`, not `waitForIdle()`, and the failed turn appears in the terminal `agent_end` event's `messages` array on the spawned `AgentSession`, not on `ctx.sessionManager`). The defect is therefore the interaction of the success-arm symmetry committed by `024215d` with the prompt-mode failure-arm rewrite committed by `d37c563`: each commit was locally coherent, but together they left the subagent failure arm visibly absent next to a fully-authored subagent success arm and a fully-authored prompt-mode failure arm.

## Solution Space

**Shape:** multiple

### Option A — Add a symmetric failure-arm clause in `subagent.md` (recommended)

**Approach.** Extend the `#subagent-untyped-query-ok-extraction` paragraph (or insert a new `<a id="subagent-untyped-query-error-detection"></a>` clause immediately before it) with the symmetric failure-detection rule, structured to mirror `#prompt-mode-error-detection`.

**Spec edits.**
- In `docs/spec_topics/pi-integration-contract/subagent.md`, **Conversation drive — subagent mode**:
  - Before the existing success-arm clause, add a *Failure detection.* clause stating:
    1. **Cancellation short-circuit (highest precedence).** If `loomAbort.signal.aborted` is true at the moment the awaited `session.sendUserMessage(text)` settles or the terminal `agent_end` arrives, the runtime synthesises `Err(QueryError { kind: "cancelled" })` per [Cancellation](../cancellation.md), with the same precedence the prompt-mode rule pins, and bypasses the two probes below.
    2. **Awaited-promise rejection (secondary).** A rejection of the awaited `session.sendUserMessage(text)` `Promise<void>` (unlike `pi.sendUserMessage`'s `void` shape) is observable at the call site and maps to `Err(QueryError { kind: "transport", message: <reason.message>, http_status: null, provider: <resolved-model-provider>, retryable: false })`, not to `loom/runtime/internal-error` — exactly the disposition the prompt-mode synchronous-throw branch pins, lifted to the rejection channel.
    3. **Terminal-event probe (primary).** After the terminal `agent_end` event (`willRetry: false`) is observed and before the success-arm extraction runs, the runtime reads the trailing `assistant` message from that event's `messages: AgentMessage[]` array (the same array the success arm reads, applying the chronological-ordering presupposition pinned at [`AgentMessage[]` chronological-ordering presupposition](./host-interfaces-core.md#messages-chronological-order-presupposition) — surface (2)) and inspects its `stopReason` and `errorMessage` fields. `stopReason: "error"` maps to `Err(QueryError { kind: "transport", message: <errorMessage>, http_status: null, provider: <resolved-model-provider>, retryable: false })`; an absent `errorMessage` yields the fixed string `"provider transport failure"`. Explicitly state that the "same failure-mapping rule on both sides" invariant holds, paralleling the existing "same final-assistant-text rule on both sides" claim on the success arm.
  - In the *Failure detection* clause, cross-reference the prompt-mode `#prompt-mode-error-detection` anchor so the symmetry is auditable from either side.
  - Add a forward-link from `host-interfaces-core.md` §`AgentSession` member surface to this new failure-detection anchor (the `sendUserMessage(text): Promise<void>` row already calls out the `Promise<void>` shape vs `pi.sendUserMessage`'s `void`; the failure mapping is the consumer of that distinction).
- In `docs/spec_topics/pi-integration-contract/version-bump-step2.md`, audit whether the new "terminal `agent_end` `messages` array carries the failed driven turn's `assistant` message with `stopReason: "error"` populated" presupposition warrants a SHOULD-level bump-checklist item paralleling item (h) (ordering) and item (n) (`AgentSession` member surface). If yes, add it; if it is already implicitly covered by item (h) plus the `AssistantMessage.stopReason` surface item (n) carries, state that explicitly in a one-line note.

**Pros.**
- Restores the success/failure-arm symmetry the spec already advertises ("same final-assistant-text rule on both sides").
- Keeps the failure-detection rule co-located with the success-extraction rule on the subagent page, so a reader of the subagent paragraph finds both halves without cross-page navigation.
- Each subagent failure detection surface (rejection vs terminal-event probe) maps cleanly onto an analogous prompt-mode surface, so the two sides stay auditable in lock-step.

**Cons.**
- Adds material to an already-dense paragraph (the subagent-mode rules block is the largest single span on the page); needs a fresh anchor and careful interleaving with the existing success-arm clause to avoid making the paragraph unscannable.

**Risks.**
- The `Promise<void>` rejection channel on `AgentSession.sendUserMessage` is currently unaudited as a behavioural surface (no bump-checklist item covers it). The new clause introduces an implicit presupposition that a rejection here is the only signal of a synchronous-throw-class failure; either the new clause must call that out (and item (n) extend to cover it), or step 2 above must add a new audit item.

### Option B — Delegate the subagent failure arm to a forward-link, leaving the rule defined once on the prompt-mode page

**Approach.** Add a one-sentence delegation in `subagent.md`'s success-arm paragraph stating that the subagent untyped-query failure-detection rule is "exactly the [prompt-mode rule](./conversation-drive.md#prompt-mode-error-detection), lifted to the session-local `subscribe` / `Promise<void>` channels described above," without restating the mapping.

**Spec edits.**
- Insert one sentence at the head of the success-arm paragraph (before the `Ok(string)` text) and a matching back-link on the prompt-mode page noting the subagent arm consumes the same rule.

**Pros.**
- Minimal text; avoids paragraph bloat.
- Single source of truth for the failure mapping — a future revision of the `kind: "transport"` shape lands in one place.

**Cons.**
- The prompt-mode rule does not transfer verbatim: the call surfaces (`session.sendUserMessage(text): Promise<void>` vs `pi.sendUserMessage(text): void`) and the completion surfaces (terminal `agent_end` event payload vs `ctx.sessionManager` after `waitForIdle()`) differ on every relevant detail. "Same rule, mechanically transposed" is a non-trivial transposition the spec leaves to the reader.
- Breaks the local "same final-assistant-text rule on both sides" pattern the success arm already establishes by inlining both halves on both pages.

**Risks.**
- A future prompt-mode-only edit (e.g. extending the `stopReason` probe to a new variant) silently bypasses the subagent arm.

### Recommendation

Adopt **Option A**. The two arms differ on enough surfaces (`Promise<void>` rejection channel, terminal-event `messages` payload as the probe substrate) that a verbatim transposition rule is more error-prone than a symmetric inline restatement, and the existing success-arm wording on the subagent page already commits the spec to the inline-symmetry pattern. The implementer should ensure the cancellation short-circuit explicitly takes precedence over both the rejection branch and the terminal-event probe (as the prompt-mode rule does), so an Esc fired while the subagent driven turn is mid-flight does not get misclassified as `kind: "transport"`. Edge cases the implementer must watch: a terminal `agent_end` event whose final assistant message has `stopReason: "error"` *and* an empty `messages` array (yields `Ok("")` under the success arm without this fix — the failure-arm clause MUST run *before* the success-arm extraction), and the `willRetry: true` interaction (a retried turn that ultimately fails must surface only the terminal-event failure, not a per-retry one, on the same footing as the existing success-arm `willRetry` filter).

## Consequence

**Severity:** correctness

A failed subagent driven turn currently has no spec-defined disposition. A conformant implementer who reads only the success arm will either return `Ok("")` (the empty-final-turn rule applied to an error turn whose `assistant` array is empty or absent), throw an uncaught rejection out of the awaited `session.sendUserMessage`, or invent an ad-hoc `QueryError` kind — three observably different behaviours for the same provider failure, none anchored in the spec. The defect also breaks the explicit "same final-assistant-text rule on both sides" invariant the success arm advertises: only half of the prompt↔subagent symmetry is actually pinned.
