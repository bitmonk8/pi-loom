## Finding

`V2a` (`docs/plan_topics/V2a-type-grammar.md`) closes three distinct grammar surfaces in one leaf. Its `Adds.` field enumerates (1) the **type-expression parser** (primitive / named / generic `array`/`Result` / inline-object / union / literal types, return-only `void`), (2) the **loom literal sublanguage** (primitive / named-value / array / object literals with the full-field requirement and the is-literal check), and (3) `array<T>` **type-sink resolution** over the sink set. The paired test leaf `V2a-T` mirrors this with three independently-firing test groups: `loom/parse/{generic-arity-mismatch,void-in-non-return-position,result-in-schema-position}` for type grammar, `loom/parse/{default-not-literal,tool-arg-not-literal,missing-object-field}` for the literal sublanguage, and `loom/parse/array-no-common-type` for sink resolution.

These surfaces occupy different grammar positions — type annotations versus value literals — and their test groups red out in isolation, so they meet the project's own bar for separate leaves: `conventions.md` §Vertical slices defines a leaf as "the smallest feature that can ship and be tested independently." Type-expression parsing ships and tests without any literal-sublanguage code; the literal sublanguage (and the `array<T>` sink resolution that resolves `[]` literals against a surrounding type sink) ships against the type grammar as a dependency. Bundling them forces a single implementer to hold two grammar fragments at once and produces one large diff in place of two scope-bounded ones.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e
**History:** `docs/plan_topics/V2a-type-grammar.md` was authored in the single plan-build commit `c6a664e` (2026-06-10) with all three concerns already bundled in its `Adds.` field. `git log --follow` shows no subsequent commit touching the file, so the leaf has carried the bundled scope unchanged since inception; no later edit narrowed, split, or re-partitioned it.

## Solution Space

**Shape:** multiple

This finding carries two obligations that cannot be discharged by a single leaf edit: the **partition** of the `V2a`/`V2a-T` pair, and the **propagation** of the resulting ID change through every document that references `V2a`. Both are required; the order below matters.

### Option A — Partition the `V2a` / `V2a-T` pair

**Approach.** Keep `V2a` (and `V2a-T`) scoped to the type-expression parser, and move the loom literal sublanguage together with `array<T>` type-sink resolution into a new implementation + tests leaf pair. Sink resolution travels with the literal sublanguage because `loom/parse/array-no-common-type` resolves an `[]` literal against its surrounding type sink — it is a property of the literal, not of the type grammar.

**Plan edits.**
- In `V2a-type-grammar.md` / `V2a-T-type-grammar.md`, retain the type-grammar tests `loom/parse/generic-arity-mismatch`, `loom/parse/void-in-non-return-position`, `loom/parse/result-in-schema-position`, and reduce the `Adds.` field to the type-expression parser only.
- Create the new leaf pair (use the literal placeholder `<new>` / `<new>-T` until the implementer picks real IDs) carrying the literal-sublanguage tests `loom/parse/default-not-literal`, `loom/parse/tool-arg-not-literal`, `loom/parse/missing-object-field` and the sink test `loom/parse/array-no-common-type`. Its `Deps.` lists `V2a` (the new leaf parses literals against the type grammar) and its own `-T`.
- Add the new implementation leaf's link to `docs/plan.md` §V2 TOC, after `V2a`.

**Spec edits.** None (`grammar.md` / `type-system.md` are read-only; the partition boundary follows their existing type-vs-literal split).

**Pros.** Each leaf becomes independently testable; the literal-sublanguage red-green cycle is no longer coupled to type-grammar work.

**Cons.** Adds a leaf pair to the plan and one TOC entry.

**Risks.** The new leaf's `Deps.` must name `V2a` or its literal tests will compile against an absent type grammar.

### Option B — Retarget downstream references to the new partition

**Approach.** Repoint every reference to the old single `V2a` scope onto whichever half it actually needs, once Option A has fixed the new IDs.

**Plan edits.**
- `coverage-matrix.md`: the `lexical.md (LEX), grammar.md (GRAM)` code-keyed row currently lists `V1a, V1b, V2a`; add the new literal-sublanguage leaf ID so the GRAM area's literal/sink codes remain closed.
- Retarget the `Deps. … V2a` edge in `V2b`, `V2b-T`, `V2c`, `V2d`, and `V5a` to the half each consumer needs (the type-expression parser, the literal/sink leaf, or both). Each consumer's required half is decided by reading what its tests assert against; do not blanket-point all five at both halves without checking.

**Spec edits.** None.

**Pros.** Keeps the dependency DAG and the coverage gate accurate after the split.

**Cons.** Touches five downstream leaf files and the coverage matrix.

**Risks.** A missed `Deps.` edge leaves a consumer pointing at the wrong or a defunct scope; an un-updated coverage row would leave the GRAM literal/sink codes unclosed.

### Recommendation

Resolve **Option A first**, then **Option B**. The partition is the scope-bounding obligation: it establishes the new leaf IDs and decides which concern lands where, and Option B's reference retargeting can only be done correctly against that settled baseline. Watch the consumer-half assignment in Option B — `V2b` (literal `L ⊑ T` per its `TYPE-3` test) plausibly needs both halves, whereas `V2d` (number rendering) needs only the type grammar; verify each against the consumer's own tests rather than retargeting uniformly.

## Consequence

**Severity:** advisory

Shipping `V2a` unsplit does not block pickup or break a coverage gate, but it violates the plan's own smallest-independently-shippable-leaf rule: one implementer carries two unrelated grammar fragments and produces a single large diff with a coupled red-green cycle, raising review burden and the chance the literal-sublanguage half is under-attended behind the type-grammar work.
