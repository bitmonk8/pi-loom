## Problem

`docs/spec_topics/schema-subset.md` § Lowering Algorithm step 3's *Union of primitives only* bullet leaves the lowered form under-pinned ("`{ "type": [...] }` form preferred for readability; falls back to `anyOf`"), and the top-of-file *Nullability* bullet is the same shape (`string | null` lowers to `{"type": ["string", "null"]}` **or** an `anyOf` with `{"type": "null"}`). Two conformant implementations can therefore emit different lowered fragments for the same source. That collides with the *Canonical schema hash* section in the same file, whose slug is a content-address over the **bytes** of the lowered fragment — the four slug-keyed synthesised-name forms and the three byte-equality collision checks all assume identical source lowers to identical bytes. A conformance suite cannot snapshot the lowered fragment, the slug, or the synthesised name for any primitive union or nullable primitive until the form is fixed normatively.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:**

- `fae85d3` (2026-05-04, *Tighten spec for implementation-plan readiness*) — added the *Lowering Algorithm* section to `spec.md` with the "form preferred for readability; falls back to `anyOf`" wording already in place at step 3's primitive-union bullet; the earlier sentence in the same commit's Nullability bullet (`{"type": ["string", "null"]}` or an `anyOf` with `{"type": "null"}`) carried the same permissiveness.

**History:** `git log --all -S 'preferred for readability' -- spec.md docs/spec_topics/schema-subset.md` returns three commits: `fae85d3` (origin), `fecb504` (the topic split that moved the bullet from `spec.md` to `spec_topics/schema-subset.md`), and `81bf52a` (the `spec/` → `spec_topics/` collapse). Neither subsequent commit modified the bullet's substance. `git log --follow` on the file's later edits (e.g. `7321f01` *Lowered JSON array element order is unpinned*, `e1f5c2a` *Result in lowered-schema positions*, `2128e5f` *Schema-slug collision posture*, `111e533` *Canonical hash recipe cites keywords the lowered subset never emits*) tightened the canonical-hash recipe and its byte-identity dependants without revisiting the primitive-union bullet, so the conflict widened over time as the hash contract grew teeth around an already-permissive bullet.

## Solution approach

Rewrite the step-3 *Union of primitives only* bullet to a single normative MUST that fixes the lowered form by the arms' kinds: the multi-type-array form when every arm is a primitive (treating `null` as a primitive, so the `string | null` case takes this form), and `anyOf` when any arm is non-primitive. Rewrite the top-of-file *Nullability* bullet to drop the `or anyOf` alternative so it names one fixed shape. Add reference vectors beneath the bullet covering the two-arm primitive case, the `T | null` case, and a mixed union, so the canonical-hash recipe is testable against a fixed rendering. The arm-order rule in step 3's *Array element order* clause already pins ordering (source order, `"null"` last when admitted) and composes unchanged.

## Solution constraints

- Out of scope: the *Canonical schema hash* recipe section — pin only the lowered primitive-union/nullability form that section hashes; the recipe's numeric-serialisation rule is owned by the sibling canonical-hash finding.
