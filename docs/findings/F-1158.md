## Finding

`discovery/package-and-settings.md` §"Package discovery" root #4 says: "When `npmCommand` is configured (per `packages.md` §npm), Pi resolves the global root and the extension reads that already-resolved root instead of the literal path; the extension does not itself invoke `npmCommand`." The sentence asserts that the extension *reads* a Pi-resolved root, but it never names the surface through which that read happens — no `ExtensionContext` member, no PIC seam, no host-interface method, no field on any object loom touches. The contract is therefore unimplementable as stated.

Inspection of `@earendil-works/pi-coding-agent` at the loom 1.0 Pi-SDK pin confirms there is no exposed surface that satisfies the description. `ExtensionContext` (enumerated in `host-interfaces-core.md`) has no field that carries package-root metadata; the `PackageManager` interface and its `DefaultPackageManager` implementation are exported at the top level of the SDK but are not handed to extensions through `ExtensionContext`, and `DefaultPackageManager`'s `globalNpmRoot` / `getGlobalNpmRoot` are private members — the only public per-source surface is `getInstalledPath(source, scope)`, which resolves a *named* package, not the enumeration root the discovery walk needs. The literal-path branch (when `npmCommand` is unset) is left only implicit: the sentence covers the configured case but never states that the default reads `~/.pi/agent/npm/` verbatim, so the same paragraph that creates the seam gap also leaves the fallback condition unstated.

The downstream effect is that root #4 of the five-source priority list has no defined enumeration mechanism whenever a user has configured `npmCommand` — an officially-supported Pi configuration (`packages.md` §npm). Two reasonable implementers will diverge: one will reach into `DefaultPackageManager`'s private state (violating the "extension does not itself invoke `npmCommand`" rule's spirit while satisfying its letter), another will quietly ignore the `npmCommand`-resolved branch and always scan `~/.pi/agent/npm/`, missing packages the user actually installed under e.g. `mise`-managed npm.

## Solution Space

**Shape:** single

### Recommendation

Make the global npm root at root #4 a fixed literal path, dropping the `npmCommand`-resolved branch entirely.

In `package-and-settings.md`, replace root #4 with: "`~/.pi/agent/npm/` — global npm packages installed via `pi install -g`. The extension scans this literal path; `npmCommand` (per `packages.md` §npm) only affects how Pi performs installs and does not change the discovery root."

Add a one-line note to **Edge cases** stating that loom-shipping packages installed under a wrapper-managed npm prefix (e.g. `mise`/`asdf`-wrapped npm) are not discovered at root #4 and SHOULD be added via `loomPaths` if the user wants them in scope. This keeps the recovery path explicit: a `loomPaths` entry that points into a wrapper-managed prefix remains in scope under root *Settings*, so affected operators have a documented, non-silent way to bring those packages into discovery.

This makes root #4 symmetric with the literal-path treatment of `~/.pi/agent/git/` at root #5, introduces no new SDK surface or test fake, and adds no presupposition on Pi internals. The only behavioural cost — a `pi install -g` performed under a wrapper produces a package that is not auto-discovered at root #4 — is bounded by the documented `loomPaths` recovery path and is small enough to live in **Edge cases**.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on a supported Pi configuration. Under `npmCommand` (`mise`/`asdf`-wrapped npm), one implementation scans the wrapper-resolved root and discovers loom-shipping packages installed there; the other scans only `~/.pi/agent/npm/` and silently fails to discover them. Users who installed loom-shipping packages via `pi install -g` under such a wrapper get an empty discovery set with no diagnostic.
