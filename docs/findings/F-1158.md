## Problem

`discovery/package-and-settings.md` §"Package discovery" root #4 says that when `npmCommand` is configured (per `packages.md` §npm), Pi resolves the global npm root and the extension reads that already-resolved root rather than invoking `npmCommand` itself. The sentence never names the surface through which the extension performs that read — no `ExtensionContext` member, no PIC seam, no host-interface method — so the contract is unimplementable as stated. At the loom 1.0 Pi-SDK pin no such surface exists: `ExtensionContext` carries no package-root metadata, and `DefaultPackageManager`'s global-root members are private, leaving only `getInstalledPath(source, scope)` which resolves a named package rather than the enumeration root the walk needs. The same paragraph also leaves the literal-path fallback (when `npmCommand` is unset) unstated.

## Solution approach

Rewrite root #4 of the §"Package discovery" priority list to drop the `npmCommand`-resolved branch and make `~/.pi/agent/npm/` the fixed literal discovery root, leaving the relationship to `npmCommand` (per `packages.md` §npm) as governing installs only, not the discovery root — symmetric with root #5's literal treatment of `~/.pi/agent/git/`. Add a note to the §"Package discovery" **Edge cases** list recording that packages installed under a wrapper-managed npm prefix (e.g. `mise`/`asdf`-wrapped npm) are not auto-discovered at root #4 and can be brought into scope via a `loomPaths` entry.
