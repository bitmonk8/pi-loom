## Finding

`discovery/package-and-settings.md` §"Package discovery" root #4 says: "When `npmCommand` is configured (per `packages.md` §npm), Pi resolves the global root and the extension reads that already-resolved root instead of the literal path; the extension does not itself invoke `npmCommand`." The sentence asserts that the extension *reads* a Pi-resolved root, but it never names the surface through which that read happens — no `ExtensionContext` member, no PIC seam, no host-interface method, no field on any object loom touches. The contract is therefore unimplementable as stated.

Inspection of `@earendil-works/pi-coding-agent` at the loom 1.0 Pi-SDK pin confirms there is no exposed surface that satisfies the description. `ExtensionContext` (enumerated in `host-interfaces-core.md`) has no field that carries package-root metadata; the `PackageManager` interface and its `DefaultPackageManager` implementation are exported at the top level of the SDK but are not handed to extensions through `ExtensionContext`, and `DefaultPackageManager`'s `globalNpmRoot` / `getGlobalNpmRoot` are private members — the only public per-source surface is `getInstalledPath(source, scope)`, which resolves a *named* package, not the enumeration root the discovery walk needs. The literal-path branch (when `npmCommand` is unset) is left only implicit: the sentence covers the configured case but never states that the default reads `~/.pi/agent/npm/` verbatim, so the same paragraph that creates the seam gap also leaves the fallback condition unstated.

The downstream effect is that root #4 of the five-source priority list has no defined enumeration mechanism whenever a user has configured `npmCommand` — an officially-supported Pi configuration (`packages.md` §npm). Two reasonable implementers will diverge: one will reach into `DefaultPackageManager`'s private state (violating the "extension does not itself invoke `npmCommand`" rule's spirit while satisfying its letter), another will quietly ignore the `npmCommand`-resolved branch and always scan `~/.pi/agent/npm/`, missing packages the user actually installed under e.g. `mise`-managed npm.

## Solution Space

**Shape:** multiple

### Option A — Declare the literal path is always used

**Approach.** Drop the `npmCommand`-resolved branch from root #4 entirely. State that the extension always scans `~/.pi/agent/npm/` verbatim and that loom-shipping packages installed via an `npmCommand`-wrapped global install are out of scope of root #4 (they remain reachable via roots 1–3 or via `loomPaths` if the user wants them discovered).

**Spec edits.**

- Replace root #4 in `package-and-settings.md` with: "`~/.pi/agent/npm/` — global npm packages installed via `pi install -g`. The extension scans this literal path; `npmCommand` (per `packages.md` §npm) only affects how Pi performs installs and does not change the discovery root."
- Add a one-line note in **Edge cases** stating that loom-shipping packages installed under a wrapper-managed npm prefix are not discovered at root #4 and SHOULD be added via `loomPaths` if the user wants them in scope.

**Pros.** No new SDK surface; no new presupposition; symmetric with the literal-path treatment of `~/.pi/agent/git/` (root #5); no test fake required.

**Cons.** Surprises users of `mise`/`asdf`-wrapped npm: a `pi install -g` under that wrapper produces a package that loom never discovers automatically.

**Risks.** Documentation surface: the limitation must be visible enough that wrapper-using operators encounter it before silent under-discovery does.

### Option B — Pin a Pi-supplied reader via a new presupposition + version-bump audit

**Approach.** Treat the existence of a Pi-supplied way to read the resolved global npm root as a host **presupposition** (in the sense of `host-prerequisites.md` §"Host prerequisites for the degraded-state branch"), record the *evidentiary site* the contributor must re-confirm on each Pi bump, and name in `package-and-settings.md` root #4 the literal-stable wrapper the extension wraps `DefaultPackageManager` around (or an equivalent named adapter), routed through the constructor-injection skeleton.

**Spec edits.**

- In `package-and-settings.md` root #4, name the adapter the runtime constructs and state: "the extension obtains the resolved root by constructing the SDK-exposed `DefaultPackageManager` against the active `npmCommand` setting and reading its resolved global-npm-root via [adapter name]; the adapter is wired through the DI skeleton and has a `FakeNpmRootResolver` test seam."
- Add a `PIC-N` entry in `host-interfaces-services.md` (DI seam) for the resolver, with its behavioural contract (input: `npmCommand` array or `undefined`; output: an absolute directory path; failure modes for spawn-failure / unparseable output map to a `loom/load/*` diagnostic).
- Add a presupposition to `host-prerequisites.md` stating the runtime's reliance on Pi's `DefaultPackageManager` (or its successor) continuing to expose enough surface to back the adapter, with a version-bump checklist item to re-confirm.
- Add a coverage entry in `V10b` for the resolver path.

**Pros.** Preserves the user-observable behaviour the current sentence reaches for (discovery follows `npmCommand`); maps the resolution onto the existing DI-seam discipline so it is conformance-testable.

**Cons.** Introduces a new presupposition the SDK does not pin as a typed contract; the only public Pi surface that would back the adapter is `DefaultPackageManager`'s constructor — `getGlobalNpmRoot` is private — so the adapter must either re-implement `npm root -g` invocation (regressing the "extension does not itself invoke `npmCommand`" rule) or rely on a Pi-side widening; widening reserves a request to Pi upstream.

**Risks.** Drift between Pi versions: the adapter sits on a surface Pi does not currently consider a stable extension contract; the version-bump audit must catch a Pi rename or refactor before it silently breaks discovery.

### Option C — Pin a single literal path *with* a fallback algorithm

**Approach.** Hybrid: state that the extension always scans `~/.pi/agent/npm/`, and additionally scans every directory found under a fixed, documented set of wrapper-managed prefixes (e.g. `$NPM_CONFIG_PREFIX/lib/node_modules` if set), without invoking `npmCommand`. The algorithm is purely environment-variable-driven and uses no subprocess.

**Spec edits.**

- Replace root #4 with the literal-path scan plus a defined extension to `$NPM_CONFIG_PREFIX/lib/node_modules` (or equivalent named env vars) when present.
- Pin the env-var set normatively in the prose; state that no other inference is performed.
- Add an edge-case for when the env-var resolves to a non-existent / non-directory path (silent skip, matching the rule for non-existent roots).

**Pros.** No new SDK surface; no presupposition on Pi internals; recovers most wrapper-using setups without requiring subprocess.

**Cons.** Environment-variable inference is fragile across `mise`/`asdf`/`nvm`/`fnm`; the spec ends up pinning a wrapper-detection algorithm that lives in loom rather than in Pi; misses any wrapper whose env-var convention the spec did not enumerate.

**Risks.** Spec maintenance: each new wrapper convention is a future spec edit; the algorithm risks scope creep into territory `packages.md` deliberately owns.

### Recommendation

Take **Option A**. The asymmetry with root #5 (git root #5 is already pinned at the literal path with no wrapper-resolution branch) is the strongest evidence that "literal path always" is the consistent rule for global roots, and root #4 was an outlier carrying an aspiration the SDK does not support. Option A is the only resolution that requires neither a Pi-side widening nor a new presupposition the SDK does not pin, and the documented limitation (wrapper-installed packages reachable only via `loomPaths` / project roots) is small enough to live in `**Edge cases**` rather than in a separate `host-prerequisites.md` presupposition. The implementer must watch for one edge case: a `loomPaths` entry that points into a wrapper-managed prefix is still in scope under root *Settings*, so the recovery path for affected operators is documented and not silently broken.

If a later loom version wants to recover wrapper-aware discovery (Option B), that is an additive widening of root #4 — the presupposition and PIC seam can be added then without first having to retract a contradictory contract.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on a supported Pi configuration. Under `npmCommand` (`mise`/`asdf`-wrapped npm), one implementation scans the wrapper-resolved root and discovers loom-shipping packages installed there; the other scans only `~/.pi/agent/npm/` and silently fails to discover them. Users who installed loom-shipping packages via `pi install -g` under such a wrapper get an empty discovery set with no diagnostic.
