## Finding

The inbound translation pass defined in `runtime-value-model.md` ("Wire-name translation") is specified as a walk over a *validated JSON value*: at every position the lowering pass's *Named-enum positions* sidecar maps to a declaring-enum name, the runtime reattaches that enum's tag to the validated string. The sidecar itself, defined in `schema-subset.md` Lowering Algorithm step 5, is "a map keyed by **JSON Pointer into the lowered schema fragment**" — i.e. keyed against the schema shape (which contains `$ref` indirection into `$defs`, `anyOf` arms, and `array`/`items` levels), not against the value shape (which contains none of those constructs).

The two key spaces are not interchangeable, and the spec never bridges them. Specifically: (a) it does not state how a position in the value walk is converted to a JSON Pointer into the lowered schema fragment, (b) it does not say how `$ref` is resolved on schema descent (the value has no `$ref` step, so a value-position pointer like `/foo/bar` does not literally exist in a schema that routes through `/properties/foo/$ref → #/$defs/X/properties/bar`), (c) it does not say how the matching `anyOf` arm is selected for a value position whose schema is a discriminated union (the value side carries no `anyOf` step, so the pointer is structurally ambiguous), and (d) recursive schemas (`schema Tree { children: array<Tree> }`, called out as supported by the depth-enforcement section) admit unbounded distinct value-position pointers reaching the same schema position, so a finite pointer-keyed lookup table cannot exhaustively cover them.

The practical consequence is that two conforming implementations will disagree on which validated strings get enum tags reattached. One implementer may build a parallel schema walk that resolves `$ref` and selects the matching `anyOf` arm by re-validating each candidate; another may pre-flatten the sidecar to value-shaped pointers (and silently miss recursive positions); a third may key the sidecar by schema-position identity instead of pointer. All three produce different runtime equality behaviour for enum variants returned through nested, union-typed, or recursive query response schemas — the very surface this contract was added to nail down.

## Solution Space

**Shape:** single

### Recommendation

Replace the JSON-Pointer-keyed sidecar with an explicit specification that the inbound pass is a **parallel value+schema walk**, and rewrite both step 5 of `schema-subset.md` and the inbound bullet in `runtime-value-model.md` to describe that walk directly. Concretely:

1. **Drop the "keyed by JSON Pointer" framing in step 5.** Redefine the *Named-enum positions* map as an annotation attached to each declaring-enum **schema node** in the lowered fragment (equivalently: a `Set` of object identities, or a sidecar field on the lowered `{"type":"string","enum":[…]}` node itself), valued by the loom-side declaring-enum name. The map is consulted by node-identity on the schema side, not by JSON Pointer.

2. **Specify the inbound pass as a parallel walk over (value, schema) pairs**, starting from `(validated_root, response_schema_root)` and descending in lockstep with these rules:
   - **`$ref` resolution.** When the schema position is a `$ref`, replace the schema cursor with the resolved `$defs/<Name>` body and continue the descent. The value cursor does not move. (Recursive `$ref` is fine: descent into a recursive position pushes the same schema node again, but only when the value side has further structure to descend into; the recursion terminates at value-side leaves.)
   - **Object descent.** For each present wire-named field on the value, descend into the schema's `properties/<wire-name>` (re-resolving `$ref` per the previous rule).
   - **Array descent.** For each value element, descend into the schema's `items`.
   - **`anyOf` arm selection.** When the schema position is `anyOf`, choose the unique arm that the validated value satisfies. Because AJV has already validated the value (with `additionalProperties: false` on every object form per step 3), a discriminated union's arm is uniquely determined by the discriminator field's `const` value; a mixed `anyOf` (e.g. `string | Author`) is uniquely determined by the value's JSON type. Specify this arm-selection rule normatively and state the diagnostic posture if zero or more than one arm matches (treat as `loom/runtime/anyof-arm-ambiguous` and let the value flow through untagged, or pick one of the existing runtime codes — name it explicitly).
   - **Tag reattachment.** When the value cursor sits on a JSON string and the schema cursor (post-`$ref` resolution, post-`anyOf` arm selection) is a node carrying a *Named-enum positions* annotation, brand the string with the annotated declaring-enum name. Anonymous string-literal-union positions carry no annotation and receive no tag.

3. **Pin the recursion contract explicitly.** State that the parallel walk's termination is bounded by the depth ceiling already enforced under *Depth Enforcement* (≤ 5 levels), so no separate cycle-detection mechanism is needed; a recursive schema's enum-tag reattachment is by construction finite because the *value* depth is capped.

4. **Add a reference vector or two** in `schema-subset.md` covering: an enum reached through a `$ref`, an enum reached through one arm of a discriminated `anyOf`, and an enum reached at depth 3 of a recursive schema (e.g. `schema Tree { tag: Kind, children: array<Tree> }`).

Edge cases the implementer must watch: (a) the parallel walk must descend into `Result.Ok` / `Result.Err` payloads as the existing inbound bullet already requires, which means the schema-side `anyOf` arm selection runs over the `Result` lowering as well; (b) the value's loom-side rebuild (renaming wire→loom) happens during the same pass, so the walk must consult the wire-name translation map for descent keys while the enum-tag reattachment consults the named-enum annotation — both maps live on the same sidecar by design; (c) defaults still bypass the pass per the existing paragraph, so this fix does not regress the frontmatter-defaults branch.

## Consequence

**Severity:** correctness

Two reasonable implementers of inbound translation diverge on which validated strings receive the declaring-enum tag whenever the response schema routes through `$ref`, an `anyOf` arm, or a recursive cycle. The observable divergence is `==` on enum variants returned from typed queries: one runtime treats `result.severity == Severity.High` as `true`, another as `false`, for the same wire bytes and the same source loom. Defaults bypass this pass per the same section, so the bug is invisible on local-only fixtures and surfaces only on real model output through non-trivial schemas — exactly the path the sidecar exists to handle.
