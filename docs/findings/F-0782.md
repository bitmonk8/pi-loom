## Problem

The inbound translation pass in `runtime-value-model.md` (Wire-name translation inbound bullet) consumes the *Named-enum positions* sidecar while walking a validated JSON value, but `schema-subset.md`'s Lowering Algorithm step 5 defines that sidecar as keyed by JSON Pointer into the lowered schema fragment. The schema key space contains `$ref` indirection, `anyOf` arms, and `array`/`items` levels that the value shape does not, and the spec never bridges the two: it does not state how a value-walk position converts to a schema pointer, how `$ref` is resolved on schema descent, how the matching `anyOf` arm is selected, or how recursive schemas (whose distinct value positions are unbounded) are covered by a finite pointer-keyed table. Two conforming implementations therefore disagree on which validated strings receive an enum tag, producing different `==` results for enum variants returned through nested, union-typed, or recursive response schemas.

## Issue introduction

**Verdict:** single-commit-introduced
**Introducing commit:** `7f217a8` (2026-06-07) — *pi-loom spec: resolve "Language-core hidden assumptions: enum-tag sidecar, AJV-config silence, Markdown claim"*
**History:** Before `7f217a8`, step 5 of the Lowering Algorithm declared only the wire-name translation map (`5. **Wire-name translation** is captured in a sidecar map per schema …`) and the inbound bullet in `runtime-value-model.md` spoke of "every position the schema annotates as a named enum" without naming a key space. The same commit (a) widened step 5 to a two-map sidecar adding "*Named-enum positions* — a map keyed by JSON Pointer into the lowered schema fragment, valued by the *loom-side* name of the declaring `enum`" and (b) rewrote the `runtime-value-model.md` inbound bullet to consume that sidecar by name. The mismatch between the sidecar's schema-pointer key space and the consumer's value-walk position space was introduced in that single edit; no later commit touched either bullet (`git log -S "Named-enum positions"` returns only `7f217a8`, and `git log --follow docs/spec_topics/runtime-value-model.md` shows no edit to the inbound bullet after `7f217a8`).

## Solution approach

Rewrite `schema-subset.md` Lowering Algorithm step 5 and the `runtime-value-model.md` inbound bullet so the spec defines, normatively, how the inbound pass maps each validated-value position to the declaring-enum annotation it must reattach — bridging the schema key space (`$ref` indirection, `anyOf` arms, `array`/`items` levels) to the value-walk positions that have none of those constructs. The specification must state how `$ref` is resolved on schema descent, how the matching `anyOf` arm is selected, and why a recursive schema's enum-tag reattachment is finite (the existing Depth Enforcement ceiling bounds it). Name the runtime diagnostic code for the case where `anyOf` arm selection matches zero or more than one arm.
