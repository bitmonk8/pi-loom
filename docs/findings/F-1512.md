## Finding

The indexed-access definition in `expressions.md` ("Supported forms" → Indexed access) pins a static type constraint on the index expression for **object** receivers — it must be of type `string`, and a non-`string` index is rejected at parse time as `loom/parse/non-string-object-index`. It pins **no** corresponding constraint on the index expression for **array** receivers. The only array-side rule is the runtime panic, defined purely as a range check: an index "outside `0..arr.length`" panics with `loom/runtime/index-out-of-bounds` (mirrored verbatim in `errors-and-results/error-model.md` and the runtime code registry as "`arr[i]` with `i < 0` or `i >= arr.length`").

Because `integer` and `number` are distinct loom types and the array index is never constrained to `integer`, a `number`-typed index expression — e.g. `a[1.5]`, or `a[x / y]` where the quotient is fractional — is admissible. Such a value is neither out of range (`1.5 < arr.length`) nor a valid element position. Its disposition is left entirely undefined: the range-check panic does not fire, no parse-time check rejects it, and no rule states whether the runtime should panic, floor/truncate, or treat the slot as absent.

The asymmetry is sharpened by the rest of the array surface, which already treats indices as integers throughout: `slice(start: integer, end?: integer)`, `indexOf(x): integer`. The natural and self-consistent contract is that an array index expression is statically `integer`, but the spec never states it, so the gap is real.

## Issue introduction

**Verdict:** single-commit introduction (asymmetric extension)
**Introducing commit:** `d6f17ca` — *"pi-loom spec: resolve 'indexed-access static semantics and runtime disposition'"* (2026-06-07).
**History:** The Indexed-access bullet first acquired any typed rule in `01ba226` (2026-06-04), which added only the receiver-side rule `loom/parse/non-indexable-receiver` — settling that strings are not indexable but pinning no index-type constraint on either branch. `d6f17ca` then extended the bullet with the object-receiver branch end-to-end (parse rule `loom/parse/non-string-object-index`, result-type rule, runtime-panic crossreference, read-order pin) and added the matching row to `code-registry-parse.md`, but did not author the symmetric array-receiver branch. The asymmetry is local to that commit's diff: prior to `d6f17ca` neither branch was pinned, so the hole is the omission of the array half of the bilateral fix `d6f17ca`'s commit title implies. `git log -S 'non-string-object-index'` confirms no later commit closes the array side.

## Solution Space

**Shape:** multiple

### Option A — Require the array index expression to be statically `integer`

- **Approach:** State at the indexing definition that for an `array<T>` receiver the index expression must be of type `integer`; a `number` (or any non-`integer`) index is a parse-time error. This mirrors the existing object-receiver rule (`string`-index → `loom/parse/non-string-object-index`) and makes a fractional index unrepresentable, so the runtime range check stays total over the reachable index domain.
- **Spec edits:** In `expressions.md` "Supported forms" → Indexed access, add a sentence: for an `array<T>` receiver the index expression must be of type `integer`; a non-`integer` index (e.g. `a[1.5]`) is rejected at parse time. Register the parse code in `diagnostics/code-registry-parse.md` (prefer extending/reusing an existing parse code's predicate over coining a new one; if a new code is unavoidable, name it symmetrically, e.g. `loom/parse/non-integer-array-index`). No change to `error-model.md`.
- **Pros:** Symmetric with the object-index rule; consistent with the array stdlib, which already types indices as `integer` (`slice`, `indexOf`); rejects the defect statically rather than deferring to runtime; leaves the runtime panic predicate untouched and total.
- **Cons:** Adds (or extends) a parse diagnostic; authors relying on a `number`-typed expression as an index must insert an explicit narrowing.
- **Risks:** Need to confirm whether an existing parse code can carry the predicate; coining a new code touches the registry and the `V3a` parse phase plus its `-T` fixtures.

### Option B — Admit `number` indices and define the runtime disposition

- **Approach:** Keep `number` indices statically legal and pin the runtime disposition of an in-range non-integer value, routing it to the existing `loom/runtime/index-out-of-bounds` panic (no new code).
- **Spec edits:** Broaden the panic predicate in `expressions.md`, `errors-and-results/error-model.md`, and `diagnostics/code-registry-runtime.md` from "`i < 0` or `i >= arr.length`" to also cover a non-integer `i`; adjust the message template if it must render a fractional `<i>`.
- **Pros:** No parse-time change; reuses the existing panic code.
- **Cons:** Pushes a statically-detectable error to runtime; broadens a panic whose name and message ("index out of bounds: `<i>` not in `0..<length>`") read awkwardly for an in-range fractional value; diverges from the object-index treatment, which rejects at parse time.
- **Risks:** The existing message template asserts `<i> not in 0..<length>`, which is false for `1.5` when length is 3; the template would need rewording, touching the diagnostics renderer fixtures.

### Recommendation

Take **Option A**: require the array index expression to be statically `integer`, rejecting a non-`integer` index at parse time. It restores symmetry with the object-index rule, matches the array stdlib's existing `integer` index typing, and keeps the runtime range check total without weakening a panic message into inaccuracy. Watch one edge case: confirm the parse code choice (reuse vs. new) and verify `integer ⊑ number` (TYPE-2) does not accidentally admit a `number` index under the compatibility relation — the constraint must demand `integer` exactly, not `⊑ number`.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on `a[1.5]`: one panics with `index-out-of-bounds`, another floors to `a[1]`, another treats it as a missing slot. The same loom source then produces different observable behaviour (different value vs. different panic) across conforming runtimes — exactly the divergence the closed panic list and parse-error catalogue exist to prevent.
