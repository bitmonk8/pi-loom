## Finding

PIC-8 in `pi-integration-contract/tool-registration-lifetime.md` and the `loom/runtime/active-set-restore-failed` row in `diagnostics/code-registry-runtime.md` both pin the same rule: "The retry MUST NOT chain back into `pi.setActiveTools` beyond the single re-attempt above (no exponential backoff, no third try)." The rule constrains the runtime's *call count* into `pi.setActiveTools`, not the shape of anything emitted. A conformant implementation issues exactly two calls and emits one `active-set-restore-failed` diagnostic + one `display: true` system note; a non-conformant implementation that performs three (or more) calls emits the same single diagnostic + note pair on the final failure. The two implementations are byte-identical on `loom-system-note`, persistent-diagnostic, and `Err` surfaces.

The spec therefore admits a falsification criterion only via a test seam that can count invocations of `pi.setActiveTools` itself — i.e., a fake `pi.*` harness. The runtime already routes every Pi SDK call through an injectable seam (per `H3a` and the `V9a` capability probe), so this is mechanically achievable, but neither the spec rule nor the `V9f-T` acceptance criterion presently pins the upper bound. `V9f-T` asserts that "a step-4 restore throw triggers one re-attempt, then `active-set-restore-failed` (E) + a `display:true` note, and propagates the original error" — which verifies the lower bound (a re-attempt happens) and the emission shape, but not that no third call is made.

The gap is asymmetric: under-counting (zero re-attempts) is caught by the existing test, since a single throw would propagate without emitting the diagnostic at all. Over-counting (three or more attempts, with or without backoff) passes silently.

## Solution Space

**Shape:** single

### Recommendation

Pin the call-count cap as a normative falsification criterion at the PIC-8 site, and add the corresponding citing assertion to `V9f-T`.

**Spec edits.**

- In `docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md`, after the existing PIC-8 sentence "(no exponential backoff, no third try)", append: "Conformance: a test fake whose `pi.setActiveTools(snapshot)` rejects unconditionally MUST observe exactly two invocations of `pi.setActiveTools` from the restore path on a single snapshot/restore window, one `loom/runtime/active-set-restore-failed` diagnostic, and one `display: true` `loom-system-note`."
- The `loom/runtime/active-set-restore-failed` row in `diagnostics/code-registry-runtime.md` already reads "the runtime's mandated single re-attempt with the same snapshot also failed" and need not be re-edited; it inherits the conformance clause via the cross-reference to PIC-8.

**Plan edits.**

- In `docs/plan_topics/V9f-T-tool-registration-lifetime.md`, replace the existing PIC-8 bullet with two assertions: (i) the existing re-attempt-and-emit assertion (lower bound), and (ii) the new upper-bound assertion — a `pi.setActiveTools`-rejecting fake observes exactly two invocations from the restore path per snapshot/restore window, with exactly one `active-set-restore-failed` and one `display: true` system note per invocation.
- No edit to `V9f` itself is required; the implementation obligation is unchanged, only the test obligation gains an upper-bound assertion.

**Edge cases the implementer must watch.**

- The cap is per snapshot/restore window, not per invocation: a prompt → prompt nested `invoke` stacks two LIFO windows per the surrounding section, and each window owes its own pair of attempts on its own failure. The conformance fake must count per-window, not process-globally.
- The setup-side `pi.getActiveTools()` / step-2 swap-install calls are PIC-19, not PIC-8, and must be excluded from the count (they never trigger this code).
- Counting the *snapshot-side* `pi.setActiveTools` invocation (step 2) against the cap is a category error; only step-4 restore-path calls count.

## Consequence

**Severity:** advisory

A spec-compliant implementer who reads PIC-8 literally still implements the call-cap. A defensive implementer who, on encountering a transient `pi.setActiveTools` rejection in production, adds a third attempt or a small exponential-backoff retry under the same `finally` will violate the rule without any conformance test catching the drift. The cost of the drift is bounded — the diagnostic and system-note are still emitted on the terminal failure — but the rule's purpose (bounding host-state-mutating retries on the user-session active set) is silently weakened.
