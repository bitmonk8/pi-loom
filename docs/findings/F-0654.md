## Problem

PIC-8's "The retry MUST NOT chain back into `pi.setActiveTools` beyond the single re-attempt above (no exponential backoff, no third try)" constrains the runtime's call count into `pi.setActiveTools`, not the shape of anything emitted. A conformant implementation (exactly two calls) and a non-conformant one (three or more calls) emit a byte-identical single `active-set-restore-failed` diagnostic, `display: true` system note, and `Err` surface on terminal failure. The only falsification path is a fake `pi.*` harness counting restore-path invocations, but neither the PIC-8 rule nor the `V9f-T` acceptance criterion pins the upper bound. The gap is asymmetric: under-counting is caught by the existing test, while over-counting passes silently.

## Solution approach

Add a conformance criterion at PIC-8 in `pi-integration-contract/tool-registration-lifetime.md` that makes the call-count cap observable to a test seam — a fake whose restore-path `pi.setActiveTools(snapshot)` rejects unconditionally observes exactly two restore-path invocations per snapshot/restore window. Add the matching upper-bound assertion to the PIC-8 test bullet in `V9f-T-tool-registration-lifetime.md`, alongside the existing re-attempt-and-emit (lower-bound) assertion. The `active-set-restore-failed` row in `diagnostics/code-registry-runtime.md` inherits the criterion via its cross-reference to PIC-8 and needs no edit.

## Solution constraints

- The count covers only step-4 restore-path `pi.setActiveTools` calls; the step-1 `pi.getActiveTools()` and step-2 swap-install calls are PIC-19 setup-side calls and are out of scope of the cap.
- The cap is per snapshot/restore window, not process-global: LIFO-nested prompt → prompt windows each owe their own attempt pair, so the conformance fake counts per-window.
