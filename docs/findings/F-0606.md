## Finding

Every path-shaped literal in `audit-resolution.md` is written with `/` as the segment separator: the include glob `src/**/*.ts`, the exclusion globs (`*.test.ts`, `*.spec.ts`, `*.d.ts`, `src/extension/**/*.assert.ts`), and the `__tests__/` directory-segment match. The *Cross-platform glob normalisation* paragraph pins case-sensitivity across Linux / macOS / Windows hosts and *Symlink handling* pins opacity, but neither paragraph — and no other paragraph on the page — states what path-separator the audit's walker compares paths against before glob, segment, or suffix matching. On Windows, Node's `fs.readdir` / `path.relative` produce paths separated by `\`, so a naive walker comparing `src\extension\index.ts` against `src/**/*.ts` (or scanning the path for the literal substring `__tests__/`) would include or exclude the wrong file set. Two compliant Windows implementations could legitimately disagree about which files are in scope.

The omission directly defeats the page's own portability guarantee — "a Linux contributor … and a Windows / macOS contributor … audit the same closed file set for a given source tree" — which is asserted for case-folding but never extended to separators. It also splits load from the sibling `audit-wire-and-canary.md` *Wire serialisation* paragraph, which already commits "lexicographic on POSIX-form path" for the deterministic file-iteration order; the audit relies on POSIX-form paths at the output stage without ever pinning POSIX-form normalisation at the matching stage that precedes it.

## Solution Space

**Shape:** single

### Recommendation

Extend the *Cross-platform glob normalisation* paragraph in `audit-resolution.md` to pin path-separator normalisation alongside the existing case-sensitivity clause: before the audit performs any glob match, directory-segment match, or extension-suffix match, the implementation MUST normalise the candidate path's separators to ASCII forward-slash `/` (i.e. replace every ASCII `\` in the path with `/`). State the rule once at the matching boundary and make it cover the include glob, every exclusion glob, the `__tests__/` segment scan, and the `src/extension/**/*.assert.ts` suffix match — i.e. every path-shaped literal currently on the page. Tie the rule explicitly to the existing "POSIX-form path" commitment in `audit-wire-and-canary.md`'s *Wire serialisation* paragraph so the matching-stage normalisation and the output-stage lexicographic-on-POSIX-form-path ordering share a single normal form. Phrase it as a behavioural rule on the audit's path comparison, not as an obligation on the host filesystem, so it holds uniformly across walker implementations (Node `fs.readdir`, `glob`, `fast-glob`, `minimatch`, etc.).

Implementer-relevant edge cases: the rule applies to relative paths the walker produces while traversing the audited source tree (paths rooted at the include glob's anchor — see the related "glob root-anchoring directory unstated" finding for what that anchor is); the audit does not need to round-trip the normalised path back to the host filesystem since file reads can keep the original host-form path. A literal `\` inside a filename (legal on POSIX, illegal on Windows) is out of scope at loom 1.0 — TypeScript source files do not carry one in practice — but if the spec wants to be paranoid the rule can say "replace every directory-separator `\`" rather than "every `\`" to leave the question open. Confirm that the `path:line` wire field in `audit-wire-and-canary.md` is emitted with the normalised separator so the output format is also platform-independent.

## Consequence

**Severity:** correctness

A Windows audit run can diverge from a Linux/macOS run on the same source tree: a `src\extension\foo\__tests__\bar.ts` file the spec intends to exclude would be included if the walker scans for the literal `__tests__/` segment without separator normalisation, and a `src\extension\index.ts` may not match `src/**/*.ts` under a naive glob engine. Either direction breaks the audit's "same closed file set" portability guarantee and makes the family (1)/(2)/(3)/(5) discriminators platform-dependent in a way the failure-surface contract assumes they are not.
