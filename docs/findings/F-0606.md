## Problem

Every path-shaped literal in `audit-resolution.md` is written with `/` as the segment separator (the include glob `src/**/*.ts`, the exclusion globs, the `__tests__/` directory-segment scan, and the `src/extension/**/*.assert.ts` suffix), but no paragraph on the page states what path-separator the audit's walker normalises candidate paths to before glob, segment, or suffix matching. On Windows hosts, `fs.readdir` / `path.relative` produce `\`-separated paths, so a naive walker comparing `src\extension\index.ts` against `src/**/*.ts` (or scanning for the literal `__tests__/` segment) would include or exclude the wrong file set, and two compliant implementations could legitimately disagree about which files are in scope. This defeats the page's own "same closed file set" portability guarantee, which is asserted for case-folding but never extended to separators.

## Solution approach

Clarify the *Cross-platform glob normalisation* paragraph in `audit-resolution.md` to pin the path-separator normal form the audit applies before any glob, directory-segment, or extension-suffix match, covering the include glob, the exclusion globs, the `__tests__/` segment scan, and the `src/extension/**/*.assert.ts` suffix match. Phrase it as a behavioural rule on the audit's path comparison rather than an obligation on the host filesystem, so it holds uniformly across walker implementations. Tie the normal form to the existing `lexicographic on POSIX-form path` commitment in the sibling `audit-wire-and-canary.md` *Wire serialisation* paragraph so the matching stage and the output stage share one normal form.
