## Problem

DIAG-2 (`docs/spec_topics/diagnostics/diagnostic-shape.md`, `id="diag-2"`) states that the registry is closed and that changing a code's trigger is a spec change, presenting a closed-trigger rule with no carve-out. The `loom/runtime/internal-error` row in `code-registry-runtime.md` carries a deliberately open-ended *Trigger* cell, which read against DIAG-2 appears to violate it. The reconciling interpretation — that DIAG-2 constrains *codes*, not triggers — is normative and load-bearing, but it appears only in the second sentence of the runtime-table preamble that introduces `loom/runtime/internal-error`; neither the DIAG-2 rule text nor the row's *Trigger* column states it. A reader who reaches DIAG-2 directly, or who lands on the `internal-error` row first, sees the apparent conflict with no carve-out at either point.

**History:** Commit 9d7b1b0 rewrote the runtime-registry intro to introduce the runtime-defect-surface framing for `loom/runtime/internal-error` and, at the same time, added the load-bearing sentence "The code is registered and stable; only its trigger condition is intentionally open-ended, which does not weaken the closed-registry rule (rule 2 above) — that rule constrains *codes*, not triggers." into that intro. A month later, 6bde0df coined the `DIAG-2` anchor on the existing closed-registry rule in `diagnostic-shape.md` without absorbing the carve-out clause, and 8d730c5 amended DIAG-2 again to add the GOV-15 disposition for code additions/removals — again without picking up the open-trigger carve-out. The clarification has therefore lived only in the runtime preamble since 9d7b1b0; the two later edits to the DIAG-2 rule text both passed over it.

## Solution approach

Add the open-trigger carve-out to DIAG-2 in `diagnostic-shape.md` so the carve-out lives at its canonical rule home. Rewrite the `code-registry-runtime.md` runtime preamble's `internal-error` carve-out sentence as a cross-reference to the DIAG-2 carve-out. Keep the carve-out grounded in the runtime-defect-surface framing via the existing cross-reference to `errors-and-results/error-model.md#runtime-panics`.

## Solution constraints

- Keep the carve-out scoped to open-ended *trigger* enumeration; do not weaken DIAG-2's spec-change discipline for code additions/removals or namespace/severity changes (GOV-15-governed).
- Do not duplicate the panic-vs-defect taxonomy from `errors-and-results/error-model.md#runtime-panics` into `diagnostic-shape.md`.
