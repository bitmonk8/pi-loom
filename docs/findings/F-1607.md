## Problem

In `docs/spec_topics/binder/defaulting-system-note-echo.md`, the
`#system-note-rendering` rule 2 ("Length cap.") describes the optional
grapheme-cluster back-off with two adjacent sentences that contradict each
other on normative status: the first grants the back-off via `MAY` (itself an
RFC 2119 / BCP 14 normative permission keyword), while the next labels the same
behaviour "non-normative" and attaches `tests MUST NOT assert cluster-aware
behaviour`. A clause cannot simultaneously confer a `MAY` permission and be
"non-normative", and a sub-clause carrying a `MUST NOT` cannot be non-normative.
The result is implementer/conformance-author friction: a test author cannot
decide whether the back-off and its `MUST NOT` side-constraint are enforceable
spec surface or merely editorial illustration.

## Issue introduction

**Verdict:** present-since-section-rewrite (single commit)
**Introducing commits:** `6efbd8a` (2026-05-05, "pi-loom spec: resolve
'System-note 120-codepoint cap truncation unit'").
**History:** Rule 2 was substantially rewritten in `6efbd8a` from a single
sentence ("Overflow is replaced with a trailing `…` (U+2026), which counts
toward the cap.") into the current multi-clause form. The new wording
introduced both the `MAY` permission and the adjacent "non-normative … tests
MUST NOT assert cluster-aware behaviour" qualifier in the same diff hunk; the
two clauses have been mutually contradictory since they landed. Subsequent
edits — `f5e89f4` (2026-06-04, the corpus-wide 100KB-cap split that moved the
text into `docs/spec_topics/binder/defaulting-system-note-echo.md`) and
`c15da5c` (2026-06-24, "supply precedence for system-note exactly-120 cap vs
cluster back-off", which reordered the precedence between the exactly-120
requirement and the back-off) — preserved both clauses verbatim and did not
touch the MAY/non-normative pairing. `git log -S "cluster-aware" --
docs/spec_topics/binder/defaulting-system-note-echo.md` confirms `f5e89f4` as
the sole introducing commit at the current path; `git log -S "grapheme" --
spec_topics/binder.md` traces the wording back to `6efbd8a` at its original
path.

## Solution approach

Rewrite the conformance sentence in `#system-note-rendering` rule 2 ("Length
cap.") so the grapheme-cluster back-off is framed as the normative optional
permission the `MAY` grammatically already confers, and so the conformance-test
prohibition reads as a parallel normative obligation rather than a footnote to
a "non-normative" label. Leave the preceding `MAY` sentence unchanged.

## Solution constraints

- The conformance text must stay inside rule 2's body; do not promote it to its
  own numbered rule (that re-opens the exactly-120-vs-back-off precedence
  ordering c15da5c established).
