## Problem

The `H5d` transitive-completeness gate arm tokenises every closing-leaf cell of `coverage-matrix.md`'s *Numbered REQ-IDs* and *Code-keyed obligation areas* tables by its backtick-delimited spans and fails CI on any cell none of whose listed leaves is in `H5b`'s expanded `Deps.`. The only stated exclusion is the bare `<new>` placeholder row. The *Code-keyed obligation areas* table carries ten retired rows (`cka-7`, `cka-12`, `cka-18`, `cka-21`, `cka-22`, `cka-23`, `cka-24`, `cka-25`, `cka-28`, `cka-29`) whose closing-leaf cell holds the literal `*(numbered above)*` with no backtick span, so each tokenises to the empty set and vacuously satisfies the failure condition — a faithful implementation reddens CI permanently against the live table. No plan text distinguishes a retired `*(numbered above)*` row from a `<new>` placeholder or from a mis-authored un-delimited single-leaf cell, leaving the always-fail and skip-every-empty readings unresolved.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `617a3ad` (gate definition with `<new>`-only exclusion) interacting with `716a13b`, `47915a2`, `ffe2cc9` (later cka-row retirements that introduced the `*(numbered above)*` cell shape)
**History:** `617a3ad` (2026-06-16, "H5a bundles gate arms spanning four data sources") created the `H5d` arm; its row-exclusion set was `<new>`-only and complete for the corpus as it then stood — no `*(numbered above)*` row existed. The first retired row carrying that literal entered in `716a13b` (2026-06-25, F-1286, query-subtree migration retiring `cka-12`), with further retirements in `47915a2` (2026-06-27, F-1890) and `ffe2cc9` (2026-06-28, F-2317, retiring `cka-7`). None of those retirement commits extended the gate's exclusion set, so the defect emerged from the interaction between the gate's fixed exclusion set and the later retirement commits rather than from any single commit in isolation.

## Solution approach

Add a third recognised empty-tokenisation case to the `H5d` arm so a closing-leaf cell whose content is exactly the literal `*(numbered above)*` is excluded from the at-least-one check, alongside the existing `<new>` exclusion. Propagate the same recogniser to the three homes that describe the arm: `H5d-transitive-completeness-gate.md` (its **Adds**, **Tests**, and **Ships when**), the *Transitive-completeness plan-maintenance* obligation in `conventions.md` §REQ-ID discipline, and §"Closing-leaf column authoring" in `coverage-matrix.md` (defining the marker as a third recognised empty-tokenising cell form distinct from `<new>` and from a mis-authored un-delimited cell). Key the recogniser on the exact literal cell content `*(numbered above)*`.

## Solution constraints

- Any empty-tokenised closing-leaf cell that is neither the literal `<new>` nor the literal `*(numbered above)*` must continue to redden the arm, preserving the mis-authored-un-delimited-cell detection the §"Closing-leaf column authoring" rule depends on.
- The spec is read-only for this fix.
