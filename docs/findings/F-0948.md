## Finding

`V11g` owns the post-default-merge AJV validation hook: after fill-if-absent
defaulting fills absent wire names with their declared defaults, the merged
`args` are re-validated through `SchemaValidator.validate()` against the lowered
`params` schema (spec `binder/defaulting-system-note-echo.md`
`#post-default-merge-ajv-validation`; coverage-matrix row `cka-40`). Both
`V11g` and `V11g-T` carry a single `Tests` bullet — "Fill-then-revalidate" —
which exercises only a well-formed merge that passes validation. Neither leaf
asserts the negative path: a merged `args` payload that violates the schema must
be rejected by the `SchemaValidator.validate()` call this leaf installs.

Because the only gate is the happy path, an implementation in which the
`validate()` call is made but its rejection verdict is dropped (or where
defaulting silently produces a structurally valid-but-wrong merge) would still
pass `V11g`'s `npm test`. The re-validation can therefore ship vacuous within
the leaf that owns it.

The failure-path *routing* is adjacent in `V11f`: `HC3-c` asserts an
AJV-on-`args` failure is not retried, and `V11f`'s depth-walk fast-fail bullet
exercises a `params`-boundary rejection at this same enforcement site. But those
`V11f` cases drive a binder-returned envelope, not a runtime default-filled
merge, and they live in a sibling leaf — so nothing in `V11g` itself proves its
own re-validation rejects an invalid post-merge result.

## Solution Space

**Shape:** multiple

### Option A — Add a dedicated negative gate to V11g / V11g-T

**Approach.** Add a failure-path `Tests` assertion to both `V11g` and its paired
`V11g-T` leaf: a post-default-merge `args` object that violates the lowered
`params` schema causes `SchemaValidator.validate()` to reject, and the rejection
is classified into the AJV-on-`args` failure class (per
`determinism-cancellation-failure.md` `#failure-class-taxonomy`).

**Plan edits.** In `docs/plan_topics/V11g-T-defaulting-revalidation.md` and
`docs/plan_topics/V11g-defaulting-revalidation.md`, add a `Tests` bullet whose
content asserts that a defaulted merge violating the lowered `params` schema
fails `SchemaValidator.validate()` and is routed to the AJV-on-`args` class;
extend `V11g`'s `Ships when` so `npm test` exercises the reject path as well as
the accept path. Cite the same spec anchors the existing bullet uses
(`#post-default-merge-ajv-validation` and the failure-class taxonomy).

**Spec edits.** None.

**Pros.** Closes the vacuous-pass risk inside the leaf that owns the hook; the
default-fill-induced invalid-merge case (distinct from `V11f`'s binder-returned
envelope cases) gets an explicit gate.

**Cons.** Adds a test the `V11f` taxonomy partly overlaps.

**Risks.** Minimal — the assertion targets behaviour the spec already pins.

### Option B — Cross-reference V11f's taxonomy test as the gate

**Approach.** Leave `V11g`'s tests success-only and add an explicit pointer
stating that the post-merge reject path is gated by `V11f`'s AJV-on-`args` /
depth-walk fast-fail assertions.

**Plan edits.** In `docs/plan_topics/V11g-defaulting-revalidation.md`, add a
note in `Tests` naming `V11f` (`HC3-c` and the depth-walk fast-fail bullet) as
the gate for the rejection path.

**Spec edits.** None.

**Pros.** No new test; relies on coverage already present.

**Cons.** `V11f`'s cases drive a binder-returned envelope, not a runtime
default-filled merge, so the specific default-fill-induced invalid-merge path
stays ungated; the pointer documents an overlap rather than closing the gap.

**Risks.** A defect that only manifests on a default-filled (not
binder-supplied) invalid merge stays undetected.

### Recommendation

Take Option A. `V11f`'s failure-taxonomy cases exercise binder-returned and
depth-walk-fast-fail `args`, not a runtime default-filled merge that violates the
schema, so a cross-reference alone leaves `V11g`'s own hook ungated against the
default-fill case it uniquely produces. Add the negative bullet to `V11g-T`
(red) and `V11g`, citing `#post-default-merge-ajv-validation` and the
failure-class taxonomy anchor. Edge case for the implementer: a defaulted field
whose declared default is itself schema-valid cannot trigger the reject path —
construct the negative fixture so the schema violation arises from the merged
result (e.g. an interaction with a sibling required field or a depth/shape
constraint), not from a well-formed default value.

## Consequence

**Severity:** advisory

If shipped unfixed, `V11g`'s `npm test` certifies only that re-validation runs
and accepts a valid merge; a re-validation that never rejects an invalid merge
passes the leaf's own gate vacuously. End-to-end failure routing is still
covered transitively by `V11f`, so a working implementation is producible, but
the leaf that owns the hook carries no negative assertion of its own.
