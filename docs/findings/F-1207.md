## Problem

The bare-`return`-legality bullet in `docs/spec_topics/return.md` says bare `return` is legal "only inside a `void` function or `void` top-level loom". The `void` function arm is constructible (a `fn` declaration carries an explicit return annotation, and `ReturnType` admits `void`), but the `void` top-level loom arm is not: a top-level loom acquires its return type only by inference, [Function Definitions — Loom return type](./functions.md#loom-return-type) states "There is no frontmatter `returns:` field", and `LoomBody` carries no return-annotation slot. No frontmatter, grammar, or inference path produces a `void` top-level loom. The bullet therefore licences a state the rest of the spec forbids, leaving each implementer to decide whether bare top-level `return` is ever legal — the divergence the spec is meant to close.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `53f5831` (2026-05-03, *spec: rewrite query primitive, system prompts, and schema syntax*) — present-since-inception introduction of the bullet text "Bare `return` (no argument) is legal only inside a `void` function or `void` top-level loom"; `8eea523` (2026-05-04, *spec: fill implementation-blocking gaps*) — added the **Loom return type** paragraph with "There is no frontmatter `returns:` field", which closed off the only candidate frontmatter mechanism that would have made a `void` top-level loom constructible.
**History:** `git log -L 20,20:docs/spec_topics/return.md` confirms the "`void` function or `void` top-level loom" wording landed in `53f5831` (then in `spec.md`, later split into `spec_topics/return.md` by `fecb504` and moved under `docs/` by `31ff060`) and has been preserved by every later edit — including `839db71` (named the diagnostic code), `023d576` (rephrased the sibling top-level-`return-expr` bullet), and `08886d9` (rewrote the **Loom return type** paragraph for early-return reconciliation). `git log -G 'no frontmatter .returns' -- docs/` locates `8eea523` as the commit that added the no-`returns:` clause to **Loom return type**. The clause was tightened further by `08886d9` (which kept "There is no frontmatter `returns:` field" verbatim) without revisiting the now-orphaned `return.md` arm. The defect crystallised across these two commits: `53f5831` introduced the bullet while loom return typing was still under-specified; `8eea523` closed the candidate path without back-propagating the change to `return.md`. Subsequent edits (`023d576`, `08886d9`) touched neighbouring lines but did not reopen the bullet.

## Solution approach

Rewrite the bare-`return`-legality bullet in `docs/spec_topics/return.md` to drop the unconstructible `void` top-level loom arm and pin the top-level outcome explicitly: bare `return` at the top level of a `.loom` file is `loom/parse/bare-return-in-non-void`, because a top-level loom has no return annotation and acquires its return type only by inference per [Function Definitions — Loom return type](./functions.md#loom-return-type).

## Solution constraints

- Preserve the `void`-annotated function arm; only the top-level loom arm is unconstructible.
