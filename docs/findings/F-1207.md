## Finding

The Return Statement page enumerates the rule:

> Bare `return` (no argument) is legal only inside a `void` function or `void` top-level loom; elsewhere it is `loom/parse/bare-return-in-non-void` ("missing return value").

The "`void` function" arm is constructible — a `fn` declaration carries an explicit return annotation, and `ReturnType ::= Type | "void"` in [Grammar Appendix — Type grammar](./grammar.md#type-grammar) admits `void` there. The "`void` top-level loom" arm is not. A top-level loom acquires its return type by inference only: [Function Definitions — Loom return type](./functions.md#loom-return-type) says "There is no frontmatter `returns:` field", and [Function Definitions — Empty-tail body](./functions.md#empty-tail-body) says an empty-tail body has inferred return type `null` (the literal type), explicitly noting that "absence of a tail expression on its own does NOT imply `void`". `LoomBody` in [Grammar Appendix — Block expressions](./grammar.md#block-expressions) carries no return-annotation slot either. There is no path — frontmatter, grammar, or inference — by which a `.loom` file's top level becomes `void`.

The bullet therefore licences a state the rest of the spec forbids. An implementer following return.md verbatim either invents a `void` top-level loom mechanism (diverging from functions.md / grammar.md / frontmatter) or correctly concludes the arm is dead and produces a diagnostic on every bare `return` at the top level — but that decision is left to each implementer, which is the divergence the spec is supposed to close.

The companion sentence two lines below ("From a top-level loom, `return expr` exits the loom with `expr` as its return value, exactly as a tail expression would") is consistent with the rest of the spec — top-level `return expr` is well-defined; only the bare-`return` arm is broken.

## Solution Space

**Shape:** single

### Recommendation

Edit the bare-`return` bullet in `docs/spec_topics/return.md` to drop the unreachable arm and pin the top-level outcome explicitly:

> Bare `return` (no argument) is legal only inside a function whose explicit return annotation is `void`; elsewhere — including the top level of a `.loom` file, since a top-level loom has no return annotation and acquires its return type only by inference per [Function Definitions — Loom return type](./functions.md#loom-return-type) — it is `loom/parse/bare-return-in-non-void` ("missing return value").

Edge cases the implementer must watch:

- A `.warp` library file's top-level `fn` declarations are still governed by the "`void` function" arm; only the loom file's *top-level block* (`LoomBody`) is excluded.
- If `docs/spec_topics/diagnostics/code-registry-parse.md`'s row for `loom/parse/bare-return-in-non-void` restates the legality condition in its Description column, mirror the edit there so the diagnostic registry and the owning rule do not drift.
- The V3d / V3d-T acceptance criteria currently read "bare `return` is allowed only in `void`"; narrow them to "bare `return` is allowed only in a `void`-annotated function, and is rejected with `loom/parse/bare-return-in-non-void` at the top level of every `.loom`".

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on whether bare `return` at the top level of a `.loom` is ever legal: one carves out a hypothetical "void top-level loom" path (perhaps by reading `void` into the type-system register), the other rejects every bare top-level `return`. The diagnostic surface (`loom/parse/bare-return-in-non-void` fires? doesn't fire?) and the V3d acceptance test (cited verbatim against this rule in `plan_topics/V3d-T-functions-and-return.md`) both bend accordingly.
