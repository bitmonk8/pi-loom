## Finding

`match` is a first-class expression form in loom, but its normative definition — pattern grammar, disambiguation rule, exhaustiveness contract, arm-syntax / LUB-typing rule, and runtime no-match panic — lives entirely in `errors-and-results/error-model.md`. `expressions.md` (the page whose lede declares itself the home for "the same grammar [that] applies wherever an expression is expected: the RHS of `let`, `if` / `match` scrutinees, function arguments, …") names `match` four times — in the lede, the *Not supported* list ("`match` inside `${...}`"), and the grammar-disambiguation rules (scrutinee parens, arm body single-expression) — but never defines it and never links forward to where it is defined. `grammar.md` carries the `MatchArm`/`ArmBody` productions and a back-reference to `error-model.md` for the surface; `control-flow.md` does not mention `match` at all.

The practical effect is one-way discoverability. A reader who treats `expressions.md` as the language reference encounters `match` four times as an undefined construct with no in-page cross-reference to its semantics. A reader of `error-model.md` finds the full surface but co-mingled with `Result`, `?`, `ERR-18`, runtime panics, and terminal outcomes — none of which are conceptually required to understand `match` itself (a `match` on a non-`Result` scrutinee is a perfectly ordinary loom expression). The implementation-side leaf `V4a` does cite both pages and so produces a correct implementation, but the spec-as-document fails to surface a canonical home that a language-side reader (or a `match`-only feature-page xref from a sibling page) can point at.

The finding's separate framing that "an agent must invent parse-time vs runtime no-match behaviour and the diagnostic" overstates the gap — the runtime `loom/runtime/match-error` panic is normatively pinned in `error-model.md` (Exhaustiveness paragraph), the parse-time `loom/parse/match-arm-type-mismatch` / `loom/parse/match-guard-not-supported` / `loom/parse/rest-pattern-not-supported` / `loom/parse/statement-in-arm-body` codes are in the parse registry, and `grammar.md` owns the arm-body production. The real defect is placement, not under-specification.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 53f5831 — spec: rewrite query primitive, system prompts, and schema syntax (2026-05-03, Thomas Andersen)
**History:** `match` entered the corpus in commit 53f5831 as part of the query/`Result`/`?` rewrite of the monolithic `spec.md`; the rewrite created the *Expression Sublanguage* and *Errors and Results* sections in the same commit and placed the full `match` definition (pattern grammar, exhaustiveness, arm syntax) under *Errors and Results* with only naming references from the expression section. The subsequent reorganisations preserved that placement: `fecb504` split the monolith into `spec_topics/errors-and-results.md` and `spec_topics/expressions.md` as separate files, and `f5e89f4` (the 30 KB / 100 KB cap split) moved the `match` paragraphs into `errors-and-results/error-model.md` without revisiting which page should own them. The placement decision has therefore never been re-examined since the construct was first authored.

## Solution Space

**Shape:** single

### Recommendation

Leave the normative `match` paragraphs (pattern grammar, disambiguation rule, exhaustiveness contract, arm-syntax LUB rule, and the `loom/runtime/match-error` runtime panic) where they are in `errors-and-results/error-model.md`, preserving the `match` ↔ `Result` ↔ `?` story (the `?`-to-`match` equivalence and the panic's bypass-of-`?`-and-`match` rule).

Add a new `## `match` expression` section to `expressions.md`, placed immediately **before** *Grammar disambiguation* so the scrutinee-parens and arm-body rules in that section read in their right context. The section is a short (~5-line) entry point containing a one-paragraph form description — `match` is an exhaustive destructuring expression, its arm bodies are expressions, and its scrutinee may be any expression — followed by an explicit forward link: "The full pattern grammar, disambiguation rule, exhaustiveness contract, arm-syntax LUB rule, and `loom/runtime/match-error` runtime panic are defined in [Errors and Results — `match` expression](./errors-and-results/error-model.md)."

The stub MUST NOT restate any of the rules (pattern grammar, exhaustiveness, arm-syntax LUB); it asserts *that* the construct exists and links forward only, kept deliberately rule-free so it cannot drift from the canonical paragraphs.

Tighten the lede of `error-model.md`'s `match` section in one sentence so it acknowledges it owns the language-wide construct (not only the error-handling instance), letting a sibling-page cross-reference to "the canonical match definition" land there legitimately.

No paragraph moves, no diagnostic-registry row touches, no plan-leaf pointer changes.

## Consequence

**Severity:** advisory

A spec reader who opens `expressions.md` expecting it to define every expression form named in its own lede is left to follow a forward chain (`expressions.md` lede → `match` undefined → search → `error-model.md`) to find pattern grammar, exhaustiveness, and arm-body typing. Two implementers will still converge on the same runtime behaviour because the plan leaf `V4a` and the diagnostics registry pin everything that matters, and `grammar.md` already cross-references `error-model.md`; nothing observable diverges. The cost is reader friction and a missing canonical anchor for sibling pages that today have to point at an error-handling page to talk about a non-error expression form.
