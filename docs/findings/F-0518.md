## Finding

The `DIAG-3` test bullet in both `V7b-code-registry.md` and `V7b-T-code-registry.md` reads: "`DIAG-3`: codes are stable identifiers — a renamed code fails the gate (renames deferred to loom 2.0)." The operative, in-scope test requirement is "a renamed code fails the gate." The trailing parenthetical "(renames deferred to loom 2.0)" is a forward-compatibility note about *migration scope*, not a property the test asserts.

The deferral is already owned authoritatively by the spec: `diagnostics/diagnostic-shape.md` §DIAG-3 states "A rename is therefore deferred to loom 2.0 migration, outside the GOV-15 diagnostic-registry carve-out." Restating it inline in the test bullet adds nothing the spec does not already carry, and it sits next to a normative assertion where an implementer expects only the gate behaviour. The juxtaposition invites the implementer to wonder whether the gate must somehow *handle* renames (a migration concern) rather than simply *reject* them, which is the opposite of the bullet's actual requirement.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** Both `V7b-code-registry.md` and `V7b-T-code-registry.md` first enter version control in `c6a664e`, the initial plan-build commit. `git log -S"renames deferred to loom 2.0"` against each file returns only `c6a664e`, so the parenthetical was present in the `DIAG-3` bullet from the moment the leaves were authored; no later edit introduced it. The subsequent commit `f284534` touched these files for an unrelated DIAG-2 resolution and did not add or remove the qualifier.

## Solution Space

**Shape:** single

### Recommendation

In the `Tests.` section of both `docs/plan_topics/V7b-code-registry.md` and `docs/plan_topics/V7b-T-code-registry.md`, strike the parenthetical from the `DIAG-3` bullet so it reads:

`` - `DIAG-3`: codes are stable identifiers — a renamed code fails the gate. ``

The deferral disposition remains owned by `diagnostics/diagnostic-shape.md` §DIAG-3; no plan-side restatement is needed. The edit is identical in both leaves and changes no asserted behaviour.

## Consequence

**Severity:** advisory

If shipped unfixed, the implementer still produces a working gate, but the parenthetical is noise next to a normative assertion and can prompt a needless detour into whether rename-handling/migration is in scope for the gate (it is not — the gate only rejects renames). No behavioural divergence; pure readability/scope-clarity cost.
