## Problem

The [Failure-class taxonomy](#failure-class-taxonomy) routes a structural `ContextOverflowError` into the *Binder model transport failure* row of [Failure-mode templates](#failure-mode-templates-normative), but `ContextOverflowError` carries no `provider` field while that row's `<provider>` placeholder is pinned to the classifier-produced `TransportError.provider`. No normative rule states how `<provider>` and `<message>` are populated when the underlying failure is a context overflow rather than a transport error. The rendered failure note is a declared spec-versioned breaking-change surface, so the absent mapping leaves it underdetermined and untestable against a fixed oracle.

**History:** The failure-mode-templates table was introduced with the LLM-driven binder at 83802db, where the transport row's `<message>` placeholder was already present and unpinned. be82734 added the rule routing structural `ContextOverflowError` into that same transport row without defining how an error variant lacking a `provider` field populates `<provider>`/`<message>`. f41af5d later added the custom-type-unsafe row carrying `<value>` without pinning its source or escaping. (A subsequent clarity pass, 2fb5783, pinned `<provider>` and `<ajv-summary>` explicitly but left `<message>` and `<value>` unpinned.)

## Solution approach

Clarify the [Failure-class taxonomy](#failure-class-taxonomy) Context-overflow-handling bullet and/or the [Failure-mode templates](#failure-mode-templates-normative) placeholder-pinning prose so a structural `ContextOverflowError` surfaced through the *Binder model transport failure* row has a normative source for both `<provider>` and `<message>`. Because `ContextOverflowError` carries no `provider` field, ground `<provider>` in the resolved binder-model `Model<Api>.api` already used by the existing transport-row pin. Whether to re-express the overflow as a `TransportError` before rendering or to map the placeholders directly from the `ContextOverflowError` plus the resolved binder model is the implementer's choice.

## Solution constraints

- The overflow-case `<provider>` value MUST stay consistent with the existing transport-row `<provider>` pin (resolved binder-model `Model<Api>.api`); do not introduce a separate provider-derivation for the overflow case.
- Out of scope: the general binder-path transport-row `<message>` source pin owned by F-2424.
