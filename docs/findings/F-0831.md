## Finding

The step-2 preamble mandates that the contributor record a per-item audit outcome for every item (a)–(ak) in the bump commit message, with verdict `pass` / `fail` / `N/A` (or `not-performed` for the SHOULD-level (f)–(aj) items) plus a one-line rationale, item (e) recording its two sub-outcomes (e.i)/(e.ii), item (e) `N/A` carrying the `N/A — superseded by <named Pi mechanism>` form, and item (e) `fail` carrying an additional second-pass outcome — declaring that "omitting the row is non-conformant" — but never defines what the structural unit it calls a "row" actually is. There is no statement of the record's keying (item letter, anchor ID, or prose heading), no field set (`outcome` / `rationale` / second-pass / sub-outcome columns), no surface form (markdown table, YAML block, fenced bullet list, free-prose paragraphs), and no ordering rule. With ~37 items, four verdict tokens, three special sub-shapes (item (e) two sub-outcomes; item (e) `N/A` superseded-by mechanism naming; item (e) `fail` second-pass), and the `not-performed` arm on (f)–(aj), the absence of a schema means two contributors writing genuinely conformant commit messages can produce records a reviewer cannot mechanically compare for "present / absent".

The downstream plan-side acceptance gate is explicit about this dependency: `V18c` *Ships when* requires editorial review to confirm "the bump commit message records a per-item outcome for every item `(a)`–`(ak)` (item `(e)` recording its two sub-outcomes) **in the shape [`version-bump-step2.md`](...) mandates**". The shape `version-bump-step2.md` actually mandates is `pass`/`fail`/`N/A`/`not-performed` plus a rationale string per item — a list of *contents*, not a layout — so the reviewer's "in the shape mandates" check decays to "did the contributor write some prose against each item?" with no mechanical handle on what counts as "the row for (af)" or whether a missing (af) row is detectable at all.

The fix is to specify, once in the step-2 preamble, the concrete layout of the record block: a single markdown structure (table or fenced list) whose primary key is the item letter (and `(e.i)`/`(e.ii)` for item (e)'s sub-outcomes, and a separate second-pass row for the item (e) `fail` arm), with explicit fields for `outcome` (one of the four verdict tokens) and `rationale` (one line, with the additional pinned forms — `N/A — superseded by <named Pi mechanism>` for item (e), and the pass-evidence-rule code-site citation for (e.i)/(e.ii) `pass` — surfaced as constraints on the rationale field rather than as separate prose). Once the record block has a single fixed shape, "omitting the row" becomes mechanically decidable, V18c's reviewer check becomes a present-set comparison against the closed item enumeration the same preamble owns, and a future leaf can wire a commit-message-lint gate against the same shape without re-litigating layout.

## Solution Space

**Shape:** single

### Recommendation

Define the per-item recording schema once in the step-2 preamble, immediately before the items list (and cross-referenced from the *Outcome recording conventions* sub-section so item (e)'s special-cases stay co-located with their parent shape):

1. **One record block per bump commit message**, fenced as a single markdown construct so a reviewer can locate it by structure rather than by prose scan. Keep the fence choice to a single canonical form across the corpus (recommended: a labelled fenced code block such as ` ```pi-bump-audit ` containing one line per item, or a markdown table — pick one and pin it; both work, but the spec must pick).
2. **Primary key = item letter** (`(a)`, `(b)`, …, `(ak)`), exactly as the items appear in the step-2 list, in the same order. The reviewer-side present-set check becomes a string-equality comparison against the closed item enumeration the preamble already owns.
3. **Fields per record:** `outcome` (exactly one of `pass`, `fail`, `N/A`, `not-performed`) and `rationale` (a single-line string). Specify that the rationale carries the additional pinned wording the existing prose already requires — `N/A — superseded by <named Pi mechanism>` for item (e), the candidate-minor code-site citation for (e.i)/(e.ii) `pass` per the pass-evidence rule — as content constraints on the `rationale` field, not as separate columns.
4. **Item (e) sub-outcomes** are recorded as two additional records keyed `(e.i)` and `(e.ii)`, on the same footing as a top-level letter; the `(e)` line itself is not required (the two sub-records carry the verdict). State this explicitly so a contributor does not record three rows where two are mandated.
5. **Item (e) `fail`-arm second-pass** is recorded as one additional record keyed `(e.i second-pass)` / `(e.ii second-pass)` (whichever sub-outcome failed), present iff the corresponding original is `fail`; absent otherwise. The merge-gate prose already in the item (e) recovery paragraph reads against the second-pass row, so the key shape must be cite-able from that paragraph.
6. **Omission predicate:** "the row for item `X` is omitted" is defined as: no record whose primary key string-equals `X` (or `(X.i)`/`(X.ii)` for item (e)'s sub-outcomes) appears in the bump commit's record block. Pin this definition once, so the existing "omitting the row is non-conformant" sentence routes to it.

Implementer edge cases to surface in the same edit: a record block accidentally placed outside the commit-message body (e.g. in a follow-up commit) does not satisfy the obligation; ordering within the block follows the step-2 item enumeration order (so a reviewer comparing two bumps reads parallel rows); the `not-performed` verdict's rationale field is mandatory (the current preamble already implies this — re-pin it under the schema for clarity).

A future leaf MAY land a commit-message-lint gate that mechanically parses the record block against the schema above; the schema being pinned here is the prerequisite, not the gate itself.

## Consequence

**Severity:** correctness

Two reasonable implementers writing fully conformant bump commit messages can produce records that disagree on layout, keying, and field placement to the point where V18c's editorial-review acceptance check — "in the shape `version-bump-step2.md` mandates" — has no mechanical predicate to evaluate against. A missing (af) row is structurally undetectable, defeating the "omitting the row is non-conformant" rule the same preamble pins; a reviewer who notices the gap cannot cite which clause was violated. The defect compounds with the item count (~37 items, 4 verdict tokens, item-(e) sub-shapes, second-pass arm), since each new item the checklist accretes widens the surface across which contributors must independently invent the same conventions.
