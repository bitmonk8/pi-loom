## Problem

The `#classifier-input-surface` paragraph and the provider overflow table leave three input edges with no determinate disposition. A non-cancellation `complete()` rejection (TCP/TLS error, DNS failure, mid-stream truncation surfaced as rejection, pi-ai-adapter throw on a malformed response) carries no `AssistantMessage` and matches no classifier branch, so it escapes unclassified — even though `binder-inference.md` already promises such rejections route through this mapping table. When `onResponse` fires more than once within a single `complete()` invocation (the SDK-driven retry path), which `ProviderResponse` feeds the classifier is undefined, and the choice flips `TransportError.retryable`. An `amazon-bedrock` overflow matches both the overflow-table row (→ `ContextOverflowError`) and the no-`onResponse` network-level rule (→ `TransportError`) with no stated precedence, so two conforming implementers disagree on the resulting `QueryError.kind` for the spec's only documented SDK-only overflow path.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `44782e9` ("pi-loom spec: resolve \"TransportError catch-all / pi-ai provider-error surface\"", 2026-06-06)
**History:** `git log --diff-filter=A --follow` puts `provider-error-mapping.md` into existence at `f5e89f4` (the spec-set sharding commit), but the three edges in this finding were all introduced together at `44782e9`, which added the `#classifier-input-surface` paragraph in a single hunk. That hunk simultaneously (i) split the classifier inputs into "HTTP status / body wording / no-HTTP-response", which is the surface that omits the rejection case; (ii) introduced the "joins the captured `{status, headers}` with the resolved `AssistantMessage`, keyed per `complete()` invocation" wording that under-specifies multi-fire `onResponse`; and (iii) added the SDK-only-provider "network-level" carve-out for amazon-bedrock that overlaps the existing `ValidationException` overflow-table row. No earlier commit defined any of the three edges differently — they have never been resolved in the corpus. `git log -S "onResponse did not fire"`, `git log -S "joins the captured"`, and `git log -S "ValidationException"` all corroborate single-commit introduction at `44782e9`.

## Solution approach

Within `provider-error-mapping.md`, specify which `onResponse` firing's `{ status, headers }` the classifier retains when a single `complete()` invocation fires it more than once (the SDK-driven retry path), so `TransportError.http_status` is determinate. Establish that the overflow-signature match takes precedence over the no-`onResponse` network-level fallback, so an `amazon-bedrock` `stopReason: "error"` resolution whose `errorMessage` matches an overflow signature classifies as `ContextOverflowError` rather than `TransportError`. Extend the no-HTTP-response network-level branch at `#transport-error-retryable` to also cover non-cancellation `complete()` rejections, leaving the `signal`-driven cancellation route at [`#complete-retry-and-cancellation-presupposition`](./conversation-drive.md#complete-retry-and-cancellation-presupposition) unchanged.

## Solution constraints

- Route the rejection branch through the existing `TransportError { retryable: true, http_status: null }` shape; do not introduce a new `QueryError` variant.
