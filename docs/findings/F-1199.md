## Finding

The `#classifier-input-surface` paragraph and the provider table together specify how an `AssistantMessage` value gets classified into a `QueryError` variant, but the three input edges below have no determinate disposition.

1. **Rejected `complete()` promises (non-cancellation).** The classifier reads `ProviderResponse.status` (from `onResponse`), `AssistantMessage.errorMessage`, and `AssistantMessage.stopReason`. Every classifier branch â€” the HTTP table, the `TransportError.retryable` rule, the no-HTTP-response "network-level" arm, and stop-reason classification â€” assumes `complete()` *resolved* with an `AssistantMessage`. The "network-level" arm is explicitly "the case where `onResponse` did not fire before `complete()` resolved with `AssistantMessage.stopReason: "error"`". A `complete()` *rejection* (TCP/TLS error, DNS failure, mid-stream truncation surfaced as rejection, pi-ai-adapter throw on a malformed provider response, server-side connection reset, etc.) carries no `AssistantMessage` at all and therefore matches no branch. The cancellation route covers only the `options.signal`-driven rejection pinned at `#complete-retry-and-cancellation-presupposition`; every other rejection escapes unclassified. `binder-inference.md` already promises such rejections are "classified through the Provider error mapping table exactly as the typed and untyped query paths classify provider failures", so the gap is at the classifier, not the caller.

2. **Multiple `onResponse` firings.** The classifier "joins the captured `{ status, headers }` with the resolved `AssistantMessage`, keyed per `complete()` invocation". With `complete()` driving inter-attempt retries internally from `StreamOptions.maxRetries` / `maxRetryDelayMs` (per the *retry-timing presupposition*), a single `complete()` invocation can produce more than one `onResponse` firing â€” a 503 attempt followed by a 200 success, a 429 followed by a 500, etc. The keying-per-invocation rule disambiguates *across* concurrent calls but not *within* a multi-fire call. Which `ProviderResponse` feeds the classifier is undefined, and the choice flips outcomes: a 5xx-then-4xx sequence resolves to `retryable: true` if first wins and `retryable: false` if last wins; a 429-then-200 success that nonetheless produced a body-envelope error chooses between `retryable: true` (from the 429) and `retryable: false` (from the 200 + body envelope).

3. **Overflow vs network-level precedence on bedrock.** The amazon-bedrock row matches "`ValidationException` with body matching `/(input is too long|context window)/i`" with no HTTP-status precondition; the *classifier input surface* paragraph also calls out amazon-bedrock as the canonical SDK-only provider whose adapter "resolves with `stopReason: "error"` without an `onResponse` invocation" and is therefore "classified as network-level". A real bedrock overflow rides exactly that path â€” no `onResponse`, resolved with `stopReason: "error"`, `errorMessage` carrying the `ValidationException` body. Both the overflow-table row (â†’ `ContextOverflowError`) and the network-level rule (â†’ `TransportError { retryable: true, http_status: null }`) match. The two have no stated precedence, so two conforming implementers will disagree on the resulting `QueryError.kind` for the most common bedrock failure mode the spec actually anticipates.

## Solution Space

**Shape:** single

### Recommendation

Make the following three edits, all within `provider-error-mapping.md` `#classifier-input-surface` (the overflow-precedence clause may instead live in the overflow-table preamble). Author them in the order below: the multi-`onResponse` join sentence first â€” it pins which `ProviderResponse` feeds the classifier â€” then the overflow-precedence sentence on top of that disambiguated input surface, and the rejection-mapping sub-clause last, since it widens the network-level branch and must reference the cancellation carve-out.

**1. Multi-`onResponse` join rule (last firing wins).** Immediately after the "keyed per `complete()` invocation" clause, add: "When `onResponse` fires more than once for a single `complete()` invocation (the SDK-driven `maxRetries` retry path), the classifier retains the last firing's `{ status, headers }` and discards earlier ones." The SDK reports the final attempt, so the final HTTP context is the matching one; this needs no schema change. When an SDK-driven retry produces two attempts with different statuses (e.g. 503 â†’ 502) the `retryable` outcome is identical, but `TransportError.http_status` must still observe the *last* status so the rule is verifiable.

**2. Overflow-table match takes precedence over the no-`onResponse` network-level arm.** Add a precedence clause: "The overflow-signature match in the table above is evaluated before the no-HTTP-response network-level fallback. An `AssistantMessage` that resolves with `stopReason: "error"`, fired no prior `onResponse`, and whose `errorMessage` matches an overflow signature is classified as `ContextOverflowError` per that row, not as `TransportError { retryable: true, http_status: null }`." The amazon-bedrock row carries no HTTP-status precondition, so this precedence is the only edit needed; it preserves `ContextOverflowError` detection on bedrock and the documented `tokens_used = tokens_limit = null` fallback. The precedence applies to both openai-completions overflow rows â€” the HTTP 400 row *and* the HTTP 200 `stopReason: "error"` row â€” not only the HTTP-200 case. The overflow-signature regex is narrow, so the precedence cannot mask a true network failure with overflow-shaped wording.

**3. Map non-cancellation `complete()` rejection to `TransportError { retryable: true, http_status: null }`.** Extend the network-level branch so it covers both shapes of "no usable resolution":

> The no-HTTP-response (network-level) class covers two paths: (i) `complete()` resolved with `AssistantMessage.stopReason: "error"` and no prior `onResponse`; and (ii) `complete()` rejected with any reason other than the `options.signal` abort cancellation pinned at [`complete()` retry-timing and cancellation presupposition](./conversation-drive.md#complete-retry-and-cancellation-presupposition) (the rejection covers TCP/TLS errors, DNS failures, end-of-stream truncation surfaced as rejection, and pi-ai-adapter throws on a malformed provider response). Both paths route through `TransportError { retryable: true, http_status: null }`. The `signal`-driven rejection continues to route through `CancelledError` per [Cancellation](../cancellation.md), unchanged by this rule.

Source the rejection's `TransportError.message` from the rejection's `Error.message`, with a fixed `"provider transport failure"` fallback when the rejection has no readable message, mirroring the absent-`errorMessage` fallback the prompt-mode mapping already uses. Reuse the existing `TransportError { retryable: true, http_status: null }` shape â€” no new `QueryError` variant. Optionally cite this branch explicitly from `binder-inference.md`'s "a rejected promise â€¦ is classified through the Provider error mapping table" sentence; that is citation polish, not a behavioural change. The cancellation carve-out must use the `signal`-aborted determination the SDK pins, not a `loomAbort.signal.aborted` check at classify time â€” the abort may fire *after* the rejection, in which case the rejection is genuinely a network failure and the call-site cancellation short-circuit handles the abort separately. A provider rejection whose underlying cause is a context-overflow that the SDK routed to rejection rather than to a resolved `stopReason: "error"` is classified as `TransportError`, consistent with the existing provider-owned-wording presupposition.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge on three observable surfaces: (a) what `QueryError.kind` a non-cancellation `complete()` rejection produces (silent crash vs. `TransportError`); (b) which retry attempt's `ProviderResponse` populates `TransportError.retryable` on multi-fire `onResponse`; (c) whether a bedrock overflow produces `ContextOverflowError` (with the `tokens_used` / `tokens_limit` contract) or `TransportError { retryable: true }`. The bedrock case is the most damaging â€” it misroutes the spec's only documented overflow path for an SDK-only provider, defeating the `ContextOverflowError` detection the binder, query, and `code-registry-runtime` paths all branch on.
