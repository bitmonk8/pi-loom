## Finding

The `#classifier-input-surface` paragraph and the provider table together specify how an `AssistantMessage` value gets classified into a `QueryError` variant, but the three input edges below have no determinate disposition.

1. **Rejected `complete()` promises (non-cancellation).** The classifier reads `ProviderResponse.status` (from `onResponse`), `AssistantMessage.errorMessage`, and `AssistantMessage.stopReason`. Every classifier branch — the HTTP table, the `TransportError.retryable` rule, the no-HTTP-response "network-level" arm, and stop-reason classification — assumes `complete()` *resolved* with an `AssistantMessage`. The "network-level" arm is explicitly "the case where `onResponse` did not fire before `complete()` resolved with `AssistantMessage.stopReason: "error"`". A `complete()` *rejection* (TCP/TLS error, DNS failure, mid-stream truncation surfaced as rejection, pi-ai-adapter throw on a malformed provider response, server-side connection reset, etc.) carries no `AssistantMessage` at all and therefore matches no branch. The cancellation route covers only the `options.signal`-driven rejection pinned at `#complete-retry-and-cancellation-presupposition`; every other rejection escapes unclassified. `binder-inference.md` already promises such rejections are "classified through the Provider error mapping table exactly as the typed and untyped query paths classify provider failures", so the gap is at the classifier, not the caller.

2. **Multiple `onResponse` firings.** The classifier "joins the captured `{ status, headers }` with the resolved `AssistantMessage`, keyed per `complete()` invocation". With `complete()` driving inter-attempt retries internally from `StreamOptions.maxRetries` / `maxRetryDelayMs` (per the *retry-timing presupposition*), a single `complete()` invocation can produce more than one `onResponse` firing — a 503 attempt followed by a 200 success, a 429 followed by a 500, etc. The keying-per-invocation rule disambiguates *across* concurrent calls but not *within* a multi-fire call. Which `ProviderResponse` feeds the classifier is undefined, and the choice flips outcomes: a 5xx-then-4xx sequence resolves to `retryable: true` if first wins and `retryable: false` if last wins; a 429-then-200 success that nonetheless produced a body-envelope error chooses between `retryable: true` (from the 429) and `retryable: false` (from the 200 + body envelope).

3. **Overflow vs network-level precedence on bedrock.** The amazon-bedrock row matches "`ValidationException` with body matching `/(input is too long|context window)/i`" with no HTTP-status precondition; the *classifier input surface* paragraph also calls out amazon-bedrock as the canonical SDK-only provider whose adapter "resolves with `stopReason: "error"` without an `onResponse` invocation" and is therefore "classified as network-level". A real bedrock overflow rides exactly that path — no `onResponse`, resolved with `stopReason: "error"`, `errorMessage` carrying the `ValidationException` body. Both the overflow-table row (→ `ContextOverflowError`) and the network-level rule (→ `TransportError { retryable: true, http_status: null }`) match. The two have no stated precedence, so two conforming implementers will disagree on the resulting `QueryError.kind` for the most common bedrock failure mode the spec actually anticipates.

## Solution Space

**Shape:** multiple

The finding bundles three independent obligations. Per the bimodal-obligation rule each gets its own option, and the recommendation pins the order: edges 2 and 3 are stable text-only clarifications inside `#classifier-input-surface`; edge 1 introduces a new mapping that should land on top of those clarifications.

### Option A — Multi-`onResponse` join rule: last-fire wins

**Approach.** Add one sentence to `#classifier-input-surface` immediately after the "keyed per `complete()` invocation" clause: "When `onResponse` fires more than once for a single `complete()` invocation (the SDK-driven `maxRetries` retry path), the classifier retains the last firing's `{ status, headers }` and discards earlier ones."

**Spec edits.** `provider-error-mapping.md` `#classifier-input-surface` only.

**Pros.** Aligns with the resolved `AssistantMessage` semantics (the SDK reports the final attempt, so the final HTTP context is the matching one); no schema change; trivially testable with a fixture that drives 503-then-200.

**Cons.** Loses the information that a transient 5xx preceded the success — but the spec already commits to classifying off the resolved value, so this is consistent with the existing posture.

**Risks.** None beyond fixture coverage.

### Option B — Overflow-table match takes precedence over the no-`onResponse` network-level arm

**Approach.** Add a precedence clause to `#classifier-input-surface` (or to the table preamble): "The overflow-signature match in the table above is evaluated before the no-HTTP-response network-level fallback. An `AssistantMessage` that resolves with `stopReason: "error"`, fired no prior `onResponse`, and whose `errorMessage` matches an overflow signature is classified as `ContextOverflowError` per that row, not as `TransportError { retryable: true, http_status: null }`." The amazon-bedrock row already has no HTTP-status precondition, so the precedence is the only edit needed.

**Spec edits.** `provider-error-mapping.md` `#classifier-input-surface` (or table preamble) only.

**Pros.** Preserves the spec's commitment to `ContextOverflowError` detection on bedrock; matches the documented `tokens_used = tokens_limit = null` fallback for bedrock; symmetric to the openai-completions HTTP-200 row, which the spec already classifies as overflow rather than HTTP-200 body-envelope `TransportError`.

**Cons.** Adds a precedence rule to a paragraph the audit has already flagged as dense.

**Risks.** None — the overflow-signature regex is narrow, so the precedence cannot mask a true network failure with overflow-shaped wording.

### Option C — Map non-cancellation `complete()` rejection to `TransportError { retryable: true, http_status: null }`

**Approach.** Extend the network-level branch in `#classifier-input-surface` so it covers both shapes of "no usable resolution":

> The no-HTTP-response (network-level) class covers two paths: (i) `complete()` resolved with `AssistantMessage.stopReason: "error"` and no prior `onResponse`; and (ii) `complete()` rejected with any reason other than the `options.signal` abort cancellation pinned at [`complete()` retry-timing and cancellation presupposition](./conversation-drive.md#complete-retry-and-cancellation-presupposition) (the rejection covers TCP/TLS errors, DNS failures, end-of-stream truncation surfaced as rejection, and pi-ai-adapter throws on a malformed provider response). Both paths route through `TransportError { retryable: true, http_status: null }`. The `signal`-driven rejection continues to route through `CancelledError` per [Cancellation](../cancellation.md), unchanged by this rule.

The rejection's `TransportError.message` is sourced from the rejection's `Error.message` (with a fixed `"provider transport failure"` fallback when the rejection has no readable message), mirroring the absent-`errorMessage` fallback the prompt-mode mapping already uses.

**Spec edits.** `provider-error-mapping.md` `#classifier-input-surface`. Optionally, soften `binder-inference.md`'s "a rejected promise … is classified through the Provider error mapping table" sentence to cite the new branch explicitly, but that is a citation polish, not a behavioural change.

**Pros.** Closes the gap `binder-inference.md` already asserts is closed; keeps the rejection-to-`Err` contract honest; reuses the existing `TransportError { retryable: true, http_status: null }` shape — no new `QueryError` variant.

**Cons.** Requires an explicit carve-out for the `options.signal` cancellation rejection (otherwise the cancellation path becomes ambiguous with the new rejection branch).

**Risks.** A provider rejection whose underlying cause is a context-overflow that the SDK formatter routed to rejection rather than to a resolved `stopReason: "error"` would be classified as `TransportError`, not `ContextOverflowError`. This is consistent with the existing "Provider-owned-wording presupposition" — provider-side overflow wording moving fields is already routed to editorial review per the `npm test` overflow fixtures.

### Recommendation

Apply all three options; they are independent and cannot be merged into a single coherent edit without confusing the reader. Land them in the order **A → B → C**:

- **A** first, because the multi-`onResponse` join is the smallest insertion and pins the "which `ProviderResponse` feeds the classifier" reading the table and `retryable` rule both quietly assume.
- **B** second, because it is one declarative precedence sentence that lands on top of A's already-disambiguated input surface.
- **C** last, because it widens the network-level branch with a new sub-clause that must reference the cancellation carve-out — easier to author once A and B have stabilised the surrounding paragraph.

Edge cases the implementer must watch:

- Edge for A: an SDK-driven retry where the *first* attempt is a 5xx and the *second* attempt is the same 5xx but with a different status (e.g. 503 → 502) — both attempts produce `retryable: true`, so the choice is moot, but the fixture should still observe the *last* status in `TransportError.http_status` to make the rule verifiable.
- Edge for B: openai-completions has both an HTTP 400 overflow row *and* an HTTP 200 `stopReason: "error"` overflow row; the precedence rule must apply to both, not only the HTTP-200 case.
- Edge for C: the cancellation carve-out must use the `signal`-aborted determination the SDK pins, not a `loomAbort.signal.aborted` check at classify time — the abort may have fired *after* the rejection, in which case the rejection is genuinely a network failure and the existing cancellation short-circuit at the call site handles the abort separately.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge on three observable surfaces: (a) what `QueryError.kind` a non-cancellation `complete()` rejection produces (silent crash vs. `TransportError`); (b) which retry attempt's `ProviderResponse` populates `TransportError.retryable` on multi-fire `onResponse`; (c) whether a bedrock overflow produces `ContextOverflowError` (with the `tokens_used` / `tokens_limit` contract) or `TransportError { retryable: true }`. The bedrock case is the most damaging — it misroutes the spec's only documented overflow path for an SDK-only provider, defeating the `ContextOverflowError` detection the binder, query, and `code-registry-runtime` paths all branch on.
