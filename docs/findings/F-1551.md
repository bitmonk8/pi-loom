## Finding

The Depth Enforcement section of `schema-subset.md` closes its narrative with an implementation directive: *"Implementation is a recursive descent over the parsed JSON value with a depth counter; the first node whose depth would exceed `5` short-circuits and produces the failure."* This pins a specific traversal strategy (recursive descent + a running counter) onto the implementer.

The observable contract is already fully specified by the surrounding normative material: the **Counting algorithm** (scalar/empty = depth 1; non-empty object/array = 1 + max child depth; `anyOf` arms are not levels; cap `depth ≤ 5`), the four worked examples, the "runs **before** AJV at each site" ordering, the five enforcement sites, and the error shape whose `path` is a *JSON Pointer to the first too-deep node*. Every caller-observable property — the accept/reject boundary, the short-circuit-on-first-violation behaviour, and the pointer to the offending node — is determined by those clauses independent of how the walk is coded.

"Recursive descent with a depth counter" therefore adds no behavioural constraint while forbidding equivalent implementations (an explicit work-stack iterative traversal, a streaming-parse depth tracker, etc.) that satisfy the same observable contract. It is implementation guidance miscast as normative prose inside a normative section.

## Issue introduction

**Verdict:** introduced-in-resolution
**Introducing commit:** `216130d` — "pi-loom spec: resolve \"Depth ≤5: counting algorithm and enforcement point unspecified\"" (2026-05-04)
**History:** The Depth Enforcement section did not exist before `216130d`; the depth cap was a single bullet ("Depth ≤ 5 levels of nesting at runtime …") in the subset list. The resolving commit added the full § Depth Enforcement block — counting algorithm, worked examples, enforcement points, error shape, edge cases — and in the paragraph that establishes the AJV-ordering contract appended "Implementation is a recursive descent over the parsed JSON value with a depth counter; the first node whose depth would exceed `5` short-circuits and produces the failure." `git log -L` and `git log -S "recursive descent"` confirm the phrase was introduced in that single commit and has not been edited since (later touches to the surrounding paragraph — `7f217a8`, `f7a505e`, `ba68984` — extended the error-shape and per-boundary table without altering this sentence). The defect is a classic over-fix: while pinning the observable contract the resolution also reached for an implementation strategy, mixing two registers in one sentence.

## Solution Space

**Shape:** single

### Recommendation

Delete the sentence *"Implementation is a recursive descent over the parsed JSON value with a depth counter; the first node whose depth would exceed `5` short-circuits and produces the failure."* from the Enforcement-point paragraph. If the short-circuit-on-first-violation intent must be retained explicitly, keep only the behavioural half — that the walk reports the first node exceeding depth 5 (which the error shape's "JSON Pointer to first too-deep node" already implies) — and drop the "recursive descent … with a depth counter" mechanism. If any implementation-strategy hint is still wanted, relocate it verbatim to a `*(Non-normative implementation note.)*` so it carries no normative force.

## Consequence

**Severity:** cosmetic

No observer is affected: the accept/reject boundary, short-circuit semantics, and first-too-deep-node pointer are all pinned elsewhere. The sentence only removes implementer freedom over an internal traversal choice that produces identical observable results.
