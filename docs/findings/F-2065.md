## Finding

GOV-23 fixes the anchor-scheme stability rule for the Session Model `sm-N-…` anchors with the clause: *"a future split of an SM-N appends new sub-letters rather than renumbering existing siblings."* The clause states what the split does **not** do (renumber siblings) but leaves underspecified what it **does** do at the anchor level, and two readings diverge:

- **(a)** A split decomposes the SM-N into per-sub-obligation sub-letter anchors of that same integer — e.g. splitting an atomic SM-1 yields `sm-1a` / `sm-1b`, with the bare `sm-1` becoming (or being retired in favour of) the umbrella anchor.
- **(b)** A split allocates a fresh top-level integer (`sm-(next)`) for the new obligation, leaving the original SM-N and its anchor untouched.

The rule's own wording ("sub-letters") points toward (a), but the gap is sharpest for the case the clause does not address: splitting an SM-N that currently has **no** sub-letters. Under reading (a) the bare `sm-N` anchor is repurposed into an umbrella over freshly-coined `sm-Na` / `sm-Nb` — which sits in tension with the same rule's "A published `sm-N-…` anchor MUST NOT be reused for a different obligation," since the atomic obligation's anchor would now denote an umbrella. The in-corpus precedents (`sm-3` → `sm-3a`/`sm-3b`, `sm-7` → `sm-7a`…`sm-7e` in `session-model-and-appendix.md`) already pair an umbrella anchor with sub-letters, but GOV-23 does not connect those examples to the split clause, nor state the fate of the bare anchor or the rule for adding a genuinely new (non-split) top-level obligation.

The rule binds the spec corpus itself (GOV-18 arm (b)); the cost of the ambiguity is borne by future maintainers performing Session Model anchor-site edits, not by runtime implementers.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** af9288d — pi-loom spec: resolve "editorial and governance meta-commentary misplaced in orientation" (2026-06-03, spec maintainer)
**History:** The rule entered in `40ffdc5` (T19a, eight SM-N sub-units) as a `spec.md` paragraph whose split clause carried a disambiguating worked example: *"a future split of an SM-N (e.g. the in-place `sm-3` → `sm-3a`/`sm-3b` and `sm-7` → `sm-7a` … `sm-7e` decompositions already below) appends new sub-letters…"*. Commit `af9288d` relocated the rule from `spec.md` into GOV-23 on the governance page and, in doing so, dropped the parenthetical worked example from the split clause, leaving the bare "appends new sub-letters" with no illustration of what a split does to the anchors — which is the ambiguity reported here.

## Solution Space

**Shape:** single

### Recommendation

Restore the disambiguating mechanic the relocation dropped: in the GOV-23 *Anchor-scheme stability* clause, make explicit that a split of an SM-N decomposes it into per-sub-obligation sub-letter anchors (`sm-Na`, `sm-Nb`, …) of the same integer — citing the existing `sm-3` → `sm-3a`/`sm-3b` and `sm-7` → `sm-7a`…`sm-7e` decompositions as the worked pattern — rather than allocating a new top-level integer. State the fate of the bare `sm-N` anchor on a split (it is preserved as the umbrella/orientation anchor over the new sub-letters, consistent with the "MUST NOT be reused for a different obligation" rule, since an umbrella over an obligation's own sub-parts is not a different obligation), and, if the corpus intends new *independent* obligations to take the next top-level integer, say so as a separate sentence so the two cases are not conflated.

## Consequence

**Severity:** advisory

Two maintainers splitting a previously-atomic Session Model obligation could diverge on the resulting anchor form — one minting `sm-Na`/`sm-Nb` sub-letters (repurposing the bare `sm-N`), another allocating a new top-level `sm-(next)` — and on whether the original bare anchor survives, is retired, or becomes an umbrella. The existing `sm-3`/`sm-7` precedents make divergence unlikely for a careful reader but the rule does not state the mechanic, so the stability guarantee GOV-23 exists to provide is weaker than intended.
