## Finding

The two sibling `loom/load/*` diagnostic codes that the strict-capability probe surfaces describe complementary states of the same indicator on `Model<Api>.strictCapable`, but they use two different naming structures:

- `loom/load/binder-model-not-strict-capable` — *negated-adjective* form, fires when `strictCapable === false`.
- `loom/load/binder-model-strict-capability-unknown` — *qualified-noun* form, fires when `strictCapable` is absent.

The two codes are introduced one row apart in the `loom/load/*` registry table on `docs/spec_topics/diagnostics/code-registry-load.md` and are cross-referenced together everywhere the probe is described (`binder/binder-model-and-context.md` §Strict-capability requirement, `frontmatter/frontmatter-fields-a.md` `bind_model` row, `pi-integration-contract/host-prerequisites.md` §2, `pi-integration-contract/capability-inventory-items.md` capability 7, `pi-integration-contract/audit-recognised-shapes.md` strict-capability-absence-pin, and `implementation-notes.md`). Aligning their structure to a single shape — `binder-model-strict-capability-{false,unknown}` — makes the relationship between the two codes immediately legible at every cite site and removes the only naming irregularity in an otherwise consistent `binder-model-*` family (`binder-model-unresolved`, the two strict-capability codes).

A pre-loom-1.0 rename is cheap: the codes are wire identifiers, but no runtime or test currently consumes them — the only consumers are the spec, the V11a plan leaf, and the V11a-T test obligations, all of which the rename re-points in lockstep with the registry row.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `0fa81fa` ("spec: pin binder model resolution; document envelope two-arm rationale (Decisions #5, #6)", 2026-05-05) and `022bff9` ("pi-loom spec: resolve 'Orientation prerequisites + Host runtime + pi-integration-contract Host prerequisites'", 2026-05-06)
**History:** Commit `0fa81fa` introduced the E-level code under the negated-adjective name `loom/load/binder-model-not-strict-capable` as part of pinning the binder-model resolution chain (Decisions #5/#6). The next day, `022bff9` added the companion W-level code under the qualified-noun name `loom/load/binder-model-strict-capability-unknown` while resolving the Pi-integration-contract host-prerequisites finding — that commit added §2 *Binder model* to `spec_topics/pi-integration-contract.md` and surfaced the absent-indicator branch under a noun-phrase shape. Neither commit considered the pair as a unit; each code is internally well-formed and the asymmetry only becomes visible when the two rows sit side-by-side in the `loom/load/*` registry table. Subsequent refactors (the spec split `f5e89f4`, the strict-capability-absence-pin consolidation `6829a79`) preserved both names verbatim.

## Solution Space

**Shape:** single

### Recommendation

Rename `loom/load/binder-model-not-strict-capable` to `loom/load/binder-model-strict-capability-false`, mirroring the noun-phrase shape of its sibling `loom/load/binder-model-strict-capability-unknown`. Concretely:

1. In `docs/spec_topics/diagnostics/code-registry-load.md`, change the code-id cell and update the Message column from `binder model '<model>' is not flagged as strict-structured-output capable` to `binder model '<model>' strict-structured-output capability is false`.
2. Re-point every spec cross-reference to the new code id: `binder/binder-model-and-context.md` (§Binder model parse rule and §Strict-capability requirement), `frontmatter/frontmatter-fields-a.md` (`bind_model` row), `discovery/package-and-settings.md` (`looms.binderModel` scalar-key paragraph), and `implementation-notes.md` (binder-model load-time gate sentence).
3. Re-point the two plan leaves `V11a` and `V11a-T` and the `cka-38` coverage-matrix row.

Do **not** rename the W-level `loom/load/binder-model-strict-capability-unknown`: it is already the noun-phrase shape and is the universal-production code under the loom 1.0 Pi-SDK pin (anchored at `pi-integration-contract/audit-recognised-shapes.md#strict-capability-absence-pin`), so it carries strictly more reader load than the E-level code.

Edge cases the implementer must watch:

- The diagnostic-codes registry is wire-contract per DIAG-3; rename it in a single commit so no intermediate state ships a hybrid pair.
- `binder-model-unresolved` is unaffected — it names a *resolution* outcome, not a *capability* outcome, and reads naturally as a noun phrase already.

## Consequence

**Severity:** cosmetic

The two codes are functionally unambiguous today; readers and implementers can follow the cross-references. The cost is one extra glance at each of the ~8 cite sites to confirm that the negated-adjective form and the qualified-noun form refer to the same probe. No observer behaviour differs.
