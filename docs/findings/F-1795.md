## Finding

The `.warp` file rules in `imports.md` twice anchor execution context to "the *calling* `.loom`'s conversation":

- The query bullet: "A query inside an imported function executes against the *calling* `.loom`'s conversation when the function is invoked."
- The `invoke(...)` bullet: "the invocation executes against the *calling* `.loom`'s conversation … exactly like a `@`...`` query inside a warp function."

Both formulations implicitly assume the function's *direct* caller is a `.loom` file. That assumption fails for a warp-to-warp call chain: in `A.loom` → `A.warp` `fn_a` → `B.warp` `fn_b`, where `fn_b` contains a query or `invoke(...)`, the direct caller of `fn_b` is a `.warp` function, not a `.loom` file. The phrase "the calling `.loom`" then has no referent at the firing site, leaving two readings open: (a) "calling `.loom`" denotes the `.loom` at the root of the current execution chain (the only conversation actually in existence, since `.warp` files have no conversation of their own), or (b) the phrase is simply undefined whenever an intervening `.warp` `fn` frame separates the query/invoke from any `.loom`.

Operationally there is only ever one conversation in play — the one owned by the executing `.loom` that initiated the chain — so reading (a) is the only behaviour that can actually be implemented. The defect is that the spec's wording names the *direct* caller rather than the conversation's owner, so a literal reader following reading (b) could treat warp-to-warp queries as an error or unspecified case. The phrasing should pin the conversation to the `.loom` at the root of the current `.warp` call chain, independent of how many `.warp` `fn` frames intervene.

## Solution Space

**Shape:** single

### Recommendation

Reword both `.warp`-rule bullets so the conversation is bound to the `.loom` at the root of the current execution chain rather than to the direct caller. A `.warp` `fn` has no conversation of its own; a query or `invoke(...)` inside it executes against the conversation of the `.loom` whose execution initiated the chain, regardless of how many intervening `.warp` `fn` frames sit between the `.loom` and the firing site.

Concretely, replace "the *calling* `.loom`'s conversation" with a phrase such as "the conversation of the `.loom` at the root of the current `.warp` call chain" (or "the active `.loom`'s conversation") in both the query bullet and the `invoke(...)` bullet, and add one clause stating that warp-to-warp nesting (`.loom` → `.warp` `fn` → `.warp` `fn`) does not introduce a new conversation — every query/invoke inside any `.warp` `fn` in the chain targets that same root `.loom` conversation. Edge case the implementer must keep aligned: a callee resolved as subagent-mode still spawns a fresh isolated conversation (already stated for the `invoke(...)` bullet); the clarification only fixes the *non-isolated* default path.

## Consequence

**Severity:** advisory

A reader taking the phrase literally finds no `.loom` direct caller for a query or `invoke(...)` inside a `.warp` `fn` that was reached from another `.warp` `fn`, and could treat warp-to-warp queries as unspecified or erroneous. Because only one conversation (the root `.loom`'s) ever exists at runtime, a working implementation is still reachable, but the wording invites divergent handling of the warp-to-warp case.
