## Finding

PIC-12 carries a runtime MUST that "the runtime MUST NOT call `Date.now`, `performance.now`, `Date.prototype.getTime`, or the global `setTimeout` / `clearTimeout` outside the `WallClock` adapter." The only enforcement mechanism the spec then names â€” a build-time grep-test, parallel to the `process.env.HOME` ban for `homedir()` â€” is immediately disclaimed in the same sentence as "a non-normative implementation note â€¦ the grep-test mechanism is an implementation choice, not an observable conformance point." The MUST therefore stands with no normative acceptance criterion of any kind: no behavioural observation through `FakeClock`, no static-analysis obligation that any conforming implementation must run, and no diagnostic that fires on violation.

The downstream cost is not abstract. The same paragraph already records that `FakeClock` is the seam that makes the watcher's 250 ms debounce, the `SHUTDOWN_AWAIT_CAP_MS` shutdown await, the `looms.scanPackagesTimeoutMs` deadline math, and the `loop-iter` macrotask yield deterministically testable. Every one of those conformance tests assumes that no runtime path reaches around the seam to a real `Date.now` / `performance.now` / `setTimeout`. A violation outside the `WallClock` adapter silently corrupts the timing assertion at the *consuming* test â€” the violating site is invisible at its own site and only surfaces as a flaky timer- or deadline-bounded test elsewhere â€” because nothing in the spec turns the ban itself into a green-or-red observation.

The defect is symmetrical with the `process.env.HOME` ban PIC-12 cites as its parallel, which has the same surface (DISC-1 / PIC-13 `homedir()`) and the same self-disclaimed enforcement â€” a separately-raised finding in this review flags the DISC-1 env-var ban as a co-bundled obligation, but neither side currently pins what a conforming implementation must do to demonstrate the ban holds. The spec needs either an observable acceptance criterion (a static-analysis assertion any conforming implementation must run, with a stated failure shape) or an explicit demotion of the MUST to an architectural / contributor-discipline note that does not pretend to bind conformance.

## Solution Space

### Recommendation

Promote the PIC-12 timing-primitive ban from a self-disclaimed grep-test note to a normative static-analysis acceptance criterion stated on the *implementation*. Rewrite the closing sentence of the PIC-12 paragraph in `host-interfaces-services.md` so it reads as an implementation-conformance MUST rather than a runtime MUST on the runtime: "A conforming implementation MUST carry a build-time check that fails the build on any direct lexical reference to `Date.now`, `performance.now`, `Date.prototype.getTime`, the global `setTimeout`, or the global `clearTimeout` from any production-code site other than the `WallClock` adapter; the check's failure shape is build failure, not a runtime diagnostic." Keep the existing "implementation choice" wording for *how* the check is realised (grep, AST scan, lint rule, architectural test) â€” pin only *that* such a check exists and *what* it must reject as the conformance point.

Enumerate the five banned identifiers (`Date.now`, `performance.now`, `Date.prototype.getTime`, `setTimeout`, `clearTimeout`) as the closed primitive list the check must cover, preserving the `scan list âŠ‡ spec ban` superset relationship `conventions.md` checklist item 9 already maintains for the plan-side scan list. Pin this closed enumeration in the spec text rather than leaving it to the implementation, so a future maintainer cannot read "MUST carry a build-time check" as licence to widen the list arbitrarily.

State explicitly that the criterion binds only *direct lexical* references: indirect forms (aliased reads, destructured reads, computed access, helper wrappers, re-export indirection â€” the residue `conventions.md` checklist items 2 / 9 enumerate) remain conceded to plan-side review and are out of scope for the spec criterion. The MUST carries a "direct lexical reference" qualifier on its face.

Apply the same shape to the parallel `process.env.HOME` / `process.env.USERPROFILE` ban under PIC-13's `homedir()` bullet in the same edit so the two bans stay symmetrical â€” either re-state the PIC-13 ban to the same acceptance-criterion shape, or mark PIC-12 as the canonical statement and have PIC-13 cross-reference it, so the two bans do not drift apart.

## Consequence

**Severity:** advisory

A non-reference implementation can ship calls to `Date.now` / `performance.now` / `setTimeout` outside the `WallClock` adapter and remain "conforming" by the letter of the spec, because the only stated enforcement is explicitly non-normative. In the reference implementation itself the practical cost is timing-test flakiness at unrelated sites â€” a violating site silently breaks `FakeClock`-driven assertions of the watcher debounce, shutdown await, or `loop-iter` yield without surfacing at its own location. Implementers can still produce a working system; the gap is the absence of a conformance hook.
