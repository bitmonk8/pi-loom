## Finding

PIC-12 carries a runtime MUST that "the runtime MUST NOT call `Date.now`, `performance.now`, `Date.prototype.getTime`, or the global `setTimeout` / `clearTimeout` outside the `WallClock` adapter." The only enforcement mechanism the spec then names — a build-time grep-test, parallel to the `process.env.HOME` ban for `homedir()` — is immediately disclaimed in the same sentence as "a non-normative implementation note … the grep-test mechanism is an implementation choice, not an observable conformance point." The MUST therefore stands with no normative acceptance criterion of any kind: no behavioural observation through `FakeClock`, no static-analysis obligation that any conforming implementation must run, and no diagnostic that fires on violation.

The downstream cost is not abstract. The same paragraph already records that `FakeClock` is the seam that makes the watcher's 250 ms debounce, the `SHUTDOWN_AWAIT_CAP_MS` shutdown await, the `looms.scanPackagesTimeoutMs` deadline math, and the `loop-iter` macrotask yield deterministically testable. Every one of those conformance tests assumes that no runtime path reaches around the seam to a real `Date.now` / `performance.now` / `setTimeout`. A violation outside the `WallClock` adapter silently corrupts the timing assertion at the *consuming* test — the violating site is invisible at its own site and only surfaces as a flaky timer- or deadline-bounded test elsewhere — because nothing in the spec turns the ban itself into a green-or-red observation.

The defect is symmetrical with the `process.env.HOME` ban PIC-12 cites as its parallel, which has the same surface (DISC-1 / PIC-13 `homedir()`) and the same self-disclaimed enforcement — a separately-raised finding in this review flags the DISC-1 env-var ban as a co-bundled obligation, but neither side currently pins what a conforming implementation must do to demonstrate the ban holds. The spec needs either an observable acceptance criterion (a static-analysis assertion any conforming implementation must run, with a stated failure shape) or an explicit demotion of the MUST to an architectural / contributor-discipline note that does not pretend to bind conformance.

## Solution Space

**Shape:** multiple

### Option A — Pin a normative static-analysis acceptance criterion

**Approach.** Promote the grep-test from "non-normative implementation note" to a normative conformance hook. The PIC-12 paragraph adds an observable acceptance criterion of the form: "A conforming implementation MUST carry a build-time check that fails the build on any direct lexical reference to `Date.now`, `performance.now`, `Date.prototype.getTime`, the global `setTimeout`, or the global `clearTimeout` from any production-code site other than the `WallClock` adapter; the check's failure shape is build failure, not a runtime diagnostic." Keep the present "implementation choice" wording for *how* the check is realised (grep, AST scan, lint rule, architectural test), but pin *that* such a check exists and *what* it must reject as a conformance point.

**Spec edits.**
- Rewrite the closing sentence of the PIC-12 paragraph in `host-interfaces-services.md` to state the acceptance criterion above as a runtime-conformance MUST on the *implementation*, not as a runtime MUST on the *runtime*.
- Enumerate the five banned identifiers as the closed primitive list the check must cover (`scan list ⊇ spec ban` — the same superset relationship `conventions.md` checklist item 9 already maintains for the plan-side scan list).
- Apply the same shape to the parallel `process.env.HOME` / `process.env.USERPROFILE` ban under PIC-13's `homedir()` bullet so the two bans stay symmetrical (or explicitly state PIC-12 is the canonical site and PIC-13 inherits).

**Pros.**
- The MUST gains an observable acceptance criterion that any conforming implementation can be tested against.
- Brings PIC-12 into line with the rest of the conformance-testable spec surface and lets the plan's `H3a` / `V8d` architectural-test acceptance criteria cite a spec REQ-ID for *what* they verify, not just for the runtime behaviour they wrap.
- Closes the silent-`FakeClock`-corruption hazard at the spec level rather than at the reference-implementation level.

**Cons.**
- The criterion is testable only for *direct lexical* references; indirect forms (aliased reads, destructured reads, computed access, helper wrappers, re-export indirection — the residue `conventions.md` checklist items 2 / 9 enumerate) remain unobservable without a heavier static-analysis surface the spec does not currently mandate. The MUST would therefore carry an explicit "direct lexical reference" qualifier, which slightly weakens the ban's reach.
- Couples the spec to a static-analysis posture other Pi extensions may not adopt, narrowing the audience for which the conformance criterion is checkable.

**Risks.** A future maintainer reading "MUST carry a build-time check" may interpret it as licensing arbitrary widening of the primitive list; the closed enumeration must be pinned in the spec text rather than left to the implementation.

### Option B — Demote the ban from a runtime MUST to an architectural / contributor-discipline note

**Approach.** Acknowledge that the ban is an internal architectural posture of the reference implementation — held to keep `FakeClock` timing assertions trustworthy — and remove the runtime MUST. Replace the closing sentences of the PIC-12 paragraph with a non-normative architectural note: "The reference implementation routes all wall-clock and deferred-work primitives through the `WallClock` adapter to keep `FakeClock`-driven timing assertions trustworthy; how a conforming implementation enforces equivalent discipline is an implementation choice." The behavioural rule that *production wiring uses `performance.now`/`Date.now`/`setTimeout` via the `WallClock` adapter and tests use `FakeClock`* stays normative on the seam's *consumers*; the prohibition on ambient access becomes architectural advice.

**Spec edits.**
- Strike the final two sentences of the PIC-12 paragraph in `host-interfaces-services.md` (the MUST NOT and the self-disclaimed grep-test note) and replace with the architectural note above.
- Update `implementation-notes.md` *Clock.* bullet to drop any reference to a ban being enforced normatively.
- Apply the same demotion to PIC-13's `process.env.HOME` ban (or leave it standing if PIC-13 has its own observable acceptance criterion — Option B'd be limited to PIC-12).

**Pros.**
- Honest about what the spec actually constrains: the seam's consumers and the production-wiring shape are observable; the ambient-primitive ban is implementation hygiene.
- Removes a normative MUST that no conformance test can witness, which removes a class of toothless-MUST cruft the broader audit flags elsewhere in this review.

**Cons.**
- Loses the architectural signal that today carries plan-side mechanical weight (the `H3a` architectural test, the `V8d` *Tests* bullet, the `conventions.md` checklist item 9 reconciliation all anchor to PIC-12 as the source-of-truth for the timing-primitive list). Each of those would need to re-anchor to a plan-internal convention.
- A non-reference implementation could legitimately reach around `WallClock` and the spec would no longer object — the silent-`FakeClock`-corruption hazard becomes a "do not do this" rather than a "MUST not do this."

**Risks.** Demoting the ban risks loosening the discipline that the rest of the seam's testability story depends on; the architectural note must be strong enough that the `H3a` / `V8d` plan citations still read as well-anchored.

### Recommendation

Option A. The PIC-12 ban is doing real work — every `FakeClock`-driven test in the spec depends on it holding, and the plan has already mechanised it at `H3a` / `V8d` against `src/**`. Promoting the grep-test to a normative "the implementation MUST carry a build-time check that rejects direct lexical references to the enumerated primitives outside the `WallClock` adapter" closes the observable-criterion gap with the smallest disturbance to the existing plan surface: `V8d`'s architectural test already discharges it, and `coverage-matrix.md` already maps PIC-12 → `V8d`. The implementer must watch two edges: (i) the criterion binds only direct lexical references — the indirect-form residue (alias / destructure / computed / wrapper / re-export indirection) stays explicitly conceded to plan-side review (`conventions.md` checklist items 2 / 9), and the spec should say so rather than imply otherwise; (ii) the parallel `process.env.HOME` / `process.env.USERPROFILE` ban under PIC-13's `homedir()` bullet should be re-stated to the same shape in the same edit, or PIC-12 should be marked as the canonical statement and PIC-13 cross-reference it, so the two bans do not drift apart.

## Consequence

**Severity:** advisory

A non-reference implementation can ship calls to `Date.now` / `performance.now` / `setTimeout` outside the `WallClock` adapter and remain "conforming" by the letter of the spec, because the only stated enforcement is explicitly non-normative. In the reference implementation itself the practical cost is timing-test flakiness at unrelated sites — a violating site silently breaks `FakeClock`-driven assertions of the watcher debounce, shutdown await, or `loop-iter` yield without surfacing at its own location. Implementers can still produce a working system; the gap is the absence of a conformance hook.
