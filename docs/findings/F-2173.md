## Finding

Throughout the spec corpus the word **anchor** has a precise, governed technical meaning: an HTML `<a id="…">` cross-reference target, regulated by GOV-3 / GOV-8 (e.g. `#pi-sdk-pin`, `#pi-sdk-pin-manifest-lock-step`). The *Pi SDK pin* section of `host-prerequisites.md` uses that exact sense repeatedly — "This anchor (`#pi-sdk-pin`) is the single source of truth", "the literal-anchor list in [Pi version bump procedure]", and the page is dense with `<a id>` declarations (`#pic-33`, `#pi-sdk-pin-manifest-lock-step`, etc.).

Inside the *Deliberate deviation from Pi's bundled-package convention* sub-paragraph the same section coins a different, unrelated sense of the word: **"The four-anchor invariant is what the build-time `peerDependencies` literal-read test … are built around."** Here "four-anchor" does not denote four `<a id>` targets — it denotes the four `@earendil-works/*` peer-dependency entries that must move together at one Pi minor line (the property PIC-33 names "Four-package lock-step co-move"). The collision is acute because the very next clause of the same sentence uses "anchor" in the spec-reserved sense ("the literal-anchor list"), so a reader meets both meanings of the word within one sentence.

The phrase is a one-off — `four-anchor invariant` appears exactly once in the corpus — and the concept it names already has a settled, non-overloaded vocabulary on the same page ("four-package lock-step", PIC-33). Renaming it removes the overload at zero conceptual cost.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** `d8da74b` (`pi-loom spec: resolve "peerDependencies ranges silently deviate from Pi documented * convention"`, 2026-05-08)
**History:** `git log -S "four-anchor invariant"` over the corpus shows the phrase first entered the spec in `d8da74b`, the commit that authored the *Deliberate deviation from Pi's bundled-package convention* sub-paragraph to resolve a peer-dependency-convention review finding. That same paragraph already used "anchor" in the spec-reserved `<a id>` sense ("the literal-anchor list in [Pi version bump procedure]"), so the overload was present from the paragraph's first authoring. The later split commit `f5e89f4` only relocated the paragraph into `host-prerequisites.md` during the corpus file-split (and carried the `@mariozechner` → `@earendil-works` rename and the `0.72.1` → `0.75.5` range bump); it did not introduce the overload. The defect is therefore attributable to the single authoring commit `d8da74b`, not present-since-inception of the spec.

## Solution Space

**Shape:** single

### Recommendation

In the *Deliberate deviation* sub-paragraph of `#pi-sdk-pin`, replace the phrase **"four-anchor invariant"** with **"four-package lock-step invariant"**, matching the PIC-33 rule title ("Four-package lock-step co-move") immediately above and the "four-package lock-step" vocabulary already used in the item-1 paragraph. This is a single-phrase edit at the one site where the phrase occurs; no other occurrence exists in the corpus and no anchor, rule ID, or citation changes. ("peer-dep lock-step invariant" is an acceptable equivalent if a non-numeric form is preferred, but "four-package" keeps it parallel with PIC-33.)

## Consequence

**Severity:** cosmetic

A reader must disambiguate two senses of "anchor" within a single sentence; the intended meaning is recoverable from the surrounding context (the build-time literal-read test, the peer-dep probe), so no implementation diverges. The cost is interpretive friction in a load-bearing rationale paragraph, not behavioural ambiguity.
