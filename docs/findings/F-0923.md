## Finding

H5b's single Tests bullet asserts the warn-only canary behaviour by "seeding one gap of each kind, confirming the canary reports it, and confirming the run's exit status stays green, **then restoring the seed**." The cleanup verb is inverted: the test's seeding step *adds* a gap, so "restoring the seed" reads naturally as "put the seed back" — i.e. re-apply the seeded gap — whereas the intended final action is the opposite: remove the seeded gaps so the corpus returns to its clean, pre-seed state.

The ambiguity is sharper here than in the sibling H6a bullets, which use the same family of phrasing coherently: H6a removes a mapping "and restor[es] it", where "restoring" correctly means putting the removed mapping back. Because H5b's operation is seed-then-unseed (not remove-then-restore), reusing "restoring the seed" describes the wrong direction. The bullet also seeds one gap of *each* kind (four kinds), so the cleanup is plural.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H5b-warn-only-canary.md`, in the `Convention:` Tests bullet, replace the trailing clause `then restoring the seed` with a clause that names the undo of the seeding as the cleanup — restoring the corpus to its pre-seed clean state, not re-applying the seed. Literal replacement text: `then removing the seeded gaps to restore the corpus to its pre-seed clean state`. Use the plural ("gaps") to match the bullet's "seeding one gap of each kind". No mechanism changes; this is wording only.

## Consequence

**Severity:** advisory

An implementer taking "restoring the seed" literally would re-apply the seeded gaps as the test's final step, leaving the corpus polluted with permanent seeded violations; a later canary or closing-gate run over that state could then report a spurious finding. The surrounding context makes the intended cleanup inferable, so the canary itself still functions, but the bullet directs the opposite of the intended teardown.
