## Problem

The binder envelope reference schema in `binder-bypass-and-envelope.md`
(`### Binder envelope schema (constructed dynamically from params:)`)
renders the `ambiguous.candidates` field (`array<string> | null`) in the
multi-type-array form `{ "type": ["array", "null"], "items": … }`. SUBS-1
in `schema-subset.md` (`#subs-1`) requires a union with any non-primitive
arm to lower to `{ "anyOf": [...] }`, and `array<string>` is non-primitive
(its own lowering rule, distinct from the primitive rule; cf. the
`string | Author` reference vector). The reference JSON therefore
contradicts the canonical lowering rule it claims to instantiate — the
"exactly what the lowering pass would produce" introduction is byte-false
for this field.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** 816c3988 (2026-05-04, "pi-loom spec: resolve \"Binder envelope schema is hand-crafted, off-subset, and fails its own AJV safety net\""); 6892aa3e (2026-06-20, "pi-loom spec-plan: fix F-0786 — pin primitive-union/nullability lowered form (SUBS-1) for canonical-hash byte-identity")
**History:** The `candidates` field was deliberately set to the multi-type-array form by 816c3988, which rewrote the envelope schema and changed `candidates` from `{ "anyOf": [{ "type": "array", … }, { "type": "null" }] }` to `{ "type": ["array", "null"], … }` on the explicit premise (carried in the resolved finding's own text) that *"`array` is in the primitive-types list"* and that the subset *"prefers the multi-type-array form … when all arms are primitive."* At that time the lowering rule was permissive — step 3 read "Union of primitives only: `{ "type": [...] }` form *preferred* for readability; falls back to `anyOf` if any arm is non-primitive" — and the primitive/non-primitive boundary for `array<T>` was not pinned. Commit 6892aa3e later hardened that bullet into SUBS-1, turning the soft preference into a MUST and fixing `array<string> | null` squarely on the non-primitive (`anyOf`) side via the `string | Author` reference vector. The binder reference JSON was never re-synchronised with the tightened rule, so the form 816c3988 introduced became a SUBS-1 violation only once 6892aa3e landed. The defect is the interaction of the two edits, not either alone.

## Solution approach

Rewrite the `candidates` property of the `ambiguous` arm in the reference
envelope JSON to the SUBS-1-conformant `anyOf` form, emitting the array
arm first and the `"null"` arm last per SUBS-1's arm-order clause:

```json
"candidates": {
  "anyOf": [
    { "type": "array", "items": { "type": "string", "maxLength": 500 } },
    { "type": "null" }
  ]
}
```

The `array<string> | null` type expression itself is unchanged; only the
lowered JSON form moves to `anyOf`.

## Solution constraints

- Out of scope: BNDR-2's normative text and its scope-tagging concern (owned by F-2087); change only the reference envelope schema JSON.
