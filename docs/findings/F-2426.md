## Problem

The custom-type-unsafe row of the Failure-mode templates (normative) table renders `'<value>'`, but `<value>` has no pinned source and no stated escaping. System-note rendering rule 1 collapses only ASCII whitespace, so a `CustomMessage.customType` containing `'`, `]`, or the `: ` sequence breaks the single-quote wrapper or the prefix/suffix boundary that a downstream renderer is contracted to trust. The row is part of a spec-versioned breaking-change surface and is not testable against a fixed byte oracle while the placeholder is unpinned.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 83802db — spec: collapse to single typed argument style; add LLM-driven slash-command binder (2026-05-04, Thomas Andersen); be82734 — pi-loom spec: resolve "Ceiling #3 (binder retry budget): provider-failure classification rule undefined" (2026-05-07, Thomas Andersen); f41af5d — pi-loom spec-plan: fix F-1131 — add the custom-type-unsafe Failure-mode-templates row (2026-06-24, Thomas Andersen)
**History:** The failure-mode-templates table was introduced with the LLM-driven binder at 83802db, where the transport row's `<message>` placeholder was already present and unpinned. be82734 added the rule routing structural `ContextOverflowError` into that same transport row without defining how an error variant lacking a `provider` field populates `<provider>`/`<message>`. f41af5d later added the custom-type-unsafe row carrying `<value>` without pinning its source or escaping. (A subsequent clarity pass, 2fb5783, pinned `<provider>` and `<ajv-summary>` explicitly but left `<message>` and `<value>` unpinned.)

## Solution approach

Pin the custom-type-unsafe row's `<value>` to the rejected `CustomMessage.customType` per BNDR-9 (`#bndr-9`). Clarify the escaping applied to that value inside the `'…'` wrapper — either binding it to an existing escaping discipline or stating that none is required and why — so the rendered note remains unambiguous for a `customType` carrying `'`, `]`, or `: `.
