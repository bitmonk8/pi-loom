## Problem

The hot-reload contract pins coalescing of *scheduled* watcher events (the 250 ms debounce in the *Caching and reload* paragraph and step 5's chokidar window) and the stability of a single in-flight invocation across one build-aside-then-publish swap (the In-flight invocation rule), but says nothing about what happens when a debounce timer fires while a rebuild from a prior window is still awaiting its asynchronous steps (re-parse, AJV recompilation, `pi.registerTool`). Two concurrent rebuilds can then run against the same `LoomRegistry`, AJV validator cache, and registration cache with no rule ordering their publishes. The unguarded outcomes the spec does not exclude are: a stale older rebuild publishing over a newer one (lost update), validator-cache eviction interleaving with another rebuild's staged-validator construction, and an undefined `ERR-7` `loom/runtime/registry-swap-failed` partition between the two publish orders.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** f5e89f4 — docs: enforce 100KB hard / 30KB recommended size cap on spec set (2026-06-04, Thomas Andersen); 6d1a949 — pi-loom spec: resolve "session-shutdown sub-step 5 hot-reload split" (2026-06-05, Thomas Andersen)
**History:** The drop-and-reschedule debounce paragraph in `discovery/package-and-settings.md` was present at the file's first commit (`f5e89f4`), and the symmetric chokidar-window paragraph in `registration-steps.md` was created the next day by `6d1a949`, which also introduced the build-aside-then-publish swap and the In-flight invocation rule. Both commits pin coalescing of *scheduled* timeouts and stability of one in-flight *invocation* against one swap; neither introduced (nor did any subsequent edit add) the symmetric obligation against multiple in-flight *rebuilds*. The gap is the interaction of the two: the debounce gives an upper bound on event-to-rebuild latency but not on rebuild duration, and the build-aside-then-publish swap is observable as atomic only when there is a single staging set.

## Solution approach

Add a rebuild-serialization rule to the *Caching and reload* paragraph in `package-and-settings.md` (anchor `caching-and-reload`) and to step 5's watcher / hot-reload registration in `registration-steps.md` (anchor `watcher-hot-reload-registration`): a debounce timer that fires while a prior window's rebuild is still awaiting must not start a concurrent rebuild against the live `LoomRegistry`, validator cache, and registration cache. Pin that no edit is lost across the deferral. Confirm the In-flight invocation rule is unchanged — running invocations continue against the pre-swap snapshot and the publish remains a single synchronous write. Address the `session_shutdown` handler (step 4) so teardown does not block on a deferred rebuild.

## Solution constraints

- Preserve the existing 250 ms drop-and-reschedule coalescing within a single debounce window unchanged; the serialization rule applies only across windows (a timer firing while a prior rebuild is in flight).
- Out of scope: the watcher post-`error` lifecycle posture on step 5, owned by F-1203.
