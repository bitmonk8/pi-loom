## Finding

The watcher / hot-reload contract specifies how *scheduled* events are coalesced — the 250 ms `Clock.setTimeout` debounce window with drop-and-reschedule semantics in [`package-and-settings.md` — Caching and reload](../spec_topics/discovery/package-and-settings.md#caching-and-reload) and the chokidar window in [`registration-steps.md` — watcher / hot-reload registration](../spec_topics/pi-integration-contract/registration-steps.md#watcher-hot-reload-registration) — but says nothing about what happens once a debounce window has *closed* and an asynchronous rebuild is already running. The rebuild is materially asynchronous: it re-parses the changed `.loom` / `.warp` file and every transitive importer, awaits AJV recompilation, and awaits `pi.registerTool` for every newly-distinct schema slug before the single synchronous publish step installs the staged registry, the staged validator cache, and the staged registration-cache entries.

A fresh chokidar event that arrives 250 ms or more after the first window's timer fires schedules a brand-new timer that elapses while the first rebuild's `await`s are still in flight. Two concurrent rebuilds are then live against the same `LoomRegistry`, the same AJV cache, and the same `pi.registerTool` registration cache, with no rule that orders their publishes. Possible observable bad outcomes the spec does not exclude: an older rebuild whose `pi.registerTool` settles second publishes *after* a newer rebuild that already published, replacing the newer staged registry with the older one (lost-update / stale-registry); the second rebuild's cache-eviction pass (per [`V15e`](../plan_topics/V15e-hot-reload-cache-eviction.md)) interleaves with the first rebuild's staged validator construction and evicts entries the first rebuild has just rebuilt; and the `ERR-7` `loom/runtime/registry-swap-failed` partition between "newer publish replaces older" and "older publish replaces newer" is undefined.

The whole point of the build-aside-then-publish swap is "atomic against a single in-flight invocation" — and the In-flight invocation rule rightly pins that reads against the entry-table are stable across one swap. The omission is the symmetric obligation against multiple in-flight *rebuilds*: nothing says rebuilds are serialised, nothing pins a monotonic generation that lets a stale publish be discarded, and nothing forbids the second-fired timer from starting until the first has published.

**History:** The drop-and-reschedule debounce paragraph in `discovery/package-and-settings.md` was present at the file's first commit (`f5e89f4`), and the symmetric chokidar-window paragraph in `registration-steps.md` was created the next day by `6d1a949`, which also introduced the build-aside-then-publish swap and the In-flight invocation rule. Both commits pin coalescing of *scheduled* timeouts and stability of one in-flight *invocation* against one swap; neither introduced (nor did any subsequent edit add) the symmetric obligation against multiple in-flight *rebuilds*. The gap is the interaction of the two: the debounce gives an upper bound on event-to-rebuild latency but not on rebuild duration, and the build-aside-then-publish swap is observable as atomic only when there is a single staging set.

## Solution Space

**Shape:** multiple

### Option A — Serialise rebuilds (wait-for-in-flight)

- **Approach.** Extend the watcher-time contract so that, when a debounce timer fires while a rebuild from a prior window is still running, the new rebuild is *queued* rather than started. The queue holds at most one pending rebuild (a third event coalesces into the queued slot rather than appending). On completion of the in-flight rebuild's publish, the queued rebuild starts; on failure of the in-flight rebuild, the queued rebuild still starts (the failure is independent).
- **Spec edits.** In the *Caching and reload* paragraph and in step 5 of `registration-steps.md`, after the debounce-window rule, add a sentence pinning the in-flight guard and the single-slot queue. State explicitly that the chokidar event that triggers the queued rebuild captures the union of files changed since the in-flight rebuild's staging-set was frozen, so no edit is lost. State that the In-flight invocation rule is unchanged (running invocations continue against the pre-swap snapshot regardless of how many rebuilds are queued behind the swap).
- **Pros.** Simplest mental model; no per-rebuild metadata; the publish step remains a single synchronous write under the existing atomicity rule; ERR-7's stale-publish question disappears by construction.
- **Cons.** A slow rebuild (a large `.warp` import graph, a slow `pi.registerTool`) delays propagation of subsequent edits by the duration of the slow rebuild plus the debounce window.
- **Risks.** The queue head needs to be cleared on `session_shutdown` so a pending rebuild does not race the teardown sub-steps; the SHUTDOWN_AWAIT cap already gives bounded waiting.

### Option B — Monotonic publish generation with stale-publish discard

- **Approach.** Each scheduled rebuild captures a monotonically-increasing generation number at the moment its staging set is frozen. The single synchronous publish step performs a compare-and-set against the current `LoomRegistry`'s published generation: install the staged set only if the candidate generation is strictly greater than the published generation; otherwise discard the staged set silently.
- **Spec edits.** Add a generation field to the `LoomRegistry`'s published snapshot in step 5; pin the compare-and-set under the *Caching and reload* paragraph and under step 5's publish step; state that a discarded publish emits no `loom/runtime/registry-swap-failed` diagnostic (it is not a failure — the registry already holds a strictly newer version) but its already-issued `pi.registerTool` calls remain in Pi's global registry as the existing tool-registration-lifetime page already accepts. Add a sentence to ERR-7 noting that the registry-swap-failed code applies only to throws, not to compare-and-set discards.
- **Pros.** No serialisation cost — rebuilds run concurrently and the freshest one wins; latency-of-propagation is bounded by the *fastest* rebuild.
- **Cons.** Wasted work on stale rebuilds; tool-registration-lifetime accumulation grows under burst conditions; the model is harder to reason about and requires the generation field to thread through every staging artefact.
- **Risks.** Cache-eviction interleaving (`V15e`) still needs an explicit rule — the spec must state that eviction is performed *at publish time* under the same compare-and-set, not eagerly when the staging set is built.

### Recommendation

Adopt **Option A (serialise)**. The implementation surface is smaller, the existing atomicity rule extends naturally to the queue, and the ERR-7 / `pi.registerTool` accumulation question collapses to its existing single-rebuild discharge. Option B's only material advantage is rebuild-latency under burst conditions, which is not a stated requirement and is already bounded by the 250 ms debounce window for the common case. Edge cases the implementer must watch: a chokidar event arriving *during* the in-flight rebuild's debounce window (before its timer fires) still drop-and-reschedules into the existing window, not into the queue — the queue only catches events whose timer would fire while a prior rebuild's awaits are still in flight; the queued rebuild's chokidar payload is the union of changes accumulated since the in-flight rebuild's staging set was frozen, not since the queued rebuild's timer fired; and `session_shutdown` must drop the queue head as part of sub-step 4's watcher-close sequence so the teardown does not block on a queued rebuild.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge: one will adopt wait-for-in-flight serialisation, the other will treat each debounce window as independent and let publishes race. The race-permitting implementation can lose an edit (an older slow rebuild publishing over a newer fast rebuild leaves the registry at the prior content), and it can interleave validator-cache eviction with staged-validator construction, producing a registry whose tools point at evicted validators. The user-observable surface is a `.loom` edit that "doesn't take" and only recovers on the next watcher event, with no diagnostic to explain the staleness.
