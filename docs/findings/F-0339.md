## Problem

`coverage-matrix.md`'s `X-n … X-m` range note (in §Numbered REQ-IDs) defines the inclusive contiguous range and carves out only the numbering-*hole* case (a number neither live nor retired surfaces as the `H5a` per-prefix-numbering-hole gate failure); it says nothing about a **retired interior** REQ-ID. The `H5a` citing-test reconciliation in `conventions.md` *REQ-ID discipline* enumerates mapped REQ-IDs from the matrix and treats a "coverage-matrix-mapped numbered REQ-ID with no citing test" as a CI failure, whereas the unmapped-REQ-ID direction enumerates the *live* executable set (retired IDs excluded). Because the range note is silent here, a naive textual expander treats a retired interior ID as mapped — and, being retired, it has no citing test by design — tripping a spurious "mapped-REQ-ID-with-no-citing-test" failure. The defect is latent: it materialises the first time an ID interior to a live `…` range (e.g. `TYPE-1 … TYPE-10`, `ERR-1 … ERR-7`, `CIO-2 … CIO-4`) is retired, reddening the warn-only canary and the loom 1.0 release gate until the range is manually split.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 37d5ac7 ("pi-loom plan: resolve \"Coverage-matrix ellipsis range notation undefined\"", 2026-06-11) — crystallising; 953e3fa ("pi-loom plan: resolve \"closing gate checks numbered-REQ-ID matrix mapping, not test assertion\"", 2026-06-10) — contributing predecessor.
**History:** The citing-test reconciliation direction ("a coverage-matrix-mapped numbered REQ-ID with no citing test … is a CI failure") landed first in `conventions.md` at 953e3fa. The formal `X-n … X-m` range-expansion semantics landed later in `coverage-matrix.md` at 37d5ac7 (a descendant of 953e3fa, confirmed by `git rev-list 953e3fa..37d5ac7`). 37d5ac7 added an explicit carve-out for the numbering-*hole* case but none for a retired *interior* ID, while the citing-test direction that now consumes the expanded set was already live. Neither commit alone is the defect: before 953e3fa there was no citing-test obligation for a range member to violate, and before 37d5ac7 the range expansion was undefined. The defect is the interaction — range expansion defined without a retired-ID carve-out on top of an already-present citing-test reconciliation. The retired/live-clash and numbering-hole machinery (`conventions.md`, 15f69aa and earlier) is background context, not an introducing change.

## Solution approach

Amend `coverage-matrix.md`'s `X-n … X-m` range note so the retired-interior-ID case is handled alongside the existing numbering-hole clause: state that a retired interior ID, accounted for by the prefix's `## Retired REQ-IDs` table, is not a coverage-matrix-mapped REQ-ID for the citing-test reconciliation. Mirror the same exclusion in `conventions.md` *REQ-ID discipline* where the citing-test reconciliation is defined, so the "coverage-matrix-mapped numbered REQ-ID" set it iterates is the matrix mappings intersected with the live executable REQ-ID set (retired IDs excluded). Retired interior IDs then carry no citing-test obligation and never redden the mapped-REQ-ID-with-no-citing-test arm.

## Solution constraints

- The exclusion applies only to the citing-test direction's range expansion: a genuine numbering hole (a number neither live nor retired) MUST still redden the `H5a` per-prefix-numbering-hole arm.
- Out of scope: the `spec_topics/**` governance corpus (including the prefix table and retirement tables) is read-only for this plan-doc fix.
