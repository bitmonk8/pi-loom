## Problem

The child-invoke arm of the **Surfacing** rules in `cancellation.md` (`#surfacing`) discriminates the two output envelopes on temporal phrases — `kind: "invoke_callee"` wrapping inner `cancelled` "when the abort originated inside the child" vs bare `kind: "cancelled"` "when the parent's own signal fired first." Those phrases are not disjoint for the case that matters most: a parent-initiated cancellation that fires while the child is already in flight. Under **Propagation** that abort travels down the derived controller and is observed at one of the child's own checkpoints, so it satisfies both arms, and two implementers diverge on which `QueryError` discriminant results. The same ambiguous two-arm phrasing is mirrored in the Cancellation row of the per-cause caller-surfaces table in `errors-and-results/error-model.md`.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** fae85d3 — Tighten spec for implementation-plan readiness (2026-05-04, Thomas Andersen)
**History:** The two-arm child-invoke Surfacing rule was authored in fae85d3, which first introduced the `## Surfacing` section into `spec.md`; the disambiguator "when the parent's own signal fired first" was never tied to a checkpoint boundary, so the in-flight-propagation case was ambiguous from the moment the arm appeared. A later same-day commit (8395c03, 2026-05-04) added the authoritative `invoke`-entry rule ("if the parent's signal is already aborted at child-spawn time, the child surfaces `cancelled` synchronously without spawning") but did not reconcile the Surfacing arm against it, leaving the gap. Subsequent edits (8455d5e) only renamed the variant (`invoke_callee_error` → `invoke_callee`) and never touched the temporal phrasing.

## Solution approach

Rewrite the child-invoke Surfacing arm at `#surfacing` to key the two cases on the pre-invoke checkpoint rather than on temporal "fired first": bare `Err(QueryError { kind: "cancelled", ... })` applies only when the parent's signal was already aborted at the pre-invoke checkpoint and the child never spawned, cross-referencing the `invoke(...)`-entry forwarding rule that already states this; `Err(QueryError { kind: "invoke_callee", inner: { kind: "cancelled", ... } })` applies whenever the abort is observed inside the child after it has spawned, including a parent-initiated abort propagated down to the in-flight child. Apply the identical disambiguation to the Cancellation row of the per-cause caller-surfaces table in `errors-and-results/error-model.md` so both pages state the boundary in the same terms.

## Solution constraints

- Respect CNCL-5 (`#cncl-5`): a parent-initiated in-flight cancellation surfacing `invoke_callee` MUST NOT be retroactively rewritten when the parent's next checkpoint subsequently observes its own abort.
