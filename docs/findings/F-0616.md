## Finding

The *Wall-clock budget* clause in `audit-wire-and-canary.md` says: "The audit MUST enforce a wall-clock budget on its own execution (the implementation pins the budget value, e.g. 60 s); on budget expiry the audit MUST emit one record under a stable `<symptom>` token in the infrastructure-failure family and exit non-zero." The MUST is normative but its argument — the budget value — is delegated wholesale to the implementation with only a parenthetical "e.g. 60 s" as guidance. No upper bound, no lower bound, no required order of magnitude.

A conformant implementation that pins the budget at 1 ms satisfies the MUST while turning the budget into a guaranteed spurious-timeout source (every real audit run trips it). A conformant implementation that pins it at 86 400 s satisfies the MUST while neutralising the fail-closed property the clause exists to provide (a hung audit will be killed by the host's `npm test`-side timeout long before the wall-clock budget fires, so the budget never actually bounds anything). Either extreme passes the spec; the "fail-closed guarantee" the rest of the section relies on becomes unenforceable at the spec level and reduces to a trust-the-implementer convention. The same problem affects the `npm test`-side timeout the *Pre-emission termination carve-out* defers to ("the implementation pinning a `npm test`-side timeout to bound the overall run"), with identical unboundedness.

The "e.g. 60 s" is the only quantitative anchor in the clause but it is non-normative, so it carries no obligation. The spec elsewhere pins concrete numeric thresholds (counter floors, the `> 0` floor on the canary, etc.) and is precise about discriminator-token shapes; the wall-clock budget is the only audit-side numeric pin left implementation-defined.

## Solution Space

**Shape:** multiple

### Option A — Pin a single concrete budget value

**Approach:** Replace the parenthetical "e.g. 60 s" with a normative pin: *"the audit MUST enforce a wall-clock budget of 60 s on its own execution"*. Apply the same treatment to the `npm test`-side timeout one sentence later (pin a single value, e.g. 120 s, that is comfortably greater than the audit's budget so the audit-side fail-closed surface fires first and produces a routable record). Optionally add a forward-compatibility note that a future minor revision MAY raise the budget if a non-pathological audit run is observed exceeding it.

**Spec edits:**

- In `audit-wire-and-canary.md`, *Infrastructure-failure handling* paragraph, *Wall-clock budget* clause: change *"(the implementation pins the budget value, e.g. 60 s)"* to *"of 60 s"* and drop the parenthetical.
- In the *Pre-emission termination carve-out* clause: change *"with the implementation pinning a `npm test`-side timeout to bound the overall run"* to a concrete value (e.g. *"with a 120 s `npm test`-side timeout that bounds the overall run; the 60 s audit budget above is the inner bound and fires first on a hung audit, producing a routable infrastructure-failure record"*).

**Pros:** Single number per timeout, trivially auditable by a reviewer, removes the divergence surface entirely. Mirrors how the spec pins other numeric thresholds (counter floors, `> 0`, etc.).

**Cons:** Locks the budget against future SDKs / source trees that may warrant a larger value; a spec revision is required to raise it. Picks a number the spec has no data to defend beyond "60 s is the example that survived the sweep".

**Risks:** A grown audited source tree could legitimately exceed 60 s, forcing a spec revision to ship a fix rather than an implementation-side adjustment. Mitigated by the optional forward-compatibility note.

### Option B — Pin an explicit numeric range

**Approach:** Keep the implementation-pins-the-value shape but constrain the choice to a normative range. The clause becomes *"MUST enforce a wall-clock budget on its own execution; the budget value MUST be in the closed interval [30 s, 300 s]; the implementation pins the concrete value within that interval"*. Apply a parallel range to the `npm test`-side timeout (e.g. [60 s, 600 s], with the additional MUST that the `npm test`-side timeout is strictly greater than the audit budget so the audit's routable infrastructure-failure record fires first).

**Spec edits:**

- In *Wall-clock budget*: insert *"the budget value MUST be in the closed interval [30 s, 300 s]"* alongside the existing MUST.
- In *Pre-emission termination carve-out*: insert an analogous range for the `npm test`-side timeout and the ordering constraint (`npm test`-timeout > audit-budget).

**Pros:** Preserves implementer latitude for source-tree size and host speed; rules out the 1 ms / 86 400 s pathologies and the inversion where the outer `npm test` timeout fires before the audit budget. Closest in spirit to the existing delegation pattern.

**Cons:** Introduces two new numeric thresholds the spec has to defend. The range bounds are themselves implementation-defined choices the spec is asserting authoritatively, with the same "where did 30 / 300 come from" question Option A raises for the single value.

**Risks:** A future SDK genuinely needing >300 s still forces a spec revision; the range only widens the window before that revision is needed.

### Recommendation

Option A. The clause already publishes "60 s" as the example, no one is reaching for a wildly different value, and the surrounding spec consistently pins single numeric thresholds rather than ranges (counter floors, `<n/a>` sentinel length, the `> 0` rule). Adding bounds (Option B) buys flexibility the project has not demonstrated it needs and asks the spec to defend three numbers instead of one. Edge cases the implementer must watch: the `npm test`-side timeout in the *Pre-emission termination carve-out* must be pinned in the same edit (the carve-out depends on it), and it must be strictly greater than the audit budget so the audit's routable infrastructure-failure record fires before the outer `npm test` kill removes the routing surface.

## Consequence

**Severity:** advisory

Two reasonable implementers will both pick a sensible value (the parenthetical 60 s is a strong hint), so divergent-but-still-working CI gates are the expected outcome rather than a correctness split. An adversarial or careless implementer can however satisfy the MUST with a value that defeats the fail-closed protection, and the spec gives a reviewer no normative anchor to reject that choice. A working system is reachable; an *unambiguously specified* fail-closed bound is not.
