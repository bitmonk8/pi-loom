## Finding

`docs/spec_topics/query/query-escapes-stringification.md` owns two distinct concerns named by its title: template escape sequences (QRY-17) and stringification of interpolated `${expr}` values (QRY-18). The file also carries two further sections — *Discarded query results are a parse error* (QRY-19) and *Observability of discarded results* (QRY-20) — that have nothing to do with escapes or stringification.

QRY-19 is a rule about how a query's `Result` must be handled at the call site: a bare `` @`...` `` expression-statement is `loom/parse/discarded-query-result`, forcing the author to choose `?`, `let _ =`, or a binding. QRY-20 specifies the operator-channel observability contract for the `Err` of a discarded query (a `loom-system-note` runtime event with `display: false`, fired exactly once). Both are about what happens to an unhandled/discarded query `Result` — i.e. query failure-handling and failure-observability — not about the lexical content of a template body or the rendering of interpolated values.

The natural home for both is `docs/spec_topics/query/query-failure-and-repair.md`, which already owns the query failure surface (QRY-8 failure modes, `QueryError` variants, `ContextOverflowError` detection, schema-validation respond-repair). QRY-20's observability contract is a direct sibling of that material; QRY-19 is tightly coupled to QRY-20 (QRY-20 references QRY-19's discard form) and should travel with it. A reader consulting the failure-handling page to learn how an unhandled query result behaves currently finds nothing and must instead discover the contract in the escapes/stringification sibling. Note that QRY-21 (*Panics during interpolation are not caught by `let _ =`*) genuinely belongs in the escapes/stringification file — it concerns `${expr}` evaluation during template rendering — so only QRY-19 and QRY-20 are misplaced.

## Solution Space

**Shape:** multiple

### Option A — Move QRY-19 + QRY-20 into `query-failure-and-repair.md`

- **Approach:** Relocate both the *Discarded query results are a parse error* (QRY-19) and *Observability of discarded results* (QRY-20) sections from `query-escapes-stringification.md` into `query-failure-and-repair.md`, keeping them adjacent and in order. Leave QRY-21 (interpolation panics) in `query-escapes-stringification.md`. Repoint any inbound links / anchors (`#qry-19`, `#qry-20`) to the new file.
- **Spec edits:** Cut the two sections from `query-escapes-stringification.md`; paste into `query-failure-and-repair.md` (e.g. after the *Failure modes* / respond-repair material, before or after the non-validation-failures subsection). Update cross-references to `query-escapes-stringification.md#qry-19` / `#qry-20` corpus-wide.
- **Pros:** Co-locates discard observability with the existing failure-observability surface (respond-repair, `QueryError` variants); keeps QRY-19/QRY-20 together; file title then accurately bounds its contents.
- **Cons:** QRY-19 is strictly a parse/syntax rule (which call-site forms are legal), which is arguably a *forms* concern rather than a *failure* concern.
- **Risks:** Anchor repointing must be complete or inbound links break; the file is near the size-cap regime that drove the original split — confirm the move keeps `query-failure-and-repair.md` under the cap.

### Option B — Move QRY-19 + QRY-20 into `query-forms.md`

- **Approach:** Relocate both sections into `query-forms.md`, on the grounds that the discard rule is a direct corollary of the query forms and their `Result<string|Schema, QueryError>` return type.
- **Spec edits:** Cut the two sections from `query-escapes-stringification.md`; paste into `query-forms.md` near the return-type / forms material. Update `#qry-19` / `#qry-20` cross-references.
- **Pros:** QRY-19 is a syntactic rule about the `` @`...` `` expression-statement position, which fits the forms page; the discard rule is a corollary of the documented return type.
- **Cons:** QRY-20 is an observability/failure contract, not a forms rule; placing it on the forms page splits the failure-observability surface across two files (respond-repair stays in failure-and-repair, discard observability lands in forms).
- **Risks:** Same anchor-repointing and size-cap considerations as Option A.

### Recommendation

Take **Option A**: move QRY-19 and QRY-20 together into `query-failure-and-repair.md`. The dominant concern across both sections is what happens to an unhandled/`Err` query `Result` — failure-handling and failure-observability — which is exactly that file's remit, and keeping the two coupled sections together (QRY-20 builds on QRY-19's discard form) outweighs the narrow syntactic argument for the forms page. Watch for: complete repointing of `#qry-19` / `#qry-20` inbound links, and verifying `query-failure-and-repair.md` stays within the spec-set size cap that motivated the original split.

## Consequence

**Severity:** cosmetic

The discard parse-error rule and its observability contract are complete and correct; only their location is wrong. The content sits in the wrong sibling within the same `query/` subdirectory, so it is still findable but unexpected — an author or implementer looking up query failure/observability behaviour on the failure-and-repair page will not find it there. No behavioural divergence results.
