## Finding

`host-interfaces-core.md` carries five forward-looking sentences embedded inline within dense normative prose, none of them marked off from the surrounding obligations the reader is trying to extract:

- *Tool execution from loom code:* "Widening to a richer return shape (preserving non-text blocks) is reserved for [Future Considerations](../future-considerations.md)." ‚Äî inside the envelope-lowering paragraph.
- *Tool execution from loom code:* "(a future revision MAY widen `cause` additively if the distinction needs to surface)" ‚Äî parenthetical at the end of the `cause: "execution"`-source paragraph.
- *Outcome routing summary:* "(loom 1.0 makes no internal timeout attempt, consistent with [...NOCEIL-1]; the deferred per-call timeout extension lands additively on the open-struct seam in [Tool Calls](../tool-calls.md))" ‚Äî embedded mid-sentence in the non-settling-Promise disposition.
- *`ExtensionContext` "Members loom does not touch" closing paragraph:* "a future widening (the deferred *Mid-loom user-session replacement* extension in [Future Considerations]...) is the only loom 1.0 surface that anticipates such use) is a spec-versioned change."
- *Per-mode override semantics ‚Äî Subagent mode:* "loom 1.0 has no per-subagent frontmatter `cwd` override; widening is reserved for [Future Considerations](../future-considerations.md)."

Each of these deferred surfaces is already owned by a normative `> **loom 1.0 seam ‚Äî <name>.**` blockquote on its proper home page (per-call timeout: `tool-calls.md` line 42 and `invocation.md` line 46; mid-loom replacement: `conversation-drive.md` line 20; the richer return shape and wider `cause` are inventoried under *Surface extensions* in `future-considerations/surface-extensions.md`). The inline mentions here add no normative content over those callouts ‚Äî they exist purely to orient the reader, but they orient them by interrupting the routing rules an implementer is consulting.

Separately, the `estimateTokens` paragraph carries a labelled "**Non-normative.**" block describing Pi's current algorithm ("a conservative character-quarter count (`Math.ceil(chars / 4)`-style, counting UTF-16 code units‚Ä¶); for orientation, a user message whose only content is `"üòÄüòÄüòÄ"` (six UTF-16 code units) estimates to roughly 2 tokens. This description is orientation only, is owned by Pi, and may drift on any Pi revision; the authoritative behaviour is Pi's implementation at the pin.") The paragraph itself already states the authoritative position ‚Äî *loom calls the function as-imported and does not redefine how it walks the message* ‚Äî and the inventoried `TokenEstimator` seam (PIC-16, consumed via `V8e`/`V11i`) explicitly uses a `FakeTokenEstimator` so test vectors do not couple to Pi's heuristic. The orientation block therefore exposes a Pi-side algorithm the spec promises not to track, with the only effect being to mislead a reader who notices `chars / 4` and starts reasoning from it on a Pi version where the heuristic has changed.

## Solution Space

**Shape:** multiple

### Option A ‚Äî delete the non-normative estimator orientation block (scope-bounding, do first)

**Approach.** In the *`estimateTokens` (named export)* paragraph of `host-interfaces-core.md`, delete the labelled "**Non-normative.**" sentence ‚Äî the entire run from "At the [loom 1.0 Pi-SDK pin], Pi's estimator‚Ä¶" through "‚Ä¶the authoritative behaviour is Pi's implementation at the pin." Keep the surrounding normative text intact (the declaration site, the `AgentMessage` parameter type, the "loom calls the function as-imported and does not redefine how it walks the message" sentence, the `PiTokenEstimator` / `TokenEstimator` PIC-16 delegation, the version-bump item (p) reference).

**Spec edits.**
- Delete the `**Non-normative.**` sentence described above, including the `"üòÄüòÄüòÄ"` orientation example.
- No other text changes.

**Pros.**
- Single contiguous deletion on one paragraph; no relocation to coordinate.
- Removes the only inline Pi-algorithm description the spec promises not to track, eliminating the misleading-anchor risk on Pi-side estimator drift.
- The retained normative text already says what loom guarantees (delegation, observable contract, non-redefinition), so nothing load-bearing is lost.

**Cons.**
- A reader looking for a sanity-check vector to understand what magnitude `estimateTokens` returns loses that ergonomic. The `FakeTokenEstimator` test fixtures in `V11i` cover what the truncation walk does with arbitrary counts; they do not give a reader an intuition for what Pi typically returns.

**Risks.**
- None at the spec-corpus level: no other paragraph in the corpus cross-references the orientation block (it carries no anchor and is not the target of any link).

### Option B ‚Äî convert the five inline future-widening sentences to `> **loom 1.0 seam ‚Äî <name>.**` cross-references or delete them in favour of the existing seam-callout homes

**Approach.** For each of the five inline sentences enumerated in **Finding**, replace the inline mention with a single short cross-reference at the end of the surrounding paragraph (e.g. "*Future widening:* the [per-call timeout seam](../tool-calls.md#‚Ä¶) reserves the additive landing."), or ‚Äî where the host paragraph already cross-references the seam owner ‚Äî delete the inline mention entirely. That page is already the inventory of record per the *Forward-compatibility seams* invariant on `overview-and-orientation.md`, so this is the intended trail rather than a regression.

### Recommendation

Apply both options in order: **Option A first**, then **Option B**.

Option A is one contiguous deletion on a paragraph that already contains the surviving normative pin, so it is the smaller scope-bounding obligation and lands on a stable baseline before Option B's five-site rewrite begins. Option B then proceeds against a page whose `estimateTokens` paragraph is already in its terminal shape, so no rebase coordination is required between the two passes.

Edge cases the implementer should watch:

- **Do not edit the live seam callouts.** The `> **loom 1.0 seam ‚Äî per-call timeout.**` blockquotes on `tool-calls.md` (line 42) and `invocation.md` (line 46), and the `> **loom 1.0 seam ‚Äî mid-loom user-session replacement.**` blockquote on `conversation-drive.md` (line 20), are the normative anchors. - **Preserve every anchor on the affected paragraphs.** `<a id="tool-execution-from-loom-code">`, the `compact`-row anchor inside the `ExtensionContext` block, the `<a id="estimatetokens-named-export">` anchor at the top of the file, and the per-mode override table itself are all link targets from `plan_topics/V14a-tool-calls.md`, `plan_topics/coverage-matrix.md` (`cka-46`), and `plan_topics/V11i-session-context-truncation.md`.
