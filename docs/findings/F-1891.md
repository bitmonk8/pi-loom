## Finding

`PIC-49. Cross-window rebuild serialization` (anchored `#pic-49` in `spec_topics/pi-integration-contract/registration-steps.md`) is an executable runtime MUST: at most one rebuild runs against the live `LoomRegistry`, AJV validator cache, and prompt-mode registration cache at a time. A debounce timer that fires while a prior window's rebuild is still awaiting its asynchronous steps (re-parse, AJV recompilation, `pi.registerTool`) MUST NOT start a concurrent rebuild; it defers until the in-flight rebuild's single synchronous publish — or its `loom/runtime/registry-swap-failed` discard (PIC-36) — completes, then runs against the now-current live state, re-parsing the changed-file set observed up to its own debounce close.

No plan leaf implements or verifies this obligation, and PIC-49 appears in neither the Numbered REQ-IDs table nor any code-keyed (`cka-<n>`) row of `coverage-matrix.md`. The adjacent leaves cover different obligations: `V9b` closes the registry-swap / structural-note / getCommands family (PIC-36–39 via `cka-23`); `V10d` closes only the within-window 250 ms drop-and-reschedule coalescing (`cka-36`) plus the settings-re-merge failure-injection arm, with no live watcher and no across-window in-flight tracking; `V15e` closes hot-reload static-resolution cache eviction. None of them serialize concurrent rebuilds across debounce windows.

The coverage-matrix header asserts every executable spec REQ-ID is mapped to a closing leaf. PIC-49 is an anchored, live numbered REQ-ID with no mapping and no citing test, so it is both a behavioural coverage gap and a matrix-completeness gap.

## Solution Space

**Shape:** multiple

The defect is a single coverage gap (PIC-49 has no closing leaf and no matrix mapping); the choice is *where* to close it. Each option below fully resolves the finding on its own — it adds the serialization behaviour plus a citing test to a leaf and adds the PIC-49 matrix mapping. The fix is plan-side only; `registration-steps.md` §`#pic-49` is read-only for this fix.

### Option A — Close PIC-49 on `V10d`

- **Approach:** `V10d` owns the `Clock`-driven 250 ms reload debounce. Add the across-window serialization concern at the debounce→rebuild boundary it already owns: a debounce timer firing while a prior rebuild is still awaiting its async steps defers rather than starting a concurrent rebuild; the deferred rebuild re-parses the changed-file set observed up to its own debounce close against post-publish live state; the in-flight guard is released on the in-flight rebuild's single synchronous publish or on its `loom/runtime/registry-swap-failed` discard.
- **Plan edits:** extend `V10d` Adds with the serialization arm and add a Tests bullet citing `registration-steps.md#pic-49`; add a Numbered REQ-IDs row `PIC-49 → V10d` in `coverage-matrix.md` with a PIC-49-citing test. `V10d` already declares `Deps. … V9b`, so the publish/discard completion signal that releases the guard needs no new dependency edge.
- **Pros:** PIC-49's trigger is literally a debounce timer firing mid-rebuild — the boundary `V10d` owns; `V10d` already drives `FakeClock` and asserts coalescing, so the deferral assertion reuses that harness; the existing `V10d → V9b` dependency already exposes the rebuild lifecycle.
- **Cons:** widens `V10d` beyond single-window coalescing + settings-re-merge injection; the guard-release-on-`registry-swap-failed` reads the swap lifecycle that `V9b` owns.
- **Risks:** the deferral must not drop an edit observed during the in-flight window; the guard must release on the failure path as well as success.

### Option B — Close PIC-49 on `V9b`

- **Approach:** `V9b` owns the build-aside-then-publish rebuild/swap mechanism and the `loom/runtime/registry-swap-failed` outcome (PIC-36). Place the at-most-one-rebuild-in-flight guard with the rebuild lifecycle: acquired at rebuild start, released at publish or `registry-swap-failed` discard.
- **Plan edits:** extend `V9b` Adds and Tests with the serialization gate citing `registration-steps.md#pic-49`; map PIC-49 in `coverage-matrix.md` pointing at `V9b` (Numbered REQ-IDs row, or an added `cka-23` facet) with a PIC-49-citing test.
- **Pros:** the in-flight/publish/discard lifecycle and `registry-swap-failed` are `V9b`'s, so the guard release co-locates with the swap owner; PIC-49 sits beside PIC-36 in the same spec section.
- **Cons:** the "debounce timer fires while a prior rebuild is in-flight" trigger lives in `V10d`'s debounce path, not `V9b`; `V9b` is already a dense multi-facet leaf (four `cka-23` facets).
- **Risks:** facet bloat on `V9b`; the timer-firing trigger still straddles `V9b`/`V10d`.

### Option C — Close PIC-49 on a new leaf `<new>`

- **Approach:** a dedicated leaf spanning the `V10d` debounce-timer trigger and the `V9b` rebuild lifecycle, closing PIC-49 end to end.
- **Plan edits:** new leaf citing `registration-steps.md#pic-49` with `Deps` on `V9b` (rebuild/swap, publish/discard signal) and `V10d` (debounce timer); add a Numbered REQ-IDs row `PIC-49 → <new>` with a PIC-49-citing test.
- **Pros:** keeps `V10d` and `V9b` scopes intact; isolates the cross-window concern that genuinely spans both seams.
- **Cons:** introduces a leaf for a single REQ-ID and requires a new ID allocation.
- **Risks:** minimal.

### Recommendation

Close PIC-49 on `V10d` (Option A). PIC-49's normative trigger is "a debounce timer that fires while a prior window's rebuild is still awaiting its asynchronous steps" — the debounce→rebuild boundary `V10d` already owns and already exercises deterministically through `FakeClock`, and `V10d` already depends on `V9b` for the rebuild/swap lifecycle, so the publish/discard completion signal that releases the in-flight guard is reachable without a new dependency. Add a PIC-49 row to the Numbered REQ-IDs table of `coverage-matrix.md` (PIC-49 is an anchored numbered REQ-ID, parallel to the PIC-36–39 family) mapping it to `V10d`, and a `V10d` test that cites `PIC-49` inline and witnesses: a window firing mid-rebuild defers rather than starting a concurrent rebuild; the deferred rebuild re-parses the changed-file set up to its own debounce close against post-publish state; and the guard releases on both the synchronous publish and the `registry-swap-failed` discard. Edge cases the implementer must watch: the deferral must preserve an edit observed during the in-flight window (no lost edit across the deferral), and the guard must release on the `registry-swap-failed` failure path as well as success — otherwise a single failed rebuild permanently blocks every later rebuild.

## Consequence

**Severity:** correctness

PIC-49's concurrency-safety MUST ships unimplemented: with no closing leaf, an implementer building the watcher reload path produces no across-window serialization, so two rebuilds triggered in successive debounce windows can run concurrently against live state — a stale older rebuild publishing over a newer one, staged-validator construction interleaving with another rebuild's validator-cache eviction, or a `registry-swap-failed` outcome ambiguously attributed. Separately, because the matrix asserts every executable REQ-ID is mapped, the `H5a`/`H6a` live-corpus reconciliation will flag PIC-49 as an unmapped executable REQ-ID and redden the release gate, and the citing-test gate never fires for it.
