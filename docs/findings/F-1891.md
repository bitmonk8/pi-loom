## Problem

`PIC-49` (anchored `#pic-49` in `spec_topics/pi-integration-contract/registration-steps.md`) is an executable runtime MUST: at most one rebuild runs against the live registry/validator/cache state at a time, and a debounce timer firing while a prior window's rebuild is still in flight must defer rather than start a concurrent rebuild. No plan leaf implements or verifies this obligation, and PIC-49 appears in neither the Numbered REQ-IDs table nor any code-keyed row of `coverage-matrix.md`. The adjacent leaves cover different obligations (`V9b`/`cka-23` the registry-swap family, `V10d`/`cka-36` only within-window coalescing, `V15e` hot-reload cache eviction); none serialize concurrent rebuilds across debounce windows. PIC-49 is therefore both a behavioural coverage gap and a matrix-completeness gap that the `H5a`/`H6a` live-corpus reconciliation will redden.

**Verdict:** present-since-inception (of the PIC-49 REQ-ID)

**History:** `git log -S 'PIC-49' -- docs/spec_topics/` shows PIC-49 entering the corpus only at `443ee4e`. `git log -S 'PIC-49' -- docs/plan_topics/` returns no commits, and `git log -S 'PIC-49' -- docs/plan_topics/coverage-matrix.md` is likewise empty; the token "cross-window" and "serializ"/"serialis" against `docs/plan_topics/` surface no PIC-49 coverage. The gap is original to PIC-49's introduction — the spec rule was added without the paired plan-side closing leaf and matrix row landing in the same or any later commit — not a later regression that removed an earlier mapping.

## Solution approach

Close PIC-49 on the `V10d` leaf, which owns the `Clock`-driven 250 ms reload debounce. Extend `V10d`'s Adds with an across-window rebuild-serialization arm at the debounce→rebuild boundary, releasing the in-flight guard on the in-flight rebuild's synchronous publish or its `loom/runtime/registry-swap-failed` discard; `V10d` already declares `V9b` in `Deps.`, so the publish/discard completion signal is reachable without a new dependency edge. Add a `V10d` Tests bullet citing `registration-steps.md#pic-49` that witnesses the deferral and guard-release behaviour via the existing `FakeClock` harness. Add a `PIC-49 → V10d` row to the Numbered REQ-IDs table of `coverage-matrix.md`.
