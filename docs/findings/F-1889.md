## Finding

The non-`-T` half of an MVP/vertical feature pair — the leaf whose ID is the bare `<id>` and whose green tests are the coverage-closure evidence — is named two different ways across the plan corpus. The two-task TDD ritual in `conventions.md` calls it the **"implementation task"** (paired with the "tests task" `<id>-T`), and `plan.md` step 4 follows that wording. The coverage-matrix intro and the `conventions.md` §Leaf-format **Deps** paragraph instead call the same artifact the **"implementation leaf"**. No single sentence states that the two surface terms denote the same `<id>` artifact.

The two terms are not cleanly split across files: `conventions.md` itself uses both — "implementation task" at the TDD-ritual pairing and "implementation leaf" at §Leaf format. Each occurrence is anchored to the `<id>` token (and the ritual line literally reads "Implementation task — leaf ID `<id>`"), so the referent is recoverable, but a reader scanning for the closure unit sees two names for one concept with the equivalence left implicit.

This is a presentational consistency gap, not a behavioural one. The `<id>` anchor means no implementer or gate is misdirected; the cost is only that one concept carries two surface names without an explicit bridge.

## Issue introduction

**Verdict:** present-since-feature-introduction
**Introducing commits:** `f9c5354` ("pi-loom plan: add scope section + two-task TDD pairing convention", 2026-06-09), `c6a664e` ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** The two-task pairing nomenclature — "tests task" / "implementation task" — entered `conventions.md` at `f9c5354`, the commit that introduced the TDD pairing convention; the same commit also added the "implementation leaf" phrasing to `conventions.md` §Leaf format. The matching "implementation leaf" wording in `coverage-matrix.md`'s intro was set the following day at `c6a664e`. Both surface terms therefore co-arrived with the pairing convention itself; the dual naming has been present since that feature entered the corpus, not introduced by a later divergent edit.

## Solution Space

**Shape:** single

### Recommendation

Establish the equivalence once in the authoritative document. At the `conventions.md` §Per-phase TDD ritual definition of the implementation task ("**Implementation task** — leaf ID `<id>`"), add a brief clause noting that the coverage matrix and §Leaf-format **Deps** refer to this same `<id>` artifact as the "implementation leaf" — so a reader meeting either term can resolve it to the other.

"leaf" is the plan's unit-of-work term ("Leaves are the unit of work") and "implementation leaf" already appears in both `coverage-matrix.md` and `conventions.md` §Leaf format, so prefer keeping "implementation leaf" wherever the closure/coverage artifact is named, and retain "implementation task" only where the tests-task/implementation-task pairing symmetry is the point (the TDD ritual and `plan.md` step 4). The one bridging clause covers the remaining cross-reference. Do not retitle leaf-ID tokens or alter the `coverage-matrix.md` closing-leaf cells the `H5d` tokeniser reads.

## Consequence

**Severity:** cosmetic

The same `<id>` artifact is described as both "implementation task" and "implementation leaf". Because every occurrence of each term is anchored to `<id>` (and the ritual defines the implementation task as "leaf ID `<id>`"), no implementer is misdirected and no coverage gate is affected; the only cost is two surface terms for one concept without an explicit equivalence.
