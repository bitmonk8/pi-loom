## Finding

The `*Pi-owned snapshot.*` editorial note in `host-interfaces-core.md` is repeated verbatim four times — at lines 43, 47, 75, and 121 — each copy carrying the identical text: "The enumerated member set is normative on the loom runtime — members deliberately omitted remain MUST-NOT-call (an arm (a) behavioural constraint per [GOV-18 arm (a)]) — while each member's TypeScript signature is a non-normative snapshot of the Pi-owned shape at the [loom 1.0 Pi-SDK pin], re-validated on each Pi minor bump per [Pi version bump procedure] below."

The note carries a single normative idea — member set normative, TypeScript signature a non-normative re-validated snapshot — and applies uniformly to every interface block in the section. Stamping the full paragraph after each of the four member-pinning blocks is pure duplication: the rule is identical at each site, so the repetition adds reading volume and four independent edit points for a single concept without adding any per-site content. A future signature- or wording-change to the rule must be applied four times to stay consistent.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** `15c0f17`
**History:** All four verbatim `*Pi-owned snapshot.*` notes were added in a single commit (`15c0f17`, 2026-06-24, "add uniform Pi-owned-snapshot markers to the four host-interfaces-core.md member-pinning surfaces"), which itself resolved an earlier finding by stamping a uniform marker onto each member-pinning surface. `git log -S 'Pi-owned snapshot.'` over the file returns only this commit, and its diff adds the phrase exactly four times, so the duplication entered wholesale at that commit. The uniformity goal was met by literal copy of identical paragraph text rather than by a single standing rule plus back-references — the defect is the chosen shape of that fix, not a later regression.

## Solution Space

**Shape:** single

### Recommendation

Promote the note to a single standing rule stated once, before the first interface block in the section — e.g. a preamble note: "Unless stated otherwise, the interface snapshots in this section are all Pi-owned: the enumerated member set is normative (members deliberately omitted remain MUST-NOT-call, an arm (a) behavioural constraint per [GOV-18 arm (a)]), while each member's TypeScript signature is a non-normative snapshot of the Pi-owned shape at the [loom 1.0 Pi-SDK pin], re-validated on each Pi minor bump per [Pi version bump procedure]." Then delete the three later verbatim repetitions, replacing each with a terse back-reference such as `*(applies per the Pi-owned snapshot rule above)*` where a local reminder helps the reader.

Preserve the GOV-18 arm (a), Pi-SDK-pin, and version-bump-procedure cross-links in the consolidated rule so the per-site normative force is unchanged. If any one of the four current sites carries a site-specific qualification beyond the shared text, keep that qualification inline at that site rather than folding it into the standing rule.

## Consequence

**Severity:** cosmetic

No implementer or runtime observer diverges: the four copies are byte-identical, so the rule reads the same wherever it is encountered. The cost is maintenance — the rule has four edit points that can drift apart under future edits — and reading overhead in an already dense section.
