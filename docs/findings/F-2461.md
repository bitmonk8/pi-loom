## Problem

The `loom/runtime/internal-error` row in `code-registry-runtime.md` populates its `hint` field from the caught throw's `error.stack`, guarded only by a falsy check (rendering `<no stack available>` when `error.stack` is falsy). JavaScript permits `throw <any value>`, so the caught value's `.stack` is not guaranteed to be a safely-readable string. A throwing `.stack` getter throws synchronously during emission of the very last-resort defect diagnostic meant to report the originating throw, and a truthy non-string `.stack` passes the falsy guard and is carried into `hint` as a non-string value. The companion message read is hardened against exactly these cases by the underlying-error coercion at `placeholder-rendering-b.md#underlying-error-coercion`, but that discipline is never applied to the `.stack` read.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `f8f4813` (2026-05-05) "pi-loom plan: resolve \"Panic routing has no defined path for unexpected interpreter exceptions\"" — introduced the `loom/runtime/internal-error` row whose `hint` reads `error.stack` guarded only by `"<no stack available>"` when `error.stack` is falsy. No coercion existed anywhere at this point.
- `bf85092` (2026-06-06) "pi-loom spec: resolve \"non-Error-throw coercion + session_start pi.getCommands() failure contract\"" — introduced the §6 underlying-error coercion that rigorously handles non-Error throws (undefined / non-string / throwing) for the *message* read and the enumerated message-bearing fields, but did not extend that discipline to the `.stack` read.

**History:** `git log -G 'for operator triage'` and `git log -S 'error.stack'` over `code-registry-runtime.md` (and its pre-split `docs/spec` ancestry) place the falsy-only `error.stack` read at the row's inception in `f8f4813`. `git show bf85092 -- code-registry-runtime.md` shows that commit added the §6 cross-reference preamble (message coercion) without touching the `.stack` read. The defect was latent from `f8f4813` and became a visible asymmetry once `bf85092` hardened the message path while leaving the stack path on the original falsy-only guard.

## Solution approach

Rewrite the `loom/runtime/internal-error` row's `.stack` handling so the read routes through the underlying-error coercion posture at `placeholder-rendering-b.md#underlying-error-coercion`, replacing the falsy-only guard. The read must resolve to the existing `<no stack available>` sentinel whenever it does not yield a non-empty string — covering the property access itself throwing, a truthy non-string value, and the already-handled falsy case — so emission can never throw and `hint` is always a string.
