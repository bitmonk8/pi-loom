## Finding

The `loom/load/invalid-tool-rename` Tests bullet in `V6c` (and its paired `V6c-T`) reads: "a `tools:` `as` rename target that is not loom-identifier-shaped (e.g. `as MyTool`) is rejected at load time." The chosen example, `MyTool`, is a conventional, well-formed-looking identifier in most languages. The leaf does not state the property that actually makes it invalid: loom identifiers must begin with a lowercase letter (the binding-case rule whose violation `V1a` fires as `loom/parse/binding-case-mismatch`). A reader working only from `V6c`/`V6c-T` cannot tell why `MyTool` is rejected and may read the example as arbitrary or even mistaken.

The spec the leaf cites, `frontmatter-fields-a.md` §`tools`, uses the identical example but states the reason inline: "The override target must obey loom's lowercase-first identifier rule (`./summarise.loom as MyTool` is `loom/load/invalid-tool-rename`)." The plan bullet dropped the rule and kept only the example, so the leaf no longer carries the discriminator that distinguishes a valid rename target from an invalid one.

## Issue introduction

**Verdict:** introduced-by-a-later-edit
**Introducing commits:** `60c9400` ("pi-loom plan: resolve \"tools:-resolution load codes have no asserting plan leaf\"", 2026-06-17)
**History:** The `loom/load/invalid-tool-rename` bullet with the `as MyTool` example was added to both `V6c-tools-set.md` and `V6c-T-tools-set.md` in commit `60c9400`, which introduced the four `tools:`-resolution load-code Tests bullets in response to a prior review finding. `git log -S 'as MyTool'` shows no earlier occurrence of the example in the plan corpus; the bullet's wording (example without the lowercase-first rule the spec carries) has been unchanged since that commit.

## Solution Space

**Shape:** single

### Recommendation

In the `loom/load/invalid-tool-rename` Tests bullet of both `V6c-tools-set.md` and `V6c-T-tools-set.md`, make the example self-explanatory by naming the rule it violates, mirroring the spec's own phrasing in `frontmatter-fields-a.md` §`tools`. Replace the parenthetical `(e.g. \`as MyTool\`)` with text that states the lowercase-first requirement, e.g.:

> `loom/load/invalid-tool-rename`: a `tools:` `as` rename target that is not loom-identifier-shaped — e.g. `as MyTool`, whose uppercase first letter violates loom's lowercase-first identifier rule — is rejected at load time.

Alternatively, the same goal is met by swapping `MyTool` for a self-evidently malformed target (such as `as my-tool` or `as "myTool"`); naming the rule is preferred because it ties the bullet to the authoritative spec statement and to `V1a`'s binding-case diagnostic. Apply the identical edit to both leaves so the `-T` tests-first pair stays byte-aligned with its implementation leaf.

## Consequence

**Severity:** advisory

An implementer reading `V6c`/`V6c-T` in isolation cannot tell why `as MyTool` is rejected and may write a test that mis-identifies the rejection condition (e.g. asserting against the wrong target shape) or search the spec to recover the rule. The spec is authoritative and `V6c-T` still pins the diagnostic code, so a working leaf can still be produced; the cost is wasted lookup and a weaker self-contained test bullet.
