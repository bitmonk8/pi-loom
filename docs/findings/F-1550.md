## Finding

The *Enforcement point* paragraph in `schema-subset.md` opens: "The depth check **lives in** the `SchemaValidator` service (the same component that owns the AJV cache; see [Implementation Notes — Runtime])." This is a normative statement about *which internal component hosts* the depth walk, not about observable behaviour.

The caller-observable contract for the depth check is fully specified by the surrounding text: it runs at the five enumerated validation sites, it runs *before* AJV at each site, and it produces the pinned `maxDepth` / `"JSON document depth exceeds 5"` error shape with the per-boundary routing table. Nothing an implementer, loom author, or tool can observe depends on whether the walk is physically hosted by the `SchemaValidator` service, by a free function called ahead of it, or by any other internal arrangement.

The mandate also sits awkwardly against the spec's own seam definition. `SchemaValidator`'s normative behavioural contract (PIC-11 in `pi-integration-contract/host-interfaces-services.md`) is an AJV-validation surface (`compile` / `invalidate`, one-pass multi-error, no convert/defaults, slug-cache byte-verify); it does **not** include the depth walk, which by contract runs *before* AJV. PIC-11 further states that the seam's member decomposition "is not itself binding," and GOV-18 arm (a) defines binding scope as externally-observable behaviour on `.loom`/`.warp` inputs — internal component layout is outside it. The "lives in `SchemaValidator`" sentence therefore prescribes a structural placement the corpus elsewhere declares non-binding.

**Verdict:** introduced-by-commit
**Introducing commits:** `216130d` (2026-05-04, "pi-loom spec: resolve \"Depth ≤5: counting algorithm and enforcement point unspecified\"")
**History:** Before `216130d` the `Depth ≤ 5` bullet in `schema-subset.md` carried no Depth-Enforcement section at all (only a one-line cap with a parenthetical about OpenAI's cap). `216130d` introduced the whole `## Depth Enforcement` section in one edit, including the *Enforcement point* paragraph whose opening sentence pins the check to "the `SchemaValidator` service (the same component that owns the AJV cache)". `git log -S 'lives in the \`SchemaValidator\`'` returns only that commit (plus the later `81bf52a` path-reshuffle that moved `spec_topics/` under `docs/` without altering the sentence and `f7a505e` which only touched an internal cross-reference target inside the same section). The named-component binding was therefore present from the moment the Depth-Enforcement section entered the corpus.

## Solution Space

**Shape:** single

### Recommendation

State the enforcement point behaviourally and drop the component-placement mandate. Replace the opening sentence of the *Enforcement point* paragraph with one that asserts only what is observable — that a depth check runs at the five enumerated sites and runs before AJV at each — and remove "lives in the `SchemaValidator` service (the same component that owns the AJV cache …)." Keep the `Implementation Notes — Runtime` cross-reference if useful, but as informative navigation rather than a placement requirement. If co-location with the AJV/validator cache is genuinely load-bearing, express it as a behavioural constraint (e.g. cache-keying or invalidation behaviour) rather than as a "lives in" rule. This aligns the paragraph with PIC-11's non-binding-decomposition note and GOV-18 arm (a).

## Consequence

**Severity:** cosmetic

A normative sentence over-constrains internal component layout with no caller-observable effect, contradicting the corpus's own non-binding-decomposition posture (PIC-11 / GOV-18 arm (a)). No implementer would diverge or produce wrong behaviour; removing the mandate changes nothing observable.
