## Finding

The opening **Provider error mapping.** paragraph of `provider-error-mapping.md` ends with an inline `*Re-validation gate (loom 1.0.0).*` sub-paragraph that mixes two kinds of content in consecutive sentences with no seam between them. First a normative loom-1.0 obligation: a contributor performing a Pi bump "SHOULD run them [the provider-error fixtures] against the candidate `@earendil-works/pi-ai` minor per item (i) of the *Editorial-review checklist for unpinned host presuppositions*". Immediately after it, a deferred-work declaration: "Wiring this re-validation into the bump procedure as a mechanical gate is a recognised post-loom 1.0.0 maintenance follow-up."

The deferred sentence describes work explicitly out of scope for loom 1.0 (CI-wiring the fixture re-run as a mechanical bump-procedure step). It carries no normative force for the loom-1.0 implementer, yet it sits inside a normative requirement paragraph with no non-goals box, no seam marker, and no bullet in the Future Considerations deferral catalogue. The project already maintains a canonical home for exactly this kind of note — `future-considerations/surface-extensions.md` §"Tooling deferrals (no loom 1.0 impact)" — whose existing bullets (e.g. "Automated cadence for re-importing Pi patches") follow the established pattern of naming a deferred tooling item and pointing back at the topic-page paragraph that owns the loom-1.0 seam. A reader scanning that catalogue for deferred work will not find this CI-gate deferral, because it is buried inside a normative paragraph on a different page.

A second paragraph on the same page (the `*Provider-owned-wording presupposition.*` paragraph) closes with a back-reference — "See *Re-validation gate (loom 1.0.0)* above for the corresponding post-loom 1.0.0 maintenance follow-up." — so the deferred note is also a citation target, which constrains how it can be cleaned up.

## Solution Space

**Shape:** single

### Recommendation

Relocate the deferred-work sentence out of the normative paragraph into the existing deferral catalogue, rather than deleting it.

- In `provider-error-mapping.md`, remove the clause "Wiring this re-validation into the bump procedure as a mechanical gate is a recognised post-loom 1.0.0 maintenance follow-up." from the `*Re-validation gate (loom 1.0.0)*` sub-paragraph, leaving that sub-paragraph stating only the loom-1.0 facts: the fixtures exist under `npm test`, the contributor SHOULD run them against the candidate minor per item (i) of the editorial-review checklist, and the checklist does not yet enumerate the run as a mechanical step.
- Add a bullet to `future-considerations/surface-extensions.md` §"Tooling deferrals (no loom 1.0 impact)" naming the deferral ("CI-wired bump-procedure step that mechanically re-runs the provider-error fixtures against a candidate `@earendil-works/pi-ai` minor") and pointing back at the *Re-validation gate (loom 1.0.0)* paragraph for the loom-1.0 seam it defers — matching the existing "Automated cadence for re-importing Pi patches" bullet's shape.
- Update the back-reference in the `*Provider-owned-wording presupposition.*` paragraph: its sentence "See *Re-validation gate (loom 1.0.0)* above for the corresponding post-loom 1.0.0 maintenance follow-up." must repoint to the new Tooling-deferrals bullet (or be removed if the *Re-validation gate* paragraph no longer carries the deferral), so the back-reference is not left dangling.

Relocation is preferred over outright deletion because the deferral is a real, recorded loom-1.0 scoping decision with an existing canonical home and a live citation; deleting it would lose the recorded deferral and orphan the back-reference.

## Consequence

**Severity:** cosmetic

No observer behaviour changes and no implementer is misled into producing a wrong system: the SHOULD is plainly normative and the maintenance-follow-up clause is plainly labelled prose. The cost is purely organisational — the loom-1.0 deferral of CI-gate wiring is undiscoverable from the Future Considerations catalogue that exists to enumerate exactly such deferrals.
