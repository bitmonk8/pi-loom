## Finding

GOV-9 in `docs/spec_topics/governance/req-id-prefix-table-active-b.md` opens with: *"Each spec page that **depends on a normative rule from another topic page** MUST either **state the rule locally** or reference it by a markdown link whose target is the specific REQ-ID anchor (`#prefix-n`) of the depended-upon rule."* Both halves of that MUST are mechanically undecidable:

- The **trigger predicate** — "depends on a normative rule from another topic page" — has no structural witness. Deciding whether a sentence on page *A* depends on a rule on page *B* requires reading both rules and judging whether *A*'s prose presupposes *B*'s obligation. No surface signal (cross-link, REQ-ID token mention, shared identifier) is necessary or sufficient: a page can depend on another's rule without naming any token from it, and a page can name a token from another's rule purely illustratively.
- The **compliance arm** — "state the rule locally" — is by construction indistinguishable from original content. A checker comparing a candidate paragraph against another page's rule has no oracle for "states the same rule"; the spec itself concedes the analogous "semantic equivalence between two prose paragraphs is not mechanically decidable" in GOV-8 and GOV-30.

The downstream universal-resolvability obligation in the same paragraph (*"Every REQ-ID anchor on a non-narrative spec page MUST resolve as a `#prefix-n` URL fragment"*) and the progressive-coinage rule in GOV-22 are both mechanically tractable and are correctly framed as such. They do not rescue the opening MUST, which still binds in its undecidable form. A reviewer asked to enforce GOV-9 on a PR has no way to falsify the "state the rule locally" arm; two reasonable reviewers will diverge on which paragraphs trigger the MUST in the first place.

## Solution Space

**Shape:** multiple

### Option A — Narrow the MUST to a structurally decidable sub-case

**Approach.** Replace the opening MUST with one phrased over surface tokens only: every occurrence of a `PREFIX-N` token on a page that does not own `PREFIX` (i.e. whose row in the REQ-ID prefix table carries a different prefix, or the `(no IDs — narrative)` cell) MUST appear either as the link text of a markdown link whose target fragment is `#prefix-n` on the owning page, or inside an inline code span, fenced code block, or HTML comment (the GOV-3 exclusion list). The "state the rule locally" alternative is dropped — a page that genuinely restates the rule locally without naming the foreign `PREFIX-N` token is already outside the trigger.

**Spec edits.** Rewrite GOV-9's opening sentence; keep the narrative-page section-link allowance (it is already structural: "row carries `(no IDs — narrative)`"); keep the universal-resolvability sentence and the GOV-22-residue clause unchanged. Add a non-normative note stating that paraphrased local restatement is permitted and out of this MUST's scope by construction.

**Pros.** Mechanically decidable (a single regex pass over `spec_topics/**` against the prefix table); composes cleanly with GOV-22's progressive coinage and GOV-1's dual-form delivery; matches the framing the spec already uses for the analogous "every numbered REQ-ID anchor MUST resolve as `#prefix-n`" downstream sentence.

**Cons.** Strictly weaker than the current text — a page that depends on another's rule without naming any `PREFIX-N` token escapes the MUST. In practice GOV-22 ensures every defining obligation site eventually carries a `PREFIX-N`, so the residue is bounded, but the gap exists during the progressive-drain window.

**Risks.** A future page that legitimately quotes a sibling's `PREFIX-N` in narrative prose (e.g. *"unlike PIC-12, which forbids …"*) is dragged under the MUST. The GOV-3 exclusion list (code spans, fences, HTML comments) covers the common escape hatches, but reviewers must remember to reach for them.

### Option B — Demote the semantic trigger to SHOULD, keep the structural sub-MUST

**Approach.** Split GOV-9's opening sentence into two: a SHOULD (or lowercase *should*) recording the editorial preference that pages either restate a rule locally or link to its `#prefix-n` anchor, plus a MUST narrowed to the structural sub-case Option A defines. Add a parenthetical that the SHOULD half is reviewer-enforced and has no mechanical witness, mirroring the disclaimer GOV-30 already uses for aggregator lock-step.

**Spec edits.** Same structural sub-MUST as Option A. Above it, a SHOULD-phrased editorial sentence covering the broader semantic intent. Update the GOV-22 cross-reference if it still cites "GOV-9's `#prefix-n` cross-link contract" — the contract is now carried by the structural sub-MUST, not by the SHOULD.

**Pros.** Preserves the spec's stated intent (cross-page citation prefers `#prefix-n` anchors) as a recorded editorial preference, without leaving an undecidable MUST in the corpus. Consistent with the spec's existing pattern (GOV-30 already carries the "documentation defect, not normative defect" disclaimer for analogous undecidable surfaces).

**Cons.** Adds a second clause to a paragraph this finding's sibling reviews already flag as over-stuffed. Reviewers must distinguish the SHOULD's editorial scope from the MUST's structural scope on every PR that touches a cross-page citation.

**Risks.** The "uppercase SHOULD vs not a normative defect" finding (sibling to this one, against GOV-30) shows that uppercase SHOULDs paired with "documentation defect" disclaimers confuse implementers — repeating that pattern here would invite the same finding against GOV-9.

### Recommendation

Adopt **Option A**. The structural sub-MUST is the actual contract that GOV-1's dual-form delivery and GOV-22's progressive coinage are designed to satisfy; recording it as the sole MUST aligns GOV-9 with the rest of the structurally decidable governance surface. Drop the "state the rule locally" arm and the semantic trigger entirely — they are unfalsifiable today, and the legitimate use case (a page restating a rule rather than linking it) is automatically out of the narrowed trigger's scope. Implementer edge cases to watch: (i) ensure the narrowed MUST's prefix-ownership predicate reads the prefix table at the *commit's* state, not a snapshot, so a GOV-7 *Add* in the same commit is honoured; (ii) confirm the existing GOV-22 back-link still resolves — it cites "GOV-9's `#prefix-n` cross-link contract," which the narrowed MUST continues to carry, but the GOV-9 paragraph text the cross-link lands in changes shape.

## Consequence

**Severity:** correctness

GOV-9's opening MUST is the spec's only obligation governing *cross-page* citation form, and as written it is unfalsifiable. Two reviewers comparing the same PR will disagree on (a) which paragraphs trigger the MUST, and (b) whether a paraphrased local restatement satisfies it. In practice the rule degrades to "reviewer intuition wearing a MUST hat" — the same posture GOV-30 explicitly carves out as editorial convention, but without GOV-9's `MUST` admitting it. The drafted contract with GOV-1 / GOV-22 (universal `#prefix-n` resolvability, progressive coinage of un-anchored sites) is the part that actually carries weight; the un-rescued semantic trigger above it is dead text that invites inconsistent enforcement.
