## Finding

`req-id-prefix-table-active-b.md` exists, by its filename and by its lead in the parent `governance.md` index ("Req id prefix table active b"), as the second physical shard of the live REQ-ID prefix table — its remit is the prefix → page binding mechanics (GOV-5 disjoint-prefix invariant, GOV-6 table-completeness, GOV-7 mutation procedures, GOV-8 REQ-ID lifecycle, GOV-9 cross-link form, GOV-22 progressive coinage, GOV-24 per-row invariant) and the live prefix-table data rows themselves.

Two rules in the same file — GOV-30 (`spec.md` aggregator paragraphs are informative; lock-step convention) and GOV-31 (integer-count aggregator literal-preservation invariant) — own a different concern. They govern how the informative orientation hub (`spec.md`) lock-steps with the topic-page obligations it forward-links to: the *multi-source-page* and *single-source-page* aggregator shapes, the bidirectional-referent-identity criterion, the integer-literal-preservation MUST, and the in-code-constant carve-out. None of that is about REQ-ID prefix allocation, prefix retirement, anchor placement, or prefix-table-row mechanics, and neither rule consults the prefix table.

A reader looking for aggregator-paragraph conventions has no signal that they live in a file named for the prefix table, and a reader of the prefix-table file gets a long topic excursion mid-shard between GOV-22 and GOV-24. The two rules were placed here mechanically when the unsplit `governance.md` hub was carved up by line range under the 30KB-per-file recommendation; their topical home was never reconsidered. The defect is purely organisational, not normative.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `126ff8a` (2026-05-06, "pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B") coined the aggregator-paragraph convention as GOV-12, co-resident with the rest of the prefix-table mechanics in the still-unsplit `docs/spec_topics/governance.md`. `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set") split `governance.md` into shards via `docs/_tools/split_spec.py`'s line-range plan; the plan entry `("req-id-prefix-table-active-b", 101, 162)` mechanically swept GOV-12 into the second prefix-table shard along with GOV-7/GOV-8/GOV-9/GOV-22/GOV-24, with no topical reconsideration. `fe8aa5c` (2026-06-05, "pi-loom spec: resolve 'GOV-12 bundles ≥3 independently-testable sub-obligations'") then renamed GOV-12 → GOV-30 and split off GOV-31 in place — preserving the misplacement and doubling the affected rule count.
**History:** The defect is the interaction of the rule's original co-location with the prefix-table mechanics (a pre-split editorial choice) and the line-range-keyed split that promoted that co-location into a separate-file misfiling. The durable fix is to relocate the two rules to a topically aligned shard; neither single commit on its own constitutes the defect.

## Solution Space

**Shape:** multiple

### Option A — Append GOV-30 / GOV-31 to `corpus-direction-and-scope.md`

**Approach.** Move the two `<a id="gov-30"></a> **GOV-30 …**` and `<a id="gov-31"></a> **GOV-31 …**` paragraphs out of `req-id-prefix-table-active-b.md` and append them at the tail of `corpus-direction-and-scope.md`, which already owns GOV-17 (corpus direction) and GOV-18 (binding scope). Update `docs/_tools/split_spec.py` so its `PLANS["spec_topics/governance.md"]` line ranges reflect the new file content; on the next regeneration the splitter will reproduce the same physical layout. Update the `governance.md` intro paragraph's "GOV-1, GOV-3 through GOV-9, GOV-15, GOV-16, GOV-30, GOV-31" enumeration only insofar as it names a containing file (it does not, so no change is required there).

**Spec edits.** Remove the two paragraphs from `req-id-prefix-table-active-b.md` (between GOV-22 and GOV-24). Append them to `corpus-direction-and-scope.md` under a new sub-section header (e.g. `## Aggregator-paragraph conventions`). Repoint every in-corpus cross-link `…/governance/req-id-prefix-table-active-b.md#gov-30` and `…#gov-31` to `…/governance/corpus-direction-and-scope.md#gov-30` / `…#gov-31` (seven files listed under *Spec Documents*).

**Pros.** Zero new files; reduces the prefix-table shard from ~29KB to ~14KB (well under the 30KB recommended cap, restoring headroom for future GOV-7/GOV-8 edits); reinforces `corpus-direction-and-scope.md` as the home for cross-corpus / orientation / binding-scope rules, which is the natural topical adjacency for the `spec.md`-aggregator convention; the file at ~24KB after the move still sits under the recommended cap.

**Cons.** Inflates `corpus-direction-and-scope.md` from ~9KB to ~24KB — a single sub-section will dominate the file by line count.

**Risks.** None beyond the routine link-repointing pass; the moved paragraphs are self-contained, all cross-link consumers are listed.

### Option B — Create a new shard `aggregator-paragraph-conventions.md`

**Approach.** Add a new file `docs/spec_topics/governance/aggregator-paragraph-conventions.md` holding GOV-30 and GOV-31 in their entirety. Add a Contents-list entry to `governance.md`. Add a new `PLANS` line-range entry for it in `docs/_tools/split_spec.py`.

**Spec edits.** Remove the two paragraphs from `req-id-prefix-table-active-b.md`. Author the new shard with H1 + lead paragraph (matching the sibling shards' shape) and the two paragraphs verbatim. Repoint the seven consumer files' cross-links to the new path. Add `- [Aggregator paragraph conventions](./governance/aggregator-paragraph-conventions.md)` to `governance.md`'s Contents list.

**Pros.** Cleanest topical separation — one rule cluster per file, mirroring how `source-language-stability.md` and `stable-inline-labels.md` are already isolated; small file (~14KB) leaves ample headroom for future amendments to either rule.

**Cons.** Adds an eighth shard under `governance/`; a casual reader scanning the Contents list sees one more entry; `split_spec.py` needs a new PLANS entry rather than only line-range adjustments.

**Risks.** None beyond Option A's risks.

### Recommendation

Take **Option A**. The two rules' topical neighbour is GOV-17/GOV-18 (corpus direction, binding scope) — both already on `corpus-direction-and-scope.md` — and Option A discharges the misplacement without growing the shard count. The file-size arithmetic favours it: it drops the over-loaded prefix-table shard from ~29KB to ~14KB and lifts the corpus-direction shard from ~9KB to ~24KB, leaving both comfortably under the 30KB recommended cap. The implementer must repoint the seven cross-link consumers in the same commit (their paths are listed under *Spec Documents*) and update `docs/_tools/split_spec.py`'s `PLANS["spec_topics/governance.md"]` line ranges so re-splits reproduce the new layout. Resolve this finding before any GOV-30 / GOV-31 wording edit (e.g. the GOV-30 SHOULD-vs-defect finding) so that downstream edits land on the relocated home.
