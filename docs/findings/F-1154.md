## Finding

`discovery-sources.md` defines the `--loom` CLI flag as a single string split on `path.delimiter`, with each component "resolved with the same rules as the settings `loomPaths` array (see [`loomPaths` entry schema](./package-and-settings.md#loompaths-entry-schema) below)". The settings entry schema in `package-and-settings.md` keys its resolution base on *which settings file the entry came from* — entries in `~/.pi/agent/settings.json` resolve relative to `~/.pi/agent/`; entries in `.pi/settings.json` resolve relative to `.pi/`. The CLI flag has no settings file, so this cross-reference pins nothing for a bare relative component such as `--loom looms/extra.loom`.

Absolute paths and `~/`-prefixed paths are covered (the latter by DISC-1's `homedir()`-via-`FileSystem` rule, listed explicitly for the `--loom` flag). The gap is purely the relative-path branch, which the spec neither defines nor rejects. The behaviour is observable: two implementers can ship `--loom looms/extra.loom` resolving against `process.cwd()`, against the captured factory-time `FileSystem.cwd()` value, against the project root carried on a `resources_discover` event, or as a hard rejection, and the spec gives no test fixture or diagnostic to discriminate. The cross-reference further inherits an unrelated semantic — that settings entries deduplicate "post-tilde-expansion" against the resolved absolute path — and that dedup is meaningless until the resolution base is pinned.

## Solution Space

**Shape:** single

### Recommendation

Amend the CLI bullet in `discovery-sources.md` so the `--loom` source pins its own resolution base rather than inheriting the settings-file-keyed one. State: each component is first home-expanded per DISC-1; an absolute path (post-expansion) is taken as-is; a relative path resolves against the project root — the value of `event.cwd` on a `resources_discover` event, falling back to the `FileSystem.cwd()` seam member ([PIC-13](../pi-integration-contract/host-interfaces-services.md#pic-13)) at factory time, matching the precedence already pinned in `registration-steps.md` step 1 for the project-local discovery scan. Carry the cross-reference to PIC-13 inline so the load-bearing fact (the seam is the single source of truth; production captures `process.cwd()` once at construction) is visible at the consumption site, not only at the inherited link.

Edge cases the implementer must handle:

- A relative component whose post-resolution absolute path lies on the same path as a settings-resolved or `pi.looms`-resolved component is silently deduplicated by the existing rule ("Settings entries that resolve to the same absolute path post-tilde-expansion are deduplicated silently"). Confirm the dedup statement reads on the *resolved* absolute path so it covers a relative-CLI match against an absolute-settings entry.
- A relative component supplied at factory time but invoked under a session whose `event.cwd` differs from the factory-captured value resolves against the factory value at factory-scan time and against `event.cwd` on the next `resources_discover` reload — already the pattern step 1 of registration-steps.md pins for the project-local scan, applied here uniformly.
- The chosen base does **not** alter the failure-modes table: a relative component resolving to a missing path remains a `loom/load/missing-source` error per the CLI row.

Reject the alternative of forbidding relative `--loom` paths (a `loom/load/invalid-extension`-style rejection): users running the loom binary from a project directory naturally write `--loom looms/extra.loom`, and a hard rejection breaks Pi's convention that CLI overrides accept whatever the shell hands them.

Add a `V10a` / `V10a-T` test bullet exercising a relative `--loom` component against a fixture cwd, asserting the resolved absolute path is the cwd-joined value and that an absolute-`loomPaths`-vs-relative-`--loom` pair pointing at the same file is silently deduplicated.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge on `--loom looms/extra.loom` invoked from an arbitrary working directory: one resolves against `process.cwd()`, another against the seam-captured factory-time `cwd`, another against the per-event `event.cwd`, another rejects. Each implementation passes `npm test` because no fixture exists; the divergence surfaces only when a user with a non-default cwd reports that `--loom` "doesn't see my file." The discovery-root computation, the active-roots membership test on `invoke` callees, and the cross-`--loom`-component dedup all key on the resolved absolute path, so any silent base mismatch fans out into invocation-time path-restriction errors with no diagnostic naming the cause.
