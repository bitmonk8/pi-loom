## Problem

The `--loom` CLI bullet in `discovery-sources.md` splits the flag value on `path.delimiter` and resolves each component "with the same rules as the settings `loomPaths` array". That settings rule keys its resolution base on which settings file the entry came from (`~/.pi/agent/` for global, `.pi/` for project). The CLI flag has no settings file, so the cross-reference pins nothing for a bare relative component such as `--loom looms/extra.loom`. Absolute and `~/`-prefixed components are covered (the latter by DISC-1), but the relative-path branch is neither defined nor rejected, so implementers can ship divergent resolution bases (`process.cwd()`, factory-time `FileSystem.cwd()`, per-event `event.cwd`, or rejection) that all pass the test suite.

## Issue introduction

**Verdict:** present-since-inception (in current corpus form)
**Introducing commits:**
- `b5ba88e` `spec: --loom CLI flag is single string with OS path-list separator (Decision #9)` — replaced the prior `--loom <path>` (repeatable, optional) one-liner with the current "resolved with the same rules as the settings `looms` array" cross-reference, on the (then-monolithic) `spec.md`. Decision #9's stated scope was the multi-path packing, not the per-component resolution base; the cross-reference inherited the settings-file-relative rule without addressing the absent-settings-file case.
- `fae85d3` `Tighten spec for implementation-plan readiness` — earlier inception of `--loom <path>` (repeatable, optional) with no resolution rule at all.

**History:** The defect predates the `spec.md` split into `spec_topics/` (`fecb504`) and the per-topic reshuffle that produced today's `discovery/discovery-sources.md` (`f5e89f4`). `git log -L 7,7:docs/spec_topics/discovery/discovery-sources.md` shows the current CLI bullet originated unchanged at the post-split baseline, with the underlying text dating to `b5ba88e`. No subsequent commit on the CLI line, on `loomPaths` entry resolution, or on DISC-1/DISC-2 has revisited the relative-path branch.

## Solution approach

Rewrite the `--loom` CLI bullet in `discovery-sources.md` so the flag pins its own resolution base for relative path components rather than inheriting the settings-file-keyed rule. Resolve a relative component against the project root using the precedence already pinned in `registration-steps.md` step 1 — `event.cwd` on a `resources_discover` event, falling back to the `FileSystem.cwd()` seam member at [PIC-13](../pi-integration-contract/host-interfaces-services.md#pic-13) at factory time — with home-expansion applied first per DISC-1. Carry the PIC-13 reference inline at the consumption site.
