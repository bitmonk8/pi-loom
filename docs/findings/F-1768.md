## Finding

The strict-capability probe in `docs/spec_topics/binder/binder-model-and-context.md` (the paragraph anchored `strict-capability-requirement`) states the behavioural rule as a literal TypeScript expression:

> The probe is duck-typed: `(model as { strictCapable?: boolean }).strictCapable` is read as a three-valued check ā€” `true` admits the model with no diagnostic; `false` emits `loom/load/binder-model-not-strict-capable` (E) and refuses to register the loom; `undefined` (the field is absent) emits `loom/load/binder-model-strict-capability-unknown` (W) and registers the loom.

The normative contract is the three-valued read of an optional `strictCapable` boolean off the resolved `Model<Api>` (`true` admits; `false` ā†’ not-strict-capable error; `undefined` ā†’ capability-unknown warning). That contract is fully captured by the field name, the three branches, and the two diagnostic codes. The inline type-assertion form `(model as { strictCapable?: boolean })` is implementation syntax, not an observable contract: a typed accessor helper, a declared local interface, or a narrowed `unknown` read are all behaviourally identical. By embedding one particular cast in the normative sentence, the spec invites an implementer to treat the exact surface form as mandated.

Every other normative sentence in the same paragraph follows the correct pattern ā€” naming the field, the branches, and the codes without pinning access syntax. The field *name* `strictCapable` is correctly a pinned SDK contract (it is the rename-detection hook for the version-bump procedure and the build-time surface-inventory assertion); only the cast form is over-specified.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the probe sentence so the behavioural rule stands alone, naming the field and the three-valued semantics without an embedded cast:

> The probe is duck-typed: the runtime reads the optional `strictCapable` boolean off the resolved `Model<Api>` as a three-valued check ā€” `true` admits the model with no diagnostic; `false` emits `loom/load/binder-model-not-strict-capable` (E) and refuses to register the loom; the field's absence emits `loom/load/binder-model-strict-capability-unknown` (W) and registers the loom.

If the cast is worth keeping as an audit-greppability hook (the original `9b9e0cb` rationale), demote it to a single parenthetical non-normative reference rendering attached to the same sentence and mark it as such, e.g. "(non-normative reference rendering: `(model as { strictCapable?: boolean }).strictCapable`; implementers MAY realise the read in any equivalent way)."

Preserve verbatim (out of scope for this edit):

- The three-valued branch table (`true` / `false` / absent ā†’ admit / E / W) and the two diagnostic codes.
- The MUST short-circuit clause for the unresolved-model case ("The runtime MUST short-circuit the `strictCapable` probe when binder-model resolution yields no model ā€¦").
- The probe-field-name pin ("The probe field name `strictCapable` is the loom-side constant ā€¦") ā€” the actual rename-detection hook.
- The bypass-eligibility carve-out (bypass-eligible looms skip both checks).

## Consequence

**Severity:** advisory

A strict reading of the spec arguably requires the literal cast `(model as { strictCapable?: boolean }).strictCapable` in implementation code, even though a typed helper or narrowed read is behaviourally identical and may be preferred by the project's TypeScript style. The runtime is not at risk of divergence; the cost is blurring the line between the normative behavioural contract and an illustrative surface rendering.
