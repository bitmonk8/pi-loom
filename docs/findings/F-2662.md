## Problem

The `recovery-mutex-runtime-contract` block in `tool-registration-lifetime.md` specifies a contingency-only contract: it is engaged only on a fail outcome of the version-bump item-(e) audit recipe, and loom 1.0 ships without it on a pass outcome. Yet it is authored inline mid-paragraph inside a consequences bullet alongside permanent must-build prose, with its contingency qualifier buried inline. An implementer can read it as must-build content and build the recovery mutex into loom 1.0.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** f5e89f4 — docs: enforce 100KB hard / 30KB recommended size cap on spec set (2026-06-04, Thomas Andersen); 19b193b — resolve "PIC-17 active-set swap/snapshot throws have no failure-mode rule" (2026-06-05, Thomas Andersen); 823ce54 — resolve "Recovery-mutex acquisition semantics and teardown-budget interaction undefined" (2026-06-06, Thomas Andersen); 8008b91 — fix F-1217: add acceptance-criteria/fixture blocks for four Shard-08 MUSTs (2026-06-25, Thomas Andersen)
**History:** The PIC-17 step-4 `finally` invariant and the PIC-8 restore-failure protocol were carried into this file at its first commit (f5e89f4, the size-cap split); 19b193b added the PIC-19 setup-failure protocol and 823ce54 authored the recovery-mutex contingent contract inline in the consequences bullet — each adding a failure-path MUST without a fixture. The gap crystallised at 8008b91, which added the `Acceptance criteria — PIC-17 active-set install vector` section but scoped its fixtures to the happy-path install vectors (a)/(b) only, never extending coverage to the exception path or the PIC-8/PIC-19 protocols. The defect is the interaction: acceptance-criteria coverage did not keep pace with the failure-path protocols, and the contingency block was never separated from must-build prose.

## Solution approach

Move the recovery-mutex runtime contract out of the `recovery-mutex-runtime-contract` consequences bullet into a visually-separated contingency block, cross-referenced from that bullet, so the contingency-only contract no longer interleaves with the permanent must-build prose. Preserve the contract's contingency framing (engaged only on a fail outcome of `version-bump-step2.md#bump-checklist-slash-dispatch-serialisation` item (e); not shipped in loom 1.0 on a pass outcome) in the relocated block.

## Solution constraints

- Out of scope: the PIC-17/PIC-8/PIC-19 failure-path acceptance-criteria coverage owned by F-2660 and F-2661.
- The relocation MUST NOT alter the normative content of the recovery-mutex contract (FIFO serialising `await`, no steady-state timeout, teardown interaction, per-extension-instance keying) or downgrade its contingency framing into must-build prose.
