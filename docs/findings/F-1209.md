## Finding

The file `docs/spec_topics/pi-integration-contract/runtime-event-channel.md` opens with H1 `# Runtime event channel` (line 1) and, partway down, introduces a bolded inner section with the same literal lead-in `**Runtime event channel.**` (line 40) — the paragraph that defines the always-log set, the occurrence/origin pair, and the group-A/group-B partition. Both the page-as-a-whole and that one paragraph share a single referent string. Within the file, four other paragraphs already use this referent ambiguously: "noted under **Runtime event channel** below" (line 14, *Subagent-mode display: false cascades*), two `details`-variant bullets that say "see **Runtime event channel** below" (lines 22–23), and the Success-side null-policy paragraph's "governed by the **Runtime event channel** paragraph above" (line 63). A reader cannot tell from the text whether each pointer targets the whole document or that one paragraph.

The ambiguity propagates outward. `docs/spec_topics/glossary.md`'s `always-log set`, `operator`, and `occurrence`/`origin` entries all close with `See: [Pi Integration Contract — Runtime event channel]`; `slash-invocation.md` SLSH-3 cites "[Pi Integration Contract — Runtime event channel](./pi-integration-contract/runtime-event-channel.md#runtime-event-channel)"; `errors-and-results/error-model.md` similarly anchors against `#runtime-event-channel`. None of these can disambiguate document-scope from paragraph-scope by inspection. The only `<a id>` anchors in the file mark *other* sub-sections (`#custom-message-context-entry-presupposition`, `#system-note-details-shapes`, `#success-side-null-policy`, `#pic-1`); the bold `**Runtime event channel.**` paragraph carries no explicit anchor at all, so the `#runtime-event-channel` fragment relied on by external cross-references resolves only to the H1 by default Markdown slug rules, even when the intended target was the paragraph definition of the always-log set.

**Verdict:** single-commit

**Introducing commits:** `f5e89f4` (2026-06-04, *docs: enforce 100KB hard / 30KB recommended size cap on spec set*).

**History:** The inline bold lead-in `**Runtime event channel.**` was introduced at `b3bc4ce` (Mon May 4 2026, *spec: pin failure-observability surface (Cluster A, Option C)*) inside the then-monolithic `spec_topics/pi-integration-contract.md`, whose H1 was `# Pi Integration Contract` — so the bold heading was unambiguous in the monolith because no document H1 shared its text. `f5e89f4` carved the section out into a new file whose H1 (`# Runtime event channel`) was set to the bold heading's lead-in verbatim, while the bold lead-in itself was retained inside the body and every "above/below" pointer that the original monolith paragraph used kept its prose form. The collision (H1 = inner heading literal) entered the corpus at that split. Later edits (`78a2f94`, *Misplaced sections within the pi-integration-contract pages*, expanded the same paragraph; `0033898`, `032dda7`, `d75f2a8`, `2e1888c`, `85daf5a` further refined the body) inherited the ambiguous referent and added new external cross-references against it (`slash-invocation.md` SLSH-3's `#runtime-event-channel` anchor, `errors-and-results/error-model.md`'s identical anchor, three `glossary.md` `See:` lines), each compounding the surface area without re-flagging the underlying same-name collision.

## Solution Space

**Shape:** single

### Recommendation

Rename the inner bold heading and give it an explicit anchor; keep the page title (`# Runtime event channel`) as is so external `runtime-event-channel.md` links and the corresponding plan-leaf citations continue to resolve.

Concrete edits inside `docs/spec_topics/pi-integration-contract/runtime-event-channel.md`:

1. Replace the bolded lead-in `**Runtime event channel.**` at line 40 with `<a id="always-log-set"></a> **Always-log set and emission routing.**` (a heading that names the paragraph's actual subject — the always-log set, occurrence/origin pair, and group-A/B partition — and gives external anchors a stable, paragraph-scoped target).
2. Rewrite the four in-file pointers that currently say "**Runtime event channel** above/below" to name the renamed section instead:
   - Line 14 ("noted under **Runtime event channel** below") → "noted under **Always-log set and emission routing** below".
   - Line 22 ("See **Runtime event channel** below for the partition rule…") → "See **Always-log set and emission routing** below for the partition rule…".
   - Line 23 ("see **Runtime event channel** below") → "see **Always-log set and emission routing** below".
   - Line 63 ("governed by the **Runtime event channel** paragraph above") → "governed by the **Always-log set and emission routing** paragraph above".

External cross-references whose intended target is the always-log paragraph (not the page as a whole) — currently `docs/spec_topics/slash-invocation.md` SLSH-3's `runtime-event-channel.md#runtime-event-channel`, `docs/spec_topics/errors-and-results/error-model.md`'s `runtime-event-channel.md#runtime-event-channel`, and the three `docs/spec_topics/glossary.md` `See:` lines on `always-log set` / `operator` / `occurrence`/`origin` — should be repointed to `runtime-event-channel.md#always-log-set` in the same edit so the rename does not leave stale fragments. References whose target is genuinely the page as a whole (e.g. `docs/spec_topics/runtime-value-model.md`'s Engine value model link to `pi-integration-contract/runtime-event-channel.md`) stay on the bare file path and need no change.

Edge cases the implementer must watch:

- The page H1 stays `# Runtime event channel`. Do not rename the H1 — the filename, plan-leaf citations (`V7d`, `V9d`, `V13c`, `V13g`), and the four in-corpus bare-file cross-references rely on the document being recognisable by that title.
- The `#runtime-event-channel` fragment that Markdown's default slug rule generates from the H1 remains valid; preserving it means any external reference outside the corpus that adopted the slug verbatim is not broken by this rename. The new `#always-log-set` anchor is *additional*, not a replacement.
- Future authors must keep the bold-heading literal distinct from the file's H1. A one-line authoring note may be added under the H1, but is not required for behavioural compliance.

## Consequence

**Severity:** advisory

Implementers and reviewers chasing a `See: Runtime event channel` link land on the H1 and must scroll the whole page to identify the always-log paragraph that the cross-reference actually meant; tooling that resolves `#runtime-event-channel` anchors lands on the document slug rather than on the paragraph that defines the always-log set, the occurrence/origin pair, and the group-A/B partition. No behavioural divergence in shipped runtime; the cost is reader friction and a fragile invariant where any future renaming of one referent without the other silently invalidates external anchors.
