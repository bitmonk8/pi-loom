## Finding

`H1a`'s **Ships when** condition â€” "A fresh checkout runs `npm install && npm run build && npm run typecheck && npm test` green" â€” is the green-install witness on which the whole scaffold rests. To satisfy it, a fresh `npm install` must resolve the four scoped `@earendil-works/pi-*` packages (`pi-coding-agent`, `pi-agent-core`, `pi-ai`, `pi-tui`) declared in both `peerDependencies` and loom's own `devDependencies`, plus `typebox`. `H1a` declares *what* is pinned and at which range, but never states *where* those packages are obtained: the registry that serves the `@earendil-works/` scope, whether it is the public npm registry or a private/scoped registry, and whether any auth token or `.npmrc` scope mapping is a build precondition.

The non-`@earendil-works` dependencies (`ajv`, `semver`, `chokidar`, `yaml`, `minimatch`, `eslint`, `vitest`, `typescript`) are ordinary public-registry packages whose resolution is uncontroversial. The scoped Pi-SDK packages are the load-bearing unknown: `H1a`'s **Adds** broadened the install reliance by additionally declaring the four in `devDependencies` so a fresh `npm install` provisions them under loom's own `node_modules` for the downstream `V18a`/`V18b`/`V18c`/`V18d` reads and the `H4a` harness â€” yet no leaf records the source/auth that fresh provisioning depends on.

The spec's [`host-prerequisites.md` Â§`#pi-sdk-pin`](../../docs/spec_topics/pi-integration-contract/host-prerequisites.md) pins the SDK *minor line* and notes the four packages "are released together from the `pi-mono` monorepo", but treats them as host-provided `peerDependencies` and likewise does not record a registry/source or auth precondition for loom's *own* fresh `npm install`. That build-environment concern is plan-side (it governs loom's checkout and CI, not the consumer-facing contract), so it is appropriately `H1a`'s to own.

## Solution Space

**Shape:** single

### Recommendation

State the source/registry precondition for the scoped Pi-SDK packages directly in `H1a`'s **Adds**, alongside the existing "two install-time reliances" passage that already scopes the `file:` plugin resolution and the `devDependencies` provisioning to the npm + flat/hoisted-`node_modules` combination. Add a third install-time reliance recording, in content terms, that the fresh `npm install` resolving the four `@earendil-works/pi-*` packages and `typebox` depends on the registry that serves the `@earendil-works/` scope being reachable at install time, and naming that source:

- If those packages are served by the public npm registry (the same channel Pi itself is distributed through), state that the `@earendil-works/` scope resolves from the default public registry with no scope-specific `.npmrc` or auth token required, so the green-install Ships-when needs only ordinary public-registry network access.
- If a private or scoped registry / auth token is required, state that source and the `.npmrc` scope-to-registry mapping (and auth) as the install-time precondition, so the green-install Ships-when is reproducible on a fresh checkout and in CI.

Tie the statement to the existing **Ships when** green-install witness so the precondition reads as a build-environment requirement of that condition rather than free-floating prose. The non-scoped public-registry dependencies need no such statement.

## Consequence

**Severity:** advisory

A contributor or CI runner on a fresh checkout cannot tell from the plan whether `npm install` resolves the `@earendil-works/` scope from the public registry with no auth, or requires a private/scoped registry and an auth token. If a scope-specific registry or token is in fact required, the `H1a` green-install Ships-when (and the downstream `V18*` SDK reads it provisions) cannot be reproduced without out-of-band knowledge; if it is the public registry, an implementer can still proceed and the gap is a documentation omission only.
