## Finding

The `always-log set` glossary entry in `docs/spec_topics/glossary.md` enumerates the dispositions across which exactly-once emission is guaranteed as three: "matched the `Err`, propagated it via `?`, or discarded it via `let _ =`." The enumeration reads as exhaustive — three named author-side dispositions joined by "or" — but the spec recognises a fourth disposition: the void-tail-expression discard form (a `@`…`` query in tail position of a void-returning function, equivalent to `let _ = @`…``). That fourth form is named on the canonical owner page (`pi-integration-contract/runtime-event-channel.md` — "Discard-site disposition" paragraph: "`let _ = @"…"`, `void`-tail-expression discard, or any other disposition that drops the `Err`") and again inside the neighbouring `query-terminating` entry in the same glossary file ("discarded via `let _ = …` or the equivalent `void`-tail-expression form"), so the omission is local to this one entry and contradicts both of its anchoring sources.

An implementer reading only the always-log set entry to determine where exactly-once emission must fire would conclude the void-tail-discard case does not need the emit, since the entry's three-form list looks closed and `loom-system-note` emission is what the entry exists to guarantee. The canonical page is correct; the glossary summary is what's narrower than the rule it summarises.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `52219c4` (2026-05-06, "Address all 41 spec-review findings")
**History:** Commit `52219c4` introduces both the `always-log set` glossary entry and the runtime-event-channel "Discard-site disposition" paragraph in a single change. The glossary entry is created with the three-form discard enumeration (`matched` / `?` / `let _ =`); the same commit's PIC paragraph already names `void`-tail-expression discard as an equivalent dropping disposition. The defect — the glossary's narrower enumeration relative to its canonical source — is therefore born at the same commit that introduces the broader rule. Subsequent edits to the `always-log set` entry (`8455d5e` 2026-05-23 renamed `invoke_callee_error` → `invoke_callee`; `038dab3` 2026-06-05 widened the set's membership to include binder-failure causes and `loom/runtime/*` panic codes) touched the entry but left the discard-form list untouched. The neighbouring `query-terminating` glossary entry, added later by `8168ea8` (2026-05-10, "T30 ceiling #4 model-driven row trichotomy carve-out"), modelled its discard clause correctly as "`let _ = …` or the equivalent `void`-tail-expression form", making the asymmetry between the two glossary entries observable from `8168ea8` onward.

## Solution Space

**Shape:** single

### Recommendation

Extend the `always-log set` entry's discard clause to mirror the wording used by both the canonical PIC page and the neighbouring `query-terminating` glossary entry. Replace "discarded it via `let _ =`" with "discarded it via `let _ =` or the equivalent `void`-tail-expression form" (matching the `query-terminating` entry's phrasing verbatim so a future grep on either phrasing finds both sites). No other change to the entry is required, and no change is needed on the canonical PIC page or the `query-terminating` entry — both already enumerate the void-tail form correctly.

## Consequence

**Severity:** advisory

A glossary summary contradicts its own canonical page on a closed enumeration. An implementer working from the glossary alone could omit `loom-system-note` emission for `Err` values dropped via void-tail discard, producing a silent operator-observability gap on the exact discard form the runtime-event-channel page took pains to cover. Readers who consult the canonical page are unaffected; readers who treat the glossary as a faithful summary diverge.
