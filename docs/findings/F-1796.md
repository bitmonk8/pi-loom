## Finding

The `loom/runtime/watcher-terminated` row in `code-registry-runtime.md` defines its trigger condition in terms of a specific implementation library: *"The **chokidar** file watcher over the discovered roots terminated — its `error` route fired or it threw such that one or more watched roots stop delivering events."* The observable condition the diagnostic actually keys off is library-agnostic — the file watcher over the discovered roots stopped delivering events for one or more roots and the adapter left it torn down rather than re-armed — yet the trigger embeds the production library's name and one of its internal routes into normative diagnostic prose.

This contradicts how the rest of the corpus frames chokidar. `host-prerequisites.md` explicitly classifies it as an implementation-side choice ("Loom 1.0's own npm-package dependencies (e.g. `semver` … `chokidar` behind the `FileWatcher` seam) are implementation-side choices rather than part of the Pi-supplied contract"), and the normative watcher contract is the `FileWatcher` / `FakeFileWatcher` DI seam in PIC-14, which exposes only three event kinds and **filters chokidar's `error` events out before they reach the runtime** ("the chokidar-adapter's own `error` events route through the `ctx.ui.notify` toast surface … never reach the runtime"). A diagnostic-trigger that fires on "its `error` route" therefore references a route the normative seam deliberately does not surface, describing the trigger against the production adapter's internals rather than the seam's observable contract.

The behavioural intent is already carried, library-neutrally, by PIC-55 (the watcher post-`error`/post-throw recovery posture), which the row references. The remedy is to state the trigger in seam-level observable terms and let the single existing chokidar pin in `host-prerequisites.md` / `implementation-notes.md` stand as the implementation note. (The sibling `loom/runtime/registry-swap-failed` row, named parenthetically in the original report, already uses the library-neutral descriptor "The watcher's …" and needs no change; the defect is confined to the `watcher-terminated` row.)

## Solution Space

**Shape:** single

### Recommendation

Reword the `loom/runtime/watcher-terminated` trigger Description in `code-registry-runtime.md` to a seam-level observable descriptor, dropping the library name and the chokidar-specific `error`-route reference while keeping every downstream clause (the torn-down-not-re-armed posture, `LoomRegistry` stays live and dispatchable, no drain-state tag written, `/reload` recovery, `loom-system-note` channel) and the existing PIC-55 reference intact. For example, open the cell with: *"The file watcher over the discovered roots stopped delivering events for one or more roots — its error route signalled an unrecoverable failure or it threw — and the adapter left it torn down rather than re-armed, so hot-reload can no longer fire for the affected roots until the operator reloads."*

Do not introduce a new chokidar pin: the implementation-side pin is already stated once in `host-prerequisites.md` and `implementation-notes.md`, and the normative watcher contract is the PIC-14 `FileWatcher` seam. Leave the `registry-swap-failed` row unchanged — it is already library-neutral.

## Consequence

**Severity:** advisory

An implementer wiring the documented `FileWatcher` seam with a non-chokidar adapter (or the in-memory `FakeFileWatcher` the seam mandates for tests) has no `error` route to key off, so the trigger as written does not map cleanly onto the only contract the runtime actually observes. The behavioural intent survives via PIC-55, so a working system is still reachable, but the over-prescriptive wording invites confusion and a test fixture coupled to chokidar internals rather than the seam.
