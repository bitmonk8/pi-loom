## Finding

Step 0 (c) reads `typeof AgentSession.prototype.abort === "function"` against the class imported from `@earendil-works/pi-coding-agent`, and the spec promises that "a missing or renamed `abort` member surfaces through the same diagnostic rather than as a runtime-time `TypeError` at the spawn site". The runtime, however, never touches the imported class again: cancellation forwards through `session.abort()` on the value `createAgentSession(...)` resolves to (see `subagent.md` — *Subagent session lifecycle*), and that handle is what `host-interfaces-core.md` pins via the `AgentSession` interface block.

The probe therefore predicts the spawn-time handle's `abort` only on an unstated behavioural premise: that `createAgentSession(...)` returns an instance whose prototype chain includes `AgentSession.prototype` (or, equivalently, a value on which `Object.getPrototypeOf(handle).abort === AgentSession.prototype.abort`). A Pi minor that re-implemented the factory as `return { abort: () => …, dispose: () => …, sendUserMessage: …, subscribe: … }` — a plain-object adapter, or a wrapper that swaps in its own implementation while still re-exporting the original class shape for typing — would pass `typeof AgentSession.prototype.abort === "function"`, satisfy the surface-inventory and editorial-review checks (including bump-checklist item (n), which audits the *declared* `AgentSession` member surface, not the runtime prototype chain), and only then trip a runtime `TypeError` (or, more insidiously, dispatch to a different `abort`) the first time `loomAbort.signal` fires. The probe-side rationale — "surfaces as `loom/load/host-incompatible` rather than as a runtime-time `TypeError`" — is silently void on that path.

The gap is narrow today but durable: nothing in the spec corpus, the editorial-review checklist, or the surface inventory closes it.

## Issue introduction

**Verdict:** single-commit

**Introducing commit:** `5c8e630` — *pi-loom spec: resolve "Subagent cancellation wiring depends on a non-existent createAgentSession({ signal }) option"* (2026-05-07)

**History:** Subagent-mode cancellation was originally wired through a (non-existent) `createAgentSession({ signal: loomAbort.signal })` option. Commit `5c8e630` replaced that wiring with the current pattern — probe `AgentSession.prototype.abort` at factory time, then invoke `session.abort()` on the spawned handle from a one-shot `loomAbort.signal` listener — and added the on-failure rationale that "a missing or renamed member surfaces as `loom/load/host-incompatible` rather than as a runtime-time `TypeError` at the spawn site". That rationale presupposes a prototype-to-handle link that the same commit did not state. Subsequent commits refined adjacent concerns — `f5e89f4` reshaped the probe under the 100 KB size cap (preserving the bullet verbatim), `aa334c2` ("AgentSession consumed member surface not pinned") pinned the *handle's* member surface in `host-interfaces-core.md` and added bump-checklist item (n) for `dispose`/`sendUserMessage`/`subscribe` (explicitly excluding `abort` on the grounds that the prototype probe already covers it), and `a33bd49` added the `details.member` discriminator — but none addressed the prototype-vs-handle linkage gap. The defect is therefore attributable to a single commit and has persisted, in identical form, across every later refactor of the surrounding text.

## Solution Space

**Shape:** single

### Recommendation

Add a one-sentence presupposition immediately under the `AgentSession` interface block in `host-interfaces-core.md` stating that the value `createAgentSession(...)` resolves to is an instance whose prototype chain includes `AgentSession.prototype` (so the factory-time `typeof AgentSession.prototype.abort === "function"` probe is observationally equivalent to a `typeof handle.abort === "function"` check at the spawn site, and the on-failure rationale in `capability-probe.md` Step 0 (c) is preserved). Cross-link the presupposition from the Step 0 (c) `AgentSession.prototype.abort` bullet (one parenthetical clause, no rule change). Extend `version-bump-step2.md` item (n) — or add a sibling item — to re-validate the linkage against each candidate Pi minor: the contributor confirms that `createAgentSession(...)`'s return value, at the candidate pin, is constructed by `new AgentSession(...)` (or equivalent) such that `Object.getPrototypeOf(handle).abort === AgentSession.prototype.abort` holds, and records `pass`/`fail`/`not-performed` per the existing recording conventions. The new clause sits on the same SHOULD-level footing as the rest of item (n) and escalates to a build-time pin under the same condition the surrounding items already declare.

Implementer note: the presupposition is observable at runtime via a single `instanceof` or `getPrototypeOf` read at the spawn site — but the spec must not require such a runtime read in the loom 1.0 hot path, because adding one would duplicate work the factory-time probe is explicitly designed to subsume. The runtime check, if any, belongs in the conformance test that pairs with `V9i`, not in the production runtime.

## Consequence

**Severity:** advisory

A Pi minor that replaces the class-instance return of `createAgentSession(...)` with a plain object or a non-`AgentSession`-prototyped wrapper would pass Step 0 (c) silently, then surface as a runtime `TypeError` (or, worse, a silently-redirected `abort`) on the first cancellation in subagent mode — the exact failure mode the probe is documented as preventing. The probability is low at the loom 1.0 Pi-SDK pin, but the spec's own rationale becomes false on that path, and the version-bump procedure has no item that would catch the regression.
