## Finding

H2a's *No globals, statics, singletons* architectural test bans a module-level binding that is, among other forms, "a top-level `const` whose initializer is a **mutable** object or array literal." The same Adds sentence then says "the AST scan decides the ban on the **observable initializer shape**, not on runtime reuse." These two qualifications point an implementer at incompatible recognisers. "Mutable ŌĆ” literal" invites runtime-mutability reasoning (is the object actually mutated / mutable at runtime?); "observable initializer shape ŌĆ” as determined by the AST" invites pure node-type reasoning (is the initializer node an `ObjectExpression` / `ArrayExpression`?). The word "mutable" survives from an earlier runtime-reuse predicate and now contradicts the node-type framing layered on top of it.

The divergence is concrete on a wrapped or frozen initializer such as `const x = Object.freeze({ ŌĆ” })`. Under the node-type reading the `const`'s initializer node is a `CallExpression` (`Object.freeze(...)`), not an object literal, so it is not flagged. Under a literal-mutability reading an implementer might either treat the frozen object as immutable (not flagged) or recurse into the call arguments to find the `{ ŌĆ” }` literal and flag it. The fourth Tests bullet repeats the same "mutable object or array literal" wording, so the fixture that proves the gate inherits the ambiguity rather than resolving it.

Two reasonable implementers would therefore build the detector ŌĆö and its proving fixture ŌĆö to classify wrapped/frozen module-level constants differently, with no text in the leaf to adjudicate which is correct.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H2a-cross-cutting-gates.md`, disambiguate the module-level-binding definition in both places it appears ŌĆö the Adds field and the fourth Tests bullet ŌĆö so a single recogniser is described. Strike the word "mutable" qualifying the literal and key the definition on the AST node type of the `const`'s direct initializer: a top-level `const` whose direct initializer node is an object literal (`{ ŌĆ” }`) or array literal (`[ ŌĆ” ]`), declared outside any class or function. Keep the existing rationale clause ("shared and mutated across calls") and the residue clause (closure-captured / lazy-cache / DI-container forms owned by the *Per-phase TDD ritual* self-review) intact.

State explicitly how a wrapped/frozen initializer is treated: a `const` whose direct initializer is a call such as `Object.freeze({ ŌĆ” })` is a `CallExpression`, not a literal node, and is therefore **not** flagged ŌĆö consistent both with the node-type rule and with the stated rationale (a frozen object cannot be shared-and-mutated). Make the fourth Tests bullet's fixture description use the same node-type wording so the proving fixture matches the recogniser.

Candidate replacement text for the defining clause (both sites): "a top-level `const` whose direct initializer node is an object literal `{ŌĆ”}` or array literal `[ŌĆ”]` (as determined by the AST node type; a call-wrapped initializer such as `Object.freeze({ŌĆ”})` is a `CallExpression`, not a literal node, and is out of scope)".

## Consequence

**Severity:** correctness
