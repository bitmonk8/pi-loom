## Finding

The `respond_repair` field bullet in `frontmatter/frontmatter-fields-b-and-templates.md` opens by stating that `respond_repair` "controls how typed queries recover from schema-validation failures (see the [Query] section)" and then immediately re-states the same thing in different words: "The block configures **respond-repair** — the response-repair mechanism specified in [Query — Schema-validation respond-repair]. The verb \"retry\" is reserved for genuinely retry-shaped behaviour elsewhere in the spec (`TransportError.retryable`, the binder's single-shot transport retry); they are different mechanisms."

This restatement carries no normative content beyond the opening sentence and the field name itself. The mechanism is named `respond-repair` consistently throughout the corpus — `glossary.md` canonicalises the term ("Hyphenated as `respond-repair` in prose; underscored as a YAML key"), and `frontmatter-fields-a.md`, `errors-and-results/queryerror-variants.md`, `hard-ceilings/ceiling-invariants-and-audit.md`, and `query/query-failure-and-repair.md` all use it. The respond-repair / retry distinction is likewise already drawn at its load-bearing sites: `binder/determinism-cancellation-failure.md` ("Respond-repair follow-ups … do not apply to a binder") and `pi-integration-contract/host-interfaces-services.md` ("a separate concept and is unaffected").

The deleted sentence is decision-log residue from the `retry:`→`coercion:` rename (Decision #11): once the `retry:` key was removed, the spec added a defensive note reserving the word "retry". With the term used consistently everywhere and the distinction drawn at its proper sites, the note adds interpretive noise to a field-definition bullet. Deleting it also removes a minor naming variant — the sentence calls the mechanism "the response-repair mechanism" while the canonical spelling is "respond-repair".

## Issue introduction

**Introducing commits:** `708a42f` (2026-05-06, *pi-loom spec: resolve "coercion overloaded: schema-validation repair vs. type conversion"*), with precursor text added by `13cf76d` (2026-05-05, *spec: rename retry: -> coercion: frontmatter key (Decision #11)*) and carried into the current path by `f5e89f4` (2026-06-04, *docs: enforce 100KB hard / 30KB recommended size cap on spec set*).
**History:** The phrase "the response-repair mechanism specified in [Query — Schema-validation coercion]" was first added by `13cf76d`, when the canonical key was `coercion` — there was no spelling clash at that point because the canonical term was a different word entirely. The clash was created by `708a42f`, which renamed the frontmatter key `coercion` → `respond_repair` and the in-prose mechanism name to **respond-repair** throughout the same bullet — but left the two earlier "response-repair mechanism" / "response-repair loop" tokens (one in the frontmatter prose, one in the glossary entry) untouched, so the same sentence now carries both spellings. The 2026-06-04 size-cap split (`f5e89f4`) moved the affected paragraph from `spec_topics/frontmatter.md` into the current `frontmatter-fields-b-and-templates.md` shard verbatim; it did not introduce or fix the defect. `git log -L` on the surviving line confirms `f5e89f4` is the only commit that touches it at the current path.

## Solution Space

**Shape:** single

### Recommendation

Delete the two sentences "The block configures **respond-repair** — the response-repair mechanism specified in [Query — Schema-validation respond-repair](../query.md). The verb \"retry\" is reserved for genuinely retry-shaped behaviour elsewhere in the spec (`TransportError.retryable`, the binder's single-shot transport retry); they are different mechanisms." from the `respond_repair` field bullet. The bullet then reads "`respond_repair` controls how typed queries recover from schema-validation failures (see the [Query](../query.md) section). `attempts` bounds the number of follow-up respond-repair turns; `methodology` selects the phrasing strategy. Recognised methodologies (loom 1.0) — …", which retains every normative element. Leave all other `respond-repair` usages across the corpus untouched.

## Consequence

**Severity:** cosmetic

A reader of the `respond_repair` field bullet encounters a redundant restatement of the opening sentence plus a "retry vs respond-repair" disambiguation already established elsewhere. No observer behaviour is affected; the cost is interpretive noise and a stray "response-repair" naming variant in an otherwise consistently-named corpus.
