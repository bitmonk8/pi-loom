## Finding

`docs/spec_topics/descriptions.md` opens with the normative claim that `///` doc comments "lower to JSON Schema `description:` fields and are passed to the model when the schema is used for structured output." The page's *No transformation* rule further pins emission to "byte-for-byte into the lowered schema." Both statements presuppose an emission step in the lowering pipeline and the presence of `description` in the loom-blessed JSON Schema subset.

Neither is supplied. `docs/spec_topics/schema-subset.md` enumerates the permitted keywords (`type`, `properties`, `required`, `additionalProperties`, `items`, `enum`, `const`, `anyOf`, `$defs`, `$ref`) and the rejected keywords (`pattern`, `format`, `minLength`, …); `description` appears in neither list. The *Lowering Algorithm* in the same file walks the per-type-form emission rules (Primitive, Object, Array, Enum, SUBS-1 primitive union, discriminated object union, mixed `anyOf`) and the four supporting steps (named-schema collection, inline-object hoist, per-query `$defs` pruning, per-schema sidecar, discriminator detection) without ever stating where `description:` is attached, what value source it draws from, or which carrier node receives it for object schemas, fields, enum declarations, enum variants, or alias schemas. The aside in `grammar.md` §`///` placement that says an alias-schema description "lowers as the description of the named type wherever it surfaces in JSON Schema output" gestures at the alias case but defers the actual mechanics to a section that does not exist.

An implementer building "exactly the loom-defined subset" reads the Schema Subset as authoritative and emits no `description:` keys at all — producing schemas that pass the loom reject gate, validate identically under AJV, but silently drop every author-written description and degrade the structured-output prompts that descriptions.md tells authors are material to output quality. Two reasonable implementers can also diverge on where to attach the description at each schema shape: at the `$defs/<Name>` entry vs. at every `$ref` site; at an enum's `{ "type": "string", "enum": [...] }` node vs. at a synthesised per-variant `const`; on the alias's named-type entry vs. on the alias's expansion; on a hoisted `__inline_<slug>` `$defs` entry vs. inline. Each choice is defensible from the current text and observationally different.

## Issue introduction

**Verdict:** introduced

**Introducing commits:**
- `fae85d3` — "Tighten spec for implementation-plan readiness" (2026-05-04)

**History:** The `description:`-lowering claim in `descriptions.md` was introduced in `53f5831` ("spec: rewrite query primitive, system prompts, and schema syntax", 2026-05-03), which also introduced the Schema Subset enumeration **without** a Lowering Algorithm and **without** `description` in the keyword set. At that moment the gap was latent: there was no algorithm to omit the step from. `fae85d3` then added the *Lowering Algorithm* section (steps 1–6: collect, hoist, emit-per-type-form, per-query pruning, wire-name sidecar, discriminator detection) but did not include a `description:` emission step and did not add `description` to the subset bullets, despite `descriptions.md` already promising the emission — the defect lands here.

Subsequent edits to the same surfaces left the gap untouched: `7321f01` pinned array element order in step 3, `e1f5c2a` added the `Result`-rejection bullet, `2128e5f` added the schema-slug collision posture, `7f217a8` rewrote step 5 into the two-map per-schema sidecar and replaced the "Markdown" rule on `descriptions.md` with the "byte-for-byte" *No transformation* rule (strengthening the emission promise without specifying the emission), and `6892aa3` added SUBS-1 for primitive unions. None added a description-emission step or a `description` subset entry. The defect has been present since `fae85d3`, with the strength of the promise (and therefore the divergence cost) increased by `7f217a8`.

## Solution Space

**Shape:** single

### Recommendation

Land a single coordinated edit covering both surfaces:

1. **Subset enumeration (`schema-subset.md` top-of-file bullets).** Add a *Descriptions* row (or extend *Validation*) listing `description` as a permitted keyword and stating that `description:` carries author-written `///` text byte-for-byte per the rule in `descriptions.md` *No transformation*. The *Explicitly not supported* list stays untouched.
2. **Lowering Algorithm (`schema-subset.md` §Lowering Algorithm, step 3 per-type-form list).** Add an emission step that attaches `description:` per schema shape, with the following normative attachment rules — each chosen so the description survives `$defs` dedup, `$ref` reuse, and per-query pruning:
   - **Named object schema (`schema X { … }`):** `description:` on the `$defs/<Name>` entry. `$ref` sites do not carry their own `description:`.
   - **Hoisted inline object schema:** `description:` on the `$defs/__inline_<slug>` entry. Authors cannot attach `///` to an anonymous inline object (no production accepts it per `grammar.md` §`///` placement), so this is for completeness; in loom 1.0 the slot is always empty.
   - **Field within an object schema:** `description:` on the field's subschema node inside the enclosing object's `properties` map. For fields whose type is a `$ref`, the description is attached to a wrapping schema with `$ref` plus `description` (JSON Schema 2020-12 permits sibling keywords alongside `$ref`).
   - **Named enum (`enum X { … }`):** `description:` on the `$defs/<Name>` entry (the `{ "type": "string", "enum": [...] }` node).
   - **Enum variant:** loom 1.0 has no JSON Schema carrier for per-variant descriptions on a string enum; preserve the variant `///` on the AST as human-facing documentation and do not lower it. Authors who need per-value documentation use named enums with descriptions on the parent declaration, or use a discriminated object union (where each variant is an object whose own `description:` does lower per the named-schema rule).
   - **Alias schema (`schema X = T | U`):** `description:` on the `$defs/<Name>` entry the alias produces; the aside in `grammar.md` §`///` placement ("wherever it surfaces") is normalised to mean "on the alias's named-type `$defs` entry," consistent with the named-object rule.
   - **Top-level `fn` declaration:** unchanged — no JSON Schema is emitted, the description stays AST-only per `descriptions.md` *Placement*.
3. **Sidecar (`schema-subset.md` §Lowering Algorithm, step 5).** No structural change; the sidecar's two maps are unaffected because `description:` is part of the lowered fragment, not a translation map.
4. **Grammar aside (`grammar.md` §`///` placement, alias sentence).** Replace "wherever it surfaces in JSON Schema output" with a cross-reference to the new §Lowering Algorithm rule, removing the under-specified hand-wave.

Edge cases the implementer must handle: (a) a field whose declared type is a `$ref` to a named schema that itself has a description — the field-site description wins for that occurrence; (b) two structurally-identical inline schemas that lower to the same `__inline_<slug>` entry are required to be byte-identical anyway (the existing slug-collision rule), so they cannot disagree on `description:`; (c) the per-query `$defs` pruning in step 4 carries descriptions transparently — descriptions live inside the `$defs` entry that survives the prune. The hash-participation question (whether `description:` contributes to the canonical schema hash) is the explicit subject of the related sibling finding and is resolved there, not here.

## Consequence

**Severity:** correctness

A subset-only implementer drops every `description:` silently, defeating the "materially improve output quality" promise descriptions.md makes to authors. Two implementers who do emit `description:` will disagree on where to attach it for fields, enum variants, alias schemas, and `__inline_<slug>` entries, producing different lowered fragments for byte-identical source — which propagates to different canonical hashes, different `$defs` dedup outcomes, and different `__inline_<slug>` / `__loom_respond_<slug>` synthesised names across implementations.
