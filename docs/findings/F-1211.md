## Problem

`descriptions.md` normatively claims `///` doc comments lower to JSON Schema `description:` fields, and its *No transformation* rule pins that emission to "byte-for-byte into the lowered schema." Both presuppose `description` in the loom-blessed subset and an emission step in the lowering pipeline. `schema-subset.md` lists `description` in neither its permitted-keyword enumeration nor its *Explicitly not supported* list, and the *Lowering Algorithm* step 3 per-type-form rules never state where `description:` attaches, what value it draws from, or which carrier node receives it for object schemas, fields, enum declarations, enum variants, or alias schemas; the `grammar.md` §`///` placement alias aside defers the mechanics to a section that does not exist. A subset-only implementer therefore emits no `description:` at all, while implementers who do emit diverge on the attachment site per schema shape — producing different lowered fragments, and so different canonical hashes and `$defs` dedup outcomes, for byte-identical source.

## Issue introduction

**Verdict:** introduced

**Introducing commits:**
- `fae85d3` — "Tighten spec for implementation-plan readiness" (2026-05-04)

**History:** The `description:`-lowering claim in `descriptions.md` was introduced in `53f5831` ("spec: rewrite query primitive, system prompts, and schema syntax", 2026-05-03), which also introduced the Schema Subset enumeration **without** a Lowering Algorithm and **without** `description` in the keyword set. At that moment the gap was latent: there was no algorithm to omit the step from. `fae85d3` then added the *Lowering Algorithm* section (steps 1–6: collect, hoist, emit-per-type-form, per-query pruning, wire-name sidecar, discriminator detection) but did not include a `description:` emission step and did not add `description` to the subset bullets, despite `descriptions.md` already promising the emission — the defect lands here.

Subsequent edits to the same surfaces left the gap untouched: `7321f01` pinned array element order in step 3, `e1f5c2a` added the `Result`-rejection bullet, `2128e5f` added the schema-slug collision posture, `7f217a8` rewrote step 5 into the two-map per-schema sidecar and replaced the "Markdown" rule on `descriptions.md` with the "byte-for-byte" *No transformation* rule (strengthening the emission promise without specifying the emission), and `6892aa3` added SUBS-1 for primitive unions. None added a description-emission step or a `description` subset entry. The defect has been present since `fae85d3`, with the strength of the promise (and therefore the divergence cost) increased by `7f217a8`.

## Solution approach

Add `description` to `schema-subset.md`'s permitted-keyword enumeration (the *Validation* row or a sibling row), drawing its value from `descriptions.md`'s *No transformation* rule. Add an emission step to the *Lowering Algorithm* step 3 per-type-form list stating, for each schema shape it already enumerates, the carrier node that receives `description:` and the value source, chosen so the description survives the step-2 `$defs` dedup, `$ref` reuse, and the step-4 per-query prune. Rewrite the `grammar.md` §`///` placement alias sentence's "wherever it surfaces in JSON Schema output" hand-wave as a cross-reference to the new step-3 rule.
