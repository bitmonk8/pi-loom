## Problem

`grammar.md`'s `///`-placement list names `FnDecl` (top-level `fn` declarations) as a production a doc comment may precede, and the Type grammar block's `ReturnType ::= Type | "void"` is annotated "function-return position only" — but no `FnDecl` production is written down anywhere in the corpus. The only treatment of `fn` declarations is the worked example in `functions.md` plus surrounding prose, leaving the surface form to inference: the parameter-list grammar (separator, trailing comma, empty list `fn f()` vs `fn f`), whether the return annotation is optional, and whether `fn` parameters may carry defaults. The default-on-parameter question is load-bearing because `loom/parse/invoke-arity-too-few` counts "non-defaulted `params:`", so whether an analogous concept exists for `fn` parameters changes both the parser and that diagnostic against `.warp`-imported `fn`s. A parser implementer building from the appendix has nowhere to attach the `ReturnType` reference.

## Issue introduction

**Verdict:** present-since-inception (of the Grammar Appendix)
**Introducing commit:** `f841b12` (2026-05-04) — *spec: resolve Cluster 2 (literal sublanguage + grammar appendix)*
**History:** `git log -S '`FnDecl`'` on `docs/spec_topics/grammar.md` returns a single commit, `f841b12`, which added `spec_topics/grammar.md` from scratch. The `///`-placement bullet referencing `FnDecl` was authored in that initial commit (`@@ -0,0 +131,1 @@`); no production for `FnDecl` was added alongside it. Subsequent edits to the appendix (`232a020`, `1c552ff`, `4c76dfb`, `816852b`, `057159c`, `c3b7c87`, `e1f5c2a`, `4e656ac`) extended the Type, Block-expression, and DocComment productions but never closed the `FnDecl` gap. On the `functions.md` side, the `fn rate_strictness(…)` example dates to `53f5831` (the schema-syntax rewrite), well before the appendix existed and never elevated to a production. The defect is the original shape of the appendix: the bullet list was written as if `FnDecl` were defined elsewhere on the same page, but it never was.

## Solution approach

Add a normative `FnDecl` production to `grammar.md` positioned so `ReturnType` and `FnBody` are in scope, resolving the parameter-list form, the optionality of the `: ReturnType` annotation, and whether `fn` parameters admit defaults or a `mut` modifier — consistent with loom 1.0 carrying no defaulted `fn` parameters and the existing `loom/parse/mut-on-immutable-context`. Tie the annotation-less case to `functions.md#empty-tail-body`, closing the dangling "annotation-less `fn` body" reference. Add a forward-link from `functions.md` §Placement to the new production and drop the bare-example reliance there. If the resolution rejects a surface form (e.g. a defaulted `fn` parameter) with a new parse diagnostic, name and register it in `code-registry-parse.md` in the same change.
