## Finding

The Grammar Appendix names `FnDecl` as one of the productions a `///` doc comment may legally precede (`grammar.md` §`///` placement, bullet 5), but no `FnDecl` production is ever written down. The neighbouring entries in the same list (`SchemaDecl`, `EnumDecl`) are defined in `schemas.md` and have BNF on the page; `FnDecl` is referenced as if it were similarly defined and is not. The only treatment of `fn` declarations anywhere in the corpus is the worked example `fn rate_strictness(p: Author): Result<ReviewScore, QueryError> { … }` in `functions.md`, plus surrounding prose about placement, naming, doc comments, and the empty-tail rule. There is no normative production at any site.

As a result, the surface form of a `fn` declaration is left to inference from one example. Specifically unresolved: the parameter-list grammar (separator, trailing comma, empty parameter list `fn f()` vs. `fn f`), whether the return annotation is optional (an annotation-less form is alluded to in `functions.md` — *"An annotation-less `fn` body infers its return type by this same rule"* — but no grammar admits it), and whether `fn` parameters may carry defaults. The default-on-parameter question is load-bearing because `params:` defaults already have a formal literal-RHS grammar (the Loom literal sublanguage on the same page) and a defaulted-vs-non-defaulted partition that the invocation grammar reads (`invoke-arity-too-few` counts "non-defaulted `params:`"); whether the analogous concept exists for `fn` parameters changes both the parser and `loom/parse/invoke-arity-too-few` against `.warp`-imported `fn`s.

The Type grammar block separately admits `ReturnType ::= Type | "void"` *"in function-/loom-return position only"*, which presupposes a `FnDecl` production that names the position. A parser implementer building from the appendix has nowhere to attach that `ReturnType` reference.

## Issue introduction

**Verdict:** present-since-inception (of the Grammar Appendix)
**Introducing commit:** `f841b12` (2026-05-04) — *spec: resolve Cluster 2 (literal sublanguage + grammar appendix)*
**History:** `git log -S '`FnDecl`'` on `docs/spec_topics/grammar.md` returns a single commit, `f841b12`, which added `spec_topics/grammar.md` from scratch. The `///`-placement bullet referencing `FnDecl` was authored in that initial commit (`@@ -0,0 +131,1 @@`); no production for `FnDecl` was added alongside it. Subsequent edits to the appendix (`232a020`, `1c552ff`, `4c76dfb`, `816852b`, `057159c`, `c3b7c87`, `e1f5c2a`, `4e656ac`) extended the Type, Block-expression, and DocComment productions but never closed the `FnDecl` gap. On the `functions.md` side, the `fn rate_strictness(…)` example dates to `53f5831` (the schema-syntax rewrite), well before the appendix existed and never elevated to a production. The defect is the original shape of the appendix: the bullet list was written as if `FnDecl` were defined elsewhere on the same page, but it never was.

## Solution Space

**Shape:** single

### Recommendation

Add a normative `FnDecl` production to `grammar.md` in a new short subsection (placement: after the Type grammar block and before the Block-expression block, so `ReturnType` is in scope and `FnBody` is in scope by forward reference) of the following shape:

```
FnDecl       ::= "fn" Ident "(" ParamList? ")" (":" ReturnType)? FnBody
ParamList    ::= Param ("," Param)* ","?
Param        ::= Ident ":" Type
```

with the following normative clauses anchored on the production:

- The parameter list is comma-separated with an optional trailing comma; an empty parameter list is written `fn f()` (the parentheses are required — `fn f { … }` does not parse).
- The return annotation `: ReturnType` is optional. When absent, the body's return type is inferred per [Function Definitions](./functions.md#empty-tail-body) (this closes the dangling "annotation-less" reference).
- `Param` admits no `= literal` default. A `fn` parameter with a default RHS is `loom/parse/default-not-on-param-allowed` (or, preferring an existing surface, fold this into a parse error reusing `loom/parse/default-not-literal`'s register slot with a distinct message — name the code at the same time so `V3d` can test it). loom 1.0 carries no notion of defaulted `fn` parameters; the `invoke-arity-too-few` "non-defaulted `params:`" rule applies to `params:` only and has no `fn`-parameter analogue.
- `Param` admits no `mut` modifier (it is `loom/parse/mut-on-immutable-context`, already defined; the production needs to reflect that by naming `Ident` rather than `Pattern` on the LHS).
- The `FnBody` reference resolves to the existing relaxed-tail production on the same page.

Cross-link from `functions.md` §Placement to the new production once added, and remove the bare-example reliance there. The `///`-placement bullet then resolves against a real referent. Edge cases for the implementer: the production above does not change the `nested-fn` parse error (that is a placement constraint on where `FnDecl` may appear, not a productions-grammar issue, and remains owned by `functions.md`); and the diagnostic code chosen for a defaulted `fn` parameter must be added to `code-registry-parse.md` in the same change so `V3d`/`V3d-T` can fixture it.

## Consequence

**Severity:** correctness

Two implementers given the current text will diverge on which `fn` surface forms parse: at minimum on the zero-parameter form (`fn f()` vs `fn f`), on whether `fn f(): T { … }` admits an annotation-less variant, and on whether `fn f(x: int = 0)` is accepted, rejected as a parse error, or silently parsed and ignored. The latter is observable through `loom/parse/invoke-arity-too-few` arithmetic against an imported `.warp` callee and through the slash-command binder's view of which params have defaults. Conformance tests cannot be written against forms the spec does not admit or reject.
