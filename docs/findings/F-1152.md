## Finding

`diagnostic-shape.md` mandates that every diagnostic outside an enumerated closed set carry **both** `file` and `range` (the inline shape comment for `range?` reads "omitted together with file", so the two fields can only be dropped as a pair). The closed set lists eight codes — `loom/load/host-incompatible`, `loom/load/extension-bootstrap-failed`, `loom/load/discovery-slow`, `loom/load/package-read-timeout`, and the four `loom/host/session-shutdown-*` teardown codes — explicitly described as having "no source location at emission time".

A scan of `code-registry-load.md` finds many more `loom/load/*` rows whose registered *Trigger* prose describes an emission site with **no** single token span, and several with **no** file at all. Concrete examples:

- **No file at all (descriptor-keyed):** `loom/load/missing-source`, `loom/load/unreadable-source`, `loom/load/wrong-type-source` carry a discovery *descriptor* (e.g. `"settings entry index 2"`, `"--loom flag #1"`, `"global looms directory"`); the offending path may not exist or may not be a file. `loom/load/manifest-invalid` and `loom/load/manifest-escapes-package` key off a `package.json`, not a `.loom`. `loom/load/settings-unreadable`, `loom/load/settings-invalid-json`, `loom/load/settings-invalid-entry`, and `loom/load/settings-value-out-of-range` key off a settings file.
- **Multiple files (no single `file`):** `loom/load/case-collision`, `loom/load/cross-source-shadow`, `loom/load/cross-format-collision`, `loom/load/invocation-cycle`, `loom/load/import-cycle` all describe collisions or cycles whose `message` template enumerates two-or-more paths.
- **A single file but no token span:** `loom/load/unreadable` (the `.loom` could not be read, so no tokens exist), `loom/load/invalid-encoding` (byte-offset only, no line/column until after a successful decode), `loom/load/invalid-extension`, `loom/load/non-canonical-extension`, `loom/load/invalid-slash-name` (judged on the filename, not on file contents).

Under the current spec each of these is forced to either fabricate a sentinel `file` and zero-valued `range` (the wire form the location-less paragraph explicitly forbids for its eight codes, and the **Serialised content format** rule has no rendering carve-out for elsewhere) or omit the fields and fail strict envelope validation. Two reasonable implementers will diverge on every code in the lists above, and on every such row authors will see either a `file:0:0:` lie or an envelope validation failure.

The "omitted together with file" wire-shape pairing is itself part of the gap: even for codes that *do* know a file (e.g. `case-collision`, `unreadable`), a strict reading forbids carrying `file` without a `range`, so a code with one file but no span has no legal shape.

## Issue introduction

**Verdict:** present-since-inception (of the location-less mechanism)
**Introducing commit:** `b7d383a` (2026-06-06) — "pi-loom spec: resolve \"Diagnostic envelope unsatisfiable for location-less codes\""
**History:** The `Diagnostic` shape originally pinned `file: string` and `range: { start, end }` as required fields for every code, an unsatisfiable contract for any code with no source location. `b7d383a` resolved that earlier defect by marking `file?` / `range?` optional, introducing the **Location-less diagnostics** paragraph, and enumerating eight codes that drop both fields. The introducing commit's enumeration was scoped narrowly to the eight codes the earlier finding (then-`T100`) named — the four bootstrap/discovery-cap `loom/load/*` codes plus the four `loom/host/session-shutdown-*` teardown codes — and did not walk the rest of `code-registry-load.md` to classify the remaining `loom/load/*` rows whose registered triggers describe descriptor-keyed, multi-file, or whole-file-unreadable emission sites. The "omitted together with file" pairing on the `range?` line was added in the same commit and inherits the same scoping gap: no per-code rule was added for codes that know a file but not a token span. No later commit revisited the enumeration; `8d730c5` and `a37d4ef` touched neighbouring paragraphs (`GOV-15` carve-out, column-legend relocation) without re-examining the location-less set. The defect is therefore present since the location-less mechanism's introduction and has been latent against the unedited `code-registry-load.md` rows ever since.

## Solution Space

**Shape:** multiple

### Option A — Extend the closed location-less set

**Approach.** Walk `code-registry-load.md` row-by-row, classify each emission site against the {has-file, has-span} matrix, and add every code with no-file (or no-single-file) to the **Location-less diagnostics** enumeration. For codes with a file but no token span, either (a) admit them to the location-less set too (drop both `file` and `range`, accepting the small loss of the known file in the rendered line) or (b) introduce a parallel "file-without-range" form.

**Spec edits.** Replace the eight-code list in **Location-less diagnostics** with the expanded set (eight existing + the descriptor-keyed and multi-file rows enumerated in the Finding). In the **Serialised content format** paragraph, keep the existing "drop the leading `<file>:<line>:<col>:`" rule for the expanded location-less set. If sub-option (b) is taken, add a third content-line form (`"<file>: <code>: <message>"`) for the file-without-range codes and document its rendering. Relax the `range?` comment from "omitted together with file" to "omitted independently of `file`" (so a code may carry `file` alone), and update the multi-file rows (`case-collision`, cycles, shadows, collisions) to specify whether they render via the location-less form or via `related?` entries plus a missing `file`/`range` on the head.

**Pros.** Mechanically simple. Preserves the existing rendering machinery. The set stays closed (DIAG-2-compatible), so the loom-1.x carve-out remains a single-paragraph spec edit per future addition. Implementers have one rule per code: look it up, render in the named form.

**Cons.** The "expanded set" still has to be exhaustive against a registry that future spec work will add to; every new `loom/load/*` code must remember to classify itself. The multi-file codes (`case-collision`, cycles, collisions) lose the discoverability of the head file in the rendered single-line summary unless `related?` is normatively required to populate every colliding site. If sub-option (b) is taken, three content-line forms instead of two adds a rendering branch.

**Risks.** Drift between the set and the registry. Multi-file codes risk dropping useful information from the head line.

### Option B — Per-code `file` / `range` source rules in each registry row

**Approach.** Replace the single closed location-less set with a per-code rule: every registry row gains a *Location* column (or a one-line trigger-prose addition) naming where `file` and `range` come from (or that they are absent). The location-less codes become the rows whose *Location* reads "n/a"; multi-file codes name a head file source (e.g. "the first colliding path, sorted") plus a `related?` obligation for the others; file-without-range codes (`unreadable`, `invalid-extension`, etc.) name the file source and explicitly state `range` is absent.

**Spec edits.** Delete the **Location-less diagnostics** enumeration. Add a *Location* column to the four registry tables, populated for every existing row. Rewrite the `range?` shape comment to "omitted per the row's *Location* rule". Rewrite the **Serialised content format** paragraph to switch on the row's *Location* rule rather than on the closed set.

**Pros.** Future-proof: new codes are classified at the registry-row site (the same site that already pins *Trigger*, *Phase*, *Sev*, *Message*, *Hint*), so the drift risk in Option A vanishes. Multi-file and file-without-range codes get a precise per-row rendering rule rather than being shoehorned into a one-size form. Implementer reads the row and knows everything.

**Cons.** Larger spec diff — adds a column to four sibling registry tables and rewrites the central rendering paragraph. The wire-shape consumer can no longer validate strictly against a single per-code list without consulting the registry. DIAG-3's stability claim now covers a per-row *Location* string too.

**Risks.** Authoring discipline: every future row must remember to populate *Location*. Existing reviewers used to "is it in the location-less set?" must learn the new column.

### Recommendation

Take **Option A** with sub-option (a) — extend the closed set to admit every code with no single token span, render them all via the existing prefix-dropping rule. The rendered single-line summary loses the head file for multi-file codes (cycles, collisions), but `related?` already exists to carry every offending site with its own `{ file, range, message }`, so no information is lost — only relocated from the head line to the related-site lines. Pair the extension with the smaller mechanical fix on the shape comment: relax `range?`'s "omitted together with file" to "omitted independently" and add a third content-line form `"<file>: <code>: <message>"` for the file-only codes (`unreadable`, `invalid-encoding` once a byte offset alone is carried, `invalid-extension`, `non-canonical-extension`, `invalid-slash-name`). Order of resolution: classify each `loom/load/*` row first (purely editorial against the existing trigger prose), then land the shape-comment and content-line edits in one diff.

Implementer edge cases to watch:
- `loom/load/invalid-encoding` already says `<offset>` is a byte offset; decide explicitly whether this code carries `file` only or also a `range` synthesised from the byte offset (the current row prose leaves it ambiguous).
- Multi-file codes' `related?` arrays must enumerate **every** colliding site (the current rows describe two-path messages but do not pin `related?` cardinality).
- The DIAG-4 *Message* templates for the affected rows must remain renderable with the chosen form (some templates currently embed `<path>` / `<paths>` placeholders that survive without a `file` field on the head).
- The future-addition obligation closing the **Location-less diagnostics** paragraph must continue to apply: any new `loom/load/*` row added under the `GOV-15` carve-out must classify itself against the expanded set in the same edit.

## Consequence

**Severity:** correctness

Roughly two dozen registered `loom/load/*` codes have no legal wire form under the current shape contract. Implementers will silently diverge — some fabricating `file:"<unknown>":0:0` strings, some shipping envelopes that fail strict consumer validation, some inventing a third per-team convention — and tests entitled by `DIAG-4` to assert on the rendered message cannot derive an expected string for any affected row. The breakage is invisible until a strict consumer (LSP integration, log pipeline, test harness) hits production data.
