## Finding

`diagnostic-shape.md` mandates that every diagnostic outside an enumerated closed set carry **both** `file` and `range` (the inline shape comment for `range?` reads "omitted together with file", so the two fields can only be dropped as a pair). The closed set lists eight codes — `loom/load/host-incompatible`, `loom/load/extension-bootstrap-failed`, `loom/load/discovery-slow`, `loom/load/package-read-timeout`, and the four `loom/host/session-shutdown-*` teardown codes — explicitly described as having "no source location at emission time".

A scan of `code-registry-load.md` finds many more `loom/load/*` rows whose registered *Trigger* prose describes an emission site with **no** single token span, and several with **no** file at all. Concrete examples:

- **No file at all (descriptor-keyed):** `loom/load/missing-source`, `loom/load/unreadable-source`, `loom/load/wrong-type-source` carry a discovery *descriptor* (e.g. `"settings entry index 2"`, `"--loom flag #1"`, `"global looms directory"`); the offending path may not exist or may not be a file. `loom/load/manifest-invalid` and `loom/load/manifest-escapes-package` key off a `package.json`, not a `.loom`. `loom/load/settings-unreadable`, `loom/load/settings-invalid-json`, `loom/load/settings-invalid-entry`, and `loom/load/settings-value-out-of-range` key off a settings file.
- **Multiple files (no single `file`):** `loom/load/case-collision`, `loom/load/cross-source-shadow`, `loom/load/cross-format-collision`, `loom/load/invocation-cycle`, `loom/load/import-cycle` all describe collisions or cycles whose `message` template enumerates two-or-more paths.
- **A single file but no token span:** `loom/load/unreadable` (the `.loom` could not be read, so no tokens exist), `loom/load/invalid-encoding` (byte-offset only, no line/column until after a successful decode), `loom/load/invalid-extension`, `loom/load/non-canonical-extension`, `loom/load/invalid-slash-name` (judged on the filename, not on file contents).

Under the current spec each of these is forced to either fabricate a sentinel `file` and zero-valued `range` (the wire form the location-less paragraph explicitly forbids for its eight codes, and the **Serialised content format** rule has no rendering carve-out for elsewhere) or omit the fields and fail strict envelope validation. Two reasonable implementers will diverge on every code in the lists above, and on every such row authors will see either a `file:0:0:` lie or an envelope validation failure.

The "omitted together with file" wire-shape pairing is itself part of the gap: even for codes that *do* know a file (e.g. `case-collision`, `unreadable`), a strict reading forbids carrying `file` without a `range`, so a code with one file but no span has no legal shape.

## Issue introduction

**Verdict:** present-since-inception (of the location-less mechanism)
**Introducing commit:** `b7d383a` (2026-06-06) — "pi-loom spec: resolve \"Diagnostic envelope unsatisfiable for location-less codes\""
**History:** The `Diagnostic` shape originally pinned `file: string` and `range: { start, end }` as required fields for every code, an unsatisfiable contract for any code with no source location. `b7d383a` resolved that earlier defect by marking `file?` / `range?` optional, introducing the **Location-less diagnostics** paragraph, and enumerating eight codes that drop both fields. The introducing commit's enumeration was scoped narrowly to the eight codes the earlier finding (then-`T100`) named — the four bootstrap/discovery-cap `loom/load/*` codes plus the four `loom/host/session-shutdown-*` teardown codes — and did not walk the rest of `code-registry-load.md` to classify the remaining `loom/load/*` rows whose registered triggers describe descriptor-keyed, multi-file, or whole-file-unreadable emission sites. The "omitted together with file" pairing on the `range?` line was added in the same commit and inherits the same scoping gap: no per-code rule was added for codes that know a file but not a token span. No later commit revisited the enumeration; `8d730c5` and `a37d4ef` touched neighbouring paragraphs (`GOV-15` carve-out, column-legend relocation) without re-examining the location-less set. The defect is therefore present since the location-less mechanism's introduction and has been latent against the unedited `code-registry-load.md` rows ever since.

## Solution Space

**Shape:** single

### Recommendation

In `diagnostic-shape.md`, extend the closed **Location-less diagnostics** set to admit every `loom/load/*` code whose registered emission site has no single token span, and render the whole expanded set via the existing prefix-dropping rule. Walk `code-registry-load.md` row-by-row, classify each emission site against the {has-file, has-span} matrix, and replace the eight-code enumeration with the expanded set: the eight existing codes plus the descriptor-keyed rows (`loom/load/missing-source`, `loom/load/unreadable-source`, `loom/load/wrong-type-source`, `loom/load/manifest-invalid`, `loom/load/manifest-escapes-package`, `loom/load/settings-unreadable`, `loom/load/settings-invalid-json`, `loom/load/settings-invalid-entry`, `loom/load/settings-value-out-of-range`), the multi-file rows (`loom/load/case-collision`, `loom/load/cross-source-shadow`, `loom/load/cross-format-collision`, `loom/load/invocation-cycle`, `loom/load/import-cycle`), and the single-file-no-span rows (`loom/load/unreadable`, `loom/load/invalid-encoding`, `loom/load/invalid-extension`, `loom/load/non-canonical-extension`, `loom/load/invalid-slash-name`).

For the multi-file codes (cycles, collisions, shadows), drop `file` and `range` on the head and require `related?` to enumerate **every** colliding site, each with its own `{ file, range, message }`. The single-line summary loses the head file, but the offending sites are carried in full on the related-site lines, so no information is lost — only relocated.

Relax the `range?` shape comment from "omitted together with file" to "omitted independently of `file`", so a code may carry `file` alone. Add a third content-line form `"<file>: <code>: <message>"` to the **Serialised content format** paragraph for the file-only codes (`loom/load/unreadable`, `loom/load/invalid-encoding`, `loom/load/invalid-extension`, `loom/load/non-canonical-extension`, `loom/load/invalid-slash-name`) and document its rendering alongside the existing prefix-dropping form. Keep the existing "drop the leading `<file>:<line>:<col>:`" rule for the expanded location-less set.

Land the row classification first (purely editorial against the existing trigger prose), then the shape-comment and content-line edits in one diff.

Edge cases to handle:
- `loom/load/invalid-encoding` already says `<offset>` is a byte offset; state explicitly whether this code carries `file` only or also a `range` synthesised from the byte offset (the current row prose is ambiguous).
- The multi-file codes' `related?` arrays must enumerate every colliding site — pin `related?` cardinality on those rows, which currently describe only two-path messages.
- The DIAG-4 *Message* templates for the affected rows must remain renderable with the chosen form; some templates embed `<path>` / `<paths>` placeholders that survive without a `file` field on the head.
- The future-addition obligation closing the **Location-less diagnostics** paragraph must continue to apply: any new `loom/load/*` row added under the `GOV-15` carve-out classifies itself against the expanded set in the same edit.

## Consequence

**Severity:** correctness

Roughly two dozen registered `loom/load/*` codes have no legal wire form under the current shape contract. Implementers will silently diverge — some fabricating `file:"<unknown>":0:0` strings, some shipping envelopes that fail strict consumer validation, some inventing a third per-team convention — and tests entitled by `DIAG-4` to assert on the rendered message cannot derive an expected string for any affected row. The breakage is invisible until a strict consumer (LSP integration, log pipeline, test harness) hits production data.
