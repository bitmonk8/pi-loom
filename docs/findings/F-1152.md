## Problem

`diagnostic-shape.md`'s **Location-less diagnostics** closed set enumerates eight codes that drop both `file` and `range`, and the `range?` shape comment ("omitted together with file") forces the two fields to be dropped only as a pair. Many other `loom/load/*` rows registered in `code-registry-load.md` describe emission sites with no single token span: some have no file at all (descriptor-keyed, settings-file, and `package.json`-keyed rows), some span multiple files (collisions and cycles), and some have one file but no span (whole-file-unreadable, filename-judged, and encoding rows). None of these has a legal wire form — each must either fabricate a sentinel `file` / zero-valued `range` (forbidden) or omit the fields and fail strict envelope validation — and the **Serialised content format** paragraph has no rendering form for a code that carries `file` without `range`.

## Issue introduction

**Verdict:** present-since-inception (of the location-less mechanism)
**Introducing commit:** `b7d383a` (2026-06-06) — "pi-loom spec: resolve \"Diagnostic envelope unsatisfiable for location-less codes\""
**History:** The `Diagnostic` shape originally pinned `file: string` and `range: { start, end }` as required fields for every code, an unsatisfiable contract for any code with no source location. `b7d383a` resolved that earlier defect by marking `file?` / `range?` optional, introducing the **Location-less diagnostics** paragraph, and enumerating eight codes that drop both fields. The introducing commit's enumeration was scoped narrowly to the eight codes the earlier finding (then-`T100`) named — the four bootstrap/discovery-cap `loom/load/*` codes plus the four `loom/host/session-shutdown-*` teardown codes — and did not walk the rest of `code-registry-load.md` to classify the remaining `loom/load/*` rows whose registered triggers describe descriptor-keyed, multi-file, or whole-file-unreadable emission sites. The "omitted together with file" pairing on the `range?` line was added in the same commit and inherits the same scoping gap: no per-code rule was added for codes that know a file but not a token span. No later commit revisited the enumeration; `8d730c5` and `a37d4ef` touched neighbouring paragraphs (`GOV-15` carve-out, column-legend relocation) without re-examining the location-less set. The defect is therefore present since the location-less mechanism's introduction and has been latent against the unedited `code-registry-load.md` rows ever since.

## Solution approach

Extend the **Location-less diagnostics** closed set in `diagnostic-shape.md` to admit every `loom/load/*` row whose registered emission site has no single token span, classifying each `code-registry-load.md` row against a {has-file, has-span} matrix. Relax the `range?` shape comment so a code may carry `file` without `range`. Add a rendering form to the **Serialised content format** paragraph for the file-only codes, and carry the colliding sites of multi-file codes in `related?` rather than on the head.

## Solution constraints

- Preserve the **Location-less diagnostics** paragraph's closing future-addition obligation so any new `loom/load/*` row classifies itself against the expanded set in the same edit (the GOV-15 diagnostic-registry carve-out requirement).
