## Finding

The `loom/load/host-incompatible` row in `docs/spec_topics/diagnostics/code-registry-load.md` summarises the probe's short-circuit discipline as `(a)` Node floor â†’ `(b)` `AbortSignal` shape â†’ `(c)+(d)` capability list and peer-dep version â†’ `(e)` `typebox`, and asserts that a simultaneous-failure host "emits the first failing check's `kind`". Collapsing `(c)` and `(d)` into one tier and then naming the verdict by *"first failing check"* leaves an implementer reading only this row unable to decide which of the seven `kind` values is emitted when a host fails both an `sdk-capability-missing` check and a `peer-dep-*` check â€” the row offers no rule for resolving the tie inside the bundled tier, even though `kind` is a normative wire field that conformance tests read.

The source-of-truth paragraph in `docs/spec_topics/pi-integration-contract/capability-probe.md` already pins the within-tier order: step `(d)` "runs after the factory-probable capability checks complete (i.e. it is the last of the four pinned-constants checks)". So the determinism exists in the spec; the registry row simply hides it behind the `(c)+(d)` shorthand. The defect is local to the row's summary sentence â€” the per-`kind` test contract elsewhere relies on a deterministic `kind`, and the row should not introduce ambiguity that the SoT does not.

## Solution Space

**Shape:** single

### Recommendation

Replace the registry row's collapsed short-circuit sentence with an explicit five-step order that mirrors the capability-probe SoT, and drop the *"first failing check"* phrasing in favour of *"the first failing check in the order above"*. Concretely, change

> The fixed short-circuit ordering is `(a)` Node floor, `(b)` `AbortSignal` shape, `(c)+(d)` capability list and peer-dep version, `(e)` `typebox` host-shape; a simultaneous-failure host emits the first failing check's `kind`.

to

> The fixed short-circuit ordering is `(a)` Node floor â†’ `(b)` `AbortSignal` shape â†’ `(c)` capability list â†’ `(d)` peer-dep version â†’ `(e)` `typebox` host-shape (see [Capability probe â€” Step 0](../pi-integration-contract/capability-probe.md#entry-capability-probe) for the within-step iteration rules); a simultaneous-failure host emits the `kind` of the first failing step in this order.

Notes for the editor:

- The capability-probe SoT must remain the authority on within-step iteration (member order inside `(c)`; package order inside `(d)`); the row enumerates step order only, not member/package order, so the dual-source surface stays minimal.
- Preserve the row's existing closing clause about routing through the **System notes** fallback chain.
- No change to the `kind` enumeration or the per-`kind` `details` schema â€” only the summary sentence.
- The closing paragraph in `capability-probe.md` ("`(c)+(d)` capability list and peer-dep version") uses the same shorthand and should be aligned in the same edit so the registry row and the SoT page agree byte-for-byte on step labels; the SoT body already pins the within-tier order, so this is a wording sync, not a normative change.

## Consequence

**Severity:** correctness

An implementer (or conformance fixture author) who consumes only the registry row when constructing the expected `details.kind` for a host that fails both an `sdk-capability-missing` check and a `peer-dep-out-of-range` / `peer-dep-malformed-version` check has no rule to pick one. Two reasonable implementers diverge on the emitted wire value; only readers who cross to the capability-probe SoT recover determinism. The blast radius is one diagnostic, but `kind` is the normative discriminator the registry contract is built around.
