## Problem

The `loom/load/host-incompatible` row in `docs/spec_topics/diagnostics/code-registry-load.md` summarises the probe's short-circuit discipline with `(c)+(d)` collapsed into one tier and names the verdict as "the first failing check's `kind`". An implementer reading only this row cannot determine which of the seven `kind` values is emitted when a host fails both an `sdk-capability-missing` check and a `peer-dep-*` check — the row offers no rule for resolving the tie inside the bundled tier, even though `kind` is a normative wire field that conformance tests read. The capability-probe SoT already pins the within-tier order (step `(d)` runs after the capability checks, "the last of the four pinned-constants checks"), so the determinism exists in the spec; the row merely hides it behind the shorthand.

## Issue introduction

**Verdict:** present-since-inception (for the registry row's `(c)+(d)` summary)
**Introducing commits:** `126ff8a` (2026-05-06, "pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B")
**History:** The collapsed `"(c)+(d)` capability list and peer-dep version; a simultaneous-failure host emits the first failing check's `kind`"* sentence was added to the `loom/load/host-incompatible` registry row in `126ff8a`, the same commit that introduced the `(d)` peer-dep sub-step and the explicit short-circuit ordering. The sentence has been carried through every subsequent edit of the row (`f072903` adding `(e)`, `9371309`, `3e6ce2e`, `81bf52a` corpus reshuffle, `d8d5ad0`, `62d49d6`) without ever re-stating the within-tier order. The within-tier order has lived in the capability-probe SoT since the same `126ff8a` rewrite ("runs after the factory-probable capability checks complete"), so the defect is solely the registry row's summary, not the SoT.

## Solution approach

Rewrite the `loom/load/host-incompatible` row's short-circuit ordering sentence to enumerate `(c)` and `(d)` as distinct ordered steps rather than the collapsed `(c)+(d)` tier, and recast the verdict clause so the emitted `kind` is named by the first failing step in that explicit order. Add a forward-link to the capability-probe SoT at `capability-probe.md#entry-capability-probe` for the within-step iteration rules. Align the matching `(c)+(d)` shorthand in that SoT page's closing short-circuit paragraph so the row and the SoT agree on step labels.

## Solution constraints

- The capability-probe SoT (`capability-probe.md#entry-capability-probe`) remains the authority on within-step iteration order — member order inside `(c)`, package order inside `(d)`; the row enumerates step order only, not member/package order.
- Out of scope: the `kind` enumeration and the per-`kind` `details` schema — only the summary sentence (and its SoT-page counterpart) changes.
