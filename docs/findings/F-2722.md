## Problem

The trigger-(2) paragraph in `real-host-smoke-gate.md` concludes that a double-vs-real divergence on the four fidelity-contract axes introduced by a loom change "is caught before it reaches `main`," asserting a pre-merge gate. Two sentences later the same paragraph disclaims exactly that: the step is a "contributor obligation, not a CI-enforced gate," loom 1.0 "has no mechanical hook that observes whether the smoke ran," trigger-(2) "has no automatic diff detector," and a skipped or unrecognised run is detectable "only after the fact" via the `H6a` commit-message audit. Both claims cannot hold. A reader taking the over-claim at face value misjudges the residual risk loom 1.0 ships with and may assume a mechanical pre-merge detector exists to build against — masking that the `ERR-8` live-surface confirmation (criterion (c)) has no automated assertion anywhere.

## Solution approach

Rewrite the trailing trigger-(2) clause in `real-host-smoke-gate.md` — the "so a divergence introduced by a loom change rather than a Pi bump is caught before it reaches `main`" clause — so it no longer asserts a pre-merge guarantee. State instead that the obligation is to run the smoke before merge while an unrecognised fidelity-axis divergence can still reach `main`, detectable only after the fact via the `H6a` commit-message audit, consistent with the "contributor obligation, not a CI-enforced gate" sentence already in the same paragraph. Note inline that the `ERR-8` live-surface confirmation (criterion (c)) consequently has no automated `npm test` assertion by design.

## Solution constraints

- Out of scope: the gate's pass/fail criteria (a)–(e), the merge-blocker, and the `V18d` revert path — leave them unchanged.
- The after-the-fact detection limits of the `H6a` commit-message audit are owned by the `H6a` same-cluster finding; introduce no new claim that the `H6a` audit can catch a dishonestly-recorded smoke result.
- The spec host-prerequisites pages the note cites (`host-prerequisites.md`) are read-only.
