## Finding

The normative paragraph pinning the strict-capability check states that the runtime probes `Model<Api>` for `strictCapable` "duck-typed: `(model as { strictCapable?: boolean }).strictCapable`", and reads that expression as a three-valued check. The observable contract is the field name (`strictCapable`), the three-valued read (`true` / `false` / absent), and the diagnostic outcome each value maps to. The cast expression itself is a TypeScript access technique — it pins how the runtime's source code is written, not what behaviour an implementer must produce.

The spec is open elsewhere about being implemented in TypeScript (the runtime is a Pi extension; `implementation-notes.md` says so), but the convention is to keep the behavioural contract language-agnostic and confine implementation technique to implementation notes. Embedding the cast in the normative probe paragraph mixes the two layers: a Pi-SDK minor that exports `strictCapable` as a declared (no longer `unknown`-typed) member would obsolete the cast without changing the behavioural contract, yet a literal reading of the current prose would require either re-editing the paragraph or carrying a redundant cast indefinitely.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `52219c4` (2026-05-06, "Address all 41 spec-review findings")
**History:** The strict-capability probe paragraph was rewritten in `52219c4` as the resolution of an earlier-cycle finding that asked for a duck-typed three-valued read. That finding's chosen "Option A" recommendation explicitly proposed `(model as { strictCapable?: boolean }).strictCapable` "so the type concession is localised and audit-greppable" — the cast was a deliberate editorial choice, not an accident. The wording has been preserved through every subsequent edit to this paragraph (`79fa3b9`, `f5e89f4` size-cap split into `binder-model-and-context.md`, `6829a79` PIC cross-reference for absence-under-the-pin, and onwards); none of those revisions revisited the cast itself. The current finding re-evaluates that choice: audit-greppability is a code-side concern that does not require the cast to live in normative prose.

## Solution Space

**Shape:** single

### Recommendation

Replace the sentence

> The probe is duck-typed: `(model as { strictCapable?: boolean }).strictCapable` is read as a three-valued check — `true` admits the model with no diagnostic; `false` emits `loom/load/binder-model-not-strict-capable` (E) and refuses to register the loom; `undefined` (the field is absent) emits `loom/load/binder-model-strict-capability-unknown` (W) and registers the loom.

with a language-neutral phrasing such as:

> The probe reads the `strictCapable` field on the resolved model as a three-valued check — `true` admits the model with no diagnostic; `false` emits `loom/load/binder-model-not-strict-capable` (E) and refuses to register the loom; absent or `undefined` emits `loom/load/binder-model-strict-capability-unknown` (W) and registers the loom.

Preserve every other element of the paragraph: the field-name pin (`strictCapable`), the three diagnostic outcomes and their severities, the universal-W production claim under the SDK pin (and the cross-reference to `audit-recognised-shapes.md#strict-capability-absence-pin`), the short-circuit when binder-model resolution yields no model, and the rename-detection clause about the build-time SDK surface-inventory assertion. The implementation may continue accessing the field via the same TypeScript cast; the spec simply no longer prescribes it. The audit-greppability rationale that motivated the original cast is satisfied by the code site itself plus the `strict-capability-probe` literal-read inventory row owned in `audit-recognised-shapes.md`, neither of which depends on this paragraph.

Edge case: keep the phrase that distinguishes the *absent* field from a present-but-`undefined` value collapsed into the same outcome (the SDK pin currently exposes neither, so the W branch is universal in production), so a future Pi minor that introduces the field with an explicitly-`undefined` reading does not silently re-route to the E branch.

## Consequence

**Severity:** cosmetic

The behavioural contract is fully specified by the field name, the three-valued read, and the diagnostic mapping; the cast adds no implementer-relevant constraint. Removing it leaves the contract intact and slightly improves layering between behavioural prose and implementation technique.
