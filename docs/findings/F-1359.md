## Problem

The binder-model parse rule ([`binder-model-and-context.md#binder-model-parse-rule`](../spec_topics/binder/binder-model-and-context.md#binder-model-parse-rule)) resolves a binder-model string by loom's own exact-match resolution over `ctx.modelRegistry.getAvailable()`, accepting either a canonical `provider/modelId` reference or a bare `modelId`. The cited spec never names the `Model<Api>` field that carries model identity (the value a bare `modelId` is matched against) nor defines what the `provider` component of a `provider/modelId` reference compares against; the only pinned members are `.strictCapable` and `.api`, and `.api` is explicitly an api-shaped value (e.g. `anthropic-messages`), "not a short provider-id form such as `openai`". No worked `provider/modelId` example exists — only the bare `bind_model: claude-haiku`. An implementer must therefore invent both the identity-field name and the provider-comparison semantics; a wrong guess silently mis-resolves model references rather than failing loudly, and because `V9b` constructs the single matcher instance that both `V6a`'s load-time `model:` resolution and `V11a`'s binder-model resolution bind, the wrong behaviour propagates uniformly to both paths.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 2424003 — pi-loom spec-plan: fix F-1311 — re-specify model-reference matcher as loom's own exact-match resolution over ctx.modelRegistry.getAvailable() (drop unimportable findExactModelReferenceMatch …) (2026-06-24, Thomas Andersen)
**History:** The parse rule originally (commit 54c4e39, 2026-06-03) delegated model-reference matching to Pi's `findExactModelReferenceMatch` helper, which owned the `provider/modelId` vs bare-`modelId` grammar and the field-comparison semantics, so the spec did not need to pin a `Model<Api>` identity field. F-1311 (2424003) dropped that unimportable helper and re-specified the matcher as "loom's own exact-match resolution", carrying the grammar prose over verbatim while transferring ownership of the field/grammar semantics to loom — but it did not supply the `Model<Api>` identity-field definition or the `provider`-comparison target, leaving the gap this finding describes.

## Solution approach

Define the two matcher inputs the resolver depends on at the binder-model parse rule (`binder-model-and-context.md#binder-model-parse-rule`) — or, equivalently, at `host-interfaces-core.md#model-registry-pin` where `Model<Api>` is already pinned, with the parse rule cross-referencing it: the `Model<Api>` field carrying model identity (compared for a bare-`modelId` match and as the `modelId` half of a `provider/modelId` reference) and what the `provider` component compares against. Reconcile the provider-comparison target with the existing api-shaped-`.api` pin, and add a worked `provider/modelId` example alongside the bare `bind_model: claude-haiku` one. Then update the Adds of `V11a`, `V6a`, and `V9b`, and their `-T` matcher-contract Tests bullets, to reference the named identity field and the defined provider-comparison in place of the bare "exact-match over `getAvailable()`" prose.

## Solution constraints

- The provider-comparison definition must reconcile with the existing pin that `Model<Api>.api` is api-shaped (e.g. `anthropic-messages`) and not a short provider-id form — do not silently adopt the `provider == .api` reading that collides with it.
