## Problem

The file `docs/spec_topics/pi-integration-contract/version-bump-triggers.md` carries the heading `# Version bump steps 5–7`, which correctly names the page's content (the later numbered steps of the version-bump procedure), but the filename token `triggers` names a topic the page is no longer about. Every cross-reference across the spec and plan corpus renders the page with the display text "Version bump steps 5–7" while its href embeds the stale `./version-bump-triggers.md` path, propagating the mismatched token into every consumer. Nothing is broken at link-time — the hrefs resolve to the real filename — but the on-disk name advertises a "triggers" topic, so the page is hard to locate by its display name and the misleading token is duplicated corpus-wide.

## Issue introduction
**Verdict:** single-commit
**Introducing commit:** a7a172c (`pi-loom spec: resolve "version-bump-triggers.md H1 mis-scopes page contents"`, 2026-06-05)
**History:** The corpus is git-tracked. The file was created at f5e89f4 (`docs: enforce 100KB hard / 30KB recommended size cap on spec set`, 2026-06-04) with heading `# Version bump triggers` — at that point filename and heading agreed on the `triggers` token (though the heading then mis-scoped the page's step-5–7 content). Commit a7a172c, resolving a prior "H1 mis-scopes page contents" finding, changed the heading from `# Version bump triggers` to `# Version bump steps 5–7` to match the actual content but did not rename the file. That commit is where the filename/heading divergence entered: it corrected the heading and left the filename token stale. `git log --follow` shows no rename across the file's history; the path has been `version-bump-triggers.md` since creation.

## Solution approach

Rename `docs/spec_topics/pi-integration-contract/version-bump-triggers.md` so its filename token reflects the page's "steps 5–7" content rather than `triggers`, and repoint every inbound href across the spec and plan corpus from the old path to the new one. The page-local fragment anchors after the `#` are unaffected — only the path segment changes.

## Solution constraints

- Keep the `# Version bump steps 5–7` heading and all existing "Version bump steps 5–7" display text unchanged; do not revert the filename or heading to a `triggers`-scoped name.
- Out of scope: the regenerated `docs/findings/*.md` and `docs/spec-review*.md` review artefacts that reference the old path.
