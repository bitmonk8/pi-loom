## Problem

The load code registry declares three error-severity `tools:`-resolution codes — `loom/load/invalid-tool-rename`, `loom/load/unresolvable-loom-path`, and `loom/load/unknown-tool` — but no plan leaf asserts any of them. `V6c` / `V6c-T` assert only `loom/load/prompt-mode-callable` and `loom/load/tool-name-collision`; `V15a` covers the `invoke`-path codes and `V9f` the `PIC` behaviours, none of which are these three. Because the codes are in the closed registry (DIAG-2) with no asserting test, the `H5a` closing-gate reconciler has no witness for them and fails the build, which also blocks the `H6a` release-gate flip.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** `V6c` was authored in `c6a664e`, the build of the current plan corpus; its Tests section never named the three `tools:`-resolution load codes. A later edit, `ae66bf1` (2026-06-11, "resolve \"V6c/V6c-T Tests bullets name no diagnostic codes\""), touched exactly this surface and added explicit codes for `loom/load/prompt-mode-callable` and `loom/load/tool-name-collision`, but did not add `invalid-tool-rename`, `unresolvable-loom-path`, or `unknown-tool`. `git log -S` over `docs/plan_topics/` confirms none of the three code strings has ever appeared in any plan leaf of the current corpus. The gap is therefore present since corpus inception, narrowed but not closed by `ae66bf1`.

## Solution approach

Assign each of the three codes a closing plan leaf and add an asserting test in that leaf's `-T` driver. Name `loom/load/invalid-tool-rename` and `loom/load/unresolvable-loom-path` in `V6c` Tests and Ships when, with failing tests added to `V6c-T`, since both arise from the `tools:`-entry resolution `V6c` already owns. For `loom/load/unknown-tool`, which the spec documents as reachable only via the file-watcher reload path, decide in `V6c` whether `V6c` asserts it or defers it to the reload-path owner (`V9f`), then add the asserting test in whichever leaf's `-T` driver owns it.

## Solution constraints

- The spec registry (`code-registry-load.md`) and the `frontmatter-fields-*` pages are read-only for this fix — the codes already exist there.
- The `coverage-matrix.md` DIAG-2 row stays mapped to `V7b`; do not edit the matrix.
- Cite only leaf IDs already in the corpus; do not introduce new leaves.
