## Finding

DISC-5 names `minimatch` as the matcher engine and pins three result-affecting knobs: the override order (plain → `!` → `+` → `-`), the three match targets (package-root-relative path, basename, POSIX-normalised absolute path), and `nocase: off`. Every other minimatch option that materially changes which paths a pattern matches is left at the engine's default by silent omission — most pointedly `dot` (whether leading-dot path segments match `*`), `noglobstar` (whether `**` crosses directory boundaries — moot only if directories themselves are part of the candidate set, but DISC-5 says the starting set is "every path under the package root", so `**` semantics decide what that set contains), `nobrace` / brace expansion (whether `{a,b}` expands), and `matchBase` (whether a pattern lacking `/` matches against the basename — DISC-5 already does basename matching as one of its three targets, so leaving `matchBase` ambiguous compounds with that).

The minimatch *major version* is also unpinned. minimatch's defaults are not contractually frozen across majors (e.g. v3 → v9 changed default-flag behaviour around `nonegate` / `noext` and tightened POSIX bracket handling); H1a declares `minimatch` as a runtime dependency but cites no version literal in the spec corpus, so two implementers can each follow DISC-5 to the letter, install different majors with different default flag values, and produce different match sets for the same `pi.looms` array.

The fallback escape — "delegate to Pi's `applyPatterns`" — is closed off by DISC-5's own wording: it locates that symbol at `dist/core/package-manager.js`, which is the compiled-output path, not a `@earendil-works/pi-coding-agent` package export. An implementer cannot import it through the SDK surface the inventory-closure audit recognises; reimplementation against minimatch directly is the only path open, and that reimplementation requires an option set DISC-5 does not enumerate.

That edit introduced the surface this finding criticises: it pinned exactly one option (`nocase`) and one match-target triple but stopped there, leaving `dot` / `noglobstar` / brace expansion / `matchBase` and the engine major version unaddressed. `git log -G "minimatch" -- docs/spec_topics/discovery/package-and-settings.md` shows only `1cb54c7` touching the `minimatch` token in this file; the gap has not narrowed or widened since. ## Solution Space

**Shape:** multiple

### Option A — Enumerate the option set DISC-5 hands to minimatch and pin the major version

**Approach.** Extend the DISC-5 paragraph with a closed enumeration of every result-affecting minimatch option loom configures, alongside the existing `nocase: off` clause. Each enumerated option is fixed to a literal value (defaults retained, defaults overridden) with a one-line rationale so a future major-version bump that flips a default cannot silently change matching behaviour. Pin the minimatch major version once in `implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)", parallel to how `semver` / `chokidar` / `yaml` are already named there (the existing clauses fix the dependency name, not the version literal; this would fix the major).

**Spec edits.** In `package-and-settings.md` §DISC-5, after the existing `nocase` clause, add a sentence of the form "Matching also runs with `dot: <value>`, `nobrace: <value>`, `noglobstar: <value>`, `matchBase: <value>`; …", choosing each value deliberately rather than implicitly. Cross-reference the major-version pin in `implementation-notes.md`. In `implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)", append a sentence stating the minimatch major version DISC-5's option set is judged against (parallel to the existing pattern that names the dependency without restating a `^N` literal — the literal lives in `H1a`'s `package.json`, which the spec can cite as the version-of-record).

**Pros.** Closed enumeration: implementers and reviewers can mechanically check option-by-option whether their construction matches DISC-5. Version-bump tests can assert that an installed minimatch major matches the pinned major and fail loudly when it drifts.

**Cons.** Couples the spec to a specific matcher library more tightly than today — a future swap to a different engine has to re-render the option set in the new library's vocabulary. Adds prose to a DISC-5 paragraph that is already dense.

**Risks.** If the option list is not exhaustive, the same divergence resurfaces for any option the list missed. The reviewer of any subsequent minimatch major bump has to re-audit the enumerated values against the new major's defaults.

### Option B — Reframe DISC-5 as behavioural parity with Pi's resource-array glob dialect

**Approach.** Demote the literal `minimatch` / `applyPatterns` references to a non-normative implementation note and state the normative requirement as behavioural parity with Pi's resource-array glob dialect (the dialect Pi applies to `pi.extensions` / `pi.skills` / `pi.prompts` / `pi.themes`) over the three match targets and the override order DISC-5 already fixes. Define parity observationally: for every `(pattern, candidate-path)` pair, loom's matcher returns the same boolean Pi's would.

**Spec edits.** In `package-and-settings.md` §DISC-5, replace the "Glob patterns are matched with the `minimatch` engine …" sentence with a behavioural-parity clause; move the `minimatch` / `dist/core/package-manager.js` / `applyPatterns` references into a parenthetical implementation note marked non-normative. Leave the option set unenumerated (it is implied by parity).

**Pros.** Resolves this finding and the related over-prescription finding in one edit. Survives a future Pi-side matcher swap without a spec change. Sidesteps the `dist/`-internal-symbol reachability problem entirely.

**Cons.** Replaces a finite enumeration problem with an unbounded conformance problem: "matches Pi" is not directly testable without a Pi instance to compare against, so the V10b tests must either ship a corpus of `(pattern, candidate-path, expected)` triples that pin the dialect by example or import Pi at test time. The first is a substantial test-corpus authoring task; the second creates a build-time dependency on Pi internals that V18b's inventory-closure audit then has to recognise.

**Risks.** Pi's dialect itself is not specified in `@earendil-works/pi-coding-agent`'s public docs — `applyPatterns` lives at the same `dist/`-internal path DISC-5 currently cites — so the parity contract reduces to "matches whatever Pi's compiled `dist/` does today", which is no firmer a target than the current minimatch-default-defer.

### Recommendation

Take **Option A**. The over-prescription concern raised by the related finding is real but is the lesser cost: DISC-5 already pins one option (`nocase`) and three match targets explicitly, so it is already in the enumeration business — extending the enumeration to cover `dot` / `noglobstar` / `nobrace` / `matchBase` and pinning the major version closes the divergence with one well-scoped edit that V10b-T can mechanically test. Option B's parity contract has no testable referent today (the only published anchor for Pi's dialect is the same `dist/`-internal symbol DISC-5 already cites), so it trades a fixable enumeration gap for an unfixable conformance gap.

Edge cases for the implementer:

- The enumerated option set must be authored against a *specific* minimatch major, picked deliberately. The current minimatch default for `dot` is `false`, which means a pattern like `*.loom` does **not** match `.foo.loom`; DISC-5 should state whether that exclusion is intended (it presumably is, since the rest of discovery treats `*.loom` byte-exactly) and pin `dot: false` explicitly rather than by default-inheritance.
- `noglobstar` is consequential because DISC-5's starting set is "every path under the package root" — with `noglobstar: false` (the default) a plain pattern `**/*.loom` walks subdirectories, which contradicts the non-recursion rule applied to the `looms/` fallback directory. Pin a value that makes the `pi.looms` recursion behaviour observable and consistent with the non-recursion rule, and state it.
- `matchBase` interacts with DISC-5's existing basename-match target: if `matchBase: true`, a slashless pattern matches against the basename via *two* paths (DISC-5's three-target loop plus minimatch's own basename fallback), which is redundant but harmless; if `matchBase: false`, the basename-match target is solely DISC-5's responsibility. Either choice works as long as DISC-5 picks one and the V10b tests assert it.
- The major-version pin in `implementation-notes.md` should name the major the option literals were chosen against (e.g. "minimatch v9"); a later bump that flips any of the enumerated defaults requires the spec edit to be re-validated, not the `H1a` version range alone.

## Consequence

**Severity:** correctness

Two implementers each following DISC-5 to the letter can produce divergent match sets for the same `pi.looms` array — most visibly for patterns containing `**`, leading-dot segments, or `{a,b}` braces. The divergence is silent: each implementation passes its own unit tests because minimatch's defaults are internally consistent, and only a cross-implementation comparison surfaces it. A user moving between two such loom installations sees their `pi.looms` array load a different set of `.loom` files with no diagnostic.
