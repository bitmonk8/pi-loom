## Finding

The closing sentence of the *Post-probe SDK-shape drift* paragraph in `capability-probe.md` is a forward-looking deferral inlined inside the loom 1.0 routing contract:

> A `details.kind` discriminator on `loom/runtime/internal-error` distinguishing host-drift from loom-defect is an additive registry-minor change deferred past loom 1.0 per [Diagnostics](../diagnostics.md)'s registry-change rules; nothing in the loom 1.0 routing contract precludes adding it later without breaking emitted shapes.

Two things are wrong with this sentence sitting where it does. First, it carries no normative content for loom 1.0 — it describes a future registry edit and re-asserts a property (additive `details.kind` additions are non-breaking) that is already governed by `diagnostics/diagnostic-shape.md` DIAG-2 ("the registry is closed … A code addition or removal under this rule has a defined loom-version disposition: it is a recognised GOV-15 diagnostic-registry carve-out admissible within a loom 1.x minor release") and by GOV-15's diagnostic-registry carve-out in `governance/source-language-stability.md`. The deferral therefore restates an existing rule rather than reserving a new one. Second, the sentence does not cite a future-considerations anchor where the deferred discriminator is recorded; the loom 1.0 future-considerations corpus (`future-considerations/model-changes-and-non-goals.md`, `future-considerations/surface-extensions.md`) has no item for it, so the deferral is *only* recorded in this inline aside on the routing page — exactly the placement the corpus elsewhere routes to a `*Recorded at:*` slot on a future-considerations bullet.

The result is forward-looking decision-log prose interleaved with the present-tense routing MUSTs the paragraph exists to state, with no inventory entry on the future-considerations side to keep it in sync with.

## Solution Space

**Shape:** single

### Recommendation

Delete the sentence. The two things it currently asserts are both already owned upstream — that additive `details.kind` additions are an admissible registry-minor change is pinned by `diagnostics/diagnostic-shape.md` DIAG-2 and the GOV-15 diagnostic-registry carve-out in `governance/source-language-stability.md`, and the absence of a present-tense routing obligation for the host-drift-vs-loom-defect distinction is already conveyed by the preceding sentence ("Operator triage of `loom/runtime/internal-error` therefore cannot, in loom 1.0, distinguish a host-drift failure from a loom-runtime defect on the `error.message` / `error.stack` payload alone"). Deleting the trailing sentence removes the redundant restatement of DIAG-2 + GOV-15 and the dangling forward-looking aside in one stroke; no future-considerations bullet needs to be coined, because deferring an additive registry minor is already the default disposition for any discriminator the loom 1.0 spec does not pin.

Edge cases for the fixer: do not delete the immediately preceding sentence (the operator-triage observation closes the loom-1.0 routing contract and is the resolution of an earlier review pass per `eb33bf9`); the `#post-probe-sdk-shape-drift` anchor immediately above the paragraph MUST be preserved (it is the inbound anchor used by other topic pages); and the edit is paragraph-internal, so no cross-references need to be retargeted.

## Consequence

**Severity:** cosmetic

The inline sentence carries no normative weight: the registry-additivity property it restates is already pinned by DIAG-2 + GOV-15, and the deferred discriminator itself is not a loom 1.0 obligation. Leaving it in place adds editorial-review surface (a forward-looking aside that must be re-read against DIAG-2 on every audit pass) and a stale-divergence risk once the registry carve-out wording evolves, but no implementer behaviour changes either way.
