## Problem

`V15e`'s eviction correctness rests entirely on the transitive-importer computation: on a `LoomRegistry` swap it drops the static-resolution parse-cache entry for the changed file and every file that transitively imports it across the `.warp` import-edge graph (`V15c`). If that computation under-counts — misses one transitive importer — the next load pass reuses a stale parsed/lowered entry, and the failure is silent (no diagnostic, no error; the served entry is structurally valid but out of date). Both `V15e` and `V15e-T` carry only a happy-path Tests bullet and Ships-when arm; neither exercises the under-eviction case. The coverage gate therefore passes without ever witnessing the one failure mode that makes this leaf risky.

## Solution approach

Add a negative-direction Tests bullet to `V15e-T` and its paired Tests / Ships-when arm to `V15e` covering a reload where a file's transitive `.warp` import edge to the changed file is newly established — a case a naive pre-swap-graph computation would miss — and asserting that file is re-parsed on the next load pass rather than served from cache. Pin the test against the ordering between the import-edge-graph rebuild and the eviction walk so it cannot pass vacuously. Alternatively, the implementer may close the gap with a per-entry staleness stamp (content hash / mtime) verified before serve, turning a silent stale serve into a detectable miss; that path additionally entails a spec obligation in `implementation-notes.md`.

## Solution constraints

- Keep `V15e` and `V15e-T` consistent: a Tests assertion added to one must have its paired arm in the other (plan TDD-pairing convention).
- If the staleness-stamp path is taken (adding a spec obligation in `implementation-notes.md`), keep `coverage-matrix.md`'s IMPL static-resolution row consistent with the new normative content.
