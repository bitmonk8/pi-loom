## Problem

The closing gate's un-anchored-MUST recogniser treats a `<new>` placeholder coverage-matrix row as "names no real closing leaf" and therefore as not satisfying the closing-leaf condition. This rule is stated semantically ("names no real leaf") in three surfaces — `coverage-matrix.md`'s *Code-keyed obligation areas (no numbered REQ-IDs)* placeholder paragraph, `conventions.md` *REQ-ID discipline*, and `H5a`'s un-anchored-MUST arm — but the mechanical recogniser the `H5a` implementer must build is never pinned. Two implementers diverge: one reads it as a literal-`<new>`-token match (so any other token, including a typo'd or non-existent closing-leaf ID like `V99z`, counts as "naming a closing leaf" and passes the gate vacuously); the other reads it as resolution against the real plan leaf-ID universe (so any non-resolving token reddens the gate). The ambiguity is compounded by `H5d`, which already pins its recogniser on backtick presence, so the two arms can classify the same cell differently.

The residual recognition-rule ambiguity is thus a single-commit introduction by the resolution `a85a913`.

## Solution approach

Pin the recogniser mechanically and propagate it to the three surfaces that restate it. In `coverage-matrix.md`'s *Code-keyed obligation areas* placeholder paragraph, state that the placeholder is exactly the literal closing-leaf token `<new>` and that any other closing-leaf token not resolving to an existing plan leaf ID reddens the gate as a defect rather than being read as a benign placeholder. Pin the same resolution-against-the-real-leaf-universe rule in `H5a`'s un-anchored-MUST arm (the *Adds* clause and the `<new>`-placeholder *Tests* bullet), and in `conventions.md` *REQ-ID discipline* replace the bare semantic phrasing with a reference to that pinned rule. Choose the literal `<new>` token form so it and `H5d`'s backtick-keyed tokeniser select the same cell.
