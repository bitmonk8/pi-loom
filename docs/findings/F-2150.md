## Problem

The closing sentence of the re-entrant `execute`-adapter paragraph in
`extension-bootstrap-and-per-loom.md` (§Per-loom registration) — "Adapter
implementations MUST NOT serialise calls on shared closure state." — has an
ambiguous prepositional attachment. "On shared closure state" can read as
*manner* (MUST NOT serialise concurrent calls by means of shared closure state)
or as *scope* (the prohibition applies only to the subset of calls that touch
shared closure state). Under the scope reading an implementer can satisfy the
literal MUST NOT while still serialising concurrent calls through a non-closure
mechanism (external queue, mutex), silently negating the re-entrancy guarantee
the preceding sentence establishes for parallel-tool-mode invocations.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** 7ce0ffa — pi-loom spec: resolve "Tool registry mutation during execution and model-driven parallel tool calls" (2026-05-04, Thomas Andersen)
**History:** Commit 7ce0ffa added the re-entrant `execute`-adapter paragraph to the then-monolithic `pi-integration-contract.md`, introducing the sentence "Adapter implementations MUST NOT serialise calls on shared closure state." verbatim. The grammatical ambiguity was present at the sentence's inception and carried unchanged through the later split that relocated the paragraph into `extension-bootstrap-and-per-loom.md`; no subsequent commit altered the wording.

## Solution approach

Rewrite the closing sentence of the re-entrant `execute`-adapter paragraph in
`extension-bootstrap-and-per-loom.md` so its prohibited target is named
unambiguously — the adapter's own per-adapter shared serialising state, not
merely closure-stored state — closing the non-closure-serialisation loophole the
current attachment ambiguity leaves open.
