## Finding

`PIC-22. Parallel subagent spawn initiation` in `docs/spec_topics/pi-integration-contract/subagent.md` states the runtime MUST initiate `createAgentSession` for all N parallel subagent-mode `.loom` callables before any one returns, and pins its conformance witness — a fake `AgentSession` whose `sendUserMessage` blocks until released — only "for any fixed N ≥ 3". Two sentences earlier the `<a id="no-invocation-cap">` paragraph independently MUSTs that the runtime "interpos[es] no admission cap, priority queue, or scheduler between sibling invocations and the event loop". PIC-22 exists specifically to make that prior no-cap MUST observable (it landed at commit `7afe9bb` resolving spec-review finding T14, "no-invocation-cap MUST NOT is unobservable").

The "N ≥ 3" lower bound defeats that purpose at N = 2. A runtime that serialises a parallel pair — admit the first `createAgentSession`, wait for `sendUserMessage` to return, then admit the second — violates the no-admission-cap MUST but never triggers PIC-22's witness, so two reasonable implementers can produce divergent N = 2 behaviour with identical conformance results. N = 1 is trivially vacuous (any "all N before any returns" rule is satisfied), but N = 2 is the smallest meaningfully observable case and the spec leaves it on the wrong side of the trigger.

## Issue introduction

**Verdict:** present-since-inception (of PIC-22)
**Introducing commits:** `7afe9bb` (2026-05-31, "pi-loom spec: resolve \"no-invocation-cap MUST NOT is unobservable\"")
**History:** `git log -S 'for any fixed N' -- docs/spec_topics/pi-integration-contract.md` identifies `7afe9bb` as the sole introducing commit; the rule was added to `pi-integration-contract.md` and later moved to `subagent.md` under the topic-split. The "small fixed N (≥ 3)" wording originated in `docs/spec-review.md`'s T14 *Solution approach* (also part of `7afe9bb`) and was carried verbatim into the normative paragraph and the downstream `V9i` / `V9i-T` plan leaves. No later commit revised the bound; the gap has been present since the witness was first defined.

## Solution Space

**Shape:** single

### Recommendation

Lower the PIC-22 trigger bound to N ≥ 2 in both the MUST sentence and its conformance-test sentence, leaving the rest of the paragraph unchanged. N = 2 is the smallest case where "all N initiated before any returns" is a non-vacuous, mechanically testable obligation, and it is the smallest case at which a capping runtime can be caught.

Concretely:

- In the MUST sentence, replace "for any fixed N ≥ 3" with "for any fixed N ≥ 2".
- In the conformance-test sentence, the existing wording ("a conformant runtime has created all N sessions and entered each session's `sendUserMessage` before any blocked call is released") needs no change beyond the bound; the fake-`AgentSession` fixture works identically at N = 2.
- Mirror the bound change in `plan_topics/V9i-subagent-isolation.md` and `plan_topics/V9i-T-subagent-isolation.md`, where the test fixture is pinned at "N=3"; lower to "N=2" so the leaf acceptance criterion exercises the new minimum.

Do **not** lower the bound to N ≥ 1: at N = 1 the obligation is vacuous (the single call's initiation trivially precedes its own return), so stating it there adds no observable constraint and invites readers to mistake the rule for one about per-call latency rather than admission ordering. Do **not** take the alternative "explicitly exclude N ≤ 2 with a named governing rule" — that would carve out the exact case that makes the no-admission-cap MUST testable, defeating PIC-22's reason for existing.

Edge case the implementer must watch: the fake `AgentSession` fixture for N = 2 must still assert both sessions' `sendUserMessage` have been *entered* (not merely that both `createAgentSession` calls resolved) before either blocked call is released — otherwise a runtime that constructs both sessions sequentially but then dispatches `sendUserMessage` in series still passes. The N = 3 wording already requires this; preserve it verbatim at N = 2.

## Consequence

**Severity:** correctness

A conformant-on-paper runtime that admits parallel subagent calls behind a cap-of-2 (or any cap ≥ 2) passes the entire conformance suite while breaking the no-admission-cap invariant the surrounding paragraph asserts. The visible symptom is that the second of two parallel subagent siblings serialises behind the first — exactly the storm-mitigation behaviour the no-cap rule was written to forbid — and the gap is invisible to any test that doesn't deliberately scale N to 3.
