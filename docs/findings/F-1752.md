## Problem

SLSH-4 row `SNK-j` defines a per-`kind` template for `invoke_callee`, and SLSH-4's umbrella rule mandates that renderers emit each row's template verbatim and permits conformance tests to assert on the exact string. SLSH-5, however, mandates that on any `invoke_callee` cascade the descriptive text is computed from the **leaf** `kind`, not from the wrapper — and because every concrete `InvokeCalleeError` carries a required `inner: QueryError`, there is no inner-less `invoke_callee`, so SNK-j is unreachable. The two rules prescribe different output for the same input and the page names no winner: a renderer satisfying SNK-j violates SLSH-5, and one satisfying SLSH-5 fails SNK-j. SNK-j is the unique row whose `kind` SLSH-5 strips out, and `V12b`'s test list bakes the conflict in by asserting SNK-j as a discrete obligation.

## Issue introduction

**Verdict:** single-commit-regression
**Introducing commit:** `2d217b1` (2026-05-06, *pi-loom spec: resolve "Call chain suffix is illustrative, not normative"*)
**History:**

- `907c029` (2026-05-04) added the `invoke_callee_error` row to the per-`kind` table with the template `"… invoked <callee_path> returned Err — <inner.kind>"`. At that point the table row was the renderer's sole output for an `invoke_callee_error`; an inline parenthetical hinted at chain-printing but was not normative ("the note identifies the leaf and prints the call chain"). No ordering conflict existed: the row was the rule.
- `2d217b1` (2026-05-06) elevated the chain-attribution paragraph to a normative MUST and added the leaf-kind clause: *"the descriptive text in the per-`kind` row is computed from the **leaf** `kind` (the innermost non-`invoke_callee_error` variant), not from the wrapper"*. This is the regression site: the new rule made the `invoke_callee_error` row unreachable for any concrete `InvokeCalleeError` (whose `inner` is non-optional), but the row was not removed and no supersession note was added.
- `27b2576` (2026-06-05) added the `SNK-a … SNK-k` inline labels for citation, including `SNK-j` for the already-dead row. This formalised the unreachable row as a separately-citable obligation (and propagated into `V12b`'s test list as a discrete bullet), amplifying the defect's surface but not introducing it.
- `1ccbb75` (2026-06-07) refined `<line>` definition and renamed `invoke_callee_error` → `invoke_callee` to match the wire-kind pattern; the leaf-kind / wrapper-row conflict was preserved verbatim through both renames.

## Solution approach

Delete the SNK-j row and its `id="snk-j"` anchor from the SLSH-4 per-`kind` table in `docs/spec_topics/slash-invocation.md`, and clarify in the post-table prose — the existing `For invoke_callee the chain-attribution suffix described in the next paragraph…` sentence is the natural home — that `invoke_callee` has no dedicated per-`kind` row because every concrete `InvokeCalleeError` triggers SLSH-5's leaf-kind selection. Remove the `SNK-j` test bullet from `docs/plan_topics/V12b-top-level-err-chain.md` and `docs/plan_topics/V12b-T-top-level-err-chain.md`, since SLSH-5 plus the leaf row now cover it. Update the `SNK-a … SNK-k` closure-rationale prose in `docs/plan_topics/coverage-matrix.md` to reflect SNK-j's retirement.

## Solution constraints

- Do not renumber SNK-k (or any other SNK label) to close the SNK-j gap; SNK labels are GOV-16-governed identifiers and renumbering breaks existing citations — leave the range non-contiguous.
- Out of scope: any change to the SLSH-5 leaf-kind selection rule itself.
