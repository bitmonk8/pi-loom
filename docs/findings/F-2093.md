## Finding

The *Binder inference call* section restates the complete shapes of two `@earendil-works/pi-ai`-owned types instead of citing loom's *use* of them ŌĆö even though the same section's opening paragraph explicitly states the opposite intent ("this paragraph pins loom's *use* of that surface, not the surface itself, which `@earendil-works/pi-ai` owns").

Two sites are affected:

- The intro paragraph carries the closed type equation `ProviderStreamOptions = StreamOptions & Record<string, unknown>`. This reproduces the full definition of an externally-owned type. The token `ProviderStreamOptions` appears nowhere else in the corpus and is never consumed by any loom rule ŌĆö the equation is dead weight whose only effect is to mirror a sibling-package type definition that loom does not own.
- The `options.onResponse` bullet names the captured value as `ProviderResponse { status, headers }`, a closed-brace form asserting that `status` and `headers` are the type's *complete* field set. Loom reads only `status` (joined per-invocation to supply the classifier's HTTP-status input); `headers` is captured but not consumed by any rule on this page.

Both restatements duplicate the surface rather than pin loom's use of it, and both can silently drift from the actual `@earendil-works/pi-ai` declaration on a Pi minor bump ŌĆö exactly the failure mode the use-not-surface principle exists to prevent. The canonical use-citation already lives on the *Classifier input surface* note in `provider-error-mapping.md`, which states the full `ProviderResponse` shape once and is the authoritative reader-side description; the binder page should reference loom's read fields plus the `dist/types.d.ts` declaration site rather than re-stating closed shapes.

## Issue introduction
**Verdict:** multi-commit-accretion
**Introducing commits:** `fb3474f` (2026-06-04, *resolve "Binder inference call ŌĆö no pi-ai entry point pinned"*); `44782e9` (2026-06-06, *resolve "TransportError catch-all / pi-ai provider-error surface"*)
**History:** The two restatements entered the corpus independently. `fb3474f` added the `ProviderStreamOptions = StreamOptions & Record<string, unknown>` equation to `pi-integration-contract.md` when first pinning the binder `complete()` entry point; that text was carried verbatim into the newly-extracted `binder-inference.md` at the `f5e89f4` size-cap file split later the same day (`--follow -S` reports the equation entering the current path at `f5e89f4`, but its content originates at `fb3474f`). `44782e9` then added the `ProviderResponse { status, headers }` closed-shape form to the `options.onResponse` bullet as part of wiring the provider-error classifier's HTTP-status input. Each commit added one complete-shape restatement that was locally coherent in isolation; the defect is the page accreting two closed external shapes rather than use-citations.

## Solution Space

**Shape:** single

### Recommendation

On the binder-inference page, replace both complete-shape restatements with use-citations that name only what loom reads, deferring the type's full shape to its `@earendil-works/pi-ai` declaration:

- For `ProviderResponse`: drop the closed `{ status, headers }` form in the `options.onResponse` bullet. Reference the field loom actually reads (`ProviderResponse.status`, joined per-invocation), note that the value is captured via `StreamOptions.onResponse`, and cite the `dist/types.d.ts` declaration site ŌĆö mirroring the reader-side phrasing already used in `provider-error-mapping.md`'s *Classifier input surface* note. Do not re-enumerate the type's fields.
- For `ProviderStreamOptions`: drop the `= StreamOptions & Record<string, unknown>` equation. The intro paragraph already names `StreamOptions` as a consumed type declared at `dist/types.d.ts`; the parenthetical equation adds nothing loom uses and should be deleted (or, if the relationship is worth retaining for orientation, reduced to a non-normative aside citing the declaration site rather than reproducing the definition).

Adopt the same use-citation form the *Classifier input surface* note uses so the two pages stay consistent. Keep the existing per-Pi-minor-bump re-validation pointer intact ŌĆö it is what guards the cited surface against drift once the restated shapes are removed.

## Consequence

**Severity:** advisory

No implementer diverges and no runtime behaviour changes today: the declaration site (`dist/types.d.ts`) is cited and loom reads only the fields it needs. The hazard is maintenance drift ŌĆö a restated closed shape (`{ status, headers }`) or a restated type equation goes stale if `@earendil-works/pi-ai` adds or renames a field on a minor bump, leaving the binder page asserting an external contract it does not own and contradicting its own use-not-surface principle.
