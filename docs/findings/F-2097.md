## Finding

Within the `binder/` topic cluster the bare word "token" carries two unrelated meanings with no disambiguation, and both senses occur in the same files (in one case the same file).

- **LLM-context-token sense (a count unit).** In `binder-model-and-context.md` → *Session-context truncation* the word denotes provider context tokens: "Token counts are computed per message", the `8000`-token budget, the `TokenEstimator` / `estimateTokens` seam, "the running token sum", "20 turns / 8000 tokens". This is a numeric resource the truncation walk sums.
- **Required-literal-string sense (an exact byte fragment).** In `binder-bypass-and-envelope.md` → *System-prompt structure (normative)* the same word names mandated prompt substrings: the intro pins "the listed tokens, line-prefixes"; item 2 forbids "no `Description:` token with an empty value"; item 4 pins "exactly one of the literal tokens `required` or `default=<literal>`", "the fixed token order `<wire-name> (<type>) <requirement>`", and "these three tokens"; item 5 references "the `User arguments:` token"; item 6 requires the prompt to "begin with the literal token `Recent session context`"; item 7 pins "their `kind`-token names" and "the three kind-name tokens". The same literal-string sense reappears in `binder-model-and-context.md` itself ("every other character of the template — including the literal token `loom(s)`").

The two senses share no referent: one is a count produced by an estimator, the other is a mandated byte sequence. `binder-model-and-context.md` uses both within one page. The spec coins no term-of-art for either, so a reader in a dense normative paragraph must infer from surrounding context which sense applies. The byte contracts and the token budget are each independently and unambiguously pinned, so the overload is a vocabulary-hygiene defect rather than an under-specification.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** literal-string "token" arm accreted across `4188412` ("Parameters block: indentation and per-field token order are not normative", 2026-05-07 — introduced "literal tokens `required`" / "fixed token order"), `5a4ec51` ("Binder system prompt information content lacks verifiable field set" — introduced the *System-prompt structure (normative)* items and "the listed tokens"), and `f88698e` ("Hot-reload looms.binderModel recovery note under-specified" — introduced "the literal token `loom(s)`"). LLM-context-token arm present since the binder session-context content's formative authoring (already present at the first consolidated review `a7cfc30`, 2026-05-05).
**History:** `git log -S` shows no single commit that introduced both senses. The context-token vocabulary in *Session-context truncation* predates the cluster's review history; the required-literal-string use of "token" was reached for independently by several later normative-sharpening commits, each landing in proximity to the older count sense without reconciling vocabulary. The collision is therefore the cumulative result of independent edits, not a single defective change.

## Solution Space

**Shape:** single

### Recommendation

Reserve "token" exclusively for the LLM-context-token sense (the estimator-counted unit in *Session-context truncation*). Everywhere the word currently names a required prompt substring, replace it with "literal string" / "literal text" / "exact byte sequence" — matching the byte-pin vocabulary the spec already uses (e.g. "the byte sequence U+0020 U+2014 U+0020"). Concretely:

- `binder-bypass-and-envelope.md` *System-prompt structure (normative)*: intro "the listed tokens" → "the listed literal strings"; item 2 "no `Description:` token" → "no `Description:` literal prefix"; item 4 "the literal tokens `required` or `default=<literal>`" → "the literal strings ...", "the fixed token order" → "the fixed field order", "these three tokens" → "these three fields"; item 5 "the `User arguments:` token" → "the `User arguments:` literal prefix"; item 6 "the literal token `Recent session context`" → "the literal string `Recent session context`"; item 7 "their `kind`-token names" / "the three kind-name tokens" → "their `kind` literal-string names" / "the three `kind` literal strings".
- `binder-model-and-context.md` recovery-note substitution rule: "the literal token `loom(s)`" → "the literal string `loom(s)`".
- Leave every use of "token" in *Session-context truncation* (and the `TokenEstimator` seam name, which is an SDK contract) unchanged.

This is a faithful one-pass rename of a single overloaded word; it introduces no new normative content and changes no pinned byte sequence or numeric cap.

## Consequence

**Severity:** cosmetic

A reader scanning the binder cluster must disambiguate "token" by context each time it appears, with the literal-string sense and the count sense colliding on a single page. Because both the prompt byte contracts and the context-token budget are independently pinned, no implementer divergence results; the cost is interpretive friction and reduced searchability, not incorrect behaviour.
