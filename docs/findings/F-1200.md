## Finding

The "Dedent and newline-trim — normative behaviour" section of `docs/spec_topics/query/query-forms.md` opens with a sentence that names *two* normative references in one breath: "The normative reference is the behaviour of CPython 3.x `textwrap.dedent` *as illustrated by the table below*; an implementer who cannot read the CPython source can still pass the conformance tests from the spec alone." This invites a divergence whenever CPython's behaviour and the table's coverage are not identical — i.e. always, because the table has eight rows and `textwrap.dedent` has an effectively unbounded input domain.

The three numbered behaviour bullets and the eight-vector table cover ASCII space and ASCII tab inputs with LF line endings only. CPython's `textwrap.dedent` (an `re.sub` over the `(?m)^[ \t]+` pattern in stripped lines, with `_whitespace_only_re` matching `[ \t]+$`) has externally observable behaviour on inputs the spec never names: CRLF and bare-CR line endings (does the LF or the whole CR/LF participate in "whitespace-only line" detection? what does the common-prefix walk do across mixed line terminators?), non-ASCII leading whitespace such as U+00A0 NO-BREAK SPACE or U+3000 IDEOGRAPHIC SPACE (CPython's whitespace-only-line regex `^[ \t]*$` does *not* match these — they participate in the literal common prefix; the spec's three bullets say "tab-only and space-only … uniformly" but do not address the wider Unicode whitespace question), and exotic inputs such as a sole `\r\n` template body. Two implementers — one who pins behaviour to the table and a Python port of `textwrap.dedent`, one who pins behaviour to the spec's three bullets plus the eight vectors — will produce divergent rendered prompt text on these inputs without either contradicting the spec text. Because the rendered text is what the model sees, this is a wire-observable divergence.

The asymmetry is structural: the table is *illustrative of* CPython behaviour, not a superset or replacement of it. Whenever the two are co-listed as "the normative reference", an implementer hitting an input outside the table is forced to guess which authority wins. The fix is to pick one — and the natural choice given the surrounding ecosystem (loom is implemented in TypeScript, not Python) is to make the spec itself authoritative and the CPython reference a non-normative pointer to a working implementation of the same algorithm.

## Solution Space

**Shape:** multiple

### Option A — Spec-primary; CPython reference demoted to a non-normative pointer

**Approach.** Rewrite the opening sentence so the spec text (three behaviour bullets + the eight-vector table + any obligations the rewrite adds) is the sole normative authority. Add a short non-normative aside saying CPython 3.x `textwrap.dedent` is one working reference implementation of the same algorithm. Extend the three behaviour bullets to close the inputs the current text leaves ambiguous: (i) line endings — state that newline-trim and dedent operate on LF (`\n`); a `\r` byte in the template body is a literal content byte that does *not* participate in line splitting and does *not* satisfy "whitespace-only line" — or, alternatively, that CRLF is folded to LF before the algorithm runs; pick one and pin it; (ii) the whitespace alphabet — pin "leading whitespace" to ASCII space (U+0020) and ASCII tab (U+0009) only (matching the alphabet used at `binder/defaulting-system-note-echo.md` § "System-note rendering" rule 1, and matching CPython's `[ \t]` regex), and state that any other code point — including U+00A0, U+3000, and the rest of the Unicode whitespace block — is treated as ordinary content for both the common-prefix walk and the whitespace-only-line predicate. Optionally add two or three vectors to the table exercising the chosen line-ending rule and a non-ASCII-whitespace leading character so the new obligations have falsifiable fixtures.

**Spec edits.**
- `docs/spec_topics/query/query-forms.md` § "Dedent and newline-trim — normative behaviour": rewrite the opening sentence; extend behaviour bullets 1–3 with the line-ending and whitespace-alphabet pins; append vectors covering the new obligations to the table and matching commentary entries.
- Cross-reference `binder/defaulting-system-note-echo.md` § "System-note rendering" rule 1 for the ASCII-whitespace alphabet rather than restating it.

**Pros.** The spec becomes self-contained; no implementer needs to consult an external project to determine wire-observable behaviour. Aligns the dedent alphabet with the already-pinned system-note alphabet (one whitespace definition across the spec, not two). The fix is local — one section.

**Cons.** A few input shapes that `textwrap.dedent` handles by happy accident (e.g. CRLF) now require explicit spec text. The table grows by two or three rows.

**Risks.** Picking the wrong CRLF rule could surprise authors who paste Windows-line-ending templates from a clipboard; the rationale for the chosen rule should sit beside it.

### Option B — CPython-primary; the table demoted to a non-normative subset

**Approach.** Rewrite the opening sentence so CPython 3.x `textwrap.dedent` is the sole normative reference and the table is labelled "non-normative examples illustrating common cases". Add a sentence stating that on any input outside the table, the rendered text MUST match what `textwrap.dedent` (CPython 3.x, with the version pinned) produces on the same input after newline-trim. Keep the three behaviour bullets as a non-normative summary.

**Spec edits.**
- `docs/spec_topics/query/query-forms.md` § "Dedent and newline-trim — normative behaviour": rewrite the opening sentence; relabel the table; pin the CPython minor version (e.g. 3.12) so the reference is stable over time; add a "tie-breaking" sentence pointing at the CPython implementation on any input the table does not cover.

**Pros.** Cheapest edit (one paragraph, no new vectors). Inherits all of CPython's edge-case behaviour for free.

**Cons.** Implementers must read and faithfully port (or invoke) a Python standard-library function to be conformant — a tooling burden in a TypeScript-implemented runtime. Pins the spec to an external project's release cadence and back-compat policy; a CPython behavioural change in 3.13+ would silently move the spec.

**Risks.** Cross-language conformance testing requires either a Python installation in CI or a vetted TypeScript port; the spec acquires a transitive dependency on the CPython release notes.

### Recommendation

Option A. The spec is normative for this runtime; an external implementation in a different language and release cadence is the wrong place to anchor wire-observable behaviour, and the existing ASCII-whitespace pin at `binder/defaulting-system-note-echo.md` already establishes the precedent that the spec defines its own whitespace alphabet rather than borrowing one. Resolve the CRLF question by stating that the parser operates on LF-terminated lines and a literal `\r` in the template body is content — this matches the behaviour of every other line-oriented rule in the spec. Edge cases an implementer should watch when reading the rewritten section: (i) the alphabet pin must apply to *both* the common-prefix walk *and* the whitespace-only-line predicate, not just one; (ii) the `descriptions.md` cross-reference ("same algorithm as query-template dedent") inherits whatever the rewritten section says, so authors of `///` doc-comment lines see the same alphabet.

## Consequence

**Severity:** correctness

Two implementers conforming to the literal spec text can produce different rendered prompt strings on inputs outside the eight tabulated vectors (CRLF templates, templates whose lines are indented with U+00A0 or U+3000, templates mixing CRLF and LF). Different rendered prompts produce different model outputs, so a conformance gap here directly weakens cross-implementation reproducibility of typed-query results.
