## Finding

The `SNK-k` test bullet in both `V12b` and its paired `V12b-T` reads: "the catch-all note renders verbatim and the renderer is total over any unlisted `kind` in the `QueryError` union." The two clauses are not the same kind of claim. "Renders verbatim" is a concrete, testable property — feed the renderer a `QueryError` whose `kind` is unmatched by SNK-a..j and assert the emitted string equals the SNK-k template. "Total over any unlisted `kind`" is a universally-quantified claim over an open domain: it asserts the renderer has a defined output for *every* possible unmatched `kind` value, not just the one a test exercises.

A runtime test that interpolates one synthetic out-of-enum `kind` demonstrates that the catch-all path is *reachable* and *correct for that input* — it does not, and cannot, establish totality. Per the spec ([`slash-invocation.md`](../spec_topics/slash-invocation.md) SLSH-4), the catch-all row "makes the renderer's contract total against any future variant added to the union, so a renderer never has 'no defined output' for a well-formed `QueryError`." That totality is a structural property of the renderer's default/fallback branch — it holds because the branch catches anything SNK-a..j did not, not because any finite set of inputs was tried. The bullet, as worded, frames a structural guarantee as something a verbatim-render assertion proves.

The risk is that an implementer satisfies the bullet with a single synthetic-`kind` test and treats the totality clause as discharged, when a renderer with a partial dispatch that happens to fall through for that one value (but throws or returns undefined for some others) would pass the test while not being total.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the `SNK-k` Tests bullet identically in both `docs/plan_topics/V12b-top-level-err-chain.md` and `docs/plan_topics/V12b-T-top-level-err-chain.md` so the verbatim-render assertion and the totality property are stated as the distinct claims they are: the test exercises a representative unlisted `kind` (reachability + correct verbatim output), and totality is attributed to the renderer's default/fallback branch (the structural catch-all), not to the finite test.

Replace the current bullet text:

> `SNK-k`: the catch-all note renders verbatim and the renderer is total over any unlisted `kind` in the `QueryError` union.

with:

> `SNK-k`: a representative unlisted `kind` (not matched by SNK-a..j) renders the catch-all note verbatim; renderer totality over `QueryError` rests on the default/fallback branch emitting this template for any `kind` SNK-a..j does not match (a structural catch-all, not a property proven by the single synthetic input).

The `<kind>` and `<message>` placeholders in the SNK-k template still interpolate the leaf values per SLSH-4; the verbatim assertion targets the surrounding template text. The spec ([`slash-invocation.md`](../spec_topics/slash-invocation.md) SLSH-4) is correct as written and is read-only for this fix — the defect is solely in the plan's test-bullet framing. The `coverage-matrix.md` SNK-k closure attribution ("carrying the catch-all totality property over the `QueryError` union") describes the spec-real property and needs no edit.

## Consequence

**Severity:** advisory

An implementer reading the SNK-k bullet may discharge the totality clause with a single synthetic-`kind` render assertion, leaving the renderer's structural completeness (a defined output for *any* unmatched `kind`) unverified. A working leaf is still producible, but the gate is weaker than its prose implies: a partial dispatch that falls through for the one tested value yet not for others would pass.
