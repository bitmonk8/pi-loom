## Problem

The `loom/parse/unsupported-feature` registry row in `code-registry-parse.md` describes its trigger as an open-ended construct set (signalled by "etc."), but the only rendering rule for its `<construct>` placeholder is §3 of `placeholder-rendering-a.md` — a closed, GOV-7/GOV-8-governed token-name table of JS-syntax constructs. Emitting sites outside that table can fire this code: the schema-subset reject gate rejects by default (an inherently open keyword universe) and the reserved hex/octal/binary/underscore numeric-literal rules emit forms the table does not list. For those constructs `<construct>` has no defined rendering, so two conformant implementations produce different diagnostic strings for the same source defect, breaking the GOV-15 byte-identical-message guarantee.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 839db71 — pi-loom spec: resolve "Diagnostic codes are declared as namespaces but never assigned to specific errors" (2026-05-04, Thomas Andersen); 61ea1a5 — pi-loom spec: resolve "Diagnostic message placeholders have no defined rendering" (2026-05-06, Thomas Andersen)
**History:** Commit 839db71 introduced the `loom/parse/unsupported-feature` registry row with the open-ended "etc." trigger description and the `unsupported syntactic feature: <construct>` template. Two days later, 61ea1a5 added the placeholder-rendering subsection that pinned `<construct>` to a *closed* token-name table (further hardened to a build-time-enforced GOV-7/GOV-8 surface in f930818, 2026-05-07). The defect is the unreconciled interaction: the open trigger phrasing was never narrowed when the rendering surface was closed. The 2026-06-04 topic-split commit (f5e89f4) merely relocated both halves into `code-registry-parse.md` and `placeholder-rendering-a.md` without changing them.

## Solution approach

Reconcile the open trigger universe with the closed rendering surface so `<construct>` rendering is total across every emitting site. In `code-registry-parse.md`, rewrite the `loom/parse/unsupported-feature` trigger description so it no longer implies an enumerable construct set, instead pointing at the emitting sites that own the construct universe and at `placeholder-rendering-a.md` §3 for the rendered token. In §3, add a deterministic fallback for emitting sites outside the closed token-name table — rendering the offending construct verbatim from source, consistent with the existing `<expr>` verbatim-span rule in the same category.

## Solution constraints

- Do not resolve this by enumerating the rejected-construct set: the schema-subset reject gate's reject-by-default semantics make that universe inherently open, so enumeration would silently drop cases.
- The §3 fallback must preserve the closed-surface GOV-7/GOV-8 posture of the rendering rule (a single deterministic rule, not an open-ended exception).
