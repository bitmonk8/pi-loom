## Finding

The step-2 editorial-review checklist in `version-bump-step2.md` partitions its 43 audit items into a MUST tier (items (a)–(e), whose detection routes to mandatory editorial review) and a SHOULD tier (items (f)–(aq), 38 items audited "per each item's SHOULD-level scope and escalation clause"). Within the SHOULD tier there is no further ranking: every item is a flat peer bullet, each closing with the identical "SHOULD-level only at loom 1.0 … escalates to a build-time pin" boilerplate, and the recording convention explicitly permits a contributor to skip any SHOULD item by recording `not-performed` with a one-line rationale.

The SHOULD items span a wide stakes range, but the checklist surfaces no signal of that spread at the tier level. Some encode correctness- or privacy-adjacent guarantees — e.g. (i) provider overflow-signature wording (a silent rewording downgrades a real overflow to `TransportError` with null token fields), (aj) `SessionManager.inMemory` transcript non-retention (a regression is a transcript leak), (ac)/(ak)/(al) the prompt-mode error-detection and `Ok(string)` extraction ordering preconditions. Others are largely observational drift-detectors — e.g. (ae) per-extension privilege-scoping facet detection, (ao) `pi.registerFlag('loom', …)` namespace behaviour, (o)-style satellite re-validations that loom does not even consume. A contributor who must skip a subset under time pressure has no guidance distinguishing the high-stakes items from the conformance-only ones; the flat presentation invites skipping a transcript-leak or overflow-misclassification check on the same footing as a no-op drift probe.

The defect is the absence of any intra-tier prioritization signal — not the SHOULD level itself, which is appropriate for editorial audits that cannot yet be mechanically gated.

## Solution Space

**Shape:** multiple

### Option A — Per-item risk-tier labels

- **Approach:** Assign each of items (f)–(aq) a risk-tier label drawn from a fixed vocabulary (e.g. `correctness` / `security-observability` / `conformance-only`, or P0/P1/P2), either as an inline tag on each bullet or by grouping the items under tier sub-headings within the checklist.
- **Spec edits:** Add a tier label to each (f)–(aq) bullet in `version-bump-step2.md`, and a one-sentence definition of the tier vocabulary in the step-2 preamble. Optionally reorder/group the bullets under tier sub-headings (note: reordering churns the stable item letters (f)–(aq) that other items and the recording conventions cite by anchor, so prefer inline tags over physical regrouping).
- **Pros:** Complete, per-item guidance; a contributor sees the stakes of every item without cross-referencing prose.
- **Cons:** 38 individual edits; introduces a tier taxonomy the spec must then keep self-consistent; risk of churning cited item letters if bullets are physically regrouped.
- **Risks:** Tier assignments become a new bikeshed surface and a new drift target on each bump.

### Option B — Highest-stakes callout in the step-2 preamble

- **Approach:** Leave items (f)–(aq) as flat peers but add a single preamble sentence naming the highest-stakes SHOULD items a time-constrained contributor SHOULD prioritize when skipping any, citing them by their stable item letters.
- **Spec edits:** Add one sentence to the step-2 preamble (alongside the existing MUST/SHOULD distinction) enumerating the prioritized item letters — e.g. the correctness/privacy-adjacent set (i), (ac), (aj), (ak), (al) — with a one-clause rationale that these guard against silent correctness or privacy regressions.
- **Pros:** Single-location, low-churn edit proportionate to an advisory-severity SHOULD checklist; touches no item letters or anchors; gives a contributor an actionable skip-priority signal.
- **Cons:** Coarser than per-item tiering; the prioritized set must be maintained as new SHOULD items are added.
- **Risks:** The callout list can go stale if a future high-stakes item is added without updating the preamble (mitigable by the existing "any future presupposition … MUST be added to this checklist in the same edit" rule, which can be extended to cover the callout).

### Recommendation

Adopt **Option B**. It resolves the prioritization gap with a single proportionate edit, avoids the maintenance and anchor-churn cost of tagging 38 items, and matches the finding's own minimum bar (name the highest-stakes SHOULD items in the preamble). The implementer should fold the callout's maintenance into the existing same-edit obligation that already governs adding new checklist items, so a newly-added high-stakes item updates the callout in the same commit.

## Consequence

**Severity:** advisory

A time-constrained contributor performing a Pi minor bump, allowed to record `not-performed` on any SHOULD item, has no basis for choosing which to skip and may drop a correctness- or privacy-adjacent audit (e.g. overflow-wording or in-memory-transcript non-retention) as readily as a purely observational one. The procedure still yields a working system — full coverage remains available and the per-item bodies describe individual stakes — so the gap is a prioritization-guidance deficiency rather than a contradiction or a missing rule.
