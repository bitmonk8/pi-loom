## Finding

The single-string bypass rule (`binder-bypass-and-envelope.md`, item 2 under `## Binder bypass`) gates the bypass on three local conditions — exactly one declared field, type is `string`, no default — and then asserts that post-bypass AJV validation "still runs as a safety net (a string passes by definition; this is just the standard validation path)." The "passes by definition" claim is only true because loom 1.0's schema subset (`schema-subset.md`, the *Explicitly not supported* enumeration) rejects every string-constraining keyword — `pattern`, `format`, `minLength`, `maxLength` — at parse time, so the lowered fragment for a `string` field cannot carry a constraint that the safety-net AJV pass could fail on. That dependency on the subset is what makes the bypass sound; the bypass paragraph does not cite it.

The result is a silent coupling. A reader of the bypass paragraph in isolation has no way to verify the "passes by definition" assertion, and a future widening of the schema subset that admitted any constrained `string` form (a `pattern`, a `maxLength`, an enum-of-strings refinement) would silently invalidate the bypass without any in-paragraph signal that the bypass language needs revisiting. No failure surface is defined for "bypass-eligible loom skips the binder, then the safety-net AJV rejects the raw slash-argument string" because, in loom 1.0, that combination is unreachable — but only because of a constraint pinned in a different file.

The schema-subset rejections (`pattern`, `format`, `minLength`, `maxLength`, …) that *make* the claim true were already pinned in `schema-subset.md` when the bypass was added, so the silent dependency has existed in the corpus from the bypass's first commit.

## Solution Space

**Shape:** single

### Recommendation

In `binder-bypass-and-envelope.md`, item 2 of `## Binder bypass`, replace the parenthetical `(a string passes by definition; this is just the standard validation path)` with a parenthetical that names the subset dependency explicitly, citing the *Explicitly not supported* enumeration in `schema-subset.md` by anchor. A conforming rendering:

> AJV validation still runs as a safety net. The validation is vacuous for a loom 1.0 `string` field — the [Schema Subset](../schema-subset.md) rejects `pattern`, `format`, `minLength`, and `maxLength` at parse time, so no string-constraining keyword can reach the lowered fragment AJV checks against — but the call is the same standard validation path every binder result takes, kept here so the bypass and non-bypass paths share one post-binding surface.

Anchor / wording can vary; the contract is (1) the link to `schema-subset.md`, (2) the four rejected keywords named so a grep over either file finds the other, and (3) keeping the "same validation path" language so the post-bypass and post-binder surfaces remain identified.

No new failure-mode template row is needed: the constrained-`string`-after-bypass case is unreachable under the loom 1.0 subset, and any later subset widening that made it reachable is the correct trigger for revisiting the bypass design, not for adding a dead template row now. The implementer of `V11c` should additionally cross-check that the leaf's *Adds* line ("the binder bypass path … skips the LLM call") still reads coherently once the paragraph cites `V5d`'s territory.

## Consequence

**Severity:** advisory

In loom 1.0 nothing breaks: the subset reject gate (`V5d`) makes constrained `string` shapes unreachable, so the "passes by definition" claim holds. Two reasonable implementers will not diverge today. The cost is a hidden cross-file coupling: a later subset relaxation could silently desynchronise the bypass paragraph from its premise without any local cue that the bypass language must be re-examined.
