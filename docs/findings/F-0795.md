## Problem

The SNK-a row of the SLSH-4 per-`kind` table renders `<n>` into the system note `loom /<name> returned Err: model failed schema after <n> respond-repair attempts`, but never names the source field that supplies it. SLSH-4 marks the templates normative and explicitly conformance-assertable, yet three plausible bindings each render a different integer for the same failure: `ValidationError.attempts`, the total provider calls (one initial turn plus the follow-ups), or the configured `respond_repair.attempts` ceiling — most visibly `0` vs `1` on the `respond_repair.methodology: none` path. SNK-a is the only SLSH-4 cell that uses `<n>`, and unlike SNK-h's `<last_tool_name>` no post-table prose binds it. Two reasonable renderers can therefore disagree on a string a conformance test is allowed to assert against.

## Issue introduction

**Verdict:** present-since-inception

**Introducing commit:** `53f5831` ("spec: rewrite query primitive, system prompts, and schema syntax", 2026-05-03) introduced the per-`kind` table in the monolithic `spec.md`, including the row `| validation | "loom /<name> returned Err: model failed schema after <n> coercion attempts" |`. The `<n>` placeholder appeared without a binding from inception.

**History:** The defect has been carried unchanged through every subsequent edit of this row: `fecb504` (2026-05-04, split spec into per-topic files) preserved the unbound `<n>`; `708a42f` (2026-05-06, "coercion overloaded") renamed "coercion attempts" → "respond-repair attempts" but did not bind the placeholder; `98ba4fe` split the row into the `schema_validation` / `empty_template` pair (the empty-template row dropped `<n>` entirely; SNK-a kept it, still unbound); `27b2576` introduced the `SNK-a`…`SNK-k` inline-label scheme and the post-table explanatory paragraph that did pin `<last_tool_name>` for SNK-h — but did not pin `<n>` for SNK-a. The most recent table touch (`40b1f17`, "SLSH-4/SLSH-5 inline-backtick emission") added a paragraph governing backtick stripping in templates and again did not touch `<n>`. No commit has ever bound the placeholder.

## Solution approach

Clarify the explanatory paragraph immediately below the SLSH-4 table (the paragraph that already pins `<last_tool_name>` for SNK-h) to bind SNK-a's `<n>` to `ValidationError.attempts`, with a forward-link to `errors-and-results/queryerror-variants.md#query-time-variants`. The defect is the unstated binding, so name the field: `<n>` is the respond-repair follow-up count carried on the surfaced `ValidationError`, rendered as the field's non-negative integer, including the `0` rendering on the `respond_repair.methodology: none` / `attempts: 0` path.

## Solution constraints

- Out of scope: the counting-rule semantics of `ValidationError.attempts` — bind `<n>` to the field by forward-link only; the field's counting rule is owned by the same-cluster query-failure-and-repair finding.
