## Finding

The SNK-a row of the SLSH-4 per-`kind` table renders the system note `loom /<name> returned Err: model failed schema after <n> respond-repair attempts`. SLSH-4's umbrella MUST declares the templates normative — wording changes are "spec-versioned breaking changes" and conformance tests "MAY assert on the exact rendered string" — yet the source field that supplies `<n>` is never named. The row is the only SLSH-4 cell that uses the `<n>` placeholder, and no nearby prose, no entry in the SLSH-5 worked examples, and no cross-reference into `errors-and-results/queryerror-variants.md` binds it.

Three plausible bindings each render a different integer for the same failure: the count of follow-up respond-repair turns that produced a re-validated response (the `attempts` field on `ValidationError`, per `queryerror-variants.md`), the total provider calls on the typed query (one initial turn plus the `attempts` follow-ups), or the configured ceiling `respond_repair.attempts` from frontmatter. Edge cases sharpen the divergence: with `respond_repair.methodology: none` (or `attempts: 0`) the three bindings render `0`, `1`, and `0` respectively; on the wrong-tool branch where `ValidationError.attempts: respond_repair.attempts` after exhaustion, the "total provider calls" binding renders one higher than the field. SLSH-4's MUST then forces two reasonable renderer implementations to disagree on a string a conformance test is explicitly allowed to assert against.

The peer placeholder `<last_tool_name>` in SNK-h carries explicit post-table prose binding it to the source field and pinning its `null` rendering; `<n>` has no such treatment.

## Issue introduction

**Verdict:** present-since-inception

**Introducing commit:** `53f5831` ("spec: rewrite query primitive, system prompts, and schema syntax", 2026-05-03) introduced the per-`kind` table in the monolithic `spec.md`, including the row `| validation | "loom /<name> returned Err: model failed schema after <n> coercion attempts" |`. The `<n>` placeholder appeared without a binding from inception.

**History:** The defect has been carried unchanged through every subsequent edit of this row: `fecb504` (2026-05-04, split spec into per-topic files) preserved the unbound `<n>`; `708a42f` (2026-05-06, "coercion overloaded") renamed "coercion attempts" → "respond-repair attempts" but did not bind the placeholder; `98ba4fe` split the row into the `schema_validation` / `empty_template` pair (the empty-template row dropped `<n>` entirely; SNK-a kept it, still unbound); `27b2576` introduced the `SNK-a`…`SNK-k` inline-label scheme and the post-table explanatory paragraph that did pin `<last_tool_name>` for SNK-h — but did not pin `<n>` for SNK-a. The most recent table touch (`40b1f17`, "SLSH-4/SLSH-5 inline-backtick emission") added a paragraph governing backtick stripping in templates and again did not touch `<n>`. No commit has ever bound the placeholder.

## Solution Space

**Shape:** single

### Recommendation

Bind `<n>` in the explanatory paragraph that already lives immediately below the SLSH-4 table (the same paragraph that pins `<last_tool_name>` for SNK-h). Add a sentence of the form:

> For `SNK-a`, `<n>` is rendered as the integer value of [`ValidationError.attempts`](./errors-and-results/queryerror-variants.md#query-time-variants) on the rendered `QueryError`. Per that field's contract, `<n>` counts the respond-repair follow-up turns that produced a re-validated assistant response — not the total provider calls on the typed query, and not the configured `respond_repair.attempts` ceiling. The integer is rendered with no thousands separator, no leading zeros, and no sign (`attempts` is non-negative). It is `0` on the `respond_repair.methodology: none` (or `attempts: 0`) path, in which case SNK-a renders the literal string `model failed schema after 0 respond-repair attempts` verbatim; this is the intended behaviour and conformance tests SHOULD cover it.

Pick `ValidationError.attempts` (not "total provider calls", not "configured ceiling") because it is the only one of the three candidates that is already a normative field on the surfaced error, whose semantics are already pinned in `queryerror-variants.md`, and whose value matches what the user can independently observe by inspecting the `RuntimeEvent` payload carried on the same `loom-system-note` (per `pi-integration-contract/runtime-event-channel.md`, the `attempts?` field on the always-log event is the same `ValidationError.attempts`). The other two candidates would force a derivation in the renderer that has no other consumer.

Edge cases the implementer must watch:

- The `0` rendering is reachable through three distinct paths — `respond_repair.methodology: none`, explicit `attempts: 0`, and the wrong-tool branch's first forced respond turn with `methodology: none` per `queryerror-variants.md` — and renders identically in all three.
- The empty-template short-circuit (SNK-b, `cause: "empty_template"`) fires with `attempts: 0` but does not use the `<n>` placeholder at all; the binding above is SNK-a-only and does not need an SNK-b carve-out.
- The conformance test on the `0` case is the cheapest way to catch a renderer that off-by-ones into "total provider calls" semantics.

## Consequence

**Severity:** correctness

Two reasonable implementers will disagree on the integer rendered into the SNK-a note. Because SLSH-4 marks the surrounding template normative and conformance-assertable, the disagreement surfaces as test failure for whichever renderer guessed the unstated binding — without any spec text to settle the dispute. The case where `respond_repair.methodology: none` makes the disagreement most visible (`0` vs `1`).
