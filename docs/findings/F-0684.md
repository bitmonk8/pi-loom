## Finding

DISC-2's failure-modes table in `discovery-sources.md` has a column headed `Path is wrong type (file vs dir)`. The parenthetical reads as "the path is a file where a directory was expected (or vice versa)" — i.e. a mismatch between what the source supplies and what the source expects.

That is not the rule. Per DISC-2's third implementation-notes bullet, "[a] `--loom` flag (or settings entry) pointing at a directory is allowed and treated like a per-source root; the wrong-type rule fires only when the path exists but is neither a `.loom` file nor a directory containing them." Both files and directories are valid for every source row; the rule fires for a *third* kind of path (a socket, a device node, a symlink target that resolves to neither, a regular non-`.loom` file). The column header steers a reader of the table — especially one skimming without dropping into the bullets — toward the wrong invariant.

The diagnostic code (`loom/load/wrong-type-source`) and the implementation notes are correct; only the table's column header is misleading.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 5b772b6 — pi-loom spec: resolve "Discovery source failure-mode policy is unspecified" (2026-05-04, Thomas Andersen)
**History:** The failure-modes table — including the `Path is wrong type (file vs dir)` column — was introduced in 5b772b6, the commit that first specified DISC-2. The same commit added the corrective implementation-notes bullet ("the wrong-type rule fires only when the path exists but is neither a `.loom` file nor a directory containing them"), so the table header and the rule it summarises have been inconsistent since the rule's first appearance. The split commit f5e89f4 (file relocation to `discovery/discovery-sources.md`) carried the text verbatim and did not perturb the wording.

## Solution Space

**Shape:** single

### Recommendation

Rename the column header from `Path is wrong type (file vs dir)` to `Path is wrong type (not a .loom file or directory)`. No other text in the table or surrounding prose changes; the diagnostic code (`loom/load/wrong-type-source`), the implementation-notes bullet, and the plan leaves continue to describe the same rule.

Implementer notes:
- The column applies to every row identically; do not introduce a per-row footnote.
- The rule's anchor for "directory" is "a directory containing `.loom` files" per the implementation notes — an empty directory still counts as a directory and does not trip wrong-type (it is a normal silent-zero-looms outcome for conventional sources, or a successful zero-looms enumeration for explicit references).

## Consequence

**Severity:** cosmetic

An implementer relying on the column header alone might assert wrong-type only when the *kind* mismatches the source's expectation (e.g. fail a `--loom` flag pointing at a directory). The bullet below the table and the diagnostic-code name correct this on a careful read, so divergent implementations are unlikely but not impossible.
