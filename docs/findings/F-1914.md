## Finding

`V5d` (Schema-subset reject gate) declares `Deps.` `V5d-T`, `V5a`, `V5b`, `V2d`, and its test partner `V5d-T` declares `V5a`, `V5b`, `V2d`. The leaf's entire observable surface is the allowlist reject gate: it fires `loom/parse/unsupported-feature` for constructs outside the permitted subset, fires `loom/parse/result-in-schema-position` for a `Result` in a schema-feeding position, and accepts the permitted subset. None of that behaviour renders numbers, so it never composes the `V2d` canonical integer/number renderer.

`V2d` is consumed only where the canonical-hash recipe runs — and that recipe was extracted into `V5f` (Schema lowering and canonical hash). `V5f` correctly declares `V2d` and is the leaf that supplies the renderer's `integer`-vs-`number` kind discriminator from the JSON-Schema `type` of the enclosing `const`/`enum` position. The `V2d` edge on `V5d`/`V5d-T` is a residue from when `V5d` still owned lowering and hashing; it now over-sequences `V5d` behind `V2d` without any behavioural justification. The reject gate validates by rejecting numeric-constraint keywords (`min*`/`max*`), not by rendering numbers, so there is no literal-validation reason to retain the edge.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commit:** `cb8737a` — "pi-loom plan: resolve \"Systemic leaf over-bundling across the plan corpus\"" (2026-06-12)
**History:** `git log -S 'V2d' --follow` on the leaf file (tracked under its earlier name `docs/plan_topics/V5d-subset-lowering.md`, renamed to `V5d-reject-gate.md` in `09293d2`) shows `V2d` predates the reject-gate split. Before `cb8737a`, `V5d` was titled "Schema-subset gate, lowering, and canonical hash" and its `Adds.` owned the lowering pass and the canonical-hash recipe — the binder-number-rendered canonical form genuinely composed `V2d`, so the `V2d` `Deps.` edge was justified. `git show cb8737a -- docs/plan_topics/V5d-subset-lowering.md` shows that commit moved the lowering pass and canonical-hash recipe out to `V5f` (de-bundling `V5d` to the reject gate only) but left the `V2d` token in the unchanged `**Deps.** V5d-T, V5a, V5b, V2d` line. The edge was orphaned at that moment. The later rename commit `09293d2` and subsequent edits carried the stale edge through unchanged.

## Solution Space

**Shape:** single

### Recommendation

Strike `V2d` from the `Deps.` line of both `docs/plan_topics/V5d-reject-gate.md` and `docs/plan_topics/V5d-T-reject-gate.md`:

- `V5d-reject-gate.md` — `**Deps.** \`V5d-T\`, \`V5a\`, \`V5b\``
- `V5d-T-reject-gate.md` — `**Deps.** \`V5a\`, \`V5b\``

Leave `V5f`/`V5f-T`'s `V2d` deps untouched — that is where the canonical-hash recipe consumes the renderer. Do not touch any spec file; this fix is internal to the two plan leaf files.

## Consequence

**Severity:** cosmetic

`V5d` carries a dependency edge on the canonical number renderer that its reject-gate behaviour never exercises, falsely sequencing `V5d` behind `V2d` and misrepresenting the build DAG. Execution is not affected — `V5a`/`V5b` are real deps that already precede it and `V2d` builds early regardless — so no implementer is misled into wrong behaviour; it is dependency-graph hygiene.
