## Problem

The "Registered loom (subagent-mode)" row of the *Return type* table in `tool-calls.md` ends with "Otherwise the runtime AJV check enforces it." The pronoun "it" has no clear antecedent — what the runtime validates is the callee's returned value against the inferred return type `T`, not "its inferred return type" or "the call site". More seriously, the "Otherwise …" branch was copied from `invoke<T>(...)`, where a not-statically-resolvable callee leaves a genuine runtime safety net; but per `invocation.md` *Static resolution*, a `tools:` `.loom` entry whose callee is not statically resolvable is a load-time error and the parent loom does not register, so a successfully registered `.loom` callable's callee is always statically resolvable. The branch describes a state that cannot occur for the row it annotates, leaving implementers to diverge over whether to wire a dead "only-when-unresolvable" validation path.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** `7cd9cc6` (spec: unify tools and loom-as-tool callables under tools:; add ToolCallError — 2026-05-04), `333ba2f` (spec: define static resolution for invoke/tools callees (Option C) — 2026-05-04)
**History:** `git log -S` traces the exact string "Otherwise the runtime AJV check enforces it" to `7cd9cc6`, which added the "Registered loom (subagent-mode)" return-type row to `spec.md`. At that point the clause was a faithful copy of the pre-existing `invoke<T>(...)` safety-net phrasing (same commit's *Loom return type* text: "otherwise the runtime AJV check is the safety net") and was internally consistent, because static-resolution failure modes were not yet differentiated. Later the same day `333ba2f` introduced the *Static resolution* section with the deliberate severity asymmetry — a `tools:` `.loom` unresolvable callee is an **error** (parent does not register), whereas `invoke(...)` is a **warning** (parent registers, AJV safety net applies). That asymmetry retroactively invalidated the copied fallback for the registered-loom row without the row being updated. The defect is therefore the interaction of the two commits, not a fault in either alone; the file later moved `spec.md` → `spec_topics/tool-calls.md` (split `fecb504`) → `docs/spec_topics/tool-calls.md` (`81bf52a`, `31ff060`) carrying the clause unchanged.

## Solution approach

Rewrite the second sentence of the "Registered loom (subagent-mode)" row so it names what the runtime validates — the callee's returned value against the inferred return type `T` — and reconciles with the static-resolution asymmetry instead of carrying over `invoke<T>(...)`'s safety-net branch. A registered `.loom` callable is always statically resolvable (a `tools:` `.loom` entry whose callee is not statically resolvable is a load-time error and the parent does not register, per [Invocation — Static resolution](./invocation.md#static-resolution)), so `T` always flows into the call site and the runtime validates on every call. Keep the cross-reference to `invoke<T>(...)` for the inference rule but do not carry over its not-statically-resolvable safety-net branch.
