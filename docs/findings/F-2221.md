## Problem

The "Canonical schema hash" section of `schema-subset.md` declares its recipe part of the on-disk and on-wire contract, pinned exactly, and specifies three independently-failable steps: canonical UTF-8 serialisation, SHA-256, and truncation to the first 16 lowercased hex characters (the schema slug). For a byte-exact contract it ships no worked reference vector tying a concrete lowered fragment to its expected canonical bytes and resulting slug; the only slug shown, `__inline_3f9a1c2b8d4e5076`, is an illustrative name shape with no input behind it. The adjacent vectors that do exist validate other stages (SUBS-1 source→lowered-fragment renderings, BNDR-6 echo value renderings), never the serialise→digest→truncate pipeline as one unit. Two conforming implementations can therefore satisfy every prose rule yet emit divergent slugs, and the divergence stays undetectable until slug-keyed caches, fixture snapshots, or replay payloads fail to interoperate.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `e21449b` ("pi-loom spec: resolve \"__inline_<hash> hash function is not pinned\"", 2026-05-04)
**History:** The "Canonical schema hash" section was created in `e21449b`, which replaced the earlier under-specified structural hash with the pinned recipe. That same commit introduced the "on-disk and on-wire contract … so the V1 spec pins it exactly" framing and all three failable steps (canonical form, SHA-256, 16-hex truncation) — and shipped them with no end-to-end reference vector; the only slug shown (`__inline_3f9a1c2b8d4e5076`) was already a free-standing illustrative name. The defect is thus contemporaneous with the recipe's introduction, not a later regression. Subsequent commits tightened individual rules without ever adding a fragment→bytes→slug vector: `c5bb5e1` pinned numeric serialisation, `7321f01` pinned array element order, and `6892aa3` (SUBS-1) added source→lowered-fragment vectors only. `git log -S "precomputed-value"` and `-S "MUST produce canonical bytes"` across all history return nothing, confirming such a vector has never been present in the corpus.

## Solution approach

Add a normative reference-vectors block to the "Canonical schema hash" section pinning the full `lowered fragment → canonical-form bytes → schema slug` pipeline, with precomputed 16-hex slugs that conforming implementations MUST reproduce. Choose vectors that exercise the serialiser disagreements the prose leaves implicit: object-key code-point sorting against declaration-ordered emitted properties, RFC-8259 minimal string escaping, and the borrowed BNDR-4/BNDR-5 numeric rendering applied inside a `const`/`enum` position rather than only in the echo formatter.
