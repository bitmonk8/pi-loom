## Problem

The *Outbound* (loom value → JSON) bullet of the `Wire-name translation` block in `runtime-value-model.md` enumerates three outbound sites: `tool input`, `query response payloads`, and `invoke arguments`. The `query response payloads` entry has no referent anywhere else in the corpus — the phrase appears at no other site. The likely intended referents are callee-side serialisation sites the other two entries do not cover: a `.loom`-callable return value exposed to a parent model, and the `invoke<T>` callee return value. Two implementers will diverge on whether those callee-side returns pass through outbound wire-name translation before AJV validation, silently emitting loom-side identifiers to the model when the rename is skipped.

**Verdict:** present-since-inception
**Introducing commits:** `fae85d3` (2026-05-04, "Tighten spec for implementation-plan readiness")

The phrase "query response payloads" has been present in this exact wording, in the same enumeration, since that initial introduction; no later commit has narrowed or expanded the enumerated outbound sites. The defect entered with the section itself.

## Solution approach

Rewrite the *Outbound* bullet of the `Wire-name translation` block in `runtime-value-model.md` so every enumerated outbound site resolves to a named section elsewhere in the spec. Replace the unreferenced `query response payloads` entry with the real callee-side outbound serialisation sites — the `.loom`-callable return value exposed to a parent model (`tool-calls.md`, *Registered loom (subagent-mode)*) and the `invoke<T>` callee return value (`invocation.md`). Align the enumeration with `schema-subset.md`'s per-boundary table (`#depth-enforcement`, rows #1–#5) rather than introducing a parallel taxonomy.

## Solution constraints

- Out of scope: the *Inbound* bullet and its named-enum-tag sidecar walk — owned by the same-cluster sidecar finding; edit only the *Outbound* bullet.
- The outbound enumeration must not name the typed-query forced-respond response as an outbound site — it is inbound (validated at the per-boundary table's row #1).
