## Finding

The `Wire-name translation` block in `docs/spec_topics/runtime-value-model.md` enumerates the *outbound* (loom value → JSON) sites as exactly three: "tool input," "query response payloads," and "`invoke` arguments." Two of the three name surfaces a reader can locate elsewhere in the spec (tool-call args at `tool-calls.md`, `invoke` args at `invocation.md`), but **"query response payloads" has no referent anywhere else in the corpus** — the phrase does not appear in `query.md`, `query/`, `schema-subset.md`, or any other topic file. A grep across `docs/` finds the literal string only at this single site.

The likely intended referent is the *callee-side* serialisation that happens when a `.loom` file is exposed to a parent model via another loom's `tools:` (per `tool-calls.md` — *Registered loom (subagent-mode)* — and the `.loom`-callable `execute` adapter pinned in `pi-integration-contract/extension-bootstrap-and-per-loom.md`). The callee's typed return value must be serialised to wire-named JSON before reaching the parent model; that is a real outbound site that the existing two enumerated entries ("tool input," "invoke arguments") do not cover — "tool input" is loom-code → tool dispatch, "`invoke` arguments" is parent → child invocation inputs, neither of which is the callee-loom → parent-model return path. The `invoke<T>` callee return value (row #5 of the `schema-subset.md` per-boundary table) likewise needs outbound translation on the callee side, and is not currently covered by either of the two unambiguous entries.

Two reasonable implementers will diverge on which sites apply outbound wire-name translation before AJV validation, in particular on whether a `.loom` callee returning a typed value to a parent model must walk and rename loom-side fields. The wire-name renaming machinery is the only mechanism between loom code and the model; mis-routing it silently passes loom-side identifiers to the model (or its strict-mode AJV decoder), producing per-loom validation drift that no diagnostic surfaces.

## Solution Space

**Shape:** single

### Recommendation

Replace the *Outbound* bullet's three-site enumeration with one whose every entry resolves to a section heading elsewhere in the spec. Concretely, rewrite:

> *Outbound* (loom value → JSON): when constructing tool input, query response payloads, or `invoke` arguments, the runtime walks the loom-side value and produces wire-named JSON before AJV validation.

as something like:

> *Outbound* (loom value → JSON): the runtime walks the loom-side value and produces wire-named JSON before AJV validation at every site where loom-declared JSON crosses outward — code-driven tool-call arguments ([Tool Calls](./tool-calls.md), AJV boundary row #3 of [Schema Subset — Depth Enforcement per-boundary table](./schema-subset.md#depth-enforcement)), `invoke` arguments ([Invocation](./invocation.md)), `invoke<T>` callee return values ([Invocation — Failures](./invocation.md), AJV boundary row #5), and the return value of a `.loom`-callable exposed to a parent model via another loom's `tools:` ([Tool Calls — Registered loom (subagent-mode)](./tool-calls.md) and the `.loom`-callable `execute` adapter in [Pi Integration Contract — Per-loom registration](./pi-integration-contract/extension-bootstrap-and-per-loom.md)).

Implementer-relevant edge cases the rewrite must keep explicit:

- The typed-query forced respond turn is **not** an outbound site — the model's `__loom_respond_<slug>` arguments travel model → loom (inbound), validated by row #1 of the per-boundary table. The rewrite must avoid re-creating the original ambiguity by naming a typed-query response as outbound.
- Frontmatter `params:` defaults remain in the existing carry-out paragraph ("Frontmatter `params:` defaults bypass the inbound translation pass…"); they are not an outbound site and need not be re-mentioned.
- The cross-reference to `schema-subset.md`'s per-boundary table is the natural authority for which AJV sites exist; the *Outbound* bullet should align its enumeration with that table rather than introduce a parallel taxonomy.

## Consequence

**Severity:** correctness

Implementers will disagree on whether `.loom`-callable return values and `invoke<T>` return values pass through the outbound wire-name translation pass before AJV validates them on the receiving side. A divergent implementation that omits the rename silently emits loom-side identifiers to the parent model (or AJV), producing per-loom-callable validation failures whose root cause is invisible to authors who declared a `field as "WireName": T` rename and reasonably expected it to apply on every boundary.
