## Problem

The binder inference call's `context.messages` bullet in `binder-inference.md` relies on every loom-1.0-supported provider accepting a `complete()` request whose message array is exactly one fixed `user` message alongside the system prompt, but states this only as inline rationale ("to satisfy providers that reject a system-only request with no non-system message"). Unlike the two sibling `complete()` presuppositions on the same call â€” `#complete-forced-tool-presupposition` and `#complete-retry-and-cancellation-presupposition` in `conversation-drive.md` â€” it is pinned by no named presupposition and routed to no Pi version-bump re-validation item. No SDK surface pins this provider behaviour, so a future provider or SDK change could silently break the binder with no detection path.

## Solution approach

Add a named, anchored presupposition asserting that each loom-1.0-supported provider accepts a `complete()` request whose `context.messages` is a single fixed non-system (`user`) message accompanying the system prompt, alongside the existing `complete()` presuppositions in `conversation-drive.md`. Add a forward-link to it from the `context.messages` bullet in `binder-inference.md`. Add a Pi version-bump editorial-review checklist item routing its re-validation in `version-bump-step2.md`, mirroring items (u) and (ab).
