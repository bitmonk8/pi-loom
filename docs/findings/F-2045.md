## Finding

The hard-ceiling-#4 per-boundary destination/surface table exists as two near-identical normative copies on two different topic pages:

- `hard-ceilings/ceilings-3-and-4.md` (`#ceiling-4-table`) — the **canonical owner**, declared as such by the orientation aggregator at `overview-and-orientation.md#hard-runtime-ceilings` ("Per-boundary destination/surface table: [Hard Runtime Ceilings — Per-boundary destination/surface table (ceiling #4)](./hard-ceilings/ceilings-3-and-4.md#ceiling-4-table)").
- `schema-subset.md` ("Depth Enforcement" → the "Per-boundary surfaces" table, rows #1–#5).

Both tables carry the same five enforcement points (typed-query response, model-driven tool-call args, code-driven tool-call args, `params` validation, `invoke<T>` return value), the same Destination/Surface column structure, and the same boundary-dependent routing dispositions. The two currently agree; the duplication is content the corpus must keep synchronised, not a one-line cross-reference.

There is no normative lock-step pin binding the two copies. The only acknowledgement of the relationship is a single trailing sentence on `schema-subset.md`: *"The full per-boundary routing-class table for ceiling #4 … lives in [`spec.md` — Hard ceilings](./overview-and-orientation.md#hard-runtime-ceilings); this section's table mirrors it with the depth-walk-specific carrier details."* That sentence is defective in two ways. First, it points at the orientation aggregator (`overview-and-orientation.md#hard-runtime-ceilings`), which holds only a one-liner forward-link, **not** the full table — the canonical table lives at `ceilings-3-and-4.md#ceiling-4-table`. Second, it is a one-directional prose aside, not a lock-step obligation: the canonical owner page makes no reciprocal mention of the `schema-subset.md` mirror, and the corpus's lock-step convention (GOV-30) governs only `spec.md`-aggregator-vs-source parity, not source-page-to-source-page duplication. Consequently a future edit to either table — for example a change to the slash-load `params` cross-routing wording or a carrier-shape revision — has no normative trigger requiring the sibling copy to be updated in the same commit, and the two can silently drift.

## Solution Space

**Shape:** multiple

### Option A — Collapse the `schema-subset.md` table to a cross-reference

**Approach:** Delete the duplicated per-boundary table from `schema-subset.md` "Depth Enforcement" and replace it with a single cross-reference to the canonical `ceilings-3-and-4.md#ceiling-4-table`, retaining inline only the depth-walk-specific carrier deltas (the `path` JSON-Pointer field, the `ModelToolError`-exclusion note, the respond-repair applicability note) that the canonical table omits.

**Spec edits:** Remove the rows-#1–#5 table; keep the "Error shape" prose; add a sentence directing the reader to the canonical table for routing and stating which depth-walk-specific carrier details `schema-subset.md` adds.

**Pros:** Eliminates the duplication outright, so there is nothing to keep in lock-step; removes the misdirecting "mirrors it" pointer.

**Cons:** `schema-subset.md` is the depth-enforcement owner; readers there lose the at-a-glance per-boundary view and must hop to the hard-ceilings page. The depth-walk-specific carrier details still have to live somewhere, so some carrier prose must be re-homed.

**Risks:** Re-homing the depth-walk-specific carrier deltas risks dropping a detail (e.g. the `path` field, the `ModelToolError` exclusion) that currently only appears in the `schema-subset.md` rows.

### Option B — Keep both copies; declare an explicit lock-step and fix the pointer

**Approach:** Treat `schema-subset.md`'s table as a deliberate depth-walk-specific mirror of the canonical `#ceiling-4-table`, and make the relationship normative. Repoint the trailing sentence's cross-link from `overview-and-orientation.md#hard-runtime-ceilings` to the canonical owner `ceilings-3-and-4.md#ceiling-4-table`; add a reciprocal note on the canonical owner naming `schema-subset.md`'s table as the depth-walk mirror; and register the pair under a lock-step obligation (a sibling to GOV-30's aggregator-vs-source convention, or an explicit extension of it) so that a PR editing either table MUST update the other in the same commit.

**Spec edits:** (1) `schema-subset.md` trailing sentence: change the link target to `ceilings-3-and-4.md#ceiling-4-table`. (2) `ceilings-3-and-4.md` `#ceiling-4-table`: add one sentence pointing at the `schema-subset.md` depth-walk mirror and asserting lock-step. (3) Governance: add/extend a lock-step rule covering this source-to-source mirror.

**Pros:** Preserves the at-a-glance per-boundary view on the depth-enforcement page where readers expect it; retains the depth-walk-specific carrier details in place; uses the existing owner-page-plus-mirror pattern the corpus already relies on.

**Cons:** Two normative copies remain; the lock-step is reviewer-enforced (GOV-30-style conventions reserve no mechanical witness), so drift is prevented by discipline, not by a gate.

**Risks:** Depends on how the GOV-30 "`spec.md` aggregator" binding is anchored (see Related Findings); the new lock-step rule must name the canonical owner page precisely so it does not inherit GOV-30's existing mis-pointing.

### Recommendation

Adopt **Option B**. The `schema-subset.md` table is not a redundant copy — it carries depth-walk-specific carrier detail (the `path` JSON-Pointer field, the `ModelToolError` exclusion, respond-repair applicability) that the canonical table omits, so the corpus deliberately maintains two views of the same routing surface. The missing pieces are a correct canonical pointer and a declared lock-step, matching the owner-page-plus-mirror pattern already used elsewhere. The implementer must repoint the cross-link at `ceilings-3-and-4.md#ceiling-4-table` (not the aggregator) and ensure the new lock-step rule names that canonical owner explicitly rather than re-using GOV-30's `spec.md`-relative phrasing.

## Consequence

**Severity:** advisory

No implementer diverges today — the two tables agree, so ceiling-#4 routing is unambiguous. The defect is a latent maintenance hazard: a future edit to either copy is not caught by any lock-step trigger, so the two normative tables can drift apart, and the only cross-link a reader can follow from `schema-subset.md` to the "full" table misdirects them to an aggregator that does not contain it.
