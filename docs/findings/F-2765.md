## Problem

`V6c`'s `loom/load/unresolvable-loom-path` test rejects a `tools:` `.loom` entry "whose path does not exist or is not readable" at load time â€” a filesystem probe. The *No globals, statics, singletons* convention bans direct ambient filesystem access in `src/**`; every file read must route through the `V8b` `FileSystem` seam. But `V6c`'s `Deps` (`V6c-T, V6a, V15a, V9f`) never names `V8b`, and neither leaf states that another leaf performs the probe on `V6c`'s behalf, so the diagnostic is attributed to `V6c` as if `V6c` itself touches the disk. The plan leaves the probe locus undetermined: one implementer adds a direct, un-seamed probe inside `V6c` (an ambient-access violation), another routes it through `V15a`'s static-resolution walk (which already reads `.loom` `tools:` entries via its `V8b` seam) and treats `V6c` as an observer.

## Solution approach

Locate the `.loom`-path existence/readability probe in `V15a`'s static-resolution walk, which already reads each `.loom` `tools:` entry through its `V8b` `FileSystem` seam, and reword the `loom/load/unresolvable-loom-path` bullet in `docs/plan_topics/V6c-tools-set.md` and `docs/plan_topics/V6c-T-tools-set.md` so the rejection is surfaced by `V15a`'s resolution and `V6c` observes the result. State in `V15a`'s `Adds`/`Tests` that its static-resolution pass over `.loom` `tools:` entries surfaces `loom/load/unresolvable-loom-path`. The existing `V6c â†’ V15a` and `V15a â†’ V8b` `Deps` edges carry the seam transitively, so no new `Deps` edge is required.
