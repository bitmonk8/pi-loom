## Problem

The normative overflow signatures in the `Provider error mapping` table live in inline-code spans inside GFM table cells, with each alternation pipe written `\|` (e.g. the `anthropic-messages` and `openai-completions` rows). A GFM table cell requires a literal pipe to be backslash-escaped even inside a code span, so the rendered regex reads as JS-regex alternation, but the raw Markdown source reads as a JS-regex escaped pipe matching a literal `|` character. The two readings of the same source text diverge: an implementer transcribing the raw source builds a classifier that never matches any real provider overflow message, silently downgrading every `ContextOverflowError` to `TransportError` with null token counts. The operative meaning of a normative regex is reader-dependent because Markdown's escaping convention and the target language's escaping convention collide on the same character.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `82be125` (2026-05-04, *spec: adopt P1 typed-query two-phase loop (Option A)*) — added the provider overflow-signature table to the then-monolithic `spec_topics/pi-integration-contract.md`, with the regexes already embedded as inline-code spans inside GFM table cells using `\|` for each alternation pipe.
**History:** `git log -S 'prompt is too long'` across the tree returns `82be125` as the oldest commit carrying the signature; the table has held the `\|`-in-table-cell form since that commit. The size-cap split `f5e89f4` (2026-05) moved the table verbatim from `spec_topics/pi-integration-contract.md` into the current `provider-error-mapping.md` (cells unchanged), and `git log --follow` / `git log -G` over the file since show no commit revising the escaping (`513a5cc` and `c0de1a7` touched adjacent classifier prose — field-sourcing and precedence — not the regex spans). The ambiguity has therefore existed unbroken from the table's inception.

## Solution approach

Restructure the `Provider error mapping` overflow-signature table so each per-provider overflow regex appears outside GFM table-cell escaping — somewhere a literal `|` reads only as JS-regex alternation and no `\|` remains in any signature. Keep each provider's condition prose alongside its regex.

## Solution constraints

- Out of scope: the classification semantics — the `openai-completions` HTTP-200 `stopReason: "error"` alternative branch, the per-provider `tokens_used` / `tokens_limit` outcomes, and the cross-reference to *Overflow token-count extraction* (`#overflow-token-extraction`) must survive the restructure unchanged.
