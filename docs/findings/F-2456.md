## Finding

The `loom/load/settings-value-out-of-range` row fires once for a settings file whose JSON root parses successfully but is not an object (a top-level array, number, boolean, string, or bare `null`). In that case the row's *Message* template `settings key <key> value is out of range; got <observed>` is rendered with `<key>` set to the document-root sentinel, which both `code-registry-load.md` and `package-and-settings.md` describe as "rendered as `(root)`".

The `<key>` placeholder is a category-5 source-derived placeholder. Its rule in `placeholder-rendering-b.md` §5 is: render the key double-quoted **only when the key string is not identifier-shaped** (does not match `^[A-Za-z_][A-Za-z0-9_]*$`), otherwise render it bare; the identifier-shape predicate is a runtime check on the key string. The string `(root)` contains parentheses, so it is not identifier-shaped, and a literal application of §5 double-quotes it — producing `settings key "(root)" value is out of range; got array`.

Nothing in the corpus states whether the `(root)` sentinel is fed through §5's identifier-shape quoting or is a pre-rendered literal that bypasses §5. The phrasing "rendered as `(root)`" reads as a final form (no quotes), but §5's mechanical rule produces `"(root)"`. The two readings yield different normative message bytes. This is the only `<key>` value in the registry that is a synthesised sentinel rather than a real source key, and it is also the only `<key>` value containing non-identifier characters, so §5 was never written with it in mind. The §8 test vectors for this row deliberately assert only the `<observed>` substring (`array` / `object` / `null`) and never the full *Message* string, leaving the `<key>` half of the rendering unpinned.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 89eabf1 — pi-loom spec-plan: fix F-0744 — define settings non-object-root outcome + array/object observed rendering (2026-06-22, Thomas Andersen)
**History:** The §5 `<key>` identifier-shape quoting rule pre-dates this defect (present at the placeholder-rendering file split, commit f5e89f4, 2026-06-04, and applied only to real settings keys). Commit 89eabf1 (F-0744) introduced the `(root)` document-root sentinel into the `<key>` slot and meticulously pinned the new `<observed>` rendering for `array`/`object`/`null` roots — adding matching test vectors — but never reconciled the synthesised `(root)` value against the pre-existing §5 identifier-shape quoting rule, leaving its `<key>` rendering undefined.

## Solution Space

**Shape:** single

### Recommendation

In `placeholder-rendering-b.md` §5, carve the document-root sentinel `(root)` out of the `<key>` identifier-shape quoting predicate. State that `(root)` is a pre-rendered literal substituted directly into `<key>`, not a key string subject to the identifier-shape check; it renders bare as the seven characters `(root)` regardless of the `^[A-Za-z_][A-Za-z0-9_]*$` predicate. Add a sentence to the §5 `<key>` bullet stating this carve-out (the parentheses already mark the value as a synthesised sentinel rather than a real source key, so the §5 double-quoting does not apply to it).

Add a `settings-value-out-of-range` full-*Message* test vector in §8 covering the non-object-root case so the `<key>` half of the rendering is byte-pinned, not just the `<observed>` half — e.g. a top-level array root renders `settings key (root) value is out of range; got array`.

## Consequence

**Severity:** correctness

Two conformant implementations diverge on a byte-exact normative message (`settings key (root) ...` vs `settings key "(root)" ...`), violating the GOV-15 byte-identicality contract that the placeholder-rendering surface exists to guarantee. Conformance tests that assert the full *Message* string for the non-object-root case would split into two incompatible expected values.
