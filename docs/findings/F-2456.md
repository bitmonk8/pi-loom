## Problem

The `loom/load/settings-value-out-of-range` row renders `<key>` as the document-root sentinel `(root)` when a settings file's JSON root parses as a non-object (array, number, boolean, string, or bare `null`). `<key>` is a §5 source-derived placeholder whose rule double-quotes any key string that is not identifier-shaped per `^[A-Za-z_][A-Za-z0-9_]*$`; `(root)` contains parentheses, so a literal application of §5 yields `"(root)"`, yet both consuming registries describe the value as "rendered as `(root)`" (bare). The corpus never states whether `(root)` is fed through §5's identifier-shape check or is a pre-rendered literal that bypasses it, so two conformant implementations diverge on the byte-exact *Message*. The §8 test vectors for this row assert only the `<observed>` substring (`array` / `object` / `null`), leaving the `<key>` half of the rendering unpinned.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 89eabf1 — pi-loom spec-plan: fix F-0744 — define settings non-object-root outcome + array/object observed rendering (2026-06-22, Thomas Andersen)
**History:** The §5 `<key>` identifier-shape quoting rule pre-dates this defect (present at the placeholder-rendering file split, commit f5e89f4, 2026-06-04, and applied only to real settings keys). Commit 89eabf1 (F-0744) introduced the `(root)` document-root sentinel into the `<key>` slot and meticulously pinned the new `<observed>` rendering for `array`/`object`/`null` roots — adding matching test vectors — but never reconciled the synthesised `(root)` value against the pre-existing §5 identifier-shape quoting rule, leaving its `<key>` rendering undefined.

## Solution approach

In `placeholder-rendering-b.md` §5, clarify the `<key>` bullet so the document-root sentinel `(root)` has a defined quoting disposition — specify whether it is a pre-rendered literal that bypasses the `^[A-Za-z_][A-Za-z0-9_]*$` identifier-shape predicate (rendering bare as `(root)`, consistent with the "rendered as `(root)`" phrasing the consuming registries already use) or is subject to it. Add a §8 full-*Message* test vector for `loom/load/settings-value-out-of-range` on a non-object root that byte-pins the `<key>` half, not just the `<observed>` half — e.g. an array root rendering `settings key (root) value is out of range; got array`.
