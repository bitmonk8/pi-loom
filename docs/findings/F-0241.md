## Finding

`V3e` (Expression stdlib members) implements four independently-specified
standard-library surfaces in a single leaf: the `String.replace` reference
vectors plus `concat`, the `string` members
(`length`/`toLowerCase`/`toUpperCase`/`trim`/`startsWith`/`endsWith`/`includes`/`split`),
the `array<T>` members (`length`/`join`/`includes`/`indexOf`/`slice`), and the
`object` members (`keys`/`values`/`has`). Each surface carries its own normative
behaviour and return-type contract — they share only the `expressions.md`
`#built-in-methods-and-properties` spec section, the `V3a`/`V2c` Deps, and the
expression-evaluator substrate.

The conventions hold that each leaf inside a slice should be the smallest
feature that can ship and be tested independently (`conventions.md` §3). The
four surfaces are cohesive but not interdependent: the `string` surface could
ship and be asserted without the `array<T>`/`object` surfaces and vice versa.
The leaf already isolates each surface into its own named `Tests` bullet (in
both `V3e` and the paired `V3e-T`), so the bundling is mild — it affects leaf
size and review surface, not assertion clarity.

## Issue introduction

**Verdict:** introduced-by-commit `20b4689`, relocated by `cb8737a`
**Introducing commits:** `20b4689` (2026-06-11, "resolve \"Stdlib members beyond replace/concat have no named assertion\""); `cb8737a` (2026-06-12, "resolve \"Systemic leaf over-bundling across the plan corpus\"")
**History:** The plan corpus is git-tracked. The stdlib surface was born inside `V3a-T-expression-evaluator.md` at `c6a664e` (2026-06-10) carrying only the `replace`/`concat` bullet. `20b4689` added the named `string`, `array<T>`, and `object` stdlib assertions — the moment all four surfaces first co-existed as named bullets in one leaf. `cb8737a`, whose stated goal was to de-bundle the corpus, extracted the whole stdlib surface out of `V3a` into the dedicated new `V3e`/`V3e-T` leaves (git copy-detected `C068` from `V3a-T`), carrying the four surfaces together; the de-bundling split stdlib from the evaluator but left the four stdlib surfaces co-located.

## Solution Space

**Shape:** single

Split `V3e` and its `V3e-T` partner along the string-versus-collection boundary
into two leaf pairs:

- A string-surface leaf pair owning the `String.replace` reference vectors,
  `concat`, and the `string` members — i.e. the existing `Tests` bullets
  `expressions.md — String.replace reference vectors` and
  `expressions.md — string stdlib members`, carried over verbatim.
- A collection-surface leaf pair owning the `array<T>` and `object` members —
  i.e. the existing `Tests` bullets `expressions.md — array<T> stdlib members`
  and `expressions.md — object stdlib members`, carried over verbatim.

Each new leaf keeps the `expressions.md` **Spec** citation and the `V3a`/`V2c`
**Deps** (the implementation leaf additionally depending on its own `-T`
partner). Link the new implementation leaves into the `### V3` slice list in
`docs/plan.md` in place of `V3e`, and replace `V3e` with the two new IDs in the
`expressions.md` (EXPR) row of `coverage-matrix.md`. Allocate the new leaf IDs
with `<new>` placeholders per the plan's ID policy; do not invent final IDs here.

Edge case: this is an advisory NIT — the bundle is correct and pickable as it
stands. Split only if the combined leaf's review burden is judged material; if
left as one leaf, no spec or coverage change is required.

## Consequence

**Severity:** advisory

The bundle ships correctly as-is: each surface already has an independent named
assertion in `V3e`/`V3e-T`, so an implementer can produce a complete, fully-tested
leaf without divergence. The only cost is review and iteration burden — one
leaf's diff spans four independent normative surfaces, enlarging the critique
surface relative to the smallest-feature convention.
