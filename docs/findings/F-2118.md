## Finding

The glossary fixes an explicit casing convention for the language name. The `Loom (language)` entry (`docs/spec_topics/glossary.md`) states: "Capitalised when used as a proper noun (\"Loom expressions\", \"Loom code\", \"Loom follows Rust's…\"); the lowercase form is reserved for sense 1" — i.e. lowercase `loom` is reserved for the `.loom` file-unit sense and the `loom 1.0` version-string pattern, while the language itself is the proper noun `Loom`.

Two shard-04 topic files violate this convention in language-behaviour subject position, mixing both forms within the same documents:

- `docs/spec_topics/expressions.md` — `Loom \`==\` is structural` (capitalised, conforming) coexists with `loom has no JS last-truthy-operand widening` and `Membership test using loom structural equality` (lowercase, non-conforming).
- `docs/spec_topics/control-flow.md` — `Loom has three loop and branch forms` (capitalised, conforming) coexists with `loom performs no truthiness coercion` (lowercase, non-conforming).

In each lowercase case the subject is the language asserting a semantic rule ("loom has no…", "loom performs no…", "loom structural equality"), which is precisely the proper-noun usage the glossary requires to be capitalised. The fix is to capitalise `Loom` at these language-behaviour callsites, leaving the `.loom` file-unit sense and the `loom 1.0` / `loom 1.0.0` version-string literals untouched.

## Issue introduction
**Verdict:** multi-commit-interaction

**History:** Commit 53b8970 established the `Loom (language)` glossary entry and its capitalise-the-proper-noun convention. The two non-conforming prose sites entered later as fresh prose by independent edits that did not honour that convention: `git show ee9f01f` adds the `loom has no JS last-truthy-operand widening` sentence to `expressions.md`, and `git show a08f166` adds the `loom performs no truthiness coercion` sentence to `control-flow.md`. The inconsistency is the interaction of the convention-establishing commit with these subsequent unaware edits, not a defect present at corpus inception.

## Solution Space

**Shape:** single

### Recommendation

Capitalise `Loom` at the language-behaviour callsites so they match the glossary convention:

- `docs/spec_topics/expressions.md`: `loom has no JS last-truthy-operand widening` → `Loom has no JS last-truthy-operand widening`; `Membership test using loom structural equality` → `Membership test using Loom structural equality`.
- `docs/spec_topics/control-flow.md`: `loom performs no truthiness coercion` → `Loom performs no truthiness coercion`.

Leave lowercase `loom` untouched wherever it denotes the `.loom` file unit, a diagnostic code (`loom/parse/...`), a fenced ` ```loom ` block, or a version-string literal (`loom 1.0`, `loom 1.0.0`). Sweep both files for any further language-as-subject lowercase occurrences while editing, applying the same rule.

## Consequence

**Severity:** cosmetic

Pure spec-prose casing inconsistency with no behavioural contract attached; no two implementers would diverge on system behaviour. It mildly weakens the corpus's own naming discipline and the credibility of the glossary convention it contradicts.
