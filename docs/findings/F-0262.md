## Finding

The `V7b` (and its paired `V7b-T`) DIAG-2 test bullet reads: *"the registry is closed — a code asserted by a test with no registry row fails the gate (and, per the `H5a` gate's `Adds.`, a registry code with no asserting test also fails the gate)."* Both arms it names reconcile the **test-asserted code set** against the **registry row set**: test→registry and registry→test. Neither arm observes production emission sites.

A diagnostic code that `src/**` emits at runtime but that is **neither** registered **nor** asserted by any test therefore appears in neither set and passes the gate silently — yet the leaf's `Ships when` and the DIAG-2 header both present the gate as enforcing that "the registry is closed." The closed-set guarantee the header claims is real only if every emitted code is also test-asserted; that bridging premise is the per-phase TDD discipline (the spec's DIAG-1 entitles tests to assert on the specific code at every documented diagnostic site, and the project's per-phase TDD ritual is what actually lands those assertions), not anything the `H5a` gate mechanically checks. The bullet states the conclusion ("the registry is closed") while leaving the load-bearing premise implicit.

The gate itself (`H5a`) is correctly specified — it genuinely reconciles only the test-asserted and registry sets. The defect is in `V7b`/`V7b-T`'s self-description, which over-claims what that reconciliation guarantees.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** f284534 — pi-loom plan: resolve "DIAG-2 src/** emission scan vs closing gate" (2026-06-10, Thomas Andersen)
**History:** The "the registry is closed" DIAG-2 framing entered at the plan's first build (c6a664e) phrased as "a code emitted by `src/**` with no registry row fails the gate," asserting a src/** emission scan. Commit f284534, resolving an earlier review finding that the gate performs no such scan, retargeted both the `V7b` and `V7b-T` DIAG-2 bullets (and `V7b`'s Ships-when) to "a code asserted by a test … reconciles against the registry." That retargeting is accurate about the gate but left the "registry is closed" header attached to a test-assertion-only reconciliation — the present gap: a production-emitted code that is neither registered nor test-asserted escapes the closed set unless the unstated TDD discipline (every emitted code is test-asserted) holds.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V7b-code-registry.md` and `docs/plan_topics/V7b-T-code-registry.md`, rewrite the DIAG-2 Tests bullet so it describes what the gate actually reconciles and states the bridging dependency inline rather than asserting bare closure. The gate's two arms (test-asserted code with no registry row fails; registry code with no asserting test fails) are accurate and stay; what must change is the "the registry is closed" framing, which should be qualified to make explicit that production-emission closure rests on every emitted code being test-asserted per the per-phase TDD discipline (spec DIAG-1).

Candidate replacement text for the DIAG-2 bullet (identical in both leaf files, since the two carry the same bullet verbatim):

> `DIAG-2`: every test-asserted code reconciles with the registry and vice versa — a code asserted by a test with no registry row fails the gate, and (per the `H5a` gate's `Adds.`) a registry code with no asserting test also fails the gate. Closure of the registry against production emissions rests on the per-phase TDD discipline that every emitted code is test-asserted (spec DIAG-1); the gate does not scan `src/**` emission sites directly.

`V7b`'s `Ships when` already reads "reconciles test-asserted codes against the registry," which is consistent and needs no change. The spec (DIAG-1/DIAG-2) is correct and is read-only for this fix; do not edit `H5a`'s gate semantics. Honour the leaf-ID scheme and authoring conventions per `conventions.md`.

## Consequence

**Severity:** advisory

An implementer reading the DIAG-2 bullet may conclude the closing gate alone enforces the closed registry and not realise that catching a production-emitted-but-unregistered code depends entirely on the separate discipline of asserting every emitted code in a test. In practice the per-phase TDD ritual covers it, so a working leaf still results; the gap is a guidance/precision one (the closure guarantee's load-bearing premise is unstated), not a divergence in the gate's behaviour.
