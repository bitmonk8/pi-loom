## Finding

Step 0 (d)'s peer-dep version probe iterates over the four `@earendil-works/*` packages and, for each, resolves its main entry via `createRequire(import.meta.url).resolve("<pkg>")`, walks parent directories to the first `package.json` whose `name` matches the iteration's `<pkg>`, and reads its `version` field. The four `peer-dep-malformed-version` routing conditions — `MODULE_NOT_FOUND` on the main-entry resolution, a root-bound parent walk, a missing `version` field, or a syntactically invalid SemVer — are all stated at the read site, but the implicit deployment shape they presuppose (each peer exists as a *discrete*, *on-disk* package with a synchronously-readable `package.json` reachable by Node's CommonJS resolver from the loom factory's module) is nowhere stated as a precondition.

That shape is genuinely load-bearing for the probe's diagnostic semantics. A loom build that inlines the four peers (e.g. a future repackaging through any bundler that traverses `peerDependencies` and ships their JS into the extension's own `dist/`, or any vendored layout in which the peers exist as in-tree source rather than as resolved package roots) makes `createRequire(...).resolve("<pkg>")` throw `MODULE_NOT_FOUND` even though the runtime is fully conformant — the four peers are present, just not as resolvable packages. The probe then emits `loom/load/host-incompatible` with `details.kind = "peer-dep-malformed-version"` and `details.observed = "<unresolvable>"` on every peer, a false-negative that misclassifies the deployment rather than the host.

The loom 1.0 packaging model that the rest of the spec describes (peer dependencies declared in `package.json#peerDependencies` for skew detection by `pnpm` / `yarn`, the extension auto-discovered from `package.json#pi.extensions`) is the deployment shape in which (d) is well-defined. The probe contract should make that scope explicit so the four routing conditions are read as gating that precondition rather than as a packaging-shape policy in disguise — and so any future repackaging is recognised as out-of-scope for the loom 1.0 probe rather than as a host-conformance bug.

The discrete-on-disk assumption was baked into (d)'s mechanic at `126ff8a` and has been carried unchanged through every subsequent edit.

## Solution Space

**Shape:** single

### Recommendation

Add a one-sentence deployment-shape precondition to `capability-probe.md` immediately before the four-package iteration in Step 0 (d), stating that the four `@earendil-works/*` peer packages are presupposed to be installed as discrete, on-disk packages each carrying a synchronously-readable `package.json` reachable by Node's CommonJS resolver from the loom factory's module, and that the four `peer-dep-malformed-version` routing conditions below gate that precondition. Reference the [Pi SDK pin / Manifest lock-step](./host-prerequisites.md#pi-sdk-pin-manifest-lock-step) framing as the corpus-wide source of the packaging model (peer dependencies declared in `package.json#peerDependencies`, install-time skew detection by `pnpm` / `yarn`) the precondition is the runtime restatement of; do not invent a new packaging contract here. Bundled-or-vendored loom builds in which the four peers are inlined rather than installed as discrete packages are explicitly out-of-scope of the loom 1.0 probe and not host-conformance failures. Implementer-visible edge case: the precondition is what makes "`MODULE_NOT_FOUND` on the main-entry resolution call" a `peer-dep-malformed-version` rather than a packaging-shape branch in (d); the routing semantics do not change.

## Consequence

**Severity:** advisory

A spec-conformant implementation that follows (d) verbatim will produce the documented routing on the deployment shape loom 1.0 actually ships under, so nothing breaks for the loom 1.0 release. The unstated precondition leaves the probe's failure semantics ambiguous for any future repackaging: a bundled-or-vendored build of loom would have its four peers silently misclassified as `peer-dep-malformed-version` on conformant hosts, with no in-spec hook explaining why or recording that the deployment shape is out of scope.
