## Problem

The *Binder context* section of `binder/binder-model-and-context.md` specifies only the parse-time side of declaring `bind_context: session` on a `mode: subagent` loom: parsing fires `loom/parse/bind-context-session-on-subagent` as a non-fatal **W**, and the loom still loads, registers, and advertises a slash command. The runtime invocation-time side is silent, yet that slash dispatch is the only point at which `bind_context` is ever read. The spec leaves two materially different readings open: the runtime degrades the combination to `bind_context: none` for this invocation, or it refuses the invocation and surfaces some failure variant (which itself has no pinned bytes). A conformance test cannot decide which transcript bytes the binder should see.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `83802db` — *spec: collapse to single typed argument style; add LLM-driven slash-command binder* (2026-05-04)
**History:** `git log -S "Declaring \`bind_context: session\` on a subagent-mode loom"` against the whole corpus returns three commits — `83802db` (initial introduction in the monolithic `spec.md`), `fecb504` (2026-05-04, mechanical split of `spec.md` into per-topic files), and `f5e89f4` (2026-06-04, the size-cap move to `docs/`). The introducing commit added the sentence in its current form: "`bind_context: session` on a subagent-mode loom is a parse warning, not an error — subagent-mode looms invoked from a slash command have no caller-session context to attach", together with the `bind_context:` field, the `none`/`session` enum, and the binder design as a whole. The runtime-side silence was present from that first introduction: the introducing commit specified the warning and explained *why* the combination is degenerate, but never specified what the runtime does at invocation time when a loom carrying that warning is dispatched. The subsequent splits (`fecb504`) and moves (`f5e89f4`) preserved the wording verbatim. No later commit narrowed or expanded the runtime side. The defect is the original framing, not a regression.

## Solution approach

Clarify the *Binder context* section to pin the runtime invocation-time behaviour for `bind_context: session` on a `mode: subagent` loom: at slash-invocation time the runtime treats the value as `bind_context: none` for binder-input construction — the `#session-context-truncation-bind_context-session` walk is skipped and no *Recent session context* block is emitted. Place the runtime pin adjacent to the existing `loom/parse/bind-context-session-on-subagent` warning sentence.
