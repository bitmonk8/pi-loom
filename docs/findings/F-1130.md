## Finding

The *Binder context* section of `binder/binder-model-and-context.md` pins one obligation about declaring `bind_context: session` on a `mode: subagent` loom: parsing fires `loom/parse/bind-context-session-on-subagent` as a **W** (warning), and the loom still loads — the code-registry row in `diagnostics/code-registry-parse.md` and the registry rule that a `W` does not block registration both confirm this. The runtime side of the same combination is silent.

The combination is reachable: a subagent-mode loom that carried only this warning at load time is registered, advertises a slash command, and can therefore be invoked from a slash dispatch — the only entry point at which `bind_context` is ever read. At that invocation point the spec leaves implementers with two materially different readings: (a) the runtime treats `bind_context: session` as if it were `bind_context: none` for this invocation — skip the session-context truncation walk, omit the *Recent session context* block from the binder system prompt, and run the binder normally on the slash text plus frontmatter; or (b) the runtime refuses the invocation (the warning's hint, "Subagent-mode looms have no caller-session context", arguably implies "there is no session to attach, so the binder cannot run"), in which case the loom does not start and *something* surfaces — but no row of the `determinism-cancellation-failure.md` failure-mode templates covers it, so the user-visible bytes are equally undefined.

Reading (a) is the only one consistent with the rest of the spec: the warning's `W` severity is explicit that the configuration is non-fatal, the *Binder context* preamble pins runtime behaviour per `bind_context:` *value* (with `none` defined as "binder sees only the slash text and the loom's frontmatter"), and `bind_context: none` is already the default — so degrading to it on this combination is a no-op for the binder-input path. But the spec never says so, the truncation-walk subsection presupposes a caller session exists without naming the prerequisite, and `V11b`'s acceptance tests cover only the parse-time warning, not the invocation outcome. A conformance test today cannot decide which transcript bytes the binder should see.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `83802db` — *spec: collapse to single typed argument style; add LLM-driven slash-command binder* (2026-05-04)
**History:** `git log -S "Declaring \`bind_context: session\` on a subagent-mode loom"` against the whole corpus returns three commits — `83802db` (initial introduction in the monolithic `spec.md`), `fecb504` (2026-05-04, mechanical split of `spec.md` into per-topic files), and `f5e89f4` (2026-06-04, the size-cap move to `docs/`). The introducing commit added the sentence in its current form: "`bind_context: session` on a subagent-mode loom is a parse warning, not an error — subagent-mode looms invoked from a slash command have no caller-session context to attach", together with the `bind_context:` field, the `none`/`session` enum, and the binder design as a whole. The runtime-side silence was present from that first introduction: the introducing commit specified the warning and explained *why* the combination is degenerate, but never specified what the runtime does at invocation time when a loom carrying that warning is dispatched. The subsequent splits (`fecb504`) and moves (`f5e89f4`) preserved the wording verbatim. No later commit narrowed or expanded the runtime side. The defect is the original framing, not a regression.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/binder/binder-model-and-context.md`, immediately after the existing sentence declaring `loom/parse/bind-context-session-on-subagent`, add one normative sentence pinning the runtime side:

> At slash-invocation time, the runtime MUST treat `bind_context: session` on a `mode: subagent` loom as `bind_context: none` for the purpose of binder-input construction — the [Session-context truncation walk](#session-context-truncation-bind_context-session) is skipped, no *Recent session context* block is emitted, and the binder runs against the slash text and frontmatter alone. The parse-time warning is the operator surface; the runtime carries no further obligation.

Edge cases the implementer must watch:

- The degrade is **per invocation**, not "rewrite the frontmatter value": the loom's declared `bind_context:` value stays whatever the author wrote, and any future re-entrant binder call on the same loom turn (per the loom 1.0 seam paragraph immediately below this sentence) re-applies the same degrade.
- The degrade applies regardless of whether the warning was actually surfaced to the operator (a watcher rebuild that re-emits the same diagnostic, a silenced diagnostic channel, etc. do not change the runtime side).
- The session-context truncation walk's own preconditions (PIC-16 estimate well-formedness, the chronological-order and leading-`user` invariants) are not consulted at all on this combination, so the related finding *session-context truncation walk undefined for non-finite/negative per-message estimate* is independent of this one.
- Update `V11b`'s **Tests.** to add an invocation-time oracle (binder system prompt for a subagent loom with `bind_context: session` contains no *Recent session context* block) so the new MUST has a closing test.

## Consequence

**Severity:** correctness

Two reasonable implementers will produce different observable behaviour on the same loom: one skips the walk and ships the binder a session-less system prompt; another rejects the invocation and renders some failure-template variant (which itself has no pinned bytes). Authors of subagent-mode looms who tripped the `W` at load time (and may well not have noticed it) will see divergent outcomes across implementations, and conformance tests have no oracle to discriminate. The same defect also blocks `V11b`/`V11i` from achieving full assertion coverage of the `bind_context` surface.
