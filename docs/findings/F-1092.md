## Problem

The `V6c` leaf owns the load-time `loom/load/unknown-tool` reject ("a `tools:` entry naming a Pi tool absent from the registry is rejected at load time"), but its `Spec.` field cites only `frontmatter/frontmatter-fields-a.md`, whose `tools` prose ("Pi tool names … resolve against Pi's tool registry at loom-load time") has no referent the leaf points at. The leaf never names where the host Pi-tool set comes from, nor states that that set is resolvable and stable at the load pass. An implementer reading only the leaf has no statement of how a Pi-tool name resolves to a `ToolDefinition` nor what set determines `loom/load/unknown-tool` membership, so two reasonable readings — a runtime host tool-registry read versus the static SDK built-in-tool factory family — diverge on observable behaviour for a `tools:` entry naming a tool another extension registered at runtime. The paired `V6c-T` tests leaf inherits the same gap.

## Solution approach

In `V6c-tools-set.md`, extend the `Spec.` field to additionally cite `../spec_topics/frontmatter/frontmatter-fields-b-and-templates.md` (Resolution snapshot) and `../spec_topics/pi-integration-contract/subagent.md` (built-in-tool factory family), and state in the `Adds.` field what set backs `loom/load/unknown-tool` membership — that a `tools:` Pi-tool name resolves against the SDK-exported built-in-tool factory family and the reject fires when no factory exists for the name, with the set statically resolvable and snapshot-stable at the load pass. Mirror the same `Spec.` citation addition in `V6c-T-tools-set.md` so the paired tests leaf cites the page defining the resolution source it asserts against.
