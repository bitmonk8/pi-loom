## Finding

The prompt-mode registration-cache paragraph in `tool-registration-lifetime.md` names the synthesised Pi tool for a `.loom` callee `__loom_callee_<slug>__<post-rename-name>`, using `<post-rename-name>` as the human-readable suffix that identifies the callee. The same concept — the callee loom's canonical, post-`pi.looms`-mapping slash-name — is named the **final derived name** in `active-invocation-registry.md` (the `loom` field "carries the invocation's canonical-key name (the *final derived name* per Discovery — post-`pi.looms` mapping, stem taken verbatim — with no leading `/`)") and is the term defined at its authoritative site in `discovery/discovery-sources.md` ("Detection runs on the **final derived name**"). The corpus therefore carries two spec-owned labels for one concept, with `<post-rename-name>` appearing only in the synthesised-name placeholder form.

The placeholder occurs in three lock-step sites, not one: the base form in `tool-registration-lifetime.md`'s prompt-mode paragraph, the collision-disambiguated counter form `__loom_callee_<slug>_<n>__<post-rename-name>` in PIC-44 of the same file, and the `__loom_callee_<slug>__<post-rename-name>` entry in `schema-subset.md`'s **Synthesised names** list (item 5), which declares itself "the source of truth for the full set" of synthesised names. Any rename must touch all three or the source-of-truth list diverges from the definition site.

The label `<post-rename-name>` is also latently misleading. The registration cache is keyed by schema slug (the lowered `parameters` shape) and dedups across all callers, so a single registration cannot carry a caller-specific name: the same `.loom` callee referenced from two callers with different `as`-renames produces one slug and one registered name. The suffix is therefore a property of the callee — its canonical slash-name (the *final derived name*) — not the caller-side `tools:`-entry name. The corpus uses "post-rename name" elsewhere (the Resolution snapshot's `{ post-rename name → resolved callable }` table, `expressions.md`, `queryerror-variants.md`) for that distinct caller-side concept; reusing it here both breaks terminology consistency and points an implementer at the wrong derivation.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** `27ec53d` (2026-05-04, "Resolve Cluster-A: per-mode tool registration"); `6750c81` (2026-05-05, "spec: consolidate QueryError variants; rename to InvokeInfraError / CodeToolError / ModelToolError")
**History:** `git log -S 'post-rename-name'` shows the `<post-rename-name>` placeholder entered the corpus in `27ec53d`, which rewrote the prompt-mode synthesised tool name from `__loom_callee_<sha>__<name>` to `__loom_callee_<sha12>__<post-rename-name>`. The competing term "final derived name" was coined a day later in `6750c81`, which introduced it in the Discovery detection text while the same commit simultaneously used "post-rename name" prose elsewhere in the file — the two labels were never reconciled against the pre-existing placeholder. The `schema-subset.md` source-of-truth occurrence was later added in `c577ab3` (2026-06-06, resolving "Glossary schema slug name enumeration cannot be complete"), propagating the stale placeholder spelling into the canonical synthesised-name list. The defect is the un-reconciled interaction of the independently-coined terms, not a single introducing edit; subsequent file splits (`f5e89f4`) carried both spellings forward unchanged.

## Solution Space

**Shape:** single

### Recommendation

Replace `<post-rename-name>` with `<final-derived-name>` at all three lock-step occurrences — the prompt-mode base form and the PIC-44 counter form in `tool-registration-lifetime.md`, and the *Synthesised names* entry (item 5) in `schema-subset.md` — and add a parenthetical at the `schema-subset.md` definition site such as "(the callee's post-`pi.looms`-mapping stem, taken verbatim for unmapped sources, no leading `/`)" so the placeholder's referent matches the `active-invocation-registry.md` `loom` field and the `discovery/discovery-sources.md` definition.

Edge case the implementer must watch: the broader prose term "post-rename name" — used in the frontmatter Resolution snapshot's `{ post-rename name → resolved callable }` table, `expressions.md`, and `queryerror-variants.md` — denotes a *different* concept (the caller-side `tools:`-entry name after `as`-rename or hyphen→underscore rewrite). It MUST NOT be touched; only the synthesised-callee-tool-name placeholder is renamed.

## Consequence

**Severity:** advisory

The schema slug guarantees dispatch correctness regardless of the suffix, so no runtime behaviour breaks. But the suffix is observable in Pi's global tool registry (`pi.getAllTools()` / `/tools` listings), and two reasonable implementers could populate it differently — one with the caller-side `as`-rename the label literally suggests, one with the callee's canonical slash-name — yielding divergent registry entries for the same callee.
