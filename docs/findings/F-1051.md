## Problem

Two paragraphs in `version-bump-step2.md` over-prescribe implementation detail where the underlying obligation is purely behavioural. The step 2(a) positive-direction paragraph (`#bump-step-2-positive`) mandates the `CAPABILITY_OBLIGATIONS` constant's concrete data structure (a closed array of seven anchor IDs) and its module colocation alongside the four Step-0 probe constants, and inline-enumerates all seven anchor IDs — duplicating what `capability-inventory-items.md` is the source of truth for. Independently, the baseline-acquisition block (`#bump-baseline-acquisition`) mandates `npm pack @earendil-works/pi-coding-agent@<pinned-version>` as the specific command (with `npm install --prefix` framed as the rejected alternative), where the behavioural requirement is only that a scratch copy of the loom 1.0-pinned `dist/*.js` tree be materialised, distinct from the candidate-minor install step 1 lays down in `node_modules/`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `5dfc9aa` — pi-loom spec: resolve "peerDependencies literal-read assertion shape and CAPABILITY_OBLIGATIONS member-anchor list unstated" (2026-06-07, Thomas Andersen); `ccc6e38` — pi-loom spec: resolve "Item (e)/(r) dist/*.js baseline source unspecified" (2026-06-07, Thomas Andersen)
**History:** Both over-prescriptions were introduced on the same day by two independent "resolve" commits that each over-shot the underspecification finding they were closing. `5dfc9aa` rewrote the step-2(a) paragraph to spell out `CAPABILITY_OBLIGATIONS` as "a closed array of seven anchor IDs … in that exact order" with all seven literal anchor IDs inlined, plus the pinned-constants-block colocation clause — going beyond the cited finding (which asked only for the member list to be stated *somewhere* canonical) and pinning the data structure and module location too. Independently, `ccc6e38` added the `bump-baseline-acquisition` block to fix an unspecified-baseline finding by mandating the specific `npm pack` command (with `--prefix` framed as the rejected alternative) where "materialise the pinned `dist/*.js` tree" was the underlying requirement. Neither commit touched the other's site, so the two over-prescriptions are independent in origin and remain independent to resolve.

## Solution approach

Rewrite the `#bump-step-2-positive` paragraph to state the `CAPABILITY_OBLIGATIONS` gate behaviourally — name the constant and its `CAPABILITY_OBLIGATIONS.length === 7` assertion, and reference `capability-inventory-items.md` items 1–7 as the source of truth for membership and ordering — leaving the data structure and the constant's module location to the implementer. Delete the inline enumeration of the seven anchor IDs and the pinned-constants-block colocation mandate. Separately, rewrite the `#bump-baseline-acquisition` block to state the requirement as a scratch copy of the loom 1.0-pin's `dist/*.js` tree distinct from the step-1 `node_modules/` install, presenting `npm pack` as one means rather than the mandated command.
