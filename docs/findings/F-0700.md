## Finding

The `system:` *Path grammar* block in `frontmatter-fields-b-and-templates.md` says the head `Ident` is a "case-sensitive, exact byte match against the declared `params` field name" — a **loom-side** name — and that resolution happens "against the validated params object," which `runtime-value-model.md` defines as a JS object keyed by loom-side names. But the very next sentence says each subsequent `.Ident` "must name a reachable field of an *object* schema in **the lowered params schema**," and the lowered schema is the wire-keyed JSON Schema produced by the lowering pass (`schema-subset.md` step 3: `properties` and `required` carry wire names). The two namespaces contradict each other on the same path.

For any renamed nested field the contradiction is observable. Given

```loom
schema Author { name as "FullName": string }
params:
  author: Author
```

the rule from the head `Ident` plus the stated resolution target make `${author.name}` the only candidate that resolves at evaluation time, but the lowered-schema reachability check on the `.name` step looks at `properties` under the wire name `FullName`, so the parse-time `loom/parse/system-interp-bad-field` step (which `code-registry-parse.md` defines as "A `.Ident` step in a `system:` `${...}` path does not name a reachable object field") fires on the loom-side spelling and not on the wire spelling — under the literal text of the spec, neither `${author.name}` nor `${author.FullName}` is well-defined. This also clashes with the global invariant in `runtime-value-model.md` ("Loom code never sees wire names; tools, the model, and external JSON Schema consumers never see loom-side names") and with `expressions.md` / `loom/parse/non-string-object-index`, which fix object navigation on the loom-side name throughout the language surface.

The intent is clearly the loom-side surface — the head `Ident`, the resolution target, and every other field-access form in loom land all use loom-side names — and the phrase "in the lowered params schema" is the lone wire-keyed clause embedded inside a loom-side-keyed sentence.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the `.Ident` reachability sentence in `frontmatter-fields-b-and-templates.md`'s *Path grammar* paragraph to gate on the **loom-side declared object schema** rather than the lowered schema, and align the `code-registry-parse.md` row for `loom/parse/system-interp-bad-field` with the same wording. Concretely:

- In `frontmatter-fields-b-and-templates.md`, change the second sentence of the paragraph beginning "`Ident` is the lexical identifier …" so each subsequent `.Ident` "must name a declared field of an *object* schema in the loom-side `params:` declaration (the same identifier used to reference the field in loom code)." Drop the phrase "in the lowered params schema" entirely; it is the sole wire-keyed clause in an otherwise loom-side paragraph.
- Add one clarifying sentence — placed at the end of the same paragraph or immediately under *Stringification* — stating explicitly that `${param.field}` paths use loom-side names throughout, never the `as "WireName"` rename target, consistent with the global invariant in `runtime-value-model.md` ("Loom code never sees wire names") and with object indexing rules in `expressions.md`. Wire-name translation applies only at the validation boundary when constructing the lowered tool input the model sees, which is not a surface this slot exposes.
- In `code-registry-parse.md`, change the `loom/parse/system-interp-bad-field` *Cause* cell from "does not name a reachable object field, or descends into an array or un-narrowed discriminated union" to "does not name a declared object field on the loom-side schema resolved one step up the path, or descends into an array or un-narrowed discriminated union." The *Message* template ("`'system:' interpolation '.<field>' does not name a reachable object field on <path>`") already renders the loom-side spelling — `<field>` is the source-text `Ident` the author wrote — and needs no change.

Edge cases for the implementer:

- A redundant rename (`field as "field"`, which carries the warning `loom/parse/redundant-wire-name`) parses successfully under either reading; the corrected rule does not change its behaviour.
- A field declared as `field as "WireName": T` is now unambiguously `.field` in `system:`; `${param.WireName}` fires `loom/parse/system-interp-bad-field` even though `WireName` appears in the lowered schema's `properties`. This matches the behaviour of every other field-access form in the language.
- Discriminated-union arms remain rejected at parse time (`system-interp-bad-field` per the second clause), unchanged.
- The corrected reachability check runs at frontmatter-parse time against the loom-side `params:` declaration, not at evaluation time — the existing ordering claim ("All four codes fire at frontmatter-parse time, before AJV validates the params payload") survives intact.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge on which path string parses and which fires `loom/parse/system-interp-bad-field`: one will gate `.Ident` on the lowered schema's `properties` keys (wire names), the other on the loom-side object schema's declared field names; a loom whose `params` carries any object-typed param with a wire-name rename will either fail to load or silently mis-resolve depending on which reading the implementer picked. The bug is invisible until a rename appears, at which point the spec gives no path that works.
