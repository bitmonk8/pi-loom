## Problem

BNDR-11 ‚Äî a `looms.binderModel`-only settings edit MUST NOT re-run binder-model resolution or the `strictCapable` probe for already-loaded looms; each retains its previously-resolved handle until its next load ‚Äî is a live numbered REQ-ID with no row in the coverage matrix's Numbered REQ-IDs table and no citing leaf. `V11a` owns binder-model resolution and the `strictCapable` probe; `V10d` owns the settings re-merge. Neither verifies the no-re-resolve guard, so the obligation ships unverified.

## Issue introduction

**Verdict:** introduced-by-later-spec-edit (two distinct commits; never propagated to the plan)
**Introducing commits:** `bc77cf6` (`pi-loom spec-plan: fix F-1130 ‚Äî coin BNDR-10 ‚Ä¶`, 2026-06-24) for BNDR-10; `1b6d2f1` (`pi-loom spec-plan: fix F-1282 ‚Äî coin BNDR-11 ‚Ä¶`, 2026-06-24) for BNDR-11.
**History:** `git log -S bndr-10` / `-S bndr-11` over `docs/spec_topics/binder/binder-model-and-context.md` traces each REQ-ID to a single spec-coining commit (`bc77cf6` and `1b6d2f1` respectively). `git log -G 'BNDR-7, BNDR-8, BNDR-9'` and `git log -S 'BNDR-10' / -S 'BNDR-11'` over `docs/plan_topics/coverage-matrix.md` show the BNDR coverage row was authored once at the plan build (`c6a664e`, `pi-loom plan: build/update plan for spec.md + review`, 2026-06-10) and BNDR-10 / BNDR-11 were never added. `git merge-base --is-ancestor` confirms `c6a664e` predates both coining commits: the coverage matrix was built when only BNDR-1‚Ä¶9 existed, and each later spec edit that coined a new BNDR REQ-ID landed without a paired coverage-matrix row or citing leaf. The defect is not present-since-inception of the matrix; it is two independent spec-after-plan additions whose plan-side propagation was omitted.

## Solution approach

Add a `BNDR-11` closing row to the coverage matrix's Numbered REQ-IDs table naming `V11a` as the closing leaf (`V10d` is an acceptable alternative if the team places the guard on the settings re-merge path). Add a Tests bullet to the chosen closing leaf and its `-T` pair citing `BNDR-11` inline and asserting that a `looms.binderModel`-only settings edit does not re-run binder-model resolution or the `strictCapable` probe for already-loaded looms (the previously-resolved handle is retained until the loom's next load).
