## Problem

The *Wire serialisation* clause of `audit-wire-and-canary.md` pins audit records as line-delimited UTF-8 on stdout with each line leading with the `audit/<class>/<family>/<symptom>` discriminator for leading-token prefix routing, and the final wiring paragraph declares the line-delimited stdout stream the sole machine-parseable surface a CI parser may consume in loom 1.0. That contract holds only if the audit's records are the only bytes on the stream, but the spec never states that precondition. The same section permits running the audit in-process to the harness (e.g. as a Vitest test case), where the test runner's own reporter writes to the same process stdout — interleaving reporter output with audit records, and worse, a non-`\n`-terminated reporter write immediately before a record splices into the same physical line so the record no longer begins with `audit/` and the leading-token router silently drops it. The spec thus offers a free choice between a clean-stream wiring (isolated child process) and a shared-stream wiring (in-process Vitest) and asserts the stdout contract holds for both, without requiring that the record stream be a channel the harness does not also write to.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `e5cd514` (2026-05-09) "pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\""
**History:** `git log -S 'line-delimited UTF-8 text on **stdout**'` against the spec corpus identifies `e5cd514` as the commit that first authored the *Wire serialisation* clause's per-line stdout obligation and the leading-token routability rule. The same commit also authored the harness-wiring epilogue (`git log -S 'in-process to the harness'`), which is the source of the in-process-Vitest permission the precondition gap rests on. Both halves of the defect — the line-routing contract that needs stdout integrity and the in-process wiring that does not guarantee it — were therefore introduced together in `e5cd514` and have travelled unchanged through `f5e89f4` (size-cap split that mechanically moved the prose into `audit-wire-and-canary.md`) and `81bf52a` (the `spec/` → `spec_topics/` rename). No later commit narrowed or revisited the precondition.

## Solution approach

Clarify the *Wire serialisation* clause and the final `npm test`-wiring paragraph of `audit-wire-and-canary.md` to add the missing precondition: the machine-parseable record stream MUST be a channel the test harness does not itself write to. Pin the obligation across both permitted wiring arms — the child-process wiring (the spawned audit's own stdout captured as a stream distinct from the harness's stdout) and the in-process wiring (records routed to a sink the in-process test runner's reporter does not also write to), re-affirming the existing "survive the harness wrapping" rule that records still reach a process stream a CI parser can consume.
