## Problem

The *Wire serialisation* clause in `audit-wire-and-canary.md` pins a leading-token routing contract: each violation, infrastructure-failure, and canary record is one `\n`-terminated stdout line whose leading whitespace-separated token is the `audit/<class>/<family>/<symptom>` discriminator a CI parser routes on without parsing the rest of the line. The clause never states the preconditions that contract depends on: that each record's bytes reach stdout as a single atomic write (so the four tab-delimited fields cannot tear across two writes) and that no other writer interleaves output between a record's discriminator token and its trailing newline. The same page's harness-wiring epilogue explicitly permits running the audit in-process to the harness (e.g. a Vitest case sharing the test runner's stdout), where a parallel reporter, a concurrent `console.log`, or a record exceeding Node's stdout chunk threshold can tear or interleave records, leaving lines that no longer satisfy the routing contract. The leading-token contract is therefore satisfiable only by accident on the in-process variant the page admits.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `e5cd514` (2026-05-09) "pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\""
**History:** `git log -S 'line-delimited UTF-8 text on **stdout**'` against the spec corpus identifies `e5cd514` as the commit that first authored the *Wire serialisation* clause's per-line stdout obligation and the leading-token routability rule. The same commit also authored the harness-wiring epilogue (`git log -S 'in-process to the harness'`), which is the source of the in-process-Vitest permission the precondition gap rests on. Both halves of the defect — the line-routing contract that needs stdout integrity and the in-process wiring that does not guarantee it — were therefore introduced together in `e5cd514` and have travelled unchanged through `f5e89f4` (size-cap split that mechanically moved the prose into `audit-wire-and-canary.md`) and `81bf52a` (the `spec/` → `spec_topics/` rename). No later commit narrowed or revisited the precondition.

## Solution approach

Rewrite the *Wire serialisation* clause in `audit-wire-and-canary.md` to pin the stdout-integrity precondition: each emitted record reaches stdout via a single atomic write, and the harness wiring guarantees no other writer interleaves between the start of a record's leading discriminator token and that record's trailing `\n` on the byte stream a CI parser consumes. Extend the harness-wiring epilogue paragraph so whichever wiring is adopted (in-process or child-process) must demonstrate the precondition holds under its concrete adoption, on the same footing as the existing single-concrete-adoption rule there.
