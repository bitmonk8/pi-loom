## Problem

The closed continuation-trigger table in `grammar.md` § Newline continuation lists `?` in the trailing binary-or-ternary operator set (`+ - * / % == != < <= > >= && || ? :`), so a line ending in `?` continues across the newline. But `?` is also loom's postfix error-propagation operator, which is a completed expression, so source like `result?\nnext_stmt()` is read literally as one statement even though every other page treats `result?` as a complete expression. The trigger set is declared closed and the table is normative, so no implementer is licensed to silently carve postfix `?` out, and two conforming front-ends diverge on whether a trailing `?` ends a statement. The defect is isolated to the trailing-operator row; the leading-operator row is unambiguous because postfix `?` never starts a line.

## Issue introduction

**Verdict:** present-since-this-section-was-authored
**Introducing commit:** `f841b12` (2026-05-04, *spec: resolve Cluster 2 (literal sublanguage + grammar appendix)*)
**History:** The `## Newline continuation` section, including the trigger table and the operator string `+ - * / % == != < <= > >= && || ? :`, was added wholesale by `f841b12` when the grammar appendix was first introduced (`git log -S "Trailing binary or ternary operator"`, `git show f841b12 -- spec_topics/grammar.md`). The immediately prior spec text (`8eea523`, 2026-05-04) said "trailing binary or unary operator" with no `?` in the set. The same commit's authors did not own the postfix `?` operator (lives in `expressions.md` / `errors-and-results/`), so the conflation entered with the appendix and has not been touched by subsequent grammar-appendix commits (`5c8041a`, `c3b7c87`, `e1f5c2a`, etc.). The sibling continuation prose in `lexical.md` later acquired the same "trailing binary or ternary" wording in `38e399f` (2026-06-05, *trailing-continuation operator set disagreement*) by aligning to grammar.md, propagating the defect to a second page without resurfacing it.

## Solution approach

In `grammar.md` § Newline continuation, drop `?` from the trailing-operator trigger string so postfix `?` becomes a statement-completing token and ternary continuation across a newline relies on the already-documented leading-operator form. Clarify in the same section that a trailing `?` is the postfix error-propagation operator and is never a continuation trigger. Mirror the change in `lexical.md` § Statement terminators inline prose so the two continuation sites do not redrift.
