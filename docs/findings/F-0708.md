## Finding

The closed continuation-trigger table in `grammar.md` § Newline continuation lists `?` as part of the trailing **binary or ternary** operator set (`+ - * / % == != < <= > >= && || ? :`). Read literally, a line ending in `?` therefore continues across the newline.

But `?` is also loom's postfix error-propagation operator (`expressions.md` § Postfix error-propagation, precedence level 1, owned by `errors-and-results/error-model.md`), which is a *completed* expression — `result?` already evaluates to the unwrapped `Ok` value or `return Err(e)`s. A literal application of the trigger table to source like

```
let x = result?
next_stmt()
```

makes the two lines one statement (parse failure / unrelated diagnostic), even though every other page treats `result?` as a complete expression and `next_stmt()` as a new statement. The trigger set is declared "closed" and the table is normative, so no implementer is licensed to silently carve postfix `?` out; two conforming front-ends will diverge on whether a trailing `?` ends a statement.

The same ambiguity does not affect the leading-operator row: a line beginning with `?` is unambiguously the ternary `?` (postfix `?` never starts a line on a fresh operand). The defect is isolated to the trailing-operator row.

**Verdict:** present-since-this-section-was-authored

**History:** The `## Newline continuation` section, including the trigger table and the operator string `+ - * / % == != < <= > >= && || ? :`, was added wholesale by `f841b12` when the grammar appendix was first introduced (`git log -S "Trailing binary or ternary operator"`, `git show f841b12 -- spec_topics/grammar.md`). The immediately prior spec text (`8eea523`, 2026-05-04) said "trailing binary or unary operator" with no `?` in the set. The same commit's authors did not own the postfix `?` operator (lives in `expressions.md` / `errors-and-results/`), so the conflation entered with the appendix and has not been touched by subsequent grammar-appendix commits (`5c8041a`, `c3b7c87`, `e1f5c2a`, etc.). The sibling continuation prose in `lexical.md` later acquired the same "trailing binary or ternary" wording in `38e399f` (2026-06-05, *trailing-continuation operator set disagreement*) by aligning to grammar.md, propagating the defect to a second page without resurfacing it.

## Solution Space

**Shape:** multiple

### Option A — Drop `?` from the trailing-operator set; require ternary line-break before the `?`

**Approach.** Remove `?` from the trailing-operator string in the table; postfix `?` is then unambiguously a statement-completing token, and ternary continuation across a newline must use the leading-operator form (`cond\n  ? a : b`), which is already a documented trigger.

**Spec edits.**
- `grammar.md` § Newline continuation, table row 2: change the trailing-operator string to `+ - * / % == != < <= > >= && || :` (drop `?`). Add one sentence after the table: "A trailing `?` is the postfix error-propagation operator and is never a continuation trigger; ternary continuation across a newline uses the leading-operator form (`cond\n  ? a : b`)."
- `lexical.md` § Statement terminators: mirror the change in the inline prose ("a trailing binary or ternary operator" → "a trailing binary operator or `:`") so the two sites do not redrift.
- `V1a` / `V1a-T`: drop the trailing-`?` positive vector if one was planned; add a vector that `result?\nnext` tokenises as two statements; keep the leading-`?` ternary vector.

**Pros.** Lexer needs no parse-state context (ternary nesting is not tracked). One closed rule per token. Matches the dominant author intuition that `result?` is a completed expression. Minimal text change.

**Cons.** Authors who would prefer trailing-form ternary across a newline lose that form; they must rewrite as leading-operator. (Anecdotally rare — leading-operator is the documented idiom for both branches.)

**Risks.** None on the lexer; one edit-coordination risk in `lexical.md` (kept by the same edit).

### Option B — Make trailing `?` context-sensitive on an in-progress ternary nesting counter

**Approach.** Add a ternary-balance counter to the lexer (analogous to bracket balance): every `?` not preceded by a postfix-eligible operand opens a pending ternary; the matching `:` closes it. A trailing `?` continues only when it opens a pending ternary; a trailing postfix `?` (the operand-already-present case) does not.

**Spec edits.**
- `grammar.md` § Newline continuation, table row 2: keep `?` in the operator string, but qualify with: "a trailing `?` continues only when it opens a ternary (i.e., a matching `:` is still expected); a postfix `?` applied to a `Result<_, QueryError>` operand is a completed expression and does not continue." Pin the disambiguation rule.
- Define the ternary-balance counter in `lexical.md` (or a new "ternary nesting" sub-section), alongside bracket balance.

**Pros.** Preserves trailing-form ternary continuation. No author-visible breakage.

**Cons.** Introduces parse-state context into the lexer (it previously needed only bracket balance and operand/operator distinction); the rule is harder to test and to state without ambiguity (what happens inside an unbalanced `?:` straddling a `{` block?). Two implementations are more likely to diverge on edge cases than under Option A.

**Risks.** Adds a second context-counter the closed-trigger philosophy of the section was designed to avoid; widens the test surface for V1a-T meaningfully.

### Recommendation

**Option A.** The postfix `?` operator is the dominant user-facing form (every `Result` consumer uses it); the trailing-form ternary is an editorial preference with a working substitute (leading-operator). Lexer simplicity is the section's stated design goal ("the trigger set is closed"). Edge case to watch in V1a-T: assert that `cond ?\n  a : b` is **not** accepted as continued (`?` no longer trails), and that `cond\n  ? a : b` **is** accepted (leading-operator path unchanged).

## Consequence

**Severity:** correctness

Two reasonable implementations of the trigger table diverge on `result?\n<next>`: one continues (per the literal trailing-`?` rule), the other closes the statement (per postfix `?` being a completed expression). The disagreement is silent — both can claim conformance — and shows up only as a parse-error mismatch on real loom code that uses the dominant `?` form.
