## Finding

The closed continuation-trigger table in `grammar.md` § Newline continuation lists `?` inside the trailing binary-or-ternary operator set (`+ - * / % == != < <= > >= && || ? :`). Read literally, this means any line ending in `?` continues across the newline. But `?` is also Loom's postfix error-propagation operator, owned by [Expressions — Postfix error-propagation](../docs/spec_topics/expressions.md) and [Errors and Results — ERR-18](../docs/spec_topics/errors-and-results/error-model.md#err-18), which every other page treats as completing the expression (a successful `Ok` unwraps in place; an `Err` early-returns from the enclosing function or loom).

Two front-ends reading only this section can therefore diverge on a single statement form. Given

```
let r = call()?
do_next()
```

one implementation treats `result?` as a complete statement and parses `do_next()` as a fresh one; another reads the trailing `?` as a continuation trigger, fuses the two lines, and then either raises a syntactic error or quietly produces `let r = call()?do_next()`. The trigger set is explicitly declared closed and the table is normative, so no implementer is licensed to silently carve postfix `?` out. The leading-operator row is unaffected (postfix `?` never starts a line); the defect is isolated to the trailing-operator row plus its mirror sentence in `lexical.md` § Statement terminators ("a trailing binary or ternary operator…"), which copied the same wording in commit `38e399f`.

## Issue introduction

**Verdict:** present-since-this-section-was-authored
**Introducing commit:** `f841b12` (2026-05-04, *spec: resolve Cluster 2 (literal sublanguage + grammar appendix)*)
**History:** The `## Newline continuation` section, including the trigger table and the operator string `+ - * / % == != < <= > >= && || ? :`, was added wholesale by `f841b12` when the grammar appendix was first introduced (`git log -S "Trailing binary or ternary operator"`, `git show f841b12 -- spec_topics/grammar.md`). The immediately prior spec text (`8eea523`, 2026-05-04) said "trailing binary or unary operator" with no `?` in the set. The same commit's authors did not own the postfix `?` operator (lives in `expressions.md` / `errors-and-results/`), so the conflation entered with the appendix and has not been touched by subsequent grammar-appendix commits (`5c8041a`, `c3b7c87`, `e1f5c2a`, etc.). The sibling continuation prose in `lexical.md` later acquired the same "trailing binary or ternary" wording in `38e399f` (2026-06-05, *trailing-continuation operator set disagreement*) by aligning to grammar.md, propagating the defect to a second page without resurfacing it.

## Solution Space

**Shape:** multiple

### Option A — Drop `?` from the trailing-operator trigger; ternary continues only via the leading-operator form

**Approach.** Remove `?` from the trailing-operator string in the `grammar.md` trigger table so the closed set reads `+ - * / % == != < <= > >= && || :`. Add a one-sentence note immediately under the table: "Trailing `?` is the postfix error-propagation operator (see [Expressions — Postfix error-propagation](./expressions.md) and [Errors and Results — ERR-18](./errors-and-results/error-model.md#err-18)) and is never a continuation trigger; the ternary `cond ? a : b` continues across a newline via the leading-operator form (`cond\n  ? a : b` or `cond ? a\n  : b`)." Mirror the change in `lexical.md` § Statement terminators inline prose so the two continuation sites do not redrift.

**Spec edits.** `grammar.md` trigger table row + new prose sentence; `lexical.md` § Statement terminators inline operator list.

**Pros.** Makes postfix `?` a statement-completing token by construction — no implementer interpretation needed. The leading-operator row already covers every legal multi-line ternary. Aligns with the dominant treatment of postfix `?` across every other page.

**Cons.** Eliminates the rarely-used `cond ?\n  a : b` style (trailing-`?` ternary split), forcing authors who want a line break around `?` to use the leading-operator form.

**Risks.** Low; the `:` half of the ternary remains a trailing trigger, so the equally common `cond ? a\n  : b` continues to parse as one statement.

### Option B — Keep `?` in the trailing list; disambiguate by parser look-ahead (require the trailing `?` to have a non-postfix follow-set)

**Approach.** Keep `?` in the trailing-operator string, but add a normative carve-out: "A trailing `?` is a continuation trigger only when it is syntactically the ternary `?` — i.e. there exists a matching `:` on the same statement (after newline continuation closes). Postfix error-propagation `?` (operand satisfies ERR-18) is never a continuation trigger." Specify a deterministic test the parser performs (e.g. one-token-back inspection: a `?` immediately following a value-returning expression that is not itself within a pending ternary becomes postfix-`?`).

**Spec edits.** `grammar.md` trigger table footnote + decision procedure; matching note in `lexical.md`; possibly a new bullet in `expressions.md` § Grammar disambiguation.

**Pros.** Preserves both ternary line-break styles (`cond ?\n  a : b` *and* `cond ? a\n  : b`).

**Cons.** Requires the lexer or parser to encode a non-trivial decision procedure (postfix-vs-ternary look-ahead) inside what is currently a closed lexical trigger table. Conflicts with the "closed trigger set, no carve-outs" framing the section opens with. Two implementers can still disagree about what counts as "syntactically the ternary `?`" in the presence of nested ternaries or in continuation-spanning bodies, leaving residual divergence.

**Risks.** Moderate. The carve-out re-opens the closed-set discipline; future ambiguities (e.g. an even-later operator overload) will accumulate similar exceptions.

### Recommendation

Adopt **Option A**. The closed-trigger section's value is precisely that a tokeniser can decide continuation from a one-line look-back without semantic context; Option B trades that property for a marginal stylistic accommodation. Pair the edit with the mirror sentence in `lexical.md` § Statement terminators (same commit) so the two continuation surfaces stay locked. Edge cases for the implementer to watch:

- The ternary `cond ? a : b` still has full multi-line support via the leading-operator form (`cond\n  ? a : b`, `cond ? a\n  : b`); only the unusual `cond ?\n  a : b` form goes away.
- The `:` operator remains in the trailing trigger set; do not also remove it.
- Both `V1a-T` and `V1a` fixtures that exercise "trailing binary/ternary operator" continuations must include a postfix-`?` vector asserting the *non-*continuation behaviour (`result?\nnext_stmt()` tokenises as two statements), and any existing trailing-`?` ternary vector must move to the leading-operator form.

## Consequence

**Severity:** correctness

Two reasonable lexer implementations diverge on a one-character source difference: whether `expr?\n…` is one statement or two. The defect is in the lexer's closed-trigger contract, so neither front-end is "wrong" against the present wording, and the divergence shows up at every postfix-`?` use site — a very common form (every `@`-query, `invoke`, or `Result` whose failure is propagated). Downstream parsing errors, error spans, and even successful-but-wrong parses are all possible outcomes.
