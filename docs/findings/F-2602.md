## Problem

The corpus specifies query-path provider-error classification two incompatible ways. QRY-10 (`query-failure-and-repair.md`) and the `binder-inference.md` cross-reference route the `sendUserMessage`-driven query paths (untyped queries, the typed-query free phase, and their subagent equivalents) through the full HTTP-status-keyed Provider error mapping table â€” which distinguishes 5xx/429 `retryable: true` from 4xx `retryable: false`, populates `http_status`, and can recognise an HTTP-400 overflow. But the only described status-capture mechanism (`ProviderResponse.status` via `onResponse`, per the `id="classifier-input-surface"` note) exists solely on the `complete()`-driven paths; the `sendUserMessage` paths return `void` and surface failure only via the trailing `assistant` message's `stopReason`/`errorMessage`. PIC-50/PIC-51/PIC-43 consequently hard-code those failures to a status-less `TransportError { http_status: null, retryable: false }`. An implementer cannot build the table-conformant classifier QRY-10 promises on paths that have no status input, and the two readings produce observably different `QueryError` values for the same failure.

## Solution approach

In `provider-error-mapping.md`'s `id="classifier-input-surface"` note, narrow the HTTP-status input (`ProviderResponse.status` via `onResponse`) to the `complete()`-driven paths and clarify that the `sendUserMessage`-driven query paths classify from the trailing `assistant` message's `stopReason`/`errorMessage` alone. Rewrite QRY-10 in `query-failure-and-repair.md` and the `binder-inference.md` cross-reference so they delegate only the overflow-signature and stop-reason arms to the table on the query paths, pinning the `http_status`-population and 5xx/429-vs-4xx `retryable`-split arms to the `complete()` paths. Specify the status-less query-path mapping PIC-50/PIC-51/PIC-43 already emit â€” `stopReason: "error"` â†’ `TransportError { http_status: null, retryable: false }`, with the existing `id="stop-reason-classification"` `stopReason: "length"` â†’ `ContextOverflowError` rule covering output-boundary overflow â€” as the intended query-path behaviour rather than a contradiction of QRY-10. State the resulting observable loss: HTTP-status-only failures are unobservable on the query paths.
