## Finding

The binder per-field rendering rule in `binder/binder-bypass-and-envelope.md` makes the ` — <description>` segment conditional on a predicate it delegates to the Descriptions page: item 4 appends the segment "iff the field carries a non-empty `description:` (per [Descriptions](../descriptions.md), after that section's normalisation)", and the fourth reference rendering pins the omitted form for "a `description:` that normalises to empty". Both phrasings assert that *Descriptions* defines when a normalised `///` description counts as empty.

It does not. The *Multi-line* rule in `descriptions.md` defines exactly three normalisation operations — consecutive `///` lines are joined with newlines, common leading whitespace is stripped, and empty `///` lines become blank lines — and the *No transformation* rule states the result is emitted byte-for-byte. None of these defines an emptiness predicate. The normalisation steps do not reduce a whitespace-only or blank-only doc comment to the empty string: `///␠␠` normalises to `"␠␠"` (common-*leading*-whitespace stripping does not remove trailing or sole-content spaces), and a run of bare `///` lines normalises to a string of one or more `\n` characters. Neither equals `""`.

Consequently "empty after normalisation" is undefined precisely at the boundary the binder rule depends on. Two reasonable implementers diverge: one treats a non-`""` whitespace/blank string as non-empty and renders ` — ␠␠` (or ` — ` followed by embedded newlines), the other treats any all-whitespace normalised result as empty and omits the segment. The same ambiguity propagates to the description-omitted reference rendering, whose pinned byte sequence (`  language (string) required`, ending after `required` with no trailing whitespace) is only reachable if the whitespace/blank-only case resolves to "empty". The spec needs a definition of the post-normalisation emptiness predicate so the consuming binder rule is well-defined.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** 53f5831 (`spec: rewrite query primitive, system prompts, and schema syntax`, 2026-05-03) introduced the three-step `///` normalisation rule without an emptiness predicate; the rule was carried verbatim into the per-topic split at fecb504 (2026-05-04) and is unchanged since.
**History:** The corpus is git-tracked. `git log -S 'become blank lines' -- spec.md docs/spec_topics/descriptions.md` shows the *Multi-line* normalisation text first appearing at 53f5831 and never subsequently defining what makes a normalised description empty. On the consumer side, the binder per-field rule has referenced a Descriptions-defined emptiness predicate since before its sharpening: the state of `spec_topics/binder.md` at 4188412~1 already read "When the field carries a non-empty `description:` (per [Descriptions](./descriptions.md)) … when absent or empty, the description segment MUST be omitted." Commit 4188412 (`resolve "Parameters block: indentation and per-field token order are not normative"`, 2026-05-07) sharpened that reference to the explicit "after that section's normalisation" / "normalises to empty" phrasing but did not backfill the missing definition in `descriptions.md`. The gap is therefore latent from the normalisation rule's inception; no single commit introduced it as a regression.

## Solution Space

**Shape:** single

### Recommendation

Add an emptiness-predicate sentence to the *Multi-line* rule (or a dedicated bullet) in `descriptions.md`, defining the post-normalisation predicate the binder already cites: after the multi-line join and common-leading-whitespace strip, a `///` description is **empty after normalisation** iff its normalised content contains no non-whitespace character — i.e. it is the empty string, contains only U+0020 SPACE / other whitespace, or consists solely of blank lines. State that a description that is empty after normalisation is treated by every consumer as if no `description:` were present (no lowering into JSON Schema `description:`, and the binder per-field ` — <description>` segment omitted per `binder-bypass-and-envelope.md` item 4).

Edge cases the implementer must watch: the predicate must cover the blank-line case (a run of bare `///` lines normalising to `"\n"`/`"\n\n"`), not just leading/trailing spaces, since the *Multi-line* rule only strips *common leading* whitespace and explicitly preserves blank lines; and the predicate is applied to the post-normalisation string, not the raw source, so a comment carrying only strippable common-leading whitespace also resolves to empty.

## Consequence

**Severity:** correctness

A whitespace-only or blank-only `///` doc comment falls into an undefined gap: one implementation emits a per-field line ending in ` — ` plus stray whitespace/newlines, another omits the segment and produces the pinned `  <wire-name> (<type>) <requirement>` form. The binder system-prompt bytes are pinned for input reproducibility, so the divergence is observable and conformance-breaking rather than cosmetic.
