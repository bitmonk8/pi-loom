## Finding

The `## Blocked obligations` section of `docs/plan.md` defines its category generically: the intro (line 208) states "Leaves whose work rests on an unresolved spec **open question**" and that a leaf "is not pickable until the **open question** is closed." The section's only entry (line 210) describes the actual blocking condition for `V9l` three times as an "**open contradiction**" — against Pi's documented teardown-and-rebind extension lifecycle. The same "open contradiction" phrasing is carried in both `V9l-session-only-degraded-branch.md` and `V9l-T-session-only-degraded-branch.md` `Deps` fields.

The section never states the relationship between the two terms — whether an "open contradiction" is a subtype of the generic "open question", or whether they are meant to denote the same thing. A maintainer reading the section cannot tell whether the intro's category term subsumes the body's instance term, and a future second entry could legitimately use either word, entrenching the drift.

This is purely a terminology reconciliation within plan prose. The corpus is otherwise consistent: every concrete reference to this `V9l` block (the section entry, the V9 slice link line, and both V9l leaf `Deps`) uses "open contradiction"; only the category intro uses the broader "open question".

## Issue introduction

**Verdict:** introduced-by-single-commit (present since section inception)
**Introducing commits:** `55ed974` — "pi-loom plan: resolve \"degraded-state obligations green over open spec contradiction\"" (2026-06-12)
**History:** `git log -S` over `docs/plan.md` shows the `## Blocked obligations` section, the intro string "unresolved spec open question", and the entry string "open contradiction" all entered in the same commit, `55ed974`, which created the section. The naming divergence has been present from the section's first commit; no later edit introduced or narrowed it. The plan corpus is git-tracked, so the history is determinate.

## Solution Space

**Shape:** single

### Recommendation

Edit the §Blocked obligations intro in `docs/plan.md` (line 208) so the relationship between the generic category term and the entry's instance term is explicit: keep "open question" as the extensible category label and state that an open contradiction (as in the `V9l` entry) is one kind of such open question. This reconciles the intro with the existing body and leaf wording while leaving the category open to future entries that block on a different kind of open question.

Because the V9 slice link line and both V9l leaf `Deps` already use "open contradiction", no leaf-file edits are required under this fix — the only change is the single intro sentence.

Edge case for the implementer: if a future maintainer prefers to narrow the category to "open contradiction" outright instead of stating the subtype relation, that is also a valid reconciliation, but it forecloses entries blocked on non-contradiction open questions — confirm no such entry is anticipated before narrowing.

## Consequence

**Severity:** cosmetic

A reader of §Blocked obligations sees the category defined generically as an "open question" but the sole entry — and both V9l leaf `Deps` — call the same condition an "open contradiction" with no stated relation, so a maintainer auditing the block, or adding a future blocked-obligation entry, must infer the two terms denote one blocking class. No leaf diverges in behaviour: the `Deps` edges (`V18c`), not the section prose, gate `V9l`'s pickability, so the impact is wording only.
