## Finding

`docs/plan.md` carries two `> **Interleave note.**` blockquotes: one under the `### V9 — Extension host integration` header and one under the `### V11 — Binder` header. The V11 blockquote states only the V11a interleaving fact — `V11a` depends on `V9b` and is a prerequisite of `V9c`/`V9i`/`V9j`, so it lands mid-V9 (seam `V9b → V11a → V9c`/`V9i`/`V9j`), and "Sequence by **Deps**, not slice number."

Every sentence of the V11 note is already present in the V9 note, which additionally documents the `V9h`/`V9g` → `V18c` brand-string dependency and the `V9l` blocked-on-clause-(a) edge. The V11 note therefore adds no information not already carried (more fully) by the V9 note; it is a strict subset stated a second time. Because the same V11a-sequencing fact lives in two places, the two surfaces can drift — and have already begun to: the V9 copy has been expanded twice since both notes were introduced together, while the V11 copy has not.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** b9b427d ("pi-loom plan: resolve \"slice-ordering narrative overpromises DAG order\"")
**History:** `git log -L` on both blockquote lines shows the V9 note and the V11 note were added together in b9b427d as near-identical text. The V9 copy was subsequently expanded in 8af3204 ("V18c bundles static gates with runtime-evidence acceptance gate") and 55ed974 ("degraded-state obligations green over open spec contradiction"), gaining the `V18c` brand-string and `V9l` blocked-edge clauses; the V11 copy was never touched after b9b427d. The duplication thus dates to the original two-note insertion, and the later one-sided expansion of the V9 copy is what turned the V11 copy into a strict subset.

## Solution Space

**Shape:** multiple

### Option A — Delete the V11 interleave blockquote

**Approach:** Remove the `> **Interleave note.** …` blockquote under `### V11 — Binder`, leaving the V9 note as the single home for the V11a-sequencing fact.

**Plan edits:** In `docs/plan.md`, delete the blockquote line under the `### V11 — Binder` header (the line beginning `> **Interleave note.** \`V11a\` depends on \`V9b\` …`).

**Spec edits:** None.

**Pros:** Eliminates the duplicated fact outright; no second copy can drift.

**Cons:** A reader scanning only the V11 section loses the local cue that V11a interleaves into V9; they must already be reading the V9 section to find it.

**Risks:** Minimal — the sequencing is enforced by each leaf's Deps regardless of the prose.

### Option B — Replace the V11 blockquote with a pointer to the V9 interleave note

**Approach:** Collapse the V11 blockquote to a one-line cross-reference to the V9 interleave note rather than restating the sequencing prose.

**Plan edits:** In `docs/plan.md`, replace the V11-section blockquote body with a pointer to the V9-section interleave note — e.g. a blockquote reading `> **Interleave note.** \`V11a\` interleaves into V9; see the [V9 interleave note](#v9--extension-host-integration). Sequence by **Deps**, not slice number.` (anchor text to match the project's existing in-file link style).

**Spec edits:** None.

**Pros:** Preserves the local navigational cue in the V11 section while keeping the normative sequencing prose in exactly one place, so the two surfaces cannot drift on the V11a fact.

**Cons:** Leaves a second blockquote to maintain (the pointer itself), and introduces an in-file anchor that must stay valid if the V9 header text changes.

**Risks:** Anchor rot if the V9 header is later retitled.

### Recommendation

Adopt Option B: replace the V11-section blockquote with a pointer to the V9 interleave note. It removes the duplicated normative prose (the actual defect) while keeping a discoverability cue for a reader who lands in the V11 section first. If the project prefers not to carry in-file anchors, Option A (outright deletion) is an acceptable fallback. Either way the V9 interleave note remains the single authoritative statement of the V11a interleaving; do not edit or trim the V9 note as part of this fix.

## Consequence

**Severity:** cosmetic

The duplication is pure organisation: both notes already say "Sequence by Deps, not slice number," so the canonical sequencing source is each leaf's Deps, not the prose. The standing risk is drift — the two copies of the V11a-sequencing fact can diverge over time (the V9 copy has already grown while the V11 copy stayed minimal), at which point a reader scanning the V11 section sees a stale or conflicting restatement.
