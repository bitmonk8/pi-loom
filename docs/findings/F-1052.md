## Finding

The version-bump procedure mandates a growing set of build-time assertions — the seven-capability literal-read on the imported `@earendil-works/pi-coding-agent` namespace, `CAPABILITY_OBLIGATIONS.length === 7`, the literal-array consistency check on the `SessionShutdownEvent['reason']` snapshot entry, the three-way `engines.node` equality, the `peerDependencies` four-entry byte-equality, the separate `typebox` `"*"` line, the provider seed-field `Api`-coverage check, the strict-capability probe, and the inventory-closure audit's wire output — all of which it routes to a single `npm test` invocation. For the *constants* these assertions read, the spec carefully pins one source of truth: "the single source-of-truth pinned-constants block in the extension module" (`version-bump-triggers.md` step 5, grouping (i)–(iv); `capability-probe.md`). For the *tests themselves*, only one — the bidirectional `SessionShutdownEvent['reason']` brand-string assertion — has a pinned location, and even there the pin is permissive (`test/` alongside the surface-inventory test, or a dedicated `src/extension/**/*.assert.ts` file co-located with the constants block, per `version-bump-triggers.md` step 5).

Every other build-time assertion the procedure names is referred to only by behaviour ("the build-time test surface", "the SDK surface-inventory tests", "the build-time `engines.node` literal-read test", "the `peerDependencies` literal-read test") with no path, module, or file-naming convention. `audit-wire-and-canary.md` confirms `npm test` is the gate but explicitly delegates harness wiring to the implementation; `audit-resolution.md` excludes `*.test.ts` / `*.spec.ts` co-located under `src/` from the inventory-closure audit, implying a `test/` convention without normatively pinning one. A reviewer reading `version-bump-step2.md` cannot tell whether the seven-capability assertion belongs in `test/`, in a sibling `src/extension/*.assert.ts`, or in a wholly new top-level test root, nor whether an implementation that places one of these assertions in a file the `npm test` configuration's glob does not actually load would still be conformant.

The gap admits a silent-pass mode: a new assertion lands in a file the `npm test` harness's configuration does not reach, the bump's "re-run the SDK surface-inventory tests" step exits green against the candidate pin, and the assertion never fires. The brand-string assertion has an explicit clause that closes this hole for itself ("the loom `tsconfig` MUST include it, and a split-tsconfig layout that excludes it is a violation of this step"); the other assertions have no analogous clause.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e5cd514` (2026-05-09, "T35 SDK capability inventory closed-set negative-direction audit" — added the seven-capability literal-read assertion); `00b2497` (2026-05-11, "Forward-compatibility-seam aggregator count is not gated by CI" — added `CAPABILITY_OBLIGATIONS.length === 7`); `75a9626` (2026-05-09, "SessionShutdownEvent.reason closed set has no build-time pin against the SDK type" — added the brand-string assertion that later acquired a placement pin); `66da202` and `05656bb` (2026-06-05, normative-MUST and verbatim-tsc fixes — added the explicit `test/` vs `src/extension/**/*.assert.ts` placement pin, but scoped it to the brand-string assertion only); `aebec4e` (2026-06-04, "Pi version-bump gate cannot detect three unstated host/provider presuppositions") and the surrounding 2026-06-04/05 sweep that landed the (h)–(an) checklist items and their associated re-validation fixtures.
**History:** Each build-time assertion was authored by a different remediation pass over a five-week window. The first wave (May 2026) established the constants block and the assertions that read it without ever specifying where the assertion files themselves live. The brand-string assertion later acquired a placement contract because its `npm run typecheck` participation forced a `tsconfig`-inclusion question the other assertions do not raise. No single commit introduced the omission; it persists because every assertion-adding pass treated "land it under `npm test`" as a plan-side concern (`H1a`/`V18c` resolve it that way) rather than a spec-side one. The defect is the missing generic placement contract, not any one commit's framing.

## Solution Space

**Shape:** single

### Recommendation

Add one spec-side paragraph — preferably under `version-bump-triggers.md` step 5 grouping (ii), where the brand-string assertion already carries its own placement pin, or as a new sub-paragraph of `version-bump-step2.md` step 2 preamble — that pins, for every build-time assertion the version-bump procedure names (step 2(a)/(b), step 3, step 4, step 6, step 7, and the inventory-closure audit), the same two-arm placement contract the brand-string assertion already enjoys:

1. The assertion's source file MUST reside either under `test/` (which is outside the inventory-closure audit's `src/**/*.ts` include glob by construction, per `audit-resolution.md`) or under `src/extension/` in a dedicated file matching the `src/extension/**/*.assert.ts` suffix (so the inventory-closure audit's existing exclusion picks it up).
2. The file MUST be reached by `npm test` (and, for the brand-string assertion, by `npm run typecheck`) on every invocation; a configuration that excludes it — a `vitest`/`tsconfig` `include` glob, a project-references split, or any equivalent — is a violation of the corresponding step rather than a silent demotion of the assertion's gate to a no-op.

The contract should explicitly cover the gates listed by behaviour today: the seven-capability presence check, the `CAPABILITY_OBLIGATIONS.length === 7` count-equality, the `SessionShutdownEvent['reason']` literal-array consistency check, the `engines.node` three-way equality, the `peerDependencies` four-entry literal-read, the `typebox` `"*"` one-line literal-read, the provider seed-field `Api`-coverage assertion, the strict-capability probe, and the inventory-closure audit's `npm test` wiring already pinned in `audit-wire-and-canary.md`. Implementer-relevant edge case: the brand-string assertion's own placement pin in `version-bump-triggers.md` step 5 should be re-pointed at the new generic paragraph rather than restating its arms, so the two pins cannot drift.

## Consequence

**Severity:** advisory

A contributor adding a new build-time assertion that the procedure mandates (e.g. when promoting a new SDK surface to `SDK_SURFACE_INVENTORY` under step 2(b) branch (2)) has no spec-side test-location obligation to land on. An assertion placed in a file the `npm test` harness's configured globs do not load passes the bump silently. The risk is bounded — the plan already routes all V18 assertions into the `npm test` configuration `H1a` scaffolds — but the spec itself does not pin the contract that backs that routing, so a later configuration drift or a parallel implementation reading only the spec is not caught.
