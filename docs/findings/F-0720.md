## Finding

The `**Loom-package implementation dependencies (loom 1.0).**` bullet in `implementation-notes.md` (anchor `#loom-package-implementation-dependencies-loom-1-0`) names three specific runtime dependencies — `semver`, `chokidar`, and `yaml` — and, for each of the two non-`semver` packages, attaches a per-package normative carve-out of the form *"`<pkg>` is neither one of the four `@earendil-works/*` peer packages nor `typebox`, so it is out of scope for the inventory-closure audit's *Target surface categories* and requires no `// allow-pi-surface:` exemption marker at any `<pkg>` import site."*

That carve-out is not new information. The general rule already lives at the audit's [*Target surface categories*](../docs/spec_topics/pi-integration-contract/audit-recognised-shapes.md#audit-target-surface-categories) anchor in `audit-recognised-shapes.md`, where category (2) is closed at exactly the four `@earendil-works/*` peer packages plus `typebox`. By construction any other package is out of scope and needs no exemption marker. The per-package paragraphs in `implementation-notes.md` re-derive that consequence by hand for each named package, with prose that is verbatim-duplicated between the `chokidar` and `yaml` paragraphs. When a third package was added in a later commit, the same paragraph shape was copy-pasted rather than collapsed; the fragility this finding flags has already manifested once in the corpus history.

Two concrete failure modes follow. First, a future adapter swap — e.g. replacing the `chokidar` `FileWatcher` adapter with a different watcher library — requires a spec edit to attach the same boilerplate to the new package's name, even though the audit semantics are unchanged. Second, a reader looking up the audit's exemption rules sees per-package assertions in `implementation-notes.md` that *appear* to be additional normative content rather than restatements of a general scope, inviting confusion when a fourth or fifth dependency is introduced silently and is mistakenly believed to need its own carve-out paragraph before it can be used.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `4cfe858` (2026-06-04, *pi-loom spec: resolve "chokidar named but never declared as a dependency"*) — introduced the original `chokidar` dependency paragraph in `pi-integration-contract.md`, including the first per-package "out of scope … no `// allow-pi-surface:` marker" sentence. The carve-out language was authored in the same paragraph as the dependency declaration rather than as a cross-reference to the general scope rule.
- `1ea0455` (2026-06-04, *pi-loom spec: resolve "DI-seam vs reference-implementation placement boundary"*) — moved the paragraph (with its per-package carve-out shape unchanged) from `pi-integration-contract.md` into `implementation-notes.md` under the loom-package-implementation-dependencies anchor, locking in the structure on the implementer-hints page.
- `1064946` (2026-06-10, *pi-loom plan: resolve "YAML frontmatter parsing mechanism is never declared as a dependency"*) — added the `yaml` dependency by copy-pasting the `chokidar` paragraph's shape, including a near-verbatim duplicate of the carve-out sentence ("Like `chokidar`, `yaml` is neither one of the four `@earendil-works/*` peer packages nor `typebox`…"). This is the multi-commit-interaction marker: the first commit established the per-package-named pattern; the third commit demonstrated the foreseeable maintenance tax by reproducing it on the next dependency that came up.

**History:** `git log -S "is out of scope for the [inventory-closure audit"` on `docs/spec_topics/` returns exactly the three commits above. `git log -S "chokidar" -- docs/spec_topics/implementation-notes.md` corroborates the move-in-place at `1ea0455` and the no-touch propagation at `1064946`. The defect is not attributable to any single commit: `4cfe858` chose a defensible local framing in a single-package context, and only the `1064946` duplication made the per-package shape's cost visible. The durable fix is to address the *pattern* (collapse the per-package claims into a single cross-reference to the existing general rule), not to revert any commit.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the `**Loom-package implementation dependencies (loom 1.0).**` bullet so that:

1. The dependency list itself (`semver`, `chokidar`, `yaml`, plus their roles and `@types/*` notes) stays in the bullet but is framed as a non-normative implementation choice — the existing "this is an implementation-side recipe rather than a normative contract" sentence already pitches `semver` this way; extend that framing to `chokidar` and `yaml`. The `dependencies` / `devDependencies` placement, the named role behind each DI seam (`FileWatcher`, YAML frontmatter parsing), and the existing cross-references to the seam definitions remain.
2. The two per-package "out of scope … no `// allow-pi-surface:` marker" sentences are removed. In their place, append one cross-reference sentence at the end of the bullet stating that the audit's scope is closed at the categories enumerated by [*Target surface categories*](./pi-integration-contract/audit-recognised-shapes.md#audit-target-surface-categories) (canonical `pi` / `ctx` carriers plus named imports from the four `@earendil-works/*` peer packages and `typebox`), and that any runtime dependency outside those categories — current or future — is therefore out of scope for the inventory-closure audit and requires no exemption marker. State this once, per-category, never per-package.
3. Preserve both existing anchors (`loom-package-implementation-dependencies-loom-1-0` and `loom-package-implementation-dependencies-v1`) verbatim so the citations from `H1a`, `V6a`, and any other leaves continue to resolve.

Edge cases the implementer must watch:

- The rewritten cross-reference must name the *category set* (four `@earendil-works/*` peers + `typebox`, plus the `pi`/`ctx` carriers), not enumerate it inline. The category set is owned by `audit-recognised-shapes.md`; restating it here re-creates the same drift surface the rewrite is meant to eliminate.
- If a future dependency happens to expose a member-access surface on a canonical-named parameter typed `ExtensionAPI` / `ExtensionContext` (vanishingly unlikely for `chokidar`/`yaml`/`semver`, but the category-(1)/(3) carrier rule keys on the textual type annotation), that surface *would* fall in scope and *would* need a marker. The cross-reference sentence should be a positive statement of the rule, not a blanket "every runtime dependency is exempt" claim that would be wrong-by-construction for such a case.
- `V18b`'s test set should gain a positive-direction case asserting that an import from a non-`@earendil-works/*` / non-`typebox` package (e.g. `import * as semver from "semver"` or `import { default as YAML } from "yaml"`) passes the audit with no marker. The test belongs in `V18b` because the property is now spec-anchored at `audit-recognised-shapes.md`'s scope paragraph rather than at the implementation-notes bullet.

## Consequence

**Severity:** advisory

The current text is internally consistent and an implementer who reads `audit-recognised-shapes.md` derives the right behaviour. The cost is editorial: every future runtime-dependency addition pays a spec-edit tax to attach a derivable carve-out, and the redundancy creates a slow drift surface (the `chokidar` and `yaml` paragraphs are already at risk of falling out of sync on any future edit that touches one and not the other).
