## Problem

`V18d` carries two independently-verifiable version-bump obligations in one leaf: the runtime-evidence acceptance gate (a green `H4a`-backed end-to-end run against the bumped Pi-SDK pin) and the revert/rollback path (on a red run, restore step 4's pin edit before merge and re-run `V18c`'s static gates against the restored prior pin, widening on red). The two are distinct code paths — a green H4a run at the bumped pin versus a simulated-red revert sequence re-running the `V18c`-gate doubles at the prior pin — sharing no fixture beyond the H4a harness and observing orthogonal outcomes. `V18d`'s **Ships when** has two arms and the paired `V18d-T` carries two corresponding red scenarios, so an agent cannot author, drive, and verify both in a single coherent TDD pass. A leaf with two Ships-when arms over two unrelated code paths exceeds the plan's leaf-as-smallest-shippable-and-testable-unit convention.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `8af3204` (2026-06-11), `dc98f78` (2026-06-17)
**History:** The plan corpus is git-tracked. `git log --follow -- docs/plan_topics/V18d-version-bump-acceptance.md` shows the leaf was created at `8af3204` (2026-06-11, "resolve 'V18c bundles static gates with runtime-evidence acceptance gate'") — itself a split that carved the acceptance gate out of `V18c`. `git show 8af3204:` confirms the leaf was born titled "runtime-evidence acceptance gate and revert path" and bundled both concerns from inception, but the revert path was then prose-only in Adds and a single Ships-when sentence, with no Tests bullet — i.e. not yet an independently-verifiable obligation. The bundling tipped into a step-atomicity defect at `dc98f78` (2026-06-17, "resolve 'revert/rollback path has no verification gate'"), whose diff added the dedicated revert-verification Tests bullet to both `V18d` and `V18d-T`, plus the second Ships-when arm gating the revert path's completion. That fix legitimately closed the revert path's verification gap; its side effect was a leaf with two fully-verified, orthogonal obligations. The durable fix is therefore to extract the now-verifiable revert path into its own leaf, not to revert `dc98f78`.

## Solution approach

Extract the revert/rollback path from `V18d` into a new paired leaf (bare + `-T`, ID picked per the plan's `V<N><letter>` scheme), leaving `V18d` scoped to the runtime-evidence acceptance gate alone. Remove the revert-path Adds clause, the second Tests bullet, and the second Ships-when arm from `docs/plan_topics/V18d-version-bump-acceptance.md`, and the second Tests bullet from `docs/plan_topics/V18d-T-version-bump-acceptance.md`; carry the revert-path obligation into the new leaf, whose **Deps** name `V18d` and `V18c`. Add the new leaf to the V18 listing in `docs/plan.md` §Vertical slices, re-point the merge-blocking facet of the `cka-19` row in `docs/plan_topics/coverage-matrix.md` to the new leaf, and, if the new leaf closes the merge-blocking MUST, add it to the **Deps** coverage-producing set in `docs/plan_topics/H5b-warn-only-canary.md`.

## Solution constraints

- Out of scope: the `V18c` static-gate definitions the revert path re-runs stay owned by `V18c` / `V18c-T` and are consumed as doubles — do not redefine them in the new leaf.
- The new leaf ID must not collide with the `V18e` proposed by the related `V18c`-too-large finding; whichever split lands second picks a distinct free letter per the `V<N><letter>` scheme.
