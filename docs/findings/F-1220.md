## Finding

`slash-invocation.md` is enumerated in `language-and-architecture.md` as a core-language page; SLSH-1 / SLSH-2 / SLSH-3 (no-params overflow, user-visible streaming, the existence of the top-level boundary system note) belong there because they describe dispatch and observability semantics of the slash surface itself. SLSH-4, SLSH-5, and the per-`kind` `SNK-a` … `SNK-k` template table do not: they specify the wire format of a Pi-extension rendering subsystem. The MUST that "renderers MUST emit the surrounding template text verbatim", the eleven normative template strings (`"loom /<name> returned Err: transport — <message>"`, etc.), the conformance hook ("Conformance tests MAY assert on the exact rendered string"), the chain-attribution suffix's leaf-first walk over `invoke_callee` hops, and the closing sentence — "Every row above emits as a `loom-system-note` carrying `details: { event: RuntimeEvent }` … See [Pi Integration Contract — System notes] for the `pi.sendMessage` call shape, the disjoint `details` payload variants, the renderer registration, and the best-effort fallback chain" — are all describing how the `loom-system-note` custom-type and its registered renderer format the always-log-set events that `pi-integration-contract/runtime-event-channel.md` already owns.

The mis-placement is bidirectional. From the core-language side, SLSH-4 / SLSH-5 inject PIC vocabulary (`pi.sendMessage`, `RuntimeEvent`, the renderer registered by the extension factory, post-`realpath` absolute paths) into a page whose other content names none of it; a reader auditing core dispatch semantics has to detour through PIC mechanics. From the PIC side, `runtime-event-channel.md` repeatedly *cross-references* the templates — the `details: { event: RuntimeEvent }` row in its per-variant `display` / `content` table cites `[Slash-Command Invocation — Top-level Err at the slash-dispatch boundary]` as the source of "the normative user-facing template", and the `display: true` / `display: false` paragraph does the same — yet the rendering rules those references resolve to live on a page whose subject is slash dispatch, not the runtime-event channel. A PIC implementer who reads only the PIC pages encounters the wire-format obligations only by following an outbound link off a sibling page.

This is the same authoring pattern flagged elsewhere in the shard for `invocation.md` ("prompt→prompt `pi.setActiveTools` snapshot/restore + LIFO/single-window invariants authored on the core-language page") and for `schema-subset.md` ("synthesised PIC names and PIC runtime diagnostic codes defined on a core-language page"): PIC-shaped normative material accumulating on core-language pages because each individual addition felt local to whichever rule already lived there.

Subsequent commits enriched the rendering subsystem in place — `8da4bc5` added the SLSH-N REQ-ID anchors, `27b2576` ("SLSH-4 covers 11 independently normative template rows under one REQ-ID") introduced the `SNK-a` … `SNK-k` inline labels and registered the `SNK` prefix on `governance/stable-inline-labels.md` against `slash-invocation.md` as the owning page, and later commits (`1ccbb75`, `40b1f17`, `0033898`) refined the chain suffix and the inline-backtick handling — but no commit re-evaluated whether the rendering subsystem belonged on the core dispatch page. ## Solution Space

**Shape:** multiple

### Option A — Relocate into `runtime-event-channel.md`

**Approach.** Move SLSH-4, SLSH-5, the `SNK-a` … `SNK-k` table, the chain-attribution worked examples, and the closing `loom-system-note` / `pi.sendMessage` paragraph into `pi-integration-contract/runtime-event-channel.md`, just after the existing per-variant `display` / `content` table (the natural anchor: that table already cites SLSH-3 as the source of the user-facing template for the `details: { event: RuntimeEvent }` top-level-cascade row). The REQ-ID anchors (`#slsh-4`, `#slsh-5`) and the `SNK-*` inline-label anchors travel verbatim with the content. GOV-16's per-page inline-label prefix table updates the `SNK` row's `Page` cell to `pi-integration-contract/runtime-event-channel.md`; the prefix and `alphabetic` tail form are unchanged. `slash-invocation.md` retains a one-line forward reference under the SLSH-3 paragraph ("The per-`kind` template format and the chain-attribution suffix are normatively defined at [Pi Integration Contract — Runtime event channel — `loom-system-note` rendering](./pi-integration-contract/runtime-event-channel.md#…)"). Every existing cross-reference to `#slsh-4` / `#slsh-5` / `#snk-*` is retargeted, including the `index/ids.md §SNK` entries flagged as a related global-traceability finding.

**Spec edits.**
- `slash-invocation.md`: delete SLSH-4, SLSH-5, the `SNK` table, the worked-example bullets, and the closing PIC paragraph; insert a one-line cross-reference under SLSH-3.
- `runtime-event-channel.md`: insert the relocated block; rewrite the SLSH-3 cross-references in the per-variant `display` / `content` table and in the `display: true` / `display: false` paragraph to point at the now-co-located anchors.
- `governance/stable-inline-labels.md`: GOV-16 table — change the `SNK` row's `Page` cell to `pi-integration-contract/runtime-event-channel.md`; update the surrounding prose ("the per-`kind` loom-system-note rendering rows `SNK-a` … `SNK-k` under [SLSH-4]") to cite the new location.
- `language-and-architecture.md`: line 56 retargets the rendering-subsystem claim.
- `cancellation.md`: retarget `#slsh-4` cross-reference.
- `_tools/` ids/xrefs: retarget all 11 `#snk-*` locators and the `#slsh-4` / `#slsh-5` locators (this option subsumes the related `index/ids.md §SNK` finding's mechanical work).
- `plan_topics/V12b-top-level-err-chain.md` and `V12b-T-…`: retarget the **Spec** field to add the new owning page.

**Pros.**
- Zero new pages; PIC implementers find the rendering subsystem inside the same page that owns the `loom-system-note` channel, the `details` payload variants, and the always-log set.
- The cross-reference traffic between SLSH-3, SLSH-4, the `details: { event }` table row, and the `display` rule collapses onto a single page, eliminating two of the bidirectional outbound links.
- GOV-16 inline-label registration is a one-cell append-only edit.

**Cons.**
- `runtime-event-channel.md` is already 40 KB and is the largest PIC page in the corpus; the relocated block adds roughly 4 KB (the table + the SLSH-5 paragraph + the worked examples), pushing it further past the 30 KB *recommended* cap enforced at `f5e89f4` ("docs: enforce 100KB hard / 30KB recommended size cap on spec set"). It stays well under the 100 KB *hard* cap.
- Co-locates two surface concerns (the always-log set / dedup / `display` rule vs. the per-`kind` template shapes) that have historically attracted independent edits, mildly raising future merge friction on this page.

**Risks.**
- Per the GOV-30 / GOV-31 lock-step concern called out in shard-01 / governance findings, any aggregator that names SLSH-4 / SLSH-5 must be touched in the same commit; this is mechanical but easy to miss.

### Option B — Relocate to a new PIC page `slash-error-rendering.md`

**Approach.** Create `docs/spec_topics/pi-integration-contract/slash-error-rendering.md`. Move SLSH-4, SLSH-5, the `SNK-a` … `SNK-k` table, the worked examples, and the closing PIC paragraph into it. Anchors travel verbatim. `slash-invocation.md` retains the same one-line forward reference described under option A. `runtime-event-channel.md` retargets its SLSH-3 / SLSH-4 cross-references to the new page. GOV-16's per-page inline-label prefix table updates the `SNK` row's `Page` cell to the new page. The `pi-integration-contract.md` hub gains a TOC entry.

**Spec edits.** Identical to option A's list except the destination is the new page rather than `runtime-event-channel.md`, plus the hub-page TOC insert.

**Pros.**
- Keeps `runtime-event-channel.md` at its current weight (40 KB), avoiding further pressure on the 30 KB recommended cap.
- The new page's subject — "how the `loom-system-note` channel renders the slash-dispatch-boundary user-facing one-liner" — is narrow and self-contained; a future addition of, e.g., parse-batch rendering rules or a binder-recovery rendering rule has a natural home.
- Separates two concerns that have historically attracted independent edits (always-log set / dedup / `display` rule on one page; per-`kind` rendering template shape on another).

**Cons.**
- Adds a small new PIC page (~5 KB), bringing the PIC sub-directory to one more file; reviewers learning the corpus structure have one more page to discover.
- Two outbound cross-references remain in `runtime-event-channel.md` (the `details: { event: RuntimeEvent }` row and the `display: true` rule), trading the slash-side ↔ PIC-side outbound for a PIC-page ↔ PIC-page outbound.

**Risks.**
- Naming the new file: `slash-error-rendering.md` (Suggested-fix wording) is reasonable but the page also owns the chain-attribution suffix for *non-error* presentation when the leaf is `cancelled`, which is a control-flow outcome rather than an error; `loom-system-note-rendering.md` would be a slightly more accurate name. This is the only naming question to settle before the move.

### Recommendation

Take **option B**. The shard's other placement findings (`schema-subset.md` synthesised names → PIC pages, `invocation.md` `pi.setActiveTools` → `tool-registration-lifetime.md`) all relocate PIC content out of core-language pages into a *dedicated PIC home*, and the same pattern fits here. The size pressure on `runtime-event-channel.md` is real enough — the page is already the heaviest in PIC — that bundling another ~4 KB of normative wire-format material into it is the worse trade against the "one more PIC file" cost of option B. Name the new page `loom-system-note-rendering.md` (slightly broader than `slash-error-rendering.md`, to legally cover the `cancelled` row and any future `Ok`-side rendering rule should one ever land). When carrying out the move, register the page in `pi-integration-contract.md`'s hub TOC in the same commit as the GOV-16 `SNK` row update and the ids/xrefs retargeting, so the GOV-9 / GOV-30 cross-link contract for the eleven `#snk-*` anchors and the two `#slsh-*` anchors never breaks during the edit. Implementer-relevant edge cases to watch: (i) the `index/ids.md §SNK` retargeting (called out as a related finding) is *the same edit* under both options and MUST land in the same commit as the relocation, not a follow-up; (ii) `cancellation.md`'s `#slsh-4` link is the only non-PIC inbound reference and is the easiest to overlook.

## Consequence

**Severity:** advisory

Two reasonable implementers reading the spec from opposite ends — one starting at `slash-invocation.md` to wire the slash surface, one starting at the PIC pages to wire `pi.sendMessage` / renderer / `RuntimeEvent` — produce the same correct runtime, because the cross-references stitch the rendering subsystem together. The cost is editorial: PIC contributors discover wire-format obligations only via outbound links, and a future change to either the always-log set or the rendering templates risks divergence between the two pages because the two authoring loci pull in opposite directions. No on-the-wire behaviour is at risk for loom 1.0.
