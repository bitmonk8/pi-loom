## Finding

`grammar.md` §*Loom literal sublanguage* / *Field rules* states that "Every declared field of the LHS schema (or the named variant schema) MUST be present. Omissions are `loom/parse/missing-object-field`; partial defaults are not supported." This rule is applied uniformly to both positions that admit a `BareObjectLit` — `params:` defaults and Pi-tool call arguments — but the LHS-schema notion behind the rule is Loom's own schema model, where optionality is encoded as `T | null` (a present field whose value may be `null`) and the explicit-`null` and absent cases are deliberately conflated (`schemas.md` §*Object schema*). Pi-tool input schemas do not share that encoding.

External Pi tools advertise their input schemas as TypeBox `TObject`s that routinely use `Type.TOptional<…>` to mark genuinely-optional fields. The pinned built-in tools demonstrate the pattern: `read` declares `offset: Type.TOptional<Type.TNumber>` and `limit: Type.TOptional<Type.TNumber>` (`@earendil-works/pi-coding-agent/dist/core/tools/read.d.ts`); `bash` declares `timeout: Type.TOptional<Type.TNumber>`; `grep`, `find`, and `ls` follow the same shape. Each of these lowers to a JSON Schema in which the optional field is in `properties` but not in `required` and is typed as a bare `number` (no `null`-union). At runtime AJV will accept the object whether the field is omitted or present-with-a-number, but it will reject `offset: null` — `null` is not a member of `{type: "number"}`. There is no mechanism for the Loom parser to mint a present-with-`null` value the tool's schema would accept.

Applied literally, the Field rules force authors to write `read({ path: "src/main.ts", offset: null, limit: null })` to call `read`, and then the runtime AJV check (`tool-calls.md` §*Argument shape*) rejects the call with `Err(CodeToolError { cause: "validation" })`. Applied non-literally — the parser silently exempts Pi-tool positions from the every-field check — `read({ path: "src/main.ts" })` parses and works, but the spec does not say so anywhere, and an implementation that takes the rule at face value is conformant to the words and broken in practice. Either reading is a real defect; the rule needs a Pi-tool carve-out that tracks the schema's `required` set instead of the full field list.

## Solution Space

**Shape:** single

### Recommendation

Split the Field rules into a position-discriminated form:

- For the `params:`-default position and for `NamedObjectLit` constructors (`Cat { ... }`), retain the current rule unchanged: every declared field MUST be present; omissions are `loom/parse/missing-object-field`. Loom-declared schemas have no absence-meaningful optionals — `T | null` is a present field with a `null` value — so the rule remains tight.
- For the Pi-tool argument position, redefine field-completeness against the tool's registered JSON Schema `required` set rather than its full `properties` set: an omitted field is `loom/parse/missing-object-field` iff the field name appears in `required` and the tool's schema is statically resolvable. Omissions of non-`required` fields MUST parse and lower as-written (the JSON object simply lacks that key); the runtime AJV check remains the safety net for both arms. When the tool's schema is not statically resolvable, defer to the same posture `type-system.md` §*Unresolvable operands* and `tool-calls.md` §*Argument shape* already pin for Pi-tool argument type-checking — the parse-time check is skipped and the runtime AJV check is the sole arbiter (Pi-tool argument type-mismatch is *always* runtime-only on Pi-tool arm per tool-calls.md, so the carve-out for `missing-object-field` is the same posture).
- Add a one-sentence note to `grammar.md` clarifying that passing `null` for a Pi-tool field whose schema does not admit `null` is a runtime AJV failure (`Err(CodeToolError { cause: "validation" })`), not a parse error — Loom's present-with-`null` encoding is not portable across the boundary. Authors who need to express "omit the field" do so by omitting it; the parser does not synthesise a `null` placeholder.

Edge cases for the implementer:
- A Pi-tool schema may declare no `required` array at all (every field optional). The carve-out must treat that as "every omission permitted at parse time," not "no field check at all" — extra-field rejection still applies via `additionalProperties: false` at runtime, but `loom/parse/missing-object-field` is unreachable in this case.
- The `.loom`-callable arm (callee declared via `params:`) is unaffected: its schema is Loom-declared, so the every-field rule continues to apply at parse time when the callee is statically resolvable, and the non-statically-resolvable case routes to runtime `InvokeInfraError { cause: "validation" }` per `tool-calls.md`.
- The diagnostic-registry row for `loom/parse/missing-object-field` currently says "Schema constructor omits a declared (required) field." Add a Pi-tool-position clause: "or a Pi-tool argument literal omits a field listed in the tool's registered `required` set (when the tool's schema is statically resolvable)."

## Consequence

**Severity:** correctness

Authors who write the most natural call form — `read({ path: "src/main.ts" })`, `bash({ command: "ls" })` — get either a parse error (`loom/parse/missing-object-field`) under a literal reading of the rule, or a runtime AJV failure (`Err(CodeToolError { cause: "validation" })`) if they comply by passing `offset: null`. Every Pi built-in with an optional input field is affected, and external Pi extensions that follow the same TypeBox convention inherit the same break. Two reasonable implementers will diverge: one will reject the omission at parse time, the other will silently exempt the Pi-tool position from the every-field rule.
