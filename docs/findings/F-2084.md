## Problem

In `binder-bypass-and-envelope.md` → *System-prompt structure (normative)*, items 2 (Description line) and 3 (Argument-hint line) gate line emission on the frontmatter value being "non-empty" and require omission when it is "absent or empty", but neither item defines "empty" for a value that is present yet whitespace-only (e.g. `description: " "`). Read literally, `" "` has length > 0 and is therefore non-empty, so items 2/3 would emit a `Description: ` / `Argument hint: ` line whose content is pure whitespace. Item 4 faces the same case and resolves it explicitly (suppress after normalisation); items 2/3 are silent, so two implementers can reasonably reach opposite conclusions. The binder system-prompt bytes feed a determinism hash, so the divergence produces different prompts — and different hashes — for the same loom.

## Issue introduction
**Verdict:** multi-commit-interaction
**Introducing commits:** 5a4ec51 — pi-loom spec: resolve "Binder system prompt information content lacks verifiable field set" (2026-05-06, Thomas Andersen); 4188412 — pi-loom spec: resolve "Parameters block: indentation and per-field token order are not normative" (2026-05-07, Thomas Andersen)
**History:** 5a4ec51 introduced items 2, 3, and 4 together in `binder.md`, all phrasing the emptiness trigger plainly ("non-empty" / "absent or empty") with no reference to normalisation — so the three items were uniform at birth. 4188412 then rewrote item 4 to qualify its predicate with "(per Descriptions, after that section's normalisation)" / "after normalisation" while leaving items 2/3 untouched, creating the inconsistency. The content was later relocated into `binder-bypass-and-envelope.md` by the f5e89f4 (2026-06-04) size-cap split, which carried the divergence forward verbatim.

## Solution approach

Clarify items 2 and 3 in *System-prompt structure (normative)* to define their emptiness predicate explicitly: a present-but-whitespace-only frontmatter value is treated as empty and suppresses the line, matching item 4's post-normalisation suppression. Define the predicate as a leading/trailing whitespace trim on the YAML scalar rather than routing through item 4's *Descriptions*-page pointer.

## Solution constraints

- Out of scope: do not route items 2/3 through the *Descriptions*-page normalisation (the `///` doc-comment join/dedent algorithm), which does not apply to frontmatter scalars — items 2/3 need their own emptiness predicate.
- The trim governs only the present-vs-omitted decision; a value that is non-empty after trimming MUST be emitted verbatim, with no interior whitespace trimming or collapsing.
