## Finding

The `YAML-shape` paragraph in `frontmatter/frontmatter-fields-b-and-templates.md` describes the two accepted spellings of `tools:` example-first ("comma-separated short form" and "YAML list form") but never states the dispatch rule the runtime must apply. The only normative-sounding clause is "The comma form is the YAML plain scalar split on commas" — which names *plain scalar* in passing, not as a discriminant, and leaves the rest of the YAML node-kind space unaddressed.

Two real input shapes fall in the gap. A flow-sequence is still a YAML sequence node:

```yaml
tools: [read, grep, bash]
```

A quoted scalar is still a YAML scalar but is not a plain scalar:

```yaml
tools: "read, grep, bash"
```

Both are well-formed YAML that an author could plausibly write (the flow-sequence is the obvious single-line alternative to the block list; the quoted scalar is what a YAML-aware editor produces when it judges the value needs quoting). The spec gives an implementer no answer for either: flow-sequence might be rejected (the prose says "YAML list form" via the block example), iterated like the block form (it is a sequence node), or fall through to "comma form" (it is a single-line value with commas). Quoted scalar might be split (it is still a scalar with commas), treated as a single entry whose name happens to contain commas and whitespace (so it is `loom/load/unknown-tool` against `read, grep, bash`), or rejected. Each plausible reading produces a different load-time outcome on the same source file.

The fix is small but the rule must be explicit: the discriminant is the parsed YAML node kind, and the behaviour on every kind in the (scalar, sequence) cross of (plain, quoted) × (block, flow) must be pinned — or the rejected kinds named with a diagnostic.

## Solution Space

**Shape:** single

### Recommendation

Replace the lead clause of the `YAML-shape` paragraph with an explicit, node-kind-keyed dispatch rule, then enumerate the four cross-product cells so no input shape is left undefined:

- **Dispatch on parsed YAML node kind.** A YAML **scalar** node (plain, single-quoted, or double-quoted) is the comma form: split on `,`, trim leading/trailing whitespace from each resulting entry, and apply the per-entry grammar from [Parameters and Frontmatter — `tools`](frontmatter-fields-a.md#tools). A YAML **sequence** node (block or flow) is the list form: one entry per sequence item, each parsed by the same per-entry grammar. `tools: []` and `tools: ""` are both equivalent to absent `tools:` (already covered for `[]` in `frontmatter-fields-a.md` — extend that sentence to cover the empty-scalar case in the same clause). Any other YAML node kind (mapping, alias, tag) is `loom/load/tools-wrong-shape` — a new load-time error that surfaces the actual node kind in the diagnostic message.
- **State that quoting is transparent.** A double-quoted scalar `"read, grep"` and the plain scalar `read, grep` produce the same callable set; YAML's quoting choices never change the dispatch. Likewise the block sequence and the flow sequence (`[read, grep]`) are equivalent at the loom layer.
- **State that the per-entry grammar is the only validator of each resulting string.** Diagnostic codes (`loom/load/unknown-tool`, `loom/load/unresolvable-loom-path`, `loom/load/invalid-tool-rename`) fire on the post-split / post-iterate entry text regardless of which YAML shape produced it — re-emphasising the existing "same load diagnostic … regardless of which spelling produced it" sentence, but now with a real definition of "spelling" behind it.

Edge cases the implementer must handle:

- A scalar entry containing an explicit YAML tag (`!!str "read"`) is rare but well-formed; treat it as a scalar of the resolved type for dispatch purposes (the tag does not change node kind).
- A flow-sequence item that is itself a mapping (`tools: [{read: yes}]`) is `loom/load/tools-wrong-shape` against the *item*, not against the outer sequence — same diagnostic code, but the placeholder identifies the item position.
- The `loom/load/tools-wrong-shape` code must be added to the diagnostic-code registry (`docs/spec_topics/diagnostics/code-registry-load.md`) and to the closing-gate inventory.

Whether to introduce a new diagnostic code or to fold mapping/alias rejection into an existing code (e.g. `loom/load/frontmatter-value-out-of-range`) is an implementer-neutral choice; the recommendation introduces a dedicated code for symmetry with the other `loom/load/*` codes in the surrounding paragraph.

## Consequence

**Severity:** correctness

Two implementers reading the current paragraph will diverge on what `tools: [read, grep]` and `tools: "read, grep"` mean — one may accept both via "split on commas", another may iterate the flow-sequence and reject the quoted scalar, a third may accept the quoted scalar as a single Pi-tool name and fire `loom/load/unknown-tool`. The same `.loom` file therefore loads, partially loads, or fails to load depending on whose runtime sees it; the divergence is silent at authoring time because both spellings are conventional YAML.
