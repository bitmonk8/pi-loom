## Finding

`placeholder-rendering-b.md` §7's *Parse-time literal-value* sub-rule pins `<value>` rendering for six parse-time rows (`loom/load/unknown-mode-value`, `loom/load/unknown-methodology-value`, `loom/load/unknown-bind-context-value`, `loom/load/model-unresolved`, `loom/parse/duplicate-enum-value`, `loom/parse/duplicate-discriminator-value`) to category 5's `<key>` shape — bare when identifier-shaped, double-quoted otherwise — **except** for "YAML scalars with no enclosing source quoting (e.g. `mode:` values)", which render unquoted regardless of identifier shape. The carve-out's discriminator is the *source-quoting form* of the scalar, but YAML has five scalar styles and only one of them ("plain") is unambiguously "no enclosing source quoting". The other four — single-quoted (`'…'`), double-quoted (`"…"`), block literal (`|`), and block folded (`>`) — all carry distinct enclosing source markers, and the rule says nothing about how to render `<value>` when the offending scalar uses any of them.

Two readings of the carve-out are equally defensible and produce non-byte-identical messages. An author who writes `mode: "wibble"` (double-quoted source) triggers `loom/load/unknown-mode-value`; one renderer reads the carve-out as "the source had quoting, so fall through to the §5 `<key>` rule" and emits `unknown 'mode:' value '"wibble"'`, another reads it as "render the parsed scalar value" and emits the bare `unknown 'mode:' value 'wibble'`. The §7 closure premise — *every* `<…>` token renders byte-identically across conformant implementations — is broken at the V1 surface, not just at some hypothetical future edge: `mode:`, `respond_repair.methodology:`, and `bind_context:` are all routinely quoted by authors who hit these errors.

The fix is to replace the source-quoting test with one that lifts cleanly to all five YAML scalar styles. The suggested wording — "YAML plain scalars (no surrounding `'`, `"`, `|`, or `>`)" — does exactly that: it names the YAML 1.2 scalar-style classification explicitly, leaving the four non-plain styles to fall through to category 5's `<key>` rule (bare-when-identifier-shaped, double-quoted otherwise), which is already well-defined on the *parsed* scalar value irrespective of source style.

## Solution Space

**Shape:** single

### Recommendation

Replace the carve-out clause in `placeholder-rendering-b.md` §7's *Parse-time literal-value* bullet with the wording the original finding suggested:

> — except that **YAML plain scalars** (no surrounding `'`, `"`, `|`, or `>`) render unquoted regardless of identifier shape.

Two implementer-relevant notes the leaf and test author must observe:

1. The replacement pivots the discriminator from "presence/absence of source quoting" (currently underdefined) to "YAML scalar style is `plain`" (YAML 1.2 §8.1.2.1, an unambiguous five-way classification covering plain / single-quoted / double-quoted / block-literal / block-folded). The four non-plain styles fall through to category 5's `<key>` rule, which already operates on the *parsed* scalar value and is well-defined regardless of how the source encoded it.
2. `V7c-T`'s `DIAG-4` vectors for at least one parse-time row (the natural choice is `loom/load/unknown-mode-value`, since `mode:` is the example the rule itself cites) must add four boundary vectors — one each for `mode: wibble` (plain → bare), `mode: 'wibble'` (single-quoted → bare per §5, identifier-shaped), `mode: "wibble"` (double-quoted → bare per §5, identifier-shaped), and `mode: "my-mode"` (double-quoted, non-identifier-shaped → double-quoted per §5) — to lock the byte-identical rendering in for each style. Block scalars (`|`/`>`) are syntactically valid for `mode:` but practically unidiomatic; one vector against a row whose value field admits multi-line content (e.g. a future row, or none in V1) is sufficient and need not be written for V1.

## Consequence

**Severity:** correctness

Two conformant implementations targeting the same author input (`mode: "wibble"`, `respond_repair.methodology: |…`, etc.) can produce non-byte-identical diagnostic messages, violating the §7 closure premise that every enumerated placeholder renders byte-identically. The `DIAG-4` per-category test vectors in `V7c-T` cannot be written unambiguously against the current rule, and conformance tests would diverge between implementations that read the carve-out as "source had quoting → §5 fall-through" versus "render parsed scalar value".
