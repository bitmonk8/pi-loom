## Finding

The closing line of the *Tool-registration lifetime and visibility* section in `tool-registration-lifetime.md` reads: *"No Pi-side change is required for loom 1.0; if Pi later adds `pi.unregisterTool`, the registration cache becomes the natural site to call it from on extension teardown without altering the per-mode wiring rule above."*

The first clause ("No Pi-side change is required for loom 1.0") is legitimate normative scope-setting — it confirms the section's design is self-contained at the loom 1.0 pin. The second clause is speculative implementation guidance for a hypothetical future Pi API (`pi.unregisterTool`) that the spec elsewhere explicitly establishes does **not** exist ("Pi's tool registry has no `unregisterTool` API"). It tells a future maintainer where they *would* wire a call that no loom 1.0 implementer can write. It constrains nothing in the shipping contract and adds carry-forward maintenance surface that becomes inaccurate the moment Pi's API changes.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 27ec53d (`Resolve Cluster-A: per-mode tool registration`, 2026-05-04)
**History:** `git log -S "if Pi later adds"` shows exactly three commits touching the string: `27ec53d` introduces the sentence on the original `docs/spec.md`; `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set") shards the spec and carries the sentence verbatim into `docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md`; `7dd8b27` is a review pass that does not modify it. The defect therefore entered with the Cluster-A resolution and has been present in this page since the page itself was created.

## Solution Space

**Shape:** single

### Recommendation

Delete the speculative second clause so the sentence terminates at the normative scope statement:

> No Pi-side change is required for loom 1.0.

Retain the first clause unchanged. No other text in the section depends on the deleted clause — the absence of a `pi.unregisterTool` API and the permanence of cached registrations are already stated normatively in the opening sentence and the preceding consequences bullets (`pi.getAllTools()` accumulation, the `/reload` cache-drop bullet).

## Consequence

**Severity:** cosmetic

No observer is affected. The clause is an informative aside that guides nothing in the loom 1.0 implementation; its only cost is reader distraction and a maintenance liability if Pi's tool-registry API later diverges from the speculation.
