## Finding

`SUBS-1` is a live numbered executable REQ-ID: `SUBS` is a registered REQ-ID prefix for `schema-subset.md` (`spec_topics/governance/req-id-prefix-table-active-a.md` §REQ-ID prefix table), and `schema-subset.md` carries the dual-form anchor `<a id="subs-1"></a> **SUBS-1.**` at its *Lowering Algorithm* (the *Union of primitives only* rule). It is therefore matched by the GOV-3 primary extractor's live-prefix alternation and is part of the spec REQ-ID universe the `H5a` closing gate reconciles against `coverage-matrix.md`.

`coverage-matrix.md`'s *Numbered REQ-IDs (runtime obligations)* table has no `SUBS-1` row. `schema-subset.md` appears only in the *Code-keyed obligation areas* table as `cka-10` (closing leaves `V5d`, `V5e`, `V5f`) — a table whose own header declares it is for pages that "own their obligations through `loom/parse/*` … diagnostic codes **rather than** numbered `PREFIX-N` REQ-IDs." A `cka-<n>` row is not a numbered-REQ-ID mapping, so it does not supply `SUBS-1` a per-REQ-ID closing mapping. `SUBS-1` is thus an unmapped executable REQ-ID.

The page legitimately carries both kinds of obligation, and the matrix already has precedent for splitting them: `control-flow.md` has `CTRL-1` in the Numbered table and a separate `cka-5` row for its reject codes (whose annotation explicitly notes "CTRL-1 closes in the numbered table above"). `schema-subset.md` is the same shape but missing its numbered-table half. The union-of-primitives lowering behaviour itself is implemented — it is part of the lowering pass owned by `V5f` — so this is an enumeration / closing-gate mapping gap, not an unimplemented requirement.

**History:** `git log -S 'id="subs-1"' -- docs/spec_topics/schema-subset.md` reports `6892aa3` as the sole commit that introduced the `SUBS-1` numbered REQ-ID anchor; that commit touched only `docs/spec-plan-review.md` and `docs/spec_topics/schema-subset.md` (`git show --stat 6892aa3`), leaving `coverage-matrix.md` untouched. Before `6892aa3`, `schema-subset.md` carried no numbered `SUBS-N` REQ-ID, so its coverage row was correctly code-keyed-only — first authored at `c6a664e` ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10) as `schema-subset.md (SUBS) | V5d, V5e`, then assigned the `cka-10` token at `7259777` ("pi-loom plan: resolve \"code-keyed obligation areas lack a canonical citable token\"", 2026-06-18). `git log -G 'SUBS-1' -- docs/plan_topics/coverage-matrix.md` returns no commit, confirming a `SUBS-1` numbered-table row has never existed. The defect entered the corpus the instant `6892aa3` made `SUBS-1` a live numbered REQ-ID without the matching plan-side numbered-table row.

## Solution Space

**Shape:** multiple

This finding carries two obligations in different files: (1) map `SUBS-1` in the Numbered REQ-IDs table, and (2) supply the inline `SUBS-1` citing test the mapped-REQ-ID citing-test arm then requires. Mapping the row alone would clear the unmapped-executable-REQ-ID arm but trip the mapped-REQ-ID-with-no-citing-test arm, so both are required. They are decomposed below; the Recommendation fixes their order.

### Option A — Map `SUBS-1` in the Numbered REQ-IDs table

**Approach.** Add a `SUBS-1` row to the *Numbered REQ-IDs (runtime obligations)* table in `coverage-matrix.md`, mapping it to its closing leaf `V5f` (the leaf that owns the lowering pass), mirroring the `control-flow.md` precedent where the page carries both a numbered-table row and a `cka` row.

**Plan edits.** In `docs/plan_topics/coverage-matrix.md`, add the row `| SUBS-1 | `` `V5f` `` |` to the Numbered REQ-IDs table; the leaf-ID token `V5f` must be backtick-delimited per the *Closing-leaf column authoring* rule. Optionally append a note to the `cka-10` row's annotation that "SUBS-1 closes in the numbered table above", matching the existing `cka-5` / `CTRL-1` cross-note.

**Spec edits.** None.

**Pros.** Closes the unmapped-executable-REQ-ID gap; the live-corpus arm reconciles `SUBS-1` once `H6a` binds.

**Cons.** Once mapped, `SUBS-1` acquires the citing-test obligation that Option B satisfies.

**Risks.** A bare (un-backticked) `V5f` token is dropped by the `H5d` tokeniser and emptied into a spurious gate failure.

### Option B — Add the inline `SUBS-1` citing test to `V5f` / `V5f-T`

**Approach.** Add a Tests bullet that asserts the SUBS-1 union-of-primitives lowering and cites the literal token `SUBS-1` inline, authored red-first in `V5f-T` then closed by `V5f`, so the `H5a` mapped-REQ-ID citing-test arm (and the `H5f` per-facet arm) finds a citing test.

**Plan edits.** In `docs/plan_topics/V5f-T-subset-lowering-hash.md` add a red-first Tests bullet, and in `docs/plan_topics/V5f-subset-lowering-hash.md` the corresponding impl Tests bullet, asserting SUBS-1's lowered form (e.g. a `string | null` union lowers to the multi-type-array form `{"type": ["string", "null"]}` and a union with a non-primitive arm lowers to `{"anyOf": […]}`), with `SUBS-1` named inline in the bullet text; reflect the assertion in `V5f`'s Ships when.

**Spec edits.** None.

**Pros.** Satisfies the citing-test arm; gives the already-implemented SUBS-1 lowering explicit coverage evidence.

**Cons.** Requires a paired-leaf edit (`-T` then impl) per the per-phase TDD ritual.

**Risks.** The recogniser is a best-effort inline-`PREFIX-N` scan, so the bullet must carry the literal `SUBS-1` token (not a `SUBS code-keyed area` paraphrase) to certify.

### Recommendation

Apply Option A first — the numbered-table row bounds the mapping and is the precondition the citing-test arm reconciles against — then Option B on that stable baseline, so the `V5f` / `V5f-T` citing test lands against an already-present `SUBS-1` mapping. Both must be in place before the `H6a` live-corpus activation flips the gate to hard-fail. Edge cases: backtick-delimit the `V5f` token in the new cell, and author the SUBS-1 assertion red-first in `V5f-T` per the TDD ritual.

## Consequence

Once the terminal `H6a` activation flips the `H5a` closing gate to its live-corpus footing, the live spec REQ-ID set reconciled against `coverage-matrix.md` reds with an unmapped-executable-REQ-ID failure for `SUBS-1`, blocking the loom 1.0 release gate from going green even though the behaviour ships in `V5f`.
