## Finding

`SUBS-1` is a live numbered executable REQ-ID: `SUBS` is a registered REQ-ID prefix for `schema-subset.md` (`spec_topics/governance/req-id-prefix-table-active-a.md` §REQ-ID prefix table), and `schema-subset.md` carries the dual-form anchor `<a id="subs-1"></a> **SUBS-1.**` at its *Lowering Algorithm* (the *Union of primitives only* rule). It is therefore matched by the GOV-3 primary extractor's live-prefix alternation and is part of the spec REQ-ID universe the `H5a` closing gate reconciles against `coverage-matrix.md`.

`coverage-matrix.md`'s *Numbered REQ-IDs (runtime obligations)* table has no `SUBS-1` row. `schema-subset.md` appears only in the *Code-keyed obligation areas* table as `cka-10` (closing leaves `V5d`, `V5e`, `V5f`) — a table whose own header declares it is for pages that "own their obligations through `loom/parse/*` … diagnostic codes **rather than** numbered `PREFIX-N` REQ-IDs." A `cka-<n>` row is not a numbered-REQ-ID mapping, so it does not supply `SUBS-1` a per-REQ-ID closing mapping. `SUBS-1` is thus an unmapped executable REQ-ID.

The page legitimately carries both kinds of obligation, and the matrix already has precedent for splitting them: `control-flow.md` has `CTRL-1` in the Numbered table and a separate `cka-5` row for its reject codes (whose annotation explicitly notes "CTRL-1 closes in the numbered table above"). `schema-subset.md` is the same shape but missing its numbered-table half. The union-of-primitives lowering behaviour itself is implemented — it is part of the lowering pass owned by `V5f` — so this is an enumeration / closing-gate mapping gap, not an unimplemented requirement.

The defect entered the corpus the instant `6892aa3` made `SUBS-1` a live numbered REQ-ID without the matching plan-side numbered-table row.

## Solution Space

**Shape:** single

### Recommendation

`schema-subset.md` carries both a numbered REQ-ID (`SUBS-1`) and code-keyed obligations, so it needs a numbered-table mapping in addition to its existing `cka-10` row. Make these two edits, in this order — add the numbered-table row first, then add the citing test against that mapping:

1. In `docs/plan_topics/coverage-matrix.md`, add a `SUBS-1` row to the *Numbered REQ-IDs (runtime obligations)* table, mapping it to its closing leaf `V5f` (the leaf that owns the lowering pass): the row reads `| SUBS-1 | `` `V5f` `` |`. The leaf-ID token `V5f` must be backtick-delimited per the *Closing-leaf column authoring* rule — a bare, un-backticked token is dropped by the `H5d` tokeniser and emptied into a spurious gate failure. Append a note to the `cka-10` row's annotation that "SUBS-1 closes in the numbered table above", matching the existing `cka-5` / `CTRL-1` cross-note.

2. Add an inline `SUBS-1` citing test, authored red-first in `docs/plan_topics/V5f-T-subset-lowering-hash.md` and then closed by `docs/plan_topics/V5f-subset-lowering-hash.md`. In `V5f-T-subset-lowering-hash.md` add a red-first Tests bullet, and in `V5f-subset-lowering-hash.md` add the corresponding impl Tests bullet, asserting `SUBS-1`'s lowered form (e.g. a `string | null` union lowers to the multi-type-array form `{"type": ["string", "null"]}`, and a union with a non-primitive arm lowers to `{"anyOf": […]}`). Name the literal `SUBS-1` token inline in the bullet text — the recogniser is a best-effort inline-`PREFIX-N` scan, so a `SUBS code-keyed area` paraphrase will not certify. Reflect the assertion in `V5f`'s Ships when.

Both edits must be in place before the `H6a` live-corpus activation flips the gate to hard-fail.

## Consequence

**Severity:** correctness

Once the terminal `H6a` activation flips the `H5a` closing gate to its live-corpus footing, the live spec REQ-ID set reconciled against `coverage-matrix.md` reds with an unmapped-executable-REQ-ID failure for `SUBS-1`, blocking the loom 1.0 release gate from going green even though the behaviour ships in `V5f`. Until that flip the gate is seeded-fixture-only and does not catch it, so the gap stays latent through the entire vertical-slice build and surfaces only at release activation.
