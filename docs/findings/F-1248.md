## Problem

`SUBS-1` is a live numbered executable REQ-ID — `SUBS` is a registered prefix and `schema-subset.md` carries the dual-form `SUBS-1` anchor at its Lowering Algorithm — so it is part of the spec REQ-ID universe the `H5a` closing gate reconciles against `coverage-matrix.md`. The *Numbered REQ-IDs (runtime obligations)* table in `coverage-matrix.md` has no `SUBS-1` row; `schema-subset.md` appears only via the `cka-10` *Code-keyed obligation areas* row, which by its own header is not a numbered-`PREFIX-N` mapping and supplies `SUBS-1` no per-REQ-ID closing mapping. `SUBS-1` is thus an unmapped executable REQ-ID. Once the terminal `H6a` activation flips the `H5a` gate to its live-corpus footing, the gate reds with an unmapped-executable-REQ-ID failure for `SUBS-1`, blocking the loom 1.0 release gate even though the lowering behaviour ships in `V5f`.

## Issue introduction

**Verdict:** single-commit (spec-edit left the plan coverage-matrix unsynchronised)
**Introducing commit:** `6892aa3` — "pi-loom spec-plan: fix F-0786 — pin primitive-union/nullability lowered form (SUBS-1) for canonical-hash byte-identity" (2026-06-20)
**History:** `git log -S 'id="subs-1"' -- docs/spec_topics/schema-subset.md` reports `6892aa3` as the sole commit that introduced the `SUBS-1` numbered REQ-ID anchor; that commit touched only `docs/spec-plan-review.md` and `docs/spec_topics/schema-subset.md` (`git show --stat 6892aa3`), leaving `coverage-matrix.md` untouched. Before `6892aa3`, `schema-subset.md` carried no numbered `SUBS-N` REQ-ID, so its coverage row was correctly code-keyed-only — first authored at `c6a664e` ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10) as `schema-subset.md (SUBS) | V5d, V5e`, then assigned the `cka-10` token at `7259777` ("pi-loom plan: resolve \"code-keyed obligation areas lack a canonical citable token\"", 2026-06-18). `git log -G 'SUBS-1' -- docs/plan_topics/coverage-matrix.md` returns no commit, confirming a `SUBS-1` numbered-table row has never existed. The defect entered the corpus the instant `6892aa3` made `SUBS-1` a live numbered REQ-ID without the matching plan-side numbered-table row.

## Solution approach

Add a `SUBS-1` row to the *Numbered REQ-IDs (runtime obligations)* table in `coverage-matrix.md`, mapping it to its closing leaf `V5f`, and cross-note the existing `cka-10` row that `SUBS-1` closes in the numbered table (matching the established `cka-5` / `CTRL-1` cross-note). Add a `SUBS-1` citing test red-first in `V5f-T-subset-lowering-hash.md` and close it in `V5f-subset-lowering-hash.md`, naming the literal `SUBS-1` token inline so the inline-`PREFIX-N` recogniser certifies it, and reflect the assertion in `V5f`'s *Ships when*.

## Solution constraints

- The spec is read-only for this fix — `schema-subset.md`'s `SUBS-1` anchor is correct; the gap is the plan coverage-matrix / `V5f` leaf sync only.
- Leaf-ID tokens in the *Numbered REQ-IDs* table's closing-leaf cell must be backtick-delimited per the *Closing-leaf column authoring* rule — the `H5d` tokeniser drops a bare token.
