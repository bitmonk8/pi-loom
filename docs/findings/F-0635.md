## Finding

Step 0 (b) of the capability probe (`docs/spec_topics/pi-integration-contract/capability-probe.md`) enumerates nine `AbortSignal` / `AbortController` members in a table but specifies neither the order in which the probe walks the table nor a first-failure short-circuit. Step 0 (c), immediately below, pins both explicitly — "in the listed order … stopping at the first member whose check returns false" — and justifies that pin with: "This fixed iteration order is normative: it makes the failing member deterministic, so a conformance test that removes exactly one of the [members] can predict which member the runtime names in the emitted payload."

The same justification applies to (b) but the discipline is missing. On a host where two step-(b) members fail simultaneously — for example a polyfilled `AbortSignal` lacking both `AbortSignal.any` and `AbortSignal.timeout`, or a polyfill missing both `aborted` and `reason` — two conformant implementations may emit different `loom/load/host-incompatible` diagnostics. When the failing members are of different `kind`s (e.g. one `static-method` and one `prototype-property`), the rendered `<observed>` / `<required>` substrings themselves diverge (`"undefined" / "function"` vs. `"absent" / "present"`), so a conformance test that constructs such a host cannot reliably assert which substring pair the runtime emits. The `<observed>` substring on (b) is also the only signal the diagnostic carries about which member failed — step (b) does not pin a sibling `details.member` field the way (c) does — so an unpinned iteration order leaves the diagnostic less debuggable on (b) than on (c) for the same class of multi-failure host.

The asymmetry with (c) is gratuitous: the table in (b) is already a fixed authored sequence, the short-circuit invariant in *On failure: refusal and diagnostic* requires exactly one emitted diagnostic across the whole probe, and the per-member rule is uniform. Pinning the iteration as table-order with first-failure short-circuit changes no observable behaviour on single-failure hosts and removes the divergence on multi-failure hosts.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `126ff8a` ("pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B", 2026-05-06) and `a33bd49` ("pi-loom spec: resolve 'host-incompatible sdk-capability-missing discriminator'", 2026-06-03)
**History:** `126ff8a` rewrote Step 0 and introduced the nine-member `AbortSignal` / `AbortController` per-member-kind table under (b). At that point neither (b) nor (c) pinned an iteration order or short-circuit: the diagnostic payload for (c) carried only `kind`, `observed`, `required`, so an unpinned order had no observable consequence on either step. `a33bd49` later refined (c) to emit a sibling `details.member = "<path>"` field naming the failing member, and at the same time added the now-quoted "in the listed order … stopping at the first member whose check returns false … makes the failing member deterministic" discipline to (c) so the new `details.member` would be testable. (c)'s justification ports verbatim to (b), but (b) was not amended in lockstep; the divergence is the residue of that one-sided refinement. The defect is the interaction, not either commit in isolation — `126ff8a` was internally consistent, and `a33bd49` correctly tightened (c).

## Solution Space

**Shape:** single

### Recommendation

In Step 0 (b), after the per-member-kind table and the `prototype-property` paragraph, add a normative sentence pinning the iteration discipline parallel to (c)'s: the probe checks the table members in the listed order (top-to-bottom: `AbortController` global, `AbortSignal` global, `AbortController.prototype.abort`, `AbortSignal.any`, `AbortSignal.timeout`, `AbortSignal.prototype.throwIfAborted`, `AbortSignal.prototype.addEventListener`, `AbortSignal.prototype.aborted`, `AbortSignal.prototype.reason`) and stops at the first member whose `kind`-keyed check (`typeof === "function"` for `constructor` / `static-method` / `prototype-method`, `"<name>" in AbortSignal.prototype` for `prototype-property`) returns false. State the justification once: this makes the failing member's `kind`-derived `<observed>` / `<required>` substring pair deterministic, so a conformance test that removes a specific member can predict the rendered diagnostic.

Implementer-relevant notes:

- The table's authored top-to-bottom order should be the normative sequence; do not re-sort by `kind`. This keeps the order the test source-of-truth file can read directly.
- The probe's existing **PIC-3** invariant ("MUST NOT use any member it is itself checking") still binds; the iteration-order pin imposes no new constraint that conflicts with it.
- Step (b)'s diagnostic does not need a sibling `details.member` field added in this fix — the iteration-order pin is sufficient to make the existing `<observed>` / `<required>` substrings deterministic. Whether to mirror (c)'s `details.member` addition for (b) is an orthogonal question and out of scope for this finding.
- The `placeholder-rendering-b.md` per-`kind` enumeration table for `abortsignal-shape` already pins two rows keyed on the member's kind; with the iteration order pinned, a conformance test for a multi-failure host can anchor on the row corresponding to the earliest-listed failing member without further table changes.

## Consequence

**Severity:** correctness

Two reasonable implementations of the (b) probe will diverge on multi-failure hosts: which member's `kind`-derived `<observed>` / `<required>` pair gets rendered, and (if a future spec change adds a `details.member` to (b) the way it exists on (c)) which member name is reported. Conformance tests that exercise multi-failure step-(b) hosts cannot be written today against the spec as written.
