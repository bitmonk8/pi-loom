## Problem

Step 0 (b) of the capability probe enumerates nine `AbortSignal` / `AbortController` members in a table but specifies neither the order in which the probe walks the table nor a first-failure short-circuit, whereas Step 0 (c) immediately below pins both explicitly. On a host where two step-(b) members of different `kind`s fail simultaneously, two conformant implementations may render different `<observed>` / `<required>` substring pairs in the `loom/load/host-incompatible` diagnostic, so a conformance test that constructs such a host cannot assert which pair the runtime emits. The `<observed>` substring is the only signal step (b) carries about which member failed — (b) pins no sibling `details.member` field the way (c) does — so the unpinned order leaves (b)'s diagnostic less debuggable than (c)'s for the same class of multi-failure host.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `126ff8a` ("pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B", 2026-05-06) and `a33bd49` ("pi-loom spec: resolve 'host-incompatible sdk-capability-missing discriminator'", 2026-06-03)
**History:** `126ff8a` rewrote Step 0 and introduced the nine-member `AbortSignal` / `AbortController` per-member-kind table under (b). At that point neither (b) nor (c) pinned an iteration order or short-circuit: the diagnostic payload for (c) carried only `kind`, `observed`, `required`, so an unpinned order had no observable consequence on either step. `a33bd49` later refined (c) to emit a sibling `details.member = "<path>"` field naming the failing member, and at the same time added the now-quoted "in the listed order … stopping at the first member whose check returns false … makes the failing member deterministic" discipline to (c) so the new `details.member` would be testable. (c)'s justification ports verbatim to (b), but (b) was not amended in lockstep; the divergence is the residue of that one-sided refinement. The defect is the interaction, not either commit in isolation — `126ff8a` was internally consistent, and `a33bd49` correctly tightened (c).

## Solution approach

In Step 0 (b)'s `AbortSignal` / `AbortController` shape section, add a normative iteration-discipline sentence parallel to (c)'s: the probe checks the table members in the table's authored top-to-bottom order and stops at the first member whose `kind`-keyed check returns false. State the justification once — this makes the failing member's `kind`-derived `<observed>` / `<required>` substring pair deterministic, so a conformance test that removes a specific member can predict the rendered diagnostic.

## Solution constraints

- Out of scope: do not add a sibling `details.member` field to step (b)'s diagnostic; pinning the iteration order alone makes the existing `<observed>` / `<required>` substrings deterministic, and mirroring (c)'s `details.member` is an orthogonal change.
