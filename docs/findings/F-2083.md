## Finding

The inventory-closure audit's *Wire serialisation* contract (`audit-wire-and-canary.md`) emits one record per line, terminated by a single `\n`, with the four fields (`path`, `line`, `symbol`, `proposed-resolution`) packed into tab-delimited (U+0009) segments. It asserts that the four field values "MUST NOT contain the ASCII tab character … file paths are tab-free by their pinned shapes." That clause treats path tab/newline-freedom as a *derived property* of the scope definition rather than as something the scope definition actually guarantees.

But *Audit scope* (`audit-resolution.md`) defines the audited source tree purely as the closed set matching the include glob `src/**/*.ts` minus enumerated exclusions. The scope paragraph pins case-sensitivity, symlink handling, extension matching, and UTF-8 encoding — but it never constrains the in-scope file *path* to be free of an ASCII tab or a line terminator. On a POSIX filesystem, both characters are legal in filenames, so a file named e.g. `src/foo<TAB>bar.ts` or one whose name embeds a newline matches the glob and is walked. There is no precondition asserting such paths cannot exist, and no detection rule routing them anywhere.

The result is an unbridged contradiction between the two sections: the wire contract assumes a property that the scope contract does not establish, and nothing in either page closes the gap. When the assumed-impossible path occurs, an embedded tab injects a spurious segment (a parser splitting on tab recovers the wrong field count) and an embedded newline splits one logical record across two physical lines — silently corrupting the machine-parseable stdout stream a CI parser is the sole permitted consumer of.

## Solution Space

**Shape:** multiple

### Option A — Detect and fail closed under an `infra`-class discriminator

**Approach:** Add a rule to *Wire serialisation* (and/or *Infrastructure-failure handling*) that a walked in-scope path containing an ASCII tab or any line terminator MUST surface as an infrastructure failure under a stable `audit/infra/<family>/<symptom>` discriminator (non-zero `npm test`), rather than being emitted into a record field.

**Spec edits:** In `audit-wire-and-canary.md`, replace the assertion that "file paths are tab-free by their pinned shapes" with a normative detection obligation; register the `infra`-class `<family>`/`<symptom>` token alongside the existing infrastructure-failure surface.

**Pros:** Aligns with the audit's existing fail-closed philosophy and the surface-set-closure guarantee — the offending file cannot be silently skipped. Keeps the gap visible to a contributor instead of producing a corrupt-but-green record.

**Cons:** Adds one more infrastructure-failure family token to maintain.

**Risks:** The detection predicate must run before record emission for that file, so it composes with the *Partial-evaluation semantics* (emit resolved violations, then the infra record, then exit non-zero).

### Option B — Exclude such paths from Audit scope

**Approach:** Add files whose path contains an ASCII tab or a line terminator to the *Audit scope* exclusion list, with an explicit normative statement that they are out of the audited source tree.

**Spec edits:** Extend the exclusions enumeration in `audit-resolution.md` — *Audit scope* with a tab/newline-in-path exclusion clause.

**Pros:** Localised to one paragraph; keeps the wire contract's tab-free assertion literally true.

**Cons:** Silently drops the file from the walk, opening a surface-set-closure hole — a Pi-touching reference inside such a file is never audited. Contradicts the audit's blind-walk closure premise.

**Risks:** A contributor could deliberately or accidentally place a Pi-surface reference in an excluded-by-name file and evade the closure guarantee.

### Recommendation

Adopt **Option A**. The audit's central property is surface-set closure enforced fail-closed; silently excluding a walked file (Option B) creates exactly the closure hole the audit exists to prevent. Surfacing the pathological path as an `infra`-class failure preserves closure and keeps the corruption out of the record stream. Edge case for the implementer: the check must also cover a path containing a bare `\r` (and a `\r\n`), not only `\n`, since a lone carriage return is equally a line terminator on the stdout surface.

## Consequence

**Severity:** correctness

A walked in-scope file whose on-disk path contains an ASCII tab or a line terminator corrupts the wire format: a tab fabricates an extra field segment and a newline splits one record into two physical lines, so a conforming CI parser mis-counts fields or mis-routes records. Two reasonable implementers diverge — one emits a corrupt record, another fails closed — and absent a detection rule the surface-set-closure guarantee can be silently violated for the offending file. The trigger is a pathological filename, so the blast radius is narrow, but the behaviour on that path is genuinely undefined.
