## Finding

The `Loom-package implementation dependencies (loom 1.0)` bullet in `implementation-notes.md` records reference-implementation packaging decisions inline in the spec body: it names the `semver`, `chokidar`, and `yaml` npm packages, pins each to a specific `package.json` block (`semver`/`chokidar`/`yaml` in `dependencies`, `@types/semver` in `devDependencies`), and notes which of them ship their own TypeScript bindings (so `yaml` needs no `@types/*` entry). The adjacent `Implementation hint (non-normative)` bullet does the same for AJV (`ajv ^8`, `ajv-formats ^3` in `dependencies`). Package-manager layout, which dependency block a library lands in, and whether a `@types/*` companion is required are reference-implementation packaging concerns with no caller-observable contract — they are `CONTRIBUTING.md` / README material, not language-spec content.

The bullets carry "non-normative" qualifiers, but they are interleaved flat among genuinely normative runtime bullets (Schema validation, Clock) on the same page rather than being fenced off in a reference-implementation appendix. The packaging trivia is therefore mixed with conformance content with no structural boundary.

Compounding this, the same bullet carries one statement that *is* conformance-relevant — that `chokidar` and `yaml` fall outside the inventory-closure audit's *Target surface categories* (only the canonical `pi`/`ctx` carriers, the four `@earendil-works/*` peers, and `typebox` are in scope), so no `// allow-pi-surface:` exemption marker is required at their import sites. Burying an audit-scope statement an implementer of the closure audit must honour inside a bullet headed "implementation dependencies" alongside dropped-as-trivia packaging guidance blurs which sentences bind a conforming implementation and which are incidental choices of the reference.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** 1ea0455 — pi-loom spec: resolve "DI-seam vs reference-implementation placement boundary" (2026-06-04, Thomas Andersen); 1064946 — pi-loom plan: resolve "YAML frontmatter parsing mechanism is never declared as a dependency" (2026-06-10, Thomas Andersen)
**History:** The `Loom-package implementation dependencies (loom 1.0)` bullet was introduced by 1ea0455, the same commit that drew the DI-seam vs reference-implementation placement boundary; it landed the `semver`/`chokidar` packaging recipe (dependency-block placement, `@types/semver`, audit out-of-scope clause) flat in the spec body. Commit 1064946 later widened the same bullet with the `yaml` clause. The AJV `Implementation hint (non-normative)` bullet predates these and was only relocated, not created, by later moves. The packaging-recipe-in-spec-body shape was complete at 1ea0455; 1064946 extended the existing pattern rather than introducing it through interaction.

## Solution Space

**Shape:** multiple

### Option A — Relocate packaging recipe to a reference-implementation doc

- **Approach:** Move the dependency-block placement, `@types/*`, and package-naming prose to a reference-implementation `README.md` / `CONTRIBUTING.md`. Retain in the spec only conformance-binding language (the inventory-closure audit out-of-scope statement, the validator-neutrality / `SchemaValidator` contract already owned elsewhere).
- **Spec edits:** Delete the `Loom-package implementation dependencies (loom 1.0)` bullet and the AJV packaging detail from `implementation-notes.md`; lift the audit out-of-scope statement to (or cite it from) `audit-target-categories.md`.
- **Pros:** Cleanest scope separation; spec body carries no packaging trivia.
- **Cons:** Breaks the `#loom-package-implementation-dependencies-loom-1-0` / `-v1` anchors that `H1a` and `V6a` cite; the conformance-relevant out-of-scope clause must be rehomed without loss.
- **Risks:** Orphaned cross-references if the anchor is dropped without repointing the two dependent plan leaves.

### Option B — Isolate the recipe under an explicit non-normative reference-implementation sub-section

- **Approach:** Keep the content in `implementation-notes.md` but fence the packaging recipe (and the AJV/Chevrotain hints) under an explicit `### Reference-implementation packaging (non-normative)` sub-heading, and lift the audit out-of-scope statement out of that sub-section into a normative line (or a forward citation to `audit-target-categories.md`). Preserve the existing anchors on the relocated sub-section.
- **Spec edits:** Add the non-normative sub-heading; move the packaging bullets under it; keep the audit out-of-scope sentence as a normative statement (cited to the audit target-categories page); retain `#loom-package-implementation-dependencies-loom-1-0` and `#loom-package-implementation-dependencies-v1` anchors on the relocated content.
- **Pros:** Preserves the anchors `H1a`/`V6a` depend on; keeps the dependency-naming discoverable next to the runtime notes; cleanly separates the conformance-relevant statement from the packaging trivia.
- **Cons:** Packaging detail still lives in the spec file (mitigated by the explicit non-normative fence).
- **Risks:** Minimal; the boundary is editorial.

### Recommendation

Take Option B. The bullet's anchors are load-bearing — `H1a` sources its `dependencies`-block placement and `V6a` its `yaml` dependency from `#loom-package-implementation-dependencies-loom-1-0` — so a clean extraction (Option A) would orphan two plan-leaf citations, whereas an explicit non-normative fence resolves the scope concern while keeping the anchors stable. The implementer must (1) preserve both `#loom-package-implementation-dependencies-loom-1-0` and `#loom-package-implementation-dependencies-v1` anchors on the relocated sub-section, and (2) keep the `chokidar`/`yaml` audit out-of-scope statement as a normative line (or a citation to `audit-target-categories.md`) rather than letting it inherit the non-normative label of the surrounding packaging prose.

## Consequence

**Severity:** advisory

An alternate implementer cannot tell at a glance which sentences in the bullet are binding (the audit out-of-scope rule) and which are incidental reference-implementation packaging choices (dependency-block placement, `@types/*` needs). Because the packaging detail is explicitly labelled non-normative, no working-system divergence results, but the spec carries contributor-doc material in its normative body and risks an audit implementer treating the conformance-relevant out-of-scope statement as droppable trivia.
