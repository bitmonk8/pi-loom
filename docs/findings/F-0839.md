## Finding

`docs/spec_topics/pi-integration-contract/version-bump-step2b.md` carries the H1 "Version bump step2b" and a matching filename stem, but its body contains three numbered list items: the `**(b) Negative direction …**` paragraph that completes step 2 (started in the sibling `version-bump-step2.md`), the numbered item **3. Re-confirm the `engines.node` floor**, and the numbered item **4. Update the version pin in `peerDependencies` and the equivalent literal here**. The parent index `docs/spec_topics/pi-integration-contract.md` lists this child as "Version bump step2b", so neither the filename nor the TOC entry hints that steps 3 and 4 live there.

The neighbouring sibling has the inverse defect (`version-bump-triggers.md` is titled "Version bump steps 5–7"); together the four files of the version-bump procedure (`-intro`, `-step2`, `-step2b`, `-triggers`) use three different naming conventions for the same series. A contributor opening the index to find step 3 or step 4 by name will not see either, and inbound cross-references to those steps necessarily reach them via anchors on this file even though its name signals a narrower scope.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `f5e89f4` ("docs: enforce 100KB hard / 30KB recommended size cap on spec set", 2026-06-04)
**History:** The child file was created by the one-shot split driver `docs/_tools/split_spec.py`, whose slice table assigns the stem `version-bump-step2b` to lines 923–930 of the pre-split monolithic `docs/spec_topics/pi-integration-contract.md`. Those eight lines of the pre-split file span the step-2(b) paragraph (line 923) plus the standalone numbered items 3 and 4 (lines 928–929) plus a closing line; the slice driver picked the stem from the *first* numbered item in the range rather than its full content. `git log --follow` on the child file returns six commits, all post-split refinements (peerDependencies assertion shape, branch (2) co-edit obligation, engines.node operand fix, T061 clarity gaps); none of them widened the scope of this file — its content covered steps 2(b), 3, and 4 from the moment it was carved out, and the title/filename was never updated to reflect that the slice straddles three steps.

## Solution Space

**Shape:** single

### Recommendation

Rename the file and its H1 to cover the full slice: `version-bump-step2b.md` → `version-bump-steps-2b-4.md`, and "Version bump step2b" → "Version bump steps 2b–4". Update the index entry in `docs/spec_topics/pi-integration-contract.md` and repoint every inbound cross-reference (the ~24 `grep -rn 'version-bump-step2b' docs/` hits across `audit-resolution.md`, `inventory-audit-intro.md`, `version-bump-step2.md`, `version-bump-triggers.md`, and the `split_spec.py` slice table) to the new stem in the same commit. The anchor IDs inside the file (`bump-step-2b-delete`, `bump-step-2b-promote`, `bump-step-2b-exempt`, `bump-step-2b-stale-rewrite`, `bump-step-2b-rewrite-shape`) keep the `2b` token because they name branches of step 2(b) specifically — they do not need to change, and inbound links that target those anchors will continue to resolve once the path stem is updated.

Edge cases the implementer must watch:

- The `split_spec.py` slice tuple `("version-bump-step2b", 923, 930)` must be updated in the same commit so a future re-run of the splitter produces a file under the new stem rather than re-introducing the old one.
- Two sibling findings on this consolidated review touch closely-related surface — `version-bump-triggers.md` is being renamed to `version-bump-steps-5-7.md` by its own finding, and the *procedure-output summary placement* finding wants the trailing summary block moved off this file's step 4 onto `version-bump-intro.md`. Land this rename either before that placement edit (so the placement edit operates on a file whose name already matches its scope) or co-resolve with it; landing it *after* the placement edit means re-touching the same file twice. The sibling-file rename and this one are independent and can land in either order.
- After the rename the four files in the version-bump series read as a uniform sequence (`-intro`, `-step2`, `-steps-2b-4`, `-steps-5-7`), provided the `version-bump-triggers.md` rename also lands.

## Consequence

**Severity:** advisory

A contributor looking up step 3 or step 4 by TOC or filename finds neither; they must already know that steps 3 and 4 live in the step-2(b) child. No runtime correctness is at stake — the inbound anchor links resolve and the procedure's mechanical gates are unaffected — but the navigation defect compounds with the sibling `version-bump-triggers.md` mismatch to make the four-file procedure read as three inconsistent naming conventions.
