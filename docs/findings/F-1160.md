## Finding

`discovery/package-and-settings.md` §Settings file reads states that both `.pi/settings.json` and `~/.pi/agent/settings.json` are **optional** (each may be absent and is then treated as `{}`), and the §Caching and reload paragraph says "A file-watcher on each of the two paths invalidates the cache on change; reads following invalidation re-apply the merge." The single word "change" leaves the creation case implicit: when a user creates one of the two settings files for the first time during a session, the cached value (`{}`) is wrong until the watcher fires and the merge is re-applied.

The underlying capability exists — `PIC-14` defines `FileWatchEventKind = "add" | "change" | "unlink"`, so the seam can deliver the creation event — but the spec's natural-language phrasing "file-watcher on **each of the two paths**" tilts the implementer toward watching the path literal, which most file-watching APIs cannot do meaningfully for a path that does not yet exist (chokidar with `ignoreInitial` defaults, `fs.watch`, etc., will only deliver `add` for a previously-absent file when the watcher subscribes to the containing directory). Without an explicit "MUST detect `add` of a previously-absent file" clause, two reasonable implementations diverge: one re-merges as soon as the missing file appears, the other only re-merges if the file already existed at extension load and is later modified.

The same concern applies symmetrically to `unlink` of an originally-present settings file (the post-`unlink` merge should fall back to `{}` for that source while preserving the other file's contribution per the existing replace-not-suppress rule), and to the structural-change note already required by `registration-steps.md` §"Structural changes" when a settings edit nets added or removed `loomPaths` entries.

## Solution Space

**Shape:** single

### Recommendation

In `discovery/package-and-settings.md` §Caching and reload, replace "A file-watcher on each of the two paths invalidates the cache on change" with an explicit three-kind obligation citing the `FileWatcher` seam:

> The extension attaches a `FileWatcher` (per [PIC-14](../pi-integration-contract/host-interfaces-services.md#pic-14)) covering each of the two settings paths, observing all three `FileWatchEventKind` values: `"add"` (the file was previously absent and now exists — including the first-time creation of an optional file), `"change"` (an existing file's contents changed), and `"unlink"` (a previously-present file was removed). Any of the three invalidates the cache for that file and re-applies the merge per §Merge semantics; an `"unlink"` reverts that file's contribution to `{}` while the other file's loaded value is unaffected, identical to the §Failure modes "missing or unreadable" path (a single `loom/load/settings-unreadable` is logged per the existing rule). The debounce window, the malformed-intermediate-state rule, and the ERR-7 surfacing of re-merge failures (per §Watcher-time reload failures) all apply unchanged across the three kinds.

Implementer-relevant edge cases:

- The seam contract is the source of truth: how the production chokidar adapter subscribes (typically to the parent directory with a filename predicate, so `"add"` for a previously-absent file is delivered) is an adapter implementation detail and stays out of the spec — only the three observed kinds are normative.
- An `"unlink"` followed by an `"add"` within the same 250 ms debounce window coalesces into one re-merge that observes the final post-window contents, consistent with the existing drop-and-reschedule rule.
- The structural-change `loom-system-note` clause in `registration-steps.md` §"Structural changes" already routes settings-edit-driven `loomPaths` changes through the watcher note path; that rule is unchanged — it consumes the post-re-merge resolved set, not the raw watcher event.
- The presupposition that the production `FileWatcher` adapter can deliver `"add"` for a path that did not exist at `watch()` time should land in `host-interfaces-services.md` §`PIC-14` as a one-line bullet (the seam interface already names the three kinds; only the "for previously-absent paths" clarification is new), so the discovery page can cite it rather than re-state it.

## Consequence

**Severity:** correctness

Without an explicit obligation, an implementer who reads "file-watcher on each of the two paths" literally will subscribe to the path that may not exist and silently never observe creation. A user who writes their first `.pi/settings.json` mid-session then expects loom to honour `loomPaths` / `looms.binderModel` immediately; on a literal-path implementation nothing happens until `/reload`, despite the spec's promise that "reads following invalidation re-apply the merge." Two conforming implementations would diverge on this user-visible behaviour.
