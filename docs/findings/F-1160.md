## Problem

The §Caching and reload paragraph (`id="caching-and-reload"`) says the file-watcher on each of the two optional settings paths "invalidates the cache on change", leaving the creation case implicit: when a user creates a previously-absent `.pi/settings.json` or `~/.pi/agent/settings.json` mid-session, the cached `{}` stays wrong until a later modification. The word "change" tilts an implementer toward watching the path literal, which most file-watching APIs cannot observe meaningfully for a not-yet-existing path, so two conforming implementations diverge: one re-merges as soon as the file appears, the other only on modification of an already-present file. The same gap applies symmetrically to `unlink` of an originally-present file. PIC-14's `FileWatchEventKind = "add" | "change" | "unlink"` already makes the obligation satisfiable, but the discovery sentence never invokes `"add"` explicitly.

The wording has never carried a clause covering creation of a previously-absent optional file; subsequent edits to the paragraph (`d488eaa`, `f88698e`, `9ce41c6`, the file relocation in `f5e89f4`) refined ERR-7 elevation, the binderModel hot-reload note, and the debounce window but did not touch the change-vs-add framing. The defect is therefore present since the §"Settings file reads" section was introduced; `PIC-14`'s three-kind `FileWatchEventKind` enumeration arrived later (`8a6d15f`, "FileSystem seam interface signature lives outside the spec corpus") and made the unstated obligation latently satisfiable, but the discovery-page sentence still does not invoke `"add"` explicitly.

## Solution approach

Rewrite the §Caching and reload sentence (`id="caching-and-reload"`) so the settings file-watcher explicitly observes all three PIC-14 `FileWatchEventKind` values — `"add"` (covering first-time creation of a previously-absent optional file), `"change"`, and `"unlink"` — each invalidating that file's cache and re-applying the merge per DISC-7 (§Merge semantics). An `"unlink"` reverts that file's contribution to `{}` while the other file's loaded value is unaffected, identical to the §Failure modes "missing or unreadable" path. Add a clause to PIC-14 (`#pic-14`) stating the production adapter delivers `"add"` for a path that was absent at `watch()` time, so the discovery page can cite it rather than restate it.

## Solution constraints

- Out of scope: the overlapping-rebuild serialisation / ordering rule (owned by F-1159).
- Keep adapter mechanism out of the normative text — only the three `FileWatchEventKind` values are normative; how the production adapter subscribes to deliver `"add"` for a previously-absent path is an implementation detail (per GOV-18 arm (a)).
