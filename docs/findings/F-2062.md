## Finding

The opening paragraph of `active-invocation-registry.md` defines how the `ActiveInvocationRegistry` entry's `loom` field is resolved across the three insertion sites (slash-command handler entry, `tool.execute(...)` adapter entry, `invoke` spawn-site entry). The normative content is complete and converges on a single rule: read the slash-name from the already-resolved catalog / static-resolution cache / `tools:` snapshot entry the arm already carries, and never re-derive the canonical name at the insertion site. The defect is purely structural: that rule is expressed as one unbroken ~400-word sentence that interleaves the three insertion arms, the two fallback sources (the per-load-pass static-resolution cache and the parent loom's load-time `tools:` snapshot), the `LoomRegistry` entry-table key, the collision-handling guarantees, and several parenthetical justifications.

Because the three arms, their source structures, and the single shared rule are braided into one sentence rather than separated, an implementer must mentally de-interleave the arms to extract the source structure for any one of them. This raises the risk that an implementer reads off the wrong arm's source, or re-derives the name from the call-site's authored identifier (the registered command name, the adapter's source loom, or the `invoke(...)` literal path) — which is the exact behaviour the sentence forbids, and the behaviour that would split operator dedup buckets on `details.event.loom` between two implementations that converged on different canonical forms.

This is a faithful-restructuring opportunity, not a contract gap: no rule needs to change, only its presentation.

## Solution Space

**Shape:** single

### Recommendation

Decompose the opening resolver sentence into a short per-arm list. Keep the lead-in stating that all three insertion sites consume the same canonical-name resolver and that the `loom` field carries the final-derived name (post-`pi.looms` mapping, stem verbatim, no leading `/`), then give one bullet per arm:

- **slash-command handler** — read the slash-name from the `LoomRegistry` `Map<slashName, parsedLoom>` key via the drain-state contract's entry-table accessor.
- **`tool.execute(...)` adapter** — for slash-registered tool-exposed (prompt-mode) looms, read from the same `LoomRegistry` arm; for subagent-mode tool-only callees that are never slash-registered, read the slash-name field the parent loom's load-time `tools:` resolution snapshot already carries.
- **`invoke` spawn-site** — for `tools:`-routed callables, read from the parent's `tools:` snapshot; for literal `invoke("./path.loom", ...)`, read the slash-name field the per-load-pass static-resolution cache entry already carries.

Close with the single shared rule stated once: **read the slash-name from the already-resolved entry; never re-derive the canonical name at the insertion site.** Retain (as separate short sentences, not inline parentheticals) the two load-time guarantees the current paragraph carries — that callee-only basename collisions and `pi.looms` collisions are caught at discovery/load time via `loom/load/cross-format-collision` before any insertion-site code runs, so the resolver never observes an unresolvable or collided callee, and that no `<callee-only>` sentinel is introduced. No new contract; this is a faithful restructuring that preserves every existing normative claim and cross-reference.

## Consequence

**Severity:** advisory

The normative rule is fully recoverable, so a careful implementer can produce a correct system. The risk is mis-application by a hurried reader: extracting the wrong arm's source structure or re-deriving the canonical name at the insertion site, which would diverge `details.event.loom` across implementations and split operator dedup buckets.
