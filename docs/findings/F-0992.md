## Problem

`grammar.md` §*Loom literal sublanguage* — *Field rules* states the missing-field rule (`loom/parse/missing-object-field`) as an absolute parse-time MUST that applies identically to `BareObjectLit` in both admitted positions, including the Pi-tool argument position whose LHS schema is the tool's registered input schema. `type-system.md` §*Type compatibility* — *Unresolvable operands* establishes a contrary posture for exactly this case: when a compatibility operand depends on a Pi-tool call whose registered schema is not visible at parse time, the parse-time check is skipped and the runtime AJV check is the safety net. The grammar rule cites no such fallback and says nothing about the unresolvable case. Two implementers therefore diverge on every Pi-tool call whose input schema is not statically resolvable — one fires `missing-object-field` unconditionally, the other defers the check entirely.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f841b12` (2026-05-04, "spec: resolve Cluster 2 (literal sublanguage + grammar appendix)"); `0226b80` (2026-05-05, "pi-loom plan: resolve 'V15c references an undefined compatibility relation'")
**History:** `f841b12` created `spec_topics/grammar.md` with the *Field rules* bullet as an absolute MUST against the LHS schema, with no qualifier for unresolvable cases. The following day, `0226b80` added the *Type compatibility* section to `spec_topics/type-system.md`, including the *Unresolvable operands* paragraph that introduced the "parser skips, runtime AJV is the safety net" posture for any compatibility check whose operands are past the parser's static view — citing a Pi-tool call whose registered schema is not visible at parse time as the prototypical case. The defect is the unaligned interaction between the two commits: the unresolvable-operands escape hatch landed after the field-completeness rule was authored and was not retro-fitted to it, even though field-completeness is structurally a parser-side compatibility judgement against the LHS schema. Subsequent edits to both files (the grammar.md bullet has not been substantively modified since `f841b12`; `type-system.md` gained TYPE-9/-10/-11 in later commits but did not revisit the field-completeness gap) preserved the misalignment.

## Solution approach

Clarify the `grammar.md` *Field rules* `missing-object-field` bullet so it defers to the *Unresolvable operands* posture for the Pi-tool argument position whose tool input schema is not statically visible at parse time: the parse-time `missing-object-field` check is skipped there and the runtime AJV check is the safety net. Add forward-links to [Type System — Type compatibility](./type-system.md#type-compatibility) and [Tool Calls](./tool-calls.md). State that the `params:` default position is unaffected because the param's declared type is always statically visible.

## Solution constraints

- The deferral applies only to the Pi-tool argument position; the `params:` default position keeps `missing-object-field` as a hard parse-time MUST.
