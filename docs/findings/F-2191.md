## Problem

The §3 syntactic-construct `<expr>` bullet in `placeholder-rendering-a.md` renders the offending span as "copied byte-for-byte from the source file … (post-newline-normalisation per [Lexical — Encoding])", mixing two reference frames: the raw on-disk bytes (which may carry `\r\n`) and the post-normalisation in-memory stream the start/end token positions index into. For an `<expr>` span that straddles a former `\r\n` boundary, one reading emits `\n` and the other emits `\r\n`, producing two distinct UTF-8 byte sequences for the same source defect. This defeats the section's stated GOV-15 byte-identical conformance guarantee for a CRLF-checked-in `.loom`. `lexical.md` already settles the frame — newline normalisation happens before span recording, so the token positions are offsets into the normalised stream.

The rule and its directional ambiguity entered the corpus in the same authoring commit; no later edit changed the wording. The 100KB-cap split (`f5e89f4`, 2026-06-04) relocated the bullet into `placeholder-rendering-a.md` and re-pointed the relative link (`./lexical.md` → `../lexical.md`) but left the defect text untouched. The defect is thus original to the rule, not a later regression.

## Solution approach

Clarify the `<expr>` bullet in §3 of `placeholder-rendering-a.md` so the copy frame is explicit: the span is extracted from the post-normalisation in-memory source stream — the same stream the start/end token positions index into — not from the raw on-disk bytes. State that, because newline normalisation precedes span recording (per the Diagnostic-spans rule in `lexical.md`), a `\r\n` or bare `\r` inside the span renders as a single `\n`, so the rendered `<expr>` is independent of the file's on-disk line-ending convention. Retain the existing internal-whitespace-preserved semantics.
