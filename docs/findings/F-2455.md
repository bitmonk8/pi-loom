## Problem

The **Extension-bootstrap SDK failures** section classifies a `pi.registerFlag` failure (step 1) and any factory-time `pi.on(...)` subscription failure (steps 1/3/4) as fatal to the whole extension and pins that steps 2–5 do not execute, but it never states the disposition of registrations that already succeeded before the fatal step. Two facets are open: the already-installed Pi-side registrations (renderer, flag, earlier subscriptions) have no stated post-fatal disposition; and when the fatal step is the step-4 `session_shutdown` subscription, the step-3 `session_start` handler is already live and will fire, calling `pi.registerCommand` against a `LoomRegistry` that step 5 never built. The spec gives no rule requiring the half-bootstrapped extension to self-disable, so an implementer is left to invent whether a fatally-failed extension still registers and dispatches per-loom commands.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `2d2fc8e` (2026-05-06, "pi-loom spec: resolve \"Pi SDK capability calls have no failure contract\"")
**History:** The corpus is git-tracked. `git log -S` over the fatal-rule tokens (`"do not execute, mirroring the step-0 failure semantics"`, `"Extension-bootstrap SDK failures"`) traces the extension-bootstrap SDK-failure granularity contract to its first authoring in `2d2fc8e`, which introduced the `pi.registerFlag` fatal bullet ("steps 2–5 do not execute") without ever specifying the disposition of registrations that had already succeeded. The omission was carried forward unchanged when the `pi.on(...)` granularity bullet was added (`79ea4e8`, 2026-06-06) and when the section was split into its own file under the size-cap commit (`f5e89f4`, 2026-06-04); subsequent relocations (`6a42c1a`, `4b4382f`) did not touch it. The tokens `rollback` and `unwind` have never appeared in the file's history. The defect is an omission present since the contract's inception, not a regression introduced by a later edit.

## Solution approach

Clarify the **Extension-bootstrap SDK failures** section's fatal `pi.registerFlag` and `pi.on(...)` bullets in `extension-bootstrap-and-per-loom.md` to state the post-fatal disposition of registrations that already succeeded: the renderer, the `--loom` flag, and earlier subscriptions remain installed for the extension-instance lifetime, Pi reclaims them on instance teardown (`/reload`, `/new`, quit), and no loom-side rollback exists because Pi exposes no `unregisterMessageRenderer` / `pi.unregisterCommand` / unsubscribe surface. Add a rule that a fatally-failed extension is non-dispatchable, routing a `session_start` handler subscribed before the fatal step through the degraded arm rather than registering per-loom commands or dispatching into a never-built `LoomRegistry`, by reusing the `LoomRegistry` drain-state contract (`drain-state-contract.md#loom-registry-drain-state-contract`).

## Solution constraints

- Out of scope: pinning the `pi.on`-failure rule's current-step position — that prose at the same bullet is owned by the `code-registry-load.md` sibling.
- The drain-state-tag write on the fatal path MUST NOT break the `session_shutdown` handler-entry short-circuit predicate, which assumes the tag is written only by the teardown path.
