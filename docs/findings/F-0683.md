## Finding

`discovery-sources.md` defines the CLI source as `--loom <paths>` whose components, after splitting on `path.delimiter`, are "resolved with the same rules as the settings `loomPaths` array (see [`loomPaths` entry schema](./package-and-settings.md#loompaths-entry-schema) below)." The cross-reference does not close the loop. The `loomPaths` entry-schema Resolution rule specifies a base **only** for relative paths originating in a settings file: "Paths in `~/.pi/agent/settings.json` resolve relative to `~/.pi/agent/`; paths in `.pi/settings.json` resolve relative to `.pi/`." A `--loom` component has no originating settings file, so a bare relative path such as `./tools.loom`, `looms`, or `../shared/code-review.loom` is left unanchored — three reasonable implementers could plausibly pick `process.cwd()`, the captured `FileSystem.cwd()` seam value, the project root supplied by `event.cwd`, or the user's home directory and each be internally consistent with the cited cross-reference.

The gap is observer-visible. Whether `pi --loom ./tools.loom` finds the file depends on which base the implementation picks; a project tutorial that documents one choice will fail under another implementation. The same uncertainty propagates into discovery-root computation (per the *Discovery roots* paragraph, each `--loom` component contributes a root, and the resulting root set is what the `invoke`/`tools:` path-restriction rule consults), so a wrongly-anchored relative `--loom` path can quietly admit or exclude callees.

The two parallel relative-path sites in the same file have already been pinned: the project `.pi/settings.json` path is "resolved relative to the factory-time working directory supplied by the [`FileSystem.cwd()` seam member]" with a defined preference for `event.cwd` on `resources_discover` events, and `~/`-prefixed entries route through DISC-1's `FileSystem.homedir()` rule. The `--loom` relative case is the only path-resolution rule in the discovery surface without a stated base.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `fae85d3` (2026-05-04, "Tighten spec for implementation-plan readiness") added the CLI source bullet as `--loom <path> (repeatable, optional)` with no resolution rule at all.
**History:** The `--loom` CLI source was introduced by `fae85d3` with no resolution rule whatsoever — the bullet read `- CLI: \`--loom <path>\` (repeatable, optional)` and the per-component anchoring was not addressed. Commit `b5ba88e` (2026-05-05, "spec: --loom CLI flag is single string with OS path-list separator") rewrote the bullet to delegate to the settings `looms` entry schema ("resolved with the same rules as the settings `looms` array"), which papered over the gap by reference rather than closing it: the settings-entry Resolution rule (added by `d261b07` on 2026-05-04, "spec: resolve 'settings.json looms array entry schema is undefined'") only defines a base relative to the originating settings file, a referent the CLI source lacks. The defect has therefore existed continuously from `fae85d3` onward; `b5ba88e` is the commit that codified the false-closure cross-reference but is not the introducing commit, and `d261b07` is the upstream commit whose narrow scope (settings-file-relative anchoring) the cross-reference silently overshoots.

## Solution Space

**Shape:** multiple

### Option A — Anchor relative `--loom` components against `FileSystem.cwd()` (with `event.cwd` on reload)

**Approach.** State that a relative `--loom` component (no leading `/`, drive-letter prefix, or `~/`) resolves against `FileSystem.cwd()` at factory time, and against `event.cwd` on every `resources_discover` event — mirroring the rule already pinned for the project `.pi/settings.json` read in `package-and-settings.md` and `pi-integration-contract/registration-steps.md` step 1.

**Spec edits.** In `discovery-sources.md`, after the existing "resolved with the same rules as the settings `loomPaths` array" cross-reference, append a sentence:

> A `--loom` component that is not absolute and not `~/`-rooted resolves against the project root: at factory time, the [`FileSystem.cwd()` seam member](../pi-integration-contract/host-interfaces-services.md#pic-13); on every `resources_discover` event, `event.cwd` per [Registration steps](../pi-integration-contract/registration-steps.md) step 1. This overrides the settings-file-relative base of the cross-referenced rule, which does not apply to a CLI flag.

**Pros.**
- Reuses the seam and `event.cwd`-preference machinery the project-local settings read already pins, with no new seam.
- Matches user mental model: `pi --loom ./tools.loom` invoked from the project root resolves the same way `pi` itself resolves project-relative paths.
- Backward-compatible with any plausible existing tutorial that assumes "cwd at invocation time."

**Cons.**
- Inherits the [factory-time `cwd == project root` presupposition](../spec_topics/pi-integration-contract/host-interfaces-services.md#cwd-project-root-presupposition); a host that launches the factory with `cwd` elsewhere mis-anchors relative `--loom` components until the first `resources_discover` event arrives.
- Subtly different from a user shell mental model on reload: a `--loom ./tools.loom` typed from `/elsewhere` will be re-resolved against `event.cwd` on the next reload, not the original invocation directory.

**Risks.** None new beyond the existing `cwd == project root` presupposition.

### Option B — Reject non-absolute, non-`~/` `--loom` components

**Approach.** Disallow bare relative `--loom` components. Each component must be absolute (`/…`, `C:\…`) or `~/`-rooted; anything else fails the entry with a load-phase diagnostic and the component contributes no loom.

**Spec edits.** In `discovery-sources.md`, after the existing cross-reference, append:

> A `--loom` component that is neither absolute nor `~/`-rooted is rejected per-component with `loom/load/invalid-cli-loom-path` (severity `error`); the offending component contributes no loom and is reported via the standard diagnostics channel naming `"--loom flag component #N"`. Other components in the same flag still process. The settings-file-relative resolution base of the cross-referenced `loomPaths` entry schema does not apply to a CLI flag.

A new code row is added to `docs/spec_topics/diagnostics/code-registry-load.md` and a `--loom` example added to `docs/spec_topics/diagnostics/placeholder-rendering-b.md` for the new code.

**Pros.**
- No dependency on the `cwd == project root` presupposition; behaviour is fully determined by the literal flag text.
- Eliminates the factory-vs-reload base-change subtlety in Option A.

**Cons.**
- Adds a new diagnostic code (registry-load row + placeholder example), expanding the closed-set fixture surface.
- Breaks shell ergonomics: users who today (in any draft implementation) type `pi --loom ./tools.loom` from the project root get an error instead of the intuitive result.
- Diverges from the settings-entry schema, where a non-absolute, non-`~/`-rooted path is accepted and anchored.

**Risks.** Authors who try the natural shell form will encounter a diagnostic before learning the rule; the diagnostic must be discoverable.

### Recommendation

Take **Option A**. The project root is already the conventional anchor for every other relative path in the discovery surface (settings paths, the project discovery root `.pi/looms/`), the seam (`FileSystem.cwd()` with `event.cwd` preference on reload) is already pinned and exercised by adjacent rules, and no new diagnostic code or fixture row is needed. Option B's principal merit — avoiding the `cwd == project root` presupposition — is moot because the same presupposition already governs the project-local settings read; pinning the `--loom` rule to a different base would introduce a new asymmetry rather than remove one. Implementer must watch: (1) the `event.cwd`-preference rule applies symmetrically to `--loom` resolution and to project-local settings reads on every `resources_discover` event, so a relative `--loom` component re-resolves on reload — this is the intended behaviour and must be tested explicitly; (2) the absolute/`~/`-rooted predicate must cover Windows drive-letter prefixes (`C:\…`, `\\server\share`) as absolute, consistent with the existing `path.delimiter` Windows guidance in the same bullet.

## Consequence

**Severity:** correctness

Two reasonable implementers can pick different anchors (`process.cwd()`, `FileSystem.cwd()` capture, `event.cwd`, `~`) and each be internally consistent with the cross-reference, producing divergent file-resolution behaviour for the same `pi --loom ./tools.loom` invocation. The discrepancy then propagates into the discovery-root set the `invoke`/`tools:` path-restriction rule consults, so a wrongly-anchored relative path can silently admit or exclude callees.
