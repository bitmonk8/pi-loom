## Finding

`discovery/discovery-sources.md` line 7 pins the `--loom` CLI flag as a single `string`-typed flag whose value is split on Node's `path.delimiter`, and registration-steps.md restates that the `pi.getFlag('loom')` string value is fed through that split. Neither file says what happens when the split yields an **empty string** component. Empty components arise from realistic operator inputs:

- A delimiter-only value: `--loom ""`, `--loom ":"`, `--loom ";;"` (the last two are easy to produce from a shell that joins a possibly-empty `$LOOM_PATHS` variable with the platform separator).
- Leading or trailing separators: `--loom ":a.loom"`, `--loom "a.loom;"`.
- Repeated separators between non-empty paths: `--loom "a.loom;;b.loom"`.

Two readings of the existing spec are equally defensible and give different behaviour. Reading A: each component is "an entry resolved by the same file-vs-directory rule as settings entries" (line 31); empty string is an entry; the path resolver normalises `""` to the process CWD; the CWD silently becomes a discovery root and is non-recursively scanned for `*.loom`. Reading B: the `--loom` row of DISC-2 says missing explicit references are errors; an empty path does not exist; the entry produces a `loom/load/missing-source` error per component. Reading C — drop empty components silently as harmless syntactic noise — is the behaviour DISC-2's symmetric handling for settings entries would suggest, but it is nowhere written.

Two reasonable implementers can land on different arms of this trichotomy from the same spec text. Reading A is the worst outcome (a `--loom ""` invocation silently registers every `*.loom` in the CWD as a slash command); Reading B floods diagnostics for benign shell-joining mistakes; Reading C is the operator-friendly behaviour but is currently unwritten. The spec MUST close the trichotomy explicitly.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `fae85d3` (2026-05-04, "Tighten spec for implementation-plan readiness") added the CLI source bullet as `--loom <path> (repeatable, optional)` with no resolution rule at all.
**History:** The `--loom` CLI source was introduced by `fae85d3` with no resolution rule whatsoever — the bullet read `- CLI: \`--loom <path>\` (repeatable, optional)` and the per-component anchoring was not addressed. Commit `b5ba88e` (2026-05-05, "spec: --loom CLI flag is single string with OS path-list separator") rewrote the bullet to delegate to the settings `looms` entry schema ("resolved with the same rules as the settings `looms` array"), which papered over the gap by reference rather than closing it: the settings-entry Resolution rule (added by `d261b07` on 2026-05-04, "spec: resolve 'settings.json looms array entry schema is undefined'") only defines a base relative to the originating settings file, a referent the CLI source lacks. The defect has therefore existed continuously from `fae85d3` onward; `b5ba88e` is the commit that codified the false-closure cross-reference but is not the introducing commit, and `d261b07` is the upstream commit whose narrow scope (settings-file-relative anchoring) the cross-reference silently overshoots.

## Solution Space

**Shape:** single

### Recommendation

Specify that empty components produced by the `path.delimiter` split are dropped silently — they contribute no discovery root, no `loom/load/missing-source` error, and no other diagnostic. The drop is a syntactic-noise filter applied to the split result before any per-entry resolution runs.

Concrete spec edits (all in `docs/spec_topics/discovery/discovery-sources.md`):

1. Extend the CLI bullet (line 7) with a trailing sentence: *"After splitting on `path.delimiter`, empty-string components are dropped silently — they contribute no discovery root and emit no diagnostic. Leading, trailing, and repeated separators are therefore syntactic noise. A `--loom` value that consists entirely of separators (including `--loom ""`) contributes no `--loom` paths to the walk and is indistinguishable from omitting the flag."*
2. Add a corresponding clarifying line under the discovery-roots fifth bullet (line 31), or rely on the CLI bullet's reference.
3. In the DISC-2 "Implementation notes" list (after the existing third bullet, lines 65–68), add: *"Empty components produced by the `--loom` `path.delimiter` split are dropped silently before per-entry classification — the `loom/load/missing-source` error for explicit references therefore never fires on an empty-string component."*

Edge cases the implementer must watch:

- The drop happens **before** the `~/` expansion rule of DISC-1, so a `~/` expansion of an empty string never runs (Node's `path.resolve('')` would otherwise reduce to CWD).
- The drop is for empty strings only; a component consisting solely of whitespace (`--loom " "`) is **not** empty after split and falls through to normal resolution, where it will fail per DISC-2's `loom/load/missing-source` / `loom/load/unreadable-source` arms. The spec does not need to special-case whitespace-only entries — DISC-2 already covers them under existing rules.
- The same empty-drop rule does **not** automatically apply to `loomPaths` array entries (those are JSON strings, not delimiter-split tokens, so an empty-string `loomPaths` entry is an author-authored empty string that DISC-2's `Settings loomPaths entry` row already routes to `loom/load/missing-source`). The asymmetry is intentional: `--loom ""` is a shell-joining artefact; `loomPaths: [""]` is an authored mistake. The new rule says so only for the CLI split arm.

## Consequence

**Severity:** correctness

A `--loom ""` invocation (trivially producible from a shell that interpolates an empty variable) silently scans the process CWD for `*.loom` and registers everything it finds as slash commands under Reading A — a real privilege-escalation-shaped surprise, since the project CWD is a place looms-from-the-CWD have no reason to be authoritative. Reading B fires `loom/load/missing-source` errors for benign trailing-separator noise. Two implementers will diverge silently, and the divergence is observable end-to-end through `pi.getCommands()`.
