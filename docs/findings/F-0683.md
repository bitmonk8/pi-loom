## Problem

The CLI source bullet in `discovery-sources.md` resolves each `--loom` component "with the same rules as the settings `loomPaths` array", cross-referencing the [`loomPaths` entry schema](./package-and-settings.md#loompaths-entry-schema). That schema's Resolution rule defines a base only for paths originating in a settings file (relative to `~/.pi/agent/` or `.pi/`). A `--loom` component has no originating settings file, so a bare relative component such as `./tools.loom` or `looms` is left unanchored and reasonable implementers could pick different bases. The gap is observer-visible and propagates into the discovery-root set the `invoke`/`tools:` path-restriction rule consults, so a wrongly-anchored relative `--loom` path can silently admit or exclude callees.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `fae85d3` (2026-05-04, "Tighten spec for implementation-plan readiness") added the CLI source bullet as `--loom <path> (repeatable, optional)` with no resolution rule at all.
**History:** The `--loom` CLI source was introduced by `fae85d3` with no resolution rule whatsoever — the bullet read `- CLI: \`--loom <path>\` (repeatable, optional)` and the per-component anchoring was not addressed. Commit `b5ba88e` (2026-05-05, "spec: --loom CLI flag is single string with OS path-list separator") rewrote the bullet to delegate to the settings `looms` entry schema ("resolved with the same rules as the settings `looms` array"), which papered over the gap by reference rather than closing it: the settings-entry Resolution rule (added by `d261b07` on 2026-05-04, "spec: resolve 'settings.json looms array entry schema is undefined'") only defines a base relative to the originating settings file, a referent the CLI source lacks. The defect has therefore existed continuously from `fae85d3` onward; `b5ba88e` is the commit that codified the false-closure cross-reference but is not the introducing commit, and `d261b07` is the upstream commit whose narrow scope (settings-file-relative anchoring) the cross-reference silently overshoots.

## Solution approach

Clarify the `--loom` CLI source bullet in `discovery-sources.md` so a relative, non-`~/`-rooted component has a defined resolution base, mirroring the project `.pi/settings.json` rule already pinned in `package-and-settings.md`: resolve against the `FileSystem.cwd()` seam ([`#pic-13`](../pi-integration-contract/host-interfaces-services.md#pic-13)) at factory time, and against `event.cwd` on each `resources_discover` event per [`registration-steps.md`](../pi-integration-contract/registration-steps.md) step 1. State that this base overrides the settings-file-relative rule of the cross-referenced `#loompaths-entry-schema`, which has no referent for a CLI flag. Absolute and `~/`-rooted components continue to route through the existing absolute-path and [DISC-1](#disc-1) home-expansion handling.

## Solution constraints

- The absolute/`~/`-rooted predicate MUST treat Windows drive-letter prefixes (`C:\…`, `\\server\share`) as absolute, consistent with the existing `path.delimiter` Windows guidance in the same bullet.
