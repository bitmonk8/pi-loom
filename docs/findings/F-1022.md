## Finding

The Depth Enforcement enforcement-point list and per-boundary table in `schema-subset.md` partition `params` validation along two call sites: `invoke(...)` (surfaces as `Err(InvokeInfraError { cause: "validation", ... })`) and slash-load (cross-routes to ceiling #3). Item #4 reads *"`params` validation at loom invocation — **whether bound via the binder or supplied directly by `invoke`**"*, and row #4 of the per-boundary table enumerates the matching two call sites.

There is a third way a loom's `params` come to be validated: the model issues a `tool_use` against a registered `.loom` callable. Per `frontmatter-fields-a.md` line 57, *"when invoked from `invoke(...)` or **as a registered tool**, arguments arrive already typed and are validated directly"*; per `tool-registration-lifetime.md` and `extension-bootstrap-and-per-loom.md`, that registered-tool's `parameters` schema **is** the lowered `params` schema of the callee. A depth-6 `arguments` payload from the model therefore trips a `params` AJV boundary that row #4's two-site enumeration does not name. Row #2 (*Tool-call args, model-driven*) is the generic owner of "model-driven `arguments` payload, JSON-decoded, before the tool body runs" — but the spec never says whether the `.loom`-callable case collapses into row #2 or constitutes a third sub-arm of row #4, so two compliant readers could route it differently: row-#2 reading routes the failure to the model as a tool-error result and counts a `tool_loop` slot; row-#4 reading routes it through `InvokeInfraError { cause: "validation", ... }` (the `invoke(...)` arm of row #4) and never crosses the model. The two are observationally distinct.

The omission is structural: enforcement-point #4's prose explicitly enumerates *"binder or `invoke`"* and stops there. Without the third site named, neither row #4 nor row #2 is unambiguously the owner.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `216130d` (Mon May 4 2026, "pi-loom spec: resolve \"Depth ≤5: counting algorithm and enforcement point unspecified\"")
**History:** `git log -L` over the enforcement-point list shows commit `216130d` introduced enforcement-point #4 with the wording *"whether bound via the binder or supplied directly by `invoke`"* — already a two-site enumeration. Commit `1750848` (Thu May 7 2026, "Ceiling #4 routing class collapses four boundaries whose destinations differ") split the prose paragraph into the per-boundary table and added row #5, but copied row #4's two-site enumeration verbatim. Later edits (`f2a948b`, `ba68984`, `f7a505e`, `f5e89f4`) refined the slash-load arm, the destination-column noun, the `cause` wording, and the path move; none added the third call site. The defect was a sin of omission at first introduction and has been preserved through every subsequent rewrite of the row.

## Solution Space

**Shape:** multiple

### Option A — collapse the third call site into row #2

**Approach.** Treat a model-driven `.loom`-callable `arguments` payload as wholly governed by row #2 (Tool-call args, model-driven). Add a single clarifying clause to row #4's left-hand-side text and a parallel back-pointer in row #2's right-hand-side text saying the row covers both Pi-tool and `.loom`-callable model-driven args, with the same destination (the model, via tool-error result).

**Spec edits.**
- `schema-subset.md` enforcement-point #4: change *"whether bound via the binder or supplied directly by `invoke`"* to *"whether bound via the binder, supplied directly by `invoke(...)`, or — for a model-driven `.loom`-callable `arguments` payload — covered by enforcement point #2 above"*.
- `schema-subset.md` table row #4 left cell: extend the parenthetical to *"(`invoke(...)` and slash-load only; the model-driven `.loom`-callable args case is row #2)"*.
- `schema-subset.md` table row #2 right cell: add a trailing sentence — *"This row covers both Pi-tool args and registered `.loom`-callable args; the loom-callable adapter's `parameters` schema is the callee's lowered `params` schema, so the depth-walk at the adapter boundary fires before the loom body and the failure feeds back to the model identically."*.
- `hard-ceilings/ceilings-3-and-4.md` per-boundary table: mirror the row #2 / row #4 wording changes to keep the two tables in lockstep.

**Pros.** No new surface; the existing row #2 destination ("the model"; `tool_loop` slot counted; loop continues) is the most natural one for any model-issued `tool_use` failure, and matches how a Pi-tool depth-6 args breach already behaves. Implementers do not need to special-case `.loom`-callable adapters at the depth-walk site.

**Cons.** The `.loom`-callable adapter's surface for *other* infra-side validation failures (per `tool-calls.md`) is `Err(InvokeInfraError { cause: "validation", ... })` — collapsing the depth case into row #2 means depth-6 args and (say) type-mismatch args from the same model call route to different destinations. This is a real asymmetry the fix should acknowledge.

**Risks.** None new; the asymmetry above already exists for Pi-tool args (depth-6 routes to the model; a tool whose `execute` later throws on validation routes elsewhere).

### Option B — add a third sub-arm to row #4

**Approach.** Keep row #2 generic to Pi tools and add an explicit third sub-arm to row #4 for the model-driven `.loom`-callable args case, with the same destination as row #2 (the model, via tool-error result; `tool_loop` slot counted; loop continues). This keeps every `params` validation site in one row.

**Spec edits.**
- `schema-subset.md` enforcement-point #4: replace *"whether bound via the binder or supplied directly by `invoke`"* with *"whether bound via the binder, supplied directly by `invoke(...)`, or supplied by the model as a `tool_use` `arguments` payload against a registered `.loom` callable"*.
- `schema-subset.md` table row #4: append a third call-site clause to the surface cell — *"Model-driven `.loom`-callable args: tool-error result fed back to the model as the next user turn per [Query — Tool calls during a query](./query.md); the round counts against `tool_loop.max_rounds` and the loop continues (same routing as row #2)."*.
- `schema-subset.md` table row #2 right cell: restrict the framing to *"(Pi-tool args)"* and forward to row #4's model-driven sub-arm for `.loom`-callable args.
- `hard-ceilings/ceilings-3-and-4.md` per-boundary table: mirror.

**Pros.** Keeps `params` validation surface enumeration in one place — a reader looking for "where does my `params` payload get validated?" finds all three sites in row #4. Matches the way `frontmatter-fields-a.md` enumerates the three `params` call sites today.

**Cons.** Row #4 becomes the longest cell in the table, with a sub-arm whose destination column differs from the row's other arms. Splitting row #4 across two destinations ("loom code" for `invoke(...)`; "the model" for the new sub-arm) means the *Destination* column for row #4 has to read *"depends on call site"* with three distinct values, not two.

**Risks.** The reader has to cross-check row #4 against row #2 to confirm the surfaces are identical for the model-driven sub-arms; a future edit that touches row #2 without touching row #4 could let them drift.

### Recommendation

**Option A.** The defect is essentially a definitional choice between "tool-call args" and "`params`" as the categorisation axis, and row #2 already has the right destination semantics for the case in question. Folding the third site into row #2 keeps the table's column invariants clean (row #4 stays "loom code"-destinationed; row #2 stays "the model"-destinationed) and matches the principle that the *runtime mechanism that fires the depth-walk* — the adapter's AJV boundary on the model's `arguments` payload — is what categorises the row, not the downstream semantic role of those bytes. Implementers must watch one edge case: the adapter's *other* validation surfaces for `.loom` callables (`Err(InvokeInfraError { cause: "validation", ... })` for code-driven calls per `tool-calls.md`) remain untouched — the collapse applies only to model-driven args, only to the depth-walk's enforcement site.

## Consequence

**Severity:** correctness

Two reasonable implementers will route the same depth-6 model-issued `tool_use` against a registered `.loom` callable differently: one returns the failure to the model as a tool-error result (row-#2 reading), the other surfaces it to loom code as `Err(InvokeInfraError { cause: "validation", ... })` (row-#4 `invoke(...)` reading). These are different observable program behaviours — the first lets the loop continue and consumes a `tool_loop` slot; the second terminates the query with an `Err`.
