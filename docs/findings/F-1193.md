## Problem

The `loom/parse/type-alias-cycle` row in `code-registry-parse.md` declares its Message template as `type-alias cycle: <path>`, but the emitting site is a chain of schema-name identifiers (`schema X = Y; schema Y = X`), not a path-literal source span. The §5 `<path>`/`<file>` rendering rule in `placeholder-rendering-b.md` ("the literal text inside the path-literal quotes") presupposes a path-literal source and does not fit, and the §7 "Path-shaped" sub-rule covers file-path stems where the ` → ` join and head-repetition are owned by the registry template — not bundled inside a single token as they are here. No placeholder-rendering category governs the intended rendering `type-alias cycle: X → Y → X` that `schemas.md` pins, so the closure clause's claim that the categories cover every registry placeholder is locally false. A conformant test cannot derive this row's expected Message string from the placeholder-rendering surface alone (DIAG-4).

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `de05433` (2026-05-04, *"pi-loom spec: resolve 'Schema declarations: empty bodies, alias cycles, and discriminator literal type unspecified'"*), `1929ed1` (2026-05-05, *"pi-loom plan: resolve 'Type-alias cycle detection has no plan leaf'"*), `61ea1a5` (2026-05-06, *"pi-loom spec: resolve 'Diagnostic message placeholders have no defined rendering'"*).
**History:** `de05433` introduced the `loom/parse/type-alias-cycle` concept in `spec_topics/schemas.md` with an inline rendering example *`"type-alias cycle: X → Y → X"`* — a type-name chain joined by ` → ` with the head repeated at the tail. One day later, `1929ed1` added the corresponding diagnostic-registry row to `spec_topics/diagnostics.md` with Message template `` `type-alias cycle: <path>` ``, bundling the whole chain into a single `<path>` placeholder (`git show 1929ed1 -- spec_topics/diagnostics.md` shows the row addition; no rendering rule for `<path>` existed at that point). The next day, `61ea1a5` introduced the eight placeholder-rendering categories and defined `<path>` in §5 as *"the literal text inside the path-literal quotes, with no realpath normalisation, no symlink resolution, no scheme prefixing"* — a definition keyed to path-literal source spans, narrower than the registry's bundled use. The defect is the interaction: neither commit on its own was wrong (the schemas.md rendering was self-consistent; the registry row predates §5's rule; §5's rule was correct for every other `<path>` site), but their composition leaves the `type-alias-cycle` row with a Message template whose placeholder is governed by a rule that does not fit the emitting site. The 2026-06-04 split commit `f5e89f4` only relocated the registry row to `code-registry-parse.md` and the §5 rule to `placeholder-rendering-b.md` without changing the substance; it did not introduce the defect.

## Solution approach

Introduce a placeholder whose rendering rule governs the type-alias-cycle chain at the surface where the other placeholders live. In `code-registry-parse.md`'s `loom/parse/type-alias-cycle` row, replace the trailing `<path>` token in the Message template with a chain-specific token, and add that token and its rendering rule to `placeholder-rendering-b.md` §7: identifier-shaped schema names (per [Lexical — Identifiers]) joined by ` → ` (a single U+2192 code point, not the ASCII `->` digraph) in walk order, with the head repeated at the tail to close the cycle. Add a §7 test vector covering `schema X = Y; schema Y = X` rendering `type-alias cycle: X → Y → X` so the `V7c-T` expected string is derivable from the placeholder surface.

## Solution constraints

- Do not modify the shared §5 `<path>`/`<file>` rule in `placeholder-rendering-b.md`; it governs every other path-literal emitting site and must stay keyed to path-literal source spans.
