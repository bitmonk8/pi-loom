## Finding

The `loom/parse/type-alias-cycle` registry row (`docs/spec_topics/diagnostics/code-registry-parse.md` line 95) declares its Message template as `type-alias cycle: <path>`, reusing the `<path>` token name as a single bundled placeholder for the cycle. By the §5 closure clause in `placeholder-rendering-a.md` ("every `<…>` token … is either (a) a placeholder enumerated in categories 1–7 (rendered byte-identically per its category's rule) …"), every occurrence of `<path>` must be rendered by §5's rule for `<path>`/`<file>`: *"the literal text inside the path-literal quotes, with no realpath normalisation, no symlink resolution, no scheme prefixing."* That rule presupposes a path-literal source span — there is none at the emitting site. The diagnostic fires on a chain of schema-name identifiers (`schema X = Y; schema Y = X`), and `schemas.md` (line 143) pins the *intended* rendering as `"type-alias cycle: X → Y → X"`: the participating type names joined by ` → ` with the head repeated at the tail. Neither §5's path-literal rule nor §7's path-shaped sub-rule (which covers `<A>, <B>` for `loom/load/import-cycle` / `loom/load/invocation-cycle`, but in those rows the tokens are file-path stems, *not* type-name identifiers, and the join string is interpolated by the registry template itself — `<A> → <B> → <A>` — not produced by the placeholder) covers the case.

The §7 "Path-shaped" sub-rule's parallel construction is also misleading on a second axis: `loom/load/import-cycle` and `loom/load/invocation-cycle` keep each cycle hop as a separate placeholder (`<A>`, `<B>`), letting the registry template own the `→` separator and the head-repetition. The `loom/parse/type-alias-cycle` row instead bundles the whole chain into a single `<path>` placeholder, so the join string and the head-repetition convention sit *inside* the placeholder's render — but no §5/§7 rule pins that grammar. A conforming `V7c-T` test cannot construct an expected message string from the placeholder-rendering surface alone; it has to consult `schemas.md`'s informal example.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `de05433` (2026-05-04, *"pi-loom spec: resolve 'Schema declarations: empty bodies, alias cycles, and discriminator literal type unspecified'"*), `1929ed1` (2026-05-05, *"pi-loom plan: resolve 'Type-alias cycle detection has no plan leaf'"*), `61ea1a5` (2026-05-06, *"pi-loom spec: resolve 'Diagnostic message placeholders have no defined rendering'"*).
**History:** `de05433` introduced the `loom/parse/type-alias-cycle` concept in `spec_topics/schemas.md` with an inline rendering example *`"type-alias cycle: X → Y → X"`* — a type-name chain joined by ` → ` with the head repeated at the tail. One day later, `1929ed1` added the corresponding diagnostic-registry row to `spec_topics/diagnostics.md` with Message template `` `type-alias cycle: <path>` ``, bundling the whole chain into a single `<path>` placeholder (`git show 1929ed1 -- spec_topics/diagnostics.md` shows the row addition; no rendering rule for `<path>` existed at that point). The next day, `61ea1a5` introduced the eight placeholder-rendering categories and defined `<path>` in §5 as *"the literal text inside the path-literal quotes, with no realpath normalisation, no symlink resolution, no scheme prefixing"* — a definition keyed to path-literal source spans, narrower than the registry's bundled use. The defect is the interaction: neither commit on its own was wrong (the schemas.md rendering was self-consistent; the registry row predates §5's rule; §5's rule was correct for every other `<path>` site), but their composition leaves the `type-alias-cycle` row with a Message template whose placeholder is governed by a rule that does not fit the emitting site. The 2026-06-04 split commit `f5e89f4` only relocated the registry row to `code-registry-parse.md` and the §5 rule to `placeholder-rendering-b.md` without changing the substance; it did not introduce the defect.

## Solution Space

**Shape:** multiple

### Option A — Introduce a new `<cycle>` placeholder under §7

**Approach.** Rename the placeholder in the `loom/parse/type-alias-cycle` row from `<path>` to a new `<cycle>` token and pin its rendering as a fresh sub-rule under §7 (closed-enum / identifier-derived placeholders): the participating schema names (identifier-shaped per [Lexical — Identifiers]) joined by ` → ` (U+2192 with a single space on each side) in walk order, with the first element repeated as the last element to close the cycle.

**Spec edits.**

- `code-registry-parse.md` row 95: change the Message template's trailing token from `<path>` to `<cycle>`.
- `placeholder-rendering-b.md` §7 "**Placeholders.**" list: add `<cycle>` to the enumeration.
- `placeholder-rendering-b.md` §7 rule list: add a new bullet — `**Cycle-shaped** (`<cycle>`) — identifier-shaped schema names joined by ` → ` (U+2192, single space on each side) in walk order, with the first element repeated at the tail to close the cycle.`
- `placeholder-rendering-b.md` §7 "**Test vectors.**": add a vector — `A `schema X = Y; schema Y = X` cycle renders `type-alias cycle: X → Y → X`.`
- `placeholder-rendering-a.md` §5 / §7: no change to existing `<path>` rule needed; the closure clause (b)–(e) carve-outs are not affected.

**Pros.** Keeps each placeholder's rule single-valued (no overloading of `<path>`); the `<cycle>` token's name matches its semantics; the join string and head-repetition convention are pinned where every other placeholder lives; sibling cycle diagnostics in `loom/load/import-cycle` / `loom/load/invocation-cycle` continue to inline the `→` in the template, but a future cleanup could migrate them onto `<cycle>` for consistency.

**Cons.** Introduces one more closed-enum placeholder to the eight-category surface (the closure clause needs no edit, but `V7c-T` gains one more byte-identical vector to maintain). Under GOV-7 / GOV-8 the addition is a versioned change to the placeholder surface, though it lands as part of pre-1.0 spec settling.

**Risks.** None notable; `<cycle>` is the natural reading of the bundled chain.

### Option B — Align with the sibling cycle diagnostics: unbundle into per-hop placeholders

**Approach.** Restructure the row to mirror `loom/load/import-cycle` / `loom/load/invocation-cycle`: replace the bundled `<path>` with per-hop placeholders inlined in the template, e.g. `` `type-alias cycle: <X> → <Y> → <X>` ``, and lean on §7's identifier-shaped sub-rule (which already covers `<X>`) for the per-hop rendering. The join string and head-repetition then live in the registry template itself, not inside a placeholder.

**Spec edits.**

- `code-registry-parse.md` row 95: change the Message template from `type-alias cycle: <path>` to `type-alias cycle: <X> → <Y> → <X>` (or an N-hop equivalent — see Cons).
- `placeholder-rendering-b.md` §7: no change to the placeholder lists or rules; `<X>` is already identifier-shaped.
- Add a clarifying note in the row's Trigger column (or in `schemas.md`) that the template generalises to N-hop cycles (`X → Y → Z → X`); the registry template itself uses a 2-hop instance as the byte-identical surround per DIAG-4.

**Pros.** Zero new placeholders; the rendering rule is unchanged; the row visually parallels its two sibling cycle diagnostics; the registry template is self-explanatory.

**Cons.** N-hop cycles longer than the template's instantiated arity (2 hops) have an under-specified rendering — the template only literally fixes `<X> → <Y> → <X>`, and DIAG-4's byte-identical contract assumes the template *is* the rendering. An implementer would need a separate rule pinning how `<X> → <Y> → <X>` expands when the cycle has 3+ hops, which reintroduces the very problem Option A solves cleanly. The sibling diagnostics (`import-cycle`, `invocation-cycle`) share this latent gap, but they are not flagged in this review.

**Risks.** Re-opens the N-hop generality question on three rows instead of one; likely produces a follow-up finding.

### Recommendation

Option A. Introduce `<cycle>` as a new §7 sub-rule, with the join string ` → ` and the head-repeated-at-tail convention pinned at the placeholder, plus the test vector `A `schema X = Y; schema Y = X` cycle renders `type-alias cycle: X → Y → X`.` This collapses the rule to a single normative location, decouples the type-alias cycle's wire format from the unrelated file-path-literal rule, and admits arbitrary N-hop cycles by construction. Implementer must watch: the U+2192 arrow must be a single code point (not the ASCII `->` digraph), and the head-repetition convention applies even on 1-hop self-cycles (`schema X = X` renders `type-alias cycle: X → X`).

## Consequence

**Severity:** correctness

Two conformant implementers reading only the diagnostics surface will diverge on the rendered Message for `loom/parse/type-alias-cycle`: one may emit the type-name chain (consulting `schemas.md`'s informal example), another may treat `<path>` literally and emit a file path, an empty string, or a fault. DIAG-4 ("the Message column is normative — every asserting test sources its expected string from the registry, not prose") cannot be satisfied for this row because the registry's expected string is not derivable from the placeholder-rendering categories. The closure clause's claim that the eight categories cover every registry placeholder is locally false.
