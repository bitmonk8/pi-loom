## Finding

`session-shutdown-semantics.md` sub-step 3 ("Await subagent disposal") states that the bounded await arms a `Clock.setTimeout(SHUTDOWN_AWAIT_CAP_MS, …)` deadline timer and then asserts: *"The pending `Clock.setTimeout` handle is cleared on the success path so a reload that completes early does not leak a timer onto the about-to-be-invalidated runtime."* This is a normative resource-hygiene requirement on the early-completion path (all `disposeBarrier`s settle before the 2000 ms deadline fires), but no fixture obligation pins it.

The cluster's only acceptance-criteria block — `session-only-degraded-state.md` items (a)–(m) — covers the `runtime-degraded`/`teardown-step-failed` emission matrix and drain-state transitions; none observes the deadline-timer handle being cleared. The paired test leaf `V9g-T` enumerates sub-step 3 only as "awaits every in-flight entry's `disposeBarrier` to settle via `Promise.allSettled`, bounded by `SHUTDOWN_AWAIT_CAP_MS`" and "`reload-teardown-timeout` fires at the cap" — i.e. it tests the timeout (deadline-fires) path and the settle-all path, but not the early-completion clear. "Does not leak a timer" is an absence claim with no stated observable outcome, so an implementation that omits the `clearTimeout` on the success path would pass every existing fixture while leaving a 2000 ms timer pending against a runtime Pi is about to invalidate.

The behavior is observable through the existing seam: the injected `Clock`/`FakeClock` (`host-interfaces-services.md`) exposes `clearTimeout`, and `FakeClock` can record clear calls and treat already-fired handles as no-ops, so a fixture can assert the success-path clear directly without new machinery.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** `52219c4` (2026-05-06, "Address all 41 spec-review findings")
**History:** Sub-step 3 was first authored by `5e8b499` (2026-05-06, "resolve 'Hot-reload teardown contract …'") with the await bounded by `Math.min(2000ms, remainingShutdownBudget)` and **no** explicit deadline-timer handle — so there was no timer to leak and no hygiene claim. `52219c4` reformulated the bound to an explicit `Clock.setTimeout(SHUTDOWN_AWAIT_CAP_MS, …)` deadline and introduced the "pending handle is cleared on the success path so … does not leak a timer" sentence (`git log -G"reload that completes early does not leak a timer"` / `git show 52219c4`). The behavioral claim entered there without an accompanying acceptance criterion, and the gap persisted through the `f5e89f4` corpus cap-split and the `c3de47a` relocation of the teardown block into `session-shutdown-semantics.md`; the later-added (a)–(m) acceptance criteria never covered it.

## Solution Space

**Shape:** single

### Recommendation

Add one fixture obligation to the *Acceptance criteria* block in `session-only-degraded-state.md` (the cluster's canonical acceptance-criteria home) covering the early-completion path, and reference it from `V9g-T`. State it against the existing `Clock`/`FakeClock` seam, e.g.: "On the early-completion path — every in-flight `disposeBarrier` settles before `Clock.setTimeout(SHUTDOWN_AWAIT_CAP_MS, …)` fires — the handle returned by that `setTimeout` MUST be passed to `Clock.clearTimeout` before sub-step 3 returns (observable in the `FakeClock` clear log), and no pending timer callback fires afterward." Edge cases the implementer must watch: the assertion is on the success/early-completion path only (the timeout path is already covered by `reload-teardown-timeout`); and `FakeClock.clearTimeout` is a no-op on an already-fired handle, so the fixture must drive settlement strictly before advancing the fake clock past the deadline to exercise the clear rather than the fire.

## Consequence

**Severity:** advisory

The success-path timer-clear is stated normatively, so two implementers are unlikely to diverge on the requirement itself; the gap is purely test coverage. Without a fixture, an implementation that drops the `clearTimeout` ships a pending 2000 ms timer that fires on an invalidated runtime after every fast reload, and no red test surfaces it.
