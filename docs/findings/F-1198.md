## Finding

The **Stop-reason classification** subsection of `provider-error-mapping.md` and the surrounding error/seed-field paragraphs carry five independent defects that the runtime classifier and the V18c bump-time gates both consume. They share a file but not an edit:

1. **"non-terminal stop reason" contradicts "content-filter terminator".** The catch-all bullet reads "Any other non-terminal stop reason — including a content-filter terminator, or any stop reason the runtime does not otherwise recognise — maps to `TransportError` with `retryable: false`". A content-filter stop is, by construction, *terminal* — the model has stopped producing — so calling the same value "non-terminal" inside the same clause is self-contradictory. The bullet's actual scope is "any `stopReason` other than `"length"` and other than the `end_turn` / `stop` / `tool_use` turn-boundary set the tool loop already consumes" (i.e. the set complement of the preceding bullet plus the section intro's turn-boundary list).
2. **The `"length"` mapping cites a pi-ai vocabulary token without a fixture pin.** The output-boundary bullet asserts "pi-ai surfaces output truncated at the model's length boundary as `stopReason: "length"`". `"length"` is in fact a member of pi-ai's `StopReason = "stop" | "length" | "toolUse" | "error" | "aborted"` literal-union (declared at `dist/types.d.ts` in `@earendil-works/pi-ai`), so the claim is currently true, but no fixture or editorial-checklist item pins it: a pi-ai minor that renamed `"length"` (or split it into `"max_tokens"` / `"output_limit"`) would silently dead-letter every output-overflow into the catch-all `TransportError` branch with neither the SDK surface-inventory test nor the existing checklist items (m), (af), (al) — all keyed on `stopReason: "error"`, not on `"length"` — going red.
3. **The seed-field `Api`-coverage assertion as specified will red on day one.** The **Provider seed-field mapping** paragraph asserts "the build-time `Api`-coverage assertion … enumerates pi-ai's exposed `Api` literal-union values and asserts every value appears as a row key in the seed-field table constant", and the four-row table keys on `openai-completions`, `mistral`, `anthropic-messages`, `amazon-bedrock`. The actual pi-ai `KnownApi` literal-union at the loom 1.0 Pi-SDK pin has nine members — `openai-completions`, `mistral-conversations`, `openai-responses`, `azure-openai-responses`, `openai-codex-responses`, `anthropic-messages`, `bedrock-converse-stream`, `google-generative-ai`, `google-vertex` (plus the `(string & {})` open arm). Two of the table's keys (`mistral`, `amazon-bedrock`) are not even members of that union; the other five `KnownApi` values have no row at all. The spec gives no default-omit rule for unenumerated members, no rule for the open `(string & {})` arm, and no mapping from the spec's short provider names to pi-ai's `Api` tokens — so the assertion as written is unimplementable against the actual surface.
4. **The mistral / bedrock rows say "body matching", contradicting the all-matches-against-`errorMessage` rule.** The *Classifier input surface* note pins "every body-wording match in the table is a match against [the `AssistantMessage.errorMessage`] string", and the anthropic-messages and openai-completions rows now read `errorMessage matching /.../i`. The mistral and amazon-bedrock rows still read `body matching /.../i` and `ValidationException with body matching /.../i` respectively. There is no "body" surface for them to match — pi-ai exposes no parsed JSON body — so an implementer following the table literally has no input to feed the regex.
5. **The capability-gap `retryable:false` pointer dangles cross-file.** The **`TransportError.retryable` population** paragraph says "The unsupported-provider typed-query case described under **Provider compatibility for typed queries** above is the one path that pins `retryable: false` independent of HTTP class". *Provider compatibility for typed queries* lives in `conversation-drive.md`, not in `provider-error-mapping.md`; the word "above" is wrong for the post-split file layout, and a reader of this shard has no path to the referenced rule.

The five defects share a page and a classifier surface but do not share a fix. Each lands on its own clause; bundling them produces a diff broad enough to redden every shard-08 lens on the next pass.

The durable shape of the defect is the absence of a single-author pass over the post-split page that reconciles classifier inputs (`errorMessage` everywhere), classifier outputs (`StopReason` token vocabulary), and the seed-field assertion (against the real `KnownApi` union).

## Solution Space

**Shape:** multiple

Each option below is a separately landable fix. Address them in the listed order — A first (smallest, scope-bounding), then B–E — so each subsequent edit lands on a baseline that the previous fix has already stabilised. Bundling them produces a single oversized diff that empirically destabilises the per-finding fix loop.

### Option A — Repair the "non-terminal stop reason" catch-all wording (defect 1)

**Approach.** Rewrite the catch-all bullet of § **Stop-reason classification** to name the excluded sets directly instead of qualifying with "non-terminal". Drop the "content-filter terminator" example (it contradicts the qualifier) and replace with a non-self-contradictory exemplar list.

**Spec edits.** In `provider-error-mapping.md` § **Stop-reason classification**, replace the second bullet's lead clause:

> Any other non-terminal stop reason — including a content-filter terminator, or any stop reason the runtime does not otherwise recognise — maps to `TransportError` with `retryable: false` …

with:

> Any `stopReason` other than `"length"` (covered by the preceding bullet) and other than the `end_turn` / `stop` / `tool_use` turn-boundary set the tool loop consumes (per the section intro) — including a content-filter terminator and any stop reason the runtime does not otherwise recognise — maps to `TransportError` with `retryable: false` …

**Pros.** Single-bullet edit; resolves F-0750 in the same touch; no plan or fixture impact.
**Cons.** None.
**Risks.** None.

### Option B — Pin the `StopReason: "length"` vocabulary via an editorial-review item (defect 2)

**Approach.** Add one SHOULD-level item to the *Editorial-review checklist for unpinned host presuppositions* (`version-bump-step2.md`) covering the `StopReason: "length"` member the output-boundary bullet keys on, mirroring existing items (m) / (af) / (al) which cover `StopReason: "error"`. No change to the runtime classifier.

**Spec edits.** Append a new item after (al) in `version-bump-step2.md` — *`StopReason: "length"` output-boundary membership* — pointing at `provider-error-mapping.md` § **Stop-reason classification**, first bullet, instructing the contributor to confirm against the candidate `@earendil-works/pi-ai` minor that `StopReason` still includes the `"length"` member and that the per-provider adapter still surfaces output-boundary truncation as `stopReason: "length"`. Add a forward-reference from the output-boundary bullet to the new checklist item.

**Pros.** Symmetric with existing checklist items; zero runtime-classifier change; gives the unpinned vocabulary an observable artefact.
**Cons.** Adds one more item to the already-long (a)–(al) list.
**Risks.** None.

### Option C — Specify the seed-field `Api`-coverage assertion against the real `KnownApi` union (defect 3)

**Approach.** Rewrite the *Api-coverage assertion* sentence and the four-row table so the assertion's contract matches what pi-ai actually exposes. Three sub-choices, of which the recommendation picks one:

- **C1 (preferred): keyed-by-`KnownApi`-token, default-omit-rule.** Re-key the table on pi-ai's actual `KnownApi` tokens (so `mistral-conversations` not `mistral`, `bedrock-converse-stream` not `amazon-bedrock`, plus rows for `openai-responses`, `azure-openai-responses`, `openai-codex-responses`, `google-generative-ai`, `google-vertex`). Add an explicit "default: omitted" rule for the open `(string & {})` arm. Reword the assertion to "every member of `KnownApi` appears as a row key; the open `(string & {})` arm receives the default-omitted treatment without enumeration".
- **C2: closed-supported-set marker, default-omit-rule.** Leave the four rows alone, demote the assertion to "every member of `KnownApi` appears as a row key *or* falls under the default-omitted rule", and explicitly mark each absent `KnownApi` value (and the open arm) as default-omitted. Cheaper edit but loses the per-`Api` audit signal.
- **C3: name-map indirection.** Keep the table on the spec's short names, add a separate `pi-ai-token → table-key` map, assert coverage on the map. Strictly more moving parts and not recommended.

**Spec edits (C1).** In `provider-error-mapping.md` § **Provider seed-field mapping**: re-key the four existing rows to the `KnownApi` tokens; add five new rows covering the remaining `KnownApi` members each marked `omitted`; add a "*Default for unknown `Api` values.* Members of pi-ai's open `(string & {})` arm — values not in `KnownApi` — receive no seed key (treated as `omitted`)" sentence. Rewrite the assertion sentence per the C1 wording above. Update `V18c-version-bump-checklist.md` and `V18c-T-version-bump-checklist.md` to cite the rewritten assertion contract.

**Pros (C1).** Aligns the assertion with the actual pi-ai surface so it can land green; preserves the per-`KnownApi`-member audit signal; makes the open-arm policy explicit.
**Cons (C1).** Five new table rows; minor V18c plan-leaf wording update.
**Risks.** None — the runtime payload for the five new rows is `omitted`, identical to the existing `anthropic-messages` / `amazon-bedrock` rows, so the runtime classifier is unchanged.

### Option D — Make the mistral / bedrock overflow rows match against `errorMessage` (defect 4)

**Approach.** Rewrite the two table rows to read `errorMessage matching ...`, identical in form to the anthropic-messages and openai-completions rows that `513a5cc` already converted. Strip "body" / "`ValidationException` with body" — `errorMessage` is the only surface the classifier reads, per the *Classifier input surface* note.

**Spec edits.** In `provider-error-mapping.md`, replace the two row predicates:

- `mistral`: `HTTP 400 with body matching /context.*length/i` → `HTTP 400 with errorMessage matching /context.*length/i`
- `amazon-bedrock`: `ValidationException with body matching /(input is too long|context window)/i` → `errorMessage matching /(input is too long|context window)/i` (drop the `ValidationException` framing or move it into the pi-ai-formatter prose; the *Provider-owned-wording presupposition* paragraph already names `ValidationException` as one of the inputs the formatter consumes)

**Pros.** Restores the all-matches-against-`errorMessage` invariant the *Classifier input surface* note already pins.
**Cons.** None.
**Risks.** Need to confirm the captured-error-body corpus the bump-procedure fixtures consume already represents the post-formatter `errorMessage` strings for mistral / bedrock; if it records raw bodies, the corpus needs a re-capture pass — but that obligation is already owned by item (i) of the editorial-review checklist and `F-1049`.

### Option E — Fix the dangling "above" pointer to *Provider compatibility for typed queries* (defect 5)

**Approach.** Replace "described under **Provider compatibility for typed queries** above" in the **`TransportError.retryable` population** paragraph with an explicit cross-file link to `conversation-drive.md` § **Provider compatibility for typed queries**.

**Spec edits.** In `provider-error-mapping.md`, rewrite the capability-gap clause as: "The unsupported-provider typed-query case described under [Conversation drive — Provider compatibility for typed queries](./conversation-drive.md#provider-compatibility-for-typed-queries) is the one path that pins `retryable: false` independent of HTTP class …".

**Pros.** One-line edit; matches the cross-file link style the *Classifier input surface* and *Provider-owned-wording presupposition* paragraphs already use.
**Cons.** None.
**Risks.** Confirm the anchor name on the target section; the conversation-drive page uses an `id="provider-compatibility-for-typed-queries"`-style anchor or the section's auto-anchor — the link target must resolve.

### Recommendation

Land the five options in the order **A → E → D → B → C**, each as a separate spec-fix commit:

1. **A** first — pure wording, zero downstream surface; smallest baseline-stabilising edit, and it resolves F-0750 along the way.
2. **E** — single-line cross-file pointer fix; clears the broken "above" referent so reviewers reading the file end-to-end aren't tripped on it before reaching D and B.
3. **D** — table-row wording fix; small and contained.
4. **B** — new editorial-review checklist item; touches `version-bump-step2.md` independently of the four classifier-surface fixes above.
5. **C** (sub-option **C1**) — largest edit; pin it last so the table re-keying and the V18c-leaf wording update land on a stable baseline. C1 is preferred because it preserves the per-`KnownApi` audit signal the assertion is intended to deliver.

Edge cases the implementer of C must watch:

- pi-ai's `KnownApi` is `KnownApi | (string & {})`; the assertion contract must explicitly cover the open arm (the table cannot enumerate it). C1 handles this with the default-omit rule.
- The runtime classifier reads `Model<Api>.api` (the resolved provider's `Api` token); make sure the spec's seed-field lookup is explicitly on `Model<Api>.api`, not on the spec's old short-name vocabulary. `binder-inference.md` already cites `Model<Api>.api`, so the seed-field paragraph's prose needs to align with that.
- The two `V18c-*-version-bump-checklist.md` leaves both reference the assertion contract — co-edit them in the same commit as the C1 spec change so the leaves' **Ships when** wording matches.

## Consequence

**Severity:** correctness

Two implementers reading the file in isolation diverge on at least three independent axes — what `stopReason` values land on `ContextOverflowError` vs `TransportError`, what input the mistral/bedrock overflow regexes consume, and what the build-time `Api`-coverage assertion is actually checking against. Defect 3 in particular: an implementer who wires the assertion against pi-ai's real `KnownApi` literal-union at the loom 1.0 Pi-SDK pin will get a red build immediately (five `KnownApi` values absent from the table, plus `mistral` / `amazon-bedrock` row keys that don't even appear in the union), and the spec offers no rule that resolves it. Defects 1 and 5 are pure clarity gaps that an attentive reader works around; defects 2, 3, and 4 are spec/SDK-surface mismatches that the bump-procedure gate cannot itself catch.
