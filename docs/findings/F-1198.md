## Finding

The **Stop-reason classification** subsection of `provider-error-mapping.md` and the surrounding error/seed-field paragraphs carry five independent defects that the runtime classifier and the V18c bump-time gates both consume. They share a file but not an edit:

1. **"non-terminal stop reason" contradicts "content-filter terminator".** The catch-all bullet reads "Any other non-terminal stop reason — including a content-filter terminator, or any stop reason the runtime does not otherwise recognise — maps to `TransportError` with `retryable: false`". A content-filter stop is, by construction, *terminal* — the model has stopped producing — so calling the same value "non-terminal" inside the same clause is self-contradictory. The bullet's actual scope is "any `stopReason` other than `"length"` and other than the `end_turn` / `stop` / `tool_use` turn-boundary set the tool loop already consumes" (i.e. the set complement of the preceding bullet plus the section intro's turn-boundary list).
2. **The `"length"` mapping cites a pi-ai vocabulary token without a fixture pin.** The output-boundary bullet asserts "pi-ai surfaces output truncated at the model's length boundary as `stopReason: "length"`". `"length"` is in fact a member of pi-ai's `StopReason = "stop" | "length" | "toolUse" | "error" | "aborted"` literal-union (declared at `dist/types.d.ts` in `@earendil-works/pi-ai`), so the claim is currently true, but no fixture or editorial-checklist item pins it: a pi-ai minor that renamed `"length"` (or split it into `"max_tokens"` / `"output_limit"`) would silently dead-letter every output-overflow into the catch-all `TransportError` branch with neither the SDK surface-inventory test nor the existing checklist items (m), (af), (al) — all keyed on `stopReason: "error"`, not on `"length"` — going red.
3. **The seed-field `Api`-coverage assertion as specified will red on day one.** The **Provider seed-field mapping** paragraph asserts "the build-time `Api`-coverage assertion … enumerates pi-ai's exposed `Api` literal-union values and asserts every value appears as a row key in the seed-field table constant", and the four-row table keys on `openai-completions`, `mistral`, `anthropic-messages`, `amazon-bedrock`. The actual pi-ai `KnownApi` literal-union at the loom 1.0 Pi-SDK pin has nine members — `openai-completions`, `mistral-conversations`, `openai-responses`, `azure-openai-responses`, `openai-codex-responses`, `anthropic-messages`, `bedrock-converse-stream`, `google-generative-ai`, `google-vertex` (plus the `(string & {})` open arm). Two of the table's keys (`mistral`, `amazon-bedrock`) are not even members of that union; the other five `KnownApi` values have no row at all. The spec gives no default-omit rule for unenumerated members, no rule for the open `(string & {})` arm, and no mapping from the spec's short provider names to pi-ai's `Api` tokens — so the assertion as written is unimplementable against the actual surface.
4. **The mistral / bedrock rows say "body matching", contradicting the all-matches-against-`errorMessage` rule.** The *Classifier input surface* note pins "every body-wording match in the table is a match against [the `AssistantMessage.errorMessage`] string", and the anthropic-messages and openai-completions rows now read `errorMessage matching /.../i`. The mistral and amazon-bedrock rows still read `body matching /.../i` and `ValidationException with body matching /.../i` respectively. There is no "body" surface for them to match — pi-ai exposes no parsed JSON body — so an implementer following the table literally has no input to feed the regex.
5. **The capability-gap `retryable:false` pointer dangles cross-file.** The **`TransportError.retryable` population** paragraph says "The unsupported-provider typed-query case described under **Provider compatibility for typed queries** above is the one path that pins `retryable: false` independent of HTTP class". *Provider compatibility for typed queries* lives in `conversation-drive.md`, not in `provider-error-mapping.md`; the word "above" is wrong for the post-split file layout, and a reader of this shard has no path to the referenced rule.

The five defects share a page and a classifier surface but do not share a fix. Each lands on its own clause; bundling them produces a diff broad enough to redden every shard-08 lens on the next pass.

## Solution Space

**Shape:** single

### Recommendation

Apply the following five spec edits to `provider-error-mapping.md` (and the named gate/checklist files), each as a separate spec-fix commit landed in the order given so each edit builds on a baseline the previous edit has stabilised.

1. **Repair the catch-all stop-reason bullet.** In `provider-error-mapping.md` § **Stop-reason classification**, replace the second bullet's lead clause:

   > Any other non-terminal stop reason — including a content-filter terminator, or any stop reason the runtime does not otherwise recognise — maps to `TransportError` with `retryable: false` …

   with:

   > Any `stopReason` other than `"length"` (covered by the preceding bullet) and other than the `end_turn` / `stop` / `tool_use` turn-boundary set the tool loop consumes (per the section intro) — including a content-filter terminator and any stop reason the runtime does not otherwise recognise — maps to `TransportError` with `retryable: false` …

   This is a single-bullet wording edit with no plan or fixture impact, and it resolves F-0750 in the same touch.

2. **Fix the dangling "above" pointer.** In `provider-error-mapping.md` § **`TransportError.retryable` population**, rewrite the capability-gap clause as: "The unsupported-provider typed-query case described under [Conversation drive — Provider compatibility for typed queries](./conversation-drive.md#provider-compatibility-for-typed-queries) is the one path that pins `retryable: false` independent of HTTP class …". Confirm the anchor resolves against `conversation-drive.md` (the section's `id="provider-compatibility-for-typed-queries"` / auto-anchor). This matches the cross-file link style the *Classifier input surface* and *Provider-owned-wording presupposition* paragraphs already use.

3. **Match the mistral / bedrock overflow rows against `errorMessage`.** In `provider-error-mapping.md`, replace the two table row predicates so they read against `errorMessage` (the only surface the classifier reads, per the *Classifier input surface* note):

   - `mistral`: `HTTP 400 with body matching /context.*length/i` → `HTTP 400 with errorMessage matching /context.*length/i`
   - `amazon-bedrock`: `ValidationException with body matching /(input is too long|context window)/i` → `errorMessage matching /(input is too long|context window)/i` (drop the `ValidationException` framing or move it into the pi-ai-formatter prose; the *Provider-owned-wording presupposition* paragraph already names `ValidationException` as one of the inputs the formatter consumes).

   Confirm the captured-error-body corpus the bump-procedure fixtures consume records post-formatter `errorMessage` strings for mistral / bedrock; if it records raw bodies, a re-capture pass is required — that obligation is already owned by item (i) of the editorial-review checklist and `F-1049`.

4. **Pin the `StopReason: "length"` vocabulary via an editorial-review item.** Append a new item after (al) in `version-bump-step2.md` — *`StopReason: "length"` output-boundary membership* — pointing at `provider-error-mapping.md` § **Stop-reason classification**, first bullet, instructing the contributor to confirm against the candidate `@earendil-works/pi-ai` minor that `StopReason` still includes the `"length"` member and that the per-provider adapter still surfaces output-boundary truncation as `stopReason: "length"`. Add a forward-reference from the output-boundary bullet to the new checklist item. This mirrors existing items (m) / (af) / (al) (which cover `StopReason: "error"`) and requires no runtime-classifier change.

5. **Specify the seed-field `Api`-coverage assertion against the real `KnownApi` union.** In `provider-error-mapping.md` § **Provider seed-field mapping**: re-key the four existing rows to pi-ai's actual `KnownApi` tokens (`mistral-conversations` not `mistral`, `bedrock-converse-stream` not `amazon-bedrock`, plus rows for `openai-responses`, `azure-openai-responses`, `openai-codex-responses`, `google-generative-ai`, `google-vertex`); add five new rows covering the remaining `KnownApi` members, each marked `omitted`; add a "*Default for unknown `Api` values.* Members of pi-ai's open `(string & {})` arm — values not in `KnownApi` — receive no seed key (treated as `omitted`)" sentence. Rewrite the coverage-assertion sentence to "every member of `KnownApi` appears as a row key; the open `(string & {})` arm receives the default-omitted treatment without enumeration". Update `V18c-version-bump-checklist.md` and `V18c-T-version-bump-checklist.md` in the same commit so their **Ships when** wording cites the rewritten assertion contract. Express the seed-field lookup explicitly on `Model<Api>.api` (the resolved provider's `Api` token), aligned with `binder-inference.md`, not on the old short-name vocabulary. The runtime payload for the five new rows is `omitted`, identical to the existing rows, so the runtime classifier is unchanged.

## Consequence

**Severity:** correctness

Two implementers reading the file in isolation diverge on at least three independent axes — what `stopReason` values land on `ContextOverflowError` vs `TransportError`, what input the mistral/bedrock overflow regexes consume, and what the build-time `Api`-coverage assertion is actually checking against. Defect 3 in particular: an implementer who wires the assertion against pi-ai's real `KnownApi` literal-union at the loom 1.0 Pi-SDK pin will get a red build immediately (five `KnownApi` values absent from the table, plus `mistral` / `amazon-bedrock` row keys that don't even appear in the union), and the spec offers no rule that resolves it. Defects 1 and 5 are pure clarity gaps that an attentive reader works around; defects 2, 3, and 4 are spec/SDK-surface mismatches that the bump-procedure gate cannot itself catch.
