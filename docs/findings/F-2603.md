## Problem

Both static lookup tables on `provider-error-mapping.md` — the **Provider error mapping** overflow-signature table and the **Provider seed-field mapping** table — head their leftmost column `Provider` while keying rows on `api`-shaped values (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`). This contradicts the `Model<Api>.provider` vs `Model<Api>.api` distinction in `host-interfaces-core.md`: an implementer keying these tables on `Model<Api>.provider` matches no row, silently downgrading every context-overflow response to `TransportError` and mis-naming or omitting the seed field. Separately, the `mistral` and `amazon-bedrock` overflow rows describe their match surface as "body matching", naming a parsed JSON error body, while the `#classifier-input-surface` note establishes loom receives overflow wording only as the `AssistantMessage.errorMessage` string — the `anthropic-messages`/`openai-completions` rows already match against `errorMessage`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 82be125 — spec: adopt P1 typed-query two-phase loop (Option A) (2026-05-04, Thomas Andersen); 513a5cc — pi-loom spec: resolve "overflow table predicates key on structured JSON fields errorMessage does not expose" (2026-06-09, Thomas Andersen)
**History:** Both tables — the `Provider`-headed column over `api`-shaped row keys, and the mistral/amazon-bedrock rows worded "body matching" — entered in 82be125 when the provider error and seed tables were first authored (then in the monolithic `pi-integration-contract.md`, later split into this page by f5e89f4 without content change). At that point all four overflow rows referenced structured fields uniformly. 513a5cc rewrote the `anthropic-messages` and `openai-completions` rows to match against `errorMessage` and added the *Classifier input surface* clause that "every body-wording match in the table is a match against that string", but left the `mistral` and `amazon-bedrock` rows saying "body matching" — converting uniform wording into the present inter-row inconsistency. The "Provider" column header was untouched by either commit.

## Solution approach

Rename the leftmost column header of both tables from `Provider` to name the `Model<Api>.api` value the rows are keyed on, and add a keying sentence to the **Provider error mapping** prose, mirroring the `#provider-seed-field-mapping` prose that already states `.api` keying. Reword the `mistral` and `amazon-bedrock` overflow rows so their match predicate names `errorMessage` rather than a parsed body, consistent with the other two rows and the `#classifier-input-surface` note.

## Solution constraints

- Preserve the existing overflow-signature regex literals unchanged; the `\|` alternation-operator disambiguation is owned by the same-cluster regex finding.
