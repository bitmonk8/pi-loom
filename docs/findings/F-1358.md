## Problem

`H5b`'s dependency-set parenthetical in `H5b-warn-only-canary.md` and the *Transitive-completeness plan-maintenance* rule in `conventions.md` both characterise `H5b`'s `Deps.` as exactly the **coverage-producing** set (the closing-leaf set named in `coverage-matrix.md`). But the actual `Deps.` is a phase-range sweep of every MVP and vertical implementation leaf, which pulls in Class-2 seam leaves that close no coverage-matrix row — `V15g`, `V15i`, and `V17c`. The enumeration is therefore a strict superset labelled as equal to the coverage-producing set. A maintainer adding a future non-coverage seam leaf cannot tell from the conflicting framings whether `H5b` membership requires it; `H5d` reads the set under an at-least-one test, so this is a plan-maintenance consistency defect, not a gate contradiction.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `ea6b1da` ("pi-loom plan: resolve \"Live-corpus gate activation has no documented rollback\"", 2026-06-11)
**History:** `H5b-warn-only-canary.md` was created in `ea6b1da` (`git log --follow`/`git log -S "complete coverage-producing set"` both bottom out there). At creation the parenthetical already labelled the set "the complete coverage-producing set" while simultaneously equating it with "every MVP and vertical implementation leaf (`V1a`–`V18c`)" — the all-implementation-leaves enumeration — so the two framings were contradictory from inception. A later edit `25b911f` ("resolve \"H1a missing from H5b's Deps\"", 2026-06-11) sharpened the characterization to "the closing-leaf set named in coverage-matrix.md, defined by coverage production rather than phase," which made the contradiction more acute, and `37733fd` ("resolve \"H6a transitive-completeness rule parked in Deps, not conventions\"", 2026-06-11) moved the maintenance obligation into `conventions.md` carrying the same label; neither introduced the underlying mismatch.

## Solution approach

Rewrite `H5b`'s dependency-set parenthetical in `H5b-warn-only-canary.md` and the matching *Transitive-completeness plan-maintenance* sentence in `conventions.md` so the label describes the set as all `M`/`V*` implementation leaves plus the named horizontal singletons, rather than as exactly the coverage-producing set. State coverage-production as a sufficient-not-necessary condition for membership, so every coverage-producing leaf is necessarily a member and the transitive-completeness obligation holds as a consequence. Leave `H5b`'s `Deps.` enumeration unchanged.

## Solution constraints

- Out of scope: the coverage-matrix status of `V15g`, `V15i`, and `V17c`, owned by their respective findings — do not add, remove, or reassign coverage-matrix rows as part of this label reconciliation.
