## Finding

The **Unknown-reason rule** in `unknown-reason-rule.md` introduces two handler-local variables by their spelled-out identifiers — `capturedEventReason` (pre-initialised at handler entry to the literal `"<unreadable>"`, overwritten by the `event.reason` read on the success arm) and `pinnedConstantReadOk` (pre-initialised to `false`, set to `true` on the snapshot lookup-and-`literals` success arm) — and then re-uses those exact names as cross-document binding handles rather than as role descriptions. The observable contract is already fully pinned independently of the names: the value of `details.event.reason` / `details.observed` on each of the four reason paths (closed-set member, unknown-reason coercion, throwing-access sentinel, snapshot-failure sentinel), and the boolean "the snapshot read succeeded". The internal variable identities and their pre-initialisation mechanics add nothing the observable contract does not already determine.

The names then leak across the cluster as load-bearing tokens. `session-only-degraded-state.md`'s *Predicate split* clause quotes `pinnedConstantReadOk && capturedEventReason ∈ {"new","resume","fork"}` and `capturedEventReason ∉ {"quit","reload"}` as the narrow and broad predicates, and its *Seam minimality (normative)* clause forbids "any handler-scoped state seam beyond `pinnedConstantReadOk` and `capturedEventReason`". `drain-state-contract.md` names both in carve-out (b) of the *Handler control-flow ordering* clause and again in the `markRuntimeDegraded` *Methods* paragraph. `version-bump-triggers.md`'s step-5 sweep pins `capturedEventReason` into the second-stage grep anchor set `event.reason|SessionShutdownEvent|capturedEventReason|entry.shutdownReason` as a required search token. Two plan leaves' **Spec** fields (`V9h`, `V9l`) cite "the unknown-reason rule's `pinnedConstantReadOk` seam" as their normative-cross-link-closure justification.

The fix is to state the behaviour the seams stand in for — a captured-reason slot whose value per path is already enumerated, and a snapshot-read-succeeded flag — and let downstream clauses reference those *behaviours* rather than the named variables. The variable names and pre-initialisation should be demoted to non-normative illustration; because the names currently serve as the cross-document binding handle, the demotion must restate every consuming site (predicates, seam-minimality ceiling, control-flow carve-out, and the version-bump grep anchor) in role terms within the same edit, and re-ground the version-bump sweep on shapes the spec pins observably (`entry.shutdownReason`, `details.event.reason`).

## Solution Space

**Shape:** single

### Recommendation

Demote the two variable names to non-normative illustration and re-express every normative reference in role terms, in one commit:

1. In `unknown-reason-rule.md`'s pre-init clause, replace the named-variable introductions with role descriptions — a handler-scoped *captured-reason slot* (pre-initialised at handler entry to `"<unreadable>"`, overwritten by the success arm of the `event.reason` read) and a handler-scoped *snapshot-read-succeeded flag* (pre-initialised to `false`, set on the snapshot lookup-and-`literals` success arm). Add a one-line note that the role labels are spec-owned and the implementation chooses the identifiers, so a future reader does not re-introduce the pin to "restore" the grep.
2. In `session-only-degraded-state.md`, restate the *Predicate split* predicates in role terms ("the snapshot-read-succeeded flag is set **and** the captured-reason slot is a member of `{"new","resume","fork"}`"; "the captured-reason slot is not a member of `{"quit","reload"}`"), and phrase the *Seam minimality (normative)* ceiling as "no handler-scoped state seam beyond the snapshot-read-succeeded flag and the captured-reason slot".
3. In `drain-state-contract.md`, restate carve-out (b) and the `markRuntimeDegraded` tag-transition predicate in the same role terms.
4. In `version-bump-triggers.md`, re-ground the step-5 second-stage anchor set on shapes the spec pins observably — drop the loom-internal `capturedEventReason` token and anchor instead on `entry.shutdownReason` (the per-entry stamped field, already in the set) and `details.event.reason` (the emission payload site), plus the prose "the handler-scoped capture site of the session-shutdown reason, whatever the implementation has named it." Re-ground **before** dropping the literal token, since the recipe is load-bearing for inbound-reference coverage.
5. In the `V9h` and `V9l` plan-leaf **Spec** fields, replace "the unknown-reason rule's `pinnedConstantReadOk` seam" with role phrasing ("the unknown-reason rule's snapshot-read-succeeded flag seam").

Edge case for the implementer: the role-rename must land atomically — the names are the current cross-document binding handle, so a partial edit leaves dangling references. The behavioural pins that genuinely constrain the discriminator (it MUST use the *captured* string and MUST NOT re-read `event.reason` at the discriminator site) are retained verbatim; only the variable-name pinning is demoted.

## Consequence

**Severity:** advisory

A conformant implementation that names the handler-local variables differently diverges from spec text without diverging from spec semantics, and a contributor renaming them inside the handler module triggers a multi-page inbound-reference sweep with no observable effect. The misalignment also lays a contributor trap on the next Pi bump: `version-bump-triggers.md`'s step-5 grep recipe treats the literal token `capturedEventReason` as a required search anchor, so an implementation that legitimately chose another name would silently break the sweep's coverage. No runtime behaviour is wrong; the cost is spec-text divergence and a brittle maintenance contract.
