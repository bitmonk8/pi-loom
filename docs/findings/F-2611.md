## Problem

The QRY-5 "Newline trim" worked example in `query-forms.md` shows a multi-line template whose raw source lines carry a four-space indent, but states the before-dedent intermediate render as `"  The author...\n  with...\n  Produce..."` with only two leading spaces. QRY-7's normative vector 1 — annotated "the worked example above, restated as a normative vector" — encodes the same example at four spaces. The same illustration is thus presented at conflicting indentation widths across the raw template, the before-dedent intermediate, and the QRY-7 vector, and the raw form reads as if dedent strips a fixed two spaces rather than the full common prefix. The shipped behaviour is governed by the QRY-7 vectors and is unaffected, but a reader calibrating dedent off QRY-5 can be misled.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `53f5831` (spec: rewrite query primitive, system prompts, and schema syntax — 2026-05-03), `66e419d` (pi-loom spec: resolve "Dedent normalisation has no normative test vectors" — 2026-05-04)
**History:** `git log -S`/`-G` and `git blame` trace the worked example to `53f5831`, which introduced it in `spec.md` (later carried verbatim into `spec_topics/query.md` at `fecb504` and into `query/query-forms.md` at the `f5e89f4` size-cap split). At `53f5831` the example used a raw four-space template indent inside the bullet list with a two-space before-dedent intermediate; in the rendered markdown the list strips two spaces, so the displayed template and the intermediate both showed two spaces and the example was self-consistent — there was no second statement to contradict. `66e419d` then added the QRY-7 normative vector table, whose vector 1 is explicitly "the worked example above, restated as a normative vector" but encodes it at four-space indent (`git show 66e419d` adds `| 1 | `` @`\n    The author...` ``). The cross-reference disagreement materialised from the interaction of the original two-space illustration and the later four-space normative restatement, not from either commit in isolation.

## Solution approach

Reconcile the QRY-5 "Newline trim" worked example so its rendered template indentation and its before-dedent intermediate string agree with QRY-7 vector 1. Because the fenced `loom` block is nested in the `- **Newline trim.**` bullet, markdown strips two spaces of list-continuation indent from the rendered view, so the raw source indent and the intermediate string must be adjusted in lockstep to display the same width (lifting the example out of the bullet is an equivalent route).
