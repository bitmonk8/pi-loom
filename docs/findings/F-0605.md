## Problem

`audit-resolution.md` carries four ordering-relative prose pointers ("the *Malformed-marker discriminator* below" on the *Minimum justification* and *Stale-marker discriminator* paragraphs; "the (s2) sub-kind below" on *Inventory-first resolution*; "the malformed-marker dual-emission rule below" on *Multi-line surface placement*) whose targets all live on the separate `audit-failures.md` page. The "below" wording presupposes intra-file order, so each pointer dangles: a reader on `audit-resolution.md` who follows it finds nothing further on the page. The defect is symmetric — `audit-failures.md`'s back-pointers ("per *Marker scope* / *Multi-line surface placement* above"; "per *Malformed-marker discriminator* above") dangle in the other direction. The references are structural, not incidental: the *Malformed-marker discriminator* clauses (a)–(g) are handed off to from several `audit-resolution.md` paragraphs, so the audit's per-discriminator routability obligation breaks at the page boundary.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `f5e89f4` — *docs: enforce 100KB hard / 30KB recommended size cap on spec set* (2026-06-04)
**History:** `git log -S 'malformed-marker discriminator below'` and `git log -S 'Malformed-marker discriminator'` show the *Malformed-marker discriminator*, *Stale-marker discriminator*, *Minimum justification*, *Inventory-first resolution*, and *Multi-line surface placement* paragraphs all entered the corpus in a single edit (`e5cd514`, 2026-05-09, *"resolve T35 SDK capability inventory closed-set negative-direction audit"*) inside the then-single `docs/spec_topics/pi-integration-contract.md` file, where every "below" pointer resolved intra-file. The 576KB-to-30-children size-cap split in `f5e89f4` placed *Malformed-marker discriminator* on the new `audit-failures.md` while leaving the *Minimum justification* / *Citation-token grammar* / *Stale-marker discriminator* / *Inventory-first resolution* / *Multi-line surface placement* paragraphs on `audit-resolution.md`, stranding the four pointers as cross-file "below" references that no longer compose. The commit message describes its rewrite as "1571 cross-references repointed across docs/" and "no content edits", confirming that the splitter was wired to fix `[link](path#anchor)` Markdown references but had no signal for naked prose pointers like "the *X* below". A `git blame` on the offending paragraphs in today's `audit-resolution.md` confirms the four "below" tokens are byte-for-byte the same strings the splitter inherited from `e5cd514`.

## Solution approach

Rewrite the four cross-file "below" pointers on `audit-resolution.md` and the matching "above" back-pointers on `audit-failures.md` as explicit inter-file links into the named target paragraphs on the other page. Add a stable HTML anchor to `audit-failures.md`'s *Malformed-marker discriminator* paragraph (which carries no co-located anchor today) so the links from `audit-resolution.md` resolve. Confirm the result with `docs/_tools/check_links.py`.

## Solution constraints

- Out of scope: re-merging the paragraph layout the size-cap split produced — the per-page size cap (`f5e89f4`) MUST continue to hold.
