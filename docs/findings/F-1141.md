## Problem

The `loom/runtime/internal-error` row in `docs/spec_topics/diagnostics/code-registry-runtime.md` enumerates the catchable host-allocation failures (`RangeError: Invalid string length`, `RangeError: Invalid array length`, `RangeError: Maximum call stack size exceeded`) as a closed list, with no "for example", no "and any other", and no back-reference to the catch boundary's source of truth. Read in isolation, an implementer can conclude the runtime must match those three exact `error.message` strings and that any other catchable host-allocation `RangeError` (or future engine-specific message string) falls outside the trigger. NOCEIL-3 in `docs/spec_topics/hard-ceilings/ceiling-invariants-and-audit.md` is the single source of truth and is explicitly exemplary — it lists the same three strings "and any other JavaScript-exception-shaped allocation failure the host throws" and pins the partition operationally (catchable JavaScript exception ⇒ `internal-error`; uncatchable engine fatal ⇒ process terminates, no diagnostic). The row drops both the exemplary clause and the operational framing, re-opening the engine-version dependency NOCEIL-3 had closed.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** cc240be — pi-loom spec: resolve "No additional V1 runtime ceiling closing paragraph" (2026-05-07, Thomas Andersen)
**History:** The three-string list and the NOCEIL-3 catch boundary were added in the same commit `cc240be`. That commit authored NOCEIL-3 in `spec.md` with the exemplary "and any other JavaScript-exception-shaped allocation failure the host throws" clause and the operational catchable/uncatchable partition, and in the same edit appended the catchable-allocation arm to the `loom/runtime/internal-error` row in `spec_topics/diagnostics.md` (now `docs/spec_topics/diagnostics/code-registry-runtime.md`) — but the row received only the three concrete strings, without the exemplary closing clause or any back-reference to NOCEIL-3 for the boundary. Subsequent edits to the row (`8455d5e`, `47fe3cd`, `2128e5f`, `9568906`, and the `f5e89f4` corpus split) preserved the three-string list verbatim; the drift between row and NOCEIL-3 has been present since the catchable arm was first authored.

## Solution approach

Rewrite the catchable host-allocation clause of the `loom/runtime/internal-error` row so the three message strings read as non-exhaustive examples rather than a closed match list, and add a forward-link delegating the catchable/uncatchable partition to NOCEIL-3 in `docs/spec_topics/hard-ceilings/ceiling-invariants-and-audit.md` as the source of truth. The edit tightens the row's prose to match the boundary NOCEIL-3 already pins; it does not introduce a new trigger surface.

## Solution constraints

- Out of scope: confine the edit to the `internal-error` row's catchable host-allocation clause — the `details.kind = "tool-return-shape"` arm (owned by F-1149) and the `hint` / `error.stack` Error-shape assumption (owned by F-1142) are not touched.
