## Finding

The `loom/runtime/internal-error` row in `docs/spec_topics/diagnostics/code-registry-runtime.md` enumerates "catchable host allocation failures" as a closed three-element list — `RangeError: Invalid string length`, `RangeError: Invalid array length`, `RangeError: Maximum call stack size exceeded` — separated from the rest of the trigger description by an em dash and terminated with a semicolon, with no "for example", no "and any other", and no back-reference. Read in isolation, an implementer can reasonably conclude the runtime must match those three exact `error.message` strings to route through `internal-error`, and that any other catchable host-allocation `RangeError` (or future engine-specific allocation failure with a different message string) falls outside the trigger.

The single source of truth for the catch boundary is **NOCEIL-3** in `docs/spec_topics/hard-ceilings/ceiling-invariants-and-audit.md`, which is explicitly exemplary: it lists the same three strings *"and any other JavaScript-exception-shaped allocation failure the host throws"*, and pins the partition operationally — catchable JavaScript exception ⇒ routed through `internal-error`; uncatchable engine fatal (the `OOMErrorCallback` / `abort()` path) ⇒ process terminates and emits no diagnostic. The operational formulation is what makes the rule engine-version-stable: classification rides on the runtime-observable "did the host throw a JavaScript exception or did the process die" distinction, not on any specific V8 build's table of message strings. The code-registry-runtime row drops both the exemplary closing clause and the operational framing, so the prose reads as a literal three-string match list and silently re-opens the engine-version dependency that NOCEIL-3 had already closed.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** cc240be — pi-loom spec: resolve "No additional V1 runtime ceiling closing paragraph" (2026-05-07, Thomas Andersen)
**History:** The three-string list and the NOCEIL-3 catch boundary were added in the same commit `cc240be`. That commit authored NOCEIL-3 in `spec.md` with the exemplary "and any other JavaScript-exception-shaped allocation failure the host throws" clause and the operational catchable/uncatchable partition, and in the same edit appended the catchable-allocation arm to the `loom/runtime/internal-error` row in `spec_topics/diagnostics.md` (now `docs/spec_topics/diagnostics/code-registry-runtime.md`) — but the row received only the three concrete strings, without the exemplary closing clause or any back-reference to NOCEIL-3 for the boundary. Subsequent edits to the row (`8455d5e`, `47fe3cd`, `2128e5f`, `9568906`, and the `f5e89f4` corpus split) preserved the three-string list verbatim; the drift between row and NOCEIL-3 has been present since the catchable arm was first authored.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the catchable-allocation clause of the `loom/runtime/internal-error` row to align with NOCEIL-3's wording and to delegate the catch boundary to NOCEIL-3 rather than restate it. Replace the current

> `; also catchable host allocation failures — RangeError: Invalid string length, RangeError: Invalid array length, RangeError: Maximum call stack size exceeded;`

with prose that (a) keeps the three message strings as **non-exhaustive examples**, (b) carries NOCEIL-3's exemplary closing clause verbatim, and (c) names NOCEIL-3 as the source of truth for the catchable/uncatchable partition. Suggested wording:

> `; also catchable host allocation failures — examples include \`RangeError: Invalid string length\`, \`RangeError: Invalid array length\`, and \`RangeError: Maximum call stack size exceeded\`, and any other JavaScript-exception-shaped allocation failure the host throws (the catchable/uncatchable partition is pinned operationally at [Hard Runtime Ceilings — NOCEIL-3](../hard-ceilings/ceiling-invariants-and-audit.md), not by enumeration of engine-specific message strings);`

The same edit leaves the existing uncatchable-arm back-reference (`[`spec.md` — Hard ceilings, NOCEIL-3]`) and the rest of the row untouched. Edge cases the implementer must watch:

- The `details.kind = "tool-return-shape"` arm immediately after this clause is unrelated and must remain in place; the edit is confined to the semicolon-bounded catchable-allocation phrase.
- The corrected wording does not introduce a new trigger surface — it tightens prose so that the row matches the catch boundary NOCEIL-3 already pins. No `V4b` / `V4b-T` acceptance criterion changes.
- The link target uses the canonical `ceiling-invariants-and-audit.md` page (which carries NOCEIL-3) rather than the `overview-and-orientation.md#no-additional-ceilings` aggregator anchor the uncatchable arm currently points at; either is acceptable provided the destination resolves to the NOCEIL-3 paragraph.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge on which catchable host throws route through `internal-error`. One reads the row literally and writes an `error instanceof RangeError && error.message === "…"` triplet match, dropping every other catchable host failure on the floor (no diagnostic, no slash-surface system note, no `InvokeInfraError` to the `invoke` parent). The other reads NOCEIL-3 and routes any catchable JavaScript exception that escapes the closed panic-source list. The divergence is silent on the happy path and only surfaces on rare host-resource pathologies, where one implementation reports nothing and the other reports `internal-error` — observably non-equivalent runtime-defect surfaces.
