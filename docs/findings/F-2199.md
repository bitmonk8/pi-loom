## Finding

QRY-18 (Stringification of interpolated values) specifies that a `${expr}` interpolation whose operand is statically a `Result` is rejected with the parse-phase diagnostic `loom/parse/interpolated-result`. Its Notes bullet then extends this to the case where the operand's type is *statically unresolvable* (an inferred binding that widens past the parser's view): "the runtime renderer falls back to a **panic** carrying the same `loom/parse/interpolated-result` diagnostic code." The diagnostics registry row for that code (`code-registry-parse.md`) records the same design: "the runtime renderer raises the same code as a panic when the type is statically unresolvable."

This collides with the runtime-panic contract in `errors-and-results/error-model.md` → *Runtime panics*, which is an explicitly **closed** enumeration: "V1 panic sources — each carrying its registered `loom/runtime/*` code … This list is closed." Six sources are listed (`match-error`, `index-out-of-bounds`, `null-member-access`, `null-index-access`, `missing-object-key`, `invoke-depth-exceeded`), each with a normative row in the six-row *Panic message string* template table, and a separate `loom/runtime/internal-error` runtime-defect surface absorbs every unexpected throw outside that closed set. The unresolvable-`Result`-interpolation panic is admitted by none of these: it carries a `loom/parse/*` code (the panic surface mandates `loom/runtime/*`), it is absent from the closed six-source list, and it has no row in the normative panic-message-template table. QRY-21, which enumerates the panics a `${expr}` interpolation can trip ("non-exhaustive `match`, OOB, null/missing-key access"), likewise omits it.

An implementer cannot satisfy both contracts as written. The three plausible reconciliations produce observably different behaviour — a panic emitting a `loom/parse/*` code with no registered message template, a route through `loom/runtime/internal-error` (an "internal error" framing for what is actually a known author error), or an invented `loom/runtime/*` code — so two reasonable implementers diverge on the diagnostic code, the message string, and the operator-facing surface.

## Issue introduction
**Verdict:** single-commit

**Introducing commit:** `0c9ce54` — "pi-loom spec: resolve \"Stringification rule for ${expr} interpolations is unspecified\"" (2026-05-04)

**History:** The spec corpus is git-tracked. The QRY-18 stringification rule — including both the "runtime renderer falls back to a panic carrying the same `loom/parse/interpolated-result` diagnostic code" wording and the registry's "raises the same code as a panic" note — was authored together in `0c9ce54` (`git log -S "falls back to a panic carrying the same"` and `git log -S "raises the same code as a panic"` each return `0c9ce54` as the earliest commit; later hits are the file-split commit `f5e89f4`). At that commit the *Runtime panics* section already read "V1 panic sources — each carrying its registered `loom/runtime/*` code" and "This list is closed" (verified via `git show 0c9ce54:spec_topics/errors-and-results.md`), and the panic-message-template table already enumerated only `loom/runtime/*` codes. The defect therefore entered fully formed in `0c9ce54`: a runtime-panic fallback carrying a `loom/parse/*` code was introduced into a corpus whose panic surface was already closed and already required per-source `loom/runtime/*` codes. The subsequent `f8f4813` (which added the `loom/runtime/internal-error` runtime-defect surface) and `f5e89f4` (the 100KB size-cap file split) neither caused nor reconciled the conflict.

## Solution Space

**Shape:** multiple

### Option A — Coin a dedicated `loom/runtime/interpolated-result` panic source

**Approach:** Treat the unresolvable-`Result` interpolation as a genuine, author-caused runtime panic (analogous to `match-error`). Register a new `loom/runtime/interpolated-result` code in `code-registry-runtime.md` with a message template; add it as a seventh row to the closed list and the panic-message-template table in `error-model.md`; add it to QRY-21's enumeration; and update QRY-18's Notes bullet and the `code-registry-parse.md` row so the runtime case names the new `loom/runtime/*` code (codes are phase-scoped, so the static parse-phase code and the runtime panic code are distinct, sharing only the author-facing message text).

**Spec edits:** new runtime registry row + message template; widen the closed-list count and table in `error-model.md`; add the source to QRY-21; repoint the runtime sentence in QRY-18 and the parse-registry row from `loom/parse/interpolated-result` to the new runtime code.

**Pros:** Preserves QRY-18's "static where possible, runtime where not" intent; keeps the panic carrying an author-meaningful code and message rather than an "internal error" framing; honours the per-source `loom/runtime/*` + message-template discipline.

**Cons:** Widens the "closed" panic set (six → seven) and every place that asserts "exactly six panic sources"; touches more files.

**Risks:** Missing one of the "six panic sources" count assertions (e.g. in `code-registry-runtime.md`) leaves a fresh inconsistency; the lock-step co-edit obligation must cover all count sites.

### Option B — Route the runtime occurrence through `loom/runtime/internal-error`

**Approach:** Declare that the statically-unresolvable case is not a distinct panic source. Narrow QRY-18's Notes bullet and the `code-registry-parse.md` row so the runtime occurrence routes through the existing `loom/runtime/internal-error` runtime-defect surface (which already absorbs throws outside the closed list), and stop describing `loom/parse/interpolated-result` as raised "as a panic." `loom/parse/interpolated-result` then names only the static parse/type-phase diagnostic.

**Spec edits:** rewrite QRY-18's runtime sentence to reference `loom/runtime/internal-error`; amend the parse-registry row's parenthetical; leave the closed six-source list and QRY-21 unchanged.

**Pros:** Smallest surface; keeps the panic list genuinely closed; reuses an already-defined catch-all and its routing.

**Cons:** `internal-error` is defined for *unexpected interpreter exceptions* and surfaces as "loom `/<name>` aborted with internal error"; an unresolvable-`Result` interpolation is a known author error, so this framing mislabels it and loses the specific "unwrap with `?` or `match`" guidance at the operator surface.

**Risks:** Implementers may read "internal error" as signalling a runtime defect and file spurious bug reports; the author gets no actionable diagnostic distinguishing this from a true interpreter throw.

### Recommendation

Adopt **Option A**. The unresolvable-`Result` case is a known, author-attributable failure that QRY-18 deliberately defers from parse-time to runtime ("static where possible, runtime where not"), exactly the shape of the existing closed-list panics; giving it its own `loom/runtime/*` code with a registered message template keeps the diagnostic actionable and preserves the per-source template discipline. Watch two edge cases: (1) update every "exactly six panic sources" / six-row assertion in lock-step (`error-model.md` and `code-registry-runtime.md`), and (2) keep the static `loom/parse/interpolated-result` and the new runtime code as distinct phase-scoped codes that share only the author-facing message string.

## Consequence

**Severity:** correctness

A runtime render of a statically-unresolvable `Result` interpolation has no well-defined diagnostic identity: the spec demands a `loom/runtime/*` panic code with a registered message template, while QRY-18 and the parse registry name a `loom/parse/*` code that the closed panic surface cannot carry. Two implementers will diverge on the emitted code, the panic message string, and whether the failure routes through the panic channel or the `internal-error` runtime-defect surface — producing different author- and operator-facing behaviour for the same loom.
