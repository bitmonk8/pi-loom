## Problem

The five-field `ToolDefinition` derivation list in `extension-bootstrap-and-per-loom.md` (Per-loom registration) leaves three derivation preconditions unpinned for a `.loom` exposed as a tool. First, the `execute` derivation states only that the adapter "spawns the loom as a subagent invocation ‚Ä¶ and returns its result envelope" ‚Äî the success path; it never says how an *in-loop* failure of the spawned invocation (a callee `Err` such as `InvokeCalleeError` / `InvokeInfraError`, a panic, or a mid-run cancellation) maps onto the Pi `AgentToolResult` (`{ content, isError }`) the model-invoked adapter must return, and `tool-calls.md` (Outcome enumeration) pins only the pre-evaluation setup throw and the post-cancel late-settlement discard. Second, `description` derives from the loom's frontmatter `description`, which `frontmatter-fields-a.md` marks optional (`required: no`, default `null`, "internal-only looms legitimately omit this"), so a description-less loom derives a `null` for a Pi-required field with no defined behaviour. Third, the `label` capitalisation transform is undefined when the basename's leading character has no uppercase form (a digit, underscore, or non-ASCII codepoint, e.g. `2fa-check.loom` or `_internal.loom`).

## Solution approach

Clarify the `execute` derivation in `extension-bootstrap-and-per-loom.md` (or pin it in `tool-calls.md`'s Outcome enumeration alongside the existing setup-throw and post-cancel bullets) so the in-loop failure dispositions of a model-invoked `.loom`-as-tool ‚Äî callee `Err`, panic, mid-run cancellation ‚Äî each map onto a defined `{ content, isError }` tool-result fed back to the model. Resolve the optional-frontmatter-`description`-versus-required-`ToolDefinition`-field tension for the `description` derivation. Define the `label` capitalisation behaviour for basenames whose leading character has no uppercase form.
