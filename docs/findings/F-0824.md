## Problem

`version-bump-step2.md` opens with a preamble paragraph declaring a two-tier split — items (a)–(e) are MUST-audited, items (f)–(an) are SHOULD-audited-but-MUST-recorded — and then drops into a flat bulleted list running (a) to (an) with no structural break between item (e) (anchor `id="bump-checklist-slash-dispatch-serialisation"`) and item (f). The only signal that item (f) sits in a different tier is the per-item SHOULD-escalation clause buried at the tail of each item body. A contributor scrolling the list cannot tell at a glance which tier an item belongs to without re-reading the preamble or counting letters back to (e). This diverges from the corpus convention elsewhere, where multi-tier checklists separate tiers with sub-headings or explicit grouping.

## Solution approach

Insert a tier-naming structural break immediately before item (f) in `version-bump-step2.md` — a sub-heading (or a horizontal rule plus a labelled lead-in) sitting between the preamble paragraph and item (f) — that names the SHOULD tier and its item range. Reuse the preamble's existing range token (`(f)–(an)`) rather than coining a fresh one, so the break stays accurate as the SHOULD tier grows. Leave the preamble paragraph in place as the authoritative tier declaration; the break is a navigation aid, not a replacement for the recording rules.
