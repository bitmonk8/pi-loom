## Finding

The Unknown-reason rule pins the `details.observed` coercion with the sentence:

> Pin the coercion to `String(event.reason)` explicitly — a template-literal coercion (`` `${event.reason}` ``) re-introduces the symbol-throw bug `String()` is chosen to avoid.

The leading verb "Pin" is a bare editorial imperative with no RFC-2119 modal. The very same paragraph repeatedly uses MUST for its other coercion obligations — "The validation read of `event.reason` MUST be wrapped in a `try`/`catch` …", "The coercion itself MUST be wrapped in `try`/`catch`", "The `String(error)` coercion MUST itself be wrapped in `try`/`catch`" — so a reader applying the spec's modal vocabulary cannot tell whether using `` `${event.reason}` `` instead of `String(event.reason)` is non-conformant or merely discouraged. The asymmetry matters because the sentence itself names a real defect the choice prevents (template-literal coercion re-introduces the `Symbol` `TypeError` path that `String()` was chosen to avoid), so the obligation is meant to bind; the wording just fails to say so.

## Solution Space

**Shape:** single

### Recommendation

Replace the sentence

> Pin the coercion to `String(event.reason)` explicitly — a template-literal coercion (`` `${event.reason}` ``) re-introduces the symbol-throw bug `String()` is chosen to avoid.

with

> The `details.observed` coercion MUST use `String(event.reason)` rather than a template-literal coercion (`` `${event.reason}` ``); a template-literal coercion re-introduces the `Symbol`-coercion `TypeError` `String()` is chosen to avoid.

Implementer-relevant edge cases:

- The MUST applies to the `details.observed` site only. The handler-scoped captured-value site already inherits the same coerced string per the existing "the same coerced string" clause, so no second MUST is needed there; the new wording should not be duplicated.
- The sibling `String(error)` coercion in the `"throw:<String(error)>"` discriminator further down the paragraph is already MUST-wrapped for its `try`/`catch` but is silent on the same template-literal alternative. That symmetric gap exists but is **not** in scope for this finding — it lives under the `"throw:<String(error)>"` grammar pin, not the **Unknown-reason rule** coercion pin, and should be tracked separately if review wants both sites normalised. Avoid bundling them into this edit.
- The anchor-stable contract-surface carve-out at the end of the paragraph lists (a)–(d) as anchor-stable substrings; `String(event.reason)` is not among them, so the wording change does not trigger an inbound-reference sweep.

## Consequence

**Severity:** advisory

Two reasonable implementers can disagree on whether `` `details: { observed: \`${event.reason}\` }` `` is conformant. The runtime difference (`` `${someSymbol}` `` throws `TypeError`, `String(someSymbol)` does not) is the exact failure the **Unknown-reason rule** exists to prevent, so the loose wording can let a non-conformant implementation pass review while still being silently fragile to a hostile `Symbol`-typed reason.
