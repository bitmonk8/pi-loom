## Finding

`V18c`'s build-time literal-read assertion reads two constants directly out of `V9a`'s single-source-of-truth pinned-constants block: `FACTORY_PROBABLE_CAPABILITIES` and the cancellation-runtime constant `SHUTDOWN_AWAIT_CAP_MS` (per `V18c-version-bump-checklist.md` Adds, and confirmed by `V9a-capability-probe.md` Adds, which states `SHUTDOWN_AWAIT_CAP_MS = 2000` "which `V18c`'s build-time literal-read assertion reads"). Despite this direct symbol-level coupling, `V18c Deps = V18c-T, V18a, V18b` and `V18c-T Deps = V18a, V18b` — neither names `V9a`.

Execution order is currently safe only by accident: `V18c → V18a → V9a` (V18a's `Deps` carry `V9a`) guarantees `V9a` builds first. But `V18a` consumes only `FACTORY_PROBABLE_CAPABILITIES` (for its partition-flag reconciliation in `CAPABILITY_OBLIGATIONS`); it does not read `SHUTDOWN_AWAIT_CAP_MS` at all. So `V18c`'s direct read of `SHUTDOWN_AWAIT_CAP_MS` rides an incidental transitive edge that `V18a` carries for an unrelated reason.

The dependency graph therefore understates a real producer→consumer edge. A future re-scoping of `V18a` that dropped its `V9a` dependency would silently strand `V18c`'s literal-read assertion (the constant `V18c` reads would no longer be guaranteed built), with nothing in `V18c`'s own `Deps` to catch it.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 8af3204 — pi-loom plan: resolve "V18c bundles static gates with runtime-evidence acceptance gate" (2026-06-11, Thomas Andersen); 622d0d3 — pi-loom plan: resolve "SHUTDOWN_AWAIT_CAP_MS physical home contradicts pinned-constants-block placement" (2026-06-11, Thomas Andersen)
**History:** `V18c` was created (c6a664e, 2026-06-10) with `Deps = V18c-T, V18a, V18b`, never declaring `V9a` directly. Commit 8af3204 split the runtime-evidence gate out to `V18d` and restored that exact three-edge `Deps` set, settling it without `V9a`. Commit 622d0d3 then added `V18c`'s direct read of the V9a-owned `SHUTDOWN_AWAIT_CAP_MS` constant to the build-time literal-read assertion, deepening the undeclared producer→consumer coupling beyond the pre-existing `FACTORY_PROBABLE_CAPABILITIES` read — the defect is the interaction of the `V9a`-less `Deps` set and the later-added direct constant read.

## Solution Space

**Shape:** single

### Recommendation

Add `V9a` to both leaves' `Deps` fields so the pinned-constants-block read is bound by a declared edge:

- In `docs/plan_topics/V18c-version-bump-checklist.md`, change the `Deps.` field from `` `V18c-T`, `V18a`, `V18b` `` to `` `V18c-T`, `V18a`, `V18b`, `V9a` ``.
- In `docs/plan_topics/V18c-T-version-bump-checklist.md`, change the `Deps.` field from `` `V18a`, `V18b` `` to `` `V18a`, `V18b`, `V9a` ``.

This is a declared-edge correction only; no architecture or mechanism changes. The leaf body already names `V9a` as the constant source, so no Adds/Tests text needs to change.

## Consequence

**Severity:** advisory

The leaf is pickable and builds correctly today, so nothing breaks at execution time. The defect is a latent dependency-declaration gap: `V18c`'s direct read of `SHUTDOWN_AWAIT_CAP_MS` depends on `V9a` building first through an edge that `V18a` carries for an unrelated purpose, so a later re-scope of `V18a` could silently strand `V18c`'s assertion with no declared edge to detect it.
