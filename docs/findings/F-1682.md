## Problem

PIC-22 (`subagent.md`, `#pic-22`) states the normative obligation as: the runtime MUST initiate `createAgentSession` for all N invocations before any one of the N returns. The immediately-following conformance test, presented as witnessing that obligation, asserts a strictly stronger property: a conformant runtime has created all N sessions **and entered each session's `sendUserMessage`** before any blocked call is released. These are not the same property — an implementation that fires all N `createAgentSession` calls concurrently but then awaits and drives them serially satisfies the literal MUST yet fails the conformance test against the blocking fake `AgentSession`. The normative sentence under-states the obligation, so the rule text and the conformance test it cites disagree.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `7afe9bb` — "pi-loom spec: resolve \"no-invocation-cap MUST NOT is unobservable\"" (2026-05-31)
**History:** `git log -G "conformance test witnesses"` and `git log -S "blocks until explicitly released"` identify `7afe9bb` as the commit that first added the parallel-spawn rule together with its inline fake-`AgentSession` witnessing recipe — the recipe was introduced precisely *to* make the previously-unobservable no-invocation-cap MUST NOT testable, and was placed directly inside the requirement body at that point. Later edits preserved the embedding rather than introducing it: `7a3b907` ("subagent parallel-initiation MUST has no closing leaf", 2026-06-10) lifted the rule out of the **Concurrent-invocation isolation** paragraph into its own `<a id="pic-22">` anchor, carrying the recipe verbatim, and `f566349` (F-1224) narrowed the bound from N ≥ 3 to N ≥ 2 without touching the recipe's placement. `f5e89f4` only relocated the text during the monolith split.

## Solution approach

Strengthen the PIC-22 MUST at `#pic-22` so its stated obligation is co-extensive with the conformance test it cites: require that the runtime create all N sessions **and** enter each session's `sendUserMessage` (reach the per-session drive point) before any one of the N invocations returns. This closes the serial-drive loophole by making the rule and its witness equivalent. PIC-22 already carries its own REQ-ID anchor, so strengthening it satisfies GOV-22 without coining a new ID.
