## Finding

Rule 1 under **Conversation drive — subagent mode** in `docs/spec_topics/pi-integration-contract/subagent.md` describes how a built-in Pi tool name is materialised into a `ToolDefinition` for the spawn-call's `customTools` array via "the corresponding per-tool factory function re-exported from the `@earendil-works/pi-coding-agent` package root", and gives `read` → `createReadToolDefinition(cwd)` and `bash` → `createBashToolDefinition(cwd)` as the two exemplary mappings. The rule deliberately leaves the construction open-ended ("e.g.", and the factory family pinned only at the wildcard path `dist/core/tools/*.d.ts`). Three things the implementer needs to write this lowering step are therefore unpinned:

1. The closed set of built-in tool names the lowering step must dispatch over.
2. The name→factory mapping for the five names not exemplified (`edit`, `write`, `grep`, `find`, `ls` at the loom 1.0 Pi-SDK pin).
3. The per-factory call signature — whether every factory takes `(cwd)` only, whether some require an `options` payload, and whether all return the same `ToolDefinition` shape rule 1 then asserts.

A reasonable implementer reading rule 1 will infer "uniform `cwd`-only call across the whole set" from the two examples and the wildcard path, and will write a single switch / loop keyed on the loom-declared built-in names that resolves each to `create<Capitalised>ToolDefinition(cwd)`. That inference is wrong-by-construction if any factory in the unenumerated remainder requires additional arguments, and the divergence surfaces only at spawn — after `customTools` has been assembled, at a site where the existing diagnostic surface routes through `loom/runtime/internal-error` rather than a load-time refusal. Rule 2 separately names a four-tool default set (`read` / `bash` / `edit` / `write`) for what Pi enables when `tools` is omitted, which does not bound the closed factory set — a loom that lists `grep` or `ls` in its callable set is still in scope, so the lowering step must dispatch over a broader set than rule 2's default-suppression argument bounds.

The `@earendil-works/pi-coding-agent` package at the loom 1.0 Pi-SDK pin in fact exports both a `ToolName` union (`"read" | "bash" | "edit" | "write" | "grep" | "find" | "ls"`) and a unified `createToolDefinition(toolName: ToolName, cwd: string, options?: ToolsOptions): ToolDef` dispatcher that closes the name→factory mapping in one named API, so the unpinning is editorial — the surface to pin already exists — rather than a substantive gap in Pi.

## Solution Space

**Shape:** multiple

### Option A — Enumerate the closed name→factory table

**Approach.** Replace rule 1's `e.g.` clause with an explicit table (or inline list) of the seven loom 1.0 built-in tool names and their factory functions, each with its signature. Pin the whole table to the named `dist/core/tools/index.d.ts` re-export point at the loom 1.0 Pi-SDK pin (in addition to the wildcard `dist/core/tools/*.d.ts` already cited).

**Spec edits.**
- In `subagent.md` rule 1, replace the two-example "e.g." fragment with an enumeration along the lines of: `read` → `createReadToolDefinition(cwd)`, `bash` → `createBashToolDefinition(cwd)`, `edit` → `createEditToolDefinition(cwd)`, `write` → `createWriteToolDefinition(cwd)`, `grep` → `createGrepToolDefinition(cwd)`, `find` → `createFindToolDefinition(cwd)`, `ls` → `createLsToolDefinition(cwd)` — each declared at `dist/core/tools/index.d.ts` at the loom 1.0 Pi-SDK pin, each taking `(cwd: string, options?: <Tool>ToolOptions)` and returning `ToolDefinition`.
- Add a sentence pinning the uniform `(cwd, options?)` signature and stating that loom 1.0 invokes every factory with `cwd` only (no `options` payload).
- Pin the closed set to item (o) (or a new sibling item) of the version-bump editorial-review checklist for re-validation on each Pi minor bump.

**Pros.** The lowering step is reducible to a literal switch; a Pi-side rename or new built-in is caught immediately by the named pin. The reader does not have to read SDK `.d.ts` files to know the closed set.

**Cons.** Verbose: seven mappings inline. The pin couples the spec text to the precise factory-function naming convention; a future Pi rename would require a textual edit at this site (the same posture every other pinned member surface on the page already has).

**Risks.** A new built-in added by Pi between minor bumps would be invisible to the loom unless the version-bump procedure catches it; mitigated by folding the closed-set pin into item (o).

### Option B — Pin the unified `createToolDefinition` dispatcher

**Approach.** Rewrite rule 1 to specify the lowering step in terms of the existing unified `createToolDefinition(toolName, cwd, options?)` function plus the `ToolName` string-literal union that bounds its first argument. Drop the per-factory exemplars in favour of one named dispatcher and one closed type.

**Spec edits.**
- In `subagent.md` rule 1, replace the per-factory "e.g." fragment with: "each materialised by calling `createToolDefinition(name, cwd)`, where the admissible `name` values are exactly the members of the `ToolName` string-literal union (`"read" | "bash" | "edit" | "write" | "grep" | "find" | "ls"` at the loom 1.0 Pi-SDK pin); both `createToolDefinition` and `ToolName` are declared at `dist/core/tools/index.d.ts` and re-exported from `dist/index.d.ts`."
- Pin the `createToolDefinition` signature and the `ToolName` union to a version-bump re-validation item, on the same footing as the existing factory-family pin.

**Pros.** One named API, one named closed type — much smaller textual surface, and the closed set is pinned by reference rather than by enumeration. A future Pi addition to `ToolName` propagates through the version-bump pin without a per-factory edit.

**Cons.** Requires the reader to dereference `ToolName` to know the closed set (mitigated by inlining the union members at the loom 1.0 pin). Drops the readable `read → createReadToolDefinition` style some readers may prefer.

**Risks.** The `createToolDefinition` dispatcher may not be the surface Pi chooses to keep stable across versions; if it is removed in favour of per-factory-only access, the spec must fall back to Option A.

### Recommendation

Adopt **Option B**, with the `ToolName` union's members inlined verbatim at the loom 1.0 Pi-SDK pin so the closed set is readable on the page without dereferencing the type. This is one named dispatcher, one named closed type, one re-validation pin — the minimum surface that closes all three sub-gaps (closed name set, complete name→factory mapping, uniform signature) without committing rule 1 to enumerating seven near-identical lines. Edge case for the implementer: when the loom's `tools:` declares a Pi-tool name that the `ToolName` union does not admit (a future Pi name the loom 1.0 pin does not enumerate, or an extension-supplied tool name), the lowering step is out of `createToolDefinition`'s domain — that path is owned by `loom/load/unknown-tool` per `frontmatter-fields-a.md` and is not changed by this fix.

## Consequence

**Severity:** correctness

Two reasonable implementers reading rule 1 will produce divergent lowering steps: one will hand-code a switch over the two exemplified names and silently drop `edit` / `write` / `grep` / `find` / `ls` calls a loom legitimately declared; another will assume uniform `(cwd)`-only signatures across the unenumerated remainder, which holds at the current pin but is not what the spec actually pins. Both bugs surface at spawn rather than load, routed through `loom/runtime/internal-error`, with no spec-level guidance on which implementation is conformant.
