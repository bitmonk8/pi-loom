## Problem

Rule 1 in `subagent.md`'s spawn block names the built-in Pi tool name→`ToolDefinition` factory family only via two exemplars (`read → createReadToolDefinition(cwd)`, `bash → createBashToolDefinition(cwd)`) and a `dist/core/tools/*.d.ts` glob; it pins neither a canonical mapping covering all seven `tools:`-admissible Pi tool names nor a convention an implementer could use to derive the remaining five. The factory signatures do not extend trivially — each per-tool factory accepts a different per-tool `options` type, and the SDK exposes multiple viable lowering surfaces (the per-name factories, a single `createToolDefinition` dispatcher, and the bulk `create*ToolDefinitions` helpers) with no spec-level preference. The `lowerLoomToolsToPi(loom.tools)` call pins neither which entry point is canonical, what `options` argument is passed, nor what `cwd` value is supplied. Two conformant runtimes can therefore produce `ToolDefinition`s whose `options`-derived behaviour (e.g. bash timeout / env defaults) silently differs for the same loom `tools:` declaration, and the drift surfaces only at tool-call time.

**History:** `git log -S "Pi built-ins (resolved by name"` and `-S "createReadToolDefinition"` against `docs/spec_topics/pi-integration-contract/subagent.md` (and its pre-split ancestor `spec_topics/pi-integration-contract.md` walked via `git log --follow`) trace rule 1 to two commits. `69b9c12` introduced the spawn-block rewrite that authored rule 1's first form — "both Pi built-ins (resolved by name from the model registry / extension API to their `ToolDefinition`) and `defineTool`-wrapped loom callees" — which left the closed name set, the dispatch API, and per-factory signatures entirely abstract. `e279e0c` later sharpened the resolution surface to "the corresponding per-tool factory function re-exported from the `@earendil-works/pi-coding-agent` package root" and added the two `read` / `bash` exemplars plus the wildcard pin to `dist/core/tools/*.d.ts`, but kept the mapping non-exhaustive and did not enumerate or cite a unifying dispatcher. The defect is the interaction of the two passes: the first introduced an unpinned dispatch concept, the second pinned the family at a wildcard but did not close the name set or the signatures the first left open. No single commit revert restores a defect-free state.

## Solution approach

Rewrite rule 1's exemplar in `subagent.md` so the lowering surface is pinned to a single named SDK entry point that covers the closed set of accepted Pi tool names, with one fixed `options` posture, replacing the two-exemplar-plus-glob form. Pin the `cwd` argument to the same `cwd` already passed to `SessionManager.inMemory(cwd)` on the same spawn call so the two `cwd`-consuming spawn-block sites cannot drift. Cross-reference the `ToolName` enumeration and the `tools:` Pi-tool-name validation home at `frontmatter-fields-a.md#tools` rather than re-authoring it here. Fold the new pin into the existing *Spawn-block satellite type pins* paragraph (`#subagent-spawn-satellite-types`) and the editorial-review checklist under `version-bump-intro.md#pi-version-bump-procedure` so a Pi-side rename, reshape, or `ToolName` widening is caught at the bump.

## Solution constraints

- The lowering surface MUST honour the loom's `tools:` allowlist: the bulk helpers `createCodingToolDefinitions` / `createReadOnlyToolDefinitions` / `createAllToolDefinitions` MUST NOT be used as the lowering surface, as they ignore the allowlist and would silently widen the callable set (breaking the rule-2 "ambient Pi tools NOT inherited" invariant).
