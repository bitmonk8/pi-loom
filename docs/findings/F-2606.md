## Problem

The **Provider error mapping** paragraph (`id="provider-error-mapping"`) and the **`TransportError.retryable` population** rule (`id="transport-error-retryable"`) both cite "a surfaced 3xx" as a captured status mapping to `TransportError` with `retryable: false`. The page's *Classifier input surface* note (`id="classifier-input-surface"`) reads the HTTP-status signal exclusively from `ProviderResponse.status` via pi-ai's `StreamOptions.onResponse` callback; because redirect-following is internal to the transport layer loom neither configures nor can configure, a bare 3xx never surfaces as a resolved `ProviderResponse.status`, so the cited case is unreachable on the described input surface. The example adds no behavioural content — the leading "regardless of HTTP status → `TransportError`" clause already subsumes any 3xx — and misleadingly implies loom must arrange 3xx visibility (e.g. by disabling redirect-following), which it cannot and need not do.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:**
- `d1b39a1` — *pi-loom spec-plan: fix F-1853 — totalise TransportError.retryable + variant-selection over full status domain* (2026-06-28)

**History:** The `3xx` token enters `docs/spec_topics/pi-integration-contract/provider-error-mapping.md` for the first and only time in `d1b39a1`; `git log -S "surfaced 3xx"` and `git log -S "3xx"` on that path each return `d1b39a1` as the sole commit. `git show d1b39a1` shows the edit replaced the pre-existing enumerated rule ("Every other 4xx/5xx response, every HTTP-200 response carrying a non-overflow body-envelope error, and every network-level failure …") with a totalised rule ("Every classifier-reaching response … maps to `TransportError` regardless of HTTP status — … and every other captured status including a non-200 2xx (201/202/204) or a surfaced 3xx …"), adding the same "non-200 2xx … or a surfaced 3xx" parenthetical to both the **Provider error mapping** paragraph and the **`TransportError.retryable` population** rule in one commit. The parent blob (`d1b39a1^`) contains neither the `2xx` nor the `3xx` example. The defect is thus a side-effect of the F-1853 totalisation: broadening the rule to cover the full status domain introduced illustrative status examples without checking each example against the page's own input surface (`ProviderResponse.status` via `onResponse`), against which a 3xx is not reachable.

## Solution approach

Remove the misleading "or a surfaced 3xx" example from both the `id="provider-error-mapping"` paragraph and the `id="transport-error-retryable"` rule, leaving the totalised "regardless of HTTP status" catch-all and the retained "non-200 2xx (201/202/204)" example to cover the residual status domain. Any 3xx that did reach the classifier remains covered by the explicit "regardless of HTTP status" clause, so no behavioural detail is lost.
