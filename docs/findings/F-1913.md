## Problem

The `TYPE-11` test bullet on leaf `V2b` states alias-schema transparency as unconditional unfolding ("identified by the `=` form, not by what `R` resolves to") with no object-schema tail, which licenses an implementer to treat `schema Y = SomeObjectSchema` as fully transparent and admit cross-named structural matches. The paired `TYPE-11` bullet on leaf `V2b-T` asserts the opposite-looking rule — "aliasing an object schema still re-enters TYPE-10's nominal case" — with no unfold head. Read side by side the two surfaces describe incompatible engines, and an implementer following `V2b` could build transparency that fails `V2b-T`'s object-schema assertion. The spec rule at `type-system.md#type-11` reconciles both halves (the alias unfolds, and an object-schema RHS then obeys TYPE-10 nominal admission), but neither plan bullet states the full reconciled rule.

**History:** The plan corpus is git-tracked. `TYPE-11` did not exist on either leaf before `b93334a`; that single commit added the `TYPE-11` bullet to both `V2b-type-compat-engine.md` and `V2b-T-type-compat-engine.md` (and the matching spec rule + coverage row). The two divergent phrasings were authored together in that commit: `V2b` received the "identified by the `=` form, not by what `R` resolves to" wording with no object-schema tail, while `V2b-T` received the "aliasing an object schema still re-enters TYPE-10's nominal case" wording with no unfold head. `git log --follow` and `git log -G 'TYPE-11'` over both files show no later edits to either bullet, so the inconsistency has been present since the rule was introduced rather than drifting apart across commits.

## Solution approach

Rewrite both `TYPE-11` test bullets so each states the full reconciled rule from `type-system.md#type-11`: extend `V2b`'s bullet with the object-schema tail (the unfolded object schema then participates under TYPE-10's nominal rules), and give `V2b-T`'s bullet the matching unfold head (the alias unfolds to its RHS first). The two bullets should read as the same rule.

## Solution constraints

- The spec is read-only for this fix; `type-system.md#type-11` already states the reconciled rule and must not be edited.
- Object-schema aliases do unfold and then land on TYPE-10's nominal rules — do not adopt a framing in which object-schema aliases are exempt from unfolding.
