## Finding

`V2b`'s `TYPE-11` test bullet states the alias-transparency rule as: a `schema X = R` named type "unfolds to its RHS for `⊑`, recursing through nested aliases (identified by the `=` form, not by what `R` resolves to)." Read in isolation, this licenses an implementer to treat *every* `=`-form alias as fully transparent — including `schema Y = SomeObjectSchema` — which would admit cross-named structural matches whenever an alias names an object schema.

`V2b-T`'s `TYPE-11` test bullet, by contrast, asserts that "aliasing an object schema still re-enters TYPE-10's nominal case." Taken at face value next to `V2b`'s wording, the two surfaces read as opposite engines: one says all `=`-form aliases unfold transparently, the other says object-schema aliases do not get structural treatment. An implementer following `V2b`'s bullet could build full transparency that fails `V2b-T`'s object-schema assertion.

The spec already resolves this cleanly. `type-system.md` `TYPE-11` says the alias is identified "solely by the `schema X = R` `=` form — **not** by what `R` resolves to: aliasing an object schema (`schema Y = SomeObjectSchema`) unfolds to that object schema, **which then participates under the nominal rules of TYPE-10**, so alias transparency never reopens TYPE-10's nominal case." Both statements are simultaneously true: the alias *does* unfold, and the unfolded object schema *then* obeys TYPE-10 nominal admission. The plan defect is that `V2b`'s bullet omits the "then re-enters TYPE-10" tail while `V2b-T`'s bullet omits the "unfolds first" head, so neither plan surface states the reconciled rule and the pair reads contradictory.

The two divergent phrasings were authored together in that commit: `V2b` received the "identified by the `=` form, not by what `R` resolves to" wording with no object-schema tail, while `V2b-T` received the "aliasing an object schema still re-enters TYPE-10's nominal case" wording with no unfold head. `git log --follow` and `git log -G 'TYPE-11'` over both files show no later edits to either bullet, so the inconsistency has been present since the rule was introduced rather than drifting apart across commits.

## Solution Space

**Shape:** single

### Recommendation

Align both plan `TYPE-11` bullets with the reconciled spec rule in `type-system.md#type-11`, which already states both halves: the alias unfolds, and an object-schema RHS then obeys TYPE-10 nominal admission. Do not adopt a "only non-object aliases unfold" framing — per the spec, object-schema aliases *do* unfold; they simply land on TYPE-10's nominal rules afterward.

- In `docs/plan_topics/V2b-type-compat-engine.md`, extend the `TYPE-11` test bullet so it carries the object-schema tail. Recommended text: ``TYPE-11`: alias-schema transparency — a `schema X = R` named type unfolds to its RHS for `⊑`, recursing through nested aliases (identified by the `=` form, not by what `R` resolves to); when `R` is an object schema, the unfolded object schema then participates under TYPE-10's nominal rules, so alias transparency never reopens the nominal case.``
- In `docs/plan_topics/V2b-T-type-compat-engine.md`, give the `TYPE-11` test bullet the matching unfold head so it reads as the same rule. Recommended text: ``TYPE-11`: alias-schema transparency — a `schema X = R` named type unfolds to its RHS for `⊑` (identified by the `=` form, not by what `R` resolves to); alias-unfold cases hold (`"low" ⊑ Severity` for `schema Severity = "low" | "medium" | "high"`, and `StringOrNumber ⊑ string | number`), recursing through nested aliases; an alias of an object schema unfolds to that object schema, which then re-enters TYPE-10's nominal case.``

Keep `V2b`'s Adds field's existing "transparent alias-schema unfolding" phrase; it is consistent with the reconciled rule. The coverage-matrix `TYPE-1 … TYPE-11 → V2b` row and the spec stay untouched.

## Consequence

**Severity:** correctness

Two reasonable implementers reading the plan diverge: one follows `V2b`'s bullet and builds full alias transparency that grants cross-named structural admission for object-schema aliases; the other follows `V2b-T` and keeps object-schema aliases nominal. The two implementations fail each other's `TYPE-11` test, and only the second matches the spec.
