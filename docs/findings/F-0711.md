## Finding

The code-side tool-call surface in the spec is described in a single vocabulary almost everywhere: the `QueryError` variant is `CodeToolError`, its wire `kind` discriminator is `"code_tool"`, the lowering site is titled *"Tool execution from loom code"*, and the surrounding prose consistently distinguishes "loom code" calls from model-driven `@`-loop calls. The one exception is the synthesised `toolCallId` prefix at `host-interfaces-core.md` §*Tool execution from loom code*: it uses the word **`loom-direct:`** — the only place in the code-side surface where the discriminating word is "direct" rather than "code".

The same `loom-direct:` token is then mirrored in `tool-calls.md`'s *loom 1.0 seam — per-call timeout* note, so the divergence is now two-sited and cross-referenced. There is no rationale in the spec for the asymmetry — "direct" was a coherent distinguisher against the then-existing `ToolCallError` / `ToolFailureError` pair, but after that pair was renamed to `CodeToolError` / `ModelToolError` (wire kinds `"code_tool"` / `"model_tool"`), nothing in the corpus distinguishes "direct" from "code" semantically. A reader greping for `code_tool`, `CodeToolError`, or any variant of "code-side tool call" does not surface the `toolCallId` prefix paragraph; an auditor mapping the code-tool boundary has to know to additionally grep for "direct".

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `fae85d3` (2026-05-04, *"Tighten spec for implementation-plan readiness"*) + `6750c81` (2026-05-05, *"spec: consolidate QueryError variants; rename to InvokeInfraError / CodeToolError / ModelToolError"*)
**History:** The `loom-direct:` prefix was introduced in `fae85d3`, the same commit that first articulated code-side tool execution against the then-monolithic `spec.md`. At that time the code-side error variant was named `ToolCallError` with wire `kind: "tool_call"`, and the sibling model-driven variant was `ToolFailureError` with wire `kind: "tool_failure"` — so the prefix word "direct" was a coherent distinguisher against the *call*/*failure* sibling pair, neither of which used the word "code". Commit `6750c81` then swept the error-variant family, renaming `ToolCallError` → `CodeToolError` (wire `"tool_call"` → `"code_tool"`) and `ToolFailureError` → `ModelToolError` (wire `"tool_failure"` → `"model_tool"`); per its own commit message it "Slash-invocation per-kind table updated; human-facing 'failed' wording preserved per decision", but the sweep did not reach the `toolCallId` prefix at the lowering site or its cross-reference in `tool-calls.md`. The two-sited `loom-direct:` token thereafter sat as residue from the pre-rename vocabulary, surviving the file split (`fecb504`) and relocation to `docs/spec_topics/` (`31ff060`) untouched. The defect is the *interaction* between the two commits, not a defect of either in isolation — `fae85d3` was internally consistent against then-current siblings; `6750c81` would have been sweep-complete had it included the prefix in its scope.

## Solution Space

**Shape:** single

### Recommendation

Rename the synthesised `toolCallId` prefix from `loom-direct:` to `loom-code:` at both citation sites (`host-interfaces-core.md` line 85 and `tool-calls.md` line 40's parenthetical reference to the synthesised field). The "code" stem aligns the prefix with `CodeToolError`, `kind: "code_tool"`, and the *"Tool execution from loom code"* section title, restoring a single grep-able discriminator across the code-side tool-call surface. The change is wire-observable in the sense that the bytes passed to Pi's `execute(toolCallId, …)` differ, but Pi treats `toolCallId` opaquely (it is keyed by the spec only as a synthesised UUID with a prefix, and no Pi surface asserts the prefix string), and no plan leaf's *Tests* / *Ships when* clause asserts the prefix, so the rename has no acceptance-surface fallout.

Implementer note: the rename is to the prefix string only — the underlying UUID portion, the field name `toolCallId`, the field's role at the `execute()` boundary, and the per-call timeout open-struct seam in `tool-calls.md` are all unchanged. If the co-resolvable companion finding *"tool-calls.md — synthesised `loom-direct:` `toolCallId` format/uniqueness undefined"* is being addressed in the same pass, fix this one first so the suffix-format text that finding adds reads against the final prefix.

## Consequence

**Severity:** advisory

Vocabulary drift across the code-side tool-call surface. The runtime behaviour does not change (Pi treats `toolCallId` opaquely; no Pi surface keys on the prefix string), but readers and auditors mapping the surface boundary by greping the "code" vocabulary miss the prefix paragraph and have to discover the second spelling independently. Two reasonable implementers wiring the synthesis primitive would still produce interoperable behaviour, so this is not a correctness defect — it costs reader / auditor cycles.
