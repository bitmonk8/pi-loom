## Finding

`CIO-5` ([`ceilings-3-and-4.md#cio-5`](docs/spec_topics/hard-ceilings/ceilings-3-and-4.md)) states that ceiling #3 (binder per-class retry budget) never interleaves with ceiling **#1, #2, or #4** — a binder-bypass / load-time-only property spanning three distinct runtime-class siblings. `V16a` explicitly defers the cross-site, never-interleaves witness for the whole of CIO-5 to `H7a` ("CIO-5's 'ceiling #3 never interleaves with #1/#2/#4' … whose end-to-end witness is `H7a`'s co-occurring-breach integration assertion"), and `coverage-matrix.md` records `H7a` as CIO-5's "cross-site never-interleaves integration witness."

`H7a`'s final Tests bullet, however, drives exactly one pairing: a binder-retry-class breach (ceiling #3, `V11f`) co-occurring with the runtime-class `tool_loop.max_rounds` round boundary (ceiling **#2**, `V13c`). The #3-vs-#1 (`invoke`-entry depth) and #3-vs-#4 (AJV-boundary) pairings are never exercised by the fixture, yet the bullet closes with the unqualified claim that this single run is "closing the live-site integration gap that `V16a`'s synthesised-candidate suite leaves open." The `Ships when` clause carries the same single-pairing assertion.

The prose, the `V16a` deferral text, and the `coverage-matrix.md` attribution all assert complete closure of CIO-5's cross-site property while only one of its three sibling pairings is witnessed end-to-end. The claim overstates what the gate observes.

## Issue introduction

**Verdict:** single-commit introduction (later remediation)
**Introducing commits:** `b73a11a` ("pi-loom plan: resolve \"CIO-5 cross-ceiling arbitration verified only in isolation\"", 2026-06-10)
**History:** The plan corpus is git-tracked. `H7a` was created at `e7e51cc` (2026-06-10) as a single-scenario integration-acceptance leaf without any CIO-5 content. `git log -S "closing the live-site integration gap" -- docs/plan_topics/H7a-integration-acceptance.md` (also across `--all`) returns the single commit `b73a11a`, and `git show b73a11a --stat` confirms that commit touched only `H7a` (and the now-superseded `plan-review.md`), adding the CIO-5 co-occurring-breach Tests bullet, the matching `Ships when` clause, and the `V16a` Deps entry as the fix for the prior "CIO-5 cross-ceiling arbitration verified only in isolation" finding. The overclaim was introduced by that remediation: it supplied a single #3/#2 co-occurrence witness but worded it as closing the whole live-site CIO-5 gap.

## Solution Space

**Shape:** multiple

### Option A — Scope the claim to the representative pairing and narrow the attribution

**Approach.** Treat the #3-vs-#1 and #3-vs-#4 non-interleave as a structural binder-bypass consequence (`invoke(...)` does not invoke the binder; the binder runs only at slash-load), for which no positive co-fire event can be constructed, and downgrade the prose from whole-gap closure to a representative-pairing witness.

**Plan edits.**
- In `H7a`'s final Tests bullet and the corresponding `Ships when` clause, replace "closing the live-site integration gap that `V16a`'s synthesised-candidate suite leaves open" with text scoping the assertion to the #3/#2 (binder-retry vs runtime-class round-boundary) co-occurrence as the representative load-time-vs-runtime pairing, and naming where the #3-vs-#1 / #3-vs-#4 non-interleave is otherwise accounted for (the binder-bypass structural fact / the release-time residue inspection in `conventions.md`).
- In `coverage-matrix.md`'s `CIO-5` row annotation and closure-rationale paragraph, narrow the `H7a` co-witness annotation from "cross-site never-interleaves integration witness" to the #3-vs-runtime-class (#2) pairing it actually exercises.
- In `V16a` (Adds "Not witnessable" list and the CIO-5 Tests bullet) and `V16a-T` (CIO-5 Tests bullet), align the deferral prose so it no longer attributes the whole cross-site CIO-5 property to `H7a`.

**Spec edits.** None.

**Pros.** Honest about what the gate observes; keeps `H7a` a single deterministic fixture run; no new leaf or fixture.
**Cons.** CIO-5's invoke-bypass and AJV-boundary facets carry no driven end-to-end witness (they rest on the binder-bypass structural argument plus residue inspection).
**Risks.** If `V16a`/`coverage-matrix.md` are not updated in lockstep with `H7a`, the overclaim re-enters through the un-narrowed copy.

### Option B — Extend the fixture so all three CIO-5 pairings are witnessed

**Approach.** Extend `H7a`'s fixture (or add a sibling fixture stimulus) to also drive a #3-vs-#1 (invoke-entry) and a #3-vs-#4 (AJV-boundary) scenario, so all three CIO-5 pairings are witnessed end-to-end and the existing whole-gap claim becomes true.

**Plan edits.** Add the additional co-occurrence stimuli to `H7a`'s fixture and assertions; leave the closure prose and the `coverage-matrix.md` attribution as-is.
**Spec edits.** None.

**Pros.** Makes the coverage-matrix attribution literally accurate; each CIO-5 facet gets a driven witness.
**Cons.** The binder runs only at slash-load and `invoke(...)` bypasses it, so a genuine #3-vs-#1 / #3-vs-#4 *co-fire* event cannot be constructed — there is no live stimulus where a binder retry and an invoke-depth or AJV-boundary breach co-occur. The added "scenarios" would have to be synthetic, duplicating what `V16a`'s seam suite already covers in isolation.
**Risks.** Likely infeasible as a live co-fire; risks re-introducing the same isolation-only coverage under an end-to-end label.

### Recommendation

Take Option A. The #3-vs-#1 and #3-vs-#4 non-interleave is a binder-bypass structural fact — `invoke(...)` never invokes the binder, which runs only at slash-load — so no positive co-fire event exists for `H7a` to drive; the accurate fix is to scope `H7a`'s CIO-5 claim to the #3-vs-runtime-class (#2) representative pairing, narrow the `coverage-matrix.md` CIO-5 annotation to match, and route the un-driven pairings to the binder-bypass leaves that already account for them or to the release-time residue inspection. Resolve the lockstep set together: `H7a` (Tests + Ships when), `coverage-matrix.md` (CIO-5 row + rationale), and `V16a`/`V16a-T` (deferral prose) must all stop asserting whole-gap closure, or the overclaim survives in whichever copy is left untouched.

## Consequence

**Severity:** advisory

`coverage-matrix.md` reads CIO-5 as fully closed end-to-end while only the #3-vs-#2 pairing is actually witnessed; the #3-vs-#1 (invoke-bypass) and #3-vs-#4 (AJV-boundary) non-interleave facets ship with no driven end-to-end witness. An auditor trusting the matrix/`Ships when` prose concludes CIO-5 is completely closed, so a binder-bypass regression on the invoke or AJV path would not be caught by this gate. The leaf remains buildable and pickable, so this does not block — the defect is an overstated claim and a traceability inaccuracy.
