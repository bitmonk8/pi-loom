## Finding

In `docs/spec_topics/pi-integration-contract/conversation-drive.md`, the **Conversation drive — prompt mode** bullet says:

> loom 1.0 prompt-mode looms always use `deliverAs: "steer"` when not idle (the loom is the active driver and wants its message delivered to *this* turn); `deliverAs: "followUp"` is reserved for future considerations and is not emitted by loom 1.0.

"Is reserved for future considerations and is not emitted by loom 1.0" is a factual description, not a normative obligation. The reader cannot tell whether a conformant loom-1.0 implementation that *does* emit `deliverAs: "followUp"` violates the spec or merely contradicts a description of what current implementations happen to do. Every adjacent clause in this bullet — `MUST NOT await pi.sendUserMessage`, `MUST NOT attach a .catch handler`, `MUST NOT subscribe to the global pi.on("agent_end", …)`, `MUST NOT wrap the await in a try/catch`, `MUST NOT introduce additional access channels` — uses a RFC-2119 modal verb. This one sentence does not, leaving the only constraint on a loom-1.0-conformant runtime's `deliverAs` value implicit.

The defect is purely modal: the surrounding text already names `"steer"` as the runtime's option value and pins the producer (the runtime, not a `.loom` author), so the only ambiguity is the modal verb to use against `"followUp"`.

**Verdict:** single-commit introduction
**Introducing commits:** `c993401` (2026-05-04, "pi-loom spec: resolve \"Prompt-mode driver calls a non-existent ctx.sendUserMessage\"")
**History:** The `deliverAs: "followUp"` sentence entered the corpus in commit `c993401` on the prompt-mode bullet of the then-single-file `spec_topics/pi-integration-contract.md`, when the broader prompt-mode driver rewrite was authored. The sentence was added verbatim in that commit with the non-modal phrasing "is reserved for future considerations and is not emitted by V1" (`git log -S 'deliverAs: "followUp"'`, `git show c993401 -- spec_topics/pi-integration-contract.md`). Subsequent edits did not touch the modal structure: commit `a6d82db` (2026-05-29, prose `V1 → loom 1.0` rename) substituted `loom 1.0` for `V1` in this sentence, and commit `f5e89f4` (2026-06-04) moved the bullet from `spec_topics/pi-integration-contract.md` into the split file `docs/spec_topics/pi-integration-contract/conversation-drive.md` without text changes. The defect — descriptive prose where surrounding clauses are RFC-2119 modal — has been present since `c993401` and is attributable to that single commit's authoring of the new clause.

## Solution Space

**Shape:** single

### Recommendation

Replace the sentence

> `deliverAs: "followUp"` is reserved for future considerations and is not emitted by loom 1.0.

with

> A loom-1.0 prompt-mode runtime MUST NOT emit `deliverAs: "followUp"`; the option is reserved for a future extension.

Notes for the implementer:

- The MUST NOT is on the *runtime emitting the option*, not on a `.loom` author's source text — there is no surface-language construct that picks `deliverAs`. Keep the subject explicit ("a loom-1.0 prompt-mode runtime") so the rule does not accidentally pick up subagent-mode (`AgentSession.sendUserMessage`) or `ReplacedSessionContext.sendUserMessage`, neither of which uses the prompt-mode driver.
- Pair the new MUST NOT with one bullet on `V9c-T` (the existing test leaf) asserting that for every prompt-mode driven turn the runtime issues either `pi.sendUserMessage(text)` (idle branch) or `pi.sendUserMessage(text, { deliverAs: "steer" })` (mid-stream branch), and never `pi.sendUserMessage(text, { deliverAs: "followUp" })`. The existing PIC-2 / PIC-17 / PIC-18 tests already exercise the call shape; adding a `deliverAs`-value assertion to the existing mid-stream fixture is sufficient and does not require a new fixture.
- The reservation half of the original sentence ("reserved for future considerations") need not migrate to `future-considerations/surface-extensions.md`; "reserved for a future extension" as a trailing clause on the MUST NOT carries the same information and keeps the reservation visible at the only site that names the value.

## Consequence

**Severity:** advisory

A reasonable implementer reads the sentence as a description of intent rather than as a prohibition, sees no test obligation, and may legitimately emit `deliverAs: "followUp"` to queue rather than interrupt (e.g. as part of a future-considerations-flavoured experiment, or simply because the option exists on Pi's `sendUserMessage`). Nothing observable breaks at the call site — Pi accepts the option — but the runtime's prompt-mode delivery semantics diverge from the single-driver framing the rest of the bullet relies on (the `"steer"` rationale "the loom is the active driver and wants its message delivered to *this* turn"). Two conformant implementations could differ on this point with no scenario distinguishing them.
