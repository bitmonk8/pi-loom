## Finding

The audit's wire contract gives a CI parser two parallel ways to identify the leading discriminator on each stdout record, and both must agree. *Wire serialisation* on `audit-wire-and-canary.md` says the `audit/<class>/<family>/<symptom>` token MUST appear as **"the leading whitespace-separated field"** on every line, with the rest of the line then split on the ASCII tab to recover the four record fields. The two parsing models — tab-split into four fields, *and* prefix-match on the leading whitespace-separated field — agree only when the discriminator token itself contains no ASCII whitespace.

`<class>` is closed to the three literals `violation` / `infra` / `canary`, so it is whitespace-free by enumeration. `<symptom>` is constrained at the *Failure-surface contract* in `audit-failures.md` to "lowercase ASCII letters, digits, and `-` only; no whitespace, no `/`". `<family>`, by the same paragraph, is only "the implementation-chosen short name for one of the five families above (one stable `<family>` token per family, fixed at first implementation and pinned as a permanent test invariant)" — there is no character class, no whitespace prohibition, and no `/` prohibition. A conformant implementation that picks a `<family>` token like `node floor`, `peer dep`, or `infra/io` produces a discriminator whose leading whitespace-separated field is `audit/violation/node` (or whose `/`-segmentation has five segments instead of four), silently breaking the leading-token routing the *Wire serialisation* clause hands a CI parser.

The fix is to give `<family>` the same character class `<symptom>` already carries (and to extend it symmetrically to the structurally implicit constraints on `<symptom>` regarding `/`, which the surrounding wire serialisation already depends on).

## Solution Space

**Shape:** single

### Recommendation

In `audit-failures.md`'s *Failure-surface contract* paragraph, extend the `<family>` sentence so it carries the same character class `<symptom>` already does. Concretely, replace

> `<family>` is the implementation-chosen short name for one of the five families above (one stable `<family>` token per family, fixed at first implementation and pinned as a permanent test invariant)

with

> `<family>` is an implementation-chosen kebab-case short name for one of the five families above (lowercase ASCII letters, digits, and `-` only; no whitespace, no `/`; non-empty; one stable `<family>` token per family, fixed at first implementation and pinned as a permanent test invariant)

and align `<symptom>`'s existing clause to also state `non-empty` for symmetry. The constraint is normative on every `<class>` (`violation` / `infra` / `canary`) — the implementation-chosen `<family>` tokens scoped under each of the three classes (per the *Three-class partition* sentence in the same paragraph, the *Infrastructure-failure handling* `infra`-class `<family>` obligation in `audit-wire-and-canary.md`, and the *Non-empty-scan canary* `canary`-class `<family>` obligation in the same file) all flow through this single grammar definition, so no per-class restatement is needed.

Add a sibling assertion to the per-`<symptom>` negative-test-fixture obligation in the same paragraph: each fixture's emitted discriminator MUST additionally be checked to contain no ASCII whitespace and no `/` outside the three structural separators. Without this, the existing fixture assertion on the structural shape alone still admits a whitespace-bearing `<family>` token. The fixture floor is otherwise unchanged.

Implementer edge cases: (i) the existing `<symptom>` clause already says "no `/`", and the new `<family>` clause is the same — together they guarantee that splitting the discriminator on `/` yields exactly the four structural segments (`audit`, `<class>`, `<family>`, `<symptom>`); (ii) the `<class>` enumeration is unchanged and remains the closed three-element set; (iii) leaf-side tables that name `<family>` tokens (the family→step-2(b) routing table and the negative-test fixtures) inherit the constraint without further edit, since they consume the same token strings.

## Consequence

**Severity:** correctness

Two implementers reading the current text can reasonably diverge on whether a `<family>` token may contain whitespace or `/`; one of them produces wire output that silently bypasses the leading-token routing protocol a CI parser is told it may rely on, and the audit's whole machine-parseable surface goes with it. Negative-test fixtures asserting only that the emitted discriminator matches the structural `audit/<class>/<family>/<symptom>` shape (the assertion the surrounding paragraph pins) do not catch the divergence — a structural match on a four-segment-with-internal-whitespace token passes.
