## Problem

The *Failure-surface contract* paragraph in `audit-failures.md` (anchor `audit-failure-surface-contract`) constrains `<symptom>` to a kebab-case character class (lowercase ASCII letters, digits, and `-` only; no whitespace, no `/`) but defines `<family>` only as the "implementation-chosen short name for one of the five families", carrying no character class. The companion *Wire serialisation* clause in `audit-wire-and-canary.md` makes the `audit/<class>/<family>/<symptom>` discriminator both the leading whitespace-separated field and a `/`-segmented four-field token, so the two parsing models agree only when every segment is whitespace-free and `/`-free. A conformant implementation choosing a `<family>` token containing whitespace or `/` silently breaks the leading-token routing a CI parser is told it may rely on. Negative-test fixtures asserting only the structural `audit/<class>/<family>/<symptom>` shape do not catch the divergence.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `e5cd514` (`pi-loom spec: resolve "T35 SDK capability inventory closed-set negative-direction audit"`, 2026-05-09)
**History:** `git log -G 'kebab-case symptom'` over `docs/` shows `e5cd514` as the earliest commit introducing the `audit/<class>/<family>/<symptom>` grammar into the spec corpus. That commit's diff already states `<symptom>` as "kebab-case symptom identifier (lowercase ASCII letters, digits, and `-` only; no whitespace, no `/`)" while introducing `<family>` only as "short name for one of the five families above (one stable `<family>` token per family, fixed at H1 land time and pinned as a permanent test invariant)" with no character class. The asymmetry was present in the discriminator's first appearance; subsequent edits (notably `81bf52a` splitting the spec into topic files and `5e73072` renaming the npm scope) carried the asymmetry forward verbatim without revisiting it. The companion *Wire serialisation* clause that makes the constraint load-bearing — defining the discriminator as "the leading whitespace-separated field" alongside the tab-split four-field shape — was added in the same `e5cd514` commit, so the dual parsing model and the under-constrained `<family>` co-introduce.

## Solution approach

Extend the `<family>` clause at anchor `audit-failure-surface-contract` to carry the same character class `<symptom>` already declares — lowercase ASCII letters, digits, and `-` only; no whitespace, no `/`. Extend the per-`<symptom>` negative-test-fixture obligation in the same paragraph so each fixture's emitted discriminator is additionally checked to contain no ASCII whitespace and no `/` outside the structural separators.
