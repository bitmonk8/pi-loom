## Finding

PIC-50 in `conversation-drive.md` derives the synthesised `TransportError.provider` field as "the `Api`-shaped `Model<Api>.api` value of the user session's currently-selected model — read from `ctx.model`, the model the driven user turn actually routes through." The same derivation is reused by PIC-51 for the post-`waitForIdle()` `errorMessage` probe ("the `provider` field is derived per PIC-50's user-session-model derivation").

These two surfaces read `ctx.model` at two different moments. For the synchronous-throw path (PIC-50) the read happens at the `pi.sendUserMessage` call site, i.e. at dispatch. For the primary `stopReason: "error"` / `errorMessage` probe path (PIC-51) the read happens *after* `ctx.waitForIdle()` resolves — a re-read at probe time, after the entire driven user turn has run. PIC-50 asserts the attribution is "identical whether the failure surfaced as a synchronous throw at the call site or via the `errorMessage` probe," which holds only if `ctx.model` returns the same value at both moments — that is, only if the user session's selected model does not change across the span of the driven turn.

The spec never states this stability assumption. The page already documents three analogous loom-side consumption postures explicitly — driven-turn session-commit ordering, user-visible streaming ordering, and `ctx.waitForIdle()` no-rejection — each recording that Pi encodes no typed contract for the behaviour loom depends on and routing detection to an editorial-review checklist item in the Pi version-bump procedure. The model-stability assumption underlying provider attribution is the same kind of unpinned Pi behavioural dependency, but it is neither stated nor routed. A consumer reading PIC-50/PIC-51 has no signal that the probe-time re-read of `ctx.model` is load-bearing, and no version-bump gate re-audits it against future Pi minors that might allow mid-turn model switching.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commit:** `6f38d17` — "pi-loom spec-plan: fix F-1145 — derive prompt-mode TransportError.provider from ctx.model + sentinel" (2026-06-25)
**History:** `git log --follow` / `git show` on `conversation-drive.md` shows that before `6f38d17` PIC-50/51 sourced `provider` from "the loom's resolved `model:`", which carried no routing presupposition (the loom's own config needs none). Commit `6f38d17` switched the derivation to the user-session `ctx.model` and added the routing claim ("the model the driven user turn actually routes through, **not** the loom's resolved `model:`"), introducing the behavioural presupposition. The same commit touched only `spec-plan-review.md`, `queryerror-variants.md`, and `conversation-drive.md` — it did not add a consumption-posture paragraph or a `version-bump-step2.md` checklist item, so the presupposition entered unpinned at that commit. The sibling consumption-posture paragraphs and their checklist items (e.g. item (al) at commit `1ca13ec`) predate or are independent of it; the defect is the omission in `6f38d17`, not a multi-commit interaction.

## Solution Space

**Shape:** multiple

### Option A — Document model-stability as a loom-side consumption posture

- **Approach:** Add a consumption-posture presupposition block alongside the existing `driven-turn-commit-ordering-presupposition` / `waitforidle-no-rejection-presupposition` blocks, stating that PIC-50/PIC-51 provider attribution presupposes the user session's selected model (`ctx.model`) does not change between turn dispatch and the post-`waitForIdle()` probe in loom 1.0 prompt mode, that loom encodes no typed contract for this, and that a Pi minor permitting mid-turn model switching would falsify it.
- **Spec edits:** One new anchored presupposition paragraph in `conversation-drive.md`; one matching editorial-review checklist item in `version-bump-step2.md` routing detection to the version-bump procedure (mirroring items (ac)/(ak)/(al)).
- **Pros:** Idiomatic — matches the page's established pattern for unpinned Pi behavioural dependencies; minimal blast radius; keeps the single `ctx.model` derivation rule intact; gives the assumption a re-audit gate.
- **Cons:** Leaves the attribution dependent on a Pi behaviour loom cannot enforce; the diagnostic field can still be misattributed if the assumption is ever false.
- **Risks:** Low — purely additive prose plus a checklist row.

### Option B — Capture the model identity at dispatch and reuse it at probe time

- **Approach:** Capture `ctx.model` (or its `Model<Api>.api`) at the moment the driven turn is dispatched (`pi.sendUserMessage`), and use that captured value for provider attribution on both the synchronous-throw and the post-`waitForIdle()` `errorMessage` paths, so attribution never depends on a re-read.
- **Spec edits:** Reword PIC-50's derivation to pin a dispatch-time capture and have PIC-51 reference the captured value rather than re-reading `ctx.model`; note the capture site in the prompt-mode driver description.
- **Pros:** Eliminates the temporal assumption entirely; makes "attributed identically … whether synchronous throw or `errorMessage` probe" true by construction.
- **Cons:** Changes the attribution mechanism (a captured value threaded through to the probe), touching the driver (`V9c`) and the mapping (`V9n`); larger surface than Option A; the captured value can still be stale relative to the *actual* provider the turn ran on if Pi switched models mid-turn server-side, so it trades one assumption for another.
- **Risks:** Moderate — introduces a new captured-state lifetime to specify and test.

### Recommendation

Adopt **Option A**. The page already treats every unpinned Pi behavioural dependency on this path as a documented consumption posture routed to the version-bump editorial-review checklist; the model-stability dependency is the same shape and belongs in that mechanism. The implementer must (1) place the new presupposition block adjacent to the existing ones and give it a stable anchor, and (2) add the matching checklist item so the assumption is re-audited against each candidate Pi minor — without the checklist row the posture is recorded but never re-checked.

## Consequence

**Severity:** advisory

If the user-session model is replaced between turn dispatch and the post-`waitForIdle()` probe, the synthesised `TransportError.provider` is attributed to the wrong provider — a misleading value in the always-logged `kind: "transport"` runtime-event emission and the `QueryError`. loom 1.0 prompt-mode looms drive a single non-replaced user session, so the assumption likely holds today; the gap is that it is implicit and unguarded rather than that current behaviour is wrong.
