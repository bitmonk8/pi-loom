## Problem

`V9a`'s **Adds.** describes the home of two exported constants in source-layout terms — `FACTORY_PROBABLE_CAPABILITIES` is "one of the four pinned probe constants in the extension module's single source-of-truth pinned-constants block," and `SHUTDOWN_AWAIT_CAP_MS = 2000` is "the single physical home" that "lives alongside the probe constants here so the block has one explicit physical owner." These phrasings prescribe how the source is physically arranged rather than any observable behaviour. Per `conventions.md` §leaf-format **Adds.**, layout wording carrying no REQ-ID and naming no consumer-bound seam is a Class-3 shape mandate that is illustrative only and is not authorised by Adds prose. The genuinely load-bearing facts are the export/import relationships between `V9a` and its consumers.

## Issue introduction

**Verdict:** introduced-by-later-edit (multi-commit interaction)
**Introducing commits:** `b9de3ee` ("pi-loom plan: resolve \"V18a partition assertion imports undeclared V9a probe constant\"", 2026-06-11) introduced "single source-of-truth pinned-constants block"; `622d0d3` ("pi-loom plan: resolve \"SHUTDOWN_AWAIT_CAP_MS physical home contradicts pinned-constants-block placement\"", 2026-06-11) added "single physical home", "lives alongside", and "one explicit physical owner".
**History:** The corpus is git-tracked. The original plan build `c6a664e` (2026-06-10) did not contain any of these phrasings (`git log -G` over each phrase returns only the two later commits). Both introducing commits were prior plan-fixes resolving earlier review findings; each piled additional Class-3 layout prose into V9a's Adds while resolving a placement/import contradiction, rather than restating the contract as export/import edges. The defect is the cumulative residue of those two fix passes.

## Solution approach

In `V9a`'s **Adds.**, delete the source-layout phrasings ("in the extension module's single source-of-truth pinned-constants block", "is the single physical home of", "it lives alongside the probe constants here so the block has one explicit physical owner") and restate the load-bearing content as the export/import edges: `V9a` exports `FACTORY_PROBABLE_CAPABILITIES` as the symbol `V18a`/`V18c` reconcile their partition against, and exports `SHUTDOWN_AWAIT_CAP_MS = 2000`, which `V18c` reads via its build-time literal-read assertion and `V9g`/`V9i`/`V17a` import rather than redeclare. Retain the citation to `patch-skew-degradation.md §session_shutdown sub-step 3` for the literal value and the `V17a`-owns-semantics note, since both are content rather than layout.
