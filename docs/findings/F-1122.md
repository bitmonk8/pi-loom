## Finding

The [*Category-(1)/(3) inventory join key (per-category specialisation)*](../../docs/spec_topics/pi-integration-contract/audit-resolution.md#audit-category-1-3-inventory-join-key) paragraph defines the lookup for `pi.<member>` and `ctx.<member>` accesses as an **exact match between `<member>` and the rightmost dot-separated segment of an entry's `path` field** — and that rule is the only one written. It never restricts the resolver's candidate set to entries of the matching category (kind). A category-(1) `pi.modelRegistry(...)` access therefore matches an `SDK_SURFACE_INVENTORY` entry whose `path` is `"ExtensionContext.modelRegistry"` (a category-(3) entry), and a category-(3) `ctx.registerCommand(...)` access matches an entry with `path: "pi.registerCommand"` (a category-(1) entry). The rule is carrier-agnostic by omission.

The page does try to keep the world consistent through a side-channel: the [duplicate-`path`-segment uniqueness rule](../../docs/spec_topics/pi-integration-contract/audit-resolution.md#audit-category-2-inventory-join-key) under *Category-(2) inventory join key* extends symmetrically to "category-(1) and category-(3) audit-participating kinds … on the rightmost dot-separated segment" and pins the `literal-read/duplicate-rightmost-segment` discriminator. Read together with the under-specified resolver, that uniqueness rule is doing implicit work: it has to be an **inter-category** uniqueness invariant (`pi.modelRegistry` and `ExtensionContext.modelRegistry` may not coexist) rather than the natural **intra-category** one, otherwise resolution is non-deterministic. Whether this inter-category reading is intended — or whether the resolver is supposed to be kind-scoped and the uniqueness rule is intra-category — is exactly what the resolution paragraph leaves unsaid.

The consequence is that two compliant implementations diverge: one filters candidates by `kind`/category before applying rightmost-segment matching (and accepts side-by-side `pi.modelRegistry` and `ctx.modelRegistry` entries), the other does not (and rejects them as a duplicate-rightmost-segment collision). Both readings can be supported by the present text.

## Solution Space

### Recommendation

Scope inventory resolution to entries of the matching category (`kind`-filtered).

In *Category-(1)/(3) inventory join key (per-category specialisation)*, add a sentence requiring the resolver to filter `SDK_SURFACE_INVENTORY` to entries whose `kind` is the category-(1) entry kind (for a `pi.<member>` access) or the category-(3) entry kind (for a `ctx.<member>` access) coined per *Target surface categories* above, **before** applying the rightmost-segment match; entries of any other `kind` MUST NOT be candidates.

Rewrite the symmetric duplicate-rightmost-segment sentence under *Category-(2) inventory join key* so the uniqueness rule is scoped per category: `literal-read/duplicate-rightmost-segment` fires only when two entries of the *same* audit-participating category share a rightmost segment (no two category-(1) entries may share one; no two category-(3) entries may share one). State explicitly that a category-(1) entry and a category-(3) entry with the same rightmost segment do not collide, so sibling `pi.modelRegistry` and `ctx.modelRegistry` entries are legal.

To keep the resolver's filter unambiguous, the spec edit must either enumerate the `kind` discriminator strings that count as "category-(1)" vs. "category-(3)", or anchor-cite the implementation-owned enumeration that does.

Implementer edge cases to honour: the resolver computes the candidate set by `kind` *before* applying the rightmost-segment match, so a `pi.X` access against an inventory containing only `ExtensionContext.X` correctly fires the family-(1) off-inventory discriminator rather than silently resolving; and the `literal-read/duplicate-rightmost-segment` assertion partitions entries by category before checking for collisions.

## Consequence

**Severity:** correctness

Two reasonable implementers produce different audit verdicts for the same source tree: under the kind-scoped reading, `pi.modelRegistry` correctly fires as off-inventory when only an `ExtensionContext.modelRegistry` entry exists; under the carrier-agnostic reading, it resolves silently. Either implementation also disagrees on whether the inventory may carry sibling `pi.<member>` and `ctx.<member>` entries with the same trailing segment without firing `literal-read/duplicate-rightmost-segment`. Surface-set closure — the audit's central contract — is implementation-defined where it should be pinned.
