## Finding

`V18c`'s version-bump editorial-review checklist (items `(a)`–`(ak)`, per [`version-bump-step2.md`](../../docs/spec_topics/pi-integration-contract/version-bump-step2.md)) has two separable verification surfaces. The **substantive** surface — whether each host-presupposition audit was *performed correctly* against the candidate Pi minor — is irreducibly human judgment; no machine cross-check can decide whether a contributor actually diffed `AgentSession.prototype.prompt` or merely typed `pass`. The finding does not contest that half.

The **structural** surface is different: it is mechanically decidable from the recorded outcomes alone. `V18c`'s **Ships when** states two structural properties — (1) the bump commit message records a per-item outcome for *every* item `(a)`–`(ak)` (with `(e)` carrying its two sub-outcomes `(e.i)`/`(e.ii)`), and (2) no MUST-level item `(a)`–`(e)` carries a `fail` (or a conservative-posture-recorded non-positive) outcome, which blocks the bump. Both of these are pure shape/membership checks over the outcome record — yet `V18c`'s **Ships when** routes *both* of them to "editorial review — not by `npm test`." A contributor who omits an item's outcome row (non-conformant per the spec's "omitting the row is non-conformant" rule) or who merges a commit carrying a MUST-level `fail` is caught only if a human reviewer notices, even though either condition is deterministically computable.

The fix is to add a mechanical backstop for the structural half while leaving the substantive audit editorial. Because the spec ([`version-bump-step2.md`](../../docs/spec_topics/pi-integration-contract/version-bump-step2.md)) currently designates the *free-prose bump commit message* as the sole record location and says acceptance is "confirmed by editorial review," giving the structural half a deterministic gate requires a recording surface a test can parse — which pulls a spec edit into scope alongside the `V18c` / `V18c-T` plan edits.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f1cda48` (primary — *pi-loom plan: resolve "V18c mislabels editorial-review checklist as a static build-time gate"*, 2026-06-11) and `828b7d3` (*pi-loom plan: resolve "V18c editorial-checklist MUST/SHOULD blocking distinction missing from exit gate"*, 2026-06-16); inception context `c6a664e` (*pi-loom plan: build/update plan for spec.md + review*).
**History:** At inception (`c6a664e`) `V18c`'s **Ships when** carried no done-condition for the editorial-review checklist at all — it was loosely listed among the leaf's "static build-time gates" in **Adds** with no Tests/Ships-when coverage. `f1cda48` correctly reclassified the checklist as *not* a build-time gate and, in doing so, gave it its first concrete done-condition: structural arm (1) — "the bump commit message records a per-item outcome for every item `(a)`–`(ak)`" — routed to "editorial review — not by `npm test`." `828b7d3` then added structural arm (2) — "no MUST-level item `(a)`–`(e)` carries a `fail`" — onto the same editorial-review path. The mechanically-decidable structural half was therefore assembled across these two commits and routed to human review at both steps; the defect is not a regression of a previously-mechanized gate but the state established when the checklist's verification was first specified.

## Solution Space

**Shape:** multiple

### Option A — Mechanical gate over a structured commit-message record

**Approach.** Have the spec designate a machine-parseable outcome record inside the bump commit message (keyed by item id → verdict + rationale), and add a `V18c-T` test that reads `HEAD`'s commit message, asserts every item `(a)`–`(ak)` (and `(e.i)`/`(e.ii)`) is present with a recognised verdict, and reddens when any MUST-level item `(a)`–`(e)` carries `fail`. Reflect the new gate in `V18c` **Tests** and **Ships when**.

**Plan edits.** `V18c-T` **Tests**: add the commit-message structural-parse assertion. `V18c` **Adds**/**Ships when**: name the structural gate and move arms (1)/(2) from "confirmed by editorial review" to the `npm test` arm, leaving only the substantive verdict-correctness editorial.

**Spec edits.** `version-bump-step2.md` preamble / Outcome recording conventions: pin the parseable record location and shape.

**Pros.** Keeps the canonical record where the spec already puts it (the commit message); single source of truth.

**Cons.** A test keyed on git commit-message text is awkward in the build-time-test model — the record does not exist until the commit is authored, so the gate cannot run pre-commit and is fragile against message reformatting.

**Risks.** CI ordering: the gate can only fire post-commit, weakening its value as a pre-merge guard unless wired into a commit/PR hook the plan does not currently model.

### Option B — Mechanical gate over a tracked in-repo audit artefact

**Approach.** Have the spec relocate (or mirror) the canonical per-bump outcome record into a tracked in-repo artefact with a parseable shape, and add a `V18c-T` test asserting that artefact contains every item `(a)`–`(ak)` with a recognised verdict and no MUST-level `fail`. The substantive audit stays editorial.

**Plan edits.** Same `V18c` / `V18c-T` edits as Option A, but the test reads the tracked artefact rather than the commit message.

**Spec edits.** `version-bump-step2.md`: designate the in-repo artefact as the (or a) canonical record location.

**Pros.** The structural gate runs inside `npm test` like every other `V18c` gate — deterministic, runnable locally and pre-merge, not coupled to git plumbing.

**Cons.** Introduces a second record location unless the spec moves the canonical record off the commit message; risk of divergence between the commit-message record and the artefact if both are retained.

**Risks.** Spec must state which record is authoritative to avoid two-source drift.

### Recommendation

Adopt **Option B**: relocate (or make canonical) the per-bump outcome record into a tracked in-repo artefact and gate its structural shape in `V18c-T`, because the resulting check runs inside `npm test` alongside the other `V18c` gates and pre-merge, rather than depending on parsing the git commit message after the fact. The spec edit in `version-bump-step2.md` is the owner of the recording-location contract and must be touched for either option — the substantive audit (verdict correctness) stays editorial and is explicitly out of scope for the mechanical gate, since it is not machine-decidable. The implementer must ensure the spec names a single authoritative record location so the commit-message and artefact records cannot diverge.

## Consequence

**Severity:** advisory

A non-conformant bump — an omitted per-item outcome row, or a commit that merges with a MUST-level item `(a)`–`(e)` recorded `fail` (a state the spec says blocks the bump) — would not be caught by CI; detection rests entirely on a human reviewer noticing a deterministically-checkable defect. `V18c` remains implementable and a careful reviewer can perform the structural check correctly; the gap is the absence of a mechanical backstop for a property that is mechanically decidable.
