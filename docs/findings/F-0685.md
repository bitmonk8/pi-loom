## Finding

The DISC-4 paragraph in `discovery-sources.md` opens with a universally-quantified rule — "**none of the colliding entries register**" — that ranges over both same-format (loom-vs-loom) collisions and cross-format (loom-vs-Pi-owned) collisions, then closes the same paragraph with a carve-out — "For the cross-format slice (`.loom` vs Pi-owned `.md`), the Pi-owned entry survives … only the colliding loom(s) drop." A clause-by-clause reading therefore yields two incompatible behaviours for the cross-format slice: DROP-ALL (read the universal opening as authoritative) versus DROP-LOOM-ONLY (read the carve-out as authoritative). The next paragraph ("When the loom extension discovers, on its `session_start` handler…") reinforces the carve-out, but a reader who stops at the bold opening — or who treats the universally-quantified sentence as the rule statement and the rest as elaboration — will implement DROP-ALL and unregister the Pi-owned prompt template / skill / sibling-extension command on collision.

The defect is the universal opening, not the carve-out. The actual intended rule is bimodal and asymmetric: loom-vs-loom same-priority collisions drop every colliding loom; cross-format collisions drop only the loom(s) and leave the Pi-owned entry registered (because the loom extension cannot unregister Pi-owned commands — see [Registration steps — *Structural changes*](../pi-integration-contract/registration-steps.md#structural-changes-no-unregister)). Encoding both arms in one universally-quantified sentence plus a trailing carve-out is what produces the contradiction; splitting them into two explicitly-scoped arms removes it.

**Introducing commit:** `d124b8c` — "spec: same-priority slash-name collisions are uniform load-time errors (Decision #10)" (Tue May 5 2026)
**History:** The pre-`d124b8c` paragraph was titled *Slash-name collisions across formats* and described only the cross-format arm (loom drops, Pi-owned survives) — no universal "none of the colliding entries register" claim existed, so no contradiction was reachable. `d124b8c` generalised the section to *Slash-name collisions at the same priority* covering both same-format and cross-format arms; in the same edit it introduced the bold universal opening "**none of the colliding entries register**" *and* the trailing cross-format carve-out "the Pi-owned entry survives — only the colliding loom(s) drop." The contradiction has therefore been present since the rule's current shape was introduced — five later edits to the same paragraph (`7e440c9`, `d488eaa`, `f2a82ad`, `4a10cde`, `f5e89f4`) refined the diagnostic code reference, the hyphen-normalisation clause, the `de-register` wording, and the file split, but none touched the opening/carve-out tension. `git log -S "none of the colliding entries register" -- "*.md"` confirms `d124b8c` as the first appearance of the universal opening.

## Solution Space

**Shape:** single

### Recommendation

Replace the single bimodal paragraph with two explicitly-scoped arms; remove the universally-quantified opening and the trailing carve-out together, so neither sentence can be read as authoritative in isolation.

Concretely, rewrite the DISC-4 lead paragraph as roughly:

> **DISC-4. Slash-name collision rules.** Detection runs on the **final derived name** (after `pi.looms` mapping and `as` rename), not the source filename, and applies to two distinct collision shapes:
>
> 1. **Loom-vs-loom (same priority).** Two or more `.loom` candidates at the same priority level deriving the same slash name — e.g. two packages each shipping `code-review.loom`, two settings entries that both expand to a `code-review.loom`, or two `--loom` components resolving to files with the same stem. **None of the colliding looms register**, and the loom extension emits `loom/load/cross-format-collision` naming every colliding path.
> 2. **Loom-vs-Pi-owned (cross-format).** A `.loom` candidate deriving the same slash name as a Pi-owned `.md` prompt, `.md` skill, or another extension's command, observed at the `session_start` `pi.getCommands()` comparison. **Only the colliding loom(s) drop**; the Pi-owned entry remains registered, because the loom extension cannot unregister Pi-owned commands (see [Registration steps — *Structural changes*](../pi-integration-contract/registration-steps.md#structural-changes-no-unregister)). The same diagnostic is emitted naming the `.loom` path(s) and the colliding Pi-owned entry.

Then keep the existing follow-up material (the "Settings entries that resolve to the same absolute path are deduplicated silently before collision detection" sentence; the "When the loom extension discovers, on its `session_start` handler…" paragraph; the `LoomRegistry`-side drop paragraph; the superseded-entry-dispatch cross-reference) as elaborations that the two arms now both anchor cleanly into.

Edge cases the implementer must watch:

- The diagnostic code stays `loom/load/cross-format-collision` (locked per DIAG-3); the rename is deferred to loom 2.0 per the sibling finding on the misleading code name. The Trigger column of the `loom/load/cross-format-collision` row in `code-registry-load.md` already describes both arms — verify it does not need to be edited.
- The `V10a` / `V10a-T` `DISC-4` test bullet must be widened to assert both arms (currently only the cross-format arm is asserted); the same-format-DROP-ALL arm must witness that *every* colliding loom drops with a single diagnostic listing all paths, not a per-pair diagnostic.
- The "three packages each shipping `lint.loom` produces a single error listing all three paths" example currently sits inside the universal opening sentence; it must move into arm (1) so it stays anchored to the same-format slice.
- The DISC-4 anchor (`<a id="disc-4"></a>`) and the `loom/load/cross-format-collision` cross-link from `drain-state-contract.md#superseded-entry-dispatch` must remain stable — both still point at the same paragraph, just with a different internal structure.

## Consequence

**Severity:** correctness

Two reasonable implementers reading the same paragraph will disagree on what happens when a loom collides with a Pi-owned `.md` prompt or skill: one drops only the loom (matching Pi's contract that the loom extension cannot unregister Pi-owned commands), the other drops the Pi-owned entry too (matching the universal opening). The DROP-ALL implementation will additionally call a non-existent `pi.unregisterCommand`, fail silently or throw at runtime depending on how it discovers Pi has no such API, and effectively make every `.loom`/`.md` name collision a Pi-side outage for the surviving prompt template.
