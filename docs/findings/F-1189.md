## Finding

The orientation page `docs/spec_topics/pi-integration.md` makes two claims about
host-observable behaviour that it does not itself ground, and that have no
forward link back here to where they would be grounded:

1. **File-watcher note content.** The file-watcher bullet says
   *"the watcher emits a one-line system note prompting the user"* but does not
   name the system-note channel, quote the rendered template, or even commit to
   `/reload` being the action the user is prompted to run — the immediately
   preceding sentence does mention `/reload`, but the prompt's wording is left
   entirely to the reader. The verbatim template
   (`loom watcher: <N> file(s) added or removed; run /reload to refresh the
   slash command list`, `display: true`) is fully pinned in
   `pi-integration-contract/registration-steps.md` step 5 *Structural changes*,
   but the orientation page does not link there. A reader using this page as
   the entry into the file-watcher contract cannot tell from it whether the
   prompt is a transient toast, a transcript note, or a diagnostic.

2. **Slash-command autocomplete rendering.** The slash-command discovery bullet
   asserts as fact that *"the `description` field from frontmatter is what
   populates the autocomplete entry that Pi's slash-command UI shows"*. The
   *converse* (the `argument-hint` field is **not** rendered, because
   `RegisteredCommand` exposes no `argumentHint` slot) is grounded in
   `frontmatter/frontmatter-fields-a.md`'s `#argument-hint-binder-only`
   paragraph and routed for re-audit nowhere. The positive claim — that the
   `description` field *is* rendered, and where in the UI — has no SDK type
   pin, no `pi.registerCommand` shape pin, and no entry on the
   `version-bump-step2.md` editorial-review checklist for unpinned host
   presuppositions. A Pi minor that re-purposed the field, moved the slot,
   or dropped extension-command description rendering altogether would
   silently invalidate the orientation framing without reddening any
   build-time gate.

Both gaps are orientation-level: the file-watcher template is pinned in the
contract page, and the autocomplete-rendering behaviour is observable enough
that loom 1.0 will work or not. They are flagged together here because they
share the same shape — a fact about host behaviour stated declaratively on the
orientation page with no anchor to where the host contract owns or audits it.

## Solution Space

**Shape:** multiple

The finding bundles two independent obligations (orientation-page gloss vs
unpinned host presupposition) that the per-finding fix loop must address
sequentially to keep each diff small. Option A is the smaller, scope-bounding
edit and resolves first; Option B then lands the new editorial-review
checklist entry against a stable orientation-page baseline.

### Option A — Anchor the watcher-note gloss to its owning contract

*Approach.* Tighten the file-watcher bullet so the *"one-line system note
prompting the user"* phrase becomes a forward link to the verbatim template
the contract page already owns, without restating the template here. The
discovery-bullet wording is untouched.

*Spec edits.* In `docs/spec_topics/pi-integration.md`, rewrite the
file-watcher bullet's second sentence to something like *"Addition or
removal of a file requires `/reload`; the watcher emits a one-line
`loom-system-note` whose verbatim template (`loom watcher: <N> file(s) added
or removed; run /reload to refresh the slash command list`) is pinned in
[Pi Integration Contract — Extension entry point step 5,
*Structural changes*](./pi-integration-contract.md)."* No change to
`registration-steps.md` or any other file.

*Pros.* One-paragraph edit; preserves the orientation page's role as a
summary; eliminates the gloss by anchoring it; no normative content
re-stated and so no risk of the two sites drifting.

*Cons.* Slightly lengthens an already long bullet. Does not address the
slash-UI rendering presupposition.

*Risks.* Future edits to the watcher template in `registration-steps.md`
must remember to refresh the parenthetical quotation here; an in-prose
restatement of a normative template always carries that risk.

### Option B — Add the slash-UI rendering presupposition to the version-bump editorial-review checklist

*Approach.* Treat the assertion *"the `description` field from frontmatter
is what populates the autocomplete entry that Pi's slash-command UI
shows"* as an unpinned Pi behavioural presupposition on the same footing
as items (f)–(ao) of `version-bump-step2.md`'s *Editorial-review checklist
for unpinned host presuppositions*, and add a SHOULD-level item to that
checklist that re-audits the rendering against each candidate Pi minor.
Forward-link the orientation page's discovery bullet to the
`argument-hint-binder-only` paragraph in
`frontmatter/frontmatter-fields-a.md` (which already grounds the
negative-`argumentHint` half) and to the new checklist item.

*Spec edits.*

1. Append one item to `version-bump-step2.md` (e.g. item `(ap)`,
   following the existing alphabetical sequence): *Pi extension-command
   autocomplete entry rendering* — *the contributor SHOULD confirm
   against the candidate `@earendil-works/pi-coding-agent` minor that
   Pi's slash-command autocomplete dropdown still renders the
   `description` field passed to `pi.registerCommand({ description,
   handler })` (per
   [Registration steps — step 3](./registration-steps.md#slash-handler-registration))
   as the entry's user-visible text*. Give the item a stable
   `<a id="bump-checklist-registercommand-description-render"></a>`.
2. Add a parenthetical to the slash-command discovery bullet in
   `docs/spec_topics/pi-integration.md` pointing at the new checklist item
   (mirroring the parenthetical the `argument-hint` half already carries
   into `#argument-hint-binder-only`).

*Pros.* Closes the audit gap symmetrically with the
`argument-hint`/`argumentHint` negative-presupposition framing the spec
already carries; makes the assertion survive a Pi minor bump without
silently going stale.

*Cons.* Grows the editorial-review checklist by one item (already
~32 items long); adds contributor-discipline burden on every minor bump.

*Risks.* The new presupposition is editorial-only (SHOULD-level
audit-but-record), matching items (f)–(ao); no build-time gate fires if a
contributor forgets to record it, so the same staleness mode the existing
checklist carries applies here too. Acceptable because the rendering
failure is operator-visible the first time anyone types `/`.

### Recommendation

Resolve **Option A first**, then **Option B** against the A-edited
baseline. Option A is the smaller, isolated edit (one paragraph on one
file) and bounds the orientation-page diff before Option B adds a new
checklist entry that lives on a different page; resolving A first prevents
Option B's editorial-review checklist update from being mixed into a
larger orientation-page rewrite. Implementer edge cases: (Option A) the
parenthetical quoting the watcher template must use the exact byte string
pinned in `registration-steps.md` step 5, including the literal `file(s)`
spelling (always parenthesised `s`, never pluralised, per the substitution
rule that step pins); (Option B) the new checklist item ID must follow the
`bump-checklist-<slug>` convention and slot alphabetically without
colliding with the existing `(a)`–`(ao)` letter space — appending as
`(ap)` is the cleanest extension.

## Consequence

**Severity:** advisory

A reader using the orientation page as the entry into the file-watcher
contract cannot reconstruct what the user sees from the prose alone, and the
slash-UI rendering presupposition is taken on faith with no per-bump audit —
so a Pi minor that re-shaped the extension-command autocomplete entry would
silently invalidate the framing without surfacing on the build-time
SDK-surface-inventory tests. Neither gap blocks implementation; both are
documentation-completeness defects.
