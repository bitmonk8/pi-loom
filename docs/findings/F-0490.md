## Finding

The plan applies the word "terminal" to two distinct leaves in the Horizontal phases without stating their relative order, so a reader cannot tell from the labels alone which leaf is actually last. `H7a` is presented as "Terminal integration-acceptance run" in the §Horizontal phases list and as "A terminal integration-acceptance gate" in its own **Adds**, while `H6a` is "the terminal `H6a` leaf" (§How to use, step 5), "the terminal step" (§Release gate), and "The terminal release-gate step" (`H6a` **Adds**). Two steps cannot both be terminal; only one can be last.

The true order is unambiguous from the dependency graph: `H6a`'s **Deps** are `H5b, H7a`, so `H6a` sequences after `H7a` and nothing depends on `H6a`. `H6a` is therefore the genuinely-last leaf, and the "terminal" label on `H7a` is the misleading one. The defect is purely in the prose labels; execution sequencing is fixed correctly by `H6a`'s `Deps`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `5353dd7` ("pi-loom plan: resolve \"Release-gate activation has no owning leaf\"", 2026-06-10 09:12) introduced `H6a` as "The terminal release-gate step"; `e7e51cc` ("pi-loom plan: resolve \"Plan has no terminal end-to-end integration-acceptance leaf\"", 2026-06-10 10:26) introduced `H7a` as "Terminal integration-acceptance run".
**History:** Neither commit was independently wrong — each named a newly-added leaf "terminal" while resolving a separate plan gap. `git log -S` confirms `H6a`'s "terminal release-gate step" label landed first (`5353dd7`); roughly an hour later `e7e51cc` added `H7a`'s "Terminal integration-acceptance run" label against the already-present `H6a` label, at which point the two-terminal collision emerged. The §Release-gate prose later edited by `ea6b1da` preserved the `H6a` "terminal step" phrasing. The ambiguity is the joint product of the two leaf-introduction commits, not a single defect-introducing edit.

## Solution Space

**Shape:** single

### Recommendation

Reserve "terminal" for `H6a` (the genuinely-last leaf, since its **Deps** `H5b, H7a` place it after `H7a`) and strike the word from `H7a`'s naming:

- In `docs/plan.md` §Horizontal phases, change the `H7a` list entry text from `Terminal integration-acceptance run (cross-slice end-to-end gate)` to `Integration-acceptance run (cross-slice end-to-end gate)`.
- In `docs/plan_topics/H7a-integration-acceptance.md`, change the heading from `# \`H7a\` — Terminal integration-acceptance run (cross-slice end-to-end gate)` to `# \`H7a\` — Integration-acceptance run (cross-slice end-to-end gate)`.
- In the same file's **Adds**, change the opening `A terminal integration-acceptance gate that runs…` to `An integration-acceptance gate that runs…`.

Leave every "terminal" reference to `H6a` unchanged: §How to use step 5 ("the terminal `H6a` leaf"), §Release gate ("the terminal step"), and `H6a` **Adds** ("The terminal release-gate step") are all correct.

Edge case for the implementer: avoid re-attaching "terminal" to `H7a` via a qualifier such as "terminal integration-acceptance gate that `H6a` depends on" — that re-introduces the colliding word. `H7a`'s **Adds** may continue to describe the leaf as the final cross-slice integration gate in non-"terminal" wording without conflict.

## Consequence

**Severity:** cosmetic

A reader scanning the plan sees two leaves both called "terminal" and cannot infer from the labels which one is the final release step. The actual order is fixed unambiguously by `H6a`'s **Deps** (`H5b, H7a`), so no implementer mis-sequences; the cost is reader confusion only.
