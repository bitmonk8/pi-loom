# Governance

*Audience.* This page is directed at spec maintainers and at the coverage-matrix closing CI gate (specified in the plan corpus), not at implementers of the loom runtime. It governs the spec corpus itself — how REQ-IDs are coined, anchored, retired, and gated — and sources no software requirement on the runtime, the binder, the type system, or the Pi integration. Implementers who restrict their reading per GOV-10 will reach this page only via the initial REQ-ID anchor pass and the coverage-matrix closing gate (both specified in the plan corpus).

This page owns the spec corpus's REQ-ID governance: the per-page prefix table, the per-prefix retirement registry, and the rules (GOV-1 through GOV-12, GOV-14, GOV-15, GOV-16; GOV-13 retired — see [Retired REQ-IDs](#retired-req-ids)) that govern how REQ-IDs are coined, anchored, retired, and gated. GOV-16 additionally governs a second cross-page-citable identifier class — *stable inline labels* (currently `HC3-a`…`HC3-e` and `NOCEIL-1`…`NOCEIL-4`, both bound to [`hard-ceilings.md`](./hard-ceilings.md)) — on terms parallel to GOV-1 / GOV-3 / GOV-8 / GOV-9. The rules apply to every non-narrative page under `spec_topics/`; tooling enforcement lives in the coverage-matrix closing CI gate (specified in the plan corpus).

## REQ-ID prefix table

**GOV-1 (anchor placement).** Each spec page that carries normative obligations is assigned a stable per-page REQ-ID prefix (table below). The initial REQ-ID anchor pass (specified in the plan corpus) inserts `PREFIX-N` anchors into each non-narrative page in the table.

- **Canonical form.** Each REQ-ID's defining anchor is the inline `**PREFIX-N.**` marker. The bold-with-period decoration is normative: it distinguishes anchor sites (`**BNDR-7.**`) from back-references (`per BNDR-7`) and is the witness pattern for the anchor-insertion gate, the Reused-ID gate, and the Dense-numbering gate (all specified in the plan corpus). Tooling that emits or consumes anchors MUST treat the bold-with-period bytes as part of the anchor form.
- **Required HTML-anchor contexts.** The `<a id="prefix-n"></a>` HTML form MUST accompany the inline marker at every REQ-ID anchor site, in each of the four enumerated contexts below. These are the cases where the inline marker is not by itself a sufficient anchor (Markdown bold-with-period either does not render, breaks the surrounding construct, or — being rendered as bold text — does not by itself produce a stable URL fragment in any common Markdown renderer):
    - inside a Markdown table cell;
    - inside an ATX heading (`#` … `######`);
    - on the line immediately preceding a fenced code block whose content is the rule's normative example;
    - in body-paragraph context, where the inline marker renders fine as bold text but does not by itself produce a stable URL fragment in any common Markdown renderer, so the HTML form is the canonical fragment source for body-paragraph REQ-IDs (the three rendering-driven contexts above each carry their own per-context placement mechanics in *Edge cases* below — heading-slug repointing for ATX headings, the table-cell leading-token rule, and the line-preceding-a-fenced-code-block placement). *Body-paragraph context* is defined operationally as the negation of the three rendering-driven contexts above: every source-line position that is neither inside a Markdown table cell, nor inside an ATX heading, nor on the line immediately preceding a fenced code block whose content is the rule's normative example. This explicitly includes prose paragraphs, top-level and nested unordered list items, ordered list items, blockquote bodies, definition-list bodies, footnote bodies, and the line *following* a fenced code block.

  The HTML form is permitted only as part of a dual-form pair (per *Dual-form layout* below — i.e. co-located on the same source line as an inline `**PREFIX-N.**` marker) in one of the four enumerated contexts above; no other source-line position permits the HTML form. A bare `<a id="prefix-n"></a>` anchor with no co-located inline marker on the same line is not a GOV-1 anchor; the anchor-insertion CI gate (specified in the plan corpus) MUST exit non-zero on any such bare HTML anchor, and the plan corpus is the normative source for the gate's failure surface (exit code, per-offence message format, accumulation semantics, and output stream), parallel to GOV-2's floor obligation. A REQ-ID inside a fenced code block, an inline code span, or an HTML comment is neither an anchor nor a back-reference and is invisible to GOV-3 extraction (per the exclusion list in GOV-3 below). *Multi-context overlap.* The dual-form anchor is required in every one of the four contexts above, so the applicability question never depends on which context a site is classified under; only the per-context *mechanics* in *Edge cases* below differ. When a REQ-ID anchor site syntactically matches more than one enumerated context (e.g. a table-cell marker whose row is followed by a fenced code block, or an ATX-heading marker whose line precedes a fenced code block), the more-specific (rendering-driven) context determines which *Edge cases* mechanics apply, in this fixed precedence order: (1) table-cell, (2) ATX-heading, (3) line-preceding-a-fenced-code-block, (4) body-paragraph (which by construction cannot overlap with any of the three rendering-driven contexts, per the operational definition above). The first context whose syntactic conditions the site satisfies wins; lower-ranked matches are suppressed for the purposes of selecting *Edge cases* mechanics.
- **Dual-form layout.** When the HTML form is used, it MUST appear on the same source line as the inline marker, in the order `<a id="prefix-n"></a> **PREFIX-N.**` — the HTML anchor first, a single ASCII space, then the inline marker. This ordering keeps the inline marker adjacent to the rule text it introduces and gives the coverage-matrix gate a single regex to anchor against (`<a id="prefix-n"></a> \*\*PREFIX-N\.\*\*`). The regex is matched line-by-line against each source line after stripping (a) any leading ASCII whitespace, and (b) a Markdown block-prefix sequence built from the closed grammar below; the dual-form pair MUST be the first non-prefix tokens on the line. Tabs anywhere in the line-prefix region (i.e. before the dual-form pair) are normalised to spaces per CommonMark before stripping. The permitted block-prefix sequence has three ordered slots, each independently optional, in this fixed order; each slot's prefix bytes MAY be preceded by ASCII whitespace per CommonMark indentation rules:

  1. **Blockquote slot.** Zero or more `> ` blockquote prefixes (so `> `, `> > `, `> > > `, etc. all match).
  2. **List-item slot.** Zero or one list-item marker, drawn from:
     - an unordered-list marker `- `, `* `, or `+ `; or
     - an ordered-list marker `<digits>.` or `<digits>)`, in either case followed by a single ASCII space.
  3. **Construct-marker slot.** Zero or one construct-marker prefix, drawn from:
     - a table-cell leading `|` with optional surrounding ASCII whitespace (per CommonMark/GFM table syntax — i.e. an optional leading-pipe at column 0 plus the inter-cell pipes that separate cell content);
     - an ATX-heading `#{1,6} ` (one to six `#` characters followed by a single ASCII space);
     - a definition-list body marker `: ` (colon followed by a single ASCII space); or
     - a footnote body marker `[^<identifier>]: ` (a footnote label `[^<identifier>]` followed by colon and a single ASCII space).

  Slots are stripped in the order 1 → 2 → 3; no other ordering is recognised. A prefix from a later slot appearing before a prefix from an earlier slot on the same line, or two prefixes drawn from the same slot 2 or slot 3 on the same line (the slot-1 `> ` is the only prefix that may repeat, per its zero-or-more cardinality), is not stripped, so the dual-form pair is not the first non-prefix tokens on the line and the witness regex fails. The legal compositions therefore include `> - `, `> > - `, `> > > 1. `, and a blockquoted ATX heading `> ## `, while `- > ` and `## - ` are not recognised.

  This strip-list MUST stay co-extensive with the union of (a) the table-cell and ATX-heading rendering-driven contexts enumerated in *Required HTML-anchor contexts* above and (b) every construct enumerated in the body-paragraph operational definition there (which is what brings definition-list bodies, footnote bodies, list items, and blockquote bodies into scope). Adding a new context to either side of *Required HTML-anchor contexts* without extending this strip-list (or vice versa) is a spec defect: the witness regex would either reject every legal anchor in the new context or silently extend by implementer guess. In the three rendering-driven contexts (table cells, ATX headings, and the line preceding a fenced-code-block normative example), the mechanics of placing the dual-form anchor are governed by *Edge cases* below. Enforcement lives in the anchor-insertion CI gate (specified in the plan corpus); the plan corpus is the normative source for the gate's failure surface (exit code, per-offence message format, accumulation semantics, and output stream). *Transitional rule (retires once the backfill commit lands).* The MUST clause in *Required HTML-anchor contexts* above binds from the commit that lands the initial REQ-ID anchor pass (specified in the plan corpus) forward — that pass owns the one-shot backfill of pre-existing REQ-IDs without the HTML form across the corpus, in any of the four enumerated contexts (every REQ-ID anchor site under every live prefix in the [REQ-ID prefix table](#req-id-prefix-table) — including GOV-1 itself, and whether in body-paragraph, table-cell, ATX-heading, or line-preceding-a-fenced-code-block context — that lacks the HTML form today). Until that commit lands, the anchor-insertion CI gate treats pre-existing REQ-IDs without the HTML form (in any of the four contexts), and pre-existing bare `<a id="prefix-n"></a>` anchors whose inline `**PREFIX-N.**` marker lives on a different source line (as for GOV-14 and GOV-15 on this page today), as review-only backlog and MUST NOT fail the build on them; only newly added or edited REQ-ID anchor sites are gated. The backfill commit owns rewriting those legacy bare anchors into proper dual-form pairs. The backfill itself is GOV-8 *Pure rewording* — adding the dual-form HTML anchor next to an existing `**PREFIX-N.**` marker leaves the rule's normative sentences unchanged and so existing REQ-IDs survive en-masse rather than retiring under GOV-8 *Split* / *Deletion*; the modal-force shift introduced by this commit applies to the page-level rule (GOV-1 itself), not to any individual rule's `**PREFIX-N.**` paragraph.
- **Edge cases.** Inserting an anchor inside an ATX heading shifts the heading's auto-generated GitHub fragment ID; where every affected in-corpus cross-link can be repointed atomically with the anchor insertion, the anchor-insertion pass MUST repoint each such cross-link in the same edit. When repointing the heading-slug fragment is impractical in the same edit (e.g. the heading slug is referenced from outside the spec corpus or the cross-link cannot be repointed atomically with the anchor insertion), the entire dual-form pair (`<a id="prefix-n"></a> **PREFIX-N.**`) MUST instead be placed on the source line immediately preceding the heading, and the heading text MUST NOT carry the inline `**PREFIX-N.**` marker. In that case the anchor site classifies as body-paragraph context per the operational definition in *Required HTML-anchor contexts* above (the line preceding an ATX heading is body-paragraph by negation, since it is not itself inside the heading), so the *Dual-form layout* same-source-line MUST and witness regex apply normally with no ATX-heading-specific carve-out. Inside a table cell, the dual-form pair (`<a id="prefix-n"></a> **PREFIX-N.**`, in the order specified by *Dual-form layout* above) MUST be the cell's leading token sequence; where carrying the rule body in the cell would distort column widths, lift the entire rule out of the table — the cell then carries only a back-reference (e.g. `per PREFIX-N`) and the dual-form pair sits at the start of the rule's body-paragraph source line outside the table, per *Dual-form layout*. In the line-preceding-a-fenced-code-block context, the dual-form pair (`<a id="prefix-n"></a> **PREFIX-N.**`, in the order specified by *Dual-form layout* above) MUST appear on the source line immediately above the opening fence, with no intervening blank line, and that line MUST NOT carry any other content. The line-preceding-a-fenced-code-block *rendering-driven context* (and its per-context placement mechanics) is reserved for code blocks whose content is the rule's normative example; the line preceding a decorative or illustrative fence does not classify as that context and so does not invoke its per-context mechanics. Such a line is body-paragraph context per the operational definition in *Required HTML-anchor contexts* above, so a dual-form REQ-ID anchor on that line is still required if the line carries an inline `**PREFIX-N.**` marker — it is governed by the body-paragraph mechanics, not by the line-preceding-a-fenced-code-block context. Body-paragraph context carries no per-context edge case beyond the *Dual-form layout* same-source-line MUST: the dual-form pair (`<a id="prefix-n"></a> **PREFIX-N.**`) sits at the start of the body-paragraph source line that carries the inline marker, and any list-item / blockquote / nested-block indentation that precedes the pair is handled by the witness-regex anchoring rule rather than by a body-paragraph-specific carve-out here.

**GOV-2.** The plan's coverage matrix is keyed per REQ-ID, mapping each ID to its closing leaf. The coverage-matrix closing CI gate treats any unmapped REQ-ID as a CI failure. The plan corpus is the normative source for the gate's failure surface (exit code, per-offence message format, accumulation semantics, and output stream). The floor obligation carried by GOV-2 itself is that the gate exits non-zero on any violation; the rest of the surface is fixed by the plan leaf.

**GOV-3.** REQ-ID prefixes are byte-exact uppercase ASCII tokens of length 2–4 (`[A-Z]{2,4}`). Prefix matching is case-sensitive; `lex-1` does not match `LEX-1`. REQ-ID extraction operates on raw Markdown source bytes, not on rendered HTML. Before regex application, the following are stripped, in order: (i) fenced code blocks (` ```…``` ` and `~~~…~~~`, inclusive of fence lines), (ii) HTML comments (`<!--…-->`), (iii) inline code spans (`` `…` `` and the multi-backtick variants). Markdown link text is in scope; link targets are out of scope. The exclusion-stripped corpus is the input that the coverage-matrix closing CI gate consumes.

Two regexes apply:

1. **Primary extractor** (used by the anchor pass and by the appropriate coverage-matrix gates per the plan corpus): `\b(<live-prefix-alternation>)-[1-9][0-9]*\b`, where `<live-prefix-alternation>` is built from the prefix table below at gate time (never hard-coded). Leading zeros in the numeric tail are forbidden.
2. **Unknown-prefix detector** (used by the unknown-prefix gate only): `\b[A-Z]{2,4}-[1-9][0-9]*\b`, applied to the same exclusion-stripped corpus. Any token that matches but whose prefix is not in the live + retired union fails the gate. The detector regex and its membership check are tightened by [GOV-16](#gov-16) to also catch stable inline labels (the 2–6 character prefix range and the single-lowercase-letter tail form); see GOV-16 for the tightened regex and the widened membership union.

Pages whose row in the prefix table below carries the literal cell `(no IDs — narrative)` are excluded from extraction; all other rows in `spec_topics/*.md` are in scope. The exclusion cell's canonical byte sequence is `(no IDs — narrative)` — open paren, `no IDs`, ASCII space, U+2014 EM DASH, ASCII space, `narrative`, close paren — and cosmetic variants (smart quotes, trailing whitespace, en-dash, ASCII double-hyphen) are not recognised and silently change the page's exclusion status; the prefix-table parser used by the coverage-matrix gate MUST compare bytes.

IDs are immutable: when a rule is split, the original ID retires and two new IDs appear; numbering never collapses to fill holes.

**GOV-4 (per-row invariant).** The prefix → page binding is append-only and immutable: once a prefix is allocated to a page (per GOV-7 *Add* or GOV-7 *Narrative-to-normative promotion*), the binding never changes silently and the prefix is never reassigned to a different page (per GOV-7 *Delete* and GOV-7 *Normative-to-narrative demotion*, retired prefixes move to the *Retired prefixes* sub-table below and are never reused). The *table rows themselves* are NOT immutable — GOV-7 *Rename* updates the Page column without changing the prefix, and GOV-7 *Delete* / *Merge* / *Normative-to-narrative demotion* remove or relocate rows. Introducing a new non-narrative page requires appending a new row whose prefix is *previously-unused* — meaning absent from both this table and the *Retired prefixes* sub-table below. The prefix → page binding is many-to-one: a single page MAY own more than one prefix (e.g. `hard-ceilings.md` owns both `CEIL` and `CIO`), in which case each prefix sits on its own row and the per-row invariants apply to each row independently. Adding a second (or further) prefix to a page that already appears in the table follows the *Add prefix to existing page* sub-clause of GOV-7 below. The live table's `Prefix` column is a key — duplicate live `Prefix` values, and any live prefix that also appears in the *Retired prefixes* sub-table, are CI failures (enforced by the prefix-uniqueness gate specified in the plan corpus).

**GOV-5 (disjoint-prefix rule).** Each row's `Prefix` value is a complete identifier token, not a search prefix. Tooling that consumes REQ-IDs MUST anchor matches at a word boundary on both ends (`\b<PREFIX>-[0-9]+\b`); two prefixes that share a common substring (e.g. `BNDS` / `BNDR`) MUST NOT be treated as aliases or as one prefix-matching the other.

**GOV-6 (table-completeness invariant).** At every commit on `main`, the set of prefixes appearing in REQ-IDs across `spec_topics/*.md` is a subset of the union of (live prefix table, Retired prefixes sub-table). The coverage-matrix closing CI gate enforces this; the plan corpus is the normative source for the gate's failure surface, as for GOV-2.

**GOV-7 (mutation procedures).**

- **Add.** New page → append a row with a previously-unused prefix.
- **Add prefix to existing page.** Existing page acquires a second (or further) prefix → append an additional row, whose Page cell repeats the existing page's filename and whose Prefix cell is *previously-unused* on the same terms as *Add*. The pre-existing row(s) for that page stay as they are (per GOV-4's prefix → page binding stability — appending a new prefix row for an existing page is not one of the GOV-7 mutations *Rename* / *Delete* / *Merge* / *Normative-to-narrative demotion* that GOV-4 enumerates as the exhaustive set of operations permitted to modify or relocate existing rows, so the existing rows' Prefix and Page columns are untouched).
- **Rename.** Prefix follows the page; the row's Page column updates, the Prefix column does not. Existing in-page anchors are not rewritten. In the same commit, update every reference to the old filename across `plan.md` and `plan_topics/**.md`; the plan-link CI gate (specified in the plan corpus) enforces this.
- **Delete.** The row is moved from the live table to the Retired prefixes sub-table. The prefix MUST NOT be reused.
- **Merge.** When `<absorbed-page>` is merged into `<surviving-page>`:
    - Every live REQ-ID on the absorbed page undergoes a GOV-8 *Merge* into a freshly allocated ID under the surviving prefix, appended at the surviving page's tail. The absorbed-page IDs retire on the absorbed page (its `## Retired REQ-IDs` section gains one row per merged ID) before the file is deleted.
    - The absorbed page's pre-existing `## Retired REQ-IDs` rows transplant verbatim into the surviving page's `## Retired REQ-IDs` section.
    - The absorbed prefix moves to the *Retired prefixes* sub-table; the *Formerly* cell records `<absorbed-page> (merged into <surviving-page> at <sha>)`.
    - The absorbed page file is deleted in the same commit (otherwise the prefix-table-completeness gate sees both a retired prefix and a live page carrying it).
- **Narrative-to-normative promotion.** Replace the `(no IDs — narrative)` cell with a freshly allocated prefix in the same edit that introduces the first obligation.
- **Normative-to-narrative demotion.** When a page that previously owned a prefix becomes pure narrative (every claim it carried is canonically owned elsewhere): (a) move its prefix from the live table to the *Retired prefixes* sub-table per GOV-7 *Delete*; (b) append a new live table row for the page carrying `(no IDs — narrative)`. Re-promotion later (per *Narrative-to-normative promotion* above) requires a *fresh* prefix — the original is retired and immutable per GOV-4 / GOV-5, and the *Retired prefixes* sub-table is itself append-only.

**GOV-8 (REQ-ID lifecycle).**

*Scope note (non-normative).* GOV-8 is a *bookkeeping* discipline: it pins how a substantive edit is recorded, not whether two V1.x releases produce the same output for the same `.loom` / `.warp` source file. The behavioural V1.x stability statement (per **GOV-15** at the page tail) is intentionally *not* derived from this rule.

<a id="gov-14"></a> **GOV-14 (review posture on the V1.x-equivalence release-process goal).** Reviews of `spec.md` and `spec_topics/*.md` SHOULD NOT cite GOV-8 as a substitute for the V1.x source-language equivalence release-process goal recorded at [GOV-15](#gov-15) and SHOULD NOT re-raise the absence of an automated equivalence gate as a V1.0 correctness finding. The V1.0 release decision treats the absence of an automated equivalence gate as a recorded scope choice, not a defect.

- **Split.** When one rule splits into N rules, the original ID retires and N fresh IDs are appended at the page's tail.
- **Merge.** When N rules merge into one, all N source IDs retire and one fresh ID is appended at the page's tail.
- **Deletion.** Rule removed without replacement → ID retires; the prefix-position number MUST NOT be reused.
- **Pure rewording.** Typo fixes, sentence restructuring, link updates leave the ID unchanged. A change that alters which inputs are accepted, which outputs are produced, which diagnostics fire, or which invariants hold is substantive and MUST be modelled as a split, merge, or deletion-plus-add — never as an in-place edit. This boundary is enforced by review on the PR that touches a `**PREFIX-N.**` paragraph; no CI gate detects substantive in-place edits, and none is planned, because semantic equivalence between two prose paragraphs is not mechanically decidable. Worked examples:
  - Modal strengthening or weakening (`should` → `MUST`, `MUST` → `should`, `MAY` → `MUST`, etc.) inside a `**PREFIX-N.**` paragraph is **substantive** — the obligation's force has shifted even when the inputs and outputs at the moment of the edit appear unchanged. Treat as a split-or-deletion-plus-add. (This is the most commonly missed case in review; flag it first.)
  - Adding or removing a diagnostic code on a `**PREFIX-N.**` paragraph is **substantive** — it alters which diagnostics fire, per the test above. Treat as a split-or-deletion-plus-add. The new code's diagnostic-registry entry (asserted by the diagnostic-code closing CI gate per the plan corpus) is a separate, additive obligation and does NOT discharge the GOV-8 retirement requirement.
  - Editing a non-normative example (a fenced code block, parenthetical illustration, or worked sample) attached to a `**PREFIX-N.**` paragraph is **pure rewording**, provided the rule's normative sentences are unchanged. Examples do not constrain implementations. When in doubt about whether a sentence is normative or illustrative, retire-and-re-add.

All retirements (per GOV-7 *Delete* / *Merge* and per GOV-8 *Split* / *Merge* / *Deletion*) MUST be recorded:

- **Per-prefix retirements** appear in the *Retired prefixes* sub-table immediately below.
- **Per-ID retirements** appear in a trailing `## Retired REQ-IDs` section on each non-narrative page (skeleton inserted by the initial REQ-ID anchor pass specified in the plan corpus). After a GOV-7 *Merge*, the surviving page's `## Retired REQ-IDs` section accumulates rows under both the surviving and the absorbed prefixes (per the *Merge* sub-bullet above); the absorbed page's section is deleted with its file.

**GOV-9 (cross-link form).** Each spec page that depends on a normative rule from another topic page MUST either state the rule locally or reference it by a markdown link whose target is the specific REQ-ID anchor (`#prefix-n`) of the depended-upon rule. Where the depended-upon page is pure-narrative (its row in the [REQ-ID prefix table](#req-id-prefix-table) carries the `(no IDs — narrative)` cell per GOV-3), a section-level link to the relevant heading on that page suffices. From the commit that lands the initial REQ-ID anchor pass (specified in the plan corpus) forward, and independently of whether any such cross-link currently exists, every REQ-ID anchor on a non-narrative spec page MUST resolve as a `#prefix-n` URL fragment, so that any cross-page link of the form `…#prefix-n` formed per the preceding sentence resolves to the depended-upon rule's anchor site; this universal-resolvability obligation is the contract that GOV-1 *Required HTML-anchor contexts* and *Dual-form layout* deliver against, and is gated by the same anchor-insertion CI gate under the same transitional rule (per GOV-1 *Dual-form layout*).

**GOV-10 (plan-leaf reading scope).** An implementer MAY restrict their reading to the topic pages listed under their plan leaf's `**Spec**` field, where a *plan leaf* is a terminal task in [`plan.md`](../plan.md) (leaf format defined in [`plan_topics/conventions.md`](../plan_topics/conventions.md#leaf-format)) and its `**Spec**` field is the list of `spec_topics/*.md` filenames the leaf implements.

**GOV-11 (Spec-field closure).** The plan leaf's `**Spec**` field MUST be closed under normative cross-link. Closure runs from each listed topic outbound: if a listed topic `T` cross-links a normative rule in topic `T'`, then `T'` MUST also appear in the field. Closure applies transitively to a fixed point — iterate adding pages until the field stops growing. Narrative cross-links — those whose row in the [REQ-ID prefix table](#req-id-prefix-table) carries the `(no IDs — narrative)` cell, which per GOV-3 is the single source of truth for narrative-page status — do not trigger the closure obligation.

The *single-source-page* shape enumerates a closed set of items all sourced (for aggregator lock-step purposes) from a single sub-bucket on a single topic page — the *aggregator-source page*, which is distinct from the per-item *normative owners* of the underlying dispositions and may differ from them (the per-item normative owners may live on other topic pages, cited in the sub-bucket's per-item `Recorded at:` lines or equivalent) (currently: the V1 non-goals aggregator paragraph on `spec.md`, whose aggregator-source page is [`future-considerations.md`](./future-considerations.md#v1-non-goals) — it enumerates the no-seam dispositions by name and forward-links the bullet list there, while the per-bullet normative owners cited in each `Recorded at:` line live on their respective topic pages; only that one sub-bucket on `future-considerations.md` is in lock-step scope — sibling sub-buckets (Tooling deferrals, Surface extensions, Model-level changes) on the same page are not; a *navigation-pointer cross-link* — defined as a cross-link whose role is wayfinding (pointing the reader at related material) rather than item enumeration; it does not name an item that is under this rule's lock-step scope — from the aggregator into one of those out-of-scope sibling sub-buckets (or to any other anchor outside the aggregator-source sub-bucket) carries no anchor-stability obligation under this rule, so a future PR that renames such a sibling sub-bucket's heading silently breaks the cross-link without a normative trigger — this is a documentation defect, not a normative defect, and the rename-PR author's responsibility (i.e. the PR that renames the heading SHOULD repoint each affected aggregator-side cross-link in the same commit, caught by reviewer / grep, not by a CI gate)). The lock-step parity criterion for the *single-source-page* shape is *bidirectional referent identity*: the aggregator's enumeration MUST be a 1:1 referent-identity mapping with the aggregator-source sub-bucket — every bullet in the source sub-bucket has exactly one item in the aggregator that refers to the same disposition, and every item in the aggregator has exactly one bullet in the source sub-bucket that refers to the same disposition. Aggregator-side stylistic rewording (prose phrasing, bullet-vs-inline rendering, ordering) is permitted if it preserves that 1:1 referent mapping — bullet-name string identity, item-count rendering parity, and content parity beyond the referent are not required (this mirrors the GOV-8 *Pure rewording* limit cited in the undecidability disclaimer below, and shares its consequence: whether a given paraphrase preserves the referent is a reviewer judgement, not a mechanical check). The bidirectional shape makes this parity criterion mutually reinforcing with the add/remove/rename lock-step trigger below: a PR that adds, removes, or renames a source-side bullet without the matching aggregator-side update breaks the 1:1 mapping and so violates this criterion in addition to the lock-step trigger. Aggregator paragraphs are maintained in lock-step with their source pages by editorial convention: a PR that adds, removes, or renames an item on a source page (e.g. introduces a fifth hard ceiling on a topic page, retires an SDK capability, splits a Host runtime obligation, adds a sixth `.warp` permitted form, adds, removes, or renames a topic-page obligation enumerated in the Session model paragraph's forward-link inventory above, or adds, removes, or renames a bullet under the V1 non-goals list on `future-considerations.md`) MUST update the corresponding `spec.md` aggregator in the same commit. Drift between an aggregator and its sources is a documentation defect, not a correctness defect; reviewers SHOULD flag it on the PR that introduced the drift, but no CI gate detects it (semantic equivalence between an aggregator paragraph and a set of topic-page paragraphs is not mechanically decidable, mirroring the GOV-8 *Pure rewording* limit). Aggregators of the *integer-count* shape (paragraphs and, in the single-bullet case noted within the enumeration below, a single Scope bullet — taking the shape of an integer count of source items; currently: the 13 typed/structural seams (the lone *single-bullet* aggregator in this enumeration — its host is the *Forward-compatibility seams* Scope bullet under [`spec.md` — Scope](../spec.md#scope), not an aggregator paragraph; the host bullet is simultaneously one of the source items counted by the *five Scope bullets* aggregator listed below in this enumeration — the dual role poses no integer-count-gate ambiguity because the gate operates per aggregator on its own source set; this entry is therefore declared as a GOV-12 aggregator by its membership in this integer-count `currently:` enumeration alone and is intentionally not registered under either of the *multi-source-page* or *single-source-page* shapes enumerated earlier in this paragraph), the four hard runtime ceilings, the five `.warp` permitted top-level forms, the seven V1 non-goals, the five Scope bullets, and the three Host runtime obligations) are an exception to the undecidability disclaimer above — specifically, they override only its "no CI gate detects it" clause (a literal-vs-count drift on an in-scope aggregator IS detected by CI), while the documentation-defect characterisation and the reviewer-SHOULD-flag obligation continue to apply on top of the CI gate (reviewers SHOULD flag drift independently of whether the gate fires; the gate is a backstop, not a substitute for review, so a reviewer who fails to flag a drift the gate also caught has not thereby satisfied the SHOULD): counting items in a fixed source set against a literal integer is mechanically decidable, and the coverage-matrix closing CI gate enforces equality between the literal and the actual source-item count (subtracting any explicitly excluded items per the gate's source table, with each exclusion citing the topic-page paragraph that justifies the exclusion); the plan corpus is the normative source for the gate's failure surface, as for GOV-2. The floor obligation carried by GOV-12 itself for this sub-rule is that the gate exits non-zero on any equality-predicate violation OR any source-table well-formedness violation; the rest of the surface is fixed by the plan leaf. For aggregators of the *integer-count* shape, this gate is the mechanical witness for the *add* and *remove* arms of the lock-step MUST stated earlier in this paragraph: a source-side add or remove without the matching aggregator-side literal update fires the gate, and remediation is either updating the literal or, if the source-side item is genuinely outside the gate's scope for stated reasons, adding a source-table exclusion entry citing the topic-page paragraph that justifies the exclusion (per the citation requirement on exclusions above); both are valid resolutions and the choice is a contributor judgement constrained by that citation requirement. Pure renames do not change the source-item count and so are not mechanically witnessed by this gate (the equality predicate `literal == |source| − |exclusions|` evaluates identically before and after a pure rename, and the matching aggregator-side update for a rename is itself a non-literal change outside the gate's witness surface); the *rename* arm of the lock-step MUST remains in force on the same editorial-convention basis as for non-integer-count aggregators, and the reviewer-SHOULD-flag obligation under the undecidability disclaimer above continues to apply to rename drift on integer-count aggregators with no mechanical backstop. Boundary cases (zero literal against an empty source set, dangling exclusion naming an item not present in the source set, duplicate exclusion where the same item is named two or more times in the exclusion list, causing it to be subtracted more than once, and negative residual where the exclusion count exceeds the source-set cardinality, yielding `|source| − |exclusions| < 0`) are *source-table well-formedness* concerns rather than equality-predicate violations — most notably the zero-literal-against-empty-source-set case trivially satisfies the equality predicate (`0 == 0`) and so is not by itself an equality-predicate violation, and is further resolved inline here as a well-formedness pass as well, yielding an overall gate-pass disposition that is NOT delegated to the plan-corpus arm. The per-case classification (gate failure vs gate pass) of the other three boundary cases (dangling exclusion, duplicate exclusion, and negative residual) is pinned in the plan-corpus arm on the same delegation as for GOV-2; the floor obligation above (gate exits non-zero on any equality-predicate violation OR any source-table well-formedness violation) guarantees that any plan-side classification of `violation` for one of these cases exits non-zero, while leaving the per-case failure-vs-pass classification itself to the plan corpus. *Source-table-coverage transitional rule (retires once the source-table population covers every integer-count aggregator currently enumerated above).* The gate's effective coverage of those listed aggregators is conditional on the plan-side leaf landing the source-table population: until that population covers each one, the gate MUST report any uncovered listed aggregator as itself a gate failure on the same exit-non-zero floor (uncovered ≡ gate failure on that aggregator), so that an empty or partial source table at any commit that merges while the source-table population remains incomplete cannot vacuously satisfy this sub-rule. *Uncovered* is defined narrowly for this rule: an aggregator is *uncovered* iff the source table has no row at all for that aggregator. An aggregator that has a source-table row whose source set is structurally well-formed but empty is *covered* under this transitional rule and falls into the boundary-cases mechanism above (where its failure-vs-pass disposition is pinned in the plan-corpus arm — except for the zero-literal-against-empty-source-set case itself, which is resolved inline above as a trivial equality-predicate pass), not into the uncovered classification here; the two mechanisms are therefore disjoint on the same source-table row, and no precedence rule between them is needed. As a narrow exception to the aggregator-side rewording permission stated earlier in this paragraph (which would otherwise let a PR rephrase the literal away and silently exit the gate's scope), each aggregator of the *integer-count* shape MUST preserve a parseable integer literal — in some parseable form (e.g. decimal-digit `13` or English-number-word `thirteen`) — anywhere within the same aggregator paragraph (or, for single-bullet aggregators, the same bullet); the literal's sentence position, list rendering, and ordering relative to other prose remain reworkable under the rewording permission above, but the literal MUST survive in parseable form somewhere in that aggregator's text. Paraphrasing the literal away entirely (e.g. rewording "the five Scope bullets" to "the Scope bullets" with no surviving parseable integer literal anywhere in the aggregator) is itself a gate failure detected by the gate's source table on the same exit-non-zero floor as a literal-vs-count mismatch. The precise lexical locator the gate uses to find the surviving literal within the aggregator is a plan-corpus choice; the floor here is solely that some parseable literal survives somewhere in the aggregator's text. Aggregators whose count is already mechanically enforced by a leaf-owned in-code constant (currently the SDK-capabilities aggregator alone, enforced by [`CAPABILITY_OBLIGATIONS.length === 7`](./pi-integration-contract.md#bump-step-2-positive) against the leaf-owned constant per [PIC — SDK capability inventory](./pi-integration-contract.md#sdk-capability-inventory) — the markdown enumeration of items 1–7 there and the [`spec.md` Prerequisites](../spec.md#prerequisites) cross-reference list are maintained by editorial convention per that anchor, not inputs to the gate above; additional aggregators may be added to this in-code-constant carve-out class only via a substantive GOV-12 amendment that names the aggregator and its in-code-constant witness, on the same `currently:` lock-step convention used by the shape enumerations above — a leaf that introduces a new `length === N`-style check elsewhere does NOT thereby remove its aggregator from the integer-count gate's scope until such an amendment lands) are out of scope of the coverage-matrix closing CI gate above; the in-code-constant assertion is the sole mechanical enforcement of those counts. The orientation prose literal that names the count on `spec.md` (e.g. "seven SDK capabilities") is likewise out of scope of this rule's literal-preservation MUST stated earlier in this paragraph (which binds only aggregators of the *integer-count* shape, not in-code-constant carve-out aggregators) and remains under editorial convention on the same terms as the lock-step MUST and reviewer-SHOULD-flag obligation under the undecidability disclaimer above — the in-code-constant assertion mechanically witnesses only the count's truth, not the orientation literal's presence on `spec.md`, so a PR that paraphrases the literal away (e.g. rewording "seven SDK capabilities" to "the SDK capabilities") is a documentation defect that reviewers SHOULD flag but no CI gate detects. This orientation-vs-constant drift gap is intentional under this rule. Separately, the Session model paragraph's forward-link inventory (registered above under the *multi-source-page* shape) is also out of scope of the integer-count gate, on different grounds: it is an open-ended forward-link inventory of topic-page obligations rather than a counted set, and so does not surface a single-integer cardinality in any lexical slot for the gate to witness; its lock-step MUST stated earlier in this paragraph remains reviewer-enforced under the undecidability disclaimer above, with no mechanical witness. This rule is the single canonical home for the lock-step convention; previous load-bearing MUSTs scattered across `spec.md` aggregator paragraphs (most notably the hard-runtime-ceilings migration MUST) are removed in favour of citing GOV-12.

<a id="gov-15"></a> **GOV-15 (V1.x source-language equivalence — release-process goal).** A `.loom` or `.warp` file that loads cleanly under V1.0 is expected to load under every V1.x release and to produce, for any given input, identical (a) return values, (b) ordered diagnostic-code sequences, and (c) `loom-system-note` content strings. Wall-clock timing, token counts, and log-line volume are explicitly excluded from this expectation. This rule states a release-process goal, not a verifiable obligation: V1.0 ships without an automated equivalence gate; equivalence between two V1.x releases is a release-process responsibility verified by reviewer inspection of the diff against the prior V1.x release. <a id="gov-15-fixture-suite"></a> A conformance fixture suite that mechanically diffs the three observables above against a frozen V1.0 baseline is a recognised post-V1.0 follow-up; see [Future Considerations](./future-considerations.md). The companion review-posture rule that bars reviewers from re-raising the missing gate as a V1.0 correctness finding is [GOV-14](#gov-14). The (a)–(c) observable list and the wall-clock / token-count / log-line-volume exclusion list are both closed at V1.0: any other observable not named in either list is, by default, neither in scope of the equivalence promise nor of the explicit-exclusion list, and adding a further observable to either list is a substantive edit to GOV-15 governed by the present rule's edit conventions, not by GOV-8's *Pure rewording* arm. When the reviewer-inspection step encounters such a third-bucket observable diverging between releases, it raises the divergence as a documentation defect against GOV-15 (to add the observable to one of the two lists in a follow-up release) but does NOT block the current release. References elsewhere in the corpus to "four ceilings", "four-item list", "four NOCEIL-1 … NOCEIL-4 claims", "four-axis closure list", and any other count-bearing phrase naming the V1.0 starting set are bound by GOV-12 lock-step: any operation that changes the count those phrases name (e.g. *Introduce*, append-trailing *Retire*, retirement form (ii) coining a new `NOCEIL-N`, or *new-axis case* introduction extending the closure axis count) also updates the affected count-bearing phrases in the same commit.

<a id="ceiling-set-carve-out"></a>

*Ceiling-set carve-out.* Introducing a new hard runtime ceiling within V1.x — or retiring or relaxing an existing one — is a recognised carve-out to this equivalence. The carve-out's in-scope input set is defined *post-hoc* over the diff between the two releases under comparison and is a property of that inter-release diff, not a property the runtime can statically derive at single-release evaluation time: an input is *in scope* of the equivalence promise iff, when run under both releases, it either (i) trips no hard runtime ceiling under either release, or (ii) trips exactly the same hard runtime ceiling (identified by its bullet ordinal in the four-item list at [`spec.md` — Hard ceilings](../spec.md#hard-runtime-ceilings), persistent across the inter-release diff) under both releases. An input that trips a ceiling absent from one of the releases, or that trips a ceiling under one release but succeeds under the other (whether via the ceiling being introduced, retired, relaxed, or never having been ceilinged on the other side), is *outside* the in-scope set and the carve-out applies to it. "Trips" here means *surfaces* — the single ceiling whose firing reaches loom code under the CIO-1 … CIO-6 evaluation order; co-fires recorded under `masked` are not separately tested. A release that reorders surfaced ceilings (via a CIO-ordering or `masked`-domain change) without performing a ceiling-set change does NOT trigger the carve-out — the carve-out is keyed to ceiling-set changes only. Such surfacing-reordering edits remain bound by the equivalence promise on observables (a)/(b)/(c) and any divergence on inputs that previously surfaced ceiling X but now surface ceiling Y is a GOV-15 release-process violation; CIO-ordering and `masked`-domain edits that change observable surfacing therefore require V2 migration unless they preserve every input's surfaced ceiling identity. For inputs outside the in-scope set defined above, observable (a) is by construction permitted to differ between releases — the success/fail outcome is the very change the ceiling-set change produces — and the carve-out also covers this (a) drift; only on the in-scope set is the equivalence promise unaltered for (a), subject to the routing-class-demotion exception below. For inputs that remain in-scope but whose routing-class is demoted under *Relax* (e.g. panic→recoverable-`Err`), observable (a) is permitted to differ between releases — the routing-class change's own (a) impact (panic-shape → `Err`-value-shape, or analogous demotion) is carve-out-covered for those inputs; only the changed-ceiling's own discriminator and message text are bound by the *Attribution test*. The [deferred conformance-fixture suite](#gov-15-fixture-suite) is the mechanism that verifies the carve-out post-hoc against the inter-release diff. The carve-out covers only the changed ceiling's own diagnostic emissions — i.e. observables (b) and (c) of GOV-15's first paragraph as they pertain to the changed ceiling — enumerated parallel to the three carve-out-eligible ceiling-set changes: for an *introduced* ceiling, the new discriminator and the canonical message it adds; for a *retired* ceiling, the removed discriminator and the canonical message it removes; for a *relaxed* ceiling, the existing discriminator (which continues to apply but fires under different conditions) and any canonical-message text change that re-states the new bound (e.g. `exceeded 50` replacing `exceeded 25`) or otherwise reflects the relaxed predicate; plus, in all three cases, any `loom-system-note` content directly attributable to the ceiling-set change (per the *Attribution test* below). On in-scope inputs, an unrelated discriminator whose multiplicity (number of emissions per query) changes solely as a behavioural consequence of the ceiling-set change — e.g. a per-round diagnostic that fires more often after a loop bound is relaxed — is treated as carve-out-covered drift, not as a GOV-15(b) violation; the canonical-message content of the *unrelated* discriminator must remain unchanged on each emission; the changed ceiling's own diagnostic content is governed by the *Attribution test* below. Collateral diagnostic-code edits or `loom-system-note` content edits landed in the same release on unrelated surfaces (per the same *Attribution test*) are forbidden in a ceiling-set release per the *Attribution test* below; they must be split into a separate V1.x release that lands no ceiling-set change. When a single V1.x release performs more than one ceiling-set change (e.g. introduces one ceiling and retires another, or introduces two ceilings at once), the carve-out applies as the union of the per-change carve-outs — each ceiling-set change is individually scoped per the *Attribution test* below, and an input that falls outside the in-scope set under any one of the constituent changes is outside the in-scope set for the release as a whole. An edit jointly attributable to two or more constituent ceiling-set changes is in-scope for the release; an edit attributable to at least one constituent change is in-scope, even if it is collateral with respect to another constituent change. The *Collateral* set is the residue. The orientation aggregator at [`spec.md` — Hard ceilings](../spec.md#hard-runtime-ceilings) names this carve-out in lock-step with this rule per [GOV-12](#gov-12). The GOV-15 reviewer-inspection step verifies the [ceiling-set invariants](./hard-ceilings.md#ceiling-set-invariants) per the canonical clause at [Hard Runtime Ceilings — Release gate](./hard-ceilings.md#release-gate).

<a id="attribution-test"></a>

*Attribution test.* The carve-out's two phrases — *directly attributable to the ceiling-set change* and *collateral … on unrelated surfaces* — are pinned by the following syntactic predicate, mechanically applicable on the release diff. A wording change to a diagnostic code or to `loom-system-note` content is *directly attributable to the ceiling-set change* iff at least one of the following holds: (i) its discriminator (`details.kind` for diagnostic-channel ceilings, the `Err`-variant `kind` for `QueryError` ceilings, or the discriminator named in ceiling #4's per-boundary table at [`hard-ceilings.md` — Per-boundary destination/surface table (ceiling #4)](./hard-ceilings.md#ceiling-4-table)) equals one of the changed ceiling's own discriminators (plural where the changed ceiling cross-routes — e.g. ceiling #4's `loom/runtime/*` and `loom/load/*` arms both count); (ii) its canonical message text contains either the ceiling's ordinal identifier (e.g. `ceiling #N`) or the literal numeric bound being introduced, retired, or relaxed; (iii) the change is the minimal edit required to keep a message-template row, header line, or `details.kind`/discriminator enumeration shared with another ceiling well-formed under the changed ceiling (the *shared-template* case — including a discriminator rename that subsumes both the changed ceiling's old discriminator and an unrelated runtime discriminator, where the rename's normative widening is the minimal edit required and any further wording change to the unrelated discriminator's surface is not the minimal edit required and falls into the *Collateral* set below); or (iv) for a routing-class demotion or enforcement-point narrowing with no numeric bound, the canonical message text contains an explicit reference to the routing-class or enforcement-point surface change (e.g. `no longer fatal`, `no longer enforced at <site>`) minimal to reflect the relaxed predicate. Each ceiling's owner page is the source of truth for that ceiling's discriminator inventory. *Collateral* is any wording change for which none of the four sufficient conditions (i)–(iv) above holds (i.e. for which *Directly attributable* is false). A release that lands collateral diagnostic-code or `loom-system-note` content edits inside the same release as a ceiling-set change is flagged by this rule's reviewer-inspection step (per the reviewer-inspection step in GOV-15's main paragraph above) as a GOV-15 release-process violation, blocking the release until the collateral edits are split into a separate V1.x release that lands no ceiling-set change.

<a id="operational-definitions"></a>

*Operational definitions.* The four ceiling-set verbs name distinct operations. *Introduce* adds the new ceiling's ordinal-bearing entry to the four-item list at [`spec.md` — Hard ceilings](../spec.md#hard-runtime-ceilings) (extending it to a five-item list), creates the per-ceiling owner-page section on the appropriate topic page (defaulting to a new sub-section on [`hard-ceilings.md`](./hard-ceilings.md) if no other topic page is the natural owner), and adds a per-axis entry under [Hard Runtime Ceilings — Audit methodology — Four-axis closure list](./hard-ceilings.md#four-axis-closure-list), all in the same commit per GOV-12; *Introduce* is a normative addition not modelled by any GOV-8 lifecycle case (which addresses only edits to existing rules), and is governed directly by the present rule and by the audit-methodology paragraph at [Hard Runtime Ceilings — Audit methodology](./hard-ceilings.md#audit-methodology). *Retire* — remove a ceiling; the retired ceiling's owner-page entry is annotated in place with a pointer to the retirement-release diff, with retirement-table recording deferred to the eventual REQ-ID anchor pass. The pointer takes the same SHA-or-tag form as the `Retired in` column of the [*Retired prefixes*](#retired-prefixes) sub-table above — either the 7-character abbreviated commit SHA of the retirement commit or the release tag of the V1.x release that performed the retirement, nothing else. For inputs that previously tripped the retired ceiling, the retired ceiling's owner-page annotation MUST document the post-retirement runtime trajectory (succeed; fall through to a documented `NOCEIL-N` surface; fall through to another ceiling; or non-terminating absent cancellation); silent unbounded behaviour is forbidden. Cardinality after retirement: the retired ceiling's bullet is left as an in-place retirement-annotated stub (preserving the persistent ordinals the carve-out's in-scope predicate relies on); the bullet MUST NOT be removed outright while any later bullet exists, since outright removal would shift subsequent ordinals and break the in-scope predicate's identity test. Bullets MAY be removed outright only when retired in append-trailing position (no later bullet exists), in which case the editor MUST also update every count-bearing reference (per GOV-12 lock-step) in the same commit. *Relax* — widen the input set the ceiling permits (raise a numeric bound, narrow the enforcement-point surface, or demote routing class — e.g. `panic` to recoverable `Err`). *Tighten* — the inverse of relax (narrow the input set the ceiling permits — lower a numeric bound, broaden the enforcement-point surface, or promote routing class) — is explicitly NOT a recognised carve-out under this rule, because it can break the equivalence promise on inputs that previously succeeded; it is permitted within V1.x only when no input in the V1.x conformance fixture suite (once it ships) is made to fail by the change. Until that suite ships, *Tighten* is forbidden under V1.x and any required tightening is deferred to V2 migration per [Future Considerations — V1 non-goals](./future-considerations.md#v1-non-goals); a tightening that would fail any input in the eventually-shipped suite remains likewise V2-only and MUST NOT land within V1.x. Once *Tighten* becomes permitted (per the conformance-fixture-suite condition above), it falls under the same GOV-12 lock-step co-edit and *Audit methodology* release-gate obligations as *Relax*: once *Tighten* is unlocked, any spec-side enumeration that lists "introduce, retire, or relax" is read as covering *Tighten* in the same shape as *Relax* — but the inheritance is procedural only (lock-step + per-axis-entry update + release gate), not equivalence: *Tighten*-affected inputs remain bound by the GOV-15 main-paragraph equivalence promise without any *Tighten*-specific carve-out (numeric-bound, enforcement-point surface, or routing-class edit to an existing ceiling without retiring its identity). The Relax/Tighten sub-case enumeration above (numeric-bound, enforcement-point surface, routing-class) is closed at V1.0; a fourth widening or narrowing shape is forbidden in-place and requires V2 migration. Both *Relax* and *Tighten* are substantive ceiling-page edits in GOV-8 terms; they are licensed for in-place edit (rather than the GOV-8 retire-and-readd path) only by the present rule's *Ceiling-set carve-out* (for *Relax*) or by the present rule's *Tighten* permission clause (for *Tighten*). Either operation is recorded by amending the ceiling owner-page entry in place; GOV-8's split-or-deletion-plus-add path is explicitly not required for either operation. V1 specifies no upgrade-time warning surface for retirement-induced dead `match` arms in loom code (e.g. a `match` arm on `Err(QueryError { kind: "tool_loop_exhausted", … })` that becomes statically unreachable if `tool_loop.max_rounds` were ever retired); such drift is a known consequence of *Retire* and is deferred to a future tooling surface (no V1 tooling specification exists; whether such a warning surface is added in a future V1.x release is out of scope here). Symmetrically, a routing-class demotion under *Relax* (e.g. `panic` → recoverable `Err`) makes a previously-unreachable `match` arm reachable on previously-panicking inputs; V1 specifies no upgrade-time warning surface for the new-arm reachability case either.

## Stable inline labels

<a id="gov-16"></a> **GOV-16 (stable inline labels — second governed identifier class).** Some normative obligations carry a *stable inline label* — a cross-page-citable identifier that is not a REQ-ID. Inline labels are a second governed identifier class with their own per-page registry, anchor form, lifecycle, and retirement table; the rule is structured to mirror GOV-1 / GOV-3 / GOV-8 / GOV-9 so that the two classes are gated by parallel machinery. Two inline-label prefixes are currently registered, both on [`hard-ceilings.md`](./hard-ceilings.md): `HC3` (the per-class binder retry sub-obligations `HC3-a` … `HC3-e`) and `NOCEIL` (the four enumerated non-existence claims `NOCEIL-1` … `NOCEIL-4`). The `CIO-1` … `CIO-6` labels on the same page are *not* inline labels under this rule — they are full REQ-IDs under the `CIO` row of the [REQ-ID prefix table](#req-id-prefix-table) (co-owned with `CEIL` per GOV-4) and are governed by GOV-1 / GOV-3 / GOV-8 / GOV-9 directly.

- **Canonical form.** Each inline label's defining anchor is the inline `**PREFIX-<tail>.**` marker, where `<tail>` is either `[1-9][0-9]*` (numeric tail, e.g. `NOCEIL-3`) or a single lowercase ASCII letter `[a-z]` (alphabetic tail, e.g. `HC3-a`). The bold-with-period decoration is normative on the same terms as for REQ-IDs (per GOV-1 *Canonical form*) — it distinguishes anchor sites from back-references and is the witness pattern for the inline-label arms of the anchor-insertion gate, the Reused-ID gate, and the Dense-numbering gate (all specified in the plan corpus).
- **GOV-1 dual-form layout applies.** Inline-label anchor sites participate in GOV-1's *Required HTML-anchor contexts* and *Dual-form layout* obligations on identical terms — the witness regex `<a id="prefix-n"></a> \*\*PREFIX-N\.\*\*` is read as `<a id="prefix-tail"></a> \*\*PREFIX-<tail>\.\*\*` with `<tail>` per the *Canonical form* above and rendered lowercase in the `id` attribute to match GitHub's slug convention. The transitional rule of GOV-1 *Dual-form layout* applies to existing inline-label anchors on the same terms — the initial REQ-ID anchor pass (specified in the plan corpus) owns the one-shot backfill of the dual-form HTML anchor at every existing inline-label anchor site.
- **Per-page inline-label prefix table.** Inline-label prefixes are registered per page below; the table is append-only and follows GOV-4 / GOV-5 / GOV-7 on the same terms as the REQ-ID prefix table — live inline-label prefixes are disjoint from each other and from the live + retired union of REQ-ID prefixes, and a retired inline-label prefix is never reused. Inline-label prefixes are uppercase ASCII letters of length 2–6, optionally with ASCII digits in non-leading positions (e.g. `HC3`, `XY7`, `NOCEIL`) — matched by `[A-Z][A-Z0-9]{1,5}`; the upper bound widens beyond REQ-IDs' 4 to admit `NOCEIL`, and admitting digits in non-leading positions is a deliberate divergence from GOV-3's letters-only REQ-ID prefix grammar (`[A-Z]{2,4}`) so that digit-bearing prefixes such as `HC3` can be registered. Matching is case-sensitive per GOV-3.

  | Page | Inline-label prefix | Tail form |
  |---|---|---|
  | `hard-ceilings.md` | `HC3` | `alphabetic` |
  | `hard-ceilings.md` | `NOCEIL` | `numeric` |

  The `Tail form` column pins each inline-label prefix to exactly one tail shape — `numeric` (matching `[1-9][0-9]*`) or `alphabetic` (matching `[a-z]`) — drawn from the two tail forms enumerated in *Canonical form* above. The column is normative: it is consulted by the *Third extractor* and the *Tightened unknown-prefix detector* below to bind each prefix to one tail form, and is part of the GOV-4 / GOV-7 append-only table contract on the same terms as the `Page` and `Inline-label prefix` columns. A row's `Tail form` cell is fixed at the prefix's allocation moment and is not silently re-pointed; switching a live inline-label prefix from one tail form to the other is a GOV-7 *Delete* of the original prefix plus a GOV-7 *Add* of a fresh prefix under the new tail form.

  *Retired inline-label prefixes.* Empty in V1; the sub-table is append-only on the same terms as the *Retired prefixes* sub-table for REQ-IDs and is materialised here when the first inline-label prefix retires.

- **Third extractor.** The coverage-matrix gate (per GOV-3) gains a third regex used by the inline-label arm: `\b(<alphabetic-tail-inline-label-prefix-alternation>)-[a-z]\b|\b(<numeric-tail-inline-label-prefix-alternation>)-[1-9][0-9]*\b`, a two-armed alternation whose prefix groups are partitioned by the `Tail form` column of the per-page inline-label prefix table above. `<alphabetic-tail-inline-label-prefix-alternation>` is built at gate time from exactly those live inline-label prefixes whose row carries `Tail form` = `alphabetic` (currently `HC3`); `<numeric-tail-inline-label-prefix-alternation>` is built at gate time from exactly those live inline-label prefixes whose row carries `Tail form` = `numeric` (currently `NOCEIL`). Each arm consumes only its registered prefixes paired with its registered tail form, so cross-class malformed shapes (e.g. `HC3-7`, `NOCEIL-a`) are not extracted by this regex — matching the *Tightened unknown-prefix detector*'s tail-aware semantics below and discharging the `Tail form` column's "consulted by the *Third extractor*" claim above. Neither alternation is hard-coded. Leading zeros in the numeric tail are forbidden on the same terms as GOV-3. Exclusion stripping (fenced code blocks, HTML comments, inline code spans) applies on the same terms as GOV-3.
- **Tightened unknown-prefix detector.** GOV-3's unknown-prefix detector is tightened to `\b[A-Z][A-Z0-9]{1,5}-(?:[1-9][0-9]*|[a-z])\b` (the prefix character class widens from `[A-Z]{2,4}` to `[A-Z][A-Z0-9]{1,5}` — both lengthening from 2–4 to 2–6 to admit `NOCEIL` and admitting ASCII digits in non-leading positions to admit `HC3`, parallel to the inline-label prefix grammar in *Per-page inline-label prefix table* above and intentionally diverging from GOV-3's letters-only REQ-ID prefix grammar — and the tail admits the single-lowercase-letter form to admit `HC3-a` … `HC3-e`). The membership check is *tail-aware* and is structured as two jointly exhaustive arms — a token of shape `<prefix>-<tail>` matching the widened regex passes the gate iff one of the following holds, and fails the gate (as an unknown-prefix detector hit) otherwise:

  - **REQ-ID arm.** `<prefix>` appears in the union of (live REQ-ID prefix table, *Retired prefixes* sub-table) AND `<tail>` matches the numeric form `[1-9][0-9]*`. REQ-ID prefixes never carry the alphabetic tail form, so a REQ-ID-table prefix paired with a single-lowercase-letter tail (e.g. `BNDR-a`, `CEIL-c`, `GOV-x`) is an unknown-prefix detector hit even though the prefix is enumerated.
  - **Inline-label arm.** `<prefix>` appears in the union of (live inline-label prefix table, retired inline-label prefix sub-table) AND `<tail>` matches the prefix's pinned tail form per the `Tail form` column of *Per-page inline-label prefix table* above (where retired inline-label prefixes carry the same pinned tail form they held when live). An inline-label-table prefix paired with the wrong tail form (e.g. `HC3-7` under `HC3`'s pinned `alphabetic`, or `NOCEIL-a` under `NOCEIL`'s pinned `numeric`) is an unknown-prefix detector hit even though the prefix is enumerated.

  The two arms are jointly exhaustive over the four (table-side × tail-form) combinations: every token matching the widened regex is classified by exactly one of (REQ-ID arm passes, inline-label arm passes, neither arm passes — fail). A gate implementer MUST evaluate the tail-form check on both arms; a membership check that consults only the prefix tables and ignores the `Tail form` column does not implement this rule and silently admits the cross-class malformed tokens enumerated above. The widened membership union (live REQ-ID prefix table, *Retired prefixes* sub-table, live inline-label prefix table, retired inline-label prefix sub-table) closes GOV-3's "we caught everything" guarantee against the new identifier class without permitting a third unenumerated prefix to slip through. The tightened detector regex supersedes the regex in GOV-3 *Unknown-prefix detector* above for the unknown-prefix gate; the GOV-3 *Primary extractor* and the inline-label *Third extractor* above are unchanged by this clause.
- **Lifecycle.** Inline labels follow the GOV-8 lifecycle (*Split*, *Merge*, *Deletion*, *Pure rewording*) on identical terms — substantive edits MUST be modelled as a split, merge, or deletion-plus-add; pure rewording leaves the label unchanged. The GOV-8 *Pure rewording* worked examples (modal strengthening / weakening, diagnostic-code addition / removal, edits to attached non-normative examples) apply verbatim to inline labels.
- **Retirement section.** All inline-label retirements MUST be recorded in a trailing `## Retired inline labels` section on each page that owns inline-label prefixes (skeleton inserted by the initial REQ-ID anchor pass per GOV-8), structurally parallel to the `## Retired REQ-IDs` section.
- **Cross-link form.** GOV-9's cross-link contract applies to inline labels on identical terms — every cross-page reference to an inline label MUST resolve as a `#prefix-tail` URL fragment (rendered lowercase per *GOV-1 dual-form layout applies* above), so any cross-page link of the form `…#hc3-a` or `…#noceil-3` resolves to the depended-upon label's anchor site.
- **GOV-12 lock-step coverage.** GOV-12's lock-step convention names `HC3-a`…`HC3-e`, `CIO-1`…`CIO-6`, and `NOCEIL-1`…`NOCEIL-4` explicitly. That enumeration is correct under this rule and stays as-written; a PR that adds, removes, or renames any of the identifiers enumerated above on `hard-ceilings.md` (whether an inline label under this rule or a `CIO-N` REQ-ID under GOV-1 / GOV-3 / GOV-8 / GOV-9) MUST update the corresponding `spec.md` aggregator in the same commit per GOV-12.
