# Patch skew degradation

   <a id="patch-skew-degradation-contract"></a>
   **Patch-skew degradation contract.** The *patch-skew window* is the interval during which a user's installed `@earendil-works/pi-coding-agent` patch is newer than the theta patch on the same machine — the [theta 1.0 Pi-SDK pin range](./host-prerequisites.md#pi-sdk-pin) permits patch-only Pi updates without re-running [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below, so the Pi-side `SessionShutdownEvent['reason']` union and the theta-side `literals` snapshot can transiently disagree. theta 1.0 assumes a Pi patch release preserves every unpinned host presupposition catalogued in [Host prerequisites](./host-prerequisites.md) — including the bundler-output *known-fragile evidence* sites those presuppositions cite — so the patch-skew allowance is not confined to the `SessionShutdownEvent['reason']` surface this contract specifies: a Pi patch that violates any catalogued presupposition reaches users with no theta-side build-time gate, and detection routes only through the next minor bump's editorial-review checklist under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below. During the window, the runtime fallback above (the `theta/host/session-shutdown-reason-unknown` diagnostic and the full five-sub-step teardown for any reason not in the snapshot's `literals`) is the **sole** user-facing contract for any new reason; no other surface promises observability or graceful degradation. The skew is detected by the bidirectional type-equality assertion below firing red on the next theta typecheck cycle that imports the candidate Pi patch — theta does **not** run an automated cadence (no Renovate, no scheduled CI re-install, no nightly cron) that proactively re-imports patches in theta 1.0, so the build-time gate is best-effort: the window's duration depends on contributor cadence and is not bounded by spec, and it detects the skew only as fast as a theta contributor next runs `npm run typecheck` against the candidate patch (the window itself closes only when a release shipping the resulting snapshot edit reaches the user; an automated cadence for re-importing Pi patches is deferred to [Future Considerations — Tooling deferrals](../future-considerations.md#tooling-deferrals-no-v1-impact)). The window is silent on the user side until a non-snapshot reason actually arrives at runtime (routed through the unknown-reason path above) or the contributor's next typecheck cycle catches the drift; this is intentional theta 1.0 behaviour. The contract is asymmetric across widening and narrowing of `SessionShutdownEvent['reason']`, with the mixed widening+narrowing case handled by the **Mixed** sub-rule below. **Widening** (a Pi patch adds a new literal): every shutdown carrying the new literal routes through the unknown-reason path above, with `details.observed` carrying the `String(event.reason)` coercion of the new literal so operator triage identifies the cause without a code change; the full five-sub-step teardown still runs and the snapshot edit step 5 below prescribes is the theta-side resolution. **Narrowing** (a Pi patch removes a literal): the snapshot's `literals` array is transiently superset to the SDK union, but Pi never delivers the removed literal at runtime, so the stale snapshot entry is harmless — no runtime regression, no diagnostic, no observable behaviour change. The build-time bidirectional assertion fires red on the next typecheck (the SDK-union side becomes the strict subset and the right-to-left direction breaks); the only required follow-up is the snapshot-edit-only sweep under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) step 5 below (which step 1's carve-out keeps prose-update-free). **Mixed** (a Pi patch both adds and removes literals in the same release — rename, replace, or swap): the patch decomposes into its widening half (the added literal is treated per the **Widening** rule above — runtime routes the new literal through the unknown-reason path with `details.observed` carrying the coerced value) and its narrowing half (the removed literal is treated per the **Narrowing** rule above — the snapshot's `literals` array is transiently superset and the stale entry is harmless until the snapshot edit ships); both halves are observed and handled independently, and no composite handling is owed. Three behaviours pin this contract and MUST be covered by build-time fixtures; the (P-a)/(P-b)/(P-c) labelling here uses a `"P-"` prefix to be deliberately distinct from both the (i)/(ii)/(iii) sub-anchor labelling of the *Sub-anchors and pinned substrings* paragraph immediately above **and** the (a)–(d) anchor-stable substring labelling of that same paragraph (the substring labels are anchor-stable surface and cannot themselves be relabelled), so the three adjacent enumerations cannot collide on cross-reference: (P-a) an *unknown-reason runtime* fixture that synthesises a `session_shutdown` event whose `reason` is not a snapshot `literals` member and asserts the handler emits exactly one `theta/host/session-shutdown-reason-unknown` with `details.observed` carrying the coerced literal, then runs the full five-sub-step teardown; (P-b) a *snapshot-only widening* fixture pair that constructs a synthetic value with the same shape as the production `SDK_SURFACE_INVENTORY`'s `SessionShutdownEvent['reason']` snapshot entry — same composite-predicate keys (`kind: "type-union-snapshot"`, `path: "SessionShutdownEvent.reason"`) and same `literals: [...] as const` shape — whose `literals` array contains one extra string `"synthetic-new-reason"` not present in the production constant (no mutation of the production `SDK_SURFACE_INVENTORY` import — the `as const` declaration on the production constant remains intact and is not written through), and exercises the runtime's set-membership check against that synthetic value through a *test-only* substitution of the production `SDK_SURFACE_INVENTORY` module export — the substitution mechanic MUST preserve the handler's static import of `SDK_SURFACE_INVENTORY` from the source-of-truth module (sub-anchor (ii) of the **Unknown-reason rule** above); the choice of primitive that achieves that preservation is an implementation choice. PIC pins only the static-import-preservation behavioural property above and the behavioural assertions below; the synthetic constant's exact source is an implementation choice. The fixture pair asserts both directions: **(P-b-1, positive)** with the substitution applied so the handler's static import resolves to the synthetic constant, a `session_shutdown` event with `reason: "synthetic-new-reason"` is accepted as a closed-set member — **no** `theta/host/session-shutdown-reason-unknown` is emitted, the full five-sub-step teardown still runs, and no other failure-mode diagnostic from the **Unknown-reason rule** above fires; **(P-b-2, negative)** a parallel test in which the substitution is **not** applied (so the handler's static import resolves to the unmodified production `SDK_SURFACE_INVENTORY`, whose `literals` array does not contain `"synthetic-new-reason"`) MUST emit exactly one `theta/host/session-shutdown-reason-unknown` for the same `reason: "synthetic-new-reason"` event. The differential between (P-b-1) and (P-b-2) — same input event, opposite outcomes, with the only varying factor being whether the substitution is applied — proves the handler reads through the live module export rather than a hardcoded copy of the snapshot's `literals` field (a hardcoded copy would not swing on the substitution state and the two halves would observe the same outcome). The pair together — and only together — proves the single-source-of-truth wiring above; (P-c) a *narrowing no-regression* fixture pair that proves the asymmetric-skew property — runtime tolerance plus build-time gating — across two separate test phases that cannot share a single process invocation (the runtime-acceptance assertion requires compiled code to run, while the build-time-red assertion requires `tsc` to fail, and a `tsc` failure aborts `npm run typecheck` before any vitest harness starts; no single-process harness can assert both, so the obligation is split into two sub-fixtures with the same logical pairing pattern as (P-b-1)/(P-b-2)). **(P-c-1, runtime)** constructs a synthetic snapshot value with the same shape as the production `SDK_SURFACE_INVENTORY`'s `SessionShutdownEvent['reason']` snapshot entry whose `literals` array is a strict superset of the SDK union (e.g., the SDK has narrowed to drop `"fork"` but the synthetic snapshot still pins it alongside the rest of the narrowed union), substitutes it through the same static-import-preserving substitution mechanic (P-b) uses, dispatches a `session_shutdown` event with `reason: "fork"` (the previously-snapshotted literal that the synthetic SDK union no longer contains), and asserts the runtime accepts it as a closed-set member — **no** `theta/host/session-shutdown-reason-unknown` is emitted, the full five-sub-step teardown still runs, and no other failure-mode diagnostic from the **Unknown-reason rule** above fires. **(P-c-2, build-time)** is a separate test (one whose harness is the `tsc` invocation itself, not a vitest case) that exercises the bidirectional type-equality assertion of [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) step 5 below against a synthetic SDK type one literal narrower than the synthetic snapshot, and asserts the assertion produces a `tsc` diagnostic carrying the literal brand string `theta/typecheck/session-shutdown-reason-snapshot` verbatim. The two sub-fixtures together — and only together — prove the asymmetric-skew property the **Narrowing** paragraph above pins. In each of (P-a), (P-b-1), and (P-c-1), the assertion that *the full five-sub-step teardown still runs* is not satisfied by witnessing a single teardown observable: the fixture MUST assert that each of the eight call sites enumerated in the closed `details.call` label set of the **Per-step isolation** paragraph below was invoked on this teardown, via spies on the injected `ThetaRegistry`, `chokidar`-watcher, `Clock`, and signal-listener seams. Sub-steps 2 and 3 carry no `details.call` label and are witnessed separately: the fixture MUST assert that sub-step 2's per-entry `ActiveInvocationRegistry` iteration ran (via a per-entry registry-iteration spy) and that sub-step 3's bounded `Promise.allSettled` await over the entries' `disposeBarrier`s ran (via an injected `disposeBarrier` or `Clock.setTimeout` spy). For (P-c-1) specifically, the fixture MUST use a fresh `ThetaRegistry` per case so sub-step 1's `thetaRegistry.drain` observable is witnessed even when an earlier shutdown in a longer-running harness already drained a shared registry. The fixtures are normative-by-reference here; the implementation owns their `<file>` placement and exact assertion shape.

   *Fixture-obligation categories.* This contract and the [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below together pin four distinct *categories* of fixture obligation the implementation MUST cover — the headline count "four" refers to categories, not individual fixtures, since each category may comprise multiple sub-fixtures. The categories are: the three *patch-skew* fixture-obligation categories named immediately above (the *unknown-reason runtime* category, the *snapshot-only widening* category whose (P-b-1, positive) and (P-b-2, negative) sub-fixtures together comprise it, and the *narrowing no-regression* category whose (P-c-1, runtime) and (P-c-2, build-time) sub-fixtures likewise together comprise it), and the *per-sub-trigger negative-test fixtures* category named in [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) step 5 below (one sub-fixture per Step-0-probe-constant failing entry under [trigger (i)](./version-bump-triggers.md#bump-step-5-trigger-i), and one sub-fixture per `SessionShutdownEvent['reason']` snapshot-entry sub-trigger under [trigger (ii)](./version-bump-triggers.md#bump-step-5-trigger-ii)). PIC does not enumerate concrete `<file>` placements or assertion shapes; those are implementation choices.
