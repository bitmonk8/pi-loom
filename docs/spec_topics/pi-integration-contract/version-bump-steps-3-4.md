# Version bump steps 3–4

3. **Re-confirm the `engines.node` floor.** The build-time `package.json` `engines.node` literal-read test asserts a three-way equality across (i) the loom `package.json#engines.node` literal, (ii) the in-repo pinned floor recorded in the `pi-engines-node` row of `SDK_SURFACE_INVENTORY`, and (iii) the `engines.node` floor read live at build time from the installed `@earendil-works/pi-coding-agent` `package.json` under `node_modules` at the candidate version. Operand (iii) is the only live read; because it reflects the installed dependency rather than an in-repo literal, a Pi minor bump that moves the upstream floor changes (iii) while operands (i) and (ii) stay pinned, so the test fails red at the `npm test` invocation rather than at user runtime and no contributor-side manual compare is required here. If the test fails red, the upstream floor has moved; update the loom `package.json#engines.node` literal (operand (i)), [Step 0 (a)](./capability-probe.md#entry-capability-probe) above (the comparator and floor reference), the informative-orientation sentence in [`spec.md` — Host runtime precondition 1](../overview-and-orientation.md#orientation), and the `pi-engines-node` row's `literal` field in `SDK_SURFACE_INVENTORY` (operand (ii), the in-repo pinned copy of the upstream floor that the assertion compares against the live read (iii)) in the same edit; if the test stays green, the floor has not moved and no edit is needed.
4. **Update the version pin in `peerDependencies` and the equivalent literal here.** Move all four `peerDependencies` entries together (the `peerDependencies` literal-read test, whose assertion shape is stated below, enforces the joint move) and update the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) literal in the same commit (this is the single in-corpus restatement-source per the [Manifest lock-step](./host-prerequisites.md#pi-sdk-pin-manifest-lock-step) sub-paragraph above: every other in-corpus reference to the pin cites the anchor by location). Splitting these into separate commits leaves a window in which that test is red against `main`; do not split. The `peerDependencies` literal-read test compares each of the four `@earendil-works/*` entries in `package.json#peerDependencies` (`pi-coding-agent`, `pi-agent-core`, `pi-ai`, `pi-tui`) against the canonical Pi-SDK-pin literal at [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin), using string-equal (byte-equal) comparison, per the [Manifest lock-step](./host-prerequisites.md#pi-sdk-pin-manifest-lock-step) sub-paragraph above; the test passes iff all four entries are byte-equal to that single source-of-truth literal. The `typebox` entry is excluded from this iteration — it is pinned to `"*"` and asserted separately by its own one-line literal-read assertion, per the [Manifest lock-step](./host-prerequisites.md#pi-sdk-pin-manifest-lock-step) carve-out above. The "joint move" property follows from this assertion shape — any one of the four entries diverging from the pin literal fails the test red — rather than being the assertion itself. The deliberate deviation from Pi's bundled-package convention recorded in **Host prerequisites — Pi SDK pin** above (the *Deliberate deviation from Pi's bundled-package convention* sub-paragraph) MUST be re-written at this step whenever the candidate Pi minor changes the lock-step expectation `packages.md` prescribes — namely if `packages.md` at the candidate minor stops naming the `@earendil-works/*` four as `"*"`-range packages, or if Pi changes the `pi-mono` release model so that the four packages are no longer guaranteed to ship together at a single minor. A satisfying re-write MUST (a) name which of those two triggering changes fired and cite its candidate-minor source (the `packages.md` revision, or the `pi-mono` release-model statement), and (b) state whether the deviation rationale stands as written or is invalidated and replaced; a re-write that names neither triggering change, or that only cosmetically rewords the existing rationale without making the (a)/(b) determination, does not satisfy this obligation. In either case, update the *Deliberate deviation* sub-paragraph together with the `peerDependencies` literal-read test, the [Step 0 (d) Peer-dep version](./capability-probe.md#entry-capability-probe) probe, and the literal-anchor list in [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) above — all four are updated together whenever this step's `packages.md` / `pi-mono` trigger fires, not conditionally per artefact — in the same commit; if `packages.md` and the `pi-mono` release model are unchanged at the candidate minor, the deviation rationale stands as written and no edit to that sub-paragraph is required.
