# Type System

The type system is JSON-native. Type expressions are built from:

- **Primitive types**: `string`, `number`, `integer`, `boolean`, `null`
- **Named types**: any schema or enum identifier in scope (`Author`, `ReviewScore`)
- **Generic types**: `array<T>`, `Result<T, E>` (and any future parameterised type) ‚ÄĒ angle-bracket type parameters are the uniform Loom convention
- **Union types**: `T | U | ...` ‚ÄĒ the `|` operator is the lowest-precedence type operator and is legal anywhere a type is
- **Literal types**: `"..."`, `42`, `true`, `false`, `null` ‚ÄĒ string, number, and boolean literals are valid type expressions (single-arm "unions" are how `kind: "validation"`-style const fields are expressed)
- **Inline anonymous objects**: `{ field: T, ... }` ‚ÄĒ legal in any type position, but named schemas are preferred for reuse and for getting a name in error messages
- **Inline arrays**: not a separate form ‚ÄĒ use `array<T>` (no `T[]`, no `[T]`)

`void` is **not** a primitive value type and is deliberately absent from the list above: it is admitted only as a function- or loom-**return** annotation meaning "intentionally produces no value", never in a value-bearing type position (see [Grammar Appendix ‚ÄĒ Type grammar](./grammar.md#type-grammar) and [Function Definitions ‚ÄĒ Empty-tail body](./functions.md#empty-tail-body)). A `void` in any other type position is `loom/parse/void-in-non-return-position`, and `void` does not participate in type compatibility ([below](#type-compatibility)).

The same type grammar applies in every type-annotation position: schema fields, frontmatter `params:`, `let x: T`, function parameters, and `@<T>`...`` explicit query schemas; the function- and loom-return position additionally admits the return-only `void` annotation.

Subsections of the type system are split into their own pages:

- [Schema Declarations](./schemas.md) ‚ÄĒ `schema X { ... }`, `schema X = ...`, `enum`, discriminated unions, recursion, wire-name renaming.
- [Descriptions](./descriptions.md) ‚ÄĒ `///` doc comments and field separator rules.
- [Schema Subset](./schema-subset.md) ‚ÄĒ the JSON-Schema subset Loom targets, plus the lowering algorithm.

## Type compatibility

Wherever the spec asks whether a value of static type `T‚āĀ` may be used in a position that expects type `T‚āā` ‚ÄĒ the RHS of a typed `let`, a function-argument slot, an `invoke<T>` return annotation, the common type of `match` arms or ternary branches, an `array<T>` element against its sink, the `+` operator's mixed-numeric case, a frontmatter `params:` default ‚ÄĒ the answer is governed by the single relation `T‚āĀ ‚äĎ T‚āā` (read "`T‚āĀ` is compatible with `T‚āā`"). The relation is normative; every site that previously phrased itself as "the same rules as `let`" or "a common type" cites this section by anchor.

**Operational definition.** Whenever `T‚āĀ ‚äĎ T‚āā` holds, every value statically typed as `T‚āĀ` AJV-validates against the lowering of `T‚āā` (see [Schema Subset ‚ÄĒ Lowering Algorithm](./schema-subset.md#lowering-algorithm)); AJV-validation is thus a *necessary* condition for compatibility, not a sufficient one. The AJV reading is the safety net at runtime; the parser is required to recognise the structural cases enumerated below without falling back to it, so that compatibility failures surface as parse errors at the offending source span rather than as runtime validation errors at a downstream call site. Because named schemas are nominal ([TYPE-10](#type-10)), the parser admits *fewer* pairs than AJV alone would: a pair the parser rejects is `T‚āĀ ‚čĘ T‚āā` even when the two lowered fragments are AJV-interchangeable (e.g. a named schema and an inline object with identical fields). The parser's rejection is authoritative; AJV admitting the lowered pair at runtime does not retroactively make the pair `‚äĎ`.

**Structural cases the parser must recognise.** The following hold without invoking AJV. The list is closed for V1 ‚ÄĒ anything outside it that the parser cannot decide statically is reported as a type mismatch (`loom/parse/*-type-mismatch` at the call site, e.g. `loom/parse/invoke-return-type-mismatch`, `loom/parse/invoke-arg-type-mismatch`, `loom/parse/array-element-type-mismatch`, `loom/parse/match-arm-type-mismatch`), unless the position is one where a runtime AJV check is documented as the safety net (e.g. an `invoke` against a callee that is not statically resolvable).

| ID | Rule | Notes |
|---|---|---|
| <a id="type-1"></a> **TYPE-1.** | Reflexivity: `T ‚äĎ T` for every type. | Identical primitives, identical named schemas, identical inline objects. |
| <a id="type-2"></a> **TYPE-2.** | `integer ‚äĎ number`. | One-way only; the reverse is `loom/parse/integer-narrowing` (see [Lexical ‚ÄĒ Number literals](./lexical.md)). |
| <a id="type-3"></a> **TYPE-3.** | Literal-to-primitive: `L ‚äĎ T` when `L` is a literal type and the value `L` would be statically typed `T` in expression position. | E.g. `"validation" ‚äĎ string`, `42 ‚äĎ integer`, `42 ‚äĎ number`, `true ‚äĎ boolean`, `null ‚äĎ null`. |
| <a id="type-4"></a> **TYPE-4.** | Variant-to-union: for any discriminated union `schema U = A \| B \| ...`, every variant satisfies `A ‚äĎ U`. | The "narrower callee under wider annotation" case (`Cat ‚äĎ Animal`). |
| <a id="type-5"></a> **TYPE-5.** | Union-widening: `T ‚äĎ T \| U` for any `U`. | Combined with [TYPE-4](#type-4), `A ‚äĎ A \| B` even when the union is anonymous. |
| <a id="type-6"></a> **TYPE-6.** | Union-distributive: `T‚āĀ \| T‚āā ‚äĎ T‚āÉ` iff `T‚āĀ ‚äĎ T‚āÉ` and `T‚āā ‚äĎ T‚āÉ`. | Each arm must individually satisfy the target. |
| <a id="type-7"></a> **TYPE-7.** | Element-wise on arrays: `array<T‚āĀ> ‚äĎ array<T‚āā>` iff `T‚āĀ ‚äĎ T‚āā`. | Covariant ‚ÄĒ `array<Cat> ‚äĎ array<Animal>`. Already implied by the array-LUB rule in [Expressions ‚ÄĒ array construction](./expressions.md#object-construction-array-construction-and-operator-rules); restated here so it can be cited from non-array sites. |
| <a id="type-8"></a> **TYPE-8.** | Field-wise on inline object types: an inline object type (`ObjectType`, `{ f‚āĀ: T‚āĀ, ... }`; see [Grammar ‚ÄĒ Type grammar](./grammar.md#type-grammar)) is `‚äĎ` another inline object type with the same declared field set `{ f‚āĀ: U‚āĀ, ... }` iff `TŠĶĘ ‚äĎ UŠĶĘ` for every `i`. This rule covers inline object types only; a `NamedType` resolving to an object schema is nominal ‚ÄĒ see [TYPE-10](#type-10). | Field sets must match exactly: every Loom-lowered object schema carries `additionalProperties: false` (see [Schema Subset](./schema-subset.md)), so excess properties never widen ‚ÄĒ there is no TS-style structural-subtyping admission of extra fields, and an object with even one extra declared field is not compatible. Field order is irrelevant. The field-wise admission here never crosses the inline/named boundary: name identity governs named schemas ([TYPE-10](#type-10)). |

Rules outside this list are deliberately **not** part of loom 1.0 compatibility: function-parameter contravariance, optional-field widening, excess-property tolerance on objects, and `number ‚äĎ integer` are all rejected. A future widening of the relation is non-breaking iff it only admits new pairs.

**`void` does not participate in `‚äĎ`.** `void` is a return-only annotation, not a value-bearing type (see [Grammar Appendix ‚ÄĒ Type grammar](./grammar.md#type-grammar)); no value is ever statically typed `void`, so `void` appears on neither side of a `T‚āĀ ‚äĎ T‚āā` check. A `void` in any non-return type position is rejected at parse time as `loom/parse/void-in-non-return-position` before any compatibility question arises.

**Unresolvable operands.** When either side of a compatibility check is past the parser's static view (e.g. an inferred binding whose RHS depends on a Pi-tool call whose registered schema is not visible at parse time, or an `invoke` against a callee that produced `loom/load/callee-has-errors`), the parse-time check is skipped and the runtime AJV check is the safety net. The skipping is the same posture documented for the unresolvable-callee case in [Invocation ‚ÄĒ Typed return](./invocation.md) and the unresolvable-tool case in [Tool Calls](./tool-calls.md); cite this paragraph rather than restating it.

<a id="type-9"></a> **TYPE-9.** Three of the compatibility check sites listed at the top of this section are not covered by the invoke / array / match codes named under *Structural cases* above; each reports its own parse-time diagnostic on a static failure (`T‚āĀ ‚čĘ T‚āā`, both operands statically resolvable). The RHS of a typed `let x: T = expr` reports `loom/parse/let-rhs-type-mismatch`. A plain function-argument slot ‚ÄĒ an argument to a top-level `fn` call that is neither an `invoke(...)` nor a `.loom`-callable call ‚ÄĒ reports `loom/parse/fn-arg-type-mismatch`. A ternary `cond ? a : b` reports through the array-and-ternary common-type machinery in [Expressions ‚ÄĒ Array construction](./expressions.md#object-construction-array-construction-and-operator-rules): `loom/parse/array-element-type-mismatch` when a branch fails against an in-scope sink, and `loom/parse/array-no-common-type` when no sink narrows two branches that share no common type. The `let`- and `fn`-site rows are registered in [Diagnostics ‚ÄĒ code registry](./diagnostics/code-registry-parse.md#code-registry).

<a id="type-10"></a> **TYPE-10.** Named schemas are nominal. A `NamedType` whose declaration is an object schema `schema X { ... }` (see [Schema Declarations ‚ÄĒ Object schema](./schemas.md#object-schema)) participates in `‚äĎ` only through reflexivity against the same named schema ([TYPE-1](#type-1)), variant-to-union membership ([TYPE-4](#type-4)), and union widening / distribution ([TYPE-5](#type-5)/[TYPE-6](#type-6)). A named-schema value is **not** `‚äĎ` an inline object type with the same field shape, and an inline-object value is **not** `‚äĎ` a named schema with the same field shape ‚ÄĒ the cross-form pair is `T‚āĀ ‚čĘ T‚āā` regardless of field shape. Two *distinct* named schemas with byte-identical lowered fragments are likewise incompatible: name identity, not lowered identity, drives `‚äĎ`, so any canonical-hash slug coincidence between them is irrelevant. A cross-form or cross-named-schema mismatch is reported at parse time on the offending site through that site's own diagnostic (`loom/parse/let-rhs-type-mismatch`, `loom/parse/fn-arg-type-mismatch`, or `loom/parse/invoke-arg-type-mismatch`; see [TYPE-9](#type-9)), not deferred to a runtime AJV failure. To move a value across the boundary, assign through an explicit construction or a typed `let` that names the target schema.
