# pi-loom — Normative Requirements Inventory

Backbone for the end-to-end test campaign. Every row is one independently
checkable, observable behaviour extracted from the specification (`docs/spec.md`,
`docs/spec_topics/**`) and the normative Reference pages (`docs/reference/**`).

## How to read this document

Each requirement row has five columns:

- **Campaign ID** — stable `REQ-<AREA>-<n>` handle for the test plan. Area
  prefixes are globally unique across this document.
- **Spec tag** — the original stable spec tag when one exists (e.g. `QRY-22`,
  `TYPE-1`, `HC3-a`, `SUBS-1`, `DISC-5`, `SNK-a`, `INV-4`, `ERR-1`), else `—`.
- **Normative statement** — one testable sentence describing observable behaviour.
- **Citation** — `path:line` or `path §section` into the spec/reference corpus.
- **Testability** — one of:
  - `offline-unit` — pure parser / binder / type-checker / schema-lowering /
    ceiling-arithmetic logic; no model, no composition harness.
  - `conformance` — production composition / registration / error-routing path
    exercised without a real model round-trip (fakes/harness).
  - `live` — needs a real (or mock) provider/model turn.
  - `manual/inspection` — editorial / terminology / host-integrator wiring
    verified by code or doc inspection; not runtime-asserted.

Requirements the spec marks **deferred / not-yet-implemented / out-of-scope /
non-goal** are collected in the final appendix so the test plan does not assert
unshipped behaviour as if it were shipped.

### Cluster 1 — Language core

- Hex / octal / binary number forms and underscore separators are not in loom 1.0 (surface as `loom/parse/unsupported-feature`). — docs/spec_topics/lexical.md §Number literals
- No integer-division operator in loom 1.0 (`/` always produces `number`); deferred. — docs/spec_topics/expressions.md §Other arithmetic
- No user-defined methods and no `this`; additional built-in methods may be added non-breakingly later. — docs/spec_topics/expressions.md §Built-in methods and properties
- loom 1.0 exposes no `charAt`/`codePointAt`; strings are decomposed via `split(...)`. — docs/spec_topics/expressions.md §Built-in methods and properties
- `match` guards (`Ok(x) if ...`) and rest patterns (`[first, ...rest]`, `{ k, ...other }`) are not in loom 1.0 (`loom/parse/match-guard-not-supported`, `loom/parse/rest-pattern-not-supported`). — docs/spec_topics/expressions.md §Pattern grammar
- `match` exhaustiveness is not statically checked in loom 1.0 (would be unsound); non-coverage is a runtime `MatchError`. — docs/spec_topics/expressions.md §`match` expression
- Arrow functions / higher-order methods (`map`/`filter`/`reduce`), spread/rest, optional chaining/nullish coalescing, strict equality, bitwise operators, `++`/`--`, comma operator, nested template strings — all unsupported in loom 1.0. — docs/spec_topics/expressions.md §Not supported
- Closures and first-class function values are not part of loom 1.0 (function names appear only in call position). — docs/spec_topics/functions.md:20 (FN-1)
- Function parameters are immutable — a deliberate loom 1.0 simplification (mutable params deferred). — docs/spec_topics/bindings.md §Immutable contexts
- `break expr` (value-carrying break) is not in loom 1.0 (`loom/parse/break-with-value`). — docs/spec_topics/control-flow.md §`break` / `continue`
- `fn` parameters carry no defaults in loom 1.0 (literal-valued defaults only on `params:` frontmatter fields). — docs/spec_topics/grammar.md §`fn` declarations
- A future widening of the `⊑` relation (contravariance, optional-field widening, excess-property tolerance, `number ⊑ integer`) is explicitly excluded from loom 1.0; non-breaking only if it admits new pairs. — docs/spec_topics/type-system.md §Type compatibility
- Bun, Deno, browser embeds, and other non-Node JavaScript hosts are out of loom 1.0 scope; adding a non-Node host is a probe re-design deferred. — docs/spec_topics/runtime-value-model.md §JavaScript engine assumptions
- If a future host-interop surface exposes a live `Result`/enum value to JS, the runtime value-model section must be revisited before that surface ships. — docs/spec_topics/runtime-value-model.md §Reference encoding (non-normative)
